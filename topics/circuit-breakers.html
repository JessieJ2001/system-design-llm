Circuit Breakers

A resilience pattern that detects failing downstream services and fails fast, preventing cascading failures from bringing down your entire system.

22 min read

What Is It?

A circuit breaker is a resilience pattern that wraps calls to downstream services and monitors them for failures. When failures exceed a configured threshold, the breaker "trips" and immediately rejects subsequent calls without even attempting them. This prevents your service from wasting threads, connections, and time waiting on a backend that is already down — and more critically, prevents the failure from cascading upstream through the entire call chain.

The analogy is an electrical circuit breaker in your home. When a short circuit causes dangerous current to flow, the breaker trips and cuts power to that circuit. This protects the wiring from melting and your house from catching fire. You don't keep trying to push electricity through a broken circuit — you stop, let things cool down, and then carefully test whether it's safe to reconnect.

A circuit breaker has three states: Closed (normal operation — requests flow through, failures are counted), Open (failing fast — all requests are immediately rejected without calling the downstream service), and Half-Open (testing recovery — a limited number of probe requests are let through to check if the downstream service has recovered). If the probes succeed, the breaker closes again. If they fail, it reopens.

Circuit Breaker State Machine

How It Works

The circuit breaker operates as a state machine with three states and well-defined transitions:

  1. Closed (normal): All requests pass through to the downstream service. The breaker maintains a rolling window of recent results. Every failure increments a failure counter. Every success resets or decrements it (depending on implementation).
  2. Threshold exceeded — trip to Open: When the failure count (or failure rate) within the sliding window exceeds the configured threshold, the breaker trips. The state transitions from Closed to Open. A timeout timer starts.
  3. Open (failing fast): All incoming requests are immediately rejected with a fallback response or error — the downstream service is never called. This protects both the caller (no wasted threads/connections) and the downstream service (no additional load while it's struggling).
  4. Timeout expires — transition to Half-Open: After a configured timeout duration (e.g., 30 seconds), the breaker transitions to Half-Open. This is the recovery-testing phase.
  5. Half-Open (probing): A limited number of requests (often just one) are allowed through to the downstream service. These are "probe" requests that test whether the service has recovered.
  6. Probe succeeds — close the breaker: If the probe request(s) succeed, the breaker transitions back to Closed. Normal traffic resumes. The failure counter is reset.
  7. Probe fails — reopen the breaker: If any probe request fails, the breaker transitions back to Open and the timeout timer resets. The cycle repeats.

This creates a self-healing loop: the system automatically detects failure, stops hammering the broken service, periodically tests for recovery, and resumes normal operation when the downstream service is healthy again — all without human intervention.

Step-by-Step State Machine Walkthrough

Let's trace through a complete lifecycle with a concrete scenario. Our circuit breaker is configured with a failure threshold of 5 consecutive failures and an open-state timeout of 30 seconds. The downstream payment service starts healthy, goes down, and eventually recovers.

Notice how step 4 is crucial for understanding the consecutive failure counting strategy: a single success resets the counter to zero. This is both a strength (avoids false positives during intermittent issues) and a weakness (one lucky success during a real outage resets the count and delays tripping). We'll explore alternative counting strategies below.

State Transition Rules Summary

  • CLOSED → OPEN: When failure count reaches the threshold (5 consecutive failures in this example). The trigger can also be a failure rate exceeding a percentage — see "Failure Counting Strategies" below.
  • OPEN → HALF-OPEN: Automatically, after the timeout period expires (30 seconds). No human intervention needed.
  • HALF-OPEN → CLOSED: If the probe request(s) succeed. The downstream service has recovered.
  • HALF-OPEN → OPEN: If any probe request fails. The downstream is still broken. The timeout timer restarts.

Note: there is no direct transition from OPEN → CLOSED. The system must pass through HALF-OPEN first. This is a safety mechanism — we never declare recovery without actually testing it with a real request.

When to Use / When NOT to Use

Use Circuit Breakers When

  • Calling external or unreliable services: Third-party APIs, payment gateways, or any service outside your direct control that may go down without warning.
  • Preventing cascade failures: In a microservices architecture where Service A calls Service B calls Service C — if C goes down, without a breaker, A and B both hang waiting, exhausting their thread pools, and the failure cascades to every upstream caller.
  • Protecting thread pools from exhaustion: Each pending request ties up a thread or connection. If a downstream service is slow or unresponsive, breakers prevent your thread pool from filling up with blocked requests.
  • Providing graceful degradation: When you can serve a degraded response (cached data, default values, reduced functionality) instead of failing completely when a dependency is down.
  • Synchronous RPC calls: Any blocking call where you're waiting for a response is a candidate for circuit breaking.

Don't Use Circuit Breakers When

  • Local in-process function calls: If the call is to local code, it won't fail due to network issues. A try/catch is sufficient.
  • Failures are expected and normal: Validation errors, 400 Bad Request, or business logic rejections are not signs of a failing service. Don't count these as "failures" for breaker purposes.
  • Idempotent retry is better suited: For transient one-off errors (a single timeout, a DNS blip), a simple retry with exponential backoff may be more appropriate than tripping a breaker. Use both together: retry first, then break if retries consistently fail.
  • Asynchronous messaging: For fire-and-forget messages via a queue, the queue itself provides the buffering. Circuit breaking is for synchronous call paths.
  • Database queries: Connection pool exhaustion and query timeouts are better handled by connection pool configuration and query timeouts than by a circuit breaker (though breakers can wrap DB calls in extreme cases).

Classic vs. Modern Approaches

Classic: Threshold-Based Circuit Breaker

The original pattern, popularized by Michael Nygard's Release It! (2007). A simple counter tracks consecutive failures. When the count exceeds a fixed threshold (e.g., 5 failures), the breaker trips. After a fixed timeout, it transitions to half-open. Simple to implement and reason about, but brittle: a fixed threshold doesn't adapt to varying traffic volumes. Five failures out of 10 requests (50% error rate) is very different from five failures out of 10,000 requests (0.05% error rate), but a basic threshold breaker treats them identically.

Adaptive Circuit Breakers

Instead of a fixed failure count, adaptive breakers use a sliding window and calculate the error rate over that window. The breaker trips when the error rate exceeds a percentage threshold (e.g., 50%) within a minimum sample size. This handles varying traffic volumes correctly. Some implementations dynamically adjust the threshold based on historical error rate trends — if the baseline error rate is 2%, a spike to 15% is significant; if the baseline is 10%, 15% might be normal.

Netflix Hystrix (now in maintenance mode)

Hystrix pioneered production-grade circuit breaking at scale. It introduced thread pool isolation (each dependency gets its own thread pool so a slow service can't exhaust the shared pool), request collapsing (batching multiple requests to the same service), request caching, and a sophisticated dashboard for real-time monitoring. Hystrix used a rolling window of 10 seconds divided into 10 one-second buckets. Netflix put Hystrix into maintenance mode in 2018, recommending Resilience4j as the successor.

Resilience4j

The modern JVM library for fault tolerance. Lightweight (no thread pool isolation by default — uses a semaphore), composable (circuit breaker, retry, rate limiter, bulkhead, and time limiter can be stacked as decorators), and designed for functional programming and reactive streams. Uses a count-based or time-based sliding window. Supports automatic state transitions and configurable wait durations in the open state.

Envoy Outlier Detection (Proxy-Level Breaking)

Instead of application-code breakers, Envoy (the sidecar proxy) implements circuit breaking at the network layer. Outlier detection monitors individual upstream hosts and ejects unhealthy ones from the load balancing pool. This is more granular than a traditional circuit breaker: instead of breaking the entire circuit to a service, it removes just the failing instances. Configured via consecutive_5xx, consecutive_gateway_errors, and success_rate ejection policies.

Service Mesh Circuit Breaking (Istio)

Istio builds on Envoy's capabilities and adds centralized policy management. Circuit breaking rules are configured declaratively via DestinationRule resources, applied uniformly across the mesh without modifying application code. Supports connection pool limits (maxConnections, maxPendingRequests) and outlier detection at the mesh level. The advantage: consistent resilience policy across polyglot services.

Failure Counting Strategies

The state machine walkthrough above used consecutive failure counting — the simplest approach. But how you count failures has a huge impact on how well the breaker detects real outages versus tripping on noise. There are three main strategies, each with distinct tradeoffs.

Strategy 1: Consecutive Failures

The breaker maintains a simple counter that increments on each failure and resets to zero on any success. When the counter reaches the threshold (e.g., 5), the breaker trips.

How it works: Imagine the downstream returns these results for 12 consecutive requests:

OK, FAIL, FAIL, OK, FAIL, FAIL, FAIL, FAIL, FAIL, —, —, —

  • Request 1: OK → counter = 0
  • Request 2: FAIL → counter = 1
  • Request 3: FAIL → counter = 2
  • Request 4: OK → counter resets to 0 (the single success clears the count)
  • Request 5: FAIL → counter = 1 (starting over)
  • Request 6: FAIL → counter = 2
  • Request 7: FAIL → counter = 3
  • Request 8: FAIL → counter = 4
  • Request 9: FAIL → counter = 5 → BREAKER TRIPS OPEN
  • Requests 10-12: REJECTED (breaker is open)

The problem: Look at the overall results — 7 failures out of 9 requests (78% failure rate). The service was clearly degraded from request 2 onward. But that single success at request 4 reset the counter, delaying the trip by 4 more failures. In the worst case, a service that fails 80% of the time but succeeds every 4th request will never trip a consecutive-failures breaker with a threshold of 5.

Strategy 2: Failure Rate Over a Time Window

Instead of counting consecutive failures, track the failure rate (percentage of failed requests) over a sliding time window. If the failure rate exceeds a threshold (e.g., 50%), the breaker trips.

How it works: The breaker looks at all requests in the last W seconds. It divides failures by total requests to get the failure rate.

Example: Window = 60 seconds, threshold = 50% failure rate.

  • In the last 60 seconds: 100 requests total, 55 failed.
  • Failure rate = 55 / 100 = 55%
  • 55% > 50% threshold → BREAKER TRIPS OPEN

This is much better at detecting degraded services that intermittently succeed. The 80%-failure service from the previous example would trip this breaker quickly.

The problem: What about low-traffic endpoints? If only 3 requests were made in the last 60 seconds and 2 failed, the failure rate is 67%. That looks bad, but with only 3 data points, it's probably just bad luck — not a real outage. Tripping the breaker on 2 failures out of 3 requests is a false positive.

Strategy 3: Failure Rate with Minimum Request Threshold

Same as Strategy 2, but the breaker does not evaluate the failure rate until at least M requests have been made in the window. This prevents false positives from statistical noise in low-traffic scenarios.

Example: Window = 60 seconds, threshold = 50%, minimum requests = 20.

  • Scenario A: 3 requests, 2 failed (67% failure rate). But 3 < 20 minimum → DO NOT TRIP. Not enough data.
  • Scenario B: 25 requests, 14 failed (56% failure rate). 25 ≥ 20 minimum, and 56% > 50% threshold → TRIP.
  • Scenario C: 200 requests, 8 failed (4% failure rate). 200 ≥ 20 minimum, but 4% < 50% threshold → DO NOT TRIP. Normal error rate.

This is the strategy used by most production circuit breaker implementations, including Resilience4j and Netflix Hystrix. It adapts to traffic volume and avoids both false positives (tripping on noise) and false negatives (ignoring gradual degradation).

Comparison

Strategy Pros Cons Best For
Consecutive failures Simple to implement. Fast to trigger when failures are truly continuous. Easy to reason about. A single success during degradation resets the count, keeping the breaker closed when it should open. Cannot detect "90% failure rate" scenarios. Binary up/down services (healthy or completely dead, no partial failures)
Rate over window Handles intermittent errors correctly. Detects gradual degradation. Adapts to partial failures. Can produce false positives with low traffic (2 out of 3 failures = 67% but not meaningful). Window size tuning is tricky. High-traffic services where the sample size is always large enough

Slow to trigger on low-traffic endpoints (must wait for minimum requests). Production systems (this is the most common strategy in practice)

Adaptive Circuit Breakers: Netflix Hystrix Deep Dive

Netflix Hystrix (now in maintenance mode, succeeded by Resilience4j) popularized adaptive circuit breakers and set the standard for how production systems implement this pattern. Understanding Hystrix's approach is essential because virtually every modern circuit breaker library follows its design.

Hystrix Configuration

Hystrix uses the "failure rate + minimum threshold" strategy with a rolling window divided into time buckets:

  • Rolling window: 10 seconds (the evaluation period)
  • Bucket count: 10 buckets (each bucket = 1 second)
  • Minimum request threshold: 20 requests in the window
  • Failure rate threshold: 50%
  • Sleep window (open-state duration): 5 seconds

How the Rolling Window Works

The rolling window is the key innovation. Instead of storing individual request timestamps (which uses a lot of memory at high traffic), Hystrix divides time into fixed-size "buckets." Each bucket is a 1-second interval that stores aggregate counts: total requests, failures, successes, and timeouts.

Every second, the oldest bucket (from 10 seconds ago) is discarded, and a new empty bucket is created for the current second. This gives a sliding 10-second view of health, using only 10 buckets of memory regardless of traffic volume.

Evaluation at t=15s

  • Total requests in window: 5 + 4 + 3 + 4 + 3 + 3 + 2 + 3 + 2 + 1 = 30
  • Total failures: 0 + 0 + 1 + 1 + 2 + 3 + 2 + 3 + 2 + 1 = 15 (this is not used directly for the trip check, but for illustration)
  • Wait — let's recalculate: failures = 0 + 0 + 1 + 1 + 2 + 3 + 2 + 3 + 2 + 1 = 15
  • Check 1 — Minimum requests: 30 ≥ 20 → PASS (enough data to evaluate)
  • Failure rate: 15 / 30 = 50%
  • Check 2 — Failure rate threshold: 50% ≥ 50% → TRIP
  • Result: Breaker transitions from CLOSED → OPEN. After a 5-second sleep window, one probe request is sent.

One second later (t=16s): Bucket 1 (the 6s-7s bucket) is discarded, and a new empty bucket is created for 16s-17s. The window now covers 7s to 17s. This is why it's called a "rolling" window — it continuously slides forward in time.

Why Buckets Instead of Individual Timestamps?

At 10,000 requests per second, storing individual timestamps would require 100,000 entries in a 10-second window. With 10 buckets, you store exactly 10 counters regardless of traffic. The tradeoff is precision — you can't tell exactly when within a 1-second bucket the failures occurred. But for circuit breaking, second-level granularity is more than sufficient.

Bulkhead Pattern

The bulkhead pattern is named after ship bulkheads — watertight compartments that divide a ship's hull into isolated sections. If the hull is breached in one section, only that compartment floods. The other compartments remain dry, and the ship stays afloat. Without bulkheads, a single leak sinks the entire ship.

In software, the bulkhead pattern means isolating resources so that a failure in one component cannot exhaust resources needed by others. It is one of the most important companions to the circuit breaker pattern.

The Problem Without Bulkheads

Consider Service A, which calls three downstream dependencies: Payment Service, Inventory Service, and Email Service. Service A has a shared thread pool of 25 threads for handling all outgoing requests.

What happens when Payment Service hangs:

  1. Requests to Payment Service start timing out (each blocks a thread for 30 seconds).
  2. More requests arrive. More threads are blocked waiting for Payment.
  3. Within minutes, all 25 threads are stuck waiting for Payment Service responses that will never come.
  4. Now Inventory Service and Email Service calls also fail — not because those services are broken, but because there are no available threads to make the call.
  5. Service A becomes completely unresponsive. One slow dependency has killed everything.

Thread Pool Bulkhead

The solution: give each dependency its own isolated thread pool.

  • Payment Service pool: 10 threads
  • Inventory Service pool: 10 threads
  • Email Service pool: 5 threads

Now when Payment Service hangs:

  1. Payment requests consume all 10 threads in the Payment pool.
  2. The 11th Payment request is rejected immediately (pool full) — a form of fast failure.
  3. Meanwhile, Inventory's 10 threads and Email's 5 threads are completely unaffected.
  4. Inventory and Email calls continue working normally.
  5. Service A is partially degraded (no payments) but still functional for inventory and email operations.

This is exactly how ship bulkheads work: the damage is contained to one compartment.

Connection Pool Bulkhead

The same principle applies to connection pools for databases and external APIs:

  • Database A: connection pool max 50
  • Database B: connection pool max 50
  • External API: connection pool max 20

If Database A becomes slow and connections pile up, it can consume at most 50 connections. Database B and the External API retain their own dedicated pools. Without separate pools, Database A could consume all 120 connections, starving every other dependency.

How Bulkhead Complements Circuit Breaker

Bulkheads and circuit breakers solve related but different problems, and they work best together:

Pattern What It Prevents How It Works Analogy
Bulkhead Resource exhaustion spreading across dependencies Limits the resources (threads, connections) any single dependency can consume Watertight compartments on a ship — contains the flooding
Circuit Breaker Wasting resources on a broken dependency Stops calling a dependency after detecting persistent failure An electrical fuse — cuts the circuit when current is dangerous

Together: The bulkhead contains the blast radius (prevents one slow dependency from starving others), while the circuit breaker speeds up recovery (stops calling the broken dependency entirely). The bulkhead buys time during the detection window — while the circuit breaker is still counting failures before tripping, the bulkhead ensures the damage is contained to that one dependency's thread pool.

Example timeline:

  1. t=0s: Payment Service starts hanging. Payment thread pool starts filling up.
  2. t=5s: Payment thread pool is full (10/10 threads blocked). New payment requests are rejected. But Inventory and Email are fine (bulkhead working).
  3. t=10s: Circuit breaker detects 5 consecutive failures. Breaker trips OPEN. Now payment requests are rejected before even entering the thread pool (circuit breaker working).
  4. t=15s: Blocked threads start timing out and returning to the pool. Payment thread pool drains back to 0/10.
  5. t=40s: Circuit breaker enters HALF-OPEN. Sends one probe. If Payment is back, breaker closes. If not, reopens for another 30 seconds.

Without the bulkhead, all 25 threads would be consumed by Payment between t=0 and t=10 (before the breaker trips), and Inventory and Email would also fail during that window. The bulkhead prevents the collateral damage during the detection window.

Envoy's Outlier Detection: Proxy-Level Circuit Breaking

Envoy proxy implements a form of passive circuit breaking called outlier detection. Instead of the application maintaining a circuit breaker in code, the sidecar proxy automatically monitors responses from each backend and ejects unhealthy instances from the load balancing pool. This happens transparently — the application code doesn't even know it's happening.

How It Works Step by Step

  1. Envoy monitors responses from each backend instance individually. Every 5xx response is counted against that specific backend host (not the whole cluster).
  2. If a backend returns consecutive 5xx errors (default: 5 in a row), it's flagged as an outlier.
  3. The outlier is ejected from the load balancing pool. Traffic is routed only to healthy instances. The ejected instance receives zero requests.
  4. The ejection lasts for a base ejection time (default: 30 seconds). During this time, the backend has a chance to recover without being hammered by traffic.
  5. Each subsequent ejection doubles the time: first ejection = 30s, second = 60s, third = 120s, and so on. This implements exponential backoff for chronically unhealthy backends.
  6. After the ejection period, the backend is automatically re-included in the pool. If it returns healthy responses, it stays. If it returns 5xx again, it's re-ejected (with a longer duration).

Safety Mechanism: Maximum Ejection Percentage

This is a critical guardrail: at most 10% of backends can be ejected at any time (configurable). Here's why this matters:

Imagine you deploy a bad version to all 20 backend pods. Without a cap, outlier detection would eject all 20 pods, leaving zero backends in the pool. Every request would fail with "no healthy upstream" — a total outage caused by the safety mechanism itself.

With max_ejection_percent: 10 and 20 backends, at most 2 backends can be ejected at once. The remaining 18 stay in the pool. Yes, those 18 are also returning errors (because the deploy is bad), but at least traffic can still flow through — and the errors are visible to monitoring systems, triggering alerts and rollback. This is far better than a silent total blackout.

Configuration Example

outlier_detection:
  consecutive_5xx: 5              # eject after 5 consecutive 5xx responses
  interval: 10s                   # evaluate health every 10 seconds
  base_ejection_time: 30s         # first ejection lasts 30 seconds
  max_ejection_percent: 10        # at most 10% of backends can be ejected
  enforcing_consecutive_5xx: 100  # 100% enforcement (vs probabilistic sampling)
  consecutive_gateway_errors: 5   # also eject on gateway errors (502/503/504)
  success_rate_minimum_hosts: 5   # need at least 5 hosts to do success rate ejection
  success_rate_request_volume: 100  # need 100+ requests to evaluate success rate
  success_rate_stdev_factor: 1900   # eject if success rate is 1.9 stdev below average

Outlier Detection vs. Application-Level Circuit Breaker

The key difference: outlier detection operates per backend instance, while a traditional circuit breaker operates per service. If one pod out of ten is unhealthy:

  • Outlier detection: Ejects that one pod. 90% of capacity preserved. Only ~10% of requests saw errors before ejection.
  • Application-level breaker: Trips the entire circuit. 0% of capacity — all requests get the fallback. The 9 healthy pods sit idle.

Outlier detection is almost always better for partial failures (one bad pod, one bad deploy to a subset of instances). Application-level breakers are better when the entire downstream service is down (all pods failing, or a network partition between your service and the downstream).

In practice, use both: Envoy outlier detection handles individual bad pods (surgical), while an application-level circuit breaker handles total downstream failure (nuclear option). They operate at different granularities and complement each other.

Assumptions & Model

Key Assumptions

  • Failures are transient and will recover: The circuit breaker model assumes that the downstream service will eventually come back. The half-open state exists to detect recovery. If the failure is permanent (service decommissioned, endpoint removed), the breaker will cycle between open and half-open indefinitely, never resolving.
  • A single backend's failure is detectable: The model assumes you can distinguish "this service is failing" from "my request was bad." If your error detection lumps client errors (4xx) with server errors (5xx), the breaker trips on normal validation failures — a classic misconfiguration.
  • Failing fast is better than waiting: The model assumes that an immediate error is more useful to the caller than a slow timeout. This is true when callers can handle the error gracefully (fallback, cached data, degraded response). It is not true if the only option is to show the user an error — in that case, waiting a few seconds for a potential recovery might be better.
  • The breaker has enough traffic to make statistically meaningful decisions: A breaker monitoring a low-traffic endpoint may trip on a single failure because the sample size is too small. Minimum request thresholds mitigate this.

When These Assumptions Break

  • Permanent failures: The breaker cycles open/half-open forever. Solution: add alerting when a breaker has been open for extended periods, indicating human intervention is needed.
  • Partial failures: Some endpoints on the backend work fine, others don't. A single breaker for the entire service is too coarse — it blocks healthy endpoints too. Solution: per-endpoint or per-operation breakers.
  • Shared breaker for independent backends: If a single breaker wraps calls to multiple independent backends, one backend's failure trips the breaker for all of them. Solution: one breaker per downstream dependency.

Sizing & Tradeoffs

Failure Threshold

Too low (e.g., 2 failures): The breaker trips on occasional, normal errors. You get false positives — the service is fine, but the breaker thinks it's broken. Traffic is unnecessarily rejected. Too high (e.g., 100 failures): Slow detection. The downstream service has been failing for a long time before the breaker notices. Hundreds of requests fail before protection kicks in.

Rule of thumb: Use a percentage-based threshold with a minimum sample size. "Trip when error rate exceeds 50% in a 10-second window, with a minimum of 20 requests." This adapts to traffic volume.

Timeout Duration (Open State)

Too short (e.g., 5 seconds): The breaker transitions to half-open too quickly and hammers the recovering service with probe requests before it's ready. Too long (e.g., 5 minutes): Extended outage — even after the downstream service has recovered, your service continues to reject requests for minutes.

Rule of thumb: Start with 30 seconds. Adjust based on the downstream service's typical recovery time. If it's a containerized service that restarts in 10 seconds, 30 seconds is generous. If it's a database cluster that takes 2 minutes to failover, use 60-90 seconds.

Half-Open Request Count

Allowing just 1 probe request is conservative — minimal risk, but a single request isn't statistically meaningful (it might succeed by luck). Allowing 3-5 probes gives more confidence but means more requests hit a potentially still-broken service.

SLO-Driven Sizing Examples

Circuit breaker parameters should never be guessed — they should be derived from your Service Level Objectives. Work backward from the availability, latency, or success-rate targets your team has committed to.

Example 1 — Payment Service Dependency: 99.9% Success Rate SLO

Your checkout service calls a payment processor and must maintain a 99.9% success rate (no more than 1 failure per 1,000 requests). The payment processor's baseline error rate is ~0.05%.

Derivation:

  • Error budget: 0.1% of requests can fail. At 500 req/s, that is 0.5 errors/s — or about 5 errors per 10-second window.
  • Failure threshold: Trip when error rate exceeds 1% over a 10-second sliding window (minimum 50 requests). This is 10x the baseline, so it clearly signals a real problem rather than noise, and trips well before you exhaust your 0.1% error budget over the SLO measurement period (30-day rolling).
  • Window size: 10 seconds (time-based). At 500 req/s this evaluates 5,000 requests — highly statistically significant. Shorter windows (e.g., 5s) risk flapping; longer windows (e.g., 30s) burn too much error budget before tripping.
  • Open-state timeout: 30 seconds. The payment processor's historical P95 recovery time is 20 seconds (container restart). 30 seconds provides margin.
  • Half-open probes: 5 requests. Payment is critical — you need high confidence before re-closing. With 5 probes, the probability of all 5 succeeding by luck against a 50%-broken service is only 3.1%.
  • Fallback: Queue the payment for async retry and return "Payment processing — you'll receive confirmation shortly."

Example 2 — Microservices Mesh: p99 < 200ms Per Service SLO

In a request path that spans 4 microservices, the end-to-end p99 target is 800ms, giving each service a budget of 200ms (p99). A slow downstream must be cut off before it consumes the entire latency budget.

Derivation:

  • Request timeout: 150ms. Set below the 200ms SLO to leave headroom for local processing (~50ms). Any request exceeding 150ms is a timeout failure from the circuit breaker's perspective.
  • Failure threshold: Trip when 25% of requests time out within a 5-second window (minimum 30 requests). A 25% timeout rate means p75 is already above your p99 target — the service is clearly degraded.
  • Window size: 5 seconds (time-based). In a mesh with high request rates (1,000+ req/s between services), 5 seconds provides 5,000+ data points. Short windows are essential here: every second of delay cascades through 4 services.
  • Open-state timeout: 15 seconds with ±3 seconds jitter. Mesh services are typically auto-scaled and recover quickly. Jitter prevents synchronized probe storms across the mesh.
  • Half-open probes: 3 requests. Lower count because the mesh has many callers — even 3 probes per caller adds up. The service mesh (Envoy/Istio) can coordinate this across instances.
  • Key insight: The timeout (150ms) is the real lever here. Without a circuit breaker, a slow downstream would cause your service to block for its full timeout (e.g., 30s default), violating SLO by 150x.

Example 3 — Third-Party API Integration: 95% Vendor Availability → 99.9% System Availability

You depend on a third-party geocoding API with a published SLA of 95% availability (up to 36 hours of downtime per month). Your system SLO is 99.9% availability. The circuit breaker must bridge the gap.

Derivation:

  • The gap: The vendor is unavailable 5% of the time, but you can only be unavailable 0.1% of the time. The circuit breaker + fallback must handle the remaining 4.9% — meaning the fallback must work reliably.
  • Failure threshold: Trip when error rate exceeds 10% in a 30-second window (minimum 10 requests). Third-party APIs have higher baseline error rates (1-3%), so the threshold must be higher to avoid false trips. But 10% is clearly above baseline and triggers before extended degradation.
  • Window size: 30 seconds (time-based). Third-party APIs often have longer, sustained outages rather than brief blips. A wider window reduces flapping during intermittent connectivity issues.
  • Open-state timeout: 60 seconds with exponential backoff on repeated trips (60s → 120s → 240s, capped at 5 minutes). Vendor outages can last hours — aggressive probing wastes quota and may trigger rate limiting.
  • Half-open probes: 2 requests. Minimal probing to conserve API quota (many vendors charge per call or enforce rate limits).
  • Fallback: Return cached geocoding results (even if stale) or use a secondary provider. The fallback quality determines whether you actually achieve 99.9% — the breaker just ensures you switch fast enough.

Example 4 — Database Connection Pool: p99 < 50ms Query Time SLO

Your service has a p99 query latency target of 50ms. When the database is under load, connection acquisition from the pool starts timing out, cascading into request failures and thread starvation.

Derivation:

  • Connection acquisition timeout: 20ms. If the pool cannot provide a connection within 20ms, the database is saturated. This leaves 30ms for the query itself (which normally completes in 5-10ms at p99).
  • Failure threshold: Trip when 15% of connection acquisitions time out in a 5-second window (minimum 100 requests). Database connection pools have near-zero baseline timeout rates, so 15% is an unambiguous signal of saturation.
  • Window size: 5 seconds (count-based, 500 calls). Database issues escalate extremely fast — a full connection pool causes 100% failure within seconds. Detection must be rapid.
  • Open-state timeout: 10 seconds. Database recovery from connection pool exhaustion is fast once load is shed — typically within 2-5 seconds. A 10-second timeout lets the pool drain and connections recycle.
  • Half-open probes: 5 requests. Higher probe count because database health is binary — either the pool has capacity or it does not. 5 probes quickly confirm recovery.
  • Key insight: The circuit breaker on connection acquisition prevents thread starvation. Without it, threads block waiting for connections (up to the pool's max-wait, often 30s), request queues grow, and the entire service becomes unresponsive — far exceeding the 50ms SLO.

SLO-Driven Capacity Planning

Circuit breaker parameters and capacity planning are deeply intertwined. The breaker's behavior during failures directly affects your required infrastructure capacity. Use these principles to plan capacity around your SLO targets:

Planning Factor How the Circuit Breaker Affects It Capacity Implication
Fallback compute When the breaker opens, fallback logic runs instead of the normal path. If your fallback queries a cache or secondary datastore, that system must handle the redirected load. Size fallback infrastructure for 100% of traffic to the protected dependency. If the dependency handles 10K req/s, your cache must sustain 10K req/s during an outage.
Thread pool headroom Without a breaker, slow downstream calls consume threads for the full timeout duration. With a breaker, threads are freed in <10ms (fast-fail). The thread pool can be smaller. Without CB: provision for concurrent_requests × timeout threads. With CB: provision for concurrent_requests × avg_response_time. Often a 10-50x reduction in required threads.
Error budget burn rate The time between failure onset and breaker trip is the "detection window" — errors during this window count against your SLO error budget. For a 99.9% SLO (43.2 min/month budget): if your breaker takes 10 seconds to trip and you have 4 incidents/month, that consumes 40 seconds — roughly 1.5% of your error budget. Budget the rest for other failure modes.
Recovery surge capacity When the breaker closes after an outage, pent-up demand floods the recovered service. This "recovery surge" can cause the downstream to fail again. Implement gradual ramp-up: after closing, route only 10% of traffic through the dependency, increasing by 10% every 5 seconds. The downstream needs capacity for only 110% of normal load, not 200%+.
Multi-dependency budget allocation If your service has 5 dependencies, each with its own breaker, and your system SLO is 99.9%, you must allocate the error budget across all of them. With 5 dependencies: each gets ~0.02% error budget (0.1% / 5). This means each dependency's breaker should trip fast enough that its failure contributes no more than 0.02% unavailability. Use this to derive per-dependency detection windows.

Capacity Planning Formula: For a service with SLO S, n dependencies, and circuit breaker detection time d seconds:

max_incidents_per_month = (1 − S) × 2,592,000 / (n × d)

For 99.9% SLO, 5 dependencies, 10s detection: (0.001 × 2,592,000) / (5 × 10) = 51.8 incidents/month before breaching SLO. If you experience more incidents, either reduce detection time, reduce dependencies, or relax your SLO.

Scaling Journey

0 – 1K users

No Protection

Monolithic app with a few dependencies. If something fails, you catch the exception and log it. Retries are ad-hoc. Failures are rare because traffic is low and there are few moving parts. You might not even notice a downstream hiccup.

1K – 100K users

Try/Catch + Retry

You start calling external APIs (payment, email, SMS). You add try/catch blocks with simple retry logic and exponential backoff. This works until a downstream service has an extended outage — your retries pile up, thread pools fill, and response times spike for all users.

100K – 10M users

In-Process Breaker

You adopt a library like Resilience4j or Polly. Circuit breakers wrap each external dependency. Fallbacks return cached data or degraded responses. You add bulkheads (separate thread pools per dependency) so a slow payment service doesn't starve the product catalog. Dashboards show breaker state in real time.

10M+ users

Service Mesh Breaker

Hundreds of microservices in multiple languages. You move circuit breaking to the sidecar proxy layer (Envoy/Istio). Policies are configured centrally via CRDs, applied consistently across all services. Outlier detection ejects individual unhealthy pods rather than breaking entire circuits. Combined with automated canary deployments and progressive rollouts.

What Can Go Wrong

Breaker Trips Too Aggressively During Deploys

What happened: During a rolling deployment, old pods are terminated and new pods are starting up. For a brief window, health check failures spike as old pods go down before new pods are ready. The circuit breaker trips, rejecting all traffic to the service — even though healthy pods are available.

Root cause: The breaker threshold was tuned for steady-state traffic, not for the transient spike in errors during deployment. The rolling update strategy didn't have sufficient readiness probe delay.

Mitigation: Tune readiness probes so pods are only added to the load balancer pool when fully ready. Use a percentage-based breaker (not absolute count) so the error rate during deployment stays below the threshold. Consider excluding deployment-related errors from the breaker's failure count.

Breaker State Not Shared Across Instances

What happened: A service has 20 replicas, each with its own in-process circuit breaker. When the downstream service starts failing, some replicas trip their breakers quickly (they happened to send more requests) while others haven't yet seen enough failures. Traffic shifts to the replicas with closed breakers, which then experience a burst of failures. The breakers trip in a staggered cascade over 30+ seconds.

Root cause: Each replica independently tracks failures. With traffic spread across 20 replicas, each individual replica sees 1/20th of the errors, so it takes 20x longer for each to reach the threshold.

Mitigation: Lower per-instance thresholds proportionally, or use a shared state store (Redis) for breaker state. Alternatively, move to a service mesh where the proxy handles breaking with cluster-wide visibility.

Half-Open Thundering Herd

What happened: After the open-state timeout expires, all 50 instances of the calling service simultaneously enter half-open and send probe requests to the recovering downstream service. The burst of 50 simultaneous probes overwhelms the service, causing it to fail again — and all breakers reopen.

Root cause: All instances had their breakers trip at approximately the same time, so their timeouts expired simultaneously.

Mitigation: Add jitter to the open-state timeout (e.g., 30s +/- 5s random). Limit the half-open probe count to 1 per instance. Use a central coordinator to stagger probe requests.

Breaker Masks Permanent Failures

What happened: A misconfigured DNS entry meant all requests to a downstream service were going to the wrong IP. The circuit breaker dutifully opened, served fallback responses, and periodically probed (which also failed). Because the fallback was "good enough," nobody noticed the breaker was open for three days.

Root cause: No alerting on breaker state duration. The fallback masked the failure from end users.

Mitigation: Alert when a breaker has been open for more than 5 minutes. Track and dashboard breaker state transitions. Distinguish between "graceful degradation" and "something is broken."

Operating in Production

Key Metrics to Monitor

Metric What It Tells You Alert Threshold
circuit_breaker_state Current state (closed/open/half-open) per dependency Alert if open > 5 minutes
circuit_breaker_trip_rate How often the breaker trips per hour Alert if > 3 trips/hour (flapping)
downstream_error_rate Percentage of failed calls to each dependency Alert if > 10% sustained over 1 minute
fallback_invocation_rate How often fallback logic is triggered Alert if > 5% of requests use fallback
downstream_p99_latency Tail latency to downstream — slow responses precede failures Alert if > 2x normal baseline

Debug Playbook

Breaker is open — requests failing fast:

1. Check circuit_breaker_state dashboard — which dependency's breaker is open?

2. Check downstream_error_rate for that dependency — is the downstream actually failing?

3. If YES: Check the downstream service's health — is it crashing, overloaded, or misconfigured? Is it a deployment in progress?

4. If NO (low error rate but breaker is open): The threshold may be too sensitive. Check the sliding window config — is the minimum request count too low? Are 4xx errors being counted as failures?

5. Check if breaker is flapping (rapid open/close cycles) — this indicates the downstream is intermittently unhealthy. Look for resource contention, GC pauses, or connection pool exhaustion on the downstream.

6. Check if half-open probes are succeeding but breaker is still opening — the probe may be hitting a healthy instance while most instances are unhealthy. Consider increasing half-open probe count.

Interactions & Pitfalls

Circuit Breakers + Load Balancers

If the load balancer doesn't know about the circuit breaker's state, it keeps routing traffic to instances that have their breakers open — those instances immediately reject the request, and the user gets an error. The LB should either integrate with the breaker state (e.g., Envoy outlier detection removes unhealthy upstream hosts from the pool) or the breaker's fallback should return a response that the LB interprets as "try another instance" (via retries). In a service mesh, Envoy handles both load balancing and circuit breaking in the same proxy, avoiding this coordination problem.

Circuit Breakers + Retries

If you have a retry policy that retries through an open circuit breaker, you're just retrying immediate rejections — burning CPU and adding latency for no benefit. Retries should be below the circuit breaker. The call chain should be: caller → circuit breaker → retry → downstream. The retry handles transient single-request failures. If retries consistently fail, the breaker trips and stops the retries too. Never put retries above the breaker.

Circuit Breakers + Timeouts

The request timeout must be shorter than the breaker's evaluation window. If your timeout is 30 seconds and your breaker window is 10 seconds, you'll have at most a few timed-out requests in the window (because each takes 30 seconds). The breaker won't trip because it hasn't seen enough failures in 10 seconds. Set timeouts aggressively (1-5 seconds for most service calls) so failures are detected quickly and the breaker has enough data points in its window.

Circuit Breakers + Message Queues

Circuit breakers are designed for synchronous request-response calls. For asynchronous communication via message queues, the queue itself acts as the buffer. If a consumer is down, messages queue up and are processed when the consumer recovers. You don't need a circuit breaker on the producer side. However, if a consumer synchronously calls a downstream service while processing a message, that call should have a circuit breaker.

Cost Considerations

Cost Analysis

Direct cost: Minimal. Circuit breakers are in-process logic — a few counters, a state variable, and a timer. No additional infrastructure unless you centralize breaker state in Redis (a small single-node Redis instance costs ~$15/month on AWS). In a service mesh, the cost is already baked into the sidecar proxy overhead.

Dominant cost driver: Engineering time for tuning and testing. Getting thresholds right requires load testing, chaos testing, and iteration. A poorly tuned breaker (too aggressive or too lenient) causes more problems than it solves.

Cost optimization: Start with library-level breakers (free). Move to service mesh only when you have enough services to justify the mesh overhead. Use sensible defaults (Resilience4j defaults are good starting points) and tune incrementally based on production data.

Cost of NOT having circuit breakers: Cascading failures can take down your entire platform. A single slow or failing downstream service — without a breaker — can exhaust thread pools, cause timeouts to propagate upstream, and result in a total outage affecting all users. One incident like this can cost hours of engineering time, SLA credits, and user trust. Netflix estimated that a single major cascading failure incident could cost millions in lost revenue.

Real-World Examples

Netflix Hystrix

Netflix pioneered the circuit breaker pattern at scale. With hundreds of microservices and billions of daily API calls, a single failing dependency could cascade through the entire streaming platform. Hystrix wrapped every inter-service call with circuit breakers, thread pool isolation, and fallbacks. When the recommendation service went down, the UI showed generic recommendations from cache instead of failing entirely. Hystrix also provided a real-time dashboard (Hystrix Dashboard + Turbine) that showed breaker state across the fleet. Though now in maintenance mode, Hystrix established the patterns that all modern implementations follow.

Envoy Proxy Outlier Detection

Envoy, the proxy at the heart of Istio and many service meshes, implements circuit breaking via "outlier detection." Instead of a binary open/closed breaker on the entire service, Envoy tracks the health of individual upstream hosts. When a host returns consecutive 5xx errors, Envoy ejects it from the healthy pool for a configurable duration. This is more granular — one bad pod doesn't break the circuit for all pods. Envoy also implements connection pool circuit breaking: when the number of pending requests or active connections to an upstream cluster exceeds limits, additional requests are rejected immediately.

AWS App Mesh Circuit Breaking

AWS App Mesh provides managed service mesh capabilities built on Envoy. Circuit breaking is configured via virtual node connection pool settings: maxConnections and maxPendingRequests for TCP, plus maxRequestsPerConnection for HTTP. When limits are exceeded, the proxy returns a 503 to the caller. This protects downstream services from being overwhelmed, and integrates with CloudWatch for monitoring breaker activity. The managed approach means teams get circuit breaking without operating the mesh control plane.

Code Example

A minimal circuit breaker state machine in Python (~40 lines). This captures the core logic: failure counting, state transitions, and timeout-based recovery testing.

import time

class CircuitBreaker:
    def __init__(self, threshold=5, timeout=30):
        self.threshold = threshold
        self.timeout = timeout
        self.failures = 0
        self.state = "closed"       # closed | open | half-open
        self.opened_at = None

    def call(self, fn, *args, **kwargs):
        if self.state == "open":
            if time.time() - self.opened_at >= self.timeout:
                self.state = "half-open"  # Try a probe
            else:
                raise Exception("Circuit is OPEN — failing fast")

        try:
            result = fn(*args, **kwargs)
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise

    def _on_success(self):
        self.failures = 0
        if self.state == "half-open":
            self.state = "closed"   # Recovery confirmed

    def _on_failure(self):
        self.failures += 1
        if self.failures >= self.threshold:
            self.state = "open"
            self.opened_at = time.time()

# Usage:
breaker = CircuitBreaker(threshold=3, timeout=10)
def call_payment_api():
    # ... HTTP call to payment service
    pass

try:
    result = breaker.call(call_payment_api)
except Exception as e:
    print(f"Fallback: {e}")  # Serve cached/default response

How to Validate Your Design

Chaos Testing

Inject failures into the downstream service and verify the breaker trips correctly. Use toxiproxy to add latency, return errors, or sever connections to a downstream dependency. Verify: (1) the breaker transitions to open after the configured failure threshold, (2) requests receive the fallback response, (3) no threads or connections are leaked during the open state. Use Chaos Monkey or Litmus to randomly kill downstream pods in a staging environment and observe breaker behavior.

Test Recovery

After injecting failures and observing the breaker trip, restore the downstream service and verify: (1) the breaker transitions to half-open after the timeout, (2) probe requests succeed, (3) the breaker closes and normal traffic resumes, (4) no request backlog floods the recovering service. Time the end-to-end recovery cycle — from failure injection to full recovery — and verify it matches your expected timeout + probe duration.

Load Testing

Use k6 or Locust to generate realistic traffic while a downstream is failing. Measure: (1) latency during the open state — it should be near-zero (immediate rejection), (2) throughput to healthy dependencies — they should be unaffected by the broken dependency, (3) thread pool utilization — it should drop when the breaker opens, not spike. Compare latency distributions with and without the circuit breaker during a simulated outage. Without the breaker, p99 latency will spike to the timeout duration (e.g., 30 seconds). With it, p99 should stay low (sub-millisecond for rejected requests).

Interview Questions

How would you prevent cascading failures in a microservices architecture?

Cascading failures happen when a failing downstream service causes its callers to hang (waiting for timeouts), exhausting their thread pools, which in turn makes them unresponsive to their callers — propagating the failure upstream through the entire call chain.

The primary defense is the circuit breaker pattern: wrap every synchronous call to a downstream service with a breaker that monitors failure rates. When failures exceed a threshold, the breaker opens and immediately rejects requests without calling the downstream — this is "failing fast." Combine with timeouts (aggressive, 1-5 seconds per hop) so individual requests don't hang, bulkheads (isolate each dependency's thread pool so one slow service can't starve others), and fallbacks (return cached data, defaults, or gracefully degraded responses when the breaker is open).

At scale, implement these patterns at the service mesh layer (Envoy/Istio) rather than in application code. This ensures consistent protection across all services regardless of language. Add load shedding so services under pressure can reject excess requests before they become overloaded.

Follow-up: "How do you choose between circuit breakers at the application level vs. the service mesh level?" — Application-level breakers give more control (custom fallbacks, business-logic-aware failure classification), while mesh-level breakers provide consistent policy across polyglot services with zero code changes.

Explain the circuit breaker states and transitions

A circuit breaker is a state machine with three states. Closed is the normal state: all requests pass through to the downstream service, and failures are recorded in a sliding window (count-based or time-based). When the failure rate in the window exceeds a threshold (e.g., 50% error rate with a minimum of 20 requests), the breaker transitions to Open.

In the Open state, all requests are immediately rejected without calling the downstream — this is the "fail fast" behavior. A timeout timer starts (e.g., 30 seconds). When the timer expires, the breaker transitions to Half-Open.

In Half-Open, a limited number of probe requests (e.g., 3) are allowed through to test if the downstream has recovered. If all probes succeed, the breaker transitions back to Closed and normal operation resumes. If any probe fails, the breaker transitions back to Open, the timer resets, and the cycle repeats.

Follow-up: "What happens to the in-flight requests when the breaker first trips to open?" — Requests already in flight continue to completion (or timeout). Only new requests are rejected. Some implementations also cancel in-flight requests, but that's more aggressive.

How do you tune circuit breaker thresholds?

Tuning is the hardest part of operating circuit breakers. The key parameters are: failure rate threshold (what percentage of failures triggers the breaker — start at 50%), sliding window size (how many requests or how long a time period to evaluate — start at 10 seconds or 100 requests), minimum request count (don't evaluate the error rate until you have enough data — start at 20), and open-state timeout (how long to wait before probing — start at 30 seconds).

The process is: (1) start with library defaults, (2) observe production behavior for a week, (3) look for false positives (breaker trips when the downstream is healthy — lower sensitivity or increase minimum request count) and false negatives (downstream is failing but the breaker hasn't tripped — increase sensitivity or shrink the window), (4) adjust incrementally. Run chaos tests in staging to validate the tuning before production.

Critical mistake: using an absolute failure count (e.g., "5 failures") instead of a rate. Five failures out of 10,000 requests is noise; five failures out of 10 requests is a crisis. Always use percentage-based thresholds with a minimum sample size.

Follow-up: "How do you tune differently for high-traffic vs. low-traffic endpoints?" — High-traffic endpoints can use smaller windows (error rate stabilizes quickly). Low-traffic endpoints need higher minimum request counts to avoid noise, and you may need to use count-based windows instead of time-based.

Circuit breaker vs. retry — when to use which?

Retries and circuit breakers solve different problems and complement each other. Retries handle transient, isolated failures — a single dropped connection, a momentary blip, a brief DNS hiccup. The assumption is that the next attempt will succeed. Use retries with exponential backoff and jitter for idempotent operations.

Circuit breakers handle sustained failures — the downstream service is down, overloaded, or degraded for an extended period. The assumption is that repeated attempts will all fail, so stop trying. Use circuit breakers when the downstream is genuinely broken, not just flaky.

In practice, use both together in layers: the circuit breaker wraps the retry. A request first hits the circuit breaker check. If the breaker is closed, the request goes through with retry logic (e.g., retry up to 3 times with exponential backoff). If all retries fail, the failure is reported to the breaker. If enough retries fail across enough requests, the breaker trips. Once open, the breaker short-circuits both the request and the retries — no retries through an open breaker.

Follow-up: "What about non-idempotent operations like payment processing?" — For non-idempotent operations, retries are dangerous (you might charge the customer twice). Use circuit breakers without retries, and implement idempotency keys if retries are needed.

How does Envoy implement circuit breaking?

Envoy implements circuit breaking at two levels: connection pool limits and outlier detection. Connection pool limits set hard ceilings on concurrent connections (max_connections), pending requests (max_pending_requests), and concurrent requests (max_requests) to an upstream cluster. When any limit is hit, additional requests get a 503 immediately — this prevents overwhelming the upstream.

Outlier detection is Envoy's equivalent of the traditional circuit breaker pattern, but more granular. Instead of breaking the circuit for an entire service, Envoy monitors individual upstream hosts. When a host returns consecutive 5xx errors (configurable via consecutive_5xx), Envoy ejects that host from the healthy pool for a configurable duration (base_ejection_time). The host is automatically re-added after the ejection period. Envoy can also eject based on success rate: if a host's success rate is statistically lower than the cluster average, it gets ejected.

This per-host approach is more precise than application-level breakers: if one pod out of ten is unhealthy, only that pod is removed from the pool, and 90% of capacity is preserved. Application-level breakers would break the entire circuit, losing 100% of capacity.

Follow-up: "How does Envoy's outlier detection interact with Kubernetes readiness probes?" — They're complementary. Readiness probes remove pods that fail health checks. Outlier detection removes pods that pass health checks but return errors on real traffic (a more realistic signal). Together, they cover both startup/crash failures and runtime degradation.

Design a fallback strategy when a circuit breaker is open

The fallback strategy depends on the dependency and the business context. The hierarchy from best to worst user experience is: (1) serve from cache — if you have a recently cached response from the downstream service, return it. For many use cases (product catalog, recommendations, user profiles), slightly stale data is far better than no data. Set a "fallback TTL" that's longer than your normal cache TTL. (2) return defaults — if no cached data is available, return sensible defaults. For a recommendation engine, show popular items. For a currency conversion service, use the last known exchange rate. (3) degrade gracefully — remove the feature that depends on the broken service. If the review service is down, show the product page without reviews. (4) queue for later — for write operations, queue the request and process it when the service recovers (requires idempotent processing). (5) return an error — last resort. Return a clear error message with a retry suggestion.

Key implementation details: fallbacks should be fast (no network calls in the fallback path), simple (the fallback itself should never fail), and monitored (track fallback invocation rate — if it's high, something is wrong). Never fall back to calling the same failing service through a different path.

Follow-up: "How do you test that your fallbacks actually work?" — Run regular chaos tests in staging that trip breakers for each dependency and verify the fallback behavior. Without testing, you discover your fallback is broken during a real outage — the worst possible time.

Key Takeaways

  • Circuit breakers prevent cascading failures by detecting failing downstream services and failing fast, protecting the entire call chain from being dragged down by one broken dependency.
  • Three states: Closed, Open, Half-Open. Closed = normal operation. Open = failing fast. Half-Open = probing for recovery. The state machine is self-healing.
  • Use percentage-based thresholds, not absolute counts. "50% error rate with a minimum of 20 requests" adapts to traffic volume. "5 failures" does not.
  • Circuit breakers and retries are complementary, not alternatives. Retries handle transient failures. Circuit breakers handle sustained failures. Layer them: breaker wraps retry.
  • Always have a fallback. An open breaker is only useful if you can serve a degraded response. Without a fallback, the user sees an error either way.
  • Monitor breaker state. Alert when a breaker has been open for more than 5 minutes. Dashboard breaker transitions. Track fallback invocation rates.
  • At scale, move to the proxy layer. In-process breakers work for small deployments. Service mesh breakers (Envoy/Istio) provide consistent, language-agnostic protection at scale.
  • Tune incrementally from production data. Start with defaults, observe, adjust. Don't guess thresholds — measure them.
