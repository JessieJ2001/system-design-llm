Observability

Understand the internal state of any distributed system through the three pillars — logging, metrics, and distributed tracing — so you can diagnose problems you never anticipated.

35 min read

What Is It?

Observability is the ability to understand the internal state of a system by examining its external outputs. Unlike traditional monitoring, which answers pre-defined questions ("Is the server up? Is CPU above 80%?"), observability lets you ask new questions you didn't anticipate when you designed the system. When an on-call engineer gets paged at 3 AM for a problem nobody has seen before, observability is what lets them diagnose it without deploying new code or adding new log statements.

The concept rests on three pillars. Logs are discrete, timestamped events — a record of what happened at a specific moment (e.g., "user 42 got a 500 error on /checkout"). Metrics are aggregated numerical measurements over time — counters, gauges, and histograms that tell you things like request rate, error percentage, and p99 latency. Traces are records of a single request's journey through multiple services, showing exactly where time was spent and where failures occurred.

Individually, each pillar is useful but limited. Metrics tell you something is wrong but not what. Logs tell you what happened in one service but not the full picture across services. Traces show the request path but don't aggregate well. The power of observability comes from correlating all three: a metric spike triggers an alert, you drill into traces to find the slow service, then you pull the correlated logs from that service to find the root cause. This correlation — usually through shared identifiers like trace_id and span_id — is what separates observability from simply having three monitoring tools.

What Each Pillar Captures That the Others Cannot

Think of the three pillars as three different lenses on the same system. Each one answers a fundamentally different question:

PillarWhat It RecordsQuestion It AnswersConcrete Example
Logs Individual events with full context WHAT happened? "User 123 failed to login from IP 1.2.3.4 at 14:32:05 — invalid password."
Metrics Aggregated numerical measurements over time HOW MUCH is happening? "Login failure rate = 2.3% in the last 5 minutes. This is 4x higher than the baseline."
Traces Request path across services with timing WHERE did time go? "Request abc-123 went Auth → UserDB → Cache → Response, total 230ms. The UserDB call took 180ms."

Why You Need All Three Together

Consider this incident scenario at an e-commerce company:

  1. Metrics ALERT you: The login_failures_total counter spikes. Your dashboard shows the login failure rate jumped from 0.5% to 15% in the last 3 minutes. An SLO-based alert fires.
  2. Logs help you DIAGNOSE: You query your structured logs: SELECT reason, count(*) FROM logs WHERE service='auth' AND level='ERROR' GROUP BY reason. You see that 95% of failures have reason: "database_timeout". The logs show which database host is timing out and which query is stuck.
  3. Traces help you PINPOINT: You pull a sample trace for a failed login. The trace shows: Auth Service (5ms) → User Database query (29,500ms — timeout!). The DB call span has attributes showing it hit replica db-replica-3. That specific replica is the problem.

Without metrics, you would not have known there was a problem until users complained. Without logs, you would have known "something is failing" but not what. Without traces, you would have known "the database is timing out" but not which replica or which specific query path.

Distributed Trace in Action

A request flows through three services. Each service generates a log entry, increments a metric counter, and produces a trace span. The spans assemble into a complete distributed trace.

How It Works

1. Structured Logging

Modern logging goes beyond printf-style text. Structured logs are emitted as JSON objects with consistent fields: timestamp, level, service, trace_id, span_id, message, and arbitrary key-value attributes. This structure enables powerful queries — "show me all ERROR logs from the payment service in the last hour where user_id = 42" — without regex gymnastics. Correlation IDs (the trace_id) let you stitch together logs from every service a single request touched.

Unstructured vs. Structured: A Concrete Comparison

Consider this log line from a login failure. Here is the unstructured version:

2024-03-15 14:32:05 ERROR User 123 failed to login from 1.2.3.4 - invalid password

And here is the structured (JSON) version of the same event:

{
  "timestamp": "2024-03-15T14:32:05Z",
  "level": "ERROR",
  "message": "Login failed",
  "userId": 123,
  "ip": "1.2.3.4",
  "reason": "invalid_password",
  "service": "auth-service",
  "traceId": "abc-123"
}

Why structured is better: You can query it. With structured logs in a centralized store, you can write: 

SELECT count(*) FROM logs
WHERE reason = 'invalid_password'
  AND ip = '1.2.3.4'
  AND timestamp > NOW() - INTERVAL '1 hour'

This instantly answers: "How many invalid password attempts came from IP 1.2.3.4 in the last hour?" With unstructured logs, you would need to write a fragile regex like /failed to login from (\d+\.\d+\.\d+\.\d+) - invalid password/, which breaks the moment anyone changes the log message wording.

Log Levels: When to Use Each

Log levels control verbosity. In production, you typically run at INFO level and above. Each level has a specific purpose:

LevelWhen to UseExample
TRACE Extremely detailed debugging; usually off in production "Entering function processPayment with args {amount: 49.99, currency: USD}"
DEBUG Development and troubleshooting; off in production by default "Cache miss for key user:123, fetching from DB"
INFO Normal operations worth recording "Order #456 created successfully, total=$89.99"
WARN Something unexpected but the system continues working "Response time 2.3s exceeded 2s threshold for /api/search"
ERROR Something failed but the system continues "Payment gateway returned 503, retrying (attempt 2/3)"
FATAL System is crashing; immediate attention needed "Database connection pool exhausted, shutting down"

A practical rule: if you get paged for it, it is ERROR or FATAL. If you want to know about it during an investigation, it is INFO. If you only care during active debugging, it is DEBUG. If you never want it in production, it is TRACE.

2. Metrics

Metrics are pre-aggregated numerical data points collected at fixed intervals. There are three fundamental types:

  • Counters — monotonically increasing values (e.g., http_requests_total). You derive rate from the delta between two scrapes.
  • Gauges — point-in-time values that go up and down (e.g., active_connections, cpu_usage_percent).
  • Histograms — distributions of values across configurable buckets (e.g., request latency). Let you compute percentiles like p50, p95, p99.

Metrics are cheap to store (a single float64 per series per scrape) and fast to query, making them ideal for dashboards and alerts. The tradeoff is that aggregation discards individual events — you know the p99 is 500ms, but you don't know which requests were slow.

Metric Types Explained with Concrete Examples

TypeWhat It MeasuresOperationsExample
Counter Cumulative total (only goes up; resets to zero on restart) Increment http_requests_total = 1,247,891. To get the rate, compute the delta: if it was 1,247,791 one minute ago, the rate is 100 requests/min.
Gauge Current value (goes up and down) Set, increment, decrement active_connections = 42. Right now there are 42 open connections. A minute ago there were 38. Unlike a counter, it can decrease.
Histogram Distribution of values across configurable buckets Observe (record a value) request_duration_seconds with buckets: {le="0.1": 5000, le="0.5": 8500, le="1.0": 9900, le="+Inf": 10000}. This means: 5000 requests were ≤ 100ms, 8500 were ≤ 500ms, 9900 were ≤ 1s, and 10000 total. From this, you can compute that 100 requests (1%) took more than 1 second.
Summary Similar to histogram but calculates quantiles client-side Observe request_duration{quantile="0.99"} = 0.45. The client computed that the p99 is 450ms. Unlike histograms, summaries cannot be aggregated across multiple instances (you cannot average p99 values to get a global p99).

Why Histograms Beat Averages for Latency

Imagine a service handling 100 requests. Average latency = 50ms. Sounds great! But consider this distribution:

  • 99 requests took 10ms each (total: 990ms)
  • 1 request took 4,010ms (total: 4,010ms)
  • Average = 5,000ms / 100 = 50ms

The average hides the fact that 1 in 100 users waited 4 seconds. This is why percentiles matter:

PercentileMeaningTypical Use
p50 (median) Half of requests are faster than this Typical user experience
p95 95% of requests are faster SLO target (e.g., "p95 < 200ms")
p99 99% of requests are faster Worst-case for most users
p99.9 1 in 1,000 requests is slower Critical for high-traffic services (1M req/day = 1,000 users hit this daily)

Real example: p50=12ms, p95=45ms, p99=230ms, p99.9=1200ms. The average might be 50ms, but that single number hides the fact that 1% of users wait 230ms or more, and 0.1% wait over a second. If you have 1 million requests per day, that means 1,000 users per day experience a 1.2-second wait. Percentiles reveal what averages conceal.

3. Distributed Tracing

A distributed trace represents the end-to-end journey of a single request through your system. Each unit of work within a service is a span — it has a trace_id (shared across all spans in the trace), a span_id (unique to this span), a parent_span_id (linking it to the calling span), a start time, duration, and key-value attributes. When Service A calls Service B, the trace context (trace ID and parent span ID) is propagated via HTTP headers or gRPC metadata, so Service B's spans are linked to Service A's.

The result is a tree of spans that shows exactly how a request flowed, where time was spent, and where errors occurred. This is indispensable for debugging latency in microservices, where a single user-facing request might touch 10+ services.

The Data Model: Spans and Traces

A span is a single unit of work. It has: a span ID (unique identifier), a parent span ID (null for the root span), an operation name, a start time, a duration, a status (OK or ERROR), and key-value tags/attributes (like http.method=GET or db.statement=SELECT...).

A trace is a tree of spans representing one end-to-end request. The root span is the initial incoming request. Each downstream call creates a child span. The tree structure reveals the full request path and where time was spent.

Example Trace: GET /api/user/123

Here is a complete trace showing a user profile lookup that hits a cache miss:

Trace ID: abc-123
|
+-- [Span 1] API Gateway: GET /api/user/123             (0ms - 230ms)  total: 230ms
|   |
|   +-- [Span 2] Auth Service: validateToken             (5ms - 15ms)   total: 10ms
|   |
|   +-- [Span 3] User Service: getUser                   (20ms - 210ms) total: 190ms
|   |   |
|   |   +-- [Span 4] Cache: GET user:123                 (22ms - 23ms)  total: 1ms  --> MISS
|   |   |
|   |   +-- [Span 5] PostgreSQL: SELECT * FROM users     (25ms - 180ms) total: 155ms <-- BOTTLENECK
|   |   |             WHERE id=123
|   |   |
|   |   +-- [Span 6] Cache: SET user:123                 (182ms - 185ms) total: 3ms
|   |
|   +-- [Span 7] Response serialization                  (212ms - 215ms) total: 3ms

Reading this trace: The total request took 230ms. The bottleneck is Span 5 (the database query, 155ms — 67% of total time). The cache miss (Span 4) forced the database query. If we improve the DB query (add an index, optimize the query) or increase the cache hit rate (longer TTL, cache warming), we fix the latency. Without this trace, we would only know "the API is slow" but not why.

Context Propagation: How Spans Link Across Services

When Service A calls Service B over HTTP, the tracing SDK injects a special header into the outgoing request: 

traceparent: 00-abc123def456789-span456abc-01
               |  |                |        |
               |  trace ID         span ID  sampled flag
               version

Service B receives this header, extracts the trace ID and parent span ID, and creates a new span with traceId = abc123def456789 and parentSpanId = span456abc. This linking mechanism is what connects all spans into one coherent trace, even when they come from completely different services written in different languages running on different machines.

This is the W3C Trace Context standard. All modern tracing libraries (OpenTelemetry, Jaeger, Zipkin) support it. The critical requirement is that every service in the call chain must propagate the header. If even one service drops it, the trace breaks into disconnected fragments.

4. OpenTelemetry: The Unified Standard

OpenTelemetry (OTel) is the industry-standard framework for instrumenting, generating, collecting, and exporting telemetry data. It emerged from the merger of OpenTracing and OpenCensus in 2019 and is now a CNCF project with broad vendor support. OTel provides a single SDK that produces logs, metrics, and traces with consistent correlation, eliminating the need to instrument your code three different ways for three different backends. Telemetry is exported via the OpenTelemetry Protocol (OTLP) to any compatible backend — Jaeger, Prometheus, Grafana, Datadog, or your own storage.

When to Use / When NOT to Use

Use Observability When

  • Production microservices — you cannot debug distributed systems by SSH-ing into servers and reading log files
  • "It's slow" isn't enough — you need to pinpoint which service, which dependency, which query is the bottleneck
  • Any system with SLOs — you need metrics to measure compliance and alerts to detect violations
  • High-stakes production systems — revenue-impacting services where MTTR (Mean Time to Resolution) matters
  • Async workflows — tracing request flow through message queues and background jobs

Don't Over-Invest When

  • Simple monolith — basic application logs + an APM agent (New Relic, Datadog APM) covers 80% of needs without custom instrumentation
  • Prototype / MVP — structured logging is premature when you're validating product-market fit; console.log is fine
  • Storage costs exceed value — if you're logging 10TB/day but only query it during incidents once a month, sample more aggressively
  • Small team, simple stack — a 3-person team with one database and one server doesn't need Jaeger; a stack trace is usually enough

Classic vs. Modern Approaches

Classic (2000s–2010s)

Syslog + flat log files — applications wrote unstructured text to /var/log. Operators used grep, awk, and tail -f to debug. Request correlation meant manually grepping for a request ID you embedded in your logs. Nagios / Zabbix polled services with health checks and sent email/pager alerts. StatsD + Graphite provided basic counters and gauges. This era had monitoring, but not observability — you could only answer questions you'd thought to ask in advance.

Modern (2015–Present)

  • Structured logging (ELK / Splunk / Loki) — JSON logs shipped to a centralized store with full-text search, field-based filtering, and log aggregation. Grafana Loki optimizes cost by only indexing labels, not the full log line.
  • Prometheus + Grafana — pull-based metrics collection with a powerful query language (PromQL). Histograms, recording rules, and alerting built in. De facto standard for Kubernetes environments.
  • Distributed tracing (Jaeger / Zipkin / Tempo) — inspired by Google's Dapper paper (2010). Visualize request flow as a waterfall of spans. Grafana Tempo provides a cost-effective trace backend using object storage.
  • OpenTelemetry — vendor-neutral instrumentation SDK that unifies logs, metrics, and traces. Auto-instrumentation libraries reduce manual effort. Context propagation baked in.
  • Continuous profiling (Pyroscope / Parca) — always-on CPU and memory profiling at low overhead (<2%). Lets you answer "why is the payment service using 80% CPU?" without reproducing the load in a dev environment.
  • eBPF-based observability (Cilium / Pixie / Coroot) — kernel-level instrumentation that requires zero application changes. Captures network flows, DNS queries, HTTP transactions, and even database queries by intercepting system calls. Ideal for polyglot environments where you can't instrument every language.

Assumptions & Model

Key Assumptions

  • Instrumentation overhead is negligible — we assume adding tracing and metrics adds <2% latency and <5% CPU overhead. This holds for well-designed SDKs (OpenTelemetry) but can break if you log excessively at high throughput, serialize large payloads, or use synchronous exporters.
  • Storage is affordable — we assume you can store or sample enough telemetry to be useful. At very high scale (millions of RPS), raw storage becomes expensive and you must sample aggressively, which means you might miss rare events.
  • Engineers know how to query — observability tools are only as good as the people using them. If nobody knows PromQL or how to read a trace waterfall, expensive tooling provides little value.
  • Clocks are approximately synchronized — distributed traces rely on timestamps from different machines. NTP typically gives <10ms accuracy, which is sufficient. If clock skew exceeds span durations, traces become misleading.

The RED Method (for Services)

For every service, measure three things: Rate (requests per second), Errors (failed requests per second), and Duration (latency distribution, especially p50/p95/p99). These three metrics provide a complete picture of service health. If Rate is flat but Errors spike, something is broken. If Rate and Errors are flat but Duration increases, something is slow. Coined by Tom Wilkie at Grafana Labs.

The USE Method (for Resources)

For every resource (CPU, memory, disk, network), measure: Utilization (how busy it is, as a percentage), Saturation (how much queued work there is beyond capacity), and Errors (count of error events). Created by Brendan Gregg. Example: a disk at 90% utilization with high saturation (long I/O queue) is a bottleneck even before errors appear.

RED vs. USE: Which Method for Which Situation?

Use RED for request-driven services — API servers, web services, microservices, anything that handles requests from users or other services. RED tells you if the service is healthy from the user's perspective.

Use USE for infrastructure resources — CPUs, memory, disks, network interfaces, thread pools, connection pools, database connections. USE tells you if the underlying resources are healthy from the system's perspective.

ScenarioMethodKey Metrics to Check
API endpoint is slow RED Duration p99 increased from 50ms to 500ms. Rate is steady. Error rate unchanged. Diagnosis: something downstream is slow but not failing.
Database is struggling USE CPU utilization 95%. Saturation: 50 queries waiting in queue. The database is maxed out — either optimize queries or scale up.
Payment service failing RED Error rate jumped from 0.1% to 15%. Rate dropped (users getting errors and leaving). Duration increased (retries and timeouts). Something is broken.
Redis running out of memory USE Memory utilization 98%. Saturation: evictions increasing (Redis is removing keys to make room). Errors: none yet, but imminent if memory hits 100%.

In practice, you use both methods together. RED alerts you that something is wrong ("checkout latency is up"). USE helps you find the root cause ("the database connection pool is saturated — all 50 connections are in use and 30 queries are waiting").

Sizing & Tradeoffs

SLO-Driven Sizing Examples

Observability infrastructure should be sized to meet your Service Level Objectives, not arbitrary volume targets. Start with the detection requirements, then derive the pipeline configuration backward from there.

Context: 500 microservices, high traffic during flash sales, p99 latency SLO of 300ms on checkout.

SLO-Driven Capacity Planning

Traditional capacity planning asks “how much data will we generate?” SLO-driven capacity planning inverts the question: “What pipeline throughput and latency do we need to protect our SLOs?”

The process follows four steps:

  1. Define detection requirements. For each SLO, determine the maximum acceptable detection time. Example: a 99.9% availability SLO with a 43-minute monthly error budget can tolerate at most 5-minute detection — otherwise a single incident could burn the entire budget before anyone notices.
  2. Derive pipeline latency budget. Break detection time into segments: ingestion latency + aggregation window + rule evaluation interval + notification delivery. Each segment constrains a different infrastructure component. For a 2-minute detection budget: 15s scrape + 30s aggregation + 15s eval + 10s notification + 50s margin.
  3. Size for the SLO, then add headroom. Calculate the minimum cardinality, throughput, and storage to achieve detection within budget. Then add 2–3x headroom for traffic spikes (flash sales, viral events). A pipeline that saturates at peak load will introduce lag — which is invisible SLO risk.
  4. Validate with failure injection. Simulate the incidents your SLOs are designed to catch (latency injection, error injection, capacity reduction) and measure actual detection time. If detection exceeds your budget, the pipeline is undersized regardless of what the capacity math says.
SLO TypeDetection BudgetKey Pipeline ConstraintSizing Rule of Thumb
Availability (99.9%)≤ 5 minScrape interval + eval interval15s scrape, 15s eval, 2x Alertmanager HA
Latency (p99 < 500ms)≤ 2 minHistogram resolution + recording rules15s scrape, 10-bucket histograms, 50 recording rules
Freshness (data < 30s old)≤ 30sStreaming pipeline end-to-end latencyKafka + Flink, 5s windows, in-memory ring buffer
Per-tenant SLO≤ 5 min per tenantRule count × tenants × eval intervalShard rulers, partition storage by tenant, cap cardinality per tenant

The common mistake is sizing observability for average load. SLOs break during peak load — which is exactly when your observability pipeline is also under maximum stress. Always size for the 95th percentile of your traffic, not the median.

Scaling Journey

0 – 1K Users

printf Debugging

Application logs to stdout. console.log / print(). Errors in the terminal. Maybe a free-tier error tracker like Sentry. No metrics, no traces. Debugging means reading logs on the single server.

1K – 100K Users

Structured Logging + Basic Metrics

Switch to JSON structured logging. Ship logs to a centralized store (ELK, CloudWatch, Loki). Add Prometheus + Grafana for basic metrics (RED method). Set up PagerDuty. Signal to transition: first incident where you couldn't find the relevant log because it was on a server that got recycled.

100K – 10M Users

Distributed Tracing

Microservices make logs alone insufficient. Add Jaeger or Grafana Tempo. Instrument with OpenTelemetry. Add correlation IDs linking logs to traces. Implement SLOs and error budgets. Signal: you have a latency spike and can't tell which of 15 services is responsible.

10M+ Users

Unified Pipeline + Profiling

OpenTelemetry Collector as a central pipeline with routing, sampling, and enrichment. Tail-based sampling for traces. Continuous profiling (Pyroscope). eBPF-based observability for uninstrumented services. Observability data lake for long-term analysis. Dedicated SRE team managing the platform.

What Can Go Wrong

1. Log Volume Explosion

What happened: A team enabled DEBUG logging in production to diagnose a subtle bug. They forgot to revert. Log volume went from 50 GB/day to 2 TB/day. The centralized logging cluster (Elasticsearch) couldn't keep up — indexing fell behind, queries timed out, and the cluster went into a red state. All teams lost access to logs for 6 hours during a critical deployment window.

Root cause: No guardrails on log volume per service. No rate limiting on the log pipeline. DEBUG-level log statements logged request/response bodies, which included large JSON payloads.

Prevention: Enforce log level controls per environment (never allow DEBUG in prod without a TTL). Rate-limit log ingestion per service. Set up alerts on log volume anomalies. Use sampling for verbose log categories.

2. Metric Cardinality Bomb

What happened: A well-intentioned developer added a user_id label to an HTTP request duration histogram. With 2 million active users, this created tens of millions of time series. Prometheus memory usage spiked from 8 GB to 120 GB, the instance OOM-killed, and all alerts stopped firing — including the critical alerts for the billing system, which was simultaneously experiencing errors that went undetected for 45 minutes.

Root cause: No cardinality checks in the metrics pipeline. No code review process that caught high-cardinality labels.

Prevention: Enforce cardinality limits (Prometheus sample_limit per scrape target). Use CI/CD linters to flag unbounded label values. Educate developers: user IDs, IP addresses, request IDs, and URLs must never be metric labels — put them in logs or trace attributes instead.

3. Trace Sampling Misses Rare Errors

What happened: A payment processing system used 0.1% head-based trace sampling. A race condition caused double-charges for 0.02% of transactions — roughly 50 per day. None of these transactions were captured in traces because the sampling rate was too low and head-based sampling doesn't bias toward errors. The issue went undetected for weeks until a customer complained.

Root cause: Head-based sampling randomly decides at the start of a request, before errors occur. Rare errors are statistically unlikely to be sampled.

Prevention: Use tail-based sampling that keeps 100% of error traces and high-latency traces. For critical paths (payments, auth), consider 100% trace collection. Set up metric-based alerts (error rate) as a safety net independent of tracing.

SLOs and Error Budgets

Define SLOs (Service Level Objectives) based on user-facing behavior, not internal metrics. Example: "99.9% of API requests return a non-5xx response within 500ms." This gives you an error budget of 0.1% — roughly 43 minutes of downtime per month. When the budget is nearly exhausted, freeze feature releases and focus on reliability. Track burn rate: if you're consuming error budget 10x faster than expected, that's an active incident.

Alert Design: Symptom-Based, Not Cause-Based

Bad alert: "CPU usage > 90% on host web-42." CPU might be high but users are perfectly fine — maybe it is running an efficient batch job. This alert fires constantly and gets ignored. Good alert: "p99 latency > 500ms for 5 minutes on /api/checkout." This directly measures user impact. The CPU alert might be useful as a low-priority warning in a Slack channel, but it should never page someone at 3 AM.

Every alert should include: a link to the relevant dashboard, a link to a runbook with diagnosis steps, and the SLO it relates to.

Alert Fatigue: The Silent Killer

If your on-call team gets 100 alerts per day, they stop reading them. Every alert becomes "snooze and move on." When the truly critical alert fires — the one that means customers are losing money — it gets lost in the noise and goes unnoticed for 45 minutes. Target: fewer than 5 actionable alerts per on-call shift. If an alert fires more than 5 times in a month without leading to action, delete it or tune it.

SLO-Based Alerting: Burn Rate

Instead of alerting on instantaneous threshold violations ("error rate > 1% right now"), alert based on how fast you are consuming your error budget. This is called burn rate alerting.

Setup: Suppose your SLO is 99.9% availability over 30 days. That gives you an error budget of 0.1% — which equals 43.2 minutes of downtime per month. A "1x burn rate" means you are consuming the budget at exactly the expected rate (you will use all 43.2 minutes over 30 days). A "14.4x burn rate" means you are burning 14.4 times faster — your entire monthly budget will be gone in about 2 days.

Burn RateMeaningBudget Exhausted InAction
1x Normal error rate 30 days (as designed) No action needed
2x Double the normal errors 15 days Create a ticket for business hours
14.4x Burning 14.4 times faster ~2 days Page on-call immediately
36x Burning 36 times faster ~20 hours Urgent page — wake people up

The key insight: a 14.4x burn rate means your 30-day error budget will be exhausted in roughly 2 days. That is urgent enough to wake someone up. A 2x burn rate means you will run out in 15 days — concerning, but it can wait until business hours. This approach eliminates false alarms from brief, harmless spikes while still catching sustained problems before they exhaust your budget.

Multi-window burn rate (recommended by the Google SRE book): use two windows to avoid false positives. For example, page only if both the 5-minute burn rate exceeds 14.4x and the 1-hour burn rate exceeds 6x. The short window catches sudden spikes. The long window confirms the problem is sustained, not a 30-second blip.

Debug Playbook

User reports "it's slow":

→ Check the service's RED metrics dashboard. Is latency elevated?

  → Yes: Is error rate also elevated?

    → Yes: Find a sample error trace. Check the failing span. Read the correlated error logs for that trace_id.

    → No (slow but not erroring): Find a slow trace (p99). Identify the bottleneck span. Is it a downstream service? A database query? External API?

  → No (latency looks normal): Check if the user is in a specific region or on a specific client version. Check CDN/edge metrics. Problem may be client-side.

Interactions & Pitfalls

Observability + Microservices

This is the most critical pairing. In a monolith, a stack trace tells you the full call path. In microservices, a request touches 5–20 services, and a stack trace only shows you one of them. Without distributed tracing, debugging a latency spike means manually correlating timestamps across 20 different log streams. Pitfall: if even one service doesn't propagate trace context, the trace breaks into disconnected fragments. Ensure all services use a compatible context propagation format (W3C Trace Context or B3).

Observability + Load Balancing

Load balancer metrics (connection count, response time per backend, 5xx rate per backend) are some of the most valuable signals. They give you a single vantage point to compare backend health. Pitfall: if you only have metrics from individual backends but not the load balancer, you might miss imbalanced load distribution. A backend can look healthy in isolation while the LB is sending it 10x the traffic of other backends.

Observability + Circuit Breakers

Circuit breaker state transitions (closed → open → half-open) should be emitted as both metrics and log events. Pitfall: if the circuit breaker opens and your metrics stop receiving data from the downstream service, a naive dashboard might show "no errors" — because no requests are being made. Track the breaker state itself as a gauge metric (circuit_breaker_state{service="payment"} = 1 for open).

Observability + Message Queues

Tracing across async boundaries (producer → queue → consumer) requires injecting trace context into message headers. The trace should show the full flow even when the consumer processes the message minutes later. Pitfall: if you create a new trace on the consumer side instead of continuing the producer's trace, you lose the end-to-end view. Also, consumer lag should be a top-line metric: kafka_consumer_group_lag is one of the most important observability signals for event-driven systems.

Cost Considerations

Cost Drivers

Observability can easily become the largest line item in your infrastructure budget. The dominant cost drivers vary by pillar:

  • Logs: Storage and ingestion volume. This is typically 60–80% of observability cost. Every byte you log costs money to ingest, index, store, and query.
  • Metrics: Cardinality (number of unique time series). Each series costs ~20 bytes/sample × 4 samples/min = 115 KB/day. 1M series = 115 GB/day of metric storage.
  • Traces: Span volume and retention. Mitigated by sampling.

SaaS Pricing Reality

Datadog, Splunk, and New Relic charge per host, per GB ingested, or per million spans. A medium-sized microservices platform (50 hosts, 100 services, 5K RPS) can easily reach $50K–$150K/year. Large enterprises (1000+ hosts) commonly pay $500K–$2M/year. The convenience is real — no infrastructure to manage, excellent UIs, ML-powered anomaly detection — but the cost scales super-linearly with growth.

Self-Hosted Alternatives

The Grafana stack (Loki for logs, Prometheus/Mimir for metrics, Tempo for traces) can reduce costs by 10–40x at scale. Tradeoffs: you need 1–3 engineers maintaining the platform, upgrades require planning, and you miss SaaS features like ML-based root cause analysis. A common hybrid approach: use SaaS for the first year (move fast), then migrate critical telemetry to self-hosted when the bill becomes painful.

Cost Optimization Tips

  • Drop verbose log fields — don't log request/response bodies in production; log them at DEBUG level only
  • Sample traces intelligently — keep 100% of errors and slow requests, sample 1–5% of successful requests
  • Use log levels — INFO for normal operations, WARN for anomalies, ERROR for failures. Production should be INFO+ only.
  • Tiered retention — 7 days hot (SSD), 30 days warm (HDD), 90+ days cold (S3 Glacier)
  • Aggregate metrics at the edge — use the OTel Collector to pre-aggregate histograms and drop unused labels before they reach your metrics backend

Real-World Examples

Google Dapper

Google's Dapper (2010 paper) pioneered distributed tracing at scale. It introduced the concepts of trace IDs, span IDs, and parent spans that all modern tracing systems use. Dapper traced billions of requests across Google's infrastructure with extremely low overhead (<0.01% performance impact) by embedding instrumentation in the RPC framework (Stubby) so every service was automatically traced. The key insight: if you instrument the RPC layer, you get tracing for free without developer effort. This idea directly inspired Zipkin (Twitter), Jaeger (Uber), and OpenTelemetry.

Uber's Jaeger

Uber built Jaeger in 2015 to trace requests across thousands of microservices. At peak, Uber's platform processes millions of traces per second. Jaeger uses adaptive sampling — it adjusts the sampling rate per service and per endpoint based on traffic volume, ensuring low-traffic endpoints get higher sampling while high-traffic endpoints are sampled more aggressively. Jaeger was donated to the CNCF and is now one of the most widely used open-source tracing backends. Uber also developed M3, a metrics platform handling billions of time series, because Prometheus alone couldn't scale to their needs.

Netflix's Atlas

Netflix built Atlas, a custom time-series metrics system, to handle their scale (hundreds of millions of time series from thousands of instances). Atlas uses an in-memory architecture with dimensional data model (like Prometheus labels) and a stack-based query language. Netflix uses Atlas to power real-time dashboards that detect streaming quality issues within seconds. Their philosophy: metrics should be cheap enough that developers never hesitate to add a new one. Atlas achieves this by keeping per-series storage cost extremely low through aggressive rollup and pre-aggregation.

Code Example

OpenTelemetry-style instrumentation in Python — creating spans, adding attributes, and context propagation across service boundaries:

from opentelemetry import trace, metrics
from opentelemetry.context import attach, detach
from opentelemetry.trace.propagation import inject, extract
import logging, json, time

tracer = trace.get_tracer("order-service")
meter = metrics.get_meter("order-service")

# Metrics: counter and histogram
order_counter = meter.create_counter("orders_total",
    description="Total orders processed")
latency_hist = meter.create_histogram("order_duration_seconds",
    description="Order processing latency")

logger = logging.getLogger("order-service")

def process_order(request):
    # Extract trace context from incoming request headers
    ctx = extract(request.headers)
    token = attach(ctx)

    try:
        with tracer.start_as_current_span("process_order") as span:
            start = time.time()
            span.set_attribute("user.id", request.user_id)
            span.set_attribute("order.total", request.total)

            # Structured log with trace correlation
            logger.info(json.dumps({
                "event": "order_started",
                "user_id": request.user_id,
                "trace_id": span.get_span_context().trace_id,
                "span_id": span.get_span_context().span_id,
            }))

            # Call downstream payment service (propagate context)
            headers = {}
            inject(headers)  # injects trace context into headers
            result = call_payment_service(request, headers)

            # Record metrics
            order_counter.add(1, {"status": "success"})
            latency_hist.record(time.time() - start)
            return result
    except Exception as e:
        order_counter.add(1, {"status": "error"})
        span.set_status(trace.Status(trace.StatusCode.ERROR, str(e)))
        span.record_exception(e)
        raise
    finally:
        detach(token)

How to Validate Your Design

Inject Known Failures

For each pillar, inject a controlled failure and verify detection:

  • Logs: Trigger a known error condition. Verify the error appears in your centralized logging within 60 seconds and contains the correct trace ID, service name, and error details.
  • Metrics: Generate a burst of 5xx errors. Verify the error rate metric spikes on dashboards and an alert fires within your configured window (e.g., 5 minutes).
  • Traces: Send a test request through the full service chain. Verify the trace shows all expected spans, with correct parent-child relationships and no gaps.

Chaos Testing

Use toxiproxy or Chaos Mesh to inject latency (e.g., add 2s delay to a downstream service). Verify that: (1) the latency appears in traces as an elongated span, (2) the p99 metric increases, (3) an SLO-based alert fires, and (4) the debug playbook leads you to the correct root cause. Also test your observability pipeline itself — what happens if the OTel Collector crashes? Do you lose telemetry? Is there a dead letter queue?

Load Testing with Observability Validation

Use k6 or Locust to generate realistic load. During the test, verify: dashboards update in real time, query latency on your metrics backend stays under 5 seconds, log search returns results within 10 seconds, and trace lookup by ID completes in under 2 seconds. Common mistake: testing observability on low traffic and being surprised when it falls over at 10x load — test your observability pipeline at the same scale as your application.

Trace Completeness Audit

Periodically sample 100 traces and manually verify: every expected service appears, no "orphan" spans (spans with missing parents), span durations are plausible (no negative durations or missing timestamps), and error spans have corresponding log entries. Automate this with a trace validator that runs as a periodic job.

Interview Questions

How would you design an observability platform for a microservices architecture?

Start by establishing the three pillars. For metrics, deploy Prometheus (or Mimir for multi-tenant) with a 15s scrape interval, using the RED method for every service. For logging, standardize on JSON structured logs with mandatory fields (timestamp, service, level, trace_id), ship via Fluent Bit to Grafana Loki or Elasticsearch. For tracing, instrument all services with OpenTelemetry SDKs, propagate context via W3C Trace Context headers, and store traces in Jaeger or Grafana Tempo.

The critical architectural decision is the OpenTelemetry Collector as a central pipeline between your applications and backends. It acts as a buffer, handles tail-based sampling, enriches telemetry with metadata (Kubernetes labels, environment), and decouples your application instrumentation from your backend choice. Deploy it as a DaemonSet (one per node) for metrics and a Deployment (centralized) for tail-based trace sampling.

Define SLOs from day one: error rate <0.1%, p99 latency <500ms for user-facing services. Set up SLO-based alerting with burn rate windows. Create standardized dashboards per service (RED metrics) and per-team golden signal dashboards.

Follow-ups: How do you handle multi-tenancy? How do you prevent cardinality explosions? What's your sampling strategy at 100K RPS?

Explain the difference between monitoring and observability

Monitoring answers known questions: "Is the server up? Is CPU above 80%? Are there more than 10 errors per minute?" You define these checks in advance, and monitoring tells you when thresholds are breached. It's reactive and check-based.

Observability lets you ask new questions you didn't anticipate. When a novel failure occurs — "why are users in Brazil seeing 3x latency on checkout, but only on iOS, but only when they have more than 5 items in cart" — observability gives you the tools to explore and correlate signals to find the answer without deploying new instrumentation.

The key enabler is high-cardinality, high-dimensionality data. Monitoring stores a few aggregated metrics. Observability stores rich, detailed events (structured logs, trace spans with attributes) that can be sliced and diced along any dimension. In practice, you need both: monitoring for known failure modes with well-tuned alerts, observability for unknown-unknowns and novel debugging.

Follow-ups: Can you have observability without monitoring? How does observability change the on-call experience? What's the cost tradeoff?

How does distributed tracing work across service boundaries?

When Service A makes an RPC call to Service B, the tracing SDK injects trace context into the outgoing request (typically as HTTP headers: traceparent and tracestate per the W3C Trace Context standard). This context includes the trace ID, the current span ID (which becomes the parent span ID for Service B), and sampling flags.

Service B's tracing SDK extracts this context from the incoming request headers. It creates a new span with the same trace ID, a new span ID, and the parent span ID from the incoming context. This linking creates a tree structure: Service A's span is the parent, Service B's span is the child. If Service B calls Service C, the same propagation continues, building a deeper tree.

Each service independently exports its spans to the tracing backend (Jaeger, Tempo). The backend assembles the full trace by collecting all spans with the same trace ID and building the tree from parent-child relationships. The result is a waterfall visualization showing the timeline of each span, making it immediately clear where time was spent and where errors occurred.

Follow-ups: How does tracing work across async boundaries (message queues)? What happens when one service doesn't propagate context? How do you handle trace context in polyglot environments?

How do you handle high-cardinality metrics without blowing up your metrics backend?

High-cardinality metrics are the single most common way teams accidentally break their monitoring. The rule is simple: never use unbounded values as metric labels. User IDs, request IDs, IP addresses, full URLs, email addresses — these are attributes for logs and trace spans, not metric labels.

Concrete defenses: (1) Set sample_limit per scrape target in Prometheus so a single service can't create millions of series. (2) Use CI/CD linting to flag new metrics with potentially high-cardinality labels. (3) Pre-aggregate in the OTel Collector — drop unnecessary labels, merge similar endpoints (e.g., /users/123 and /users/456 become /users/:id). (4) Use recording rules to pre-compute frequently-queried aggregations. (5) For metrics backends like Mimir or Cortex, configure cardinality limits per tenant.

When you genuinely need per-user metrics (e.g., "how many requests did user X make?"), use exemplars — attach a few sample trace IDs to your aggregated metric so you can jump from the metric to specific traces without creating per-user series.

Design an alerting system that avoids alert fatigue

Alert fatigue is the number one operational failure mode: when on-call engineers get 200 alerts per shift, they ignore all of them, including the critical one. The fix is a layered approach.

Layer 1 — SLO-based alerts only page. Only alert on user-facing symptoms, not infrastructure causes. "p99 latency SLO burn rate > 10x for 5 minutes" pages. "CPU > 90%" does not page — it goes to a low-priority Slack channel. Use multi-window, multi-burn-rate alerting per the Google SRE book: a fast burn (2% of error budget in 1 hour) pages immediately; a slow burn (5% in 6 hours) creates a ticket.

Layer 2 — Deduplication and grouping. If 50 pods in a deployment are all failing the same health check, that's one alert, not 50. Alertmanager's group_by and inhibition rules are essential. A cluster-wide network failure should suppress all downstream service alerts and produce one root-cause alert.

Layer 3 — Every alert must be actionable. If an alert fires and the on-call can't do anything about it, delete the alert. Every alert needs: a runbook link, clear severity (page vs. ticket vs. info), and an owner. Review alert frequency monthly; if an alert fires >5 times without leading to action, delete or tune it.

Follow-ups: How do you set error budgets for new services with no baseline? What's the difference between alerting on raw error counts vs. error rates? How do you handle cascading failures in alerting?

How do you trace a request through an async message queue?

The key is context propagation through message headers. When the producer publishes a message, the tracing SDK injects the current trace context (trace ID, span ID) into the message headers (Kafka headers, RabbitMQ message properties, SQS message attributes). The producer's span represents "publish message" and ends when the message is accepted by the broker.

On the consumer side, the SDK extracts trace context from the message headers and creates a new span linked to the producer's span. The consumer span represents "process message." This creates a trace that spans the producer, the queue (visible as the time gap between producer span end and consumer span start — this gap is your consumer lag), and the consumer's processing.

Challenges: (1) Batching — if the consumer processes 100 messages in a batch, you can't link to 100 parent spans. Use span links (OpenTelemetry feature) to reference the producer spans without creating a direct parent-child relationship. (2) Fan-out — one message triggers processing in 5 consumers. Each consumer creates its own child span. (3) Long delays — if a message sits in the queue for hours, the trace becomes hard to visualize. Consider starting a new trace on the consumer and using span links to reference the original producer trace.

Follow-ups: How do you handle dead-letter queue messages in traces? How do you measure end-to-end latency across async boundaries? What about exactly-once processing and trace duplication?

Key Takeaways

  • Observability is not monitoring. Monitoring answers known questions; observability lets you ask new ones. You need both.
  • Three pillars, correlated. Logs, metrics, and traces are individually useful but transformative when linked by correlation IDs (trace_id, span_id).
  • OpenTelemetry is the standard. It unifies instrumentation across all three pillars and all major languages. Instrument once, export anywhere.
  • RED for services, USE for resources. Rate/Errors/Duration covers service health. Utilization/Saturation/Errors covers infrastructure health.
  • Cardinality kills metrics. Never use unbounded values (user IDs, IPs) as metric labels. High-cardinality data belongs in logs and traces.
  • Sampling is mandatory at scale. Prefer tail-based sampling that keeps 100% of errors and outliers.
  • Alert on symptoms, not causes. Page on SLO burn rate, not CPU usage. Every alert must be actionable with a runbook link.
  • Observability costs real money. Log volume is the dominant cost driver. SaaS pricing can be 10–40x more expensive than self-hosted at scale.
  • Your observability pipeline is infrastructure too. Monitor your monitoring. If Prometheus goes down, you lose all alerts.
