Exactly-Once Payment Processing

When a network hiccup can double-charge a customer $50, how do you build a payment pipeline that processes every transaction exactly once — even when components crash and retry?

NALSD Walkthrough • ~15 min read

The Scenario

You are designing the core payment processing pipeline for an e-commerce platform. Every payment involves three atomic operations:

  1. Debit the customer's account (reduce their balance).
  2. Credit the merchant's account (increase their balance).
  3. Record the transaction in the ledger (audit trail).

Scale

Payment rate1,000 payments/sec

The Non-Negotiable Requirements

  • Never double-charge a customer. If a payment is processed twice, the customer loses $50 they should not have been charged. At scale, even a tiny duplicate rate is catastrophic.
  • Never lose a payment. If the customer is debited but the merchant is not credited, money vanishes. The ledger must be consistent.
  • Survive crashes and retries. Networks drop packets. Consumers crash. Kafka redelivers messages. The system must handle all of these without corrupting financial data.

Back-of-Envelope Math

Let us quantify the cost of duplicates to understand why "mostly correct" is unacceptable for payments.

Cost of Duplicate Processing

Average payment:            $50

If 0.01% are duplicated (1 in 10,000):
  Duplicates per day:       8,640
  Double-charged amount:    8,640 x $50 = $432,000/day

If 0.1% are duplicated (1 in 1,000):
  Duplicates per day:       86,400
  Double-charged amount:    86,400 x $50 = $4,320,000/day

If 1% are duplicated:
  Double-charged amount:    $43,200,000/day

How Often Do Failures Actually Happen?

At 1,000 payments/sec, even rare events happen frequently:

Network packet loss rate:  ~0.01% (typical cloud)
  = 0.1 payments/sec affected = 8,640/day

Consumer crash rate:       ~1 crash per hour (aggressive but realistic)
  Each crash redelivers the last batch (~100 messages)
  = 24 x 100 = 2,400 redeliveries/day

Kafka redelivery (ack lost): ~0.001%
  = 0.01/sec = 864/day

Total events that could cause duplicates: ~12,000/day

Without idempotency protection, each of these events could produce a duplicate charge. The system will encounter these failures daily.

Even 0.01% duplicates cost $432,000 per day. This is not a rounding error. Customers will notice, dispute charges, lose trust, and regulators will intervene. The duplicate rate must be effectively zero.

Three Failure Scenarios That Cause Duplicates

Before designing solutions, let us understand exactly how duplicates arise. Each scenario is a different point in the pipeline where a failure causes a retry, and the retry produces a duplicate.

Scenario 1: Producer Retry → Duplicate Message in Kafka

Timeline:
  T1: Producer sends payment P1 to Kafka.
  T2: Kafka writes P1 to the partition log. Success.
  T3: Kafka sends ACK back to producer.
  T4: ACK is lost in the network (packet drop, timeout).
  T5: Producer waits for ACK, times out, assumes failure.
  T6: Producer retries: sends P1 again.
  T7: Kafka writes P1 (again) to the partition log.

Result: Kafka now has TWO copies of payment P1.
Consumer will process P1 twice = double charge.

Scenario 2: Consumer Crash → Kafka Redelivery

Timeline:
  T1: Consumer reads payment P1 from Kafka.
  T2: Consumer processes P1: debits customer $50. Balance: $500 → $450.
  T3: Consumer crashes BEFORE committing its Kafka offset.
  T4: Consumer restarts.
  T5: Kafka sees no offset commit for P1. Redelivers P1.
  T6: Consumer reads P1 again.
  T7: Consumer processes P1 again: debits customer $50. Balance: $450 → $400.

Result: Customer debited twice. Balance should be $450 but is $400.

Scenario 3: Partial Processing → Inconsistent State

Timeline:
  T1: Consumer reads payment P1 from Kafka.
  T2: Consumer debits customer $50. Balance: $500 → $450. Success.
  T3: Consumer tries to credit merchant $50. Merchant service is down.
  T4: Consumer retries credit later. Also retries debit (defensive retry).
  T5: Debit executes again: $450 → $400.
  T6: Credit executes: merchant receives $50.

Result: Customer charged $100 instead of $50.
  Merchant received $50. $50 has vanished.

Each scenario involves a different component (producer, consumer, downstream service), but the root cause is the same: a retry re-executes an operation that already succeeded.

Naive Design

  1. Payment API receives a payment request, publishes to Kafka.
  2. Consumer reads from Kafka, debits the customer's account in the database.
  3. Consumer credits the merchant's account in the database.
  4. Consumer commits its Kafka offset.

No idempotency checks. No transactional guarantees across the debit and credit. Every failure scenario above will produce duplicates or inconsistencies.

What Breaks

FailureWhat HappensImpact

At 1,000 payments/sec with a conservative 0.1% failure rate, that is 1 problematic payment every second, or 86,400 per day. Each one is a potential double-charge or lost payment.

Iteration 1: Idempotency at Every Layer

Fixing Scenario 1: Kafka Idempotent Producer

Kafka provides a built-in mechanism for producer-side deduplication. When you enable enable.idempotence=true, Kafka assigns each producer a unique Producer ID (PID) and each message a sequence number. The broker tracks the last sequence number for each (PID, partition) pair.

How it works:

  Producer sends:  { PID: 42, seq: 7, payload: "payment P1" }
  Kafka writes it: partition log now has (PID=42, seq=7, P1)
  Kafka sends ACK.

  ACK is lost. Producer retries:
  Producer sends:  { PID: 42, seq: 7, payload: "payment P1" }  (same seq!)
  Kafka checks:    PID=42, last seq for this PID = 7. Already seen!
  Kafka responds:  ACK (but does NOT write a duplicate)

Result: Kafka topic has exactly one copy of P1, even though the producer sent it twice.
Cost: Enabling idempotent producers has near-zero performance overhead. It is a one-line configuration change. There is no reason not to enable it for any Kafka producer.

Fixing Scenario 2: Consumer-Side Idempotency Key

Even with an idempotent producer, the consumer can still process a message twice (if it crashes after processing but before committing its offset). The fix: make the consumer itself idempotent.

Every payment has a unique payment_id assigned by the upstream API. The consumer uses this as an idempotency key:

def process_payment(payment):
    payment_id = payment['payment_id']

    with db.transaction() as txn:
        # Step 1: Check if already processed
        if txn.exists("processed_payments", payment_id):
            log.info(f"Payment {payment_id} already processed. Skipping.")
            return  # idempotent: no-op on retry

        # Step 2: Debit customer
        txn.execute("""
            UPDATE accounts
            SET balance = balance - %s
            WHERE account_id = %s
        """, [payment['amount'], payment['customer_id']])

        # Step 3: Credit merchant
        txn.execute("""
            UPDATE accounts
            SET balance = balance + %s
            WHERE account_id = %s
        """, [payment['amount'], payment['merchant_id']])

        # Step 4: Record in ledger
        txn.execute("""
            INSERT INTO transactions (payment_id, customer_id, merchant_id, amount, ts)
            VALUES (%s, %s, %s, %s, NOW())
        """, [payment_id, payment['customer_id'],
              payment['merchant_id'], payment['amount']])

        # Step 5: Mark as processed (idempotency record)
        txn.execute("""
            INSERT INTO processed_payments (payment_id, processed_at)
            VALUES (%s, NOW())
        """, [payment_id])

        # All five steps commit atomically.
        txn.commit()

Why This Works for Scenario 2

Normal flow:
  T1: Consumer reads P1 from Kafka.
  T2: BEGIN transaction.
  T3: Check processed_payments → not found → proceed.
  T4: Debit customer: $500 → $450.
  T5: Credit merchant: $1000 → $1050.
  T6: Insert into transactions ledger.
  T7: Insert into processed_payments: {payment_id: "abc123"}.
  T8: COMMIT. All changes are now durable.
  T9: Commit Kafka offset.

Crash-and-retry flow:
  T1: Consumer reads P1, starts transaction.
  T2: Debit customer: $500 → $450.
  T3: CRASH before COMMIT.
  T4: Transaction rolls back automatically. Customer balance returns to $500.
  T5: Consumer restarts, Kafka redelivers P1.
  T6: Consumer reads P1, starts new transaction.
  T7: Check processed_payments → not found (previous insert was rolled back).
  T8: Process normally: debit, credit, record, mark processed.
  T9: COMMIT. Success.

Result: Payment processed exactly once. No double-charge.

What About Scenario 3?

Notice that in the code above, the debit and credit happen in the same database transaction. This works only if both accounts are in the same database. If the merchant's account is in a different service (which is common in microservices), we need a different approach. That leads us to Iteration 2.

Critical subtlety: The idempotency check (step 3) and the idempotency record (step 5) MUST be in the same database transaction as the business logic (steps 4-6). If they are separate, a crash between the business logic and the idempotency record creates a window where a retry would re-execute the business logic.

Iteration 2: The Outbox Pattern for Cross-Service Consistency

The Problem: Dual Writes

In a microservices architecture, the customer account and merchant account are likely in different services with different databases. The consumer cannot use a single DB transaction to update both. It must write to its own database AND publish a message to the merchant service. This is the dual write problem:

The dual write problem:

  Step 1: Write to local DB (debit customer).     → succeeds
  Step 2: Publish to Kafka (credit merchant).      → fails (network error)

  Result: Customer debited, merchant never credited. Money lost.

  OR:

  Step 1: Publish to Kafka (credit merchant).      → succeeds
  Step 2: Write to local DB (debit customer).      → fails (DB error)

  Result: Merchant credited, customer never debited. Free money.

No matter which order you do them, a failure between the two steps
leaves the system inconsistent. You cannot make two separate systems
(DB and Kafka) atomically consistent without a coordination protocol.

The Outbox Pattern

The insight: instead of writing to the DB and Kafka, write everything to the DB in one transaction. Include an outbox table that acts as a local message queue. A separate process reads the outbox and publishes to Kafka.

The Consumer Code

def process_payment(payment):
    payment_id = payment['payment_id']

    with db.transaction() as txn:
        # Idempotency check
        if txn.exists("processed_payments", payment_id):
            return

        # Debit customer (local to this service's DB)
        txn.execute("""
            UPDATE accounts SET balance = balance - %s
            WHERE account_id = %s
        """, [payment['amount'], payment['customer_id']])

        # Write to outbox (NOT directly to Kafka)
        txn.execute("""
            INSERT INTO outbox (id, aggregate_type, aggregate_id, event_type, payload)
            VALUES (%s, 'merchant', %s, 'credit', %s)
        """, [uuid4(), payment['merchant_id'],
              json.dumps({
                  'payment_id': payment_id,
                  'merchant_id': payment['merchant_id'],
                  'amount': payment['amount']
              })])

        # Record transaction and idempotency key
        txn.execute("""
            INSERT INTO transactions (payment_id, customer_id, amount, status)
            VALUES (%s, %s, %s, 'debited')
        """, [payment_id, payment['customer_id'], payment['amount']])

        txn.execute("""
            INSERT INTO processed_payments (payment_id, processed_at)
            VALUES (%s, NOW())
        """, [payment_id])

        txn.commit()
        # At this point, the debit AND the outbox entry are atomically committed.

The Outbox Relay

A separate process (or use Debezium with Kafka Connect for zero custom code) reads new rows from the outbox table and publishes them to Kafka:

def relay_outbox_events():
    while True:
        rows = db.query("""
            SELECT * FROM outbox
            WHERE published = FALSE
            ORDER BY created_at
            LIMIT 100
        """)

        for row in rows:
            kafka_producer.send(
                topic=f"{row.aggregate_type}-events",
                key=row.aggregate_id,
                value=row.payload
            )
            db.execute("""
                UPDATE outbox SET published = TRUE WHERE id = %s
            """, [row.id])

        time.sleep(0.1)  # poll interval
Why Debezium is better: Instead of polling the outbox table, Debezium uses Change Data Capture (CDC) to stream database changes directly to Kafka. It reads the database's transaction log (WAL in Postgres, binlog in MySQL), so it captures outbox inserts in real-time with no polling delay and no extra load on the database.

The Merchant Service

The merchant service consumes from the merchant-events Kafka topic and applies its own idempotency check:

def process_merchant_credit(event):
    payment_id = event['payment_id']

    with db.transaction() as txn:
        # Merchant-side idempotency check
        if txn.exists("processed_credits", payment_id):
            return  # already credited, skip

        txn.execute("""
            UPDATE merchant_accounts SET balance = balance + %s
            WHERE merchant_id = %s
        """, [event['amount'], event['merchant_id']])

        txn.execute("""
            INSERT INTO processed_credits (payment_id, credited_at)
            VALUES (%s, NOW())
        """, [payment_id])

        txn.commit()

Failure Recovery: Step-by-Step Trace

Let us trace through every possible crash point and verify that no duplicate or lost payment occurs.

Happy Path

Step 1: Payment event {payment_id: "abc123", customer: C1, merchant: M1, amount: $50}
        arrives in Kafka topic "payments".

Step 2: Consumer reads event.

Step 3: BEGIN transaction.

Step 4: Check processed_payments for "abc123" → NOT FOUND → proceed.

Step 5: Debit C1: balance $500 → $450.

Step 6: Insert into outbox: {merchant: M1, amount: $50, payment_id: "abc123"}.

Step 7: Insert into processed_payments: {payment_id: "abc123"}.

Step 8: COMMIT transaction. All three writes are now durable.

Step 9: Commit Kafka offset for this message.

Step 10: Outbox relay reads the outbox row, publishes to "merchant-events" topic.

Step 11: Merchant consumer reads event, checks processed_credits → NOT FOUND.

Step 12: Credits M1: balance $1000 → $1050. Inserts processed_credits record. COMMIT.

Final state: C1 balance = $450, M1 balance = $1050. Correct.

Crash at Step 5 (During Debit, Before Commit)

Step 5: Debit C1 executes in-memory but...
        Consumer process crashes.
        Transaction was never committed → automatic rollback.
        C1 balance remains $500.
        Outbox row was never written.
        processed_payments record was never written.

Recovery:
  Consumer restarts. Kafka redelivers "abc123".
  Step 4: Check processed_payments → NOT FOUND (rollback erased it).
  Processing proceeds normally from scratch.

Result: Payment processed exactly once. No double-charge.

Crash at Step 9 (After DB Commit, Before Kafka Offset Commit)

Step 8: DB transaction committed successfully.
        C1 debited. Outbox row written. Idempotency record saved.
Step 9: Consumer crashes before committing Kafka offset.

Recovery:
  Consumer restarts. Kafka redelivers "abc123" (offset not committed).
  Step 4: Check processed_payments → FOUND ("abc123" exists).
  Consumer skips this message. No re-processing.

Result: Payment processed exactly once. Idempotency key saved us.

Crash at Step 10 (Outbox Relay Fails)

Step 8: DB committed. Outbox row exists with published=FALSE.
Step 10: Relay crashes before publishing to Kafka.

Recovery:
  Relay restarts. Queries outbox WHERE published = FALSE.
  Finds the row. Publishes to "merchant-events".
  Marks published = TRUE.

  Merchant consumer processes the credit normally.

Result: Credit arrives late but is not lost. No duplicate.

Crash at Step 12 (Merchant Consumer Crashes After Credit)

Step 11: Merchant consumer reads event.
Step 12: Credits M1, but crashes before committing Kafka offset.

Recovery:
  Kafka redelivers the event to merchant consumer.
  Step 11: Check processed_credits → FOUND ("abc123" exists).
  Consumer skips. No double credit.

Result: Merchant credited exactly once.
The pattern: Every crash scenario is covered by the combination of (1) database transactions that make writes atomic, and (2) idempotency keys that prevent reprocessing on retry. These two mechanisms, applied consistently at every service boundary, guarantee exactly-once processing.

Iteration 3: Kafka Transactions (Alternative Approach)

If your pipeline is Kafka-to-Kafka (read from one topic, process, write to another topic), Kafka provides built-in exactly-once semantics (EOS) using transactions.

How Kafka Transactions Work

A Kafka transaction allows you to atomically: (1) consume messages from input topics, (2) produce messages to output topics, and (3) commit consumer offsets — all as a single atomic operation.

from kafka import KafkaConsumer, KafkaProducer

producer = KafkaProducer(
    transactional_id="payment-processor-1",  # enables transactions
    enable_idempotence=True
)
producer.init_transactions()

consumer = KafkaConsumer(
    'payments',
    group_id='payment-processors',
    enable_auto_commit=False,  # we commit offsets inside the transaction
    isolation_level='read_committed'  # only read committed messages
)

for message in consumer:
    payment = deserialize(message.value)

    producer.begin_transaction()
    try:
        # Process: produce result to output topic
        producer.send('merchant-credits', value=serialize({
            'payment_id': payment['payment_id'],
            'merchant_id': payment['merchant_id'],
            'amount': payment['amount']
        }))

        # Commit consumer offset as part of the transaction
        producer.send_offsets_to_transaction(
            {TopicPartition('payments', message.partition):
             OffsetAndMetadata(message.offset + 1)},
            consumer.group_id
        )

        producer.commit_transaction()
    except Exception:
        producer.abort_transaction()

What Happens on Failure

If crash before commit_transaction():
  - No messages are published to merchant-credits (transaction aborted).
  - Consumer offset is not committed.
  - On restart, Kafka redelivers the message.
  - Consumer processes it again from scratch.
  - No duplicate because the previous output was never committed.

If crash after commit_transaction():
  - Transaction was committed. Output is in merchant-credits.
  - Consumer offset was committed.
  - On restart, consumer picks up from the next message.
  - No duplicate, no lost message.

Outbox Pattern vs Kafka Transactions

DimensionOutbox PatternKafka Transactions
Atomicity scopeDB + Kafka (via outbox relay)Kafka-to-Kafka only
DB integrationNatural (writes to same DB)Requires separate DB writes outside the Kafka transaction
LatencyRelay adds ~100msSub-millisecond (native Kafka)
ComplexityOutbox table + relay processKafka configuration only
Best forServices with a database (most services)Stream processing (Kafka Streams, Flink)

In practice, most payment systems use the outbox pattern because they need to update a database (account balances) as part of processing. Kafka transactions are ideal for pure stream-processing stages where no external database is involved.

Idempotency Key Lifecycle

How long to keep idempotency records?

  Payment processing window: typically complete within minutes.
  Kafka consumer retry window: at most a few hours.
  Safety margin: keep records for 7 days.

  Storage cost:
    86.4M payments/day x 7 days = 604.8M records
    Each record: payment_id (16B) + timestamp (8B) + index overhead (~40B) = ~64B
    Total: 604.8M x 64B = ~38.7 GB

  This fits comfortably in a single database with a B-tree index on payment_id.
  Older records can be archived or deleted with a daily cleanup job.

Key Takeaways

  1. Exactly-once is achieved by making every step idempotent. There is no magic "exactly-once" switch. It is the combination of idempotent producers, idempotency keys, and atomic transactions that creates the end-to-end guarantee.
  2. The idempotency check and business logic must be in the same transaction. If they are separate, a crash between them creates a window where retries cause duplicates.
  3. The outbox pattern solves dual writes. Never write to a database and a message broker in two separate operations. Write everything to the database (including outbox events), then relay to the broker.
  4. Every service boundary needs its own idempotency. The payment consumer has its idempotency table. The merchant consumer has its own. Each service is responsible for protecting itself from duplicate deliveries.
  5. Do the dollar math. Abstract failure rates become concrete when translated to dollars. "0.01% duplicate rate" sounds acceptable until you calculate it equals $432,000/day in double-charges.
  6. Retries are inevitable. Networks lose packets. Processes crash. Kafka redelivers. Design for retries from the start, not as an afterthought.
