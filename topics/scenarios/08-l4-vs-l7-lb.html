Scenario #8: L4 vs L7 Load Balancing

A video streaming platform with three traffic types—API requests, multi-gigabyte uploads, and static assets—each demanding different load balancing strategies. One size does not fit all.

~15 min read

The Scenario

You are architecting the load balancing layer for a video streaming platform (think YouTube or Vimeo). The platform handles three fundamentally different traffic types:

Traffic Type Volume Payload Size Pattern Special Requirements
1. API Requests 5,000 QPS 5 KB avg response Short-lived request/response Route by URL path, auth validation
2. Video Uploads 500 concurrent 1–5 GB each Long-lived connections (200+ sec) Sticky sessions (multi-part upload)
3. Static Assets 50,000 QPS 30 KB avg (thumbnails) High-frequency, cacheable Serve from CDN, minimal routing

Back-of-Envelope Math

Bandwidth per Traffic Type

API Requests:
  5,000 QPS × 5 KB avg response = 25 MB/sec = 200 Mbps
  Standard request-response. Each request: ~50-200ms lifecycle.

Video Uploads:
  500 concurrent uploads × avg 2 GB file
  Average upload speed: 10 MB/sec per client
  Total inbound bandwidth: 500 × 10 MB/sec = 5 GB/sec = 40 Gbps
  Duration per upload: 2 GB / 10 MB/sec = 200 seconds
  Connections held: 500 simultaneous, each for ~3.3 minutes

Static Assets:
  50,000 QPS × 30 KB avg (thumbnails, JS, CSS)
  = 1,500 MB/sec = 1.5 GB/sec = 12 Gbps outbound

Total bandwidth: ~52 Gbps (dominated by uploads and static assets)
Total QPS: ~55,000 (dominated by static assets)

L4 vs L7: What Each Can See

The fundamental tradeoff: L4 is 10-20x faster but blind to application-level details. L7 can make intelligent routing decisions but must parse every HTTP request, adding latency and limiting throughput. Most production architectures use both.

Naive Design: One L7 Load Balancer for Everything

Nginx Config for Naive Approach

# Nginx config for naive single-LB approach
upstream api_servers {
    server api-1:8080;
    server api-2:8080;
    server api-3:8080;
}

upstream upload_servers {
    server upload-1:8080;
    server upload-2:8080;
}

upstream static_servers {
    server static-1:80;
    server static-2:80;
}

server {
    listen 443 ssl;

    location /api/ {
        proxy_pass http://api_servers;
    }

    location /upload/ {
        proxy_pass http://upload_servers;
        # Sticky sessions for multi-part uploads
        sticky cookie srv_id expires=1h;
        client_max_body_size 5g;
    }

    location /static/ {
        proxy_pass http://static_servers;
    }
}

What Breaks (With Numbers)

Problem 1: TLS Termination CPU Overload

L7 must terminate TLS for every connection (it needs to read HTTP headers).

TLS handshake cost:
  RSA 2048-bit: ~1ms per handshake on modern CPU
  ECDHE: ~0.5ms per handshake

  New connections per second (assuming keep-alive reuse of 10 requests):
    55,000 QPS / 10 = 5,500 new TLS handshakes/sec
    5,500 × 0.5ms = 2.75 CPU-seconds per second of TLS work

  With video uploads (large data):
    500 uploads × 5 GB/sec total = 5 GB/sec of data
    L7 must decrypt, inspect, re-encrypt all of this
    AES-256-GCM throughput: ~2 GB/sec per CPU core
    TLS data processing: 5 GB/sec / 2 GB/sec = 2.5 CPU cores just for upload TLS

  Total CPU for TLS: ~5 CPU cores dedicated to TLS alone
  This is manageable with one beefy LB, but leaves little headroom.

Problem 2: HTTP Parsing Overhead for Static Assets

The L7 LB must parse every HTTP request to read the URL path for routing.

  50,000 static asset requests/sec
  Each parse: ~10-50 microseconds (header parsing, URL matching)
  Total: 50K × 30μs = 1.5 seconds of CPU per second

  These 50K requests are just thumbnails. The L7 LB reads the URL,
  sees "/static/", and forwards to the static server.
  That URL parsing provides no value — the same routing could be done
  at L4 by port number or IP, without parsing HTTP at all.

  Result: 50K QPS of unnecessary HTTP parsing

Problem 3: Upload Connections Consume LB Resources

Each upload holds an L7 connection for ~200 seconds.
L7 LBs maintain per-connection state:
  - HTTP parser state
  - SSL session
  - Buffering (L7 may buffer request body)
  - Timeout tracking

  500 concurrent uploads × connection overhead per upload
  Each connection: ~50KB of memory (SSL buffers, HTTP state)
  500 × 50KB = 25MB  ← not a lot of memory, but...

  The problem is connection TIME:
  500 connections × 200 seconds = 100,000 connection-seconds
  The LB's connection pool is occupied by slow uploads

  If the LB has a max connection limit of 10,000:
    500 uploads use 5% of connections (for 200 sec each)
    API + static use 95% of connections (for ~100ms each)
    Utilization is fine... until upload traffic spikes during events.

The root issue: A single L7 LB is doing expensive work (TLS termination, HTTP parsing, header inspection) for 50K static asset requests that do not need any of this intelligence. It is also holding long-lived connections for uploads that consume resources disproportionately. We are using a smart (expensive) tool for dumb (cheap) work.

Iteration 1: Two-Tier Architecture

Use L4 in front for raw packet forwarding at high throughput. L7 behind for smart routing where it is actually needed.

How Traffic Is Split

L4 splits traffic WITHOUT parsing HTTP. How?

Option A: DNS-based splitting
  static.example.com → CDN IPs (Cloudflare, CloudFront)
  api.example.com    → L4 LB → L7 API pool
  upload.example.com → L4 LB → L7 upload pool

Option B: Port-based splitting (if single domain needed)
  example.com:443/static/* → DNS points to CDN
  example.com:443/api/*    → L4 forwards to L7 API LB on port 8080
  example.com:443/upload/* → L4 forwards to L7 upload LB on port 8081
  (L4 can't see URL paths, so this requires the client or DNS to split)

Option C: Separate L4 listeners
  L4 listener on port 443 → forwards to L7 LBs that do URL routing
  CDN configured as origin-pull from static servers
  (Static never hits your L4 LB at all)

Traffic Flow After CDN Offload

Before CDN offload:
  L7 LB handles: 55,000 QPS total
  L7 LB bandwidth: 52 Gbps

After CDN offload:
  CDN handles: 50,000 QPS of static assets (12 Gbps)
  L4 LB handles: 5,500 QPS of API + upload (40.2 Gbps)
  L7 LBs handle: 5,500 QPS split across 2 LBs
    - API L7 LB: 5,000 QPS (25 MB/sec) ← trivial
    - Upload L7 LB: 500 concurrent (5 GB/sec) ← manageable

  L7 capacity needed: 2 Nginx instances (each handles 100K+ req/sec)
  Headroom: 20x for API, plenty for uploads

Best practice: Static assets should never hit your load balancers at all. Use a CDN. Configure DNS so that static.example.com resolves to CDN edge IPs. The CDN handles 50K QPS of thumbnails, JS, and CSS. Your load balancers only see API + upload traffic = 5.5K QPS. That is a 10x reduction.

Sticky Sessions for Uploads

Multi-part video uploads send multiple chunks to the server. Each chunk must go to the same backend server because that server holds the partial upload state (chunks received so far, checksum, metadata). If a chunk goes to a different server, the upload fails.

Upload session lifecycle:
  1. Client initiates upload → server returns upload_session_id = "abc123"
  2. Client sends chunk 1 (256MB) with session_id header → routed to Server A
  3. Client sends chunk 2 (256MB) with session_id header → must go to Server A
  4. ... (8 chunks for a 2GB file)
  5. Client sends "finalize" request → Server A assembles chunks

If chunk 3 goes to Server B:
  Server B: "I don't have chunks 1 and 2. Upload failed."
  Client must restart from scratch. 768MB wasted.

Nginx Sticky Session Config

# L7 upload LB with cookie-based sticky sessions (Nginx)
upstream upload_servers {
    server upload-1:8080;
    server upload-2:8080;

    # Sticky sessions via cookie
    sticky cookie upload_session expires=1h path=/upload;
}

server {
    listen 8081 ssl;

    location /upload/ {
        proxy_pass http://upload_servers;

        # Support large uploads
        client_max_body_size 5g;
        proxy_request_buffering off;    # Stream directly, don't buffer
        proxy_read_timeout 300s;        # 5 min timeout for slow uploads
        proxy_send_timeout 300s;
    }
}

Iteration 2: Resilient Uploads and Smart Routing

Problem: Upload Server Crashes Mid-Upload

Scenario:
  User uploading 5GB video to Server A
  At 3GB (60% complete), Server A crashes

  With simple sticky sessions:
    L7 LB notices Server A is down (health check fails)
    Next chunk goes to Server B
    Server B has no record of this upload
    All 3GB of progress is lost
    User must restart from scratch

  At 10 MB/sec upload speed:
    3GB uploaded = 300 seconds of uploading = 5 minutes wasted
    User is frustrated. May not retry.

Fix: Consistent Hashing + Resumable Uploads

Two-part solution:

1. Resumable upload protocol:
   - Server stores each chunk on shared storage (S3 / NFS)
   - Server keeps metadata in Redis: "upload abc123: chunks 1-12 of 20 received"
   - If server crashes, any other server can read the metadata and resume
   - Client tracks last successful chunk, resumes from there

2. Consistent hashing for upload routing:
   - Hash(upload_session_id) → consistent hash ring → server
   - Even if L7 LB restarts, same session_id maps to same server
   - If a server is removed, only its uploads need to migrate
   - Other servers' uploads are unaffected
# Resumable upload with shared state
def upload_chunk(upload_id, chunk_number, chunk_data):
    # Store chunk on shared storage (any server can access)
    s3.put_object(
        Bucket="uploads-temp",
        Key=f"{upload_id}/chunk_{chunk_number:04d}",
        Body=chunk_data
    )

    # Update metadata in Redis (shared across all servers)
    redis.sadd(f"upload:{upload_id}:chunks", chunk_number)
    redis.expire(f"upload:{upload_id}:chunks", 86400)  # 24hr expiry

    # Check if all chunks received
    received = redis.scard(f"upload:{upload_id}:chunks")
    total = redis.get(f"upload:{upload_id}:total_chunks")

    if received == int(total):
        # Trigger assembly job (can run on any server)
        queue.publish("upload.complete", {"upload_id": upload_id})

    return {"chunk": chunk_number, "received": received, "total": int(total)}

def get_upload_status(upload_id):
    """Client calls this to know where to resume."""
    received = redis.smembers(f"upload:{upload_id}:chunks")
    total = int(redis.get(f"upload:{upload_id}:total_chunks"))
    missing = set(range(total)) - {int(c) for c in received}
    return {"received": len(received), "total": total, "missing": sorted(missing)}

Health Checking at Both Layers

L4 health checks (fast, coarse):
  - TCP connect to backend every 5 seconds
  - If 3 consecutive failures: mark as down
  - If 2 consecutive successes: mark as up
  - Checks: "Is the port open?" (not "Is the app healthy?")

L7 health checks (slower, precise):
  - HTTP GET /health every 10 seconds
  - Expects 200 OK with JSON body: {"status": "healthy", "connections": 45}
  - Checks application-level health:
    - Can it connect to its database?
    - Is disk space available for uploads?
    - Is memory usage below threshold?
  - If unhealthy: stop sending new uploads (but don't kill existing ones!)

Why both layers:
  L4 catches: server crashes, network partitions, port conflicts
  L7 catches: application bugs, disk full, DB connection exhausted, OOM
Health Check Layer Interval Threshold Catches
TCP connect L4 5 sec 3 failures = down Server crash, network issue
HTTP GET /health L7 10 sec 2 failures = drain App errors, resource exhaustion
Active connection drain L7 On failure Wait 5 min, then force Graceful shutdown of uploads

Connection Draining for Uploads

When an upload server needs to go down (deploy, maintenance):

  1. L7 LB marks server as "draining" (accepts no NEW uploads)
  2. Existing uploads continue until complete or timeout
  3. Drain timeout: 5 minutes (longest expected upload segment)
  4. After drain: server is removed from pool
  5. If drain times out: remaining uploads are interrupted
     (client can resume thanks to resumable upload protocol)

  Without draining:
    Server stops immediately → 50 uploads interrupted → 50 angry users

  With draining:
    New uploads go elsewhere → existing 50 uploads complete → 0 interrupted
    Deploy takes 5 min longer but no user impact

Final Design

Capacity Planning

Component Type Capacity Load Headroom
CDN Global edge network Tbps+ 12 Gbps, 50K QPS Effectively unlimited
L4 NLB AWS NLB or DPDK 1M+ conn/sec, 100 Gbps 5.5K QPS, 40 Gbps 180x connections, 2.5x bandwidth
L7 API LB Nginx / Envoy 100K+ req/sec 5K QPS 20x
L7 Upload LB Nginx / Envoy 10K concurrent conn 500 concurrent 20x
API Servers 3 instances ~2K QPS each ~1.7K QPS each 1.2x (add more for safety)
Upload Servers 3 instances ~200 conn each ~167 conn each 1.2x

Cost Comparison

Naive (single L7 for everything):
  1 large L7 LB handling 55K QPS + 52 Gbps
  Must overprovision for peak: ~$2,000/month (large instance)
  Static assets: paying for HTTP parsing of 50K req/sec of thumbnails
  Risk: single point of failure, limited headroom

Optimized (L4 + CDN + L7 split):
  CDN: ~$500/month for 12 Gbps static (usage-based pricing)
  L4 NLB: ~$200/month (AWS NLB pricing, very cheap per connection)
  L7 API LB: ~$300/month (small-medium instance, only 5K QPS)
  L7 Upload LB: ~$300/month (medium instance, long-lived connections)
  Total: ~$1,300/month

  Savings: $700/month, plus:
  - 20x more headroom on API routing
  - CDN eliminates 50K QPS from your infrastructure
  - Each component can be scaled independently
  - No single point of failure

Key Takeaways

  1. L4 for high-throughput, simple routing. L4 load balancers operate at the TCP level, forwarding packets without parsing HTTP. They handle millions of connections per second with minimal overhead. Use L4 as the front door.
  2. L7 for content-based routing, header inspection, and session affinity. L7 can read URLs, cookies, and headers. This enables smart routing (/api/* vs /upload/*), authentication, and sticky sessions. But it costs 10-20x more CPU per request than L4.
  3. Static assets belong on a CDN, not your load balancers. CDNs serve cached content from edge servers close to users. Thumbnails, JS, and CSS should never touch your infrastructure. This removes 90% of QPS from your LB layer.
  4. Use both L4 and L7 together. L4 in front for raw performance and initial traffic splitting. L7 behind for intelligent routing within each traffic type. This is the standard two-tier pattern used by Google, AWS, and most large-scale systems.
  5. Long-lived connections (uploads, WebSockets) need special handling. Sticky sessions ensure multi-part operations go to the same backend. Connection draining ensures deploys do not interrupt active uploads. Resumable protocols ensure crashed servers do not lose progress.
  6. Health checks at both layers. L4 checks "is the port open?" (fast, coarse). L7 checks "is the application healthy?" (slower, precise). Both are needed to catch different classes of failures.
