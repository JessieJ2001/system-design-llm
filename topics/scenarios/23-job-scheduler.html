Design a Distributed Job Scheduler

10 million jobs per day, exactly-once execution, sub-30-second accuracy, and high availability — from a single polling loop to a lease-based, multi-scheduler architecture with Redis priority queues.

NALSD Walkthrough • ~20 min read • LC #621 (Task Scheduler) / min-heap-by-next-fire-time + delay queue

LC kernel. A min-heap keyed by next-fire-time — the same priority-queue at the heart of "Task Scheduler" (LC #621) and "Meeting Rooms II" (LC #253): keep a heap of (next_fire_time, job); pop the minimum, run it; for a recurring job, recompute its next fire and push it back; sleep until the new head's time, or until a new shorter-delay job arrives (a delay queue / timing wheel). On one machine that's O(log n) per fire and "what's due now" is the heap top — O(1) peek. Three properties that break at millions of jobs / thousands of executions/sec: (1) the heap is in RAM and the process is reliable — a real scheduler crashes mid-run, so the heap must be a cache over a durable store, and a job a worker was running when it crashed must be detected (lease expiry) and re-dispatched; (2) one heap, one consumer — millions of pending jobs don't fit one process's heap and thousands of fires/sec exceed one process's throughput, so the timing structure becomes a sharded, time-bucketed durable index ("what's due in the next second" = scan one small bucket, not the whole heap) and many workers pull from it; (3) a function call — "run the job" is fn() in the LC version; in production it's an at-least-once execution with a fencing token handed to an idempotent worker, because exactly-once across a network is a fiction (a worker can finish and die before acking). The C8 production cousin replaces the bare in-RAM min-heap with: a durable job store with an indexed next_fire_time (the heap is a hint), SELECT ... FOR UPDATE SKIP LOCKED (or a Redis sorted-set delay queue) for many-worker pull, a lease + fence-token execution contract, and a standby scheduler so the cron-evaluator + lease-reaper aren't a SPOF. (This is the canonical "task scheduler" scenario; the generic task queue it builds on is #9, and the exactly-once machinery is #14.)

The Scenario

You are building the job scheduling infrastructure for a large SaaS platform. Customers create scheduled tasks that must execute reliably: sending emails at a future time, generating reports on a schedule, running data pipelines every hour, or triggering webhooks at precise moments.

System class

Class: infra primitive / automation pipeline — see functional-requirements.html. It is an internal building block other services depend on (the "cron + delay-queue" the rest of the platform calls), with an automation-pipeline flavour: a stream of time-triggered events flows through it to a worker pool. The class fixes the FR vocabulary — operations (schedule / cancel / pause / update a job; execute via a worker pool) plus semantics (delivery guarantee, fire-time accuracy, fairness) — and puts these NFRs at the centre: scheduling accuracy (fire-time skew), execution-delivery guarantee, availability of the trigger path, and fairness across tenants. It is not a workflow engine (no DAGs / job dependencies — that's Airflow/Temporal-class, out of scope) and not the worker fleet that runs the jobs (that's the task queue, #9).

Functional requirements (cite functional-requirements.html)

  • Schedule a recurring (cron) job. Accept (cron_expr, timezone, priority, payload_ref, tenant); the system fires it on every matching wall-clock instant until cancelled/paused.
  • Schedule a one-shot delayed job. Accept (fire_at, priority, payload_ref, tenant); the system fires it once at fire_at.
  • Cancel / pause / update a job. By job_id — cancel removes it; pause stops it firing without deleting it; update changes the schedule/payload/priority (next fire recomputed).
  • Execute via a worker pool with leases. A worker leases a due job, runs the payload, renews the lease while running, releases (or fails) it; a crashed worker's lease expires and the job is re-dispatched.
  • Deliver each scheduled execution at-least-once with a fencing token. Every fire is handed to a worker with a monotonically-increasing fence_token (the execution_id); the worker presents it on every side-effecting call; downstream rejects a stale token. (Exactly-once-effect = at-least-once delivery + idempotent worker keyed on the token; see #14.)
  • Fairness across tenants. No single tenant's burst of jobs can starve another tenant's due P0/P1 jobs.

Out of scope (named, not silently dropped): the jobs' own logic (the worker code is the caller's; we trigger and lease, we don't define what runs); workflow DAGs / job dependencies ("run B after A completes" — that's a workflow engine, Airflow/Temporal/Cadence-class, a different system); the result store (job output goes wherever the worker writes it; we keep an execution-history audit row, not the payload output); the worker fleet's own autoscaling (the task queue, #9, owns that). Flagged assumptions (would confirm with the interviewer): (1) job count ≈ ~5 M one-shot pending + ~1 M cron schedules; execution rate ~116/sec average, ~1,000/sec peak (batch hour) — if it's 100× this, the single Redis + single Postgres assumption breaks (see Pushbacks); (2) max acceptable fire-time skew is 30 s (so a poll-based design is borderline OK; a 1-s SLA would force an event-driven queue from the start); (3) at-least-once + idempotency is acceptable for most jobs — the rare "must run exactly once" job (a billing run) gets the fence token + a downstream dedupe table (see Pushbacks); (4) max job duration is minutes, not hours — the 60-s lease + 20-s renewal assumes this; an hours-long ETL needs a much longer lease and aggressive renewal (see Assumptions callout); (5) NTP keeps clock drift between schedulers and Postgres under ~1 s — lease expiry is clock-dependent.

Requirements

  • Volume: 10,000,000 jobs per day (~116 jobs/sec average).
  • Job types:
    • One-time delayed jobs: "Run this at 2026-03-21 14:30:00 UTC" (5M/day).
    • Recurring cron jobs: "Run every 15 minutes" or "Run at 9am daily" (1M cron schedules generating 5M executions/day).
  • Priority levels: P0 (critical, run immediately when due), P1 (high), P2 (normal), P3 (low, can be delayed up to 5 minutes).
  • Exactly-once execution: A job must not run twice. Running a job zero times (missing it) is also unacceptable for P0/P1.

  • Availability: 99.9% (8.7 hours downtime per year max).

Back-of-Envelope Math

Storage

Active jobs (pending + running):
  At any moment, ~1 hour of jobs are "active" (pending or running).
  10M jobs/day / 24 hours ≈ 417K jobs/hour
  417K × 500 bytes = 208MB of active job metadata

Total storage (30-day retention):
  10M × 500B × 30 days = 150GB

This fits comfortably on a single PostgreSQL instance with:
  - 500GB SSD storage
  - 32GB RAM (most active job metadata fits in the buffer pool)
  - Read throughput: PostgreSQL handles 10K+ queries/sec easily
  - Write throughput: 116 inserts/sec + 116 status updates/sec = 232 writes/sec
    (well within PostgreSQL's capability of 10K+ writes/sec on SSD)

Throughput

Average: 116 jobs/sec → one scheduler can handle this trivially.

Peak hour (batch processing, start of business):
  Up to 10× average = 1,000 jobs/sec
  Still manageable for a single well-optimized process.

But: a single scheduler is a single point of failure.
  If it crashes, no jobs run until it restarts.
  99.9% availability requires redundancy.

Cron Evaluation

1,000,000 cron schedules must be evaluated periodically.

Every minute, check which cron schedules are due:
  1M cron expressions × ~1μs per evaluation = 1 second
  A single thread handles this in 1 second. No parallelism needed.

Each due cron job generates a "one-time" job instance:
  If average cron fires 5×/day: 5M / (24 × 60) ≈ 3,472 cron triggers/minute
  That's ~58 job insertions/sec from cron alone — trivial.

Naive Design: Single Scheduler with Database Polling

The simplest possible approach: one scheduler process that polls the database for due jobs.

Database Schema

CREATE TABLE jobs (
    job_id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    job_type        VARCHAR(10) NOT NULL,  -- 'one-time' or 'cron'
    scheduled_time  TIMESTAMP NOT NULL,
    priority        SMALLINT NOT NULL DEFAULT 2,  -- 0=critical, 3=low
    payload         JSONB,
    status          VARCHAR(10) NOT NULL DEFAULT 'pending',
    retry_count     SMALLINT DEFAULT 0,
    created_at      TIMESTAMP DEFAULT now(),
    updated_at      TIMESTAMP DEFAULT now()
);

CREATE INDEX idx_jobs_pending ON jobs (scheduled_time, priority)
    WHERE status = 'pending';

Scheduler Loop

import time
import psycopg2

def scheduler_loop():
    while True:
        # Poll for due jobs, ordered by priority then scheduled time
        jobs = db.execute("""
            SELECT * FROM jobs
            WHERE status = 'pending'
              AND scheduled_time <= now()
            ORDER BY priority ASC, scheduled_time ASC
            LIMIT 100
        """)

        for job in jobs:
            # Mark as running
            db.execute("""
                UPDATE jobs SET status = 'running', updated_at = now()
                WHERE job_id = %s AND status = 'pending'
            """, (job.job_id,))

            # Dispatch to a worker
            worker_pool.submit(execute_job, job)

        # Sleep before next poll
        time.sleep(5)  # poll every 5 seconds

What This Gets Right

  • Extremely simple to understand and implement.
  • PostgreSQL handles the storage, indexing, and ordering.
  • At 116 jobs/sec with batch size 100, the scheduler polls ~1.2 times per second. PostgreSQL handles this without breaking a sweat.
  • The partial index on (scheduled_time, priority) WHERE status = 'pending' keeps the poll query fast even with millions of historical rows.

Alternatives considered (C4) — the store

  • A SQL table with an indexed next_fire_time (Postgres) — chosen for the Naive design. The schedule is a durable record that survives a crash (the LC heap doesn't); a B-tree index on next_fire_time WHERE status='pending' makes "what's due now" a range scan, not a full-heap rebuild (cite storage-engines#btree); and Postgres' SELECT ... FOR UPDATE SKIP LOCKED (Iter 1) gives multi-worker pull for free. The cost: poll latency = poll interval, and a poll query even when nothing's due — both fixed in Iter 2 with a Redis delay queue layered over Postgres (Postgres stays the system of record).
  • Redis sorted-set as a delay queue (score = fire-time epoch) — rejected as the only store, adopted as a layer in Iter 2. O(log n) ZADD/ZPOPMIN, ~100 k ops/sec, near-zero dispatch latency (cite storage-engines#kv-engine) — perfect as the timing index, but on its own it's a single point of data loss (a Redis crash drops the queue) and has no audit trail; you want the durable record in Postgres and the fast queue in Redis.
  • A dedicated timer service (Cassandra-backed time-bucket rows) — rejected at this scale, the answer at 100×. Partition the schedule by a time bucket ((bucket_minute, job_id)): "what's due in minute M" = read one partition. Scales horizontally to billions of timers (cite sharding-strategies#range) — this is what Uber's Cherami / a "distributed delay queue" does — but it's overkill for 5 M pending jobs (a single Postgres index handles that), and Cassandra's lack of SKIP LOCKED means you re-implement leasing with lightweight transactions (slower) or per-bucket ownership. Named in Pushbacks as the 100× move.
  • Kafka with delayed delivery — rejected. Kafka has no native "deliver at time T" — you'd bucket into per-delay topics and shuttle messages forward, which is a worse version of the Cassandra-bucket approach with a log's append-only constraints. Kafka is the right tool for the execution stream downstream of the scheduler, not the timing index.

Concrete technology (C10) — the durable store

Postgres 16 for the job store: the working set (~208 MB of active jobs — see BOE) fits the buffer pool; the partial B-tree index on next_fire_time WHERE status='pending' serves the only hot query; SELECT ... FOR UPDATE SKIP LOCKED (Postgres 9.5+) is exactly the leasing primitive; synchronous_commit=on with one sync standby (Iter 3) for RPO = 0 on the schedule. Versus competitors: MySQL 8 — also has SKIP LOCKED (8.0+), would work; Postgres preferred for partial indexes + jsonb payload-ref + richer EXPLAIN; DynamoDB — rejected: no SKIP LOCKED equivalent (you'd poll-and-conditional-update, hot-key risk on the GSI of due jobs), 5–10× unit cost at 300 writes/sec sustained; CockroachDB — rejected: distributed-SQL overhead buys nothing at 150 GB / single-region; revisit only if multi-region. Operability: managed (RDS/Cloud SQL) so failover + PITR are operator-button, not a project; observability via pg_stat_statements (the poll query), pg_stat_activity (lock waits), autovacuum lag on the high-churn jobs table (status updates churn rows — tune autovacuum_vacuum_scale_factor down for this table). Cite storage-engines#btree, replication-decisions#single-sync.

What Breaks (C5 — the Naive design; cite failure-modes.html)

The Naive single-poller hits four failure modes — named against the canonical catalog, with the scenario-specific blast radius inline. Problem 1 is failure-modes#leader-loss (the sole scheduler is the leader, with no failover); Problem 2 is the at-least-once-without-a-fence hazard (cf. #poison-message for the retry side and #14 for the fix); Problem 3 is a throughput/latency cliff (bottlenecks#backlog-growth — due jobs pile up faster than one poller drains them); Problem 4 is a durability gap in the cron-fan-out path.

Problem 1: Single Point of Failure

If the scheduler process crashes, all jobs stop executing until someone manually restarts it. With a 99.9% availability target (8.7 hours downtime per year max), relying on a single process is unacceptable.

Failure scenarios:
  - Process OOM killed: all in-flight jobs lost, queue stalls
  - Host machine reboots: 2-5 minutes to come back online
  - Deployment: even a rolling deploy takes 30-60 seconds of downtime
  - Bug causes crash loop: jobs pile up until manual intervention

Impact: during any outage, jobs accumulate. A 5-minute outage means
  116 jobs/sec × 300 sec = 34,800 jobs are late.
  P0 (critical) jobs like fraud alerts or SLA-bound webhooks are missed.

Problem 2: No Double-Execution Prevention

The naive design has a race condition. Consider this scenario:

1. Scheduler reads job #42 (status = 'pending')
2. Scheduler starts executing job #42
3. Scheduler crashes mid-execution
4. Scheduler restarts
5. Scheduler reads job #42 again — status is still 'pending'
   (the UPDATE to 'running' and the crash happened in sequence,
    but if the crash occurred before the UPDATE committed, the
    job is still 'pending')
6. Scheduler executes job #42 again → DUPLICATE EXECUTION

Even worse: if we had two scheduler instances (for availability),
both could read the same 'pending' job simultaneously:
  Scheduler A: SELECT → gets job #42
  Scheduler B: SELECT → also gets job #42 (before A's UPDATE commits)
  Both execute job #42 → DUPLICATE EXECUTION

Problem 3: Polling Latency

Poll interval: 5 seconds
Worst-case scheduling delay: up to 5 seconds after the job is due.

For a job scheduled at 14:30:00:
  Best case: poll happens at 14:30:00 → delay = 0s
  Worst case: poll happened at 14:29:56, next poll at 14:30:01 → delay = 1s
  Average delay: 2.5 seconds

This is within the 30-second target, but:
  - During a peak spike of 1,000 jobs/sec with batch size 100,
    the scheduler needs 10 polls per second. With 5-second intervals,
    it can only do 0.2 polls/sec → jobs back up.
  - Reducing the interval to 100ms "fixes" this but hammers the DB
    with 10 queries/sec even when no jobs are due.

Problem 4: Cron Jobs Are Mixed with One-Time Jobs

The naive design treats cron jobs by re-inserting the next execution after each run. But if the scheduler crashes between "execute the cron job" and "insert the next execution," that cron schedule is lost forever — it will never fire again.

Iteration 1: Multiple Schedulers with Lease-Based Locking

Bottleneck (cite bottlenecks#coordinator-bottleneck)

The Naive design's shape-determining failure is the single coordinator: one process is the only thing that fires jobs (loss = total outage — a 5-min outage = 116×300 ≈ 34,800 late jobs, including P0/P1 misses, failing the 99.9% target), and it has no safe way to hand a job to itself-after-a-restart without re-running it. Per bottleneck-to-decisions#coordinator-bottleneck, the first move for a single-coordinator bottleneck is: run several, make the shared state arbitrate ownership — here, multiple scheduler instances that compete for due jobs via row-level locks on the Postgres jobs table, each claim time-boxed by a lease so a crashed claimant's jobs are reclaimable. This is the write-heavy-uniform pattern on the lease columns (every claim is an UPDATE; every renewal is an UPDATE). It closes Problems 1 and 2 together — availability (any surviving scheduler keeps firing) and double-execution (the lock serializes the claim; the fence token guards the worker-after-lease-expiry case).

We solve the single-point-of-failure and double-execution problems simultaneously by running multiple scheduler instances that compete for jobs using row-level locking with leases.

Alternatives considered (C4) — ownership & dispatch

  • Leases (time-boxed claims) with a fencing token — chosen. A lease (claimed_by + lease_expires) is self-healing: a crashed scheduler's lease just expires; no liveness gossip needed, no membership protocol. The fence token (execution_id, monotone per job — bumped on every re-claim) covers the gap a bare lease doesn't: a worker that's still running after its scheduler's lease expired and got re-claimed by another scheduler — the worker presents its stale token on the next side-effecting call and is rejected (cite #14 for the fencing-token machinery; we don't re-derive it). This is the standard distributed-lease pattern (ZooKeeper/etcd leases, Chubby, DynamoDB lock client).
  • Heartbeats (workers ping a registry; a reaper notices a missed heartbeat) — rejected as the primary mechanism. Equivalent in spirit but needs a separate liveness store and a tunable "how many missed beats = dead" — more moving parts than "the lease column has a timestamp; if it's in the past, it's free." We do use the same idea internally as the lease-renewal loop (a renewal that fails ≈ a missed heartbeat), but the source of truth is the lease timestamp in Postgres, queried with WHERE lease_expires < now().
  • Push dispatch (a dispatcher hands jobs to specific workers) — rejected. A dispatcher that assigns due jobs to chosen workers is itself a single coordinator (the thing we're trying to eliminate) and needs to track worker liveness/load. Pull (workers/schedulers poll for due jobs and claim what they find) — chosen — needs no central assignment: SKIP LOCKED is the arbiter, and a slow/dead consumer simply doesn't claim anything (cite #3 push-vs-pull). The trade is poll latency, which Iter 2 erases with the Redis delay queue while keeping pull.
  • Cron expansion: materialize-the-next-N-fires vs compute-on-demand — deferred to Iter 2/3. Two ways to turn a cron schedule into concrete fires: (a) compute the next fire on demand each tick (one row per cron_schedules entry; a cron-evaluator loop computes croniter(expr, last_fired).next()); (b) materialize the next N fires as concrete jobs rows ahead of time (the schedule becomes N pending one-shots; refilled in batches). (a) is less storage and simpler; (b) makes a cron job indistinguishable from a one-shot to the dispatch path (uniform handling) and survives a long cron-evaluator outage (the next N fires are already in the queue). We use (a) with a leader-elected evaluator (Iter 3); a 100×-scale variant would switch to (b) to take the evaluator off the critical path. (Constraint-flip: 🔀 if the finest cron granularity were sub-minute or the schedules were millions of "every 5 seconds" jobs, (b) materialization + a time-bucketed index becomes mandatory — the per-tick evaluator can't keep up.)

The Lease Mechanism

A "lease" is a time-limited claim on a job. A scheduler claims a job by writing its own ID and a lease expiration time into the job row. If the scheduler crashes, the lease eventually expires, and another scheduler can reclaim the job.

-- Updated schema with lease fields
ALTER TABLE jobs ADD COLUMN claimed_by     VARCHAR(64);
ALTER TABLE jobs ADD COLUMN lease_expires   TIMESTAMP;
ALTER TABLE jobs ADD COLUMN execution_id    UUID;
import uuid

LEASE_DURATION = 60  # seconds
SCHEDULER_ID = f"scheduler-{uuid.uuid4().hex[:8]}"

def claim_jobs(batch_size=100):
    """Atomically claim a batch of due jobs using UPDATE ... RETURNING"""
    execution_batch_id = uuid.uuid4()
    rows = db.execute("""
        UPDATE jobs
        SET status = 'running',
            claimed_by = %s,
            lease_expires = now() + interval '%s seconds',
            execution_id = %s,
            updated_at = now()
        WHERE job_id IN (
            SELECT job_id FROM jobs
            WHERE status = 'pending'
              AND scheduled_time <= now()
            ORDER BY priority ASC, scheduled_time ASC
            LIMIT %s
            FOR UPDATE SKIP LOCKED  -- critical: skip rows locked by others
        )
        RETURNING *
    """, (SCHEDULER_ID, LEASE_DURATION, execution_batch_id, batch_size))
    return rows

Why FOR UPDATE SKIP LOCKED Is Critical

The FOR UPDATE SKIP LOCKED clause is the key to making multiple schedulers work without conflicts:

  • FOR UPDATE locks the selected rows, preventing other transactions from modifying them.
  • SKIP LOCKED tells PostgreSQL: "if a row is already locked by another transaction, do not wait for it — skip it and return the next available row."
  • Result: Scheduler A and Scheduler B can run the same query simultaneously. A gets rows 1-100, B gets rows 101-200. No overlap, no waiting, no deadlocks.

Exactly-Once Execution with Execution IDs

The lease prevents two schedulers from simultaneously claiming the same job. But what about this scenario?

1. Scheduler A claims job #42, execution_id = "abc-123"
2. Scheduler A dispatches job #42 to Worker W1
3. Scheduler A's lease renewal fails (network partition)
4. Lease expires after 60 seconds
5. Scheduler B reclaims job #42, execution_id = "def-456"
6. Scheduler B dispatches job #42 to Worker W2
7. Now W1 and W2 are both running job #42!

Solution: Workers check the execution_id before performing side effects.

Worker W1 starts, checks: is execution_id still "abc-123"?
  → If yes, proceed (it still owns the job)
  → If no (it's now "def-456"), abort immediately

This is called a "fencing token" — the execution_id acts as proof
of ownership. Any operation that modifies external state (sending an
email, writing to a database, calling an API) must first verify the
execution_id matches.

def execute_job_with_lease(job):
    """Execute a job while periodically renewing its lease"""
    # Start a background thread to renew the lease every 20 seconds
    lease_renewer = LeaseRenewer(job.job_id, SCHEDULER_ID, interval=20)
    lease_renewer.start()

        result = worker_pool.submit(job.payload)
        result.get(timeout=LEASE_DURATION - 10)  # timeout before lease expires

        # Mark as completed
        db.execute("""
            UPDATE jobs SET status = 'completed', updated_at = now()
            WHERE job_id = %s AND claimed_by = %s
        """, (job.job_id, SCHEDULER_ID))

    except Exception as e:
        # Mark as failed, allow retry
        db.execute("""
            UPDATE jobs
            SET status = CASE WHEN retry_count < 3 THEN 'pending' ELSE 'failed' END,
                claimed_by = NULL,
                lease_expires = NULL,
                retry_count = retry_count + 1,
                updated_at = now()
            WHERE job_id = %s AND claimed_by = %s
        """, (job.job_id, SCHEDULER_ID))

    finally:
        lease_renewer.stop()

Architecture with 3 Schedulers

What breaks (C5; cite failure-modes.html)

  • Lease-expiry storm — THE one. If a scheduler dies (or GCs / network-blips) holding many leases, all of them expire at roughly the same instant, and every surviving scheduler's reclaim loop sees a flood of status='running' AND lease_expires < now() rows and re-dispatches them in a thundering wave — a spike of duplicate-ish executions (the workers from the dead scheduler may still be alive) and a load spike on the worker fleet. Defence: stagger lease durations (jitter the 60 s by ±10 s) so they don't all expire together; cap the reclaim batch size per loop (re-dispatch at most N/sec, drain the backlog); the fence token makes the duplicate executions safe (the dead scheduler's workers present a stale token). Blast radius: one scheduler's worth of in-flight jobs (~batch×workers ≈ a few hundred), delayed by up to one lease duration; detection: a spike in reclaim_count + fence_token_rejected.
  • Poison job — a job whose payload crashes its worker every time. Worker dies → lease expires → re-dispatched → worker dies → ... a crash-loop that consumes a worker slot forever and never makes progress (and at-least-once means it keeps getting retried). Defence: the retry_count cap (3, already in the schema) routes it to status='failed' — a dead-letter state for human review; exponential backoff between retries so a poison job doesn't hot-loop; alert on the failed rate. Blast radius: one worker slot until the retry cap trips (~3 attempts); detection: retry_count≥2 gauge + the failed-state rate.
  • Clock skew between a scheduler and Postgres. Lease expiry compares a scheduler-written lease_expires against Postgres' now() — if a scheduler's clock is fast, its leases look expired early (jobs reclaimed while still running → duplicates); slow, and crashed-scheduler jobs sit stuck longer. Defence: write lease_expires = now() + interval '60 s' using the database's clock (never the scheduler's), so all comparisons are against a single clock; NTP on all hosts with drift alarms <1 s as defence-in-depth. Blast radius: bounded by the drift (seconds of early/late reclaim); detection: NTP offset metric per host.
  • SKIP-LOCKED contention at higher scale. With many schedulers polling the same hot range of due rows, the FOR UPDATE SKIP LOCKED scan has to skip more and more locked rows before finding a free one — wasted index scanning; at extreme scale the partial index on due rows becomes a hot page. Defence (now): keep the scheduler count small (3) and batch size modest (100), so the hot range is small; (at 100×): shard the due-jobs index by a hash of job_id mod K so scheduler i only scans shard i (cite bottleneck-to-decisions#lock-contention, sharding-strategies#hash) — or move the timing index to Redis (Iter 2) where ZPOPMIN is atomic and lock-free. Blast radius: poll latency creep; detection: poll-query p99 + index-scan rows-examined.

Iteration 2: Replace Polling with a Redis Priority Queue

Bottleneck (cite bottlenecks#lock-contention + access-patterns#bursty)

Iter 1's shape-determining failure is the poll loop: scheduling latency = the poll interval (2.5 s average at a 5-s interval — OK against the 30-s SLA but wasteful), and during the batch-hour burst of ~1,000 jobs/sec the scheduler needs ~10 polls/sec of batch-100 just to keep up — at a 5-s interval it does 0.2 polls/sec, so due jobs back up (the bursty access pattern the DB poll can't track). Shrinking the interval to 100 ms "fixes" the burst but hammers Postgres with empty polls when idle and worsens the SKIP LOCKED contention from the previous What-Breaks. Per bottleneck-to-decisions#lock-contention the move when a hot-row poll is the bottleneck is: replace the periodic scan with an event-driven, lock-free dequeue — a Redis sorted-set delay queue keyed by fire-time, layered over Postgres (Postgres stays the durable record + audit; Redis is the timing index, rebuildable from Postgres). ZPOPMIN is atomic, so no FOR UPDATE; the dequeue loop is continuous, so latency is near-zero; idle = a cheap BZPOPMIN block, not a query.

Database polling works but has two drawbacks: (1) it adds latency equal to the poll interval, and (2) it generates unnecessary database load when no jobs are due. A Redis sorted set gives us an event-driven alternative.

Redis Sorted Set as a Scheduling Queue

A Redis sorted set stores members with associated scores. Members are sorted by score. We use the scheduled timestamp (as a Unix epoch) for the score. The member with the lowest score is the next job to execute.

# Enqueue a job: score = scheduled_time as Unix timestamp
def enqueue_job(job_id, scheduled_time, priority):
    # Encode priority into the score: multiply timestamp by 10, add priority
    # This ensures P0 jobs at the same time are dequeued before P3 jobs.
    score = scheduled_time.timestamp() * 10 + priority
    redis.zadd("job_queue", {job_id: score})

# Dequeue the next due job (atomic pop)
def dequeue_next_job():
    now_score = time.time() * 10 + 9  # 9 = higher than any priority
    # ZPOPMIN atomically removes and returns the member with lowest score
    result = redis.zpopmin("job_queue")
    if result:
        job_id, score = result[0]
        if score <= now_score:
            return job_id
        else:
            # Job is not due yet — put it back
            redis.zadd("job_queue", {job_id: score})
            return None
    return None

The Complete Dequeue Loop

def scheduler_loop():
    """Continuously dequeue and execute due jobs"""
    while True:
        # Atomically pop the job with the lowest score (earliest due time)
        results = redis.zrangebyscore(
            "job_queue",
            min=0,
            max=time.time() * 10 + 9,  # all jobs due now or earlier
            start=0,
            num=1,          # just one at a time
            withscores=True
        )

        if not results:
            time.sleep(0.1)  # no jobs due — sleep 100ms to avoid busy-wait
            continue

        job_id, score = results[0]

        # Atomically remove it (only if it's still there — another scheduler
        # may have grabbed it between our ZRANGEBYSCORE and ZREM)
        removed = redis.zrem("job_queue", job_id)
        if removed == 0:
            continue  # Another scheduler got it first — try again

        # We now own this job. Fetch metadata from PostgreSQL.
        job = db.execute("SELECT * FROM jobs WHERE job_id = %s", (job_id,))

        # Claim with lease in PostgreSQL (for crash recovery)
        claimed = db.execute("""
            UPDATE jobs
            SET status = 'running',
                claimed_by = %s,
                lease_expires = now() + interval '60 seconds',
                execution_id = gen_random_uuid()
            WHERE job_id = %s AND status = 'pending'
            RETURNING execution_id
        """, (SCHEDULER_ID, job_id))

        if not claimed:
            continue  # Job was already claimed or cancelled

        # Execute the job
        execute_job_with_lease(job, claimed.execution_id)

Handling Cron Jobs Separately

Cron jobs need a dedicated evaluation process that runs independently of the job dispatch loop.

def cron_evaluator():
    """Runs every 60 seconds. Evaluates all cron schedules and enqueues due jobs."""
    while True:
        start = time.time()

        # Fetch all active cron schedules
        cron_schedules = db.execute("""
            SELECT cron_id, cron_expression, last_fired_at
            FROM cron_schedules
            WHERE active = true
        """)

        enqueued = 0
        for schedule in cron_schedules:
            next_fire = croniter(schedule.cron_expression,
                                 schedule.last_fired_at).get_next()
            if next_fire <= time.time():
                # Create a one-time job instance for this cron execution
                job_id = uuid.uuid4()
                db.execute("""
                    INSERT INTO jobs (job_id, job_type, scheduled_time, priority,
                                     payload, status)
                    VALUES (%s, 'cron', %s, %s, %s, 'pending')
                """, (job_id, next_fire, schedule.priority, schedule.payload))

                # Enqueue in Redis
                enqueue_job(job_id, next_fire, schedule.priority)

                # Update last_fired_at to prevent re-evaluation
                db.execute("""
                    UPDATE cron_schedules SET last_fired_at = %s
                    WHERE cron_id = %s
                """, (next_fire, schedule.cron_id))

                enqueued += 1

        elapsed = time.time() - start
        print(f"Cron evaluation: {len(cron_schedules)} schedules, "
              f"{enqueued} enqueued, {elapsed:.2f}s")

        # Sleep until the next minute boundary
        sleep_time = 60 - (time.time() % 60)
        time.sleep(sleep_time)

Concrete technology (C10) — the delay queue

Redis 7 (single instance + Redis Sentinel for failover) for the timing index: the sorted-set ops (ZADD, ZRANGEBYSCORE, BZPOPMIN) are O(log n) and the instance does ~100 k ops/sec — ~290 ops/sec here is <0.3% of headroom; persistence: AOF with everysec fsync (a crash loses ≤1 s of ZADDs, which Postgres re-supplies on recovery — the sorted set is a cache); maxmemory with noeviction (never evict a pending job — a full queue is a hard failure to alarm on, not silently drop). Key shape: a single sorted set sched:queue with member = job_id, score = fire_at_epoch × 10 + priority (priority packed into the low digit so P0 wins ties at the same instant); plus per-tenant counters for the fairness controller (Iter 3-area). Versus competitors: RabbitMQ delayed-message plugin — rejected: it holds delayed messages in an Erlang timer that doesn't survive a broker restart cleanly and doesn't give you "peek the next 1,000 due" cheaply; Memcached — no sorted set, no atomic pop, dead on arrival; a dedicated timer service (Cassandra time-buckets) — the 100× answer, overkill here (see Naive's C4). (Why not just Postgres' LISTEN/NOTIFY to push due jobs instead of Redis? It works for low rates but NOTIFY payloads are tiny, there's no priority ordering, and a slow listener drops notifications — the sorted set is the durable-enough, ordered, peekable structure we want.) Operability: managed (ElastiCache/MemoryStore) so Sentinel/failover is operator-button; observability via Redis SLOWLOG, keyspace notifications (for queue-depth alarms), INFO memory/ops; known hazard — the AOF rewrite fork can OOM on a memory-tight box, so size for 2× the dataset. Cite storage-engines#kv-engine, redis, and the SoT message-queues.

What breaks (C5; cite failure-modes.html)

  • Redis lost — the timing index evaporates. If Redis crashes and the AOF tail is behind, the sorted set is gone — due jobs stop firing until it's rebuilt. Defence: Sentinel fails over to a replica (1–2 s); on promotion (or if the replica is stale too) the scheduler rebuilds the sorted set from Postgres — SELECT job_id, scheduled_time, priority FROM jobs WHERE status='pending' → bulk ZADD (5 M rows ≈ a few seconds of ZADD pipelining); during the rebuild, the Iter-1 DB-poll path is the fallback (slower, but jobs still fire). Postgres is the source of truth precisely so this is a rebuild, not a data loss. Blast radius: a few seconds of "scheduling via the slow path"; detection: Redis-up health check + rebuild-in-progress flag.
  • Midnight thundering herd — 1 M cron jobs all * 0 * * *. Tenants overwhelmingly schedule "daily at midnight" / "hourly on the hour," so at 00:00:00 the cron evaluator enqueues a huge batch and every scheduler's ZRANGEBYSCORE 0 now returns thousands at once — a spike that swamps the worker fleet and Postgres' status-update rate (cite bottlenecks#stampede and the herd cousins #26 / #27). Defence: smear — for low-priority (P3, "delayed up to 5 min") cron jobs, when materializing the fire, add a deterministic jitter hash(job_id) mod 300 seconds, so 1 M midnight jobs spread over a 5-min window (~3,300/sec instead of a 1 M instantaneous spike); P0/P1 fire on time (they're a small fraction). Plus the worker pool's own backpressure (a bounded queue; shed P3 first — cite #13). Blast radius without the smear: a multi-minute backlog + a Postgres write spike at every round hour; detection: ZCARD sched:queue spike + worker-queue-depth gauge.
  • Lost-update race between ZRANGEBYSCORE and ZREM. Two schedulers both see job_id=j at the head; both try to claim it — the loser of the ZREM (returns 0) just moves on, and the loser of the Postgres UPDATE ... WHERE status='pending' (returns 0 rows) also moves on. Safe by construction (both checks are atomic), but if a scheduler crashes between the successful ZREM and the Postgres claim, the job is out of Redis but still pending in Postgres — orphaned. Defence: the lease-reaper's scan is widened to status='pending' AND scheduled_time < now() - interval '5 min' (a pending job whose fire-time has long passed and that isn't in Redis — re-ZADD it); this is the same reaper, one more clause. Blast radius: a handful of jobs delayed by the reaper interval; detection: count of "re-enqueued orphaned pending" per reaper run.
🔀 Constraint flip — if the fire-time SLA were 1 s instead of 30 s
  • The Iter-1 DB-poll path stops being a viable fallback (a 5-s poll blows a 1-s SLA) — Redis becomes load-bearing, so it needs a hot-standby that's caught up enough to fail over within the budget, and the rebuild-from-Postgres path needs to be fast enough or pre-warmed.
  • The midnight-smear window for P3 jobs shrinks (or P3-with-a-5-min-grace stops existing as a tier) — less herd-smoothing headroom.
  • Clock-skew tolerance tightens: NTP drift now eats a meaningful fraction of the 1-s budget.

Iteration 3: Leader Election, Fairness, and Partition Handling

Bottleneck (cite bottlenecks#hot-partition + failure-modes#leader-loss)

Iter 2 has good dispatch availability (any scheduler pops from Redis) but two residual problems. (1) Singleton operations are running N times. The cron evaluator and the lease-reaper aren't idempotent-by-construction: 3 schedulers each evaluating 1 M cron schedules = 3× the work and 3 duplicate jobs rows per firing (the last_fired_at guard catches most, but it's a race); 3 reapers fighting over the same expired rows. One scheduler must own those — the standard answer is leader election (cite bottleneck-to-decisions#coordinator-bottleneck). (2) A noisy tenant starves others — the FR "fairness across tenants" isn't met: one tenant scheduling a million 1-second jobs floods the Redis queue head; ZPOPMIN hands those to every worker, and another tenant's due P0 sits behind them — a hot-partition on the shared queue. The move: a leader-elected cron-evaluator + lease-reaper (etcd lease) and a per-tenant fairness gate on dequeue (weighted fair queuing across tenants).

With 3 schedulers all competing for jobs via ZPOPMIN, we have good availability. But some operations should only be performed by one scheduler at a time to avoid conflicts: specifically, the cron evaluation loop and the lease expiry reclamation loop. And no single tenant should be able to monopolize the worker fleet.

Why Leader Election for Cron Evaluation

If all 3 schedulers run the cron evaluator simultaneously, each cron schedule is evaluated 3 times. This causes 3 duplicate job insertions for every cron firing. We need exactly one scheduler to be the "cron leader."

Alternatives considered (C4) — leader election & fairness

  • etcd lease (or ZooKeeper ephemeral node) for leadership — chosen. A short-TTL lease + a watch gives "exactly one leader, fast failover" with a battle-tested consensus core; the leader's TTL (15 s) bounds how long the cron evaluator pauses on a leader crash. Versus a Postgres advisory lock (pg_try_advisory_lock(key)) — tempting (no new infra; the lock auto-releases when the session ends) and a fine choice if you want to avoid running etcd, but failover detection is "the TCP session dropped," which is murkier under a network partition than a lease TTL, and it couples leadership to a single Postgres connection's liveness. Versus Redlock (the Redis-based lock) — rejected: Redlock's safety under partitions is contested, and we already need a fence token (execution_id) for the data path, so we don't want a second, weaker lock for the control path. We'd accept the Postgres-advisory-lock variant if "no etcd" were a hard constraint; etcd is the cleaner answer when it's available.
  • Per-tenant weighted fair queuing on dequeue — chosen for fairness. Instead of one global ZPOPMIN, the dispatch loop round-robins (weighted by tenant tier) across per-tenant due-sets: each tenant gets its own sorted set sched:queue:{tenant} (or a single set tagged by tenant with a per-tenant in-flight counter), and the scheduler dequeues at most cap(tenant) in-flight per tenant before moving on — a flooding tenant fills its own quota and waits, others flow. Versus per-tenant rate limiting at submit time — complementary, not a substitute (cite #20): a token-bucket on POST /v1/jobs per tenant stops the creation flood; the dequeue gate handles a tenant that legitimately has a million already-scheduled jobs all coming due at once. Versus priority-only (no tenant dimension) — rejected: priority is across-tenant; a tenant could schedule a million P0 jobs and still starve everyone. You need both axes: priority within a tenant, fair share across tenants.

Using etcd for Leader Election

import etcd3

etcd_client = etcd3.client()

def run_as_leader(task_name, task_function):
    """Run task_function only while this instance holds the leader lease"""
    lease = etcd_client.lease(ttl=15)  # 15-second lease

    # Try to become leader by creating a key with our lease
    # put_if_absent: only succeeds if the key doesn't exist
    success, _ = etcd_client.transaction(
        compare=[etcd_client.transactions.create(f"/leader/{task_name}") == 0],
        success=[etcd_client.transactions.put(
            f"/leader/{task_name}",
            SCHEDULER_ID,
            lease=lease
        )],
        failure=[]
    )

    if success:
        print(f"Became leader for {task_name}")
        # Renew lease in background (keeps leadership)
        lease_keeper = lease.keepalive()
        try:
            task_function()  # Run the task
        finally:
            lease.revoke()   # Release leadership on shutdown
    else:
        print(f"Another scheduler is leader for {task_name}. Standing by.")
        # Watch for leader key deletion (leader crashed)
        events_iterator, cancel = etcd_client.watch(f"/leader/{task_name}")
        for event in events_iterator:
            if isinstance(event, etcd3.events.DeleteEvent):
                print("Leader lost. Attempting to become leader...")
                cancel()
                run_as_leader(task_name, task_function)  # retry

What breaks (C5; cite failure-modes.html)

  • Leader loss / split-brain on the cron evaluator. A leader that's network-partitioned-but-alive still thinks it's leader until its etcd lease expires; a new leader is elected in the majority partition; for a window both may run the evaluator. Defence: the last_fired_at guard per cron schedule (an UPDATE ... WHERE last_fired_at < next_fire — the loser of the race no-ops); the evaluator does nothing irreversible (it inserts pending jobs; a dup insert is caught by the same guard or de-duped by an idempotency key on the materialized fire). Blast radius: at most one cron tick's worth of duplicate pending rows for the window (~15 s of lease TTL), all guarded; detection: the etcd leader-change event + a "cron eval ran twice in one minute" alarm. (This is the canonical leader-loss mode — the fix is "the singleton operation must be idempotent anyway"; the lease just makes the overlap rare.)
  • Bad deploy of the scheduler — a crash-looping new version. A rolling deploy of a broken scheduler binary takes all schedulers down one by one; the leader role flaps; due jobs stop firing. Defence: deploy one scheduler at a time, health-gated (it must successfully claim+complete a canary job before the next instance rolls); keep N−1 on the old version until the new one is proven; instant rollback (it's stateless — the durable state is in Postgres + Redis + etcd). Blast radius without the gate: a full scheduling outage until rollback (minutes); with it: one instance's worth of capacity during the canary. Detection: the deploy pipeline's canary-job check + the standing "0 schedulers firing" alarm.
  • Fairness-gate evasion — a tenant spreads a flood across many tenant IDs. If "tenant" is the only fairness key, a tenant that controls many sub-accounts can multiply its fair share. Defence: the fairness key is the billing account (the unit you actually want to be fair across), not the API-key; a per-account submit-rate limit (cite #20) on top; alert on a single account's queue-depth share > X%. Blast radius: temporary unfairness until the rate limit/quota kicks in; detection: per-account queue-depth-share gauge.
  • Postgres↔Redis disagreement under partition. If a scheduler can reach Redis but not Postgres, it can ZREM a job but not write the claim — the orphan case (covered by the widened reaper, above). If it can reach Postgres but not Redis, it can fall back to the Iter-1 DB-poll path. Defence: the fallback hierarchy is explicit (Redis dequeue → DB poll → if neither, this scheduler is useless and should be shed by the LB / restart); the orphan-reaper closes the gap. End-to-End Flow: One-Time Job

1. API server receives: "Send welcome email to user-42 at 14:30:00"
2. API writes to PostgreSQL:
     INSERT INTO jobs (job_id='j-001', scheduled_time='14:30:00',
                       priority=2, payload='{"email": "welcome", "user": 42}',
                       status='pending')
3. API enqueues in Redis:
     ZADD job_queue 17110980002 j-001  (score = 1711098000 × 10 + 2)

4. At 14:30:00, Scheduler B's dequeue loop runs:
     ZRANGEBYSCORE job_queue 0  LIMIT 0 1
     → returns ("j-001", 17110980002)
     ZREM job_queue j-001  → returns 1 (success, we own it)

     UPDATE jobs SET status='running', claimed_by='scheduler-B',
            lease_expires=now()+60s, execution_id='exec-abc'
     WHERE job_id='j-001' AND status='pending'

6. Worker thread sends the email:
     - Check: is execution_id still 'exec-abc'? Yes → proceed.
     - Call email API: send_welcome_email(user=42)
     - Email sent successfully.

7. Scheduler B marks complete:
     UPDATE jobs SET status='completed' WHERE job_id='j-001'

Degradation contract

When a scheduler dies → its leases expire (jittered, ~60 s); a surviving scheduler's reaper re-dispatches its in-flight jobs (rate-capped to avoid a lease-expiry storm); dispatch continues uninterrupted on the survivors; if it held a singleton role (cron-evaluator / reaper), etcd elects a new leader in ~15 s. When Redis is lost → Sentinel fails over (1–2 s); if the replica is stale, the schedulers rebuild sched:queue from Postgres (a few seconds) and run on the Iter-1 DB-poll fallback meanwhile — jobs still fire, just on the slower path. When Postgres is lost → failover to the sync standby (~10 s); new submissions fail (the API returns 503, the client retries) but jobs already in Redis keep firing; on recovery the orphan-reaper reconciles. When the cron evaluator overlaps (leader split-brain) → the last_fired_at guard makes the dup a no-op. When the worker fleet backs up (more due jobs than workers can drain) → the scheduler sheds/delays P3 first (with a reason — never silent), keeps P0/P1 flowing; for P3 cron jobs the midnight-smear already spread the herd. When a tenant floods → the per-account dequeue cap + submit rate-limit confine it to its fair share; others flow. When a job is poison (crashes its worker every time) → the retry_count cap (3, with backoff) routes it to failed for human review. The system never silently drops a scheduled job (a full Redis = alarm, not eviction; the durable record is in Postgres), never fires a P0/P1 zero times within the 30-s SLA absent a multi-component outage, and never double-executes an effect for a fence-aware (billing-class) job. Cite failure-modes#lease-expiry-storm + #leader-loss + #backlog-overflow + #poison-message.

Operations (C6)

  • Metrics (SLIs → NFRs). Fire-time skew — the headline: actual_fire_time − scheduled_time, p50/p99, alarm if p99 > 30 s; broken down by priority (P0/P1 separately — the SLA tiers). Late-job count — jobs whose scheduled_time has passed and that aren't running/completed (the backlog gauge); split by priority. Lease health — reclaim_count/min (a spike = a scheduler died), fence_token_rejected/min (workers running after re-claim — expected during a reclaim, alarming otherwise), reaper-batch size. Queue depth — ZCARD sched:queue (the herd gauge), per-tenant share of it (the fairness gauge), idx_jobs_due rows-examined per poll (the SKIP LOCKED-contention gauge). Cron evaluator — eval duration (should be ~1 s), "ran twice in a minute" alarm (leader split-brain), enqueued-per-tick. Dead letter — failed-state rate, retry_count≥2 gauge (poison jobs). Availability — "0 schedulers firing in the last N s" (the standing page), leader-change rate, Redis/Postgres failover events. Worker feed — worker-queue depth + shed-by-priority rate.
  • Rollout. Schedulers are stateless — deploy one instance at a time, health-gated: the new instance must successfully claim+complete a synthetic canary job before the next rolls; keep N−1 on the old version until the new one is proven; instant rollback (the durable state is external). Schema migrations on jobs are additive-then-backfill-then-switch (add the nullable column, deploy code that writes it, backfill, add the index CONCURRENTLY, then make it NOT NULL) so a high-churn table is never locked. Redis/etcd are managed — failover is operator-button. Rollback triggers: a fire-time-skew p99 regression, a fence_token_rejected spike not explained by a reclaim, a rise in the failed rate, or a canary-job failure.
  • Migration (single-cron-box → distributed scheduler). Most teams start with one box running cron (or a Quartz/APScheduler process). Phase 1: stand up the distributed scheduler in shadow — it reads the same schedule definitions, computes what it would fire, logs it, but doesn't dispatch; diff against the old box's actual fires for a week (chase down timezone/DST edge cases — the usual culprit). Phase 2: cut over per-job-class — move the low-risk, idempotent jobs (report generation, cache warming) to the new system first; watch a week; then the rest; the old box keeps running the not-yet-migrated jobs. Phase 3: the old box runs nothing — decommission it, but keep its config in version control as a fallback for a month. The risk is entirely Phase 1's diff (and the fact that cron "fires once per matching minute on this host's clock" while the distributed scheduler fires "once globally" — double-firing during overlap is the thing the fence token + idempotent workers exist to make safe). Cite api-design-patterns#idempotency.
  • Headroom. +50% on scheduler instances (a rolling deploy + a host failure must not drop firing capacity below burst-hour demand — ~1,000 jobs/sec); Redis sized for 2× the dataset (AOF-rewrite fork) and ~10× the burst ops/sec; Postgres' jobs-table autovacuum tuned aggressively (status updates churn rows) and sized for ~10× the write rate; etcd at 5 nodes across AZs (lease + leader-key write rate ×3). Reason: a deploy during a host failure during the batch hour is a Tuesday; the scheduler must not be the thing that falls over when the fleet most needs jobs fired.

Anticipated pushbacks

Stress tests against the derivation above — depth rehearsal, not interview rehearsal. If a question can't be answered in 3–6 sentences from the text above, the page is missing a probe.

What if there were 100× the jobs — 500 M one-shot pending, 100 M cron schedules, ~100 k executions/sec? (scale flip)

Three things move. (1) The due-jobs index is now a hot partition — shard the jobs table by job_id mod K (K = 16+), give scheduler i shard i, so the SKIP LOCKED scan and the partial index aren't shared (cite sharding-strategies#hash); the Redis sorted set splits the same way (sched:queue:{shard}) — or you graduate to a dedicated timer service: a Cassandra-backed time-bucketed index ((bucket_minute, job_id)), "what's due this minute" = read one partition, scales to billions of timers (cite sharding-strategies#range — the Naive C4 named this). (2) The cron evaluator can't scan 100 M schedules every tick from one leader — switch to materialize-the-next-N-fires (the Iter-1 C4's option (b)): a cron schedule becomes N pending one-shots, refilled in batches by a sharded, parallel refiller — the per-tick critical path goes away. (3) Postgres at ~100 k writes/sec needs sharding anyway, or you move the system-of-record to Cassandra/ScyllaDB and keep Postgres only for the small cron_schedules definitions. The bottleneck shifts from "coordinator" (long since solved with leases + cells) to "the timing-index partition" — which is a hot-partition problem, solved by sharding the time/key dimension. The +50% scheduler headroom is unaffected; what grows is the store fleet.

Why at-least-once + idempotency rather than true exactly-once? (semantics)

Because exactly-once delivery across a network is provably impossible — the worker can finish the side effect and die before acking, and the scheduler has no way to tell "done but un-acked" from "never ran" (the FLP / two-generals problem). So you choose: at-least-once delivery (a fire may be handed to a worker twice) + an idempotent worker (keyed on the fence token, the second run is a no-op) = exactly-once effect, which is what anyone actually wants. The fence token (execution_id, monotone per job) is the linchpin: every side-effecting downstream call carries it; for ordinary jobs (send an email, warm a cache) re-doing the call is harmless; for the rare job where it isn't, the downstream keeps a (job_id → max fence_token) row and rejects a stale token — a tiny dedupe table, not a distributed transaction. Cite #14 for the full machinery; this page leans on it rather than re-deriving it. The alternative — a 2-phase commit between the scheduler and every worker on every fire — is slower, fragile (a stuck participant blocks), and unnecessary for 99% of jobs.

What changes if a job MUST run exactly once — it's a billing run that charges customers? (semantics / regulatory-ish)

The scheduling side is unchanged — you still get at-least-once delivery with a fence token. What changes is the worker contract: the billing worker must be idempotent on the fence token, and the thing it talks to (the ledger / payment processor) must dedupe on an idempotency key derived from (job_id, execution_id) — a charge with a seen key is a no-op that returns the original result (cite api-design-patterns#idempotency, #14). Concretely: the billing worker, before charging, writes a (idem_key, status=pending) row in its own DB in the same transaction as reading "what to charge"; charges; updates to done with the result; a re-delivery finds the row and returns the recorded result instead of charging again. The scheduler also bumps this job's priority to P0 (it must fire within the SLA) and might mark it no_smear (don't jitter its fire-time). The cost: the billing worker is more complex (a dedupe table, a careful transaction) — but that complexity lives in the one worker that needs it, not in the scheduler, which stays a dumb at-least-once trigger.

The scheduler dies at exactly 00:00:00 — right when the midnight cron herd is due. What happens? (failure)

Two cases. (1) A non-leader scheduler dies — the other two keep dequeuing from Redis; its in-flight jobs' leases expire (~60 s, jittered) and the reaper (running on the leader) re-dispatches them, rate-capped; net effect: ~one scheduler's worth of in-flight jobs delayed up to a minute. (2) The leader dies — on top of the above, the cron-evaluator stops; etcd elects a new leader in ~15 s; the new leader's first eval picks up where last_fired_at left off, so the midnight herd is enqueued ~15 s late — within the 30-s SLA for P0/P1, and the P3 herd was smeared over 5 min anyway so ~15 s of lateness is noise. The danger isn't the herd itself (the smear + worker backpressure handle it); it's a crash-loop (a bad deploy taking all schedulers down repeatedly) — which is why deploys are one-at-a-time and health-gated, and why "0 schedulers firing" is a standing page. If the worst happens (all schedulers down for minutes), nothing is lost — the durable record is in Postgres; the queue refills from it on recovery; jobs fire late, not never. Cite failure-modes#stampede + #leader-loss.

The midnight thundering herd — 1 M jobs all scheduled for 00:00:00. How do you not fall over? (failure / load)

Three layers. (1) Smear at materialization: when the cron evaluator turns a "0 0 * * *" schedule into a concrete fire, for P2/P3 jobs it adds a deterministic jitter hash(job_id) mod jitter_window (300 s for P3, ~30 s for P2), so 1 M midnight jobs land spread over 5 min — ~3,300/sec instead of a 1 M instantaneous spike; P0/P1 fire on time (they're a small fraction). (2) Worker backpressure: the dispatch loop checks the worker fleet's queue depth and stops dequeuing (or sheds P3) when it's deep — the herd waits in Redis (durable, ordered), it doesn't crater the workers (cite #13, failure-modes#backlog-overflow). (3) Postgres write-spike absorption: the status updates from the herd are batched (mark 100 jobs running in one statement), and the jobs table is autovacuum-tuned for the churn. The herd cousins #27 (cache stampede) and #26 (the downstream meltdown) cover the same shape from the read and cascade angles. Without the smear, every round hour is a multi-minute backlog + a Postgres write spike — with it, the herd is just "slightly elevated for 5 minutes."

Where's the cost — what dominates the bill, and what's the cost-vs-latency knob? (cost)

The scheduler itself is ~$1,900/mo — Postgres is ~half of it (the durable store + audit log), then Redis, schedulers, etcd; the worker fleet (the thing that costs real money) isn't part of this system (it's the task queue, #9). So "the bill" of a job-scheduler is a rounding error; what you're really buying with the iterations isn't lower cost — it's correctness under failure (the Naive one-box version is ~$700/mo and is a SPOF with a double-execution race). The one cost-vs-latency knob worth naming: the Redis delay queue (~$300/mo) buys ~5 s → ~0 s dispatch latency; you'd drop it only if the 30-s SLA is comfortable and the burst hour is mild. A second: the sync standby (~half the Postgres cost) buys RPO = 0; an async standby is cheaper but loses the last few writes on failover — not acceptable for billing-class jobs. Cite cost-shapes.html.

How do you migrate from a single cron box to the distributed scheduler without double-firing or dropping jobs? (migration)

Phase 1 — shadow: the distributed scheduler reads the same schedule definitions and logs what it would fire, but dispatches nothing; diff against the old box's actual fires for a week (the divergences are almost always timezone/DST edge cases — the old box used the host's local time, the new one uses explicit per-job timezones). Phase 2 — cut over per job-class: move the low-risk, idempotent jobs first (report generation, cache warming, webhook pings); the old box stops running those, the new system starts; watch a week; proceed class by class; the riskiest (billing) last, after its worker's dedupe table is in place. During any overlap, double-firing is safe — that's exactly what the fence token + idempotent workers exist for. Phase 3 — the old box runs nothing; decommission it but keep its crontab in version control as a fallback for a month. Rollback at any phase: move a job-class back to the old box (its config is still there). The risk is concentrated in Phase 1's diff; if it matches for a week, the cutover is boring. Cite api-design-patterns#idempotency.

A job didn't fire (or fired twice) — where do you look? (monitoring / debug)

Didn't fire: pull the jobs row by job_id — what's status? pending with next_fire_time in the past → it's stuck in the queue: is it in Redis (ZSCORE sched:queue job_id)? If not, the orphan case — check the orphan-reaper's last run (it should re-ZADD it). If it's in Redis but old → the dispatch loop isn't draining: check per-tenant fairness gate (is this tenant's in-flight cap pinned by a flood?), worker-queue depth (backpressure shedding P3?), and the "0 schedulers firing" gauge. paused → someone paused it. failed → it hit the retry cap — check executions for the crash reason (poison job). Fired twice: check executions for two rows with different execution_id — that's a re-dispatch after a lease expiry (a scheduler died, or a clock-skew false-expiry, or a network blip during renewal); correlate with reclaim_count and NTP-offset spikes around that timestamp; the effect should still be once if the worker honored the fence token (check fence_token_rejected on the downstream — it should show the stale-token rejection). If the effect happened twice, the worker isn't idempotent — that's a worker bug, not a scheduler bug, but the scheduler's job is to make it survivable, so file it. Each path maps to a specific What-Breaks entry + a runbook line; the "decompose by status, then by which stage" discipline is so this is a lookup, not a guess.

A tenant schedules a million 1-second jobs — fairness, abuse, what happens? (adversary / abuse)

Three gates. (1) Submit-side rate limit (cite #20): a per-billing-account token bucket on POST /v1/jobs caps the creation rate — a million-job burst is throttled to a sane rate (or returns 429), so the flood never fully lands. (2) Dequeue fairness: even if a tenant legitimately has a million already-scheduled jobs all coming due, the weighted-fair dequeue gives them at most cap(tenant) in-flight at a time — their jobs fire at their fair share's rate, everyone else flows; their backlog grows (in their queue, visibly), but it doesn't touch another tenant's due P0. (3) Quota: a per-account ceiling on total pending jobs (and on cron schedules) — past it, new submissions are rejected with "you have N pending; cancel some or raise your quota." Abuse signals: a single account's queue-depth share > X%, its submit-429 rate, its failed-rate (a million jobs that all crash = a poison flood — the retry cap + backoff confine it). The fairness key is the billing account, not the API-key, so a tenant can't multiply its share by spawning sub-accounts. Cite failure-modes#hot-partition-meltdown.

Key Takeaways

1. Lease-based locking prevents double execution

Leases are the standard mechanism for distributed resource ownership. The scheduler writes its ID and an expiration time into the job row. If it crashes, the lease expires, and another scheduler reclaims the job. The execution_id (fencing token) provides a second layer of protection for workers that may still be running after a lease expires.

2. Redis sorted sets are better than database polling

ZPOPMIN is atomic and O(log N), giving near-zero scheduling latency. Database polling requires periodic queries that waste resources when idle and add latency equal to the poll interval when busy. Use the database for durable storage and Redis for the scheduling queue.

3. Separate cron evaluation from job dispatch

Cron schedule evaluation is a distinct concern: it runs on a fixed interval, scans all schedules, and generates one-time job instances. Job dispatch is event-driven: it dequeues and executes individual jobs. Mixing them creates complexity. Separating them lets you scale and reason about each independently.

4. Multiple schedulers with leader election give high availability

Three scheduler instances give us fault tolerance. Leader election (via etcd) ensures singleton operations like cron evaluation run on exactly one instance. Competing consumers (via ZPOPMIN) let all instances participate in job dispatch without coordination overhead.

5. Exactly-once is at-least-once + idempotency + a fence token — don't try to do better

Exactly-once delivery across a network is impossible (a worker can finish and die before acking). So the scheduler delivers at-least-once, hands the worker a monotone fence token (execution_id), and the worker is idempotent on it; downstreams that touch money dedupe on it. The complexity lives in the one worker that needs it — the scheduler stays a dumb trigger. See #14; cite it, don't re-derive it.

6. The midnight herd is a stampede — smear it at materialization

Tenants overwhelmingly schedule "daily at midnight" / "hourly on the hour," so every round hour is a thundering herd. For low-priority cron jobs, add a deterministic jitter (hash(job_id) mod jitter_window) when materializing the fire — 1 M midnight jobs spread over 5 min instead of a 1 M instantaneous spike. Pair with worker-fleet backpressure (shed P3 first) and batched status updates. See #27.

7. A multi-tenant primitive needs fairness on TWO axes

Priority is across-tenant; fairness is per-tenant. Without the tenant axis, a tenant scheduling a million P0 jobs starves everyone. Without the priority axis, a tenant's P3 batch delays its own P0. You need both: weighted-fair dequeue across billing accounts (not API keys — sub-accounts evade) plus a per-account submit rate limit (see #20). The cost of a job-scheduler is a rounding error; what the iterations buy is correctness under failure and isolation under abuse, not a lower bill.

Assumptions and When They Break

  • 116 jobs/sec fits in a single Redis + single PostgreSQL: If job volume grows 100× to 11,600 jobs/sec, you need to shard the Redis sorted set by priority level or time bucket, and use a PostgreSQL read replica for metadata lookups.
  • Jobs are independent: If jobs have dependencies (job B must run after job A completes), you need a DAG scheduler — a fundamentally different architecture (topological sort, dependency tracking, event-driven execution).
  • 60-second lease is appropriate: If jobs run for hours (e.g., ETL pipelines), the lease must be much longer, and lease renewal becomes critical. If jobs run for milliseconds, a 60-second lease means crashed jobs are "stuck" for a full minute.
  • Clock synchronization: Lease expiry depends on clocks. If clocks drift between schedulers and PostgreSQL, leases may expire early or late. Use NTP and keep drift under 1 second.
