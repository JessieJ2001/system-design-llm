Designing a Calendar Availability Service

200 M users, 50k availability-lookup QPS, 5k bookings/sec, p95 read ≤ 80 ms, p99 ≤ 800 ms, 5-region active-active. We start from a single Postgres events table with a B-tree on (user_id, start_ts) — the LC #7 "merge intervals" kernel lifted directly — and watch p99 collapse on heavy users when the 5-attendee fan-out turns into 5 random-IO range scans. Then iterate to a sharded interval store, a per-user 14-day busy-bitmap in Redis (672 bits = 84 bytes/user), single-flight + CAS-on-bitmap for transactional booking, regional cells with home-region routing, and refresh-ahead for popular shared calendars.

NALSD Walkthrough • ~50 min read • LC #7 (Merge Intervals)

LC kernel. Merge Intervals (LC #7), and its inverse "free-slot gaps from busy intervals." Textbook structure: sort intervals by start_ts; sweep left-to-right; merge overlaps; gaps between merged intervals are free slots. Per-attendee work is O(E log E) sort + O(E) sweep, where E = events in the window. Multi-attendee free-slot is the union of busy intervals across attendees: O((N×E) log(N×E)) sort. Three properties that break at scale: (1) the "merge" runs at read time, so a 5-attendee free-slot lookup over 14 days does 5×E range scans + an in-app merge over the union — with E ~400 events per heavy user, that's 2,000 rows materialized, sorted, merged on every read; the p99 user has 10k events/year ≈ 380 events in any 14-day window, and the slowest of 5 fan-out scans defines the request's p99. (2) The merged intervals get recomputed on every lookup even though the underlying events change rarely — the same 5-attendee set queried 100 times in an hour does the same merge work 100 times. (3) The textbook ignores time-zone / DST / RRULE expansion, but production must expand recurring rules ("every Monday at 09:00 forever") into concrete intervals before the merge can run; the LC kernel has no write path and no concurrency model for "two people booking the same slot at the same time." The C8 production cousin replaces read-time merge with per-user pre-materialized busy bitmaps at 30-min granularity for the next 14 days (672 bits = 84 bytes per user), refreshed on event create/update; replaces SQL-on-Postgres with Redis MGET + bitwise AND for the read path; adds CAS-on-bitmap for transactional booking; and uses regional cells + refresh-ahead to keep p99 inside 800 ms across 5 regions.

S0. Scenario & Requirements

A consumer-product calendar service (Google Calendar / Calendly / Microsoft Bookings tier): users have personal calendars; the platform exposes a "find common free 30-min slots in the next 14 days across N attendees" API for meeting scheduling, plus a transactional booking endpoint for taking a slot. We assume users are globally distributed across 5 regions (active-active); per-user event volume tops out at ~10k events/year for power users.

System class

Class: read-heavy, low-latency, derived-view-shaped — "users mutate a small write-set; readers query an aggregated/derived view of the writes with strict latency SLOs" (cite functional-requirements.html#class-derived-view). The class fixes the NFR additions: read p99 dominates (50k QPS × 5-attendee fan-out vs 5k bookings/sec); write-amplification at materialization time is OK (event creates are rare; reads are constant); and active-active multi-region raises a freshness vs latency knob the LC kernel doesn't model.

Functional requirements (cite functional-requirements.html)

  • Create / update / delete an event. Owner writes (owner_user_id, slot_start_ts, slot_end_ts, title, attendees[]); the platform persists, fans out invitations, and updates each attendee's busy view.
  • Get free slots across N attendees. Given (attendee_user_ids[], window_start_ts, window_end_ts, slot_duration=30min), return the ordered list of slot_start_ts where every attendee is free. Window capped at 14 days; slot_duration fixed at 30 min for the canonical case (the LC #7 read-merge endpoint).
  • Book a slot transactionally. Given a free-slot result + a chosen slot_start_ts, atomically take the slot for all N attendees; reject if any attendee became busy in the window between lookup and book (concurrency; cite api-design-patterns#concurrency).
  • List a user's upcoming events. Standard "show me my next 7 days" calendar UI request; pagination by start_ts.
  • Expand recurring rules. RRULE-style ("every Monday 09:00") expanded into concrete intervals out to the 14-day window for free-slot lookups.

Out of scope (named, not silently dropped): rich event metadata + attachments (separate metadata service; the slot-availability path only needs intervals); video-conferencing integration (call out, not designed); shared-calendar ACL beyond owner+attendee (per-organization ACLs are Probe A); cross-system free-busy (Google + Outlook federation; Probe B — we model only first-party); machine learning for "best meeting time" suggestion (separate ranking service; this page returns all common free slots, not a ranked subset); calendar-of-rooms / resource booking (modeled identically to user calendars but with separate IDs; we only show user calendars).

Flagged assumptions. (1) Average shared-availability lookup spans N=5 attendees (interview-typical "find a meeting time among 5 people"); we'll re-derive cost/perf for N=10 in a constraint flip. (2) 90% of lookups span attendees in the same region (commute / co-located teams); 10% are cross-region (the interesting tail; Iter 3 addresses). (3) Slot granularity is 30 min for the shared-availability endpoint (the canonical Calendly setting); custom granularities exist but are out-of-scope optimization (Iter 4 probes). (4) RRULE expansion happens at write-time into the 14-day forward window; we don't store infinite recurrences in the read path (would explode the bitmap).

Sister scenarios

  • Resource-booking (rooms / desks / equipment) — same merge-intervals kernel; the entity is a room not a user, but the bitmap shape is identical (busy/free per slot). Drastic flip: one room is shared by many bookers; concurrency is higher per-entity (a room can be CAS-fought-over by all of an org's calendars at once). The Iter 2 single-flight + CAS move applies more aggressively.
  • Doctor / hairdresser appointment booking (Calendly-pro tier) — same shared-availability kernel; but the lookup is "given one provider's calendar + many parallel candidate bookers, find slots." Drastic flip: read fan-out is N=1 (one provider), so the per-user bitmap suffices and Iter 0's sharded interval store collapses to a single-shard read.
  • Movie / event ticketing flash sale (cite #28 Flash Sale) — same CAS-on-counter pattern but at extreme contention (one popular event = millions of bookers fighting for thousands of seats). Drastic flip: contention dominates; we'd add a queueing layer; a per-user bitmap doesn't apply — the contended resource is the seat count, not a per-user calendar.
  • Distributed rate limiter (cite #20 Distributed Rate Limiter) — sister via "per-user counter / bitmap mutated under CAS at high QPS." Both compress per-user state to a small bitstring + atomic op; both face cross-region replication of mutations.

The fundamental tension. Three constraint-pairs collide. (a) Read latency vs write freshness — events are mutable (people RSVP, reschedule, decline) but lookups must complete in 80 ms across 5 attendees; we can't merge intervals at read-time, but we also can't let bitmaps drift more than a few seconds behind events. (b) Per-user state size vs cross-region replication cost — 84 bytes per user fits in Redis trivially, but 200 M users × 5 regions = 1 B replicated bitmaps; mutation propagation has to be lazy (last-writer-wins or home-region-leader). (c) 5-fan-out tail vs 5-region active-active — a 5-attendee lookup is "slowest of 5 reads"; cross-region adds a second tail-multiplier on the 10% of queries that span regions; we need N×1 region-local lookups, not N×K cross-region. The arc resolves the tension: shard the events (Iter 0), materialize the bitmap (Iter 1), CAS-on-bitmap for booking + single-flight on lookup (Iter 2), regional cells (Iter 3), refresh-ahead for popular calendars (Iter 4).

The partition-key heuristic: the dominant access pattern is "give me user X's busy view in the next 14 days" (every availability lookup hits N attendees once each). Hash partition by attendee_user_id for the events table and the bitmap so a single-user lookup is a single-shard read. This prevents cross-shard scans for the per-user view at the cost of making "all events for a meeting" require a fan-out scatter (rare; only the meeting-owner UI uses it — we co-locate with meeting_id in the Bookings table).

S4. Naive Architecture — the simplest thing that meets the FRs

One Postgres database; one events table with a B-tree on (user_id, slot_start_ts). A 5-attendee free-slot lookup runs 5 range scans (one per attendee) over the next 14 days, materializes the union of busy intervals in the application, sorts + sweeps to merge, then returns the gaps. This is LC #7's reference solution lifted directly: read each user's events, merge the union, return free slots. Booking is a transaction: INSERT booking IF NOT EXISTS WHERE no conflicting event exists for any attendee in [start_ts, end_ts].

Architecture

Connections (numbered above) and line-style legend. Solid slate = sync hot path. The Naive design has no async paths and no cache — every lookup blocks on Postgres.

  • ① Client → API Workers — HTTPS GET /v1/availability/free-slots?attendees=u1,...,u5&....
  • ② API Workers → Postgres `events` — 5 parallel range scans: SELECT slot_start_ts, slot_end_ts FROM events WHERE attendee_user_id=$1 AND slot_start_ts BETWEEN ? AND ? AND is_busy=true ORDER BY slot_start_ts. App-side merge intervals, gap-detect, return.
  • ③ API Workers → Postgres (Booking) — BEGIN; SELECT ... FOR UPDATE; INSERT booking; INSERT events for each attendee; COMMIT. SERIALIZABLE isolation prevents double-book.

This is the simplest architecture that satisfies all FRs — one table, one engine, one place. No cache, no derived view, no fanout-coordinator. Booking is one ACID transaction.

Why it's broken (per failing NFR)

  • Read p95 / p99. ✗ 5-attendee fan-out's tail is dominated by the slowest of 5 range scans. With a heavy user (384 events / 14d) and cold cache, a single scan is 150–200 ms; the slowest-of-5 tail makes p95 ≈ 200 ms (over 80 ms NFR by 2.5×) and p99 ≈ 1,500 ms (over 800 ms NFR by ~2×). The B-tree+merge structure forces this even before contention enters.
  • Write throughput & correctness. &warning; 5,000 bookings/sec under SERIALIZABLE on a sharded Postgres works in p50 but tail-explodes under contention (deadlock retries, FK lookups, cross-shard 2PC); p99 booking latency > 1 sec under load. Correctness is fine (SERIALIZABLE ensures no double-book) but latency fails.
  • Read-time merge waste. &warning; The same (sorted-attendee-set, window) gets queried repeatedly — Calendly-style "share my availability link" gets opened 100x by a recipient deciding when; the merge runs 100 times. No caching layer exists in Naive.
  • 5-region active-active. ✗ This Naive design has no multi-region story; events live in one Postgres cluster. Cross-region lookups pay 80–200 ms RTT on every range scan, blowing the SLO open by an order of magnitude on cross-region attendees.

Cost of naive at NFR-target load: ~$17k/month and STILL fails read-p99 NFR structurally on 5-attendee fan-out + heavy-user range scans. The first failing NFR is read latency (the 800-ms p99). The B-tree on (user_id, start_ts) is exactly LC #7's data structure; the failure is that we're paying read-time merge cost on every lookup — we must change the unit of read (precomputed bitmap) and localize lookups to one engine (Redis), not scale Postgres.

Architectural decisions to revisit

Alternatives considered (C4)

  • Cassandra sharded by attendee_user_id + materialized busy intervals — chosen. Cassandra excels at "give me one partition's rows in time order" — exactly what "user's events in 14d" looks like. The materialized busy_intervals_14d rolls a per-user merge into a single partition row (~50 merged intervals); read is one point-read. Cite storage-engines#lsm.
  • Spanner / CockroachDB — rejected. Strong consistency we don't need (5s convergence is fine cross-region); ~5× the cost of self-hosted Cassandra. Reconsider if booking required true global serializability (it doesn't — per-user CAS is enough).
  • FoundationDB — rejected. Transactional KV; transactions are unused for the read path (which dominates QPS); txn-coordinator overhead is pure cost. Useful for booking but Postgres covers that more cheaply at 5k writes/sec.
  • Postgres + materialized view — rejected. Postgres MVs need REFRESH (not incremental); on every event mutation we'd re-run the merge over the full table. Cassandra's partition-write model fits a per-user "rebuild this partition" workflow better; it's also where the events live.
  • Skip the materialization; just shard better — rejected. Sharding fixes the QPS-per-node ceiling; it doesn't fix the per-call p99. The merge cost stays at read-time. Materialization is structural — it changes the work done per read by an order of magnitude.

The chosen move

Three coordinated changes, all aimed at the same read-time-merge bottleneck: (a) move events to Cassandra partitioned by attendee_user_id so a single user's 14d window is a partition scan in time order; (b) add a derived busy_intervals_14d column-family that stores each user's pre-merged busy interval list; (c) on event create/update/delete, an async BusyInterval Refresher recomputes the affected user's partition row (incremental: only the affected user; ~5 ms per refresh). The read path becomes 5 point-partition reads on busy_intervals_14d (not 5 range scans on events) + a small union/gap-detect in the API worker.

What breaks (C5; cite failure-modes.html)

  • Refresher lag under burst event creation. If a user creates 50 events in 30 seconds (e.g., bulk import), the Refresher can lag by minutes; reads see stale busy_intervals_14d. Alternatives considered (C4)

  • Redis cluster + 84-byte bitmap per user — chosen. Native bitwise ops (BITAND, BITCOUNT); MGET amortizes 5 keys into ONE round-trip (vs Cassandra's 5 separate hops); ~80 GB usable per cache.r6g.4xlarge fits 200M users in 6 nodes. Sub-ms p99 per MGET. Cite storage-engines#kv-engine.
  • Memcached — rejected. No native bitwise ops; would force read-modify-write of the bitmap from app; lose the server-side AND. Also: no Lua scripting for atomic CAS (needed in Iter 2 for transactional booking).
  • DynamoDB DAX — rejected. Hosted in-memory tier in front of DynamoDB; pricing for 50k MGET/sec is much higher than self-hosted Redis; cross-region multi-active-replication is opaque (we want explicit control for Iter 3).
  • Aerospike — rejected. Fits the hot-key + small-value workload but adds a vendor license cost without a competitive QPS edge at our scale; Redis Cluster covers the throughput.
  • RocksDB on NVMe (instead of in-memory) — rejected. 84 bytes × 200M = 17 GB working set; Redis fits it in RAM directly. RocksDB would add an SSD-tier I/O hop on every read for no benefit.
  • Bitmap stored in Cassandra as a blob column — rejected. Same coordinator-hop tail; storing it in Cassandra wouldn't help the read latency. The Redis hop's value is in-memory + MGET batching; that's the structural win.

The chosen move

For each user, materialize a per-window bitmap: 14 days × 48 30-min slots = 672 bits = 84 bytes. Each bit is 1 if the user is busy (any event with is_busy=true covers that slot), else 0. Key: bitmap:<user_id>:<window_start_day>; value: 84 raw bytes; cluster: Redis 7 OSS Cluster on 6 cache.r6g.4xlarge nodes (RF=2; shaped by user_id hash slot). Read: MGET 5 keys in one round-trip; bitwise AND_NOT in the API worker; enumerate runs of zeros ≥ 1 bit (30 min) and emit slot_start_ts. Write: when an event mutates, the Bitmap Refresher reads that user's 14d events from Cassandra, OR-folds them into a fresh 84-byte bitmap, SETs with a version bump.

Cite sketches.html for the bitmap-as-membership pattern; cite storage-engines#kv-engine for Redis. Cassandra remains the SoR (events are still authoritative; bitmap is rebuildable in O(events_in_14d) ≈ 5 ms per user).

(b) CAS-on-bitmap: booking is no longer a SERIALIZABLE Postgres transaction across 5 events; instead, a Lua script runs atomically on each attendee's bitmap key — "if bitmap_version == expected and bit at slot_idx == 0, then set bit + bump version, else return CONFLICT." 5 CAS ops; if any returns CONFLICT, we 409 the client and let it re-fetch. Postgres bookings becomes a record-keeper (idempotency + audit), not the contention point.

Bottleneck

Iter 1 left two wounds open. (a) Booking concurrency: Postgres SERIALIZABLE across 5 attendees has p99 ~1 sec under contention; the contended resource is actually the bitmap bit (a single slot), not "all of attendees' rows in events" — we're using a heavier primitive than needed. Wrong-concurrency-primitive: SERIALIZABLE was chosen for global ordering we don't need; per-bitmap-bit CAS is the right grain. (b) Redundant identical lookups: Calendly's "find a time" link is opened 100× by a recipient cohort; each open re-fetches 5 bitmaps and reruns the merge despite no underlying change. Redundant-work bottleneck: the same answer is computed 100 times; single-flight + short-TTL cache fixes it for ~$0 cost.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#wrong-concurrency-primitive: match the lock grain to the contended resource. Per bottleneck-to-decisions.html#redundant-work: coalesce concurrent identical work via single-flight; cache results with a TTL just long enough to absorb the burst. The combined move: bitmap-bit-grain CAS for booking + 2-second single-flight on lookup.

Alternatives considered (C4)

  • Redis Lua CAS script + single-flight in Redis — chosen. Lua scripts run atomically inside Redis (single-threaded per slot); 5 sequential CAS attempts (one per attendee) have p99 ~5 ms total; no cross-shard 2PC. Single-flight is a 1-line Redis SETNX. Both reuse the existing Redis cluster (different keyspace prefix). Cite single-flight.html.
  • Postgres SERIALIZABLE across 5 attendees (Iter 1) — rejected. Heavy under contention; deadlock retries inflate tail; the global serialization order isn't required — we only need "no two bookings book the same slot for the same user," which is per-bitmap-key.
  • Distributed lock service (etcd / Consul / ZooKeeper) — rejected. Adds a separate cluster; per-lock latency is 5–20 ms; we'd pay it 5 times per booking. Redis Lua gives us atomicity per key in <1 ms.
  • Optimistic locking via Postgres UPDATE WHERE version=? — rejected. The bitmap is in Redis (the read path); routing booking through Postgres means we'd need to keep Postgres + Redis in lockstep, doubling write paths. CAS on the source-of-truth-for-reads (Redis) is simpler.
  • Two-phase commit across Redis + Postgres + Cassandra — rejected. 2PC adds a coordinator + WAL; the latency budget can't absorb it. Instead: do CAS on Redis first; if all 5 succeed, async write to Postgres + Cassandra; if Postgres write later fails (rare), the booking_id reconciler reverts the bitmap (idempotency_key key allows safe replay).
  • Per-(sorted-attendees,window) materialized free-slot view (longer cache) — rejected. Would cache too long (5 sec+) and risk stale-slot bookings; a 2s TTL with single-flight gives the burst-absorption without the staleness.

The chosen move

(a) Single-flight on lookup. Each lookup hashes (sorted(attendee_user_ids), window_start_day) to a single-flight key sf:<sha1>. The first lookup does SET sf:<sha1> "in-flight" NX EX 2; on success it computes the result (MGET 5 bitmaps + AND), then SET sf:<sha1> <result-blob> EX 2. Concurrent lookups within 2 seconds GET the cached blob (sub-ms). Burst on a popular Calendly link ≈ 200 concurrent opens → 1 actual computation + 199 cache reads. ~30% global hit rate (Calendly-style links concentrated; long-tail unique queries don't hit single-flight).

(b) CAS-on-bitmap for booking. The booking flow:

  1. Client sends POST /v1/bookings with expected_bitmap_versions[u_i] from the prior lookup response.
  2. Booking Worker computes slot_idx = (slot_start_ts - day0) / 30min.
  3. For each attendee in turn (or in parallel via pipelined EVAL), runs the Lua CAS script on bitmap:u_i:day0: 
    -- KEYS[1]: bitmap:<user_id>:<day0>
-- KEYS[2]: bitmap:<user_id>:<day0>:version
-- ARGV[1]: expected_version
-- ARGV[2]: slot_idx (0-671)
-- Returns: "OK" | "VERSION_MISMATCH" | "ALREADY_BUSY"
local cur_v = redis.call("GET", KEYS[2]) or "0"
if cur_v ~= ARGV[1] then return "VERSION_MISMATCH" end
local b = redis.call("GET", KEYS[1])
local byte_idx = math.floor(tonumber(ARGV[2]) / 8)
local bit_idx  = tonumber(ARGV[2]) % 8
local cur_byte = string.byte(b, byte_idx + 1)
if bit32.band(cur_byte, bit32.lshift(1, bit_idx)) ~= 0 then
  return "ALREADY_BUSY"
end
local new_byte = bit32.bor(cur_byte, bit32.lshift(1, bit_idx))
local new_b = string.sub(b, 1, byte_idx) .. string.char(new_byte) .. string.sub(b, byte_idx+2)
redis.call("SET", KEYS[1], new_b)
redis.call("INCR", KEYS[2])
return "OK"
  4. If any CAS returns non-OK, revert prior CAS-OK attendees (small compensating Lua), 409 to client.
  5. If all 5 succeed: INSERT into Postgres bookings (idempotency_key UNIQUE); INSERT 5 events into Cassandra. Bitmap Refresher will independently confirm via CDC (defensive consistency).

Bottleneck

Iter 2 is single-region. Architectural decisions to revisit

Per cross-cutting#cells: each region is a self-sufficient cell; cross-cell traffic uses explicit cross-region RPCs; data ownership is pinned per-entity to a home cell. The combined move: pin each user to a home_region; replicate writes to peer regions via Kafka MirrorMaker; serve reads region-locally where possible.

Alternatives considered (C4)

  • Spanner-style globally-strong-consistent — rejected. Spanner gives serializable cross-region commits but at 5× the cost; we don't need strong consistency on availability lookups (last-writer-wins on bitmap is fine for 5s convergence). Reconsider for a banking-grade calendar where over-booking is a regulatory hazard.
  • Active-active multi-master with CRDTs on bitmap — rejected.   • Read-from-anywhere with leader-elected writes — rejected. One leader region per user (per home_region) is what we picked. The global-leader variant would force all writes through one region — latency tail-explosion on writes from non-leader regions.
  • DNS-based geo-routing only (no per-user home_region) — rejected. Geo-routes the API call to the closest region but still leaves the question of "where does this user's bitmap live?" If everywhere, replication cost is 5× storage; if only locally-cached, EU user traveling to US pays cold-cache penalty. The home_region approach pins the truth; geo-routing handles edge serving.

Bookings owned by the meeting-owner's home region. (c) Cross-region replication via Kafka MirrorMaker: each region's Cassandra CDC stream publishes to a global topic; peer regions consume and apply to their replica view of foreign users (not authoritative; canonical lookups still RPC to the home region for fresh data, but a stale-OK local replica serves common cases sub-ms).

The peer-region RPC is bounded by 80 ms p99 RTT; the local fan-out is parallel; total p99 ≈ 80 ms (vs 200 ms if we'd routed the whole call to one region).

Plus: RRULE expansion moves to write-time (was lazy in Iter 0; sync now) so the 14-day window always has all expected occurrences ready — no read-time gap on recurring meetings.

Bottleneck

Iter 3's cross-region path costs 80–200 ms when the local replica is stale or for users whose calendars are mutating. ~10% of lookups span regions, but a small fraction of users account for a disproportionate share of cross-region demand: top-1% of users (popular Calendly links, executives, popular meeting templates) generate ~50% of all lookups. Power-law access pattern on the user-popularity dimension; per bottleneck-to-decisions#read-amplification: cache the hot subset; tier hot/cold; populate via observed access frequency. Refresh-ahead-replication for the popular 1% pays for itself: 2M users × 84 B × 5 regions = 840 MB extra storage cluster-wide, in exchange for cutting cross-region RTT on half of all reads.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#read-amplification + access-patterns#power-law: identify the hot subset by observed frequency; pre-replicate it; let the long tail pay full lazy-replication cost. The streaming top-K identifies the popular subset; refresh-ahead applies the pre-replication; RRULE expansion at write-time eliminates a read-time computation gap for recurring events.

Alternatives considered (C4)

  • Streaming top-K + refresh-ahead for top-1% — chosen. Top-1% by lookup-QPS captured by Count-Min + heavy-hitters sketch (cite #15 Top-K Frequent); rolling 5-min window. Refresh-Ahead Worker pushes those users' bitmaps to all 5 regions on every event change. Bandwidth: 2M users × 5 regions × ~5 mutations/day per popular user ≈ 50M small writes/day; trivial.
  • Replicate all 200M users to all 5 regions — rejected. 200M × 84 B × 5 regions × overhead = ~250 GB cluster-wide; cost ≈ 5× current Redis fleet. The 99% of users who don't generate cross-region lookups don't pay for this. Power-law-tail-replication is wasteful.
  • Lazy on-demand replication only (Iter 3 baseline) — rejected. First cross-region lookup pays full RTT; subsequent cached. But for popular users with constant churn (their event count changes faster than the cache TTL), every lookup pays the RTT.
  • Separate "popular" Redis tier — rejected. Adds a new cluster + a routing decision per lookup. The is_popular flag in user_metadata is a per-user bit; refresh-ahead writes to the same regional Redis; routing logic stays simple ("local first; if missing, cross-region").
  • Pre-expand RRULEs to 1-year horizon — rejected. Bitmap is 14d (672 bits); pre-expanding 1 year's worth would mean we'd recompute the bitmap when the window slides. Daily RRULE roll-forward (existing pattern) is sufficient; the 14d window is the boundary.

The chosen move

Three coordinated changes targeting cross-region tail + recurring-event read freshness: (a) Streaming top-K Popularity Tracker: every lookup increments a Count-Min sketch + heavy-hitter list; rolling 5-min window emits the top-1% (~2M users) by lookup-QPS. Stored as is_popular bit in user_metadata. (b) Refresh-Ahead Worker: for is_popular users, on every event change, push the rebuilt bitmap to all 5 regions in parallel (vs Iter 3's home-region-only + lazy replication). Plus probabilistic early refresh on TTL: if a popular user's bitmap is approaching its 21-day TTL, proactively refresh in the last 1 hour. (c) RRULE Expansion at write-time: when a Meeting with a recurrence rule is created, the RRULE Expander runs synchronously into Cassandra events for the 14-day horizon; the Refresher updates the bitmap immediately. No read-time gap for recurring meetings.

Degradation contract

When the Redis bitmap cluster in a region is unavailable: API falls back to Cassandra busy_intervals_14d (Iter 0 path) at ~50 ms p99; SLO degrades from p99 30 ms → 80 ms; still meets the 800 ms SLO. When Kafka MirrorMaker is lagging > 30 sec: cross-region lookups bypass local replicas and call peer regions directly; cross-region p99 jumps from 12 ms (popular users hit) to 80–200 ms (full RTT), but SLO still met. When a region fails entirely: home_region fails over to failover_region; affected users' bookings briefly read-only (~5 min RTO) but lookups continue served from any peer region's replica. When Postgres bookings cluster fails: bookings rejected with 503 + Retry-After; bitmap CAS continues but no booking row written → client sees 503; on Postgres recovery, idempotency_key dedupes retries.

Real-system comparison

Google Calendar's free-busy API is the canonical reference; their public posts describe a per-user materialized busy view at coarse granularity, with Bigtable as the SoR and a bitmap-equivalent in their Borg-managed cache. Our 5-region cell shape mirrors Google's spanner-region pattern (with the simplification that we use Cassandra + Kafka instead of Spanner since we don't need globally-strong-consistent bookings).

Calendly's published engineering blog posts describe a Postgres-on-RDS shape with per-org sharding; at their scale (~100k orgs, sub-billion-events) the Postgres B-tree path works without our bitmap layer. Their move to "scheduling links" with refresh-ahead happened at the 1M-org scale where Calendly-link bursts started saturating Postgres on shared hot tenants.

Microsoft Outlook / Exchange use a per-mailbox calendar with synchronization protocols (EWS, ActiveSync); the free-busy API there is server-side merge over the per-mailbox SQL store — a Naive-shape kept alive by Outlook's enterprise-scale per-tenant deployment (a single tenant rarely exceeds the heavy-user threshold).

Cal.com (the open-source Calendly competitor) uses a single Postgres database with naive merge; runs at ~10k QPS comfortably. The bitmap layer is what we'd reach for if their consumer scale grew 100×.

Where this design diverges from the published systems: (a) we use Redis with a 14-day bitmap as a first-class derived view; production systems commonly use Bigtable / Cassandra-backed views with similar bit-encoding but disk-backed; (b) we explicitly model the 5-region active-active with refresh-ahead for popular users; production systems often use a single-region for the SoR with read replicas in other regions; (c) our CAS-on-bitmap is more aggressive than typical "Postgres SERIALIZABLE for booking" — production Calendly stays with SERIALIZABLE because their per-shard contention is low enough. Anticipated pushbacks

Stress tests against the derivation above. If any answer doesn't fit in 3–6 sentences, the page is missing a probe — close it before declaring done.

Why not just put it in Postgres with a B-tree on (user_id, start_ts)? (naive trap)

  • Heavy-user range scan: 10k events/yr ≈ 384 events in any 14-day window; per-attendee scan p99 ~150 ms cold cache; 5-attendee fan-out tail = max of 5 = ~200 ms p95 / ~1500 ms p99; misses the 80 ms p95 / 800 ms p99 NFR.
  • Read-time merge wastes CPU: Calendly link clicked 100× in 5 minutes → 100 redundant merges. Bitmap + AND eliminates this; SF-cache eliminates redundant lookups.
  • SERIALIZABLE booking degrades under contention: 5k bookings/sec across 5 attendees in a single Postgres SERIALIZABLE txn → deadlock retries inflate p99 to 1+ sec. Per-bitmap-bit CAS scales linearly.
  • No multi-region story: Postgres logical replication is per-cluster; cross-region active-active needs custom CDC + conflict resolution. Cassandra + Kafka is the off-the-shelf shape.

Why a 84-byte bitmap and not store events directly in Redis? (encoding choice)

  • Fixed size: 84 bytes regardless of event count; cluster sizing is predictable. Storing event lists per user means 384 events × ~32 B = 12 KB for heavy users — 150× the bitmap.
  • Bitwise AND at memory bandwidth: 5 × 84 B AND across 5 attendees ≈ 420 B; AVX2 SIMD does it in ~1 µs. Merging interval lists is O(N events log N) per call.
  • Loss of resolution: 30-min granularity rounds events; a 1-minute meeting in slot[i] occupies the whole 30-min slot. For consumer calendar UX, 30 min is the user-facing slot anyway; for higher resolution, flip to 5-min granularity at 6× the bitmap size.
  • Loss of metadata: bitmap encodes only busy/free; no event titles, no decline status. The bitmap is the read-path optimization; Cassandra events remains the SoR for full event metadata.

How do you handle DST / timezone edge cases? (temporal correctness)

  • Storage in UTC: events stored as UTC slot_start_ts + slot_end_ts; bitmap built in UTC slots (672 30-min slots starting at day0 midnight UTC).
  • RRULE expansion handles DST: a rule "every Monday 9 AM Pacific" interpreted via IANA TZ rules at expansion time; America/Los_Angeles 9 AM is 17:00 UTC in standard time, 16:00 UTC during DST — both expand to correct UTC slot indices.
  • Twice-a-year edge case: a "fall-back" Sunday has 25 hours; the bitmap day0 is fixed at UTC midnight, so the local calendar UI shows two 1:30 AM slots that point to different UTC bits. Acceptable; documented.
  • Display in user-local TZ: API response includes UTC timestamps; client converts to user's TZ for rendering. Server doesn't store user TZ in the bitmap; only the User table has tz_iana.

What if N attendees were 50 (org-wide all-hands)? (scale flip)

  • Bitmap MGET stays cheap: 50 keys via MGET; ~2 ms p99 cluster-internal; bitwise AND of 50 × 84 B = 4.2 KB; ~5 µs SIMD.
  • SF-cache hit rate higher for org-wide bursts; one all-hands query absorbs N concurrent opens.
  • CAS booking with N=50: 50 sequential CAS Lua scripts; ~20 ms; under the 200 ms NFR. 409 conflict probability higher (the Birthday paradox); clients see more retries.
  • Common-free slot rarity: with 50 attendees in different time zones, common-free slots in working hours are rare; the API may return zero results often. UX flip: rank-by-fewest-conflicts (out of scope for this page).

How does the system serve booking under 1M concurrent click-bookers (viral event)? (extreme contention)

  • Per-slot CAS contention: 1M clicks on the same 1-hour slot for one popular user: 1 wins, 999,999 see 409. CAS Lua is ~1 ms per attempt; the wave clears in ~1 sec wall-clock (Redis throughput ~200k Lua/sec/slot).
  • Booking SLO degraded: p99 booking jumps from 30 ms to ~1 sec for the wave; clients see "slot taken" 99.9999% of the time and redirect to next-available.
  • SF-cache absorbs lookup load: 1M lookups for the same slot → ~10 actual computations after de-dup at SF cache.
  • Real-world: the Iter 4 design is shaped for 5k bookings/sec sustained; viral events > 100k/sec are rare and accept degraded SLO; queueing layer (a la flash-sale, cite #28) is a Probe.

What if the bitmap drifts from events? (consistency)

  • Refresher rebuilds from Cassandra: the bitmap is a derived view; on every event mutation, the Refresher rebuilds the affected user's 84 bytes from a 14-day partition scan of Cassandra (the SoR).
  • Defensive consistency: after a CAS booking, the Refresher's CDC consumer also runs a rebuild; converges within ~200 ms even if CAS already updated the bit (idempotent: second SET overwrites identical bytes).
  • 409 retry: if a lookup returned a free slot but the bit was actually busy (drift), CAS fails with VERSION_MISMATCH or ALREADY_BUSY; client retries with fresh bitmap. The SoR (Cassandra events) is authoritative; bitmap drift never causes incorrect booking.
  • Total rebuild: if the bitmap cluster is lost entirely, full rebuild from events takes ~30 minutes (200M users / 5 regions / 5 worker nodes × 5 ms/user).

How does this differ from Google Calendar's free-busy API? (real system)

  • Same kernel: per-user materialized busy view at coarse granularity; multi-region async replication; CAS-class booking primitive.
  • Engine substitution: Google uses Bigtable + Spanner internally; we use Cassandra + Postgres to stay within the interview-budget assumption. Spanner gives globally-strong-consistent bookings (no double-book on partition); we accept the rare-edge-case via CAS-409 retry.
  • Granularity: Google exposes free-busy in 5-minute slots for premium users; we picked 30-min for the brief. Bitmap size scales 6× at 5-min; achievable but more Redis fleet.
  • Pricing model flip: Google amortizes the cost across Workspace seat licensing; standalone calendar SaaS faces the per-call cost we computed ($6 per million bookings).

Where does the bill come from? What's the cost knob? (cost)

  • ~40% Cassandra events SoR: 6 nodes × 5 regions × $986 = $30k. Compute-heavy: 10B events at 50 B/row = 500 GB; storage tiny but compute (writes + reads + replication) drives the bill.
  • ~13% Redis bitmap: 2 nodes × 5 regions × $980 = $10k. The size win pays off massively here: 84 B/user fits in 1 node per region.
  • Top knob: region count. 3-region instead of 5-region cuts ~$30k (40% drop) at the cost of higher cross-region latency for outlying regions.
  • Next knob: Cassandra RF. RF=2 instead of RF=3 cuts compute fleet by 1/3 (~$10k); accepts higher data-loss probability.
  • Hardest to cut: the bitmap is structural for the read-p99 NFR; can't remove without breaking the SLO.

How do you migrate from Naive (Postgres) to Final without downtime? (migration)

  • Phase 1 — events to Cassandra: dual-write events to both Postgres and Cassandra for 30 days; backfill historical events; cut over reads to Cassandra; deprecate Postgres events table.
  • Phase 2 — bitmap layer: stand up Redis bitmap; Refresher backfills from Cassandra (~30 min for 200M users); dual-read (live merge AND bitmap lookup) for 14 days; assert match; cut over reads to bitmap.
  • Phase 3 — CAS booking: add Lua scripts; switch booking endpoint from SERIALIZABLE Postgres txn to CAS-on-bitmap; dual-write to bookings table for audit; deprecate the SERIALIZABLE path after 30 days.
  • Phase 4 — regional cells: stand up secondary regions one at a time; backfill via Kafka MirrorMaker; geo-route 1% → 10% → 100% per region.
  • Phase 5 — refresh-ahead: deploy Popularity Tracker; verify top-1% list stability; enable refresh-ahead replication.
  • Total: 90–180 days for full migration; SLO improvements ramp at each phase boundary.

S7. Key Takeaways

  • The merge-intervals kernel is a read-time-merge artifact. Once realistic event volumes enter (heavy users with 384 events per 14-day window), the per-call sort+merge dominates p99 and the 5-attendee fan-out tail explodes the latency. The right move is to change the unit of read to a pre-materialized per-user bitmap so reads become O(N×byte) bitwise AND, not O(N×E log E) merges.
  • Pick the encoding by access shape, not by name. A 14-day × 30-min bitmap is 672 bits = 84 bytes/user — predictable, fits memory, supports server-side BITAND. Storing event lists in Redis would have been larger by 100×, lose the bitwise op, and miss the structural win.
  • Match the lock grain to the contended resource. SERIALIZABLE Postgres for booking grabs locks on "all attendees' rows in events"; the actual contended resource is the per-bitmap-bit. CAS-on-bitmap via Redis Lua is per-key atomic and 10× faster than Postgres SERIALIZABLE under contention.
  • Single-flight is free latency win on shared-link workloads. Calendly-style "find a time" links get opened 100× by a recipient cohort; SETNX with 2-sec TTL coalesces them into 1 actual computation. ~30% real-world hit rate; cost = adding a few key-prefix conventions.
  • Multi-region is a structural NFR, not a performance one. 5-region active-active demands per-region home cells with per-user home_region; cross-region replication via Kafka MirrorMaker (~200 ms p99). Spanner is the alternative if you need strong-consistent bookings; for our ~5-sec freshness SLO, async replication suffices and saves 5× the cost.
  • Power-law popularity dictates a small refresh-ahead set. Top-1% of users (Calendly-link celebrities, executives) drive ~50% of cross-region demand; pre-replicating 2M users' 84-byte bitmaps to all 5 regions costs ~3 GB cluster-wide and cuts cross-region p99 from 60 ms to sub-ms. Refresh-ahead is the canonical "tier the hot subset" move.
  • RRULE expansion is write-time work. Recurring rules ("every Monday 9 AM") expand into concrete events at write-time into a 14-day horizon; daily roll-forward keeps the horizon current. Read-time RRULE expansion would put parsing + DST + IANA-TZ math on the hot path; doing it at write keeps reads simple.
  • Cost shape is compute-heavy, not storage-heavy. Final $74k/mo is dominated by Cassandra event SoR (40%) and per-region API/Booking fleets. Storage is trivial (~500 GB total events). The diagnostic question: "where's the elephant — compute, storage, network, or memory?" For calendar availability, compute (per-call read fan-out) is the elephant; the bitmap re-encoding is what tames it.
