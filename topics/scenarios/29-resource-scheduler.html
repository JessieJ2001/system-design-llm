Resource Scheduler: Multi-Tenant Cloud Platform

Placing 500,000 virtual machines across 50,000 physical servers — bin packing, fragmentation, fault domains, and why a greedy first-fit algorithm leaves 30% of your hardware idle.

NALSD Walkthrough • ~25 min read • LC #253 (Meeting Rooms II)

LC kernel. Meeting Rooms II (LC #253 / #252). Textbook structure: sort meetings by start time; keep a min-heap of end-times of rooms in use; for each meeting, if the heap's smallest end-time ≤ this meeting's start, pop it (that room frees and is reused), else allocate a new room; push this meeting's end-time. The heap size at any moment is the number of concurrent rooms. O(n log n). Lift it to a cluster: rooms → servers, end-times → "server free at time t", a meeting → a VM/job wanting some resources for a while — pop the server with the earliest free-time, place, push back. Three properties that break at 50,000 servers: (1) one dimension — a meeting needs "a room"; a VM needs (cpu, ram, disk) simultaneously, so "any free server" becomes multi-dimensional vector bin-packing, which is NP-hard (no optimal poly-time algorithm) — we use heuristic best-fit, not "earliest-free"; (2) a static set — the LC rooms exist forever and never change; a real fleet churns (servers fail, get added, drain for maintenance), so the in-memory node-state is a cache of reality that must be reconciled, not a ground truth; (3) a function return — LC returns "which room"; production returns a reservation that can be preempted, under a priority/fairness policy, with anti-affinity and fault-domain constraints — and the "scheduler" is no longer one process but a hierarchy of cell schedulers with leader-election and failover. The C8 production cousin replaces the bare min-heap with a filter-and-score multi-dimensional best-fit-decreasing placer (Iter 1), wrapped in cached node-state + optimistic concurrency (Iter 2), partitioned into cells with leader-elected schedulers (Iter 3), with a background defragmenter (Iter 4) and an ILP batch-rebalancer for the long pole (Iter 5). For the GPU-topology-aware variant (NVLink domains, rail fabric, gang scheduling — where slots are not interchangeable and every placement formula is topology-weighted) see #39 GPU cluster scheduler. And note this is the "where to run" scheduler — the "when to run" scheduler (cron/DAG/dependency triggering) is a different problem: see #23 job scheduler.

Placement Constraints

  • Hard constraints: A VM must fit on the host (enough free CPU, RAM, disk). Memory cannot be overcommitted.
  • Soft constraints: CPU can be overcommitted up to 2:1 ratio. Prefer spreading VMs across racks for fault tolerance.
  • Anti-affinity: Replicas of the same service must not land on the same physical host (ideally not the same rack).
  • Data locality: Some VMs prefer hosts near specific storage volumes.

How this differs from job scheduling (#23): #23 is the "when to run" scheduler — it owns the trigger (cron, a DAG dependency, a queue arrival) and decides at what time a task fires; tasks run and finish. This page is the "where to run" scheduler — it owns the placement: given a workload that must run now (or soon), pick a machine. The two compose (a job scheduler decides "run this batch job now" then asks a resource scheduler "on which node?"), but the hard parts are disjoint: #23's is time/dependency/retry semantics; #29's is multi-dimensional bin packing, fragmentation, fault-domain awareness, and preemption. (And #39 is #29 specialized to a GPU fleet where the slots are not interchangeable — topology decides throughput — which re-weights every placement formula.)

System class

Class: resource scheduler / placement service — an infrastructure primitive (cite functional-requirements.html#class-infra; NFR profile nfr-checklist.html#class-infra-nfr). It takes a stream of placement requests against a fixed-but-churning pool of machines, decides placements (and preemptions) honoring hard constraints, priorities and anti-affinity, maintains an allocation map, and emits assign/evict commands; it does not run the workloads (the hypervisors / node agents do that), it does not own the lifecycle trigger (#23 does), and — in this generic version — the resources are interchangeable (4 free cores is 4 free cores); see #39 when they aren't. The scheduler framing puts placement quality (fragmentation/utilization), decision latency, fault-domain correctness, and the hierarchical-fleet scaling problem at the centre; the algorithmic-kernel framing brings the C8 min-heap → multi-dimensional-best-fit swap.

Functional requirements (cite functional-requirements.html)

  • Place a workload. Accept (cpu, ram, disk, service_id, anti_affinity?, locality_hint?, priority); assign a physical host with enough free resources, honoring hard constraints; return the host (and a reservation). Slowest acceptable: p99 < 2 s.
  • Honor constraints & preferences. Hard: fits on host (no RAM overcommit), anti-affinity (replicas of one service not co-located). Soft (scored): CPU overcommit up to 2:1, rack-spread for fault tolerance, data locality, resource balance.
  • Honor priority & preempt. When the cluster can't host a higher-priority request but a lower-priority running workload's resources would let it fit, preempt the lower-priority one (drain / migrate / kill-and-reschedule), place the high-priority one, re-queue the victim. (Modeled in Iter 4/5; for VMs "preempt" = live-migrate where possible, evict-and-reschedule where not.)
  • Free on delete / failure. When a workload is deleted, or its host fails, or its node agent reports it dead, reclaim its resources into the free pool and (on host failure) re-place the surviving workloads.
  • Report state. A request's status (queued / placed / failed); a host's resource state; a fleet view (per-rack/per-DC free capacity, utilization, fragmentation). Operators can drain a host/rack, set per-tenant priority quotas.

Out of scope (named, not silently dropped): running / monitoring the workload itself (hypervisor + guest OS own that); the image/volume provisioning system (we trigger a boot and trust storage); workload autoscaling (request sizes are fixed at submit time; elastic VMs that resize are a probe); chargeback / billing (priority tiers are an internal proxy); the network fabric / SDN programming (we assume the SDN re-routes on migration). Flagged assumptions (would confirm with the interviewer): (1) memory cannot be overcommitted — if guest-OS ballooning / KSM page-sharing is in play, RAM becomes soft and the math changes; (2) 2:1 CPU overcommit is safe — assumes workloads aren't correlated (Black Friday breaks this); (3) live migration works for most VMs — pinned VMs (GPU passthrough, SR-IOV, DPDK) can't migrate and create permanent fragmentation; (4) servers are homogeneous (96 cores / 384 GB / 2 TB) — a multi-generation fleet needs the scoring function to normalize across SKUs (a probe); (5) "30% wasted" is fragmentation, not headroom — some slack is intentional (burst capacity, rolling-deploy buffer); we treat the target utilization (85–88%) as the headroom.

The question is: how do you pack VMs onto servers to maximize utilization while respecting fault domains and anti-affinity rules? This is a variant of the multi-dimensional bin packing problem, which is NP-hard. We need good heuristics.

Naive Design: Random Placement

The simplest scheduler: pick a random server that has enough resources, and place the VM there.

# Naive random placement
import random

def place_vm(vm, servers):
    """Pick a random server that has enough resources."""
    eligible = [
        s for s in servers
        if s.free_cpu >= vm.cpu
        and s.free_ram >= vm.ram
        and s.free_disk >= vm.disk
    ]
    if not eligible:
        raise NoCapacityError("No server has enough resources")
    return random.choice(eligible)

Why Random Seems OK at First

  • Simple: No state to maintain, no scoring, O(N) filter + O(1) pick.
  • Fast: Placement takes microseconds.
  • Statistically balanced: With many placements, load should average out... right?

At low utilization (30-40%), random placement works. Servers have plenty of headroom and fragmentation does not matter. The problems emerge as utilization climbs past 60%.

What Breaks

Random placement causes three serious problems at high utilization.

Problem 1: Resource Fragmentation

A server has 96 cores and 384 GB RAM.
After random placement, a typical server might have:
  - 3 medium VMs (12 cores, 48 GB) + 2 large VMs (32 cores, 128 GB)
  - Used:  44 cores, 176 GB RAM
  - Free:  52 cores, 208 GB RAM

Can we fit an x-large VM (32 cores, 128 GB)? Yes.
Can we fit TWO x-large VMs? No (64 > 52 cores).

Now imagine 10,000 servers each with ~50 free cores but in
awkward combinations. Total "free" CPU: 500,000 cores.
But we can't place a workload that needs 64 cores on any
of them — every server has the resources scattered across
multiple dimensions in a way that doesn't fit.

This is fragmentation: resources exist but are unusable.
At 80% cluster utilization, fragmentation wastes 15-30% of
the remaining capacity — equivalent to 2,000-4,000 servers.

Problem 2: No Fault-Domain Awareness

A customer deploys 3 replicas of their database.
Random placement might put all 3 on the same rack.

Probability of all 3 on the same rack (500 racks, random):
  First replica: any rack.
  Second on same rack: 99/49,999 ≈ 0.2%
  Third on same rack:  98/49,998 ≈ 0.2%
  Combined: ~0.2% × 0.2% = 4×10⁻⁶ per customer.

But with 50,000 customers each running 3 replicas:
  Expected customers with all replicas on same rack:
  ~50,000 × 4×10⁻⁶ ≈ 0.2 customers.

Seems rare — but when that rack loses power, that customer
has a TOTAL outage. And rack failures happen: 2-3 per year
across 500 racks. Over a year, this is almost certain to hit
someone. One customer losing all replicas simultaneously is
an SLA violation and a front-page incident.

Problem 3: Scheduling Conflicts at Scale

Multiple schedulers running in parallel (for HA) can race:

  T=0ms   Scheduler A reads Server 17: 32 cores free
  T=1ms   Scheduler B reads Server 17: 32 cores free
  T=2ms   Scheduler A places 16-core VM on Server 17 → 16 free
  T=3ms   Scheduler B places 16-core VM on Server 17 → 0 free
  T=4ms   Scheduler B places ANOTHER 16-core VM on Server 17
          → -16 cores! OVERCOMMITTED (not the good kind)

Without coordination, parallel schedulers can double-book
resources. This causes VMs to fight for CPU/RAM, leading to
performance degradation or OOM kills.

The fundamental tension: Good placement requires global knowledge (what is running where), but centralized scheduling is a single point of failure. We need placement quality close to a centralized optimizer, with the availability of a distributed system.

Iteration 1: Filter-and-Score Scheduler

Instead of random selection, use a two-phase approach inspired by the Kubernetes scheduler: first filter out ineligible servers, then score the remaining ones and pick the best.

Scoring Functions

def score_server(server, vm, all_placements):
    """Score a server for placing this VM. Higher = better."""
    score = 0.0

    # 1. Best-fit: prefer servers where the VM fits tightly
    #    (minimizes leftover fragments)
    cpu_remaining = (server.free_cpu - vm.cpu) / server.total_cpu
    ram_remaining = (server.free_ram - vm.ram) / server.total_ram
    # Lower remaining = tighter fit = higher score
    fit_score = 1.0 - max(cpu_remaining, ram_remaining)
    score += 0.40 * fit_score

    # 2. Rack spread: prefer racks with fewer of this
    #    customer's VMs (anti-affinity)
    same_rack_count = count_customer_vms_on_rack(
        server.rack_id, vm.customer_id, all_placements
    )
    spread_score = 1.0 / (1 + same_rack_count)
    score += 0.25 * spread_score

    # 3. Resource balance: prefer servers where CPU and RAM
    #    utilization are close (avoids stranded resources)
    cpu_util = (server.total_cpu - server.free_cpu + vm.cpu) / server.total_cpu
    ram_util = (server.total_ram - server.free_ram + vm.ram) / server.total_ram
    balance_score = 1.0 - abs(cpu_util - ram_util)
    score += 0.20 * balance_score

    # 4. Data locality: prefer servers near the VM's storage
    locality_score = 1.0 if server.rack_id == vm.preferred_rack else 0.0
    score += 0.15 * locality_score

    return score

Why Best-Fit Beats Random

Example: Server has 48 cores free. Two VMs to place: 32-core and 16-core.

Random: Might place 16-core first → 32 free.
  Then 32-core fits → 0 free. Lucky!
  Or might place both elsewhere, wasting this server.

Best-fit: Prefers this server for the 32-core VM (tightest fit).
  Then places 16-core VM → 0 free. Deterministic.

At scale (50K servers, 500K VMs):
  Random placement: ~65% average utilization (35% wasted)
  Best-fit scoring: ~78% average utilization (22% wasted)

Improvement: 13 percentage points = 6,500 fewer servers needed.
At $10K/server/year = $65M/year saved.

Fragmentation Reduction

Best-fit + balance scoring reduces fragmentation by grouping
similar-sized VMs together and keeping CPU/RAM ratios aligned.

Before (random):
  Server A: 52 cores free, 24 GB RAM free
  → Can't fit any Large VM (needs 64 GB RAM)
  → Can fit 12 Small VMs, but wastes 40 cores

After (best-fit + balance):
  Server A: 32 cores free, 128 GB RAM free
  → Can fit 2 Large VMs perfectly (32 cores, 128 GB)

The balance score prevents "stranded resources" — servers
where one dimension is full but others have plenty left.

This is how Kubernetes schedules pods. The kube-scheduler uses exactly this filter-then-score pipeline. Filters (called "predicates") check hard constraints. Scoring functions (called "priorities") rank the remaining nodes. The weights are configurable. Our version adds multi-dimensional resource awareness and rack-level anti-affinity.

What breaks (C5; cite failure-modes.html)

  • The scheduler scans all 50K hosts from the DB on every request (a scan-heavy access pattern on the data store) — ~50 ms/request and ~200K row-reads/sec at peak. Fine at 4 req/sec; the cliff is ~10× growth or a recovery burst. → fixed in Iter 2 (in-RAM node cache).
  • Two schedulers double-book a host — no coordination, so concurrent placements race and over-commit (the bad kind). → fixed in Iter 2 (CAS / optimistic concurrency); see lock-contention for why a global lock isn't the answer.
  • The single scheduler is a SPOF — coordinator-bottleneck; one process whose per-decision work grows with the fleet, and whose loss stops all placements fleet-wide. → fixed in Iter 3 (cells + leader-elected schedulers; see failure-modes#leader-loss).
  • The node-state in the DB drifts from reality — a host crashes or a VM dies and the scheduler still thinks the resources are taken (or free). data-corruption of the cache-of-reality. → mitigated by the periodic full-sync reconciliation introduced alongside the cache in Iter 2, and the orphan-reaper.

Iteration 2: Cached State and Optimistic Concurrency

Iteration 1 scans all 50,000 servers on every placement. This works at 4 req/sec, but we want headroom and high availability (multiple schedulers). Two problems to solve: efficiency and concurrency.

Problem: Scanning 50K Servers Per Request

Each placement:
  Read 50K server records from DB: ~50ms (batch read)
  Filter to eligible: ~5ms
  Score eligible (~8K): ~1ms
  Total: ~56ms per placement

At 4 req/sec: 56ms × 4 = 224ms of compute per second. Fine.
At 40 req/sec (10x growth): 56ms × 40 = 2.2 seconds. Still OK.
At 400 req/sec (during burst or recovery): problem.

But the real issue is DB load:
  4 requests/sec × 50K rows per request = 200K row reads/sec
  That is a lot of DB traffic for a scheduling decision.

Solution: In-Memory Server State Cache

The scheduler maintains an in-memory cache of all server states.

Cache size: 50,000 servers × ~200 bytes each = 10 MB
  (server_id, free_cpu, free_ram, free_disk, rack_id, dc_id)

Update strategy:
  - On placement: update cache immediately (optimistic)
  - On VM deletion: receive event from VM lifecycle service
  - Periodic full sync: every 30 seconds, reconcile with DB
  - On conflict: re-read from DB, re-attempt placement

With in-memory cache:
  Filter 50K servers: ~1ms (array scan, no DB)
  Score 8K eligible: ~1ms
  Total: ~2ms per placement (28x faster)

Optimistic Concurrency for Multiple Schedulers

Compare-and-Swap (CAS) Update

def place_vm_with_cas(vm, cache, db):
    """Place VM using cached state + CAS for safety."""
    for attempt in range(3):  # max 3 retries
        # 1. Filter and score from in-memory cache
        eligible = filter_servers(vm, cache)
        best = max(eligible, key=lambda s: score_server(s, vm))

        # 2. Attempt atomic placement in DB
        success = db.compare_and_swap(
            table="servers",
            key=best.server_id,
            expected_version=best.version,
            updates={
                "free_cpu": best.free_cpu - vm.cpu,
                "free_ram": best.free_ram - vm.ram,
                "free_disk": best.free_disk - vm.disk,
            }
        )

        if success:
            # 3. Update local cache
            cache.update(best.server_id, vm)
            return best

        # 4. Conflict — refresh cache for this server, retry
        cache.refresh(best.server_id, db)

    raise PlacementError("Failed after 3 attempts")

Why Optimistic Beats Pessimistic Locking

Pessimistic (lock the server row during scoring):
  Lock acquired → score 8K servers → pick one → update → release
  Lock held for ~5ms. Other schedulers blocked during this time.
  At 4 req/sec: 20ms of lock contention per second. Tolerable.
  At 40 req/sec: 200ms. Schedulers queue up. Latency spikes.

Optimistic (CAS on write):
  No locks during read/score. Only contend at write time.
  Write takes ~1ms. Conflict only if two schedulers pick the
  SAME server within ~1ms of each other.
  With 50K servers, P(same server) ≈ 1/50,000 per pair.
  At 4 req/sec with 3 schedulers: virtually zero conflicts.

Optimistic concurrency scales linearly with scheduler count.

🔀 Alternatives considered (C4)

  • Optimistic concurrency (CAS on write) — chosen. No locks during the scan/score; conflict only if two schedulers pick the same host within ~1 ms (P ≈ 1/50,000 per pair). Scales linearly with scheduler count. Cite api-design-patterns#concurrency.
  • Pessimistic locking (lock the candidate host row during scoring) — serializes placement; lock held ~5 ms; tolerable at 4 req/sec, queues up at 40+. Rejected — doesn't scale with scheduler replicas.
  • A single coordinator with a placement queue (all requests funnel through one scheduler; no concurrency at all) — trivially correct, but a coordinator-bottleneck and a SPOF; this is what Iter 3's cells decompose. Rejected for HA.
  • Two-level offers (Mesos-style) — a resource-master makes offers of free resources to per-framework schedulers; no shared state. Avoids the cache-drift problem but at the cost of offer-staleness and poor cross-framework packing (each framework sees only its offers). Considered; we keep a shared cached state + CAS because we want global packing quality. (Re-examined as a C4 alternative in Iter 3.)

What breaks (C5; cite failure-modes.html)

  • Even the in-RAM scan of 50K hosts is wasteful past ~100K servers — the cache stays cheap (10 MB) but the per-decision scan grows linearly with fleet size; the coordinator-bottleneck is structural, not a cache problem. → fixed in Iter 3 (cells: each scheduler scans only its slice).
  • Cache drift under churn — a missed VM-deletion event or a silent host failure leaves the cache thinking resources are taken/free; the 30-s full-sync bounds the staleness but doesn't eliminate the window. A black-hole node — one that accepts placements but doesn't actually run the VM — is the nasty case; mitigated by node-agent heartbeats + an orphan-reaper that reclaims allocations whose lease lapsed.
  • The state DB is a SPOF — if the single PostgreSQL is down, no placements (CAS can't commit). → per-cell DB in Iter 3 limits the blast radius to one cell; the DB itself runs leader+replica (cite replication-decisions#single-sync).
  • Retry storms on a hot host-class — if a popular SKU (say, the GPU hosts) is the only one that fits a wave of similar requests, every scheduler races for the same few rows — a hot-partition-style contention spike. Mitigated by jittered retry + the scoring function spreading across racks; the deeper fix (a reservation/admission step) is a probe.

Iteration 3: Cell-Based Hierarchical Scheduling

Iteration 2 works well, but as the fleet grows past 100K servers, even scanning an in-memory cache of all servers becomes wasteful. Most placements don't need global knowledge. We partition the fleet into cells.

How It Works

  1. Global scheduler receives the VM request. It picks the best cell (not server) based on: which DC is closest to the customer, which cells have capacity, and anti-affinity at the DC level.
  2. Cell scheduler receives the request within its cell. It runs filter-and-score on only its 10K servers (not 50K). Placement decisions are local and fast.
  3. Overflow: If a cell scheduler cannot place the VM (cell is full), it returns a rejection. The global scheduler tries the next-best cell.

Performance Improvement

Before (single-tier, 50K servers):
  Filter: scan 50K servers = ~5ms
  Score:  ~8K eligible = ~1ms
  Total:  ~6ms

After (cell-based, 10K servers per cell):
  Global scheduler: pick cell = ~0.1ms (compare 5 cells)
  Cell scheduler: scan 10K servers = ~1ms
  Score: ~1.6K eligible = ~0.2ms
  Total: ~1.3ms (4.6x faster)

But the real win is isolation:
  - Cell scheduler crash only affects 1/5 of the fleet
  - Cell state cache is 2 MB instead of 10 MB
  - Cell scheduler can be colocated with its servers (low latency)
  - Each cell can run 3 scheduler replicas independently

Blast Radius Containment

Bug in scoring function causes bad placements:

Single scheduler: Affects all 50,000 servers. Every VM placed
  during the bug window gets a bad placement. Blast radius: 100%.

Cell-based: Bug rolls out to Cell 1 first (canary).
  Only 10,000 servers affected. If metrics degrade, halt rollout.
  Blast radius: 20%. Other cells continue with the old version.

This is why Google Borg and AWS use cell-based architectures.
Each cell is an independent failure domain for the control plane.

Cell sizing rule of thumb: A cell should be large enough that most placements can be satisfied within it (avoiding cross-cell overflow), but small enough that a cell-level outage is survivable. 5,000-20,000 servers per cell is typical. Google Borg uses ~10,000 machines per cell.

Iteration 4: Background Defragmentation

Even with best-fit scoring, fragmentation accumulates over time as VMs are created and deleted in unpredictable patterns. We need an active defragmentation process.

How Fragmentation Accumulates

Day 1: Server has 4 medium VMs (16 cores, 64 GB each).
  Used: 64 cores, 256 GB. Free: 32 cores, 128 GB.

Day 3: Customer deletes 2 medium VMs (middle ones).
  Used: 32 cores, 128 GB. Free: 64 cores, 256 GB.

Day 5: New large VM request (16 cores, 64 GB). Placed here.
  Used: 48 cores, 192 GB. Free: 48 cores, 192 GB.

Day 7: Another medium deleted.
  Used: 32 cores, 128 GB. Free: 64 cores, 256 GB.
  But: 1 large + 1 medium remain, leaving irregular gaps.

Over weeks, servers accumulate "Swiss cheese" patterns:
  Server A: 30 cores, 48 GB free (can't fit Large)
  Server B: 22 cores, 180 GB free (can't fit X-Large by CPU)
  Server C: 64 cores, 12 GB free (can't fit anything useful by RAM)

Total free: 116 cores, 240 GB.
Could fit: 3 Large VMs (48 cores, 192 GB) if consolidated.
Actually fits: 0 Large VMs. All fragmented.

The Defragmenter

A background process that live-migrates VMs to consolidate free
resources. Runs continuously at low priority.

Algorithm:
  1. Find "source" servers with low utilization (<30%).
  2. For each VM on the source, find a "target" server where
     it would improve overall packing (higher score).
  3. Live-migrate the VM: copy memory pages to target while VM
     runs, brief pause (<100ms) to copy final dirty pages,
     resume on target. Total downtime: <100ms.
  4. Once source server is empty, mark it as "drainable"
     (available for maintenance, power-off, or new workloads).

Constraints:
  - Max 1 migration per server at a time (migration is I/O heavy)
  - Max 50 concurrent migrations per cell (limit network impact)
  - Never migrate during customer-defined maintenance windows
  - Respect anti-affinity rules at all times

Live Migration Mechanics

Migrating a medium VM (4 cores, 16 GB RAM):

Phase 1 — Pre-copy (VM keeps running on source):
  Copy all 16 GB of memory to target.
  Speed: 10 Gbps network → 16 GB / 1.25 GB/sec = ~13 seconds.
  During copy, VM dirties ~200 MB of pages (write rate: ~15 MB/s).

Phase 2 — Iterative re-copy:
  Copy the 200 MB of dirty pages. Takes ~0.2 seconds.
  During this, ~3 MB more dirtied.
  Copy 3 MB. Takes ~3 ms. Almost converged.

Phase 3 — Stop-and-copy:
  Pause VM. Copy final ~50 KB of dirty pages. <1 ms.
  Update network routing (ARP, SDN rules). ~10 ms.
  Resume VM on target. Total pause: ~50-100 ms.

Customer impact: <100 ms of pause. Most applications don't notice.
Connections survive (MAC address unchanged, SDN handles routing).

Defragmentation Impact

Without defragmentation:
  Cluster utilization plateaus at 75-78% (fragmentation ceiling)
  15-20% of servers have "stranded" resources

With continuous defragmentation:
  Cluster utilization sustained at 85-88%
  Recovered capacity: 10 percentage points = 5,000 servers
  At $10K/server/year = $50M/year in recovered value

Migration budget:
  ~500 migrations/day across the cluster (1% of VMs)
  Network cost: 500 × 16 GB = 8 TB/day
  On a 10 Gbps backbone, this is ~6.4 seconds of full bandwidth.
  Negligible compared to total inter-rack traffic.

🔀 Alternatives considered (C4)

  • Continuous greedy live-migration — chosen. Drain low-utilization servers one VM at a time; cheap (<100 ms pause/VM), incremental, self-throttling. Locally-greedy → can be globally suboptimal → that's exactly the gap Iter 5's ILP closes.
  • No defragmentation — accept the ceiling. Utilization plateaus at 75–78% regardless of placement quality; ~5,000 servers ($50M/yr) left on the table. Rejected — the migration budget (~500/day, ~8 TB/day, ~6 s of backbone) is trivial against $50M/yr.
  • Preempt-and-reschedule instead of live-migrate — kill the VM, re-place it. Works for VMs that can't be live-migrated (GPU passthrough, SR-IOV), but causes customer-visible downtime — reserved for pinned VMs and for preemption (a higher-priority workload needs the host now), not for routine defrag.
  • Gang-only / batched migration — move several VMs as one atomic plan. Closer to optimal but heavier; this is the ILP path (Iter 5) — greedy single-VM moves are the cheap continuous baseline, batched-optimal moves are the periodic cleanup.

What breaks (C5; cite failure-modes.html)

  • Migration storms — an aggressive defragmenter (or a bug in the "improves packing?" test) live-migrates far more than the ~500/day budget, saturating inter-rack links and degrading every VM's network. Bounded by the hard caps (1 migration/server, 50 concurrent/cell) and a circuit-breaker on aggregate migration bandwidth; a runaway defragmenter is a bad-deploy-class incident — canary the defrag policy too.
  • Migration thrash — VM A migrated to host X, then later migrated off X to make room for VM B; locally each move scored well, the sequence churned. The greedy defragmenter can't see this; Iter 5's ILP plans the whole batch and avoids it.
  • Pinned-VM fragmentation — GPU/SR-IOV/DPDK VMs can't be live-migrated, so the servers they sit on can never be drained; permanent fragments accumulate. Mitigated by placing pinned VMs onto a dedicated pool (a hard constraint at placement time) so they don't fragment the general pool.
  • A migration that won't converge — a write-heavy VM dirties memory faster than the pre-copy can ship it (write rate > ~bandwidth); the iterative re-copy never gets small enough for a sub-100 ms stop-and-copy. Detected by a max-iterations cap → either accept a longer pause, or abort the migration (the VM stays put, defrag picks another).

Iteration 5: Optimal Placement with Integer Linear Programming

The greedy defragmenter from Iteration 4 moves VMs one at a time. Each move looks locally good, but the sequence can be globally suboptimal — it might move VM A to Server X, only to later need to move VM B off Server X to make room for VM C. What if we could compute the mathematically optimal rebalancing plan — the minimum number of migrations that achieves the best packing? This is where Integer Linear Programming (ILP) enters the picture.

Linear Programming (LP) — The Foundation

Before we get to ILP, we need to understand Linear Programming (LP). LP is a mathematical technique for finding the best outcome (maximum profit, minimum cost, etc.) when you have limited resources and choices to make.

The Core Idea

Every LP problem has three ingredients:

  1. Decision variables — the things you control. “How much of X should I produce? How many workers should I assign?”
  2. An objective function — what you want to maximize or minimize. It must be a linear formula (just additions and multiplications by constants — no squares, no exponents, no if statements).
  3. Constraints — the limits you cannot violate. Each constraint is a linear inequality (e.g., 2x + 3y ≤ 120).

“Linear” means the objective and every constraint are straight lines (or flat planes in higher dimensions). No curves, no multiplication of two variables together.

Concrete Example: A Small Factory

Suppose a factory makes chairs and tables:

  • A chair needs 2 hours of wood-shop and 1 hour of finishing. Profit: $40.
  • A table needs 3 hours of wood-shop and 2 hours of finishing. Profit: $70.
  • The wood shop has 120 hours/week available.
  • Finishing has 80 hours/week available.

Question: how many chairs and tables should the factory make per week to maximize profit?

Translating to Math

Decision variables:
  x = number of chairs per week
  y = number of tables per week

Objective (maximize):
  Profit = 40x + 70y          ← linear: just additions & multiplications

Constraints:
  2x + 3y ≤ 120               ← wood-shop hours limit
  1x + 2y ≤  80               ← finishing hours limit
  x ≥ 0,  y ≥ 0               ← can't make negative furniture

That's it. This is a complete LP formulation.

How LP Solves It — The Feasible Region

Each constraint cuts the xy-plane into a "half":
  2x + 3y ≤ 120  →  everything below/left of the line 2x+3y=120
  x + 2y  ≤ 80   →  everything below/left of the line x+2y=80
  x ≥ 0, y ≥ 0   →  first quadrant only

The overlap of all these halves is a polygon called the
FEASIBLE REGION — every point inside it is a valid plan.

The corners (vertices) of this polygon are:
  A = (0, 0)        →  Profit = $0
  B = (60, 0)       →  Profit = 40×60 = $2,400
  C = (24, 24)      →  Profit = 40×24 + 70×24 = $2,640   ← BEST
  D = (0, 40)       →  Profit = 70×40 = $2,800

Wait, let's check D. Does (0, 40) satisfy both constraints?
  Wood-shop:  2(0) + 3(40) = 120 ≤ 120  ✓
  Finishing:  0 + 2(40) = 80 ≤ 80       ✓
  Profit = $2,800.  That's higher than C!

Let me recheck C = (24, 24):
  Wood-shop:  2(24) + 3(24) = 48+72 = 120  ✓
  Finishing:  24 + 2(24) = 24+48 = 72 ≤ 80 ✓
  Profit = 960 + 1680 = $2,640

So actually D = (0, 40) gives $2,800. Is there a better corner?
The intersection of 2x+3y=120 and x+2y=80:
  Multiply 2nd equation by 2: 2x+4y=160
  Subtract 1st: y = 40, x = 80-2(40) = 0
  That IS point D. So D is the best.

Optimal plan: make 0 chairs and 40 tables → $2,800/week profit.

Key Insight: Why Check Only Corners?

A fundamental theorem of LP says: if an optimal solution exists, at least one optimal solution is at a vertex (corner) of the feasible region. Why? Because the objective function is a flat plane. As you slide this plane in the “better” direction, the last place it touches the feasible polygon is always a corner (or an entire edge, in which case both endpoints are equally optimal).

This means LP solvers don’t need to check every possible point — they jump between vertices, improving the objective at each step. The Simplex algorithm does exactly this and typically solves problems with thousands of variables in seconds.

Integer Linear Programming (ILP) — When Fractions Don’t Make Sense

In the factory example, the LP solution said “make 0 chairs and 40 tables.” Those are whole numbers — we got lucky. But what if the LP solution was “make 23.7 chairs and 18.4 tables”? You can’t make 0.7 of a chair. You need integer solutions.

Why Not Just Round the LP Solution?

Suppose LP says: x = 23.7, y = 18.4

Rounding options:
  Round down (23, 18): Feasible? Maybe. Optimal? No — leaves
    resources on the table.
  Round up (24, 19): Feasible? Maybe not — might violate a
    constraint.
  Round nearest (24, 18): Feasible? Maybe. Optimal? Not
    necessarily — could be far from the true integer optimum.

In fact, the gap between the LP optimum and the true integer
optimum can be ARBITRARILY large. Rounding is a gamble.

For resource scheduling, "x = 0.7" means "assign 70% of a VM
to this server." That's meaningless. A VM is either on a server
or it isn't. We NEED integer solutions.

ILP = LP + “Variables Must Be Integers”

An ILP adds one line to the LP formulation:
  x, y ∈ {0, 1, 2, 3, ...}      ← integers only

This single addition makes the problem MUCH harder:
  LP:  solvable in polynomial time (fast, even for millions of
       variables)
  ILP: NP-hard in general (no known fast algorithm for all cases)

Why is it harder? The feasible region is no longer a smooth
polygon — it's a scatter of integer dots inside the polygon.
The simplex trick of walking along edges doesn't work on
scattered dots. Solvers must use techniques like:
  - Branch and bound: systematically explore subproblems
  - Cutting planes: add extra constraints to tighten the LP
    relaxation
  - Branch and cut: combine both approaches

Modern solvers (Gurobi, CPLEX, OR-Tools) use all of these and
can solve ILPs with 10,000+ variables in seconds to minutes.

Binary Variables — The Yes/No Workhorse

A special case of integer variables is binary (0/1) variables. They represent yes/no decisions:

x_ij = 1  if VM i is assigned to Server j
x_ij = 0  if VM i is NOT assigned to Server j

This is exactly what resource scheduling needs. Each placement
decision is binary: a VM is on a server, or it isn't.

For 6 VMs and 3 servers, we have 6 × 3 = 18 binary variables:
  x_11, x_12, x_13    (VM 1's assignment)
  x_21, x_22, x_23    (VM 2's assignment)
  ...
  x_61, x_62, x_63    (VM 6's assignment)

Mixed Integer Linear Programming (MILP)

Sometimes you need both integer and continuous (fractional) variables in the same problem. That’s a Mixed Integer Linear Program (MILP).

When Do You Need Mixed Variables?

In our resource scheduler:
  Integer (binary) variables:
    x_ij ∈ {0, 1} — "is VM i on Server j?" (can't be 0.5)

  Continuous variables:
    u_j ∈ [0, 1] — "what fraction of Server j's CPU is used?"
    b   ∈ [0, 1] — "the worst-case utilization imbalance"
    m_i ∈ {0, 1} — "was VM i migrated?"  (also binary)

You might want to minimize total migrations (integer count)
while also minimizing utilization imbalance (continuous measure).
The placement decisions are binary, but the utilization metrics
are continuous. This makes it a MILP.

MILP is also NP-hard (it includes ILP as a special case), but
modern solvers handle it well for moderate sizes.

The Scenario: Optimal Batch Rebalancing

The greedy defragmenter (Iteration 4) moves VMs one at a time: find a low-utilization server, pick a VM, move it somewhere better, repeat. This is fast but can produce suboptimal plans. Here’s a case where greedy fails and ILP shines:

Why Greedy Defragmentation Fails

3 servers, each with 32 cores total. Current state:

  Server 1:  VM-A (8 cores) + VM-B (8 cores)  → 16 used, 16 free
  Server 2:  VM-C (24 cores)                   → 24 used,  8 free
  Server 3:  VM-D (8 cores)                    → 8 used,  24 free

Goal: consolidate to free up entire servers (so we can power
them off for maintenance or cost savings).

Greedy approach (move VMs off lowest-utilized server first):
  Step 1: Server 3 is lowest (25% used). Move VM-D (8 cores).
    Where? Server 1 has 16 free → fits.  Server 2 has 8 free → fits.
    Greedy picks Server 2 (tightest fit).
    → Server 2: VM-C(24) + VM-D(8) = 32 used, 0 free.  FULL.
    → Server 3: empty. ✓ Can power off.

  Result: 1 server freed, 2 migrations possible but greedy
  stopped after emptying Server 3.

  But we can't do better… or CAN we?

  What if we had moved differently:
    Move VM-D(8) to Server 1.  Server 1: A(8)+B(8)+D(8)=24, 8 free.
    Move VM-A(8) to Server 2.  Server 2: C(24)+A(8)=32, 0 free.
    Move VM-B(8) to Server 2.  Doesn't fit (0 free). Stuck.

  So greedy actually did OK here. Let's see a harder case...

Formulating the Rebalancing Problem as ILP

Let’s build the ILP formulation step by step. We’ll first define it for a small example (6 VMs, 3 servers), then show the general form.

Step 1: Decision Variables

For each VM i and each server j, define:

  x_ij = 1  if VM i is assigned to Server j in the NEW plan
  x_ij = 0  otherwise

We also know where each VM is RIGHT NOW. Let's call that:

  curr_ij = 1  if VM i is CURRENTLY on Server j
  curr_ij = 0  otherwise
  (these are constants, not variables — we know the current state)

From these, we can derive:
  m_i = 1 if VM i is migrated (its server changed)
      = 0 if VM i stays where it is

  m_i = 1 - x_i,curr(i)
  where curr(i) is VM i's current server.
  (If x_ij = 1 for the current server, no migration. Otherwise, migration.)

Concrete example — 6 VMs, 3 servers:
  Variables: x_11, x_12, x_13, x_21, x_22, ..., x_63
  Total: 6 × 3 = 18 binary variables

Step 2: Constraints

Constraint 1: Each VM must be on exactly one server.
  For each VM i:  x_i1 + x_i2 + x_i3 = 1

  "VM 1 is on Server 1 OR Server 2 OR Server 3, but not two."

  Example for VM 1:  x_11 + x_12 + x_13 = 1
  This gives us 6 constraints (one per VM).


Constraint 2: Server CPU capacity cannot be exceeded.
  For each server j:
    Σ (cpu_i × x_ij) ≤ capacity_j
    for all VMs i

  In English: "The total CPU of all VMs assigned to Server j
  must not exceed Server j's capacity."

  Example for Server 1 (32 cores):
    cpu_1·x_11 + cpu_2·x_21 + cpu_3·x_31
    + cpu_4·x_41 + cpu_5·x_51 + cpu_6·x_61 ≤ 32

  This gives us 3 constraints (one per server).
  (Add similar constraints for RAM, disk if needed.)


Constraint 3: Anti-affinity — VMs of the same service can't
share a server.
  If VM 2 and VM 5 are replicas of the same service:
    x_2j + x_5j ≤ 1   for each server j

  "At most one of {VM 2, VM 5} can be on Server j."
  This gives us 3 constraints (one per server per anti-affinity pair).


All constraints are LINEAR:
  - Each is a sum of variables times constants, compared to a constant.
  - No x_ij × x_kl terms. No x_ij². Just additions.

Step 3: Objective Function

What do we want to optimize? Two common objectives:

Objective A — Minimize number of migrations:
  Minimize  Σ m_i  =  Σ (1 - x_i,curr(i))

  For each VM, we pay a cost of 1 if it moves. We want the plan
  that achieves our goals with the fewest moves.


Objective B — Minimize number of active servers:
  Introduce a new binary variable for each server:
    s_j = 1 if server j has ANY VMs assigned to it
    s_j = 0 if server j is completely empty

  Link s_j to the x variables:
    x_ij ≤ s_j   for all VMs i, server j
    "If any VM is on server j, then s_j must be 1."

  Minimize  Σ s_j
  "Use as few servers as possible" = bin packing!


Objective C — Multi-objective (weighted combination):
  Minimize  α × (number of active servers)
          + β × (number of migrations)

  α = 100 (heavily penalize using an extra server)
  β = 1   (lightly penalize each migration)

  This says: "Free up servers aggressively, but prefer plans
  with fewer migrations when multiple plans free the same
  number of servers."

Worked Example: 6 VMs on 3 Servers

Let’s trace through a complete ILP formulation and solve it by hand for a small instance.

Setup

3 servers, each with 32 CPU cores:
  Server 1 (S1): 32 cores capacity
  Server 2 (S2): 32 cores capacity
  Server 3 (S3): 32 cores capacity

6 VMs with their CPU demands and current placement:
  VM-A:  8 cores,  currently on S1
  VM-B: 12 cores,  currently on S1
  VM-C:  4 cores,  currently on S2
  VM-D: 16 cores,  currently on S2
  VM-E:  8 cores,  currently on S3
  VM-F: 12 cores,  currently on S3

Current state:
  S1: VM-A(8) + VM-B(12)  = 20 used, 12 free
  S2: VM-C(4) + VM-D(16)  = 20 used, 12 free
  S3: VM-E(8) + VM-F(12)  = 20 used, 12 free

Total demand: 8+12+4+16+8+12 = 60 cores
Total capacity: 3 × 32 = 96 cores
Theoretical minimum servers: ⌈60/32⌉ = 2 servers

So it should be POSSIBLE to pack all 6 VMs onto just 2 servers,
freeing 1 server entirely. But can the ILP find a valid packing?

Full ILP Formulation

Variables:
  x_A1, x_A2, x_A3   (VM-A's assignment: 1 if on that server)
  x_B1, x_B2, x_B3   (VM-B)
  x_C1, x_C2, x_C3   (VM-C)
  x_D1, x_D2, x_D3   (VM-D)
  x_E1, x_E2, x_E3   (VM-E)
  x_F1, x_F2, x_F3   (VM-F)
  s_1,  s_2,  s_3     (1 if server is active)
  All variables ∈ {0, 1}

Objective: Minimize 100·(s_1 + s_2 + s_3) + 1·(migrations)
  where migrations = (1-x_A1) + (1-x_B1) + (1-x_C2) + (1-x_D2)
                    + (1-x_E3) + (1-x_F3)
  (each VM pays cost 1 if it moves from its current server)

Assignment constraints (each VM on exactly 1 server):
  x_A1 + x_A2 + x_A3 = 1
  x_B1 + x_B2 + x_B3 = 1
  x_C1 + x_C2 + x_C3 = 1
  x_D1 + x_D2 + x_D3 = 1
  x_E1 + x_E2 + x_E3 = 1
  x_F1 + x_F2 + x_F3 = 1

Capacity constraints (CPU must fit):
  8·x_A1 + 12·x_B1 + 4·x_C1 + 16·x_D1 + 8·x_E1 + 12·x_F1 ≤ 32
  8·x_A2 + 12·x_B2 + 4·x_C2 + 16·x_D2 + 8·x_E2 + 12·x_F2 ≤ 32
  8·x_A3 + 12·x_B3 + 4·x_C3 + 16·x_D3 + 8·x_E3 + 12·x_F3 ≤ 32

Server-active linking (if any VM on server j, then s_j = 1):
  x_A1 ≤ s_1,  x_B1 ≤ s_1,  x_C1 ≤ s_1,  ...  (6 per server)
  x_A2 ≤ s_2,  x_B2 ≤ s_2,  ...
  x_A3 ≤ s_3,  x_B3 ≤ s_3,  ...

Total: 18 + 3 + 6 + 3 + 18 = 48 constraints, 21 variables.

Solving It (Step by Step)

The ILP solver explores possible assignments. Let's trace its
reasoning (simplified — real solvers use branch-and-bound):

Can we achieve s_1+s_2+s_3 = 2 (only 2 active servers)?
  Total demand = 60 cores. Two servers have 64 cores. Feasible!

Try packing everything onto S1 and S2 (freeing S3):
  S1 capacity: 32.  S2 capacity: 32.  Need to fit 60 cores total.

  Try S1: VM-D(16) + VM-B(12) + VM-C(4) = 32. Full!  ✓
  Try S2: VM-A(8) + VM-F(12) + VM-E(8) = 28. Fits!   ✓
  Total used: 32 + 28 = 60. S3 is empty.

  Migrations needed:
    VM-A: currently S1, assigned S1? No, assigned S1. Wait...
    Let me re-check. VM-D is currently on S2, assigned to S1 → migrated.
    VM-B stays on S1. VM-C currently on S2, assigned to S1 → migrated.
    VM-A currently on S1, assigned to S2 → migrated.
    VM-F currently on S3, assigned to S2 → migrated.
    VM-E currently on S3, assigned to S2 → migrated.

    Migrations: VM-D, VM-C, VM-A, VM-F, VM-E = 5 migrations.

Can we do it with fewer migrations? Try keeping more VMs in place:
  Keep VM-A(8), VM-B(12) on S1 → 20 used, 12 free.
  Can we fit remaining 40 cores on S2 alone? 40 > 32. No.

  Keep VM-C(4), VM-D(16) on S2 → 20 used, 12 free.
  Need to fit VM-A(8)+VM-B(12)+VM-E(8)+VM-F(12) = 40 across S1+S2.
  S1 has 32. S2 has 12 free. Total free = 44. Fits in principle.
  S1: VM-A(8)+VM-B(12)+VM-E(8) = 28 ≤ 32 ✓ (moved VM-E to S1)
  S2: VM-C(4)+VM-D(16)+VM-F(12) = 32 ≤ 32 ✓ (moved VM-F to S2)
  Migrations: VM-E (S3→S1) + VM-F (S3→S2) = only 2 migrations!

ILP OPTIMAL SOLUTION:
  S1: VM-A(8) + VM-B(12) + VM-E(8) = 28 cores
  S2: VM-C(4) + VM-D(16) + VM-F(12) = 32 cores
  S3: empty (powered off) ← saves $10K/year
  Migrations: 2 (VM-E and VM-F move off S3)
  Objective: 100×2 + 1×2 = 202

No plan can do better: we need at least 2 active servers,
and at least 2 migrations to empty S3.

Extending to MILP: Balancing Utilization

The ILP above minimizes servers and migrations. But the winning plan leaves S1 at 87.5% and S2 at 100%. That’s unbalanced — S2 has zero headroom for bursts. We can add a continuous variable to balance utilization, making it a MILP.

Adding a Balance Objective

New continuous variable:
  u_max ∈ [0, 1] — the highest utilization across active servers

New constraints:
  For each server j:
    (total CPU assigned to j) / 32 ≤ u_max

  In linear form (multiply both sides by 32):
    8·x_Aj + 12·x_Bj + 4·x_Cj + 16·x_Dj + 8·x_Ej + 12·x_Fj
    ≤ 32 · u_max

Updated objective (MILP — mixed integer + continuous):
  Minimize 100·(s_1+s_2+s_3) + 1·(migrations) + 10·u_max

This penalizes plans where any server is near 100%.
Now the solver might prefer:
  S1: VM-A(8)+VM-B(12)+VM-D(16) = 36. Doesn't fit!

  S1: VM-A(8)+VM-B(12)+VM-C(4)+VM-E(8) = 32  (100%)
  S2: VM-D(16)+VM-F(12) = 28  (87.5%)
  Migrations: VM-C(S2→S1), VM-E(S3→S1), VM-F(S3→S2) = 3
  u_max = 1.0.  Objective = 200 + 3 + 10 = 213.

  vs. our earlier plan:
  S1: VM-A(8)+VM-B(12)+VM-E(8) = 28  (87.5%)
  S2: VM-C(4)+VM-D(16)+VM-F(12) = 32  (100%)
  u_max = 1.0.  Objective = 200 + 2 + 10 = 212.

Both have u_max = 1.0, so balance penalty is the same.
Can we find a plan where no server exceeds, say, 93.75% (30/32)?

  S1: VM-A(8)+VM-D(16)+VM-C(4) = 28  (87.5%)
  S2: VM-B(12)+VM-E(8)+VM-F(12) = 32  (100%). Still 100%.

With 60 cores and 64 capacity, at least one server must hold
≥ 30 cores (93.75%). We can't avoid u_max ≥ 0.875.

The MILP correctly identifies this tradeoff: using only 2
servers forces high utilization. If we allow 3 servers:
  u_max = 20/32 = 0.625.  Objective = 300 + 0 + 6.25 = 306.25.
  More servers (higher cost) but better balance.

The weight α=100 ensures the solver strongly prefers fewer
servers unless the utilization penalty is extreme.

Implementation with Google OR-Tools

Here’s a complete, runnable Python implementation using Google’s free ortools solver. This is what the background defragmenter would run periodically on each cell.

from ortools.linear_solver import pywraplp

def solve_rebalancing(vms, servers):
    """
    Find the optimal VM-to-server assignment that minimizes
    active servers and migrations using ILP.

    Args:
        vms: list of dicts with 'id', 'cpu', 'ram', 'current_server', 'service_group'
        servers: list of dicts with 'id', 'cpu_capacity', 'ram_capacity'
    Returns:
        dict mapping vm_id → server_id (the new assignment)
    """
    solver = pywraplp.Solver.CreateSolver("SCIP")  # free ILP solver
    if not solver:
        raise RuntimeError("SCIP solver not available")

    n_vms = len(vms)
    n_servers = len(servers)

    # --- Decision variables ---
    # x[i][j] = 1 if VM i is assigned to Server j
    x = {}
    for i in range(n_vms):
        for j in range(n_servers):
            x[i, j] = solver.IntVar(0, 1, f"x_{i}_{j}")

    # s[j] = 1 if Server j has any VM assigned (is "active")
    s = {}
    for j in range(n_servers):
        s[j] = solver.IntVar(0, 1, f"s_{j}")

    # --- Constraints ---
    # 1. Each VM on exactly one server
    for i in range(n_vms):
        solver.Add(
            sum(x[i, j] for j in range(n_servers)) == 1
        )

    # 2. CPU capacity
    for j in range(n_servers):
        solver.Add(
            sum(vms[i]["cpu"] * x[i, j] for i in range(n_vms))
            <= servers[j]["cpu_capacity"]
        )

    # 3. RAM capacity
    for j in range(n_servers):
        solver.Add(
            sum(vms[i]["ram"] * x[i, j] for i in range(n_vms))
            <= servers[j]["ram_capacity"]
        )

    # 4. Link server-active variable: x[i][j] <= s[j]
    for i in range(n_vms):
        for j in range(n_servers):
            solver.Add(x[i, j] <= s[j])

    # 5. Anti-affinity: VMs in the same service_group can't share a server
    from collections import defaultdict
    groups = defaultdict(list)
    for i, vm in enumerate(vms):
        if vm.get("service_group"):
            groups[vm["service_group"]].append(i)
    for group_vms in groups.values():
        for j in range(n_servers):
            solver.Add(
                sum(x[i, j] for i in group_vms) <= 1
            )

    # --- Objective ---
    # Minimize: 100 * (active servers) + 1 * (migrations)
    migrations = []
    for i, vm in enumerate(vms):
        curr_j = next(
            j for j, srv in enumerate(servers)
            if srv["id"] == vm["current_server"]
        )
        # Migration cost = 1 if VM moves (i.e., x[i][curr_j] = 0)
        migrations.append(1 - x[i, curr_j])

    solver.Minimize(
        100 * sum(s[j] for j in range(n_servers))
        + 1 * sum(migrations)
    )

    # --- Solve ---
    solver.SetTimeLimit(30_000)  # 30-second timeout
    status = solver.Solve()

    if status not in (pywraplp.Solver.OPTIMAL, pywraplp.Solver.FEASIBLE):
        raise RuntimeError("No feasible solution found")

    # --- Extract solution ---
    assignment = {}
    for i, vm in enumerate(vms):
        for j in range(n_servers):
            if x[i, j].solution_value() > 0.5:
                assignment[vm["id"]] = servers[j]["id"]
    return assignment

Scaling ILP in Practice

Problem: A cell has 10,000 servers and 100,000 VMs.
  ILP variables: 100,000 × 10,000 = 1 BILLION binary variables.
  No solver can handle this. We need decomposition strategies:

Strategy 1 — Scope reduction:
  Only consider VMs on servers with < 40% utilization (candidates
  for consolidation). If 2,000 servers qualify, with 20,000 VMs,
  we have 20,000 × 2,000 = 40M variables. Still too many.

Strategy 2 — Column generation (restrict target servers):
  Each VM can only move to servers in the same rack or nearby
  racks (say, 200 servers). Variables: 20,000 × 200 = 4M.
  Getting closer but still large.

Strategy 3 — Decompose by rack:
  Solve each rack independently (100 servers, ~1,000 VMs).
  Variables: 1,000 × 100 = 100,000. Solves in seconds!
  Then solve cross-rack moves separately.

Strategy 4 — Rolling horizon:
  Solve for 500 VMs at a time. Apply migrations. Repeat.
  Each batch: 500 × 500 = 250,000 variables. Solves in < 1 second.
  Run 40 batches to cover 20,000 VMs. Total: ~40 seconds.

In practice, Strategy 3 or 4 gives near-optimal results.
Google's Borg uses LP relaxation + rounding (not pure ILP)
for batch scheduling, which is another viable approach.

ILP in the real world: Google uses LP/ILP techniques in Borg for batch workload placement. Microsoft Azure uses MILP for VM defragmentation. AWS uses constraint programming (a cousin of ILP) for placement optimization. These are always run as background optimization passes, not on the real-time placement path. The real-time path uses the fast heuristic (filter-and-score); the ILP runs periodically to clean up what the heuristic left behind.

🔀 Alternatives considered (C4) — exact vs approximate batch rebalancing

  • ILP / MILP on a decomposed scope — chosen. Optimal within the batch; decomposed by rack / rolling-horizon so it solves in seconds (per-rack ~100K vars). The strategies in "Scaling ILP in Practice" above are the C4 sub-options (scope reduction / column generation / decompose-by-rack / rolling-horizon).
  • Pure greedy (Iter 4) only — ~80% of optimal, milliseconds, no solver dependency. Good enough on the hot path; on the periodic batch pass, the gap (extra freed servers = $10K/yr each) justifies the solver.
  • LP relaxation + rounding — solve the LP (polynomial), round fractional assignments, repair constraint violations. What Borg actually does for batch. Faster than ILP, near-optimal in practice; a valid alternative to pure ILP — the page uses ILP for pedagogical clarity but LP+rounding is the production-pragmatic choice.
  • Constraint programming (CP-SAT) — what AWS uses; better for the messy logical constraints (licensing, affinity webs) that linearize awkwardly in ILP. Equivalent expressive power for this problem; tool choice, not a different design.

What breaks (C5; cite failure-modes.html)

  • The solver times out — a batch too large or a pathological instance; the 30-s time-limit returns the best feasible solution found (or none). Falls back to the greedy defragmenter for that batch; never blocks — the ILP is advisory, not on any critical path.
  • The solver's plan races reality — the cluster state changed between "snapshot → solve" and "apply migrations"; a target host filled up. Each migration in the plan is re-validated against current state at apply-time (and CAS'd, like a normal placement); a stale step is dropped, not forced.
  • Migration cascade from a big plan — an optimal plan with hundreds of moves, applied too fast, is itself a migration storm (see Iter 4's What-Breaks). The plan is fed through the same migration-bandwidth caps; "optimal packing" doesn't get to bypass the throttle.
  • Backlog growth on the pending queue — if placements fail faster than the cluster can free capacity (genuine capacity exhaustion, not fragmentation), no rebalancer helps; the pending-request queue grows unbounded — backlog-overflow / backlog-growth. The fix isn't algorithmic — it's admission control (shed/delay low-priority requests, signal "buy more servers") + preemption of low-priority workloads; the queue depth and the placement-failure rate are the "out of capacity" alarm.

X3DRDurable state is small — per-cell Postgres (replicated, PITR) + cell-scheduler leases in etcd (replicated across AZs). A region loss reconstructs from those + node-agent re-registration; RPO ≈ one Postgres commit (placement state is transactional); RTO ≈ seconds (warm-standby cell scheduler) / minutes (cold rebuild from Postgres + heartbeat reconcile). Running VMs don't lose state in a scheduler outage — the hypervisor keeps them alive. Quarterly drill: kill a cell scheduler + its standby, watch cold rebuild. Cite cross-cutting#dr.
X4CapacityThe scheduler fleet autoscales on per-cell scheduler CPU + pending-queue depth (subdivide a hot cell if a scheduler runs hot); +30% headroom on control-plane nodes (a deploy + an AZ loss promoting standbys). Cold-start of a cell scheduler = rebuild from Postgres + reconcile heartbeats (~seconds). The real capacity question is the server fleet's: the placement-failure rate + pending-queue trend + per-DC utilization are the "buy more servers" feed; the server fleet runs at ~85–88% by design — the headroom is the slack above that (burst, rolling deploys, DR re-placement). Cite cross-cutting#capacity.

Cell sizing: big enough that most placements stay in-cell, small enough that a cell outage is survivable (5K–20K typical). Cite cross-cutting#cells.

Degradation contract (the final design's failure posture)

When a cell scheduler dies → its warm standby promotes in ~seconds (etcd-fenced); only new placements/migrations/preemptions in that cell pause for that window; running VMs keep running (the hypervisor doesn't need the scheduler); standby-also-gone → cold rebuild from the cell Postgres + heartbeat reconcile. When the cell Postgres is degraded → that cell can't CAS-commit placements → it stops accepting new placements (the router routes elsewhere) and keeps serving reads from its in-RAM state; running VMs untouched. When the global router is down → new submissions are rejected (caller retries; nothing was written); already-routed requests are still served by their cells. When the pending queue grows faster than capacity frees → admission control sheds/delays low-priority requests (with a reason — never silent), keeps high-priority flowing, signals capacity pressure, and preemption reclaims low-priority workloads' hosts. When a preempted/migrated VM won't drain → the deadline escalates (graceful → force-evict → fence the host, page). When the node-state cache drifts → the periodic full-sync + heartbeat reconcile catches it; a host whose heartbeat lapsed is marked down and its allocations re-placed; a black-hole host (accepts but doesn't run) is caught by the accepted-vs-running gap and quarantined. The system never double-allocates a host, never silently drops a placement request, never lets a priority-0 workload starve indefinitely (it preempts), and never exceeds the 2-s decision SLO (cell-bounded scan). Cite failure-modes#leader-loss + #backlog-overflow + #black-hole-node + #data-corruption.

Operations (C6)

  • Metrics (SLIs → NFRs). Cluster utilization — the headline: allocated CPU/RAM/disk vs total, per DC and per cell; plus fragmentation = (free capacity that can't host the largest queued request) / total free — the "wasted hardware" gauge. Placement decision latency p99 per cell vs the 2-s SLO (and the filter-vs-score breakdown). Pending queue depth by priority + the placement-failure rate — the "out of capacity" alarm. CAS conflict rate (rises with scheduler count / hot SKUs — the contention signal). Migration rate, pause-ms p99, aggregate migration bandwidth (the storm guard). Preemption rate, per-workload preempt-count (anti-thrash), priority-inversion incidents. Cache health — full-sync drift count, stale-heartbeat / marked-down rate, accepted-vs-running gap per host (black-hole detector). Availability — cell-scheduler promotion rate, placement-pause-per-cell, orphan-reap rate. Router — cell-selection retry rate, per-cell capacity-gauge staleness.
  • Rollout. A new cell-scheduler binary canaries on one cell first — ideally on that cell's standby in shadow mode (reads the lease + DB, computes placements, diffs against the live primary for 24 h, doesn't act); then promote it (it becomes primary; old binary becomes standby — instant rollback); roll cell-by-cell. The router + admission controller are stateless — blue-green behind the LB. Rollback triggers: a shadow-decision divergence, any regression in placement-decision p99 / utilization / CAS-conflict rate, any rise in migration bandwidth or the accepted-vs-running gap. The defrag policy and the scoring weights are config, not code — they get the same canary-one-cell-first treatment (a bad weight is a slow-burning utilization regression).
  • Migration (single-scheduler → cellular — the Iter 2→3 cutover). Stand up the cell schedulers in read-only shadow: each watches its DC's node-agent heartbeats + the placement queue, builds its node-state from current state, computes what it would place — the old single scheduler still owns the fleet. Diff shadow placements vs the old scheduler's for a week; chase divergences (usually a tie-break or a stale-cache edge case). Then cut over DC-by-DC: atomically (via the etcd lease) hand DC 1's servers to cell 1's scheduler — the old scheduler stops managing them, the cell scheduler takes the lease; node agents check the lease epoch on every command, so there's no window where both could act on the same host. Watch a day, proceed; the global router goes live once all DCs are cellular. Keep the old scheduler as a fallback (it can re-take a DC's lease) for a week, then decommission. The risk is entirely the shadow-diff; if it matches for a week, the cutover is boring.
  • Headroom. +30% on control-plane nodes (a binary roll during an AZ loss is routine; the scheduler must not be the thing that falls over when the fleet most needs placing); per-cell Postgres sized for the CAS write rate ×3 + a replica; etcd for the lease churn ×3; the router for ~3× the submission peak. The server fleet runs at ~85–88% utilization by design — that target is the headroom (the ~12–15% slack absorbs burst creates, rolling deploys, DR re-placement, and the cell-boundary tax); a preemptible/spot tier (a probe) soaks any slack above that. Reason: the scheduler's own cost is a rounding error (see below) — over-provision it freely; the fleet's headroom is the expensive knob — tune it deliberately.

Real-system comparison

The closest production cousins: Google Borg (the "Large-scale cluster management at Google with Borg" paper) — the cell model (a Borgmaster per cell of ~10K machines, primary + Paxos-replicated state, a stateless link to machine-resident Borglets), priority tiers, preemption, alloc-sets, and the "scheduler is cheap, the fleet is expensive" economics — this page is essentially Borg for a generic VM fleet; Omega (shared-state, optimistically-concurrent schedulers over a common cell state — the alternative to Borg's monolithic-per-cell that Iter 2's CAS design leans toward); Apache Mesos + Marathon/Aurora (the two-level offer model — a resource-master makes offers to per-framework schedulers; chosen when frameworks are independent and global packing is secondary); Kubernetes scheduler (the filter-and-score "predicates + priorities" pipeline this page's Iter 1 mirrors directly, plus the descheduler for defrag and Volcano for gang scheduling) — what you'd extend if you already run K8s, with OpenAI's "Kubernetes to 7,500 nodes" post documenting the patches the hyperscale regime forces; HashiCorp Nomad (a lighter single-binary scheduler — bin-packing + spread + preemption out of the box, multi-region by federation — the pragmatic mid-scale choice); Hadoop YARN (the central ResourceManager + per-app ApplicationMasters — built for batch analytics, a coordinator-bottleneck for long-lived services at this scale); and on the algorithms side, the multi-dimensional bin-packing literature (best-fit-decreasing, first-fit-decreasing, and the LP-relaxation-plus-rounding that Borg actually uses for batch). Where a real team diverges from this page: most start from something (Borg, K8s, Mesos, Nomad), so the build-vs-adopt call in Iter 3 is usually "extend what we have"; and a real fleet is heterogeneous (multiple SKUs, pinned hosts, partial-failure servers), so the scoring function and the cell map are living things. Anticipated pushbacks

Stress tests against the derivation above — depth rehearsal, not interview rehearsal. If a question can't be answered in 3–6 sentences from the text above, the page is missing a probe.

What if the fleet were 10× tomorrow — 500K servers? (scale flip)

The cell model scales by adding cells: 500K servers at ~10K/cell ≈ ~50 cells, ~100 cell-scheduler nodes — still ~$0.5 M/yr, still rounding error. The per-cell scan is unchanged (each cell is still ~10K hosts). The new pressure is two-fold: (1) the global router now picks among 50 cells — still cheap (a gauge read) but its capacity-gauge fan-in grows, so the gauges become coarse summaries refreshed every few seconds, not live; (2) cross-cell anti-affinity at the DC level (replica-spread across 50 cells) needs the router to coordinate — manageable, but a request that needs N replicas in N distinct cells is the kind of cross-cell decision the router (not a single cell scheduler) must own. The bottleneck stays "the per-decision scan" (cell-bounded, so flat) and "the router's coordination of cross-cell constraints" (grows with cell count, but slowly). The +30% scheduler headroom is unaffected; the server fleet's headroom is still "the ~12–15%."

Why heuristic best-fit instead of optimal bin-packing? (algorithm)

Multi-dimensional vector bin-packing is NP-hard — there is no polynomial-time optimal algorithm, and the cluster state changes (VMs created/deleted at 4/sec) faster than an exact solver finishes. So the hot path uses best-fit-decreasing scoring (within ~10–15% of optimal, microseconds), and the gap is recovered by a background ILP/LP-rounding rebalancer (Iter 5) that has the time budget — it runs nightly per rack, plans the optimal consolidation, frees the servers a greedy mover left stranded ($10K/yr each). The key enabler is that a placement can be corrected later (live-migration is cheap) — that's what makes "good enough now, optimal eventually" the right split. If migration were impossible (every placement permanent), you'd want a much better hot-path algorithm and the ILP wouldn't help — but it isn't, so it does.

What changes for a GPU / topology-aware fleet? (constraint flip → #39)

Everything in the placement formula. On this generic fleet, slots are interchangeable — 4 free cores is 4 free cores regardless of which server — so "fits" is the only placement question and best-fit is about leaving big holes. On a GPU fleet, which GPUs matters: a 1024-GPU training gang scattered across 30 racks in 5 zones trains at ~30–50% of the throughput of one packed into adjacent racks on the same rail fabric — so placement becomes a subtree cover over an NVLink-domain / rail / rack / zone topology tree, the scoring function is topology-weighted, and requests are gangs (all K GPUs together or none — a partial allocation is wasted idle GPUs, and naive first-fit deadlocks under contention). The cells must be topology-coherent (a cell ≥ the largest gang it should host tight), preemption needs checkpoint-awareness, and there's a speculative cross-cell hedging move for the long pole of a big gang. The skeleton (filter-and-score, cells, defrag, ILP, leader-elected schedulers) is the same; every number is re-derived. That's #39.

The scheduler dies right now — what happens? (failure)

A cell-scheduler loss: its warm standby (in another AZ) reads the cell's etcd lease + the cell Postgres + the node-agent heartbeats and promotes in ~seconds, etcd-fenced (the old primary's commands are rejected once the lease lapsed). Only new placements / migrations / preemptions in that one cell pause for the handoff window — ~1/5 of the fleet; running VMs are untouched (the hypervisor keeps them alive without the scheduler). If the standby is also gone, a cold scheduler rebuilds its node-state from the cell Postgres + a heartbeat-reconcile sweep (~seconds-to-minutes). The other 4 cells are entirely unaffected. The global router (stateless ×3) being down is separate: new submissions are rejected (caller retries; nothing was written), already-routed requests still flow. The on-call sees: a placement-pause spike on one cell, a promotion event, and (if cold) a reconcile sweep; runbook: confirm it's a scheduler loss not a partition, let the failover proceed, escalate if the standby didn't promote. Cite failure-modes#leader-loss.

A VM has been stuck "queued" for an hour — where do you look? (monitoring/debug)

Decompose: is it (1) not picked up, (2) picked up and failing, or (3) genuinely no capacity? Check the cell scheduler the router routed it to — is it healthy (placement-decision rate > 0, not mid-failover)? If the scheduler is stuck (a hung scan, a deadlocked CAS-retry loop), that's a scheduler bug — page; the standby should have promoted. If it's healthy but the request keeps failing the filter: which hard constraint? An anti-affinity group with no eligible host (every host in the cell already runs a replica), a locality hint that's unsatisfiable, a size no host can fit (a request bigger than the largest free block — check fragmentation: is the cell fragmented, so a defrag pass would help, or genuinely full?). If genuinely full: is preemption enabled for this priority tier — should it have evicted a lower-priority workload? (If not, that's a priority-config gap.) And is the request stuck in one cell when others have room — should the router's overflow-to-next-cell have fired? Each branch maps to a metric (placement-failure rate by reason, fragmentation per cell, queue depth by priority, router retry rate) and a runbook entry; the "decompose the queue-wait" discipline is so this is a dashboard read, not a guess. Cite bottlenecks#backlog-growth.

Where does the bill come from, and what's the one knob that moves it most? (cost)

~99.95% of the bill is the server fleet (~$500 M/yr of amortized capacity); the scheduler is ~$0.24 M/yr. So the knob that moves the bill is fleet utilization, and the single biggest lever — by a wide margin — is the placement algorithm (Iter 1): random → best-fit took utilization ~65% → ~78% (~$65 M/yr reclaimed — 6,500 fewer servers at $10K each) with one algorithm change and zero new hardware. After that, defragmentation (Iter 4, ~+10 pt to ~85–88%, ~$50 M/yr more) and the ILP cleanup (Iter 5, more of the last ~4,000 servers) are the next levers; cells (Iter 3) cost a small utilization tax (cell-boundary fragmentation) but are non-negotiable for operability. On the system side the only knob worth naming is the warm standby (~$2.6 k/mo for ~second failover vs ~minute) — trivially worth it. The cost-vs-quality knob is real here: the ILP cadence/scope trades solver compute + migration churn for packing tightness — tune it (nightly per rack, not continuous fleet-wide). Cite cost-shapes.html (the "managed-resource-is-the-bill" shape).

How do you roll out a new scheduling policy without a stampede of re-placements? (migration / rollout)

The danger: a new scoring weight (or a new defrag aggressiveness) that the cluster "disagrees" with — the defragmenter wakes up, decides thousands of VMs are now mis-placed, and live-migrates them all at once, saturating inter-rack links and degrading every VM's network. Mitigations stack: (1) the scoring weights and defrag policy are config, canaried on one cell first — watch utilization, migration rate, and the accepted-vs-running gap for 24 h before rolling further; (2) the defragmenter's hard caps (1 migration/server, 50 concurrent/cell) and the aggregate-migration-bandwidth circuit-breaker bound the rate regardless of how "wrong" the policy thinks the world is; (3) the new policy only changes future placements and opportunistic migrations — it never triggers a "reconcile everything now" sweep; existing well-placed VMs stay put. The principle: a policy change is a slow, throttled drift toward the new optimum, not a flash cut. Rollback is reverting the config (instant) + letting the throttled drift undo itself. Cite failure-modes#bad-deploy.

A tenant floods the queue with thousands of requests — fairness? (adversary / fairness)

Two layers. Quota at submit time: a tenant places only within its per-priority-tier max_cpu (the quotas table) — requests above quota get a 403, so a flood is rejected at the door, not queued. Fairness in the queue: the placement queue is ordered by priority, then submit-time — a tenant can't jump the line by submitting more, and within a priority tier a per-tenant rate cap + a max-in-flight prevents one tenant's burst from starving others (a Borg-style "the queue is per-tenant fair-shared, not FIFO" discipline). If a tenant has a legitimate burst within quota that still floods, the pending-queue-depth-by-priority + per-tenant in-flight metrics flag it, and admission control delays (not drops) the overflow with a reason. The audit story: every placement and preemption is logged with the tenant, so a tenant gaming priority tiers (submitting research work as production) is detectable after the fact. Cite failure-modes#backlog-overflow + #pool-starvation.

Key Takeaways

  • The LC min-heap treats rooms as interchangeable; a fleet's slots aren't — and the set isn't static. "Meeting Rooms II" gives you a min-heap of free-times; a real scheduler needs multi-dimensional vector bin-packing (NP-hard → heuristic best-fit), over a constantly-churning node set (the in-RAM state is a cache of reality, reconciled, never trusted), returning a preemptible reservation, run by a hierarchy of leader-elected cell schedulers. Every one of those is a response to a property the LC problem doesn't have.
  • Resource scheduling is about placement quality, not speed. At 4 placements/sec, even a slow algorithm has time. The difference between random and best-fit placement is ~$65M/year in hardware costs at this scale.
  • Multi-dimensional bin packing is NP-hard. You cannot find the optimal placement on the hot path. But best-fit-decreasing scoring gets within 10-15% of optimal, and a background ILP/LP-rounding pass — which has the time budget — claws back the rest. The split works because placements are correctable later (live-migration is cheap).
  • Fragmentation is the hidden cost. Without active defragmentation, utilization plateaus at 75-78% regardless of how good your placement algorithm is. Live migration is essential for sustained high utilization — and pinned VMs (GPU/SR-IOV) that can't migrate need a dedicated pool so they don't fragment the general one.
  • Cell-based architecture limits blast radius. Partitioning the fleet into independent cells means a scheduler bug, a bad config change, or a DB outage only affects 20% of customers (one cell). This is how Google Borg, AWS, and Azure operate; the cell must be big enough that most placements stay in-cell, small enough that a cell outage is survivable.
  • Optimistic concurrency scales with multiple schedulers. Pessimistic locking serializes placement decisions. CAS-based optimistic concurrency lets multiple schedulers run in parallel with near-zero conflict rate — the same shape as Omega's shared-state model.
  • Anti-affinity must be a first-class constraint. Customers expect replicas to survive single-rack and single-DC failures. This means tracking placement at the rack and DC level, not just the server level — and respecting it during defrag and preemption, not just initial placement.
  • Sometimes the thing you're optimizing isn't your system — it's the resource your system manages. This scheduler costs ~$0.24M/yr; the fleet it places costs ~$500M/yr. So the cost discipline inverts: spend freely on the scheduler (warm standbys, defrag workers, an ILP pool, an extra cell), because one percentage point of fleet utilization is ~$5M/yr — ~20× the whole scheduler fleet's cost — so any scheduler-side spend that buys even ~0.05 pt breaks even (the Iter 1 algorithm change alone bought ~13 pt, ~$65M/yr). "Spend on the scheduler, save on the fleet" — name that ROI before drawing the first box.

Assumptions and When They Break

  • Homogeneous servers: We assumed all servers have the same spec (96 cores, 384 GB). Real fleets have multiple generations with different specs. The scoring function must normalize resources across server types.
  • Stable VM sizes: We assumed fixed tiers. If customers can request arbitrary sizes (e.g., 37 cores, 91 GB), fragmentation worsens significantly. Consider rounding up to standard sizes.
  • CPU overcommit works: 2:1 CPU overcommit assumes VMs don't all burst simultaneously. If workloads are correlated (e.g., Black Friday traffic), overcommit causes noisy-neighbor performance issues.
  • Live migration is feasible: Some workloads (GPU VMs, SR-IOV, DPDK) cannot be live-migrated. These "pinned" VMs create permanent fragmentation that cannot be defragmented.
