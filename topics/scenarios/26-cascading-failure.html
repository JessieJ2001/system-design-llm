Cascading Failure in a Microservice Stack

One service deploys a bad ML model. Within 30 seconds, the entire three-tier architecture is down. Walk through the cascade mechanics, the math behind thread pool exhaustion, and the layered defenses that contain the blast radius.

NALSD Walkthrough • ~15 min read

The Scenario

You operate a three-tier microservice architecture for a product search platform. Each service calls the next in a synchronous chain:

The end-to-end path is sequential: A receives a request and does 5ms of routing/auth work, then calls B synchronously. B does 20ms of its own logic, then calls C synchronously. C runs the recommendation model (50ms) and returns. The total end-to-end latency is 5 + 20 + 50 = 75ms.

The Incident

At 14:32, Service C deploys a new version with an updated ML recommendation model. The new model has a memory leak that triggers increasingly frequent garbage collection pauses. Over the course of 2 minutes, C's latency climbs from 50ms to 10 seconds.

Back-of-Envelope: The Cascade Math

Let's trace the impact of C's latency spike through each layer, using concrete numbers.

Service C: The Origin of the Problem

Before incident:
  C processes 10,000 req/sec
  Each request takes 50ms
  Concurrent requests = 10,000 * 0.05s = 500
  Threads available: 2,000
  Utilization: 500 / 2,000 = 25% — healthy

After incident (C latency = 10s):
  C still receives 10,000 req/sec
  Each request now takes 10,000ms
  Concurrent requests = 10,000 * 10s = 100,000
  Threads available: 2,000
  Thread pool exhausted by factor of 50x

  Actual throughput with 2,000 threads at 10s each:
    2,000 threads / 10s = 200 req/sec

  9,800 req/sec are queued or rejected.

Service B: The First Domino

Service B calls C synchronously. When C is slow, B's threads are stuck waiting for C's response.

Before incident:
  B's total time = 20ms own + 50ms waiting on C = 70ms
  Concurrent requests = 10,000 * 0.07s = 700
  Threads: 2,000. Utilization: 35% — healthy

After incident (C takes 10s):
  B's total time = 20ms own + 10,000ms waiting on C = 10,020ms
  Concurrent requests = 10,000 * 10.02s = 100,200
  Threads: 2,000

  Actual throughput:
    2,000 threads / 10.02s = ~200 req/sec

  B can handle 200 req/sec instead of 10,000.
  9,800 req/sec are queued or rejected.
  That's a 98% drop in capacity.

Service A: The Final Domino

Before incident:
  A's total time = 5ms own + 70ms waiting on B = 75ms
  Concurrent requests = 10,000 * 0.075s = 750
  Threads: 2,000. Utilization: 37.5% — healthy

After incident (B takes 10s due to C):
  A's total time = 5ms own + 10,020ms waiting on B = 10,025ms
  Concurrent requests = 10,000 * 10.025s = 100,250
  Threads: 2,000

  Actual throughput: 2,000 / 10.025 = ~200 req/sec

  A can handle 200 req/sec.
  9,800 req/sec get queued, then timeout, then 503.

The Cascade Timeline

The core mechanic: In a synchronous call chain, a slow downstream service consumes upstream threads. Each upstream service's thread pool becomes a bottleneck at the same rate as the downstream delay, because threads are held for the duration of the downstream call. One slow service can consume all threads in the entire stack.

The Naive Design: No Protection

What's Missing

  • No timeouts: B waits indefinitely for C. A waits indefinitely for B.
  • No circuit breakers: B keeps calling C even though C is clearly failing.
  • No bulkheads: B uses the same thread pool for its own work and for C calls. When C calls consume all threads, B can't even handle requests that don't need C.
  • No load shedding: Every request is accepted and queued, even when the queue is hopelessly long.

The Result

A single bad deployment in Service C takes down the entire platform in 30 seconds.

Iteration 1: Timeouts

The simplest defense: set a timeout on every downstream call. If C doesn't respond within a reasonable time, B gives up and either returns an error or a degraded response.

Choosing the Timeout Value

C's normal latency: 50ms
C's p99 latency: ~150ms (some requests take 3x normal)
C's p99.9 latency: ~300ms

Timeout guideline: 2-4x the p99 latency
  Too short (100ms): clips normal slow requests, causes unnecessary errors
  Too long (5s): threads still held for 5 seconds, cascade still happens
  Good range: 200-500ms

Choose: 200ms timeout on B → C calls

Impact of 200ms Timeout

When C is slow (10s latency):
  Every B → C call times out at 200ms (instead of waiting 10s)
  B thread is held for: 20ms (own work) + 200ms (timeout) = 220ms

  B's throughput with timeouts:
    2,000 threads / 0.22s = ~9,090 req/sec

  Compare to no timeouts:
    2,000 threads / 10.02s = ~200 req/sec

  45x improvement! B can handle nearly full load.

But: every request that reaches B still gets a 200ms penalty (the timeout wait), and then returns an error or degraded response. Users experience 220ms latency with no recommendations, instead of the normal 75ms with recommendations.

Implementation

import requests
from requests.exceptions import Timeout

def get_recommendations(user_id, product_ids):
    try:
        response = requests.get(
            f"http://service-c/recommendations",
            params={"user_id": user_id, "products": product_ids},
            timeout=0.2  # 200ms timeout
        )
        return response.json()
    except Timeout:
        # C is slow — return empty recommendations
        # The product page still works, just without "You might also like"
        return {"recommendations": [], "source": "fallback"}
    except Exception:
        return {"recommendations": [], "source": "fallback"}

Limitation

Timeouts help, but B is still calling C on every request. Every call waits 200ms and fails. That's 10,000 failed calls per second to a service we already know is broken. We're wasting B's time and adding load to C (it still receives the connection attempts). We need to stop calling C entirely when it's down.

Iteration 2: Circuit Breaker

A circuit breaker monitors the success/failure rate of downstream calls. When failures exceed a threshold, it opens the circuit — immediately rejecting calls to C without even attempting the connection. This eliminates the 200ms timeout penalty entirely.

The State Machine

Walk-Through of the Incident with Circuit Breaker

T=0s:   C deploys. Breaker is CLOSED. All calls go through normally.

T=5s:   C latency at 500ms. Timeout fires at 200ms.
        B sees timeouts on calls to C.
        In the last 10 seconds: 100,000 calls to C.
        Failures (timeouts): ~50,000 (calls made after T=3s when C > 200ms)
        Failure rate: ~50%. Threshold is 50%. Breaker is right at the edge.

T=7s:   C latency at 1s. All calls timeout.
        Failure rate in rolling window: 70%. Exceeds 50%.
        *** CIRCUIT BREAKER OPENS ***

T=7s+:  All calls to C are immediately rejected by the breaker.
        B does NOT call C. No 200ms timeout wait. No network call.
        B returns degraded response (no recommendations) in 20ms.

        B's throughput:
          2,000 threads / 0.02s = 100,000 req/sec
          B is actually FASTER than normal because it skips the C call!

T=37s:  30 seconds after opening, breaker enters HALF-OPEN.
        Allows ONE test request through to C.

T=37s:  Test request to C: still times out (C hasn't been fixed yet).
        Breaker goes back to OPEN for another 30 seconds.

T=40s:  C team identifies the memory leak, rolls back to previous version.
        C latency drops back to 50ms.

T=67s:  Breaker enters HALF-OPEN again.
        Test request to C: succeeds in 48ms.
        *** CIRCUIT BREAKER CLOSES ***

T=67s+: All traffic to C resumes. Full functionality restored.
        Users see recommendations again.

Implementation

import time
import threading

class CircuitBreaker:
    def __init__(self, failure_threshold=0.5, window_size=10,
                 min_requests=20, open_timeout=30):
        self.failure_threshold = failure_threshold  # 50%
        self.window_size = window_size              # 10 seconds
        self.min_requests = min_requests            # need at least 20 calls
        self.open_timeout = open_timeout            # 30 seconds in OPEN

        self.state = "CLOSED"
        self.failures = []    # timestamps of failures
        self.successes = []   # timestamps of successes
        self.opened_at = None
        self.lock = threading.Lock()

    def allow_request(self):
        with self.lock:
            if self.state == "CLOSED":
                return True
            elif self.state == "OPEN":
                # Check if timeout has elapsed
                if time.time() - self.opened_at >= self.open_timeout:
                    self.state = "HALF_OPEN"
                    return True  # Allow one test request
                return False  # Reject
            elif self.state == "HALF_OPEN":
                return False  # Only one test request at a time

    def record_success(self):
        with self.lock:
            now = time.time()
            self.successes.append(now)
            if self.state == "HALF_OPEN":
                self.state = "CLOSED"  # Recovery confirmed
                self.failures.clear()
                self.successes.clear()

    def record_failure(self):
        with self.lock:
            now = time.time()
            self.failures.append(now)
            if self.state == "HALF_OPEN":
                self.state = "OPEN"  # Still broken
                self.opened_at = time.time()
                return
            # Check failure rate in window
            self._cleanup_old_entries(now)
            total = len(self.failures) + len(self.successes)
            if total >= self.min_requests:
                failure_rate = len(self.failures) / total
                if failure_rate >= self.failure_threshold:
                    self.state = "OPEN"
                    self.opened_at = time.time()

    def _cleanup_old_entries(self, now):
        cutoff = now - self.window_size
        self.failures = [t for t in self.failures if t > cutoff]
        self.successes = [t for t in self.successes if t > cutoff]


# Usage in Service B
breaker = CircuitBreaker(failure_threshold=0.5, open_timeout=30)

def get_recommendations(user_id, product_ids):
    if not breaker.allow_request():
        # Circuit is OPEN — don't even try calling C
        return {"recommendations": [], "source": "circuit_open"}

    try:
        response = requests.get(
            f"http://service-c/recommendations",
            params={"user_id": user_id, "products": product_ids},
            timeout=0.2
        )
        breaker.record_success()
        return response.json()
    except Exception:
        breaker.record_failure()
        return {"recommendations": [], "source": "fallback"}

Iteration 3: Bulkhead Isolation

Circuit breakers help when C is completely down. But what about partial failures? Suppose C is slow for some request types but fast for others. Or suppose B calls multiple downstream services (C, D, E), and only C is failing. Without isolation, C's failures can consume all of B's threads, starving calls to D and E.

The Ship Bulkhead Analogy

A ship's hull is divided into compartments by bulkheads — watertight walls. If one compartment floods, the bulkheads prevent water from spreading to other compartments. The ship stays afloat even with a breach.

In software, a bulkhead is a separate resource pool for each downstream dependency. If one dependency is slow and its pool fills up, the other pools are unaffected.

Sizing the Bulkhead

How many threads for the C pool?

Normal C call volume: 10,000 req/sec
Normal C latency: 50ms
Normal concurrency: 10,000 * 0.05 = 500 across all 10 B instances
Per instance: 500 / 10 = 50 concurrent C calls

Bulkhead size per instance: 50 threads
  - At normal load: 100% utilized (tight but fine with the circuit breaker
    as overflow protection)
  - When C is slow: all 50 threads stuck. But the other 150 threads in B
    are free for non-C work.
  - With circuit breaker: breaker opens after 7 seconds, 50 threads freed.

Shared Pool vs Bulkheaded Pools

Iteration 4: Load Shedding

Even with timeouts, circuit breakers, and bulkheads, there's one more problem: when the system is overloaded, queued requests pile up. A request that waits 10 seconds in a queue before being processed is useless — the client has already timed out and retried. Processing it wastes server resources on work nobody will receive.

The Problem with Slow Failures

Slow failure path (no load shedding):
  T=0:   Request arrives, enters queue
  T=8s:  Request reaches front of queue, starts processing
  T=8.2s: Request processed, response sent
  T=???:  Client already gave up at T=5s (client timeout)

  Result: Server spent 200ms processing a request nobody wanted.
          In that 200ms, it could have processed a fresh request.

Fast failure path (with load shedding):
  T=0:   Request arrives, queue length > 1,000 (threshold)
  T=0ms: Server immediately returns 503 Service Unavailable

  Result: Client gets a fast failure. Can retry immediately.
          Client retries, maybe hits a different (healthier) server.
          Server spent ~0ms on the rejected request.

CoDel: The Smart Approach

CoDel (pronounced "coddle") was originally designed for network router queues, but applies perfectly to request queues. The idea: measure how long each request has been in the queue. If the minimum queue wait time over the last interval exceeds a target (e.g., 5ms), the queue is persistently overloaded. Start dropping requests from the head (oldest first).

# Simplified CoDel for request queues
class CoDelQueue:
    def __init__(self, target_delay=0.005, interval=0.1):
        self.target_delay = target_delay  # 5ms
        self.interval = interval          # 100ms measurement window
        self.queue = collections.deque()
        self.dropping = False
        self.first_above_time = 0

    def enqueue(self, request):
        request.enqueue_time = time.time()
        self.queue.append(request)

    def dequeue(self):
        if not self.queue:
            return None

        request = self.queue.popleft()
        sojourn_time = time.time() - request.enqueue_time

        if sojourn_time < self.target_delay:
            # Queue is healthy
            self.dropping = False
            self.first_above_time = 0
        else:
            # Request waited too long
            if self.first_above_time == 0:
                self.first_above_time = time.time()
            elif time.time() - self.first_above_time > self.interval:
                # Persistent overload — start dropping
                self.dropping = True

        if self.dropping and sojourn_time > self.target_delay:
            # Drop this request (return 503)
            return None  # Caller sends 503

        return request

  - 30 seconds to complete outage
  - 10+ minutes of total outage (until C is rolled back + pools drain)

Key Takeaways

  1. Cascading failures are fast. A slow downstream service can take down an entire stack in 30 seconds through thread pool exhaustion. The cascade propagates upstream at the speed of timeout accumulation.
  2. The math is simple: concurrent connections = QPS x latency. When latency increases 200x (50ms to 10s), you need 200x more threads. You don't have them. The pool fills. Everything stops.
  3. Timeouts are the minimum viable defense. Every downstream call must have a timeout. A system without timeouts is a system waiting to cascade. Set timeouts at 2-4x the p99 latency.
  4. Circuit breakers prevent wasting time on broken services. When a service is down, stop calling it. Return degraded results. Test periodically. This turns a 200ms penalty (timeout) into a 0ms penalty (instant fallback).
  5. Bulkheads prevent cross-contamination. Isolate thread pools per downstream dependency. One failing service should not consume threads needed for other services.
  6. Fast failure beats slow failure. A client that gets a 503 in 1ms can retry on another server. A client that waits 10 seconds for a timeout has wasted 10 seconds and consumed a thread the entire time. Load shedding ensures fast failures when the system is overloaded.
  7. Defense in depth. No single mechanism is sufficient. Timeouts + circuit breakers + bulkheads + load shedding together provide robust protection. Each layer catches failures the others miss.
  8. Degraded is better than dead. A search page without recommendations is far better than a 503 error page. Design every feature to have a degraded fallback mode.
