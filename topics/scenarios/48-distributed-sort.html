Designing a Distributed Sort of a 100 TB Array

Sort a 100 TB key–value array (a GraySort/TeraSort run, or “sort all of yesterday's events by (user_id, ts)” for a downstream batch consumer) on commodity nodes (32 vCPU, 128 GB RAM, 25 Gbps NIC, 4×4 TB NVMe), wall-clock ≤ 30 min, emitting 1024 range-partitioned sorted files with deterministic ordering, on a spot fleet that can be reclaimed mid-run. We start from the textbook external merge sort on one beefy box — which doesn't even fit the input, would take days of disk IO, and restarts from zero if the box dies — and iterate to the TeraSort shape: sample → range-partition → parallel local sort → ordered concat, with skew-aware split keys, a compressed rack-aware shuffle, speculative re-execution, materialized shuffle + reducer checkpoints for spot-tolerance, and a decentralized coordinator driving a hierarchical two-round shuffle.

NALSD Walkthrough • ~55 min read • LC kernel: external merge sort / k-way merge / bucket sort

LC kernel. External merge sort (and its cousins k-way merge and bucket sort) — the textbook way to sort data that doesn't fit in RAM. On one machine: split the input into chunks small enough to sort in memory; sort each chunk and write it back as a sorted run file; then do a k-way merge of the run files (a min-heap over the run heads) streaming the globally-sorted output. O(n log n) comparisons; the merge of the run files is the bottleneck pass. Bucket sort is the same idea with a known key distribution: route keys to value-range buckets, sort each bucket, concatenate. Four properties that break at 100 TB: (1) fits-on-one-node — 100 TB is ≈780× one node's RAM and ≈6× one node's local NVMe; you cannot hold the input, let alone input + run files + output, on one box, so the sort must distribute — which immediately raises “how do you produce a global order across nodes?” (2) uniform-ish data — bucket sort needs to know the key distribution; real keys (Zipf on user_id) make equal-range buckets wildly uneven → straggler reducers, so you must sample the data to pick split keys and handle heavy keys specially; (3) a reliable machine — a distributed run on a spot fleet has tasks fail and get reclaimed mid-sort, so you need speculative re-execution, a recoverable (materialized) shuffle, and reducer checkpoints — none of which the single-box algorithm has; (4) the merge pass is the cost — once you range-partition, the disjoint per-range files concatenate with no merge at all; the cost moves entirely to the shuffle — every byte crosses the network, bounded by the cluster's bisection bandwidth, not by comparisons. The production cousin is the TeraSort/MapReduce shape: sample → range-partition → parallel local (external) sort → ordered concatenation.

S0. Scenario & Requirements

A data platform must sort a 100 TB key–value array and write the sorted result back to an object store. Two framings, same job: (a) the GraySort/TeraSort benchmark workload — 100 byte records, 10 byte keys, 1 trillion records, sort by key; (b) the production batch shape — “take all of yesterday's events (~100 TB of (user_id, ts, payload)) and produce them sorted by (user_id, ts) so the downstream consumer can scan one user's history without a full pass.” The output is 1024 range-partitioned sorted files: file 0 holds the smallest keys in sorted order, file 1 the next range, … file 1023 the largest — concatenating the 1024 files in order is the globally-sorted result. The cluster is commodity spot capacity: a node can be reclaimed with ~2 min notice mid-run, and the same input must always produce byte-identical output (deterministic ordering, even for ties, even across re-runs). The hard wall: a single beefy box doesn't fit the data and would take days; sorting 100 TB in 30 minutes means roughly ~167 GB/s of sustained aggregate throughput across the fleet — the design exists to find the cheapest fleet that hits that, and the dominant cost is the shuffle: every byte crosses the network exactly once, bounded by the cluster's bisection bandwidth.

System class

Class: batch data pipeline (a one-shot map → shuffle → reduce job), with an algorithmic kernel — see functional-requirements.html#class-pipeline. It is not a queryable service: there is no QPS, no read-latency SLO, no online API hit per request. The pipeline framing puts wall-clock, throughput (aggregate bytes/sec), fault tolerance under reclamation, and determinism at the centre; the algorithmic-kernel framing brings the C8 swap (single-box external merge sort → distributed sample-and-range-partition).

Functional requirements (cite functional-requirements.html#fr-pipeline)

  • Sort the array. Given a 100 TB input in the object store (a set of unsorted shards), produce the records in non-decreasing key order. Comparator is configurable: by key, or by a composite (user_id, ts).
  • Emit range-partitioned output. Write exactly 1024 output files, file i holding a contiguous, sorted key range, ranges disjoint and covering the whole key space, so concatenation = the global sort. A downstream consumer can binary-search to a file and stream one range.
  • Deterministic ordering. Equal keys are ordered by a stable tie-break (input position); the same input always produces the same 1024 files, byte-for-byte, on a re-run — the partition split keys must be reproducible.
  • Survive node reclamation. The job runs to completion despite spot interruptions of mappers, reducers, or the coordinator — without restarting from zero.
  • Report progress. A driver/console shows phase (sample / shuffle / sort / write), per-task status, and ETA, so an operator can tell a straggler from a stuck job.

Out of scope (named, not silently dropped): incremental / streaming sort — this is a one-shot batch job; an “append today's delta to yesterday's sorted output” variant is a probe. Sorting within a database (an ORDER BY over a SQL engine's tables) — that engine owns its own shuffle; this is a bare bytes-in/bytes-out sort. Multi-key secondary indexes / arbitrary projections — this produces the sorted records, not a queryable structure. Encryption-at-rest / column-level masking of the payload — assumed handled by the object store's SSE; named under X1.

Flagged assumptions (would confirm with the interviewer): (1) the input is already in the object store and readable in parallel by many nodes (not a single tape drive); (2) 100 TB is the logical size; if it's gzip'd on disk the de-compressed working set is larger — we size for 100 TB of records; (3) 1024 output files is a fixed downstream contract (a power of two, picked so a consumer can route by the top 10 bits of the key); if it were “as many as you like” we'd pick it from the data size; (4) the 30-min wall-clock is a soft deadline with a hard outer bound (~45 min) — missing it by 20% is a warning, by 2× is a page; (5) keys are comparable bytes (lexicographic) — a custom comparator is a hook, not a redesign.

The fundamental tension: a global sort is inherently all-to-all — you can't know which output file a record belongs in until you know where its key falls in the global order, which means every record must be routed against a globally-agreed set of boundaries, which means every byte crosses the network exactly once. The whole design is: agree on the 1023 boundaries cheaply (sample, don't scan twice), and then move 100 TB across a fabric whose bisection bandwidth is the real ceiling — so “shuffle fewer bytes” (compress, co-locate, hierarchical rounds) and “don't let one straggler or one reclaimed node stall the all-to-all” are the only two levers that matter.

Internal worker protocol (sketch) & SLOs

SLOs (these are wall-clock-budget allocations, not request latencies):
  sample phase    p99 <= ~60 s   (one fast pass drawing a reservoir; the cluster's read bandwidth, not CPU)
  shuffle phase   p99 <= ~20 min  (100 TB / bisection bandwidth -- this IS most of the wall-clock)
  sort+write tail p99 <= ~5 min   (each reducer's in-range external sort + one output PUT)
  coordinator API p99 <= 200 ms  (status reads; tiny)

S3. Back-of-Envelope — the two ratios that govern every iteration

One number sizes the fleet; one ratio decides what's worth optimizing. No optimization techniques here (CLAUDE.md Iron Rule #1) — these are the raw physics; the iterations name the techniques.

Ratio 1 — the work-to-do (sizes the fleet)

A global sort moves every byte exactly 3 times:
  read   D from the object store           = 100 TB
  shuffle D over the network               = 100 TB   (every record goes from its mapper to its range's reducer)
  write  D back to the object store         = 100 TB
  total                                     = ~300 TB of IO + network

Over the wall-clock budget T = 1800 s:
  required aggregate throughput = 300 TB / 1800 s = 166.7 GB/s   (call it ~167 GB/s sustained, fleet-wide)

At ~3 GB/s of NIC per node, the network alone needs:
  167 GB/s / 3 GB/s/node = ~56 nodes        <- the absolute floor, zero headroom, zero stragglers, zero re-runs

Real runs need headroom for: the sample pass, the per-reducer sort tail, speculation duplicates,
5-10% of nodes reclaimed, and the bisection cut being < the summed NIC bandwidth -> ~3-4x the floor:
  ~56 x ~3.5 ~= ~200 nodes   <- the working number this page uses (anchored to the GraySort 100 TB winners,
                                 which used hundreds-to-low-thousands of nodes for minutes-scale 100 TB sorts).

Ratio 2 — shuffle : everything-else (decides what to optimize)

Split the 300 TB by where it happens, and time each on a ~200-node fleet (sum-of-NICs ~= 200 x 3 = 600 GB/s;
bisection ~= 0.5 x 600 = ~300 GB/s; aggregate disk ~= 200 x 6 = 1200 GB/s; object-store read/write ~= many hundreds GB/s):
  read 100 TB from object store  : 100,000 GB / ~500 GB/s (parallel reads)  ~= ~200 s
  shuffle 100 TB over the network: 100,000 GB / ~300 GB/s (bisection)       ~= ~333 s    <- the dominant single phase
  write 100 TB to object store    : 100,000 GB / ~500 GB/s (parallel writes) ~= ~200 s
  sort the data (CPU)             : 100,000 GB / (200 x ~6 GB/s/node)        ~= ~83 s     (cheap; CPU is not the gate)
  sum, if phases were serial      : ~820 s; overlapped (read || shuffle || write pipelined): ~the shuffle dominates.

So: the SHUFFLE is the wall-clock. read/write/sort run concurrently with it on a node that's reading,
sorting, sending, and receiving at once. The two levers are therefore:
  (a) shuffle fewer bytes  -> compress the stream (2-3x), co-locate (skip ~1/M of the bytes), hierarchical rounds (fatter cross-rack transfers)
  (b) don't let the all-to-all stall -> no straggler reducer, no reclaimed node that re-shuffles from zero
Everything in Iterations 1-3 is one of those two.

The trap, named up front: if you hash-partition (route record by hash(key) mod 1024) you get the same 100 TB shuffle and the same per-reducer sort — but the output is not globally sorted: file 0 has a random scatter of keys, just grouped. Concatenating them gives garbage. To get global order from hash partitions you'd need a final k-way merge over all 1024 sorted shards — another full pass over 100 TB. Range-partitioning makes the output trivially concatenable: that's why Iteration 0 picks range, and why it must sample first.

S4. Naive Architecture — the simplest thing that meets the FRs

One beefy box. Run the textbook external merge sort: stream the 100 TB input from the object store through one machine; chop it into ~80 GB chunks that fit in RAM; sort each chunk and write it back to local NVMe as a sorted run file; once all runs are written, do a k-way merge (a min-heap over the run heads) streaming the globally-sorted output; chop the output into 1024 equal-byte slices and PUT each as part-NNNNN.sorted. One process, one disk, no network shuffle, no coordinator. It satisfies every FR — it produces the sorted records, in 1024 files, deterministically (one process, one merge order). The numbers say it's not even buildable, and even if it were, it's ~100× too slow.

Architecture

Connections and line-style legend. Solid slate = the data path (read/write to the object store); dashed gray = local-disk spill/read of run files; dashed red = none here.

  • ① Object store → chunk & in-RAM sort — stream the 100 TB input; fill an ~80 GB buffer; sort it (introsort / radix); that's one chunk.
  • ② chunk & sort → local NVMe — write each sorted chunk back to disk as a run file; ~100 TB / 80 GB ≈ ~1250 run files. (This is where capacity blows up — ~100 TB of run files on a 16 TB disk.)
  • ③ local NVMe → k-way merge — open all ~1250 run files; a min-heap over their current heads emits the next-smallest record; refill from the run it came from; repeat. One linear pass producing the global sort.
  • ④ k-way merge → Object store — chop the sorted stream into 1024 equal-byte slices; PUT each as part-NNNNN.sorted. (Equal-byte, not equal-key-range — the naive box doesn't know the key distribution; this already fails the “each file is a clean key range” spec, but never mind — it never finishes anyway.)

Numbers (the gap)

Capacity (the hard wall):
  one beefy box's local disk = 16 TB (4 x 4 TB NVMe).  Even a fat 24-drive box ~= ~50-100 TB.
  the sort needs, concurrently: input staging (~some TB) + ~100 TB of sorted run files + output staging (~some TB)
  -> ~100+ TB of local disk -> DOESN'T FIT any single box.  (RAM is 128 GB; you can't even hold 0.13% of the input.)
  Workaround: stream from the object store instead of staging -> still need ~100 TB for the run files. Still doesn't fit.
  Real workaround: a multi-PB SAN attached to one box -> now you've built a slow, single-failure-domain "cluster" of one.

Disk IO (even if it fit):
  external merge sort touches the data ~4 x: read input (100 TB) + write runs (100 TB) + read runs (100 TB) + write output (100 TB)
                                          = ~400 TB of disk IO
  one box's disk = ~6 GB/s aggregate (4 NVMe, shared between the read and write legs -> effectively ~3 GB/s useful)
  -> 400 TB / 3 GB/s = 400,000 GB / 3 GB/s ~= 133,000 s ~= ~37 hours    (and that's the optimistic, all-sequential number)
  -> wall-clock target is 1800 s.  Gap: ~74x too slow.   And a fatter SAN doesn't help -- one box's PCIe/memory bus caps you.

CPU (not the gate, but):
  comparison sort of 1e12 records ~= 1e12 x log2(1e12) ~= 1e12 x 40 ~= 4e13 comparisons; at ~1e9 cmp/s/core x 32 cores ~= 3e10/s
  -> ~1300 s of pure compare time -- fits in 1800 s ON ITS OWN, but it's serialized behind 37 h of disk. CPU is not the bottleneck.

Fault tolerance:
  a 37-hour job on one box.  Box dies at hour 30 -> start over from zero.  No checkpoint, no shuffle to re-pull, nothing.
  Spot reclamation makes this strictly impossible: you'd be reclaimed ~18 times during one run.

Determinism:
  one process, one merge order -> deterministic.  PASSES (the one NFR the naive design gets right).

Why it's broken (per failing NFR)

  • Aggregate throughput / wall-clock — FAILS, and capacity FAILS first. ~100 TB of intermediate run files doesn't fit one box's disk — the only single boxes that hold it are a SAN-backed monster that's a single failure domain and still disk-IO-bound. This is the shape-determining failure: the only way out is to split the data across a fleet → it becomes a distributed sort → which immediately raises “how do you produce a global order across the pieces?” Classified memory-capacity-bound (the working set — here, local-disk — exceeds one node and scales with the data, not the request rate); the slowness symptom is disk-io-bound (~400 TB of spill IO at ~3 GB/s = ~37 h).
  • Wall-clock — FAILS, ~74×. ~400 TB of disk IO at one box's ~3 GB/s useful disk bandwidth ≈ ~37 hours vs the 30-minute target. A fatter SAN doesn't fix it — one box's PCIe/memory bus caps the useful throughput; you need many disks and NICs working in parallel.
  • Fault tolerance under reclamation — FAILS. A multi-hour job with no checkpoint; a single box death = start over; on spot you'd be reclaimed ~18× mid-run → the job never completes.
  • Cost per run — FAILS implicitly. A SAN-backed multi-PB single box running for ~37 hours is both expensive and idle most of the time you'd want it (you sort once a day).
  • Correctness & determinism — PASSES. One process, one merge order → the output is the exact sort, ties broken by read order, byte-identical on a re-run. The one thing the naive design gets right — and the distributed version will have to work to keep it.

Cost of naive “at NFR-target load”: there is no buildable single-box answer that meets the 30-min wall-clock — even an idealized SAN monster is ~37 h and a single failure domain. So the baseline is “impossible / ~days,” not a dollar figure. Iteration 0 — The distributed-sort skeleton: sample → range-partition → local sort → ordered concat

System after this iteration

Connections (numbered above) and line-style legend. Solid slate = the data path (read input, shuffle records, write output); dashed gray = control / sample / broadcast; dashed red = none yet.

  • ① Object store → Mappers — the coordinator assigns each mapper a disjoint set of input shards (~D/M each); the mapper streams them.
  • ② Mappers → Coordinator (sample) — on the first pass each mapper draws a reservoir sample of keys (cite sketches#reservoir — a fixed-size uniform sample of a stream of unknown length, no second pass) — ~50k–500k keys per mapper → ~tens of MB merged — and ships it to the coordinator.
  • ③ Coordinator → Mappers (split keys) — the coordinator sorts the merged sample of n keys, picks b_i = sample[round(i · n / 1024)] for i = 1..1023 — the empirical quantiles — so the 1024 ranges hold ~equal record count; broadcasts the 1023 keys to every mapper. (Seeded reservoirs → reproducible split keys → deterministic output.)
  • ④ Mappers → Shuffle — the mapper re-reads its slice; for each record, reducer_id = upper_bound(split_keys, sort_key) (a binary search over 1023 keys = ~10 comparisons); it streams the record to that reducer over the network.

Delta vs Naive: the one box becomes ~200 commodity nodes; the “sort the whole 100 TB” step is replaced by “agree on 1023 boundaries from a sample, then each node sorts only its ~97 GB range”; the k-way merge of run files is replaced by an ordered concatenation of disjoint range files (no merge at all); a coordinator appears to pick the boundaries and assign work. The per-reducer in-range sort is still the LC external merge sort — just on 1/1024 of the data.

Bottleneck

Naive's shape-determining failure is capacity: ~100 TB of intermediate state doesn't fit one node's disk — classified memory-capacity-bound (working set scales with the data, exceeds one node), with the slowness symptom disk-io-bound (~37 h of spill IO). The signal that picks this entry: a per-job working set sized in the same order as the input (100 TB), not in the order of one machine (10s of TB). Per bottleneck-to-decisions#memory-capacity: revisit the partition key (split the data across nodes — which here is also the sharding strategy: range vs hash, the central decision), the data structure (the per-piece sort is still external merge sort — tractable now), and whether an intermediate durable buffer belongs between phases (Iteration 2). Iteration 0 takes the partition-key lever — and the choice of which partitioning is the whole point.

Architectural decisions to revisit

Per bottleneck-to-decisions#memory-capacity, the levers are: partition strategy (range vs hash — this iteration), data structure per partition (external merge sort, unchanged), intermediate persistence (push-streamed shuffle now; materialized later, Iter 2), and coordination (a coordinator picks the boundaries; it's a single point we'll decentralize in Iter 3). The load-bearing one here is the partition strategy — and the answer is forced by the FR “output concatenable to the global sort.”

Alternatives considered (C4)

  • Range-partition via a sampled quantile estimate, then parallel local sort, then ordered concat — chosen. Sample the data, pick 1023 split keys so the 1024 ranges hold ~equal record count, route each record to its range's reducer, sort within the reducer, concatenate. The output is trivially the global sort — no final merge — because the ranges are disjoint. This is the TeraSort / MapReduce-sort-example shape. Cite sharding-strategies#range (range partition; supports ordered scans — here, ordered concatenation).
  • Hash-partition (reducer = hash(key) mod 1024), sort within each shard — rejected (the trap). Same 100 TB shuffle, same per-reducer sort — but the output is not globally sorted: each file is a random scatter of keys, merely grouped. Concatenating them is garbage. To recover global order you'd need a final k-way merge over all 1024 sorted shards — another full ~100 TB pass — which is exactly the expensive merge step range-partitioning avoids. Hash-partition is the right answer for a distributed GROUP BY / aggregation (you want all records with the same key co-located, you don't care about order); it's the wrong answer for a sort. Cite sharding-strategies#hash. (Constraint-flip: 🔀 if the requirement were “group equal keys together, order doesn't matter” instead of “globally sorted output,” hash-partition wins — no sample pass, perfectly uniform load, and you skip the “handle skewed ranges” work in Iteration 1.)
  • No sampling — split the key space into 1024 equal value ranges (e.g., for 64-bit keys, range i = [i·2⁶⁴/1024, (i+1)·2⁶⁴/1024)) — rejected. Works only if keys are uniformly distributed over the value space. Real keys never are (a user_id column is dense in some ranges, empty in others; English text clusters around common prefixes) → some reducers get 10 GB, others 1 TB → the 1-TB reducer is the wall-clock. The whole reason to sample is to make the ranges equal by count, not by value width.
  • An exact two-pass histogram (first pass: count records per fine-grained value bucket; compute exact 1023 boundaries; second pass: shuffle) — rejected. Pass 1 is a full ~100 TB scan just to count — doubling the read cost — for a marginal accuracy gain over a sample. A reservoir sample of ~1–10M keys gives ~few-% bucket skew, well within tolerance (and Iteration 1 corrects the residual skew anyway). Cite sketches#reservoir; the t-digest (cite sketches#t-digest) is a streaming alternative if you can't afford even one buffered sample — here we can, so the simple “sort the merged sample, take every (n/1024)-th key” wins on simplicity and exactness-of-the-estimate.

The chosen move

Insert a sample phase and a range-partition shuffle. Phase 1 (sample): each of M mappers reads its slice of the input, draws a fixed-size reservoir sample of keys (cite sketches#reservoir) with a seed derived from (job_id, mapper_id) — deterministic — and ships it to the coordinator. Phase 2 (boundaries): the coordinator merges the ~M×sample-size keys, sorts them, and picks the 1023 empirical quantiles as the split keys — so the 1024 ranges hold ~equal record count; it broadcasts the 1023 keys. Phase 3 (shuffle & sort): each mapper re-reads its slice, binary-searches each record's key against the 1023 split keys to find its reducer, and streams it there; each of R=1024 reducers (co-located on the M nodes) receives its range's records, runs the LC external merge sort over its ~97 GB (chunk → sorted runs → k-way merge — tractable now), and PUTs one sorted file. Concatenating the 1024 files in range order is the global sort — no final merge, because the ranges are disjoint. Why range, not hash: hash gives co-location of equal keys (good for GROUP BY) but no global order — you'd need a k-way merge over all 1024 sorted shards, another full pass; range gives disjoint ranges → concatenation. Sizing: from S3, the work is ~300 TB of IO+network over 1800 s ≈ ~167 GB/s → ~56 nodes is the network floor; with headroom for the sample pass, the sort tail, and skew → ~200 commodity nodes; R = 1024 reducers is fixed by the output contract (5 reducers per node, on average); the sample is ~M×~250k = ~50M keys ≈ ~0.5 GB — the coordinator sorts it in seconds. Cite sharding-strategies#range and sketches#reservoir.

What breaks (C5; cite failure-modes.html)

  • Lost mapper or reducer (black-hole node). Bottleneck

Iteration 0's wall-clock is ~800 s, of which the shuffle is ~333 s — the single dominant phase — and a skewed range can blow it to hours (one reducer with ~5 TB). Both are the same root: too many bytes through the wrong path — 100 TB across a bisection cut that's half the summed NIC bandwidth, plus a hot key that dumps a disproportionate slice onto one reducer. Classified network-bandwidth-bound (the throughput is set by a network cut, not by CPU or disk — the signal: every byte crosses the network and the bottleneck link is the inter-switch fabric) and hot-partition (a partition-key value with disproportionate load → an uneven reducer). Per bottleneck-to-decisions#network-bandwidth: revisit how much data crosses the cut (compress, co-locate, aggregate-before-shuffle), topology awareness (rack-local routing), and the partition function (skew correction). Co-location is free (a record whose reducer is local never serializes). Rack-awareness is free (intra-rack links aren't oversubscribed). Heavy-key isolation is the standard skew fix (cite bottleneck-to-decisions#hot-partition). The chosen move

Two parts, one root (the all-to-all is too heavy and too lopsided). (a) Skew correction. Each mapper, during the sample pass, also runs a Space-Saving heavy-hitters sketch over its slice's keys (cite sketches#space-saving — bounded memory, tracks the top-k keys by approximate count); it ships the top counters; the coordinator merges them and flags any key whose estimated count exceeds a fair reducer share (~D/1024 records). Each flagged key gets a dedicated reducer (just that key's records), or — for a composite comparator (user_id, ts), where one user_id is hot — gets split across several reducers by ts-range (user_id=X, ts ∈ [t0,t1), [t1,t2), …); the output still concatenates in order because the splits are themselves disjoint ranges. (b) Cheaper bytes through the cut. Compress each mapper→reducer stream with zstd (~2–3× on sort-ish data; ~1–2 GB/s/core decompress — CPU is idle, S3); co-locate (a record whose reducer is on the same node is handed over in-process, never serialized — saves ~1/M of the bytes); rack-aware routing (records to in-rack reducers go over the unblocked intra-rack links; cross-rack traffic — which is what actually contends for the bisection cut — is batched into fewer, fatter streams). Cite cost-shapes#stream-processing. Sizing: effective shuffle bytes on the wire ≈ 100 TB × (1 − 1/M) / ~2.5 ≈ ~40 TB; cross-rack bytes are a fraction of that. The deadline (shuffle ≤ ~20 min) now needs M ≥ 40 TB / (0.5 × 3 GB/s × 1200 s) ≈ ~22 nodes for the network floor → with the same ~3.5× headroom, ~120 nodes meets the deadline (down from ~200) — or, keep ~200 nodes and the wall-clock drops to ~8–9 min. We pick ~120 nodes (cheaper) and accept a slightly tighter wall-clock margin (~11 min).

Iteration 2 — Materialized shuffle + reducer checkpoints + speculative re-execution (run on spot)

System after this iteration

Connections (numbered above) and line-style legend. Solid slate = data path; dashed gray = control / checkpoint; dashed red = failure recovery (re-fetch, reassign, speculate).

  • ① Object store → Mappers — unchanged: each mapper reads its input slice.
  • ② Mappers → Coordinator — the reservoir sample + heavy-hitters (as before), plus a MapDone reporting the size of each of its 1024 (+hot) on-disk partitions; heartbeats every 5 s.
  • ③ Mappers → local disk (materialize) — instead of streaming records straight to reducers, the mapper writes its per-reducer partitions (compressed) to its own NVMe: shuffle/<mapper_id>/part-<reducer_id>.zst. Cost: ~1× D of extra local writes (~833 GB/node; trivial on a 16 TB disk). Benefit: a reducer that dies can re-pull; a mapper that dies after writing can have its partitions re-read by a replacement that re-mounts the disk (or, if the disk is gone, just re-map that mapper's input slice — one node's worth, not the job).
  • ④ Reducers ↔ Mappers (PULL shuffle) — each reducer FETCHes part-<reducer_id>.zst from every mapper's local disk; the mapper serves it as a file read. Pull (not push) is the fault-tolerant choice: the mapper holds the data durably (on its disk) until every reducer has confirmed the fetch, so a transient reducer failure is “fetch again,” not “re-run the map.” (This is Spark's shuffle model; Hadoop MapReduce's too.)
  • ⑤ Reducers → Object store (output) — sort the pulled+decompressed records; write the output file with write-then-rename (PUT to a temp key, then atomic rename to part-NNNNN.sorted) so a partially-written file is never visible — idempotent: a re-run of a reduce task overwrites cleanly. Exactly-once commit: the coordinator records a reducer’s commit only after the rename succeeds.
  • ⑥ Reducers → Object store (checkpoint) — every ~30–60 s a reducer flushes its in-progress state — the sorted runs it has produced so far + a vector of “received bytes up to offset X” per source mapper — to ckpt/<reducer_id>/<ts>/. A reclaimed reducer's replacement loads the latest checkpoint and re-fetches only from offset X onward — it doesn't re-shuffle its whole ~97 GB.
  • ⑦ Coordinator: reassign on loss — a node misses 3 heartbeats → its tasks are marked lost; a map task is reassigned (re-read that slice); a reduce task is reassigned to a fresh node that loads the lost reducer's last checkpoint and continues. One node's worth of re-work, bounded by the checkpoint interval.
  • ⑧ Coordinator: speculate — when a task's elapsed time exceeds, say, ~1.5× the median of its cohort (and there's spare capacity), the coordinator launches a duplicate attempt on another node; whichever attempt finishes first commits; the other is cancelled. MapReduce's “backup tasks” — the same idea as a hedged request: pay a little extra compute to cap the tail.

Delta vs Iter 1: the shuffle goes from push-streamed-and-ephemeral to materialized on the mapper's disk and pulled by the reducer — so a task loss is recoverable from a survivor, not a job restart; reducers checkpoint their progress, so a reclaimed reducer resumes; the coordinator tracks the task DAG, reassigns lost tasks, and speculatively duplicates slow ones. With those, the fleet moves to spot (~3–5× cheaper) — which is the whole point of the materialization. Cost: ~1× D of extra local disk writes (the mapper's materialized partitions) + a few checkpoint PUTs — cheap. The split keys, the per-reducer sort, and the ordered-concat output are unchanged.

Bottleneck

Iteration 1's wall-clock is ~700 s — but two things still blow it: stragglers (one slow disk, a noisy neighbour, a GC pause, a NIC at half speed → one task runs 5× the median → the job's wall-clock = that task) and reclamation (a spot interruption of any mapper or reducer = restart the whole job → on spot, where you'd be reclaimed ~10–20× in 30 min, the job never completes). Both are tail-latency in the “job-finishes-when-the-last-task-does” sense (the signal: completion time = max over tasks, and the max is far above the median), compounded by the fact that the all-or-nothing failure model turns one node loss into a full restart. Per bottleneck-to-decisions#tail-latency: revisit redundant execution (speculate/hedge the slow task), checkpointing (so a lost task resumes, not restarts), and making intermediate state durable (so a survivor can serve it). All three address the same root: the job's completion is held hostage by its slowest/unluckiest task — make a task loss cheap and the tail short.

Architectural decisions to revisit

Per bottleneck-to-decisions#tail-latency: redundant execution (speculative backup tasks — this iteration), intermediate persistence (materialize the mapper→reducer partitions on local disk — this iteration), checkpointing (reducer flushes partial state — this iteration), and commit protocol (write-then-rename, idempotent task output — this iteration). Not touched: the coordinator is still a single point that picks split keys and drives the whole DAG — Iteration 3; and the all-to-all's M×R connection count is still large — also Iteration 3.

Alternatives considered (C4)

  • Materialized shuffle (mapper writes partitions to disk) + pull-based fetch + reducer checkpoints + speculative re-execution — chosen. Materialization makes the intermediate state durable on the producer, so a consumer (reducer) loss is “fetch again” and a producer (mapper) loss is “re-read one node's worth” — not a job restart. Pull (reducer fetches) beats push (mapper streams) for fault tolerance: with push, a reducer dying means re-running every mapper that fed it; with pull, the data sits on the mappers' disks until confirmed. Checkpoints bound the re-work of a reclaimed reducer to one interval. Speculation caps the straggler tail. Cite failure-modes#bad-deploy (a task reclaimed mid-run is the closest mode — an unexpected restart of a worker) and #black-hole-node.
  • Keep the push streaming, just retry the whole job on failure — rejected. On on-demand capacity with rare failures that's almost fine (Iter 0/1 lived with it); on spot it's a non-starter — you'd be reclaimed many times per run and never finish. The materialization cost (~1× D of local writes, ~$0 incremental) is far cheaper than the ~3–5× price difference between spot and on-demand that it unlocks. (Constraint-flip: 🔀 if the cluster were dedicated on-demand and node failures were ~zero over a 30-min run, you'd skip materialization, keep the lighter push streaming, and accept the rare full restart — the materialization is bought specifically for spot-tolerance.)
  • Replicate every reducer (run two copies of each, take the first to finish) — rejected. That's 2× the fleet for the whole run, not just for the slow tasks — ~$180/sort instead of ~$90. Speculation gives you the same tail protection by duplicating only the ~few-% of tasks that are actually slow, for a few-% cost bump. Full replication is what you'd do if you couldn't detect a straggler — but the coordinator can (it has per-task progress).
  • Checkpoint the mappers too (so a mapper can resume mid-read) — rejected as not worth it. A map task is cheap to redo — it's one node's input slice (~833 GB), re-read and re-partitioned in ~minutes — and its output is already materialized once it's done. Checkpointing the read offset would save a few minutes on a rare mid-read reclaim; not worth the complexity. Reducers are the ones worth checkpointing — they accumulate ~97 GB of pulled state that's expensive to re-pull. (Iteration 3's hierarchical shuffle changes the math a bit; revisited there.)

The chosen move

(a) Materialized, pull-based shuffle. Each mapper writes its 1024 (+hot) per-reducer partitions — compressed — to its own local NVMe (shuffle/<mapper_id>/part-<reducer_id>.zst) before any reducer reads them; reducers then pull their partition from every mapper's disk. A reducer that dies re-pulls from the survivors (the data is still on the mappers' disks); a mapper that dies after writing has its partitions re-read by a replacement that re-mounts the disk, or — if the disk is gone with the node — its input slice (one node's worth) is simply re-mapped. (b) Reducer checkpoints. Every ~30–60 s a reducer flushes to the object store: the sorted runs it has produced so far + a per-source “received up to byte X” vector. A reclaimed reducer's replacement loads the latest checkpoint and re-fetches only from X onward — bounding re-work to one checkpoint interval, not its whole ~97 GB. (c) Speculative re-execution. The coordinator tracks per-task progress; a task elapsed > ~1.5× its cohort median (with spare capacity available) gets a duplicate attempt on another node; first to finish commits, the other is cancelled — MapReduce’s “backup tasks,” the batch analogue of a hedged request. (d) Idempotent commit. Reduce output is written to a temp key then atomically renamed; the coordinator records a commit only on rename success — so a re-run, a speculative duplicate, or a reclaimed-and-resumed reducer all produce the same file exactly once. With (a)–(d), the fleet runs on spot (~$0.30–0.45/hr/node instead of ~$1.5) — the materialization is the price of that ~3–5× discount. Sizing: add ~10% extra mappers (~120 → ~130) as standing headroom to absorb reclaimed nodes and run speculative duplicates without slowing the cohort; ~1× D = ~100 TB of extra local writes spread over ~130 nodes = ~770 GB/node — trivial on 16 TB; checkpoint volume per reducer is small (the runs it would re-produce anyway) × a few PUTs ≈ negligible cost. Cite pricing-constants#compute (spot vs on-demand).

What breaks (C5; cite failure-modes.html)

  • A spot reclamation cascade (many nodes pulled at once). Blast radius: if the spot pool yanks 20% of the fleet in one event (a capacity reclaim), that's ~26 nodes' worth of map output and ~5% of reducers gone simultaneously. Detection: a burst of missed heartbeats. Recovery: the lost mappers' slices are re-mapped (the input is durable in the object store) and their on-disk partitions re-produced; the lost reducers resume from their last checkpoints. With ~10% standing headroom this is absorbed but the wall-clock stretches; if the reclaim is > the headroom, the coordinator requests replacement spot (or falls back to on-demand for the stragglers) — the job slows, doesn't fail. Worst case (the whole spot pool evaporates): fall back to on-demand and eat the cost.
  • Mapper local disks fill with materialized partitions. Blast radius: a node, then its reducers. ~770 GB/node of materialized partitions is fine on 16 TB — but if a mapper got an oversized slice (skew in shard sizes, not key distribution) or partitions aren't deleted after all reducers confirm the fetch, the disk fills → the mapper can't write → reducers can't pull. Detection: per-node disk-free metric. Recovery: the coordinator only tells a mapper to delete part-<r>.zst after reducer r has confirmed its fetch (or checkpointed past it); slices are sized by bytes, not shard count, so no mapper gets a 5 TB slice; a near-full disk triggers eager cleanup of confirmed partitions.
  • A speculative duplicate and the original both commit. Blast radius: one output file — potentially two versions, or a half-written one. Detection: the coordinator sees two commit attempts for the same reducer_id. Recovery: write-then-rename makes the rename the commit point, and the coordinator accepts exactly one (compare-and-swap on the manifest entry: first rename wins, the loser's temp file is garbage-collected); since both attempts sort the same input with the same comparator and the same tie-break, the files are byte-identical anyway — the CAS just picks one. No corruption, no duplication.

🔀 Constraint flip — if a STABLE sort weren't required (ties may be reordered)

If equal keys could come out in any order, the input_seq tie-break tag disappears — records are ~10% smaller (no 8-byte sequence number) → ~10% fewer bytes everywhere (read, shuffle, sort, write), so the deadline is met with slightly fewer nodes. But determinism across re-runs still requires a fixed ordering for ties (else the output isn't byte-identical) — so in practice you keep a deterministic tie-break even without a “stable wrt input position” requirement (e.g., break ties by a hash of the payload). The interesting flip is the other way: if you don't need byte-identical re-runs either (just “a valid sort”), you can drop the seed-reproducibility constraint on the sample — mappers can use random reservoirs — which simplifies the coordinator but means two runs of the same input produce different (both correct) split keys and thus different file boundaries.

Iteration 3 — Thin, checkpointed coordinator + per-rack sub-schedulers + a hierarchical two-round shuffle

System after this iteration

Connections (numbered above) and line-style legend. Solid slate = data path; dashed gray = control / sample / broadcast / checkpoint; dashed red = re-elect / failover.

  • ① Object store ↔ Mappers / Reducers — unchanged: read input, write the 1024 sorted files.
  • ② Coordinator → Rack sub-schedulers (plan) — the coordinator's only hot-path job is the merged sample → 1023 split keys + hot-key map + range→reducer assignment; it ships that plan to the rack sub-schedulers and the object-store checkpoint, then steps off the hot path. Its state is <~MB — the split keys, the hot-key map, the rack→reducer layout.
  • ③ Standby coordinator ← checkpoint (re-elect) — a standby holds an etcd lease; if the primary coordinator's lease expires, the standby reads the latest coordinator checkpoint from the object store and resumes — a ~seconds re-elect, not a job restart, because the coordinator's state is tiny and the in-flight work (tasks, materialized partitions) is owned by the workers and rack schedulers, not the coordinator.
  • ④ Rack sub-scheduler ↔ Mappers/Reducers (in-rack) — heartbeats, task assignment, speculation decisions, and disk-cleanup orders are handled per rack by the rack's sub-scheduler (a two-level scheduler — the coordinator delegates within-rack scheduling; cf. the Borg/Omega/cell-scheduler split). Cross-rack reassignments (e.g., a whole rack lost) escalate to the coordinator.
  • ⑤ Mappers → Round-1 shuffle (within rack) — each mapper's materialized 1024 partitions are first shuffled within its own rack: the mappers in rack g exchange partitions so the rack collectively holds, for each reducer r, the rack's combined records for range r — one “rack-partial-partition” per reducer per rack. These are rack-local moves over the unblocked top-of-rack links.
  • ⑥ Round-2 shuffle (between racks) → Reducers — now each reducer r pulls one fat transfer per rack (the rack's combined records for range r) instead of one thin transfer per mapper. Cross-rack streams: R_racks × 1024 instead of M × 1024 — e.g., ~10 racks × 1024 = ~10k fat flows vs ~130 × 1024 = ~133k thin flows (and millions at thousand-node scale). The spine switches carry a few fat flows efficiently; they choke on a flood of tiny ones (per-connection overhead, slow-start, head-of-line). Same ~40 TB on the wire — but the bisection cut is now actually saturated instead of fragmented.
  • ⑦ Reducers ↔ Object store (checkpoint + commit) — unchanged: periodic partial-run checkpoints; write-then-rename of the final file; CAS commit.

Delta vs Iter 2: the coordinator goes from “drives the whole DAG, single point, loss = restart” to “picks the boundaries, checkpoints a tiny state, delegates scheduling to per-rack sub-schedulers, re-elects in seconds”; the shuffle goes from a flat M×1024 all-to-all (~133k thin cross-rack flows) to a two-round hierarchy (within-rack, then ~R_racks×1024 fat cross-rack flows). The byte volume on the wire is unchanged (~40 TB); what changes is that the bisection cut carries fat flows it can saturate, the coordinator is no longer a SPOF or a scaling gate, and a rack loss has a smaller, better-isolated blast radius. The per-reducer sort and ordered-concat output are unchanged. Bottleneck budget exhausted (Naive + 4 iterations).

Bottleneck

Iteration 2's wall-clock is ~11–12 min and the job survives reclamation — but two structural weaknesses remain. (1) The coordinator is a single point and a scaling gate: its loss restarts the job (its state isn't checkpointed); and it processes M heartbeats + tracks the M×1024 task DAG + drives all speculation — fine at ~130 nodes, a bottleneck at thousands. Classified coordinator-bottleneck (a central component that every task talks to → its throughput / availability caps the cluster's). (2) The flat all-to-all shuffle is connection-heavy: M×1024 = ~133k streams now, millions at thousand-node scale — each a TCP connection with setup, slow-start, and head-of-line-blocking overhead; the bisection cut ends up fragmented across a flood of tiny transfers instead of saturated by a few fat ones, so the effective bisection bandwidth is well below the nominal. Still network-bandwidth-bound — but now the limit is the structure of the all-to-all (too many tiny flows), not the byte volume (Iter 1 fixed that). Per bottleneck-to-decisions#coordinator-bottleneck: make the coordinator's state small + checkpointed (fast re-elect), delegate the per-node work (a two-level scheduler), keep it off the hot path. Per bottleneck-to-decisions#network-bandwidth: a hierarchical shuffle to coalesce many tiny transfers into few fat ones — the topology-aware structural fix on top of the byte-volume fix.

Architectural decisions to revisit

Per bottleneck-to-decisions#network-bandwidth: shuffle structure (hierarchical / multi-round — this iteration). Alternatives considered (C4)

  • Thin checkpointed coordinator + per-rack sub-schedulers + a two-round (within-rack then between-rack) shuffle — chosen. Decentralizing the coordinator removes the SPOF (loss = ~seconds re-elect from a tiny checkpoint) and the scaling gate (per-rack sub-schedulers absorb the heartbeat/DAG/speculation load locally; the coordinator only does the global boundary decision). The hierarchical shuffle coalesces M×1024 thin cross-rack flows into ~R_racks×1024 fat ones — far less per-connection overhead, a bisection cut that's actually saturated. This is the standard large-cluster shape (Dryad/DryadLINQ's hierarchical exchange, Spark's tiered shuffle services, the Sort-Benchmark winners' rack-aware multi-round shuffles). Cite failure-modes#leader-loss (the coordinator-loss mode the checkpoint+standby addresses).
  • Fully decentralized — no coordinator at all; nodes gossip the split keys via a distributed agreement — rejected. The split-key decision needs a global view of the merged sample; doing it by gossip/consensus over ~130 nodes is slower and far more complex than “everyone sends ~0.5 GB of sample to one small node that sorts it in seconds and broadcasts 1023 keys.” The coordinator is cheap and the decision is genuinely centralized — what hurts is keeping it on the hot path; the fix is to take it off, not to eliminate it. (Constraint-flip: 🔀 at extreme scale — tens of thousands of nodes — even gathering the sample to one node is a fan-in bottleneck; then you'd do a hierarchical sample-merge: each rack merges its mappers' samples to a rack sample, the coordinator merges the ~R_racks rack samples — a tree, same idea as the hierarchical shuffle.)
  • Skip the hierarchy; just buy a non-blocking fabric (bisection = sum of NICs) — rejected as not under our control, and not sufficient. On a non-blocking fabric the cross-rack bandwidth stops being the limit — but M×1024 tiny TCP flows still carry per-connection overhead and head-of-line blocking; the hierarchy helps regardless of oversubscription by making each transfer fat. And we're given a commodity (~2:1) fabric; we design for what we have. (If the fabric were non-blocking, the hierarchy's value drops to “fewer connections to manage” — still worth it at thousand-node scale, marginal at ~130.)
  • Three or more shuffle rounds (a deeper tree: node → rack → pod → reducer) — rejected at this scale, kept in reserve. Each extra round adds a full re-materialization pass (~1× D of disk writes) — worth it only when the cross-tier connection count at the previous level is itself the bottleneck. At ~130 nodes / ~10 racks, two rounds (within-rack, between-rack) is plenty; a third round is for thousand-rack clusters — a probe.

The chosen move

(a) Thin, checkpointed coordinator + per-rack sub-schedulers. The coordinator's hot-path job shrinks to: gather the merged sample, pick the 1023 split keys + hot-key map, assign ranges→reducers and the rack→reducer layout, write that (<~MB) state to the object-store checkpoint, then step off. Per-rack sub-schedulers own within-rack heartbeats, task assignment, speculation, and materialized-partition cleanup — a two-level scheduler, so the coordinator no longer fields M heartbeats or tracks every attempt. A standby coordinator holds an etcd lease; on the primary's lease expiry it reads the latest coordinator checkpoint and resumes — a ~seconds re-elect, because the coordinator's state is tiny and the in-flight work lives in the workers/rack-schedulers, not the coordinator. (b) Hierarchical two-round shuffle. Round 1 (within rack): the mappers in each rack exchange their materialized partitions so the rack collectively holds, for each reducer r, one combined “rack-partial-partition” for range r — rack-local moves over the unblocked top-of-rack links. Round 2 (between racks): each reducer r pulls one fat transfer per rack (that rack's combined records for range r) — so cross-rack streams go from M × 1024 (thin) to R_racks × 1024 (fat). The bytes on the wire are the same ~40 TB; the difference is that the bisection cut is saturated by a few fat flows instead of fragmented across a flood of tiny ones (per-connection setup, slow-start, head-of-line blocking all amortized). Cite bottleneck-to-decisions#coordinator-bottleneck and #network-bandwidth. Sizing: ~10 racks of ~13 nodes; coordinator state <~MB → checkpoint + re-elect < ~5 s; cross-rack flows ~10×1024 = ~10k (down from ~133k) → the effective bisection bandwidth rises (fewer overheads) → the shuffle phase tightens; round 1 adds a within-rack re-materialization (~1× D of rack-local disk+net, overlapped, ~minutes) — the price of the coalescing. Net wall-clock ~stable or slightly better at ~130 nodes; the big win is that this design keeps working at 5–10× the node count, which the flat coordinator + flat all-to-all would not.

What breaks (C5; cite failure-modes.html)

  • Coordinator and standby both lost (or a split-brain on the lease). Blast radius: scheduling stalls (the cluster keeps running in-flight tasks but no new assignments / speculation decisions). Detection: no node holds the coordinator lease. Recovery: a fresh coordinator is started (the etcd lease + the object-store checkpoint are the source of truth — whoever wins the lease reads the checkpoint and resumes); split-brain is prevented by the lease being a fenced token (an old coordinator that comes back can't act on a stale token). Worst case (etcd itself down): the job pauses until etcd recovers; no data is lost (everything's materialized / checkpointed). Cite failure-modes#lease-expiry-storm for the lease-handling discipline.
  • A rack partitions off mid-round-1. Blast radius: that rack — its mappers' input not yet read, its in-progress round-1 partials, any reducers it hosts. Better-isolated than Iter 2's rack loss because the rack's contribution is now a coherent unit (the rack's combined partials for each reducer). Detection: the rack sub-scheduler stops heartbeating to the coordinator. Recovery: the coordinator re-assigns that rack's input slices to other racks (re-map ~1/R_racks of the input) and the affected reducers re-pull the now-missing rack-partials once re-produced; the rest of the cluster is untouched. ~1 rack's worth of re-work, not the job.
  • A slow rack drags round 2. Blast radius: round-2 latency — every reducer waits on its slowest source rack. If one rack's nodes are uniformly slow (a bad TOR switch, an oversubscribed uplink), every reducer's fat transfer from that rack lags → the whole round-2 tail stretches. Detection: per-rack round-1 completion times; round-2 fetch progress per (rack,reducer). Recovery: speculation now operates at the rack-partial grain too — a slow rack's partials for a reducer can be re-produced from the underlying mapper materializations on faster nodes (the source data is still durable on disk); or, if a rack is consistently slow, drain it and re-spread its mappers. The hierarchy added a new straggler grain (the rack) — so speculation got a new grain to watch.

Probe iterations (the interviewer pulls the thread — beyond the 5-Core budget)

Each in a <details> block; same NFR-driven structure, one level deeper.

Probe A — Incremental re-sort (yesterday's sorted output + today's small delta)

Bottleneck. If the job runs daily and the input changes by only ~1% (today's new events), re-sorting the whole 100 TB from scratch is ~$27 and ~11 min of fleet you mostly don't need — you're paying full price for a 1% change. Move. Treat it as a merge of two sorted streams: yesterday's output is already 1024 sorted range-files; sort just today's delta (~1 TB) the normal way into the same 1023 split keys (reuse yesterday's split keys — or re-pick if the distribution drifted > a threshold), producing 1024 small sorted delta-files; then, per range, do a streaming 2-way merge of part-NNNNN.sorted (yesterday) + delta-NNNNN.sorted (today) → the new part-NNNNN.sorted — one linear pass per range, embarrassingly parallel across the 1024 ranges, no shuffle of yesterday's data at all. Cost ≈ (sort ~1 TB delta) + (read 100 TB + write 100 TB for the per-range merge) ≈ ~$10–15 and ~5 min — and if you keep the delta separate (a “base + delta” layout) and only periodically compact, the daily run is just “sort the delta” (~$1, ~1 min) and the consumer merges base+delta at read time. This is the LSM-style trade (cite storage-engines#lsm — append a sorted delta, merge lazily) applied to a batch sort. The constraint that drives it: a small delta + a daily cadence; if every run were a fresh 100 TB, this buys nothing.

Probe B — Columnar output (Parquet/ORC) instead of row-oriented sorted files

Bottleneck / opportunity. The downstream consumer doesn't want whole rows — it wants “SUM(amount) for users in this range” or “the ts column, sorted.” A row-oriented sorted file makes it read every byte of every row to touch one column. Move. Each reducer, after sorting its range, writes its output as a columnar file (cite storage-engines#columnar): column chunks, per-chunk min/max statistics (which are tight because the data is sorted — a sorted column's chunks have non-overlapping ranges, so a range scan reads only the relevant chunks), dictionary + run-length encoding (sorted data RLE-compresses heavily — long runs of the same user_id), and a footer index. The 1024 files are still range-partitioned and concatenable logically (the manifest gives the order), but the physical layout is column-major. The trade: columnar write costs more CPU at the reducer (encoding, stats) and the output is no longer a simple byte concatenation — a consumer that wants whole rows in order now pays a row-reconstruction cost. So: pick columnar if the downstream is analytical (scans, aggregations, projections); pick row-oriented if it's a sequential whole-row replay (e.g., feeding another system that wants the original records). Default for “sort yesterday's events for a batch consumer”: columnar (the consumer is almost always analytical). Default for GraySort: row-oriented (the benchmark wants the records back, byte-for-byte).

Probe C — Secondary sort (by user_id THEN ts), and stable-sort mechanics

The mechanics. The partitioner needs only the primary key: a record's reducer is determined by where its user_id falls among the split keys (which are themselves user_id values, or (user_id, ts) prefixes). The reducer's in-range sort uses the full composite (user_id, ts) as the comparator — so a user's events come out in time order within the file. The output still concatenates in (user_id, ts) order globally. A hot user_id is split across several reducers by ts-range (Iter 1) — (user_id=X, ts ∈ [t0,t1)), [t1,t2), … — and those splits concatenate in ts order, slotting into the user's position. Stable sort: when even (user_id, ts) has ties (two events at the same millisecond), break by input_seq — the deterministic (shard_index << 40) | offset tag every record carries (S1). The comparator is effectively (user_id, ts, input_seq); since input_seq is unique and a pure function of the input layout, the sort is total and byte-identical on a re-run. The cost: an extra ~8 B per record (the tag) — ~8% on 100 B records — carried through the shuffle; cheap insurance for determinism (and you'd want a deterministic tie-break even if “stable” weren't a stated requirement, just to make re-runs reproducible).

Probe D — Sort then GROUP BY in one job (the sort already groups equal keys)

Opportunity. A common downstream is “sort by user_id, then for each user compute COUNT(*), SUM(amount), MAX(ts).” The sort has already done the hard part: after the sort, all of a user's records are adjacent (within one range-file, or across the user's hot-key splits in order). Move. The GROUP BY is then a single linear pass over the sorted output: scan, accumulate per-user aggregates, emit a row when the user_id changes — O(n), one pass, no shuffle, embarrassingly parallel across the 1024 ranges (each range's file is a contiguous set of complete user-groups, except possibly a user straddling a range boundary — handled by the next range's reducer carrying a tiny “continuation” for a boundary user, or by the consumer stitching). So a combined “sort + aggregate” job is ~the sort's cost + one cheap linear pass — far cheaper than “hash-shuffle for the GROUP BY” as a separate job, which would re-shuffle all 100 TB. This is why sort-based aggregation is the engine's choice when the data's already going to be sorted (and why hash-based aggregation wins when it isn't): the sort pays for the grouping. Cite sharding-strategies#composite for the “partition by the group key” framing.

Probe E — Cross-region / cross-cloud sort

Bottleneck. If the 100 TB input is spread across regions (or the output must land in a different region), the shuffle — already the wall-clock — now crosses a WAN link that's both slow (~Gbps, not ~Tbps) and expensive (~$0.02/GB egress → 100 TB × ~$0.02/GB = ~$2,000 just to move the data once across the boundary — ~75× the whole on-spot compute bill). Classified cross-region. Move. First choice: don't — sort within each region (each region's data → its own 1024-way range-partition, using globally-agreed split keys so the per-region outputs are mergeable), then either leave the result as “1024 files per region, merge at read time” or do a final cross-region streaming merge per range (which moves 100 TB across the WAN exactly once, unavoidable if the consumer needs a single global file). If the data must be co-located in one region: pre-stage it (a one-time bulk copy, ideally during off-peak / via a cheaper bulk-transfer path) and then run the normal single-region sort. The principle: a global sort across a WAN is dominated by the WAN egress, so the design is “sort locally, merge globally, move bytes across the boundary at most once.” Cite cross-cutting#dr for the buddy-region framing and cost-shapes#egress-heavy for the egress-dominates shape.

Architecture

Connections (numbered above) and line-style legend. Solid slate = data path (read input, shuffle, write output); dashed gray = control / sample / broadcast / checkpoint / status; dashed red = re-elect / failover / speculation.

  • ① Object store ↔ Mappers / Reducers — mappers read disjoint input slices (~770 GB/node); reducers write the 1024 sorted files (write-then-rename, CAS commit).

  • ③ Mappers → Hierarchical shuffle — route each record (binary-search split keys, or hot-key map), zstd-compress per-reducer streams, materialize to local NVMe; round 1 = within-rack exchange → rack-partials; round 2 = cross-rack fat flows.
  • ④ Shuffle → Reducers (pull) — reducer r pulls one fat transfer per rack (the rack's combined records for range r); ~10 fetches instead of ~130; producers hold the data on disk until every reducer confirms.
  • ⑤ Reducers → Object store (checkpoint) — every ~30–60 s, partial sorted runs + per-source “received up to” offsets; a reclaimed reducer resumes from the latest checkpoint.
  • ⑥ Reducers → Object store (output) — the LC external merge sort over the pulled records (~97 GB per range); write the sorted file to a temp key, atomic rename, CAS the manifest entry → exactly-once.
  • ⑦ Job-control API ↔ Coordinator — submit (pre-checks required GB/s vs the deadline → 507 if the cluster's too small), status (phase / progress / ETA / stragglers), output manifest (the 1024 files + split keys + per-file key ranges).
  • ⑧ Coordinator ↔ Rack sub-schedulers — the coordinator delegates within-rack heartbeats, task assignment, and speculation to per-rack sub-schedulers (2-level scheduler); only whole-rack losses and cross-rack reassignments escalate. The standby coordinator re-elects from the etcd lease + the checkpoint on primary loss.

Degradation contract

When the spot pool shrinks below the standing headroom: the job slows, doesn't fail — the rack sub-schedulers request replacement spot; if none is available within a deadline budget, stragglers near the wall-clock are duplicated onto on-demand capacity (the job costs more, finishes ~on time). When the object store throttles reads/writes: the affected phase (input read or output write) backs off and retries with jitter; the wall-clock stretches. When etcd is unavailable: scheduling pauses (no new task assignments / speculation), in-flight tasks continue, no data is lost (everything's materialized/checkpointed); the job resumes when etcd recovers. When the coordinator and standby are both lost: scheduling stalls until a fresh coordinator wins the lease and reads the checkpoint — ~seconds. The job only truly fails if the input itself becomes unreadable (object-store regional outage) — then it's “re-run when the input is available again.” Cite failure-modes.html for the contract pattern.

  • Rollout. The “deploy” here is the sort-engine binary version — canary it on a small (~1 TB) sort first, compare the output against a known-good reference (byte-identical, since deterministic) before promoting to the 100 TB job; the binary is pinned per-job (a job runs the version it started with, even through worker restarts).
  • Migration. No persistent data store to migrate — each run is fresh. Changing the engine (e.g., hand-rolled → Spark, see below) is a re-implementation behind the same job-control API; run both on the same input, diff the outputs.
  • Headroom. +~10% mappers (reclaim absorption + speculation); +~15% wall-clock budget (the 30-min target vs the ~11-min typical — lots of slack for a bad spot day or a residual straggler); the hard outer bound (~45 min) is the page-the-operator line.

Real-system comparison

This is the TeraSort / MapReduce-sort shape, hardened. The MapReduce paper (Dean & Ghemawat) uses sort as its canonical example: the map emits (key, record), the partitioning function range-partitions on a prefix of the key (so the per-reducer outputs are already globally ordered), each reducer sorts its bucket, and the outputs concatenate — exactly Iteration 0; the paper's “backup tasks” for stragglers and its “map output written to local disk, fetched by reducers” are Iteration 2's speculation and materialized pull-shuffle. Hadoop's TeraSort (Owen O'Malley's 2008 record) is the same with the explicit sampling step to pick range boundaries (it samples ~100k keys, sorts them, picks 1023 split points — Iteration 0's reservoir-sample-and-quantiles, almost verbatim) and a trie over the split keys for fast routing. Google's “Sorting 1 PB with MapReduce” (2008) ran 1 PB on ~4000 machines — the “hundreds-to-thousands of nodes for minutes-scale, big-data sorts” anchor; they note the run was dominated by the shuffle and by disk, and that stragglers/failures over a long run forced the recovery machinery. Apache Spark's 2014 GraySort win (“Spark sets new sort record,” Reynold Xin et al.) sorted 100 TB on ~206 EC2 nodes in ~23 min and 1 PB in ~4 hr — their ~206 nodes for 100 TB is almost exactly this page's ~130–200, and their write-up is mostly about the shuffle: a sort-based shuffle that writes one file per map task (not one per reduce task) to cut file count, a network-layer rewrite (Netty zero-copy) to saturate the NICs, and pull-based fetch — Iterations 1–3's “fewer/fatter transfers” theme. Dryad / DryadLINQ (Microsoft, OSDI '08) is the lineage for the hierarchical exchange (a DAG of stages with explicit, topology-aware connectors) and for partition-skew handling via dynamic re-partitioning — Iterations 1 and 3. The TritonSort / Themis academic papers (UCSD) push the “balanced sort” idea — size every resource (disk, NIC, CPU, RAM) so none idles — which is exactly S3's “CPU is idle, so spend it on compression” reasoning. Where this page diverges: it picks a hand-rolled MapReduce-style sort engine (or a thin layer on a batch framework) over a heavyweight general engine — at 100 TB / 30 min / spot, a Spark or Hadoop job works (and is what most teams would reach for), but a purpose-built sort can skip the general DAG machinery and tune the shuffle/materialization/checkpoint exactly to this one job; a cloud-native sort (EMR/Dataflow/BigQuery ORDER BY) is the “just submit it” option if you don't want to run the cluster — you pay the managed premium and lose the spot-cost lever, which for a daily 100 TB sort is real money. Did the real teams make a different call? Mostly no — they used Hadoop/Spark because those were the tools at hand and the benchmark rules; the “hand-rolled” choice here is a judgment call for the specific case of a recurring, well-understood, cost-sensitive sort, and TritonSort is the precedent that it's a defensible one.

What if it's 1 PB, or the deadline is 10 min? (scale flip)

1 PB at the same 30-min deadline = 10× the bytes → ~10× the aggregate throughput → ~1300–2000 nodes (and the Sort-Benchmark PB winners used ~4000) — the design holds, but the flat coordinator and the flat all-to-all (~2000×1024 = ~2M thin flows) would choke, which is exactly why Iteration 3's hierarchy exists; at PB scale you'd add a third shuffle round (node→rack→pod→reducer) and a hierarchical sample-merge (rack samples → pod samples → coordinator) — Probe E's 🔀 flip. A 10-min deadline at 100 TB = ~3× the throughput → ~400 nodes, ~$50–75 spot — the cost-vs-speed knob (S6) turned up; the bisection-bandwidth floor (cite bottlenecks#network-bandwidth-bound) is what sets it, and “shuffle fewer bytes” is what lets you hit it with the fewest nodes. The ~15% wall-clock headroom (S6) buys you nothing against a 3× tighter deadline — that's a re-size, not a slack absorption.

Why a hand-rolled engine and not just Spark? (alt-engine)

Spark works — its 2014 GraySort win is this exact workload — and for most teams it's the right answer (you get the shuffle, the speculation, the materialization, the fault tolerance for free, and a huge ecosystem). The case for a purpose-built sort engine here: (1) a recurring daily 100 TB sort on spot, where the ~3–5× spot-vs-on-demand cost lever is real money — a hand-rolled engine controls its own task placement, materialization, and reclaim handling exactly for spot; (2) you skip the general-DAG machinery (catalyst, RDD lineage, the JVM heap pressure of a general engine) and tune the shuffle/checkpoint to this one job; (3) determinism — byte-identical re-runs — is easier to guarantee when you own the whole pipeline than to bolt onto a general engine. Where Spark would win: anything but a pure sort (joins, aggregations, ML), or a one-off (the dev cost of a custom engine never amortizes), or a team without the SRE bandwidth to run their own cluster engine. A cloud-native sort (EMR/Dataflow/BigQuery ORDER BY) wins if you want zero ops and will pay the managed premium — but you lose the spot lever. So: hand-rolled for a recurring, cost-sensitive, well-understood sort; Spark for everything broader; cloud-native for "I never want to think about it." Cite storage-engines#heap-append for why the shuffle/materialization is a log-append problem either way.

What changes if the records are PII (a user_id column under GDPR)? (regulatory)

Surprisingly little in the pipeline, because it never interprets the payload, never logs records, and deletes all transient state (materialized shuffle, checkpoints) at job end — the data-protection obligations live with the input and output datasets, not this fleeting fleet. What changes: (1) the sample sent to the coordinator must be keys only (not whole records) — already the case — and is discarded after the split keys are computed; (2) instance volumes (where the materialized shuffle lands) must be encrypted — X1; (3) an erasure request is satisfied by re-running the sort over the post-erasure input — cheap via incremental re-sort (Probe A) if the deletions are a small delta; (4) audit: log the job (who ran it, over which input, when, what comparator), not the records. What stays the same: the entire iteration arc — sampling, range-partition, shuffle, speculation, hierarchy — is data-content-agnostic. Cite cross-cutting#privacy. Naming what stays the same is the point: a regulated sort is the same sort.

The spot pool evaporates 5 min into the run — now what? (failure)

The degradation contract (S6): the job slows, doesn't fail. Mappers that finished have their per-reducer partitions materialized on disk — if those nodes are gone, their input slices (one node's worth each) are re-mapped on whatever capacity is available; reducers resume from their last ≤60-s checkpoints. The rack sub-schedulers request replacement spot; if there's none, near-the-deadline stragglers are duplicated onto on-demand capacity (the bill rises, the wall-clock holds). The coordinator's state is checkpointed, so even if the coordinator node was in the evaporated pool, the standby re-elects in ~seconds. Worst case — no capacity of any market available — the job pauses (everything materialized/checkpointed, nothing lost) and resumes when capacity returns; the input is never at risk (it's in the durable object store, untouched). Detection: a burst of missed heartbeats + a spot-reclaim-notice flood; the runbook entry is "check spot pool health, fall back to on-demand for the tail, expect the wall-clock to stretch toward the ~45-min outer bound." Cite failure-modes#bad-deploy (a worker reclaimed mid-run) and #leader-loss (the coordinator case).

Where does the bill actually come from? (cost)

~$27/sort on spot, of which ~$26 (96%) is the ~130-node cluster's compute-hours — everything else (coordinator, rack sub-schedulers, object-store I/O, output PUTs) is <~$1 combined; intra-region object-store transfer is free. So the bill is the cluster size, and the cluster size is set by the network bisection bandwidth (shuffle 100 TB / bisection in ≤ ~20 min), not by CPU or disk. The cost-vs-speed knob is cluster size: ~2× nodes → ~½× wall-clock → ~2× bill (~$50 spot for ~5–6 min); ~½× nodes → ~2× wall-clock (~22 min, still under 30) → ~½× bill (~$13 spot). The free lever is "shuffle fewer bytes" — compression (spare CPU), co-location, rack-awareness, hierarchy — which Iteration 1 showed is worth ~40% of the bill (fewer bytes → fewer nodes meet the deadline). Cite cost-shapes#stream-processing — shuffle bytes dominate both the wall-clock and the bill; the storage cost of the output (~$0.023/GB-mo × 100 TB) is the consumer's choice (how long to keep it), not this job's.

How do you ship a new sort-engine version without risking a bad run? (migration)

There's no persistent data store to migrate — each run is fresh — so "migration" here is engine-version rollout. The deterministic-output property is the lever: canary the new binary on a small (~1 TB) sort, then diff its output against the known-good reference byte-for-byte; if identical, promote. A job pins its engine version at submission (workers that restart re-launch the same version), so a mid-run upgrade is impossible by construction. Switching engine families (hand-rolled ↔ Spark ↔ cloud-native) is a re-implementation behind the same job-control API: run both on the same input, diff the outputs (they should be byte-identical if both honor the same comparator + tie-break), cut over the daily job once they match for a week, keep the old one as the rollback for one cycle. The rollback trigger: any output mismatch, or a wall-clock regression past the ~45-min outer bound on the canary.

The run is slow right now — where do you look first? (monitoring/debug)

Phase first (the status API): which phase is over budget? Sample over budget → the object-store read is throttled, or the reservoir is mis-sized (too large). Shuffle over budget → check per-reducer received_bytes (one growing >> mean → bad split keys / missed hot key → the heavy-hitters sketch under-sampled; recovery is a larger sample + re-run, or dynamic re-split) and per-rack round-1 completion spread (one rack slow → a bad TOR switch / oversubscribed uplink → speculate the rack-partials onto faster nodes or drain the rack). Sort/write tail over budget → check the speculation count (high → pervasive stragglers, maybe noisy spot neighbours → add headroom; low but the tail's long → one genuine straggler the threshold didn't catch → lower the speculation threshold). Cross-cutting signals: spot-reclaim count (high → the pool's churning → expect the wall-clock to stretch), per-node disk-free (a node near full → a materialized-partition leak → the cleanup-after-confirm logic), coordinator-lease-changes (flapping → the coordinator node's unhealthy). Each of these maps to a C5 failure mode; the runbook is "phase → the over-budget phase's SLI → that SLI's failure mode → its recovery."

S7. Key Takeaways

  • "It doesn't fit one box" is the whole opening move — and what distribution forces is the global-order problem. 100 TB ≫ one node's disk → the sort must distribute → you can't sort it in one place → how do you produce a global order across the pieces? That question, not "make it parallel," is what determines the architecture.
  • Range-partition, not hash — because the merge step is the cost. Range-partition into disjoint ranges → the per-reducer outputs concatenate with no merge; hash-partition gives co-location (good for GROUP BY) but no order → you'd need a full k-way merge over all shards, another whole pass. "Hash-partition then sort within each shard" is the trap: it works for the wrong problem.
  • Sample, don't scan twice. A reservoir sample of ~1–10M keys, sorted centrally, gives the 1023 quantile boundaries to within a few percent — an exact two-pass histogram doubles the read cost for a marginal gain. And on real (Zipf) data you must also detect heavy keys (a Space-Saving sketch over the sample) and give them their own / split reducers — equal-count ranges aren't enough when one key owns 5% of the records.
  • The shuffle is the wall-clock; bisection bandwidth is the ceiling. Every byte crosses the network exactly once, bounded not by the summed NIC bandwidth but by the cut between halves of the cluster (a fraction of it, on a commodity fabric). So the two levers are "shuffle fewer bytes" (compress — spend the spare CPU; co-locate — free; rack-aware — free; hierarchical rounds — coalesce thin flows into fat ones) and "don't let the all-to-all stall" (speculate the straggler, make a task loss local).
  • Spot-tolerance is bought with a materialized shuffle. Writing the mapper→reducer partitions to local disk (and pulling, not pushing) costs ~1× the data in extra local writes — and buys you a recovery story (a survivor serves the data; a reducer resumes from a checkpoint) that turns "any node loss = restart" into "one node's worth of re-work" — which is what makes a ~3–5×-cheaper spot fleet usable. The cheap insurance unlocks the big saving.
  • The coordinator should decide the boundaries and then get out of the way. Picking the 1023 split keys from a merged sample is genuinely centralized and cheap — keep it. Fielding M heartbeats and tracking the M×1024 DAG on the hot path makes it a SPOF and a scaling gate — delegate that to per-rack sub-schedulers, checkpoint the tiny global state, re-elect in seconds. "Centralized decision, decentralized execution."
  • Determinism is a property you keep, not one you add. The textbook single-box sort is byte-identical on a re-run for free (one merge order); the distributed version works to preserve it — seeded reservoirs → reproducible split keys, a fixed comparator with a deterministic input_seq tie-break, idempotent write-then-rename commits so speculative duplicates and re-runs produce the same file exactly once. The FR demanded it; on ~130 nodes it takes effort.
  • This is the batch-pipeline shape. Map → shuffle → reduce, with every load-bearing batch decision visible: sampling for partition boundaries, skew handling, shuffle-byte minimization, straggler speculation, materialize-and-checkpoint for fault tolerance, hierarchical topology-aware data movement, a thin checkpointed coordinator. Learn it once on sort; it's the same skeleton for any large batch job.
