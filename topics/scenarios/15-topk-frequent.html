Top-K Most Frequent URLs

Finding the 100 most visited pages from a billion page views per hour — when exact counts do not fit in memory on a single machine.

NALSD Walkthrough • ~25 min read • LC #347 (Top K Frequent Elements)

LC kernel. Top K Frequent Elements (LC #347). Textbook structure: a HashMap<item, count> built by one pass over the array, then either a min-heap of size k over the entries (O(n log k)) or bucket-sort by count (O(n)) to pull the top k. Three properties that break on an unbounded high-volume stream: (1) memory is O(distinct keys) — the array is finite, but a page-view stream has no end and (worst case) every URL is distinct, so the hashmap grows to 100 GB+; (2) no mergeability — you cannot split the stream across machines and add up partial hashmaps cheaply, and you cannot merge two partial top-k lists into a correct global top-k (a key that's #101 on every shard can be #1 globally); (3) no freshness / windowing — LC counts a static array once; "top-100 in the last hour" needs old events to stop counting. The C8 production cousin replaces the exact hashmap with a fixed-size frequency sketch (Count-Min / Space-Saving — cite sketches.html#cms, #space-saving) that is O(1/ε), mergeable (associative — the property the distributed fan-in needs), and a rotating bank of them for windowing, paired with the same min-heap for the top-k extraction.

The Scenario

You are building a real-time analytics dashboard for a large web platform. The primary query: "What are the top 100 most visited URLs in the past hour?" — the trending list every product surface shows.

Reused by. The trending-topics overlay in #45 (search autocomplete) and the trending-topics / mute-filter surfacing in #41 (home-timeline fan-out) are this design — they consume the mergeable sketch + min-heap derived here. If you change the sketch sizing or the partition scheme on this page, those pages inherit it.

System class & functional requirements (cite functional-requirements.html)

Class: automation pipeline (stream-processing) with an algorithmic kernel — it folds an unbounded event stream into a small derived structure and serves a fixed query; it is not a queryable analytics DB (that would bring an ad-hoc-query FR and a per-row read-latency NFR — out of scope). Functional requirements:

  • Ingest the page-view stream. Accept (url, ts) events at the firehose rate; fire-and-forget.
  • Maintain an approximate top-K. Track the k most-frequent URLs over the active window, updated continuously.
  • Serve "current top-K" queries. Return the ranked list with (approximate) counts, on demand, in milliseconds.
  • Window the counts. "Last hour" means events older than the window stop contributing; granularity ~1 minute is fine.

Out of scope (named, not silently dropped): exact counts (a separate batch path handles billing/SLA reporting — covered in the MapReduce alternative below); per-user / personalized top-K (a different cardinality story — flagged as a probe); arbitrary group-bys ("top URLs per country") — the dashboard query is fixed; click-through / dwell-time ranking — this counts visits only; raw-event storage and replay — that's the event log's job (Kafka / S3), not this pipeline's.

Flagged assumptions (would confirm with the interviewer): (1) the URL popularity distribution is heavy-tailed / Zipfian — the top ~10 M URLs carry ~99% of traffic (cite access-patterns.html#power-law); the design must still survive the worst case (every event distinct); (2) K ≈ 100 — small relative to the distinct-key count; (3) error tolerance ε ≈ 0.0001 (an item that's within ~100K of the threshold on a 1B-event window can be a false positive — acceptable for a trending widget, not for billing); (4) window = sliding 1 hour, ~1 min granularity (not all-time; all-time would drop the rotation machinery); (5) a single internal tenant (the platform's own traffic) — multi-tenant top-K-per-customer is a probe.

Clarification ledger

  • Approximate OK, or must counts be exact? → approximate for the dashboard; exact only on the batch path (drives sketch vs. hashmap).
  • Window: last hour sliding, or fixed clock-hour buckets, or all-time? → sliding 1 h, ~1 min granularity (drives rotating sketches vs. a single sketch).
  • How big is K, and is it fixed? → ~100, fixed (drives heap size; K=10⁴ would still be a heap, K="all above a threshold" would be Space-Saving's natural output).
  • One global list, or per-segment (country / device / tenant)? → one global list (per-segment multiplies cardinality — the personalized-top-K probe).
  • Traffic today and growth? → ~1B events/h now (~278K/s); plan for 10× (drives the shard count and the partition-by-key decision).
  • Is there already a durable event log we can read? → yes, Kafka (drives the ingest transport and the batch path's source).

The Memory Challenge

To find the top-100 most frequent URLs, you need to count how many times each unique URL appears. The question is: how much memory does that take?

Worst case: every page view is a unique URL.
  1 billion unique URLs x (100 bytes URL + 8 bytes count) = 108 GB
  Does not fit in a single machine's memory (typical: 64 GB).

Realistic case: 90% are duplicates.
  100 million unique URLs x 108 bytes = ~10.8 GB
  Fits in one machine, but barely. And what if traffic doubles?

Optimistic case: 99% are duplicates (heavy-tailed distribution).
  10 million unique URLs x 108 bytes = ~1.08 GB
  Comfortable for one machine.

We need a design that works even in the worst case, and ideally scales horizontally as traffic grows.

Back-of-Envelope Math

This section establishes only the memory facts the Naive Design needs (how big exact counts are) and forward-references the sketch that Iteration 1 introduces — it does not use the sketch in any Naive-Design calculation.

Approach 1: Exact Counts with Hash Map

Data structure: HashMap<URL, count>
Memory per entry: 100B (URL) + 8B (count) + 40B (hash map overhead) = ~148B
Unique URLs (worst case): 1 billion
Total memory: 1B x 148B = 148 GB → does NOT fit on one machine.

Unique URLs (realistic): 100 million
Total memory: 100M x 148B = 14.8 GB → fits, but tight.

Approach 2: Count-Min Sketch (Approximate) — introduced in Iteration 1

A Count-Min Sketch estimates per-item frequencies in O(1/ε) memory by hashing each item into d rows of w counters and reading back the minimum of the d hit counters (it never underestimates; over-count ≤ ε·N with probability ≥ 1−δ). Full mechanism, the why-the-minimum argument, and the Space-Saving / Heavy-Keeper alternatives are in sketches.html#cms — we don't restate them. What this page contributes is the sizing for this workload and the consequences (windowing, partitioning, the heap pairing); those are below and in Iteration 1.

Sizing the Sketch

The Count-Min Sketch has two parameters that control accuracy:

  • ε (epsilon): error bound. The estimated count for any item is at most the true count + ε × N, where N is the total number of events.
  • δ (delta): failure probability. The probability that the error exceeds ε × N.
Width:  w = ⌈e / ε⌉      (e = 2.718, Euler's number)
Depth:  d = ⌈ln(1/δ)⌉

For ε = 0.001 (0.1% error), δ = 0.01 (99% confidence):
  w = ⌈2.718 / 0.001⌉ = 2,718
  d = ⌈ln(100)⌉ = ⌈4.605⌉ = 5

  Total cells: 2,718 x 5 = 13,590
  Memory: 13,590 x 4 bytes/counter = 54,360 bytes ≈ 54 KB

For ε = 0.0001 (0.01% error), δ = 0.01 (99% confidence):
  w = ⌈2.718 / 0.0001⌉ = 27,183
  d = 5

  Total cells: 27,183 x 5 = 135,915
  Memory: 135,915 x 4 bytes = 543,660 bytes ≈ 540 KB

What Does the Error Mean Concretely?

With ε = 0.001 and N = 1 billion events:
  Maximum overestimate for any URL: 0.001 x 1,000,000,000 = 1,000,000

  If the true count for "example.com/popular" is 5,000,000,
  the sketch might report up to 6,000,000 (overestimate by 1M).

  If the true 100th most popular URL has 3,000,000 views,
  a URL with only 2,000,000 views could appear to have 3,000,000
  (2M + 1M overestimate) → false positive in the top-100.

With ε = 0.0001 and N = 1 billion:
  Maximum overestimate: 0.0001 x 1,000,000,000 = 100,000

  Much tighter. A URL needs to be within 100K of the threshold to
  be a false positive, which is unlikely for URLs with millions of views.

Naive Design

The absolute simplest thing that could work — the LC #347 kernel, run once per minute on the last hour of events: collect all ~1 billion page-view events of the window, build an exact HashMap<URL, count>, sort by count, return the top 100. No sketch, no streaming, no partitioning — just the textbook algorithm at scale.

Why This Fails

Step 1: Collect all 1B events in memory.
  Memory needed: 1B x 100B = 100 GB → does not fit.

Step 2 (alternative): Write all events to disk, external sort.
  Disk: 100 GB of data.
  External merge sort: O(N log N) with disk I/O.
  At 200 MB/sec disk throughput: multiple passes over 100 GB.
  Time estimate: 10-30 minutes for the sort alone.

Step 3: Scan sorted file, count consecutive URLs, keep top 100.
  One more pass over 100 GB: ~500 seconds.

Total time: ~30-40 minutes. Not real-time.

What Breaks (cite bottlenecks.html / failure-modes.html)

  • Memory capacity bound (cite bottlenecks.html#memory-capacity-bound — aggravated by #cardinality-explosion): 100 GB of raw events — or even just the exact HashMap over the distinct URLs (up to 148 GB worst case) — does not fit in one machine's RAM. The bound is set by the distinct-key count, which the stream can drive arbitrarily high.
  • Latency / freshness: the external-sort path is ~30 min; the dashboard needs a refresh every minute. The whole batch shape is wrong for the freshness NFR.
  • No horizontal scaling path: doubling traffic doubles the single-machine load with nowhere to put it. The matching failure mode when the box does fall over is failure-modes.html#backlog-overflow — events pile up faster than the one node drains them.
  • No windowing: "top-100 in the last hour" needs old events to stop counting; re-sorting the whole dataset every minute re-does O(N log N) work for a result that barely moved.

Iterations 1–2 close these in order: the sketch fixes the memory/cardinality bound and enables streaming (freshness); per-key partitioning gives the horizontal path; rotating sketches give windowing.

Iteration 1: Count-Min Sketch + Min-Heap (Single Machine)

Bottleneck in the Naive Design

The dominant bottleneck is memory capacity, set by the distinct-key count (cite bottlenecks.html#memory-capacity-bound + #cardinality-explosion): the exact HashMap is O(distinct URLs) = up to 148 GB — it doesn't fit, and even when it does (~14.8 GB realistic) it's one traffic-doubling away from not fitting. The architectural decision to revisit (cite bottleneck-to-decisions.html#cardinality): does this count need to be exact? — if not, swap the exact structure for a fixed-size sketch. Secondary bottleneck: the batch shape is wrong for freshness — fold the stream incrementally instead. Both are addressed by the same move: a streaming sketch + heap.

Alternatives considered (C4)

  • Exact HashMap<URL, count> (the LC kernel as-is) — rejected: memory is O(distinct keys), unbounded on this stream (the disqualifying constraint). Survives only if you can bound distinct URLs and over-provision for growth — neither holds here. Kept for the batch path (bounded slice per mapper) — see the MapReduce alternative.
  • Count-Min Sketch (chosen) — O(1/ε) memory, over-counts only (never under), linear & mergeable (adding two CMS arrays cell-wise gives the CMS of the union — the property Iteration 2's fan-in needs), trivially supports rotation for windowing. Cite sketches.html#cms. Disqualifying weakness it doesn't have for us: its over-count grows with total stream volume N — fine at ε=10⁻⁴ for a million-view threshold, would be a problem if the threshold were small.
  • Space-Saving (Stream-Summary) — viable alternative: tracks exactly the top-m candidates in O(m) counters, gives a tighter error for the heavy hitters specifically and emits the top-K directly (no separate heap needed). Rejected as the primary because it's harder to make mergeable across shards (merging two Stream-Summaries is not the clean cell-wise add CMS gives) — and Iteration 2 leans on mergeability. Cite sketches.html#space-saving. (A real system often runs both — CMS for mergeable counts, Space-Saving per shard for the candidate set; that's the final design's refinement.)
  • Heavy-Keeper — viable alternative: a count-with-exponential-decay sketch tuned for heavy hitters with much lower error than CMS at the same size, by aggressively decaying cold keys. Rejected as the primary for the same mergeability reason and because its decay interacts awkwardly with our explicit rotation-based windowing; noted as a per-shard refinement. Cite sketches.html#heavy-keeper.

The chosen move: a Count-Min Sketch for approximate frequency counting + a min-heap of size 100 for the current top-K, fed by the live stream.

Process the event stream in real-time using two data structures: a Count-Min Sketch for approximate frequency counting, and a min-heap of size 100 for tracking the current top-K.

The Algorithm

class TopKTracker:
    def __init__(self, k=100, cms_width=27183, cms_depth=5):
        self.k = k
        self.cms = CountMinSketch(width=cms_width, depth=cms_depth)
        self.heap = MinHeap()          # min-heap ordered by count
        self.in_heap = {}              # URL → current count in heap

    def process_event(self, url):
        # Step 1: Update the Count-Min Sketch
        self.cms.increment(url)
        estimated_count = self.cms.query(url)

        # Step 2: Update the top-K heap
        if url in self.in_heap:
            # URL already in heap: update its count
            self.in_heap[url] = estimated_count
            self.heap.update(url, estimated_count)

        elif len(self.heap) < self.k:
            # Heap not full yet: add directly
            self.heap.push(url, estimated_count)
            self.in_heap[url] = estimated_count

        elif estimated_count > self.heap.peek_min_count():
            # New count exceeds the smallest in the heap: replace
            evicted_url = self.heap.pop_min()
            del self.in_heap[evicted_url]
            self.heap.push(url, estimated_count)
            self.in_heap[url] = estimated_count

        # If count <= heap minimum: this URL is not in top-K, ignore.

    def get_top_k(self):
        return self.heap.get_all_sorted_descending()

Performance Analysis

Per-event cost:
  CMS update:    O(d) = O(5) hash computations + 5 counter increments
  CMS query:     O(d) = O(5) hash + min of 5 values
  Heap update:   O(log K) = O(log 100) ≈ O(7)
  Total per event: O(d + log K) ≈ O(12) operations

Throughput:
  278,000 events/sec x 12 operations = 3.34 million operations/sec
  A single modern CPU core handles ~100M simple operations/sec.
  Processing uses ~3.3% of one core. Trivially fast.

Memory:
  CMS: 540 KB
  Heap: 100 entries x ~108 bytes = ~10.8 KB
  in_heap map: 100 entries x ~148 bytes = ~14.8 KB
  Total: ~566 KB — less than 1 MB.
From 148 GB to 566 KB. The Count-Min Sketch + min-heap approach uses roughly 260,000x less memory than exact counting, processes 278K events/sec on a fraction of one CPU core, and provides results at any moment without waiting for a batch to complete.

Handling the 1-Hour Window

The requirement is "top-100 in the last hour." Events older than 1 hour must stop contributing to counts. Two approaches:

Approach A: Rotating Sketches

Maintain 60 sketches, one per minute.

Minute 0-1:   Write to sketch_0.
Minute 1-2:   Write to sketch_1.
...
Minute 59-60: Write to sketch_59.
Minute 60-61: Reset sketch_0 and reuse for the new minute.

To query the last hour: sum the counts across all 60 sketches.

Memory: 60 x 540 KB = 32.4 MB. Still tiny.

Trade-off: 1-minute granularity for window eviction.
Some events might be counted for up to 61 minutes.

Approach B: Sliding Window with Decay

Use a single sketch but apply exponential decay. Multiply all counters by a decay factor every minute. Recent events contribute more; old events fade away. This is less precise but uses only one sketch.

What breaks (C5; cite failure-modes.html)

  • The one node dies → failure-modes.html#bad-deploy / process crash: the sketch + heap live only in RAM, so a crash loses the current window. Detection: heartbeat / no-emit alarm. Recovery: replay the last hour of events from the Kafka log into a fresh sketch (minutes) — or, until Iteration 2, accept a gap. This is the "no failure story" property of the LC kernel, unfixed at this iteration — it's a single-machine design.
  • Ingest outpaces the consumer → failure-modes.html#backlog-overflow: at 278K events/s the fold uses ~3% of one core, so this is comfortable now — but a 10× spike or a GC pause stalls the consumer and the Kafka lag grows. Detection: consumer-lag metric. Recovery here: none structural (one node) — Iteration 2's sharding is the real fix; in the interim, shed (sample) the stream and count the drop.
  • The sketch saturates / over-counts blow up → failure-modes.html#data-corruption (logical, not bit-level): if the operator under-sizes the sketch (too small w) for the actual distinct-key count, collisions inflate every count and the heap fills with false positives. Detection: sample-and-compare against an exact count on a small slice (a canary). Recovery: re-size the sketch (an offline parameter change) — see Operations in the Final Design for the canary protocol.

Iteration 2: Distributed Top-K Across Multiple Machines

Bottleneck in Iteration 1

Two limits remain. (1) Single-node throughput & the matching failure mode — one consumer folds 278K events/s at ~3% of a core today, but a 10× growth (2.78M/s) or a runaway-hot URL pushes it toward bottlenecks.html#network-bandwidth-bound (the NIC, not the CPU, becomes the wall — 2.78M × ~100 B = ~280 MB/s of event payload, well into multi-Gbps territory), and a crash blinds the whole window. (2) No redundancy — the design has a single point of failure. The architectural decision to revisit (cite bottleneck-to-decisions.html#hot-partition): partition the stream — but by what key? — so each shard sees a complete sub-problem. The move is mechanically clean precisely because the CMS is mergeable (the property we chose it for in Iteration 1).

Alternatives considered (C4)

  • Partition by hash(URL), per-shard CMS + heap, merge top-K lists (chosen) — every event for a URL lands on one shard, so each shard's local top-K is exact-within-CMS-error for its key range, and the global top-K is the merge of N local lists of size ≥K (a key in the global top-K can't be missing from its owning shard's top-K). Cite sharding-strategies.html#hash. Disqualifying weakness it can have but doesn't here: a single mega-hot URL still pins one shard — tolerable because that one URL is one counter increment, not state growth (and a salted-key split is the probe if it ever isn't).
  • Broadcast the whole stream to every shard, each keeps a full sketch — rejected: it doesn't reduce per-node ingest at all (every node still eats 278K × ... events/s) — the throughput bottleneck (the disqualifying constraint) is untouched; it only buys redundancy, expensively. Useful only as a degenerate "replicate for HA" option, dominated by per-key sharding + replicating each shard.
  • One fat node, scaled vertically — rejected: there's no machine large enough for the worst-case exact count, and even with the sketch the NIC bandwidth (the disqualifying constraint) caps a single box; no failure story either.
  • Partition randomly, coordinator re-aggregates per-URL counts — rejected: a URL's count is split across shards, so no shard sees the true count and the coordinator would have to receive all distinct URLs from all shards to re-sum (cite bottlenecks.html#coordinator-bottleneck) — the coordinator's fan-in is the disqualifying constraint, and it re-creates the cardinality problem at the coordinator.

A single machine handles 278K events/sec easily. But if traffic grows to 10 billion events/hour (2.78M events/sec), or if we want redundancy, we distribute the work.

Architecture

Why Partition by hash(URL)?

If we partition events randomly, the same URL could be split across multiple workers. Worker A might see 3,000 visits to /home and Worker B might see 2,500 visits. Neither worker sees the true count of 5,500. The coordinator would need to re-aggregate, which defeats the purpose.

By partitioning on hash(URL), all events for the same URL go to the same worker. Each worker sees the complete count for its assigned URLs. Its local top-100 is exact (within CMS approximation error) for its partition.

The Coordination Step

Every 60 seconds:
  1. Each worker emits its local top-100:
     [{url: "/home", count: 5500}, {url: "/pricing", count: 4200}, ...]

  2. Coordinator collects 10 x 100 = 1,000 candidates.

  3. Coordinator picks the top-100 from these 1,000 candidates.
     Simple: sort 1,000 items by count, take first 100.
     Time: O(1000 log 1000) ≈ trivial (microseconds).

  4. Coordinator publishes the global top-100 to the dashboard.

Is This Exact?

Almost. The global top-100 is correct if every globally top-100 URL appears in at least one worker's local top-100. Since each URL is assigned to exactly one worker (hash partitioning), the URL with the globally 100th-highest count must be in its assigned worker's local top-100. This is guaranteed as long as each worker tracks at least 100 candidates — which it does.

Edge case: If we did NOT partition by URL (random distribution), then a URL's count is split across workers. No single worker sees the full count, and a globally popular URL might not appear in any worker's local top-100. With hash partitioning, this cannot happen.

Scaling

10 workers, each handling ~28K events/sec:
  Per-worker CPU: ~0.3% of one core (trivial)
  Per-worker memory: ~566 KB (CMS + heap)
  Total memory: ~5.7 MB

To handle 10x traffic (10B events/hour):
  100 workers, each handling ~28K events/sec
  Total memory: ~57 MB
  Coordination: merge 100 x 100 = 10K candidates → trivial

What breaks (C5; cite failure-modes.html)

  • A shard dies mid-window → failure-modes.html#bad-deploy / node loss: that shard's key range goes dark — the top-K is missing whatever was popular in that ~10% of the hash space. Fix: run each shard with a warm replica that consumes the same Kafka partitions (Kafka's offset bookkeeping makes the failover clean) and checkpoints its sketch bank to S3 every minute; a fresh replica rebuilds from the last checkpoint + replay-since-checkpoint (seconds, not minutes). Detection: per-shard heartbeat + Kafka consumer-group rebalance. Until then, the coordinator emits the merged list flagged "shard j stale" rather than silently dropping it.
  • A single mega-hot URL pins one shard → failure-modes.html#hot-key-meltdown: hash partitioning sends all of one URL to one shard; if that URL alone is, say, 30% of all traffic, that shard does 30% of the work. It's only a counter increment (no state growth), so it's far less dangerous than a hot key in a stateful store — but at extreme skew the fix is salted-key splitting (send (url, rand()%S) to S shards, sum the S counters at query time). Flagged as a probe; detection is per-shard ingest-rate imbalance.
  • The coordinator stalls or its emit is slow → failure-modes.html#slow-dependency: the dashboard goes stale, but the shards keep folding — nothing is lost, just not published. The coordinator is stateless (it merges N lists of K), so a standby takes over instantly; the dashboard shows the last good list with its age. Detection: emit-age gauge on the dashboard.
  • The sketch is mis-sized after a traffic shape change → failure-modes.html#data-corruption (logical): distinct-URL count climbed, w no longer big enough, over-counts inflate — same as Iteration 1's mode, now fleet-wide. Fix: the canary-compare-against-exact protocol (Operations, Final Design) plus a per-shard "estimated distinct keys" gauge (an HLL alongside the CMS — cheap) that trips a re-size alarm.

Alternative: Exact Count with MapReduce

If approximate counts are not acceptable (e.g., for billing, SLA reporting), use a batch approach with MapReduce or a distributed SQL engine.

The MapReduce Job

Map phase:
  - 100 mappers, each processing 1B / 100 = 10M events.
  - Each mapper builds a local HashMap<URL, count>.
  - Worst case: 10M unique URLs x 148B = ~1.48 GB per mapper. Fits in memory.
  - Each mapper emits its top-100.

Shuffle phase:
  - 100 mappers x 100 entries = 10,000 candidate (URL, count) pairs.
  - Shuffled to a single reducer (small enough).

Reduce phase:
  - Reducer receives 10,000 (URL, count) pairs.
  - Groups by URL, sums counts from all mappers.

  Wait — if mappers each count their local slice independently and emit
  top-100, a URL that is #101 on every mapper might actually be the true #1
  globally (if its traffic is evenly distributed across mappers).

  Fix: Mappers emit ALL (URL, count) pairs, not just top-100.
  Reducer groups by URL, sums counts, sorts, takes top-100.

  Shuffle data: 100M unique URLs x (100B + 8B) x 100 mappers?
  No — each unique URL appears on multiple mappers, but the reducer
  sums them. Shuffle size: 100M unique x 108B x avg_mappers_per_url.

  Better approach: partition by hash(URL) so each mapper gets a
  disjoint set of URLs. Then each mapper's top-100 is globally correct
  for its partition (same as the distributed streaming approach).

Timing

Map phase: Read 10M events x 100B = 1GB per mapper.
  At 500 MB/sec read speed: 2 seconds to read.
  Counting: 10M hash lookups ≈ ~50ms.
  Map phase total: ~2-3 seconds per mapper (parallelized).

Shuffle: 100 x 100 entries = 10K records = ~1 MB. Instant.

Reduce: Sort 10K entries = microseconds.

Total: ~3-5 seconds for the batch job.
But: only available AFTER the hour ends (batch, not streaming).

Resource Summary

Real-time path (10 workers):
  Memory:    10 x 60 sketches x 540 KB = 324 MB total
  CPU:       < 1 core total
  Network:   10 x 100 entries every 60s = negligible

Batch path (100 mappers, runs hourly):

Concrete technology choices (C10)

  • Stream transport: Kafka. Need: durable, partitioned-by-key, replayable (the failure story is "replay the last hour"), high throughput. Kinesis — managed and simpler ops, but per-shard 1 MB/s ingest / 2 MB/s egress caps make the shard count for 280 MB/s awkward (~280 shards) and 24 h–365 d retention is fine but pricier per GB at this volume; viable if you want zero Kafka ops. Pulsar — tiered storage and per-message TTL are nice, but it adds BookKeeper as a second system to operate for no win here. We pick Kafka: the key-ordered partition is our shard, consumer-group rebalance is our failover, and the ecosystem (Kafka Streams / Flink) is the obvious home for the fold. Cite message-queues.html.
  • Per-shard sketch runtime: a custom in-process sketch inside a Flink (or Kafka Streams) operator. Need: O(d) per-event update, mergeable, checkpointable, ~ns per op. A library (Apache DataSketches / stream-lib) — yes, use the library's CMS/Space-Saving implementation inside the operator; "custom" means "ours runs in our process," not "we re-implement the math." Redis with a CMS module (RedisBloom's CMS.*) — rejected: every event becomes a network round-trip to Redis (~ms vs ~ns in-process) — at 278K events/s that's 278K Redis ops/s per shard, and you've moved the bottleneck onto Redis for nothing. Druid / ClickHouse approximate-topK (topN / topK()) — great for the query side if the dashboard also wants ad-hoc top-K-per-segment, but it's a columnar store ingesting rows, not a streaming fold; it'd duplicate the pipeline. We keep the fold in-process; Druid/ClickHouse is the optional ad-hoc-query annex (a probe). Cite sketches.html#cms.
  • Serving store for the materialized top-K: Redis (or even just the coordinator's RAM). The output is one tiny list (≤ a few KB) refreshed every ~60 s and read by the dashboard. Redis — one SET trending:hourly <json> per minute; fast reads, trivially HA, and it doubles as a place to stash a few historical snapshots; chosen. A small Postgres — overkill for a single key written once a minute; only worth it if the trending list must be queried relationally (joined to URL metadata) — then it's a tiny table, not a load concern. In-memory on the coordinator — fine and the cheapest, but a coordinator restart loses the last list for one cycle; Redis costs ~nothing and removes that. Cite caching.html.

Real-system shape: this is essentially what Twitter/X's "trends" and similar trending pipelines do — a partitioned stream into a streaming engine running heavy-hitter sketches, with a tiny materialized list served from a cache; the batch path is the warehouse re-running an exact GROUP BY ... ORDER BY count DESC LIMIT k for reporting.

State, topic config & query API (C9; cite api-design-patterns.html)

# --- Kafka topic for the page-view firehose ---
topic: page-views
  partitions: 100                 # = shard count; key = hash(url) so all events for a URL co-locate
  replication.factor: 3           # survive a broker loss
  retention.ms: 7_200_000         # 2 h -- covers the 1 h window + replay-since-checkpoint headroom
  compression.type: lz4
  # consumers: one consumer group "topk-shards"; each shard owns ~1 partition; rebalance = failover.

# --- Per-shard in-RAM state (lives on the owning shard; checkpointed to S3 every 60 s) ---
struct ShardState {
    sketch_bank: [CountMinSketch; 60]   # one per minute; ~540 KB each -> ~32 MB; index = (now_minute % 60)
    distinct_est: HyperLogLog           # cheap; trips the "re-size the CMS" alarm if distinct keys climb
    heap: MinHeap<(url, est_count)>     # size K (=100..few-hundred); the local top-K candidate set
    in_heap: HashMap    # K entries; membership + current count
    consumed_offsets: map   # checkpointed alongside, for clean replay-since-checkpoint
}

# --- Checkpoint object (S3) ---
s3://topk-checkpoints/{shard_id}/{epoch_minute}.snap   # flat dump of ShardState + consumed_offsets

# --- Materialized output (Redis), refreshed by the coordinator every ~60 s ---
SET trending:hourly  {"as_of": "2026-05-10T12:34:00Z", "window_sec": 3600,
                      "items": [{"url": "...", "approx_count": 5_500_000, "rank": 1}, ...]}   # <= a few KB

# --- Read API: "show me the current top-K"  (cite api-design-patterns#pagination, #errors, #versioning) ---
GET /v1/trending?window=1h&k=100&cursor=...
  -> 200 { as_of, window_sec, items:[{url, approx_count, rank}], next_cursor }   # cursor pagination per #pagination
  -> 400  { type:"about:blank#bad-window", title:"unsupported window", ... }     # problem+json per #errors
  -> 503  (coordinator has no fresh list; client may serve last-known with the stale flag)
  # window in {5m, 1h} (= which subset of the 60 sketches to sum); k <= 1000; results are approximate (header X-Approx: true).
  # versioning: URL prefix /v1/ per #versioning. No idempotency/concurrency concerns -- it's a read-only endpoint.

API conventions (pagination, error envelope, versioning) cite api-design-patterns.html#pagination / #errors / #versioning — not restated; only the scenario-specific parameters (window, k, the X-Approx header) are inline.

Cross-cutting coverage (X1–X8; cite cross-cutting.html)

Degradation contract

When a shard dies → its key range goes dark until the standby (warm replica on the same Kafka partitions, checkpointing every 60 s) takes over (seconds); during the gap the coordinator publishes the merged list flagged {"degraded": ["shard 7"]} rather than silently dropping ~10% of the hash space. When the coordinator stalls → shards keep folding (nothing lost); the dashboard serves the last list with an as_of timestamp and goes amber past a staleness threshold; a stateless standby coordinator takes over instantly. When ingest spikes 10× → if the shard fleet can't be scaled out fast enough, each shard samples the stream (count the dropped fraction, surface it as a per-shard sampled_pct gauge so the published counts can be scaled / flagged) rather than letting Kafka lag unbounded — the trending ranking survives sampling far better than absolute counts do. When the sketch is found mis-sized (the distinct-keys HLL or the canary-vs-exact compare trips) → raise an alarm; the counts stay usable (over-count is bounded, just looser) until the next maintenance window applies a bigger w (a config push + rolling restart, replaying from checkpoint). The query API never throws on a hot-path read: no fresh list → 503 + the client serves last-known; unknown window → 400. The invariant: the trending list degrades (staler, looser-counted, missing a slice and saying so) but is never silently wrong and the exact batch path is always available as the source of truth. Cite failure-modes#slow-dependency + #backlog-overflow.

Operations (C6)

  • Metrics (SLIs → NFRs). Per-shard Kafka consumer lag (→ the freshness NFR — lag × event-rate is the staleness); coordinator emit-age / publish-interval (→ "updated every minute"); per-shard estimated-distinct-keys (HLL) and CMS load factor (→ the accuracy NFR — trips a re-size alarm before over-counts blow up); canary error rate = |CMS estimate − exact count| on a small sampled slice, per shard (→ accuracy, ground-truthed); per-shard ingest-rate imbalance (→ hot-key detection); per-shard sampled_pct (→ load-shedding visibility); query API p99 + 503 rate; checkpoint-write success + age (→ the recovery-time NFR); standby-lag for each shard's replica.
  • Rollout / canary for sketch sizing. Sketch parameters (w, d, the rotation count, K) are config, not code. To re-size: deploy the new parameters to one canary shard (it covers ~1/N of the key space), run it alongside the old shard on the same partitions for a window, compare the canary's top-K and counts against (a) the old shard's and (b) an exact count of that slice; promote fleet-wide only if the canary error is within the ε budget; rolling-restart shards (each rebuilds from its last checkpoint, so no gap). The picker binary itself rolls out as a standard service deploy gated on the lag / emit-age / canary-error SLIs.
  • Migration (single-node → sharded). From the Iteration-1 single-node tracker to the Iteration-2 sharded fleet: stand up the N-partition Kafka topic and the N shards consuming it in parallel with the old single-node consumer (both read the firehose — Kafka lets you have two consumer groups); the coordinator initially just mirrors the old single node's list, then switches to merging the N shard lists once they've warmed (one window); cut the dashboard over; retire the single node. Re-keying isn't an issue — the sketch is approximate and rebuilds in one window anyway, so there's no "migrate the old state" step. Rollback = point the coordinator back at the single node.
  • Headroom. Fleet sized at +30% over ceil(peak_qps / per-node-rate) — reason: burst traffic (a viral event spikes the firehose), rolling deploys (a fraction of shards restarting), and a node-down without immediate replacement. The sketch is sized for ~10× today's distinct-key estimate (the cost of being generous is linear in w and the absolute numbers are KB) — reason: a traffic-shape change (a crawler, a misconfigured client) can multiply distinct URLs overnight, and an under-sized sketch silently degrades accuracy whereas an over-sized one just wastes a few MB.

Anticipated pushbacks

Stress tests against the derivation above — depth rehearsal, not interview rehearsal.

What if the stream is 100× tomorrow — 100B events/hour (~28M events/sec)? (scale flip)

The memory side is a non-event: the sketch is O(1/ε), not O(N) — 60 rotating sketches per shard is still ~32 MB no matter how many events flow through them (the over-count bound ε·N grows, so to hold a 100×-larger N at the same absolute slack you'd want w ~100× bigger — that's still only ~50 MB per sketch, ~3 GB per shard for the bank; fine). The throughput side is just more shards: nodes = ceil(28M / per-node-rate) — if a node comfortably folds ~280K events/s (10× today's per-node load, still well under a core but now NIC-relevant at ~28 MB/s), that's ~100 shards; the coordinator merges 100×100 = 10K candidates, microseconds. The one genuinely new decision is hot-key skew at this scale: one mega-popular URL is now 28M/s×(its share) of events on one shard — if that's > a single NIC, salted-key splitting (the probe). Nothing structural breaks; you turn the shard knob and maybe the salt knob.

Why not just use exact counts — aren't sketches a premature optimization? (alt-engine)

Exact counting means a HashMap sized to the distinct-key count, which on an unbounded stream is unbounded — up to 148 GB worst case, and "realistic" 14.8 GB is one traffic-doubling from not fitting. The sketch is ~540 KB. That's not a marginal optimization; it's the difference between "fits on a laptop" and "doesn't fit on one machine." And the sketch buys two things exact counts can't: mergeability (the distributed fan-in and the windowing both depend on associativity — an exact hashmap doesn't merge cheaply and partial top-K lists don't merge correctly) and bounded streaming cost (O(d) per event, no rehashing). If a particular interview really has a bounded distinct-key count (say "at most 1M product SKUs ever") and exactness matters — then yes, an exact HashMap + heap is correct and simpler; we'd say so. Here the premise (every page view potentially a new URL) makes that bound false.

What changes if the top-K must be exact — e.g. it feeds billing or an SLA report? (regulatory / correctness)

That's a different SLO, and it's already on the page: the batch path (MapReduce / a warehouse GROUP BY url ORDER BY count DESC LIMIT k, partitioned by hash(url) so each reducer's slice is exact). The real-time sketch path stays as the dashboard's live view; the exact path produces the authoritative numbers after the hour closes (~seconds of compute over the event log). The two coexist: approximate-and-now for the widget, exact-and-soon for the ledger — never try to make one structure do both. If the exactness requirement extends to real-time exact counts (rare), you're back to "bound the distinct keys and run an exact hashmap, sharded by key" — possible only if that bound exists. Cite the streaming-vs-batch tradeoff table above.

A shard dies mid-window — what does the dashboard actually show? (failure)

For the seconds until the standby takes over: the coordinator still publishes a list, but it's the merge of the N−1 live shards, flagged {"degraded": ["shard 7"]} — so whatever was popular in shard 7's ~10% of the hash space is temporarily under-represented, and the dashboard can show a banner instead of pretending the list is complete. The standby (warm replica on the same Kafka partitions, checkpointing its sketch bank to S3 every 60 s) rebuilds from the latest checkpoint + replay-since-checkpointed-offset — that's ~32 MB of state + ~1 min of events, so seconds, not minutes. Kafka's consumer-group rebalance is what makes the failover clean (the standby just resumes the partitions). The exact batch path is unaffected — it reads the event log, which lost nothing. Cite failure-modes#bad-deploy.

Where's the cost, and which knob moves it? (cost)

There isn't a meaningful one. The real-time fleet is ~10 small nodes (a few hundred MB of sketch total, < 1 core total) — tens of dollars a month; the batch path borrows ~100 cores for ~5 s an hour — cents; the Redis serving store holds one tiny key. The dominant nearby cost is Kafka retention — storing 1B events/hour × ~100 B for the 2 h retention window is ~200 GB resident, trivial, and that log exists for the whole platform anyway, not for this pipeline (cite cost-shapes.html#stream-processing + #telemetry). The cost-shape lesson: a streaming-aggregation primitive is cheap; its value is in what it avoids — here, the 148 GB box you'd otherwise need, and the 30-minute batch latency. The only knob with a real bill is sketch width w (accuracy vs. RAM — linear, and the absolute numbers are KB–MB), and the rotation count (windowing granularity vs. RAM — 60 vs. 600 sketches is 32 MB vs. 320 MB; still nothing). Don't size this by its own bill.

How do you roll out the sharded version from the single-node design without an incident? (migration)

Run them in parallel. Kafka lets two consumer groups read the same firehose, so stand up the N-partition topic (key = hash(url)) and the N shards alongside the existing single-node consumer; the coordinator starts by just echoing the single node's list (zero behaviour change), then — once the shards have folded one full window — switches to merging the N shard lists; cut the dashboard over; decommission the single node. There's no "migrate the old state" step because the sketch is approximate and rebuilds in one window anyway. Rollback at any point = point the coordinator back at the single node (it's still running until you decommission it). Then re-size the per-shard sketches via the canary protocol if the per-shard distinct-key count warrants it.

The top-K looks wrong — some URL that shouldn't be there is #3. Where do you look? (monitoring / debug)

Decision tree off the SLI dashboard: (1) Is it a near-threshold false positive? — check the canary error metric (|CMS estimate − exact| on a sampled slice) and the CMS load factor / distinct-keys HLL for that URL's shard; if distinct keys climbed past what w can hold, collisions are inflating counts → re-size (canary protocol). (2) Is a shard degraded? — check the coordinator's degraded flag and per-shard heartbeat; a missing shard skews the merge. (3) Is the list stale? — check as_of / coordinator emit-age; a stuck coordinator shows an old list that "looks wrong" because reality moved. (4) Is the stream being sampled? — check per-shard sampled_pct; under load-shed, absolute counts are scaled estimates and the ranking can wobble at the margins. (5) Is it actually right? — cross-check against the exact batch path's last completed hour; the sketch never under-counts, so a true heavy hitter is never missing — a spurious extra entry is always over-count noise or a stale/degraded read, and every one of those has a gauge. Cite failure-modes#data-corruption.

Key Takeaways

  1. Probabilistic data structures trade accuracy for massive memory savings. A Count-Min Sketch uses 540 KB to approximate what would take 148 GB with exact counting. The error is bounded and quantifiable.
  2. Partition by the grouping key. When distributing, partition events by hash(URL) so each worker sees the complete count for its URLs. Random partitioning splits counts across workers, making local top-K unreliable.
  3. Streaming and batch serve different needs. Use streaming (CMS + heap) for real-time dashboards where approximate results are fine. Use batch (MapReduce) for exact counts needed for billing or SLA reporting.
  4. The min-heap is the key to top-K. You do not need to sort all items. A min-heap of size K lets you find the top-K from a stream in O(N log K) time and O(K) space. For K=100, log K ≈ 7, which is almost free.
  5. Time windows require rotation. To support "last hour" queries, maintain multiple sketches (one per minute) and rotate them. Querying sums across all active sketches. This adds 60x memory but 60 x 540 KB = 32 MB is still tiny.
  6. Always do the memory math first. The entire design hinges on whether exact counts fit in memory. If they do (10M uniques = 1.5 GB), use exact counting. If they might not (100M+ uniques), probabilistic structures become necessary.
  7. The LC #347 kernel survives at scale by trading three properties. The textbook "hashmap of counts → heap" becomes a Count-Min/Space-Saving sketch + min-heap, distributed and mergeable: memory goes from O(distinct keys) to O(1/ε); exactness becomes a bounded over-count; and you gain associativity — which is precisely what the distributed fan-in and the rotating-window machinery are built on. Name that trade in an interview; don't just say "use a Count-Min Sketch."
