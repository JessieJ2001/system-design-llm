Serving Llama-3 70B to 2 Million Users

Design a real-time inference system that serves a 70-billion-parameter LLM with streaming output, multi-turn conversations, and sub-second time-to-first-token — at scale.

NALSD Walkthrough • ~20 min read

The Scenario

Your company is launching a ChatGPT-style product powered by Llama-3 70B. Users send text messages and receive streaming responses. Conversations are multi-turn — the model must see the full conversation history for every message. The interview question: design the GPU inference serving layer — from the moment a tokenized prompt arrives to the moment the last token streams back.

The Problem

LLM inference is unlike any traditional web service. A single request can occupy a GPU for 5–30 seconds, consume gigabytes of memory, and produce output one token at a time. The model itself is too large to fit on a single GPU. You need to split it across multiple GPUs, batch requests efficiently, and manage a massive, dynamically-growing cache of intermediate computations — all while maintaining sub-second responsiveness for streaming output.

The core tension: GPU memory is the bottleneck, not compute. A single H100 has 989 TFLOP/s of FP16 compute but only 80 GB of memory. The model weights alone eat 140 GB. The intermediate state (KV cache) for each active conversation eats hundreds of megabytes to gigabytes. Every design decision revolves around fitting more useful work into limited GPU memory.

Constraints

  • Model: Llama-3 70B — 80 transformer layers, 64 query heads, 8 KV heads (Grouped Query Attention), d_model = 8192, head_dim = 128
  • Precision: FP16 (2 bytes per parameter)
  • GPU: NVIDIA H100 80 GB HBM3 — 3.35 TB/s memory bandwidth, 989 TFLOP/s FP16
  • DAU: 2 million
  • Avg messages/user/day: 40 (mix of short and long conversations)
  • Avg input: 1,000 tokens; avg output: 500 tokens
  • SLO: p99 time-to-first-token (TTFT) < 1 second, p99 inter-token latency < 50 ms
  • Context window: up to 32K tokens

Essential Formulas

Every calculation in this page derives from these formulas. Refer back here when you see a number you can’t trace.

Hardware Constants (H100 80GB)

HBM capacity:     80 GB per GPU
HBM bandwidth:    3,350 GB/s (= 3.35 TB/s) per GPU
Peak FP16 compute: 989 TFLOP/s per GPU
MFU (utilization): 0.4 (40%, conservative estimate)
Effective compute: 989 × 0.4 = 395.6 TFLOP/s per GPU

Model Constants (Llama-3 70B)

Parameters:      70B
Model size (FP16): 70B × 2 bytes = 140 GB
Model size (INT8): 70B × 1 byte  = 70 GB
Layers:          80
d_model:         8,192
d_head:          128 (numbers per head per token)
num_q_heads:     64 (= d_model / d_head)
num_kv_heads:    8 (GQA: 64/8 = 8 Q heads per KV head)

Core Formulas

=== Per-GPU Memory Budget ===
  weights_per_GPU   = model_size / TP
  available_for_KV  = 80 GB - weights_per_GPU - overhead (~10 GB)

=== KV Cache ===
  KV_per_token      = 2 × layers × num_kv_heads × d_head × 2 bytes
                    = 2 × 80 × 8 × 128 × 2 = 320 KB (total across GPUs)
  KV_per_token/GPU  = 320 KB / TP
  KV_per_req/GPU    = KV_per_token/GPU × avg_tokens

=== Max Batch Size ===
  max_batch = available_for_KV / KV_per_req/GPU

=== Decode Step Time (per decode step, per GPU) ===
  Load time    = (weights_per_GPU + batch × KV_per_req/GPU) / 3,350 GB/s
                 [weights: constant. KV reads: scales with batch]
  Compute time = (2 × params × batch + attn_FLOPs) / TP / 395.6 TFLOP/s
                 [weight matmuls + attention compute, per GPU]
  Decode step time = max(Load, Compute)   [double buffering]

=== Attention FLOPs (per layer, per sequence, c cached tokens) ===
  scores (q×K^T):   2 × d_head × c × num_q_heads
  weighted sum (w×V): 2 × d_head × c × num_q_heads
  Total per layer:    4 × d_head × c × num_q_heads
  Total all layers:   4 × 128 × c × 64 × 80 × batch = 3.93G × c_avg × batch

=== Attention HBM Reads (per layer, per sequence) ===
  K_all + V_all:      2 × num_kv_heads × d_head × c × 2 bytes
  (scales with num_kv_heads=8, NOT num_q_heads=64 — GQA saves 8×)

=== Prefill Time ===
  FLOPs  = 2 × params × input_tokens
  Load   = model_size / TP / 3,350 GB/s
  Compute = FLOPs / TP / 395.6 TFLOP/s
  Prefill = max(Load, Compute)   [compute-bound for >~189 tokens]

=== Request Latency & Throughput ===
  Latency    = prefill_time + output_tokens × decode_step_time
  Throughput = batch / latency  (per node)
  Nodes      = peak_QPS / throughput_per_node

=== Crossover Points ===
  Load-bound ↔ Compute-bound crossover:
    Set Load time = Compute time, solve for batch (or tokens).
    At TP=4 with avg 1,500 cached tokens: crossover ≈ batch 189.
    Below: load-bound (good for batching).
    Above: compute-bound (throughput plateaus).

=== Chunked Prefill ===
  chunk_size: chosen so mixed step time < inter-token SLO (50ms)
    Mixed step time = max(10.4ms, 2×70B×(chunk_size + decode_reqs) / 1,582T)
    (mixed = prefill chunk + decode tokens in same forward pass)
  chunk_count = ceil(total_input_tokens / chunk_size)
  Total prefill = chunk_count × mixed_step_time

=== Speculative Decoding ===
  E[tokens per iteration] = 1 / (1 - α)   [α = acceptance rate]
  Iteration cost = K × draft_step_time + 1 × target_step_time
  Effective ms/token = iteration_cost / E[tokens]
  Speedup = target_step_time / effective_ms_per_token

Back-of-Envelope Math

Before designing anything, we need to understand the fundamental resource constraints. The model’s 70 billion parameters dominate GPU memory, and the forward pass computation determines how fast we can generate tokens. Let’s quantify the basics.

Model Memory

Model weights:
  70B params × 2 bytes (FP16) = 140 GB

Single H100 = 80 GB HBM
  140 GB > 80 GB → doesn't fit on 1 GPU

Tensor Parallelism with TP=2 (2 H100s):
  140 / 2 = 70 GB per GPU
  Remaining per GPU: 80 - 70 = 10 GB for activations

  Without KV cache (our starting point in the Naive Design),
  10 GB is plenty for activations — the model processes one
  layer at a time, holding only that layer's intermediates:
    Peak activation memory ≈ batch × seq_len × d_model × 2B × ~4
    = 1 × 1,500 × 8,192 × 2 × 4 ≈ 98 MB  (well within 10 GB)

    (2B = 2 bytes per value — FP16 is 16 bits = 2 bytes.
     The ×4 is a rough empirical multiplier — NOT related to TP.
     It means: peak activation memory during one layer ≈ 4× the
     base activation footprint [batch, seq_len, d_model, 2B].

     This is NOT literally "4 tensors of the same size." During a
     layer's forward pass, several intermediate tensors exist at
     different times — the residual stream (kept for the skip
     connection), attention outputs, and the MLP intermediate.
     The MLP intermediate is the largest: the FFN expands the
     hidden state from d_model (8,192) to 28,672 (≈3.5× d_model)
     via a linear layer, applies SiLU, then projects back down.
     These tensors vary in size and don't all coexist, but their
     combined peak footprint works out to roughly 4× the base.

     TP=2 splits these across 2 GPUs (~49 MB each), but the
     total is still ~98 MB.)

  TP=2 works when we DON'T store KV cache. Once we introduce
  KV cache (Iteration 0), we'll need TP=4 for the extra memory.

"Node" = one physical server with 2 H100 GPUs,
connected via NVLink (900 GB/s inter-GPU bandwidth).
The 2 GPUs work together (tensor parallelism) to serve ONE
model replica. Node ≠ GPU. Node = 2 GPUs = 1 model copy.

Forward Pass Basics

=== What Happens Each Step ===

A forward pass through Llama-3 70B:
1. Loads ALL 140 GB of weights from GPU HBM (layer by layer)
2. Processes the input tokens through 80 transformer layers
3. Produces output logits for the next token

Why ALL weights every step? GPU compute units (registers, SRAM)
are tiny (~20-40 MB). They can't hold 140 GB. So each step
streams weights layer by layer:
  Load layer 0 attention weights → compute → discard
  Load layer 0 FFN weights → compute → discard
  Load layer 1 attention weights → compute → discard
  ... repeat through all 80 layers ...
  Done: 140 GB streamed, one forward pass complete.

Each layer is ~1.75 GB (140 GB / 80 layers), far too large
for SRAM (~20-40 MB). So the GPU streams each layer's weights
from HBM through SRAM in small tiles (a few MB at a time),
computes on each tile, then discards it. The conceptual unit
of work is "one layer at a time," but SRAM never holds a full
layer — just small pieces of it in flight.

=== Two Bottlenecks ===

  Memory bandwidth: loading 140 GB of weights from HBM
    At TP=2: 140 GB / (2 × 3.35 TB/s) = 140 / 6.7 = 20.9 ms

  Compute: matrix multiplications across all layers
    FLOPs = 2 × num_params × num_tokens_processed
    (See “Deriving 2 × params × tokens” below.)
    At TP=2: capacity = 2 × 989 × 0.4 MFU = 791 TFLOP/s
    (MFU = Model FLOPs Utilization — fraction of peak FLOP/s
     achieved in practice. 40% is conservative; range 30-60%.)

Which bottleneck dominates depends on num_tokens_processed:

  Small (1 token):
    Load:    20.9 ms
    Compute: 2 × 70B × 1 / 791T = 0.18 ms
    → Memory-bandwidth-bound (load is 116× slower)

  Large (1,000 tokens):
    Load:    20.9 ms
    Compute: 2 × 70B × 1,000 / 791T = 177 ms
    → Compute-bound (compute is 8.5× slower)

This distinction is critical: without KV cache, every
generation step processes ALL tokens (compute-bound).
With KV cache (Iteration 0), each step processes just 1
new token (memory-bandwidth-bound). This fundamentally
changes how batching works.

FLOPs = "2 × params × tokens"

X = one token's activation vector [1 × 8192] — the token's representation flowing through the model. (Layer 1: X comes from embedding. Later layers: X is the previous layer's output.)

Consider one matmul, e.g. Q = X × W_Q:

X is [1 × 8192], W_Q is [8192 × 8192] (67M parameters).

Each output element:
  Q[j] = X[0]×W[0][j] + X[1]×W[1][j] + ... + X[8191]×W[8191][j]
  = 8192 multiplies + 8192 adds = 16,384 FLOPs

Q has 8192 elements → total = 2 × 8192 × 8192 = 134M FLOPs.
W_Q has 8192 × 8192 = 67M parameters → 2 FLOPs per parameter.

Each of W_Q's 67M params is used exactly once. W_K is a separate matrix with its own 8M params, each used once too. Across the full model:

Per layer — 7 separate matmuls, each with its own weights:
  Q = X × W_Q       (67M params, used once per token)
  K = X × W_K       ( 8M params, used once)
  V = X × W_V       ( 8M params, used once)
  out = attn × W_O   (67M params, used once)
  up = X × W_up     (235M params, used once)
  gate = X × W_gate (235M params, used once)
  down = h × W_down (235M params, used once)
  ──────────────────────
  Per layer total: ~855M params

80 layers × 855M = ~68B + ~2B embeddings = ~70B params
Every param used once → 2 × 70B = 140 GFLOP per token
For 1,000 tokens: 140 GFLOP × 1,000 = 140 TFLOP

(Q×K^T and scores×V use activations, not weight params,
so they're not counted. Softmax/layernorm <5% — ignored.)

Naive Design: No KV Cache

Let’s start with the simplest possible approach: load the model onto 2 GPUs and generate tokens by running a full forward pass over the entire sequence at each step. No caching, no batching.

Why a Request Queue in Front of the GPU?

GPU compute and memory are fixed, but request arrival is bursty and unpredictable. The queue solves four problems:

  1. Admission control — each in-flight request consumes GPU memory (activations, KV cache). Letting unlimited requests hit the GPU simultaneously causes OOM or thrashing. The queue only admits what the GPU can handle right now. (Note: admission control is not the same as rate limiting. Rate limiting caps requests per unit of time per client, regardless of GPU state — it’s about fairness across callers. Admission control is capacity-aware: it checks current resource state, e.g., “the GPU has 12 GB free, this request needs 8 GB, admit it.” Both typically coexist — rate limiting at the API gateway, admission control at the GPU queue.)
  2. Decoupling arrival from processing — the GPU processes at a roughly fixed rate, but requests arrive in bursts. The queue absorbs spikes so excess requests wait instead of being dropped.
  3. Backpressure signal — a growing queue tells the load balancer or autoscaler “we’re saturated” so it can reject, reroute, or scale up. Without a queue, overload is only discovered when the GPU fails.
  4. Fairness — FIFO ordering ensures requests are served in arrival order so no request starves while later ones jump ahead.

In this naive design, the GPU processes 1 request at a time, so every other request must wait somewhere — the queue is that “somewhere.” Later iterations add batching, but even then the batch size is bounded by GPU memory, so a queue remains necessary.

How It Works

  1. Requests queue up in a FIFO queue.
  2. The model runs on 2 H100 GPUs (TP=2). 140 GB / 2 = 70 GB per GPU, leaving 10 GB for activations.
  3. To generate each output token, the model runs a full forward pass over the entire sequence so far (all input tokens + all previously generated tokens). There is no cache — the model recomputes K,V for every previous token at every step.
  4. Response is returned only after all tokens are generated. No streaming.

Why This Is Catastrophically Slow

Without KV cache, the model has no memory of previous tokens.
To generate each new output token, it must re-process the ENTIRE
sequence from scratch — like re-reading an entire book every
time you want to write the next word of a summary.

Key regime: no KV cache = many tokens per step = COMPUTE-BOUND.
(With KV cache, each step processes just 1 token, flipping to
memory-bandwidth-bound. That regime change is why KV cache matters.)

For Llama-3 70B at TP=2 (791 TFLOP/s compute capacity):

  Token 1 (prefill 1,000 input tokens):
    FLOPs: 2 × 70B × 1,000 = 140 TFLOP
    Time: 140 / 791 = 177 ms (compute-bound)

  Token 2 (re-process 1,001 tokens): 177.2 ms
  Token 3 (re-process 1,002 tokens): 177.4 ms
  ...
  Token 500 (re-process 1,499 tokens):
    FLOPs: 2 × 70B × 1,499 = 210 TFLOP
    Time: 265 ms

  Total: sum of 177 ms .. 265 ms ≈ 500 × 221 ms = ~110 seconds

Compare: weight loading takes only 20.9 ms per step, but
the COMPUTATION takes 177-265 ms. Every step is compute-bound.

Batching Doesn’t Help

In a compute-bound regime, adding more sequences to the batch
proportionally increases compute per step:

  batch=1:  load = 21 ms, compute = 177 ms  → 1 token out
  batch=16: load = 21 ms, compute = 2,832 ms (177 ms x 16) → 16 tokens out

  Per-token: 177 ms either way!

The weight loading IS amortized (21 ms shared across 16
sequences), but since compute (177 ms) >> load (21 ms),
the saving is negligible (<1%).

This changes dramatically with KV cache (Iteration 0):
each step processes just 1 token per sequence, making it
memory-bandwidth-bound — and batching becomes powerful.

The cost of no caching. At ~0.009 req/sec per node (1 request / 110 sec), we need 2,778 / 0.009 ≈ 309,000 nodes (618,000 GPUs). At ~$2/GPU/hr, that’s ~$890M/month. KV cache (next) is not an optional optimization — it’s a necessity.

Iteration 0: KV Cache + Static Batching

The naive design recomputes attention from scratch at every step — 375,000 redundant K,V projections for 500 output tokens. The fix: cache the Key and Value vectors so each decode step processes only 1 new token.

KV Cache: Why It Exists

During prefill, the model computes Q (query), K (key), and V (value) for all input tokens in parallel. The attention output for token i uses Q[i] against ALL keys K[0..i] and values V[0..i]. The full Q matrix is computed, used immediately for attention, and then discarded within this same step — Q₀, Q₁, Q₂ are never needed again. After prefill, only K and V are kept in GPU memory.

During generation (decode), we produce one new token at a time. We no longer compute a full Q matrix — we compute a single Q vector [1 × d_model] for just the new token (from its own hidden state; it cannot reuse any previous token’s Q). That single Q vector is then multiplied against the full cached K from all previous tokens to get attention scores, and the scores are applied to the full cached V to get the attention output. K and V have not changed (the model’s weights are frozen, and the inputs that produced them are unchanged).

Why not cache Q too? Q is only used by the token that created it — Q₀ was used by “The” to compute its attention, and no future token ever references Q₀ again. K and V are the opposite: every future token attends to them. Think of it as: Q = “what am I looking for?” (used once, by its own token), K = “what do I contain?” (queried by every future token), V = “what do I provide?” (read by every future token). Caching Q would waste memory on values that are never reused.

=== Prefill vs Decode: Same Operations, Different Sizes ===

Both phases perform the SAME operations. The difference is input size.

PREFILL (all input tokens at once, e.g. n=1000):
  1. Q = X × W_Q        [n × d] × [d × d]     → [n × d]
  2. K = X × W_K        [n × d] × [d × d_kv]  → [n × d_kv]
  3. V = X × W_V        [n × d] × [d × d_kv]  → [n × d_kv]
  4. scores = Q × K^T   [n × d] × [d × n]     → [n × n]
  5. weights = softmax(scores / √d_k)
  6. attn = weights × V [n × n] × [n × d_kv]  → [n × d_kv]
  7. out = attn × W_O   (project back)
  8. FFN: W_up, W_gate, W_down
  → Store K, V into cache. Discard Q.

DECODE (1 new token, cache has c previous tokens):
  1. q = x × W_Q        [1 × d] × [d × d]     → [1 × d]
  2. k = x × W_K        [1 × d] × [d × d_kv]  → [1 × d_kv]
  3. v = x × W_V        [1 × d] × [d × d_kv]  → [1 × d_kv]
  4. Append k,v to cache → K_all=[K_cached; k], V_all=[V_cached; v]
  5. scores = q × K_all^T  [1 × d] × [d × (c+1)] → [1 × (c+1)]
  6. weights = softmax(scores / √d_k)
  7. attn = weights × V_all  [1 × (c+1)] × [(c+1) × d_kv] → [1 × d_kv]
  8. out = attn × W_O
  9. FFN: W_up, W_gate, W_down

Steps 1-3 shrink from n tokens to 1 token — that's why decode
is so much cheaper and flips from compute-bound to
memory-bandwidth-bound.

Where the KV CACHE is actually used during decode:
  Step 4: append new k,v to the cache
  Step 5: q × K_all^T  ← reads ALL cached K vectors
  Step 7: weights × V_all ← reads ALL cached V vectors
  These two reads are why it's called the "KV" cache —
  K is read for scoring, V is read for the weighted sum.
Without KV cache (the naive design):
  To generate token 501, we must recompute K,V for ALL 500
  previous tokens → 500 × (W_K + W_V) matrix projections.
  At token 1000: 1000 projections.
  Total for 500 output tokens:
    501 + 502 + 503 + ... + 1000 ≈ 375,000 projections

With KV cache:
  At each step, compute K,V for ONLY the 1 new token,
  append to cache, then attend over the full cached K,V.
  Total for 500 output tokens: 500 projections

  Speedup: 375,000 / 500 = 750× fewer K,V projections

The Memory vs Compute Tradeoff

The KV cache trades memory for compute. We store intermediate results in GPU HBM so we don’t have to recompute them.

=== Latency Formula (applies to BOTH prefill and decode) ===

A "step" = one full forward pass through ALL 80 layers.
Each request requires two SEPARATE phases, done sequentially:

  1. PREFILL (1 step): process all input tokens in parallel.
     Input shape: [1,000 × d]. Produces first output token + fills KV cache.
     Compute-bound (many tokens → compute >> load).

  2. DECODE (499 steps): produce 1 new token per step, reading KV cache.
     Input shape: [1 × d]. Each step appends to KV cache.
     Memory-bandwidth-bound (1 token → load >> compute).

  Total: 1 prefill + 499 decode = 500 forward passes.

The SAME formula applies to both phases (only "tokens" changes).
Each step is composed of:

  For each of 80 layers:
    1. Load layer's attention weights (~0.9 GB) from HBM
    2. Attention matmuls: Q, K, V, O projections
    3. All-reduce (sync partial results across TP GPUs)
    4. Load layer's FFN weights (~0.9 GB) from HBM
    5. FFN matmuls: W_up, W_gate, W_down
    6. All-reduce
  = 80 layers × 7 matmuls × 2 all-reduces = one step done

The formulas below roll all 80 layers into aggregate numbers:
  "Load 140 GB" = sum of all layers' weights (80 × ~1.75 GB)
  "Compute 2 × 70B × tokens" = sum of all matmuls across all layers

BOTH prefill and decode load ALL 140 GB of weights every step.
The weights don't change between phases — same load, same time.
The only difference is how much compute happens per byte loaded:
  Prefill (1,000 tokens): compute >> load → compute-bound
  Decode (1 token):       load >> compute → memory-bandwidth-bound

Every step has two costs:

  Load    = model_size / (bandwidth_per_GPU × TP)
          = 140 GB / (3.35 TB/s × TP)
          CONSTANT — always load ALL weights, regardless of token count.

  Compute = 2 × params × tokens_processed / (peak_TFLOP/s × TP × MFU)
          = 2 × 70B × tokens / (989 TFLOP/s × TP × 0.4)
          SCALES with how many tokens are processed in this step.

  Decode step time ≈ max(Load, Compute)

  Why max() and not sum()? The GPU overlaps load and compute
  using DOUBLE BUFFERING.

  All 140 GB of weights are ALREADY in HBM. "Load" means streaming
  weight tiles from HBM → SRAM (the tiny on-chip memory where
  compute happens). SRAM is split into two buffers (A and B).
  While compute units work on a tile in buffer A, the memory
  controller loads the next tile into buffer B. Then they swap:

    Buffer A: [compute tile 1] [ load tile 3  ] [compute tile 3] ...
    Buffer B: [ load tile 2  ] [compute tile 2] [ load tile 4  ] ...

  This works because:
    • SRAM is partitioned — two buffers, not one shared block
    • Load units and compute units are SEPARATE hardware —
      they physically operate in parallel
    • Each buffer only holds one small tile, not a whole layer

  Prefill (many tokens): compute per tile is slow, so the next
    tile arrives before compute finishes → load hidden behind compute.
  Decode (1 token): compute per tile is instant, so you just wait
    for tiles to arrive from HBM → compute hidden behind load.

  max() is an approximation — double buffering doesn't perfectly
  overlap (switching overhead, first tile has no overlap). This
  imperfect overlap is part of why MFU is ~40%, not 100%.

At TP=4:
  Load    = 140 / 13.4 (= 3.35 * 4) = 10.4 ms  (same for both phases)
  Compute = 140 GFLOP × tokens / 1,582 TFLOP/s

  Prefill (1,000 tokens):
    Compute = 140G × 1,000 / 1,582T = 88 ms
    max(10.4, 88) → 88 ms  ← COMPUTE-BOUND (many tokens)

  Decode (1 token):
    Compute = 140G × 1 / 1,582T = 0.09 ms
    max(10.4, 0.09) → 10.4 ms  ← MEMORY-BANDWIDTH-BOUND (1 token)

The crossover point (where load = compute) is at ~118 tokens:
  10.4 ms = 140G × N / 1,582T → N ≈ 118 tokens
  <118 tokens: memory-bandwidth-bound (decode regime)
  >118 tokens: compute-bound (prefill regime)
Without KV cache (500 output tokens, TP=2):
  Re-prefill every step: ~110 sec per request
  Compute-bound throughout. Batching doesn't help.

With KV cache:
  Prefill once (88 ms at TP=4) + 499 decode steps
  Total: 88 ms + 499 × 10.4 ms ≈ 5.3 sec per request

  Per-request speedup: 110 sec → 5.3 sec = ~20× faster
  KV cache is a LATENCY optimization, not just throughput.

But we PAY memory: 480 MB per request at 1,500 tokens
(or up to 10.24 GB pre-allocated for 32K max context).

Why TP=2 is now IMPOSSIBLE (purely a memory problem):
  TP=2: each GPU holds 70 GB weights, leaving only 10 GB free.
  KV per request per GPU: 10.24 GB / 2 = 5.12 GB
  Max batch = 10 / 5.12 ≈ 1 request. Unusable!

  It's NOT a latency or throughput issue — a single request
  would still be fast. The problem is we can only fit ONE
  request's KV cache in memory. No batching = no throughput.

Why TP=4 WORKS (more GPUs = more total memory):
  TP=4: each GPU holds 35 GB weights, leaving 45 GB free.
  KV per request per GPU: 10.24 GB / 4 = 2.56 GB
  Max batch = ~35 GB / 2.56 GB ≈ 13-16 concurrent requests.
  Now we can batch, and throughput follows.

How concurrency drives throughput (Little's Law):
  Global concurrency needed:
    2,778 req/sec × 5.3 sec = ~14,723 in-flight requests

  Per node (TP=4, batch=16):
    node_throughput = batch / latency = 16 / 5.3 = ~3 req/sec

What Is All-Reduce?

With tensor parallelism, each GPU computes a partial result (e.g., GPU 0 computes heads 0-7, GPU 1 computes heads 8-15). After each layer, the partial results must be combined so every GPU has the full output to feed into the next layer. This combining step is called all-reduce.

Example: TP=4, output vector = [a, b, c, d] split across 4 GPUs

  GPU 0 has [a, 0, 0, 0]  (computed its 1/4)
  GPU 1 has [0, b, 0, 0]
  GPU 2 has [0, 0, c, 0]
  GPU 3 has [0, 0, 0, d]

  All-reduce: every GPU sends its piece to all others.
  After: all 4 GPUs have [a, b, c, d].

  Cost per all-reduce:
    Data moved: d_model × batch × 2 bytes
    = 8192 × 16 × 2 = 256 KB (tiny)
    (batch=16 = max concurrent requests, derived from
     the GPU memory budget above: ~35 GB / 2.56 GB per req.)
    At 900 GB/s NVLink: <0.001 ms per sync

  But it happens TWICE per layer (after attention + after FFN):
    80 layers × 2 = 160 syncs per forward pass
    Total: 160 × 0.001 ms = 0.16 ms (negligible at TP=4)

  At TP=8: more GPUs in the ring → more hops per sync.
    Each sync takes ~2× longer: 160 × 0.002 = 0.32 ms.
    Still small vs 5.2 ms decode. But at TP=16 over
    InfiniBand (50 GB/s): 160 × 0.04 ms = 6.4 ms
    — that DOUBLES the decode step time. Unacceptable.

  Weight load: 140 GB / (3.35 TB/s × 4) = 10.4 ms
  Compute:     2 × 70B × 1 / 1,582 TFLOP/s = 0.09 ms
  Load is 116× slower → memory-bandwidth-bound!

  batch=16: 10.4 ms/step → 1,538 tokens/sec
  (Same load time per step, but more tokens produced!)

  2 × 70B × B / 1,582 TFLOP/s = 10.4 ms → B ≈ 117
  Up to batch≈117, decode is memory-bound (linear scaling).

Per request (1K input + 500 output):
  Prefill: 2 × 70B × 1,000 / 1,582 = 88 ms (compute-bound)
  Decode: 499 steps × 10.4 ms = 5,190 ms
  Total: 88 + 5,190 = 5,278 ms ≈ 5.3 sec per request

  (vs 110 sec without KV cache — 21× faster!)

Iteration 1: Continuous Batching

KV cache (Iteration 0) made each request 21× faster, but static batching still wastes GPU slots — finished requests wait for the slowest one before their slot can be reused. Continuous batching fixes this.

Continuous Batching (Orca)

Static batching waits for the entire batch to finish before releasing any slot. Continuous batching (introduced in the Orca paper, 2022) fixes this by checking after every decode step which sequences have finished, immediately evicting them and admitting new requests from the queue.

Each sequence maintains its own KV cache at its own length — no padding needed. A sequence at 200 tokens uses 200 tokens of cache; a sequence at 2,000 tokens uses 2,000. They share the same batch dimension for the matrix multiply but have independent lifetimes.

def serve_loop(model, queue, max_batch=16):
    active = []

    while True:
        # Decode one step for all active sequences
        for seq in active:
            token = model.decode_one(seq.kv_cache)
            seq.output.append(token)
            seq.stream_to_client(token)  # SSE push

        # Evict finished sequences, admit new ones
        active = [s for s in active if not s.done()]

        while len(active) < max_batch and not queue.empty():
            new_seq = queue.get()
            new_seq.kv_cache = model.prefill(new_seq.prompt)
            active.append(new_seq)

Impact: Each request uses its ACTUAL output length (avg 500 tokens, 5.3 sec) instead of waiting for the longest. Throughput: 16 / 5.3 = 3 req/sec per node — a 2× improvement over static batching, with zero hardware changes. Iteration 2: PagedAttention — Virtual Memory for KV Cache

Continuous batching solved scheduling waste. But KV cache memory management is still naive: each sequence pre-allocates a contiguous block for its maximum possible length. This wastes 60–90% of GPU memory.

Contiguous Allocation Waste

Without PagedAttention, the serving engine must reserve a contiguous memory region for each sequence’s KV cache. Since it doesn’t know how long the output will be, it pre-allocates for the worst case — the full context window.

Scenario: max context = 32K tokens
KV cache per token: 320 KB

Without PagedAttention:
  Each sequence pre-allocates: 32K × 320 KB = 10.24 GB
  With 45 GB per GPU: only 4 sequences fit
  But most sequences use ~1,500 tokens = 480 MB
  Waste: (10.24 GB - 0.48 GB) / 10.24 GB = 95.3% wasted!

  batch=4 → only 4 requests in flight → 4/5.3s = 0.75 req/sec
  Way below our target of 3 req/sec.

Even "right-sizing" to avg 1,500 tokens doesn't help much:
  Some sequences are 200 tokens, some 8,000.
  When a 200-token sequence finishes, it leaves a 480 MB
  "hole" that an 8,000-token sequence (2.56 GB) can't use.
  This is external fragmentation — just like OS memory
  before paging was invented.

Paging: Allocate on Demand

PagedAttention (introduced in the vLLM paper, 2023) borrows the page table idea from operating system virtual memory. Instead of allocating one contiguous region per sequence, it divides KV cache into fixed-size blocks (pages) and allocates them on demand as the sequence grows.

Each GPU manages its own block pool independently.
With TP=4, each GPU stores its share of the KV heads (2 out of 8).

Block size: 16 tokens per block, per GPU:
  16 tokens × 80 KB/token = 1.28 MB per block
  (80 KB/token/GPU = 320 KB total per token ÷ 4 GPUs)

How it works:
  1. Each GPU maintains its own free pool of 1.28 MB blocks
  2. Each sequence has a block table (like an OS page table)
     mapping logical block index → physical block address
  3. When a sequence needs a new token's KV cache:
     - If current block has room (<16 tokens), append there
     - If full, allocate a new block from the free pool
  4. When a sequence finishes, return all its blocks to the pool
  (All 4 GPUs allocate/free blocks in lockstep for the same sequence.)

Sequence of 1,500 tokens:
  Blocks needed per GPU: ceil(1500/16) = 94 blocks
  Memory per GPU: 94 × 1.28 MB = 120 MB  (exact fit, no waste!)

Sequence of 200 tokens:
  Blocks per GPU: ceil(200/16) = 13 → 16.6 MB per GPU

45 GB per GPU ÷ 1.28 MB/block = 35,156 blocks per GPU
  (block per GPU = 16 tokens × 80 KB = 1.28 MB; 80 KB = 320 KB / TP=4)
  Per request: ceil(1500/16) = 94 blocks × 1.28 MB = 120 MB per GPU

  Max batch = available_mem / KV_per_req_per_GPU
    KV_per_req_per_GPU = 120 MB = 0.12 GB (from line above)
    Without overhead: 45 GB / 0.12 GB = 375
    With ~10 GB overhead: 35 GB / 0.12 GB ≈ 291

  → Or ~4 sequences at 32K + many short sequences
  → Memory adapts to actual workload mix

Iteration 3: Flash Attention + Chunked Prefill

Our system handles steady-state load. But what happens when a user with a 32K-token conversation sends a new message? Prefill takes 2.8 seconds, during which all other requests in the batch are blocked from decoding. We need to fix the attention computation itself.

Standard Attention Is Memory-Wasteful

During prefill, the attention mechanism computes the full N×N score matrix: scores = Q @ KT. For a 32K-token prefill, this matrix is enormous:

Attention matrix size per head:
  32,768 × 32,768 × 2 bytes (FP16) = 2.15 GB

Llama-3 70B has 64 attention heads across 80 layers.
That's 64 × 80 = 5,120 attention computations per forward pass.

Even computing one head at a time:
  2.15 GB written to HBM + 2.15 GB read back = 4.3 GB of HBM traffic
  At 3.35 TB/s bandwidth: 4.3 GB / 3.35 TB/s = 1.28 ms per head

  Total HBM traffic for attention alone:
    1.28 ms × 5,120 heads = 6,554 ms ≈ 6.6 seconds

This is JUST the memory traffic for attention scores —
the actual compute on top of this makes 32K prefill take ~2.8s
even with the optimized GPU kernels.

The core problem: the N×N matrix is materialized in HBM (GPU main memory). HBM is large (80 GB) but relatively slow (3.35 TB/s). The GPU has a much faster memory — SRAM — but it is tiny.

Flash Attention: Tile in SRAM

Flash Attention (Dao et al., 2022) never writes the N×N attention matrix to HBM. Instead, it tiles the computation into blocks that fit entirely in GPU SRAM:

GPU memory hierarchy (H100):
  HBM:  80 GB capacity,  3.35 TB/s bandwidth
  SRAM: 50 MB capacity, ~33 TB/s bandwidth (10× faster!)

Flash Attention algorithm:
  1. Split Q into blocks of B_r = 128 rows  (Block-row: query tile size)
  2. Split K, V into blocks of B_c = 128 rows  (Block-col: key/value tile size)
     Why 128? All tiles must fit in SRAM simultaneously:
       Q tile:  128×128×2B = 32 KB
       K tile:  128×128×2B = 32 KB
       V tile:  128×128×2B = 32 KB
       Output:  128×128×2B = 32 KB
       Total: ~128 KB per tile operation << 50 MB SRAM
     Too large → won't fit in SRAM. Too small → poor tensor core utilization.
     128 is the sweet spot (power of 2, good matmul efficiency).

  3. For each Q block:
     For each K/V block:
       a. Load Q tile + K tile + V tile into SRAM  (~96 KB)
       b. Compute attention for this tile entirely in SRAM
       c. Accumulate partial result using online softmax
  4. Write only the final output: N × d_head to HBM

Key insight: "online softmax" lets us compute softmax
incrementally across tiles without ever materializing
the full row of scores.
Why O(N²) vs O(N)?

  Standard attention must STORE the full N×N matrix in HBM:
    Q = [N × d_head]    K^T = [d_head × N]
    scores = Q × K^T → [N × N]  (context_len × context_len)
    One score per (query token, key token) pair → written to HBM (O(N²))
    At 32K tokens: 32,768 × 32,768 × 2B = 2.15 GB per head!
    softmax(scores) needs ALL scores in a row to normalize
    → must keep entire [N × N] matrix in HBM before softmax

  Flash Attention NEVER writes N×N to HBM:
    Yes, there are (N/128)² tile operations — same O(N²) FLOPs.
    But each tile's [128×128] score matrix is:
      1. Computed in SRAM
      2. Immediately consumed (accumulated via online softmax)
      3. Discarded in SRAM — never written to HBM
    Only the accumulated output [128 × d_head] per Q tile is
    written to HBM. Total HBM writes: N/128 tiles × 128 × d_head
    = N × d_head → O(N).

    Flash Attention doesn't reduce compute (still O(N²) FLOPs).
    It reduces HBM I/O by keeping scores in SRAM.

Memory savings:
  Standard: O(N²) — 32K tokens → 2.15 GB per head
  Flash:    O(N)  — 32K tokens → N(32K) × d_head(128) × 2B = 8 MB per head
  Reduction: 268× less HBM usage

Speed improvement (fewer HBM round-trips):

  "HBM pass" = one read or write between HBM and SRAM.
  Standard attention runs separate GPU kernels per step,
  so intermediates bounce back to HBM between steps:

  Standard attention (each arrow = HBM read or write):
    1. READ  Q, K from HBM
    2. WRITE scores = Q×K^T to HBM       [N×N to HBM!]
    3. READ  scores back from HBM          [N×N from HBM!]
    4. WRITE softmax(scores) to HBM        [N×N to HBM again!]
    5. READ  softmax weights + V from HBM  [N×N + N×d from HBM!]
    6. WRITE output to HBM
    The N×N matrix bounces HBM ↔ SRAM multiple times.
    Total HBM I/O: ~4 passes over O(N²) data + 2 over O(N).

  Flash Attention (one fused kernel, tiles stay in SRAM):
    1. READ  Q, K, V tiles from HBM    [O(N) data]
    2. WRITE final output to HBM        [O(N) data]
    Scores computed and consumed entirely in SRAM — zero
    HBM round-trips for the N×N data.
    Total HBM I/O: 2 passes over O(N) data only.

  Result: 2-4× faster in practice (empirical, from Flash Attention
  papers — depends on N, d_head, hardware, implementation).

  Prefill 32K tokens (derived):
    Weight matmuls: 2 × 70B × 32,768 / 1,582 TFLOP/s = 2.9s
    + attention HBM overhead (standard): N² data bouncing through HBM
      adds significant time at 32K tokens (N² = 1 billion entries)
    + attention compute: 4 × 128 × 32K × 64 × 80 layers = 84 TFLOP
      84 / 1,582 = 53 ms (small vs weight matmuls)
    Standard attention HBM overhead (the N×N matrix bouncing through HBM):
      N×N matrix: 32,768² × 2B = 2.15 GB per head
      4 HBM passes per head per layer (write scores, read scores,
        write softmax, read softmax): 4 × 2.15 = 8.6 GB
      Per GPU (TP=4): 16 heads × 80 layers × 8.6 GB = 11,008 GB
      Time: 11,008 / 3,350 GB/s = 3.3s

    Standard total ≈ 2.9s (weights) + 3.3s (attn HBM) = 6.2s
    Flash Attention total ≈ 2.9s (weights) + 53 ms (attn compute) = 3.0s
      (attention HBM overhead eliminated — tiles stay in SRAM)
    Speedup: 6.2 / 3.0 ≈ 2.1×

  Flash Attention doesn't speed up the weight matmuls (2.9s).
  It speeds up the ATTENTION part by eliminating N² HBM I/O.
  The win is largest for long sequences where N² dominates.

  Flash Attention primarily benefits PREFILL, not decode:
    Prefill: Q=[N × d_head], so Q×K^T = [N × N] — huge matrix.
      Flash avoids materializing this in HBM. Big win.
    Decode:  q=[1 × d_head], so q×K^T = [1 × c] — just a vector.
      No N×N matrix to avoid. Standard attention is already
      cheap here (one dot product per cached token).

Chunked Prefill: The Problem

Even with Flash Attention, a 32K-token prefill still takes ~3 seconds. With continuous batching, prefill and decode run in the same forward pass — the GPU processes the prefill tokens alongside the decode tokens. But the prefill’s compute dominates the step:

Normal decode step (64 reqs × 1 token each): 12.7 ms
If one request prefills 32K tokens in the same step:
  Weight matmuls: 2 × 70B × (32,000 + 63) / 1,582T ≈ 2.8s
  The step takes 2.8s instead of 12.7ms.
  ALL 63 other users wait 2.8s for their next token!

The problem isn't memory — it's compute interference.
The prefill doesn't take a separate GPU; it runs in the SAME
batch, stretching step time for everyone.

This creates catastrophic latency spikes.

Why not decode during the prefill? A forward pass is synchronous — all tokens in the batch move through all 80 layers in lockstep. The GPU’s tensor cores and HBM bandwidth are shared resources; one forward pass uses all of them. You can’t have request A at layer 40 while request B is at layer 10 on the same GPU.

Chunked Prefill: The Solution

Solution: split the prefill into small chunks and process each chunk + decode tokens in one forward pass.

Common misconception: a chunk does not emit a prediction. The model predicts the first output token once, after the whole prompt is in the KV cache — never mid-prompt. Chunking changes only how the prefill compute is scheduled, never what is computed or what is output.

Concrete example — prompt "I like to go to" → model predicts "park"
tokens = [I][like][to][go][to]   (5 tokens),  chunk_size = 2

WITHOUT chunking — one prefill pass:
  process [I][like][to][go][to] through all 80 layers
    → KV written for positions 0..4
    → read logits at LAST position (4) → predict "park"
  (every position attends causally to all earlier positions)

WITH chunked prefill — chunk_count = ceil(5/2) = 3:
  Chunk 1 = [I][like]  → KV for pos 0..1.   NO prediction.
  Chunk 2 = [to][go]   → KV for pos 2..3.   NO prediction.
                         (chunk 2 still attends to chunk 1's KV —
                          chunks are NOT independent prompts)
  Chunk 3 = [to]       → KV for pos 4. Prompt fully cached.
                         → read logits at last pos → predict "park"
  then decode: "park" → next token → ...   (1 token/step)

Why this matters:
  - Prediction happens ONCE, at the end of the LAST chunk —
    not per-chunk, not mid-prompt.
  - Output is identical with or without chunking. Same prompt
    → same KV → same logits → same "park".
  - Chunking does not speed up X — X's prefill gets slightly
    SLOWER (more passes, more weight reloads). It exists so the
    OTHER batched users (A,B,C below) get a decode token between
    X's chunks instead of stalling for one huge prefill pass.
Choosing chunk size (derive from SLO):
  Each step processes: chunk_size prefill tokens + decode_reqs decode tokens
  Step time = max(Load, Compute)
    Load:    35 GB / 3,350 GB/s = 10.4 ms  (constant)
    Compute: 2 × 70B × (chunk + decode_reqs) / 1,582 TFLOP/s

  We need step time < 50ms. Why? From the scenario requirements:
  "p99 inter-token latency < 50 ms." Each step produces one token
  per user, so step time = inter-token latency for all decode users.
  With 63 decode requests in the batch (batch=64 minus 1 prefilling):
  Total tokens per step = chunk_size + 63 (each decode req = 1 token)

    chunk=512: total=512+63=575  2×70B×575/1,582T = 51 ms (exceeds SLO!)
    chunk=256: total=256+63=319  2×70B×319/1,582T = 28 ms (ok)
    chunk_size=128: total=128+63=191  2×70B×191/1,582T = 17 ms (comfortable)

  We use chunk_size=128 (17ms per step, well within 50ms SLO).
  chunk_count = ceil(total_tokens / chunk_size)
    1K context:  ceil(1000/128) = 8 chunks
    32K context: ceil(32768/128) = 256 chunks
  Tradeoff: smaller chunks = better decode latency, but more
  steps to finish prefill.

Without chunked prefill (User X sends 32K-token request):
  t=0ms:    Start prefill for User X (32K tokens in one pass)
  t=0-3.0s: GPU processes all 32K tokens.
            Users A, B, C: BLOCKED. No tokens generated.
  t=3.0s:   X's prefill done. Resume decode for A, B, C, X.
            → A, B, C experienced a 3-SECOND gap between tokens!

With chunked prefill (chunk_size=128, chunk_count=ceil(32K/128)=256):
  Each forward pass processes chunk + decode tokens together:

  t=0ms:   Forward pass 1: X's chunk 1 (128 tok) + A,B,C decode (63 tok)
           = 191 tokens through all 80 layers                    [~17ms]
           A, B, C each get their next decode token!
  t=17ms:  Forward pass 2: X's chunk 2 (128 tok) + A,B,C decode (63 tok)
           = 191 tokens                                          [~17ms]
           A, B, C get another decode token!
  ...
  (continues for 256 steps = chunk_count)

  X's total prefill: chunk_count(256) × 17ms = 4.4s (vs 3.0s unchunked)
  A, B, C get a decode token every ~17ms (within 50ms SLO)
  (vs one 3.0s stall without chunking)

Latency Impact

Chunked prefill trades slightly higher TTFT for one request to prevent catastrophic latency spikes for every other concurrent request:

Metric Without Chunked Prefill With Chunked Prefill
User X's TTFT (1K context) 88 ms (one forward pass: 2×70B×1000/1,582T) 136 ms (ceil(1000/128)=8 chunks × 17ms)
User X's TTFT (32K context) ~3.0s (one forward pass: 2×70B×32K/1,582T) ~4.4s (ceil(32768/128)=256 chunks × 17ms)
Users A, B, C inter-token latency 88 ms stall (1K) / 3.0s stall (32K) 17 ms per step (chunk_size=128, combined with decode)

The key tradeoff: Chunked prefill trades slightly higher TTFT for one request to prevent catastrophic latency spikes for everyone else. This is individual latency vs. system-wide latency — almost always worth it in production.

Fleet impact: Flash Attention + Chunked Prefill don't increase steady-state throughput (still 12 req/sec/node, still 928 H100s). Their value is latency quality — they fix p99 inter-token latency from >1 second to <50ms and enable long-context requests without disrupting the batch. This is what makes the system production-ready.

Iteration 4: INT8 Quantization — Halve the Fleet

Our bottleneck is memory bandwidth: loading 140 GB of model weights through HBM every single decode step. The weights dominate the data movement. What if we made the weights smaller?

Weight Quantization

Quantization reduces each weight from FP16 (16 bits, 2 bytes) to INT8 (8 bits, 1 byte). The idea is simple: most weights cluster in a narrow range, so we can represent them with fewer bits and a per-group scale factor.

Per-group quantization (group_size = 128):

For each group of 128 consecutive weights:
  1. Find the maximum absolute value:
     scale = max(|W_group|) / 127

  2. Quantize: divide by scale and round to integer
     W_int8 = round(W_group / scale)
     (result fits in [-127, 127], stored as 8-bit integer)

  3. At inference: dequantize on-the-fly before matmul
     W_approx = W_int8 × scale

Worked example:
  W_group = [0.52, -1.20, 0.81, 2.10, -0.33, ...]

  scale = 2.10 / 127 = 0.01654

  W_int8 = [round(0.52/0.01654),   →  31
            round(-1.20/0.01654),  → -73
            round(0.81/0.01654),   →  49
            round(2.10/0.01654),   →  127
            round(-0.33/0.01654),  → -20, ...]

  Reconstructed:
  W_approx = [31×0.01654,  -73×0.01654,  49×0.01654, ...]
           = [0.513,       -1.207,       0.810,      ...]

  Max error: |0.52 - 0.513| = 0.007 (1.3%)
  Average error across all weights: ~0.5%
  Perplexity increase: <1% — negligible quality impact
Memory savings:
  FP16: 70B parameters × 2 bytes = 140 GB
  INT8: 70B parameters × 1 byte  =  70 GB  (+ ~0.5 GB for scales)

  Per GPU (TP=4): 70 GB / 4 = 17.5 GB (down from 35 GB)
  Remaining HBM per GPU: 80 - 17.5 = 62.5 GB for KV cache
  (was 45 GB before — 39% more KV cache budget!)

Throughput Impact

INT8 quantization delivers a triple win: faster decode, larger batches, and lower cost — all at once.

1. FASTER DECODE (2×)
   Before (FP16):
     Weight load per step = 140 GB / 13.4 TB/s = 10.4 ms/token
   After (INT8):
     Weight load per step = 70 GB / 13.4 TB/s = 5.2 ms/token
     → 2× faster decode!

   For 500 output tokens:
     FP16: 500 × 10.4ms = 5,200ms decode
     INT8: 500 × 5.2ms  = 2,600ms decode

2. LARGER BATCHES (2×)
   Extra KV budget per GPU: 62.5 - 45 = 17.5 GB more

   KV cache per sequence (1,500 tokens avg):
     1,500 × 320 KB = 480 MB total, 120 MB per GPU (TP=4)

   Old max batch (FP16): 45 GB / 120 MB = 375 sequences
     (limited to ~64 by compute in practice)
   New max batch (INT8): 62.5 GB / 120 MB = 520 sequences
     batch=128 easily fits: 128 × 120 MB = 15.4 GB/GPU

3. COMBINED THROUGHPUT
   Total request time:
     Prefill: 88ms (1K input, UNCHANGED — prefill is compute-bound,
       and quantization doesn't reduce FLOPs. INT8 weights are
       dequantized to FP16 before matmul, same compute.
       Load halves (10.4→5.2ms) but was never the bottleneck:
       max(5.2, 88) = 88ms, same as max(10.4, 88) = 88ms.)
     Decode:  500 × 5.2ms = 2,600ms
     Total:   2,688ms ≈ 2.7s per request

   batch=128 concurrent:
     128 slots / 2.7s per request ≈ 47 req/sec per node

4. FLEET SIZING
   Nodes needed: 2,778 / 47 = 59 nodes
   H100s: 59 × 4 = 236 GPUs
   Cost: 236 × $2/hr × 24hr × 30 days = $340K/month

INT8 quantization is the single biggest win in this entire optimization stack. It delivers 2× faster decode, 2× larger batch capacity, and 4× cheaper fleet — all for less than 1% quality degradation. The fleet drops from 928 to 236 H100s, saving nearly $1M/month. This is why every production LLM serving system uses quantization.

Iteration 5: Speculative Decoding — Break the 1-Token-per-Step Barrier

Every optimization so far accepts a fundamental constraint: the target model produces one token per forward pass. Speculative decoding challenges this directly — by using a small, fast "draft" model to guess multiple tokens, then verifying them all in a single pass of the large model.

Draft-then-Verify

The mechanism uses a small draft model (Llama-3 8B, fits on 1 GPU with TP=1) to rapidly generate K=5 candidate tokens. Then the full 70B target model verifies all 5 tokens in a single forward pass.

Why verification costs the same as generating 1 token: the target model processes a sequence of 5 tokens in parallel — exactly like prefill. It produces logits (probability distributions) for each of the 5 positions simultaneously. We then compare the draft model's choice against what the target model would have chosen.

Acceptance Rule (Modified Rejection Sampling)

For each drafted token at position i, given the draft model's probability Pdraft(t) and the target model's probability Ptarget(t) for token t:

For position i = 1 to K:
  t = draft_token[i]

  If P_target(t) ≥ P_draft(t):
    ACCEPT token t (target agrees or is even more confident)

  Else:
    Draw random number r ~ Uniform(0, 1)
    If r < P_target(t) / P_draft(t):
      ACCEPT token t (probabilistic acceptance)
    Else:
      REJECT token t
      Sample a corrected token from an adjusted distribution:
        P_corrected(x) = max(0, P_target(x) - P_draft(x))
                         (normalized)
      Discard all remaining positions i+1 through K.
      STOP.

After processing all K positions:
  If all K were accepted, sample one BONUS token from
  the target model's distribution at position K+1.

Result: accepted tokens + 1 corrected/bonus token

This is mathematically lossless — the output distribution is identical to running the 70B model alone, token by token. The rejection sampling procedure guarantees this: accepted tokens match the target distribution exactly, and rejected tokens are replaced by samples from the corrected distribution that compensates for the draft model's bias.

Concrete Example

The draft model (8B) generates 5 tokens. The target model (70B) verifies them all in one forward pass:

Pos Draft Token Pdraft Ptarget Accept? Reasoning
1 "The" 0.35 0.42 YES Ptarget ≥ Pdraft → always accept
2 "model" 0.28 0.31 YES Ptarget ≥ Pdraft → always accept
3 "processes" 0.22 0.08 NO ratio = 0.08/0.22 = 0.36, drew r=0.71 > 0.36
4 "each" — — SKIP Discarded — pos 3 was rejected, so pos 4+ are invalid (they were conditioned on the wrong token "processes")
5 "token" — — SKIP Discarded — same reason, built on wrong context 
Result: 2 accepted + 1 corrected token = 3 tokens from ONE target forward pass

Cost breakdown:
  Draft:  5 steps × ~0.5 ms/step = 2.5 ms

  Verify: 1 forward pass of 70B      = 5.2 ms

  Total:  7.7 ms for 3 tokens → 2.6 ms per token

  Speedup on this example:            5.2 / 2.6 = 2.0×

Speedup Math

The expected number of tokens generated per target model forward pass depends on the per-token acceptance rate α:

Expected tokens per verify step:

  E[accepted] = α + α² + α³ + ... + α^K  (geometric series)
  Plus 1 corrected or bonus token at the end.

  For large K:  E[tokens] ≈ 1 / (1 - α)

Llama-3 8B drafting for Llama-3 70B:
  These models share architecture and training data.
  Typical per-token acceptance rate: α ≈ 0.70-0.80

  At α = 0.75:
    E[tokens per verify step] = 1 / (1 - 0.75) = 4.0 tokens

Time per speculative iteration:
  Draft:  K=5 tokens × 0.5 ms  = 2.5 ms
  Verify: 1 forward pass          = 5.2 ms
  Total:  7.7 ms per iteration (always the same cost)

  The concrete example above got 3 tokens in 7.7 ms (that specific case).
  On AVERAGE with α=0.75: E[tokens] = 4.0 per iteration.
  Effective: 7.7 ms / 4.0 tokens = 1.93 ms/token

  Speedup: 5.2 ms / 1.93 ms = 2.7×

For 500 output tokens:
  Without speculative: 500 tokens × 5.2 ms     = 2,600 ms
  With speculative:    500/4 = 125 iterations
                       125 × 7.7 ms              =   963 ms

  Total request time:
    Prefill:  88 ms
    Decode:  963 ms
    Total: 1,051 ms ≈ 1.05 seconds (down from 2.7s — 2.6× faster!)

When It Doesn't Help

  • Creative writing / brainstorming: low predictability, α < 0.50 → less than 1.5× speedup. The draft model's guesses are frequently wrong, so most tokens get rejected and we pay the draft cost for little benefit.
  • The draft model adds GPU cost: each serving node now needs 1 extra H100 for the 8B draft model (it runs on a separate GPU to avoid contending with the target model's memory bandwidth).
  • High batch sizes (>64): the verify step processes K=5 tokens for every sequence in the batch simultaneously. At batch=128, this is 640 token positions — the verify step shifts from memory-bandwidth-bound to compute-bound, reducing the speed advantage.
  • Very short outputs: the fixed overhead of running the draft model matters more when there are fewer tokens to amortize it over.

Key Takeaways

  1. Start with the bottleneck. LLM decode is memory-bandwidth-bound. Every optimization in this scenario targets one of three strategies: (a) reducing bytes loaded per step (quantization), (b) amortizing the load across more useful tokens (batching, PagedAttention), or (c) producing more tokens per load (speculative decoding).
  2. KV cache is the enabling trick AND the central challenge. It turns O(n²) recomputation into O(n) cached lookup, but the cache itself grows linearly with sequence length × batch size and becomes the memory bottleneck. Managing this cache efficiently (PagedAttention) is the single highest-ROI serving optimization.
  3. PagedAttention is the answer you give in every inference interview. It is pure software — no quality loss, no hardware changes — and it roughly doubles serving capacity by eliminating memory fragmentation. This is why vLLM became the default serving framework.
  4. Quantization is the highest-ROI for cost. INT8 halves weight memory, doubles decode speed, AND frees memory for larger batches — a triple win for less than 1% quality degradation. It is the single largest cost reduction in the stack.
  5. Speculative decoding breaks the 1-token-per-step wall. Verification is parallel (same cost as generating 1 token). The draft model's acceptance rate α determines the speedup: 1/(1−α). The output distribution is mathematically lossless.
  6. Flash Attention + Chunked Prefill are about tail latency, not throughput. They don't increase steady-state req/sec, but they prevent catastrophic p99 spikes when long-context requests arrive. Without them, a single 32K-token request stalls every other user for a full second.
  7. Layer the optimizations. Each technique addresses a different bottleneck (scheduling, memory management, memory bandwidth, compute, decode parallelism). Combined, they compound multiplicatively: 3× (batching) × 4× (paging) × 4× (quantization) × 2.7× (speculative) ≈ 130×, plus the latency fixes from Flash Attention and chunked prefill that make the system production-ready.
