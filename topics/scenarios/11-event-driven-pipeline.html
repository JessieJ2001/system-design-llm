Event-Driven Order Processing Pipeline

Processing 500 orders/sec through a multi-step pipeline where each step can fail independently — using the Saga pattern, idempotency, and the outbox pattern to guarantee correctness without distributed transactions.

NALSD Walkthrough • ~15 min read

The Scenario

You are designing the order processing backend for an e-commerce platform. When a customer places an order, the system must execute five steps in sequence. Each step calls a different service, and each can fail independently. If a later step fails, all previous steps must be undone (compensated).

The Five Steps

StepActionServiceAvg LatencyFailure Rate
1Create order recordOrder Service (internal)20ms0.01%
2Reserve paymentPayment Gateway (external)2,000ms1%
3Reserve inventoryInventory Service (internal)50ms0.1%
4Schedule shippingShipping API (external)500ms0.5%
5Send confirmation emailEmail Service (internal)100ms0.01%

Compensation Actions (Undo)

Each step has a corresponding compensation action that reverses its effect:

StepForward ActionCompensation (Undo)
1Create orderCancel order (mark as cancelled)
2Reserve paymentRefund payment
3Reserve inventoryRelease inventory
4Schedule shippingCancel shipment
5Send confirmationSend cancellation email

Scale

Peak order rate:   500 orders/sec
Daily orders:      500 × 86,400 = 43.2 million orders/day
Average order value: $50
Daily GMV:         43.2M × $50 = $2.16 billion/day

Back-of-Envelope Math

How Often Do Orders Need Compensation?

Probability that ALL steps succeed:
  P(all succeed) = P(step1) × P(step2) × P(step3) × P(step4) × P(step5)
                 = 0.9999 × 0.99 × 0.999 × 0.995 × 0.9999
                 = 0.9840

  P(any step fails) = 1 - 0.9840 = 1.6%

At 500 orders/sec:
  Compensation needed: 500 × 0.016 = 8 orders/sec
  Per day: 8 × 86,400 = 691,200 orders need compensation

This is NOT an edge case. 700K orders per day require at least
one compensation action. The compensation path must be as
reliable as the happy path.

Latency Budget

Synchronous chain (worst case):
  Step 1:   20ms
  Step 2: 2,000ms
  Step 3:   50ms
  Step 4:  500ms
  Step 5:  100ms
  ─────────────────
  Total:  2,670ms per order

At 500 orders/sec with 2,670ms processing time:
  Concurrent orders in flight: 500 × 2.67 = 1,335
  Threads needed: 1,335 (if one thread per order)

If step 2 (payment) is slow (5s at p99):
  Concurrent orders: 500 × 5 = 2,500 threads
  Thread pool exhaustion is a real risk.

Async approach:
  API response time: 20ms (create order + publish event)
  Steps 2-5 happen asynchronously.
  User sees "Order placed!" immediately.

Event Volume

Happy path: 5 events per order
  order-created → payment-reserved → inventory-reserved
  → shipping-scheduled → confirmation-sent

Failure + compensation: varies
  If step 3 (inventory) fails after step 2 (payment) succeeds:
    Events: order-created → payment-reserved → inventory-failed
            → payment-refund-requested → payment-refunded → order-cancelled
    = 6 events

Average events per order:
  98.4% happy path: 5 events
  1.6% failure path: avg 6 events
  Weighted: 0.984 × 5 + 0.016 × 6 = 5.016 events/order

Total event throughput:
  500 orders/sec × 5.016 events = 2,508 events/sec

Kafka capacity: one partition handles 10K-100K msgs/sec.
  A single partition could handle our entire load.
  Use 8-16 partitions for parallelism and headroom.

Naive Design: Synchronous REST Chain

The simplest approach: the Order Service calls each downstream service in sequence via REST. On failure at step N, it synchronously calls the compensation for steps 1 through N-1.

def place_order(order):
    try:
        # Step 1: Create order
        order_id = order_service.create(order)

        # Step 2: Reserve payment
        payment_id = payment_api.reserve(order_id, order['amount'])

        # Step 3: Reserve inventory
        inventory_service.reserve(order_id, order['items'])

        # Step 4: Schedule shipping
        shipping_api.schedule(order_id, order['address'])

        # Step 5: Send confirmation
        email_service.send_confirmation(order_id, order['email'])

        return {"status": "confirmed", "order_id": order_id}

    except PaymentError:
        order_service.cancel(order_id)
        raise

    except InventoryError:
        payment_api.refund(payment_id)    # ← What if this fails?
        order_service.cancel(order_id)
        raise

    except ShippingError:
        inventory_service.release(order_id)
        payment_api.refund(payment_id)    # ← What if this fails?
        order_service.cancel(order_id)
        raise

What Breaks

Problem 1: Compensation Can Fail

The code above assumes compensation always succeeds. But what if the payment API is down when we try to refund? We have already reserved the customer's money AND failed to deliver the item. The customer is charged but gets nothing.

Failure scenario:

  t=0ms     Create order #1234                    ✓ Success
  t=2000ms  Reserve $50 from payment gateway      ✓ Success
  t=2050ms  Reserve inventory for 3 items         ✗ FAIL (out of stock)
  t=2051ms  Attempting compensation: refund $50...
  t=4051ms  Payment API returns 503 (gateway down) ✗ FAIL

  Result: Customer charged $50. No items reserved. No refund.
  Order is in an INCONSISTENT state.

  The order service has no retry mechanism. The refund is lost.
  Manual intervention needed. At 8 failures/sec, that is
  ~700K orders/day needing manual review. Impossible.

Problem 2: Crash Between Steps

If the Order Service process crashes between step 2 (payment reserved) and step 3 (inventory), neither the forward step nor the compensation executes. The payment is reserved, but nobody knows to proceed or refund.

Crash scenario:

  t=0ms     Create order #5678                    ✓ Success
  t=2000ms  Reserve $75 from payment gateway      ✓ Success
  t=2001ms  ORDER SERVICE PROCESS CRASHES (OOM, deploy, etc.)

  Result: Order exists in DB. Payment reserved (customer charged).
  Nobody will ever call inventory_service.reserve() or
  payment_api.refund(). The order is stuck in limbo.

  The payment gateway will eventually release the hold
  (after 7 days), but the customer sees a pending charge
  for a week with no order confirmation.

Problem 3: High Latency and Tight Coupling

Each order holds a thread for 2.67 seconds. At 500 orders/sec, that is 1,335 concurrent threads. If the payment API slows down (4s instead of 2s), concurrent threads jump to 2,250. If the shipping API also slows down, we might need 3,000+ threads. The Order Service becomes a bottleneck, and its health depends on the health of every downstream service.

The fundamental issue: Synchronous compensation is unreliable because the failure that caused the compensation might also prevent the compensation from succeeding (e.g., network partition affects both the forward call and the undo call). We need a mechanism that persists the intent to compensate and retries until it succeeds.

Iteration 1: Saga Pattern with Orchestration

The Saga pattern breaks a distributed transaction into a sequence of local transactions, each with a compensating action. Instead of trying to make the entire operation atomic (which requires distributed locking and is slow), each step commits independently. If a later step fails, the system executes compensating actions for all completed steps.

What Makes Sagas Different from the Naive Approach?

The critical difference: events are persisted in a durable message queue. If a compensation fails, the event stays in the queue and is retried with exponential backoff. Unlike the synchronous approach, the intent to compensate is never lost.

The Orchestrator

An Order Orchestrator service manages the saga. It maintains the state of each order (which steps have completed) and decides what happens next based on incoming events.

Event Flow: Happy Path

Step-by-step trace for Order #1234 ($50, 2 items):

1. API receives order request.
   → Orchestrator creates order record (state: CREATED).
   → Publishes event: {topic: "order-commands", type: "reserve-payment",
                        order_id: 1234, amount: $50}

2. Payment Consumer reads event from "order-commands".
   → Calls payment gateway: reserve $50 for order 1234.
   → Payment succeeds. Payment ID: PAY-789.
   → Publishes event: {topic: "order-events", type: "payment-reserved",
                        order_id: 1234, payment_id: PAY-789}

3. Orchestrator reads "payment-reserved" event.
   → Updates order state: PAYING → RESERVING_INVENTORY.
   → Publishes event: {topic: "order-commands", type: "reserve-inventory",
                        order_id: 1234, items: [{sku: "A", qty: 2}]}

4. Inventory Consumer reads event from "order-commands".
   → Reserves 2 units of SKU "A" in inventory DB.
   → Publishes event: {topic: "order-events", type: "inventory-reserved",
                        order_id: 1234}

5. Orchestrator reads "inventory-reserved" event.
   → Updates order state: RESERVING_INVENTORY → SHIPPING.
   → Publishes event: {topic: "order-commands", type: "schedule-shipping",
                        order_id: 1234, address: "123 Main St"}

6. Shipping Consumer reads event.
   → Calls shipping API. Tracking: SHIP-456.
   → Publishes event: {topic: "order-events", type: "shipping-scheduled",
                        order_id: 1234, tracking: SHIP-456}

7. Orchestrator reads "shipping-scheduled" event.
   → Updates order state: SHIPPING → CONFIRMING.
   → Publishes event: {topic: "order-commands", type: "send-confirmation",
                        order_id: 1234, email: "alice@example.com"}

8. Email Consumer reads event.
   → Sends confirmation email.
   → Publishes event: {topic: "order-events", type: "confirmation-sent",
                        order_id: 1234}

9. Orchestrator reads "confirmation-sent" event.
   → Updates order state: CONFIRMING → COMPLETED.
   → Done!

Total events: 9 (5 commands + 4 results). Elapsed: ~3 seconds.

Event Flow: Failure with Compensation

Order #5678: Payment succeeds, but inventory fails (out of stock).

1. Orchestrator creates order (state: CREATED).
   → Publishes "reserve-payment" command.

2. Payment Consumer: reserves $75. Succeeds.
   → Publishes "payment-reserved" event.

3. Orchestrator → RESERVING_INVENTORY.
   → Publishes "reserve-inventory" command.

4. Inventory Consumer: tries to reserve 5 units of SKU "B".
   → Only 2 units in stock. FAILS.
   → Publishes "inventory-failed" event (reason: "insufficient stock").

5. Orchestrator reads "inventory-failed".
   → Order cannot proceed. Must compensate step 2.
   → Updates state: RESERVING_INVENTORY → COMPENSATING_PAYMENT.
   → Publishes "refund-payment" command: {order_id: 5678, payment_id: PAY-321}

6. Payment Consumer reads "refund-payment" command.
   → Calls payment gateway: refund $75 for PAY-321.

   SCENARIO A: Refund succeeds.
     → Publishes "payment-refunded" event.
     → Orchestrator updates state → CANCELLED. Done.

   SCENARIO B: Payment gateway is DOWN (503 error).
     → Consumer does NOT publish a success event.
     → Consumer does NOT acknowledge the Kafka message.
     → After visibility timeout, Kafka redelivers the message.
     → Consumer retries with exponential backoff.
     → Eventually payment gateway recovers.
     → Refund succeeds. "payment-refunded" published.
     → Orchestrator → CANCELLED.

     KEY INSIGHT: Because the "refund-payment" command is persisted
     in Kafka, it will be retried until it succeeds. The refund is
     NEVER lost, even if the payment gateway is down for hours.
Why this is better than synchronous compensation: In the synchronous approach, if the refund fails, the in-memory intent to refund dies with the failed HTTP call. In the saga approach, the intent is persisted as a durable event in Kafka. The event survives process crashes, network outages, and downstream failures. It will be processed eventually.

Iteration 2: Idempotency and the Outbox Pattern

Why Idempotency Is Essential

Because Kafka provides at-least-once delivery, any consumer might process the same event twice. Without idempotency, this causes real damage:

Without idempotency — double charge scenario:

  1. Payment Consumer reads "reserve-payment" for order #1234.
  2. Calls payment gateway: charge $50. Succeeds.
  3. Consumer tries to commit Kafka offset... but crashes before commit.
  4. Kafka redelivers the same "reserve-payment" message.
  5. Payment Consumer reads it again.
  6. Calls payment gateway: charge $50 AGAIN.
  7. Customer is charged $100 for a $50 order.

With idempotency:

  1. Payment Consumer reads "reserve-payment" for order #1234.
  2. Checks idempotency table: SELECT * FROM processed_events
     WHERE idempotency_key = 'order-1234-reserve-payment'
     → No rows found. Proceed.
  3. Calls payment gateway with idempotency key: charge $50.
  4. Inserts into idempotency table:
     INSERT INTO processed_events (key, result, created_at)
     VALUES ('order-1234-reserve-payment', 'PAY-789', now())
  5. Consumer crashes before Kafka offset commit.
  6. Kafka redelivers the message.
  7. Payment Consumer reads it again.
  8. Checks idempotency table: row EXISTS.
     → Already processed. Skip. Return cached result PAY-789.

Implementing Idempotency

def process_payment_command(event):
    """Idempotent payment reservation."""
    idempotency_key = f"order-{event['order_id']}-reserve-payment"

    # Check if already processed
    existing = db.query(
        "SELECT result FROM processed_events WHERE key = %s",
        idempotency_key
    )
    if existing:
        # Already processed — publish cached result
        publish_event('order-events', {
            'type': 'payment-reserved',
            'order_id': event['order_id'],
            'payment_id': existing['result'],
        })
        return

    # Not yet processed — do the work
    payment_id = payment_gateway.reserve(
        amount=event['amount'],
        idempotency_key=idempotency_key,  # gateway also supports idempotency
    )

    # Record that we processed this event
    db.execute(
        "INSERT INTO processed_events (key, result, created_at) "
        "VALUES (%s, %s, NOW())",
        idempotency_key, payment_id
    )

    # Publish result
    publish_event('order-events', {
        'type': 'payment-reserved',
        'order_id': event['order_id'],
        'payment_id': payment_id,
    })

The Dual-Write Problem

There is a subtle bug in the code above. The consumer does two things that must both succeed or both fail:

  1. Write to the database (business data + idempotency record)
  2. Publish an event to Kafka

These are two different systems. If the DB write succeeds but Kafka publish fails (network blip), the orchestrator never learns that payment was reserved. The order is stuck. If Kafka publish succeeds but DB write fails (crash), the event is published but the idempotency record is missing, risking double-processing on retry.

The dual-write problem:

  DB write ✓  +  Kafka publish ✗  →  Order stuck (event lost)
  DB write ✗  +  Kafka publish ✓  →  Inconsistency (no idempotency record)

We cannot make "write to DB" and "publish to Kafka" atomic
because they are different systems. A transaction in one system
does not span the other.

The Outbox Pattern: Solving Dual Writes

The outbox pattern eliminates the dual-write problem by writing both the business data and the outgoing event to the same database in a single local transaction. A separate process (the relay) reads events from the outbox table and publishes them to Kafka.

def process_payment_command_with_outbox(event):
    """Idempotent payment processing with outbox pattern."""
    idempotency_key = f"order-{event['order_id']}-reserve-payment"

    # Check if already processed
    existing = db.query(
        "SELECT result FROM processed_events WHERE key = %s",
        idempotency_key
    )
    if existing:
        return  # already processed; relay will re-publish if needed

    # Do the external work
    payment_id = payment_gateway.reserve(
        amount=event['amount'],
        idempotency_key=idempotency_key,
    )

    # Single transaction: business data + idempotency + outbox event
    with db.transaction():
        db.execute(
            "INSERT INTO payments (order_id, payment_id, amount, status) "
            "VALUES (%s, %s, %s, 'reserved')",
            event['order_id'], payment_id, event['amount']
        )
        db.execute(
            "INSERT INTO processed_events (key, result) VALUES (%s, %s)",
            idempotency_key, payment_id
        )
        db.execute(
            "INSERT INTO outbox (topic, payload) VALUES (%s, %s)",
            'order-events',
            json.dumps({
                'type': 'payment-reserved',
                'order_id': event['order_id'],
                'payment_id': payment_id,
            })
        )

Final Design

Throughput Verification

Total events: 500 orders/sec × 5.016 events = 2,508 events/sec

Kafka sizing:
  8 partitions on "order-commands" topic
  8 partitions on "order-events" topic
  Each partition: ~313 events/sec (well under 10K limit)

Consumer scaling:
  Payment: 500 orders/sec × 2s avg = 1,000 concurrent
           → 4 consumer instances × 250 threads each
  Inventory: 500/sec × 50ms = 25 concurrent → 1 instance
  Shipping: 500/sec × 500ms = 250 concurrent → 2 instances
  Email: 500/sec × 100ms = 50 concurrent → 1 instance

Orchestrator DB:
  500 state updates/sec × 5 updates per order = 2,500 writes/sec
  Each update is a single row UPDATE (order_id, state).
  Easily handled by a single PostgreSQL instance with connection pooling.

Outbox relay latency:
  Polling interval: 100ms.
  Adds 50-100ms to end-to-end latency.
  For async order processing, this is acceptable.

Failure Recovery Summary

Failure ScenarioWhat HappensRecovery
Consumer crashes mid-processing Kafka offset not committed Another consumer picks up the message. Idempotency prevents double-processing.
External API down (payment, shipping) Consumer retries with backoff Event stays in Kafka. Retried until success. No manual intervention.
Orchestrator crashes Events queue up in Kafka Orchestrator restarts, reads from last committed offset, resumes processing.
Kafka broker failure Other brokers in the cluster serve the partitions Kafka replication factor 3. Tolerates 2 broker failures.
DB failure Consumer cannot write outbox event Transaction rolls back. Kafka redelivers message. Retry after DB recovers.
Relay crashes Outbox events not published Relay restarts, picks up un-published rows from outbox table.
End-to-end guarantee: No order is ever stuck. Every order eventually reaches either COMPLETED or CANCELLED. The saga state machine, idempotent consumers, and durable event log guarantee that progress is always made, even in the presence of arbitrary failures.

Key Takeaways

  • Partial failure is the norm, not the exception. At 500 orders/sec with a 1.6% failure rate, you get 8 compensation events per second. The compensation path must be as reliable and automated as the happy path.
  • Synchronous compensation is unreliable. If the failure that caused the need for compensation also prevents the compensation from succeeding (e.g., network partition), you are stuck. Persist the intent to compensate in a durable queue.
  • The Saga pattern trades atomicity for availability. Unlike 2PC (two-phase commit), sagas do not lock resources across services. This means higher throughput and lower latency, but the system is temporarily inconsistent between steps.
  • Idempotency is non-negotiable in event-driven systems. At-least-once delivery means duplicate events will happen. Every consumer must handle duplicates gracefully. Use idempotency keys derived from the business operation (order_id + step_name).
  • The outbox pattern solves the dual-write problem. Never write to a database AND publish to a message queue independently. Write both to the database in one transaction, then relay the event to the queue asynchronously.
  • Design the state machine first. Before writing any code, draw the saga state machine with all states and transitions. This is the contract that guarantees every order reaches a terminal state.
  • Calculate your event throughput. Events per order times orders per second gives you the total event rate. Kafka can handle millions of events per second, but your consumers' processing capacity is the bottleneck.
