Designing WhatsApp / E2E Messaging

2B users, 100B messages/day, server cannot read a single byte of content. We start from a synchronous plaintext push and iterate to a per-device store-and-forward queue + Signal-protocol E2E + multi-device sealed sender + mobile-push edge wakeup + group sender-keys.

Sister scenario. This page is the highest-contrast sister of #41 Twitter home-timeline-fanout within Group F (Twitter / fanout). The two share the LC kernel (LC #10 — one writer fans out to N readers) but diverge on four load-bearing constraints (cite the 14-factor list in the plan): trust/adversary model (server is untrusted — F5 vs trusted server #41), access-pattern shape (bounded recipients — group ≤1024 — vs power-law celebrity skew), connectivity (mobile-first offline-tolerant vs always-online), persistence model (per-device queue of opaque ciphertext vs ranked inbox of post IDs). Result: 0 of #41's 5 iterations transfer verbatim. The shape of the loop transfers (naive → durability → encryption → fanout → connection scale → group optimization); every move is new.

LC kernel. Design Twitter (LC #10). Textbook structure: k-way merge of follower feeds at read time. Property that breaks at scale (and breaks in a new way vs #41): in F5 the server is blind — it cannot read post content because every byte is E2E-encrypted. The k-way merge cannot run on the server because the server has no plaintext to merge. The C8 production cousin replaces read-time merge with a per-device store-and-forward queue of opaque ciphertext blobs; the client decrypts and assembles its own view. The push-direction asymmetry also flips: #41's celebrity fanout disappears (group cap = 1024 by product rule), but a new fanout cliff appears on the sender's mobile network — one group message becomes 1,024 × 1.5 device ciphertexts uploaded over 4G. Iter 4 (Sender's Keys) collapses that 1,500×.

S0. Scenario & Requirements

A consumer mobile messaging app at WhatsApp scale. Users send text + media references in 1:1 or group conversations; messages arrive on every device the recipient owns; recipients may be offline for hours or days and must still receive everything in order on reconnect; and the server — ours, run by us — is treated as untrusted: it must never see a byte of message plaintext, even if an operator with root access reads every database. WhatsApp, Signal, iMessage — same shape.

System class

Class: product-shaped (user-facing capabilities) with a hard infra-primitive overlay (the cryptographic guarantee). The FR vocabulary is product-class (cite functional-requirements.html#class-product); the NFR vocabulary adds an infra-class confidentiality guarantee that is binary, not a knob — either server can decrypt or it cannot. This single NFR reshapes every iteration vs #41.

Functional requirements (cite functional-requirements.html)

  • Send message. User sends text (≤4 KB) or media reference (URL to separately-encrypted blob) to a peer or group. Returns delivery receipt per recipient device.
  • Receive message. User's online devices get pushed; offline devices buffer server-side and drain on reconnect with no loss and no reorder within (sender, recipient_device) pair.
  • Group chat. Up to 1,024 members per group (product cap; not a scale limit — a trust limit, since every member can leak content). Same E2E guarantees as 1:1.
  • Multi-device. Same user account on up to 4 devices (phone + tablet + desktop + web). All devices see the same conversation in the same order; messages composed on one device echo to others.
  • Delivery receipts. Sender sees ✓ when each recipient device acknowledges receipt, ✓✓ when client confirms decryption.

Out of scope (named): voice/video calls (separate WebRTC + STUN/TURN pipeline); status/stories (separate ephemeral-content service); business API + payments; cloud backup of message history (separate user-key-encrypted blob product — we discuss this in Probe B); end-user contact discovery via phone numbers (separate privacy-sensitive lookup service); content moderation (impossible by construction — covered in pushbacks).

Flagged assumption. E2E is non-negotiable and applies to every message (no "opt-in"). If the product allowed plaintext mode for some users, Iter 1 (E2E) becomes optional and the server keeps its rank/mute/dedupe capabilities — the design converges back toward #41. The whole arc below assumes E2E-by-default.

Sister scenarios (see Group F landing page)

  • #41 Twitter home-timeline-fanout — the canonical sibling. Read-heavy, power-law (celebrity 200 M followers), public, ranker. Inverts on: trust model (trusted server), access pattern (power-law), persistence (ranked inbox).

The fundamental tension. We need the server to be the only way 2 billion devices reach each other, but we need the server to be incapable of reading what they say. Every server-side feature #41 uses to scale (server-side rank, server-side mute filter, server-side dedupe, server-side fanout-on-write to per-user inbox indexed by post content) is forbidden here. Every iteration below pays for the privacy guarantee in some way: extra round trips for key agreement (Iter 1), N× fanout for multi-device addressing (Iter 2), push notifications as the only way to wake an offline phone we can no longer reach over TCP (Iter 3), per-group sender keys to keep mobile uploads tractable (Iter 4).

S1. Core Entities (cite entities-and-data.html)

Eight entities. Device is first-class (not an attribute of User) because (recipient_user_id, recipient_device_id) is the partition that the entire delivery path traverses; every message lands on a per-device queue. PreKeyBundle earns first-class status because it's the only entity the server gives to the sender (the rest are sender→server writes); its replenishment cycle is a separate access pattern. SenderKey (Iter 4) is a per-(group, sender_device) cryptographic state that the server stores opaquely.

ID scheme (cite identifier-schemes.html#snowflake): Snowflake (64-bit) for user_id, conv_id, group_id; monotonic per-(user, device) sequence (32-bit ascending) for msg_seq within the device queue partition. Why msg_seq is per-partition monotonic and not Snowflake: the recipient device drains in msg_seq ASC order with no gaps, and a partition-local counter is cheaper to gap-detect (client compares last_seen_seq vs server's current; missing seqs = ratchet desync flag). Snowflake's worker-skew would produce non-monotonic seqs within a single partition. Rejected: Snowflake for msg_seq (within-partition gap detection becomes statistical instead of exact); UUIDv7 for everything (16 B vs 8 B doubles per-message overhead on a 1 KB ciphertext — 1.6% on the wire, 1.6% in the queue, and the queue is the dominant byte sink).

The partition-key heuristic (cite entities-and-data.html): the entire delivery path starts from "where does this ciphertext go?" → (recipient_user_id, recipient_device_id). 10 M ciphertexts/sec all partition on this composite key. Contrast with #41 where reads partition by user_id (single dimension) and the read path is a fan-in over many followees. Here writes partition by (recipient_user_id, recipient_device_id) because the read path is a single point-drain by the recipient device itself — no merge, no fan-in.

S4. Naive Architecture — the simplest thing that meets the FRs

One API server. Sender opens TCP, sends plaintext message + recipient_user_id. Server looks up the recipient's currently-connected socket (in-process map). If online, push plaintext immediately. If offline, drop. No encryption, no queue, no fanout, no multi-device, no push notifications. This is the dumbest plausible thing that satisfies "user can send and receive messages."

Why it's broken (per failing NFR)

  • Throughput fails by 200× (50k vs 10M). The first failing NFR by raw magnitude.
  • Durability fails: 40% of messages dropped (recipient offline). RPO is unbounded loss, not 0.
  • Confidentiality fails: server has plaintext access to every message. Binary failure — not a degree.
  • Availability on receive: when sender's destination is offline, no delivery is ever possible — receive availability = 60% best case.
  • Multi-device fails: only one connection per user can be active.
  • Send latency ✅ passes when recipient is online (single hop). This is the only NFR the naive design happens to satisfy.

The naive design's failures are mutually entangled: we cannot fix confidentiality without changing the fanout model (each device becomes its own crypto endpoint); we cannot fix durability without a persistent queue (which then forces a partition decision); we cannot fix throughput without horizontal API + a state layer behind it. Iter 0 addresses durability + throughput + multi-device together because they share the same first move (per-device persistent queue + horizontal API tier). Confidentiality is left for Iter 1 because the queue's shape doesn't depend on whether the payload is plaintext or ciphertext — that's a payload-format change, not a topology change.

Cost of naive at NFR-target load: not deployable. One server cannot reach 10 M msg/sec at any cost (NIC saturates at ~12 M packets/sec hard ceiling regardless of CPU). The gap is ∞ on the throughput axis and binary on the confidentiality axis — both motivate the next iteration. The anchor cost ("$/mo to make Naive 'work'") is meaningless because the architecture is structurally unable to serve the workload.

Bottleneck

The Naive design (S4) fails three NFRs at once with a shared root cause: there is nowhere to put a message addressed to an offline recipient. Throughput fails 200× because one TCP-relaying pod tops at 50k msg/sec while the gate is 10 M/sec. Durability fails because there is no persistent backing store — offline-recipient drops are 40% of traffic. Multi-device fails because in-process socket map only addresses one connection per user. Classified as bottlenecks#backlog-growth — "variable consumer rate, must persist until ACK"; combined with bottlenecks#fanout-asymmetry — "one producer, N recipient devices." The signal that picks these entries: producer rate is constant (10 M/sec), consumer drain rate is bursty (0 to bulk-drain on reconnect), and they share no synchronization — the canonical backlog-growth signature.

Architectural decisions to revisit

Per bottleneck-to-decisions.html for backlog-growth: (a) introduce a durable queue with backpressure; (b) partition the queue so per-recipient ordering is preserved without global coordination; (c) add horizontal stateless processing tier in front; (d) pick a retention/TTL bound matched to consumer offline tolerance. This iteration addresses all four with one coherent move (Cassandra per-device partition queue + Envoy/WS pod tier). Replication topology decision is deferred to within-this-iteration (RF=3 LOCAL_QUORUM); cross-region is Probe A.

Alternatives considered (C4)

  • Cassandra per-(user, device) partition queue — chosen. LSM write-heavy, native time-window compaction matches the TTL exactly, ~3 B partitions is fine (Cassandra has no upper bound on partition count, only on rows-per-partition which is bounded by 30-day TTL × in-flight rate per device).
  • Kafka, one topic per device — rejected. Kafka caps at ~10k partitions per cluster (controller metadata cost); 3 B device-partitions would require ~300k Kafka clusters — un-operable. (🔀 if the access pattern were "broadcast to ~10k channels" instead of "point-delivery to ~3B devices," Kafka would win.)
  • Redis queue per device — rejected. Durability story is too thin (AOF + RDB at 25 PB of active queue is operationally brutal); per-device memory cost dominates (each device needs at least the key allocation in Redis hash). Redis serves the connection registry instead (ephemeral state).
  • DynamoDB — rejected on unit cost. ~10 M write/sec × $1.25/M-WCU = ~$108k/hr ≈ $78 M/mo for writes alone. Self-managed Cassandra at $330k/mo for the same rate is ~240× cheaper (cite C7).

The chosen move

Two coupled changes: (1) introduce a durable per-device queue in Cassandra 5.x, partitioned on (recipient_user_id, recipient_device_id) with clustering on msg_seq DESC and TWCS 1-day windows + 30-day TTL; (2) introduce a horizontal stateless API/WebSocket tier (Envoy front, Go pods behind) that terminates WebSocket connections, validates short-lived sender certs, persists incoming messages to the queue, and pushes drains to connected recipient devices. The connection-state-to-pod mapping lives in a Redis cluster (ephemeral, 60s TTL on heartbeat) so any pod can route messages to any active connection regardless of which pod terminates that connection. Why this addresses the bottleneck shape: per-device partitioning gives each recipient its own backlog (no head-of-line blocking across users; one user offline for 30 days does not slow another user's drain); LSM matches the write-heavy append-then-tail-read access pattern; TWCS aligns sstable lifecycle with TTL so deletions never produce tombstone storms (the classic Cassandra failure mode of TTL workloads under SizeTieredCompactionStrategy).

What breaks (C5; cite failure-modes.html)

  • Backlog overflow. A device offline 30 days reconnects; its partition has accumulated up to ~30 days × 1 msg/sec from a chatty group = 2.6 M rows. Drain dumps all at once, saturates the pod's NIC and the device's mobile data. Blast radius: one device (the offline one); other devices on the same pod see ~200 ms latency spike during the burst. Detection: per-partition row count metric in Cassandra; alert at > 10k. Recovery: drain in 1k-row chunks with client-paced ACKs; pod-side rate-limiter per recipient device.
  • Hot-partition meltdown. A device in 100 active groups gets 100× the average write rate. At 1k msg/sec to one partition, Cassandra's per-partition lock contention starts producing 100 ms p99 spikes. Detection: partition-heat metric (Cassandra nodetool tablestats by partition); alert at > 500 writes/sec single partition. Recovery: shard the queue partition with a synthetic bucket key for the top-1000 hottest devices — ((user_id, device_id, week_bucket), msg_seq); clients track bucket boundary on drain.
  • Network partition (cross-AZ). AZ split during writes — LOCAL_QUORUM tolerates one AZ down (2 of 3 acks). If two AZs split off, writes stall on QUORUM and the partition is rejected (CP under partition). Degradation contract: SEND returns 503 with Retry-After; client buffers locally and retries on partition heal. RPO=0 maintained (no message accepted-but-lost).
  • Poison message. A malformed SEND frame crashes a pod. Currently affects all in-flight messages on that pod. Mitigation: deserialize SEND in a sandbox + return 400; circuit-break the offending sender after 3 consecutive 400s. Cite failure-modes#poison-message.

Iteration 1 — Signal Protocol E2E: X3DH initial agreement + Double Ratchet ongoing

System after this iteration

Legend. Solid slate = synchronous critical path. Dashed gray = async/lookup. Accent = new/changed (sender + recipient now do all crypto client-side; prekey-bundle store and identity-key registry are new server-side stores).

Bottleneck

This iteration is the F5-defining move and is not driven by a numeric NFR-gap the way Iter 0 was. The failing NFR is confidentiality: Iter 0 fixed durability and throughput but left the server holding plaintext. Confidentiality is binary — there is no number that "shows" it failing because the failure is structural (anyone with root can SELECT payload FROM device_inbox). This is the analog of #41's GDPR probe but elevated to a Core iteration because the entire product premise depends on it. Why a Core iteration and not a Probe: in #41 GDPR was a Probe because the design served its NFRs without it; in F5, the product collapses without E2E (it stops being WhatsApp and becomes SMS) — so it shapes the architecture, it doesn't decorate it. This iteration also surfaces a new bottleneck dimension that was numeric: the prekey-bundle fetch rate becomes a new hot service we must size for.

Architectural decisions to revisit

This iteration does not match a performance entry in bottleneck-to-decisions.html because the bottleneck shape is constraint-driven (a binary NFR), not performance-driven. Decisions to revisit: (a) what compute moves from server to client (all crypto + all content-aware features); (b) what new server-side stores are needed to support the client-side protocol (prekey bundles, identity-key registry, sender certificates — the support infrastructure E2E requires); (c) which server features become permanently impossible (server-side ranking, server-side mute, server-side spam-detection-by-content) and what compensating mechanisms replace them (client-side mute, sender-certificate rate limiting in Iter 2, sealed-sender abuse reporting via recipient-initiated unwrap).

Alternatives considered (C4)

  • Signal Protocol (X3DH + Double Ratchet) — chosen. The de facto standard for asynchronous E2E with forward + future secrecy. Open spec, audited, deployed at the same scale by Signal Foundation. Library: libsignal (Rust). Why it wins: asynchrony (recipient can be offline; sender doesn't wait), forward secrecy on every message rotation, mature multi-device extensions in v2.
  • OTR (Off-the-Record Messaging) — rejected. Synchronous (requires both peers online to establish session). Doesn't fit the mobile offline-tolerance NFR. Was the previous-generation answer (Pidgin, Adium); Signal protocol superseded it.
  • PGP / S/MIME — rejected. Long-lived keys (no forward secrecy — key compromise reveals every past message). Designed for email asynchrony, but the lack of ratcheting is unacceptable for chat where threads run for years.
  • MLS (Messaging Layer Security; RFC 9420) — deferred (probe). Newer (2023), designed for large groups with sub-linear sender cost; would simplify Iter 4. But: protocol churn risk (recent ratification), library maturity behind libsignal, and group cap is already 1024 (we'll see in Iter 4 that Sender's Keys are sufficient). (🔀 if the product expanded to 10k-member channels, MLS would replace Signal Protocol entirely.)

The chosen move

Adopt the Signal Protocol for all message encryption. On device registration, each device generates an Ed25519 identity key (long-term), an X25519 signed prekey (medium-term, rotated weekly, signed by identity key), and 100 X25519 one-time prekeys; the bundle (public halves) is uploaded to a new prekey_bundles + one_time_prekeys Cassandra table. On first contact between sender device A and recipient device B, A fetches B's bundle (consumes one of B's one-time prekeys via Cassandra LWT to guarantee single-use), performs X3DH (four DH exchanges combining identity, signed prekey, one-time prekey, ephemeral ephemeral) to derive the initial root key. From there, every subsequent message rotates: the chain key advances per message (forward secrecy — old chain keys are deleted), and a DH ratchet step happens on every received message acknowledgment (future secrecy — a compromised chain key is healed by the next inbound message). The server's only job is to: store + dispense prekey bundles (with LWT on one-time consume), store ciphertext blobs in device_inbox.payload exactly as in Iter 0 (now an opaque byte string from the server's view), and store + serve identity public keys for safety-number verification.

What breaks (C5; cite failure-modes.html)

  • Prekey exhaustion. Recipient device offline; sender fetches its bundle; all 100 one-time prekeys consumed by 100 senders before recipient comes back to replenish. New sender gets 410 (exhausted). Degradation: fall back to signed-prekey-only session (forward secrecy preserved via Double Ratchet, but the initial agreement loses one DH "leg" so the very first message has weaker forward secrecy — acceptable per Signal spec). Detection: per-device prekey-count metric in Cassandra; alert when median user has <20 unclaimed prekeys (replenishment lagging). Recovery: client uploads new batch of 100 on next connect; until then, signed-prekey fallback.
  • Ratchet desynchronization. Client device wipes its session state (uninstall/reinstall, restore from backup that doesn't include sender keys). All incoming messages encrypted under the old ratchet are undecryptable. Blast radius: one device, all peers it has open sessions with. Detection: client-side "cannot decrypt" counter; surface to user as "🔒 message could not be decrypted — start a new conversation." Recovery: session reset (drop ratchet state, fetch new prekey bundle, X3DH again). Note: messages encrypted under the old ratchet that are still in queue are lost — this is acceptable (forward secrecy is the trade-off).
  • Identity-key rotation MITM window. User reinstalls app; new identity key generated. Old peers see "safety number changed." A real attacker can rotate identity keys server-side to insert themselves. Mitigation: client UI warns on safety-number change; opt-in "block messages until verified" mode for high-threat users; identity-key history is server-stored for forensic detection. Detection: rate of safety-number-change events per user; abnormally high rate (e.g., daily) flags account-compromise.
  • Poison ciphertext. Malformed cipher_header crashes recipient's decrypt path. Mitigation: bounded ratchet-state size; client refuses to advance past a sanity limit; bad messages logged + dropped without affecting other peers' sessions.
  • Mass signed-prekey rotation storm. Signed prekeys are rotated weekly. If all 3 B devices rotate on the same day-of-week-at-midnight-UTC, that's 3 B writes in <1h = 800k writes/sec just for rotation. Mitigation: client randomizes rotation jitter (uniform over 7 days from registration); server enforces this server-side as a refusal if the previous bundle is <5 days old.

Iteration 2 — Multi-device addressing + sealed sender

System after this iteration

Legend. Solid slate = sync critical path. Dashed gray = out-of-band (cert verify, identity ref). Accent = new/changed (sender now fans out N times; sender-cert service is new; user_devices is new lookup).

Bottleneck

Iter 1 supports E2E for one device per user. The product needs multi-device: 1.5 devices/user average, max 4. Iter 1's flow encrypts to one recipient device key — the recipient's other devices receive nothing. Failing NFR: multi-device consistency — phone and laptop disagree on conversation state because only one of them gets the message. The fix forces sender-side fanout: sender must encrypt the message N times (once per recipient device) and upload N ciphertexts. This numerically shows up as a 1.5× multiplier on Cassandra write rate and sender upload bandwidth. Classified as bottlenecks#fanout-asymmetry — the same shape as #41's celebrity write fanout, but inverted: in #41 fanout was per-follower (cardinality 200 M, mitigated by hybrid push/pull); here fanout is per-recipient-device (cardinality 1–4 for 1:1, 50–1024 for groups), small enough that we can do the full fanout in Iter 2 and defer the group-scale optimization to Iter 4. The second bottleneck this iteration also addresses (in the same move) is a privacy gap: even with content encrypted, the server sees a per-message sender_user_id → recipient_user_id tuple, which is a powerful metadata leak (social graph reconstruction). Sealed sender is the canonical fix.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#fanout-asymmetry: (a) change the sender protocol from single-recipient to recipient-set; (b) introduce a recipient-device discovery step (sender needs to know how many devices to encrypt for); (c) rate-limit at a layer that doesn't require server to see sender identity (sender certificate, not user_id). Replication topology and queue schema are unchanged from Iter 0; this iteration's changes are on the sender protocol + a new sender-cert service.

Alternatives considered (C4)

  • Sender encrypts N times, server fans out to per-device queues — chosen. Preserves the security model (server cannot link, server cannot decrypt, server cannot impersonate). Sender bears the N× encrypt cost. Acceptable for 1:1 + small groups; Iter 4 fixes large groups.
  • Server reencrypts per-device on the sender's behalf — rejected. Would require server to hold a re-encryption key — breaks E2E entirely. Non-starter.
  • One per-user key shared across devices — rejected. Breaks forward secrecy (device-lost scenarios reveal all past messages on all devices); breaks per-device safety-number guarantee (user can't verify which specific device is in the session).
  • Sealed sender via Tor/Mixnet relay — rejected. Vastly improves metadata privacy but adds 500 ms+ latency and operational dependency on a third-party network. Sealed sender as Signal designed it (encrypted inner envelope, server sees only "anonymous-but-validated") is the cost-effective middle ground.

The chosen move

Three coupled changes:

  • Per-device addressing. SEND frame now carries an array [(device_id, cipher_payload, cipher_header), ...]. Sender first resolves GET /v1/users/{recipient_id}/devices (response cached client-side for 1h with cache-busting on safety-number change), then performs Signal session lookup per device (X3DH if no session, Double Ratchet step if session exists), then encrypts the plaintext N times with N distinct session keys. Server iterates the array and writes one row per (recipient_user_id, recipient_device_id) partition.
  • Sealed sender. The outer SEND frame no longer carries sender_user_id in plaintext-to-server. Sender wraps an inner envelope {sender_user_id, sender_device_id, ts, cipher_payload} and encrypts the inner envelope to the recipient's identity public key (a separate sealed layer using a one-shot ephemeral key + ChaCha20-Poly1305). The server sees only {recipient_user_id, recipient_device_id, sealed_envelope, sender_cert} — no link to sender identity. Recipient unwraps to recover sender identity. Cite Signal Sealed Sender.
  • Sender certificate. Server still needs a basis to rate-limit and to ban abusive senders. We introduce a short-lived (24-hour) sender certificate: a JWT-ish token signed by the server's sender-cert-issuer service, issued after the device proves account ownership (phone-number verification + recent device-attestation challenge), embedded in the sealed envelope but also exposed as a plaintext field in the SEND frame for rate-limit basis. Rate-limit table at the API tier keys on sender_cert_id, not sender_user_id — the server can ban a cert without learning who's sending. (If abuse-investigation requires identity, the recipient's spam-report unwraps the sealed envelope on their device and surrenders sender_id with consent.)

Why this addresses the bottleneck shape: per-device addressing makes multi-device consistency a direct property of the protocol (server reliably delivers to every addressed device). Sealed sender closes the metadata leak that Iter 1 didn't address. Sender-cert rate-limiting decouples abuse mitigation from identity exposure, preserving sealed-sender's privacy while keeping spam under control.

What breaks (C5)

  • Sender-cert theft. Compromised cert can be used to send messages for 24h until expiry. Detection: rate-limit per cert (10/sec burst); abnormal-pattern detection per cert.

Mitigation: certificate-id rate-limit + recipient-initiated spam reports (recipient unwraps envelope, attaches consent, ships to abuse-team-only audit log). Detection: recipient-side "report spam" buttons; server tracks per-cert spam-report rate.
  • Device-list staleness. Sender's cached device list for a recipient is 1h old; recipient added a new device 30 min ago. Sender's new message doesn't reach the new device until cache expires. Mitigation: server includes a device_set_version in the SEND response; sender invalidates cache + retries with new list on mismatch. Cost: ~1% of sends do a refresh round trip.
  • Hot-key meltdown on user_devices. Celebrity-account-equivalent: a user added to a viral broadcast list has every sender resolving its device list. Mitigation: Redis L1 cache in front of user_devices; refresh-ahead at >100 QPS for the same key (cite single-flight).
  • Cascading timeout on multi-device send. Sender SEND blocks until all N partition writes ACK. If one partition is slow, the whole SEND blocks. Mitigation: server returns ACK after >=1 partition succeeds (eventually-delivered-to-rest pattern); slow-partition writes complete in the background.

Iteration 3 — Edge POP pinning + FCM/APNs push wakeup for background devices

System after this iteration

Legend. Solid slate = sync. Dashed gray = async/lazy (background WS only on wake). Dashed red = wake signal across Apple/Google infra (best-effort, retryable). Accent = new/changed (foreground/background split, edge POP cluster, push relay, FCM/APNs integration).

Bottleneck

Iter 2 assumes every online device holds a WebSocket. Concurrent online devices ≈ 1 B (from constraints). Each WebSocket consumes per-connection memory (~5 KB for buffers + socket state) + a heartbeat slot. Per-pod cap from realistic Linux + JVM/Go runtime: ~200k WS per pod. 1 B / 200k = 5,000 WS pods just for connection termination — plus the ~1,500 pods we had for SEND throughput. Heartbeat alone is 1 B × 100 B every 15 s = ~6 GB/sec aggregate (50 Gbps), more than a small region's interconnect can absorb. Classified as bottlenecks#connection-saturation + network-bandwidth-bound. The signal: connection count is >100× the request rate, so the cost-per-connection (memory + heartbeat) dominates the cost-per-request (CPU on each SEND). Cross-cutting X5 subsetting and X7 push apply (cite cross-cutting#subsetting, cross-cutting#real-time-push).

Architectural decisions to revisit

Per bottleneck-to-decisions.html#connection-saturation: (a) reduce concurrent connection count via push wakeup (only keep WS for actively-in-app devices); (b) geo-distribute connection termination so heartbeat traffic doesn't traverse the WAN (edge POPs); (c) separate connection state from message state (push tokens persistent; WS-to-pod mapping ephemeral). Cross-cutting X6 (cells) earns a place — each POP becomes a cell with its own connection pool and a bounded blast radius.

Alternatives considered (C4)

  • Keep all 1B devices on WebSocket + edge POPs — partially chosen for foreground. Only sustainable for the ~100M actively-using-the-app subset; would need 5,000 pods globally for the full 1B.
  • FCM/APNs push wakeup for background — chosen. Trades a ~5 s wake-and-drain latency for a 10× reduction in WS pod fleet. Acceptable because background devices are by definition not actively expecting instant delivery.
  • Long-poll (HTTP/2 long-poll instead of WebSocket) — rejected. Same connection-count problem; HTTP/2 streams have higher per-stream overhead than raw WebSocket frames for our message shape.
  • Direct UDP / QUIC custom protocol — rejected. Would help NAT-traversal cost (no TCP keepalive needed; QUIC connection migration handles network changes), but adds protocol-complexity that isn't justified at the latency we currently meet (p95 80 ms). (🔀 if we needed sub-50ms p99 globally, QUIC connection-migration becomes attractive.)

The chosen move

Four coupled changes:

  • Foreground/background split. Devices report their lifecycle state on each WS frame and on app-suspend/resume. Foreground devices (~10% of installed base, ~100 M concurrent) maintain a persistent WebSocket. Background devices (~90%, ~900 M) close the WebSocket and rely on push wakeup.
  • Edge POP termination. WebSocket connections terminate at the geographically nearest POP (DNS + Anycast routing). Each POP is one cell with its own Envoy fleet, connection-registry Redis cluster (per-POP local), and routing layer that forwards SEND messages to the home-region queue tier over a smaller pool of pod-to-region long-lived gRPC streams.
  • Push relay tier. A new ~200-pod tier consumes a Cassandra CDC stream (or a Kafka topic the queue tier writes to) of "new ciphertext for offline device" events and dispatches an opaque wake payload (no ciphertext — just {msg_seq_high_water}) to the recipient's FCM/APNs token. Recipient device wakes, opens WS, sends last_seen_seq, drains.
  • POP-to-region async commit. POP buffers SEND locally for ~50 ms before batching to home region; if home region is unreachable, POP holds the batch (up to 5 s) and retries before degrading. RPO degrades from 0 to ≤5 s in the POP-isolated failure mode — explicit trade for the latency saving.

Why this addresses the bottleneck shape: 90% of the connection-count problem disappears when background devices stop holding WebSockets. Edge POPs localize heartbeat traffic (each POP's interconnect handles only its own ~20 M concurrent foreground devices' heartbeats, not the global 100 M). FCM/APNs absorb the wake-fanout into infrastructure we don't pay to operate (Google + Apple subsidize push delivery; the only cost we pay is APNs above 1 B pushes/day).

Iteration 4 — Group Sender's Keys: one ciphertext per group message, server fans out

System after this iteration

Legend. Solid slate = sync critical path. Dashed gray = lookup. Accent = new/changed (server-side group fanout; cipher_blob_store; group_members.sender_key_version).

Bottleneck

Iter 2 made the cost of large groups visible: sender uploads 1,536 ciphertexts (1.5 MB) for a 1024-member group message, taking ~2.4 s on 4G upload and ~1.5 s on phone CPU to encrypt. Effective fanout-weighted Cassandra writes/sec is 1.36 B (cite Iter 2 numbers) — would need ~27k Cassandra nodes at $26 M/mo, an order of magnitude above the rest of the system's budget. Classified as bottlenecks#fanout-asymmetry — the same pattern as #41's celebrity write fanout, but here the asymmetry is per-group (one sender produces one logical message; group cardinality multiplies the per-recipient-device write cost) rather than per-account. Sender CPU + sender mobile-network bandwidth are also failing — this is one of the rare bottlenecks where the constraint sits on the user's device, not on our infra.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#fanout-asymmetry for asymmetric fanout: (a) separate the "logical message" from the "per-recipient delivery record"; (b) let the server fan out the cheap part (delivery records) while sender produces the expensive part (encryption) only once; (c) share a symmetric key across the recipient set safely (forward-secret, rotation-on-membership-change). The cryptographic decision is the load-bearing one: how do we let N devices decrypt the same ciphertext without violating per-device forward secrecy? Sender's Keys (Signal Group Messaging) is the canonical answer.

Alternatives considered (C4)

  • Signal Sender's Keys — chosen. Per-(group, sender_device) symmetric chain key delivered out-of-band via 1:1 E2E channels; rotated on member add/remove. Forward-secret within the sender's chain (each message advances the chain; past messages cannot be decrypted by a future-leaked key). Acceptable trade: any current member can leak the current sender key — no cryptographic prevention possible (the threat model accepts member-trust within the group).
  • MLS (Messaging Layer Security, RFC 9420) — deferred. Sub-linear member-add cost (log N) and stronger group-membership guarantees. Would handle 1024-groups more elegantly. Why deferred: protocol is recent (2023 RFC); libsignal-MLS bridge is still maturing; current Sender's Keys flow with 1024-cap is good enough for our product cap. Probe C revisits.
  • Sender encrypts N times (Iter 2 baseline) — rejected for groups. Demonstrably infeasible above ~50-member groups (sender upload + sender CPU breaks the UX).
  • Server-side key escrow + symmetric re-encrypt — rejected. Breaks E2E (server holds the key). Non-starter for the product.

The chosen move

Adopt Signal's Sender's Keys protocol for all group messages. The protocol works in four phases:

  • Initial distribution. When a device joins a group (either created the group or was added by an admin), each existing sender device generates (or rotates) its per-group sender key (a 256-bit chain key + chain index 0) and delivers it to the new member via the new member's 1:1 E2E channel (Iter 2's flow). A 1024-member group with all-already-keyed senders requires N×(N-1) ≈ 1 M 1:1 messages on member-add — amortized over future messages, this is acceptable. New devices initially decrypt nothing until they receive the sender keys.
  • Sending. Sender encrypts the plaintext once with the current sender key (AES-CBC + HMAC-SHA256 as in Signal's group protocol; rotates the chain key per message). Sends one frame: {group_id, msg_id, cipher_payload, sender_cert}.
  • Server fanout. Server looks up group_members for the group, expands to all recipient devices via user_devices, persists the ciphertext blob once in cipher_blob_store, then writes a small reference row per (recipient_user_id, recipient_device_id) to device_inbox — reference is just {group_id, msg_id, ts}, ~50 bytes vs the previous 1 KB. Recipient devices drain references, fetch the blob once per group-message they receive.
  • Rotation on membership change. When a member leaves (or is removed), every existing sender device generates a fresh sender key and redistributes via 1:1 to all remaining members. Removed member cannot decrypt new messages. Also rotate periodically (every 7 days for chatty groups) to bound the damage of an undetected leak.

Why this addresses the bottleneck shape: sender's upload bandwidth drops from O(group_size × devices/user) to O(1) for the main hot path; the N× fanout is server-side and cheap (small reference rows, not 1 KB ciphertexts). The expensive cryptographic work is amortized over the group's lifetime (sender-key distribution happens once per group-membership-epoch, not once per message). The trade-off is a weaker trust model — any member can leak the sender key — but this is acceptable because any member can also copy/paste the plaintext anyway; the security boundary is the group's membership, which is what the user signs up for when they join a group.

  • Missing sender key on first message. Recipient joins a group, sender messages before recipient has fetched the new sender key. Recipient sees "cannot decrypt message"; requests sender key out-of-band via 1:1 from the sender. UX: "🔒 Waiting for key from {sender}..." with retry. Mitigation: sender pre-emptively pushes sender key to new member as part of the membership-add flow.
  • cipher_blob_store hot key. Viral message in a 1024-group: every recipient device fetches the blob. 1,536 reads/sec on one blob partition. Mitigation: Redis L1 cache in front of cipher_blob_store with single-flight (cite single-flight); blob is content-addressable by (group_id, msg_id) so cache-key is stable. Cache hit-rate > 95% on hot blobs.
  • Reference row + blob inconsistency. Server writes N references, then blob write fails. Recipients drain references pointing to nothing. Mitigation: write order is blob-first, then references; references include blob_etag so recipient verifies; orphaned blobs are GC'd by daily sweep (no references = TTL).

Probe iterations (the interviewer pulls the thread)

Beyond the 5-Core budget. Each in a <details> block.

Probe A — Multi-region active-active (EU residency + RPO ≤ 5 s globally)

Three home regions (US, EU, APAC). User's home region is determined by phone-number country code; user data (account, prekey bundles, queue) is anchored to home region. Cross-region messaging routes through home-region pairing (sender's POP forwards to sender's home, which dispatches via DCI to recipient's home). Cassandra cross-region with NetworkTopologyStrategy RF=3 per region + asynchronous repair stream; per-message commit is LOCAL_QUORUM in sender's home + async replication to recipient's home (≤5 s RPO cross-region). GDPR EU-residency: account + prekey bundles + queue stay in EU region for EU-rooted users; non-EU peer messaging recipients' queues are EU-resident still (since recipient's home region anchors them); EU user's outbound to non-EU recipient: the recipient's queue holds the message in non-EU but the message originated from EU. Audit log of cross-region transit per regulation. Cite cross-cutting#privacy.

Probe B — Cloud backup of message history (user-encrypted)

Messages are E2E, but if a user loses their device, history is gone (queue retention is 30 days, plus the device's local store). Optional cloud backup: client encrypts daily-rollup of plaintext history under a backup encryption key (BEK) which is encrypted under a password-derived key (PDK = Argon2id(passphrase, salt, m=64MB, t=3)) and stored in an HSM-backed service. User remembers passphrase or stores it in a hardware token; recovery requires both the backup blob (in object store) and the passphrase. Server cannot decrypt without passphrase. Threat: brute-force on the passphrase — mitigated by Argon2 cost + HSM rate-limit on PDK derivation. WhatsApp's E2E backup launched in 2021 follows this shape.

Probe C — MLS migration (for future 10k-channel groups)

If the product evolves to support 10k-member channels (Telegram-style), Sender's Keys' rotation cost (10k × 9999 1:1 dispatches per member-add) becomes infeasible. Migrate to MLS (RFC 9420): a TreeKEM-based group key agreement with O(log N) cost per member-add. Migration strategy: dual-protocol period where both Sender's Keys and MLS sessions coexist per group; new groups created with MLS; existing groups migrate on rekey events.

Probe D — Spam without server-side content visibility

Server cannot read messages → cannot classify content as spam. Mitigations: (1) sender-cert rate-limit (Iter 2); (2) recipient-initiated spam report (recipient client decrypts and ships the message to a separate abuse-handling service with explicit consent and consent-revocable timeout); (3) graph-based spam detection (sender_user_id frequency to recipients who never replied; metadata-only signal); (4) honeypot accounts that report all received messages; (5) network-effect priors (group-add by a stranger triggers a friction prompt). No single move catches everything; layered defense is the practice.

Degradation contract

  • FCM/APNs down: background devices delivery latency degrades from ~1.5 s to "next time user opens app." Foreground unaffected. SEND continues to succeed (durably queued). Cite failure-modes#slow-dependency.
  • POP failure: Anycast withdraws BGP advertisement; clients DNS-retarget within ~10 s. In-flight POP-buffered SEND lost (≤5 s window) — client retries on reconnect via msg_id idempotency. Cite failure-modes#bad-deploy.
  • Cassandra cross-AZ split: writes return 503-Retry-After (LOCAL_QUORUM fails with 1 of 3 AZs lost — that's the contract). Clients buffer + retry. RPO=0 maintained (no accepted-but-lost messages).
  • Prekey-bundle service down: first-contact between new peers cannot establish session — degrades to "🔒 Setting up secure connection..." until service recovers. Already-paired devices unaffected.
  • Sender-cert service down: can't issue new certs; existing 24h certs continue working. Newly-registered devices cannot send until recovery.

Real-system comparison

WhatsApp's own published architecture (Erlang-based; Yaws + Mnesia + custom routing) achieved similar scale on much smaller hardware due to Erlang's per-process concurrency model (millions of WebSocket-equivalent processes per node). The architecture above uses Go/Envoy + Cassandra because that's a more transferable stack to interview at FAANG-shaped employers; the structural decisions (per-device store-and-forward, Signal protocol for E2E, Sender's Keys for groups, mobile-push wake) match Signal's Sealed Sender + WhatsApp's E2E rollout publications. Where we diverge from Signal/WhatsApp: we use Cassandra where WhatsApp uses Mnesia (operational maturity preference); we use FCM/APNs the same way; we use libsignal for the crypto rather than re-implementing.

The non-obvious public lesson from Signal/WhatsApp engineering: the operational cost of E2E is not the cryptography — it's the loss of every server-side feature that a non-E2E IM has (search, server-side spam, content-based ML, etc.). Each of those is replaced by a client-side or metadata-side approximation that's strictly weaker. The interview signal Iter 1 tests for is whether the candidate names these losses rather than ignoring them.

Anticipated pushbacks

Eight stress tests against the derivation above. Each answer in 3–6 sentences; if a probe requires consulting another section, the iteration's prose hasn't earned its depth and needs more inline.

What if message volume tripled to 30 M/sec? (scale flip)

Bottleneck shifts further into Cassandra queue write rate. Iter 4's 9,100-node fleet scales to ~27,300 (linear in write rate) costing ~$27M/mo, plus cross-AZ egress 3× to ~$5M/mo. Net ~$45M/mo. Before scaling Cassandra linearly, consider: (a) raising group cap to fewer-and-larger groups by reducing 1024 to 256 (4× saving on group-write rate); (b) compressing reference rows to 30B via a binary format; (c) introducing a queue-write coalescing layer (one Cassandra write per (recipient, 100 ms batch)). The current +30% headroom buys ~2 months of organic growth before this becomes load-bearing.

Why Cassandra for the queue and not ScyllaDB, FoundationDB, or DynamoDB? (alt-engine)

ScyllaDB would handle ~2× the throughput per node (C++ shard-per-core) — saving ~50% of the Cassandra fleet ($4.5 M/mo). We chose Cassandra because our operator team has institutional knowledge of Cassandra operations (Reaper, Sidecar repair, TWCS tuning) and the migration cost (8–12 quarters of operator-team work) exceeds the savings horizon. If we were greenfield today, ScyllaDB would likely win. FoundationDB's transactional model is overkill (we don't need cross-partition transactions on the queue) and per-partition throughput is lower. DynamoDB at this scale would cost ~$78 M/mo just for queue writes (cite C4 in Iter 0). Cite storage-engines.html and replication-decisions.html.

What changes under GDPR + EU residency? (regulatory)

Probe A's multi-region active-active design becomes load-bearing. EU users' accounts, prekey bundles, group state, and queues stay in EU region; cross-region replication is selective (EU↔non-EU traffic only when the recipient's home region is non-EU). Account-deletion (right-to-erasure) is already handled by 30d TTL + cascade-delete in user-bound stores; the new addition is an audit log of cross-border data transit per Schrems II. Critically, message content is unaffected by GDPR because the server can't read it — the regulatory burden falls on metadata (account info, push tokens, sender_cert audit, push-relay logs). Cite cross-cutting#privacy.

The entire Cassandra queue cluster goes down right now. What happens? (failure)

SEND fails with 503-Retry-After at the home-region API tier; clients buffer locally and retry. Cross-region async replica continues serving its own region's traffic, so a single-region failure is contained (assuming Probe A's multi-region is live). In-region failure: client-side message buffer (~100 MB per client; days of typical use) bridges the recovery window. Push relay halts (no new events to consume). Foreground devices' WS stays connected with no new MESSAGE frames. RTO target: 30 min for a full-cluster restart (Cassandra restart sequence for 9,100 nodes is ~10 min in waves). Cite failure-modes#bad-deploy.

Where does the bill come from? What's the cost knob? (cost)

Cassandra queue dominates at ~$9 M/mo (70% of total), driven by 1.36 B writes/sec including small-row references from Iter 4's group fanout. The cost knob is retention TTL: dropping from 30 days to 14 days saves ~$2M/mo (storage + smaller compaction working set). The product cost is reduced offline-tolerance — users returning from a 3-week vacation lose unread messages. Second-dominant: cross-AZ egress ($1.6 M) which is structurally tied to RF=3 across 3 AZs — reducing to RF=2 cross-2-AZ saves 33% of egress but raises the chance of permanent message loss on AZ-pair failure. Cite cost-shapes.html.

How do you roll out Iter 1 (E2E) to a live plaintext-server product without downtime? (migration)

Dual-protocol period (WhatsApp took ~6 months for this in 2014–2016). Server accepts both plaintext SEND (legacy clients) and ciphertext SEND (new clients) into the same device_inbox; cipher_header column NULL means plaintext. New client versions ship with E2E on by default; old versions remain plaintext. Force-upgrade phase: gradually refuse plaintext sends from active clients with prompts to update. Historical plaintext messages stay plaintext until 30d TTL aging-out — cannot be retroactively E2E-encrypted because the server would need to re-encrypt them with peer keys it doesn't have access to. Rollback trigger: cipher-decrypt-error-rate > 1% at the client surface (indicates protocol bug). After 12 months, drop plaintext acceptance.

What's the abuse / spam path when the server can't read content? (adversary)

Five layers, all weaker individually than content-scanning would be: (1) sender-cert rate-limit at the edge (Iter 2) — bounds messages-per-second per device; (2) recipient-initiated spam report — recipient client decrypts and ships the message to an abuse-only service with consent; (3) graph-anomaly signal — sender-cert messaging many recipients who never reply within 24h; (4) honeypot accounts that report all received messages; (5) friction prompts when a stranger adds a user to a group. The cost-of-spam is higher per-message for attackers (they can't bulk-send via one cert because of rate-limits), but content can't be classified before delivery the way Gmail does. This is the structural cost of E2E and we name it explicitly — staff+ candidates who pretend the server can still do content-scanning under E2E fail this signal.

Production delivery is slow right now — where do you look first? (monitoring/debug)

Triaging order on the SLI dashboard (from C6 ops): (1) edge POP WS connection count + heartbeat-loss — per-POP; if one POP is hot, DNS retarget runbook. (2) Cassandra queue write p99 + partition-heat top-K — if a partition is hot, the Iter 0 hot-partition-shard runbook (synthetic bucket key). (3) FCM/APNs push delivery success rate — if <99%, fall back to WS-only foreground until recovery (background users see degraded latency — honest tradeoff). (4) cipher_blob_store read hit-rate on Redis L1 — if dropped, group-message fetch path degrades. (5) prekey-bundle fetch p99 — only matters for new-contact establishment, isolatable failure surface. Each SLI maps to a specific iteration's bottleneck so the on-call knows where the load-bearing code is.

S7. Key Takeaways

  • The trust-model flip changes everything. Same LC kernel as Twitter (#41) but the server-untrusted constraint makes none of #41's iterations transfer. Every "server-side cleverness" #41 used (rank, mute filter, dedupe, content-based fanout) becomes structurally impossible. Recognizing this before diving into the iteration arc is the staff+ signal.
  • E2E is not an iteration on top of a non-E2E design; it's the design. Iter 1 in F5 isn't "add encryption to messages"; it's a re-shaping of where compute lives (all crypto moves to clients), what server-side stores exist (prekey-bundle service + identity-key registry are not optional), and what server-side features become permanently impossible.
  • Per-device, not per-user, is the right partition unit when multi-device is real. The queue partitions on (recipient_user_id, recipient_device_id) — a user has 1–4 devices, each is an independent delivery endpoint, each has its own cryptographic session. Treating "user" as the partition unit (the natural #41 reflex) silently breaks multi-device.
  • Sealed sender is the missing piece that makes E2E "actually private." Without it, the server reconstructs the social graph from message-routing metadata even though it cannot read content. Sealed sender + sender-cert rate-limiting decouples abuse mitigation from identity exposure — a non-obvious design that pays for itself in privacy at modest cost.
  • The sender-side cost (encrypt, upload bandwidth) becomes a real bottleneck. Iter 2's fanout would have made 1024-group messages take 4 seconds on the sender's phone — UX-breaking even though server cost was technically feasible. Sender's Keys (Iter 4) is a cryptographic move that prioritizes the constraint sitting on the user's device. Knowing when the bottleneck moves off your infra is a staff+ skill.
  • Offline-tolerance reshapes both ends. The queue's 30-day TTL is a product decision masquerading as an infra decision; the cost of the TTL choice is the dominant cost knob in C7. Pricing UX features in infra dollars is the right discipline.
  • Foreground/background split is the cheap way to scale WebSocket fleets. The temptation is to scale WS pods linearly with online users; the win is to redefine "online" via app lifecycle (~10% foreground at any moment) and lean on mobile-push for the rest. Each platform (Apple, Google) absorbs the wake-fanout cost subsidized.
  • Iterations expose the next iteration's cliff. Iter 2 made the group-fanout cliff visible (would cost $27 M/mo). Iter 4 collapsed it. Naming the "this iteration creates the next bottleneck" pattern aloud is how to keep the arc honest.
