Distributed Rate Limiter at Scale

Enforce per-customer rate limits across 10,000 API servers with less than 5% error and under 1 ms of added latency — without melting your Redis cluster. We start from the LC #362 / #359 single-machine kernels (a sliding-window hit counter and a 1-in-N-seconds gate), lift them onto one process, then iterate to local-first counting with periodic async reconcile, adaptive sync, and a fail-open degradation contract.

NALSD Walkthrough • ~18 min read • LC #362 (Design Hit Counter) + LC #359 (Logger Rate Limiter)

LC kernel — two of them, one primitive. A rate limiter is a hit counter plus a gate. LC #362 “Design Hit Counter”: count hits in the last 5 minutes — a deque (or ring) of timestamped buckets; on hit(t) push/merge into the bucket for t; on getHits(t) evict buckets older than t−window and sum the rest. O(1) amortized, O(window/bucket) space per series — that is the sliding-window counter. LC #359 “Logger Rate Limiter”: print a message only if it hasn't been printed in the last N seconds — a hashmap key → last_allowed_time; on shouldPrint(t, key), allow iff t ≥ last[key] + N, then set last[key] = t. O(1), O(#distinct keys) space — that is the 1-in-N gate, which is just a token bucket of capacity 1 refilling one token every N seconds. They are the same primitive at different granularities: a deque of buckets is the fine-grained version; a single (count, window_start) pair is the coarse version; a (tokens, last_refill) pair is the token-bucket version. Properties that break at 10,000 servers / 50 M req/sec: (1) one counter doesn't fit / can't take the QPS — one customer's limit is a single hot key; a global atomic increment on it serializes the whole platform; (2) a per-request round trip to a central store eats the latency budget — 0.5 ms × 50 M/sec is both a bandwidth wall on the store and 0.5 ms of head-of-line latency on every API call; (3) sliding-window-log (the exact version — store every hit's timestamp) is exact but O(window×rate) memory per key — a 1 M-req/sec customer would need a million timestamps in flight; you must approximate; (4) the limiter can't take down the API — it must fail open (admit everything) on its own outage, never fail closed. The C8 production cousin: a deque-on-one-machine becomes a local per-pod approximate counter (a count + a window start, or a token bucket) checked in-process with zero network, periodically reconciled against a sharded regional counter (Redis Cluster + Lua) that holds the eventually-consistent global view — with a fail-open fallback to a conservative local limit when that store is unreachable. The Logger-Rate-Limiter 1-in-N gate is the same machinery applied per-(IP, endpoint, message) instead of per-tenant — see #42 for it applied to per-customer error quotas.

The Scenario

You run an API platform (like Stripe, Twilio, or AWS API Gateway). Customers purchase rate limit tiers. A large enterprise customer might have a limit of 1 million requests per second. A free-tier user might have 100 requests per second. You must enforce these limits globally across your entire fleet — and hierarchically: a tenant has a global limit and per-API-key sub-limits, plus per-IP and per-endpoint limits layered on top.

System class: infra primitive — an embedded library in every API server (the per-request allow/deny check, which must be local and lock-free) plus a thin distributed-counter backend (the reconciliation store). The FR vocabulary is operations + semantics (what calls it exposes, what guarantees they make), not user-facing capabilities; the load-bearing NFRs are per-request added latency, the QPS the backend can take, approximation error, and fail-open behaviour.

Why This Is Hard

If all 10,000 servers were one machine, rate limiting is trivial — it's exactly LC #362 / #359: keep a counter (or a deque of buckets, or a token bucket) per customer, increment on each request, reject when over. But with 10,000 servers, a single customer's requests are scattered across many machines; no single server sees the full picture. How do you count globally without adding latency to every request — and without a single hot counter becoming the bottleneck?

Requirements & Scope (C1)

Functional requirements (operations + semantics)

  • allow(scope_key, cost=1) → ALLOW | DENY(retry_after_s, limit, remaining, reset_at) — called in-process on every API request, for each applicable scope; must be O(1), no network call on the hot path. scope_key = (tenant_id) or (tenant_id, api_key_id) or (ip) or (endpoint, tenant_id) — the same machinery, different keys. Hierarchical: a request is allowed only if it passes every scope it falls under (tenant global AND its key's sub-limit AND per-IP AND per-endpoint).
  • set_limit(scope_key, limit_per_sec, window) — control plane sets/updates a scope's limit; propagates to all API servers within seconds (eventual).
  • peek(scope_key) → {limit, remaining, reset_at} — for the X-RateLimit-* response headers; reads local state, no round trip.
  • backend: report_and_fetch(scope_key, local_delta) → global_count — the reconciliation call: a server tells the regional counter how many it's allowed since last sync and reads back the approximate global total. Called from a background thread, batched, never on the request path.

Out of scope (named, not silently dropped): the API gateway / auth / routing the limiter lives inside (upstream); quota as opposed to rate (monthly request budgets with billing semantics — that's a metering pipeline, a different durability class, see #42); WAF / bot-detection / abuse-scoring (a security layer that may feed the limiter but isn't it); the load balancer's choice of which server sees a request (we consume that distribution, we don't control it); cross-region strong consistency on the count (we deliberately accept slight over-admission — see flagged assumptions); long-term retention of per-request decisions (the analysis/billing pipeline, downstream).

Flagged assumptions (would confirm with the interviewer): (1) “< 5% error” is a soft SLO and slight over-admission is preferred to under-admission (rejecting a paying customer's legitimate traffic is worse than letting a few extra through) — the design optimizes for “don't under-admit”; (2) a customer's traffic clusters on a subset of servers (~100–500 of 10,000), not uniformly — this is the number that breaks naive local-only limiting; (3) NTP keeps clock skew < ~10 ms fleet-wide — windows depend on clocks; (4) the limit catalog is small (~10,000 tenants × a few scope types ≈ tens of thousands of limits, ~tens of KB) and changes rarely; (5) the limiter must fail open — its own outage admits everything rather than denying everything (a revenue-generating API can survive un-enforced limits for a few minutes; it cannot survive a self-inflicted total outage).

Back-of-Envelope Math

Ideal vs Reality

Ideal distribution for a customer with 1M req/sec limit:
  1,000,000 req/sec ÷ 10,000 servers = 100 req/sec per server.

  If we set each server's local limit to 100 req/sec, it works IF
  traffic is perfectly uniform across all 10,000 servers.

Reality: traffic is NOT uniform.
  A customer's traffic might hit only 100-500 of the 10,000 servers
  (depending on load balancer configuration, client behavior, etc.)

  If traffic hits 200 servers instead of 10,000:
    Each of those 200 servers sees: 1M / 200 = 5,000 req/sec.
    But local limit is set to: 1M / 10,000 = 100 req/sec.
    Each server thinks: "5,000 > 100 → reject!"
    Customer effectively gets: 200 servers × 100 req/sec = 20,000 req/sec.
    That is 2% of their 1M limit. Absurdly under-allowing.

The Fundamental Problem

The fundamental problem:
  Local limits = global_limit / total_servers works only with
  perfectly uniform distribution. In practice, traffic clusters
  on a subset of servers, causing massive under-allowance.

Algorithm Taxonomy (cite, don't restate)

Fixed-window-counter, sliding-window-log, sliding-window-counter, token-bucket, leaky-bucket, GCRA — the comparison (memory, burst behaviour, smoothness, edge cases) is the source-of-truth page rate-limiting.html; this page picks among them with scenario-specific numbers and won't re-derive the algorithms. What matters here for the BOE: sliding-window-log is the only exact one (store every hit's timestamp, evict the old ones, count) — and it costs O(window×rate) memory per key, so a 1 M req/sec customer over a 1-second window needs ~1 M timestamps in flight. That disqualifies it at scale; everything else is an approximation, which is fine given the < 5% soft SLO. We'll name the specific choice (sliding-window-counter, with token-bucket as the alternative) when we get to the iteration that needs it — the Naive Design below uses the dumbest exact thing (a per-request central INCR with a fixed 1-second window) so every later improvement is measurable.

The core tension: Accurate rate limiting requires knowing the global request count. But checking a global counter on every request is expensive. We need to find a middle ground between “check globally on every request” (accurate but slow) and “count locally only” (fast but inaccurate).

Naive Design: Centralized Redis Counter

The most straightforward approach: every request, before processing, increments a counter in Redis. If the counter exceeds the limit, reject the request.

Redis Counter Code

import redis
import time

r = redis.Redis(host='rate-limiter.redis.cluster')

def check_rate_limit(customer_id, limit):
    """Check if request is within rate limit. Returns True if allowed."""
    current_second = int(time.time())
    key = f"rate:{customer_id}:{current_second}"

    # Atomic increment + get
    # Using Lua script to avoid race condition between INCR and GET
    lua_script = """
    local current = redis.call('INCR', KEYS[1])
    if current == 1 then
        redis.call('EXPIRE', KEYS[1], 2)
    end
    return current
    """
    count = r.eval(lua_script, 1, key)

    if count > limit:
        return False  # Rate limited!
    return True  # Allowed

Why Lua Script Instead of Separate INCR + EXPIRE?

Race condition with separate commands:

  Thread A: INCR "rate:cust1:1710000000" → returns 1 (first request this second)
  Thread A: (about to call EXPIRE, but gets preempted)
  Thread B: INCR "rate:cust1:1710000000" → returns 2
  Thread B: (key already exists, doesn't need EXPIRE)
  Thread A: EXPIRE "rate:cust1:1710000000" 2 → sets TTL

  This particular interleaving works fine. But consider:

  Thread A: INCR "rate:cust1:1710000000" → returns 1
  — Server crashes before calling EXPIRE —

  The key has no TTL → it persists forever → counter never resets →
  customer is permanently rate-limited!

  The Lua script runs atomically on the Redis server:
  INCR + conditional EXPIRE happen as one operation.
  No partial execution possible.

Alternatives considered (C4) — the algorithm

The Naive Design's one decision: which algorithm, run centrally, on every request. We pick the dumbest exact one (fixed-window INCR) on purpose. The space (comparison details → rate-limiting.html):

  • Fixed-window counter (central INCR, 1-second key) — chosen for Naive. One key per (scope, second), INCR + compare, EXPIRE for cleanup. Exact within a window, trivial, O(1) memory per key. Disqualifying flaw for production: a burst straddling a window edge (900 k at 0:59.5, 900 k at 1:00.5) is ~2× the limit over any 1-second span — fixed in the Final Design with sliding-window-counter. (And it's central, the bottleneck this whole page fixes.)
  • Sliding-window log (sorted set of hit timestamps) — rejected. Exact, no boundary problem — but O(window×rate) memory per key (a ZADD per request, a ZREMRANGEBYSCORE to evict) → ~1 M members in flight for a 1 M req/sec / 1-second customer, and the per-request work is now O(log n) not O(1). The exactness isn't worth that cost given the soft SLO.
  • Sliding-window counter (weighted blend of current + previous fixed window) — deferred to the Final Design. Two counters per key; estimated count = prev × overlap_fraction + curr. O(1) memory, bounds the boundary burst to < 2× the limit, no per-hit storage. The right approximation — but it doesn't fix the central problem, so it's a Final-Design refinement on top of the local-first architecture, not a Naive choice.
  • Token bucket / leaky bucket (a (tokens, last_refill) pair) — the production alternative. Refill rate tokens/sec up to capacity; allow iff tokens ≥ cost, then deduct. O(1) memory, smooths to a true rate (no boundary burst at all — capacity is the explicit burst allowance), and it's exactly the LC #359 1-in-N gate generalized (capacity 1, refill 1/N). We'd choose this over sliding-window-counter when callers want a configurable burst allowance; the two are interchangeable in this design — both are O(1)-per-key local state. (GCRA — the “virtual scheduling time” reformulation of leaky-bucket — is the same thing with one number instead of two; rate-limiting.html has it.)

What Breaks

Redis Cannot Handle the Load

Every request triggers a Redis call:
  50 million req/sec × 1 Redis operation = 50 million Redis ops/sec.

Redis capacity (single instance):
  ~100,000-200,000 ops/sec for simple commands.

Instances needed:
  50,000,000 / 150,000 = 333 Redis instances.

  That's 333 Redis servers JUST for rate limiting.
  At ~$500/month each = $166,500/month.
  And this doesn't account for Redis cluster overhead,
  failover, or the complexity of managing 333 instances.

Latency Is Too High

Each Redis call:
  Network round trip (same DC): ~0.2-0.5 ms
  Redis processing:              ~0.01 ms
  Total:                         ~0.3-0.5 ms

  Our budget: < 1 ms. A single Redis call is within budget,
  but BARELY. And that's best case — under load with 333 instances,
  some calls will be slower.

  The real problem: 0.5 ms added to EVERY request, 50 million times/sec.
  That's 25,000 seconds of aggregate latency per second.
  Each request is 0.5 ms slower than it needs to be.

Redis Is a Single Point of Failure

If Redis is down, what do you do?

  Option A: Block all traffic.
    → 50 million req/sec rejected. Every customer is down.
    → Unacceptable for a revenue-generating API platform.

  Option B: Allow all traffic.
    → Rate limits not enforced. Customers can consume unlimited resources.
    → A single abusive customer could overload your entire backend.

  Option C: ???
    → We need a fallback that degrades gracefully.

Classified against the canonical taxonomies

  • One counter, the whole platform's QPS → hot key + lock contention. One customer's limit is a single key; a 1 M req/sec customer means 1 M atomic INCRs/sec on one key on one shard — the key (and the shard owning it) is pegged regardless of how many Redis nodes you add. Per bottleneck-to-decisions: client-side coalescing — count locally, sync periodically — is the first move (Iteration 1).
  • A round trip on every request → network-RTT-bound. p50 added latency ≈ 1×RTT ≈ 0.2–0.5 ms intra-DC; the API server's CPU is idle waiting. First move: move the decision closer (in-process) and batch the sync — Iteration 1.
  • The store as a hard dependency → slow dependency / leader loss. Redis failover takes seconds; during it, every API request blocks on the limiter. The fix is a degradation contract that treats unreachable-counter as “use the local fallback,” never “block” — Iteration 3.
  • Clients retrying a 429 in a tight loop → retry storm. A denied client that ignores Retry-After and re-sends immediately doubles its own offered load — mitigated by the response contract (mandatory Retry-After; cite api-design-patterns#rate-limit-response) plus per-caller backoff guidance.
  • Windows depend on wall clocks → clock skew. Two servers disagreeing on “which second is it” by > the sync interval over-admit; NTP keeps skew < ~10 ms, which the accuracy math (Iteration 1) folds in as a small term.
  • Access pattern. The increment stream is write-heavy (all writes, no reads on the hot path) but power-law across keys (a few huge tenants dominate) and bursty (10× spikes) — which is why the design is local-first (absorbs the write rate and the burst) plus adaptive sync (spends backend ops on the few keys near their limit).

Iteration 1: Local Counters with Periodic Sync

The key insight: instead of checking Redis on every request, count locally and sync periodically. Each server maintains local counters per customer. Every sync interval (e.g., 500 ms), it reports its local count to Redis, fetches the global count, and adjusts its local allowance.

How It Works

Each API server maintains:
  - local_count[customer_id]: requests seen since last sync
  - local_limit[customer_id]: how many more requests to allow before next sync

Sync cycle (every 500 ms):

  1. REPORT: Server sends its local_count to Redis.
     Redis command: INCRBY "global:cust1:window" local_count
     → Redis now has the approximate global count.

  2. FETCH: Server reads the global count from Redis.
     Redis command: GET "global:cust1:window"
     → Server knows: "globally, 450,000 requests have been counted this window."

  3. ADJUST: Server calculates its local limit for the next 500 ms.
     remaining_global = global_limit - global_count
     my_fair_share = remaining_global / estimated_active_servers
     local_limit = my_fair_share × sync_interval

  4. RESET: local_count = 0. Start counting fresh.

Between syncs: pure local operations, zero network calls.
  if local_count < local_limit:
      allow request
  else:
      reject request

Redis Load Calculation

With periodic sync every 500 ms:

  Operations per sync per server: 2 (INCRBY + GET)
  Servers: 10,000
  Syncs per second: 2 (every 500 ms)

  Total Redis ops/sec: 10,000 servers × 2 ops × 2 syncs/sec = 40,000 ops/sec

  Compare to naive approach: 50,000,000 ops/sec

  Reduction: 50,000,000 / 40,000 = 1,250× fewer Redis operations!

  One Redis instance (150K ops/sec capacity) handles this trivially.
  We need exactly 1 Redis instance (with a replica for failover).

The tradeoff: We reduced Redis load by 1,250×, but we introduced a window of uncertainty. Between syncs (up to 500 ms), each server makes decisions based on stale global data. How much error does this cause?

Accuracy Analysis

Worst-case over-allowance scenario:
  Customer limit: 1,000,000 req/sec.
  Customer traffic hits 200 servers evenly: 5,000 req/sec each.
  Sync interval: 500 ms.

  Between two syncs (500 ms window):
    Each server allows up to: local_limit requests.
    After a sync, each server gets a fair share:
      remaining = 1,000,000 - current_global_count
      local_limit = (remaining / 200 servers) × 0.5 seconds

  At the start of a second (global_count ≈ 0):
    local_limit per server = (1,000,000 / 200) × 0.5 = 2,500 per server per 500ms

  In the first 500 ms, all 200 servers allow 2,500 each:
    Total allowed = 200 × 2,500 = 500,000 (exactly half the limit for half the second) ✓

  After sync at 500 ms:
    Global count reported: 500,000.
    Remaining: 500,000. Each server gets 2,500 again.
    Total for the second: 1,000,000. Exactly at the limit. ✓

  But what if servers sync at different times (clock skew)?
    Worst case: all servers count locally for a full sync interval
    BEFORE any of them report to Redis.
    Over-allowance per interval ≈ local_limit × num_active_servers × clock_skew_fraction

    With NTP-synchronized clocks (skew < 10 ms):
    Over-allowance ≈ 2,500 × 200 × (10/500) = 10,000 requests.
    That's 10,000 / 1,000,000 = 1% error. Within the 5% budget. ✓

Implementation

import threading
import time
import redis

class DistributedRateLimiter:
    def __init__(self, redis_client, sync_interval_ms=500):
        self.redis = redis_client
        self.sync_interval = sync_interval_ms / 1000.0
        self.local_counts = {}    # customer_id → count since last sync
        self.local_limits = {}    # customer_id → allowed until next sync
        self.customer_limits = {} # customer_id → global limit per second
        self.lock = threading.Lock()

        # Start background sync thread
        self.sync_thread = threading.Thread(target=self._sync_loop, daemon=True)
        self.sync_thread.start()

    def allow_request(self, customer_id):
        """Returns True if request is allowed. O(1), no network call."""
        with self.lock:
            count = self.local_counts.get(customer_id, 0)
            limit = self.local_limits.get(customer_id, self._default_limit(customer_id))

            if count < limit:
                self.local_counts[customer_id] = count + 1
                return True
            return False

    def _sync_loop(self):
        """Background thread: sync with Redis every sync_interval."""
        while True:
            time.sleep(self.sync_interval)
            self._sync_all_customers()

    def _sync_all_customers(self):
        """Report local counts to Redis, fetch global, adjust limits."""
        with self.lock:
            customers_to_sync = dict(self.local_counts)
            self.local_counts = {}  # Reset local counts

        pipe = self.redis.pipeline()
        window_key = f"window:{int(time.time())}"

        for customer_id, local_count in customers_to_sync.items():
            key = f"rate:{customer_id}:{window_key}"
            pipe.incrby(key, local_count)  # Report local count
            pipe.get(key)                   # Fetch global count
            pipe.expire(key, 3)             # Auto-cleanup

        results = pipe.execute()

        with self.lock:
            for i, (customer_id, _) in enumerate(customers_to_sync.items()):
                global_count = int(results[i * 3 + 1] or 0)
                global_limit = self.customer_limits.get(customer_id, 1000)
                remaining = max(0, global_limit - global_count)
                # Estimate active servers from the ratio
                estimated_servers = max(1, global_count / max(1, local_count))
                fair_share = remaining / estimated_servers
                self.local_limits[customer_id] = int(fair_share * self.sync_interval)

    def _default_limit(self, customer_id):
        """Conservative default before first sync."""
        global_limit = self.customer_limits.get(customer_id, 1000)
        # Assume 1000 active servers (conservative)
        return int((global_limit / 1000) * self.sync_interval)

Alternatives considered (C4) — the topology

The bottleneck (from What Breaks): one hot counter, a round trip per request. The decision: where does the count live, and when does coordination happen. Options:

  • Local-first with async reconcile — chosen. Each server counts in-memory (the LC kernel, lifted), checks locally (zero network on the hot path), and a background thread periodically reports its delta to a shared counter and reads back the global total to re-budget. Properties: hot path is ~1 µs and lock-free; backend QPS drops by the sync ratio (~1,250×); the cost is bounded over-admission proportional to the sync interval. The standard pattern (this is what Cloudflare, Stripe, Envoy's global rate limiter all do in spirit).
  • Central store on every request (the Naive Design) — rejected. Exact, but it's the hot-key + RTT bottleneck this iteration exists to fix.
  • Local-only with no backend, just gossip — rejected. Servers gossip their counts peer-to-peer; no central store. Removes the SPOF but: a 10,000-node all-to-all gossip is its own scaling problem (O(N²) messages), convergence is slow (seconds), and the “how many servers are active for this customer” estimate — the thing that makes local limits accurate — is exactly what gossip is bad at delivering quickly. A central counter is the aggregator; it's cheap because each server talks to it ~twice per sync interval, not to every peer.
  • Redis Cluster with a Lua check-and-increment per request (no local layer) — rejected. Moves the atomic INCR+compare server-side (one round trip instead of two, no race) — but it's still a round trip per request and still a hot key per customer. It's the right backend for the reconcile store (we use it for that), just not on the hot path.
  • Hierarchical reconcile (per-rack aggregator → regional counter) — deferred. A two-tier sync (servers → rack-local aggregator → region) would cut the regional store's connection count further; worth it only if the regional store's connection count (not its op rate) becomes the limit — it doesn't at 10 k servers. Named here so it's not a surprise at 100 k.

What breaks (C5; cite failure-modes.html)

  • Bad active-server estimate → divergent local budgets. The my_fair_share = remaining / estimated_active_servers step depends on guessing how many servers are currently serving this customer. Guess too high → each server's slice is tiny → the customer is under-admitted (gets < 90% of their paid limit); guess too low → over-admission. Defence: derive the estimate from the ratio Redis reports back (global_count / local_count ≈ #active servers) rather than a static constant; floor it at 1; cap the per-server slice at the global limit. Blast radius: one customer, until the next sync (≤ the sync interval); detection: per-customer over/under-admission gauges (already in the monitoring list).
  • Server restart loses local counts → brief over-admission. A server that restarts re-starts its local_count at 0; until its next sync it hasn't reported its in-flight traffic, so the global count is slightly low and everyone's slice is slightly generous. Bounded by (traffic on that server) × (sync interval) — tiny. Defence: on startup, sync immediately (don't wait a full interval); detection: a spike in a customer's sync-reported-delta after a deploy.
  • Sync thread stalls / GC pause → stale budget held too long. If a server's background thread is blocked (GC, a slow Redis call), it keeps using its last local_limit — which is fine for under-admission but causes over-admission proportional to the stall length. Defence: a watchdog — if the last successful sync is older than 5× the interval, shrink the local limit toward the conservative fallback (treat “stale” like “degraded”); detection: sync-age gauge per server (alert at p99 > 5× interval).

Iteration 2: Adaptive Sync Frequency

Not all customers need the same sync frequency. A customer using 10% of their limit does not need frequent syncs. A customer at 90% of their limit needs tight synchronization to avoid over-allowance.

Adaptive Strategy

Utilization = global_count / global_limit

  Utilization < 50%:   sync every 2,000 ms (2 seconds)
    Customer is well under limit. Coarse-grained counting is fine.
    Error margin at 2s sync: up to ~4% over-allowance. Acceptable.

  Utilization 50-80%:  sync every 500 ms
    Getting close. Standard sync interval.
    Error margin: ~1% over-allowance.

  Utilization 80-95%:  sync every 100 ms
    Near the limit. Need tight control.
    Error margin: ~0.2% over-allowance.

  Utilization > 95%:   sync every 50 ms
    At the limit. Maximize accuracy.
    Error margin: ~0.1% over-allowance.

Redis Load with Adaptive Sync

Assume customer distribution:
    8,000 customers at <50% utilization:  sync every 2s
    1,500 customers at 50-80%:             sync every 500ms
    400 customers at 80-95%:               sync every 100ms
    100 customers at >95%:                 sync every 50ms

  Redis ops per second per server:
    8,000 × (2 ops / 2s)    = 8,000 ops/s
    1,500 × (2 ops / 0.5s)  = 6,000 ops/s
    400 × (2 ops / 0.1s)    = 8,000 ops/s
    100 × (2 ops / 0.05s)   = 4,000 ops/s
    Total per server:         26,000 ops/s

  Wait — that's per server. With 10,000 servers?
  No! Each server only tracks customers it has seen recently.
  A server typically sees traffic from ~100-500 customers, not all 10,000.

  Revised calculation per server (tracking 300 customers):
    ~240 at <50%:  240 × 1 = 240 ops/s
    ~45 at 50-80%: 45 × 4  = 180 ops/s
    ~12 at 80-95%: 12 × 20 = 240 ops/s
    ~3 at >95%:    3 × 40  = 120 ops/s
    Total per server: ~780 ops/s

  Total across all servers: 10,000 × 780 = 7,800,000 ops/s.

  Hmm, that's too many. We need Redis sharding.

  With a Redis cluster of 50 shards (each handling 200K ops/s):
    7.8M / 50 = 156,000 ops/s per shard. Feasible.

Important correction: Adaptive sync increases accuracy for high-utilization customers but increases Redis load. The 50-shard Redis cluster costs more than the single instance in Iteration 1, but far less than the 333 instances of the naive approach. The tradeoff is justified because high-utilization customers (those near their limit) are the ones who care most about accuracy.

Batching Reduces Redis Load Further

Instead of one Redis call per customer per sync:
  Batch all customers into a single Redis pipeline per sync.

  Server syncs 300 customers in one pipeline:
    300 INCRBY + 300 GET = 600 commands in 1 pipeline.
    Redis processes pipelines much faster than individual commands:
    1 pipeline ≈ 1 network round trip, regardless of command count.

  Effective Redis ops/sec with pipelining:
    10,000 servers × ~5 pipeline calls/sec = 50,000 pipeline calls/sec.
    Each pipeline has ~100 commands.

    Redis throughput with pipelining: 500K-1M commands/sec per instance.
    Need: 50,000 × 100 = 5,000,000 commands/sec.
    With 10 Redis instances: 500K each. Feasible with pipelining.

  Down from 333 instances (naive) to 10 instances (batched adaptive sync).

Sharding the counter space (cite, don't restate)

The 10-shard cluster shards keys by hash(scope_key) — hash sharding, with consistent hashing so adding a shard moves only ~1/N of keys (detail: consistent-hashing.html). Hash, not range: there's no range-scan query here, only point increments, and uniform load is the priority. A single huge tenant is still one key on one shard — a hot key — but the local-first layer (Iteration 1) already absorbed the per-request rate; the shard now sees only ~2 ops per server per sync interval for that key, not 1 M/sec. If that ever became hot (it won't at these numbers), the move is to shard the customer's counter into k sub-counters (scope_key + ":" + rand(k)) and sum on read — the standard hot-key-with-salt move.

What breaks (C5; cite failure-modes.html)

  • A customer oscillating around the utilization band thresholds → sync-rate flapping. A customer hovering at exactly 80% utilization flips between 500 ms and 100 ms sync — 5× the backend ops, oscillating. Defence: hysteresis on the band edges (enter the faster tier at 80%, leave it only below 75%) and a minimum dwell time per tier (don't change the interval more than once per few seconds). Blast radius: one customer's worth of extra backend ops; detection: sync-interval-change rate per customer.
  • A burst pushes thousands of customers into the fast tier simultaneously → backend stampede. A platform-wide traffic spike (a popular event) raises everyone's utilization at once → the adaptive logic switches a large fraction of scopes to 50 ms sync → backend ops jump ~10×. Defence: a fleet-wide ceiling on aggregate sync rate — if the regional store's load is near its budget, cap the fastest tier for everyone (accept ~4% error fleet-wide rather than overloading the store); this is exactly the DEGRADED state in Iteration 3. Detection: regional-store ops/sec vs its budget.
  • Pipeline batches a slow customer behind a fast one → head-of-line in the sync. Batching all of a server's scopes into one pipeline means a near-limit customer's reconcile waits for the whole pipeline to complete; a slow shard delays the urgent update. Defence: split the pipeline by shard (one per Redis shard, sent in parallel) so a slow shard only delays its own keys; and put near-limit scopes in their own small high-priority pipeline. Detection: per-shard sync latency.

Iteration 3: Graceful Degradation and Failover

What happens when Redis is unreachable? The system must degrade gracefully rather than fail catastrophically.

Failover Strategy

Redis health states and corresponding behavior:

State 1: HEALTHY (normal operation)
  - Sync with Redis at adaptive intervals.
  - Full accuracy (<1% error for near-limit customers).

State 2: DEGRADED (Redis responding slowly, >10ms latency)
  - Increase sync interval to 2 seconds for ALL customers.
  - Reduce Redis load. Accept up to ~4% error for all customers.
  - Alert the on-call team.

State 3: UNREACHABLE (Redis connection failed)
  - Fall back to LOCAL-ONLY rate limiting.
  - Each server enforces: local_limit = global_limit / estimated_num_servers.

  Estimated servers: use a reasonable default, e.g., 1,000 (not 10,000).
    Why 1,000 and not 10,000? Because traffic isn't uniform.
    A customer's requests hit ~100-500 servers, not all 10,000.
    Using 1,000 is a conservative middle ground.

    Customer with 1M limit: local_limit = 1M / 1,000 = 1,000 req/sec per server.

    If customer hits 200 servers: effective limit = 200 × 1,000 = 200,000.
    That's 20% of their actual limit — under-allowing by 80%.

    Not ideal, but the customer gets SOME service.
    Better than Option A (block all) or Option B (allow all).

State 4: RECOVERING (Redis back online)
  - Gradually resume syncing (one customer group at a time).
  - Don't slam Redis with all 10,000 servers syncing simultaneously.
  - Stagger reconnection over 30 seconds.

The “Allow or Block?” Decision

When rate limiting infrastructure fails, you face a choice:

  "Fail open" (allow all traffic):
    + Customers experience no disruption.
    - An abusive customer could overload your backend.
    - You are charging for rate limits you're not enforcing.
    → Use for: low-risk APIs, internal services.

  "Fail closed" (block all traffic):
    + No risk of backend overload.
    - ALL customers are down. This is a self-inflicted outage.
    → Use for: almost never. This is worse than the problem it prevents.

  "Fail conservative" (local-only with conservative limits):
    + Customers get partial service.
    + Backend is protected (conservative limits prevent overload).
    - Customers may see more rejections than expected.
    → Use for: most production API platforms. This is the right default.

Monitoring and Alerting

Key metrics to monitor:

  1. Sync latency (p50, p99):
     Normal: <1 ms. Alert if p99 > 5 ms.

  2. Sync success rate:
     Normal: >99.9%. Alert if <99%.

  3. Over-allowance rate per customer:
     Track: (actual requests allowed - limit) / limit.
     Alert if any customer exceeds 5% over-allowance.

  4. Under-allowance rate per customer:
     Track: (limit - actual requests allowed) / limit,
     when customer is actively sending at their limit.
     Alert if any customer gets <90% of their paid limit.

  5. Redis memory usage:
     Each key ≈ 100 bytes. Keys per second window × TTL of 3s:
     10,000 customers × 3 windows = 30,000 keys × 100B = 3 MB.
     Redis memory usage for rate limiting is negligible.

Degradation contract — fail OPEN, with a cap

The headline guarantee: a rate-limiter outage admits traffic; it never denies it. Concretely — (1) regional counter slow (> 10 ms or timing out) → DEGRADED: every scope syncs at the slow interval (2 s), accepting ~4% over-admission fleet-wide, alert on-call; the hot path is unaffected (it never touched the counter). (2) regional counter unreachable → LOCAL-ONLY: each server enforces local_limit = global_limit / estimated_servers with a deliberately generous estimate (e.g., 1,000, not 10,000 — assume traffic clusters), so a customer keeps getting partial service rather than being throttled to nothing; the backend is still protected (the cap is conservative enough that even a customer hitting all 10,000 servers gets at most 10× their limit, not unbounded). (3) config-push down → keep the last-known limit catalog indefinitely; new tenants just don't get a limit until config flows. (4) the limiter library itself faults (a bug, an exception in allow()) → the API server's middleware catches it and admits the request — a limiter bug must never 500 a customer's API call. (5) recovery → servers re-sync on a jittered schedule over ~30 s, not all at once (no thundering herd on the recovering counter). The one exception: a per-scope policy bit can mark a security scope (login attempts, password reset, signup) as fail-closed — for those, “can't check the counter” means “deny”, because the cost of over-admitting an attacker outweighs the cost of blocking a few legit logins. The invariant: the limiter degrades to staler / coarser / more-permissive — never to “the API is down”. Cite failure-modes#slow-dependency + #stampede.

Alternatives considered (C4) — multi-region

The platform spans regions; a customer's traffic hits multiple. Where does the global count live? Options:

  • Per-region quotas with periodic rebalance — chosen. Split a customer's global limit into per-region sub-limits (proportional to recent regional traffic); each region runs the local-first design above against its own regional counter (a regional Redis cluster); a slow background job (every few seconds to a minute) measures actual per-region usage and rebalances the split — a region using 10% of its quota donates headroom to a region at 95%. Properties: zero cross-region traffic on the hot path or the per-request sync; bounded over-admission = the sum of per-region over-admissions plus the rebalance lag. The standard approach. Worked: a 1 M req/sec customer with traffic 70% in us-east, 30% in eu-west → quotas 700 k / 300 k; if eu-west traffic doubles to 60% before the next rebalance, eu-west over-admits by up to (its real usage − its 300 k quota) for one rebalance interval — bounded, and the rebalance closes it.
  • One global CP counter (Spanner / a Raft group) the per-request path reads — rejected. Exact globally — but a cross-region read on every request is 60–200 ms (cross-region latency), violating the < 1 ms budget by two orders of magnitude, and the consensus group is a global SPOF. Exactness across regions isn't worth that; the soft SLO permits the per-region split's small slop.
  • One global eventually-consistent CRDT counter (a g-counter replicated cross-region) — rejected as the primary mechanism. A grow-only counter merges cleanly across regions with no coordination — appealing — but each region still needs the count fast on the reconcile path, so you're back to per-region replicas of it; and a pure g-counter never decreases, so windowing (resetting the count each second) needs a per-window counter, which is what the per-region quota scheme already is, minus the cross-region replication overhead. CRDTs are the right tool when regions must see each other's count; here they don't — they only need their own quota.

Final Design

Architecture

Request Path (Hot Path — No Network Call)

1. API server receives request from customer C.
2. Look up C in local rate limiter:
     local_counts[C] += 1
     if local_counts[C] > local_limits[C]:
         return HTTP 429 Too Many Requests
     else:
         process request normally
3. Total added latency: ~0.001 ms (in-memory hash map lookup).
   Well under the 1 ms budget.

Sync Path (Background — Async)

Every sync_interval (adaptive: 50ms to 2000ms):
  1. Batch all local counts into a Redis pipeline.
  2. Send pipeline: INCRBY + GET for each active customer.
  3. Receive global counts.
  4. Recalculate local_limits based on remaining budget and utilization.
  5. Reset local counts.
  6. Adjust sync_interval based on new utilization levels.

Sliding Window Counter Implementation

The local per-pod counter is a sliding-window counter — the O(1)-memory approximation that avoids the fixed-window boundary burst without storing every hit (the algorithm comparison is rate-limiting.html; token bucket is the interchangeable alternative, see the Naive-Design C4). Per-pod state is two integers per scope: the current 1-second window's count and the previous window's count. Scenario-specific worked numbers:

Fixed window problem:
  Window 1: [0:00, 1:00)  — Customer sends 900K requests at 0:59.
  Window 2: [1:00, 2:00)  — Customer sends 900K requests at 1:01.
  Both windows under 1M limit individually.
  But in the 2-second span [0:59, 1:01): 1.8M requests. Nearly 2× the limit!

Sliding window counter solution:
  At time T, calculate:
    count = (previous_window_count × overlap_fraction) + current_window_count

    overlap_fraction = 1 - (seconds_into_current_window / window_size)

  Example at time 1:15 (15 seconds into the current 60-second window):
    previous_window_count = 800,000
    current_window_count = 200,000
    overlap_fraction = 1 - (15/60) = 0.75

    estimated_count = (800,000 × 0.75) + 200,000 = 800,000

    If limit is 1,000,000: 800,000 < 1,000,000 → allow.

  This is an approximation but eliminates the boundary burst problem.
  Error is bounded by the distribution of requests within the previous window.

C9 — schema / API sketch

// --- Local per-pod state (in-RAM, lock-free; request threads use atomics) ---
struct ScopeCounter {                         // one per active scope_key on this pod
  curr_window_start: i64;  curr_count: u32;   // sliding-window-counter form ...
  prev_count: u32;                            // ... estimated = prev*overlap + curr
  // (token-bucket alt: tokens: f32; last_refill: i64;  refill = rate*(now-last_refill))
  local_limit: u32;                           // budget until next sync (re-derived each reconcile)
  reported_delta: u32;                        // allowed-since-last-sync, to report
}
map[scope_key -> ScopeCounter]  local;        // ~100-500 entries per pod; LRU-evict cold scopes
map[scope_key -> {limit_per_sec, window, policy}]  limits;   // synced from the control plane

// --- Regional counter store (Redis Cluster, sharded by hash(scope_key)) ---
// Key shape:  rl:{scope_key}:{window_id}     window_id = floor(epoch_sec / window)
// Lua: atomic report-delta + read-global  (one round trip per scope per sync, batched in a pipeline):
//   local c = redis.call('INCRBY', KEYS[1], ARGV[1])      -- ARGV[1] = local reported_delta
//   if c == tonumber(ARGV[1]) then redis.call('EXPIRE', KEYS[1], ARGV[2]) end   -- ARGV[2] = TTL (2-3 windows)
//   return c                                              -- the approximate global count

// --- Control-plane store (limit catalog; tiny, relational; ~tens of KB) ---
CREATE TABLE limits ( scope_type text, scope_id text, limit_per_sec bigint, window_sec int,
                      fail_policy text DEFAULT 'open',   -- 'open' | 'closed' (security scopes)
                      parent_scope text,                 -- hierarchy: a key's parent is its tenant
                      version bigint, PRIMARY KEY (scope_type, scope_id) );

// --- The rate-limit RESPONSE (cite api-design-patterns#rate-limit-response -- do not restate) ---
// On DENY:  HTTP 429
//   Retry-After: (seconds until the scope's window resets / next token)     -- mandatory
//   X-RateLimit-Limit: (limit_per_sec for the binding scope)
//   X-RateLimit-Remaining: max(0, limit - estimated_count)                  -- from peek(), local read
//   X-RateLimit-Reset: (epoch when the window resets)
//   body: RFC 7807 Problem Details { type, title, status:429, detail, retry_after_seconds, trace_id }
// On ALLOW: the same X-RateLimit-* headers (so well-behaved clients self-throttle before they hit 429).

Headers and body shape: api-design-patterns#rate-limit-response (and the error model for the 7807 body). Note peek() reads local state — the X-RateLimit-* numbers are this pod's view, approximate, never a round trip.

C10 — concrete technology, justified

  • Regional counter store: Redis Cluster + Lua — chosen. An atomic INCRBY+EXPIRE+read in one server-side script, ~150 k–200 k ops/sec/node, sharded by key, replica for failover, microsecond-class ops, ubiquitous ops experience. The reconcile traffic (~50 k pipeline calls/sec, ~10 nodes) fits comfortably. Versus: a CRDT g-counter (Riak / Redis Enterprise CRDTs) — merges across regions without coordination, but we don't need cross-region merge (per-region quotas), and a g-counter doesn't decrease, so windowing reintroduces per-window counters anyway; net complexity for no benefit here. DynamoDB atomic counters (UpdateItem ADD) — managed, durable, scales — but ~5–10 ms per write (vs <1 ms Redis), partition-key throughput caps make a hot tenant's key a hot partition, and we don't want durability we're paying for (the count is ephemeral, rebuilt every window). A custom in-memory sharded counter service — could be leaner, but it's a new stateful fleet to operate for a job Redis already does; build it only if Redis's per-key hot-shard ceiling is genuinely hit (it isn't). Envoy's global rate limiter / lyft​/ratelimit — this is essentially that design (a gRPC ratelimit service in front of a Redis backend); we'd use it off-the-shelf in practice; the from-scratch derivation here is to understand its knobs (descriptor keys = our scope keys; the local cache = its “local rate limit” filter; the shadow mode = our DEGRADED state).
  • Local cache: in-process library, not a sidecar — chosen. The hot-path check must be a function call (~1 µs, no IPC); a sidecar adds a localhost round trip (~50–100 µs) × 50 M/sec — tolerable latency-wise but a needless CPU and complexity tax for state that's tiny and pod-local. A sidecar is the right call only when the limiter must be polyglot-shared across many runtimes that can't link the same library — then accept the localhost hop. (Envoy's design splits it: the local rate-limit filter is in-process in the proxy; the global one calls the ratelimit service — mirroring our local-first split.)
  • Control-plane store: a small relational DB (Postgres) — chosen. The limit catalog is ~tens of KB, changes rarely, has a hierarchy (a key's parent is its tenant) the validation walks — relational fits; pushed to pods via a config-distribution stream. Not etcd (the catalog isn't a coordination problem), not in Redis (it's config, not a hot counter).

Operations (C6)

  • Metrics (SLIs → NFRs). Hot-path added latency p50/p99 (→ the <1 ms NFR — should be ~1 µs; if it's not, you've accidentally put a network call on the path); per-customer over-admission rate (allowed − limit)/limit and under-admission rate (limit − allowed)/limit when sending at the limit (→ the ≤5% accuracy NFR — alert per-customer breach); sync success rate + sync age p99 per server (→ staleness — alert at p99 > 5× interval); regional-counter ops/sec vs its budget (→ backend QPS NFR + the stampede detector); active-scope count per server (memory bound on the local map); fraction of fleet in DEGRADED / LOCAL-ONLY (→ the fail-open path is engaged); config-version histogram across servers (→ limit-freshness NFR); 429 rate per scope + the Retry-After-honoured fraction (retry-storm detector).
  • Rollout. Two surfaces: (1) the limiter binary/library — canary a few API servers, watch hot-path latency + accuracy + ejection-to-LOCAL-ONLY rate, ramp; rollback is a redeploy. (2) a limit change — set_limit is validated (positive, hierarchy-consistent — a key's sub-limit ≤ its tenant's total), versioned, pushed; a tenant-facing limit decrease rolls out behind a soft window (the new lower limit logs-but-doesn't-enforce for a grace period, then enforces) so a misconfigured decrease doesn't instantly throttle a customer; a one-command rollback re-publishes the prior version.
  • Migration. From a single-Redis per-request limiter (the Naive Design) to the local-first design without an incident: ship the new library with the local layer behind a flag and the regional sync pointed at the same Redis (now used for reconcile, not per-request); turn on the local layer for a subset of API servers — they stop hammering Redis per request and start syncing; watch Redis ops/sec drop and accuracy hold; ramp server-by-server; once the whole fleet is local-first, swap the single Redis for the sharded cluster (consistent-hash add-a-shard moves only 1/N of keys). A bad batch of servers flips the flag back — they revert to per-request Redis, which still works (just hot). The migration is “flag per server, then re-shard the backend” — no big-bang.
  • Headroom. The regional counter cluster carries headroom for the worst case — a platform-wide burst pushing a large fraction of scopes to the 50 ms tier at once (~10× baseline reconcile); plus rolling-deploy churn (1/3 of API servers re-syncing on startup); plus 2× organic fleet growth. The local map has a hard cap (LRU-evict to a conservative default) so a server seeing an unusually large customer fan-in can't OOM. Reason stated inline at each: burst, deploy churn, growth.

  Naive (central INCR per request):    333 Redis nodes x ~$500/mo  = ~$166,500/mo
  Iteration 1 (1 Redis + replica):     2 nodes x ~$500/mo          = ~$1,000/mo
  Iteration 2 (10-shard cluster, HA):  ~20 nodes x ~$500/mo        = ~$10,000/mo  (adaptive sync trades $ for accuracy where it matters)
  Final (~10 shards + replicas + a small Postgres for the catalog): ~$5,000-10,000/mo

Dominant cost shape (cite cost-shapes.html): this is a cache-heavy / coordination-light primitive -- the backend is a small fleet of
  in-memory counters, not a database; its job is to make the EXPENSIVE thing (the API backend it protects) cheaper by keeping abusive
  load off it. Don't size it by its own bill; size it by the backend overload it prevents -- an un-rate-limited API needs the backend
  provisioned for "any customer might send 10x" instead of "limits hold", which is the real money.

Cost-vs-accuracy knob: the sync interval / the adaptive bands. Tighter sync = more backend ops = more $ = less over-admission. We spend
  the ops on the few customers near their limit (50ms sync) and almost nothing on the many well under it (2s sync) -- the adaptive bands
  ARE the cost/accuracy dial. There is no latency knob -- the hot path is already a ~1us local lookup.

Real-system comparison

This is essentially Envoy's rate-limiting subsystem re-derived: the in-process local rate limit filter (a token bucket per descriptor, no network) is our local-first layer; the global rate limit filter calling the lyft​/ratelimit gRPC service backed by Redis is our reconcile path; descriptor keys are our hierarchical scope keys; shadow mode is our DEGRADED state. Cloudflare's rate limiter does the same shape at the edge (per-POP local count + eventual aggregation). Stripe's published design (a token-bucket per key in Redis with a Lua script, plus a local in-process layer for the highest-volume endpoints) is the same local-first/central-reconcile split. The from-scratch derivation here exists to understand why each of those has a local layer and an eventual-sync layer and a fail-open story — so when Envoy's accuracy looks off you know to check the local-cache TTL and the descriptor granularity, and when Redis is hot you know it's a hot-key-per-tenant question, not a “need more Redis” one.

Anticipated pushbacks

Stress tests against the derivation above — depth rehearsal, not interview rehearsal.

What if offered load were 100× — 5 billion req/sec? (scale flip)

The hot path is unbothered — it's a local lookup, it scales with the API fleet, which is sized for the API anyway. The reconcile load scales with #servers × #active-scopes ÷ sync_interval, not with request rate — 100× the requests on roughly the same fleet means roughly the same reconcile traffic. What does grow: the API server count (say 1 M servers) → 1 M × 10 shards = 10⁶ connections to the regional cluster, which is the subsetting threshold — now you want the hierarchical reconcile (servers → per-rack aggregator → regional counter) from the Iteration 1 C4, so the regional cluster sees ~thousands of aggregator connections, not millions. And per-key hot-shard pressure on a single mega-tenant: shard that tenant's counter into k sub-counters with a random suffix, sum on read — standard hot-key-with-salt. Nothing structural breaks; two known levers (hierarchical reconcile, counter salting) come off the shelf.

Why approximate at all — why not exact? (correctness)

Exact means one of two things, both disqualified at this scale: (1) sliding-window-log — store every hit's timestamp, evict, count — which is O(window×rate) memory per key (a million timestamps for a 1 M req/sec customer over a 1-second window) and turns the per-request check from O(1) into O(log n); (2) a global exact count read per request — a round trip on every call, which is both the latency budget blown by 100× and a bandwidth wall on the store. The < 5% soft SLO exists precisely so we don't have to pay either. And approximate in the right direction: the design biases toward slight over-admission (let a few extra through) over under-admission (reject paying customers' legit traffic) — because the latter is the worse failure for an API platform.

A tenant disputes their bill — what if it must be exact for billing? (exact-for-money)

Then it's no longer a rate limiter — it's a usage meter, a different system with a different durability class. Rate limiting answers “allow this request right now?” in microseconds, approximately; billing answers “how many requests did tenant X make in May?” durably, exactly, reconcilably — that's an append-only event log (every request emits a metered event), aggregated offline, with idempotent dedup and a correction path. They share nothing but the input: don't make the rate limiter authoritative for money. The limiter can emit a per-request counter as a side-effect that feeds the meter, but the meter's source of truth is the durable event stream, not the limiter's ephemeral counters. (This is the boundary with #42's metering side and a metrics/billing pipeline generally.)

The regional counter cluster dies right now — fail open or closed? (failure)

Open, with a cap — per the degradation contract. Concretely: each API server immediately falls back to LOCAL-ONLY: local_limit = global_limit / 1,000 (a deliberately generous active-server estimate — assume traffic clusters, don't divide by 10,000), so a customer keeps getting partial service rather than being throttled to ~2% of their limit; the backend is still protected because even a customer hitting all 10,000 servers gets at most ~10× their limit (10,000 × limit/1,000), not unbounded. Servers detect the outage by sync failures, switch within one failed sync, and on recovery re-sync on a jittered 30-second ramp (no herd on the recovering cluster). The single exception: scopes marked fail_policy = closed (login/signup/password-reset abuse limits) deny on “can't check” — for those, over-admitting an attacker is the worse outcome. What you do not do: block all traffic (a self-inflicted total outage, strictly worse than un-enforced limits for a few minutes).

Where's the cost, and which knob moves it? (cost)

This component is ~$5–10 k/mo (a ~10-shard Redis cluster with replicas + a tiny Postgres for the limit catalog) — cheap; the limiter rides inside API servers that exist for the API regardless, at <1% of their CPU. The relevant lever is the adaptive sync bands: tighter sync → more backend ops → more $ → less over-admission — and the design already spends those ops only on the few customers near their limit (50 ms) and almost none on the many well under it (2 s). The real money is elsewhere: an API without working rate limits has to be provisioned for “any customer might 10× us,” so the limiter's job is to make the expensive backend cheaper by holding load flat — size the limiter by the overload it prevents, not its own bill. Cite cost-shapes#cache-heavy.

How do you migrate from a single-Redis limiter to the local-first design? (migration)

Flag per server, then re-shard the backend — no big-bang. Ship the new library with the local layer behind a per-server flag, pointing its reconcile at the existing single Redis (now doing INCRBY-deltas, not per-request INCRs). Flip the flag on a handful of servers: they stop hitting Redis per request and start syncing ≈ twice per interval — watch Redis ops/sec fall and per-customer accuracy hold (the over/under-admission gauges). Ramp server-by-server; a bad batch flips back to per-request Redis, which still works (hot, but correct). Once the whole fleet is local-first, swap the single Redis for the 10-shard cluster — consistent-hash add-a-shard moves only ~1/N of keys, and a wrong global count for a moment during the cutover just means slightly stale budgets, self-healing within a window. The control-plane limit catalog migrates separately (it's tiny config).

A tenant says they're being throttled wrongly — where do you look? (monitoring/debug)

Decision tree off the SLI dashboard: (1) Check their under-admission gauge — if they're getting < 90% of their paid limit while sending at it, the estimated_active_servers is too high (each server's slice is too small) → check the global_count/local_count ratio the reconcile uses, and whether their traffic is hitting an unusual number of servers (an LB change?). (2) Check which scope is binding — the 429's X-RateLimit-Limit header names it; a tenant fine on their global limit can be hitting a per-key sub-limit, a per-IP limit (shared NAT?), or a per-endpoint cap — the hierarchy means “throttled” isn't always the obvious scope. (3) Check the fleet's DEGRADED/LOCAL-ONLY fraction — if the regional counter is down, everyone is on the conservative fallback; the fix is restoring the counter, not touching this tenant. (4) Check the config-version histogram — if a limit change is half-propagated, some servers enforce the old (lower?) limit. (5) Check their sync-age — a wedged sync thread on the servers they hit holds a stale (possibly too-tight) budget. Each branch maps to a What-Breaks entry and a runbook step; the over/under-admission and per-scope-429 instruments exist precisely so this is a dashboard read, not archaeology.

A burst right at the window boundary doubles the effective rate — how do you handle it? (boundary burst)

This is the fixed-window flaw the Final Design's sliding-window-counter fixes: 900 k requests at 0:59.5 and 900 k at 1:00.5 are each under a 1 M/sec limit per fixed window but ~1.8 M over the 1-second span straddling the edge. The sliding-window counter estimates count = prev_window × overlap_fraction + curr_window — at 1:00.5, overlap_fraction = 0.5, so the estimate is 900k×0.5 + 900k = 1.35M > 1 M → deny — the boundary burst is bounded to < 2× (and in practice well under, since real traffic isn't a single spike). If even < 2× is too loose, switch the per-key state to a token bucket (refill at the rate, capacity = the explicit burst allowance you actually want) — it smooths to a true rate with no boundary at all; it's the interchangeable O(1)-per-key alternative (Naive-Design C4). Note the approximation here is purely local — it's orthogonal to the local-first/reconcile architecture, which handles the distribution problem, not the windowing one.

Key Takeaways

  • A rate limiter is a hit counter plus a gate — LC #362 and LC #359, same primitive at different granularities. A deque of timestamped buckets (fine), a (count, window_start) pair (coarse), and a (tokens, last_refill) token bucket (rate-smoothed) are three views of the same O(1)-per-key state. Production keeps the O(1) form (a count + a timestamp), not the deque — the exact version (sliding-window-log) is O(window×rate) memory per key.
  • Pure centralized rate limiting does not scale — it's a hot key plus an RTT tax. Checking a central store on every request at 50M req/sec is 333 Redis nodes and 0.5 ms on every API call; one big customer's limit is a single key one shard owns. Cost and latency both unacceptable.
  • Pure local rate limiting is too inaccurate. Without coordination, each server must guess how many others serve the same customer; traffic clusters on ~100–500 of 10,000 servers, so naive limit/N under-admits massively (customers get ~2% of their limit).
  • Local-first with async reconcile is the middle ground. Count in-process (the LC kernel, lifted — zero network on the hot path, ~1 µs), sync a batched delta to a sharded regional counter ~twice per interval. Backend load drops ~1,250×; over-admission is bounded by the sync interval; the hot path is a lock-free hash lookup.
  • Adaptive sync is the cost/accuracy dial. Sync fast (50 ms) for the few customers near their limit (accuracy matters); slow (2 s) for the many well under it (it doesn't). Spends backend ops where they buy accuracy; there is no latency knob — the hot path is already a local lookup.
  • Fail OPEN, with a cap — not fail closed, not fail unbounded. A limiter outage admits traffic on a conservative local fallback (a generous active-server estimate so customers get partial service; conservative enough that the backend is still protected). A limiter-library exception admits the request — never 500s a customer's API call. Security scopes (login/signup) can opt into fail-closed per-scope. Blocking all traffic on a limiter outage is a self-inflicted outage, strictly worse than the problem.
  • Don't make the rate limiter authoritative for money. Rate limiting is microsecond, approximate, ephemeral; billing is durable, exact, reconcilable — a separate metering pipeline over an append-only event log. They share only the input.
