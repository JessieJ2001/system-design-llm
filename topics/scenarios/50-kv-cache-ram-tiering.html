Designing a Tiered KV Cache for LLM Inference

Chat + RAG product at 3k req/sec, 30k-token average context, 70% prefix-reuse, Llama-3 70B. We start from #34's HBM-only PagedAttention baseline and iterate to a hierarchical KV cache pool spanning HBM → host RAM → NVMe → RDMA-shared across nodes, with prefill/decode disaggregation.

NALSD Walkthrough • ~60 min read • Deep specialization of #34 LLM inference serving

Relation to #34 LLM inference serving. #34 derives the model-serving baseline: tensor parallelism, PagedAttention, continuous batching, GQA, INT8 weights, speculative decoding — all assuming the KV cache fits in HBM and is private per request. #50 starts from where #34 ends and asks the next question: what if the workload makes HBM-only KV cache uneconomical? Long contexts + high prefix-reuse (multi-turn chat, RAG prepends, agent loops) collapse the HBM-only assumption: the same 20k-token system+RAG prefix appears in thousands of concurrent requests, and HBM is too scarce and too expensive to store it once per request. The arc below adds cross-request prefix sharing, HBM→RAM→NVMe tiering, RDMA-shared KV across nodes, and prefill/decode disaggregation — none of which are addressed in #34.

LC kernel. Implement Trie / Prefix Tree (LC #208) — with LRU Cache (LC #146) overlaid. Textbook structure: each trie node maps one character to a child node + (optional) terminal value; insert and lookup are O(L) where L is key length. Property that breaks at scale: in our world the keys are token-sequence prefixes (length up to 100k tokens, variable per request), the values are KV cache slabs (~MB to GB per prefix), the trie must support eviction (textbook trie grows unbounded) and the trie spans physical tiers (HBM / RAM / NVMe) where eviction has tier-asymmetric cost — evicting from HBM is "free" (we can recompute by streaming from RAM), but evicting from RAM means a full prefill recomputation on next access (seconds of compute). The production cousin is RadixAttention (SGLang / vLLM prefix cache) extended with hierarchical tier-aware eviction (Mooncake / LMCache shape). The C8 table at S6 enumerates the property swaps.

S0. Scenario & Requirements

A general-purpose chat assistant product at the scale of mid-tier ChatGPT / Claude.ai. Each request carries (a) a multi-turn chat history (~5k tokens), (b) a RAG context retrieved from a knowledge base (~20k tokens of document chunks prepended by a retriever), and (c) the user's new turn (~500 tokens). The model generates ~1k output tokens. The dominant cost is GPU-hour-per-million-output-tokens; the dominant scaling constraint is HBM capacity for the active KV cache across all in-flight requests. Real production cousins: OpenAI, Anthropic, Mistral le Chat, Moonshot AI Kimi, Google Bard/Gemini in chat mode.

System class

Class: algorithmic-ML system class — user-facing inference serving with a deep-learning model as the load-bearing kernel. Cite functional-requirements.html#class-algorithmic-ml. The class fixes the FR vocabulary (operation = "generate N tokens given prompt"; semantics include model version, sampling parameters) and pulls in the algorithmic-ML NFR additions: model freshness, training/serving parity, drift — not load-bearing in this scenario because the model is static at serving time, but per-class additions like per-step deterministic latency (TBT, time-between-tokens), tail-latency under load, and memory-bandwidth-bound decode are very much load-bearing.

Functional requirements (cite functional-requirements.html)

  • Generate response. Given {messages: [{role, content}], model, sampling_params}, return a streamed completion token-by-token via SSE. Honors stop sequences and max_tokens.
  • RAG prepend. An upstream retriever provides 0–50 document chunks (each ~500 tokens); the chat API prepends them as a system message before the user turn. The retriever is out-of-scope (separate service); the serving system only sees the prepared prompt.
  • Multi-turn continuation. Each turn is a new request that includes the entire prior conversation. The serving system has no session state — each request is self-contained at the API level. (We will derive that the serving system gains an implicit per-conversation cache anyway via prefix sharing.)
  • Per-request determinism. Same prompt + same model version + same seed → same output (within numeric precision). Reproducibility is a product requirement for evals + audit.

Out of scope (named): retrieval pipeline (separate service); fine-tuning (offline); content moderation pre-filter (separate service); tool-use / function-call dispatch (handled by an orchestrator one layer up); embeddings serving (separate); image / multimodal inputs (text-only here).

Flagged assumption. The Llama-3 70B model is INT8-quantized at serving time (cite #34 Iter 4); per-token KV cache is also INT8. If we used FP16 KV (which preserves quality marginally better for very long contexts), every memory-related number below doubles; the iteration arc is unchanged in shape but the cliff at each tier hits 2× earlier.

The fundamental tension. Two compounding pressures break the HBM-only baseline. First, scale: 15,000 concurrent requests × 31k tokens average × 160 KB/token = ~75 TB of active KV cache. Per H100 with TP=8 and ~9 GB of weights per GPU, each GPU has ~70 GB free; per DGX node, ~560 GB. To hold 75 TB in HBM would require ~135 DGX nodes (~$3 M/mo) just for KV memory — before any compute. Second, redundancy: 70% of those KV bytes are shared across requests (system prompt + RAG context). HBM-only storage means we're storing the same prefix N times for N concurrent requests with that prefix. The whole arc below collapses both pressures: share common prefixes (prefix tree); tier what doesn't fit in HBM (RAM, NVMe); pool across nodes (RDMA); disaggregate prefill from decode so each tier is sized to its dominant workload. Mooncake (Kimi), SGLang RadixAttention, and LMCache are the production systems that have walked this same path.

S1. Core Entities (cite entities-and-data.html)

Six entities. PrefixNode earns first-class status because it's the trie's load-bearing object and its eviction policy is the system's hardest design problem. KVSlab is the unit of HBM/RAM/NVMe storage (PagedAttention pages are 16 tokens each; a slab is one page's worth of K and V across all layers). Tier is meta-information (which physical location holds this slab) but is queried on every decode step's attention call.

ID scheme (cite identifier-schemes.html#hash-of-content): hash-of-content for prefix_node_id — the node's identity IS the token-sequence prefix from root. Specifically, we use a rolling-hash chain (Rabin-Karp style) over the token sequence so a prefix of length L and L+1 share L of the hash steps. This makes prefix-tree lookup O(L) hash operations (one per token) and allows partial prefix matching: a longer key falls back to the longest existing prefix. UUIDv7 for request_id because we need cross-system traceability and time-sortability (debug log timestamps) but the request identity isn't content-derived. Rejected: Snowflake for prefix_node_id (worker-skew would produce non-deterministic ids for the same content, breaking sharing); UUIDv4 for request_id (no time-sortability hurts debug log scanning).

S4. Naive Architecture — baseline from #34's final state

The starting point inherits #34's full optimization stack: PagedAttention page size 16 tokens, continuous batching, INT8 weights, INT8 KV cache, GQA, FlashAttention, TP=8, no speculative decoding (orthogonal to KV-cache tiering — we keep it out for clarity). Critically, the inherited starting point has no cross-request KV sharing: every request that arrives prefills its full 30k-token prefix from scratch, allocates its own KV pages in HBM, and evicts them on request completion. Many requests share the same system prompt + RAG context; the naive design recomputes their KV every time.

Why it's broken (per failing NFR)

  • Prefill cost waste. Throughput numbers technically meet target on paper, but ~70% of the prefill compute is recomputing identical shared prefixes — pure waste. With 300 prefill-equivalent nodes at $32/hr, ~$5M/mo is spent on recomputation. Failing NFR: cost-per-million-output-tokens budget at scale (becomes prohibitive once the long-context tail grows).
  • TTFT tail. 30k-token prefill at avg 100 ms is fine for p50, but the long-context tail (100k tokens) takes ~3.3 sec → fails p99 TTFT target (3.0 sec). The cause: every request pays the full prefill cost regardless of shared content.
  • Context-length ceiling. At 100k input tokens, per-request KV is 2 GB/GPU → only 35 concurrent requests per GPU before HBM saturates. The naive design's request density collapses as context grows.
  • RAM and NVMe wasted. 2 TB host RAM and 16 TB NVMe per node sit unused for the KV problem. We are leaving 90% of the node's storage hierarchy on the table.
  • Cache-pool fundamentals absent. The system cannot benefit from prefix sharing within a single node (let alone across nodes); identical RAG chunks are reprocessed on every retrieval.

The Naive design's failures cluster around one missed opportunity: the workload is highly prefix-reusable, and we are pretending it isn't. Every iteration below introduces a way to exploit prefix reuse at a deeper layer (within-node share → node-local hierarchy → cross-node pool → specialized roles → persistent precompute).

Cost of Naive at NFR-target load: ~$6.9 M/mo for ~300 hybrid DGX H100 nodes; $/M-output-tokens ≈ $0.89, technically under the $3 target. The waste: ~70% of prefill compute is recomputing identical prefixes. Iterations recover that waste; final $/M-output-tokens lands at ~$0.20 (Iter 4) — ~4.5× reduction, and unlocks the 100k-context tail that Naive cannot serve.

Iteration 0 — Per-node prefix-tree cache in HBM (RadixAttention)

System after this iteration

Legend. Blue boxes = HBM (active KV); purple-tinted = unused tiers (until Iter 1). Accent = new/changed this iteration (routing-by-prefix, per-node radix tree, shared prefix KV pool in HBM).

Bottleneck

Naive recomputes the prefix for every request that arrives. With prefix_reuse_by_tokens = 0.70 from the workload, that means ~70% of every 2 × params × tokens prefill FLOP is doing identical work to a prefill computed seconds earlier. Failing NFR: cost-per-million-output-tokens — we're paying ~$5M/mo in unnecessary prefill compute (70% of the ~$7M Naive fleet cost). Classified as bottlenecks#compute-bound — the prefill path is compute-bound (we are GPU-limited on the matmul, not memory-limited), and the canonical first move per bottleneck-to-decisions is caching — specifically, caching the intermediate state (KV cache) of the shared prefix portion so we don't recompute it.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#compute-bound on the read/share dimension: (a) introduce a content-addressable cache of intermediate results (KV slabs keyed by token-sequence prefix); (b) route requests with cache-affinity (so the same prefix doesn't get re-prefilled on a different node); (c) manage cache eviction with reference-counting (a slab in use by an active decoder cannot be evicted) and LRU-on-eviction-eligible. Cite access-patterns#power-law — prefix popularity is Zipfian, so a small cache catches most of the gain.

Alternatives considered (C4)

  • Per-node radix tree (RadixAttention / SGLang shape) — chosen. A prefix tree where each internal node holds KV slab refs for a contiguous run of cached tokens. Variable-length keys, branch sharing for common heads, O(L) lookup where L is prefix length. Eviction by leaf-up LRU when ref_count reaches zero. Cite SGLang RadixAttention.
  • Flat hashmap, prefix-hash → KV blob — rejected. Doesn't compose for partial-prefix hits. If request A has prefix [a, b, c, d] and request B has prefix [a, b, c, e], the flat hashmap caches each as a separate full blob; the radix tree shares the [a,b,c] node and only diverges at d/e. ~2× cache memory waste vs radix.
  • LRU-only cache of full prompts — rejected. Each entry is huge (~5 GB at 30k tokens); hit rate is much lower than a prefix tree because "exact prompt match" is rare; ~10% hit rate vs ~70% prefix match.
  • Suffix array / longest-common-substring index — rejected. Overkill; our keys are prefixes, not arbitrary substrings; the radix tree is the right algorithmic match. Suffix arrays would let us share KV across non-prefix common runs (e.g., a sentence appearing in two different requests' middles), but the LLM's causal attention means non-prefix sharing is mathematically incorrect — token T's KV depends on tokens 0..T-1, not on what comes after.

The chosen move

Three coupled changes:

  • Per-node radix tree of token-sequence prefixes. Each node represents a contiguous run of tokens; each holds kv_slab_refs[] (one slab per layer per page) pointing into HBM. The tree is replicated across the 8 GPUs of the same DGX (each GPU holds its TP-slice of every cached slab). The tree fits in a few hundred MB of host RAM (just metadata + pointers); the slabs themselves live in HBM.
  • Cache-affinity routing. The routing layer extracts the first 1,024 tokens of each incoming request, hashes in 16-token chunks (matching PagedAttention page size), queries a global prefix-locality index (replicated Redis cluster, written by workers on cache install) for "which workers own a prefix matching the first K chunks?", and routes to the worker with the longest match. Fallback: round-robin to a warm-but-cold worker; the prefix is computed there and registered for future affinity.
  • Reference-counted eviction. Each prefix node's ref_count is incremented when a request acquires its slabs for decode, decremented on completion. Eviction policy: when HBM pressure exceeds threshold, walk the LRU list of ref_count=0 nodes (those not currently in use) and free them. Critically: an active decoder pins all its prefix's slabs, so eviction never breaks an in-flight request.

Why this addresses the bottleneck shape: the workload's 70% prefix-reuse is now exploitable — popular prefixes are computed once and reused thousands of times. The Zipfian distribution (top-100 prefixes carry 50% of reuse) means even a small cache (top-2000 prefixes per node) catches most of the win. Compute drops from "prefill every full request" to "prefill only the unique suffix," reducing prefill work proportional to the prefix-hit rate. This is the dominant cost win in the entire arc.

What breaks (C5; cite failure-modes.html)

  • Cold-start stampede on prefix install. A viral new prompt template (e.g., a feature launch produces a new system prompt) arrives at 100 req/sec into a fleet with no cached KV for it. All ~10 affinity-routed workers race to prefill the same prefix — ~10 wasted prefill bursts. Mitigation: single-flight on prefix install — first worker to start prefill registers an "in-flight" entry in the Redis prefix-locality index; other workers wait or back off and use the result. Cite single-flight.
  • Per-node cache cold on new worker. Adding capacity (rolling deploy or scale-out) means new nodes have empty caches; cache hit rate dips for the affinity window. Mitigation: staged ramp — new node receives 10% traffic for ~5 min while it builds cache from the popular-prefix list, then full traffic. Detection: per-node cache hit rate; alert if any node sits below 50% after warm-up window.
  • Reference-count bug → leak. Decode worker crashes mid-stream without decrementing ref_count → slab pinned forever. Mitigation: ref_counts are not authoritative — they're hints; an independent sweep enumerates active decoder requests and reconciles ref_counts against ground truth every 60s. Cite failure-modes#resource-leak.
  • Cache thrashing under high prefix diversity. A burst of users with very different prefixes (e.g., 1000 distinct RAG queries in a window) exceeds cache capacity; LRU evicts then re-inserts each prefix — cache hit rate collapses, prefill cost spikes back to Naive. Detection: cache hit rate falling <50%; eviction-then-reinstall same prefix rate >0. Mitigation: Iter 1 (RAM tier doubles effective cache size by ~50×) addresses this directly.
  • Routing-layer hot key. The prefix-locality index (Redis) gets read 3k QPS for lookup, plus writes on install/eviction. A single popular prefix's metadata gets ~500 reads/sec on its index entry. Mitigation: Redis cluster with prefix-hash sharding (cite sharding-strategies#hash); replicate hot keys.

Iteration 1 — HBM ↔ host-RAM tiering with PCIe-overlapped chunked prefill

System after this iteration

Legend. Blue = HBM (Tier 1); purple = RAM (Tier 2). Solid = hot data flow; dashed gray = metadata/control. Accent = new this iteration (RAM-tier slab pool, tier-aware tree, chunked-prefill PCIe-overlap scheduler).

Bottleneck

Iter 0 caches the top ~100 prefixes per node in HBM. But the prefix distribution has a long warm tail: there are ~500–1,000 medium-popularity prefixes (account-specific RAG contexts, multi-turn chats) that each appear in <1% of requests but together represent ~20–25% of the prefix-reuse opportunity Iter 0 leaves on the floor. With HBM-only caching, those prefixes thrash — LRU evicts them between accesses, forcing recompute. The cache hit rate plateaus at ~70% by tokens; we want to get to ~90%. Classified as bottlenecks#memory-capacity-bound — cache is too small to catch the warm tail. Critical observation: the same DGX node has 2 TB of host RAM sitting unused for the KV problem — ~30× the HBM capacity, ~1/8th the bandwidth (400 GB/s vs HBM's 3.35 TB/s/GPU), but plenty of bandwidth if we are careful about when we cross PCIe.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#memory-capacity-bound: (a) extend the cache into the next tier of the storage hierarchy (host RAM); (b) pay the bandwidth cost only when promoting cold-to-hot (not on every access); (c) overlap the bandwidth cost with useful compute so the GPU never stalls; (d) design eviction-cost-aware policies that account for tier-asymmetric recompute cost.

Alternatives considered (C4)

  • HBM + host RAM tier, PCIe-overlapped fetch — chosen. RAM as the warm tier; promotion via PCIe DMA overlapped with chunked prefill. Captures ~80% of the additional hit-rate gain at modest engineering cost. Cite Mooncake paper.
  • Larger HBM (H200 / B200) — rejected for now. H200 (141 GB) and B200 (192 GB) would 2–3× HBM per GPU, raising cache capacity. But: (a) cost per GPU is 1.5–2×, with comparable HBM bandwidth → ROI is worse than tiering; (b) supply is limited; (c) the tiering architecture wins twice (cheaper per byte AND scales to NVMe in Iter 4) and is preferable over hardware substitution. (🔀 if H200 became spot-available at H100 prices, we'd reconsider; ROI flips.)
  • RDMA to peer-node RAM (cross-node tier directly) — deferred to Iter 2. Local RAM is faster and operationally simpler; we earn the per-node win first, then aggregate.
  • NVMe tier instead of RAM — rejected for warm tier. NVMe at 12 GB/s per drive (96 GB/s aggregate) is plenty for capacity but ~4× the latency of RAM; warm tier should be lower latency. NVMe is the right answer for cold tier (Iter 4).

The chosen move

Four coupled changes:

  • Tier-aware radix tree. Each prefix node now carries tier_residency (HBM, RAM, or BOTH during transition). Lookup returns the slab handles + their current tier; the prefill path dispatches accordingly. The radix tree itself lives in host RAM (small, ~few hundred MB metadata).
  • RAM-backed KV pool. 1.5 TB of each node's 2 TB host RAM is allocated as a tier-2 KV pool, structured as mmap'd 1 GB huge pages (avoiding TLB miss storms on large transfers). Slab allocator dispenses page-sized (16-token) extents indexed by (prefix_node_id, layer_id).
  • PCIe-overlapped promotion. When a request lands and its prefix is RAM-resident, the worker issues async cudaMemcpyAsync calls layer-by-layer, fetching slabs from RAM into HBM. Crucially, these are scheduled in parallel with chunked prefill of the unique suffix: while layer L of the suffix prefills (compute-bound on the GPU), layer L+1 of the cached prefix DMAs in (PCIe-bound on the bus). At PCIe Gen5 x16 = 64 GB/s and per-layer slab size 320 KB / GPU, one layer transfers in ~5 µs; 80 layers in ~400 µs total — well under the prefill latency it overlaps with. The GPU never stalls waiting for KV.
  • Tier-cost-aware eviction. When HBM is pressured, demote the least-recently-used prefix slabs from HBM to RAM (cheap: PCIe write, then mark HBM page free; the slab still exists, just in RAM now). When RAM is pressured, evict the LRU RAM slab entirely (expensive: requires recompute on next access). Bias the eviction calculus toward keeping prefixes in RAM as long as possible; only evict HBM (effectively free) under any pressure.

Why this addresses the bottleneck shape: we 30× the effective cache capacity per node (40 GB HBM → 1.5 TB RAM warm tier) at negligible latency cost — the PCIe transfer is hidden by overlap with prefill compute that has to happen anyway. Cache hit rate rises from ~70% (HBM-only) to ~88% (HBM + RAM caches a long tail of warm prefixes). The unused 2 TB of RAM per node finally becomes load-bearing.

What breaks (C5)

  • PCIe contention under simultaneous many-promotion bursts. Mitigation: per-node PCIe scheduler that prioritizes the longest-prefix promotion first; bounded concurrent promotions per GPU (~10) with overflow queued. Detection: PCIe utilization > 80% on a node for > 1 sec; alert.
  • RAM-tier corruption / leak. A bug in the slab allocator leaves freed pages not actually freed; RAM fills until OOM and the whole worker pod crashes. Mitigation: page-granularity accounting + slab inventory sweep every 5 min; canary worker with 50% smaller RAM pool detects growth before fleet-wide. Cite failure-modes#resource-leak.
  • Demote-then-immediately-re-need thrashing. Worker demotes prefix P to RAM because HBM is pressured; 200 ms later, a new request arrives needing P. Pay the promotion cost again. Mitigation: hysteresis — only demote slabs with last-access age > 10 sec; bias toward keeping in HBM when recent access. Detection: demote-then-promote-same-slab rate > 1 per minute.
  • NUMA misalignment. Dual-socket host: half the GPUs' PCIe roots are on CPU socket 0, half on socket 1. RAM allocated on the wrong socket halves effective PCIe throughput (cross-socket QPI hop). Mitigation: NUMA-aware slab allocator; pin each GPU's slab pool to its local CPU socket; document the slot-to-socket mapping in node bringup.
  • vLLM internal allocator interaction. vLLM's own paged-attention allocator manages HBM pages; our tier system competes for the same HBM. Bugs here look like "vLLM thinks it has 30 GB free, our allocator thinks it has 30 GB free, total > 30 GB → OOM crash." Mitigation: a single unified HBM allocator in our custom backend; vLLM gets a fixed-size sub-allocator; our tier manager owns the rest.

  • Range-partitioned (by token-sequence prefix range) — rejected. Token sequences don't have meaningful natural ordering; range partitioning gives skewed distribution. Hash is the right partition for content-addressable data. Cite sharding-strategies#hash.
  • Move slabs to a separate KV-cache-only tier (cacheOnly nodes; like Mooncake's cache nodes) — deferred to Iter 3. A natural next step. We do it implicitly as part of prefill/decode disaggregation in Iter 3, where prefill workers double as cache pool hosts (lots of HBM + lots of NIC).
  • NCCL collectives for KV transfer — rejected. NCCL is optimized for training (synchronized all-reduce, broadcast). Our access pattern is point-to-point on-demand fetch — nothing like all-reduce; the API and overhead are mismatched.

The chosen move

Five coupled changes:

  • Consistent-hash prefix ownership. Each prefix's node_id (rolling hash of token sequence) maps deterministically to an owner worker via consistent-hash ring with 150 virtual nodes per worker. Adding/removing a worker reshuffles only ~1/N of the prefix space. Cite consistent-hashing.html.
  • Distributed prefix-locality directory. A Redis Cluster (16 shards) maps each prefix_hash to {owner_node_id, tier, etag, replica_node_ids}. Routing layer queries it (~50 µs RTT); workers update it on install/evict.
  • RDMA serving endpoint. Each worker exposes a libfabric RDMA endpoint that lets peer workers issue RDMA-READ against their RAM-tier slabs. The reads are one-sided — no CPU involvement on the source side beyond initial setup; the NIC DMA-reads from RAM directly to the target node's NIC, which then DMAs into the target's GPU HBM via GPUDirect (no host-CPU stage on the target either). Per-port 200 Gbps = 25 GB/s; per-node 200 GB/s aggregate.
  • Hot-prefix replication. A prefix that receives > 100 QPS of RDMA fetch traffic is automatically replicated to K=2 owners (the original + one more, picked by hash second-pass). Subsequent requests load-balance between the two. Hot-key reads spread; one-of-two-owner failure still serves.
  • Lazy local promote. When worker A repeatedly fetches prefix H from owner B (e.g., > 10 times in 5 min), A promotes a local copy into its own RAM tier and registers as a co-owner in the directory. The directory bumps replica count; future routing prefers locality.

Why this addresses the bottleneck shape: aggregate cache capacity rises from ~1.5 TB (one node) to ~135 TB (90 nodes × 1.5 TB) — 90×. Cache hit rate ceiling rises to ~95% because the warm long tail now fits comfortably. Duplicate-storage waste drops to the hot-replicate set (K=2 for hot keys) which is by design and accepted. The RDMA fetch cost (~500 µs per layer) is comparable to local PCIe (~5 µs per layer) on raw transfer but is acceptable when overlapped with chunked prefill compute (same trick as Iter 1, just with RDMA instead of PCIe).

Iteration 3 — Prefill / decode disaggregation (Splitwise / DistServe shape)

System after this iteration

Legend. Accent = role-specialized workers (new this iteration). Blue = decode (HBM-heavy); purple = cache pool (RAM-heavy); amber = prefill (compute-heavy). Solid = sync request. Dashed gray = directory updates. Dashed red = RDMA cross-worker KV transfer.

Bottleneck

Iter 2's hybrid workers (each node does both prefill and decode) hit two compounding inefficiencies: (a) hardware mismatch — prefill is compute-bound (GPU FLOPs), decode is memory-bandwidth-bound (HBM reads per step); the same H100 fits both, but its expensive 3.35 TB/s HBM is wasted during prefill, and its 989 TFLOP/s compute is wasted during decode; (b) batch interference — a node running both prefill and decode concurrently has its decode batch interrupted by prefill bursts; effective decode throughput is ~30% lower than a decode-only node. Classified as bottlenecks#tail-latency for the batch interference shape (prefill bursts pollute decode tail) plus a hardware-utilization-inefficient shape (not in the canonical taxonomy — we could add it — but observable as compute or bandwidth headroom on each worker depending on the moment). The DistServe and Splitwise papers identify this directly.

Architectural decisions to revisit

Per bottleneck-to-decisions.html on the batch-interference dimension: (a) separate the two phases onto separate worker pools (so each is sized to its dominant resource); (b) introduce a KV-cache handoff mechanism (since the prefill output must reach the decoder); (c) separate the storage tier from compute tiers so prefill workers don't carry decode's hot KV (and vice versa); (d) ensure routing decisions consider the prefill/decode pair allocation, not just one worker. Cross-cutting X6 cells applies — each role becomes its own cell with its own scaling story.

Alternatives considered (C4)

  • Prefill / decode disaggregation with cache-pool tier — chosen. Three worker roles: prefill-only DGX (compute-heavy), decode-only DGX (HBM-bandwidth-heavy), and cache-pool m7i.48xl (CPU+RAM-heavy, no GPU). Mooncake / DistServe / Splitwise shape. Captures ~30% efficiency gain on each side plus the cost-arbitrage win of moving the RAM tier off the (expensive) DGX node onto a (cheap) m7i node.
  • Keep hybrid workers, just isolate prefill batches in time — rejected. A "prefill-only minute, decode-only minute" cycle was tried (DistServe baseline comparison) — loses ~50% throughput because of cold-start effects on each phase. Continuous co-execution of prefill and decode in the same model server is what the disaggregation paper specifically argues against.
  • Custom prefill ASICs (Cerebras, Groq, Etched) — rejected for now. Compute-heavy prefill is a great fit for non-H100 silicon, but adopting it is a multi-quarter integration project. H100 prefill workers are fine; ASIC is a Probe.
  • Multi-instance GPU (MIG) on H100 to host prefill + decode side-by-side — rejected. MIG partitions a single H100 into isolated GPU instances. Provides hard QoS between prefill and decode, but: (a) Halves effective HBM per instance, killing batch size; (b) Doesn't let us mix hardware types between roles. Disaggregation is more flexible.

The chosen move

Five coupled changes:

  • PREFILL role. ~15 DGX H100 nodes designated as prefill-only. They run vLLM in "prefill mode" — large continuous-batched prefill of many concurrent unique-suffix sequences. No decode is scheduled here. HBM cached prefix slabs are looked up from the cache pool via RDMA on demand; computed unique-suffix KV is appended; the full request KV cache is then RDMA-transferred to the decode worker.
  • DECODE role. ~45 DGX H100 nodes designated as decode-only. They run vLLM in "decode mode" — many concurrent in-flight requests, each generating tokens step-by-step. HBM is full of in-flight requests' full KV (prefix + accumulated decode tokens). Because no prefill bursts steal bandwidth, decode batch size rises 30% and throughput per node rises proportionally.
  • CACHE-POOL role. ~20 m7i.48xlarge (CPU-and-RAM only, no GPU; 768 GB RAM each, 200 Gbps NIC, ~$3/hr) carved out as a dedicated cache pool. They hold the RAM-tier prefix cache aggregate (15 TB total) and expose RDMA-read endpoints. They have no decode HBM and no prefill compute. Cost arbitrage: $3/hr × 20 = $60/hr replaces what would have been spread across the DGX HBM+RAM of more expensive hybrid nodes.
  • Prefill→decode KV handoff. When a prefill worker completes prefill for request R, it RDMA-pushes R's full KV cache (prefix-slabs-referenced + unique-suffix-slabs) into the assigned decode worker's HBM. The decode worker starts decoding while the last few layers' KV still arrive. Layer-by-layer overlap (as in Iter 1) means the first decode token can emit ~50 ms after prefill completes. Cite DistServe for the disaggregation pattern + handoff overlap math.
  • Cache-pool pull as primary, not local RAM tier. Decode workers no longer maintain a host-RAM tier of their own — they rely on the cache pool. Their local RAM (2 TB on DGX) goes to spill buffers + activations + a small (~200 GB) on-decoder hot-prefix cache for the requests they're currently serving. This frees the rest of DGX RAM that was getting underutilized on decode-only workloads.

Why this addresses the bottleneck shape: hardware utilization rises on both ends (prefill workers are now consistently 70%+ MFU; decode workers are consistently 80%+ HBM-bandwidth-utilized). Disentangling the workloads means each side's tail-latency is bounded by only its own queue, not the other's bursts. Cost arbitrage on the cache pool: replacing 15 TB of DGX-attached RAM with 15 TB of m7i.48xl RAM saves ~$1k/hr ≈ $720k/mo — nearly half of Iter 2's total bill.

Bottleneck

Iter 3's 12 TB RAM cache pool catches the warm prefix tail but the cold-but-predictable tail still misses. Two specific cold-but-predictable sources fall through every cache: (a) the RAG corpus long tail — we have 10 M RAG chunks but only the top 1% (~100k chunks) are reused frequently; the remaining 99% appear in < 0.1% of requests but those misses still cost full prefill ($80k/mo waste); (b) account-specific contexts — users' long-lived custom system prompts that appear once a day per user; not "popular" enough for RAM-tier persistence but predictable enough to be worth keeping warm. The remaining 3–5% cache-miss rate translates to ~$80k/mo of pure prefill recomputation cost — the last efficiency to unlock. Classified as bottlenecks#cold-start — predictable cold-cache miss with deterministic content; and as bottlenecks#disk-io-bound implicit (we're moving from "need RAM for everything" to "can serve some from disk").

Architectural decisions to revisit

Per bottleneck-to-decisions.html#cold-start: (a) add a deeper, cheaper storage tier (NVMe) for cold-but-predictable data; (b) introduce offline precompute for predictable artifacts (RAG corpus chunks) so they live in cache from day one; (c) introduce content-addressable storage so the same RAG chunk text is served from the same cached slabs regardless of which retriever surfaced it. NVMe access latency (~5 ms for a full prefix) is acceptable when it replaces a full prefill (200 ms+); the ratio is 40× latency improvement and cost reduction proportional to per-byte storage cost (NVMe is ~10× cheaper than RAM per TB).

Alternatives considered (C4)

  • NVMe cold tier + content-addressable RAG precompute — chosen. Adds Tier 3 (NVMe) under existing RAM tier; precomputes the popular RAG corpus offline; routing layer becomes RAG-aware. Captures the last ~3–5% miss rate at ~$50/mo precompute cost + small NVMe nodes.
  • Object store (S3) instead of NVMe — rejected. S3 first-byte latency is ~50 ms p99; we need ~5 ms to overlap with prefill. S3 fits for offline batch loading (one-shot, in the precompute job) but not online tier.
  • External KV store (Redis large; Memcached) for cold tier — rejected. Adds an RPC hop (~1 ms); 1 KB-keyed value semantics don't match our paged-attention slab shape; net slower than NVMe + RDMA.
  • Per-user persistent KV (long-lived multi-turn user sessions) — deferred to Probe B. Could pre-prefill each user's conversation history into NVMe nightly; we don't because the long-tail UX value isn't worth the operational complexity (per-user state lifecycle, GDPR erasure, etc.); per-request prefix-tree caching already captures the within-day reuse.

Slab access is via io_uring + O_DIRECT reads against memory-mapped file regions; per-page (16-token) access latency ~30 µs (random read on hot NVMe); full prefix (1,250 pages) ~5 ms with full PCIe-NVMe bandwidth aggregation. Eviction LRU on age; pages younger than ~7 days stay; older are reclaimed.
  • Content-addressable RAG corpus precompute. A nightly offline batch job runs on 1–2 reserved prefill nodes (4 hr/night). For each of the top-1% (~100k) RAG corpus chunks: prefill the chunk through the model, capture its KV slabs (per layer per page), store as a content-addressable blob on NVMe keyed by SHA-256 of the chunk's tokens. Total storage: ~8 TB of precomputed slabs (100k chunks × 500 tokens × 160 KB).
  • RAG-aware routing. The retrieval layer (upstream service, out-of-scope for the iteration but the API contract changes) passes chunk_ids[] in addition to the assembled prompt. Routing layer queries the precompute index for each chunk; for found chunks, the request's prefill plan is "skip these chunks; assemble their cached KV; only prefill non-cached chunks + user turn." Per-request prefill cost drops to "user turn only" for requests where all retrieved chunks are pre-cached — ~30% of all requests achieve this.

Why this addresses the bottleneck shape: the most predictable cold misses (RAG corpus access) are eliminated by offline precompute; the remaining cold long tail is served from NVMe at 40× less cost than recompute. Final cache hit rate rises to 97% by tokens with a tiny tail-recompute residual.

What breaks (C5)

  • NVMe wear-out under high write churn. Frequent eviction-and-rewrite churns NVMe TBW (terabytes written) budget. Gen5 NVMe is rated ~1 DWPD (drive-writes-per-day); at our write rates we should be ~0.2 DWPD — safe. Monitoring: SMART data (Percentage_Used, Available_Spare); alert at 80% wear.
  • Precompute drift. RAG corpus updates daily but precompute job runs nightly; for a few hours, the precomputed-KV index may reference stale chunks (chunk text edited between snapshots). Mitigation: precompute job stores chunk content_hash; serve-time lookup verifies hash matches current corpus before using. If mismatch, fall back to online prefill.
  • NVMe-tier latency spikes under flash-translation-layer GC. NVMe SSDs occasionally pause for internal GC; p99.9 read latency can hit 50 ms. Mitigation: over-provision NVMe (8 drives, use 75% capacity); use enterprise NVMe with low-tail-latency firmware (e.g., Solidigm D7-PS1010, Micron 7500 Pro); align serving SLO to p99 not p99.9.
  • Pool node failure with NVMe data loss. NVMe drives are local, not replicated. A pool node dies → its NVMe content is gone until rebuilt. Mitigation: the NVMe tier is cache (reproducible from prefill); recovery is "evict from directory, reprefill on access" — user-visible as a temporary cache-miss spike for affected prefix ranges. K=2 replication on hot prefixes already provides some redundancy. Total durability for KV: not required (reproducible). For RAG precompute, the source-of-truth is the RAG corpus (separately durable); precompute is regenerable from a single offline job.
  • Routing layer becomes critical SPOF for RAG awareness. If routing layer fails to query RAG precompute index, all requests fall back to online recompute; prefill load triples. Mitigation: routing layer has Redis Cluster index (16 shards); failure of one shard degrades 1/16 of lookups; full Redis outage triggers automatic mode-switch to "no RAG precompute lookups, online recompute on all chunks." Cite failure-modes#cascading-timeout.

Probe iterations (the interviewer pulls the thread)

Beyond the 5-Core budget. Each in a <details> block.

Probe A — Multi-region active-active (latency-pinned per-region serving)

Three regions (US, EU, APAC). Each region has its own full fleet (prefill + decode + cache pool + NVMe). Routing layer is geo-DNS + Anycast. The challenge: prefix-cache pool sharing across regions is impractical (cross-region RDMA latency ~50 ms vs ~500 µs intra-region). Solution: per-region cache pools; popular prefixes precomputed in all three regions independently (each region's precompute job runs locally). RAG corpus is the same globally but its precompute is replicated. EU residency for user-PII prompts: PII detection at API edge; PII-tagged requests pin to EU region's serving fleet only. Cross-region replication uses async pull (a cold-cache request in APAC for a prefix only in US can pull via WAN ~150 ms, comparable to recompute — not advantageous, but a fallback).

Probe B — Per-user persistent conversation state

The Iter 0–4 design caches by prefix-content; per-user multi-turn conversations share content across turns but the cache has no concept of "this is User U's session." A per-user persistent KV tier could pre-pin each active user's conversation history (most recent N turns) in their assigned region's cache pool, surviving days of idle time. Cost: ~1 GB per active user × 10M users × $0.023/GB-mo S3-equivalent = ~$200k/mo just for storage. Operational complexity (per-user lifecycle, GDPR right-to-erasure, account deletion cascade) is significant. We defer because the within-day reuse (already caught by Iter 0–4) covers 90% of the value at 5% of the cost.

Probe C — Speculative decoding integration

#34 Iter 5 introduces speculative decoding. In this scenario, spec decoding adds another KV-cache axis: the draft model's KV cache (smaller, runs on the same decode worker or a co-located cheap GPU). Naively this multiplies KV cache by ~1.5×; in practice the draft model's KV is tiny (e.g., 1.5B model: ~10× smaller) and can be co-resident. Integration with prefix sharing is direct: the draft model also benefits from prefix caching (its draft path also starts from the same prompt). Estimated additional win: ~20% throughput uplift; cost-per-token drops to ~$0.15 / M-output.

Probe D — What if the model were 405B (Llama-3.1 405B class)?

Per-token KV at FP8 for a 405B model with similar GQA: ~600 KB / token (vs 160 KB for 70B). Per-30k-context request: 18 GB KV (vs 5 GB for 70B). HBM-only model would not fit a single request at 100k context (60 GB). RAM tiering becomes mandatory, not optional. Cache-pool nodes need 2× the storage. Cross-region precompute is necessary for SLO. The arc above scales linearly with model size in storage cost; latency and TTFT are unaffected because they're dominated by prefill compute (which scales linearly with model size and is mitigated identically by prefix-cache hits). Mooncake's published 405B-class results vindicate this.

Degradation contract

  • Cache pool partial failure: affected prefix range falls back to online prefill; TTFT spikes for those requests; routing layer marks affected workers unhealthy. Cite failure-modes#slow-dependency.
  • Prefill capacity exhausted: routing layer returns 503 with Retry-After; clients buffer and retry. Cite failure-modes#backlog-overflow.
  • Decode capacity exhausted: same 503 from routing; alternatively, degrade ITL target temporarily (let p99 ITL rise to 150 ms vs 100 ms target).
  • Directory failure: routing falls back to round-robin (no cache awareness); cache hit rate drops to ~0% briefly; prefill load triples. Detected within 10 sec; failover to standby Redis cluster within 30 sec.
  • NVMe tier loss on one pool node: evicted prefixes recomputed on next access (acceptable; the NVMe tier is cache).
  • RAG precompute drift: serve-time hash check catches stale precomputed entries; falls back to online prefill on mismatch.

Real-system comparison

Mooncake (Moonshot AI / Kimi; arXiv 2406.17565) is the closest published production cousin. They report similar architecture: a disaggregated KV cache pool with HBM/RAM/NVMe tiers, RDMA-based cross-node fetch, and prefill/decode separation. Their published numbers at similar scale show comparable cache hit rates (90%+) and similar cost reductions vs HBM-only baselines. Where we converge: nearly all major decisions (consistent-hash partitioning, content-addressable storage, RDMA over libfabric, NVMe for cold tier, hot-key replication). Where we diverge: Mooncake's published design uses a separate dedicated "cache cluster" type whereas our cache pool is just CPU+RAM+NVMe nodes (no GPU) — same idea, simpler hardware procurement.

SGLang (arXiv 2312.07104) pioneered RadixAttention — the per-node prefix-tree caching of Iter 0. Our Iter 0 matches their design directly; subsequent iterations layer tiering on top of their model.

LMCache (lmcache.github.io) is a third-party open-source library that adds CPU/disk KV cache offload to vLLM — matches Iter 1 + Iter 4 directly. Adopting it would reduce our engineering surface but we run a custom backend for the full disaggregation pattern.

Splitwise (arXiv 2311.18677) and DistServe (arXiv 2401.09670) are the prefill/decode disaggregation papers; Iter 3 directly applies their findings (DistServe reports ~30% throughput improvement from disaggregation; we measured ~30% in Iter 3 sizing).

Anticipated pushbacks

Eight stress tests against the derivation above.

What if prefix-reuse rate dropped from 70% to 30%? (workload flip)

The whole iteration arc collapses. Iter 0's value drops by 60%; Iter 1's RAM tier becomes overprovisioned; Iter 2's distributed pool becomes overhead. Estimated impact: $/M-output rises from $0.19 to ~$0.50 because we're paying for cache infrastructure that doesn't help. The right move would be to revert toward #34's design (per-request KV; PagedAttention; speculative decoding) and minimize cache infrastructure. Recognizing this regime requires monitoring the workload-level cache hit rate; if it drops below 50% for > 1 week, redesign decision is on the table. The architecture above is right for our actual measured workload (70%+), not for a counterfactual workload.

Why not just buy H200 / B200 GPUs and skip tiering? (hardware alt)

H200 (141 GB HBM) and B200 (192 GB HBM) raise per-GPU HBM 2–2.4× vs H100. At equal HBM-only sizing, our 110-hybrid-node fleet (Iter 0) would shrink to ~50 nodes — saving ~$1M/mo on hardware. But: B200 cost per GPU is ~$50k vs H100's ~$30k (a 67% per-unit premium); MFU on B200 for our workload is ~15% lower (production teams report; new architecture, less mature kernels); supply is constrained (8-month lead times in 2024). Net economic case for B200-only: ~10% cheaper than tiered H100. Tiering wins on flexibility (extends down to NVMe), works on already-deployed H100s, and would compose with B200 when supply allows. We adopt B200 incrementally for new decode capacity. Cite storage-engines.html.

What changes under GDPR + EU residency? (regulatory)

Per-user prompts containing PII must stay EU-resident. Solution: PII detection at API edge (regex-based for emails/phones plus content-classifier for free text); PII-tagged requests pin to EU region's serving fleet. The prefix-cache is content-addressable hash — the hash itself is not PII, but the cached KV cache is derived from PII content. So cached prefix slabs for PII-tagged content must also stay in EU region. Implementation: directory tracks region-affinity per prefix; PII prefixes never replicate cross-region (K=1 only in source region). Non-PII content (system prompts, RAG corpus) replicates freely across regions. Cite cross-cutting#privacy.

The entire cache pool fails. What happens? (failure)

Cache pool failure (all 20 m7i nodes down due to control-plane outage) means no cross-node prefix sharing; per-decode-worker HBM still has its own active prefix cache (~250 GB per node, ~3,000 prefixes resident). System degrades to "Iter 0 equivalent" — per-node caching only. Cache hit rate drops to ~60% (the per-decode-worker resident set); prefill load triples; routing layer's prefix-locality lookups return empty so it round-robins. TTFT p95 rises from 700 ms to ~1.5 sec. The system is still working; just degraded. Recovery: rebuild cache pool on standby instances; takes ~30 min to repopulate the warm tier from prefill traffic. Cite failure-modes#bad-deploy.

Where does the bill come from? (cost)

~70% of the bill is decode worker DGX nodes. This is fundamental: decode is memory-bandwidth-bound on HBM, and HBM bandwidth costs what it costs. The cost-vs-latency knob is decode batch size: doubling batch size cuts decode-node count ~40%, but doubles ITL p99 (HBM read load per step doubles). We hold batch at 80 to meet ITL target. The next-biggest opportunity is speculative decoding (Probe C; ~20% throughput uplift expected). Beyond that: better HBM bandwidth per dollar requires the next GPU generation. Cite cost-shapes.html.

How do you roll this out from #34's baseline without downtime? (migration)

Iter 0 (prefix cache): vLLM backend update; canary cohort first. Backward-compatible: routing layer's affinity hint is optional; falling back to round-robin works at degraded hit rate. Iter 1 (RAM tier): per-worker rollout; capacity stays same. Iter 2 (RDMA pool): introduce directory + RDMA endpoints alongside; new workers register and start serving as both peer-RDMA-source and -client; rolling deploy. Iter 3 (disaggregation): the biggest change — carve out some nodes as prefill-only, others as decode-only, and add cache-pool nodes; new request routing flows through prefill→decode pair; old hybrid path stays available as fallback for 4 weeks. Iter 4 (NVMe + RAG precompute): pure addition; precompute job runs on offline batch capacity; live serving picks up precomputed slabs as bonus. Total migration time: ~16 weeks at steady cadence.

Production TTFT is high right now — where do you look first? (debug)

Triage order on the dashboards: (1) cache hit rate per tier — if <60%, directory or pool is failing; (2) prefill QPS per node — if <50% target, prefill nodes are starved (network? bad deploy?); (3) RDMA fetch p99 per direction — if >5 ms, NIC or top-of-rack switch issue; (4) NVMe wear / saturation — if drives saturated, evict aggressively or add capacity; (5) directory Redis latency — if > 5 ms, routing layer queues stack up; (6) decode batch size per node — if at cap, fleet too small for current load. Each SLI maps to a specific iteration's bottleneck; the runbook entry corresponds.

What's the attack surface? Could an attacker poison the cache? (adversary)

Attack vectors: (1) Cache-key collision — rolling-hash chunks can collide; if an attacker can craft a prompt whose hash matches a legitimate cached prefix, they'd get a "match" that decodes to nonsense. Mitigation: use SHA-256 for prefix hash (collision-resistant), not just rolling-hash. (2) Cache pollution — submitting many distinct never-reused prefixes to consume cache budget. Mitigation: per-tenant rate limits at API edge; cache admission policy weights frequency-of-reuse heavily. (3) Side-channel timing — TTFT for "cache hit" is lower than "miss"; an attacker can measure to infer "did some other user just send a similar prompt?" Mitigation: add jitter to first-byte response; treat as low-severity privacy issue for now (cite Spectre/Meltdown analog). (4) RDMA man-in-the-middle — intra-DC traffic on private VLAN; mTLS at routing layer; physical access control on rack networks.

S7. Key Takeaways

  • Prefix-reuse is the load-bearing workload assumption. The whole arc collapses without it. Measure it before designing; revisit periodically; if it drops below 50% for sustained periods, redesign.
  • Tiering is a way to spend RAM and NVMe on a problem that nominally needs HBM. HBM is the scarcest, most expensive memory; everything below it is 10–100× cheaper per GB. If 70% of your working set is shareable, ~90× effective capacity is unlocked by tiering at low marginal cost.
  • The radix tree is the right algorithmic structure for prefix-cache. Hash-based flat caches lose ~2× capacity to non-shared sub-prefixes; the trie shares branch heads. Cite SGLang RadixAttention.
  • Disaggregation is a hardware-utilization play, not a complexity play. Prefill and decode want different hardware; co-located, each underutilizes the GPU; separated, each runs at high utilization. Splitwise / DistServe pattern.
  • The cache pool can run on CPU-only nodes — carve it off DGX. A cache pool is RAM + NIC; you don't need GPUs. Replacing DGX-attached RAM with m7i-attached RAM at ~1/10 the per-node cost is the cheapest move in the arc.
  • NVMe is for cold-but-predictable; RAM is for warm-but-popular. Not interchangeable: NVMe random read latency (~30 µs) is fine when overlapped with prefill; RAM random read is ~100 ns and serves the active working set.
  • RDMA + GPUDirect makes cross-node KV transfer practical. Without RDMA (200 Gbps direct GPU↔GPU), the cross-node pool wouldn't work; we'd be stuck at single-node caches. The hardware unlock makes the architecture.
  • Iterations expose the next bottleneck. Iter 2 surfaced the hardware-utilization gap that Iter 3 fixed; Iter 3's cache-pool concentration enabled Iter 4's NVMe tier. The arc doesn't have shortcuts; each move builds on the prior.
