Designing the Home-Timeline Fan-Out (Twitter/X)

500M DAU, 400M posts/day, 1M timeline reads/sec, one celebrity with 200M followers. We start from a naive read-time JOIN and iterate to a hybrid push/pull pipeline with cached celebrity feeds, hedged reads, and tiered inbox storage.

NALSD Walkthrough • ~50 min read • LC #10 (Design Twitter)

LC kernel. Design Twitter (LC #10). Textbook structure: k-way merge of follower feeds at read time (a min-heap pulling the next-newest post from each followee's posts list). Property that breaks at scale: the merge fans in over millions of followees and a single celebrity write fans out over 200M follower inboxes. Both directions exceed any single coordinator. The C8 production cousin replaces the read-time merge with precomputed inboxes for the bulk of users and inverts to read-time pull only for celebrities — a hybrid that no LC variant exposes.

S0. Scenario & Requirements

A consumer social product. Users post short text + media references; followers see those posts in their home timeline, ordered by recency-blended-with-engagement. The whole product collapses if either direction breaks: either a normal user composes and their followers don't see it (write path failed), or a user opens the app and the timeline takes 3 seconds to assemble (read path failed). Twitter, X, LinkedIn feed, Meta News Feed, Threads — same shape.

System class

Class: product-shaped — user-facing, "user can <verb> <noun>" capabilities; cite functional-requirements.html#class-product. The class fixes the FR vocabulary (capabilities, not operations + semantics) and the NFR additions (user-facing latency, fan-out tolerance, per-entity consistency).

Functional requirements (cite functional-requirements.html)

  • Compose. User posts text (≤280 chars) + optional media URL. Returns a post id.
  • Read home timeline. User opens the app, gets the latest 20–100 ranked posts from the people they follow, paginated.
  • Follow / unfollow. User follows another user; effect on home timeline is bounded-eventually-visible.
  • Engage. Like / repost / reply. Engagement signals feed the ranker.
  • View profile timeline. All posts by one author, recency order. (Distinct from home timeline; cheap.)

Out of scope (named, not silently dropped): search (separate index pipeline), DMs (separate messaging service), ads (separate ranker), trending (separate top-K service — see #15), video transcoding, moderation, edits, account onboarding, payments.

Flagged assumption. Ranking is a black-box scorer applied at read assembly time; the page treats it as a constant-cost step. If it became a 100 ms ML inference, Iteration 3 (tail-latency) would add a candidate-generation layer in front of the ranker; the rest of the arc is unchanged.

The fundamental tension. Compose is one-to-many: one writer produces a post that N readers must see. We can pay the cost at write time (precompute every reader's view) or read time (assemble on demand). For a typical user with 200 followers, write-time precomputation is cheap (200 small writes per post). For a celebrity with 200,000,000 followers, write-time precomputation is catastrophic (200 M small writes for one post; meltdown). The arc resolves this asymmetry: push for normals, pull for celebrities, cache the celebrity pulls, hedge the assembly tail, tier the inbox storage.

S1. Core Entities (cite entities-and-data.html)

ID scheme (cite identifier-schemes.html#snowflake): Snowflake (64-bit) for user_id and post_id. Why: 8 bytes vs UUIDv7's 16 bytes matters at 12 B post-rows × inbox-windows of 1 k entries × 500 M users = inbox storage halves; sortable by ms-precision timestamp (so chronological inbox queries don't need a separate created_at column); predictable but not user-guessable per worker. Rejected: UUIDv7 (16 B doubles every inbox entry, which is the dominant byte budget); UUIDv4 (no sortability, defeats range scans on author timeline); KSUID (20 B + second-precision is too coarse for 10,000 posts/sec which exceeds 1/sec per ms granularity).

The partition-key heuristic (cite entities-and-data.html — "spotting the partition key"): the user-facing query path is "fetch this user's home timeline." 1 M reads/sec all start from user_id. Partition every read-path entity (Inbox, MuteFilter) by user_id — this is the access locality. Posts and CelebrityFeed partition by author_id because compose and celebrity-pull are author-locality.

SQL vs NoSQL — per-dataset choices (cite storage-engines.html)

The product owns six datasets with very different access patterns; one engine doesn't fit all. We pick per-dataset, citing the engine catalog. Five of six are NoSQL because the dominant access pattern is point-read or sorted-set range; only the User profile dataset needs relational ACID and is SQL.

DatasetEngine choiceFamilyWhy this engine (cite anchor)Rejected alternative

Memcached — lacks sorted-set type; we'd reimplement it in app code. DynamoDB — managed alternative, but at 1M ops/sec the unit cost is 5–10× higher than self-managed Redis.

Plain disk (HDD) — IOPS too low for even 1% of 1M reads. S3 — latency too high (~50 ms p95) for warm-tier reads.
Celebrity Feed (per-author sorted-set, hot key) Redis (5× replicated for top-100) NoSQL (KV in-memory) Same primitive as Inbox HOT (sorted-set ZRANGE), but read pattern is power-law concentrated — cite access-patterns#power-law. Replication addresses the hot-key bottleneck (Iter 2). Total data is tiny (~1.2 GB) so storage is not the cost driver. Cassandra — per-key replication is awkward in LSM; Cassandra's per-row replication factor is fleet-wide, not per-key. Memcached — same as Inbox HOT.

Summary. 1 SQL (User profile) + 5 NoSQL stores. The dividing line: does this dataset need cross-row ACID transactions? Only User does (handle uniqueness, follower_count update under concurrent follows). Everything else is either (a) append-only system-of-record where LSM wins, or (b) regenerable cache/inbox where in-memory KV wins. The page does NOT use a single-engine architecture (e.g., "everything in Postgres") because the access patterns are too divergent — cite storage-engines.html's "system-of-record + derived-stores" pattern: Posts KV is the SoR; Inbox tiers, Celebrity Feed, and follower-graph materializations are derived stores rebuildable from the SoR + Follow Graph.

Configuration specifics that drive the design (partition key, clustering key, replication factor, persistence/durability mode, eviction policy) are stated, not inferred. Posts KV: Cassandra 4.1 LTS (vs ScyllaDB vs DynamoDB vs FoundationDB)

OptionVerdictWhy (constraint)
Cassandra 4.1 LTSchosenLSM-optimized for write-heavy time-clustered data; partition by author_id + clustering by post_id DESC gives O(1) "latest N posts per author" range scan. Mature operator tooling (nodetool, Reaper for repairs). Self-managed at our scale beats DynamoDB on $/op by 5–10×.
ScyllaDBrejected4–6× per-node throughput vs Cassandra (C++ rewrite, shard-per-core) is real, but operational ecosystem (Reaper, Medusa, kubectl operators, observability dashboards) is thinner; community size is <1/10 of Cassandra. DynamoDBrejectedOperational simplicity is real but unit cost at 10k writes/sec sustained = ~$1,250/M writes × 26B writes/mo ≈ $32k/mo just for writes — 2× our entire Posts KV cluster cost on Cassandra. Vendor lock-in adds re-architecture cost on exit.
FoundationDBrejectedStrong ACID is a feature we don't need (Posts are append-only-immutable); the transaction overhead of FDB is paid every write without benefit. Better fit for the User profile dataset (we don't use it there either; see Postgres rationale).

  Partition key:           author_id (hash; uniform load; avoids hot-author within-partition)
  Clustering key:          post_id DESC (Snowflake; sortable; range-scan latest 1000 posts O(log N))
  Replication factor:      RF=3 in-region; LOCAL_QUORUM reads/writes for sync replication
  Compaction strategy:     TimeWindowCompactionStrategy (TWCS) with 1-day windows
                           — posts immutable + time-clustered → TWCS bounds compaction churn,
                             avoids LCS write amp on append-heavy workload.
  Read repair:             dclocal_read_repair_chance=0.1 (10% of reads do background repair)
  Bloom filter:            FP rate = 0.01 (default); skips disk for 99% of partition misses
  Hint window:             3 hours (allows brief node outage without full repair)
  Memtable flush:          memtable_cleanup_threshold=0.4 (back-pressure on write storms)

Modern alternatives we evaluated & would reconsider in 2–3 years (principal-bar lookahead). Per the article's "capacity to educate the interviewer" beat, the staff+ designer surfaces forward-looking options the interviewer may not have considered, even when rejecting them today.

  • Pelikan (Twitter's open-source Redis/Memcached modernization, BSD) — we use it for the warm tier; on a 2–3 year horizon Pelikan-Pingora could replace Redis 7 for Inbox HOT if their Redis-protocol implementation reaches feature parity. Twitter's internal numbers: 5× per-node throughput vs Redis; 50% lower memory overhead.
  • DragonflyDB (multi-threaded Redis-compatible) — today rejected on AGPL + ecosystem maturity; in 2 years if commercial license clarifies and operators mature, it would shrink the Inbox HOT cluster from 38 nodes to ~12.
  • ScyllaDB Enterprise — today rejected on operational maturity; in 2–3 years if their managed offering (ScyllaDB Cloud) matures we'd reconsider for Posts KV; 5× cluster-size reduction would save ~$80k/mo.
  • Tigerbeetle (financial-grade transaction DB) — not relevant here, but mentioned as an example of purpose-built specialization beating general-purpose; we should watch for similar specializations in the timeline-fanout space (a "social-fanout-as-a-product" primitive could emerge).
  • RedPanda Community — today rejected on license + community size; if their open-source tier matures, the operational savings (single binary, no JVM) would be material at our 200-partition fanout topic.
  • FoundationDB layered architecture — not a current fit, but principal-level designers should know FDB is the substrate Apple, Snowflake, and others build on. If we were building a new system today and could choose a unified ACID-KV substrate, FDB + custom layers would be a compelling architecture; we don't because retrofitting onto an existing 6-store design would be a multi-year migration with no compelling forcing function.

The page treats today's choices as defensible at staff-bar and the modern-alternatives list as defensible at principal-bar lookahead — the design choice would re-open if any of these alternatives reach our maturity threshold.

How this system fits in the company (principal-bar context). Timeline doesn't live alone; it has standing contracts with at least 9 adjacent systems. Naming them surfaces the org-level couplings a staff+/principal designer must reason about.

  • Search (separate inverted-index pipeline; planned scenario #45 autocomplete-suggest) — Posts KV emits a CDC stream (Debezium → Kafka) that search ingests; search is downstream-eventual.
  • Ranking / ML platform (planned scenario #49 realtime-ml-ranker) — we treat the ranker as a black-box scorer with ≤ 50 ms inference budget; contract is ranker-model-version per A/B experiment; we don't pin version, the ranker does.
  • Notifications / push (separate fan-out service) — we publish to compose-events; notification service consumes the same Kafka topic and routes to push-notification providers.
  • Trust & Safety / abuse moderation — T&S has a synchronous compose-time veto API (block compose if author is suspended/spam-flagged); we call this on the compose path. T&S also issues retroactive purgatory via a separate Kafka topic that suppresses already-fanned-out posts from inboxes.
  • Ads & monetization — ads injection at read-time is a separate service that we call after timeline assembly, before ranker; ads contract: 1–3 ads inserted per 100 organic posts. Latency budget: 30 ms.
  • Observability platform — we emit metrics to the company's central platform (cite #30); SLI/SLO definitions live there, not on this team's runbook.
  • Customer support tooling — "why is this post not showing for user X?" debugging requires read access to inbox, celebrity feed, and posts; we expose a per-user "timeline trace" API that walks the assembly path.
  • Data warehouse / analytics — daily ETL from Posts KV (CDC) and from inbox-fanout topic (engagement events) into the analytics warehouse for product metrics, A/B test analysis, and ML training data.
  • Identity / auth — user_id and session validation come from the central identity service (separate); we trust JWTs at the edge gateway.

The boundaries above are Conway's-Law lines: each adjacent system is a different team. The contract with each is a stable interface; couplings beyond the interface are tactical and need explicit governance. A principal-level designer maintains the contract list and watches it for drift.

Multi-year evolution (principal-bar trajectory).

  • 1 year: stabilize the design above; complete strangler-fig migration off the existing Postgres-IN-query path (cite the migration plan in C6); land Probe B (live-tail SSE) for engagement lift; pay down Iter 2's 5× replication operational burden via auto-tuning.
  • 3 years: Probe A (multi-region active-active) becomes mandatory as EU residency + APAC growth force it; hybrid push/pull threshold may need revisiting if bot-driven content pushes celebrity-rate tweets to 100× today's count; the ranker is now an ML model with 50 ms inference budget that materially eats into our 200 ms p95.
  • 5 years: the inbox-storage architecture may be obsolete — CXL-attached memory + DPU offload could replace the Redis HOT tier with a unified large-RAM substrate; the LSM-vs-cache distinction blurs. Modern alternatives (DragonflyDB, ScyllaDB, RedPanda) likely mature into adoption; expect a 30–50% cluster shrink. Our exit strategy: system-of-record + derived-stores pattern means any tier can be swapped without affecting the SoR contract.

S4. Naive Architecture — the simplest thing that meets the FRs

One Postgres holds two tables: posts(post_id, author_id, body, created_at) with index on (author_id, created_at), and follows(follower_id, followee_id, state) with index on follower_id. Compose is one INSERT into posts. Read timeline is the textbook k-way merge expressed in SQL: pick the followee set for the reader, pull the latest posts from each, sort by recency, return top 20. This is exactly the LC #10 reference solution at a single node.

Why it's broken (per failing NFR)

  • Read throughput. 5,000 QPS/node vs 1,000,000 QPS target → 200× short. Even 200 read replicas would meet QPS in aggregate but cost ≈ 200 × $700/mo ≈ $140k/mo just for the read fleet, and each replica still pays the 200-key fan-in cost (CPU + buffer cache pressure compound).
  • Read latency. Under load, a single PG node's 200-key IN-query saturates the planner; p95 climbs from 50 ms (idle) to several seconds (under contention). The 200 ms p95 NFR is missed by 10×+ at target QPS.
  • Compose throughput. ✅ passes. 10k INSERT/sec on one node is fine.
  • Availability. ❌ Single Postgres = ~99.9%; we need 99.95%.
  • Consistency. ✅ trivially passes (single-node, strong).

Cost of naive at NFR-target load: infeasible. Even a 200-read-replica cluster at ≈ $200k+/mo wouldn't hit p95 ≤ 200 ms because every query does 200-key fan-in. The first failing NFR is read throughput, dominated by the per-request fan-in cost. We must change the shape of the read — not scale Postgres.

Bottleneck

Naive fails read-throughput by 200× (5,000 QPS/node × any reasonable fleet vs 1,000,000 QPS target). The shape of the failure is per-request fan-in compounded by single-coordinator throughput — each timeline read does 200 index seeks, multiplied by 1 M req/sec = 200 M index seeks/sec across the cluster, and the PG primary's planner CPU caps at ~5,000 such queries/sec. Classified as fan-in-read (the access pattern: many keys per query) compounded with coordinator-bottleneck (one PG primary as the throughput ceiling). The corresponding write-side asymmetry — one write event must reach many readers — matches fan-out-write; iteration 0 inverts the cost direction by precomputing the readers' views.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#fanout: push vs pull (decide per-record), time-bucketed materialization, differential prefetch. The dominant lever for read-heavy fan-out workloads is to shift the cost to write time: precompute each reader's timeline so the read becomes a single key fetch.

Alternatives considered (C4)

  • Fanout-on-write to per-user inbox — chosen. Each compose triggers N inbox-writes (one per follower); reads become a single Redis ZRANGE. Wins because N writes-per-compose × 10,000 compose/sec ≈ 2 M writes/sec is well within a sharded Redis cluster, while reads collapse to 1 key fetch each.
  • More read replicas — rejected. Each replica still does 200-key fan-in (same per-query CPU); fleet scales linearly with QPS but per-request latency remains 50–500 ms under load. The shape didn't change. (🔀 If avg_followees were 5 instead of 400, this would win — the fan-in cost wouldn't dominate.)
  • Materialized view in Postgres — rejected. A timeline_inbox table with INSERT-ON-COMPOSE is structurally the same as our chosen move, but Postgres write throughput for 2 M small INSERTs/sec requires partitioning + write-optimized engine; we'd reinvent the Redis cluster on top of PG. Use the right tool.

The chosen move

For each compose event, look up the author's follower set and write the new post_id into each follower's inbox:<user_id> sorted-set (Redis ZADD with score = ms timestamp). Cap the sorted-set at the most recent 1,000 entries (ZREMRANGEBYRANK on overflow). Reads become a single ZRANGE returning 20 post IDs, then a batched MGET against Posts KV to hydrate bodies.

Cite sharding-strategies.html#hash: hash partition by user_id for the inbox cluster (uniform load; no locality needed since reads are point-fetches). Cite storage-engines.html#kv-engine for the Redis choice (sub-ms point reads + atomic sorted-set ops are the win; durability best-effort is fine because inbox is regenerable from Posts KV via offline rebuild).

What breaks (C5; cite failure-modes.html)

  • Hot-key meltdown on follow-graph read. Compose-time SMEMBERS for a celebrity author returns 200 M follower IDs — a single list read of ~1.6 GB. The follow-graph node serving that author saturates NIC. Detection: per-key bytes-read p99 > 100 MB. Recovery: paginate the follow-graph read into chunks of 10k; defer to Iter 1 where we'll suppress the SMEMBERS entirely for celebrities.
  • Backlog overflow on the fanout dispatcher. A celebrity post creates a 200 M-job burst; the dispatcher queue (in-memory or Kafka topic) backs up faster than workers drain. Detection: queue depth monotonically increasing for > 30s. Recovery: shed by priority (drop low-priority "engaged-but-not-recently-active" follower writes); DLQ messages over 5 min old; address structurally in Iter 1.
  • Stampede on cold-start of inactive-user inbox. If we evict an inactive user's inbox (Iter 4), they return and trigger a rebuild from Posts KV; under flash-mob conditions (a viral article re-engages 10M users in 10 min), the cold-start rate exceeds origin throughput. Detection: rebuild QPS > baseline × 10. Recovery: refresh-ahead on login signal; single-flight per user_id rebuild.

Bottleneck

Iter 0's fanout dispatcher chokes on celebrity posts: a 200 M-follower compose generates 200 M ZADDs, which at 50k ZADDs/node/sec across 50 inbox shards = 1 M ZADDs/sec aggregate, takes 200 seconds to drain — and the 5 s consistency NFR is blown by 40×. The signal: the top 0.01% of authors generate > 70% of total fanout-write bytes, while accounting for ≈ 5% of compose volume. Classified as fanout-asymmetry per the diagnostic taxonomy: top-percentile fan-out orders of magnitude above median.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#fanout: push vs pull (pick per-record), time-bucketed materialization, differential prefetch. The canonical first move for fanout-asymmetry is hybrid push/pull — choose, per author, whether their post is pushed to followers' inboxes at compose time or pulled from a per-author feed at read time. Push the post for a normal-author compose (cheap: write to ~200 follower inboxes once); pull the post at read time for a celebrity-author's compose (cost amortized across many readers; one ZRANGE on the celebrity's per-author feed during timeline assembly, instead of 200 M inbox writes at compose).

Alternatives considered (C4)

  • Hybrid: push for normals, pull for celebrities — chosen. Suppress fanout-on-write when author.follower_count > 10,000; instead, write to a per-author celebrity_feed:<author_id> sorted-set. At read time, the timeline service merges the reader's inbox with the celebrity feeds of the celebrities they follow.
  • Slow-path async fanout for celebrities — rejected. Drain the 200 M-write burst over 30 minutes via low-priority workers. Honors most NFRs but pushes the consistency window from 5 s to 30 min — users notice when a celebrity tweet doesn't appear after 5 min. Rejected because it trades off the NFR we're trying to protect.
  • Per-celebrity inbox bypass (each celebrity has their own dedicated infrastructure tier) — rejected. Operationally complex; doesn't scale to "rising celebrities" who cross threshold dynamically; the threshold-based hybrid achieves the same isolation without per-celebrity provisioning.

The chosen move

Compose-time decision tree:

on_compose(author_id, post):
    write Posts KV[author_id, post_id] = post          // always
    if user_record(author_id).is_celebrity:            // follower_count > 10_000
        ZADD celebrity_feed:<author_id> (post_id, ts)  // ONE write
        // skip fanout entirely
    else:
        followers = SMEMBERS follows-by-followee:<author_id>
        for f in followers:                            // ~200 typical
            ZADD inbox:<f> (post_id, ts)               // batched

Read-time assembly:

on_timeline_read(user_id, cursor):
    inbox_ids   = ZRANGE inbox:<user_id> (cursor) 100               // 1 call
    celeb_set   = user.celebrities_followed                          // ≤10 typical
    celeb_ids   = parallel for c in celeb_set:                       // ~3 calls avg
                      ZRANGE celebrity_feed:<c> (cursor_per_celeb) 100
    merged      = top-K merge(inbox_ids ∪ celeb_ids by score) :: 100
    bodies      = MGET Posts KV[merged]                              // 1 batched fetch
    return rank(bodies) :: 20

Cite access-patterns.html#power-law: this is the textbook hybrid for power-law fan-out. Cite sharding-strategies.html#hash for both inbox (by reader user_id) and celebrity_feed (by author user_id); they're separate clusters because their access patterns differ (inbox = uniform random reads; celebrity feed = power-law concentrated reads, becomes hot-key in Iter 2).

Why the threshold is 10,000 followers

Crossover analysis. Push cost per post: F writes (where F = author's follower count). Pull cost per post: Ractive reads where Ractive = followers who actually read their timeline within the post's relevance window. Empirically Ractive/F ≈ 0.3 (30% of followers check within 24h), and each pull-read is ~1× the cost of a push-write (both are sorted-set ops on Redis). So push is cheaper iff F < Ractive × cost_ratio; with cost_ratio = 1, push wins for F < 0.3 × F — never. The real lever is burst smoothing: pulls are amortized across reader requests (steady), pushes are bursts per compose (spiky). Push wins when the burst doesn't blow the consistency budget. With 50 inbox shards × 50k ZADD/sec = 2.5M ZADD/sec aggregate and a 5s consistency window, max sustainable burst = F ≤ 2.5M × 5 = 12.5M ZADDs — round down to 10,000 followers as the threshold (gives 1,250× safety margin for concurrent celebrity bursts).

Bottleneck

Architectural decisions to revisit

Per bottleneck-to-decisions.html#hot-key: caching, single-flight, key replication via salting, admission policy, hot-key isolation tier. The canonical combination — L1 cache + single-flight + key replication — is pulled in together because each handles a different cliff: L1 absorbs steady state, single-flight prevents cache-miss stampede, replication eliminates the per-key write bottleneck on cache invalidation.

Alternatives considered (C4)

  • L1 in-app cache + single-flight + 5× replication of celebrity feeds — chosen. Three coherent moves, each addressing a related cliff of the hot-key shape. L1 absorbs 95% of steady-state read traffic per app pod.   • Salt the key (write to multiple shards via random suffix) — rejected. Works for write-hot keys (e.g., a counter incremented at high rate), but our hot-key is read-hot on a low-write-rate key (celebrity composes a few posts/day). The simpler move is just to replicate the read replicas; salting adds write-side coordination cost we don't need.
  • Push celebrity feeds to client-side cache only — rejected. The "follow Drake" set is dynamic; clients can't subscribe efficiently to N celebrity feeds and stay current. Server-side L1 is on the same machine as the timeline assembler — near-zero RTT, server controls invalidation centrally.

Bottleneck

Iter 2's read path makes 4 parallel calls (1 inbox + ~3 celebrity feeds). At 12 ms p95 per component, the assembly p99 amplifies via tail multiplication: p99assembly = 1 − (1 − Fp99(t))4 evaluated at each component's distribution. Empirically, with each component's p99 ≈ 50 ms (a 4× tail-vs-p95 fan-out), the 4-way max gives p99assembly ≈ 100–150 ms — still within 800 ms but it eats most of the budget, leaving no room for the ranker or downstream slowdowns. Classified as tail-latency. Signal: p99system ≫ p99component; result requires N parallel calls.

Secondary bottleneck: the mute filter is a per-user blacklist of ~100 author_ids checked on every timeline read. Currently a Redis SISMEMBER per request — 1M extra ops/sec to mute-filter cluster. We can move this to the edge.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#tail-latency: hedging, fan-out reduction, deadline propagation, bounded-result-set. Combined here with edge-tier filtering for the mute list (cite cross-cutting.html#edge) because it both reduces backend QPS by 1M/sec and shaves the mute-RTT off the critical path.

Alternatives considered (C4)

  • Hedged requests at p95 + Bloom mute filter pushed to client cookie — chosen. Hedging: re-issue a parallel request to a replica when the first is past the p95 threshold (e.g., 20 ms); take whichever returns first. Cost: ~5% extra calls (only the slow tail hedges). Bloom: mute list of ~100 author_ids fits in a 1 KB Bloom filter at 1% false-positive rate; rendered server-side, sent to client as a cookie, refreshed on mute-list change. Cite sketches.html#bloom.
  • Fewer celebrity feed lookups (cap at top-3 most-active celebs) — rejected. Saves a call but breaks completeness: the user follows 10 celebs; capping at 3 silently omits 7. Acceptable for engagement-ranked feeds (the ranker would deprioritize them anyway), but the page treats ranking as a black-box; this iteration shouldn't depend on ranker semantics.
  • Server-side mute filter via materialized exclude list — rejected. A user could mute on one device and read on another; the server-side check is correct but it's the QPS we're trying to remove. Bloom-on-cookie keeps the check at the client (close to free) with 1% false-positive rate (a muted post occasionally slips through — acceptable; the user can mute again or report; lossy is fine for blocklists).

Alternatives considered (C4)

  • Three-tier inbox: hot Redis (recent 100) → warm NVMe-backed KV (next 900) → cold pull from author feeds — chosen. Hot tier serves the typical refresh (1 ZRANGE returning latest 20 of 100). What breaks (C5)

  • Cold-start cliff for returning inactive users. An inactive user returns; their hot tier is full of recent posts (refilled by ongoing fanout), but their warm tier entries 101–1000 may have been evicted under memory pressure. Detection: NVMe miss rate spikes during reactivation events. Recovery: refresh-ahead on login signal warms the warm tier from Posts KV pulls; takes ~50 ms, acceptable for a returning-user latency budget.
  • Spill-queue backlog between hot and warm tiers. If warm tier is slow (e.g., NVMe instance failure), spill writes back up; hot Redis grows past 100 entries; eventually OOM. Detection: spill_queue.depth > threshold for > 1 min. Recovery: backpressure on compose path (5% of composes shed); replace failed warm node; drain backlog.
  • Data corruption at warm tier. NVMe write fails silently → entry 200 in user X's inbox is wrong. Detection: end-to-end checksum on inbox row (post_ids hash) compared at read time. Recovery: re-fetch the corrupted row from Posts KV (slow but correct); periodic anti-entropy scrub.

Probe A — Multi-region active-active

Bottleneck. Single-region serves global DAU at 200 ms cross-region RTT — EU users see 200 ms+ on every refresh. Probe B — Live-tail SSE for active sessions

Bottleneck. Active users currently poll on refresh. SSE/WebSocket push delivers new posts to followers' open sessions in real time, removing perceived staleness. Probe C — Anti-spam / poison-author quotas

Bottleneck. A spam author posting 1,000/sec triggers fanout-on-write to 200 followers/post = 200,000 inbox-writes/sec from one user — rate-limit at compose. Cite rate-limiting.html. Per-user GCRA at 100 posts/min; fanout-time check on author's reputation score (precomputed nightly). Suspended authors: posts go to a "purgatory" tier that's not fanned out, only visible on profile timeline.

What if compose were 100× higher (1M posts/sec)? (scale flip)

  • Fanout scales linearly: 1M × 200 = 200M ZADDs/sec → 4,000 inbox shards at 50k/node.
  • Celebrity threshold drops from 10k → ~1k followers (a 10k-follower author now creates a meaningful spike).
  • L1 hit rate holds (Zipf head doesn't broaden); celebrity-feed cluster sizes up ~10× because origin QPS tracks reads, not writes.
  • New dominant bottleneck: cross-AZ egress on fanout writes — 200M × 50 B × $0.01/GB ≈ $90k/mo (matches storage).
  • Fix: co-locate inbox shards in the same AZ as the fanout dispatcher, sized per cell (cite cross-cutting#cells).

Why Redis sorted sets and not Cassandra/DynamoDB for the inbox? (alt-engine)

  • Cassandra (cite storage-engines#lsm) is the natural alternative: append-friendly, partition-by-user_id, clustering by post_id.
  • Latency trade: Cassandra p99 for a 100-row partition scan ≈ 10–20 ms (LSM read-amp + bloom + on-disk SSTables); Redis ≈ 0.5 ms (RAM).
  • Budget pressure: 200 ms p95 has headroom for Cassandra today, but tightens once we add ranking + hedging + hydration + the ranker itself.
  • Cassandra wins if storage cost (Iter 4 tier) becomes 10× more painful — skip the warm tier, put everything on LSM.
  • DynamoDB wins if we value fully-managed ops at 10× unit cost — operational simplicity an interviewer might trade for.

What changes under GDPR (EU residency + right-to-erasure)? (regulatory)

  • EU residency: Probe A (multi-region active-active) becomes mandatory — EU users' Posts KV + inbox shards stay in EU.
  • Replication policy: follow-graph still replicates cross-region (small, write-light); EU posts do not replicate out.
  • Erasure of own posts (cite cross-cutting#privacy): delete in Posts KV; LSM tombstone propagates to inboxes within the warm-tier compaction window (≤ 7 days).
  • Erasure of traces (followees' stale follower-counts): per-user-id partition + cascade-delete job; SLA 30 days; audit log to S3 immutable.
  • Arc unchanged — only cluster geography + back-end deletion sweeps differ.

The celebrity Redis cluster goes down right now. What happens? (failure)

  • Degradation contract: assembly falls back to inbox-only — normal-author posts at full latency, celebrity posts disappear.
  • User-visible: banner "Some posts unavailable; refreshing soon."

  • RTO ≈ 30 s; RPO undefined — feeds are regenerable from posts:<celeb_id> in Posts KV (rebuild ≈ 5 min).

S7. Key Takeaways

  • The k-way merge is a single-machine artifact. Once F (followees) is large and reads dominate writes by 100:1, the right move is to invert the merge: precompute timelines on write so reads become point lookups. The LC kernel is exact; the production cousin is exact-on-the-typical-path with a fallback for the celebrity tail.
  • Push and pull aren't a choice; they're a per-record decision. The threshold (10k followers in our derivation) is set by the burst budget on the fanout cluster, not by an aesthetic preference. When the access pattern is power-law, the answer is always both; the only question is where the threshold lives and how it adjusts to load.
  • Hot keys are different from hot partitions. Hot partitions can be split or salted; hot keys can't. The only moves that work for read-hot keys are caching, single-flight, and replication of the hot key itself. Drake's celebrity feed is the canonical read-hot key in social systems.
  • Tail latency from N parallel calls is amplified by N. Hedging at p95 is a high-leverage fix: ~5% extra calls turn a p99 of N-way assembly into something close to single-component p99. Don't pay for it on the critical path; pay for it on the tail.
  • Edge filters are free latency. A 1 KB Bloom filter on a cookie removes 1 M ops/sec of mute-list lookups from the backend AND removes the RTT from the critical path. The cost (1% false-positive on hidden posts) is trivially acceptable for blocklists; the same is not true for allowlists.
  • Storage tiering is the cheapest money you can save. Active users' typical refresh hits the most-recent ~20 entries; the trailing 980 entries are ~99% never-read but cost ~10× more on RAM than NVMe. Tiering buys ~3× cost reduction at the cost of ~5 ms higher p95 on deep pagination.
  • Each iteration cited a canonical bottleneck. Naive → fan-in-read + coordinator-bottleneck. Iter 0 → fanout-asymmetry. Iter 1 → hot-key. Iter 2 → tail-latency. Iter 3 → cardinality-explosion. The arc is the canonical bottlenecks list, traversed in order; that's not coincidence — that's what NALSD looks like when the diagnostic taxonomy is the ladder.
