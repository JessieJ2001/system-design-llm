URL Shortener: Collision-Free ID Generation at Scale

Generate 100 million unique 7-character short URLs per day across 5 data centers — without coordination on every write and without collisions, ever.

NALSD Walkthrough • ~15 min read

The Scenario

You are designing the ID generation layer for a URL shortener service like bit.ly or tinyurl. Every time a user submits a long URL, the system must generate a unique 7-character short code (e.g., https://short.ly/aB3kQ9z). The short code must be globally unique across all data centers, generated instantly, and never collide with any previously generated code.

Back-of-Envelope Math

Write throughput:
  100M URLs/day ÷ 86,400 sec/day ≈ 1,157 writes/sec average
  Peak (3× average): ~3,500 writes/sec

Read throughput:
  100× writes = 115,700 reads/sec average
  Peak: ~350,000 reads/sec

ID space exhaustion:
  62^7 = 3,521,614,606,208 ≈ 3.52 trillion possible codes
  100M/day × 365 days/year = 36.5 billion/year
  3.52 trillion ÷ 36.5 billion = 96.4 years until exhaustion
  → More than enough capacity.

Storage (over 5 years):
  Total records: 100M/day × 365 × 5 = 182.5 billion
  Per record: 7B (short code) + 100B (avg long URL) + 50B (metadata) ≈ 160B
  Total: 182.5B × 160B = 29.2 TB
  With replication factor 3: ~88 TB across all data centers.

Per data center:
  88 TB ÷ 5 DCs = 17.6 TB per DC
  → Fits on a modest database cluster.

Naive Design: Random Generation with Collision Check

The simplest approach: for each new URL, generate a random 7-character base62 string. Check if it already exists in the database. If it does, generate another. Repeat until unique.

import random
import string

BASE62 = string.ascii_letters + string.digits  # a-z, A-Z, 0-9

def generate_short_url(long_url, db):
    while True:
        # Generate random 7-character code
        code = ''.join(random.choices(BASE62, k=7))

        # Check for collision
        if not db.exists(code):
            db.insert(code, long_url)
            return code
        # Collision! Try again.
        # How often does this happen? Let's calculate...

The Birthday Paradox Kills This Approach

The birthday paradox is a counterintuitive result from probability theory: in a group of just 23 people, there is a 50% chance that two share a birthday (out of 365 possible days). The same principle applies to hash collisions. The collision probability is far higher than the naive 1/S estimate because any pair of generated codes can collide, not just the new code with one specific existing code.

Birthday paradox formula:
  P(collision) ≈ 1 - e^(-N² / 2S)

  N = number of generated codes so far
  S = size of the code space = 62^7 = 3.52 trillion

Let's calculate collision probability at different scales:

After 1 day (100M codes):
  N = 10^8
  P ≈ 1 - e^(-(10^8)² / (2 × 3.52 × 10^12))
  P ≈ 1 - e^(-10^16 / 7.04 × 10^12)
  P ≈ 1 - e^(-1,420)
  P ≈ 1.0  (effectively 100% — collision is CERTAIN)

Wait — a collision is certain after just ONE DAY?

Let's verify with smaller numbers. After 1 hour (~4.2M codes):
  N = 4.2 × 10^6
  P ≈ 1 - e^(-(4.2 × 10^6)² / (2 × 3.52 × 10^12))
  P ≈ 1 - e^(-17.64 × 10^12 / 7.04 × 10^12)
  P ≈ 1 - e^(-2.5)
  P ≈ 1 - 0.082
  P ≈ 91.8%

After just 1 HOUR, there is a 92% chance of at least one collision.

After 10 minutes (~700K codes):
  N = 7 × 10^5
  P ≈ 1 - e^(-(7 × 10^5)² / (2 × 3.52 × 10^12))
  P ≈ 1 - e^(-4.9 × 10^11 / 7.04 × 10^12)
  P ≈ 1 - e^(-0.0696)
  P ≈ 6.7%

Even after 10 minutes, there's a 7% chance of collision.

The birthday paradox in action: With 3.52 trillion possible codes, you might expect collisions to be rare. But at 100M codes per day, collisions become certain within hours. The reason: you are not checking one code against one other code — you are checking each new code against ALL previous codes. The number of pairs grows as N2, which vastly outpaces the linear growth of N.

What Breaks

Collisions Cause Retries, Retries Cause Latency

After 1 billion codes (10 days of operation):
  Collision probability per generation attempt:
    P(single collision) = N / S = 10^9 / 3.52 × 10^12 = 0.000284 = 0.028%

  That seems low, but at 1,157 writes/sec:
    Expected collisions per second: 1,157 × 0.000284 = 0.33
    Expected collisions per hour: ~1,183

  Each collision requires:
    1. Generate random code
    2. Database lookup (does this code exist?)  → 2-5 ms
    3. Collision! Generate another code.
    4. Another database lookup                  → 2-5 ms
    5. Maybe another collision...

  Average retries per write ≈ 1 / (1 - P) ≈ 1.000284 (barely over 1)
  But worst case: multiple retries → tail latency spikes.

After 100 billion codes (2.7 years):
  P(single collision) = 10^11 / 3.52 × 10^12 = 2.84%
  Expected retries per write: 1 / (1 - 0.0284) = 1.029
  ~3% of writes need at least one retry.
  At 1,157 writes/sec: ~34 retries/sec, each adding 2-5 ms latency.

The Real Problem: Database Lookups on Every Write

Even without collisions, the collision check requires a database read before every write. At 3,500 writes/sec (peak), that is 3,500 extra database reads just for uniqueness verification. This halves the effective write throughput and adds latency to every single write.

Multi-DC Coordination is Impossible at This Speed

With 5 data centers, two DCs could simultaneously generate the same random code. To prevent this, you would need a distributed lock or consensus protocol for every write. Cross-DC latency is 50–200 ms. A distributed lock adds this latency to every write, blowing past the 50 ms SLA.

Iteration 1: Counter-Based IDs with Pre-Allocated Ranges

Instead of random generation, use a monotonically increasing counter. Counter value 1 becomes short code 0000001, value 62 becomes 0000010, and so on. No collisions ever — each number maps to exactly one base62 string.

Base62 Encoding

BASE62 = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"

def encode_base62(num):
    """Convert an integer to a 7-character base62 string."""
    if num == 0:
        return BASE62[0] * 7

    chars = []
    while num > 0:
        chars.append(BASE62[num % 62])
        num //= 62

    # Pad to 7 characters
    while len(chars) < 7:
        chars.append(BASE62[0])

    return ''.join(reversed(chars))

# Examples:
# encode_base62(0)       → "0000000"
# encode_base62(1)       → "0000001"
# encode_base62(61)      → "000000Z"
# encode_base62(62)      → "0000010"
# encode_base62(3521614606207) → "ZZZZZZZ" (max 7-char value)

The Coordination Problem

A single global counter works perfectly on one machine. But we have 5 data centers, each with multiple application servers. If every write must contact a single global counter to get the next number, that counter becomes:

  • A bottleneck — 3,500 writes/sec all hitting one service.
  • A single point of failure — counter down = no new URLs.
  • A latency source — cross-DC round trip to the counter adds 50–200 ms.

Solution: Pre-Allocated Ranges

Instead of handing out one number at a time, hand out ranges of numbers. Each application server gets a block of 1 million sequential IDs. It uses them locally (no network calls) until the block is exhausted, then requests a new block.

Central Range Service (one instance, backed by a durable counter in a DB):

  State: next_available = 0  (starts at zero)

  allocate_range(size=1_000_000):
    start = next_available
    next_available += size
    return (start, start + size - 1)
    # Persisted in DB to survive restarts.

How it plays out:

  App Server A requests range → gets [0, 999,999]
  App Server B requests range → gets [1,000,000, 1,999,999]
  App Server C requests range → gets [2,000,000, 2,999,999]
  ...

  App Server A processes writes:
    Write 1: counter = 0       → encode_base62(0) = "0000000"
    Write 2: counter = 1       → encode_base62(1) = "0000001"
    Write 3: counter = 2       → encode_base62(2) = "0000002"
    ...
    Write 1,000,000: counter = 999,999 → encode_base62(999999) = "0004C91"
    Range exhausted → request new range from Central Service.

How often does each server contact the Central Service?
  At 1,157 writes/sec across all servers.
  Say 20 app servers total (4 per DC).
  Each handles ~58 writes/sec.
  Range of 1M IDs lasts: 1,000,000 / 58 = 17,241 seconds ≈ 4.8 hours.

  Central Service contacted: 20 servers / 4.8 hours = ~4 requests/hour.
  This is trivially low. The Central Service could be down for HOURS
  and existing ranges would keep working.

No coordination on the write path. Each application server generates IDs from its local range with zero network calls. The Central Range Service is contacted once every 4.8 hours per server. Even if it is down for an entire day, each server has enough IDs to keep operating.

Iteration 2: Multi-DC Range Allocation

With 5 data centers, we want each DC to operate independently for writes. The Central Range Service gives each DC a large block, and each DC’s internal coordinator sub-divides it among its app servers.

Two-Level Range Allocation

Level 1: Central Service → Data Centers
  Each DC gets a block of 100 million IDs.

  DC 1 (US-East):   [0,          99,999,999]
  DC 2 (US-West):   [100,000,000, 199,999,999]
  DC 3 (EU):        [200,000,000, 299,999,999]
  DC 4 (Asia):      [300,000,000, 399,999,999]
  DC 5 (Australia): [400,000,000, 499,999,999]

  How long does 100M IDs last per DC?
    100M URLs/day ÷ 5 DCs = 20M/day per DC
    100M ÷ 20M/day = 5 days of capacity per block.

Level 2: DC Coordinator → App Servers
  Within each DC, a local coordinator sub-divides the DC's block.

  DC 1 has 4 app servers:
    Server 1A: [0,         24,999,999]   (25M IDs)
    Server 1B: [25,000,000, 49,999,999]
    Server 1C: [50,000,000, 74,999,999]
    Server 1D: [75,000,000, 99,999,999]

  Each server's 25M IDs last:
    25M ÷ (20M/day ÷ 4 servers) = 25M ÷ 5M/day = 5 days.

Exhaustion timeline for the ENTIRE ID space:
  3.52 trillion ÷ 100M/day = 35,200 days = 96.4 years.

  Number of 100M-block allocations before exhaustion:
    3.52 trillion ÷ 100M = 35,200 allocations.
    At 5 DCs requesting every 5 days: 1 request/day from all DCs.
    35,200 allocations ÷ 1/day = 96.4 years. Consistent. ✓

Handling DC Coordinator Failure

If the DC coordinator is down:
  - App servers continue using their existing sub-ranges.
  - At 5 days of capacity per sub-range, there is no urgency.
  - The coordinator can be restarted or failed over within hours.

If the Central Range Service is down:
  - DCs continue using their existing 100M blocks.
  - At 5 days per block, there is time to recover.
  - The Central Service's state is just one number (next_available).
  - Stored in a replicated database (e.g., etcd, ZooKeeper).
  - Recovery: read the counter from the replica, resume allocation.

Wasted IDs on server crash:
  If Server 1A crashes with 20M unused IDs in its range,
  those IDs are "lost" (never used). Is this a problem?

  20M wasted ÷ 3.52 trillion total = 0.000006%.
  Even if servers crash daily, losing 20M IDs each time:
    20M × 365 × 100 years = 730 billion wasted.
    Still only 20% of the 3.52T space. Plenty of room.

Iteration 3: Making Sequential IDs Look Random

Counter-based IDs have a security issue: they are predictable. If a user sees short.ly/0004C91, they can guess that short.ly/0004C90 and short.ly/0004C92 also exist. An attacker could enumerate all URLs by incrementing the counter. This reveals private URLs and total volume information.

Solution: Format-Preserving Encryption (FPE)

Apply a bijective permutation to the counter before base62 encoding. A bijection (one-to-one mapping) means every counter value maps to exactly one output, and every output maps back to exactly one counter value. No collisions possible by construction.

How format-preserving encryption works conceptually:

  Think of it as a "shuffle" of all numbers from 0 to 3.52 trillion.

  Without FPE:
    Counter:  0    1    2    3    4    5    6    7   ...
    Code:     000  001  002  003  004  005  006  007 ...
    (sequential → predictable)

  With FPE (using a secret key K):
    Counter:  0    1    2    3    4    5    6    7   ...
    FPE(K):   2.8T 47M  981B 12   3.1T 555M 78B  1.2T ...
    Code:     xQ9z aB3k mN7p 000c zZZa h8Kj bC2m pR5v ...
    (looks random, but is actually deterministic and reversible)

  Properties:
    - Deterministic: same counter always gives same output.
    - Reversible: given the output + key K, recover the counter.
    - Bijective: no two counters map to the same output. Zero collisions.
    - Pseudorandom: without key K, output looks random.
from cryptography.hazmat.primitives.ciphers import Cipher, algorithms, modes

# Simplified FPE using Feistel network on the counter value
# (production systems use FF1 or FF3-1 from NIST SP 800-38G)

SECRET_KEY = b'your-32-byte-secret-key-here!!'  # Keep secret!

def feistel_permute(counter, rounds=6):
    """
    Simple Feistel network that permutes a counter value
    within the range [0, 62^7) without collisions.
    """
    # Split counter into two halves
    modulus = 62 ** 4  # ~14.7 million (half of 7-digit base62)
    left = counter // modulus
    right = counter % modulus

    for i in range(rounds):
        # Round function: hash the right half with the round number
        f = hash_round(right, i, SECRET_KEY) % modulus
        left, right = right, (left + f) % modulus

    return left * modulus + right

def generate_short_url(counter):
    """Counter → FPE → Base62 → short URL."""
    permuted = feistel_permute(counter)
    return encode_base62(permuted)

# Result:
# generate_short_url(0)  → "kR3mQ7x"  (looks random)
# generate_short_url(1)  → "aB9pL2w"  (looks random)
# generate_short_url(2)  → "zN5vT8c"  (looks random)
# But all are deterministic and collision-free!

Why this works: A Feistel network is a bijection by construction. No matter what the round function produces, the overall mapping is always one-to-one. This is the same principle that makes DES and other block ciphers invertible. We are effectively using a block cipher with a block size of 7 base62 digits instead of the usual 64 or 128 bits.

Final Design

Architecture

Write Path (Creating a Short URL)

1. App server receives request: shorten("https://example.com/long/path")

2. App server increments its LOCAL counter: counter = 47,382,951
   (No network call. No database lookup. No coordination.)

3. Apply FPE: feistel_permute(47382951) → 2,891,004,573

4. Encode to base62: encode_base62(2891004573) → "kR3mQ7x"

5. Write to database: INSERT (code="kR3mQ7x", url="https://...")
   (No uniqueness check needed — the counter guarantees uniqueness.)

6. Return to user: "https://short.ly/kR3mQ7x"

Total write latency:
  Counter increment:  ~0 ms (in-memory)
  FPE computation:    ~0.01 ms (CPU-only)
  Base62 encoding:    ~0 ms
  Database write:     ~5 ms (SSD-backed, local DC)
  Total:              ~5 ms  ✓  (well under 50 ms SLA)

Read Path (Redirecting a Short URL)

1. User visits: https://short.ly/kR3mQ7x

2. Look up "kR3mQ7x" in cache (Redis/Memcached).
   → Cache hit (90%+ of the time): return long URL. Done.
   → Cache miss: query database.

3. Database: SELECT url FROM urls WHERE code = "kR3mQ7x"
   → Return long URL. Cache it for future reads.

4. HTTP 301 redirect to the long URL.

Read latency:
  Cache hit:  ~1 ms   ✓
  Cache miss: ~5 ms   ✓  (both under 10 ms SLA)

Handling Edge Cases

Custom short URLs (user picks their own code):
  - Store in a separate "custom codes" table.
  - Before assigning a counter-generated code, check it doesn't
    collide with any custom code. (Rare — custom codes are uncommon
    and the space is 3.52 trillion.)

Analytics (click counting):
  - Append click events to a Kafka topic: (code, timestamp, IP, user-agent).
  - Consume into a time-series database for per-URL analytics.
  - Decoupled from the redirect path — no latency added.

Expiration:
  - Add expires_at column. Default: never.
  - Background job deletes expired entries.
  - Counter values are NOT recycled (too complex, not needed with 96-year space).

Rate limiting:
  - Per-user write limit: 1,000 URLs/day (prevents abuse).
  - Per-IP read limit: 10,000 reads/minute (prevents scraping).

Key Takeaways

  • Never use random generation for high-volume unique IDs. The birthday paradox makes collisions certain far sooner than intuition suggests. With 3.52 trillion possible codes and 100M generated per day, collisions begin within hours.
  • Pre-allocated ranges eliminate coordination on the write path. Each server gets a block of 1 million IDs. It generates IDs locally with zero network calls. The range service is contacted once every 4.8 hours — if it is down for days, the system keeps running.
  • Format-preserving encryption hides sequential patterns. Apply a bijective permutation (Feistel network) to the counter before base62 encoding. The output looks random but is deterministic and collision-free by construction. This prevents URL enumeration attacks.
  • Two-level range allocation enables multi-DC operation. Central service allocates large blocks to DCs. DC coordinators sub-divide for app servers. No cross-DC communication on the write path.
  • Back-of-envelope math prevented a bad design. The birthday paradox calculation immediately disqualified random generation. The ID space exhaustion calculation (96 years) confirmed that counter-based generation is viable. Always do the math first.
