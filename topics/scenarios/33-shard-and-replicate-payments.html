Shard & Replicate a Global Payment Ledger

Designing the storage layer for a fintech payment processing system that requires both sharding (for capacity) and replication (for availability) — and walking through every challenge that arises when you combine both.

NALSD Walkthrough • ~20 min read

The Scenario

You are designing the storage layer for a global fintech company’s payment processing ledger. Every payment — card charge, refund, transfer, payout — must be recorded as an immutable transaction in this ledger. The ledger is the source of truth for all financial data.

Data Model

Each transaction record is approximately 500 bytes and contains:

CREATE TABLE transactions (
  txn_id        UUID PRIMARY KEY,       -- globally unique
  account_id    BIGINT NOT NULL,         -- the account this belongs to
  amount_cents  BIGINT NOT NULL,         -- e.g., 1999 = $19.99
  currency      CHAR(3) NOT NULL,        -- ISO 4217: USD, EUR, JPY
  txn_type      VARCHAR(16) NOT NULL,    -- CHARGE, REFUND, TRANSFER_IN,
                                         -- TRANSFER_OUT, PAYOUT
  status        VARCHAR(16) NOT NULL,    -- PENDING, CONFIRMED, FAILED
  counterpart   BIGINT,                  -- other account (for transfers)
  idempotency_key VARCHAR(64) UNIQUE,    -- prevents double-processing
  created_at    TIMESTAMP NOT NULL,
  updated_at    TIMESTAMP NOT NULL,
  metadata      JSONB                    -- merchant info, notes, etc.
);

-- Indexes for read patterns
CREATE INDEX idx_txn_account_time
  ON transactions (account_id, created_at DESC);
CREATE INDEX idx_txn_status
  ON transactions (account_id, status);

Access Patterns

  • Account balance lookup — Sum of all CONFIRMED transactions for a given account_id. This is the most critical read. Must return the correct, up-to-date balance (strong consistency). Used before every payment to check sufficient funds.
  • Transaction history — Paginated list of transactions for a given account, ordered by created_at DESC. Powers the user’s statement view. Filtered by date range, type, or status.
  • Write: append new transaction — Insert a new row with status=PENDING, then update to CONFIRMED or FAILED. Append-only (no deletes, no edits to amount). Immutable once CONFIRMED.
  • Cross-account transfer — Debit account A, credit account B. Must be atomic: both happen or neither does. The hardest operation.

Back-of-Envelope Math

Before picking any architecture, let us calculate storage, throughput, and availability requirements. These numbers will drive every decision.

Storage: Why One Machine Is Not Enough

1 year of transactions:
  100M txns/day × 365 days = 36.5 billion rows
  36.5B rows × 500 bytes   = 18.25 TB raw data

Indexes add ~30-50% overhead:
  B-tree on (account_id, created_at DESC): ~6 TB
  B-tree on (account_id, status):          ~3 TB
  Primary key index (txn_id):              ~3 TB
  Total with indexes:                      ~30 TB

PostgreSQL practical limit per instance:
  - VACUUM (garbage collection) on a 30 TB table takes hours
  - pg_dump backup of 30 TB: ~8-12 hours
  - Point-in-time recovery: replay WAL from last base backup
    → could take days for 30 TB
  - ALTER TABLE (add column, reindex): locks or takes hours

Rule of thumb: keep PostgreSQL instances under 1-2 TB for
  comfortable operations. Above that, everything gets painful.

Need at minimum: 30 TB / 1.5 TB per shard ≈ 18-20 shards

Throughput: Can One Machine Handle the Writes?

Single PostgreSQL write capacity (with fsync=on):
  - fsync ensures data hits disk before confirming the write.
    Without fsync, a power failure can lose committed transactions.
    For financial data, fsync=on is non-negotiable.
  - With SSD + fsync: ~3,000-5,000 writes/sec per instance
  - Our peak: 10,000 writes/sec
  - One instance cannot handle peak writes alone.

With 18 shards:
  - Peak writes per shard: 10,000 / 18 ≈ 556 writes/sec
  - Well within a single instance's capacity (3,000-5,000)
  - Headroom factor: 5,000 / 556 ≈ 9x → very comfortable

Read throughput per shard:
  - Balance query: SUM(amount_cents) WHERE account_id = X
    AND status = 'CONFIRMED'
  - With index on (account_id, created_at), this is an index
    scan. Avg account has 36.5B / 10M = 3,650 txns/year.
  - Scanning 3,650 rows to compute a sum: ~2-5 ms
  - Each shard handles ~556K accounts (10M / 18)
  - Even at 5,000 balance queries/sec per shard → fine

Availability: Why Replication Is Mandatory

Target: 99.999% uptime = 5.26 minutes downtime per year

Single PostgreSQL instance availability:
  - Hardware MTBF: ~3-5 years (mean time between failures)
  - Unplanned downtime: ~1-4 hours per incident
  - Planned maintenance: ~2-4 hours per quarter
  - Realistic availability: 99.9% = 8.76 hours downtime/year

99.9% is 100x worse than our 99.999% target.

With replication (1 primary + 2 replicas per shard):
  - All 3 must be down simultaneously for shard to be unavailable
  - P(all 3 down) = (0.001)³ = 0.000000001 = 99.9999999%
  - Even accounting for correlated failures (same rack, same DC):
    P(shard down) ≈ 0.00001 → 99.999% per shard
  - This meets our target.

Total node count:
  18 shards × 3 replicas = 54 database instances

Why Not Just Replicate Without Sharding?

Could we use 1 primary + 2 replicas (3 nodes total)?

Storage: 30 TB on each node.
  ✗ VACUUM takes 12+ hours on 30 TB
  ✗ Backup takes 8-12 hours
  ✗ Recovery takes days
  ✗ Schema changes are nightmarish

Write throughput: 10,000 peak writes on 1 primary.
  ✗ Exceeds single-instance capacity (3,000-5,000)

Conclusion: replication alone solves availability but NOT
  storage capacity or write throughput. We need BOTH.

The core tension: 18 TB of data forces us to shard. 99.999% availability and zero data loss forces us to replicate. When we combine both, we inherit all the challenges of each — cross-shard queries, hot shards, and resharding difficulty (from sharding) plus replication lag, consistency tradeoffs, split-brain, and failover complexity (from replication) — plus new combined challenges like cross-shard transactions with replicated shards and coordinated failover across 54 nodes.

Naive Design: Single PostgreSQL

Start with the simplest thing that could work. One PostgreSQL instance. One transactions table. All reads and writes go to the same machine.

Why This Works Initially

Month 1 numbers:
  Transactions/day: ~300K (ramp-up period)
  Storage: 300K × 500 B × 30 days = 4.5 GB
  Write rate: 300K / 86,400 = ~3.5 writes/sec

  A single PostgreSQL handles this trivially.
  Balance queries on a 4.5 GB table: < 1 ms.
  Backup: seconds. Recovery: minutes.

  Single server, simple operations, easy debugging.
  This is the right design for month 1.

When This Stops Working

Month 6: 10M txns/day
  Storage: ~75 GB → still fine
  Writes: ~115/sec avg, ~1K peak → still fine
  VACUUM: ~30 min → getting noticeable

Month 12: 50M txns/day
  Storage: ~3 TB → queries slowing, VACUUM takes 2+ hours
  Writes: ~580/sec avg, ~5K peak → approaching limit
  Backup: 2-3 hours → risky window

Month 18: 100M txns/day
  Storage: ~9 TB → VACUUM fails to keep up, index bloat
  Writes: ~1,200/sec avg, ~10K peak → exceeds capacity
  Backup: 6+ hours → unacceptable
  Single failure = total outage → violates 99.999% SLA

  ✗ Storage too large for one machine
  ✗ Write throughput exceeds capacity
  ✗ Single point of failure
  ✗ Operational nightmare (backup, recovery, maintenance)

What Breaks

Three things break simultaneously, and each one demands a different solution. The challenge is that the solutions interact.

Problem 1: Storage Exceeds Single Node

30 TB with indexes on one PostgreSQL instance:

VACUUM autovacuum_naptime = 1 min (default)
  - Scans 30 TB table for dead tuples
  - With 1,200 writes/sec = 1,200 dead rows/sec
  - 103M dead rows/day to clean up
  - VACUUM can process ~10K rows/sec
  - Time to vacuum: 103M / 10K = ~10,000 sec ≈ 2.8 hours
  - But new dead rows accumulate during vacuum → never catches up

Table bloat:
  - Dead rows occupy space until vacuumed
  - If VACUUM falls behind: table grows unboundedly
  - Queries slow down (scanning dead rows)
  - Index bloat further compounds the problem

Solution needed: SHARDING (split data across multiple machines)

Problem 2: Write Throughput Limit

Why fsync limits PostgreSQL writes:

1. Client sends INSERT
2. PostgreSQL writes to WAL (Write-Ahead Log) buffer
3. PostgreSQL calls fsync() → forces WAL buffer to disk
4. Disk confirms write is durable → PostgreSQL ACKs client

Step 3 is the bottleneck:
  - SSD fsync latency: ~0.1-0.5 ms
  - With group commit (batching): ~0.2 ms per batch
  - Max throughput: 1000ms / 0.2ms = ~5,000 writes/sec

Our peak: 10,000 writes/sec > 5,000 capacity

You cannot disable fsync for financial data. A power failure
between step 2 and step 3 would lose committed transactions.
For a payment ledger, this means lost money.

Solution needed: SHARDING (split writes across machines)

Problem 3: Single Point of Failure

99.999% availability = 5.26 minutes downtime/year

What happens when the single PostgreSQL server fails?

Scenario: SSD dies at 3:14 AM
  - Detection: monitoring alerts in 30 seconds
  - On-call engineer responds: 5-15 minutes
  - Spin up new server: 2-5 minutes
  - Restore from backup: 6+ hours for 30 TB
  - Replay WAL since last backup: 1-4 hours
  - Total downtime: 7-10 hours

That is ~500 MINUTES of downtime from one incident.
Our annual budget is 5.26 minutes.
One failure consumes 100x our annual budget.

Even with hot standby (async replica):
  - Failover time: 30-60 seconds
  - But async replication → some transactions lost
  - Lost transactions = lost money = regulatory violation

Solution needed: SYNCHRONOUS REPLICATION (zero data loss)
  + automatic failover (minimize downtime)

Key insight: We cannot solve these problems independently. Sharding alone gives capacity but not durability. Replication alone gives durability but not capacity. We need both — and combining them creates new challenges that neither technique has in isolation.

Iteration 1: Shard by account_id + Replicate Each Shard

The fundamental idea: split the 18 TB across 18 shards using hash(account_id) % 18, and give each shard a primary + 2 synchronous replicas for durability. All transactions for one account live on one shard, so balance queries never need to cross shards.

Why hash(account_id) Is the Right Shard Key

Shard key options for a payment ledger:

Option 1: hash(txn_id)
  ✓ Uniform distribution (UUIDs are random)
  ✗ Balance query (SUM WHERE account_id = X) must hit ALL 18
    shards — every single balance check is a scatter-gather.
    At 5,000 balance checks/sec: 5,000 × 18 = 90,000 queries/sec
    across the cluster. Unacceptable.

Option 2: hash(account_id) ← CHOSEN
  ✓ All txns for one account on one shard
  ✓ Balance query is single-shard (fast, consistent)
  ✓ Transaction history is single-shard (fast pagination)
  ✗ Cross-account transfers require cross-shard coordination
  ✗ Whale accounts (merchants with millions of txns) create hot shards

Option 3: range(created_at) — partition by date
  ✓ Good for time-range queries
  ✗ All writes go to the latest shard (hot spot!)
  ✗ Balance query spans ALL shards (full account history)
  ✗ Terrible for write distribution

hash(account_id) wins because the two most frequent operations
(balance lookup and transaction history) both filter by account_id.
Single-shard reads with strong consistency are simple and fast.

How Synchronous Replication Works

Write path for a single transaction on Shard 5:

1. Client sends INSERT to Shard 5 Primary
2. Primary writes to its WAL (Write-Ahead Log)
3. Primary sends WAL record to Sync Replica 1 AND Sync Replica 2
4. Replica 1 writes WAL to its disk, sends ACK
5. Replica 2 writes WAL to its disk, sends ACK
6. Primary waits for BOTH ACKs (synchronous)
7. Primary commits the transaction
8. Primary responds to client: "confirmed"

Timeline:
  t=0ms    Client → Primary: INSERT
  t=0.5ms  Primary writes WAL to local disk
  t=1.0ms  Primary → Replica 1: WAL record (network: ~0.2ms)
  t=1.0ms  Primary → Replica 2: WAL record (parallel, ~0.2ms)
  t=1.5ms  Replica 1 writes WAL to disk
  t=1.6ms  Replica 2 writes WAL to disk
  t=2.0ms  Both ACKs received by Primary
  t=2.1ms  Primary commits, responds to client

Total write latency: ~2-3 ms (same-region, same-datacenter)

Compared to no replication (~0.5-1 ms), sync replication adds
~1-2 ms of latency. Still well within our 100 ms target.

But this means: if EITHER replica is slow or down, writes stall.
This is the price of zero data loss.

What This Design Gets Right

  • Single-shard balance queries — SUM(amount_cents) WHERE account_id = X hits exactly one shard. No scatter-gather. 2-5 ms latency.
  • Single-shard transaction history — Paginated query by account + date on one shard. Index scan, very fast.
  • Zero data loss — Synchronous replication to 2 replicas. Even if the primary’s disk catches fire, both replicas have the data.
  • Automatic failover — If primary dies, promote a sync replica. No data lost because replica is fully caught up.
  • Per-shard operations — Each shard is 1.7 TB. VACUUM: ~15 min. Backup: ~30 min. Recovery: ~1 hour. All manageable.

What This Design Gets Wrong

  • Cross-shard transfers — Transfer from account A (shard 5) to account B (shard 12) requires atomic debit + credit across two independent databases. This is the hardest problem.
  • Synchronous replication latency across regions — If replicas are in different regions (US → EU), each write waits for a 100-200 ms round trip. Our 100 ms write target is blown.
  • Hot shards from whale accounts — A major merchant processing 1M txns/day lands on one shard. That shard gets 10x the writes of other shards.
  • Failover coordination — When shard 7’s primary dies, we need to promote a replica, update routing, and ensure no split-brain — all without affecting the other 17 shards.

Iteration 2: Solving Cross-Shard Transfers

The most dangerous operation in a sharded payment ledger is a transfer between two accounts on different shards. Debit A and credit B must both succeed or both fail. If one succeeds and the other fails, money is either created or destroyed.

Why 2PC (Two-Phase Commit) Is Risky Here

How 2PC Would Work

Transfer $50 from Account A (Shard 5) to Account B (Shard 12):

Phase 1 — PREPARE:
  Coordinator → Shard 5: "Can you debit A by $50?"
  Coordinator → Shard 12: "Can you credit B by $50?"
  Shard 5: checks A's balance ≥ $50, locks A's row → YES
  Shard 12: locks B's row → YES

Phase 2 — COMMIT:
  Coordinator → Shard 5: COMMIT
  Coordinator → Shard 12: COMMIT
  Both release locks.

The problem: during Phase 1, rows are LOCKED.
  - If coordinator crashes between Phase 1 and Phase 2:
    both rows remain locked indefinitely.
  - No other transaction can touch accounts A or B.
  - For a payment system processing 10K txns/sec,
    locked rows cause cascading failures.

2PC Failure Scenarios

Failure 1: Coordinator crashes after collecting YES votes
  - Shard 5 and Shard 12 both voted YES and hold locks
  - They cannot commit (don't know coordinator's decision)
  - They cannot abort (coordinator might have committed)
  - Rows locked until coordinator recovers → minutes to hours
  - This is the "blocking" problem of 2PC

Failure 2: Shard 12 crashes after Shard 5 commits
  - Shard 5: debited $50 from A (committed)
  - Shard 12: never received COMMIT (crashed)
  - When Shard 12 recovers: must replay and commit
  - But if recovery takes hours, B's credit is delayed
  - A sees -$50 but B doesn't see +$50 → user panic

Failure 3: Network partition between coordinator and Shard 12
  - Coordinator sent COMMIT to both shards
  - Shard 5 received and committed
  - Shard 12 never received (partition)
  - System is in an inconsistent state until partition heals

The Saga Approach: Our Solution

Instead of trying to make two shards commit atomically (2PC), we break the transfer into a sequence of independent local transactions with compensating actions. Each step succeeds or fails independently, and if a step fails, we undo the previous steps.

Saga Step-by-Step Trace

Transfer $50 from Account A (Shard 5) to Account B (Shard 12)
Transfer ID: TXF-9001 (also the idempotency key)

Before transfer:
  Account A balance (Shard 5): $200 (sum of CONFIRMED txns)
  Account B balance (Shard 12): $75

Step 1: PENDING debit on Shard 5
  INSERT INTO transactions (txn_id, account_id, amount_cents,
    txn_type, status, counterpart, idempotency_key)
  VALUES (uuid(), A, -5000, 'TRANSFER_OUT', 'PENDING',
    B, 'TXF-9001-debit');

  A's balance is still $200 (PENDING excluded from SUM).
  A's available balance: $200 - $50 pending = $150.
  → This "hold" prevents double-spending even if another
    transfer starts before this one completes.

Step 2: PENDING credit on Shard 12
  INSERT INTO transactions (txn_id, account_id, amount_cents,
    txn_type, status, counterpart, idempotency_key)
  VALUES (uuid(), B, 5000, 'TRANSFER_IN', 'PENDING',
    A, 'TXF-9001-credit');

  B's balance still $75 (PENDING excluded).

Step 3: CONFIRM debit on Shard 5
  UPDATE transactions SET status = 'CONFIRMED'
  WHERE idempotency_key = 'TXF-9001-debit';

  A's balance: $200 - $50 = $150 ✓

Step 4: CONFIRM credit on Shard 12
  UPDATE transactions SET status = 'CONFIRMED'
  WHERE idempotency_key = 'TXF-9001-credit';

  B's balance: $75 + $50 = $125 ✓

Total time: ~8-15 ms (4 sequential DB operations, each 2-4 ms)

Handling Failures: The Compensating Actions

What if Step 2 fails? (Shard 12 is down or B's account is frozen)

  Step 1 succeeded: A has a PENDING debit of -$50.
  Step 2 failed: B never got the credit.

  Compensate Step 1:
    UPDATE transactions SET status = 'FAILED'
    WHERE idempotency_key = 'TXF-9001-debit';

  A's PENDING hold is released. Balance returns to $200.
  Transfer marked as FAILED in the orchestrator.
  No money was created or destroyed. ✓

What if Step 3 fails? (Shard 5 crashes during CONFIRM)

  Step 1: PENDING debit on A ✓
  Step 2: PENDING credit on B ✓
  Step 3: CONFIRM debit fails (Shard 5 down)

  The orchestrator retries Step 3 when Shard 5 recovers.
  The idempotency key ('TXF-9001-debit') ensures that even
  if the CONFIRM was actually applied before the crash
  (but the ACK was lost), retrying is safe — it either
  updates the row (if still PENDING) or is a no-op
  (if already CONFIRMED).

What if the ORCHESTRATOR crashes?

  The orchestrator persists the transfer state machine to a
  durable store (another DB table or a message queue).
  On restart, it reads all in-progress transfers and resumes
  from the last completed step.

  Transfer state: INITIATED → DEBIT_PENDING → CREDIT_PENDING
                  → CONFIRMING → COMPLETED

  Each state transition is persisted before executing the
  next step. Recovery = re-read state, resume from there.

Idempotency: Preventing Double-Processing

# Every step uses an idempotency key derived from the transfer_id.
# If a step is retried (after a timeout or crash recovery),
# the database rejects the duplicate insert.

def execute_pending_debit(transfer_id, account_id, amount):
    idempotency_key = f"{transfer_id}-debit"
    try:
        db.execute("""
            INSERT INTO transactions
              (txn_id, account_id, amount_cents, status,
               idempotency_key)
            VALUES (gen_random_uuid(), %s, %s, 'PENDING', %s)
        """, [account_id, -amount, idempotency_key])
        return "SUCCESS"
    except UniqueViolation:
        # Already inserted — this is a retry. Check current status.
        row = db.query("""
            SELECT status FROM transactions
            WHERE idempotency_key = %s
        """, [idempotency_key])
        if row.status == 'PENDING':
            return "SUCCESS"  # Already done, move to next step
        elif row.status == 'CONFIRMED':
            return "ALREADY_CONFIRMED"
        elif row.status == 'FAILED':
            return "ALREADY_FAILED"

The balance query must account for PENDING holds. The “confirmed balance” is SUM(amount_cents) WHERE account_id = X AND status = 'CONFIRMED'. But the “available balance” (used to decide if a new payment can proceed) must also subtract PENDING debits: confirmed_balance - SUM(ABS(amount_cents)) WHERE status = 'PENDING' AND amount_cents < 0. Without this, two concurrent transfers from the same account could both pass the balance check and overdraw the account.

Iteration 3: Multi-Region with Strong Consistency

The system must operate in 3 regions (US, EU, Asia) for low latency to global users and regulatory compliance (EU data residency under GDPR). But financial data demands strong consistency — a stale balance could allow double-spending. This creates a fundamental tension.

The Consistency vs. Latency Tradeoff

Option A: Single-Region Primary

All 18 shard primaries in US-East.
Async replicas in EU and Asia for disaster recovery.

Write from US user:
  → US Primary (2 ms) → Sync Replica US (3 ms) → done
  Total: ~5 ms ✓

Write from EU user:
  → EU app server → US Primary (120 ms cross-Atlantic RTT)
  → Sync Replica US (3 ms) → back to EU (120 ms)
  Total: ~245 ms ✗ (exceeds 100 ms target)

Write from Asia user:
  → Asia app server → US Primary (180 ms cross-Pacific RTT)
  → Sync Replica US (3 ms) → back to Asia (180 ms)
  Total: ~365 ms ✗✗ (way over target)

Reading from async replica:
  EU read from EU replica: 2-5 ms (fast!)
  But... async replica might be 50-200 ms behind primary.
  EU user checks balance → sees $200 (stale) → approves payment.
  Meanwhile, US primary already processed a $150 debit.
  True balance is $50. EU payment of $100 causes overdraft.

  ✗ Unacceptable for financial data.

Option B: Raft Consensus Across Regions

Each shard uses Raft consensus with nodes in all 3 regions.
This is the CockroachDB / Google Spanner approach.

Raft requires majority quorum: 2 of 3 regions must ACK.

Write from US:
  → US node (leader): writes locally (2 ms)
  → Sends to EU node (120 ms RTT) and Asia node (180 ms RTT)
  → Needs 1 of 2 remote ACKs (takes the faster one = EU)
  → Total: ~125 ms ✗ (slightly over 100 ms target)

Write from EU:
  → EU node (if leader): writes locally (2 ms)
  → Sends to US (120 ms) and Asia (200 ms EU→Asia)
  → Fastest remote ACK: US at 120 ms
  → Total: ~125 ms ✗

The problem: every write, no matter where, pays at least
one cross-ocean RTT (~120 ms). This is physics — the speed
of light through fiber from New York to London is ~65 ms
one way, ~130 ms round trip. No amount of engineering can
make light travel faster.

Works for CockroachDB because many workloads tolerate
125 ms writes. But our SLA is <100 ms p99.

Option C: Region-Pinned Accounts (Our Choice)

The key insight: accounts have a “home region.” A European customer’s account is pinned to the EU region. Their shard primary lives in EU. All writes for that account are local — no cross-ocean latency. Cross-region transfers use the saga from Iteration 2.

How Region Pinning Works

Account creation:
  1. User signs up from IP address in Frankfurt, Germany
  2. System determines home region: EU
  3. Shard assignment: hash(account_id) % 6 + 6
     (Shards 6-11 are EU shards)
     e.g., hash(ACC-12345) % 6 = 3 → Shard 9
  4. Account metadata stored: {account_id: ACC-12345,
     home_region: "EU", shard: 9}

Routing logic:
  Global routing table (replicated to all regions via DNS/config):
    Shard 0-5:   primary in US
    Shard 6-11:  primary in EU
    Shard 12-17: primary in Asia

  When a request arrives:
    1. Extract account_id from request
    2. Compute shard = hash(account_id) % 18
    3. Look up which region owns that shard
    4. If same region → route to local primary (2-5 ms)
    5. If different region → route cross-region (120-200 ms)

Cross-region writes (EU account accessed from Asia):
  The write MUST go to the EU primary (strong consistency).
  Latency: 200 ms cross-region RTT.
  This is rare: most users access from their home region.
  We accept higher latency for this edge case (~2-5% of traffic).

Cross-Region Transfer Example

Transfer $50 from EU Account A (Shard 9, EU) to
  Asia Account B (Shard 14, Asia)

Step 1: PENDING debit on Shard 9 (EU)
  Orchestrator (runs in EU, near A's shard)
  → Shard 9 Primary (EU): INSERT PENDING -$50
  → Sync replica (EU): replicated
  Latency: ~3 ms (local)

Step 2: PENDING credit on Shard 14 (Asia)
  Orchestrator (EU) → Shard 14 Primary (Asia)
  Network latency EU → Asia: ~200 ms RTT
  → Shard 14 Primary (Asia): INSERT PENDING +$50
  → Sync replica (Asia): replicated
  Latency: ~205 ms (cross-region)

Step 3: CONFIRM debit on Shard 9 (EU)
  → UPDATE PENDING → CONFIRMED (local)
  Latency: ~3 ms

Step 4: CONFIRM credit on Shard 14 (Asia)
  Orchestrator (EU) → Shard 14 Primary (Asia)
  Latency: ~205 ms (cross-region)

Total: ~3 + 205 + 3 + 205 = ~416 ms

This is above our 100 ms write target, but cross-region
transfers are a small fraction of total traffic (~5%).
The business accepts this tradeoff:
  - 95% of payments: local, < 5 ms ✓
  -  5% of transfers: cross-region, ~400 ms (acceptable)

Optimization: run the orchestrator in the region of the
DEBIT account (A's region). Steps 1 and 3 are local.
Only steps 2 and 4 cross regions. We can parallelize
steps 3 and 4 if we accept the (extremely rare) case
where one CONFIRM succeeds and the other needs retry.

Why not just use CockroachDB or Spanner? They solve multi-region strong consistency with Raft. But they pay the cross-region RTT on every write (not just cross-region transfers). With region-pinned accounts, 95% of writes are local and fast. We only pay the cross-region penalty for the 5% of operations that genuinely span regions. For a payment ledger with strict latency SLAs, this tradeoff is usually better.

Iteration 4: Operational Challenges

Having 54 primary/sync-replica nodes (plus 36 async DR replicas = 90 nodes total) introduces operational complexity that a single PostgreSQL never had. Each of these operations must work without downtime.

Resharding: Adding Shard 19

After 2 years, data has grown to 36 TB. Each shard now holds
~2 TB. VACUUM is getting slow again. Time to add more shards.

The problem with hash(account_id) % 18 → % 19:
  Almost every account changes shard assignment.
  Account with hash = 37: 37 % 18 = 1, but 37 % 19 = 18
  Roughly 17/18 = 94% of accounts would need to move.
  Moving 94% of 36 TB = 34 TB of data. Days of migration.

Better approach: logical shards (virtual shards)

Instead of 18 physical shards, create 256 logical shards:
  logical_shard = hash(account_id) % 256

Then map logical shards to physical shards:
  Logical shards 0-13   → Physical shard 0
  Logical shards 14-27  → Physical shard 1
  ...
  Logical shards 242-255 → Physical shard 17

To add Physical shard 18:
  Move logical shards 242-255 from Physical shard 17 → 18
  That is only 14/256 = 5.5% of data ≈ 2 TB

  Migration steps:
  1. Create Physical shard 18 with replicas (empty)
  2. Start replicating logical shards 242-255 from shard 17 → 18
  3. Once caught up, enable double-writes (write to both)
  4. Switch routing: logical 242-255 → physical shard 18
  5. Stop writes to old location on shard 17
  6. Delete migrated data from shard 17 (reclaim space)

  Zero downtime. Accounts only experience ~50 ms extra latency
  during the double-write phase (step 3).

  The 256 logical shards give us room to grow to 256 physical
  shards without ever doing a full reshard.

Replica Promotion: Shard 7 Primary Dies

Scenario: Shard 7's primary (US region) crashes at 14:32:07

Timeline:
  14:32:07.000  Primary crashes (process killed / disk failure)
  14:32:07.200  Sync Replica 1 stops receiving WAL stream
  14:32:07.200  Sync Replica 2 stops receiving WAL stream
  14:32:08.000  Health check detects: primary not responding
                (health checks every 1 sec)
  14:32:09.000  Second consecutive failure confirmed
  14:32:09.100  Shard Manager initiates failover:
                1. Compare LSN (Log Sequence Number) on both replicas
                   R1 LSN: 0/4A003F8 (last WAL position received)
                   R2 LSN: 0/4A003F8 (same — both are sync replicas)
                2. Choose R1 (arbitrary, both are equally caught up)
                3. Promote R1 to primary: pg_ctl promote
                4. R1 accepts writes
  14:32:10.000  Update shard routing table:
                Shard 7 primary: old_ip → R1_ip
  14:32:10.500  R2 reconfigured to replicate from new primary (R1)
  14:32:11.000  New primary accepting traffic

Total failover time: ~4 seconds
Data lost: 0 bytes (synchronous replication was fully caught up)

Preventing split-brain:
  What if the old primary isn't actually dead, just slow?
  (network glitch, long garbage collection pause)

  Fencing token approach:
  - Each primary holds a lease with an incrementing epoch number
  - Old primary: epoch 5. New primary (R1): epoch 6.
  - All replicas and clients only accept writes from the
    highest epoch they've seen.
  - If old primary "wakes up" and tries to write with epoch 5,
    replicas reject it: "I've seen epoch 6, you're stale."
  - Old primary must step down and become a replica.

  STONITH (Shoot The Other Node In The Head):
  - As a last resort, the shard manager sends a power-off
    command to the old primary's physical/virtual machine.
  - Guarantees it cannot issue any more writes.
  - Harsh but eliminates all split-brain risk.

Monitoring: What to Watch on 90 Nodes

Per-shard metrics (18 shards × 3 replicas = 54 sync nodes):

  Replication lag (per replica):
    - Sync replica lag should be 0 bytes / 0 ms
    - If sync lag > 0 for > 5 seconds → alert
    - Indicates replica is falling behind → write stalls imminent
    - Dashboard: lag_bytes = primary_lsn - replica_lsn

  Write latency (per shard primary):
    - p50, p95, p99 of INSERT/UPDATE latency
    - p99 > 50 ms → investigate
    - p99 > 100 ms → SLA breach alert

  Disk usage (per node):
    - Current usage vs capacity
    - Growth rate: GB/day
    - Alert at 70% → plan resharding
    - Alert at 85% → urgent resharding

  Connection count (per shard primary):
    - Active connections vs max_connections
    - Alert at 80% of max

  VACUUM progress (per shard):
    - Dead tuple count
    - Time since last successful vacuum
    - If dead tuples > 1M → vacuum is falling behind

Cross-shard transaction metrics:
  - Saga completion rate: should be > 99.9%
  - Saga average duration: p50, p95, p99
  - In-progress sagas count (stuck sagas > 60 sec = problem)
  - Compensation (rollback) rate: should be < 0.1%

Per-region metrics:
  - Cross-region query count (should be small)
  - Cross-region latency p99
  - Async DR replica lag (should be < 1 second)

Backup Strategy: Per-Shard Independent

Why per-shard backup (not cluster-wide):

  Cluster-wide consistent backup of 90 nodes requires
  freezing all writes simultaneously → downtime.
  Per-shard backup: each shard backs up independently.

  Each shard: ~1.7 TB
  pg_basebackup time: ~30 min per shard

  Schedule: staggered across shards
    00:00 - Shard 0 backup starts
    00:10 - Shard 1 backup starts
    00:20 - Shard 2 backup starts
    ...
    02:50 - Shard 17 backup starts
    03:20 - All backups complete

  Backup runs on sync replica (not primary) → zero impact
  on write performance.

  Point-in-time recovery per shard:
    1. Restore base backup (~30 min for 1.7 TB)
    2. Replay WAL from backup time to target time
    3. Shard is recovered independently
    4. Cross-shard transactions that span the recovered shard:
       - Check saga orchestrator for in-progress transfers
       - Resume any interrupted sagas
       - Idempotency keys prevent double-processing

  Recovery Time Objective (RTO): ~1 hour per shard
  Recovery Point Objective (RPO): 0 (synchronous replication
    means the surviving replica has all committed data)

Hot shard problem: A single merchant (e.g., a large e-commerce platform) processing 1M transactions/day through one account creates a hot shard. That shard receives ~12 writes/sec from this one account alone, but more critically, the account’s balance query must scan 1M/365 ≈ 2,740 rows per day × 365 days = ~1M rows. Solutions: (1) pre-computed running balance in a separate table (updated on each write), (2) sharding within an account by date (monthly sub-accounts that are summed for balance), (3) rate limiting per account to cap writes at a sustainable level.

Final Design

Putting all four iterations together into the complete architecture.

Design Decision Summary

DecisionChoiceWhyAlternative Rejected
Shard key hash(account_id) % 256 (logical shards) Balance query is single-shard. Transaction history is single-shard. No scatter-gather for the two most common reads. hash(txn_id) — would require 18-shard scatter-gather for every balance check.
Replication mode Synchronous (same-region) Zero data loss for financial records. Write latency only +1-2 ms within same datacenter. Async replication — risks losing committed transactions on primary failure.
Cross-shard transfers Saga with compensating actions No cross-shard locks. Each step is a local transaction. Idempotent retries. Handles failures gracefully. 2PC — coordinator is SPOF, holds locks across shards, blocking on coordinator failure.
Multi-region strategy Region-pinned accounts 95% of writes are local (<5 ms). Only cross-region transfers pay RTT penalty. GDPR compliance (EU data stays in EU). Raft across regions — every write pays 120+ ms RTT. Spanner — requires atomic clocks (TrueTime).
Failover mechanism Raft-based leader election per shard Automatic failover in ~4 sec. Fencing tokens prevent split-brain. No human intervention needed. Manual failover — requires on-call response, 5-15 min minimum downtime.
Resharding 256 logical shards mapped to 18 physical Adding a physical shard only moves ~5% of data. No full reshard ever needed. hash % N directly — adding 1 shard reshuffles ~94% of data.

Code: Shard Router Implementation

import hashlib
from typing import Dict, Tuple

class ShardRouter:
    """Routes account operations to the correct shard and region."""

    NUM_LOGICAL_SHARDS = 256

    def __init__(self):
        # Logical shard → (physical shard, region, primary_host)
        self.routing_table: Dict[int, Tuple[int, str, str]] = {}
        self._build_routing_table()

    def _build_routing_table(self):
        """Map 256 logical shards to 18 physical shards across 3 regions."""
        # ~14 logical shards per physical shard (256 / 18 ≈ 14.2)
        for logical in range(256):
            if logical < 86:        # 0-85 → US (shards 0-5)
                physical = logical // 15  # ~14-15 logical per physical
                region = "us-east-1"
            elif logical < 172:     # 86-171 → EU (shards 6-11)
                physical = 6 + (logical - 86) // 15
                region = "eu-west-1"
            else:                   # 172-255 → Asia (shards 12-17)
                physical = 12 + (logical - 172) // 14
                region = "ap-southeast-1"

            primary_host = f"pg-shard-{physical}-primary.{region}.internal"
            self.routing_table[logical] = (physical, region, primary_host)

    def get_shard(self, account_id: int) -> Tuple[int, str, str]:
        """Return (physical_shard, region, primary_host) for an account."""
        # Consistent hash to logical shard
        hash_val = int(hashlib.sha256(
            str(account_id).encode()
        ).hexdigest(), 16)
        logical_shard = hash_val % self.NUM_LOGICAL_SHARDS
        return self.routing_table[logical_shard]

    def route_query(self, account_id: int, query_type: str):
        """Route a query to the correct shard primary."""
        physical, region, host = self.get_shard(account_id)
        print(f"Account {account_id} → Logical shard "
              f"{int(hashlib.sha256(str(account_id).encode()).hexdigest(), 16) % 256}"
              f" → Physical shard {physical} ({region})")
        return host

# Example usage:
router = ShardRouter()

# Balance query: single-shard
router.route_query(12345, "balance")
# → Account 12345 → Logical shard 42 → Physical shard 2 (us-east-1)

# Transaction history: same shard
router.route_query(12345, "history")
# → Same shard — no cross-shard needed

# Transfer: may cross shards
src_shard = router.get_shard(12345)   # Shard 2, US
dst_shard = router.get_shard(67890)   # Shard 14, Asia
if src_shard[0] != dst_shard[0]:
    print("Cross-shard transfer — use saga orchestrator")

Code: Saga Transfer Orchestrator

import enum
import uuid
from dataclasses import dataclass
from typing import Optional

class TransferState(enum.Enum):
    INITIATED = "INITIATED"
    DEBIT_PENDING = "DEBIT_PENDING"
    CREDIT_PENDING = "CREDIT_PENDING"
    CONFIRMING = "CONFIRMING"
    COMPLETED = "COMPLETED"
    COMPENSATING = "COMPENSATING"
    FAILED = "FAILED"

@dataclass
class Transfer:
    transfer_id: str
    src_account: int
    dst_account: int
    amount_cents: int
    state: TransferState
    src_shard: int
    dst_shard: int

class TransferOrchestrator:
    def __init__(self, router: ShardRouter, state_store):
        self.router = router
        self.state_store = state_store  # durable storage for saga state

    def execute_transfer(self, src: int, dst: int, amount: int) -> str:
        transfer_id = f"TXF-{uuid.uuid4().hex[:8]}"
        transfer = Transfer(
            transfer_id=transfer_id,
            src_account=src, dst_account=dst,
            amount_cents=amount,
            state=TransferState.INITIATED,
            src_shard=self.router.get_shard(src)[0],
            dst_shard=self.router.get_shard(dst)[0],
        )
        # Persist initial state (survives orchestrator crash)
        self.state_store.save(transfer)

        try:
            # Step 1: PENDING debit
            self._pending_debit(transfer)
            transfer.state = TransferState.DEBIT_PENDING
            self.state_store.save(transfer)

            # Step 2: PENDING credit
            self._pending_credit(transfer)
            transfer.state = TransferState.CREDIT_PENDING
            self.state_store.save(transfer)

            # Step 3 + 4: CONFIRM both
            transfer.state = TransferState.CONFIRMING
            self.state_store.save(transfer)
            self._confirm_debit(transfer)
            self._confirm_credit(transfer)

            transfer.state = TransferState.COMPLETED
            self.state_store.save(transfer)
            return transfer_id

        except Exception as e:
            # Compensate: undo completed steps
            self._compensate(transfer)
            transfer.state = TransferState.FAILED
            self.state_store.save(transfer)
            raise

    def _pending_debit(self, t: Transfer):
        """Insert PENDING debit on source shard."""
        host = self.router.get_shard(t.src_account)[2]
        db = connect(host)
        db.execute("""
            INSERT INTO transactions
              (txn_id, account_id, amount_cents, txn_type,
               status, counterpart, idempotency_key)
            VALUES (gen_random_uuid(), %s, %s, 'TRANSFER_OUT',
                    'PENDING', %s, %s)
        """, [t.src_account, -t.amount_cents,
              t.dst_account, f"{t.transfer_id}-debit"])

    def _pending_credit(self, t: Transfer):
        """Insert PENDING credit on destination shard."""
        host = self.router.get_shard(t.dst_account)[2]
        db = connect(host)
        db.execute("""
            INSERT INTO transactions
              (txn_id, account_id, amount_cents, txn_type,
               status, counterpart, idempotency_key)
            VALUES (gen_random_uuid(), %s, %s, 'TRANSFER_IN',
                    'PENDING', %s, %s)
        """, [t.dst_account, t.amount_cents,
              t.src_account, f"{t.transfer_id}-credit"])

    def _confirm_debit(self, t: Transfer):
        host = self.router.get_shard(t.src_account)[2]
        db = connect(host)
        db.execute("""
            UPDATE transactions SET status = 'CONFIRMED',
              updated_at = NOW()
            WHERE idempotency_key = %s AND status = 'PENDING'
        """, [f"{t.transfer_id}-debit"])

    def _confirm_credit(self, t: Transfer):
        host = self.router.get_shard(t.dst_account)[2]
        db = connect(host)
        db.execute("""
            UPDATE transactions SET status = 'CONFIRMED',
              updated_at = NOW()
            WHERE idempotency_key = %s AND status = 'PENDING'
        """, [f"{t.transfer_id}-credit"])

    def _compensate(self, t: Transfer):
        """Undo completed steps in reverse order."""
        if t.state.value >= TransferState.CREDIT_PENDING.value:
            # Undo credit
            host = self.router.get_shard(t.dst_account)[2]
            db = connect(host)
            db.execute("""
                UPDATE transactions SET status = 'FAILED'
                WHERE idempotency_key = %s AND status = 'PENDING'
            """, [f"{t.transfer_id}-credit"])

        if t.state.value >= TransferState.DEBIT_PENDING.value:
            # Undo debit
            host = self.router.get_shard(t.src_account)[2]
            db = connect(host)
            db.execute("""
                UPDATE transactions SET status = 'FAILED'
                WHERE idempotency_key = %s AND status = 'PENDING'
            """, [f"{t.transfer_id}-debit"])

Key Takeaways

When You Need Both Sharding AND Replication

  • Sharding is for capacity — when data is too large for one machine (storage) or write throughput exceeds one machine (I/O). Our 18 TB + 10K writes/sec required 18 shards.
  • Replication is for availability and durability — when a single copy is not safe enough. Our 99.999% SLA and zero-data-loss requirement demanded 3 copies per shard.
  • Combining them multiplies complexity — you inherit all challenges of each technique, plus new combined challenges. This is not 1 + 1 = 2; it is 1 + 1 = 5 in terms of operational burden.

Challenges Inherited from Each

  • From sharding: cross-shard queries (solved with shard key choice), hot shards (solved with account-level rate limiting or sub-sharding), resharding (solved with 256 logical shards), shard key selection (hash(account_id) for our access patterns).
  • From replication: replication lag (solved with synchronous replication, accepting the latency cost), consistency tradeoffs (solved by reading only from primaries for balance), split-brain (solved with fencing tokens and STONITH), failover complexity (solved with Raft-based leader election).
  • New combined challenges: cross-shard transactions with replicated shards (saga pattern), coordinated failover across 54+ nodes (per-shard Raft, independent), region-pinned resharding (move logical shards between physical shards within a region), monitoring 90 nodes (per-shard + per-replica + cross-shard metrics).

The Numbers That Drove Every Decision

Storage:
  18 TB raw → 30 TB with indexes → 18 shards × 1.7 TB each
  Rule: keep PostgreSQL under 1-2 TB per instance

Throughput:
  10K peak writes/sec ÷ 18 shards = 556 writes/shard
  Single PG capacity: ~5K writes/sec → 9x headroom per shard

Availability:
  Single node: 99.9% → 8.7h downtime/year
  3 replicas per shard: 99.999% → 5 min downtime/year
  Failover time: ~4 seconds with Raft + fencing tokens

Latency:
  Local write: 2-5 ms (sync replication adds ~1-2 ms)
  Balance query: 2-5 ms (single-shard index scan)
  Cross-region transfer: 250-420 ms (4 saga steps, 2 cross-ocean)

Total nodes: 18 shards × 3 sync + 18 × 2 async DR = 90 nodes
Total storage footprint: 30 TB × 5 copies = 150 TB

Design Principles Applied

  • Choose the shard key based on access patterns, not data distribution. hash(account_id) gives uneven distribution (some accounts are bigger) but makes the two most critical queries (balance + history) single-shard.
  • Use synchronous replication sparingly. Same-region sync replicas: yes (adds 1-2 ms, guarantees zero data loss). Cross-region sync replicas: no (adds 120+ ms per write, kills latency).
  • Avoid distributed transactions when possible. Sagas with idempotency keys are more resilient than 2PC. Accept eventual consistency for cross-shard operations (PENDING → CONFIRMED is eventually atomic, not instantly atomic).
  • Plan for resharding from day one. 256 logical shards cost almost nothing upfront but save weeks of downtime when you inevitably need to add physical shards.
  • Pin data to regions for compliance and latency. Region-pinned accounts solve both GDPR data residency and write latency in one design decision.
