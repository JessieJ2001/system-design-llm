Thundering Herd After a Cache Node Failure

One of 100 Redis nodes dies. 1% of the keyspace goes cold. The database, which was comfortably handling 5K reads/sec, is suddenly hit with 50K reads/sec. Walk through the mechanics of the thundering herd and the layered mitigations that tame it.

NALSD Walkthrough • ~15 min read

The Scenario

You operate a read-heavy web application backed by a distributed cache cluster and a PostgreSQL database.

The Architecture

Steady-State Numbers

MetricValue
Total read traffic500,000 reads/sec
Cache hit rate99%
Reads served from cache495,000/sec
Cache misses (hit DB)5,000/sec
DB read capacity20,000 queries/sec (4 replicas x 5K each)
DB utilization5,000 / 20,000 = 25%
Redis nodes100
Keys per node~100,000 (10M total / 100 nodes)
Cache key TTL5 minutes (300 seconds)

The system is healthy. DB is at 25% utilization with comfortable headroom.

The Incident: One Node Dies

At 14:23, Redis node #47 crashes due to a hardware failure. It held approximately 100,000 keys (1% of the total keyspace). Those keys are now gone. Via consistent hashing, requests for those keys are routed to the next node in the ring, which has none of them cached. Every request for those 100K keys is now a cache miss.

Back-of-Envelope: Why "1% More Misses" Is Misleading

The Naive Calculation (Wrong)

1% of keys are uncached.
If traffic is uniformly distributed:
  Additional cache misses = 1% of 500,000 = 5,000/sec
  New DB load = 5,000 (normal misses) + 5,000 (lost node) = 10,000/sec
  DB capacity = 20,000/sec
  DB utilization = 10,000 / 20,000 = 50%

This seems fine. The DB has plenty of headroom.
But this calculation is WRONG.

The Thundering Herd (What Actually Happens)

The naive calculation assumes the 5,000 additional misses arrive smoothly, spread evenly across each second. In reality, they arrive in a burst.

Why It Bursts

Before the node died, all 100,000 keys were cached. Zero requests for those keys hit the DB. At the instant the node dies:

  • Every app server that requests any of those 100K keys simultaneously discovers a cache miss.
  • There is no coordination between app servers — they all independently decide to fetch from the DB.
  • For a single popular key that's requested 200 times per second, all 200 requests hit the DB simultaneously in the first second.

The Duplicate Query Problem

Consider one hot key: "product:12345" (a popular product page)
  Normal: served from cache, 200 requests/sec, 0 DB queries

  Node dies at T=0:
  T=0.000s: Request from App Server #1: cache miss → query DB
  T=0.001s: Request from App Server #2: cache miss → query DB
  T=0.002s: Request from App Server #3: cache miss → query DB
  ...
  T=0.005s: App Server #1's DB query returns. Writes result to cache.
  T=0.005s+: Subsequent requests find the key in cache again.

  In those 5 milliseconds: ~1 request arrived per millisecond = 1 DB query each.
  Total DB queries for this ONE key in the first 5ms: ~5

  That's not terrible for one key. But there are 100,000 keys.

The Aggregate Burst

Key distribution follows Zipf's law (a few keys are very hot):
  Top 1% of keys (1,000 keys): ~50% of traffic for those keys
  Next 9% (9,000 keys): ~30% of traffic
  Bottom 90% (90,000 keys): ~20% of traffic

Traffic to lost keys: 500,000/sec * 1% = 5,000 req/sec
  Hot keys (1,000): 2,500 req/sec for 1,000 keys
    Average: 2.5 req/sec per key
    But the hottest key might get 200 req/sec

In the FIRST SECOND after the node dies:
  - All 5,000 req/sec for lost keys are cache misses
  - For each key, MULTIPLE requests hit the DB before any one
    request can fetch the result and repopulate the cache
  - A DB query takes ~5ms. In 5ms, a key requested 200/sec
    gets 1 duplicate query.
  - A key requested 50/sec gets ~0.25 duplicates (sometimes 1)

Total DB queries in first second (estimated):
  5,000 unique key misses + ~2,000-10,000 duplicate queries
  = 7,000 - 15,000 queries in the first second

In the FIRST 100ms (the real danger zone):
  The burst is even more concentrated because all misses happen
  at once, not spread across the second.
  Peak instantaneous rate: potentially 20,000-50,000 queries/sec

The DB Perspective

Steady state: 5,000 queries/sec, 25% utilization, 5ms avg latency

Burst hits:
  T=0s:     DB receives 20,000-50,000 queries/sec
  T=0.01s:  Connection pools start filling
  T=0.1s:   DB query latency rises from 5ms to 50ms (10x)
            (disk I/O saturated, buffer cache thrashing)
  T=0.5s:   Query latency at 200ms. Some queries timing out.
  T=1s:     Timeouts cause retries. Retries add MORE load.
            DB query latency: 500ms+
  T=2s:     Connection pools on app servers exhausted.
            New requests can't get a DB connection.
            Even CACHED requests might fail if they need any DB call.
  T=5s:     Cascading failure in progress.
The key insight: The thundering herd problem is not about the total number of additional cache misses. It's about the concurrency of identical queries. When 200 app servers all discover the same cache miss at the same moment, they all make the same DB query simultaneously. 200 identical queries return the same row, but each one consumes DB resources (CPU, I/O, connections).

The Naive Design: Cache-Aside with No Coordination

The Code

# Standard cache-aside pattern — no thundering herd protection
def get_product(product_id):
    # Step 1: Check cache
    cached = redis.get(f"product:{product_id}")
    if cached:
        return json.loads(cached)

    # Step 2: Cache miss — query DB
    product = db.query("SELECT * FROM products WHERE id = %s", product_id)

    # Step 3: Write to cache for next time
    redis.setex(f"product:{product_id}", 300, json.dumps(product))

    return product

This code is correct for a single request. The problem is what happens when 200 concurrent requests all execute Step 1 at the same time, all find a cache miss, and all proceed to Step 2 simultaneously. There is no coordination between them.

The Wasted Work

Timeline for one hot key ("product:12345"):

App Server 1:  [cache miss] → [DB query.......] → [write cache] → done
App Server 2:  [cache miss] → [DB query.......] → [write cache] → done
App Server 3:  [cache miss] → [DB query.......] → [write cache] → done
App Server 4:  [cache miss] → [DB query.......] → [write cache] → done
...            ...
App Server 50: [cache miss] → [DB query.......] → [write cache] → done

All 50 queries return the SAME data. 49 of them were wasted.
The cache is written 50 times with the same value. 49 writes were wasted.

Iteration 1: Request Coalescing (Single-Flight)

The idea: when multiple requests need the same cache key at the same time, only one of them should query the database. The others should wait for that one query to complete, then share the result.

How It Works

Step-by-step for key "product:12345":

T=0ms:  Request A arrives. Cache miss.
        A checks the in-flight map: is anyone fetching "product:12345"?
        No. A registers itself as the fetcher. Starts DB query.

T=1ms:  Request B arrives. Cache miss.
        B checks in-flight map: yes, A is fetching "product:12345".
        B subscribes to A's result. Does NOT query DB. Waits.

T=2ms:  Request C arrives. Same story. Subscribes to A's result.

T=3ms:  Requests D, E arrive. Subscribe to A's result.

T=5ms:  A's DB query returns the product data.
        A writes result to cache.
        A broadcasts the result to B, C, D, E.
        All five requests return simultaneously.

DB queries made: 1 (instead of 5)
Total time for all requests: 5ms (instead of 5 queries x 5ms each)

Implementation

import threading
import time

class SingleFlight:
    """Ensures only one in-flight request per key."""

    def __init__(self):
        self.lock = threading.Lock()
        self.in_flight = {}  # key -> Event + result

    def do(self, key, fetch_fn):
        """
        If no one is fetching `key`, call fetch_fn() and share the result.
        If someone is already fetching `key`, wait for their result.
        """
        with self.lock:
            if key in self.in_flight:
                # Someone else is already fetching this key
                entry = self.in_flight[key]
            else:
                # We're the first — register ourselves
                entry = {"event": threading.Event(), "result": None, "error": None}
                self.in_flight[key] = entry
                # Release lock, then fetch
                # (we must release the lock before the slow DB call)

        if entry["event"].is_set() or key not in self.in_flight:
            # Result already available (race condition edge case)
            entry["event"].wait()
            if entry["error"]:
                raise entry["error"]
            return entry["result"]

        # Check if WE are the fetcher (we created the entry)
        is_fetcher = False
        with self.lock:
            if self.in_flight.get(key) is entry and not entry["event"].is_set():
                is_fetcher = True

        if is_fetcher:
            try:
                result = fetch_fn()
                entry["result"] = result
            except Exception as e:
                entry["error"] = e
            finally:
                entry["event"].set()  # Wake up all waiters
                with self.lock:
                    del self.in_flight[key]

            if entry["error"]:
                raise entry["error"]
            return entry["result"]
        else:
            # We're a waiter
            entry["event"].wait(timeout=5.0)
            if entry["error"]:
                raise entry["error"]
            return entry["result"]


# Usage
flight = SingleFlight()

def get_product(product_id):
    # Check cache first
    cached = redis.get(f"product:{product_id}")
    if cached:
        return json.loads(cached)

    # Cache miss — use single-flight to coalesce DB queries
    def fetch_from_db():
        product = db.query("SELECT * FROM products WHERE id = %s", product_id)
        redis.setex(f"product:{product_id}", 300, json.dumps(product))
        return product

    return flight.do(f"product:{product_id}", fetch_from_db)

Impact

Without coalescing:
  100,000 lost keys
  Each key: average 1-5 duplicate DB queries in first second
  Total DB queries: 5,000 + 5,000-25,000 duplicates = 10,000-30,000/sec
  DB is overwhelmed (capacity: 20,000/sec)

With coalescing (per app server):
  Each app server only sends 1 query per key
  50 app servers × 1 query per key = 50 queries per key
  Wait — coalescing is PER APP SERVER, not global!

  A hot key requested 200/sec across 50 servers = 4 req/sec/server
  Each server coalesces to 1 query per 5ms window = ~1 query
  50 servers × 1 query = 50 DB queries per key (not 200)

  Still a 4x reduction, but not as dramatic as it first seems.

  For ALL 100K keys in the first second:
    Without: ~30,000 DB queries/sec (burst)
    With per-server coalescing: ~7,000-10,000 DB queries/sec

  10,000/sec is within DB capacity (20,000/sec). Situation is stable!

Limitation: Per-Server vs Global Coalescing

The implementation above runs on each app server independently. If you have 50 app servers, a popular key still gets 50 DB queries (one per server). For truly global coalescing, you need a centralized mechanism:

# Global coalescing using Redis as a distributed lock
def get_product_global_coalesce(product_id):
    key = f"product:{product_id}"
    lock_key = f"lock:{key}"

    # Check cache
    cached = redis.get(key)
    if cached:
        return json.loads(cached)

    # Try to acquire the fetch lock
    acquired = redis.set(lock_key, "1", nx=True, ex=5)  # NX = only if not exists

    if acquired:
        # We won the lock — we fetch from DB
        try:
            product = db.query("SELECT * FROM products WHERE id = %s", product_id)
            redis.setex(key, 300, json.dumps(product))
            return product
        finally:
            redis.delete(lock_key)
    else:
        # Someone else is fetching — poll the cache
        for _ in range(50):  # Try for 500ms
            time.sleep(0.01)  # 10ms
            cached = redis.get(key)
            if cached:
                return json.loads(cached)

        # Timeout — fetch from DB ourselves (fallback)
        product = db.query("SELECT * FROM products WHERE id = %s", product_id)
        redis.setex(key, 300, json.dumps(product))
        return product

With global coalescing, each key generates exactly 1 DB query regardless of how many servers need it. 100,000 unique keys = 100,000 DB queries total over the warm-up period, instead of millions.

Iteration 2: Probabilistic Early Expiration

Request coalescing handles the burst after a node failure. But there's another thundering herd scenario: mass TTL expiration. If many keys were written at the same time, they all expire at the same time, causing a synchronized stampede.

The Problem: Synchronized Expiry

Suppose a cache warm-up job runs at midnight, writing 1M keys with TTL=300s.
All 1M keys expire at 12:05:00 AM simultaneously.
At 12:05:00, every request for those keys is a cache miss.

This is the same thundering herd problem as a node failure,
except it's 10x worse (1M keys instead of 100K).

Solution: Jittered TTLs

Add randomness to the TTL so keys expire at different times:

import random

def cache_set_with_jitter(key, value, base_ttl=300):
    """
    Instead of all keys expiring at exactly base_ttl seconds,
    spread expirations over a range.

    base_ttl = 300s
    jitter range: 0.8 to 1.2 (±20%)
    actual TTL: between 240s and 360s
    """
    jitter = 0.8 + random.random() * 0.4  # 0.8 to 1.2
    actual_ttl = int(base_ttl * jitter)
    redis.setex(key, actual_ttl, json.dumps(value))
Without jitter:
  1M keys expire at T=300s: all 1M keys miss simultaneously
  DB spike: 1M queries in the first few seconds

With ±20% jitter:
  Keys expire between T=240s and T=360s
  Spread over 120 seconds
  Rate: 1M / 120s ≈ 8,333 misses/sec (smooth, manageable)

Probabilistic Early Recomputation (XFetch)

An even smarter approach: instead of waiting for the key to expire and then having a thundering herd, proactively refresh it before it expires. The trick: not every request refreshes early — only a probabilistic subset.

import math
import random
import time

def xfetch_get(key, fetch_fn, base_ttl=300, beta=1.0):
    """
    XFetch: Probabilistic early recomputation.

    As the key approaches its TTL, an increasing probability of
    requests will trigger a background refresh BEFORE expiry.
    This ensures the key is always warm.

    beta controls aggressiveness: higher = refresh earlier.
    """
    cached = redis.get(key)
    if cached is None:
        # True cache miss — must fetch
        value, compute_time = _timed_fetch(fetch_fn)
        expiry = time.time() + base_ttl
        redis.setex(key, base_ttl, json.dumps({
            "value": value,
            "expiry": expiry,
            "compute_time": compute_time
        }))
        return value

    entry = json.loads(cached)
    ttl_remaining = entry["expiry"] - time.time()
    compute_time = entry["compute_time"]

    # The XFetch formula:
    # Recompute if: current_time - (compute_time * beta * ln(random())) >= expiry
    # Equivalent: refresh if ttl_remaining < compute_time * beta * ln(1/random())
    should_refresh = ttl_remaining < compute_time * beta * math.log(1 / random.random())

    if should_refresh:
        # Refresh in background (don't block this request)
        threading.Thread(target=_background_refresh,
                        args=(key, fetch_fn, base_ttl)).start()

    return entry["value"]

def _timed_fetch(fetch_fn):
    start = time.time()
    result = fetch_fn()
    compute_time = time.time() - start
    return result, compute_time

def _background_refresh(key, fetch_fn, base_ttl):
    value, compute_time = _timed_fetch(fetch_fn)
    expiry = time.time() + base_ttl
    redis.setex(key, base_ttl, json.dumps({
        "value": value,
        "expiry": expiry,
        "compute_time": compute_time
    }))

How XFetch Works Intuitively

Key with TTL=300s, compute_time=5ms:

TTL remaining  | Probability of refresh per request
300s           | ~0.000%  (key was just written, no need to refresh)
200s           | ~0.000%  (still plenty of time)
100s           | ~0.001%  (very unlikely)
 10s           | ~0.05%   (slight chance)
  1s           | ~0.5%    (some requests will trigger refresh)
  0.1s         | ~5%      (likely refreshed by now)

With 500 requests/sec for this key:
  At 1s remaining: 500 * 0.005 = ~2.5 requests trigger refresh/sec
  One of them succeeds (single-flight prevents duplicates)
  Key is refreshed BEFORE it expires

Result: zero cache misses, zero thundering herd for TTL expiry

Iteration 3: Circuit Breaker on the Database

Even with coalescing and jittered TTLs, an unexpected surge might overwhelm the database. The last line of defense: a circuit breaker that protects the DB by stopping queries when it's overloaded.

Why Protect the Database?

When the DB is at 90% capacity:
  Query latency: ~50ms (10x normal)
  Adding more queries pushes it to 100%
  At 100%: latency spikes to 500ms+
  At 110% (more queries than it can handle):
    Connection pool exhausts → queries queue → timeouts → retries → more load
    This is the death spiral.

The circuit breaker breaks the cycle:
  DB latency > 50ms (2x normal)? → Trip the breaker.
  Stop sending new queries. Return degraded results.
  DB load drops. DB recovers. Breaker half-opens. Test. Resume.

What to Return When the DB Breaker Is Open

StrategyHow It WorksTrade-off
Stale cache dataKeep expired cache entries in a "stale" tier. Serve them when DB is unavailable.Data might be minutes/hours old
Default valuesReturn a pre-defined default (e.g., "product not found" or generic product info)Bad user experience
Secondary data storeRead from a less-fresh but always-available store (e.g., a search index)Eventual consistency
Graceful error"This page is temporarily unavailable. Try again in a few seconds."Honest but not ideal

Stale-While-Revalidate Pattern

def get_product_with_stale_fallback(product_id):
    key = f"product:{product_id}"
    stale_key = f"stale:{key}"

    # Check fresh cache
    cached = redis.get(key)
    if cached:
        return json.loads(cached), "fresh"

    # Cache miss — check if DB breaker allows queries
    if not db_breaker.allow_request():
        # DB is overloaded — serve stale data if available
        stale = redis.get(stale_key)
        if stale:
            return json.loads(stale), "stale"
        return None, "unavailable"

    # Fetch from DB (with single-flight coalescing)
    try:
        product = flight.do(key, lambda: db.query(
            "SELECT * FROM products WHERE id = %s", product_id
        ))

        # Write to fresh cache (5 min TTL)
        redis.setex(key, 300, json.dumps(product))
        # Write to stale cache (24 hour TTL — backup for emergencies)
        redis.setex(stale_key, 86400, json.dumps(product))

        db_breaker.record_success()
        return product, "fresh"
    except Exception as e:
        db_breaker.record_failure()

        # Fallback to stale
        stale = redis.get(stale_key)
        if stale:
            return json.loads(stale), "stale"
        return None, "error"

The stale cache uses a much longer TTL (24 hours vs 5 minutes). Under normal operation, the fresh cache handles everything. When the DB is under stress, stale data keeps the site functional. Users see slightly outdated product info (which is better than a broken page).

Cache Warm-Up: How Long Until Normal?

After the node dies, a new node takes its place (or the keys are redistributed to other nodes). How long until all 100K keys are back in cache?

Passive Warm-Up (Wait for Requests)

Traffic to lost keys: 5,000 unique-key requests/sec
  (5,000 req/sec for 100K keys, but many are duplicates)

Unique keys populated per second:
  Depends on request distribution (Zipf's law).

  First second: the hottest ~2,000 keys are requested and cached.
  By 10 seconds: ~10,000 keys cached (the popular ones).
  By 60 seconds: ~30,000 keys cached.
  By 200 seconds: ~80,000 keys cached.
  The long tail (rarely accessed keys): might take 10+ minutes.

  Practical impact:
    - After 10 seconds: 90% of TRAFFIC is served from cache
      (because hot keys are cached first, and they handle most traffic)
    - After 60 seconds: 99% of traffic from cache
    - Full warm-up: 3-5 minutes for all keys

Active Warm-Up (Pre-Load Keys)

# If you maintain a list of hot keys, pre-load them into the new node
def warm_up_cache(lost_keys, batch_size=100):
    """
    Pre-fetch hot keys from the DB and load them into cache.
    Do this in batches to avoid overwhelming the DB.
    """
    for i in range(0, len(lost_keys), batch_size):
        batch = lost_keys[i:i + batch_size]

        # Batch query: fetch 100 products at once (1 DB query, not 100)
        products = db.query(
            "SELECT * FROM products WHERE id IN %s",
            tuple(batch)
        )

        # Write all to cache
        pipeline = redis.pipeline()
        for product in products:
            key = f"product:{product['id']}"
            pipeline.setex(key, 300, json.dumps(product))
        pipeline.execute()

        # Rate limit: don't overwhelm the DB
        time.sleep(0.01)  # 10ms between batches

# 100K keys / 100 per batch = 1,000 batches
# 1,000 batches * 10ms = 10 seconds to warm up
# DB queries: 1,000 batch queries (very manageable)

Key Takeaways

  1. The thundering herd is a concurrency problem, not a volume problem. 5,000 extra cache misses per second sounds manageable. But when those misses all happen simultaneously and generate duplicate DB queries, the burst can be 5-10x the steady-state rate.
  2. Request coalescing is the most impactful mitigation. Ensuring only one DB query per key per cache miss eliminates the duplicate query problem. Per-server coalescing is simple to implement; global coalescing (via distributed locks) is more effective but more complex.
  3. Jittered TTLs prevent synchronized expiry. Add randomness to cache TTLs so keys don't all expire at the same instant. A 20% jitter window spreads the load over time.
  4. XFetch (probabilistic early refresh) eliminates TTL-based stampedes entirely. By refreshing keys before they expire, you avoid cache misses for popular keys altogether.
  5. Stale-while-revalidate is your safety net. Keep a backup copy of cached data with a long TTL. When the DB is overloaded, serve stale data instead of adding more load. Stale data is better than no data.
  6. Circuit-break the database. The DB is a shared resource. When it's under stress, stop sending it more queries. Protect it so it can recover.
  7. Warm-up time follows Zipf's law. The hottest 10% of keys handle 90% of traffic. After a node failure, the most impactful keys are re-cached within seconds. Full warm-up takes minutes, but user impact is minimal after the first few seconds.
  8. The best time to add thundering herd protection is before you need it. By the time you're in a 3 AM incident, it's too late to add coalescing. Build these patterns into your cache-aside implementation from day one.
