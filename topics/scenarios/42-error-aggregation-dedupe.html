Designing Error Aggregation & Dedupe (Sentry-style)

100 B error events/day (1.16 M/sec avg, 5 M/sec peak), 50 k tenants with 10%+ volume skew, ~50 M steady-state grouping fingerprints ("issues"), per-customer storage quotas, dashboard p95 ≤ 1 s. We start from a naive "write every raw event to one Postgres table and GROUP BY fingerprint on demand" — the LC #12 Logger Rate Limiter kernel sits right inside it — and iterate to a Kafka-buffered, fingerprint-at-ingest pipeline with an issues rollup store, per-customer token-bucket rate limiting + Count-Min sketches for per-issue rates, per-customer issue-count and tag-cardinality quotas with a negative cache, stack-trace canonicalization, and a ClickHouse (Snuba-style) analytical query layer.

NALSD Walkthrough • ~50 min read • LC #12 (Logger Rate Limiter)

LC kernel. Logger Rate Limiter (LC #12). Textbook structure: a hashmap last_printed[message] = timestamp; shouldPrintMessage(ts, msg) returns true iff msg hasn't been printed in the last 10 seconds. Insert is O(1); the suppression decision is O(1) hashmap lookup. Three properties that break at scale: (1) the key space is unbounded — a malformed release can spray millions of distinct fingerprints in minutes; the hashmap grows without limit and a counter-per-key is infeasible at ~50 M issues. (2) The hashmap is shared — one noisy customer doing 10%+ of total volume floods the consumers and evicts/starves everyone else's rate-limit state if the state is global. (3) The "message" isn't a stable string — line numbers shift between releases, anonymous functions and minified frames have no name, ANR/OOM stacks are truncated — so a naive hash(stacktrace) fragments one logical bug into thousands of issues, inflating the issue count, hiding release regressions, and burying signal in noise. The production cousin replaces the shared hashmap with a Kafka-buffered fingerprint-at-ingest pipeline + per-customer token buckets + per-customer Count-Min sketches for per-issue rates without a counter per fingerprint; bounds cardinality with per-customer issue-count and tag-cardinality quotas + a negative cache for known-bad fingerprints; replaces hash(stacktrace) with a canonicalized grouping fingerprint; and serves the ranked / time-windowed / group-by dashboard from a ClickHouse analytical layer.

How this differs from #30 (general log ingest): #30 ingests and indexes a firehose of opaque log lines for ad-hoc search. #42 is about dedupe — collapsing a firehose into a bounded set of grouped issues — and per-tenant fairness under quota: a noisy tenant or a malformed release must not starve everyone or blow the shared issues table. The fingerprint algorithm, the per-customer rate limiting, the issue-count cap, and the Count-Min sketch are all #42-specific moves that #30 doesn't have.

S0. Scenario & Requirements

A crash/error reporting SaaS (Sentry / Bugsnag / Rollbar tier): customer applications run an SDK that captures unhandled exceptions, ANRs, OOMs, and manual captureException calls and ships them as raw error events (stack trace + breadcrumbs + context). The platform groups events into a bounded set of issues by a grouping fingerprint, rate-limits per customer against their paid-tier quota, and surfaces signal not noise: per-issue event counts, first-seen / last-seen, affected-user counts, and "this release regressed" alerts. The product collapses if either side breaks: either the firehose overwhelms ingest and events are silently lost (write path failed), or the dashboard takes 8 seconds to list a busy project's issues (read path failed), or one customer's flood starves everyone else (fairness failed).

System class

Class: stream-aggregation-shaped — "ingest stream X, maintain bounded aggregate view Y, answer queries against Y" capabilities (cite functional-requirements.html#class-aggregation); the FRs are operations on a derived rollup + the freshness and accuracy guarantees on it. The class fixes the NFR additions: ingest throughput (firehose absorbed without loss), bounded state (per-key state × cardinality must not grow without limit), per-tenant isolation, and query latency on the rollup.

Functional requirements (cite functional-requirements.html)

  • Ingest an event. An SDK posts (customer_id, project_id, exception, stacktrace, breadcrumbs, context, release, user_id?); the platform either accepts it (and groups it into an issue) or rejects it (rate limit / quota / malformed).
  • Group into an issue. Every accepted event is assigned a grouping fingerprint computed from the (normalized) stack trace; the issue is the (customer, fingerprint) tuple. New fingerprints create new issues; existing ones increment counts and update last-seen.
  • List issues. Given a project, return issues ranked by last_seen / event_count (over a time window) / release, paginated, with first-seen, last-seen, count, affected-user count. p95 ≤ 1 s even for a project with 200 k live issues.
  • Drill into an issue. Given an issue, return its metadata + a sample of recent raw events (full payloads) + the count time series + which releases it appears in.
  • Release regression alert. When an issue's event rate in release R+1 spikes vs R (or a brand-new issue first appears in R+1), fire an alert.

Out of scope (named, not silently dropped): distributed tracing / spans (separate APM product); session-replay video; source-control integration (issue→commit blame is a thin overlay); the SDK itself (client libraries are a separate workstream); uptime/cron monitoring; alerting fan-out channels (PagerDuty/Slack delivery is a separate notification service); the billing engine (we expose billed_events / quota but don't compute invoices here).

Flagged assumptions. (1) Avg raw event payload ≈ 5 KB (stack trace ~1–3 KB + breadcrumbs + context + tags). (2) ~50 M steady-state grouping fingerprints across the fleet after canonicalization; a naive hash(raw_stacktrace) would be 30–100× that. (3) Heavy tenant skew: the top customer can be 10%+ of total volume (≈ 500 k events/sec at peak from one tenant). (4) Per-issue popularity is Zipfian: a handful of issues (a hot null-deref in a hot code path) can be millions of events/sec; the long tail is ~1 event each. (5) Affected-user counts are approximate (HLL) — "12,431 users affected ± 2%" is what the dashboard needs, not an exact set. (6) Event sampling is acceptable: we keep all issue metadata + counts but only a rotating sample of K≈100 raw events per issue (30-day retention on the samples; issue metadata kept longer).

Sister scenarios (see plan)

  • #30 Logging system — the firehose-ingest sister where the payload is an opaque log line and the surface is ad-hoc search, not dedupe. Drastic flip: no grouping fingerprint, no per-issue rollup — #30 indexes for search; #42 collapses for signal.
  • #36 Streaming percentiles — the sketch-shaped sister: per-(series, window) DDSketch instead of per-(customer, window) Count-Min. Same cardinality-explosion bottleneck, same mergeable-sketch fix, different error model (value-error quantiles vs frequency estimates).
  • #20 Distributed rate limiter — the per-tenant-fairness sister: token buckets at the edge. #42 reuses the token-bucket primitive but the "key" is a customer and the question is "which of this customer's issues do I sample" not "should this request pass."
  • Audit-log aggregation (planned; SOC2/compliance event store) — same ingest shape but exact counts are mandatory and retention is years not 30 days. Drastic flip: sampling and Count-Min are forbidden; the design swaps to a Cassandra+exact-counter side-car and accepts the cost.

The fundamental tension. The naive "remember every fingerprint and its count" kernel is exact and per-key; the workload is "a firehose of events whose fingerprints are mostly unstable, occasionally adversarial in cardinality, and Zipfian in popularity." Three constraint-pairs collide: (a) exactness vs bounded state — an exact counter per fingerprint is infeasible at 50 M churning issues; a Count-Min sketch bounds memory at the cost of over-estimates proportional to ε×N (fine, we only need "which issues are high-rate"). (b) per-tenant fairness vs a shared limiter — a global rate-limit hashmap is simple, but one tenant doing 10%+ of volume starves everyone; per-customer token buckets + per-customer sketches + per-customer issue caps make a flood's blast radius the offender. (c) fine-grained grouping vs noise — hash(raw_stacktrace) is the finest possible grouping but fragments one bug into thousands of issues; canonicalizing the stack before hashing trades fragmentation for a few false merges. The arc resolves the tension: fingerprint at ingest + rollup store (Iter 0), per-customer token bucket + Count-Min (Iter 1), per-customer issue-count & tag quotas + negative cache (Iter 2), canonicalize the fingerprint (Iter 3), ClickHouse for the analytical query layer (Iter 4).

ID scheme (cite identifier-schemes.html#hash-id): issue id = (customer_id, fingerprint) where fingerprint = blake2b-128(canonicalized_frame_list). Why a content-derived fingerprint and not a Snowflake: two events for the same bug, ingested by different consumer workers with no coordination, must land on the same issue — the fingerprint is the dedupe key. 128 bits is plenty: collision probability at 50 M fingerprints is < 10-20; we don't re-verify on collision (a fingerprint collision merges two unrelated bugs — annoying, astronomically rare, and the customer can split via a fingerprint rule). Why scoped to customer_id: the same exception in two different customers' apps is two different issues; the customer prefix is the tenant boundary and the partition prefix.

The partition-key heuristic. The dominant write is "upsert the issue for (customer, fingerprint)" (every accepted event does one, micro-batched) and the dominant read is "list this project's issues, ranked." Writes are point-upserts keyed by (customer, fingerprint) → hash partition (256 shards) for uniform write spread, with a per-customer salt so one customer's 10 M issues don't all land on one shard. Reads are range scans within a (customer, project) ordered by last_seen or event_count → that's a columnar / analytical access pattern, which is why the dashboard's ranked queries move to ClickHouse (Iter 4) while the issue-metadata store stays the KV/wide-column source of truth.

S2. API contracts (cite api-design-patterns.html)

Two surfaces: the ingest API (the SDK→platform firehose — one endpoint, must be cheap and fast to reject) and the dashboard API (list / drill-in / regression — the read surface that the p95 ≤ 1 s SLO is about). Internal: the consumer→rollup upsert and the rollup→ClickHouse stream are not public APIs.

Per-endpoint SLO

  • POST /store/ — p95 ≤ 50 ms (DSN auth + Redis token-bucket op + Kafka append; everything else is async). This is the firehose; it must be cheap to reject too — a rate-limited customer gets a 429 in < 5 ms without touching Kafka.
  • GET /issues/ — p95 ≤ 1 s on a 200 k-issue project (the load-bearing read SLO; the iteration arc tunes for this and Iter 4 is where it's met).
  • GET /issues/{id}/ — p95 ≤ 1 s (Cassandra point-read + S3 multi-get of samples + ClickHouse time-series query, in parallel).
  • GET /releases/{r}/regressions/ — p95 ≤ 2 s (two-window aggregation; less hot than list-issues).
  • Event → issue freshness: "issue exists / first-seen" visible < 1 s after ingest; "count time series" lags one micro-batch flush ≈ 10 s. Stated as a freshness contract, not a latency SLO.

Count-Min / quota / fleet / cost formulas

// --- Token bucket (per customer; Iter 1) ---
// bucket(cust): { tokens, last_refill_ts }; capacity = quota_per_min; refill = quota_per_min / 60 tokens/sec.
// On a micro-batch of n events for cust: tokens -= n (clamp 0); accept min(n, tokens_available); the rest -> dropped (counted).
// Authoritative bucket in Redis; consumer batches the decrement per (customer) per ~1 s -> Redis op rate ~ N_customers / 1 s = 50k ops/s. Tiny.

// --- Count-Min sketch (per customer per time window; Iter 1; cite sketches#cms) ---
// Goal: estimate per-ISSUE event rate within ONE customer's in-window stream, so we
//       can decide which of that customer's issues to keep sampling vs drop -- WITHOUT
//       a counter per fingerprint (50M of them).
// w = ceil(e / eps); d = ceil(ln(1/delta)).
// Pick eps = 0.001 (over-estimate <= 0.1% of the customer's in-window total),
//      delta = 0.001 (>= 99.9% confidence):
//   w = ceil(2.71828 / 0.001) = 2719 columns
//   d = ceil(ln(1000))        = 7 rows
//   counter = 4 B
//   bytes_per_sketch = w * d * counter = 2719 * 7 * 4 = 76,132 B = ~74 KB / (customer, window)
// Error bound: est(issue) <= true(issue) + eps * N_window  (prob >= 1-delta), N_window = customer's in-window event count.
// Mergeable: element-wise sum of same-shape sketches. Consumers keep partials in RAM, merge to Redis at window close.
// Total live CMS state: N_customers * 2 windows (current + previous) * 74 KB = 50_000 * 2 * 76_132 = ~7.6 GB.

// --- HLL for affected-users (per issue) ---
// HLL with 14-bit registers: 2^14 = 16384 registers * 1 B = 16 KB/issue at +/-0.8% error.

S4. Naive Architecture — the simplest thing that meets the FRs

Every raw event is inserted as a row into one Postgres table, events, with a fingerprint column = md5(stacktrace_text) computed by the ingest server. There is no issues table: the dashboard's "list issues" query is an on-demand SELECT fingerprint, count(*), max(received_at), min(received_at) FROM events WHERE project_id=? AND received_at > now()-'24h' GROUP BY fingerprint ORDER BY count(*) DESC LIMIT 25. Rate limiting is a single in-memory hashmap on the ingest server: last_seen[fingerprint] = ts, drop if seen in the last 10 s — this is LC #12's Logger Rate Limiter lifted directly. Drill-in is SELECT * FROM events WHERE fingerprint=? LIMIT 100.

Architecture

Connections (numbered above) and line-style legend. Solid slate = sync hot path. The Naive design has no async paths and no buffer — every event blocks on a Postgres INSERT, and every dashboard load blocks on a full-table aggregation.

  • ② Ingest Server → Postgres `events` — INSERT INTO events (event_id, project_id, fingerprint, received_at, release, user_id, payload) VALUES (...). One row per raw event.

This is the simplest architecture that satisfies all five FRs — one INSERT per event, one GROUP BY per dashboard load, one hashmap for rate limiting. No buffer, no rollup, no sketches, no canonicalization, no analytical store.

Why it's broken (per failing NFR)

  • Ingest throughput. ✗ A single Postgres tops out at ~15 k inserts/sec; peak is 5 M/sec — 333× over. Sharding by project doesn't save it: a single hot project (the 10%-of-volume tenant) is 500 k/sec into one shard. There's no buffer to absorb the burst; the SDK gets 500s and drops events.
  • Dashboard query latency. ✗ "List issues ranked by 24 h event count" is a GROUP BY fingerprint over ~100 B rows. Even a 64-core parallel scan is ~26 minutes; the p95 ≤ 1 s SLO is missed by 3–4 orders of magnitude. No index helps — the aggregation touches every recent row.
  • Bounded state. ✗ The last_seen hashmap and the events table both grow with the input stream, not with the issue count. A malformed release that sprays 10 M distinct md5(stacktrace) values OOMs the ingest server and adds 10 M useless "issues" to the dashboard.
  • Per-tenant isolation. ✗ The rate-limit hashmap is either per-process (so the 10 s limit leaks ×N_servers) or, if moved to one Redis, global (so one customer's fingerprint flood evicts everyone else's entries). Either way one tenant degrades the whole fleet.
  • Storage / retention. &warning; Keeping every raw event for 30 days is 15.36 PB on EBS ≈ $1.23 M/mo just for the volumes — and 99%+ of those rows are duplicates of an issue the dashboard would show as "×4.2 M".

Cost of naive at target load: ~$1.41 M/month (333 sharded Postgres primaries at $554 + 15.36 PB EBS at $0.08/GB) and it still cannot ingest 5 M/sec, still cannot run the dashboard query in under half an hour, and still OOMs on a fingerprint flood. The first failing NFR is ingest throughput — a synchronous "INSERT then ack" can't absorb a firehose. The second is dashboard latency — an on-demand GROUP BY over the firehose is hopeless. We must change the unit of storage from "raw event" to "issue rollup", put a durable buffer in front, and move the ranked query to a columnar store — not scale Postgres.

Bottleneck

Naive's first failing NFR is ingest throughput (a synchronous INSERT can't absorb 5 M/sec), and its second is dashboard latency (an on-demand GROUP BY over 100 B rows is ~26 min). Both are the same root cause: backlog-growth + the wrong unit of storage. The producer rate (5 M events/sec) vastly exceeds any single-store consumer rate, and the thing the dashboard wants — counts per issue — is being re-derived from scratch every read. Iteration 0 fixes both at once because they're one fix: put a durable buffer (Kafka) in front so the burst is absorbed, and fold the firehose into a pre-computed rollup (one row per issue) so the dashboard reads, not aggregates. Per bottleneck-to-decisions#backlog-growth: add a queue + backpressure; pre-aggregate to bound the read cost.

Why Kafka and not "just scale Postgres"

  • Kafka (durable partitioned log) — chosen. ~100 MB/s/broker write; a flood pins to a customer's partitions instead of fanning out across all shards; consumers replay from offset on a bug; the SDK gets a fast 200 because Kafka append is μs-class. Cite storage-engines#log. (Sentry runs Kafka in front of its consumers; Snuba/Relay topology.)
  • Sharded Postgres — rejected. 333 shards just to ingest peak; a hot project still slams one shard; no replay; the dashboard's GROUP BY is still cross-shard scatter-gather over the firehose. Sharding moves the bottleneck, it doesn't remove it.
  • Kinesis — rejected. 1 MB/s/shard write → 25 GB/s peak / 1 MB/s = 25,000 shards; the per-shard limits and resharding ops dominate. Kafka's per-broker throughput is ~100× higher.
  • Cassandra for the issues rollup — chosen over DynamoDB and Postgres. LSM write path eats the upsert firehose; (customer, fingerprint) partition key gives O(1) "does this issue exist?"; counter columns for times_seen; wide rows hold the K sample-event-ids and the (sparse) HLL blob. DynamoDB fits the access pattern but bills a write-unit per counter increment — at 50 M churning issues that exceeds a self-hosted Scylla cluster. Postgres is the Naive failure. Cite storage-engines#wide-column.

The chosen move

The ingest worker becomes a thin, fast accept layer: authenticate the DSN, PII-scrub, append to Kafka (partition key = customer_id), return 200 in < 50 ms. A pool of partitioned consumers reads the Kafka topic and, per event: (a) computes fp = hash(stacktrace_text) (the naive fingerprint — we don't canonicalize until Iter 3); (b) micro-batches per-(customer, fp) aggregates in RAM over a 10 s flush window; (c) on flush, upserts each touched issue row in Cassandra (insert + first_seen on a new fingerprint; times_seen += delta, last_seen = max otherwise) and updates the issue's sparse HLL of affected users; (d) reservoir-samples K≈100 raw events per issue into S3 with a 30-day lifecycle rule. The dashboard reads issue rows for a project, ordered by last_seen (a clustering column). The micro-batch matters: under the firehose, ~10% of the 2 B issue universe fires in any 10 s window ≈ 5 M issue-rows touched → 5 M / 10 s = 500 k Cassandra upserts/sec instead of 5 M (one per event); a hot issue seeing 1 M events/sec becomes one += 10,000,000 upsert per consumer per flush instead of 1 M increments. Cite sharding-strategies#hash (partition the rollup by (customer, fingerprint)) and cost-shapes (the firehose-ingest shape: most of the bill is Kafka + consumers).

What breaks (C5; cite failure-modes.html)

Bottleneck

Iter 0's gap: per-tenant isolation. The 10%-of-volume tenant doing 500 k events/sec lands on ~20 Kafka partitions; the consumers owning those partitions fall behind; that tenant's count-freshness degrades to minutes and, if sustained, the lag never drains — and there's no mechanism to throttle a single tenant. Worse, even within that one tenant, we have no cheap way to decide which of its (say) 30,000 active issues deserve fresh raw samples during the burst — an exact counter per fingerprint is the LC #12 hashmap problem (unbounded keys). Classified as backlog-growth (one producer's rate > the consumer rate it can claim) compounded by cardinality-explosion (per-fingerprint state). Iteration 1 fixes both at the tenant boundary: a token bucket per customer caps the inflow before Kafka, and a Count-Min sketch per customer per window estimates per-issue rates within that customer's bounded budget — so a flood's blast radius is the offending tenant, and sampling decisions cost ~74 KB of sketch per (customer, window) instead of a counter per issue.

Why Redis token bucket + Count-Min, and not the alternatives

  • Per-customer token bucket in Redis — chosen. The authoritative bucket must be shared across all ingest workers (an event for customer C can hit any worker) and all consumers handling C's partitions, so it can't be in-process. Redis: a single Lua script does atomic refill + decrement; ~200 k ops/sec/node and the ingest worker batches the check per (customer) per ~1 s → ~50 k Redis ops/sec total — trivial. Cite storage-engines#in-memory. (Sentry uses Redis for quota/rate-limit state.)
  • In-process rate limit on each ingest worker — rejected. Each worker would see only a fraction of a customer's traffic; the limit leaks ×N_workers (the same bug the Naive design has). Token-bucket-per-worker only works if you also pin a customer to one worker, which destroys load balancing.
  • A dedicated quota microservice — rejected (for now). Extra hop + extra fleet for what is one Redis key op. Revisit if quota logic gets hierarchical (per-project sub-quotas, burst credits, overage billing tiers).
  • Count-Min sketch per (customer, window) — chosen over HLL and exact counters. We need per-issue frequency estimates within a customer's stream; HLL answers "how many distinct issues" (a count, not a per-key frequency) — wrong question. Exact counters are the unbounded-key problem (50 M of them). CMS bounds memory at w×d×4 B regardless of how many issues; over-estimates are bounded by ε×N_window, which is fine because we only care about which issues are high-rate (the over-estimate on the long tail doesn't change the ranking of the head). Mergeable (element-wise sum) → consumers keep partials, merge at window close. Cite sketches#cms.
  • Why not Space-Saving / HeavyKeeper (top-K sketches)? Those give you "the K hottest issues" deterministically; we'd want that and a frequency estimate for an arbitrary issue (e.g., "is issue X high-rate enough to keep sampling?"). CMS answers the point query; a small top-K heap on top (cheap) gives the head set if needed. CMS is the more general primitive here; Space-Saving is the better pick when you only want the head (that's #15).

The chosen move

Token bucket. Each customer has a bucket {tokens, last_refill_ts} in Redis with capacity = event_quota_per_min and refill = event_quota_per_min / 60 tokens/sec (the quota comes from the plan). The ingest worker batches arrivals per customer over ~1 s and runs one Lua script: refill by elapsed×rate (clamp to capacity), then accepted = min(n, tokens); tokens -= accepted; the n - accepted overflow events get a 429 with Retry-After and a dropped_events counter increments — they never enter Kafka. A flood is now bounded to the offender's quota: a customer provisioned for 100 k events/min who sprays 500 k/sec gets ~1,667 accepted per second and 498,333 rejected, and the consumers never see the flood.

Count-Min for per-issue rates within a customer. Each consumer holds, per customer it's seeing, a partial CMS for the current time window (say 60 s): w = ⌈e/ε⌉ = ⌈2.71828/0.001⌉ = 2719 columns, d = ⌈ln(1/δ)⌉ = ⌈ln(1000)⌉ = 7 rows, 4-byte counters → 2719 × 7 × 4 = 76,132 B ≈ 74 KB per (customer, window). On each event for customer C with fingerprint fp: increment cms_C[i][hash_i(fp)] for i in 1..7. Query "rate of issue fp in C": min over i of cms_C[i][hash_i(fp)] — over-estimates by at most ε × N_window (if C did 100 k events in the window, that's ±100 — noise for a long-tail issue, irrelevant for a head issue doing 50,000). At window close each consumer merges its partial into C's authoritative CMS in Redis (element-wise sum); the next window's "should we keep refreshing samples for issue fp?" decision reads it: high-rate issues keep getting fresh samples even under a burst; low-rate issues aren't churned out of the reservoir. Total live CMS state: 50,000 customers × 2 windows (current + previous) × 74 KB = ~7.6 GB — fits comfortably in a small Redis cluster. Cite sketches#cms.

What breaks (C5; cite failure-modes.html)

  • Token-bucket cold-start stampede. If Redis loses a shard and the buckets reset to capacity, every customer on that shard briefly gets a full burst allowance → a coordinated flood. Detection: Kafka append rate spike right after a Redis failover. Recovery: persist buckets with AOF + replica failover; on cold start, initialize tokens = 0 (deny-then-refill) rather than tokens = capacity — the design choice is "fail closed" on the limiter.
  • CMS staleness across a window boundary. A brand-new issue that appears mid-window has CMS estimate ≈ its in-window count so far, which can under-represent its true rate for the first window — the consumer might not start sampling it aggressively until the next window. Detection: "issue first-seen but < K samples after 2 windows". Recovery: a brand-new fingerprint always gets its first ~10 events sampled regardless of CMS (the CMS gates sustained sampling, not the first-look samples).
  • CMS hash collision inflating a tail issue. Two unrelated fingerprints colliding on all d=7 rows would make a quiet issue look hot → we'd waste sample-store writes on it. Detection: probability is (1/w)^d = (1/2719)^7 ≈ 10-25 — effectively impossible; if observed, it's a hash-function bug, not a collision. Recovery: rotate the d seeds per window (already done at window roll).

3. Token-bucket Lua (the atomic decrement).

# EVAL'd in Redis; KEYS[1] = rl:tb:{customer_id} ; ARGV = [now_ms, capacity, refill_per_ms, n_arrived]
def token_bucket(key, now_ms, capacity, refill_per_ms, n):
    tokens, last = unpack(redis.get(key) or (capacity, now_ms))
    tokens = min(capacity, tokens + (now_ms - last) * refill_per_ms)   # refill by elapsed time
    accepted = min(n, int(tokens))
    tokens -= accepted
    redis.set(key, pack(tokens, now_ms))
    return accepted          # the ingest worker forwards `accepted` events to Kafka, 429s the rest

Bottleneck

Iter 1's gap: the token bucket caps a customer's event rate but not their issue cardinality. A customer provisioned for 100 k events/min, plus a release that embeds a request-id in an exception message, produces 100 k distinct fingerprints/min — 100 k new issue rows/min in the shared Cassandra table, each costing a read-before-write to check "is this a new fingerprint?", all charged against that customer's storage quota but bloating the table everyone shares and burying the dashboard in junk. Classified as cardinality-explosion at the per-tenant granularity. Per bottleneck-to-decisions#cardinality: cap N with eviction; route the overflow to a single bucket; negative-cache the known-bad keys so the hot path skips the lookup. Iteration 2: a per-customer live-issue-count cap (overflow → one synthetic bucket), a per-issue tag-cardinality cap (the same explosion one level down), and a TTL'd negative cache of overflow fingerprints in Redis so a sustained malformed release costs O(1) per event, not O(Cassandra read).

Why a hard cap + negative cache, and the alternatives

  • Hard per-customer issue-count cap (~10 k live issues) with overflow → one synthetic bucket — chosen. A real project has tens to low-thousands of distinct active issues; 10 k is generous headroom and a malformed release blowing past it is a signal worth surfacing, not silently absorbing. Overflow to one bucket means the cost of "the 11,000th distinct fingerprint" is a counter increment, not a row. (Sentry's "spike protection" / per-project event and issue limits work this way.)
  • Negative cache (Redis TTL'd set of overflow fingerprints per customer) — chosen. Without it, every event for an over-cap fingerprint does a Cassandra read ("is live_issue_count ≥ cap? does this fingerprint exist?") before being diverted — under a sustained flood that's the read-before-write thrash all over again. With it, the consumer checks a Redis set membership (O(1), batched per micro-batch) and drops/diverts; the TTL means the entry ages out if the customer fixes the release. Cite the negative-caching pattern (cache the absence / the known-bad, not just the present).
  • Per-issue tag-cardinality cap — chosen. The same explosion one level down: a tag like request_id or session_id with a unique value per event blows up the per-issue tag index (used for "filter this issue's events by tag value"). Cap tags_seen_count per issue; over the cap, stop indexing new tag values (the events still count toward times_seen).
  • Soft cap / "just bill them for it" — rejected. The storage quota already bills the customer; but the shared issues table, the consumer's read-before-write rate, and the dashboard's signal-to-noise are not the customer's to spend — they're shared resources. A hard cap protects the shared substrate; billing handles the per-tenant accounting separately.
  • Why not just canonicalize the fingerprint first (Iter 3)? — ordering. Canonicalization reduces natural fragmentation (line numbers, minified frames) but it doesn't stop an adversarial / pathological release that puts genuinely-distinct data in the stack trace (a request-id baked into a frame name). The cap is the backstop; canonicalization is the everyday reduction. We need both, and the cap should land first because it bounds the worst case regardless of how good canonicalization gets.

The chosen move

Add a Cassandra counter live_issue_count per customer (decremented when an issue is deleted/merged) and a counter tags_seen_count per issue. In the consumer, before creating a row for a brand-new fingerprint fp for customer C: (1) check fp ∈ neg:{C} in Redis (batched per micro-batch) — if yes, route the event to C's <too-many-issues> issue (times_seen += delta) and stop; (2) else read live_issue_count(C) — if ≥ cap, route to <too-many-issues>, do SADD neg:{C} fp with a (say) 1-hour TTL, and stop; (3) else create the issue row, live_issue_count(C) += 1, and proceed normally. For an existing issue: on a new tag value, if tags_seen_count(issue) ≥ tag_cap (say 1,000), don't index the new value (the event still counts). The <too-many-issues> bucket is surfaced on the dashboard with a "issue cap reached — fix your grouping (likely a variable in a frame name or message) or raise the limit" banner, so it's a visible, actionable signal, not a silent drop. What breaks (C5; cite failure-modes.html)

  • Negative-cache TTL too long — legitimate new issues suppressed. If a customer fixes the malformed release but the neg-cache entries still have an hour to live, genuinely-new issues that happen to collide with an old overflow fingerprint are diverted to <too-many-issues>. Detection: divert rate per customer staying high after their issue count drops. Recovery: short TTL (minutes, not hours) + an explicit "flush my negative cache" admin action; the cap re-fills the set in seconds if the release is still bad.
  • live_issue_count counter contention. Many consumers incrementing one customer's live_issue_count on a new-fingerprint burst — Cassandra counters are CRDTs (conflict-free) so correctness holds, but the read-modify-decision pattern ("read count, decide, increment") is not atomic, so under a burst the cap can overshoot by ~(N_consumers) issues. Detection: live_issue_count drifting above cap + N_consumers. Recovery: that overshoot is harmless (a few hundred extra rows); a periodic reconciler recomputes the true count from the issue rows and corrects the counter.
  • Cap-raise stampede. Support raises a big customer's cap from 10 k to 100 k mid-flood → the consumer suddenly creates 90 k issue rows for that customer in one flush window. Detection: new-issue rate spike right after a cap change. Recovery: ramp cap changes (raise by 2× per flush window, not in one jump) so the row-creation rate stays bounded.

Bottleneck

Iter 2's gap: the fingerprint is still hash(raw_stacktrace_text), so the same logical bug fragments into many issues — a deploy that bumps every line number makes every issue "new"; a minified bundle gives every frame an opaque name like a/n/t that changes per build; a recursive function appears with a stack-trace depth that varies per input. The result: the issue count is inflated (toward the cap, for no good reason), release regressions are invisible (release R+1 looks like 1,000s of brand-new issues, not "the same 200 issues, one of which got 5× worse"), and the dashboard is noisy. Classified as cardinality-explosion rooted in the wrong fingerprint — the dedup key is too sensitive to incidental variation. Per bottleneck-to-decisions#cardinality: normalize the key so incidental variation collapses; group at the granularity that matches "is this the same bug?". Iteration 3: canonicalize the stack trace before hashing — strip line numbers, collapse recursion, de-minify via source maps, group by the in-app frame subset, apply customer fingerprint rules — trading a few false merges for ~40× less fragmentation, a meaningful releases_seen, and a possible regression view.

The canonicalization algorithm (worked example)

The fingerprint is blake2b-128 over a normalized list of frame keys, computed by:

  1. De-minify / symbolicate. If a frame is from a minified JS bundle or a stripped native binary, look up the release's source map / debug symbols (cached in Redis, cold in S3) and replace the opaque name (a.n.t) with the original (UserStore.fetchProfile). If no source map is available, fall back to (filename, basename) and tag the issue "needs source map" on the dashboard.
  2. Pick the in-app frames. Keep frames marked in_app: true (your code); drop framework/runtime/library frames (React, the JVM, libc) — two bugs in your code with the same call site but different library-internal stacks are the same bug. If no frame is in-app (a pure library crash), fall back to the full stack.
  3. Strip incidental positions. Drop lineno and colno from each frame's key — a line shifting from 142 to 147 across a deploy is not a new bug. Keep (module, function, normalized_filename).
  4. Collapse recursion. Replace a run of ≥ 2 identical adjacent frames with one (a recursive function blowing the stack at depth 4,000 vs 4,200 is the same bug).
  5. Normalize anonymous / dynamic frames. <anonymous>, eval'd code, generated frames → a stable placeholder keyed by the enclosing scope, not the per-instance address.
  6. Apply customer fingerprint rules. The customer can supply rules ("group everything under vendor/* as one issue"; "split this issue by error.code") — applied last so the customer can override the default grouping for known-quirky paths.
  7. Hash. fp = blake2b-128("\n".join(frame_keys)).

EVENT A (release 1.3.0):                       EVENT B (release 1.4.0):
  at a.n.t (app.min.js:1:24187)                  at a.q.r (app.min.js:1:31002)      <- minifier renamed; line/col moved
  at handleClick (app.min.js:1:24050)            at handleClick (app.min.js:1:30910)

// 1. de-minify via 1.3.0 / 1.4.0 source maps:
//      a.n.t  -> UserStore.fetchProfile        a.q.r -> UserStore.fetchProfile
// 2. in_app frames only: drop HTMLButtonElement. (vendor)
// 3. strip line/col ; keep (module, function): [UserStore.fetchProfile, handleClick]
// 4-5. no recursion, no anonymous in the in-app subset
// 6. no customer rule for this path
// 7. fp = blake2b-128("UserStore.fetchProfile|UserStore\nhandleClick|app") = 0x9c...   (SAME for A and B)
//  -> ONE issue. "Seen in releases: 1.3.0, 1.4.0." If its rate in 1.4.0 is 5x its rate in 1.3.0 -> regression alert.

// --- Fragmentation reduction ---
// Empirically (Sentry-grade): hash(raw_stacktrace) over a year of a busy project produces ~30-100x more distinct
// fingerprints than the canonicalized version. We model the fleet steady-state as ~2e9 (naive) -> ~50e6 (canonical), ~40x.

Cite Sentry's published grouping docs/blog ("How Sentry groups issues" / the grouping engine) and the analogous Bugsnag "error grouping" and Rollbar fingerprinting docs — all do this same normalize-then-hash pipeline; the differences are which frames count as "in-app" and how aggressive the de-minification is. The cost of getting it wrong: too aggressive → two different bugs merge into one issue (a false merge; the customer splits via a rule); too timid → the fragmentation we started with. The default leans slightly timid (prefer not to merge) because a false merge hides a bug, while a false split is merely noisy.

What breaks (C5; cite failure-modes.html)

  • Missing source map → opaque grouping. If we ship a new canonicalization algorithm, events processed by old vs new consumers produce different fingerprints for the same bug — the issue "splits" for the duration of the rollout and merges back after. Detection: a transient spike in new-issue rate during a consumer deploy. Recovery: version the canonicalization algorithm; pin all consumers to one version per deploy wave (no mixed-version processing of the same Kafka partition); accept a short-lived split during the cutover, then a one-time merge job re-keys the affected issues.

Bottleneck

Iter 3's gap: the dashboard's headline queries — "list this project's issues ranked by event count over the last 24 h," "filter to issues seen in release 1.4.2," "which issues regressed in 1.4.2 vs 1.4.1" — are analytical aggregations over many issue rows, and the Cassandra rollup is a point-/range-by-partition store: to rank a 200 k-issue project by 24 h count, you'd scatter-read 200 k partitions and sort in the app → multi-second p95, blowing the ≤ 1 s SLO. Classified as wrong-data-structure for the access pattern: the data is right (canonical issues, real releases_seen) but the storage layout (one partition per issue, no columnar pre-aggregation by time/release) doesn't match "scan-and-aggregate-then-top-N." Per bottleneck-to-decisions#wrong-data-structure: add a read-optimized projection in an engine whose layout matches the query — here, a columnar OLAP store fed from the same write stream. Iteration 4: stream per-(issue, minute) aggregates + sampled-event headers into ClickHouse; serve the dashboard's ranked / time-windowed / by-release queries from there; keep Cassandra as the metadata + dedupe-state SoR.

Why ClickHouse, and the alternatives

  • ClickHouse — chosen. Column-oriented, vectorized; a GROUP BY issue ORDER BY sum(count) DESC LIMIT 25 over a 24 h window touches a few columns of a few hundred million highly-compressible rows and scans at billions of rows/sec/node; MergeTree partitioning by day + ordering by (customer, project, minute) means a project's 24 h window is a contiguous range. Real-time ingestion via batched inserts is mature. Cite storage-engines#columnar. (This is Sentry's architecture: Snuba is a query service in front of ClickHouse, serving the issues/events API.)
  • Elasticsearch — rejected. Its inverted index is built for full-text search; we rank by numbers (event count), not search by token — the index is wasted weight, aggregations over hundreds of millions of docs are slower than ClickHouse's columnar scan, and JVM GC pauses make the p99 spiky. Keep ES (if at all) for the secondary "search issue titles/messages by text" feature, not the primary ranked list.
  • Druid — rejected (close call). Also columnar OLAP with real-time ingestion; viable. But the operational surface (coordinator/overlord/historical/broker node types, segment management) is heavier than ClickHouse's, and the join story for "ClickHouse aggregates × Cassandra metadata" is weaker — we'd denormalize more into Druid. ClickHouse's single-node-type cluster + dictionary-join support wins on operational simplicity at this scale.
  • Postgres (with materialized views) — rejected. A 200 k-issue project's 24 h ranked query is 200 k×1,440 = 288 M rows; no Postgres index makes that < 1 s without a pre-aggregated MV per ranking dimension (count / users / by-release) refreshed continuously — which is a column store, badly reinvented.
  • Why not put everything in ClickHouse and drop Cassandra? ClickHouse is bad at the write pattern of the dedupe path: high-rate point-upserts to (customer, fingerprint) with a counter and a mutable status/HLL — MergeTree doesn't do efficient in-place updates (ALTER ... UPDATE is a heavyweight async mutation). Cassandra's LSM + CRDT counters are exactly right for that. Two engines, two access patterns: the canonical "columnar projection alongside the OLTP-ish SoR" split.

The chosen move

The consumer, on each flush, in addition to upserting Cassandra, batch-inserts into ClickHouse: (a) issue_minute — one row per touched (customer_id, project_id, fingerprint, minute_bucket, release, environment) with the count delta and an affected_users_hll partial for that minute (ClickHouse has a native AggregateFunction(uniqCombined, ...) for mergeable HLL); (b) sampled_events — a flat row per K-sample with (fingerprint, received_at, release, environment, level, tags Map(String,String)) for "show this issue's recent events filtered by tag/release." The dashboard's list-issues becomes SELECT fingerprint, sum(count) AS c, uniqCombinedMerge(affected_users_hll) AS u FROM issue_minute WHERE customer_id=? AND project_id=? AND minute_bucket >= now()-24h GROUP BY fingerprint ORDER BY c DESC LIMIT 25 → then a Cassandra multi-get for the 25 issues' title/culprit/status. By-release filtering adds AND release=?. Regression: two windows, GROUP BY fingerprint, release, compare rate_after / rate_before; "new in release" = fingerprints with rows in R+1 and none in R. What breaks (C5; cite failure-modes.html)

  • ClickHouse ingest lag → stale ranked list. If the consumer→ClickHouse insert path falls behind (a bad query saturating the cluster), the ranked list shows counts that are minutes stale — date-sorted (from Cassandra last_seen) is still fresh. Detection: ClickHouse insert lag metric. Recovery: dedicate a query node pool separate from the insert pool; rate-limit expensive ad-hoc queries; the Cassandra SoR is unaffected so "issue exists / first_seen" stays correct.
  • Cassandra/ClickHouse divergence on issue identity. If a consumer upserts Cassandra but its ClickHouse insert fails (or vice versa), the ranked list can reference an issue whose metadata Cassandra doesn't have yet, or vice versa. Detection: list-issues returning a fingerprint with no Cassandra row. Recovery: ClickHouse is the derived store — the dashboard tolerates "issue X has 4,000 events in the last hour, metadata loading…" for a few seconds; a reconciler backfills any ClickHouse rows whose Cassandra row is missing from the issue table.
  • Hot project query melting a ClickHouse shard. One huge customer's "list 200 k issues, no filter, 30-day window" is a much bigger scan than the 24 h SLO case (200 k×43,200 = 8.6e9 rows). Detection: per-query rows-scanned histogram. Recovery: cap the dashboard window selector at the SLO'd range for the live list (longer windows go through an async "export" path); pre-aggregate to a day-rollup table for > 7-day windows so the scan is 200 k×30 = 6 M rows instead of 8.6 B.

Final absorbs the firehose through Kafka into ~210 canonicalizing consumers, holds a bounded ~50 M-issue rollup with per-customer event-rate / issue-count / tag-cardinality caps and a Count-Min sketch for per-issue rates, and serves the dashboard's ranked / time-windowed / by-release queries from ClickHouse at p95 well under the 1 s SLO.

Bottleneck arc: Naive (synchronous INSERT can't absorb the firehose; on-demand GROUP BY over 100 B rows) → Iter 0 (no per-tenant isolation — one customer's flood lags the consumers; no per-issue rate signal) → Iter 1 (issue cardinality still unbounded — a malformed release bloats the shared table) → Iter 2 (the same bug still fragments ~40× across releases / minified frames — regressions invisible) → Iter 3 (the ranked / time-windowed / by-release dashboard queries are analytical scans Cassandra can't serve in < 1 s) → Iter 4 (all NFRs met).

Dominant cost: the firehose ingest — Kafka (~$122 k, 42%) + consumers (~$116 k, 40%) = ~82% of the bill. ClickHouse 6%, Cassandra 4%, Redis 3%, app/ingest/Postgres ~4%, S3 < 1%. This is the canonical firehose-ingest cost shape (cite cost-shapes): the bytes-in-per-second dwarfs everything stored or queried. The dominant cost-vs-correctness knob is the canonicalization budget — doing it cheaper (skip source-map de-minification) would drop consumers from ~210 to ~150 nodes (−$33 k/mo) at the cost of coarser grouping on minified codebases; the dominant cost-vs-freshness knob is the micro-batch flush interval (10 s → 1 s would 3× the Cassandra fleet for 10× fresher counts).

Degradation contract

When a Cassandra shard is unavailable: the consumer can't upsert issues for that (customer, fingerprint) range — it parks those events in a Kafka dead-letter topic and replays when the shard recovers; the dashboard's date-sorted list degrades for affected projects but ClickHouse-served ranked queries still work (different store). When ClickHouse is unavailable: the dashboard falls back to the Cassandra date-sorted list ("newest 25 issues"); ranked / by-release / regression views show "temporarily unavailable"; ingest and dedupe are unaffected. When Redis is unavailable: token buckets fail closed (the ingest worker rejects with 429 until Redis is back — we'd rather shed than let a flood through), the CMS reverts to "sample everything new" (slightly more sample-store writes), and the negative cache misses (the consumer falls back to a Cassandra read on new fingerprints — slower under a flood but correct). When Kafka is unavailable: ingest workers return 503; SDKs buffer locally and retry; once Kafka is back, the backlog drains (consumers replay from offset). When the consumer fleet falls behind: Kafka lag grows; count freshness degrades from 10 s to minutes; the "issue exists / first_seen" fast path still fires within seconds; no events are lost (they're durable in Kafka).

Real-system comparison

Sentry's public "Grouping in Sentry: tradeoffs and how it works" describes the same canonicalize-then-hash pipeline — in-app frame selection, line-number stripping, recursion collapse, source-map de-minification, and user-supplied grouping enhancements (our "fingerprint rules") — and explicitly frames it as a tradeoff: too aggressive merges distinct bugs, too timid fragments one bug. Sentry's ingest path runs Relay → Kafka → consumers, and its analytical layer is Snuba, a query service in front of ClickHouse serving the issues/events API — the Iter 4 split here ("metadata in the KV store, numbers in ClickHouse") is that architecture. Sentry's "spike protection" and per-project event/issue rate limits are the Iter 1–2 token-bucket + issue-count cap; quota state lives in Redis. Bugsnag's "Grouping and deduplication of errors" and Rollbar's fingerprinting docs document the same normalize-the-stack-trace approach with different in-app heuristics.

Where this design diverges from the published systems: (a) we use Count-Min sketches explicitly for per-issue rate estimation under cardinality pressure — Sentry's published material doesn't detail the exact data structure behind its sampling/spike-protection decisions, but the constraint (decide which issues to keep sampling without a counter per fingerprint) is the same and CMS is the textbook fit (cite sketches#cms); (b) we keep the metadata SoR on Cassandra/Scylla — Sentry has used Postgres for issue metadata at smaller scale and a mix at larger scale; the access pattern (high-rate point-upserts + counters) is what argues for an LSM/wide-column engine at 50 M churning issues; (c) S3 for sample bodies vs Sentry's node-store abstraction — same role, different backing. The cribs/42.md file carries verbatim quotes from the Sentry / Bugsnag / Rollbar posts to diff against.

Why not exact counts? (cardinality)

  • Where exact is kept: times_seen per issue is a Cassandra CRDT counter — that's exact (modulo the micro-batch flush lag). The ClickHouse issue_minute counts are exact sums. So per-issue totals are exact-ish.
  • Where the sketch is used: deciding, within a customer's stream during a burst, which of its (say) 30,000 active issues are high-rate enough to keep refreshing raw samples for — that needs a per-issue rate estimate, and an exact counter per fingerprint is the LC #12 unbounded-key problem (50 M of them, churning). CMS bounds memory at ~74 KB/(customer,window) regardless; the over-estimate is ≤ ε×N_window, which is noise for a tail issue and irrelevant for a head issue.
  • The audit case: if a customer needs provably-exact event counts (billing disputes, compliance), the Cassandra counter + ClickHouse sums are the exact record; the CMS never feeds a billed number — it only feeds the sampling decision. A "tagged-series exact audit path" (a parallel exact store for flagged projects) is Probe B.

Why Count-Min and not HyperLogLog here? (sketch choice)

  • Different questions. HLL answers "how many distinct issues did this customer have this window?" (a cardinality). CMS answers "how many events did this specific issue get?" (a per-key frequency). The sampling decision needs the second.
  • We use HLL too — for a different field: affected_users per issue is exactly "how many distinct user ids touched this issue" → HLL (sparse, ~1–4 KB/issue, mergeable). Right tool, right question.
  • And not Space-Saving / HeavyKeeper? Those give the top-K hottest issues deterministically; we'd want that and a frequency estimate for an arbitrary issue ("is issue X hot enough to keep sampling?"). CMS answers the point query; a tiny top-K heap on top is cheap if we also want the head set. Space-Saving is the better pick when you only want the head — that's #15, a different scenario.

Why Kafka and not direct-to-store? (buffer)

  • The burst. Peak is 5 M events/sec, ~4× the average; an incident that breaks a popular app sends a wall of events. A synchronous "INSERT then ack" couples the SDK's latency (and the SDK's retry storm) to the slowest downstream store; Kafka append is μs-class, so the SDK always gets a fast 200, and the burst sits in the log while consumers drain it.
  • Replay. A consumer bug that mis-canonicalizes a release's events? Fix it, rewind the offset, reprocess. Direct-to-store has no replay — the bad data is already written.
  • Per-tenant ordering & isolation. partition key = customer_id means a customer's events are ordered, and a flood concentrates on that customer's partitions instead of fanning out across every shard of a direct store.
  • The cost. Kafka is ~42% of the bill — not free. But it's the load-bearing decoupling: without it the consumers (and Cassandra, and ClickHouse) would have to be sized for peak with no smoothing, and the SDK contract ("we'll take your event in < 50 ms or tell you to retry") would be impossible.

Per-tenant fairness vs a global rate limit? (isolation)

  • A global limit doesn't isolate. If the rate limit is "5 M events/sec total," one tenant doing 4 M starves the other 49,999 to 1 M combined — and if the limiter is a shared hashmap, that tenant's fingerprint flood evicts everyone's entries. The product promise ("your errors show up regardless of what other customers do") requires per-tenant.
  • The per-customer token bucket is the boundary. Each customer's quota comes from their plan; their flood is rejected at the ingest door once their bucket empties; the consumers never see it. Plus per-customer Count-Min sketches (a flood can't corrupt another tenant's sketch) and a per-customer issue-count cap (a malformed release's blast radius is the offender's <too-many-issues> bucket).
  • The escape hatch: a fleet-wide circuit breaker still exists for the "everything is on fire" case (a coordinated DDoS across many tenants) — but it's the last resort, not the first line; per-tenant limits handle the everyday "one noisy customer."

Why ClickHouse and not Elasticsearch? (query engine)

  • We rank by numbers, not text. The headline query is "issues sorted by event count over 24 h," "issues seen in release 1.4.2," "which release regressed" — numeric aggregations over hundreds of millions of rows. ClickHouse's columnar scan does that at billions of rows/sec/node; Elasticsearch's inverted index is built for full-text and its aggregations over that volume are slower and JVM-GC-spiky.
  • ES has a role — secondary. "Search issue titles/messages for the word 'timeout'" is a full-text query; if we ship that feature, ES (or ClickHouse's own full-text/ngram functions) handles it. It's not the primary ranked list.
  • And not Druid? Close call — also columnar OLAP with real-time ingestion. ClickHouse wins on operational simplicity (one node type vs Druid's coordinator/overlord/historical/broker) and on the join story for "ClickHouse aggregates × Cassandra metadata" at this scale. Both beat ES for this workload.

Grouping by stack trace vs by message? (fingerprint)

  • Message is the unstable one. The same bug throws "User 8123 not found" / "User 9904 not found" — the message has the variable in it. Grouping by message fragments one bug into thousands; you'd have to strip numbers/IDs from the message, which is canonicalization on a weaker signal.
  • Stack trace is stable + informative. The call path doesn't change when the data changes; canonicalizing it (strip line numbers, in-app subset, de-minify) gives a stable id and the culprit (which function). The message is kept as the issue title (the first event's), not as the dedup key.
  • The fallback: a generic Error from a hot library path with no in-app frames has no good stack signal — that's the case where canonicalization falls back to the full stack + a normalized message. Stack-first, message-as-fallback.

Sampling vs full retention — aren't you losing events? (sampling)

  • We don't lose counts. Every accepted event increments the issue counter and the ClickHouse aggregate — the dashboard's "×4.2 M in the last hour" is real (modulo the micro-batch lag). What we don't keep is the full 5 KB payload of all 4.2 M — we keep K≈100 rotating samples per issue, because the 4.2 millionth instance of the same null-deref tells you nothing the 100th didn't.
  • What is dropped: events rejected at the token bucket (over the customer's quota) or diverted to <too-many-issues> (over the issue cap). Those are dropped on purpose — a customer over quota gets a 429 with Retry-After and a "you're dropping events, upgrade or fix your volume" dashboard banner; it's visible, not silent.
  • The full-fidelity escape hatch: a customer can mark a specific issue or project "keep all payloads" (at the cost of their storage quota) — useful when actively debugging a hard issue. The default is sample-K because 30-day full retention of 100 B events/day is 15 PB and ~$1.2 M/mo of storage for ~zero marginal debugging value.

The GDPR / PII angle on stored stack traces? (privacy)

  • The risk is real. A frame argument can be a user's email; a breadcrumb can be a URL with a session token; a context value can be a customer record. The stored sample payloads (S3) and the queryable headers (ClickHouse sampled_events.tags) can carry PII.
  • Layered scrubbing. SDK-side before_send (the customer's first line); server-side scrubbing rules at the ingest worker (drop user.email, mask body fields matching CC/SSN regexes, configurable per customer) before the event hits Kafka; 30-day retention on the PII-bearing sample bodies (they age out).
  • Deletion. "Delete customer X" = a purge job over Cassandra rows + counters, ClickHouse partitions (by customer_id prefix), the S3 prefix, and Redis keys — SLA 30 days. "Delete this end-user's data" = a tagged-event search-and-purge (find sample events where user.id = ..., delete the S3 objects and ClickHouse rows). A sub-day purge SLA / HIPAA-grade regime is Probe C (it would force PII out of the stored payloads entirely — store only a hash, fetch the body from the customer on demand).

What if the issues table itself is hot — a single fingerprint at 5 M events/sec? (hot key)

  • One Cassandra partition. A single (customer, fingerprint) seeing the whole firehose maps to one partition. The micro-batch saves it: 210 consumers, each flushing times_seen += ~24,000 per 10 s, is ~21 counter increments/sec on that partition — trivial. The K-sample reservoir writes for that issue go to per-issue-per-consumer sub-keys merged lazily so they don't contend.
  • ClickHouse side: that issue produces one issue_minute row per minute per consumer (210 rows/min, summed by SummingMergeTree) — a rounding error on a cluster doing ~87 k rows/sec.
  • The real defense is upstream: if one fingerprint is 5 M/sec, that customer's token bucket is the gate — they're provisioned for some quota, the rest is 429'd. A single issue genuinely doing the whole firehose under quota means the customer bought a huge quota; we're sized for it because the consumers are sized for peak regardless.

How do you migrate from Naive without losing events? (migration)

  • Phase 1 — dual ingest: stand up Kafka + consumers + Cassandra rollup; the ingest worker writes to both the Naive Postgres and Kafka for 30 days; the dashboard still reads Postgres. Zero risk — the new path is shadow.
  • Phase 2 — flip reads: dashboard reads the Cassandra rollup; Postgres becomes the fallback. Compare rollup counts vs Postgres GROUP BY for a week.
  • Phase 3 — layer in the rest: Redis token buckets + CMS, then issue-count caps + neg-cache, then canonicalization (with a one-time re-key job on existing issues), then ClickHouse + the regression worker — each behind a per-customer flag, canaried 1% → 10% → 100%.
  • Phase 4 — decommission: drop the Postgres events table once the rollup has ≥ 30 days of data and the comparison holds. Total ~90 days; no event-loss window (the dual-ingest phase means the new path is always behind, never ahead).

S7. Key Takeaways

  • The Logger-Rate-Limiter hashmap is a single-property artifact. "Don't print this message more than once per 10 s" assumes a bounded, shared, stable key space and a discard-don't-count semantics. Real error reporting has an unbounded key space (malformed releases spray fingerprints), a key space that must be per-tenant (one customer can't starve everyone), an unstable key (line numbers / minification fragment one bug), and it must count, not discard. Each broken assumption is one iteration.
  • Fingerprint quality is the product. Everything downstream — issue count, storage, query cost, regression detection, dashboard signal-to-noise — is gated by whether "same fingerprint" means "same bug." Canonicalize before hashing (strip line numbers, collapse recursion, de-minify via source maps, group by the in-app frame subset, allow customer rules), and lean slightly timid (a false merge hides a bug; a false split is merely noisy). hash(raw_stacktrace) fragments one bug ~40×.
  • Per-tenant isolation needs per-tenant state, full stop. A token bucket per customer (event rate), a Count-Min sketch per customer (per-issue rates), an issue-count cap per customer (cardinality) — three different per-tenant structures because a flood, a cardinality bomb, and a rate question are three different failure modes. A shared limiter isolates none of them.
  • Sketches buy bounded state at the cost of bounded error — pick the sketch by the question. "How many events did issue X get?" → Count-Min (per-key frequency; over-estimates ≤ ε×N). "How many distinct users hit issue X?" → HyperLogLog (cardinality). "Which are the K hottest issues?" → Space-Saving. Using HLL where you need CMS (or vice versa) answers the wrong question precisely.
  • Pre-aggregate at write time; don't re-aggregate at read time. The Naive design re-runs a GROUP BY over 100 B rows on every dashboard load. Folding the firehose into an issue rollup at ingest turns the read into a lookup. The cost moved to the consumers (which you were going to need anyway) and the buffer (Kafka) that decouples spiky-in from steady-out.
  • One engine per access pattern: the metadata-vs-numbers split. "Does this fingerprint exist? what's the issue's title/status?" is a high-rate point-upsert + counter workload → LSM/wide-column (Cassandra/Scylla). "Rank 200 k issues by 24 h event count / by release / regression" is an analytical scan-and-aggregate → columnar OLAP (ClickHouse). Same data, two layouts; the consumer is the single writer to both, and the columnar store is replay-rebuildable.
  • Negative caching turns an O(store-read) flood into O(set-membership). Once a fingerprint is known-bad (over the issue cap), remember that — cache the absence/the rejection, with a TTL so it self-heals. Without it, every event of a sustained malformed release does a read-before-write; with it, it's a Redis set lookup and a counter on one bucket.
  • Know the cost shape before you optimize. #42 is ingest-heavy: ~82% of the bill is Kafka + consumers (bytes-in-per-second), barely 13% is storage + query. That's the inverse of #43's storage-heavy (petabytes on S3) and #34's compute-heavy (GPU inference). The diagnostic question: "where's the elephant — the firehose in, the data at rest, the query, or the compute?" For this one it's the firehose, so the knobs that matter are the per-event consumer cost and the buffer sizing, not the storage tier.
