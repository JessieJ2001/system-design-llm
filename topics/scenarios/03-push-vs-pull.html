Push vs Pull: Real-Time Stock Ticker

Designing a system that delivers price updates to 1 million connected users with sub-second latency — and why the naive approach for each model fails.

NALSD Walkthrough • ~15 min read

The Scenario

You are building a real-time stock price service for a brokerage app. Users see live price tickers for stocks they are watching.

MetricValue
Stock symbols tracked10,000
Price update rate~100 updates/sec total (avg 10/sec per symbol, bursty)
Connected subscribers1,000,000 (1M concurrent WebSocket connections)
Symbols per user (watchlist)5–50 (average 20)
Latency target<500 ms from exchange to user's screen
Update payload size~100 bytes (symbol, price, volume, bid, ask, timestamp)

Back-of-Envelope Math

The key question is: how many messages per second does the system need to deliver, and how much bandwidth does that require? The answer is radically different for push vs pull. (Reference: BOE cheat sheet.)

Subscriber Distribution per Symbol

1M users, each watching avg 20 symbols out of 10,000 total.
Total "subscriptions" = 1M × 20 = 20 million.

Average subscribers per symbol:
  20M subscriptions / 10K symbols = 2,000 subscribers/symbol

But the distribution is highly skewed (Zipf's law):
  - Top 10 stocks (AAPL, TSLA, AMZN, etc.): ~100,000 subscribers each
  - Top 100 stocks:                           ~10,000 subscribers each
  - Top 1,000 stocks:                         ~2,000 subscribers each
  - Bottom 9,000 stocks:                      ~100 subscribers each

Push Model: Message Volume

For average stocks:
  100 updates/sec × avg 2,000 subscribers = 200,000 messages/sec
  200K msgs × 100 bytes = 20 MB/sec outbound

For popular stocks (AAPL with 100K subscribers):
  10 updates/sec × 100,000 subscribers = 1,000,000 messages/sec
  Just for ONE stock!

Worst case (market open, all stocks active):
  Total fan-out messages: up to 2-5 million messages/sec
  Bandwidth: 200-500 MB/sec outbound

Pull Model: Request Volume

Polling every 1 second:
  1M users × 1 request/sec = 1,000,000 requests/sec
  Each response: 20 symbols × 100 bytes = 2 KB
  Bandwidth: 1M × 2 KB = 2 GB/sec

Polling every 5 seconds:
  200,000 requests/sec
  Bandwidth: 400 MB/sec
  But data is up to 5 seconds stale — unacceptable for traders.

Polling every 100 ms (near real-time):
  10,000,000 requests/sec
  Bandwidth: 20 GB/sec
  → Catastrophically expensive.

Pull does not scale for real-time data. At 1-second polling intervals, we need 1M req/sec and 2 GB/sec of bandwidth. Most of those responses contain no new data (the stock price has not changed in the last second). This is wasted work.

MetricPushPull (1s interval)
Messages/sec200K–5M (proportional to actual changes)1M (constant, regardless of changes)
Bandwidth20–500 MB/sec2 GB/sec
Latency<100 ms (immediate push)0–1,000 ms (depends on poll timing)
Wasted workNone (only send when data changes)High (most polls return unchanged data)
Connection overhead1M persistent WebSockets (~10 GB RAM)Stateless HTTP (but 1M req/sec)

Push wins for real-time, high-frequency data. Push only sends data when something changes, and it delivers instantly. Pull wastes bandwidth on empty polls and cannot match the latency.

Naive Design: Direct WebSocket Push

Each user opens a WebSocket connection to the server. The server maintains a mapping of which user is watching which symbols. When a price update arrives, the server looks up all subscribers and pushes the update to each one.

# Naive push implementation (single server)
import asyncio
import websockets

# symbol -> set of connected websocket clients
subscriptions = {}  # {"AAPL": {ws1, ws2, ...}, "TSLA": {ws3, ...}}

async def handle_client(websocket):
    """Client connects and sends their watchlist."""
    watchlist = await websocket.recv()  # e.g., "AAPL,TSLA,GOOG"
    symbols = watchlist.split(",")
    for symbol in symbols:
        subscriptions.setdefault(symbol, set()).add(websocket)
    try:
        await websocket.wait_closed()
    finally:
        for symbol in symbols:
            subscriptions[symbol].discard(websocket)

async def handle_price_update(symbol, price_data):
    """Called when a new price arrives from the exchange."""
    subscribers = subscriptions.get(symbol, set())
    message = json.dumps({"symbol": symbol, **price_data})
    # Fan-out: send to every subscriber
    await asyncio.gather(
        *[ws.send(message) for ws in subscribers]
    )

What Breaks

Problem 1: Fan-Out for Popular Stocks

AAPL has 100,000 subscribers.
AAPL updates 10 times/sec.

Fan-out per second for AAPL alone:
  10 updates × 100K subscribers = 1,000,000 sends/sec

Each WebSocket send ≈ 10 μs (including kernel buffer copy)
Time to fan out 1 update to 100K clients:
  100,000 × 10 μs = 1 second

But the next update arrives in 100 ms!
We can't even finish sending one update before the next arrives.
The server falls behind and latency grows without bound.

Problem 2: Connection Limits

1M WebSocket connections on a single server:
  - Each connection: ~10 KB memory (kernel socket buffers + app state)
  - 1M × 10 KB = 10 GB RAM just for connections
  - File descriptors: 1M (need to tune ulimit)
  - Typical Linux server: ~100K-500K concurrent connections practical limit

A single server cannot hold 1M connections.
Even with 500K per server, we need at least 2 servers.
And the fan-out problem exists on each server independently.

Problem 3: Single Point of Failure

All subscription state is in memory on one server.
If the server crashes:
  - All 1M connections drop
  - All subscription mappings are lost
  - Every client must reconnect and re-subscribe
  - Thundering herd: 1M reconnections hit the new server simultaneously

Iteration 1: Pub-Sub with Edge Servers

The key insight: separate the publishing of price updates from the delivery to subscribers. Use a pub-sub message broker in between, and multiple edge servers to handle connections.

Architecture

How It Works

  1. Price update arrives from the exchange feed. The ingestion service publishes it to the relevant Kafka topic (one topic per symbol).
  2. Edge servers subscribe to topics based on which symbols their connected users are watching. If any user on Edge Server 1 watches AAPL, that edge server subscribes to the AAPL topic.
  3. Edge server receives the update once from Kafka, then fans out locally to all connected users watching that symbol.

Why This Helps: Fan-Out Math

Without pub-sub (naive):
  AAPL update → fan out to 100K subscribers directly = 100K sends

With pub-sub (20 edge servers):
  AAPL update → Kafka → 20 edge servers (20 sends from Kafka)
  Each edge server: 100K / 20 = 5,000 local subscribers for AAPL
  Local fan-out per edge server: 5,000 sends

  Per-server work: 5,000 sends × 10 μs = 50 ms ✓ (well within budget)
  Compare to naive: 100,000 sends × 10 μs = 1,000 ms ✗

The broker multiplies the update 20x (one per edge server).
Each edge server only fans out to its own connected users.

Edge Server Sizing

Target: 1M connections across N edge servers.
Per-server limit: ~50,000 connections (conservative for good perf).
Servers needed: 1M / 50K = 20 edge servers.

Memory per edge server:
  50K connections × 10 KB = 500 MB for connections
  Subscription map: 50K users × 20 symbols = 1M entries × 24 bytes = 24 MB
  Total: ~600 MB — trivial.

CPU per edge server (fan-out for AAPL update):
  5,000 sends × 10 μs = 50 ms per AAPL update
  10 AAPL updates/sec × 50 ms = 500 ms of CPU per second
  Plus other symbols: ~1-2 CPU cores for fan-out
  → Comfortable on a 4-core server.

The pub-sub tier decouples the publisher from the subscribers. The price feed does not need to know how many users are watching each symbol. It just publishes to a topic. The edge servers handle the last-mile delivery. Adding more users = adding more edge servers, without changing the publisher.

Iteration 2: Batching and Delta Compression

Iteration 1 solves the fan-out problem, but we can reduce bandwidth and message count further.

Problem: Too Many Small Messages

Each user watches 20 symbols.
If all 20 symbols update within 1 second:
  20 individual WebSocket messages per user per second.

1M users × 20 msgs/sec = 20M WebSocket frames/sec total.

Each WebSocket frame has overhead:
  - 2-10 bytes framing
  - ~50 bytes TCP/IP headers (with Nagle disabled for low latency)
  - 100 bytes payload
  Total per frame: ~160 bytes

20M frames × 160 bytes = 3.2 GB/sec
Most of it is overhead, not payload!

Solution: Batch Updates Every 100 ms

Instead of pushing each update individually, buffer updates on the
edge server for 100 ms, then send a single batched message per user.

Every 100 ms:
  - Collect all price changes for symbols this user watches
  - If any changed, send ONE WebSocket message with all changes

Example batch message for a user watching 20 symbols:
{
  "ts": 1711036800123,
  "updates": [
    {"s": "AAPL", "p": 178.52, "v": 1523400},
    {"s": "TSLA", "p": 171.03, "v": 892100},
    {"s": "GOOG", "p": 153.21, "v": 445000}
  ]
}
Only 3 out of 20 symbols changed in this 100ms window.

Message count reduction:
  Before: 20 individual messages per user per second
  After:  10 batched messages per second (one per 100ms window)
  But many windows have 0 changes → realistically ~3-5 messages/sec

Bandwidth per user:
  Before: 20 × 160 bytes = 3,200 bytes/sec
  After:  5 × 250 bytes = 1,250 bytes/sec (61% reduction)

Delta Compression

Most price updates change only price and volume.
The symbol name, bid/ask spread structure don't change.

Full update: {"s":"AAPL","p":178.52,"v":1523400,"b":178.50,"a":178.54}
  = ~65 bytes

Delta update: {"s":"AAPL","p":178.52}
  = ~22 bytes (66% smaller)

Only send fields that changed since the last update sent to this user.
Edge server keeps track of last-sent state per user per symbol.

Combined savings (batching + delta):
  Before: 3,200 bytes/sec/user × 1M users = 3.2 GB/sec
  After:  ~400 bytes/sec/user × 1M users = 400 MB/sec
  → 8x bandwidth reduction

Latency Trade-Off

Batching adds up to 100 ms of latency:
  - Price change occurs at t=0
  - Kafka delivers to edge server at t=5ms
  - Edge server buffers until next 100ms batch at t=5-105ms
  - WebSocket send at t=105ms
  - Client receives at t=110ms

Total worst case: ~110 ms (within 500 ms target)
Total best case: ~15 ms (if change arrives just before batch flush)
Average: ~60 ms

For a brokerage app, 60 ms average is excellent.
For high-frequency trading, you'd skip batching entirely
and push every update immediately (they have dedicated feeds anyway).

Batching interval is configurable per user tier. Premium users (day traders) get 50 ms batching. Free tier gets 500 ms. This lets you scale server costs with revenue.

Final Design Summary

ComponentDetailsSizing
Price ingestion Consumes exchange feed, publishes to Kafka 2 instances (active/standby). 100 updates/sec, trivial load.
Message broker Kafka with 10K topics (one per symbol) 3-node cluster. 100 msgs/sec ingest, 2K msgs/sec fan-out to edge servers.
Edge servers WebSocket servers with local fan-out + 100ms batching 20 servers, 50K connections each. ~600 MB RAM, 2 CPU cores each.
Load balancer L4 (TCP) LB for WebSocket connections Sticky sessions (same user reconnects to same edge server).
Subscription service Manages user watchlists, persists to DB Watchlist changes are infrequent. Simple CRUD service.

Request Flow: Price Update End-to-End

Request Flow Steps

1. NYSE sends AAPL price update: $178.52
2. Ingestion service receives, publishes to Kafka topic "AAPL"      (2 ms)
3. Kafka delivers to all 20 edge servers that have AAPL subscribers  (3 ms)
4. Edge Server 7 receives update, has 5,000 users watching AAPL
5. Edge Server 7 buffers update in current 100ms batch               (0-100 ms)
6. Batch flushes: for each user watching AAPL, include in their batch
7. WebSocket send to each user (batched with other symbol updates)   (5 ms)
8. User's browser receives and renders: "$178.52"

Total latency: 10-110 ms (avg ~60 ms) ✓ well under 500 ms target

Scaling Considerations

If users grow to 10M:
  Edge servers: 10M / 50K = 200 servers
  Kafka fan-out: 200 edge servers × 100 updates/sec = 20K msgs/sec
    → Still trivial for Kafka (handles 100K+ msgs/sec per partition)
  Bandwidth: 400 MB/sec × 10 = 4 GB/sec total outbound
    → Need CDN-like infrastructure or regional edge deployments

If update rate grows to 10,000/sec (more symbols, options, crypto):
  Kafka throughput: 10K msgs/sec ingest → still fine
  Edge server CPU: 10x more updates to batch → ~20 CPU cores per server
    → Scale up to 8-core servers or add more edge servers

Key Takeaways

  • Push for real-time, high-frequency data. Push only sends data when something changes, delivering it instantly. Polling wastes bandwidth on empty responses and adds latency equal to the poll interval.
  • Pull for infrequent or on-demand data. If data changes rarely (e.g., user settings, static content), polling is simpler and avoids the cost of maintaining persistent connections.
  • Use a pub-sub tier for high fan-out. A message broker decouples publishers from subscribers. The publisher does not need to know about 100K subscribers; it publishes once to a topic. Edge servers handle last-mile delivery.
  • Batching reduces overhead dramatically. Sending one message every 100 ms instead of 20 individual messages per second reduces frame overhead by 75%+ and saves bandwidth at the cost of small added latency.
  • Delta compression reduces payload size. Sending only changed fields (instead of the full record every time) can reduce bandwidth by 60%+. The edge server tracks last-sent state per user.
  • Design for the skewed case. The top 10 stocks may have 50x more subscribers than the median stock. If your design works for AAPL's 100K subscribers, it works for everything.
