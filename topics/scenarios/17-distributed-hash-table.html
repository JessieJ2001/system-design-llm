Distributed Hash Table: Chord-Style Key-Value Store

Design a peer-to-peer key-value store where 1,000 nodes cooperate to store 10 billion keys — with O(log N) lookups and automatic key redistribution when nodes join or leave.

NALSD Walkthrough • ~15 min read

The Scenario

You are building a distributed key-value store that does not rely on a central coordinator. There is no “master” node — every node is equal. The system must self-organize so that any node can route a lookup to the correct destination in a small number of network hops.

Concrete Numbers

ParameterValue
Total nodes1,000
Total key-value pairs10 billion
Average value size1 KB
Total data10 TB
Node churn rate1 join or leave per minute
Lookup latency target< 10 ms within a data center

Requirements

  • O(log N) lookups — a lookup should contact at most ~10 nodes, not all 1,000.
  • Automatic key redistribution — when a node joins or leaves, keys move to maintain balance without any manual intervention.
  • No central coordinator — the system must be fully decentralized.
  • Fault tolerance — if a node crashes, its keys should still be available.

Back-of-Envelope Math

Storage & Latency Estimates

Storage per node:
  10 TB / 1,000 nodes = 10 GB per node
  → Fits comfortably in RAM (modern servers have 64-256 GB)
  → Even on SSD, 10 GB is trivially small

Keys per node:
  10 billion / 1,000 = 10 million keys per node
  → Hash map with 10M entries ≈ 10M × (8B key hash + 8B pointer) = 160 MB overhead
  → Plus 10M × 1 KB values = 10 GB. Total ≈ 10.2 GB per node.

Lookup & Churn Estimates

Lookup hops (Chord with finger tables):
  O(log₂ N) = log₂(1,000) ≈ 10 hops

Latency per hop:
  Same data center: ~0.5 ms RTT (network round-trip)

Total lookup latency:
  10 hops × 0.5 ms = 5 ms  ✓  (well under 10 ms target)

Node churn:
  1 node change per minute = 1 every 60 seconds
  When a node leaves, its 10M keys (10 GB) must transfer to the successor
  At 10 Gbps network: 10 GB / 1.25 GB/sec = 8 seconds to transfer
  → Churn is manageable (60 sec between events, 8 sec to handle each)

Key insight: The math tells us this is feasible. Each node stores a manageable amount of data, lookups finish in milliseconds, and the network can handle key transfers faster than new churn events arrive. The challenge is purely algorithmic: how do we route lookups efficiently without a central directory?

Naive Design: Successor-Only Ring

The simplest decentralized approach: arrange all nodes in a ring. Each node only knows its successor (the next node clockwise). To look up a key, forward the request around the ring one hop at a time until you reach the node that owns the key.

How the Ring Works

Both nodes and keys are hashed onto a circular identifier space [0, 2m). For SHA-1, m = 160, giving a ring of 2160 positions. A key is stored on the first node whose position is equal to or follows the key’s hash (its successor).

Naive Lookup: Walk the Ring

Looking up key at position 30, starting at Node 1:

  Step 1: Node 1 receives query for key 30.
          Node 1's successor is Node 8.
          30 is not in (42, 1], so forward to Node 8.

  Step 2: Node 8's successor is Node 14.
          30 is not in (1, 8], forward to Node 14.

  Step 3: Node 14's successor is Node 21.
          30 is not in (8, 14], forward to Node 21.

  Step 4: Node 21's successor is Node 32.
          30 is not in (14, 21], forward to Node 32.

  Step 5: Node 32 owns range (21, 32]. Key 30 is in this range!
          Node 32 returns the value.

  Result: 4 hops. With 6 nodes this is already O(N).

What Breaks

O(N) Lookups Are Unacceptable

With 1,000 nodes and successor-only routing:

  Average hops per lookup: N/2 = 500 hops
  Latency per hop: 0.5 ms
  Total lookup latency: 500 × 0.5 ms = 250 ms

  Compare to our target: < 10 ms

  This is 25× too slow.

Linear Scanning Wastes Network Bandwidth

Every lookup that starts far from the target traverses hundreds of nodes. Each hop consumes network bandwidth and CPU on an intermediate node that has nothing to do with the requested key. Under load (millions of lookups per second), intermediate nodes become bottlenecks even though they are just forwarding messages.

The Core Problem

Each node only knows about its immediate neighbor. It has no way to “skip ahead” to a node closer to the target. We need each node to maintain a small routing table that lets it jump large distances around the ring.

Iteration 1: Chord Finger Tables

The Chord protocol (Stoica et al., 2001) solves the O(N) problem by giving each node a finger table — a routing table with m entries (where m is the number of bits in the hash, e.g., 160 for SHA-1). Each entry lets the node jump exponentially farther around the ring.

Finger Table Construction

For a node at position n, the i-th finger table entry (0-indexed) is:

finger[i] = successor of (n + 2^i) mod 2^m

For our simplified ring [0, 48) with m = 6 bits (2^6 = 64 ≈ 48):

Node 1's finger table:
  finger[0]: successor of (1 + 1)  = successor of 2  → Node 8
  finger[1]: successor of (1 + 2)  = successor of 3  → Node 8
  finger[2]: successor of (1 + 4)  = successor of 5  → Node 8
  finger[3]: successor of (1 + 8)  = successor of 9  → Node 14
  finger[4]: successor of (1 + 16) = successor of 17 → Node 21
  finger[5]: successor of (1 + 32) = successor of 33 → Node 42

Node 21's finger table:
  finger[0]: successor of (21 + 1)  = successor of 22 → Node 32
  finger[1]: successor of (21 + 2)  = successor of 23 → Node 32
  finger[2]: successor of (21 + 4)  = successor of 25 → Node 32
  finger[3]: successor of (21 + 8)  = successor of 29 → Node 32
  finger[4]: successor of (21 + 16) = successor of 37 → Node 42
  finger[5]: successor of (21 + 32) = successor of 5  → Node 8

The intuition: Finger table entries point to nodes at distances 1, 2, 4, 8, 16, 32, … around the ring. This means a node can jump halfway around the ring in one hop, then a quarter, then an eighth, and so on. Each hop halves the remaining distance — exactly like binary search. That is why lookup takes O(log N) hops.

Lookup with Finger Tables: Concrete Trace

Looking up key at position 30, starting at Node 1:

  Step 1: Node 1 wants to find the successor of position 30.
          Scan finger table from largest to smallest:
            finger[5] → Node 42 at position 42.  42 > 30?
              But 42 is past the target (in ring terms). Skip.
            finger[4] → Node 21 at position 21.  21 < 30? Yes!
              Node 21 is the largest finger that doesn't overshoot.
          Forward query to Node 21.

  Step 2: Node 21 wants to find successor of 30.
          Scan finger table:
            finger[4] → Node 42 at position 42.  Overshoots. Skip.
            finger[3] → Node 32 at position 32.  Overshoots. Skip.
              Wait — does 32 overshoot? 32 > 30, but Node 32 might be
              the successor of 30. Check: Node 21's successor for range
              (21, 32] is Node 32. Is 30 in (21, 32]? Yes!
          Key 30 belongs to Node 32. Forward to Node 32.

  Step 3: Node 32 is the owner. Return the value.

  Result: 2 hops! (Down from 4 with naive routing.)
  With 1,000 nodes: ~10 hops instead of ~500.

Lookup Algorithm in Pseudocode

def find_successor(node, target_id):
    """Find the node responsible for target_id."""
    if target_id is in range (node.id, node.successor.id]:
        return node.successor

    # Find the closest preceding finger
    closest = closest_preceding_node(node, target_id)
    # Recursively (or iteratively via RPC) ask that node
    return find_successor(closest, target_id)

def closest_preceding_node(node, target_id):
    """Search finger table for the largest node that precedes target_id."""
    for i in range(m - 1, -1, -1):  # m-1 down to 0
        if node.finger[i].id is in range (node.id, target_id):
            return node.finger[i]
    return node  # fallback: no finger precedes target

The Stabilization Protocol

Finger tables must stay accurate as nodes join and leave. Chord uses a stabilization protocol that runs periodically on every node (e.g., every 10 seconds):

Node Join — Node P joins the ring at position P:

  1. P contacts any existing node (a "bootstrap" node it already knows).
  2. P asks that node: "Who is the successor of position P?"
     → This is a standard Chord lookup: O(log N) hops → returns node S.
  3. P sets its successor = S.
  4. P's predecessor = S's old predecessor (call it Q).
  5. S updates its predecessor = P.
  6. Keys in range (Q, P] that were owned by S are transferred to P.

  Key transfer:
    10M keys × 1 KB = 10 GB per node on average.
    But P only takes keys in its range, not all of S's keys.
    On average, P takes 1/N of total keys = 10M keys.
    Transfer time at 10 Gbps: 10 GB / 1.25 GB/s = 8 seconds.

Periodic stabilization (every 10 seconds, on every node):

  1. Ask successor: "Who is your predecessor?"
     If successor's predecessor is a new node between me and successor,
     update my successor to that new node.
  2. Tell my successor: "I think I'm your predecessor."
     Successor updates its predecessor if I'm closer.
  3. Refresh one random finger table entry:
     finger[i] = find_successor(node.id + 2^i)

  With 1,000 nodes, each refreshing one finger every 10 seconds,
  all finger tables converge within ~160 seconds (m = 160 entries,
  one refreshed per cycle).

During stabilization, lookups may fail. If a node has just joined but its predecessor’s finger table hasn’t updated yet, a lookup may be routed to the wrong node. The system is eventually consistent — finger tables converge, but there is a window where lookups may return “key not found” even though the key exists. This is addressed in Iteration 2 with replication.

Iteration 2: Replication and Fault Tolerance

In Iteration 1, each key lives on exactly one node. If that node crashes, the key is gone. We need replication.

Successor-List Replication

Each key is stored on its successor and the next r − 1 successors in the ring. With r = 3:

If Node 32 crashes:
  1. Node 21 detects its successor (32) is dead via heartbeat timeout (5 sec).
  2. Node 21 updates its successor to Node 42 (next in successor list).
  3. Node 42 becomes the primary for keys in range (21, 32].
     It already has these keys (it was Replica 2). No data transfer needed!
  4. Node 42 creates a new 3rd replica on Node 8 (its successor's successor).
     Transfer: ~10M keys × 1 KB = 10 GB at 10 Gbps → 8 seconds.

Availability during failure:
  - Read: Any of the 3 replicas can serve reads. As long as 1 survives, data is available.
  - Write: Writes go to the primary. If primary dies, the new primary (next successor) takes over.
  - Simultaneous failures: To lose data, all 3 consecutive nodes must crash.
    Probability with independent failures (1% failure rate per node):
    0.01³ = 0.000001 = 0.0001% per key. Acceptable.

Read Load Balancing

With 3 replicas per key, read requests can be sent to any replica. The client (or routing layer) can pick the closest or least-loaded replica. This triples read capacity for popular keys.

Consistency Model

Write path (quorum-based):
  1. Client sends write to primary (Node 32).
  2. Primary writes locally and forwards to Replicas 2 and 3.
  3. Primary waits for W acknowledgments before responding to client.
     With W=2 (out of R=3 replicas): tolerates 1 slow/dead replica.

Read path:
  - For strong consistency: read from primary only (or use read quorum).
  - For eventual consistency: read from any replica (faster, but may be stale).
  - Read-repair: if a read finds stale data on a replica, update it in the background.

Quorum condition for strong consistency:
  W + R > N  →  2 + 2 > 3  →  True ✓
  (Write quorum 2 + Read quorum 2 > Replication factor 3)

Iteration 3: Virtual Nodes for Load Balance

With 1,000 physical nodes on a ring, the arc lengths between adjacent nodes vary widely. Some nodes own a large arc (many keys), others own a tiny arc. The standard deviation of load is proportional to 1/√N — with N = 1,000, some nodes may have 2–3× more keys than average.

The Problem

Ideal: each node stores 10M keys (10 billion / 1,000).
Reality with 1,000 random positions on the ring:
  - Expected max load ≈ (ln N / ln ln N) × average
  - For N = 1,000: ≈ (6.9 / 1.9) × 10M ≈ 36M keys on the most loaded node.
  - That's 3.6× the average — 36 GB instead of 10 GB.
  - The least loaded node might have only 2-3M keys.

Solution: Virtual Nodes

Each physical node creates V virtual nodes at different ring positions. Instead of 1,000 points on the ring, we have 1,000 × V points.

With V = 100 virtual nodes per physical node:

  Total ring positions: 1,000 × 100 = 100,000 virtual nodes

  Each physical node "owns" 100 non-contiguous arcs on the ring.
  The keys assigned to any of those 100 virtual nodes are stored
  on the same physical machine.

  Load balance improvement:
    Standard deviation ∝ 1/√(N×V) = 1/√100,000 ≈ 0.3%
    Max load ≈ average × (1 + a few percent) ≈ 10.3M keys
    vs. without virtual nodes: up to 36M keys

  Memory overhead for finger tables:
    Each virtual node needs its own finger table.
    100 virtual nodes × 160 entries × 8 bytes = 128 KB per physical node.
    Negligible.

Benefit During Node Departure

Without virtual nodes:
  When a node leaves, ALL its 10M keys go to its single successor.
  That successor temporarily holds 20M keys (2× average) until
  stabilization redistributes them. Spike in load on one node.

With 100 virtual nodes:
  The departing node's 10M keys are spread across 100 virtual nodes.
  Each virtual node's keys go to a DIFFERENT successor.
  100 different physical nodes each absorb ~100K keys (1% increase).
  No load spike on any single node.

Final Design

Putting all iterations together, here is the complete Chord-style DHT:

Architecture Summary

ComponentDescription
Ring space[0, 2160) using SHA-1 hashes
Physical nodes1,000 servers, each with 64+ GB RAM, SSD
Virtual nodes100 per physical node = 100,000 ring positions
Finger tables160 entries per virtual node for O(log N) routing
Replicationr = 3 replicas on consecutive successors
ConsistencyQuorum reads/writes (W=2, R=2, N=3)
StabilizationEvery 10 seconds: verify successor, refresh one finger
Failure detectionHeartbeat every 2 seconds, timeout after 3 missed = 6 sec

Request Flow: Lookup

Client wants to GET key "user:12345":

  1. Client hashes "user:12345" → SHA-1 → position P on the ring.
  2. Client sends request to any known node (e.g., Node A).
  3. Node A checks: is P in my range? If yes, return the value.
     If no, use finger table to find closest preceding node → forward.
  4. After O(log N) ≈ 10 hops, request reaches Node X that owns P.
  5. Node X looks up key in its local hash map → returns value.

  Total latency: 10 hops × 0.5 ms = 5 ms.

Optimization: Iterative Routing

Instead of each node forwarding to the next (recursive),
each node returns the next hop's address to the CLIENT.
The client contacts each hop directly.

Advantage: if a hop fails, the client can retry with
a different finger. No request "stuck" inside the ring.

Same number of hops, but the client controls the timeout.

Key Takeaways

  • Finger tables are the key optimization. By storing O(log N) routing entries that point to exponentially increasing distances around the ring, each node can halve the remaining distance to the target on every hop — giving O(log N) lookups with O(log N) state per node.
  • Stabilization handles churn. The periodic protocol (verify successor, refresh one finger) ensures the ring self-heals when nodes join or leave. Convergence happens within minutes, and the system remains functional (with slightly suboptimal routing) during convergence.
  • Replication handles failure. Storing each key on r = 3 consecutive successors means any single node crash causes zero data loss and zero downtime. The successor list lets the ring route around dead nodes immediately.
  • Virtual nodes handle load balance. Mapping each physical node to 100 ring positions reduces the max-to-average load ratio from 3.6× to ~1.03×. It also distributes key transfers across many nodes when a physical node departs.
  • The math drove every decision. O(N) = 500 hops told us we needed finger tables. The 3.6× load variance told us we needed virtual nodes. The 10 GB transfer time told us churn is manageable.
