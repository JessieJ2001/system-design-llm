Designing Search Autocomplete / Query-Suggest

Google-search-style as-you-type suggestions: every keystroke returns the top-10 completions of the typed prefix, ranked by historical frequency × recency, blended with what's trending now, lightly personalized — 100 k suggest QPS over 10 B unique historical queries, top-10 in ≤ 50 ms p99 (network + compute) from a 50-region edge, trending events visible within ≤ 5 min. We start from one node holding the LC #212 "Word Search II" pointer trie in RAM (every historical query inserted, each node a hash-map of children; on a keystroke, walk to the prefix node, DFS the subtree, sort every completion by stored frequency, return 10) — which doesn't fit a node, blows the latency budget the moment someone types "a", lives in one region 150 ms from half the planet, and can't reflect a trending event until the next rebuild — and iterate to a precomputed prefix→top-10 map packed into a minimal-acyclic FST (mmap-able, read-only, tens of GB not terabytes), replicated to 50 edge POPs with single-flight cache-fill, layered with a Count-Min trending overlay and a per-user re-rank, with a hot-prefix shard pinned in every POP.

NALSD Walkthrough • ~55 min read • LC #212 (Word Search II / trie + DFS)

LC kernel. Word Search II (LC #212). Textbook structure: a trie — a tree where each node is a hash-map (or array) of child characters; insert a word by walking/creating one node per character and marking the last node terminal; "find all words with prefix P" by walking to P's node then DFS-ing the subtree, collecting every terminal you pass. O(corpus chars) space; a prefix query is O(|P| + size-of-subtree). Four properties that break at 10 B queries / 100 k QPS / 50 ms / 50 regions: (1) size — a pointer trie over 10 B queries (~250 GB of raw text) with a hash-map child node per character is ~1–3 TB; it doesn't fit a node, and a pointer-chasing DFS over it is cache-hostile; (2) it enumerates, it doesn't rank — "find all completions of a" walks millions of subtree nodes and only then sorts by frequency; the LC trie has no notion of returning just the best 10 in O(|P|); (3) it's built once and queried offline — the real thing serves 100 k QPS at the edge with a 50 ms (network + compute) budget across 50 regions, and a trending event must surface within 5 min — a monolithic in-RAM trie rebuilt on a schedule can't do that; (4) no typo / normalization handling — users type "resturant", "CAFÉ", mixed scripts. The C8 production cousin replaces the pointer trie with a minimal acyclic finite-state transducer (FST) — byte-labeled transitions, shared suffixes, the output at the end of each prefix's path is a fixed-size precomputed top-10 — mmap-able read-only, tens of GB, an O(|P|) walk that returns the ranked answer with no DFS and no sort; replicated to a 50-POP edge tier; merged with a streaming trending overlay and a per-user re-rank. (Cite storage-engines#fst.)

S0. Scenario & Requirements

The query box of a large web search engine (or a large e-commerce / video / maps search). As a user types each character, a request goes out with the current prefix; the service returns the top-10 query completions, ordered by a ranking score that blends how often this query has been searched historically (with a recency decay, so last year's trends fade), what's trending right now (a celebrity name, a breaking-news term, a just-launched product — visible within minutes of the spike), and a light personalization signal (the user's own recent searches, their language, their region). This is the suggest service — not the search backend that runs the query once the user hits enter (that's a separate, much heavier inverted-index system; see #44's neighbours). The load-bearing tension: a suggestion must come back faster than the user types the next key (the 50 ms p99 budget includes the round trip from wherever they are), and the index it's served from is sized by 10 B historical queries while needing to reflect events that happened five minutes ago.

System class

Class: product, with an algorithmic kernel — see functional-requirements.html#class-product. It's a user-facing capability ("type, see suggestions"); the product framing brings the latency, freshness, and edge-presence NFRs to the front. The algorithmic-kernel framing brings the C8 trie→FST swap — the LC #212 trie is the structure that breaks, and the iterations are about what replaces it. (An offline pipeline — the nightly job that rolls query logs into the FST — sits behind it; that pipeline is a supporting actor, not the system class.)

Functional requirements (cite functional-requirements.html)

  • Suggest. Given a prefix string P (and an optional user/session context), return up to 10 query completions of P, ranked by the blended score, with the score's components visible enough that the client can highlight "trending".
  • Rank by frequency × recency. A completion's base score is a function of how many times it has been searched (heavily damped — log-scaled) times a recency-decay factor (a query searched a million times last week beats one searched a million times two years ago).
  • Reflect trending. When a query's recent search rate spikes, it appears among the suggestions of its prefixes within ≤ 5 min, even though the offline index is rebuilt far less often.
  • Personalize lightly. Blend a small per-user signal — the user's own recent searches, their UI language, their coarse region — into the ranking, without making any guarantee that suggestions are unique per user.
  • Ingest the query stream. Every executed search feeds back into the system: into the offline log (for the next FST rebuild) and into the live trending counter.
  • Manage the corpus. Operators can trigger a rebuild, adjust the inclusion threshold (which prefixes get a precomputed list), and apply a blocklist (some prefixes/completions must never be suggested).

Out of scope (named, not silently dropped): the actual search execution after enter — that's the search backend, a different system; spelling-correction "did you mean" on the results page — we do typo-tolerant prefix matching (a probe), not full query rewriting; voice / image query input — text-prefix only here; per-user search history UI — we read a small per-user feature blob, we don't own the history product; ads / sponsored suggestions — a separate ranked slot, out of scope; full CJK input-method-editor integration — we handle CJK segmentation for the index (a probe), not the IME.

Flagged assumptions (would confirm with the interviewer): (1) the 50 ms p99 budget is end-to-end including the network from the user's device — if it were "server-side only" the edge tier would be a nice-to-have, not load-bearing; (2) "trending freshness ≤ 5 min" means from the spike being detectable in the global query stream to the suggestion appearing — not from the very first search of a brand-new term; (3) personalization is a re-rank of a global candidate set, not a per-(prefix, user) precomputation — 10 B prefixes × hundreds of millions of users is not a thing you precompute; (4) the corpus is ~10 B unique queries averaging ~25 characters — the long tail (queries searched once) may be pruned out of the precomputed index, falling back to a degraded path or no suggestion; (5) one global service with regional/language sharding — not a separate product per market.

ID scheme (cite identifier-schemes.html): query_id = hash64(normalized_query_text) — a 64-bit content hash of the normalized query string. Why: deterministic, so the offline build, the trending stream, and the serving tier all derive the same id with zero coordination and no registry lookup; fixed-size; uniform; and content-addressed means an identical query searched in two regions is the same entity automatically (cite identifier-schemes#hash-content). Rejected: a monotonic sequence / Snowflake (needs a coordinator + a registry on the build path, and "the same query" would get two ids if two pipelines saw it independently); UUIDv7 (random — can't be re-derived from the text, so every reference needs a lookup table). The prefix has no separate id — it is its byte path through the FST; that's the whole point of the FST (the structure is the index of prefixes). Build epochs are monotonic integers (a counter bumped per rebuild), used to detect "is this POP serving a stale FST?".

Per-endpoint SLO

  • GET /v1/suggest — p99 ≤ 50 ms, p50 ≤ 20 ms, end-to-end including the network (the headline SLO; drives the edge tier, the FST being mmap-resident, and the per-request work budget).
  • POST /v1/searches — p99 ≤ 20 ms to 202 (just "appended to Kafka + bumped the sketch"; the search backend doesn't wait).
  • POST /v1/rebuild — returns 202 immediately; the base rebuild completes within a few hours, the recent rebuild within a few minutes (it's the freshness path).
  • PUT /v1/blocklist — p99 ≤ 200 ms; propagated to all POPs within ~30 s (a blocklist change is urgent — you're suppressing something harmful — so it ships out-of-band of the FST, faster than a recent rebuild).

S3. Back-of-Envelope — the ratios that govern every iteration

Three numbers shape the whole arc. No optimization technique appears here — the FST, the edge tier, the trending overlay, and the per-user re-rank are named as future moves but are not used in any calculation below; the math is about the naive trie and the raw constraints only.

The corpus — how big is the trie?

N_queries = 10 B unique queries, avg ~25 chars  ->  ~250 GB of raw query *text* (a lower bound on anything indexing it).
A pointer trie inserts ~one node per character (shared prefixes collapse some, but the long tail of distinct queries
keeps most characters distinct):  ~250 B-nodes worth of structure.  Each interior node is a hashmap of children:
  - a small hashmap (Java HashMap, Go map, C++ unordered_map): ~48-64 B object header + buckets + per-entry ~16-32 B
  - amortized over a node's children: easily ~50-150 B per trie node beyond the 1 B character.
=> trie size ~= 250 GB (chars) x (1 + ~5-12x overhead) ~= 1-3 TB.
The biggest box on the menu (x2idn.16xlarge) is 1 TB -> DOESN'T FIT, and even if it did, a pointer-chasing DFS over a 1+ TB
structure is ~100 ns per hop x millions of hops for a short popular prefix's subtree.   Memory-capacity-bound, hard wall.
(The fix that shrinks ~1-3 TB -> ~tens of GB is the FST -- Iteration 0 -- not used in this section.)

The popular-prefix DFS — how slow is one keystroke?

latency_budget = 50 ms (network + compute), and the network alone from a far user to one region is ~100-150 ms RTT
(US-east <-> Australia ~= 150 ms; US-east <-> Europe ~= 80-100 ms).  So a SINGLE-REGION naive design fails the budget
for half the planet on the network leg ALONE, before any compute.   (The fix is the 50-POP edge -- Iteration 1.)

And the compute: for a popular short prefix like "a", the LC trie's "find all completions" DFS-es the WHOLE subtree.
"a..." matches a large fraction of all queries: say ~5-10% of 10 B = ~500 M-1 B completions.  Walking that subtree =
~500 M pointer hops x ~100 ns = ~50 SECONDS, then a sort of 500 M items.  Per keystroke.   FAILS the 50 ms budget by ~1000x.
The LC trie has no way to return "just the best 10" cheaply -- it enumerates then sorts.   (The fix is precompute top-10
per prefix -- Iteration 0.)   This is the C8-relevant break: the LC structure RANKS NOTHING.

The clock — freshness vs rebuild cadence

freshness_budget = 300 s.   A monolithic trie/FST rebuilt nightly (or even hourly) means a trending event is invisible for
up to the rebuild period -- HOURS, not 5 minutes.   So freshness CANNOT come from "rebuild faster" alone (a base rebuild over
10 B queries is a multi-hour batch job; you can't run it every 5 min).   It must come from a SEPARATE fast path: a small
"recent" structure rebuilt every few minutes PLUS a live streaming counter (a Count-Min sketch over the query stream) whose
result is merged into the precomputed top-10 at query time.   100 k searches/s into a CMS is trivial (the sketch is MB-class);
shipping the resulting "trending overlay" (also MB-class) to 50 POPs every 1-2 min is ~tens of MB/min -- negligible bandwidth.
Every iteration's freshness column is checked against this 300-s clock; the trending path is Iteration 2.

S4. Naive Architecture — the simplest thing that meets the FRs

One process, one region. At startup it reads the query log and inserts every one of the 10 B unique queries into an in-RAM trie — each node a hash-map of child characters; each query's terminal node stores its lifetime search count. On GET /suggest?q=P: walk character-by-character to P's node (O(|P|)), then DFS the subtree collecting every terminal you pass (each with its count), sort the collected completions by count descending, return the top 10's text. There is no caching, no edge, no trending, no personalization — it's the LC #212 trie, lifted directly, with a count field bolted on for ranking. It satisfies every FR in the loosest sense (suggest ✓, rank by frequency ✓ — ignore recency for now, "manage corpus" = restart with a new log). The numbers tell us how far short of the NFRs it falls.

Architecture

Connections and line-style legend. Solid slate = sync hot path; dashed gray = async / offline rebuild; dashed red = hedge / retry (not used here).

  • ① User browser → suggest handler — GET /suggest?q=P per keystroke (after a client debounce); for a user far from the one region this round trip is ~80–150 ms before any compute.
  • ② suggest handler → trie (walk) — descend |P| nodes following P's characters; if the path doesn't exist, return []; else hold P's node.
  • ③ suggest handler ↔ trie (DFS + sort) — depth-first walk of P's subtree, collecting (completion_text, count) for every terminal; sort by count desc; take the first 10; return their text. For a short popular prefix this visits hundreds of millions of nodes.
  • ④ Query log → nightly rebuild — a batch job reads the log, counts queries, rebuilds the whole trie from scratch, swaps it in (with a brief stall or a double-RAM blue/green). Anything searched after the last rebuild — including every trending event — is invisible until the next one.

Why it's broken (per failing NFR)

  • Per-request latency — FAILS, and it's the headline. Two independent ways: (a) a far user's network RTT alone (~80–150 ms) exceeds the 50 ms budget — one region can't serve the planet; (b) the LC trie enumerates then sorts — a DFS over a short popular prefix's subtree visits hundreds of millions of nodes (~50 s) before it can pick the top 10. The trie has no notion of returning just the best 10 in O(|P|). This is the shape-determining failure: the only way out is to precompute the per-prefix top-10 (so a request is an O(|P|) lookup, no DFS, no sort) and serve it from near the user (so the network leg fits). Classified compute-bound on the per-request DFS — the signal is "request work scales with the data, not with the result size, and a popular key makes the work explode". (The network half is network-rtt-bound — addressed by the same "serve from near the user" move once the per-request work is small enough that the network is the budget.)
  • Index size — FAILS. ~1–3 TB doesn't fit the largest box on the menu; the only single boxes that hold it are a single failure domain and a cache-hostile pointer maze. (This is a co-symptom of the same "the LC structure is the wrong structure" problem — the FST fixes both size and request work in one move, so we treat the per-request latency as the headline and the size as its travelling companion.)
  • Trending freshness — FAILS. A nightly (or hourly) rebuild can't reflect a five-minute-old event; freshness needs a separate fast path, not a faster rebuild.
  • Availability — FAILS. One node, one region; its loss is a global outage; its restart is tens of minutes (re-read the log, rebuild the trie). 99.95% is not achievable.
  • Ranking quality — FAILS. Raw lifetime count only — no recency decay, no trending, no personalization; three of the six FRs are unmet.

Cost of naive "at NFR-target load": ~$15–30k/mo for a custom 2–4 TB-RAM box — and that box still fails latency (both the network leg and the DFS), throughput, freshness, and availability. It's not a real option; it's off the menu. The arc from here will spend — an edge fleet of ~150 nodes across 50 POPs is the dominant line item — but it spends to buy ~1000× lower per-request latency, global presence, minutes-not-hours freshness, and a degradation story. A staff designer names which way the bill goes (up — this is an edge/cache-heavy + search-index shape, cite cost-shapes#search-index + #cache-heavy) before drawing the first box.

Iteration 0 — Precompute the per-prefix top-10 offline; pack it into a minimal-acyclic FST

System after this iteration

Delta vs Naive: the per-request work flips from "DFS the whole subtree, sort everything, take 10" (O(subtree-size), seconds for a popular prefix) to "walk |P| FST transitions, dereference one pointer" (O(|P|), µs); the structure flips from a ~1–3 TB pointer maze to a ~tens-of-GB mmap-able read-only blob with shared suffixes; ranking gains recency decay. Still one region, still nightly — latency-for-far-users and freshness untouched.

Bottleneck

Naive's shape-determining failure is per-request latency on the compute path: the LC trie enumerates-then-sorts, so a popular short prefix's "find all completions" is a multi-second subtree DFS — ~1000× over the 50 ms budget — and there's no way to make it return "just the best 10" cheaply, because the structure ranks nothing. (Index size and availability also fail, but they're symptoms of "the LC structure is the wrong structure" — the same move that kills the DFS also shrinks the structure ~100×.) Classified compute-bound — the signal that picks this entry: request work that scales with the data (subtree size), not the result size, with a popular key making it explode. Per bottleneck-to-decisions#memory-capacity and the compute-bound entry: revisit the data structure (can the answer be precomputed so the request is a lookup, not a search?), the read path shape (is the structure mmap-able / cache-friendly?), and whether materialization belongs offline (yes — the top-10 per prefix is stable between rebuilds; compute it once, serve it many times). Iteration 0 takes the "precompute + change the structure" lever; the size shrink comes along for free because the FST shares suffixes and stores fixed-size outputs instead of a hash-map per node.

Alternatives considered (C4)

  • Precompute prefix→top-10 offline; pack into a hand-built minimal-acyclic FSA / FST (Lucene-FST-style: byte-labeled transitions, shared suffixes, output = a packed pointer) — chosen. The FST is the textbook answer for "map a huge set of strings to small outputs, read-only, mmap-able, cache-friendly" (cite storage-engines#fst): ~tens of GB for 10 B queries (suffix sharing is huge on natural language), an O(|P|) walk, no per-node allocation, and the whole thing is a single file you can replicate. Lucene's FST implementation is the reference; we'd use it (or a faithful port) rather than re-invent it — building a minimal acyclic FSA correctly (the Daciuk/Mihov incremental-minimization algorithm) is subtle, and Lucene's is battle-tested.
  • RocksDB with a prefix-scan iterator over a prefix → top10 KV table — rejected. RocksDB would store the precomputed lists fine, and a point-get of an exact prefix is O(log) on an LSM — but it's a general LSM (block cache, bloom filters, compaction, WAL) where we need a static read-only map: ~3–5× the on-disk footprint of an FST for the same data, no suffix sharing, and a point-get touches an SSTable block + a bloom check + possibly a few levels rather than ~30 byte-compares. For a structure that never changes between rebuilds, an LSM's mutability machinery is pure overhead. (Constraint-flip: 🔀 if the index needed point-updates between rebuilds — not our case; updates here are whole-file swaps — an LSM's incremental writes would start to win.)
  • A Redis sorted-set per prefix (ZADD prefix score completion; ZREVRANGE prefix 0 9) — rejected. Beautifully simple for a small corpus, but: 10 B queries × ~25 prefixes each ≈ hundreds of billions of sorted-set members — that's tens of TB of Redis even at a few bytes/member, and every suggest is a network round trip to the Redis cluster (~0.5 ms minimum), which is fine within a 50 ms budget but multiplies the fleet you have to run and shard. The FST is ~1000× smaller and serves from local mmap'd RAM with no network hop. Redis-per-prefix is the right answer at, say, 10 M autocomplete entries; at 10 B it's the wrong shape.
  • Keep a trie but make it succinct (a LOUDS / level-order-unary-degree-sequence bit-packed trie) — rejected as the primary, kept in mind. A succinct trie also gets you to ~tens of GB and O(|P|) lookups; the FST wins because its outputs compose along the path (you can store the top-10 pointer compactly) and Lucene's implementation is mature; a succinct trie would be a defensible alternative if we needed the trie's "all completions" capability too — but we don't, we precompute the 10.

The chosen move

Move the work that doesn't change between rebuilds — "for prefix P, which 10 completions are best?" — offline, and store the answer in a structure built for read-only lookup. The build pipeline (a transient cluster of ~20–40 m6i.4xlarge running a Spark/Beam job nightly): (1) read the query log; (2) normalize (lowercase, Unicode-NFC, trim — the heavier normalization is a probe); (3) for each prefix P of each query, accumulate (completion, score) where score = log(1 + count) × exp(-λ · age_days) (the recency decay — λ tuned so a query goes "stale" over ~weeks); keep the top-10 per P; prune prefixes whose best completion's score is below an inclusion threshold (the long tail of one-off queries gets no precomputed list — a tunable knob); (4) build a minimal acyclic FSA over the surviving prefixes, byte-labeled, suffixes shared, output at each terminal a varint offset into a packed suggestion block store appended to the same file; emit fst-{epoch}.bin to S3. The suggest server mmaps the latest file; a request walks the FST (≤ ~30 byte-transitions, ~5 ns each, all in L2/page-cache → ~150 ns), reads the ~200 B block at the output offset, resolves the 10 id refs via the id→string table (also in the file), returns the top-10. Sizing: the FST + block store + id table for the full 10 B-query corpus ≈ ~30 GB (empirically FSTs over natural-language string sets land in the "few-percent-of-raw-text" range thanks to suffix sharing; the fixed-size outputs add the block store; we plan for ~30 GB and note it could be ~20–40 GB) — fits comfortably in one r6i.4xlarge's 128 GB RAM with the page cache hot. Per-request compute ≈ ~1–5 µs (FST walk + one block read + ten id lookups). Build time: a few hours for the base rebuild over 10 B queries on ~30 nodes — well inside "nightly". Cite storage-engines#fst and access-patterns#read-heavy-uniform (the read pattern: tiny point lookups over a static structure).

What breaks (C5; cite failure-modes.html)

  • Bad build / corrupt FST. Blast radius: the suggest server mmaps a corrupt or incompatible fst-{epoch}.bin → either it refuses it (the file header carries a format version + a checksum — the server validates before swapping) and keeps serving the old epoch (graceful), or, if validation is missing, it mis-parses and returns garbage (catastrophic). Detection: the build pipeline writes a manifest with the checksum + a few canary lookups' expected results; the server verifies; a mismatch alerts. Recovery: stay on the last-good epoch; re-run the build. This is why the file format is versioned and checksummed (S2) — "validate before swap" is the contract.
  • Object store slow / unavailable at swap time. Blast radius: the server can't download the new epoch → it keeps serving the old one (suggestions get staler, but nothing errors). Detection: epoch-age metric climbs. Recovery: retry the download with backoff; if S3 is down for hours, the staleness is the degradation, not an outage — the suggest path keeps working off mmap'd RAM.
  • Serving node loss. Blast radius: total global suggest outage — it's one node. Detection: trivial (health checks). Recovery: a replacement boots, downloads ~30 GB from S3 (~30 GB / ~1 GB/s ≈ ~30 s + mmap warm), is live — but that's ~minutes of global blackout, and there's nothing to fail over to. This open wound is exactly what Iteration 1 closes (50 POPs × multiple nodes each).

Bottleneck

Iter 0's remaining failure is per-request latency on the network leg: the compute is now ~µs, so the 50 ms budget is entirely the round trip — and from one region that's ~80–150 ms for users on the far side of the planet, blowing the budget for a large fraction of traffic. Classified network-rtt-bound — the signal that picks this entry: the latency budget is dominated by speed-of-light round-trip distance, not by any compute or queue. Per bottleneck-to-decisions#network-rtt: revisit where the data lives (replicate the read path closer to the reader), the read-replica topology (how many edge locations, how do they get the data), and the cache-fill path (a near-the-user cache that's empty on a cold key is a stampede risk). The data is read-only and small — the textbook conditions for "replicate it to the edge". Alternatives considered (C4)

  • Self-run (or CDN-co-located) edge POP fleet: ~3 nodes per POP, each holding the FST mmap-resident in RAM — chosen. What breaks (C5; cite failure-modes.html)

  • Stampede on a cold prefix at a POP. Blast radius: a brand-new prefix (a breaking-news term) typed by a million users in a minute, all hitting the same POP, all missing the same not-yet-resident block → without protection, a million origin fetches. With single-flight: one origin fetch per POP; the rest await it (≤ a few ms) or get the fast no-result fallback and retry. Detection: origin-fetch rate per block; the single-flight "coalesced N requests into 1" counter. Recovery: automatic — the fill completes, the block is resident, subsequent requests are local. (When every POP holds the whole FST this is moot — the block is always resident; it bites once we range-shard or add the overlay miss path, which is why single-flight goes in now, before it's needed.)
  • Bad epoch propagated to all POPs. Blast radius: a corrupt/bad fst-{epoch}.bin reaches the origin caches and from there all 50 POPs → if validation is weak, global garbage suggestions. Mitigation: each POP independently validates (checksum + canary lookups) before swapping; a failed validation keeps the old epoch and alerts — so a bad build degrades to "everyone's on yesterday's index", not "everyone serving garbage". Recovery: pin all POPs to the last-good epoch (a manifest rollback), re-run the build. Detection: a spike in "validation failed" across POPs; canary-lookup mismatch alarms.
  • POP ↔ origin partition during a rebuild. Blast radius: a POP can't pull the new epoch → it keeps serving the previous one; suggestions there are one rebuild stale until the partition heals — degradation, not outage. Detection: per-POP epoch-age metric (and the lag between the manifest's current_epoch and the POP's served epoch). Recovery: retry with backoff; if a POP is stuck stale for > N rebuilds, alert and consider draining it to a healthy sibling.

Bottleneck

Iter 1's remaining failure is trending freshness: the FST is rebuilt nightly, so a query whose rate spiked five minutes ago is invisible until tomorrow's build — the freshness gap is hours against a 5-minute target. This isn't a capacity or latency bottleneck in the usual sense; it's a derived-data-staleness bottleneck — the served structure is a stale projection of a stream that's moving fast. Per the diagnostic taxonomy the closest entries are backlog-growth (the gap between "what's true" and "what's served" grows because the refresh job runs far slower than the change rate) and the general "stale derived store" anti-pattern; per bottleneck-to-decisions#backlog the levers are: shorten the refresh cycle (but a base rebuild over 10 B queries can't run every 5 min — physics), split the refresh into a fast lane and a slow lane (a small "recent" structure rebuilt fast + a big base rebuilt slow), and compute the fast-changing part incrementally / via a sketch instead of a full rebuild (a CMS over the stream is O(1) per search, no rebuild). Iteration 2 takes the split-into-fast-and-slow-lanes lever and the sketch lever together: a streaming Count-Min sketch (the #15 structure) feeds both a few-MB overlay and a few-GB recent FST, each rebuilt every ~1–2 min and merged with the slow base at query time.

Alternatives considered (C4)

  • A Count-Min-sketch trending processor (Kafka in front, a small Flink job — or a few stateful sketch nodes — maintaining CMS + per-prefix heaps over a sliding window) producing a few-MB overlay + a few-GB minutes-cadence "recent" FST, merged at query time — chosen. The CMS gives O(1)-per-search counting in MB-class memory with a known error bound (cite sketches#cms) — exact counting of "recent rate per query" over a high-cardinality stream would be GB-class and pointless (the rate is noisy anyway). The overlay is tiny → cheap to push to 50 POPs every minute. The recent FST handles the "brand-new query, no base entry yet" case (the overlay alone can't surface a completion that isn't in the base FST's prefix paths — the recent FST adds the new prefix paths). The "split fast/slow lane" is exactly how production query-suggest systems handle this.
  • Just rebuild the base FST more often (every ~15 min) — rejected. A full build over 10 B queries is a multi-hour batch job on ~30 nodes; running it every 15 min would mean ~12× the build cluster running ~continuously (~$15–20k/mo just for build compute) and you'd still be ~15 min stale — over the 5-minute target. Worse, you'd be re-deriving the entire stable 10 B-query distribution every 15 min just to capture the tiny moving slice. The right move is to separate the moving slice (sketch + recent FST, minutes) from the stable bulk (base FST, nightly). (Constraint-flip: 🔀 if the corpus were ~100 M queries, a full rebuild is ~minutes — then "just rebuild every 5 min" is genuinely viable and you might skip the overlay entirely; the split is forced by the 10 B-scale build time.)
  • Push trending counts into Redis (per-prefix sorted set of recent counts, decremented by a sliding-window job) and have the POP query Redis on every suggest — rejected. Adds a network hop to every suggest (the trending merge can't be a local read if the trending data lives in a regional Redis) — ~0.5–1 ms, which is survivable in the 50 ms budget but is a hard dependency on a stateful service in the hot path, and at 100 k QPS × (a few prefixes per request, since you'd query trending for every prefix of the typed string) it's a lot of Redis ops. Shipping a snapshot overlay to the POPs every minute keeps the hot path a pure local-RAM read — a minute of staleness on trending is well inside the 5-minute budget. (Redis-in-the-hot-path would win if trending needed sub-second freshness — it doesn't.)
  • A heavier heavy-hitter sketch (Space-Saving / HeavyKeeper) instead of plain Count-Min — considered, a reasonable swap. Space-Saving (cite sketches#space-saving) directly maintains the top-K heavy hitters with tighter guarantees on which items are heavy; HeavyKeeper trades a little accuracy for less memory. For "what's trending" we mostly care about the heaviest few hundred recent queries per active prefix, which Space-Saving does cleanly; we picked CMS because the #15 page already derives CMS + heap and we reuse it, but Space-Saving is a defensible substitute — the rest of the design is unchanged.

The chosen move

Split the index refresh into a slow lane (the nightly base FST — the stable 10 B-query distribution, unchanged) and a fast lane (a streaming sketch + a minutes-cadence "recent" structure — the moving slice), and merge them at query time. The fast lane: every executed search → Kafka searches → a stream processor (a small Flink job, or ~3–5 stateful sketch nodes; the choice is a footprint trade — Flink for the checkpointing/HA out of the box, the lighter design if you want fewer moving parts) that maintains over a sliding ~1-hour window a Count-Min sketch of per-query recent counts plus per-prefix heaps of the heaviest recent completions (the #15 mechanism); every ~1–2 min it (a) emits a trending overlay — for each prefix whose top recent completion exceeds a moving threshold, a small list [(completion_id, recent_rate)]; total a few MB because most prefixes aren't moving — and (b) runs a fast "recent" FST build over the last few hours of searches (a ~few-GB FST, ~1–2 min on a small cluster — this catches brand-new queries that have no base-FST entry yet). Both ship to the origin caches and are pulled by the 50 POPs every ~1–2 min. At query time the POP, for suggest(P): gets the base top-10 from the base FST; gets trending candidates from overlay[P] and recent_FST(P); merges into one candidate set; re-scores each as final = base_score(c) + recency_boost(recent_rate(c)) where recency_boost is calibrated so (i) a candidate spiking and with real history clearly outranks a stale one, (ii) a brand-new spiking term with ~zero base score still cracks the top-10 if its rate is high enough, (iii) an old query with no recent activity is unaffected; takes the top-10; sets the trending flag on any whose place is recency-driven. Freshness re-derivation: a spike becomes visible to the sketch within ~seconds (it's a streaming count); the overlay/recent-FST is rebuilt every ~1–2 min; the POP pulls it within ~1–2 min more; the query-time merge is instantaneous → worst-case spike-to-suggestion ≈ ~2–4 min < the 5-minute budget (with margin for a slow POP pull). Bandwidth: the overlay is a few MB × 50 POPs × (every ~1–2 min) ≈ ~hundreds of MB/min — noise; the recent FST is a few GB × 50 × (every ~1–2 min) ≈ ~tens of GB/min aggregate cross-region — still small (~hundreds of MB/s). Compute: ~one extra FST walk + a ~10-element merge per request ≈ ~1–5 µs — the latency budget is untouched. Cite sketches#cms, #15, and access-patterns#bursty (the trending query stream is bursty by nature).

What breaks (C5; cite failure-modes.html)

  • Trending pipeline stalls / Kafka backs up. Blast radius: the overlay/recent-FST stop refreshing → trending goes stale (a spike that happened during the stall isn't surfaced) but the base FST still serves — suggestions degrade to "frequency × recency only", which is a fine fallback. Detection: overlay-epoch-age metric at the POPs; Kafka consumer lag on the trending job. Recovery: the job catches up (the CMS is a sliding window — it self-heals as fresh data flows); if it's wedged, restart it (Flink checkpoints, or the sketch nodes rebuild their windows from Kafka). The contract: stale trending is silently dropped from the merge once its epoch-age exceeds a threshold — better to show no "trending" flag than a 30-minute-old "trend".
  • Poison query in the stream. Blast radius: a malformed or adversarial entry (a query string crafted to blow up the normalizer, or a spam campaign artificially inflating a query's recent count to game trending). For the malformed case: the stream job skips un-parseable entries and counts them. For the spam case: this is where the blocklist (Iter 3 / probes) and rate-aware filtering matter — a query whose recent rate jumps implausibly from a narrow set of sources gets dampened; and the blocklist hard-suppresses anything harmful regardless of how trending it looks. Detection: spike-source-concentration metrics; the blocklist hit rate. Recovery: blocklist push (≤ ~30 s to all POPs — faster than a recent rebuild, by design).
  • Bad recent-FST or overlay propagated. Blast radius: a corrupt recent-{epoch}.bin or trending-{epoch}.bin reaches the POPs → if validated (same checksum + canary-lookup contract as the base FST), it's rejected and the POP keeps the previous one (trending one cycle stale — degradation); if not, garbage in the merge. Mitigation: the validate-before-swap contract applies to all three artifact families. Detection: per-artifact validation-fail counters. Recovery: the next ~1–2 min cycle produces a good one; the bad epoch is skipped.

Bottleneck

Iter 2's remaining FR gap is personalization — suggestions are global, identical for every user of a (prefix, lang); and the carried-forward concern is the hot prefix — a single very-popular prefix concentrates a large slice of the 100 k QPS, and once the FST is range-sharded (when the corpus grows) or under a 3× burst, whichever node owns that range is a hot key. The hot-prefix half is classified hot-key — the signal: one key's traffic is a meaningful fraction of total and falls on one shard. Per bottleneck-to-decisions#hot-key the levers are: replicate the hot key more widely (since it's read-only, replication is free — pin it in every node), don't shard it onto a single node, and cache it harder (lock it RAM-resident, never evict). The personalization half isn't a "bottleneck" in the diagnostic sense — it's an unmet FR — but it has a sharp shape constraint: you cannot precompute per-(prefix, user) — 10 B prefixes × hundreds of millions of users is absurd — so personalization must be a re-rank of a global candidate set with a small per-user feature blob, applied late (at the edge, after the global merge), and it must degrade to the global answer when the blob is missing (so it's never a hot-path dependency). Iteration 3 takes the hot-key "replicate-it-everywhere" lever and the personalization "late re-rank, fail-open" lever together (they're independent fixes for two carried-forward issues, packaged in one iteration because the bottleneck budget is Naive + 4 iters — this is the last one).

Alternatives considered (C4)

  • Personalization as a late, edge-side re-rank over a small per-user feature blob, with a global-top-10 fallback — chosen. The candidate set comes from the (already-computed) global merge expanded to top-20; the re-rank is a handful of arithmetic ops per candidate (does this completion match one of my recent queries? is it trending in my locale? — small boosts) — ~µs at the POP, no network hop (the blob is cached at the POP after one fetch). Critically it's fail-open: no blob → return the global top-10; this keeps the suggest path's availability decoupled from the per-user KV's. This is the only shape that scales — per-(prefix, user) precomputation is a non-starter, and a per-request call to a personalization model service would add a network hop and a hard dependency to a 50 ms-budget hot path.
  • A per-request call to an ML ranking service that scores the candidates with a learned model — rejected for the hot path, plausible offline. A learned ranker (gradient-boosted trees, a small NN) over richer features would rank better — but it's a network hop + a model-serving dependency in the critical path of a 100 k QPS, 50 ms-budget read; and the marginal quality over "boost my recent queries + my locale's trends" is modest for autocomplete (the prefix already pins the candidates hard). Where the learned ranker does belong: offline, computing the base FST's score (a learned blend of frequency, recency, CTR-on-suggestion, etc.) and the recency_boost function — that's a build-time computation, not a hot-path call. (Constraint-flip: 🔀 if autocomplete quality were a top business metric and the budget were 100 ms, a co-located edge model-serving sidecar doing a ~1–5 ms inference on the top-20 becomes worth it — that's the #49 real-time-ranker shape applied here.)
  • For the per-user feature KV: a regional DynamoDB table (or Cassandra) keyed by user_id — chosen; vs. Redis — rejected. The blob is read once per session and cached at the POP, so the KV sees ~session-start rate, not suggest rate — a few k QPS per region, point-gets by key, ~1–4 KB values, written async from the search-history service. DynamoDB on-demand (or self-run Cassandra) fits: durable, single-key point-get latency ~single-digit ms (and we're off the hot path anyway), no fleet to babysit per region for Dynamo. Redis would be faster per-get but it's a cache, not a store — we'd need a backing store behind it anyway, and the blob doesn't need sub-ms reads (it's a session-start fetch). At the per-session read rate, Dynamo's per-request cost is fine; Redis's per-region cluster ops aren't worth it. (If session-start rate were huge — tens of k/s/region — a Redis read-through in front of Dynamo would start to pay.)
  • Hot-prefix mitigation: pin the hot-prefix ranges RAM-resident in every POP node + don't range-shard those ranges — chosen; vs. a separate "hot tier" of dedicated nodes — rejected. Since the FST is read-only and small, the cheapest hot-key defense is "every node has the hot key, locked in RAM" — no special tier, no routing complexity, the busiest prefix is always a local read on whatever node a request lands. A dedicated hot tier (a few extra nodes per POP that only serve the top prefixes) would work but adds routing logic and a tier to operate, for no gain over "just pin it everywhere" when "everywhere" is already only ~3 nodes/POP. We'd consider a hot tier only if the FST grew so large that range-sharding became unavoidable and the hot ranges were too big to also full-replicate — not the case at ~30 GB.

The chosen move

Personalization — a late, edge-side re-rank, fail-open. On session start (or lazily on the first suggest), the POP fetches the user's feature blob from the regional user-feature KV (DynamoDB/Cassandra keyed by user_id; ~1–4 KB: the last ~50 normalized queries, UI lang, coarse region, a few interest tags; written async by the search-history service) and caches it in an in-RAM LRU (TTL ~minutes–hours). For each suggest(P): the FST + trending merge produces the top-20 global candidates (a slightly larger set than 10, so the re-rank has room to promote a personal favourite without dropping a strong global one); then a lightweight per-user re-rank: personal_score(c) = global_score(c) + α·[c matches one of my recent queries] + β·[c is trending in my region/lang] with small α,β (calibrated so a personal match can lift a candidate a few positions but a clearly-dominant global suggestion still leads); take the top-10; set the personalized flag on any whose place is personal-driven; return. If the blob is missing or stale (new user, anonymous session, KV slow/down) → skip the re-rank, return the global top-10 — personalization never blocks or errors the suggest path. Compute: ~one KV cache lookup (RAM) + ~20 candidates × a few ops ≈ ~1–5 µs more — budget untouched. Hot-prefix mitigation. The build/trending tier emits, each cycle, a hot-prefix list (the top ~few-hundred prefixes by QPS — single letters, common bigrams, the current mega-trends; computed from the same recent-rate-per-prefix data the trending sketch already has). Every POP node keeps those prefixes' FST sub-paths + suggestion blocks locked RAM-resident (never evicted), and the FST byte-ranges covering them are full-replicated to every node in the POP rather than range-sharded — so the busiest prefix on the planet is always a local RAM read on whichever of the POP's ~3 nodes a request hits, never a routed call to one "owner" node. (When the FST fits one node entirely — the current ~30 GB case — this is already true for free; the pin list makes it stay true if/when the corpus grows past one-node and range-sharding kicks in.) Cite bottlenecks#hot-key + bottleneck-to-decisions#hot-key (replicate the read-only hot key everywhere), and sharding-strategies#range (the FST is range-sharded by leading-byte when it must be sharded at all; the hot ranges are exempted into full replication — a hybrid, cite #hybrid).

What breaks (C5; cite failure-modes.html)

  • Per-user feature KV slow / down. Blast radius: blob cache misses can't be filled → those users get the global top-10 (no personalization) — a degradation, not an error or a slowdown (on a miss the POP serves global immediately and warms the blob async, so the user's request isn't delayed). Detection: blob-cache hit rate drops; KV error/latency metrics. Recovery: the KV recovers; cache hit rate climbs back; in the meantime suggest works fine globally. This is the whole point of "fail-open": the per-user store is the least-critical dependency, so its failure has the smallest blast radius.
  • Hot-prefix burst beyond pinned capacity. Blast radius: a mega-event makes a prefix so hot that even with it pinned RAM-resident on every node, a single POP's ~3 nodes are CPU-saturated by that one prefix's QPS. Mitigation: (a) the ~3× burst headroom in the fleet sizing absorbs the normal case; (b) for an extreme case, the CDN's 2-second response cache (Cache-Control: max-age=2 on the suggest response, keyed on (prefix, lang) — trending/personalized variants bypass it) collapses repeated identical requests at the CDN layer before they reach the serving nodes; (c) autoscale adds nodes to the hot POP (cold-start ~minutes to download the FST — see the cold-POP-start probe). Detection: per-POP, per-prefix QPS; CPU saturation alarms. Recovery: the 2-s cache + autoscale; in the worst case, briefly serving a slightly-stale (2-s-cached) top-10 for the mega-prefix is an acceptable degradation.
  • Stale / corrupt per-user blob. Blast radius: a user's blob is from a corrupted write or is months old → the re-rank boosts queries they no longer care about — mildly annoying, not harmful (the global suggestions are still in the candidate set; the personal boosts only re-order). Mitigation: the blob carries a write timestamp; the POP ignores boosts from a blob older than ~N days (treats it as missing); the search-history service's writes are idempotent and versioned. Detection: blob-age distribution. Recovery: the next session-start fetch (or the history service's next write) refreshes it.

🔀 Constraint flip — if personalization had to be guaranteed (every user sees a meaningfully personal list)

If "personalize lightly" became "personalize hard — the list must reflect this specific user's intent", the fail-open re-rank isn't enough: you'd need richer per-user features (not just recent queries — clicked results, dwell time, cross-device profile), a learned ranker (not a couple of boosts), and you couldn't fall back to the global list silently (a guarantee means you have to have the data). That pushes the per-user KV into the critical path (or forces a much fatter per-session blob fetched eagerly), adds a model-serving dependency, and likely blows past the 50 ms budget — so you'd renegotiate the budget to ~100 ms and accept a co-located edge inference sidecar (the #49 real-time-ranker shape). "Personalize lightly + fail open" is the assumption that lets personalization be a free ~µs re-rank with zero added critical-path dependency; "guarantee it" turns it into the dominant complexity of the system.

Probe iterations (the interviewer pulls the thread — beyond the 5-Core budget; each closes a refinement, not a failing NFR)

Each in a <details> block; not rendered inline by default. These are real concerns a production query-suggest system handles — they just don't re-architect the design, they bolt onto it.

Probe A — Typo-tolerant prefix matching (the user types "resturant")

Two complementary moves. (1) Levenshtein-automaton ∩ FST. Build a small automaton accepting all strings within edit-distance 1 (or 2) of the typed prefix; intersect it with the base FST (the FST is a DFA — intersection is a standard product construction, and because most edits prune fast it visits few states); the surviving terminals are the fuzzy-prefix matches, which then go through the normal merge + re-rank. This is exactly how Lucene's fuzzy queries work. Cost: ~10–100× the exact-walk's state visits for an edit-distance-1 automaton over a ~10–15-char prefix — still ~µs–tens-of-µs, fine in the budget; we'd only run the fuzzy intersection when the exact walk returns < N results (a typed prefix that has plenty of exact completions doesn't need fuzziness). (2) A small "common misspellings" map baked into the build: from query-log analysis, the offline pipeline learns frequent (misspelled → corrected) pairs and adds the misspelled forms as alias prefixes pointing at the corrected query's suggestions — covers the high-frequency typos with zero query-time cost. Use (2) for the common cases, (1) as the general fallback.

Probe B — Query normalization (case, accents, CJK segmentation, RTL, Unicode)

All normalization happens offline in the build pipeline and is mirrored identically at the POP on the incoming prefix — the FST is keyed on normalized bytes, so the runtime must normalize the user's prefix the same way before walking. The pipeline: Unicode NFC; lowercase (locale-aware — Turkish ı/I, German ß); strip or fold accents per language policy (French users want "café" and "cafe" to match; some languages don't fold); collapse whitespace; for CJK (Chinese, Japanese), where there are no word boundaries, segment into tokens and key the FST on the segmented form (and handle the case where the user has typed a partial trailing token — you walk the FST allowing the last segment to be a prefix); for RTL (Arabic, Hebrew), normalize bidi control characters and key on logical order. The FST is also sharded by language (Probe C) so each language's normalization rules apply to its own corpus. The hard part isn't the FST — it's getting the build-time and runtime normalizers to be byte-for-byte identical (a mismatch silently breaks matching for that locale); ship them as one shared library, version it, and include locale-specific canary lookups in the build manifest.

Probe C — Per-language / per-region corpora (the FST is sharded by language)

One global FST is wrong: a French prefix shouldn't surface English completions (mostly), normalization rules differ, and a 10 B-query global corpus is mostly noise to any single user. So the build emits one FST per language (or per language × coarse-region for big markets): fst-{lang}-{epoch}.bin; a POP holds the FSTs for the languages its region serves (a European POP: ~10–20 language FSTs; a single-language market's POP: 1–2); the suggest request carries lang (from the browser's Accept-Language / the user's blob) and walks that language's FST. This makes each FST smaller (the per-language corpora sum to ~the global one but each is a fraction), makes per-POP RAM scale with the languages it serves (not the global total), and localizes a rebuild (a bad French build doesn't touch English). The trending overlay and recent FST are similarly per-language. A user typing in a language their POP doesn't hold (a tourist) falls back to the nearest POP that does, or to a degraded English/Latin-script result — rare, acceptable.

Probe D — Abuse / safety filtering (don't autocomplete to slurs, PII, illegal content)

A blocklist applied after the FST walk + merge + re-rank, just before returning: (1) a set of blocked prefixes — typing one of them returns [] (an empty list, not a 404; some prefixes must yield no suggestions at all — e.g. a prefix that only completes to harmful queries); (2) a set of blocked completions — any suggestion matching one is dropped from the list (and if dropping it leaves fewer than N, the next-best candidates fill in — that's why the FST returns top-20 before filtering); (3) pattern filters — regexes/automata for PII shapes (looks-like-an-SSN, looks-like-a-credit-card), profanity classes, and known-illegal-content terms. The blocklist is small, replicated whole to every POP, and shipped out-of-band of the FST on its own fast channel (≤ ~30 s to all POPs — when you're suppressing something harmful you can't wait for the next recent-FST cycle, let alone the nightly base build). It's also applied upstream: the trending sketch ignores blocked queries (so a spam campaign can't make a slur "trend" and the build pipeline never precomputes it into the FST in the first place — defense in depth, but the runtime blocklist is the backstop because the FST is built before the latest blocklist edits). Auditing: every blocklist edit is logged (who, when, why) — this is a content-policy surface.

Probe E — Cold start of a new POP (it has to load a ~30 GB FST)

When a new POP comes online (or all its nodes are replaced after a rack failure), each node must download the base FST (~30 GB) + the recent FST (~few GB) + the overlay (MB) from the nearest origin cache and mmap them before it can serve — ~30 GB / ~1 GB/s ≈ ~30 s for the transfer, plus warming the page cache (a few more seconds of first-touch faults, or pre-fault the hot pages eagerly) — call it ~1–2 min to fully warm. During that window the POP returns 503 for suggest, and anycast/GeoDNS health-checks route its traffic to the next-nearest POP (which absorbs it — the ~3× burst headroom in every POP's sizing is partly for exactly this: a sibling POP going dark). To make a planned add faster, you can pre-stage the FST onto the new POP's nodes before flipping them into rotation (download in the background, then health-check green). The cold-start cost is bounded by the FST size — another reason the Iter-0 shrink (1–3 TB → ~30 GB) matters: a terabyte-scale structure would be ~15–30 min to cold-load, a much worse failure-recovery story. Cite bottlenecks#cold-start + failure-modes#black-hole-node.

Degradation contract

When a POP is unavailable → anycast/GeoDNS routes its users to the next-nearest healthy POP (slightly higher RTT for that slice; the burst headroom absorbs the load); a cold-starting POP returns 503 and stays out of rotation until it's warm. When an origin cache is unavailable → its POPs pull from S3 directly (slower per-pull, irrelevant on the hot path). When S3 / the build pipeline is unavailable → every POP keeps serving the last good base FST + recent FST + overlay it has mmap'd — suggestions get staler (trends fade, the recent FST ages) but nothing errors; the index just stops updating until S3 recovers. When the trending pipeline stalls → the overlay/recent FST go stale; once their epoch-age exceeds a threshold the POP drops them from the merge — suggestions degrade to "frequency × recency only", no trending flag (better than a 30-minute-old "trend"). When the per-user feature KV is slow/down → the per-user re-rank is skipped — users get the global top-10 (fail-open; their request isn't delayed — on a blob cache miss the POP serves global immediately and warms async). When a prefix is below the inclusion threshold or on the blocklist → the suggest API returns an empty list (200 {suggestions:[]}), not a 404. The suggest path never errors on "no good suggestions", never exceeds the 50 ms budget by blocking on a non-critical dependency (per-user KV, trending pipeline, S3), and never serves a known-harmful completion (the blocklist is the backstop, applied last, shipped fastest). Cite failure-modes#slow-dependency + #stampede + #bad-deploy.

Operations (C6)

  • Metrics (SLIs → NFRs). suggest_latency_p50/p99 end-to-end and split by POP, by region, by "did it RTT-or-compute dominate" (→ the headline 50 ms SLO — alert at p99 > 45 ms); per-POP, per-prefix QPS + CPU saturation (→ throughput, hot-prefix); fst_walk_p99_us + the merge/re-rank time (→ the per-request work budget — should be ~µs; a spike means a corrupt FST or a fuzzy-walk runaway); index_epoch_age (base / recent / overlay / blocklist) per POP (→ freshness — alert if recent/overlay age > ~5 min, or base age > ~36 h, or any POP > N epochs behind the manifest); trending_spike_to_visible (a synthetic canary query injected into the stream, measured to appearance at a sample of POPs → the 5-min budget — this is the real test); sf_coalesced/s + origin-fetch rate (→ cache-fill health / a viral cold prefix); per_user_blob_hit_rate + KV latency/errors (→ personalization health — a drop is a degradation, not a page); validation_fail count per artifact (→ bad-build detection); blocklist hit rate + edit-to-propagation latency (→ the ~30-s policy SLO); 503 rate per POP (→ cold-start / capacity).
  • Rollout (binary / FST-format / ranking changes). Canary one POP (or a few nodes within several POPs) on the new binary serving real traffic, with its suggest stream diffed against a control POP for the same prefixes — chase any divergence (a ranking change should diverge; a binary change shouldn't); ramp POP-by-POP. An FST-format change ships a build that writes both the old and new format for a transition, and POPs are upgraded to read the new before the old is dropped. A ranking-formula change (new score function / new recency_boost) is gated behind an offline eval (NDCG against a held-out click set) and a small online A/B (the per-user re-rank already gives a place to feature-flag it). Rollback trigger: a p99-latency regression, a spike in 503s or validation-fails, or an NDCG drop in the A/B.
  • Migration (current trie → this design). Phase 1: stand up the offline build pipeline, produce the first FST, deploy a single FST-serving node behind the existing trie (shadow — mirror a copy of suggest traffic, diff the top-10s, chase divergences for a week; mostly ranking gaps you decide are improvements). Phase 2: bring up the 50-POP fleet pulling that FST; route a small % of real traffic to it via the CDN, watch latency + the diff; ramp to 100%. Phase 3: turn on the trending tier (shadow the merge first — compute the merged result, diff against base-only, verify the recency_boost calibration); ramp. Phase 4: turn on per-user re-rank for a % of sessions (it's already fail-open); ramp. Keep the old trie warm as an instant fallback through Phase 2, then decommission. Rollback at any phase: pin the CDN routing back to the old trie.
  • Headroom. +~3× burst headroom per POP (a mega-event concentrates traffic and a sibling POP can go dark — both at once is plausible); +~50% on the origin caches and the trending tier over steady state; the build cluster is transient (sized for "finish the nightly base build in a few hours"). Reason: viral events are the normal failure mode here, not the exotic one; a celebrity-news term can 5× a POP's QPS on a single prefix in minutes, and the design has to absorb that with the 2-s response cache + the pinned hot-prefix shard + spare node capacity, not fall over.

Real-system comparison

The closest production cousin is Google's query autocomplete ("Suggest"): it serves as-you-type completions ranked by a blend of historical query frequency, freshness/recency, and personalization (your prior searches, location), with trending-event responsiveness measured in minutes — the exact FR set this page derives; Google has published on the suggest infrastructure and on the FST-style data structures underneath search (and Lucene's FST — a minimal acyclic FSA mapping terms to outputs, mmap-able — is the open-source reference for the structure; Elasticsearch's "completion suggester" is literally an FST per shard). Algolia and other search-as-a-service vendors document FST-backed prefix lookup with query-time ranking on top. LinkedIn has written on typo-tolerant prefix search (the Levenshtein-automaton ∩ FST trick in Probe A). Where this page diverges from what those teams describe: (1) the explicit edge POP tier with the FST RAM-resident — many real autocomplete deployments serve from a handful of big regions and accept higher RTT for far users, because their corpus is small enough not to force the Iter-0 shrink, or because they don't own a 50-POP CDN footprint to piggyback on; the 50-POP design is the right one given "50 ms including the network" + "we already run 50 POPs" + "the FST is only ~30 GB". (2) The fail-open per-user re-rank at the edge rather than a per-request call to a personalization model service — a real team optimizing autocomplete quality hard might run an edge inference sidecar (the #49 shape); we kept it a ~µs boost because the prefix already pins the candidates and the budget is tight. (3) The CMS-overlay-snapshot-to-the-edge trending path — some teams keep trending in an in-path low-latency store (the rejected Redis option) for sub-second freshness; we took the snapshot approach because the 5-minute budget allows it and it keeps the hot path a pure local read. If a real team running this told me they put trending in-path and ran 5 regions not 50, I'd ask: is your corpus small enough to skip the FST shrink, and is your latency budget server-side-only? — because those are the two constraints that flip those decisions. Sources: see cribs/_sources.md for this scenario (Google Suggest, Lucene/Elasticsearch FST docs, Algolia query-processing, LinkedIn typo-tolerance) and cribs/45.md for the diff notes.

Anticipated pushbacks

Stress tests against the derivation above — depth rehearsal, not interview rehearsal. If a question can't be answered in 3–6 sentences from the text above, the page is missing a probe.

What if suggest QPS were 10× tomorrow — 1 M QPS? (scale flip)

The FST is unchanged (it's sized by the corpus, not the QPS — still ~30 GB, still fits a node). The edge fleet grows: 1 M QPS / 50 POPs ≈ ~20 k QPS/POP average, ~60 k peak — one r6i.4xlarge at ~µs walks can still do ~200–500 k QPS, so it's still ~1 node for the QPS, but you'd run ~5–6 nodes/POP for N+1 + burst at that level → ~250–300 nodes (~$200k/mo). The new pressure is on the per-user feature KV (session-start rate scales with QPS → ~100 k reads/s global → the Dynamo read bill jumps ~10× to ~$65k/mo, at which point a Redis read-through in front of Dynamo per region starts to pay) and on the 2-second CDN response cache doing more of the work (a bigger fraction of 1 M QPS is repeated identical prefixes). The bottleneck shape doesn't change — it's still "RAM in 50 regions" + "the per-user store"; you'd add nodes and put a cache in front of the KV. The ~3× headroom buys roughly the time to provision the bigger fleet before it's load-bearing.

Why not Elasticsearch's completion suggester per region instead of a hand-rolled FST fleet? (alt-engine)

Elasticsearch's completion suggester is an FST per shard — conceptually exactly this — and if autocomplete were one feature of a search product you were already running ES for, you'd absolutely use it. As a standalone 50-region autocomplete service it's heavyweight: you pay for the full ES cluster machinery (indexing path, JVM, shard allocation/rebalancing, cluster coordination, the master nodes) in each of 50 regions, all to serve a structure that never changes between rebuilds; a purpose-built "mmap a read-only FST file, walk it, serve it" fleet has a fraction of the operational surface and cost. ES also doesn't naturally give you the trending-overlay merge or the fail-open per-user re-rank at query time — you'd be writing those as ES plugins/scripts anyway. Cite storage-engines#fst (the FST is the right engine; the question is how much extra machinery you wrap it in) and #inverted-index (ES's home turf — full-text search, which this isn't). The constraint that disqualifies ES-per-region: "standalone autocomplete, 50 regions, read-only index" — flip it to "autocomplete inside an existing ES search product" and ES wins.

What changes if this has to honor GDPR / strict regional data residency? (regulatory)

Most of it doesn't change — the FST is built from query counts (aggregate, not personal), the searches stream carries normalized text + coarse region + lang only (no IP, no user id, no raw text), and the suggest API is request/response with a 2-s private cache. What gains: (1) the per-user feature blob — the only PII-bearing store — becomes strictly regional (an EU user's blob lives only in EU; the EU POPs read only the EU KV; no global table that replicates it out), TTL'd, and wired into the search-history product's erasure cascade (a deletion there must remove the blob); (2) the per-language/per-region corpus split (Probe C) becomes a requirement — an EU-built FST from EU query logs, served from EU POPs — not just an optimization; (3) audit logging of blocklist edits (already there) and of any access to the per-user KV; (4) the "coarse region" in the stream gets coarsened further or dropped in jurisdictions that consider it identifying. Naming what stays the same: the FST structure, the edge tier, the trending sketch, the latency math — residency adds governance and a regional partitioning of the one PII store, not new bottlenecks. Cite cross-cutting#privacy.

A whole continent's worth of POPs (and its origin cache) goes dark right now — blast radius and recovery? (failure)

Anycast/GeoDNS immediately re-routes that continent's suggest traffic to the next-nearest healthy POPs (on adjacent continents) — users there see higher RTT (maybe ~60–100 ms instead of ~10–40, so some p99s now exceed 50 ms for that slice — a degradation, not an outage), and the absorbing POPs lean on their burst headroom. The dead continent's origin cache being gone is irrelevant to serving (POPs already have the FST mmap'd) and only matters for the next rebuild — new POPs in that continent (once it's back) pull from S3 directly or from another continent's origin cache until the local one is rebuilt. Recovery sequence: continent comes back → POPs there cold-load the latest FST (~1–2 min/node) + recent + overlay → health-check green → anycast re-attracts the local traffic. No data is lost (the index is derived); the only user-visible effect was a window of higher latency for one region. Cite failure-modes#network-partition + #black-hole-node.

Where does the ~$130k/mo bill come from, and which knob moves it most? (cost)

~84% is the edge serving fleet — ~150 r6i.4xlarge across 50 POPs — and that's almost entirely "RAM in 50 regions to keep the ~30 GB FST (+ recent + overlay + per-user-blob LRU) hot for a 50-ms-including-the-network budget"; everything else (origin caches ~3%, the per-user KV ~5%, the trending tier ~5%, the build/egress/S3 the rest) is rounding error. So the single biggest knob is how much RAM you commit to the edge: (a) the FST pruning threshold — only precompute prefixes with ≥ K historical searches; a higher K shrinks the FST (~30 GB → ~few GB at a high K), so each POP node can be a smaller/cheaper instance and a new POP cold-starts faster — at the cost of rare prefixes getting no precomputed list (they fall back to the recent FST, a fuzzy walk, or an empty result); (b) nodes per POP — 2 instead of 3 saves ~$36k/mo but loses the N+1 (a node loss during a burst can saturate the survivor). We keep K low (completeness on rare prefixes is a product value) and 3 nodes (viral bursts + a node loss is normal), but those are the two dials, and they're the same dial: edge RAM = the bill. Cite cost-shapes#cache-heavy + #search-index.

How do you migrate from the current production trie to this without breaking suggestions? (migration)

Phase 1 (build + shadow): stand up the offline pipeline, produce the first FST, run one FST-serving node behind the existing system mirroring a sample of suggest traffic — diff the top-10s for a week, triage divergences (most are ranking improvements from the recency decay; some are bugs in the build/normalizer — fix them; some are the pruning threshold dropping a rare prefix — decide the K). Phase 2 (edge ramp): bring up the 50-POP fleet pulling that FST; route a small % of real traffic via the CDN, watch latency + the diff; ramp to 100%, keeping the old trie warm as an instant fallback (CDN routing pinned back). Phase 3 (trending): turn on the Kafka stream + CMS processor; shadow the merge (compute base+trending, diff against base-only, verify the recency_boost calibration on real spikes) before it affects responses; ramp. Phase 4 (personalization): enable the per-user re-rank for a % of sessions — it's already fail-open, so the risk is "a bad boost re-orders something" not "an outage"; ramp. Rollback at any phase: pin CDN routing to the old trie (Phases 1–2) or feature-flag off the trending/personalization (Phases 3–4). The data-migration story is trivial — the index is derived; there's no state to migrate, only a build pipeline to stand up and a serving tier to ramp.

"Autocomplete is slow / showing weird suggestions right now" — where do you look first? (monitoring/debug)

For slow: (1) suggest_latency_p99 split by POP — if it's one POP, it's local (CPU saturation from a viral prefix → check per-prefix QPS + the sf_coalesced rate; or the POP is mid-cold-start → 503 rate; or a node died → capacity); if it's global, look at fst_walk_p99_us (should be ~µs — a spike means a corrupt/bad FST got swapped in, or a fuzzy-walk is running away) and the merge/re-rank time (a slow per-user KV with a low cache hit rate dragging the path — though fail-open should have caught that, so check that the fail-open timeout is actually firing). For weird suggestions: (1) index_epoch_age per POP — a POP serving a stale base/recent epoch shows yesterday's-but-not-today's; (2) the validation_fail counter — if a bad build was rejected, POPs are on the last-good epoch (expected); if validation is somehow being skipped, garbage; (3) the blocklist epoch/version per POP — if a harmful completion is showing, is the blocklist current there? (it should propagate in ~30 s); (4) the trending_spike_to_visible canary — if it's lagging, the trending pipeline is stalled (Kafka lag, Flink wedged) → trending is stale or being dropped from the merge. Each signal maps to a specific failure mode from the What-Breaks blocks and a runbook entry; the validate-before-swap + epoch-age + canary discipline exists precisely so this triage is a dashboard read.

What stops a spam campaign from making a slur or a scam term "trend"? (adversary)

Defense in depth, with the runtime blocklist as the non-negotiable backstop. (1) The trending sketch ignores blocked queries — a term on the blocklist (slurs, known scams, PII patterns) never accumulates a recent-rate count, so it can't surface no matter how many bots search it. (2) Spike-source heuristics — a query whose recent rate jumps from a narrow, suspicious set of sources (few IPs/ASNs, no organic spread, bot-like timing) gets dampened or held out of the overlay; this is the "is this trend real?" filter. (3) The offline base build also applies the blocklist, so a harmful query that did accumulate history (before it was blocked) gets pruned out of the FST on the next rebuild — but because the FST is built before the latest blocklist edits, the runtime blocklist (applied last, on every response, shipped to all POPs in ~30 s) is the authoritative one. (4) Some prefixes are configured to return nothing (an empty list) regardless — for prefixes whose only completions are harmful. (5) Every blocklist edit is audited (who, when, why, ticket ref) — this is a content-policy surface, not just an engineering one. The residual risk is "a harmful term that's new, not yet blocked, and genuinely viral" — the spike-source heuristics catch most; the rest is a fast-response operational loop (detect → blocklist push in ~30 s). Cite failure-modes#poison-message.

S7. Key Takeaways

  • The LC trie ranks nothing — that's the property that breaks. Word Search II's trie enumerates: "find all completions of P" DFS-es the subtree, then you sort. At 10 B queries a popular short prefix's subtree is hundreds of millions of nodes — ~50 s per keystroke against a 50 ms budget. The fix isn't "a faster trie"; it's precompute the per-prefix top-10 offline so a request is an O(|P|) lookup, not a search — and once you precompute, the structure can be a minimal-acyclic FST (suffix-shared, fixed outputs, mmap-able): ~30 GB instead of ~1–3 TB. One move kills the DFS and the size.
  • "≤ 50 ms including the network" is what forces the edge. Once the per-request work is ~µs (the FST), the whole budget is the round trip — and one region is ~150 ms from half the planet. Because the FST is small and read-only, "replicate it to all 50 POPs we already run" is the cheap answer; with a terabyte-scale structure you'd be stuck with a few big regions and a blown budget for far users. Read the latency NFR for where the budget goes: if it's distance, you replicate the read path; if it's compute, you precompute.
  • Freshness needs a fast lane, not a faster rebuild. A base rebuild over 10 B queries is a multi-hour batch job — you can't run it every 5 min, and even hourly you'd be hour-stale. So split the moving slice from the stable bulk: a streaming Count-Min sketch over the query stream + a minutes-cadence "recent" FST (the #15 top-K-over-a-stream machinery) feed a few-MB overlay, merged with the slow base at query time → ~2–4 min spike-to-suggestion. A 30-s target would instead force a real-time push to the edge; 5 min lets trending be a cheap snapshot-to-the-edge job.
  • Personalization that scales is a late, fail-open re-rank — never a precompute, never a hot-path dependency. You cannot precompute per-(prefix, user) — 10 B prefixes × hundreds of millions of users is absurd. So the FST gives a global top-20; a small per-user feature blob (cached at the POP after one fetch) re-ranks to top-10 with a couple of boosts; if the blob is missing, you serve the global top-10 unchanged. The per-user store is the least-critical dependency, so its failure has the smallest blast radius — that's the design goal, not an accident.
  • A read-only hot key is the easiest hot key — replicate it everywhere. The single most-typed prefix ("a", "the") concentrates a big slice of the QPS; because the FST is read-only, the fix is "every POP node holds that prefix's subtree pinned RAM-resident, never evicted, and its byte-range is full-replicated rather than range-sharded" — the busiest prefix on the planet is always a local read on whatever node a request lands. No hot tier, no routing tricks. Cite bottlenecks#hot-key.
  • Not every NALSD arc cuts the bill — this one falls then rises. Iter 0 cut cost ~10× (the naive pointer trie was monstrously wasteful); Iters 1–3 then spent ~60× from there to ~$130k/mo — ~84% of it the ~150-node edge fleet — to buy global presence + minutes-fresh trending + personalization. The dominant cost is "RAM in 50 regions to keep the index hot"; the one knob that moves it is the FST pruning threshold (prune harder → smaller FST → cheaper edge → but rare prefixes lose suggestions). A staff designer names which way the bill goes, what dominates it, and which knob moves the dominant line.
  • Validate before you swap; ship the blocklist fastest. The index is a blob replicated to ~150 nodes — a bad build that propagates is a global-garbage event unless every node independently validates (checksum + canary lookups) and keeps the last-good epoch on failure: a bad build degrades to "everyone's on yesterday's index", not "everyone serving garbage". And the content-policy blocklist ships on its own channel (~30 s to all POPs, faster than any rebuild) because when you're suppressing something harmful you can't wait for the next cycle — it's applied last, on every response, and it's the backstop the build-time filter doesn't fully cover.
