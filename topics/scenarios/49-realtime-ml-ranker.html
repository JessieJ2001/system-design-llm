Designing a Real-Time ML Ranker

A personalized ranker that scores 500 candidate items per request at 100 k requests/sec with a p99 ≤ 80 ms budget — where each scoring needs ~1 k features per (user, item) and the model is retrained daily on 30 days of impression+engagement events (10 B events/day) — while ten-plus A/B experiments each pin their own model version and online and offline features must agree exactly. We start from a synchronous Python server that fetches every feature from a Postgres table per request, loads the model per-pod, and batch-retrains nightly on one VM — which melts the database, blows the latency budget, costs millions a month, and silently fails on training/serving skew, model freshness, and experiment isolation — and iterate to a dedicated online feature store with item-feature caching, a microbatched GPU inference tier, a write-path feature pipeline that logs served features for point-in-time-correct training, and a continuous-training loop with a model registry, per-experiment model pinning, and drift-gated rollback.

S0. Scenario & Requirements

A personalized ranker — the thing behind a Meta/X feed, a Google/LinkedIn search-results re-rank, a YouTube watch-next, an Amazon/DoorDash “recommended for you,” an ad-auction quality score. A retrieval layer (out of scope — ANN / two-tower / heuristics) hands us ~500 candidate items per request; we score each with a learned model on ~1 k features per (user, item), sort, and return the top K, all inside an 80 ms user-facing budget. There are four subsystems, not one: the online inference path (request → features → model → sort), the feature store (online for serving + offline for training, and they must agree), the offline training pipeline (build the dataset from logs, train, evaluate, register a new model), and the experiment platform (assign each request to an A/B arm; each arm pins a model version). The tension that shapes every iteration: a ranker looks like a stateless RPC service, but it isn't — its features go stale, its training and serving paths skew, a bad model is a silent outage, ten experiments need ten models at once, and the bottleneck shape changes every day when the model retrains.

System class

Class: algorithmic-ML — a request-time scoring API plus an offline training/refresh pipeline — see functional-requirements.html#class-algorithmic. The FR vocabulary combines product-shaped (the Rank call) with pipeline-shaped (the training loop); the load-bearing NFRs are the per-class additions from nfr-checklist.html — training cadence, drift tolerance, online↔offline parity, model freshness, cold-start, fairness bounds, experiment latency — alongside the usual online latency/throughput.

Functional requirements (cite functional-requirements.html#class-algorithmic)

Online (the scoring API):

  • Rank(user_id, [candidate_id], context, experiment_overrides?) → [{candidate_id, score}] — for each candidate, assemble its feature vector, run the model assigned to this request's experiment arm, sort by score, return the top K (K is the caller's page size, ≤ the 500 candidates in). The features and the model version used must be recoverable for this exact request (logged for training + debugging).
  • Experiment assignment — each Rank call is deterministically bucketed into the experiment arms it qualifies for (sticky per user, salted, mutually-exclusive per layer); each arm names a model version. (The bucketing machinery itself is the subject of #40 — we consume it, we don't re-derive it.)

Offline (the training/feature pipeline):

  • Feature definition + materialization — a feature is defined once (its transformation, entity, freshness SLA, owner); a pipeline computes it from raw events and writes it to the online store (for serving) and the offline store (for training); the values served at inference time are logged so the training set is point-in-time correct.
  • Train a model version — on a cadence (and on-demand): read the training window from the offline store, train, run offline eval (including fairness/segment guardrails), and register the resulting model with its metadata (training window, feature-schema version, eval metrics) in a model registry. A model that fails a guardrail is not registered as deployable.
  • Promote / roll back a model — move a registered model into production (or an experiment arm), and roll back to last-known-good on a drift or guardrail alarm.

Out of scope (named, not silently dropped): candidate retrieval — the layer that produces the 500 candidates (ANN/two-tower/heuristic recall) is upstream and has its own hot-item bottleneck; the 80 ms budget explicitly excludes it (a probe names it). The experiment analysis pipeline — computing whether arm B beat arm A is downstream of “which arm did this request get.” The training algorithm itself — we treat “train a model” as a black box with a FLOP/row cost; whether it's a DLRM-style DNN or a GBDT is a tunable (we take the DNN-on-GPU path and name the GBDT-on-CPU fork). The serving UI / clients. Multi-region — each region runs its own serving stack + feature-store replica; training is global.

Flagged assumptions (would confirm with the interviewer): (1) the 1 k features per (user, item) decompose — in practice ~400 are user features (one vector per user, shared across all 500 candidates), ~500 are item features (one vector per item, shared across all users requesting it), ~100 are context/cross features computed in-request — this decomposition is what makes feature retrieval and caching tractable; if features were genuinely per-(user, item) pair (50 B distinct pairs) the design changes radically. (2) Item-popularity is Zipfian (α ≈ 1) — a small head of items appears in most requests, so an item-feature cache has a high hit rate. (3) Average ~1 k items in the active catalog per user-segment that actually get requested; ~100 M users, ~500 M items in the catalog. (4) “Daily retrain” is the starting cadence; the model's predictive quality decays over hours, so this is itself an NFR to tighten. (5) The model is “modest” — ~20 MFLOP per scored row (a few MLP layers + embedding lookups); the model file is ~500 MB.

The fundamental tension: a ranker is a stateful, time-varying machine wearing the costume of a stateless RPC service. The Rank call looks like “function in, list out” — but the function depends on features that are decaying, on a model that was trained on a different computation of those features unless you're disciplined, on a model version that's one of ten in flight, and on a training loop whose cadence determines how wrong the answer is allowed to get. Every iteration buys back one of those properties — tractable feature fan-in, cheap model compute, exact parity, fresh-enough models with safe rollback — and the bill (feature-store reads + inference compute + training compute) is the price of treating it as the system it actually is, not the service it pretends to be.

S2. API contracts (cite api-design-patterns.html)

Three surfaces: the online Rank RPC (called per request), the feature-store read API (called by the ranker, internal), and the offline training-pipeline interface (called by an orchestrator on a cadence). Patterns — versioning, idempotency, errors — cite the canonical; only the scenario-specific shape is inline.

Online — the Rank RPC + the feature-store read API

// Online scoring (gRPC; called 100k/s):
Rank(RankRequest{ user_id, candidate_ids: [<= ~500 item ids], context: {surface, device, session_features...},
                  experiment_overrides?: {experiment_id -> arm_id}  // for forced-arm debugging
                }) -> RankResponse{ ranked: [{candidate_id, score}], model_id_used, feature_schema_version, request_id }
  // p99 <= 80 ms, server-side, EXCLUDING the upstream retrieval call. request_id is logged with the served features.
  // On a feature-store miss for an item: serve a "no features" fallback vector (cold-start item) -- never error.
  // On a model-load failure for the assigned arm: fall back to the production model -- log it, alarm it -- never error.

// Feature-store read API (internal; called by the ranker per Rank):
GetFeatures(entity_type, entity_ids: [...], feature_groups: [...]) -> { entity_id -> {feature_group -> vector} }
  // ONE multi-get for the user vector; ONE batched multi-get for the (uncached) item vectors. p99 <= ~5-10 ms (with hedge).
  // Returns the feature_schema_version alongside; the ranker checks it matches the assigned model's expectation.

Versioning:  cite api-design-patterns#versioning -- RankRequest/Response are versioned; model_id and feature_schema_version
             are returned so the caller (and the impression log) records exactly what scored this request.
Idempotency: Rank is a pure read (no side effect beyond logging the impression, which is keyed by request_id). cite api-design-patterns#idempotency.
Errors:      cite api-design-patterns#errors -- the hot path NEVER throws on a missing feature / missing model; it degrades to a
             fallback (cold-start vector / production model) and emits a metric. A schema-version MISMATCH is the one hard error
             (refuse to serve garbage) -- it should be impossible if the pipeline is disciplined; the tripwire catches a bad deploy.

Offline — the training-pipeline interface

// Feature definition + materialization (the pipeline is config-driven by the registry):
RegisterFeature(FeatureDefinition{ name, entity_type, transform, freshness_sla, owner, schema_version }) -> ok | conflict
  // conflict if a feature of this name already exists with a different transform -- "compute it once" is enforced here.
GetTrainingDataset(window: [from, to], feature_schema_version, sample?) -> a columnar dataset URI (point-in-time-correct rows)

// Train + register + promote:
TrainModel(TrainSpec{ base_model_id?  // warm-start from this (mini-retrain) or null (cold full retrain)
                      window: [from, to], feature_schema_version, hyperparams }) -> training_job_id
  // on completion: runs offline eval (NDCG@K, per-segment engagement deltas, fairness metrics vs the incumbent),
  // writes the artifact to the model registry with status = "deployable" iff every guardrail passes, else "registered" (NOT deployable).
PromoteModel(model_id, target: "production" | experiment_arm)  -> ok    // requires status = "deployable"
RollbackModel(target)  -> ok    // sets target back to last-known-good model_id; the inference tier reloads within seconds.
GetModel(model_id) -> { artifact_uri, metadata }                       // the inference tier pulls this on assignment to an arm.

Cadence: a scheduler (Airflow/Argo) calls TrainModel(base=prod_model, window=last 24h) hourly (mini-retrain, warm-start)
         and TrainModel(base=null, window=last 30d) daily (full retrain). On-demand: an operator calls it ad hoc; SLA <= 6 h.
Errors:  a training job that fails a guardrail is NOT promotable; a job that errors is retried; the scheduler alarms on a missed cadence.

Per-call SLO

  • Rank — p99 ≤ 80 ms server-side (target steady-state ~15–25 ms; the 80 ms is the budget the design must guarantee under tail conditions), p999 ≤ ~150 ms; excludes candidate retrieval.
  • GetFeatures — p99 ≤ ~5–10 ms for the batched item multi-get (with a hedge on the slowest shard); p99 ≤ ~2 ms for the single user-vector get.
  • Training: a model registered every ≤ 1 h (mini) / ≤ 24 h (full); on-demand ≤ 6 h; offline-eval gate runs before any promotion; promotion/rollback take effect on the inference tier within ≤ ~30 s.

S3. Back-of-Envelope — the ratios that govern every iteration

No optimization technique appears here. The online feature store, the item-feature cache, microbatched GPU inference, the write-path feature pipeline, continuous training, and per-experiment model pinning are named as future moves — not used in any calculation below. The BOE math is what the Naive design will actually compute.

The feature fan-in — the headline

100k Rank RPS  x  500 candidates/request  =  50 M candidate-scorings/sec.
50 M scorings/sec  x  ~1,000 features/scoring  =  50 BILLION feature-lookups/sec  (if you literally look up 1k features per (u,i)).
But the features decompose: ~400 user features (1 vector per USER, reused for all 500 candidates) + ~500 item features (1 vector per
  ITEM, reused for every user who gets that item) + ~100 context features (computed in-request from the request itself).
=> per request you actually fetch:  1 user vector (~2 KB)  +  500 item vectors (~2 KB each = ~1 MB).
=> at 100k RPS:  100k user-vector reads/sec  +  50 M item-vector reads/sec  +  ~100 GB/s of item-feature bytes off the wire.
The 50 M item-vector reads/sec is THE number every iteration revolves around: a single OLTP DB serves maybe ~5-50 k such reads/sec
  before its disk/CPU/network saturates -- ~1000x short. The fix is (a) a purpose-built KV store for the reads and (b) caching the
  ITEM vectors (they're shared -- a Zipfian head means a small cache absorbs most of the 50 M/sec). Both are Iteration 0.

The model compute — second

model ~20 MFLOP/scored row  x  50 M rows/sec  =  1 x 10^15 FLOP/sec  =  1 PFLOP/sec of inference compute, sustained, at peak.
On a CPU core doing numpy/BLAS for a tiny model: ~10-30 GFLOP/s effective -> 1 PFLOP/s needs ~30-100 THOUSAND cores -> absurd as
  "one core per Python request." On an accelerator (L4) at ~48 TFLOP/s usable: ~21 L4 of raw compute -- IF the model runs batched on
  the GPU, not row-by-row in Python. Naive runs it in Python; Iteration 1 moves it to a batched GPU tier.
Per-request: 500 rows x 20 MFLOP = 10 GFLOP. On one L4 alone: 10e9 / 48e12 = ~0.2 ms of compute -- but weight-load (~500 MB / 300 GB/s
  = ~1.7 ms) dominates at batch=500; microbatching several requests amortizes the weight-load (the #34 trick) -> ~ms-scale per request.

The four ML-class hazards a stateless-web-service mindset misses

(1) PARITY:  if training computes a feature one way (a SQL aggregation over 30 days of logs) and serving computes it another way
    (a live read from some table), they disagree -> the model is trained on numbers it never sees in prod -> it performs WORSE online
    than offline -> training/serving skew. Fix (Iter 2): compute each feature ONCE on the write path, write it to BOTH stores, and LOG
    the value actually served so the training set is the served features by construction. Naive has two code paths -> skew by default.
(2) FRESHNESS:  a nightly retrain means the model is up to ~24 h stale. If engagement shifts overnight (a trending topic, a holiday,
    a viral event), the model is wrong and the system doesn't know. Fix (Iter 3): hourly mini-retrains + a "recency" sub-model + drift
    detection. Naive: ~24 h stale, no detection -> a wrong model for hours.
(3) DRIFT / ROLLBACK:  a bad model is a SILENT outage -- the system returns 200 OK with garbage rankings; engagement craters; nobody
    is paged unless someone is watching realized-vs-predicted engagement (a delayed signal). Fix (Iter 3): drift monitors + auto-rollback
    to last-known-good. Naive: deploy-and-pray (CLAUDE: a model with no quality NFR has no rollback criterion).

S4. Naive Architecture — the simplest thing that meets the FRs

A FastAPI/Flask ranking service. Each pod loads the model file into RAM at startup. Per request: receive (user_id, [500 candidate_ids], context); issue one big SELECT ... WHERE entity_id IN (...) against a Postgres table that stores feature vectors as JSON/array columns — one query for the 500 item vectors, one for the user vector; assemble a 501×~1k feature matrix in Python; call model.predict(matrix) (numpy/BLAS for a small DNN, or lightgbm.predict for a tree); sort the 500 scores; return the top K. Offline: a nightly cron on one big VM runs a SQL job that reads 30 days of impression+engagement logs, computes features with aggregations, joins labels, trains the model, writes the new model file to a shared path; pods pick it up on the next restart. It answers every FR (rank, assign-an-experiment? — well, no, there's only one model; train; promote = “copy the file”). The numbers say where it breaks — and the ML-class NFRs say it was never going to work.

Why it's broken (per failing NFR)

  • Ranking latency — FAILS, structurally (the shape-determining failure). The feature fan-in — 500 item-vector reads per request × 100 k RPS = 50 M reads/sec + ~100 GB/s of bytes — cannot come from a single OLTP DB; p99 of the feature fetch alone is seconds, not the ≤80 ms budget. Classified tail-latency driven by a fan-in the storage can't serve (it's also a round-trip-and-bandwidth problem — ~100 GB/s of feature bytes — and a disk-IO one). This is the only failure that determines the architecture's shape — the only fix is to stop reading 500 vectors from an OLTP DB per request → Iteration 0. (The model-compute failure is also fatal but is a separate, second move.)
  • Throughput / cost — FAILS. ~$7–25 M/mo of slow Python pods, and the DB melts at any pod count — a non-starter.
  • Online↔offline parity — FAILS, by construction. Two code paths compute “the same” feature → training/serving skew → a model that works in eval and underperforms in prod, undetectably.
  • Feature staleness & model freshness & drift — FAIL. Nightly batch → ~24 h-stale features and a ~24 h-stale model, with no drift monitor → a wrong model serves for hours and nobody is paged.
  • Fairness / eval guardrails — FAIL. No offline-eval gate → a regression ships silently → deploy-and-pray.
  • Per-experiment model pinning — FAILS. One model file for everyone → zero experiments possible.
  • The sort step (the LC-analog) — ✅. Heapifying 500 floats and taking the top K is ~µs — the LC-analog step is, as promised, the least interesting part. It stays fine through every iteration.

Cost of naive at NFR-target load: ~$7–25 M/mo of slow Python pods — and even that doesn't work, because the single Postgres melts under 50 M item-vector reads/sec regardless of how many pods you add, and the design fails six of the eight NFRs outright. Call it ~$15 M/mo and broken — the anchor for the iterations, which take it to ~$200 k/mo and meeting every NFR (≈70× cheaper, and actually functional). The lesson the cost number under-states: most of the gap isn't dollars, it's that a ranker built like a stateless web service is wrong about features, training, and experiments — the four iterations are mostly buying correctness, and the cost falls out as a side effect of doing it right.

Iteration 0 — A dedicated online feature store + item-feature caching + request-time batching

Bottleneck

Naive's shape-determining failure is the feature fan-in — 500 item-vector reads per request × 100 k RPS = 50 M reads/sec + ~100 GB/s of bytes — served from a single OLTP DB that delivers maybe ~5–50 k reads/sec, so p99 of the feature fetch alone is seconds, ~1000× over the ≤80 ms budget. Classified tail-latency from a fan-in the storage can't serve, with a round-trip-and-bandwidth and disk-IO component — the signal: a per-request fan-out of N reads to a store that's sized for OLTP point-reads, where N×RPS dwarfs the store's QPS and N×RPS×value_size dwarfs its bandwidth. Per bottleneck-to-decisions#tail-latency the levers are: change the store (an OLTP row store → a horizontally-sharded KV store built for high-QPS small-value point reads); cache the hot subset (the item vectors are shared across requests — a Zipfian head means a small cache absorbs most reads); collapse duplicate work (single-flight the concurrent misses for the same key); and batch the per-request fan-out (one multi-get, not 500 round trips). This iteration does all four — they all target the one bottleneck.

Architectural decisions to revisit

Per bottleneck-to-decisions#tail-latency for a fan-in the storage can't serve: the storage engine for online features (cite storage-engines#kv-engine — this is the textbook KV-engine workload: keyed point reads of a small fixed-shape value at high QPS), the caching layer (per-pod vs a shared cache tier — cite caching.html), the request batching (multi-get vs N round trips — cite network-rtt-bound), and miss collapse (cite single-flight.html). It does not touch the model-serving tier (still Python, Iteration 1), the feature-computation path (still the cron's SQL, Iteration 2), or the training/registry/experiment machinery (Iteration 3) — one bottleneck per iteration.

Alternatives considered (C4)

  • Self-managed ScyllaDB as the online feature store + a per-pod LRU over item vectors — chosen. Sharded by entity_id (cite sharding-strategies — hash on the entity id; uniform, no hot partitions because a popular item is one key, not one shard), value = the feature-group vectors, RF=3 for durability. ScyllaDB does ~1 M small reads/sec/node and has predictable single-digit-ms p99 → ~6 nodes of raw capacity for 5.1 M reads/sec, ~15–20 with RF=3 + headroom; storage = ~100 M users × ~2 KB + ~500 M items × ~2 KB ≈ ~1.2 TB × 3 = ~3.6 TB — comfortably on those nodes. The per-pod LRU is what makes it cheap: item vectors are shared across all users who get that item, item popularity is Zipfian (α≈1), so the top ~1 M items (×~2 KB ≈ ~2 GB, fits in a 64 GB pod's RAM) cover ~90% of the 500 lookups per request → the store only sees the ~10% tail. Single-flight (cite single-flight.html) collapses the burst when a hot-new item is requested by many concurrent requests on the same pod. Operability: self-managed (one team owns it), nodetool + Scylla Monitoring; known hazard = compaction storms (size for it), and a hot key is just a key (not a shard) so it's a cache problem, not a partition problem.
  • DynamoDB on-demand — rejected on cost. The math is brutal: ~5 M item-vector reads/sec (after caching) × ~2.6 M sec/mo = ~1.3×1013 reads/mo × $0.25/M (cite pricing-constants#storage) = ~$3.3 M/mo just for the feature reads — ~25× the entire self-managed-ScyllaDB bill. DynamoDB is the right call for spiky, low-QPS, ops-averse workloads; a steady 5 M reads/sec is the opposite. (Constraint-flip: 🔀 if the read rate were ~100 k/sec instead of ~5 M, or extremely bursty with long idle stretches, DynamoDB's zero-ops + pay-per-use wins.)
  • Redis Cluster as the online store — rejected as the primary, used as a hint. Redis is faster per op but it's RAM-only — storing ~1.2 TB of feature vectors (×replication) in RAM is ~$10 k+/mo of cache.r6g nodes (cite pricing-constants#managed-services) and you'd still want a durable store behind it for the cold tail (~500 M items, mostly cold). ScyllaDB keeps the working set in its row cache and pages the cold tail from NVMe at ~the same latency for far less money. (A small Redis is fine as a shared L2 cache in front of ScyllaDB if the per-pod LRU's hit rate disappoints — that's a tuning escape hatch, not the base design.)
  • Feast (open-source feature store) on Cassandra/Redis, or a managed feature platform (Tecton, SageMaker/Vertex Feature Store) — rejected for now, revisited in Iter 2. Feast/Tecton are frameworks — their value is the feature registry + the online/offline materialization discipline (the parity story), not raw read throughput; under the hood they sit on a KV store like ScyllaDB anyway. We want the registry/parity machinery, but that's the Iteration 2 move (the bottleneck there is skew, not read latency); here the bottleneck is purely the read fan-in, so we install the KV store now and the registry later. (When we get to Iter 2 we adopt the Feast-style registry pattern on top of this ScyllaDB.)

The chosen move

Move online features into ScyllaDB, sharded by entity_id: user feature vectors keyed by user_id, item feature vectors keyed by item_id, each value a small fixed-shape blob of the feature groups (~2 KB). The ranking pod becomes an orchestrator: per Rank it (a) does one multi-get for the user's vector, (b) checks its per-pod LRU for each of the 500 item ids, (c) does one batched multi-get to ScyllaDB for the ~50 misses (single-flighting concurrent misses for the same id), (d) fills the LRU, (e) assembles the 501×~1k matrix, (f) — still — scores it in Python, sorts, returns top-K. Why this resolves the bottleneck: the 50 M item-vector reads/sec that melted Postgres become ~5 M reads/sec to a store built for exactly that (sharded, ~1 M small reads/sec/node, page-cache-friendly), and ~100 GB/s of feature bytes off the wire become ~10 GB/s (after the cache) spread across ~15–20 nodes — both well within capacity, with p99 batched-multi-get ~5–10 ms (hedge the slowest shard to keep the tail tight, cite tail-latency). The cache works because the read pattern is power-law / long-tail on the item dimension — a small head, fat tail; the head fits in RAM. Cite storage-engines#kv-engine, caching.html, single-flight.html.

What breaks (C5; cite failure-modes.html)

  • Slow dependency — the feature store. A ScyllaDB compaction storm, a hot shard, or a node loss spikes the batched-multi-get p99 to ~50–100 ms → Rank p99 blows the budget for the affected fraction. Defence: hedge the slowest shard in the multi-get (send to a replica after a short delay); a per-request deadline — if the item multi-get hasn't returned by ~T ms, score with whatever item vectors are in the LRU + a cold-start fallback vector for the rest (degraded ranking quality, but on-budget and a metric fires); single-flight + the LRU already shield against a hot-new-item burst. Blast radius: the fraction of requests touching the slow shard; detection: feature-store p99 SLI + the “served-with-fallback-vectors” counter; recovery: shed/throttle the compaction, or fail the shard over.
  • Stampede on a cache cold-start. A wave of new ranking pods (a deploy) starts with empty LRUs → ~100% miss for a few minutes → ~50 M item-vec reads/sec briefly hit ScyllaDB — ~10× steady-state. Defence: single-flight (within a pod); rolling deploys (don't replace the whole fleet at once); ScyllaDB sized with headroom for a ~2–3× burst; optionally pre-warm a new pod's LRU from a snapshot of the hot-item set on startup. Detection: feature-store QPS spike correlated with a deploy; recovery: the LRUs refill in minutes.
  • Hot key — a viral item. One item id suddenly in every request (a celebrity post, a flash-sale product) → that one ScyllaDB key is read ~100 k+/sec. But it's one key, not one shard — and the per-pod LRU caches it after the first miss, so each pod reads it ~once, not ~thousands of times; single-flight collapses the concurrent first misses. The defence is the design, not a bolt-on. Detection: per-key read-rate outliers; if a key ever does saturate a shard's replica set, replicate the hot key to extra replicas (cite hot-key). Blast radius: ~one shard, briefly, before the caches fill.

Bottleneck

After Iter 0, the feature fetch is ~5–10 ms but Rank p99 is still ~0.3–1 s — because the pod scores 500 rows of a DNN row-orchestrated in Python, single-threaded per request, no batching across requests: ~10 GFLOP/request at ~10–30 GFLOP/s effective ≈ ~0.3–1 s, and ~5 req/s/pod → ~tens of thousands of pods → ~$12 M/mo. This is a compute-bound hot path running on the wrong hardware in the wrong shape — the signal: a forward pass that is intrinsically a dense matrix multiply (500×~1k) being executed as a Python loop on a general-purpose core instead of a batched kernel on an accelerator (or a vectorized BLAS/AVX call). Per bottleneck-to-decisions#compute-bound the levers are: move to the right hardware (a GPU/TPU for a DNN; a vectorized-CPU runtime for a GBDT), batch the work to fill it (microbatch concurrent requests so the accelerator runs near its roofline and the weight-load is amortized — exactly #34's continuous-batching argument), and separate the tier (a dedicated inference service so the batching layer can pool work from all orchestrator pods). It also forks on the model type: a DNN wants a GPU; a GBDT wants vectorized CPU — we take the DNN path and name the fork.

Architectural decisions to revisit

Per bottleneck-to-decisions#compute-bound: the compute substrate (general-purpose CPU loop → batched accelerator kernel or vectorized-CPU SIMD), the batching strategy (per-request → cross-request microbatching — cite #34 for the mechanics; we do not re-derive batching/roofline math), the tier topology (model-in-the-app-pod → a dedicated inference service), and the serving framework. It does not touch the feature store (Iter 0, untouched), the feature-computation path (Iter 2), or training/registry/experiments (Iter 3).

  • TF-Serving / TorchServe — rejected (close call). Both do batched serving and multi-version serving. Triton wins on: hardware-agnostic backends (run the GBDT fork on the same server), better dynamic-batching ergonomics, and being the de-facto pairing with the #34-style microbatching story. If the model were pure TensorFlow, TF-Serving's tighter integration would make it a wash; we don't bet the design on the framework, but Triton is the safest single choice for a heterogeneous model zoo.
  • A custom Rust/C++ scorer in the orchestrator pod (no separate tier) — rejected. A hand-rolled SIMD/CUDA scorer embedded in each orchestrator pod removes a network hop (~1 ms) — but it can't pool work across pods (each pod batches only its own concurrent requests ≈ far below the accelerator's efficient batch size), every model update is a binary deploy, and you've rebuilt Triton badly. The ~1 ms hop is cheap insurance against all of that. (Constraint-flip: 🔀 if the latency budget were ~5 ms total — an ad-auction inner loop — the network hop would be unaffordable and an in-process vectorized scorer (with per-pod batching only) becomes the answer, at a much larger fleet.)
  • GBDT-on-vectorized-CPU instead of DNN-on-GPU — the model-type fork (named, not taken). If the model is a gradient-boosted-tree (XGBoost/LightGBM) rather than a DNN, predict over a 500×~1k matrix is a vectorized tree-traversal — ~1–5 ms on a single AVX-512 core, no GPU needed. Fleet: ~100k RPS ÷ (~200–1000 such batches/s/core) ≈ ~100–500 cores ≈ ~10–50 c6i/c7g instances ≈ ~$5–25 k/mo — cheaper than the GPU fleet, because a tree model's FLOP/row is effectively tiny and CPUs are cheap per FLOP at that scale. You'd serve it from Triton's CPU backend or a vectorized GBDT runtime (Treelite, ONNX Runtime). The cross-over: a DNN's accuracy-per-FLOP can beat a GBDT on rich feature sets (embeddings, sequence features), which buys engagement — so the choice is “cheaper tree vs more-accurate DNN at higher inference $”, which is exactly the cost-vs-quality knob (S6). This page takes the DNN-on-GPU path because it's the more demanding compute story (and reuses #34); a real team picks per their model's measured accuracy lift.

The chosen move

Lift the model out of the Python pod into a dedicated Triton inference tier on ~50 L4 GPUs, fronted by a microbatching layer. The orchestrator pod, per Rank, now: fetches features (Iter 0), assembles the 500×~1k matrix, makes one gRPC call to the inference tier, gets 500 scores back, sorts, returns top-K. Triton's dynamic batcher coalesces concurrent orchestrator calls into one accelerator batch — why this resolves the bottleneck: a DNN forward pass is a dense matmul, which is exactly what a GPU is for; running it batched (thousands of rows per step) puts the accelerator near its compute roofline (the ~1.7 ms model-weight-load gets amortized over the whole batch — the #34 argument applied here; cite 34-llm-inference-serving.html), so the 1 PFLOP/s of inference compute fits in ~21 L4 of raw capacity (×~2.4 headroom ≈ ~50). The orchestrator pods are now mostly I/O wait (feature fetch + inference RPC), so each handles ~thousands of Rank/s → ~75 r6i.2xlarge glue pods (down from tens of thousands of CPU-bound Python pods). End-to-end p99: ~5–10 ms (features) + ~2–5 ms (model microbatch) + ~µs (sort) + ~2–3 ms (assemble + serialize + RPC) ≈ ~15–20 ms — comfortably inside the 80 ms budget with ~60 ms of slack for tail conditions.

What breaks (C5; cite failure-modes.html)

  • Slow dependency — the inference tier. A GPU node dies, or the microbatcher's queue backs up (a load spike), and model-score p99 jumps to ~50–200 ms → Rank p99 eats into (or blows) the 80 ms budget. Defence: a per-request deadline on the inference RPC — if it doesn't return by ~T ms, return the candidates in retrieval order (or a cheap heuristic order) and mark the request “unranked” (degraded quality, on-budget, a metric fires); autoscale the GPU tier on queue depth; cap the microbatch max-delay so the batcher never adds more than a few ms of latency. Blast radius: the affected fraction; detection: inference-tier p99 + queue-depth SLIs + the “unranked” counter; recovery: scale out / fail the node.
  • Bad deploy — a bad model binary or a wrong feature-schema-version. A new model file (still the cron's, this iter) is corrupt, or expects a different feature layout → the inference tier either crashes on load or scores garbage. Defence (partial this iter, completed in Iter 3): the inference tier verifies the model's feature_schema_version against the online store's current schema on load — mismatch = refuse to load, keep serving the previous model, alarm (the parity tripwire); a content-hash check on the artifact catches a corrupt file. Real fix is Iter 3's registry + canary; here the tripwire at least prevents serving garbage. Detection: load-failure alarm; recovery: the tier keeps the last-good model; operator investigates.
  • Stampede / retry storm onto the inference tier. If orchestrator pods retry a slow inference RPC aggressively, a brief GPU-tier hiccup turns into a retry amplification that keeps it down. Defence: bounded retries with jittered backoff; a circuit breaker on the inference client (if p99 is bad, stop retrying, go straight to the “unranked” fallback); the deadline above caps how long any one call waits. Detection: inference RPC error/retry rate; recovery: the breaker sheds load until the tier recovers.

Bottleneck

After Iter 1 the online path meets latency, throughput, and cost — but the model is being trained on the wrong numbers: the cron computes “clicks_last_7d” one way (a SQL GROUP BY over the log), the online store has it computed another way (whatever populated it — now even the cron itself, a third path), and what the model actually saw at serve time was whichever-was-in-the-store-at-that-instant — none of these agree on cutoffs, timezones, late-arriving events, or null handling. The result is training/serving skew: the model's offline NDCG is optimistic and its online engagement is worse, and nothing detects it because there's no comparison point. And the same nightly batch makes every sub-day feature ~24 h stale. This is a data-correctness failure — the data the model trains on doesn't match the data it serves on — with a freshness-lag component on the sub-day features. There's no canonical “bottleneck” entry for “training/serving skew” because it's an ML-class hazard, not a capacity bottleneck (a gap in the canonicals — flagged in the report); the closest analogues are the system-of-record / derived-stores discipline (one source of truth, derived views kept consistent by one pipeline — cite storage-engines#kv-engine for the online derived store and storage-engines#columnar for the offline one) and replica-divergence (two copies of “the same” data drift apart unless one writer owns both). The lever: compute the feature once, on the write path, from one definition, and write it to both stores — plus log what was served so the training set is the served features, not a re-derivation. That's the move.

Architectural decisions to revisit

The feature-computation topology (read-time / nightly-batch / N divergent jobs → one write-path streaming pipeline + a feature registry as the single definition); the offline store for training data (cite storage-engines#columnar — a columnar, time-keyed store is exactly right for “scan a 30-day window of feature values, point-in-time joined”); the event-log substrate (an OLTP table → a partitioned, replayable log — the training pipeline needs replay); and the training-data assembly (re-derive features → join the logged served features). It does not touch the online inference path (Iter 0–1, untouched) or the model-training cadence / registry / experiments (Iteration 3).

Alternatives considered (C4)

  • A Feast-style feature registry + an Apache Flink streaming feature pipeline writing to both stores + logged served features — chosen. The registry (Feast OSS, or a hand-rolled equivalent) is the source of truth for what a feature is — one definition (transform DSL/SQL/UDF, entity, freshness SLA, owner, schema version); registering a second definition of the same name is a conflict. The pipeline is Flink: a stateful stream processor — the right tool for windowed aggregations (“clicks_last_1h” is a sliding window over the event stream), exactly-once sinks, and event-time/watermark handling for late events; it consumes the Kafka firehose (~500 k/s peak) on ~20–40 TaskManagers (~10–30 k events/s/TM) and dual-writes each computed value to ScyllaDB (latest) and the offline store (timeline). The logged-features trick — the orchestrator attaches the feature vectors it actually scored to the impression event — is what makes parity exact: the training set is literally the served features + the realized label, not a best-effort re-derivation. Cite storage-engines#kv-engine (online derived store) + storage-engines#columnar (offline store). Operability: Flink has the well-known hazards (checkpoint storms, state-backend pressure — RocksDB state backend, size for it; backpressure when a sink is slow — ScyllaDB write capacity must keep up); the registry is small and relational.
  • Spark Structured Streaming instead of Flink — rejected (close call). Spark Structured Streaming does micro-batch streaming and would also dual-write; Flink wins on true event-time semantics with watermarks (sub-second windows; correct late-event handling), lower per-event latency (record-at-a-time vs micro-batch), and richer stateful operators — all of which matter for “clicks_last_1h”-style features with a tight freshness SLA. If the org already runs a Spark shop and the freshest feature SLA were ~minutes (not seconds), Spark is a defensible reuse. Either way the pattern — one pipeline, both stores, registry-driven — is the same.
  • Kafka Streams (a library, not a cluster) — rejected at this scale. Kafka Streams is great for “a few stateful transforms close to the topic” with no separate cluster — but at ~500 k events/s with hundreds of feature definitions, complex windowed state, and a dual-sink fan-out, you want a managed dataflow engine with proper state backends and operational tooling, not a library scaled by adding app instances. Kafka Streams is the right call for a small feature surface, not this one.
  • Keep batch feature computation (nightly Spark/SQL), just centralize the definition (one job, both stores) — rejected on freshness. Centralizing the definition fixes parity (one computation, both stores) — but a nightly batch still makes every sub-day feature ~24 h stale, failing the staleness NFR for “clicks_last_1h”-class features (and those are exactly the high-signal ones for a real-time ranker). Streaming buys both parity and freshness; batch buys only parity. (A hybrid is real and fine: streaming for sub-day features, a nightly batch for genuinely-daily-or-slower ones — both go through the same registry; we'll note that in S6.) (Constraint-flip: 🔀 if every feature were genuinely daily-or-slower — e.g. a pure long-term-preference model — a nightly batch through the registry is enough and the Flink cluster is unjustified.)
  • A managed feature platform (Tecton, SageMaker / Vertex Feature Store, Databricks Feature Store) — rejected for ownership, would reconsider for a small team. These bundle the registry + the materialization pipelines + the online/offline stores into one product — less to build and operate, and they get the parity discipline right. The reasons to self-assemble here: at ~50 M item-vec reads/s + ~500 k events/s the managed online-store tier becomes very expensive (the same DynamoDB-style math as Iter 0), you want control over the Flink job's logic and the ScyllaDB topology, and a platform team already owns Kafka/Flink/ScyllaDB. A smaller org with no platform team should buy, not build.

The chosen move

Install a feature registry (Feast-style: one definition per feature — transform, entity, freshness SLA, owner, feature_schema_version) and an Apache Flink streaming feature pipeline driven by it: the pipeline consumes the Kafka event firehose, computes each feature exactly once from its single definition, and writes the value to both the online ScyllaDB store (the latest value, for serving) and a columnar offline store (Iceberg/Parquet on S3, keyed by entity + as_of timestamp, for point-in-time-correct training). The orchestrator pod, after returning each Rank response, emits an impression event to Kafka carrying the feature vectors it actually scored (refs + a sampled subset of full vectors) plus the model_id_used; the engagement stream is joined to it later to produce labels; the training job builds its dataset from these logged events — so a training example is literally the served features + the realized label. Why this resolves the bottleneck: there is now exactly one computation of each feature (the registry forbids a second definition; the pipeline is the one writer of both stores), so online and offline can't disagree — parity is structural, not reconciled; and the served-features log removes even the “re-derive the training features” step, so a feature-definition change can't retroactively skew old training data. Sub-day features go from ~24 h stale (nightly) to seconds-to-minutes (streaming). The Postgres event log becomes Kafka because the pipeline (and the training job) need a partitioned, replayable log, not an OLTP table. Cite storage-engines#kv-engine + storage-engines#columnar (the system-of-record → derived-stores discipline applied to features).

What breaks (C5; cite failure-modes.html)

  • Slow dependency — the feature pipeline falls behind. A Flink backpressure event (a slow ScyllaDB sink, a checkpoint storm, a TaskManager loss) → the pipeline's lag grows → sub-day features in the online store go stale (back toward the nightly-batch failure mode) — the serving path keeps working (it reads whatever's in the store), but ranking quality degrades quietly. Defence: a per-feature freshness SLI (online-store value's as_of vs now) with an alarm; autoscale Flink on lag; size the ScyllaDB write capacity for the firehose with headroom; partition the pipeline so a slow feature doesn't stall the others. Detection: feature-freshness SLI; recovery: scale Flink / fix the slow sink — lag drains.
  • Data corruption — a bad feature definition. Someone registers a feature transform with a bug (a wrong window, a unit error, a null that should be zero) → the pipeline writes garbage to both stores → the model serves on garbage and trains on garbage — and parity makes it consistent garbage (no skew alarm fires!). Defence: the registry requires a definition to pass validation (a backfill on a sample window + a distribution check vs the previous definition) before it goes live; a per-feature distribution monitor in the pipeline (alarm if a feature's distribution shifts abruptly); feature_schema_version bumps on a breaking change so a model trained on the old version refuses the new layout (the tripwire). Blast radius: every model and request using that feature; detection: the distribution monitor / a downstream model-quality regression; recovery: roll back the definition, backfill the offline store from the event log (it's replayable), re-materialize the online store.
  • Poison message — a malformed event. A corrupt or unparseable event in the Kafka firehose stalls a Flink operator (a parse exception in a loop). Defence: the pipeline routes unparseable events to a dead-letter topic and continues (never let one bad message stop the stream); a DLQ-size alarm; a schema on the event topic (a registry / Avro/Protobuf) so most malformations are caught at produce time. Detection: DLQ rate; recovery: investigate the producer; replay from the DLQ once fixed.

The flip clarifies why parity is an NFR here: a high-stakes ranker (feed, ads, search) where a few percent of engagement is millions of dollars can't afford skew, so it pays for the discipline; a low-stakes one can. The trap the page exists to break — “treat it like a stateless web service” — is specifically the high-stakes case where that mindset costs you the parity, the freshness, and the rollback story all at once.

Iteration 3 — Continuous training + model registry + per-experiment model pinning + drift-gated rollback

Bottleneck

After Iter 2 the model trains on parity-correct, fresh-enough features — but it's still a once-a-day, ungated, registry-less, one-for-everyone model. Three NFRs are still failing, and they're related (all about the model lifecycle, not the data): (1) freshness/drift — ~24 h stale, and a bad or stale model is a silent outage (200 OK with garbage rankings; engagement craters; nobody is paged) because there's no quality NFR with a rollback criterion; classified bad-deploy with no detection/rollback; (2) fairness/eval guardrails — a model ships regardless of offline metrics (deploy-and-pray); (3) per-experiment model pinning — one model file → zero experiments, when the NFR demands ≥10 concurrent, each with its own version on the same fleet. There's no single canonical bottleneck entry for “the model lifecycle is missing” (again an ML-class gap — flagged); the closest are bad-deploy (the cell/canary/automated-rollback discipline, applied to a model artifact) and cold-start (a freshly-promoted model warming, a new experiment arm loading). The lever: treat a model exactly like a versioned, canaried, rollback-able deploy artifact — a registry (every version + metadata + status), an eval gate (a model isn't deployable until it passes), continuous retraining (close the freshness gap), per-arm pinning (the experiment platform decides which version each request gets; the inference tier holds K of them), and drift-gated rollback (the missing quality NFR + the missing rollback criterion). All five target the one bottleneck (the absent model lifecycle), so they're one iteration.

Architectural decisions to revisit

Per failure-modes#bad-deploy (canary + automated rollback on SLO regression, applied to a model) and bottlenecks#cold-start (warming a promoted/new model): the training cadence + cluster (a cron-on-a-VM → a scheduled distributed-training cluster doing warm-started mini-retrains + a full retrain); the model artifact lifecycle (a file → a registry with versions, metadata, status, and an eval gate); the experiment↔model binding (cite #40 for the bucketing — we add the “arm pins a model_id” layer); the inference-tier memory budget (one model → K≈12–20 versions resident); and the drift/rollback path (none → monitors + auto-revert). It does not touch the feature store, the feature pipeline, or the online hot path's structure (Iter 0–2, untouched).

Alternatives considered (C4)

  • Hourly warm-started mini-retrains + a daily full retrain, on a distributed-PyTorch training cluster, with a model registry (MLflow-style / a managed equivalent), an offline-eval gate, per-arm model pinning via the #40 experiment platform, and drift-monitor-gated auto-rollback — chosen. Cadence: warm-starting from the current prod model on just the last ~24 h of data converges in ~10–30 min on ~10–20 nodes → a fresh model every hour (the freshness NFR); a daily cold full retrain on the 30-day window (~few hours on ~50–100 nodes) prevents warm-start drift accumulation. Registry: every TrainModel output is registered with its metadata; the eval gate sets status=deployable only if NDCG@K, per-segment engagement, and fairness metrics don't regress vs the incumbent past their thresholds — a regression is registered but unpromotable. Pinning: the experiment platform (#40) assigns an arm; the arm's config (in the registry) names a model_id; the request carries it; the inference tier (Triton, which natively serves multiple model versions) holds K≈12–20 of them (~500 MB × 20 ≈ ~10 GB < the L4's 24 GB — the headroom we sized into Iter 1's ~2.4× factor). Drift: a streaming monitor on the online prediction-score distribution vs the offline-eval distribution (immediate; catches a broken model) + a batch monitor on realized-vs-predicted engagement per arm (delayed by the feedback window; catches a quietly-wrong model); a threshold crossing → RollbackModel → the registry reverts → the inference tier reloads in ~30 s. Cite #40 for bucketing; cite failure-modes#bad-deploy for the canary/rollback shape.
  • Pure online learning / contextual bandits instead of batch retrain — rejected as the base, named as a probe. An online-learning model updates per (or per-mini-batch-of) interactions — freshness goes to ~seconds and you get exploration vs exploitation for free (a Thompson-sampling / LinUCB bandit explores under-served items). But: the infra is very different (a low-latency model-update path co-located with serving, careful handling of the delayed-and-biased feedback loop, much harder to A/B and to roll back — “roll back” means reverting a continuously-mutating model's state), the eval story is weaker (no clean held-out set), and a bug corrupts the live model directly. Hybrid is what real systems do: a batch-retrained base model + a thin online layer (a recency/bandit head) on top — we name that as a probe, not the Core path. (Constraint-flip: 🔀 if the dominant signal were extremely time-sensitive — breaking news, live-event ranking — a bandit/online-learning head becomes load-bearing, not optional.)
  • Just retrain nightly but add the eval gate + registry + per-arm pinning (skip the hourly mini-retrain) — rejected on freshness. The registry + gate + pinning are non-negotiable (they're three failing NFRs); the question is only the cadence. A nightly-only cadence leaves the model up to ~24 h stale — failing the model-freshness NFR for any workload where engagement shifts intra-day (a trending topic, a holiday). The hourly mini-retrain is cheap (warm-start, last 24 h, ~10–30 min) and closes that gap; the daily full retrain still happens for robustness. (For a slow-moving domain — long-term-preference rec with no intra-day dynamics — nightly-only is fine and the mini-retrain is over-engineering.)
  • A managed ML platform (SageMaker, Vertex AI, Databricks) for the whole training+registry+pipeline — rejected for ownership, would reconsider for a small team. These give you the registry, the training orchestration, the eval/monitoring tooling, and the deployment/rollback machinery as a product — far less to build. The reasons to self-assemble here: a platform team already owns the Kafka/Flink/ScyllaDB/Triton stack and the training cluster; the cost of a managed platform at this scale (training compute markup + per-prediction monitoring fees) is significant; and you want control over the eval-gate logic (the fairness/segment thresholds are business-specific) and the inference tier's K-versions topology. A startup with no ML-platform team should buy the platform and skip most of the assembly — the discipline (parity, registry, eval gate, drift rollback) is what matters, not whether you built it.

The chosen move

Replace the nightly cron with a training cluster on a scheduler (Airflow/Argo): a warm-started hourly mini-retrain on the last ~24 h (warm-start from the current prod model → ~10–30 min on ~10–20 distributed-PyTorch nodes) plus a daily full retrain on the 30-day window (cold → ~few hours on ~50–100 nodes); each training run is followed by an offline-eval gate (NDCG@K + per-segment engagement deltas + fairness metrics, computed on a held-out set vs the incumbent) that registers the model status=deployable only if every guardrail passes. A model registry holds every version with its metadata (training window, feature_schema_version, eval metrics, status) and is the source of truth for what each experiment arm pins. The experiment platform (#40 — deterministic salted layered bucketing, consumed not re-derived) assigns each Rank request to an arm; each arm's config names a model_id; the request carries it; the Triton inference tier holds K≈12–20 model versions resident (~10 GB < the L4's 24 GB — the headroom we already sized in) and routes the request to the pinned one. Drift monitors — online prediction-score distribution vs offline-eval distribution (immediate) + realized-vs-predicted engagement per arm (delayed by the feedback window) — auto-call RollbackModel on a threshold crossing; the registry reverts to last-known-good; the inference tier reloads in ~30 s. Why this resolves the bottleneck: a model becomes a versioned, evaluated, canaried, rollback-able artifact — the same discipline a code deploy gets (cite failure-modes#bad-deploy): the freshness gap closes (hourly), a regression can't ship (the gate), ten experiments can run at once (the registry + per-arm pinning + K-resident-versions), and a model going wrong in prod is detected and reverted automatically (drift monitors) instead of being a silent multi-hour engagement outage.

What breaks (C5; cite failure-modes.html)

  • Bad deploy — a bad model slips past the gate. A model passes offline eval (the held-out set didn't catch the failure mode) and is promoted to prod → online engagement craters. Defence: don't promote to 100% — canary the new prod model to ~1–5% of traffic (a #40 arm) first; the online guardrail metrics (per-arm engagement, error rate, prediction-distribution drift) gate the ramp; the drift monitors auto-roll-back on a regression; a one-command manual rollback always works. Blast radius: the canary fraction until promotion (or the whole prod arm if it slipped through — until the drift monitor's feedback window elapses, ~hours); detection: the canary's guardrail metrics / the realized-vs-predicted monitor; recovery: RollbackModel → ~30 s to revert.
  • A bad feature definition or schema bump corrupts every model overnight — “the bottleneck shape changes daily.” The pipeline (Iter 2) starts writing a subtly-wrong feature; the next mini-retrain trains on it and serves on it; parity makes it consistent (no skew alarm) → quality degrades and the system “looks fine.” This is the headline ML-class hazard: tomorrow's model is a different system than today's. Defence: the feature-distribution monitors in the pipeline (Iter 2); the offline-eval gate catches a quality drop before promotion (a model trained on a corrupted feature should fail NDCG-vs-incumbent); the feature_schema_version tripwire (the inference tier refuses a model whose schema doesn't match the online store's); a model can always be rolled back to a version trained on the pre-corruption data (the registry keeps them; the offline store is replayable so the feature can be backfilled and the model retrained). Detection: eval-gate failures / model-quality dashboards / the feature monitors; recovery: roll back the feature definition AND the model, backfill, retrain.
  • Cold-start — a new experiment arm or a freshly-promoted model warming. An experiment pins a model_id the inference tier hasn't loaded → the first requests to that arm wait on a model load (pull from registry, verify, load into GPU memory — ~seconds for ~500 MB); a freshly-promoted prod model similarly. Defence: pre-load a new arm's model on the inference tier before the experiment goes live (the experiment platform notifies the tier on arm creation); during the warm a few requests to that arm fall back to the prod model (logged as “arm-warming”); the K-resident-versions budget means evicting the least-recently-pinned version, not thrashing. If K experiments would exceed the memory budget (K×model_size > ~14 GB), the tier model-caches with LRU eviction and a warm cost on a cache miss — size K so that's rare. Detection: arm-warming counter / model-load latency; recovery: warm completes in seconds.
  • Retrain backlog / a missed cadence. A training run hangs or the offline store is slow → the hourly cadence slips → the model ages past the freshness SLA. Defence: the scheduler alarms on a missed cadence; a training job has a deadline (kill + retry); the daily full retrain is the backstop (even if mini-retrains fail, the model is at most ~24 h stale, which is the Iter-2 state — degraded but not broken). Detection: model-age SLI vs the SLA; recovery: unblock the pipeline; the next mini-retrain catches up.

Tiny / cheap to train: then “continuous training” collapses toward online learning — you'd update the model every few minutes (or per-mini-batch), the freshness NFR is trivially met, and the interesting problems shift to the feedback-loop bias and the rollback-of-a-mutating-model story (the bandit/online-learning probe becomes the base). Probe iterations (the interviewer pulls the thread — beyond the 5-Core budget)

Each in a <details> block; same NFR-driven structure, deeper bottleneck. Not rendered inline by default.

Probe A — Candidate retrieval (the layer that produces the 500 candidates — out of scope, but named)

The 80 ms budget explicitly excludes this, but a real interview goes here. Retrieval turns “the whole ~500 M-item catalog” into ~500 plausible candidates for this user, fast. The standard shapes: a two-tower model (a user-tower and an item-tower each producing an embedding; item embeddings precomputed offline into an ANN index — FAISS/ScaNN/HNSW; at request time, embed the user, ANN-search the top ~few-hundred), plus heuristic recall sources (followed accounts, recent interactions, trending in the user's segment, freshness pools) unioned and de-duplicated. Its bottleneck is hot-partition / hot-key (a few items dominate every user's candidate set — the ANN index shard holding the popular cluster, and the “trending” source, get hammered) plus memory-bandwidth (an ANN search over a multi-GB index is bandwidth-bound). The fixes rhyme with this page: shard the ANN index by embedding-space region, replicate the hot region, cache the trending set. It feeds the ranker; the ranker doesn't care how the candidates were chosen, only that there are ~500 and each has an item-feature vector in the store.

Probe B — Two-stage ranking (a cheap model over 500 → top-50, an expensive model over 50)

The standard funnel: a light ranker (a tiny MLP, or even a linear model, on a feature subset) scores all 500 candidates → keep the top ~50; an expensive ranker (the full ~20 MFLOP/row model, or a bigger one) re-scores those 50 → final order. The bottleneck this addresses: the expensive model's FLOP/s load. With one stage, the expensive model does 50 M rows/s; with the funnel it does 100k RPS × 50 = 5 M rows/s — 10× less expensive-model compute → ~10× smaller (or ~10×-bigger-model-at-same-cost) on the expensive tier, for a small accuracy hit (the light model occasionally drops a candidate the expensive one would have ranked #1 — recall@50 is the metric to watch). The light model's 50 M rows/s is cheap precisely because it's tiny (~1–2 MFLOP/row → ~1–2 L4 of compute). This is the single biggest inference-cost lever and the natural next iteration if the expensive-model bill grows; it interacts with the cost-vs-quality knob (S6) directly. Cite bottlenecks#compute-bound.

Probe C — Real-time / session features (computed in-request, not pre-materialized)

Some features depend on the current session — “items viewed in the last 30 s,” “searches this session,” “dwell on the previous item” — and can't be pre-materialized (they don't exist until the session is happening). These come in two flavours: (a) features in the request — the client/edge passes the session state in the RankRequest.context, the orchestrator turns it into the ~100 context/cross features in-process (no extra fetch — this is the cheap path, and where the ~100 context features live); (b) features needing a session store — a small, very-fresh KV (Redis, or a session table) holding “this user's last N events,” updated within milliseconds of each event by a thin session-update path, read by the orchestrator per Rank as a third multi-get (a ~few-ms add to the latency budget — affordable given the ~60 ms slack). The freshness SLA on these is “real-time-in-request” (the registry's tightest tier); they're logged with the served features like everything else (parity holds). Cite storage-engines#kv-engine.

Probe D — Cold-start (a brand-new user or item with no features)

A new user (no history → the user feature vector is mostly nulls) or a new item (just created → not yet in the item-feature store, or only the static features known) — the model trained on populated features performs badly on near-empty ones. The “no features” path: (a) on a feature-store miss for an item, serve a cold-start fallback vector — either zeros (the model learned to handle that, if cold-start examples were in the training set — they are, because new items appear in the log) or a content-based vector derived from the item's metadata (category, creator, text/image embedding — available immediately); (b) for a new user, fall back to a popularity model / a segment-default ranking (rank by global or segment-level engagement) until enough history accrues; (c) optionally a dedicated cold-start model arm. The quality bar for cold-start is explicitly lower (a flagged NFR — “cold-start behavior: a popularity fallback at a lower quality bar”); the key is it never errors — a miss is a fallback, not a 500. Cite bottlenecks#cold-start.

Probe E — Fairness / eval guardrails as a hard gate (and the online-guardrail story)

Iter 3's eval gate is sketched; the depth: the gate runs, on a held-out set, (1) overall quality (NDCG@K, or the business's calibrated objective) — must not regress > threshold vs the incumbent; (2) per-key-segment engagement (new users, low-activity users, each major locale/market, each major content category) — no segment may regress > its threshold (a model that lifts the aggregate by hurting a segment is rejected); (3) fairness metrics if there are protected attributes (demographic parity / equalized-odds-style measures on the relevant outcome) within declared bounds; (4) calibration (predicted-vs-observed engagement rates). A model failing any of these is registered status=registered — visible, inspectable, not promotable. Online, the same metrics are guardrails: an experiment arm whose per-segment engagement or error rate regresses past a guardrail is auto-disabled (its traffic returns to prod) before the experiment “completes” — you don't wait for statistical significance to stop a harmful arm. This is the difference between “we have a fairness policy” and “a model that violates it cannot ship.” Explainability (per-prediction feature attribution) is a separate, out-of-scope-here side path — added if regulation requires it (a SHAP/integrated-gradients service alongside the inference tier, sampled, not per-request).

Probe F — The bandit / online-learning alternative (exploration vs exploitation)

Batch retraining only ever learns from items it showed — it can't learn that an item it never ranked highly would have done well (a feedback-loop / selection-bias trap that compounds: the rich get richer, the long tail starves). A contextual bandit (Thompson sampling / LinUCB over a model's uncertainty) deliberately explores — occasionally ranking an uncertain item higher to gather signal — trading a little immediate engagement for long-run model quality and catalog coverage. The infra difference from the Core design: a low-latency model-update path co-located with serving (the bandit's posterior updates per interaction), careful handling of the delayed/biased reward, and a much harder rollback story (you're reverting a continuously-mutating state, not swapping a frozen artifact). The realistic shape is hybrid: the Core batch-retrained model is the base; a thin bandit/online head sits on top for exploration on a fraction of slots; the base model's eval gate + registry + drift rollback still own the bulk. We name it because every staff+ ranking interview eventually asks “how do you avoid only learning from what you showed” — the answer is exploration, and the cost is infra complexity.

  • ⑫ Eval gate → Model registry — every version recorded with metadata + status; deployments table holds the prod pointer + each experiment arm's pin.
  • ⑬ Model registry → Inference tier — on a pin the tier hasn't loaded: pull the artifact, verify content-hash + feature_schema_version, load (LRU-evict over K); promotion flips the prod pointer → reload in ~30 s.

  • ⑮ Drift monitors → Auto-rollback — a threshold crossing fires an alarm.

Degradation contract

The invariant: Rank always returns within budget with some ordering, never an error or a hang; ranking quality degrades gracefully; the model lifecycle fails safe. Concretely: feature store slow/down → the per-request deadline trips; score with whatever item vectors are in the LRU + a cold-start fallback vector for the rest (degraded quality, on-budget, a metric fires); if the user vector is unavailable, score with a near-empty user vector (cold-start path). Inference tier slow/down → the inference-RPC deadline + circuit breaker trip; return the candidates in retrieval order (or a cheap heuristic order) and mark the request “unranked.” Feature pipeline behind → serving uses the (now-staler) features that are present; a freshness alarm fires; quality decays toward the nightly-batch baseline but nothing breaks. Model is bad → the offline-eval gate stops a regression from being promoted; a canary catches one that slipped through; the drift monitors auto-roll-back to last-known-good in ~30 s; a one-command manual rollback always works; an experiment arm tripping a guardrail is auto-disabled (its traffic returns to prod). Control plane (registries / training / experiment platform) down → serving continues on the last-known-good models + the current feature definitions; no new models, no definition changes, no new experiments — routing is unaffected (fail-static, not fail-empty). A feature definition is bad → the validation gate (backfill + distribution check) should catch it pre-live; if not, the pipeline's per-feature distribution monitor + the model-eval gate + the feature_schema_version tripwire are the layered defences; recovery = roll back the definition + the model, backfill the offline store from Kafka replay, re-materialize. Cite failure-modes#slow-dependency + #bad-deploy.

Operations (C6)

  • Metrics (SLIs → NFRs). Rank p99/p999 + per-stage breakdown (feature fetch, inference RPC, assemble, sort, serialize) → the latency NFR; feature-store batched-multi-get p99 + per-shard p99 + the “served-with-fallback-vectors” counter → the feature-fetch tail; per-pod item-LRU hit rate → the cache assumption (the math assumes ~90% — verify it); inference-tier model-score p99 + GPU-queue depth + MFU → the inference tier's health and the headroom factor; ML-specific: model age per arm (now − the model's training-window end) vs the freshness SLA; training cadence — time since the last successful mini-retrain / full retrain, with an alarm on a missed cadence; offline-eval-gate outcomes (pass/fail rate, NDCG-vs-incumbent, per-segment deltas, fairness metrics — a fail is a signal, not just a block); drift dashboards — online vs offline prediction-score distribution distance, realized-vs-predicted engagement per arm (the delayed signal), with the alarm thresholds visible; feature-freshness SLIs — per-feature-group now − as_of in the online store vs each feature's declared SLA; online/offline parity checks — a periodic job re-scores a sample of logged impressions with the served features vs the offline-rebuilt features and asserts they match (parity is “by construction” — this is the tripwire that proves the construction held); K-resident-versions — how many model versions are loaded, eviction rate, arm-warming latency. Per-arm engagement / error-rate / prediction-distribution guardrails (auto-disable triggers).
  • Rollout. Three rollout surfaces. (1) A model — trained → offline-eval-gated (“deployable”) → canaried as a #40 arm at ~1–5% → online guardrails must hold → PromoteModel ramps it to prod (per-region first, then global); a one-command rollback re-points deployments to prev_model_id. (2) A feature definition — registered “proposed” → validated (backfill on a sample window + distribution check vs the previous definition) → “live” (the Flink pipeline picks it up); a breaking change bumps feature_schema_version so old models refuse it; rollback = re-mark the previous definition “live”, backfill from Kafka. (3) A binary (orchestrator / inference server / Flink job) — standard service rollout (canary a few nodes, watch the SLIs, ramp), with the relevant SLIs gating it.
  • Migration. From a Naive synchronous-Python ranker: (a) stand up ScyllaDB + dual-write the features (the cron also populates ScyllaDB) → flip the read path to ScyllaDB behind a per-region flag, watching feature-fetch p99 and the LRU hit rate; (b) stand up the Triton inference tier with the current model loaded → flip the model-call from in-pod to the tier behind a flag, watching model-score p99 (shrink the Python pods to orchestrators as the tier proves out); (c) stand up Kafka + the Flink pipeline + the registry, run it in shadow (writing the offline store, not yet trusted) → cut training over to the offline store + logged features, validating the new model's offline metrics match the old pipeline's before trusting it; (d) stand up the training cluster + model registry + drift monitors, run the hourly mini-retrain in shadow (registered, not promoted) → enable per-arm pinning + auto-rollback once the drift monitors are calibrated. Each step is flag-gated and reversible; the order is “fix the read fan-in, then the model compute, then parity, then the lifecycle” — the iteration order.
  • Headroom. Inference tier: ~2.4× raw-FLOP capacity (MFU realism + rolling deploys + K resident model versions + burst). ScyllaDB: sized for ~3× the steady read rate (a cache-cold deploy burst). Flink: scales on lag. Training cluster: sized for the daily full retrain's peak (the mini-retrains fit in the troughs). Reason: traffic bursts, rolling deploys, a new batch of experiments (more resident model versions), and a cache-cold deploy storm — each is a routine event, not an emergency.

Anticipated pushbacks

Stress tests against the derivation above — depth rehearsal, not interview rehearsal. If you can't answer in 3–6 sentences, the page is missing a probe.

What if the model drifts overnight — engagement patterns shift while everyone's asleep? (ML-flavoured: freshness/drift)

This is exactly what Iter 3 closes. Three layers catch it: (1) the hourly warm-started mini-retrain — the model is at most ~1 h behind the data, so a Tuesday-3am trending event is in the 4am model; (2) the drift monitors — the online prediction-score distribution diverging from the offline-eval distribution fires immediately (it doesn't need the feedback loop), and the realized-vs-predicted-engagement-per-arm monitor catches a quietly-wrong model within the feedback window (~hours); (3) auto-rollback — a threshold crossing reverts to last-known-good in ~30 s, so even a bad new model is self-limiting. The residual exposure: a slow drift that's within the freshness window but the mini-retrain doesn't fully correct, and that's gradual enough not to trip a threshold — the daily full retrain (cold, 30 days) is the backstop, and the dashboards (NDCG-vs-incumbent trend, per-segment trends) surface it for a human. Pre-Iter-3 (a nightly-only model, no monitors) this is a multi-hour silent engagement outage — which is why “the bottleneck shape changes daily” is the headline ML-class hazard.

Online/offline skew — the model is great in offline eval and disappoints in prod. Where did it leak? (ML-flavoured: parity)

In the Naive/Iter-0/Iter-1 design it leaks everywhere: the cron computes “clicks_last_7d” one way (a SQL GROUP BY), something else populates the online store, and what the model saw at serve time was whichever-was-in-the-store-then — none agree on cutoffs, timezones, late events, nulls — so the offline NDCG is optimistic. Iter 2 makes that structurally impossible: one registry definition per feature → one Flink computation → written to both stores; and the orchestrator logs the served features, so a training example is literally what the model saw, not a re-derivation. The remaining ways skew can still creep in (and the defences): a feature-definition change that's applied online but the offline backfill lags (→ the periodic parity-check job re-scores a sample of logged impressions with served-vs-offline-rebuilt features and alarms on mismatch); a sampling bias in which impressions get the full served-feature vector logged vs just refs (→ sample uniformly, or log refs always and the offline store is the authoritative timeline); a feature-schema-version mismatch (→ the inference-tier tripwire refuses to serve a model whose schema doesn't match the online store). The discipline is “compute once, log what you served, and verify the construction held with a tripwire.”

A bad model ships — how bad does it get before someone notices, and how do you stop it? (ML-flavoured: bad deploy)

Layered, by Iter 3: a regression should never get past the offline-eval gate (NDCG / per-segment / fairness vs incumbent — a model that fails is registered but not promotable); if it slips through (the held-out set didn't represent the failure), the canary catches it — a new prod model goes to ~1–5% of traffic first, and the online guardrails (per-arm engagement, error rate, prediction-distribution drift) gate the ramp; if it still slips through to 100%, the drift monitors auto-roll-back within the feedback window (~hours for the realized-engagement signal; immediately for the distribution-distance signal). Worst case — a subtle quality drop that's within the guardrail thresholds — degrades engagement by a few percent for a few hours until the dashboards/trends surface it for a human, who one-command-reverts. The thing that makes this not a deploy-and-pray system is that there's a quality NFR with a rollback criterion (CLAUDE's rule) — pre-Iter-3 there wasn't, so a bad model was a silent 200-OK outage with no detection path at all.

The feature store is the bottleneck — 50 M item-vector reads/sec is a lot. What if the cache hit rate is worse than ~90%? (alt-design: the feature store)

The ~90% is a Zipf(α≈1) estimate for item popularity over a top-~1 M-item cache — defensible but it's an assumption, monitored as an SLI. If it disappoints (say it's ~70%): item-vec reads to ScyllaDB go from ~5 M/s to ~15 M/s — still well within ~15–20 nodes' capacity (~1 M reads/s/node), so the immediate answer is “the headroom absorbs it.” If it's catastrophically worse (the catalog is huge and flat, no head): add a shared Redis L2 in front of ScyllaDB (a few hundred GB of the warm item set across the fleet — this is the “L2 cache” escape hatch named in Iter 0's alternatives), or bump the per-pod LRU (more RAM per orchestrator), or shard the orchestrator fleet by item-id-region so each pod's cache covers a slice of the catalog densely (a #40-style subset, applied to the cache). Beyond that — if reads genuinely can't be cached — you scale ScyllaDB horizontally (it's linear: ~1 M reads/s/node, add nodes), and the cost grows linearly with it (still ~$700/mo/node — 50 M reads/s uncached is ~50 nodes ≈ ~$35 k/mo, not a catastrophe). The DynamoDB-style alternative remains rejected on cost at any of these read rates. The structural point: the read fan-in is the headline, and the design has three independent levers against it (cache, single-flight, horizontal store) plus a horizontal-scaling fallback — it's not a single point of failure.

Why not just buy a managed feature store + a managed ML platform and skip all this assembly? (alt-engine)

You should — if you're a small team without a platform org. Tecton / SageMaker / Vertex / Databricks bundle the registry + the online/offline materialization + the training/registry/deployment machinery, get the parity discipline right, and save you most of this build. The reasons to self-assemble at this scale: (1) cost — the managed online-store tier at ~50 M item-vec reads/s is the same brutal DynamoDB-style math (Iter 0's rejected alternative); (2) control — the Flink pipeline's logic, the ScyllaDB topology, the eval-gate's segment/fairness thresholds (business-specific), and the inference tier's K-resident-versions are things you want to own; (3) you already run Kafka/Flink/ScyllaDB/Triton and a training cluster, so the marginal build is the glue, not the components. The thing that matters either way is the discipline — parity by construction, a feature registry, an offline-eval gate, a model registry, drift-gated rollback — not whether you built or bought it; a team that buys the platform and skips the discipline still has a deploy-and-pray ranker. Cite storage-engines#kv-engine for the online store choice (it's a KV-engine workload regardless of who operates it).

What changes if this is regulated — the ranker affects credit, hiring, housing, or it's in the EU/under the DSA? (regulatory)

The fairness/eval guardrails become hard (they already are NFRs here; under regulation they're legally binding — the offline-eval gate's disparity bounds are not negotiable, and an external auditor may need the eval reports), explainability moves in-scope (a per-prediction feature-attribution side path — a sampled SHAP/integrated-gradients service alongside the inference tier; for a high-stakes decision, every prediction; for a feed, sampled), data residency constrains the feature store (EU users' feature vectors stay in EU — a per-region feature-store partition, and the offline store / training cluster scoped per-jurisdiction or the model trained on de-identified/aggregated data), and auditability tightens (the impression log's “user U got ranking R from model M with features F” must be retained and queryable for the regulatory window — the log already has it; add retention + access controls). What stays the same: the feature store, the inference tier, the feature pipeline, the training/registry machinery — regulation adds gates and side paths and residency partitions, not a new bottleneck. Cite cross-cutting#privacy.

Production is slow / mis-ranking right now — where do you look first? (monitoring/debug)

Decision tree off the SLI dashboard. (1) Rank p99 elevated? — check the per-stage breakdown: feature-fetch p99 (a ScyllaDB shard slow? a compaction storm? the LRU hit rate dropped?), inference-RPC p99 (GPU-queue depth up? a node down? the microbatcher's max-delay misconfigured?), or assemble/serialize (a CPU-bound orchestrator pod?). (2) Rankings look wrong (engagement down, not latency)? — check model age per arm (is a stale model serving? did a mini-retrain fail?), the drift dashboards (online-vs-offline prediction distribution distance up? realized-vs-predicted engagement diverging?), and the offline-eval-gate history (did a recently-promoted model have a marginal NDCG-vs-incumbent? a per-segment regression?). (3) A specific segment/market is bad? — the per-segment engagement guardrail dashboard; check if a recent model or feature-definition change regressed that segment. (4) Features look wrong? — the feature-freshness SLIs (is a feature group stale → the Flink pipeline behind?), the per-feature distribution monitors (did a feature's distribution shift → a bad definition?), and the parity-check job (served-vs-offline-rebuilt mismatch → a backfill lag). (5) An experiment's numbers look off? — the per-arm dashboards + the #40 bucketing-integrity checks (is the arm getting its pinned model? is the bucketing sticky?). Each branch maps to a What-Breaks entry and a runbook step; the ML-specific instruments (model age, drift distance, feature freshness, parity check) exist precisely so the “is the model the problem” question is a dashboard read, not a forensic exercise.

S7. Key Takeaways

  • A ranker is a stateful, time-varying machine wearing the costume of a stateless RPC service — and that costume is the trap. Treat it as a web service and you ignore feature freshness, training/serving skew, model-version pinning per experiment, drift detection, and the fact that the bottleneck shape changes daily when the model retrains. The four iterations are mostly buying correctness (parity, freshness, drift rollback, experiments); the ~70× cost drop falls out as a side effect of doing it right.
  • The feature fan-in is the headline: 100 k RPS × 500 candidates × ~1k features looks like 50 B lookups/sec — until the user/item decomposition turns it into 50 M item-vector reads/sec, which a KV store built for it (sharded, page-cache-friendly) serves, and a per-pod LRU over the Zipfian item head cuts to ~5 M. “1k features per (user, item)” almost always means “1k features derivable from a user vector + an item vector + the context” — spotting that decomposition is what makes the problem tractable; not spotting it is how you end up trying to materialize 50 B vectors.
  • A cheap model with a huge fan-out is a batched-accelerator problem, not a per-request one. 50 M scorings/sec × ~20 MFLOP = 1 PFLOP/s — that's ~21 L4 of compute if the model runs batched on the GPU (the #34 weight-load-amortization trick), and ~tens of thousands of CPU pods if it runs row-by-row in Python. Move the model to a dedicated microbatched inference tier; the orchestrator pods become thin I/O glue. (And fork on model type: a DNN wants a GPU, a GBDT wants vectorized CPU and is cheaper — the choice is engagement-per-FLOP.)
  • Parity is a property you get by construction, not by reconciliation: compute each feature once (one registry definition, one streaming pipeline), write it to both the online and offline stores, and log the features you actually served so the training set is the served features. Two code paths for “the same” feature is training/serving skew by default — a model that works in eval and underperforms in prod, undetectably. A streaming pipeline buys parity and freshness (sub-day features in seconds-to-minutes vs ~24 h); a nightly batch buys only parity.
  • A model is a deploy artifact — give it the same discipline: a registry (every version + metadata + status), an offline-eval gate (a regression can't ship), continuous retraining (close the freshness gap), per-arm version pinning (ten experiments, ten models, one fleet, the request carries which), and drift-gated auto-rollback. Without a quality NFR with a rollback criterion, “ship the model” is deploy-and-pray — a bad model is a silent 200-OK engagement outage that nobody is paged for.
  • The bottleneck shape changes daily — you're operating a moving target. Tomorrow's model trained on tomorrow's features (which a pipeline change could subtly corrupt) is a different system than today's; every iteration's correctness has to survive the next retrain. That's why the feature-distribution monitors, the offline-eval gate, the schema-version tripwire, the parity-check job, and the drift monitors are layered — each catches the corruption at a different stage, and the registry + replayable event log make “roll the model AND the feature back, backfill, retrain” a recoverable operation rather than a catastrophe.
  • The cost shape of real-time ML serving is a three-way split: inference compute + feature-store reads + training compute — with the feature pipeline (the parity machinery) a fourth heavy line a stateless-web-service mindset wouldn't have budgeted at all. The cost-vs-quality knob is the funnel depth / model size: a bigger model costs linearly more inference compute and buys some engagement; a two-stage funnel cuts the expensive-model load ~10× for a small recall hit. A staff designer states the marginal-engagement-per-FLOP target and sizes the funnel to it.
