Top-K Largest Log Entries

Finding the 100 largest log entries by size from a billion entries spread across 50 machines — within a 10-minute SLA. A static, batch, exact top-K — the opposite of #15's streaming-approximate sibling.

NALSD Walkthrough • ~25 min read • LC kernel: LC #215 Kth Largest Element + its MapReduce generalization

The Scenario

LC kernel. This is LeetCode #215 — "Kth Largest Element in an Array": given an array, return the K-th largest value. The textbook answers are quickselect (partition around a pivot, recurse only into the side that holds rank K — O(n) expected, in-place, needs the whole array in RAM) or a min-heap of size K (scan once; keep the K largest seen so far with the smallest at the root — O(n log K), streaming-friendly, O(K) memory). The MapReduce generalization — the production cousin — is: each of M shards computes its local top-K independently (a per-shard min-heap), then a coordinator does a k-way merge of the M sorted-K lists into the global top-K. Three properties that break at 1 billion entries on 50 machines: (1) fits-on-one-node — 1 TB doesn't fit one machine's RAM, and moving it to one machine is 13 min over a 10 Gbps link — so the work must be M parallel passes, not one; (2) a reliable machine — a 50-node batch run has tasks fail / get reclaimed mid-scan, which the single-array algorithm has no story for — you re-run the lost shard's task; (3) no coordination — one array needs no coordinator; M shards need one to split the input and merge the M local results. What does not break: exactness. The global top-K is provably exactly the top-K of {all per-shard top-Ks} — any of the true K largest is in its shard's local top-K, so the merge loses nothing. (Contrast #15: top-K-frequent on a Zipfian stream needs a Count-Min sketch and is only approximate; here we keep exactness for free — the only cost is the merge fan-in, not error. See sketches.html — not needed here.)

You are building a log analysis tool for a large distributed system. One common diagnostic query: "Find the 100 largest log entries by size (in bytes)." Large log entries often indicate serialized stack traces, oversized payloads, or data corruption — all worth investigating.

System class

Class: an algorithmic kernel (the selection problem) wrapped in a batch data pipeline (a one-shot map → merge job over a static dataset in an object store / on log volumes) — see functional-requirements.html#class-pipeline and #class-algorithmic. It is not a queryable service in the steady state — there's no per-request QPS, no read-latency SLO; the pipeline framing puts wall-clock, throughput (bytes scanned/sec), and fault tolerance under reclamation at the centre. (A thin GET /v1/top-k API in front, S2, is the operator's submit/poll surface, not a hot path.) The access pattern is a scan-heavy full-pass batch job over read-heavy, roughly uniform data — not the power-law shape that forces #15's sketch.

Key Difference from Top-K Frequent (#15)

This problem is fundamentally different from "top-K most frequent." In the frequency problem you must count occurrences of each item — state for potentially every unique item, on a Zipfian stream — which forces a sketch (Count-Min) and an approximate answer. Here the comparison value (size in bytes) is already attached to each item; you just compare it against the current top-100. No counting, no hash maps, no sketches, and the answer is exact. The #15/#16 axis: streaming + approximate + Zipfian (#15) vs static + exact + roughly uniform (#16). #16 is also a special case of #48 distributed sort — if you fully sort the dataset you get top-K for free — but you never need a full sort for top-K; selection is O(n log K) vs sort's O(n log n), and the merge is a tiny k-way merge of M sorted-K lists, not a 100 TB shuffle.

The good news: Top-K by a pre-existing attribute (size, timestamp, score) is much simpler than top-K by frequency. You never need to store counts for all items — just maintain a small heap of the K largest seen so far.

Functional requirements (cite functional-requirements.html#fr-pipeline + #fr-algorithmic)

  • Exact top-K by size. Given the static set of ~1B log entries across the 50 machines, return the 100 largest by size_bytes — exactly, not approximately; ties broken deterministically (by (size, machine_id, file, offset)) so a re-run on the same data yields the same 100.
  • Within the SLA. The query completes end-to-end in ≤ 10 min for the full 1 TB / 1B-entry dataset.
  • Return the entries (metadata, then content on demand). The result is the 100 entries' identifiers + sizes; on request, fetch the actual entry content (up to 1 MB each) from the machine that holds it.
  • Survive a shard failure mid-run. If a machine dies or its scan task is reclaimed during the run, that shard's work is re-run; the job still completes within the SLA (with headroom for one re-run).
  • Report progress. An operator can see which shards have finished, which are running, and an ETA — enough to tell a straggler from a stuck job.

Out of scope (named, not silently dropped): streaming / continuous top-K over a live event stream — that's #15's shape (a persistent-heap variant is sketched as Iteration 2's extension and as a probe); top-K by frequency — #15; arbitrary ad-hoc analytical queries over the logs (this returns the top-K, not a queryable index — that's a log-search system, observability); multi-region — the 50 machines and the coordinator are one region; per-tenant fairness — one query, one tenant per run; full-content encryption / masking — assumed handled by the log store's SSE (named under X1).

Flagged assumptions (would confirm with the interviewer): (1) K ≈ 100 — small relative to N, so log K ≈ 7 is effectively constant; if K were ~1M (top-1M largest) the per-shard heap is ~tens of MB and the merge is 50×1M = 50M items — still fine, but the "metadata is tiny" claim weakens (see the pushbacks); (2) N ≈ 1B entries / ~1 TB — if it were 100× bigger we'd add more shards, not change the algorithm; (3) size is the attribute and it's already stored per entry — no recomputation; (4) the data is roughly uniform-to-heavy-tailed by size but not adversarially skewed across machines (every machine holds ~20M entries) — if one machine held 90% of the data its scan is the wall-clock (see the pushbacks); (5) one-shot query, possibly repeated daily — if repeated, only the shards whose data changed need re-scanning (incremental, see the pushbacks); (6) the 10-min SLA is a soft target with a hard outer bound (~15 min) — missing by 50% is a warning, by 2× is a page.

Non-functional requirements (cite nfr-checklist.html#class-pipeline-nfr)

The general checklist plus the batch-pipeline per-class additions (wall-clock / throughput, fault tolerance under reclamation, exactly-once / deterministic output, cost-per-run) were walked; the ones that bind: wall-clock (≤ 10 min for 1 TB → ~1.7 GB/s aggregate scan; the per-machine 20 GB disk read is the floor); aggregate throughput (50 machines × ~500 MB/s sequential read = ~25 GB/s — 15× over the floor, so disk-I/O is comfortable); fault tolerance under reclamation (lose any one machine mid-run → re-run its shard, still meet the SLA); correctness / determinism (exact top-K, deterministic tie-break, byte-identical result on a re-run over the same data); cost per run (minimize $/query — the knob is whether you scan full content or just an index). Out-of-scope NFRs (named): read latency / online QPS (no online surface); strong consistency / ACID (a batch read of static data); multi-region; multi-tenancy.

Per machine: ~20 GB of log data.

  No. We only need to scan entries and check their sizes.
  We can stream through the data, keeping only the 100 largest in memory.

Memory for the heap:
  100 entries x up to 1 MB each = up to 100 MB.
  In practice, much less (we only store metadata + a reference,
  not the full entry content, during the scan phase).

Scan Time per Machine

Data per machine: 20 GB
Disk read speed (sequential SSD): ~500 MB/sec
Time to read all data: 20 GB / 500 MB/sec = 40 seconds

Entries per machine: 20 million
Per-entry processing: read size, compare with heap minimum
  Min-heap operations: O(log 100) ≈ 7 comparisons (only when replacing)
  Most entries are small and skip the heap entirely.

CPU time for comparisons:
  20M entries x 1 comparison (size > heap_min?) = 20M comparisons
  At 1 billion comparisons/sec on modern CPU: 0.02 seconds. Negligible.

Bottleneck: Disk I/O (40 seconds), not CPU.

Total Time if Sequential

If we process all 50 machines one at a time:
  50 machines x 40 seconds = 2,000 seconds = 33 minutes.
  Exceeds the 10-minute SLA.

If we process all 50 machines in parallel:
  40 seconds (all machines scan simultaneously).
  Plus coordination time: merge 50 x 100 = 5,000 entries → trivial.
  Total: ~40-45 seconds. Well within SLA.
Parallelism is the key insight. The data is already distributed across 50 machines. Each machine finds its local top-100 independently. The coordinator merges the results. No data needs to move between machines during the scan phase — only the final 5,000 candidate entries at the end.

Naive Design

Collect all log entries on a single machine, sort by size, take the top 100.

Step 1: Transfer all data to one machine.
  Data to transfer: 1 TB over the network.
  At 10 Gbps network (1.25 GB/sec): 1 TB / 1.25 GB/sec = 800 seconds ≈ 13 min.
  Already exceeds the SLA, and we have not started processing yet.

Step 2: Sort 1 billion entries by size.
  Cannot fit 1 TB in memory (typical machine: 64-256 GB).
  External merge sort required.
  O(N log N) with multiple disk passes.
  Sorting 1 TB with 64 GB memory:
    Passes: ⌈log₂(1TB / 64GB)⌉ = ⌈log₂(16)⌉ = 4 passes
    Each pass reads+writes 1 TB = 2 TB I/O per pass
    Total I/O: 4 x 2 TB = 8 TB
    At 500 MB/sec: 8 TB / 500 MB/sec = 16,000 sec ≈ 4.4 hours.

Step 3: Read last 100 entries from sorted file.
  Trivial, but we already failed at step 1.

This design fails on network transfer time alone, before we even start processing.

What Breaks

  • Network transfer: Moving 1 TB to a single machine takes 13+ minutes at 10 Gbps. This alone exceeds the SLA.
  • Memory: 1 TB does not fit in a single machine's RAM. External sort is required, which adds hours of processing.
  • Wasted work: Sorting all 1 billion entries when we only need the top 100 is extraordinarily wasteful. Sorting is O(N log N); finding top-K is O(N log K) where K is tiny.
  • Single point of failure: The central machine is a bottleneck. If it fails, the entire job must restart.
The key mistake: We do not need to sort. We do not need to move data. We do not need exact ordering of all entries. We only need the 100 largest. This is a selection problem, not a sorting problem.

Iteration 1: Distributed Min-Heap

The optimal approach: each machine scans its local data and maintains a min-heap of size 100. No data leaves the machine until the very end.

The Min-Heap Algorithm

A min-heap of size K maintains the K largest items seen so far, with the smallest of those K items at the root. This is counterintuitive — why a min-heap for the largest items? Because the root tells us the threshold: if a new item is smaller than the root, it cannot possibly be in the top-K, and we skip it immediately. If it is larger, we replace the root and re-heapify.

import heapq

def find_local_top_k(log_files, k=100):
    """Find the top-K largest log entries on this machine."""
    heap = []  # min-heap: smallest of the top-K is at index 0

    for filepath in log_files:
        for entry in read_log_entries(filepath):
            size = entry.size_bytes

            if len(heap) < k:
                # Heap not full yet: add everything
                heapq.heappush(heap, (size, entry.id, entry.metadata))

            elif size > heap[0][0]:
                # New entry is larger than the smallest in our top-K.
                # Replace the smallest with this entry.
                heapq.heapreplace(heap, (size, entry.id, entry.metadata))

            # else: entry is smaller than all top-K entries. Skip.

    return heap

Notice that most entries are skipped with a single comparison (line 5, 7, 9, 10). The heap replacement operation (O(log K)) only happens when an entry is actually larger than the current threshold. For a heavy-tailed distribution (most entries are small), this happens rarely.

The Coordination Step

After all 50 machines finish their local scans:

  Each machine sends its top-100 to the coordinator.
  Data transferred: 50 machines x 100 entries x metadata (~200 bytes each)
                  = 50 x 100 x 200 = 1 MB. Trivial.

  Coordinator receives 5,000 candidate entries.
  Finds top-100 from 5,000: build a min-heap of 100, scan 5,000 entries.
  Time: 5,000 x log(100) = 35,000 operations → microseconds.

Total wall-clock time:
  Parallel scan:  ~40 seconds
  Coordination:   ~1 second (including network)
  Total:          ~41 seconds

SLA: 10 minutes. We finish in under 1 minute. 15x headroom.

Alternatives considered (C4)

  • Per-shard local top-K (min-heap) → k-way merge at the coordinator — chosen. Each machine scans its own ~20 GB once, keeping a size-100 min-heap; ships its 100 candidates; the coordinator merges 50×100 = 5,000 down to 100. No data moves during the scan; the algorithm is O(N log K) per shard with K tiny; the merge is trivial; and it's exact (any true top-100 entry is in its shard's local top-100, so the merge drops nothing). This is the MapReduce generalization of LC #215.
  • Sort everything, take the top-100 — rejected. Sorting 1B entries is O(N log N); even distributed across 50 machines it's ~80 s (2× the heap's ~40 s) and writes 20 GB of sorted output back to disk per machine — pure wasted I/O, since we discard all but 100 entries. Sorting is the right tool when you need global order (that's #48); for top-K you need selection, not order. Quantified in the "Why Not External Sort?" section below.
  • Single fat node — collect all 1 TB, run quickselect — rejected (the Naive design). Quickselect on one array is O(n) expected — faster than the heap per pass — but it needs the whole array in RAM, and getting 1 TB to one machine is 13 min over 10 Gbps (already over SLA) and doesn't fit RAM. The "you have many machines" reality is exactly the property that breaks the textbook one-array assumption: M parallel O(n/M) passes beat one O(n) pass that can't even start.
  • Quickselect on a distributed array (e.g., distributed median-of-medians / a parallel partition) — rejected. You can do a distributed select — broadcast a pivot, each shard counts how many of its elements exceed it, sum the counts, recurse — but it takes O(log N) rounds of all-to-coordinator communication and re-reads (or buffers) the data each round. For top-K with K << N it's strictly worse than "each shard finds its local top-K in one pass, coordinator merges once" — one pass, one round. (A distributed select is the right shape for "the exact 500-millionth-largest with O(1) memory per shard," not "the top 100.")
  • Quickselect within each shard instead of a min-heap — viable but rejected for this workload. If a shard's data fit in RAM you could load all 20M sizes and quickselect (~40M ops, ~3.5× fewer than the heap's ~140M). But the data streams off disk; the heap processes each entry as it arrives with O(K) memory and never materializes 20M sizes. Disk I/O (~40 s) dwarfs both CPU figures (~0.02 s vs ~0.04 s) — so the streaming-friendly heap wins on memory at zero wall-clock cost. (Detailed in Iteration 2 Optimization B.)
  • An approximate fast-path (sample each shard, estimate the top-K) — rejected: there's no reason to. Unlike #15, exactness here costs nothing — O(K) memory per shard, one pass. Sampling would introduce error to save resources we aren't short of. We keep the exact path. (If the dataset were 1000× bigger and the SLA 10× tighter, sampling a fraction of shards for an approximate answer becomes a knob — see the pushbacks.)
🔀 If the data were heavily skewed across machines (one machine holds 90% of the entries)
  • 90% of 1 TB ≈ 900 GB on one machine → ~900 GB / 500 MB/s ≈ ~1,800 s = 30 min just to read it → that machine's scan is the wall-clock and blows the 10-min SLA (the hot-partition case).
  • Fix: split that machine's files across more scan workers (re-shard by byte range, not by machine) — the merge still works because top-K-of-tops is associative; the only change is "shard" no longer means "machine."
  • This is exactly why hash-partitioning the input is fine for top-K (no global order needed, just balance) — contrast #48, which needs range-partitioning to keep the output concatenable.

What breaks (C5; cite failure-modes.html)

  • A scan machine dies or hangs mid-scan (black-hole node). Blast radius: that shard's ~20M entries are unaccounted for — missing, which would silently corrupt the (supposedly exact) result. Detection: missed heartbeats / the coordinator's progress view shows the shard stuck. Recovery this iteration: re-assign that shard's files to a spare worker and re-scan (~40 s) — the 15× SLA headroom absorbs one re-run easily. Because each shard is independent and stateless until it ships its 100 candidates, a re-scan is exactly idempotent.
  • A scan task is reclaimed mid-run (spot interruption / rolling deploy). Same as above — the work is lost, not corrupted, because nothing was committed until the candidate list shipped. Recovery: the coordinator detects the lost task (heartbeat) and re-queues it on whatever capacity is free; if it can't get capacity before the SLA, it falls back to on-demand for that one shard. (This is why the SLA has a hard outer bound: a re-run near the deadline pushes you toward it.)
  • The coordinator dies during the merge (leader loss). Blast radius: the whole job — but recovery is cheap because the coordinator's only state is "which shards have reported" + their 5,000 candidates (a few hundred KB). Re-elect a coordinator (lease); re-request candidates from any shards that already finished (their 100-lists are still in memory, or persisted) and re-queue the rest. Detection: lease expiry. The slowness flavour is coordinator-bottleneck — and the fix (don't put the coordinator on the hot path of the scan; checkpoint its tiny state) is built in: it does ~1 s of work at the end, not 40 s of scanning.
  • A corrupt or truncated log file. Blast radius: one file's entries; if the file is unparseable past byte X, the entries after X are missed (a correctness hit on an "exact" answer). Detection: a parse error / checksum mismatch the scanner surfaces. Recovery: skip-and-flag the bad region, log it; if the operator needs a guaranteed-complete answer they restore the file from a replica and re-scan. The degradation contract: a partial-but-flagged result beats a stalled job.

Degradation contract: a shard loss → re-run that shard (SLA absorbs it); a coordinator loss → re-elect + re-collect (cheap, tiny state); a corrupt file → skip-flag-continue, with a "result may be incomplete in region [file]" note — never a silently-wrong "exact" answer, never a stalled job. The only true failure is "more than ~1–2 shards lost simultaneously near the deadline" → the run misses the SLA and is retried (or scaled up).

Why This Works So Well

  • No data movement during scan. Each machine reads its own disk. The only network traffic is 1 MB of results at the end.
  • O(N) per machine, not O(N log N). We do one pass through the data with a constant-size heap. No sorting required.
  • Perfect parallelism. All 50 machines are independent during the scan. No coordination until the end.
  • Tiny memory footprint. The heap holds 100 entries. Even if each entry is 1 MB, the heap is at most 100 MB. In practice, we store metadata (file offset + size) rather than the full entry content, so the heap is a few KB.
  • Exact, by construction. The global top-100 = top-100 of {all per-shard top-100s}. Any of the true 100 largest lives in some shard, and it's in that shard's local top-100 (a shard's local top-100 contains its 100 largest), so the coordinator's merge cannot drop it. No sketch, no error term — the only cost is the 1 MB merge fan-in. (This is the property #15 can't have on a Zipfian frequency stream.)

Iteration 2: Optimizations and Streaming Extension

Optimization A: Two-Phase Scan (Metadata First)

If log entries are stored with a separate index (common in structured logging systems like Elasticsearch or cloud logging services), we can avoid reading the full entry content during the scan:

Phase 1: Scan index only.
  Index contains: (entry_id, offset, size_bytes) for each entry.
  Index size: 20M entries x 32 bytes = 640 MB per machine.
  Read speed: 640 MB / 500 MB/sec = 1.3 seconds.
  Find top-100 by size using min-heap.

Phase 2: Fetch the actual top-100 entries.
  Read 100 entries from specific offsets.
  Even at 1 MB each: 100 MB of random reads.
  At 100 MB/sec (random SSD reads): ~1 second.

Total: ~2.3 seconds per machine. 17x faster than full scan.

Optimization B: Quickselect (Alternative to Heap)

For in-memory data, Quickselect finds the Kth largest element in O(N) expected time without maintaining a heap. It partitions the array around a pivot, similar to Quicksort, but only recurses into the partition that contains the Kth element.

Quickselect vs Min-Heap:

  Min-Heap:
    Time: O(N log K) — for each of N items, potentially O(log K) heap work.
    For N = 20M, K = 100: 20M x 7 = 140M operations.
    Does NOT modify the input array.
    Works with streaming data (can process items one at a time).

  Quickselect:
    Time: O(N) expected — each element visited ~2 times on average.
    For N = 20M: ~40M operations. ~3.5x fewer than heap.
    DOES modify the input array (partitions in-place).
    Requires all data in memory at once.

For our use case (streaming from disk), the min-heap is better:
  - Data streams from disk, we don't load it all into memory.
  - Heap processes each entry as it arrives.
  - Quickselect would require loading all 20M sizes into memory first.

Extension: Continuous Streaming Top-K

If the system needs top-K continuously (not just as a one-time query), each machine maintains a persistent min-heap:

Each agent only sends a message to the coordinator when its local heap changes — which happens rarely after the initial fill. The coordinator maintains a global heap from agent updates, not from the raw event stream.

How Often Does the Local Heap Change?

After processing the first ~10,000 entries (a few seconds of data),
the heap threshold stabilizes. For the remaining 20M entries:

  Probability that a random entry exceeds the 100th-largest so far
  depends on the distribution, but for heavy-tailed log sizes:
    ~0.01% of entries trigger a heap update.
    20M x 0.0001 = ~2,000 updates per machine.
    Over the full scan: ~2,000 / 40 seconds = ~50 updates/sec per machine.
    50 machines x 50 updates/sec = 2,500 updates/sec to coordinator.
    Each update is ~200 bytes. Total: 500 KB/sec. Negligible.

Alternatives considered (C4)

  • Index scan + targeted fetch when an index exists — chosen when available. Scan only the (id, offset, size) index (~640 MB/machine → ~1.3 s) instead of the full content (~20 GB → ~40 s); fetch the actual top-100 entries by offset at the end. ~17× faster. The trade: it only helps if the log store already maintains a size index — building one just for this query would itself be a ~40 s full scan. So this is conditional on the storage design, which is why "build the index you'll query against" is a takeaway.
  • Min-heap of K vs quickselect within a shard. Heap: O(N log K) (~140M ops for N=20M, K=100), no array materialization, streams from disk, O(K) memory. Quickselect: O(N) expected (~40M ops), needs all 20M sizes in RAM at once, partitions in place. Heap wins here because the data streams off disk and disk I/O (~40 s) dominates both CPU figures (~0.02 s heap vs ~0.04 s quickselect) — the ~3.5× CPU win buys nothing while the memory cost (materialize 20M sizes) is real. Quickselect would win only if the sizes were already in memory (e.g., a pre-loaded columnar block).
  • Persistent-heap streaming top-K (the extension above) vs re-running the batch job — chosen only if "continuous top-K" is a requirement. If top-K must always reflect the latest data, each machine keeps a live size-100 heap and pushes a tiny update to the coordinator only when its heap changes (~50 updates/s/machine after warm-up; ~500 KB/s fleet-wide). This is exactly the #15 shape minus the counting. The trade: extra always-on machinery (per-machine agents, a live coordinator heap) you don't need for a one-shot query. We default to the batch job; the persistent version is a probe.
  • Hand-rolled MapReduce job vs Spark vs a columnar warehouse ORDER BY ... LIMIT K — see "Engine choices (C10)" below. For a one-shot exact top-K over 1 TB, a SQL SELECT ... ORDER BY size DESC LIMIT 100 on a columnar warehouse (BigQuery / ClickHouse / Snowflake) is often the right answer — the engine already does the per-shard top-K + merge, and a columnar scan reads only the size column. The hand-rolled job wins only when the data isn't in such a warehouse or you need exact control of the scan.
🔀 If this query is re-run daily and only ~5% of shards changed since yesterday
  • Re-scan only the changed shards; keep the unchanged shards' cached top-100 lists.
  • The coordinator merges (re-scanned top-100s) ∪ (cached top-100s) — still exact, because top-K-of-tops is associative and order-independent.
  • Wall-clock drops from ~41 s to ~2–3 s; cost drops proportionally. The only new state: a per-shard "(data version, top-100 list)" cache — tiny.

What breaks (C5; cite failure-modes.html)

  • Stale or out-of-sync index (Optimization A). If the index says an entry has size 800 KB but the content was rewritten/truncated, the heap admits a phantom. Blast radius: the top-100 may include an entry that no longer matches its indexed size. Detection: at fetch time the actual size disagrees with the indexed size. Recovery: drop the mismatched candidate, re-scan that shard's index region (or fall back to a content scan for that shard). Mitigation: trust the index only as far as its freshness SLO; for a "must-be-exact" answer, verify the final 100 against content (100 random reads ≈ 1 s).
  • Slow / dead coordinator in the streaming extension (backlog overflow). If the live coordinator falls behind the ~2,500 updates/s, its inbound queue grows. Blast radius: the live top-100 view goes stale. Detection: queue depth / lag metric. Recovery: the updates are idempotent and monotone (a heap-update message is "shard S's new 100th-largest is X") — drop duplicates, coalesce per shard (keep only the latest), and the coordinator catches up; worst case it rebuilds its global heap from the 50 shards' current heaps (~5,000 items, microseconds). Bounded-queue + coalesce is the degradation contract here, same pattern as slow-dependency.
  • A machine holding a top-100 entry dies before the content fetch. Blast radius: that entry's content is unavailable (its metadata — id, size — is already at the coordinator). Detection: the fetch-by-offset RPC fails. Recovery: serve the result with that entry marked "content unavailable, restoring from replica," and pull the content from the replica/backup; the top-100 ranking is unaffected (metadata sufficed). If the log store has no replica, the entry is reported by id+size only.

Degradation contract: index drift → verify-against-content for the final 100; coordinator lag (streaming) → bounded queue + per-shard coalesce + rebuild-from-heaps; lost content holder → metadata-only result + async restore. The ranking stays exact in every case; only content availability degrades, and visibly.

Why Not External Sort?

For completeness, let us analyze the external sort approach and see exactly where it falls apart.

Central Sort (1 Machine)

External merge sort on one machine:

  Data: 1 TB. Available memory: 64 GB.

  Step 1: Create sorted runs.
    Read 64 GB chunks, sort in memory, write to disk.
    Chunks: 1 TB / 64 GB = 16 sorted runs.
    Each run: read 64 GB (128s) + sort (seconds) + write 64 GB (128s)
    Total: 16 x 256s = 4,096 seconds ≈ 68 minutes.

  Step 2: K-way merge.
    Merge 16 sorted runs into one sorted output.
    Read all 1 TB + write 1 TB = 2 TB of I/O.
    At 500 MB/sec: 2 TB / 500 MB/sec = 4,000 seconds ≈ 67 minutes.

  Total: ~135 minutes = 2.25 hours. Far exceeds 10-minute SLA.

Distributed Sort (50 Machines)

Even distributed across 50 machines (sorting 20 GB each):

Per machine: sort 20 GB.
  Read + sort + write: ~80 seconds per machine (parallelized).
  Then extract top-100 from each sorted file: trivial.
  Merge 50 x 100 = 5,000 entries: trivial.

Total: ~80 seconds. This works, but it's 2x slower than the heap approach
(40 seconds) and does O(N log N) work instead of O(N).

The sort also writes 20 GB back to disk per machine (the sorted output),
which is wasted I/O since we only need the top 100 entries.

If the User Wants Full Entry Content

After identifying the top-100 (by metadata):

  Coordinator knows which machine holds each entry (from the metadata).
  Sends fetch requests to the relevant machines.
  Each machine reads the specific entries from disk using file offsets.

  Worst case: all 100 entries are 1 MB each = 100 MB total transfer.

  Total including content fetch: ~42 seconds.

Operations, cross-cutting & cost

Operations (C6)

  • Metrics (SLIs). Phase + per-phase elapsed vs budget (assign / scan / merge / fetch); per-shard scan progress (bytes read / total) — the straggler detector: one shard far behind the rest = a slow disk, a hot machine (too much data), or a stuck task; aggregate scan bytes/sec across the fleet; heap-update rate per shard (anomalously high = a shard with unusually many huge entries, fine; flatlined early = the threshold stabilized, expected); candidates-received / 5,000 at the coordinator; re-run count this job (high = a churning fleet); per-machine disk-free (a leak in the optional index). Each maps 1:1 to a C5 failure mode.
  • Rollout. The "deploy" is the scan-worker binary version. Because the result is deterministic, canary the new binary on one shard (or a small synthetic dataset), diff its top-K against the known-good reference byte-for-byte, then promote. A job pins its worker version at submission (restarted workers re-launch the same version) — no mid-run upgrade.
  • "A shard dies mid-job" recovery (the headline ops question). Missed heartbeat → coordinator marks the shard's task lost → re-queues its files on a spare worker (or splits them across two if the shard was a hot machine) → re-scan (~40 s; ~1.3 s if scanning an index) → merge proceeds once that shard reports. Idempotent by construction: nothing is committed until the 100-list ships, so a re-scan can't double-count. If a re-run lands near the SLA, fall back to on-demand capacity for that one shard; if ≥~2 shards are lost simultaneously near the deadline, the run misses the SLA and is retried or scaled up — the only true failure mode.
  • Migration. No persistent store to migrate; each run is fresh. Switching engines (hand-rolled ↔ Spark ↔ warehouse SQL) is a re-implementation behind the same POST /v1/top-k API — run both on the same dataset snapshot, diff the outputs (byte-identical if both honor the same tie-break), cut over once they match.
  • Headroom. +~1 spare scan worker per ~20 shards (absorb one or two reclaimed/dead machines without re-shrinking the fleet); the SLA's own ~15× slack (~41 s typical vs 10 min) is the real headroom — it buys ~14 sequential re-runs before the deadline bites; the hard outer bound (~15 min) is the page-the-operator line.

Anticipated pushbacks

Stress tests against the derivation above. If you can't answer in 3–6 sentences, the page is missing a probe — close it before declaring done.

What if the data is heavily skewed — one machine holds 90% of the entries? (scale flip)

The algorithm is unchanged but the fleet shape isn't: 90% of 1 TB = ~900 GB on one machine → ~1,800 s = 30 min just to read it → that machine's scan is the wall-clock and blows the SLA — the hot-partition case. Fix: re-shard the input by byte range, not by machine — split that machine's ~900 GB across, say, ~45 scan workers reading disjoint byte ranges (each ~20 GB → ~40 s), and the other 49 machines scan their ~2 GB each. The merge is unaffected because top-K-of-tops is associative and order-independent — "shard" just stops meaning "machine." This is exactly why top-K can hash-partition (or byte-range-partition) the input arbitrarily: you need balance, not global order — the contrast with #48, which must range-partition by key to keep the output concatenable. Detection: per-shard progress shows one shard far behind; the planner pre-checks shard sizes and re-shards before starting.

Why not just sort everything and take the top 100? (alt-algorithm)

Because sorting solves a harder problem than you asked. A full sort is O(N log N); selection (the heap) is O(N log K) with K = 100, so log K ≈ 7 — effectively O(N). Distributed, sorting 20 GB/machine is ~80 s vs the heap's ~40 s, and the sort writes ~20 GB of sorted output back to disk per machine — pure wasted I/O, since you discard all but 100 entries (quantified in "Why Not External Sort?"). Sort when you need global order (which is the entire point of #48, and which does give you top-K as a free byproduct of the first/last K of the sorted output); for top-K alone, selection wins on both work and I/O. The one case sorting is fine: a columnar warehouse's ORDER BY ... LIMIT 100 — but there the engine internally does the per-shard top-K + merge, so it's not "sort everything" in the naive sense.

What if K is huge — top-1,000,000 largest? (scale flip)

The algorithm survives but two assumptions weaken. (1) The per-shard heap is now ~1M×~32 B = ~32 MB of metadata per machine — fine in RAM, but no longer "negligible"; and log K ≈ 20, so the per-entry comparison is still cheap but the occasional heap-replace is ~3× costlier (still dwarfed by disk I/O). (2) The merge is 50×1M = 50M candidates → ~1.6 GB shipped to the coordinator (vs 1 MB) and a 50M-element k-way merge (~50M×20 ops ≈ ~1 s) — still trivial, but you'd want the merge fan-in to be hierarchical (rack-coordinators merge to ~K each, then a root merge) if M also grew. (3) If K approached N you'd just sort — the heap loses its edge once K is no longer << N (the crossover: heap's O(N log K) vs sort's O(N log N) are the same when K ≈ N). For top-1M out of 1B, K/N = 0.001 — the heap still wins comfortably. Returning 1M entries' content (up to 1 TB) is a different problem — you'd paginate the result, not stream it all.

What changes if it's a live stream, not a static dataset? (scale flip → #15)

You move to the persistent-heap shape (Iteration 2's extension): each machine keeps a live size-100 min-heap over its incoming entries and pushes a tiny update to a live coordinator only when its heap changes (~50 updates/s/machine after warm-up, ~500 KB/s fleet-wide); the coordinator maintains a global size-100 heap from those updates, not from the raw stream. The selection part is identical — you still just compare a stored attribute against a threshold — so it's much simpler than #15, which on a Zipfian frequency stream needs a Count-Min sketch and an approximate answer because exact counts are O(unique keys). What you add: failure handling for the live coordinator (bounded queue + per-shard coalesce + rebuild-from-heaps), and a decision about windowing — "top-100 ever" needs no eviction; "top-100 in the last hour" needs the heap to age out old entries (a heap doesn't support deletion well → use a sliding-window structure, which is #37's territory). Static→streaming is a real change of shape; static→static-but-bigger isn't.

A scan machine is reclaimed (spot) 30 s into its scan — now what? (failure)

Nothing was committed — the machine's partial heap is gone, but it never shipped a candidate list, so there's no double-count risk. The coordinator misses its heartbeat (3 missed beats), marks the shard's task lost, and re-queues that shard's files on a spare worker (one of the ~1-per-20 spares); the re-scan takes ~40 s (or ~1.3 s if scanning an index), and the merge proceeds once that shard reports. The SLA's ~15× slack absorbs this trivially — even ~10 sequential re-runs fit in 10 min. If a re-run lands close to the deadline, fall back to on-demand capacity for that shard. The only true failure: ≥~2 shards lost simultaneously near the deadline (or a re-scanned shard's data itself unreadable) → the run misses the SLA and is retried or scaled up. Detection: missed heartbeats + (on spot) a reclaim notice; runbook: "re-queue the lost shard, watch the ETA, fall back to on-demand if it'll miss." Cite failure-modes#bad-deploy (a worker reclaimed mid-run) and #black-hole-node (a hung scan).

Where does the cost actually come from? (cost)

For a one-shot query, essentially nowhere — pennies, however you run it: ~$0.06 hand-rolled (or ~$0.002 scanning an index), ~$0.04 as a BigQuery LIMIT (it scans only the ~8 GB size column, not 1 TB), ~$0.15 on a transient Spark cluster (where ~1–2 min of cluster startup dominates a ~40 s job). The cost shape only bites if the query is continuous or the dataset is 100× bigger — then it's "bytes scanned × rate," and the levers are (a) read a column, not the row (columnar projection: 8 GB not 1 TB), and (b) read an index, not the content (~640 MB/machine not ~20 GB). "Buy more machines" is a non-lever — you're already 15× under the SLA; nobody's waiting. Cite cost-shapes#telemetry (scan-bytes-dominated) and pricing-constants#compute for the rates.

How do you make this incremental — re-run only the changed shards? (migration / efficiency)

Cache, per shard, the tuple (data_version, local_top_100) after each run. On the next run, the coordinator re-scans only the shards whose data_version changed; for the rest it reuses the cached local_top_100. The merge is merge(re-scanned top-100s ∪ cached top-100s) — still exact, because top-K-of-per-shard-tops is associative and order-independent (a cached top-100 from yesterday is just as valid an input to the merge as a fresh one, as long as that shard's data didn't change). If ~5% of shards changed, wall-clock drops from ~41 s to ~2–3 s and cost proportionally. The only new state is that tiny per-shard cache (~3 KB each). This is the same trick a columnar warehouse does for free via partition pruning — only re-scan partitions whose data changed since the last query's snapshot.

Why a min-heap and not quickselect, since quickselect is O(n)? (alt-algorithm / why-this-number)

Quickselect is O(n) expected and ~3.5× fewer ops than the heap's O(n log K) — but it needs the whole array of sizes in RAM at once (it partitions in place), which means materializing 20M sizes per shard. The heap streams: it consumes each entry as it's read off disk, holds only K = 100 tuples, and skips >~99.99% of entries with a single comparison against the root. Disk I/O is ~40 s; the heap's CPU is ~0.02 s and quickselect's ~0.04 s — the CPU difference is invisible next to the I/O, while the memory difference (3 KB vs materializing 20M×8 B = 160 MB) is real. So the heap wins on memory at zero wall-clock cost. Quickselect would win only if the sizes were already in memory — e.g., a columnar engine that's already read the size column into a block; that engine does exactly that internally. (LC #215 accepts either answer; the system context picks the streaming one.)

Key Takeaways

  1. Top-K by attribute is a selection problem, not a sorting problem. You do not need to sort N items to find the K largest. A single-pass min-heap reduces the work from O(N log N) to O(N log K), and since K is small (100), log K ≈ 7 is essentially constant.
  2. Use the parallelism you already have. Data is already distributed across 50 machines. Process locally, merge globally. The merge step handles only 5,000 items regardless of the input size.
  3. Minimize data movement. The naive approach moves 1 TB across the network. The distributed heap approach moves 1 MB. That is a 1,000,000x reduction in network traffic.
  4. Know your bottleneck. For this problem, the bottleneck is disk I/O (40 seconds to read 20 GB), not CPU (0.02 seconds for comparisons). Optimizing the algorithm further does not help — you need faster disks or an index.
  5. Indexes change the game. If the log system has a size index, the problem goes from 40 seconds to 3 seconds per machine. When designing log storage, consider what queries you will need and build indexes proactively.
  6. The min-heap is the universal tool for streaming top-K. It works for top-K largest, top-K most recent, top-K highest score — any problem where you compare items by a single attribute. Memory is O(K), time is O(N log K), and it processes items one at a time (streaming-friendly).
  7. Top-K-of-per-shard-tops is exact, for free. Unlike #15 (top-K frequent on a Zipfian stream — needs a Count-Min sketch, only approximate), top-K by a stored attribute stays exact with O(K)-per-shard memory: any true top-K element is in its shard's local top-K, so the merge drops nothing. The only cost is the merge fan-in (~1 MB), not error. Know which axis your problem is on — static/exact/uniform vs streaming/approximate/Zipfian.
  8. Hash- (or byte-range-) partition the input arbitrarily — you need balance, not order. Because top-K doesn't produce a global ordering, you can split the input any way that balances the shards. Contrast #48, which must range-partition by key to keep its output concatenable. If one machine holds most of the data, re-shard by byte range — the associative merge doesn't care.
  9. The right "engine" is often a SQL LIMIT on a columnar warehouse. BigQuery / ClickHouse / Snowflake's ORDER BY size DESC LIMIT 100 is the MapReduce top-K (per-shard bounded queue + k-way merge), and a columnar scan reads only the one attribute column — seconds, pennies, zero ops. Hand-roll the distributed heap only when the data is stuck on the source volumes or you need exact control of the scan (e.g., scanning a size index instead of content).

Scenario: Top-K Most Frequent (#15)

The streaming / approximate / Zipfian sibling — top-K by frequency, which needs a Count-Min sketch and gives up exactness. The #15/#16 contrast (static-exact-uniform vs streaming-approximate-Zipfian) is the central pedagogy of both pages.

Scenario: Distributed Sort (#48)

The more general MapReduce-shaped batch sibling. A full sort gives you top-K for free — but you never need a full sort for top-K; selection is O(n log K), and you can hash-partition (no range-partition needed, because there's no global order to preserve).

Scenario: Real-Time Leaderboard (#24)

The "per-entity score, get the rank" cousin — a Redis sorted set instead of a per-shard heap; top-N is a ZREVRANGE, the inverse query (a player's rank) is a ZREVRANK.

Sharding strategies

Hash vs range — the decision here: hash- (or byte-range-) partition the input arbitrarily because top-K needs balance, not global order; range-partition only when the output must be ordered (#48).

Canonical bottlenecks

disk-io-bound (the per-machine scan), coordinator-bottleneck (keep the coordinator off the per-byte path), hot-partition (a machine holding most of the data), network-bandwidth-bound (the merge fan-in — tiny here).
