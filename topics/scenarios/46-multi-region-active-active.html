Designing a Multi-Region Active-Active E-Commerce Service

User profile + cart + order ledger + inventory for 100 M users across 3 regions, with one load-bearing constraint that nothing else lets you dodge: every dataset has a different CAP requirement — profile/cart tolerate ~5 s staleness (AP), the order ledger must never double-charge (CP), the inventory count must be CP for the last unit but AP for the common case — on top of p95 in-region read ≤ 100 ms, EU-data-stays-in-EU residency, and RPO ≤ 5 s / RTO ≤ 30 s. We start from a single us-east primary with read replicas — which fails the read SLO for 60% of users, fails residency outright, and turns a region outage into a multi-hour total outage — and iterate to read-local-everywhere, a per-dataset CAP audit that gives each dataset its own topology + conflict-resolution, residency-aware geo-partitioning with an explicit per-dataset partition-behaviour rule, and a CP global control plane that does fast home-region reassignment on a region loss.

NALSD Walkthrough • ~45 min read • No LC kernel — constraint-driven design

No LC kernel. This is the gap the 15 LeetCode problems don't reach: there is no single data-structure trick to upgrade. It is a constraint-driven design — the spine is the per-dataset CAP audit in Iteration 1: you do not make one global consistency choice; you classify each dataset (profile, cart, order, inventory, session) and pick its replication topology + consistency model + conflict-resolution separately. Each iteration closes one constraint: Iter 0 — the read SLO (cross-region RTT); Iter 1 — write latency & the CAP-per-dataset audit; Iter 2 — EU residency & explicit partition behaviour; Iter 3 — RPO/RTO via fast home-region reassignment. See each iteration for the constraint (RTT, residency, RPO/RTO, partition behaviour) that drove the decision.

S0. Scenario & Requirements

An e-commerce platform — think Amazon retail, Shopify storefronts, or Zalando — serving 100 M users from 3 regions (us-east, eu-west, ap-southeast), active-active: every region serves reads and writes for the users near it. The user-facing surface is four datasets that happen to span every CAP point — profile (name, addresses, prefs; read-heavy, occasional edits), cart (per-user item list; edited from the user's current device), order ledger (append-only record of placed orders + payments; must never double-charge), inventory count (units-in-stock per SKU; the genuinely hard one — needs CP-ish for the last unit but AP for the common case) — plus a session/auth-token store. The tension that shapes the whole page: a junior answer says "multi-region active-active with strong consistency"; that forces the worst latency on every dataset and is illegal for EU data. The staff answer is a per-dataset CAP audit: classify each dataset, give each its own topology, conflict-resolution, and partition behaviour.

System class

Class: product — user-facing capabilities (browse, edit cart, place orders), see functional-requirements.html. The load-bearing NFRs are in-region read latency, per-dataset consistency, regulatory residency, and the DR pair (RPO/RTO) — the things a single-region product never has to confront.

Functional requirements (cite functional-requirements.html)

  • Sign up / sign in. Create an account with a residency tag (EU / US / APAC, set at signup from billing address); authenticate; maintain a session.
  • View & edit profile. Read addresses/prefs (the hot read path); edit them occasionally.
  • Manage a cart. Add/remove items; the cart is per-user, edited from whichever device the user is on; it must survive a device switch and a brief region failover without losing an added item.
  • Place an order. Checkout: reserve the inventory, write an order row, charge the payment, return a confirmation — exactly once, never double-charged, never showing two orders for one checkout.
  • See stock. Show "in stock / N left / out of stock" on the product page; decrement on checkout; the last few units must not oversell.

Out of scope (named, not silently dropped): the product catalog (treated as a read-only globally-replicated tier — non-personal, not the interesting part); payment-processor integration internals (we call a PSP, idempotency-keyed); search/recommendations (separate system); the warehouse/fulfilment side; CDN/static assets; analytics warehouse (a downstream consumer of the order stream).

Flagged assumptions (would confirm with the interviewer): (1) "active-active" means every region takes writes for its own users, not that any region can take any user's write — each user has a home region; (2) residency is a hard legal constraint for personal data (GDPR), not a soft preference; (3) ≤ 5 s bounded staleness is acceptable for profile/cart reads from a non-home region; (4) the catalog and the inventory global-pool metadata are non-personal and may be replicated everywhere; (5) 3 regions today, but the design must absorb a 4th without a maintenance window.

The fundamental tension: the speed of light is fixed — a synchronous cross-region quorum costs ~80–200 ms of RTT per write — but most of our datasets can't afford that and most don't need it; so the design's whole job is to spend the strong-consistency budget (one cross-region round trip per write) only on the datasets that require it (the order ledger, the inventory last-unit, the region-home catalog) and nowhere else.

ID scheme (cite identifier-schemes.html#uuidv7): region-prefixed UUIDv7 for users and orders — the region prefix (us_ / eu_ / ap_) guarantees an id minted in one region can never collide with one from another (no coordination needed), and UUIDv7's time component keeps it roughly sortable for index locality. Rejected: bare UUIDv4 (no time ordering → index fragmentation); Snowflake (needs a coordinated worker-id allocator — an extra cross-region dependency we don't want); plain auto-increment (impossible without a single sequence owner — the thing we're trying to avoid).

S2. API Contracts (cite api-design-patterns.html)

Eight endpoints. The cross-region-specific shape: every endpoint is served by the region nearest the caller; writes to CP datasets (order, inventory commit) may be forwarded to the user's home region; checkout is idempotency-keyed so a retry across a failover doesn't double-charge.

POST /v1/session           {email, credentials}            -> 201 {token}            (read-your-writes in-region)

Error model: cite api-design-patterns.html#errors (problem+json). Versioning: URL prefix /v1/.
Idempotency on POST /v1/orders: cite api-design-patterns.html#idempotency — key stored with the order in the home region.
Concurrency on PATCH /v1/profile: last-writer-wins by (updated_at, region) — NOT If-Match (cite #concurrency; we accept LWW here).

Per-endpoint SLO

  • GET /v1/profile, GET /v1/cart, GET /v1/.../stock — p95 ≤ 100 ms (in-region replica read).
  • PATCH /v1/profile, PUT /v1/cart/items — p95 ≤ 100 ms (local write, async replication off the hot path).
  • POST /v1/orders — p95 ≤ 300 ms when the user is in their home region; p99 ≤ 500 ms (includes inventory reservation + payment call); +1 cross-region RTT (~80–200 ms) if forwarded to a non-local home region or if the order needs the last-unit CP path.
  • POST /v1/session — p95 ≤ 150 ms (CP token write in home region).

S3. Back-of-Envelope — the three numbers that govern every iteration

No optimization techniques here — just the ratios. (Forward-references named, not used: "we'll fix this with read-local replicas in Iter 0, the CAP audit in Iter 1".)

1. Cross-region RTT eats the read budget

Budget: p95 read ≤ 100 ms, measured at the serving region.
In-region: RTT_intra (1 ms) + DB_read (1–10 ms) ≈ 2–11 ms  -> 10× under budget ✓
Single-region (data in us-east only):
  EU user read  = RTT_us_eu (80 ms ×2 round trip if a request/response pair) ≈ 80–160 ms + DB 10 ms  -> OVER ✗
  APAC user read = RTT_us_ap (150 ms) ≈ 150–300 ms + DB                       -> WAY OVER ✗
60% of users (EMEA + APAC) fail the SLO on RTT alone. -> data must be in-region. (Iter 0.)

2. The replication stream is small; the egress price is the only thing to manage

Change-data: 30 k writes/s × 1 KB = 30 MB/s global.
Each region replicates to up to 2 followers: ≈ 30 MB/s × 2 ≈ 60 MB/s of cross-region traffic (pre-residency-sharding).
Monthly: 60 MB/s × 86400 × 30 ≈ 155 TB/mo × $0.02/GB ≈ $3.1 k/mo.   Tiny next to the 3× compute bill.
But: if we replicated *snapshots* not deltas, or didn't compress, this 10×s. And residency-sharding (Iter 2)
cuts it further — EU deltas never cross to US/AP. So: replicate deltas, compress, residency-shard.

3. RPO/RTO is a budget, not a wish

RPO ≤ 5 s  ⇒ async replication lag of the slowest follower must stay < 5 s  ⇒ monitor it; lag > 5 s = page.
            CP datasets: RPO = 0 within the home region; but the home region can be LOST → that data is the
            replicated copy in the follower, so RPO there = the same async lag. (Iter 3.)
RTO ≤ 30 s ⇒ detect (~5–10 s health checks) + decide (~1–5 s quorum) + reassign home regions (~1–10 s, small catalog)
            ≈ 10–25 s ✓ — IF the home-region map is small and CP. (Iter 3 builds that control plane.)

S4. Naive Architecture — the simplest thing that meets the FRs

One region, us-east. A single primary database (Postgres-class) plus a couple of in-region read replicas; an app tier in front; a global anycast/GeoDNS that just points everyone at us-east. It satisfies all the functional requirements — you can sign up, edit a cart, place an order, see stock — with the simplest possible consistency story (one primary, strongly consistent, no conflicts ever). It is also the absolute worst case for every NFR, which is exactly why we start here.

Why it's broken (per failing NFR)

  • In-region read latency ✗ — 60% of users (EMEA+APAC) are over the 100 ms budget on cross-ocean RTT alone, before any DB work.
  • Residency ✗ — EU personal data sits in us-east. Not a performance problem; a legal one. No amount of caching fixes it.
  • RPO ✗ / RTO ✗ — the region is a single point of failure. RPO = backup lag (hours); RTO = restore-from-backup time (hours-to-a-day). A region outage = a total outage.
  • Consistency ✓ — trivially: one primary, strongly consistent, zero conflicts. (Iterations will weaken this for AP datasets — on purpose, per the audit — and we'll re-check it each time.)

Shape-determining failure: it is single-region. There is no in-region patch — the only fix is to put serving and data in multiple regions, which immediately forces the question this whole page answers: what happens to a write in region A vs region B? That's Iteration 0.

Alternatives considered (C4)

  • Read replica in every region + single global write primary — chosen. Smallest step that fixes reads everywhere with zero new consistency complexity (still one primary → still linearizable, no conflicts). Cite replication-decisions#single-async.
  • Edge CDN cache of reads — rejected (insufficient). Helps the catalog (already out of scope as a separate tier) but profile/cart/stock are per-user and write-through — CDN TTLs can't bound staleness to 5 s without near-zero TTL, which defeats the cache. (🔀 if reads were public & cacheable, this would be the move.)
  • Jump straight to multi-leader active-active — rejected (premature). That's the right destination but it lands conflict-resolution, residency, and the home-region concept all at once with no contrast. We get there over Iters 1–3.

The chosen move

Stand up 3 regions. Each region runs the app tier plus a full read replica of every dataset, fed by asynchronous single-leader replication from the us-east primary (cite replication-decisions#single-async; consistency-model background in 06-strong-vs-eventual, database-replication.html). Reads are served from the caller's region → in-region latency. Writes are forwarded to us-east → still one writer, still linearizable, still zero conflicts. This resolves the read bottleneck because the read path no longer crosses an ocean; it deliberately does not touch the write path, residency, or DR — those are the next three iterations.

What breaks (C5; cite failure-modes.html)

  • Leader / lease loss. us-east primary dies. Blast: all writes globally stop. Detection: write errors + replication-lag-to-infinity alarm. Recovery: promote a replica — manual or semi-automated, minutes; RTO ✗. (This is exactly what Iter 3 fixes.)
  • Replica divergence. A follower falls > 5 s behind (network blip, vacuum storm). Blast: that region serves stale reads beyond budget; RPO at risk. Detection: per-follower lag SLI. Recovery: shed reads from the lagging replica to another region until it catches up.
  • Cascading timeout. A cross-region forwarded write times out (transient WAN congestion); the app retries, amplifying load on the already-stressed link. Detection: forwarded-write p99 + retry rate. Recovery: idempotency key (already on POST /v1/orders) + bounded retries + circuit-breaker on the forward path.

Schema/API delta

No schema change. API: GET endpoints now served by the nearest region's replica (router/DNS change); mutating endpoints forward to us-east transparently. The Idempotency-Key on POST /v1/orders (cite api-design-patterns#idempotency) is now load-bearing — a forwarded write that times out must be retryable without double-charging.

Concrete tech choices.

  • AP datasets (profile, cart) → DynamoDB Global Tables. Multi-leader async with built-in LWW; cart's OR-Set lives in the item value (we do the merge in the app on read, or use a CRDT library). Rejected: Cassandra multi-DC (LWW only at the cell level — no native OR-Set, and tunable-consistency footguns); CockroachDB (it's CP-by-default — we'd be paying for consensus we explicitly don't want on these datasets); Cosmos DB (good multi-region consistency knobs, but ties us to Azure and its residency story is harder to reason about than DynamoDB's explicit per-table region list). DynamoDB wins on: native multi-leader, explicit per-table replica-region control (matters for Iter 2's residency), and the lowest ops burden.
  • Order ledger → single-region-owner relational store (Aurora-class per region, ledger semantics). Each region owns its users' orders with strong consistency; cross-region replicas are async. Rejected: Calvin-style deterministic DB (great theory, but a global deterministic log adds cross-region latency to every order — we don't need global ordering, only per-order); a global Raft group per shard (overkill: cross-region consensus on every checkout when the user's writes are naturally single-region).
  • Cart's OR-Set → Redis CRDT (or an app-level OR-Set on top of the DynamoDB item). Redis Enterprise's Active-Active CRDTs implement OR-Sets natively; Riak's Datatypes do too. We pick app-level merge on the DynamoDB item so we don't run a second multi-region datastore — the OR-Set logic is ~30 lines and the cart is small.
  • Inventory CP pool → a small global Raft group (etcd-style) holding per-SKU counts, or the relational ledger store with a single owner per SKU. The pool is small (one row per SKU), updated rarely (top-ups, not per-checkout), so global consensus is affordable here.

What breaks (C5; cite failure-modes.html)

Bottleneck

Iter 1 still leaks EU personal data: the DynamoDB Global Table replicates EU profile/cart to us-east and ap-southeast; Orders[EU] replicas live in US/AP. That's a residency violation — not a latency bottleneck but a hard-constraint failure (GDPR). Cite bottlenecks#cross-region (the consistency/placement axis) and failure-modes#network-partition.

Architectural decisions to revisit

Per bottleneck-to-decisions#cross-region: residency policy (the one we deferred from Iter 1) and the partition-tolerance behaviour implied by each dataset's CAP choice. Per cross-cutting#privacy: data residency is a placement constraint on the partition key. We add the residency dimension to the shard key and write the partition-behaviour rule for each dataset (the CAP audit's "P" column, finally explicit).

Alternatives considered (C4)

  • Geo-shard by residency tag; one pair of regions per jurisdiction; a global tier for non-personal data — chosen. Cite sharding-strategies#geo + #composite (the key becomes (residency, user_id)). EU data physically can't land outside the EU because no EU partition's replica set includes a non-EU region. Cleanest, auditable, and it cuts cross-region egress (EU deltas never cross to US/AP).
  • Encrypt EU data with EU-only keys, replicate the ciphertext everywhere — rejected. Tempting (one global topology, "it's just ciphertext") but most regulators treat encrypted personal data still located abroad as a transfer, and key-access patterns leak metadata; legal risk too high. (🔀 if the regulator explicitly blessed "encrypted-at-rest-with-local-keys = compliant", this would be simpler — some do for some data classes.)
  • One global multi-leader table with a per-row "allowed regions" filter on the replication stream — rejected (fragile). A misconfigured filter = a fine. Geo-sharding makes the boundary structural, not a config knob.

The chosen move

(a) Residency geo-sharding. The partition key for every personal entity is (residency, user_id). Each residency value maps to a fixed set of regions: EU → {eu-west, eu-central}; US → {us-east, us-west}; APAC → {ap-southeast, ap-northeast}. (The second region per shard gives in-region durability + a failover target without leaving the jurisdiction — and gives the AP datasets a quorum and the CP datasets a place to put the home-region replica.) Replication for personal data happens only within a shard. A separate global tier holds non-personal data — the product catalog and the InventoryPool metadata — replicated to all regions freely. Iteration 3 — Fast failover: home-region reassignment via a CP global control plane

System after this iteration

Dashed gray = health checks & in-shard async replication; dashed red = failover command (fence + reassign). Connections:

  • ① Every region → control plane — liveness heartbeats; the control plane forms a quorum view of which regions are up.
  • ② Control plane → failover region (in the same shard) — on a confirmed region death: bump the fencing token, reassign the home-region map entries, tell the failover region to take over.
  • ③ home region → failover region (async, in-shard) — the steady-state replication that keeps RPO < 5 s; on takeover the failover region replays its last applied state and resumes.

Delta vs Iter 2: added a small CP global control plane (home-region map + fencing + health-check quorum + the small strongly-consistent catalog: auth-customer index, billing-customer, residency table); the failover region per shard now has a defined takeover procedure; region death is detected and acted on automatically within the RTO budget.

Bottleneck

Iter 2 still fails RTO on a region loss (as opposed to a partition): if eu-west is destroyed, every EU order and session homed there is orphaned — eu-central has the data (≤ 5 s behind, so RPO ✓) but nothing tells it "you're the home region now", and nothing fences the dead region against a zombie return. Classified as leader / lease loss at region granularity, and the missing piece is a coordinator — we deliberately have no global coordinator yet, and now we need a small one. RTO target 30 s; current RTO = however long manual operator action takes (minutes). Gap: ~10×+.

Architectural decisions to revisit

Per bottleneck-to-decisions#cross-region & the DR pattern in cross-cutting#dr: who owns the home-region map, how a failover is decided (quorum, not a single node), how the dead region is fenced (so it can't come back and double-write — the classic split-brain hazard, cite failure-modes#network-partition), and where the genuinely-global small state (auth index, billing) lives. This iteration introduces the one thing we make strongly consistent across all regions: the control plane.

Alternatives considered (C4)

  • A small CP global control plane (5-region Raft / etcd-style, or the cloud's global config service) holding the home-region map + fencing token — chosen. The map is tiny (one entry per user-or-shard, not per row), changes rarely (only on region events), so global consensus is cheap. Cite replication-decisions#single-sync (synchronous quorum) and cross-cutting#dr.
  • Spanner / TrueTime as the control plane — viable alternative. Spanner gives globally-consistent transactions with bounded clock uncertainty — a clean way to hold the home-region map and even the order ledger. We don't use it as the default because (a) cloud lock-in, (b) we only need consensus on a tiny config blob, not full SQL semantics, (c) the order ledger is fine single-region-owned. But "put the control plane on Spanner" is a perfectly defensible choice; we'd quote the +RTT-per-transaction cost. (🔀 if we were already all-in on GCP and wanted the order ledger globally consistent too, Spanner would carry both.)
  • No control plane — gossip the home-region map between regions — rejected. Gossip is eventually consistent; during the window where regions disagree on "who's the home for entity X", two regions could both think they're it → split-brain on the CP datasets. The whole point of the CP datasets is to never do that. The control plane has to be CP.
  • Self-run 5-region Raft vs the cloud's global config service — a wash. Self-run = no lock-in, more ops; managed = less ops, lock-in. Pick the managed one unless multi-cloud is a hard requirement; either way the design is identical.

The chosen move

Stand up a small CP global control plane — a 5-region Raft group (etcd-style) or a managed global config service — that holds: the home_region map (per user, or per shard-slice; small), a monotonic fencing-token sequence, the health-check quorum view, and the small genuinely-global tables (auth-customer index, billing-customer, the residency → [regions] table, the InventoryPool per SKU). It is the only thing we make strongly consistent across all regions — because it's the coordination substrate, and it's small enough that global consensus is affordable. Failover procedure: (1) health checkers in every region heartbeat the control plane; a quorum decides "eu-west is dead" (not just slow — a partition where it's alive on the other side is handled by the partition rules from Iter 2, and the fencing token makes a zombie return safe either way); (2) the control plane bumps the fencing token and reassigns eu-west's homed entities to eu-central (a small map update); (3) eu-central replays its last replicated state (it's ≤ 5 s behind → RPO ✓), starts serving as the new home; (4) inventory: eu-west's allocation is returned to the CP pool and re-distributed. A dead region that later comes back gets a stale fencing token and is rejected until it re-syncs as a follower — no double-write. This resolves the RTO bottleneck because the slow part (operator paging + manual promotion) is replaced by a quorum decision + a tiny map update, both seconds.

What breaks (C5; cite failure-modes.html)

Probe iterations (the interviewer pulls the thread — beyond the 5-Core budget)

Probe A — A user travels (a US user flies to the EU)

Their home_region stays us-east — their personal data is US-residency-tagged and physically in the US shard. From eu-west: reads of their profile/cart go to the US shard cross-region (~80 ms — over the 100 ms in-region budget, but "in-region" SLO is defined for the user's home region; a traveler accepting one cross-Atlantic hop is the right trade); writes (cart edit) are forwarded to the US shard (~95 ms); checkout is forwarded to us-east (the order's home), idempotency-keyed. Alternative: migrate the user's home region — a rare, coordinated, CP operation: copy their data to the EU shard, update the HomeRegionMap under a fencing-token bump, point them at eu-west. Don't do this on a single trip; do it if they relocate (re-tag residency, which itself may have legal implications — their data now must move to the EU). Caveat: a US user whose data is US-tagged but who lives in the EU is a residency question, not just a latency one — flag it to legal.

Probe B — Onboarding a 4th region (e.g., sa-east) with no maintenance window

New region = new (or extended) residency shard. (1) Decide which residency it serves — new "SA" residency for new users, plus optionally migrate existing SA-located users (the home-region-migration procedure from Probe A, batched). (2) Stand up the region's stores empty; start backfilling from the relevant shard's existing region(s) via a snapshot + catch-up-on-the-replication-log (the standard add-a-replica procedure, cite database-replication.html) — no downtime, the new region just isn't serving yet. (3) When caught up (lag < 5 s), add it to the shard's region set in the control plane, start routing SA users to it. (4) Add it to the inventory allocation re-balancer's region list. The control plane's home-region map and the CP Raft group don't need re-sizing for a data-plane region — only if you also want it to vote on failovers (then it's a Raft membership change, online).

Probe C — Hard mode: "exactly N units, never oversell, ever" for inventory

If the business cannot tolerate any overselling even transiently (e.g., limited-edition drops, ticketing), the reservation pattern's "AP local decrement" is no longer acceptable — a partition could let two regions each sell their last allocated unit when the true total was one. Then inventory becomes fully CP: a single global Raft group (or a single-region owner) for the count, and every checkout does a cross-region CP round trip (~80–200 ms p99 added to checkout). Throughput ceiling: a single Raft group commits maybe ~10–50 k ops/s — fine for one hot SKU, a problem if you do it for the whole catalog (so: only the SKUs that need it; the rest stay on the reservation pattern). This is the cost-vs-correctness knob made concrete: you can have local-fast inventory or never-oversell-the-last-unit-under-partition, not both, for free.

Probe D — Cross-region read-your-writes

A user writes their profile in eu-west, then a request lands in eu-central (load-balancer reshuffle) before replication catches up — they could read their old profile. Fixes, in order of cost: (1) sticky routing — pin a user to one region for a few seconds after a write (a cookie / session affinity), so reads land where the write went; cheap, covers ~all cases (a user rarely flips regions sub-second). (2) read-from-home on a "fresh" flag — the client sets a header on the next read after a write; the server reads from the home/leader replica for that one read. (3) monotonic read tokens — the write returns a version vector; the read passes it; the replica waits until it's caught up to that version. We default to (1) + (2) — (3) is the heavyweight option for the rare client that needs a hard guarantee. Note this is intra-shard (same residency); a traveling user (Probe A) reading cross-shard is already accepting staleness by being outside their home region.

Probe E — The cost of "strong everywhere" (why we did the audit)

If we'd ignored the audit and made every dataset CP-cross-region: every cart edit, every profile change becomes a cross-region quorum write → +~100 ms p99 on writes that today take 16 ms (a ~6× regression on the most common write); the replication topology becomes "quorum of N regions" instead of "async to the in-shard peer", multiplying cross-region egress (a quorum round-trip per write vs one async delta); and a single slow region drags down every write globally. The audit is the act of not paying that — we pay it on the order ledger (one CP write per checkout, in the home region, ~16 ms because the home region is local + the rare cross-region case), the inventory last-unit (rare), and the control plane (a tiny config blob, region-events only). Everything else stays AP-and-fast. That's the whole page in one paragraph.

Degradation contract

When the control plane is unreachable: every region keeps serving on its last-known home-region map — reads, AP writes (profile/cart), and CP writes to entities homed in a reachable region all continue; only failover decisions and home-region migrations pause. A control-plane outage must never stop in-region serving (cite failure-modes#network-partition). When a region is lost: AP datasets keep serving everywhere (the data is in the failover region); CP datasets homed in the dead region return 503 retry until the control plane reassigns them (~15 s). When a network partition splits two regions: per-dataset rules from Iter 2 — AP stays available on both sides (reconcile on heal), CP serves only the home/quorum side, the other returns read-only/retry for those entities. When the CP inventory pool is unreachable: regions keep selling from their local allocations until exhausted, then checkouts for those SKUs fail (better than overselling).

Operations (C6)

  • Metrics (SLIs mapped to the NFRs). Per-region read p95 (→ read-latency NFR); per-dataset write p95 (→ write-latency); per-shard, per-replica replication lag (→ RPO — the paging metric, alert at 3 s); home-region reassignment time on drills (→ RTO); CRDT conflict-resolution rate (→ consistency health — a spike means a partition or a clock issue); residency audit: count of personal rows whose location ≠ their residency mapping (must be 0 — alert on any non-zero); control-plane Raft commit latency + quorum health; checkout 503-rate (→ partition / pool-exhaustion signal).
  • Rollout. Canary the control-plane config to one region first (it validates the map includes itself); ramp app changes per region (one shard at a time, failover region before home region); rollback trigger: read p95 > 80 ms or 503-rate > 0.1% in any region.
  • Migration (from current single-region prod). (1) Stand up the read replicas in eu/ap (Iter 0) — read-only, no behaviour change. (2) Per dataset, flip to its target topology behind a flag, dual-writing (old primary + new topology) and comparing, then cut over (Iter 1). (3) Backfill EU/APAC users into their residency shards; once a user's data is in their shard, flip their routing; then stop replicating their data to the old region and delete it there (Iter 2). (4) Stand up the control plane, populate the home-region map, enable automatic failover (Iter 3). Each step is reversible; the order ledger's idempotency key makes the dual-write phase safe.
  • Headroom. +30% per region (burst + rolling deploys); the failover region per shard sized at ~40% of peak, auto-scaling to 100%+ on takeover.

Real-system comparison

The closest production cousins: Netflix's "Active-Active for Multi-Regional Resiliency" — they run their services active-active across AWS regions with per-service replication (Cassandra multi-region for the eventually-consistent stuff, EVCache for caches) and explicit "evacuate a region" tooling; their key lesson is the same as ours — the hard part isn't running things twice, it's deciding per-dataset what's allowed to be stale and what isn't, and having a fast region-evacuation drill. AWS's own building blocks instantiate the audit: DynamoDB Global Tables = our multi-leader-async-LWW choice for profile/cart (AWS's conflict-resolution post documents the LWW-with-region-tiebreak we use); Aurora Global Database = a single-region writer + read replicas elsewhere with ~1 s lag and a managed cross-region failover — our order-ledger single-region-owner pattern, productized. Google Spanner / TrueTime is the "make it strongly consistent globally and eat the latency" option (our Probe C / control-plane alternative) — CockroachDB and YugabyteDB give the same Spanner-style per-table CP with explicit "survival goals" (region-survivable vs zone-survivable per table) which is literally a per-dataset CAP knob. Cosmos DB exposes five consistency levels you pick per container — again, the per-dataset audit as a product feature. Facebook's Akkio is the home-region concept we use, generalized: it migrates a user's data ("μ-shard") to the region accessing it most, which is exactly our "home-region migration" (Probe A) run automatically and continuously. Shopify/Stripe-style regional isolation matches our cell-per-residency-shard blast-radius story.

Where we diverged from any one of these: we don't use Spanner-class global consistency for the order ledger (single-region-owner is cheaper and the user's orders are naturally single-region) — did we miss a constraint? If the business needed cross-user global ordering of orders (we don't — orders are per-user) or genuinely-global transactions spanning users in different regions (we don't — checkout touches one user + one SKU's allocation), Spanner-class would be the call. We also pin users to a home region rather than letting any region take any write (true Akkio-style fluidity) — that's a deliberate simplification: a fixed home region makes the CP datasets' ownership trivial; Akkio's automatic migration is the optimization you add when access patterns shift enough to justify the migration machinery.

Anticipated pushbacks

Stress tests against the derivation. If any can't be answered in 3–6 sentences, the page is missing a probe — close it before declaring done.

What if traffic 10×'d — 1 B users? (scale flip)

The structure is unchanged — you add more regions per residency shard (now 3–4 regions per jurisdiction instead of 2) and scale each region's app+DB fleet horizontally. The bottleneck that appears: hot SKUs — at 10× the checkout rate, a popular SKU's per-region allocation drains fast and the top-up rate against the CP pool climbs; if a single SKU's top-up rate exceeds what one Raft group commits (~10–50 k ops/s), you shard the inventory pool by SKU (cite bottlenecks#hot-key) — many small Raft groups, one per SKU-bucket. The control plane's home-region map also grows 10× — still small (it's per-user-or-shard-slice, not per-row), so still fine on a 5-region Raft group. The +30% headroom buys ~weeks of organic growth before you need the next region.

Why not just put everything on Spanner (or CockroachDB) and be done? (alt-engine)

You could — and for the order ledger + control plane it's a defensible choice (Probe C). But Spanner-class systems give you globally linearizable as the default, which means every write that isn't local pays the cross-region commit latency (cite replication-decisions#single-sync); for profile/cart — 95%+ of writes — that's ~+100 ms p99 for a consistency guarantee the NFR doesn't ask for. CockroachDB/YugabyteDB's per-table "survival goals" let you dial it down per table, which is the per-dataset CAP audit again — if you're already running Cockroach, expressing this design as Cockroach survival-goals + locality is a perfectly good instantiation. The disqualifier for "everything on Spanner": you'd be paying global-consensus latency on the AP datasets where bounded-staleness-async is both cheaper and faster, and Spanner's residency story (configurable but a config knob) is less structurally bulletproof than geo-sharding where the boundary is the schema.

What changes if a new jurisdiction (India's data-localization, China's PIPL) demands data stays in-country? (regulatory)

It's already the shape — add a residency value (IN, CN), a shard (a pair of in-country regions), and a routing-table entry; tag those users; backfill (Probe B). What stays the same: the topologies, the conflict-resolution, the control plane, the API. What changes: that shard's regions must be physically in-country (cloud-region availability becomes a constraint), and for China specifically you likely need a separate cloud account / partner (Ningxia/Beijing) which makes that shard nearly a separate deployment that shares only the global catalog — cite cross-cutting#privacy. The control plane spanning into China is itself a residency question (the home-region map for CN users is CN-data) — you'd run a CN-local control plane slice. Naming what doesn't change matters: this is why we made residency a schema dimension and not a config flag.

An entire region (eu-west) just died — walk me through the next 30 seconds. (failure)

t=0 eu-west goes dark. t≈5–10 s: health checkers in eu-central, us-east, ap-southeast stop seeing eu-west heartbeats; the control plane's quorum confirms "dead" (not "partitioned-but-alive" — and even if it were a partition, the fencing token makes the next step safe). t≈10–15 s: the control plane bumps the fencing token for all eu-west-homed entities and updates the home-region map to eu-central; pushes it. t≈15–25 s: eu-central, which is ≤ 5 s behind on async replication (= RPO), replays its last applied state, accepts the new home role, starts writing orders/sessions for EU users; eu-west's inventory allocation is returned to the CP pool and re-leased. EU users saw: reads continued throughout (eu-central was always serving them); AP writes (cart/profile) continued (multi-leader); order/session writes got a brief 503/retry for ~10–20 s. RPO ✓ (≤ 5 s lost), RTO ✓ (~15 s). When eu-west returns, it presents a stale fencing token, gets rejected as a writer, re-syncs as a follower. Cite failure-modes#leader-loss + #network-partition.

Where's the bill, and which knob moves it? (cost)

~$107 k/mo, ~95% of it compute+storage run ~4.2× (3 full home regions + 3 ~40%-sized failover regions); cross-region egress is ~1% (~$1 k/mo) because we replicate compressed deltas, not snapshots, and residency-sharding keeps EU/US/AP streams from crossing. Cite cost-shapes.html — product-class is compute-dominated, multi-region multiplies by ~(#regions). The knob: how strongly-consistent you make each dataset. Moving any AP dataset to CP-cross-region adds ~+100 ms p99 on its writes and roughly doubles cross-region egress (a quorum round-trip per write vs one async delta) — the audit spends that knob only on orders/inventory-last-unit/control-plane. Cheapest real savings: shrink the failover regions further (riskier RTO) or co-locate the failover region's compute to spin up on-demand only (cheaper, slower RTO) — both trade the DR budget for dollars.

How do you migrate from today's single-region prod without an outage? (migration)

Four reversible phases mirroring the iterations (full plan in Operations above): (1) add read replicas in eu/ap, read-only, no behaviour change; (2) per dataset, flip to its target topology behind a flag with dual-write + compare, then cut over — the order ledger's idempotency key makes dual-write safe; (3) backfill EU/APAC users into their residency shards, flip their routing once their data's there, then stop replicating it to the old region and delete it there (this step is the one with a compliance deadline — until it's done you're not residency-compliant, so it's prioritized); (4) stand up the control plane, populate the home-region map, enable automatic failover. Rollback trigger at every phase: read p95 > 80 ms or error-rate > 0.1% in any region → revert the flag. The risky phase is (3) because it touches data location; do it shard-by-shard, smallest first.

Production is slow / a region looks wrong right now — where do you look first? (monitoring/debug)

The dashboard from Operations, in order: (1) per-region read p95 — if one region is high, is it a replica behind (check that region's replication-lag SLI) or a DB-node hotspot? (2) per-shard replication lag — if > 3 s, RPO is at risk; find the lagging replica's cause (vacuum, network, a write spike). (3) CRDT conflict-resolution rate — a spike means a partition (correlate with cross-region link health) or a clock jump (correlate with NTP drift). (4) checkout 503-rate — a spike means either a CP dataset's home region is unreachable (region/partition event — check region_health) or the inventory pool is exhausted for a hot SKU (check pool levels). (5) residency-audit counter — if non-zero, a row is in the wrong shard: page immediately, it's a compliance incident. Each signal maps to a specific iteration's failure mode (C5 lists); the runbook entry point is "which of these five is red?"

S7. Key Takeaways

  • There is no global consistency choice — there's a per-dataset CAP audit. "Multi-region active-active with strong consistency" is the trap. The staff move is a table: each dataset → consistency model × topology × conflict-resolution × partition behaviour. Profile/cart = AP multi-leader; order ledger = CP single-region-owner; inventory = AP-decrement + CP-pool reservation; session = CP per-region; control plane = the one globally-strong thing.
  • The Naive design failed because it was single-region — and there was no in-region patch. The read SLO, residency, and RPO/RTO all fail for the same root cause; the only fix is multi-region, which immediately forces "what happens to a write in region A vs B?" That question is the design.
  • Iterate in this order: read-local (RTT) → per-dataset write topology (CAP audit) → residency (geo-sharding) → fast failover (control plane). Each closes exactly one constraint; "before" = previous "after". Doing the CAP audit and residency and the control plane in one step destroys the contrast.
  • Residency is a schema dimension, not a config flag. Make the partition key (residency, user_id) so an EU row physically cannot land outside the EU — the boundary is structural, auditable, and unmissable. A replication-stream filter is a fine you're one misconfiguration away from.
  • A "home region" per user makes the CP datasets cheap. Set it at signup to the user's nearest region; then the order ledger and session-issue are local for the common case — you only pay cross-region CP cost for travelers and the inventory last-unit. (Akkio is this, automated.)
  • RPO is a paging metric, RTO is a control-plane decision. RPO ≤ 5 s means "alert when in-shard replication lag > 3 s". RTO ≤ 30 s means "detect (~10 s) + quorum decide (~5 s) + reassign a tiny home-map (~10 s)" — which only works if the home-map is small and CP. Fence the dead region with a monotonic token, never a clock-lease.
  • The cross-region egress is the cheap part — minimize it anyway. The bill is ~4× single-region compute (you run it 3× + failover); egress is ~1%. Replicate deltas not snapshots, compress, and residency-shard so EU/US/AP streams never cross — that's what keeps egress at 1%.
  • "Strong everywhere" costs ~+100 ms p99 per write and ~2× egress — that's WHY you do the audit. The audit isn't bureaucracy; it's the act of spending the strong-consistency budget only where an NFR demands it (orders, inventory last-unit, control plane) and leaving everything else AP-and-fast.
