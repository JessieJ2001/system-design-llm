Debugging a Slow Database Query at 3 AM

Your monitoring system pages you. A 200ms query now takes 5 seconds. Search pages are timing out, conversion is dropping, and the connection pool is filling up. Walk through the systematic diagnosis and fix.

NALSD Walkthrough • ~15 min read

The Scenario

It is 3:07 AM. Your phone buzzes with a PagerDuty alert: "Product search p99 latency > 3s for 5 minutes." You open the dashboard. The product search page — the most critical page on your e-commerce site — is timing out for roughly 40% of users. Conversion rate has dropped from 3.2% to 0.4%. Revenue is bleeding.

The root cause is a single database query that powers the search page. Normally it completes in 200ms. Right now it is taking 5 seconds.

The Query

SELECT p.id, p.name, p.price, c.name AS category
FROM products p
JOIN categories c ON p.category_id = c.id
WHERE p.name ILIKE '%wireless%'
  AND p.price BETWEEN 10 AND 500
  AND p.is_active = true
ORDER BY p.relevance_score DESC
LIMIT 20;

The query uses ILIKE, which is PostgreSQL's case-insensitive pattern matching operator. The %wireless% pattern has a leading wildcard, meaning it searches for "wireless" anywhere in the product name. This detail turns out to be critical.

The Numbers

Back-of-Envelope Math

Before diagnosing the query itself, let's understand the blast radius of a 5-second query at 2,000 QPS.

Connection Pool Exhaustion

Each query holds a database connection for its duration. Under normal conditions:

Concurrent connections needed = QPS x latency
Normal:  2,000 req/sec x 0.2s = 400 concurrent connections
Current: 2,000 req/sec x 5.0s = 10,000 concurrent connections

Your connection pool has 150 connections. Under normal load, you need 400 concurrent connections — already tight, likely spread across multiple app servers (e.g., 10 servers with 15 connections each = 150 total). But at 5-second latency, you need 10,000 concurrent connections. The pool is exhausted by a factor of 67x.

What happens when the pool is exhausted:

  1. New requests queue up waiting for a free connection.
  2. The queue grows at a rate of 2,000 requests/sec minus the drain rate.
  3. Drain rate = 150 connections / 5s = 30 queries/sec (instead of the needed 2,000).
  4. Queue growth = 2,000 - 30 = 1,970 requests/sec accumulating.
  5. After 10 seconds, 19,700 requests are queued. Most will hit the application-level timeout (typically 10-30s) and return errors.

Thread Pool Exhaustion

Each queued request holds a thread (in thread-per-request models). With 200 threads per app server and 10 app servers = 2,000 threads total. At 5s/query, throughput = 2,000 / 5 = 400 req/sec. But you're receiving 2,000 req/sec. After the thread pool fills, new requests are immediately rejected with 503 Service Unavailable.

The Cascading Effect

A single slow query has turned into a complete site outage in under 2 minutes. This is why query performance monitoring matters.

Step 1: Check the Query Plan

The first thing to do with any slow query is run EXPLAIN ANALYZE. This tells you exactly what the database is doing.

The EXPLAIN ANALYZE Command

EXPLAIN ANALYZE
SELECT p.id, p.name, p.price, c.name AS category
FROM products p
JOIN categories c ON p.category_id = c.id
WHERE p.name ILIKE '%wireless%'
  AND p.price BETWEEN 10 AND 500
  AND p.is_active = true
ORDER BY p.relevance_score DESC
LIMIT 20;

The Output (simplified)

Limit  (cost=2847239.52..2847239.57 rows=20)
  (actual time=4892.331..4892.412 rows=20 loops=1)
  -> Sort  (cost=2847239.52..2848489.52 rows=500000)
       Sort Key: p.relevance_score DESC
       Sort Method: top-N heapsort  Memory: 29kB
       -> Nested Loop  (cost=0.00..2835739.52 rows=500000)
            -> Seq Scan on products p  (cost=0.00..2834489.52 rows=500000)
                 Filter: ((name ILIKE '%wireless%') AND (price >= 10)
                          AND (price <= 500) AND (is_active = true))
                 Rows Removed by Filter: 49500000
            -> Index Scan on categories c  (cost=0.28..0.30 rows=1)
                 Index Cond: (id = p.category_id)
Planning Time: 0.245 ms
Execution Time: 4892.631 ms

Reading the Query Plan

The critical line is:

Seq Scan on products p  (cost=0.00..2834489.52 rows=500000)
  Filter: ((name ILIKE '%wireless%') AND ...)
  Rows Removed by Filter: 49,500,000

Seq Scan means a sequential scan — the database is reading every single row in the products table, all 50 million of them, one by one. It reads each row, checks if the name contains "wireless" (case-insensitive), checks the price range, checks is_active, and discards 49.5 million rows that don't match. Only 500,000 rows survive the filter.

This is like searching for a specific book in a library by walking down every aisle and reading the title of every single book, instead of using the card catalog. With 50 million "books," it takes a while.

Why Not an Index Scan?

A B-tree index (the default index type in PostgreSQL) works like a sorted phone book. If you search for names starting with "Smith," the index takes you directly to the "S" section. But if you search for names containing "smith" anywhere, the sorted phone book doesn't help — "Blacksmith" is in the "B" section, "Goldsmith" is in "G," "Smithfield" is in "S." You'd have to read the entire phone book.

That's the leading wildcard problem. ILIKE '%wireless%' has a % before "wireless," meaning the match can start at any position in the string. A B-tree index is useless here, and the query planner correctly ignores it.

Index Usability by Pattern Type

PatternB-tree Index Usable?Reason
LIKE 'wireless%'YesPrefix search — index can seek to "wireless" and scan forward
LIKE '%wireless'NoSuffix search — could appear anywhere
LIKE '%wireless%'NoSubstring search — could appear anywhere
ILIKE '%wireless%'NoCase-insensitive substring — worst case

Step 2: But This Query Was Fast Before

If the query always did a sequential scan on 50M rows, it would always have been slow. Something changed. Here are the most common culprits:

Hypothesis A: The Table Grew Significantly

Perhaps the table was 10M rows three months ago, and the sequential scan took ~1 second (still slow, but below the timeout). After the holiday season, the catalog grew to 50M rows. The scan time grew proportionally: 5x more rows = 5x slower.

-- Check table size over time (if you track it)
SELECT relname, n_live_tup, last_analyze
FROM pg_stat_user_tables
WHERE relname = 'products';

Hypothesis B: Statistics Are Stale

PostgreSQL's query planner uses statistics (histograms of column value distributions) to estimate how many rows each filter will match. These statistics are collected by the ANALYZE command. If ANALYZE hasn't run in a while, the planner might have wrong estimates.

For example, the planner might estimate that ILIKE '%wireless%' matches 100 rows (because the statistics are from when the table had fewer rows with "wireless" in the name). With that estimate, a sequential scan looks cheap. In reality, it matches 500,000 rows — but the planner doesn't know that.

Stale statistics can also cause the planner to choose a bad join order or a bad index. Check when statistics were last collected:

-- When was ANALYZE last run?
SELECT schemaname, relname, last_analyze, last_autoanalyze,
       n_live_tup, n_dead_tup
FROM pg_stat_user_tables
WHERE relname = 'products';

-- If last_analyze is NULL or weeks old, run it:
ANALYZE products;

Hypothesis C: A Critical Index Was Dropped

Someone ran a migration that dropped an index. Maybe there was a trigram index that supported the ILIKE query, and it was accidentally removed during a schema change. Check the current indexes:

-- List all indexes on the products table
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'products';

Hypothesis D: Lock Contention

A long-running batch job (e.g., a nightly import that updates all product prices) might be holding a lock that blocks the search query. The query itself isn't slow — it's waiting for a lock.

-- Check for blocked queries
SELECT blocked_locks.pid AS blocked_pid,
       blocked_activity.query AS blocked_query,
       blocking_locks.pid AS blocking_pid,
       blocking_activity.query AS blocking_query
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity
  ON blocked_activity.pid = blocked_locks.pid
JOIN pg_catalog.pg_locks blocking_locks
  ON blocking_locks.locktype = blocked_locks.locktype
  AND blocking_locks.database IS NOT DISTINCT FROM blocked_locks.database
  AND blocking_locks.relation IS NOT DISTINCT FROM blocked_locks.relation
  AND blocking_locks.pid != blocked_locks.pid
JOIN pg_catalog.pg_stat_activity blocking_activity
  ON blocking_activity.pid = blocking_locks.pid
WHERE NOT blocked_locks.granted;

Hypothesis E: Connection Pool Exhaustion (Feedback Loop)

This is a vicious cycle. The query is a bit slow (say 1s instead of 200ms due to table growth). This causes connections to be held longer. The pool fills up. New queries wait for a connection. By the time they get one, the database is under heavier load (more concurrent queries), making each query even slower. The cycle accelerates until everything times out.

-- Check active connections and their state
SELECT state, count(*), avg(now() - state_change) AS avg_duration
FROM pg_stat_activity
WHERE datname = 'mydb'
GROUP BY state;

-- Look for long-running queries
SELECT pid, now() - query_start AS duration, query
FROM pg_stat_activity
WHERE state = 'active'
  AND now() - query_start > interval '5 seconds'
ORDER BY duration DESC;

Iteration 1: The Trigram Index

The core problem is that B-tree indexes can't handle ILIKE '%wireless%'. We need a different kind of index.

What Is a Trigram?

A trigram is a sequence of three consecutive characters. PostgreSQL's pg_trgm extension breaks strings into trigrams and indexes them.

The word "wireless" breaks into these trigrams:
  "  w", " wi", "wir", "ire", "rel", "ele", "les", "ess", "ss "

The word "Sony Wireless Headphones" breaks into:
  "  s", " so", "son", "ony", "ny ", "y w", " wi", "wir", "ire",
  "rel", "ele", "les", "ess", "ss ", "s h", " he", "hea", "ead",
  "adp", "dph", "pho", "hon", "one", "nes", "es "

A trigram index stores a mapping from each trigram to the set of rows that contain it. When you search for ILIKE '%wireless%', PostgreSQL:

  1. Breaks "wireless" into trigrams: "wir", "ire", "rel", "ele", "les", "ess"
  2. Looks up each trigram in the index to find which rows contain it
  3. Intersects the results — only rows containing ALL trigrams are candidates
  4. Does a final recheck on the candidates to confirm the full pattern matches

This is dramatically faster than a sequential scan because most rows don't contain any of these trigrams, and the index eliminates them without reading the actual row data.

Creating the Index

-- Enable the extension (one-time, requires superuser)
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- Create the trigram index
CREATE INDEX CONCURRENTLY idx_products_name_trgm
ON products USING gin (name gin_trgm_ops);

Key details:

  • CONCURRENTLY: Build the index without locking the table. This is essential for a 50M-row table in production — without it, the table would be locked for writes during the entire index build (potentially hours).
  • USING gin: GIN (Generalized Inverted Index) is the index type designed for "contains" queries. It maps each trigram to the list of rows containing it.
  • gin_trgm_ops: The operator class that tells PostgreSQL to break the column into trigrams for indexing.

How Long Does Index Creation Take?

50M rows, average name length ~50 characters:
  ~15 trigrams per name (after deduplication)
  50M x 15 = 750M trigram entries in the index

Build time estimate: 30-90 minutes for CONCURRENTLY on 50M rows
  (depends on disk speed, RAM, CPU)

Index size: approximately 2-4 GB

This is too slow for a 3 AM emergency. The index should be created proactively, but right now we need a faster fix.

After the Trigram Index Is Built

Run EXPLAIN ANALYZE again:

Limit  (cost=1520.42..1520.47 rows=20)
  (actual time=45.892..45.974 rows=20 loops=1)
  -> Sort  (cost=1520.42..2770.42 rows=500000)
       -> Nested Loop  (cost=384.28..1234.56 rows=500000)
            -> Bitmap Heap Scan on products p  (cost=384.00..1200.00 rows=500000)
                 Recheck Cond: (name ILIKE '%wireless%')
                 Filter: ((price >= 10) AND (price <= 500) AND is_active)
                 -> Bitmap Index Scan on idx_products_name_trgm
                      Index Cond: (name ILIKE '%wireless%')
            -> Index Scan on categories c  (cost=0.28..0.30 rows=1)
Execution Time: 46.152 ms

Query time dropped from 4,892ms to 46ms — a 100x improvement. The database now uses the trigram index to find the ~500K matching products directly, instead of scanning all 50M.

Iteration 2: Composite and Partial Indexes

The trigram index solves the ILIKE problem, but we're still filtering 500K rows for price range and is_active, then sorting by relevance_score. We can optimize further.

Partial Index for Active Products

Since 85% of searches only look at active products, create an index that only includes active rows:

-- Partial index: only indexes rows where is_active = true
CREATE INDEX CONCURRENTLY idx_products_price_active
ON products (price, relevance_score DESC)
WHERE is_active = true;

This index is smaller (only active products, perhaps 40M of 50M rows) and covers both the price filter and the sort order.

Covering Index Strategy

If we include all needed columns in the index, PostgreSQL can answer the query entirely from the index without touching the table data (an "index-only scan"):

-- Covering index with INCLUDE (PostgreSQL 11+)
CREATE INDEX CONCURRENTLY idx_products_search_covering
ON products USING gin (name gin_trgm_ops)
INCLUDE (price, is_active, relevance_score, category_id);

Note: GIN indexes with INCLUDE are supported in PostgreSQL 14+. For earlier versions, you'd need a different approach.

The ILIKE filter is by far the most selective — it eliminates 99% of rows. The optimizer should apply it first. If statistics are accurate, it will. If statistics are stale (hypothesis B from earlier), it might choose a worse plan. Always ensure ANALYZE has been run.

Iteration 3: The Emergency Response (Right Now at 3 AM)

Creating a trigram index on 50M rows takes 30-90 minutes. Users are experiencing outages right now. Here's the immediate triage, in order:

Step 1: Reduce the Bleeding (0-2 minutes)

Action: Enable circuit breaker or serve degraded results
  - If you have a circuit breaker: trip it for the search endpoint
  - Serve cached search results (even if stale)
  - Or: return a "Search is temporarily unavailable" message
  - This stops the thread pool exhaustion cascade immediately

Step 2: Kill Long-Running Queries (2-5 minutes)

-- Find and kill queries running longer than 10 seconds
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE state = 'active'
  AND query ILIKE '%products%wireless%'
  AND now() - query_start > interval '10 seconds';

This frees up database connections immediately. But new queries will still be slow.

Step 3: Run ANALYZE (5-10 minutes)

-- Update statistics for the products table
ANALYZE products;

This takes 1-5 minutes on a 50M-row table. If the problem was stale statistics causing a bad query plan, this fixes it immediately.

Step 4: Check for Lock Contention (Parallel with Step 3)

-- If a batch job is blocking, consider killing it
SELECT pid, query, now() - xact_start AS transaction_duration
FROM pg_stat_activity
WHERE state = 'active'
  AND now() - xact_start > interval '30 minutes'
ORDER BY transaction_duration DESC;

Step 5: Start the Index Build (10+ minutes)

CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX CONCURRENTLY idx_products_name_trgm
ON products USING gin (name gin_trgm_ops);

This runs in the background without locking the table. Once complete (30-90 min), the query will be fast.

Step 6: Gradually Re-enable Traffic

After Steps 2-3, if the query is faster (check with EXPLAIN ANALYZE):
  - Partially re-enable search (10% of traffic)
  - Monitor latency
  - If stable, increase to 25%, 50%, 100%

If query is still slow (waiting for index build):
  - Keep serving degraded results
  - Add a cache layer: cache search results for 5 minutes
  - At 2,000 QPS, most searches are for common terms
  - Even a 60-second cache TTL reduces DB load by 98%+

Iteration 4: Connection Pool Recovery

Even after fixing the query, recovery isn't instant. The connection pool has queued hundreds or thousands of requests. These all fire at once when connections become available, creating a thundering herd on the database.

The Pool Drainage Problem

State when query is fixed:
  - 150 connections busy with slow queries (some still running)
  - 5,000+ requests queued waiting for a connection
  - Each queued request has been waiting 5-30 seconds

When the slow queries finish:
  - 150 connections free up simultaneously
  - 150 queued requests immediately get connections
  - They all hit the DB at once
  - If the DB can handle it, great
  - If not, latency spikes again (mini-cascade)

Controlled Recovery

  1. Drain the queue first: Don't immediately release all queued requests. Many have been waiting longer than the client timeout (30s). These will get a DB result but have no client to send it to — wasted work.
  2. Drop aged-out requests: If a request has been queued for longer than the client timeout, drop it. There's no point executing it.
  3. Ramp up gradually: Start with 10% of the connection pool active, measure latency, increase.
# Pseudo-code: connection pool with queue aging
class SmartConnectionPool:
    def get_connection(self, timeout=30):
        queued_at = time.now()
        conn = self.pool.acquire(timeout=timeout)

        # How long did we wait?
        wait_time = time.now() - queued_at

        # If we waited longer than the client timeout,
        # the client has already given up. Release the connection.
        if wait_time > CLIENT_TIMEOUT:
            self.pool.release(conn)
            raise TimeoutError("Request aged out in queue")

        return conn

Final Design: Preventing the Next 3 AM Page

The immediate crisis is resolved. Here's the architecture to prevent it from happening again.

1. Query Performance Monitoring

-- Enable pg_stat_statements to track query performance
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- Find the slowest queries by total time
SELECT query,
       calls,
       mean_exec_time,
       max_exec_time,
       stddev_exec_time,
       rows
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 20;

2. Proactive Index Management

-- Detect missing indexes: find sequential scans on large tables
SELECT relname, seq_scan, seq_tup_read,
       idx_scan, idx_tup_fetch,
       n_live_tup,
       round(seq_tup_read::numeric / GREATEST(seq_scan, 1)) AS avg_rows_per_seq_scan
FROM pg_stat_user_tables
WHERE seq_scan > 0
  AND n_live_tup > 100000
ORDER BY seq_tup_read DESC;

3. Automatic Statistics Maintenance

-- In postgresql.conf, tune autovacuum for large tables:
autovacuum_analyze_scale_factor = 0.02  -- analyze after 2% of rows change
                                        -- (default is 10%, too lazy for 50M rows)
                                        -- 2% of 50M = 1M row changes triggers ANALYZE

autovacuum_analyze_threshold = 50000    -- minimum changes before analyze

4. Search Architecture for Scale

For a 50M-product search, the long-term solution is a dedicated search engine. Elasticsearch advantages for text search: inverted index built specifically for text search, distributed across multiple nodes, supports fuzzy matching, synonyms, and relevance scoring, handles ILIKE '%wireless%' natively and efficiently, and can serve 50K+ search queries/sec from a small cluster.

Trade-offs: data duplication (products in both Postgres and ES), synchronization complexity (what if ES is out of date?), and additional infrastructure to manage.

5. Caching Layer for Common Searches

Top 1,000 search terms account for ~80% of searches (Zipf's law)

Cache strategy:
  Key:   search:{normalized_query_hash}
  Value: serialized result set (20 product objects)
  TTL:   5 minutes (products don't change that fast)

  Cache hit rate estimate: 80%+ for common terms
  DB load reduction: 80% * 2,000 QPS = 1,600 QPS saved
  Remaining DB load: 400 QPS (easily handled)

Key Takeaways

  1. EXPLAIN ANALYZE is your first tool. Always check the query plan before guessing. The difference between a sequential scan and an index scan can be 100x.
  2. Leading wildcards kill B-tree indexes. LIKE '%foo%' cannot use a standard B-tree index. Use trigram indexes (pg_trgm) for substring search, or move to a dedicated search engine.
  3. Slow queries cascade. A 25x latency increase (200ms to 5s) causes connection pool exhaustion, thread pool exhaustion, and then a complete outage. The blast radius of one slow query is the entire application.
  4. Connection pool math matters. Concurrent connections = QPS x latency. If latency increases 25x, you need 25x more connections. If you don't have them, requests queue and timeout.
  5. Keep statistics fresh. Stale statistics cause the query planner to make bad decisions. Tune autovacuum_analyze_scale_factor for large tables.
  6. Have a degraded mode ready. When the database is in trouble, serve cached or degraded results rather than letting everything fail. A search page that shows "popular products" is better than a 503 error.
  7. Monitor query latency percentiles, not averages. The average might be 200ms while the p99 is 3 seconds. Alert on p99 before it reaches the timeout threshold.
  8. For text search at scale, use a search engine. PostgreSQL is a great database, but Elasticsearch/OpenSearch is purpose-built for text search. At 50M+ rows with complex text queries, the right tool matters.
