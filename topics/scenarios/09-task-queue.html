Task Queue: Image Processing Pipeline

Designing a reliable task queue that processes 100K image tasks per minute with at-least-once delivery, exponential backoff retries, and dead letter queues — all under a 5-minute SLA.

NALSD Walkthrough • ~15 min read

The Scenario

You are building the backend for a photo-sharing platform. Every time a user uploads an image, the system must produce multiple derived versions: a resized copy for mobile, a thumbnail for previews, and a watermarked copy for sharing. These transformations are CPU-intensive and cannot block the upload API.

Concrete Numbers

MetricValue
Task arrival rate100,000 tasks/min (peak)
Task processing time2–30 seconds (average 10 seconds)
Task typesResize, thumbnail, watermark (1 upload = 3 tasks)
Delivery guaranteeAt-least-once
Retry policyExponential backoff, max 3 attempts
Dead letter queueAfter 3 failures, move to DLQ for manual inspection
SLA95% of tasks complete within 5 minutes of submission

Why a Task Queue?

The upload API must return a response to the user in under 500ms. Image processing takes 2–30 seconds. If the API did processing synchronously, users would stare at a spinner for 10 seconds on average. Worse, during traffic spikes the API servers would be overwhelmed with CPU-bound work, leading to timeouts on all requests — including simple ones like viewing a feed.

A task queue decouples the "accept the work" step from the "do the work" step. The API server says "got it, I will process this later" and immediately responds. A separate pool of workers pulls tasks from the queue and processes them asynchronously.

Back-of-Envelope Math

Before writing any code, let us figure out how many machines we need and whether our SLA is achievable.

Throughput and Worker Count

Task arrival rate:
  100,000 tasks/min = 1,667 tasks/sec

Average processing time per task:
  10 seconds

Little's Law: L = λ × W
  L = number of tasks being processed concurrently
  λ = arrival rate = 1,667 tasks/sec
  W = average processing time = 10 sec

  L = 1,667 × 10 = 16,670 concurrent workers needed

If each VM runs 4 workers (image processing is CPU-bound,
and most VMs have 4 cores):
  16,670 / 4 = 4,168 VMs

Cost at $0.10/hr (on-demand):
  4,168 × $0.10 × 24 × 30 = $300,096/month
That is expensive. 4,168 VMs running 24/7 costs $300K/month. But 100K tasks/min is the peak rate. Average might be 30K tasks/min. We need a strategy that scales up during peaks and down during troughs.

Optimized Fleet Sizing

Strategy: Base fleet + auto-scaling + spot instances

Base fleet (handles average load):
  30,000 tasks/min = 500 tasks/sec
  Workers needed: 500 × 10 = 5,000
  VMs (4 workers each): 1,250 VMs
  Use reserved instances: $0.04/hr
  Cost: 1,250 × $0.04 × 24 × 30 = $36,000/month

Auto-scale fleet (handles peak):
  Additional 70,000 tasks/min = 1,167 tasks/sec
  Additional workers: 1,167 × 10 = 11,670
  Additional VMs: 2,918 VMs (spot instances at $0.03/hr)
  Running ~8 hrs/day during peaks:
  Cost: 2,918 × $0.03 × 8 × 30 = $21,010/month

Total: ~$57,000/month (vs $300K with naive approach)

SLA Feasibility Check

SLA: 95% of tasks complete within 5 minutes of submission.

At steady state (arrival rate = processing rate):
  Queue wait time ≈ 0 (workers available immediately)
  Processing time: avg 10s, p95 ≈ 25s
  Total: p95 ≈ 25 seconds. SLA met easily.

During a burst (arrival rate > processing rate):
  Suppose tasks arrive at 2,000/sec but we can process 800/sec.
  Queue growth rate: 2,000 - 800 = 1,200 tasks/sec
  After 5 minutes: 1,200 × 300 = 360,000 tasks backlogged

  Time to drain backlog at 800/sec: 360,000 / 800 = 450 sec = 7.5 min
  A task arriving at minute 5 of the burst waits 7.5 min in queue
  + 10 sec processing = 7.7 minutes total. VIOLATES SLA.

  Fix: Auto-scaling must kick in within 2 minutes.
  If we scale to 2,000/sec processing by minute 2:
    Backlog at minute 2: 1,200 × 120 = 144,000 tasks
    After scaling, processing rate matches arrival rate.
    Backlog drains at 2,000/sec: 144,000 / 2,000 = 72 sec
    Max wait: 2 min (scale-up) + 72 sec (drain) + 10 sec = ~3.2 min
    SLA MET.

Queue Storage Requirements

Each task message contains:
  - task_id (UUID): 36 bytes
  - image_url (S3 path): ~100 bytes
  - task_type (resize/thumb/watermark): 10 bytes
  - parameters (dimensions, quality): ~200 bytes
  - metadata (user_id, upload_time, retry_count): ~100 bytes
  Total: ~450 bytes per message, round up to 500 bytes

Max queue depth during burst (360K messages):
  360,000 × 500 bytes = 180 MB

This is trivial for any message broker.
Even Redis can hold this in memory with ease.

Naive Design

The simplest possible task queue: a Redis list. Producers push tasks to the left; workers pop from the right.

Architecture

Producer Code

import redis, json, uuid

r = redis.Redis(host='queue-redis', port=6379)

def enqueue_task(image_url, task_type, params):
    task = {
        'task_id': str(uuid.uuid4()),
        'image_url': image_url,
        'task_type': task_type,   # 'resize', 'thumbnail', 'watermark'
        'params': params,
        'created_at': time.time(),
    }
    r.lpush('image_tasks', json.dumps(task))
    return task['task_id']

Worker Code

def worker_loop():
    while True:
        # BRPOP blocks until a task is available (timeout 30s)
        _, raw = r.brpop('image_tasks', timeout=30)
        if raw is None:
            continue

        task = json.loads(raw)
        process_image(task)  # resize, thumbnail, or watermark
        print(f"Completed task {task['task_id']}")

This is dead simple. LPUSH/BRPOP gives us a FIFO queue. BRPOP is blocking, so workers don't busy-wait. Redis handles thousands of operations per second.

What Breaks

The naive design has five critical failure modes that will surface immediately in production.

Problem 1: Task Loss on Worker Crash

BRPOP removes the task from the list the moment a worker receives it. If the worker crashes mid-processing (out-of-memory, segfault, power loss), the task is gone forever. The image is never processed. The user never gets their thumbnail.

Timeline of a lost task:

t=0.0s  Worker calls BRPOP, receives task T1
        Redis list: [T2, T3, T4]   (T1 is GONE from Redis)
t=3.2s  Worker starts processing T1 (downloading image from S3)
t=5.7s  Worker process crashes (OOM while resizing 50MB image)
        T1 is lost. Not in Redis. Not completed.
        Nobody knows T1 ever existed.

Problem 2: No Retry Logic

If image processing fails (corrupt file, unsupported format, temporary S3 error), the task is simply lost. There is no mechanism to try again. Transient errors (S3 returning 503 for 2 seconds) cause permanent failures.

Problem 3: No Dead Letter Queue

Suppose a particular image is corrupt and will always fail processing. Without a DLQ, one of two things happens: (a) with no retries, the task silently disappears; (b) with naive retries, the task retries forever, consuming worker capacity endlessly. A DLQ captures these "poison pill" tasks so an engineer can investigate without wasting worker time.

Problem 4: Redis Data Loss

Redis stores data in memory. If the Redis instance crashes or restarts, the entire queue is lost. Even with Redis persistence (RDB snapshots or AOF), there is a window of data loss: RDB might be 5 minutes old, and AOF with fsync every second can lose 1 second of writes. At 1,667 tasks/sec, that is 1,667 lost tasks.

Problem 5: No Priority or Ordering

All tasks are treated equally. A user-initiated "resize my profile picture" task waits behind 10,000 batch-processing tasks. No way to prioritize urgent work.

The core issue: Redis lists were designed as a data structure, not a message broker. They lack acknowledgment, visibility timeouts, retry policies, and durability guarantees — all features that a production task queue needs.

Iteration 1: Proper Message Broker with Visibility Timeouts

Replace the Redis list with a proper message broker that supports acknowledgment-based delivery. Both RabbitMQ and Amazon SQS provide this out of the box. Let us use SQS-style semantics (the concepts are the same in RabbitMQ with manual ACKs).

The Visibility Timeout Mechanism

Instead of permanently removing a task when a worker receives it, the broker hides the task for a configurable timeout (called "visibility timeout"). During this period, the task is invisible to other workers. If the worker completes the task and sends an ACK (acknowledgment) before the timeout expires, the task is permanently deleted. If the worker crashes or takes too long, the visibility timeout expires and the task reappears in the queue for another worker to pick up.

Visibility Timeout = 60 seconds

Happy path (worker succeeds):
  t=0s    Worker receives task T1. T1 becomes invisible.
  t=8s    Worker finishes processing T1.
  t=8s    Worker sends ACK for T1. T1 is permanently deleted.

Crash path (worker dies):
  t=0s    Worker receives task T1. T1 becomes invisible.
  t=3s    Worker crashes.
          ...
  t=60s   Visibility timeout expires. T1 reappears in queue.
  t=61s   Another worker receives T1 and processes it.

Slow path (processing takes longer than expected):
  t=0s    Worker receives task T1. T1 becomes invisible.
  t=55s   Worker is still processing (large image).
  t=55s   Worker sends EXTEND_VISIBILITY(T1, +60s).
          Timeout reset to t=115s.
  t=70s   Worker finishes. Sends ACK. T1 deleted.
Choosing the visibility timeout: Set it to 2–3x the maximum expected processing time. Our max is 30 seconds, so 60 seconds is a reasonable starting point. Too short: tasks reappear while still being processed, causing duplicate work. Too long: if a worker crashes, the task sits invisible for too long before being retried.

Exponential Backoff Retries

When a task fails (not a crash, but an explicit error), the worker should NOT immediately re-enqueue it. The failure might be due to a transient issue (S3 outage, downstream service overloaded). Retrying immediately would just fail again and waste resources.

Instead, we use exponential backoff: each retry waits longer than the previous one, giving the system time to recover.

Retry schedule with exponential backoff + jitter:

  Attempt 1: Immediate (first try)
  Attempt 2: Wait 30 seconds  (base_delay × 2^0 = 30s)
  Attempt 3: Wait 2 minutes   (base_delay × 2^1 = 60s, ×2 = 120s)
  Attempt 4: NONE — move to Dead Letter Queue

Jitter: Add random(0, delay × 0.5) to each wait.
  Without jitter: if 1,000 tasks all fail at the same time,
  they all retry at exactly the same time → thundering herd.
  With jitter: retries spread over a window, reducing burst load.

Example timeline for a task that fails twice then succeeds:

  t=0s      Worker picks up task T1, attempt 1.
  t=8s      Processing fails (S3 returns 503).
            Worker re-enqueues T1 with delay=30s, retry_count=1.
  t=38s     T1 becomes visible again.
  t=39s     Worker picks up T1, attempt 2.
  t=47s     Processing fails again (S3 still down).
            Worker re-enqueues T1 with delay=120s, retry_count=2.
  t=167s    T1 becomes visible again.
  t=168s    Worker picks up T1, attempt 3.
  t=176s    Processing succeeds! Worker ACKs. T1 deleted.

Dead Letter Queue (DLQ)

After 3 failed attempts, the task is moved to a separate queue called the Dead Letter Queue. The DLQ is not actively consumed by workers. Instead, it serves as a holding area for tasks that consistently fail. An engineer or automated system can later inspect these tasks, fix the underlying issue, and optionally re-drive them back into the main queue.

DLQ flow:

  Main Queue: [T1, T2, T3, ...]
                 │
                 ▼
  Worker processes T1 → FAIL (attempt 1)
  Worker processes T1 → FAIL (attempt 2)
  Worker processes T1 → FAIL (attempt 3, max reached)
                 │
                 ▼
  Dead Letter Queue: [T1]

  T1 includes metadata:
  {
    "task_id": "abc-123",
    "original_queue": "image_tasks",
    "failure_count": 3,
    "last_error": "UnsupportedImageFormat: .heic not supported",
    "first_failure_at": "2026-03-21T10:00:00Z",
    "last_failure_at": "2026-03-21T10:02:30Z"
  }

Common DLQ inhabitants:
  - Corrupt images that crash the decoder
  - Unsupported formats (HEIC, AVIF before codec update)
  - Images referencing deleted S3 objects
  - Tasks with malformed parameters

Worker Code with Retries and DLQ

import boto3, json, time, random

sqs = boto3.client('sqs')
MAIN_QUEUE_URL = 'https://sqs.../image-tasks'
DLQ_URL = 'https://sqs.../image-tasks-dlq'
MAX_RETRIES = 3

def worker_loop():
    while True:
        response = sqs.receive_message(
            QueueUrl=MAIN_QUEUE_URL,
            MaxNumberOfMessages=1,
            VisibilityTimeout=60,
            WaitTimeSeconds=20,  # long polling (reduces empty receives)
        )
        for msg in response.get('Messages', []):
            task = json.loads(msg['Body'])
            retry_count = task.get('retry_count', 0)

            try:
                process_image(task)
                # Success — delete from queue (ACK)
                sqs.delete_message(
                    QueueUrl=MAIN_QUEUE_URL,
                    ReceiptHandle=msg['ReceiptHandle']
                )
            except Exception as e:
                # Delete the failed message from main queue
                sqs.delete_message(
                    QueueUrl=MAIN_QUEUE_URL,
                    ReceiptHandle=msg['ReceiptHandle']
                )
                if retry_count + 1 >= MAX_RETRIES:
                    # Move to DLQ
                    task['last_error'] = str(e)
                    task['failure_count'] = retry_count + 1
                    sqs.send_message(
                        QueueUrl=DLQ_URL,
                        MessageBody=json.dumps(task)
                    )
                else:
                    # Re-enqueue with backoff
                    task['retry_count'] = retry_count + 1
                    delay = min(30 * (2 ** retry_count), 900)  # cap at 15 min
                    jitter = random.uniform(0, delay * 0.5)
                    sqs.send_message(
                        QueueUrl=MAIN_QUEUE_URL,
                        MessageBody=json.dumps(task),
                        DelaySeconds=int(delay + jitter)
                    )

Architecture After Iteration 1

What this solves: Task loss on crash (visibility timeout), transient failures (exponential backoff retries), poison pills (DLQ), and durability (SQS replicates messages across multiple AZs).

What is still missing: Task priority, deduplication, and observability.

Iteration 2: Priority, Deduplication, and Observability

Task Priority

Not all tasks are equal. A user who just uploaded a profile picture and is waiting for the thumbnail to appear should not wait behind 50,000 batch-processing tasks from the nightly job.

We create separate queues for different priority levels:

Priority queues:

  HIGH priority queue:   User-initiated tasks (profile pic, post image)
  NORMAL priority queue: Standard processing (feed thumbnails)
  LOW priority queue:    Batch jobs (reprocessing old images, new format)

Worker polling strategy:
  1. Check HIGH queue first (always).
  2. If HIGH is empty, check NORMAL queue.
  3. If NORMAL is empty, check LOW queue.

  Workers spend 70% of time on HIGH, 25% on NORMAL, 5% on LOW.
  Or: dedicate separate worker pools per priority.

Alternative: Weighted fair queuing
  - Poll HIGH with 60% probability, NORMAL 30%, LOW 10%
  - Prevents LOW starvation while still prioritizing HIGH

Task Deduplication

If the upload API server retries (due to a timeout or network blip), the same image might be enqueued twice. Processing the same image twice wastes worker time and could cause bugs (double watermark, duplicate thumbnails).

We use an idempotency key: a hash of (image_url + task_type). Before processing, the worker checks a deduplication store (Redis or DynamoDB) to see if this task has already been completed.

def process_with_dedup(task):
    dedup_key = f"dedup:{task['image_url']}:{task['task_type']}"

    # Check if already processed (atomic SET-if-not-exists)
    if not redis_client.set(dedup_key, 'processing', nx=True, ex=3600):
        # Key already exists — task was already processed or is in progress
        print(f"Skipping duplicate task {task['task_id']}")
        return

    try:
        result = process_image(task)
        redis_client.set(dedup_key, 'completed', ex=86400)  # keep for 24h
    except Exception:
        redis_client.delete(dedup_key)  # allow retry
        raise

Observability

Without metrics, we are flying blind. We need to monitor three things to know if the system is healthy and meeting its SLA.

MetricWhat It Tells YouAlert Threshold
Queue depth How many tasks are waiting. If growing, workers can't keep up. > 50,000 for 5 min: scale up workers
Task age (oldest message) How long the oldest task has been waiting. Directly predicts SLA violations. > 3 min: warning. > 4 min: critical (SLA at risk).
Processing latency (p50/p95/p99) How long tasks take to process once a worker picks them up. Detects slow images or degraded workers. p95 > 25s: investigate
Failure rate Percentage of tasks that fail on first attempt. Spike means something is wrong upstream. > 5%: warning. > 20%: critical.
DLQ depth How many tasks have permanently failed. Should be near zero. > 100: investigate
End-to-end latency Time from task submission to completion. The SLA metric. p95 > 4 min: scale up immediately

Auto-Scaling Based on Queue Depth

Auto-scaling policy:

  Metric: ApproximateNumberOfMessagesVisible (queue depth)

  Scale-out:
    If queue_depth > 10,000 for 1 minute:
      Add 20% more workers (up to max 5,000 VMs)
    If queue_depth > 50,000 for 2 minutes:
      Scale to max immediately (emergency)

  Scale-in:
    If queue_depth < 1,000 for 10 minutes:
      Remove 10% of workers (down to base fleet of 1,250 VMs)
    Cool-down period: 5 minutes between scale-in actions
      (prevents flapping)

Example scale-out timeline:

  t=0 min   Traffic spike begins. 2,000 tasks/sec arriving.
            Base fleet processes 800/sec. Queue growing at 1,200/sec.
  t=1 min   Queue depth = 72,000. Auto-scale triggers.
            Requests 500 additional VMs (spot instances).
  t=2 min   Queue depth = 144,000. New VMs launching.
  t=3 min   300 new VMs online (1,200 workers). Total: 9,200 workers.
            Processing rate: 920/sec. Still behind, but closing gap.
  t=4 min   All 500 VMs online (2,000 workers). Total: 10,000 workers.
            Processing rate: 1,000/sec. Still slightly behind.
  t=5 min   Auto-scale adds another batch. Total: 12,000 workers.
            Processing rate: 1,200/sec. Queue starts draining.
  t=8 min   Queue depth back to near-zero. SLA maintained for 95%+.

Iteration 3: Handling Edge Cases at Scale

The Poison Pill Problem

A "poison pill" is a task that crashes the worker every time it is processed. For example, an image with a maliciously crafted header that triggers a buffer overflow in the image library. Without protection, this task cycles between the main queue and workers forever: worker picks it up, crashes, task reappears, another worker picks it up, crashes again.

With our retry limit of 3, the poison pill moves to the DLQ after 3 worker crashes. But it has already killed 3 worker processes. If 100 poison pills arrive at once, we lose 300 workers. The fix: run image processing in a sandboxed subprocess. If the subprocess crashes, the parent worker process catches the failure, increments the retry count, and continues without dying.

import subprocess, json

def process_image_sandboxed(task):
    """Run image processing in a subprocess to isolate crashes."""
    result = subprocess.run(
        ['python', 'image_processor.py', json.dumps(task)],
        timeout=45,           # hard kill after 45 seconds
        capture_output=True,
    )
    if result.returncode != 0:
        raise ProcessingError(
            f"Worker subprocess failed: {result.stderr.decode()}"
        )

Task Batching for Efficiency

Each image upload generates 3 tasks (resize, thumbnail, watermark). Processing them separately means downloading the original image from S3 three times. Instead, we can batch all three tasks for the same image into one unit of work.

Without batching (3 tasks per image):
  Download image from S3:    200ms  × 3 = 600ms
  Process (resize):          8,000ms
  Process (thumbnail):       2,000ms
  Process (watermark):       4,000ms
  Upload results to S3:      150ms  × 3 = 450ms
  Total:                     15,050ms for 3 tasks
  S3 bandwidth:              3 downloads + 3 uploads = 6 transfers

With batching (1 compound task per image):
  Download image from S3:    200ms  × 1 = 200ms
  Process (all 3 in memory): 14,000ms
  Upload results to S3:      150ms  × 3 = 450ms
  Total:                     14,650ms for 3 tasks
  S3 bandwidth:              1 download + 3 uploads = 4 transfers

Savings: 2 fewer S3 downloads per image = 66,667 fewer S3 GETs/min
At $0.0004 per 1,000 GETs: saves $0.027/min = $38.40/day = $1,152/month

Graceful Shutdown

When scaling down or deploying new code, workers must shut down gracefully. A worker that is killed mid-task causes a visibility timeout wait before the task is retried. With a 60-second visibility timeout, that is up to 60 seconds of unnecessary delay.

import signal, sys

shutdown_requested = False

def handle_sigterm(signum, frame):
    global shutdown_requested
    shutdown_requested = True
    print("SIGTERM received. Finishing current task, then exiting.")

signal.signal(signal.SIGTERM, handle_sigterm)

def worker_loop():
    while not shutdown_requested:
        msg = receive_message()
        if msg:
            process_and_ack(msg)
    # After loop: no more tasks picked up. Process exits cleanly.
    print("Worker shut down gracefully.")
    sys.exit(0)

Key Takeaways

  • Little's Law is your first tool. L = lambda x W gives you the number of concurrent workers needed. Do this math before anything else — it tells you if the problem is even tractable.
  • Visibility timeouts solve the "pop and crash" problem. Never permanently remove a task from a queue until it is confirmed complete. This one mechanism is the difference between "tasks sometimes disappear" and "at-least-once delivery."
  • Exponential backoff with jitter is not optional. Without it, transient failures cause retry storms that amplify the original problem. The jitter prevents synchronized retries from creating thundering herds.
  • Dead letter queues separate "broken tasks" from "broken workers." Without a DLQ, a single corrupt image can consume infinite worker capacity. With a DLQ, it fails three times and gets quarantined.
  • Auto-scaling must be faster than your SLA. If 95% of tasks must complete in 5 minutes, your auto-scaling must detect overload and add capacity within 2 minutes. Do the math: how fast does the queue grow, how fast can you add workers, how long to drain the backlog?
  • Batch related tasks. Three tasks that all process the same image should download it once, not three times. This is a simple optimization that saves real money at scale.
  • Idempotency is the safety net. At-least-once delivery means tasks might be processed more than once. Idempotency keys ensure that duplicate processing has no side effects.
