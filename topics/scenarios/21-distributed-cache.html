Design a Distributed Cache

10TB of key-value data, 1M reads/sec, sub-5ms latency, 99.99% availability — and a celebrity tweet that aims 50,000 reads/sec at one key. Start from the LC #146 doubly-linked-list + hashmap LRU; iterate to a sharded, replicated, hot-key-resistant cache. (Absorbs the former "Imbalanced Cache" scenario as Iteration 4.)

NALSD Walkthrough • ~25 min read • LC #146 (LRU Cache)

LC kernel. LRU Cache (LC #146). Textbook structure: a hashmap (key → node) over a doubly-linked list ordered by recency — get moves the node to the head, put inserts at head and evicts the tail when over capacity, both O(1). System class: infrastructure primitive (cite functional-requirements#class-infra) — a building block other services depend on, not a product. Four properties that break at scale, each motivating an iteration: (1) one node's RAM ≪ the working set — 10TB doesn't fit in one box's 64GB; you must shard, which loses the global recency order (Iter 2 — consistent hashing + per-shard CLOCK approximates global LRU); (2) a single-node LRU has no failure story — a crash erases the whole cache, and a cold restart thunders the backend (Iter 1 — replication; Iter 3 — warm-up + eviction monitoring); (3) a hot key melts its shard — one node sees all of a key's traffic, so 50K reads/sec on one key drives that shard to its knees and the 20M innocent co-tenants with it (Iter 4 — detect → replicate-the-key → L1 local cache; consistent hashing with bounded loads); (4) a stampede on a cold key thunders the backend — a million simultaneous misses on one freshly-evicted key become a million DB queries (Iter 4 — single-flight, cite single-flight.html). The C8 production cousin: the dll+hashmap becomes a per-shard CLOCK / W-TinyLFU approximation plus a consistent-hashing ring across shards, with key-replication and an in-process L1 in front of the hot keys.

The Scenario

You are the infrastructure lead at a large e-commerce company. Your application layer makes millions of database queries per second. The database is buckling under the load — query latency has crept from 5ms to 200ms during peak traffic. Your team decides to place a distributed cache between the application and the database to absorb the read-heavy workload.

Requirements

  • Dataset size: 10TB of active key-value data must be cache-resident at all times.
  • Read throughput: 1,000,000 reads/sec (1M reads/sec).
  • Write throughput: 100,000 writes/sec (100K writes/sec).
  • Read latency: <5ms at p99.
  • Write latency: <10ms at p99.
  • Availability: 99.99% (52 minutes of downtime per year maximum).

Data Characteristics

  • Key-value pairs. Average key size: 50 bytes. Average value size: 1KB.
  • Total number of keys: 10TB / (50B + 1KB) ≈ 10 billion keys. However, with Redis overhead per key (~100 bytes for internal structures), effective capacity per key is ~1.15KB.
  • Access pattern: read-heavy, ~10:1 reads:writes (cite access-patterns#read-heavy-uniform). Underlyingly Zipf, not uniform — a small head of keys takes a large share of traffic (cite access-patterns#power-law); the hot-key tail is the subject of Iteration 4.

System class & functional requirements

Class: infrastructure primitive (cite functional-requirements#class-infra) — a look-aside KV cache other services call; it is an optimization in front of the database, not a system of record. FRs (cite functional-requirements#fr-infra): GET(k) → value | miss; SET(k, v, ttl); DEL(k) (for cache-aside invalidation); per-key TTL; bounded memory with automatic eviction; transparent failover so callers never special-case a dead shard.

Out of scope (named, not silently dropped): being a database — no durability guarantee, no transactions across keys (Redis MULTI within one slot only); secondary indexes / range scans / SQL — this is a flat KV store; cross-region replication — single region with multi-AZ here (the geo variant is a probe); the cache-aside application logic that fills and invalidates keys — that belongs to the calling service, we provide GET/SET/DEL; auth / per-tenant quotas — the cache lives inside the trust boundary, callers are first-party services on mTLS.

Flagged assumptions (would confirm with the interviewer): (1) the dataset is working-set-resident — 10TB is what's hot, not the whole 200TB of DB rows, so a >95% hit rate is plausible; (2) values are small (~1KB) — if they were 100KB serialized blobs, NIC bandwidth, not RAM, would bind first; (3) ms-level staleness is acceptable on a read — this is a cache, the DB is authoritative (if the caller needs read-your-writes it does cache-aside DELETE-on-write, or bypasses the cache); (4) the backend (the DB) can survive a full cache miss-storm at degraded latency — i.e. the cache is an optimization, not a load-bearing dependency. We revisit (4) in the degradation contract; if the DB cannot survive a cold cache, the cache becomes load-bearing and the SLO tightens (see Pushbacks).

Back-of-Envelope Math

Before drawing a single architecture diagram, we need to figure out how many cache instances we need. There are three potential bottlenecks: storage, read throughput, and write throughput. Whichever requires the most instances is our bottleneck.

Storage Bottleneck

A single Redis instance can hold data up to the size of its RAM. We will use machines with 64GB RAM each (a common production choice — large enough to be efficient, small enough that failure impact is limited).

Total data:       10TB = 10,240 GB
RAM per instance:  64 GB
Usable RAM:        ~60 GB (reserve 4GB for OS + Redis overhead)

Primary shards needed = 10,240 GB / 60 GB = 170.7 → 167 shards
  (167 × 60 GB = 10,020 GB ≈ 10 TB; close to capacity but workable)

With 3× replication (each key stored on 3 nodes):
  Total instances = 167 × 3 = 501 instances
  (Each node stores ~60GB of keys — matches our 60GB usable RAM)

Read Throughput Bottleneck

A single Redis instance handles approximately 100,000 simple GET operations per second for 1KB values (this is a well-known benchmark figure that accounts for network overhead and serialization).

Total reads:           1,000,000 ops/sec
Throughput per Redis:  100,000 ops/sec

Instances needed = 1,000,000 / 100,000 = 10 instances

With replicas serving reads (3× replication):
  Effective read capacity = 501 × 100K = 50.1M reads/sec
  (Massively overprovisioned for reads)

Write Throughput Bottleneck

Total writes:          100,000 ops/sec
Throughput per Redis:  100,000 ops/sec

Instances needed = 100,000 / 100,000 = 1 instance

With 501 instances, writes are spread across primaries:
  167 primary nodes × 100K = 16.7M write capacity
  (Also massively overprovisioned)

The Bottleneck Is Storage

Key insight: We need 167 primary shards for storage alone, but only 10 for read throughput and 1 for write throughput. Storage is the dominant constraint. With 3× replication for availability, that’s 501 instances total. Note this just barely covers 10 TB — if data grows, we’ll need to either add shards or use larger nodes.

Naive Design

The simplest possible architecture: 500 Redis instances, consistent hashing to distribute keys, no replication.

How It Works

  1. Each of the 500 Redis instances is assigned a position on a hash ring (a circular space from 0 to 232 - 1).
  2. When the client wants to read or write key K, it computes hash(K) and walks clockwise on the ring until it finds the first Redis node.
  3. That node is the owner of key K. The client sends GET or SET directly to that node.

Naive Client Code

import hashlib

class NaiveDistributedCache:
    def __init__(self, nodes):
        """nodes: list of (host, port) tuples for 500 Redis instances"""
        self.ring = {}
        self.sorted_keys = []
        for node in nodes:
            hash_val = self._hash(f"{node[0]}:{node[1]}")
            self.ring[hash_val] = node
            self.sorted_keys.append(hash_val)
        self.sorted_keys.sort()

    def _hash(self, key):
        return int(hashlib.md5(key.encode()).hexdigest(), 16) % (2**32)

    def get_node(self, key):
        """Find which Redis node owns this key"""
        h = self._hash(key)
        for ring_key in self.sorted_keys:
            if h <= ring_key:
                return self.ring[ring_key]
        return self.ring[self.sorted_keys[0]]  # wrap around

    def get(self, key):
        node = self.get_node(key)
        return redis_client(node).get(key)

    def set(self, key, value, ttl=3600):
        node = self.get_node(key)
        return redis_client(node).set(key, value, ex=ttl)

What This Gets Right

  • Simple to understand and implement.
  • Keys are spread across 500 nodes — each node holds ~20GB.
  • Read and write throughput are well within per-node limits.
  • Latency for individual operations is sub-millisecond within the same data center.

What Breaks

This naive design has several critical failure modes. Let us walk through each one with concrete numbers.

Problem 1: Node Failure Causes Data Loss and DB Spikes

With no replication, when a node dies, all data on that node is gone. The cache does not have a copy anywhere else.

Keys per node:       10 billion / 500 = 20 million keys
Data per node:       10TB / 500 = 20GB

When node #237 crashes:
  - 20 million keys become cache misses instantly
  - 1M reads/sec × (1/500) = 2,000 reads/sec now hit the database
  - These 2,000 reads/sec were previously ~1ms (cache hit)
    Now they're ~50ms (database query)
  - Database gets a sudden 2,000 QPS spike on cold queries

Problem 2: Node Failures Are Frequent at Scale

Commodity server hardware has an annual failure rate of 2-5%. With 500 servers:

Expected failures per year = 500 × 3% = 15 failures/year
That's roughly one failure every 24 days.

Each failure:
  - Causes 2,000 extra DB queries/sec
  - Lasts until a replacement node is provisioned and warmed up
  - Warm-up time: 20M keys × 50ms per DB read = way too long sequentially
    Even at 2,000 reads/sec (organic traffic refilling the cache),
    warm-up takes: 20M / 2,000 = 10,000 seconds ≈ 2.8 hours

Problem 3: Availability Math Does Not Work

Our target is 99.99% availability (52 minutes downtime per year). But a single node failure means 1/500 = 0.2% of the cache is unavailable. If we define "available" as "all cache reads succeed," a single node failure already violates our SLA.

With 15 failures/year, each causing 2.8 hours of degradation:
  Total degraded time = 15 × 2.8 hours = 42 hours/year
  Availability = (8,760 - 42) / 8,760 = 99.52%

  That's 99.52% — far below the 99.99% target.

Problem 4: Uneven Load Distribution

With only 500 points on the hash ring (one per physical node), the distribution of keys is uneven. Some nodes get 30% more keys than average, others get 30% fewer. This means some nodes are at 26GB (overflowing 20GB budget) while others sit at 14GB (wasted capacity).

Iteration 1: Add Replication

The single biggest improvement we can make is storing each key on multiple nodes. We replicate each key to 3 consecutive nodes on the hash ring: one primary and two replicas.

How Replication Works on the Ring

When key K hashes to position P on the ring, we walk clockwise and find the first three distinct physical nodes. The first is the primary (handles writes), and the next two are replicas (can serve reads).

class ReplicatedCache:
    def get_nodes(self, key, replica_count=3):
        """Return [primary, replica1, replica2] for this key"""
        h = self._hash(key)
        nodes = []
        seen_physical = set()

        for ring_key in self.sorted_keys:
            if h <= ring_key:
                node = self.ring[ring_key]
                physical = node['physical_id']
                if physical not in seen_physical:
                    nodes.append(node)
                    seen_physical.add(physical)
                    if len(nodes) == replica_count:
                        return nodes
        # Wrap around the ring if needed
        for ring_key in self.sorted_keys:
            node = self.ring[ring_key]
            physical = node['physical_id']
            if physical not in seen_physical:
                nodes.append(node)
                seen_physical.add(physical)
                if len(nodes) == replica_count:
                    return nodes
        return nodes

Alternatives considered (C4)

  • No replication, persistence instead (Redis AOF/RDB to disk). Survives a process restart but not a node loss (the disk dies with the node), and AOF fsync hurts write latency. Replication gives availability and read fan-out; persistence gives neither. Rejected. (We add RDB snapshots in Iter 3 only as a warm-up aid, not as the durability story.)
  • 2× replication. Halves the replication storage tax (334 vs 501 nodes) but a key goes unavailable if both copies are down — during a rolling deploy you routinely have one replica down, so 2× means "one bad node from a blind spot." 3× tolerates one planned + one unplanned loss. Keep 3×.
  • Synchronous replication (wait for 2-of-3 acks before SET returns). Eliminates the read-stale window but adds a cross-node RTT to every write (cite replication-decisions#single-sync) — for a cache, where the DB is authoritative anyway, that's paying DB-grade latency for nothing. Async (cite replication-decisions#single-async). Revisited in Iter 2.
  • Erasure coding instead of full replicas. Cuts storage overhead (1.5× vs 3×) but a read on a lost node requires reconstructing from k fragments — CPU + multi-node fan-in on the hot path, unacceptable for a sub-5ms cache. Erasure coding is for cold blob storage, not a cache.

What breaks (C5; cite failure-modes.html)

  • Shard-primary loss (failure-modes#leader-loss) — a replica is promoted; writes to that range pause for the ~5s detection window, reads keep flowing from the survivors. Detected by missed heartbeats; the promoted replica re-replicates a third copy in the background.
  • Replica divergence (failure-modes#replica-divergence) — async replication means a replica can lag the primary by 1–5ms; a read to the lagging replica returns the pre-write value. Bounded by the replication lag; acceptable because the DB is the source of truth and cache-aside DELETE-on-write makes the worst case "one extra cache miss," not "wrong forever."
  • Correlated node loss — if all 3 replicas of a shard land in the same rack/AZ and that fails, the shard goes dark. Mitigation: rack/AZ-aware replica placement (the next two distinct physical nodes on the ring must also be in distinct fault domains). Without it the per-shard availability math above is optimistic.

Iteration 2: Consistency Model and Virtual Nodes

Choosing a Consistency Model

With 3 replicas per key, we must decide how reads work. There are two options:

ModelHow It WorksLatencyStaleness
Strong consistency All reads go to the primary Higher (primary may be farther away; no load spreading) None — always latest value
Eventual consistency Reads go to any replica (nearest or least-loaded) Lower (choose closest replica) May read a value that is milliseconds behind the latest write

Decision: Eventual consistency for reads. Here is why:

  • This is a cache, not the source of truth. The database is authoritative. If the cache returns a slightly stale value, the consequence is minor (e.g., a user sees a product price that was updated 10ms ago).
  • With async replication, replicas lag behind the primary by 1-5ms in the same data center. This staleness is invisible to humans.
  • Eventual consistency lets us spread 1M reads/sec across all 500 nodes instead of concentrating them on 167 primaries. Each node handles 1M / 500 = 2,000 reads/sec instead of 1M / 167 = 6,000 reads/sec. Both are well within limits, but the lower per-node load gives us more headroom for spikes.

Writes always go to the primary, which asynchronously replicates to the two replicas.

Virtual Nodes for Even Distribution

The naive design placed one point per physical node on the hash ring. With 500 points, the distribution of keys is lumpy — some nodes get 2× the keys of others. Virtual nodes solve this.

Instead of placing each physical node at one position on the ring, we place it at 200 positions. Each position is a "virtual node" (vnode). The client hashes the key, walks clockwise, finds a virtual node, and maps it back to the physical node.

500 physical nodes × 200 virtual nodes each = 100,000 points on the ring

Load variance (mathematical bound):
  With N virtual nodes per physical node, the standard deviation of
  load per physical node is approximately:

    std_dev ≈ average_load / sqrt(N)
    std_dev ≈ (10TB / 500) / sqrt(200)
    std_dev ≈ 20GB / 14.14
    std_dev ≈ 1.41GB

  So most nodes hold 20GB ± 1.4GB (between 17.2GB and 22.8GB).
  With 60GB usable RAM, even 22.8GB is well within limits.

Memory cost of virtual nodes:
  100,000 entries × (4 bytes hash + 8 bytes node pointer) = ~1.2MB
  Trivial. Stored in memory on every app server.

Virtual Node Code

class VirtualNodeRing:
    def __init__(self, physical_nodes, vnodes_per_node=200):
        self.ring = {}
        self.sorted_keys = []
        self.vnodes_per_node = vnodes_per_node

        for node in physical_nodes:
            for i in range(vnodes_per_node):
                # Create a unique virtual node identifier
                vnode_key = f"{node.host}:{node.port}#vnode{i}"
                hash_val = self._hash(vnode_key)
                self.ring[hash_val] = node  # maps back to physical node
                self.sorted_keys.append(hash_val)

        self.sorted_keys.sort()

    def _hash(self, key):
        digest = hashlib.md5(key.encode()).digest()
        return int.from_bytes(digest[:4], 'big')

        h = self._hash(key)
        idx = bisect.bisect_left(self.sorted_keys, h)
        if idx == len(self.sorted_keys):
            idx = 0
        return self.ring[self.sorted_keys[idx]]

What breaks (C5; cite failure-modes.html)

  • Ring drift / split-brain ring (failure-modes#bad-deploy) — if clients hold different versions of the ring during a topology change, the same key routes to two shards and writes split. Iteration 3: Eviction, Failure Recovery, and Monitoring

LRU Eviction Strategy

Since this is a cache (not the source of truth), we can evict entries when memory is tight. Redis implements an approximated LRU eviction policy. We configure each node to start evicting when memory usage reaches 90% of available RAM:

maxmemory 57gb          # 90% of 64GB (leave headroom)
maxmemory-policy allkeys-lru  # Evict any key using approximate LRU

When a node hits 57GB:
  - Redis samples 10 random keys
  - Evicts the one with the oldest last-access time
  - Repeat until memory drops below threshold

Consequence of eviction:
  - The evicted key's next read becomes a cache miss
  - The miss triggers a DB read (50ms) + cache refill
  - This is fine: the key was rarely accessed (LRU guarantees this)
  - If the key was popular, it would have been recently accessed
    and would NOT have been evicted

Monitoring Eviction Rates

Evictions are normal. But excessive evictions mean the cache is too small for the workload.

Healthy state:
  Eviction rate per node: < 100/sec
  Cache hit rate: > 95%
  DB queries caused by misses: < 50K/sec (5% of 1M reads)

Warning threshold:
  Eviction rate per node: > 1,000/sec
  Cache hit rate: < 90%
  DB queries caused by misses: > 100K/sec

Action: if eviction rate > 1% of reads sustained for 10 minutes,
        trigger auto-scaling to add more cache nodes

Node Failure Recovery

When a node dies, the system must recover smoothly. Here is the timeline:

TimeEventImpact
T+0s Node #237 crashes No immediate impact — replicas still serve reads
T+5s Health check detects failure (3 missed heartbeats at 2s intervals) Node marked as down in the ring
T+5s Successor node takes ownership of dead node's key range Writes for that range now go to the successor as new primary
T+10s Ring update propagated to all app servers All clients route around the dead node
T+60s New replacement node provisioned (auto-scaling) Empty node joins the ring
T+60s - T+120s New node warm-up via organic traffic Cache misses gradually refill the new node

Warm-Up Math

Keys owned by the dead node: ~20M keys
Reads hitting the new (empty) node: 1M / 500 = 2,000 reads/sec

But the new node also has 2 replicas that still have the data!
Reads can still go to replicas during warm-up.

Warm-up strategy:
  Option A: Organic — wait for reads to naturally populate the new node
    Time to fill: 20M keys / 2,000 keys/sec = 10,000 sec ≈ 2.8 hours
    (Only keys that are actually read get cached — Pareto effect means
     80% of traffic hits 20% of keys, so the hot keys fill in ~30 min)

  Option B: Background copy from a replica
    Copy rate: 50,000 keys/sec (throttled to avoid impacting the replica)
    Time: 20M / 50K = 400 seconds ≈ 7 minutes
    Network: 50K keys × 1KB = 50MB/sec — easily within data center bandwidth

  → Use Option B with Option A as fallback

Alternatives considered (C4)

  • Eviction policy — LRU vs LFU vs ARC vs W-TinyLFU vs FIFO/CLOCK (see caching.html for the full taxonomy). We keep allkeys-lru: it's the LC kernel, it's what callers reason about, and Redis's sampled-LRU is cheap. W-TinyLFU (Caffeine) would give a measurably higher hit rate on this Zipf workload at the same memory — it admits new keys only if they're frequent enough to beat the eviction victim, so a one-off scan can't flush the cache; if a future hit-rate audit shows scan-flush, switch. ARC self-balances recency vs frequency but is patent-encumbered and not in Redis. FIFO/CLOCK is cheaper than LRU bookkeeping but evicts hot keys that haven't been touched recently — worse hit rate; only worth it on a CPU-starved box. volatile-* variants (evict only TTL'd keys) risk OOM if some keys have no TTL — we set a default TTL so allkeys-lru is safe.
  • Consistency / write path — cache-aside vs write-through vs write-behind (cite caching.html). We use cache-aside: the writer updates the DB then DELs the key; the next reader misses, fetches fresh, fills. Why DELETE not SET-the-new-value: two concurrent writers doing read-DB-then-SET-cache can reorder on the network and leave the cache holding the older value forever; DELETE is idempotent and the next read self-heals. Write-through (write the cache synchronously on every DB write) removes the read-after-write miss but couples write latency to the cache and doubles the write fan-out. Write-behind (cache absorbs writes, flushes to DB async) is fastest but loses un-flushed writes on a crash — unacceptable when the cache has no durability. Stale-window: cache-aside has a tiny race (reader fetches old DB row, then writer's DELETE lands, then reader's SET writes the old value) — bounded by the DB-read latency, and a default TTL caps it; for keys that can't tolerate even that, use versioned keys (key suffixed with a monotonically-increasing version; readers always read the current version, old versions TTL out) — cite api-design-patterns#concurrency.
  • Invalidation — TTL vs explicit-DEL vs versioned-keys. TTL alone (no DEL) bounds staleness to the TTL but serves a known-stale value until then; explicit DEL is exact but needs the writer to know the key; versioned keys make staleness impossible at the cost of a version registry. We use explicit DEL + a safety-net default TTL; per-key shorter TTLs for keys that change fast.

What breaks (C5; cite failure-modes.html)

  • Cold-start thundering herd (failure-modes#stampede, bottlenecks#cold-start) — a node restart (or a flushed cache after a deploy) means 20M–10B keys are cold; misses pour onto the DB at a rate it wasn't sized for. Mitigation: warm-up from a replica (Option B above) before the new node takes read traffic; rate-limit miss-driven DB reads; the deeper fix (single-flight, no two misses on the same key in flight) is Iter 4.
  • Eviction storm (bottlenecks#memory-capacity-bound) — if the working set grows past capacity, every node sits at maxmemory and evicts faster than reads refill; hit rate collapses, DB load explodes. Detected by per-node eviction rate > 1% of reads; auto-scale adds shards. The signal is in the Operations block.
  • Slow / dead backend on a miss (failure-modes#slow-dependency) — a cache miss now waits on a struggling DB; the app's threads block; retries pile on. Mitigation: short DB-read timeout on the miss path, circuit-breaker, serve-stale-on-error if the key still has an expired-but-present copy.

Iteration 4: Hot-Key Mitigation — Detect, Replicate the Key, L1 in Front

(This iteration absorbs the material that used to be a separate page, 22-imbalanced-cache.html — the hot-key / imbalanced-shard problem and its mitigations. "Make a hot key not melt one shard" is a natural NALSD iteration on "build a distributed cache," so it lives here now.)

Bottleneck analysis of the Iteration-3 design

Iterations 1–3 made keys evenly distributed — 200 vnodes per node, ±1.4GB load variance. But virtual nodes balance key counts, not traffic per key. The access pattern is Zipf (cite access-patterns#power-law): the head is brutally concentrated. Concretely — 0.1% of keys (10M of them) take 40% of reads (400K reads/sec); the top 1,000 keys take 10% (100K reads/sec); and during a viral event one single key can pull 50,000 reads/sec. That key hashes to exactly one shard. Cite bottlenecks#hot-key for the steady-state shape and failure-modes#hot-key-meltdown for the failure shape.

A shard handles ~100,000 GET/sec (from BOE: per-Redis throughput).
Uniform background load on a shard: 1M reads/sec / 500 nodes = 2,000 reads/sec
  → utilization 2,000 / 100,000 = 2%, p99 ≈ 0.5ms.

Now the viral key (50,000 reads/sec) hashes to Node-42:
  Node-42 load = 2,000 (background) + 50,000 (hot key) = 52,000 reads/sec
  Utilization = 52,000 / 100,000 = 52% → p99 ≈ 3–5ms (queueing).
And if two top keys collide on one node (birthday paradox over 500 nodes
is not negligible with 1,000 hot keys):
  Node load = 2,000 + 50,000 + 25,000 = 77,000 → 77% → p99 ≈ 10–20ms — SLA violated.

Collateral damage: the ~20M innocent keys co-located on Node-42 all get the
slow p99 too. And a slow shard makes app threads block → thread pools fill →
timeouts → retries → 2× the load on the already-hot shard → death spiral
(cite failure-modes#retry-storm). No amount of ring rebalancing fixes this —
the hot key lives on one node.

System after this iteration

Three new things vs Iter 3: (a) a count-min sketch on each shard + a coordinator that maintains the global hot-key list (detection); (b) hot-key replication — for keys on that list, the client appends a random #r0..#r9 suffix so the same logical key lives on 10 shards; (c) an L1 in-process cache on every app server in front of the hot keys, plus single-flight on the miss path. Plus a small change to the ring: consistent hashing with bounded loads — if a key's owner is over a load cap, spill to the next node.

Step 4a — Detect hot keys (count-min sketch)

You can't replicate a key you don't know is hot, and you can't keep an exact counter for 10B keys. A count-min sketch (cite sketches.html) estimates per-key access frequency in fixed memory, O(1) per increment and per query — it adds essentially zero latency to a GET, unlike a top-N min-heap which is O(log N) per read. Each Redis shard runs one (5 rows × 10,000 cols × 4B = 200KB); every 10s it scans for keys whose estimate exceeds the hot threshold (~500 reads/sec ≈ 5,000 over the 10s window) and ships those candidates to a coordinator, which unions them across all 500 shards into a global hot-key list (typically 1K–10K entries) and broadcasts it to the app servers.

Memory budget for detection:
  Per shard:        200KB count-min sketch
  Fleet of 500:     500 × 200KB = 100MB
  Hot-key list:     10,000 entries × 100B = 1MB (broadcast to app servers)
  → negligible against 10TB of cached data.

Detection lag: ~10s (one report interval) + a few ms to broadcast.
During that window one shard absorbs the spike — at the T+10s traffic level
(say 10,000 reads/sec) the shard is at 10% utilization: elevated but safe.
And the L1 cache (next step) already absorbs ~99% of the spike even before
the key is "officially" hot, so the 10s gap is not the first line of defence —
it's what enables the replication strategy and the monitoring.

Step 4b — Replicate the hot key (key suffixing)

Once a key is on the hot list, the client appends a random replica suffix before hashing — k#r0 .. k#r9 — so 10 physical copies of the value land on (up to) 10 different shards, and the key's read load is divided ~10×. Writes to a hot key fan out to all 10 copies (write amplification: ~1,000 hot keys × ~10 writes/sec × 10 copies = 100K extra writes/sec spread over 500 nodes ≈ 200/node/sec — negligible). The staleness window is the same eventual-consistency tradeoff as the base design: a hot-key write hits the primary copy first, then the replicas, so for 1–5ms some #rN copies have the old value. A replica miss (the copy hasn't been written yet, e.g. just after a key goes hot) falls through to the un-suffixed key. Alternative: split the key. Replicating fans out reads; splitting a key (sharding its value — e.g. a counter into k:part0..k:part9 summed at read time) is what you do when the key is write-hot (a counter incremented 50K times/sec), not read-hot. We're read-hot, so replicate.

Before:  "product:viral-sneaker" → 50,000 reads/sec on shard 42 → 52% util.
After R=10: 50,000 / 10 = 5,000 reads/sec per replica shard.
  shard 42 now: 2,000 (background) + 5,000 (its 1/10 share) = 7,000 reads/sec
  → utilization 7% — well within limits, p99 back to ~0.5ms,
    and the 20M innocent co-tenants are no longer collateral damage.

Step 4c — L1 in-process cache (the biggest lever)

The best defence against a hot key is to never send the request over the network. Each app server keeps a small in-process cache (~100MB ≈ 100K entries at 1KB) with a short TTL (1s for hot keys; a few seconds for warm ones). For the viral key at 50K reads/sec spread over 200 app servers, each server sees 250 reads/sec; with a 1s TTL, the first read each second misses L1 and hits L2, the other ~249 are L1 hits — a ~99.6% L1 hit rate. That collapses L2 traffic for the key to 200 reads/sec total (one miss/sec/server × 200 servers), a 250× reduction; combine with R=10 and each replica shard sees ~20 reads/sec — essentially zero. Memory cost: 200 servers × 100MB = 20GB fleet-wide, 0.2% of the 10TB L2 — the highest-ROI line in the whole design. The cost is bounded staleness: a hot-key value can be up to 1s old. Fine for view counts, product info, news, profiles; for inventory-during-flash-sale or anything that can't tolerate 1s, use a 100ms TTL or bypass L1 for those keys.

class ThreeLayerCache:
    def __init__(self):
        self.l1 = L1Cache(max_entries=100_000, default_ttl=1.0)   # in-process
        self.l2 = DistributedRedisCache(nodes=500)                # Iter 1–3
        self.hot_keys = set()                                     # refreshed every 10s
        self.inflight = {}                                        # key -> Future (single-flight)

    def get(self, key):
        v = self.l1.get(key)
        if v is not None:
            return v                                              # L1 hit (~0.01ms, no network)
        if key in self.hot_keys:
            v = self.l2.get(f"{key}#r{random.randrange(10)}") or self.l2.get(key)
        else:
            v = self.l2.get(key)                                  # ~1ms
        if v is not None:
            self.l1.put(key, v, ttl=1.0 if key in self.hot_keys else 5.0)
            return v
        # L1+L2 miss → single-flight: only the first caller hits the DB
        return self._single_flight(key)

    def _single_flight(self, key):                                # cite design-moves/single-flight.html
        if key in self.inflight:
            return self.inflight[key].result()                    # piggyback on the in-flight load
        fut = self.inflight[key] = Future()
        try:
            v = database.query(key)                               # ~50ms — the ONE DB hit
            if v is not None:
                self.l2.set(key, v, ttl=3600); self.l1.put(key, v, ttl=1.0)
            fut.set_result(v); return v
        finally:
            del self.inflight[key]

Step 4d — consistent hashing with bounded loads + single-flight (defense for the unlucky cases)

Two residual risks. (1) A key goes hot faster than 10s (an instantaneous spike from a bot or a DDoS) — before detection kicks in, one shard takes the hit. Consistent hashing with bounded loads (cite consistent-hashing.html, and the "power of two choices" idea) caps any node at (1+ε)× the mean load; when the owner of a key is over the cap, the client walks the ring to the next under-cap node — the hot key's overflow reads spill to neighbours without waiting for the coordinator. It's a blanket safety net, cheap (a per-node load counter the client samples), and it composes with replication (spillover targets are themselves bounded). (2) A cold key gets a million simultaneous misses — a freshly-evicted or freshly-written key, or one whose TTL just expired, with a million callers all racing to the DB. Single-flight (cite single-flight.html — the canonical move for this; we don't re-derive it) collapses those into one DB read: the first caller fetches and fills, every other caller waiting on that key piggybacks on the same in-flight future. This is the stampede / thundering-herd defence (cite failure-modes#stampede, sister scenario 27-thundering-herd.html); it caps DB miss-load at "distinct keys missing concurrently," not "callers missing concurrently."

Without single-flight: viral key's TTL expires → 50,000 callers in that second
  all miss → 50,000 DB queries for ONE key. With 1,000 hot keys all expiring
  in a burst → 50M DB queries/sec → DB falls over (failure-modes#stampede).
With single-flight: 1 DB query per distinct key in flight.
  1,000 hot keys → 1,000 DB queries total, regardless of caller count.
  Then L2 is filled, L1 is filled, the herd is served from cache.

Alternatives considered (C4) — the hot-key fix

  • Replicate the key (chosen, with L1 in front) — spreads read load; cheap; write fan-out is small. The lever that does the heavy lifting is the L1 cache; replication handles the residual that escapes L1.
  • Split the key — shard a key's value across N sub-keys (counter → k:part0..partN, summed at read). The right move when a key is write-hot; useless when it's read-hot (every read still touches all N parts). Out of scope for this read-heavy workload; flagged for the write-heavy variant.
  • Local-LRU-in-front (L1) only, no replication — L1 alone cuts the viral key's L2 traffic 250× (to ~200 reads/sec), which is already under a shard's limit. Replication is belt-and-suspenders for the case where L1 TTL is forced very short, or where there are many simultaneous hot keys. We keep both because they're cheap; if you must pick one, pick L1.
  • Consistent hashing with bounded loads (power of two choices) — a blanket cap on per-node load with ring-walk spillover. Doesn't need detection (it reacts in <1ms, not 10s) but it only spreads the overload to neighbours — if the spike is huge it just makes 3 nodes hot instead of 1. It's the fast-reacting net under the detection-based replication, not a replacement for it.
  • Single-flight / request coalescing — orthogonal: it kills the miss-side stampede on a cold key, where the others kill the hit-side overload on a hot key. Both are needed. Cite single-flight.html.

What breaks (C5; cite failure-modes.html)

  • Hot-key meltdown during the detection gap (failure-modes#hot-key-meltdown) — if a key goes from cold to 100K reads/sec in <1s (faster than the 10s detection cycle), L1 (1s TTL, blanket on all keys) still absorbs ~99% and bounded-loads spillover reacts immediately; the shard takes the residual for one report interval. The blast radius is one shard's other keys getting elevated p99 for ~10s; recovery is automatic when the key gets replicated.
  • L1 staleness bites a consistency-sensitive key (failure-modes#replica-divergence) — a key that genuinely can't tolerate 1s of staleness (inventory during a flash sale) read through L1 returns an oversell-causing stale count. Mitigation: a per-key "no-L1" / short-L1-TTL flag set by the owning service; the cache client honours it.
  • Write-hot key (bottlenecks#hot-partition) — a counter incremented 50K times/sec doesn't benefit from read-replication or L1; the writes still all hit one shard. Out of scope here (this is a read-heavy cache); the fix is key-splitting + eventual aggregation, flagged for the write-heavy sister.
  • Coordinator down — if the hot-key coordinator dies, the hot list goes stale; new hot keys aren't detected until it recovers. L1 + bounded-loads keep the system safe in the meantime (L1 doesn't need the coordinator). The coordinator is a small, restartable service; run two and let app servers consume whichever is up.

# --- Redis key shape (the only "schema" a flat KV cache has) ---
  ::            # e.g. product:12345, user:88f3a, config:flags
  TTL: per-key (default 3600s; shorter for fast-changing keys); allkeys-lru evicts under pressure

  Versioned-key variant (for keys that can't tolerate cache-aside's race): "#v"

# --- Cache-aside read/write (caller's logic; the cache provides GET/SET/DEL) ---
  read(k):  v = cache.GET(k); if v: return v
            v = single_flight(k, lambda: db.SELECT(k))         # one DB hit per key in flight
            cache.SET(k, v, ttl=3600); return v
  write(k): db.UPDATE(k, v); cache.DEL(k)                      # DEL not SET — idempotent, self-heals on race
            # for a hot key: also cache.DEL("k#r0".."k#r9")    # invalidate the replicas

# --- The consistent-hashing ring (client-side, control-plane-pushed) ---
  ring = sorted list of 100,000 (hash32, shard_id) pairs   # 500 shards × 200 vnodes
  shard_for(k): i = bisect_left(ring, hash32(k)); return ring[i % len(ring)].shard_id
  reads: pick the least-loaded of {primary, replica1, replica2}; writes: primary only
  bounded-loads (Iter 4): if load[shard_for(k)] > (1+ε)·mean_load: walk the ring to the next under-cap shard

Operations (C6)

  • SLIs / dashboards: per-shard hit rate, eviction rate (alert > 1% of reads sustained 10 min), p99 GET latency, ops/sec, memory used / maxmemory; fleet cache-hit rate (target > 95%); DB queries/sec attributable to misses (target < 50K); hot-key list size and the top-key reads/sec; L1 hit rate per app server; single-flight collapse ratio (callers served / DB queries). These map 1:1 to the C5 failure modes: hit-rate dip → eviction storm or imbalance; DB miss-spike → cold-start or stampede; one-shard p99 → hot key or rack-correlated loss.
  • Rollout: cache-tier changes (Redis version, maxmemory-policy) are canaried on one shard group, then 10%, then fleet; the smart-client library is rolled like any client dependency (the ring epoch protects against split-brain during the rollout). New count-min-sketch / hot-key thresholds are config-pushed and watched for false-positive hot keys (over-replicating).
  • Migration / scaling (8 → 16 shards, the classic): add the new shards to the ring with zero vnodes, then bleed vnodes from old shards to new ones in small steps, copying the affected key ranges from a replica before flipping ownership — so each step moves only ~1/N of one shard and the new shard is warm before it takes reads. Never re-hash the whole ring at once (that's a fleet-wide cold start). Bound the copy rate (~50K keys/sec/shard) so it doesn't impact live traffic. Ring epoch bumps once per step; clients refresh.
  • Headroom: +20% RAM headroom per shard above the ±1.4GB load tail (so an unlucky shard isn't permanently evicting); read-throughput is ~50× over-provisioned by construction so burst headroom is free there; the hot-key path has its own elastic headroom (L1 + replication scale with the spike).

Degradation contract

The rule: the cache is an optimization, not a dependency. If the entire cache tier is unavailable, every read becomes a DB read — so the DB must be sized to survive that: it absorbs 1M reads/sec at degraded latency (p99 ≈ 50ms instead of 1ms), shedding non-critical reads first if it can't. We accept reduced performance, not an outage. Concretely, by failure scope: one shard-group lost → 0.6% of keys fall through to the DB (~6K extra QPS) — invisible. One AZ lost → rack/AZ-aware placement means each shard still has a surviving replica; reads re-route, no fall-through. Whole cache cold (deploy flush / region restart) → warm-up-from-replica + rate-limited miss path + single-flight keep DB QPS bounded; hit rate climbs back to >95% within minutes (the Zipf head fills in ~30 min from organic traffic, faster with background copy). Hot-key coordinator down → L1 + bounded-loads carry the load; degraded only in that new hot keys aren't auto-replicated until it recovers. The one place this contract changes: if the DB cannot survive a cold cache (e.g. it's already at capacity), the cache becomes load-bearing — then you add AOF persistence, warm-standby shards, and tighten the cache's own availability SLO toward 99.999%. State which world you're in; don't leave it implicit.

Cost (C7; cite cost-shapes#cache-heavy + pricing-constants.html)

Cache-heavy cost shape: it's dominated by RAM-hours — you're paying for memory, not CPU or I/O. 501 self-managed 64GB memory-optimized instances — an r6i.2xlarge-class node (half of r6i.4xlarge's $1.01/hr ≈ ~$0.50/hr, cite pricing-constants#compute): 501 × $0.50 × 24 × 30 ≈ ~$180k/mo for the cache tier, plus ~20GB of L1 RAM that's already in the app servers (rounding error) and a tiny coordinator. 1 sig fig: ~$200k/mo. (Managed ElastiCache on cache.r6g.4xlarge @ $1.36/hr would be ~$500k/mo — the ~30–50% managed premium times the node count; we self-manage.) The cost-vs-availability knob: 3× replication is 2/3 of the bill — dropping to 2× saves ~$60k/mo but turns "one bad node from a blind spot" into reality during deploys; that's the trade. (Bigger nodes lower instance count but raise blast radius per failure; we keep 64GB.) The hot-key machinery (Iter 4) costs essentially nothing — 100MB of sketches fleet-wide and L1 RAM that already exists — which is exactly why it's the highest-ROI line.

Real-system comparison

This is roughly Facebook's memcache architecture (the NSDI 2013 paper, "Scaling Memcache at Facebook"): a huge look-aside fleet, leases/single-flight to kill stampedes, regional pools, and mcrouter doing the client-side routing and per-key replication that we fold into the smart client. The hot-key + L1 layer mirrors Netflix's EVCache (in-process L1 + remote L2) and the general "L1/L2 cache" pattern. The consistent-hashing-with-bounded-loads idea is Google's "Consistent Hashing with Bounded Loads" (2017), used in Vimeo's Maglev-style load balancer and YouTube. The cache-aside-DELETE-not-SET race is the one Facebook's paper calls out as the reason for leases. Anticipated pushbacks

What if the working set is 10× bigger — 100TB? (scale flip)

Storage was the only binding bottleneck, so 100TB just multiplies shard count: ~1,670 primary shards × 3 ≈ 5,000 instances, ~$2M/mo. The smart client now opens 5,000 connections — this is where X5 subsetting becomes mandatory (each client talks to a deterministic subset, say 500, of the 5,000). Read throughput is still over-provisioned. Eviction monitoring matters more (more nodes → more eviction events). Nothing structural changes; the cost shape (RAM-hours) is the same, just larger. The thing to revisit: is 100TB really the working set, or are you trying to cache the whole DB? If the latter, you don't have a cache problem, you have an "the DB should be faster" problem.

Why not just one big Redis? (alt-topology)

The biggest single instance you can buy is ~1–2TB RAM (x2gd-class); 10TB doesn't fit, full stop — that's the LC-kernel "one node's RAM ≪ the working set" property. Even if it fit: one node is one failure domain (a crash blacks out 100% of the cache, a cold start thunders the DB), one throughput ceiling (~100K ops/sec for one Redis vs 50M across the fleet), and one hot-key victim (every hot key on one box). Sharding is forced by capacity and bought you availability, throughput, and blast-radius isolation as a bonus.

What changes if the cache must be strongly consistent with the DB? (consistency flip)

Then it's no longer a cache in the usual sense. Options, worst-to-best: (a) read-through with synchronous write-through and sync replication — every read sees the latest write, but you've paid DB-grade write latency and lost replica read fan-out (back to the strong-consistency row of Iter 2); (b) versioned keys — the writer bumps a monotonic version on the key, readers always read the current version, so a stale L1 entry is just a miss on the new version — this gets you read-your-writes without sync replication, at the cost of a version registry; (c) invalidation-via-CDC — tail the DB's binlog and DEL changed keys, so the staleness window is the CDC lag (sub-second) not the TTL. For most "I think I need strong consistency" cases, (b) or (c) is enough — true linearizable cache reads are rarely the actual requirement. Also: L1 must go (1s staleness is incompatible with strong consistency) or be CDC-invalidated too.

A shard's primary dies right now — walk me through it. (failure)

T+0: primary stops responding. T+0–5s: clients fail GETs to it, retry against a replica (read traffic uninterrupted); writes to that range pause. T+5s: 3 missed heartbeats → the cluster promotes replica-1 to primary; ring epoch bumps; clients refresh and resume writes to the new primary. T+5–60s: a replacement node is provisioned, joins with 0 vnodes. T+60s–7min: it copies the shard's key range from a replica (throttled ~50K keys/sec) before taking read traffic; then it becomes replica-2 and the shard is back to 3×. Blast radius: zero data loss, ~5s of write-pause on ~0.6% of keys, no read impact. The thing that would make this worse: all 3 replicas in the same rack — hence rack-aware placement (Iter 1 What-Breaks).

The hot-key meltdown — walk me through it. (failure)

Before Iter 4: celebrity tweets, one key → 50K reads/sec, all to shard 42. Shard 42 hits 52% util, p99 climbs to 15–30ms; its 20M other keys get that p99 too (collateral damage). App threads waiting on shard 42 pile up, thread pools fill, requests time out, timeouts retry, retries double the load on shard 42 → death spiral; eventually shard 42's misses spike DB load too. With Iter 4: L1 (1s TTL, blanket) already absorbs ~99.6% before the key is even flagged, so shard 42 sees ~250 reads/sec for the key, not 50K — no meltdown. By T+20s the key is replicated ×10 and L1-aware, and shard 42 sees ~20 reads/sec. Bounded-loads is the instant net if the spike outpaces the 10s detection. Detection: per-shard p99 alert + the hot-key list size; recovery: automatic (replication + L1 + the key dropping off the list when the event ends).

Where does the bill come from? (cost)

RAM-hours, ~$200k/mo for the 501-instance cache tier — cache-heavy cost shape, you're buying memory. The dominant line within that is the 3× replication factor: 2/3 of the spend exists for availability, not capacity. The cost-vs-availability knob is exactly that 3× → 2× (saves ~$60k/mo, costs you a deploy-time blind spot). Bigger nodes lower instance count but raise per-failure blast radius — not free. Managed (ElastiCache) would be ~$500k/mo — the managed premium times node count. The L1 / hot-key machinery is ~$0 (RAM that's already in app servers + 100MB of sketches) — the cheapest, highest-ROI part of the design.

How do you grow from 8 to 16 shards without a stampede? (migration)

Never re-hash the whole ring at once — that flushes everything and is a fleet-wide cold start. Instead: add the 8 new shards to the ring with 0 vnodes (they own nothing yet); then, in small steps, move vnodes from old shards to new ones — for each vnode moved, first background-copy that key range from a replica into the new shard (throttled), then flip ownership and bump the ring epoch. Each step migrates ~1/16 of one shard and the new shard is already warm when it starts taking reads, so DB miss-load barely moves. Clients refresh the ring on each epoch bump. Total migration is many small, warm, reversible steps — no cliff. (This is the same Operations#migration procedure as a node replacement, just repeated across many ranges.)

Cache hit rate dropped — where do you look? (monitoring/debug)

In order: (1) per-shard eviction rate — if one shard is evicting hard, it's an imbalance (an unlucky vnode allocation) or that shard is undersized; if all shards are evicting, the working set outgrew the cluster → auto-scale. (2) Recent deploys — a cache flush or a serialization-format change that invalidated keys. (3) TTL changes — someone shortened a default TTL. (4) Upstream — the app changed its key naming (now everything's a "miss" on the new keys) or its access pattern (a batch job scanning cold keys, polluting LRU — this is the scan-flush case W-TinyLFU would fix). (5) A node that's up but slow (black-hole-ish) — reads to it time out and look like misses. The SLI that disambiguates: eviction rate (capacity), key-cardinality drift (naming), and per-shard p99 (a sick node) — all three are on the C6 dashboard.

A cold key gets a million simultaneous misses — the stampede. (failure)

Without single-flight: a key's TTL expires (or it was just evicted/written), and in that instant a million callers all GET it, all miss, all query the DB — a million DB queries for one key, repeated for every popular key whose TTL lapses in that burst. The DB falls over (cite failure-modes#stampede; sister #27). With single-flight (Iter 4): the first caller to miss a key registers an in-flight future and does the one DB read; every other caller for that key in that window awaits the same future; when it resolves, it fills L2 and L1 and everyone is served from cache. DB miss-load is bounded by "distinct keys missing concurrently" (thousands) not "callers missing" (millions). Belt-and-suspenders: stagger TTLs with jitter so popular keys don't all expire at the same instant, and serve-stale-on-error if the DB read fails (return the just-expired value rather than nothing). The canonical move and its variants are in single-flight.html — we cite it rather than re-deriving.

Key Takeaways

1. Storage, not throughput, usually determines cache cluster size

We needed 171 instances for storage but only 10 for read throughput. Most distributed cache designs are storage-bound. Always do the BOE math for all three dimensions (storage, reads, writes) and let the largest one drive your instance count.

2. Replication gives you both availability and read throughput

3× replication costs 3× the storage, but it eliminates single-node-failure data loss, enables instant failover, and triples your per-key read capacity. For a cache, this is almost always worth it.

3. Eventual consistency is almost always right for caches

A cache is not the source of truth — the database is. If a read returns a value that is 5ms behind the latest write, the impact is negligible. Strong consistency (reading only from the primary) wastes replica capacity and increases latency for no meaningful benefit in most cache use cases.

4. Virtual nodes solve the uneven distribution problem

200 virtual nodes per physical node gives load variance under 8%. The memory cost is trivial (1.2MB for 100K entries). Always use virtual nodes in production consistent hashing deployments.

5. LRU eviction with monitoring is your safety valve

LRU eviction ensures the cache never runs out of memory. But eviction rate is a critical metric: if evictions are high, the cache is undersized and you are sending too many requests to the database. Monitor eviction rate and auto-scale when it exceeds thresholds. (W-TinyLFU is the upgrade if a hit-rate audit shows scan-flush.)

6. Virtual nodes balance key counts; they do nothing for traffic skew

A hot key lives on one shard no matter how evenly the ring is balanced. Fix the hot key separately: detect it (count-min sketch, O(1) per GET), replicate it across shards (random suffix), and — the biggest lever — put a small in-process L1 cache in front of it (250× fewer L2 reads for ~20GB fleet-wide). Bounded-loads is the instant net; single-flight handles the inverse problem (a stampede on a cold key). See Iteration 4 (which absorbed the former separate scenario).

7. State which world you're in: optimization or dependency

"The cache is an optimization, not a dependency" means the DB is sized to survive a cold cache. If it can't be, the cache is load-bearing — add persistence, warm-standby, and a tighter availability SLO. The degradation contract is the most important paragraph on the page; don't leave it implicit.

Assumptions and When They Break

  • Working-set-resident: If 10TB is the whole DB rather than the hot set, hit rate falls and you're solving the wrong problem.
  • Single data center: Cross-region caching adds replication lag (50–100ms) and needs geo-aware routing (a probe).
  • 1KB average value: If values are 100KB blobs, NIC bandwidth binds before RAM.
  • Read-heavy (10:1): At 1:1, stale reads become frequent; at write-hot, key-splitting (not replication or L1) is the fix.
  • The DB survives a cold cache: If not, the cache is load-bearing — see the degradation contract.
