LLM Inference Serving — Interview Ready

Concise, logical, derivation-complete. Every number traces to a formula. Every technique traces to a bottleneck.

~8 min read • Full walkthrough with detailed explanations

1. Setup

The Problem

Serve Llama-3 70B to 2M DAU with streaming output. Each request: ~1K input tokens, ~500 output tokens, up to 32K context.

Peak QPS:    2,778 req/sec (80M msgs/day, 3× peak)
SLOs:        TTFT < 1s, inter-token latency < 50ms
GPU:         H100 80GB (3,350 GB/s HBM, 989 TFLOP/s FP16)
MFU:         0.4 (40% of peak, conservative; real range 30-60%)
Effective compute per GPU: 989 × 0.4 = 395.6 TFLOP/s

"Node" = 1 physical server = all GPUs for 1 model replica (via TP).
  Naive (TP=2): node = 2 GPUs.  Iter 0+ (TP=4): node = 4 GPUs.
  GPUs in a node connected via NVLink (900 GB/s).
  Node ≠ GPU. Node = TP GPUs = 1 model copy.

Key Formulas

Architecture constants (Llama-3 70B):
  layers = 80             d_model = 8,192
  head_dim = 128          d_ff = 28,672 (FFN intermediate dimension:
                           d_model → expand to d_ff → compress back to d_model.
                           ≈3.5× d_model. Larger = more learning capacity.)
  num_q_heads = 64        num_kv_heads = 8 (GQA: 8 Q heads per KV head)
  vocab_size = 128,256

  Token embedding: [1 × d_model] = [1 × 8,192] (NOT [1 × head_dim])
  After W_Q projection: [1 × 8,192] (same size, different values)
  Reshape into heads:   [head 0: 128] [head 1: 128] ... [head 63: 128]
    head_dim = 128 = how many numbers each head works with per token.
    64 heads × 128 dims = 8,192 = d_model (adds back up).

Deriving 70B params from architecture:
  Formula: total_params = layers × (attn_params + ffn_params) + embedding_params

  Attention per layer:
    W_Q: d_model × num_q_heads × head_dim  = 8,192 × 8,192 = 67M
    W_K: d_model × num_kv_heads × head_dim = 8,192 × 1,024 = 8.4M
    W_V: d_model × num_kv_heads × head_dim = 8,192 × 1,024 = 8.4M
    W_O: num_q_heads × head_dim × d_model  = 8,192 × 8,192 = 67M
    attn_params = 151M per layer

  FFN per layer (SwiGLU: 3 weight matrices):
    W_up:   d_model × d_ff = 8,192 × 28,672 = 235M
    W_gate: d_model × d_ff = 8,192 × 28,672 = 235M
    W_down: d_ff × d_model = 28,672 × 8,192 = 235M
    ffn_params = 705M per layer

  Per layer: 151M + 705M = 856M
  80 layers: 856M × 80 = 68.5B
  Embeddings: vocab_size × d_model = 128,256 × 8,192 = 1.05B
  Total: 68.5 + 1.05 ≈ 69.5B ≈ 70B ✓

Model size: 70B × 2B (FP16) = 140 GB  |  70B × 1B (INT8) = 70 GB

Per-GPU (TP=4):
  Weights:    140/4 = 35 GB (FP16)  |  70/4 = 17.5 GB (INT8)
  Available:  80 - 35 - 10(overhead) = 35 GB for KV (FP16)
            | 80 - 17.5 - 10 = 52.5 GB for KV (INT8)
    Each Q head sees a 128-number slice of the 8,192-number token.

Attention params per layer:
  Formula: attn_params = (num_q_heads + 2×num_kv_heads + num_q_heads) × head_dim × d_model
    W_Q: d_model × (num_q_heads × head_dim)  = 8,192 × 8,192 = 67M  (Q projection)
    W_K: d_model × (num_kv_heads × head_dim) = 8,192 × 1,024 = 8.4M (K projection)
    W_V: d_model × (num_kv_heads × head_dim) = 8,192 × 1,024 = 8.4M (V projection)
    W_O: (num_q_heads × head_dim) × d_model  = 8,192 × 8,192 = 67M  (output projection)
    Total per layer: 67 + 8.4 + 8.4 + 67 = 151M
    All 80 layers: 151M × 80 = 12.1B (out of 70B total)
  GQA savings: KV params = 16.8M/layer (vs 134M without GQA = 8× less)
    → 8× smaller KV cache, 8× less KV bandwidth

Formula: KV_per_token = 2(K+V) × layers × num_kv_heads × head_dim × 2B(FP16)
KV per token: 2 × 80 × 8 × 128 × 2 = 327,680 bytes ≈ 320 KB total
  (this is the K,V output size from W_K and W_V above, stored per token)
  Per GPU (TP=4): 320 KB / 4 = 80 KB  (KV heads split across GPUs)
  Per request (1,500 avg tokens): 80 KB × 1,500 = 120 MB = 0.12 GB
    (1,500 = 1,000 input + 500 generated. Cache stores ALL tokens:
     input K,V from prefill + generated K,V from decode.)
  Per request (32K max context):  80 KB × 32,768 = 2.56 GB
    (32K = max context window, from scenario constraints)

Formula: weight_matmul_FLOPs = 2 × params × tokens_processed
  (each of the 70B params does 1 multiply + 1 add = 2 FLOPs per token.
   All params used once per token across 80 layers' 7 matmuls.)
Formula: attn_FLOPs = 4 × head_dim × c × num_q_heads × layers × batch
  c = cached tokens per request (all tokens in KV cache so far:
      input tokens from prefill + previously generated tokens.
      Avg c ≈ 1,500 for our workload: 1,000 input + 500 output.)
  (scores q×K^T + weighted sum w×V; this is the attention COMPUTATION,
   separate from the Q,K,V,O weight matmuls counted in weight_matmul_FLOPs)
  (scores q×K^T + weighted sum w×V; small at low batch, grows with c)

Decode step time = max(Load, Compute)
  Load    = (weights_per_GPU + batch × KV_per_req_per_GPU) / 3,350 GB/s
          = (35 + batch×0.12) / 3,350
  Compute = (weight_matmul_FLOPs + attn_FLOPs) / TP / 395.6 TFLOP/s

Prefill time  = max(Load, Compute)  [compute-bound: many tokens]
Decode step   = max(Load, Compute)  [load-bound: 1 token]
Latency       = prefill + output_tokens × decode_step_time

2. Bottleneck → Fix → Re-derive

Start with the dumbest possible design. Identify one bottleneck. Fix it with one technique. Re-derive all numbers. Repeat.

Naive: No KV Cache

Bottleneck: RECOMPUTATION. Every decode step reprocesses ALL tokens.
  Token 500: reprocess 1,499 tokens just to produce 1 new token.
  Total: 375,000 redundant K,V projections for 500 output tokens.

Why: Without caching, the model has no memory of previous tokens.
  Each step runs a full forward pass over the entire sequence.

No prefill/decode split here — without KV cache, every step
reprocesses ALL tokens (1K input + previously generated).
  Step 1:   1,000 tokens.  Step 500: 1,499 tokens.
  Average:  ~1,250 tokens per step.

Numbers (TP=2, 2 GPUs per node):
  Formula: Load    = model_size / (TP × 3,350 GB/s)
  Formula: Compute = 2 × params × tokens / (TP × 395.6 TFLOP/s)
  Formula: Step time = max(Load, Compute)

  Load:    140 GB / (2 × 3,350) = 140 / 6.7 = 20.9 ms
  Compute: 2×70B×~1,250 / (2 × 395.6T) = 221 ms
  Step = max(20.9, 221) = 221 ms (compute-bound: many tokens per step)

  Formula: Latency = output_tokens × step_time  (no separate prefill)
  Per request: 500 steps × 221 ms = ~110 sec
    (each of the 500 steps already reprocesses all 1,000 input tokens
     — that's why avg tokens/step is ~1,250, not 1. The input "prefill"
     cost is baked into every step. That's the waste KV cache fixes.)

  Formula: Throughput = batch_per_node / latency_per_node
  Throughput: 1/110 = 0.009 req/sec/node  (batch=1, no batching benefit)

  Formula: Nodes = peak_QPS / throughput_per_node
  Fleet: 2,778 / 0.009 = 309,000 nodes = 618,000 GPUs

  Formula: Cost = GPUs × $2/hr × 24 × 30
  Cost: 618,000 × $2 × 720 = $890M/month

Iter 0: KV Cache + Static Batching

Bottleneck solved: RECOMPUTATION
  Cache K,V after prefill. Each decode step: compute K,V
  for ONLY the 1 new token, attend over all cached K,V.
  375,000 projections → 500. (750× fewer)

New regime: decode is now LOAD-BOUND
  Formula: Prefill = max(Load, Compute) + N² HBM overhead
  Prefill (1K tokens):
    Load:    model_size / (TP × 3,350) = 140 / (4×3.35T) = 140 / 13.4 = 10.4 ms
    Compute: 2 × params × tokens / (TP × 395.6T) = 2×70B×1000 / 1,582T = 88 ms
    N² HBM: 16 q_heads × 80 layers × 4 passes × (1,000²×2B = 2 MB)
            = 10 GB / 3,350 GB/s = 3 ms (negligible at 1K tokens)
    Prefill ≈ 88 ms (compute dominates; N² overhead only 3 ms)
    (At 32K tokens this overhead becomes 3.3 sec — fixed in Iter 3)

  Formula: Decode step time = max(Load, Compute)  [load-bound for 1 token]
  Decode (1 token, batch=16):
    Load:    (weights_per_GPU + batch × KV_per_req_per_GPU) / BW
             = (35 + 16×0.12) / 3,350 = 36.9 / 3,350 = 11.0 ms
             (weights 35 GB + KV reads 1.9 GB. KV reads are small at batch=16
              but grow significant at higher batch — see Iter 2.)
    Formula: Compute = (weight_matmul_FLOPs + attn_FLOPs) / TP / 395.6T
      (Why split into two terms now? Without KV cache, every step processes
       ALL tokens through everything → one formula: 2×params×tokens.
       With KV cache, only NEW tokens go through weight matrices, but
       attention reads ALL CACHED tokens. They scale differently:
         weight_matmuls: scales with batch (new tokens only)
         attn_FLOPs:     scales with batch × c (new × cached)
       So they must be computed separately.)
    Compute:
      weight_matmuls = 2 × 70B × 16 = 2,240 GFLOP
      attn_FLOPs     = 4 × 128 × 1,500 × 64 × 80 × 16 = 62.9 GFLOP
      total = (2,240 + 62.9) / 4 GPUs / 395.6T = 1.5 ms
    Decode step = max(11.0, 1.5) = 11.0 ms (load-bound)

Why TP=4? KV cache eats memory.
  Formula: available_for_KV = 80 - model_size/TP - overhead
  TP=2: 80 - 70 - 10 = 0 GB free. Unusable.
  TP=4: 80 - 35 - 10 = 35 GB free.

  Formula: max_batch = available_for_KV_per_GPU / KV_per_req_per_GPU
  (With TP, all GPUs process the SAME batch — each holds its 1/4
   KV shard for ALL requests. So batch_per_GPU = batch_per_node.)
  Without PagedAttention: pre-allocate for 32K max.
    KV_per_req_per_GPU = 320 KB × 32K / 4 = 2.56 GB
    max_batch = 35 / 2.56 ≈ 13 → use batch=16.

  Formula: Latency = prefill + output_tokens × decode_step_time
  Latency: 88 + 499×11.0 = 5.6 sec/req (was 110 sec)

Static batching: wait for slowest in batch.
  Avg longest output ≈ 1,000 tokens.
  Batch time: 88 + 1,000×11.0 = 11.1 sec

  Formula: Throughput = batch_per_node / latency_per_node
  Throughput: 16 / 11.1 = 1.4 req/sec/node

  Formula: Nodes = peak_QPS / throughput; Cost = GPUs × $2 × 720
  Fleet: 2,778/1.4 = 1,984 nodes = 7,936 GPUs
  Cost: 7,936 × $2 × 720 = $11.4M/month

Iter 1: Continuous Batching

Bottleneck solved: STATIC BATCHING PADDING WASTE
  Static: all 16 sequences must wait for the longest.
  If longest = 2,000 tokens, 15 others (avg 500) wait idle.

Fix: after each decode step, check which sequences finished.
  Evict done sequences, admit new ones immediately.
  Each sequence uses its ACTUAL length, not the longest.

  Formula: Throughput = batch_per_node / avg_latency_per_node (not longest)
  Throughput: 16 / 5.6 = 2.9 req/sec/node (2× vs static's 1.4)
    (5.6 sec = average request time: 88 + 499×11.0 ms)

  Formula: Nodes = peak_QPS / throughput; Cost = GPUs × $2 × 720
  Fleet: 2,778/2.9 = 958 nodes = 3,832 GPUs
  Cost: 3,832 × $2 × 720 = $5.5M/month

Iter 2: PagedAttention

Bottleneck solved: MEMORY FRAGMENTATION
  Each request pre-allocates contiguous KV for 32K max context.
  Actual avg: 1,500 tokens (480 MB) vs allocated: 10.24 GB.
  Waste: 95%. Only 16 sequences fit despite plenty of memory.

Fix: borrow OS paging. Divide KV into fixed-size blocks per GPU.
  Block size: 16 tokens per block (standard vLLM default, like OS page size)
  Formula: block_size_bytes = tokens_per_block × KV_per_token_per_GPU
    = 16 × 80 KB = 1.28 MB per block per GPU
  Allocate blocks on demand. Return blocks when sequence finishes.
  Formula: KV_per_req_per_GPU = KV_per_token_per_GPU × avg_tokens
    = 80 KB × 1,500 = 120 MB = 0.12 GB
  1,500-token req: ceil(1500/16)=94 blocks × 1.28 MB = 120 MB per GPU.

  Formula: max_batch = available_for_KV / KV_per_req_per_GPU
  max_batch = (80 - 35 - 10) / 0.12 = 291 (theoretical)
  Practical: ~64 (headroom for variable lengths, safety margin)

  Formula: Decode step time = max(Load, Compute)
  At batch=64 (KV reads now significant):
    Load:    (weights_per_GPU + batch×KV_per_req_per_GPU) / BW = (35+64×0.12)/3,350 = 12.7 ms
    Compute: (2×params×batch + attn_FLOPs) / TP / 395.6T = 5.9 ms
      attn_FLOPs = 4 × d_head × c × num_q_heads × layers × batch
                 = 4 × 128 × 1,500 × 64 × 80 × 64 = 252 GFLOP
      weight_matmuls = 2 × 70B × 64 = 8,960 GFLOP
      total = (8,960 + 252) / 4 GPUs / 395.6T = 5.9 ms
    Decode step = max(12.7, 5.9) = 12.7 ms (still load-bound)

  Formula: Latency = prefill + output_tokens × decode_step_time
  Latency: 88 + 499×12.7 = 6.4 sec/req

  Formula: Throughput = batch_per_node / latency_per_node; Nodes = QPS / throughput
  Throughput: 64/6.4 = 10 req/sec/node
  Fleet: 2,778/10 = 278 nodes = 1,112 GPUs
  Cost: 1,112 × $2 × 720 = $1.6M/month

Iter 3: Flash Attention + Chunked Prefill

Bottleneck solved: LONG PREFILL BLOCKS ALL DECODE USERS
  A long-context prefill monopolizes compute in the shared forward
  pass, spiking inter-token latency for all other batched requests.
  (Flash Attention reduces the prefill cost; Chunked Prefill prevents
   it from blocking decode users even when the cost is still large.)
  N² HBM overhead only affects PREFILL (not decode):
    Prefill: Q=[N × d_head], so Q×K^T = [N × N] matrix → huge at large N.
    Decode:  q=[1 × d_head], so q×K^T = [1 × c] vector → no N² problem.
  At 1K tokens: 1,000²×2B = 2 MB per head (tiny, not a bottleneck).
  At 32K tokens: 32,768²×2B = 2.15 GB per head (huge!).
  Standard attention bounces this 4× through HBM:
  HBM overhead: 16 q_heads_per_GPU × 80 layers × 4 passes × 2.15 GB
    = 11,008 GB per GPU / 3,350 GB/s = 3.3 sec.
  During prefill: ALL decode users blocked (shared forward pass).

Flash Attention: tile into [128×128] blocks in SRAM (50 MB).
  Online softmax: scores computed and consumed in SRAM,
  never written to HBM. HBM I/O: O(N²) → O(N).
  Prefill 32K tokens (max context, from scenario constraints):
    Formula: weight_matmul_FLOPs = 2 × params × tokens
    weight matmuls: 2×70B×32K / 1,582T = 2.9s

    Formula: attn_FLOPs = 4 × d_head × tokens × num_q_heads × layers
    attn compute: 4×128×32K×64×80 / 1,582T = 84T / 1,582T = 53 ms

    Formula: N² HBM overhead = q_heads_per_GPU × layers × passes × N²×2B / BW
    N² HBM overhead per GPU:
      N² matrix size: 32,768² × 2B = 2.15 GB per head
      q_heads_per_GPU = num_q_heads / TP = 64 heads / 4 GPUs = 16
      (NOT batch_size! "64" here = query heads, not requests.)
      16 q_heads × 80 layers × 4 HBM passes × 2.15 GB = 11,008 GB
      11,008 GB / 3,350 GB/s = 3.3s

    Standard: 2.9s + 3.3s (N² HBM) + 53ms (attn compute) = 6.2s
    Flash:    2.9s + 53ms (attn compute only, N² HBM eliminated) = 3.0s
    Speedup: ~2×

  Chunked Prefill: derive chunk_size from 50 ms SLO.
    Split INPUT TOKENS into chunks (not the model). Each chunk still
    goes through ALL 80 layers (the full model). "2×70B×tokens"
    already includes all layers (70B = 856M/layer × 80 layers).

    Each forward pass: 1 prefill chunk + all decode tokens, through full model.
    Formula: Prefill_chunk_step = max(Load, Prefill_chunk_compute)
      Load = weights_per_GPU / BW = 35 / 3,350 = 10.4 ms (no KV reads for prefill tokens)
      Prefill_chunk_compute = 2×params×(chunk_size + decode_reqs) / 1,582T

    Constraint: Prefill_chunk_step < 50 ms (inter-token latency SLO, from scenario)
    Total tokens per step = chunk_size + 63 (decode reqs, batch=64 minus 1)
    Formula: Prefill_chunk_compute = 2×70B×(chunk_size + 63) / 1,582T < 50 ms
      Solve: chunk_size + 63 < 50 × 1,582T / (2×70B) = 50 × 1,582 / 140 = 565
      chunk_size < 565 - 63 = 502
    chunk_size=512 → 575 tokens → 51 ms (barely exceeds SLO!)
    chunk_size=256 → 319 tokens → 28 ms (ok)
    chunk_size=128 → 191 tokens → 17 ms (comfortable margin)
    We use chunk_size=128 (17 ms, well within 50 ms SLO).

  Verify: Prefill_chunk_step = max(10.4, 17) = 17 ms < 50 ms ✓

  Formula: chunk_count = ceil(input_tokens / chunk_size)
  Tradeoff: smaller chunk_size = better inter-token latency for
    EXISTING decode users, but worse TTFT for the NEW prefilling user
    (more chunks → more steps to finish prefill):
      chunk_size=256: ceil(32K/256)=128 chunks × 28ms = 3.6s TTFT
      chunk_size=128: ceil(32K/128)=256 chunks × 17ms = 4.4s TTFT
      chunk_size=64:  ceil(32K/64) =512 chunks × 10ms = 5.1s TTFT
    (vs unchunked: 3.0s TTFT but 3.0s stall for everyone else)
  No more multi-second stalls.

Throughput: unchanged (~10 req/sec/node). This is a LATENCY fix.
  p99 inter-token: >1s → <50ms.

Iter 4: INT8 Quantization

Bottleneck solved: WEIGHT LOAD TIME
  Decode is load-bound. 140 GB weights loaded every step.
  Must load → must wait for HBM bandwidth.

Fix: INT8 (1 byte/param). 140 GB → 70 GB. Dequantize on-the-fly.
  Quality: <1% perplexity increase.

Triple win:
  Formula: Load = model_size / (TP × 3,350 GB/s)
  1. Decode 2× faster: 70 / 13.4 = 5.2 ms (was 140/13.4 = 10.4)

  Formula: available_for_KV = 80 - model_size/TP - overhead
  2. More KV budget: 80 - 70/4 - 10 = 52.5 GB free (was 35)
     Formula: max_batch = available / KV_per_req_per_GPU
     batch=128 fits: 128×0.12 = 15.4 GB << 52.5 GB

  Formula: Prefill = max(Load, Compute) [compute-bound, unchanged]
  3. Prefill unchanged: max(5.2, 88) = 88 ms
     (quantization halves Load, but Compute was the bottleneck)

  Formula: Latency = prefill + output_tokens × decode_step_time
  Latency: 88 + 499×5.2 = 2.7 sec/req

  Formula: Throughput = batch_per_node / latency_per_node; Nodes = QPS / throughput
  Throughput: 128/2.7 = 47 req/sec/node
  Fleet: 2,778/47 = 59 nodes = 236 GPUs
  Cost: 236 × $2 × 720 = $340K/month

Iter 5: Speculative Decoding

Bottleneck solved: 1-TOKEN-PER-STEP WALL
  Each 70B forward pass = 1 token. Can't go below 5.2 ms/tok.

Fix: draft model (8B INT8) generates K=5 candidates fast.
  Target model (70B) verifies ALL 5 in one forward pass.
  Accept/reject via modified rejection sampling (lossless).

  Formula: Iteration cost = K × draft_step + 1 × target_step
  Formula: Load = model_size / (TP × 3,350 GB/s)
  Draft model: 8B params × 1 byte (INT8) = 8 GB
  Draft step:  8 GB / 4 GPUs / 3,350 GB/s = 0.6 ms
  Draft:  5 × 0.6 = 3.0 ms
  Verify: 1 forward pass of 70B target model = 5.2 ms
    Processes all 5 drafted tokens at once (like a mini-prefill).
    Produces logits at each position to verify draft correctness.
    Formula: Verify = max(Load, Compute)
    Load:    70 GB / 13.4 TB/s = 5.2 ms (same 70 GB weights regardless of token count)
    Compute: 2×70B×5 / 1,582T = 0.44 ms (5 tokens is tiny)
    Verify = max(5.2, 0.44) = 5.2 ms (load-bound: same cost for 1 or 5 tokens)
  Total: 3.0 + 5.2 = 8.2 ms per iteration

  Formula: E[tokens per iteration] = 1 / (1 - α)
  E = 1/(1-0.75) = 4.0 tokens/iteration (α=0.75 typical)

  Formula: effective_ms_per_tok = iteration_cost / tokens_per_iteration
  Effective: 8.2 / 4.0 = 2.05 ms/tok (2.5× speedup vs 5.2)

  Formula: num_iterations = output_tokens / tokens_per_iteration
  num_iterations = 500 / 4.0 = 125

  Formula: Latency = prefill + num_iterations × iteration_cost
  Latency: 88 (prefill) + 125 × 8.2 (decode) = 1,113 ms ≈ 1.1 sec/req
    (prefill still required: process 1K input tokens to build KV cache.
     Speculative decoding only speeds up decode, not prefill — the input
     tokens are given, not generated, so there's nothing to "draft.")

  Formula: Throughput = batch_per_node / latency_per_node; Nodes = QPS / throughput
  Throughput: 128/1.1 = 116 req/sec/node
  Fleet: 2,778/116 = 24 nodes
    Target GPUs: 24 nodes × 4 GPUs/node (TP=4) = 96 GPUs
    Draft GPUs:  24 nodes × 1 GPU/node (8B fits on 1 GPU) = 24 GPUs
    Total: 96 + 24 = 120 GPUs
  Cost: 120 × $2 × 720 = $173K/month

3. Final Design Summary

Architecture

Client → Load Balancer → Request Queue (admission control)
  → Scheduler (continuous batching + chunked prefill)
    → 24 nodes (4×H100 each, TP=4, INT8 weights)
        Per node:
          PagedAttention KV manager (1.28 MB blocks, allocate on demand)
          Flash Attention 2 (tiled SRAM, O(N) HBM)
          Speculative decode: Llama-3 8B draft (K=5) → 70B verify → accept/reject
    → 24 draft GPUs (1×H100 each, Llama-3 8B INT8 = 8 GB, fits on 1 GPU)
  → Streaming output (SSE, token-by-token)

The One Insight

LLM decode is MEMORY-BANDWIDTH-BOUND, not compute-bound.

Every optimization targets one of three strategies:
  1. Reduce bytes loaded per step    (INT8 quantization)
  2. Amortize load across more work  (batching, paging, flash attention)
  3. Produce more tokens per load    (speculative decoding)

The GPU has 989 TFLOP/s of compute but only 3.35 TB/s of bandwidth.
Decode uses <1% of compute capacity. The bottleneck is moving data.
