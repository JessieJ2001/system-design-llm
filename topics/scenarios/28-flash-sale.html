Flash Sale: 10x Traffic Spike with Limited Inventory

100,000 requests per second, 100 limited-quantity items, 10 million eager buyers. The system must not crash, must not oversell, and must be fair. Walk through the architecture that makes it work.

NALSD Walkthrough • ~15 min read

The Scenario

You run an e-commerce platform. Marketing has announced a flash sale starting tomorrow at noon: 100 products at deep discounts, each limited to 1,000 units. The sale runs for 30 minutes or until items sell out (whichever comes first). It has been promoted heavily — 10 million users have registered for sale reminders.

The Numbers

MetricNormalFlash Sale Peak
Total requests/sec10,000100,000
Product page views/sec7,00080,000
Add-to-cart operations/sec2,00015,000
Checkout attempts/sec1,0005,000
App servers20Need: 200
DB write capacity10,000 writes/sec (max)

The Constraints

  • Must not crash: The site must stay up for the full 30 minutes. Non-sale pages (account, order history, customer support) must also work.
  • Must not oversell: If an item has 1,000 units, exactly 1,000 orders can succeed. Not 1,001. Not 999 (unless genuinely unclaimed). Inventory must be atomically decremented.
  • Must be fair: First-come, first-served. A user who clicks "Buy" at 12:00:01 should get the item before a user who clicks at 12:00:02. No one should get an unfair advantage from retrying or using bots (within reason).

Inventory Math

Total inventory: 100 items x 1,000 units = 100,000 units
Checkout attempts: 5,000/sec x 30 min = 9,000,000 attempts
Successful checkouts: 100,000 (at most)
Failure rate: (9,000,000 - 100,000) / 9,000,000 = 98.9%

98.9% of checkout attempts will fail because the item is sold out.
These MUST fail fast — not after a slow DB roundtrip.
Every millisecond spent on a doomed checkout is wasted capacity.

Back-of-Envelope Math

Compute Requirements

Normal capacity: 20 app servers x 500 req/sec each = 10,000 req/sec
Flash sale requirement: 100,000 req/sec
Servers needed: 100,000 / 500 = 200 app servers

Auto-scaling from 20 to 200 servers:
  Cloud provider VM boot time: ~60 seconds
  Container (Kubernetes): ~10-30 seconds
  But: 180 new instances need to:
    - Pull container image
    - Start application
    - Warm JVM / establish DB connections
    - Pass health checks
  Realistic time: 3-5 minutes

Those first 3-5 minutes:
  20 servers handling 100,000 req/sec = 5,000 req/sec each
  Normal capacity per server: 500 req/sec
  Overload factor: 10x
  Result: servers crash, 90% of requests fail

Database Write Capacity

Normal write load:
  Cart operations: 2,000/sec
  Checkout (order creation + inventory update): 1,000/sec
  Total: 3,000 writes/sec. DB capacity: 10,000. Utilization: 30%.

Flash sale write load:
  Cart operations: 15,000/sec
  Checkout: 5,000/sec
  Total: 20,000 writes/sec
  DB capacity: 10,000 writes/sec
  Overload: 2x

The database cannot handle the raw write load.
Even if servers can handle the traffic, the DB becomes the bottleneck.

Read Amplification

80,000 product page views/sec, but only 100 products in the sale.
  Average: 800 views/sec per product.
  Hottest item: potentially 5,000-10,000 views/sec (Zipf distribution).

Without caching: 80,000 DB reads/sec for 100 products.
  DB read capacity: 20,000 reads/sec (with replicas). Overwhelmed.

With caching: 100 product pages cached. Hit rate: ~100%.
  DB reads for product pages: ~0/sec.
  Caching is essential.

The Naive Design: Hope for the Best

Plan: Rely on auto-scaling. Every request goes through the normal path.

  User → Load Balancer → App Server → PostgreSQL
                                     (reads AND writes)

Checkout flow:
  1. Start DB transaction
  2. SELECT quantity FROM inventory WHERE product_id = 123 FOR UPDATE
  3. If quantity > 0: UPDATE inventory SET quantity = quantity - 1
  4. INSERT INTO orders (user_id, product_id, ...)
  5. Process payment (external API call, ~500ms)
  6. COMMIT transaction

Problems with this approach:
  - Step 2: FOR UPDATE takes a row lock. With 5,000 concurrent checkouts
    for the same product, 4,999 of them wait on the lock.
  - Step 5: Payment processing inside the transaction holds the lock for 500ms.
    Lock hold time: 500ms. Throughput: ~2 orders/sec per product.
    For 1,000 units: 1,000 / 2 = 500 seconds = 8+ minutes to sell out ONE item.
  - Auto-scaling lag: first 3-5 minutes overwhelm 20 servers.

What Actually Happens

T=0 (noon):     Sale starts. 100,000 req/sec hit 20 servers.
T=0-5s:         Servers at 10x capacity. Response times go from 100ms to 10s.
T=5-10s:        Thread pools full. 503 errors for 90% of users.
T=10-30s:       DB connection pools exhausted. Even non-sale pages fail.
T=1-3min:       Auto-scaler starts spinning up new instances.
T=3-5min:       New instances coming online, but DB is the bottleneck.
                Lock contention on inventory rows = ~2 orders/sec/product.
T=5-10min:      Traffic starts subsiding (users give up).
                Social media fills with "worst sale ever" posts.

Result: Most users see errors. Few orders succeed. Brand damage.

Iteration 1: Pre-Scaling

The sale is announced 24 hours in advance. You know exactly when the spike will happen. Auto-scaling is for unpredictable load. For a known event, pre-scale.

Pre-Warm the Caches

# Run at T-1 hour: load all sale products into cache
def pre_warm_sale_cache(sale_products):
    """
    Load sale product pages into Redis BEFORE the sale starts.
    This ensures 100% cache hit rate from the first request.
    """
    for product in sale_products:
        # Cache the product page data
        key = f"product:{product['id']}"
        redis.setex(key, 3600, json.dumps(product))  # 1 hour TTL

        # Cache the inventory count
        inv_key = f"inventory:{product['id']}"
        redis.set(inv_key, product['quantity'])  # 1000 units

    print(f"Pre-warmed {len(sale_products)} products")

# Also pre-warm: sale landing page, category pages, static assets

Cost

Extra compute cost:
  180 additional app servers x 2 hours x $0.10/hour = $36
  4 additional DB read replicas x 6 hours x $0.50/hour = $12
  Extra Redis memory: negligible

Total: ~$50-100 for the pre-scaling

Revenue from a successful flash sale: $50K-500K+

The ROI is obvious. Pre-scale.

Iteration 2: Atomic Inventory in Redis

The database bottleneck for checkout is the inventory row lock. Every checkout attempt needs to read and decrement the inventory count. With 5,000 concurrent checkouts/sec for the same product, the database lock becomes a serialization point.

Solution: move inventory to Redis. Redis is single-threaded and processes commands sequentially, which means atomic operations are trivially safe — and extremely fast (100,000+ operations/sec per instance).

The DECR Trick

# Redis-based inventory management
def attempt_checkout(user_id, product_id):
    inv_key = f"inventory:{product_id}"

    # Atomically decrement inventory
    # DECR returns the NEW value after decrementing
    remaining = redis.decr(inv_key)

    if remaining >= 0:
        # Success! We got a unit.
        # remaining=999 means we were the 1st buyer (1000 → 999)
        # remaining=0 means we were the last (1 → 0)
        return {"status": "reserved", "remaining": remaining}
    else:
        # remaining is negative — we went below 0.
        # Undo the decrement (increment back).
        redis.incr(inv_key)
        return {"status": "sold_out"}

# How it works, step by step:
#
# Redis key "inventory:12345" = 1000
#
# T=0.000s: User A: DECR inventory:12345 → 999 (>= 0, success!)
# T=0.001s: User B: DECR inventory:12345 → 998 (>= 0, success!)
# ...
# T=0.050s: User 1000: DECR inventory:12345 → 0 (>= 0, success! Last one.)
# T=0.051s: User 1001: DECR inventory:12345 → -1 (< 0, sold out!)
#           User 1001: INCR inventory:12345 → 0 (undo)
# T=0.052s: User 1002: DECR inventory:12345 → -1 (< 0, sold out!)
#           User 1002: INCR inventory:12345 → 0 (undo)

Why This Is Fast

PostgreSQL with row locks:
  Lock acquisition: ~0.1ms
  Disk I/O for UPDATE: ~1-5ms
  Payment processing in transaction: ~500ms (if included)
  Lock hold time: 1-500ms depending on design
  Throughput: 2-1,000 orders/sec per product (serial)

Redis DECR:
  Network round trip to Redis: ~0.5ms
  Redis command execution: ~0.001ms (in-memory, single-threaded)
  No disk I/O, no locks, no transactions
  Throughput: 100,000+ operations/sec

  Sold-out check: 0.5ms (one Redis roundtrip)
  No DB involvement at all for sold-out items.

  98.9% of checkouts (sold-out) return in 0.5ms instead of 5ms+.
  Massive reduction in system load.

But What About Durability?

Redis is in-memory. If it crashes, inventory state is lost. Solutions:

StrategyHowData Loss Risk
AOF (Append-Only File)Redis writes every command to disk. appendfsync everysecUp to 1 second of operations
Redis ClusterReplicate to secondary nodes. Automatic failover.Commands in-flight during failover (milliseconds)
DB reconciliationWrite confirmed orders to PostgreSQL asynchronously. After sale, reconcile Redis count with DB orders.None (DB is the eventual source of truth)
 
# Complete checkout flow with Redis inventory + async DB write
def checkout(user_id, product_id):
    # Step 1: Check inventory in Redis (fast path)
    inv_key = f"inventory:{product_id}"
    remaining = redis.decr(inv_key)

    if remaining < 0:
        redis.incr(inv_key)  # Undo
        return {"status": "sold_out", "message": "This item is no longer available."}

    # Step 2: Generate order ID
    order_id = generate_uuid()

    # Step 3: Enqueue the order for async processing
    order = {
        "order_id": order_id,
        "user_id": user_id,
        "product_id": product_id,
        "reserved_at": time.time(),
        "status": "pending"
    }
    message_queue.publish("orders", json.dumps(order))

    # Step 4: Return immediately to the user
    return {
        "status": "reserved",
        "order_id": order_id,
        "message": "Your order is being processed. You'll receive confirmation shortly."
    }
    # User sees: "Order placed! Confirmation coming in a few seconds."
    # Actual DB write + payment happens asynchronously in a worker.

Iteration 3: Queue-Based Load Leveling

The checkout path has three expensive operations: inventory decrement (now in Redis, fast), payment processing (external API, ~500ms), and order creation in the database (5-10ms). The payment and DB write don't need to happen synchronously with the user's request.

Synchronous vs Asynchronous Checkout

Synchronous checkout (traditional):
  User clicks "Buy"
  → Reserve inventory (Redis: 0.5ms)
  → Process payment (Stripe API: 500ms)
  → Write order to DB (10ms)
  → Return confirmation
  Total: ~510ms per checkout. Thread held for 510ms.

  At 5,000 checkouts/sec with 510ms processing:
    Threads needed: 5,000 × 0.51 = 2,550
    This is feasible but expensive.

Asynchronous checkout:
  User clicks "Buy"
  → Reserve inventory (Redis: 0.5ms)
  → Enqueue order (message queue: 1ms)
  → Return "Processing..." (HTTP 202 Accepted)
  Total: ~1.5ms per checkout. Thread freed almost instantly.

  At 5,000 checkouts/sec with 1.5ms processing:
    Threads needed: 5,000 × 0.0015 = 7.5
    Virtually no thread pool pressure.

The Worker Pipeline

But wait — 98.9% of those 9M checkout attempts already failed at the Redis inventory check. Only ~100,000 orders actually make it to the queue. Those process in: 100,000 / 10,000 = 10 seconds. The queue drains almost instantly for successful orders.

User Experience

Happy path (got the item):
  T=0ms:     User clicks "Buy Now"
  T=1.5ms:   Response: "Order placed! Confirmation coming shortly."
             Page shows: order status = "Processing"
  T=200ms:   (Optional) WebSocket push: "Payment processing..."
  T=700ms:   Worker completes payment + DB write
             WebSocket push: "Order confirmed! #ORD-12345"

Total user wait for confirmation: ~1 second.
User perception: instant.

Sold-out path:
  T=0ms:     User clicks "Buy Now"
  T=0.5ms:   Redis DECR returns -1. Sold out.
  T=1ms:     Response: "Sorry, this item is sold out."

Total: 1 millisecond. No queue, no DB hit, no wasted resources.

Handling Payment Failures

# Worker: process orders from the queue
def process_order(order):
    try:
        # Step 1: Process payment
        payment_result = payment_gateway.charge(
            user_id=order["user_id"],
            amount=get_product_price(order["product_id"]),
            idempotency_key=order["order_id"]  # Prevent double-charge on retry
        )

        if payment_result["status"] == "success":
            # Step 2: Write to DB
            db.execute("""
                INSERT INTO orders (id, user_id, product_id, status, paid_at)
                VALUES (%s, %s, %s, 'confirmed', NOW())
            """, (order["order_id"], order["user_id"], order["product_id"]))

            # Step 3: Notify user
            notify_user(order["user_id"], "Order confirmed!", order["order_id"])
        else:
            # Payment failed — release the inventory back
            redis.incr(f"inventory:{order['product_id']}")

            # Notify user
            notify_user(order["user_id"], "Payment failed. Item released.",
                       order["order_id"])

    except Exception as e:
        # Transient error — retry (the queue handles retries)
        raise RetryableError(f"Order processing failed: {e}")

Iteration 4: Graceful Degradation

Even with pre-scaling, some requests might exceed capacity during the initial surge. Prioritize the critical path (sale checkout) over non-essential features.

Priority Tiers

PriorityFeatureDuring Sale
P0 (Critical)Sale product pages, cart, checkoutFull functionality. All resources allocated.
P1 (Important)Non-sale product pages, user accountServed from cache. No DB writes.
P2 (Nice-to-have)Recommendations, "You might like", reviewsDisabled entirely. Return empty.
P3 (Deferrable)Analytics tracking, email triggers, search indexingQueued for later processing.

Feature Flags for Degradation

Iteration 5: Anti-Abuse and Fairness

Flash sales attract bots and resellers who try to buy all available inventory before real customers can. Fairness measures are essential.

Rate Limiting Per User

# Rate limit checkout attempts: max 1 per product per user per 5 seconds
def checkout_with_rate_limit(user_id, product_id):
    rate_key = f"rate:{user_id}:{product_id}"

    # Redis: SET NX with TTL = rate limit window
    allowed = redis.set(rate_key, "1", nx=True, ex=5)

    if not allowed:
        return {
            "status": "rate_limited",
            "message": "Please wait a few seconds before trying again.",
            "retry_after": 5
        }

    return attempt_checkout(user_id, product_id)

Quantity Limits

# Limit: max 2 units of any sale item per user
def checkout_with_limits(user_id, product_id, quantity=1):
    limit_key = f"purchased:{user_id}:{product_id}"

    # Check how many they've already bought
    current = int(redis.get(limit_key) or 0)

    if current + quantity > 2:
        return {"status": "limit_exceeded",
                "message": "Maximum 2 units per customer."}

    # Attempt checkout
    result = attempt_checkout(user_id, product_id)

    if result["status"] == "reserved":
        # Record the purchase count
        redis.incr(limit_key)
        redis.expire(limit_key, 86400)  # Keep for 24 hours

    return result

Virtual Waiting Room

Instead of letting all 10 million users hit the site simultaneously at noon, put them in a queue and release them gradually:

T-5min:  Users start arriving at the sale page.
         They see: "Sale starts in 5:00. You're in line."
         The page is a static HTML page served from CDN — zero server load.

T=0:     Sale starts. Users are assigned random positions in the queue.
         (Random, not first-come, to prevent "refresh at 11:59:59" bots.)

T=0s:    Release first 5,000 users into the sale.
         They can browse and checkout.

T=1s:    Release next 5,000 users.

T=2s:    Release next 5,000.
         ...and so on.

Rate: 5,000 users/sec × 20 seconds = 100,000 users admitted.
The remaining 9.9M see: "You're in line. Estimated wait: 2 minutes."
Most items sell out in 20-60 seconds. Late arrivals are told
"The sale is over" — no wasted server capacity.
# Virtual waiting room implementation
class WaitingRoom:
    def __init__(self, release_rate=5000):
        self.release_rate = release_rate  # users per second
        self.queue = []
        self.admitted = set()

    def join(self, user_id):
        """User arrives at the sale page."""
        position = random.random()  # Random position (not arrival order)
        redis.zadd("waiting_room", {user_id: position})
        rank = redis.zrank("waiting_room", user_id)
        estimated_wait = rank / self.release_rate
        return {"position": rank, "estimated_wait_seconds": estimated_wait}

    def release_batch(self):
        """Called every second by a scheduler."""
        # Pop the next batch of users from the sorted set
        users = redis.zpopmin("waiting_room", self.release_rate)
        for user_id, _ in users:
            # Give them a time-limited admission token
            token = generate_token(user_id, ttl=300)  # 5 min to complete purchase
            redis.setex(f"admitted:{user_id}", 300, token)
            notify_user(user_id, "It's your turn! You have 5 minutes to shop.")

    def is_admitted(self, user_id):
        """Check if user has been released from the waiting room."""
        return redis.exists(f"admitted:{user_id}")

Key Takeaways

  1. Pre-scale for known events. Auto-scaling is for surprises. A flash sale announced 24 hours in advance is not a surprise. Scale up hours before, not minutes after.
  2. Move hot data to Redis. Inventory checks at 5,000/sec with row locks in PostgreSQL will create a serialization bottleneck. Redis DECR is atomic, in-memory, and handles 100K+ ops/sec. Use it as the real-time inventory ledger during the sale.
  3. Fail fast for sold-out items. 98.9% of checkout attempts will fail (sold out). Each one should take 0.5ms (Redis check), not 5ms (DB roundtrip) or 500ms (full checkout flow). Fast rejection preserves capacity for real orders.
  4. Decouple checkout from payment. Accept the order synchronously (1ms), process payment asynchronously (500ms). The user sees "Order placed!" instantly. The queue buffers the payment processing at the DB's sustainable rate.
  5. A virtual waiting room is a rate limiter for humans. Instead of 10M users hitting the site at once, release 5,000/sec. The system never exceeds its capacity. Users wait in a fair queue instead of experiencing crashes.
  6. Offload reads to CDN. 80% of flash sale traffic is product page views. Pre-render them as static HTML and serve from CDN. Zero app server load for the majority of traffic.
  7. Degrade gracefully under load. Disable non-essential features (recommendations, analytics, reviews) to free up resources for the critical path (browsing and checkout).
  8. Plan for reconciliation. When Redis is the real-time source of truth, reconcile with the database after the event. Check that Redis inventory counts match DB order counts. Handle any discrepancies (payment failures, race conditions).
