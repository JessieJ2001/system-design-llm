Design a Real-Time Gaming Leaderboard

50 million players, 100K score updates per second, and "What's my rank?" queries that must answer in under 50ms — a lesson in sorted sets, fan-out queries, and approximate ranking.

NALSD Walkthrough • ~20 min read • LC #380 / #215 (order-statistic / sorted-set)

The Scenario

LC kernel. The textbook structure behind a leaderboard is an order-statistic tree (or a skip list — what backs a Redis ZSET): keyed by score, it supports insert/update in O(log n), rank-of(member) in O(log n), and range-by-rank(i, i+m) in O(log n + m). A "Kth Largest in a Stream" (LC #703) heap answers only "top K" — the order-statistic tree additionally answers arbitrary rank queries, which is the whole point of a leaderboard. Four properties that break at scale: (1) one node's tree doesn't take the write rate — a single Redis instance caps at ~100K ops/sec on a 50M-entry sorted set, below our 260K ops/sec; (2) rank requires a global order, but sharding splits the data — ZREVRANK on a shard gives you your rank within that shard, not globally; (3) in-RAM only has no durability story — a crash loses 50M entries; (4) "random" queries aren't free at 100K QPS — exact rank fanned out to 10 shards is 1M ops/sec. The C8 production cousin: a sharded skip list (hash-by-player) + a coarse bucket histogram that turns the global-rank query into a sum of per-shard counters — exact for the top, approximate (and Redis-free) for everyone else — behind AOF/replica durability.

You are building the leaderboard service for a massively multiplayer online game. Players earn scores through in-game actions, and the leaderboard must reflect these changes in real time. This is a stateful infra primitive — an ordered-set service with a query API — sitting between the game servers (which write scores) and the clients (which read ranks); the game itself, matchmaking, and anti-cheat are upstream and out of scope. See functional-requirements.html (infra-primitive class — the FR vocabulary is operations + semantics, not user-facing features).

Requirements

  • Players: 50,000,000 registered accounts.
  • Score updates: 100,000 updates/sec during peak hours (each update sets a player's new score).
  • Score range: 0 to 10,000,000,000 (10 billion). Scores are monotonically increasing (new score is always ≥ old score).
  • Latency target: <50ms for all query types.

Data Size

Per player: player_id (8 bytes) + score (8 bytes) = 16 bytes
Total: 50,000,000 × 16 bytes = 800 MB

With Redis sorted set overhead (~64 bytes per entry for skip list pointers):
  50M × (16 + 64) = 4 GB

This fits in memory on a single machine with 8+ GB RAM.

Query Types

QueryDescriptionExpected QPS

My rank "I have a score of 1,234,567. What is my rank out of 50M players?" 100,000/sec
Window around me "Show ranks 1,234,560–1,234,580 (10 above and 10 below me)." Folded into "my rank" QPS — same fan-out shape, just ZRANGE by rank after the rank is known. (part of 100,000/sec)

Functional requirements (cite functional-requirements.html)

  • submit_score(leaderboard_id, player_id, score) — set a player's score; idempotent on (player, score) since scores are monotone (re-submitting the same value is a no-op). Latency p99 ≤ 50 ms.
  • get_rank(leaderboard_id, player_id) → {rank, score, exact?} — the player's position (1 = top). Exact for the top tier, best-effort (±a small window) below it — see the flagged assumption.
  • get_top_k(leaderboard_id, k) → [{player, score, rank}] — the K highest, ordered; k ≤ ~1,000 (clients page beyond that — cite api-design-patterns#pagination).
  • get_window(leaderboard_id, player_id, n) → [...] — the n players above and n below the caller (the "around me" strip).
  • Per-segment leaderboards — the same operations scoped to a friends list, a region, or a time window (daily / weekly / season). A segment is just another leaderboard_id; friends segments are derived (no separate store).

Out of scope (named, not silently dropped): anti-cheat / score validation (an upstream service decides what a legitimate score is — we trust the writer); the game itself and matchmaking; player identity / profile (we key on an opaque player_id); historical rank-over-time charts (a downstream analytics pipeline owns the request log); reward fulfilment for top finishers.

Flagged assumptions (would confirm with the interviewer): (1) exact rank is required only for the top tier (~top 1,000 — where a one-position difference is a prize boundary); below that, “rank 4,237,000 vs 4,234,567 out of 50M” is indistinguishable to a player, so an approximate rank is acceptable — if the interviewer insists on exact rank for everyone, the cost jumps (every rank query becomes a 10-shard ZCOUNT fan-out, see Iteration 2's alternatives). (2) How many leaderboards? — we assume one global + a handful of region/season segments + on-the-fly friends segments; if the game has thousands of distinct ladders, multiply shard count and memory. (3) Score-update rate 100K/sec at peak, scores monotone — if scores can decrease (penalties, decay) the histogram maintenance needs the previous score (a side hash, addressed in Takeaways). (4) Read:write ratio ≈ 1.6:1 (160K read QPS : 100K write QPS) — write-heavy for a leaderboard; this is why the bottleneck is write throughput, not read latency.

Back-of-Envelope Math

The key question: can a single Redis instance handle all operations?

Redis Sorted Set Operation Complexity

Sorted set with N = 50,000,000 entries:

ZADD (update score):
  Time complexity: O(log N) = O(log 50M) = O(26)
  Each operation: ~1-2μs of CPU time
  At 100K updates/sec: 100K × 26 = 2.6M skip list comparisons/sec

ZREVRANGE 0 99 (top-100):
  Time complexity: O(log N + 100) ≈ O(126)
  Very fast: traverse to the tail of the skip list, then walk 100 entries.

ZREVRANK (my rank):
  Time complexity: O(log N) = O(26)
  At 100K queries/sec: 100K × 26 = 2.6M skip list traversals/sec

Total operations per second:
  100K ZADD + 10K ZREVRANGE + 100K ZREVRANK + 50K friends queries
  ≈ 260K+ operations/sec on sorted sets

Redis Single-Instance Throughput

Redis benchmark for sorted sets with 50M entries:
  Small sorted set (1K entries):  ~200K ops/sec
  Medium sorted set (1M entries): ~150K ops/sec
  Large sorted set (50M entries): ~80K-100K ops/sec

  (Throughput decreases with set size because each O(log N) operation
   does more work, and the skip list doesn't fit in L1/L2 cache.)

Our load: ~260K ops/sec on a single sorted set of 50M entries.
Redis capacity: ~100K ops/sec.

Result: a single Redis instance cannot handle the load.
We need to shard.

Access patterns & storage engine

The workload (cite access-patterns.html): score updates are write-heavy-uniform across players (uniform because player_id is opaque — no skew); the leaderboards themselves are power-law (the one global ladder gets the lion's share of reads — the hot-key shape that bites in Iteration 2); friends segments are high-cardinality-query (50M distinct small sets, queried by a key the store doesn't index on). The store is a sorted-set engine — an in-RAM skip list + hash — which is the KV-engine entry's "sorted-set" variant; we choose it (over a B-tree index, see Iteration 1's alternatives) because all three operations — update, rank-of, range-by-rank — are O(log n) on it and it lives entirely in memory at this size.

The bottleneck is throughput, not storage — bottlenecks#hot-key in disguise: it's not a hot row, it's a hot data structure (one giant sorted set every op touches). 4 GB fits on a single machine easily, but 260K ops/sec exceeds a single Redis instance's capacity for large sorted sets. This is the opposite of the distributed-cache scenario (where storage was the bottleneck).

Naive Design: Single Redis Sorted Set

The simplest approach: one Redis instance, one sorted set containing all 50M players.

import redis

r = redis.Redis()
LEADERBOARD_KEY = "leaderboard:global"

def update_score(player_id: str, new_score: int):
    """Update a player's score. O(log N)."""
    r.zadd(LEADERBOARD_KEY, {player_id: new_score})

def get_top_100():
    """Get the 100 highest-scoring players. O(log N + 100)."""
    return r.zrevrange(LEADERBOARD_KEY, 0, 99, withscores=True)

def get_rank(player_id: str):
    """Get a player's rank (0-indexed, 0 = highest). O(log N)."""
    rank = r.zrevrank(LEADERBOARD_KEY, player_id)
    return rank + 1 if rank is not None else None

def get_friends_top_100(player_id: str, friend_ids: list):
    """Get top 100 among friends. O(F × log N) where F = number of friends."""
    pipe = r.pipeline()
    for fid in friend_ids:
        pipe.zscore(LEADERBOARD_KEY, fid)
    scores = pipe.execute()

    friends_with_scores = [
        (fid, score) for fid, score in zip(friend_ids, scores)
        if score is not None
    ]
    friends_with_scores.sort(key=lambda x: x[1], reverse=True)
    return friends_with_scores[:100]

What This Gets Right

  • Redis sorted sets are the natural data structure for leaderboards. They maintain elements in score-sorted order with O(log N) operations.
  • Under the hood, Redis implements sorted sets using a skip list (for ordered traversal) plus a hash table (for O(1) lookups by member). This gives O(log N) for rank, range, and update queries.
  • The API is minimal: ZADD, ZREVRANGE, ZREVRANK, ZSCORE. Each maps directly to a leaderboard operation.

What Breaks

Problem 1: Throughput Saturation

Required: ~260K sorted set ops/sec
Capacity: ~100K ops/sec for a 50M-entry sorted set

At 260K ops/sec:
  Redis command queue grows faster than it drains.
  Latency rises: p50 goes from 0.5ms → 5ms → 50ms → timeout.
  Clients start timing out at 50ms.
  Retry storms make it worse.

The single Redis instance becomes a hard throughput ceiling.

Problem 2: Single Point of Failure

If the Redis instance crashes, the entire leaderboard is unavailable. With 50M entries, rebuilding from the database takes minutes. During that time, all leaderboard queries fail.

Problem 3: Writes Block Reads

Redis is single-threaded. A burst of 100K ZADD operations per second consumes CPU time that cannot be spent on ZREVRANK queries. During peak write load, read latency spikes because reads wait in the command queue behind writes.

Problem 4: Friends Query Is Expensive

Friends top-100 for one player:
  100 friends × 1 ZSCORE each = 100 Redis commands
  Pipelined: 1 round trip, 100 commands = ~2ms total

At 50,000 friends queries/sec:
  50K × 100 = 5,000,000 ZSCORE commands/sec
  This alone exceeds the 100K ops/sec limit by 50×.

Even with pipelining (which reduces network overhead but not CPU),
the server still processes 5M commands.

Which failure modes (cite failure-modes.html)

  • hot-key-meltdown — blast radius: the entire leaderboard. One sorted set absorbs 100% of ops; at 260K > 100K capacity the command queue grows monotonically, p99 walks past the 50 ms SLO into timeouts. Detection: Redis instantaneous_ops_per_sec > ~100K and latency histogram tail rising. Recovery: none without sharding — this is the bottleneck Iteration 1 fixes.
  • leader-loss — trigger: the single Redis process dies. Blast radius: all queries fail. Recovery: reload 50M entries from the game DB — minutes of full outage (no replica yet). Iteration 1 adds a replica per shard; the degradation contract is in the Final Design.
  • retry-storm — once p99 crosses the client timeout, every game server retries; effective load multiplies, the queue never drains. Needs the throughput headroom that sharding + caching provide, plus client-side backoff (Operations block).
  • pool-starvation — the 5M friends-ZSCORE/sec (50× capacity) saturates the single thread; submit_score and get_rank wait behind it — writes block reads block writes. Iteration 2's friends-score cache cuts this to 500K/sec.

Iteration 1: Shard the Leaderboard by Player ID

Bottleneck (cite bottlenecks.html)

The Naive design's dominant bottleneck is bottlenecks#hot-key — one sorted set takes 260K ops/sec against ~100K capacity (2.6× over). The first lever for a hot key that is also a capacity ceiling is to split it (cite bottleneck-to-decisions#hot-key → revisit the sharding decision). The critical question: shard by what?

Alternatives considered (C4) — the store, and the shard key

The store (cite storage-engines.html):

  • Redis-Cluster ZSETs (chosen) — the sorted-set engine; O(log n) update / rank-of / range-by-rank, all in RAM; Cluster gives hash-slot sharding + per-shard replica failover out of the box. Cost: in-RAM only (durability via AOF/replica), and cross-shard rank needs work (this whole page).
  • A SQL table with an indexed score column (Postgres / MySQL) — rejected. SELECT count(*) WHERE score > ? is the exact rank, and a B-tree index makes it a range scan, but: at 100K updates/sec the index write-amp (bottlenecks#index-write-amp) and MVCC bloat crush a single Postgres; get_rank on a 50M-row B-tree is a count over potentially millions of index tuples = tens of ms even with an index, not the <1 ms a ZSET ZREVRANK gives; and you'd shard it anyway. Window functions (rank() OVER (ORDER BY score)) compute the whole ranking — O(n), useless per-query. SQL is right only if you also need rich relational queries on scores, which we don't.
  • A bespoke order-statistic-tree service — rejected for v1. A custom in-process augmented BST (subtree-size counters → O(log n) rank-of directly, no ZCOUNT) is what we'd build if Redis's per-op overhead became the wall; it's strictly more code and ops for the same big-O. Keep it in the back pocket (it's essentially the C8 production cousin, just home-grown).
  • A search engine (Elasticsearch) — rejected. Inverted indexes rank documents by a relevance score, not by an arbitrary numeric field with exact-rank-of-element semantics; aggregations approximate; write throughput at 100K/sec needs heavy tuning. Wrong tool.

The shard key (cite sharding-strategies.html):

StrategyHowVerdict
range — by scoreshard 1 = scores 0–1B, shard 2 = 1B–2B, …Rejected. A score update can cross a boundary → cross-shard transaction (DEL old + ADD new) on the hot path; score distributions are skewed (most players cluster low) → hot shards (failure-modes#hot-partition-meltdown); rebalancing a boundary is a bulk move. Upside it doesn't pay off: "my rank" would be one shard's ZREVRANK + a constant offset — nice, but not worth the write-path pain.

Cost: every global query (top-K, my rank, window) fans out to all shards — but parallel fan-out to 10 shards is <1 ms, and the bucket-histogram trick (Iter 2) removes the fan-out for the 99% case.
hybrid — hash + per-shard approx countshash-by-player, plus each shard publishes a coarse score histogramThis is the destination — Iteration 2 adds the histogram on top of the hash sharding. Listed here so the arc is visible.

Why Shard by Player ID (Not by Score)

The decision, in one line: scores change, player IDs do not. Sharding by player ID makes a score update always local to one shard (no cross-shard transactions) and the distribution uniform. The price — global queries need fan-out — is cheap because the fan-out is parallel.

Per-Shard Capacity

10 shards, each with:
  Players: 50M / 10 = 5M entries
  Memory: 4GB / 10 = 400MB per shard
  Score updates: 100K / 10 = 10K ZADD/sec per shard
  Redis capacity for 5M-entry sorted set: ~150K ops/sec

  Utilization per shard: 10K / 150K = 6.7% — plenty of headroom.

Shard Assignment Code

NUM_SHARDS = 10

def get_shard(player_id: str) -> int:
    """Deterministic shard assignment based on player_id hash"""
    return hash(player_id) % NUM_SHARDS

def update_score(player_id: str, new_score: int):
    """Update score — goes to exactly one shard"""
    shard = get_shard(player_id)
    redis_shards[shard].zadd(f"leaderboard:{shard}", {player_id: new_score})

After sharding, global queries require fan-out across all shards. Here is how each query type works.

Query: Global Top-100

No single shard has the global top-100. We must query all 10 shards and merge.

import asyncio

async def get_global_top_100():
    """Fan-out to all shards, merge results"""
    # Query all shards in parallel
    tasks = []
    for shard_id in range(NUM_SHARDS):
        tasks.append(
            redis_shards[shard_id].zrevrange(
                f"leaderboard:{shard_id}", 0, 99, withscores=True
            )
        )
    shard_results = await asyncio.gather(*tasks)

    # Merge: combine all shard results (10 × 100 = 1,000 entries)
    all_entries = []
    for entries in shard_results:
        all_entries.extend(entries)

    # Sort by score descending, take top 100
    all_entries.sort(key=lambda x: x[1], reverse=True)
    return all_entries[:100]
Performance:
  10 parallel ZREVRANGE queries × ~0.1ms each = 0.1ms (wall clock)
  Merge 1,000 entries = ~0.01ms (in-memory sort of 1K items)
  Total: ~1ms — well within 50ms target ✓

Query: My Rank

This is where sharding gets interesting. ZREVRANK tells you your rank within your shard, but you need your rank globally. A player ranked #42,000 in shard 3 might be globally ranked #420,000 or #400 — it depends on how many players in other shards have higher scores.

async def get_global_rank(player_id: str):
    """Get player's global rank by counting players
    with higher scores across all shards"""
    shard = get_shard(player_id)

    # Step 1: Get the player's score from their shard
    score = await redis_shards[shard].zscore(
        f"leaderboard:{shard}", player_id
    )
    if score is None:
        return None

    # Step 2: For each shard, count how many players
    #         have score > this score
    tasks = []
    for shard_id in range(NUM_SHARDS):
        tasks.append(
            redis_shards[shard_id].zcount(
                f"leaderboard:{shard_id}",
                f"({score}",   # exclusive lower bound
                "+inf"         # to infinity
            )
        )
    counts = await asyncio.gather(*tasks)

    # Step 3: Sum counts across all shards + 1
    global_rank = sum(counts) + 1
    return global_rank
Performance:
  Step 1: 1 ZSCORE = 0.1ms
  Step 2: 10 parallel ZCOUNT queries = 0.1ms (wall clock)
  Step 3: sum 10 integers = negligible

  Total: ~0.5ms per rank query — well within 50ms ✓

But: at 100K rank queries/sec, each fans out to 10 shards:
  100K × 10 = 1,000,000 ZCOUNT commands/sec across the cluster
  Per shard: 1M / 10 = 100K ZCOUNT/sec
  Plus 10K ZADD/sec from writes = 110K ops/sec per shard.
  Redis capacity: 150K ops/sec for 5M entries.
  Utilization: 110K / 150K = 73% — approaching the limit.

Query: Friends Top-100

async def get_friends_top_100(player_id: str, friend_ids: list):
    """Fetch scores for all friends, sort locally"""
    # Group friends by shard
    shard_groups = {}
    for fid in friend_ids:
        shard = get_shard(fid)
        shard_groups.setdefault(shard, []).append(fid)

    # Fetch scores from each shard in parallel (pipelined within each shard)
    tasks = []
    for shard_id, fids in shard_groups.items():
        pipe = redis_shards[shard_id].pipeline()
        for fid in fids:
            pipe.zscore(f"leaderboard:{shard_id}", fid)
        tasks.append((fids, pipe.execute()))

    # Collect results
    results = await asyncio.gather(*[t[1] for t in tasks])
    friends_with_scores = []
    for (fids, _), scores in zip(tasks, results):
        for fid, score in zip(fids, scores):
            if score is not None:
                friends_with_scores.append((fid, score))

    # Sort and take top 100
    friends_with_scores.sort(key=lambda x: x[1], reverse=True)
    return friends_with_scores[:100]
Performance for 100 friends across 10 shards:
  ~10 friends per shard (uniform distribution)
  10 parallel pipelined requests × 10 ZSCORE each = 0.1ms wall clock
  Sort 100 entries locally = negligible
  Total: ~0.5ms ✓

At 50K friends queries/sec:
  50K × 100 friends = 5M ZSCORE commands/sec
  Per shard: 5M / 10 = 500K/sec ← exceeds capacity!

This is a problem. We'll address it in Iteration 2.

Schema / API delta

Redis Cluster, 10 hash-slot shards, RF=2 (1 primary + 1 replica per shard).
Per leaderboard L, the sorted set is keyed by player-hash slot — Cluster
routes automatically; logically:

  ZSET  lb:{L}                       member = player_id, score = score
        ZADD lb:{L}             # submit_score      O(log n)
        ZSCORE lb:{L}                  # my score          O(1)
        ZREVRANK lb:{L}                # rank WITHIN slot  O(log n)
        ZREVRANGE lb:{L} 0 99 WITHSCORES       # top-100 of slot   O(log n+100)
        ZCOUNT lb:{L} ( +inf            # count above, per slot

Persistence: AOF appendfsync everysec (RPO ≈ 1 s); the game DB is the
system of record — Redis is rebuildable. Replica: async (asymp <1 ms lag),
auto-failover via Cluster. (REST surface lands in the Final Design's C9.)

What breaks (C5; cite failure-modes.html)

  • leader-loss — a shard's primary dies mid-tournament. Blast radius: 1/10 of players' writes + every fan-out query (which touches that slot) until failover (~seconds with Cluster). Detection: Cluster FAIL gossip; client redirect errors. Recovery: replica promoted; the async replica may have lost <1 s of writes → a few players' scores reappear slightly stale and self-correct on their next submit_score (scores are monotone, so a stale lower value can't beat a fresh higher one).
  • replica-divergence — async replication means a read served by a replica (if you ever route reads there) can be behind the primary. We keep get_rank on primaries; the only replica reads are in DR. Stated in the degradation contract.
  • tail-cascade / network-rtt-bound — a fan-out query is as slow as its slowest shard; one GC pause on shard 7 stalls every "my rank" globally. Mitigation: the fan-out has a tight per-shard deadline (e.g., 30 ms); a laggard shard's count is dropped and the rank returned with a "≥" qualifier — and Iteration 2 removes the fan-out for 99% of queries entirely, which is the real fix.
  • retry-storm still latent — friends queries are 500K ZSCORE/sec/shard, 3× over capacity; if friends QPS spikes, that shard saturates and the storm returns. Iteration 2's friends-score cache is required, not optional.

Why this iteration doesn't fully close the loop: sharding bought throughput headroom for submit_score (10K/shard vs 150K capacity = 7%) and made global queries possible, but it did not fix the two NFR failures that survive — (a) "my rank" at 100K QPS × 10 fan-out = 100K ZCOUNT/sec/shard pushes a shard to 73% (no headroom for a deploy or a spike), and (b) friends queries are still 3× over. Both go to Iteration 2.

Iteration 2: Cache Top-100, Approximate Rank, and Friends Optimization

Bottleneck (cite bottlenecks.html)

After Iteration 1, two NFRs still fail: (1) "my rank" fans out 100K QPS × 10 shards = 100K ZCOUNT/sec/shard → 73% utilization, no headroom — this is bottlenecks#network-rtt-bound + bottlenecks#fanout-asymmetry (a cheap read amplified by fan-out into a fleet-wide load); (2) friends queries are 500K ZSCORE/sec/shard, 3× over; (3) the global top-100 ladder is the hot-key read — 10K QPS all reading the same handful of slots. The lever for a fan-out-amplified read and a hot read key is the same: stop hitting the store — precompute/cache the slow-changing part (cite bottleneck-to-decisions#fanout, #hot-key). Three caches/precomputations, one per failing read.

Alternatives considered (C4) — exact rank when the data is sharded

The hard problem: ZREVRANK on a shard gives rank within that shard, but a global rank needs a global order, which sharding destroyed. Options:

  • Exact via 10-shard ZCOUNT fan-out (the Iteration-1 mechanism) — correct but expensive. 100K QPS × 10 = 1M ZCOUNT/sec; pushes shards to 73%; one slow shard stalls every rank query. Keep it — but only for the top tier (~1K players where a one-position difference is a prize boundary): 1K QPS × 10 = 10K ZCOUNT/sec, negligible.
  • A single fat Redis with replicas (un-shard the rank) — rejected. Brings back exact ZREVRANK in one op — but it's exactly the Naive bottleneck (one sorted set, 260K ops/sec > 100K capacity); replicas scale reads but not the single-threaded write path. Doesn't work.
  • Don't shard the rank index at all — keep scores sharded by player but maintain ONE unsharded "rank tree" (Redis or a custom order-statistic tree) updated async — rejected. The rank tree still takes all 100K writes/sec on one node — same wall. (A custom OST with lock-striping might push the wall higher, but it's a project, not an iteration.)
  • Approximate via a coarse bucket histogram (chosen for the 99%). Partition the score space [0, 10B] into B buckets; each shard maintains a count-per-bucket (incrementally, on every ZADD); the global histogram is the elementwise sum; rank(score) ≈ (sum of higher-bucket counts) + (interpolated position within the score's bucket) + 1 — zero Redis ops per query, just a local array lookup. Error is bounded by the count in the player's bucket. This is the scenario-specific bit; it's derived below.

The chosen combination: bucket-histogram for the approximate-rank 99% case + exact ZCOUNT fan-out for the top-1K — this is the hybrid sharding strategy: hash-by-player for writes, plus a coarse per-shard summary for the cross-shard aggregate query.

Optimization A: Cache the Global Top-100

The global top-100 changes slowly. The #1 player does not change every second. We can compute it once per second and serve it from a cache.

import time, threading

# Background thread recomputes top-100 every second
cached_top_100 = []
cached_top_100_time = 0

def top_100_refresh_loop():
    global cached_top_100, cached_top_100_time
    while True:
        cached_top_100 = compute_global_top_100()  # fan-out to 10 shards
        cached_top_100_time = time.time()
        time.sleep(1)

threading.Thread(target=top_100_refresh_loop, daemon=True).start()

def get_global_top_100():
    """Served from cache. Always <0.01ms. Updated every 1 second."""
    return cached_top_100
Before: 10K top-100 queries/sec × 10 shard fan-out = 100K ZREVRANGE/sec
After:  1 query/sec × 10 fan-out = 10 ZREVRANGE/sec

Reduction: 10,000× fewer Redis operations for this query type.
Staleness: top-100 is at most 1 second behind. Imperceptible to users.

Optimization C: Friends Score Cache

Friends queries generate 5M ZSCORE/sec across the cluster. Most friends' scores do not change every second. We can cache friend scores with a short TTL.

# Application-level cache for individual player scores
# TTL: 5 seconds — scores update at most every few seconds per player
score_cache = {}  # player_id → (score, expiry_time)

def get_player_score_cached(player_id):
    entry = score_cache.get(player_id)
    if entry and time.time() < entry[1]:
        return entry[0]  # cache hit

    # Cache miss — fetch from Redis
    shard = get_shard(player_id)
    score = redis_shards[shard].zscore(f"leaderboard:{shard}", player_id)
    score_cache[player_id] = (score, time.time() + 5)
    return score

def get_friends_top_100_cached(player_id, friend_ids):
    scores = [(fid, get_player_score_cached(fid)) for fid in friend_ids]
    scores = [(fid, s) for fid, s in scores if s is not None]
    scores.sort(key=lambda x: x[1], reverse=True)
    return scores[:100]
With 5-second cache TTL:
  Each unique player's score is fetched at most once every 5 seconds.
  50M players / 5 seconds = 10M unique fetches/sec in the worst case.

  But: most players' friends overlap with other players' friends.
  And: most players are not being queried every 5 seconds.

  Realistic cache hit rate: ~90% (most friend score lookups are cached).

  Before: 5M ZSCORE/sec → After: 500K ZSCORE/sec
  Per shard: 500K / 10 = 50K/sec. Manageable.

Optimization B: Approximate Rank Using Histograms

The "my rank" query is the most expensive: 100K queries/sec, each fanning out to 10 shards. We can reduce this dramatically by pre-computing score distribution histograms.

The Histogram Approach

Divide the score range [0, 10B] into 10,000 buckets:
  Bucket 0:     scores 0 to 999,999
  Bucket 1:     scores 1,000,000 to 1,999,999
  ...
  Bucket 9,999: scores 9,999,000,000 to 9,999,999,999

Each bucket width: 10B / 10,000 = 1,000,000 (1M points)

Every second, each shard computes its local histogram:
  "How many players have scores in each bucket?"

Example for Shard 3 (5M players):
  Bucket 0:     125,000 players
  Bucket 1:     118,000 players
  ...
  Bucket 5,000: 2,100 players  ← your score (5,234,567) is here
  ...
  Bucket 9,999: 3 players

Accuracy Analysis

Bucket width: 1,000,000 score points
Players per bucket (average): 50M / 10,000 = 5,000 players

The approximation error is at most the number of players in your bucket:
  Maximum error: ±5,000 positions (for average-density buckets)
  For most players: ±2,000 positions

  At rank ~25,000,000 (median player): ±5,000 = ±0.02% error
  At rank ~100 (top player): this bucket likely has 0-10 players,
    so the error is ±5 positions.

For 99% of players (rank > 1,000), ±5,000 error is imperceptible.
  "You are ranked #4,234,567 out of 50,000,000" vs
  "You are ranked #4,237,000 out of 50,000,000" — no user cares.

For top-1,000 players (where exact rank matters):
  Use the exact fan-out query (ZCOUNT across all shards).
  At 1K exact queries/sec × 10 fan-out = 10K ZCOUNT/sec.
  This is negligible.

Histogram Code

# Background: each shard computes its histogram every second
def compute_shard_histogram(shard_id):
    """Count players in each score bucket for this shard"""
    histogram = []
    for bucket in range(NUM_BUCKETS):
        low = bucket * BUCKET_WIDTH
        high = (bucket + 1) * BUCKET_WIDTH - 1
        count = redis_shards[shard_id].zcount(
            f"leaderboard:{shard_id}", low, high
        )
        histogram.append(count)
    return histogram

# Global histogram: sum across all shards
def compute_global_histogram():
    shard_histograms = [compute_shard_histogram(i) for i in range(NUM_SHARDS)]
    global_histogram = [0] * NUM_BUCKETS
    for bucket in range(NUM_BUCKETS):
        for shard in range(NUM_SHARDS):
            global_histogram[bucket] += shard_histograms[shard][bucket]
    return global_histogram

# Approximate rank lookup: no Redis queries needed!
def get_approximate_rank(player_id: str, score: int):
    """Estimate rank using the pre-computed histogram"""
    bucket = int(score / BUCKET_WIDTH)

    # Count all players in buckets above this one
    players_above = sum(global_histogram[bucket + 1:])

    # Estimate position within this bucket (linear interpolation)
    bucket_count = global_histogram[bucket]
    position_in_bucket = (score % BUCKET_WIDTH) / BUCKET_WIDTH
    players_above_in_bucket = int(bucket_count * (1 - position_in_bucket))

    return players_above + players_above_in_bucket + 1

Histogram Computation Cost

Computing the histogram: 10,000 ZCOUNT queries per shard per second.
  Per shard: 10,000 ZCOUNT/sec (each O(log N + bucket_count))

  This is expensive! 10K ops/sec per shard just for histograms.

Optimization: use Redis ZRANGEBYSCORE to scan once instead of 10K ZCOUNTs.
  Or: maintain the histogram incrementally.
  On each ZADD, update the histogram bucket count (decrement old bucket,
  increment new bucket). This adds 2 operations per score update:
  100K updates/sec × 2 = 200K histogram ops/sec (spread across 10 shards
  = 20K per shard). Each op is a simple counter increment — O(1).

Total per-shard load with incremental histograms:
  10K ZADD (score updates)
  + 20K counter increments (histogram maintenance)
  + 10K ZCOUNT (exact rank for top-1K players)
  + 500 ZSCORE (friends queries — see below)
  ≈ 40K ops/sec per shard

  Utilization: 40K / 150K = 27% — healthy headroom ✓

Schema / API delta

New per shard, per leaderboard L — the bucket histogram (auxiliary structure):

  HASH  hist:{L}:           field = bucket_index (0..B-1), val = count
        HINCRBY hist:{L}:  1     # on submit_score
        HINCRBY hist:{L}:  -1    #   (needs the old score —
                                                #    monotone scores, so the
                                                #    OLD score is the one we
                                                #    just overwrote; ZSCORE
                                                #    before ZADD, in a MULTI)
  B (bucket count) = 10,000  →  width = 10B / 10,000 = 1,000,000 score pts
                            →  ~5,000 players/bucket avg  →  ±5,000 rank err

API-server local (per leaderboard L):  global_hist[L] = elementwise sum of the
  10 shard hists, refreshed every 1 s  (10 HGETALL/sec/shard — negligible);
  top100_cache[L]  (1-s refresh);  score_cache (player→score, 5-s TTL).

The Redis ZSET schema and persistence (AOF everysec, RF=2) are unchanged
from Iteration 1.

What breaks (C5; cite failure-modes.html)

  • replica-divergence — the histogram and the ZSET can drift if the incremental updates aren't atomic with the ZADD (e.g., a crash between ZADD and the two HINCRBYs). Blast radius: a few buckets off by 1 — sub-position rank error, invisible. Recovery: a slow background job re-derives each shard's histogram from ZCOUNTs once an hour and overwrites it (the "compute_shard_histogram" path), self-healing the drift. Wrapping the three commands in a MULTI makes it atomic on the primary.
  • stampede — end-of-season: a few minutes before reset, every player checks their final rank → the top-100 ladder reads spike (and clients that bypass the 1-s cache hammer the fan-out). Detection: top-100-cache hit-rate drop, ZREVRANGE QPS spike on the slots. Mitigation: the 1-s top-100 cache and the Redis-free histogram rank already absorb 99.99% of it; for the exact-top-1K path, single-flight the fan-out (one in-flight recompute per second, all callers share it — cite single-flight.html). Stale-while-revalidate on the cache so a slow recompute never causes a miss.
  • leader-loss (still relevant) — a shard primary dying loses its in-RAM histogram too; the replica has it (replicated like any key); on failover the hourly re-derive job re-syncs anything stale. If both are lost, the histogram rebuilds from ZCOUNTs after the ZSET reloads from the game DB.
  • hot-key-meltdown on a single mega-popular leaderboard — if one ladder (e.g., "global") is 100× hotter than the rest, its 10 slots are hot even after sharding. Mitigation: the caches make reads on it nearly free; for writes, a single ladder's writes are already spread across 10 slots; if one ladder alone exceeds a shard's write capacity, split that ladder into more slots (a per-ladder shard count) — cross-link #18 hotspot hashing.

Degradation contract

  • A shard primary fails: Cluster promotes its replica in seconds; up to ~1 s of writes for ~1/10 of players may be lost and self-correct on their next submit_score (monotone scores). During the failover window, fan-out rank for the top-1K returns with an "≥" qualifier (one shard's count missing); approximate rank is unaffected (it reads the local histogram).
  • A shard is slow (GC pause): the fan-out has a per-shard 30 ms deadline; a laggard's count is dropped, the rank returns slightly low with exact:false. Never a hard fail.
  • The whole Redis Cluster is unreachable: API servers serve the last cached top-100 and the last global histogram — reads degrade to "as of N seconds ago"; submit_score returns 503 + Retry-After (the game server buffers and replays; the game DB still has the truth). No data loss (game DB is the SoR).
  • End-of-season read storm: 99.99% served from the 1-s top-100 cache and the Redis-free histogram; the exact-top-1K fan-out is single-flighted (one recompute/sec shared by all callers). Worst case: rank reads are 1 s stale. Acceptable.
  • Histogram drift: bounded to a few buckets; the hourly re-derive job overwrites it from ZCOUNTs. Rank error stays ±~5,000; the design already assumes that.

Operations (C6)

  • Key SLIs / alerts: per-shard instantaneous_ops_per_sec (alert > 80% of ~150K cap), p99 command latency (alert > 5 ms), Cluster health (alert on any FAIL), AOF fsync lag, replica lag (> 2 s), top-100-cache hit rate (< 99%), histogram-drift gauge (max |bucket - re-derived| > threshold), fan-out p99 + per-shard-deadline-exceeded rate, rank-API exact:false fraction (should be ~0 for the top-1K segment).
  • Rollout: the API tier is stateless — rolling deploy, one instance at a time (each rebuilds its caches on start in ~ms; keep ≤10% out at once so cache-miss bursts stay small). Redis upgrades: replica first, then failover, then ex-primary — one shard at a time.
  • Migration single-Redis → sharded (the actual cutover): (1) stand up the 10-shard Cluster empty; (2) dual-write: submit_score writes both the old single Redis and the new shard (the new write is fire-and-forget at first); (3) backfill: ZSCAN the old set, ZADD each member to its hash-shard, also seeding the histograms; (4) when the backfill catches up and the shadow read path matches the old one (compare ranks for a sample), flip reads to the Cluster; (5) stop writing the old Redis, decommission. Reversible at every step (the old Redis is still being written until step 5).
  • Adding shards (10 → 20): add Cluster nodes, migrate hash slots online (Redis Cluster does this live); the histogram bucket count B is unchanged (it's score-space, not shard-space); fan-out width goes 10→20 — still <5 ms parallel.
  • Headroom & why: shards target ≤54% of the ~150K ops/sec cap — the ~2× spare covers a rolling deploy (a server's cache rebuild = one extra fan-out), an end-of-season burst, and a slow shard shifting load. Don't run a leaderboard shard hot; a sorted-set op slows as the set grows, so the cap drifts down over a season of new players.

Cost (C7; cite cost-shapes#cache-heavy, pricing-constants.html)

This is a cache-heavy cost shape — the bill is RAM, not storage or egress. Naive baseline: the single Redis can't actually serve the load at all, so "baseline" is the smallest fleet that would have to brute-force it — ~26 large memory instances just to hit 260K ops/sec on 50M-entry sorted sets with no caching (it never converges — the point is it's an order of magnitude off). Final design: 20 Redis instances (10 primary + 10 replica), each a small memory-optimized box (4 GB working set, <1 vCPU of work) — an r6g.large-class node at ~$0.13/hr (cite pricing-constants#compute) × 20 × 24 × 30 ≈ ~$1.9K/mo for Redis; plus ~20 API servers (c6g.large-class, ~$0.07/hr) ≈ ~$1K/mo. Order of magnitude: ~$3K/mo for the leaderboard service — roughly 10× cheaper than the brute-force fleet, and dominated by per-instance RAM. The cost–latency trade-off: the approximate-rank histogram trades a ±0.01% rank error for not needing the ~26-instance fleet — the alternative (exact rank for everyone) would put shards at 73%+ and force more shards / read replicas, roughly doubling the Redis bill for an accuracy nobody perceives.

Key Takeaways

1. Sorted sets (skip lists) are the natural data structure for leaderboards

Redis sorted sets give O(log N) for score updates, rank queries, and range retrievals. No other data structure provides all three efficiently. The skip list backing structure makes ordered traversal fast (unlike a hash table) while keeping insertion fast (unlike a sorted array).

2. Shard by player_id, not by score

Scores change; player IDs do not. Sharding by player_id means a score update is always local to one shard (no cross-shard transactions). The tradeoff is that global queries (top-100, my rank) require fan-out across all shards, but parallel fan-out is fast (<1ms for 10 shards).

3. Approximate rank eliminates the most expensive query

Pre-computed score histograms let you answer "what's my rank?" without any Redis query at all — just a local array lookup. The accuracy trade-off (±5,000 positions) is invisible to 99.9% of players. For the top 1,000 where exact rank matters, fall back to the exact (but expensive) fan-out query.

4. Cache aggressively for slow-changing data

The global top-100 changes at most a few times per second. Caching it with 1-second refresh eliminates 10,000 fan-out queries per second. Friend scores with a 5-second cache reduce 5M ZSCORE/sec to 500K. In leaderboard systems, the biggest wins come from recognizing which data can tolerate short staleness.

Assumptions and When They Break

  • Scores are monotonically increasing: If scores can decrease (e.g., penalties, decay), the histogram maintenance becomes more complex (must track previous score for every update, not just the new score). Use a separate hash to store the current score for each player.
  • 50M players fit in 10 shards: If the player count grows to 500M, increase to 100 shards. Fan-out increases from 10 to 100 queries, but parallel execution keeps latency under 5ms. The histogram approach scales perfectly (just a larger sum).
  • Single leaderboard: If the game has multiple leaderboards (per-region, per-game-mode, weekly reset), each needs its own set of sorted sets. Multiply the shard count and memory accordingly. Weekly reset leaderboards can use a separate key prefix and be deleted en masse.
  • Friends list is small (~100): If a user can have 10,000 friends (social network style), the friends query becomes a full table scan. At that scale, consider pre-computing each user's friends leaderboard as a materialized view.

Anticipated pushbacks

What if there are 100× the players tomorrow — 5 billion accounts? (scale flip)

The structure doesn't change — the numbers do. Per-shard entries: scale shards 10→1,000 to keep ~5M entries/shard (the size at which a sorted-set op stays ~150K ops/sec). Fan-out width 10→1,000 — that's the only thing that gets worse: a 1,000-way parallel fan-out for the exact-top-1K rank is ~tens of ms even parallel, so you'd add an intermediate aggregation tier (each of ~30 aggregators fans out to ~33 shards, then a coordinator sums 30 partials) — classic tree fan-out. The histogram approach scales perfectly: the bucket count B stays 10K (it's score-space), each shard still publishes a 10K-int array, the global histogram is still an elementwise sum — the 99% rank path is unchanged. Bucket error: ~5M players/bucket now → ±0.001% of 5B — still invisible; if you wanted tighter, raise B to 1M buckets (4 MB/shard array — fine). Cost is linear in shards. Cite bottlenecks#fanout-asymmetry.

Why not just SELECT count(*) WHERE score > ? on a Postgres table with a B-tree on score? (alt-engine)

Three numbers kill it. (1) Write rate: 100K updates/sec, each rewriting an indexed column → B-tree page splits + WAL + MVCC dead tuples; a single Postgres tops out an order of magnitude below that and bloats. (2) Rank latency: even with the index, count(*) over scores above a median player visits ~25M index tuples — tens of ms, not the <1 ms a ZSET ZREVRANK gives (the ZSET keeps subtree-size-equivalent skip pointers; the B-tree doesn't, so it must walk). (3) You'd shard it anyway — and a sharded Postgres has the exact same cross-shard-rank problem as our sharded Redis, minus the in-RAM speed. Window functions (rank() OVER (ORDER BY score)) compute the entire ranking — O(n) per call. SQL wins only if you also need rich relational queries over scores; we don't. Cite bottlenecks#index-write-amp, storage-engines#btree.

What if exact rank isn't actually needed for anyone — how much cheaper? (requirement flip)

Drop the exact-top-1K ZCOUNT fan-out entirely; everything is histogram + the cached top-100 list. Per-shard load loses the ~1K ZCOUNT/sec (it was already tiny); the real win is operational — no fan-out path means no per-shard deadline, no tail-cascade risk, no single-flight machinery, and "my rank" is a pure local-array lookup at 100K QPS with zero Redis ops. You could even drop a couple of API servers. Conversely, if exact rank were needed for everyone: 100K QPS × 10-shard ZCOUNT = 1M ZCOUNT/sec → 73% per shard → you'd add ~2 more shards or read replicas — roughly 2× the Redis bill. The histogram is the cheap middle.

The Redis primary for shard 3 just died mid-tournament — what actually happens? (failure)

Cluster's gossip marks it FAIL within ~1–2 s; the shard-3 replica is promoted (it has the ZSET, the histogram, the AOF tail — everything, async-replicated, so at most ~1 s of writes behind). During the window: submit_score for the ~5M players hashing to shard 3 gets a brief Cluster redirect/error → the game server retries with backoff (and the game DB still has the truth). Fan-out exact-rank for the top-1K returns with one shard's count missing → reported as "≥ N" (or just exact:false with the histogram estimate). Approximate rank for everyone else: completely unaffected — it reads the API server's local histogram, which had shard 3's last-second snapshot. The few players whose shard-3 writes were lost: their score reappears at the slightly-older value and self-heals on their next submit_score (monotone — a stale lower value can't beat a fresh higher one). Net: a few seconds of degraded writes for 10% of players, zero data loss, zero impact on reads. Cite failure-modes#leader-loss.

Where's the cost, and which knob moves it? (cost)

Cache-heavy shape (cite cost-shapes#cache-heavy): the bill is per-instance RAM × number of Redis instances — ~$3K/mo total (20 Redis + 20 API servers, single region). The knob: shard count (drives Redis instance count linearly) and replication factor (RF=2 doubles Redis cost vs RF=1; we keep RF=2 because a leaderboard outage mid-season is bad PR). Going RF=3 (some games want a sync replica + an async one) ≈ +50% Redis. Multi-region = ×(number of regions) for the API + region/season ladders, plus a thin global-ladder replication path. The histogram itself is essentially free (40 KB/shard). What does not move the bill: more read QPS (the caches absorb it), more friends queries (5-s cache), edge-cached top-K.

How do you migrate from the single-Redis Naive design to the sharded one without an incident? (migration)

Dual-write + backfill, reversible at every step: (1) stand up the empty 10-shard Cluster; (2) make submit_score write the new shard in addition to the old single Redis (new write fire-and-forget at first, so a Cluster hiccup can't break live writes); (3) backfill — ZSCAN the old set, ZADD each member to hash(player)%10, seeding the histograms as you go; (4) shadow-read — for a sample of rank queries, compute the answer from both systems and compare; when they agree, flip reads to the Cluster; (5) stop writing the old Redis; decommission. Rollback before step 5 is just "flip reads back" — the old Redis is still current. (Same shape as adding shards later: online slot migration, no dual-write needed because Cluster does the move.) See the Operations block.

A player says their rank looks wrong — where do you look? (monitoring/debug)

Decision tree: (1) Is the player in the top-1K segment? — then the answer came from the exact fan-out; check the per-shard-deadline-exceeded metric (did a shard time out and get dropped, biasing the count low?) and the shard-failover log. (2) Below the top-1K? — the answer is the histogram estimate; expected error is ±~5,000 — if it's way off, check the histogram-drift gauge for that shard (a missed HINCRBY from a non-atomic write? the hourly re-derive job should have caught it — did it run?) and whether the API server's global_hist is fresh (its 1-s refresh wedged?). (3) Their score looks wrong, not just the rank — then it's a write-path problem: check the game DB (the SoR) vs Redis; a recent shard failover may have lost their last write — tell them to re-trigger it (monotone, so harmless). (4) Everyone's ranks are stale — Redis Cluster unreachable; API servers are serving cached snapshots; check Cluster health. Every branch maps to a What-Breaks entry and an alert; the drift gauge and deadline-exceeded counter exist precisely so this is a dashboard read.

End-of-season, everyone checks their final rank at once — thundering herd on reads? (stampede)

The design is already built for it. "My rank" for 99% of players is a local-array lookup — zero Redis ops — so the 100K→500K QPS spike hits API-server CPU, not Redis. "Top-100" is the 1-s cache — the underlying fan-out runs once/sec regardless of QPS, with stale-while-revalidate so a slow recompute never causes a miss. The only Redis-touching path is the exact-top-1K ZCOUNT fan-out — single-flight it (one in-flight recompute per second per leaderboard; all callers in that second share the result — cite single-flight.html, failure-modes#stampede). Net: a season finale is an API-tier autoscaling event, not a Redis incident. (Also: pre-warm by bumping the API fleet ahead of a scheduled reset.)
