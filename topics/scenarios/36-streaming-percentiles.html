Designing Streaming Percentiles for SRE Observability

200M time series, 50M samples/sec peak, 100 ms ingest-to-query freshness, ≤1% relative error at p99, 5k dashboard QPS. We start from a Postgres-with-raw-samples naive that the LC two-heap kernel fits perfectly — and that fails by 1000× on every NFR — and iterate to a sharded DDSketch pipeline with watermark-driven seals, on-demand window merging, and per-tenant cardinality control.

NALSD Walkthrough • ~45 min read • LC #295 (Find Median from Data Stream)

LC kernel. Find Median from Data Stream (LC #295). Textbook structure: two heaps — a max-heap of the lower half of the stream and a min-heap of the upper half. Insert is O(log N); the median is O(1) at any moment. Three properties that break at scale: (1) two heaps compute one quantile, but SLOs need p50, p95, p99, p999 simultaneously; replicating per-quantile heaps is brittle. (2) Per-series state is O(samples in window); 50M sps × 60 s = 3 B values to store for one 1-minute window across the cluster — not 200M, the cardinality, but 3 B, the sample count. (3) The structure is single-machine; nothing in two-heap composes across shards. The C8 production cousin replaces both heaps with a single mergeable summary (DDSketch) of bounded size that answers every quantile, ships across the wire, and reduces with bucket-wise sum — properties no LC variant exposes.

S0. Scenario & Requirements

An internal observability platform: every service in the company emits latency samples for every request. SREs build dashboards that ask "p99 latency for service=checkout, endpoint=/charge, region=us-east, over the last 5 minutes," and the answer must update within 100 ms of a real anomaly. Datadog, Honeycomb, Lightstep, Prometheus + Thanos, and the SRE platforms at Google / Meta / Twitter solve this same shape. The dashboard is the SRE's primary alerting surface; staleness or wrong percentiles cause real on-call incidents.

System class

Class: infrastructure-shaped — "operation X with semantic Y" capabilities (cite functional-requirements.html#class-infrastructure); the FRs are operations on a measurement stream + their semantic guarantees (window definitions, error tolerance, ordering), not user-facing verbs. The class fixes the NFR additions: ingest throughput dominates, error tolerance is a first-class NFR, and the bottlenecks lean toward cardinality-explosion + memory-capacity rather than user-facing latency.

Functional requirements (cite functional-requirements.html)

  • Ingest a sample. A service publishes (series_id, ts_ms, value_f64); the platform durably accepts and accounts for it within the freshness budget.
  • Query a quantile over a window. Given (tag_filter, quantile q ∈ {0.5, 0.95, 0.99, 0.999}, window ∈ {1m, 5m, 1h}), return the estimated value at that quantile within the relative-error guarantee.
  • List series. Given a tag filter (e.g., service=checkout), return the matching series_ids for dashboard auto-complete.
  • Subscribe to alerts. Given a panel + threshold (e.g., "p99 of /charge over 5m > 800 ms"), fire when crossed; tolerates the same staleness as queries.
  • Define a series. Tag schema registration: a tenant declares the dimensions of a metric (service, endpoint, region, instance, …); the catalog assigns the series_id for every observed (tag tuple).

Out of scope (named, not silently dropped): exact percentiles to floating-point precision (we trade for sketch error); event tracing (separate system; cf. Jaeger/Tempo); log search (separate system; cf. #30); metric forecasting / anomaly detection (separate ML pipeline; #37 if it exists); custom-percentile picking (q is restricted to {0.5, 0.95, 0.99, 0.999} so the sketch error budget can be sized once).

Flagged assumptions. (1) Sample distribution is roughly log-normal latency-shaped (most values in [1 ms, 10 s]); the sketch sizing assumes this. If a tenant emits values across 10 orders of magnitude (e.g., bytes 1 → 1010), DDSketch bucket count grows logarithmically — flagged so the cardinality-control iteration handles it. (2) Out-of-order skew is bounded at 30 s (consistent with NTP-synced fleet + collector buffering); a clock-skew tail of minutes would force the watermark math in Iteration 1 to be re-derived.

Sister scenarios (see Group K: Streaming percentiles)

  • K2 financial tick percentiles per symbol (planned) — sub-ms p99, in-memory per-symbol two-heap, exact-ish over a small window. Drastic flip: latency budget collapses; cardinality is small (~10k symbols); exact heap re-becomes the right answer.
  • K3 billing aggregations (planned) — exact, batch nightly, SQL roll-up. Drastic flip: approximate is forbidden; nightly batch is acceptable; sketches are the wrong tool entirely.
  • K4 ML drift-detection percentiles (planned) — online, drift-aware, alerting on distribution shift, not percentile values. Drastic flip: the value of p99 is less important than the change in the whole distribution; KS-test or earth-mover-distance over rolling sketches.
  • 24-realtime-leaderboard — the rank-not-value sister; t-digest beats DDSketch when uniform rank error matters.
  • #37 metric-window anomaly (planned) — sliding-window max/min on the same ingest stream; reuses partitioning + watermark plumbing.

The fundamental tension. The two-heap kernel is exact and per-stream; observability is approximate and 200M concurrent streams. Three constraint-pairs collide: (a) cardinality vs memory — per-series exact state grows with both N (samples) and K (series); (b) freshness vs persistence — 100 ms freshness rules out batch jobs and disk-buffered writes; (c) error tolerance vs queryability — we can trade exactness for bounded memory only if a single sketch answers every supported quantile and merges across shards. DDSketch is the only structure on the menu that satisfies all three: bounded size (geometric bucketing), bucket-wise mergeable (additive counts), and value-relative-error guaranteed at every quantile (constant-relative-width buckets). The arc resolves the tension: shrink per-series state to a sketch, partition by series, seal on watermark, merge at query time, cap the cardinality.

ID scheme (cite identifier-schemes.html#hash-id): series_id = murmur3_128(canonical(tag_set)) truncated to 64 bits. Why hash-of-tags rather than Snowflake: the same (service=checkout, endpoint=/charge, region=us-east, instance=i-1234) tuple from two collectors must produce the same series_id without coordination — Snowflake gives a unique-per-write id, which is the wrong primitive when multiple writers contribute to one series. Collisions: 64-bit hash gives ~1 in 4 B birthday collision at 200M cardinality ≈ 0.5% expected collisions; we accept this and detect via tag-set fingerprint check on series-catalog write.

The partition-key heuristic: the dominant access pattern is "update sketch for series X on every sample" (50M times/sec) and "merge sketches for series matching tag filter" (5k times/sec). Both are series_id-locality at the leaf, with a fan-in from many ingest pods to one (series, window) sketch (write-side hot-key) and a fan-out from one query to many series (read-side fan-in). Hash by series_id at the sketch tier so the per-series hot-spot is a single Redis slot; range partition the sealed-sketch S3 layout by (shard_id, hour) so historical bulk reads are a few large objects.

The hot Redis tier runs at RF=1 (in-RAM only) precisely because the SoR is Kafka, not Redis; this halves the hot-tier cost vs RF=2.

S4. Naive Architecture — the simplest thing that meets the FRs

One Cassandra cluster holds every raw sample as a row: samples((series_id), ts_ms), partition by series_id, clustering by ts_ms. Each ingest collector writes every sample; each query SELECTs samples in the requested (series, window) and computes the percentile by streaming the values into the LC two-heap structure on the query coordinator. This is the LC #295 reference solution lifted directly: per query, fetch all samples in the window, run two heaps, return the median. For p95 / p99 / p999 we run a sort-and-index step over the fetched window (O(N log N)) since two-heap only gives the median.

Why it's broken (per failing NFR)

  • Freshness. ❌ ingest→query lag is 200+ ms p99 even at idle (Cassandra commit-log + range scan); the 100 ms target is missed by 2×. Worse: under load (real ingest at 50M sps), the LSM compaction stalls push p99 to seconds.
  • Storage cost. ❌ 7-day retention costs $655k/mo just for storage; 30-day retention is structurally infeasible at this hardware shape. Even with 5× LSM compression and dropping to 1-day retention, the storage floor is $90k/mo — and you've now lost the historical query path.
  • Quantile error. ❌ the LC two-heap is exact and computes median only. p95/p99/p999 require either separate per-quantile structures (memory blows up further) or a full sort over the fetched window (CPU and latency multiply). The "1% relative error at p99" NFR isn't just unmet — it's not even definable in this design (the answer is exact when it works, but the design doesn't work at scale).
  • Ingest throughput. ❌ Cassandra at 30k writes/node × 250 nodes = 7.5M sustained — plus burst headroom, ~10M with batching — misses 50M target by 5×. Aggressive batching (1000 samples / batch) helps but degrades freshness further.
  • Query throughput. ⚠️ 5k QPS × 30 fan-in = 150k partition reads/sec is just barely feasible at idle but collapses under contention with the ingest write load on the same nodes.

Cost of naive at NFR-target load: infeasible. Even at the floor of $400k+/mo (1-day retention, 5× compression, no replication headroom for failures), this design still misses the freshness NFR by 2× and the quantile-error NFR by definition. The first failing NFR is quantile error — the LC kernel only computes the median; we've left p95/p99/p999 structurally unaddressed. The second is freshness, dominated by Cassandra's commit-log + range-scan path. We must change the data structure — not scale Cassandra.

Naive schema (Cassandra)

CREATE TABLE samples (
  series_id    bigint,                       -- partition key (hash; uniform per cardinality)
  ts_ms        bigint,                       -- clustering key (ASC)
  value        double,

Bottleneck

Naive fails freshness (200+ ms p99 ingest→query lag), storage cost ($655k/mo on Cassandra alone), and quantile-error (LC two-heap computes only the median, p95/p99/p999 require sort over the fetched window). Classified as cardinality-explosion compounded with coordinator-bottleneck on the query path: per-series state grows with sample count and series count (the canonical "per-key state × N grows unboundedly with input cardinality"). The corresponding access pattern matches fan-in-read for queries (one panel → many series→samples) compounded with write-heavy-uniform for ingest. Iteration 0 fixes both axes simultaneously: shrink per-series-per-window state to a constant-sized sketch, and shard by series_id so the per-shard load drops linearly with cluster size.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#cardinality: sketch swap (CMS / HLL / t-digest / DDSketch / Bloom); cap N with eviction; ingress sampling; tiered (exact for top-K hot, sketch for long tail). The dominant lever for "per-key state grows unboundedly" is the sketch swap; sampling is rejected because it would forfeit the relative-error guarantee at p99 (sample-rate determines tail accuracy directly).

Alternatives considered (C4)

  • DDSketch — chosen. Per sketches#dd-sketch: bucket index = ⌊logγ(value)⌋ for γ = (1+α)/(1−α); at α=0.01, γ ≈ 1.0202, ~1500 buckets cover [1 µs, 100 s] of latency. Critical property: strict relative-error guarantee on the value, not the rank — the SLO says "p99 = 142 ms ± 1%" not "p99 within 1% of true rank position." Mergeable: bucket-wise sum.
  • t-digest — rejected. Per sketches#t-digest: bounded relative rank error, not value error. For a latency distribution where p99 might be 80 ms and p999 might be 800 ms, t-digest's rank error means the value at p99 could be reported anywhere in the upper tail. Wins for the leaderboard rank sister (cite #24); loses here. (🔀 If the SLO were "alert when the rank of 200 ms drops below the 95th percentile" rather than "alert when p99 > 200 ms," t-digest would win.)
  • HDR Histogram — rejected. Fixed pre-allocated buckets at 2 significant figures; ~256 KB per histogram (vs DDSketch's 6 KB). 200M series × 256 KB = 51 TB hot state — 14× bigger. HDR's "configured value range" also forces a per-tenant configuration burden (latency vs throughput vs queue-depth distributions need different ranges); DDSketch's logarithmic bucketing is range-agnostic.
  • Approximate quantile via reservoir sample (Algorithm R) — rejected. Cite sketches#reservoir: O(K) memory per series, O(1) per sample. For K=10,000 the relative error at p999 is ≈ 30% — misses the 1% NFR by 30×. Reservoir wins when you want to retain example values (a stratified-sample of slow requests), not when you want a quantile.

The chosen move

For each sample (series_id, ts_ms, value): (a) compute the DDSketch bucket index k = ⌊logγ(value)⌋ with γ = 1.0202 (one multiplication + one log; ~10 ns); (b) atomically increment the bucket count in the Redis hash sk:<series_id>:1m:<ts_ms / 60000>. The whole per-sample path is one HINCRBY. For each query: load the sketches for the relevant (series, window) tuples, merge bucket-wise, and answer the quantile by walking the merged buckets to find the one containing the target rank.

Cite sharding-strategies.html#hash: hash partition by series_id for Redis Cluster (uniform load; the dominant access is per-series). Cite storage-engines.html#kv-engine for Redis as the engine: the hot-path operation (atomic field-increment under a key) is exactly what Redis's hash type provides at sub-ms latency. Why RF=1 instead of RF=2: Kafka is the system of record (RF=3 for the sample stream). Redis going down means we replay the last 6 hours of Kafka into a fresh sketch tier; recovery time ~10 min for a single shard. Halving the Redis fleet from RF=2 saves ~$50k/mo.

What breaks (C5; cite failure-modes.html)

  • Hot-key meltdown on top-1% series. A high-volume series (e.g., the company-wide HTTP /healthz endpoint) generates 100k samples/sec to a single Redis HINCRBY field. At 100k ops on one slot, Redis single-shard maxes at ~80k HINCRBY/sec; the queue depth on that shard's command pipeline grows. Detection: per-key ops/sec p99 exceeds 50k. Recovery (this iteration): noisy-neighbor protection — client-side coalesce (each ingest pod keeps a per-(series, 1m-window) local counter, flushes every 100ms or 1k samples to one HINCRBY). 100× reduction. Iter 1 makes the sketch lock-free (atomic-add bucket counts) so per-key concurrency stops being a wall.
  • Backlog overflow on Kafka samples topic. A burst (a noisy-neighbor tenant emits 200M samples in 30 seconds) backs up the Kafka log faster than ingest workers consume. Detection: Kafka consumer-group lag > 5 min, monotonically growing. Recovery: per-tenant ingest quota at the gRPC gateway with 429 backoff (cite api-design-patterns#rate-limiting); in extreme cases shed by tenant priority (free-tier first). Address structurally in Iter 3 (cardinality control).
  • Stale-data inconsistency on sketch tier reboot. A Redis sketch node reboots; the sketch keys it owns are gone (RF=1). Recovery: replay the last 6 hours of Kafka samples for the partitions that map to the rebooted shard's slot range. Detection: shard-up health probe shows 0 keys; query coord receives 206 Partial Content for queries hitting that shard. Recovery time: 5–10 min for one shard, gated by Kafka replay throughput. Side effect: queries during recovery return partial results (the warning header surfaces this); SREs accept ~1% degraded coverage during a single shard rebuild.

Bottleneck

Iter 0 carries 3 sketches per series in hot RAM (1m + 5m + 1h concurrently) for queries that span every supported window. The 5m and 1h sketches are derivable from N consecutive 1m sketches via DDSketch's bucket-wise additive merge — cite sketches#dd-sketch's mergeability section. Carrying them in hot RAM is wasteful: memory-capacity-bound, where the per-key state is 3× what the freshness NFR actually demands. Secondarily, with many ingest pods feeding samples to the same (popular series, current 1m) sketch, the per-key write rate hits Redis's HINCRBY single-slot ceiling of ~30k/sec — classic lock-contention on the per-(series, current-1m) hot key.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#memory-capacity: tier to slower-larger medium (NVMe, S3); shrink representation (compact serialization, delta); shard the dataset; sketch (give up exactness for bounded memory). We've already swapped to a sketch (Iter 0); the next move is tier the historical state to S3 while keeping the live (currently-updating) sketch in RAM. Per bottleneck-to-decisions.html#lock-contention: lock granularity (shard the lock); lock-free / CAS structures. DDSketch buckets are independent atomic counters — the per-bucket lock granularity is already as fine as it can go; concurrent ingest writes to different buckets within the same sketch don't contend. The remaining contention is per-bucket, which Redis HINCRBY handles natively (single-threaded server with atomic command processing).

Alternatives considered (C4)

  • Watermark-driven seal-and-persist + on-demand window merge — chosen. The watermark = max_observed_ts − allowed_skew; window N is "closed" when the watermark advances past N's end timestamp. At that point the 1m sketch is immutable and can be sealed (read-only, packed, persisted). Late samples (after seal, within the 30-second skew bound) merge into a delta sketch that's reconciled on the next compaction pass. Wins because: (a) hot state shrinks 3× (1.2 TB → cluster fits in 15 Redis nodes); (b) historical state moves to S3 at $0.023/GB-mo (vs Redis at ~$12/GB-mo); (c) on-demand window merge of 5 × 1m sketches is ~25 µs — well under the query budget.
  • Concurrent 1m + 5m + 1h hot sketches — rejected. Iter 0's design. Pays 3× RAM for state that's deterministically derivable from the 1m sketches. The argument for keeping it would be "5m/1h queries are slow if we merge on demand" — but on-demand merge of 5 sketches is 25 µs and 60 sketches is 300 µs, both well within the 200 ms p95 budget. (🔀 If 1h queries dominated 1m queries 100:1, pre-aggregating 1h would win on read latency — but at 5k QPS spread across windows, on-demand merge remains cheap.)
  • Pre-aggregate to NVMe (instead of S3) — rejected. NVMe r6id at $1.30/GB-mo is ~50× more expensive than S3, while the read pattern for sealed sketches is power-law (most queries hit recent sketches; older ones rarely accessed). The Iter 2 sealed-sketch cache covers the warm-read case; S3 covers the cold tail at minimal cost.
  • Force per-(series, 1m) update lock at app layer — rejected. Would prevent concurrent updates to the same sketch from different ingest pods, but at the cost of throughput. DDSketch's bucket-wise atomic counts are the lock-free primitive; serializing at the sketch level instead of the bucket level is a 1500× worse granularity. Cite bottlenecks#lock-contention: "shard the lock" — we've already sharded all the way to the bucket; that's the natural granularity.

The chosen move

Three tightly-related techniques, all addressing the same bottleneck:

  1. Drop 5m and 1h hot sketches. Hot Redis stores only the current-1m sketch per series. 5m queries merge 5 consecutive 1m sketches on the fly; 1h queries merge 60. Both fit in the query budget.
  2. Watermark-driven sealing. Each ingest worker tracks the max sample ts_ms it has seen on its assigned partitions. The cluster-wide watermark is the min across all workers' max (a standard min-of-max watermark; cite the streaming-systems literature). When the watermark advances past a 1m window-end, the Sealer signals that minute's sketches are immutable. Late samples within the 30-second skew bound merge into a delta sketch, reconciled at next compaction.
  3. Lock-free atomic bucket updates. Each DDSketch bucket count is a Redis hash field; HINCRBY is atomic per-field. Concurrent ingest pods writing to different buckets of the same sketch don't contend. Per-bucket contention (rare except at the very tail of latency distributions) is bounded by Redis's single-threaded command pipeline, ~80k HINCRBY/sec/slot. Client-side coalescing (introduced in Iter 0's failure-modes block) absorbs the residual hotness.

Cite sketches#dd-sketch for DDSketch mergeability; cite the cardinality-explosion → tiered: exact for top-K hot, sketch for long tail decision pattern (we generalize: hot for current window, cold-storage sketch for sealed windows).

What breaks (C5; cite failure-modes.html)

  • Stale-data window misalignment on watermark stall. If a single ingest worker stalls (GC pause, OOM kill), its partition's watermark stops advancing; the cluster-wide min-of-max watermark stalls with it. Detection: per-partition watermark progress monitor; alert if any partition's watermark lags the median by >30s for >1 min. Recovery: restart the stalled worker; partitions rebalance via Kafka cooperative-sticky reassignment; the watermark catches up within seconds. Workers maintain a "tentative seal" mode: they can advance a sketch's seal optimistically when their local watermark is past window-end and they've received heartbeats from peers within the last 10s; if peer heartbeats fail, they hold the seal.
  • Data loss on Sealer crash mid-write. If the Sealer crashes after reading a sketch from Redis but before completing the S3 PUT, the sketch may not be persisted; the source-of-truth Kafka log still has the underlying samples, so re-derivation is possible. Detection: completed-seal manifest in Postgres; missing sealed-window entries trigger a replay from Kafka. Recovery: re-run the sealer on the affected window; the in-memory sketch is rebuilt by replaying Kafka samples through a fresh DDSketch update path; idempotent because DDSketch update is commutative.
  • Cache-miss stampede on cold sealed-sketch read. A sudden interest in a 24-hour-old time range (say, an SRE investigating yesterday's incident) triggers many parallel S3 GETs for the same set of sealed sketches; without single-flight, 50 query-coord pods all issue the same GET. Detection: sealed-cache miss rate spike + S3 GET rate spike on identical keys. Recovery: single-flight per (sealed_object_key) in the sealed-sketch cache — cite single-flight.html; introduced cleanly in Iter 2.

Iteration 2 — Panel-result cache + single-flight on miss + per-cell co-location

System after this iteration

Per-cell legend. Each cell holds ~1/3 of the series (hash-partitioned by series_id) plus its own ingest, sketch tier, query coord, panel cache, and sealer. Within-cell traffic is the dominant hot path. Cross-cell traffic appears only when a query's tag filter spans series resident in multiple cells (which is a small minority — tag filters are typically tenant-scoped, and tenants are deliberately co-resident on one cell). Delta vs Iter 1: Panel-result cache (TTL = window/6) sits in front of the query coord; single-flight (cite single-flight.html) on cache miss prevents stampede. Cell co-location collapses cross-AZ egress on the hot path: ingest writes to cell-local Kafka, sketch tier is cell-local, query coord reads cell-local sketch tier. Cross-cell only on multi-cell tag filters (~2% of queries).

Bottleneck

Iter 1 still does a full sketch-merge on every query: 5,000 QPS × 30 series × 5 minute-sketches = 750,000 sketch fetches/sec. Most queries are repeat queries — the dashboard auto-refreshes every 10s, so the same panel issues the same query thousands of times per minute. Caching the panel result (the answered quantile value, not the raw sketches) at TTL = window/6 collapses 90%+ of the read load. Without single-flight, a cache miss on a popular panel would have N pods all racing to recompute it — cache-stampede on the hot panel. Separately, Iter 1's $31k/mo cross-AZ egress is dominated by ingest→sketch traffic that crosses AZs in a single-cell layout; cells co-locate ingest and sketch tier in the same AZ to drop this cost. Combined classification: fan-in-read + stampede + network-bandwidth-bound on cross-AZ.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#stampede: single-flight per key, jittered TTL, request coalescing. Per bottleneck-to-decisions.html#network-bandwidth: co-locate writers with readers; compress on the wire; reduce read fan-out; cache hot intermediate results. Per cross-cutting#cells: cells are the canonical anti-correlation move — one cell's failure caps blast radius at 1/N.

Alternatives considered (C4)

  • Panel-result cache + single-flight + cells — chosen. Three techniques, one bottleneck-cluster. The cache stops 90% of sketch-merges; single-flight bounds the worst-case stampede; cells drop the egress.
  • Pre-merge "hot panels" at ingest time — rejected. Subscribe each ingest pod to the active panel definitions and pre-merge sketches into the panel-result on each sample. Wins on read latency (panel is always hot in cache). Loses on coupling: 1M panel definitions × ingest pods would explode the per-sample work; the panel set is dynamic (SREs add/remove panels constantly), so subscriptions churn. (🔀 If panel count were fixed at <10k and rarely changed, this would win — but the dashboard ecosystem creates panels at SRE typing speed.)
  • CDN/edge cache instead of internal panel cache — rejected. Dashboard data is per-tenant, often privacy-sensitive (latency curves can leak business volume); CDN caching requires careful tenant-scoping and cache-key design. The internal Redis panel cache is simpler and the latency saving is small (the network from CDN POP back to origin already dominates over a tens-of-ms internal cache lookup).
  • Compress sketches on the wire (zstd) instead of cells — rejected as a stand-alone fix. A sparse 1500-bucket sketch with ~50 active fields compresses well (~5×), but the wire-cost is a fraction of the cross-AZ egress on writes. Cells fix the dominant cost; we'll still apply zstd to query-time sketch transfers as a free secondary win.

The chosen move

(a) Panel-result cache. A small Redis cluster per cell, keyed by panel_cache_key = sha1(panel_id, q, window, ts_bucket) where ts_bucket = floor(now / (window / 6)). TTL = window/6 (10s for 1m, 50s for 5m, 600s for 1h). Cache stores the answered scalar value (a double) plus the sketch_version + count, so the dashboard can render an "as-of timestamp" and an "n samples" annotation. ~250 bytes per entry; with ~1M unique panels and TTL-rotation, cache size ~3 GB per cell.

(b) Single-flight on cache miss. On miss, query coord acquires a Redis-Lua mutex on the panel_cache_key (TTL = max-query-time = 1s). Concurrent misses block on the mutex via a short blocking-pop on a per-key Redis list (the Resolver pattern). The first miss does the work; followers receive the result via the list. Cite single-flight.html for the canonical move; this page reuses the move (does not redefine it).

(c) Per-cell co-location. Each cell holds ~1/3 of series_ids (hash-partitioned). Ingest in cell A only consumes the Kafka partitions for series resident in cell A; the sketch tier in cell A holds those series; query coord in cell A serves panels whose tag filter is satisfiable from cell-A series. Tenants are pinned to one cell per cell-affinity tag (their dashboards naturally hit cell-local data). Cross-cell queries (rare; they happen when a tag filter is intentionally global) fan out via cell-to-cell merge, paying the cross-AZ egress only on those.

Bottleneck

Iter 2 holds 200M series in hot tier today; the trajectory is 500M+ within a year as Kubernetes pod churn, customer-id dimensions, and bot-driven endpoint URLs add ephemeral series. Cardinality-explosion — the canonical "'just add more RAM' — the input grows faster" anti-pattern. Single noisy-neighbor tenants who instrument with high-cardinality tags (e.g., user_id as a metric tag) can 100× the platform's series count overnight, evicting other tenants' sketches under memory-capacity-bound. Secondarily, sealed-sketch storage grows linearly with retention × cardinality — without rollup-and-evict, 1m sketches kept for 13 months would be ~5 PB of S3 ($115k/mo just for storage).

Architectural decisions to revisit

Per bottleneck-to-decisions.html#cardinality: Sketch swap (we did Iter 0); cap N with eviction (LRU); ingress sampling; tiered (exact for top-K hot, sketch for long tail). Alternatives considered (C4)

  • Per-tenant cardinality quota + dimension-drop fallback + rollup retention — chosen. Three techniques, one cluster of bottlenecks. Quota is the gate (rejects overflow); dimension-drop is the graceful-degradation path (preserves a useful aggregate when quota is hit); rollup-and-evict bounds storage growth.
  • "No quota; just buy more RAM" — rejected. The canonical anti-move. Cardinality grows superlinearly with users + dimensions; capacity-driven scaling alone shifts the failure mode from "platform OOM" to "billing surprise."
  • Reservoir sampling at ingress — rejected. Would bound the per-series sample volume but not the series count itself; the explosion is in K (distinct tag tuples), not N (samples per series). Wrong dimension to sample on.
  • Strict reject above quota (no aggregation fallback) — rejected. Wins on simplicity; loses on user experience: an SRE who instruments user_id as a tag is surprised when the panel disappears entirely. The dimension-drop path "user_id excluded; service+endpoint preserved" is the graceful degradation.

The chosen move

(a) Cardinality Enforcer. A horizontally-scalable in-line check at the ingest gateway. Maintains a per-(tenant, day) HyperLogLog of distinct series_ids (cite sketches#hll: ~12 KB per tenant-day at 1% rel err on cardinality estimate). On each batch, the enforcer estimates whether adding this batch's series would push the tenant over quota. If under quota: pass. If over quota: try dimension-drop — strip one tag (selected per tenant config; default is the highest-cardinality tag observed) and re-estimate; if the dimension-dropped series fit, pass with a warning header. If still over: 429 with Retry-After, error body explains which dimension(s) caused the overflow.

(b) Rollup Worker. A scheduled job (one per cell, runs every minute) that drives the 1m → 5m → 1h cascading rollup:

// Rollup cadence:
every 5 minutes:
  for each (series_id, last 5 1m-sketches in S3):
    merged_5m = bucket_wise_sum(1m_sketches)        // DDSketch is mergeable
    write_s3 sealed/5m/{series_id}/{w_start_5m}.parquet
    delete_s3 sealed/1m/{series_id}/{w_start_1m}.parquet  // evict prior tier

every hour:
  for each (series_id, last 12 5m-sketches in S3):
    merged_1h = bucket_wise_sum(5m_sketches)
    write_s3 sealed/1h/{series_id}/{w_start_1h}.parquet
    delete_s3 sealed/5m/{series_id}/{w_start_5m}.parquet

every day:
  scan_for series with last_sample_at < now - 30d
  → mark_inactive in series_catalog (stop scoring against tenant quota; sketches expire naturally)

(c) Tiered retention with eviction. 1m sketches kept 6 hours (covers the freshness query window for "the last hour at 1m granularity"); 5m sketches kept 7 days (dashboard zoom-out); 1h sketches kept 13 months (capacity-planning, on-call retros). Retention is enforced by S3 lifecycle rules in addition to the rollup worker's explicit deletes (defense in depth).

What breaks (C5; cite failure-modes.html)

  • Data loss on quota rejection. A tenant exceeding quota loses samples (429 from the enforcer, no buffering). Detection: per-tenant rejected-sample count metric. Recovery: client-side retry isn't valid because the rejection is a steady-state condition (the tenant's tag schema needs adjustment); customer-success engages with the tenant to either (a) bump quota (paid plan), (b) drop a high-cardinality dimension, or (c) consolidate via metric-renaming. The enforcer surfaces "which dimension caused the overflow" in the rejection so the SRE can fix it without log-spelunking.
  • Rollup miss. If the Rollup Worker for a cell crashes for hours, 1m sketches accumulate beyond the 6-hour retention; S3 storage spikes. Detection: 1m sketch count in S3 vs expected (rate × retention); alert if 50% above expected. Recovery: restart rollup worker; it catches up by processing a backlog from the sealed_manifest in Postgres. Idempotent because rollup is "merge N sketches into one" — running it twice produces the same result; an aborted rollup leaves the inputs untouched.
  • HLL drift on enforcer restart. An enforcer pod restart loses its in-memory HLL until it rehydrates from Redis. During the rehydrate window (~10s) the pod under-counts cardinality. Detection: per-pod HLL state-age metric. Recovery: pod warmup blocks the live ingest path until HLL is rehydrated; a 10s freshness sacrifice on a quota check is preferable to under-enforcement.

Probe A — Multi-region active-active

Bottleneck. Single-region serves global SREs; cross-Atlantic dashboard-load adds 80 ms RTT to every cache miss. Per-dataset CAP audit: sketches are CRDT-friendly (bucket-wise sum is a commutative monoid; multi-region async replication with bounded staleness is fine for an SRE dashboard; the "1% relative error" NFR absorbs the slight inflation from late merges); panel cache is per-region (no replication); tenant config requires CP (single-leader writes; eventually-consistent reads region-locally with bounded lag). Cost: +1.5× for second region (per-region active-active sketch tiers; replicate sealed sketches via S3 Cross-Region Replication). Cite replication-decisions#multi-leader-async.

Probe B — Exact-percentile audit path

Bottleneck. Some compliance regimes (financial trading; medical-device latency reporting) require exact percentiles, not 1%-relative-error sketches. We can't make the sketch tier exact, but we can ship a parallel sample-store for tagged-as-audit series: every Nth sample (head sampling at 1/100) goes to a Cassandra raw-sample table for that series; nightly batch jobs compute the exact percentile and store the audit answer. Cost: ~$5k/mo per audit-tagged tenant for the sample storage; deliberately expensive to disincentivize over-tagging.

Probe C — Anomaly detection on the same ingest stream

Bottleneck. The same samples that feed sketch tier could feed an anomaly detector (sliding-window max/min over 5m, alert on Nσ deviation). Reuses the Kafka samples topic + ingest-worker plumbing; adds an "anomaly worker" consumer group. Cite bottlenecks#backlog-growth + cross-link to the planned #37 metric-window anomaly scenario where this is the primary kernel.

Probe D — Cost consolidation toward the brief's $30k target

Bottleneck. Iter 3 lands at ~$84k; the brief's $30k target requires aggressive consolidation: (a) co-locate sealer + rollup + enforcer on shared pods (3 worker tiers → 1) saving ~$10k; (b) Kafka KRaft on denser r6g.8xlarge nodes (16 brokers → 6 brokers at 2.5× throughput) saving ~$8k; (c) self-host the Postgres tier on a single multi-AZ replica pair (already done) saving negligibly; (d) shrink panel cache to LRU on hot tier (panel cache as a first-class tier disappears) saving ~$8k; (e) retire the m6i.4xlarge for the lighter workers (m6i.2xlarge at $277/mo for sealer/rollup) saving ~$5k. Real-system comparison

Datadog's metrics platform, per their DDSketch engineering post and the underlying DDSketch paper (Masson, Rim, Lee, 2019), took the same arc but with their proprietary M3-style storage. The DDSketch error model and merge algorithm in this page are taken directly from that paper. Their "100ms freshness" target maps to ours; their cardinality control story is similar (per-tenant tag quota), and they publish that they aggressively rollup-and-evict 1m sketches after 24 hours.

Druid's DataSketches quantiles extension uses t-digest as the default; the moments-sketch extension is a different mergeable structure. Druid's choice of t-digest reflects a different constraint mix (rank-error-tolerant analytical workloads) than ours (value-error-bounded SLO alerts).

Prometheus + Thanos take an entirely different architectural stance: they ship histogram buckets with the sample (the user pre-decides bucket boundaries) rather than an adaptive sketch. This works for pre-defined Grafana dashboards but breaks for ad-hoc queries on un-bucketed data — one of the drivers behind Prometheus's native histograms work, which moves toward DDSketch-like adaptive bucketing. Where this design diverges: we ship adaptive sketches per (series, window) so any quantile is queryable from any window without prior bucket choice. Why DDSketch and not t-digest? (alt-sketch)

  • The NFR phrasing. The constraint says "≤ 1% relative error at p99" — that's an error on the value, not the rank. DDSketch's bucket index = ⌊logγ(value)⌋ gives uniform relative-on-value error; t-digest gives uniform relative-on-rank error.
  • Concrete failure mode for t-digest. If the actual p99 value is 80 ms but t-digest places the rank-99 boundary at a centroid covering [60, 200]ms, the reported p99 could be anywhere in that range; SLO alerts on "p99 > 100 ms" become noisy.
  • When t-digest wins. Leaderboard percentile-rank queries (#24): "what rank does value 200 sit at?" → exact value-bound is irrelevant; you want uniform rank precision.

What if cardinality grew 10× to 2 billion series? (scale flip)

  • Hot tier scales linearly: 2 B × 6 KB = 12 TB → 150 Redis nodes (was 11). Cost ~$150k just for hot tier.
  • Cardinality Enforcer load amplifies: HLL state per-tenant grows; a tenant with 10M series per day needs ~12 KB HLL (still small), but the enforcer's CPU at 5µs/check × 500M sps = 2,500 cores → 80 enforcer nodes (was 16).
  • Sealed S3 storage: ~7 TB → still <$200/mo. Storage isn't the constraint; ingest CPU is.
  • New dominant bottleneck: per-tenant fairness becomes acute — the enforcer needs adaptive per-tenant rate-limiting (TGCRA or fair-share scheduling at the gateway), not just static quotas.

The sketch tier crashes; what does the dashboard see? (failure)

  • For affected shards: queries return 206 Partial Content; dashboard renders "data partial" banner with the cells/shards reporting.
  • Recovery: Kafka replay rebuilds the lost shard's sketches in ~5–10 minutes; the SoR is the Kafka log (RF=3 across AZs).
  • RPO: <5 seconds (Kafka acks=all + sync replicas); RTO ≈ 10 minutes for a single shard, ≈ 30 minutes for whole-cluster.
  • The risk-mitigation knob: we run sketch tier at RF=1 because Kafka is the SoR; running RF=2 would halve recovery time at 2× the cost. Kept at RF=1 because 10-min RTO is acceptable for a dashboard (alert evaluators have their own watchdog / dead-man-switch).

How do you handle a tenant who instruments user_id as a metric tag? (noisy-neighbor)

  • First detection: the Cardinality Enforcer's HLL spikes per (tenant, day); the tenant hits cardinality_quota within minutes.
  • Graceful degradation: the enforcer's dimension_drop_priority column is consulted — user_id is the default top-priority drop; the enforcer aggregates over user_id (i.e., strips the user_id tag and re-counts) before rejecting samples.
  • Customer-success path: the rejected-with-reason header lands in the tenant's settings page; SREs see "user_id excluded; service+endpoint preserved"; they can either remove the tag at source or upgrade their plan.
  • Why dimension-drop, not strict reject: the typical accident is "added user_id for one ad-hoc debug session"; strict reject would lose all data; dimension-drop preserves the aggregate without unbounded cardinality.

Where does the bill come from? What's the cost knob? (cost)

  • ~90% is compute + cache: Kafka $16k, sketch tier $11k, panel cache $9k, ingest workers $13k, enforcer + workers $19k. Storage on S3 is ~$16/mo (negligible).

How do you migrate from Naive (Cassandra raw samples) to Final without dashboard downtime? (migration)

S7. Key Takeaways

  • The two-heap kernel is a single-stream artifact. Once K (number of streams) and N (samples per stream) both grow, the right move is to shrink per-stream state to a constant-sized mergeable summary; the LC kernel's exactness is incidental at the system-design scale.
  • Pick the sketch by error model, not by name. DDSketch (uniform relative-on-value), t-digest (uniform relative-on-rank), HDR (fixed-resolution bucketed), reservoir (sampled-with-retention) all compute "percentiles." The constraint — "1% rel err at p99" vs "rank precision" vs "exact retained samples for compliance" — picks one, not aesthetics.
  • Mergeability is the unlock. A non-mergeable sketch forces all-data-on-one-node, defeating the bounded-memory goal. DDSketch, t-digest, HLL, CMS, Bloom all merge with a commutative monoid; reservoir and cuckoo do not. When in doubt, draw the merge function.
  • Watermarks separate "freshness" from "completeness." A 1m sketch is "ready to seal" when the watermark says no more in-bound samples are expected for that window. The watermark math, not clock-wall time, is what makes streaming-percentile pipelines correct under bounded skew.
  • Cardinality is the silent killer. The platform-wide failure mode isn't "too many samples per second" — it's "too many distinct (tag-tuple) series." The Cardinality Enforcer with HLL + per-tenant quota is the gate; without it, one noisy tenant evicts everyone's sketches.
  • Tiered retention with rollup is the answer to "store everything forever." 1m sketches at 6 hours; 5m at 7 days; 1h at 13 months. Each tier is the merge of the prior tier; storage is bounded by retention × cardinality, not by sample volume.
  • Each iteration cited a canonical bottleneck. Naive → cardinality-explosion + coordinator-bottleneck. Iter 0 → memory-capacity-bound (3-window waste). Iter 1 → fan-in-read on query path. Iter 2 → cardinality growth + per-tenant fairness.   • Cost shape inverts vs cache-heavy systems. Where #41 (timeline) is dominated by RAM + storage for fast reads, #36 is dominated by compute for atomic sketch updates and the Kafka SoR.
