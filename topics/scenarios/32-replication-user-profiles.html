Database Replication: Global User Profile Service

Designing the storage layer for 200M user profiles across 3 regions — why replication is the right tool, and the challenges it introduces.

NALSD Walkthrough • ~20 min read

The Scenario

You are designing the storage layer for a global user profile service. Every time a user logs in, views someone’s profile, or changes their settings, this service handles the request. It must be fast, reliable, and serve users on three continents.

Data

  • 200 million user profiles. Each profile contains: name, email, avatar URL, preferences (language, theme, notification settings), timezone. Average size: ~1 KB per record.
  • Total data size: 200M × 1 KB = 200 GB.
  • Schema is simple — this is a key-value-like workload. Almost every query is SELECT * FROM profiles WHERE user_id = ?.

Access Patterns

  • Reads (50,000/sec peak): Profile lookups by user_id (login, profile page views, avatar rendering, authorization checks). Read-heavy — 100:1 read-to-write ratio.
  • Writes (500/sec peak): Profile updates (name change, email change, avatar upload, settings toggle). Infrequent and small — typically updating 1–3 fields at a time.
  • Geographic distribution: Users are spread across 3 regions — US (40%), EU (35%), Asia (25%). Each region needs <50 ms read latency.

Back-of-Envelope Math

Before choosing any architecture, let’s calculate whether a single server can handle this workload. These numbers will drive every decision.

Storage: Does It Fit on One Machine?

Total data: 200M profiles × 1 KB = 200 GB

A modern database server has 256–512 GB of RAM.
200 GB fits entirely in memory on a single machine.

PostgreSQL with shared_buffers = 200 GB can serve
every profile lookup from RAM — no disk I/O needed.

Disk for persistence: 200 GB on SSD = trivial.
WAL (write-ahead log): 500 writes/sec × 1 KB × 86,400 sec
  = 43 GB/day. Standard SSD handles this easily.

Throughput: Can One Server Handle the QPS?

Read capacity of a single PostgreSQL instance:
  Simple primary-key lookups from RAM → ~10,000 reads/sec
  (B-tree index lookup: O(log n), depth 3-4 for 200M rows,
   all index pages cached in shared_buffers → ~0.1 ms each)

We need: 50,000 reads/sec
We have: 10,000 reads/sec per server

Servers needed for reads: 50,000 / 10,000 = 5 servers

Write capacity of a single PostgreSQL instance:
  Simple UPDATE by primary key → ~5,000-10,000 writes/sec
  (WAL write + index update, fsync in batches)

We need: 500 writes/sec
We have: 5,000+ writes/sec per server

Writes are trivial — one server can handle 10× our peak.

Key insight: The data is small (200 GB), writes are low (500/sec), but reads are high (50K/sec). We need 5× the read capacity of a single PostgreSQL. Sharding would split 200 GB into, say, 10 × 20 GB shards — pointless overhead for data this small. Each shard would still need its own server, backup strategy, and failover. Replication is the right tool: keep the full dataset on multiple servers, spread reads across them, and get high availability as a bonus.

Naive Design: Single PostgreSQL

Start with the simplest thing that could work. One PostgreSQL instance serving all reads and writes.

Schema

CREATE TABLE profiles (
  user_id      BIGINT PRIMARY KEY,
  name         VARCHAR(128),
  email        VARCHAR(256) UNIQUE,
  avatar_url   TEXT,
  preferences  JSONB,       -- {language, theme, notifications}
  timezone     VARCHAR(64),
  updated_at   TIMESTAMP DEFAULT NOW(),
  created_at   TIMESTAMP DEFAULT NOW()
);

-- Most queries:
SELECT * FROM profiles WHERE user_id = 12345;
-- ~0.1 ms from RAM (B-tree depth 4 for 200M rows)

What This Gets Right

  • Simple to operate — one server, one database, no replication complexity.
  • Strong consistency — every read sees the latest write (single node, no lag).
  • ACID transactions — email uniqueness constraints, atomic profile updates.

What Breaks

The single-server design fails on three fronts. Let’s quantify each.

Problem 1: Read Throughput

Single PostgreSQL capacity:   10,000 reads/sec
Peak demand:                  50,000 reads/sec

50,000 / 10,000 = 5× over capacity.

At 50K reads/sec, PostgreSQL's connection pool saturates.
Each connection consumes ~10 MB RAM for buffers.
At 200 connections × 0.1 ms/query = 2,000 QPS.
At 500 connections × 0.1 ms/query = 5,000 QPS.
(Past ~500 connections, contention on locks, buffers,
 and process scheduling causes throughput to plateau
 or even decline — the "connection scaling wall.")

Result: requests queue up, latency spikes from 0.1 ms
to 100+ ms, then timeouts and errors cascade.

Problem 2: Availability

A single server gives us roughly 99.9% availability:
  - Planned downtime: OS patches, PostgreSQL upgrades,
    hardware maintenance → ~4 hours/year
  - Unplanned downtime: disk failure, OOM, kernel panic,
    network issues → ~4 hours/year
  - Total: ~8.7 hours/year → 99.9%

We need 99.99%:
  - Allowed downtime: 52 minutes/year
  - A SINGLE hardware failure exceeds our entire budget

With one server, any failure = total outage.
No reads, no writes, no logins.

Profiles are on the critical path for authentication.
If the profile service is down, users cannot log in.

Problem 3: Geographic Latency

Network round-trip times (RTT) to a US-based server:

  US users  → US server:    ~2 ms     ✓ under 50 ms
  EU users  → US server:    ~120 ms   ✗ over 50 ms target
  Asia users → US server:   ~200 ms   ✗ over 50 ms target

Physics constraint: light in fiber travels at ~200,000 km/sec.
  US-EU distance:   ~6,000 km → 30 ms one-way → 60 ms RTT
  US-Asia distance: ~10,000 km → 50 ms one-way → 100 ms RTT

Add TCP overhead, TLS handshake, and processing time:
  US-EU total:    ~120 ms
  US-Asia total:  ~200 ms

No amount of code optimization can fix the speed of light.
The only solution is to put the data closer to the user.

Replication solves all three problems: (1) Multiple replicas multiply read capacity — 5 servers × 10K reads/sec = 50K reads/sec. (2) If one server dies, the others keep serving — no single point of failure. (3) Replicas in EU and Asia serve local reads in <5 ms instead of 120–200 ms. But replication introduces its own set of challenges — the rest of this page walks through them.

Iteration 1: Primary + Read Replicas (Same Region)

The simplest replication setup: one primary that handles all writes, and multiple replicas that handle reads. All servers are in the same region (US) for now.

How Async Replication Works

1. Client sends UPDATE profiles SET name='Alice B'
   WHERE user_id = 12345 to the PRIMARY.

2. Primary writes the change to its WAL (Write-Ahead Log)
   — a sequential, append-only file on disk.

3. Primary applies the change to its data pages in memory
   and returns "UPDATE 1" to the client.

4. In the background (async), the primary streams the WAL
   record to each replica over a TCP connection.

5. Each replica receives the WAL record and replays it:
   applies the same change to its own copy of the data.

Timeline:
  t=0 ms    Client sends UPDATE to primary
  t=0.5 ms  Primary writes WAL, applies change, responds
  t=1-5 ms  WAL record arrives at Replica 1 (same rack)
  t=1-5 ms  WAL record arrives at Replicas 2-4
  t=5-10 ms All replicas have the new data

The gap between t=0.5 ms (primary responds) and
t=1-5 ms (replicas catch up) is the REPLICATION LAG.

Capacity Math

Servers: 1 primary + 4 replicas = 5 servers total

Read distribution:
  Each server handles ~10,000 reads/sec
  5 servers × 10,000 = 50,000 reads/sec ✓

Write handling:
  All 500 writes/sec go to the primary only.
  Primary capacity: 5,000+ writes/sec → 10% utilized ✓

Storage per server: 200 GB (full copy of all data)
Total storage: 5 × 200 GB = 1 TB
  (Compare: sharding into 5 shards would also use 5 servers
   but each would hold only 40 GB. Same server count,
   more operational complexity, no availability benefit.)

Availability:
  Any single server can die without service interruption.
  P(one server up) = 99.9% (annual)
  P(all 5 down simultaneously) ≈ (0.001)^5 ≈ 0
  P(at least one read server up) = 1 - (0.001)^5 ≈ 99.9999%
  With automated failover: 99.99%+ achievable ✓

This works for US-based users. But there are two problems hiding in async replication: (1) replication lag causes stale reads, and (2) failover can lose the most recent writes. Let’s examine both.

The Replication Lag Problem

Async replication means replicas are always slightly behind the primary. Usually this lag is 1–5 ms (imperceptible). But it creates a real user experience bug.

The “Read-Your-Own-Writes” Bug

Scenario: Alice updates her display name.

  t=0 ms      Alice sends: UPDATE profiles SET name='Alice B.' WHERE user_id=42
              → routed to PRIMARY

  t=0.5 ms    Primary applies the change, responds "success" to Alice.
              Primary now has: name = 'Alice B.'
              Replicas still have: name = 'Alice A.'

  t=1 ms      Alice's browser reloads the profile page.
              → GET /profile/42 is routed to REPLICA 2 (round-robin)
              Replica 2 hasn't received the WAL record yet.
              Returns: name = 'Alice A.'

  t=3 ms      WAL record arrives at Replica 2. It now has 'Alice B.'
              But Alice already saw her old name and thinks the update failed.

Alice sees: "I changed my name but it didn't save!"
She tries again. And again. Files a support ticket.

How Often Does This Happen?

Replication lag in the same data center: 1-5 ms typical.

A user who updates their profile and immediately
refreshes the page will hit this bug if the page load
arrives at a replica within that 1-5 ms window.

Page reload time:
  DNS lookup:     ~0 ms (cached)
  TCP + TLS:      ~0 ms (connection reuse)
  HTTP request:   ~1 ms (same data center)
  Total:          ~1 ms

So the reload arrives ~1 ms after the write completes.
Replication lag is 1-5 ms. The race condition is:

  P(stale read) ≈ P(replica lag > reload time)
                ≈ P(lag > 1 ms)
                ≈ 50-80% of the time!

This is not a rare edge case — it happens to roughly
half the users who update and immediately check.

Failover Data Loss

Scenario: primary crashes with async replication.

  t=0        Primary has committed writes up to LSN 1000.
  t=-3 ms    Replicas have received up to LSN 997.
             3 ms of WAL records (LSN 998, 999, 1000) are
             in flight — sent by primary but not yet received.

  t=0        Primary crashes (hardware failure, OOM, etc.)
             Those 3 WAL records are LOST — they were in the
             primary's send buffer, which is now gone.

  t=30 sec   Automated failover promotes Replica 1 to primary.
             New primary's latest LSN: 997.
             Writes at LSN 998, 999, 1000 are permanently lost.

How many profile updates were lost?
  500 writes/sec × 0.003 sec lag = 1.5 writes lost
  In a worst case (lag spike to 100 ms): 50 writes lost.

For most profile data (name, avatar), losing 1-2 updates
is tolerable — the user just re-submits. For email changes
tied to verification flows, it could cause confusion.

The core tension: Synchronous replication guarantees zero data loss (the primary waits for the replica to confirm before responding to the client) but adds latency to every write. Async replication is fast but risks losing the most recent writes on failover. The next iteration solves both problems.

Iteration 2: Solving Read-Your-Own-Writes

There are two practical approaches to guarantee a user sees their own writes. Both are widely used in production.

Approach A: Sticky Routing After Write

Idea: After a user writes, route that user's reads
to the PRIMARY for the next N seconds.

Implementation:
  1. User sends a write (UPDATE profile).
  2. Application server sets a flag:
       cache.set("read-from-primary:{user_id}", true, TTL=5s)
  3. On subsequent reads from this user:
       if cache.get("read-from-primary:{user_id}"):
           route to PRIMARY
       else:
           route to any REPLICA (round-robin)

Timeline with sticky routing:
  t=0 ms    Alice updates name → routed to PRIMARY
  t=0.5 ms  Primary responds. App sets sticky flag (TTL=5s).
  t=1 ms    Alice reloads profile → sticky flag is set
            → routed to PRIMARY (not a replica)
            → sees the updated name ✓
  t=5 sec   Sticky flag expires. Future reads go to replicas.
            By now, all replicas have caught up (lag < 5 ms).

Tradeoff:
  + Simple to implement (just a cache flag)
  + No changes to PostgreSQL
  - Increases primary load after bursts of writes
  - If user has multiple browser tabs, only the tab that
    wrote gets the sticky flag (unless stored in a shared
    session or cookie)
  - TTL is a guess — if replication lag exceeds TTL (rare
    but possible under heavy load), the bug reappears

Approach B: LSN-Based Read-After-Write

Idea: After a write, the primary returns its current
LSN (Log Sequence Number). Subsequent reads include
this LSN, and the replica waits until it's caught up
to that LSN before responding.

What is an LSN?
  The LSN (Log Sequence Number) is a monotonically
  increasing position in the WAL. Every write advances it.
  LSN 1000 means "the 1000th byte of WAL written."
  If a replica's replay position is at LSN 997, it's
  3 bytes (or 3 records) behind the primary.

  PostgreSQL exposes this:
    Primary: SELECT pg_current_wal_lsn();  → '0/174A8B0'
    Replica: SELECT pg_last_wal_replay_lsn(); → '0/174A890'

Implementation:
  1. User sends UPDATE → primary applies it.
  2. Primary returns the LSN of this write:
       response.headers['X-Read-After'] = '0/174A8B0'
  3. Client stores this LSN (cookie, header, or session).
  4. On next read, client sends:
       GET /profile/42
       X-Read-After: 0/174A8B0
  5. Application server picks a replica and checks:
       SELECT pg_last_wal_replay_lsn();
       If replica_lsn >= requested_lsn:
           serve the read from this replica ✓
       Else:
           wait (up to 50 ms) for replica to catch up
           or fall back to reading from primary

Timeline:
  t=0 ms    Alice writes. Primary LSN advances to 0/174A8B0.
  t=0.5 ms  Response includes X-Read-After: 0/174A8B0.
  t=1 ms    Alice reads. Request includes the LSN.
  t=1 ms    App server asks Replica 2: "Are you at 0/174A8B0?"
            Replica 2 is at 0/174A890 (not caught up yet).
  t=3 ms    Replica 2 catches up to 0/174A8B0.
  t=3 ms    App server reads from Replica 2 → returns new data ✓

Tradeoff:
  + Precise — no wasted primary reads, no TTL guessing
  + Works across browser tabs (LSN stored in session)
  - More complex to implement (LSN propagation in headers)
  - Adds 2-3 ms latency on reads that must wait for replay
  - Requires PostgreSQL-specific LSN functions

Which approach do production systems use? Both. AWS Aurora uses LSN-based tracking internally. Google Cloud SQL uses a variant of sticky routing. For a user profile service where reads vastly outnumber writes and the consistency window is tiny (1–5 ms), sticky routing is simpler and usually sufficient. LSN-based is more precise and worth the complexity if you have strict consistency requirements (e.g., financial data).

Solving Failover Data Loss: Semi-Synchronous Replication

The problem: with pure async replication, the primary
responds to the client BEFORE any replica confirms.
If the primary crashes, those in-flight writes are lost.

Solution: make ONE replica synchronous.

Semi-synchronous replication:
  Primary sends WAL to ALL replicas.
  Waits for ONE replica to confirm (sync replica).
  Other replicas are still async.

  Write path:
    t=0 ms    Client sends UPDATE.
    t=0.2 ms  Primary writes WAL to local disk.
    t=0.3 ms  Primary sends WAL to Replica 1 (sync)
              AND Replicas 2-4 (async) simultaneously.
    t=1 ms    Replica 1 writes WAL to its disk, sends ACK.
    t=1 ms    Primary receives ACK from Replica 1.
    t=1 ms    Primary responds to client: "UPDATE 1"

  Write latency: ~1 ms (was ~0.5 ms with pure async)
  — only 0.5 ms slower, because we wait for ONE LAN ACK.

Failover with semi-sync:
  Primary crashes at LSN 1000.
  Replica 1 (sync) is guaranteed to have LSN 1000.
  Promote Replica 1 → ZERO data loss.

  Replicas 2-4 might be at LSN 997-999 (async).
  They catch up from new primary (Replica 1).

Configuration: 1 primary + 1 sync replica + 3 async replicas
  Total servers: 5
  Write latency: ~1 ms (acceptable for 500 writes/sec)
  Failover guarantee: zero data loss when promoting sync replica

Iteration 2 Architecture

1 Primary (US)
  ├── 1 Sync Replica (US, same AZ or nearby AZ)
  │     → zero-loss failover target
  │     → serves reads + ready for instant promotion
  ├── 1 Async Replica (US, different AZ)
  │     → additional read capacity
  ├── 1 Async Replica (US, different AZ)
  │     → additional read capacity
  └── 1 Async Replica (US, different AZ)
        → additional read capacity

Read routing:
  - After a write: route to primary for 5 sec (sticky)
  - Otherwise: round-robin across all 5 servers

Write routing:
  - Always to primary
  - Primary waits for sync replica ACK before responding

Total read capacity: 5 × 10K = 50K reads/sec ✓
Total write capacity: limited by sync round-trip (~1 ms)
  = ~1000 writes/sec throughput → 500/sec fits ✓

Remaining problem: All 5 servers are in the US. EU users still see 120 ms latency. Asia users see 200 ms. We need replicas closer to those users.

Iteration 3: Multi-Region Replicas

To serve EU and Asia users with <50 ms latency, we place read replicas in those regions. But cross-ocean replication introduces much higher lag.

Read Latency: Solved

With local replicas in each region:

  US users  → US replica:     ~2 ms    ✓
  EU users  → EU replica:     ~3 ms    ✓
  Asia users → Asia replica:  ~5 ms    ✓

All under 50 ms target. The data is physically close
to the user — network RTT is minimal.

The New Problem: Cross-Ocean Replication Lag

Within a data center, replication lag = 1-5 ms.
Across the Atlantic (US → EU), lag = 80-100 ms.
Across the Pacific (US → Asia), lag = 120-150 ms.

This is physics, not a software problem:
  US → EU:   6,000 km of fiber → 30 ms one-way propagation
             + TCP overhead + WAL processing = 80-100 ms RTT
  US → Asia: 10,000 km → 50 ms one-way = 120-150 ms RTT

Read-your-writes is now much harder:

  Scenario: EU user updates profile.
    t=0 ms     Write goes to US primary (120 ms RTT).
    t=120 ms   Primary responds "success."
    t=121 ms   User reloads page → reads from EU replica.
    t=121 ms   EU replica is 80-100 ms behind primary.
               It hasn't received this write yet!
    t=200 ms   EU replica catches up.

  The stale-read window is 80-100 ms — much worse than
  the 1-5 ms within a single region.

Sticky routing to PRIMARY doesn't work well here either:
  If we route the EU user's reads to the US primary,
  reads take 120 ms RTT — worse than the 50 ms target.
  We're right back to the original latency problem.

Three Options for Multi-Region Writes

OptionHow It WorksWrite LatencyConsistencyComplexity
A: Single primary (US) All writes go to US. Accept 120–200 ms write latency from EU/Asia. US: 1 ms, EU: 120 ms, Asia: 200 ms Strong (with LSN-based read-after-write) Low
B: Region-local primary (multi-primary) Each region has its own primary. Writes are local (~1 ms). Cross-region replication syncs data. All regions: ~1 ms Eventual (conflicting writes possible) High
C: Single primary + local write buffer Writes go to a local queue, forwarded to US primary async. Client gets immediate ACK. All regions: ~1 ms (apparent) Eventual (write may fail at primary) Medium

Why Option A Is Best for User Profiles

User profiles have specific properties that favor Option A:

1. WRITES ARE RARE.
   500 writes/sec globally. Per-user, a profile update
   happens maybe once a week. The 120-200 ms write latency
   affects only the rare update — the user clicks "Save"
   and waits 200 ms. That's imperceptible in a UI flow
   that already has form validation, button animation, etc.

2. READS ARE FREQUENT AND LATENCY-SENSITIVE.
   50,000 reads/sec. Every page load, every API call.
   This is where sub-50 ms matters. Local replicas solve this.

3. CONFLICTS ARE DANGEROUS.
   Profile data includes email (used for login) and
   preferences (used for authorization scoping). If two
   regions accept conflicting writes to the same profile:
     Region US: SET email = 'alice@new.com'
     Region EU: SET email = 'alice@other.com'
   Which email is correct? Last-writer-wins (LWW) might
   pick the wrong one. The user can't log in.

   With a single primary, writes are serialized — no conflicts.

4. SIMPLICITY.
   Multi-primary replication requires conflict detection,
   conflict resolution strategies, and extensive testing.
   For data that's written 100x less than it's read,
   this complexity isn't justified.

Decision: Option A — single US primary, multi-region read replicas.
  EU/Asia users accept 120-200 ms write latency.
  All users get <5 ms read latency from local replicas.
  Read-your-writes: after a write, route that user's reads
  to the PRIMARY for 5 seconds (sticky routing via cookie).
  Since writes are rare (once a week per user), the
  occasional 120 ms read from primary is acceptable.

When IS multi-primary worth it? When writes are frequent AND latency-sensitive AND conflicts are rare or easily resolvable. Examples: collaborative editing (CRDTs handle merges), shopping carts (merge = union of items), counters (commutative). User profiles don’t fit this pattern — writes are rare, and email conflicts are costly.

Iteration 4: Handling Split-Brain and Failover

With servers spread across 3 regions, network partitions between regions become a real concern. What happens when the US primary can’t communicate with the EU and Asia replicas?

What Is Split-Brain?

Split-brain occurs when a network partition causes
two parts of the system to each believe the other is down,
and both attempt to act as the primary.

Scenario: Network cable between US and EU is severed.

  US side sees: Primary (US) is healthy. EU replica is
  unreachable. "EU must have crashed."

  EU side sees: EU replica is healthy. US primary is
  unreachable. "US primary must have crashed."

  If the EU side automatically promotes its replica to
  primary, we now have TWO primaries accepting writes:

    US Primary:  UPDATE email = 'alice@us.com' WHERE user_id = 42
    EU Primary:  UPDATE email = 'alice@eu.com' WHERE user_id = 42

  When the network heals, which email is correct?
  Both writes were committed. Both sides believe they're
  authoritative. This is split-brain — the most dangerous
  failure mode in replicated systems.

Why it's dangerous for user profiles:
  - User can't log in (which email is the real one?)
  - Email verification flows break
  - Account recovery sends to the wrong address
  - Preferences conflict (one region says dark mode,
    the other says light mode)

Prevention: Fencing Tokens

A fencing token is a monotonically increasing number
that identifies the "current" primary. Only the holder
of the latest token can perform writes.

How it works:
  1. Primary is elected with fencing token = 7.
  2. Primary includes token=7 in every write to storage.
  3. Storage layer rejects writes with token < current.

  Scenario with fencing:
    t=0      US Primary holds token 7. Writes normally.
    t=100ms  Network partition. EU can't reach US.
    t=30s    EU failover system tries to promote EU replica.
             It requests a new token from the token service
             (which uses a consensus protocol like Raft).
             Token service is split — US side has 2 nodes,
             EU side has 1 node. Majority (2/3) is on US side.
             EU CANNOT get a new token. Promotion blocked. ✓

    If token service nodes are: US(2), EU(1), Asia(1):
             EU + Asia = 2 nodes, US = 2 nodes.
             Neither side has majority (3/4). Neither can
             elect a new primary. System stays read-only
             until partition heals. This is the correct
             behavior — availability is sacrificed to prevent
             split-brain (CP choice in CAP theorem).

  The token service itself must be distributed and use
  consensus (e.g., etcd, ZooKeeper, Consul) — otherwise
  it becomes a single point of failure.

Automated vs. Manual Failover

AUTOMATED FAILOVER:
  A health checker pings the primary every 1 second.
  If 3 consecutive pings fail (3 seconds), declare primary dead.
  Promote the sync replica to primary.

  Pros: Fast recovery (~5-10 sec total downtime).
  Cons: False positives. The primary might be:
    - Temporarily overloaded (GC pause, slow query)
    - Experiencing a brief network blip (1-2 sec)
    - Actually fine but the health checker's network failed

  A false positive causes:
    1. Promote replica to primary → now TWO primaries briefly
    2. Old primary comes back → split-brain until it's fenced
    3. Clients connected to old primary send writes → lost or
       conflicting when old primary realizes it's been deposed

  Mitigation: use fencing tokens, so even if old primary
  accepts writes briefly, the storage layer rejects them.

MANUAL FAILOVER:
  On-call engineer gets an alert. Investigates.
  Decides whether to promote or wait.

  Pros: No false positives — a human verifies.
  Cons: Slow — minutes to hours depending on response time.
        At 99.99% target (52 min/year), manual failover
        might consume the entire budget on one incident.

HYBRID (recommended for our service):
  Automated failover with fencing tokens + human review.
  1. Health checker detects failure (3 consecutive misses).
  2. Verify with secondary signal (can other services reach
     the primary? Is the primary's host reporting healthy
     in the infrastructure monitoring system?).
  3. If confirmed: automated promotion with fencing token.
  4. Alert on-call engineer for post-incident review.
  5. If ambiguous: alert and wait for human decision.

  Expected downtime per incident: 10-30 seconds.
  With 2-3 incidents/year: 30-90 sec total → 99.999% ✓

False Positive Detection

The hardest problem in failure detection: distinguishing
a dead server from a slow server or a network partition.

Signals that the primary is truly dead:
  1. Health check fails from MULTIPLE vantage points
     (not just one checker — use 3+ checkers in different
      network zones). If all 3 fail, it's likely real.
  2. The primary's host-level metrics stop updating
     (CPU, memory, disk I/O all flatline in monitoring).
  3. The primary's TCP connections are RST or timeout
     (not just slow).

Signals that might be a false positive:
  1. Only ONE health checker fails → could be checker's
     network, not the primary's.
  2. Health check is slow but eventually responds →
     primary is overloaded, not dead.
  3. Other services in the same data center are also
     unreachable → data center network issue, not
     individual server failure.

Decision matrix:
  All checkers fail + host metrics dead → auto-promote
  Some checkers fail → wait 10 more seconds, recheck
  Health check slow but responding → do NOT promote
  Data center-wide issue → do NOT promote (would make
    things worse — the new primary can't reach clients
    in that data center either)

The CAP tradeoff in practice: During a network partition between US and EU, our system chooses consistency over availability for writes. EU users can still read (from their local replica), but writes are rejected or queued until the partition heals. For a user profile service, this is the right tradeoff — a brief period of read-only mode is far less damaging than conflicting profile data.

Availability Calculation

Target: 99.99% (52 minutes downtime/year)

Failure modes and their impact:

1. Single replica dies:
   Impact: None. Reads redistribute to remaining replicas.
   Recovery: Replace server, resync (200 GB at 1 Gbps = ~27 min).

2. US async replica dies:
   Impact: US read capacity drops from 30K to 20K.
   20K is at capacity for US peak (20K). No immediate issue
   unless there's a traffic spike.
   Recovery: Auto-replace, resync.

3. Sync replica dies:
   Impact: Writes continue (primary falls back to async).
   Failover safety degrades until sync replica is restored.
   Recovery: Promote another async replica to sync role.

4. Primary dies:
   Impact: Writes fail for ~10-30 seconds (failover time).
   Reads continue from replicas (stale by a few ms).
   Recovery: Promote sync replica (zero data loss).
   Duration: 10-30 seconds automated.

5. Network partition (US ↔ EU):
   Impact: EU reads continue locally (stale by 80-100ms).
   EU writes fail (can't reach US primary).
   Recovery: Automatic when partition heals.
   Duration: Minutes to hours (depends on network repair).

6. Entire US region down:
   Impact: No primary → no writes anywhere.
   Reads in EU and Asia continue from local replicas.
   Recovery: Manual promotion of EU replica to primary.
   Duration: 5-30 minutes.

Availability math:
  P(primary or sync replica available) for writes:
    Each server: 99.95% uptime individually.
    Both US servers down simultaneously:
      P(both down) = (0.0005)² = 0.00000025 = 99.999975%
    Write availability: ~99.999%

  P(at least one replica per region available) for reads:
    US (3 replicas): 1 - (0.0005)³ ≈ 99.9999999%
    EU (1 replica): 99.95% (add second for 99.9999%)
    Asia (1 replica): 99.95% (add second for 99.9999%)

  Overall: write availability (~99.999%) is the bottleneck.
  Adding the second EU and Asia replicas makes read
  availability effectively 100%. Target 99.99% → achieved ✓

Key Takeaways

  • Replication solves three problems at once: read scalability (5× throughput from 5 replicas), high availability (no single point of failure), and geographic latency (local replicas in each region). For read-heavy workloads with small data, it’s the right tool — simpler than sharding.
  • Replication introduces real challenges: replication lag (stale reads), read-your-writes consistency bugs, failover data loss risk, multi-region write latency, and split-brain danger. Every benefit has a cost.
  • Async replication is fast but lossy. The primary responds before replicas confirm. This means sub-millisecond write latency but up to N milliseconds of data loss on failover. Semi-synchronous replication (wait for one replica) gives you zero-loss failover with minimal latency overhead.
  • Read-your-own-writes is a user experience bug, not a system crash. But it causes support tickets, confusion, and lost trust. Sticky routing (route writes to primary for N seconds) is the simplest fix. LSN-based tokens are more precise.
  • Multi-region replication amplifies the lag problem. Cross-ocean lag is 80–150 ms (physics). For write-rare data like profiles, accepting higher write latency from remote regions is the simplest and safest tradeoff. Multi-primary replication is only worth the complexity when writes are frequent AND latency-sensitive AND conflict-free.
  • Split-brain is the most dangerous failure mode. Two primaries accepting conflicting writes can corrupt data permanently. Fencing tokens (monotonically increasing IDs from a consensus-based token service) prevent stale primaries from writing. Always use fencing in any automated failover system.
  • Start with the math. 200 GB fits in RAM → no sharding needed. 50K reads ÷ 10K per server = 5 replicas. 500 writes/sec × 0.003 sec lag = 1.5 lost writes on failover. Every design decision was driven by a concrete number, not intuition.
