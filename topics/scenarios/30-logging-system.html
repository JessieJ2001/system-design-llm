Designing a Distributed Logging System

Ingesting 2 million log lines per second from 10,000 microservices, making them searchable within 30 seconds, and retaining 30 days of history — without breaking the bank or the services producing the logs.

NALSD Walkthrough • ~22 min read • LC #12 (Logger Rate Limiter)

LC kernels. Two textbook structures sit inside log ingest. (1) Design a search index / inverted index — a hashmap token → sorted list of doc-ids; querying level=ERROR AND service=payment is an intersection of postings lists; building it is "merge K sorted streams" of per-segment postings. The textbook version assumes a fixed corpus that fits in RAM. (2) A ring buffer / time-bucketed append log for the ingest side — producers append; the buffer absorbs bursts; the consumer drains at its own pace. Three properties that break at scale: (a) the inverted index doesn't fit and is expensive to keep hot — 17 TB/day of index on SSD for data 95% of which is never queried; (b) ingest at TB/hr saturates the index-write path — one Elasticsearch node indexes ~30 k docs/sec; 2 M lines/sec would need 67 nodes just to keep up, 333 at peak; (c) a query over cold archives can't hit the index — it was aged out to free SSD, so cold queries become full scans with no postings lists. The production cousin replaces the in-memory inverted index over a fixed corpus with a tiered, time-partitioned, write-optimized log index: an LSM-style index that absorbs the firehose, hot data fully indexed on SSD / warm data label-indexed-only on object store / cold data scan-on-demand, the "merge K sorted streams" becoming "merge results across time-partitions and storage tiers," and the ring buffer becoming a Kafka-class durable log with at-least-once-plus-sampling semantics that drops under overload rather than blocking the apps. The #42 Logger Rate Limiter (LC #12) lives inside Iteration 3's per-service rate limiting.

The three observability pillars. Logs (this page), metrics (#36 streaming percentiles, #37 metric-window anomaly), and traces (a separate APM product, out of scope here). This page is the logs pillar: unstructured-to-semi-structured text events, ad-hoc full-text + structured search, high write volume, occasional reads. For the error-tracking specialization — where the "log lines" are stack traces and the value is the dedup/grouping — see #42 (Sentry-style error aggregation), which folds the firehose into one row per error group instead of keeping every line, adds fingerprinting, per-customer quotas, and an issues rollup. #30 indexes every line for ad-hoc search; #42 collapses lines for signal.

The Scenario

You are building the centralized logging platform for a large microservices architecture. Every service emits structured logs. Engineers need to search, filter, and tail logs in near-real-time during incidents. Compliance requires 30-day retention.

Query Patterns

  • Incident debugging: service=payment-service AND level=ERROR AND timestamp > now()-1h — high urgency, needs sub-5s response.
  • Trace lookup: trace_id=abc123def456 — find all log lines across all services for one request. Needs to search all services.
  • Pattern analysis: message CONTAINS "timeout" AND timestamp > now()-24h — broader query over more data. Can tolerate 10-30s latency.
  • Live tail: Stream logs matching service=checkout-service AND level=WARN in real-time. WebSocket-based, continuous.

System class

Class: automation pipeline (log ingest) + a query surface. The ingest side is an ingest-stream-X-into-store-Y pipeline (cite functional-requirements.html#class-pipeline): a firehose in, a tiered index/archive out, freshness and durability guarantees on it. The read side is an interactive search/retrieval API. The class fixes the NFR additions (cite nfr-checklist.html#class-pipeline-nfr): ingest throughput (the firehose absorbed without blocking producers), freshness (ingest-to-searchable delay), durability under overload (drop/sample, never block the apps), query latency on the hot window, and cost (logs are huge and mostly never read — the design must match storage cost to access frequency).

Functional requirements (cite functional-requirements.html)

  • Ingest structured log lines at the firehose rate. Every container emits structured JSON log lines; the platform ships them off-host asynchronously, batched, never blocking the app, at 2 M lines/sec sustained and 10 M/sec peak.
  • Sample / drop per a policy. Per-service rate limits; when over quota, keep all ERROR/FATAL, sample lower severities at a configured ratio, and stamp the sampled line with its sample rate so the true count is recoverable. Under pipeline overload, the policy decides what to drop — it never falls back to blocking producers.
  • Store tiered (hot / warm / cold). Hot = the last few days, fully indexed, searchable in seconds; warm = the rest of the retention window, columnar on object storage, scan-on-demand; cold = beyond retention, compressed/archived for compliance, retrievable in hours.
  • Full-text + structured search over hot + warm. Structured filters (service=, level=, trace_id=) and substring/full-text on message, over an arbitrary time range; the hot window meets the <5 s SLO; warm queries are scans (10–60 s).
  • Retrieve a cold archive on demand. Given a time range + service, fetch the raw compressed log files from cold storage (hours, async); optionally re-index a slice for ad-hoc search.
  • Live tail. Stream logs matching a filter in near-real-time (~150 ms) via WebSocket — read off the ingest buffer, not the index.

Out of scope (named, not silently dropped): metrics (the second observability pillar — counters/gauges/histograms over time; see #36 / #37); distributed traces (the third pillar — spans, parent/child causality; a separate APM product); alerting on logs (log-based alert rules → a downstream alerting engine; we expose a query API it can poll/stream from); the log SDK / structured-logging library in each service (a separate client workstream — we assume services emit well-formed JSON to stdout); the error-tracking specialization (fingerprinting stack traces into issues + per-customer issue quotas → #42); SIEM/security analytics correlation; log-to-BI ETL.

Flagged assumptions. (1) Ingest volume ≈ 2 M lines/sec avg (~500 MB/s raw ≈ 1.8 TB/hr), 10 M/sec peak (~2.5 GB/s ≈ 9 TB/hr) — if this is 10× higher, every fleet number scales and the index-only-labels (Loki-style) alternative in Iter 2 becomes the default not the alternative. (2) Hot-window length = 3 days searchable — if the SLO needed 7-day fast search, the hot ES tier roughly doubles ($5.2 k → ~$12 k/mo SSD). (3) Retention = 30 days warm-retrievable; some compliance regimes want 1–7 years — the cold tier (Glacier, ~$1/TB-mo) handles it cheaply but re-indexing for search takes hours. (4) Query QPS ≈ ~200 concurrent engineers during an incident, a few queries/sec each — tens of QPS, not thousands; bursty, incident-correlated. (5) What fraction of logs is ever read ≈ ~1–5% (95% of queries hit the last 3 days; days 4–30 are touched only during audits/postmortems) — this is the cost lever: it's why the warm tier drops the inverted index entirely and why keeping 30 days in ES is the wrong design.

Sister scenarios

  • #42 Error aggregation & dedupe (Sentry-style) — the error-tracking specialization: same firehose-ingest shape, but the "log lines" are stack traces and the value is dedup. Drastic flip: #42 computes a grouping fingerprint and folds the firehose into one row per error group with per-customer issue quotas; #30 indexes every line for ad-hoc search. The fingerprint algorithm, the issues rollup, and per-customer issue caps are #42-specific moves #30 doesn't have.
  • #11 Event-driven pipeline — the generic ingest-buffer-consumer pipeline; #30 is the logs-specialized instance (Kafka buffer + indexer pool is exactly that pattern, plus the tiered store and search surface).
  • #36 Streaming percentiles / #37 metric-window anomaly — the metrics pillar. Same firehose-in / tiered-storage / sampling shape, but the unit is a numeric sample over (series, window) not a text line; the query is a quantile/rollup not a search.
  • Audit-log store (planned; SOC2/compliance event store) — same ingest shape, but completeness is mandatory: no sampling, immutable storage, retention in years, exact counts. Drastic flip: drops the sampling iteration entirely and adds a write-once archive with a Merkle digest.

The fundamental tension: Logs are a write-heavy workload (2M writes/sec) with an occasional read pattern (searches happen during incidents, not constantly). The system must handle massive sustained write throughput without compromising read performance during the moments that matter most — when things are on fire. The arc resolves it iteration by iteration: agent + Kafka buffer (Iter 1 — decouple producers from the index-write path), hot/warm/cold tiering (Iter 2 — stop paying SSD prices for data nobody reads), per-service rate limiting + sampling + live tail (Iter 3 — one noisy service can't saturate the pipeline; real-time tail reads the buffer), search isolation + index-per-service-group + query cache (Iter 4 — search and ingest stop competing).

Back-of-Envelope Math

Let us quantify the storage, throughput, and cost constraints before choosing any technology. (Reference: BOE cheat sheet.) Engine primer (cited, not restated): the search index is an inverted index — token → sorted postings list, queries are list-intersections (see storage-engines#inverted-index); the ingest buffer and the raw archive are heap/append stores (storage-engines#heap-append); the warm tier is columnar (storage-engines#columnar). The dominant bottleneck the iterations attack: index-write-amp (the index-write path saturates under the firehose) and memory-capacity-bound (the hot index must fit on SSD). Access patterns at play: write-heavy-uniform (the ingest stream), scan-heavy (warm/cold queries), power-law (a few noisy services), bursty (incident spikes). Unit prices cite pricing-constants.html; the cost shape is cost-shapes#telemetry ("keep alerts/aggregates, sample the raw, tier the storage").

Daily Ingestion Volume

Average: 2M lines/sec × 250 bytes = 500 MB/sec
Daily:   500 MB/sec × 86,400 sec   = 43.2 TB/day (raw)

With compression (typical 5:1 for JSON text):
  43.2 TB / 5 = 8.6 TB/day compressed

30-day retention:
  8.6 TB × 30 = 258 TB compressed storage

Index Size

Full-text search index (inverted index) is typically
30-50% of the raw data size for structured JSON logs.

Index size per day: 43.2 TB × 0.4 = 17.3 TB/day
Hot index (3 days): 17.3 × 3 = 51.9 TB
  → Must fit on fast SSDs for sub-5s search.

At $0.10/GB/month for SSD:
  51.9 TB × $100/TB = $5,190/month for hot index storage.

Warm storage (days 4-30, compressed, no index):
  8.6 TB × 27 days = 232 TB on cheap object storage (S3).
  At $0.023/GB/month: 232 TB × $23/TB = $5,336/month.

Ingestion Throughput

A single Elasticsearch node can index ~20,000-50,000 docs/sec
(depending on document size and mapping complexity).

At 2M lines/sec average:
  2,000,000 / 30,000 = ~67 indexing nodes needed (average)

At 10M lines/sec peak:
  10,000,000 / 30,000 = ~333 indexing nodes needed (peak)

We cannot scale to 333 nodes for peak.
We need a buffer (queue) to absorb bursts.

Network Bandwidth

50,000 containers → logging infrastructure:
  500 MB/sec raw = 4 Gbps sustained

Peak: 2.5 GB/sec = 20 Gbps sustained

This is significant. If logs flow over the same network as
production traffic, logging can cause network congestion during
incidents — exactly when production traffic matters most.

Key insight: logs must NEVER impact production services.
If the logging system is slow or down, services must not block.

The 5x peak problem: Incidents cause a 5x spike in log volume (more errors, more retries, more debug logging). This is precisely when engineers need to search logs. The system must handle 5x write load AND serve search queries simultaneously — during the worst possible moment.

Naive Design: Direct-to-Elasticsearch

The simplest architecture: each service writes logs directly to an Elasticsearch cluster via HTTP.

# Naive: each service pushes logs directly to Elasticsearch
import requests
import logging

ES_URL = "http://elasticsearch:9200"

class ElasticsearchHandler(logging.Handler):
    def emit(self, record):
        log_entry = {
            "timestamp": record.created,
            "level": record.levelname,
            "service": SERVICE_NAME,
            "message": record.getMessage(),
        }
        # Blocks the application thread on every log line!
        requests.post(
            f"{ES_URL}/logs-{today()}/_doc",
            json=log_entry,
            timeout=5
        )

Why This Seems OK

  • Simple — no intermediate components.
  • Logs are immediately searchable (no pipeline delay).
  • Elasticsearch handles indexing and search in one system.

What Breaks

Direct-to-Elasticsearch has four fatal flaws (failure modes cite failure-modes.html). Problems 1–3 are variants of backlog-overflow and slow-dependency — the index can't keep up, so the producers pay; Problem 2 also includes bad-deploy (a rolling restart = a connection storm) and cascading-timeout (logging-on-the-hot-path turns into the incident). Problem 4 is the telemetry cost shape ignored: SSD prices for cold data.

Problem 1: Logging Blocks Application Threads

Each log line triggers an HTTP POST to Elasticsearch.
Average latency: 5-10ms per log.
If a service logs 200 lines/sec per instance:
  200 × 10ms = 2,000 ms of thread blocking per second.

That is 2 seconds of a thread's time spent logging every second.
If the service has 4 worker threads, logging consumes 50% of
the service's capacity.

Worse: during an incident, Elasticsearch is overloaded.
Log writes take 500ms-5s instead of 10ms.
The application service backs up, response times spike,
and logging — meant to help debug — BECOMES the incident.

Problem 2: 50K Direct Connections

50,000 containers each maintaining an HTTP connection to ES.
Elasticsearch coordinating node handles ~10K connections max.
We need 5 coordinating nodes just for connections.

Each connection: ~10 KB memory
50K × 10 KB = 500 MB just for connection state.

Worse: during a rolling deploy, all 50K containers restart
within minutes. 50K new connections hit ES simultaneously.
Connection storm + write spike = ES coordinating node OOM.

Problem 3: No Buffer for Bursts

Normal: 2M logs/sec. ES cluster sized for 2M/sec.
Incident: 10M logs/sec (5x spike).

Options when ES can't keep up:
  (a) Drop logs — lose visibility during the incident.
  (b) Block services — services slow down, making incident worse.
  (c) Queue in memory — OOM on the application side.

None of these is acceptable. We need a buffer that absorbs
the burst and lets ES index at its own pace.

Problem 4: ES Is Expensive for Cold Data

30-day index in Elasticsearch:
  Raw data: 43.2 TB/day × 30 = 1,296 TB
  Index: ~518 TB
  Total: ~1,814 TB on SSD

Cost: 1,814 TB × $0.10/GB/month = $181,400/month

Most of this data (days 4-30) is rarely searched.
95% of queries hit the last 3 days.
We are paying SSD prices for data that's accessed
once a month during compliance audits.

Rule #1 of logging: the logging system must never impact production services. If writing a log line can block an application thread, slow down a response, or cause an OOM, the logging system is a liability, not an asset. Logging must be fire-and-forget from the application's perspective.

Iteration 1: Local Agent + Kafka Buffer

Decouple log production from log ingestion with two components: a lightweight local agent on each host and a Kafka buffer between producers and Elasticsearch.

The Log Agent (Fluent Bit)

Each host runs a lightweight log agent (Fluent Bit, ~30 MB RAM).

The agent:
  1. Tails container stdout/stderr log files (inotify-based, no polling)
  2. Parses JSON structure, adds host/container metadata
  3. Batches logs into chunks of 1,000 lines or 1 MB (whichever first)
  4. Sends batches to Kafka asynchronously (non-blocking)
  5. On Kafka failure: buffers to local disk (1 GB ring buffer)

Application impact: ZERO.
  Services write to stdout (a kernel pipe, ~1μs per write).
  The agent reads from files — services never wait for the agent.
  If the agent crashes, logs accumulate on disk. No data loss,
  no application impact. Agent restarts and catches up.

Kafka as a Buffer

Kafka absorbs the write burst that Elasticsearch cannot handle.

Normal load: 2M msgs/sec
  Kafka: 2M/sec in, 2M/sec out to indexers. No lag.

Incident load: 10M msgs/sec
  Kafka: 10M/sec in, 2M/sec out (ES limit). Lag builds.
  Backlog: (10M - 2M) × 250 bytes = 2 GB/sec accumulating.
  Kafka retention: 24 hours = plenty of buffer.

After incident (load drops to normal):
  Kafka: 2M/sec in, but indexers can consume faster (catch-up).
  Indexers process backlog at 3M/sec.
  8M backlog msgs → catch-up in ~4 seconds per second of lag.
  1-hour incident → ~15 min to drain backlog.

During the lag, logs are delayed (not lost). Engineers can still
search the most recent data (within ES), and the backlog will
become searchable within minutes of the incident ending.

Kafka Sizing

Topic: "logs", 64 partitions (one per indexer × 2 for headroom)
Replication factor: 2 (durability without 3x cost — logs are
  not financial transactions; losing a few lines is acceptable)

Throughput: 2M msgs/sec × 250 bytes = 500 MB/sec
  Per broker: 500 MB / 5 brokers = 100 MB/sec (well within limits)

Storage (24h retention):
  500 MB/sec × 86,400 = 43.2 TB/day × 2 replicas = 86.4 TB
  With compression (snappy, ~2x): 43.2 TB
  5 brokers × 9 TB NVMe each = 45 TB total. Fits.

Broker count: 5 brokers (each handles 400K msgs/sec easily)
  Kafka can handle 1M+ msgs/sec per broker on modern hardware.

Alternatives considered (C4)

The buffer. Kafka (chosen) — durable append log, replay-from-offset, per-partition ordering, ~1 M+ msgs/sec/broker, and the live-tail consumers (Iter 3) read it directly. Kinesis — managed, but 1 MB/s/shard write cap means 500 MB/s avg = 500 shards (2,500 at peak) — an operational and cost nightmare at this volume; reasonable only well below 100 MB/s. Redis Streams / SQS — SQS has no replay, no ordering; Redis Streams isn't built for 43 TB/day of durable retention. Direct-to-storage (agent → S3) — cheap and durable, but loses the streaming consumer model (no live tail, no incremental indexers); a viable design for a logs-as-cold-archive product, not for one with a hot search SLO. The agent. Fluent Bit (chosen, ~30 MB RAM, C, inotify tailing) over Fluentd (Ruby, ~200 MB+, heavier plugins) or Vector (Rust, comparable footprint, also fine) — on 50 k hosts the per-host footprint matters; Fluent Bit and Vector are the two right answers, Fluentd is the legacy heavyweight.

What breaks (C5; cite failure-modes.html)

  • Kafka backlog overflow during a sustained incident. 10 M/sec in, 2 M/sec out → lag grows at (10M−2M)×250 B ≈ 2 GB/sec; 24 h retention ≈ 173 TB of headroom, but a multi-hour incident eats it. Detection: per-partition consumer lag (alert at > 30 min). Recovery: auto-scale indexers to burst (32 → 64), and if lag still grows, the agent-side rate limiter (Iter 3) sheds — never block producers. Fully addressed once Iter 3's per-service caps + sampling cap the inflow.
  • A malformed log line that crashes the indexer. A line that breaks the JSON parser or the ES mapping (e.g. a field that's a string in one line and an object in the next — "mapping explosion") → the indexer batch fails and retries forever, blocking the partition. Detection: indexer error rate per partition. Recovery: parse-and-validate at the agent (drop or quarantine malformed lines); on the indexer, a poison-pill detector routes a repeatedly-failing record to a dead-letter topic and continues. (This is why structured-JSON-from-an-SDK is a flagged assumption — arbitrary text makes this far more likely.)
  • Connection storm on a fleet-wide rolling deploy — mostly fixed here. The agent holds a small pooled set of long-lived Kafka connections per host (not one per container); 5 k hosts × a handful of connections ≈ tens of thousands, vs the Naive's 50 k. Residual: a Kafka broker restart drops its share; producers reconnect with backoff. Detection: Kafka connection-count + producer reconnect rate.
  • Agent disk-buffer loss. If a host's local disk buffer (1 GB ring) fills (Kafka unreachable for > ~30 min at that host's rate) it wraps — oldest buffered lines are lost. Detection: per-host buffer-utilization metric. Recovery: this is the intentional degradation contract — logs are best-effort; we lose old buffered lines, not new ones, and we never block the app. (An audit-log variant would size the buffer for the worst-case outage and alert hard.)

Kafka decouples write speed from index speed. Services produce logs at whatever rate they want. Kafka absorbs it. Elasticsearch indexes at its own pace. During bursts, Kafka's backlog grows. After bursts, indexers catch up. This is the single most important architectural decision in the logging pipeline. (Cite bottleneck-to-decisions#backlog: a producer faster than the consumer → put a durable buffer between them and let the consumer drain at its pace.) Schema/API delta: Kafka topic logs — ~64 partitions (one per indexer task ×2 headroom), partition key = service_id (per-service ordering; lets the agent-side rate limiter and the live-tail consumers concentrate by service), RF=2, snappy compression, 24 h retention. The agent emits batches of ≤ 1,000 lines / ≤ 1 MB; the indexer consumes, batches 5,000 docs, and calls the ES _bulk API. The end-to-end "log→searchable" path is now agent (~100 ms batch) → Kafka (~5 ms) → indexer batch (~5 s) → ES refresh (~1 s) ≈ ~6 s — well under the 30 s SLO.

Iteration 2: Hot/Warm/Cold Tiered Storage

Keeping 30 days of indexed data in Elasticsearch is wasteful. 95% of queries hit the last 3 days. Older data should move to cheaper storage.

Hot Tier: Elasticsearch (Days 0-3)

Index strategy: daily indices (logs-2026.03.22, logs-2026.03.21, ...)
Shard count: 64 shards per daily index (matches Kafka partitions)
Replica count: 1 (data is in Kafka for 24h, so one replica is enough)

Write path:
  Indexers consume from Kafka → batch 5,000 docs → ES _bulk API
  Bulk request size: 5,000 × 250 bytes = 1.25 MB
  400 bulk requests/sec across 32 indexers = 2M docs/sec

Read path:
  Search API fans out to shards covering the queried time range.
  1-hour query: hits 1 daily index, 64 shards.
  p99 search latency: 2-3 seconds.

Rollover: at midnight, create new daily index. Delete index
from 4 days ago (data already in warm tier).

Warm Tier: Parquet on S3 (Days 4-30)

When an ES daily index ages past 3 days:
  1. Export index to Parquet files (columnar format, ~5:1 compression).
  2. Upload Parquet files to S3, partitioned by service and hour:
     s3://logs/service=payment-service/date=2026-03-19/hour=14/data.parquet
  3. Register partitions in a metadata catalog (AWS Glue / Hive).
  4. Delete the ES index to free SSD space.

Query engine: Athena / Presto / Spark SQL
  - Queries scan only relevant partitions (service + time range).
  - Columnar format means scanning only needed columns.
  - A query over 1 service × 24 hours scans ~1.5 GB: ~10 seconds.
  - A query over ALL services × 7 days scans ~300 GB: ~60 seconds.

Cost: S3 storage ($0.023/GB) + Athena scan ($5/TB scanned).
  Typical query cost: $0.008. Thousands of queries/month: ~$50.

Cold Tier: Glacier + a manifest (Days 31+)

Past the 30-day retention SLO, the Parquet files move from S3
Standard → Glacier (or Glacier Deep Archive) via a lifecycle rule.
We keep the partition manifest (the Glue catalog) hot — it's tiny —
so "what archives exist for service X, March 2026?" is a cheap lookup.

Retrieve-on-demand: a restore job pulls the requested partitions
back to S3 Standard (Glacier: minutes-to-hours; Deep Archive: ~12h),
then an ad-hoc query (or a one-off re-index into a temporary ES index)
runs over them. The manifest is the index for the cold tier — the
inverted index was aged out long ago.

Cost: ~10 TB/month highly-compressed × $1/TB-mo (Glacier) = ~$10/mo
per month of cold data; a year of cold = ~$120/mo. Negligible vs hot.

Splunk / Datadog Logs / CloudWatch Logs (managed)Proprietary index; you pay per GB ingested ($1–$3/GB at list).Rejected on cost — 43 TB/day × $2/GB = ~$86 k/day at list. Fine at small scale where ops headcount > infra cost; uneconomical at 2 M lines/sec.

Cold store: S3 Glacier + the Glue manifest (chosen) over S3 Glacier Deep Archive (cheaper but ~12 h retrieval — use it for >1 yr compliance archives) or keeping a manifest-only index in Postgres (works, but the Glue/Hive catalog already does partition metadata). Sharding the time-partitioned store is range by time (cite sharding-strategies#range) — the natural partition for logs; combined with hash by service within a day it's a composite key, which is exactly the service/date/hour partition path.

What breaks (C5; cite failure-modes.html)

  • The ES→Parquet export job falls behind. The midnight rollover exports a day's index (~17 TB) to Parquet before deleting it. If the export is slow (S3 throttling, a huge day), the ES index can't be deleted on schedule → SSD fills → ES goes read-only. Detection: free-SSD-percent + export-job lag. Recovery: the export is incremental (per-shard, resumable); a low-water-mark on SSD triggers an emergency older-index delete (the data is already in warm) before ES wedges.
  • A bad export — Parquet written, ES deleted, files corrupt. If the export writes garbage and we delete the ES index, that day is gone. Recovery: the export verifies (row count + checksum vs the ES index) before the delete; the ES index is also still in Kafka for 24 h as a last resort within that window.
  • A warm query that scans 232 TB (a scan-heavy request with no partition filter; failure shape: slow-dependency — Athena saturates and every other warm query queues). An engineer runs message CONTAINS "error" over all services × 30 days with no service filter → Athena scans the whole warm tier → ~hours, ~$1,160 in scan cost. Detection: pre-query estimate (partitions × avg size). Recovery: the Search API rejects queries whose estimated scan exceeds a cap (e.g. 1 TB) with a "narrow the time range or add a service filter" message — the same circuit breaker as the hot tier (Iter 4).
  • Cold restore is slow — by design. A compliance request for March data → Glacier restore takes hours. This is the documented degradation contract for the cold tier: it's an archive, not a search index. Detection/UX: the restore job returns a job-id and an ETA; the requester polls.

Partition pruning is the key. Without partitioning, a warm-tier query would scan all 232 TB — hours of work. With service + date + hour partitioning, a typical incident query scans 1-2 GB. The partition scheme must match query patterns: most queries filter by service name and time range. This is the telemetry cost shape in action: the 95%-never-read fraction (a flagged assumption) is the lever — it justifies dropping the inverted index for everything past 3 days, which is a 35× storage-cost reduction on those 27 days ($175 k/mo of ES-on-SSD → ~$5 k/mo of Parquet-on-S3). Schema/API delta: the warm partition layout is s3://logs/service={svc}/date={yyyy-mm-dd}/hour={hh}/part-*.parquet + a Glue catalog; the cold tier is the same layout in Glacier + the manifest stays hot.

Iteration 3: Volume Control and Live Tail

Two remaining challenges: (1) some services produce too many logs, overwhelming the pipeline, and (2) engineers want real-time log tailing during incidents.

Per-Service Rate Limiting

Without limits, one misbehaving service can flood the pipeline.

Example: A retry bug causes payment-service to log 500K lines/sec
(normally 20K). This single service consumes 25% of total capacity.

Solution: per-service rate limits at the agent level.

Each agent enforces:
  - Max lines/sec per container: 1,000 (configurable per service)
  - Burst allowance: 5x for 10 seconds (handles startup logs)
  - When exceeded: sample (keep 1-in-N) + log a warning

Sampling strategy when rate-limited:
  - Always keep ERROR and FATAL lines (never sample these)
  - Sample INFO lines at 1-in-10
  - Sample DEBUG lines at 1-in-100
  - Add "sampled: true, sample_rate: 10" to sampled lines
    so engineers know the true count = displayed × sample_rate

Dynamic Log Levels

Most services run at INFO level in production.
During incidents, engineers want DEBUG logs for specific services.

Implementation:
  - Central config service stores per-service log level.
  - Agents poll config every 10 seconds.
  - When log level changes to DEBUG for "payment-service":
    → Agent adjusts filter: now forwards DEBUG lines for that service.
    → Pipeline capacity check: if cluster has headroom, allow it.
    → If not: auto-enable sampling for INFO on OTHER services
      to make room.

This gives engineers DEBUG visibility during incidents without
overwhelming the pipeline. Automatic rebalancing prevents overload.

Cost: DEBUG logging typically 10x the volume of INFO.
  One service at DEBUG: 20K → 200K lines/sec. Affordable.
  All services at DEBUG: 2M → 20M lines/sec. Pipeline cannot cope.
  → Limit DEBUG to 5 services simultaneously.

Live Tail via Kafka

Engineers want: "show me payment-service ERROR logs RIGHT NOW."

Architecture: connect directly to Kafka, not Elasticsearch.

Live tail service:
  1. Engineer opens WebSocket: /tail?service=payment-service&level=ERROR
  2. Tail service creates a Kafka consumer starting at "latest" offset.
  3. Consumer reads from all 64 partitions.
  4. Filters in-flight: discard lines that don't match the query.
  5. Forward matching lines to the WebSocket.

Latency: Kafka end-to-end latency is 5-50ms.
  Log written by service → agent batches (100ms) → Kafka (5ms)
  → tail consumer reads (10ms) → WebSocket → engineer's screen.
  Total: ~150ms. True real-time.

Compare to Elasticsearch:
  Log → agent → Kafka → indexer batch (5s) → ES refresh (1s)
  → query → result.
  Total: ~6-30 seconds. Not acceptable for live debugging.

Tail Service Scaling

Each tail session consumes from all 64 Kafka partitions.
Kafka consumer throughput per session: reads 2M msgs/sec,
filters to ~100 matching lines/sec.

Problem: 200 concurrent tail sessions × 64 partitions =
  12,800 partition consumers. That is a lot of Kafka readers.

Optimization: shared consumers.
  - Group tail queries by their service filter.
  - 10 engineers all tailing "payment-service" share ONE
    Kafka consumer group that filters for payment-service.
  - Matching lines are broadcast to all 10 WebSocket sessions.
  - 200 engineers typically tail ~30 distinct services.
  - 30 shared consumers instead of 200 individual ones.

Memory per consumer: ~50 MB (read buffers).
30 consumers × 50 MB = 1.5 GB. One tail server handles it.

Alternatives considered (C4) — the sampling strategy

The decision: what gets dropped when volume exceeds capacity? (The bottleneck this iteration attacks is hot-key at the service granularity — one noisy service is a hot partition on the pipeline; cite bottleneck-to-decisions#hot-key: isolate the offender, don't let it steal everyone's capacity.)

  • No sampling, just scale to peak (rejected on cost): provision the whole pipeline (Kafka, indexers, ES) for 10 M/sec all the time — ~5× the steady-state cost for headroom used during incidents. Wrong unless logs are an audit record.
  • Head sampling at the agent (chosen for the volume-control role): the agent decides keep/drop per line as it's read, using a fixed per-severity ratio (keep all ERROR/FATAL, 1-in-10 INFO, 1-in-100 DEBUG when over the per-service quota) + stamp the kept line with its sample_rate. Cheap (the decision is local, O(1) per line), bounded (per-service quota), and the stamp makes the true count recoverable (displayed × sample_rate). The cost: a head decision can't be trace-aware — if you sample out an INFO line that belonged to a trace whose ERROR you kept, the trace is incomplete.
  • Tail sampling (considered, used for traces not logs): buffer all lines for a request/trace, then decide keep/drop once you've seen the whole trace (keep it iff it had an error or was slow). Trace-complete, but needs a buffering tier keyed by trace_id and a decision window — that's the traces pillar's design, not logs'. We borrow the idea only for the "always keep the whole trace if any line in it is ERROR" enhancement, applied best-effort at the agent.
  • No-sampling-but-cheap-storage (the Loki/ClickHouse philosophy): don't sample at all — just make ingest+storage cheap enough (label-only index, aggressive compression) that keeping everything is affordable. A legitimate alternative; at 10× this volume it's the better answer. We sample here because ES's full inverted index makes "keep everything hot" expensive, and because rate-limiting a runaway service is desirable independent of cost.

What breaks (C5; cite failure-modes.html)

  • A retry bug makes one service log 500 k lines/sec (25× normal). Without the per-service cap it'd consume 25% of pipeline capacity and lag everyone. With it: the agent caps that service at its quota (e.g. 1,000 lines/sec/container with a 5× burst), keeps all its ERRORs, samples the rest, and the dropped-count is recorded. Detection: per-service ingest rate vs quota; per-service drop rate. Recovery: alert the owning team ("you're dropping 480 k lines/sec, fix your retry loop"); the cap protects the pipeline meanwhile.
  • "A log line is missing — was it sampled or lost?" Ambiguity is itself a failure mode. Mitigation: sampled lines are stamped (sampled:true, sample_rate:N) and the per-service-per-minute dropped-count is emitted as its own log line, so a gap in the data has a number next to it ("INFO from svc-X, 14:03–14:04: 12,400 kept of ~124,000"). Lost (agent buffer wrap, indexer poison-pill) is a separate metric. The two are never conflated.
  • Dynamic DEBUG on too many services at once. If 50 engineers each flip a different service to DEBUG during a big incident, 50 × 10× volume = pipeline overload. Mitigation: the config service caps concurrent DEBUG-enabled services (e.g. 5); a 6th request either queues or auto-enables INFO sampling on others to make room. Detection: count of DEBUG-enabled services; total ingest rate vs capacity.
  • Live-tail consumer leak. A WebSocket that closes without cleanup leaves its Kafka consumer running → consumer-group churn. Mitigation: tail sessions have a TTL (idle > 15 min → close); the shared-consumer model means at most ~30 consumers exist regardless of session count. Detection: tail-consumer count vs distinct-service-filter count.

Degradation contract (the load-shedding order). Under ingest overload the priority is fixed and producers are never blocked: (1) keep ERROR/FATAL from every service, always; (2) sample INFO at the configured ratio; (3) sample/drop DEBUG hard; (4) if Kafka lag still grows, drop oldest agent-buffered lines (the 1 GB ring wraps) — new lines win over old; (5) the dropped count is always recorded so a gap has a number. The contract for the cold tier is separate: it's an archive, restores take hours, by design. Schema/API delta: the log line gains optional sampled/sample_rate fields; a synthetic __sampling_summary log line per (service, minute) carries kept/dropped counts; the live-tail endpoint is WS /v1/tail?service=&level=&query= reading the Kafka logs topic at the latest offset, filtering in-flight.

Iteration 4: Search Optimization

With 52 TB of hot data and 200 concurrent searchers during incidents, Elasticsearch needs careful tuning to meet the <5s search target.

Index-Per-Service vs Shared Index

Option A: One index per day (all services mixed)
  - logs-2026.03.22: 64 shards, ~14 TB
  - Query for one service scans ALL shards (service is just a filter)
  - 67 nodes each scan their local shards
  - Problem: 90% of data scanned is irrelevant to the query

Option B: Index per service per day
  - logs-payment-service-2026.03.22: 4 shards, ~200 GB
  - Query for payment-service scans ONLY its 4 shards
  - 4 nodes involved instead of 67
  - Problem: 10,000 services × 3 days = 30,000 indices.
    Elasticsearch cluster state grows to ~3 GB. Slow.

Option C (best): Index per service-group per day
  - Group 10,000 services into ~50 groups by volume:
    - High-volume (top 20 services): 1 index each = 20 indices/day
    - Medium (next 200): grouped by team = 20 indices/day
    - Low-volume (remaining 9,780): 10 catch-all indices
  - Total: ~50 indices/day × 3 days = 150 indices. Manageable.
  - Queries for a high-volume service scan only its dedicated index.
  - Queries for a low-volume service scan a shared index (small).

Query Performance Analysis

Query: service=payment-service AND level=ERROR AND last 1 hour

With dedicated index for payment-service:
  Index size for 1 hour: 14 TB/day / 24h / 50 groups = ~12 GB
  Shards: 4 (each ~3 GB)
  Nodes involved: 4
  Per-shard scan time: ~500ms (3 GB data, filtered by level)
  Parallel across 4 shards: ~500ms total
  ✓ Well within 5s target.

With shared catch-all index (low-volume service):
  Index size for 1 hour: ~600 MB
  Shards: 4 (each ~150 MB)
  Per-shard scan: ~50ms
  ✓ Even faster (less data to scan).

Trace query (trace_id=abc123):
  Must search across ALL indices (trace spans multiple services).
  150 indices × 4 shards = 600 shards.
  But trace_id is a keyword field (exact match, not full-text).
  Each shard: O(1) hash lookup = ~1ms.
  Fan-out to 600 shards in parallel: ~50ms.
  ✓ Fast because trace_id is indexed as a keyword.

Preventing Search from Killing Ingestion

During incidents: 200 engineers searching + 10M logs/sec ingesting.
Both compete for the same ES cluster resources (CPU, I/O, memory).

Isolation strategies:
  1. Dedicated coordinating nodes for search (3 nodes, no data).
     Ingestion goes through separate coordinating nodes.
     Search and ingestion never compete for the same coordinator.

  2. Thread pool separation:
     ES "search" thread pool: 50 threads.
     ES "bulk" thread pool: 32 threads.
     OS-level CPU pinning: search threads on cores 0-15,
     bulk threads on cores 16-31. No interference.

  3. Circuit breaker on heavy queries:
     If a query would scan > 100 GB, reject it with a suggestion
     to narrow the time range or add a service filter.
     Prevents one broad query from saturating the cluster.

  4. Read replicas for search:
     3 read-only replicas dedicated to search queries.
     Ingestion writes to primary shards only.
     Search reads from replicas. Complete isolation.

Caching Frequent Queries

During incidents, many engineers run the same query:
  "service=payment-service AND level=ERROR AND last 15 min"

Without caching: 50 identical queries each scan 3 GB. Wasteful.

ES request cache: automatic for identical queries on same shards.
  First query: 500ms. Next 49: 5ms (cache hit).
  Cache invalidated on shard refresh (every 1 second by default).

Additional: query result cache at the API gateway level.
  Cache key: hash(query + time_bucket).
  Time bucket: round "last 15 min" to nearest minute.
  TTL: 10 seconds (results are slightly stale but good enough).
  Hit rate during incidents: ~80% (many engineers, same query).

  50 queries/sec × 80% cache hit = 40 served from cache.
  Only 10 queries/sec actually hit Elasticsearch.

Alternatives considered (C4) — isolating search from ingest

The bottleneck this iteration attacks: search and ingest compete for the same ES cluster CPU/IO/memory, and an incident hits both at once (lock-contention / resource contention on the shared cluster). Options to isolate them:

  • Dedicated coordinating nodes + thread-pool separation + read replicas (chosen, all three): search queries land on coord nodes that hold no data and never participate in indexing; the ES search and bulk thread pools are sized and CPU-pinned separately; 1 replica per primary serves reads while primaries take writes. Layered isolation — cheap (config + 1 extra replica) and effective.
  • Two physically separate ES clusters (write cluster + read cluster, cross-cluster replication) (rejected as overkill here): total isolation, but doubles the cluster count and adds CCR lag; warranted only when even the replica-on-the-same-node contention is unacceptable.
  • Index-per-service-group (chosen) vs index-per-day (rejected: 90% wasted scan) vs index-per-service (rejected: 30 k indices → ~3 GB cluster state, slow): grouping the 10 k services into ~50 volume-tiered groups gives targeted scans (a high-volume service hits only its own ~12 GB/hr index) without index explosion. Cite sharding-strategies#composite — the index key is (service-group, date), a composite of hash-by-service and range-by-time.
  • Query result cache: API-gateway cache + ES request cache (chosen, both layers) over no cache (50 identical incident queries each scan 3 GB) or cache-only-at-ES (misses the cross-shard dedup the gateway cache gives when many engineers run the same time-bucketed query). The gateway cache keys on hash(query + rounded_time_bucket), 10 s TTL — ~80% hit during incidents, so ~10 of ~50 QPS actually reach ES. Cite caching.html.
  • Heavy-query circuit breaker (chosen): reject a query whose estimated scan exceeds a cap (100 GB hot / 1 TB warm) with a "narrow it" message — one broad query can't saturate the cluster. The same breaker covers the warm tier (Iter 2's 232 TB-scan failure mode).

What breaks (C5; cite failure-modes.html)

  • A "show me everything" query during an incident. An engineer runs level=ERROR with no service filter over 3 days → fans out to all 150 hot indices × their shards → every data node is busy → ingest indexers (sharing the cluster) slow → Kafka lag grows. Mitigation: the circuit breaker rejects it pre-execution (estimated scan > 100 GB); read replicas mean even an accepted broad query doesn't touch the indexing primaries. Detection: per-query estimated-scan-bytes; search-thread-pool queue depth.
  • A hot service-group index. If the volume-tiering is stale (a service grew 10× and is still in a shared catch-all index) that index's shards are overloaded on both write and read. Detection: per-index write+search rate vs the group's expected share. Recovery: a daily re-tiering job moves a grown service into its own index at the next midnight rollover.
  • Cache stampede on the gateway query cache. When the 10 s TTL expires during an incident, 50 engineers' identical query all miss simultaneously → 50 concurrent ES queries instead of 1. Mitigation: single-flight on the cache key (the first miss runs the query; the rest wait for its result). Detection: cache-miss burst rate.
  • A slow ES shard drags p99. One garbage-collecting or disk-throttled data node makes the slowest shard in a 64-shard fan-out dominate latency (the classic tail-at-scale). Mitigation: ES adaptive replica selection (route to the faster replica) + a per-shard timeout that returns partial results with a "incomplete" flag rather than hanging. Detection: per-shard search-latency histogram; partial-result rate.

Search and ingest stop competing. Dedicated coord nodes, thread-pool + CPU isolation, read replicas, index-per-service-group, a two-layer query cache, and a heavy-query circuit breaker together let 200 engineers search a 52 TB hot tier at p99 < 3 s while 10 M lines/sec are being indexed. Schema/API delta: ES index mapping below (C9); the search API is GET /v1/search?q=&from=&to=&cursor=&limit= — routes to ES (hot) or Athena (warm) by time range, cursor pagination (cite api-design-patterns#pagination), rejects over-cap scans (cite api-design-patterns#errors).

Degradation contract

When Kafka is unavailable: agents buffer to their local 1 GB disk ring and retry; once Kafka is back, the backlog drains; if a host's ring fills (Kafka down > ~30 min at that host's rate) oldest buffered lines wrap — new lines win. Producers never block. When Elasticsearch (hot) is degraded: search over the last 3 days returns partial results with a partial:true flag; live tail (reads Kafka, not ES) is unaffected; ingest indexers back off and Kafka lag grows (count of un-indexed lines), draining when ES recovers. When the indexer fleet falls behind: ingest-to-searchable lag grows from ~6 s toward minutes; no lines are lost (durable in Kafka); auto-scale to burst, and if lag still grows, the agent rate limiter sheds. When S3 / Athena (warm) is unavailable: warm queries fail fast with "warm tier unavailable, try again"; hot search and ingest are unaffected. When the whole pipeline is overloaded: the fixed load-shed order — keep all ERROR/FATAL, sample INFO, drop DEBUG, drop-oldest-agent-buffered, always record the dropped count — runs automatically; the cold-tier contract (restores take hours) is unchanged. The one invariant: logging is best-effort and must never impact production services.

Operations (C6)

  • Metrics (SLI dashboard). Ingest-to-searchable lag p99 (the <30 s SLO); hot search p99 (the <5 s SLO); per-partition Kafka consumer lag; per-service ingest rate vs quota + per-service drop rate (the "sampled vs lost" accounting); agent disk-buffer utilization per host; ES free-SSD-percent (rollover/export health); ES search-thread-pool queue depth; gateway query-cache hit rate; live-tail consumer count vs distinct-filter count; warm-query estimated-scan-bytes distribution; Athena scan-cost/month.
  • Alerts. Kafka lag > 30 min on any partition; ES free-SSD < 15%; ingest-to-searchable lag > 60 s; any service dropping > X lines/sec for > 5 min (page the owning team, not the logging team); > 5 services at DEBUG simultaneously; export job behind schedule.
  • Rollout. Agent config (rate limits, sampling ratios, log levels, redaction rules) is pushed via the central config service, polled every 10 s, canaried by host cohort (1% → 10% → 100%). ES mapping changes go through a new daily index (next rollover) — never re-map a live index. The volume-tiering job (which services get a dedicated index) reruns daily and takes effect at midnight rollover.
  • Migration (Naive → Final). Phased, no log-loss window: (1) deploy Fluent Bit agents alongside the direct-to-ES path; agents dual-write (still POST to ES and to Kafka) for a soak period; (2) stand up the indexer pool reading Kafka, writing to a parallel set of ES indices; compare; (3) flip the search API to read the indexer-fed indices; turn off the direct-to-ES path; (4) layer in tiering (the midnight export job + warm/cold), then per-service rate limiting + sampling, then live tail, then the search-isolation tuning — each behind a flag. (5) Decommission the old all-hot ES cluster once the new hot tier + warm export have soaked. ~60–90 days.
  • Headroom. +30% on ES data nodes and indexers (autoscale on lag); the 24 h Kafka retention is itself ~173 TB of burst headroom; +50% on the live-tail servers; the per-service rate limits leave a 5× per-container burst allowance for startup-log floods.

Real-system comparison

This shape is what the major logging platforms converge to. Elastic's own reference architecture uses hot/warm/cold/frozen ILM tiers (hot on NVMe, frozen searching directly off S3 via "searchable snapshots") — the Iter 2 tiering here, plus a fourth tier. Grafana Loki is the "label-index-only, grep the rest, everything in object storage" alternative discussed in Iter 2 — it explicitly trades full-text index cost for cheaper storage and a simpler operational model, and pairs with Tempo (traces) and Mimir (metrics) for the three pillars. ClickHouse (used for logs at scale by Uber, Cloudflare, and others) is the columnar-with-bloom-skip-index alternative — one engine, great compression, fast scans, no relevance ranking. OpenObserve / SigNoz are recent open-source platforms built directly on the "object storage + columnar + bloom skip indexes" model. The ingest side — a per-host agent (Fluent Bit / Vector / the OTel Collector) → Kafka → an indexer pool — is the de-facto standard; Confluent, Datadog, and most internal platforms run it. The error-tracking specialization (#42) is what Sentry/Bugsnag/Rollbar do: same agent→Kafka→consumer ingest, but the consumer fingerprints and folds the firehose into an issues rollup instead of indexing every line.

Anticipated pushbacks

Stress tests against the derivation above. If an answer doesn't fit in 3–6 sentences, a section is missing — close it before declaring done.

What if ingest is 10× this — 20 M lines/sec average? (scale-out)

  • Linear-ish on the ingest path: Kafka ~50 brokers, indexers ~320, Kafka storage ~430 TB — all proportional, nothing structurally changes.
  • The hot ES tier is the problem: ~170 TB/day of index × 3 days ≈ 520 TB on SSD ≈ ~$52 k/mo just for the hot index, and the search fan-out gets wider. This is the point where the warm-tier alternative becomes the default for the whole stack: switch to Loki (label-only index) or ClickHouse (columnar + bloom skip) for everything, drop the full inverted index, and accept slower-but-cheaper full-text. The flagged assumption "ingest volume" calls this out.
  • Also reconsider the hot-window length — at 10× volume, "3 days searchable" might shrink to "1 day fast, the rest on the columnar tier."

Why sample at all — why not keep everything? (sampling)

  • We don't lose counts — sampled lines carry their sample_rate and the dropped-count is emitted as its own line, so "the rate of this INFO line" is recoverable. What we don't keep is every byte of every duplicate INFO line during a flood — the 100,000th identical "request handled" line tells you nothing the 100th didn't.
  • Sampling exists for two reasons: cost (ES's full inverted index makes "keep everything hot" expensive) and protecting the pipeline from a runaway service — rate-limiting a buggy retry loop is desirable independent of cost. The "no-sampling-but-cheap-storage" answer (Loki/ClickHouse) is legitimate and is the right call at much larger scale; it doesn't address the runaway-service problem, which is why a per-service cap still belongs in that design too.
  • The escape hatch: a service (or an on-call engineer during an incident) can flip "keep all lines for service X" — full fidelity when you actually need it, at the cost of pipeline headroom.

What changes if logs are an audit record — must be complete & immutable for compliance? (compliance)

  • No sampling, ever — the per-service "cap" becomes a "alert hard and provision for it" instead of "shed." Kafka RF ≥ 3 (was 2), and the agent disk buffer is sized for the worst-case Kafka outage with a hard alert if it nears full (the "lose old buffered lines" contract is unacceptable).
  • The archive becomes write-once / immutable (S3 Object Lock in compliance mode) with a Merkle digest per partition so tampering is detectable; retention extends to years (Glacier Deep Archive).
  • A separate exact-count side-channel: per (service, minute) line counts written to an append-only store independent of the index, so "did we receive all of service X's logs for 14:03?" has an authoritative answer that doesn't depend on the index being healthy.
  • Cost roughly doubles (RF 2→3 on Kafka, no sampling means provisioning for peak, immutable archive overhead). That's the price of "audit record"; "debugging aid" is much cheaper. This is the first clarification-ledger question for a reason.

What if Elasticsearch falls over under an ingest spike? (failure)

  • It doesn't take the system with it — that's the whole point of the Kafka buffer. ES going read-only or slow means ingest-to-searchable lag grows (a number on the dashboard), not lines lost (they're durable in Kafka for 24 h) and not producers blocked (the agent never waits on ES).
  • Indexers detect the slowdown (bulk-write latency / error rate), back off, and let Kafka lag build; auto-scale adds indexer capacity for the catch-up once ES recovers. If ES is wedged because SSD filled (rollover/export behind), the low-water-mark trigger emergency-deletes the oldest hot index (data's already in warm).
  • Search degrades to partial results (partial:true) with adaptive replica selection routing around the slow node; live tail is unaffected (it reads Kafka). Recovery is "re-index from Kafka" within the 24 h window, or "re-export from warm" beyond it.

Where's the cost? (cost shape)

  • It's the hot index in RAM/SSD + the ingest path, not the data at rest. ES data nodes (SSD-resident inverted index for the searchable window) are ~66% of the bill; Kafka + indexers ~17%; warm/cold storage for the 95%-never-read bulk is ~11%. The diagnostic question — "where's the elephant: the firehose in, the index hot, the data at rest, or the query?" — answers "the hot index, then the firehose."
  • So the knobs that move the bill are: hot-window length (3d→7d ≈ +$30 k/mo), the hot-tier engine (ES inverted index vs Loki label-only vs ClickHouse columnar), and the sampling ratio (less data → smaller hot index). Tweaking the cold-storage tier barely moves anything.
  • This is the telemetry cost shape: huge volume, mostly never read → keep the alerts/aggregates hot, sample the raw, tier the storage. The mistake is treating logs like a database (everything queryable, everything fast) — that's the Naive design's $181 k/mo.

How do you migrate from a single Elasticsearch to the tiered design? (migration)

  • Phase 1 — dual-write: deploy Fluent Bit agents alongside the existing direct-to-ES path; agents POST to ES and publish to Kafka for a soak period. Zero risk — the new path is shadow.
  • Phase 2 — parallel index: stand up the indexer pool reading Kafka, writing to a parallel set of ES indices (with the new service-group layout); compare query results against the old indices for a week.
  • Phase 3 — flip: point the search API at the new indices; turn off the direct-to-ES write path; the old monolithic cluster keeps serving historical queries until its data ages out.
  • Phase 4 — layer in the rest: the midnight export job (hot→warm Parquet), then per-service rate limiting + sampling, then live tail, then the search-isolation tuning — each behind a flag, canaried. Decommission the old cluster once the new hot tier + warm export have soaked ≥ 30 days. ~60–90 days, no log-loss window.

A log line is missing — was it sampled or lost? (observability of the logger)

  • The system makes this answerable. Sampled: the kept lines carry sampled:true, sample_rate:N, and a synthetic __sampling_summary line per (service, severity, minute) carries "kept K of ~M" — so a gap has a number next to it. Lost: separate metrics for agent-buffer-wrap (per host), indexer poison-pill drops (per partition), and Kafka-side loss (should be ~0 with RF=2 + idempotent producer). The two are never conflated; the dashboard shows both.
  • If neither metric fired and the line is genuinely gone, that's a bug (a parse failure that silently dropped instead of quarantining, a mapping rejection) — the indexer's dead-letter topic captures these for inspection.

A tenant logs 100× normal — fairness and cost attribution? (noisy neighbor)

  • The per-service rate limit at the agent is the first defense — that service is capped at its quota (with a 5× burst allowance), keeps all its ERRORs, samples the rest, and the dropped-count is recorded. Its 100× spike can't steal the pipeline's capacity from the other 9,999 services (isolate the hot key).
  • Cost attribution: Iter 3's per-service ingest accounting (which the rate limiter already needs) gives a per-service GB/day number; in a charge-back model that's the noisy team's bill, and it's their incentive to fix the retry loop. Without charge-back, an alert pages the owning team.
  • If the spike is legitimate (a real incident generating real errors), the ERRORs all flow through (never sampled) and the engineer can temporarily lift that service's quota or flip it to "keep everything" — the cap is a default, not a hard wall.

Key Takeaways

  • Never let logging block application threads. Services write to stdout (a kernel pipe, microseconds). A local agent handles everything else asynchronously. If the logging pipeline is down, services must not notice.
  • Kafka is the shock absorber. Incidents cause a 5x spike in log volume — exactly when engineers need search. Kafka absorbs the burst and lets indexers catch up afterward. Without a buffer, you must provision for peak (5x the cost) or drop logs during incidents.
  • Tiered storage saves 73%. 95% of log queries hit the last 3 days. Keeping 30 days in Elasticsearch wastes $133K/month. Hot (ES on SSD), warm (Parquet on S3), cold (Glacier) matches storage cost to access frequency.
  • Index per service-group, not per service. 10K services × 3 days = 30K indices would overwhelm ES cluster state. Grouping into 50 service-groups gives targeted search performance (scan only relevant data) without index explosion.
  • Live tail goes through Kafka, not Elasticsearch. Kafka delivers logs in 150ms. Elasticsearch takes 6+ seconds (batching + indexing + refresh). For real-time debugging, read directly from the message bus.
  • Per-service rate limiting prevents cascade. One buggy service should not be able to saturate the logging pipeline. Rate limit at the agent, always keep ERROR lines, sample lower-severity lines when over quota.
  • The textbook inverted index assumes a fixed corpus that fits in RAM — logs break all three. The corpus is a firehose (so the index-write path is the bottleneck → LSM/append-optimized index + a Kafka buffer in front), it's 258 TB and growing (so tier it — full index hot, label/columnar warm, scan-on-demand cold), and 95% of it is never read (so don't index what nobody queries). "Merge K sorted streams" of postings becomes "merge results across time-partitions and storage tiers."
  • Know the cost shape: this is telemetry, not a database. The bill is the hot index (~66%) + the ingest path (~17%), not the data at rest (~11%). The mistake is making everything queryable-and-fast; the fix is keep alerts/aggregates hot, sample the raw, tier the storage. The Naive "30 days all in ES on SSD" is $181 k/mo; the tiered design is ~$48 k/mo for the same retention.
  • "Sampled" and "lost" must be different words. A gap in the data needs a number next to it: sampled lines carry their sample_rate, dropped counts are emitted per (service, minute), and lost lines (buffer wrap, poison-pill) are a separate metric. Ambiguity ("is this line missing because we sampled it or because we lost it?") is its own failure mode.
  • The error-tracking specialization folds the firehose into one row per group. When the "log lines" are stack traces and the value is dedup — #42 — you don't index every line; you fingerprint it, increment an issue's counter, and keep K samples. Same agent→Kafka→consumer ingest; the consumer collapses instead of indexes.

Assumptions and When They Break

  • Structured JSON logs: We assumed all services emit structured JSON. If services emit unstructured text, the agent must parse it (regex-based, expensive). Enforcing a logging SDK is critical for pipeline efficiency.
  • Elasticsearch for search: At even larger scale (10x), consider purpose-built log search engines like ClickHouse or Grafana Loki that use label-based indexing instead of full-text inverted indices, reducing index storage by 5-10x.
  • Single-region deployment: We assumed all services are in one region. Multi-region logging adds cross-region shipping costs ($0.01/GB) and requires regional Kafka clusters with a global search federation layer.
  • 30-day retention is enough: Some compliance regimes (SOX, HIPAA) require 1-7 years. The cold tier (S3 Glacier) handles this cheaply (~$1/TB/month), but re-indexing old data for search takes hours.
