Designing an Object-Store Metadata Service

S3/GCS-style metadata layer (NOT the data plane). 1 T objects; 200 k metadata QPS (GET-by-key dominant + LIST-prefix); 10 k writes/sec; LIST p95 ≤ 200 ms even on prefixes with 100 M children; eleven nines durability for the metadata. We start from a single Postgres table keyed by (bucket, key) — the LC #14 "in-memory file system" tree-of-maps lifted directly — that doesn't fit on one node and serializes writes on a single primary, then iterate to a range-sharded FoundationDB cluster with auto-splitting hot ranges, a paginated LIST path with an immediate-children index, hedged-read tail-tolerance, and regional cells.

NALSD Walkthrough • ~50 min read • LC #14 (Design In-Memory File System)

LC kernel. Design In-Memory File System (LC #14). Textbook structure: tree of maps — each path component is a key in a hash-map at one level, the value is either a leaf (file metadata) or another map (directory). LS at /a/b/ walks the tree to the right node and returns the keys at that level; PUT at /a/b/c walks the tree, creates intermediate maps if missing, sets the leaf. Insert is O(path-depth); LS is O(children-at-that-level); GET-by-path is O(path-depth). Three properties that break at 1 T objects: (1) the tree doesn't fit on one node — 1 T rows × ~500 B/row ≈ 500 TB raw, way past a single node's NVMe; (2) hot prefixes form — date-stamped keys (logs/2026/05/08/), customer-id prefixes, and write-once-then-list workloads concentrate writes on a single subtree; (3) LIST-prefix and PUT-by-key have opposing layout preferences — LIST wants contiguous-on-disk children (range partition), PUT wants uniform load (hash partition). The C8 production cousin replaces the single-tree-of-maps with a range-sharded ordered KV (FoundationDB / Bigtable / Spanner style), auto-splits hot ranges by per-range write-QPS threshold, and adds a per-prefix immediate-children index so a LIST under a 100 M-child prefix doesn't scan all 100 M rows.

S0. Scenario & Requirements

An S3/GCS-style object store has two planes: a data plane (the bytes of each object, stored as immutable blobs on erasure-coded disk) and a metadata plane (the path-to-blob mapping, ACL, versioning, multipart upload state, lifecycle rules). This page designs only the metadata plane — the data plane is out of scope (that's a sister scenario; see #43 for content-addressed blob storage). Path lookup, listing under a prefix, ACL evaluation, version chain walk, and multipart-upload state machine all live on this layer.

System class

Class: storage-shaped — "persist X durably and surface Y view of it later" capabilities (cite functional-requirements.html#class-storage); the FRs are operations on a key-ordered map + their durability and ordering guarantees. The class fixes the NFR additions: durability (loss is unacceptable; metadata loss orphans the data plane), reconstruction latency on LIST (not just point GET), and ordered-prefix iteration as a first-class operation (not a side-effect of B-tree leaf walks).

Functional requirements (cite functional-requirements.html)

  • GET object metadata by key. Given (bucket, key), return {etag, size_bytes, content_type, version_id, acl_ref, blob_locator, last_modified}. The dominant 95% of metadata QPS.
  • PUT object metadata. On data-plane upload completion, atomically write a new metadata row keyed by (bucket, key) with a fresh version_id; invalidate any cached LIST result for the parent prefix.
  • DELETE object. Tombstone the metadata row (the data-plane blob is GC'd later by a reaper); preserves prior versions if versioning is enabled on the bucket.
  • LIST prefix. Given (bucket, prefix_string, delimiter, continuation_token, max_keys), return up to max_keys children of prefix_string in lexicographic order, plus a next_continuation_token. Common prefixes (when delimiter='/') are returned as pseudo-directories. This is the load-bearing FR — LIST p95 ≤ 200 ms even on 100 M-child prefixes is the SLO that drives the entire iteration arc.
  • List versions of a key. Walk the version chain for (bucket, key) in time-descending order; needed for restore-from-prior-version.
  • Multipart upload state. CreateMultipartUpload → UploadPart (n times) → CompleteMultipartUpload. The metadata layer holds the in-flight parts list and atomically swaps to the final object on Complete; AbortMultipartUpload reaps the parts. Each upload session is its own row with a TTL.

Out of scope (named, not silently dropped): the data plane itself (bytes, erasure coding, durability of the blob — see #43); cross-region replication of metadata (we run in one cell; multi-region is Iter 4); object-locking / WORM compliance (separate enterprise SKU); event notifications (S3 Event Notifications — built on top of the metadata change log; out of scope this page); per-object encryption with customer-managed keys (server-side encryption with our keys is in scope; KMS-CMK is Probe A); inventory reports (large-scale offline LIST; built from a nightly snapshot of the metadata; out of scope).

Flagged assumptions. (1) Average key length is ~200 B (path components plus bucket plus version_id); (2) average row size is ~500 B (key + small JSON metadata + ACL ref + blob_locator); (3) the read:write ratio is ~20:1 (200 k GETs/sec : 10 k writes/sec); (4) within reads, GET-by-key dominates at ~95%; LIST is ~5% but its tail latency on hot prefixes is the load-bearing failure mode; (5) hot-prefix patterns are real and known: date-stamped keys (logs/<YYYY>/<MM>/<DD>/...), customer-id prefixes (cust/<customer_id>/...), and sequential ETLs (events/<hour>/<sequence>) all concentrate writes on a single contiguous range.

Sister scenarios (see plan)

  • #43 Content-defined dedup for backup storage — the data-plane sister: byte storage, dedup, restore. The chunk index there is roughly 12 B keys; the metadata service here is 1 T keys with prefix-listing. Drastic flip: there the dominant access is point-read on a 32-byte content hash (no order needed); here LIST-prefix forces ordered storage.
  • Filesystem metadata service (planned; HDFS NameNode / GFS master / Tectonic) — same shape with stronger consistency (POSIX semantics: rename is atomic, hardlink, fsync). Drastic flip: add atomic rename across prefixes, which forces multi-key transactions at higher write cost.
  • Container-image registry (planned) — metadata for image layers; layer count per registry is ~10 B vs our 1 T; LIST is rare (humans browse, machines pull by digest). Drastic flip: hot-prefix problem disappears; hash-shard wins outright.
  • #35 Web crawler — the per-URL-by-prefix sharding sister: the URL frontier is keyed by domain prefix and concentrates on hot domains exactly the way our hot prefixes concentrate on hot dates.

The fundamental tension. A metadata service for 1 T objects is squeezed by three constraint-pairs: (a) cardinality vs single-node fit — 500 TB raw cannot live on one node; sharding is mandatory. (b) LIST locality vs PUT load-balance — LIST-prefix wants contiguous-on-disk neighbors (range partition); PUT wants uniform write load (hash partition). The two preferences are opposite; a single sharding choice can't satisfy both. (c) hot-prefix concentration vs per-shard QPS ceilings — date-stamped keys, customer-id prefixes, and sequential ETLs concentrate writes on one contiguous range; range partitioning makes the hot prefix into a hot shard, capping at the per-node QPS limit. The arc resolves the tension: range-shard for LIST locality (Iter 1), auto-split hot ranges by write-QPS threshold (Iter 2), add a per-prefix immediate-children index + paginate + hedge (Iter 3), regional cells for cross-region latency (Iter 4).

S1. Core Entities (cite entities-and-data.html)

Six entities. The ObjectMetadata row is the load-bearing one (it's the unit of read, write, and prefix-listing) and gets a separate engine. The PrefixIndex entity (Iter 3) is the second load-bearing one because it's how 100 M-child prefixes get listed in 200 ms.

ID scheme (cite identifier-schemes.html#composite): partition_key = (bucket_id, key) for the ObjectMetadata table, where bucket_id is a UUIDv4 and key is the customer-supplied path string (up to 1024 bytes). Why a composite key and not a content hash: LIST requires ordered storage on the key — a content hash spreads keys uniformly and destroys lexicographic order. Why not just the key: prefix collisions across buckets would conflate two customers' namespaces (e.g., two customers each with logs/2026/...); the bucket_id is the tenant boundary, prefixed onto every key. The version_id is appended as a clustering suffix so a point GET on (bucket, key, latest) reads the most-recent version directly; a version-chain walk reads adjacent rows.

The partition-key heuristic: the dominant access patterns are point GET on (bucket, key) (95% of reads) and LIST on (bucket, prefix) (5% of reads but the SLO that defines the system). Both want ordered storage on (bucket_id, key): GET is a point-read into the ordered keyspace, and LIST is a contiguous range scan of the same keyspace. Range-partition by (bucket_id, key) with auto-splitting — the FoundationDB / Bigtable / Spanner core trick — gives both. Hash-partition would be cheaper to implement but would destroy LIST locality; we explicitly reject it in Iter 0→1 with numbers.

Six datasets, three engines. The dividing line is per-dataset access pattern: ordered KV with auto-splitting for the metadata table; hash KV for upload sessions; B-tree SQL for low-cardinality config.

Summary. 1 SQL (Postgres for bucket config) + 1 ordered-KV (FoundationDB for ObjectMetadata; introduced in Iter 1) + 2 hash-KV (RocksDB for multipart upload state and ACL policies) + 1 cache (Redis; from Iter 3) + 1 log (Kafka; from Iter 3). System-of-record: FoundationDB — a metadata row's existence is authoritative there; everything else (Redis cache, Postgres bucket config) is either a derived store or a configuration store. Loss of any cache or Kafka topic does not lose metadata; loss of FoundationDB is metadata loss (and orphans the data-plane bytes).

Per-endpoint SLO

  • GET object — p99 ≤ 10 ms; the dominant volume.
  • PUT object — p99 ≤ 50 ms (synchronous quorum write to FoundationDB across 3 AZs).
  • LIST prefix — p95 ≤ 200 ms even on 100 M-child prefixes (the load-bearing SLO).
  • DELETE object — p99 ≤ 50 ms (quorum write to tombstone the row).
  • CompleteMultipartUpload — p99 ≤ 200 ms (atomic transaction across the upload-session row + new metadata row).

Error model: S3-compatible error codes (NoSuchKey, NoSuchBucket, InvalidContinuationToken, SlowDown); 503 with Retry-After on per-bucket throttle; 412 (PreconditionFailed) for conditional writes. Versioning: /v1 URL prefix.

S4. Naive Architecture — the simplest thing that meets the FRs

One Postgres table keyed by (bucket_id, key), B-tree-indexed on the composite key, with a single primary node and 2 read replicas. GET is a B-tree point lookup; LIST prefix is a B-tree range scan starting at the first key with that prefix and walking forward; PUT is an INSERT/UPSERT on the primary; DELETE sets a tombstone column. This is LC #14's tree-of-maps reference solution lifted directly: a single ordered map keyed by path, with prefix listing as ordered iteration.

Architecture

Connections (numbered above) and line-style legend. Solid slate = sync hot path; dashed gray = async replication. The Naive design has no fan-out and no fallback — every byte traverses the same blocking path through one primary.

  • ① Client → API Gateway — HTTPS S3 SDK request; the gateway parses bucket + key out of the URL and forwards.
  • ② Gateway → Postgres PRIMARY (write path) — INSERT INTO objects (bucket_id, key, version_id, ...) VALUES (...) on the single primary node. All writes serialize through this one box.
  • ③ Gateway → Read Replica (read path) — SELECT ... FROM objects WHERE bucket_id=? AND key=? for GET; ... WHERE bucket_id=? AND key >= ? AND key < ? ORDER BY key LIMIT 1000 for LIST. Replicas serve reads to keep load off the primary.
  • ④ Primary → Replica 1 (async stream) — standard Postgres logical/physical replication; ~100 ms replication lag at peak.
  • ⑤ Primary → Replica 2 (async stream) — same.

This is the simplest architecture that satisfies all five FRs — one ordered map keyed by (bucket_id, key), with prefix listing as B-tree leaf walk. No sharding, no auto-split, no LIST-side index.

Why it's broken (per failing NFR)

  • Storage capacity. ✗ 500 TB raw vs 3 TB on a single i4i.4xlarge NVMe. Even with the most aggressive LSM compression (3×) we'd need 57 nodes for storage with RF=3 = 171 nodes — which requires sharding. A single-primary Postgres simply cannot hold 1 T metadata rows.
  • Write throughput. ✗ ~5 k writes/sec on a single primary; the SLO needs 10 k sustained, 40 k burst. The primary is the serialization point on the WAL fsync; no amount of replicas helps.
  • LIST p95 on 100 M-child prefix. ✗ ~20 sec on a hot-cached single-node B-tree leaf walk vs 200 ms SLO — 100× over. LIST returns at most 1000 keys per page, but the B-tree leaf walk has to traverse 20 M leaf pages just to gather the first page (because each leaf has only 5 rows and most leaves contain rows from other prefixes interleaved). Even on hot cache, traversal is O(children-in-prefix), not O(page-size).
  • Hot-prefix isolation. ✗ Date-stamped keys (logs/2026/05/08/) all land in the same B-tree leaf range; the leaf-page lock contention is fatal. There's no notion of a "shard" to split off the hot range to.
  • Durability. &warning; Postgres physical replication is async by default; a primary crash before WAL ships can lose seconds of writes. Synchronous replication is configurable but caps write throughput further (already inadequate at ~5 k/sec).

Cost of naive at NFR-target load: can't even be stood up — the data doesn't fit. If we assume a hypothetical Postgres-on-110-NVMe-volumes monstrosity with $400/TB-mo EBS, raw storage is 330 TB × $400 = $132 k/mo just for disk; the single primary is the LIST and write bottleneck regardless. The first failing NFR is storage capacity (the data physically doesn't fit on one node). The second is write throughput (single primary can't sustain 10 k/sec). The third is LIST p95 (B-tree leaf walk on a 100 M-child prefix is O(N) regardless of node count). We must shard the metadata table — not scale the primary vertically.

Delta vs Naive: the single-primary Postgres becomes 32 hash-sharded RocksDB clusters keyed by bucket_id. Bottleneck

The Naive single-primary Postgres physically can't hold 500 TB raw and serializes writes through one WAL (~5 k writes/sec ceiling). Classified as memory-capacity-bound on the storage axis and coordinator-bottleneck on the write axis. The corresponding access pattern is write-heavy-uniform across buckets — many tenants writing concurrently to disjoint buckets — for which the canonical fix is shard the write path by tenant key. Iteration 0 fixes the capacity + write-throughput failures: hash-shard by bucket_id across 32 physical clusters so storage and writes scale linearly with shard count.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#coordinator-bottleneck: shard the write path; remove the single serialization point. The canonical first move on a "doesn't fit + writes serialize" diagnosis is hash-sharding by the natural tenant key (here, bucket_id). It's the cheapest, simplest sharding strategy — uniform load by construction (UUIDs are uniform), no auto-split logic, no rebalancer. Cite sharding-strategies#hash.

Alternatives considered (C4)

  • Hash-shard by bucket_id — chosen for this iteration. Simplest sharding strategy; uniform load; per-bucket LIST stays on one shard (good). All metadata for one bucket lands on one shard, so within-bucket prefix LIST still walks an ordered keyspace.
  • Hash-shard by (bucket_id, key) — rejected. Spreads each bucket's keys across all 32 shards; uniform load on writes and reads, but destroys LIST locality entirely — a LIST under logs/2026/05/08/ would scatter-gather across all 32 shards, do a per-shard local prefix scan, merge the results, and paginate. Even at 5 ms p99 per shard, scatter-gather of 32 shards has tail-latency p95 closer to 30 ms just for the scatter, before any per-shard scanning. We reject this option here precisely because the next iteration (Iter 1) shows that even hash-by-bucket has a fatal problem with hot prefixes; the right answer is range-sharding, not finer-grained hashing.
  • Range-shard by (bucket_id, key) with auto-split — rejected for THIS iteration only. The right long-term answer (Iter 1), but skipping straight to it would skip the pedagogical "what does hash-by-tenant get us, and what does it still miss?" step. Hash-by-bucket is the textbook "first sharding move" that an engineer reaches for; we show why it doesn't go far enough.
  • Single Postgres with logical sharding via partitioning — rejected. Postgres declarative partitioning helps with table size but not with single-primary write serialization (the WAL is per-cluster, not per-partition). Citus (Postgres-based sharded SQL) is closer but lacks the auto-split feature we need in Iter 1.

The chosen move

32 RocksDB-backed metadata shards (each running on i4i.4xlarge with NVMe), partitioned by hash(bucket_id) % 32. Each shard holds ~31 k buckets (10 M / 32) and the ~31 billion rows belonging to those buckets. RF=3 across 3 AZs via RocksDB's built-in replication (or via a Raft layer like ScyllaDB; we name RocksDB for the engine and assume a thin Raft replication layer). API Gateway computes hash(bucket_id) % 32 and routes to the shard's leader for writes / a follower for reads. Within a shard, RocksDB stores keys lexicographically, so LIST under a prefix on a single bucket walks contiguous SSTs on that shard's NVMe.

Cite sharding-strategies.html#hash; cite storage-engines.html#lsm.

NFR check. Storage capacity ✓ (170 TB fits across 192 nodes). Write throughput ✓ (~960 k aggregate; 96× the 10 k SLO). GET p99 ✓ (~3 ms). Durability ✓ (RF=3 quorum). LIST p95 ✗ on 100 M-child single-bucket prefix — still ~20 s because hash-by-bucket keeps all of a bucket's keys on one shard, and that shard still does an O(N) leaf walk on the prefix. But also: hash-by-bucket spreads buckets uniformly across shards, but a single hot bucket (the top 0.01% of customers running ETLs) keeps all its writes on one shard — capping at 30 k writes/sec, fine for now. The real failure is LIST p95 on prefixes with 100 M children — the load-bearing SLO still fails by 100×.

What breaks (C5; cite failure-modes.html)

  • Hot bucket caps at one shard's QPS ceiling. If one customer runs an ETL writing 50 k objects/sec to a single bucket, all those writes hash to one shard and saturate its 30 k/sec ceiling. Detection: per-shard write-QPS p99; alert at >25 k. Mitigation: not addressed here — Iter 1's range-sharding with auto-split is the structural fix.
  • Hot prefix on date-stamped keys within one bucket. A customer's logs/2026/05/08/ prefix is one B-tree leaf range on one shard; sequential writes there serialize on the leaf-page lock. Hash-by-bucket doesn't help — the shard owns the whole bucket. Mitigation: not addressed here.
  • Cross-bucket scatter-gather LIST. Admin LIST across all of a customer's buckets fans out to all 32 shards; tail-latency p95 dominated by the slowest shard. Detection: cross-bucket LIST p99; alert at >500 ms. Mitigation: cross-bucket admin LIST is rare; tolerate.

  • Writes spread perfectly uniformly: even a single-bucket-50k/sec ETL parallelizes across all 32 shards.

Iteration 1 — Range-shard by (bucket_id, key) on FoundationDB

Delta vs Iter 0: 32 hash shards merge back into one logical FoundationDB cluster with ~340 internal range shards. The keyspace is globally ordered on (bucket_id, key, version_id); FoundationDB's internal range-shard table maps key-ranges to storage nodes. Strict-serializable cross-key transactions enable atomic CompleteMultipartUpload (swap parts list to final object atomically).

Bottleneck

Iter 0's hash-by-bucket lands all of a bucket's keys on one shard, but a LIST on a 100 M-child prefix within that bucket is still an O(N) leaf walk on one shard's RocksDB — ~20 sec, 100× over the 200 ms SLO. The deeper problem is that hash-by-bucket destroys cross-shard parallelism for in-bucket LIST: even if we wanted to fan out the leaf walk across multiple shards to parallelize, hash-by-bucket put all of a bucket's keys on one shard. Classified as hot-partition on LIST + disk-IO-bound on the prefix scan. The corresponding access pattern is scan-heavy — few LIST queries, big result sets, ordered — for which the canonical fix is range-partitioning + clustering keys, not hash-partitioning.

Iteration 1 fixes the structural mismatch: range-shard the global keyspace by (bucket_id, key) so LIST under a prefix scans contiguous shards (1–2 typically; many shards on huge prefixes) and is O(prefix_size / page_size / shard_count) — much better than hash-by-bucket's O(prefix_size / page_size) on one shard.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#hot-partition: change partition strategy to one whose locality matches the dominant access pattern. The dominant read pattern is GET (point-read; 95% of QPS) which doesn't care about ordering, BUT the dominant SLO-defining read pattern is LIST (5% of QPS but 200 ms SLO on 100 M-child prefixes), which absolutely cares about ordering. The right partition strategy is the one whose locality matches LIST — range-sharding with auto-split. Per bottleneck-to-decisions.html#wrong-storage-engine: swap engine to one whose access pattern matches the dominant query.

Alternatives considered (C4)

  • FoundationDB 7.x — chosen. Range-sharded ordered KV with strict-serializable transactions over arbitrary key sets. Open-source (Apache 2). FDB's "shard" is a contiguous key range owned by a storage server team; the cluster controller auto-splits a shard when it exceeds a byte threshold (~500 MB) or a write-QPS threshold. Cite the FoundationDB SIGMOD 2021 paper for the auto-split + transaction-coordinator design.
  • Bigtable — rejected only on managed-lock-in grounds. Same architectural shape (range-sharded ordered KV with auto-split tablets). The 2006 OSDI paper is the canonical reference for the row-key → tablet → tablet-server design we're using here. Pick Bigtable on GCP-native deployments; FoundationDB on AWS / self-host.
  • Spanner — rejected. Same shape as FDB + TrueTime for global serializable transactions. Spanner is GCP-only and ~3× the per-row cost; we don't need cross-region serializability for this scope (single-region; cross-region in Iter 4 uses regional cells with bounded staleness).
  • CockroachDB — rejected. Range-sharded with auto-splitting (same shape as FDB), excellent SQL compat. Per-row CPU cost is ~2× FDB at this point-read mix because the SQL layer has overhead even on KV-only workloads. Pick CockroachDB if SQL semantics on the metadata are required (e.g., complex joins for billing).
  • Cassandra — rejected. Hash-partitions inside each table by token-range; range scans across partitions (LIST across a bucket) require ALLOW FILTERING or a secondary table indexed differently. Cassandra's WITH CLUSTERING ORDER works within a single partition but doesn't give us cross-partition prefix listing.
  • DynamoDB — rejected. Composite-key DynamoDB with sort-key gives ordered storage within a single partition key, but DynamoDB partition keys cap each partition at ~10 GB and ~3 k WCU; a 100 M-child prefix would blow past that. The Query operation on a sort-key range works but is throttled per partition.

The chosen move

Deploy a single FoundationDB cluster: 200 i4i.4xlarge storage nodes × RF=3, 12 m6i.4xlarge proxy/log nodes for the transaction-coordinator tier, 5 m6i.4xlarge cluster-controller nodes. The keyspace is globally ordered on (bucket_id, key, version_id); FoundationDB's internal "shard map" partitions this keyspace into ~340 contiguous logical shards at startup (one logical shard ≈ one storage-server-team, sized to ~500 GB; 170 TB / 500 GB ≈ 340). Each logical shard is replicated 3× across 3 AZs. Auto-splitting is configured but with conservative thresholds (Iter 2 tightens these); for now, we accept that hot ranges remain intact.

Cite sharding-strategies.html#range; cite the FoundationDB SIGMOD 2021 paper "FoundationDB: A Distributed Unbundled Transactional Key Value Store".

Bottleneck

Iter 1's range-sharding is structurally correct for LIST but creates a hot shard when writes concentrate on a contiguous prefix — the canonical "range-sharding's price" problem. A bucket whose key pattern is date-stamped (logs/<YYYY>/<MM>/<DD>/...) drives 40 k writes/sec into the one shard owning the current-date range; the shard's per-node write ceiling (~30 k/sec on FDB storage with RF=3 quorum + WAL) caps throughput; FDB queues the rest and write p99 spikes from 5 ms to 50+ ms. Classified as hot-partition on the write path.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#hot-partition: subdivide the range; redistribute the sub-shards to underloaded nodes. The canonical fix is "auto-split when a shard is hot, by either bytes or QPS" — the same trick Bigtable, Spanner, and CockroachDB use. The split point is the median key in the hot range (so each sub-shard gets roughly half the load); the data-move job streams half the bytes to a new owner with a two-phase atomic cutover (writes briefly buffer; reads continue from the old owner until cutover signal).

Alternatives considered (C4)

  • Auto-split on per-shard write-QPS — chosen. Cluster-controller monitors per-shard write-QPS over a 60 sec window; on breach, picks the median write-key in the rolling sample and splits. Splits are eventual but bounded (~30 sec from breach to new shard live). This is the Bigtable / Spanner / CockroachDB / FoundationDB consensus design.
  • Pre-emptive sharding by client-side key salting — rejected. Have the client agent prepend a random 2-char hex prefix to every key (logs/2026/05/08/... → {a3}logs/2026/05/08/...); spreads writes across 256 shards immediately. BUT: destroys LIST-prefix entirely — LIST under logs/2026/05/08/ would have to scatter-gather across all 256 salt values and merge-sort the results. Salting is the right move when LIST isn't a load-bearing SLO; here it isn't.
  • Manual range-pinning for known-hot prefixes — rejected. Operator declares "logs/<YYYY>/<MM>/<DD>/" as a hot pattern and pre-creates 4 sub-shards per day. Works for known patterns but doesn't generalize to unforeseen hotspots (a customer's brand-new events/<hour>/ pattern). Auto-split adapts to any access pattern.
  • Replicate hot shard to multiple read nodes — rejected for the write path. More replicas don't help write throughput (writes still serialize through the leader's WAL). Replicas help read tail-latency — addressed via hedged reads in Iter 3.
  • Increase shard density (smaller default shard size) — rejected. Smaller logical shards (e.g., 100 GB instead of 500 GB) improve hot-prefix granularity at startup but explode the shard map (340 → 1700 shards); FDB's locator service caches the shard map and benefits from a moderate cardinality. The right balance is "default logical shards stay 500 GB; hot ones split on demand."

The chosen move

Tighten the FoundationDB cluster-controller's split policy: split a logical shard when (a) bytes > 500 GB (existing default for storage-server-teams), OR (b) sustained write-QPS over a 60-second window > 5 k writes/sec, OR (c) sustained read-QPS > 25 k/sec. The split-point selection: median key in the rolling write-sample for the hot shard (so each half gets ~50% of the load). The data-move job: for shard S = [k_lo, k_hi) being split into S1 = [k_lo, k_mid) and S2 = [k_mid, k_hi), the cluster-controller (1) picks a new owner team for S2, (2) streams S2's bytes to the new owner over the storage network (~40 min for a 250 GB half-shard at 100 MB/s; the bulk of the data-move can run async while the hot key-range is fenced into a smaller slice that moves first in seconds), (3) atomically updates the shard map and signals all FDB clients, (4) old owner drops S2's data after a 5-min grace period. Reads during the move continue from the old owner; writes briefly buffer (~50 ms p99 spike during cutover).

Cite sharding-strategies.html#range; cite Bigtable OSDI 2006 for the canonical auto-split-on-tablet-bytes pattern; cite the FoundationDB SIGMOD 2021 paper for the storage-server-team architecture.

Bottleneck

Iter 2's auto-split solves write hot-shards but doesn't address the fundamental cost of LIST on a 100 M-child prefix: enumerating the children requires touching all 100 M rows even if we only return 1000 per page. The first page's p95 sits at ~120 ms on a warm cache; cold-cache p95 drifts to 250 ms (slowest of 7-shard scatter dominates). Two compounding failure modes: tail-latency on the scatter-gather (slowest shard dominates) and disk-IO-bound on each shard's range scan when the SSTs aren't in cache. The structural fix is to not scan all 100 M rows: maintain a per-prefix immediate-children index that records the keyspans of children directly under (bucket, prefix); LIST hits this index first to learn "you have 100 M children; the first 1000 of them are in keyspan [A…B]"; then a single range scan on [A…B] returns the page.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#scan-heavy: precompute / index the access pattern; turn O(N) scan into O(page_size) index lookup + bounded scan. Per bottleneck-to-decisions.html#tail-latency: hedge / speculative retries on the slowest of N parallel calls. Per bottleneck-to-decisions.html#read-amplification: cache the hot subset; populate via observed access frequency. Three targeted moves, all on the LIST hot path.

Alternatives considered (C4)

  • PrefixIndex co-located in FDB — chosen. Per-prefix entry records the keyspans of immediate children under (bucket, prefix) + the child count. A LIST first reads the index for (bucket, prefix) (1 point-read; ~5 ms), then range-scans the keyspan it returned (bounded to ~1000 keys). The index entry is updated by the PrefixIndex Updater consuming the Kafka change log. Eventual consistency: index lags writes by ~1–5 sec; LIST tolerates this per the consistency NFR.
  • Per-prefix counter only (no keyspans) — rejected. A counter alone tells us "100 M children" but not where the first 1000 live; we'd still scan the full prefix range to find them. Keyspans (10 KB per prefix entry) are the structural payload.
  • Index every key (full inverted index) — rejected. A separate index entry per key is just another copy of the keyspace; no win. The index gains only by being coarser than the keys themselves — one entry per prefix, not per key.
  • Materialize each (bucket, prefix) full LIST result — rejected. Storage explodes: 100 M keys per prefix × many prefixes = orders of magnitude more bytes than the metadata itself. Only feasible for the top-N hot prefixes; we use that strategy in the LIST cache (Redis), not the index.
  • Hedged reads on the scatter-gather slowest shard — chosen. Per Dean & Barroso 2013: at p95-elapsed (~80 ms), issue a duplicate range read to a different RF=3 replica of the same shard; first response wins. Costs ~5% extra reads; drops p99 chunk-fetch ~2×. Same play as #43's restore tail.
  • LIST cache in Redis — chosen. Top-1% of LIST queries by frequency (typically the same 100k prefix-page combinations) get cached for 30 sec; absorbs ~80% of LIST QPS without touching FDB.
  • Range-aware caching on FDB internal page cache — complementary, not a substitute. FDB's page cache helps within a node; Redis fronts the whole cluster and survives node restarts.

The chosen move

Three coordinated changes on the LIST hot path:

  1. PrefixIndex: a co-located FDB keyspace under prefix "px/". For each (bucket_id, prefix_string) that has been LIST'd in the last 30 days, store an entry {child_count: int64, keyspans: [(start_key, end_key, count)], updated_at: u64}. The keyspans list partitions the children into ~100-key buckets so the LIST first-page query can pick the right bucket and read up to max_keys from that bucket. Rebuilt on demand (first LIST under a prefix: do a full scan and populate the index; subsequent LISTs read the index and scan the right keyspan).
  2. Kafka change log + PrefixIndex Updater: every metadata write emits a CDC record keyed by (bucket_id, parent_prefix) via FDB's change-feed plumbing. A 3-node consumer fleet (PrefixIndex Updater) reads the topic and applies updates to the affected PrefixIndex entries: increment/decrement child_count, adjust keyspan boundaries, mark updated_at. Consumer lag is the consistency lag for LIST; we target < 5 sec.
  3. Hedged reads + LIST cache: gateway issues hedged duplicate range-reads when a scatter-gather shard exceeds the p95-elapsed timer. Hot LIST results (top-1% prefix-page combinations) cached in Redis for 30 sec.

What breaks (C5; cite failure-modes.html)

  • PrefixIndex staleness on bursty writes. A flash burst of 100 k writes/sec to one prefix overwhelms the PrefixIndex Updater; consumer lag climbs to 30 sec. LIST returns slightly-old child_count and slightly-old keyspans; some recently-written children appear on later pages or as "missing" from the first-page bucket. Detection: Kafka consumer lag p99; alert at >10 sec. Mitigation: LIST result includes updated_at timestamp from the index; clients that need fresh results can re-issue with x-consistency: strong which falls back to direct FDB scan (slower but fresh).
  • Hedge storm under FDB brownout. If FDB enters partial degradation (shard server slow), every LIST hits the p95 timer; hedge rate jumps from 5% to 60%; FDB read load doubles; brownout deepens. Detection: hedge-rate metric; alert at >25%. Mitigation: clamp hedge rate at 20%; cite circuit-breaker.html.
  • Cache stampede on hot prefix expiration. Top-1% prefix's 30-sec cache TTL expires; 1000 concurrent gateways all miss and hit FDB simultaneously. Detection: per-prefix cache miss spike. Mitigation: jittered TTL (30 +/- 5 sec); single-flight at the gateway (cite single-flight.html): first miss does the FDB read, others wait on a per-prefix promise.
  • PrefixIndex divergence from FDB. If Kafka loses messages or the Updater has a bug, the PrefixIndex's child_count can drift. Detection: nightly audit job samples 1% of prefixes, scans the FDB main keyspace, and compares to PrefixIndex; alert on divergence >1%. Recovery: rebuild affected PrefixIndex entries from the main keyspace.

Bucket-config (which lives in the rare-write Postgres) replicates async cross-region so any cell can resolve bucket_name → home_region.

Bottleneck

Iter 3's single-region cluster meets all in-region SLOs, but a US-West client paying 70 ms RTT to a US-East cell breaks the 10 ms GET SLO immediately. Even with hedged reads inside the cell, the network floor is unavoidable. Classified as cross-region latency / consistency. The fix is structural: regional cells with home-region routing — each bucket's metadata lives in one cell (its home_region), and clients are routed to the bucket's home cell. Cross-region access is the exception (audit, inventory) and runs async.

Architectural decisions to revisit

Per cross-cutting.html#cells: cells make latency, blast-radius, and capacity isolation work in one move. Per bottleneck-to-decisions.html#cross-region: partition by region; route at the edge; replicate config (low-volume) and not data (high-volume) cross-region. The S3 / GCS production design is the same: each bucket has a region, set at creation; LIST and GET on that bucket route to its home region.

Alternatives considered (C4)

  • Regional cells with bucket home_region — chosen. Each bucket has a fixed region (set at creation); object metadata lives only in that cell; cross-region clients pay RTT but in-region traffic dominates. Matches S3 / GCS production behavior.
  • Globally-replicated metadata via Spanner-style synchronous quorum — rejected. Spanner gives strict-serializable global writes via TrueTime + Paxos quorum; works, but synchronous quorum across regions adds 70 ms RTT to every PUT — PUT p99 jumps from 50 ms to 150 ms. Plus the Spanner cost ~3× FDB. Rejected unless the SLO required global strict-serializability (it doesn't).
  • Active-active multi-master with eventual consistency — rejected. Conflict resolution on (bucket, key) creates last-writer-wins anomalies that violate the "strong read-after-write on the same key" NFR. CRDTs would work but add complexity not justified by the per-bucket-pinned-region pattern.
  • Single global cluster with read-locality replicas — rejected. A client could read from the local replica (~5 ms) but writes would still go to the global leader (70 ms). Asymmetric latency is confusing for SDKs; the regional cell model has uniform behavior per bucket.
  • Cross-region metadata replication for DR — deferred to Probe. If the home region is lost, all metadata for that region's buckets is lost. The data plane has 11-nines durability in-region; for the metadata to match, we need cross-region async replication of metadata as a DR copy, but that's a separate scope (~1h RPO; cite replication-decisions.html). The metadata cluster is itself the SoR; a derived async replica is OK for DR. Out of scope this iteration; flagged as Probe A.

The chosen move

Deploy 2 regional cells (US-East-1, US-West-2; growing to 5 in production). Each cell has its own complete stack: API Gateway, FoundationDB cluster (sized for that cell's metadata: ~half the global keyspace each, so ~100 storage nodes per cell), PrefixIndex, LIST cache, Kafka change-log, RocksDB ACL + multipart. Buckets are created with a home_region; metadata writes/reads route to that cell only. Bucket-config (Postgres) has a single master in US-East-1 and async-replicated readonly replicas in every cell so any cell can resolve bucket_name → home_region + bucket settings without cross-region calls. Audit / Inventory rollups happen offline cross-region (Kafka MirrorMaker pulls all cells' CDC streams to a central audit cluster nightly).

Cite cross-cutting.html#cells; cite replication-decisions.html for the active/passive bucket-config + per-cell-leader object metadata pattern.

Degradation contract

When a single FDB shard is unavailable: GET/PUT for keys in that range fail with 503; FDB transparently retries on RF=3 quorum, recovering within ~30 sec on shard rebalance. When the LIST Cache is unavailable: LIST falls back to PrefixIndex + FDB direct (Iter 3 path) at the cost of higher FDB load and ~2× the LIST p95 (still inside SLO, just no margin). When the PrefixIndex is stale (Updater lag): LIST returns slightly-old keyspans; clients can request x-consistency: strong for fresh-but-slow LIST. When a whole cell is down: buckets in that cell are unavailable; cross-region DR replica failover takes ~30 min (RTO).

Real-system comparison

AWS S3's published "How Amazon S3 Works Under the Hood" blog series documents the same architectural arc on custom storage: range-sharded metadata across many "index nodes", per-bucket regional pinning, and a separate data plane. S3's metadata layer is famously not Cassandra (despite frequent guesses); the published descriptions point to a custom distributed KV with auto-splitting that is effectively the FoundationDB / Bigtable shape we've derived.

Meta's Tectonic file system (FAST 2021) is the canonical academic reference for an exabyte-scale metadata service on top of an erasure-coded data plane; their Metadata Store is a sharded ZippyDB (RocksDB-based) cluster with similar range-shard + auto-split mechanics, separated from the storage tier exactly the way we separate FDB metadata from the data plane.

Google's Colossus (the next-gen GFS) and Bigtable papers are the OG: range-shard a sorted keyspace, auto-split on bytes/QPS thresholds, replicate via Paxos quorum. Our design is FoundationDB-shaped because FDB is the closest open-source analog to that internal stack.

Where this design diverges: (a) we use FoundationDB instead of a fully custom KV because at the brief's interview-budget scale FDB's operational maturity beats a from-scratch build; (b) we deliberately reject Spanner-class global serializability because S3's documented per-region read-after-write model is the right SLO; (c) the PrefixIndex is our explicit move — some real systems do this implicitly via the LSM's bloom filters + key-range stats, but for 100 M-child prefixes the explicit index is much cheaper than relying on bloom-skip. Why FoundationDB and not Spanner / CockroachDB / Cassandra? (engine)

  • vs Spanner. Same architectural shape (range-sharded ordered KV with auto-split + serializable transactions). Spanner adds TrueTime-backed global serializability we don't need (S3-compatible per-region model). Spanner is GCP-only and per-row cost is ~3× FDB. We pick Spanner if the workload demands global strict-serializability or we're already on GCP.
  • vs CockroachDB. Same shape; great Postgres-compat SQL surface. CockroachDB's per-row CPU cost is ~2× FDB at this point-read mix because the SQL layer has overhead even on KV-only workloads. Pick CockroachDB if SQL semantics on the metadata are required.
  • vs Cassandra. Cassandra hash-partitions inside each table by token-range; range scans across partitions need ALLOW FILTERING or secondary tables. LIST under logs/2026/05/08/ spanning billions of rows would scatter-gather; LIST p95 fails the SLO structurally. Cassandra fits per-partition ordered workloads (Twitter timelines per user); not bucket-wide ordered scans.
  • vs Bigtable. Identical shape (range-shard, auto-split, ordered KV). The choice is operational: managed on GCP (Bigtable) vs self-host (FDB). On AWS we self-host FDB.

Why range-shard and not hash-shard? (sharding)

  • The LIST SLO is load-bearing. 200 ms p95 on 100 M-child prefixes requires the ability to find contiguous matching rows; hash-sharding spreads each prefix's children across all shards, turning every LIST into a fan-out + merge.
  • Hash-shard's win is uniform write load. We get uniform load via auto-split (Iter 2) instead — hot prefixes are detected and split, redistributing load across nodes while preserving the ordered keyspace.
  • The hybrid alternative (hash for writes, range for reads) doubles the write cost by writing to both layouts and adds an eventually-consistent reconciliation; the FDB auto-split approach is structurally simpler.
  • When hash-shard wins. If there were no LIST-prefix SLO (e.g., a plain object KV like a session store), hash-shard would be the right move; not our case.

What if a single bucket has 10 B objects with no prefix structure? (scale flip)

  • Storage: 10 B × 500 B = 5 TB metadata for one bucket; ~10 k FDB shards; the bucket's keyspace spans the entire FDB cluster.
  • LIST under the bucket root: 10 B children; PrefixIndex helps for the first page (~115 ms cold) but full enumeration is 10 M pages × 115 ms = 13 days. That's expected; clients use prefix narrowing.
  • Auto-split: the bucket spreads naturally across many shards because the keyspace is huge; no special case needed.
  • The catch: a 10 B-object bucket with hot writes (e.g., 100 k/sec) puts 100 k/sec on the date-stamped sub-prefix; auto-split slices into 30+ sub-shards. FDB's shard-map cardinality grows; we might need to bump the locator service's tier.

Strong vs eventual consistency on LIST — what are the trade-offs? (consistency)

  • Strong (per-key) is mandatory: GET-after-PUT on the same (bucket, key) MUST return the new version. FDB gives this via single-key strict-serializable transactions. No SLO trade.
  • Eventual (per-LIST) is acceptable: S3's documented model is "LIST may not show the most recent PUT for several seconds"; we match that. The PrefixIndex Updater lag (target < 5 sec) is the consistency lag.
  • Why not strong LIST? Would require LIST to read at the latest commit-timestamp on every shard it touches, blocking on in-flight writes; LIST p95 doubles. The SLO doesn't require this; we don't pay the cost.
  • Escape hatch: x-consistency: strong request header forces a fresh FDB scan (skips the PrefixIndex and cache); rare; ~5× the latency.

Why a separate PrefixIndex and not just rely on FDB's range scan? (index)

  • FDB's range scan is O(matching rows) regardless of indexing. A LIST on a 100 M-child prefix has 100 M matching rows; the scan must read all of them (even if it returns only the first 1000 sorted) because it doesn't know which rows are children of this prefix vs grandchildren without examining each key.
  • The PrefixIndex pre-bucketizes children into ~100 keyspans, so the LIST query reads one index entry to find "the first 1000 children live in keyspan [A, B)" and then range-scans only that bounded keyspan (~1000 keys instead of 100 M).
  • The cost is ~7 GB of index storage (across 1 M hot prefixes) and the Kafka-driven Updater fleet to keep it fresh. Trivial relative to 170 TB main storage.
  • What if all prefixes are warm? Then the index grows; we cap entries at 30-day staleness (LRU-evict prefixes that haven't been LIST'd in 30 days); cold prefixes do a full FDB scan on the rare LIST that hits them (still inside SLO when the prefix is small).

Where does the bill come from? What's the cost knob? (cost)

  • ~77% is FoundationDB storage: 200 nodes × $986 = $197 k single-region; 169 nodes across both cells = $169k. Storage drives the count (170 TB × RF=3 / 3 TB per node = 171 nodes).
  • Top knob: row-level compression. If we get LSM compression from 3× to 4×, storage shrinks from 170 TB to 125 TB; nodes drop from 171 to 125; ~$45k/mo savings.
  • Next knob: shard density. Smaller default shards (250 MB instead of 500 MB) double the shard count; FDB locator service costs grow but not linearly — small operational hit. Larger default shards (1 GB) reduce shard count but increase per-split data-move cost. 500 MB is the operational knee.
  • Hardest to cut: RF=3 across 3 AZs is structural to the 11-nines durability NFR; can't reduce without losing durability.

How do you migrate 1 T existing rows from Naive to Final? (migration)

  • Phase 1 — stand up FDB alongside Naive: deploy FDB cluster; dual-write all new metadata to both Naive Postgres and FDB; reads continue on Naive. Run for 30 days; assert byte-equality on dual-written rows daily.
  • Phase 2 — backfill: stream all 1 T rows from Naive Postgres to FDB via per-shard parallel copy; ~7 days at 1.6 M rows/sec aggregate.
  • Phase 3 — shadow reads: 1% of GETs and LISTs read from both Naive and FDB; assert equality; ramp to 100% over 7 days.
  • Phase 4 — cutover: reads switch to FDB primary; Naive becomes a fallback for 14 days; deprecate Naive.
  • Phase 5 — PrefixIndex + cache + cells: bring online incrementally per Iter 3 + 4. Cells bring up region-by-region with ~10% bucket migration per week.
  • Total: 90–120 days; storage savings ramp linearly.

S7. Key Takeaways

  • The tree-of-maps kernel is a single-node artifact. 1 T objects don't fit on one node, full stop. The right move is to shard the ordered keyspace, not to scale the primary vertically. The naive Postgres design fails on three independent NFRs (capacity, write throughput, LIST p95); the data physically doesn't fit.
  • Pick sharding by the SLO that defines the system. Hash-shard is uniformly the best write-throughput answer; range-shard is uniformly the best LIST-locality answer. The two are opposite. Range-shard wins here because LIST p95 is the load-bearing SLO; we recover write uniformity via auto-split (which preserves order while redistributing load).
  • Auto-split is the structural fix for hot prefixes under range-sharding. Date-stamped keys, customer-id prefixes, and sequential ETLs all concentrate writes on one contiguous range — the canonical "price of range-sharding." Auto-split on per-shard write-QPS > 5 k slices the hot range and redistributes; the cluster adapts to whatever access pattern the customer brings, not just patterns we anticipated.
  • LIST on 100 M-child prefixes is O(N) without indexing — build the index. Even on a perfectly-sharded ordered keyspace, returning the first 1000 children of a 100 M-child prefix scans 100 M rows unless we precompute the immediate-children keyspans. The PrefixIndex (one entry per prefix; ~7 KB per entry) turns O(N) into O(page_size + log(prefix_count)).
  • Hedged reads are the canonical move on multi-shard scatter-gather. A LIST that fans out to 7 shards has tail-latency dominated by the slowest. Issue a duplicate at p95-elapsed; first response wins. Same play as #43's restore tail; same Dean & Barroso "Tail at Scale" paper.
  • Power-law on LIST QPS justifies a small Redis cache. Top-1% of LIST queries by frequency get ~80% of LIST QPS; a 50 GB Redis with 30-second TTL absorbs them with sub-ms latency. The cache is small because the popular set is small.
  • Strict-serializable transactions belong on writes, not on LIST. S3-compatible read-after-write requires strict per-key consistency on GET-after-PUT (single-key FDB txn covers this); LIST is allowed to be eventually-consistent within seconds. The PrefixIndex can lag the write path by ~250 ms via Kafka without violating the SLO.
  • Regional cells with home-region routing is the default S3-compatible answer. Cross-region synchronous quorum (Spanner-style) breaks the 50 ms PUT SLO via 70 ms RTT; per-bucket regional pinning matches both S3's published consistency model and the latency floor. Cross-region async DR replica handles the 11-nines durability target via a separate code path.
