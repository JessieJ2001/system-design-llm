Scenario #6: Strong vs Eventual Consistency

Three data stores in one company—banking ledger, social media likes, shopping cart—each with radically different consistency needs. Quantify the actual cost of stale reads to make the right tradeoff.

~15 min read

The Scenario

You are the architect for a large company that runs three distinct services. Each service stores data across multiple replicas (for availability and read throughput). The question: what consistency level does each service actually need?

Service Data Scale Read/Write Ratio Cost of a Stale Read
A: Banking Ledger Account balances, transfers 10M transactions/day 60% reads / 40% writes Double-charges, lost money
B: Social Media Likes Like counts on posts 100K likes/sec 90% reads / 10% writes Counter off by 1 for 500ms
C: Shopping Cart Items in user's cart 5M active sessions 50% reads / 50% writes Cart shows wrong items briefly

Key question for each: If a user writes data and then immediately reads it back, do they need to see their own write? And what is the business cost if they do not?

Back-of-Envelope Math: The Cost of Staleness

Service A: Banking Ledger

Scenario: User transfers $500 from Account X to Account Y.

Timeline with eventual consistency:
  T=0ms    User initiates transfer. Write goes to primary replica.
  T=1ms    Primary confirms: Account X = $500 (was $1000)
  T=5ms    User checks balance (reads from a replica that hasn't replicated yet)
  T=5ms    Replica still shows: Account X = $1000  ← STALE
  T=5ms    User thinks: "Transfer didn't go through. Let me try again."
  T=6ms    User initiates SECOND transfer of $500
  T=7ms    Primary processes: Account X = $0 (deducted again!)
  T=100ms  Replication catches up. Both replicas now show $0.

Result: User lost $1000 instead of $500.

Scale of damage:
  10M transactions/day
  Even if only 0.1% cause double-transfers:
    10,000 errors/day × avg $200 per error = $2,000,000/day in errors

  That is $2M per day. Even $730M per year if not caught.

Verdict for banking: Strong consistency is mandatory. The cost of a stale read is financial loss, regulatory violations, and destroyed customer trust. You must pay whatever performance penalty strong consistency costs.

Service B: Social Media Like Counter

Scenario: Post has 10,423 likes. User taps "like."

Timeline with eventual consistency:
  T=0ms    User taps like. Write goes to nearest replica.
  T=1ms    Local replica: count = 10,424
  T=50ms   Another user loads the post. Reads from a different replica.
  T=50ms   That replica still shows: 10,423  ← STALE (by 1)
  T=300ms  Replication catches up. All replicas show 10,424.

Cost of this stale read: $0.00

  - Nobody notices 10,423 vs 10,424
  - The count will be correct within a few hundred milliseconds
  - No business logic depends on the exact count being real-time
  - No user makes a financial decision based on like counts

Scale of the alternative (strong consistency):
  100K likes/sec × consensus round per like
  Each consensus round: ~100-200ms cross-DC latency
  Concurrent consensus operations: 100K × 0.15 = 15,000 simultaneous
  Infrastructure cost: massive distributed lock coordination
  All to prevent... a like count being off by 1 for 300ms.

Verdict for likes: Eventual consistency is the obvious choice. The cost of staleness is zero. The cost of strong consistency is enormous. This is the easiest consistency decision you will ever make.

Service C: Shopping Cart

Scenario: User adds a laptop to their cart, then navigates to checkout.

Timeline with eventual consistency:
  T=0ms    User clicks "Add to cart." Write goes to primary.
  T=1ms    Primary: cart = [laptop]
  T=20ms   User clicks "Checkout." Read goes to a replica.
  T=20ms   Replica still shows: cart = []  ← STALE
  T=20ms   User sees: "Your cart is empty"  ← CONFUSING
  T=300ms  Replication catches up. Replica shows: cart = [laptop]
  T=300ms  If user refreshes, they see the laptop.

Cost analysis:
  - User is confused ("I just added that!")
  - Some users will add the item again → duplicate in cart
  - Some users will abandon checkout → lost revenue
  - Support tickets: "Your site is broken"

  Not catastrophic (no money lost), but annoying.
  Estimated impact: 2-5% cart abandonment increase during stale windows

  At $50M/day GMV, 3% abandonment increase = $1.5M/day lost revenue.
  That is significant, but does NOT require full strong consistency.

Verdict for shopping cart: You need read-your-writes consistency—a middle ground. The user who wrote the data must see their own writes immediately. Other users can see stale data (they don't care what is in someone else's cart). This is weaker than strong consistency but stronger than eventual.

Naive Design: Strong Consistency for Everything

The simplest approach: use synchronous replication across all 3 data centers for every service. Every write waits for all replicas to confirm before returning success.

What Breaks (With Numbers)

Performance Impact

Metric Strong Consistency Eventual Consistency Difference
Write latency 121 ms (cross-DC sync) 1 ms (local write) 121x slower
Read latency 80-120 ms (quorum) 1 ms (local read) 80-120x slower
Write throughput (3-node) ~3K writes/sec (bottlenecked by slowest replica) ~30K writes/sec (each node independent) 10x less throughput
Availability All replicas must be reachable Any one replica suffices Lower availability

Applying This to Our Services

Service B (Likes) with strong consistency:
  100K likes/sec × 121ms write latency
  = 12,100 concurrent writes waiting for cross-DC sync
  Each requires coordination across 3 DCs
  Infrastructure cost: ~$50K/month extra for consensus overhead
  User benefit: like count is never off by 1 for 300ms
  ROI: Spending $600K/year to solve a problem nobody notices

Service A (Banking) with eventual consistency:
  Savings: ~$20K/month in infrastructure
  Risk: $2M/day in double-transfer errors
  ROI: Saving $240K/year while risking $730M/year in errors
  This is obviously insane.

The lesson: Consistency is not free. It costs latency, throughput, and availability. Using the same consistency level for all data is either wasteful (over-consistent for non-critical data) or dangerous (under-consistent for critical data). You must match the consistency level to the business cost of staleness.

Iteration 1: Match Consistency to Business Requirements

Service A (Banking): Linearizable Consistency

Every read returns the result of the most recent write. No stale reads, ever. This is the strongest possible guarantee.

Implementation: PostgreSQL with synchronous_commit = on
  - Writes replicate to at least one standby before confirming
  - All reads go to the primary (or use synchronous read replicas)
  - Trade: higher latency (50-100ms per write), lower throughput

Write path:
  Client → Primary DB → write to WAL → sync to standby → confirm
  Latency: 1ms (local) + 50ms (sync replication) = 51ms

Read path:
  Client → Primary DB → read from local storage
  Latency: 1ms

Throughput: ~5K writes/sec (limited by sync replication)
  10M transactions/day = 115 transactions/sec
  5K capacity vs 115 needed = 43x headroom  ← Plenty

Banking: Code Example

# Banking: read from primary, synchronous replication
import psycopg2

def transfer(from_account, to_account, amount):
    conn = psycopg2.connect(
        host="primary-db.bank.internal",  # Always primary
        options="-c synchronous_commit=on"  # Wait for replica
    )
    cursor = conn.cursor()

    try:
        cursor.execute("BEGIN")
        # Pessimistic lock: prevent concurrent modifications
        cursor.execute(
            "SELECT balance FROM accounts WHERE id = %s FOR UPDATE",
            (from_account,)
        )
        balance = cursor.fetchone()[0]

        if balance < amount:
            cursor.execute("ROLLBACK")
            return {"error": "Insufficient funds"}

        cursor.execute(
            "UPDATE accounts SET balance = balance - %s WHERE id = %s",
            (amount, from_account)
        )
        cursor.execute(
            "UPDATE accounts SET balance = balance + %s WHERE id = %s",
            (amount, to_account)
        )
        cursor.execute("COMMIT")
        return {"success": True}
    except Exception:
        cursor.execute("ROLLBACK")
        raise

Service B (Likes): Eventual Consistency

Writes go to the nearest replica. Replication is asynchronous. Reads may return stale data for a few hundred milliseconds. Nobody cares.

Implementation: Cassandra with consistency level ONE
  - Write to nearest node, replicate asynchronously
  - Read from nearest node (may be stale)
  - Trade: may see stale counts for 100-500ms

Write path:
  Client → nearest Cassandra node → local write → confirm
  Latency: 1ms
  Replication: async to other nodes (100-500ms)

Read path:
  Client → nearest Cassandra node → local read
  Latency: 1ms

Throughput: each node handles 10K writes/sec independently
  3 nodes × 10K = 30K writes/sec
  100K likes/sec → need 10 nodes (with replication factor 3)
  Very achievable with Cassandra.

Likes: Code Example

# Likes: eventual consistency with Cassandra
from cassandra.cluster import Cluster
from cassandra import ConsistencyLevel

cluster = Cluster(["cass-node-1", "cass-node-2", "cass-node-3"])
session = cluster.connect("social_media")

def increment_like(post_id):
    # Counter column in Cassandra - eventually consistent
    stmt = session.prepare(
        "UPDATE post_likes SET count = count + 1 WHERE post_id = ?"
    )
    stmt.consistency_level = ConsistencyLevel.ONE  # Fastest
    session.execute(stmt, [post_id])

def get_like_count(post_id):
    stmt = session.prepare(
        "SELECT count FROM post_likes WHERE post_id = ?"
    )
    stmt.consistency_level = ConsistencyLevel.ONE  # May be stale
    result = session.execute(stmt, [post_id])
    return result.one().count

Service C (Shopping Cart): Read-Your-Writes Consistency

The user always sees their own writes. Other users may see stale data (but they have no reason to read your cart).

Implementation: Session affinity (sticky sessions)
  - Route each user's reads to the same node they wrote to
  - That node always has the latest data for that user
  - No cross-DC synchronization needed for reads

Write path:
  Client → assigned node → local write → confirm
  Latency: 1ms
  Replication: async to other nodes

Read path:
  Client → SAME assigned node → local read (always fresh for this user)
  Latency: 1ms

Fallback: If assigned node is down, read from another node
  with a "consistency token" (a version number from the last write).
  If the replica's version is too old, wait briefly for replication
  or read from primary.

Cart: Code Example

# Shopping cart: read-your-writes via session affinity
import hashlib

def get_cart_node(user_id):
    """Route user to a consistent node for read-your-writes."""
    node_index = int(hashlib.md5(user_id.encode()).hexdigest(), 16) % NUM_NODES
    return nodes[node_index]

def add_to_cart(user_id, item_id):
    node = get_cart_node(user_id)  # Always same node for this user
    node.execute(
        "INSERT INTO cart_items (user_id, item_id, added_at) VALUES (?, ?, NOW())",
        [user_id, item_id]
    )
    # Async replication to other nodes for fault tolerance
    return {"success": True, "consistency_token": node.get_version()}

def get_cart(user_id, consistency_token=None):
    node = get_cart_node(user_id)  # Same node → sees own writes
    if not node.is_healthy():
        # Fallback: read from another node, but check freshness
        fallback = get_fallback_node(user_id)
        if consistency_token and fallback.get_version() < consistency_token:
            # Replica is behind. Wait up to 100ms for replication.
            fallback.wait_for_version(consistency_token, timeout_ms=100)
        return fallback.execute("SELECT * FROM cart_items WHERE user_id = ?", [user_id])
    return node.execute("SELECT * FROM cart_items WHERE user_id = ?", [user_id])

Iteration 2: Understanding the Consistency Spectrum

Consistency is not binary. There is a spectrum of guarantees, each with different costs and properties. Understanding where each level sits helps you make better tradeoffs.

The Consistency Hierarchy

Level Guarantee Example Cost
Linearizable Every read sees the most recent write globally. Operations appear to happen instantaneously. Bank transfers Highest: cross-DC sync on every write
Sequential All clients see operations in the same order, but not necessarily real-time. Distributed log High: total ordering required
Causal If operation A caused operation B, everyone sees A before B. Concurrent operations may be seen in any order. Comment threads (reply always appears after parent) Medium: track causal dependencies
Read-your-writes You always see your own writes. Others may not. Shopping cart Low: session affinity or version tokens
Eventual All replicas converge to the same value eventually. No timing guarantee. Like counters, DNS Lowest: async replication

Concrete Read/Write Timeline

The following timeline shows what different users see under each consistency level. Two users (Alice and Bob) interact with a counter that starts at 0. Alice increments it to 1 at T=10ms.

Timeline:
  T=0ms    Counter = 0 (all replicas)
  T=10ms   Alice writes: Counter = 1 (write goes to Replica A)
  T=15ms   Bob reads from Replica B

What does Bob see?

  Linearizable:        Bob sees 1.  (Write is visible globally before read)
  Sequential:          Bob sees 1.  (Write happens-before read in global order)
  Causal:              Bob sees 0.  (Alice's write is not causally related to Bob's read)
                       UNLESS Bob knew about Alice's write (e.g., Alice sent Bob a link)
  Read-your-writes:    Bob sees 0.  (Bob didn't write it, so no guarantee)
                       Alice reads: sees 1.  (She wrote it, she sees it)
  Eventual:            Bob sees 0.  (Replica B hasn't replicated yet)
                       At T=300ms, Bob reads: sees 1.  (Replication caught up)

PACELC: The Full Tradeoff

The CAP theorem tells you what happens during a network partition. But partitions are rare. PACELC extends the model: even when there is no partition, you still choose between latency and consistency.

PACELC for our three services:

  Banking (PA/EC):
    During Partition:   Choose Consistency over Availability
                        (Refuse writes rather than risk inconsistency)
    Else (no partition): Choose Consistency over Latency
                        (Pay 50-100ms for sync replication)

  Likes (PA/EL):
    During Partition:   Choose Availability over Consistency
                        (Accept likes even if replicas disagree)
    Else (no partition): Choose Latency over Consistency
                        (1ms local writes, replicate later)

  Cart (PA/EL with read-your-writes):
    During Partition:   Choose Availability over Consistency
                        (Cart still works, might be stale for other users)
    Else (no partition): Choose Latency over Consistency
                        (1ms local writes, session affinity for own reads)

Final Design

Cost Comparison

Approach Monthly Infrastructure Cost Monthly Risk Cost Total
Strong consistency for everything $150K (consensus overhead, cross-DC sync for all) $0 (no stale reads) $150K/mo
Eventual consistency for everything $30K (simple async replication) $60M+ (banking errors at $2M/day) $60M+/mo
Right consistency per service $50K (strong only where needed) $0 (banking is consistent, others acceptable) $50K/mo

The right answer is almost never "one consistency level for everything." It is always "the cheapest consistency level that satisfies the business requirement for each piece of data."

Key Takeaways

  1. Consistency is not free. Strong consistency costs latency (121x slower writes) and throughput (10x fewer writes/sec). Only pay for it when the business requires it.
  2. Quantify the cost of a stale read. For banking: $2M/day in errors. For likes: $0. For cart: moderate user frustration. The numbers make the decision obvious.
  3. Most data does not need strong consistency. In a typical application, only 5-10% of data (financial transactions, inventory counts, user credentials) truly needs linearizability. The other 90-95% is fine with eventual or read-your-writes consistency.
  4. Read-your-writes is often the sweet spot. Users expect to see their own actions reflected immediately. They rarely notice if other users' data is slightly stale. Session affinity or version tokens give you this guarantee cheaply.
  5. The consistency spectrum matters. Linearizable > Sequential > Causal > Read-your-writes > Eventual. Each step down the spectrum is cheaper but provides weaker guarantees. Pick the weakest level that satisfies your requirements.
  6. PACELC extends CAP. Even without network partitions, you still trade off latency vs consistency. This is the tradeoff you face every day, not just during rare partition events.
