Designing Content-Defined Dedup for Backup Storage

100 PB stored, 5 PB ingested per month, 10k uploads/sec peak; 50% intra-customer dedup target; restore p95 ≤ 30 s for ≤10 GB; cross-customer dedup forbidden by policy. We start from a whole-file SHA-256 naive that the LC #13 kernel fits perfectly — and that collapses to ~0% dedup on a single-byte edit — and iterate to FastCDC content-defined chunking with a sharded chunk index, per-user Bloom miss-skip, hedged parallel-fetch restore, a hot-chunk read cache, and Glacier-IA tiering for cold chunks.

NALSD Walkthrough • ~50 min read • LC #13 (Find Duplicate File in System)

LC kernel. Find Duplicate File in System (LC #13). Textbook structure: hash each file's content (e.g., SHA-256) and group by hash. Insert is O(file_size); duplicate detection is O(1) hash lookup. Three properties that break at scale: (1) the hash is over the whole file, so a single byte changed in a 10 GB file invalidates the entire hash — backup workloads (where users edit small parts of large files daily) collapse the dedup ratio to ~0% on edited files. (2) The hash table is single-machine; at 100 PB / 4 MiB chunks ≈ 24 billion chunks × 64 B/entry ≈ 1.5 TB of index — doesn't fit one node, and uniform-hash sharding loses locality. (3) Restore reassembles a deduped file from many small chunks scattered across the chunk store; the slowest of N parallel S3 GETs dominates latency by tail. The C8 production cousin replaces whole-file hashing with content-defined chunking (FastCDC) so edits only invalidate the chunk that contains them; replaces the single hash table with a sharded chunk index + per-user Bloom filter so the common case (chunk already exists for this customer) skips the index lookup; and uses hedged parallel-fetch restore + hot-chunk caching to tame the long-tail recombination.

S0. Scenario & Requirements

A cloud backup product (Backblaze / Dropbox / Carbonite tier): customers run an agent that snapshots their disks daily (or on file change) and uploads to our storage. The product detects and stores duplicate file content exactly once within each customer's account, but cross-customer dedup is forbidden by policy (privacy: dedup-as-a-side-channel leaks file existence across tenants — cite Harnik/Pinkas/Shulman-Peleg 2010). Restores must complete the body of a typical ≤10 GB folder in 30 s p95.

System class

Class: storage-shaped — "persist X durably and surface Y view of it later" capabilities (cite functional-requirements.html#class-storage); the FRs are operations on a content store + their durability and integrity guarantees. The class fixes the NFR additions: durability (loss is unacceptable), capacity-density (cost-per-byte dominates), and reconstruction latency (not just write throughput).

Functional requirements (cite functional-requirements.html)

  • Upload a file. A customer agent uploads (customer_id, file_path, file_bytes, mtime); the platform durably persists the bytes (deduped within the customer) and assigns a file_version_id.
  • Restore a file (or folder). Given (customer_id, file_path, version_id), the platform returns the file's full bytes within the restore SLO.
  • List file versions. Given a customer's file path, return the chronological list of versions and their sizes / mtimes (browse-restore-points UI).
  • Delete a customer / file. When a customer cancels (or deletes a file), the platform must reclaim the storage that's only referenced by them — chunks shared with other files of the same customer remain.
  • Verify integrity. Periodic background scrub: read a chunk, hash it, compare to its stored hash; alert on mismatch (bit rot detection).

Out of scope (named, not silently dropped): end-to-end encryption with customer-held keys (we do server-side encryption with our keys; client-keyed E2E is Probe A); cross-customer dedup (forbidden by policy — the privacy paper above documents the side-channel that side-effect creates); incremental file-system delta protocols like rsync (the agent uploads whole files; CDC handles the dedup); compliance-grade WORM / immutable-buckets (separate enterprise SKU; Probe B); media-specific dedup like image-perceptual-hashing (we only do byte-exact CDC; near-duplicates aren't merged).

Flagged assumptions. (1) Average file size is ~2 MB — consistent with consumer backup mixes (many KB-sized OS files, MB-sized docs/photos, occasional GB videos; the bytes are dominated by the few large files but the count is dominated by small files). (2) Edit pattern is "1-byte changed in a large file" (the canonical worst case for whole-file hashing). (3) 50% intra-customer dedup is achievable with content-defined chunking; the canonical sources of dedup are: re-uploads of unchanged files (most common), edited-then-saved files (CDC catches the unchanged regions), shared OS / app binaries within one user's machines (Time-Machine-style cross-machine dedup is in scope; cross-customer is not).

Sister scenarios (see plan)

  • Photo-sharing dedup (planned) — cross-user dedup permitted; perceptual-hashing instead of byte-exact CDC. Drastic flip: the privacy policy that forbids dedup here is the load-bearing constraint; flipping it would unlock another 5× storage savings (most photos in cloud have many uploaders).
  • Container-image registry (planned; e.g., ECR / Docker Hub) — layer-level dedup; layers are fixed boundaries (defined by Dockerfile commands); CDC isn't needed. Drastic flip: chunk boundaries are pre-known; the index doesn't need a Bloom filter (every layer is named).
  • Distributed file system (planned; HDFS/Ceph-style) — block-level storage with fixed-size blocks; dedup is opt-in and rarely on. Drastic flip: read-write workload (not write-once); CDC's "edit invalidates one chunk" win disappears because writes happen at the block level by design.
  • #35 Web crawler — the dedup sister where dedup is on URLs (Bloom filter for "have we crawled this?"); shares the per-prefix sharding decision but the dedup fingerprint is a URL string, not a content hash.

The fundamental tension. Whole-file hashing is exact and per-file; backup is "many files, mostly unchanged across uploads, occasionally edited in place." Three constraint-pairs collide: (a) edit-amplification vs hash-granularity — a 1-byte edit invalidates a whole-file hash, but per-byte hashes are infeasible (storage explodes); CDC picks a chunk size that tracks edit locality. (b) dedup ratio vs cross-customer privacy — global dedup would 5× the savings, but leaks file existence as a side-channel; we trade savings for privacy. (c) reconstruction latency vs storage cost — small chunks favor dedup ratio + edit-locality but explode the chunk count and the parallel-fetch tail; the chunk-size choice is a Pareto curve we tune via FastCDC's stochastic Gear-hash boundary detection. The arc resolves the tension: chunk by content (Iter 0), shard the index + Bloom-skip the lookup (Iter 1), hedge the parallel restore (Iter 2), cache the hot chunks (Iter 3), tier the cold chunks (Iter 4).

ID scheme (cite identifier-schemes.html#hash-id): chunk_id = BLAKE3-256(chunk_bytes). Why a content hash and not a Snowflake: the same bytes uploaded twice must produce the same id without coordination (otherwise we'd store duplicates). BLAKE3 over SHA-256 because (a) BLAKE3 is ~2.5× faster per core (1 GB/s vs 0.4 GB/s) which dominates the per-sample CPU cost; (b) BLAKE3's parallelizable Merkle-tree mode lets a single large chunk be hashed across multiple cores (irrelevant for 4 MiB chunks, useful for small-file fast-path); (c) BLAKE3 has the same 256-bit security as SHA-256 against second-preimage attacks. Collisions: 2128-birthday on 256-bit; at 1013 chunks the collision probability is < 10-50 — effectively zero (we still re-verify on chunk insert by reading the existing chunk and comparing bytes if a hash matches; cite the rare-but-paranoid path).

The partition-key heuristic: the dominant access pattern is "chunk-exists check by hash" (every uploaded chunk does one) and "fetch chunk bytes by hash" (every restore does N). Both are point-read on chunk_hash. Hash by the first byte (8 bits → 256 shards) of chunk_hash for the chunk index so the index is uniformly spread; range partition the chunk-store S3 layout by chunk_hash[0..2]/chunk_hash[2..4]/... prefix so listing operations are bounded and lifecycle rules are per-prefix.

S2. API contracts (cite api-design-patterns.html)

Five endpoints — upload is gRPC for streaming-throughput; restore is REST so the agent can tee directly into a file. Manifest assembly and chunk-existence checks are internal RPCs the agent calls before sending bytes (the "tell me which chunks I need to upload" pre-flight saves bandwidth on already-stored chunks).

Pre-flight (gRPC; agent → server)

// Agent computes chunk hashes locally (BLAKE3 on the client) and asks
// the server which it needs to upload. Saves ingress bandwidth on
// already-stored chunks (the dominant 50% in steady-state).
rpc CheckChunks(ChunkCheckBatch) returns (ChunkCheckResp);

message ChunkCheckBatch {
  string customer_id  = 1;
  repeated bytes chunk_hashes = 2;   // up to 1000 per request
}
message ChunkCheckResp {
  repeated bool exists_for_customer = 1;   // 1:1 with input order
  bytes  upload_token = 2;                  // signed JWT for next-step PutChunk
}

// Per-customer Bloom filter check on the server side: a "false" answer
// is authoritative (chunk doesn't exist for this customer); a "true"
// answer triggers the chunk-index point-read for confirmation.
// Cross-customer dedup is forbidden, so the per-customer scope is
// the security boundary as well as the optimization boundary.

Upload (gRPC; streaming)

rpc PutChunk(stream ChunkUpload) returns (PutChunkResp);

message ChunkUpload {
  bytes upload_token = 1;        // from CheckChunks; 5-min TTL

  bytes chunk_bytes  = 3;        // up to 16 MiB

Per-endpoint SLO

  • CheckChunks — per-batch p95 ≤ 50 ms (Bloom check + index point-read on misses); the dominant client-perceived latency.
  • PutChunk — per-chunk ack p95 ≤ 200 ms (S3 PUT for new chunks; immediate ack for dedup hits).
  • CommitFile — p95 ≤ 100 ms (Cassandra write to FileCatalog).
  • GET /v1/files/.../bytes — p95 ≤ 30 s for ≤10 GB (the load-bearing restore SLO; the entire iteration arc tunes for this).
  • GET /v1/files/.../versions — p95 ≤ 100 ms (incidental read; Cassandra partition scan).

CDC / Bloom / restore / fleet / cost formulas

// FastCDC boundary: rolling Gear-hash; declare a boundary at byte i
// when (gear_hash(window_at_i) & MASK) == 0. MASK is sized so the
// expected chunk length matches chunk_size_bytes_target.
// Cite Xia/Jiang "FastCDC: a Fast and Efficient Content-Defined
// Chunking Approach for Data Deduplication" (USENIX ATC 2016).
// Property: a 1-byte edit invalidates ~1 chunk on average (the
// chunk that contains the edit), not the whole file.

// Bloom filter sizing (per-customer):
//   chunks_per_customer (median, 10 PB / 10M cust × dedup) ≈ 250
//   chunks_per_customer (p99) ~ 100,000  // top-1% backup-heavy customers
//   target false-positive rate p = 0.01 (1% FP at p99-customer size)
//   bits_per_entry m/n = -ln(p)/ln(2)^2 = 9.6
//   for n=100k: m = 9.6 × 100k = 0.96 Mbits ≈ 120 KB
//                (rounded up to 150 KB capacity for headroom + power-of-2 alignment)
//   k_hashes = ln(2) × m/n = 6.6 → use k=7

// Restore time (parallel-fetch model):
chunks_per_restore       = restore_max_GB × 1e9 / chunk_size_bytes_target
                         = 10 × 1e9 / 4.19e6 ≈ 2,386 chunks
parallel_degree D        = restore worker fetches D chunks concurrently
per_chunk_fetch_ms       = chunk_size_bytes_target / S3_per_stream_MBps
                         = 4 MB / 100 MB/s ≈ 42 ms
sequential_restore_sec   = chunks_per_restore × per_chunk_fetch_ms / 1000
                         = 2386 × 0.042 ≈ 100 sec  (no parallelism)
parallel_restore_sec     = ceil(chunks_per_restore / D) × per_chunk_fetch_ms / 1000
// At D=64: 2386/64 = 38 batches × 42 ms = 1.6 sec ideal-tail
// But the slowest of N parallel calls dominates (tail-latency).
// p95 of any-of-N at p99=150ms: P(any > 150ms) = 1 - (1 - 0.01)^N
// For N=2386: ~100% of restores see at least one straggler chunk.
// Hedged reads (Iter 2) cap this: at p95-latency wait, send a
// duplicate request to a second S3 prefix; whichever returns
// first wins. Drops p99 chunk-fetch from 150ms to ~80ms.

// Fleet sizing:
ingest_nodes_hash_bound   = ingest_GBps_peak / (BLAKE3_per_core × cores_per_node × util)
                          = 7.7 / (1 × 16 × 0.5) ≈ 1 (tiny; CPU is not the bottleneck)
                          // dominated instead by NIC: 7.7 GB/s / 1.5 GB/s/node = 6 nodes
ingest_nodes_NIC_bound    = ceil(ingest_GBps_peak / 1.5 GB/s) = 6 nodes (peak)
                          + headroom 30% = 8 nodes

chunk_index_nodes         = ceil(chunk_index_bytes_with_overhead / per_node_NVMe_capacity)

S4. Naive Architecture — the simplest thing that meets the FRs

Each upload computes SHA-256 of the whole file; the file is stored as a single S3 object keyed by that hash. The Cassandra files table has one row per file_version with a whole_file_sha256 column; on commit, we look up whole_file_sha256 against an existing row from the same customer — if it exists, we don't upload bytes (just refcount). Restore is one S3 GET that streams the whole object back. This is LC #13's reference solution lifted directly: hash each file, group by hash, dedup by hash equality.

Bottleneck

Naive's whole-file SHA-256 collapses dedup ratio to ~5% on edited files; the storage line balloons to $2.3 M/mo. Classified as wrong-data-structure (the dedup unit is too coarse for the edit pattern). The corresponding access pattern is write-once-read-many with localized writes — a 1-byte edit modifies a single byte range; the data structure that respects this locality is content-defined chunking, where boundary positions are determined by the byte content (not by absolute offsets). Iteration 0 fixes the dedup-ratio failure: shrink the dedup unit from "whole file" to "content-defined chunk" so edits invalidate one chunk, not the file.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#wrong-data-structure: change the data structure to one whose locality matches the access / mutation pattern. The structure here is the dedup unit. CDC (FastCDC, Rabin–Karp) is the canonical "boundaries determined by content, not offset" trick that turns "1-byte edit invalidates the whole file" into "1-byte edit invalidates one chunk." Per bottleneck-to-decisions.html#storage-density: reduce redundancy via dedup at the right granularity.

Alternatives considered (C4)

  • FastCDC (Gear-hash rolling boundary) — chosen. Per Xia/Jiang USENIX ATC 2016: a rolling Gear-hash window scans the file byte-by-byte; declare a chunk boundary when (gear_hash & MASK) == 0. MASK chosen so expected boundary every 4 MiB. ~3× faster than Rabin–Karp at the same cut-distribution quality. Property that wins: a 1-byte edit shifts the rolling window by 1 byte, but boundaries are self-resynchronizing — within a chunk-size of the edit, boundaries align with the previous version's boundaries. Net: 1 chunk invalidated, ~99% of file shares chunks with prior version.
  • Rabin–Karp rolling polynomial — rejected. Same boundary property (self-resynchronizing) but ~3× slower per byte; the per-byte cost matters at 7.7 GB/s peak. Cite the FastCDC paper's published comparison.
  • Fixed-size 4 MiB chunks — rejected. Simplest possible chunking; loses on the canonical "insert 1 byte at offset 0" case. After the insert, every chunk's boundary shifts by 1 byte; no chunk hashes match the prior version; dedup collapses to 0% on prepended/inserted edits. CDC's content-defined boundaries don't have this problem.
  • Variable-size chunks at file-format-aware boundaries — rejected. A .docx-aware chunker would cut at zip-entry boundaries; a .vmdk-aware chunker would cut at VM-block boundaries. Wins on a per-format basis but loses on heterogeneity (every format needs a custom chunker; we have thousands of formats); CDC is format-agnostic and almost as good.
  • BLAKE3 over SHA-256 — chosen. 2.5× faster (~1 GB/s/core vs 0.4 GB/s/core); same 256-bit security; parallelizable Merkle-tree mode (irrelevant at 4 MiB chunk size, useful for larger blocks). Cite official BLAKE3 benchmarks.

The chosen move

For each upload: (a) the agent runs FastCDC locally over the file's bytes, computing chunk boundaries via rolling Gear-hash and per-chunk BLAKE3-256 hashes; (b) sends a CheckChunks(customer_id, chunk_hashes[]) RPC; (c) the server queries the chunk index for each hash, returning exists_for_customer[]; (d) the agent uploads only the chunks marked missing; (e) the agent calls CommitFile with the ordered list of chunk_hashes (the manifest), which writes a manifest row + a file_version row in Cassandra. For each restore: load the manifest by manifest_id, parallel-GET each chunk by hash from S3, stream them back in manifest order.

Cite sharding-strategies.html#hash: hash partition by chunk_hash for the chunk index (uniform load; the dominant access is per-chunk point-read). Cite storage-engines.html#object-store for S3 as the chunk store.

Bottleneck

Iter 0's chunk index lives in Cassandra co-located with the FileCatalog. Cassandra's strengths are ordered range-scans within a partition (e.g., "list this customer's file versions in time order") — not per-row point-reads on a 32-byte hash. The per-CheckChunks call traverses Cassandra's coordinator + token-aware routing + read-repair, adding 1–2 ms per chunk. At peak ingest of 7.7 GB/s ≈ 1,837 chunks/sec + restore of 333 MB/s × ~3 active restore streams = 80,000+ chunks/sec on the index, a single Cassandra cluster's coordinator becomes the long pole. Coordinator-bottleneck compounded with wrong-storage-engine for the access pattern (point-read on hash with 90% hit rate is RocksDB's sweet spot, not Cassandra's). Secondarily: every CheckChunks call hits the index even for chunks the customer has already uploaded — 50% of chunks are dedup hits; a Bloom filter would skip those lookups entirely.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#wrong-storage-engine: swap engine to one whose access pattern matches the dominant query. Per bottleneck-to-decisions.html#point-lookup-overhead: front a hot lookup with a Bloom / cache that absolves the lookup on the common path. The combined move: dedicated point-read engine (RocksDB) + Bloom miss-skip for the dedup-hit common case.

Alternatives considered (C4)

  • RocksDB sharded by chunk_hash prefix + per-customer Bloom — chosen. RocksDB is built for KV point-reads on a binary key; the LSM has built-in Bloom filters per SST that skip 99% of disk seeks for nonexistent keys. Embedded (in-process) avoids the network coordinator hop. Per-customer Bloom (separate from the SST-level Bloom) skips even the in-process check for the dedup-hit case — ~50% of chunks. Cite storage-engines#kv-engine + sketches#bloom.
  • Cassandra (Iter 0) — rejected for the chunk index. Cassandra's coordinator hop adds ms-class latency on every CheckChunks; the read path is also commit-log + memtable + SSTable triple-check (write-path optimized, not read-path). LSM compaction is the same in RocksDB but without the coordinator overhead. We keep Cassandra for the FileCatalog (where range-scan-within-partition is the dominant access).
  • FoundationDB — rejected. Transactional KV; transactions are unused (chunk insert is naturally idempotent on hash). The transaction-coordinator overhead is pure cost.
  • DynamoDB — rejected. Hosted, fits the access pattern; at 12 billion items the on-demand cost exceeds self-hosted RocksDB by ~3× once you include sustained write-units.
  • Cuckoo filter instead of Bloom — rejected. Cuckoo supports deletion (Bloom doesn't). We do need delete (when a customer's last reference to a chunk is removed, we remove from the Bloom too) — but the per-customer Bloom is rotated every N days (rebuilt from chunk index + customer_chunks) so deletion at element-level is unnecessary; we delete by rebuilding the whole Bloom. Cuckoo's space overhead per insertion (~10 bits/key vs Bloom's 9.6) doesn't pay back the complexity.

The chosen move

Three tightly-related techniques addressing the same coordinator + lookup-overhead bottleneck:

  1. Move chunk index to RocksDB. 256 logical shards keyed by chunk_hash[0..7] (first byte). Each shard is a RocksDB column-family on an i4i.4xlarge NVMe node. 8 physical nodes × RF=3 = 24 nodes; each node hosts ~32 shards. RocksDB SST-level Bloom filters skip ~99% of misses without a disk seek. Point-read p99 ≈ 1 ms (in-process; no coordinator hop).
  2. Per-customer Bloom filter for miss-skip. One Bloom per (customer, day) holding the hashes of chunks this customer has already uploaded. Sized at p=1% FP for 100k chunks (the p99 customer): m=1.2 Mbits = 150 KB; k=7 hashes. Stored on the same RocksDB shard that owns the customer's chunks (sharded by customer_id[0..7] for the Bloom side). On CheckChunks: Bloom miss (50% of cases for active backups) → skip the RocksDB point-read; Bloom hit (50% + 1% FP) → do the point-read for confirmation. Bloom is rebuilt nightly from the customer_chunks table (false-positives cap is preserved).
  3. Cross-customer dedup is forbidden — the per-customer Bloom is the security boundary. The chunk index can know that some customer has already uploaded a chunk with hash H; but the per-customer Bloom enforces that customer X is told "you have already uploaded H" only if customer X uploaded it. A side-channel attack (Harnik et al. 2010) where an attacker uploads a chunk with hash H and observes a fast "exists" response would leak that some other customer has H — the per-customer Bloom returns "not for you" so the customer-visible response is identical regardless of cross-customer state.

System after this iteration

Delta vs Iter 1: Restore worker tier introduced as a dedicated fleet (was bundled into ingest workers). Hedged-read logic added to chunk-fetch path: send a duplicate request to the same S3 prefix when the primary exceeds the p95-latency threshold. Wins ~10x reduction in p99 chunk-fetch latency for ~5% extra S3 GET cost. Cite Dean & Barroso "The Tail at Scale" CACM 2013 — the canonical hedged-reads paper.

Bottleneck

Iter 1's restore p95 sits at ~5 s because parallel-fetch of N=2386 chunks is dominated by tail-latency: with per-chunk p99 ~150 ms on S3 GET, the probability that at least one chunk lands in the tail is 1 - (1 - 0.01)^2386 ≈ 100%. Every restore sees a straggler; the 200 ms+ chunk defines the batch-completion latency. Tail-latency bottleneck on the canonical "slowest of N parallel calls" pattern (Dean & Barroso 2013). The single straggler doesn't represent a deep S3 problem — it's GC, network reroute, or noisy-neighbor on a single S3 partition; the S3 service itself is healthy.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#tail-latency: hedge / speculative retries; backup-on-timeout; replicate the hot path so any of N answers wins. Cite bottlenecks.html#tail-latency: hedged reads cost ≈1/N extra calls and reduce p99 by an order of magnitude when the tail is from independent stragglers (not systemic). S3 stragglers are independent (GC events, network reroutes), so hedging applies cleanly.

Alternatives considered (C4)

  • Hedged reads at p95-elapsed timer — chosen. Issue the primary GET; start a timer at the p95 of recent S3 GETs (~80 ms). If the timer fires before the response, issue a duplicate GET. First response wins; sender cancels the loser via TCP RST + S3 client-side discard. Cost: ~5% extra GETs (the fraction that hit the tail). Win: p99 chunk-fetch drops from 150 ms to ~80 ms. Cite Dean & Barroso "The Tail at Scale" CACM 2013.
  • Read from N replicas; first wins (full-fan-out) — rejected. S3 doesn't expose per-replica reads; the GET hits whichever S3 partition owns the key. To get N copies we'd need to write each chunk to N keys (e.g., chunk_hash || copy=0..2), tripling our PUT cost and storage cost. Hedged reads at p95-timer give 90%+ of the win for 5% of the cost.
  • Increase D (parallelism degree) — rejected as a stand-alone fix. D=128 instead of 64 halves the in-ideal-tail batch count, but the slowest chunk's tail-latency is unchanged. The marginal gain is small relative to the doubling of in-flight requests (which hits per-restore-worker connection limits). Hedged reads attack the tail directly.
  • Pre-fetch / range-prefetch warmup — rejected. Useful when chunk-access patterns are predictable (e.g., streaming a video sequentially); the restore is a one-shot, fan-out-then-finish workload. Pre-fetching adds latency to the first byte without reducing whole-restore latency.
  • Use S3 Express One Zone (faster but per-AZ) — rejected. Express One Zone has lower per-GET latency (single-digit-ms) but limited to a single AZ; would force per-restore-stream cell affinity (which we don't have for reads since restores can come from anywhere). Reconsider if AWS adds multi-AZ Express; out of scope today.

The chosen move

Each Restore Worker maintains a rolling p95 of recent S3 GET latency (one estimator per S3 prefix bucket; bucket = chunk_hash[0..2], 256 buckets). For each chunk fetch in the restore stream: (a) issue primary GET on bucket chunk_hash[0..2]; (b) start a timer at p95_estimator + 10ms safety margin (typically ~80 ms); (c) on timer: issue hedge GET on the same key; (d) on first response: cancel the other (HTTP client abort + TCP RST) and consume the body. Hedge GETs add ~5% extra GET volume at the trigger threshold — cite the Dean & Barroso math: hedging at p95-elapsed adds 1/p95-tail-rate ≈ 5% extra calls.

Cite single-flight.html for the canonical hedged/coalesced-fetch family of moves; cite bottlenecks.html#tail-latency for the pattern; cite Dean & Barroso CACM 2013 for the empirical tail-reduction characterization.

// In-memory state on each Restore Worker:
struct PrefixLatencyEstimator {
  prefix_bucket: u8,            // chunk_hash[0..2] mapped to 256-bucket id
  p95_window_ms: SlidingWindow<1024 samples>,  // last 1024 GETs to this prefix
  hedge_rate:    AtomicF32,     // current % of GETs that triggered hedge
}

// Hedge decision (per chunk GET):
fn fetch_chunk(chunk_hash: [u8; 32]) -> Bytes {
  let prefix = chunk_hash[0];
  let p95 = estimator[prefix].p95_window_ms.percentile(0.95);
  let hedge_at_ms = p95 + 10;   // 10 ms safety margin

  let primary  = spawn(s3_get(chunk_hash));
  let hedge_fn = || spawn(s3_get(chunk_hash));

  match select(primary.timeout(hedge_at_ms)) {
    Ok(body) => body,
    Timeout  => {
      let hedge = hedge_fn();
      // first to complete wins; cancel the other
      first_of(primary, hedge)
    }
  }
}

Bottleneck

Iter 2's restore is fast (~3 s p95) but every chunk fetch costs an S3 GET (~$0.0004 per 1000). Restore-driven GETs are dominated by a small set of popular chunks: every Windows machine has the same kernel32.dll bytes; every Mac has the same Foundation.framework chunks; every Photoshop install has identical installer chunks. Per-shard QPS distribution is heavily Zipfian: top-0.001% of chunks (~120k) account for ~80% of restore-side GET volume. Hitting S3 on every popular-chunk fetch wastes GET budget AND inflates the tail-latency variance because hot chunks share S3 partition slots. Power-law access pattern meets read-amplification on the same dataset; both dictate "front the hot tier with a memory cache."

Architectural decisions to revisit

Per bottleneck-to-decisions.html#read-amplification: cache the hot subset; tier hot/cold; populate via observed access frequency. Per cross-cutting#cells: cells make cache locality work (per-cell cache is hot for cell-local restores). The popularity skew is the canonical "top-1% serves 80% of traffic" Zipfian distribution; an LFU cache with bounded RAM exactly fits.

Alternatives considered (C4)

  • Redis 7 OSS Cluster + LFU eviction — chosen. ~250 GB cache holds ~60k of the most popular 4 MiB chunks (top-0.0005%); LFU keeps popularity-driven chunks hot vs LRU which would evict yesterday's-popular for today's-burst. Sub-ms hit latency vs S3's 50 ms p50. Cite storage-engines#kv-engine.
  • Memcached — rejected. Eviction policy is LRU only; LFU is the right policy for power-law-popular content where today's popular is correlated with yesterday's. We'd lose ~10% hit rate.
  • S3 Express One Zone instead of Redis — rejected. Express OZ has single-digit-ms latency but is per-AZ; the cache layer in front gives multi-AZ access from any restore worker. Express OZ is also more expensive per-GB stored than Standard (paid for low-latency, not for cache use case).
  • EFS / FSx / EBS-backed cache — rejected. Block-storage caches add 10× the latency and don't speed up the bottleneck (which is per-key get-by-hash, not bulk-throughput).
  • Cache larger fraction (1 TB instead of 250 GB) — rejected by cost analysis. Hit rate vs cache-size curve flattens after the top-0.001%: 250 GB → 80% hit rate; 1 TB → 85% hit rate; 10 TB → 90% hit rate. Diminishing returns; 250 GB is the knee.
  • Per-customer cache instead of global — rejected. Cache content is keyed on chunk_hash (not customer); the popular-chunks set is uniform across customers (everyone has the same OS DLLs). A per-customer cache would lose the cross-customer hit rate, even though cross-customer dedup is forbidden — reading a hot chunk from cache doesn't violate the privacy policy because the chunk has already been requested by an authorized customer.

The chosen move

A Redis 7 OSS Cluster with maxmemory-policy allkeys-lfu sits in front of S3 on the restore-read path. Restore Workers run a wrapper that does MGET on the cache first; cache hits return immediately (~0.5 ms); cache misses fall through to the existing chunk-index + S3 GET path (with hedging from Iter 2) and write-through-populate the cache. Eviction is LFU at 250 GB total cluster size (3 cache.r6g.4xlarge nodes × ~80 GB usable each).

An offline Cache Warmer job runs nightly: scans the prior day's S3 GET access logs, computes the top-N chunks by access count, and pre-loads them via cache SET. This cold-start protection ensures the cache is hot at start-of-day even after a Redis cluster restart (Redis RF=1; not durable; rebuild from S3).

RF=1 because S3 is the SoR for chunk bytes; if the cache cluster fails, restores fall back to direct S3 fetch (the Iter 2 path) at the cost of higher GET volume + slightly higher tail. Cite access-patterns#power-law; cite storage-engines#kv-engine.

What breaks (C5; cite failure-modes.html)

  • Cache miss storm on cluster restart. Redis cluster reboots; cache is empty; 100% of restore reads fall through to S3. Detection: cache hit rate drops to ~0%. Recovery: Cache Warmer kicks off automatically on cluster-up; restores hot working set from prior-day's access log within ~5 minutes. During those 5 minutes restore p95 reverts to Iter 2's ~3 s — still inside SLO.
  • Cross-customer cache poisoning. The cache is keyed on chunk_hash (not customer); a malicious actor uploads a chunk with hash collision (theoretical at 256-bit; practically impossible) and pollutes another customer's restore. Mitigation: integrity check on cache GET — the restore worker re-verifies BLAKE3(cache_value) == chunk_hash before returning. Adds ~5 ms per chunk for re-hashing (parallel with consumption); acceptable. Detection: hash mismatch → eject cache entry + alert.
  • Eviction of chunks-being-restored. A restore in progress reads chunk X from cache; LFU evicts X mid-restore; subsequent reads of X by the same restore stream miss. Mitigation: per-restore-stream chunk pin via Redis OBJECT FREQ bumping. Detection: per-restore-stream cache miss rate > 30%; alert.

Iteration 4 — Glacier IR tiering for cold chunks (the storage-cost move)

System after this iteration

Connections (numbered above) and line-style legend. Solid slate = sync hot path; dashed gray = async / lifecycle / metadata; dashed red = hedge (Iter 2; not depicted here for clarity).

  • ⑥ Tier Manager → Chunk Index — update tier column on the chunk row; restore workers route GETs based on current tier.

Delta vs Iter 3: S3 chunk store now has two storage classes — Standard (hot) and Glacier IR (cold). Lifecycle rule transitions chunks past 90-day idle threshold. Glacier IR is the right cold tier (vs Glacier Deep Archive) because IR retains ms-class retrieval (~150 ms p99) at $0.004/GB-mo; Deep Archive at $0.00099/GB-mo would require 3-12h restore jobs and break the 30s NFR.

Bottleneck

Iter 3's bill is 96% storage ($1.15 M/mo S3 Standard out of $1.19 M total). The compute side is fully optimized; further compute cuts would save < $5k/mo. The dominant lever is the storage line. Access-frequency analysis on a typical backup workload shows: ~10% of chunks account for 95% of GETs (the "currently-active" working set); ~90% of chunks are written-once and re-fetched only on rare full-restore events (cite Backblaze's published "less than 5% of customers ever restore" stat). The chunks in the latter set are valid candidates for cheaper storage at the cost of slightly higher GET latency on the rare restore. Storage-cost bottleneck with power-law access is the canonical "tier by temperature" pattern.

Architectural decisions to revisit

Per bottleneck-to-decisions.html#storage-cost: tier the cold majority to a cheaper medium; size the hot tier to the working set. Cite cost-shapes#storage-heavy: in storage-heavy systems, the storage line is the primary cost; tier-by-temperature is the highest-leverage move.

Alternatives considered (C4)

  • S3 Glacier IR (Instant Retrieval) — chosen. $0.004/GB-mo (5.75× cheaper than Standard); ms-class retrieval (~150 ms p99 GET) — no async restore job; same API as Standard. The threshold (90 days unaccessed) preserves the recent-access working set on Standard (sub-ms via cache, ~80 ms via direct S3) while moving cold chunks to the cheap tier without breaking restore SLO.
  • S3 Glacier Flexible Retrieval — rejected. $0.0036/GB-mo (slightly cheaper than IR) but 1–5 minute or 3–5 hour async restore jobs. A restore would need to issue a "restore object" request, wait, then GET. Breaks the 30s NFR for cold chunks.
  • S3 Glacier Deep Archive — rejected. $0.00099/GB-mo (~4× cheaper than IR) but 12-hour restore SLA. Even rare cold-chunk restores would block the user-visible restore by 12 hours. Use Deep Archive only for "compliance retention; reads expected zero" workloads, not for backup-restore.
  • S3 Intelligent Tiering — rejected. Auto-tiers based on access patterns; pays per-object monitoring fee ($0.0025 per 1k objects per month). At 12 billion objects, the monitoring fee alone is $30k/mo — eats the savings. Worth it if we had ~10M objects with hot-cold uncertainty; at 12 B objects with predictable cold majority, manual lifecycle rules are cheaper.
  • Compress before storage (zstd level 9) — secondary win, applied independently. Compresses chunks on the wire before S3 PUT; ~30% reduction on a typical chunk-bytes mix (zip-archived files don't compress further; raw text/log/database files compress 3×). Apply on top of tiering; saves another ~30% on storage. Not the primary move (tiering is 5× the savings) but a free secondary.

The chosen move

(a) S3 Lifecycle Rule. Configure on the chunks/ prefix: "transition objects to GLACIER_IR after 90 days since last access." S3 evaluates this internally on its access-time metadata; we don't have to issue per-object transitions. (b) Chunk Index tier column. Add tier: enum {STD, GIR} to the RocksDB chunk row; updated by the Tier Manager nightly based on the prior-day's S3 access logs (the lifecycle rule transitions; the Tier Manager mirrors the state in the index so restore workers know which tier to expect). (c) Restore Worker tier dispatch. On cache miss, restore worker reads tier from the chunk index; routes GET to S3 Standard or Glacier IR endpoint as appropriate (URL prefix differs). (d) Promotion on access. A Glacier IR chunk read by a restore is auto-promoted back to Standard if it's read more than 3 times in 7 days (the "currently-active" criterion); the cost of the back-and-forth tier transition ($0.005 per 1k GB transitioned) is amortized by the savings on the next 90-day idle window.

Cite cost-shapes.html#storage-heavy for the canonical tiered-storage pattern; cite Backblaze's storage-pod blog series for the empirical "<5% of stored data is ever read after 30 days."

Degradation contract

When the chunk-index shard is unavailable: CheckChunks returns 503 for the affected hash range; agent retries with exponential backoff; the SoR (S3) is still accepting PutChunks but they'll be rejected as "duplicate" if a hashed chunk later turns out to exist (S3 PUT is idempotent on key). When the Hot-Chunk Cache cluster is unavailable: restore worker falls back to direct S3 GET (Iter 2 path); restore p95 reverts to ~3 s — still inside SLO. When S3 Standard is unavailable: writes fail with 503; reads on cold-only chunks may still succeed via Glacier IR. When the chunk index is fully lost: rebuild takes ~24h via S3 LIST; uploads paused, restores degraded but possible (manifest tells us which chunks; LIST gives us where).

Why FastCDC and not Rabin–Karp? (alt-CDC)

  • Per-byte cost. Rabin–Karp computes a polynomial rolling hash per byte; FastCDC's Gear-hash is a single XOR + table lookup per byte — ~3× faster on x86_64 (cite Xia/Jiang's published comparison).
  • Cut quality. Both produce content-defined boundaries with similar dedup-ratio properties; FastCDC adds normalized chunking (skip-the-min-distance + jump-on-no-cut) that bounds chunk size variance tighter than Rabin's pure stochastic distribution.
  • When Rabin wins. Inside a constrained CPU env (legacy embedded systems where Gear's 256-byte table doesn't fit in cache); not our case.

Why BLAKE3 and not SHA-256? (hash-alg)

  • Throughput. BLAKE3 ~1 GB/s/core sustained vs SHA-256's ~0.4 GB/s/core (with SHA-NI hardware acceleration on x86_64). At 7.7 GB/s peak ingest, BLAKE3 needs 8 cores; SHA-256 needs 19. Per-byte cost matters.
  • Security. Both have 256-bit collision resistance; both are post-quantum-equivalent at 128 bits. No security trade.
  • Parallelism. BLAKE3's Merkle-tree mode lets a single large chunk be hashed across multiple cores; useful for the rare > 16 MiB chunk (FastCDC max) and irrelevant at the common 4 MiB.
  • When SHA-256 wins. Compliance regimes (FIPS 140-2; some healthcare) require SHA-2 family; BLAKE3 isn't FIPS-validated as of 2026. The flip is: FIPS-required tenants get SHA-256 at 2.5× the per-byte cost (we'd accept this cost for the regulated subset).

Why RocksDB sharded by hash prefix and not Cassandra? (engine)

  • Access pattern. Chunk-existence check is a 32-byte key → 100-byte value point-read with 90% hit rate. RocksDB's SST-level Bloom filters give sub-ms p99 with no coordinator hop; embedded in-process avoids the ms-class network RTT.
  • Cassandra is for range-scan within a partition — e.g., "list this customer's file versions in time order." We keep Cassandra for FileCatalog (where range-scan is dominant) and use RocksDB for the chunk index (where point-read is dominant).
  • FoundationDB rejected: transactional KV; transactions are unused (chunk insert is naturally idempotent on hash) so the txn-coordinator overhead is pure cost.

What if a single byte is changed in a 100 GB VMware snapshot? (edit-pattern)

  • Naive (whole-file SHA): 100 GB re-uploaded; dedup ratio = 0%; bandwidth + storage doubled overnight.
  • FastCDC: 1 chunk invalidated (the chunk containing the changed byte); 4 MiB re-uploaded; dedup ratio = 99.996% on this file. Backups complete in 2 seconds instead of 30 minutes.
  • Why CDC localizes the invalidation: rolling Gear-hash boundaries are self-resynchronizing — within a chunk-size of the edit, boundaries align with the previous version's. Net: only the chunk that contains the edit is unique; everything else is a dedup hit.
  • The catch: if the edit happens to fall on a chunk-boundary byte (rare, <0.001% probability), 2 chunks are invalidated instead of 1 — still 99.99% dedup.

How does cross-customer-dedup-forbidden actually get enforced? (privacy)

  • The threat (Harnik et al. 2010): if customer A uploads chunk H and gets back "exists" (because customer B previously uploaded H), A learns that B (or some other customer) has H. This is a side-channel that leaks file existence — e.g., "this celebrity has the leaked-photo chunk."
  • The defense: per-customer Bloom filter is the customer-visible scope. The chunk index can know "some customer has H"; but the API only tells customer A "you have H" if A's per-customer Bloom says so. A and B see identical responses regardless of cross-customer state.
  • Storage savings forfeit: ~5× (cross-customer dedup ratio is > 80% on backup workloads vs 50% intra-customer). We pay $1.15M/mo Standard storage where global-dedup would pay $230k/mo. The privacy NFR is worth $920k/mo.
  • Probe path: if compliance regime relaxed (B2B contracts where customers share corporate boundaries), enable per-org dedup as a tier; the per-customer Bloom expands to per-org Bloom.

What if 1M concurrent restores happen at once (mass disaster recovery)? (scale flip)

  • Restore Worker fleet: autoscales 5 → 250 nodes (we sized for this in Iter 2); 1M concurrent restores at 333 MB/s each = 333 TB/s aggregate — way past any single-region S3 throughput cap (S3 region tops out around 5 PB/day ≈ 60 GB/s on read).
  • Bottleneck flips to S3 read throughput. The hot-chunk cache absorbs 80% but the remaining 20% (66 GB/s) hits S3's region cap.
  • Mitigation: per-customer restore quota (e.g., 100 MB/s per concurrent stream during regional incident); SREs hand-tune during the actual event.
  • Real-world: mass DR events are rare (one per several years); when they happen, restore SLO degrades from 30s to several minutes — customers accept this during a regional outage.

Where does the bill come from? What's the cost knob? (cost)

  • ~88% is S3 storage: $115k Standard (5 PB hot) + $180k Glacier IR (45 PB cold) = $295k. Compute + cache is $40k.
  • Shape: storage-heavy (cite cost-shapes#storage-heavy) — the inverse of #36's compute-heavy SRE percentile shape.
  • Top knob: tier threshold (90d → 30d unaccessed). Saves ~$60k on storage at the cost of slightly more cold-restore latency.
  • Next knob: compress before storage (zstd level 9). ~30% reduction at decompression cost ≈ 30 MB/s/core CPU added. Saves ~$90k storage at +$3k compute.
  • Hardest to cut: the per-customer Bloom storage on RocksDB (~1.5 TB across the fleet) is structural to the privacy requirement; halving it would either accept a higher false-positive rate (cost: more index point-reads) or violate the privacy NFR.

How do you migrate from Naive (whole-file SHA) to Final without downtime? (migration)

  • Phase 1 — chunk index alongside Naive: stand up RocksDB chunk index + S3 Chunk Store; new uploads dual-write (legacy whole-file path AND new FastCDC path) for 30 days; agent flag flips per-customer.
  • Phase 2 — backfill: for existing customers, agent runs FastCDC over the next backup-window's data; existing whole-file-uploaded files migrate lazily on first re-upload (within 90 days for active customers; up to a year for inactive). Old whole-file objects deleted after both paths confirm content match.
  • Phase 3 — cache + tier: bring online the Hot-Chunk Cache and Glacier IR lifecycle rules; cache-warmer pre-loads from a synthetic GET histogram for the first 7 days.
  • Phase 4 — restore-path cutover: per-customer flag; canary 1% → 10% → 100%; rollback trigger = restore p95 > 1.5× baseline.
  • Total: 90–180 days for full migration; storage savings ramp linearly as backfill proceeds.

S7. Key Takeaways

  • The whole-file-hash kernel is a single-edit-pattern artifact. Once realistic edits enter (1-byte change in a 10 GB file), whole-file hashing collapses dedup ratio to ~0%. The right move is to change the dedup unit to one whose locality matches the edit pattern — content-defined chunks, not file-sized hashes.
  • Pick the chunker by stochastic boundary quality, not by name. FastCDC vs Rabin–Karp share the load-bearing property (boundaries determined by content, self-resynchronizing under edits). FastCDC wins on per-byte cost; Rabin wins where pure stochastic distribution matters more than CPU. The 4 MiB target chunk size is the Pareto knee — smaller helps dedup and hurts restore tail; bigger does the opposite.
  • Sharding gives you QPS and failure-isolation; Bloom gives you the common-path skip. Sharded RocksDB by chunk_hash[0] is 256 logical shards on 8 physical nodes; per-customer Bloom filter co-located on each shard skips the index lookup on the dedup-hit (50%) common case. Both moves are needed: sharding alone exposes coordinator-bottleneck-equivalent costs on aggregation; Bloom alone doesn't help the cardinality of the index.
  • Cross-customer dedup forbidden is a privacy NFR, not a technical one. The Harnik 2010 side-channel is real; per-customer Bloom is the security boundary. The 5× storage-savings forfeit ($920k/mo) is the price of the privacy NFR — flag this explicitly when interviewers push on cost.
  • Tail-latency dominates parallel-fetch restore. 2,386 chunks fanning out means at least one will land in the S3 GET tail with ~100% probability. Hedged reads at p95-elapsed timer cost ~5% extra GETs and drop p99 chunk-fetch ~2× — the canonical Dean & Barroso "Tail at Scale" play.
  • Power-law popularity dictates a small hot cache. Top-0.0005% of chunks (60k of 12B) absorb 80% of restore reads; a 250 GB Redis LFU cache pays for itself in latency (sub-ms vs S3's 80 ms p99). Cache larger than the knee gives diminishing returns.
  • Storage cost is the bill in storage-heavy systems. 88% of Final's $337k bill is S3 storage; compute is barely 11%. The dominant lever is tier by temperature — Glacier IR at 90d unaccessed gives 5.75× cheaper storage with ms-class retrieval (no async restore job; SLO held).
  • Cost shape inverts vs compute-heavy systems. Where #36 (streaming percentiles) is dominated by atomic sketch updates and Kafka SoR, #43 (backup dedup) is dominated by petabytes-on-S3 storage. The diagnostic question to ask before iterating: "where's the elephant — compute, storage, network, or memory?"
