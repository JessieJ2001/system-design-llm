Fan-Out Notification System

Delivering notifications from a user with 10 million followers to every follower's feed — navigating the tradeoffs between push, pull, and hybrid fan-out strategies.

NALSD Walkthrough • ~15 min read

The Scenario

You are designing the notification delivery system for a social media platform. When a user creates a post, every one of their followers must see it in their notification feed. The challenge: follower counts range from 0 to 10 million, and the system must handle both extremes efficiently.

Concrete Numbers

MetricValue
Total users500 million
Daily Active Users (DAU)100 million
Posts per second (platform-wide)10,000
Average followers per user200
Celebrity users (>5K followers)~5,000 users
Max followers (top celebrity)10 million
Average notification size200 bytes
Feed loads per DAU per day10

The Core Challenge: The Fan-Out Problem

When a user with 200 followers posts, the system writes 200 notifications. When a celebrity with 10 million followers posts, the system must write 10 million notifications. This is a 50,000x difference in work for the same user action. Any design that treats all users equally will either be wasteful for small users or collapse under celebrity posts.

Back-of-Envelope Math

Fan-Out Write Volume

Average fan-out (push-on-write for all users):
  10,000 posts/sec × 200 avg followers = 2,000,000 notification writes/sec

Celebrity impact:
  One celebrity post = 10,000,000 writes
  At 100,000 writes/sec to notification store: takes 100 seconds
  If 10 celebrities post in the same minute:
    100,000,000 extra writes in 60 seconds
    = 1,667,000 additional writes/sec
  Total during celebrity burst: 2M + 1.67M = 3.67M writes/sec
  That is 83% above baseline. Can our storage handle it?

Storage Requirements

Each notification entry:
  - recipient_user_id:  8 bytes
  - post_id:            8 bytes
  - author_user_id:     8 bytes
  - timestamp:          8 bytes
  - notification_type:  1 byte
  - read_flag:          1 byte
  - padding/overhead:   ~166 bytes
  Total:                ~200 bytes

Daily notification writes:
  2,000,000/sec × 86,400 sec = 172.8 billion notifications/day
  172.8B × 200 bytes = 34.56 TB/day

With 7-day retention:
  34.56 TB × 7 = 241.9 TB

This is a LOT of storage. Even with cheap SSDs at $0.10/GB,
that is $24,190 just for notification storage.
And 2M writes/sec requires a distributed storage layer.

Read Volume (Feed Loads)

100M DAU × 10 feed loads/day = 1 billion feed reads/day
  = 1,000,000,000 / 86,400 ≈ 11,574 reads/sec

Each feed read returns the latest 50 notifications.
  50 × 200 bytes = 10 KB per feed load.
  11,574 reads/sec × 10 KB = 115 MB/sec read throughput.

This is very manageable for a distributed cache or database.

Two Fundamental Approaches

PUSH model (fan-out-on-write):
  Write amplification:  10,000 posts/sec → 2,000,000 writes/sec
  Read cost per feed:   1 query (read user's pre-built notification list)
  Latency at read time: Very fast (~1ms, it is a simple key lookup)
  Storage:              241.9 TB/week (every follower gets a copy)

PULL model (fan-out-on-read):
  Write cost:           10,000 posts/sec (just store the post, no fan-out)
  Read cost per feed:   For each user, query posts from all N people they follow
                        11,574 reads/sec × 200 followed users = 2.3M DB reads/sec
  Latency at read time: Higher (must query 200 users and merge-sort results)
  Storage:              Just the posts: 10K/sec × 200B × 86,400 = 172 GB/day
The fundamental tradeoff: Push trades write amplification and storage for fast reads. Pull trades read amplification and latency for low storage and simple writes. Neither is universally better. The right choice depends on the read-to-write ratio and the follower count distribution.

Naive Design: Pure Push (Fan-Out-on-Write for Everyone)

The simplest approach: when any user posts, immediately write a notification entry to every follower's notification list.

Architecture

Fan-Out Worker Code

def fan_out_on_write(post):
    """Called when a new post is created."""
    # 1. Store the post
    posts_db.insert(post)

    # 2. Get all followers of the author
    followers = followers_db.get_followers(post['author_id'])

    # 3. Write notification to each follower's feed
    for follower_id in followers:
        notifications_db.insert({
            'recipient_id': follower_id,
            'post_id': post['post_id'],
            'author_id': post['author_id'],
            'timestamp': post['created_at'],
            'read': False,
        })

def get_feed(user_id):
    """Called when a user opens their notification feed."""
    # Simple: just read from the pre-built list
    return notifications_db.query(
        recipient_id=user_id,
        order_by='timestamp DESC',
        limit=50
    )

For regular users with 200 followers, this works beautifully. The fan-out completes in milliseconds. The feed read is a single indexed query. Latency is excellent.

What Breaks

Problem 1: Celebrity Posts Cause Write Storms

A celebrity with 10M followers posts a photo.

Fan-out begins:
  t=0s     Fan-out worker starts writing 10,000,000 notifications.
  t=1s     10,000 writes completed (100K writes/sec to DB cluster).
           Remaining: 9,990,000.
  t=10s    1,000,000 writes completed.
           Remaining: 9,000,000.
  t=100s   All 10,000,000 writes completed.

During these 100 seconds:
  - 1,000 other users posted (10K posts/sec × 100s = 1M posts).
  - Each needs fan-out too: 1M × 200 = 200M notification writes.
  - Total write demand: 10M (celebrity) + 200M (regular) = 210M in 100s.
  - Required: 2.1M writes/sec. Normal baseline: 2M writes/sec.
  - Only 5% overload from ONE celebrity post.

But what if 10 celebrities post in the same minute?
  10 × 10M = 100M celebrity writes in ~100s.
  Plus regular: 200M writes.
  Total: 300M in 100s = 3M writes/sec.
  That is 50% over capacity. Queue backs up. Latency spikes.
  Followers of non-celebrities also experience delayed notifications
  because the DB is overwhelmed by celebrity fan-outs.

Problem 2: Wasted Writes for Inactive Users

Of the 10M followers of a celebrity:
  - 2M are DAU (20% of users are active daily)
  - 8M are inactive (haven't opened the app in days or weeks)

We wrote 8M notification entries that nobody will ever read.
At 200 bytes each: 1.6 GB of wasted storage per celebrity post.
Over a day with 100 celebrity posts: 160 GB of wasted writes.

Problem 3: Latency Inequality

A follower who is 1st in the fan-out list gets the notification in milliseconds. A follower who is 10,000,000th gets it after 100 seconds. This creates a visible inconsistency: two followers sitting next to each other see the same post at different times.

The root cause: Pure push treats all users equally, but users are NOT equal. The distribution of follower counts is extremely skewed (power law). 99.999% of users have fewer than 5,000 followers, and the push model works perfectly for them. The problem is the 0.001% with millions of followers.

Iteration 1: Hybrid Fan-Out (Push + Pull)

The key insight: use push for regular users (where it is cheap and fast) and pull for celebrities (where push is prohibitively expensive).

The Threshold

Define a threshold: 5,000 followers.

  Users with ≤ 5,000 followers → PUSH model (fan-out-on-write)
  Users with > 5,000 followers → PULL model (fan-out-on-read)

How many users fall into each category?
  Follower count distribution (power law):
    < 100 followers:       400M users (80%)
    100 - 1,000:            80M users (16%)
    1,000 - 5,000:          15M users (3%)
    5,000 - 100,000:         4.5M users (0.9%)
    100,000 - 1M:            450K users (0.09%)
    > 1M:                     50K users (0.01%)

  Celebrity tier (> 5,000): ~5M users = 1% of total users
  Regular tier (≤ 5,000):  ~495M users = 99% of total users

How Hybrid Fan-Out Works

When a regular user posts, the system fans out notifications to all their followers immediately (push). When a celebrity posts, the system does NOT fan out. Instead, it simply stores the post in the celebrity's post list.

When a follower opens their feed, the system does two things:

  1. Read the pre-built notification list (contains pushed notifications from regular users they follow).
  2. Query the latest posts from each celebrity they follow (pull on demand).
  3. Merge the two lists by timestamp and return the top 50.
Feed read for User Alice (follows 200 regulars + 10 celebrities):

Step 1: Read pre-built list
  Query: notifications WHERE recipient_id = Alice ORDER BY time DESC LIMIT 50
  Returns: 50 pushed notifications
  Latency: ~2ms (indexed lookup)

Step 2: Pull celebrity posts
  For each of 10 followed celebrities:
    Query: posts WHERE author_id = celebrity_X ORDER BY time DESC LIMIT 5
  Returns: up to 50 recent celebrity posts
  Latency: 10 parallel queries × ~1ms each = ~1ms (with caching)

Step 3: Merge and sort
  Merge 50 pushed + 50 pulled, sort by timestamp, take top 50.
  Latency: ~0.1ms (in-memory sort of 100 items)

Total feed read latency: ~3ms

Write Volume Savings

Without hybrid (pure push):
  All fan-out on write: 2,000,000 writes/sec baseline
  + celebrity bursts: up to 3,670,000 writes/sec

With hybrid:
  Only regular users fan out on write.
  Regular posts: 10,000 posts/sec × 99% regular = 9,900 posts/sec
  Avg fan-out: 9,900 × 200 = 1,980,000 writes/sec

  Celebrity posts: 10,000 × 1% = 100 posts/sec
  No fan-out for these. Just 100 writes/sec to posts table.

  Total writes: 1,980,100 writes/sec (steady, no spikes!)

  Savings on celebrity bursts:
    Before: 10M writes per celebrity post.
    After:  1 write per celebrity post.
    Reduction: 99.99999%

Implementation

CELEBRITY_THRESHOLD = 5000

def on_new_post(post):
    """Called when any user creates a post."""
    # Always store the post
    posts_db.insert(post)

    author_id = post['author_id']
    follower_count = followers_db.count(author_id)

    if follower_count <= CELEBRITY_THRESHOLD:
        # PUSH: fan out to all followers
        followers = followers_db.get_followers(author_id)
        for follower_id in followers:
            notifications_db.insert({
                'recipient_id': follower_id,
                'post_id': post['post_id'],
                'author_id': author_id,
                'timestamp': post['created_at'],
            })
    else:
        # PULL: do nothing extra. The post is already in posts_db.
        # Followers will pull it at read time.
        pass

def get_feed(user_id):
    """Merge pushed notifications with pulled celebrity posts."""
    # Step 1: Get pushed notifications
    pushed = notifications_db.query(
        recipient_id=user_id,
        order_by='timestamp DESC',
        limit=50
    )

    # Step 2: Get followed celebrities
    followed_celebrities = get_followed_celebrities(user_id)

    # Step 3: Pull latest posts from each celebrity
    pulled = []
    for celeb_id in followed_celebrities:
        celeb_posts = posts_cache.get(celeb_id)  # from Redis cache
        if celeb_posts is None:
            celeb_posts = posts_db.query(
                author_id=celeb_id,
                order_by='timestamp DESC',
                limit=5
            )
            posts_cache.set(celeb_id, celeb_posts, ttl=60)
        pulled.extend(celeb_posts)

    # Step 4: Merge and return top 50
    combined = pushed + pulled
    combined.sort(key=lambda x: x['timestamp'], reverse=True)
    return combined[:50]

Iteration 2: Caching Celebrity Posts and Delivery Optimization

Celebrity Post Cache

Every feed read for users following a celebrity will pull that celebrity's latest posts. If a celebrity has 10M followers and 2M are DAU loading their feed 10 times/day, that is 20M reads of the same celebrity posts per day. Caching these in Redis eliminates redundant database queries.

Celebrity post cache sizing:

  Number of celebrities (> 5K followers): ~5,000
  Posts cached per celebrity: 100 (latest)
  Size per post: ~200 bytes

  Total cache size: 5,000 × 100 × 200 = 100 MB

  This is TINY. A single Redis instance with 1 GB of RAM
  can hold this 10 times over.

Cache hit rate:
  Celebrity posts are read millions of times each.
  A post is written once and read by 2M+ DAU.
  Reads-to-writes ratio: > 20,000:1
  Expected cache hit rate: > 99.99%

Without cache:
  Feed reads pulling celebrity posts: 11,574 reads/sec × avg 10 celebrities
  = 115,740 DB queries/sec to the posts table.

With cache:
  99.99% hit rate → 115,740 × 0.01% = ~12 DB queries/sec.
  Reduction: 99.99%

Async Fan-Out via Message Queue

The push fan-out for regular users should not happen synchronously in the post creation API call. If a user has 5,000 followers, the API would block for 50ms while writing 5,000 notifications. Instead, the API publishes a "new post" event to a message queue, and fan-out workers consume the event asynchronously.

Notification Delivery via WebSocket

So far we have focused on the feed (user opens app, loads notifications). But users also expect real-time push notifications (the red badge, the toast notification). For online users, we can deliver notifications via WebSocket connections.

Real-time delivery:

  When a fan-out worker writes a notification to the DB,
  it also publishes to a "notify" channel.

  WebSocket servers subscribe to these channels.
  If the recipient is currently connected, the WebSocket server
  pushes the notification in real-time.

  If the recipient is offline, the notification sits in the DB
  until the next feed load.

Online users at any moment:
  100M DAU, assume 10% online at any instant = 10M connections
  Each WebSocket server handles ~50K connections.
  10M / 50K = 200 WebSocket servers.

Notifications per second to deliver in real-time:
  2M notification writes/sec × 10% online = 200K real-time pushes/sec
  200K / 200 servers = 1,000 pushes/sec per server. Easy.

Handling the "Following" List for Pull

When a user's feed is loaded, we need to quickly determine which celebrities they follow. This should not require a full scan of the follower graph.

Optimization: Maintain a separate "followed_celebrities" list per user.

  When user A follows celebrity B:
    1. Add to follower graph (A → B)
    2. Add B to A's followed_celebrities list (if B has > 5K followers)

  When loading feed for user A:
    followed_celebrities = cache.get(f"celeb_follows:{A}")
    Typically 0-20 celebrities. Stored as a small set in Redis.

  What if a user crosses the 5K threshold?
    Background job runs daily:
    1. Check all users near the threshold.
    2. If a user crossed 5K, mark them as "celebrity."
    3. Stop fanning out their future posts.
    4. For existing followers, add to followed_celebrities lists.

    This is a rare event (few users cross 5K/day) and can be
    processed in a batch job overnight.

Final Design

Performance Comparison

MetricPure PushPure PullHybrid
Write volume (steady) 2M/sec 10K/sec 1.98M/sec
Write volume (celebrity burst) 3.67M/sec 10K/sec 1.98M/sec (no spike!)
Feed read latency ~1ms ~50-100ms ~3-5ms
Feed read DB queries 1 per load 200+ per load 1 + ~10 cache hits
Storage (7-day) 241.9 TB ~1.2 TB (posts only) ~240 TB + 100 MB cache
Celebrity post delivery time 0-100 seconds Instant at read Instant at read
Why not pure pull? Pure pull gives excellent write performance but requires 200+ queries per feed load. At 11,574 feed loads/sec, that is 2.3M DB queries/sec. Even with caching, the tail latency (p99) for feed reads would be high because some followed users' posts won't be cached. The hybrid approach keeps feed reads fast for the common case (mostly pre-built lists) while avoiding the celebrity write storm.

Edge Cases and Considerations

Edge CaseHow It Is Handled
User unfollows someone Old pushed notifications remain (stale but harmless). Celebrity pull stops immediately. Optional: background cleanup of old pushed entries.
User crosses celebrity threshold Daily batch job detects crossover. Future posts use pull model. No retroactive changes needed.
Celebrity deletes a post Remove from celebrity cache immediately. Push notifications for the post (if any) are left as-is or lazily filtered at read time.
New user with zero followers posts Zero fan-out work. Post is stored. If they gain followers later, future posts will fan out.
Notification deduplication At merge time, deduplicate by post_id (same post could appear in pushed list if a regular user was later promoted to celebrity).

Key Takeaways

  • One-size-fits-all is the enemy of scale. A design that works for 200 followers collapses at 10 million. Identify the distribution of your data (power law in this case) and design different paths for different segments.
  • Push is cheap for small fan-outs, expensive for large ones. 200 writes is trivial. 10 million writes from a single event is a disaster. The hybrid approach puts each user on the right path.
  • Caching solves the pull cost. The pull model's weakness (many DB queries per read) disappears when celebrity posts are cached. 100 MB of Redis eliminates 99.99% of pull queries.
  • Async fan-out decouples the user experience from the backend work. The user sees their post published instantly. The fan-out happens in the background via message queue workers.
  • Always do the math on both approaches before choosing. "Push is simpler" is not a sufficient argument. Calculate the write volume, the read volume, the storage, and the latency for both. The numbers will tell you which approach (or which hybrid) is right.
  • Real-time delivery is a separate concern. The notification feed (pull on app open) and the real-time push (WebSocket) are different systems with different requirements. Solve them independently.
