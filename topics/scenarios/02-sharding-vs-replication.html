Sharding vs Replication: E-Commerce Product Catalog

When your data fits on one machine, replication is almost always the answer. Learn to recognize when sharding is necessary — and when it is premature complexity.

NALSD Walkthrough • ~15 min read

The Scenario

You are designing the database layer for an e-commerce product catalog. The requirements are:

MetricValue
Total products10 million
Product record size~5 KB (title, description, price, images, attributes, reviews summary)
Writes per day100,000 (price updates, new listings, inventory changes)
Reads per day10 million (product page views)
Read latency target<100 ms p99
Availability target99.99% (downtime < 53 min/year)

The question: do you need sharding, replication, or both?

Back-of-Envelope Math

Let us calculate whether a single database server can handle this workload. (Reference: BOE cheat sheet.)

Storage

Total storage:
  10M products × 5 KB/product = 50 GB

With indexes (typically 20-30% overhead):
  50 GB × 1.3 = 65 GB

Modern server RAM: 128-256 GB
  → The entire dataset + indexes fit in memory.
  → Every read can be served from RAM (no disk I/O).
  → This is a crucial insight: if data fits in RAM, reads are fast.

Read Throughput

Read QPS:
  Average: 10M reads/day ÷ 100,000 sec/day = 100 reads/sec
  Peak (3x):                                = 300 reads/sec

PostgreSQL capacity for simple key lookups from RAM:
  10,000 - 50,000 reads/sec (depending on query complexity)

Our load: 300 reads/sec
Capacity: 10,000+ reads/sec
Utilization: 300 / 10,000 = 3%

→ A single PostgreSQL server is at 3% utilization for reads.
  We have 30x headroom.

Write Throughput

Write QPS:
  Average: 100K writes/day ÷ 100,000 sec/day = 1 write/sec
  Peak (10x): 10 writes/sec

PostgreSQL write capacity:
  1,000 - 10,000 writes/sec (with fsync, depending on durability)

Our load: 10 writes/sec peak
Capacity: 1,000+ writes/sec
Utilization: 10 / 1,000 = 1%

→ Writes are trivial. Not even close to a bottleneck.

Bottom line: A single PostgreSQL server can handle this workload at 3% read utilization and 1% write utilization. The data fits entirely in RAM. From a pure capacity standpoint, one server is more than enough.

Naive Design: Single PostgreSQL Server

-- Product catalog schema
CREATE TABLE products (
  product_id   BIGINT PRIMARY KEY,
  title        VARCHAR(256),
  description  TEXT,
  price_cents  INTEGER,
  currency     CHAR(3) DEFAULT 'USD',
  category_id  INTEGER,
  brand        VARCHAR(128),
  image_urls   TEXT[],
  attributes   JSONB,
  rating_avg   DECIMAL(2,1),
  review_count INTEGER DEFAULT 0,
  in_stock     BOOLEAN DEFAULT true,
  created_at   TIMESTAMP DEFAULT NOW(),
  updated_at   TIMESTAMP DEFAULT NOW()
);

-- Indexes for common queries
CREATE INDEX idx_products_category ON products(category_id);
CREATE INDEX idx_products_brand ON products(brand);
CREATE INDEX idx_products_price ON products(price_cents);
CREATE INDEX idx_products_rating ON products(rating_avg DESC);

One server, one database. All reads and writes go to the same instance. Simple, easy to reason about, easy to operate. Every query is consistent (no replication lag). Backup with pg_dump or WAL archiving.

Performance Check

Query TypeExpected LatencyMeets Target?
Get product by ID0.5-2 ms (in-memory PK lookup)Yes (<100 ms)
List by category2-10 ms (index scan)Yes
Search by brand + price range5-20 ms (composite index scan)Yes
Update price1-5 ms (PK update)N/A (write)

All latency targets are met. Throughput is well within capacity. So what is wrong?

What Breaks: Availability

The naive design has a fatal flaw, and it is not performance. It is availability.

Single Point of Failure

Single server availability (typical):
  99.9% uptime = 8.7 hours of downtime per year

Our target: 99.99% = 53 minutes of downtime per year

Gap: 8.7 hours vs 53 minutes → we're 10x too much downtime.

Sources of downtime for a single server:
  - Hardware failure (disk, RAM, NIC): ~1-2 events/year
  - OS updates requiring reboot: ~4-6 times/year
  - PostgreSQL major version upgrade: ~1-2 times/year
  - Unexpected crashes: ~1-2 times/year

Each event: 5-60 minutes of downtime
Total: easily 8+ hours/year

What About Backups?

Restoring from a backup after server failure:
  1. Detect failure:              1-5 minutes
  2. Provision new server:        5-15 minutes
  3. Restore 65 GB from backup:   10-30 minutes (depends on I/O)
  4. Replay WAL to catch up:      5-15 minutes
  Total recovery time:            20-65 minutes

For 99.99% uptime (53 min/year budget), a SINGLE failure event
can consume the entire annual downtime budget.

The bottleneck is not throughput or latency — it is availability. When this single server goes down, the entire product catalog is unavailable. No one can browse products, which means no one can buy anything. For an e-commerce site, this is catastrophic.

Iteration 1: Add Read Replicas

The fix for availability is redundancy. Add two read replicas using PostgreSQL streaming replication.

Architecture: 1 Primary + 2 Replicas

Availability Calculation

For reads (served by any of the 3 nodes):
  P(all 3 nodes down simultaneously):
  = (1 - 0.999)^3
  = (0.001)^3
  = 0.000000001
  = 99.9999999% availability for reads

  (In practice, correlated failures like data center outages
   reduce this, but with nodes in different availability zones
   we can still achieve 99.99%+ easily.)

For writes (only the primary):
  If the primary fails, promote a replica:
  - Automatic failover detection: 5-10 seconds
  - Replica promotion: 1-5 seconds
  - DNS/connection update: 5-30 seconds
  Total failover time: ~30 seconds

  With 2-3 failover events per year:
  = 3 × 30 sec = 90 seconds of write downtime/year
  = 99.999% write availability

Replication Lag: Is It a Problem?

Async replication lag: typically 10-100 ms

For a product catalog, what does 100 ms of lag mean?
  - A price update takes 100 ms to reach replicas
  - A user might see the old price for 100 ms
  - This is acceptable for catalog browsing

When is lag NOT acceptable?
  - Inventory count during checkout (use primary for this)
  - "Just purchased" confirmation (read from primary)

Rule: read from replicas for browsing, read from primary
for transactional operations (cart, checkout, payment).

Cost

Single server (naive design):
  1 × (256 GB RAM, 500 GB SSD) ≈ $500/month (cloud)

With replicas:
  3 × $500/month = $1,500/month

Additional cost: $1,000/month for 99.99% availability.
For an e-commerce site doing $10M+/year in revenue,
this is 0.018% of revenue — an obvious yes.

Replicas also improve read performance (even though we do not need it here). If traffic grows 10x to 3,000 reads/sec, we can spread the load across 3 nodes = 1,000 reads/sec per node, still well within capacity.

Why NOT Shard

A common mistake in system design interviews is to shard immediately. Let us examine why sharding is the wrong choice for this scenario.

What Sharding Costs You

ConcernWith Replication (our choice)With Sharding
Browse by category Single query, single database Scatter-gather across all shards (if sharded by product_id, categories span shards)
Search by brand + price range Single query with composite index Query all shards, merge results
JOINs (product + reviews) Standard SQL join Cross-shard join or denormalization required
Transactions Standard ACID Distributed transactions (2PC) or give up ACID
Schema migrations Run on 1 primary, replicas follow Coordinate across N shards, rollback is harder
Operational complexity Well-understood primary-replica setup Shard routing, rebalancing, monitoring per shard

The Sharding Decision Criteria

Sharding solves exactly two problems:

  1. Data does not fit on one machine. Our data is 50 GB. A single server can hold terabytes. Not applicable.
  2. Write throughput exceeds a single machine's capacity. We have 10 writes/sec peak. A single PostgreSQL handles 1,000+. Not applicable.
Sharding decision checklist:

  ☐ Data size > 1 TB?                      No  (50 GB)
  ☐ Data size > available RAM?              No  (50 GB < 256 GB RAM)
  ☐ Write QPS > single node capacity?       No  (10 << 1,000)
  ☐ Read QPS > replica-scaled capacity?     No  (300 << 30,000 across 3 nodes)
  ☐ Single-tenant isolation required?       No

  All "No" → Do NOT shard. Replication is sufficient.

Premature sharding is one of the most common architectural mistakes. It adds complexity to every query, every migration, and every operational procedure. Only shard when the math proves you must.

When You WOULD Shard This System

Let us calculate the thresholds where sharding becomes necessary.

Why sharding is fundamentally about write scalability. With replication, every write must go through a single primary. Adding replicas does not help — the primary still handles every write, then does extra work replicating to each replica. Three replicas means write capacity of one node (not three). Replication multiplies read capacity and gives availability, but the write bottleneck stays at one machine.

Sharding is different: it splits the data so different writes go to different nodes. Each shard only handles writes for its subset. Three shards means write capacity of three nodes. This is the only way to scale writes beyond what a single machine can handle.

Replication (3 replicas):
  Write "product A" → Primary → replicate to Replica 1, Replica 2
  Write "product B" → Primary → replicate to Replica 1, Replica 2
  All writes funnel through ONE node. Write capacity = 1 node.

Sharding (3 shards):
  Write "product A" → Shard 1 (handles products A-F)
  Write "product X" → Shard 3 (handles products S-Z)
  Writes spread across 3 nodes. Write capacity = 3 nodes.

Scenario A: Data Size Growth

Current: 10M products × 5 KB = 50 GB

If the platform becomes a marketplace (like Amazon):
  1 billion products × 5 KB = 5 TB

  5 TB exceeds what you want in a single PostgreSQL instance.
  (Technically possible, but indexes get huge, vacuuming takes
   hours, backups are slow, recovery takes forever.)

  Rule of thumb: consider sharding when a single instance
  exceeds 500 GB - 1 TB of data.

  Naive sharding: hash(product_id) % num_shards
  With 10 shards: 500 GB per shard → comfortable.

  But what happens when you need to add an 11th shard?
    hash(product_id) % 10  →  hash(product_id) % 11
    ~90% of keys remap to a different shard!
    You must migrate ~4.5 TB of data. Hours of downtime.

  Better: consistent hashing.
    Put shards on a hash ring. Each product hashes to a position
    on the ring and maps to the next shard clockwise.
    Adding shard 11: only ~1/11 ≈ 9% of keys move (the ones
    between the new shard and its neighbor on the ring).
    Migration: ~450 GB instead of 4.5 TB. 10× less data moved.

  Even better: virtual shards (logical partitions).
    Create 256 logical shards mapped to 10 physical shards.
    Each physical shard owns ~25 logical shards.
    To add an 11th physical shard: move ~23 logical shards to it.
    Each move is a self-contained chunk — no key remapping needed.
    This is how CockroachDB, Vitess, and DynamoDB work.

  Use hash(key) % N only if N will NEVER change.
  Use consistent hashing or virtual shards if you might rescale.

Scenario B: Write Throughput Growth

Current: 10 writes/sec peak

If the platform grows to a multi-vendor marketplace with
real-time inventory updates from 100K sellers:
  100K sellers × 100 inventory updates/day = 10M writes/day
  10M / 100K sec = 100 writes/sec average
  Peak: 500-1,000 writes/sec

  A single PostgreSQL: 1,000-5,000 writes/sec
  At 1,000 writes/sec we're hitting 50-100% capacity.
  Add replication overhead: primary must replicate WAL too.

  This is the threshold where sharding for writes makes sense.
  Shard by seller_id: each seller's products are on one shard.
  Writes are local to the seller's shard (no cross-shard txn).

Scenario C: Multi-Tenant SaaS

If the catalog is a SaaS product (Shopify model):
  10,000 stores × 100K products each = 1B products = 5 TB
  Each store is independent — natural shard boundary.

  Shard by tenant_id (store_id):
  - No cross-shard queries needed (stores are isolated)
  - Each shard: 1,000 stores × 100K products = 500 GB
  - 10 shards for the whole platform
  - Each store's queries hit only one shard

Final Design Summary

ComponentConfigurationJustification
Primary database PostgreSQL, 256 GB RAM, 500 GB SSD All writes. 50 GB dataset fits entirely in memory. Handles 10 writes/sec (capacity: 1,000+).
Read replicas 2 replicas in different AZs 99.99%+ read availability. Handles 300 reads/sec across 3 nodes (capacity: 30,000+).
Replication Async streaming, ~50 ms lag Acceptable for catalog reads. Checkout reads go to primary.
Failover Automatic promotion (Patroni or cloud-managed) 30-second failover. Write downtime: ~90 sec/year.
Sharding None 50 GB data, 10 writes/sec. No sharding trigger met.

The best architecture is the simplest one that meets your requirements. Replication gives us availability. Our data is small enough and our writes are few enough that sharding adds no benefit — only complexity. When the math changes, then shard.

Sharding vs Replication: The Complete Comparison

Beyond this specific e-commerce scenario, here is a comprehensive comparison of sharding and replication as general database scaling strategies. This is the mental model you want in an interview.

The Core Difference

They solve fundamentally different bottlenecks, but are almost always used together in production — each shard is replicated.

Primary Goal

DimensionShardingReplication
Primary goalWrite scalability + storage capacityRead scalability + high availability
Core mechanismPartition data so each node handles a fraction of writesCopy data so multiple nodes can serve reads and survive failures
Scaling axisHorizontal write throughput scales ~linearly with shard countRead throughput scales ~linearly with replica count; writes do NOT scale (all replicas must process every write)

When to Use Each

Use Replication When:

  • High availability / fault tolerance — survive node failures with zero downtime
  • Read throughput is the bottleneck — e.g., a 100:1 read-to-write ratio, add replicas to absorb reads
  • Geographic distribution — replicas in US, EU, Asia for low-latency reads
  • Disaster recovery — a replica in another data center ensures you survive a full DC outage
  • Zero-downtime maintenance — patch replicas one by one, failover primary
  • Analytics isolation — run expensive reporting queries on a replica without impacting production

Use Sharding When:

  • Data exceeds a single machine’s storage — hundreds of GB to TB range
  • Write throughput exceeds a single node — e.g., 50K writes/sec when one Postgres maxes at 15K
  • Tenant isolation — compliance or performance reasons (one tenant per shard)
  • Index size exceeds RAM — smaller per-shard indexes fit in memory, keeping queries fast
  • Backup/restore windows are too long — 10 TB takes hours; 10 × 1 TB can be parallelized

Use Both When:

You have high write AND read throughput at scale. This is the norm for any large system — each shard gets its own set of replicas.

What Problems They Solve

ProblemSharding Solves It?Replication Solves It?
Storage limits (single disk/node can't hold all data)YesNo
Write throughput ceilingYesNo
Connection limits (single server maxes out at ~5-10K connections)YesPartially (replicas absorb read connections)
Index size too large for RAMYes (per-shard indexes are smaller)No (replicas have the same indexes)
Slow backups/restoresYes (parallelizable per shard)Partially (restore from replica, not from backup)
Noisy neighbor / tenant isolationYes (separate shards per tenant)No
Single point of failureNo (losing a shard = losing that data)Yes

Geographic latency for readsNoYes (replicas near users)

Zero-downtime maintenancePartially (rolling per shard)Yes (failover primary, patch one at a time)
Analytics isolationNoYes (run heavy queries on a replica)

New Problems Introduced by Sharding

Every scaling technique creates new problems. In an interview, showing that you understand the tradeoffs — and know how to mitigate them — is often more valuable than proposing the solution in the first place.

1. Cross-Shard Queries Are Expensive

Problem: A query like WHERE category='electronics' ORDER BY price LIMIT 20 must hit every shard (scatter-gather), multiplying latency by the shard count.

Solutions:

  • Denormalize — secondary indexes per shard; each returns local top-N, router merges.
  • Better shard key — shard by your most common query filter (e.g., tenant_id) so queries hit one shard.
  • Global search index — Elasticsearch spans all shards, finds IDs, then fetch full records from the right shard.
  • CQRS — writes go to write-optimized shards, reads go to a separate read-optimized view organized differently.

2. Cross-Shard Transactions

Problem: Transferring inventory between two sellers on different shards requires atomically updating both. Standard ACID transactions don't span databases.

Solutions:

  • Colocate related data — design shard key so transactional data lives on one shard (e.g., shard by seller_id).
  • Two-phase commit (2PC) — guarantees atomicity but slow (2 RTTs) and coordinator is a SPOF. Use for financial txns.
  • Saga pattern — local transactions + compensating actions. Trades strict atomicity for performance. Use when eventual consistency is OK.
  • Accept eventual consistency — some ops (e.g., category transfer) tolerate a brief inconsistent window.

3. Hotspots & Uneven Load

Problem: A celebrity seller generates 100x the traffic. That shard becomes a bottleneck while others sit idle.

Solutions:

  • Key jitter — append random suffix (key_0…key_9) to spread a hot key across 10 shards. Reads scatter and merge.
  • Split the hot shard — detect via monitoring, split one shard into two. Others stay untouched.
  • Virtual nodes — consistent hashing with 100+ virtual nodes per physical shard for even distribution.
  • Cache hot data — Redis/Memcached in front of the hot shard absorbs read load before it reaches the DB.
  • Noisy neighbor isolation — move the celebrity to a dedicated, higher-capacity shard.

4. Resharding Is Painful

Problem: hash(key) % 4 growing to 8 shards remaps nearly every key. Migrating TBs live without losing writes is hard.

Solutions:

  • Consistent hashing — adding a node moves only ~K/N keys (~12.5% for 4→5 nodes).
  • Virtual/logical shards — create 1,024 logical shards on 4 servers. Growing = moving logical shards, no re-hashing. (Vitess, DynamoDB)
  • Double-write migration — dual-write to old + new, backfill, then cut over. Zero downtime, 2x write load temporarily.
  • Shadow reads — read from both old and new, compare results, log discrepancies before switching.
  • Over-provision initially — start with 32 logical shards on 4 servers. Grow by rebalancing, not re-hashing.

5. Operational Complexity

Problem: N shards = N databases to monitor, back up, patch, and migrate. One schema migration failure leaves the system inconsistent.

Solutions:

  • Sharding proxy — Vitess (MySQL), Citus (PostgreSQL) abstract routing, coordinate migrations, pool connections.
  • IaC + automation — Terraform/Ansible ensures identical config. Rolling migrations: apply to shard 1, verify, proceed.
  • Online schema tools — gh-ost / pt-online-schema-change modify tables without locking, run in parallel per shard.
  • Managed services — CockroachDB, Spanner, DynamoDB handle sharding internally. Trade control for simplicity.

6. Loss of Relational Features

Problem: Foreign keys, UNIQUE constraints, auto-increment IDs, and cross-table JOINs don't work across shards.

Solutions:

  • Distributed IDs — Snowflake IDs (timestamp + worker + seq) or UUIDs. Globally unique, no coordination.
  • Global lookup table — for unique constraints (email), keep an unsharded email→shard map. Check before insert.
  • Denormalize — store product name/price on the order row to avoid cross-shard JOINs. Accept data duplication.
  • Eventual uniqueness — tolerate brief violations, resolve with a background reconciliation job.

7. Shard Key Is Nearly Irreversible

Problem: Changing the shard key means re-partitioning all data. For 5 TB, this takes days and risks inconsistency.

Solutions:

  • Upfront analysis — analyze top 10 query patterns against production logs. Shard key should appear in 80%+ of WHERE clauses.
  • Compound keys — (tenant_id, product_id) gives tenant isolation + distribution. Covers more query patterns.
  • Secondary routing tables — global index maps email→user_id→shard for non-shard-key lookups.
  • Full re-shard (last resort) — double-write migration to new shards with new key. Budget weeks for multi-TB.

8. Application Complexity

Problem: Every DB call must know which shard to hit. Routing bugs cause silent data corruption or missing results.

Solutions:

  • Proxy layer — Vitess/ProxySQL parses the shard key from queries and routes them. App stays shard-unaware.
  • ORM-level sharding — Django custom routers, Rails ActiveRecord sharding handle connection routing at the ORM layer.
  • Shard-per-tenant — trivial routing: look up tenant's shard once per request, use that connection for everything.
  • Integration tests — verify cross-shard queries, routing correctness, and failover before production.

New Problems Introduced by Replication

1. Replication Lag

Problem: Async replicas can be ms-to-seconds behind. A user writes then reads from a stale replica — the “read-your-own-writes” problem.

Solutions:

  • Read from primary after write — route that user's reads to primary for ~5s. Simple but increases primary load.
  • Causal consistency tokens — primary returns an LSN; replica waits until it has replayed past that LSN. Precise, no over-routing. PostgreSQL supports this natively.
  • Semi-sync replication — primary waits for 1 replica ACK. Guarantees one replica is current. Adds ~10-20ms write latency.
  • App-level routing — transactional reads (cart, checkout) go to primary; browsing reads go to replicas.
  • Accept it — 100ms lag is invisible for catalog browsing. Only mitigate where users would notice.

2. Split-Brain

Problem: Network partition → replicas promote a new primary → two nodes accept writes → divergent datasets that can't auto-merge.

Solutions:

  • Quorum election (Raft/Paxos) — leader needs majority votes. A 2/3 split means only the 3-node side can elect. Gold standard.
  • Fencing tokens — monotonically increasing epoch. Storage rejects writes from stale leaders with older tokens.
  • STONITH — forcibly power off old primary via IPMI / cloud API. Brutal but effective with hardware control.
  • Lease-based leadership — leader holds a time-limited lease; if it can't renew (partition), it stops accepting writes.
  • Read-only mode — during ambiguity, reject all writes until partition heals. Sacrifices availability for correctness.

3. Failover Complexity

Problem: Primary dies. Must detect, choose best replica, promote, redirect clients, ensure no writes lost. Wrong → data loss or minutes of errors.

Solutions:

  • Patroni / Orchestrator — auto-detect failure, promote most up-to-date replica, update DNS/VIP. Failover in 10-30s.
  • Cloud-managed — AWS RDS, Cloud SQL handle failover automatically. Less control, zero ops burden. 30-60s.
  • Health-aware connection pooling — PgBouncer/ProxySQL detects unreachable primary, reroutes to new one.
  • Deterministic replica selection — always promote the replica with smallest lag. Semi-sync replica is guaranteed current.
  • Runbooks + drills — step-by-step manual failover, practiced quarterly. Better than buggy automation.

4. Write Amplification

Problem: With 3 replicas, every write is performed 4x (1 primary + 3 replicas). 4x disk I/O, 4x bandwidth. Writes do NOT scale with more replicas.

Solutions:

  • Accept it — 3-5x amplification is an acceptable cost for availability + read scaling in most systems.
  • Async replication — primary doesn't wait for replicas, so write latency is unaffected. Replicas catch up in background.
  • Right-size replica count — if you only need 2 for HA, drop from 5. 2x vs 5x amplification.
  • Batch WAL shipping — ship WAL segments in batches to amortize network overhead.
  • When writes bottleneck, shard — replication can't scale writes. That's sharding's job.

5. Conflict Resolution (Multi-Leader)

Problem: Two leaders accept conflicting writes to the same row. User A sets price to $10, User B sets $15. Which wins?

Solutions:

  • Avoid multi-leader — single-leader eliminates conflicts by design. Only use multi-leader when you need writes in multiple regions.
  • Last-write-wins (LWW) — highest timestamp wins. Simple but lossy. OK for non-critical data (last-seen time).
  • CRDTs — data structures that merge without conflict (G-Counter, PN-Counter, LWW-Register). Limited to specific types.
  • App-level merge — surface both versions to the app or user. Most flexible, most complex (Google Docs approach).
  • Conflict-free by design — partition writes so each row has one “home” leader. Leader-US for US data, Leader-EU for EU data.

6. Consistency vs Latency Tradeoff

Problem: Sync replication = strong consistency + high latency. Async = fast + stale reads. Can't have both (PACELC).

Solutions:

  • Per-query tuning — read_strong() goes to primary, read_eventual() to any replica. Each feature chooses.
  • Semi-sync — wait for 1 replica ACK (~10-20ms), keep others async for read scaling.
  • Quorum reads/writes — W + R > N guarantees freshness. W=2, R=2 with N=3 gives consistency. Dynamo/Cassandra model.
  • 90/10 rule — browsing, search, recommendations tolerate staleness. Cart, checkout, payment read from primary.

7. Storage Cost

Problem: N replicas = Nx storage. 1 TB with 3 replicas = 4 TB. Trivial at small scale, significant at petabyte scale.

Solutions:

  • Right-size replicas — 2-3 for HA is usually enough. Replace read-scaling replicas with a cache layer (Redis).
  • Tiered storage — hot data on SSDs, cold replicas on cheaper HDDs or cloud cold storage.
  • Partial replicas — replicate only the tables an analytics replica needs, not the full database.
  • Compression — storage-level compression on read-heavy replicas. 50-70% reduction, minimal CPU cost.

8. Cascade Failure During Failover

Problem: Primary dies, traffic redistributes to fewer nodes. 60% utilization → 90%. New primary must handle writes + replication + rebuild. Remaining nodes can be overwhelmed.

Solutions:

  • Over-provision — size each node for 100% load. ~33% utilization normally, but survives any single failure gracefully.
  • Load shedding — disable non-critical jobs, serve staler cache, degrade gracefully rather than fall over.
  • Throttle rebuilds — limit I/O for new replica sync. Slow steady rebuild > fast rebuild that starves live traffic.
  • Auto-scale replicas — cloud auto-provisions replacement in minutes, shrinking the degraded window.
  • Circuit breaker — if replicas overload, route reads to cache or return degraded results instead of queueing.

The Standard Database Scaling Progression

When an interviewer asks “how would you scale the database?”, the standard progression is:

  1. Vertical scaling — bigger machine. Buys time. No code changes. Hits a ceiling.
  2. Read replicas — handles read-heavy workloads. Simple, well-understood.
  3. Caching layer — Redis/Memcached offloads repeated reads. Reduces DB load by 90%+ for common queries.
  4. Sharding — when writes or storage hit the ceiling that replication and caching can't solve.
  5. Each shard gets replicas — combines both strategies for write scale + read scale + availability.

The interview one-liner: Replication copies data for safety and read speed. Sharding splits data for write speed and storage. Production systems at scale do both — every shard is replicated.

Key Takeaways

  • Replicate for availability and read scaling. Read replicas are the first tool for most databases. They add redundancy and can absorb read traffic. Simple, well-understood, low operational burden.
  • Shard for data size and write scaling. Sharding is the answer when data does not fit on one machine or when write throughput exceeds a single node's capacity. It is not the answer for read scaling (that is replication) or for availability (also replication).
  • Always do the math first. Our 50 GB dataset with 300 reads/sec does not justify the complexity of sharding. Calculate your data size, read QPS, and write QPS before reaching for distributed architectures.
  • Premature sharding is expensive. Every query must know which shard to hit. Cross-shard queries require scatter-gather. Schema migrations must be coordinated. Do not pay this cost until the numbers force you.
  • Every solution creates new problems. Sharding introduces cross-shard query complexity, resharding pain, and operational overhead. Replication introduces lag, split-brain risk, and failover complexity. Knowing the problems — and their mitigations — is what separates senior from junior answers in interviews.
  • The right answer can be "keep it simple." In an interview, demonstrating that you know when NOT to use a complex technique is often more impressive than using it. Show the math, show that a single server (with replicas) handles the load, and explain when you would revisit that decision.
