Pub/Sub with Ordering Guarantees: Chat System

Delivering 500K messages/sec across 100 million conversations with strict per-conversation ordering — while handling consumer rebalancing, hot partitions, and duplicate delivery gracefully.

NALSD Walkthrough • ~15 min read

The Scenario

You are designing the messaging backend for a chat application. The core invariant: messages within a single conversation must be delivered to all participants in the exact order they were sent. If Alice sends "Hello" then "How are you?", every participant must see those messages in that order, never reversed.

Concrete Numbers

MetricValue
Concurrent users10 million
Messages per second (platform-wide)500,000
Average message size200 bytes
Active conversations100 million
Average participants per conversation2–5
Ordering requirementStrict order within a conversation
Cross-conversation orderingNot required

Why Ordering Matters

In a chat application, out-of-order messages break the conversation. If Bob asks "Where should we eat?" and Carol replies "Italian sounds good!", but the messages arrive reversed, it looks like Carol randomly says "Italian sounds good!" and then Bob asks "Where should we eat?" The conversation is nonsensical.

Ordering is easy with a single server: messages arrive sequentially and are stored/forwarded in order. The challenge is maintaining order when the system is distributed across multiple servers for scalability.

Back-of-Envelope Math

Throughput

Message throughput:
  500,000 msgs/sec × 200 bytes = 100 MB/sec inbound

Delivery throughput (each message goes to all participants):
  Average 3 participants per conversation.
  500K msgs/sec × 3 recipients = 1,500,000 deliveries/sec

If each delivery is a WebSocket push of 200 bytes:
  1.5M × 200 bytes = 300 MB/sec outbound

Kafka Partition Sizing

Kafka provides ordering WITHIN a partition, not across partitions.

To guarantee per-conversation ordering:
  All messages for conversation X must go to the same partition.
  Partition key = conversation_id
  Kafka routes: partition = hash(conversation_id) % num_partitions

How many partitions?

  Option A: 100 partitions
    500K msgs/sec / 100 = 5,000 msgs/sec per partition
    Each partition handles: 100M conversations / 100 = 1M conversations
    Kafka per-partition throughput limit: 10K-100K msgs/sec → fine

  Option B: 1,000 partitions
    500K / 1,000 = 500 msgs/sec per partition
    Each handles 100K conversations → more headroom

  Going with 1,000 partitions for safety and parallelism.

Consumer instances:
  100 consumer instances (each handles 10 partitions)
  Each consumer: 500K / 100 = 5,000 msgs/sec
  Each consumer delivers to: 5,000 × 3 = 15,000 WebSocket pushes/sec

Storage

Message storage (for history):
  500K msgs/sec × 200 bytes × 86,400 sec = 8.64 TB/day

With 30-day retention:
  8.64 TB × 30 = 259.2 TB

Indexed by (conversation_id, sequence_number) for retrieval.
Sharded by conversation_id across a distributed DB
(Cassandra or DynamoDB).

WebSocket Server Sizing

10M concurrent connections.
Each WebSocket server handles ~50,000 connections.
  10M / 50K = 200 WebSocket servers.

Deliveries per WebSocket server:
  1.5M deliveries/sec / 200 servers = 7,500 deliveries/sec per server.
  At 200 bytes each: 1.5 MB/sec outbound per server. Trivial.

Naive Design: Single Kafka Topic with Partition Key

How Ordering Works

All messages for conversation "conv_42" hash to partition 7.

  Kafka partition 7 (in order):
    offset 0: {conv: "conv_42", seq: 1, sender: "Alice", text: "Hello"}
    offset 1: {conv: "conv_99", seq: 5, sender: "Dave",  text: "OK"}
    offset 2: {conv: "conv_42", seq: 2, sender: "Bob",   text: "Hi!"}
    offset 3: {conv: "conv_42", seq: 3, sender: "Alice", text: "How are you?"}

  Consumer 1 reads partition 7 in offset order:
    → Processes conv_42 seq 1, then conv_99 seq 5, then conv_42 seq 2, ...
    → For conv_42: messages arrive in order 1, 2, 3. Correct!

  Different conversations on the same partition are interleaved,
  but WITHIN a conversation, order is preserved because:
    1. Kafka preserves order within a partition.
    2. All messages for a conversation go to the same partition.
    3. The consumer reads sequentially from the partition.

Sequence Number Assignment

import time, uuid

# Per-conversation sequence counter (in Redis)
def assign_sequence(conversation_id, message):
    """Atomically assign the next sequence number for this conversation."""
    seq = redis_client.incr(f"seq:{conversation_id}")
    message['sequence_number'] = seq
    message['message_id'] = str(uuid.uuid4())
    message['timestamp'] = time.time()
    return message

def send_message(conversation_id, sender_id, text):
    message = {
        'conversation_id': conversation_id,
        'sender_id': sender_id,
        'text': text,
    }
    message = assign_sequence(conversation_id, message)

    # Publish to Kafka with conversation_id as partition key
    kafka_producer.send(
        topic='chat-messages',
        key=conversation_id.encode(),  # determines partition
        value=json.dumps(message).encode()
    )

This design works correctly for the common case. Messages are ordered within each conversation, consumers process them sequentially, and WebSocket servers deliver them to recipients.

What Breaks

Problem 1: Consumer Rebalancing Causes Downtime and Duplicates

When a consumer crashes or a new consumer joins the group, Kafka triggers a rebalance: it reassigns partitions among the remaining consumers. During rebalancing, affected partitions are not consumed. Messages queue up.

Rebalance scenario: Consumer 3 crashes.

  Before crash:
    Consumer 1: partitions 0-9     (handles 5K msgs/sec)
    Consumer 2: partitions 10-19   (handles 5K msgs/sec)
    Consumer 3: partitions 20-29   (handles 5K msgs/sec)  ← CRASHES
    ...

  Rebalance triggered (takes 5-30 seconds with eager assignor):
    t=0s    Consumer 3 crashes.
    t=3s    Kafka detects missing heartbeat (session.timeout.ms = 10s).
    t=10s   Consumer 3 declared dead. Rebalance starts.
            ALL consumers stop consuming during rebalance.
    t=15s   Rebalance completes. Partitions 20-29 assigned to others.
            Consumption resumes.

  During 15-second rebalance:
    ALL 1,000 partitions paused (eager rebalance stops everything).
    Messages queued: 500K msgs/sec × 15s = 7.5M messages.

  After rebalance:
    Consumers race to drain 7.5M backlog.
    Users experience 15+ seconds of no messages, then a burst.

Problem 2: Duplicate Messages on Crash

Duplicate delivery scenario:

  t=0ms    Consumer 3 reads message M1 from partition 22, offset 100.
  t=5ms    Consumer 3 pushes M1 to WebSocket server for delivery.
  t=6ms    WebSocket server delivers M1 to recipients. ✓
  t=7ms    Consumer 3 about to commit offset 101 to Kafka...
  t=7ms    CRASH. Offset 100 is the last committed offset.

  After rebalance:
  t=15s    Consumer 5 is assigned partition 22.
           Reads from last committed offset: 100.
           Reads message M1 AGAIN.
           Pushes M1 to WebSocket server.
           Recipients see M1 TWICE.

  In a chat app: "Hello" appears twice. Annoying but not catastrophic.
  In a payment system: double-charge. Much worse.

Problem 3: Hot Partitions from Popular Group Chats

A group chat with 1,000 members generates 100 msgs/sec.
All 100 msgs/sec go to ONE partition (same conversation_id).

The consumer assigned to that partition must:
  1. Read 100 msgs/sec from Kafka (easy).
  2. For each message, deliver to 1,000 recipients via WebSocket.
     100 msgs/sec × 1,000 recipients = 100,000 deliveries/sec.
  3. The consumer's other 9 partitions also need processing:
     9 × 500 = 4,500 msgs/sec of normal traffic.

Total for this consumer: 4,500 + 100 = 4,600 msgs/sec
But delivery load: 4,500 × 3 + 100 × 1,000 = 113,500 deliveries/sec
vs a normal consumer: 5,000 × 3 = 15,000 deliveries/sec

This consumer is 7.5x more loaded than its peers.
It falls behind. Messages in the hot conversation are delayed.
Messages in 9 other normal partitions are ALSO delayed
(collateral damage from sharing a consumer).

Problem 4: Sequence Number Gaps

If the API server assigns sequence number 5, publishes to Kafka,
but the Kafka publish fails (broker down), the client retries
and gets sequence number 6. The conversation now has:
  seq 1, 2, 3, 4, 6 (seq 5 is missing).

Clients that buffer by sequence number will wait forever for seq 5.
The conversation appears "stuck" on the recipient's device.
Summary of problems: (1) Rebalancing pauses ALL consumption for seconds, (2) crashes cause duplicate deliveries, (3) hot conversations overload a single consumer, (4) sequence number gaps can freeze the client.

Iteration 1: Client-Side Deduplication and Cooperative Rebalancing

Client-Side Deduplication

Since at-least-once delivery means duplicates will happen, the client must handle them. Each message has a unique message_id. The client tracks the last N message IDs it has seen for each conversation and silently discards duplicates.

# Client-side (JavaScript pseudocode)
class ChatClient:
    def __init__(self):
        # Track last 1000 message IDs per conversation
        self.seen_messages = {}  # conversation_id -> set of message_ids

    def on_message_received(self, message):
        conv_id = message['conversation_id']
        msg_id = message['message_id']

        # Initialize set for this conversation if needed
        if conv_id not in self.seen_messages:
            self.seen_messages[conv_id] = OrderedSet(max_size=1000)

        # Check for duplicate
        if msg_id in self.seen_messages[conv_id]:
            return  # duplicate, ignore

        # Not a duplicate — add to seen set and display
        self.seen_messages[conv_id].add(msg_id)
        self.display_message(message)

Client-Side Reordering Buffer

For handling out-of-order delivery (which can happen briefly during rebalancing), the client maintains a small reorder buffer per conversation.

Reordering example:

  Client expects next sequence number: 5

  Messages arrive: seq 5, seq 7, seq 6

  Processing:
    Receive seq 5: expected=5, match! Display. Next expected: 6.
    Receive seq 7: expected=6, mismatch. Buffer seq 7.
                   Buffer: {7: msg}
    Receive seq 6: expected=6, match! Display. Next expected: 7.
                   Check buffer: seq 7 exists! Display. Next expected: 8.
                   Buffer: {} (empty)

  Result: User sees messages in order 5, 6, 7. Correct!

  Timeout: If seq 6 never arrives after 2 seconds, display
  seq 7 anyway (showing a gap is better than waiting forever).
  Request missed messages via HTTP fallback.

Cooperative Sticky Rebalancing

The default Kafka rebalance protocol (eager) stops ALL consumers during rebalancing. The cooperative sticky assignor performs incremental rebalances: only the affected partitions are revoked and reassigned. Other partitions continue being consumed without interruption.

Eager rebalance (default):
  Consumer 3 crashes.
  ALL 100 consumers stop. ALL 1,000 partitions paused.
  15 seconds of total downtime.
  7.5M messages queued.

Cooperative sticky rebalance:
  Consumer 3 crashes.
  Only partitions 20-29 (Consumer 3's partitions) are revoked.
  Consumers 1, 2, 4-100 continue processing normally.
  Partitions 20-29 are reassigned to other consumers.
  Only 10 out of 1,000 partitions affected.

  Impact during rebalance:
    10 partitions paused × 500 msgs/sec × 5 seconds = 25,000 messages queued.
    vs 7,500,000 with eager rebalance.
    Reduction: 99.7%

  Users in 990 unaffected partitions: zero disruption.
  Users in 10 affected partitions: ~5 second delay.

Kafka Consumer Configuration

Kafka consumer configuration:

  partition.assignment.strategy = CooperativeStickyAssignor
  session.timeout.ms = 10000      # 10s until declared dead
  heartbeat.interval.ms = 3000    # heartbeat every 3s
  max.poll.interval.ms = 300000   # max time between polls

Handling Sequence Number Gaps

Instead of assigning sequence numbers BEFORE publishing to Kafka, we use Kafka's own offset as the ordering mechanism. Within a partition, offsets are sequential and gap-free. The per-conversation sequence number is assigned AFTER the message is durably stored in Kafka, using the Kafka offset to derive ordering.

Revised approach:

  1. API server publishes message to Kafka (no sequence number yet).
  2. Kafka assigns offset 4,721 to the message on partition 7.
  3. Consumer reads from partition 7 in offset order.
  4. For each message, consumer assigns a per-conversation sequence number:
     SELECT MAX(sequence_number) FROM messages
       WHERE conversation_id = 'conv_42'
     → Returns 4. Assign seq 5 to this message.
  5. Store message with seq=5 in the messages DB.
  6. Deliver to recipients with seq=5.

  Why this is better: Kafka offsets are always sequential and
  gap-free within a partition. No message is ever "lost" in
  the ordering sequence because the consumer processes them
  in exact offset order.

Iteration 2: Solving Hot Partitions with Fan-Out Separation

The Problem Restated

A group chat with 1,000 members produces 100 msgs/sec. Each message must be delivered to 1,000 WebSocket connections. The Kafka consumer that processes this partition is responsible for both reading messages in order AND delivering them to recipients. The delivery work (100K pushes/sec) dwarfs the ordering work (100 msgs/sec).

Solution: Separate Ordering from Delivery

Split the consumer's job into two stages:

  1. Ordering consumer: Reads from Kafka partition in order. Assigns sequence numbers. Stores messages in the database. Publishes to a per-conversation delivery channel. This is lightweight (no fan-out).
  2. Delivery service: Reads from delivery channels. Looks up conversation participants. Pushes to each participant's WebSocket connection. This is parallelized and scales independently.

Why This Solves the Hot Partition Problem

Before (single-stage):
  Consumer for partition 7:
    Read 4,600 msgs/sec from Kafka           ← CPU: low
    Deliver 113,500 pushes/sec via WebSocket  ← CPU: HIGH (bottleneck)
    One consumer does everything. Overloaded.

After (two-stage):
  Ordering consumer for partition 7:
    Read 4,600 msgs/sec from Kafka           ← CPU: low
    Publish 4,600 events to Redis Pub/Sub    ← CPU: low
    Total: very lightweight. No bottleneck.

  Delivery service (pool of N instances):
    Subscribe to delivery channels.
    Process fan-out in parallel.
    Hot conversation (conv_42): 100 msgs × 1,000 recipients
      = 100K deliveries/sec, spread across multiple delivery instances.
    Each delivery instance handles a subset of recipients.

  The ordering guarantee is maintained because:
    1. Stage 1 publishes messages in order (sequential per partition).
    2. Redis Pub/Sub delivers in order per channel.
    3. Each recipient receives messages in the order published.

Handling Very Hot Conversations

For extremely popular group chats (10K+ members), even the delivery fan-out per message is expensive. Additional optimizations:

Optimization 1: Batch deliveries
  Instead of pushing each message individually,
  batch messages in 100ms windows.
  10 messages in 100ms × 10K recipients = 10K batch pushes
  vs 10 × 10K = 100K individual pushes.
  Each push carries 10 messages instead of 1.

Optimization 2: Tree-based fan-out
  Instead of one service sending to 10K WebSocket servers:

  Level 1: Delivery coordinator sends to 10 fan-out nodes.
  Level 2: Each fan-out node sends to 1,000 WebSocket servers.

  Latency: 2 hops instead of 1, but coordinator is not a bottleneck.
  Parallel delivery at each level.

Optimization 3: Server-side subscription
  For a 10K-member group, most members are offline.
  Only push to currently connected members.
  If 10% are online: 1,000 pushes instead of 10,000.
  Offline members pull missed messages on reconnect.

Consumer Lag Monitoring

If a consumer falls behind (processing slower than message arrival), messages queue up in Kafka. We need to detect this early.

Consumer lag = Latest offset in partition - Consumer's committed offset

Example:
  Partition 7 latest offset: 1,000,000
  Consumer's committed offset: 999,500
  Lag: 500 messages

  At 500 msgs/sec per partition, 500 messages = 1 second of lag.
  Acceptable.

Alert thresholds:
  Lag > 5,000 (10 seconds):    WARNING — consumer may be slow
  Lag > 50,000 (100 seconds):  CRITICAL — messages delayed by 1.5+ minutes
  Lag > 500,000 (1000 seconds): EMERGENCY — 16+ minutes behind

  Action on CRITICAL: add more consumer instances (rebalance
  distributes partitions among more consumers) or investigate
  the slow consumer (is the delivery service overwhelmed? is
  the DB slow?).

Iteration 3: Message History and Offline Delivery

Persistent Message Storage

Kafka provides temporary storage (retention period), but chat messages need permanent storage for history (scroll up to see old messages). The ordering consumer writes every message to a persistent store, indexed by conversation and sequence number.

Messages table (Cassandra, partitioned by conversation_id):

  Partition key: conversation_id
  Clustering key: sequence_number (ascending)

  ┌─────────────────┬───────┬──────────┬──────────────────┬───────────┐
  │ conversation_id │ seq   │ sender   │ text             │ timestamp │
  ├─────────────────┼───────┼──────────┼──────────────────┼───────────┤
  │ conv_42         │ 1     │ alice    │ Hello            │ 10:00:01  │
  │ conv_42         │ 2     │ bob      │ Hi!              │ 10:00:03  │
  │ conv_42         │ 3     │ alice    │ How are you?     │ 10:00:05  │
  │ conv_42         │ 4     │ bob      │ Good, you?       │ 10:00:08  │
  │ conv_42         │ 5     │ alice    │ Great!           │ 10:00:10  │
  └─────────────────┴───────┴──────────┴──────────────────┴───────────┘

Query for history (scrolling up):
  SELECT * FROM messages
  WHERE conversation_id = 'conv_42'
  AND sequence_number < 5
  ORDER BY sequence_number DESC
  LIMIT 20

  Returns: seq 4, 3, 2, 1 (reverse order for display).
  Cassandra returns this efficiently because sequence_number
  is the clustering key.

Offline Message Delivery

When a user is offline, they miss real-time WebSocket pushes. On reconnect, the client must fetch all messages it missed.

Client reconnection flow:

  1. Client connects via WebSocket.
  2. Client sends: {"type": "sync", "conversations": {
       "conv_42": {"last_seq": 3},
       "conv_99": {"last_seq": 15},
       ...
     }}
  3. Server queries for each conversation:
     SELECT * FROM messages
     WHERE conversation_id = 'conv_42'
     AND sequence_number > 3
     ORDER BY sequence_number ASC
     → Returns: seq 4, 5 (missed messages while offline)
  4. Server pushes missed messages to client.
  5. Client applies deduplication and reordering (in case
     some messages arrived via push before disconnect).

Optimization: If user was offline for a long time (>7 days),
  don't sync all conversations. Only sync when user opens
  a specific conversation (lazy loading).

Read Receipts and Typing Indicators

These are high-frequency, low-importance signals. They do NOT need ordering guarantees or persistence.

Typing indicators:
  - Sent as fire-and-forget UDP-style messages.
  - No Kafka, no persistence.
  - Direct WebSocket push to conversation participants.
  - If lost: no harm (user just doesn't see "Alice is typing...").
  - Rate limited: max 1 typing event per 3 seconds per user.

Read receipts:
  - Slightly more important (want to persist "Alice read up to seq 5").
  - Store in DB: UPDATE read_receipts
    SET last_read_seq = 5 WHERE user_id = 'alice' AND conv_id = 'conv_42'
  - Broadcast to other participants via WebSocket.
  - Don't need Kafka ordering — last-write-wins is fine.
  - If duplicate: idempotent (max of current and new sequence number).

Key Takeaways

  • Partition-level ordering is Kafka's fundamental guarantee. If you need messages in order, route related messages to the same partition via a consistent partition key. Do not rely on cross-partition ordering — it does not exist.
  • At-least-once delivery means duplicates are inevitable. Design for it. Client-side deduplication using unique message IDs is simple and effective. Never assume exactly-once delivery from the infrastructure.
  • Cooperative sticky rebalancing is not optional for latency-sensitive applications. The default eager rebalance pauses ALL consumers. For a chat app, this means ALL users experience a 15-second blackout. Cooperative sticky limits the blast radius to only the affected partitions.
  • Separate ordering from delivery. Ordering is inherently sequential (must be done by one consumer per partition). Delivery is embarrassingly parallel (can be fanned out to many workers). Putting both in the same consumer makes the sequential part a bottleneck for the parallel part.
  • Hot partitions are unavoidable with partition-key routing. If one conversation generates 100x more traffic than average, its partition is 100x hotter. You cannot solve this by adding more partitions (all messages for that conversation MUST go to one partition). You solve it by making the per-partition consumer lightweight and offloading heavy work to a scalable layer.
  • Sequence numbers should be assigned after durable storage, not before. Assigning a sequence number before publishing to Kafka risks gaps (if the publish fails). Assigning after Kafka storage (using Kafka offsets for ordering) guarantees gap-free sequences.
  • Always calculate the rebalance impact. How many messages queue up? How long to drain? How many users are affected? These numbers determine whether your design meets latency requirements during operational events.
