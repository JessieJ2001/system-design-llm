Scenario #7: Caching Strategies

Three workloads—product catalog, session store, leaderboard—each demanding a different caching strategy. Cache-aside, write-through, and write-behind are not interchangeable. The numbers show you why.

~15 min read

The Scenario

You run a large e-commerce platform. Three distinct workloads need caching, but each has a fundamentally different access pattern:

Workload Data Size Read/Write Ratio Latency SLA Staleness Tolerance
A: Product Catalog 10M products, 5KB each = 50GB 95% reads / 5% writes <50ms p99 5-10 min OK
B: Session Store 5M sessions, 2KB each = 10GB 50% reads / 50% writes <10ms p99 0 (must never lose session)
C: Leaderboard Top-100 sellers, ~10KB total 99.99% reads / 0.01% writes <5ms p99 1 min OK

The question is not "should we cache?" (yes, always for these workloads). The question is: which caching pattern for each workload? Cache-aside? Write-through? Write-behind? The wrong choice causes stale data, cache misses, or data loss.

Back-of-Envelope Math

Workload A: Product Catalog

Total data:         10M products × 5KB = 50GB
Hot products (20%): 2M products × 5KB = 10GB  ← fits in one Redis instance (26GB max)

Read QPS:           5,000/sec peak
Write QPS:          ~250/sec (5% of traffic, price/description updates)

Without cache:
  DB read latency:  5-20ms per query
  At 5,000 QPS:     DB can handle it, but p99 at 50ms+ under load

With cache (10GB, 80% hit rate):
  Cache hits:       5,000 × 0.80 = 4,000/sec from Redis (sub-1ms)
  Cache misses:     5,000 × 0.20 = 1,000/sec from DB
  DB load reduced:  5× less traffic
  p99 latency:      ~2ms (most reads from cache)

Workload B: Session Store

Total data:         5M sessions × 2KB = 10GB
All data is hot:    Every active session is accessed

Request pattern:    Every page view = 1 read (load session)
                    Every action  = 1 write (update session)
                    5M users × 10 page views/hour = 50M requests/hour
                    = ~14,000 requests/sec
                    Split: ~7,000 reads/sec + ~7,000 writes/sec

Without cache:
  DB handles 14K ops/sec — possible but expensive (need beefy DB)
  Every page view waits for DB: 5-20ms latency

With cache:
  Redis handles 100K+ ops/sec easily
  14K ops/sec = 14% of Redis capacity
  Sub-1ms latency for all operations

Workload C: Leaderboard

Total data:         Top-100 sellers × ~100 bytes = 10KB
Read QPS:           10,000/sec (homepage widget)
Write QPS:          1/min (batch computation)

Without cache:
  10K SQL queries/sec to compute top-100 = absurd
  The query itself (GROUP BY, ORDER BY across millions of sellers) takes 500ms+

With cache:
  Compute once per minute, store result in Redis
  10K reads/sec × 10KB = 100MB/sec from cache
  One Redis instance handles this trivially
  Write: 1 per minute (overwrite the cached leaderboard)

Naive Design: Cache-Aside for Everything

Cache-aside (also called "lazy loading") is the most common pattern. It works like this:

Read Path

Read:
  1. Check cache for key
  2. If HIT:  return cached value
  3. If MISS: read from DB, write to cache, return value

Write:
  1. Write to DB
  2. Invalidate (delete) the cache entry
  (Next read will be a cache miss, which reloads from DB)

Code Example

# Cache-aside pattern (generic)
def get_product(product_id):
    # Step 1: Check cache
    cached = redis.get(f"product:{product_id}")
    if cached:
        return json.loads(cached)  # Cache HIT

    # Step 2: Cache MISS - read from DB
    product = db.query("SELECT * FROM products WHERE id = %s", [product_id])

    # Step 3: Populate cache with TTL
    redis.setex(f"product:{product_id}", 600, json.dumps(product))  # 10 min TTL

    return product

def update_product(product_id, updates):
    # Step 1: Write to DB
    db.execute("UPDATE products SET ... WHERE id = %s", [product_id])

    # Step 2: Invalidate cache (don't update - avoids race conditions)
    redis.delete(f"product:{product_id}")

What Breaks (With Numbers)

Cache-Aside Fails for Session Store (Workload B)

Problem: 50/50 read/write ratio means constant cache invalidation.

Timeline for one user session:
  T=0ms    User loads page → read session → CACHE MISS (not cached yet)
           → read from DB (5ms) → populate cache
  T=100ms  User clicks button → update session → write to DB + INVALIDATE cache
  T=200ms  User loads next page → read session → CACHE MISS (just invalidated!)
           → read from DB again (5ms) → populate cache
  T=300ms  User clicks button → update session → write to DB + INVALIDATE cache
  T=400ms  User loads next page → read session → CACHE MISS again!

Cache hit rate with cache-aside on 50/50 workload:
  Every write invalidates the cache.
  Next read is always a miss.
  Effective hit rate: ~0-10%  ← The cache is nearly useless!

  7,000 writes/sec each invalidate a cache entry
  7,000 reads/sec following those writes → 7,000 cache misses/sec
  DB still handles nearly all traffic → cache provides almost no benefit

Cache-aside is designed for read-heavy workloads. When writes are frequent, the constant invalidation-then-miss cycle makes the cache nearly useless. You need a pattern that keeps the cache warm after writes.

Cache-Aside Has Race Conditions

Race condition with cache-aside UPDATE (why we invalidate, not update):

  Thread A: reads product from DB (price = $10)
  Thread B: updates product in DB (price = $15)
  Thread B: writes to cache (price = $15)
  Thread A: writes to cache (price = $10)  ← STALE! Thread A's slower DB read
                                              overwrites Thread B's correct cache

  Cache now shows $10 (stale), DB has $15 (correct).
  This stale value persists until TTL expires.

  Fix: always INVALIDATE (delete) on write, never update.
  With invalidation, the worst case is an extra cache miss.

Leaderboard With Cache-Aside Is Wasteful

Cache-aside for the leaderboard:
  10K reads/sec check cache → usually HIT (great!)
  Every 60 seconds, the leaderboard is recomputed
  But cache-aside doesn't know about the recomputation
  Old leaderboard sits in cache until TTL expires

  If TTL = 60s, worst case = 60 seconds of stale data
  If leaderboard changes significantly, users see old rankings

  Better approach: proactively WRITE the new leaderboard to cache
  after each computation. No need for lazy loading at all.

Iteration 1: Right Strategy for Each Workload

Workload A (Product Catalog): Cache-Aside

Cache-aside is actually the right choice here. The 95/5 read/write ratio means cache entries stay warm for a long time between invalidations.

Why cache-aside works for product catalog:
  5,000 reads/sec vs 250 writes/sec
  Average time between writes for one product:
    10M products, 250 writes/sec = each product updated every ~40,000 sec (11 hours)
  Cache TTL: 10 minutes (600 seconds)
  Probability of write invalidating cache before TTL: 600/40000 = 1.5%

  So 98.5% of cache entries expire naturally via TTL, not invalidation
  Cache hit rate: 80-90% (dominated by hot products)
  Cache miss penalty: one DB read every ~10 min per hot product

Cache-Aside Code + Why Invalidate

# Product catalog: cache-aside with TTL safety net
PRODUCT_CACHE_TTL = 600  # 10 minutes

def get_product(product_id):
    cache_key = f"product:{product_id}"
    cached = redis.get(cache_key)
    if cached:
        return json.loads(cached)

    product = db.query("SELECT * FROM products WHERE id = %s", [product_id])
    if product:
        redis.setex(cache_key, PRODUCT_CACHE_TTL, json.dumps(product))
    return product

def update_product(product_id, updates):
    db.execute("UPDATE products SET ... WHERE id = %s", [product_id])
    redis.delete(f"product:{product_id}")  # Invalidate, don't update

Why invalidate, not update? Invalidation is safe against race conditions (see above). The next read will pay a cache miss (~5ms), but that is rare (once every 11 hours per product on average). The simplicity is worth the occasional extra milliseconds.

Now consider Workload B, where cache-aside fails. Write-through keeps the cache warm on every write.

Workload B (Session Store): Write-Through

Write-through writes to the cache AND the database on every write, synchronously. The cache is always warm because every write populates it.

Write-Through Flow

Write-through flow:
  Write:
    1. Write to cache
    2. Write to DB (synchronously)
    3. Return success only after both succeed

  Read:
    1. Read from cache (always a HIT, since all writes go to cache)
    2. If cache miss (cold start or eviction): read from DB, populate cache

Why this works for sessions:
  Every write populates the cache → next read is always a hit
  No invalidation-then-miss cycle
  Cache hit rate: ~100% (all data written to cache first)

  7,000 writes/sec: each writes to Redis (0.5ms) + DB (5ms) = 5.5ms total
  7,000 reads/sec: each reads from Redis (0.5ms) = 0.5ms total

Write-Through Code

# Session store: write-through pattern
def save_session(session_id, data):
    serialized = json.dumps(data)

    # Write to BOTH cache and DB (synchronous)
    redis.setex(f"session:{session_id}", 3600, serialized)  # 1hr TTL
    db.execute(
        "INSERT INTO sessions (id, data, updated_at) VALUES (%s, %s, NOW()) "
        "ON CONFLICT (id) DO UPDATE SET data = %s, updated_at = NOW()",
        [session_id, serialized, serialized]
    )
    return True

def get_session(session_id):
    # Read from cache (almost always a hit)
    cached = redis.get(f"session:{session_id}")
    if cached:
        return json.loads(cached)

    # Cache miss (cold start) — load from DB
    row = db.query("SELECT data FROM sessions WHERE id = %s", [session_id])
    if row:
        redis.setex(f"session:{session_id}", 3600, row.data)
        return json.loads(row.data)
    return None

Why not write-behind (async DB write) for sessions? Write-behind would be faster (skip the synchronous DB write), but if Redis crashes before the async flush to DB, the session data is lost. The user gets logged out. For critical session data, the synchronous DB write in write-through guarantees durability even if Redis fails.

Workload C (Leaderboard): Refresh-Ahead / Proactive Write

The leaderboard is computed once per minute by a batch job. We proactively write the result to cache. All reads come from cache. There is no lazy loading and no cache misses (except during the very first computation).

Refresh-Ahead Flow

Refresh-ahead flow:
  Every 60 seconds:
    1. Batch job queries DB: "SELECT seller_id, SUM(revenue) ... ORDER BY ... LIMIT 100"
    2. Job writes result to Redis: redis.set("leaderboard:top100", json.dumps(result))
    3. TTL: 120 seconds (2× refresh interval, safety net)

  Reads (10K/sec):
    1. Read from Redis: redis.get("leaderboard:top100")
    2. Always a hit (proactively populated)
    3. No DB query needed

Why this is optimal:
  DB queries:   1 per minute (the batch computation)
  Redis reads:  10K/sec × 10KB = 100MB/sec (trivial)
  Staleness:    max 60 seconds (acceptable for a leaderboard)
  DB load:      essentially zero for leaderboard reads

Refresh-Ahead Code

# Leaderboard: refresh-ahead with proactive cache write
import schedule

def compute_leaderboard():
    """Runs every 60 seconds via scheduler."""
    result = db.query("""
        SELECT seller_id, seller_name, SUM(revenue) as total_revenue
        FROM sales
        WHERE sale_date >= NOW() - INTERVAL '30 days'
        GROUP BY seller_id, seller_name
        ORDER BY total_revenue DESC
        LIMIT 100
    """)

    leaderboard = [
        {"rank": i + 1, "seller": row.seller_name, "revenue": float(row.total_revenue)}
        for i, row in enumerate(result)
    ]

    # Proactively write to cache
    redis.setex("leaderboard:top100", 120, json.dumps(leaderboard))  # 2min TTL
    return leaderboard

# Schedule: run every 60 seconds
schedule.every(60).seconds.do(compute_leaderboard)

def get_leaderboard():
    """Called 10K times/sec by the homepage."""
    cached = redis.get("leaderboard:top100")
    if cached:
        return json.loads(cached)

    # Fallback: compute on demand (only during cold start)
    return compute_leaderboard()

Iteration 2: Edge Cases and Hardening

Thundering Herd on Product Catalog

When a hot product's cache entry expires (or is invalidated), many concurrent requests all see a cache miss simultaneously and all hit the database at once.

Scenario: iPhone 16 product page, 500 reads/sec
  T=0:     Cache entry expires (TTL reached)
  T=1ms:   Request 1 → cache miss → query DB
  T=1ms:   Request 2 → cache miss → query DB
  T=1ms:   Request 3 → cache miss → query DB
  ...
  T=1ms:   Request 50 → cache miss → query DB
  (All 50 requests in the same millisecond hit the DB)

  DB receives 50 identical queries simultaneously
  If product query takes 20ms, DB processes 50 × 20ms of work
  For a very hot product, this can be 100+ simultaneous DB queries

Fix: Request Coalescing (Singleflight)

# Singleflight: only one request fetches from DB, others wait
import threading

_inflight = {}  # key → (condition, result)
_lock = threading.Lock()

def get_product_coalesced(product_id):
    cache_key = f"product:{product_id}"

    # Check cache first
    cached = redis.get(cache_key)
    if cached:
        return json.loads(cached)

    # Cache miss — check if another thread is already fetching
    with _lock:
        if cache_key in _inflight:
            condition, _ = _inflight[cache_key]
        else:
            condition = threading.Condition()
            _inflight[cache_key] = (condition, None)
            # This thread will do the fetch
            product = db.query("SELECT * FROM products WHERE id = %s", [product_id])
            redis.setex(cache_key, 600, json.dumps(product))
            _inflight[cache_key] = (condition, product)
            with condition:
                condition.notify_all()  # Wake up all waiting threads
            del _inflight[cache_key]
            return product

    # Another thread is fetching — wait for it
    with condition:
        condition.wait(timeout=5)
    _, result = _inflight.get(cache_key, (None, None))
    return result
Result with singleflight:
  T=0:     Cache expires
  T=1ms:   Request 1 → cache miss → starts DB query
  T=1ms:   Request 2 → cache miss → sees Request 1 in-flight → WAITS
  T=1ms:   Request 3 → cache miss → sees Request 1 in-flight → WAITS
  T=20ms:  Request 1 finishes → writes to cache → wakes all waiters
  T=20ms:  Requests 2-50 all get the result from cache

  DB queries: 1 (instead of 50)
  Thundering herd eliminated.

Session Store: What If Redis Goes Down?

With write-through, the DB always has the latest session data.
If Redis fails:
  1. Reads fall through to DB (higher latency: 5ms instead of 0.5ms)
  2. Writes go directly to DB (bypassing dead Redis)
  3. When Redis recovers, sessions are loaded on demand (cache-aside fallback)

No data loss. Users experience slightly higher latency (5ms vs 0.5ms)
but are never logged out. This is the advantage of write-through over
a cache-only approach (like storing sessions only in Redis).

Cache Warming on Deploy

Problem: After a Redis restart or deploy, the cache is empty.
  All 5,000 product reads/sec hit the DB simultaneously.
  DB load spikes from 1,000/sec (normal) to 5,000/sec (all misses).

Solution: Pre-warm the cache before routing traffic.
  1. Query DB for the top 10,000 most-accessed products
  2. Load them into Redis in a batch operation
  3. Takes ~30 seconds to warm 10K products
  4. Then start accepting traffic

  Warm cache: 80% hit rate from first request
  Cold cache: 0% hit rate, DB stampede

Final Design

Summary Comparison

Strategy Best For Read/Write Cache Always Warm? Data Loss Risk
Cache-aside Read-heavy, infrequent writes 90%+ reads No (cold after invalidation) None (DB is source of truth)
Write-through Balanced read/write, data critical 50/50 Yes (every write populates) None (sync DB write)
Write-behind Write-heavy, staleness OK 20%+ writes Yes Yes (async DB write)
Refresh-ahead Computed data, batch updated 99%+ reads Yes (proactively populated) None (DB computed)

Key Takeaways

  1. Cache-aside for read-heavy workloads with infrequent writes. Simple, safe, and effective when 90%+ of operations are reads. The occasional cache miss after a write is negligible.
  2. Write-through for balanced read/write where the cache must always be consistent. Every write populates the cache, so reads never miss. Pay the extra latency of a synchronous DB write for guaranteed durability.
  3. Write-behind for write-heavy workloads where some data loss is acceptable. Fastest writes (cache-only), but if the cache crashes before flushing, recent writes are lost. Use for non-critical counters or analytics.
  4. Refresh-ahead for pre-computed data. When a batch job produces the data, write it directly to cache. No lazy loading needed. Trivially simple and maximally efficient.
  5. Always invalidate, never update on write (for cache-aside). Updating the cache on write creates race conditions. Invalidation is safe: the worst case is one extra cache miss.
  6. Protect against thundering herds with request coalescing (singleflight). When a hot key expires, only one request should fetch from DB; all others wait for it.
  7. The right strategy depends on three factors: read/write ratio, staleness tolerance, and data criticality. There is no universal best strategy.
