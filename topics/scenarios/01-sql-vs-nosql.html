SQL vs NoSQL: Social Media News Feed

Choosing the right storage model for user profiles, follower graphs, and feed timelines — driven by concrete numbers, not dogma.

NALSD Walkthrough • ~15 min read

Operation Complexity: PostgreSQL vs Cassandra

Operation PostgreSQL Cassandra Why the difference
Point read by PK
WHERE id = 42		 O(log n) B-tree index lookup.
~0.5–2 ms		 O(1) to locate partition + O(1) to read one row.
~1–5 ms		 PostgreSQL walks a B-tree (depth ~3–4 for millions of rows). Cassandra hashes the partition key to find the exact node and SSTable — that’s the O(1) part. Once the partition is located, reading a single row within it is also O(1). The higher constant overhead (~1–5 ms vs ~0.5–2 ms) comes from the network hop to the correct node.
Range scan (sorted)
WHERE user_id = X ORDER BY time DESC LIMIT 50		 O(log n + k) index scan, k = rows returned.
~2–10 ms		 O(1) locate partition + O(k) read k rows sequentially.
~1–5 ms		 PostgreSQL finds the start via B-tree (O(log n)), then scans k rows. Cassandra first locates the partition via hash (O(1)), then reads k rows sequentially within it — rows are already sorted by clustering key on disk, so no sorting or tree traversal needed.
Insert (single row) O(log n) per index to update.
~2–5 ms (with fsync)		 O(1) append to memtable + commit log.
~0.5–2 ms		 PostgreSQL must update the heap + every index (B-tree insert each). Cassandra uses an LSM-tree: writes go to an in-memory memtable and an append-only commit log. No index update at write time — indexes are built during background compaction.
Batch insert (N rows) O(N log n) per index.
Scales linearly with indexes		 O(N) appends.
Near-constant per row		 This is why Cassandra excels at fan-out: 500 inserts cost ~500 × 0.1 ms = 50 ms. PostgreSQL would pay index update costs on each insert.
JOIN (multi-table)
posts JOIN follows		 O(n × m) worst case, but optimized with indexes to O(n log m).
~50–100 ms for feed query		 Not supported.
Must denormalize or do client-side.		 Cassandra has no JOIN — data must be pre-joined (denormalized) at write time. This is a fundamental trade-off: Cassandra sacrifices query flexibility for write/read speed on known access patterns.
Secondary index query
WHERE email = 'x'		 O(log n) B-tree scan.
~1–5 ms		 O(n) scatter-gather across all nodes.
~50–500 ms		 PostgreSQL indexes are local and efficient. Cassandra’s secondary indexes must query every node (the data is partitioned by a different key), making non-partition-key lookups very expensive.
Update (single row) O(log n) find + in-place update.
~2–5 ms		 O(1) append (new version).
~0.5–2 ms		 PostgreSQL does MVCC: creates a new row version, marks old one dead. Cassandra just writes a new timestamped entry; the latest timestamp wins at read time. Old versions cleaned up during compaction.
Delete O(log n) find + mark dead.
~2–5 ms		 O(1) write a tombstone.
~0.5–2 ms		 Cassandra can’t delete in place (data is on disk in immutable SSTables). Instead it writes a “tombstone” marker. The actual data is removed during compaction. Too many tombstones can slow reads.
Transaction (multi-row) Full ACID.
BEGIN / COMMIT / ROLLBACK		 Limited.
Lightweight transactions (CAS) on a single partition only.		 PostgreSQL supports arbitrary multi-table transactions. Cassandra only offers compare-and-set on a single partition via Paxos — much slower (~30 ms) and no cross-partition atomicity. This is why we keep user profiles in PostgreSQL.
Horizontal scaling Hard. Read replicas easy, write scaling requires sharding (manual). Built-in. Add nodes, data auto-rebalances. Cassandra was designed for horizontal scaling from day one (consistent hashing, no single primary). PostgreSQL scaling requires external tools (Citus, manual sharding).

Key takeaway: PostgreSQL wins when you need flexible queries (JOINs, secondary indexes, transactions). Cassandra wins when you know your access patterns upfront and need high write throughput + fast partition-key reads at scale. That’s why we use both: PostgreSQL for profiles and follows (flexible, transactional), Cassandra for timelines (known pattern, high write volume).

The Scenario

You are designing the storage layer for a social media application. The platform has three core data types and two dominant access patterns:

Data

  • User profiles — 50 million registered users. Each profile contains name, bio, email, settings (~1 KB per record).
  • Posts — 500 million total posts. Each post contains author ID, text content, timestamp, media URLs (~500 bytes per record).
  • Follower relationships — each user follows an average of 200 other users. That is 50M × 200 = 10 billion directed edges. Each edge is two 8-byte user IDs = 16 bytes.

Access Patterns

  • Profile lookups — get a user by user_id. Simple key-value read. Needs ACID for account operations (email change, password reset).
  • News feed generation — for a given user, show the latest 50 posts from all users they follow. This is the hard query.

Back-of-Envelope Math

Before choosing any technology, let us calculate raw storage sizes and the cost of the feed query. These numbers will drive every decision. (Reference: BOE cheat sheet.)

Storage Requirements

  50M users × 1 KB/user            = 50 GB

Posts:
  500M posts × 500 B/post          = 250 GB

Follower edges:
  50M users × 200 avg follows      = 10 billion edges
  10B edges × 16 B/edge            = 160 GB

Total raw storage:                  ≈ 460 GB
  (fits on one beefy server's SSDs, but we'll need
   multiple copies for availability)

Feed Query Cost (The Critical Calculation)

When user Alice opens her feed, the system must find the latest 50 posts from all users she follows. Alice follows 200 users on average.

Naive SQL approach:
  SELECT p.* FROM posts p
  JOIN follows f ON p.author_id = f.followed_id
  WHERE f.follower_id = ?
  ORDER BY p.timestamp DESC
  LIMIT 50;

What this query does under the hood:
  1. Look up Alice's 200 followed user IDs (index scan on follows)
  2. For each of 200 users, find their recent posts
     Avg user has 10 posts (500M / 50M), but post counts follow
     a power-law distribution: most users post rarely, while a
     small fraction post heavily. The top 10% of active users
     typically account for 80%+ of content, putting them at 100+
     posts each. Assume we scan ~50 recent posts per user.
  3. Rows touched: 200 users × 50 posts = 10,000 rows
  4. Sort 10,000 rows by timestamp, return top 50

Per-request cost:  ~10,000 row reads + sort
Peak QPS:          1,500 feed requests/sec
Total row reads:   1,500 × 10,000 = 15,000,000 rows/sec

15 million row reads per second is the number that kills the naive design. A single PostgreSQL instance (one server on one machine — internally multi-process, with one backend process per connection sharing the same memory and disk) can handle roughly 50,000 simple indexed reads per second on modern hardware (multi-core CPU, SSDs, working set fitting in RAM). A “simple indexed read” is a point lookup that hits an index and returns one row — e.g., SELECT * FROM users WHERE id = 42. The database looks up 42 in the B-tree index (a few page reads), jumps straight to that row, and returns it. No joins, no sorting, no scanning thousands of rows. But this is not a simple indexed read — it is a multi-table join with a sort. Realistically, PostgreSQL can handle maybe 100–200 of these complex feed queries per second.

Naive Design: Everything in PostgreSQL

Start with the simplest thing that could work. One PostgreSQL instance with three tables:

-- Users table: 50M rows, ~50 GB
CREATE TABLE users (
  user_id   BIGINT PRIMARY KEY,
  username  VARCHAR(64) UNIQUE,
  email     VARCHAR(256),
  bio       TEXT,
  created_at TIMESTAMP DEFAULT NOW()
);

-- Posts table: 500M rows, ~250 GB
CREATE TABLE posts (
  post_id    BIGINT PRIMARY KEY,
  author_id  BIGINT REFERENCES users(user_id),
  content    TEXT,
  media_urls TEXT[],
  created_at TIMESTAMP DEFAULT NOW()
);
CREATE INDEX idx_posts_author_time ON posts(author_id, created_at DESC);

-- Follows table: 10B rows, ~160 GB
CREATE TABLE follows (
  follower_id  BIGINT REFERENCES users(user_id),
  followed_id  BIGINT REFERENCES users(user_id),
  created_at   TIMESTAMP DEFAULT NOW(),
  PRIMARY KEY (follower_id, followed_id)
);
CREATE INDEX idx_follows_follower ON follows(follower_id);

What does REFERENCES mean? It’s the SQL keyword that creates a foreign key constraint. Writing author_id BIGINT REFERENCES users(user_id) is shorthand for FOREIGN KEY (author_id) REFERENCES users(user_id). It tells the database: “this column must always point to a valid row in the referenced table.” The database will reject any insert or update that would break this link.

This is clean, normalized, and relational. Profile lookups are a simple primary key read — fast. Follow/unfollow is a single INSERT/DELETE with ACID guarantees — correct. The only problem is the feed query.

What Breaks

The feed query is the bottleneck. Let us prove it with numbers.

Feed Query Performance

Single feed query cost:
  - Index lookup on follows: ~200 rows (Alice's followed users)
  - For each followed user, index scan on posts(author_id, created_at):
    ~50 rows per user × 200 users = 10,000 rows
  - Sort 10,000 rows, return top 50
  - Estimated execution time: 50-100 ms on warm cache

Peak load:
  - 1,500 feed queries/sec
  - Each query: 50-100 ms of CPU + I/O time
  - A PostgreSQL connection can handle: 1000ms / 75ms avg = ~13 queries/sec
  - Connections needed: 1,500 / 13 = ~115 concurrent connections
  - PostgreSQL practical limit: ~100-200 connections
    (each connection ≈ 10 MB RAM for work_mem, sort buffers)

Result: A single PostgreSQL server is at or beyond its limit
  just for feed queries alone, ignoring all other traffic.

Can We Just Add Read Replicas?

Each replica handles ~13 feed queries/sec (same as primary).
Need: 1,500 / 13 = ~115 replicas.

But wait — each replica needs the full dataset:
  50 GB (users) + 250 GB (posts) + 160 GB (follows) = 460 GB
  115 replicas × 460 GB = 53 TB of total storage

This "works" but has two serious problems:

1. Storage waste: 53 TB just to serve one query type.

2. Consistency: replicas lag behind the primary. With async
   replication (the only practical option at 115 replicas),
   a user might post, then load their feed from a replica
   that hasn't received the write yet — their own post is
   missing. Synchronous replication to all 115 replicas
   would mean every write waits for 115 ACKs, making writes
   unacceptably slow (~seconds). You cannot have full ACID
   across async replicas — you sacrifice the "C" (consistency)
   the moment replication is asynchronous.

The deeper problem is not read capacity on simple queries —
it's that the feed query is inherently expensive in a
normalized relational schema.

Key insight: The feed query is expensive because it assembles the feed at read time by joining two large tables. What if we pre-assembled the feed at write time instead?

Iteration 1: SQL for Profiles, NoSQL for Timelines

The core idea: use each storage engine for what it does best.

DataStorageWhy
User profiles PostgreSQL Structured, relational. Needs ACID for account operations (email change + verification in one transaction). 50 GB fits easily. Simple key lookups are fast.
Follower graph PostgreSQL Needs transactions for follow/unfollow (especially mutual follows, block lists). Queries are simple: "who does user X follow?" is an index scan. 160 GB is large but manageable with sharding by follower_id.
Feed timeline Cassandra Pre-materialized. Each user has a row containing their feed. Reads are a single partition scan — O(1) per user. Optimized for high write throughput (fan-out).

Fan-Out-on-Write: How It Works

When a user publishes a post, the system writes that post into the timeline of every follower. This trades write amplification for fast reads.

User Bob publishes a post. Bob has 500 followers.

1. Write the post to the posts table (PostgreSQL):
   INSERT INTO posts (post_id, author_id, content, created_at)
   VALUES (9876, bob_id, 'Hello world', NOW());

2. Look up Bob's followers (PostgreSQL):
   SELECT follower_id FROM follows WHERE followed_id = bob_id;
   → Returns 500 follower IDs

3. Fan-out: write to each follower's timeline (Cassandra):
   For each follower_id in [follower_1, follower_2, ..., follower_500]:
     INSERT INTO feed_timeline (user_id, post_time, post_id, author_id)
     VALUES (follower_id, NOW(), 9876, bob_id);

   500 Cassandra writes, each ~100 bytes.

Cassandra Timeline Table

-- Cassandra schema for the feed timeline
CREATE TABLE feed_timeline (
  user_id    BIGINT,          -- partition key
  post_time  TIMESTAMP,       -- clustering key (DESC)
  post_id    BIGINT,
  author_id  BIGINT,
  content    TEXT,
  media_urls LIST<TEXT>,
  PRIMARY KEY (user_id, post_time)
) WITH CLUSTERING ORDER BY (post_time DESC);

New Feed Read Cost

Reading Alice's feed:
  SELECT * FROM feed_timeline
  WHERE user_id = alice_id
  LIMIT 50;

Cost: Single partition read in Cassandra.
  - 1 disk seek (or memory lookup if cached)
  - Read 50 rows sequentially (already sorted by post_time DESC)
  - Latency: 1-5 ms

Compare to the naive SQL approach: 50-100 ms
  → 10-50x faster read latency

Write Amplification Cost

New posts per day:        5M
Avg followers per author: 200
Fan-out writes per day:   5M × 200 = 1 billion Cassandra writes/day
Writes per second:        1B / 100K sec = 10,000 writes/sec

Each write: ~200 bytes (post content + metadata)
Write throughput: 10,000 × 200B = 2 MB/sec

Cassandra capacity:
  A Cassandra "node" = one machine running the Cassandra process.
  Unlike PostgreSQL, there is no primary/standby — every node is
  an equal peer that handles both reads and writes.

  A single node handles 10,000-30,000 writes/sec.

  Replication factor 3 (RF=3) means every write is stored on 3
  different nodes for durability. All 3 copies are active and can
  serve reads — this is not "1 primary + 2 standbys." If any node
  dies, the other 2 still have the data and keep serving traffic.

  The throughput cost: each client write becomes 3 internal writes
  (one per replica). So a 3-node cluster where each node can do
  ~10K writes/sec has an effective client throughput of ~10K
  writes/sec (not 30K), because each write hits all 3 nodes.

  Our 10,000 writes/sec fits comfortably.

Storage estimate:
  Each timeline row: ~200 bytes (user_id, post_time, post_id,
    author_id, content snippet, media_urls)
  Fan-out writes per day: 1 billion rows × 200 bytes = 200 GB/day
  Per year: 200 GB × 365 = ~73 TB/year
  With RF=3: 73 × 3 = ~219 TB/year across 3 nodes = ~73 TB/node

  A single machine with SSDs typically holds 1-8 TB. So 73 TB/node
  does NOT fit on a single machine. We need to shard (partition)
  across more nodes.

  Cassandra handles this automatically — it distributes partitions
  (each user_id hashes to a token range) across nodes. With 73 TB
  per replica set and 4 TB usable per node, we need:
    73 TB / 4 TB = ~18 nodes per replica set
    × 3 replica sets = ~54 nodes total (for 1 year of data)

  In practice, you also set a TTL (e.g., 30 days) on timeline rows
  since users rarely scroll past recent posts. With a 30-day TTL:
    200 GB/day × 30 = 6 TB per replica set
    × 3 = 18 TB total → fits on 5-6 nodes (3 TB each)

  This is far more reasonable and is what production systems do.

Trade-off summary: We turned an expensive read (10,000 row join, 50–100 ms) into a cheap read (single partition scan, 1–5 ms) by doing extra work at write time (500 Cassandra inserts per post). This works because there are far fewer writes than reads — typical read-to-write ratio in social media is 100:1 or higher.

Iteration 2: The Celebrity Problem (Hybrid Fan-Out)

Iteration 1 works great for normal users, but it breaks for celebrities.

The Problem: Celebrity Fan-Out

Consider a celebrity with 5 million followers who posts 10 times/day.

Fan-out per post:     5,000,000 Cassandra writes
Fan-out per day:      5M × 10 = 50,000,000 writes just for ONE celebrity
Fan-out per second:   50M / 100K ≈ 500 writes/sec sustained

Time to fan out 1 post: 5M writes at 10K writes/sec = 500 seconds
  → 8+ minutes before all followers see the post!

If 100 celebrities each have 5M+ followers:
  100 × 50M = 5 billion extra writes/day
  That's 5× the total writes from all normal users combined.

Write throughput needed: 5B / 100K = 50,000 writes/sec
  → Now we need 15+ Cassandra nodes just for celebrity fan-out.

Two problems: (1) Fan-out for celebrity posts is slow — 8 minutes is unacceptable for "real-time" feeds. (2) It wastes enormous storage and write bandwidth. Most followers may never even scroll to that post.

Solution: Hybrid Fan-Out

Use fan-out-on-write for normal users and fan-out-on-read for celebrities. At read time, merge the two.

Classification threshold: users with >10,000 followers = "celebrity"

When a NORMAL user posts (followers ≤ 10K):
  → Fan-out-on-write (same as Iteration 1)
  → Write to each follower's Cassandra timeline

When a CELEBRITY posts (followers > 10K):
  → Write only to a global "celebrity posts" table (Cassandra)
  → No fan-out

When reading Alice's feed:
  1. Read Alice's pre-materialized timeline from Cassandra (1-5 ms)
  2. Look up which celebrities Alice follows (cache this — rarely changes)
  3. For each celebrity Alice follows, fetch their latest posts
     from the celebrity_posts table (Cassandra or Redis — NOT a SQL
     table scan). Each celebrity is a separate partition lookup by
     user_id. Since these reads are independent, the feed service
     fires all of them concurrently (async I/O), so 5 reads take
     the same wall-clock time as 1 read (~1-3 ms), not 5× longer.
  4. Merge the two lists by timestamp, return top 50

Merge cost:
  - Pre-materialized timeline: 50 posts (1 Cassandra partition read)
  - Alice follows ~5 celebrities on avg: 5 concurrent reads × 10
    posts each. Fired in parallel → wall-clock time = 1 read.
  - Merge 50 + 50 = 100 posts, sort, return top 50
  - Total: 6 reads but only ~2 round trips of latency → <10 ms

Why Cassandra for Celebrity Posts (Not PostgreSQL)?

The celebrity_posts table is partitioned by celebrity user_id,
clustered by post_time DESC — the same pattern as feed_timeline.
This is a read-heavy, append-only workload (celebrities post,
millions of followers read). Cassandra excels here because:
  1. Partition-key lookups are O(1) — no JOINs needed
  2. Already in the stack — no extra operational overhead
  3. Scales horizontally if celebrity count grows

PostgreSQL could work too (it's just key lookups), but since
we already run Cassandra for timelines, keeping celebrity posts
there avoids adding another read path to a different system.

Redis sits in front as a cache for the hottest celebrity posts
(most followers read the same posts → high cache hit rate).

Why 10,000 Followers as the Threshold?

Fan-out cost for a single post:

  Assumption: ~0.1 ms per Cassandra write (typical for a single
  async write with consistency level ONE on local SSD).

  What is "consistency level"? In Cassandra, it controls how many
  replica nodes must acknowledge a write before it's considered
  successful:
    - ONE:    1 out of 3 replicas. Fastest, but if that node dies
              before replicating, the write could be lost.
    - QUORUM: Majority (2 of 3). Survives 1 node failure. Good
              balance of speed and durability.
    - ALL:    All 3 replicas. Slowest — if any node is down, the
              write fails entirely.

  For fan-out writes, ONE is the right choice: speed matters more
  than per-write durability. If a few timeline entries are lost,
  the user just misses one post in their feed — tolerable compared
  to the latency cost of waiting for 2-3 nodes across millions of
  fan-out writes.

  These times
  assume sequential writes from a single worker — in practice,
  fan-out workers batch and parallelize (e.g., 50-100 concurrent
  writes), so wall-clock time is much lower. But total cluster work
  (and the time until the LAST follower receives the post) scales
  linearly:

  - 100 followers:      100 writes → ~10 ms    ✓ instant
  - 1,000 followers:    1K writes  → ~100 ms   ✓ fast
  - 10,000 followers:   10K writes → ~1 sec    ✓ acceptable
  - 100,000 followers:  100K writes → ~10 sec  ✗ too slow
  - 1,000,000 followers: 1M writes → ~100 sec  ✗ way too slow

  "Too slow for whom?" The celebrity does NOT wait — the post is
  already saved and the response returned. Fan-out is fully async
  via Kafka. The problem is on the FOLLOWER side:

  1. Delivery delay: the LAST of 1M followers doesn't see the post
     in their feed for ~100 sec (or ~1 sec with 100x parallelism).
     For a breaking-news tweet, followers expect to see it within
     seconds, not minutes.

  2. Cluster resource hogging: 1M writes consume Cassandra capacity
     that other fan-outs are competing for. If 10 celebrities each
     post at the same time, that's 10M writes queued up — normal
     users' fan-outs get delayed too. The whole system slows down.

  3. Wasted work: most of those 1M followers won't open the app in
     the next hour. Pre-writing to their timelines is work that may
     never be read. Fan-out-on-read for celebrities avoids this —
     only followers who actually open their feed trigger the read.

  Even with 100x parallelism, 1M followers still takes ~1 sec of
  wall-clock time and consumes 1M writes of cluster capacity.

The threshold is where follower delivery delay and cluster load
become unacceptable. 10K is a reasonable boundary — adjust based
on your write capacity.

Users above the threshold (Zipf's law):
  ~0.1% of users have >10K followers = 50,000 celebrity accounts
  These 50K accounts are responsible for ~30% of all post views
  but would cause ~80% of fan-out writes without the hybrid model.

This is exactly what Twitter does. Twitter (now X) uses a hybrid approach: fan-out-on-write for most users, fan-out-on-read for accounts with millions of followers. The system that does the fan-out is called Fanout Service, and the threshold is configurable per account.

Why Kafka? What Does It Do Here?

Kafka is a distributed commit log — a durable, ordered, append-only queue that sits between producers (services that write events) and consumers (services that read and process those events). In this design, Kafka solves one specific problem: decoupling post creation from fan-out.

Without Kafka, the API server has to handle the fan-out itself. You have two bad options:

  • Synchronous (in the request path): The API server writes all 500 Cassandra rows (one per follower) before returning 201 Created. Reliable — every write is guaranteed to happen — but the user waits 500+ ms for a “post created” response. For a celebrity with millions of followers, this is impossible. Under load, API threads are blocked doing fan-out work instead of serving new requests.
  • Async in-process (fire-and-forget): The API server spawns a background thread or in-process task queue to do the writes, and returns immediately. Fast response — but not durable. If the API server crashes or restarts mid-fan-out (deploy, OOM kill, hardware failure), the remaining writes are silently lost. There’s no record of what was done and what wasn’t. Some followers get the post, others don’t, and nobody knows which.

Kafka gives you the best of both: fast response AND durable delivery. The API server does only two things: (1) write the post to PostgreSQL, (2) publish a small event ({post_id, author_id}) to a Kafka topic. Both take single-digit milliseconds. The API server responds immediately. The event is now durably persisted in Kafka (replicated to 3 brokers on disk) — even if every API server crashes right now, the event survives. A separate pool of fan-out workers consumes events from Kafka at their own pace and handles the heavy Cassandra writes. If a worker crashes mid-fan-out, Kafka re-delivers the event to another worker (the offset hasn’t been committed). If a burst of posts arrives (e.g., during a live event), Kafka buffers them — the workers catch up gradually without dropping events or overloading the API layer.

Kafka Provisioning: How Many Servers?

Kafka sizing for this workload
================================

Input:
  New posts per day:         5 million
  Posts per second (avg):    5M / 100K sec ≈ 50 msg/sec
  Posts per second (peak):   ~150 msg/sec (3× average)
  Message size:              ~200 bytes (post_id, author_id, timestamp, metadata)

Throughput:
  Write throughput (peak):   150 msg/sec × 200 B = 30 KB/sec
  Read throughput (peak):    same 30 KB/sec
    ↳ Why same? Each post is written to Kafka once and read once by
      the consumer group. The fan-out amplification (1 post → 200
      follower writes) happens AFTER the Kafka read, when workers
      write to Cassandra — not inside Kafka itself.

  This is trivially low for Kafka. A single Kafka broker on modern
  hardware can handle 200-500 MB/sec of writes. We are at 0.03 MB/sec.

Partitions:
  We need enough partitions to parallelize fan-out workers.
  Rule: each partition is assigned to exactly one worker in a consumer
  group. This assignment is STABLE — it lasts until a "rebalance" is
  triggered (worker crashes, new worker joins, or partitions change).
  During normal operation the mapping is fixed, not rotating.
  One worker CAN be assigned multiple partitions — it calls poll(),
  which returns a batch of messages from whichever assigned partitions
  have data ready, processes them, then polls again. So:
    - 12 partitions, 3 workers → each worker handles 4 partitions
    - 12 partitions, 12 workers → each worker handles 1 partition
    - 12 partitions, 15 workers → 3 workers sit idle (wasted!)
  Partitions set the MAXIMUM parallelism — you can't have more
  active workers than partitions.

  Time to fan out one post:
    avg 200 followers × 0.1 ms per Cassandra write = 20 ms per post
  Worker throughput (sequential processing — one post at a time):
    1 second (1000 ms) ÷ 20 ms per post = ~50 posts/sec per worker
    ↳ Each Kafka partition is consumed in order by one worker,
      so it finishes one post's fan-out before starting the next.
  Workers needed at peak:
    150 posts/sec ÷ 50 posts/sec/worker = 3 workers minimum
    ↳ Remember: one partition → one worker, but we have MULTIPLE
      partitions working in parallel. Posts are distributed across
      partitions by hash(author_id) % num_partitions, so:

        150 posts/sec total across 3 partitions
        → ~50 posts/sec per partition
        → each partition's dedicated worker handles ~50 posts/sec
        → 3 workers drain 3 partitions in parallel = 150 posts/sec

      The parallelism comes from having multiple partitions, not
      multiple workers on one partition. That's exactly at capacity
      though — too tight, so we add headroom:

  Why more than 3? Not all users have exactly 200 followers.
  Users just below the celebrity threshold (~10K) are still fanned
  out on write, and their posts take longer:
    9,000 followers × 0.1 ms = 900 ms per post (with batching, ~90 ms)
    A single slow post blocks that worker's partition for ~90 ms,
    causing other posts in the same partition to queue up.
  (Users ABOVE 10K are celebrities — they skip fan-out entirely
   and go to the celebrity_posts table, as described in Iteration 2.)
  Use 6-12 partitions → 6-12 workers to absorb these spikes.

Brokers:
  Rule of thumb: 1 broker per ~2,000-4,000 partitions.
  We have 6-12 partitions — 1 broker handles this easily.

  But for durability, Kafka replicates partitions across brokers.
  With replication factor 3, we need at least 3 brokers so each
  partition has copies on 3 different machines.

  Minimum: 3 brokers (for replication factor 3)

Storage:
  Messages per day: 5M × 200 B = 1 GB/day
  Retention period: 7 days (standard default)
  Storage per broker: 7 GB × RF3 / 3 brokers ≈ 7 GB/broker
  → Negligible. Even the smallest SSDs handle this.

  │ Partitions (topic)   │ 6-12       │
  │ Fan-out workers      │ 6-12       │
  │ Storage per broker   │ ~7 GB      │
  └──────────────────────┴────────────┘

  3 brokers is the minimum production setup. This workload doesn't
  need more — Kafka is dramatically over-provisioned even at 3 nodes.
  The real scaling challenge is on the Cassandra side (fan-out writes),
  not the Kafka side (event routing).

Redis Provisioning: How Many Cache Servers?

Redis serves two roles in this design: (1) caching the celebrity list per user (which celebrities does Alice follow?), and (2) caching hot celebrity posts so millions of followers don’t all hit Cassandra for the same data. Let’s size each. 

Redis sizing for this workload
================================

=== Dataset 1: Celebrity list per user ===

What it stores:
  For each user, the set of celebrity user_ids they follow.
  Most users follow 0-10 celebrities (out of ~50K celebrity accounts).
  Stored as a Redis Set (SADD/SMEMBERS) — unordered, deduped,
  and supports O(1) membership checks with SISMEMBER.

Size per entry:
  Key:   "celeb_list:{user_id}" → ~20 bytes
  Value: Redis Set of ~5 celebrity IDs × 8 bytes = ~40 bytes
  Overhead (Redis hash entry): ~80 bytes
  Total per user: ~140 bytes

Total users (DAU — only cache active users):
  50M DAU × 140 bytes = 7 GB

=== Dataset 2: Hot celebrity posts ===

What it stores:
  The latest ~20 posts per celebrity, so followers can read
  them in <1 ms instead of hitting Cassandra.

  Why only 20? A user's feed page shows ~20-50 posts total (from
  ALL followed accounts, not just one celebrity). So 20 posts per
  celebrity easily covers several feed loads. If a user scrolls
  far enough to exhaust the cached posts, the app falls back to
  Cassandra — rare and acceptable. Caching more (e.g. 200) would
  use 10× memory for data almost no one reads.

Celebrity accounts: ~50,000
Posts per celebrity cached: 20
Size per post: ~500 bytes (content + metadata)

Total: 50,000 × 20 × 500 B = 500 MB

=== Combined memory requirement ===

  Celebrity lists:     7.0 GB
  Celebrity posts:     0.5 GB
  Redis overhead:      ~30% (fragmentation, internal structures)
  ────────────────────────────
  Total:               ~10 GB of usable memory needed

=== QPS calculation ===

Every feed request hits Redis twice:
  1. Read celebrity list     (1 small GET)
  2. Read celebrity posts    (MGET for ~5 keys)
  → ~6 Redis commands per feed request

Peak feed QPS:     1,500
Redis commands:    1,500 × 6 = 9,000 commands/sec

A single Redis instance handles 100,000-200,000 commands/sec.
We are at 9,000 — well under 10% of one server's capacity.

=== How many Redis servers? ===

Memory: 10 GB fits on a single server (Redis instances commonly
  run on machines with 16-64 GB RAM).

Throughput: 9,000 cmd/sec is trivial for one instance.

But for availability (what if that one server crashes?), we need
replication:
  - 1 primary (handles all reads + writes)
  - 1 replica (hot standby — promotes to primary on failure)
  - Optional: 1 more replica for read scaling or multi-AZ

Minimum: 2 Redis servers (1 primary + 1 replica)
Recommended: 3 Redis servers (1 primary + 2 replicas for multi-AZ)

Summary:
  ┌──────────────────────┬────────────┐
  │ Component            │ Value      │
  ├──────────────────────┼────────────┤
  │ Redis servers        │ 2-3        │
  │ Memory per server    │ ~10 GB     │
  │ Peak commands/sec    │ ~9,000     │
  │ Cache hit rate (est) │ ~95%+      │
  └──────────────────────┴────────────┘

  Like Kafka, Redis is not the bottleneck. 10 GB of memory and
  9K cmd/sec is a light workload. The value of Redis here is
  latency (0.5 ms vs 1-5 ms from Cassandra) and shielding
  Cassandra from thundering-herd reads on celebrity posts.

  Can Cassandra handle celebrity reads WITHOUT Redis?

  A single Cassandra node handles ~5,000-10,000 simple partition
  reads/sec (small partitions, data cached in memory). Let's check:

  Celebrity with 5M followers posts:
    How many will read this post? Not all 5M — only active users.
    DAU / total users = 50M / 500M = 10% of any user's followers
    are active on a given day. So ~500K followers will load their
    feed and see this post (not all at once, but within hours).
    Worst case: 500K spread over 5 min = 500K / 300 sec ≈ 1,700 reads/sec

  One celebrity alone? Cassandra can handle it (1,700 < 5,000).

  The problem: HOT PARTITIONS.
    All reads for one celebrity hit the SAME Cassandra node
    (the one owning that celebrity's partition). You can't
    spread load across the cluster — partitioning is by
    celebrity_id, so all requests for celebrity X go to one node.

    If 10 celebrities post around the same time:
      10 × 1,700 = 17,000 reads/sec — but only IF their
      partitions are on different nodes. If 3-4 land on the
      same node, that node gets 5,000-7,000 reads/sec from
      celebrity posts alone, plus its regular workload.

  Redis solves this:
    - 100K+ ops/sec per instance (vs 5-10K for Cassandra)
    - Sub-millisecond latency (vs 1-5 ms)
    - No hot-partition problem: all celebrity data can be
      served from the same Redis instance without bottleneck
    - Cassandra only gets hit on cache misses:
        → user scrolls past the 20 cached posts (most common)
        → cache cold start (after Redis restart or eviction)
        → TTL expiry before refresh
      Miss rate depends on user scrolling behavior (a product
      metric). Industry data suggests ~90-95% of social media
      users only view the first screen, so miss rates are low —
      but the exact number varies by product.

Final Design Summary

ComponentTechnologyWhy This Choice
User profiles PostgreSQL (primary + 2 replicas) ACID for account ops. 50 GB fits in memory. Simple PK lookups. Read replicas for availability.
Follower graph PostgreSQL (sharded by follower_id) Needs transactions. Shard only if 160 GB becomes a bottleneck; initially fits on one instance.
Feed timeline (pre-materialized) Cassandra (3+ nodes, RF=3) High write throughput for fan-out. Single-partition reads for fast feed loading. Scales horizontally.
Celebrity posts Cassandra (separate table) + Redis cache Avoids massive fan-out. Celebrity posts are hot data — cache in Redis for <1 ms reads.
Fan-out service Kafka + worker pool New post → Kafka topic → workers fan out to Cassandra. Decouples post creation from fan-out. Handles bursts via buffering.

Key Takeaways

  • Use the right database for the right access pattern. SQL (PostgreSQL) excels at structured, relational data that needs ACID guarantees. NoSQL (Cassandra) excels at high-throughput, denormalized reads with known access patterns.
  • Fan-out-on-write trades write amplification for read performance. Pre-materializing timelines makes feed reads a single-partition scan instead of a multi-table join. This works when reads vastly outnumber writes (which they do in social media).
  • Celebrity accounts break fan-out-on-write. A user with millions of followers creates millions of writes per post. The hybrid approach (fan-out-on-write for normal users, fan-out-on-read for celebrities) solves this.
  • Always start with the math. The 15M row reads/sec number immediately told us the naive SQL approach was infeasible. Without that calculation, you might waste weeks building the wrong thing.
  • Decouple writes from fan-out using a message queue. Kafka absorbs bursty post creation and lets fan-out workers process at their own pace. The user gets a fast response; the fan-out happens asynchronously.
