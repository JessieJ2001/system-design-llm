Designing a Web Crawler

Crawl 1 billion URLs every 30 days — politely, without re-fetching duplicates, while keeping news fresh and archives undisturbed. We start with a single Python loop that would take 17 years and iterate to a polite, deduplicated, freshness-aware fleet.

NALSD Walkthrough • ~20 min read

The Scenario

You are building a general-purpose web crawler that feeds a search index. The crawler discovers, downloads, and parses HTML pages, extracts outgoing links, and re-crawls each page on a schedule. Output is a compressed page archive plus a stream of (url, content_hash, extracted_text, links) records consumed by an indexer (out of scope here).

What “Polite” Means

A polite crawler does not make the web operators’ life worse. Three rules:

  • Honor robots.txt: do not fetch disallowed paths; cache it for 24 h per host.
  • Throttle: at most 1 request/sec to any single host (or whatever Crawl-Delay says).
  • Identify: send a real User-Agent with a contact URL so admins can ask you to stop.

Violating these is how crawlers get banned, blacklisted, or sued. Politeness is not an optimization — it is a hard constraint.

The Quality Goals

  • Coverage: every URL in the corpus is fetched at least once per 30 days.
  • Freshness: high-importance pages that change frequently (news homepages) should be re-crawled within hours, not weeks.
  • No waste: never fetch the same URL twice in one cycle; never re-index identical content.
  • Resilience: a fetcher crash must not lose URLs or restart the corpus.

The fundamental tension: we want to crawl fast but the web wants us to crawl slowly. Politeness caps each host at 1 req/sec, which means raw throughput is achievable only by spreading concurrency across many hosts. Every architectural decision in this design is shaped by that asymmetry.

Essential Formulas

Every number on this page is derived from these constants and formulas.

Hardware Constants

Fetcher box (m5.4xlarge):
  vCPU              = 16 cores
  NIC               = 1 Gbps    (= 125 MB/s = 1.25e8 B/s)
  RAM               = 64 GB
  Cost              = $0.77/hr  (= $554/month at 24x30)

Object store (S3):
  Storage cost      = $0.023/GB/month

Frontier service (Redis + RocksDB):
  Operational cost  = ~$300/month (3 nodes, fixed)

Workload Constants

Corpus            N        = 1e9 URLs
Cycle             T        = 30 days = 2.592e6 sec
Page raw size     S_raw    = 100 KB = 1e5 B
Page gzip size    S_gz     = 25 KB  = 2.5e4 B
HTTP latency      L_http   = 0.5 sec
Parse CPU         L_parse  = 0.05 sec/page (single-core)
Hosts             H        = 1e7
Politeness        R_host   = 1 req/sec/host

Throughput Formulas

Average URL rate           = N / T
                           = 1e9 / 2.592e6 = 386 URLs/sec

Per-box concurrent capacity (three potential bottlenecks):
  Network bound  = NIC_bytes_per_sec / S_raw
                 = 1.25e8 / 1e5  = 1,250 URLs/sec
  Conn bound     = max_concurrent_conns / L_http
                 = 1,000 / 0.5   = 2,000 URLs/sec
  Parse bound    = vCPU / L_parse
                 = 16 / 0.05     = 320 URLs/sec

Per-box throughput = min(Network, Conn, Parse)

Fleet size  = ceil(target_rate / per_box_throughput)
Fleet cost  = fleet_size x $554/month + frontier $300/month

Wall-clock to crawl 1B URLs:
  T_crawl = N / fleet_throughput

The min() rule. A fetcher box is bottlenecked by whichever of {network, connection slots, parse CPU} is smallest. Every iteration on this page asks: which one is binding? Then attacks that one. We will see the binding constraint move from parse (Iter 1) to politeness (Iter 2) to duplicate work (Iter 4) to scheduling (Iter 5).

Back-of-Envelope Math

Quantify the budget before designing anything. (See the BOE cheat sheet for hardware reference numbers.)

Required URL Rate

Average rate:
  N / T = 1e9 (corpus) / 2.592e6 sec (30 days)
        = 386 URLs/sec

Practical rate (target sustained capacity):
  We need headroom for retries (~10%), rolling deploys
  (one box out of fleet during deploys), and bursts.
  Pick: 1,000 URLs/sec sustained = 2.6x the average.

  This is the design target the iterations must meet.

Bandwidth Budget

Average download:
  386 URLs/sec (avg rate, derived above) x 100 KB (S_raw, from constants)
  = 38.6 MB/sec
  = 309 Mbps

At 1,000 URLs/sec target:
  1,000 x 100 KB = 100 MB/sec = 800 Mbps

  This fits inside one 1 Gbps NIC, but only just.
  A single-box design must compete with parse for that
  bandwidth; a multi-box design distributes it.

Storage Footprint

Compressed page archive:
  N x S_gz = 1e9 x 25 KB = 25 TB
  Cost: 25,000 GB x $0.023/GB/mo = $575/month (S3)

Extracted text + metadata:
  ~5 KB per page (text) + ~200 B (URL, hashes, timestamps)
  N x 5.2 KB = 5.2 TB
  Cost: 5,200 x $0.023 = $120/month

Frontier (URL queue) state:
  ~100 B per scheduled URL (URL string + priority + last_seen)
  N x 100 B = 100 GB

URL-seen set (exact, RocksDB):
  ~32 B per entry (SHA-256 hash + scheduling fields)
  N x 32 B = 32 GB

Politeness Headroom

Per-host rate cap: 1 req/sec
Total hosts available: H = 1e7

If every host had a URL ready at all times, the politeness
ceiling on aggregate throughput would be:
  H x R_host = 1e7 x 1 = 10,000,000 URLs/sec

We need only 1,000 URLs/sec. So politeness is NOT the global
ceiling. It only bites when too few hosts are active at once.

Active hosts needed at target rate:
  1,000 URLs/sec / 1 req/sec/host = 1,000 simultaneously active hosts

  The frontier must keep at least 1,000 distinct hosts in
  the ready queue at all times. With 10M hosts in the
  corpus, this is easy — if we schedule right.

The 17-year problem. We have 30 days to crawl 1 billion URLs. At 1 URL/sec a single thread would take 31.7 years. At 386 URLs/sec we just barely make the budget. Every iteration must defend the headroom we built up — and politeness will keep trying to take it away.

Naive Design: One Synchronous Loop

The simplest crawler that could possibly work: a single Python process with an in-memory queue, blocking urllib fetches, and writes to local disk.

import urllib.request, lxml.html

q = ["https://en.wikipedia.org/wiki/Main_Page"]
seen = set()

while q:
    url = q.pop(0)
    if url in seen:
        continue
    seen.add(url)
    html = urllib.request.urlopen(url, timeout=10).read()  # blocks ~500 ms
    open(f"pages/{hash(url)}.html", "wb").write(html)       # blocks ~5 ms
    doc = lxml.html.fromstring(html)                        # 50 ms CPU
    for link in doc.xpath("//a/@href"):
        q.append(link)

Per-URL Time Budget

One URL costs:
  HTTP fetch (blocking):  500 ms  (L_http, from constants)
  Parse (single core):     50 ms  (L_parse, from constants)
  Disk write:               5 ms
  ----                    ------
  Total:                  555 ms  per URL

Throughput = 1 / 0.555 sec = 1.8 URLs/sec

What That Buys Us

Time to crawl 1B URLs:
  N / throughput = 1e9 / 1.8 URLs/sec
                 = 5.56e8 sec
                 = 6,430 days
                 ~= 17.6 YEARS

Bandwidth used:
  1.8 URLs/sec x 100 KB = 180 KB/sec = 1.4 Mbps
  NIC capacity is 1 Gbps. Utilization: 0.14%.
  We are leaving 99.86% of the network idle.

Storage:
  25 TB compressed pages, but local disk is 2 TB. Overflows.

Cost:
  1 box x $554/month = $554/month
  Rate per useful URL: $554 / (1.8 x 86400 x 30) = $0.000119
  But we never finish, so this number is meaningless.

Why It’s Broken (Beyond Speed)

  • No politeness. The first time the queue surfaces 100 links from one site, we hammer it at 100 req/sec. We get banned.
  • No robots.txt. We crawl pages site owners explicitly disallow.
  • No URL canonicalization or dedup beyond the in-memory seen set. Crash = restart from scratch.
  • No content dedup. Mirror sites get re-fetched and re-indexed.
  • No re-crawl strategy. Once a URL is in seen, it is never refreshed.

Dominant bottleneck: synchronous I/O. The thread spends 500 ms blocked on each HTTP call while the CPU sits idle. Fix that single waste before anything else.

Iteration 1: Async I/O on a Single Box

Replace the blocking loop with async coroutines and a process pool for parsing. One box, but the CPU is no longer idle while the network works.

Bottleneck Recap (from Naive)

Naive: 1.8 URLs/sec, network 0.14% utilized.
The thread is blocked 500 ms / 555 ms = 90% of the time
on a synchronous HTTP call. Wasted CPU and wasted NIC.

Fix: overlap many fetches at once with async/await, and
parse on multiple cores in parallel.

The Three Capacities

With 1,000 concurrent in-flight HTTP requests on one box:

  Network bound  = NIC / S_raw
                 = 1.25e8 B/s (1 Gbps NIC, from constants)
                   / 1e5 B (S_raw, from constants)
                 = 1,250 URLs/sec

  Conn bound     = max_concurrent_conns / L_http
                 = 1,000 conns
                   / 0.5 sec (L_http, from constants)
                 = 2,000 URLs/sec

  Parse bound    = vCPU / L_parse
                 = 16 cores (m5.4xlarge, from constants)
                   / 0.05 sec/page (L_parse, from constants)
                 = 320 URLs/sec

  Throughput     = min(1250, 2000, 320) = 320 URLs/sec
                                          (parse-bound)

Per-Box Numbers

Throughput:    320 URLs/sec  (parse-bound; CPU saturated)
Bandwidth:     320 x 100 KB = 32 MB/s = 256 Mbps
                              (out of 1 Gbps; 25% NIC use)
Time for 1B:   1e9 / 320 / 86400 = 36 days
Cost:          1 box x $554/month = $554/month
Storage:       must move to S3 ($575/month for 25 TB)

Why Parse Becomes the Bottleneck

The async event loop happily juggles 1,000 in-flight HTTP
requests. They return at 2,000/sec aggregate (Conn bound).
But each returned page must be parsed before its links can
be enqueued, and that takes 50 ms of pure CPU.

  CPU:    16 cores at 100% gives 16 / 0.05 = 320 parses/sec
  Inflow: 2,000 URLs/sec returning from the network

The parse pool builds a backlog of 2,000 - 320 = 1,680
pages/sec until back-pressure kicks in and the fetcher
throttles itself to 320 URLs/sec.

We are now CPU-bound. The NIC is 75% idle.

From 1.8 to 320 URLs/sec (180× speedup). But 320 URLs/sec for 1 B URLs takes 36 days — we miss the 30-day budget, and we have zero headroom. The bottleneck is now parse CPU on one box. The fix is more boxes; that requires a shared frontier so they don’t step on each other.

Iteration 2: Distributed Fleet + Shared URL Frontier

Pull the URL queue out of process memory into a shared URL Frontier service so multiple fetcher boxes can work in parallel.

Bottleneck Recap (from Iter 1)

Iter 1: 320 URLs/sec on one box, parse-bound.
  Time for 1B URLs at 320/sec: 36 days > 30-day target.
  We need at least 1,000 URLs/sec sustained capacity (the
  BOE-derived target), with headroom for retries.

The bottleneck is now “single box of CPU.” You cannot
fix this on one box — you have to shard the work
across many boxes, which means the URL queue can no
longer live in one process’s memory.

Fleet Sizing

Per-box throughput (carried from Iter 1): 320 URLs/sec
Target: 1,000 URLs/sec sustained

  Boxes_min = ceil(1,000 / 320) = 4 boxes

Per-box throughput is unchanged at 320 URLs/sec
  (still parse-bound: min(1250, 2000, 320)).
Fleet throughput = 4 x 320 = 1,280 URLs/sec
                            (28% headroom over 1,000 target)

Time for 1B URLs:
  N / fleet_throughput = 1e9 / 1,280 / 86400
                       = 9 days  (ample budget vs 30)

The URL Frontier Service

A central service that holds the “to-fetch” queue.

Required operations:
  enqueue(url, priority)  — called by fetchers when they
                            extract new links from a parsed page
  dequeue() -> url        — called by fetchers to pull work

Backing store:
  Redis sorted set (priority queue) for the ready queue.
  RocksDB / disk-backed log for durability past memory cap.

Sizing (BOE):
  Frontier holds N = 1e9 URLs x ~100 B = 100 GB.
  3-node Redis cluster with RDB persistence: ~$300/month.

Failure model:
  If a fetcher dies mid-fetch, the leased URL times out
  after 60 sec and is re-enqueued. No URL is permanently
  lost as long as the frontier is durable.

The New Bottleneck: Politeness Violations

The frontier hands out URLs in roughly priority order
without grouping by host. So at any moment:

  Fetcher #1 might pull 50 URLs from amazon.com
  Fetcher #2 might pull 30 URLs from amazon.com
  Fetcher #3 might pull 10 URLs from amazon.com
  Fetcher #4 might pull 12 URLs from amazon.com

Aggregate: 102 simultaneous requests to one host across
the fleet. The host’s rate limit is 1 req/sec.
Result: bans, captchas, retroactive IP blocklists.

The fleet has the right total capacity, but it’s
distributed wrong. We need to make the host the unit
of scheduling, not the URL.

From 320 to 1,280 URLs/sec. 4 boxes meet the 1,000 URLs/sec target with 28% headroom and finish a 1B-URL crawl in 9 days. The frontier service costs $300/month; the fleet costs $2,217/month for a total of ~$2,500/month. But the design is illegal — we are violating politeness across multiple hosts. That is the next bottleneck.

Iteration 3: Host-Sharded Frontier + Politeness Queues

Make the host the unit of scheduling. All URLs from amazon.com go to exactly one fetcher, which throttles itself to 1 req/sec for that host.

Bottleneck Recap (from Iter 2)

Iter 2: 1,280 URLs/sec aggregate across 4 fetchers, but
the frontier hands out URLs without coordinating by host.
Worst-case: 100+ concurrent requests to one host. Bans.

To fix this we need TWO things on the same scheduling
boundary: (1) shard the frontier by host so only one
fetcher ever sees a given host, and (2) have each
fetcher rate-limit per-host to 1 req/sec.

These are one bottleneck (host-as-unit-of-work) addressed
by two coordinated techniques.

Frontier Sharding by Host Hash

Compute fetcher_id = consistent_hash(host(url)) % num_fetchers.

  url = "https://en.wikipedia.org/wiki/X"
  host = "en.wikipedia.org"
  fetcher_id = consistent_hash("en.wikipedia.org") % 4

All wikipedia URLs land on the same fetcher. That fetcher
is the ONLY component that ever sends a request to
en.wikipedia.org — politeness becomes a local
property of one process.

Hosts per fetcher: H / 4 = 1e7 / 4 = 2.5 million each.
URLs per fetcher: N / 4 = 250 million each.

Per-Host Token Bucket Inside a Fetcher

Each fetcher maintains a small map:
  host -> (token_bucket, queue_of_pending_urls)

Token bucket rules (per host):
  capacity   = 1 token
  refill     = 1 token/sec  (R_host, from constants)
  cost       = 1 token per request

Scheduler loop on a fetcher:
  1. Pick a host whose bucket has ≥1 token AND queue
     is non-empty.
  2. Take a URL from that host’s queue.
  3. Spend 1 token; refill timer running.
  4. Issue async fetch.

Memory cost:
  2.5M hosts x ~80 B (state + small queue head) = 200 MB
  Easily fits in the 64 GB RAM box.

Active-Host Math (Why We Don’t Lose Throughput)

A fetcher can issue at most 1 req/sec/host. So to drive
the fetcher at 320 URLs/sec we need 320 hosts to be
“active” (have a URL queued AND a token available)
at any moment.

Per fetcher: 2.5M hosts assigned, 250M URLs assigned.
On average each host has 100 URLs in flight over the cycle.

Across the 30-day cycle, every host is reachable. The
scheduler interleaves so a steady stream of distinct
hosts is “ready” at any time. With 2.5M hosts and
~100 URLs per host, keeping 320 hosts hot is trivial.

  active_hosts_needed = 320 URLs/sec / 1 req/sec/host = 320
  active_hosts_available >> 2,500 (typical cycle saturation)

So the per-fetcher throughput stays at 320 URLs/sec,
identical to Iter 2 — only now it is polite.

DNS Cache (Coupled Optimization)

Without a cache, each fetch issues a DNS query:
  1,280 URLs/sec x 1 query/url = 1,280 DNS queries/sec
  fan-out across 10 M hosts; resolver overload likely

Host-sharding makes a per-fetcher DNS cache extremely
effective: each fetcher repeatedly hits its own 2.5M hosts.

LRU cache per fetcher:
  size  = 1 M entries x ~50 B per entry = 50 MB RAM
  TTL   = honor DNS TTL (typical 300 sec to 24 h)

Hit rate: with 2.5M hosts cycled across 30 days and an
LRU cache holding 1M, the long-lived top hosts always hit.
Empirical hit rate: ~99%.

  DNS queries after cache: 1,280 x 0.01 = 13 queries/sec
  100x reduction. No external resolver pressure.

robots.txt cache: same shape. Per fetcher, cache robots.txt
per host with a 24 h TTL. 2.5M hosts x ~2 KB = 5 GB on
disk; small in-memory hot set.

What Did and Didn’t Change

Per-fetcher throughput: 320 URLs/sec (UNCHANGED —
                        still parse-bound, min(1250, 2000, 320))
Fleet throughput:       1,280 URLs/sec (UNCHANGED)
Cost:                   $2,517/month (UNCHANGED)

What CHANGED:
  - No host receives more than 1 req/sec.
  - DNS queries dropped from 1,280/sec to ~13/sec (cache).
  - robots.txt enforced; disallowed paths skipped.
  - Politeness state is local to each fetcher: simple,
    no distributed coordination needed.

Throughput is preserved because we have many hosts. Politeness only bites if too few hosts are active. With 2.5M hosts per fetcher and a steady interleaved stream, keeping 320 hosts “hot” per fetcher is easy — we can stay parse-bound while staying polite.

The New Bottleneck: Wasted Work on Duplicates

As pages are parsed, ~30% of extracted links are URLs we
already have in the corpus (cycles, mirrors, navigation
re-references). Without a global “URL seen” check,
those duplicates are re-enqueued and re-fetched.

Worse: ~5% of fetched pages have IDENTICAL content to
something we already have (mirror sites, syndicated news,
parameter variations of the same page).

Concrete cost:
  N_useful = 1B unique URLs, but
  N_fetched = 1.43B (with 30% link-level duplication)
  Excess work = 430M wasted fetches per cycle
              = 430M / 30 days / 86400 = 166 URLs/sec wasted
              = ~13% of our 1,280/sec capacity

We are paying for a 4-box fleet to do work we could skip.

Iteration 4: URL and Content Dedup

Cut wasted fetches by checking if a URL has been seen before enqueueing, and if its content has been seen before downstream re-indexing.

Bottleneck Recap (from Iter 3)

Iter 3: 1,280 URLs/sec capacity, polite, 4 boxes, $2,517/mo.

To cover 1B unique URLs at 30% URL-level duplication
we issue 1.43B fetches per cycle. Avg fetch demand:
  1.43e9 / (30 x 86400) = 552 URLs/sec  (carried from Iter 3)
Of that, 30% wasted on duplicates:
  552 x 0.30 = 166 URLs/sec wasted on average
                = 13% of our 1,280/sec fleet capacity.

If we kill the wasted work, the unique-URL fetch demand
drops to:
  1e9 / (30 x 86400) = 386 URLs/sec  (no duplicate inflation)

We can shrink the fleet from 4 to 3 boxes and still meet
this lower demand with comparable headroom.

URL Bloom Filter (Front Filter)

A bloom filter answers “is this URL probably new?”
in O(k) hash ops with no disk hit and a small RAM footprint.

Sizing for false-positive rate p = 1%:
  bits  = -n x ln(p) / (ln 2)^2
        = -1e9 x ln(0.01) / (0.693)^2
        = 1e9 x 4.605 / 0.480
        = 9.6e9 bits = 1.2 GB

  k     = -log_2(p) = log_2(100) ≈ 7 hashes
  lookup = 7 hashes x ~100 ns = ~1 us per check

False positive: we say “seen” on a URL we have NOT
seen. Effect: we miss it. Tolerable; it will likely be
discovered again from another inbound link.

False negative: impossible. If the bloom filter says
“not seen,” the URL is definitely new.

Where it lives: 1.2 GB shared bloom filter colocated with
the URL Frontier service (the place that decides whether
to enqueue). Out-of-process check via Redis BITCOUNT-style
ops, or in-process via a sidecar. Either fits in RAM.

For exact membership we ALSO keep a RocksDB “url_seen”
table (32 GB on disk; 32 B per entry x 1B). The bloom
filter is a fast first-line check to avoid touching disk.

Content Fingerprint (Back Filter)

After parse, compute SHA-256 of the canonicalized content
(strip whitespace, ads, timestamps; tokenize main body).

  hash_size = 32 B  (SHA-256)
  total     = N x 32 B = 32 GB

If we’ve seen this hash before, the page is a content
duplicate (mirror, syndicated). We still archive it (in
case the URL becomes the canonical), but we do NOT push
it downstream to the indexer or extract its links twice.

Why we hash AFTER fetch and not before: we cannot know
the content without fetching. Content dedup saves
downstream cost (indexing, link extraction, storage churn)
but not fetch cost. URL dedup saves all of them.

Re-Sized Fleet

Post-dedup demand: 386 URLs/sec avg (from recap above).

Per-box throughput unchanged at 320 URLs/sec
  (still min(1250, 2000, 320) — parse-bound).

3 boxes x 320 = 960 URLs/sec capacity
  Headroom: 960 / 386 = 2.49x  (similar to BOE’s 2.6x
            target, just applied to the lower demand).

Resilience check: with 1 box out for rolling deploy,
  remaining 2 x 320 = 640 URLs/sec » 386 demand. OK.

Time for 1B URLs at full capacity:
  1e9 / 960 / 86400 = 12 days
  (well under 30-day budget; leaves headroom for re-crawls)

Cost (compute + frontier; storage is constant across iters):
  3 fetchers x $554/mo                = $1,663
  + Frontier (now also hosts the bloom
    filter and 32 GB url_seen RocksDB
    — both small relative to its
    existing 100 GB priority queue)   = $300
  ----------------------------------    ------
  = $1,963/month  vs. $2,517 in Iter 3
  Saved: $554/month (22% reduction; one box’s worth)

What Didn’t Change

Per-box throughput: 320 URLs/sec (UNCHANGED — the
  per-box bottleneck is still parse CPU, min(1250, 2000, 320),
  and dedup happens at the frontier, not the fetcher).

Politeness: still 1 req/sec/host (UNCHANGED).
Page archive size: still 25 TB (we still archive content
  duplicates against the URL key — we just don’t
  re-index them).
robots.txt and DNS caches: unchanged.

What CHANGED:
  Fleet size: 4 -> 3 boxes
  Cost: $2,517 -> $1,963/mo (22% saving = one box’s cost)
  Fetch demand: 552 URLs/sec avg -> 386 URLs/sec avg
                (no duplicate inflation)
  Time at full capacity: 9 -> 12 days for 1B URLs

Dedup is leverage on the frontier, not the fetchers. The bloom filter check happens once per discovered link, before enqueue. We pay 1 us of CPU to save a 500 ms fetch + 50 ms parse + bandwidth + politeness budget. The ROI is 5 orders of magnitude.

The New Bottleneck: Uniform Re-Crawl Wastes Budget

After Iter 4 we comfortably crawl 1B unique URLs in 12 days.
But the cycle is 30 days — what do we do with the
remaining 18 days of capacity?

Naively, re-crawl every URL once on day 31. Every URL gets
the same refresh schedule.

Problem: pages have wildly different change rates.
  - news homepage: changes every minute
  - product page: changes every week
  - Wikipedia archive: changes every year
  - PDF spec: never changes

A uniform 30-day schedule is wrong on both ends:
  - News pages are stale up to 30 days = bad for users
  - Static PDFs are re-fetched 12 times/year for nothing
    = wasted budget that could go to fresh content

Iteration 5: Adaptive Re-Crawl Scheduling

Spend the same crawl budget more wisely: crawl important + frequently changing pages often, and rarely changing pages rarely.

Bottleneck Recap (from Iter 4)

Iter 4: 960 URLs/sec capacity on 3 boxes, dedup in place,
12 days at full capacity for 1B URLs — 18 days of
remaining cycle that go to uniform re-crawls.

Quality bottleneck: every URL has the same refresh
cadence. News is stale; static PDFs are over-crawled.

Fix: make the re-crawl interval a function of
(importance, change_rate, time_since_last_crawl).

The Priority Score

For each URL we maintain three signals:

  importance(url)    — PageRank-style signal computed
                       offline by a separate batch job;
                       refreshed weekly.
  change_rate(url)   — EWMA of binary “content
                       changed since last crawl?” over
                       the last 5 crawls.
  age(url)           — current_time - last_crawl_time.

Re-crawl priority:
  score = importance(url) x change_rate(url) x age(url)

Frontier ready queue is a Redis ZSET ordered by score.
Highest score gets pulled first.

Re-Crawl Cadence by Tier

Bucketing 1B URLs by (importance x change_rate):

  Tier H (top 5%):    50 M URLs, refresh every 3 days
                      = 50 M / 3 days = 16.7 M crawls/day
                      = 500 M crawls/month

  Tier M (next 30%): 300 M URLs, refresh every 30 days
                      = 300 M / 30 = 10 M crawls/day
                      = 300 M crawls/month

  Tier L (bottom 65%): 650 M URLs, refresh every 90 days
                      = 650 M / 90 = 7.2 M crawls/day
                      = 217 M crawls/month

  Total: 500 M + 300 M + 217 M = 1,017 M crawls/month
       ≈ same 1B/month budget we had with uniform sched

Average rate:
  1,017 M / 30 / 86400 = 393 URLs/sec
  (vs 386 URLs/sec demand in Iter 4 — essentially the
   same budget, just spent differently across tiers)

Fleet capacity (carried from Iter 4): 3 x 320 = 960 URLs/sec
Headroom: 960 / 393 = 2.44x. Same fleet, same cost,
but news is fresh within 3 days and static content is
no longer over-fetched.

Storage of Per-URL Scheduling State

Per URL we now store:
  url            ~80 B
  content_hash    32 B  (SHA-256)
  last_crawl       8 B  (unix timestamp)
  importance       4 B  (float32)
  change_rate      4 B  (float32 EWMA)
  --- total ----  128 B / URL

  N x 128 B = 1e9 x 128 = 128 GB

This is the URL state table (RocksDB on local SSD).
The bloom filter (1.2 GB) is the in-RAM front filter
on top of this. The Redis ZSET (~80 GB priority
queue with score+url) is the next-fetch index.

What Didn’t Change

Throughput: 960 URLs/sec capacity (UNCHANGED, same 3 boxes).
Per-box bottleneck: still parse CPU at 320 URLs/sec each.
Politeness: still 1 req/sec/host.
Cost: $1,963/month (UNCHANGED — same fleet).
Aggregate crawls/month: ~1B (UNCHANGED — same budget).

What CHANGED:
  Quality. News pages are at most 3 days stale, not 30.
  Static content is no longer over-crawled.
  Per-URL state grew from 32 B to 128 B (+96 GB total).

Adaptive scheduling buys quality without buying capacity. The throughput target was already met in Iter 4. Iter 5 reallocates the same budget so it produces a better index. The signal that drives the priority — observed change rate — comes for free from the dedup hash already computed in Iter 4.

End-to-End Lifecycle of One URL

1. Discovered as a link in another page’s parse output.
2. Bloom filter check at frontier (~1 us). New? Continue.
3. Hash(host(url)) -> assigned fetcher #k.
4. Enqueued in fetcher #k’s per-host queue, with priority
   score = importance(url) x change_rate(url) x age(url).
5. When token bucket for host has ≥1 token AND host’s
   robots.txt allows the path, fetcher pulls the URL.
6. DNS cache lookup (~1 us, 99% hit).
7. Async HTTP GET (~500 ms wall time, but overlapped with
   ~1,000 other in-flight fetches).
8. Page returned -> parse pool (~50 ms CPU).
9. SHA-256 of content -> content hash table check.
   Already seen? Archive but skip downstream emit.
10. Extract links -> for each, back to step 2.
11. URL state updated: last_crawl=now, change_rate=EWMA,
    next priority computed by scheduler.

Total improvement. From 1.8 URLs/sec on a single sync thread (17.6 years for 1B URLs, $554/mo compute, no politeness, no dedup, no freshness) to 960 URLs/sec capacity on 3 boxes (12 days at full capacity for 1B URLs, $1,963/mo compute + frontier plus $695/mo storage = $2,860/mo all-in, polite, deduplicated, with adaptive freshness). Throughput jumped 533× (= 960 / 1.8) while the design became simpler per component — because each iteration isolated one bottleneck.

Key Takeaways

  • Politeness, not throughput, is the hard constraint. The web has rate limits. You can buy more boxes for raw throughput; you cannot buy your way past amazon.com’s 1 req/sec rule. Make the host the unit of scheduling.
  • Shard the frontier by host hash. Politeness becomes a local property of one fetcher, no distributed locks needed. Same trick makes DNS and robots.txt caches hit ~99%.
  • Dedup pays for itself five orders of magnitude over. A 1 us bloom filter check saves a 500 ms fetch. ~30% of links you discover are duplicates — killing that lets you shrink the fleet.
  • URL dedup saves fetch cost; content dedup saves index cost. They are different filters at different stages of the pipeline. Both are cheap; both compound.
  • Adaptive re-crawl is freshness without throughput. The same 1B-crawl/month budget reallocated by (importance x change_rate) gives news pages 10× the freshness without buying any new boxes.
  • The bottleneck moves — follow it. Sync I/O -> parse CPU -> politeness -> duplicate work -> scheduling quality. Each iteration named the binding constraint, attacked it, and found the next one.
  • min(network, conn, parse) is the per-box capacity formula. On every iteration on this page the answer was “parse-bound at 320/sec.” That is not luck — it is the consequence of large pages + 16-core boxes + a 50 ms parse cost. Different constants would put the bottleneck elsewhere.
  • Frontier durability is non-negotiable. Losing a fetcher is fine; losing the frontier means restarting the corpus. Replicate it, persist it, and make it the boring part of the design.

Assumptions and When They Break

  • Long-tail host distribution: we assumed 10 M hosts each with ~100 URLs. If 80% of URLs come from 1,000 mega-hosts (Reddit, Wikipedia, YouTube), politeness becomes the true global ceiling: 1,000 hosts x 1 req/sec = 1,000/sec hard cap. The fix is to negotiate higher rate limits with those operators.
  • Static page sizes: we used 100 KB. Modern JavaScript-heavy pages weigh 2–5 MB rendered. Crawling them with a real browser (Playwright) costs ~5 sec and 500 MB RAM each — an entirely different bottleneck (compute, not network) and a different fleet (CPU+RAM-heavy, not NIC-heavy).
  • Cooperative web: we assumed sites serve normal HTML. Many serve bot-detection pages, captchas, or rate-limit responses (HTTP 429). A real crawler needs back-off, IP rotation, and an ethics policy for which sites it refuses to evade detection on.
  • 30-day cycle: a search engine like Google crawls high-importance pages every few minutes. At that scale the “static” tier from Iter 5 grows, the “hot” tier explodes, and you need a hierarchical crawler with multiple priority lanes feeding multiple fleets.
  • Single region: hosting in one region adds 100–200 ms RTT to far-away sites. A real crawler runs from a few regions for latency-friendly fetches and to amortize bandwidth costs.
  • Bloom filter false positives: 1% FP rate means we miss ~1% of new URLs on first discovery. Acceptable because most URLs are linked from many sources; the next link rediscovers them.
