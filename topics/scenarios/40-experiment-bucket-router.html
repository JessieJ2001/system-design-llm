Designing an Edge Backend Picker & Experiment Bucket Router

An in-process routing primitive in every edge-router pod at 2 M RPS: for each request, pick a healthy backend from a constantly-mutating 5 k-member pool (200 churns/sec from rolling deploys) and deterministically assign the user to an A/B variant from ~200 live experiments — routing decision p99 ≤ 100 µs, fleet-wide config convergence ≤ 1 s, a control-plane outage must never stop routing. We start from the LC #380 array-plus-hashmap with uniform GetRandom — which connection-saturates the backends, ignores load, and corrupts experiments on every edit — and iterate to deterministic per-pod subsetting, weighted power-of-two-choices with outlier ejection, layered salted mutex-aware bucketing, and a versioned-delta config plane with staged activation and a last-known-good fallback.

NALSD Walkthrough • ~40 min read • LC #380 (Insert / Delete / GetRandom O(1))

LC kernel. Insert / Delete / GetRandom O(1) (LC #380). Textbook structure: an array of items + a hashmap item→index. Insert: append to the array, record the index. Delete: swap the item with the array's last element, pop, fix the moved element's index in the map. GetRandom: array[rand() % len]. All three O(1); GetRandom is uniform over the current set. Five properties that break in a production router: (1) uniform isn't load-aware — a cold-cache just-deployed backend, or one in a busy zone, gets the same share as a warm idle one → tail latency and cascading overload; (2) items aren't weightless — backends advertise capacity, variants have allocation weights; (3) items have health state — a failing backend must be temporarily ejected and probed back, not just deleted; (4) the set isn't the whole pool — if every router pod routes to every backend, each backend pays connection cost proportional to the entire router fleet; production needs each pod to use a bounded subset; (5) “random” for A/B can't be rand() — an A/B assignment must be a deterministic, sticky-per-user, salted, layered function so the same user gets the same variant on every pod for the life of the experiment, and editing one experiment doesn't re-roll everyone. The C8 production cousin: a deterministic balanced subset per pod, picked by weighted power-of-two-choices with outlier ejection, and a parallel experiment instance whose “GetRandom” is hash(user ‖ experiment ‖ salt) mapped through layered mutex-group ranges.

S0. Scenario & Requirements

Every edge-router pod — the L7 proxy that terminates TLS and forwards to the right backend — embeds a small library that answers two questions per request, in-process, in microseconds: which backend? (pick a healthy member of the target service's pool, load-aware) and which experiment variant? (deterministically bucket this user into A/B variants of the live experiments). The backend pools mutate constantly (rolling deploys churn ~200 backends/sec); the experiment set changes when product teams edit experiments; both must converge across the whole router fleet within ~1 second, and a control-plane outage must leave routing fully functional on the last-known config. This is the routing primitive, not the proxy — TLS, HTTP parsing, the health-check mechanism itself, and the service-discovery source-of-truth are upstream of us; the experiment analysis pipeline is downstream. The tension: the picker must be O(1)-cheap and lock-free on the hot path, yet load-aware, ejection-aware, connection-bounded, and — for A/B — bit-for-bit deterministic across the fleet.

System class

Class: infra primitive — an embedded library + a thin control plane — see functional-requirements.html. The FR vocabulary is operations + semantics (what calls it exposes and what guarantees they make), not user-facing capabilities; the load-bearing NFRs are per-call latency, the connection bound it imposes on its dependencies, and config-convergence consistency.

Functional requirements (cite functional-requirements.html)

  • pick_backend(pool_id, ctx) → {backend, attempt_token} — choose a backend from the pool's currently-healthy, non-ejected members; modes per pool: uniform, capacity-weighted, load-aware (power-of-two-choices), or sticky (hash a session key). Returns a token the caller uses to report the outcome.
  • report_outcome(attempt_token, ok | conn_refused | timeout | 5xx) — feeds the local in-flight-count estimate and the outlier-ejection counter for that backend.
  • assign_bucket(user_id, experiment_id) → variant | not_in_experiment — deterministically assign a user to a variant; same user → same variant for the life of the experiment (until re-salted); respect mutual-exclusion groups (a user is in at most one experiment per group).
  • eject(backend) / probe-back loop — outlier ejection: a backend that accumulates consecutive failures is removed from rotation with exponential backoff; a probe loop returns it when healthy.
  • update_pool / update_experiments — the control plane pushes the current pool membership (with weights, zones) and the experiment config; pods reconcile to the latest version within ~1 s.

Out of scope (named, not silently dropped): the L7 proxy's TLS / HTTP parsing / connection pooling beyond the subset choice; the health-check mechanism (we consume a health signal); the service-discovery source of truth (we consume its feed); the experiment analysis / stats pipeline (downstream of “which variant did this request get”); cross-region routing (“route to the nearest healthy region” is a higher layer — GeoDNS / anycast); per-request authn/authz.

Flagged assumptions (would confirm with the interviewer): (1) “consistency within 1 s” is eventual — brief windows where pods disagree on the pool are fine (a request to a just-removed backend gets a fast retry); (2) backends advertise a stable capacity weight and a zone; (3) a session key (for sticky pools) is cheaply extractable from the request; (4) the experiment config is small (~200 experiments × ~10 fields ≈ tens of KB); (5) the platform spans ~10 regions, ~70 router pods per region → ~700 router pods globally, and a backend pool may be regional or global — a global pool sees connections from all ~700 pods, which is the structural pressure.

Per-call SLO

  • pick_backend — p99 ≤ 100 µs, p999 ≤ 500 µs (a few array reads + a counter compare; no allocation, no lock, no syscall).

S3. Back-of-Envelope — the ratios that govern every iteration

No optimization technique appears here — subsetting, P2C, and layered bucketing are named as future moves, not used in any calculation below.

The hot path is cheap; the dependency bill isn't

pick_backend at 2M RPS x ~5 array/cache ops x ~few ns = ~tens of ns/call -> ~0.05 core-seconds/sec across a region. Trivial.
Even a naive O(M) weighted scan of M=5000 ~= 5000 x ~2 ns ~= 10 us/call -> 2M x 10 us = 20 core-sec/sec -> ~3 of the 70 pods. Affordable.
=> The picker's CPU is NOT the bottleneck. What it COSTS is downstream: connections, backend overload, and corrupted experiments.

The connection explosion

Naive "every pod -> every backend":  conns_per_backend = N_pods.
  Regional pool, N=70:   70 conns/backend     (tolerable)
  GLOBAL pool, N=700:    700 conns/backend     -> ~700 x ~80 KB = ~56 MB of conn state + 700 keepalives + FD pressure
  GLOBAL pool with only M=50 backends, N=700:  STILL 700 conns/backend  -> a tiny popular service pays the WHOLE fleet's cost.
=> conns_per_backend = N (the router-fleet size), independent of how much traffic the backend actually gets. This is the canonical
   case for SUBSETTING (Iteration 0): make it N*S/M and tune S so it's bounded and decoupled from N.

The experiment-corruption hazard

Naive bucketing: variant = variants[ hash(u, e.id) % total_weight ].  Edit experiment e (add a variant, change weights) ->
  total_weight changes -> the modulo lands differently -> EVERY user can flip variant -> a customer in "treatment" yesterday is in
  "control" today -> the A/B comparison is comparing two different (mixed) populations -> the experiment's conclusion is garbage,
  AND the user sees the UI change under them. And two experiments with the same weights correlate (always-treatment or always-control
  in both) -> their effects are confounded.
=> A/B "randomness" must be salted (per-experiment, per-group), boundary-stable on edits, and layered for mutual exclusion (Iteration 2).
   It is NOT rand(); it is a deterministic function the whole fleet computes identically.

S4. Naive Architecture — the simplest thing that meets the FRs

The LC #380 structure, lifted. Each router pod, per pool, holds an array of healthy backends + a hashmap backend→index; pick_backend returns array[rand() % len] (uniform); on a backend add/remove, do the O(1) array swap. For A/B, a hashmap experiment_id → config; assign_bucket(u, e) = variants[ argmin_i cum_weight[i] > hash(u, e.id) % total_weight ]. The control plane pushes the full member list / full experiment config; each pod rebuilds. Every pod opens a connection to every backend in every pool it serves. Ejection = “delete from the array” (no probe-back — once gone, the only way back is the next control-plane push). It answers all the FRs (pick, bucket, eject, update). The numbers say where it breaks.

Numbers (the gap)

Hot-path latency:
  pick_backend  = array[rand()%len]: ~1 array read + a PRNG step ~= ~5 ns          -> p99 << 100 us  ✅
  assign_bucket = one hash + a small scan over <= ~10 variants ~= ~20-50 ns         -> p99 << 20 us   ✅
  -> latency is FINE in the Naive design. (It stays fine through every iteration -- the moves cost ~tens of ns more, not more.)

Churn handling:
  200 add/remove/sec x O(1) array swap ~= ~100 ns each = 20 us/sec of work          -> FINE  ✅  (this is exactly the LC trick's payoff)

Connection bound (THE failure):
  conns_per_backend = N_pods.   Global pool, N = 700:  700 conns/backend.
    ~700 x ~80 KB conn state ~= 56 MB just for router connections; 700 keepalive timers; FD limit pressure;
    a 50-backend "tiny popular" pool ALSO sees 700 conns/backend -> the backend pays the whole fleet's cost, not its traffic share.
  -> connections-per-backend grows with the ROUTER FLEET, unbounded.   FAILS the "<= C ~= 64-128, independent of fleet size" NFR.

Load-awareness:
  uniform random -> max-load bin ~ ln n / ln ln n above the mean; a cold-cache fresh-deploy backend gets full share immediately
  -> tail latency spikes + cascading overload on the unlucky backend.                FAILS the implicit "don't overload a backend" goal.

A/B correctness:
  edit any experiment -> total_weight changes -> hash % total_weight lands differently -> EVERY user can flip variant.
  -> experiment data corrupted on every edit; user sees UI flicker; two same-weight experiments confound each other.   FAILS A/B determinism.

Fail-static:
  control plane down -> pods keep their last config (the rebuild was on push, so nothing actively breaks)... EXCEPT an ejected
  backend can never come back (no probe loop) -> a transient blip permanently shrinks the pool until the next push.   PARTIAL FAIL.

Why it's broken (per failing NFR)

  • Connection bound — FAILS, structurally. conns_per_backend = N_pods grows with the router fleet and is independent of how much traffic the backend gets; a global pool, or any small pool, is hammered by the whole fleet. This is the shape-determining failure — the only fix is to stop every pod from talking to every backend. Classified connection-saturation.
  • Load-awareness — FAILS. Uniform random over the pool gives a hot bin ~ ln n / ln ln n above the mean and gives a cold/just-deployed backend instant full share → tail latency and cascading overload.
  • A/B determinism — FAILS. hash(u, e) % total_weight is brittle to any experiment edit; the population in “treatment” changes under you; same-weight experiments correlate.
  • Fail-static — PARTIAL. No probe-back loop, so a transient ejection is permanent until the next full push — a blip slowly shrinks the pool.
  • Latency & throughput — ✅. The O(1) structure is genuinely fine here, and stays fine through every iteration. The work isn't in the picker's CPU; it's in everything the picker's output touches.

What the Naive design actually costs: not dollars — the picker's own footprint is ~0% of the router pod, which exists for TLS+L7 anyway. It costs backend stability (connection saturation overloads the very services it routes to) and experiment validity (every edit corrupts the data). This is an infra primitive whose value is avoiding those costs, not generating one. The arc below adds four properties — bounded connections, load-awareness, sticky deterministic bucketing, fail-static config — and the dollar line barely moves (a small control plane). Name that up front: the optimization target here is the blast radius on dependencies, not the system's infra bill.

Iteration 0 — Deterministic per-pod subsetting

System after this iteration

Connections and line-style legend. Solid slate = sync hot path; dashed gray = async / config push; dashed red = (none).

  • ① Control plane → Router pods — pushes the pool membership in a canonical (sorted) order so every pod, given the same membership + its own pod_id, computes the same subset deterministically — no coordinator, no assignment service.
  • ② Router pod A → Backend pool — pod A connects to its subset of S backends only; it routes (still uniform within the subset — load-aware picking is Iteration 1) over those S.
  • ③ Router pod B → Backend pool — pod B's subset overlaps A's only on the ~S²/M backends both hash to; across the fleet each backend lands in ~N·S/M subsets → conns/backend ≈ N·S/M, which we tune via S to stay ≤ C regardless of N.

Delta vs Naive: a pod no longer routes to the whole pool — it routes to a deterministic, balanced, churn-stable subset of size S. Connections-per-backend goes from N (the fleet size, unbounded) to N·S/M (tunable). Nothing else changes — the picker within the subset is still uniform, ejection is still crude, bucketing is untouched.

Bottleneck

Naive's shape-determining failure is connection saturation: conns_per_backend = N_pods — a backend pays connection cost (memory, FDs, keepalive CPU) proportional to the entire router fleet, not to its traffic share, and it gets worse every time the platform adds router pods or a small pool gets popular. Classified connection-saturation — the signal: connection count on a downstream that scales with the client fleet, not the workload. Per bottleneck-to-decisions#connection-saturation: the first move is subsetting — have each client talk to a bounded, deterministically-chosen, balanced subset of the backends, so the per-backend connection count is N·S/M and you size S down as N grows. (The other levers — connection pooling, HTTP/2 multiplexing, a proxy layer — help the constant but don't change the O(N) shape.)

Architectural decisions to revisit

Per bottleneck-to-decisions#connection-saturation: the client–backend topology (full mesh → subsetted), the subset-selection algorithm (must be deterministic so pods agree without a coordinator, balanced so connection load is uniform, and minimal-disruption so a churn doesn't reshuffle every subset), and the subset size (a redundancy-vs-connection-budget knob). The iteration's move addresses the first two; the third becomes an explicit tunable.

Alternatives considered (C4)

  • Deterministic subsetting via a rendezvous (HRW) hash — chosen. For pod p, score each backend b by hash(p ‖ b) and take the top S. Every pod computes its own subset from the (sorted) membership + its pod_id — no coordinator. It's balanced (each backend is top-S for ~N·S/M pods in expectation) and minimal-disruption: a backend leaving removes it from the ~N·S/M subsets it was in, and each of those pods promotes the next-highest-scored backend — ~1 element changes per affected subset, not the whole subset, so connections are stable under the 200/sec churn. Cite consistent-hashing.html (rendezvous variant).
  • Round-robin / index-range subsetting (pod i gets backends [i·S, (i+1)·S) of the sorted list) — rejected. Deterministic and balanced, but not minimal-disruption: insert one backend near the front and every index shifts → every pod's window slides → the whole fleet re-connects — a connection storm on every deploy. (This is the “why consistent hashing” argument applied to subsets.)
  • A central subset-assignment service (the control plane computes each pod's subset and pushes it) — rejected. Works, but it's a new SPOF on the routing hot path's config, and it has to recompute and re-push ~700 subsets every time one backend churns — the deterministic-local computation gets the same result with zero coordination. (Constraint-flip: 🔀 if subsets had to respect cross-cutting constraints the pod can't see — e.g., “each pod's subset must span ≥ 2 zones for AZ-fault-tolerance, and zone topology is control-plane-only” — a central computation (or pushing zone tags to pods) becomes necessary.)
  • No subsetting; put a layer of aggregator proxies in front of each pool — rejected for pools in general, kept for tiny ones. Aggregators collapse N router connections into (#aggregators) connections per backend — but it's a whole extra hop, an extra fleet, and an extra failure domain. For a tiny pool where ceil(C·N/M) would force S below the redundancy floor, “don't subset, accept N connections” is fine (the absolute count is bounded and tiny pools are cheap); an aggregator layer is the escape hatch only if even that is too much.

The chosen move

Each router pod computes its subset of a pool with a rendezvous hash: score every backend b in the pool by h = hash(pod_id ‖ b.backend_id), take the S highest-scoring — that's the subset; route (uniformly, for now) over those S; open one connection to each. S is chosen as S = clamp(redundancy_floor=20, ceil(C_target · N_pods / M), M): at least 20 so a pod survives several backend failures in its subset; at most M; otherwise just big enough that conns/backend = N·S/M ≤ C_target. Worked: a global pool with N=700 pods, M=5000 backends, C_target=64 → ceil(64·700/5000)=9, below the floor, so S=20 → conns/backend = 700·20/5000 = 2.8 — far under C, and S=20 gives each pod a 20-deep set to fail over within. A bigger pool or fewer pods only makes the ratio smaller. The hot path costs nothing extra: the subset is recomputed only on a membership delta (~200/sec × an O(M) partial re-score with a small heap ≈ tens of µs, off the request path); pick_backend is still subset_array[rand() % S] — S small, one read, ~5 ns. Cite consistent-hashing.html (rendezvous / HRW) for the algorithm.

Iteration 1 — Weighted power-of-two-choices + outlier ejection with probe-back

System after this iteration

Connections and line-style legend. Solid slate = sync hot path; dashed gray = async / feedback; dashed red = probe / canary.

  • ② picker → Backends — weighted power-of-two-choices: draw two distinct backends from the subset with probability ∝ capacity_weight, read each one's pod-local in-flight-request counter, route to the lower; skip any backend whose ejected_until > now.   • ③ Backends / response → report_outcome — on completion the caller calls report_outcome(token, status); the picker decrements that backend's in-flight counter and, on a failure status, bumps its consecutive-failure count.
  • ④ report_outcome → picker — the updated counters feed the next pick (load) and the ejection decision.
  • ⑤ probe loop → ejected backend — a background loop sends an ejected backend a low-rate canary request; on success it clears ejected_until and the backend re-enters the pick set (with a brief slow-start ramp so it isn't slammed cold).

Delta vs Iteration 0: the within-subset picker changes from uniform-random to weighted power-of-two-choices; an explicit ejection side-array + exponential backoff + a probe-back loop replaces “delete on failure, never return.” The subset machinery, the connection bound, and the bucketing are unchanged.

Bottleneck

After Iteration 0 the connections are bounded, but the picker is still uniform over the subset — so the max-loaded backend runs ~ ln S / ln ln S above the mean (a noticeably hot one), and a backend that's just been deployed (cold cache, JIT warming, connection ramp) or is in a momentarily slow zone gets its full uniform share immediately → its latency spikes, callers retry, and the overload cascades. Classified tail-latency on the backend side (a long pole among otherwise-equivalent replicas), with a side of cold-start (a fresh replica can't absorb full share yet). Per bottleneck-to-decisions#tail-latency: load-aware selection (pick the less-loaded of a small random sample — power-of-two-choices), weighting (route in proportion to advertised capacity), and outlier ejection with slow-start (pull a bad one out, ramp a fresh one in). All three target the same bottleneck; they're one coherent move.

Alternatives considered (C4)

  • Weighted power-of-two-choices (P2C) + local in-flight counts + outlier ejection — chosen. P2C gets you ~all of the benefit of “always pick the global least-loaded” (max load ~ ln ln S vs ~ ln S / ln ln S for uniform) at the cost of one extra array read and a compare — no global state, no coordination, O(1). In-flight count is a good local load proxy for request-routing (it captures both slowness and queueing). Cite bottlenecks#tail-latency (P2C / "the power of two random choices").
  • Join-shortest-queue / least-loaded over the whole subset — rejected. Marginally better tail than P2C, but it's O(S) per pick (read all S counters) and — worse — it herds: every pod, seeing the same “least loaded”, piles onto it together until the counters update, causing oscillation. P2C's randomness is precisely what avoids the herd.
  • EWMA-of-latency weighting (route ∝ 1/recent_latency) — rejected as the primary signal, kept as a tiebreaker. Latency is laggy and noisy as a routing signal (it reflects load you caused several requests ago); in-flight count reacts immediately. We use in-flight as the P2C comparand and let the slower-moving latency/error EWMA influence the weight and the ejection threshold.
  • Round-robin within the subset — rejected. Deterministic and cheap, but not load-aware at all — a slow backend in the rotation gets exactly its 1/S share regardless of how slow it is, so it stays on the critical path. P2C with a load comparand routes away from it without ejecting it.

The chosen move

pick_backend within the subset becomes: draw two distinct backends, each with probability ∝ its capacity_weight (a small alias-table over the subset makes this O(1)); read each one's pod-local in-flight-request counter (a plain integer in the subset array, incremented on pick, decremented on report_outcome); route to the one with the lower count (ties → the one with the better recent latency/error EWMA); increment its counter. Backends with ejected_until > now are excluded from the draw. Outlier ejection: report_outcome(token, conn_refused | timeout | 5xx) increments a consecutive-failure counter; at a threshold (e.g., 5 in a row, or a 50%+ error rate over the last k) the backend is moved to an “ejected” side-list with ejected_until = now + base × 2^(ejection_count) (capped); a probe loop periodically sends an ejected backend a single canary request — on success it clears the ejection, halves the backoff toward base, and re-enters the pick set with a brief slow-start (its effective weight ramps from ~0 to full over a few seconds, so a cold replica isn't immediately slammed). All of this is pod-local (no shared state — each pod ejects based on what it observed; a backend bad for everyone gets ejected everywhere independently). Cost on the hot path: ~2 array reads + a compare + a counter increment ≈ ~tens of ns — the 100 µs budget is untouched. Cite bottlenecks#tail-latency (P2C) and failure-modes#slow-dependency (outlier ejection as the local degradation move).

Iteration 2 — Layered, salted, mutex-aware bucketing (the experiment side)

System after this iteration

Connections and line-style legend. Solid slate = sync hot path; dashed gray = config read / downstream; dashed red = (none).

  • ① Request → Step 1 (mutex group) — for experiment e in mutex group g: compute x = hash(user_id ‖ g.salt) normalized to [0,1); g has partitioned [0,1) into a fixed range per member experiment; if x falls in e's range the user is “in” e, else not_in_experiment (they're in some sibling, or unallocated). Adding an experiment to g carves a slice from the unallocated remainder — existing members' ranges don't move.
  • ② Step 1 → Step 2 (variant) — if the user is in e: y = hash(user_id ‖ e.id ‖ e.salt) in [0,1); the variant is the first one whose cumulative weight exceeds y. Editing e's variant weights without changing e.salt only moves users whose y sits near a shifted boundary — bounded churn; editing with a new salt is a deliberate full re-bucket (used when you want a clean restart, e.g., after a bug).
  • ③ Step 1/2 → synced config — every input (g.salt, the range table, e.id, e.salt, the variant weights, activate_at) comes from the config the control plane synced to every pod → assign_bucket is a pure function → the whole fleet computes the same answer for the same user.
  • ④ (downstream) analysis pipeline — consumes “user u was assigned variant v of experiment e” (logged with the request); out of scope here, but the determinism above is exactly what makes its population comparison valid.

Delta vs Iteration 1: assign_bucket changes from a single edit-brittle hash % total_weight to a two-step layered scheme keyed on per-group and per-experiment salts. The backend picker is untouched.

Bottleneck

The A/B side still uses variant = variants[hash(u, e.id) % total_weight]: editing any experiment changes total_weight (or the range layout) → the modulo lands differently → users flip variant mid-experiment → the “treatment” and “control” populations are no longer the same cohorts, so the experiment's measured effect is confounded with the cohort change, and the user sees the UI change under them; and two experiments with the same weights correlate (a user always-treatment in both), confounding their effects too. This is a correctness failure of the A/B FR, not a performance one — there's no canonical-bottleneck entry for “the math is wrong,” but the fix is a known pattern (Google/Microsoft “overlapping experiments” infrastructure; Optimizely/Statsig-style layered bucketing): make assignment a salted, boundary-stable, layered function. Per the data-model lens (api-design-patterns#idempotency — same input, same output, stable under unrelated changes): the assignment must depend only on (user, experiment-or-group, salt, weights) and must move the minimum number of users when any of those changes.

Alternatives considered (C4)

  • Layered, salted, mutex-group bucketing — chosen. A mutex group owns [0,1) and gives each member experiment a fixed sub-range (keyed on the group salt); within an experiment, a per-experiment salt + a cumulative-weight cut maps the user to a variant. Properties: deterministic and pod-independent (pure function of synced config); sticky-per-user (no per-user storage); salts decorrelate experiments; weight edits move only near-boundary users; mutex groups give “at most one experiment per group per user” with adding-an-experiment-disturbs-nobody. The industry-standard shape.
  • Store each user's assignment in a KV store on first sight (assignment-as-data) — rejected. Perfectly sticky — but it needs a write on first sight of every user×experiment (billions of rows), a read on every request (a network hop in the 20 µs budget — impossible), and it doesn't actually help with edits (you'd still have to decide whether an edit re-assigns). The deterministic-function approach gets stickiness for free with zero storage and zero hops.
  • Consistent-hash the user onto a ring of variant slots (Optimizely's older “bucketing”) — rejected as the primary mechanism. It's a fine way to make weight edits move few users within an experiment — but it doesn't compose for mutex groups (you'd need a ring per group and per experiment) and the simple “split [0,1) by cumulative weight + a salt” achieves the same boundary-stability for variant edits with no ring to maintain. We use the salted-interval-cut.
  • Per-request random assignment with a sticky cookie — rejected. Pushes the determinism into a client cookie — breaks for cookieless clients, breaks on cookie clear, and can't be recomputed server-side for offline analysis. The server-side pure function is the source of truth; a cookie can cache it, not replace it.

The chosen move

Model experiments in mutex groups (a.k.a. layers): a group g has a 64-bit salt and partitions [0,1) into a fixed range per member experiment plus an unallocated remainder; the range table is part of the synced config and changes only when an experiment is added (carving from the remainder) or removed (returning its range to the remainder — existing members untouched). assign_bucket(u, e): let g = e.mutex_group (or a degenerate group containing only e, owning all of [0,1), if e is unlayered). Compute x = hash64(u ‖ g.salt) / 2&sup6;⁴; if x ∉ range_g(e) return not_in_experiment. Else compute y = hash64(u ‖ e.id ‖ e.salt) / 2&sup6;⁴ and return variants[ first i where cum_weight[i] > y ]. Properties: deterministic (pure function of synced inputs → every pod, every request, agrees — satisfies the A/B-determinism NFR); sticky (depends on u, not on time or pod — no per-user storage); decorrelated (different salts → a user's membership in e₁ tells you nothing about e₂); boundary-stable on weight edits (changing variant weights without changing e.salt moves only users whose y lies between the old and new cumulative-weight boundaries — if you go 50/50→40/60 you move ~10% of e's users, the minimum possible); clean-restartable (bump e.salt to deliberately re-bucket everyone — rare, e.g., after a bug invalidated the data). Cost: 1–2 hash64 calls (~5–10 ns each) + a small scan over the variant cumulative weights (≤ ~10) + a range lookup ≈ ~20–50 ns — far inside the 20 µs budget. The control plane validates on edit: ranges within a group are non-overlapping and sum ≤ 1; variant weights within an experiment sum to 1; an edit that would re-bucket > a configured fraction without a salt bump is rejected with a warning (force the operator to choose: a small intentional shift, or an explicit salt-bumped restart). Cite the “overlapping experiments” pattern (Google, 2010) and api-design-patterns#idempotency for the “stable under unrelated changes” discipline.

Iteration 3 — Versioned-delta config distribution, staged activation, last-known-good fallback

System after this iteration

Connections and line-style legend. Solid slate = sync hot path (request-time); dashed gray = async config flow; dashed red = gap-recovery full-fetch.

  • ① SREs / discovery → Config store — pool membership churns (~200/s from rolling deploys) and experiment edits land in a versioned store; each write is validated (subset/range/weight invariants from Iter 0–2) and gets a monotonic version; experiment edits get an activate_at.
  • ② Config store → Fan-out tier — the store publishes versioned deltas to a small fan-out tier (a pub/sub or a streaming gRPC service, ~3 cheap nodes) that holds the current config and a tail of recent deltas.
  • ③ Fan-out tier → Router pods — each pod holds an open stream and applies deltas to its local cache; a new pool or experiment config is rolled to ~1% of pods first (a canary cohort) and a divergence/error-rate gate (no spike in 503s, no surge in re-bucketed users) must pass before the full rollout.
  • ④ local config cache → picker + bucketer — the picker and bucketer (Iter 0–2) read the local cache only; they never make a network call on a request and never block on the control plane.
  • ⑤ picker → Backends — unchanged; the local ejection loop keeps working even if the control plane is gone (a dead backend is ejected on local outcomes; a new backend just doesn't get traffic until the next config update).
  • ⑥ Router pod → Fan-out (gap recovery) — if a pod detects a version gap (a delta is missing) or the stream drops, it full-fetches the current config — with per-pod jitter so ~700 pods don't full-fetch simultaneously on a fan-out blip (stampede avoidance).

Delta vs Iteration 2: the config path becomes versioned-delta streaming with a small fan-out tier, canary rollout, staged activate_at, jittered gap-recovery, and an explicit fail-static contract (control plane down → pods run on last-known-good forever). No change to the picker or bucketer logic; +~3 fan-out nodes + the config store (a small Postgres + a pub/sub).

Bottleneck

After Iteration 2 every algorithm is right, but the distribution of the config that drives them is naive: full pushes (re-send the whole pool/experiment set on every change) at 200 churns/sec × ~700 pods, no version negotiation, no canary, no fail-static guarantee, and no atomic flip. Two failure shapes: (a) stampede — a config event (or a fan-out restart) makes all ~700 pods fetch the full config at once → a thundering herd on the control plane; (b) blackhole — the control plane down means pods either keep a stale config (acceptable!) or, worse, treat “can't reach control plane” as “empty config” and stop routing (catastrophic). Plus a correctness shape: a non-atomic experiment edit contaminates data for the ~1 s propagation window. Per bottleneck-to-decisions#stampede: versioned deltas (send the change, not the whole), jittered gap-recovery (don't all retry at once), a fan-out tier (absorb the read fan-out), a canary (a bad config can't take down the fleet), staged activation (flip atomically), and an explicit last-known-good contract (a missing control plane is fail-static, never fail-empty).

Alternatives considered (C4)

  • Versioned-delta streaming via a small fan-out tier + jittered gap-recovery + canary + activate_at + last-known-good — chosen. Deltas keep the per-event work tiny; the fan-out tier decouples ~700 long-lived streams from the config store; jitter kills the stampede; the canary bounds a bad-config blast radius to 1%; activate_at makes edits atomic; last-known-good makes the control plane non-load-bearing for routing. Standard config-distribution engineering (xDS/Envoy-control-plane shape).
  • Pods poll the config store directly every T — rejected. Simple, but ~700 pods polling every (say) 500 ms is 1.4 k QPS of full-config reads on the store, freshness is bounded by T (vs. ~ms for a stream), and a config-store hiccup either stalls all pods or (if you set a short timeout) triggers the very stampede you're avoiding. Streaming + a fan-out tier is strictly better at this scale.
  • Distribute config via a CDN / edge KV (signed config blobs at config.cdn/{version}) — rejected as the primary, kept as a backstop. Great for the “fail-static” story (the CDN is independently durable, pods can always fetch some recent version), and zero infra to run — but CDN propagation is seconds-to-minutes, not the ≤1 s the NFR wants for pool churn. Use the streaming path for freshness; mirror each version to the CDN as a disaster backstop a pod can fall back to if the fan-out tier is unreachable.
  • No fan-out tier; pods stream straight from the config store — rejected. Couples ~700 long-lived streams to the store's connection budget and restart behaviour; a store failover drops all 700 at once → mass gap-recovery. The fan-out tier is a cheap shock absorber (3 nodes ≈ $300/mo) that's worth it.

The chosen move

The control plane is a small config store (Postgres — the data is tiny and relational: pools, backends, experiments, mutex_groups, all versioned; chosen over a KV store because the validation queries are relational and the write rate — 200/s of single-row updates — is trivial for Postgres; over etcd because etcd's value-size and watch-fanout limits make it awkward at hundreds of churns/sec across hundreds of watchers) plus a fan-out tier (~3 stateless nodes streaming versioned deltas; chosen over reusing Kafka because the consumers want a compacted current state + a short delta tail, not a long log, and over a CDN-only approach because of the ≤1 s freshness need — though the CDN mirror is kept as a disaster backstop). Each router pod: holds an open delta stream; applies deltas to a local in-RAM config (the picker/bucketer read this, never the network); on a version gap or a dropped stream, full-fetches with per-pod random jitter (so a fan-out restart doesn't trigger 700 simultaneous fetches); on an experiment edit, the delta carries activate_at and the pod stages the new variant split, flipping at that wall-clock instant (within NTP skew). Rollout safety: a new pool or experiment config is delivered to a ~1% canary cohort of pods first; an automated gate (no 503 spike, no surge in re-bucketed users vs. the predicted churn, no error-rate regression) must pass before the fan-out promotes it to the rest. Fail-static contract, stated explicitly: if the control plane and the fan-out tier and the CDN backstop are all unreachable, every pod keeps routing on its last-known-good config indefinitely — a pool that's a few minutes stale is fine (the local ejection loop drops dead backends; a new backend just waits for traffic), and experiments simply don't change. The picker/bucketer never throw on a hot-path call: an unknown pool routes to a static fallback pool; an unknown experiment returns not_in_experiment. Cite failure-modes#stampede and cross-cutting#capacity (cold-start of a freshly-deployed pod = full-fetch from the fan-out tier).

What breaks (C5; cite failure-modes.html)

  • Bad config gets past the canary. A config that's wrong in a way the gate doesn't measure (e.g., it routes correctly but to a sub-optimal subset) reaches the fleet. Defence: keep the previous config version on every pod; a one-command rollback re-publishes version N−1 as the latest; the canary gate's metric set is itself reviewed when a miss happens. Detection: whatever the missed symptom was — add it to the gate.
  • Clock skew breaks activate_at. A pod with a 2-minute-skewed clock flips an experiment 2 minutes early or late → that pod's traffic is briefly on the wrong split. Defence: pods reject an activate_at that's in their past beyond a small tolerance (treat it as “activate now” and alert); rely on NTP keeping skew < ~50 ms; monitor per-pod clock offset. Detection: a clock-offset gauge; a config-version-vs-wall-clock check.
  • Fan-out tier total loss. All ~3 fan-out nodes gone → pods can't get deltas. Defence: this is exactly what last-known-good covers — routing continues on the stale config; pods fall back to the CDN config mirror for a coarse refresh; the fan-out tier is restored from the config store (it's stateless). Blast radius: config freshness degrades from ~ms to “frozen until recovery”; routing is unaffected. Detection: per-pod “config age” gauge climbing fleet-wide.
  • Stale-pod blackhole. One pod's stream silently wedges (it thinks it's up to date but isn't); it routes to backends that have been removed → elevated conn_refused. Defence: a heartbeat in the stream + a max-config-age watchdog (a pod whose config is older than T with no successful refresh marks itself unhealthy and is pulled from the LB); the local ejection loop also masks it (removed backends start refusing → get ejected). Detection: the per-pod config-age gauge; conn_refused rate by pod.

🔀 Constraint flip — if config had to be strongly consistent across the fleet (no version skew ever allowed)

You'd need a coordinated commit: the control plane proposes a new version, every pod acks it, then the control plane signals “activate” (a two-phase flip) — which means a single unreachable pod blocks the change, so you'd need a quorum-with-fencing and a way to evict a non-acking pod. That's a big cost for a guarantee this workload doesn't need: a few hundred ms of pool-membership skew is harmless (the retry covers it) and experiment skew is already handled by activate_at's wall-clock flip (which is “strong enough” — bounded by NTP skew, not by propagation). You'd only pay for true strong consistency if, say, the config carried a security policy where a stale pod is a vulnerability.

Probe iterations (the interviewer pulls the thread — beyond the 5-Core budget)

Each in a <details> block; same NFR-driven structure, one level deeper.

Probe A — Sticky sessions (session affinity to a backend)

Need. Some pools want the same session (cart, in-progress upload, a stateful protocol) to land on the same backend — but not at the cost of a connection storm when backends churn. Move. Maglev hashing: build a fixed-size lookup table (e.g., 65 537 slots) over the subset by Maglev's permutation algorithm; route by table[ hash(session_key) mod table_size ]. Maglev's whole point is minimal disruption: removing a backend reassigns only its ~1/S of slots, so ~1/S of sessions move, not all of them — vs. plain modulo-S (a churn re-maps everyone) and vs. ring/consistent-hash (more disruption than Maglev for the same table size, and a slower lookup). Build cost: O(S × table_size) once per membership change, off the request path; lookup: one array read, ~5 ns. Cite consistent-hashing.html (Maglev) and bottlenecks#hot-partition (a hot session key is a hot partition — mitigate by allowing the table to route a known-hot key to a small set, not a single slot).

Probe B — Cross-pod load awareness

Gap. P2C's in-flight count is pod-local — a backend being hammered by 50 other pods looks idle to this one, so this pod cheerfully adds to the pile. Move. Two options, used together: (1) backend-reported load — the backend includes a coarse load hint (queue depth, CPU%, “please back off” bit) in its responses / health pings; the picker folds it into the P2C comparand and the weight (Envoy's ORCA / “load report” mechanism); (2) error feedback — a backend that starts shedding (429 / 503-with-Retry-After) immediately drops in effective weight via the same EWMA that drives ejection. Don't gossip per-pod load directly (a 700-pod all-to-all gossip is its own scaling problem); let the backend, which sees all its load, be the aggregator. Cite bottlenecks#tail-latency.

Probe C — Experiment ramp & permanent holdback

Need. Roll a treatment variant 1% → 5% → 25% → 50% over days (catch problems while small) and keep a permanent ~1% holdout (measure the long-run cumulative effect of all shipped experiments). Move. A ramp is a sequence of weight edits with activate_at timestamps — and crucially, boundary-stable ones (Iter 2): 1%→5% moves the 1–5% slice of users into treatment, no one moves out, no one in the existing 1% flips — so the early cohort's data stays valid through the ramp. The holdback is a reserved range in a top-level mutex group (“global holdout owns [0, 0.01); no experiment may allocate there”) — those users are in no experiment, ever, by construction. Cite the Iter 2 layered-bucketing mechanism.

Probe D — Multi-region routing & global vs regional pools

Scope. “Which region” is a layer above this one (GeoDNS / anycast routes the client to the nearest healthy region; the router fleet there does the within-region pick). But a backend pool can be global (one logical service, instances in many regions) — then the pool's members span regions, and the subset/picker should prefer same-region members (a cross-region hop adds ~tens of ms): the picker tags each backend with its region and, in the P2C draw, biases toward in-region (or strictly in-region with cross-region as a fallback only when no in-region backend is healthy). The config plane is per-region, but a global pool's membership is the union (synced from all regions' discovery). Cite bottlenecks#cross-region; this is the boundary with #46 (multi-region active-active), which owns the “which region” question.

Degradation contract

When the control plane / fan-out / CDN are all unreachable → every pod keeps routing on its last-known-good config indefinitely; the local outlier-ejection loop still drops dead backends; new backends just don't get traffic until config flows again; experiments don't change. When a backend fails → it's ejected locally on consecutive failures (per pod, independently), with exponential backoff and a probe-back loop; if > 50% of a subset is ejected (a fleet-wide problem), the pod stops ejecting and routes to all of them anyway (don't route to nothing). When a pod's config wedges (stale beyond a watchdog threshold) → the pod marks itself unhealthy and is pulled from the load balancer; the local ejection loop also masks it (it's routing to removed backends → conn_refused → ejects them). When a config edit is bad → the 1% canary cohort catches it (no promotion) or a one-command rollback to version N−1; experiment edits flip atomically at activate_at, so there's no half-flipped fleet. The picker/bucketer never throw on a hot-path call: unknown pool → static fallback pool; unknown experiment → not_in_experiment. The invariant: routing degrades gracefully (staler, less load-aware) but never stops, and an experiment is never silently corrupted. Cite failure-modes#slow-dependency + #stampede.

Operations (C6)

  • Metrics (SLIs → NFRs). pick_backend and assign_bucket latency p99/p999 (→ the latency NFRs); per-backend connection count distribution (→ the connection-bound NFR — the math assumes balance, verify it); per-pod “healthy backends in subset” gauge + percent-of-subset-ejected (→ routing health, panic-mode trips); config age per pod + config-version histogram across pods (→ the freshness + atomic-flip NFRs); re-bucketed-users-per-edit (predicted vs. actual — a canary gate signal and an A/B-integrity alarm); P2C imbalance (max-backend in-flight / mean — should stay near a small constant); ejection rate, probe-back success rate; fan-out stream connection count + snapshot-fetch rate (stampede detector).
  • Rollout. Two rollout surfaces: (1) config — every new pool/experiment config canaries to ~1% of pods, an automated gate (latency flat, 503s flat, re-bucketed-users ≤ predicted) must pass, then promote; a one-command rollback re-publishes the previous version. (2) the picker binary — standard service rollout (canary a few pods, watch the SLIs, ramp), with the picker's own metrics (latency, imbalance, ejection rate) gating it.
  • Migration. From a Naive “uniform over the whole pool” picker: ship the new picker with subsetting + P2C behind a per-pool flag (still uniform over the subset = behaviourally close to before), flip pools one at a time watching connection counts and backend tails; ship the layered bucketer with a compatibility mode that reproduces the old hash % total_weight assignment for existing experiments (re-key only new ones) until each old experiment naturally ends; then drop the compat mode.
  • Headroom. Subset size S carries the redundancy headroom (a pod survives losing a chunk of its subset before it has zero healthy backends — S=20 floor); the fan-out tier is sized for ~2× the router fleet's streams + a deploy-time snapshot burst; the config store is wildly over-provisioned for ~200 writes/s (it's a small Postgres). Reason: router-fleet growth, region additions, and a deploy that restarts many pods at once.

  whole subset fails is stranded. We pick S = clamp(20, ceil(C*N/M), M): 20 is the redundancy floor; on big pools the C bound is what
  binds. Turning S up to, say, 64 on a 5000-backend / 700-pod pool gives 9 conns/backend instead of 2.8 -- still far under C -- and a
  64-deep failover set; we'd do that if backend-failure correlation were higher. There is no latency knob -- both calls are already ~ns.

Real-system comparison

The picker side is essentially Envoy's load-balancing subsystem re-derived from the LC kernel: Envoy does subsetting (its “subset load balancer” and the deterministic-aperture/subsetting work in the Envoy/gRPC ecosystem), power-of-two-choices (“least request” with a configurable choice count), outlier ejection with exponential backoff, slow-start, and a panic threshold — all the moves above, in production, at scale; the config plane is xDS (a streaming control-plane protocol — versioned, delta-capable, with ADS for ordering), which is the Iter 3 design. The bucketing side is the Google “overlapping experiments” infrastructure (Tang et al., 2010 — layers, domains, exclusivity, salted hashing) and its commercial descendants (Optimizely, LaunchDarkly, Statsig — deterministic bucketing keyed on a user id + a per-experiment salt, with mutual-exclusion groups). Where this page diverges from any of them: it's intentionally one coherent component (picker + bucketer in the same edge process) because the premise put them there; in reality the LB lives in the proxy (Envoy) and the experiment SDK is a separate library (often in the application, not the proxy) — the design moves are identical, the packaging differs. And a real Envoy deployment has many more knobs (zone-aware routing, retry budgets, circuit breakers, hedging) — several of which are this page's probes. Sources: see cribs/_sources.md for this scenario.

Anticipated pushbacks

Stress tests against the derivation above — depth rehearsal, not interview rehearsal.

What if the router fleet were 10× — 7,000 pods globally? (scale flip)

The whole point of subsetting is that this is a non-event for the backends: conns/backend = N·S/M, and you shrink S as N grows to keep the product under C — at N=7000, M=5000, C=64, S = clamp(20, ceil(64·7000/5000)=90, M) = 90 → conns/backend = 7000·90/5000 = 126, right at C; if you want margin, accept that the redundancy floor (20) on tiny pools now means tiny pools see up to 7000 conns — so put an aggregator layer in front of pools below ~C·floor backends. The fan-out tier scales linearly (more stateless nodes for more streams) and stays a rounding error. The config store is unbothered (still ~200 writes/s — that's a property of the deploy rate, not the fleet size). Nothing structural breaks; the only new decision is “aggregators in front of small pools.”

Why not just use Envoy + xDS and an off-the-shelf experiment SDK? (alt-engine)

You should — and the page says so (Real-system comparison): Envoy gives you subsetting, P2C/least-request, outlier ejection, slow-start, the panic threshold, and xDS streaming config out of the box; an SDK like Statsig/Optimizely/LaunchDarkly gives you deterministic salted layered bucketing. The reason to derive it here is to understand what those products are doing and why each knob exists — so when Envoy's tail latency spikes you know it's a P2C-comparand or a panic-threshold question, and when an experiment's numbers look wrong you know to check whether someone edited weights without a salt bump. The one place a from-scratch build is defensible: when the picker and the bucketer genuinely must be the same hot-path component (a constraint the premise imposed) and the SDK can't run inside the proxy — but even then you'd lift Envoy's algorithms, not reinvent them. Cite storage-engines#kv-engine only loosely — the config store is a small relational DB, not the interesting choice.

What changes if this routes regulated traffic (PCI / HIPAA / data-residency)? (regulatory)

The picker gains a placement constraint dimension: a request tagged (say) “EU-resident” may only be routed to backends in EU zones — so the subset is computed within the allowed zone set (the rendezvous top-S over the eligible backends only), and a pool that has no eligible backend for a request fails closed (a 451-style refusal), never falls back across the boundary. The experiment side: an experiment may be scoped to a jurisdiction (its mutex-group range only “applies” if the user is in-scope), and the “user u got variant v” log inherits the request's residency tag for the analysis pipeline. Audit: the config_versions change-log already gives who-changed-what-when; add who-routed-what-where sampling. What stays the same: subsetting, P2C, ejection, the bucketing math — regulation adds a filter on the candidate set and a fail-closed rule, not a new bottleneck. Cite cross-cutting#privacy.

The control plane is down right now — what actually happens? (failure)

Nothing visible to clients. Pods keep routing on their last-known-good config; the local outlier-ejection loop keeps dropping dead backends and probing them back, so transient backend failures still self-heal; a backend that gets added during the outage simply doesn't receive traffic until config flows again; experiments don't change variant splits or memberships. The only degradation is freshness: a backend permanently removed during the outage will be discovered “dead” by ejection (it starts refusing) within seconds anyway, so even that is masked. Recovery sequence: bring the fan-out tier back (stateless — it rehydrates from the config store or, if the store is also down, from the CDN snapshot); pods reconcile via the snapshot endpoint (jittered, so no stampede) and resume the delta stream. The watchdog (a pod whose config exceeds a staleness threshold marks itself unhealthy) is the backstop if a pod's stream is wedged in a way that isn't caught. Cite failure-modes#slow-dependency.

Where's the cost, and which knob moves it? (cost)

This component costs ~$1k/mo (a small Postgres pair + ~3 fan-out nodes + a CDN mirror that's effectively free) — a rounding error against the ~$146k/mo router fleet, which exists for TLS+L7 regardless and gains no pods from this primitive. So there's no “dominant cost” to optimize; the relevant lever is elsewhere: subset size S is a connections-vs-redundancy knob (bigger S → a deeper failover set per pod, but more connections and more connection memory on each backend — and connection memory on thousands of backends is real money), and the picker's quality (P2C, ejection, slow-start) is what lets backends run at higher utilization safely, which is where the actual savings are — a fleet that can run at 85% instead of 65% because the tail is controlled is ~25% fewer backend machines. So the “cost story” for this component is: it's cheap, and its job is to make the expensive things (backends) cheaper by behaving well toward them. Cite cost-shapes.html (an infra primitive's bill is small; its leverage is on its neighbours').

How do you roll this out from a Naive “uniform over the whole pool” picker without an incident? (migration)

Picker side: ship the new picker with subsetting + P2C behind a per-pool flag; with the flag off it's uniform-over-the-whole-pool (byte-identical behaviour); flip one pool to subsetting and watch its backends' connection counts drop and their tail latency; ramp pool-by-pool; then flip P2C on per pool watching the imbalance metric. A bad pool flips back instantly (it's a flag). Bucketer side: the hazard is re-bucketing existing experiments — so ship the layered bucketer with a compat shim that, for any experiment created before the cutover, reproduces the old hash(u, e.id) % total_weight assignment exactly (so no live user moves); only new experiments use the layered/salted path; as old experiments end naturally, their compat entries age out; once none remain, delete the shim. Rollback trigger at any step: a backend connection-count or tail-latency regression (picker), or any re-bucketed-users signal on an existing experiment (bucketer). The migration is “flag-flip per pool, compat-shim per old experiment” — no big-bang.

Production is slow / mis-routing right now — where do you look first? (monitoring/debug)

Decision tree off the SLI dashboard: (1) pick_backend p99 elevated? — almost never the picker itself (~ns); it's the backend the picker chose — look at per-backend tail latency and the P2C imbalance metric (is one backend hot? then check its ejection state — should it have been ejected?). (2) Elevated conn_refused from a pod? — that pod is routing to removed backends → check its config-age gauge (is its stream wedged? the watchdog should have pulled it — did it?). (3) Backend connection counts uneven / climbing? — subset-balance problem → check the rendezvous-hash distribution and whether S is sized right for the current N. (4) An experiment's numbers look off? — check the re-bucketed-users metric for that experiment around any recent edit (someone changed weights without a salt bump?), and the config-version histogram (did the activate_at flip converge, or is the fleet half-flipped?). (5) Everything stale fleet-wide? — the fan-out tier or config store is down → the degradation-contract path applies; restore the fan-out tier (stateless). Each branch maps to a What-Breaks entry and a runbook step; the “count the re-bucketed users” and “config-version histogram” instruments exist precisely so the A/B-integrity questions are a dashboard read.

S7. Key Takeaways

  • A cheap O(1) hot path can still be expensive — for its dependencies. The LC #380 picker costs nanoseconds, but its output (uniform over the whole pool) imposes connections-proportional-to-the-whole-router-fleet on every backend, ignores load, and corrupts experiments on every edit. Optimize what the primitive imposes on its neighbours, not its own CPU.
  • Subsetting decouples a dependency's connection count from your client-fleet size. Full mesh → conns/backend = N (unbounded). Deterministic, balanced, churn-stable subsetting → conns/backend = N·S/M, and you shrink S as N grows. Use a rendezvous (HRW) hash, not index-ranges, so a churn moves ~one element of each subset, not the whole window — the same “why consistent hashing” argument, applied to subsets.
  • Power-of-two-choices: ~all the benefit of “pick the global least-loaded” for the cost of one extra read. Uniform random over S backends leaves a hot one ~ln S / ln ln S above the mean; pick two at random and take the less-loaded and the max load collapses to a small constant above the mean — without the herding that “always pick the least-loaded” causes. Pair it with capacity weights, outlier ejection + probe-back, and slow-start for cold replicas.
  • “Random” for A/B is a deterministic function, not rand(). An assignment must be the same on every pod and stable across unrelated edits, or the experiment compares mixed cohorts. hash(user ‖ experiment ‖ salt) mapped through layered mutex-group ranges gives stickiness with zero per-user storage, decorrelated experiments, “at most one experiment per group per user,” minimal-disruption weight edits, and a deliberate clean-restart (bump the salt). The boundary-stability is what keeps a ramp's early cohort's data valid.
  • An infra primitive's config plane must be fail-static, not fail-empty. Versioned deltas (not full pushes), a fan-out tier (absorb the read fan-out), jittered gap-recovery (no stampede), a 1% canary (bound a bad config's blast radius), atomic activate_at flips (no half-flipped fleet), and — the load-bearing one — “control plane down ⇒ keep last-known-good forever, never treat unreachable-config as empty-config.”
  • Not every NALSD arc is a cost-reduction story. Here the dollar line barely moves (~$1k/mo of control plane on top of a router fleet that exists anyway). What the arc buys is bounded blast radius on dependencies, a controlled backend tail, valid experiments, and routing that survives a control-plane outage. A staff designer names the optimization target — here, “what we impose on our neighbours” — rather than reflexively chasing the bill.
