Sharding Chat Message History

Designing the storage layer for a chat application that must handle 200K message writes per second and 50 billion stored messages — and why sharding is the only option.

NALSD Walkthrough • ~20 min read

The Scenario

You are designing the storage layer for a large-scale chat application. Users send messages in conversations (1:1 chats and group chats). The primary operations are: (1) write a new message, and (2) retrieve message history for a conversation, paginated, newest first.

Scale Requirements

Users:                500 million registered
Daily active users:   100 million
Messages sent/day:    ~17 billion (200K/sec average, 500K/sec peak)
Stored messages:      50 billion (growing)
Total data size:      ~10 TB (avg 200 bytes/message)
Conversations:        ~2 billion total (1:1 and group)
Avg messages/conv:    25 messages (long tail: most convos are short,
                      some have 100K+ messages)

Latency Targets

Write latency (send message):       < 50 ms  (p99)
Read latency (load conversation):   < 100 ms (p99)

Access patterns:
  1. Send a message to conversation C
     → INSERT into messages WHERE conversation_id = C
  2. Load recent messages for conversation C (paginated)
     → SELECT * FROM messages WHERE conversation_id = C
       ORDER BY timestamp DESC LIMIT 50 OFFSET ...
  3. Get unread message count across all conversations for user U
     → COUNT(*) across ALL conversations user U belongs to
  4. Search messages (keyword search within a conversation)
     → Full-text search within conversation_id = C

Key insight: The primary access pattern (patterns 1 and 2) is conversation-centric — most queries target a single conversation. But pattern 3 (unread counts) is user-centric. This tension drives the entire design.

Back-of-Envelope Math

Before jumping to solutions, let’s calculate the hard numbers to prove that sharding is necessary — not just a nice-to-have.

Storage Calculation

Average message size:
  message_id (UUID):       16 bytes
  conversation_id (UUID):  16 bytes
  sender_id (UUID):        16 bytes
  body (text):             120 bytes avg
  timestamp:               8 bytes
  metadata (JSON):         24 bytes avg
  ─────────────────────────────────
  Total per message:       ~200 bytes

Total storage:
  50 billion messages × 200 bytes = 10 TB raw data

With indexes (conversation_id + timestamp composite index):
  Index overhead: ~30% of data = 3 TB
  Total with indexes: ~13 TB

With 3x replication:
  13 TB × 3 = 39 TB across the cluster

A single PostgreSQL instance can hold ~13 TB.
Storage alone is NOT the bottleneck. Write throughput IS.

Write Throughput Calculation

Required write throughput:
  Average: 200,000 writes/sec
  Peak:    500,000 writes/sec

What can a single PostgreSQL handle?
  With fsync (durability on): ~5,000-10,000 writes/sec
  With batching + async commit: ~15,000-20,000 writes/sec
  Absolute max with SSDs:      ~20,000 writes/sec

Gap:
  200,000 / 10,000 = 20x more than one node can handle
  500,000 / 10,000 = 50x at peak

Can replication help?
  NO. Replication adds read replicas, but ALL writes still
  go to the single primary node. The write throughput of a
  replicated cluster equals the write throughput of one node.

  Primary ← all 200K writes/sec (bottleneck!)
    ├── Replica 1 (handles reads only)
    ├── Replica 2 (handles reads only)
    └── Replica 3 (handles reads only)

Only option: SPLIT writes across multiple independent nodes.
  = Database sharding.

Minimum shards needed:
  200,000 / 10,000 = 20 shards (average load)
  500,000 / 10,000 = 50 shards (peak load)
  With 50% headroom: 40-75 shards

Let's plan for 64 shards (power of 2, convenient for hashing).

Read Throughput Calculation

Read patterns:
  100M DAU × 20 conversation opens/day = 2 billion reads/day
  2,000,000,000 / 86,400 ≈ 23,000 reads/sec average

Each read: SELECT ... WHERE conversation_id = C
           ORDER BY timestamp DESC LIMIT 50
  → Index scan on (conversation_id, timestamp DESC)
  → ~2-5 ms per query on PostgreSQL with warm cache

Single PostgreSQL can handle: ~5,000-10,000 reads/sec
23,000 reads/sec needs: 3-5 shards (reads alone are manageable)

But reads are bursty. A viral group chat might get
10,000 reads/sec on a single conversation (all on one shard).
So we need reads distributed too.

Conclusion: sharding is driven by WRITES, not reads.
Reads benefit from sharding as a side effect.

The math is clear: A single PostgreSQL maxes out at ~10K writes/sec. We need 200K writes/sec (20x gap). Replication cannot help because all writes funnel through one primary. We must shard the database across at least 40-75 independent nodes, each handling its own subset of writes.

Naive Design: Single PostgreSQL

Before sharding, let’s establish the simplest possible design. This is where every chat app starts — and it works fine until you outgrow one machine.

Schema

CREATE TABLE conversations (
  conversation_id  UUID PRIMARY KEY,
  type             VARCHAR(10),  -- '1:1' or 'group'
  created_at       TIMESTAMP DEFAULT NOW()
);

CREATE TABLE conversation_members (
  conversation_id  UUID REFERENCES conversations(conversation_id),
  user_id          UUID,
  joined_at        TIMESTAMP DEFAULT NOW(),
  last_read_at     TIMESTAMP,   -- for unread count
  PRIMARY KEY (conversation_id, user_id)
);

-- Index for "get all conversations for user X"
CREATE INDEX idx_members_user ON conversation_members(user_id);

CREATE TABLE messages (
  message_id       UUID PRIMARY KEY,
  conversation_id  UUID REFERENCES conversations(conversation_id),
  sender_id        UUID,
  body             TEXT,
  created_at       TIMESTAMP DEFAULT NOW()
);

-- The critical index: conversation history, newest first
CREATE INDEX idx_messages_conv_time
  ON messages(conversation_id, created_at DESC);

Query Performance

Load conversation history:
  SELECT * FROM messages
  WHERE conversation_id = 'abc-123'
  ORDER BY created_at DESC
  LIMIT 50;

  → Uses idx_messages_conv_time
  → B-tree index scan: O(log n) to find start + O(50) to read 50 rows
  → ~2-5 ms with warm cache ✓

Send a message:
  INSERT INTO messages (message_id, conversation_id, sender_id, body)
  VALUES (gen_random_uuid(), 'abc-123', 'user-456', 'Hello!');

  → Insert into heap + update 2 indexes (PK + conv_time)
  → ~3-5 ms with fsync ✓

Get unread count for user:
  SELECT cm.conversation_id, COUNT(m.message_id) AS unread
  FROM conversation_members cm
  JOIN messages m ON m.conversation_id = cm.conversation_id
    AND m.created_at > cm.last_read_at
  WHERE cm.user_id = 'user-456'
  GROUP BY cm.conversation_id;

  → Joins across tables, scans multiple conversations
  → ~50-200 ms depending on number of conversations ✓
  → Acceptable at small scale

Naive Architecture

This design works perfectly for a startup with <1M users and <5K writes/sec. Most chat apps never outgrow it. The problems only appear at massive scale, and they are all write-throughput problems.

What Breaks: The Write Throughput Ceiling

Let’s be precise about why a single database node fails and why replication is not the answer.

Why Single-Node Writes Have a Ceiling

Every INSERT into PostgreSQL must:
  1. Write to WAL (write-ahead log) on disk       → ~0.1 ms
  2. Insert row into the heap (table data)         → ~0.05 ms
  3. Update the primary key index (B-tree insert)  → ~0.05 ms
  4. Update idx_messages_conv_time (B-tree insert)  → ~0.05 ms
  5. fsync the WAL to guarantee durability          → ~0.05 ms
  ─────────────────────────────────────────────────
  Total per write: ~0.3 ms best case

  But these writes contend for:
    - Disk I/O bandwidth (WAL writes are sequential, but still bounded)
    - Lock contention on B-tree pages (concurrent inserts to same index leaf)
    - Buffer pool pressure (50B rows = 10 TB, only ~64 GB RAM cache)
    - CPU for index maintenance

  Practical ceiling: 5K-10K writes/sec per node
  Our requirement: 200K writes/sec
  Gap: 20-40x

Why Replication Doesn’t Help

Primary-replica replication:

  Client writes → Primary (accepts ALL writes)
                    ├── async replicate → Replica 1 (read-only)
                    ├── async replicate → Replica 2 (read-only)
                    └── async replicate → Replica 3 (read-only)

  Write throughput = throughput of Primary alone = 10K/sec
  Adding replicas increases READ throughput, not WRITE throughput.

  Even with 100 replicas:
    Write throughput is still 10K/sec ← the primary is the bottleneck

Multi-primary replication:
  Two primaries accepting writes → conflict resolution nightmare.
  Two users send messages to the same conversation simultaneously.
  Which message gets sequence number 42? Both primaries assigned it.
  Conflict resolution is complex and adds latency.

  This is why single-primary is the standard for strong consistency.

The ONLY way to scale writes:
  SPLIT the data so different subsets go to different nodes.
  Each node is a primary for its subset.
  = Sharding.

The fundamental constraint: In a replicated setup, every write must be processed by the primary. The primary is a single machine with finite I/O, CPU, and memory. Once you exceed what one machine can handle for writes, the only option is to split the data across multiple independent primaries — that is sharding.

Iteration 1: Shard by conversation_id

The most natural shard key for a chat application is conversation_id. All messages for a conversation land on the same shard, making conversation queries fast single-shard lookups.

Sharding Scheme

# Hash-based sharding: conversation_id → shard number
import hashlib

NUM_SHARDS = 64

def get_shard(conversation_id: str) -> int:
    """Deterministically map a conversation to a shard."""
    hash_bytes = hashlib.md5(conversation_id.encode()).digest()
    hash_int = int.from_bytes(hash_bytes[:4], 'big')
    return hash_int % NUM_SHARDS

# Example:
# conversation "conv-abc-123" → MD5 → 0x3A7F... → 3A7F hex = 14975 decimal
# 14975 % 64 = 31 → Shard 31

# All messages for "conv-abc-123" go to Shard 31.
# All messages for "conv-def-456" might go to Shard 12.
# The hash function distributes conversations roughly evenly.

Per-Shard Load

With 64 shards and even distribution:

  Write load per shard:
    200,000 writes/sec ÷ 64 shards = 3,125 writes/sec per shard
    Well within PostgreSQL's 10K/sec capacity ✓

  Data per shard:
    50 billion messages ÷ 64 shards = 781 million messages per shard
    10 TB ÷ 64 = 156 GB per shard
    Easily fits on one node with room to spare ✓

  Conversations per shard:
    2 billion conversations ÷ 64 = 31.25 million per shard

  Read load per shard:
    23,000 reads/sec ÷ 64 = 360 reads/sec per shard
    Trivial for PostgreSQL ✓

Sharded Architecture

Conversation Query (Single-Shard)

def get_conversation_history(conversation_id, page=0, page_size=50):
    """Load paginated message history for a conversation.
    This is a SINGLE-SHARD query — fast!"""

    shard = get_shard(conversation_id)  # hash → shard number
    conn = get_connection(shard)        # connect to correct PostgreSQL

    query = """
        SELECT message_id, sender_id, body, created_at
        FROM messages
        WHERE conversation_id = %s
        ORDER BY created_at DESC
        LIMIT %s OFFSET %s
    """
    return conn.execute(query, [conversation_id, page_size, page * page_size])

# Performance: index scan on (conversation_id, created_at DESC)
# Latency: 2-5 ms → well under 100 ms target ✓
# This hits exactly ONE shard — no scatter-gather needed.

Send Message (Single-Shard Write)

def send_message(conversation_id, sender_id, body):
    """Write a message. This is a SINGLE-SHARD write — fast!"""

    shard = get_shard(conversation_id)
    conn = get_connection(shard)

    message_id = uuid4()
    query = """
        INSERT INTO messages (message_id, conversation_id, sender_id, body)
        VALUES (%s, %s, %s, %s)
    """
    conn.execute(query, [message_id, conversation_id, sender_id, body])
    return message_id

# Performance: single INSERT + 2 index updates
# Latency: 3-5 ms → well under 50 ms target ✓
# This hits exactly ONE shard.

So far so good. Sharding by conversation_id makes the two most common operations (send message, load history) fast single-shard queries. But there are two serious problems lurking that this design cannot handle.

Problems with Shard-by-Conversation

Sharding by conversation_id optimizes for conversation-level queries. But two critical access patterns break badly.

Problem 1: Cross-Shard User Queries

Query: "Get all unread messages across all conversations for User Alice"

Alice is a member of 30 conversations.
These 30 conversations are spread across different shards
(because hash(conv_id) distributes them randomly).

To answer this query, we must:
  1. Look up which conversations Alice belongs to
     (from conversation_members table — but which shard is it on?)
  2. For EACH conversation, query the shard that holds it
  3. Count unread messages in each
  4. Aggregate results

This is a SCATTER-GATHER query:

  Router → Shard 3:   "unread count for conv-A where Alice is member?"
         → Shard 17:  "unread count for conv-B where Alice is member?"
         → Shard 42:  "unread count for conv-C where Alice is member?"
         → Shard 8:   "unread count for conv-D where Alice is member?"
         → ... (26 more shards)

  Wait for ALL 30 responses. Aggregate. Return to client.

Performance:
  Each individual query: 2-5 ms
  But we send 30 queries in parallel
  Latency = max(all 30) = ~10-20 ms (not terrible IF all succeed)

  But at 100M DAU, each user opens the app and wants unread counts:
    100M users × 1 scatter-gather request/day = 100M scatter-gathers/day
    Each scatter-gather fans out to ~10-30 shards
    Total shard queries: 100M × 20 avg = 2 billion extra queries/day
    = 23,000 extra queries/sec spread across shards

  This DOUBLES the read load on every shard.

Even worse: the conversation_members table is also sharded by
conversation_id, so "find all conversations for user X" is ITSELF
a scatter-gather across ALL 64 shards.

Problem 2: Hot Conversations

Not all conversations are equal.

Average conversation: 2 messages/day → negligible load
Active 1:1 chat:     50 messages/day → still fine
Active group (50 members): 500 messages/day → manageable

But consider:
  Company all-hands group: 10,000 members, 200 messages/hour
  Celebrity fan group:     50,000 members, 1,000 messages/hour
  Breaking news group:     100,000 members, 5,000 messages/hour

A single breaking-news group generates:
  5,000 messages/hour = 83 messages/min = 1.4 writes/sec (writes OK)
  BUT 100K members × open conversation = 100K reads when a message arrives
  All reads hit ONE shard (because all messages for this conv are there)

  100,000 reads in a burst = instant spike on one shard
  If 10 such hot conversations hash to the same shard:
    10 × 100K reads = 1M reads hitting one shard → overload

The hash function doesn't guarantee even LOAD distribution.
It guarantees even KEY distribution.
A shard with one hot conversation can receive 100x more traffic
than a shard with a million quiet conversations.

This is the "hot shard" problem — the bane of every sharded system.

Two fundamental tensions with sharding by conversation_id: (1) User-centric queries (unread counts, conversation list) require expensive scatter-gather across all shards because user data is scattered. (2) Hot conversations create hot shards because all traffic for one conversation hits one node. Both problems need explicit engineering to solve.

Iteration 2: Solving Cross-Shard Queries with Denormalization

The scatter-gather problem exists because user-centric data is spread across conversation-centric shards. The solution: maintain a second, denormalized copy of the data, sharded by user_id.

The Denormalization Strategy

Maintain TWO sharded tables:

1. messages table — sharded by conversation_id (already have this)
   Purpose: load conversation history (paginated, newest first)
   Write path: 1 write per message

2. user_inbox table — sharded by user_id (NEW)
   Purpose: "get all conversations with unread counts for user X"
   Write path: N writes per message (one for each participant)

Schema for user_inbox (on each shard):
  CREATE TABLE user_inbox (
    user_id          UUID,
    conversation_id  UUID,
    last_message_at  TIMESTAMP,
    last_message_preview TEXT,    -- "Alice: Hey, are you..."
    unread_count     INTEGER DEFAULT 0,
    PRIMARY KEY (user_id, conversation_id)
  );

  CREATE INDEX idx_inbox_user_time
    ON user_inbox(user_id, last_message_at DESC);

When User A sends a message to conversation C (with members A, B, C, D):
  1. Write to messages table → shard = hash(C) % 64  (1 write)
  2. Update user_inbox for EACH participant:
     → Update inbox for A → shard = hash(A) % 64     (1 write)
     → Update inbox for B → shard = hash(B) % 64     (1 write)
     → Update inbox for C → shard = hash(C_user) % 64 (1 write)
     → Update inbox for D → shard = hash(D) % 64     (1 write)

Total writes per message: 1 (messages) + N (inbox updates) = N+1
For a group with 50 members: 51 writes per message

Write Amplification Cost

Trade-off: fast reads at the cost of more writes.

Average conversation size: 2.5 participants (weighted average of
  1:1 chats and groups)

Without inbox denormalization:
  200K messages/sec × 1 write = 200K writes/sec

With inbox denormalization:
  200K messages/sec × 1 (message) = 200K writes/sec to message shards
  200K messages/sec × 2.5 avg participants = 500K writes/sec to inbox shards
  Total: 700K writes/sec across both shard sets

  But these writes are SPREAD across 64 shards per set:
    Message shards: 200K / 64 = 3,125 writes/sec per shard ✓
    Inbox shards:   500K / 64 = 7,812 writes/sec per shard ✓
    Both within PostgreSQL's 10K/sec capacity ✓

Storage overhead:
  Inbox table per user: 30 conversations × 100 bytes = 3 KB per user
  500M users × 3 KB = 1.5 TB total inbox data
  Small compared to 10 TB of messages.

The win: "get conversation list for user X" is now a SINGLE-SHARD query:
  SELECT conversation_id, last_message_at, last_message_preview, unread_count
  FROM user_inbox
  WHERE user_id = 'alice'
  ORDER BY last_message_at DESC
  LIMIT 20;

  → 1 shard, 1 index scan, ~2 ms ✓ (was 30-shard scatter-gather)

The Dual-Write Problem

When a message is sent, we need to write to TWO different shard sets:
  1. messages table (shard by conversation_id)
  2. user_inbox table (shard by user_id) — for EACH participant

These are on DIFFERENT shards (possibly different physical machines).
We CANNOT use a database transaction across shards (no distributed ACID).

What could go wrong:
  1. Write to messages shard succeeds
  2. Network error before writing to inbox shards
  3. Message exists in conversation history but inbox shows no new message
  → User B never sees the notification for the new message

This is the "dual-write" problem: writing to two independent data stores
atomically is impossible without a protocol.

Three approaches to solve it:
  A. Outbox pattern (most reliable)
  B. Kafka-based fan-out (most scalable)
  C. Accept eventual consistency and use retries (simplest)

Solution A: Outbox Pattern

def send_message_with_outbox(conversation_id, sender_id, body, participants):
    """Write message + outbox entry in ONE local transaction.
    A background worker reads the outbox and fans out to inbox shards."""

    shard = get_shard(conversation_id)
    conn = get_connection(shard)

    message_id = uuid4()

    # Single LOCAL transaction on the message shard
    with conn.begin():
        # 1. Insert the message
        conn.execute("""
            INSERT INTO messages (message_id, conversation_id, sender_id, body)
            VALUES (%s, %s, %s, %s)
        """, [message_id, conversation_id, sender_id, body])

        # 2. Insert into outbox table (SAME shard, SAME transaction)
        conn.execute("""
            INSERT INTO outbox (event_id, event_type, payload, created_at)
            VALUES (%s, 'new_message', %s, NOW())
        """, [uuid4(), json.dumps({
            'message_id': str(message_id),
            'conversation_id': str(conversation_id),
            'sender_id': str(sender_id),
            'body': body[:100],  # preview
            'participants': [str(p) for p in participants]
        })])

    # Transaction committed atomically.
    # If it fails, NEITHER the message NOR the outbox entry is written.
    # If it succeeds, BOTH are guaranteed to exist.

    return message_id

# Background worker (separate process) reads outbox and fans out:
def outbox_worker(shard):
    """Poll outbox table, process pending events."""
    while True:
        events = conn.execute("""
            SELECT * FROM outbox
            WHERE processed = FALSE
            ORDER BY created_at
            LIMIT 100
        """)
        for event in events:
            payload = json.loads(event.payload)
            for user_id in payload['participants']:
                update_user_inbox(user_id, payload)  # write to inbox shard
            mark_processed(event.event_id)

Solution B: Kafka-Based Fan-Out

"""
Kafka-based fan-out: write message to DB, publish event to Kafka,
consumers fan out to inbox shards.

This is more scalable than the outbox pattern because Kafka handles
the fan-out queue, and consumers can be horizontally scaled.
"""

# Producer (in the chat API server):
def send_message_kafka(conversation_id, sender_id, body, participants):
    shard = get_shard(conversation_id)
    conn = get_connection(shard)

    message_id = uuid4()

    # Step 1: Write message to DB
    conn.execute("""
        INSERT INTO messages (message_id, conversation_id, sender_id, body)
        VALUES (%s, %s, %s, %s)
    """, [message_id, conversation_id, sender_id, body])

    # Step 2: Publish to Kafka topic "message-events"
    # Kafka partition key = conversation_id (preserves ordering per conversation)
    kafka_producer.send('message-events', key=conversation_id, value={
        'message_id': str(message_id),
        'conversation_id': str(conversation_id),
        'sender_id': str(sender_id),
        'body_preview': body[:100],
        'participants': [str(p) for p in participants],
        'timestamp': datetime.utcnow().isoformat()
    })

    return message_id

# Consumer (separate service, horizontally scaled):
def inbox_fanout_consumer():
    """Kafka consumer that updates user_inbox shards."""
    for message in kafka_consumer.poll('message-events'):
        payload = message.value
        for user_id in payload['participants']:
            inbox_shard = get_shard(user_id)
            conn = get_connection(inbox_shard)
            conn.execute("""
                INSERT INTO user_inbox (user_id, conversation_id,
                    last_message_at, last_message_preview, unread_count)
                VALUES (%s, %s, %s, %s, 1)
                ON CONFLICT (user_id, conversation_id)
                DO UPDATE SET
                    last_message_at = EXCLUDED.last_message_at,
                    last_message_preview = EXCLUDED.last_message_preview,
                    unread_count = user_inbox.unread_count + 1
            """, [user_id, payload['conversation_id'],
                  payload['timestamp'], payload['body_preview']])

Dual-Write Fan-Out Architecture

Why Kafka Over Direct Writes?

Why not just write to all inbox shards directly from the API server?

Direct writes (no Kafka):
  API server sends message
    → write to message shard (5 ms)
    → write to inbox shard for User A (5 ms)
    → write to inbox shard for User B (5 ms)
    → write to inbox shard for User C (5 ms)
    → write to inbox shard for User D (5 ms)
  Total: 25 ms if sequential, 5 ms if parallel
  BUT: if any inbox write fails, message is sent but inbox is inconsistent.
  AND: the API server blocks until all writes complete.
  AND: for a group with 10,000 members, the API server must make
       10,000 network calls. Even at 1 ms each, that's 10 seconds
       of work per message.

Kafka-based fan-out:
  API server sends message
    → write to message shard (5 ms) ← user waits for this only
    → publish to Kafka (2 ms)       ← fire and forget
  Total user-perceived latency: 7 ms

  Kafka consumers (separate service, auto-scaled):
    → read event from Kafka
    → write to inbox shards for all N participants
    → if a write fails, Kafka retries (at-least-once delivery)
    → if consumer crashes, Kafka re-assigns partition to another consumer

Benefits:
  1. Decouples fast path (message write) from slow path (inbox fan-out)
  2. Automatic retries on failure (Kafka guarantees at-least-once)
  3. Backpressure: if inbox writes are slow, Kafka buffers events
  4. Horizontal scaling: add more consumers for higher throughput
  5. Replay: if inbox data gets corrupted, replay from Kafka topic

Handling Idempotency

"""
Kafka guarantees at-least-once delivery, which means the same
message event might be delivered to the consumer MORE than once
(e.g., consumer crashes after processing but before committing offset).

If we naively increment unread_count, a redelivered event would
increment it twice → wrong unread count.

Solution: make the inbox update idempotent using the message_id.
"""

def update_user_inbox_idempotent(user_id, payload):
    """Idempotent inbox update — safe to call multiple times."""
    inbox_shard = get_shard(user_id)
    conn = get_connection(inbox_shard)

    # Use message_id as an idempotency key.
    # The processed_messages table tracks which messages we've already handled.
    already_processed = conn.execute("""
        SELECT 1 FROM processed_messages
        WHERE user_id = %s AND message_id = %s
    """, [user_id, payload['message_id']])

    if already_processed:
        return  # Skip — already handled this event

    with conn.begin():
        # Update inbox
        conn.execute("""
            INSERT INTO user_inbox (user_id, conversation_id,
                last_message_at, last_message_preview, unread_count)
            VALUES (%s, %s, %s, %s, 1)
            ON CONFLICT (user_id, conversation_id)
            DO UPDATE SET
                last_message_at = EXCLUDED.last_message_at,
                last_message_preview = EXCLUDED.last_message_preview,
                unread_count = user_inbox.unread_count + 1
        """, [user_id, payload['conversation_id'],
              payload['timestamp'], payload['body_preview']])

        # Record that we processed this message (idempotency key)
        conn.execute("""
            INSERT INTO processed_messages (user_id, message_id, processed_at)
            VALUES (%s, %s, NOW())
        """, [user_id, payload['message_id']])

    # Both writes in one LOCAL transaction — atomic ✓
    # If replayed, the SELECT check catches it ✓

Iteration 2 summary: Denormalize user inbox data into a separate shard set keyed by user_id. Use Kafka-based fan-out to keep the inbox updated asynchronously. This turns cross-shard scatter-gather queries into fast single-shard lookups. The cost: 3.5x write amplification on average, and eventual consistency (inbox updates lag message writes by 50-200 ms).

Iteration 3: Solving Hot Shards

Even with 64 shards and good hash distribution, some conversations generate orders of magnitude more traffic than others. A single hot conversation can overwhelm the shard it lives on.

Quantifying the Hot Shard Problem

Normal load distribution (64 shards, 200K writes/sec):
  Average: 3,125 writes/sec per shard
  Capacity: 10K writes/sec per shard
  Headroom: 69% free ✓

Now add a breaking news event:
  A viral group chat (100K members) gets 5,000 messages/hour
  = 83 writes/sec to ONE shard

  Write load on that shard: 3,125 + 83 = 3,208 writes/sec
  Still fine for writes ✓

  BUT: 100K members open the conversation to read messages.
  Each member loads the conversation history:
    100,000 reads hitting ONE shard in a 10-second window
    = 10,000 reads/sec on that single shard

  Normal read load per shard: 360 reads/sec
  Hot shard read load: 360 + 10,000 = 10,360 reads/sec
  PostgreSQL capacity: ~5,000-10,000 reads/sec
  → Hot shard is OVERWHELMED ✗

  Even with 2 read replicas per shard:
    10,360 / 3 = 3,453 reads/sec per node
    Manageable, but just barely.

  If 3-5 hot conversations hash to the same shard:
    5 × 10,000 = 50,000 reads/sec on one shard
    50,000 / 3 = 16,667 per node → WAY over capacity ✗

Mitigation Strategy 1: Cache Hot Conversations

"""
For hot conversations, cache the recent message history in Redis.
Reads hit the cache instead of the PostgreSQL shard.

Redis can handle 100K+ reads/sec on a single instance — 10x more
than PostgreSQL.
"""

CACHE_TTL = 300  # 5 minutes
HOT_CONV_CACHE_SIZE = 50  # cache last 50 messages

def get_conversation_history_cached(conversation_id, page=0, page_size=50):
    """Check Redis cache first for hot conversations."""

    if page == 0:  # only cache first page (most requested)
        cache_key = f"conv_history:{conversation_id}"
        cached = redis.get(cache_key)
        if cached:
            return json.loads(cached)  # cache HIT → ~0.5 ms ✓

    # Cache miss → query PostgreSQL shard
    shard = get_shard(conversation_id)
    conn = get_connection(shard)
    messages = conn.execute("""
        SELECT message_id, sender_id, body, created_at
        FROM messages
        WHERE conversation_id = %s
        ORDER BY created_at DESC
        LIMIT %s OFFSET %s
    """, [conversation_id, page_size, page * page_size])

    if page == 0:
        redis.setex(cache_key, CACHE_TTL, json.dumps(messages))

    return messages

def on_new_message(conversation_id, message):
    """Invalidate/update cache when a new message arrives."""
    cache_key = f"conv_history:{conversation_id}"
    cached = redis.get(cache_key)
    if cached:
        messages = json.loads(cached)
        messages.insert(0, message)     # prepend new message
        messages = messages[:50]        # keep only 50
        redis.setex(cache_key, CACHE_TTL, json.dumps(messages))

Mitigation Strategy 2: Sub-Sharding by Time Range

For conversations with millions of messages (e.g., large long-running
group chats), even the data size on one shard can be problematic.

Sub-sharding: split a single conversation's messages across multiple
shards based on time ranges.

  Conversation "all-hands" has 10 million messages over 3 years:
    Sub-shard 0: messages from 2024-01 to 2024-06 (2M messages)
    Sub-shard 1: messages from 2024-07 to 2024-12 (2M messages)
    Sub-shard 2: messages from 2025-01 to 2025-06 (3M messages)
    Sub-shard 3: messages from 2025-07 to present  (3M messages) ← HOT

  Routing logic:
    - "Load recent messages" → always goes to the LATEST sub-shard
    - "Load older messages (pagination)" → goes to the relevant time-range sub-shard
    - "Send new message" → always goes to the LATEST sub-shard

  Benefits:
    - The hot sub-shard (latest) can be on beefier hardware
    - Old sub-shards are cold (rarely read) → can use cheaper storage
    - Reduces index size on the hot sub-shard → faster queries

  Cost:
    - More complex routing logic
    - Pagination across sub-shard boundaries requires stitching
    - Need a metadata table mapping conversation → sub-shard ranges

Mitigation Strategy 3: Dedicated Hot-Shard Capacity

Instead of sub-sharding, detect hot conversations and MOVE them
to a dedicated high-capacity shard pool.

Tier 1 shards (64 shards): Standard hardware, handle normal conversations
  - 16 CPU cores, 64 GB RAM, 1 TB SSD
  - Capacity: 10K writes/sec, 5K reads/sec per shard

Tier 2 shards (4 shards): Beefy hardware, handle hot conversations
  - 64 CPU cores, 256 GB RAM, 4 TB NVMe SSD
  - Capacity: 50K writes/sec, 30K reads/sec per shard
  - 8 read replicas each (vs 2 for Tier 1)

When a conversation is detected as "hot" (>100 reads/sec sustained):
  1. Mark it in a routing table: conversation_id → tier2_shard_X
  2. Start copying messages to the Tier 2 shard (background migration)
  3. Once caught up, update routing to point to Tier 2 shard
  4. Client requests for this conversation now go to Tier 2
  5. When it cools down (< 10 reads/sec for 1 hour), migrate back

Routing table (stored in Redis for fast lookup):
  hot_conversations = {
    "conv-breaking-news": {"shard": "tier2_shard_0", "since": "2025-03-22T10:00Z"},
    "conv-world-cup":     {"shard": "tier2_shard_1", "since": "2025-03-22T14:00Z"},
  }

The router checks this table FIRST. If the conversation is listed,
route to the Tier 2 shard. Otherwise, use hash(conv_id) % 64.

Iteration 3 summary: Hot shards are inevitable when data access is skewed. Three mitigation strategies work together: (1) Redis caching for hot conversations absorbs read spikes. (2) Time-based sub-sharding splits large conversations across multiple nodes. (3) Dedicated Tier 2 shards handle the hottest conversations with more resources. Detection via per-shard and per-conversation monitoring triggers automatic migration.

Iteration 4: Resharding Without Downtime

You launched with 64 shards. A year later, message volume doubled. You now need 128 shards. How do you add 64 more shards without dropping messages or taking the system offline?

Why hash(key) % N Is Disastrous for Resharding

Current mapping: shard = hash(conversation_id) % 64
New mapping:     shard = hash(conversation_id) % 128

Let's trace what happens to 10 conversation IDs:

  conv_id      hash value    % 64    % 128    Moved?
  ─────────    ──────────    ────    ─────    ──────
  conv-001     27391102       30      62       YES ✗
  conv-002     58204817       33      33       no ✓
  conv-003     91028445       29      93       YES ✗
  conv-004     12847293       29      93       YES ✗
  conv-005     74519283        3      67       YES ✗
  conv-006     33291847       55      55       no ✓
  conv-007     50182934       22      86       YES ✗
  conv-008     88391042       18      18       no ✓
  conv-009     41029385       41      105      YES ✗
  conv-010     67208421       21      85       YES ✗

Result: 7 out of 10 conversations must move to a different shard.

In general: going from N to 2N shards with modular hashing,
  roughly (N-1)/N ≈ 50% of all data must move.
  Going from N to M shards (non-power-of-2), it's even worse:
  up to ~(M-N)/M ≈ 90% of data moves.

For our system:
  50 billion messages × 50% = 25 billion messages to migrate
  At 200 bytes each = 5 TB of data to copy between shards
  At 100 MB/sec network throughput = 50,000 seconds = ~14 hours
  During which BOTH old and new locations must handle queries
  → Complex, error-prone, and risky

Solution: Virtual Shards (Logical-to-Physical Mapping)

Instead of mapping directly to physical shards:
  shard = hash(conv_id) % NUM_PHYSICAL_SHARDS     ← BAD

Map to a LARGE number of VIRTUAL (logical) shards, then map
virtual shards to physical shards via a lookup table:
  virtual_shard = hash(conv_id) % 1024             ← FIXED forever
  physical_shard = mapping_table[virtual_shard]     ← can change

Initial setup (64 physical shards, 1024 virtual shards):
  Virtual shards 0-15   → Physical shard 0
  Virtual shards 16-31  → Physical shard 1
  Virtual shards 32-47  → Physical shard 2
  ...
  Virtual shards 1008-1023 → Physical shard 63

  Each physical shard handles 1024/64 = 16 virtual shards.

Resharding from 64 to 128 physical shards:
  DON'T change the hash function. DON'T change virtual shard assignment.
  ONLY change the virtual → physical mapping.

  Before:
    Virtual shards 0-15 → Physical shard 0   (16 virtual shards)

  After:
    Virtual shards 0-7  → Physical shard 0   (8 virtual shards — stays)
    Virtual shards 8-15 → Physical shard 64  (8 virtual shards — moves)

  Only 50% of virtual shards move (the minimum possible).
  And the move is a well-defined, small unit: one virtual shard at a time.

Data movement:
  Each virtual shard contains: 50B / 1024 ≈ 49M messages ≈ 10 GB
  Moving one virtual shard: copy 10 GB at 100 MB/sec = 100 seconds
  Total: 512 virtual shards × 100 seconds = ~14 hours total
  BUT we can migrate 10 virtual shards in parallel = ~85 minutes
  AND we migrate one at a time, verifying each before proceeding.

Live Migration Procedure

"""
Migrate virtual shard V from physical shard A to physical shard B
without downtime.
"""

def migrate_virtual_shard(virtual_shard, source_physical, target_physical):
    # Phase 1: COPY — replicate data from source to target
    # During this phase, reads go to source, writes go to BOTH.

    set_migration_state(virtual_shard, 'COPYING')

    # Start copying existing data (background)
    copy_data(source_physical, target_physical, virtual_shard)

    # IMPORTANT: During copy, new writes go to BOTH shards.
    # The shard router checks migration state:
    #   if state == 'COPYING':
    #     write to source AND target (dual-write)
    #     read from source

    wait_for_copy_complete(virtual_shard)

    # Phase 2: VERIFY — ensure target has all data
    set_migration_state(virtual_shard, 'VERIFYING')

    row_count_source = count_rows(source_physical, virtual_shard)
    row_count_target = count_rows(target_physical, virtual_shard)
    assert row_count_source == row_count_target, "Row count mismatch!"

    # Phase 3: SWITCH — atomically update routing
    set_migration_state(virtual_shard, 'SWITCHING')

    # Update the mapping table (stored in ZooKeeper/etcd for consistency)
    update_mapping(virtual_shard, target_physical)

    # Now reads go to target. Writes go to target only.
    set_migration_state(virtual_shard, 'COMPLETE')

    # Phase 4: CLEANUP — delete data from source (delayed)
    schedule_cleanup(source_physical, virtual_shard, delay_hours=24)

# The shard router uses the migration state for routing:
def route_write(conversation_id):
    vs = hash(conversation_id) % 1024
    state = get_migration_state(vs)
    physical = get_mapping(vs)

    if state == 'COPYING':
        # Dual-write to ensure no data loss during migration
        source = get_source_shard(vs)
        target = get_target_shard(vs)
        write_to(source, conversation_id)
        write_to(target, conversation_id)
    else:
        write_to(physical, conversation_id)

Consistent Hashing Alternative

Virtual shards are the practical choice for databases.
Consistent hashing is the alternative used by Cassandra, DynamoDB, etc.

Consistent hashing places both keys and servers on a ring [0, 2^32):

  Server positions (hashed):
    Server A → position 1000
    Server B → position 3500
    Server C → position 7200
    Server D → position 9100

  To find the server for a key:
    hash(key) → position on ring
    Walk clockwise until you hit a server
    That server owns the key

  Adding Server E at position 5000:
    Only keys between 3500 and 5000 move (from C to E)
    All other keys stay put

  On average, adding 1 server to N servers moves only 1/N of the keys.
  For 64 → 65 servers: only 1/65 ≈ 1.5% of data moves.

Why we prefer virtual shards over consistent hashing for this system:
  1. Simpler to reason about (each virtual shard is a fixed set of keys)
  2. Migration is deterministic (you choose exactly which virtual shards move)
  3. No "ring rebalancing" complexity
  4. Works with any database (not just those with built-in consistent hashing)
  5. Can balance load precisely by assigning more virtual shards to beefier nodes

Iteration 4 summary: Never use hash % N directly for sharding — resharding is catastrophic. Use 1024 virtual (logical) shards mapped to physical shards via a routing table. Resharding becomes a matter of reassigning virtual shards and migrating their data. Each virtual shard is ~10 GB, a manageable migration unit. Live migration uses a three-phase process: COPY (dual-write) → VERIFY → SWITCH → CLEANUP.

What We Traded

Sharding gave us write scalability. But we paid for it with:

1. COMPLEXITY
   - 384+ database nodes to operate (vs 1 in naive design)
   - Shard routing logic in every query path
   - Virtual shard mapping table that must be consistent
   - Migration tooling for resharding

2. WRITE AMPLIFICATION
   - Every message generates N+1 writes (1 message + N inbox updates)
   - 3.5x average amplification (200K → 700K writes/sec)
   - For 10K-member groups: 10,001x amplification per message

3. EVENTUAL CONSISTENCY
   - Inbox updates lag message writes by 50-200 ms
   - User might send a message but inbox list doesn't update for 200 ms
   - Need to handle edge cases (show message immediately on sender's client)

4. NO CROSS-SHARD TRANSACTIONS
   - Can't atomically write a message and update all inboxes
   - Outbox pattern / Kafka provides at-least-once (not exactly-once)
   - Need idempotency keys to prevent duplicate inbox updates

5. HOT SHARD MANAGEMENT
   - Need monitoring, detection, and migration for hot conversations
   - Tier 2 shards are expensive (beefy hardware)
   - Migration between tiers has risk of data inconsistency

6. OPERATIONAL BURDEN
   - Resharding requires careful planning and execution
   - Each shard needs backups, monitoring, failover
   - Schema changes must be rolled out to 128+ shards
   - More failure modes (partial shard failures, network partitions between shards)

Key Takeaways

When to Shard

Shard when (and only when):
  1. Write throughput exceeds what one node can handle
  2. You've already optimized the single node (indexes, caching, query tuning)
  3. Replication can't help (it only scales reads, not writes)

Do NOT shard prematurely:
  - A single PostgreSQL handles 5K-10K writes/sec
  - That's enough for most applications (even with millions of users)
  - Sharding adds enormous complexity — it's a last resort
  - Start with one node, add read replicas, optimize queries
  - Only shard when the math proves you need it

Choosing a Shard Key

The shard key determines EVERYTHING:
  - Which queries are fast (single-shard) vs slow (scatter-gather)
  - Which operations can be transactional (same shard) vs eventually consistent
  - How evenly data and traffic are distributed

Good shard key properties:
  ✓ High cardinality (many unique values → even distribution)
  ✓ Aligns with the dominant access pattern (most queries are single-shard)
  ✓ Avoids hotspots (no single value gets disproportionate traffic)

For our chat app:
  conversation_id is good for message queries (pattern 1, 2)
  user_id is good for inbox queries (pattern 3)
  Neither is good for both → denormalize into two shard sets

Common mistake: sharding by user_id for messages
  → All of Alice's messages across ALL conversations on one shard
  → Conversation history query requires scatter-gather (Alice + Bob in same conv
    but on different shards)
  → Far worse than sharding by conversation_id

Sharding Introduces Five Hard Problems

1. CROSS-SHARD QUERIES
   Any query that doesn't align with the shard key requires
   scatter-gather across all shards. Solve with denormalization.

2. HOT SHARDS
   Skewed access patterns cause some shards to get 100x more traffic.
   Solve with caching, sub-sharding, and dedicated capacity.

3. RESHARDING
   Adding shards requires migrating data. hash % N is catastrophic.
   Solve with virtual shards or consistent hashing.

4. DISTRIBUTED TRANSACTIONS
   Can't use ACID across shards. Solve with outbox pattern,
   Kafka-based eventual consistency, or saga pattern.

5. OPERATIONAL COMPLEXITY
   N shards = N times the monitoring, backups, failover, and
   schema migration complexity. Automate everything.

Summary of Iterations

Iteration What Changed Problem Solved New Problem Introduced
Naive Single PostgreSQL Simple, correct, ACID Max 10K writes/sec (need 200K)
Iteration 1 Shard messages by conversation_id (64 shards) 200K writes/sec capacity Cross-shard user queries + hot shards
Iteration 2 Denormalized inbox sharded by user_id + Kafka fan-out Single-shard inbox queries Write amplification + eventual consistency
Iteration 3 Redis cache + Tier 2 shards + hot shard detection Hot shard overload More operational complexity
Iteration 4 Virtual shards (1024 logical → 64 physical) + live migration Resharding without downtime Routing table management
  • Sharding is a last resort — it solves write scalability but introduces five hard distributed systems problems. Only shard when the math proves a single node cannot handle the load.
  • The shard key dictates your query performance — operations aligned with the shard key are fast (single-shard); everything else is slow (scatter-gather). If you need fast queries on two different keys, denormalize into two shard sets.
  • Never use hash % N directly — use virtual shards or consistent hashing so that resharding moves only a fraction of data instead of nearly all of it.
  • Hot shards are inevitable — real-world access patterns are skewed. Build detection and mitigation (caching, dedicated capacity, sub-sharding) from day one.
  • Denormalization trades write amplification for read performance — writing the same data in multiple shard sets (once by conversation, once by user) costs more writes but eliminates expensive scatter-gather reads.
  • Eventual consistency is the price of sharding — without cross-shard transactions, updates to denormalized copies are asynchronous. Design your application to tolerate stale reads for short periods (50-200 ms).
