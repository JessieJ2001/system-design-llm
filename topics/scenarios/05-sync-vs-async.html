Scenario #5: Sync vs Async Communication

An e-commerce order pipeline that works fine under normal load — until a payment provider has a bad day and takes your entire site down with it. The fix: decouple with async messaging.

~15 min read

The Scenario

You are designing the order processing pipeline for an e-commerce platform. When a customer places an order, the system must perform five steps in sequence:

Step Operation Latency Dependency
1 Create Order 10 ms (local DB write) None
2 Process Payment 2–5 sec (external Stripe/PayPal API) Order must exist
3 Reserve Inventory 100 ms (internal service) Payment must succeed
4 Schedule Shipping 500 ms (external shipping API) Inventory reserved
5 Send Confirmation Email 200 ms (SMTP relay) Shipping scheduled

Traffic: 1,000 orders/min at peak (e.g., flash sale). That is roughly 17 orders per second.

Key observation: Step 2 (payment) dominates latency. It is an external API you do not control, and its latency can swing wildly—from 2 seconds on a good day to 30+ seconds when the provider is degraded.

Back-of-Envelope Math

Normal Conditions

End-to-end latency (synchronous):
  Create order:    10 ms
  Payment:       2000 ms  (best case)
  Inventory:      100 ms
  Shipping:       500 ms
  Email:          200 ms
  ─────────────────────
  Total:        ~2810 ms  ≈ 3 seconds

Orders per second:          17 orders/sec
Thread hold time:           3 sec per order
Concurrent threads needed:  17 × 3 = 51 threads

Typical thread pool size:   200 threads
Utilization:                51 / 200 = 25.5%   ← Looks fine!

At 25% utilization, the synchronous approach seems perfectly viable. The thread pool has plenty of headroom. So why worry?

Degraded Conditions (Payment API Slow)

Payment API latency spikes to 30 seconds (not unusual during outages).

End-to-end latency:
  Create order:     10 ms
  Payment:       30000 ms  (10x slower)
  Inventory:       100 ms
  Shipping:        500 ms
  Email:           200 ms
  ─────────────────────
  Total:        ~30810 ms  ≈ 31 seconds

Concurrent threads needed:  17 × 31 = 527 threads

Thread pool size:           200 threads
Utilization:                527 / 200 = 263%   ← OVERLOADED

Requests queuing up:        527 - 200 = 327 requests waiting
Queue grows at:             17/sec - (200/31) = ~10.5 requests/sec
Time to exhaust queue:      Within seconds

The cascade: The thread pool is exhausted. New orders queue up. Users see timeouts after 30+ seconds. But here is the real disaster: the order API typically shares a server (and thread pool) with other endpoints—product pages, search, cart. When the order thread pool fills up, every endpoint on the server becomes unresponsive. The entire site goes down because of a slow payment provider.

Naive Design: Synchronous Chain

The simplest implementation: each step makes a synchronous REST call to the next service and waits for the response before proceeding.

# Naive synchronous order processing
def place_order(request):
    # Step 1: Create order in DB
    order = db.insert("orders", {
        "user_id": request.user_id,
        "items": request.items,
        "status": "created"
    })

    # Step 2: Process payment (BLOCKS for 2-30 seconds!)
    payment = payment_api.charge(
        amount=order.total,
        card_token=request.card_token
    )  # ← Thread sits idle here, waiting for Stripe

    if not payment.success:
        db.update("orders", order.id, {"status": "payment_failed"})
        return Response(400, "Payment failed")

    # Step 3: Reserve inventory (blocks 100ms)
    inventory = inventory_service.reserve(order.items)

    # Step 4: Schedule shipping (blocks 500ms)
    shipping = shipping_api.create_shipment(order.id, request.address)

    # Step 5: Send confirmation email (blocks 200ms)
    email_service.send_confirmation(request.email, order.id)

    db.update("orders", order.id, {"status": "completed"})
    return Response(200, {"order_id": order.id})

Architecture Diagram

What Is Wrong With This?

  1. Tight coupling: If any downstream service is slow or down, the order API is directly affected. The slowest service dictates overall latency.
  2. Resource waste: The thread is blocked for 3–31 seconds doing nothing—just waiting for network responses.
  3. No partial progress: If shipping fails after payment succeeded, you have a payment with no shipment. Recovering requires manual intervention.
  4. Blast radius: Thread pool exhaustion affects all endpoints, not just orders.

What Breaks (With Numbers)

Failure Timeline

Here is exactly what happens when the payment API degrades, minute by minute:

Time Payment Latency Threads in Use Thread Pool (200) User Experience
T+0 2 sec (normal) 51 25% used Normal
T+1 min 10 sec (degrading) 187 93% used Slightly slower
T+2 min 20 sec 357 100% — SATURATED Timeouts begin
T+3 min 30 sec 527 needed, 200 available 100% + queue overflow Product pages down
T+5 min 30 sec (still degraded) All blocked Connection refused Entire site unreachable

The key insight: A slow dependency is often worse than a failed dependency. If payment returned an error in 100ms, you could show an error message quickly and release the thread. But a 30-second hang holds the thread hostage, consuming resources while providing no value.

Partial Failure Problem

Even when things are fast, the synchronous chain has a partial-failure problem:

Step 1: Create order      ✓  (order exists in DB)
Step 2: Process payment   ✓  (card charged $150)
Step 3: Reserve inventory ✗  (out of stock!)

Now what?
  - Customer was charged $150
  - No inventory reserved
  - No shipping scheduled
  - Manual intervention needed to issue refund
  - At 17 orders/sec, even 0.1% partial failures = 1 failure/min = 61/hour

Iteration 1: Make It Async

The core idea: instead of the order API doing everything synchronously, it does the minimum necessary work and returns immediately. Everything else happens asynchronously via a message queue.

New Architecture

How It Works Step by Step

  1. Order API writes the order to the database with status "pending" and publishes an order.created event to the message queue. Returns 202 Accepted to the user in ~50ms.
  2. Payment Worker picks up the order.created event, calls the payment API, and publishes payment.succeeded or payment.failed.
  3. Inventory Worker picks up payment.succeeded, reserves inventory, publishes inventory.reserved.
  4. Shipping Worker picks up inventory.reserved, calls the shipping API, publishes shipping.scheduled.
  5. Email Worker picks up shipping.scheduled, sends the confirmation email.
# Async order processing - Order API (fast path)
def place_order(request):
    # Step 1: Create order in DB (10ms)
    order = db.insert("orders", {
        "user_id": request.user_id,
        "items": request.items,
        "status": "pending",
        "idempotency_key": request.idempotency_key
    })

    # Step 2: Publish event to queue (5ms)
    queue.publish("order.created", {
        "order_id": order.id,
        "user_id": request.user_id,
        "items": request.items,
        "card_token": request.card_token,
        "address": request.address,
        "email": request.email
    })

    # Return immediately - user does NOT wait for payment
    return Response(202, {
        "order_id": order.id,
        "status": "pending",
        "message": "Order received. You will be notified when confirmed."
    })


# Payment Worker (runs independently)
def handle_order_created(event):
    order_id = event["order_id"]

    try:
        payment = payment_api.charge(
            amount=calculate_total(event["items"]),
            card_token=event["card_token"]
        )
        db.update("orders", order_id, {"status": "payment_succeeded"})
        queue.publish("payment.succeeded", {
            "order_id": order_id,
            "payment_id": payment.id,
            "items": event["items"],
            "address": event["address"],
            "email": event["email"]
        })
    except PaymentError as e:
        db.update("orders", order_id, {"status": "payment_failed"})
        queue.publish("payment.failed", {
            "order_id": order_id,
            "reason": str(e)
        })

Why This Is Better: The Numbers

User-facing latency:
  Before (sync):  2810 ms best case, 30810 ms degraded
  After (async):    50 ms always (DB write + queue publish)

Thread hold time:
  Before (sync):  3 sec best, 31 sec degraded
  After (async):  50 ms always

Threads needed at 17 orders/sec:
  Before (sync):  51 threads (normal), 527 threads (degraded)
  After (async):  17 × 0.05 = ~1 thread

Impact of payment API degradation:
  Before (sync):  Entire site goes down
  After (async):  Queue grows. Users still browse, add to cart, place orders.
                  Orders just take longer to confirm.

Queue Depth During Degradation

Normal: Payment API processes at ~5 orders/sec (2s each)
        Queue production:   17/sec
        Queue consumption:   5/sec
        Net queue growth:   12/sec

After 5 minutes of degradation:
        Queue depth: 12 × 300 = 3,600 items
        Memory per item: ~1 KB (JSON payload)
        Total queue memory: 3,600 × 1 KB = 3.6 MB  ← trivial for Kafka/RabbitMQ

Recovery time after payment API returns to normal:
        Backlog: 3,600 items
        Processing rate: 5/sec (even at normal 2s payment time)
        Drain time: 3,600 / 5 = 720 sec = 12 minutes

        With consumer scaling (10 workers × 5/sec = 50/sec):
        Drain time: 3,600 / 50 = 72 seconds  ← much better

The queue acts as a shock absorber. Traffic spikes and downstream slowdowns are absorbed by the queue growing. Nothing crashes. Nothing times out. The system degrades gracefully: orders take longer to process, but the website stays up and users can keep shopping.

Iteration 2: Idempotency, Retries, and Compensation

The async design solves the performance and coupling problems, but introduces new challenges. Distributed systems are messy: messages can be delivered more than once, workers can crash mid-processing, and partial failures still happen.

Problem 1: Duplicate Messages

If a payment worker processes a message, calls Stripe, and then crashes before acknowledging the message, the queue will redeliver it. Without protection, the customer gets charged twice.

# Idempotent payment processing
def handle_order_created(event):
    order_id = event["order_id"]
    idempotency_key = f"payment-{order_id}"

    # Check if already processed
    existing = db.query(
        "SELECT status FROM orders WHERE id = ? AND status != 'pending'",
        [order_id]
    )
    if existing:
        return  # Already processed, skip

    try:
        # Stripe accepts idempotency keys natively
        payment = payment_api.charge(
            amount=calculate_total(event["items"]),
            card_token=event["card_token"],
            idempotency_key=idempotency_key  # Stripe deduplicates
        )
        db.update("orders", order_id, {
            "status": "payment_succeeded",
            "payment_id": payment.id
        })
        queue.publish("payment.succeeded", {"order_id": order_id, ...})
    except PaymentError as e:
        db.update("orders", order_id, {"status": "payment_failed"})
        queue.publish("payment.failed", {"order_id": order_id, ...})

Idempotency key rule: Every operation that can be retried must have an idempotency key derived from the input (e.g., payment-{order_id}). If the same key is seen twice, the operation is a no-op. This makes at-least-once delivery safe.

Problem 2: Partial Failures Need Compensation

What if payment succeeds but inventory reservation fails (item went out of stock between order placement and reservation)? You need compensating actions—the distributed equivalent of a database rollback.

# Inventory worker with compensation
def handle_payment_succeeded(event):
    order_id = event["order_id"]

    try:
        reservation = inventory_service.reserve(event["items"])
        db.update("orders", order_id, {"status": "inventory_reserved"})
        queue.publish("inventory.reserved", {
            "order_id": order_id,
            "reservation_id": reservation.id,
            ...
        })
    except OutOfStockError:
        # Compensate: refund the payment
        db.update("orders", order_id, {"status": "inventory_failed"})
        queue.publish("refund.requested", {
            "order_id": order_id,
            "payment_id": event["payment_id"],
            "reason": "out_of_stock"
        })
        queue.publish("notification.send", {
            "order_id": order_id,
            "email": event["email"],
            "template": "out_of_stock"
        })

Problem 3: Poison Messages

Some messages may never succeed no matter how many times you retry—a corrupt payload, a bug in the handler, an invalid card token. Without a safety valve, the worker retries forever and blocks the queue.

Retry policy:
  Attempt 1:  immediate
  Attempt 2:  after 1 sec   (exponential backoff)
  Attempt 3:  after 4 sec
  Attempt 4:  after 16 sec
  Attempt 5:  GIVE UP → move to Dead Letter Queue (DLQ)

DLQ processing:
  - Alert on-call engineer
  - Store full event payload for debugging
  - Provide a "replay" button in admin dashboard
  - Daily report: "14 orders failed permanently today"

Order Status Tracking

Since the user got a 202 Accepted immediately, they need a way to check order status. Each worker updates the order status in the database as it processes:

Status Set By Meaning
pending Order API Order created, waiting for payment
payment_processing Payment Worker Payment API called, waiting for response
payment_succeeded Payment Worker Card charged successfully
payment_failed Payment Worker Card declined or payment error
inventory_reserved Inventory Worker Items reserved in warehouse
shipping_scheduled Shipping Worker Shipping label created
confirmed Email Worker Confirmation email sent, order complete

The frontend polls GET /orders/{id}/status every few seconds, or you use WebSockets/SSE for real-time updates.

Iteration 3: Hybrid Sync + Async

There is one problem with fully async: the user does not know if their payment was declined until later. For many e-commerce flows, the user needs immediate feedback on payment. They are sitting on the checkout page waiting.

The Hybrid Approach

Make payment synchronous (user waits for it), but everything after payment is async. This gives the user immediate feedback on the critical step while still decoupling the rest of the pipeline.

# Hybrid: sync payment, async everything else
def place_order(request):
    # Step 1: Create order (10ms)
    order = db.insert("orders", {
        "user_id": request.user_id,
        "items": request.items,
        "status": "pending"
    })

    # Step 2: Payment - SYNCHRONOUS (user waits)
    try:
        payment = payment_api.charge(
            amount=order.total,
            card_token=request.card_token,
            timeout=10  # 10-second timeout, don't wait forever
        )
    except TimeoutError:
        db.update("orders", order.id, {"status": "payment_timeout"})
        return Response(202, {
            "order_id": order.id,
            "status": "payment_pending",
            "message": "Payment is being processed. We will email you."
        })
    except PaymentDeclined:
        db.update("orders", order.id, {"status": "payment_failed"})
        return Response(400, {"error": "Card declined. Please try another card."})

    # Payment succeeded - everything else is async
    db.update("orders", order.id, {
        "status": "payment_succeeded",
        "payment_id": payment.id
    })

    # Publish for async processing
    queue.publish("payment.succeeded", {
        "order_id": order.id,
        "payment_id": payment.id,
        "items": request.items,
        "address": request.address,
        "email": request.email
    })

    # User sees success immediately
    return Response(200, {
        "order_id": order.id,
        "status": "confirmed",
        "message": "Payment processed! Your order is being prepared."
    })

Comparing All Three Approaches

Property Fully Sync Fully Async Hybrid
User-facing latency 3–31 sec 50 ms 2–10 sec
Payment feedback Immediate Delayed (minutes) Immediate
Blast radius of slow payment Entire site down Queue grows, site fine Checkout slower, site fine
Threads held per order 1 for 3–31 sec 1 for 50 ms 1 for 2–10 sec
Complexity Low High Medium
Partial failure handling Manual Compensating actions Compensating actions

The hybrid approach is the most common in production. Amazon, Shopify, and most e-commerce platforms use synchronous payment (user needs to know if the card works) and asynchronous fulfillment (inventory, shipping, email happen in the background).

Final Design

Putting it all together: the hybrid async architecture with idempotency, compensation, dead letter queues, and status tracking.

Final Numbers

Normal operation:
  User latency:           2-5 sec (payment) + 50ms overhead = ~2-5 sec
  Order confirmation:     Background in ~800ms after payment
  Throughput:             17 orders/sec, easily handled
  Queue depth:            Near zero (consumers keep up)

Payment API degraded (30s):
  User latency:           10 sec timeout → "payment pending" response
  Queue depth:            Grows at ~12/sec
  Site health:            100% operational (product pages, search, cart unaffected)
  Recovery:               Queue drains in ~72 sec with 10 workers

Payment API down:
  User experience:        "Payment is processing" after timeout
  Circuit breaker:        Opens after 5 failures, fast-fails for 30 sec
  Queue:                  Messages accumulate, processed when API recovers
  No data loss:           All orders in DB + queue, nothing dropped

Key Takeaways

  1. Synchronous for operations where the user needs an immediate answer (payment declined? try another card). Async for everything else (inventory, shipping, email).
  2. Async decouples failure domains. A slow payment provider does not take down product pages. A crashed email service does not block shipping. Each service fails independently.
  3. Calculate queue growth rate to ensure your queue can handle degraded periods. At 12 messages/sec growth, a 5-minute outage produces only 3.6 MB of backlog—trivial for any modern message broker.
  4. Always plan compensating actions for partial failures. If step 3 fails after step 2 succeeded, you need an automated way to undo step 2 (refund the payment).
  5. Idempotency keys are non-negotiable in async systems. Messages will be delivered more than once. Every handler must be safe to run multiple times with the same input.
  6. A slow dependency is worse than a dead dependency. Dead services fail fast (connection refused in milliseconds). Slow services hold threads for seconds or minutes. Use timeouts and circuit breakers to convert slow failures into fast failures.
