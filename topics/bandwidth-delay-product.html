Bandwidth-Delay Product

The single number that tells you how much data can be “in flight” at once — whether across a network, into a disk, or through a memory bus. Misunderstanding it means you leave most of your hardware’s throughput on the table.

25 min read

What Is It?

The Pain Point: Why Isn’t My Fast Link Actually Fast?

Imagine you upgrade your server’s network from 1 Gbps to 10 Gbps. You expect 10x throughput. But when you measure an actual file transfer across the country, it barely improves. The link is 10x wider, so where did the speed go?

The problem is that bandwidth alone doesn’t determine throughput. You also need to account for how long data takes to travel from sender to receiver and back (the round-trip time, or RTT). If you send one chunk of data and then stop to wait for an acknowledgement before sending the next chunk, most of the time is spent waiting, not transmitting. The fat pipe sits idle while you wait for the "OK, got it" signal to travel all the way back.

This is exactly the problem the Bandwidth-Delay Product (BDP) quantifies.

The Definition

The Bandwidth-Delay Product is the maximum amount of data that can be “in transit” (in the pipe) at any instant between a sender and a receiver:

BDP = Bandwidth × Round-Trip Time

Think of it like a water pipe. Bandwidth is how wide the pipe is (gallons per second). RTT is how long the pipe is (how many seconds the water takes to travel from one end to the other and back). The BDP is the volume of the pipe — the total amount of water that fits inside it at any moment.

To keep the pipe full (and thus achieve maximum throughput), the sender must have at least BDP bytes of data in flight — sent but not yet acknowledged. If the sender has less data in flight, the pipe has “bubbles” — idle capacity that’s being wasted.

A Concrete Example

Suppose you have a 1 Gbps link between New York and London with a round-trip time of 80 ms:

BDP = 1 Gbps × 80 ms
    = 1,000,000,000 bits/s × 0.080 s
    = 80,000,000 bits
    = 10,000,000 bytes = 10 MB

This means to fully utilize your 1 Gbps link, you need 10 MB of data in flight at all times. If your TCP window is capped at the default 64 KB (without window scaling), you’ll use only 64 KB / 10 MB = 0.6% of your link’s capacity. That 1 Gbps link behaves like a 6.4 Mbps link. Ouch.

Why This Concept Is Universal

BDP isn’t just for networks. Any system where a producer sends requests to a consumer over a channel with latency has a bandwidth-delay product. The concept applies to:

  • Network I/O — TCP connections across LANs, WANs, and the internet
  • Disk I/O — NVMe/SSD command queues; how many I/O requests must be in-flight to saturate the drive
  • Memory I/O — CPU memory bus; why CPUs use prefetching and out-of-order execution to hide DRAM latency
  • GPU pipelines — keeping the shader pipeline fed with enough work
  • Distributed systems — pipelining RPCs, async replication windows, message queue prefetch counts

In every case, the core insight is the same: to maximize throughput, you must keep enough requests in flight to cover the round-trip latency.

Animated Diagram: Filling the Pipe

Top: A sender with a small window (1 packet at a time). Most of the pipe sits empty — the sender idles waiting for each ACK before sending the next packet. Bottom: A sender with a window sized to the BDP. The pipe stays full — new packets enter as ACKs return. The throughput difference is dramatic.

The Math, Step by Step

The Core Formula

BDP = B × RTT

Where:
• B = bandwidth of the link (bits per second or bytes per second)
• RTT = round-trip time (seconds) — time for data to go from sender to receiver and for the acknowledgement to come back

Unit Conversions (The Tricky Part)

The most common mistake is mixing units. Here’s a reliable recipe:

  1. Convert bandwidth to bytes per second: divide bits/s by 8
  2. Convert RTT to seconds: divide milliseconds by 1000
  3. Multiply to get BDP in bytes

Worked Examples Across Different Scenarios

Scenario Bandwidth RTT BDP Meaning
Same datacenter 25 Gbps 0.1 ms 312 KB Small — easy to fill
Cross-country (US) 10 Gbps 60 ms 75 MB Large — needs big TCP windows
Trans-Atlantic 1 Gbps 80 ms 10 MB Default TCP window (64 KB) wastes 99.4%
Satellite link 100 Mbps 600 ms 7.5 MB Huge pipe despite modest bandwidth
Home WiFi (local) 100 Mbps 2 ms 25 KB Tiny — almost always full

Key insight from this table: BDP grows fastest when both bandwidth and latency are large. A high-bandwidth, high-latency path (called a Long Fat Network, or LFN, pronounced “elephant”) is the hardest to utilize. This is where BDP-awareness matters most.

The Utilization Formula

If the sender’s window (the amount of data it can have in flight) is W bytes, then:

Utilization = min(W, BDP) / BDP

Effective throughput = Utilization × Bandwidth

For the trans-Atlantic example: with a 64 KB TCP window and a 10 MB BDP, utilization = 64 KB / 10 MB = 0.006 = 0.6%. Effective throughput = 0.006 × 1 Gbps = 6.4 Mbps out of 1 Gbps. You’re paying for a highway but driving in one lane.

Network I/O: TCP Windows and Long Fat Networks

How TCP’s Window Relates to BDP

TCP uses a sliding window to control how much data is in flight. The sender can have up to min(cwnd, rwnd) bytes of unacknowledged data, where:

  • cwnd (congestion window) — set by the sender’s congestion control algorithm; starts small and grows
  • rwnd (receive window) — advertised by the receiver; how much buffer space it has

For maximum throughput, the effective window must be at least as large as the BDP. If the window is smaller than the BDP, the sender will finish transmitting its window’s worth of data before the first ACK arrives, and it must sit idle until an ACK comes back — leaving the pipe partially empty.

The Problem with Early TCP

Original TCP (RFC 793) used a 16-bit window size field, capping the window at 65,535 bytes (64 KB). This was fine in the 1980s when links were slow and local. But as bandwidth and distances grew, 64 KB became far too small. On a 1 Gbps trans-Atlantic link (BDP = 10 MB), 64 KB is only 0.6% of the pipe.

The Fix: TCP Window Scaling (RFC 7323)

TCP window scaling adds a scale factor negotiated during the three-way handshake. The 16-bit window field is left-shifted by up to 14 bits, allowing windows up to:

65,535 × 214 = 65,535 × 16,384 = 1,073,725,440 bytes ≈ 1 GB

This is large enough for essentially any link. Modern operating systems enable window scaling by default.

Step-by-Step: How Slow Start Ramps Up to the BDP

Even with large windows, TCP doesn’t immediately send BDP bytes. It starts small (typically 10 segments = ~14 KB) and doubles the window each RTT during slow start. Here’s a trace for our 1 Gbps / 80 ms link (BDP = 10 MB):

RTT # cwnd Data in flight Utilization Cumulative time
114 KB14 KB0.14%80 ms
228 KB28 KB0.27%160 ms
356 KB56 KB0.55%240 ms
4112 KB112 KB1.1%320 ms
...............
107.2 MB7.2 MB72%800 ms
1114.3 MB10 MB100%880 ms

It takes ~11 RTTs (880 ms) just to ramp up to full utilization. For short-lived HTTP requests, the connection may finish before slow start fills the pipe — which is why connection reuse (HTTP keep-alive, connection pooling) matters so much for high-BDP paths.

Long Fat Networks (LFNs): Special Challenges

Networks with large BDPs (> 100 KB) are called Long Fat Networks. They create specific problems:

  • Slow start penalty — as shown above, it takes many RTTs to ramp up. For a 10 MB BDP, ~11 RTTs = ~1 second wasted on ramp-up.
  • Loss recovery is slow — when a packet is lost, TCP may cut cwnd in half. Recovering from cwnd = 5 MB back to 10 MB in congestion avoidance (adding ~1 MSS per RTT) takes thousands of RTTs.
  • Sequence number wraparound — at 10 Gbps, TCP’s 32-bit sequence number space wraps in ~3.4 seconds. The PAWS (Protection Against Wrapped Sequences) extension in RFC 7323 uses timestamps to disambiguate.
  • Buffer requirements — routers and switches along the path need enough buffer to absorb BDP-sized bursts. Under-buffered switches cause loss, triggering congestion control and killing throughput.

Congestion Control Algorithms and BDP

Different TCP congestion control algorithms handle high-BDP paths very differently:

  • TCP Reno / Cubic — loss-based. They fill the pipe until they cause a packet drop, then cut the window. On LFNs, the sawtooth pattern means the window oscillates between ~0.5×BDP and ~1×BDP, averaging ~75% utilization.
  • BBR (Bottleneck Bandwidth and RTT) — Google’s model-based algorithm. Instead of probing for loss, BBR directly estimates the bottleneck bandwidth and the minimum RTT, then sets cwnd = bandwidth × RTT = BDP. It literally computes the BDP and targets it. This makes BBR excellent on LFNs.
  • QUIC (HTTP/3) — runs over UDP, implements its own congestion control (often BBR). Its 0-RTT connection establishment means it can start sending application data immediately, reducing the ramp-up penalty.

Tuning TCP for High-BDP Paths

On Linux, the key kernel parameters are:

# View current TCP buffer sizes (min, default, max) in bytes
cat /proc/sys/net/ipv4/tcp_rmem   # receive buffers
cat /proc/sys/net/ipv4/tcp_wmem   # send buffers

# Example: allow up to 16 MB per socket (for a 10 MB BDP)
# Format: min  default  max
sysctl -w net.ipv4.tcp_rmem="4096 131072 16777216"
sysctl -w net.ipv4.tcp_wmem="4096 131072 16777216"

# Enable window scaling (usually on by default)
sysctl -w net.ipv4.tcp_window_scaling=1

# Enable BBR congestion control (great for high-BDP)
sysctl -w net.ipv4.tcp_congestion_control=bbr

The max value of tcp_rmem and tcp_wmem must be ≥ BDP. Linux auto-tunes within these bounds, but if the max is too low, it caps the window below the BDP.

Disk I/O: Queue Depth as a Bandwidth-Delay Product

The Same Principle, Different Domain

When a CPU sends a read or write command to a storage device (SSD, NVMe drive, HDD), it doesn’t get an instant response. There’s latency — the device needs time to look up the data, read it from flash cells (or spin the platter), and send it back. This latency is the “delay” in the BDP formula.

The “bandwidth” is the device’s maximum throughput (e.g., 3.5 GB/s for a fast NVMe SSD). If you send one I/O request at a time and wait for it to complete before sending the next, you get:

Throughput (single request) = Request size / Latency

Example: 4 KB read with 100 µs latency = 4 KB / 0.0001 s = 40 MB/s

But the device can do 3,500 MB/s. You’re using 1.1%.

Queue Depth: The Disk I/O “Window”

The fix is the same as for TCP: send multiple requests without waiting. Queue depth is how many I/O requests are in flight simultaneously. It’s the disk analog of TCP’s congestion window.

Required queue depth = Target throughput × Latency / Request size

Or equivalently: QD = BDP / Request size

Where BDP = Device bandwidth × Device latency

Worked Example: NVMe SSD

A modern NVMe SSD has:

  • Sequential read bandwidth: 3,500 MB/s
  • Average read latency: 100 µs (for 4 KB random reads)
  • Typical I/O size: 4 KB
BDP = 3,500 MB/s × 100 µs = 3,500 × 106 B/s × 100 × 10-6 s = 350 KB

Required QD = 350 KB / 4 KB = ~88 concurrent requests

With a queue depth of 1, you get ~40 MB/s. With a queue depth of 88, you can approach 3,500 MB/s. This is why NVMe supports up to 65,535 queue entries per queue and up to 65,535 queues — it’s designed to keep a massive number of requests in flight.

Comparison Across Storage Technologies

Device Bandwidth Latency BDP QD needed (4 KB)
HDD (7200 RPM) 200 MB/s 8 ms 1.6 MB 400
SATA SSD 550 MB/s 100 µs 55 KB 14
NVMe SSD 3,500 MB/s 100 µs 350 KB 88
NVMe Gen5 SSD 12,000 MB/s 80 µs 960 KB 240

Notice the HDD: despite being the slowest device, its high latency (8 ms for seeks) means it actually has the largest BDP. But HDDs can’t truly parallelize random I/O (one physical head), so high queue depths only help with sequential or NCQ-reordered access. SSDs, by contrast, have massive internal parallelism (multiple flash dies) and genuinely benefit from deep queues.

Why io_uring and AIO Exist

Traditional read()/write() system calls are synchronous — they block until the I/O completes. That’s queue depth = 1 per thread. To achieve higher queue depths, you need either:

  • Many threads — each thread blocks on its own I/O. Works but has context-switching overhead (see OS Concepts).
  • Linux AIO (libaio) — submit multiple I/O requests asynchronously, poll for completions. Limited to direct I/O (bypasses page cache).
  • io_uring — Linux’s modern async I/O framework. Uses shared ring buffers between user space and kernel to submit and complete I/O with minimal system calls. Supports all I/O types. Explicitly designed to let applications maintain high queue depths efficiently.

Code Example: Measuring BDP Impact on Disk I/O

"""
Demonstrates how queue depth affects throughput.
Models the disk I/O BDP concept with a simulated storage device.

The simulation shows: with QD=1, throughput is latency-limited.
As QD increases toward BDP/request_size, throughput approaches
the device's maximum bandwidth.
"""

def simulate_disk_throughput(
    device_bandwidth_MBs: float,   # max device bandwidth in MB/s
    device_latency_us: float,      # per-request latency in microseconds
    request_size_KB: float,        # size of each I/O request in KB
    queue_depth: int               # number of concurrent requests
) -> dict:
    """
    Simulate effective throughput at a given queue depth.

    The key insight: with QD concurrent requests, the device
    can start a new request every (latency / QD) microseconds,
    up to its bandwidth ceiling.
    """
    latency_s = device_latency_us / 1_000_000
    request_size_MB = request_size_KB / 1024

    # BDP = bandwidth * latency
    bdp_MB = device_bandwidth_MBs * latency_s
    bdp_KB = bdp_MB * 1024
    optimal_qd = bdp_KB / request_size_KB

    # With QD requests in flight, effective throughput is:
    #   min(QD * request_size / latency, device_bandwidth)
    throughput_MBs = min(
        queue_depth * request_size_MB / latency_s,
        device_bandwidth_MBs
    )
    utilization = throughput_MBs / device_bandwidth_MBs

    return {
        "queue_depth": queue_depth,
        "throughput_MBs": round(throughput_MBs, 1),
        "utilization_pct": round(utilization * 100, 1),
        "bdp_KB": round(bdp_KB, 1),
        "optimal_qd": round(optimal_qd)
    }

# NVMe SSD: 3500 MB/s, 100µs latency, 4 KB requests
print("NVMe SSD (3500 MB/s, 100µs latency, 4KB requests)")
print(f"{'QD':>4} {'Throughput':>12} {'Utilization':>12}")
print("-" * 32)
for qd in [1, 2, 4, 8, 16, 32, 64, 128]:
    r = simulate_disk_throughput(3500, 100, 4, qd)
    print(f"{qd:>4} {r['throughput_MBs']:>9.1f} MB/s {r['utilization_pct']:>9.1f}%")
print(f"\nBDP = {r['bdp_KB']} KB, Optimal QD = {r['optimal_qd']}")

# Output:
# NVMe SSD (3500 MB/s, 100µs latency, 4KB requests)
#   QD   Throughput  Utilization
# --------------------------------
#    1       40.0 MB/s        1.1%
#    2       80.0 MB/s        2.3%
#    4      160.0 MB/s        4.6%
#    8      320.0 MB/s        9.1%
#   16      640.0 MB/s       18.3%
#   32     1280.0 MB/s       36.6%
#   64     2560.0 MB/s       73.1%
#  128     3500.0 MB/s      100.0%

Memory and Cache I/O: Hiding DRAM Latency

Why Memory Has a BDP Problem Too

Modern CPUs can execute instructions in under 1 nanosecond (a 4 GHz core has a cycle time of 0.25 ns). But reading from main memory (DRAM) takes ~50-100 ns — that’s 200-400 CPU cycles of waiting. During that time, a CPU doing nothing wastes enormous potential throughput.

The memory bus has high bandwidth. A DDR5 dual-channel system might deliver 50 GB/s. With a 100 ns DRAM latency, the BDP is:

BDP = 50 GB/s × 100 ns = 50 × 109 B/s × 100 × 10-9 s = 5,000 bytes

A single cache-line fetch is 64 bytes. To keep the memory bus fully utilized, you need 5,000 / 64 = ~78 cache-line requests in flight simultaneously. The CPU must have 78 outstanding memory requests at any given moment.

How CPUs Fill Their Memory BDP

CPUs use several mechanisms to keep enough memory requests in flight, all of which are hardware-level solutions to the BDP problem:

1. Hardware Prefetching

The CPU detects sequential or strided memory access patterns and speculatively fetches future cache lines before the program asks for them. When you iterate through an array, the prefetcher recognizes the pattern after a few accesses and starts loading cache lines ahead of your loop. This turns what would be a sequence of serial 100 ns waits into overlapped requests.

Without prefetching (serial):
Load A (100 ns) → Use A → Load B (100 ns) → Use B → Load C (100 ns) → ...
Time for 3 loads: ~300 ns

With prefetching (pipelined):
Load A, start prefetching B and C → Use A (B arrives) → Use B (C arrives) → Use C
Time for 3 loads: ~100 ns (only the first load pays full latency)

2. Out-of-Order Execution

When the CPU hits a cache miss, it doesn’t just stall. It looks ahead in the instruction stream and executes other independent instructions while waiting for memory. If the program has two independent loads (say, accessing two different arrays), the CPU issues both memory requests in parallel, cutting the effective latency in half.

Modern CPUs can track dozens of outstanding memory requests simultaneously. Intel’s cores have Line Fill Buffers (LFBs) that can track 12-16 outstanding cache-line requests. This is the CPU’s “window” — its hardware queue depth for memory operations.

3. Memory-Level Parallelism (MLP)

The total number of simultaneous memory requests a CPU can sustain is called its Memory-Level Parallelism. It’s the memory analog of TCP’s window size. Typical values:

  • Simple in-order CPU (e.g., ARM Cortex-A53): MLP ≈ 2-4
  • Modern out-of-order CPU (e.g., Intel, AMD Zen 4): MLP ≈ 10-20
  • GPU: MLP ≈ thousands (hundreds of threads, each with outstanding loads)

GPUs solve the memory BDP problem by brute force: they have so many threads that even though each thread stalls on memory, the GPU switches to another thread instantly. Collectively, thousands of threads keep enough memory requests in flight to saturate the GPU’s memory bus (which has an even larger BDP — e.g., 1 TB/s bandwidth × 200 ns latency = 200 KB BDP, requiring ~3,000+ outstanding cache-line requests).

Cache Hierarchy as a BDP Reducer

The CPU cache hierarchy (L1/L2/L3) exists precisely to reduce the “delay” component of the memory BDP:

Level Latency BDP (at 50 GB/s) Cache lines needed
L1 cache ~1 ns 50 bytes 1
L2 cache ~5 ns 250 bytes 4
L3 cache ~20 ns 1,000 bytes 16
Main memory (DRAM) ~100 ns 5,000 bytes 78

When your data fits in L1, the BDP is trivially small (1 cache line) and the CPU never stalls. As your working set grows beyond each cache level, the effective latency jumps and the BDP grows, requiring more parallelism to keep throughput high. This is why cache-friendly data structures (arrays, contiguous memory) dramatically outperform pointer-chasing structures (linked lists, trees with scattered nodes) — prefetchers can only help with predictable patterns, and the cache hierarchy only helps when locality is good.

The System Design Connection

In system design, the memory BDP shows up as:

  • In-memory cache design — Redis, Memcached. Batch operations (MGET instead of many GETs) reduce round trips to the cache server, which is the same principle as filling the BDP pipe.
  • Database buffer pool — PostgreSQL, MySQL. The buffer pool keeps frequently-accessed pages in memory (reducing “delay”). The prefetch strategy (effective_io_concurrency in PostgreSQL) controls how many I/O requests the DB issues in parallel — it’s setting the queue depth to fill the disk’s BDP.
  • Application-level batching — instead of making 100 sequential RPC calls to a service (each with 2 ms latency = 200 ms total), batch them into one call or pipeline them. You’re filling the network BDP.

BDP Thinking in Distributed System Design

Once you internalize BDP, you start seeing it everywhere in system design. The pattern is always the same: if you do things one-at-a-time and wait for each response, latency dominates throughput. The fix is always some form of pipelining, batching, or concurrency.

Pattern: Request Pipelining

Redis supports pipelining — sending multiple commands without waiting for each response. Without pipelining, each command pays the full network RTT. With pipelining, you fill the BDP between your app and Redis. 

import redis

r = redis.Redis(host='redis-server', port=6379)

# BAD: Sequential — each GET pays full RTT (say 0.5ms)
# 1000 GETs × 0.5ms = 500ms total
for key in keys:          # 1000 keys
    value = r.get(key)    # blocks for RTT each time

# GOOD: Pipelined — all 1000 GETs in one network round trip
# Total time ≈ 1 RTT + server processing ≈ 1-2ms
pipe = r.pipeline()
for key in keys:
    pipe.get(key)         # queued locally, not sent yet
values = pipe.execute()   # sends all at once, one RTT

The speedup factor is approximately N * RTT / (RTT + N * processing_time), which for small operations approaches N (in this case, ~250-500x faster).

Pattern: Async Replication Windows

In database replication (see Database Replication), asynchronous replication sends writes to replicas without waiting for acknowledgement. The amount of unacknowledged data is limited by the replication window — essentially the BDP between the primary and its replicas. A larger window means higher replication throughput but more data at risk if the primary crashes.

Pattern: Message Queue Prefetch

In RabbitMQ, the prefetch_count setting controls how many messages a consumer can have in flight (received but not yet acknowledged). This is a BDP setting: you want enough messages in flight to cover the processing time plus the network RTT for the ACK, so the broker’s delivery bandwidth is fully utilized.

If each message takes 10 ms to process and the network RTT is 2 ms:

• prefetch_count = 1: consumer processes 1 message per 12 ms = 83 msg/s
• prefetch_count = 10: consumer has 10 messages buffered, processes back-to-back = 100 msg/s (limited by processing speed, not latency)

The “BDP” here = delivery_rate × (processing_time + RTT) / 1 message ≈ 1.2 messages. Setting prefetch to 2+ eliminates the idle time.

Pattern: gRPC Streaming vs Unary Calls

If you need to send 10,000 items to a service, you have two choices:

  • 10,000 unary RPCs: each pays network RTT. Total time ≈ 10,000 × RTT.
  • 1 bidirectional stream: items flow continuously through the pipe. Total time ≈ RTT + (10,000 × per-item processing time). The stream fills the BDP automatically.

This is why gRPC streaming and HTTP/2 multiplexing exist — they’re both BDP-filling mechanisms.

Anti-Pattern: The Chatty Microservice

A common system design mistake is a microservice that makes many sequential calls to other services to fulfill a single request. If Service A calls Service B 5 times in sequence (each with 5 ms RTT), it adds 25 ms of pure waiting to the request. This is a BDP failure — the network pipe between A and B sits empty most of the time.

Fixes:

  • Batch API — Service B exposes a batch endpoint that handles all 5 operations in one call
  • Parallel fan-out — if the 5 calls are independent, make them concurrently (BDP-filling!)
  • Data locality — move the data closer (caching, denormalization) to reduce the “delay” component

Assumptions & When BDP Thinking Breaks Down

The BDP model assumes:
  • The bottleneck is the pipe, not the endpoints — if the sender can’t generate data fast enough or the receiver can’t process it fast enough, filling the pipe doesn’t help. BDP only matters when the channel bandwidth exceeds end-to-end throughput.
  • Latency is roughly constant — if latency varies wildly (e.g., mobile networks), the optimal window size changes constantly. Algorithms like BBR periodically re-probe to adapt.
  • The path has a single bottleneck — real networks have multiple hops. The BDP should be computed using the bottleneck link’s bandwidth, not the sender’s or receiver’s link speed.
  • Shared congestion isn’t a factor — if many flows share a link, each flow gets a fraction of the bandwidth. The effective BDP per flow is smaller. Overshoot (sending more than your share) causes congestion and packet loss.
  • Processing time is negligible — for disk and memory BDP, the “delay” is device latency. But if the requester spends significant time processing each response before issuing the next request, the effective RTT is longer than just the device latency.

Key Takeaways

  1. BDP = Bandwidth × RTT tells you how much data must be “in the pipe” to achieve full throughput. It applies to networks, disks, memory, and any producer-consumer system with latency.
  2. If your window < BDP, you waste capacity. This is the #1 reason fast links don’t deliver fast transfers. Utilization = min(window, BDP) / BDP.
  3. For network I/O: ensure TCP buffers ≥ BDP, use window scaling, prefer BBR on LFNs, and reuse connections to amortize slow start.
  4. For disk I/O: the queue depth must be high enough to fill the device’s BDP. Use async I/O (io_uring) instead of synchronous read/write to maintain high queue depths without thread-per-request overhead.
  5. For memory I/O: hardware prefetching, out-of-order execution, and cache hierarchies all exist to fill the memory BDP. Use cache-friendly data layouts to help the hardware help you.
  6. In system design: pipeline requests, batch operations, use streaming RPCs, and fan out in parallel. Every sequential wait on a remote resource is an unfilled BDP.
  7. The universal fix is always some form of concurrency: TCP windows, disk queue depth, CPU out-of-order execution, message prefetch, request pipelining — they’re all the same idea wearing different costumes.
