Radix Tree (Patricia Trie)

A trie whose single-child chains are collapsed into one edge, so the tree’s height tracks how much keys differ, not how long they are. Its native operation — longest-prefix match in time linear in the query, independent of how many keys are stored — is exactly what IP routers, autocomplete, and LLM prefix caches need.

~12 min read • data-structure reference

What Is It?

Start with a trie (prefix tree): a tree where each edge is labelled with one character and the path from the root to a node spells a string. All keys sharing a prefix share that prefix’s path, so “does any stored key share a prefix with X?” is answered by walking down from the root — no scan of the key set.

The trie’s flaw: a key of length k with no branching still costs k single-child nodes. The word internationalization is a 20-node chain even if it is the only key. Most nodes carry no decision — they exist only because the edge alphabet is one character.

A radix tree (a.k.a. Patricia trie, radix trie, compressed trie) applies one rule: any node with exactly one child is merged into its parent, and the two edge labels are concatenated. Edges are now labelled with strings (runs), not single characters. A node exists only where keys actually branch or a key terminates. The result stores the same set, answers the same prefix queries, and has height bounded by the number of distinguishing positions among the keys — not by key length.

The Longest-Prefix-Match Walk

The defining query: given a string s, find the longest stored key that is a prefix of s. Walk from the root, consuming whole edge-labels; remember the deepest is_key node reached; stop when no edge matches the next characters. Below, the query cares matches the stored key care (4 chars) and then halts because no edge leaves care for s.

  • ■ On the query path — nodes/edges the walk for cares traverses (root → ca → car → care).
  • ■ Stored-key node (is_key) — a path that spells an inserted key (cat, card, dog here).
  • ■ Internal node — exists only because keys branch here (e.g. ·ca).
  • → Edge label — the run of characters consumed (a trie would split each into single-char edges).

Operations & Complexity

Let k = query length, N = number of stored keys, Σ = alphabet size. The headline property: cost depends on the key, not on N.

OperationCostNote
Lookup / insert / deleteO(k)One downward walk; independent of N. A hash map is also O(k) for the hash but loses the next two rows.
Longest-prefix matchO(k)The native query (IP routing, prefix cache). A hash map cannot do this without trying all k prefixes.
Ordered / range / autocompleteO(k + m)Keys are stored in sorted order by construction; a subtree is a prefix class of m results.
Insert with edge splitO(k)If a new key diverges mid-edge, split that edge into two (the only structurally interesting case — see code).
SpaceO(total distinguishing chars)≤ a trie always; nodes only at branches/terminals, not per character.

Versus alternatives. A hash map wins for pure exact-key lookup with no prefix/order needs (O(1) expected, simpler) but cannot answer longest-prefix or range queries. A balanced BST gives order but at O(k log N) per comparison-heavy operation. A plain trie is simpler but pays O(key length) nodes; the radix tree is the trie with that waste removed.

Where It’s Used

  • IP routing tables. A router picks the route with the longest matching address prefix — literal longest-prefix match. The Linux kernel’s routing trie (LPC-trie / LC-trie) is a radix-tree variant.
  • LLM prefix caching (RadixAttention). Cached KV blocks are keyed by a chained block hash; finding how much of an incoming prompt is already cached is a longest-prefix walk over a radix tree. This is the data structure behind K2 — KV Cache (Role 2: inter-request reuse & cache-aware routing) (SGLang’s RadixAttention); subtree-LRU eviction maps cleanly onto radix subtrees.
  • Autocomplete & dictionaries. The subtree under a typed prefix is exactly the candidate set, in sorted order.
  • Databases & in-memory indexes. The Adaptive Radix Tree (ART) is a production main-memory index (used in DuckDB, HyPer-style engines) — a radix tree with node sizes that adapt to fan-out.
  • Filesystem / config / key-value paths. Hierarchical, slash-delimited keys share long prefixes; etcd and many KV stores keep range/prefix scans cheap this way.

Variants & Names

  • Trie / prefix tree — the uncompressed parent; one character per edge.
  • Radix tree / radix trie / compressed trie / Patricia trie — single-child chains collapsed; the subject of this page. (“PATRICIA” historically meant the radix-2/binary form for bitstrings; today “Patricia trie” is used loosely for any compressed trie.)
  • Adaptive Radix Tree (ART) — a radix tree whose internal nodes switch representation (4/16/48/256 children) by fan-out to stay cache-efficient; the common database form.
  • Ternary search tree — a hybrid (BST of characters) trading the per-node child array for three pointers; relevant when Σ is large and tries waste space.

Code — Insert & Longest-Prefix Match

The only non-obvious case is edge splitting: a new key shares part of an edge label, so the edge is broken at the divergence point and a new branch node is inserted.

class RadixNode:
    def __init__(self):
        self.edges = {}      # first_char -> (label: str, child: RadixNode)
        self.is_key = False  # True if a stored key ends at this node


class RadixTree:
    def __init__(self):
        self.root = RadixNode()

    def insert(self, key: str) -> None:
        node, i = self.root, 0
        while i < len(key):
            c = key[i]
            if c not in node.edges:                 # no edge: attach the rest
                leaf = RadixNode(); leaf.is_key = True
                node.edges[c] = (key[i:], leaf)
                return
            label, child = node.edges[c]
            j = 0                                   # longest common run
            while (j < len(label) and i + j < len(key)
                   and label[j] == key[i + j]):
                j += 1
            if j == len(label):                     # whole edge consumed
                node, i = child, i + j
                continue
            mid = RadixNode()                       # split edge at j
            node.edges[c] = (label[:j], mid)
            mid.edges[label[j]] = (label[j:], child)
            if i + j == len(key):
                mid.is_key = True                   # new key ends at split
            else:
                leaf = RadixNode(); leaf.is_key = True
                mid.edges[key[i + j]] = (key[i + j:], leaf)
            return
        node.is_key = True                          # key was an existing path

    def longest_prefix(self, s: str) -> str:
        node, i, best = self.root, 0, 0
        while i < len(s):
            c = s[i]
            if c not in node.edges:
                break
            label, child = node.edges[c]
            if s[i:i + len(label)] != label:        # partial edge: stop
                break
            i += len(label); node = child
            if node.is_key:
                best = i                            # deepest stored key so far
        return s[:best]


t = RadixTree()
for w in ["car", "card", "care", "cat", "dog"]:
    t.insert(w)
print(t.longest_prefix("cares"))   # -> "care"
print(t.longest_prefix("cab"))     # -> ""      (no stored key is a prefix)
print(t.longest_prefix("cards!"))  # -> "card"

Key Takeaways

  • A radix tree is a trie with single-child chains collapsed — edges carry runs, nodes sit only at branches and key-ends.
  • Lookup, insert, delete, and longest-prefix match are all O(query length), independent of how many keys are stored.
  • Longest-prefix match is the operation a hash map cannot do — it is why routers, autocomplete, and LLM prefix caches use this structure rather than a map.
  • Edge-splitting on divergent insert is the one structurally interesting operation; everything else is a downward walk.
  • In LLM serving it underlies RadixAttention: walk the prompt’s block-hash chain, the deepest existing node is the longest already-cached prefix — see K2 (Role 2).
