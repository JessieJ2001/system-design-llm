Consistent Hashing

A hash-based distribution technique that minimizes key remapping when the set of nodes changes, enabling elastic scaling of distributed caches, databases, and load balancers.

~35 min read

What Is It?

Consistent hashing is a technique for distributing keys (data, requests, or cache entries) across a changing set of nodes in a way that minimizes disruption when nodes are added or removed. Unlike naive approaches, only a small fraction of keys need to move when the cluster topology changes.

Consider the simplest approach to distributing keys: take the hash of a key and compute hash(key) % N, where N is the number of servers. This works well when N is fixed, but the moment you add or remove a single server, nearly every key maps to a different server. For a cache cluster, this means an almost complete cache miss storm — every key is suddenly on the wrong node. With 10 servers, adding one server remaps roughly 90% of keys.

Consistent hashing solves this by mapping both nodes and keys onto the same circular hash space — a "ring" of values from 0 to 232−1 (or any large range). Each key walks clockwise around the ring until it finds its nearest node. When a node joins or leaves, only the keys in the affected arc of the ring need to move — typically just K/N keys (where K is total keys and N is the number of nodes), rather than reshuffling everything.

The Hash Ring From Scratch: A Concrete Example

Let's build a consistent hash ring step by step with real numbers. The hash space is the range [0, 232) — that is, integers from 0 to 4,294,967,295. Imagine bending this number line into a circle so that position 0 and position 4,294,967,295 are adjacent. This circle is the hash ring.

Step 1: Place Servers on the Ring

We hash each server's identifier (e.g., its IP address or hostname) using a hash function like MurmurHash3. The result is a position on the ring:

  • Server A: hash("server-A") = 500,000,000 (position 500M on the ring)
  • Server B: hash("server-B") = 1,800,000,000 (position 1.8B)
  • Server C: hash("server-C") = 3,200,000,000 (position 3.2B)

These three servers divide the ring into three arcs:

  • Arc 1: 3.2B → 500M (wrapping through 0) — owned by Server A
  • Arc 2: 500M → 1.8B — owned by Server B
  • Arc 3: 1.8B → 3.2B — owned by Server C

Step 2: Map Keys to Servers

The rule is simple: for any key, walk clockwise on the ring until you hit a server. That server owns the key. The ring wraps around, so after position 4,294,967,295 comes position 0.

Let's hash 6 keys and see where they land:

The clockwise search in code is a binary search through a sorted list of server positions. Given a key's hash value, find the smallest server position that is greater than or equal to the key's hash. If no such position exists (the key's hash is larger than all server positions), wrap around to the first server in the sorted list.

Step 3: Why This Works When Servers Change

Notice that each key's assignment depends only on the nearest clockwise server — not on any other servers. If Server B goes down, only the keys in B's arc (700M "user:2" and 1.5B "user:3") need to move to C (the next clockwise server after B's arc). The keys assigned to A and C are completely unaffected. Compare this to hash(key) % 3 changing to hash(key) % 2 — that would remap nearly every key.

How It Works

  1. Choose a hash function that maps both node identifiers and keys into the same large integer space (e.g., 0 to 232−1). Common choices include MurmurHash3, xxHash, or SHA-1 truncated.
  2. Form the ring. Conceptually arrange the hash space as a circle. Each node is hashed and placed at its corresponding position on the ring.
  3. Assign keys. Hash each key and walk clockwise around the ring. The first node encountered is the owner of that key.
  4. Node addition. When a new node joins, it takes responsibility for the keys between itself and its counter-clockwise predecessor. Only those keys migrate — all other assignments remain unchanged.
  5. Node removal. When a node leaves, its keys transfer to its clockwise successor. Again, only the departing node's keys move.
  6. Virtual nodes for balance. In practice, a small number of physical nodes on a large ring creates uneven partitions. The solution is to hash each physical node multiple times (e.g., "NodeA-0", "NodeA-1", ..., "NodeA-149"), creating 100–200 virtual nodes (vnodes) per physical node. This spreads each node's responsibility across many small arcs, producing a much more uniform distribution.

Only 1 out of 6 keys moved. The only keys that move are those in the arc between B (1.8B) and D (2.5B) — because D is now the nearest clockwise server for keys in that arc instead of C. All other keys are completely unaffected.

Proof That Only K/(N+1) Keys Move on Average

With N servers on the ring, the ring is divided into N arcs. When a new server D is added, it lands somewhere on the ring — splitting exactly one existing arc into two. Only the keys in the portion of that arc that is now closer to D than to the old owner need to move.

If keys are uniformly distributed, each arc contains approximately K/N keys (where K is the total number of keys). The new server D splits one arc, and the keys that move are those between D's counter-clockwise predecessor and D itself — approximately K/(N+1) keys on average.

Concrete numbers: With 1,000,000 keys and 100 servers, adding one server moves approximately 1,000,000 / 101 ≈ 9,901 keys (about 1%). Compare this to modulo hashing (hash(key) % N), where changing N from 100 to 101 would remap approximately 990,000 keys (99%). Consistent hashing reduces disruption by two orders of magnitude.

Removing a Node: The Same Logic in Reverse

If Server B (1.8B) is removed, its keys need a new home. The rule: all of B's keys move to B's clockwise successor. In our example, B's successor is C (3.2B). So "user:2" (700M) and "user:3" (1.5B), which were assigned to B, now map to C. No other keys are affected.

Again, only K/N keys move — roughly 1/N of the total. The other (N-1)/N keys stay exactly where they are.

Virtual Nodes: Solving the Load Imbalance Problem

The Problem Without Virtual Nodes

With our 3 servers on the ring, look at the arc sizes:

  • Server A owns the arc from 3.2B to 500M (wrapping through 0). Arc size: (4,294,967,295 − 3,200,000,000) + 500,000,000 = 1,594,967,295. That is 37.1% of the ring.
  • Server B owns the arc from 500M to 1.8B. Arc size: 1,300,000,000. That is 30.3% of the ring.
  • Server C owns the arc from 1.8B to 3.2B. Arc size: 1,400,000,000. That is 32.6% of the ring.

This looks reasonably balanced — but only by luck. With different hash values, one server could easily own 50% of the ring while another owns only 15%. With just 3 points on a circle, the variance is inherently high. In the worst case, Server B gets 2.5× the load of Server C.

The Solution: Multiple Points Per Server

Instead of placing each server at one position on the ring, we create multiple virtual nodes (vnodes) for each physical server. For example, with 3 vnodes per server:

  • Server A places vnodes at: hash("A-0"), hash("A-1"), hash("A-2")
  • Server B places vnodes at: hash("B-0"), hash("B-1"), hash("B-2")
  • Server C places vnodes at: hash("C-0"), hash("C-1"), hash("C-2")

Now there are 9 points on the ring instead of 3. The ring is divided into 9 smaller arcs. Each server owns 3 arcs scattered around the ring, and the total ownership for each server averages out to approximately 33% (±5% with just 3 vnodes each).

In practice, you use 100–200 vnodes per physical server. With 150 vnodes and 10 servers, the ring has 1,500 points — and the law of large numbers ensures each server's total arc coverage is very close to 1/10 = 10% of the ring.

The Memory Tradeoff

Each virtual node requires storing a (position, server) entry in a sorted data structure. The memory cost is straightforward:

  • With 200 vnodes per server and 1,000 servers: 200,000 entries × 32 bytes per entry = 6.4 MB. This is negligible — it fits in L2 cache on most machines.
  • With 150 vnodes per server and 50 servers: 7,500 entries × 32 bytes = 240 KB. Easily fits in L1/L2 cache.

The lookup cost goes from O(log N) to O(log(N × V)), where V is the vnode count. With 200 vnodes and 1,000 servers, that is log2(200,000) ≈ 18 comparisons — still sub-microsecond. The balance improvement far outweighs this tiny extra lookup cost.

Secondary Benefit: Better Failure Handling

Without vnodes, when a server dies, all its keys go to one successor — potentially doubling that successor's load and risking a cascading failure. With vnodes, a server's keys are spread across many small arcs, so when it dies, the keys are absorbed by many different successors. With 150 vnodes, the departing server's load spreads across up to 150 different successors, adding only a tiny increment to each.

When to Use / When NOT to Use

Use When

  • Distributed caches where nodes are added/removed frequently (e.g., Memcached clusters, Redis rings) and cache miss storms from full redistribution would overload the database.
  • Database sharding in systems that need elastic scaling — adding or removing shards should not require a full data migration.
  • Load distribution for stateful routing — e.g., sticky sessions to a set of application servers that may scale up or down.
  • CDN edge routing where you need to direct requests for the same URL to the same edge server for cache efficiency.
  • Any system where the node count is dynamic and you want to minimize the "blast radius" of topology changes.

Don't Use When

  • Static environments with a fixed, known number of servers that never change. Simple modulo hashing is easier to implement and reason about.
  • Very small clusters (2–3 nodes) where the overhead of virtual nodes and a ring data structure is not justified. Modulo or explicit mapping works fine.
  • Ordered range queries are needed — consistent hashing scatters keys. Use range-based partitioning instead (e.g., Bigtable-style tablets).
  • When you need perfect balance and nodes never change — Jump Hash gives perfect 1/N balance with zero memory overhead, but does not support arbitrary removal well.

Choose Consistent Hashing Over Modulo When

The deciding factor is whether your node count changes. If nodes are added or removed during normal operations (auto-scaling, rolling deployments, node failures), consistent hashing is worth the added complexity. If the cluster is truly static, modulo hashing is simpler and equally effective.

Classic vs. Modern Approaches

Original Consistent Hashing (Karger et al., 1997)

Introduced in the paper "Consistent Hashing and Random Trees" at MIT. Maps nodes and keys onto a ring. With N nodes, adding or removing one node moves only O(K/N) keys. Virtual nodes were added later to fix the load balance problem. This is the version used by Amazon Dynamo, Apache Cassandra, and Akamai's CDN.

Jump Hash (Lamping & Veach, Google, 2014)

A remarkably simple algorithm: given a key and the number of buckets, it returns a bucket number in O(ln N) time with zero memory overhead (no ring structure needed). It achieves perfect balance — every bucket gets exactly 1/N of the keys. The catch: it only works when buckets are numbered 0..N-1 and you can only add or remove the last bucket. You cannot remove an arbitrary node. This makes it ideal for scenarios where nodes are homogeneous and failures are handled at a different layer (e.g., an intermediary remaps failed buckets).

How Jump Hash Works: The "Jumping" Procedure

The core idea is elegant: start at bucket 0, then "jump" forward to a higher-numbered bucket. The jump distance is random but deterministic (seeded by the key). You keep jumping until you would land past the last bucket, then return the last valid bucket you were on.

The key insight: when you add bucket N (going from N to N+1 buckets), each existing key has a 1/(N+1) probability of jumping to the new bucket. This means exactly 1/(N+1) of keys move — the theoretically optimal amount.

def jump_hash(key: int, num_buckets: int) -> int:
    """Jump consistent hash: O(ln n) time, O(1) memory."""
    b, j = -1, 0
    while j < num_buckets:
        b = j
        # Deterministic pseudo-random jump based on key
        key = ((key * 2862933555777941757) + 1) & 0xFFFFFFFFFFFFFFFF
        j = int((b + 1) * (1 << 31) / ((key >> 33) + 1))
    return b

Why O(ln n) Time

Each iteration roughly doubles the jump distance (the expected next jump is about 2× the current position). This means the total number of iterations is approximately ln(n). For 1,000 buckets, that is about 7 iterations. For 1,000,000 buckets, about 20 iterations. No memory is used — no ring, no sorted array, no lookup table.

The Critical Limitation

Buckets must be numbered 0 through n-1. You can only add bucket n (at the end) or remove bucket n-1 (from the end). You cannot remove an arbitrary server in the middle. If server 3 out of 10 crashes, you cannot simply remove it — you would need to remap server 3's keys to another server at a layer above the hash function. This makes jump hash unsuitable for systems where specific servers fail independently. It is designed for systems where you scale by adding or removing capacity from the end, like adding more shards to a sequentially numbered shard set.

Maglev Hashing (Eisenbud et al., Google, 2016)

Designed for Google's Maglev network load balancer. Each backend generates a permutation table, and a lookup table is built so that every slot maps to a backend. It provides near-perfect uniformity and minimal disruption: when one of M backends goes down, only 1/M of traffic shifts, and the disruption is spread evenly. The lookup table is O(M) in size and lookups are O(1). Maglev hashing supports arbitrary node removal, unlike Jump Hash.

How Maglev Hashing Works: Step by Step

Maglev builds a fixed-size lookup table where each slot maps to a backend server. The table size M must be a prime number (e.g., M = 7 for this example; real systems use M = 65,537). The algorithm has three steps.

Step 1 — Generate preference permutations. For each backend, generate a permutation of [0, M-1] using two hash functions. This permutation represents the backend's "preference list" — the order in which it would like to claim slots in the lookup table.

def generate_permutation(backend_name, M):
    """Generate a permutation of [0, M-1] for this backend."""
    offset = h1(backend_name) % M
    skip   = h2(backend_name) % (M - 1) + 1  # skip is in [1, M-1]
    permutation = []
    for j in range(M):
        permutation.append((offset + j * skip) % M)
    return permutation

With M = 7 and 3 backends (B0, B1, B2):

Each row is a valid permutation of [0, 1, 2, 3, 4, 5, 6] — every slot appears exactly once. The offset determines where the backend starts, and skip determines the stride. Because M is prime and skip is in [1, M-1], the sequence (offset + j * skip) % M always produces a full permutation (it visits every slot before repeating).

Step 2 — Fill the lookup table using round-robin. Cycle through the backends in order. Each backend claims its next preferred slot (from its permutation) that is not already taken. If its next preferred slot is taken, it skips to the next one in its permutation. Continue until all M slots are filled.

B0 has 3 slots, B1 has 2 slots, B2 has 2 slots. With M = 7 and 3 backends, perfect balance would be 7/3 ≈ 2.33 slots each. The actual distribution (3, 2, 2) is as close to perfect as integer division allows.

Step 3 — Lookup. To find the backend for a key, compute hash(key) % M to get a table index, then read the backend from that slot. This is an O(1) operation — a single hash, a modulo, and an array access.

def maglev_lookup(key, lookup_table, M):
    """O(1) lookup: hash the key, index into the table."""
    index = hash(key) % M
    return lookup_table[index]

# Example: hash("user:4") % 7 = 3 → lookup_table[3] = B0 → route to B0

Why Maglev Gives Better Balance Than Ring-Based Hashing

The lookup table has exactly M entries, and the round-robin filling process ensures each of N backends gets approximately M/N entries. The distribution is nearly perfect because every backend gets an equal number of turns to claim slots, and the permutation-based ordering ensures their preferences are spread uniformly across the table. With M = 65,537 and 100 backends, each backend gets 655 ± 1 slots — a variance of less than 0.2%.

When a Backend Goes Down

Rebuild the lookup table without the failed backend. The new table reassigns only the failed backend's slots to other backends. Since the remaining backends' permutations have not changed, most of their existing slot assignments are preserved. In practice, only about 1/N of the table entries change — the slots that previously belonged to the failed backend.

Multi-Probe Consistent Hashing (Appleton & O'Reilly, 2015)

Instead of placing multiple vnodes per physical node, you probe the ring multiple times for each key and pick the closest node across all probes. With k probes, the peak-to-mean load ratio drops to 1 + 1/k. This achieves better balance than vnodes with less memory overhead (no need to store vnode positions), at the cost of k hash computations per lookup.

How Multi-Probe Works

Each physical server is placed at just one position on the ring (no vnodes — saving memory). For each key lookup, instead of hashing the key once, you hash it k times using different hash functions (or the same hash function with different salts: hash(key + "probe-0"), hash(key + "probe-1"), etc.). Each probe gives a different position on the ring. For each position, find the nearest clockwise server and compute the distance. Pick the server with the smallest distance across all k probes.

Worked Example: k=2, 3 Servers

Servers: A at 500M, B at 1.8B, C at 3.2B (same as our earlier example).

Probe 2 has the smaller distance (100M < 300M), so "user:7" is assigned to Server A.

The Tradeoff

With k = 2 probes, load variance drops by roughly 50% compared to k = 1 (a single hash). With k = 7, the peak-to-mean ratio is about 1.14 — very close to perfect balance. But each lookup requires k hash computations and k binary searches through the sorted server list. Typical values of k are 2 to 7.

Memory advantage: With 1,000 servers and no vnodes, the ring has just 1,000 entries (32 KB). Compare this to 200 vnodes per server, which would require 200,000 entries (6.4 MB). Multi-probe trades a small amount of CPU per lookup (extra hash computations) for a large memory saving.

Rendezvous Hashing (HRW — Highest Random Weight)

For each key, compute a score for every node using hash(key, node) and pick the node with the highest score. This guarantees minimal disruption (only keys assigned to a departing node move) and perfect consistency. The downside is O(N) lookup time (you must evaluate every node), which becomes expensive for very large clusters. It excels in small-to-medium node sets where the O(N) scan is acceptable and the simplicity is appealing.

Assumptions & Model

Key Assumptions

  • Uniform hash function. The hash function distributes keys and node identifiers uniformly across the hash space. If the hash function has clustering behavior, the ring will have uneven arcs and load imbalance.
  • Random key distribution. Analysis assumes keys are drawn uniformly at random. Skewed access patterns (e.g., a few hot keys) will cause hot spots regardless of ring structure.
  • Independent node positions. Each node's position on the ring is determined solely by its hash. If node identifiers are sequential (e.g., "node1", "node2"), some hash functions may cluster them.
  • Homogeneous node capacity. The basic model assumes all nodes have equal capacity. Heterogeneous nodes require weighted vnodes (more vnodes for higher-capacity nodes).

When assumptions break down: Skewed access patterns create hot nodes even with perfect ring balance. Non-uniform hash functions cause arcs of vastly different sizes. Virtual nodes mitigate ring imbalance but add memory overhead proportional to the number of vnodes. Under extreme skew (e.g., a viral celebrity's data), application-level mitigation (secondary splitting, request coalescing) is needed — no hashing scheme alone can fix it.

Sizing & Tradeoffs

Virtual Node Count

The central parameter in consistent hashing is the number of virtual nodes (vnodes) per physical node. There is a direct tradeoff:

  • More vnodes = better balance. With V vnodes per physical node, the standard deviation of load across N nodes is approximately 1 / sqrt(V) of the mean load. Doubling vnodes reduces imbalance by ~30%.
  • More vnodes = more memory. Each vnode is an entry in a sorted data structure (typically a balanced BST or sorted array). Each entry is roughly 20–32 bytes (hash value + pointer to physical node).

The Math

With N physical nodes and V vnodes each, the ring has N×V points. The expected load per node is K/N (where K is total keys). The standard deviation of load is:

σ ≈ (K/N) × (1/√V)

The coefficient of variation (CV = σ/mean) is 1/√V.

SLO-Driven Sizing Examples

In production, ring parameters are not chosen in a vacuum — they are derived from service-level objectives. Below are four real-world scenarios showing how to work backwards from SLOs to concrete hash ring configuration.

Example 1: Distributed Cache Cluster

SLOs: p99 latency < 5 ms, throughput 100K RPS, load variance across nodes < 10%.

  • Cluster: 20 Memcached nodes, each rated for 6K RPS with headroom.
  • Load variance target: CV ≤ 5% so that 2σ (95th percentile deviation) stays under 10%. Solve: 1/√V ≤ 0.05 → V ≥ 400.
  • Ring size: 20 × 400 = 8,000 vnodes. Memory: 8,000 × 32 bytes = 256 KB — negligible.
  • Lookup latency: log2(8,000) ≈ 13 comparisons ≈ 0.2 μs. Well within the 5 ms budget (network RTT dominates).
  • Result: 400 vnodes/node, ring fits in L2 cache, p99 dominated by network — SLO met.

Example 2: CDN Edge Routing

SLOs: p95 origin-fetch latency < 50 ms, cache miss rate increase < 5% during scale-out, rebalance completes in < 30 s.

  • Cluster: 50 edge PoPs, scaling to 55 during traffic spikes (add 5 nodes).
  • Key redistribution on scale-out: Ideally K/N′ keys move (K = total cached objects). With 5 new nodes added to 50, expected movement is 5/55 ≈ 9.1% of keys — within the 5% miss-rate budget if existing nodes retain ~91% of their content.
  • Vnode count: 200 vnodes/node keeps CV at 1/√200 ≈ 7%. During scale-out, the 5 new nodes (1,000 vnodes) interleave with 10,000 existing vnodes, distributing the load drain evenly.
  • Rebalance strategy: Background pull with rate limiting — new nodes fetch content lazily on first miss. Prewarming top-1000 hot keys cuts cold-start misses by ~80%.
  • Result: 200 vnodes/node, lazy pull rebalance with hot-key prewarming, ring size 11,000 after expansion.

Example 3: Sharded Database Router

SLOs: p99 routing lookup < 2 ms, 1,000 shards, shard load imbalance < 5%.

  • Shard count: 1,000 physical shards across 100 database hosts (10 shards/host).
  • Imbalance target: CV ≤ 2.5% (so 2σ < 5%). Solve: 1/√V ≤ 0.025 → V ≥ 1,600 vnodes per shard.
  • Ring size: 1,000 × 1,600 = 1.6M vnodes. Memory: 1.6M × 32 bytes = 51.2 MB. This is significant — use a precomputed lookup table (Maglev-style) instead of a sorted ring to get O(1) lookups.
  • Lookup table: A Maglev table of size 65,537 (prime) maps each hash slot to a shard. Lookup is a single modulo + array index ≈ 50 ns, well under 2 ms.
  • Trade-off: The Maglev table sacrifices the minimal-disruption property (more keys move on resize), but with 1,000 fixed shards that rarely change, this is acceptable.
  • Result: Maglev lookup table (65,537 entries × 2 bytes = 128 KB), O(1) lookup, shard balance within 0.5%.

Example 4: Session Affinity for WebSocket Gateways

SLOs: 99.9% session preservation during rolling deploys, connection migration completes in < 500 ms.

  • Cluster: 12 WebSocket gateway nodes, rolling deploy replaces 1 node at a time.
  • Session preservation: When 1 of 12 nodes is replaced, only 1/12 ≈ 8.3% of sessions should migrate. With consistent hashing, the departing node's sessions move to successors, while the other 91.7% are untouched — satisfying the 99.9% SLO for non-affected sessions.
  • Vnode count: 300 vnodes/node ensures the 8.3% of migrating sessions spread across all remaining 11 nodes (each absorbs ~0.75%), preventing any single node from being overwhelmed.
  • Migration protocol: The new node joins the ring before the old one leaves (two-phase swap). During the overlap window (~500 ms), both nodes accept the session's traffic, and the old node forwards state to the new one via a handoff RPC.
  • Ring size: 12 × 300 = 3,600 vnodes. Memory: 115 KB. Lookup: log2(3,600) ≈ 12 comparisons.
  • Result: 300 vnodes/node, two-phase node swap, 99.9% sessions unaffected per deploy step.

SLO-Driven Capacity Planning

When designing a consistent hashing deployment, start from the SLO and work backwards to the ring parameters:

  1. Define the load variance budget. Express your SLO as a maximum acceptable deviation from mean load (e.g., ±10% at p95). Convert this to a target CV: if you want 2σ ≤ X%, then CV ≤ X/2%.
  2. Derive vnode count. Use V ≥ (1 / CV)2. For CV = 5%, V ≥ 400. For CV = 3%, V ≥ 1,111.
  3. Check memory budget. Total ring entries = N × V. Each entry ≈ 32 bytes. If the ring exceeds L2 cache (typically 256 KB–1 MB), consider a precomputed lookup table (Maglev-style) for O(1) lookups instead of O(log(N×V)) binary search.
  4. Check lookup latency budget. Binary search over a sorted ring takes log2(N×V) comparisons. Each comparison is ~5–10 ns. For 10,000 ring entries, that is ~14 comparisons ≈ 100 ns. If your SLO requires sub-microsecond lookups, this is fine. If the ring exceeds 100K entries and you need < 1 μs, switch to a hash table.
  5. Model scaling events. When adding M nodes to an N-node cluster, approximately M/(N+M) of keys will redistribute. Ensure this fraction does not exceed your cache miss-rate SLO during scale-out. If it does, use prewarming or phased rollout.
  6. Validate with simulation. Use the interactive graph above or a Monte Carlo simulation to verify that your chosen vnode count achieves the target CV with your specific node count. Small clusters (N < 10) may need higher V than the formula suggests due to higher variance.

Lookup Performance

Finding the responsible node for a key is a binary search over the sorted ring: O(log(N×V)). With 50 nodes and 150 vnodes each, that is log2(7,500) ≈ 13 comparisons — sub-microsecond on modern hardware.

Scaling Journey

0–1K users

Simple Modulo Hash

A single application server with a local in-memory cache, or a fixed set of 2–3 cache servers with hash(key) % N. The cluster never changes size, so modulo works fine. No ring data structure needed.

1K–100K users

Consistent Hash Ring

You add a Memcached or Redis cluster that scales up and down. Switching to consistent hashing with virtual nodes means adding a cache node does not flush the entire cluster. Each node gets 100–150 vnodes. Library-based (e.g., ketama in Memcached clients).

100K–10M users

Maglev / Jump Hash

At scale, you need tighter load balance and faster lookups. Adopt Maglev hashing for your load balancer tier (uniform distribution, O(1) lookup via precomputed table). Use consistent hashing with bounded loads (Google 2017) to cap the maximum load any single node receives at (1+ε) × average.

10M+ users

Multi-Tier Hashing

Consistent hashing for coarse-grained routing (which region/cluster), then a secondary hash within each shard for fine-grained distribution. Weighted vnodes accommodate heterogeneous hardware. Custom rebalancing controllers manage migrations during topology changes, throttling data movement to protect latency.

Migration Signals

  • Modulo → Consistent Hash: You notice that adding a cache node causes a spike in database load (cache miss storm). This is the primary signal.
  • Basic Ring → Bounded Loads: You see significant load imbalance across nodes, with one or two nodes running hot despite uniform key distribution.
  • Single-Tier → Multi-Tier: Your cluster spans multiple data centers or regions and you need geographic affinity in routing.

What Can Go Wrong

1. Cascading Hot Spots on Node Failure

What happens: When a node goes down, all its keys transfer to its clockwise successor. That successor now handles double its normal load. If it cannot keep up and also fails, its keys cascade to the next node — a domino effect.

Root cause: The ring assigns the entire arc of a failed node to one successor. Without replication or load shedding, the successor is overloaded.

Blast radius: In the worst case, cascading failures can take down the entire ring.

Detection: Monitor per-node request rate and latency. Alert when a node's load exceeds 1.5× its baseline.

Mitigation: Use virtual nodes (so a failed node's keys spread across many successors rather than one). Implement circuit breakers and load shedding. Replicate data to N successors on the ring so failover has warm data.

2. Vnode Imbalance with Few Physical Nodes

What happens: With only 3 physical nodes and 50 vnodes each, random placement can still create arcs where two of the three nodes are under-loaded and one holds 45% of keys instead of 33%.

Root cause: Statistical variance is high with few total vnodes (150 in this case). The law of large numbers has not kicked in.

Mitigation: Increase vnodes per node (200+) or use deterministic placement algorithms like Maglev hashing which guarantee near-perfect balance.

3. Hash Collisions in Small Key Spaces

What happens: If the key space is small (e.g., only 50 distinct keys), the granularity of the ring does not help — the same few ring positions are always hit, and adding vnodes does not improve balance because the keys themselves are clustered.

Root cause: Consistent hashing assumes a large, uniformly distributed key space. With few keys, random chance determines balance.

Mitigation: For small key spaces, use explicit mapping or rendezvous hashing, which evaluates all nodes for each key and is more robust with few keys.

Interactions & Pitfalls

Consistent Hashing + Caching

When a new cache node joins the ring, the keys it takes over from its neighbors will be cold (empty cache). Until those keys are populated, requests will fall through to the database, causing a temporary load spike. Mitigation: Pre-warm the new node by copying data from its neighbors before routing traffic to it. Some systems (e.g., Groupcache) use a "hot cache" layer that temporarily redirects cache misses to the previous owner.

Consistent Hashing + Replication

A common pattern is to replicate each key to the N successors on the ring (e.g., DynamoDB stores data on the coordinator node plus the next N−1 clockwise nodes). Pitfall: with virtual nodes, those N successors might map to fewer than N physical nodes. The replication logic must skip vnodes that belong to an already-selected physical node to ensure data is spread across distinct machines.

Consistent Hashing + Load Balancing

Google's "Consistent Hashing with Bounded Loads" (Mirrokni et al., 2017) extends the basic ring to cap the maximum load on any node at (1+ε) × average. When a key's natural node is at capacity, the key is forwarded clockwise to the next node with room. This prevents hot spots but introduces a dependency between the load balancer and real-time load metrics.

Cost Considerations

Cost Breakdown

  • Memory for the ring: Minimal. Even with 1,000 nodes × 200 vnodes × 32 bytes = 6.4 MB. This is negligible compared to the data being routed.
  • CPU for hash computation: MurmurHash3 hashes at ~4 GB/s on a single core. For typical key sizes (64–256 bytes), a single hash costs ~50 ns. Even at 1M lookups/sec, hash computation uses <5% of a single CPU core.
  • Operational cost of rebalancing: The dominant cost. Moving keys involves reading from the old node, transferring over the network, and writing to the new node. At scale (terabytes of data), a single node addition can mean moving hundreds of GB. Throttling rebalance bandwidth (e.g., to 100 Mbps) means a 100 GB migration takes ~2.5 hours.
  • Cost of NOT having consistent hashing: Without it, every topology change triggers a near-complete redistribution. For a 10-node cache cluster holding 50 GB each (500 GB total), adding one node with modulo hashing means ~450 GB of cache data becomes invalid simultaneously. The resulting thundering herd to the database can cause minutes of downtime and require significant DB over-provisioning (~5× headroom) to absorb.

Real-World Examples

Amazon DynamoDB

DynamoDB (originally described in the 2007 Dynamo paper) uses consistent hashing as its core partitioning mechanism. Each partition key is hashed onto a ring, and data is stored on the coordinator node plus N−1 successor nodes for replication. Virtual nodes allow DynamoDB to balance load across heterogeneous hardware and to perform fine-grained rebalancing when nodes join or leave. The system uses a gossip-based protocol to propagate ring membership changes.

Apache Cassandra

Cassandra partitions data across nodes using consistent hashing with configurable vnodes (default: 256 per node, though recent versions recommend 16–32 with token-aware allocation). The partition key is hashed using MurmurHash3 to a 64-bit token space, and each node owns a set of token ranges. When a node joins, it receives tokens from existing nodes, triggering streaming of the associated data. Cassandra's nodetool status command shows each node's ownership percentage, letting operators detect imbalance.

Akamai CDN

Akamai was co-founded by one of the original consistent hashing paper authors (David Karger). Their CDN uses consistent hashing to map content URLs to edge servers. When a server goes down, only the content it was responsible for needs to be re-fetched from origin, preventing a surge of cache misses across the entire fleet. This was one of the first large-scale production uses of consistent hashing and remains one of the most impactful.

Code Example

A minimal consistent hash ring in Python with virtual nodes:

import hashlib
from bisect import bisect_right

class ConsistentHashRing:
    def __init__(self, nodes=None, vnodes=150):
        self.vnodes = vnodes
        self.ring = {}       # hash_value -> node
        self.sorted_keys = []
        for node in (nodes or []):
            self.add_node(node)

    def _hash(self, key):
        return int(hashlib.md5(key.encode()).hexdigest(), 16)

    def add_node(self, node):
        for i in range(self.vnodes):
            vnode_key = f"{node}:{i}"
            h = self._hash(vnode_key)
            self.ring[h] = node
        self.sorted_keys = sorted(self.ring.keys())

    def remove_node(self, node):
        for i in range(self.vnodes):
            vnode_key = f"{node}:{i}"
            h = self._hash(vnode_key)
            del self.ring[h]
        self.sorted_keys = sorted(self.ring.keys())

    def get_node(self, key):
        if not self.ring:
            return None
        h = self._hash(key)
        idx = bisect_right(self.sorted_keys, h)
        if idx == len(self.sorted_keys):
            idx = 0  # wrap around the ring
        return self.ring[self.sorted_keys[idx]]

# Usage
ring = ConsistentHashRing(["server-A", "server-B", "server-C"])
print(ring.get_node("user:1234"))   # -> "server-B"
print(ring.get_node("session:abcd")) # -> "server-A"

ring.add_node("server-D")  # only ~25% of keys remap
print(ring.get_node("user:1234"))   # likely unchanged

How to Validate Your Design

Load Testing

Simulate realistic key distributions (uniform, Zipfian, and hot-key skewed) and measure key distribution across nodes. Use tools like k6 or Locust to drive cache requests through your consistent-hash routing layer. Verify that p99 lookup latency stays under 1 ms and that load variance across nodes stays within your target CV.

Chaos Testing

Use Chaos Monkey or toxiproxy to randomly kill cache nodes and measure:

  • What percentage of keys actually moved (should be close to K/N, not K).
  • How long the rebalance takes and whether it impacts live traffic.
  • Whether cascading failures occur when a node's successor absorbs its load.

Inject network partitions between ring members to test how your gossip-based membership protocol handles split-brain scenarios.

Benchmarking Methodology

  • Key variable to sweep: Number of vnodes (50, 100, 150, 200, 400). Measure load variance vs. lookup latency vs. memory usage.
  • Common mistake: Benchmarking with uniform keys only. Real workloads are skewed — always test with Zipfian distributions (s=1.0 and s=1.5).
  • Common mistake: Measuring average lookup time instead of p99. The sorted-ring binary search has consistent latency, but if you are rebuilding the ring during benchmarking (from add/remove operations), you may see latency spikes.
  • Tool: Write a simple script that hashes 10M keys, distributes them across N nodes, and reports min/max/mean/stddev of per-node counts. This takes seconds and gives immediate feedback on balance quality.

Interview Questions

How does consistent hashing minimize data movement?

In consistent hashing, both nodes and keys are mapped onto the same circular hash space. Each key is assigned to the nearest clockwise node. When a node is added, it only absorbs keys from the arc between itself and its counter-clockwise predecessor. When a node is removed, only its keys transfer to its clockwise successor. In both cases, the number of keys that move is approximately K/N (total keys divided by number of nodes), compared to modulo hashing where nearly all keys remap on any topology change.

The mathematical intuition is simple: adding one node to an N-node ring splits one arc into two. Only the keys in that one arc are affected — roughly 1/N of all keys. All other arcs (and their key assignments) remain unchanged.

Follow-up: What is the exact expected number of keys that move when adding one node to a ring with N nodes and K keys? (Answer: K/(N+1))

What are virtual nodes and why do we need them?

Virtual nodes (vnodes) are multiple hash positions assigned to each physical node. Instead of hashing a node once and placing it at one point on the ring, you hash it many times (e.g., "NodeA-0", "NodeA-1", ..., "NodeA-149") to create 150 positions per physical node.

Without vnodes, a small number of nodes on a large ring creates uneven arc sizes. With just 3 nodes, one might own 50% of the ring while another owns only 15%. Virtual nodes break each node's responsibility into many small arcs scattered around the ring, making the distribution much more uniform. The standard deviation of load drops as 1/sqrt(V) where V is the vnode count.

A secondary benefit: when a node goes down, its vnodes are scattered across the ring, so its keys are absorbed by many different successors rather than overloading a single successor.

Follow-up: What is the memory overhead of virtual nodes? (Answer: V entries per node in a sorted data structure, each ~20-32 bytes. With 200 vnodes and 100 nodes, that is 20,000 entries = ~640 KB.)

Compare consistent hashing vs. rendezvous hashing.

Consistent hashing uses a ring with O(log N) lookup (binary search on sorted ring positions). It requires a data structure (the ring) to be maintained, and virtual nodes to achieve good balance. It scales well to large node counts because lookup time is logarithmic.

Rendezvous hashing (Highest Random Weight) computes hash(key, node) for every node and picks the one with the highest value. Lookup is O(N) — you must check every node. It requires no ring data structure and achieves perfect minimal disruption (when a node leaves, only its keys move). Balance is naturally very good without virtual nodes.

Use consistent hashing when N is large (hundreds to thousands of nodes) because O(log N) << O(N). Use rendezvous hashing when N is small (fewer than ~50 nodes), where the O(N) scan is cheap, and you value the simplicity of not maintaining a ring data structure.

Follow-up: Can you combine the two? (Answer: Yes — use consistent hashing at the top tier to select a cluster, then rendezvous hashing within the cluster to pick a node.)

Design a distributed cache using consistent hashing.

Start with the high-level architecture: a fleet of cache nodes (e.g., Memcached or Redis instances), a consistent hash ring maintained by each client (or by a routing proxy), and a backing data store.

Write path: The client hashes the cache key, finds the responsible node on the ring, and writes the value there. Optionally replicate to the next N-1 clockwise nodes for durability.

Read path: Hash the key, find the responsible node, and read. On cache miss, fetch from the data store, populate the cache, and return.

Topology changes: Use a coordination service (ZooKeeper, etcd, or gossip protocol) to maintain the ring membership. When a node joins or leaves, clients update their local ring copy. Keys that now map to a different node will experience a cache miss on first access (lazy migration) or can be proactively migrated (eager migration).

Hot key handling: For extremely hot keys, replicate them to multiple nodes or add a small local (in-process) cache layer in front of the distributed cache.

Follow-up: How do you handle the thundering herd problem when a cache node goes down? (Answer: Use request coalescing — only one request fetches from the data store while others wait. Also, replicate to successor nodes so failover serves warm data.)

What happens during a rebalance and how do you minimize impact?

When a node joins or leaves, keys need to migrate. During this transition window, some keys are "in flight" — they might be on the old node, the new node, or being transferred. This creates a consistency window where reads might miss data that is mid-transfer.

Minimizing impact:

  • Throttle data transfer to avoid saturating network bandwidth and causing latency spikes for live traffic. Limit rebalance traffic to 10-20% of available bandwidth.
  • Double-read during migration: During the transition, route reads to both the old and new owner. Return whichever has the data. Once migration is confirmed complete, route only to the new owner.
  • Background streaming: Transfer data in the background while both old and new owners serve requests. Cassandra's "streaming" protocol does exactly this.
  • Staged rollout: Don't add multiple nodes simultaneously. Add one at a time and let rebalancing complete before adding the next.

Follow-up: What if a node fails during rebalancing? (Answer: The system must handle partial migrations. Using replicated data on successor nodes provides a safety net — the successor already has the data and can serve it.)

How does DynamoDB use consistent hashing?

DynamoDB (based on the original Dynamo paper) uses consistent hashing as its primary data partitioning strategy. Each item's partition key is hashed to determine its position on the ring, and the item is stored on the node responsible for that ring segment.

Key details of DynamoDB's implementation:

  • Replication: Each item is replicated to N nodes (typically N=3). The coordinator node and its N-1 clockwise successors on the ring form a "preference list." Reads and writes go to W and R nodes respectively (configurable quorum).
  • Virtual nodes: Each physical node owns multiple tokens (positions on the ring), allowing heterogeneous hardware to take proportionally more load.
  • Membership: Ring membership is disseminated via a gossip protocol. Nodes periodically exchange membership state, and the system converges on a consistent view.
  • Hinted handoff: When a node is temporarily down, writes destined for it are stored on another node with a "hint." When the target recovers, the hinted data is forwarded to it. This maintains write availability during transient failures.

Follow-up: How does DynamoDB handle the case where virtual nodes of the same physical node appear consecutively on the ring? (Answer: The preference list skips vnodes belonging to physical nodes already in the list, ensuring replicas are on distinct physical machines.)

Key Takeaways

  • Consistent hashing maps both keys and nodes onto a ring, so only O(K/N) keys move when a node is added or removed — a massive improvement over modulo hashing's near-total redistribution.
  • Virtual nodes are essential for load balance. Use 100–200 vnodes per physical node as a default. The standard deviation of load drops as 1/√V.
  • The ring lookup is O(log(N×V)) via binary search — extremely fast even for large clusters.
  • Modern alternatives (Jump Hash, Maglev, multi-probe) improve on the original in specific dimensions: perfect balance, O(1) lookup, or lower memory. Choose based on whether you need arbitrary node removal.
  • The biggest operational risk is cascading failure: when a node dies, its successor absorbs all its load. Virtual nodes spread this impact across multiple successors.
  • Consistent hashing interacts with caching (cold starts on new nodes), replication (skip duplicate physical nodes), and load balancing (bounded loads extension).
  • Memory cost is trivial (KBs to low MBs). The real cost is data migration during rebalancing — throttle it to protect live traffic.
