Rate Limiting

Controlling the rate of incoming requests to protect services from overload, abuse, and unfair resource consumption.

25 min read

What Is It?

Rate limiting is a technique for controlling the number of requests a client can make to a service within a given time window. When a client exceeds the allowed rate, the service rejects additional requests — typically with an HTTP 429 "Too Many Requests" status code — rather than processing them. The rejected client is told when it can retry, usually via a Retry-After header.

Think of rate limiting like a bouncer at a nightclub. The venue has a maximum occupancy. Once it is full, the bouncer turns people away and tells them to come back later. This prevents the venue from becoming dangerously overcrowded, ensures the people inside have a good experience, and stops any one group from monopolizing the space.

Why it matters:

  • Prevent abuse: Stops malicious actors from flooding your API with requests, whether for scraping, credential stuffing, or denial-of-service attacks.
  • Ensure fairness: In a multi-tenant system, prevents one noisy tenant from consuming all the shared capacity, starving other tenants.
  • Protect downstream services: Your API might handle 10K RPS, but the database behind it might only handle 2K. Rate limiting at the edge prevents the cascade of overload from reaching fragile backends.
  • Cost control: In cloud environments where you pay per request or per compute-second, runaway traffic from a bug or a misconfigured client can generate enormous bills. Rate limiting puts a ceiling on this.

How It Works

Token Bucket (Step-by-Step)

The token bucket is the most widely used rate limiting algorithm. Here is how it works:

  1. Initialize: Create a bucket with a maximum capacity (e.g., 10 tokens) and a refill rate (e.g., 5 tokens per second). Start with the bucket full.
  2. Token refill: At a fixed interval, add tokens to the bucket. If the bucket is already full, new tokens are discarded — they do not accumulate beyond capacity.
  3. Request arrives: When a request comes in, check if there is at least one token in the bucket.
  4. Accept: If a token is available, remove one token from the bucket and process the request.
  5. Reject: If the bucket is empty, reject the request immediately with HTTP 429. Set Retry-After and X-RateLimit-Remaining: 0 headers.
  6. Burst handling: Because tokens accumulate, a client that has been idle can send a burst of requests up to the bucket capacity. This is a feature, not a bug — it allows legitimate burst behavior.

Other Algorithms

Leaky Bucket

Requests enter a queue (the bucket) and are processed at a fixed rate, like water leaking from a hole. Unlike the token bucket, the leaky bucket enforces a strict, smooth output rate — no bursts are allowed. If the queue is full, new requests are dropped. This is ideal when you need a perfectly smooth request rate to protect a downstream service that cannot handle any burstiness at all.

Leaky Bucket — Deep Dive

How it differs from the token bucket: The token bucket controls input — it decides whether to accept or reject each incoming request. The leaky bucket controls output — it accepts requests into a queue and processes them at a fixed, constant rate, smoothing out any burstiness in the input. Think of it literally: a bucket with a small hole at the bottom.

  • Water (requests) pours in at varying, unpredictable rates — sometimes a trickle, sometimes a flood.
  • Water leaks out through the hole at a constant rate R (e.g., 1 request per second).
  • The bucket has a finite capacity B (the queue size). If the bucket is full and more water arrives, the excess overflows and is discarded (request dropped).

This means: no matter how bursty the input, the output is always smooth. A downstream service behind a leaky bucket sees a perfectly steady stream of requests, never more than R per second.

Key difference from token bucket: The token bucket says "you can send up to B at once, then R per second after that." The leaky bucket says "output is always exactly R per second, regardless of how bursty the input is." Token bucket permits bursts. Leaky bucket absorbs bursts.

Worked example: Leak rate R = 1 request/second, bucket capacity B = 4.

Queue-based implementation:

import time
from collections import deque

class LeakyBucket:
    def __init__(self, rate, capacity):
        self.rate = rate              # requests leaked per second
        self.capacity = capacity      # max queue size
        self.queue = deque()          # timestamps of queued requests
        self.last_leak = time.monotonic()

    def _leak(self):
        """Remove requests that have been 'processed' since last check."""
        now = time.monotonic()
        elapsed = now - self.last_leak
        leaked = int(elapsed * self.rate)
        for _ in range(min(leaked, len(self.queue))):
            self.queue.popleft()
        if leaked > 0:
            self.last_leak = now

    def allow(self):
        self._leak()
        if len(self.queue) < self.capacity:
            self.queue.append(time.monotonic())
            return True    # accepted into the queue
        return False       # bucket full — request dropped

# Usage: leak rate = 1 req/s, capacity = 4
bucket = LeakyBucket(rate=1, capacity=4)

Fixed Window Counter

Divide time into fixed windows (e.g., each minute). Maintain a counter for each window. Increment the counter for each request; reject if the counter exceeds the limit. Simple to implement but has a boundary problem: a client can send the full limit at the end of one window and the full limit at the start of the next, doubling the effective rate across the boundary.

Sliding Window Log

Store a timestamp for every request in a sorted set. When a new request arrives, remove all timestamps older than the window, then count the remaining entries. This eliminates the boundary problem but requires storing every timestamp — memory usage scales linearly with traffic, making it expensive at high rates.

Sliding Window Log — Deep Dive

The data structure: Store the timestamp of every request in a sorted set (e.g., a Redis ZSET, where the score is the timestamp). To check if a new request should be allowed:

  1. Remove all timestamps older than (now - window_size). These are requests that have "expired" — they happened outside the current sliding window and should no longer count.
  2. Count the remaining timestamps. This is the number of requests that happened within the current window.
  3. If count < limit, allow the request and add the current timestamp to the set.
  4. If count ≥ limit, reject the request.

Why it is perfectly accurate: Unlike fixed window counters (which have a boundary problem) or sliding window counters (which use an approximation), the sliding window log tracks the exact timestamp of every request. There is no approximation — the count is always exactly the number of requests in the last window_size seconds.

Memory analysis — why it does not scale for high-rate limits:

  • For a limit of 100 requests per 60 seconds, each user stores up to 100 timestamps. Each timestamp is 8 bytes. For 1M users: 1M × 100 × 8 = 800 MB.
  • For an API gateway limiting at 10,000 requests/sec per user: 10K × 8 bytes × 1M users = 80 GB.
  • This is why the sliding window log is practical only for low-volume, high-accuracy use cases (e.g., login attempt limiting at 5 per minute) — not for high-throughput API rate limiting.

Redis implementation:

import time
import redis

def sliding_window_log(redis_client, key, limit, window_secs):
    """
    Returns True if the request is allowed, False if rejected.
    Uses a Redis sorted set (ZSET) where each member is a unique
    request ID and the score is the timestamp.
    """
    now = time.time()

    # Use a pipeline to send all commands in one round-trip
    pipe = redis_client.pipeline()

    # Step 1: Remove all entries older than the window
    pipe.zremrangebyscore(key, 0, now - window_secs)

    # Step 2: Add the current request's timestamp
    pipe.zadd(key, {str(now): now})

    # Step 3: Count how many entries remain
    pipe.zcard(key)

    # Step 4: Set a TTL so the key auto-expires
    pipe.expire(key, window_secs)

    results = pipe.execute()
    count = results[2]  # zcard result

    if count > limit:
        # Undo the add — this request should not count
        redis_client.zrem(key, str(now))
        return False  # rejected

    return True  # allowed

Note: This pipeline is not fully atomic — another request could slip in between the zadd and the zcard. For strict atomicity, wrap this logic in a Lua script (see the distributed rate limiting section below).

Sliding Window Counter

A hybrid approach. Maintain counters for the current and previous window. Weight the previous window's count by the overlap fraction. For example, if you are 30% into the current minute, the effective count is current_count + previous_count * 0.7. This approximation eliminates the boundary spike with minimal memory overhead — just two counters per window.

Sliding Window Counter — Deep Dive

The core idea: Maintain two simple counters — one for the current fixed window and one for the previous fixed window. To estimate how many requests occurred in the last window_size seconds, compute a weighted combination of both counters.

The formula:

estimated_count = previous_window_count × overlap_fraction + current_window_count

Where: overlap_fraction = 1 - (elapsed_time_in_current_window / window_size)

The overlap fraction represents how much of the previous window still falls within our sliding window. If we are 15 seconds into a 60-second window, 45 seconds of the previous window still overlap — so the overlap fraction is 1 - (15/60) = 0.75.

Worked example:

  • Window size: 60 seconds. Limit: 100 requests.
  • Previous window (T=0 to T=60): 84 requests
  • Current window (T=60 to T=120): 36 requests so far
  • Current time: T=75 (15 seconds into the current window)

Calculation:

  • Overlap fraction = 1 - (15 / 60) = 0.75
  • Estimated count = 84 × 0.75 + 36 = 63 + 36 = 99
  • 99 < 100 → ALLOW

Another request arrives at T=76:

  • Current window count is now 37
  • Overlap fraction = 1 - (16 / 60) = 0.733
  • Estimated count = 84 × 0.733 + 37 = 61.6 + 37 = 98.6 → round to 99 → ALLOW

Why this is approximate: We assume requests in the previous window were uniformly distributed across the window. If all 84 requests arrived in the last 15 seconds of the previous window, our estimate would be too low — the true count within the sliding window would be much higher. But in practice, the error is small. According to Cloudflare's analysis, the error rate is less than 0.003% for typical traffic patterns.

Memory advantage: Only 2 counters per key (16 bytes total) vs. the sliding window log which stores every timestamp (potentially thousands of entries). This is why Cloudflare uses sliding window counters for their global rate limiting — it scales to millions of keys with negligible memory.

When to Use / When NOT to Use

Use Rate Limiting When

  • API protection: Any public-facing API should have rate limits. It is the first line of defense against abuse, scraping, and DDoS.
  • Multi-tenant fairness: When multiple customers share infrastructure, rate limits prevent one tenant from monopolizing capacity.
  • Cost control: In pay-per-use cloud architectures, rate limits cap runaway costs from bugs or misconfigured clients.
  • DDoS mitigation: Rate limiting at the edge (CDN/API gateway) absorbs volumetric attacks before they reach your application servers.
  • Compliance requirements: Some regulatory frameworks require demonstrable controls on data access rates.

Don't Use Rate Limiting When

  • Internal trusted services where backpressure is better: Between your own microservices, backpressure (e.g., gRPC flow control, TCP window sizing, or queue-based buffering) is usually better than hard rejection. Rejecting internal requests means you have to build retry logic everywhere.
  • When you need guaranteed delivery, not rejection: If every request must eventually be processed (e.g., payment processing, audit events), use a queue or backpressure instead of dropping requests.
  • Latency-critical paths with predictable load: If traffic is well-understood and auto-scaling handles demand, rate limiting adds unnecessary overhead and latency to the hot path.
  • Single-user tools or batch systems: If there is only one client and you control it, throttling at the client side (e.g., a semaphore or sleep loop) is simpler.

Classic vs. Modern Approaches

Classic Approaches

Token Bucket (1990s, ATM networks): Simple, allows bursts up to bucket capacity, widely implemented. Used in Linux traffic shaping (tc), Nginx, and most API gateways. The main limitation is that in distributed settings, each node maintains its own bucket, so the aggregate rate can exceed the intended limit by a factor of N (number of nodes).

Leaky Bucket: Guarantees a perfectly smooth output rate. Used in network traffic policing (ATM and frame relay). Less common for API rate limiting because most APIs can tolerate bursts and the strict smoothing unnecessarily penalizes legitimate traffic.

Modern Approaches

GCRA (Generic Cell Rate Algorithm)

GCRA is mathematically equivalent to the token bucket but is easier to implement correctly in distributed systems. Instead of tracking token count and last refill time, GCRA tracks a single value: the TAT (Theoretical Arrival Time) — the earliest time the next request should arrive. If a request arrives before TAT, it is rejected. If it arrives after, TAT is updated to max(now, TAT) + emission_interval. Only one value to store, one comparison to make, and one atomic update — perfect for Redis. Used by Shopify's rate limiter and the governor Rust crate.

GCRA — Deep Dive

The core concept: Instead of counting tokens or tracking request counts, GCRA tracks a single value: the TAT (Theoretical Arrival Time) — the earliest time the next request "should" arrive if requests were perfectly spaced. If a request arrives too early (before the system has had time to "recover"), it is rejected.

The algorithm, step by step:

  1. When a request arrives at time now:
  2. Compute the emission interval: emission_interval = window / limit. This is the ideal spacing between requests. For 5 requests per 10 seconds: 10 / 5 = 2.0 seconds.
  3. Compute the new TAT: new_tat = max(tat, now) + emission_interval. The max(tat, now) handles the case where the client has been idle — if TAT is in the past, we reset to "now" so idle time is not penalized.
  4. Check burst tolerance: if new_tat - now > burst_tolerance, the request arrived too early and would exceed the burst limit. REJECT the request.
  5. Otherwise, ALLOW the request and update tat = new_tat.

Where burst_tolerance = emission_interval × burst. The burst parameter controls how many back-to-back requests are allowed.

Worked example: limit = 5 requests per 10 seconds, burst = 2.

  • emission_interval = 10 / 5 = 2.0 seconds
  • burst_tolerance = 2.0 × 2 = 4.0 seconds

Reading the table: Request 1 arrives at T=0. TAT starts at 0, so new_tat = 0 + 2 = 2.0. The difference (2.0 - 0.0 = 2.0) is within the burst tolerance of 4.0, so it is allowed. Request 2 arrives just 0.5 seconds later. TAT is 2.0 (in the future), so new_tat = 2.0 + 2 = 4.0. The difference (4.0 - 0.5 = 3.5) is still within tolerance. But request 3 at T=1.0 pushes new_tat to 6.0, and 6.0 - 1.0 = 5.0 exceeds the burst tolerance of 4.0 — rejected. By T=3.0, enough time has passed that request 4 is allowed again.

Why GCRA is equivalent to a token bucket: GCRA's TAT tracks the same state as a token bucket's token count — it is just inverted. When TAT is in the future, it means you have "used up" tokens. When TAT is in the past, tokens have been accumulating. The burst_tolerance maps directly to the bucket capacity.

Why GCRA is better for distributed systems: Only one value needs to be stored — the TAT. A single Redis key with a GETSET (or SET ... GET in Redis 6.2+) is atomic. There is no need for multi-key transactions like INCR + EXPIRE, no race conditions between reading and writing two separate values, and no Lua script required for basic operation.

import time

class GCRA:
    def __init__(self, rate, burst):
        """
        rate:  allowed requests per second
        burst: max burst size (extra requests beyond the steady rate)
        """
        self.emission_interval = 1.0 / rate     # seconds between requests
        self.burst_tolerance = self.emission_interval * burst
        self.tat = 0.0  # Theoretical Arrival Time

    def allow(self):
        now = time.monotonic()
        new_tat = max(self.tat, now) + self.emission_interval

        if new_tat - now > self.burst_tolerance:
            return False  # too early — reject

        self.tat = new_tat
        return True       # allowed

# 5 requests per 10 seconds, burst of 2
limiter = GCRA(rate=0.5, burst=2)

Adaptive / Dynamic Rate Limiting

Instead of static limits, the rate adjusts dynamically based on system health. If CPU utilization is above 80% or p99 latency exceeds the SLO, the rate limiter tightens. If the system is healthy, limits relax. Netflix's concurrency-limits library implements this using TCP-congestion-control-inspired algorithms (Vegas, Gradient). The key insight: the optimal rate is not a fixed number you configure — it is a function of current system capacity, which changes constantly.

Distributed Rate Limiting with Redis + Lua

For distributed enforcement, the standard approach is Redis with Lua scripts for atomicity. A Lua script running inside Redis can check the current count, increment it, and set the TTL in a single atomic operation — no race conditions. The sliding window counter algorithm is commonly implemented this way. Cloudflare, Stripe, and GitHub all use variants of this pattern. The tradeoff: Redis becomes a dependency on the critical path. If Redis is down, you must decide: fail open (allow all traffic) or fail closed (reject all traffic).

Distributed Rate Limiting — Deep Dive

Why local rate limiters fail: If your limit is 100 req/s and you have 10 servers, you might configure each server with a local limiter of 10 req/s. If load balancing distributes requests perfectly evenly, a client gets exactly 100 req/s total — correct. But load balancing is never perfectly even. Some servers might see 15 req/s from a client (rejected at 10) while others see only 5 req/s. The effective global limit becomes unpredictable — the client may get anywhere from 50 to 100 req/s depending on how requests are distributed. This is why you need a centralized counter.

The race condition with INCR and EXPIRE:

The naive Redis approach looks simple but has a critical bug:

# WRONG — Race condition!
count = redis.incr("rate:user42")    # count is now 1
if count == 1:
    redis.expire("rate:user42", 60)  # set TTL only on first request

The problem: If the process crashes (or the network drops) between the INCR and the EXPIRE, the key has no TTL. The counter persists forever. The user is permanently rate-limited — every subsequent request increments the counter further, and it never resets because there is no expiration. This is a real production bug that has caused outages.

The fix — Lua script for atomicity:

RATE_LIMIT_SCRIPT = """
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2])

local current = redis.call('INCR', key)
if current == 1 then
    redis.call('EXPIRE', key, window)
end

if current > limit then
    return 0  -- rejected
end
return 1  -- allowed
"""

def check_rate_limit(redis_client, user_id, limit=100, window=60):
    """
    Returns True if allowed, False if rate-limited.
    The Lua script runs atomically inside Redis —
    no other command can execute between INCR and EXPIRE.
    """
    key = f"rate:{user_id}"
    result = redis_client.eval(RATE_LIMIT_SCRIPT, 1, key, limit, window)
    return result == 1

Why this works: Lua scripts execute atomically in Redis — the Redis server is single-threaded, and while a Lua script is running, no other command can execute. This means INCR and EXPIRE happen as one indivisible operation. There is no window for a crash or race condition between them.

Alternative approaches:

  • SET key value EX window NX for the first request (sets the key with a TTL only if it does not already exist), then INCR for subsequent requests. This avoids the race because the TTL is set in the same command as the key creation.
  • MULTI/EXEC transaction block: Groups INCR and EXPIRE into a transaction. Redis guarantees both commands execute without interleaving. However, MULTI/EXEC does not support conditional logic (you cannot read the count and branch on it within the transaction), so Lua scripts are more flexible.

Client-Side Rate Limiting with Exponential Backoff

Server-side rate limiting is the safety net; well-behaved clients should self-limit. The standard pattern is exponential backoff with jitter: after a 429 response, wait base * 2^attempt + random_jitter before retrying. AWS SDKs, Google API clients, and Stripe's client libraries all implement this. The jitter is critical — without it, all throttled clients retry at the same time, causing a thundering herd that re-triggers the rate limit.

Assumptions & Model

Token Bucket Model Assumptions

  • Uniform token refill: Tokens are added at a constant rate. In practice, refill is computed lazily — when a request arrives, the algorithm calculates how many tokens should have been added since the last request, rather than running a background timer.
  • Requests consume 1 token each: All requests are treated as equal cost. This breaks when you have requests that vary wildly in cost — a simple GET vs. a complex aggregation query. Some implementations assign different token costs to different endpoints.
  • Single point of enforcement: The algorithm assumes one authoritative counter. In a distributed system with multiple enforcement points, the same key is checked by multiple nodes, introducing race conditions and clock skew.
  • Independent, memoryless arrivals: The algorithm does not model request patterns. It works fine for steady or random traffic but can be gamed by an adversary who carefully times requests to maximize throughput.

When Assumptions Break

  • Distributed enforcement: With N nodes each running a local token bucket, the aggregate rate is up to N times the intended limit. Centralizing state (e.g., in Redis) fixes this but adds latency and a single point of failure.
  • Clock skew across nodes: If nodes have different clocks, sliding window boundaries and token refill calculations diverge. NTP drift of even a few milliseconds can cause inconsistent enforcement during window transitions.
  • Variable-cost requests: A request that triggers a 10-second database query should cost more tokens than a request served from cache. Without variable costing, rate limiting can under-protect expensive endpoints.
  • Bursty legitimate traffic: A mobile app that batches requests (e.g., sync on app open) may legitimately send 50 requests in 1 second, then nothing for 5 minutes. A low burst capacity penalizes this pattern even though the average rate is well within limits.

Sizing & Tradeoffs

How to Pick the Rate

  1. Start from SLOs: Determine the maximum request rate your service can handle while meeting its latency SLO (e.g., p99 < 100ms).
  2. Measure p99 capacity: Load test to find the throughput at which p99 latency exceeds your SLO. This is your capacity ceiling.
  3. Set at 80%: Set the global rate limit at 80% of the capacity ceiling. The 20% headroom absorbs natural variance, background tasks, and retry storms.

Per-User vs. Global vs. Per-Endpoint Limits

  • Global limit: Protects the service as a whole. Set based on total capacity. Example: 8,000 RPS.
  • Per-user/per-tenant limit: Ensures fairness. Divide global capacity by expected concurrent tenants, with headroom. Example: 100 RPS per tenant.
  • Per-endpoint limit: Protects expensive endpoints independently. A search endpoint that hits the database might have a limit of 10 RPS per user, while a simple status endpoint allows 1,000 RPS.

Burst Size

The burst size (token bucket capacity) determines how much spike you tolerate. A burst size of 1x the per-second rate allows no burst at all. A burst size of 2x allows a client to briefly double its rate if it has been idle. Rule of thumb: set burst capacity to 2-3x the per-second rate limit. Higher burst capacity is more forgiving to legitimate clients but offers less protection against sudden spikes.

SLO-Driven Capacity Planning

Rate limits should never be guessed — they should be derived from your Service Level Objectives. The process is: define the SLO, measure system behavior under load, and mathematically derive the rate limit parameters that keep you within the SLO. The following examples walk through this derivation for four common scenarios.

📐 SLO-Driven Example: Public API with 99.9% Availability

Target SLO: 99.9% availability (no more than 0.1% of requests rejected due to overload per month).

Given: Backend capacity is 12,000 RPS at p99 < 300ms. Average traffic is 4,000 RPS. Peak traffic (95th percentile daily) reaches 9,000 RPS. There are 200 API consumers.

Step 1 — Capacity ceiling: 12,000 × 0.80 = 9,600 RPS global limit (20% headroom for GC pauses, deploys, retry storms).

Step 2 — Burst allowance: Peak-to-average ratio = 9,000 / 4,000 = 2.25×. Set token bucket burst capacity to 3× the per-second rate to absorb legitimate spikes without rejections.

Step 3 — Per-consumer limit: 9,600 / 200 = 48 RPS baseline. Round up to 60 RPS per consumer (allows headroom since not all consumers peak simultaneously). Burst bucket: 60 × 3 = 180 tokens.

Step 4 — Rejection budget check: At 99.9% availability, you can reject at most 0.1% of ~10.4B monthly requests = ~10.4M rejections/month. With the 3× burst allowance, simulated rejection rate at peak is 0.02%, well within budget.

Result: Global limit 9,600 RPS; per-consumer 60 RPS sustained / 180 burst. Projected rejection rate 0.02% — 5× margin on the 0.1% SLO.

📐 SLO-Driven Example: Payment Processing (p99 < 200ms)

Target SLO: p99 latency < 200ms for all payment API calls.

Given: Each payment request takes 40ms average processing time. The payment service runs 8 worker threads. Database connection pool is 20 connections with 150ms average query time.

Step 1 — Throughput ceiling from workers: 8 threads × (1000ms / 40ms) = 200 RPS max throughput at average latency.

Step 2 — Queue depth bound: At p99, you need 99% of requests to complete in < 200ms. Using Little's Law: max queue depth = (200ms − 40ms processing) / 40ms = 4 requests queued. Beyond 4 queued requests, p99 exceeds the SLO.

Step 3 — Derive token bucket parameters: Sustained rate = 200 × 0.80 = 160 RPS. Burst capacity = queue depth bound + workers = 4 + 8 = 12 tokens. This keeps the queue from growing past the latency budget.

Step 4 — DB connection pool check: At 160 RPS with 150ms avg query time, concurrent DB connections = 160 × 0.15 = 24. This exceeds the pool size of 20. Either increase the pool to 25 or reduce the rate limit to 130 RPS (20 / 0.15 = 133, with margin).

Result: Token bucket at 130 RPS sustained / 12 burst. DB pool increased to 25. Load-tested p99 = 172ms at 130 RPS — within the 200ms SLO.

📐 SLO-Driven Example: Real-Time Chat (p95 < 100ms)

Target SLO: p95 message delivery latency < 100ms.

Given: Chat service handles 50,000 connected users. Average message rate is 2 messages/sec/user during active typing. Message fan-out averages 5 recipients per message. WebSocket server capacity is 500,000 messages/sec.

Step 1 — Peak message volume: Assume 10% of users are actively typing at any time: 5,000 users × 2 msg/s = 10,000 inbound msg/s. After fan-out: 10,000 × 5 = 50,000 outbound msg/s.

Step 2 — Per-user rate limit: To prevent spam and keep latency low, cap each user at 5 msg/s (sustained) with 10 msg burst. This allows rapid-fire replies while bounding the total inbound rate.

Step 3 — Global rate limit: At 5 msg/s per active user with 5,000 active users: 25,000 inbound msg/s × 5 fan-out = 125,000 outbound msg/s. This is 25% of server capacity — safe margin. Set global inbound limit to 30,000 msg/s (60% of capacity / fan-out factor).

Step 4 — Latency verification: At 125,000 outbound msg/s on a 500,000 msg/s capacity server, utilization is 25%. Using M/M/1 queue model: avg latency = 1 / (capacity − arrival rate) = 1 / 375,000 = 2.7μs queueing delay. p95 dominated by network RTT (~20-40ms), well within the 100ms SLO.

Result: Per-user 5 msg/s sustained / 10 burst. Global cap 30,000 inbound msg/s. Projected p95 = 40ms — 2.5× margin on the 100ms SLO.

📐 SLO-Driven Example: CDN Origin Protection (Cost SLO)

Target SLO: Origin infrastructure cost < $5,000/month.

Given: Origin server cost is $0.10 per 10,000 requests (compute + bandwidth). CDN cache hit ratio is 92%. Total edge request volume is 500M requests/month. Average origin response is 250KB.

Step 1 — Current origin load: 500M × (1 − 0.92) = 40M origin requests/month. Cost = 40M / 10,000 × $0.10 = $400/month — well within budget.

Step 2 — Budget ceiling: $5,000 / $0.10 × 10,000 = 500M origin requests/month max. That is ~193 origin RPS sustained.

Step 3 — Derive rate limit for origin shield: Set origin rate limit to 150 RPS (78% of 193 RPS ceiling). This caps monthly origin requests at 150 × 86,400 × 30 = 388M requests, costing ~$3,888 — within the $5,000 budget with 22% margin.

Step 4 — Burst for cache misses: Cache miss storms (e.g., after a purge) can spike origin traffic 10×. Set burst to 500 tokens to absorb a 3-second cache-miss storm without dropping requests: 500 tokens ÷ 150 RPS = 3.3s of burst capacity.

Result: Origin shield rate limit at 150 RPS sustained / 500 burst. Monthly cost capped at $3,888, with burst capacity for cache purge events. Alert at 120 RPS (80% of limit) to investigate cache hit ratio drops.

Worked Example

Scenario: Your service handles 10,000 RPS at p99 < 100ms. You have 100 tenants.

  • Global limit: 10,000 * 0.8 = 8,000 RPS
  • Per-tenant limit: 8,000 / 100 = 80, rounded up to 100 RPS (allows some tenants to burst while others are idle)
  • Per-tenant burst capacity: 2 * 100 = 200 tokens (allows a 2-second burst at full rate)
  • Expensive endpoint (/search): 10 RPS per tenant, burst 20
  • Redis memory: ~100 bytes per key * 100 tenants * 5 endpoints = ~50 KB (negligible)

Scaling Journey

0 – 1K users

No Rate Limiting

At low scale, your service can handle all traffic. You might add a simple time.sleep() in a script or a basic Nginx limit_req directive, but rate limiting is not a priority. Your biggest risk is a single misbehaving script, not systemic overload.

1K – 100K users

In-Process Counter

As you gain API consumers, add an in-process rate limiter (e.g., a middleware with an in-memory dictionary of counters per API key). Works if you have a single server or sticky sessions. Breaks when you add a second server — each server enforces independently, so the effective limit doubles.

100K – 10M users

Redis INCR/EXPIRE

Centralize counters in Redis. Use INCR + EXPIRE for fixed windows, or Lua scripts for sliding windows. All application servers query the same Redis, so limits are enforced consistently. Redis handles 100K+ ops/sec on a single node. Migration signal: you notice different servers enforcing different effective limits.

10M+ users

Hierarchical / Edge + Service

At massive scale, use layered rate limiting: (1) edge-level limiting at the CDN/API gateway for DDoS and volumetric attacks, (2) service-level limiting in the application for per-endpoint protection, (3) per-tenant limiting with quotas managed in a control plane. Use Redis Cluster or cell-based Redis for scalability. Consider local token buckets with periodic sync to reduce Redis round-trips.

What Can Go Wrong

1. Rate Limit Too Aggressive — Blocking Legitimate Spikes

What happened: A company set per-user rate limits based on average traffic. During a product launch, legitimate users spiked well above average. The rate limiter rejected the launch traffic, and the company's own marketing campaign resulted in error pages for excited new users.

Root cause: Rate limits were set for steady-state traffic without accounting for legitimate spikes (marketing events, viral moments, batch operations).

Blast radius: All users attempting to access the service during the spike received 429 errors. Support tickets surged.

Mitigation: Emergency increase of rate limits. Added a "burst" allowance to the token bucket configuration.

Prevention: Use adaptive rate limiting that adjusts based on system health rather than static limits. Maintain an override mechanism for planned events. Set burst capacity to at least 3x the sustained rate.

2. Race Conditions in Distributed Counters

What happened: A multi-region API used local Redis instances for rate limiting. During a traffic spike, the counter in us-east-1 showed 90/100, while eu-west-1 showed 85/100 — but the actual aggregate rate was 175 RPS, nearly 2x the intended limit. The downstream database was overwhelmed.

Root cause: Each region maintained its own counter. There was no cross-region synchronization. The rate limit was effectively multiplied by the number of regions.

Blast radius: Database overload caused cascading failures across all regions.

Mitigation: Immediate switch to a global Redis with cross-region replication for rate limiting keys. Accepted the added latency (30-80ms) as a tradeoff for correctness.

Prevention: Use a single authoritative counter (global Redis or a coordination service). Alternatively, divide the global limit across regions statically (e.g., each of 3 regions gets 1/3 of the limit) and accept the inefficiency.

3. Cached 429 Responses Blocking Traffic

What happened: A CDN was configured to cache all API responses by default. When a rate-limited 429 response was generated, the CDN cached it. Subsequent requests from all users hit the cached 429 and were rejected — even though the rate limit had long since reset.

Root cause: The 429 response did not include Cache-Control: no-store headers, and the CDN's default caching policy cached it based on URL pattern.

Blast radius: Complete API outage for the affected endpoint until the cached response expired or was manually purged.

Prevention: Always set Cache-Control: no-store, no-cache on 429 responses. Configure CDN rules to never cache 4xx responses. Add monitoring for unexpected cache hit rates on error responses.

4. Clock Skew Across Nodes

What happened: In a sliding window rate limiter, two application servers had clocks drifting by 2 seconds. Requests near window boundaries were counted in different windows by different servers, creating gaps where requests were allowed past the limit and periods where legitimate requests were incorrectly rejected.

Root cause: NTP synchronization was misconfigured, and the rate limiting algorithm relied on wall-clock time for window boundaries.

Prevention: Use centralized time (Redis server time via TIME command in Lua scripts). Alternatively, use algorithms like GCRA that are less sensitive to clock differences because they track relative intervals rather than absolute timestamps.

Critical Configuration

Config Recommended Value Why It Matters
Redis key TTL Match your window size + small buffer (e.g., 62s for 60s windows) Without TTL, keys accumulate forever and Redis runs out of memory. Too-short TTL causes premature key eviction and under-counting.
Lua script atomicity Always use EVAL/EVALSHA, never separate GET+SET Separate GET and SET have a race condition — two requests can read the same count, both pass, and only one increment is recorded.
X-RateLimit-Remaining Include in every response (not just 429s) Well-behaved clients use this header to self-throttle before hitting the limit. Reduces 429s and improves client experience.
Retry-After Include in every 429 response Tells the client exactly when to retry. Without it, clients guess and either retry too aggressively (making things worse) or too conservatively (hurting their throughput).
Fail-open vs. fail-closed Fail-open for most APIs; fail-closed for security-critical endpoints If Redis is down, fail-open allows all traffic (risking overload), fail-closed rejects all traffic (causing an outage). For most APIs, a brief period without rate limiting is better than a total outage.

Interactions & Pitfalls

Rate Limiting + Caching

Never cache 429 responses. This is the single most dangerous interaction. If a CDN or reverse proxy caches a 429 response, subsequent requests from all clients hit the cached 429 — even though the rate limit has reset. Always set Cache-Control: no-store on 429 responses. Configure your CDN to exclude 4xx/5xx responses from caching. This seems obvious, but it has caused real outages at multiple companies.

Rate Limiting + Load Balancing

Where to enforce: before or after the load balancer? If you enforce after (at the application server), each server sees only a fraction of the client's traffic. A client sending 100 RPS across 10 servers appears to send only 10 RPS to each server. You must either centralize the counter (Redis) or enforce before the load balancer (at the API gateway/edge). The API gateway approach is simpler and more correct, but it means your gateway must handle the full request rate and becomes a scaling bottleneck.

Rate Limiting + API Gateway

Centralized enforcement is the standard pattern. API gateways like Kong, AWS API Gateway, and Envoy have built-in rate limiting. This is where rate limiting naturally belongs — the gateway sees all traffic, can authenticate the client, and can enforce limits before requests reach backend services. The pitfall: if you have multiple API gateways (e.g., one per region), you are back to the distributed enforcement problem. Either share state across gateways or partition limits by region.

Rate Limiting + Auto-Scaling

Rate limiting and auto-scaling are complementary but can conflict. If your auto-scaler adds capacity in response to high traffic, but the rate limiter is rejecting that traffic, the auto-scaler may not see enough load to scale up — creating a deadlock where the system is under-provisioned but traffic is being rejected. Solution: scale based on incoming request rate (including rejected requests), not just processed requests.

Cost Considerations

Cost Analysis

  • Redis memory for counters: Tiny. A sliding window counter requires ~100 bytes per key. Even with 1M unique clients and 10 endpoints each, that is only ~1 GB. A small Redis instance (e.g., AWS ElastiCache cache.r6g.large, ~$200/month) handles this easily with room to spare.
  • Compute overhead: Negligible. A Redis round-trip adds 0.1-0.5ms to each request. The Lua script execution is sub-microsecond. At 10K RPS, rate limiting consumes less than 1% of a single CPU core.
  • Network: Each rate limit check is a small Redis request (~100 bytes) and response (~50 bytes). At 10K RPS, that is ~1.5 MB/s — well within any network capacity.
  • Dominant cost driver: The Redis instance, not the rate limiting logic itself. If you are already running Redis for caching or sessions, the marginal cost of adding rate limiting is essentially zero.

Cost of NOT Rate Limiting

  • DDoS costs: Without rate limiting, a volumetric attack scales your cloud bill linearly. A sustained 100K RPS attack on a serverless function at $0.20 per million invocations costs $1,200/hour.
  • Cascade failures: A single overloaded service can cascade to the entire system. The blast radius of an unprotected API is your entire infrastructure.
  • Customer impact: Without per-tenant limits, one noisy tenant can degrade the experience for all tenants — causing churn that dwarfs any infrastructure cost.

Real-World Examples

Stripe API Rate Limiting

Stripe enforces rate limits at 100 requests per second per API key in live mode and 25 per second in test mode. They use a token bucket algorithm and return X-RateLimit-Limit, X-RateLimit-Remaining, and Retry-After headers on every response. Stripe's approach is notable for its developer experience: clear documentation, generous burst allowance, and client libraries that automatically implement exponential backoff with jitter on 429 responses. They also offer the ability to request higher limits for specific use cases.

GitHub API Limits

GitHub's REST API allows 5,000 requests per hour for authenticated users and 60 per hour for unauthenticated requests. Their GraphQL API has a point-based system — each query has a cost based on its complexity, and users get 5,000 points per hour. This is a sophisticated form of variable-cost rate limiting: a simple query might cost 1 point, while a query that fetches 100 repositories with their issues and pull requests might cost 50 points. This prevents clients from using cheap requests to mask expensive operations.

Cloudflare Rate Limiting at the Edge

Cloudflare's rate limiting operates at the edge — in data centers distributed across 300+ cities worldwide. This means rate-limited requests never reach the origin server, saving both bandwidth and compute. Cloudflare uses a combination of fixed window and sliding window algorithms, with the ability to define rules based on URL patterns, HTTP methods, headers, and client IP. At their scale, they process millions of rate limiting decisions per second globally, using a distributed counting system that synchronizes across edge locations with eventual consistency — accepting that the effective limit may briefly exceed the configured limit by a small margin during synchronization delays.

Code Example — Token Bucket

A clean token bucket implementation in Python. The key insight is lazy refill: instead of a background timer adding tokens, we compute how many tokens should have been added since the last request.

import time

class TokenBucket:
    def __init__(self, rate, capacity):
        self.rate = rate          # tokens per second
        self.capacity = capacity  # max tokens
        self.tokens = capacity    # start full
        self.last_refill = time.monotonic()

    def _refill(self):
        now = time.monotonic()
        elapsed = now - self.last_refill
        new_tokens = elapsed * self.rate
        self.tokens = min(self.capacity, self.tokens + new_tokens)
        self.last_refill = now

    def allow(self, tokens=1):
        self._refill()
        if self.tokens >= tokens:
            self.tokens -= tokens
            return True
        return False

    def retry_after(self):
        """Seconds until a token is available."""
        if self.tokens >= 1:
            return 0.0
        deficit = 1 - self.tokens
        return deficit / self.rate

# Usage
limiter = TokenBucket(rate=10, capacity=20)  # 10 req/s, burst of 20

for i in range(25):
    if limiter.allow():
        print(f"Request {i}: allowed")
    else:
        wait = limiter.retry_after()
        print(f"Request {i}: rejected (retry after {wait:.2f}s)")

How to Validate Your Design

Load Testing

Use k6 to simulate clients at varying request rates. Start below the rate limit and gradually increase. Verify that:

  • Requests below the limit always succeed (0% rejection rate).
  • Requests above the limit are rejected with 429 (not 500 or timeout).
  • The actual rejection threshold matches the configured limit within 5%.
  • X-RateLimit-Remaining headers decrement correctly.
  • Retry-After values are accurate — retrying after the specified time succeeds.

Burst Behavior

Test burst handling explicitly: let a client idle for 10 seconds, then send requests as fast as possible. The first N requests (up to burst capacity) should succeed. Requests beyond burst capacity should be rejected. Repeat to verify tokens refill correctly.

Distributed Consistency

Run multiple k6 instances simultaneously against the same API key, through different application servers. The total accepted requests should not significantly exceed the configured limit. A margin of 5-10% is acceptable for distributed systems; more than that indicates a synchronization problem.

Chaos Testing

  • Kill Redis: Verify the system fails open (or closed, per your policy). Measure the impact on latency and error rates.
  • Add Redis latency (toxiproxy): Inject 50ms of latency to Redis. Verify the application still responds within its SLO. If not, consider local caching of rate limit decisions.
  • Clock skew (Chaos Monkey): Skew the system clock on one application server by 5 seconds. Verify the rate limiter still enforces limits correctly (it should, if you are using Redis server time).

Interview Questions

Design a rate limiter for a multi-tenant API

Start by clarifying requirements: how many tenants, what is the SLO, and whether tenants have different tiers (free vs. paid). The core design uses a centralized counter in Redis, keyed by tenant_id:endpoint:window. Use a sliding window counter algorithm implemented in a Lua script for atomicity.

The architecture has three layers: (1) An API gateway that authenticates the request and extracts the tenant ID. (2) A rate limiter middleware that checks Redis. (3) The application server. For different tiers, store the rate limit configuration per tenant in a database, and cache it in the gateway. Use X-RateLimit-Limit, X-RateLimit-Remaining, and Retry-After headers in every response.

The key tradeoff is where to enforce: at the gateway (simpler, sees all traffic) vs. at each service (finer-grained, per-endpoint limits). In practice, use both: coarse limits at the gateway, fine-grained limits at the service level.

Follow-up: How do you handle it when Redis is unavailable? (Fail-open vs. fail-closed.) How do you handle rate limiting across multiple regions?

Token bucket vs. sliding window — what are the tradeoffs?

Token bucket allows bursts (up to the bucket capacity), requires storing two values (token count and last refill time), and is simple to implement. It is ideal when you want to allow some burstiness — a mobile app that syncs on open, a batch job that sends 50 requests then idles.

Sliding window counter provides smoother enforcement (no burst spikes), requires storing two counters (current and previous window), and has a slight approximation at window boundaries. It is better when you need a steady, predictable request rate — protecting a database that does not tolerate bursts.

In distributed systems, the sliding window counter is easier to implement correctly in Redis (two INCR operations with TTL) than the token bucket (which requires a compare-and-swap on the token count). GCRA is the best of both worlds — single-value storage, correct distributed behavior, and configurable burstiness.

Follow-up: What about the fixed window boundary problem? How does the sliding window counter solve it? What is the approximation error?

How would you implement distributed rate limiting?

The standard approach is Redis with Lua scripts. The Lua script atomically checks the current count, increments it, and sets the TTL — all in a single Redis command with no race conditions. For a sliding window counter, the script accesses two keys (current and previous window), computes the weighted count, and returns the decision.

The main challenge is Redis availability. Options: (1) Redis Sentinel for automatic failover (adds ~1s of unavailability during failover). (2) Redis Cluster for horizontal scaling (shards keys by client ID — no cross-shard operations needed). (3) Local fallback — if Redis is unavailable, use an in-memory token bucket with a conservative limit until Redis recovers.

For global deployments across regions, you have two options: (a) Global Redis with cross-region replication (correct but adds latency), or (b) partition the limit across regions (e.g., 3 regions each get 1/3 of the limit) and accept some inefficiency when traffic is unevenly distributed.

Follow-up: How do you handle the case where a Lua script takes too long? (Redis is single-threaded — a long script blocks all other operations.) What is your Redis sizing?

How do you handle rate limiting for WebSocket connections?

WebSocket rate limiting is fundamentally different from HTTP rate limiting because a WebSocket connection is long-lived. You need to limit at two levels: (1) connection rate — how many new connections a client can open per time window, and (2) message rate — how many messages per second a client can send on an established connection.

For connection rate limiting, use the same approach as HTTP — a token bucket or sliding window at the connection handshake phase. For message rate limiting, use an in-process token bucket per connection (since messages flow through a specific server). If the client exceeds the message rate, you can either drop messages, send a rate limit notification message over the WebSocket, or close the connection.

The key design decision is whether to enforce server-side per connection (simple but requires coordination if clients reconnect to different servers) or per client ID across all connections (requires Redis, but correct if a client opens multiple connections).

Follow-up: How do you handle broadcast messages vs. client-to-server messages? Should they have different limits?

Design rate limiting that is fair across tenants of different sizes

Static per-tenant limits are inherently unfair: giving every tenant 100 RPS penalizes large tenants who generate more revenue and underutilizes capacity when small tenants are idle. The solution is a hierarchical token bucket with weighted fair queuing.

Design: (1) Assign each tenant a weight based on their tier (e.g., Enterprise = 10, Pro = 3, Free = 1). (2) Allocate a global token pool (e.g., 8,000 tokens/sec). (3) Each tenant's share is proportional to their weight relative to active tenants. If only one Enterprise tenant and one Free tenant are active, they get 10/11 and 1/11 of the capacity. (4) Unused capacity is redistributed — if the Enterprise tenant is idle, the Free tenant can use more than their base allocation.

Implementation: use Redis sorted sets to track active tenants and their weights. Compute each tenant's effective limit dynamically (e.g., every 10 seconds). Update the per-tenant token bucket rates. This is the approach used by cloud providers like AWS for API Gateway throttling with usage plans.

Follow-up: How do you handle the cold-start problem — a new tenant joins and suddenly all existing tenants' limits decrease? How do you prevent a malicious tenant from inflating their weight?

A service is returning 429s but Redis shows the client is under the limit. How do you debug this?

This is a classic distributed systems debugging problem. Work through the possible causes systematically:

1. Check which layer is rejecting. Is it the API gateway, the application, or a downstream service? Check logs and response headers at each layer. The 429 might not be from your rate limiter at all — a third-party API or database connection pool might be returning 429.

2. Check key mismatch. The rate limiter might be using a different key than what you are checking in Redis. Common mistakes: the client IP changed (NAT, proxy), the API key was regenerated, or the rate limiter is keying on a different identifier (e.g., user ID vs. API key).

3. Check clock skew. If the rate limiter uses wall-clock time for window boundaries and the application server clock is ahead of Redis, requests may be counted in a different window than expected.

4. Check for cached 429s. A CDN or reverse proxy might be serving a cached 429 response. Check the Age and X-Cache headers.

Follow-up: How would you add observability to prevent this from happening again? What rate limit headers would help the client self-diagnose?

Key Takeaways

  • Rate limiting is a safety mechanism, not a performance optimization. It protects your system from overload, abuse, and unfairness. Every public API should have it.
  • The token bucket algorithm is the most common approach. It is simple, allows bursts, and has well-understood behavior. GCRA is a modern equivalent that is easier to implement in distributed systems.
  • In distributed systems, centralize the counter — typically in Redis with Lua scripts for atomicity. Local counters lead to the limit being multiplied by the number of nodes.
  • Set limits based on measured capacity, not guesses. Load test to find your p99 capacity, set the limit at 80%, and size burst capacity at 2-3x the per-second rate.
  • Never cache 429 responses. Always set Cache-Control: no-store on rate-limited responses.
  • Include rate limit headers in every response — X-RateLimit-Remaining, X-RateLimit-Limit, and Retry-After on 429s. Well-behaved clients use these to self-throttle.
  • Decide your fail-open/fail-closed policy before Redis goes down. For most APIs, fail-open is better than a total outage.
  • Use layered rate limiting at scale: edge (DDoS), gateway (per-tenant), service (per-endpoint). Each layer protects against different failure modes.
  • Adaptive rate limiting (adjusting limits based on system health) is the modern best practice, but static limits are a perfectly good starting point.
