Caching

Store frequently accessed data closer to the consumer so you can serve it faster and reduce load on your origin data store.

25 min read

What Is It?

A cache is a temporary storage layer that sits between the consumer of data and the authoritative source of that data. Its entire purpose is speed: by keeping a copy of recently or frequently requested data in a faster medium (RAM instead of disk, local memory instead of a network call), a cache lets you serve responses in microseconds instead of milliseconds, or milliseconds instead of seconds.

Caches appear at every layer of a modern system. Your CPU has L1/L2/L3 caches. Your browser has a local cache for static assets. Your CDN is a geographically distributed cache. Your application server might hold an in-process cache in a hash map, and you probably also have a dedicated caching service like Redis or Memcached sitting between your app and your database. Each layer trades a small amount of memory and consistency for a large reduction in latency and backend load.

Caching patterns split into two orthogonal axes: how reads are served, and how writes propagate. A real system picks one from each side (e.g., "cache-aside reads + write-around writes" is a common combo).

Read patterns:

  • Cache-aside (lazy loading) — The application checks the cache first. On a miss, it fetches from the database, writes the result into the cache, and returns it. The cache never talks directly to the database. This is the default in most production systems.
  • Read-through — The cache itself is responsible for fetching from the database on a miss. The application only ever talks to the cache. Simpler app code, but requires a cache that knows how to load from your data source (e.g., a library or a cache-as-a-service like DAX).
  • Refresh-ahead — The cache proactively reloads keys before they expire, based on a configurable threshold (e.g., refresh when 80% of the TTL has passed). Avoids miss latency for predictably hot keys, at the cost of refreshing keys that may never be read again.

Write patterns:

  • Write-through — Every write goes to both the cache and the database synchronously. Guarantees cache–DB consistency, but write latency = max(cache, DB).
  • Write-behind (write-back) — Writes go to the cache immediately and the cache asynchronously flushes to the database in the background, often coalescing many writes into one. Fastest writes, but the cache crashing before flush means lost data.
  • Write-around — Writes go directly to the database, skipping the cache. The cache is only populated by subsequent reads (then it caches normally and serves hits like any other entry). The win comes from not writing data into the cache that no one is about to ask for — useful when writes vastly outnumber reads or when reads usually arrive long after the write (logs, audit records, large blobs that may never be fetched).
  • Write-invalidate — A common variant pairing any write strategy with cache eviction: on write, DEL the affected key(s) from cache. The next read repopulates from the DB. This is what "cache-aside writes" almost always means in practice.

How It Works

The most common pattern in practice is cache-aside. Here is the step-by-step flow:

  1. Application receives a request for data identified by a key (e.g., user:12345).
  2. Application checks the cache using the key. This is a fast O(1) lookup in Redis or an in-process hash map.
  3. Cache hit: If the key exists and has not expired, the cached value is returned immediately. Total latency: sub-millisecond for local cache, 1-2 ms for Redis over the network.
  4. Cache miss: If the key is not found (or has expired), the application queries the primary data store (e.g., PostgreSQL).
  5. Database responds with the data. Total latency for this step: 5-50 ms depending on query complexity and database load.
  6. Application writes the result into the cache with a TTL (time-to-live). For example, SET user:12345 "{...}" EX 300 stores the data for 5 minutes.
  7. Application returns the data to the caller. Subsequent requests for the same key within the TTL window will be served from cache (step 3).

The key insight is that step 6 is what "warms" the cache. The first request after a TTL expiry or a cold start pays the full database cost, but every subsequent request benefits from the cache. If 95% of your requests are cache hits, you have effectively reduced your database load by 95%.

When to Use / When NOT to Use

Use Caching When

  • Read-heavy workloads: Your read-to-write ratio is 10:1 or higher. Classic examples: user profiles, product catalogs, news feeds.
  • Expensive computations: Aggregations, recommendations, or complex joins that take hundreds of milliseconds. Cache the result instead of recomputing.
  • Latency-sensitive paths: Your SLA requires p99 < 50 ms, but your database p99 is 200 ms. A cache brings it under 5 ms.
  • Repeated access patterns: A small subset of data (hot set) receives the majority of requests. Zipfian distributions are common in practice.
  • Protecting a fragile backend: Your database is at capacity. Caching offloads read traffic and prevents cascading failures.

Don't Use Caching When

  • Data changes every request: If the data is different for every read (e.g., real-time stock prices in a trading system), caching adds complexity with no benefit.
  • Strong consistency is required for every read: Financial account balances, inventory counts during checkout — stale data can cause real money loss.
  • Write-heavy with no read benefit: If you write 100x more than you read, the cache is constantly invalidated before anyone reads it.
  • Data is too large to fit in memory: If your working set is 500 GB and your cache budget is 16 GB, the hit rate will be too low to justify the cost.
  • Debugging is a priority over performance: Caches add a layer of indirection. In early-stage systems, simplicity may matter more than speed.

Classic vs. Modern Approaches

LRU (Least Recently Used) — The Classic

LRU evicts the item that was accessed least recently. It works well when access patterns have strong temporal locality — if you used something recently, you are likely to use it again soon. Most introductory caching discussions focus on LRU, and Redis defaults to an approximated LRU policy (allkeys-lru).

Why O(1) for Both Get and Put

A naive approach might store cached items in an array sorted by access time. Getting an item is O(n) (scan the array), and every access requires moving the item to the front — also O(n). This is far too slow for a cache handling millions of requests per second.

The trick is combining two data structures: a hash map for O(1) key lookup, and a doubly linked list for O(1) reordering. The hash map stores key → pointer to the list node. The linked list maintains items in access order: the most recently used item is at the head, and the least recently used is at the tail.

  • get(key): Look up the key in the hash map (O(1)). If found, unlink the node from its current position in the list and move it to the head (O(1) pointer surgery — no traversal needed because we have a direct pointer to the node). Return the value.
  • put(key, value): If the key exists, update its value and move it to the head. If the key is new and the cache is full, remove the node at the tail (that is the least recently used item), delete it from the hash map, then insert the new node at the head and add it to the hash map. Every step is O(1).

Limitation — scan pollution: LRU is blind to frequency. A one-time full scan of your data (e.g., a batch job or analytics query) can evict the entire hot working set, causing a burst of cache misses. This is called cache pollution. For example, if a batch job reads 10,000 keys sequentially, every one of those keys enters the cache and pushes out a hot key — even though those batch keys will never be accessed again.

LFU (Least Frequently Used)

Where LRU tracks when an item was last accessed, LFU tracks how many times each item has been accessed and evicts the item with the lowest count. This makes LFU resistant to scan pollution: a batch job that touches 10,000 keys once each won't evict a hot key that has been accessed 500 times.

The Naive Approach (and Why It's Slow)

The simplest LFU implementation stores a frequency counter with each item and, on eviction, scans all items to find the one with the lowest count. This is O(n) per eviction — unacceptable for a high-throughput cache.

You could use a min-heap keyed by frequency to get O(log n) eviction. But every get() call must increment the frequency and re-heapify, which is also O(log n). For a cache serving millions of requests per second, even O(log n) per access is costly.

The O(1) LFU Trick: Frequency Buckets

The key insight (from the paper by Shah, Mitra, and Matani, 2010) is to group items by their frequency into buckets. Each bucket is a doubly linked list of all items with the same frequency. The buckets themselves are linked in order of frequency. We also keep a pointer to the minimum frequency bucket.

The data structures:

  • Hash map: key → (value, frequency, pointer to node in bucket list)
  • Frequency buckets: a linked list of buckets, where bucket f contains a doubly linked list of all items with frequency f
  • min_freq: an integer tracking the current minimum frequency

Operations:

  • get(key): Look up key in hash map (O(1)). Remove the node from its current frequency bucket f. Insert it into bucket f+1 (create the bucket if needed). If bucket f is now empty and f == min_freq, increment min_freq. All O(1).
  • put(key, value): If full, go to the min_freq bucket. Remove the item at the tail of that bucket's list (least recently used among the least frequently used). Insert the new item into bucket 1. Set min_freq = 1. All O(1).

Limitation: LFU has a "frequency inertia" problem. An item that was heavily accessed in the past accumulates a high count, making it nearly impossible to evict — even if it's no longer popular. Some implementations decay frequencies over time (halving all counts periodically) to address this.

ARC (Adaptive Replacement Cache)

The problem with choosing between LRU and LFU is that neither works best for all workloads. LRU handles temporal locality well but suffers from scan pollution. LFU handles frequency well but can't adapt when popular items change. What if the cache could automatically figure out which strategy is working better and lean into it?

That's exactly what ARC does. Introduced by Megiddo and Modha (IBM, 2003), ARC maintains four lists with a total budget of c items (the cache capacity):

  • T1 (recency): LRU list of items that have been accessed exactly once recently. These are "newcomers."
  • T2 (frequency): LRU list of items that have been accessed at least twice recently. These are "proven valuable."
  • B1 (ghost list for T1): Stores only the keys (not values) of items recently evicted from T1. This is ARC's memory of "recency evictions."
  • B2 (ghost list for T2): Stores only the keys of items recently evicted from T2. This is ARC's memory of "frequency evictions."

Only T1 and T2 hold actual data. Together they hold exactly c items: |T1| + |T2| = c. B1 and B2 are metadata-only and cost almost nothing.

The Adaptive Parameter p

ARC maintains a parameter p that controls the target size of T1. The target size of T2 is c - p. Initially, p = 0 (give all space to frequency). The key: p adapts based on which ghost list gets hit:

  • Hit in B1 (we evicted a recency item and now it's needed again): This means the recency list was too small. Increase p — give more space to T1. Specifically, p = min(c, p + max(1, |B2|/|B1|)).
  • Hit in B2 (we evicted a frequency item and now it's needed again): This means the frequency list was too small. Decrease p — give more space to T2. Specifically, p = max(0, p - max(1, |B1|/|B2|)).

The adaptation rate is proportional to the relative sizes of the ghost lists. If B2 is much larger than B1, a hit in B1 causes a big increase in p (aggressive shift toward recency). This self-tuning is what makes ARC so powerful.

A pure LRU cache would have lost all of A-D to the scan and kept the useless E, F, G.

Used in: ZFS file system, IBM storage controllers. Not common in web caching due to patent concerns (now expired).

W-TinyLFU (Window Tiny Least Frequently Used)

W-TinyLFU is the algorithm behind Caffeine, the dominant in-process caching library in the Java ecosystem. It combines the best of LRU and LFU while using minimal memory. To understand it, we first need to understand the data structure at its core.

Count-Min Sketch: Approximate Frequency Counting

The problem: we need to know how often each item has been accessed, but storing an exact counter for every item ever seen would use unbounded memory. If you've seen 100 million distinct keys, that's 100 million counters.

A Count-Min Sketch is a probabilistic data structure that estimates frequencies using fixed memory, regardless of how many distinct items you see. It trades perfect accuracy for bounded space.

What it looks like in memory: A 2D array of counters with d rows and w columns, plus d independent hash functions (one per row). In Caffeine's implementation, d = 4 rows and w is sized to the cache capacity. Each counter is just 4 bits (values 0-15), so for a cache of 10,000 items the sketch uses only 10,000 × 4 × 4 bits = 20 KB.

How insertion works: To record an access to item X, hash X with each of the d hash functions to get d column indices. Increment the counter at each of those d positions.

How query works: To estimate the frequency of item X, hash X with the same d hash functions. Read the counter at each position. Return the minimum of those d values. (This is the "Min" in Count-Min — the minimum is the least over-counted estimate.)

How W-TinyLFU Works

The cache is split into two regions:

  • Window cache (1% of total): A small LRU region where all new items enter first. This gives brand-new items a chance to prove themselves without immediately competing against established items.
  • Main cache (99% of total): A segmented LRU (SLRU) divided into a "probation" segment (20%) and a "protected" segment (80%). Items in probation that are accessed again get promoted to protected.

The admission filter: When an item is evicted from the window cache, it doesn't automatically enter the main cache. Instead, W-TinyLFU compares the incoming item's estimated frequency (from the Count-Min Sketch) against the main cache's eviction victim (the LRU item in the probation segment). If the incoming item's frequency is higher, it replaces the victim. If lower, the incoming item is rejected and discarded.

This admission filter is the key insight: instead of always admitting new items and hoping the eviction policy sorts things out, W-TinyLFU only admits items that have earned entry through demonstrated frequency. A one-time scan of 10,000 keys will enter the tiny window, but when evicted from the window, each key's frequency (1) will likely be lower than any established item in the main cache. The scan items are rejected at the gate — main cache is unaffected.

Periodic reset: To prevent stale frequency counts from keeping old items forever, the Count-Min Sketch halves all counters periodically (when the total count reaches a threshold). This acts as a decay mechanism, letting the sketch adapt to changing access patterns.

Performance: Caffeine (W-TinyLFU) consistently achieves near-optimal hit rates across diverse workloads, often 5-10% higher than LRU. It is the default caching library for Spring Boot, Apache Druid, and many other Java-based systems.

CLOCK (Second-Chance Algorithm)

LRU gives great eviction decisions, but it has a hidden cost: every single access requires updating the linked list (moving the accessed node to the head). Under high concurrency, this means every read must acquire a lock on the list, serializing access and creating contention. CLOCK is an approximation of LRU that avoids this overhead.

The Circular Buffer

A circular buffer (or ring buffer) is an array that wraps around: when you reach the end, you continue from the beginning. Imagine the cache slots arranged in a circle, with a "clock hand" pointer that sweeps around the circle. Unlike a regular array, there is no "beginning" or "end" — only the current position of the hand.

Each slot in the circular buffer holds: the cached item, and a reference bit (a single bit: 0 or 1). The reference bit is set to 1 whenever the item is accessed. That's the only bookkeeping on access — no linked list to update, no lock contention.

How CLOCK Eviction Works (Second-Chance)

When the cache is full and we need to evict, the clock hand sweeps around the buffer:

  1. Look at the item under the hand. Check its reference bit.
  2. If the reference bit is 1 (item was accessed recently): clear it to 0. The item gets a "second chance." Advance the hand to the next slot.
  3. If the reference bit is 0 (item was NOT accessed since the last sweep): evict this item. Insert the new item in this slot with reference bit = 1. Stop.
  4. If the hand completes a full revolution without finding a 0-bit, it means every item was accessed recently. On this second pass, all bits have been cleared to 0, so the hand will evict the next item it encounters.

On the second pass (6b), slot 0 (A) now has R=0, so A is evicted and replaced by E. Notice that B was accessed in step 5 (R=1) but still got its bit cleared — if nothing accesses B before the hand reaches it again, B will be evicted next.

Why CLOCK approximates LRU: Items that are accessed frequently keep getting their reference bit set back to 1, so the hand skips over them. Items that haven't been accessed are left with R=0 and get evicted. The net effect is similar to LRU (recently used items survive) but without the per-access linked list overhead. CLOCK's per-access cost is O(1) — just set a bit — with no lock contention.

CLOCK-Pro

Basic CLOCK only considers recency (was the item accessed since the last sweep?). CLOCK-Pro (Jiang, Chen, Zhang, 2005) extends CLOCK to also consider reuse distance — the number of distinct items accessed between two consecutive accesses to the same item. An item with a short reuse distance (accessed frequently) is more valuable than one with a long reuse distance.

CLOCK-Pro classifies pages as hot (short reuse distance, likely to be accessed again soon) or cold (long reuse distance). It uses three clock hands sweeping the same circular buffer:

  • Hand-hot: Scans for hot pages that have become cold (their reference bit was 0 for too long). Demotes them to cold.
  • Hand-cold: Scans for cold pages to evict. Cold pages with R=0 are evicted. Cold pages with R=1 are promoted to hot (they've proven they have short reuse distance).
  • Hand-test: Tracks "test" entries (metadata for recently evicted cold pages, similar to ARC's ghost lists). If a test entry is accessed, the corresponding page is re-admitted as hot.

The result is an eviction policy that adapts to both recency and frequency patterns, using only a circular buffer and a few bits per entry — no linked lists, no hash maps of frequency counters. This makes CLOCK-Pro attractive for systems where per-operation overhead matters (e.g., OS page caches).

Used in: Linux kernel (a variant), NetBSD. Attractive when you need a high-performance eviction policy with low per-operation overhead.

Cache Invalidation Strategies

Caching is easy. Invalidation — knowing when to remove or update stale data — is the hard part. Phil Karlton famously said there are only two hard things in computer science: cache invalidation and naming things. Here are the main strategies, each with a different tradeoff between consistency, latency, and complexity.

Cache-Aside (Lazy Loading)

The application manages the cache explicitly. On read: check cache → if miss, read from DB → fill cache. On write: update DB → invalidate (delete) cache entry. The cache is never written to directly on writes — it's only filled on reads.

Why delete instead of update? Deleting is simpler and avoids race conditions. If two concurrent writes try to update the cache, the cache might end up with the wrong value (the slower write overwrites the faster one). Deletion ensures the next read fetches fresh data from the DB.

Aren't the DB write and cache DEL atomic? No — and this is the source of cache-aside's eventual-consistency window. They are two independent operations against two independent systems (your DB and your cache, e.g. Postgres and Redis), with no shared transaction. The app code looks like:

db.execute("UPDATE users SET age=31 WHERE id=42")  # step 1
cache.delete("user:42")                             # step 2

Between step 1 and step 2 — even if just for a millisecond — another request can hit the cache and read the old value. Worse, several things can break the pair entirely:

  • App crashes between the two calls → DB has new value, cache still has old. The stale entry persists until its TTL expires.
  • Cache is down when the DEL fires → same outcome. The DB write must not fail just because the cache is unreachable (that would defeat the resilience benefit), so the app typically logs and moves on.
  • Concurrent reader interleaves: reader misses cache → reads old value from a replica → writer commits new value → writer deletes cache → reader fills cache with the old value it already fetched. Now the cache holds stale data with no pending invalidation.

Because there's no cross-system transaction, production systems mitigate (not eliminate) this with: a TTL safety net so stale entries self-heal, delete-after-commit ordering, or a CDC/outbox pattern that drains DB changes into cache invalidations reliably. If you need true atomicity, you've left cache-aside territory — that's what write-through is for.

Pros: Simple to implement — the application controls all cache logic, no special cache infrastructure needed. Resilient to cache failures (a dead cache just means slower reads, not broken writes). Only data that is actually read gets cached, so memory is used efficiently.

Cons: The first read after a write or eviction is always a cache miss (higher tail latency). Cache logic is duplicated across every read path in the app, which can drift over time. Vulnerable to the thundering herd problem — if a hot key expires, many concurrent requests can hit the DB simultaneously. Brief inconsistency window between DB write and cache delete if they aren't atomic.

Mitigating the thundering herd (see What Can Go Wrong for full implementations):

  • Lease / single-flight lock — on miss, the first request acquires a short-lived lock (Redis SET NX EX) and is the only one to query the DB. Others wait and retry the cache. Reduces N concurrent DB queries to 1.
  • Request coalescing in-process — within a single app server, a singleflight-style helper deduplicates concurrent calls for the same key so only one goroutine/thread hits the DB; the rest await its result. Eliminates intra-host duplication even without a distributed lock.
  • Probabilistic / early recomputation (XFetch) — just before the TTL expires, one request is statistically chosen to refresh the entry while others still serve the cached value. Avoids the synchronized-cliff at TTL boundary.
  • TTL jitter — instead of a fixed TTL like 300s, use 300s ± random(30s). Spreads expirations across time so 200 servers don't all miss at the same instant.
  • Stale-while-revalidate — serve the stale value past TTL while a single background task refreshes it. Readers never block on a miss for a hot key.
  • Pre-warm / refresh-ahead — a background job repopulates known-hot keys before they expire. Best for a small set of well-known hot keys (homepage, trending list).
  • Negative caching — if the DB returns "not found," cache that fact briefly so a flood of misses for a non-existent key doesn't slam the DB either.

Best for: Read-heavy workloads with tolerable staleness (user profiles, product catalogs, social feeds). The default choice when in doubt — it's what most production systems use.

Write-Through

Every write goes to both the cache and the database, synchronously. The write is only considered successful when both are complete.

Request flow:

Pros: Cache is always consistent with the database. Reads never get stale data. Simple mental model.

Cons: Every write pays the latency of both cache + DB (higher write latency). If the DB is slow, writes are slow. Also, you may cache data that is rarely read (wasteful memory usage).

Best for: Systems where read-after-write consistency is critical and write volume is moderate (e.g., user account settings).

Write-Behind (Write-Back)

Writes go to the cache immediately. The cache asynchronously flushes changes to the database in the background, often batching multiple writes together.

Request flow:

Pros: Extremely fast writes (cache-speed). Can coalesce multiple writes to the same key (e.g., 100 increments to a counter become a single +100 write to DB). Reduces DB write load.

Cons: Risk of data loss if the cache crashes before flushing to DB. Reads from the DB directly will see stale data until the flush completes.

Best for: Write-heavy workloads where some data loss is acceptable (analytics counters, gaming leaderboards, activity feeds).

Write-Around

Writes go only to the database, bypassing the cache entirely. The cache is only populated on reads (like cache-aside).

"May or may not be invalidated" — what does that mean? Write-around is a family of patterns, not one fixed rule. Some implementations explicitly delete the cache entry on write (same as cache-aside on writes); others skip the cache entirely and let the stale entry sit until its TTL expires. The "pure" form does no invalidation — and yes, that means readers can see stale data for up to one TTL after a write. Teams accept this when staleness is cheap and write throughput is precious.

Isn't write-around-with-invalidation just cache-aside? The read path is identical (check cache → miss → load from DB → fill). The distinction is in intent and emphasis on the write path. Cache-aside requires a DEL after every DB write — it's part of the contract, and forgetting it is a bug that causes stale reads. Write-around treats invalidation as optional and is happy to leave the stale entry in place, relying on TTL to clean up. So if you always invalidate, you've effectively implemented cache-aside; the "write-around" label is most meaningful in its pure (no-invalidation) form, where you're explicitly choosing to bypass cache logic on writes for speed.

Why are writes fast? Compared to write-through, the write path touches only the DB — no synchronous cache update, no extra network hop, no waiting on two systems. Compared to cache-aside, you also skip the DEL round-trip to the cache. The write latency is exactly one DB write, nothing more.

Why is it suited to write-heavy, rarely-re-read data? The cache only earns its keep when an entry is read multiple times before it changes. For data like log entries or audit records — written constantly, almost never read — populating the cache on every write would burn memory and bandwidth on entries no one will ever request. Write-around says: don't speculate. Let reads pull data into the cache only when someone actually wants it. If the same key is read soon after a write, you eat one cache miss; that's a fine trade when the read-to-write ratio is low.

Request flow:

Pros: Fast writes (only DB). Cache doesn't get polluted with data that may never be read.

Cons: If the written data is immediately read, it's a cache miss (higher read latency). Stale data risk if TTL is long.

Best for: Write-heavy data that is rarely re-read immediately (e.g., log entries, audit trails).

Assumptions & Model

Key Assumptions

  • Temporal locality: Data accessed recently is likely to be accessed again soon. This is the foundational assumption of all caching. If your access pattern is uniformly random with no repetition, caching provides zero benefit.
  • Zipfian access distribution: A small fraction of keys receive a disproportionate share of traffic. In practice, web workloads often follow a Zipfian distribution where the top 10-20% of keys account for 80%+ of reads.
  • Cache is faster than origin: The cache medium (RAM, SSD, local process memory) must be meaningfully faster than the origin (database, remote API, disk). If your "cache" is on the same disk as your database, the benefit disappears.
  • Data is tolerant of staleness: Cached data may be slightly out of date. The system must tolerate serving data that is up to TTL seconds old. If every read must reflect the latest write, caching is counterproductive.

When These Assumptions Break

  • Uniform random access: If every key is equally likely to be requested (e.g., a hash-based lookup across a very large keyspace with no hot keys), cache hit rates will be proportional to (cache size / total keys) and will be very low.
  • Write-heavy workloads: If data changes faster than it is read, TTLs expire or invalidations fire before anyone reads the cached value. The cache churns without providing value.
  • Adversarial access patterns: An attacker deliberately requesting uncacheable keys (unique, random IDs) will bypass the cache entirely and hit the origin directly. This is a cache-busting attack.

Sizing & Tradeoffs

Working Set Analysis

The first step in cache sizing is identifying your working set — the subset of data that accounts for most of the read traffic. If your database holds 10 million records but 80% of reads hit the top 50,000 records, your working set is 50K records. Cache those 50K, and you capture 80% of reads.

Hit-Rate Curves

The relationship between cache size and hit rate follows a curve of diminishing returns. Going from no cache to caching 1% of your data might give you a 50% hit rate (if traffic is highly skewed). Doubling to 2% might give you 65%. Doubling again to 4% gives 75%. Eventually, each doubling yields only 1-2% more hits. Plot your own hit-rate curve by replaying production access logs against different cache sizes.

The Square-Root Rule

A useful rule of thumb for cache sizing: if the total number of unique items is N, a cache of size sqrt(N) is often a reasonable starting point for Zipfian workloads. For N = 10,000,000 items, this gives sqrt(10M) = ~3,162 items. In practice, you will likely need more than this, but it gives a lower bound.

SLO-Driven Cache Sizing

Cache size is not a guess — it's derived from your latency SLO and traffic profile. The framework:

  1. Define the SLO: "p99 read latency < X ms."
  2. Measure uncached latency: What's the DB p99? (e.g., 50ms)
  3. Measure cached latency: What's the cache p99? (e.g., 2ms for Redis)
  4. Compute required hit rate: To meet the SLO, what fraction of requests must hit the cache?
  5. Size the cache to achieve that hit rate using your access distribution.

The Hit Rate Formula

For a p99 SLO, you need at least 99% of requests to complete within the target. If cache hit latency = Lhit and cache miss latency = Lmiss, and your SLO target = LSLO:

  • If Lhit < LSLO and Lmiss > LSLO: then every cache miss potentially violates the SLO.
  • Required hit rate h ≥ 1 - (target percentile violation budget). For p99 < LSLO, you can tolerate at most 1% of requests being misses (if misses always exceed the SLO).
  • More precisely: h ≥ 1 - (SLO violation budget / P(miss exceeds SLO)).

Example 1: Product Catalog API

SLO: p99 read latency < 20ms. Traffic: 50,000 reads/s. DB p99 = 45ms. Redis p99 = 1.5ms. Database: 10M products, avg 2 KB each.

Step 1: Required hit rate. Every cache miss (45ms) violates the 20ms SLO. For p99, at most 1% of requests can violate. So hit rate must be ≥ 99%.

Step 2: Items to cache for 99% hit rate. Assuming Zipfian access distribution with s=1.0 (typical for product catalogs), a 99% hit rate on 10M items requires caching the top ~200,000 items. (For Zipf with s=1.0: hit rate ≈ (cache_size/total)^(1-1/s) — more precisely, h = H(k,s)/H(N,s) where H is the generalized harmonic number and k = cache size. In practice, you simulate with your access logs.)

Step 3: Memory. 200K items × 2 KB = 400 MB raw data. With Redis overhead (2×): 800 MB. Add 25% growth headroom: 1 GB Redis instance.

Step 4: Verify DB load. Miss rate = 1% × 50K = 500 misses/s hitting DB. Plus 500 writes/s = 1,000 QPS on DB — easily handled by a single PostgreSQL instance. Without the cache: 50,500 QPS — would require read replicas.

Step 5: TTL. 500 writes/s across 10M products = any given product updated once per 20,000s (5.5 hours). TTL = 5 minutes gives at most 5 minutes of staleness. For a product catalog, this is acceptable.

Cost: AWS ElastiCache r6g.large (13 GB) = $146/month. Avoids needing 3-4 read replicas ($600-800/month). Net savings: ~$500/month.

Example 2: Session Store for Authentication

SLO: p99 auth check < 5ms (on the critical path of every API request). Traffic: 100,000 requests/s. DB p99 = 30ms. Redis p99 = 0.8ms. Active sessions: 2M, session size: 500 bytes.

Step 1: Required hit rate. DB latency (30ms) massively exceeds the 5ms SLO. Every miss violates. For p99: hit rate ≥ 99%. But since this is on the critical path of every request, you probably want p99.9: hit rate ≥ 99.9%.

Step 2: Items to cache. For 99.9% hit rate: you need essentially all active sessions in cache. Sessions have a natural TTL (expiry), so cache = all active sessions = 2M.

Step 3: Memory. 2M × 500B = 1 GB raw. Redis overhead: 2 GB. With headroom: 3 GB Redis.

Step 4: What about cold starts? After a Redis restart, cache is empty — 100% miss rate. All 100K RPS hit DB at 30ms — SLO violated for every request. Mitigation: cache warming script that loads all sessions from DB on startup (~2M × 500B = 1GB, takes ~30s at 100MB/s). During warm-up: route traffic through a backup Redis replica.

Cost: ElastiCache r6g.medium (6.4 GB) with replica = $150/month. This is non-negotiable — without cache, you'd need a DB that can serve 100K reads/s at <5ms, costing $2,000+/month.

Example 3: Social Media Feed (Relaxed SLO, Cost-Optimized)

SLO: p95 feed load < 200ms (users tolerate some slowness). Traffic: 20,000 reads/s. DB p95 = 150ms (complex join query). Redis p95 = 3ms. Users: 50M, feeds cached as JSON: avg 10 KB. Active users (daily): 5M.

Step 1: Required hit rate. DB p95 = 150ms < 200ms SLO. So even cache misses don't always violate the SLO! We have more flexibility. For p95: we can tolerate 5% being over 200ms. Since only some DB calls exceed 200ms (say 30% of them), we need: miss_rate × 0.30 ≤ 0.05, so miss_rate ≤ 16.7%. Hit rate ≥ 83%.

Step 2: Items to cache for 83% hit rate. 5M daily active users with Zipfian access: caching 500K feeds (10% of DAU) achieves ~85% hit rate. Why? The most active 10% of users generate ~85% of feed requests (power law).

Step 3: Memory. 500K feeds × 10 KB = 5 GB raw. Redis overhead: 10 GB. With headroom: 13 GB.

Step 4: Could we cache less? 200K feeds (4% of DAU): ~70% hit rate. Miss rate = 30%, SLO violation = 30% × 30% = 9% > 5%. Doesn't meet p95 SLO. 300K feeds: ~77% hit rate. 23% × 30% = 6.9% — still over. 500K is the minimum.

Cost: ElastiCache r6g.xlarge (26 GB) = $290/month. Alternatively: cache only 200K feeds (4 GB, $150/month) and accept the SLO miss, adding read replicas to reduce DB p95. The SLO drives the cost tradeoff.

Key Tradeoffs

  • Memory vs. hit rate: More memory = higher hit rate, but with diminishing returns (see curve above). The slope flattens — doubling cache from 1 GB to 2 GB might only improve hit rate from 92% to 95%.
  • TTL vs. freshness: Short TTLs keep data fresh but lower hit rates and increase DB load. Long TTLs improve hit rates but increase staleness. Choose TTL based on your staleness tolerance SLO (e.g., "product prices must be < 5 minutes stale").
  • Consistency vs. performance: Write-through caching keeps cache consistent but adds write latency. Cache-aside with TTL is faster but allows stale reads.
  • SLO drives everything: A stricter latency SLO requires a higher hit rate, which requires a larger cache, which costs more. Quantify the tradeoff: what's the cost of a larger cache vs. the cost of SLO violations?

Scaling Journey

0 – 1K users

No Cache

Your database handles all reads directly. With a few hundred users, a single PostgreSQL instance can serve reads in 5-10 ms. Adding a cache at this stage is premature optimization. Focus on correct data modeling and indexing.

Signal to evolve: Database CPU consistently above 60%, or p99 read latency exceeding your SLA.

1K – 100K users

Local In-Process Cache

Add a simple in-memory cache in your application (e.g., Python functools.lru_cache, Java Caffeine, Go sync.Map). No network hop, sub-microsecond reads. Works well for single-server deployments.

Limitation: Each app server has its own cache. Cold starts after deployments. No sharing between instances.

Signal to evolve: You have multiple app servers with inconsistent caches, or your working set exceeds available process memory.

100K – 10M users

Single Redis / Memcached

Deploy a dedicated caching service. All app servers share the same cache. Consistent view of cached data. Redis adds persistence options, pub/sub for invalidation, and data structures beyond simple key-value.

Limitation: Single point of failure. Memory limited to one machine (typically 64-256 GB). Network latency (~1 ms) per cache call.

Signal to evolve: Cache memory exhausted, single Redis CPU at 80%+, or availability requirements demand redundancy.

10M+ users

Redis Cluster + Multi-Tier

Redis Cluster shards data across multiple nodes. Add a local L1 cache (in-process, small, short TTL) in front of Redis (L2) for the hottest keys. A CDN (L3) caches static and semi-static content at the edge.

Architecture: Client → CDN → L1 (local) → L2 (Redis Cluster) → Database. Each layer filters traffic so the next layer sees less load.

Migration pitfall: Moving from single Redis to Redis Cluster requires key slot awareness. Keys with {hash_tag} prefixes must be planned in advance for multi-key operations.

Scaling the Cache Tier

The Scaling Journey above ends at "Redis Cluster + multi-tier" without explaining why a single cache node runs out or how the cache itself scales. A cache that can't keep up with traffic is just a slower database, so this matters.

Why One Cache Node Runs Out

A single Redis node has three independent ceilings, and you hit whichever comes first:

  • CPU / throughput. Redis executes commands on a single thread (detailed in "Redis vs. Memcached" below). A simple GET/SET workload tops out around 100,000–120,000 ops/sec on one core. Pipelining and multi-key commands push higher, but the per-core ceiling is real and you cannot add cores to escape it.
  • Network bandwidth. Cost = value_size × ops/sec. A 10 KB value at 100K ops/sec = 1 GB/s = 8 Gbps, which saturates a 10 GbE NIC before the CPU ceiling.
  • Memory. The working set must fit in RAM on one box — typically 64–256 GB. Beyond that, you must partition.

Replication — Scale Reads, Survive Failure

One primary plus N read replicas. Reads fan out across all N+1 nodes ((N+1)× read throughput); writes go to the primary and propagate asynchronously. Buys: read scaling and automatic failover. Costs: replication lag (a replica may serve slightly stale data), and writes still bottleneck on the single primary. Right when the read:write ratio is high and the entire working set still fits one box.

Sharding — Scale Writes and Memory

Partition the keyspace across M independent shards; each shard owns 1/M of the keys and serves its own reads and writes. This gives M× throughput and M× memory — the only technique here that scales writes. The key→shard mapping is the design decision: hash(key) % M remaps almost every key when M changes, so production systems use consistent hashing, where adding or removing a shard moves only ~1/M of keys. Redis Cluster implements exactly this with 16,384 hash slots.

But sharding balances load only if traffic is spread evenly across keys. Real traffic is not — it is Zipfian, and the head of that distribution breaks the assumption. That is the hot key problem.

The Hot Key Problem

A hot key is a single key whose request rate is a large fraction of total traffic: a celebrity profile, a viral post, a feature-flag config read on every request. Sharding deterministically maps one key to one shard — that is the entire point of a stable key→shard function. So a hot key concentrates its load on exactly one node, no matter how many shards you add. Adding shards cannot help, by construction. Consistent hashing solves rebalancing, not skew — it still sends every request for a given key to the same node.

This is not the thundering herd. The thundering herd (next section) is a miss storm: an expired key sends every server to the database. The hot key problem is a hit storm: the data is cached, hit rate ≈ 100%, the database is idle — yet one cache node melts. Different failure, different fix. Leases and request coalescing solve the herd; they do nothing for a hot key that is always a hit.

Worked Example: A Post Goes Viral

Setup: social app, 500,000 cache reads/sec, 10 shards via consistent hashing, value size 5 KB.

Uniform baseline: 500K / 10 = 50K ops/sec/shard. Network: 50K × 5 KB = 250 MB/s = 2 Gbps. Comfortable on every shard.

Spike: one post post:9f3 goes viral and alone draws 200K reads/sec (40% of all traffic). Consistent hashing maps it to shard 7.

Shard 7 load: its share of the other 300K traffic (300K / 10 = 30K) + the 200K hot reads = 230K ops/sec. Single-thread ceiling ≈ 120K → shard 7 is saturated. Its p99 climbs to seconds, and every other key that hashes to shard 7 is now slow too (noisy neighbor).

The tell: the other 9 shards sit at ~30K ops/sec (fine), cache hit rate is ~100%, and the database is idle. The cache tier itself is the bottleneck — a signature no DB-side mitigation will fix.

Detecting Hot Keys

  • redis-cli --hotkeys — samples the keyspace; requires an LFU maxmemory-policy so Redis tracks per-key access frequency.
  • Client/proxy top-K — a Count-Min sketch or heavy-hitters counter on the cache client samples keys and flags any exceeding, say, 1% of traffic (here, 5K ops/sec). Cheap and always-on.
  • Per-shard p99 — a latency spike isolated to one shard while hit rate stays ~100% is the hot-key signature, distinct from a herd (which shows DB load).

Mitigations (Read-Hot Keys)

The unifying idea: sharding spreads distinct keys; it cannot spread a single key. To spread one key you must copy it.

  • 1. Promote to L1 (local in-process cache). The highest-leverage fix, and exactly why the Scaling Journey's top tier has an L1. Each of the 200 app servers caches post:9f3 in process memory with a 1s TTL. The 200K reads/sec are now served with zero network hops; Redis sees at most ~200 refreshes/sec (one per server per second). Tradeoff: up to 1s of cross-server staleness. Cost: one 5 KB object per server — trivial. Best for read-heavy, staleness-tolerant data.
  • 2. Hot-key replication (key splitting). Store the value under K replica keys post:9f3#0 … post:9f3#K−1; consistent hashing scatters them across shards, so a reader picking a random replica spreads the load: 200K / 20 = 10K ops/sec/shard.
    • A write only has to reach all K replicas eventually — you choose where that cost lands:
      • Synchronous fan-out — K× write amplification, smallest skew. Still not atomic: Redis Cluster has no cross-slot transaction, so the K writes land at different instants and a reader can catch a half-updated set.
      • Async / pub-sub propagation — the writer does 1 op (update one replica, or publish once); a subscriber fans it to the other K−1. While that hop is in flight (pub-sub delivery, typically a few ms) the replicas hold different values — a bounded divergence window: the span over which the K copies disagree, capped by the propagation delay, so it self-heals within that fixed bound rather than staying stale indefinitely. The natural fit, since a cache is already only eventually consistent with the DB — a second eventual hop changes nothing fundamental.
      • Short-TTL lazy refill — the writer touches nothing; each replica expires and refills from the DB on its next miss, so staleness ≤ TTL. Cheapest writes.
    • Read-heavy keys only — write amplification or divergence grows with K.
  • 3. Replicas of the hot shard. Leave the keyspace alone; attach read replicas to the entire hot shard and round-robin reads across primary + replicas ((N+1)× read capacity). In the example, shard 7's 230K ops/sec split across primary + 3 replicas ≈ 57K each — under the ceiling. How it differs from key-splitting (#2):
    • What gets copied — #2 duplicates one key; #3 mirrors the shard's whole keyspace, including cold keys you don't care about.
    • Hardware — #2 reuses the existing M shards (no new nodes); #3 adds N dedicated nodes that do nothing but mirror shard 7.
    • Layer — #2 is app-managed (your code builds key#i and fans out writes); #3 is the cache's built-in primary/replica replication, transparent to the app.
    • Writes — #2 needs app-side write fan-out; #3 needs none (one write, auto-replicated) but every write still serializes on the single primary, so it stays write-bound.
    • When — #3 when the shard is hot diffusely (or for several keys) or you also want failover redundancy; #2 when one known key is hot and you want no extra hardware.
  • 4. Isolate known-hot keys. Route them to a dedicated cache tier so a viral spike can't starve unrelated keys sharing its shard.
import random, json

K = 20  # replica fan-out for a known hot key

def read_hot(r, key, fetch_fn, ttl=60):
    replica = f"{key}#{random.randrange(K)}"      # pick one of K copies at random
    val = r.get(replica)
    if val is not None:
        return json.loads(val)
    data = fetch_fn()                             # miss path — protect with a lease
    pipe = r.pipeline()
    for i in range(K):                            # warm all K replicas
        pipe.set(f"{key}#{i}", json.dumps(data), ex=ttl)
    pipe.execute()
    return data

def write_hot(r, key, data, ttl=60):
    # synchronous fan-out variant: simplest, K x write amplification.
    # alternatives: write one replica + propagate the rest async/via pub-sub,
    # or skip this entirely and let a short ttl + read_hot refill from the DB.
    pipe = r.pipeline()
    for i in range(K):                            # write fan-out: every replica
        pipe.set(f"{key}#{i}", json.dumps(data), ex=ttl)
    pipe.execute()

Write-Hot Keys Are Different

Replication mitigations all assume the value is mostly read. A key written 100K/sec — a global view counter, a "likes" tally — cannot be fixed by copying, because every copy must converge. Decompose it instead: keep N sub-counters likes:9f3#0 … #N−1, increment a random one (writes spread N-way), and sum all N on read (one fan-in). Or batch increments in the app and flush periodically, or use an approximate/CRDT counter. The principle mirrors the read case: spread a single key by copying for reads, by decomposing for writes.

What Can Go Wrong

1. Thundering Herd (Facebook Memcache, 2013)

What happened: When a popular cache key expired, hundreds of application servers simultaneously detected the miss and all queried the database for the same data. The database was overwhelmed by the sudden spike.

Root cause: All servers used the same TTL for the same key. When the TTL expired, the cache miss was replicated across the entire fleet at the same instant.

Blast radius: Database CPU spiked to 100%, causing cascading failures across dependent services. Response times went from 5 ms to 10+ seconds.

Mitigation 1: Locking / Leases (Redis SETNX)

The first server to detect a cache miss acquires a short-lived distributed lock. All other servers that detect the same miss are told to wait and retry. 

import redis, time, json

r = redis.Redis()

def get_with_lease(key: str, fetch_fn, ttl=300, lease_ttl=5):
    """Get from cache with thundering herd protection."""
    # 1. Try the cache first
    value = r.get(key)
    if value is not None:
        return json.loads(value)

    # 2. Cache miss — try to acquire a lease
    lease_key = f"lease:{key}"
    acquired = r.set(lease_key, "1", nx=True, ex=lease_ttl)

    if acquired:
        # 3a. We got the lease — we are the only one fetching from DB
        data = fetch_fn()
        r.set(key, json.dumps(data), ex=ttl)
        r.delete(lease_key)
        return data
    else:
        # 3b. Another server holds the lease — wait and retry
        for _ in range(50):  # retry up to 50 times
            time.sleep(0.1)  # 100ms between retries
            value = r.get(key)
            if value is not None:
                return json.loads(value)
        # Fallback: lease holder may have crashed. Fetch ourselves.
        return fetch_fn()

The nx=True flag (SETNX: "SET if Not eXists") ensures only one server acquires the lease. The ex=lease_ttl ensures the lease auto-expires if the holder crashes, preventing deadlocks.

Mitigation 2: Probabilistic Early Recomputation

Instead of waiting for the TTL to expire (hard cliff), each request has a small probability of refreshing the cache before the TTL expires. The probability increases as the TTL approaches zero, making it nearly certain that some server will refresh the key before the stampede.

The formula (from the "Optimal Probabilistic Cache Stampede Prevention" paper):

should_refresh = (current_time - (expiry_time - TTL * beta * ln(random()))) > expiry_time

Where beta controls how early the refresh can happen (typically 1.0). Let's trace through:

  • If the key has 200s remaining out of 300s TTL: probability of refresh is near 0%.
  • If the key has 30s remaining: probability is ~5% per request.
  • If the key has 5s remaining: probability is ~30% per request.
  • With 100 requests/sec, a key with 30s remaining will be refreshed within ~0.2 seconds. The TTL never actually expires.
import math, random, time

def should_early_recompute(expiry_time: float, ttl: float, beta: float = 1.0) -> bool:
    """Return True if we should refresh the cache early."""
    remaining = expiry_time - time.time()
    if remaining <= 0:
        return True  # already expired
    # XFetch algorithm: probability increases as remaining decreases
    threshold = remaining - (ttl * beta * math.log(random.random()))
    return threshold <= 0

Mitigation 3: Request Coalescing (Singleflight)

If multiple goroutines/threads request the same key at the same time, only one actually executes the fetch. The rest wait for the result. This is called the "singleflight" pattern (named after Go's golang.org/x/sync/singleflight package). 

import threading

class Singleflight:
    """Ensure only one fetch per key runs at a time."""

    def __init__(self):
        self._lock = threading.Lock()
        self._in_flight: dict[str, threading.Event] = {}
        self._results: dict[str, any] = {}

    def do(self, key: str, fetch_fn):
        with self._lock:
            if key in self._in_flight:
                # Another thread is already fetching this key — wait for it
                event = self._in_flight[key]
            else:
                # We are the first — register and proceed
                event = threading.Event()
                self._in_flight[key] = event
                event = None  # signal that WE should fetch

        if event is not None:
            event.wait()  # block until the fetcher is done
            return self._results.get(key)

        # We are the fetcher
        try:
            result = fetch_fn()
            self._results[key] = result
            return result
        finally:
            with self._lock:
                self._in_flight[key].set()  # wake up all waiters
                del self._in_flight[key]

With singleflight, 200 concurrent cache misses for the same key result in 1 DB query instead of 200. The other 199 threads wait for the result.

Facebook's solution: Facebook combined leases with staggered TTLs. They introduced a lease in their Memcache deployment: the first server to detect a miss gets a lease token, and all subsequent servers that detect the same miss are told to wait and retry (or serve slightly stale data if available). This reduced their peak DB load by over 10x during cache expiry events.

2. Cache Poisoning

What happened: A bug in the application wrote incorrect data to the cache with a long TTL. Every subsequent read returned the wrong value. Because the cache was trusted and the TTL was 24 hours, the bad data persisted for an entire day before being detected.

Root cause: No validation on cache writes. The application serialized a partially constructed object due to a race condition and stored it in Redis.

Blast radius: All users saw corrupted data for the affected keys. Customer support tickets spiked.

Prevention: Validate data before caching. Use shorter TTLs as a safety net (even if it costs hit rate). Implement cache versioning — include a version number in the key (e.g., v3:user:12345) so you can invalidate all old entries by incrementing the version. Monitor cache contents with periodic consistency checks against the database.

3. Cold Start After Fleet Restart

What happened: A Redis cluster was restarted for a maintenance upgrade. When it came back online, the cache was empty. All traffic hit the database directly, overwhelming it. The database became so slow that cache fill operations timed out, creating a vicious cycle where the cache could never warm up.

Root cause: No cache warming strategy. The system relied on organic traffic to fill the cache, but the database could not handle the full un-cached load.

Mitigation: Rate-limited the traffic to the database. Manually ran a cache warming script that pre-loaded the top 10,000 most-accessed keys.

Prevention: Use Redis persistence (RDB snapshots or AOF) so the cache survives restarts. Implement a cache warming pipeline that pre-loads the working set before cutting over traffic. Use circuit breakers on the database to prevent overload during cold starts.

Interactions & Pitfalls

Cache + Database Replication Lag

You invalidate a cache key after writing to the primary database. The application re-reads the data, but the read goes to a replica that hasn't received the write yet. The stale data from the replica is written back into the cache. Now your cache is stale until the next TTL expiry, even though the primary has the correct data.

Fix: After a write, either read from the primary for a short grace period (equal to your max replication lag), or use a "local write-through" that updates the cache directly from the write payload without re-reading from any database.

Cache + CDN Double-Caching

Your API response is cached in Redis with a 5-minute TTL. The CDN also caches the HTTP response for 10 minutes based on Cache-Control headers. After you invalidate the Redis cache, the CDN still serves the old response for up to 10 more minutes. Effective staleness: up to 15 minutes instead of 5.

Fix: Coordinate TTLs across layers. The outer layer (CDN) should have a shorter or equal TTL to the inner layer (Redis). Use CDN purge APIs for critical invalidations. Consider using stale-while-revalidate headers for graceful freshness.

Cache + Rate Limiting

A rate limiter returns HTTP 429 (Too Many Requests) to a client. If this response is cached (by an intermediate proxy, CDN, or application cache), all subsequent requests — including legitimate ones from other clients — may receive the cached 429. Legitimate traffic is blocked.

Fix: Never cache 4xx or 5xx error responses. Set Cache-Control: no-store on all error responses. If using a CDN, configure it to never cache responses with non-2xx status codes.

Redis vs. Memcached

These are the two open-source in-memory key–value stores that dominate production caching. Both serve sub-millisecond GET/SET from RAM over TCP, both are horizontally scalable, and both are battle-tested at hyperscale (Redis at Twitter/GitHub/Stack Overflow, Memcached at Facebook/YouTube/Wikipedia). They differ sharply in everything else — data model, threading, persistence, replication, eviction, and what kind of workload they're optimized for. Picking the wrong one means either over-paying for features you don't use, or hitting a wall at scale because the simple choice can't grow with you.

Origins and Philosophy

Memcached (2003, Brad Fitzpatrick at LiveJournal) was designed to do one thing exceptionally well: cache opaque bytes, indexed by string key, distributed across a fleet of cheap commodity boxes. Every design decision — multi-threaded architecture, slab allocator, no persistence, no replication, no data structures — flows from "be the fastest, simplest possible cache and let the application handle everything else." The protocol is a few text commands. The codebase is small and stable.

Redis (2009, Salvatore Sanfilippo) was designed as a "data structure server" — a remote, in-memory home for the kinds of data structures you'd otherwise build in your application: lists, hashes, sets, sorted sets, bitmaps, HyperLogLogs, geo indexes, streams. Caching is one use case; the broader pitch is "treat shared mutable state as a service." Redis added persistence, replication, clustering, scripting, transactions, and pub/sub on top of that core abstraction.

The Threading Difference, Visualized

The single biggest architectural difference is the threading model, and it shapes what each is good at:

Redis turns the single-threaded model from a weakness into a feature: there are no locks, no race conditions inside the server, and atomic ops like INCR, ZADD, or a Lua script are actually atomic without any cooperation from the client. Memcached pays for its multi-core throughput with a much weaker consistency model (single-key CAS only).

What "Redis is single-threaded" really means

A common misconception: people hear "single-threaded" and assume Redis is slow. It isn't — it's per-instance single-threaded for command processing, which is exactly what gives it deterministic ordering and lock-free semantics. To scale beyond one core, you run multiple Redis processes (shards) on the same machine and shard keys across them. Redis Cluster handles this in production: 16,384 hash slots distributed across nodes, clients route directly to the correct shard via CRC16(key) mod 16384.

Modern Redis (6.0+) does use multiple threads — but only for network I/O (reading bytes from sockets, parsing protocol, writing responses, TLS). The actual command execution is still serialized through one thread per shard. This typically buys 2–3× throughput on network-bound workloads without changing the consistency model.

Where Each One Hurts

Redis pain points:

  • One slow command kills throughput — KEYS *, SMEMBERS on a million-element set, big Lua scripts. Use SCAN, SSCAN, etc., and watch SLOWLOG.
  • Cluster operations are not transparent — multi-key ops require all keys hash to the same slot (use {tag} hash tags). Lua scripts can only touch keys on one slot.
  • Memory accounting is sneaky — maxmemory only counts dataset; replication buffers, AOF buffers, client output buffers are extra. OOM kills happen with maxmemory set "correctly."
  • Persistence + replication interactions — BGSAVE forks the process, doubling memory in the worst case (copy-on-write). Has to be planned for.
  • Single-shard write throughput ceiling — one shard maxes around 100–200K writes/sec. Past that, you're sharding.

Memcached pain points:

  • No persistence, no replication — cold restart = thundering herd to the database. You build the durability story yourself.
  • Slab calcification — if your value-size distribution changes over time, slabs configured for the old distribution waste memory. Modern Memcached supports slab rebalancing but it's not free.
  • No data structures — want to atomically push to a list? Fetch the whole list, append, write back, hope nobody else did the same. Use Redis instead.
  • 1 MB value limit by default — raising it is possible but discouraged.
  • Sharding is client-side — consistent hashing implementations vary across libraries, leading to subtle re-sharding pain when you add nodes.

Modern Alternatives Worth Knowing

  • DragonflyDB — Redis-protocol-compatible, multi-threaded command execution, claims 10–25× single-box throughput. Rewrites the data structures to be lock-friendly. Worth evaluating for write-heavy workloads.
  • KeyDB — A multi-threaded fork of Redis maintained by Snap. Drop-in replacement; ~5× throughput on multi-core boxes. (Snap was acquired by Redis Labs in 2022; future is unclear.)
  • Garnet (Microsoft) — Cache server compatible with the Redis protocol, written in C#, optimized for tiered storage (hot in RAM, cold on SSD). Designed for cloud-native workloads.
  • Hazelcast / Ignite — JVM-based distributed in-memory data grids. Different ecosystem, but solve overlapping problems.
  • Mcrouter — Facebook's Memcached protocol router that adds replication, failover, and pool management on top of vanilla Memcached. Lets you keep Memcached's simplicity while getting some HA story.

Bottom line: If you're starting today and don't have specific reasons to pick Memcached, choose Redis. The richer feature set rarely hurts and frequently saves you from rebuilding leaderboards, queues, locks, or rate limiters yourself. Pick Memcached when you specifically need its multi-threaded throughput on a single box and your data is genuinely just opaque blobs — this is most common in classic web-app HTML caching at very large scale (Facebook's pattern).

Code Example: LRU Cache in Python

A simple LRU cache using a dictionary (which preserves insertion order in Python 3.7+) and a maximum size limit. The animation on the right traces through the same operations as the step-by-step table from the LRU section above, using the same keys (A, B, C, D, E) and capacity of 3.

class LRUCache:
    """A simple Least Recently Used cache."""

    def __init__(self, capacity: int):        # Step: create cache
        self.capacity = capacity
        self.cache: dict = {}  # ordered dict (Python 3.7+)
        self.hits = 0
        self.misses = 0

    def get(self, key: str):
        if key in self.cache:                 # Step: lookup in hash map
            self.hits += 1
            # Move to end (most recently used)
            self.cache[key] = self.cache.pop(key)  # Step: move to head
            return self.cache[key]
        self.misses += 1                      # Step: cache miss
        return None

    def put(self, key: str, value, ttl_seconds: float = 300):
        import time
        if key in self.cache:
            del self.cache[key]               # Step: remove old position
        elif len(self.cache) >= self.capacity:
            # Evict least recently used (first item)
            evicted_key = next(iter(self.cache))  # Step: find LRU item
            del self.cache[evicted_key]           # Step: evict LRU
        self.cache[key] = {                   # Step: insert at head
            "value": value,
            "expires_at": time.time() + ttl_seconds,
        }

    def hit_rate(self) -> float:
        total = self.hits + self.misses
        return self.hits / total if total > 0 else 0.0


# Usage: cache-aside pattern
cache = LRUCache(capacity=1000)

def get_user(user_id: str) -> dict:
    cached = cache.get(f"user:{user_id}")     # Step: check cache
    if cached is not None:
        return cached["value"]                # Cache hit — return

    # Cache miss — fetch from database
    user = db.query(f"SELECT * FROM users WHERE id = {user_id}")
    cache.put(f"user:{user_id}", user, ttl_seconds=300)  # Fill cache
    return user

How to Validate Your Design

Load Testing

Use k6 or Locust to simulate realistic read traffic against your cached endpoint. Key things to measure:

  • Hit rate under load — does it match your expected working set analysis?
  • p50, p95, p99 latency for cache hits vs. cache misses
  • Database QPS with and without the cache — verify the expected load reduction
  • Behavior during cache warm-up: how long until hit rate stabilizes?

Generate traffic with a Zipfian distribution (not uniform random) to simulate realistic access patterns. Most load testing tools support custom distributions or you can pre-generate a key list following a Zipfian curve.

Chaos Testing

Inject failures to validate your system degrades gracefully:

  • Kill a cache node: Does the application fall back to the database? Does it recover when the cache comes back? Use toxiproxy to simulate a Redis outage.
  • Simulate full cache eviction: Run FLUSHALL on a staging Redis and observe the thundering herd on the database. Verify your circuit breaker trips before the DB dies.
  • Add network latency: Use tc netem to add 50 ms latency to the Redis connection. Verify your application timeouts and fallbacks work correctly.
  • Expire all keys simultaneously: Set all TTLs to 1 second and watch what happens. This simulates the worst-case thundering herd.

Benchmarking Methodology

  • Warm the cache first: Never benchmark against a cold cache unless you are specifically testing cold-start behavior. Run a warm-up phase of 5-10 minutes before collecting measurements.
  • Use realistic data sizes: Benchmarking with 10-byte values tells you nothing about production performance with 2 KB JSON documents.
  • Measure tail latency: p99 and p99.9 matter more than average. A GC pause or network hiccup can cause a 100x latency spike that the average hides.
  • Account for connection overhead: Use persistent connections (connection pooling) in your benchmark, just like you would in production.
  • Test with the right tool: redis-benchmark is useful for raw Redis throughput, but for end-to-end application benchmarking, use k6 or wrk hitting your actual API endpoints.

Interview Questions

How would you design a cache invalidation strategy for a social media feed?

A social media feed aggregates posts from hundreds of followed accounts. The naive approach — invalidating the entire cached feed on every new post — would cause constant cache misses since someone you follow posts every few seconds.

Approach: Use a write-behind fan-out model. When a user publishes a post, push the post ID into each follower's cached feed list (stored as a Redis sorted set keyed by timestamp). This is an append, not an invalidation. The cached feed stays warm.

For celebrity accounts with millions of followers, use a hybrid approach: don't fan-out at write time. Instead, when a reader requests their feed, merge their pre-assembled feed (from normal followed accounts) with a fresh query for celebrity posts. This is the "fan-out on read" optimization that Twitter famously uses.

Set a TTL of 5-15 minutes on the cached feed. Use stale-while-revalidate logic: serve the stale feed immediately while recomputing in the background.

Follow-ups: How do you handle deletions? (Tombstones in the feed list.) What about feed ranking changes? (Re-rank on read with a short TTL on the ranking cache.) How do you handle a user unfollowing someone? (Remove that user's posts from the cached feed lazily on next read.)

Explain the thundering herd problem and how to prevent it.

The thundering herd occurs when a popular cache key expires and many servers simultaneously detect the miss. All of them query the database for the same data, creating a load spike that can overwhelm the backend.

Prevention strategies:

  • Locking / leases: The first server to detect a miss acquires a distributed lock (or lease) and is the only one allowed to fetch from the database. Other servers either wait and retry, or serve a slightly stale value.
  • Staggered TTLs: Add random jitter to each TTL. Instead of all copies expiring at T=300s, they expire between T=270s and T=330s, spreading the load.
  • Probabilistic early recomputation: Each request has a small probability of refreshing the cache before the TTL expires. The probability increases as the TTL approaches expiry. This ensures the cache is almost always fresh without a hard cliff.
  • Background refresh: A dedicated worker proactively refreshes hot keys before they expire, so application servers never see a miss.

Follow-ups: How would you implement the lease in a distributed system? (Use Redis SETNX with a short TTL.) What happens if the lease holder crashes? (The lease TTL expires and another server can acquire it.) How does this interact with circuit breakers? (If the DB is down, the lease holder should fail fast and serve stale data.)

When would you choose write-through vs. write-behind caching?

Write-through: Every write goes to both the cache and the database synchronously. The write is only considered successful when both are complete. Use this when data consistency between cache and database is critical, and you can tolerate higher write latency.

Write-behind (write-back): Writes go to the cache immediately, and the cache asynchronously flushes to the database in batches. Use this when write performance is critical (e.g., a gaming leaderboard, analytics counters) and you can tolerate the risk of data loss if the cache crashes before flushing.

Decision factors:

  • Can you lose the last few seconds of writes? If no, use write-through. If yes, write-behind gives 10-100x better write throughput.
  • Is your write rate high enough that the database can't keep up in real-time? Write-behind lets you batch and coalesce writes (e.g., 100 increments to the same counter become a single +100 write).
  • Do reads need to see writes immediately? Write-through guarantees this. Write-behind guarantees it for cache reads but not for direct database reads.

Follow-ups: How do you handle write-behind with multiple cache nodes? (Use a write-ahead log per node.) What happens during failover? (The WAL must be replicated or the unflushed writes are lost.) Can you combine write-through with cache-aside? (Yes — write-through for writes, cache-aside for reads, is a common hybrid.)

How do you handle cache warming for a new data center or after a full cache flush?

A cold cache means 100% of traffic hits the database. If the database cannot handle the full un-cached load (which is typical at scale), you need a warming strategy.

Approaches:

  • Pre-loading: Before routing traffic, run a batch job that loads the top N most-accessed keys from the database into the cache. You need access logs or analytics to know which keys are hot.
  • Gradual traffic shifting: Route only 5% of traffic to the new data center initially. The cache warms organically under low load. Gradually increase to 25%, 50%, 100% over hours.
  • Cache replication: If you have an existing warm cache, replicate its contents to the new cache before cutover. Redis supports replication natively. Memcached does not, but you can build a migration tool.
  • Request coalescing: During warm-up, use request coalescing (singleflight pattern) so that duplicate cache misses for the same key result in only one database query.

Follow-ups: How long does warming take? (Depends on working set size and DB throughput. For 1M keys at 10K writes/sec to Redis, ~100 seconds.) How do you prioritize which keys to warm first? (By access frequency from logs. Hot keys first.) What if you don't have access logs? (Start with a small traffic slice and let organic warming happen.)

Design a caching strategy for a product catalog with 10 million items.

Start with the access pattern: product catalogs follow a heavy Zipfian distribution. A few thousand popular products (bestsellers, trending, homepage features) account for the vast majority of views.

Architecture: Use a multi-tier cache. L1: in-process cache (Caffeine/Guava) on each app server holding the top 500 products, TTL = 30 seconds, ~1 MB. L2: Redis cluster holding the top 100K products, TTL = 10 minutes, ~200 MB. L3: CDN for product detail pages with Cache-Control: max-age=60, stale-while-revalidate=300.

Invalidation: When a product is updated (price change, inventory update), publish an invalidation event to a Kafka topic. Each app server subscribes and evicts the key from its L1 cache. A separate consumer invalidates the Redis key. For CDN, use the provider's purge API for critical changes (price drops) and rely on TTL for non-critical changes (description edits).

Sizing: 10M products × 2 KB = 20 GB total. Caching the top 1% (100K products) covers 90%+ of reads and costs ~400 MB in Redis (with overhead). ROI is excellent.

Follow-ups: How do you handle flash sales where a cold product suddenly becomes hot? (Detect traffic spikes and pre-warm. Use request coalescing for the initial burst.) How do you handle personalized pricing? (Cache the base product data; compute personalized pricing on top. Don't cache personalized responses in shared caches.)

What are the tradeoffs between Redis and Memcached for caching?

Memcached is simpler: it is a pure key-value store with a multi-threaded architecture. It excels at simple caching workloads with many concurrent connections. It has no persistence, no replication, and no data structures beyond strings. Its memory allocator (slab allocator) is highly efficient and predictable.

Redis is richer: it supports data structures (lists, sets, sorted sets, hashes, streams), persistence (RDB/AOF), replication, Lua scripting, pub/sub, and clustering. It is single-threaded (per shard) for command processing, which simplifies consistency but limits per-node throughput on CPU-bound workloads.

Choose Memcached when: You need simple key-value caching at high concurrency, your values are small (< 1 MB), you don't need persistence or data structures, and you want multi-threaded performance on a single node.

Choose Redis when: You need data structures (sorted sets for leaderboards, lists for queues), persistence for cache durability across restarts, pub/sub for invalidation, or built-in clustering. Most teams choose Redis because its richer feature set justifies the small performance overhead.

Follow-ups: How does Redis 7.0's multi-threaded I/O change this? (It handles network I/O on multiple threads but still processes commands single-threaded. Improves throughput by 2-3x for network-bound workloads.) What about DragonflyDB? (A modern Redis-compatible store with multi-threaded command processing. Claims 10-25x throughput improvement.)

Key Takeaways

  • Caching trades memory and consistency for latency and throughput. The tradeoff is almost always worth it for read-heavy workloads.
  • Cache-aside is the most common pattern: the application checks the cache, falls back to the database on a miss, and fills the cache for next time.
  • Your cache only needs to hold the working set, not all your data. For Zipfian workloads, caching the top 1% of keys can capture 80%+ of reads.
  • The thundering herd is the most dangerous failure mode. Prevent it with lease-based locking, staggered TTLs, and request coalescing.
  • Always add jitter to TTLs. A fixed TTL across all keys causes synchronized expiry storms.
  • Monitor hit rate as your primary health metric. If it drops below 85%, investigate immediately.
  • Modern eviction policies (W-TinyLFU, ARC) significantly outperform classic LRU, especially on workloads with scan pollution.
  • Cache invalidation is one of the two hard problems in computer science. When in doubt, use short TTLs rather than complex invalidation logic.
  • Plan for cold starts. Have a warming strategy before you need one.
  • The cost of caching is almost always lower than the cost of scaling your database to handle un-cached load.
