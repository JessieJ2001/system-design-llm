Interview Quick Reference

Exhaustive clarification checklist + concise design tradeoff comparisons. One page to scan before any system design interview.

15 min read

Clarifications to Seek

Before drawing a single box, pin down these dimensions. Each one eliminates entire categories of design choices.

Traffic & Scale

  • QPS (Queries Per Second): Average and peak? Peak-to-average ratio? (typically 3–10x)
  • Data size: Total storage now and growth rate? Does it fit one machine?
  • Read/write ratio: 100:1 read-heavy (caching/replicas) or 1:1 write-heavy (sharding/queues)?
  • User count: DAU (Daily Active Users) vs total? Concurrent sessions?
  • Object size: Average payload size? Affects bandwidth, storage tier, and caching decisions.
  • Data growth: How much new data per day/month? When do you hit the next capacity cliff?

Latency & Performance

  • Latency SLO (Service Level Objective): p50 vs p99 target? <50ms interactive, <500ms background, <5s batch.
  • Throughput target: Sustained vs burst? How long do bursts last?
  • Real-time requirement: Is <100ms delivery needed (WebSocket / SSE — Server-Sent Events) or is polling acceptable?
  • Latency budget breakdown: If end-to-end is 200ms across 5 services, each gets ~35ms after network overhead.

Consistency & Correctness

  • Staleness tolerance: Can reads be stale? For how long? 300ms stale likes = fine; stale bank balance = catastrophic.
  • Business cost of staleness: Quantify in dollars — $0 for social likes, $2M/day for double-charges.
  • Exactly-once semantics: Is duplicate processing acceptable? (notifications yes, payments no)
  • Ordering guarantee: Global order? Per-user? Per-conversation? Per-entity?
  • Conflict resolution: Last-write-wins? CRDTs (Conflict-free Replicated Data Types — data structures like counters and sets that automatically merge concurrent updates without coordination)? Application-level merge?

Availability & Durability

  • Availability target: 99.9% (8.7h/yr), 99.99% (52min/yr), 99.999% (5min/yr)? Each 9 costs ~10x.
  • RPO (Recovery Point Objective): How much data can you lose, measured in time. RPO=0 → synchronous replication (zero loss). RPO=5min → async replication is fine. Financial: 0. Social: minutes.
  • RTO (Recovery Time Objective): How fast must the system be back online after failure. RTO=30s → automated failover, leader election, pre-provisioned standbys. RTO=1h → manual failover, cold standby acceptable.
  • Multi-region: Required for disaster recovery (DR)? If so: active-active (both regions serve traffic) or active-passive (standby region takes over on failure)?
  • Failure modes: What can go wrong and how often? Shapes retry, idempotency, and replication factor.

Access Patterns

  • Hot spots: Uniform distribution or Zipfian/power-law? (celebrity followers, viral content, flash sales)
  • Query patterns: Point lookups vs range scans vs full-text search vs aggregations?
  • Temporal patterns: Time-of-day spikes? Seasonal? Predictable or sudden?
  • Fan-out factor: One write triggers how many reads/notifications? (1 celebrity post → 10M deliveries)
  • Access recency: Do users access recent data mostly, or uniformly across history?

Operational Context

  • Team size: <20 engineers → monolith; >50 → microservices (Conway's Law).
  • Budget: Cost ceiling? Optimize for cost vs latency vs simplicity?
  • Existing infra: What's already in place? Cloud provider constraints?
  • Compliance: Data residency? Encryption at rest? Audit logs? Retention period?
  • What does "failure" look like? 503 error page vs degraded results vs queued-for-later?

Data Layer Tradeoffs

SQL vs NoSQL

SQL when you need ACID (Atomicity, Consistency, Isolation, Durability) transactions, complex joins, and strong consistency (banking, e-commerce orders). NoSQL when you need horizontal write scalability (>50K writes/sec — PostgreSQL's ceiling), flexible schemas, or simple key-value access at massive scale. Many systems use both: SQL for transactional core, NoSQL for read-heavy denormalized views.

CriterionSQLNoSQL
Complex joinsNative (1–5ms)No joins — must denormalize (see below)
ACID transactionsNative multi-record (BEGIN; UPDATE A; UPDATE B; COMMIT;)Single-document atomic only. Cross-record needs workarounds: Saga (compensating actions, not truly atomic) or CRDTs (merge-friendly structures, limited to counters/sets)
Horizontal write scaleSharding works but painful (see below)Built-in (DynamoDB partitions, Cassandra vnodes)
Write throughput ceiling~50K inserts/sec single node1M+ writes/sec (Cassandra LSM — Log-Structured Merge-tree, optimized for sequential writes)
Schema flexibilityRigid, migrations requiredSchema-on-read, flexible

What "must denormalize" means

In SQL, data is normalized: split across tables, joined at query time. NoSQL has no joins, so you denormalize — pre-flatten data by storing redundant copies together. Example: instead of joining users + orders + products, store one document with all fields embedded. Reads are fast (one lookup), but writes must update every copy — miss one and data goes stale.

Why SQL sharding is painful

NoSQL databases are distributed from the start — sharding is built into the storage engine. SQL databases bolt it on, which introduces several challenges:

  • Cross-shard queries: A query touching data on multiple shards requires scatter-gather — send query to all shards, merge results. Latency = slowest shard.
  • Cross-shard transactions: ACID doesn't span shards natively. Need 2PC (Two-Phase Commit — coordinator asks all shards to prepare, then commit; adds latency, holds locks) or Saga patterns (adds compensating logic).
  • Shard key is permanent: The column you shard by (user_id, order_id, etc.) determines which queries are fast (local to one shard) and which are slow (scatter-gather). Changing the shard key later means migrating all data — effectively rebuilding the database.
  • Resharding is painful: When shards outgrow capacity, you must split them. With naive hash % N, changing N remaps nearly every key — massive data migration. Mitigation: use virtual shards (256 logical → N physical) so resharding only moves a fraction. But this must be planned upfront; retrofitting is much harder.
  • Operational overhead: N independent databases to monitor, back up, upgrade, and failover instead of one.

Sharding vs Replication

Replication multiplies read capacity and availability — same data on multiple nodes, survives failures. Sharding splits data across nodes for write scalability and storage capacity. Ask: "Is my bottleneck read throughput / availability (→ replicate) or write throughput / data size (→ shard)?" Most production systems do both: shard for write scale, replicate each shard for durability.

DimensionReplicationSharding

Failure impactOther replicas serve readsOne shard's data unavailable
ComplexityConsistency model (async/sync/semi-sync)Shard key choice, cross-shard queries, resharding

Strong vs Eventual Consistency

Strong (linearizable) guarantees everyone sees the latest write — required when staleness has measurable business cost (double-charges, overselling). Eventual is faster and more available but allows stale reads. Middle ground: read-your-writes for shopping carts and user profiles.

ModelLatencyAvailabilityUse Case

Shard Key Selection

The shard key determines which queries are local (fast) and which require scatter-gather (slow). Pick the key that matches your dominant access pattern.

  • By user_id: Per-user queries are local. Risk: power-law users (celebrities) create hot shards.
  • By entity_id: Uniform distribution. Risk: user-scoped queries require scatter-gather across all shards.
  • By time: Range scans by time are fast. Risk: recent shard gets all writes (hot shard).
  • Compound key (user + time): Per-user time-range queries are local. Best of both when access pattern fits.

Resharding rule: Use virtual shards (e.g., 256 logical → N physical) so resharding moves a fraction of data instead of rehashing everything. Never use hash % N directly.

Consistent Hashing

Maps keys to nodes on a ring so that adding/removing a node only moves ~1/N of keys (vs hash % N which remaps nearly everything). Use virtual nodes (100–400 per physical node) to reduce load imbalance from ~3.6x to <5%. Memory overhead is negligible (~32 bytes per vnode).

Where it's used — anywhere you have N nodes that can change (scale up, crash, deploy) and want to minimize disruption:

Communication Patterns

Sync vs Async Communication

Sync when the user is waiting for the result and it must succeed before responding (payment, inventory check). Async when the operation can happen later or the caller doesn't need the result (notifications, analytics). Rule: "Is the user staring at a spinner?" Yes → sync. No → queue it. Async decouples services: if downstream is slow/down, requests queue instead of cascading.

Push vs Pull

Push gives low latency but write-amplification scales with subscriber count (one celebrity post = millions of writes). Pull is simpler but adds polling latency and wastes bandwidth when nothing changed. Hybrid is common: push for normal users (<5K followers), pull for celebrities (millions). Decision variable: updates/sec × subscribers_per_update.

Rate Limiting vs Queues

Rate limiting rejects excess traffic (429) — protects services from overload, enforces tenant fairness. Queues absorb excess traffic and process it later — smooths bursts, enables async processing. Use rate limiting at the API gateway for external traffic. Use queues internally when work can be deferred. They often work together: rate limit at the edge, queue behind the limit.

Rate Limiting Algorithms

Sizing rule: Set rate limit at 80% of capacity at SLO latency. Burst = 2–3x sustained rate.

Message Queue Patterns

  • Utilization sweet spot: 60–70%. Above 70%, tail latency amplifies non-linearly (M/M/c queuing). At 90%, response time is ~5–10x baseline.
  • At-least-once + idempotency = exactly-once. Use idempotency keys for payment/financial systems.
  • Ordering: Partition by the entity that needs ordering (conversation_id for chat, not user_id). Ordering within partition is guaranteed; cross-partition is not.
  • Dead letter queue (DLQ): Route permanently failing messages (corrupt data, missing fields) to a separate queue after N retries. Prevents infinite retry loops. Engineers inspect the DLQ manually or with alerting.
  • Backpressure: When consumers can't keep up — slow producers (backpressure) or drop oldest data (lossy buffer). Choice depends on data importance.

L4 vs L7 Load Balancing

L4 (TCP/UDP) routes by IP+port. Fast (~0.1ms overhead), stateless, handles any protocol including WebSockets. Use when you don't need to inspect content. L7 (HTTP) routes by URL path, headers, cookies. Slower (1–5ms) but enables content-based routing, A/B testing, auth at the edge. Use when different endpoints need different backends.

Caching Strategies

StrategyHow It WorksBest For
Cache-asideApp checks cache; on miss reads DB and populatesRead-heavy (95:5), infrequent updates
Write-throughEvery write goes to cache + DB togetherReads ≈ writes, cache freshness critical (sessions)
Write-behindWrite to cache, async flush to DBWrite-heavy + acceptable data loss (counters)

TTL (Time To Live) tradeoff: Longer TTL = higher hit rate + staler data. Product catalog: 5min fine. Payment state: <1sec. Immutable assets: TTL=1yr with filename hash for cache-busting.

Cache sizing: For Zipfian workloads, cache √N items as lower bound. Hit rate has diminishing returns. Derive required hit rate from SLO: if cache hit = 1.5ms but miss = 45ms (DB lookup), and your p99 SLO is 20ms, then every miss (45ms) violates the SLO. Since p99 allows at most 1% of requests to exceed 20ms, at most 1% can be misses → need ≥99% hit rate.

Cache warmup — pre-populate cache before it takes traffic (avoids cold-start 100% miss rate hitting DB):

  • Startup script: On deploy/restart, query DB for most popular keys and SET them in cache before accepting traffic.
  • Shadow traffic: Route a copy of production reads to the new cache node before adding it to the ring. It warms up without serving real users.
  • Replicate from peer: Copy hot keys from a healthy cache node (Redis MIGRATE or snapshot restore).

Gradual invalidation — avoid expiring all keys at once (thundering herd on DB):

  • Jittered TTL: TTL = base ± random(0, spread). Base=5min, spread=30s → keys expire between 4:30–5:30 instead of all at 5:00. Spreads DB load evenly.
  • Probabilistic early refresh: On each read, with small probability (e.g., 0.1%), refresh the key before it expires. Prevents synchronized expiry — by the time TTL hits, the key is already refreshed.
  • Stale-while-revalidate: Serve the stale value immediately, fetch fresh value in background. User gets fast response, cache gets updated, DB never sees a spike.

CDN

Edge caches static/semi-dynamic content close to users. Target >95% hit rate for static, 60–80% for semi-dynamic. Below 50% hit rate, CDN adds latency without benefit — remove it. Use stale-while-revalidate for grace periods. For sub-100ms dynamic content, consider edge compute (workers + edge KV).

API Gateway

Central entry point handling TLS termination (1–3ms), auth (0.1–20ms — cache tokens!), rate limiting (0.5–2ms), routing + transforms (0.1–1ms). Total overhead: 2–5ms well-tuned. Account for fanout: Backend RPS = Client RPS × Average Fanout Ratio.

Circuit Breakers

Prevents cascading failure by failing fast when a downstream is unhealthy. Without it: slow downstream blocks threads → thread pool exhaustion → your service becomes unresponsive too.

  • Failure threshold: % error rate with minimum sample size. Payment: 1%. Third-party API: 10%.
  • Open timeout: 30s for local services, 60–90s for DBs, 60–300s exponential backoff for third-party.
  • Half-open probes: 3–5 requests to confirm recovery. Critical paths: 5 probes.

Architecture Patterns

Monolith vs Microservices

Monolith when team is <20 engineers, product is evolving fast, and simplicity matters more than independent scaling. Microservices when >50 engineers need to deploy independently, services have different scaling needs, and team can absorb operational complexity. Transition zone (20–50 engineers): extract services incrementally at proven domain boundaries.

DimensionMonolithMicroservices
Team size<20 engineers>50 engineers
Deploy speedFast (single artifact)Independent per service
ScalingAll-or-nothingPer-service
DebuggingOne process, stack traceDistributed tracing required
Data consistencySingle DB, ACID transactionsSaga patterns, eventual consistency

2PC vs Saga (Distributed Transactions)

2PC (Two-Phase Commit) when all participants are local (<5ms RTT — Round-Trip Time), transaction is fast (<100ms), and you need atomicity (bank transfer in same DB). Saga when operations span external APIs, are long-running, or cross regions (2×80ms RTT makes 2PC prohibitive). Saga trades atomicity for availability — compensating transactions (refund, cancel) instead of rollback.

Pessimistic vs Optimistic Concurrency

Pessimistic (acquire lock before write) when contention is high (>30% conflicts) — prevents wasted work. Optimistic (write, detect conflict at commit, retry) when contention is low (<5%) — avoids lock overhead. Gray zone (5–30%): benchmark both. Lock TTL: set to 3–5x the p99 operation time (50ms op → 150–250ms TTL).

Leader Election

Raft/Paxos for consensus with automatic failover and zero data loss — higher latency (10–50ms/write) but strongest guarantees. Lease-based (TTL lock in Redis/etcd) when you just need "one active worker" — simpler, faster, split-brain risk if TTL too aggressive. Failover budget = detection (3–5 missed heartbeats) + election (500ms–2s) + client reroute (1–3s).

Fan-Out Strategies

  • Fan-out on write (push): Pre-compute and store results for each consumer. Fast reads, expensive writes. Works when follower count is bounded (<5K).
  • Fan-out on read (pull): Compute at read time from source of truth. Cheap writes, expensive reads. Works for celebrities (millions of followers).
  • Hybrid: Push for normal users, pull for celebrities. Threshold: fan-out cost vs read cost crossover.

Hot Key / Hot Shard Mitigation

  • Detection: Count-Min Sketch (O(1), ~540 KB for 100M keys) beats min-heap (O(log N) per request).
  • Replication: Replicate hot key to 5–10 nodes. Reads spread, writes go to all replicas.
  • L1 local cache: For extreme hot keys (50K+ reads/sec), cache in app-server memory. Tradeoff: staleness (small TTL acceptable).
  • Virtual nodes don't solve hot keys — they improve distribution of which keys go where, but a single hot key still maps to one physical node.

Resilience Patterns

Cascading Failure Prevention

Service C slows down → B's threads block waiting for C → B's thread pool exhausts → A can't reach B → entire chain fails. Prevention stack:

  1. Timeouts: Fail fast (200ms) instead of waiting 30s default. Every request reaching a slow dependency gets a bounded penalty.
  2. Circuit breakers: After N failures, stop calling the failing service entirely. Return fallback/cached/degraded response.
  3. Bulkheads: Isolate thread pools per dependency. Slow dependency A can't starve calls to healthy dependency B.
  4. Graceful degradation: Serve partial results. Product page without recommendations is better than 503.

Thundering Herd

Cache node crashes → 100K keys expire simultaneously → millions of requests hit DB at once. Mitigations:

  • Request coalescing: 5 concurrent requests for same expired key → 1 DB query, 4 wait. Impact: 100K queries instead of millions.
  • Stale-while-revalidate: Serve slightly stale data while refreshing in background.
  • Probabilistic early refresh: 0.1% of requests trigger refresh before TTL expires, preventing synchronized expiry.
  • Jittered TTL: TTL = base ± random(0, 30s). Prevents all keys expiring at the same moment.

Flash Sale / Traffic Spikes

  • Virtual waiting room: Queue users at the edge, release 5K/sec. System never exceeds capacity. Users wait in fair queue instead of experiencing crashes.
  • Fast-fail sold-out check: In-memory inventory check (0.5ms) before DB (5ms+). 98.9% of attempts are doomed — reject them instantly.
  • Pre-warm caches: Load product data and inventory into cache before sale starts.

Exactly-Once Processing

True exactly-once is impossible in distributed systems. Instead: at-least-once delivery + idempotency = effectively exactly-once. Use idempotency keys (payment_id, request_id) to detect and deduplicate replayed messages. For financial systems, a 0.01% duplicate rate = $432K/day in double-charges — idempotency is mandatory, not optional.

Bloom Filters

Probabilistic set membership test. "Definitely not in set" (no false negatives) or "probably in set" (tunable false positive rate). Formula: m = -n × ln(p) / (ln 2)² bits, k = (m/n) × ln 2 hash functions. Use cases: URL dedup (0.1% false positive rate, 1.8 GB for 1B URLs vs storing URLs at 60 GB), LSM-tree lookups (1% false positive, 12 MB per SSTable), spam filtering (0.01% false positive).

Logging at Scale

Write-heavy workload (2M+ lines/sec) with occasional reads (searches during incidents). Key tradeoffs:

  • Tiered sampling: 1% INFO, 25% WARN, 100% ERROR cuts volume 10x while preserving critical data.
  • Hot/warm/cold: 3 days searchable (Elasticsearch), 30 days queryable (object storage + Parquet), archive beyond.
  • At extreme scale (10x): Consider label-based indexing (Grafana Loki) over full-text inverted indices — reduces index storage 5–10x.

Real-Time Leaderboard

Redis sorted set for top-K queries (O(log N) updates, O(K) top-K retrieval). For "my rank" queries at scale: pre-computed histograms (bucket scores into ranges, local array lookup) give ±5K accuracy — invisible to 99.9% of players. Use exact rank (expensive fan-out) only for top 1,000 where precision matters.

Quick Decision Formulas

Little's Law & Queue Latency

Little's Law: L = λ × W. It says: the average number of requests "in flight" (queued + being processed) equals the arrival rate multiplied by the average time each request spends in the system. Holds for any stable system — no assumptions about distribution needed.

VariableMeaningConcrete Example
LAverage requests in the system (waiting in queue + being processed)50 RPC requests in flight
λArrival rate (requests/sec)1,000 RPC requests/sec arriving
WAverage time each request spends in system (queue wait + processing)50ms per request

Check: L = 1,000 req/s × 0.05s = 50 requests in flight at any moment. If you see 50 in-flight requests and know they arrive at 1,000/s, each one must be spending 50ms in the system on average.

Example 1: "How many requests are stuck waiting?"

Setup: A single-threaded worker pulls from a queue and processes each request in 1ms. Requests arrive at the queue at 800 req/s.

Given:
  λ = 800 req/s          (arrival rate)
  μ = 1 / W_service      (μ is the inverse of service time)
    = 1 / 1ms
    = 1 / 0.001s          (1ms = 0.001 seconds)
    = 1,000 req/s          (max capacity of this worker)
  ρ = λ / μ = 800 / 1,000 = 0.8  (worker is busy 80% of the time)

Step 1: How long does each request ACTUALLY spend in the system?
  Formula (M/M/1 — Kendall's notation: M=random arrivals,
  M=random service times, 1=single worker):
  W = (1/μ) / (1 - ρ)
  W = 1ms / (1 - 0.8) = 1ms / 0.2 = 5ms

  The request takes 1ms to process, but spends 5ms total.
  Where did the other 4ms go? Waiting in the queue.

Step 2: How many requests are sitting in the queue?
  Formula (Little's Law): L = λ × W

  L_total = λ × W_total
         = 800 req/s × 5ms (from Step 1)
         = 800 × 0.005s = 4 requests in the system

  L_processing = λ × W_service
              = 800 req/s × 1ms (pure processing time, from Given)
              = 800 × 0.001s = 0.8 requests being processed (on average)

  L_queue = L_total - L_processing = 4 - 0.8 = 3.2 requests waiting

  → Most of the time, ~3 requests are queued doing NOTHING.
    Each request spends 4ms waiting and only 1ms being processed.

Example 2: "How many workers do I need?"

Setup: A task queue receives 500 image resize tasks/sec. Each takes 2 sec.

Given:
  λ = 500 tasks/sec       (arrival rate)
  W_service = 2 sec       (processing time per task)

Step 1: How many tasks are in-flight at any moment?
  Formula (Little's Law): L = λ × W
  L = 500 × 2 = 1,000 tasks in the system

  → Need at least 1,000 workers to keep up (one task per worker).

Step 2: What if you only have 200 workers?
  Processing rate = 200 workers / 2 sec = 100 tasks/sec
  Backlog growth = 500 (arriving) - 100 (processed) = 400 tasks/sec

  After 10 minutes (600s):
    Queue depth = 400 tasks/sec × 600s = 240,000 tasks queued

    A new task joining now must wait for all 240,000 tasks
    ahead of it to be processed first.
    Formula: wait = queue_depth / processing_rate
    Wait = 240,000 tasks / 100 tasks/sec = 2,400 sec (40 min!)

    (This is Little's Law rearranged: W = L / λ_effective)

  → System is unstable: queue grows without bound.

Example 3: "What happens when latency spikes?"

Setup: A service has 200 threads and gets 1,000 req/s.

Normal operation (service time = 10ms):
  Formula (Little's Law): L = λ × W
  L = 1,000 × 0.01s = 10 requests in flight
  Thread pool (200) is barely touched. Everything is fine.

Latency spike (service time jumps to 500ms — slow DB, GC pause):
  L = 1,000 × 0.5s = 500 requests in flight
  Thread pool (200) is exhausted. 300 requests/sec have no thread.
  → Those requests queue up or get rejected (503).
  → Queued requests add MORE load, making latency worse.
  → This is how a latency spike cascades into a full outage.

The Response Time Multiplier (Why Worker Utilization Matters)

From Examples 1–3, the pattern is: as worker utilization (ρ) rises, wait time grows non-linearly. ρ = λ/μ is the fraction of time a worker is busy processing (not "how full the queue is" — the queue grows as a consequence of high worker utilization). The exact formula:

Formula (M/M/1):
  ρ = λ / μ  (worker utilization = arrival rate / worker's processing rate)
  W = service_time × 1/(1 - ρ)

  The "1/(1-ρ)" is the response time multiplier — how many times
  SLOWER the request is compared to its pure processing time:

  ρ=50%  → 1/(1-0.5)  = 2×    (comfortable)
  ρ=60%  → 1/(1-0.6)  = 2.5×  (good target for production)
  ρ=70%  → 1/(1-0.7)  = 3.3×  (upper target)
  ρ=80%  → 1/(1-0.8)  = 5×    (starting to hurt — Example 1)
  ρ=90%  → 1/(1-0.9)  = 10×   (dangerous — small spike = collapse)
  ρ=95%  → 1/(1-0.95) = 20×   (broken)

Example (from Example 1): service_time = 1ms, ρ = 0.8
  W = 1ms × 5 = 5ms (request waits 5ms for a 1ms operation)

This is why 60–70% utilization is the sweet spot. The curve is a hyperbola — it doesn't degrade linearly, it explodes. At 90%, a 5% traffic bump pushes you to 95% and response time doubles from 10x to 20x.

Multiple Consumers (M/M/c)

Adding consumers has dramatically diminishing returns because parallelism reduces per-server ρ:

Example: λ=80/s, μ=100/s per consumer, service=10ms

  c=1: ρ=80%, Wq=40ms    (baseline)
  c=2: ρ=40%, Wq=2.7ms   (15× better)
  c=3: ρ=27%, Wq=0.4ms   (7× better)
  c=4: ρ=20%, Wq=0.06ms  (diminishing returns)

Variance matters: Real systems use M/G/c (general service time). Multiply M/M/c wait time by (1 + CS2) / 2 where CS = σ/mean. High variance (CS=2) gives 2.5× worse queuing than exponential. Consider separate queues for heavy tasks.

peak    = 3-10x average

Powers of 2 ↔ Human Units

The key insight: 2^10 ≈ 10^3, so every 10 bits ≈ 3 decimal digits. This lets you convert between machine units (bytes, always powers of 2) and human units (thousands, millions, billions) instantly.

Unit Conversion Ladder:

  1 B   = 1 byte
  1 KB  = 10^3 B    = 1,000 bytes
  1 MB  = 10^6 B    = 10^3 KB   = 1,000 KB
  1 GB  = 10^9 B    = 10^6 KB   = 10^3 MB  = 1,000 MB
  1 TB  = 10^12 B   = 10^9 KB   = 10^6 MB  = 10^3 GB
  1 PB  = 10^15 B   = 10^12 KB  = 10^9 MB  = 10^6 GB  = 10^3 TB

Quick mental math — each step up is ×10^3 (÷10^3 going down):
  bytes → KB: drop 3 zeros     (5,000,000 B = 5,000 KB = 5 MB)
  KB → MB:    drop 3 zeros
  MB → GB:    drop 3 zeros
  GB → TB:    drop 3 zeros

Powers of 2 ↔ Powers of 10:
  2^10 ~= 10^3  (1,024 ≈ 1K)     →  1 KB
  2^20 ~= 10^6  (1,048,576 ≈ 1M) →  1 MB
  2^30 ~= 10^9                    →  1 GB
  2^40 ~= 10^12                   →  1 TB
Example 1: "1 billion users × 1 KB each = how much storage?"
  1B × 1 KB = 10^9 × 10^3 bytes = 10^12 bytes = 1 TB

Example 2: "64 GB server, 1 KB sessions — how many fit?"
  64 GB = 64 × 10^9 bytes.  Each session = 10^3 bytes.
  64 × 10^9 / 10^3 = 64 × 10^6 = 64 million sessions.

Example 3: "100 million messages/day, 2 KB each — daily storage?"
  10^8 × 2 × 10^3 = 2 × 10^11 bytes
  = 200 × 10^9 bytes = 200 GB/day

Example 4: "1 billion requests/day — what QPS?"
  10^9 / 10^5 sec/day = 10,000 QPS
