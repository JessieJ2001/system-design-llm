Database Sharding

Horizontally partition your data across multiple database instances to scale beyond the limits of a single machine.

Intermediate · 25 min read

What Is It?

Database sharding is the practice of splitting a single logical database into multiple smaller, independent databases called shards. Each shard holds a distinct subset of the data, and together they represent the complete dataset. Unlike vertical scaling (buying a bigger machine), sharding is a form of horizontal partitioning — you add more machines rather than upgrading a single one.

The core motivation is simple: a single database server has hard limits on storage capacity, write throughput, and connection count. When your data grows to hundreds of gigabytes or your write rate exceeds what one server can handle, no amount of read replicas or caching will help — you need to split the data itself. Sharding distributes both the storage burden and the query load across multiple nodes, enabling near-linear write scaling.

Every shard is a fully functional database instance. A shard key (such as user_id or tenant_id) determines which shard holds a given row. A routing layer sits in front of the shards, inspecting the shard key in each query and forwarding it to the correct shard. When done well, most queries touch only a single shard, giving you the illusion of a single database while distributing the work across many.

How It Works

  1. Choose a shard key. Pick a column (or combination of columns) that will determine how rows are distributed. Good shard keys have high cardinality and even distribution — user_id is a classic choice. Bad shard keys cluster data unevenly (e.g., country where 60% of users are in one country).
  2. Apply a partitioning function. The shard key is passed through a function that maps it to a shard number. Common approaches:
    • Hash-based: shard = hash(key) % num_shards. Gives uniform distribution but makes range queries hard.
    • Range-based: Shard 0 holds keys A–F, shard 1 holds G–M, etc. Supports range queries but can create hot spots.
    • Directory-based: A lookup table maps each key (or key range) to a shard. Most flexible but the directory is a single point of failure.
  3. Route the query. A shard router (which may be a proxy like Vitess, a library in your application, or a database-native feature) reads the shard key from the query, applies the partitioning function, and sends the query to the correct shard.
  4. Execute within the shard. The shard processes the query using its local data. Because each shard is a regular database, it supports indexes, transactions, and joins — as long as all the data needed is on that shard.
  5. Handle cross-shard operations (when unavoidable). If a query needs data from multiple shards, the router issues the query to all relevant shards (scatter), collects and merges the results (gather). This is expensive and should be rare in a well-designed sharding scheme.

Shard Key Selection: A Deep Dive

The shard key is the single most important decision in your sharding design. A poorly chosen shard key leads to hot spots, cross-shard queries, and painful resharding. A well-chosen shard key makes the sharded system feel like a collection of independent databases — simple and fast.

A good shard key has three properties:

  1. High cardinality — many distinct values, so data spreads across shards. A key with only 5 unique values cannot be distributed across 8 shards.
  2. Even distribution — no single value dominates. If 60% of rows have the same shard key value, 60% of your data lands on one shard.
  3. Query alignment — your most common queries include the shard key, so they can be routed to a single shard without scatter-gather.

Here is a comparison of shard key candidates for an e-commerce system:

Shard KeyCardinalityDistributionRange QueriesRisk
user_id High (millions) Even (if hash-based) No cross-user queries possible on single shard Large sellers (power sellers) create hot shards — one seller with 10M products overloads their shard
order_id Very high (billions) Even No range queries on orders (order_id is random) Good for write distribution, but "find all orders for user X" becomes scatter-gather across all shards
country Low (~200) Highly skewed (60% of orders in top 5 countries) Good for geo-based queries ("all orders from Germany") Hot shards for popular countries. US/China shard handles 10x the load of an Iceland shard
(user_id, order_date) Very high Even Range queries within a user's orders (e.g., "user_42's orders from January") Compound key adds routing complexity. Application must include both fields in every query.

In this example, user_id is usually the best choice: it has high cardinality, even distribution when hashed, and aligns with the most common query pattern ("show me my orders"). The hot-seller risk can be mitigated with caching or by placing large sellers on dedicated shards.

order_id is a good choice if writes are the bottleneck and you don't need user-scoped queries (or you can tolerate scatter-gather for them). country is almost always a bad choice for a shard key because of its extremely low cardinality and skewed distribution — but it works well as a secondary partition for data residency requirements (e.g., "EU user data must stay in EU data centers").

The compound key (user_id, order_date) is the most sophisticated option. It gives you the benefits of user-based sharding (single-shard user queries) while also enabling efficient range scans within a user's data. The tradeoff is that every query must include both the user ID and a date range, adding complexity to the application layer.

When to Use / When NOT to Use

Use Sharding When

  • Data volume exceeds a single node. Your database is approaching the storage or IOPS limits of the largest available machine.
  • You need horizontal write scaling. Read replicas solve read bottlenecks, but writes still funnel through a single primary. Sharding distributes writes.
  • Multi-tenant isolation. You want each tenant's data on a separate shard for performance isolation, compliance, or data residency requirements.
  • Queries are naturally partitionable. Most queries include the shard key (e.g., all queries filter by user_id), so cross-shard operations are rare.

Don't Use Sharding When

  • Your data fits on one machine. If a single Postgres instance handles your load comfortably, sharding adds complexity for no gain. Most applications never need it.
  • You need heavy cross-shard joins. If your query patterns require joining data across different shard keys, sharding will make things worse, not better.
  • Read replicas solve the problem. If your bottleneck is read throughput, try read replicas first — they're far simpler than sharding.
  • Vertical scaling is still viable. Upgrading to a bigger machine with more RAM and faster storage is often cheaper and simpler than reshaping your entire data architecture.

Classic vs. Modern Approaches

Classic Sharding Strategies

Hash-Based Sharding

Hash-based sharding applies a hash function to the shard key and takes the modulus of the shard count: shard = hash(key) % num_shards. It distributes data uniformly but makes range queries expensive (you must scatter to all shards). Resharding requires redistributing most data because changing the modulus changes most key-to-shard mappings.

Let's walk through a concrete example. Suppose we have 4 shards and 10 user keys. We hash each key and compute hash(key) % 4 to determine placement:

The distribution looks good: Shard 0 has 3 keys, Shard 1 has 3 keys, Shard 2 has 2 keys, Shard 3 has 2 keys. Fairly balanced.

8 out of 10 keys changed shards. Adding one shard reshuffles ~80% of keys. This is why naive hash sharding makes scaling painful — nearly every key must be migrated to a different shard, which means copying the vast majority of your data. For a database with 1 billion rows, adding a single shard could mean moving 800 million rows. This is the fundamental limitation that consistent hashing was designed to solve, where adding a node moves only K/N keys (roughly 1/N of the total).

Range-Based Sharding

Range-based sharding assigns contiguous key ranges to shards. It supports efficient range scans but is prone to hot spots — if new users always get the highest IDs, the last shard absorbs all writes.

Consider a concrete example: a user database with IDs from 1 to 1,000,000, split across 4 shards:

ShardKey RangeInitial Row Count
Shard 0IDs 1 – 250,000250,000
Shard 1IDs 250,001 – 500,000250,000
Shard 2IDs 500,001 – 750,000250,000
Shard 3IDs 750,001 – 1,000,000250,000

This looks balanced initially. But there is a critical problem: if new users get sequential IDs, user_1,000,001 goes to Shard 3, then user_1,000,002 goes to Shard 3, and so on. Shard 3 absorbs ALL new writes. After 6 months of growth:

ShardKey RangeRow Count (6 months later)Write Rate
Shard 0IDs 1 – 250,000250,000 (unchanged, mostly churned users)~0 writes/sec
Shard 1IDs 250,001 – 500,000250,000 (unchanged)~0 writes/sec
Shard 2IDs 500,001 – 750,000250,000 (unchanged)~0 writes/sec
Shard 3IDs 750,001 – 2,500,0001,750,000 (70% of all data)ALL writes

Shard 3 has 70% of the data while Shards 0–1 are nearly empty. This is the hot spot problem with range-based sharding on monotonically increasing keys.

Boundary selection strategies help mitigate this:

  1. Equal-count splits (static). Analyze your current data distribution and split so each shard has an equal number of rows. For example, if 80% of your users have IDs below 500,000, you might set boundaries at 100K, 200K, 350K, and 500K to balance the load. The downside: as data grows, the splits become unbalanced again and you need to re-evaluate.
  2. Dynamic splitting (auto-split). When a shard exceeds a size threshold (e.g., 100 GB or 50 million rows), automatically split it into two shards at the median key. This is what HBase and CockroachDB do. The system continuously monitors shard sizes and splits/merges as needed. This adapts to changing data distributions without manual intervention, but requires infrastructure that supports automatic range management.

Directory-Based Sharding

Directory-based sharding uses an external lookup table (the directory) that maps each key or key range to a shard. It offers maximum flexibility for rebalancing but introduces a single point of failure and adds latency for every lookup.

Here is what the directory lookup table looks like:

KeyShard
user_1Shard 2
user_2Shard 0
user_3Shard 2
user_4Shard 1
user_5Shard 3
user_6Shard 0
......

Every read and write requires a lookup to the directory first: the application asks "which shard holds user_3?", the directory answers "Shard 2", and then the application sends the query to Shard 2. This adds one extra network round-trip to every operation.

The critical weakness: the directory is a single point of failure. If the directory service goes down, no key can be located and the entire sharded database is effectively offline. Mitigations:

  • Replicate the directory. Run the directory on a replicated cluster (e.g., a 3-node Raft group) so no single node failure takes it down.
  • Cache it client-side. Each application server caches the directory locally. Cache entries are invalidated when the directory changes (via pub/sub or polling). This eliminates the extra round-trip for cache hits, but introduces a staleness window where a stale cache entry sends a query to the wrong shard.
  • Use a distributed consensus store. Store the directory in etcd, ZooKeeper, or Consul — systems specifically designed for highly available, strongly consistent configuration data. These stores handle replication, leader election, and failure recovery automatically.

The upside of directory-based sharding is unmatched flexibility: you can move any individual key to any shard at any time by updating a single row in the directory. This makes rebalancing trivial compared to hash-based or range-based approaches, where moving data requires rehashing or splitting ranges.

Modern Approaches

Vitess (YouTube/Google). Originally built to shard MySQL for YouTube. Vitess sits between the application and MySQL, handling shard routing, connection pooling, and online resharding. Its VReplication engine copies data between shards in the background, allowing you to split or merge shards with zero downtime. Vitess popularized the concept of vindexes — pluggable shard-key functions that can be hash-based, functional, or lookup-based.

CockroachDB auto-sharding. CockroachDB uses a range-based partitioning scheme where the database automatically splits ranges when they grow too large and rebalances them across nodes. The application never thinks about shards — it connects to any node and the system routes queries internally using a distributed KV layer built on Raft consensus.

TiDB. TiDB splits data into fixed-size regions (~96MB) and distributes them across TiKV storage nodes. As data grows, regions split automatically. A Placement Driver (PD) tracks which region holds which key range and handles rebalancing.

Google Spanner. Spanner uses a hash-range hybrid: data is organized into "splits" (range partitions), and splits are distributed using a consistent hashing scheme. Spanner provides globally consistent reads using TrueTime (GPS + atomic clocks), enabling cross-shard transactions without the typical performance penalty.

How Auto-Resharding Works

Modern systems like Vitess, CockroachDB, and TiDB follow a common pattern for online resharding:

  1. Start replication. Begin copying data from the source shard(s) to the destination shard(s) in the background, while the source continues serving reads and writes.
  2. Catch up on changes. Apply the write-ahead log (WAL) or binlog from the source to the destination, replaying any changes that occurred during the copy.
  3. Cut over. Once the destination is caught up, briefly pause writes, update the routing table, and redirect traffic. The pause is typically sub-second.
  4. Clean up. Remove the old data from the source shard and update metadata.

Manual Resharding: The Double-Write Migration

When you don't have auto-resharding tooling (or need to change shard key strategies), you must perform a manual resharding migration. The safest approach is the double-write migration, which proceeds in 7 steps:

Virtual Shards: Future-Proofing Against Resharding Pain

The fundamental problem with resharding is that changing the number of shards changes the hash-to-shard mapping, requiring massive data movement. Virtual shards (also called logical shards) solve this by decoupling the hash space from the physical infrastructure.

Instead of 4 physical shards, create 256 logical shards mapped to 4 physical nodes (64 logical shards per physical node):

Logical ShardPhysical Node
0 – 63Node A
64 – 127Node B
128 – 191Node C
192 – 255Node D

The application always hashes to one of 256 logical shards: logical_shard = hash(key) % 256. This mapping never changes. When you need to scale to 8 physical nodes, you simply reassign logical shards:

Logical ShardNew Physical Node
0 – 31Node A
32 – 63Node E (new)
64 – 95Node B
96 – 127Node F (new)
128 – 159Node C
160 – 191Node G (new)
192 – 223Node D
224 – 255Node H (new)

No data rehashing is needed — just metadata remapping. You move 32 logical shards from each existing node to a new node, which is a straightforward data copy. Instagram uses this approach, encoding the logical shard ID directly into their auto-generated IDs. Each Instagram ID contains the logical shard number in its high bits, so any service can determine the shard for a record by simply inspecting the ID — no directory lookup required.

Assumptions & Model

Key Assumptions

  • Uniform shard key distribution. The sharding model assumes that the shard key values are uniformly distributed so each shard receives roughly equal data and traffic. If this breaks (e.g., a celebrity user generates 1000x the activity of a normal user), one shard becomes a hot spot.
  • Queries are single-shard routable. The model assumes most queries include the shard key and can be routed to a single shard. When this breaks (e.g., "find all orders across all users from last month"), the router must scatter-gather across every shard, which is O(N) in shard count.
  • Cross-shard operations are rare. Transactions spanning multiple shards require distributed coordination (2PC or sagas), which is dramatically slower than single-shard transactions. If your workload frequently needs cross-shard transactions, sharding may hurt more than it helps.
  • Shard key is immutable. The model assumes the shard key for a row doesn't change. If a user's shard key changes, the row must be moved between shards — an expensive and error-prone operation.

When assumptions break: Hot keys (the "celebrity problem") overload a single shard while others sit idle. Cross-shard joins force full scatter-gather queries that are slower than a single-node query would be. Uneven data growth causes some shards to fill up while others have plenty of headroom, requiring premature resharding.

Sizing & Tradeoffs

How Many Shards?

The fundamental formula is:

num_shards = ceil(total_data / max_data_per_shard)

But this data-only view misses the point. The right question is: how many shards do I need to meet my SLO? You must factor in latency targets, throughput requirements, growth rate, and operational headroom:

# SLO-driven shard sizing
max_qps_per_shard = max_qps_at_latency_target  # e.g., 3,000 QPS at p99 < 10ms
num_shards_for_throughput = ceil(total_qps / max_qps_per_shard)
num_shards_for_storage = ceil(projected_data / (target_utilization * max_per_shard))
num_shards = max(num_shards_for_throughput, num_shards_for_storage)

Where target_utilization is typically 0.6–0.7 to leave headroom for spikes and rebalancing, and max_qps_at_latency_target comes from load testing a single shard at your target percentile latency.

SLO-Driven Sizing Examples

SLO-Driven Example: E-Commerce Product Catalog

Target SLO: p99 query latency < 10ms for product lookups

Given: 50M products, average row size 2 KB (100 GB total), 20,000 peak QPS, 30% annual growth

Step 1 — Benchmark single shard: Load test a single PostgreSQL instance with realistic product data. At p99 < 10ms, a single shard sustains ~3,000 QPS when the working set fits in memory.

Step 2 — Throughput shards: 20,000 QPS / 3,000 QPS per shard = 6.7 → 7 shards minimum for throughput.

Step 3 — Storage shards: Year-2 projection: 100 GB × 1.3² = 169 GB. At 70% utilization with 50 GB per shard: 169 / 35 = 4.8 → 5 shards for storage.

Step 4 — Binding constraint: Throughput dominates. Round up to 8 shards (power of 2). Each shard holds ~12.5 GB (fits in RAM), serves ~2,500 QPS — well within the 3,000 QPS p99 envelope.

Result: 8 shards, keyed on product_id, with hash-based routing. Headroom supports growth to ~24,000 QPS before resharding to 16.

SLO-Driven Example: Social Media Posts

Target SLO: p95 read latency < 20ms for user timeline queries

Given: 1B posts, average 1.5 KB per post (1.5 TB total), 200,000 read QPS, timeline queries fetch 50 most recent posts per user

Step 1 — Choose shard key: Shard by author_user_id so a user's timeline can be served from a single shard (no scatter-gather). This keeps per-user reads local.

Step 2 — Benchmark single shard: At p95 < 20ms with an indexed query on (user_id, created_at DESC), a single shard handles ~4,000 QPS when hot data fits in buffer pool.

Step 3 — Throughput shards: 200,000 / 4,000 = 50 shards minimum.

Step 4 — Storage shards: 1.5 TB at 70% utilization on 100 GB shards: 1,500 / 70 = 21.4 → 22 shards for storage.

Step 5 — Hot-user skew buffer: Power-law follower distribution means top 1% of users generate ~20% of read traffic. Add 25% headroom: 50 × 1.25 = 62.5 → 64 shards (power of 2).

Result: 64 shards keyed on author_user_id. Celebrity users with extreme read amplification are additionally served via a read-through cache layer to protect their home shard.

SLO-Driven Example: Financial Transactions

Target SLO: p99 write latency < 5ms, zero data loss (synchronous replication)

Given: 100,000 TPS peak, average transaction row 500 bytes, must support ACID per-account operations

Step 1 — Benchmark with sync replication: With synchronous replication enabled (required for zero data loss), a single shard sustains ~5,000 TPS at p99 < 5ms on NVMe SSDs.

Step 2 — Throughput shards: 100,000 / 5,000 = 20 shards minimum.

Step 3 — Shard key selection: Shard by account_id so all transactions for one account are on the same shard, preserving single-shard ACID. Cross-account transfers use 2PC or saga pattern.

Step 4 — Compliance buffer: Financial regulations require 7-year retention. Year-7 projection: ~110 TB. At 70% utilization on 2 TB shards: 110,000 / 1,400 = 78 → need storage-based expansion plan.

Result: Start with 32 shards (throughput-driven with headroom). Each shard has 1 synchronous replica. Archive transactions older than 1 year to cold storage to manage per-shard size. Plan resharding to 64 at year 3.

SLO-Driven Example: Gaming Leaderboard

Target SLO: p99 read latency < 15ms for leaderboard queries, p99 write < 10ms for score updates

Given: 10M active players, 50,000 score updates/sec during peak, leaderboard queries return top 100 globally and top 100 per region

Step 1 — Identify the hot-shard problem: A naive global leaderboard on a single shard creates extreme write contention. Sorted set operations (Redis ZADD) sustain ~30,000 ops/sec on a single instance, but a global sort requires all data in one place.

Step 2 — Shard by region: Partition into 8 regional shards (NA-East, NA-West, EU-West, EU-East, Asia-East, Asia-South, SA, OCE). Per-region leaderboards are served from a single shard with no scatter-gather.

Step 3 — Regional throughput check: Largest region (NA-East) has ~25% of traffic: 12,500 writes/sec. Well within the 30,000 ops/sec limit at p99 < 10ms.

Step 4 — Global leaderboard strategy: A background aggregator merges regional top-1000 lists every 5 seconds into a cached global leaderboard. This avoids cross-shard queries on the hot path and the global view is acceptably stale (5s).

Result: 8 regional shards using Redis sorted sets. Global leaderboard is a materialized view refreshed asynchronously. Hot-shard mitigation via regional partitioning keeps each shard at ~40% capacity.

SLO-Driven Capacity Planning

Traditional shard sizing starts from data volume. SLO-driven sizing starts from the user experience and works backward. Follow these steps when planning shard capacity:

  1. Define your SLOs first. What latency percentile (p95, p99) and throughput must you guarantee? Write these down before touching any infrastructure.
  2. Benchmark a single shard under realistic load. Use production-like data volumes and query patterns. Measure the QPS at which your target percentile latency is breached.
  3. Identify the binding constraint. Compare throughput-driven shard count vs. storage-driven shard count. The larger number wins.
  4. Add skew and headroom buffers. If your access pattern has skew (power-law, time-based hot spots), add 20–30% headroom. If you have seasonal spikes, size for peak.
  5. Round to a power of 2. This simplifies future resharding — doubling from N to 2N splits each shard exactly in half.
  6. Validate with chaos testing. Kill one shard and verify the remaining shards still meet SLOs with the redistributed load. If they cannot, you are running too hot.
  7. Set resharding triggers. Define the metric thresholds (QPS per shard, p99 latency, storage utilization) that trigger a resharding operation, and automate the alert.

Scaling Journey

0 – 1K Users

Single Database

One PostgreSQL or MySQL instance handles everything. All tables in one database. Simple, easy to reason about, easy to back up. This handles more than most people think — a well-tuned Postgres can serve 10K+ QPS.

Signal to evolve: Write throughput hitting limits, or storage approaching single-machine capacity.

1K – 100K Users

Read Replicas

Add read replicas to offload SELECT queries from the primary. Application routes writes to the primary and reads to replicas. This 3–5x your read capacity cheaply. Consider connection pooling (PgBouncer, ProxySQL) to handle more concurrent connections.

Signal to evolve: Primary write throughput is the bottleneck, or data volume exceeds single-node storage.

100K – 10M Users

Vertical Partitioning + Sharding

First try vertical partitioning: move large tables (e.g., analytics, audit logs) to separate database instances. Then shard the remaining hot tables by a natural key (user_id, tenant_id). Start with a small number of shards (4–8) using hash-based routing.

Signal to evolve: Operational burden of manual sharding, need for auto-rebalancing.

10M+ Users

Auto-Sharding (Vitess / CockroachDB)

Adopt a sharding middleware like Vitess (for MySQL) or migrate to a natively distributed database like CockroachDB, TiDB, or Spanner. These handle shard routing, automatic rebalancing, and online resharding. Your application code no longer needs to be shard-aware.

Migration pitfall: Moving from manual sharding to Vitess requires careful schema review — Vitess imposes constraints on cross-shard queries and foreign keys.

What Can Go Wrong

Hot Shard from Viral Content

What happened: A social media platform sharded user data by user_id. A celebrity with millions of followers posted viral content. Every follower's timeline query hit the shard holding the celebrity's data, overwhelming it while other shards sat idle.

Root cause: The shard key (user_id) distributes data uniformly, but access patterns are not uniform. Popular users generate disproportionate read traffic on their shard.

Blast radius: All users whose data lived on the hot shard experienced elevated latency and timeouts. Users on other shards were unaffected.

Mitigation: Added a read-through cache (Redis) in front of the hot shard to absorb repeated reads. Longer term, denormalized celebrity data into a separate fast-path cache layer.

Prevention: Implement per-shard rate limiting. Use caching for read-heavy access patterns. Consider secondary sharding for high-fanout entities.

Accidental Cross-Shard Scatter-Gather

What happened: A developer wrote an analytics query that filtered by date range but did not include the shard key (tenant_id). The query router had no choice but to send the query to all 64 shards, each of which performed a full table scan. The query took 45 seconds and consumed significant resources on every shard simultaneously.

Root cause: The query router's fallback behavior was scatter-gather when no shard key was present, rather than rejecting the query.

Mitigation: Killed the query and added a query analysis layer that rejects queries without a shard key (or requires an explicit ALL_SHARDS hint).

Prevention: Enforce shard key presence in queries at the router level. Log and alert on scatter-gather queries. Provide a separate analytics replica or data warehouse for cross-shard queries.

Resharding Data Corruption

What happened: During a manual resharding operation (splitting 4 shards into 8), a race condition in the migration script caused some rows to be written to both the old and new shard. When the cutover happened, some rows had divergent data — the old shard had newer writes that hadn't been replicated to the new shard.

Root cause: The migration script didn't properly fence writes to the source shard during the final cutover phase. A brief window existed where writes went to the source after the destination was declared authoritative.

Mitigation: Rolled back to the original 4-shard configuration using the pre-migration backup. Re-ran the migration with a proper write-pause during cutover.

Prevention: Use battle-tested resharding tools (Vitess VReplication, gh-ost) rather than custom scripts. Always implement a write-fence or brief read-only period during cutover. Validate data checksums before and after.

    → Check: Is one key driving most traffic? (celebrity problem)
    → Fix: Add caching for hot keys, or split the hot shard
  → No: Check cross-shard query percentage → If high, queries are missing shard keys
    → Fix: Add shard key to query, or route to analytics replica

Symptom: One shard running out of storage

→ Check shard key distribution — is data skewed?
  → Yes: Re-evaluate shard key choice; consider composite keys
  → No: All shards growing → time to reshard (double shard count)

Interactions & Pitfalls

Sharding + Caching

When you shard your database, you typically run a per-shard cache (or a cache layer in front of all shards). The pitfall: per-shard caches have different hit rates if traffic is skewed. A hot shard's cache may thrash (constant evictions) while cold shard caches waste memory sitting mostly idle. Solution: use a shared distributed cache (Redis Cluster) rather than per-shard local caches, and monitor hit rates per shard independently.

Sharding + Transactions

Single-shard transactions work exactly like normal database transactions. Cross-shard transactions require distributed coordination — typically two-phase commit (2PC) or a saga pattern. 2PC adds a prepare phase that doubles latency and introduces a blocking failure mode (if the coordinator crashes after prepare but before commit). Sagas avoid the blocking problem but require compensating transactions for rollback, which can be complex to implement correctly. The key takeaway: design your shard key so that entities that transact together live on the same shard.

Sharding + Joins (Cross-Shard Queries)

SQL joins across shards are either impossible or extremely expensive (requiring the router to fetch data from multiple shards and perform the join in memory). The standard mitigation is denormalization: duplicate the data you need for joins into each shard. For example, if you shard orders by user_id but need to display product names alongside orders, store a copy of the product name in the order record. This trades storage for query simplicity.

Worked Example: The Scatter-Gather Problem

Consider this query: "Find all orders for user_42 where the product category is 'electronics'." If orders are sharded by order_id, then user_42's orders are scattered across all shards — there is no way to know which shard has user_42's orders without asking every single shard.

The router must use the scatter-gather pattern:

StepActionLatency
1Coordinator sends query to all 8 shards in parallel1ms (network fan-out)
2Each shard executes the local query (WHERE user_id = 42 AND category = 'electronics')5–50ms (varies by shard load and data size)
3Coordinator waits for ALL shard responsesmax(shard latencies) = 50ms
4Coordinator merges and sorts results2ms
Total53ms (dominated by the slowest shard)

The key insight: latency = max(all shard latencies), not average. One slow shard makes the entire query slow. With 8 shards, you are almost guaranteed to hit at least one shard at its p95 or p99 latency on every scatter-gather query. With 64 shards, this problem is dramatically worse — the probability that all 64 shards respond quickly approaches zero.

Denormalization to Avoid Cross-Shard Joins

To avoid the scatter-gather penalty, duplicate data so that queries can be answered from a single shard. For example, instead of having a separate users table and orders table that require a join:

ApproachQueryShards TouchedTradeoff
Normalized (orders + users join) SELECT o.*, u.name FROM orders o JOIN users u ON o.user_id = u.id 2+ shards (orders shard + users shard) No duplication, but cross-shard join is slow
Denormalized (user info embedded in orders) SELECT * FROM orders WHERE user_id = 42 1 shard Duplicated data (user name/email stored in orders table). Updates to user info must update both tables.

Denormalization trades storage and write complexity for read performance. When a user changes their name, you must update the name in the users table AND in every order row that contains a copy. This is an acceptable tradeoff when reads vastly outnumber writes (which is true for most applications — read:write ratios of 10:1 or 100:1 are common).

Sharding + Database Replication

Each shard typically has its own set of read replicas. This means you now have num_shards * replicas_per_shard database instances to manage. Replication lag becomes a per-shard concern — one shard's replicas might be current while another's lag behind. Monitor replication lag per shard, not just as an aggregate.

Cost Considerations

Cost Analysis

Dominant cost driver: Compute and storage scale linearly with shard count. N shards = N database instances (or 2N–3N including replicas). Each instance needs CPU, memory, and storage.

Approximate cloud costs (AWS RDS):

  • 4 shards (db.r6g.xlarge, 500GB each): ~$2,800/month compute + ~$460/month storage = ~$3,300/month
  • 16 shards (same instance type): ~$11,200/month compute + ~$1,840/month storage = ~$13,000/month
  • 64 shards with replicas (128 instances): ~$90,000/month — this is when you seriously evaluate CockroachDB Serverless or Aurora.

The hidden cost is operational complexity. More shards means more things to monitor, backup, patch, failover, and debug. A 64-shard cluster might need a dedicated database team. Factor in the engineering hours: a team of 2 spending 50% of their time on shard management at $200K/year = $200K/year in people cost on top of infrastructure.

Cost optimization tips:

  • Right-size shard instances — not all shards need the same instance type if you can identify hot vs. cold shards.
  • Use reserved instances or savings plans for predictable shard infrastructure (40–60% savings).
  • Archive cold data to cheaper storage (S3 + Athena) to reduce per-shard storage growth.
  • Consider managed sharding solutions (Vitess Cloud, CockroachDB Serverless) which handle operational overhead.

Cost of NOT sharding: If you've outgrown a single database, the alternative is vertical scaling — and the largest RDS instance (db.x2idn.24xlarge) costs ~$30,000/month. Sharding 4 ways on smaller instances is 10x cheaper and scales further.

Real-World Examples

YouTube / Vitess

YouTube built Vitess in 2010 to shard its MySQL database, which had grown beyond what a single instance could handle. Vitess introduced a proxy layer that routes queries to the correct shard, handles connection pooling (critical for MySQL's per-connection thread model), and enables online resharding via VReplication. Today Vitess is open source and used by Slack, GitHub, HubSpot, and many others. Key lesson: Vitess solved the operational burden of manual sharding by automating the hardest parts (routing, resharding, schema changes).

Instagram's PostgreSQL Sharding

Instagram sharded its PostgreSQL database by user_id using a custom sharding layer in their Django application. They used a logical shard ID embedded in every primary key (the shard ID is encoded in the high bits of a 64-bit ID, similar to Twitter's Snowflake). This made it possible to determine which physical shard holds a record just by looking at its ID, without a directory lookup. They ran thousands of logical shards mapped to a smaller number of physical databases, which made rebalancing easier — you move logical shards between physical servers without changing the application.

Pinterest's MySQL Sharding

Pinterest shards MySQL across hundreds of instances, using a custom sharding framework. Their key insight was keeping the shard-routing logic in the application layer (a Python library) rather than a proxy. They chose to shard by object_id with the shard number embedded in the ID itself. They deliberately avoided cross-shard joins entirely — all data needed for a single page render is fetched by multiple parallel single-shard queries, then assembled in the application. This made the system simpler to reason about at the cost of some denormalization.

Code Example: Shard Router

A minimal Python shard router that hashes a key and routes queries to the correct shard connection.

import hashlib

class ShardRouter:
    """Routes queries to the correct shard based on a hash of the shard key."""

    def __init__(self, shard_connections):
        # shard_connections: list of DB connection objects, one per shard
        self.shards = shard_connections
        self.num_shards = len(shard_connections)

    def _get_shard_index(self, shard_key):
        """Hash the key and map to a shard index."""
        key_bytes = str(shard_key).encode('utf-8')
        hash_val = int(hashlib.md5(key_bytes).hexdigest(), 16)
        return hash_val % self.num_shards

    def get_connection(self, shard_key):
        """Return the DB connection for the shard that owns this key."""
        idx = self._get_shard_index(shard_key)
        return self.shards[idx]

    def execute(self, shard_key, query, params=None):
        """Route a query to the correct shard and execute it."""
        conn = self.get_connection(shard_key)
        cursor = conn.cursor()
        cursor.execute(query, params or ())
        return cursor.fetchall()

    def scatter_gather(self, query, params=None):
        """Execute on ALL shards and merge results (expensive!)."""
        results = []
        for conn in self.shards:
            cursor = conn.cursor()
            cursor.execute(query, params or ())
            results.extend(cursor.fetchall())
        return results

How to Validate Your Design

Load Testing

Use k6 or Locust to generate realistic traffic against your sharded database. Key approach:

  • Generate traffic with a realistic shard key distribution (not uniform — use a Zipfian distribution to simulate hot keys).
  • Test each shard independently: measure p50/p95/p99 latency and QPS per shard.
  • Ramp up until you find the breaking point for each shard, then verify that the router correctly distributes load.
  • Measure cross-shard query latency separately — it should be significantly higher than single-shard queries.

Chaos Testing

Inject failures to validate resilience:

  • Kill a shard. Verify that only the users on that shard are affected, and that the rest of the system continues serving requests. Test failover to the shard's replica.
  • Simulate a hot key. Send 100x normal traffic for a single shard key. Verify that the hot shard degrades gracefully (rate limiting, circuit breaker) without affecting other shards.
  • Corrupt the routing table. Send queries to the wrong shard and verify that the system detects the mismatch (e.g., "row not found" rather than returning wrong data). Tools: toxiproxy to inject latency or errors between the router and specific shards.

Resharding Validation

  • Test resharding with shadow traffic: duplicate production read queries to the new shard configuration and compare results against the old configuration.
  • Verify data integrity with row-count checksums before and after resharding.
  • Practice the cutover procedure in a staging environment at least 3 times before doing it in production.
  • Measure the duration of the write-pause during cutover. It should be sub-second; if it's longer, your replication catch-up phase needs optimization.

Interview Questions

How would you shard a user database for a social network?

The natural shard key is user_id, since most queries are scoped to a single user (fetch profile, fetch feed, fetch messages). Use hash-based sharding (hash(user_id) % num_shards) for uniform distribution. Store the user's posts, likes, and followers on the same shard as the user to keep most operations single-shard.

The challenge is the social graph: when user A follows user B, and they're on different shards, fetching A's feed requires reading from B's shard. Two approaches: (1) fan-out on write — when B posts, push the post to a timeline table on every follower's shard; (2) fan-out on read — when A requests their feed, the router queries all shards holding A's followed users and merges results. Fan-out on write is faster for reads but expensive for users with millions of followers. Most systems use a hybrid: fan-out on write for normal users, fan-out on read for celebrities.

Follow-up: "What about direct messages between users on different shards?" — Store messages on both the sender's and recipient's shard (dual-write), or use a separate message service with its own sharding scheme.

What happens when you need to add more shards?

Adding shards (resharding) is one of the hardest operational challenges in a sharded system. With naive hash-based sharding (hash(key) % N), changing N redistributes nearly all keys. Doubling from N to 2N is the easiest case: each existing shard splits exactly in half, so only 50% of data moves.

The process: (1) Provision new shard instances. (2) Begin copying data from source shards to destination shards while the source continues serving traffic. (3) Apply ongoing writes from the source's WAL/binlog to the destination (catch-up replication). (4) Once caught up, briefly pause writes, update the routing table, and redirect traffic. (5) Validate and clean up old data.

Modern tools like Vitess VReplication automate this process. CockroachDB does it continuously and transparently. For manual sharding, the biggest risk is data loss during the cutover window — always use a write-fence and verify checksums.

Follow-up: "How do you minimize downtime during resharding?" — Use online resharding with background replication and a sub-second write pause at cutover. Vitess and gh-ost demonstrate this pattern.

How do you handle cross-shard queries?

There are three strategies, in order of preference:

1. Avoid them. Design the shard key so that queries are single-shard. If you shard by user_id, ensure all queries include user_id. For queries that can't include the shard key (e.g., admin dashboards), maintain a separate analytics replica or data warehouse (e.g., export to BigQuery or ClickHouse).

2. Scatter-gather. The router sends the query to all (or a subset of) shards in parallel, then merges results. This works for simple aggregations (COUNT, SUM) but is expensive for complex queries (ORDER BY with LIMIT, joins). Latency is dominated by the slowest shard (tail latency).

3. Global tables. For small, frequently-joined reference data (country codes, product categories), replicate the entire table to every shard. This allows local joins but requires a mechanism to keep the replicated data consistent (e.g., broadcast updates).

Follow-up: "What's the performance impact of scatter-gather?" — Latency = max(latency across all shards) + merge time. With 64 shards, you're likely to hit the p99 latency of at least one shard on every query. Scatter-gather should be rare (<5% of queries).

How would you design shard key selection for a multi-tenant SaaS?

tenant_id is the natural shard key for multi-tenant SaaS. It provides perfect query isolation (every query includes tenant_id), data isolation (a tenant's data lives entirely on one shard), and enables per-tenant resource controls. You can even offer premium tenants a dedicated shard for guaranteed performance.

The risk is tenant size skew: if one tenant has 100x the data of a typical tenant, their shard becomes a bottleneck. Mitigation strategies: (1) Large tenants get their own shard (or multiple shards with a secondary shard key like tenant_id + entity_id). (2) Use a directory-based approach where the tenant-to-shard mapping is explicit, so you can move tenants between shards as they grow. (3) Monitor per-tenant data size and set alerts when a tenant exceeds a threshold.

An alternative is compound shard keys (tenant_id + table_name or tenant_id + user_id), which spreads a single tenant across multiple shards. This helps with hot tenants but makes tenant-wide queries (billing reports, data export) require scatter-gather.

Follow-up: "How do you handle tenant onboarding and offboarding?" — For onboarding, assign the new tenant to the least-loaded shard. For offboarding, soft-delete the tenant's data, then run a background cleanup job.

How does Vitess handle resharding?

Vitess uses a feature called VReplication for online resharding. The process works in four phases:

Phase 1 (Copy): Vitess creates the new shard tablets and begins a bulk copy of existing data from source to destination shards using consistent snapshots. This runs in the background while the source shards continue serving traffic normally.

Phase 2 (Catch-up): While copying, Vitess also streams the MySQL binlog from the source shards and applies changes to the destination. This ensures the destination stays in sync with ongoing writes.

Phase 3 (Switchover): Once the destination is caught up (replication lag near zero), Vitess performs a "SwitchTraffic" operation. It first switches read traffic to the new shards, verifies correctness, then switches write traffic. The write switch involves a brief lock (typically <1 second) to ensure no writes are lost.

Phase 4 (Cleanup): After verifying everything works, Vitess drops the old shard data. The entire process can be rolled back at any point before cleanup.

Follow-up: "What if something goes wrong during the switch?" — Vitess supports "ReverseTraffic" to switch back to the original shards. The rollback is safe because VReplication also streams changes back from the new shards to the old ones during the transition.

When would you choose range sharding vs. hash sharding?

Choose hash sharding when: (1) You need uniform data distribution across shards. (2) Your queries are point lookups (fetch by exact key) rather than range scans. (3) You want to avoid hot spots from sequential key generation (auto-increment IDs, timestamps). Hash sharding is the default choice for most workloads.

Choose range sharding when: (1) You frequently run range queries (e.g., "all orders from January" when sharding by date). (2) You want to archive or drop old data easily (drop the oldest shard). (3) Your access patterns are time-based and you can tolerate the newest shard being hotter. (4) You need locality — related keys should be on the same shard for efficient scans.

Hybrid approach (Spanner-style): Use range-based partitioning on the primary key, but prefix the key with a hash of a high-cardinality column to distribute writes. For example, shard key = hash(user_id)[0:4] + timestamp. This spreads writes across shards while still allowing range scans within a user's data.

Follow-up: "What about directory-based sharding?" — Directory-based is best when you need maximum flexibility (e.g., per-tenant placement for compliance) and the overhead of a directory lookup is acceptable. It's common in multi-tenant SaaS where tenants need to be on specific shards for data residency.

Key Takeaways

  • Sharding is a last resort, not a first step. Exhaust vertical scaling, read replicas, and caching before introducing sharding complexity.
  • Shard key choice is the most important decision. A bad shard key leads to hot spots, cross-shard queries, and painful resharding. Optimize for the most common query pattern.
  • Design for single-shard queries. If most queries can be routed to a single shard, the system behaves like a collection of independent databases — simple and fast.
  • Cross-shard operations are expensive. Joins, transactions, and aggregations across shards are orders of magnitude slower than single-shard equivalents. Denormalize aggressively.
  • Use powers of 2 for shard counts. Doubling the shard count simplifies resharding because each shard splits exactly in half.
  • Modern tools reduce operational pain. Vitess, CockroachDB, and TiDB automate routing, rebalancing, and resharding. Don't build custom sharding infrastructure if you can avoid it.
  • Monitor per-shard metrics independently. Aggregate metrics hide hot shards. Track QPS, data size, latency, and replication lag per shard.
  • Plan for resharding from day one. Embed the shard ID in your primary keys (like Instagram) so resharding doesn't require rewriting IDs.
