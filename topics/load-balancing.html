Load Balancing

Distributing incoming network traffic across multiple servers to ensure no single machine becomes a bottleneck.

25 min read

What Is It?

A load balancer sits between clients and a pool of backend servers, forwarding each incoming request to one of the available servers. Its purpose is straightforward: spread the work so that no single server is overwhelmed while others sit idle. The result is higher throughput, lower latency, and better resilience—if one server crashes, the load balancer stops sending traffic to it and the remaining servers absorb the load.

Load balancers operate at two primary layers of the network stack. An L4 (transport-layer) load balancer routes traffic based on IP addresses and TCP/UDP port numbers without inspecting the payload. It is fast and efficient—think HAProxy in TCP mode or AWS Network Load Balancer. An L7 (application-layer) load balancer understands HTTP, can inspect headers, cookies, and URL paths, and can make routing decisions based on content—think Nginx, Envoy, or AWS Application Load Balancer. L7 gives you flexibility (route /api to one pool and /static to another) at the cost of higher per-request overhead.

Why does it matter? Without a load balancer, scaling beyond a single server means relying on DNS round-robin, which has no health awareness and caches stale records for minutes. A proper load balancer gives you health checking, connection draining, and the ability to do zero-downtime deploys—capabilities that are table stakes for any production system serving real users.

How It Works

A load-balanced request follows these steps from client to server and back:

  1. DNS Resolution. The client resolves the service hostname (e.g., api.example.com) to the load balancer's IP address. This IP is either a virtual IP (VIP) owned by the LB or an Anycast IP that routes to the nearest LB instance.
  2. Client Connects. The client opens a TCP connection (or sends a UDP packet) to the load balancer's IP and port. For L7, a TLS handshake may occur here—the LB terminates TLS, meaning it holds the certificate and decrypts traffic.
  3. LB Selects a Backend. The load balancer consults its algorithm (round-robin, least-connections, consistent hashing, etc.) and the current health status of backends to pick a target server.
  4. LB Forwards the Request. For L4, the LB rewrites packet headers (destination IP/port) and forwards at the TCP level. For L7, the LB establishes a separate TCP connection to the chosen backend and proxies the HTTP request, often adding headers like X-Forwarded-For.
  5. Backend Processes and Responds. The backend server handles the request and sends its response back to the load balancer.
  6. LB Returns the Response. The LB forwards the response back to the client over the original connection. The client has no idea which backend actually served the request.

When to Use / When NOT to Use

Use When

  • Horizontal scaling: You need more throughput than a single server can provide. Adding servers behind a load balancer is the standard horizontal scaling pattern.
  • High availability: You cannot tolerate a single point of failure. With multiple backends, one can go down without affecting users.
  • Zero-downtime deploys: Rolling deployments require draining connections from old servers while routing new traffic to updated ones. A load balancer makes this possible.
  • A/B testing and canary releases: L7 load balancers can route a percentage of traffic to a new version, enabling gradual rollouts.

Don't Use When

  • Single-server applications: If your app comfortably runs on one machine and you don't need redundancy, a load balancer adds unnecessary complexity and a new failure mode.
  • The bottleneck is the database, not compute: If your servers are underutilized but your database is saturated, adding more app servers behind a load balancer won't help. Fix the data layer first (caching, read replicas, sharding).
  • Sub-millisecond latency budgets: Every hop adds latency. L4 adds ~0.1ms; L7 can add 1-5ms. For ultra-low-latency systems (HFT, real-time control), direct connections may be necessary.
  • Peer-to-peer architectures: Systems where nodes communicate directly (gossip protocols, mesh networks) don't route through a central balancer.

Classic vs. Modern Approaches

Classic Algorithms

Round-Robin: Requests are distributed to servers in cyclic order: 1, 2, 3, 1, 2, 3... Simple, zero state, but ignores server load and request cost. A heavy query and a health check get equal treatment.

Weighted Round-Robin: Each server gets a weight proportional to its capacity. A server with weight 3 receives three times more requests than one with weight 1. Useful when servers have different hardware specs, but weights are static and don't adapt to runtime load.

Weighted Round-Robin: Smooth Distribution (Nginx-Style)

A naive implementation of weighted round-robin would send all of Server A's requests in a burst (AAAAABBBCC), causing uneven load spikes. Nginx solves this with a smooth weighted round-robin algorithm that interleaves servers proportionally. Here is how it works:

Each server has three values:

  • effective_weight — the configured weight (stays constant)
  • current_weight — a running counter that changes each round (starts at 0)
  • total_weight — the sum of all effective weights

Each round: (1) Add effective_weight to each server's current_weight. (2) Pick the server with the highest current_weight. (3) Subtract total_weight from the chosen server's current_weight.

Let's trace this with servers A (weight=5), B (weight=3), C (weight=2). Total weight = 10.

class SmoothWeightedRR:
    def __init__(self, servers_with_weights):
        """servers_with_weights: list of (name, weight) tuples."""
        self.servers = []
        for name, weight in servers_with_weights:
            self.servers.append({
                'name': name,
                'effective_weight': weight,
                'current_weight': 0,
            })
        self.total_weight = sum(s['effective_weight'] for s in self.servers)

    def next_server(self):
        # Step 1: Add effective_weight to each server's current_weight
        for s in self.servers:
            s['current_weight'] += s['effective_weight']

        # Step 2: Pick the server with the highest current_weight
        best = max(self.servers, key=lambda s: s['current_weight'])

        # Step 3: Subtract total_weight from chosen server
        best['current_weight'] -= self.total_weight

        return best['name']

Least Connections: Route each new request to the server with the fewest active connections. Better than round-robin for requests with variable processing times. However, it requires the LB to track per-server connection counts in real time, and it can oscillate (the "herd" effect where many requests pile onto a just-freed server).

If R1 had been a slow query, it would not receive new requests until it caught up. This is the key advantage over round-robin — least connections inherently reacts to variable request durations.

The herd problem: If many requests complete simultaneously on one server, that server's count drops to 0, and a burst of new requests all pile onto it. This is the "thundering herd" variant for least connections. Mitigations include adding randomized tie-breaking (when counts are equal, pick randomly instead of always picking the first) and combining with P2C (pick two random servers, then choose the one with fewer connections).

Modern Approaches

Power of Two Random Choices (P2C): Instead of checking all servers, pick two at random and send the request to the one with fewer connections. This simple change is transformative: Mitzenmacher proved that the maximum load drops from O(log n / log log n) with one random choice to O(log log n) with two. That's the difference between 5x imbalance and 2x imbalance at 1000 servers. P2C is used by Envoy, gRPC, and Nginx Plus. It requires minimal state (just two random lookups) and scales effortlessly.

Power of Two Choices: The Balls-into-Bins Analogy

Imagine you have N bins (servers) and N balls (requests), and you're dropping them in one at a time. The question is: how full does the most-loaded bin get?

  • 1 random choice: Each ball goes into a uniformly random bin. The most-loaded bin ends up with O(log n / log log n) balls. For n = 1,000, this means the heaviest bin has roughly 5–7 balls when the average is 1. Some bins are nearly empty while one is severely overloaded.
  • 2 random choices: Each ball inspects two random bins and goes into the one with fewer balls. The most-loaded bin now has only O(log log n) balls. For n = 1,000, the heaviest bin has roughly 2–3 balls. The overload nearly vanishes.

The insight is that the second choice creates a feedback mechanism: heavily loaded bins are more likely to be sampled (there are more of them), and when they are, they lose the comparison. This pushes load away from hot spots automatically.

Why not three choices? Azar et al. proved that going from 1 to 2 choices gives an exponential improvement, but going from 2 to 3 gives diminishing returns. The max load with d choices is O(log log n / log d), so three choices only shaves a constant factor. Two choices captures nearly all the benefit while keeping the implementation trivially simple.

Maglev Hashing (Google, 2016): Google's software load balancer uses a custom consistent hashing scheme that maps connections to backends via a lookup table. The table is constructed so that each backend gets an equal share, disruption from backend changes is minimal (only ~1% of connections remap), and lookup is O(1). Maglev handles 10M+ packets per second per machine. It's used for Google's front-end serving infrastructure.

Join-Idle-Queue (JIQ): Idle servers proactively advertise their availability to a dispatcher. When a request arrives, the dispatcher picks from the idle queue instantly—no need to probe or sample. JIQ achieves near-perfect load balance with zero queueing delay, but requires servers to push state to the dispatcher, which adds implementation complexity.

Join-Idle-Queue: How It Works Step by Step

JIQ inverts the traditional model. Instead of the dispatcher probing servers to find one that's available, servers tell the dispatcher when they're idle. This eliminates the querying overhead entirely.

The full flow:

  1. Server finishes a request and has no more work in its local queue.
  2. Server sends an "I'm idle" message to the dispatcher (load balancer).
  3. Dispatcher adds the server to an idle queue (a simple FIFO queue of servers with no work).
  4. New request arrives at the dispatcher.
  5. If the idle queue is non-empty: pop a server off the queue and send the request directly. This server starts processing immediately with zero queueing delay.
  6. If the idle queue is empty: all servers are busy. Fall back to a secondary algorithm (typically P2C) to pick the least loaded server. The request will queue on that server.

Compare this to P2C, where even with two random choices, requests might go to a server with an existing queue. JIQ achieves near-optimal dispatching because it has perfect knowledge of which servers are ready for work.

Tradeoff vs. P2C: JIQ requires servers to send "I'm idle" notifications to the dispatcher, which adds a message per request completion. In a system handling millions of requests per second, this is significant overhead. P2C requires no server-to-dispatcher communication at all — just two random picks and two load lookups. JIQ shines when processing times are long (seconds, not milliseconds), because the notification overhead is amortized over the request duration.

Consistent Hashing with Bounded Loads: A 2017 paper from Google combines consistent hashing (for session affinity) with a load cap. Each server has a maximum load of (1 + epsilon) * average_load. If a server is at its cap, the request "overflows" to the next server on the hash ring. This prevents hot spots while preserving the benefits of consistent hashing. Used in HAProxy and Vimeo's load balancers.

Consistent Hashing with Bounded Loads: Full Walkthrough

The problem it solves: Standard consistent hashing maps each key to a fixed server on the hash ring. This gives you session affinity (the same key always goes to the same server), but it has no protection against hot spots. If one key generates 100x more traffic than others, its server drowns while others are idle.

The idea: Add a load cap. Each server can handle at most (1 + ε) × (total_keys / num_servers) keys. When a server reaches its cap, additional keys "overflow" to the next server clockwise on the ring. The parameter ε controls the tradeoff: smaller ε means tighter balance but more keys displaced from their "natural" server; larger ε means more affinity but allows more imbalance.

Worked Example: 4 Servers, 12 Keys, ε = 0.25

Setup: 4 servers (S1–S4) placed on a hash ring. 12 keys to assign. Average load = 12 / 4 = 3 keys per server. Load cap = (1 + 0.25) × 3 = 3.75, so each server can hold at most 3 keys (since keys are integers, we floor the cap; some implementations round up to 4).

When a server is added or removed, the algorithm re-balances by overflowing keys from overloaded servers, but the bounded guarantee ensures no server ever gets more than (1 + ε) times the average.

L4 vs L7 Load Balancing: In Depth

To understand the difference between L4 and L7 load balancing, you need to understand what the load balancer can "see" at each layer. The OSI model defines how data is encapsulated: application data is wrapped in an HTTP message, which is wrapped in a TCP segment, which is wrapped in an IP packet. A lower-layer load balancer can only inspect outer layers — it cannot see inside the payload.

OSI LayerNameProtocol ExamplesInfo Available to LB
3NetworkIPSource/destination IP address
4TransportTCP, UDPSource/destination port, connection state (SYN, ACK, FIN), sequence numbers
7ApplicationHTTP, gRPC, WebSocket, SMTPHTTP method, URL path, headers, cookies, request body, gRPC service/method

What L4 Sees vs. What L7 Sees

When a client sends GET /api/users HTTP/1.1 with a cookie session_id=abc123, here's what each layer sees:

  • L4 (Transport) LB sees: source IP 203.0.113.50, destination port 443, TCP SYN flag. It does not see the URL path, the HTTP method, or the cookie. The HTTP request is encrypted inside the TLS payload, which is inside the TCP segment's data field. L4 treats this as opaque bytes.
  • L7 (Application) LB sees everything: the method (GET), the path (/api/users), all headers (Host, Cookie, Authorization), and even the request body. To get this visibility, the L7 LB must terminate the TCP connection and (if using TLS) decrypt the traffic. It then establishes a new TCP connection to the chosen backend and re-transmits the request.

When to Use Each

Use CaseL4L7Why
Raw throughput (millions of packets/sec)YesNoL4 is 10–100x faster because it doesn't parse application data
Non-HTTP protocols (database, game server, custom TCP)YesNoL7 only understands HTTP/gRPC. For raw TCP/UDP, use L4.
TLS passthrough (end-to-end encryption)YesNoL4 forwards encrypted bytes without decrypting. L7 must terminate TLS.
Path-based routing (/api vs /static)NoYesL4 can't see the URL path.

In practice, use both: Most production systems run a two-tier architecture. An L4 LB (AWS NLB, Maglev, IPVS) sits at the edge and distributes raw TCP connections across a pool of L7 LBs (Nginx, Envoy, HAProxy in HTTP mode). The L4 tier handles connection distribution, DDoS absorption, and Anycast routing. The L7 tier handles intelligent routing, TLS termination, header manipulation, and rate limiting. This is exactly how Google (Maglev → GFE), AWS (NLB → ALB), and GitHub (GLB Director → HAProxy) architect their stacks.

Health Checking: Active vs. Passive

Health checking determines whether a backend server is alive and able to serve requests. There are two fundamentally different approaches, and most production systems use both simultaneously.

Active vs. Passive: Comparison

False positivesLow: dedicated health endpoint can check app readiness, DB connectivity, etc.Higher: a burst of 5xx errors might be caused by bad input, not server failure.

Concrete Health Check Configuration

Typical production settings:

  • Interval: 10 seconds — the LB sends a probe every 10s.
  • Unhealthy threshold: 3 consecutive failures — a server must fail 3 checks in a row before being marked unhealthy. This prevents transient blips (a single dropped packet, a momentary GC pause) from causing premature removal.
  • Time to detect failure: 3 × 10s = 30 seconds. During these 30 seconds, the LB is still sending real traffic to the failing server. This is why passive checks complement active checks — passive detection can be much faster.
  • Healthy threshold: 2 consecutive successes — a server must pass 2 checks before being re-added. This prevents a flapping server (up-down-up-down) from rapidly toggling.
  • Time to recover: 2 × 10s = 20 seconds after the server starts passing checks again.

What Happens During a Rolling Deploy

Imagine you're deploying a new version to 4 servers (S1–S4) one at a time, with interval = 10s, unhealthy threshold = 3, healthy threshold = 2:

The critical window: Between t=30s and t=55s, only 3 of 4 servers are handling traffic. Each server carries 33% more load than normal. If this pushes any server past its capacity, latency spikes. This is exactly why SLO-driven sizing must account for rolling deploys (the "N-1" or "25% unavailable" scenario from the sizing examples above).

Best practice: Use shorter intervals (5s) with lower thresholds (2) for faster detection (10s), but pair with passive health checks for instant detection of real-traffic failures. Always configure connection draining timeout to match your longest expected request duration (typically 30–60s). And always test the full deploy cycle under load before relying on it in production.

Assumptions & Model

Key Assumptions

  • Homogeneous servers: Most balancing algorithms assume all backends have the same capacity. If servers differ in CPU, memory, or network bandwidth, you need weighted algorithms or the model breaks down—equal distribution does not mean equal utilization.
  • Independent requests: Each request is treated as independent and can be served by any backend. This breaks down with stateful applications (WebSocket connections, in-memory sessions). Sticky sessions partially fix this but sacrifice even distribution.
  • Health checks are accurate: The LB assumes health check responses reflect actual server health. If a server passes health checks but is degraded (e.g., high GC pressure, saturated thread pool), the LB will keep sending traffic to it. This is the "grey failure" problem.
  • Uniform network latency: The model assumes latency between the LB and all backends is roughly the same. In multi-AZ or cross-region deployments, this assumption fails and you need topology-aware routing.

When assumptions break down: Grey failures cause cascading load imbalance. Heterogeneous hardware with uniform weights leads to slow servers getting overwhelmed. Cross-AZ traffic adds 0.5-2ms per hop, which accumulates on high-fan-out requests.

Sizing & Tradeoffs

How Many Servers?

The basic formula:

servers_needed = ceil(peak_RPS / per_server_RPS) * (1 + headroom)

Where headroom is typically 0.3–0.5 (30–50% extra capacity) to handle traffic spikes and allow for individual server failures without overloading the remaining pool.

SLO-Driven Capacity Planning

The correct way to size a load-balanced fleet is to start from your SLO and work backward to the required backend count and LB configuration. Generic formulas like "add 40% headroom" are a starting point, but SLO-driven sizing tells you exactly how much capacity you need.

  1. Define the SLO: "p99 latency < X ms" or "availability ≥ 99.9%" or "throughput ≥ Y RPS sustained."
  2. Measure per-server capacity: Load-test a single backend to find its max RPS before the latency SLO is violated (not before it crashes — before the SLO breaks).
  3. Account for LB overhead: L4 adds ~0.1ms per hop; L7 adds 1–5ms. Subtract this from the backend's latency budget.
  4. Model failure scenarios: How many servers can fail simultaneously while keeping remaining servers under SLO? This gives your N+K redundancy requirement.
  5. Add spike headroom: What is your peak-to-average traffic ratio? Size so that peak traffic still meets the SLO with one server down.
  6. Validate with load testing: Run synthetic load at 1.2× expected peak through the actual LB to confirm the SLO holds end-to-end.

Worked Examples (SLO-Driven)

📐 SLO-Driven Example: E-Commerce Checkout

Target SLO: p99 latency < 200ms for checkout API calls

Given: Peak traffic = 30,000 RPS during flash sales. Each backend (4 vCPU, 8 GB) handles 4,000 RPS before p99 exceeds 150ms. L7 ALB adds ~5ms overhead. Must survive 2 simultaneous server failures.

Step 1 — Latency budget: 200ms SLO − 5ms LB overhead = 195ms backend budget. At 4,000 RPS per server, p99 is 150ms — within budget (45ms margin).

Step 2 — Base server count: ⌈30,000 / 4,000⌉ = 8 servers minimum.

Step 3 — Failure tolerance (N+2): With 2 servers down, remaining 6 would each handle 5,000 RPS. At 5,000 RPS, p99 = 220ms — exceeds our 195ms budget. Need more servers: with 10 servers, losing 2 leaves 8, each handling 3,750 RPS — p99 stays ~130ms. Safe.

Step 4 — Spike headroom: Flash sale traffic can spike 1.5× to 45,000 RPS. With 10 servers: 4,500 RPS each — approaching the 4,000 RPS limit. With 12 servers: 3,750 RPS each even during spike, and losing 2 gives 4,500 RPS on remaining 10 — still under the budget.

Result: 12 backend servers behind the ALB. Monthly compute cost: 12 × ~$140 (c5.xlarge) = $1,680 + ~$200 ALB = $1,880/month.

📐 SLO-Driven Example: API Gateway (High Throughput)

Target SLO: Sustained throughput ≥ 200,000 RPS with p50 latency < 10ms

Given: API gateway layer (Envoy sidecars) fronting 50 microservices. Each gateway instance (8 vCPU) handles 25,000 RPS at p50 = 3ms. L4 NLB adds ~0.1ms. Must maintain SLO during rolling deploys (25% of instances unavailable).

Step 1 — Latency budget: 10ms SLO − 0.1ms NLB = 9.9ms per gateway. At 25,000 RPS, p50 = 3ms — well within budget.

Step 2 — Base count: ⌈200,000 / 25,000⌉ = 8 gateway instances.

Step 3 — Rolling deploy tolerance: During deploy, 25% are draining. With 8 instances, only 6 serve traffic: 33,333 RPS each. At that load, p50 rises to ~8ms — barely under SLO. Too risky.

Step 4 — Safe count: With 12 instances, deploy drains 3, leaving 9: 22,222 RPS each, p50 = 2.5ms. Comfortable margin. Even a 1.3× traffic spike during deploy (260K RPS across 9 instances = 28,889 each) keeps p50 at ~5ms.

Result: 12 gateway instances behind NLB. At $0.085/hr (c6gn.2xlarge): 12 × $62/month = $744/month + ~$150 NLB = $894/month.

📐 SLO-Driven Example: Microservices Mesh (Availability)

Target SLO: 99.95% availability (max 21.9 minutes downtime/month)

Given: Internal service mesh with 3 critical services chained (A → B → C). Each service has 5 instances behind an internal L7 LB. Individual server uptime = 99.9% (8.7 hours downtime/year). Health check interval = 10s, 3 failures to mark unhealthy = 30s detection time.

Step 1 — Per-service availability with LB: With 5 instances at 99.9% individual uptime, probability all 5 are down = (0.001)5 ≈ 0. But the real risk is capacity loss. Losing 2 of 5 servers means remaining 3 handle 1.67× load — may violate latency SLOs.

Step 2 — Effective availability: P(2+ failures in a 1-hour window with 5 servers at 99.9%) ≈ 0.001%. But with 30s detection delay, each failure causes 30s of requests to a dead server — ~0.2% error rate per failure event. With ~2 failures/month per server, that is 10 failure events × 30s = 5 minutes of degraded service.

Step 3 — Chain availability: Three chained services: 99.95%3 = 99.85% — below our 99.95% target.

Step 4 — Improve: Reduce health check detection to 10s (interval=5s, threshold=2). Add retries at LB layer (retry on 503, max 1 retry). With retries, a single server failure is masked. Increase to 6 instances per service to tolerate 2 failures without latency impact. Per-service availability becomes ~99.99%, chain = 99.97%.

Result: 6 instances per service × 3 services = 18 total instances, with fast health checks (5s interval, 2-failure threshold) and 1 automatic retry on 503.

📐 SLO-Driven Example: Video Streaming CDN Origin

Target SLO: p95 time-to-first-byte < 100ms, sustained bandwidth ≥ 80 Gbps

Given: CDN origin servers serving video segments. Each origin server (16 vCPU, 64 GB RAM, 10 Gbps NIC) can saturate its NIC at ~9.5 Gbps with p95 TTFB of 40ms. L4 NLB adds ~0.1ms. Peak concurrent viewers = 500,000 with average bitrate 4 Mbps.

Step 1 — Bandwidth requirement: 500,000 viewers × 4 Mbps = 2 Tbps total, but CDN edge caches absorb 97.5%. Origin traffic = 2 Tbps × 2.5% = 50 Gbps steady, peaking at 80 Gbps during cache misses on new content.

Step 2 — Base count: ⌈80 Gbps / 9.5 Gbps⌉ = 9 origin servers.

Step 3 — Latency check: At 9 servers handling 80 Gbps, each runs at ~94% NIC capacity. At that load, p95 TTFB rises to ~85ms. With NLB overhead: 85.1ms — under 100ms but no margin.

Step 4 — Headroom: With 12 servers, each handles 6.67 Gbps (70% NIC utilization), p95 TTFB = 45ms. Losing 2 servers leaves 10 at 8 Gbps (84% utilization), TTFB = 75ms. Still under SLO.

Result: 12 origin servers across 2 availability zones (6 per AZ). At $0.45/hr (m5n.4xlarge): 12 × $328/month = $3,936/month.

Scaling Journey

0 – 1K users

Single Server

Everything on one machine. No load balancer needed. Your app, database, and static files are all on the same box. When this server goes down, your site is down. This is fine for prototypes and side projects.

Signal to evolve: You need uptime guarantees, or your single server is maxing out CPU during traffic spikes.

1K – 100K users

DNS Round-Robin / Simple LB

Add 2-3 servers and use DNS round-robin or a simple L4 load balancer (HAProxy, AWS NLB). Health checks remove dead servers from rotation. TLS termination stays on the app servers or moves to the LB.

Signal to evolve: You need path-based routing, A/B testing, or canary deployments. DNS round-robin's lack of health awareness is causing user-facing errors.

100K – 10M users

L7 LB with Routing Rules

Deploy Nginx or an ALB. Route /api to your API fleet, /static to CDN origin servers, /ws to WebSocket servers. Add rate limiting at the LB layer. Use connection draining for zero-downtime deploys. The LB itself is redundant (active-passive or active-active pair).

Signal to evolve: Users in multiple regions experience high latency. A single LB region is a SPOF for global availability.

10M+ users

Global LB (Anycast + Geo)

Deploy load balancers in multiple regions. Use Anycast IP addressing so clients automatically connect to the nearest LB. A global traffic manager (AWS Global Accelerator, GCP Traffic Director, Cloudflare) routes by geography and health. Each regional LB fans out to local backend pools. You now have a multi-tier LB architecture: global tier (geo routing) → regional tier (L7 routing) → per-service tier (P2C or least-connections).

What Can Go Wrong

1. Health Check Misconfiguration: All Backends Marked Unhealthy

What happened: A team deploys a new version that changes the health check endpoint from /healthz to /health, but the LB configuration still checks /healthz. All backends return 404. The LB marks every server as unhealthy and returns 503 to all clients.

Blast radius: Complete outage. 100% of traffic gets errors.

Detection: Alert on active healthy backend count dropping to zero. This should fire within one health check interval (typically 10-30 seconds).

Mitigation: Most LBs have a "panic mode" or "all-unhealthy-is-healthy" fallback: if every backend fails health checks, send traffic to all of them anyway. HAProxy calls this option allbackups. AWS ALB does this automatically. Configure it proactively.

Prevention: Include the health check endpoint path in your infrastructure-as-code alongside the application deployment. Test the health check endpoint in CI before deploying.

2. Thundering Herd on Backend Recovery

What happened: A backend server restarts after a crash. Its health check passes. The LB immediately sends the full share of traffic to it. The server has cold caches, uninitialized connection pools, and un-JIT'd code. It immediately gets overwhelmed and fails again, creating a crash loop.

Root cause: No warm-up period. The LB treats a just-recovered server identically to a long-running one.

Mitigation: Use "slow start" mode (supported by HAProxy, Nginx, and ALB). When a server first passes health checks, the LB gradually increases its traffic share over 30-60 seconds instead of sending the full load immediately.

3. Uneven Load from Consistent Hashing with Skewed Keys

What happened: A social media platform uses consistent hashing on user_id to route requests, ensuring each user's requests go to the same server (for in-memory session state). A celebrity with 50M followers goes live. All requests for that user hash to a single backend, which gets 100x more traffic than any other. The server saturates while others are idle.

Root cause: Consistent hashing distributes keys uniformly, not load. When some keys generate dramatically more traffic than others, the assumption of uniform load per key breaks down.

Mitigation: Use consistent hashing with bounded loads (cap each server at 1.25x average). Alternatively, detect hot keys and replicate them across multiple servers, using a secondary balancing algorithm for hot-key traffic.

Interactions & Pitfalls

Load Balancer + Circuit Breakers

If your services use circuit breakers (e.g., Hystrix, resilience4j) but the load balancer is unaware of their state, the LB will keep routing traffic to a backend whose circuit breaker to a downstream dependency is open. The backend will fast-fail every request, returning errors. The LB sees these as application-level errors and doesn't consider the server unhealthy (the server itself is fine; it's the dependency that's broken).

Solution: Have the backend's health check reflect circuit breaker state. If a critical circuit breaker is open, the health check should return unhealthy or degraded. Alternatively, use a service mesh (Istio, Linkerd) where the sidecar proxy integrates outlier detection with routing.

Load Balancer + Session Stickiness vs. Stateless Design

Sticky sessions (routing all requests from the same user to the same backend) seem convenient but create several problems: uneven load distribution (some users generate far more traffic), inability to scale down (draining a sticky server means waiting for all its sessions to expire), and loss of sessions on server failure. The better approach is stateless backends with externalized session storage (Redis, DynamoDB). The load balancer can then use any algorithm freely.

If you must use stickiness: Use cookie-based stickiness (the LB sets a cookie with the backend ID) rather than IP-based (which fails behind NATs and CDNs). Set a short TTL (5-15 minutes) so sessions naturally rebalance.

Load Balancer + Autoscaling Lag

When traffic spikes, the autoscaler launches new servers. But there's a lag: VM boot (30-60s), container start (5-15s), application warm-up (10-30s), health check passes (10-30s). During this 1-2 minute window, the existing servers must absorb the entire spike. If the spike is steep enough, servers can fail before new ones come online, causing a cascading failure.

Mitigation: Over-provision by 30-50% baseline. Use predictive autoscaling (scale up before the predicted spike based on historical patterns). Pre-warm instances with synthetic traffic before adding them to the LB pool.

Cost Considerations

Cloud Cost Breakdown

AWS ALB: ~$22/month base + $8 per LCU-hour. An LCU (Load Balancer Capacity Unit) measures new connections, active connections, bandwidth, and rule evaluations. A service handling 10K RPS typically costs $150-300/month for the ALB alone.

AWS NLB: ~$22/month base + $4.50 per NLCU-hour. Cheaper than ALB for pure TCP traffic since it processes at L4. A high-throughput service doing 100K connections/sec costs ~$500-800/month.

Self-hosted HAProxy/Nginx: Runs on a $50-200/month VM pair (active-passive). At scale, this is dramatically cheaper than managed load balancers. The tradeoff is operational burden: you manage updates, monitoring, failover, and TLS certificate rotation yourself. This makes sense when your LB bill exceeds ~$1,000/month.

Dominant cost driver: For managed LBs, it's the per-connection and per-rule charges, not bandwidth. Reducing connection count (via HTTP/2 multiplexing, connection pooling) directly reduces LB cost.

Cost of NOT having a load balancer: Without one, you can't scale horizontally. Vertical scaling hits a ceiling quickly: a 96-vCPU instance costs 10x more than twelve 8-vCPU instances with equivalent total compute. You also pay in reliability—a single server failure means total downtime.

Real-World Examples

Google Maglev

Google's Maglev is a distributed software load balancer that runs on commodity Linux servers. Each Maglev machine can handle over 10 million packets per second. It uses a custom consistent hashing algorithm (Maglev hashing) that distributes connections evenly and minimizes disruption when backends are added or removed. Maglev sits at the edge of Google's network and is the first thing every request to Google services hits. It's described in a 2016 NSDI paper and handles traffic for Search, YouTube, Gmail, and all GCP services.

Netflix Zuul + Eureka

Netflix uses Zuul as an L7 edge proxy and Eureka as a service registry. Zuul handles authentication, rate limiting, and dynamic routing. Eureka provides service discovery: backends register themselves and Zuul discovers available instances via Eureka's REST API. When a new server starts, it registers with Eureka; when it shuts down, it deregisters. Zuul periodically fetches the registry and updates its routing table. This architecture handles Netflix's entire API traffic—over 2 billion edge requests per day.

GitHub's GLB (GitHub Load Balancer)

GitHub built GLB, a custom L4 load balancer using DPDK (Data Plane Development Kit) for kernel-bypass packet processing. GLB uses Maglev-style consistent hashing to distribute TCP connections to backend servers. It replaced their previous HAProxy setup when they needed to handle more connections with lower latency. GLB is open-sourced and documented in GitHub's engineering blog. A key design choice: GLB uses a "GLB Director" tier that does consistent hashing, followed by backend HAProxy instances that do L7 routing—a two-tier architecture separating L4 and L7 concerns.

Code Example

Two load balancing implementations in Python: basic round-robin and power-of-two-choices. Notice how P2C achieves much better load distribution with minimal additional complexity.

Round-Robin Load Balancer

class RoundRobinLB:
    def __init__(self, servers):
        self.servers = servers
        self.index = 0

    def next_server(self):
        server = self.servers[self.index]
        self.index = (self.index + 1) % len(self.servers)
        return server

# Usage
lb = RoundRobinLB(["server-1", "server-2", "server-3"])
for _ in range(6):
    print(lb.next_server())
# Output: server-1, server-2, server-3, server-1, server-2, server-3

Power of Two Choices

import random

class P2CLB:
    def __init__(self, servers):
        self.servers = servers
        self.connections = {s: 0 for s in servers}

    def next_server(self):
        # Pick two servers at random
        a, b = random.sample(self.servers, 2)
        # Send to the one with fewer active connections
        choice = a if self.connections[a] <= self.connections[b] else b
        self.connections[choice] += 1
        return choice

    def release(self, server):
        self.connections[server] = max(0, self.connections[server] - 1)

# Simulate: 10 servers, 1000 requests with variable duration
lb = P2CLB([f"server-{i}" for i in range(10)])
for _ in range(1000):
    server = lb.next_server()
    # Simulate some completing (randomly release connections)
    if random.random() > 0.3:
        lb.release(random.choice(lb.servers))

# Check distribution — P2C keeps max load close to average
print("Connections:", dict(lb.connections))

How to Validate Your Design

Load Testing

Use vegeta or wrk to generate realistic HTTP load. Key measurements: p50/p95/p99 latency, throughput (RPS), and error rate. Gradually increase load until you find the breaking point of your LB + backend pool.

# vegeta: send 10K RPS for 60 seconds
# echo "GET http://your-lb:80/api/health" | vegeta attack -rate=10000 -duration=60s | vegeta report

# wrk: 100 connections, 4 threads, 30 second test
# wrk -c 100 -t 4 -d 30s http://your-lb:80/api/health

While running the test, monitor per-backend metrics. Verify that load is distributed evenly (within 10-15% variance for round-robin, within 5% for P2C).

Chaos Testing

  • Kill a backend server mid-test. Verify the LB detects the failure within one health check interval and stops routing traffic to it. Verify in-flight requests are retried (for idempotent endpoints) or return clean errors.
  • Slow down a backend using toxiproxy (add 2 seconds of latency). Verify the LB detects the slow server and shifts traffic away (if using least-connections or P2C). Round-robin won't react—this is one of its weaknesses.
  • Restart all backends simultaneously. Verify panic mode activates and the LB doesn't drop all traffic. Then verify slow-start kicks in as backends recover.

Benchmarking Methodology

Common mistakes to avoid:

  • Testing from the same machine as the LB. Loopback traffic doesn't exercise network stack. Use a separate client machine (or fleet).
  • Ignoring connection reuse. Testing with new TCP connections per request (no keepalive) measures connection setup overhead, not routing performance. Test both modes.
  • Not warming up. The first 30 seconds of a benchmark hit cold JIT, cold caches, and unfilled connection pools. Discard the warm-up period from results.
  • Not sweeping concurrency. Run tests at 10, 100, 500, 1000, and 5000 concurrent connections. The LB may behave differently at each level (e.g., hitting file descriptor limits at high concurrency).

Interview Questions

How would you design a load balancer for a real-time gaming platform?

Real-time games require persistent connections (WebSockets or UDP), sub-50ms latency, and session affinity (a player must stay connected to the same game server for the duration of a match). This rules out stateless round-robin.

Approach: Use an L4 load balancer (NLB or Maglev-style) for the initial connection, with consistent hashing on a match_id or session_id to route players in the same game to the same server. For matchmaking, a separate service assigns players to game servers and returns the server's direct address—subsequent communication bypasses the LB entirely (direct server connection). Health checks should be aggressive (1-2 second intervals) since a stale game server causes visible lag.

Key considerations: Connection draining is critical—when shutting down a server, you must wait for all active matches to complete (which could be 30+ minutes). Capacity planning must account for per-server player limits (not just RPS). Geographic routing is essential to minimize latency—players should connect to the nearest regional server.

Follow-ups: How do you handle server crashes mid-game? (Game state checkpointing + reconnection protocol.) How do you handle global matchmaking where players from different regions play together? (Pick the server geographically closest to the centroid of all players, or use relay servers.)

Explain the "power of two choices" and why it works so well.

With one random choice, if you throw n balls into n bins, the most loaded bin has O(log n / log log n) balls with high probability. This is the classic "birthday problem" intuition—collisions are inevitable and some bins get disproportionately loaded.

The power of two choices says: instead of picking one random bin, pick two and place the ball in the less loaded one. This tiny change drops the maximum load to O(log log n)—an exponential improvement. For 1,000 servers, the max load drops from ~7x average to ~3x average. For 1,000,000 servers, from ~10x to ~4x. The result was proven by Mitzenmacher (1996) and Azar et al. (1999).

Why it works: The second choice creates a feedback loop. Overloaded servers are more likely to be sampled (because there are more of them, probability-wise), and when they are, they lose the comparison. This naturally pushes traffic away from hot spots. The beautiful part is that it requires almost no state—just two random number generations and two load lookups per request.

Follow-ups: Does choosing three or more options help further? (Yes, but with severely diminishing returns. Two choices capture almost all the benefit.) How is this implemented in practice? (Envoy uses P2C with a per-host active request count. gRPC uses P2C as its default balancing policy.)

L4 vs L7 load balancing — when to use which?

L4 (Transport Layer): Operates on IP addresses and TCP/UDP ports. Doesn't inspect packet contents. Extremely fast (millions of packets per second). Use L4 when: you need maximum throughput and minimum latency, you're balancing non-HTTP protocols (databases, game servers, custom TCP), or you're building the first tier of a multi-tier LB architecture.

L7 (Application Layer): Understands HTTP. Can inspect headers, cookies, URL paths, and even request bodies. Can make content-based routing decisions. Use L7 when: you need path-based routing (/api vs /web), header-based routing (A/B tests via cookie), TLS termination, request/response transformation, rate limiting per endpoint, or WebSocket upgrade handling.

In practice: Most production architectures use both. An L4 LB (NLB, Maglev) sits at the edge and distributes TCP connections to a pool of L7 LBs (Nginx, Envoy, ALB). The L4 tier handles raw connection distribution and DDoS absorption; the L7 tier handles intelligent routing. This is exactly how Google (Maglev → GFE), AWS (NLB → ALB), and GitHub (GLB → HAProxy) architect their stacks.

Follow-ups: What about L3 (network layer) load balancing? (Used for Anycast/ECMP routing at the BGP level, even lower overhead than L4.) Can you do TLS termination at L4? (No, because TLS requires understanding the application layer. L4 can do TLS passthrough, where the backend terminates TLS.)

How do you handle sticky sessions without sacrificing scalability?

The ideal solution is to eliminate the need for sticky sessions entirely by making backends stateless. Move session state to an external store (Redis, Memcached, DynamoDB). Each request includes a session token (JWT or session ID), and any backend can look up the session from the shared store. This gives you full flexibility in load balancing and makes scaling, draining, and failover trivial.

If stickiness is required (e.g., WebSocket connections, in-memory caches that are expensive to rebuild per-request): Use cookie-based affinity. The LB sets a cookie containing the backend server ID on the first response. Subsequent requests from the same client include this cookie, and the LB routes to the specified backend. If that backend is unavailable, the LB routes to another server and sets a new cookie.

Consistent hashing is a middle ground: Hash the session key to a backend. This naturally provides stickiness without explicit cookies, and when a server is added/removed, only ~1/N of sessions are disrupted. Combine with bounded loads to prevent hot spots.

Follow-ups: What happens to sticky sessions during a rolling deploy? (Sessions migrate to new instances as old ones drain. If session state is in-memory, users may need to re-authenticate.) How does this interact with autoscaling? (New servers start with zero sticky sessions; old servers may be stuck with too many. Use TTL-based session expiry to naturally rebalance.)

Your load balancer is adding 50ms of latency. How do you diagnose and fix this?

Start by isolating where the latency is introduced. Measure three things: (1) client-to-LB latency, (2) LB processing time, (3) LB-to-backend latency.

If the LB processing time is high: The LB CPU may be saturated. Check CPU utilization. L7 LBs doing TLS termination, header parsing, and regex-based routing rules are CPU-intensive. Solutions: upgrade the LB instance, add more LB instances behind a DNS/L4 tier, offload TLS to dedicated hardware (or use TLS 1.3 with 0-RTT), simplify routing rules (a regex match per request is expensive at scale).

If LB-to-backend latency is high: Check if the LB and backends are in the same availability zone. Cross-AZ traffic adds 0.5-2ms per hop. Also check if the LB is creating new TCP connections for every request instead of reusing keepalive connections (look at upstream keepalive settings in Nginx). Connection pooling alone can cut 5-20ms per request.

If client-to-LB latency is high: This is network/geographic latency, not an LB issue. Solutions: deploy LBs in regions closer to users, use a CDN for static content, enable HTTP/2 to multiplex requests over a single connection (reducing round trips).

Follow-ups: Could the LB's connection queue be full? (Yes—check listen backlog settings and SYN queue overflow metrics.) What about GC pauses on the LB? (If using a JVM-based LB like Zuul, yes. Prefer C/Rust/Go-based LBs like Envoy, HAProxy, or Nginx for predictable latency.)

Design a global load balancing strategy for a service with users on 6 continents.

Use a three-tier architecture: (1) DNS-based geo-routing at the top, (2) regional L4 load balancers in each PoP, (3) per-service L7 load balancers within each region.

Tier 1 — Global DNS: Use a managed DNS service (Route 53, Cloud DNS) with latency-based or geolocation routing. When a user in Tokyo resolves api.example.com, DNS returns the IP of the Tokyo PoP. For better precision, use Anycast: advertise the same IP from all PoPs and let BGP route users to the nearest one. Anycast also provides automatic failover—if a PoP goes down, BGP reroutes to the next-nearest.

Tier 2 — Regional L4: Each PoP has L4 load balancers (NLB or Maglev) that distribute TCP connections across L7 LB instances. This tier handles DDoS mitigation, TCP connection management, and basic health checking.

Tier 3 — Per-Service L7: Within each region, L7 LBs (Envoy, Nginx) route requests to service-specific backend pools. They handle path-based routing, authentication, rate limiting, and canary deployments. Use P2C or least-connections at this tier.

Cross-region failover: If all backends in a region fail, the regional health check fails, the global DNS/Anycast removes that region, and traffic automatically routes to the next-closest region. This adds latency (e.g., Tokyo fails over to Singapore, adding ~50ms) but maintains availability.

Follow-ups: How do you handle data locality? (If the database is in us-east-1, requests from Tokyo routed to the local PoP still need to make cross-Pacific database calls. Solutions: regional read replicas, edge caching, or a globally distributed database like CockroachDB/Spanner.) How do you test failover? (Regularly run "region evacuation" drills by marking a region unhealthy and verifying traffic shifts correctly.)

Key Takeaways

  • A load balancer distributes traffic across servers to improve throughput, latency, and availability. It's one of the first infrastructure components you add when scaling beyond a single machine.
  • L4 vs L7: L4 is faster and protocol-agnostic; L7 understands HTTP and enables content-based routing. Most production architectures use both in a multi-tier setup.
  • Power of Two Choices (P2C) should be your default algorithm for most workloads. It dramatically reduces load imbalance with almost zero overhead compared to random or round-robin.
  • Health checks are your lifeline. Configure them carefully, include a panic mode fallback, and use slow-start to prevent thundering herd on recovery.
  • Stateless backends are the goal. Externalize session state to avoid sticky-session problems. If you must use stickiness, prefer cookie-based affinity with short TTLs.
  • The LB itself must be redundant. A single load balancer is a single point of failure. Run at least an active-passive pair, or use a managed service.
  • Monitor per-backend metrics, not just aggregates. Aggregate p99 can look fine while one backend is on fire. Watch for imbalance in connections, RPS, and latency across your pool.
  • Size your pool as ceil(peak_RPS / per_server_RPS) * 1.3-1.5. Test your capacity assumptions under realistic load before relying on them in production.
