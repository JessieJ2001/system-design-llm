Technologies

Per-category comparison-and-contrast pages for every technology a candidate needs to defend in an interview. The question is never "explain Cassandra" — it is always "you said Cassandra, why not DynamoDB or Mongo or Postgres?" These pages answer the second question.

15 categories • comparison pages first, per-tech reference pages second

How to use these pages

Each category has a comparison page (the centerpiece) and a set of per-tech reference pages.

  1. Start at the comparison page — see the decision flowchart, feature matrix, access-pattern table, and interview decision rules.
  2. Drill into per-tech pages only when you need depth on one product — mechanism, replication topology, deployment modes, scaling for hot keys, ops knobs.
  3. The matrix shape is consistent across categories — same column ordering, same scoring scale, same workload-pattern shapes — so two categories are scannable side by side.

1. Datastores

SQL relational, NoSQL key-value, document, wide-column, graph — all the systems where data lives. The single most-asked category in any interview.

Datastores: Comparison

Postgres, MySQL, Spanner, CockroachDB, Redis, Memcached, DynamoDB, MongoDB, Cassandra, ScyllaDB, Bigtable, Neo4j — decision flowchart, feature matrix, access-pattern matrix, hybrid patterns, interview rules.

Centerpiece

Per-tech reference pages

PostgreSQL

MVCC + B+tree heap + WAL. The default SQL choice.

MySQL (InnoDB)

Clustered B+tree + undo/redo log.

Spanner

Paxos groups + TrueTime + 2PC. Global SQL textbook example.

CockroachDB

Distributed SQL: Raft per range, MVCC, HLC.

Redis

Event loop + hash table + skiplist. Cache, queue, leaderboard, lock.

Memcached

Slab + LRU, multi-threaded. Pure ephemeral cache.

DynamoDB

Consistent-hash partitioning + per-partition leader.

MongoDB

WiredTiger B-tree, oplog tailing, replica sets.

Cassandra

LSM + Dynamo-style ring + tunable quorum.

ScyllaDB

Cassandra-compatible C++ rewrite, shard-per-core.

Bigtable / HBase

LSM + tablets + Chubby/ZK.

Neo4j

Index-free adjacency for graph traversals.

2. Search

Full-text and vector search — both are "ranked retrieval" engines. Pick the comparison page first, then drill into the engine you'd actually deploy.

Search: Comparison

Elasticsearch, Solr, Algolia, Typesense, MeiliSearch, Postgres tsvector, pgvector, Pinecone, Milvus, Weaviate, Qdrant, FAISS — full-text vs vector vs hybrid; managed vs self-host.

Centerpiece

Per-tech reference pages

Elasticsearch / OpenSearch

Lucene inverted index + segments + BM25; HNSW for vectors.

pgvector

Postgres extension. IVFFlat + HNSW.

Pinecone (and Milvus / Weaviate / Qdrant)

Dedicated vector stores: HNSW or IVF+PQ.

3. Analytics / OLAP / Warehousing

Cloud data warehouses, real-time OLAP engines, time-series databases. They overlap heavily — the question is which to pick and what is the cost shape.

Analytics: Comparison

BigQuery, Snowflake, Redshift, ClickHouse, Druid, Pinot, DuckDB, Prometheus, InfluxDB, TimescaleDB.

Centerpiece

Per-tech reference pages

BigQuery

Dremel + Capacitor columnar + Colossus.

Snowflake

Micro-partitions + virtual warehouses.

ClickHouse

MergeTree columnar + vectorized exec.

DuckDB

Embedded in-process columnar OLAP.

Prometheus

Pull-based scrape + TSDB chunks per series.

InfluxDB

TSI inverted index + TSM log-structured.

5. Messaging (queues, streams, pub/sub)

The single biggest "what should I pick" category after datastores. Log vs queue vs pub/sub; ordering; delivery semantics; retention.

Messaging: Comparison

Kafka, Pulsar, RabbitMQ, SQS, Kinesis, Pub/Sub, NATS, Redis Streams, ActiveMQ.

Centerpiece

Per-tech reference pages

Kafka

Distributed append-only log + ISR.

RabbitMQ

AMQP smart broker (queues + exchanges).

SQS

Managed pull-based queue, visibility timeout.

Pulsar

Broker + BookKeeper, tiered storage.

Cloud Pub/Sub

Global, push or pull.

Kinesis

Kafka-like, AWS managed shards.

NATS / JetStream

Lightweight pub/sub + log.
