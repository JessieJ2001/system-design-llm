CAP Theorem

In a distributed data store, you can only guarantee two of three properties — Consistency, Availability, and Partition Tolerance — and since partitions are inevitable, the real choice is between C and A during a partition.

Advanced • 25 min read

What Is It?

The CAP theorem, formulated by Eric Brewer in 2000 and formally proved by Seth Gilbert and Nancy Lynch in 2002, states that a distributed data store cannot simultaneously provide all three of the following guarantees:

  • Consistency (C): Every read receives the most recent write or an error. All nodes see the same data at the same time. This is linearizability — not to be confused with the "C" in ACID, which refers to database integrity constraints.
  • Availability (A): Every request (read or write) to a non-failing node receives a response — without guaranteeing it reflects the most recent write. The system never refuses to answer.
  • Partition Tolerance (P): The system continues to operate despite arbitrary message loss or delay between nodes. A network partition means some nodes cannot communicate with others.

Here is the critical insight most people miss: partitions are not optional. Networks fail. Switches die. Cables get cut. Data centers lose connectivity. In any real distributed system, you must tolerate partitions. This means you are never actually choosing between C, A, and P — you are choosing between consistency and availability during a partition. When the network is healthy, you can have both.

A CP system will refuse to serve requests (or serve errors) during a partition to ensure no stale or inconsistent data is returned. An AP system will continue serving requests during a partition, accepting the risk of returning stale data that may need reconciliation later. Neither choice is inherently better — it depends on what your application can tolerate.

How It Works

Consider a concrete scenario with two database nodes, A and B, that replicate data between each other:

  1. Normal operation: A client writes balance = 100 to Node A. Node A synchronously (or asynchronously) replicates this write to Node B. Both nodes now have balance = 100. A read from either node returns the correct value. Life is good.
  2. Network partition occurs: The network link between Node A and Node B fails. They can no longer communicate. Node A cannot replicate writes to Node B, and vice versa.
  3. A write arrives: A client sends balance = 50 to Node A. Node A processes the write locally. It tries to replicate to Node B but fails because of the partition. Node A now has balance = 50; Node B still has balance = 100.
  4. A read arrives at Node B: A different client sends a read request to Node B asking for balance. Now the system must choose:

CP: Reject the read

Node B knows it may not have the latest data (it detected the partition). It returns an error or times out, refusing to serve potentially stale data. The client gets no answer but is never misled.

Result: Consistent but unavailable.

AP: Serve stale data

Node B returns balance = 100 even though the true value is now 50. The client gets a response immediately, but the data is stale. When the partition heals, the system must reconcile the conflict.

Result: Available but inconsistent.

  1. Partition heals: Once network connectivity is restored, the nodes must reconcile their divergent state. In a CP system, little reconciliation is needed because writes were blocked. In an AP system, conflict resolution kicks in — last-writer-wins, vector clocks, CRDTs, or application-level merging.

When to Use / When NOT to Use

Choose CP When

  • Financial transactions: Double-charging or double-spending is unacceptable. A bank transfer must not show different balances to different readers.
  • Inventory systems: Overselling items (especially limited or expensive ones) has real financial cost. Better to reject orders than sell what you do not have.
  • Leader election / coordination: Two nodes must never both believe they are the leader. This requires consensus, which is inherently CP.
  • Configuration management: All nodes must act on the same configuration. Stale config can cause outages or security issues.

Choose AP When

  • Social media feeds: Showing a slightly stale feed is acceptable. Users tolerate seeing a post 30 seconds late.
  • Shopping carts: Amazon famously chose AP for carts. A lost item in a cart is recoverable; a completely unavailable cart loses sales.
  • DNS: DNS is inherently AP. Stale records for a few minutes are tolerable; a completely unavailable DNS would break the internet.
  • Metrics and analytics: Approximate counts and slightly delayed dashboards are acceptable.

When CAP Does Not Apply

  • Single-node systems: CAP only applies to distributed systems. A single PostgreSQL instance is not subject to network partitions between replicas.
  • Non-distributed data: If your data fits on one machine and you have no replication, there is no partition to worry about.
  • Systems without state: Stateless compute nodes are not affected by CAP. It only governs distributed data stores.

Classic vs. Modern Approaches

Classic: Brewer's Conjecture and the Gilbert-Lynch Proof

Eric Brewer presented the CAP conjecture as a keynote at PODC 2000. In 2002, Seth Gilbert and Nancy Lynch formalized and proved it: in an asynchronous network model, it is impossible for a distributed read/write data store to simultaneously provide consistency, availability, and partition tolerance. The original framing as a simple triangle ("pick 2 of 3") was memorable but misleading — it implied these were equal, interchangeable choices.

The Impossibility Proof (Intuitive Walkthrough)

Imagine the simplest possible distributed system: two nodes, Node 1 and Node 2, connected by a network link. This is the minimum setup needed to call something "distributed." The proof works by showing that even in this trivially small system, you cannot have all three properties simultaneously.

Step 1: A client writes a value. A client sends a WRITE request to Node 1: SET balance = 42. Node 1 accepts the write and stores balance = 42 locally.

Step 2: Replication must happen (for consistency). For the system to be consistent (linearizable), any subsequent read from any node must return 42. This means Node 1 must replicate the value to Node 2. Under normal conditions, Node 1 sends a message to Node 2: "update balance to 42." Node 2 receives it, updates its local copy, and acknowledges.

Step 3: A network partition occurs. The link between Node 1 and Node 2 is severed. Messages cannot get through in either direction. Node 1 cannot reach Node 2, and Node 2 cannot reach Node 1. This is a network partition — and in real networks, this happens due to switch failures, cable cuts, firewall misconfigurations, or data center connectivity issues.

Step 4: A client sends a READ request to Node 2. A different client (or the same client routed differently) connects to Node 2 and asks: GET balance. Node 2 must now decide what to do.

Node 2 faces an impossible choice:

Option A: Sacrifice Availability (CP)

Node 2 refuses to respond. Its reasoning: "I cannot reach Node 1, so I do not know if any writes have happened since my last synchronization. If I return my local value, it might be stale. I would rather return an error than return wrong data."

Result: The system is consistent (no stale reads are possible) but unavailable (Node 2 rejected a request from a client, violating the availability guarantee).

Option B: Sacrifice Consistency (AP)

Node 2 responds with its last known value. If Node 2 still has balance = 0 (the value before the client wrote 42 to Node 1), it returns 0. The client receives a response, but the data is stale — the true value is 42.

Result: The system is available (Node 2 responded to the request) but inconsistent (it returned an outdated value that does not reflect the most recent write).

There is no third option. Node 2 cannot magically learn about the write to Node 1 — the partition prevents any communication. It must either respond with potentially stale data (sacrifice C) or refuse to respond (sacrifice A). This is the CAP theorem: during a partition, you must choose between consistency and availability. No algorithm, no matter how clever, can avoid this choice. It is a consequence of the laws of physics — information cannot travel across a severed network link.

The timeline below summarizes the scenario:

Time:     T0          T1              T2            T3
          |           |               |             |
          v           v               v             v
        Client A    Node 1 tries    PARTITION!    Client B
        writes      to replicate    Link is       reads from
        balance=42  to Node 2...    severed.      Node 2.
        to Node 1.  Message lost.
                                                  Node 2 must choose:
                                                  CP → return ERROR
                                                  AP → return stale value

This proof generalizes to any number of nodes. With N nodes, a partition can always split them into two groups that cannot communicate. Writes in one group cannot be seen by the other group until the partition heals, forcing the same impossible choice on every node in the disconnected group.

Modern: PACELC

Daniel Abadi proposed the PACELC extension in 2010, which captures a crucial nuance that CAP misses: what happens when there is no partition?

PACELC states: if there is a Partition, choose between Availability and Consistency; Else (when the system is running normally), choose between Latency and Consistency.

This is more useful than CAP alone because partitions are rare events. Most of the time, your system is running normally, and the real tradeoff is between latency and consistency. A system can synchronously replicate writes to all replicas (consistent but slow) or acknowledge writes after writing to one node (fast but eventually consistent). PACELC captures both the partition-time and normal-operation tradeoffs:

  • Spanner: PC/EC — always consistent, pays the latency cost with TrueTime and synchronized clocks.
  • Cassandra: PA/EL — available during partitions, favors low latency during normal operation (tunable per-query).
  • MongoDB (majority write concern): PC/EC — consistent at the cost of higher write latency.

Consistency Levels: A Spectrum

The CAP theorem talks about "consistency" as a single property, but in practice, consistency is not binary — it exists on a spectrum from strongest to weakest. The "C" in CAP specifically refers to the strongest level (linearizability), but real systems often choose a weaker consistency model that is "good enough" for their use case while being cheaper and faster. Understanding these levels is essential for making informed tradeoffs.

Linearizable (Strongest)

Linearizability is the gold standard of consistency. It means: every operation appears to take effect at a single instant in time between its invocation and its response. Once a write completes, every subsequent read — from any node, by any client — must return that written value (or a later one). The system behaves as if there is only one copy of the data, even though it may be replicated across many nodes.

Here is a concrete timeline showing what linearizability requires:

Time:     0     1     2     3     4     5     6
          |     |     |     |     |     |     |
Client A: [---W(x=1)---]
                              Client B: [---R(x)---]
                                         must return 1

Client A's write starts at T=0 and completes (is acknowledged) at T=2.
Client B's read starts at T=3 — AFTER the write completed.
Under linearizability, Client B MUST see x=1.
There are no exceptions. No "maybe." No "it depends on which replica."

What about overlapping operations? If a read and write overlap in time (the read starts before the write finishes), linearizability says the read may return either the old or new value — but once any read returns the new value, all subsequent reads must also return the new value. The system cannot "go back in time."

Time:     0     1     2     3     4     5
          |     |     |     |     |     |
Client A: [------W(x=1)------]
Client B:       [---R(x)---]    → may return 0 OR 1 (overlaps with write)
Client C:                   [---R(x)---] → MUST return 1 (starts after write)

Systems that provide linearizability: ZooKeeper, etcd, Google Spanner, CockroachDB (serializable mode).

Cost: Requires coordination between nodes on every operation (typically a consensus protocol like Raft or Paxos). This adds latency — every read or write may need to contact a majority of nodes. Cross-region linearizability adds the full network round-trip time (80–300ms depending on distance).

Sequential Consistency

Sequential consistency is slightly weaker than linearizability. It guarantees that all operations appear to execute in some sequential order, and each individual client's operations appear in the order that client issued them. However, there is no requirement that this order respects real-time — a read might not see the very latest write if it has not propagated yet, as long as the ordering is consistent.

The key difference from linearizability: sequential consistency does not require that if operation A finishes before operation B starts (in real time), then A must appear before B in the sequential order. It only requires that each client's own operations appear in the order they were issued.

Client A: W(x=1) then W(x=2)   — A writes 1, then writes 2

Client B: R(x)=2 then R(x)=1   ← INVALID under sequential consistency
                                   (B sees A's writes out of order — A wrote 1 before 2,
                                    so B cannot see 2 before 1)

Client B: R(x)=1 then R(x)=2   ← VALID (B sees A's writes in the order A issued them)

Client B: R(x)=2 then R(x)=2   ← VALID (B may have missed the first write entirely,
                                    but the order is preserved — once it sees 2, it
                                    never goes back to 1)

Why it is weaker than linearizable: Under linearizability, if Client A's W(x=2) completes and then Client B starts a read, B must see x=2. Under sequential consistency, B might still see x=1 if the write has not propagated to B's node yet — as long as B eventually sees writes in A's order.

Systems: Most single-leader databases provide sequential consistency by default. Zookeeper provides sequential consistency for reads (reads may be served from followers that lag behind the leader).

Cost: Lower than linearizability because reads do not need to contact the leader or a quorum. A follower can serve reads from its local state.

Causal Consistency

Causal consistency captures the intuitive notion of "cause and effect." It guarantees that if operation A causally precedes operation B (A "happened before" B), then every node sees A before B. But operations that are concurrent (no causal relationship between them) may be seen in different orders by different nodes.

What creates a causal relationship? If Client B reads a value that Client A wrote, then B's subsequent operations are causally dependent on A's write. If two clients never interact (never read each other's writes), their operations are concurrent and can be ordered arbitrarily.

Client A: W(x=1)                    — A writes x=1
Client B: R(x)=1, then W(y=2)      — B reads A's write, then writes y=2.
                                       B's write CAUSALLY DEPENDS on A's write
                                       (B saw x=1 before writing y=2).

Client C must see:  x=1 BEFORE y=2   — because there is a causal chain:
                                         A wrote x=1 → B read x=1 → B wrote y=2
                                         The cause (x=1) must be visible before
                                         the effect (y=2).

But if Client D writes z=3 concurrently (without reading x or y):
  Client C may see z=3 before x=1, or after y=2, or anywhere — because
  z=3 has no causal relationship with the other operations.

Causal consistency is tracked using vector clocks or version vectors. Each node maintains a vector of counters (one per node), and every operation carries this vector as metadata. When node X sends data to node Y, Y merges the vectors and can determine which operations happened before others. This is a lightweight form of coordination — much cheaper than consensus.

Systems: MongoDB (with causal consistency sessions enabled), CockroachDB read transactions, some configurations of Riak.

Cost: Lower than sequential consistency. No need for a single leader or quorum reads. The overhead is carrying vector clock metadata with every operation (typically a few dozen bytes per operation).

Eventual Consistency (Weakest)

Eventual consistency is the weakest useful guarantee. It states: if no new writes occur, all replicas will eventually converge to the same value. There is no guarantee on when convergence happens — it could be milliseconds, it could be seconds, it could (in theory) be much longer. During the convergence window, different clients reading from different replicas may see different values.

Time:     0     1     2     3     4     5     6     7
          |     |     |     |     |     |     |     |
Primary:  W(x=1)
Replica1:       x=?   x=?   x=1   ← converged after 3 time units
Replica2:       x=?   x=?   x=?   x=?   x=1   ← converged after 5 time units

During T=1 to T=2, reads from Replica1 return stale (or no) data.
During T=1 to T=4, reads from Replica2 return stale (or no) data.
After T=5, all replicas agree: x=1. The system has converged.

Eventual consistency provides no ordering guarantees. Two clients reading from two different replicas at the same wall-clock time might see completely different values. A single client reading twice from different replicas might see a value "go back in time" (read 1 returns x=5, read 2 returns x=3 because it hit a less-updated replica).

Systems: Cassandra (default consistency level ONE), DynamoDB (default read mode), DNS, most CDN caches.

Cost: Cheapest and fastest. No coordination between nodes is needed for reads or writes. Write to any node, read from any node. This is why eventually consistent systems have the highest throughput and lowest latency.

Summary: Comparing Consistency Levels

Level Guarantee Real-time? Coordination Cost Example Systems
Linearizable Latest write visible immediately after completion Yes High (consensus protocol on every operation) Spanner, ZooKeeper, etcd
Sequential Writes ordered per client; all clients see same order No Medium (single leader serializes writes) Most single-leader databases
Causal Cause visible before effect; concurrent ops unordered No Low–Medium (vector clocks) MongoDB sessions, CockroachDB reads
Eventual Replicas converge eventually; no timing guarantee No None Cassandra, DynamoDB, DNS

The progression from top to bottom is a tradeoff: stronger consistency costs more coordination, which means higher latency and lower throughput. Most production systems do not need linearizability for every operation. The art of distributed systems design is choosing the weakest consistency level that still gives your application correct behavior. For example, a social media feed can use eventual consistency (who cares if a "like" count is 3 seconds stale?), while a bank balance transfer needs linearizability (you cannot show different balances to different clients).

Modern: Harvest and Yield

Armando Fox and Eric Brewer later proposed the harvest and yield model, which quantifies the tradeoff more gracefully. Yield is the probability of completing a request (similar to availability). Harvest is the fraction of the total data reflected in the response. A search engine during a partition might return results from 9 out of 10 shards (harvest = 0.9, yield = 1.0) rather than failing entirely. This model admits partial results as a valid middle ground.

Modern: Jepsen Testing

Kyle Kingsbury's Jepsen project empirically tests whether distributed systems live up to their claimed consistency guarantees. By injecting network partitions, clock skew, and process crashes into real databases, Jepsen has revealed that many systems claiming strong consistency actually violate it under partition. Jepsen tests have found bugs in PostgreSQL, MongoDB, CockroachDB, Cassandra, Redis, Elasticsearch, and many others. It moved the conversation from theoretical proofs to empirical evidence.

Assumptions & Model

Assumptions Behind CAP

  • Asynchronous network model: Messages can be delayed, dropped, or reordered. There are no timing guarantees. This is a realistic model of real-world networks.
  • Arbitrary message delays: A partition is modeled as messages being delayed indefinitely. You cannot distinguish a slow network from a partitioned one.
  • Linearizable consistency: The "C" in CAP specifically means linearizability — the strongest consistency model. Weaker models (sequential consistency, causal consistency, eventual consistency) are not covered by the theorem's impossibility result.
  • Complete availability: The "A" in CAP means every non-failing node must respond. Even if 99.999% of requests succeed, technically CAP availability requires 100%.

Common Misconceptions

  • "Pick 2 of 3" is misleading: You do not statically choose two properties at design time. Partition tolerance is mandatory in any distributed system, so the real choice is between C and A during a partition. Most of the time (no partition), you can have both.
  • CAP is not binary: Real systems operate on a spectrum. Cassandra lets you tune consistency per query (ONE, QUORUM, ALL). You can be "mostly CP" with occasional AP relaxation.
  • "CA" systems do not exist (in distributed form): Any system claiming to be CA is either a single-node system or will silently lose data during a partition. You cannot wish away network failures.
  • CAP says nothing about latency: A system can be "consistent" but take 30 seconds to respond. CAP does not distinguish between 1ms and 30s responses — hence why PACELC is more practical.

Sizing & Tradeoffs

Rather than sizing in terms of raw throughput alone, the most effective approach is to start from your SLOs and derive the CAP configuration that satisfies them. Each combination of availability target and consistency requirement implies a specific replication topology, quorum strategy, and conflict resolution mechanism.

SLO-Driven Sizing Examples

Example 1: E-Commerce Inventory Service

SLO: 99.9% availability, eventual consistency within 500ms staleness window.

Derived architecture:

  • Replication strategy: 3 replicas with WRITE=LOCAL_QUORUM, READ=LOCAL_ONE. Writes are durable within the local region (2 of 3 nodes acknowledge), while reads are fast and served from any single replica.
  • Conflict resolution: Last-writer-wins with vector clocks for inventory decrements. On conflict detection, the lower inventory count wins (fail-safe for overselling).
  • Staleness budget: With async cross-region replication, p99 replication lag must stay under 500ms. Monitor replication_lag_ms and fall back to READ=QUORUM if lag exceeds the SLO.
  • Availability math: 99.9% uptime = 8.76 hours downtime/year. With 3 nodes per region and 2 regions, the system tolerates 1 node failure per region without violating the SLO. Annual partition budget: ~3 partitions of 2-3 hours each.

Example 2: Banking Transaction Ledger

SLO: Strong consistency (linearizable reads), 99.95% availability.

Derived architecture:

  • Partition handling: CP mode — during a network partition, the minority partition rejects writes with an explicit error rather than risking inconsistency. The client retries or queues the transaction.
  • Quorum parameters: 5-node Raft cluster. Writes require 3-of-5 majority. This tolerates 2 simultaneous node failures while maintaining strong consistency.
  • Availability math: 99.95% = 4.38 hours downtime/year. With a 5-node cluster, you lose availability only when 3+ nodes fail simultaneously. At a per-node failure rate of 0.1%/month, P(3+ failures) < 0.0001% — well within the SLO.
  • Latency tradeoff: Every write requires 3 disk fsyncs across nodes. Budget 5-15ms write latency within a single region. Cross-region strong consistency pushes this to 80-150ms (US-EU round trip).

Example 3: Social Media Likes Counter

SLO: 99.99% availability, eventual consistency acceptable within 5 seconds.

Derived architecture:

  • Data structure: PN-Counter CRDT (positive-negative counter). Each node maintains its own increment/decrement counters. The global count is the sum of all node-local counters, which converges without coordination.
  • Replication: WRITE=ONE, READ=ONE — maximum availability. Anti-entropy gossip synchronizes counters across nodes every 1-2 seconds.
  • Sizing the CRDT state: Each counter requires O(N) space where N = number of nodes. With 12 nodes across 4 regions, each counter is 12 × 16 bytes = 192 bytes. For 100M posts, total CRDT state ≈ 19.2 GB — fits in memory on a single node.
  • Availability math: 99.99% = 52.6 minutes downtime/year. With WRITE=ONE, the system is available as long as any single node is reachable. The probability that all 12 nodes are down simultaneously is effectively zero.

Example 4: Global Configuration Store

SLO: Strong consistency (linearizable), p99 read latency < 50ms globally.

Derived architecture:

  • Cluster placement: 5-node Raft cluster spread across 3 regions (2-2-1 topology). Leader is placed in the region closest to the majority of clients. Alternatively, use a consensus algorithm with leader leases (like etcd) to serve reads locally without contacting the leader.
  • Read strategy: Follower reads with read index protocol. The follower asks the leader for the current commit index (1 RTT within region, ~1-5ms), then serves the read from local state. This avoids full cross-region round trips for reads.
  • Latency math: If the leader is in US-East and a follower is co-located with the client, read latency = intra-region RTT to leader (~2ms) + local disk read (~0.5ms) = ~2.5ms p99. This is well within the 50ms SLO.
  • Write path: Writes always go through the leader and require majority replication. Write latency = max(RTT to 2 nearest replicas). With 2-2-1 placement, writes complete within the two co-located nodes in ~5ms.

SLO-Driven Capacity Planning

Once you have chosen your consistency model based on SLOs, capacity planning follows directly. The table below maps SLO requirements to concrete infrastructure decisions:

SLO RequirementCAP PostureMin. NodesReplicationCapacity Overhead
99.9% avail, eventual consistency AP 3 per region Async, W=1 / R=1 3x storage, 1x write IO
99.95% avail, strong consistency CP 5 (Raft/Paxos) Sync, majority quorum 5x storage, 3x write IO
99.99% avail, eventual consistency AP (CRDT) 3+ per region, 3+ regions Async gossip, CRDTs 9-12x storage, 1x write IO
99.95% avail, bounded staleness CP-adjacent 5 per region Sync local, async cross-region 5x storage, 3x local write IO

Key planning rules:

  • Each "nine" of availability roughly doubles your node count. Going from 99.9% to 99.99% typically requires adding a full extra region.
  • Strong consistency multiplies write IO by your quorum size. A 5-node Raft cluster requires 3 synchronous disk writes per committed entry.
  • Staleness budget determines your monitoring granularity. A 500ms staleness SLO requires sub-100ms replication lag monitoring; a 5s SLO allows coarser 1s polling.
  • Cross-region consistency is the most expensive axis. Global linearizability (like Spanner) requires GPS/atomic clocks and dedicated hardware. Evaluate whether bounded staleness (e.g., Azure Cosmos DB's "consistent prefix") meets your actual needs at a fraction of the cost.

Scaling Journey

0 – 1K Users

Single Server

One database, one server. No replication, no distribution. CAP does not apply. You have full consistency and availability because there are no network partitions between nodes (there is only one node). Enjoy it while it lasts.

1K – 100K Users

Primary-Replica

You add read replicas for scalability. This is mostly CP: writes go to the primary, reads can go to replicas. If a replica is partitioned from the primary, reads to that replica return stale data. Many teams unknowingly become AP here by reading from replicas without checking replication lag.

Migration signal: Primary CPU hitting 70%+ on reads.

100K – 10M Users

Multi-Region Active-Passive

Primary in one region, async replicas in others. This is CP with a latency cost for writes (they must reach the primary region) and AP for cross-region reads (replicas lag behind). A full region outage triggers failover, during which writes are unavailable for minutes. You have chosen partition tolerance and are paying the availability or consistency cost depending on the operation.

Migration signal: Users in distant regions experiencing high latency.

10M+ Users

Multi-Region Active-Active

Multiple regions accept writes simultaneously. This is fundamentally AP — conflicting writes can occur in different regions during a partition (or even during normal operation due to replication lag). You need conflict resolution: last-writer-wins (DynamoDB), CRDTs (Riak), or custom application logic (Amazon shopping cart). The CAP tradeoff is now a daily operational reality, not a theoretical concern.

Migration signal: Single-primary failover is too slow; users demand low-latency writes globally.

What Can Go Wrong

Split-Brain in CP Systems

What happened: A network partition divides a cluster into two halves, and both halves elect their own leader. Each half accepts writes independently, creating divergent state. When the partition heals, data conflicts are irreconcilable without data loss.

Root cause: Insufficient quorum requirements. The system used a two-node cluster without a witness/arbiter, so each node believed it was the surviving majority.

Blast radius: Data corruption affecting all writes during the split-brain period. In one public incident, an Elasticsearch cluster lost acknowledged writes because both sides accepted indexing operations during a network split.

Prevention: Always use odd-numbered clusters (3, 5, 7 nodes). Require a strict majority quorum for leader election. Use fencing tokens to prevent stale leaders from accepting writes.

Stale Reads in AP Systems

What happened: An e-commerce platform using Cassandra with consistency=ONE served inventory counts from a replica that had not yet received a recent decrement. Multiple customers purchased the "last" item, resulting in overselling 340 units of a limited product.

Root cause: The team chose AP for all operations without distinguishing between operations where staleness was acceptable (product descriptions) and where it was not (inventory counts).

Mitigation: For critical operations like inventory checks, use consistency=QUORUM or LOCAL_QUORUM. Accept the latency cost for these specific queries while keeping ONE for non-critical reads.

"CA" Systems That Silently Drop Writes

What happened: A team ran a two-node MySQL cluster with synchronous replication, believing they had a "CA" system. During a brief network partition, the system buffered writes and then silently discarded some when the buffer overflowed. The team only discovered the data loss weeks later through a billing discrepancy.

Root cause: The system was not designed for partition tolerance. Rather than explicitly choosing CP (reject writes) or AP (accept divergence), it pretended partitions could not happen and failed silently.

Prevention: There is no such thing as a CA distributed system. If your system has multiple nodes, decide explicitly what happens during a partition. Monitor replication health and alert on replication buffer usage.

Overconfidence in Network Reliability

What happened: A company ran all services within a single data center and assumed network partitions would never occur. A faulty switch created a partial partition — some nodes could communicate with each other but not with the database cluster. The system had no partition detection logic, no timeouts, and no fallback behavior. It hung indefinitely, requiring a full manual restart.

Prevention: Partitions happen within data centers too (not just between regions). Implement connection timeouts, health checks, and graceful degradation. Test with chaos engineering.

How to Detect You Are in a Partition

Heartbeat failure? → Yes → Can you reach ANY other node?

  Yes → Suspected node failure (not partition). Initiate failover for that node.

  No → Can external monitors (different network path) reach the other nodes?

    Yes → You are partitioned. Activate partition protocol (CP: stop serving; AP: serve stale).

    No → Multiple node failures or a widespread outage. Escalate immediately.

Interactions & Pitfalls

CAP + Caching

Caches are inherently AP. A cache is a local copy of data that can become stale the moment the source of truth changes. Adding a cache layer to a CP database effectively makes reads AP unless you implement synchronous cache invalidation (which is very hard to do correctly). Common pitfall: the team believes their system is CP because the database is CP, but they forgot they are reading from a Redis cache with a 60-second TTL.

CAP + Distributed Transactions (2PC)

Two-phase commit is inherently CP. If the coordinator cannot reach a participant during the prepare phase, the transaction blocks or aborts. This means 2PC sacrifices availability during partitions. If a participant is partitioned from the coordinator after voting "yes" but before receiving the commit/abort decision, it is stuck holding locks indefinitely (the "blocking" problem of 2PC). This is why Saga patterns (which are AP with compensating transactions) are preferred in microservices architectures.

CAP + Message Queues

Message queues provide an eventual consistency bridge between CP and AP worlds. A CP system can publish events to a queue, and an AP system can consume them asynchronously. This decouples the consistency requirements: the source system maintains strong consistency for writes, while downstream consumers accept eventual consistency. Pitfall: if the queue itself is partitioned from either producer or consumer, you need to handle message duplication and ordering issues. This is why idempotent consumers are essential.

Cost Considerations

Cost Tradeoffs by CAP Choice

  • CP = Higher latency (synchronous replication): Synchronous replication across regions requires a full network round-trip for every write. In US-East to EU-West, that is 80-100ms added to every write. At 10K writes/sec, your write throughput may require more nodes to handle the longer request duration, increasing compute costs by 2-3x.
  • AP = Conflict resolution complexity: Conflict resolution logic is engineering time — often the most expensive cost. Custom resolution logic for every data type, extensive testing, and ongoing maintenance. DynamoDB charges for conflict resolution storage (multiple versions of conflicting items). Last-writer-wins is cheap but lossy; CRDTs are correct but constrain your data model.
  • Multi-region = Network transfer costs: Replicating every write across regions incurs data transfer costs. At AWS, cross-region transfer is ~$0.02/GB. A system doing 50K writes/sec with 1KB average payload transfers ~4.3 TB/month cross-region = ~$86/month per region pair. With 3 regions, that triples.
  • Dominant cost driver: For CP, the dominant cost is latency and its impact on user experience and throughput. For AP, the dominant cost is engineering complexity of conflict resolution and the risk of subtle correctness bugs.

Real-World Examples

CP: Google Spanner

Spanner is Google's globally distributed, strongly consistent database. It achieves CP by using TrueTime, a global clock with bounded uncertainty based on GPS receivers and atomic clocks in every data center. Writes use two-phase commit with Paxos-based replication. The tradeoff: writes have higher latency (commit wait for clock uncertainty), but reads can be lock-free by reading at a consistent timestamp. Spanner is PC/EC in PACELC terms — it always prioritizes consistency and pays the latency cost.

CP: MongoDB (Majority Write Concern)

With writeConcern: { w: "majority" }, MongoDB will not acknowledge a write until a majority of replica set members have confirmed it. During a partition where the primary cannot reach a majority, writes are rejected. The primary steps down, and the system is unavailable for writes until a new primary is elected by a majority of nodes. Reads with readConcern: "majority" are also linearizable. However, with weaker settings (w: 1), MongoDB becomes AP and can lose acknowledged writes during failover.

CP: HBase

HBase runs on top of HDFS and ZooKeeper. Each region of data has exactly one RegionServer responsible for it. If the RegionServer fails or is partitioned, that region becomes unavailable until ZooKeeper detects the failure and assigns the region to another server. This can take 10-30 seconds. During that window, reads and writes to that region fail. HBase chose CP: strong consistency for every read and write, at the cost of availability during region transitions.

AP: Apache Cassandra

Cassandra is designed AP-first with tunable consistency. With consistency=ONE, reads and writes are acknowledged by a single replica, making the system highly available and low-latency. During a partition, the replicas that are reachable continue serving requests. Writes to unreachable replicas are stored as "hints" and replayed when the node returns. Anti-entropy repair processes (Merkle trees) detect and fix inconsistencies. For operations that need stronger guarantees, you can use QUORUM at the cost of latency and availability.

AP: Amazon DynamoDB

DynamoDB defaults to eventually consistent reads for maximum throughput and lowest latency. It offers optional strongly consistent reads (which cost 2x the RCU and have higher latency). For writes, DynamoDB uses last-writer-wins by default but also supports conditional writes (optimistic locking with version numbers). DynamoDB Global Tables replicate across regions asynchronously, making cross-region reads eventually consistent. Conflict resolution uses last-writer-wins based on timestamps.

AP: CouchDB

CouchDB was designed for an "offline-first" world where devices may be disconnected for extended periods. It uses multi-version concurrency control and stores all conflicting versions of a document. When replicas reconnect, conflicts are detected automatically, and the application (or a deterministic algorithm) resolves them. This makes CouchDB an excellent fit for mobile apps that must work offline and sync later.

Code Example

A Python simulation of a two-node system handling a network partition, demonstrating both CP and AP behavior:

import time

class Node:
    def __init__(self, name):
        self.name = name
        self.data = {}
        self.peer = None
        self.partitioned = False

    def write(self, key, value, mode="CP"):
        self.data[key] = value
        if self.peer and not self.partitioned:
            self.peer.data[key] = value  # Replicate
            return f"[{self.name}] Write '{key}={value}' replicated OK"
        if self.partitioned and mode == "CP":
            del self.data[key]  # Roll back — can't confirm replication
            return f"[{self.name}] REJECTED: partition detected, write refused"
        # AP mode: accept write locally, reconcile later
        return f"[{self.name}] Write '{key}={value}' accepted (may diverge)"

    def read(self, key, mode="CP"):
        if self.partitioned and mode == "CP":
            return f"[{self.name}] REJECTED: partition detected, read refused"
        val = self.data.get(key, "NOT_FOUND")
        stale = " (possibly stale)" if self.partitioned and mode == "AP" else ""
        return f"[{self.name}] Read '{key}' = {val}{stale}"

# Setup
a, b = Node("A"), Node("B")
a.peer, b.peer = b, a

print(a.write("balance", 100, "CP"))   # Replicated OK
print(b.read("balance", "CP"))         # 100

# Simulate partition
a.partitioned = b.partitioned = True
print("\n--- PARTITION ---")

print(a.write("balance", 50, "CP"))    # REJECTED
print(a.write("balance", 50, "AP"))    # Accepted locally
print(b.read("balance", "CP"))         # REJECTED
print(b.read("balance", "AP"))         # 100 (stale!)

How to Validate Your Design

Jepsen-Style Testing

The gold standard for validating distributed system consistency is Jepsen-style testing. Write a test that performs concurrent reads and writes against your system while injecting faults (partitions, node crashes, clock skew). Then check the history of operations against a linearizability checker (like Knossos or Elle). If the history cannot be explained by a linear ordering of operations, your system has a consistency bug.

Network Partition Injection with Toxiproxy

Toxiproxy (by Shopify) lets you simulate network conditions between services. Create a proxy between your database nodes and inject:

  • Full partition: Drop all traffic between nodes for 30 seconds.
  • Asymmetric partition: Node A can send to B, but B cannot send to A.
  • Partial partition: Drop 50% of packets (this is nastier than full partition because health checks intermittently succeed).
  • High latency: Add 500ms delay to simulate cross-region replication.

During each scenario, measure: Do reads return stale data? Do writes get lost? How long until the system detects the partition? How long until it recovers after the partition heals?

Measure Consistency Violations Under Partition

Write a checker that performs "read-your-own-write" tests: write a unique value to the system, then immediately read it back from a different node. Track how often the read returns the old value. Plot this rate over time, especially during injected partitions. For a CP system, this rate should be 0% (reads are rejected instead). For an AP system, measure the window of inconsistency and verify it matches your SLA.

Chaos Engineering Tools

  • Chaos Monkey / Chaos Mesh / Litmus: For Kubernetes-based systems, inject pod failures, network policies, and DNS failures.
  • tc (traffic control): Linux kernel tool for adding latency, packet loss, and bandwidth limits at the network level.
  • iptables: Block traffic between specific nodes to simulate targeted partitions.

Interview Questions

Explain the CAP theorem and why you cannot have all three.

The CAP theorem states that a distributed data store cannot simultaneously guarantee Consistency, Availability, and Partition Tolerance. This is not a design choice but a proven impossibility in asynchronous networks.

The key insight is that network partitions are inevitable in any distributed system. When a partition occurs, a node that receives a request must make a choice: respond with potentially stale data (sacrificing consistency for availability) or refuse to respond until it can verify the data is up to date (sacrificing availability for consistency). There is no third option — you cannot verify data freshness through a severed network link.

Importantly, CAP uses strict definitions: "consistency" means linearizability (not ACID consistency), and "availability" means every non-failing node must respond (not 99.99% uptime). These strict definitions are why the impossibility holds mathematically.

Follow-up: "Does this mean we can never have a system that is both consistent and available?" Answer: You can — most of the time! When there is no partition (the common case), you can have both. The tradeoff only manifests during a partition.

Design a system that needs strong consistency across regions — what is the latency cost?

Strong consistency across regions requires synchronous replication: a write is not acknowledged until it is durable in all (or a quorum of) regions. The minimum latency cost is the network round-trip time between the writing region and the farthest replica.

For US-East to EU-West, this adds ~80-100ms to every write. For US to Asia, ~150-250ms. This is physics (speed of light in fiber) and cannot be optimized away. Google Spanner achieves this with TrueTime, accepting 10-20ms of "commit wait" for clock uncertainty on top of replication latency.

Strategies to manage this cost: (1) Use Paxos/Raft with a quorum of 3 regions, so you only wait for the 2 closest. (2) Route writes to the nearest region when possible and use consensus to replicate. (3) Accept higher write latency but enable lock-free reads at consistent timestamps (Spanner's approach). (4) Limit cross-region strong consistency to critical operations; use eventual consistency for the rest.

Follow-up: "How does Spanner's TrueTime help?" It provides globally consistent timestamps with bounded uncertainty, allowing reads without locks and enabling consistent snapshots across regions.

When would you choose AP over CP?

Choose AP when the cost of unavailability exceeds the cost of temporary inconsistency. Concrete examples:

Shopping carts: Amazon's Dynamo paper explicitly chose AP for shopping carts. If the cart is unavailable, the customer leaves and you lose the sale immediately. If the cart temporarily has a stale item, you can reconcile later (merge cart versions). The business cost of "add to cart is down" far exceeds "cart has a duplicate item."

Social media feeds: A user seeing a post 30 seconds late is invisible to them. The feed being completely unavailable for 30 seconds is a very visible outage. Eventual consistency is the right choice because the data is time-insensitive.

DNS: The entire internet's naming system is AP. A stale DNS record for a few minutes is tolerable; an unavailable DNS would make the internet unusable.

The general principle: choose AP when data has a low "cost of being wrong" and a high "cost of being unavailable." Choose CP when a single incorrect answer causes irreversible harm (financial transactions, medical records, inventory).

Follow-up: "Can you be AP for some operations and CP for others in the same system?" Yes! Cassandra lets you set consistency per query. You might use QUORUM for inventory decrements but ONE for product page views.

What is PACELC and why is it more useful than CAP?

PACELC extends CAP by addressing what happens during normal operation (when there is no partition). It states: if there is a Partition, choose between Availability and Consistency; Else (normal operation), choose between Latency and Consistency.

This is more useful because partitions are rare events — the system operates normally 99.99%+ of the time. During normal operation, the real tension is between latency and consistency. A synchronous replication system (like Spanner) is consistent but adds latency to every write. An asynchronous replication system (like Cassandra with ONE) is fast but eventually consistent. CAP says nothing about this normal-operation tradeoff.

PACELC classifies systems more precisely: DynamoDB is PA/EL (available during partition, low-latency normally). Spanner is PC/EC (consistent always, pays the latency cost). PNUTS (Yahoo) was PC/EL (consistent during partition via rejecting writes, but low-latency normally via async replication).

Follow-up: "Give an example of a system that is PA/EC." This is unusual but possible: a system that serves stale data during a partition (AP) but synchronously replicates during normal operation (EC). Some configurations of MongoDB with w: majority but degraded-mode fallback behave this way.

How does Cassandra handle the CAP tradeoff?

Cassandra is designed as an AP system but provides tunable consistency, allowing per-query control over the CAP tradeoff. The key mechanism is the consistency level parameter on each read and write operation.

With consistency=ONE, a single replica acknowledgment is sufficient. This maximizes availability and minimizes latency but risks stale reads. With consistency=QUORUM (majority of replicas), Cassandra ensures that at least one replica in the quorum has the latest data (since write quorum + read quorum > replication factor). With consistency=ALL, every replica must respond, giving strong consistency but sacrificing availability if any replica is down.

Cassandra handles partitions through hinted handoff (writes for unreachable nodes are stored locally and replayed later), read repair (inconsistencies detected during reads are fixed in the background), and anti-entropy repair (Merkle tree comparison to detect and fix divergence). These mechanisms converge data eventually but do not prevent stale reads during the convergence window.

Follow-up: "What happens if you use QUORUM reads and QUORUM writes — is that CP?" In practice, it provides strong consistency for individual keys during normal operation. But during a partition where you cannot reach a quorum, those operations will fail (timeout), making it CP for those specific operations. The system degrades gracefully: critical operations use QUORUM, non-critical use ONE.

Is CAP relevant for a single-datacenter deployment?

CAP is less impactful but still relevant in a single datacenter. Network partitions within a datacenter are rarer than cross-region partitions, but they do happen: switch failures, rack-level network issues, misconfigured VLANs, and firewall rules can all create intra-datacenter partitions.

For a single-datacenter deployment with a standard primary-replica database (e.g., PostgreSQL with streaming replication), the practical CAP concern is: what happens when the primary cannot reach its replicas? With synchronous replication, writes block (CP). With asynchronous replication, the primary continues and replicas catch up later (AP for reads from replicas).

Where CAP becomes truly critical in a single DC is with clustered databases (Cassandra, CockroachDB) or distributed caches (Redis Cluster). A network partition between cluster nodes within the same DC forces the same C-vs-A choice as a cross-region partition. The only difference is that intra-DC partitions are shorter (seconds to minutes vs. minutes to hours) and less frequent.

Follow-up: "At what point should I start worrying about CAP?" When you have more than one copy of your data on more than one node. The moment you replicate (for availability, read scaling, or disaster recovery), CAP applies.

Key Takeaways

  • Partitions are inevitable in any distributed system. The real choice is between consistency and availability during a partition.
  • "Pick 2 of 3" is misleading. When there is no partition (the common case), you can have both consistency and availability. The tradeoff is between latency and consistency in normal operation (see PACELC).
  • There is no such thing as a CA distributed system. Any system claiming CA either operates on a single node or will fail unpredictably when a partition occurs.
  • The choice is not all-or-nothing. You can be CP for critical operations (financial transactions, inventory) and AP for non-critical ones (analytics, social feeds) within the same system.
  • Quantify the tradeoff. Calculate the cost of inconsistency (double-charges, overselling) vs. the cost of unavailability (lost revenue per minute). Let the business case drive the technical decision.
  • PACELC is more practical than CAP. It captures the latency-consistency tradeoff that dominates normal operation, not just the rare partition scenario.
  • Test empirically. Jepsen-style testing and chaos engineering reveal whether your system actually behaves according to its claimed consistency model. Many systems fail to deliver their promises under partition.
  • The CAP choice evolves with scale. A single-server system has no CAP concern. Primary-replica is mostly CP. Multi-region active-active is fundamentally AP. Your architecture will shift along this spectrum as you grow.
