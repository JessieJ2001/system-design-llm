Content Delivery Networks

Distribute content to geographically dispersed edge servers so users get faster responses and your origin infrastructure stays protected from traffic spikes.

30 min read

What Is It?

A Content Delivery Network (CDN) is a geographically distributed network of servers — called edge nodes or Points of Presence (PoPs) — that cache and serve content on behalf of an origin server. Instead of every user request traveling hundreds or thousands of miles to a single data center, the CDN intercepts the request and serves a cached copy from a nearby edge node. The result is dramatically lower latency, higher throughput, and reduced load on your origin infrastructure.

The core components of a CDN are: edge nodes (the servers closest to end users, deployed in dozens or hundreds of locations worldwide), PoPs (the data centers housing clusters of edge nodes), and the origin server (your actual web server or object storage that holds the authoritative version of the content). Some CDNs also employ a shield or mid-tier cache between edge and origin to reduce origin load further.

Why use a CDN? Three primary reasons: reduce latency by serving content from a server geographically close to the user (turning a 200ms cross-continent round trip into a 10ms local hop); offload origin traffic so your origin servers handle only cache misses instead of every single request; and absorb traffic spikes because the CDN's massive distributed capacity can handle surges that would overwhelm your own infrastructure — think a product launch, breaking news, or a viral social media post.

Who Owns the CDN?

Almost no one builds their own — CDNs are overwhelmingly run as shared, multi-tenant infrastructure. The economics are brutal: a competitive CDN needs hundreds of PoPs in dozens of countries, peering agreements with thousands of ISPs, anycast IP space, DDoS scrubbing, and 24/7 NOC staff. No single customer's traffic justifies that capex, so CDNs are operated by specialist providers who amortize the buildout across thousands of customer tenants. There are four ownership models you'll see in practice:

  • Commercial multi-tenant CDNs (the default) — Cloudflare, Akamai, Fastly, AWS CloudFront, Google Cloud CDN, Azure Front Door. They own the edge servers and PoPs; you sign up, point your DNS at them, and your content becomes one tenant on their shared cache fleet. The same physical edge box in Tokyo simultaneously serves your site, a news outlet, a SaaS app, and thousands of others — each isolated by hostname/cache-key namespacing. You pay per GB egress and per request.
  • Hyperscaler-owned CDNs — Netflix Open Connect, Google Global Cache, Meta FNA, Apple's edge fleet. These companies built private CDNs because their traffic dwarfs any commercial provider's economics (Netflix alone is ~15% of global internet traffic). They place their own appliances inside ISP networks — an ISP hosts a Netflix box for free because it saves the ISP transit costs. These CDNs cache only their owner's content; they're not for sale.
  • Telco / ISP CDNs — Verizon's Edgecast (now Edgio), Level 3, Telefónica. ISPs run CDNs to keep traffic on-net (cheaper than paying transit) and resell capacity to third parties. Multi-tenant, similar to commercial CDNs but with deeper ISP integration.
  • Private / on-prem CDNs — Some large enterprises (banks, governments) run CDN software (Varnish, NGINX, Apache Traffic Server) on their own globally-distributed infrastructure. Rare, expensive, usually justified by regulatory or sovereignty requirements rather than performance.

So no — companies do not typically deploy their own CDN. The standard model is: you keep your origin (S3 bucket, web servers, API), you sign a contract with a CDN provider, and your content becomes one of millions of cache namespaces sharing the same edge fleet. Cache isolation is by hostname and cache key, not by physical machine. Your cdn.example.com and a competitor's cdn.competitor.com may literally hit the same edge server — the CDN just serves whichever origin the hostname maps to.

How It Works

When a user requests a resource served through a CDN, the following sequence unfolds:

  1. DNS resolution — The user's browser resolves the hostname. The CDN's authoritative DNS server uses geo-routing or anycast to return the IP address of the nearest edge node.
  2. Edge check — The edge node receives the HTTP request and checks its local cache. The cache key is typically the URL plus relevant Vary headers (e.g., Accept-Encoding).
  3. Cache HIT — If the content is cached and the TTL has not expired, the edge node responds directly. The response includes an X-Cache: HIT header. Round-trip time: single-digit milliseconds.
  4. Cache MISS — If the content is not cached (or expired), the edge node sends a request to the origin server (or to a shield/mid-tier cache first). The origin responds, and the edge node caches the response according to the Cache-Control headers before forwarding it to the user.
  5. Subsequent requests — Other users in the same region will now get a cache HIT for the same content, until the TTL expires or the cache is purged.

Concrete End-to-End Request Trace

To truly understand CDN performance, let's trace a single request with concrete timing. A user in Tokyo requests https://cdn.example.com/images/hero.jpg (a 250 KB image).

StepWhat HappensLatencyRunning Total
1 DNS query — Browser asks "what IP is cdn.example.com?" ISP's recursive resolver has the answer cached (TTL hasn't expired). 2 ms 2 ms
2 DNS response — Returns IP 203.0.113.45. Thanks to anycast routing, this IP resolves to the Tokyo edge server (even though the same IP is advertised by servers in London and Virginia). — 2 ms

Cache hit: 14 ms total. Cache miss (cold): 225 ms total. That's a 16× difference — which is why CDN hit rate matters enormously. Even a shield hit (74 ms) is 5× faster than a full origin fetch. Every layer of caching you add shaves off latency.

Pull vs. Push CDN

A pull CDN is lazy: it only fetches content from the origin when a user requests it and it is not in cache. This is the most common model (Cloudflare, Fastly, CloudFront). A push CDN requires you to proactively upload content to the CDN's edge nodes before users request it. Push CDNs give you more control and guarantee that content is available at the edge immediately, but they require explicit management. Push is common for large static assets like video segments (Netflix Open Connect) or software update files.

Anycast Routing

The request trace above glossed over a critical detail: how does a single IP address (203.0.113.45) route the user to a nearby server? The answer is anycast routing.

In normal (unicast) routing, each IP address maps to exactly one server in one location. If the server is in Virginia and the user is in Tokyo, every packet crosses the Pacific Ocean. In anycast, multiple servers in different locations all advertise the same IP address using BGP (Border Gateway Protocol) — the protocol that Internet routers use to share reachability information.

How Anycast Works Step by Step

  1. The CDN deploys edge servers in Tokyo, London, and New York.
  2. All three servers announce the same IP prefix (e.g., 203.0.113.0/24) to their local Internet exchange points via BGP.
  3. BGP propagates these announcements across the Internet. Every ISP router now has three routes to 203.0.113.45: one through Tokyo, one through London, one through New York.
  4. When a user in Tokyo sends a packet to 203.0.113.45, their ISP's router checks its BGP routing table. It sees three paths with different AS path lengths (number of network hops) and metrics.
  5. The router picks the "closest" path — fewest hops, lowest latency. For a Tokyo user, this is the Tokyo edge server.
  6. The packet is delivered to Tokyo, not London or New York. The user never knows (or cares) that the same IP exists in three places.

Why UDP Is Simple with Anycast

Each UDP packet is independent — it has no "connection state." If consecutive packets happen to go to different servers (due to a BGP route change), each server processes its packet independently. This is why DNS uses UDP + anycast extensively: the 13 root DNS server addresses (a.root-servers.net through m.root-servers.net) are each served by hundreds of physical servers worldwide using anycast.

Why TCP Needs Care with Anycast

TCP is stateful: it maintains a connection with sequence numbers, window sizes, and retransmission state. If a BGP route changes mid-connection, packets could suddenly go to a different server that has no knowledge of the existing TCP connection. That server would respond with a TCP RST (reset), breaking the connection.

Solutions:

  • ECMP with flow pinning — Equal-Cost Multi-Path routing hashes the 5-tuple (source IP, destination IP, source port, destination port, protocol) to pick a consistent path. As long as the 5-tuple doesn't change, all packets in a TCP connection follow the same path to the same server.
  • BGP route stability — CDN providers carefully manage their BGP announcements to minimize route flapping. Once a route is established, it tends to stay stable for hours or days.
  • Connection migration (QUIC/HTTP3) — QUIC identifies connections by a Connection ID rather than the 5-tuple. If the underlying path changes, the connection can survive because the new server can look up the connection by its ID (if servers share state).

When to Use / When NOT to Use

Use a CDN When

  • Static assets dominate — images, CSS, JS, fonts, PDFs. These are perfectly cacheable.
  • Global audience — users are spread across continents and latency matters.
  • High traffic spikes — product launches, flash sales, viral content. The CDN absorbs load your origin cannot.
  • Video streaming — HLS/DASH segments are ideal CDN content — small, cacheable, consumed sequentially.
  • DDoS protection needed — CDNs can absorb and filter malicious traffic at the edge.

Don't Use a CDN When

  • Highly personalized content — if every response is unique per user (dashboard data, authenticated APIs), caching provides no benefit and risks leaking private data.
  • Real-time data — WebSocket streams, live stock tickers, or data that changes every second. TTL of zero defeats the purpose.
  • Small local audience — if all your users are in one city and your server is in the same city, a CDN adds cost without meaningful latency improvement.
  • Content that changes every request — CSRF tokens, nonces, or dynamic pricing where staleness is unacceptable.

Classic vs. Modern Approaches

Classic: Static File Caching

The original CDN model (Akamai, circa 1998) was simple: cache static files (HTML, images, CSS) at edge servers close to users. The origin server set Cache-Control headers, and the CDN obeyed them. Cache invalidation was done by changing file URLs (cache-busting via versioned filenames or query strings). This model works well but treats the CDN as a passive, dumb cache.

Modern: Dynamic Content Acceleration & Edge Compute

  • Edge-Side Includes (ESI) — Allows a cached HTML page to include dynamic fragments that are assembled at the edge. The page skeleton is cached, but personalized sections (e.g., user name, cart count) are fetched separately. Supported by Akamai and Varnish.
  • Edge compute (Cloudflare Workers, Lambda@Edge, Deno Deploy) — Run arbitrary JavaScript/WASM at the edge, enabling dynamic personalization, A/B testing, authentication, and even full API endpoints — all without reaching the origin. This transforms the CDN from a cache into a compute platform.
  • Tiered caching (edge → shield → origin) — Instead of every edge node independently fetching from the origin on a miss, requests go through a shield (a regional mid-tier cache). If the shield has the content, it serves it to the edge node. This collapses N edge-miss requests into at most one origin request per shield region, dramatically reducing origin load.
  • Stale-while-revalidate — The edge serves stale content immediately while asynchronously revalidating with the origin in the background. The user gets a fast response, and the cache is refreshed for the next request. Enabled via the stale-while-revalidate directive in Cache-Control.
  • Prefetching and predictive caching — Modern CDNs analyze traffic patterns and proactively fetch content that is likely to be requested soon, warming the cache before the first request arrives.

Inside a Single Edge Node: RAM → SSD → (Disk/Peer)

An "edge node" is itself a multi-tier cache hierarchy — not a single uniform store. When the trace diagram says "the edge checks its cache," it actually walks several layers, fastest first, before declaring a miss and reaching out to the shield. The same hit/miss logic you see between edge and shield repeats inside each edge box at hardware-tier granularity.

Tier Medium Lookup latency Capacity per node Typical contents
L1 / Hot RAM (DRAM) ~10–100 µs ~64–512 GB The hottest few percent of objects: small files (HTML, CSS, JS), top video segments, hot API responses.
L2 / Warm NVMe SSD ~100–500 µs ~10–100 TB The long tail of cacheable objects: most images, less-popular video segments, larger files.
L3 / Cold (optional) SATA SSD or HDD ~1–10 ms ~100 TB–1+ PB Used by some CDNs (notably Netflix Open Connect) for massive video libraries where most content is rarely watched.
L4 / Peer (optional) Sibling node in same PoP ~0.5–2 ms Sum of peers "Cluster cache" — if peer node X already has the object, fetch from X over the rack network instead of going to the shield.

On a request, the cache logic is: check RAM → if miss, check SSD (if hit, optionally promote to RAM) → if miss, check peer nodes (some CDNs) → if miss, go to the shield. Eviction runs in the opposite direction: cold objects in RAM get demoted to SSD; cold objects in SSD get evicted entirely. Promotion and demotion typically use LRU, LFU, or a hybrid like S3-FIFO/SIEVE.

Pure-HDD tiers are mostly gone — modern NVMe is fast enough and dense enough (15–30 TB per drive) that few CDNs bother with spinning disks anymore. The exception is video CDNs like Netflix's Open Connect Appliance, which packs ~200 TB across SSDs plus optional HDDs to hold the entire catalog locally so almost nothing escapes to origin.

Why this matters for the request trace: when we said "edge cache lookup: 1 ms," that's the worst case (SSD hit). A RAM hit is closer to 100 µs. The "14 ms total HIT" number in the diagram is dominated by network RTT (TCP + TLS + GET), not the cache-tier lookup itself. So adding more in-node tiers improves hit ratio (fewer trips to the shield), not raw hit latency.

Cache Hierarchies: Why a Shield Layer Matters

Above we saw the tiers inside a single edge node. Now zoom out one level: even when every edge node has its own RAM+SSD stack, every edge cache miss at the edge goes directly to your origin server. With 50+ edge PoPs worldwide, a single popular-but-uncached object triggers 50+ simultaneous requests to your origin. A shield (also called a mid-tier cache) adds a third layer between edge and origin, collapsing those 50 requests into a handful.

The hierarchy looks like this: User → Edge PoP → Regional Shield → Origin. Each edge PoP is assigned to a specific regional shield (e.g., all Asia-Pacific edges route through the Singapore shield, all European edges through the Frankfurt shield, all North American edges through the Virginia shield).

Worked Example: Quantifying the Shield Benefit

Scenario: A popular new video (100 MB) is released globally. 50 edge PoPs each receive 1,000 requests in the first minute.

WITHOUT a shield layer:

  • Each of the 50 PoPs gets a cache miss on the first request → 50 requests to origin
  • Origin serves 50 × 100 MB = 5 GB
  • Origin bandwidth: 5 GB / 60 s ≈ 700 Mbps
  • After filling, each PoP serves the remaining 999 requests from cache

WITH 3 regional shields (US, EU, Asia):

  • 50 PoPs miss → requests go to 3 shields (not origin)
  • 3 shields miss → only 3 requests reach origin
  • Origin serves 3 × 100 MB = 300 MB
  • Origin bandwidth: 300 MB / 60 s ≈ 40 Mbps

The shield reduced origin load by 94% (50 requests → 3 requests, 5 GB → 300 MB). After the shields cache the file, subsequent edge misses never reach origin at all — they are served by the shield. For a viral piece of content that triggers misses across hundreds of PoPs, the shield is the difference between your origin staying healthy and your origin crashing under load.

MetricWithout ShieldWith 3 ShieldsImprovement

Cache Invalidation at CDN Scale

Cache invalidation is famously one of the two hard problems in computer science. At CDN scale — with 200+ edge PoPs each holding independent caches — it becomes even harder. There are three main approaches:

1. URL-Based Purge

Send a purge request for a specific URL: PURGE /images/hero.jpg. The CDN propagates this to all edge nodes. Simple, but doesn't scale when you need to invalidate thousands of related URLs (e.g., "all product pages in the electronics category").

2. Surrogate Keys (Cache Tags)

Surrogate keys let you tag cached responses with arbitrary labels, then purge by label. This is the most powerful invalidation mechanism for content management systems. 

# When your origin serves a product page, include surrogate keys in the response header:
# Surrogate-Key: product-123 category-electronics store-us
#
# The CDN caches the response AND indexes it by these tags.
#
# Later, when product 123's price changes, purge by tag:
# curl -X POST https://api.cdn.com/purge \
#   -H "Authorization: Bearer TOKEN" \
#   -d '{"surrogate_key": "product-123"}'
#
# This purges ALL URLs tagged with "product-123" across all edge nodes:
#   /products/123
#   /products/123/reviews
#   /category/electronics   (contains product 123 in its listing)
#   /search?q=widget        (if this response included product 123)
#
# One API call, hundreds of URLs invalidated instantly.
#
# You can also purge broader tags:
# curl -X POST https://api.cdn.com/purge \
#   -d '{"surrogate_key": "category-electronics"}'
# This purges EVERY page tagged "category-electronics" — thousands of URLs.

3. Cache-Busting via Versioned Filenames

Never purge at all — just deploy new URLs. Instead of /style.css, use /style.a1b2c3d4.css where the hash changes when the file changes. Old entries expire naturally; new URLs are always cache misses (which is fine — they only miss once). This is the most reliable approach for static assets.

Purge Propagation: The Consistency Window

When you call a purge API, the invalidation doesn't reach all 200+ edge PoPs simultaneously. Here's what actually happens:

Time After PurgeWhat's HappeningPoPs Updated
T + 0.1 sCDN control plane receives the purge request and begins fanning out invalidation messages~10 nearby PoPs
T + 1 sInvalidation messages reach most well-connected PoPs (North America, Europe)~100 PoPs
T + 5 sMessages propagate to Asia-Pacific, South America~180 PoPs
T + 15–30 sFinal PoPs in remote regions (Africa, Oceania) receive the invalidationAll 200+ PoPs

During T=0 to T=30, some users see old content and others see new content. This is the propagation window. Mitigations:

  • Versioned URLs — Use /v2/product/123 or fingerprinted filenames so old and new content never share the same cache key. No propagation window because users immediately request the new URL.
  • stale-while-revalidate — Edge serves stale content immediately but revalidates in the background. Users get a fast response, and within one revalidation cycle the content is fresh. This doesn't eliminate the window but makes it invisible to users (they always get a fast response).
  • Short TTLs for mutable content — If content changes frequently, set max-age=60 so even without an explicit purge, stale content expires within a minute.

TLS Termination at the Edge

When a user connects to a CDN over HTTPS, the TLS handshake (key exchange, certificate verification) must happen with some server. There are two fundamentally different approaches, and the choice has major implications for performance, security, and caching.

Option 1: TLS Passthrough (Terminate at Origin)

The CDN acts as a TCP proxy — it forwards encrypted bytes without decrypting them. The TLS handshake happens directly between the user's browser and the origin server.

  • Problem: high latency — The TLS handshake requires round trips to the origin (100–200 ms away), not a nearby edge (5 ms away). A TLS 1.2 handshake needs 2 round trips, so that's 200–400 ms just for the handshake.
  • Problem: no caching — Since the edge can't read the encrypted content, it can't cache anything. The CDN becomes an expensive passthrough with no caching benefit.
  • Benefit: privacy — The CDN operator never sees your plaintext data.

Option 2: TLS Termination at Edge (Standard Approach)

The CDN terminates TLS at the edge server. The edge decrypts the request, serves from cache if possible, or forwards to origin over a separate connection. This is what virtually all CDNs do by default.

  • Lower latency — TLS handshake happens with a nearby edge server (5 ms round trip instead of 100+ ms to origin). With TLS 1.3 (1-RTT), the total TLS overhead is just 5 ms.
  • Caching works — Since content is decrypted at the edge, it can be inspected, cached, and served to subsequent users.
  • Trust tradeoff — The CDN operator can see your plaintext request and response content. You must trust the CDN provider with your data.

Edge-to-Origin Connection Options

ModeHow It WorksSecurityPerformanceWhen to Use
HTTP (unencrypted) Edge connects to origin over plain HTTP Data exposed between edge and origin Fastest (no TLS overhead on backend) Public, non-sensitive content only (e.g., public images, open-source docs)
HTTPS (re-encrypted) Edge establishes a separate TLS connection to origin Data encrypted end-to-end (user → edge is TLS, edge → origin is TLS) Small added latency (~5–10 ms for backend TLS) Most common default. Use for any content with even moderate sensitivity.
mTLS (mutual TLS) Both edge and origin present certificates to each other Origin verifies the request actually came from the CDN, not an attacker Same as HTTPS + certificate verification overhead High-security environments. Prevents direct-to-origin attacks that bypass the CDN.

Certificate Management at Scale

The CDN must have a valid TLS certificate for your domain (e.g., cdn.example.com) deployed to ALL edge servers. With 200+ PoPs, this is non-trivial:

  • Customer uploads certificate — You generate a certificate (from a CA like DigiCert or Let's Encrypt), then upload the certificate and private key to the CDN via their API or dashboard. Cloudflare, CloudFront, and Fastly all support this. You're responsible for renewal.
  • CDN auto-provisions via ACME — The CDN automatically requests and renews certificates using the ACME protocol (Let's Encrypt or similar). You just point your DNS to the CDN and it handles everything. Cloudflare does this for free-tier customers.
  • Shared wildcard certificate — The CDN uses its own wildcard certificate (e.g., *.cdn-provider.com) and your content is served from your-site.cdn-provider.com. Simplest but means your brand isn't in the URL.

With any approach, certificate rotation must be fully automated. A certificate expiring on even one of 200+ PoPs causes TLS errors for some users — the kind of bug that's intermittent and extremely hard to debug because it depends on which edge server the user reaches.

Assumptions & Model

Assumptions Behind CDN Models

  • Content is cacheable — The CDN model assumes a meaningful portion of your traffic can be served from cache. If every response is unique, the CDN becomes an expensive proxy with no caching benefit.
  • Geographic user distribution — CDN value is proportional to how far users are from the origin. If all users are in the same region as the origin, the latency reduction is minimal.
  • TTL is well-chosen — The model assumes you set TTLs that balance freshness against hit rate. A TTL of 1 second yields nearly zero cache hits; a TTL of 24 hours risks serving very stale content after an update.
  • Uniform hash function for cache keys — Cache key design assumes the Vary header and URL capture all meaningful variation. Miss this (e.g., forgetting Vary: Accept-Encoding), and users get corrupted responses.

When Assumptions Break Down

  • Cache-busting query strings everywhere — If your frontend appends a random query param to every asset URL, every request is a cache miss. The CDN becomes a passthrough.
  • Unpredictable content changes — If content changes at random intervals with no pattern, any fixed TTL is either too long (stale content) or too short (low hit rate).
  • Single-region users — If 95% of your users are in one city, a CDN with 200+ PoPs is wasted cost. A single regional cache or a reverse proxy in front of your origin may suffice.

Sizing & Tradeoffs

Cache Hit Ratio Targets

The single most important CDN metric is the cache hit ratio (CHR). For static assets (images, CSS, JS), you should target a CHR of >95%. For dynamic or semi-dynamic content (API responses with short TTLs), a CHR of 60–80% is good. Below 50%, the CDN is adding latency (extra hop) without sufficient benefit.

TTL Tradeoff

TTL is the fundamental dial. Longer TTL = higher cache hit ratio = staler content. Shorter TTL = fresher content = more origin load. The sweet spot depends on how often your content changes and how tolerant your users are of staleness.

SLO-Driven Sizing Examples

Real-world CDN sizing starts with Service Level Objectives — not raw capacity. Work backwards from your latency, availability, and freshness targets to derive cache sizing, PoP count, and TTL strategy.

Example 1: Video Streaming Platform

SLO: p95 video start time < 2 seconds, supporting 1M concurrent viewers.

Derivation: At 1M concurrent viewers with an average bitrate of 6 Mbps, total throughput is 6 Tbps. To keep p95 start time under 2s, the first 2–3 segments (~2–4 MB each) must be served from edge cache — any origin round-trip would blow the budget. Popular catalog analysis shows the top 5% of titles account for 80% of views.

  • Edge cache per PoP: Top 5% of catalog × 4 bitrate tiers × avg. 5 GB per title = ~50 TB per PoP in hot storage (NVMe SSD).
  • PoP count: To serve 6 Tbps with 200 Gbps capacity per PoP, you need at minimum 30 PoPs. For p95 latency, place PoPs within 50 ms RTT of 95% of your audience — typically 40–60 PoPs across NA, EU, and APAC.
  • Shield tier: 5 regional shields cache the full catalog (~1 PB each) to absorb long-tail misses without hitting origin.

Example 2: E-Commerce with Global Reach

SLO: p99 page load < 3 seconds, 50M daily active users across 40+ countries.

Derivation: A 3-second budget at p99 means the CDN must absorb all static asset delivery within ~500 ms (leaving time for DNS, TLS, HTML generation, and API calls). With 50M DAU and a peak-to-average ratio of 5x, peak traffic hits ~30K RPS for static assets.

  • CDN tier: A premium-tier CDN (Cloudflare Enterprise, CloudFront) with 200+ PoPs to ensure <30 ms edge-to-user latency globally. Budget CDNs with 30–50 PoPs would breach p99 in underserved regions (Africa, South America, Southeast Asia).
  • Origin shield strategy: Deploy 3 origin shields (US-East, EU-West, APAC-Singapore) to collapse miss traffic. Product catalog (~2M SKUs × 15 images each × 200 KB avg = ~6 TB) fits in shield-tier SSD. Enable request coalescing at the shield to survive flash sales.
  • Cache rules: Fingerprinted static assets get max-age=1y, immutable. Product pages get max-age=300, stale-while-revalidate=3600 to balance freshness with hit rate during sales events.

Example 3: Static Asset Serving

SLO: 99.9% cache hit ratio, 500 TB/month egress.

Derivation: A 99.9% CHR means only 0.1% of requests (misses) hit origin. At 500 TB/month with an average object size of 250 KB, that is ~2B requests/month or ~770 RPS average. Misses = 770 × 0.001 = ~0.77 RPS to origin — trivial origin load.

  • TTL strategy: Achieving 99.9% CHR requires aggressive TTLs. All assets must use fingerprinted filenames with max-age=31536000, immutable. Any asset using short TTLs (<1 hour) will drag the overall CHR below target. Audit cache key configuration to avoid unnecessary variation.
  • Purge budget: With 2B requests/month and 99.9% CHR target, you can tolerate at most ~2M misses/month. Each purge of a popular asset could generate 50–200 misses (one per PoP). Budget: <10K purges/month. This forces you toward filename-based cache busting rather than explicit purges.
  • Monitoring alert: Alert if hourly CHR drops below 99.5%. Common causes: deploy without fingerprinted filenames, misconfigured Vary header fragmenting cache, or Set-Cookie on static responses making them uncacheable.

Example 4: Real-Time Sports Scores API

SLO: p99 < 100 ms globally, 10M RPS during peak events (World Cup final, Super Bowl).

Derivation: At 100 ms p99, there is zero budget for origin round-trips on cache misses (origin RTT alone would be 50–200 ms). Every response must come from the edge. But sports scores change every few seconds — traditional caching with long TTLs is impossible.

  • Edge compute + cache strategy: Deploy edge workers (Cloudflare Workers / CloudFront Functions) at 200+ PoPs. Origin pushes score updates via WebSocket/SSE to all edge PoPs every 1–2 seconds. Edge workers serve the latest pushed state from in-memory KV store — no per-request origin fetch.
  • Cache TTL: Set max-age=1, stale-while-revalidate=5 as a fallback. The primary serving path is edge compute reading from a replicated KV store, not cache lookups.
  • Capacity at 10M RPS: Distributed across 200 PoPs = 50K RPS per PoP. Each edge worker handles ~10K RPS per instance, so provision 5–8 worker instances per PoP with auto-scaling. Total edge compute cost: ~$15K–25K/month at this scale.
  • Failover: If edge KV replication lags >5s, fall back to direct origin fetch with stale-if-error=30 to avoid serving stale scores beyond one play.

SLO-Driven Capacity Planning

When sizing a CDN in a system design interview, always start from the SLO and work backwards to infrastructure. Here is the framework:

Step Question to Answer What It Determines
1. Define latency SLO What is the target p95/p99 latency for end users? PoP placement: If p95 < 50 ms, you need PoPs within every major metro. If p95 < 200 ms, continental PoPs suffice.
2. Compute throughput What is peak RPS and peak bandwidth? PoP count & capacity: Total peak bandwidth ÷ per-PoP capacity = minimum PoP count. Add 2x headroom for bursts.
3. Analyze content working set What fraction of content is hot/warm/cold? Edge cache size: Hot + warm content size = minimum per-PoP cache. Cold content handled by shield tier.
4. Set CHR target What cache hit ratio does the latency SLO require? TTL strategy: If CHR must be >99%, you need long TTLs and fingerprinted filenames. If 80% suffices, short TTLs with stale-while-revalidate work.
5. Size origin for misses At the target CHR, how much traffic reaches origin? Origin capacity: Peak RPS × (1 − CHR) = origin RPS. Add request coalescing to reduce further by 5–10x.
6. Plan for failure modes What happens during CDN outage or cache flush? Multi-CDN, stale-if-error, origin auto-scaling: Origin must survive a temporary 100% miss rate or you need a secondary CDN.

Rule of thumb: If your latency SLO is tight (p99 < 100 ms), caching alone is not enough — you need edge compute. If your freshness SLO is tight (data <5s old), you need push-based edge replication rather than pull-based caching. If your availability SLO is >99.99%, you need multi-CDN with active-active traffic steering.

Scaling Journey

0 – 1K users

Direct from Origin

Serve everything from your single web server. No CDN needed — latency is acceptable, and traffic is low. Maybe set some Cache-Control headers for browser caching.

1K – 100K users

CDN for Static Files

Put images, CSS, JS, and fonts behind a CDN (CloudFront, Cloudflare). Use versioned filenames for cache busting. Your origin handles only HTML and API requests. Signal to migrate: static assets consume most of your bandwidth and origin CPU.

100K – 10M users

Custom Cache Rules

Configure per-path cache rules, enable tiered caching (shield), cache API responses with short TTLs, use stale-while-revalidate. Add purge automation to your CI/CD pipeline. Signal: origin overloaded during traffic spikes despite CDN.

10M+ users

Multi-CDN + Edge Compute

Deploy across multiple CDN providers for redundancy and cost optimization. Use edge compute for personalization, A/B testing, and authentication. Implement real-time traffic steering between CDNs based on performance and cost. Signal: single CDN outage impacts business, or single vendor lock-in becomes a cost issue.

What Can Go Wrong

1. Cache Purge Propagation Delay

What happened: A company deployed a critical bug fix to production and purged their CDN cache. But purge propagation took 30–90 seconds across 200+ PoPs. During that window, some users received the old buggy version while others got the fix, causing inconsistent behavior and confused support tickets.

Root cause: CDN purge is not instantaneous — it must propagate to every edge node. Some CDNs process purges asynchronously with no SLA under 30 seconds. Blast radius: All users in regions where purge had not yet propagated. Mitigation: Use versioned filenames (content-addressed assets) instead of purging. Deploy new URLs, and let old URLs expire naturally. For HTML, use short TTLs (60 seconds) combined with stale-while-revalidate so users always get a fast response even if it is briefly stale.

2. Origin Overload on Cache Miss Storm

What happened: A CDN provider had a partial outage that cleared the cache on a large number of edge nodes simultaneously. When the CDN recovered, all edges simultaneously requested the same popular content from the origin. The origin — sized for 5% of total traffic (normal miss rate) — was hit with 100% of traffic and collapsed.

Root cause: No request coalescing at the edge. Each edge independently sent a miss request to origin. Detection: Origin CPU and connection count spiked to saturation. Mitigation: Enable request coalescing (also called "request collapsing" or "coalesce on miss") — when multiple edge nodes request the same URL simultaneously, the CDN or shield forwards only one request to origin and fans out the response. Also enable stale-if-error to serve stale content when origin is down.

3. Cache Poisoning Attack

What happened: An attacker sent requests with crafted headers (e.g., X-Forwarded-Host) that the origin reflected in the response body. The CDN cached this poisoned response and served it to all subsequent users, injecting malicious content or redirects.

Root cause: The cache key did not include the headers that the origin used to generate the response. The Vary header was missing or incomplete. Blast radius: All users requesting that URL until the poisoned entry expired or was purged. Prevention: Validate and strip unrecognized headers at the edge. Ensure the origin's Vary header matches all headers that affect the response. Regularly audit cache key configuration.

4. CDN Outage as Single Point of Failure

What happened: In June 2021, Fastly experienced a global outage lasting approximately one hour. Major websites including Reddit, The New York Times, and the UK government's GOV.UK went offline because they relied on Fastly as their sole CDN provider.

Root cause: A software deployment triggered a bug in Fastly's configuration system that caused 85% of their network to return errors. Detection: Immediate — global monitoring lit up. Prevention: Multi-CDN architecture with automatic failover. DNS-level traffic steering that can reroute to a backup CDN within seconds.

Interactions & Pitfalls

CDN + Application Cache (Double Caching)

When you have both a CDN cache and an application-level cache (Redis, Memcached), invalidation must clear both layers. A common mistake: you invalidate Redis after a database write, but the CDN still serves the old cached response for the remaining TTL. The fix is to either purge the CDN as part of your invalidation flow, or use short CDN TTLs with stale-while-revalidate and rely on the application cache for freshness.

CDN + Rate Limiting

Rate limiting at the edge and rate limiting at the origin behave very differently. If you rate-limit only at the origin, the CDN absorbs most traffic and the origin rarely sees high volume — but a cache miss storm can bypass the CDN and overwhelm the origin before rate limits kick in. If you rate-limit at the edge, be careful not to cache 429 Too Many Requests responses, or you will block legitimate users who share the same cache key.

CDN + Authentication

Never cache authenticated responses without proper Vary header handling. If user A requests a page with an Authorization header and the CDN caches the response without varying on that header, user B will see user A's private data. The safe approach: set Cache-Control: private for authenticated responses, or use Vary: Authorization (which effectively disables shared caching). For public content that requires auth only for access control, consider signed URLs or token-based edge authentication.

Cost Considerations

CDN Cost Model

  • Bandwidth pricing — The dominant cost driver. CDNs charge per GB of data transferred from edge to user. Typical rates: $0.02–$0.08/GB for high-volume customers, $0.08–$0.15/GB at lower volumes. Premium PoPs (Asia, South America, Africa) can be 2–3x more expensive than North America/Europe.
  • Request pricing — Some CDNs (CloudFront) charge per HTTP/HTTPS request: ~$0.01 per 10,000 HTTPS requests. At high request volumes with small responses, request costs can exceed bandwidth costs.
  • Origin shield — Reduces origin egress (often the most expensive cloud cost) by 80–95%. Even though it is an additional CDN feature, it typically saves more on origin egress than it costs.
  • Multi-CDN arbitrage — Different CDNs have different pricing per region. By steering traffic to the cheapest CDN per region (e.g., Cloudflare for Europe, CloudFront for Asia), you can reduce costs 20–40%.
  • Cost of NOT using a CDN — Without a CDN, your origin bandwidth bill scales linearly with user traffic. Cloud egress costs ($0.08–$0.12/GB from AWS/GCP) are often more expensive than CDN bandwidth. At 100 TB/month, that is $8,000–$12,000 in cloud egress vs. $2,000–$5,000 through a CDN.

Real-World Examples

Netflix Open Connect

Netflix built its own CDN called Open Connect rather than relying on third-party CDNs. Open Connect Appliances (OCAs) are deployed directly inside ISP networks worldwide. During off-peak hours, Netflix proactively pushes (push CDN model) popular content to these appliances. During peak hours, over 95% of traffic is served from within the ISP network, never traversing the internet backbone. This is an extreme example of a push CDN tailored for a specific workload (large video files with predictable popularity curves).

Cloudflare's Network

Cloudflare operates one of the largest CDN networks with 300+ PoPs in 100+ countries. They use anycast routing — every PoP advertises the same IP address, and BGP routing sends users to the nearest one. Beyond caching, Cloudflare offers Workers (edge compute), DDoS protection, WAF, and DNS. Their free tier makes CDN accessible even to small projects, and their architecture demonstrates how modern CDNs have evolved from simple caches to full edge compute platforms.

Akamai at Scale

Akamai, the pioneer of CDNs (founded 1998), delivers 15–30% of all global web traffic through 365,000+ servers across 135 countries. Akamai invented many CDN techniques still used today: Edge-Side Includes (ESI), SureRoute (dynamic path optimization), and tiered distribution. Their scale demonstrates the ceiling of CDN architecture: serving multiple Tbps of aggregate bandwidth with sub-100ms latencies worldwide.

Code Example

Setting proper Cache-Control headers and a simple CDN cache simulation in Python:

# CDN cache simulation with Cache-Control header logic
import time
from collections import OrderedDict

class EdgeCache:
    """Simulates a CDN edge node with TTL-based caching."""

    def __init__(self, capacity=100):
        self.store = OrderedDict()  # url -> (response, expires_at)
        self.capacity = capacity
        self.hits = 0
        self.misses = 0

    def get(self, url, origin_fn):
        """Fetch from cache or origin. Returns (response, hit?)."""
        if url in self.store:
            resp, expires = self.store[url]
            if time.time() < expires:
                self.hits += 1
                self.store.move_to_end(url)
                return resp, True
            del self.store[url]  # expired

        # Cache miss — fetch from origin
        self.misses += 1
        resp, ttl = origin_fn(url)
        if ttl > 0 and len(self.store) >= self.capacity:
            self.store.popitem(last=False)  # evict LRU
        if ttl > 0:
            self.store[url] = (resp, time.time() + ttl)
        return resp, False

    @property
    def hit_ratio(self):
        total = self.hits + self.misses
        return self.hits / total if total else 0

# Usage: origin returns (body, ttl) based on Cache-Control
def origin(url):
    headers = {
        "/style.css":  3600,   # Cache-Control: max-age=3600
        "/api/data":   10,     # Cache-Control: max-age=10
        "/private":    0,      # Cache-Control: no-cache
    }
    ttl = headers.get(url, 60)
    return f"content of {url}", ttl

How to Validate Your Design

Test Cache Hit Ratio

Use curl -I to inspect CDN response headers. Look for X-Cache: HIT or X-Cache: MISS, the Age header (seconds since the response was cached), and Cache-Control values. Request the same URL twice: the first should be a MISS, the second a HIT. 

# Quick validation script
# curl -sI https://cdn.example.com/style.css | grep -E "X-Cache|Age|Cache-Control"
# Expected: X-Cache: HIT, Age: > 0, Cache-Control: public, max-age=...

Test Purge Propagation

After issuing a purge, poll the URL from multiple regions (use tools like updown.io, KeyCDN Tools, or multi-region curl scripts). Measure the time until all regions return the new content. Target: under 30 seconds globally.

Load Test Origin with CDN Disabled

Temporarily bypass the CDN (point DNS directly to origin) and run a load test with k6, vegeta, or Locust at your expected peak traffic. This measures the "CDN offload benefit" — how much origin capacity you would need without the CDN. Compare origin resource usage with and without CDN to quantify the savings.

Chaos Testing

Inject failures to validate resilience: simulate a CDN outage (block CDN IPs at your DNS), trigger a cache miss storm (purge all content and send a traffic spike), add latency between edge and origin with toxiproxy. Verify your system degrades gracefully — stale-if-error headers should let the CDN serve stale content when origin is unreachable.

Interview Questions

Design a CDN for a video streaming platform

Video streaming is the ideal CDN workload. Video files are large, immutable once encoded, and consumed sequentially in small segments (HLS/DASH). The design should use a push CDN model: after transcoding, proactively push popular content to edge nodes during off-peak hours. Use popularity prediction (based on subscriber count, trending signals) to decide what to pre-position.

For the edge tier, deploy dedicated video caching appliances with large SSD storage (tens of TB per node) inside ISP networks or at peering points. Use a tiered architecture: edge PoPs serve hot content, regional hubs store the full catalog, and origin stores the master copies. Implement byte-range request support so the CDN can serve partial segments without caching the entire file.

Key considerations: adaptive bitrate streaming means the same video exists at multiple quality levels — cache the most popular bitrates preferentially. Implement request coalescing to prevent thundering herd on new content releases. Set long TTLs (days to weeks) since video segments never change once created. Monitor cache fill rate during releases to ensure edge nodes warm up before viewership peaks.

Follow-up: How would you handle live streaming versus on-demand? Live segments have a very short useful lifetime (seconds), requiring ultra-low-latency origin-to-edge propagation and different caching strategies (very short TTLs or server push).

How do you invalidate CDN cache reliably?

There are three invalidation strategies, each with tradeoffs. URL-based purge: send a DELETE/PURGE request for a specific URL to the CDN's API. Simple but does not scale when thousands of URLs need purging. Surrogate key (tag) purge: tag cached responses with logical keys (e.g., product-123, homepage), then purge by tag. One API call invalidates all URLs tagged with that key. This is the preferred approach for content management systems. Cache-busting via filename changes: never purge — deploy new URLs (e.g., style.a1b2c3.css) and let old entries expire. This is the most reliable approach for static assets.

For reliability, combine approaches: use fingerprinted filenames for static assets (never need purging), use surrogate key purges for dynamic content, and set short TTLs with stale-while-revalidate as a safety net. Always verify purge completion by polling the URL from multiple regions. Build purge operations into your CI/CD pipeline as a post-deploy step with a verification check.

Follow-up: What happens if the purge API itself is down? You should have a fallback: either a secondary CDN with its own purge path, or reliance on TTL expiry as a last resort (which is why TTLs should never be excessively long for content that might need invalidation).

Pull vs. push CDN — when to use which?

Pull CDN (lazy, on-demand): The CDN fetches content from origin only when a user requests it and the cache does not have it. Best for: websites with diverse content (millions of URLs where only a small fraction is popular), content with unpredictable popularity, and teams that want minimal operational overhead. Downsides: the first request for any content is always a cache miss (cold start), and sudden popularity spikes can cause origin load.

Push CDN (proactive, pre-positioned): You upload content to the CDN before users request it. Best for: large static files where the first-request latency penalty is unacceptable (video streaming, software downloads, game patches), content with predictable access patterns, and scenarios where origin reliability is a concern. Downside: operational complexity — you must manage the upload pipeline, handle failures, and ensure consistency across edge nodes.

Hybrid approach: Use pull for the long tail of content and push for high-value predictable content. Netflix uses push for popular titles and pull for the long tail of their catalog.

Follow-up: How would you implement a push CDN if the vendor only supports pull? Pre-warm the cache by programmatically requesting content from every edge PoP after deployment — effectively simulating push behavior over a pull infrastructure.

How to handle personalized content with a CDN?

Personalized content is the nemesis of caching — if every user gets a different response, the cache hit ratio drops to zero. But there are several techniques to still extract value from a CDN:

Edge-Side Includes (ESI): Cache the page skeleton and assemble personalized fragments at the edge. The header, footer, and product descriptions are shared (cached); the user name, cart, and recommendations are fetched dynamically and stitched in at the edge. Edge compute: Use Cloudflare Workers or Lambda@Edge to run personalization logic at the edge, reading user cookies to customize the response without hitting origin. Client-side personalization: Serve a generic cached page and let the browser JavaScript personalize it after load (fetch user data via API). This gives the fastest initial page load. Cache segmentation: If personalization is limited to a few dimensions (country, language, A/B test bucket), use those as part of the cache key via the Vary header. This creates a manageable number of cached variants.

Follow-up: What is the privacy risk of cache segmentation? If you segment by user ID, you effectively have one cache entry per user — no benefit. If you segment too broadly, users might see each other's personalized data. Always validate that your cache key granularity matches your personalization granularity.

Design a multi-CDN failover strategy

A multi-CDN setup requires three components: traffic steering, health monitoring, and consistent configuration.

Traffic steering: Use DNS-based steering (e.g., NS1, Route 53 with health checks) to distribute traffic across CDN providers. Weight traffic based on performance (latency, availability) and cost. In normal operation, distribute 70/30 between primary and secondary. On failure, shift 100% to the healthy provider within seconds. Alternatively, use a meta-CDN layer (Cedexis/Citrix, Constellix) that performs real-user measurements and routes each request to the best-performing CDN in real time.

Health monitoring: Deploy synthetic monitoring from multiple vantage points (Catchpoint, ThousandEyes). Check both availability (HTTP 200) and performance (latency, throughput) from every region. If a CDN's p95 latency exceeds a threshold or error rate exceeds 1%, trigger failover automatically. Use short DNS TTLs (30–60 seconds) to enable fast failover.

Consistent configuration: This is the hardest part. Both CDNs must have identical cache rules, header handling, and TLS certificates. Use infrastructure-as-code (Terraform, Pulumi) to manage CDN configuration across providers. Test both paths regularly — do not let the secondary CDN go untested for months.

Follow-up: How do you handle cache warming on failover? When traffic shifts to the secondary CDN, its cache is cold. Use origin shield on both CDNs (pointed at the same shield) to avoid a miss storm, or implement background cache warming on the secondary CDN during normal operation.

What cache headers would you set for an API response?

It depends entirely on the API endpoint's characteristics. Here is a decision framework:

Public, rarely changing data (e.g., /api/countries, /api/config): Cache-Control: public, max-age=3600, stale-while-revalidate=86400. Cache for 1 hour, serve stale for up to 1 day while revalidating. Add Vary: Accept-Encoding.

Semi-dynamic data (e.g., /api/products): Cache-Control: public, max-age=60, stale-while-revalidate=300. Cache for 1 minute, revalidate in background for 5 minutes. Add ETag for conditional requests.

Private/authenticated data (e.g., /api/me, /api/orders): Cache-Control: private, no-store. Never cache in shared CDN. private allows browser caching but not CDN caching.

Real-time data (e.g., /api/stock-price): Cache-Control: no-cache, no-store, must-revalidate. No caching anywhere. Consider whether this endpoint should bypass the CDN entirely.

Follow-up: What is the difference between no-cache and no-store? no-cache allows caching but requires revalidation before every use. no-store prohibits caching entirely. For sensitive data, always use no-store.

Key Takeaways

  • A CDN is a geographically distributed cache layer. Its value is proportional to your cache hit ratio and the geographic distance between users and origin.
  • The cache hit ratio is the single most important metric. Target >95% for static assets. Every percentage point below that means more origin load and higher latency.
  • TTL is the fundamental tradeoff dial: longer TTL = higher hit rate = staler content. Use stale-while-revalidate to get the best of both worlds.
  • Use fingerprinted filenames for static assets — this makes cache invalidation trivial and eliminates purge propagation delays.
  • Enable origin shield (tiered caching) to collapse cache miss storms from N edge nodes into one origin request per region.
  • Never cache authenticated/private responses without proper Vary header handling and Cache-Control: private directives.
  • Modern CDNs are evolving from passive caches into edge compute platforms (Workers, Lambda@Edge). Use them for personalization, A/B testing, and security logic at the edge.
  • Plan for CDN outages: use multi-CDN architecture or at minimum have a failover plan. No single provider has 100% uptime.
  • A CDN often pays for itself: CDN bandwidth is typically cheaper than cloud provider egress, and the origin cost savings from offloading traffic are substantial.
