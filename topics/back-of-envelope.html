Back-of-Envelope Calculations

The reference numbers every system designer needs at their fingertips. These are the constants, rules of thumb, and quick-math techniques used throughout every NALSD scenario on this site.

Reference page — bookmark this

Why Back-of-Envelope Math Matters

In a system design interview — and in real engineering — you need to make quick decisions about whether an approach is feasible before investing time in designing it. Can one server handle this load? Will this dataset fit in memory? How many machines do we need?

You don't need exact numbers. You need to be within an order of magnitude. If your estimate says you need 10 servers and the real answer is 15, that's fine — you made the right architectural choice. If your estimate says 10 and the real answer is 10,000, you chose wrong.

The technique is simple: start with known constants, multiply through the chain, round aggressively. Every number in this reference page is approximate. That's the point.

The "Divide by 100K" Rule

Since 1 day ≈ 105 seconds, converting from daily volume to QPS is simple: divide by 100,000 (or equivalently, multiply by 10-5).

Daily Volume ÷ 105 QPS (average) Peak QPS (×2–3)
1 million/day1M / 100K~10 QPS~20–30 QPS
10 million/day10M / 100K~100 QPS~200–300 QPS
100 million/day100M / 100K~1,000 QPS~2,000–3,000 QPS
1 billion/day1B / 100K~10,000 QPS~20,000–30,000 QPS

Peak factor: Real traffic is not uniform. Use a 2–3× peak multiplier over the average. For spiky workloads (flash sales, viral events), use 10× or more.

Assumptions behind the peak multiplier:

  • 2–3× assumes a typical consumer web/mobile traffic pattern where usage peaks during evening hours in the primary timezone and is low overnight. If your users span many timezones (global service), the peak-to-average ratio is lower (~1.5–2×) because peaks smooth out across regions.
  • "Divide by 100K" assumes traffic is spread uniformly across the day. In reality, most of the day's traffic happens in ~8 active hours, not 24. So the "real average during active hours" is already ~3× the 24-hour average. The peak multiplier stacks on top of that.
  • 10× is a rough upper bound for planned spikes (Black Friday, game launches). Truly viral events (celebrity tweet, breaking news) can hit 50–100× — but these are unplannable and handled by autoscaling + shedding, not pre-provisioning.

The Latency Hierarchy

For quick mental math, remember this order-of-magnitude ladder:

1 ns          — CPU cache
100 ns        — RAM (100×)
10 μs         — SSD (100×)
1 ms          — Disk seek / network same-DC (100×)
100 ms        — Cross-continent network (100×)
1 s           — Unacceptable for user-facing (10×)

Each rung is roughly 100× slower than the previous. When you're choosing where to put data (cache vs. DB vs. disk), you're choosing which rung of this ladder you're on.

These numbers assume a standard cloud instance: 8–16 vCPUs, 32–64 GB RAM, SSD storage, 10 Gbps network — something like an AWS m6i.2xlarge to m6i.4xlarge, GCP n2-standard-8 to n2-standard-16, or a bare-metal equivalent. These numbers are order-of-magnitude reference points, not benchmarks. Real throughput depends on query complexity, payload size, connection count, OS tuning, and dozens of other factors. Use them to check feasibility ("is 50K QPS plausible on one box?"), not to size production infrastructure.

Key assumptions in the throughput table:

  • Web server (50K–100K req/sec): assumes small static responses (<10 KB), HTTP keep-alive connections, and the bottleneck is CPU/event-loop, not disk or network. Serving large files (video, large images) drops this dramatically because the bottleneck shifts to disk I/O or network bandwidth.
  • App server (5K–20K req/sec): assumes in-memory computation with no external I/O per request. The moment you add a DB call or API call, each request blocks for 1–10 ms, and throughput drops to 500–2,000 req/sec (see the "heavy logic" row).
  • DB reads (10K–50K/sec): assumes simple indexed point lookups (e.g., SELECT * FROM users WHERE id = ?) and that the working set fits in the buffer pool (RAM). Complex joins, full-table scans, or working sets larger than RAM drop this to 100–1,000 reads/sec.
  • DB writes (1K–10K/sec): assumes row-level inserts/updates with a reasonable number of indexes (<5). Each additional index adds overhead per write. fsync on every commit (durable writes) is at the low end; batching commits or relaxing durability pushes toward the high end.
  • Redis (100K–200K ops/sec): assumes simple GET/SET on small values (<1 KB). Pipelining can push to 500K+. Large values, Lua scripts, or sorted-set operations (ZRANGEBYSCORE) reduce throughput to 10K–50K ops/sec.
  • Kafka (100K–500K msgs/sec per broker): assumes small messages (<1 KB), replication factor 3, and acks=1. With acks=all (strongest durability), throughput drops ~30–50%. Large messages (100 KB+) reduce throughput proportionally due to network and disk bandwidth limits.

Server Capacity: The "How Many Machines?" Math

The most common BOE question: given a workload, how many servers do you need? Here's the framework.

Typical Server Specs (Cloud, 2024)

Resource Standard High-Perf Notes
CPU8–16 cores64–128 coresEach core: ~1B simple ops/sec
RAM32–64 GB256–512 GBRedis instance: usually 32–64 GB used
Disk (SSD)500 GB – 2 TB4–16 TBNVMe for IOPS-heavy workloads
Network1–10 Gbps25–100 Gbps

The Formula

For any resource (CPU, memory, disk, network), the number of servers needed is:

servers_needed = total_demand / capacity_per_server

Then multiply by a safety factor:
  servers_deployed = servers_needed × 1.3  (30% headroom)

And for availability:
  servers_deployed = servers_needed × replication_factor (usually 3)

Take the max across all resource dimensions (CPU, memory, disk, network).

Worked Example: 10M Users, 100 QPS Average

Given:
  10M users, each user profile = 1 KB
  100 QPS average reads, 10 QPS writes
  Want < 50ms p99 latency

Storage:
  10M × 1 KB = 10 GB (fits on one machine easily)

Memory (see "80/20 Rule" section below for why we use 20%):
  Hot data = assume 20% of users active = 2M × 1 KB = 2 GB cache
  One Redis instance handles this

Throughput:
  100 reads/sec → one PostgreSQL server handles ~10K reads/sec → fine
  10 writes/sec → easily within one server

Result: 1 primary + 2 replicas (for availability) = 3 DB servers
  Plus 1 Redis server for caching
  Plus 2 app servers behind a load balancer (for redundancy)
  Total: ~6 servers (quite modest)

Assumptions in this worked example:

  • "20% of users active" means 20% of the 10M total users are DAU. This is a typical ratio for consumer apps. B2B/SaaS apps often have higher DAU/MAU ratios (40–60%). Social media varies (30–50% for mature platforms).
  • 100 QPS from 10M users implies each DAU makes ~5 requests/day (2M DAU × 5 req/user = 10M requests/day ÷ 100K sec = 100 QPS). Highly interactive apps (messaging, feeds) can see 50–200 requests/user/day.
  • 10:1 read-to-write ratio is typical for content-heavy platforms (social media, blogs, e-commerce). Messaging apps can be closer to 1:1. Analytics pipelines can be 1:100 (write-heavy).
  • "2 app servers for redundancy" means the system can survive one app server failing. This is a minimum for availability — production systems often use N+2 (two spare) to handle a failure during a deploy.

Bandwidth Calculations

How much network capacity does your system need? Multiply QPS by response size.

The Formula

Assumption: This formula calculates egress (outbound) bandwidth only — the data your servers send back to clients. Ingress (inbound) is usually much smaller for read-heavy workloads (small HTTP requests in, large responses out). For write-heavy workloads (file uploads, log ingestion), calculate ingress separately. Also note: this is application-layer throughput. TCP/IP headers, TLS overhead, and HTTP framing add ~5–15% on top, so a "1 Gbps" NIC delivers ~850–950 Mbps of application data in practice.

bandwidth = QPS × average_response_size

Example: Serving API responses
  10,000 QPS × 5 KB/response = 50,000 KB/sec = 50 MB/sec = 400 Mbps
  → One 1 Gbps NIC handles this (at 40% utilization)

Example: Serving images
  5,000 QPS × 200 KB/image = 1,000,000 KB/sec = 1 GB/sec = 8 Gbps
  → Need a 10 Gbps NIC or multiple servers

Example: Video streaming (1080p)
  1,000 concurrent streams × 5 Mbps/stream = 5 Gbps
  → Need multiple edge servers, CDN

Storage Calculations

How much disk space does your system need over time? Multiply daily ingestion rate by retention period.

The Formula

total_storage = daily_new_data × retention_days × replication_factor

Example: Twitter-like service
  500M tweets/day × 250 bytes/tweet = 125 GB/day of tweet data
  × 365 days retention = ~45 TB/year
  × 3 (replication) = ~135 TB/year

Example: Photo-sharing service
  10M photos/day × 500 KB/photo = 5 TB/day
  × 365 days = ~1.8 PB/year
  × 3 (replication) = ~5.4 PB/year (this is why CDNs exist)

Example: Logging / metrics
  100K servers × 10 KB/sec of logs = 1 GB/sec = 86 TB/day
  × 30 days retention = ~2.5 PB
  → Need to compress (typically 10:1), so ~250 TB actual storage

Where do these example numbers come from?

  • 500M tweets/day: Derived from an assumed 200M DAU × ~2.5 posts per active user per day. Real Twitter/X reportedly sees 500M–700M tweets/day. The 250 bytes/tweet assumes text + metadata (user_id, timestamp, IDs) — media (images, video) is stored separately in blob storage and would be 1,000× larger per item.
  • 10M photos/day: Derived from an assumed 50M DAU × 20% posting rate. The 500 KB/photo assumes compressed JPEG at moderate resolution. Platforms that store multiple resolutions (thumbnail + medium + original) multiply this by 3–5×.
  • 100K servers × 10 KB/sec of logs: The 100K servers assumes a large-scale company (Google, Netflix scale). The 10 KB/sec per server assumes structured application logs at moderate verbosity — debug-level logging can easily be 100 KB/sec. The 10:1 compression ratio assumes text logs with gzip/zstd; structured binary formats (Protobuf) can achieve 20:1 or better.
  • Replication factor of 3: The standard for most distributed storage systems (HDFS, Cassandra, Kafka). This is a durability/availability choice — 3 copies means you tolerate 2 simultaneous disk/node failures. Some systems use erasure coding (e.g., Reed-Solomon 6+3) for ~1.5× overhead instead of 3×, at the cost of higher CPU for encoding/decoding.
  • 365 days retention: Assumed for user-generated content that must be preserved. Logs and metrics typically have much shorter retention (7–90 days). Time-series databases often use tiered retention: full resolution for 7 days, downsampled for 1 year, aggregates kept indefinitely.

Memory & Cache Sizing

How much RAM or cache do you need? The key insight: you rarely cache everything. Most workloads follow a power law where a small fraction of items receives most of the traffic.

The 80/20 Rule (Pareto Principle)

Assumption: power-law (Zipfian) access distribution. The 80/20 rule assumes that item popularity follows a power-law distribution — a small number of items are very popular, while most items are rarely accessed. This holds for social media feeds (viral posts dominate), e-commerce (bestsellers), and web content (popular pages). It does not hold for uniform-random access patterns (e.g., a lookup table where every key is equally likely) — in that case, caching 20% of data serves only ~20% of reads. Before applying this rule, ask: "Are some items much more popular than others?" If the answer is no, use linear cache-hit scaling instead.

If 20% of items get 80% of traffic:
  Cache the hot 20% and you serve 80% of reads from cache.

For a 10 TB dataset:
  Cache size = 10 TB × 0.20 = 2 TB
  → Could fit in ~64 Redis instances (32 GB each)
  → Or a few machines with 256 GB RAM each

For a 100 GB dataset:
  Cache size = 100 GB × 0.20 = 20 GB
  → Fits in a single Redis instance

Cache Hit Rate vs Cache Size

Diminishing returns: going from 10% cache to 20% cache gives huge gains. Going from 80% to 90% is much less impactful per GB added.

Cache Size (% of data) Approximate Hit Rate DB Load Reduction
1%~20–40%Moderate
5%~50–70%Significant
20%~80–90%Large
50%~95–98%Very large
100%~99%+Nearly all (cache misses only on new/evicted data)

Assumptions in the cache hit rate table:

  • Power-law (Zipfian) access distribution — a small fraction of items receives most of the traffic. The exact hit rates depend on the Zipf exponent (s ≈ 1.0 is typical for web workloads). A steeper exponent (s > 1) means even smaller caches work well; a flatter one (s < 0.8) makes caching less effective.
  • Perfect cache admission — assumes your eviction policy (LRU, LFU, etc.) correctly keeps the hottest items in cache. A poorly-tuned policy (e.g., LRU under scan workloads) can give significantly worse hit rates than this table suggests.
  • Uniform-random access would give linear scaling: caching 20% of data → ~20% hit rate, 50% → ~50%, etc. If your workload is uniform-random, these numbers do not apply.
  • Cache warm-up is not accounted for. A cold cache starts at 0% hit rate and takes time to warm up to steady state. The time depends on traffic volume and cache size — for a 20 GB cache at 10K reads/sec with 1 KB responses, warm-up takes ~2,000 seconds (~33 min) to fill the cache.

Availability Math: The Nines

When someone says "five nines," what does that actually mean in downtime?

Availability "Nines" Downtime / Year Downtime / Month Downtime / Day
99%2 nines3.65 days7.3 hours14.4 min
99.9%3 nines8.76 hours43.8 min1.44 min
99.95%3.5 nines4.38 hours21.9 min43.2 sec

99.999%5 nines5.26 min26.3 sec0.86 sec

Combining Availability

Assumptions: These formulas assume independent failures — each component fails independently of the others. In reality, correlated failures are common: a bad deploy takes down all instances of a service, a network partition isolates an entire rack, or a cloud provider AZ outage takes down multiple components simultaneously. Independent-failure math gives optimistic availability estimates. Also, the "parallel" formula assumes perfect failover with zero detection/switchover time — in practice, failover takes seconds to minutes, during which the system is unavailable.

Components in series (ALL must be up):
  A_total = A1 × A2 × A3

  Example: Web → App → DB, each 99.9%
  A_total = 0.999 × 0.999 × 0.999 = 0.997 = 99.7% (~26 hours downtime/year)
  Three-nines components in series give less than three nines!

Components in parallel (ANY can serve, e.g., replicas):
  A_total = 1 - (1 - A1) × (1 - A2)

  Example: 2 replicas, each 99.9%
  A_total = 1 - (0.001 × 0.001) = 1 - 0.000001 = 99.9999% (~31 sec/year)
  Two three-nines replicas give six nines!

Combined example: 2 app servers (parallel) → 1 DB (series)
  A_app = 1 - (1 - 0.999)² = 0.999999
  A_total = A_app × A_db = 0.999999 × 0.999 = 0.998999 ≈ 99.9%

Common Estimation Patterns

These patterns come up repeatedly across system design scenarios. Internalize them and you can estimate anything.

Pattern 1: "How Many Servers for X QPS?"

servers = peak_QPS / throughput_per_server

API servers:    peak_QPS / 5,000  (conservative, with DB calls)
Cache servers:  peak_QPS / 100,000 (Redis)
DB servers:     peak_QPS / 10,000 (reads) or peak_QPS / 5,000 (writes)

Example: 50K QPS peak API traffic
  50,000 / 5,000 = 10 app servers
  × 1.3 headroom = 13 app servers

Pattern 2: "How Much Storage for N Years?"

storage = items_per_day × bytes_per_item × 365 × years × replication

Example: Chat app, 1B messages/day, 200 bytes each, 5 years, 3× replication
  1B × 200 B = 200 GB/day
  × 365 × 5 = 365 TB
  × 3 = ~1.1 PB

  Derivation of 1B messages/day:
  Assume 100M DAU, each sends ~10 messages/day = 1B messages/day.
  200 bytes/message = sender_id (8B) + recipient_id (8B) + timestamp (8B)
    + message text (~150 bytes avg) + metadata (flags, status, ~26 bytes).

Pattern 3: "Will This Fit in Memory?"

total_memory = num_items × size_per_item

If total_memory < 64 GB → fits on one machine
If total_memory < 500 GB → fits on one big machine
If total_memory < 10 TB → need ~200 Redis instances (sharded)
If total_memory > 10 TB → probably shouldn't be all in memory

Example: 100M user sessions × 2 KB each = 200 GB
  → 4–7 Redis instances (32–64 GB each)

Pattern 4: "Fan-out Cost"

total_writes = events_per_sec × avg_fan_out_factor

Example: Social media, 1K posts/sec, average user has 500 followers
  Fan-out writes = 1,000 × 500 = 500K writes/sec to timelines

But: if a celebrity has 10M followers:
  One post = 10M writes. At 1 KB per timeline entry = 10 GB of writes.
  Time to fan out at 100K writes/sec = 100 seconds (too slow!)
  → This is why hybrid fan-out exists

Assumptions in the fan-out example:

  • 1K posts/sec: Derived from an assumed ~100M DAU × ~1 post per user/day = 100M posts/day ÷ 100K sec ≈ 1,000 posts/sec.
  • 500 average followers: This is the mean, but follower counts follow a heavy-tailed distribution (most users have <100 followers, a few celebrities have millions). The median is much lower (~50–100). Using the mean overstates the cost for typical posts and understates it for celebrity posts — which is exactly why hybrid fan-out exists (fan-out on write for normal users, fan-out on read for celebrities).
  • 100K writes/sec throughput: Assumes a single Redis or Cassandra instance handling timeline writes. In practice you'd shard timelines across many nodes, increasing aggregate throughput — but the celebrity problem remains because those 10M writes still go to 10M different shards.

Pattern 5: "Connection/Thread Pool Sizing"

Assumption: This uses Little's Law (L = λ × W), which holds for any stable system in steady state. It assumes arrivals don't exceed capacity (if they do, the queue grows without bound and the formula breaks). It also assumes one connection per in-flight request — connection multiplexing (HTTP/2, gRPC) changes the math because many requests share one connection.

connections_needed = QPS × avg_latency_seconds

Example: 5,000 QPS to a DB, avg query latency = 5ms
  5,000 × 0.005 = 25 concurrent connections needed
  Pool size = 25 × 1.5 (headroom) ≈ 40

Example: 5,000 QPS, but latency spikes to 500ms under load
  5,000 × 0.5 = 2,500 concurrent connections
  → Connection pool explodes, cascading failure!
  This is why latency spikes are so dangerous.

Pattern 6: "Queue Depth / Worker Sizing"

workers_needed = arrival_rate × avg_processing_time

Example: 1,000 tasks/sec, each takes 2 sec to process
  Workers = 1,000 × 2 = 2,000 workers needed to keep up

Queue depth at steady state (Little's Law):
  L = λ × W
  L = average items in system
  λ = arrival rate
  W = average time in system (wait + processing)

If workers = 2,000 and arrival = 1,000/sec:
  Processing time = 2 sec, no waiting → L = 1,000 × 2 = 2,000 items in system

If workers = 500 (under-provisioned):
  Queue builds up by 500/sec (1,000 arriving - 500 processing)
  After 10 minutes: 300,000 items queued → latency = 300K / 500 = 600 sec wait

Hash & Collision Math

ID generation, hash tables, Bloom filters — these all depend on collision probability.

The Birthday Problem

Probability of at least one collision after k items in a space of N:
  P(collision) ≈ 1 - e^(-k² / 2N)

For a 50% chance of collision:
  k ≈ 1.2 × √N

Example: 64-bit random IDs (N = 2^64 ≈ 1.8 × 10^19)
  50% collision at k ≈ 1.2 × √(1.8 × 10^19) ≈ 5 billion IDs
  → Safe for most systems, but not for multi-decade URL shorteners

Example: 128-bit UUIDs (N = 2^128 ≈ 3.4 × 10^38)
  50% collision at k ≈ 2.2 × 10^19 (22 quintillion)
  → Effectively collision-free for any real workload

Bloom Filter Sizing

Given n items and desired false positive rate p:
  Optimal bit array size: m = -(n × ln(p)) / (ln(2))²
  Optimal hash functions: k = (m/n) × ln(2)

Example: n = 1 billion items, p = 1% false positive rate
  m = -(10^9 × ln(0.01)) / (ln(2))²
  m = -(10^9 × -4.605) / 0.4805
  m ≈ 9.585 × 10^9 bits ≈ 1.2 GB
  k = (9.585 × 10^9 / 10^9) × 0.693 ≈ 7 hash functions

At 0.1% FP rate: m ≈ 1.8 GB, k ≈ 10
At 0.01% FP rate: m ≈ 2.4 GB, k ≈ 13

Consensus & Replication Math

How many nodes do you need for quorum? What's the latency cost of replication?

Quorum Math

For a cluster of N nodes:
  Majority quorum = ⌊N/2⌋ + 1

  N=3: quorum = 2, tolerates 1 failure
  N=5: quorum = 3, tolerates 2 failures
  N=7: quorum = 4, tolerates 3 failures

General: N nodes tolerates ⌊(N-1)/2⌋ failures

For Raft/Paxos: writes need majority, so write latency =
  time until ⌊N/2⌋+1 nodes acknowledge (NOT all N)

Replication Latency

Synchronous replication:
  Write latency = local write + slowest replica RTT
  In same DC: local (1ms) + replica (0.5ms RTT) ≈ 1.5ms
  Cross-region: local (1ms) + replica (100ms RTT) ≈ 101ms → painful!

Asynchronous replication:
  Write latency = local write only ≈ 1ms
  Replication lag = typically 10ms–1s (same DC), seconds–minutes (cross-region)
  Risk: data loss if primary fails before replication completes

Assumptions in the replication latency numbers:

  • "1ms local write" assumes an SSD-backed database with fsync on commit (durable write). Without fsync (e.g., writing to OS page cache only), local writes can be ~0.01ms — but you risk data loss on power failure. With spinning disks, a durable write takes ~5–10ms (disk seek + rotation).
  • "0.5ms replica RTT" assumes replicas are in the same datacenter on a low-latency network (<0.5ms RTT). Replicas in a different availability zone within the same region typically add 1–2ms RTT.
  • "100ms cross-region RTT" is typical for US East → US West (~60–80ms) or US → Europe (~80–120ms). US → Asia-Pacific is ~150–250ms. These are speed-of-light constraints that no amount of hardware can improve.
  • Synchronous replication latency = "slowest replica" assumes all replicas must acknowledge (acks=all). With majority quorum (e.g., Raft), the write completes when ⌊N/2⌋+1 replicas acknowledge, so latency = the median replica RTT, not the slowest.

The BOE Checklist

When you start a system design problem, run through this checklist to frame the numbers:

  1. Users & traffic: How many users? DAU vs MAU? Peak QPS? Use the "divide by 100K" rule.
  2. Data size: How big is each record? How many records? Total dataset size?
  3. Storage growth: How much new data per day? Retention period? Replication factor?
  4. Memory: What fraction of data is hot? Will the working set fit in RAM?
  5. Bandwidth: QPS × response size. Will the NIC be the bottleneck?
  6. Server count: For each resource dimension (CPU, RAM, disk, network), how many servers?
  7. Latency budget: End-to-end SLA. How much can each component use? Where's the bottleneck?
  8. Availability: What's the target? How many replicas? What happens during failure?

Pro tip: In an interview, state your assumptions explicitly. "I'm assuming 10M DAU, 10:1 read-to-write ratio, and 1 KB average record size. Does that sound reasonable?" This shows you know how to frame the problem, and the interviewer can correct you if needed.

Key Takeaways

  • 1 day ≈ 105 seconds. This is the single most useful conversion for QPS estimates.
  • Each latency rung is ~100× the previous. RAM → SSD → disk/network → cross-continent.
  • A single modern server can handle more than you think. 10K QPS for an API, 100K for Redis, 50K reads for PostgreSQL. Don't over-shard.
  • Cache 20% of data, serve 80% of reads. Power-law distributions are your friend.
  • Components in series multiply failure rates. Three 99.9% services in series = 99.7%. Redundancy is a must.
  • Always check: CPU, memory, disk, network. The bottleneck is whichever resource runs out first.
  • State your assumptions. No estimate is useful without stated assumptions. Round aggressively — the goal is order-of-magnitude correctness.
