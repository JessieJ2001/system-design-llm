SQL vs NoSQL

Relational databases and non-relational databases are not better or worse than each other — they are fundamentally different tools optimized for different access patterns, consistency requirements, and scaling strategies.

30 min read

What Is It?

SQL databases (also called relational databases) store data in tables with predefined schemas. Every row in a table has the same columns. Relationships between tables are expressed through foreign keys and realized through JOIN operations. SQL databases guarantee ACID properties — Atomicity, Consistency, Isolation, and Durability — meaning your transactions are reliable, and the database will never leave your data in a half-committed state. The most popular examples are PostgreSQL, MySQL, Oracle, and SQL Server.

NoSQL databases (non-relational databases) is an umbrella term for any database that does not use the traditional tabular relational model. There are four major categories, each optimized for a different access pattern:

  • Document stores (MongoDB, CouchDB) — Store data as flexible JSON/BSON documents. Each document can have a different structure. Great for content management, product catalogs, and user profiles where the schema varies or evolves frequently.
  • Key-value stores (Redis, DynamoDB, Memcached) — The simplest model: a key maps to a value (which can be a string, list, set, or blob). Optimized for ultra-low-latency lookups. Ideal for session data, caching, and feature flags.
  • Column-family stores (Cassandra, HBase, ScyllaDB) — Data is organized by column families rather than rows. Designed for write-heavy workloads and time-series data where you write many columns per row key and read specific column ranges. Excel at massive-scale distributed writes.
  • Graph databases (Neo4j, Amazon Neptune, TigerGraph) — Store data as nodes and edges (relationships). Optimized for traversing relationships — "find all friends of friends who live in New York" is a single query, not a chain of joins. Essential for social networks, recommendation engines, and fraud detection.

The critical insight: this is not a "which is better" question. It is a "which tool fits this particular job" question. Many production systems use multiple database types simultaneously — a pattern called polyglot persistence. Your order processing might live in PostgreSQL while your product search index lives in Elasticsearch and your session data lives in Redis.

MongoDB vs Postgres JSONB

Neo4j vs everything else (and what ">2 hops" means)

  • "Hops"how many relationships a query traverses. SQL handles 1 hop (one JOIN) or 2 hops (two JOINs) fine. Past 2 hops, JOIN cost multiplies; past 3, the planner gives up.
  • Use cases > 2 hopsfriend-of-friend-of-friend recs; fraud rings (account → device → account → payment → account); shortest-path ("you are 3 connections away from"); transitive impact analysis.
  • Neo4jnative graph storage; index-free adjacency — each edge is a physical pointer, so 1000 hops = 1000 pointer dereferences (not 1000 index lookups). Cypher query language.
  • SQL fallbackrecursive CTEs (Common Table Expressions: WITH RECURSIVE x AS (... UNION ALL ... FROM x ...) — the query references itself to walk a hierarchy).
  • Decision: 1–2 hops — stay in SQL with recursive CTEs. 3+ hops, large graph, hot path — use Neo4j.

Elasticsearch vs Postgres tsvector

  • Postgres tsvectorfull-text search inside Postgres; GIN index on a tsvector column. Fine up to ~10M docs at modest QPS.
  • Elasticsearch / OpenSearchpurpose-built: Lucene inverted index per field; BM25 relevance; faceted aggregations; near-real-time indexing; kNN for vectors.
  • Critical caveatES is never source of truth: no transactions; refresh interval makes reads stale; lossy on crash without proper translog config.
  • Decision: PG tsvector if data already lives in PG and volume is modest. ES when search volume, relevance tuning, or faceted aggregations justify the ops cost — with PG / DynamoDB as truth and CDC into ES.

How It Works

The Relational Model (SQL)

Relational databases are built on E. F. Codd's relational model (1970). Data is organized into relations (tables), each with a fixed schema:

  1. Schema definition — You define tables with typed columns upfront: CREATE TABLE users (id INT PRIMARY KEY, name VARCHAR(100), email VARCHAR(255) UNIQUE). The schema is enforced — you cannot insert a row that violates column types or constraints.
  2. Normalization — Data is decomposed to eliminate redundancy. A user's address goes in an addresses table linked by user_id, not duplicated in every order row. This ensures a single source of truth — update the address once and it is correct everywhere.
  3. Joins — To reassemble related data, you use JOIN queries: SELECT u.name, a.city FROM users u JOIN addresses a ON u.id = a.user_id. The database optimizer decides the most efficient execution plan (hash join, nested loop, merge join).

Normalization: Why It Exists and How It Works

Normalization solves a specific problem: update anomalies. When the same data is stored in multiple places, updating it in one place but forgetting another leaves the database in an inconsistent state. Let us see this concretely.

The Denormalized Starting Point

Suppose we store all order data in a single flat table:

order_idcustomer_namecustomer_addressproduct_nameprice

Three anomalies emerge:

  • Update anomaly: Alice moves to "789 Pine Rd, LA." You must update her address in rows 1001, 1002, and 1004. Miss one row and the database now contains two different addresses for Alice — which is correct? With 10,000 orders, this is error-prone.
  • Insertion anomaly: You want to add a new product "Monitor, $350" to the catalog, but there are no orders for it yet. You cannot insert a product without creating a fake order, because every row requires an order_id.
  • Deletion anomaly: If you delete order 1003 (Bob's only order), you lose all information about Bob — his name and address vanish from the database entirely.
Normal forms: the progression to 3NF

Codd's relational model defines a sequence of normal forms: progressively stricter rules that eliminate redundancy. Each level subsumes the previous one. Most production schemas target Third Normal Form (3NF) — it eliminates the three anomalies above without going so far that joins become painful. Higher forms (BCNF, 4NF, 5NF) exist but are usually overkill outside academic settings.

  • 1NF (First Normal Form) — atomic values: every cell holds a single, indivisible value. No comma-separated lists in one cell, no repeating column groups (phone1, phone2, phone3), no JSON dumped into a "data" column. Violation example: a tags column with the value "red, urgent, refund". Fix: put each tag in its own row in a related tags table.
  • 2NF (Second Normal Form) — full functional dependence on the primary key: 1NF plus: every non-key column must depend on the whole primary key, not just part of it. Only matters when you have a composite PK. Violation example: table with PK (order_id, product_id) and a column product_name. The product name depends only on product_id, not the full PK. Fix: move product_name to a products table keyed on product_id.
  • 3NF (Third Normal Form) — no transitive dependencies: 2NF plus: non-key columns must depend directly on the PK, not on another non-key column. Violation example: in the flat orders table above, customer_address depends on customer_name (a non-key column), which depends on order_id. That's a transitive chain: order_id → customer_name → customer_address. Fix: move customer info to its own table; the orders table just stores customer_id.
  • The 3NF rule of thumb: "every non-key column depends on the key, the whole key, and nothing but the key — so help me Codd." Once a schema is in 3NF, the three anomalies disappear automatically.
  • Higher forms (FYI, rarely used in practice): BCNF strengthens 3NF by also requiring every dependency's left side to be a superkey. 4NF eliminates multi-valued dependencies. 5NF eliminates join dependencies. In an interview, "we normalize to 3NF" is almost always sufficient; "we denormalize selectively for read-heavy workloads" is the next sentence.

The tradeoff: Reads now require JOINs. To get "Alice's orders with product names and prices," we must join three tables. This is slower than reading a single denormalized row. This is exactly why NoSQL document stores embed related data — they optimize for read speed at the cost of update complexity. The choice depends on whether your workload is read-heavy or write-heavy, and whether data consistency or read performance is your priority.

Join Algorithms: How the Database Executes JOINs

When you write SELECT * FROM users u JOIN orders o ON u.id = o.user_id, the database must match every order to its user. The algorithm it chooses has an enormous impact on performance. Consider two tables: Users (1 million rows) and Orders (10 million rows), joining on user_id.

Nested Loop Join: The Brute-Force Approach

For each row in the outer table, scan every row in the inner table looking for matches:

for user in users:          # 1M iterations
    for order in orders:     # 10M iterations each
        if user.id == order.user_id:
            emit(user, order)

Cost: 1M × 10M = 10 trillion comparisons. At 1 billion comparisons per second, this takes ~10,000 seconds (nearly 3 hours). This is O(n × m) and completely impractical for large tables. The database only uses nested loop joins when one side is very small (e.g., a single row from a WHERE clause) or when an index is available on the inner table (making each inner lookup O(log n) instead of O(n)).

Hash Join: The Smart Approach

Build a hash table from the smaller table, then scan the larger table, looking up each key in the hash table:

# Phase 1: Build hash table from smaller table (Users)
hash_table = {}
for user in users:                         # 1M iterations
    hash_table[user.id] = user             # O(1) insert

# Phase 2: Probe hash table with larger table (Orders)
for order in orders:                        # 10M iterations
    user = hash_table.get(order.user_id)   # O(1) lookup
    if user:
        emit(user, order)

Cost: 1M inserts into hash table + 10M lookups = 11 million operations. At 100M operations per second, this takes ~0.1 seconds. That is 100,000x faster than the nested loop. The time complexity is O(n + m), and the space complexity is O(n) for the hash table (where n is the smaller table).

  1. ACID transactions — Multiple operations can be grouped atomically. Transfer money from account A to B: debit A, credit B — both happen or neither does. Isolation levels (Read Committed, Repeatable Read, Serializable) control how concurrent transactions interact.

ACID Properties: A Complete Walkthrough

ACID is an acronym for four guarantees that relational databases provide for every transaction. To understand why each property matters, let us walk through a concrete example: transferring $100 from Account A (balance: $500) to Account B (balance: $300).

-- The transaction we want to execute:
BEGIN;
  UPDATE accounts SET balance = balance - 100 WHERE id = 'A';  -- Debit A: $500 → $400
  UPDATE accounts SET balance = balance + 100 WHERE id = 'B';  -- Credit B: $300 → $400
COMMIT;
Atomicity: All or Nothing

What it means: Either both the debit and credit happen, or neither does. There is no state where money is debited from A but not credited to B.

What goes wrong without it: Imagine the power fails right after the debit but before the credit. Account A now has $400 and Account B still has $300. The total went from $800 to $700 — $100 vanished into thin air. The customer lost money.

How the database implements it: The database writes all changes to a write-ahead log (WAL) before modifying actual data pages. If a crash occurs mid-transaction, on restart the database reads the WAL and rolls back any uncommitted changes. The debit to A is undone, restoring it to $500.

StepActionAccount AAccount BTotal
1BEGIN transaction$500$300$800
2Debit A by $100$400$300$700 (temporary)
3aCrash here (no atomicity)$400$300$700 — $100 lost!
3bWith atomicity: rollback$500$300$800 — safe
3Credit B by $100 (normal path)$400$400$800
4COMMIT$400$400$800 — correct
Consistency: Rules Are Never Broken

What it means: Every transaction moves the database from one valid state to another. Database constraints (CHECK, FOREIGN KEY, UNIQUE, NOT NULL) are always satisfied after a transaction commits.

What goes wrong without it: Suppose we have a constraint: CHECK (balance >= 0). If Account A has $50 and we try to transfer $100, the balance would go to -$50. Without consistency enforcement, the database would allow this — money was created from nothing.

-- This transaction violates the CHECK constraint and is REJECTED:
BEGIN;
  UPDATE accounts SET balance = balance - 100 WHERE id = 'A';
  -- Account A balance would be $50 - $100 = -$50
  -- ERROR: new row for relation "accounts" violates check constraint "balance_non_negative"
ROLLBACK;  -- Entire transaction is aborted, A stays at $50
Isolation: Concurrent Transactions Cannot Interfere

What it means: Even when multiple transactions run concurrently, each behaves as if it were the only one running. The database uses locks or multi-version concurrency control (MVCC) to prevent interference.

What goes wrong without it (the "lost update" problem): Two transfers happen simultaneously — Transfer 1 moves $100 from A to B, Transfer 2 moves $50 from A to C. Both read A's balance as $500 at the same instant.

TimeTransfer 1 (A→B, $100)Transfer 2 (A→C, $50)A's Actual Balance
T1READ A.balance → $500—$500
T2—READ A.balance → $500$500
T3WRITE A.balance = 500 - 100 = $400—$400
T4—WRITE A.balance = 500 - 50 = $450$450 — wrong!

Transfer 1's debit was overwritten. Account A should have $350 ($500 - $100 - $50) but instead has $450. Account A got $100 for free. With proper isolation (e.g., Serializable level), the database detects the conflict and aborts one of the transactions, forcing it to retry with the updated balance.

Durability: Committed Data Survives Crashes

What it means: Once a transaction is committed and the database returns success to the client, the data is permanently saved — even if the server immediately crashes, loses power, or the disk fails.

What goes wrong without it: You transfer $100, the database says "COMMIT successful," you see the new balance in the app. Then the server crashes. When it restarts, your balance is back to the old value. The $100 transfer vanished despite confirmation.

How the database implements it: Before returning COMMIT to the client, the database flushes the WAL to durable storage (fsync). Even if the data pages in memory have not been written to disk yet, the WAL contains enough information to replay the committed transaction on restart. Replication to standby servers provides durability against disk failure.

B-Tree Indexes: How They Work

A B-tree (balanced tree) is the default index structure in virtually every SQL database. It is not a binary tree — each node can hold many keys and have many children. This is a deliberate design choice because databases live on disk, and disk I/O is the bottleneck. By packing many keys into each node (typically matching the disk page size of 4–16KB), each disk read fetches an entire node, and we need very few reads to find any key.

Structure: In a B-tree with branching factor b (also called "order"), each internal node holds up to b−1 keys and has up to b children. Keys within each node are sorted. All leaf nodes are at the same depth. Here is a B-tree with branching factor 4 (each node holds up to 3 keys and has up to 4 children), storing 15 keys:

                        [25 | 50 | 75]                    ← Root (level 0)
                       /    |      |    \
             [5|10|15]  [30|35|40]  [55|60|65]  [80|85|90]  ← Leaves (level 1)

Lookup example — finding key 42:

  1. Start at the root: Compare 42 with root keys [25, 50, 75]. 42 is between 25 and 50, so follow the second child pointer (between 25 and 50). Disk read #1.
  2. Reach child node [30, 35, 40]: Compare 42 with [30, 35, 40]. 42 is greater than 40, so it would be in the next child — but this is a leaf node and 42 is not present. Key 42 does not exist. Disk read #2.

We found our answer in 2 disk reads. With a binary tree holding 15 keys, we would need up to 4 reads (log2(15) ≈ 4).

Why B-Trees Scale: The O(log n) Derivation

The depth of a B-tree determines how many disk reads a lookup requires. With branching factor b and n keys, the depth is logb(n). In real databases, the branching factor is much larger than 4:

This is why adding an index to a table transforms a slow sequential scan (read all 1M rows) into a near-instant lookup.

Insert and Node Splitting

When inserting a key into a full node (one that already has the maximum number of keys), the node must split. Here is what happens when we insert key 37 into our tree:

  1. Navigate to the correct leaf: [30, 35, 40]. This node already has 3 keys (the maximum for branching factor 4).
  2. Insert 37 to get [30, 35, 37, 40] — this is one key too many.
  3. Split: The middle key (35) is promoted to the parent. The leaf splits into [30] and [37, 40].
  4. The parent [25, 50, 75] now becomes [25, 35, 50, 75] — but this also has too many keys (4 keys in a max-3 node). The parent splits as well, promoting 50 to a new root.

Splits cascade upward in the worst case, but this is rare. Each insert is still O(log n) because we only touch nodes along the path from root to leaf.

The Document Model (NoSQL)

Document databases take a fundamentally different approach: store data the way the application uses it.

  1. Denormalized documents — Instead of splitting a user's data across three tables, store a single JSON document: { "name": "Alice", "email": "alice@example.com", "address": { "city": "SF", "zip": "94102" }, "orders": [...] }. One read fetches everything — no joins needed.
  2. Flexible schema — Documents in the same collection can have different fields. User A might have a phone field while user B does not. Schema evolution is trivial — add new fields without migrating existing documents.
  3. Embedded data and references — Frequently accessed together? Embed it in the document. Shared across many documents? Store a reference (like a foreign key). This is a modelling decision with major performance implications.
  4. Querying — Document databases support rich queries on nested fields: db.users.find({"address.city": "SF"}). But cross-document queries (equivalent to joins) are either unsupported or expensive. The data model must match your query patterns.

Key-Value Model

The simplest abstraction: GET(key) → value and PUT(key, value). Redis keeps data in memory for sub-millisecond reads. DynamoDB provides single-digit-millisecond reads at any scale with automatic partitioning. The trade-off: you can only query by key. If you need to search by any other attribute, you need a secondary index (which DynamoDB supports) or a separate search layer.

Column-Family Model

Think of it as a two-dimensional key-value store: data is addressed by a row key and a column name. Cassandra organizes data into partitions by hashing the partition key, then sorts rows within a partition by clustering columns. This makes it extremely efficient for time-series access patterns: "give me all sensor readings for device X between 2pm and 3pm" is a single sequential read of a partition. Writes are append-only (LSM trees), making Cassandra capable of handling hundreds of thousands of writes per second per node.

Why This Layout Matters for Performance

SQL (row-oriented): Rows for device-A, device-B, and device-C are interleaved on disk in insertion order. To fetch all of device-A's readings, the database must either scan every row (slow) or use a B-tree index to hop between scattered disk locations (random I/O).

Reading "all of device-A's data from 14:00 to 15:00" is a single sequential disk read — the fastest I/O pattern possible. This is why Cassandra dominates time-series and IoT workloads.

The tradeoff: Want to query "all devices where temp > 24"? In SQL, without an index this is a full table scan — every row is checked. But you can add a B-tree index on temp after the fact: the database builds a sorted structure that lets it jump directly to matching rows in O(log n), skipping everything below 24. You pay for that index on every write, but reads become fast without changing your schema or application code. In Cassandra, this query requires scanning every partition across every node (a cluster-wide scatter), because the data is partitioned by device_id, not by temp. Cassandra has secondary indexes, but they are local to each node and perform poorly at scale. You must know your query patterns at design time and model the data around them.

LSM-Trees: Why Cassandra Writes Are So Fast

An LSM-tree (Log-Structured Merge-tree) is a write-optimized data structure used by Cassandra, HBase, RocksDB, and LevelDB. It is fundamentally different from the B-tree used by SQL databases. While B-trees update data in place (random I/O to find and modify the right disk page), LSM-trees convert all writes into sequential I/O, which is 100–1000x faster on both HDDs and SSDs.

The full write lifecycle, step by step:

Step 1: Write to the memtable (in-memory)

Every write goes first to an in-memory sorted data structure called a memtable (typically a red-black tree or skip list). Writes are also appended to a commit log on disk for durability. Because the memtable is in memory and the commit log is append-only (sequential I/O), writes are extremely fast — typically sub-millisecond.

Step 2: Flush memtable to disk as an SSTable

When the memtable reaches a threshold size (typically 64MB in Cassandra), it is flushed to disk as an SSTable (Sorted String Table). An SSTable is an immutable file where keys are sorted. Because the memtable was already sorted in memory, writing the SSTable is a single sequential disk write — very fast.

Step 3: SSTables accumulate on disk

Over time, many SSTables pile up. Each represents a snapshot of writes from a specific time period. If key "user:42" was written 5 times across 5 flushes, there are 5 different SSTables that might contain data for that key.

Step 4: Compaction merges SSTables

A background process called compaction periodically merges multiple SSTables into one, discarding older versions of the same key and removing deleted data (tombstones). This is a merge-sort operation — since both input SSTables are already sorted, merging is efficient sequential I/O.

The Read Tradeoff

Reads in an LSM-tree can be slower than B-tree reads because a key might exist in any of several places: the current memtable, an older memtable being flushed, or any of the on-disk SSTables. In the worst case, a read must check all of them:

  1. Check the memtable (fast, in memory)
  2. Check each SSTable on disk, from newest to oldest, until the key is found

With 10 SSTables, that could mean 10 disk reads for a single key lookup. This is called read amplification and it is the main weakness of LSM-trees.

How Bloom Filters Rescue Read Performance

Each SSTable has an associated Bloom filter — a compact probabilistic data structure that can answer "is key X definitely NOT in this SSTable?" in O(1) time with no disk I/O. Before reading an SSTable from disk, the database checks its Bloom filter. If the Bloom filter says "not here" (which is always accurate for negatives), the SSTable is skipped entirely. In practice, Bloom filters eliminate 99%+ of unnecessary SSTable reads, bringing LSM-tree read performance close to B-tree performance for point lookups.

Bottom line: B-trees (SQL) are read-optimized — reads are fast (O(log n) with one tree), writes are slower (random I/O to update in place). LSM-trees (NoSQL column-family) are write-optimized — writes are fast (sequential I/O), reads may be slower (multiple SSTables to check). This is why Cassandra excels at write-heavy workloads like IoT sensor data and event logging.

Graph Model

Data is stored as nodes (entities) and edges (relationships) with properties on both. Neo4j uses index-free adjacency — each node physically stores pointers to its neighbors — so traversals are O(1) per hop regardless of graph size. This makes queries like "shortest path between user A and user B" or "all products purchased by friends of friends" efficient even in billion-node graphs. SQL can model graphs with join tables, but performance degrades dramatically beyond 3-4 hops because each hop requires another join.

When to Use / When NOT to Use

Use SQL (Relational) When

  • Complex queries with joins — Reporting, analytics, and any workload that needs ad-hoc queries across multiple related entities
  • ACID transactions are required — Financial systems, inventory management, anything where partial updates are unacceptable
  • Schema is well-defined and stable — The data model is known upfront and unlikely to change fundamentally
  • Data integrity is paramount — Foreign keys, unique constraints, check constraints enforce correctness at the database level
  • Moderate scale — Single-node or read-replica PostgreSQL handles most workloads up to ~100K QPS

Don't Use SQL When

  • Schema changes frequently — ALTER TABLE on a 500M-row table can lock it for hours. If your schema changes weekly, document stores are more forgiving.
  • You need horizontal write scaling — SQL databases scale reads easily (replicas) but scaling writes requires sharding, which is painful and often application-managed
  • Data is hierarchical/nested — Representing deeply nested JSON trees in normalized tables creates join explosions
  • You need global distribution with low latency — Traditional SQL databases struggle with multi-region writes (though NewSQL solves this)

Classic vs. Modern Approaches

The Classic Divide: MySQL/PostgreSQL vs MongoDB/Cassandra

For over a decade, the database landscape was neatly split. Need consistency and structured data? Use MySQL or PostgreSQL. Need scale and flexibility? Use MongoDB or Cassandra. This was a real tradeoff — you genuinely could not have both. MongoDB before version 4.0 had no multi-document transactions. PostgreSQL could not horizontally shard writes natively. You picked a side and lived with the consequences.

NewSQL: The Best of Both Worlds

NewSQL databases provide the SQL interface and ACID transactions of relational databases with the horizontal scalability of NoSQL. The key innovations:

  • Google Spanner (2012) — The pioneer. Uses TrueTime (atomic clocks + GPS) to provide externally consistent distributed transactions. Proves you can have strong consistency at global scale. The CAP theorem is not violated — Spanner chooses CP but achieves such high availability through its infrastructure that it behaves like CA in practice.
  • CockroachDB — Open-source Spanner-inspired database. Uses hybrid logical clocks instead of atomic clocks. PostgreSQL-compatible wire protocol. Automatically rebalances data across nodes. Survives region-level failures without downtime.
  • TiDB — MySQL-compatible NewSQL from PingCAP. Separates storage (TiKV, a distributed key-value store using Raft) from compute (TiDB SQL layer). Can serve as a drop-in MySQL replacement with horizontal scaling.

Blurring the Lines: Multi-Model Databases

Modern databases increasingly support multiple data models:

  • PostgreSQL as a document store — PostgreSQL's JSONB column type stores binary JSON with GIN indexing. You can query WHERE profile->>'city' = 'SF' with index support. For many workloads, PostgreSQL-with-JSONB eliminates the need for a separate document store entirely.
  • MongoDB with ACID — Since version 4.0, MongoDB supports multi-document transactions. Since 5.0, it supports time-series collections natively. The gap between MongoDB and SQL databases has narrowed significantly.
  • DynamoDB single-table design — Instead of one table per entity, DynamoDB power users put all entities into a single table with carefully designed partition keys and sort keys. This eliminates joins entirely (since DynamoDB cannot join) and optimizes for specific access patterns. It requires upfront query planning but achieves remarkable performance at scale.
  • Redis as a multi-model store — Redis supports strings, hashes, lists, sets, sorted sets, streams, and even graph data (via RedisGraph) and JSON documents (via RedisJSON). It has evolved far beyond simple key-value caching.

Assumptions & Model

Assumptions Behind the SQL Model

  • Schema is known upfront — You can define your tables and columns before writing data. Breaks down when: requirements change weekly, or you are ingesting semi-structured data from external sources where the schema is not under your control.
  • Relationships are important — Your data has meaningful relationships that need referential integrity. Breaks down when: data is largely independent (logs, events, sensor readings) or relationships are deeply nested.
  • Transactions are needed — Operations must be atomic and isolated. Breaks down when: eventual consistency is acceptable (analytics, non-critical reads) and the transaction overhead harms throughput.
  • Single-node or few-node is sufficient — Traditional SQL assumes vertical scaling or simple read replicas. Breaks down at: ~10TB+ data, ~50K+ write QPS, or multi-region active-active requirements.

Assumptions Behind the NoSQL Model

  • Schema will evolve — Flexibility is valued over enforcement. Breaks down when: data integrity issues creep in silently because nothing enforces constraints. "Schemaless" really means "schema is in the application code" — and application code has bugs.
  • Horizontal scaling is needed — NoSQL databases are designed to distribute data across many nodes. Breaks down when: your data fits on one server (most applications!) and you pay the complexity cost of distribution for no benefit.
  • Eventual consistency is acceptable — Many NoSQL databases default to eventual consistency for performance. Breaks down when: a user updates their profile and immediately sees stale data, or two concurrent purchases oversell inventory because neither sees the other.
  • Query patterns are known at design time — NoSQL databases require you to model data around your access patterns. Breaks down when: new query requirements emerge and the data model cannot support them without a costly migration.

Both models break when misapplied. A startup with 1,000 users that chooses Cassandra "for scale" will spend months fighting operational complexity they do not need. A rapidly growing company that puts everything in a single PostgreSQL instance will eventually hit a wall where vertical scaling runs out and sharding is an emergency project.

Sizing & Tradeoffs

The most common mistake in database selection is choosing based on data volume alone. Your SLOs — latency targets, throughput requirements, and consistency guarantees — should drive every sizing and technology decision. Start with the contract you owe your users, then work backwards to infrastructure.

SLO-Driven Sizing Examples

Example 1: User Profile Service

SLOs: p99 read latency < 5ms • 500K RPS reads • 10K RPS writes • 200M user profiles (~2KB each = ~400GB)

Analysis: The 50:1 read-to-write ratio and sub-5ms p99 target eliminate SQL at this RPS without heavy caching. At 500K RPS, a single PostgreSQL instance tops out around 50–80K simple reads/sec even with connection pooling. You would need 8–10 read replicas plus an application-level cache (Redis) in front, adding operational complexity and staleness risk.

Derivation: Use DynamoDB (on-demand or provisioned). Each partition handles ~3,000 RCU; 500K RPS of eventually-consistent 2KB reads = 250K RCU → ~84 partitions, well within DynamoDB's auto-split capabilities. p99 read latency is typically 3–4ms. Writes at 10K RPS = 10K WCU, comfortably handled. DAX (DynamoDB Accelerator) can reduce hot-key reads to <1ms if needed.

Instance sizing: DynamoDB on-demand — no instances to size. Estimated cost: ~$3,200/month (reads) + ~$4,700/month (writes) + ~$100/month (storage) = ~$8,000/month. Compare to a PostgreSQL fleet: 10x r6g.2xlarge replicas + primary + Redis cluster ≈ $12,000/month with higher operational burden.

Example 2: Product Search with Facets

SLOs: p95 search latency < 100ms • 50M products • 20+ filterable attributes • 10K searches/sec

Analysis: Faceted search (filter by brand + price range + rating + color + size) requires inverted indexes on every filterable field. SQL can handle this with composite indexes, but 20+ attributes means an explosion of index combinations — a B-tree index per attribute works, but multi-facet intersection is slow at 50M rows. A full-text/inverted index engine is purpose-built for this.

Derivation: Use Elasticsearch as the search layer backed by a MongoDB source-of-truth for product data. Elasticsearch inverted indexes handle arbitrary facet combinations in O(1) per posting list intersection. At 50M docs (~3KB each = ~150GB index), a 3-node cluster of 64GB RAM / 8 vCPU machines keeps hot shards in the filesystem cache.

Index strategy: 10 primary shards × 1 replica = 20 shards across 3 nodes. Each shard holds ~5M docs (~15GB). The _source field stores denormalized product data for display, avoiding a round-trip to MongoDB on every query. Facet aggregations use doc_values (column-oriented on disk) — no heap pressure. CDC via MongoDB Change Streams keeps Elasticsearch within 2–5s of the primary store.

Example 3: Time-Series IoT Data

SLOs: p99 write latency < 10ms • 1M events/sec ingest • 90-day retention • hourly rollup queries < 30s

Analysis: 1M writes/sec rules out any single-node solution. Each event is ~200 bytes, so daily ingest = ~17TB raw. 90-day retention = ~1.5PB. SQL databases cannot sustain 1M inserts/sec even with batching; PostgreSQL tops out at ~50K inserts/sec per instance. The write pattern is append-only (no updates), time-ordered, and partitioned naturally by device ID + timestamp.

Derivation: Use Apache Cassandra with a partition key of (device_id, day_bucket) and clustering key of timestamp. Cassandra’s LSM-tree storage engine is optimized for sequential writes — writes go to an in-memory memtable and commit log, then flush to SSTables asynchronously. p99 write latency is typically 2–5ms at this throughput.

Partition strategy: With 100K devices × 90 days = 9M partitions, each partition holds ~1 day of one device’s data (~1.7MB). This keeps partition sizes well under Cassandra’s recommended 100MB limit. A 12-node cluster of i3.2xlarge instances (8 vCPU, 61GB RAM, 1.9TB NVMe) with RF=3 provides ~4x the required write throughput for headroom. TTL of 90 days handles retention automatically — expired data is purged during compaction with no manual cleanup.

Example 4: Financial Reporting

SLOs: p95 report generation < 2s • Complex joins across 12+ tables • ACID required for all writes • 50M transactions/month • 7-year retention (regulatory)

Analysis: Financial reporting requires joining transactions, accounts, ledger entries, audit logs, and regulatory metadata in a single consistent snapshot. NoSQL is disqualified: denormalizing 12 interrelated tables creates update anomalies that violate audit requirements. ACID is legally mandated (SOX, PCI-DSS). The challenge is making complex analytical joins fast enough.

Derivation: Use PostgreSQL with a dedicated read replica fleet for reporting. The primary handles OLTP writes (~20 TPS for transaction processing). Reporting queries run on read replicas configured with hot_standby_feedback = on to prevent query cancellation from vacuum conflicts.

Sizing with read replicas: Primary: r6g.2xlarge (8 vCPU, 64GB RAM) handles write workload. Reporting replicas: 2x r6g.4xlarge (16 vCPU, 128GB RAM) — the extra memory keeps the working set (~80GB of hot indexes) in shared_buffers + OS page cache. Materialized views pre-compute common joins (daily account summaries, monthly P&L) and refresh every 15 minutes. Complex ad-hoc reports hitting raw tables target the p95 < 2s SLO by using covering indexes on (account_id, transaction_date) and partition pruning on the 7-year data set (range-partitioned by quarter = 28 partitions). Estimated cost: ~$2,800/month for the primary + 2 replicas + 4TB gp3 storage.

Capacity planning should start with SLOs, not hardware specs. Use this framework to translate business requirements into infrastructure decisions:

Step 2: Back-of-Envelope Capacity Math

Formula: Required instances = (Peak RPS × Safety margin) / (Single-instance throughput at target p99 latency)

The critical insight: a database instance can handle far more requests at p50 than at p99. If your SLO is p99 < 10ms, you must size for the throughput where the 99th percentile stays under 10ms — not the maximum throughput before the database falls over. Typically this means provisioning at 40–60% of maximum throughput.

Example: PostgreSQL on r6g.2xlarge handles ~80K simple reads/sec at p50 < 2ms, but only ~35K reads/sec when you constrain p99 < 5ms. If your SLO requires 100K RPS at p99 < 5ms, you need at least 3 read replicas (100K / 35K = 2.86 → 3) plus a 30% safety margin = 4 replicas.

Scaling Journey

0 – 1K Users

Single PostgreSQL

Start with PostgreSQL for everything. It handles JSON (JSONB), full-text search, and even pub/sub (LISTEN/NOTIFY). One database, one connection string, minimal ops. You do not need NoSQL yet.

Signal to evolve: Write latency exceeds your SLO, or schema changes block deploys for hours.

1K – 100K Users

Read Replicas + Redis

Add read replicas for read-heavy queries. Introduce Redis for session data and caching hot queries. PostgreSQL handles writes; replicas handle reads. This typically gives you 10x headroom.

Signal to evolve: Different data types have conflicting access patterns. Your product catalog queries are slowing down order processing.

100K – 10M Users

Vertical Partitioning

Move different data domains to purpose-built databases: product catalog to MongoDB, analytics events to Cassandra, search to Elasticsearch. PostgreSQL keeps transactional data. This is polyglot persistence.

Signal to evolve: A single domain (e.g., order data) exceeds the capacity of one PostgreSQL instance.

10M+ Users

NewSQL + Polyglot

Migrate transactional data to CockroachDB or TiDB for horizontal SQL scaling. Use DynamoDB for high-throughput key-value workloads. Full polyglot persistence with dedicated ops per database. Consider a data mesh architecture.

Risk: Operational complexity is the bottleneck now, not scale.

What Can Go Wrong

1. SQL: Schema Migration Nightmare at Scale

What happened: A team needed to add a NOT NULL column with a default value to a 2-billion-row MySQL table. The ALTER TABLE acquired a metadata lock, blocking all reads and writes for 4 hours. The application queued up thousands of connections and eventually crashed.

Root cause: MySQL's online DDL capabilities were insufficient for this particular ALTER operation on this table size. The operation required a full table copy internally.

Blast radius: Complete application outage for 4 hours. Background jobs accumulated a 6-hour backlog.

Prevention: Use tools like gh-ost (GitHub Online Schema Migration) or pt-online-schema-change (Percona) for large tables. They create a shadow table, copy data incrementally using triggers, then atomically swap. Test schema changes on a production-size replica first. PostgreSQL handles many ALTERs without full table locks but still requires caution with large tables.

2. NoSQL: Data Duplication Inconsistency

What happened: An e-commerce platform stored product prices denormalized in both the product catalog (MongoDB) and inside each order document. When a price change was applied, it updated the catalog but a bug caused the order-level prices to also be updated retroactively, changing historical order records and triggering incorrect refund calculations.

Root cause: Denormalization means the same data exists in multiple places. Without ACID transactions spanning collections, updates can be partially applied. The application code was the only enforcement of consistency, and it had a bug.

Prevention: Treat denormalized data as immutable snapshots. Order documents should capture the price at time of purchase and never update it. Use event sourcing or change data capture (CDC) to propagate updates in a controlled, auditable way rather than direct multi-document updates.

3. NoSQL: Query Flexibility Discovered Too Late

What happened: A startup built their entire platform on DynamoDB with a single-table design optimized for three query patterns. Six months later, the product team needed a new report: "all orders by geographic region for the last 90 days." This query was not supported by any existing partition key or GSI. Adding a new GSI on a 500M-item table took 48 hours to backfill and added $800/month to the AWS bill.

Root cause: DynamoDB (and NoSQL generally) requires you to design your data model around access patterns. Unknown future access patterns cannot be efficiently served.

Prevention: Mirror DynamoDB data to a SQL-compatible analytics store (Redshift, Athena, or even a PostgreSQL replica) using DynamoDB Streams + Lambda. Keep the operational workload on DynamoDB but run ad-hoc analytics on the SQL copy. Accept that you will not predict every future query pattern.

Interactions & Pitfalls

Database Choice + Caching

Key-value stores like Redis blur the line between "cache" and "database." When you use Redis as your primary session store, is it a cache or a database? If the answer is "database" (data loss is unacceptable), you need Redis persistence (RDB snapshots + AOF). If it is "cache" (data can be rebuilt), you can run it with no persistence for better performance. The same Redis instance should not be both — cache eviction (maxmemory-policy allkeys-lru) will delete your session data if you mix workloads.

Database Choice + Sharding

NoSQL databases like Cassandra, DynamoDB, and MongoDB have sharding (or partitioning) built in. You choose a partition key and the database handles distribution automatically. SQL databases require external sharding solutions: Vitess (used by YouTube, Slack) for MySQL, Citus (now part of Azure) for PostgreSQL. The operational difference is enormous — Cassandra just works with 100 nodes; sharding MySQL across 100 nodes requires significant engineering investment and ongoing maintenance.

Database Choice + Replication

SQL databases typically use primary-replica replication: one node accepts writes, replicas asynchronously follow. This gives strong consistency on reads from the primary but eventual consistency on replica reads. Cassandra uses leaderless replication — any node can accept writes, and consistency is tunable per query (ONE, QUORUM, ALL). This is more available but requires understanding of conflict resolution (last-write-wins by default).

Consistency Levels in NoSQL: A Cassandra Walkthrough

Cassandra lets you choose a consistency level per query, trading off between performance and correctness. To understand how this works, let us trace through a concrete example with replication factor RF=3 (every piece of data is stored on 3 nodes).

Write with QUORUM Consistency

A QUORUM requires acknowledgment from a majority of replicas: floor(RF/2) + 1 = floor(3/2) + 1 = 2. So the coordinator must get an ack from at least 2 of the 3 replicas before telling the client "write successful."

A QUORUM read contacts 2 of the 3 replicas and returns the most recent value (determined by timestamp). If the replicas disagree, the coordinator picks the value with the latest timestamp.

Why QUORUM Write + QUORUM Read Guarantees Strong Consistency

With RF=3, QUORUM = 2. A QUORUM write writes to at least 2 nodes. A QUORUM read reads from at least 2 nodes. Since 2 + 2 = 4 > 3 (the total number of replicas), there is always at least 1 node that participated in both the write and the read. That node has the latest data, so the read is guaranteed to see the most recent write. This is the mathematical foundation: W + R > RF guarantees strong consistency.

Read Repair: Fixing Stale Replicas

When a QUORUM read discovers that replicas disagree (one has an older version), Cassandra performs read repair — it sends the latest version to the stale replica in the background. Here is a scenario:

Read repair is how Cassandra "self-heals" stale replicas over time without requiring any manual intervention. Combined with anti-entropy repair (a periodic full-replica comparison using Merkle trees), Cassandra ensures that all replicas eventually converge to the same state.

Database Choice + Transactions

SQL provides native transactions: BEGIN; UPDATE ...; UPDATE ...; COMMIT;. When your data spans multiple NoSQL tables or services, you need distributed transaction patterns: Saga pattern (sequence of local transactions with compensating actions on failure) or outbox pattern (write events to a local outbox table, then publish asynchronously). These add significant application complexity. This is the hidden cost of NoSQL that teams often underestimate.

Putting a complex analytical workload on DynamoDB can cost 5x more than PostgreSQL because every "join" becomes multiple API calls.

Real-World Examples

Instagram — PostgreSQL at Scale

Instagram is one of the largest PostgreSQL deployments in the world. They chose PostgreSQL early and scaled it through aggressive use of Django ORM optimizations, pgbouncer for connection pooling, and eventually horizontal sharding using a custom sharding framework built on PostgreSQL. Their rationale: the team knew SQL, transactions were essential for social graph operations (follow/unfollow must be atomic), and PostgreSQL's reliability was worth the scaling investment. At their scale (~2 billion monthly active users), they run thousands of PostgreSQL shards.

Discord — Cassandra to ScyllaDB Migration

Discord stored billions of messages in Cassandra. As the dataset grew, they hit severe performance issues: garbage collection pauses in Cassandra's JVM caused tail latency spikes exceeding 5 seconds, compaction backlog caused read amplification, and large partitions (channels with millions of messages) created hot spots. In 2022, they migrated to ScyllaDB — a C++ rewrite of Cassandra that eliminates JVM GC pauses and uses a shard-per-core architecture. The result: p99 read latency dropped from 40-125ms to 15ms, and tail latency spikes disappeared. The data model stayed the same (ScyllaDB is Cassandra-compatible), so the migration was primarily an infrastructure swap.

Airbnb — Polyglot Persistence

Airbnb uses MySQL as their primary transactional database for bookings, payments, and user data. They augment it with: Redis for caching and rate limiting, Elasticsearch for search, Hive/Spark on S3 for analytics, and a custom service mesh that routes queries to the right data store. Their key insight: rather than migrate away from MySQL, they built specialized services around it. Each service owns its database and exposes an API. This avoids the "one database to rule them all" trap.

Netflix — Cassandra for Availability

Netflix runs one of the largest Cassandra deployments: thousands of nodes across multiple AWS regions storing trillions of rows. They chose Cassandra because availability is their top priority — users must always be able to browse and play content, even during regional failures. Cassandra's leaderless replication means any region can accept writes, and data is eventually consistent across regions. Netflix built extensive tooling around Cassandra, including Priam (for automated cluster management) and a custom data pipeline that feeds Cassandra from their microservices.

Code Example

The same data modeled in SQL (normalized) vs. document store (denormalized). Notice how the SQL approach requires a join while the document approach returns everything in one read — but updates to the author name must be applied in two places in the document model.

# === SQL Approach: Normalized (PostgreSQL) ===
import psycopg2

def get_post_with_author_sql(conn, post_id):
    """One query with a JOIN — single source of truth for author data."""
    cur = conn.cursor()
    cur.execute("""
        SELECT p.title, p.content, p.created_at,
               u.name AS author_name, u.email AS author_email
        FROM posts p
        JOIN users u ON p.author_id = u.id
        WHERE p.id = %s
    """, (post_id,))
    return cur.fetchone()

def update_author_name_sql(conn, user_id, new_name):
    """Update once — every post query reflects the change."""
    cur = conn.cursor()
    cur.execute("UPDATE users SET name = %s WHERE id = %s", (new_name, user_id))
    conn.commit()  # ACID: guaranteed to be atomic

# === NoSQL Approach: Denormalized (MongoDB-style) ===
def get_post_with_author_nosql(db, post_id):
    """Single document read — no join needed, everything is embedded."""
    return db.posts.find_one({"_id": post_id})
    # Returns: {"title": "...", "content": "...", "author": {"name": "...", "email": "..."}}

def update_author_name_nosql(db, user_id, new_name):
    """Must update in TWO places — users collection AND every embedded copy."""
    db.users.update_one({"_id": user_id}, {"$set": {"name": new_name}})
    db.posts.update_many(
        {"author._id": user_id},
        {"$set": {"author.name": new_name}}
    )  # No transaction — if the app crashes between these two, data is inconsistent

# === The Tradeoff ===
# SQL:   1 read = 1 join (slower read, consistent data)
# NoSQL: 1 read = 1 document (faster read, but updates are complex and risky)

How to Validate Your Design

Benchmark Both Approaches

Before committing to SQL or NoSQL, prototype both with realistic data and query patterns. Use pgbench for PostgreSQL benchmarking and cassandra-stress for Cassandra. For DynamoDB, use NoSQL Workbench to model access patterns before writing code. Key measurement: not just throughput, but p99 latency under concurrent load. A database that handles 50K QPS at p50=2ms but p99=500ms is not the same as one with p50=5ms and p99=10ms.

Test Schema Migration Process

For SQL: simulate an ALTER TABLE on a production-sized replica. Measure lock time. Test rolling migration tools (gh-ost, pt-online-schema-change). For NoSQL: test adding a new field to existing documents and verify your application handles both old and new document shapes gracefully. Use tools like mongomigrate or write custom migration scripts.

Test at Expected Scale

Generate realistic test data at 10x your expected year-one volume. Tools: Faker (Python) for realistic test data generation, k6 or Locust for load testing, toxiproxy for injecting network latency between your application and database. Key scenarios to test:

  • Hot key test — Send 50% of traffic to a single key. Does the database handle it or does the partition become a bottleneck?
  • Consistency test — Write a value, immediately read it from a different connection/node. Is it there? (Tests read-after-write consistency.)
  • Failover test — Kill the primary node. How long until writes resume? Is any data lost? Use Chaos Monkey or Litmus for systematic fault injection.
  • Connection storm — Open 1,000 connections simultaneously. SQL databases are particularly sensitive here — each connection consumes RAM. This is where PgBouncer or ProxySQL earns its keep.

Interview Questions

SQL vs NoSQL for a social network?

A social network has two dominant data patterns: the social graph (who follows whom) and the content feed (posts, comments, likes). The social graph is inherently relational — follow/unfollow operations need ACID guarantees (you cannot have a half-completed follow), and friend-of-friend queries are naturally expressed as joins. This argues for SQL or a graph database.

The content feed, however, is read-heavy (100:1 ratio), benefits from denormalization (embed author info in each post to avoid joins on every feed render), and grows rapidly. This fits a document store or a denormalized cache.

The production answer is polyglot persistence: PostgreSQL for the social graph and transactional operations, a denormalized feed service backed by Redis or Cassandra for timeline rendering, and Elasticsearch for search. Instagram does exactly this — PostgreSQL for the graph, Redis/Memcached for feed caching. Facebook uses TAO (a graph-aware cache over MySQL shards).

Follow-ups: How do you handle the fan-out problem when a user with 10M followers posts? (Write fan-out vs. read fan-out.) How do you keep the denormalized feed consistent with the social graph? (Eventual consistency with CDC.)

When would you use a graph database?

Use a graph database when your queries are defined by relationships rather than entities, and when the depth of traversal is variable or deep (>3 hops). Specific use cases: social networks (friend-of-friend recommendations), fraud detection (finding cycles of transactions between accounts), knowledge graphs (entity relationships in search), network topology (finding shortest paths), and access control (traversing permission hierarchies).

The key signal: if you find yourself writing recursive CTEs (Common Table Expressions — WITH RECURSIVE queries that walk a hierarchy by referencing themselves) in SQL or doing multiple round-trips to the database to "walk" relationships, a graph database will be both simpler to code and orders of magnitude faster. Neo4j's index-free adjacency means each hop is O(1) regardless of total graph size, while each SQL join scales with table size.

Do NOT use a graph database for: simple CRUD applications where relationships are shallow (1-2 levels), high-throughput write-heavy workloads (graph databases are generally optimized for reads), or when your data is naturally tabular or document-shaped. Also avoid if your team has no graph modeling experience — the learning curve is steep and a poorly modeled graph performs worse than SQL.

Follow-ups: How does a graph database handle horizontal scaling? (Sharding a graph is notoriously hard because edges cross partitions.) What is the difference between property graphs (Neo4j) and RDF triple stores?

Design the data layer for an e-commerce platform — which databases and why?

Start by identifying the data domains and their access patterns: (1) User accounts and authentication — relational, needs ACID (PostgreSQL). (2) Product catalog — heterogeneous schema (laptops vs. clothing have different attributes), read-heavy — document store (MongoDB) or PostgreSQL with JSONB. (3) Inventory and orders — transactional, needs strong consistency to prevent overselling — PostgreSQL with row-level locking. (4) Shopping cart / sessions — ephemeral, fast access — Redis. (5) Search — full-text with faceted filtering — Elasticsearch. (6) Recommendations — relationship-based — graph database or precomputed in a batch pipeline.

The key design decision is how to handle the checkout transaction: decrement inventory, create order, capture payment. This MUST be atomic. If your inventory is in PostgreSQL, this is a single transaction. If inventory is in a separate service, you need the Saga pattern with compensating transactions.

At small scale, PostgreSQL alone handles all of this. The product catalog can use JSONB columns. Add Redis for caching. Only split into separate databases when a specific domain's access pattern conflicts with others (e.g., catalog search queries are slowing down order writes).

Follow-ups: How do you handle flash sales (sudden 100x traffic spike on one product)? How do you keep the search index consistent with the product catalog? What happens when Redis goes down — does the cart disappear?

What is NewSQL and when does it make sense?

NewSQL databases provide the SQL interface and ACID transactions of traditional relational databases with the horizontal scalability and fault tolerance of distributed NoSQL systems. The three major examples are Google Spanner, CockroachDB, and TiDB.

NewSQL makes sense when: (1) you need ACID transactions at a scale that exceeds single-node SQL (e.g., >10TB of transactional data, >50K write QPS), (2) you need multi-region active-active with strong consistency, or (3) you want to avoid the operational complexity of manually sharding MySQL/PostgreSQL.

NewSQL does NOT make sense when: (1) your data fits on one machine (most applications!) — the distributed overhead adds latency, (2) you need the specific extensions of PostgreSQL or MySQL (PostGIS, full-text search, mature ORMs), or (3) your team does not have the expertise to operate a distributed database. CockroachDB is simpler than manual sharding but still more complex than a single PostgreSQL instance.

The key innovation in Spanner is TrueTime — using atomic clocks to order transactions globally without coordination. CockroachDB approximates this with hybrid logical clocks. TiDB separates compute (SQL layer) from storage (TiKV), allowing each to scale independently.

Follow-ups: How does Spanner's TrueTime work and what happens if the clock uncertainty interval is too large? What are the latency implications of distributed transactions in CockroachDB?

How did Discord migrate from Cassandra to ScyllaDB and why?

Discord stored trillions of messages in Cassandra, partitioned by channel ID and clustered by message timestamp. As data volume grew to petabytes, they hit three major problems: (1) GC pauses — Cassandra runs on the JVM, and garbage collection pauses caused read latency spikes up to 5 seconds, which was unacceptable for a real-time chat application. (2) Compaction backlog — Cassandra's LSM-tree architecture requires compaction to merge SSTables. Under heavy write load, compaction fell behind, causing read amplification (reads had to check many SSTables). (3) Hot partitions — Popular Discord channels (e.g., servers with millions of members) created partitions with millions of messages, causing uneven load distribution.

They migrated to ScyllaDB, a C++ rewrite of Cassandra. ScyllaDB eliminates GC pauses entirely (no JVM), uses a shard-per-core architecture (each CPU core manages its own data partition, avoiding cross-core coordination), and has more efficient compaction. The data model was Cassandra-compatible, so the migration was primarily an infrastructure replacement. They used dual-writes during migration: write to both Cassandra and ScyllaDB, read from ScyllaDB, and compare results.

Results: p99 read latency dropped from 40-125ms to 15ms. Tail latency spikes disappeared. Operational overhead decreased because ScyllaDB requires fewer nodes for the same throughput. The lesson: sometimes the right database type (column-family) is correct, but the implementation matters enormously.

Follow-ups: How did they handle data consistency during the dual-write migration period? What is shard-per-core architecture and why does it help?

Single-table design in DynamoDB — explain the approach.

Single-table design is a DynamoDB modeling technique where all entity types (users, orders, products) are stored in one table with carefully designed partition keys (PK) and sort keys (SK). Instead of separate tables, you use key prefixes to distinguish entity types: PK=USER#123, SK=PROFILE for user data, PK=USER#123, SK=ORDER#2024-001 for that user's order. A single query with PK=USER#123 returns both the user profile and all their orders — no join needed.

This works because DynamoDB has no JOIN operation and cross-table queries are expensive (multiple API calls). Single-table design collocates related data in the same partition, enabling single-request access patterns. Global Secondary Indexes (GSIs) provide additional access patterns: a GSI with GSI-PK=ORDER#2024-001 lets you query an order by order ID.

The tradeoff: you must know ALL your access patterns upfront. Adding a new access pattern may require a new GSI (which has cost and time implications for backfill). The data is optimized for specific queries, not for ad-hoc exploration. Teams that need analytical flexibility should mirror DynamoDB data to a SQL-compatible store using DynamoDB Streams.

Single-table design is powerful but has a steep learning curve. It inverts the normal database design process: instead of "what data do I have?" you start with "what queries do I need?" Rick Houlihan's re:Invent talks are the canonical resource.

Follow-ups: How do you handle many-to-many relationships in single-table design? (Adjacency lists.) What are the limits of DynamoDB GSIs? (20 per table, each consumes additional write capacity.) How do you migrate a single-table design when access patterns change?

Key Takeaways

  • SQL and NoSQL are not competing technologies — they are different tools for different jobs. The right question is "which access patterns does my data have?" not "which is better?"
  • SQL (relational) excels at complex queries, ACID transactions, and data integrity. Choose it when relationships matter and consistency is non-negotiable.
  • Document stores (MongoDB) excel at flexible schemas and denormalized reads. Choose them when your data varies in structure or when read performance on a single entity is critical.
  • Key-value stores (Redis, DynamoDB) excel at low-latency lookups by key. Choose them for caching, sessions, and high-throughput simple access patterns.
  • Column-family stores (Cassandra, ScyllaDB) excel at write-heavy and time-series workloads. Choose them when write throughput and availability matter more than read flexibility.
  • Graph databases (Neo4j) excel at relationship traversal. Choose them when queries are defined by relationships, not entities.
  • Most production systems use polyglot persistence — multiple database types, each handling the workload it is best at.
  • NewSQL (CockroachDB, TiDB, Spanner) offers SQL semantics with horizontal scaling. Consider it when you outgrow single-node SQL but still need ACID.
  • The lines are blurring: PostgreSQL has JSONB, MongoDB has transactions, DynamoDB has SQL-like PartiQL. Evaluate capabilities, not just categories.
  • Start with PostgreSQL. It handles an astonishing range of workloads. Split into specialized databases only when you have concrete evidence of a bottleneck.
