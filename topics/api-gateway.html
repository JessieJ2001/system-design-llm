API Gateway

The single entry point that sits between clients and your backend services, handling routing, authentication, rate limiting, and request transformation so your microservices don't have to.

30 min read

What Is It?

An API gateway is a server that acts as the single entry point for all client requests directed at your backend services. Instead of having mobile apps, web frontends, and third-party integrators each call individual microservices directly, every request flows through the gateway first. The gateway then routes each request to the appropriate backend service based on the URL path, HTTP method, headers, or other criteria.

Beyond simple routing, an API gateway handles cross-cutting concerns that would otherwise be duplicated across every service: authentication and authorization (validating JWT tokens or API keys), rate limiting (throttling abusive clients), request and response transformation (converting between protocols, reshaping payloads), response aggregation (combining data from multiple services into a single response), and protocol translation (accepting REST from clients while communicating via gRPC to backends).

Think of the API gateway as the "front door" of a microservices architecture. Just as a hotel concierge handles guest requests — checking IDs, directing people to the right room, calling different departments — the gateway handles the messy details of client-to-service communication so that individual services can focus purely on business logic. Without a gateway, each service must independently implement auth, logging, CORS headers, and rate limiting, leading to inconsistency, duplication, and a much larger attack surface.

How It Works

When a request arrives at an API gateway, it passes through a pipeline of processing stages before reaching a backend service. Here is the typical flow:

  1. Receive: The gateway accepts the incoming HTTP/HTTPS request from the client. TLS termination happens here — the gateway decrypts the request so backends don't need to manage certificates individually.
  2. Authenticate: The gateway validates the client's identity. This might involve checking a JWT token signature and expiration, validating an API key against a registry, or forwarding to an OAuth2 authorization server. Invalid credentials result in an immediate 401 Unauthorized response — the request never reaches the backend.
  3. Rate limit: The gateway checks whether this client or tenant has exceeded their allowed request quota. If the rate limit is hit, the gateway returns 429 Too Many Requests with a Retry-After header. This protects backend services from being overwhelmed.
  4. Route: Based on the URL path, HTTP method, and potentially headers (like API version), the gateway determines which backend service should handle this request. Route tables can be static (configured in YAML/JSON) or dynamic (pulled from a service registry).
  5. Transform: The gateway may modify the request before forwarding: stripping or adding headers, rewriting the URL path, converting the payload format (e.g., REST to gRPC protobuf), or injecting metadata like trace IDs and tenant context.
  6. Forward: The gateway sends the (potentially transformed) request to the selected backend service. This step involves service discovery and load balancing if multiple instances of the backend are running.
  7. Aggregate (optional): For some endpoints, the gateway fans out to multiple backend services in parallel, collects their responses, and merges them into a single response. For example, a product detail page might need data from the Product Service, Inventory Service, and Reviews Service.
  8. Respond: The gateway receives the backend response, optionally transforms it (removing internal fields, adding CORS headers, compressing), and sends the final response back to the client. The gateway also logs the request, records metrics, and reports trace spans.

Response Aggregation: Parallel vs. Sequential

When the gateway needs data from multiple backends, it calls them in parallel rather than one at a time. The total latency equals the slowest service, not the sum of all services. Here is a concrete example for a product detail page that aggregates data from 4 services:

Time (ms) Service A (Catalog) Service B (Inventory) Service C (Pricing) Service D (Reviews) Gateway Status

Parallel total latency = max(100, 80, 60, 70) = 100ms.
If the gateway had called them sequentially: 100 + 80 + 60 + 70 = 310ms — over 3x slower.

Handling Partial Failures with Fallbacks

What if Service D (Reviews) fails or times out? The gateway should not fail the entire request. Instead, it returns a partial response with fallback data for the failed service:

Time (ms) Service A (Catalog) Service B (Inventory) Service C (Pricing) Service D (Reviews) Gateway Status
0Request sentRequest sentRequest sentRequest sentAll 4 dispatched
60Processing...Processing...Response OKProcessing...1 of 4 complete
80Processing...Response OK—Processing...2 of 4 complete
100Response OK——Processing...3 of 4 complete; D still pending
200———TIMEOUT (200ms limit hit)D timed out — use fallback

The gateway returns a merged response where Services A, B, and C provide real data, and Service D is replaced with fallback content:

{
  "product": { "id": "P-123", "name": "Wireless Headphones" },  // Service A
  "inventory": { "in_stock": true, "quantity": 42 },             // Service B
  "pricing": { "price": 79.99, "currency": "USD" },              // Service C
  "reviews": { "available": false, "message": "Reviews temporarily unavailable" }  // Fallback for D
}

The client still gets a usable response (200 OK, not 500 or 504), and the product page renders with a "Reviews unavailable" placeholder instead of a blank error page. The total latency is 200ms (bounded by D's timeout), not the 100ms it would have been if D had responded. This is why per-service timeouts matter: setting D's timeout to 200ms caps the damage from a slow Reviews service.

When to Use / When NOT to Use

Use an API Gateway When

  • Microservices needing a unified entry point: You have 5+ backend services and don't want clients to know about or connect to each one individually.
  • Multiple client types (BFF pattern): Mobile apps need slim payloads, web apps need rich data, and third-party partners need a stable, versioned API. A gateway (or multiple BFF gateways) can tailor responses for each.
  • Cross-cutting concerns: Authentication, authorization, rate limiting, request logging, CORS, and compression need to be applied consistently across all services. Centralizing these in the gateway avoids duplicating them in every service.
  • Protocol translation: Your clients speak REST/HTTP but your internal services communicate via gRPC, Thrift, or WebSockets. The gateway bridges the protocol gap.
  • API versioning: You need to support multiple API versions simultaneously, routing /v1/* to legacy services and /v2/* to new ones.

Don't Use an API Gateway When

  • Monolith with a single API: If you have one backend service, a gateway adds latency and operational complexity with no benefit. A reverse proxy (Nginx) is sufficient.
  • Internal service-to-service communication: For east-west traffic between microservices, use a service mesh (Istio, Linkerd) instead. The API gateway is for north-south (client-to-service) traffic.
  • Gateway becomes a bottleneck: If your gateway's response aggregation logic is so complex that it adds significant latency, consider pushing that logic to a dedicated BFF service or using GraphQL federation instead.
  • Small team, simple architecture: A 3-person team with 2 services doesn't need the operational overhead of running and maintaining a gateway. Start with a simple reverse proxy and evolve when the pain justifies it.

Classic vs. Modern Approaches

Classic: Reverse Proxy Gateways

The first generation of API gateways were essentially beefed-up reverse proxies. Nginx with Lua scripting (OpenResty) was one of the earliest approaches — you'd write Lua modules for auth checks and rate limiting in the request pipeline. Netflix Zuul (2012) was the first purpose-built API gateway for microservices, handling routing and filter chains for Netflix's 1,000+ microservices. Kong (built on Nginx/OpenResty, open-sourced 2015) added a plugin architecture, making it easy to add auth, rate limiting, and logging without writing proxy code.

These gateways work well but have limitations: configuration is often static (requiring restarts for route changes), complex aggregation requires custom code, and they are designed primarily for HTTP REST traffic.

Modern: BFF Pattern (Backend For Frontend)

Instead of a single, monolithic gateway handling all client types, the BFF pattern deploys a separate gateway for each client type: one for mobile (optimized for low bandwidth, slim payloads), one for web (richer data), and one for third-party APIs (stable, versioned). Each BFF is owned by the frontend team that uses it, eliminating the "gateway team bottleneck" where every frontend change requires a gateway deployment. SoundCloud and Netflix popularized this pattern.

Modern: GraphQL Federation as API Layer

Apollo Federation lets you use a GraphQL gateway that composes schemas from multiple backend services into a single graph. Clients send one GraphQL query, and the gateway figures out which services to call and how to assemble the response. This eliminates the need for explicit response aggregation logic in the gateway — the schema defines the data relationships, and the federation runtime handles the rest. Expedia, Netflix, and The New York Times use this approach.

Modern: gRPC Gateway for Protocol Translation

Tools like grpc-gateway and Envoy's gRPC-JSON transcoder automatically translate RESTful HTTP/JSON requests into gRPC calls and back. You define your service in Protocol Buffers, and the gateway generates a REST API from the protobuf definitions. This lets you build all internal services with gRPC (for performance and type safety) while still exposing a REST API to external clients.

Modern: Serverless API Gateways

AWS API Gateway, Azure API Management, and Google Cloud Endpoints provide fully managed gateway-as-a-service. You pay per request, get auto-scaling for free, and configure routing via infrastructure-as-code (CloudFormation, Terraform). The tradeoff: vendor lock-in, limited customization, and higher per-request costs at scale.

Modern: Envoy-Based Gateways

Envoy Proxy (created at Lyft) has become the foundation for a new generation of gateways. Projects like Emissary-Ingress (formerly Ambassador) and Gloo Edge use Envoy as the data plane, adding Kubernetes-native configuration and advanced traffic management (canary routing, traffic shifting, fault injection). Envoy's xDS APIs enable fully dynamic configuration — route changes without restarts — which is critical at scale.

Assumptions & Model

Key Assumptions

  • The gateway can handle all traffic: The model assumes the gateway has sufficient capacity (CPU, memory, network bandwidth) to process every incoming request without becoming a bottleneck. This breaks at very high scale if the gateway fleet is undersized or if complex response aggregation consumes too much CPU.
  • The gateway is highly available: Since all traffic flows through the gateway, it is assumed to be deployed in a redundant, multi-instance configuration behind a load balancer. If the gateway is a single instance, it is a single point of failure — a crashed gateway means the entire system is down.
  • Routing rules are correct and up-to-date: The model assumes route tables accurately reflect the current set of backend services and their paths. Stale routes (pointing to decommissioned services) cause 502/503 errors. This assumption breaks during deployments if route updates are not atomic or if service discovery has lag.
  • Backend services respond within the timeout budget: The gateway assumes each backend will respond within a configured timeout (e.g., 5 seconds). If a backend hangs, the gateway must timeout and return an error — but response aggregation complicates this (one slow service can hold up the entire aggregated response).
  • Authentication state is externally verifiable: JWT validation assumes tokens are self-contained and can be verified using a public key without calling an auth server on every request. If you need to check token revocation, you need a revocation list or auth server call, adding latency.

Sizing & Tradeoffs

Gateway Capacity Planning

The core metric for sizing a gateway is requests per second (RPS) — but you must account for the fanout ratio. Each client request to the gateway may result in multiple backend calls if the gateway does response aggregation.

Fanout Formula

Backend RPS = Client RPS x Average Fanout Ratio

If you serve 10,000 client RPS and each request fans out to an average of 3 backend services, the gateway must handle 30,000 outbound backend requests per second plus the 10,000 inbound client requests — a total of 40,000 request-level operations per second through the gateway.

Latency Budget

The gateway adds latency to every request. Budget for it:

StageTypical LatencyNotes
TLS termination1-3 msOne-time per connection if keep-alive is used
Auth (JWT validation)0.1-0.5 msLocal crypto operation, no network call
Auth (OAuth introspection)5-20 msNetwork call to auth server; cache tokens
Rate limit check0.5-2 msLocal or Redis-based lookup
Routing + transform0.1-1 msCPU-bound, depends on transformation complexity
Response aggregationBounded by slowest backendParallelize fan-out; set per-service timeouts

Rule of thumb: A well-tuned gateway adds 2-5 ms of overhead for simple proxy requests. Response aggregation latency is dominated by the slowest backend call. Set individual timeouts per backend (e.g., 200ms for user service, 500ms for recommendations) rather than one global timeout.

SLO-Driven Sizing Examples

Real-world gateway sizing starts with your SLOs, not hardware specs. Work backward from the latency, availability, or throughput target to derive infrastructure requirements.

1. E-commerce API — p99 < 100ms added latency, 50K RPS

SLO: The gateway must add no more than 100ms of latency at the 99th percentile, while sustaining 50K client RPS. Product pages fan out to 4 services (catalog, inventory, pricing, reviews).

  • Latency budget breakdown: TLS (2ms) + JWT auth (0.3ms) + rate limit (0.5ms) + routing (0.2ms) = 3ms baseline. That leaves 97ms for response aggregation at p99 — set per-service timeouts at 80ms with a 15ms buffer for GC pauses and queuing.
  • Backend fanout: 50K × 4 = 200K backend RPS. Total gateway operations = 250K/sec.
  • Instance count: Each 8-core Envoy instance handles ~40K RPS under the p99 latency ceiling when running auth + rate limiting plugins. Need ceil(250K / 40K) + 2 = 9 instances (2 for N+2 redundancy).
  • Connection pool size: Each instance maintains persistent connections to 4 backends. At 50K RPS spread across 9 instances (~5.6K RPS/instance), with average backend latency of 50ms: concurrent_connections = 5600 × 0.050 / 4 backends ≈ 70 connections per backend per instance. Set pool size to 100 per backend (1.4x headroom). Fleet total: 9 × 4 × 100 = 3,600 backend connections.

2. Mobile App Backend — p95 < 200ms end-to-end, 10K RPS

SLO: End-to-end latency from mobile client to response must stay below 200ms at p95. The gateway performs JWT auth, per-user rate limiting (Redis-based), and request transformation (stripping fields for bandwidth).

  • Latency budget: Network round-trip (mobile to edge): ~60ms. That leaves 140ms for gateway + backend. Backend services average 80ms at p95. Gateway budget: 140 - 80 = 60ms.
  • Gateway overhead breakdown: TLS (2ms) + JWT (0.3ms) + Redis rate-limit (3ms p95, cross-AZ) + request transform (1ms) + response transform (1.5ms) = ~8ms. Well within the 60ms budget, leaving 52ms of safety margin for tail latency.
  • Resource allocation: At 10K RPS with lightweight operations, 3 gateway instances on 4-core / 8GB machines provide ample capacity. Redis rate-limit cluster: 2 replicas with read-from-replica to keep p95 under 3ms.
  • Key insight: The bottleneck is not gateway compute but the Redis rate-limit lookup. Moving to a local sliding-window counter (synced asynchronously) drops gateway overhead to ~4ms, doubling the safety margin.

3. Multi-tenant SaaS — 99.99% availability, 100K RPS

SLO: 99.99% availability = maximum 52.6 minutes of downtime per year. The gateway must survive any single failure (instance, AZ, or deployment) without breaching the SLO.

  • HA topology: Deploy across 3 availability zones. Each AZ runs an independent gateway fleet sized to handle 50% of total traffic (so any single AZ loss is survivable by the remaining two).
  • Per-AZ sizing: 50% of 100K = 50K RPS capacity per AZ. With 8-core instances at ~30K RPS each (headroom for failover spike): ceil(50K / 30K) = 2 instances per AZ. Fleet total: 6 instances across 3 AZs.
  • Deployment strategy: Rolling deployment, one AZ at a time, with 5-minute bake period between AZs. Each deployment takes ~20 minutes total. At 99.99%, you have budget for ~26 deployments per year if each causes 2 minutes of degraded capacity — so use canary deployments to minimize blast radius.
  • Health checking: L4 load balancer with 3-second health check intervals, 2 consecutive failures to mark unhealthy. Recovery detection within 6 seconds keeps failover time well under the 52.6-minute annual budget.

4. Real-time Gaming API — p99 < 20ms gateway overhead, 500K concurrent connections

SLO: Gateway overhead must stay under 20ms at p99 for latency-sensitive game state updates, while maintaining 500K persistent WebSocket connections.

  • Connection sizing: Each gateway instance can hold ~50K concurrent WebSocket connections (limited by file descriptors and memory: ~30KB per connection = 1.5GB memory for 50K connections). Need 500K / 50K = 10 instances minimum, deploy 13 instances for N+3 redundancy.
  • Latency budget: With 20ms total budget, eliminate all network calls from the hot path. Use local JWT validation (0.2ms), local rate limiting with in-memory token buckets (0.1ms), and direct L4 routing (0.1ms). Total overhead: ~0.5ms, leaving 19.5ms of headroom.
  • CPU sizing: WebSocket frame parsing + routing at 500K connections with an average of 20 messages/sec per connection = 10M messages/sec. Each 8-core instance handles ~800K messages/sec. Need ceil(10M / 800K) = 13 instances — aligns with connection-based sizing.
  • Key constraint: Connection draining during deployments. With 50K connections per instance, graceful drain at 1,000 reconnections/sec takes 50 seconds. Build this into the deployment window and ensure clients implement exponential backoff reconnection.

SLO-Driven Capacity Planning

Traditional capacity planning asks "how many requests can this hardware handle?" SLO-driven planning inverts the question: "what infrastructure do I need to meet my latency / availability / throughput commitments?"

The SLO Sizing Process

  1. Define the SLO: e.g., p99 latency < 100ms, 99.99% availability, or 50K sustained RPS.
  2. Decompose the latency budget: Subtract network time, backend time, and client processing from the end-to-end target. What remains is the gateway budget.
  3. Profile the gateway overhead: Benchmark your specific plugin chain (auth + rate limit + transform) to measure per-request CPU and memory cost.
  4. Model the queuing inflection: Use the M/M/c queuing model or load-test data to find the RPS at which latency exceeds your budget. This is your per-instance capacity ceiling — typically 60-70% of max throughput.
  5. Add redundancy for the availability SLO: If your SLO is 99.99%, you must survive any single failure. Deploy N+2 instances across multiple AZs.
SLO TypeGateway Sizing DriverKey Formula
Latency (p99 < X ms) Keep utilization below queuing inflection point instances = totalRPS / (capacity_at_p99_target) + redundancy
Availability (99.9%+) Survive N failures without breaching error budget min_AZs = 3; instances_per_AZ = ceil(peakRPS / (AZs - 1) / per_instance_cap)
Throughput (sustained RPS) Horizontal scaling with connection pool limits instances = ceil(totalOps / per_instance_RPS); pool = (RPS/instances) × avg_latency
Concurrent connections Memory and file descriptor limits per instance instances = ceil(connections / max_conn_per_instance) + N_redundancy

Critical practice: Load-test your gateway with the exact plugin chain you will run in production. A bare proxy benchmarks 5-10x higher than one running auth + rate limiting + request transformation + logging. Size against the realistic benchmark, not the marketing number.

Scaling Journey

0 – 1K users

Direct Client-to-Service

With a monolith or 1-2 services, clients call backends directly through a simple reverse proxy (Nginx). Auth is handled in each service. No dedicated gateway needed — the complexity isn't justified.

Trigger to evolve: You split the monolith into 3+ microservices and find yourself duplicating auth middleware and CORS headers in every service.

1K – 100K users

Single API Gateway

Deploy a single API gateway (Kong, Express Gateway, or a custom Node.js gateway). Centralize auth, rate limiting, and logging. All clients hit one gateway endpoint. Route configuration is static, loaded from a config file.

Trigger to evolve: Mobile and web teams fight over gateway configuration. Mobile needs slim payloads; web needs different aggregation. The gateway becomes a deployment bottleneck.

100K – 10M users

BFF Gateways per Client

Deploy separate BFF gateways: one for mobile, one for web, one for partner APIs. Each is owned by the team that uses it. Route configuration moves to a service registry (Consul, etcd). Gateway fleet auto-scales behind a load balancer.

Trigger to evolve: You need a unified data graph across dozens of services, and manually writing aggregation logic in each BFF is unsustainable.

10M+ users

Federated + Edge Gateways

Deploy GraphQL federation or an Envoy-based mesh gateway. Add edge gateways in multiple regions for latency-sensitive traffic. Edge gateways handle TLS, auth, and rate limiting; regional gateways handle routing and aggregation. Configuration is fully dynamic via xDS or a control plane.

What Can Go Wrong

Gateway as a Single Point of Failure

What happened: A company deployed a single gateway instance without redundancy. During a routine deployment, the gateway process crashed during config reload, taking down all API traffic for 12 minutes.

Root cause: The gateway was not deployed behind a load balancer, and the deployment strategy did not use rolling updates. A syntax error in a new route config caused the gateway process to abort on startup.

Blast radius: 100% of client traffic was affected — no requests reached any backend service.

Mitigation: Deploy at least 2 gateway instances behind an L4 load balancer. Use config validation (dry-run mode) before applying changes. Implement rolling deployments with health checks — the new instance must pass health checks before the old one is drained.

Configuration Drift & Routing Errors

What happened: After decommissioning an old user service, the team forgot to update the gateway route table. Requests to /api/users/profile were still routed to the decommissioned service's IP, resulting in connection timeouts and 504 errors for 45 minutes before alerting caught it.

Root cause: Route configuration was managed manually in a YAML file. There was no automated sync between the service registry and gateway routes. The decommissioning runbook did not include a gateway route update step.

Prevention: Use dynamic service discovery (Consul, Kubernetes service endpoints) so the gateway automatically discovers and routes to healthy service instances. Add automated tests that verify all routes resolve to reachable backends.

Response Aggregation Timeout Cascade

What happened: A product detail page aggregated data from 4 services. The Recommendations Service experienced a latency spike (from 50ms to 8 seconds) due to a database issue. Because the gateway waited for all 4 responses before replying, every product detail request timed out, even though the other 3 services responded in under 100ms.

Root cause: The gateway used a single global timeout (10s) for aggregated requests rather than per-service timeouts with partial-response fallbacks.

Prevention: Set per-service timeouts for aggregated responses (e.g., 500ms for recommendations). If a non-critical service times out, return a partial response with a fallback (e.g., empty recommendations list). Integrate circuit breakers so that a failing service is quickly short-circuited rather than consuming gateway threads waiting for timeouts.

Gateway Failure Cascade: A Timeline

One of the most dangerous failure modes is when a backend slows down instead of failing outright. A slow backend ties up gateway resources (threads, connections) waiting for responses. Here is a step-by-step timeline of how a backend latency spike cascades into a full gateway outage:

Time Event Backend Latency Gateway Connection Pool Client Experience
T=0s Normal operation 50ms (healthy) 20/100 connections in use (20%) Responses in ~55ms (50ms backend + 5ms gateway overhead)
T=0s Backend database hits a lock contention issue 50ms → 500ms (10x spike) 20/100 in use No visible change yet (in-flight requests still using old connections)
T=1s New requests pile up; each holds a connection for 500ms instead of 50ms 500ms 60/100 in use (filling fast) Latency rises to ~505ms. Users notice slowness.
T=3s Connection pool nearly exhausted 500ms 95/100 in use New requests start queuing, waiting for a free connection. Latency: 500ms + queue wait.

The cascade: One slow backend consumed all gateway connections, which blocked requests to every backend. A failure in the Reviews Service caused the Users Service, Products Service, and Orders Service to all become unreachable through the gateway.

Circuit Breaker Intervention

A circuit breaker stops this cascade by detecting the failure early and failing fast instead of waiting for slow responses:

Time Event Circuit Breaker State Gateway Connection Pool Client Experience
T=0s Backend latency spikes to 500ms CLOSED (normal, passing requests through) 20/100 Latency rises to 505ms
T=1s 5 requests to this backend exceed the 200ms timeout CLOSED → failure count: 5 40/100 Some 504 Gateway Timeout responses
T=2s Failure threshold reached (5 failures in 10s window) OPEN (all requests to this backend rejected immediately) 40/100 → dropping back to 20/100 Requests to the slow backend fail fast in ~2ms (not 500ms). Other backends unaffected.
T=12s Half-open timeout expires (10s after opening) HALF-OPEN (allow 1 test request through) 20/100 One request tests the backend
T=12s Test request still slow (500ms) HALF-OPEN → OPEN (back to rejecting) 20/100 Continue failing fast. Retry in another 10s.
T=30s Backend recovers (DBA fixes the lock contention) HALF-OPEN → test request succeeds in 50ms → CLOSED 20/100 Traffic to backend resumes normally

The difference: Without the circuit breaker, the gateway returned 503 to all clients after 10 seconds (complete outage). With the circuit breaker, only requests to the one failing backend were affected, and they failed in ~2ms instead of waiting 500ms. The gateway's connection pool stayed healthy, and all other backends continued serving normally. The circuit breaker turned a cascading total outage into an isolated partial degradation.

Overly Complex Transformation Logic

What happened: A team implemented complex business logic in the gateway — payload validation, data enrichment from a database, and conditional routing based on user segments. The gateway's CPU usage spiked to 95%, and p99 latency jumped from 5ms to 300ms during peak traffic.

Root cause: The gateway was being used as an application server rather than a thin routing and auth layer. Complex transformations turned the gateway into a CPU bottleneck.

Prevention: Keep gateway logic thin: auth, rate limiting, routing, simple header transforms. Move complex business logic (validation, enrichment, conditional workflows) into dedicated backend services or BFF services. If transformation logic exceeds ~10 lines, it probably belongs in a service, not the gateway.

Debug Playbook

Symptom: High latency on a specific route

1. Check gateway_latency_p99 for the route — is the latency in the gateway or the backend?

2. If gateway latency is normal but total latency is high → the backend is slow. Check backend metrics.

3. If gateway latency itself is high → check gateway CPU utilization. If CPU > 80%, the gateway is overloaded (scale out) or running heavy transformation logic (move logic to backend).

4. Check active_connections — connection pool exhaustion causes requests to queue waiting for a connection.

5. Check if the route has response aggregation — one slow sub-request drags up the total. Look at per-service latency within the aggregation.

Symptom: Spike in 502/503 errors

1. Check error_rate per backend — which backend is returning errors?

2. If one backend has high errors → check if that backend is healthy (health check status, recent deployments, pod restarts).

3. If all backends show errors → check gateway→backend network connectivity. DNS resolution issue? Network partition?

4. Check if circuit breaker has tripped — if yes, the gateway is correctly rejecting traffic to a failed backend. Focus on fixing the backend.

5. Check for configuration drift — was a route recently changed? Did a service IP change without route table update?

Interactions & Pitfalls

Gateway + Rate Limiting

The API gateway is the ideal centralized enforcement point for rate limiting because all traffic flows through it. However, in a multi-instance gateway fleet, rate limit state must be shared (typically via Redis). If each gateway instance maintains local counters, a client can exceed the limit by distributing requests across gateway instances. Pitfall: Redis-based rate limiting adds 1-2ms per request. If Redis becomes unavailable, you must decide: fail-open (allow all traffic, risking overload) or fail-closed (reject all traffic, causing an outage).

Token Bucket at the Gateway: Concrete Walkthrough

Imagine a rate limit rule for a specific API key: bucket capacity = 10 tokens, refill rate = 5 tokens/second. The bucket starts full (10 tokens). Each request consumes 1 token. If the bucket is empty, the request is rejected with 429 Too Many Requests.

Key insight: The bucket allows a burst of up to 10 requests instantly (the bucket capacity), but the sustained rate is limited to 5 requests/second (the refill rate). Requests 12 and 13 are rejected because the bucket emptied and not enough time passed for tokens to refill.

Why Distributed Rate Limiting Needs Redis

If you run 4 gateway instances, each with a local token bucket (capacity=10, rate=5/s), a client can send 10 requests to each instance for a total of 40 requests before any are rejected — 4x the intended limit. The solution: store the token count in Redis, where all gateway instances read and write the same counter.

The INCR + EXPIRE Race Condition

A naive Redis implementation uses two commands per request:

# BROKEN: Race condition between INCR and EXPIRE
count = redis.incr(f"ratelimit:{client_id}")   # Step 1: increment counter
if count == 1:
    redis.expire(f"ratelimit:{client_id}", 1)  # Step 2: set 1-second TTL
if count > 10:
    return "429 Too Many Requests"

The problem: If the gateway crashes between Step 1 and Step 2, the key is incremented but never gets a TTL. It persists forever, and eventually every request is rejected because the counter never resets. Even without crashes, a network hiccup can cause the same issue.

The Lua Script Fix

Redis Lua scripts execute atomically — the entire script runs as a single operation with no other commands interleaved. This eliminates the race condition:

# CORRECT: Atomic Lua script for rate limiting
LUA_SCRIPT = """
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2])

local current = redis.call('INCR', key)
if current == 1 then
    redis.call('EXPIRE', key, window)
end
if current > limit then
    return 0  -- rejected
end
return 1      -- allowed
"""

# Usage: check rate limit atomically
allowed = redis.eval(LUA_SCRIPT, 1, f"ratelimit:{client_id}", 10, 1)
if not allowed:
    return "429 Too Many Requests"

Because INCR and EXPIRE run inside the same Lua script, they are guaranteed to both execute or neither execute. No race condition is possible.

Gateway + Authentication

Centralizing auth at the gateway means backend services receive pre-authenticated requests with identity context (usually in headers like X-User-Id, X-Tenant-Id). Pitfall: Backend services must trust that these headers were set by the gateway, not by a malicious client. If a service is accidentally exposed without going through the gateway (e.g., via a misconfigured load balancer), the headers can be spoofed. Always strip incoming identity headers at the gateway before setting them from the validated token.

JWT Validation: Step-by-Step Trace

When a request arrives with Authorization: Bearer eyJhbGci...payload...signature, the gateway performs these steps in order. If any step fails, the gateway immediately returns 401 Unauthorized and the request never reaches a backend service.

Worked Example: Validating a JWT

Suppose the client sends this token (dots separate the three parts):

// The raw JWT (three base64-encoded parts separated by dots):
// HEADER.PAYLOAD.SIGNATURE

// Encoded token:
"eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJ1c2VyXzQyIiwiZXhwIjoxNzQxNDgyMDAwLCJhdWQiOiJhcGkubXlhcHAuY29tIiwic2NvcGUiOiJyZWFkIHdyaXRlIn0.<signature_bytes>"

Step 1: Extract the token from the Authorization header.
The gateway reads the Authorization header, strips the Bearer prefix, and splits the remaining string on the . character into three parts: header, payload, and signature.

Step 2: Base64-decode the header to get the algorithm.
Decoding the first part (eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9) yields:

{ "alg": "HS256", "typ": "JWT" }

This tells the gateway: "This token was signed with HMAC-SHA256." The gateway checks that alg is in its allowed list (e.g., ["HS256", "RS256"]). If the algorithm is "none" or an unexpected value, reject immediately — this prevents the "algorithm confusion" attack where an attacker crafts a token with "alg": "none" to bypass signature verification.

Step 3: Base64-decode the payload to get the claims.
Decoding the second part yields:

{
  "sub": "user_42",       // Subject: who this token identifies
  "exp": 1741482000,      // Expiration: Unix timestamp (2025-03-09 02:00 UTC)
  "aud": "api.myapp.com", // Audience: which service this token is for
  "scope": "read write"   // Scopes: what this token is authorized to do
}

Step 4: Verify the signature with HMAC-SHA256.
The gateway computes: HMAC-SHA256(secret_key, base64(header) + "." + base64(payload)) and compares the result to the signature part of the token. This uses a constant-time comparison (not ==) to prevent timing attacks. If the computed signature does not match, the token has been tampered with — reject with 401.

Note: With RS256 (asymmetric), the gateway uses the identity provider's public key instead of a shared secret. The public key is fetched once from the provider's JWKS endpoint (e.g., https://auth.myapp.com/.well-known/jwks.json) and cached.

Step 5: Check exp > current_time.
The gateway reads the exp claim (1741482000) and compares it to the current Unix timestamp. If exp is in the past, the token has expired — reject with 401. Most gateways allow a small clock skew tolerance (e.g., 30 seconds) to account for clock differences between the issuing server and the gateway.

Step 6: Check aud matches this service.
The gateway verifies that the aud claim ("api.myapp.com") matches its own configured audience identifier. This prevents a token issued for a different service (e.g., "admin.myapp.com") from being used here. Without this check, a token for your admin API could be used to access the public API, or vice versa.

If all six steps pass, the gateway extracts identity data (sub, scope) from the payload and passes them downstream as trusted headers: X-User-Id: user_42, X-Scopes: read write.

Gateway + Load Balancing

The gateway handles routing (which service?), while the load balancer handles distribution (which instance of that service?). These are complementary but can conflict. Pitfall: If the gateway does client-side load balancing to backend instances but doesn't integrate with the health check system, it may route traffic to unhealthy instances. Use the same service discovery source for both gateway routing and load balancing.

Gateway + Caching

The gateway can cache responses for GET requests, reducing backend load. This is especially effective for responses that are identical across users (public catalog data, feature flags). Pitfall: Caching authenticated responses is dangerous — a cached response for User A might be served to User B. Only cache responses where the Vary header is correctly set, and never cache responses with Set-Cookie or user-specific data unless the cache key includes the user identity.

Cost Considerations

Cost Analysis

AWS API Gateway (managed): $3.50 per million requests + $0.09/GB data transfer. At 100M requests/month, that is ~$350/month just for request processing — plus data transfer costs. At 1B requests/month, it becomes $3,500/month, at which point self-hosted solutions become significantly cheaper.

Self-hosted Kong/Envoy: Run on 2-4 EC2 instances (c5.xlarge, ~$120/month each). Total: $240-$480/month for a fleet that can handle 50K+ RPS — far cheaper than managed at high scale. The cost driver is CPU (TLS termination, auth validation) and network (data transfer through the gateway).

Dominant cost driver: At low scale, managed services win because you pay zero operational cost. At high scale (>10M requests/day), self-hosted wins because the per-request cost of managed services exceeds the infrastructure + ops cost of running your own fleet.

Cost optimization tips:

  • Use connection keep-alive aggressively — TLS handshakes are expensive.
  • Cache auth token validation results (e.g., cache validated JWT for 60s) to reduce crypto operations.
  • Compress responses at the gateway to reduce data transfer costs.
  • Use HTTP/2 multiplexing between gateway and backends to reduce connection overhead.

Cost of NOT having a gateway: Without a gateway, every service independently implements auth, rate limiting, logging, and CORS. Across 10 services, that's 10x the development effort for cross-cutting concerns, plus inconsistent security enforcement (missed auth checks = data breaches), and no centralized rate limiting (one abusive client can overwhelm any service directly).

Real-World Examples

Netflix Zuul

Netflix built Zuul as their edge service gateway, handling all traffic entering the Netflix streaming platform. Zuul processes billions of requests per day, performing authentication, routing, canary testing, and dynamic traffic management. Zuul's filter architecture allows Netflix engineers to deploy custom routing rules (e.g., routing 5% of traffic to a canary build) without redeploying the gateway. Netflix open-sourced Zuul 2, which uses Netty for non-blocking I/O, enabling significantly higher throughput than the original Zuul 1 (built on Tomcat with blocking I/O).

Kong at Scale

Kong is one of the most widely adopted open-source API gateways, used by companies including Nasdaq, Cisco, and the US Department of Defense. Built on Nginx and OpenResty (Lua scripting), Kong handles the gateway's core functions (auth, rate limiting, logging) via plugins. Kong's declarative configuration (via YAML or a database) and its admin API make it popular for teams that want a production-ready gateway without building one from scratch. Kong Gateway handles over 300 billion API calls per month across its user base.

AWS API Gateway

AWS API Gateway is a fully managed service that handles REST, HTTP, and WebSocket APIs. It integrates natively with AWS Lambda (for serverless backends), IAM (for auth), and CloudWatch (for metrics). Many companies use it as their initial gateway because the operational burden is zero — no servers to manage, automatic scaling, and pay-per-use pricing. The tradeoff is cost at scale and limited customization compared to self-hosted solutions.

Envoy at Lyft

Lyft created Envoy Proxy as their edge and service mesh proxy. As an edge gateway, Envoy handles TLS termination, HTTP/2 support, rate limiting, and advanced load balancing (including zone-aware routing). Envoy's xDS APIs allow fully dynamic configuration — Lyft's control plane can update routing rules across hundreds of Envoy instances without any restarts. Envoy has since been adopted by Google, Apple, Pinterest, and many others, and it forms the data plane for Istio service mesh.

Code Example

A minimal API gateway router in Python with auth middleware. This demonstrates the core idea: intercept requests, validate credentials, route to the correct backend.

from http.server import HTTPServer, BaseHTTPRequestHandler
import json, urllib.request, hmac, hashlib, time

# Route table: path prefix -> backend URL
ROUTES = {
    "/users":    "http://user-service:8001",
    "/products": "http://product-service:8002",
    "/orders":   "http://order-service:8003",
}

SECRET = b"my-jwt-secret"  # In production, use RS256 with public keys

def validate_token(token):
    """Minimal JWT-like check: verify HMAC signature and expiry."""
    try:
        payload_b64, sig = token.rsplit(".", 1)
        expected = hmac.new(SECRET, payload_b64.encode(), hashlib.sha256).hexdigest()[:16]
        if not hmac.compare_digest(sig, expected):
            return None
        payload = json.loads(payload_b64)
        if payload.get("exp", 0) < time.time():
            return None
        return payload  # {"sub": "user123", "exp": 17...}
    except Exception:
        return None

class GatewayHandler(BaseHTTPRequestHandler):
    def do_GET(self):
        # 1. Authenticate
        token = self.headers.get("Authorization", "").removeprefix("Bearer ")
        user = validate_token(token)
        if not user:
            self.send_error(401, "Unauthorized")
            return

        # 2. Route to backend
        backend = next((url for prefix, url in ROUTES.items()
                        if self.path.startswith(prefix)), None)
        if not backend:
            self.send_error(404, "No route matched")
            return

        # 3. Forward request with user context
        dest = backend + self.path
        req = urllib.request.Request(dest, headers={
            "X-User-Id": user["sub"],  # Pass identity downstream
            "X-Request-Id": str(time.time_ns()),
        })
        try:
            with urllib.request.urlopen(req, timeout=5) as resp:
                body = resp.read()
                self.send_response(resp.status)
                self.send_header("Content-Type", "application/json")
                self.end_headers()
                self.wfile.write(body)
        except Exception as e:
            self.send_error(502, f"Backend error: {e}")

HTTPServer(("0.0.0.0", 8000), GatewayHandler).serve_forever()

How to Validate Your Design

Load Testing the Gateway

The gateway itself must be load tested independently from backend services. Use k6 or wrk to send traffic directly at the gateway, with backends replaced by mock services (that return fixed responses in 1ms) so you isolate gateway overhead.

  • Measure gateway-only latency (p50, p95, p99) at increasing RPS until the gateway saturates.
  • Test with auth enabled — JWT validation adds CPU load. Measure the RPS drop compared to no-auth.
  • Test with response aggregation — fan-out to multiple backends and measure how aggregation logic scales.
  • Test connection limits — open thousands of concurrent connections to find the breaking point.

Routing Rule Testing

Treat route configuration like code — version it in Git and write automated tests. For each route, send a request and verify it reaches the expected backend. Test edge cases:

  • Overlapping routes: Does /api/users/admin match the /api/users route or a more specific /api/users/admin route?
  • Missing routes: Requests to undefined paths should return 404, not be proxied to a random backend.
  • Route changes during deployment: Verify that in-flight requests are not dropped when routes are updated.

Chaos Testing

Use toxiproxy or Chaos Monkey to simulate failures and verify the gateway degrades gracefully:

  • Kill one backend: Verify the gateway returns appropriate errors for routes to that backend without affecting other routes. Verify the circuit breaker trips.
  • Add latency to one backend: Verify the per-service timeout fires and the gateway returns a partial response (for aggregation endpoints) rather than timing out entirely.
  • Kill a gateway instance: With multiple instances behind a load balancer, verify that traffic fails over to healthy instances without client-visible errors.
  • Simulate Redis failure: If rate limiting uses Redis, verify the gateway's fail-open/fail-closed behavior matches your policy.

Design an API gateway for an e-commerce platform

Start by identifying the client types: web frontend, mobile apps, and partner APIs. Deploy a BFF pattern with separate gateways for each, owned by the respective frontend teams. The mobile BFF returns slim payloads (product name, price, thumbnail URL), while the web BFF returns richer data (full descriptions, related products, reviews).

Integrate a circuit breaker on each backend — if Inventory Service fails, show "Check availability" instead of stock counts.

API gateway vs. service mesh — when to use which?

The API gateway handles north-south traffic (client-to-service): external clients entering your system. It manages concerns that matter at the edge — client authentication, rate limiting, API versioning, response aggregation, and protocol translation (REST to gRPC).

A service mesh handles east-west traffic (service-to-service): internal communication between microservices. It manages concerns that matter internally — mutual TLS, retries, circuit breaking, traffic shifting (canary deployments), and distributed tracing between services.

They are complementary, not competing. In a mature architecture, you use both: the API gateway sits at the edge (entry point for all external traffic), and the service mesh handles everything behind the gateway. Envoy blurs this line because it can serve as both — as an edge proxy (API gateway) and as a sidecar proxy (service mesh data plane, as used in Istio).

Use only a gateway when you have a moderate number of services (5-20) and east-west traffic is simple. Add a service mesh when you have 50+ services, need mutual TLS everywhere, need fine-grained traffic control for canary deployments, or need consistent observability across all service-to-service calls.

How to handle API versioning at the gateway?

There are three common strategies, each with tradeoffs:

1. URL path versioning (/v1/users, /v2/users): The gateway routes /v1/* to the legacy service and /v2/* to the new service. This is the simplest approach and the most commonly used in practice. Downside: URL pollution and clients must update their base URL to adopt a new version.

2. Header-based versioning (Accept: application/vnd.api+json;version=2): The gateway inspects a custom header or the Accept header to determine which backend to route to. Cleaner URLs, but harder to test (can't just change the URL in a browser) and easy for clients to forget the header.

3. Query parameter versioning (/users?version=2): Similar to URL path versioning but with a query parameter. Rarely used in practice because it interacts poorly with caching (different query parameters = different cache keys).

At the gateway level, the implementation is the same: a routing rule that matches on path prefix, header value, or query parameter and forwards to the appropriate backend. The gateway can also perform version transformation — accepting v2 requests and translating them to v1 format if the backend hasn't been updated yet.

Concrete Routing Example: URL Path Versioning

Consider two API versions running simultaneously. The gateway maintains this routing table:

Incoming Request Route Match Backend Service Forwarded URL
GET /api/v1/users /api/v1/* user-service-legacy (v1.8.3) http://user-svc-v1:8001/users
GET /api/v1/users/42 /api/v1/* user-service-legacy (v1.8.3) http://user-svc-v1:8001/users/42
GET /api/v2/users /api/v2/* user-service-new (v2.1.0) http://user-svc-v2:8002/users
GET /api/v2/users/42 /api/v2/* user-service-new (v2.1.0) http://user-svc-v2:8002/users/42
GET /api/v2/users/42/preferences /api/v2/* user-service-new (v2.1.0) http://user-svc-v2:8002/users/42/preferences

The gateway strips the version prefix (/api/v1 or /api/v2) and forwards the remaining path to the appropriate backend. The v1 and v2 services can run completely different code, different database schemas, even different languages. The client only changes its base URL to adopt the new version. The /api/v2/users/42/preferences endpoint is a new feature in v2 that does not exist in v1 — the gateway simply routes it to the v2 backend, which knows how to handle it.

BFF pattern — design gateways for mobile vs. web

The Backend For Frontend (BFF) pattern recognizes that different client types have fundamentally different needs. A mobile app on a slow 3G connection needs minimal data and as few round trips as possible. A web app on a fast connection can afford richer data and parallel requests.

Mobile BFF: Aggressively aggregates data server-side so the mobile app makes one request instead of five. Returns only the fields the mobile app needs (e.g., product name, price, and a thumbnail URL — not full HTML descriptions or related product arrays). Compresses responses aggressively (gzip/brotli). Implements request batching so the app can send multiple logical requests in one HTTP call.

Web BFF: Returns richer data structures because the web app can render them. May support GraphQL to let the frontend specify exactly which fields it needs. Can afford to return paginated lists (the web app handles pagination UI). May include server-side rendering support.

Team ownership: The critical insight is that each BFF is owned by the team that consumes it. The mobile team deploys the mobile BFF; the web team deploys the web BFF. This eliminates the "gateway team bottleneck" where every frontend change requires coordinating with a central gateway team. Each BFF evolves independently at the speed of its consuming team.

How to prevent the API gateway from becoming a bottleneck?

The API gateway is on the critical path of every request, so it must be kept thin and fast. Here are the key strategies:

1. Keep gateway logic minimal: The gateway should handle only cross-cutting concerns (auth, rate limiting, routing, basic header transforms). No business logic, no database queries, no complex payload transformations. If you find yourself writing more than 10 lines of custom logic in the gateway, it belongs in a backend service.

2. Horizontal scaling: Run multiple stateless gateway instances behind an L4 load balancer (NLB). Stateless means no in-memory sessions — all state (rate limit counters, auth caches) lives in external stores (Redis). Add instances as traffic grows.

3. Connection pooling: Maintain persistent HTTP/2 connections between the gateway and each backend service. Creating a new TCP connection + TLS handshake for each request adds 10-50ms of latency. Connection pools eliminate this overhead for all but the first request.

4. Async, non-blocking I/O: Use Envoy, Nginx, or Netty-based gateways that use event-driven architectures. Thread-per-request gateways (like Zuul 1 on Tomcat) hit thread pool limits under high concurrency. Envoy handles thousands of concurrent connections on a single thread.

5. Offload to edge: Move TLS termination, DDoS protection, and static content to a CDN or edge proxy (Cloudflare, AWS CloudFront) so the gateway only handles dynamic API traffic. This can reduce gateway traffic by 40-60% for apps that serve static assets.

How would you implement request authentication at the gateway?

The most common pattern is JWT (JSON Web Token) validation at the gateway. The client authenticates with an identity provider (Auth0, Okta, Keycloak) and receives a signed JWT. On each request, the client sends the JWT in the Authorization: Bearer <token> header.

The gateway validates the JWT by: (1) checking the signature using the identity provider's public key (fetched once and cached), (2) verifying the token hasn't expired (exp claim), (3) checking the audience (aud) matches this API, and (4) checking any required scopes/roles for the requested route. This is a local operation — no network call to the auth server, so it adds only 0.1-0.5ms of latency.

The gateway then extracts identity information from the JWT (user ID, tenant ID, roles) and passes them to backends via headers (X-User-Id, X-Tenant-Id, X-Roles). Backend services trust these headers because they only accept traffic from the gateway (enforced via network policies or mutual TLS).

Token revocation is the hard part: JWTs are stateless — once issued, they're valid until they expire. If a user's account is compromised, you can't revoke a JWT without either (a) maintaining a server-side revocation list that the gateway checks on each request (adding latency and state), or (b) using short-lived tokens (5-15 min expiry) with refresh tokens. Most systems use approach (b): short-lived access tokens + longer-lived refresh tokens.

Key Takeaways

  • An API gateway is the single entry point for all client-to-service traffic, centralizing routing, authentication, rate limiting, and request transformation.
  • Keep the gateway thin: auth, routing, rate limiting, simple transforms. Move business logic to backend services. A fat gateway becomes a bottleneck and a single point of failure.
  • The BFF pattern (one gateway per client type) prevents the gateway team from becoming a deployment bottleneck and lets each frontend team optimize its own API experience.
  • For response aggregation, set per-service timeouts with fallbacks. Never let one slow backend hold up the entire aggregated response.
  • The gateway is on the critical path — deploy it as a stateless, horizontally scaled fleet behind a load balancer. A single gateway instance is a single point of failure.
  • Use dynamic service discovery (not static config files) for route tables. Static routes drift, causing 502 errors when services are redeployed.
  • The API gateway handles north-south traffic (client-to-service). For east-west traffic (service-to-service), use a service mesh instead.
  • At low traffic (<10M requests/month), managed gateways (AWS API Gateway) are cheaper due to zero ops cost. At high traffic, self-hosted (Envoy, Kong) saves money.
  • Monitor latency per route, error rate per backend, and auth failure rate. These three metrics catch 90% of gateway problems.
