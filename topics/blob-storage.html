Blob Storage

Store and retrieve massive amounts of unstructured data — images, videos, backups, and logs — using a flat key-value namespace with an S3-like API that scales to exabytes.

28 min read

What Is It?

Blob storage (also called object storage) is a data storage architecture designed for unstructured data: images, video files, application backups, log archives, machine learning datasets, and anything else that doesn't fit neatly into rows and columns. Unlike relational databases that organize data into tables with schemas, or file systems that use hierarchical directory trees, blob storage uses a flat namespace where every object is identified by a unique key (often a string that looks like a path, e.g., images/2024/photo-001.jpg, but there are no real directories).

The core abstraction is simple: each object consists of the blob (the raw bytes — could be 1 KB or 5 TB), metadata (key-value pairs like content type, creation date, custom tags), and a unique key used for retrieval. The API surface is deliberately minimal: PUT to upload, GET to download, DELETE to remove, and LIST to enumerate objects by prefix. This simplicity is what lets object stores scale to exabytes across thousands of nodes.

How does this differ from other storage paradigms? A file system (ext4, NTFS, NFS) provides POSIX semantics — open, seek, read, write, close — and supports in-place updates, locks, and directory hierarchies. It's optimized for small, frequently modified files. Block storage (EBS, iSCSI) exposes raw disk blocks and lets the OS layer a file system on top; it's used for databases and VMs that need low-latency random I/O. Blob storage sacrifices in-place updates and POSIX semantics in exchange for near-infinite horizontal scalability, built-in replication, and dirt-cheap per-GB pricing. If your data is "write once, read many," blob storage is almost always the right choice.

How It Works

PUT — Uploading an Object

  1. The client sends an HTTP PUT request with the object key, metadata headers (content type, custom tags), and the binary payload.
  2. The storage service front-end validates the request (authentication, authorization, quota checks) and computes a content hash (SHA-256 or MD5) for integrity verification.
  3. The service determines placement: which storage nodes will hold the replicas. Typically 3 replicas across different failure domains (racks, availability zones, or regions).
  4. The object is written to the primary node, then synchronously or asynchronously replicated to secondary nodes depending on the durability guarantee. S3, for instance, synchronously writes to at least 2 AZs before returning 200 OK.
  5. Metadata (key, size, content hash, storage class, ACL, custom metadata) is stored in a separate metadata store — often a distributed key-value store or database optimized for point lookups.
  6. For objects larger than a threshold (typically 5-100 MB), multipart upload is used: the client splits the file into parts, uploads them in parallel, then sends a "complete" request. The service assembles the parts and creates a single logical object.

Multipart Upload Mechanics — Step by Step

Why does multipart upload exist? Imagine uploading a 5 GB video file over a single HTTP PUT request. If the connection drops at 4.8 GB (96% complete), you must restart the entire 5 GB upload from byte zero. On a 50 Mbps connection, that 5 GB upload takes about 13 minutes — and you just lost all of it. Multipart upload solves this by splitting the file into independently uploadable parts. If one part fails, you retry only that part, not the whole file.

Here is a concrete walkthrough with a 5 GB video file split into 4 parts of 1.25 GB each:

Step Action Details
1 InitiateMultipartUpload Client calls the API with the target key (videos/concert.mp4). The service returns an upload_id (e.g., abc-123-xyz). This upload_id ties all subsequent part uploads together. No data is transferred yet.
2 Upload parts 1–4 in parallel The client reads the file in 1.25 GB chunks and uploads each as a separate HTTP PUT:
• Part 1: bytes 0 – 1,342,177,279 (1.25 GB)
• Part 2: bytes 1,342,177,280 – 2,684,354,559
• Part 3: bytes 2,684,354,560 – 4,026,531,839
• Part 4: bytes 4,026,531,840 – 5,368,709,119
Each part upload returns an ETag (content hash). All 4 uploads run in parallel — on a 100 Mbps connection, each 1.25 GB part takes ~100 seconds, so total wall-clock time is ~100 seconds (vs. ~400 seconds sequential).
3 Part 3 fails at 600 MB — retry only part 3 The connection drops while uploading Part 3. Parts 1, 2, and 4 completed successfully. The client retries only Part 3 from byte 0 of that part (not byte 0 of the file). Cost of failure: re-upload 1.25 GB (one part), not 5 GB (the whole file). Without multipart: restart 5 GB. With multipart: restart 1.25 GB. That is a 4x reduction in wasted bandwidth on failure.
4 CompleteMultipartUpload Client sends the upload_id along with a list of (part_number, ETag) pairs for all 4 parts. The service verifies all parts are present, assembles them in order into a single 5 GB object, computes the final content hash, stores the metadata, and returns 200 OK with the final ETag. The object is now accessible via a normal GET request.

Resumability example: Suppose you are uploading a 20 GB database backup in 10 parts of 2 GB each. After parts 1–7 complete successfully, your server restarts. When it comes back, you call ListParts with the upload_id to discover which parts already uploaded. The service returns parts 1–7 with their ETags. You only need to upload parts 8, 9, and 10 — saving 14 GB of re-upload. Without multipart, you would discard all 14 GB of progress and start over.

Important detail: incomplete multipart uploads consume storage but are not visible as objects. If a client initiates a multipart upload but never completes it, the uploaded parts sit in storage indefinitely, accruing costs. Always set a lifecycle rule to abort incomplete multipart uploads after 7 days: this automatically cleans up orphaned parts.

GET — Retrieving an Object

  1. The client sends an HTTP GET request with the object key.
  2. The service looks up the key in the metadata store to find which storage nodes hold the object and its storage class.
  3. The service routes the request to the nearest or least-loaded replica node.
  4. The node reads the object from disk and streams it back to the client. Range requests (Range: bytes=0-1023) are supported for partial reads.
  5. The response includes the content hash in an ETag header so the client can verify integrity.

DELETE — Removing an Object

  1. A DELETE request removes the metadata entry. With versioning enabled, the object is not physically deleted but marked with a delete marker; previous versions remain accessible.
  2. Physical garbage collection runs asynchronously to reclaim storage from deleted or overwritten objects.

LIST — Enumerating Objects

LIST returns objects matching a key prefix, paginated by continuation token. This operation is eventually consistent in many implementations — a newly uploaded object may not appear in a LIST result for a few seconds. S3 achieved strong read-after-write consistency for GET and LIST in December 2020, but many S3-compatible stores still use eventual consistency for listings.

Content Addressing & Deduplication

Some blob storage systems use content-addressable storage (CAS), where the object's key is derived from its content hash. If two users upload the same file, only one copy is stored. This is how Git stores objects internally and how backup systems like Restic and Borg achieve deduplication.

How Content-Addressable Storage Works

The core idea is simple: instead of naming objects by user-chosen keys, you name them by the hash of their content. When you store an object, the system computes SHA-256(object_bytes) and uses that hash as the storage key. Two identical files, uploaded by different users at different times, produce the same hash and therefore map to the same storage location. The system stores only one copy.

But whole-file hashing only deduplicates exact copies. What about files that are mostly identical? This is where chunked deduplication comes in. The system splits each file into fixed-size or variable-size chunks (typically 1 MB each), hashes each chunk independently, and stores only unique chunks.

Worked Example: Chunked Deduplication

Consider two files uploaded by different users:

  • File A: 100 MB video, split into 100 chunks of 1 MB each. Each chunk is hashed with SHA-256. All 100 chunks are unique (no prior data in the system), so all 100 are stored. Storage used: 100 MB.
  • File B: 100 MB video — a slightly re-edited version of File A. 95% of the content is identical. Split into 100 chunks of 1 MB each. The system hashes each chunk and checks: does this hash already exist in storage?
Metric Without Dedup With Chunked Dedup
Total chunks across both files 200 200
Duplicate chunks (same hash) 0 (stored twice) 95 (stored once, referenced twice)
Unique chunks stored 200 105 (100 from File A + 5 new from File B)
Total storage used 200 MB 105 MB
Dedup ratio 1.0x 1.9x (200 MB logical / 105 MB physical)

The dedup ratio formula: dedup_ratio = total_logical_size / total_physical_size. A ratio of 2.0x means you are storing half the raw bytes. For VM snapshots and code repositories, dedup ratios of 5x–20x are common because successive versions share most of their content. For unique media files (photos, audio recordings), the ratio is close to 1.0x — dedup provides little benefit.

How retrieval works with CAS: The system stores a manifest (also called a recipe) for each logical file. The manifest is an ordered list of chunk hashes. To reconstruct File B, the system reads the manifest, fetches each chunk by hash from the chunk store, and concatenates them in order. The 95 shared chunks are read from the same physical location as File A's chunks — there is only one copy on disk.

Erasure Coding: Durability Without 3x Replication

The problem: The simplest way to protect against data loss is triple replication — store 3 complete copies of every object on different disks/nodes. If any one copy is lost (disk failure, node crash), two copies remain. This is simple and fast to read (any copy can serve a request), but it has a brutal cost: 3x storage overhead. Storing 10 PB of data requires 30 PB of raw disk capacity.

The insight: What if instead of storing complete copies, you could store mathematical fragments that allow you to reconstruct the original data even if some fragments are lost? This is erasure coding, and it achieves the same (or better) durability as triple replication at a fraction of the storage cost.

Reed-Solomon Erasure Coding: How It Works

Reed-Solomon is the most widely used erasure coding algorithm (used by S3, Azure Blob Storage, Google Colossus, Ceph, and HDFS). The configuration is described as RS(k, m) where:

  • k = number of data fragments the original object is split into
  • m = number of parity fragments computed from the data fragments
  • Total fragments stored = k + m
  • You can lose any m fragments and still reconstruct the original object from the remaining k fragments

Worked Example: RS(6, 3)

Let us walk through storing a 10 MB object with Reed-Solomon (6, 3):

Step Action Result
1. Split Divide the 10 MB object into k=6 equal data fragments 6 data fragments, each 10 MB / 6 = 1.67 MB (D1, D2, D3, D4, D5, D6)
2. Encode Compute m=3 parity fragments using Reed-Solomon matrix math 3 parity fragments, each 1.67 MB (P1, P2, P3). Each parity fragment is a mathematical combination of all 6 data fragments.
3. Distribute Store each of the 9 fragments on a different disk/node/AZ 9 fragments × 1.67 MB = 15 MB total stored
4. Overhead Storage overhead = total stored / original size 15 MB / 10 MB = 1.5x overhead

Fault tolerance: With RS(6, 3), you can lose any 3 of the 9 fragments and still reconstruct the original 10 MB object from the remaining 6 fragments. This means the system survives the simultaneous failure of 3 disks, 3 nodes, or even 3 entire availability zones.

Reconstruction example: Suppose fragments D2, D5, and P1 are lost (their disks failed). The system reads the remaining 6 fragments: D1, D3, D4, D6, P2, P3. Using the Reed-Solomon decoding algorithm (which inverts the encoding matrix), it recomputes the original data and regenerates D2 and D5. The reconstruction is exact — no data loss, no approximation.

Cost Comparison: Replication vs. Erasure Coding at Scale

Strategy 10 PB of Logical Data Raw Storage Required Overhead Monthly Cost (at $0.023/GB)
3-way replication 10 PB 30 PB 3.0x 30,000 TB × $0.023 = $690,000
RS(6, 3) erasure coding 10 PB 15 PB 1.5x 15,000 TB × $0.023 = $345,000
Savings — 15 PB saved — ~$345,000/month saved

The tradeoff: Erasure coding is not free. Reading an erasure-coded object requires fetching k fragments from k different nodes and decoding them, which is slower than reading a single complete replica. This is why hot data (frequently accessed) often uses replication for speed, while cold/warm data uses erasure coding for cost efficiency. S3 uses replication for Standard class and erasure coding for Infrequent Access, Glacier, and internal cold tiers.

Why not RS(10, 2) for even lower overhead? With k=10 and m=2, the overhead drops to 1.2x, but you can only tolerate 2 simultaneous failures. The choice of k and m is a tradeoff between storage efficiency and fault tolerance. Most production systems use m=3 or m=4 to handle multiple concurrent failures during the repair window (the time it takes to rebuild a failed fragment onto a new disk).

When to Use / When NOT to Use

Use Blob Storage When

  • Storing images, videos, or audio — media files are large, immutable, and read-heavy; blob storage excels here.
  • Application backups & snapshots — cheap, durable, lifecycle policies auto-expire old backups.
  • Serving static assets — CSS, JS, fonts served via CDN with S3 as origin; pay for storage, not compute.
  • Data lake / analytics raw zone — Parquet, ORC, CSV files queried by Spark, Presto, or Athena directly from S3.
  • Log archives — centralized, compressed log storage; lifecycle moves to Glacier after 30 days.
  • ML training data — large datasets (TBs) that training jobs read sequentially.

Don't Use Blob Storage When

  • Frequently updated small records — a relational or key-value database is far better for small, mutable data. Objects are not designed for in-place partial updates.
  • Structured queries (SQL-like) — you can't JOIN or WHERE across objects natively. Use a database. (S3 Select is limited to single-file scans.)
  • Low-latency transactional access — blob storage GETs are 10-100ms, not microseconds. Use Redis, DynamoDB, or a local cache for hot-path reads.
  • POSIX file system needs — applications that use seek(), file locks, or in-place writes need EBS/EFS, not S3.
  • Small, high-frequency writes — writing millions of 1 KB objects/sec is expensive (request costs dominate). Use a database or append-only log.

Classic vs. Modern Approaches

Classic: Simple File Storage

In the early days, unstructured data lived on NFS shares or SAN volumes. Files were organized in directories, accessed via POSIX APIs, and backed up with rsync or tape. This approach broke down at scale: NFS has limited concurrency, directory listings slow down with millions of files, and replication/durability required expensive hardware RAID or manual scripts. Capacity was bounded by the largest disk array you could buy.

Modern: Content-Addressable Storage & Deduplication

Modern blob stores like S3, GCS, and Azure Blob Storage hash object content to detect duplicates. Sub-file deduplication (chunking the file and hashing each chunk) is used by backup tools like Restic and enterprise storage systems. This can reduce storage costs by 30-70% for workloads with redundant data (e.g., VM snapshots, code repositories).

Tiered Storage & Lifecycle Policies

S3 introduced storage classes — Standard, Infrequent Access (IA), One Zone-IA, Glacier Instant Retrieval, Glacier Flexible Retrieval, and Glacier Deep Archive — with pricing that drops from $0.023/GB/month to $0.00099/GB/month. Lifecycle policies automatically transition objects between tiers based on age. A common pattern: keep data in Standard for 30 days, move to IA for 90 days, then archive to Glacier. This single feature can cut storage costs by 60-80%.

But the retrieval fees and wait times mean you should only use it for data you almost never read — compliance archives, legal holds, disaster recovery backups. The key insight is to match the storage class to the access pattern: use lifecycle policies to automatically transition objects as they age and get accessed less frequently.

S3 Select & Query-in-Place

Instead of downloading an entire object to filter it, S3 Select pushes a SQL expression to the storage layer. The service scans CSV, JSON, or Parquet files and returns only matching rows. This reduces data transfer and processing time by up to 400%. For heavier workloads, Athena and Presto provide full SQL over S3.

Table Formats on Object Storage

Delta Lake, Apache Iceberg, and Apache Hudi add ACID transactions, schema evolution, and time travel on top of Parquet files stored in S3. This turns blob storage into a data lakehouse — combining the low cost of object storage with the query capabilities of a data warehouse.

Pre-Signed URLs for Secure Direct Upload

Rather than proxying uploads through your application server (which becomes a bottleneck), generate a pre-signed URL that lets the client upload directly to blob storage. The URL embeds an expiring signature, so the client never needs storage credentials. This pattern is standard for user-uploaded content (profile photos, file attachments, video uploads) and eliminates the application server from the data path entirely.

Assumptions & Model

Key Assumptions

  • Objects are immutable (or rarely updated). Blob storage is optimized for write-once-read-many workloads. Overwriting an object creates a new version; there's no partial update. If your data changes every second, use a database instead.
  • Eventual consistency for listings is acceptable. While individual GETs are strongly consistent (read-after-write), LIST operations may not immediately reflect recent writes in all implementations. Design your application to tolerate this (e.g., don't rely on LIST right after PUT to confirm upload success; use the returned ETag instead).
  • Objects are accessed by key, not queried by content. There's no index on object contents. If you need to search "all images tagged sunset," maintain a separate metadata index in a database, not in the blob store itself.
  • Durability over latency. Blob storage trades latency (10-100ms per GET) for extreme durability (11 nines). If you need sub-millisecond access, add a caching layer in front.
  • Network egress is a significant cost. Many cost models assume storage is cheap but data retrieval (egress) is expensive. Always factor in how often objects are read, not just how much data is stored.

Sizing & Tradeoffs

SLO-Driven Sizing Examples

Storage architecture decisions should be driven by concrete SLOs — not gut feel. Each example below starts with the business requirement and derives the infrastructure from it.

1. Photo Sharing Platform

SLOs: p95 upload < 3 s • p99 download < 500 ms • 10M photos/day • avg 4 MB each

Daily ingest: 10M × 4 MB = 40 TB/day ≈ 1.2 PB/month.

Upload latency ≤ 3 s at p95: At 4 MB per photo the minimum sustained throughput per upload must be ≥ 1.33 MB/s. Multipart upload with a 4 MB chunk size (single PUT) avoids multipart overhead for 90% of images. For the top-10% larger images (> 8 MB), use 3 parallel 4 MB parts — total wall-clock stays under 3 s on a 10 Mbps uplink.

Download latency ≤ 500 ms at p99: S3 Standard p99 TTFB is ~200 ms, leaving 300 ms for network transfer. A 4 MB image at 300 ms requires ≥ 13 MB/s — feasible within-region but not cross-continent. Derived decision: deploy a CDN (CloudFront / Cloudflare) with regional edge caches. Target cache-hit ratio ≥ 92% to keep origin load manageable. Pre-warm popular content at upload time via cache-push.

Storage tier: S3 Standard for the first 30 days (hot access window), then lifecycle to S3 Infrequent Access. At steady state (12 months): ~1.2 PB hot + ~12 PB warm. Monthly cost: 1,200 TB × $0.023 + 12,000 TB × $0.0125 ≈ $177,600/month.

2. Video Platform

SLOs: p95 first-byte time < 2 s • 100 PB total storage • 99.999% durability (11-nines at object level)

First-byte ≤ 2 s at p95: Range-GET on the first segment of an HLS/DASH manifest must return in < 2 s. S3 Standard GET p95 is ~150 ms; CDN edge-hit adds ~20 ms. The bottleneck is cold miss at edge: origin fetch + TLS + transfer of a 2–6 MB segment. Derived decision: keep the first 3 segments of every video in a hot cache tier (origin shield), and use S3 Standard for all content accessed within the last 7 days.

Durability 99.999%: S3 already provides 99.999999999% (11-nines) durability via 3-AZ replication internally. To guard against regional failure, enable cross-region replication (CRR) to a second region for the canonical copy. Replication factor: 2 regions × 3 AZ = 6 effective copies.

Storage class mix (100 PB): Access-pattern analysis shows 5% hot (last 7 days), 15% warm (8–90 days), 80% cold (> 90 days). Derived layout: 5 PB Standard + 15 PB IA + 80 PB Glacier Instant Retrieval. Monthly cost: 5,000 TB × $0.023 + 15,000 TB × $0.0125 + 80,000 TB × $0.004 ≈ $622,500/month (vs. $2.3M all-Standard).

3. Document Management SaaS

SLOs: p99 retrieval < 1 s • 50 TB total • 99.99% availability • compliance: 7-year retention

Retrieval ≤ 1 s at p99: Average document is 2 MB (PDF/DOCX). At p99, S3 Standard GET + transfer of 2 MB completes in ~250 ms within-region. The budget leaves 750 ms for application-layer lookup, auth, and pre-signed URL generation. Derived decision: metadata index in DynamoDB (single-digit ms), generate pre-signed URLs server-side, serve via S3 Standard for all documents accessed in the last 180 days.

99.99% availability: S3 Standard is designed for 99.99% availability. No additional redundancy needed at the storage layer. Derived decision: focus availability budget on the metadata/auth layer — deploy across 2 AZs with automated failover.

Hot/warm/cold tiering (50 TB): Usage analysis shows 20% of documents accessed monthly (hot), 30% accessed quarterly (warm), 50% untouched for > 1 year but must be retrievable in < 1 s for compliance audits. Layout: 10 TB Standard + 15 TB IA + 25 TB Glacier Instant Retrieval. Monthly cost: 10,000 GB × $0.023 + 15,000 GB × $0.0125 + 25,000 GB × $0.004 = $518/month.

Retention: Enable S3 Object Lock (Compliance mode) with a 7-year retention period. Use S3 Lifecycle to transition to Glacier Deep Archive after 2 years for cost savings on the coldest 50%.

4. ML Training Data Lake

SLOs: sustained read throughput > 10 GB/s • 500 TB dataset • training job start-to-first-batch < 60 s

10 GB/s sustained reads: A single S3 GET achieves ~100 MB/s. To hit 10 GB/s you need ≥ 100 parallel streams. Derived decision: shard the dataset into 100–500 prefix groups (e.g., s3://bucket/shard-000/ through s3://bucket/shard-499/). S3 partitions throughput by prefix — distributing reads across prefixes avoids 503 SlowDown throttling at 5,500 GET/s per prefix.

Storage layout: Store data as 256 MB–1 GB Parquet/TFRecord files (not millions of small files). This maximizes sequential read throughput and minimizes LIST/GET request overhead. At 500 TB with 512 MB average file size = ~1M objects — manageable for S3.

Start-to-first-batch < 60 s: Pre-compute a shuffled manifest file listing all object keys. At job start, each worker reads its manifest shard and issues parallel GETs immediately. Use S3 Select or byte-range GETs to fetch only needed columns, reducing bytes transferred by up to 80%.

Cost optimization: Training data is write-once, read-many during a training window (days–weeks), then cold. Use S3 Standard during active training, then lifecycle to IA after 30 days. 500 TB × $0.023 = $11,500/month (active) vs. 500 TB × $0.0125 = $6,250/month (idle).

Parallel read architecture: Deploy training nodes in the same region and AZ as the S3 bucket. Use S3 Transfer Acceleration only for cross-region scenarios. For GPU clusters, mount via S3 Mountpoint (FUSE) or use a caching layer (Alluxio / FSx for Lustre backed by S3) to avoid re-reading the same data across epochs.

SLO-Driven Capacity Planning

Capacity planning for blob storage should work backward from SLOs, not forward from estimated data volume. The process follows three phases:

Phase Input Output Key Question
1. SLO Definition Business requirements, user expectations Latency targets (p50/p95/p99), durability, availability What does the user experience if we miss this?
2. Architecture Derivation SLOs + estimated data volume + access patterns Storage tier mix, replication factor, CDN strategy, chunk size What is the cheapest architecture that meets every SLO?
3. Cost Envelope Derived architecture + pricing model Monthly cost range, cost-per-user, cost scaling curve At what scale does this architecture break the budget?

Common mistakes in capacity planning:

Over-provisioning durability: Cross-region replication doubles storage cost. Only enable it when the durability SLO exceeds what a single region provides (11-nines), or when the availability SLO requires surviving a full region outage.

Ignoring egress: A 100 TB/month egress bill ($9,000) can exceed storage costs. CDN cache-hit ratios above 90% are not free — they require careful cache-key design and origin-shield placement.

Wrong chunk size: Too small (e.g., 1 MB) increases request count and per-request overhead. Too large (e.g., 64 MB) wastes bandwidth on partial reads and hurts upload latency on slow connections. Derive chunk size from the upload latency SLO and the target connection speed at p95.

Flat-tiering everything: Storing 100% of data in S3 Standard is 3–5x more expensive than a lifecycle-managed mix. Run access-frequency analysis before launch, then re-validate monthly.

Scaling Journey

0 – 1K users

Local Filesystem

Store uploads in /var/uploads on your single server. Simple, zero cost, no external dependencies. Backup with rsync to a second machine. This works fine for a small app with hundreds of files.

Signal to evolve: You need a second server (load balanced), and now uploads live on only one machine.

1K – 100K users

Single NFS or Object Store

Move files to a shared NFS mount or a single MinIO/S3 bucket. Both app servers can now access uploads. S3 is preferable — it handles replication, durability, and capacity automatically. Start using pre-signed URLs for direct client uploads.

Signal to evolve: Storage exceeds 1 TB, or you need to serve content globally with low latency.

100K – 10M users

S3 + CDN + Lifecycle

Add CloudFront or Cloudflare in front of S3. Enable lifecycle policies to tier old data to IA/Glacier. Use S3 event notifications to trigger image resizing, video transcoding, or virus scanning via Lambda or SQS. Enable versioning and cross-region replication for disaster recovery.

Signal to evolve: Egress costs are massive, or you need multi-region active-active writes.

10M+ users

Multi-Region + Custom Storage

Run S3 with multi-region access points for geo-local reads and writes. Consider building custom storage (like Dropbox's Magic Pocket or Meta's f4) if you're at exabyte scale where S3 pricing doesn't make sense. Implement intelligent tiering (S3 Intelligent-Tiering) to automate class transitions based on access patterns. Use S3 Batch Operations for bulk migrations.

What Can Go Wrong

Accidental Deletion Without Versioning

What happened: A developer ran a cleanup script with an overly broad prefix match, deleting 3 million production images from S3. With no versioning enabled, the objects were permanently gone. The only recovery path was restoring from a 36-hour-old backup, meaning all uploads in that window were lost.

Root cause: No versioning, no MFA Delete, and the IAM policy granted s3:DeleteObject to the application role without requiring confirmation. The cleanup script used a prefix of images/ instead of images/temp/.

Prevention: Enable S3 versioning on all production buckets. Enable MFA Delete for permanent version removal. Use Object Lock for compliance-critical data. Restrict DeleteObject permissions to a dedicated cleanup role, never the application role.

Egress Cost Surprise

What happened: A startup used S3 to serve user-generated video content directly to browsers. Monthly storage cost was $2,000, but the first egress bill was $47,000 — 500 TB of outbound transfer at $0.09/GB. Nobody had modeled the read-to-write ratio or egress pricing.

Root cause: No CDN layer, so every video view was a direct S3 fetch. Popular videos were fetched thousands of times from S3 instead of being cached.

Prevention: Always put a CDN in front of public-facing blob storage. Model egress costs before launch: monthly_views × avg_object_size × $0.09/GB. Set up billing alerts at 50% and 80% of budget. Consider S3-compatible alternatives with free egress (Cloudflare R2, Backblaze B2 + Cloudflare).

Public Bucket Data Leak

What happened: Multiple companies (Capital One, Twitch, Accenture, the US DoD) have exposed sensitive data through misconfigured S3 bucket policies that allowed public read access. In Capital One's case, a misconfigured WAF combined with an overly permissive IAM role let an attacker exfiltrate 100 million credit card applications from S3.

Root cause: S3 buckets historically defaulted to private, but it was easy to accidentally add "Principal": "*" to a bucket policy. Legacy ACLs added confusion.

Prevention: Enable S3 Block Public Access at the account level. Use AWS Config rules to detect and auto-remediate public buckets. Never rely on bucket names being "secret" — they're enumerable. Audit bucket policies regularly. Use AWS Access Analyzer to identify unintended external access.

S3 Prefix Partitioning and Request Throttling

S3 automatically partitions your bucket's key space to distribute load across many backend servers. Each partition can handle approximately 5,500 GET/HEAD requests per second and 3,500 PUT/COPY/POST/DELETE requests per second. When all your keys share the same prefix, they land on the same partition — and you hit the throttle limit far sooner than you might expect.

The Problem: Sequential Keys

Suppose you name your images with sequential numeric keys: 

images/001.jpg
images/002.jpg
images/003.jpg
...
images/999999.jpg

All of these keys share the prefix images/0, images/00, etc. S3 partitions by prefix, so all these objects are likely served by the same partition. Your maximum throughput is capped at 5,500 GETs/second for the entire bucket — regardless of how many objects it contains.

The Solution: Random Prefixes

Add a random hash prefix to distribute keys across many partitions: 

a3f2/images/001.jpg
b1x5/images/002.jpg
7e9d/images/003.jpg
c4a1/images/004.jpg

With a 4 hex-character prefix, you create up to 164 = 65,536 distinct prefix partitions. Each partition supports 5,500 GETs/sec, so the theoretical maximum throughput becomes:

65,536 partitions × 5,500 GETs/sec = ~360 million GETs/second

In practice, you will not reach 360M GETs/sec (network and other limits apply), but the point is that randomized prefixes remove the partition bottleneck entirely. S3 also auto-splits hot partitions over time, but this takes minutes to hours — random prefixes give you the throughput immediately.

Key Strategy Example Key Partitions Max GET/sec
Sequential numeric images/000001.jpg ~1 (all share prefix) ~5,500
Date-prefixed 2024/03/09/photo.jpg ~1 per day (today's traffic on 1 partition) ~5,500
2 hex-char random prefix a3/images/001.jpg 256 ~1.4M
4 hex-char random prefix a3f2/images/001.jpg 65,536 ~360M
UUID as key f47ac10b-58cc-4372.jpg Effectively unlimited Effectively unlimited

Implementation tip: compute the prefix as the first N characters of SHA-256(original_key) or use a UUID. Store the mapping in your metadata database. When generating pre-signed URLs or building CDN paths, include the prefix. The prefix is invisible to end users if your CDN rewrites the URL.

Interactions & Pitfalls

Blob Storage + CDN

The most common pattern: S3 as origin, CloudFront/Cloudflare as CDN cache. The CDN caches objects at edge locations worldwide, reducing both latency and S3 egress costs. Pitfall: CDN cache invalidation is not instant. If you overwrite an S3 object, the CDN may serve the stale version for minutes or hours until the TTL expires. Solutions: use versioned URLs (image-v2.jpg or image.jpg?v=abc123) instead of relying on cache purges. Also, CDN cache hit rates are low for long-tail content (unique per-user files) — don't expect CDN savings if every object is accessed only once.

Blob Storage + Database

Golden rule: never store blobs in a database. Store the blob in S3, store the S3 key (and metadata like size, content type, upload timestamp) in the database. This keeps your database small, fast, and easy to back up. A PostgreSQL table with 100M rows of metadata is manageable; 100M rows of 500 KB BLOBs is a 50 TB database — backups take days, queries are slow, and costs skyrocket. Pitfall: If the database insert succeeds but the S3 upload fails (or vice versa), you get orphaned records. Use the outbox pattern: upload to S3 first, then insert the key into the DB. If the DB insert fails, the S3 object becomes garbage (cleaned up by a periodic sweep). This is safer than the reverse because an orphaned S3 object is cheap; a database record pointing to nothing causes user-visible errors.

Blob Storage + Message Queues

S3 event notifications trigger processing pipelines: an image upload fires an SNS/SQS message, which triggers a Lambda function that resizes the image, generates thumbnails, runs OCR, or scans for malware. Pitfall: S3 event notifications guarantee at-least-once delivery, not exactly-once. Your processing Lambda must be idempotent — re-processing the same image should produce the same result without duplicating database entries or generating duplicate thumbnails. Also, event notifications can be delayed by seconds during high-throughput bursts; don't use them for time-critical workflows.

Cost Considerations

Dominant Cost Drivers

Blob storage costs break down into three components: storage (per GB/month), requests (per 1,000 operations), and egress (per GB downloaded). For read-heavy workloads, egress often dominates. For write-heavy workloads with many small objects, request costs dominate. For archive-heavy workloads, storage is the primary cost.

S3 Pricing Breakdown

Operation Cost Example at Scale
PUT, COPY, POST, LIST $0.005 / 1,000 requests 100M PUTs/month = $500
GET, SELECT $0.0004 / 1,000 requests 1B GETs/month = $400
Data Transfer Out $0.09/GB (first 10 TB), decreasing at volume 100 TB/month = ~$8,500
S3 to CloudFront $0.00 (free) Use CloudFront to eliminate S3 egress charges

Cost Optimization Strategies

  • CloudFront in front of S3: S3 → CloudFront transfer is free. CloudFront egress ($0.085/GB) is cheaper than direct S3 egress ($0.09/GB), and the CDN absorbs repeat requests entirely.
  • Lifecycle policies: Automatically transition data from Standard to IA after 30 days, to Glacier after 90 days. A 50 TB bucket drops from $1,150/month to ~$300/month.
  • S3 Intelligent-Tiering: For unpredictable access patterns. AWS automatically moves objects between tiers based on usage. Small per-object monitoring fee ($0.0025/1000 objects) but no retrieval charges.
  • Compression: Gzip or Zstd compress text-based objects (logs, JSON, CSV) before upload. Typical 3-5x size reduction. Store compressed; decompress on read.
  • Batch small objects: Aggregate tiny writes into larger objects. 1M × 1 KB objects costs $5 in PUTs; 1 × 1 GB object costs $0.005.
  • Free-egress alternatives: Cloudflare R2 offers S3-compatible API with zero egress fees. Backblaze B2 + Cloudflare Bandwidth Alliance also eliminates egress costs.

Cost of NOT Using Blob Storage

Storing files on EBS volumes costs $0.08-$0.10/GB/month (4x more than S3 Standard), requires manual capacity management, and offers no built-in replication. Storing blobs in a database is even worse: RDS storage costs $0.115/GB/month, and large BLOBs destroy database performance, bloat backups, and make point-in-time recovery impractical.

Real-World Examples

Netflix on S3

Netflix stores every video asset in S3 — the master source files, encoded renditions at multiple bitrates and resolutions, subtitles, and artwork. When a new show or movie is ingested, the content goes through a pipeline: upload to S3, transcode into hundreds of renditions (via a fleet of encoding workers pulling from SQS), and push the encoded files to Netflix's Open Connect CDN appliances deployed in ISP data centers worldwide. S3 serves as the durable "source of truth" while the CDN handles the actual streaming. Netflix uses S3's cross-region replication for disaster recovery and lifecycle policies to archive older renditions.

Dropbox: From S3 to Custom Storage and Partly Back

Dropbox originally stored all user files on S3. By 2015, they were one of S3's largest customers, spending tens of millions of dollars annually. They built Magic Pocket, a custom exabyte-scale storage system running on bare metal in their own data centers. The migration saved them significant costs — at their scale, owning hardware was cheaper than renting S3. However, Dropbox continues to use S3 for specific workloads (cold storage, certain regions) where their own infrastructure doesn't have presence.

GitHub: Git Objects in Blob Storage

Git itself uses content-addressable storage: every file, commit, and tree is stored as a blob identified by its SHA-1 hash. GitHub stores these Git objects in a distributed storage layer. For large files, Git LFS (Large File Storage) replaces blobs with lightweight pointers and stores the actual binary content in a blob store (backed by S3 or equivalent). This keeps Git repositories fast while supporting multi-GB assets like game textures or trained ML models.

Code Example

A Python client demonstrating the core blob storage operations using the boto3 S3 SDK: upload, download, and pre-signed URL generation.

import boto3
from botocore.config import Config

# Initialize client with retry config
s3 = boto3.client('s3', config=Config(
    retries={'max_attempts': 3, 'mode': 'adaptive'}
))

BUCKET = 'my-app-assets'

def upload_object(key, data, content_type='application/octet-stream'):
    """Upload bytes to S3 with server-side encryption."""
    s3.put_object(
        Bucket=BUCKET, Key=key, Body=data,
        ContentType=content_type,
        ServerSideEncryption='AES256',
        Metadata={'uploaded-by': 'api-service'}
    )
    return f"s3://{BUCKET}/{key}"

def download_object(key):
    """Download an object; returns bytes or None if missing."""
    try:
        resp = s3.get_object(Bucket=BUCKET, Key=key)
        return resp['Body'].read()
    except s3.exceptions.NoSuchKey:
        return None

def generate_presigned_upload_url(key, expires_in=3600):
    """Generate a pre-signed URL for direct client upload."""
    return s3.generate_presigned_url(
        'put_object',
        Params={'Bucket': BUCKET, 'Key': key},
        ExpiresIn=expires_in
    )

def generate_presigned_download_url(key, expires_in=3600):
    """Generate a pre-signed URL for time-limited download."""
    return s3.generate_presigned_url(
        'get_object',
        Params={'Bucket': BUCKET, 'Key': key},
        ExpiresIn=expires_in
    )

# Usage
uri = upload_object('photos/user42/avatar.jpg', image_bytes, 'image/jpeg')
url = generate_presigned_upload_url('videos/upload-abc.mp4')
# Client POSTs directly to `url` without needing AWS credentials

How to Validate Your Design

Load Testing

  • Upload throughput: Use s3-benchmark or a custom k6 script to measure max upload rate. Test with realistic object sizes (not just 1 KB). Measure at different concurrency levels (10, 100, 1000 parallel uploads).
  • Download throughput: Test single-object download speed vs. parallel multi-object downloads. Measure the impact of byte-range requests for large objects.
  • Multipart upload reliability: Upload a 5 GB file using multipart (100 × 50 MB parts). Simulate partial failures by killing the client mid-upload. Verify that incomplete multipart uploads are cleaned up (set a lifecycle rule to abort incomplete uploads after 7 days).
  • Tools: s3-benchmark, warp (for S3-compatible stores), k6 with aws-sdk extension, or vegeta hitting pre-signed URLs.

Chaos Testing

  • Simulate S3 unavailability by pointing the SDK at a nonexistent endpoint or injecting 503 errors with a proxy (toxiproxy). Verify your application degrades gracefully — e.g., returns a placeholder image instead of crashing.
  • Test what happens when a pre-signed URL expires mid-upload. The client should detect the 403 and request a fresh URL.
  • For self-hosted object stores (MinIO, Ceph), kill a storage node and verify the remaining replicas continue serving reads. Restore the node and verify rebalancing completes.

Benchmarking Methodology

  • Sweep object sizes: Performance characteristics differ dramatically between 1 KB, 1 MB, 100 MB, and 1 GB objects. Test each.
  • Measure from the right location: Latency from an EC2 instance in the same region is very different from latency from a user's browser across the ocean.
  • Account for S3 partitioning: S3 auto-partitions by key prefix. Sequential keys (001.jpg, 002.jpg) may hit the same partition and get throttled. Use randomized prefixes for high-throughput benchmarks.
  • Test lifecycle transitions: Upload objects, apply a lifecycle rule, advance the clock (or wait), and verify objects moved to the correct storage class. Check that retrieval from each tier works as expected.

Interview Questions

Design an image hosting service like Imgur

Start with the core flow: users upload images via a web form or API. The upload goes directly to S3 (using a pre-signed URL generated by your backend), bypassing your application server for the data transfer. On successful upload, the backend records the S3 key, original filename, content type, size, and uploader ID in a PostgreSQL database. The image URL is a deterministic function of the key: https://cdn.example.com/{key}.

For serving, put CloudFront in front of S3. Most images will be served from the CDN cache, keeping S3 egress near zero. Generate thumbnails asynchronously: S3 event notification triggers a Lambda that creates 3 sizes (small, medium, large) and writes them back to S3 under a thumbs/ prefix. The database record is updated with thumbnail keys.

For scale: use S3's automatic partitioning (random prefixes like a3f2/{uuid}.jpg), lifecycle policies to tier old images to IA, and content-based deduplication by hashing uploads before storing (if the hash already exists, return the existing URL instead of storing a duplicate).

Follow-ups: How would you handle NSFW content detection? (Add a moderation Lambda in the pipeline.) How do you handle hot images that go viral? (CDN absorbs the load; S3 auto-scales.) How do you delete images reliably? (Soft delete in DB, S3 object deletion via background job, CDN cache invalidation.)

How does S3 achieve 11 nines of durability?

S3 promises 99.999999999% (11 nines) durability, meaning if you store 10 million objects, you'd statistically expect to lose one object every 10,000 years. This is achieved through multiple redundancy layers: every object is automatically replicated across at least three physically separated Availability Zones within a region. Each AZ has independent power, cooling, and networking.

Under the hood, S3 uses a combination of replication and erasure coding. For frequently accessed data, full replicas may be kept. For infrequently accessed data, erasure coding (e.g., Reed-Solomon) splits the object into data and parity fragments across devices. Even if multiple disks or entire AZs fail simultaneously, the object can be reconstructed from the surviving fragments.

S3 continuously monitors data integrity: every object has a checksum verified on write and periodically re-verified on disk. If a bit rot or disk failure is detected, S3 automatically re-replicates the affected data from surviving copies. The system is designed so that no single failure (disk, server, rack, or entire AZ) causes data loss.

Follow-ups: What is the difference between durability and availability? (Durability = data not lost; availability = data accessible. S3 Standard offers 99.99% availability, meaning ~53 minutes of downtime per year.) Can you get higher than 11 nines? (Cross-region replication adds geographic redundancy but is about surviving region-level disasters, not increasing per-object durability.)

How to optimize blob storage costs at scale?

Cost optimization is a multi-lever problem. First, classify your data by access pattern: hot (accessed frequently), warm (accessed occasionally), cold (rarely accessed), and archive (accessed almost never). Apply lifecycle policies to automatically move data through tiers: Standard → IA (after 30 days) → Glacier (after 90 days) → Deep Archive (after 1 year). This alone can cut storage costs by 60-80% for data with a decaying access curve.

Second, tackle egress costs. Put a CDN in front of S3 for public content (S3 → CloudFront transfer is free). For internal services, keep compute in the same region as storage to avoid cross-region transfer fees. Consider Cloudflare R2 or Backblaze B2 for workloads where egress dominates costs — both offer zero or near-zero egress fees.

Third, reduce request costs by batching. Instead of storing millions of tiny objects, aggregate them into larger files (e.g., tar archives or Parquet files) with an index. Use multipart upload for large objects to improve reliability without increasing cost. Finally, compress text-based data (logs, JSON) with Zstd before uploading — 3-5x reduction in both storage and egress.

Follow-ups: When does it make sense to build your own storage? (At exabyte scale, like Dropbox's Magic Pocket. Below ~100 PB, managed services are almost always cheaper when you factor in engineering time.) How do you handle data that's infrequently accessed but needs fast retrieval? (S3 Glacier Instant Retrieval: $0.004/GB with millisecond access.)

Design a video processing pipeline using object storage

The pipeline has four stages: ingest, transcode, store, and deliver. For ingest, the client obtains a pre-signed URL from the API, then uploads the raw video directly to S3 (avoiding proxying through your servers). Use multipart upload for files over 100 MB — this allows parallel chunk uploads and resumability on network failure.

When the upload completes, S3 sends an event notification to an SQS queue. A fleet of transcode workers (EC2 Spot instances or Kubernetes jobs) polls the queue, downloads the raw video, and encodes it into multiple renditions (1080p, 720p, 480p) in formats like H.264/H.265 with HLS packaging. Each rendition is uploaded back to S3 under a structured key: videos/{id}/1080p/manifest.m3u8.

For delivery, a CDN serves the HLS manifests and video segments. The manifest file points to segment URLs under the CDN domain. For adaptive bitrate streaming, the player switches between renditions based on network conditions. Use S3 lifecycle policies to delete raw uploads after 30 days (keeping only the encoded renditions), and move old renditions to IA after 90 days.

Follow-ups: How do you handle failed transcode jobs? (Dead letter queue + retry with backoff. Store the error in a status table. Alert if DLQ depth exceeds threshold.) How do you prioritize premium users? (Separate SQS queues with different worker pool sizes, or priority-based message attributes.)

Pre-signed URLs — how do they work and when to use them?

A pre-signed URL is a regular S3 URL with an embedded cryptographic signature that grants temporary access to a specific operation (GET or PUT) on a specific object. The signature encodes: the AWS access key ID of the signer, the allowed operation, the object key, an expiration timestamp, and any conditions (e.g., max upload size, required content type). The URL can be used by anyone — no AWS credentials needed.

The flow: (1) Client requests an upload URL from your API. (2) Your backend generates a pre-signed PUT URL using the AWS SDK (requires IAM credentials with s3:PutObject permission). (3) Client uploads directly to S3 using the URL. (4) S3 verifies the signature, checks expiration, and if valid, accepts the upload. (5) Your backend is notified via S3 event or the client calls back.

Use pre-signed URLs when: clients need to upload/download large files (avoids proxying through your server), you want to grant temporary access without sharing credentials, or you need to serve private content with time-limited links (e.g., paid download links that expire after 1 hour). Don't use them when: the content is public (just use a public CDN), or when you need server-side validation of the content before it's stored (use a proxy upload instead and validate before forwarding to S3).

Follow-ups: What happens if the signing IAM credential is rotated? (URLs signed with the old key stop working. Use IAM roles with session credentials, not long-lived access keys.) Can you restrict the upload size? (Yes, use presigned POST with conditions that limit content-length-range.)

How would you migrate from local storage to S3?

Migration happens in phases to avoid downtime. Phase 1 (dual-write): Modify the upload path to write to both local storage and S3. New uploads exist in both places. Verify S3 writes by comparing checksums. This phase validates that S3 integration works correctly in production without risking existing data.

Phase 2 (backfill): Run a background migration job that copies all existing files from local storage to S3. Use aws s3 sync or a custom script with parallelism. For large datasets (TBs), use AWS Snowball or S3 Transfer Acceleration. Track progress in a database table (file path, S3 key, migration status, checksum). Verify every migrated file's checksum matches.

Phase 3 (read fallback): Modify the read path to check S3 first. If the object doesn't exist in S3 (migration hasn't reached it yet), fall back to local storage. This is a read-through migration pattern. As the backfill completes, fewer and fewer reads hit local storage.

Phase 4 (cutover): Once backfill is 100% complete and verified, remove the local storage read fallback. Stop dual-writes. Decommission local storage volumes. Keep a backup of the local storage for 30 days as insurance. Update all URL references (database records, configs) from local paths to S3 keys or CDN URLs.

Follow-ups: How do you handle files uploaded during migration? (Dual-write ensures new files go to both places. The backfill only needs to cover files created before dual-write started.) What if the migration takes weeks and you can't afford downtime? (The dual-write + fallback approach is zero-downtime by design.)

Key Takeaways

  • Blob/object storage is the default choice for unstructured data — images, videos, backups, logs, and data lake files. Its flat key-value model trades POSIX semantics for infinite horizontal scalability.
  • The API is simple by design: PUT, GET, DELETE, LIST. This simplicity enables the extreme durability (11 nines) and scale that S3-compatible stores achieve.
  • Never store blobs in a database. Store the S3 key in the database and the blob in object storage. This keeps your database fast and your storage costs low.
  • Egress is the hidden cost killer. Always put a CDN in front of public-facing blob storage, and consider zero-egress alternatives like Cloudflare R2 for read-heavy workloads.
  • Lifecycle policies are the single most impactful cost optimization. Automatically tiering data from Standard to IA to Glacier can cut storage costs by 60-80%.
  • Use pre-signed URLs for client uploads to avoid proxying large files through your application server. This eliminates a bandwidth bottleneck and reduces server costs.
  • Enable versioning on production buckets. Without it, an accidental delete is permanent. With it, recovery is a single API call.
  • Batch small objects. Writing millions of tiny objects is expensive in request costs and inefficient for storage. Aggregate into larger files where possible.
