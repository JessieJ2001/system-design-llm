Failure Modes

The canonical catalog of how distributed systems break — what triggers each failure, the blast radius, how you detect it, how you recover. This is the source-of-truth page; scenario "What Breaks" sections classify against this taxonomy instead of redefining failures.

~12 min reference

How to Use This Page

The C5 rubric requires every iteration to enumerate ≥3 failure modes with blast radius, detection, and recovery. This page is the menu. Use it during S5:

  1. For each iteration's new design, walk this catalog and ask: which of these does the new design newly expose, leave unaddressed, or amplify?
  2. For each applicable mode, the page gives you the canonical blast radius / detection / recovery shape. The scenario fills in the scenario-specific numbers.
  3. Cite this page instead of redefining a mode. Scenario pages link to failure-modes.html#stampede, #retry-storm, etc.

Failure modes vs bottlenecks

A bottleneck is what saturates first under expected load (steady-state). A failure mode is what breaks under stress — overload, partial outage, transient surge, deploy. Some entries appear in both pages (stampede, retry storm, backlog overflow) because the steady-state version is a bottleneck and the under-stress version is a failure. The two pages cross-reference each other on those.

The three required answers per failure

For each failure a scenario page enumerates, the C5 rubric asks for three numbers/answers, not one:

  • Blast radius. What fraction of users / what data is affected? "All requests to one shard for 5 min" vs "global write outage" are very different conversations.
  • Detection mechanism. What signal fires, and at what threshold? "p99 latency > 200 ms for 1 min" vs "error rate > 1%" — be specific.
  • Recovery action. Automatic (system self-heals) or operator (page someone)? If automatic, what's the recovery time bound? If operator, what's the runbook step 1?

The degradation contract

Beyond per-failure enumeration, every Final Design states a degradation contract: under overload or partial failure, what does the system serve, and what does it refuse to do? Examples: "Serve stale top-K (last successful), never compute a fresh one." "Shed reads to 429 first; never silently drop writes." "Per-tenant isolation: a single tenant's failure cannot starve others."

Per-Failure Detail

Each entry: trigger (what causes it), symptom (what you observe), blast radius (who/what is affected), detection (signal + threshold), recovery (automatic vs operator), design-time prevention, scenarios exhibiting this mode.

Hot-key meltdown

  • Trigger. Power-law key access. Celebrity user, viral post, popular product. Top-1% keys taking > 50% of QPS.
  • Symptom. One node at 100% CPU; all other nodes idle. Reads of the hot key spike in latency. Reads of cold keys on the same node are also slow (head-of-line blocking).
  • Blast radius. All readers of any key on the hot node, until the hot key's load relents.
  • Detection. Per-key request count via CMS sketch (continuous); per-node CPU / per-node QPS > 5× cluster median for > 5 min.
  • Recovery. Auto: hot-key replication kicks in once detection fires (typ. 30–60 s lag). Operator: bump replication factor, isolate the hot key.
  • Prevention. Single-flight on miss; key replication with salt suffix; client-side L1 cache.
  • Cross-link. Steady-state version: bottlenecks#hot-key. Scenarios: 21-distributed-cache (hot-key iteration), 41-home-timeline-fanout.

Hot-partition meltdown

  • Trigger. Partition key correlated with insertion time, with a popular tenant, or with a non-uniform attribute.
  • Symptom. One shard at 100% I/O; others idle. Per-shard write latency rises; replication lag on the hot shard.
  • Blast radius. All tenants/users whose data is on the hot shard.
  • Detection. Per-shard QPS / bytes-in / write-latency > 5× cluster median for > 5 min.
  • Recovery. Auto: range auto-split (CockroachDB, Spanner) or hash re-shuffle; typically minutes. Operator: forced split, manual re-partition.
  • Prevention. Salt the partition key; choose a high-cardinality partition key; avoid date prefixes.
  • Cross-link. bottlenecks#hot-partition; scenarios 02, 18, 44.

Tail-latency cascade

  • Trigger. One downstream component's p99 spikes; the caller waits the full deadline; callers of the caller hit their deadlines too.
  • Symptom. p99 (or even p50) for the front service climbs even though no service is fully down.
  • Blast radius. Every user request whose call graph touches the slow component.
  • Detection. RED metrics per service hop; flame graph showing where the time went.
  • Recovery. Auto: hedged requests reroute to a faster replica. Operator: drain the slow node from the LB pool.
  • Prevention. Per-hop deadlines (deadline propagation in headers); hedged requests at p95; outlier ejection.
  • Cross-link. bottlenecks#tail-latency.

Cache stampede / thundering herd

  • Trigger. Synchronized cache miss for a hot key — TTL expires, all N readers miss in the next millisecond, all N hit the origin.
  • Symptom. Origin QPS for one key spikes from 1/TTL to N × QPSread. Origin overloads; cache stays cold; readers retry; pile-up cascades.
  • Blast radius. All readers of the stampeding key; any other key sharing the origin's capacity.
  • Detection. Origin QPS spike on a single key; cache miss rate spike correlated with TTL boundaries.
  • Recovery. Auto: single-flight allows only one miss through; subsequent misses wait or get stale-while-revalidate. Without it: operator must manually warm the cache or drop traffic.
  • Prevention. Single-flight on miss; jittered TTL (±20%); refresh-ahead before expiry.
  • Cross-link. bottlenecks#stampede; 27-thundering-herd, 21-distributed-cache.

Retry storm

  • Trigger. A failure (timeout, 5xx) triggers retries from N callers; retries themselves overload, causing more failures, causing more retries.
  • Symptom. Load grows during the outage instead of shrinking. Recovery time extends well beyond root-cause fix because the retry queue itself is the load.
  • Blast radius. Whatever the failing component serves; can cascade upstream if retry amplification is high.
  • Detection. Total request rate on the failing component > original rate; retry-after-N-attempts ratio rising.
  • Recovery. Auto: circuit breaker opens after error-rate threshold, fails fast (drains the retry queue), allows recovery, then probes. Without it: operator must shed traffic upstream.
  • Prevention. Exponential backoff with full jitter; per-caller retry budget (max 10% of original calls); idempotency keys for safe server-side dedupe; classify errors as retryable vs not.
  • Cross-link. bottlenecks#retry-storm; 26-cascading-failure.

Cascading timeout / death spiral

  • Trigger. A → B → C → D call chain. D slows down; C waits; C's threadpool fills; B's calls to C time out; B's threadpool fills; A's calls to B time out. Site down.
  • Symptom. Errors at the front-end caller, but root cause is several hops deep. Each layer has resource exhaustion symptoms (thread pools, connection pools).
  • Blast radius. Whole call chain unavailable, even though only the leaf was originally slow.
  • Detection. Distributed tracing shows time spent waiting in each hop; per-service threadpool saturation alarms.
  • Recovery. Auto: circuit breakers at each hop trip and shed load. Bulkheads keep one bad dependency from starving the rest. Operator: manually shed traffic; restart wedged services.
  • Prevention. Per-hop deadlines (deadline propagated in headers, decremented at each hop); circuit breakers per dependency; bulkheads (separate pools per downstream); fail fast.

Poison message

  • Trigger. One malformed event in a stream. Consumer crashes processing it; on restart, picks up the same event and crashes again. Endless loop, no progress.
  • Symptom. Backlog grows behind the poison message. Consumer pool wedged. From outside, looks like backlog growth (but the cause is one stuck record, not throughput).
  • Blast radius. All downstream of this stream until the poison is removed.
  • Detection. Same offset retried > N times; consumer error rate > threshold on a stationary offset.
  • Recovery. Auto: dead-letter queue (DLQ) — after N retries, move the message to a DLQ and continue processing. Operator: inspect the DLQ, fix the data or the consumer, replay.
  • Prevention. Bounded retry per message; DLQ as a first-class queue; idempotent consumers (so retries don't double-process).

Slow dependency / partial degradation

  • Trigger. A downstream service slows from p50 = 10 ms to p50 = 500 ms. Healthcheck still passes (it returns, just slowly).
  • Symptom. Caller threads / connections held longer than designed. Pool exhausts; queue grows; eventual full outage. Looks like the caller is at fault, but root cause is downstream.
  • Blast radius. Caller fleet; can cascade per cascading timeout.
  • Detection. Per-dependency p50/p95/p99 latency monitoring; per-dependency error rate; threadpool / connection pool saturation alarms.
  • Recovery. Auto: circuit breaker on the slow dependency; adaptive concurrency limits reduce in-flight requests; outlier ejection. Operator: drain the slow node; route around.
  • Prevention. Per-call deadlines (not just connection timeouts); adaptive concurrency limits; bulkheads.

Network partition / split-brain

  • Trigger. Network split between zones or regions. Both sides remain operational and may believe they're authoritative.
  • Symptom. Diverged writes on each side; conflicting state on heal. Or, if quorum-based: the minority side stops accepting writes (CP behavior); reads may go stale.
  • Blast radius. Half the cluster (in CP systems: writes blocked on the minority side). In AP systems: divergent writes, conflicts on heal.
  • Detection. Quorum failure alarms; cross-zone heartbeat failures; replication lag monotonically growing.
  • Recovery. Auto: quorum systems automatically continue on majority side; AP systems converge via vector clocks / CRDT / LWW. Operator: investigate root cause; sometimes manual conflict resolution.
  • Prevention. Per-dataset CAP choice (some datasets pick CP — block writes on minority — others AP — accept divergence with merge). Quorum (R+W > N). Fencing tokens to prevent two-leader scenarios.

Leader / lease loss

  • Trigger. Leader process dies; lease expires; election begins.
  • Symptom. Writes paused for the election timeout (typ. 1–10 s for Raft). Reads may continue from followers (stale).
  • Blast radius. All writes to that consensus group, for the duration of the election.
  • Detection. Election counter increments; quorum loss alarm; write-error spike.
  • Recovery. Auto: new leader elected; service resumes. Operator: only if quorum cannot be formed (need to investigate node health).
  • Prevention. Short heartbeat / longer election timeout (tradeoff: liveness vs spurious elections). Pre-elect / hot-standby leader. Lease-based delegation so non-critical work doesn't depend on the consensus path.

Lease-expiry storm

  • Trigger. Coordinator hiccups; many leases expire simultaneously. Mass re-election attempts overload the consensus log.
  • Symptom. Consensus log saturated; election attempts fail; cluster wedged in a re-election loop.
  • Blast radius. Whole consensus group; in cell architecture, one cell.
  • Detection. Election rate; consensus log queue depth.
  • Recovery. Auto: backoff on re-election; staggered lease lifetimes spread out the next storm. Operator: temporarily increase consensus capacity.
  • Prevention. Stagger lease lifetimes from creation (random initial offset); cap re-election rate; prioritize the consensus log over non-critical traffic.

Replica divergence

  • Trigger. Async replication drops or reorders updates. Replicas drift over time.
  • Symptom. Stale or inconsistent reads. May go undetected for a long time.
  • Blast radius. Readers from the lagging replica; correctness violation if writes depend on stale reads.
  • Detection. Replication lag metric; periodic cross-replica checksum (anti-entropy / read-repair).
  • Recovery. Auto: read-repair on conflict; anti-entropy merges divergent state. Operator: rebuild the diverged replica from a snapshot.
  • Prevention. Vector clocks / version numbers per record; quorum reads to detect divergence; periodic background scrubbing.

Clock skew / jump

  • Trigger. NTP sync glitch; VM live-migration with paused clock; leap second; misconfigured node.
  • Symptom. Lease misordering ("expired" leases that aren't actually expired); TTL early expiry; ID collisions; events ordered wrong in logs.
  • Blast radius. Anything depending on wall-clock for ordering: leases, distributed locks, "expired" tokens, log ordering.
  • Detection. NTP offset alarms; cross-node clock-difference monitoring; uniqueness violations on time-based IDs.
  • Recovery. Auto: re-sync NTP. Operator: investigate, possibly quarantine the node, replay affected operations.
  • Prevention. Logical clocks (Lamport, vector); HLC (hybrid logical clock) for systems that mix wall and logical time; never use wall-clock for monotonic ordering. Quarantine nodes whose clock is > threshold off.

Connection / thread pool starvation

  • Trigger. Slow dependency holds resources (connection, thread, semaphore); new requests queue forever.
  • Symptom. Caller appears hung; healthcheck still passes. Eventually queue fills; new requests fail with timeout.
  • Blast radius. Caller fleet; if shared pool (e.g., DB connection pool), all callers of that pool.
  • Detection. Pool utilization > 80% sustained; pool wait time rising; queue depth.
  • Recovery. Auto: circuit breaker opens; bulkhead isolates the slow dependency; adaptive concurrency reduces in-flight. Operator: drain the slow dependency; bounce wedged caller instances.
  • Prevention. Bounded pool with timeout (never unbounded queue); semaphore per dependency (bulkhead); separate pools for hot vs cold paths.

Backlog overflow / OOM

  • Trigger. Producer rate > consumer rate sustained; queue fills RAM (in-memory) or disk (Kafka), eventually crashes the consumer or rejects new writes.
  • Symptom. Latency rises with queue depth; eventually OOM crash or write rejections.
  • Blast radius. All consumers of this stream; can cascade to producers when shedding kicks in.
  • Detection. Queue depth monotonically growing; consumer-lag > threshold.
  • Recovery. Auto: backpressure to producers (HTTP 429 / TCP slow-down); shed by priority. Operator: scale consumer; investigate root cause.
  • Prevention. Bounded queue with explicit shed policy (never silently buffer to OOM). Auto-scale on queue-depth signal, not CPU.
  • Cross-link. bottlenecks#backlog-growth; 13-backpressure.

Resource leak (FD / memory / threads)

  • Trigger. Bug or unclosed resource. Slow degradation over hours/days.
  • Symptom. Single-instance crash on hitting OS limits (FD exhaustion, OOM kill); auto-restart cycles.
  • Blast radius. Single instance initially; can become fleet-wide if leak is in a shared library/runtime.
  • Detection. Resource counters (open FDs, RSS, thread count) trending up between restarts. The trend, not the absolute value, is the diagnostic.
  • Recovery. Auto: instance restart on crash. Operator: investigate, patch, redeploy.
  • Prevention. Bounded resources per instance (FD limit, max threads, heap limit). Periodic recycling / restart for leaky processes ("the cron of last resort"). Observability on resource counters.

Bad deploy / cell-wide failure

  • Trigger. Bug or config error rolled out to all instances simultaneously.
  • Symptom. Global outage if deployed everywhere; one-cell outage if cell-architected.
  • Blast radius. Whatever fraction of users the deploy reached. With cells: ≤ one cell. Without: 100%.
  • Detection. SLO regression on the canary cell; error-rate spike post-deploy; trace anomalies.
  • Recovery. Auto: automated rollback on SLO regression. Operator: manual rollback; investigate; root-cause; redeploy.
  • Prevention. Cell-based architecture — bug deployed to one cell cannot affect others. Canary deploys (1% → 10% → 50% → 100% with bake time at each). Automated rollback triggered by SLO regression. Pre-prod / staging environments matching prod load.

Black-hole node (alive but failing silently)

  • Trigger. Process up; healthcheck endpoint returns 200; but real request handling broken (corrupted state, hung internal thread, partition-isolated).
  • Symptom. Fraction of traffic to the bad host hangs or returns wrong results. Other hosts fine. Hard to diagnose because healthcheck looks healthy.
  • Blast radius. 1/N of traffic (where N = fleet size).
  • Detection. End-to-end synthetic probes (real-traffic-shaped); per-host error rate / p99; outlier ejection on real-traffic SLO violations.
  • Recovery. Auto: outlier ejection removes the host from the LB pool. Operator: bounce / replace the host; investigate root cause.
  • Prevention. Healthchecks that exercise the real path (not just GET /health). Outlier ejection by error rate, not just by healthcheck. Synthetic monitoring.

Silent data corruption

  • Trigger. Bit flip on disk / network; bad disk; bug writing wrong key/value; replication ordering bug.
  • Symptom. Wrong data served indefinitely. Often discovered late (customer complaint, audit failure).
  • Blast radius. Whatever data was corrupted; trust impact extends far beyond the immediate data.
  • Detection. End-to-end checksums (write + read); periodic background scrubbing; cross-replica consistency audits; immutable audit log to compare against.
  • Recovery. Auto: read-repair from a healthy replica; restore from backup. Operator: identify scope of corruption; restore range; communicate to affected users.
  • Prevention. Checksums end-to-end (CRC, hash); ECC memory; replication with verification on read; immutable audit log to reconstruct from. Never trust replicas blindly.

Degradation Contract Patterns

A degradation contract states what the system serves under overload or partial failure, and what it refuses to do. The contract makes the system's behavior predictable to its callers — they can build their own recovery plans on top of a known degradation. Common patterns:

  • Stale-on-failure. "If we cannot compute fresh data, serve the last successful answer with a freshness header." Common for top-K, rankings, dashboards. Trade: bounded staleness for unbounded availability.
  • Shed-reads-first. "Under overload, return 429 to a fraction of reads; never silently drop writes." Reads are usually retryable; lost writes are usually not.
  • Per-tenant isolation. "A noisy tenant cannot starve other tenants." Implemented via per-tenant quotas, per-tenant connection pools, per-tenant rate limits.
  • Sample-on-overload. "If we cannot ingest 100% of events, sample 10% with provenance for stat-correct down-sampling." Common for logging, telemetry, analytics. Trade: precision for availability.
  • Read-only mode. "If write path is degraded, serve reads from cached / replicated state; return 503 on writes." Common during failover, partitions, leader elections.
  • Best-effort partial result. "If we fan out to N and only k < N return in time, return what came back marked partial." Common for fan-in reads (timelines, search, aggregations).
  • Hard-fail-on-correctness. "If we cannot guarantee correctness (e.g., rate-limit decision under partition), refuse the request rather than allow." Use sparingly — only where a wrong answer is much worse than no answer (financial transactions, security decisions).

Pick one (or compose two non-conflicting ones) per design. State it explicitly in S6 Final Design.
