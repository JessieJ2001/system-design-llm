Storage Engines

B-tree vs LSM vs columnar vs heap-append vs FST/immutable vs vector index — what each is optimized for, what breaks each, and how to pick by access pattern. This is the source-of-truth page; scenarios cite this when picking a database.

~12 min reference

How to Use This Page

Picking a database is mostly picking a storage engine. Postgres / MySQL / SQL Server are B-tree engines. Cassandra / DynamoDB / RocksDB-backed systems are LSM. ClickHouse / BigQuery / Snowflake are columnar. Elasticsearch / Lucene-based systems are inverted-index. Pinecone / Milvus are vector-index.

The brand name matters less than the engine. "DynamoDB" isn't an answer — "LSM, because writes dominate and reads are point-by-PK" is. Use this page in two ways:

  • While clarifying requirements (S0 → S1). Once you know the dominant access pattern, walk the decision matrix below and pick the engine. The brand follows.
  • While iterating (S5). If the bottleneck is in the storage layer, this page tells you which engine to consider switching to and what trade you're accepting.

Per-Engine Detail

B-tree

  • Mechanism. A balanced tree of fixed-size pages on disk. Each node holds key→value (or key→child-pointer); lookup traverses logB(N) levels. Updates rewrite the page in place (or out-of-place with WAL).
  • Wins. Random lookup is O(log N) with predictable latency. Range scan is O(K) (sequential leaf traversal). Secondary indexes are themselves B-trees over (indexed-column, primary-key). Strong support for ACID transactions (page-level locking + WAL).
  • Loses. Each write touches at least one page (often more if rebalancing). At high write throughput, the page-write amplification dominates I/O. Large values cause page bloat; pages that exceed a threshold are split.
  • Real implementations. Postgres, MySQL/InnoDB, SQL Server, Oracle. The default OLTP engine.
  • Surfaces. index write-amp when many secondary indexes are added; hot-row contention on heavily-updated rows.

LSM (Log-Structured Merge)

  • Mechanism. Writes go to an in-memory memtable; when full, flush to an immutable on-disk SSTable. Background compaction merges SSTables across "levels" to bound read amp. Reads check memtable then each level's SSTables (with Bloom filters to skip).
  • Wins. Writes are sequential appends — orders of magnitude faster than B-tree random writes. Compression friendly (SSTables compress well per block). Append-only nature pairs well with replication and snapshots.
  • Loses. Read amp: a point read may probe N levels. Compaction debt: if compaction can't keep up with writes, levels grow and reads degrade. Range scan latency varies based on compaction state. Updates and deletes are tombstones — they cost at write time AND on every later read until compaction clears them.
  • Real implementations. Cassandra, ScyllaDB, RocksDB, LevelDB, HBase, BigTable, DynamoDB (LSM-like), CockroachDB (uses RocksDB).
  • Surfaces. disk-IO bound when compaction can't keep up; tail latency on reads during compaction.
  • Bloom-per-SSTable is key to making point reads viable; cite sketches#bloom.

Columnar

  • Mechanism. Stored column-by-column rather than row-by-row. Each column is its own file (or block range), allowing per-column compression (RLE, dictionary, delta, bit-packing) that achieves 10–100× ratios on low-cardinality data.
  • Wins. Aggregations (SUM, AVG, COUNT) read only the relevant columns — orders of magnitude less I/O than row stores. Compression dramatically reduces storage cost and network I/O. Vectorized execution on contiguous column data is CPU-efficient.
  • Loses. Point lookups must seek across many column files to assemble a row. Updates rewrite affected column blocks (often append + tombstone, like LSM). Secondary indexes are unusual; queries scan + filter.
  • Real implementations. ClickHouse, BigQuery, Snowflake, Redshift, Druid, Pinot, Apache Parquet (file format).
  • Surfaces. network-bandwidth bound on huge scans; hot-partition when partitioning by date in OLAP (every new write hits the latest partition).

Heap / Append-Only Log

  • Mechanism. A pure append log, often partitioned by time or by key-hash. Records are written in arrival order. Reads are by offset / position, not by content.
  • Wins. Maximum write throughput (no index maintenance, no rewrite). Natural fit for event-sourcing, audit, replay. Compression-friendly (sequential blocks).
  • Loses. Random read by anything other than append position is hard — usually requires a separate index (which itself may be in another engine).
  • Real implementations. Kafka topic log, Pulsar, AWS Kinesis, Postgres heap files (under the B-tree index), append-only journaling filesystems.
  • Pattern. Often paired with a materializer downstream — the log is the system of record; downstream consumers maintain their own (typically B-tree or LSM) indexes for query.

FST / Immutable Read-Only

  • Mechanism. Finite-state transducer: a deterministic graph that compactly represents many strings (and their associated values). Built once as a batch from a source list; immutable. Lookup is O(prefix length).
  • Wins. Memory efficiency for prefix-rich datasets (autocomplete, dictionary, gazetteer): a 10 GB raw word list might compress to 100 MB FST. Lookup is fast (constant work per character of input). Naturally cacheable in memory across a fleet — same artifact, no per-server state.
  • Loses. Updates require rebuilding the FST from scratch. Acceptable when "fresh enough" is hourly or daily; not viable for real-time write-heavy.
  • Real implementations. Lucene's term dictionary; OpenStreetMap geocoders; some autocomplete services. Pattern: build FST in batch, ship to fleet, swap atomically.
  • Surfaces. Pairs with a cache and edge tier for read-only distribution.

Inverted Index (Lucene-style)

  • Mechanism. Per-term posting list of (doc-id, position, score-stat). Query intersects posting lists for matching terms; ranks by relevance (BM25, TF-IDF) or custom scoring.
  • Wins. Full-text search; multi-field filters (AND, OR, range); relevance ranking; faceting / aggregation alongside search.
  • Loses. Index size often exceeds data size (each indexed field has its own posting list). Updating one field on one doc may invalidate many posting list entries. Index-build cost is significant.
  • Real implementations. Elasticsearch, OpenSearch, Apache Solr (all Lucene-based); Vespa.
  • Surfaces. index write-amp; pattern: use as a secondary store fed asynchronously from a primary OLTP source.

Vector Index (HNSW / IVF)

  • Mechanism. Stores high-dimensional vectors (typ. 128–1536d) and supports approximate nearest-neighbor (ANN) queries. HNSW uses a multi-layer navigable graph (sub-linear search). IVF uses inverted-file lists over coarse quantizer cells (sub-linear via probing).
  • Wins. Sub-linear ANN in high dimensions, where exact nearest-neighbor is impractical (curse of dimensionality). Tunable accuracy-vs-speed (efSearch / nprobe). Compression-friendly via product quantization (PQ).
  • Loses. Approximate by design — recall < 1.0 (typ. 0.95–0.99). Updates are tricky — graphs and inverted lists are designed for batch build; per-update insertions degrade quality without periodic rebuild. Memory-resident for fast queries (vectors don't compress as well as low-card columns).
  • Real implementations. Pinecone, Milvus, Weaviate, Qdrant; FAISS (library); Elasticsearch / OpenSearch with kNN plugin.
  • Surfaces. memory-capacity bound; pairs with PQ to compress.

Pure KV (in-memory)

  • Mechanism. Hash-table-or-skiplist-backed store, often in process or as a remote service. Memory-resident; durability is optional (snapshots, AOF, replication).
  • Wins. Sub-ms point reads. Atomic operations (INCR, CAS, SETNX) for primitives. Native data structures (sorted sets, lists, streams) for specialized cases.
  • Loses. Range scans require ordered structures (sorted set) and don't scale beyond one node's memory. No relational integrity. Persistence is best-effort if used at all.
  • Real implementations. Redis, Memcached, KeyDB, Dragonfly, in-process maps.
  • Surfaces. memory-capacity bound; hot-key on celebrity keys.

The Polyglot Reality

Production systems rarely use one engine. The pattern is system of record + derived stores:

  • System of record (SoR). The authoritative state, usually a B-tree (Postgres) or LSM (Cassandra). Strong consistency; transactional; durable.
  • Derived stores. Read-optimized projections fed asynchronously from the SoR via CDC (change data capture) or event log:
    • Inverted index for search (Elasticsearch).
    • Cache for hot reads (Redis / Memcached).
    • Columnar for analytics (ClickHouse / BigQuery).
    • Vector index for semantic search (Pinecone).
    • FST shipped to edge for autocomplete.

Each derived store is eventually consistent with the SoR (lag = CDC pipeline latency, typ. seconds). When the design needs strong consistency on the derived view, that's a major constraint that re-shapes the architecture (often: serve only from SoR; or use synchronous replication, with the latency / availability trade).

Where to put the SoR

  • Append-friendly + needs-replay → Kafka log as SoR; everything downstream is derived.
  • Transactional + relational → Postgres / MySQL.
  • Write-heavy + key-partitioned → Cassandra / DynamoDB.
  • Per-key strong consistency at scale → Spanner / CockroachDB (Raft per partition).

Anti-Patterns

  • Brand-name first. "Use DynamoDB" before establishing the access pattern. Pick the engine class, then the brand.
  • Inverted index as primary store. Elasticsearch is wonderful as a derived store; as the SoR it has weak consistency, slow ingest under update, and operational pain. Always pair with a real SoR.
  • OLAP queries on OLTP store. "Run analytics on Postgres" — kills OLTP latency at scale. Replicate to columnar.
  • OLTP queries on OLAP store. "Look up one user in BigQuery" — wrong tool; full-table scan to find one row.
  • LSM tombstone trap. Heavy delete workloads on LSM build up tombstones that aren't cleared until compaction; reads slow down. If your workload is delete-heavy, B-tree may be a better fit, or tune compaction aggressively.
  • Vector index for exact-match. ANN gives approximate answers. If you need exact, use a B-tree or filter post-hoc.
  • Treating Redis as a database. Redis is great as a cache / counter / session store; durability is best-effort (AOF can lose 1 s of writes by default). For SoR, use a real database and cache in front.
  • Indexing every column "for completeness." Each B-tree secondary index is a parallel structure that pays cost on every write. Index only what queries require.
