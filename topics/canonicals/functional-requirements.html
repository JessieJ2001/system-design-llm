Functional Requirements

How to seek and structure functional requirements (the "users / clients should be able to…" statements) before drawing any boxes. This is the source-of-truth page; scenario S0 sections cite this for the FR clarification protocol.

~6 min reference

What Functional Requirements Are

Functional requirements describe what the system does — the behavior its callers can rely on. They are the contract the system commits to, expressed in caller-visible terms, not in implementation terms.

FRs are the first thing you discuss in S0 (Scenario & Requirements). Before drawing a single box, before any BOE math, before any architecture, you settle which behaviors the system must support. Everything later — non-functional requirements, bottleneck analysis, design choices — is judged against this list.

FR vs NFR

  • FR: what the system does. "User can post a tweet." / "Cache supports GET, SET, CAS, LEASE_GET." / "Crawler fetches URL, follows redirects, respects robots.txt."
  • NFR: how well it does it. "p95 timeline read ≤ 200 ms." (See NFR checklist.)

Step 1: Identify the System Class

"Design X" interview prompts come in four distinct shapes. Each has a different FR vocabulary. Ask yourself (or the interviewer) which kind of thing am I being asked to design before listing capabilities — the wrong vocabulary produces the wrong S0.

A. Product-shaped (user-facing application)

  • Examples. Twitter, Calendar, Yelp, ride-sharing, e-commerce.
  • Caller. End users via web / mobile / SDK.
  • FR vocabulary. "User can <verb> <noun>" — capabilities at the user level.

B. Infrastructure primitive

  • Examples. Distributed cache, message queue, file system, distributed lock, ID generator, rate limiter, leader-election service, KV store.
  • Caller. Other services (not human end-users).
  • FR vocabulary. "API exposes <operation> with <semantics>" — operations and their guarantees. Often: "GET key returns the most-recently-written value" / "SUBSCRIBE delivers each message at-least-once."
  • Trap. Forgetting to specify semantics (exactly-once vs at-least-once vs at-most-once; linearizable vs sequential vs eventual; serializable vs read-committed). Half the design choices flow from these.

C. Automation / pipeline

  • Examples. Web crawler, ETL job, search-index builder, backup job, log-shipping pipeline, video-transcode pipeline, ML feature pipeline.
  • Caller. No interactive caller; runs on a schedule or trigger.
  • FR vocabulary. "Given <input source>, produce <output> subject to <policy>" — input → output transformation plus the policy that drives it (frequency, completeness target, ordering guarantees, retry behavior).
  • Trap. Skipping the policy (how often? complete or sampled? what if a source fails?). Without it, the pipeline has no stop condition.

D. Algorithmic / batch / ML

  • Examples. Recommender, fraud scorer, ranking model, search-quality system, ad auction.
  • Caller. Mix — a request-time scoring API plus an offline training/refresh pipeline.
  • FR vocabulary. "For <input>, return <output> with <correctness target>; refresh model on <cadence>." Combines product-shaped (the scoring API) with pipeline-shaped (the training loop).
  • Trap. Conflating the online and offline paths. Each has its own FRs and NFRs.

Mixed systems. Many real systems are a mix — Twitter is mostly product but its trending pipeline is automation; Sentry is product but its dedupe core is an infra primitive. When mixed, decompose into sub-systems and walk each through the right vocabulary separately.

Step 2: The Clarification Protocol (Universal)

Eliciting FRs is a back-and-forth with your interviewer (or product manager / customer / consumer team). You ask targeted questions; they confirm, deny, or scope. Treat this as a real conversation, not a checklist.

The two question shapes (work for any system class)

  • "Does the system need to do X?" — clarifies whether a candidate feature is in scope. Answer is yes / no / out-of-scope-but-noted.
  • "What would happen if Y?" — clarifies edge-case behavior of a feature already in scope. Answer reshapes the feature's definition.

Five universal lenses

Walk these in order, regardless of system class. The per-class checklists below tell you what to ask within each lens.

  1. Caller / actor. Who or what calls this system? Humans, services, schedules, triggers. Each caller has its own FR set.
  2. Inputs. What goes in? Format, source, cardinality, mutability, ordering.
  3. Operations / capabilities. What does the system do with those inputs? The 3–6 core operations.
  4. Outputs / effects. What comes out? Format, destination, durability, side-effects on other systems.
  5. Edge cases & out-of-scope. What looks adjacent but isn't this system's job? Failure modes (covered by FRs vs left to NFRs / failure-modes catalog).

Step 3: Per-Class FR Checklists

For each system class, the universal lenses translate to a specific question set. Use the matching checklist; mark items as in-scope, out-of-scope, or N/A.

A. Product-shaped systems

  • Caller. "Mobile, web, or both?" "Authenticated users only or also anon?" "Internal admin operations needed?" "Third-party developers via public API?"
  • Inputs. "What kinds of content can users create — text, image, video, structured?" "Size limits per item?"
  • Operations. "What 3–6 things can a user do?" Phrase each as "<user-type> can <verb> <noun>". Cover create, read, update, delete, list, search where applicable.
  • Outputs / effects. "Does an action notify other users?" "Are there derived feeds / inboxes / aggregates that must update?"
  • Out-of-scope. "Notifications — separate service?" "Search — in-scope or separate?" "Monetization / ads?" "Moderation / abuse-reporting?"

B. Infrastructure primitives

  • Caller. "Which services call this primitive?" "Synchronous or async callers, or both?" "How many distinct callers — 10s or 1000s?" "In-process library, sidecar, or remote service?"
  • Inputs. "What's the key shape — string, numeric, structured?" "Value shape and size — small (≤ 1 KB), medium, blob?" "Are inputs persisted, ephemeral, both?"
  • Operations + semantics. The most important per-class question: not just what operations, but with what guarantees.
    • "Read-after-write semantics — strong, monotonic, eventual?"
    • "Linearizable, sequential, causal, or eventual ordering?"
    • "Exactly-once, at-least-once, at-most-once delivery?"
    • "Per-key serializability or only per-operation atomicity?"
    • "Are conditional operations needed (CAS, If-Match)?"
    • "TTL / expiration semantics — best-effort or precise?"
  • Outputs / effects. "Synchronous responses only, or also pub-sub / change feed?" "Observability / metrics callers can read?" "Admin operations — drain, rebalance, snapshot?"
  • Out-of-scope. "Schema management — caller's job or ours?" "Authentication — assume mTLS upstream?" "Encryption at rest — caller-supplied keys or service-managed?"

C. Automation / pipeline

  • Caller / trigger. "Scheduled (cron) or event-driven?" "On-demand backfill capability needed?" "Manual replay of historical inputs?"
  • Inputs. "What are the input sources — DB, API, files, web, stream?" "Bounded or unbounded input?" "What if a source is slow / unavailable / malformed?"
  • Operations + policy. The most important per-class question: not just what the pipeline computes, but the policy that governs it.
    • "Frequency — every minute, hourly, daily, on-demand?"
    • "Completeness target — process 100% of inputs, or sample / skip on error?"
    • "Freshness target — output reflects inputs up to T seconds old?"
    • "Ordering — does input order matter? Output order?"
    • "Idempotency — must re-running on the same input produce the same output?"
    • "Retry policy — on transient failure, retry forever or bounded?"
    • "Politeness / rate-limiting — when fetching from external sources, throttle?"
    • "Filters — which inputs to skip (URLs in robots.txt; events older than X)?"
  • Outputs / effects. "Where does output land — DB, S3, Kafka, downstream API?" "Atomicity of output — partial output OK on partial failure?" "Notifications / metrics on completion?"
  • Out-of-scope. "Source onboarding — automated or manual?" "Schema evolution of input?" "Output consumption — our problem or downstream's?"

D. Algorithmic / batch / ML

  • Caller. Two callers usually: "Online: who calls the scoring API?" "Offline: who triggers training / refresh — schedule, data-arrival, manual?"
  • Inputs. "Online: what features are available at request time?" "Offline: what training data — labeled, unlabeled, feedback-derived?"
  • Operations + correctness target.
    • "Online: what does the scorer return — single label, ranked list, probability, top-K?"
    • "Correctness target — accuracy, precision/recall, AUC, NDCG, MRR? At what threshold?"
    • "Offline: training cadence — daily, weekly, on-demand on data arrival?"
    • "Cold-start behavior — what does the scorer return for unseen users / items?"
    • "Drift detection — how do we notice the model has gone stale?"
    • "Feedback loop — does scorer output get logged for next round of training?"
  • Outputs / effects. "Scoring response shape; explanation / feature attribution required?" "Training output — model artifact location, metadata, version?" "A/B-testing or shadow-deploy of new models?"
  • Out-of-scope. "Feature engineering — caller's job or ours?" "Label collection — separate system?" "Fairness / bias auditing — separate workstream?"

Step 4: The Output

What you produce

A prioritized list of 3–6 capabilities, each phrased in the right vocabulary for the system class, plus an explicit out-of-scope list. More than 6 means you haven't prioritized; fewer than 3 usually means you've collapsed too much.

Example A: Product (Twitter)

System class: product-shaped.

Functional requirements (prioritized):
  1. User can post a tweet (text, ≤ 280 chars, optional image).
  2. User can follow / unfollow another user.
  3. User can see a ranked timeline of posts from people they follow.
  4. User can see another user's profile and posts.
  5. User can like / reply to a tweet.

Out of scope:
  - DM / private messaging — separate system.
  - Notifications — handled by an upstream service.
  - Ads / monetization — v1 only.

Flagged assumption: reads:writes ≈ 100:1. Would confirm.

Example B: Infra primitive (distributed cache)

System class: infrastructure primitive.
Caller: backend services (synchronous, ~1k caller pods).

Functional requirements (prioritized):
  1. API: GET key, SET key value [ttl], DELETE key, CAS key value version.
  2. LEASE_GET key (returns lease token on miss; for stampede prevention).
  3. Semantics: read-your-writes within a single client; eventual across.
  4. TTL: best-effort expiration; up to ±1 s late OK.
  5. Admin: GET stats per shard; rebalance op (drain + reshuffle).

Out of scope:
  - Schema validation — value is opaque bytes.
  - AuthN — assumed mTLS-stamped caller identity.
  - Persistence — purely in-memory; loss on full-cluster restart OK.

Flagged assumption: working-set fits in ~10 TB across cluster.

Example C: Automation (web crawler)

System class: pipeline.
Trigger: continuous; seed list refreshed daily.

Functional requirements (prioritized):
  1. Given a seed URL list, fetch each URL, follow redirects (max 5),
     extract content + outlinks.
  2. Respect robots.txt per host; skip disallowed URLs.
  3. Politeness: ≤ 1 req/sec/host by default; per-host override allowed.
  4. Output: HTML + extracted text + outlinks → S3 + Kafka topic.
  5. Frequency: re-crawl each known URL at least once per 30 days;
     priority queue for high-rank pages (more frequent).
  6. Idempotency: re-fetching the same URL produces the same output
     (modulo the actual page changing).

Out of scope:
  - JS rendering — static-HTML only in v1.
  - Search index — downstream of crawler, separate system.
  - Login-walled content.

Flagged assumption: corpus ~10 B URLs; ~100 M new URLs/day.

Example D: Algorithmic (recommender)

System class: algorithmic / ML (online API + offline training).

Functional requirements (prioritized):
  Online (scoring API):
    1. Given (user_id, candidate_set ≤ 1000 items, context),
       return ranked top-K (K ≤ 50) with score.
    2. Cold start: for new users / items, return popularity-based fallback.
    3. Explanation: optional feature attributions per recommendation.
  Offline (training):
    4. Re-train daily from past 30 days of (impression, click) events.
    5. Shadow-deploy new model for 24 h before promotion;
       auto-rollback if AUC regresses > 1%.

Out of scope:
  - Candidate generation — upstream system; we re-rank only.
  - Feature engineering pipeline — separate workstream.
  - Fairness / bias audits — separate review process.

Flagged assumption: 100M DAU, 100 scoring requests/user/day.

Prioritization

If forced to cut, which goes first? The prioritization signals which capability drives the architecture. The dominant capability shapes the design; secondary capabilities are addressed where they bite.

Anti-Patterns

  • Wrong vocabulary for the class. Asking "who are the users?" when designing a distributed cache (the "users" are services); asking "what's the API?" when designing a crawler (it's policy + outputs, not an API). Identify the class first.
  • FRs in implementation terms. "The system has a Redis cache." Not an FR — that's a design choice. FRs are caller-visible behavior.
  • Vague capabilities. "User can interact with content." / "Cache supports basic operations." Ambiguous FRs hide design ambiguity.
  • Operations without semantics (infra primitives). "Cache supports SET and GET." What's the consistency model? Is GET-after-SET on a different node guaranteed to see the SET? Without semantics, half the design is undefined.
  • Operations without policy (pipelines). "Crawler fetches URLs." How often? With what completeness? What if a host is down for a day? Without policy, the pipeline has no stop condition.
  • Conflating online and offline (algorithmic systems). The scoring API and the training pipeline have different FRs and different NFRs. Treat them as two sub-systems.
  • Unbounded FR list. 15 capabilities in S0 means the design is unfocused. 6 max; the rest are out-of-scope.
  • No out-of-scope list. Without explicit non-goals, every interviewer pushback ("what about JS rendering?" / "what about persistence?") is a problem you haven't bounded.
  • Drawing boxes before FR alignment. The most common reason a candidate solves the wrong question well — the architecture optimizes for capabilities the interviewer didn't ask for, while missing ones they did.
  • Skipping prioritization. "All five are equally important" tells the interviewer you can't read the room. There's always a primary capability that drives the architecture.
