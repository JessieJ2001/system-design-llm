API Design Patterns

The canonical patterns every C9 API sketch reaches for: pagination, idempotency, concurrency control, error model, versioning, long-running operations, bulk, rate-limit response, REST vs gRPC. This is the source-of-truth page; scenario C9 blocks specify which patterns apply and the scenario-specific shape, not the pattern definitions.

~12 min reference

How to Use This Page

The C9 rubric requires every page (with an external API) to specify endpoints with method, path, request/response shape, SLO, pagination, idempotency, error model, versioning, and concurrency control. Most of those are fixed conventions across systems; the scenario-specific work is choosing which conventions and naming the scenario-specific deviations.

This page is the catalog. Scenario C9 blocks say: "cursor pagination per api-design-patterns#pagination; idempotency-key required on POST per api-design-patterns#idempotency", then add scenario-specific bits.

Pagination

Three options, one default

Cursor (opaque token) — the default

  • Shape. Server returns { items: [...], next_cursor: "opaque-token" }. Client passes ?cursor=<token> on the next page.
  • Token contents. Encrypted/signed encoding of (last-position, query-context). Opaque so server can change scheme without breaking clients.
  • When to use. Default for any list endpoint. Survives writes during pagination; works on any backing store.
  • Why default. Decouples API contract from storage layout. Server can change indexing without breaking clients.

Keyset (sort-key + tiebreaker)

  • Shape. ?after_id=<id>&after_ts=<ts>. Page is WHERE (ts, id) > (after_ts, after_id) ORDER BY ts, id LIMIT N.
  • When to use. When clients want to express "give me items after this specific record" without server-side cursor state. Common in admin / debugging APIs.
  • Trade. Couples API to storage sort order. If you re-shard or change clustering keys, the API breaks.

Offset (LIMIT/OFFSET)

  • Shape. ?page=N&page_size=K.
  • When to use. Almost never for new APIs. O(N) cost on the database for high page numbers; results shift if items are inserted during pagination.
  • Acceptable use. Small bounded result sets where total count is needed (admin lists, search results with "page 5 of 100"). Cap the max offset hard.

Required for any pagination

  • Bounded result size. Server enforces a hard cap (typ. 100–1000 items). A request with no limit still returns the cap, not the entire dataset.
  • Stable sort. Sort key + tiebreaker (always include the primary key as last sort field). Without a tiebreaker, two items with the same sort value can be skipped or duplicated across pages.
  • Cursor expiry. Cursors live for a bounded TTL (typ. 24 h). Beyond that, return 410 Gone — don't pretend infinite cursor lifetime.

Anti-patterns

  • Returning unbounded results — DOS-vulnerable.
  • Offset pagination on a large append-only collection — O(offset) per page.
  • Cursor that exposes storage internals (e.g., Postgres CTID) — locks the API to that storage.
  • Mixing pagination with filtering by changing the cursor encoding — breaks resume.

Idempotency

The contract

Any mutation that may be retried (network failure, timeout, 5xx) must be idempotent — repeated requests must produce the same effect as one. The pattern: client generates a unique idempotency key per logical operation; server dedupes on that key.

Mechanism

  • Header. Idempotency-Key: <client-generated-uuid> on POST/PUT/PATCH/DELETE.
  • Server-side dedupe. Server maintains a table keyed by (tenant_id, idempotency_key) with: original status code, response body, expiry. On repeat: return the original response without re-executing.
  • Dedupe window. Typ. 24 h. Beyond that, the client must generate a new key (or the operation isn't safe to retry that long).
  • Body fingerprint. Server should also hash the request body. If same key + different body, return 422 (idempotency-key reuse with conflicting body) — protects against client bugs.

Naturally idempotent operations

  • PUT by definition (replace the resource at this URL).
  • DELETE by definition (idempotent if you accept that 404 on the second call is fine).
  • GET always.
  • POST and PATCH are not naturally idempotent — must use idempotency keys.

Consumer-side patterns

  • Generate idempotency key before the first attempt; reuse across retries.
  • Persist the key locally so a client crash doesn't lose it (the most common bug: regenerating on retry, defeating the dedupe).

Anti-patterns

  • Server generating the idempotency key — can't dedupe across the network failure that made the client retry.
  • Idempotency by URL path alone (e.g., POST /orders/<client-supplied-id>) — works but exposes internal IDs and conflates idempotency with identity.
  • No expiry on dedupe table — unbounded growth.
  • Trusting POST + ID to be idempotent without server-side dedupe — the client can race itself.

Concurrency Control

Optimistic via ETag / If-Match (default)

  • Read. Server returns the resource with ETag: "<version>" (typ. a hash of the resource state or a version number).
  • Write. Client sends If-Match: "<version>". Server compares; if mismatch, returns 412 Precondition Failed; client re-reads, re-applies, retries.
  • Why default. No locks held across user-think-time; scales to high concurrency on cold rows.

Optimistic via version column (server-side)

  • Storage. Each row has a version column. Updates: UPDATE ... SET ..., version = version + 1 WHERE id = ? AND version = ?.
  • API contract. Either the version is exposed as the ETag, or the API requires the client to pass the previous version explicitly.

Pessimistic locks (rare on REST APIs)

  • Reserved for short-lived backend operations (e.g., SELECT ... FOR UPDATE within a transaction). Don't expose pessimistic locks at the API surface — clients can't be trusted to release.

Create-only via If-None-Match

  • PUT /resource/{id} with If-None-Match: * creates the resource if absent, returns 412 if present. Used for "create if not exists" semantics.

Anti-patterns

  • Last-write-wins without ETag — silently overwrites concurrent edits.
  • Pessimistic lock held over the network — client crashes leave the lock orphaned.
  • ETag based on timestamp alone — two writes within one timestamp tick collide.

Error Model

Status code categories

  • 2xx success. 200 OK, 201 Created, 202 Accepted (async), 204 No Content.
  • 4xx client error — never retryable without changing the request.
    • 400 Bad Request — malformed.
    • 401 Unauthorized — auth missing/invalid; client must re-auth.
    • 403 Forbidden — auth valid but not permitted.
    • 404 Not Found — resource doesn't exist.
    • 409 Conflict — state conflict (concurrent edit, dup create).
    • 410 Gone — was here, intentionally removed.
    • 412 Precondition Failed — ETag mismatch.
    • 422 Unprocessable Entity — well-formed but semantically invalid (validation error).
    • 429 Too Many Requests — rate-limited; retryable after Retry-After.
  • 5xx server error — generally retryable with backoff.
    • 500 Internal Server Error — unexpected; classify as retryable cautiously.
    • 502 Bad Gateway — upstream failure.
    • 503 Service Unavailable — overload / maintenance; retry per Retry-After.
    • 504 Gateway Timeout — upstream timeout.

Body shape (RFC 7807 Problem Details)

{
  "type": "https://api.example.com/errors/rate-limit",
  "title": "Rate limit exceeded",
  "status": 429,
  "detail": "Tenant tnt_123 exceeded 1000 req/min",
  "instance": "/v1/events/abc",
  "retry_after_seconds": 30,
  "trace_id": "0af7651916cd43dd8448eb211c80319c"
}

Retry semantics

  • Retryable: 408, 429, 502, 503, 504 (and 500 with caution).
  • Not retryable: all 4xx except 408, 429.
  • Retry-After header is mandatory on 429 / 503; gives the client a precise time-to-wait. Without it, clients pick arbitrary backoffs.
  • Always pair retry guidance with idempotency keys (otherwise retries can double-execute on the server).

Anti-patterns

  • 200 OK with { "success": false, "error": "..." } body — defeats HTTP semantics; load balancers, monitoring, and client retry logic all break.
  • Using 5xx for client errors — clients retry forever.
  • Using 4xx for server errors — clients give up when they shouldn't.
  • No Retry-After on 429 — clients hammer with their own arbitrary backoff.
  • No trace_id in error body — debugging across services becomes archaeology.

Versioning

Three options

  • URL path: /v1/foo, /v2/foo. Simplest, most explicit. Default for public APIs.
  • Accept header: Accept: application/vnd.example.v2+json. Cleaner URLs; can be cumbersome on debugging. Common for content-negotiation-aware APIs.
  • Header: X-API-Version: 2. Pragmatic middle ground; not standardized.

Compatibility rules within a version

  • Adding a new optional field: backward-compatible, no version bump.
  • Adding a new required field: breaks old clients; bump version.
  • Renaming or removing a field: breaks old clients; bump version.
  • Changing a field's type or semantics: breaks old clients; bump version.
  • Adding a new endpoint: backward-compatible.

Deprecation policy

  • Announce deprecation N months before sunset (typ. 6–12 months for public APIs).
  • Deprecation: true + Sunset: <date> response headers per RFC 8594.
  • Per-tenant usage telemetry on deprecated endpoints; outreach to top users.

Anti-patterns

  • Breaking changes within a version — clients break unannounced; trust eroded permanently.
  • Multiple "current" versions long-term — every change is N× the work.
  • No deprecation telemetry — flying blind on who uses what.

Long-Running Operations (LROs)

The pattern

Operations that take longer than a request can reasonably wait (typ. > 1 s synchronous; mandatory for > 30 s) return a job handle and are polled or webhooked.

Three flavors

  • 202 Accepted + polling. POST returns 202 with Location: /v1/jobs/{job_id}. Client polls GET /v1/jobs/{job_id} until status = completed | failed.
  • Webhook on completion. Client registers a callback URL on submit; server POSTs to it on completion. Better for long jobs (hours/days). Required: HMAC signature on the callback for authenticity.
  • Server-Sent Events / WebSocket subscription. Client subscribes to job-status updates. Best UX for short-medium jobs the user is actively watching.

Job resource shape

GET /v1/jobs/{job_id} →
{
  "job_id": "job_abc",
  "state": "running" | "queued" | "completed" | "failed" | "cancelled",
  "progress": 0.42,                    // optional, [0, 1]
  "started_at": "2026-05-04T12:00:00Z",
  "completed_at": null,
  "result_url": null,                  // populated on success
  "error": null                        // populated on failure
}

Cancellation

  • POST /v1/jobs/{job_id}:cancel; idempotent.
  • Server best-effort; if the job has already started a non-cancellable phase, returns 409.

Anti-patterns

  • Synchronous request held open for > 30 s — load balancers cut the connection; clients retry; the job runs N times.
  • Polling without bounded interval — clients hammer the status endpoint.
  • Webhook without signature verification — anyone can spoof completion.

Bulk Operations & Partial Success

When to expose bulk endpoints

  • When N items per logical operation regularly > 10 and per-item RTT is the bottleneck.
  • When clients need atomic-or-nothing semantics across N items (rare).

Two modes

  • All-or-nothing (transactional). Either all items succeed or none. Server response is 200 or 4xx. Use sparingly — increases server-side complexity dramatically.
  • Best-effort with per-item status. Server returns 207 Multi-Status with per-item results: 
    {
  "results": [
    { "item_index": 0, "status": 201, "id": "item_a" },
    { "item_index": 1, "status": 422, "error": "validation failed" },
    { "item_index": 2, "status": 201, "id": "item_b" }
  ]
}

Required

  • Bound on N. Hard cap (typ. 100–1000 items) on a single request. Otherwise DOS-vulnerable.
  • Idempotency-key on the bulk operation + per-item idempotency where individual items might be retried.

Anti-patterns

  • Bulk endpoint that returns 200 even if all items failed.
  • Bulk endpoint with no cap — single request can consume server-side memory unboundedly.

Rate-Limit Response Semantics

The headers

  • X-RateLimit-Limit: the limit (per minute / per hour / per day, as documented).
  • X-RateLimit-Remaining: how many requests left in the current window.
  • X-RateLimit-Reset: when the window resets (Unix timestamp or seconds-from-now).
  • Retry-After: seconds until the next request will succeed (mandatory on 429).

Standardization

RFC 9239 defines RateLimit and RateLimit-Policy structured headers. Adoption is partial; most APIs still use the X- prefixed forms above for compatibility.

The 429 body

Per the error model: include retry_after_seconds in the body too (some clients can't read headers).

Anti-patterns

  • 429 without Retry-After — clients pick arbitrary backoffs and hammer.
  • 503 (service unavailable) instead of 429 — semantically wrong; clients can't distinguish "service is down" from "you specifically are over quota."

REST vs gRPC vs GraphQL

REST (HTTP/JSON)

  • Default for public APIs. Maximum interoperability; debug with curl; cacheable at HTTP layer; works with every client and proxy.
  • Latency overhead ≈ 1–5 ms per call from JSON parsing + HTTP overhead. Tolerable for human-scale operations; problematic at high RPS.

gRPC

  • Default for service-to-service. Protobuf is 5–10× more compact than JSON, with stronger schemas; HTTP/2 multiplexing reduces connection overhead; bidirectional streaming for free.
  • Tradeoff: harder to debug (binary on wire), browser support requires gRPC-Web proxy, no HTTP cache.

GraphQL

  • Default for client-driven UIs where the client knows best which fields it needs (mobile apps with bandwidth concerns; web with deep nested data).
  • Tradeoffs: server-side query complexity (depth + breadth limits required to prevent DOS); cache layer must be GraphQL-aware (Apollo, Relay).

How to choose

  • Public-facing user-installable client (third-party developers): REST.
  • Internal microservice mesh: gRPC.
  • Mobile/web UI with diverse query shapes: GraphQL as a BFF layer in front of REST/gRPC.
  • Real-time bidirectional: WebSocket or gRPC streaming (see cross-cutting#real-time-push).
