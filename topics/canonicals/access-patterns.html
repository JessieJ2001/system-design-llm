Access Patterns

The canonical taxonomy of read/write/skew shapes a distributed system can face — and the first techniques you reach for under each. This is the source-of-truth page; scenario pages link here instead of redefining patterns.

~10 min reference

How to Use This Page

Every staff+ design probe eventually becomes some variant of "you said X — what if reads are 100× writes?" or "what if 1% of keys take 50% of traffic?" The 12 patterns below are the canonical answers to that question. They are the dimensions any interviewer is implicitly comparing against when they ask about scale.

Use this page in two ways:

  • While clarifying requirements (S0). Walk down the table. For each pattern, ask: "does this apply to my system?" If yes, that pattern's first/second levers are pre-validated starting points. If no, mark it explicitly as out-of-scope.
  • While iterating (S5). When a bottleneck appears, classify it as one of the 12 patterns; that classification tells you which lever to reach for next, and what number tells you the lever has run out.

Other pages on this site link here for definitions of these patterns. Do not redefine them in scenario pages — link instead.

Per-Pattern Signature

For each pattern: the signal that tells you you're in it, the first lever (what you reach for), the cliff (the number that says the lever has run out), and the last resort.

Read-heavy uniform

  • Signal: R:W ratio > 10:1; no measurable skew across keys.
  • First lever: Async read replicas; route reads to replica pool.
  • Cliff: Replica replication lag > read freshness SLO, or replica fan-out > primary's outbound bandwidth.
  • Last resort: Shard the dataset; or move to a different read-optimized engine (e.g., search index, OLAP store).

Write-heavy uniform

  • Signal: R:W < 1:1; sustained ingest dominates the workload; uniform key distribution.
  • First lever: Shard the write path by hash(key); batch + async commit; consider an LSM engine if currently on a B-tree.
  • Cliff: Cross-shard write fan-out blowing the network; per-shard fsync queue depth; compaction debt unbounded.
  • Last resort: Append-only log + downstream materializers (Kafka + consumers); abandon point-write semantics.

Power-law / hot keys

  • Signal: Top-1% keys account for > 50% of QPS (Zipf's law). Hot-key detection (CMS sketch) shows 1–10 keys at 100× the median.
  • First lever: Client-side L1 cache with short TTL; request coalescing (single-flight) on the read path.
  • Cliff: One node pegged at 100% CPU regardless of cluster size (single-thread bottleneck on the hot key); coalescing window saturates.
  • Last resort: Replicate hot keys with key suffixing (`key#salt`); on writes, fan-out to all replicas; on reads, route to one of N replicas.

Hot partition / hot shard

  • Signal: Per-shard QPS or storage skewed > 5× the median for > 5 minutes sustained.
  • First lever: Salt the partition key with a short hash prefix; or subdivide the hot range with auto-split.
  • Cliff: Splits can't keep up with growth; cross-shard query cost dominates.
  • Last resort: Move the hot range to dedicated nodes; carve into its own keyspace.

Bursty / spiky

  • Signal: Peak-to-average ratio > 5×; bursts shorter than the autoscaling response time.
  • First lever: Smooth with a queue (write-behind via Kafka); auto-scale the read tier on a leading metric.
  • Cliff: Queue depth grows faster than drainage; cold-start latency > SLO.
  • Last resort: Provision for peak (warm capacity); pre-warm via shadow traffic.

Scan-heavy / range

  • Signal: Few queries return many rows; OLAP-shaped workload (aggregations, time-range scans).
  • First lever: Range partitioning with clustering keys aligned to scan order; columnar storage.
  • Cliff: Per-query scanned bytes > per-query memory budget; concurrent scans starve point reads.
  • Last resort: Push scans to a separate replica fleet; pre-compute materialized aggregates.

Fan-out write (one → N reads)

  • Signal: One write triggers N downstream materializations; N varies wildly (celebrity follower count).
  • First lever: Fan-out-on-write to per-recipient inboxes (push pattern).
  • Cliff: Top-percentile fan-out cost dominates total write cost; celebrity write pegs the worker pool.
  • Last resort: Hybrid — fan-out-on-write for normals, fan-out-on-read with cache for celebrities.

Fan-in read (N → one)

  • Signal: Read assembles result from many sources (timeline, dashboard, leaderboard).
  • First lever: Pre-aggregate / rollup tables maintained on write.
  • Cliff: Roll-up freshness lag > SLO; combinatorial explosion of pre-aggregations.
  • Last resort: Stream processor (Flink) maintaining materialized views; query the view, not the source.

Long-tail / cold

  • Signal: > 80% of keys account for < 5% of QPS; storage growth dominates compute cost.
  • First lever: Tiered storage (RAM → NVMe → S3); bloom filter to skip cold lookups.
  • Cliff: Cold-tier read latency > user-facing SLO; bloom false-positive rate too high.
  • Last resort: Compression-friendly columnar format; aggressive TTL; archive to immutable storage.

Large items

  • Signal: p99 item size > 1 MB; per-row read amplification > 10× the requested bytes.
  • First lever: Store blob in object store (S3 / GCS); KV holds the pointer + metadata only.
  • Cliff: Latency for the indirect lookup blows the SLO; metadata table itself becomes large.
  • Last resort: Don't put it in this database. Switch to a blob-native store (S3 + presigned URLs).

High-cardinality query

  • Signal: Filter by > 5 distinct rare attributes; queries are not by primary key.
  • First lever: Inverted index (Lucene/Elasticsearch); secondary indexes for the most common predicates.
  • Cliff: Index size > data size; index update lag > freshness SLO.
  • Last resort: Move to a dedicated search engine; treat the OLTP store as source-of-truth, the search engine as derived.

Strict ordering / monotonic

  • Signal: Per-entity events must be totally ordered (account ledger, per-user state machine).
  • First lever: Single partition per ordering key (per-user, per-account); sequential write within partition.
  • Cliff: One ordering key's QPS exceeds single-partition throughput; cross-partition transactions needed.
  • Last resort: Sequence number / fencing token assigned by a coordinator; or move to a strongly-consistent log (Kafka with single-partition topic, or a Raft-replicated log).

Combinations & Precedence

Real systems exhibit multiple patterns at once. When they conflict, the order in which they manifest dictates the design path:

  • Power-law + Read-heavy uniform. Replicas help only the uniform tail; the hot 1% needs key replication on top. Don't conflate them — they are two separate fixes.
  • Bursty + Write-heavy. The queue absorbs the burst and reshapes write distribution; one design move (Kafka in front) addresses both. Show the math for each.
  • Hot partition + Strict ordering. Direct conflict — you cannot salt the partition key without giving up per-key ordering. Resolve by introducing a sequence number or a per-key partition with a coordinator (sacrifices throughput for order).
  • Long-tail + Large items. The combination is dangerous: a cold large item is the worst case for both latency and cost. Indirect via blob store + bloom-filter lookup is the canonical answer.

When you list multiple patterns in a scenario's S0, name which one dominates for the system's primary path; the rest become secondary considerations addressed where they bite.
