Probabilistic Sketches

The canonical bounded-memory data structures for streaming aggregates: set membership, cardinality, frequency, top-K, quantiles, sampling, similarity. This is the source-of-truth page; scenarios cite a sketch instead of redefining it.

~15 min reference

How to Use This Page

Sketches are the canonical answer to cardinality explosion. When per-key state × N grows unboundedly with input, the exact data structure (hashmap, heap, sorted set) is replaced with a sketch that bounds memory at O(1/ε) instead of O(N), at the cost of bounded relative error.

Diagnose the question first; then pick the sketch:

Mergeability matters because real systems shard ingest — many partial sketches get reduced into one. A non-mergeable sketch forces all data to one node, defeating the bounded-memory goal.

Bloom Filter

What it does

Set membership: "have I seen this key before?" Returns either "definitely not" or "probably yes" (with a tunable false-positive rate). Never false-negatives.

Mechanism

An array of m bits, all zero initially. k independent hash functions per element. Insert: hash the element with each of the k functions, set those k bits. Query: hash with the same k functions; if all bits are 1, return "probably yes"; if any is 0, return "definitely not."

Sizing

  • For n expected elements with target false-positive rate ε: m = -n × ln(ε) / (ln 2)² bits, k = (m/n) × ln 2 hashes.
  • Quick rule: ~10 bits per element for 1% FP; ~14 bits for 0.1%; ~20 bits for 0.001%.

Memory

  • For 1B elements at 1% FP: ~1.25 GB.
  • For 1M elements at 1% FP: ~1.25 MB.
  • Memory is independent of element size — each element costs m/n bits regardless of key length.

Mergeability

Two Bloom filters of the same size and same hashes can be merged by bitwise OR. The merged filter answers membership for the union of the two underlying sets.

Anti-uses

  • When you need exact answers — Bloom always has FP > 0.
  • When you need deletion — standard Bloom doesn't support delete (use Counting Bloom or Cuckoo).
  • When the cost of FP > cost of doing the actual lookup — Bloom is only useful when the underlying lookup is expensive (disk, network, slow origin).

Canonical use

Skip-disk for "probably-not-here" cases. LSM trees use Bloom per SSTable to skip files that don't contain the key. Backup dedup uses Bloom on the chunk index to skip lookup of new chunks.

Cuckoo Filter

What it does

Bloom filter + delete. Same "definitely-not / probably-yes" semantics; supports deletion at a small memory cost over Bloom.

Mechanism

Hash table where each slot holds a small fingerprint (typ. 8 bits) of the key. Two candidate buckets per key (cuckoo hashing); if both full, evict and re-insert the displaced fingerprint.

Sizing

  • For 1% FP: ~12 bits/element. (Bloom: ~10 bits/element.)
  • Slightly worse memory, but supports delete and is more cache-friendly on lookup (only 2 hash probes).

Mergeability

Not mergeable in general — two filters can't be merged into one without re-inserting all elements.

When to use

  • Set membership with deletion (Bloom can't).
  • When lookup latency matters (Cuckoo is faster than Bloom for high k Bloom).

HyperLogLog (HLL)

What it does

Estimate the cardinality of a multiset — how many distinct elements have been seen — in nearly constant memory regardless of the cardinality.

Mechanism

Hash each element to a 64-bit value. Use the first p bits to select one of 2^p "registers"; use the position of the first 1-bit in the rest as the register's value (keep the max). The estimate of cardinality is a harmonic-mean-derived function of the registers.

Sizing

  • 2^p registers, each ~6 bits. With p = 14: 16,384 registers × 6 bits = ~12 KB total, error ≈ 1.04 / sqrt(16384) ≈ 0.8% relative.
  • Estimates 1 to 10⁹ distinct elements with the same 12 KB.
  • Redis HLL: ~12 KB per HLL, ~0.81% error.

Mergeability

Yes — to merge two HLLs of the same size, take the per-register max. The merged HLL estimates the cardinality of the union.

Anti-uses

  • When you need the actual elements (HLL only counts; doesn't store).
  • When you need exact counts (HLL has bounded relative error, typ. 0.5–2%).
  • When the cardinality is small (< ~1000) — exact set is cheaper and accurate.

Canonical use

Per-page unique visitors, per-tenant unique IPs in a window, distinct query terms in a search index, dedup ratio for a backup chunk store.

Count-Min Sketch (CMS)

What it does

Estimate the frequency of any key seen in a stream. Memory bounded; error is one-sided (always over-estimates, never under).

Mechanism

A 2D matrix of d rows × w columns of counters. d independent hash functions, one per row. Insert: increment matrix[i][hash_i(key)] for each row i. Query: return min(matrix[i][hash_i(key)]) over all rows — the smallest counter is the closest to the true count (collisions can only inflate, not deflate).

Sizing

  • w = ⌈e/ε⌉; d = ⌈ln(1/δ)⌉. Memory = w × d × counter-size.
  • For ε = 0.01 (1% relative error vs total count) and δ = 0.001 (99.9% confidence): w ≈ 272, d ≈ 7; with 4-byte counters, ~7.6 KB total.
  • Error bound: estimate ≤ true_count + ε × N, where N is the total stream count, with probability ≥ 1 - δ.

Mergeability

Yes — element-wise sum of two same-shape CMSs gives the CMS for the concatenated stream.

Anti-uses

  • Estimating low-frequency items — error is proportional to total stream size, so rare items get drowned.
  • When you need under-counting (CMS only over-counts).
  • When the stream has heavy hitters and you want the heavy hitters specifically — use Space-Saving instead.

Canonical use

Per-key request-frequency tracking for hot-key detection, rate-limiting under cardinality explosion, top-K candidate filtering before exact verification.

Space-Saving (Top-K)

What it does

Maintain the top-K most frequent items in a stream, with a deterministic guarantee: "if an item's true frequency exceeds N/K, it will appear in the top-K." (No probability bound; this is a deterministic guarantee on the heavy hitters.)

Mechanism

Maintain a hashmap of size K (capacity). On a new element: if it's already in the map, increment. If not in the map and the map isn't full, add with count 1. If full, find the item with the minimum count, replace its key with the new key, and increment its count. The replacement count over-estimates the new key's true count by at most the minimum (which is bounded by N/K).

Sizing

  • K + small overhead. For top-100: a 100-entry hashmap.
  • Error: every reported count is an over-estimate by at most N/K (where N = total stream length).

Mergeability

Yes — merge by combining the two hashmaps and re-applying the eviction rule. Useful for sharded ingest with central reduction.

Anti-uses

  • When you need exact counts for the top-K — over-estimates by up to N/K.
  • When the distribution is uniform — Space-Saving works best for skewed (Zipfian) distributions; on uniform streams, no item exceeds N/K and the result is meaningless.

Canonical use

Top-K trending URLs, top-K busiest endpoints, top-K queries in autocomplete log analysis.

HeavyKeeper

What it does

Top-K with adversary-resistance. Newer than Space-Saving; better behavior under adversarial input (where an attacker tries to flood with fake hot items).

Mechanism

CMS-like structure with a probabilistic decay rule: hot items resist eviction proportionally to how hot they already are. Combined with a small min-heap to track the top-K candidates.

When to use vs Space-Saving

  • Space-Saving: simpler, well-understood, deterministic guarantee on heavy hitters. Default choice.
  • HeavyKeeper: when input may be adversarial (anti-cheat for leaderboards, abuse-resistant trending), or when accuracy on near-threshold items matters.

t-digest

What it does

Estimate any quantile (median, p95, p99, p999) of a stream of values. Bounded memory; relative error best at the tails (which is what you want for SLOs).

Mechanism

Adaptive histogram of weighted centroids. Centroids are denser in the tails (small bins near the tails, large bins near the median) so quantile error is uniformly small in relative rank, even at p999.

Sizing

  • Compression parameter δ (typ. 100–1000). Memory: ~10 KB at δ = 100.
  • Error: ≤ 1% in relative rank near tails; ≤ 0.5% near median.

Mergeability

Yes — concatenate centroids and re-compress. Order matters slightly (insertion order can affect centroid shape) but error bound is preserved.

Anti-uses

  • When relative error in value matters more than in rank — use DDSketch.
  • When the value distribution has very long tails (latencies in microseconds to seconds) and you want a strict relative-error guarantee on the tail — use DDSketch.

Canonical use

Per-(service, endpoint) latency p99 for SLOs; per-leaderboard rank histogram; per-tenant resource usage distribution.

DDSketch

What it does

Quantile estimation with a strict relative-error guarantee on the value (not just the rank). The key property: for any quantile q, the estimated value is within (1 ± α) of the true value, with α a fixed parameter (typ. 0.01).

Mechanism

Logarithmic bucketing: bucket index = ⌊log_γ(value)⌋ for some γ > 1. Bucket boundaries grow geometrically, so the relative width of each bucket is constant — that's where the relative-error guarantee comes from.

Sizing

  • For 1% relative error: ~1500 buckets; ~6 KB.
  • Memory bounded if value range is bounded; can grow if values span many orders of magnitude.

Mergeability

Yes — element-wise sum of bucket counts. Critical for distributed ingest.

When to use vs t-digest

  • t-digest: best when you want uniform rank error (the difference between p99 and p999 is what you care about).
  • DDSketch: best when you want uniform value error (the latency value at p99 must be within X% of the truth).

Canonical use

SRE observability — Datadog uses DDSketch for latency quantiles. The relative-error-on-value guarantee is exactly what you want for "p99 = 142 ms ± 1%."

Reservoir Sampling

What it does

Maintain a uniform random sample of size K from a stream of unknown length, in O(K) memory and O(1) per-element time.

Mechanism (Algorithm R)

Keep first K elements as the initial reservoir. For element i > K: with probability K/i, replace a random reservoir element with this one. Result: every element seen has equal probability of being in the final reservoir.

Mergeability

Yes, with weights. Two reservoirs of size K from streams of lengths n₁ and n₂ can be merged into a reservoir of size K from the concatenated stream by weighted random selection.

Anti-uses

  • When you need a deterministic sample (use head sampling: take every Nth).
  • When you need the most recent N — use a sliding window, not a reservoir.
  • When you need stratified sampling — basic reservoir is unstratified.

Canonical use

Sample logs / events / queries for analysis when full ingest is too expensive. Sample errors per fingerprint to keep a "representative example" without keeping all events.

MinHash

What it does

Estimate the Jaccard similarity (|A ∩ B| / |A ∪ B|) of two sets in O(K) memory per set, where K is the signature size.

Mechanism

For a set, compute K independent hashes of every element; for each hash function, keep the minimum hash value. The signature of the set is the K minima. Jaccard similarity ≈ fraction of signature positions where two sets agree.

Sizing

  • K = 128 to 256 typically. Each hash is 4–8 bytes; signature is 1–2 KB per set.
  • Standard error: ≈ 1/sqrt(K). For K = 256, error ≈ 6%.

Canonical use

Document deduplication (near-duplicate detection); recommendation similarity; locality-sensitive hashing (LSH) for nearest-neighbor.

Selection Decision Tree

What do you need to know?

├── Is X in the set?
│   ├── No deletes needed         → Bloom filter
│   └── Need to delete            → Cuckoo filter
│
├── How many distinct things?    → HyperLogLog
│
├── How frequently has X been seen?
│   ├── Any X (point query)      → Count-Min Sketch
│   ├── Top-K most frequent      → Space-Saving (default)
│   └── Top-K under adversary    → HeavyKeeper
│
├── What's the p99 / quantile?
│   ├── Want uniform rank error  → t-digest
│   └── Want uniform value error → DDSketch
│
├── A representative sample      → Reservoir sampling
│
└── How similar are these sets?  → MinHash

Mergeability and Distributed Ingest

Mergeability (associative + commutative reduction) is what makes a sketch scalable. Without it, your only option is to send all data to one node — defeating the bounded-memory goal at the system level.

The standard distributed-sketch architecture:

  1. Per-shard ingest. Each ingest worker maintains its own sketch over the events it sees.
  2. Periodic emit. Every T seconds, emit the current sketch state to a reducer / aggregator.
  3. Merge. Reducer merges the per-shard sketches into one global sketch.
  4. Query. Queries hit the merged sketch.

This pattern works for: HLL (cardinality), CMS (frequency), Space-Saving (top-K), t-digest / DDSketch (quantiles), Bloom (set membership). It does not work for Cuckoo (deletes break merge) or any sketch with non-associative state.

The trade is staleness — query results are as fresh as the last merge. For interactive systems, a hybrid pattern: query the latest and the most recent partial sketches in parallel, merge online.

Sketch-Wide Anti-Patterns

  • Using a sketch when exact is cheap. If N is small and bounded (< 10⁵), the exact data structure is simpler, faster, and accurate. Don't reach for a sketch out of habit.
  • Picking a sketch by name without checking error model. CMS over-estimates; Space-Saving over-estimates the count (under-reports the rank); t-digest is rank-error-bounded but not value-error-bounded. Pick by what kind of error your application can tolerate.
  • Storing per-(tenant × time-bucket) sketches without TTL. Cardinality of (tenant × time-bucket) is itself unbounded over time. Add explicit TTL.
  • Mixing sketches of different sizes / parameters. A 12 KB HLL and a 64 KB HLL can't be merged. Pin sketch size cluster-wide; document the migration path when you change it.
  • Treating the sketch result as exact downstream. A "top-K" returned from Space-Saving has count over-estimates that propagate into any further analysis. Either verify exactly for the heavy hitters that matter, or carry the error bound forward in the API.
