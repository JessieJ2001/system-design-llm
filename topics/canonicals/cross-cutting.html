Cross-Cutting Concerns (X1–X8)

Eight concerns that cut across every staff+ system design — security, privacy, DR, capacity, subsetting, cells, real-time push, edge. This is the source-of-truth page; scenario pages cite the relevant subsection rather than restating the canonical patterns.

X1 — Security: Auth, mTLS, Secrets, Multi-Tenancy

Authentication: where and how

  • End-user auth at the public surface. JWT or OIDC token validated at the API gateway / edge. The token rides in Authorization: Bearer; downstream services trust the gateway-stamped identity, not the raw token. Latency budget: 1–2 ms per validation including signature check + revocation lookup.
  • Service-to-service auth: mTLS. Every internal call is mutual-TLS. Certs rotated every 24h via SPIFFE/SPIRE or equivalent; no service has long-lived credentials. The mesh sidecar (Envoy / Linkerd) handles the handshake; application code sees plain HTTP.
  • Token freshness vs revocation. JWT is opaque-on-wire but stale-on-revoke. Mitigations: short TTL (5–15 min); refresh-token flow; introspection endpoint hit on sensitive operations; revocation list cached at the gateway with bounded staleness (e.g., 60 s).

Authorization model

  • RBAC — role-based, simplest, works for most tenant systems. "Admin can do X, Member can do Y."
  • ABAC — attribute-based, scales to per-resource decisions. "User can read this doc if its owner = user_id OR shared_with contains user_id."
  • ReBAC — relationship-based (Zanzibar-style), scales to social-graph permissions. "User can read this doc if user is in a group that has access."
  • Where the policy decision point (PDP) lives. In-process for low-latency / high-volume paths (~ μs); sidecar for medium-volume; central PDP for rare, high-stakes decisions. Caching policy at the PDP is critical — uncached can add 5–10 ms per call.

Secret rotation

  • KMS-backed per-tenant data encryption keys (DEKs); a master key (KEK) rotates monthly; per-tenant DEKs rotate quarterly.
  • Application reads keys via short-lived IAM credentials (instance profile / workload identity); never embeds long-lived secrets in config.
  • Rotation must be zero-downtime: dual-key window where new and old are both accepted; cutover after fleet has refreshed.

Multi-tenancy isolation

  • Per-tenant quotas on rate limits, connections, storage, query cost. Without these, one noisy tenant can starve all others (a pool-starvation failure).
  • Per-tenant data isolation via partition key prefix or per-tenant database/keyspace. Enforce at the API gateway (set tenant_id from the validated token; all queries scoped to it).
  • Noisy-neighbor budget — an explicit number ("a single tenant cannot consume > 5% of cluster capacity at any moment") that drives the quota.

What every page must state

Minimum: "Auth handled at <layer>, latency budget X ms, failure mode = <fail-closed | fail-open with audit>."

Anti-patterns

  • App-level auth without a gateway — every service re-implements token validation, drift between them, security regression on the next release.
  • Long-lived service credentials baked into config — leak window is forever.
  • Authz "we'll add it later" — adding fine-grained permissions to a system in production requires a data backfill that's typically the worst migration of the year.
  • Wall-clock token expiry without revocation — a compromised token is valid until expiry, regardless of when you noticed.

X2 — Privacy: GDPR Erasure, Residency, Audit, Retention

GDPR right-to-erasure

  • Soft-delete with tombstone is the canonical pattern. A user-deletion request stamps tombstones across all stores keyed by user_id; periodic compaction reclaims the data. SLA: ≤ 30 days from request to physical deletion.
  • Storage layout must support per-user deletion. If user_id is a partition key, deletion is O(1). If user data is intermixed with others' (no user_id in partition key), deletion requires a full scan — multiple orders of magnitude more expensive.
  • Backup handling. Backups have their own retention; either include backups in deletion, or document that a deleted user's data may persist in backups until backup TTL (typ. 30–90 days). Be explicit which.
  • Derived data. Aggregates / sketches / embeddings derived from user data must also delete or invalidate. Per-user-attributable derived data is GDPR-relevant; aggregate-only data (e.g., "10M users clicked X") usually isn't.

Data residency

  • EU-user data stays in EU storage. Per-region partitioning enforced at the data layer; cross-region access goes through a federation tier that respects residency.
  • Cross-region read is allowed for specific use cases (the user travels) with explicit logging.
  • Partition key must include region or be derivable from it. Otherwise residency cannot be guaranteed structurally.

Audit log

  • Append-only, separate from the operational store, of every mutation: who, when, what changed, from-value, to-value (or pointer to it).
  • Required for SOX, HIPAA, SOC2, GDPR (data-access logging).
  • Storage: WORM (write-once read-many) like S3 Object Lock; or an append-only Kafka topic + S3 archive.
  • Retention: typ. 7 years for financial systems; 6 years HIPAA; per-regulation.

PII classification & redaction

  • Per-field tagging at schema time — every column annotated as pii: yes/no, encrypted: yes/no, retention_days.
  • Redaction at egress — log lines, exports, debug dumps strip PII fields based on the schema annotation. Best done at the structured-logging layer (single point of enforcement).
  • Encryption at rest with per-tenant keys for the highest-sensitivity tier (medical records, payment data, government IDs).

Retention policy per data class

Hot / warm / cold / delete, automated. Each data class has a TTL; storage tier transitions are scheduled (S3 lifecycle rules, ClickHouse TTL clauses, partition drops). Manual deletion is an anti-pattern — it doesn't scale and leaves orphaned data.

What every page with persistent user data must state

A one-line erasure plan in the schema sketch: "Delete-by-user_id traverses partition X in O(K) writes; tombstone TTL = 30 days; aggregate sketches re-derived weekly so deletion propagates within 7 days."

Anti-patterns

  • Logging PII at ingest — once it's in logs, deletion costs become unbounded.
  • Per-user data spread across stores keyed by anything but user_id — deletion becomes a multi-system audit.
  • Backups without an erasure story — a 7-year backup retention with GDPR users in it is a compliance time-bomb.

X3 — Disaster Recovery: RPO, RTO, Drills

RPO and RTO as numeric SLOs

  • RPO (Recovery Point Objective): how much data can you afford to lose, in seconds/minutes? Defined by the business, not by engineering. Financial transactions: RPO = 0. Telemetry/logs: RPO = 5 min often acceptable. Cache state: RPO = ∞ (rebuild from source).
  • RTO (Recovery Time Objective): how long can you be down? Measured in seconds (active-active) to hours (cold backup restore).
  • RPO/RTO ladder. Different datasets in the same system have different RPO/RTO. Don't pick a single number for the whole system; pick per-dataset and design accordingly.

Backup of metadata stores

  • Operational-DB backup cadence: typ. continuous WAL streaming + nightly full snapshot.
  • Off-region copy: encrypted snapshot in a different region from the primary.
  • Restoration test cadence. Untested backups are not backups. Test monthly minimum.

Cross-region failover plan

  • Active-active — both regions serve traffic simultaneously; failover is a DNS / Anycast cutover (10s–60s). RPO/RTO low; cost ≈ 2× single-region.
  • Active-passive (warm standby) — standby region has data replicated but doesn't serve; failover requires standby promotion and DNS cutover (1–10 min). RPO/RTO medium; cost ≈ 1.5×.
  • Cold backup — region restored from snapshot on demand. RPO/RTO measured in hours; cost ≈ 1.0–1.1×. Not viable for user-facing systems.

DR drill cadence

  • Quarterly game-day exercise pre-committed. Without it, RTO is fiction — you find out what's broken at the worst possible time.
  • Drill scope: kill the primary region; observe failover; measure actual RPO/RTO; capture every manual step that should have been automated.
  • Different scenarios: full-region loss, regional brown-out (slow but not down), single-AZ loss, dependent-service outage.

What every multi-region page must state

"RPO = X s, RTO = Y min, drill cadence = Z, failover mode = active-active|active-passive|cold."

Anti-patterns

  • Multi-region "for resilience" but all writes go to one region's leader — you have geo-redundant storage, not a survivable system.
  • Untested DR plan — RTO claims that no one has measured.
  • Confusing replication (a property of the data layer) with DR (a property of the entire system, including config, secrets, traffic management).

X4 — Capacity Planning & Autoscaling

Autoscaling signal

  • Queue depth for async / consumer systems. Direct measure of "are we keeping up?"
  • Custom request budget for tail-latency-sensitive systems. e.g., "scale up when p99 > SLO for 1 min."
  • CPU only for CPU-bound systems with no queueing. Anti-pattern for tail-latency systems — by the time CPU saturates, p99 is already broken.

Scale-up vs scale-down asymmetry

  • Scale up fast (prevent overload) — typically 30–60 s reaction time to add capacity.
  • Scale down slow (prevent flapping) — typically 5–30 min cooldown before removing capacity. Saves cost without paying the cold-start tax twice.

Predictive vs reactive

  • Predictive pre-scaling for known patterns: daily peak, weekly cron, marketing event, sports finals. Pre-warm to peak capacity 15 min before.
  • Reactive autoscaling for surprises only. The reactive path costs you the cold-start tax; minimize by pre-scaling whenever possible.

Cold-start cost

  • New instance time-to-serve: T seconds. (Java JVM warmup ≈ 30 s; LLM model load ≈ 60–300 s; simple Go binary ≈ 5 s.)
  • T × scale-up rate = upper bound on burst absorption. If T = 60 s and you can spin 10 instances/min, you can absorb 10 × steady-state-per-instance new load per minute, no faster.
  • Pre-warm via shadow traffic — new instance receives mirrored production load before cutover.

Capacity headroom

  • Steady-state headroom: typ. +30%. Reasons: rolling deploys (one-third of fleet rebooting), bursts, gradual organic growth.
  • DR headroom: lose one zone, remaining must serve 100%. If 3 AZs, each AZ provisioned at 50% utilization at peak (so 2 AZs can serve 100%).
  • Always state the reason inline. "+30% headroom for rolling deploys + traffic burst."

Autoscaler failure mode

  • Autoscaler itself can fail (control-plane outage, faulty metric). Capacity floor (a minimum number of instances that's always running) prevents over-aggressive scale-down from compounding the failure.
  • Manual override path: operator can pin capacity at a number for a duration.

What every page must state

"Autoscaling on <signal>, scale-up T-up s, scale-down T-down min, headroom = X% (reason), capacity floor = N."

Anti-patterns

  • CPU-based autoscaling on a tail-latency system — scales after the SLO is already breached.
  • Scale-down faster than scale-up — fleet flaps with traffic ripple.
  • No capacity floor — autoscaler error scales fleet to zero; recovery requires a deploy.
  • "Auto-scale will handle it" with no derivation of T-up × steady-state-per-instance vs burst rate.

X5 — Subsetting & Connection Management

When subsetting is required

If caller_pods × backend_pods > 10⁶ connections, deterministic subsetting is mandatory, not optional. (e.g., 1k caller pods × 10k backend pods = 10⁷ connections — impossible.)

Deterministic subsetting

  • Each caller pod gets a subset of size k (typ. 20–50) of the backend pool.
  • Subset chosen by a hash of (caller_pod_id, backend_pool_version) using Maglev / rendezvous hashing — stable under churn (one backend leaving doesn't reshuffle others).
  • Trade: load uniformity vs connection count. Larger k = more uniform load + more connections.

Connection pool tuning

  • Max idle: typ. 10. Higher wastes resources; lower causes handshake on bursts.
  • Max active: typ. 50–100. The bound that prevents one slow dependency from holding too much.
  • Idle timeout: typ. 60–300 s. Long enough to amortize handshake; short enough to free resources.
  • Acquire timeout: typ. ≤ 1 s. Bounded so caller doesn't queue forever.

Keepalive vs new-connection

  • Keepalive saves the TLS handshake (~10 ms intra-region). For high-QPS callers, keepalive is right.
  • For low-QPS callers (< 1 req/s), keepalive holds resources for nothing — let the connection close.
  • HTTP/2 / gRPC stream multiplexing makes one connection serve many concurrent requests — the right default for service-to-service.

Anti-patterns

  • Per-request new connection — pays TLS handshake every call (~10 ms intra-region).
  • Unbounded pool — slow dependency consumes all caller threads, causes pool starvation.
  • Random subset that reshuffles on every backend deploy — caller-side cache misses cascade.

X6 — Cell-Based Architecture & Blast-Radius

What a cell is

A cell is a self-contained shard of users (or tenants) with its own data, compute, and control plane. A bug deployed to one cell cannot affect users in another cell. Cells are the strongest defense against the bad-deploy failure mode.

When you need cells

  • > 10M users on a single architecture — cell-loss blast radius is too large without partitioning.
  • Multi-tenant SaaS where one tenant's workload could affect another's — cells provide isolation.
  • Regulated environments where blast radius bounds are part of compliance.

Cell sizing

  • Small enough that loss of one cell affects ≤ X% of users. (Typ. X ≤ 10%.)
  • Large enough that operational overhead is bounded. Per-cell deploys, monitors, on-call rotation must scale.
  • Sweet spot: 4–16 cells. Beyond 16, ops cost dominates.

Cross-cell traffic

  • Should be the exception, not the rule. If most traffic crosses cells, you don't have cells — you have one big system with extra hops.
  • Common cross-cell needs: shared metadata (registry, schema), inter-user social features (one user in cell A friends one in cell B). Handle via a thin coordination tier outside cells, with cached lookups.
  • Rate-limit cross-cell traffic explicitly — the absence of a limit means a bug can saturate the coordination tier.

What every page with > 10M users must state

"N cells × users-per-cell; cell-loss blast radius = M%; cross-cell-traffic budget = K% of total."

Anti-patterns

  • "Cells" that share a database — not cells, just app-server pools.
  • Cross-cell calls in the hot path — the coordination tier becomes the bottleneck.
  • One cell that's "the special cell" (everyone's metadata, the admin tool, the analytics pipeline) — it's the new single point of failure.

X7 — Real-Time Push (WebSocket / SSE / Long-Poll)

Protocol choice

  • SSE (Server-Sent Events) — one-way push, HTTP-native, auto-reconnect built in. Cheaper than WebSocket; right default for live tail, dashboards, notifications.
  • WebSocket — bidirectional. Right for chat, collaborative editing, multiplayer games. Heavier than SSE.
  • Long-poll — fallback for clients that don't support SSE/WebSocket. Higher latency (one full request per event).

Connection state

  • Sticky LB required so a connection lands on a specific pod and stays there.
  • Connection draining on deploy — old pod stops accepting new connections, lets existing ones drain over T seconds, then exits. T = max acceptable disruption (typ. 30–300 s).
  • Per-pod connection cap — typ. 50k–100k WS connections per pod. File-descriptor limits, ephemeral port range, and memory-per-connection set the ceiling.

Backpressure on the push side

  • What if the client can't drain as fast as the server pushes? Two policies:
  • Drop with high-water mark — if outgoing buffer > threshold, drop messages (ideally with a "dropped N messages" notice on reconnect). Right for telemetry / log tail where loss is tolerable.
  • Buffer with cap — buffer up to N messages; on overflow, kill the connection and let the client reconnect with a since-cursor. Right for chat / collaborative state where loss is not tolerable.

Reconnect storm

  • When a pod dies, all 50k clients reconnect simultaneously. Without admission control, the new pod (and the LB) is overwhelmed.
  • Mandatory: backoff with jitter at the client. Server-side: connection-establishment rate limit, with priority for paid tier / specific user classes.

Anti-patterns

  • WebSocket without sticky LB — every reconnect lands on a different pod, app state lost.
  • Unbounded server-side buffer per connection — slow consumer fills RAM until OOM.
  • No reconnect-storm protection — pod death cascades into a cluster-wide outage.

X8 — Edge Compute (Workers / Lambda@Edge / Fastly Compute)

When to terminate at the edge

  • Cacheable read with high cardinality and bounded staleness — autocomplete, anonymous timeline.
  • Stateless decision with small input — rate-limit decision, A/B bucket assignment, geo-routing, simple transform.
  • Latency-sensitive validation — JWT validation, header rewriting, basic auth.

Edge runtime constraints

  • Cold start: Cloudflare Workers ≈ 5 ms; Lambda@Edge ≈ 100–500 ms; Fastly Compute ≈ 1–5 ms.
  • Memory: 128 MB typical; 256 MB max on most providers.
  • CPU time: 50 ms per request typical; 200 ms hard cap.
  • Network: subrequest count limited (e.g., 50 fetches per Worker invocation).
  • Sandboxing: V8 isolate (Workers), microVM (Lambda@Edge), WASM (Fastly Compute) — each has different ecosystem support.

State at the edge

  • Workers KV — eventually consistent, ~ 1 min global propagation. Right for config, A/B bucket assignments, feature flags.
  • Durable Objects — single-region strongly-consistent state. Right for per-key state (rate limit counter, single-flight coordinator).
  • Edge Cache — HTTP cache with manual purge / tag-based invalidation. Right for cacheable reads.
  • Not a substitute for the origin DB. Edge state is auxiliary; canonical state lives at origin.

Edge cache invalidation

  • TTL-based: simplest, accept the staleness window.
  • Surrogate-Key / Cache-Tag: invalidate by tag (e.g., user_id); origin sends purge on mutation. Most common pattern.
  • Stale-while-revalidate: serve stale on miss, refresh in background. Defends against origin slowness.

What every page where edge applies must state

"Edge: terminate <operation> at edge with <runtime>; state at edge = <KV / DO / cache / none>; invalidation = <TTL / tag-purge / SWR>; latency = X ms vs Y ms at origin."

Anti-patterns

  • Database calls from edge — round-trip-back-to-origin defeats the purpose.
  • Strongly-consistent state across edge POPs — physically impossible at low latency; pick AP.
  • Edge logic too complex to fit in 50 ms / 128 MB — gets cut off mid-request.
  • Personalized cache without varying the cache key on user identity — leaks data across users.
