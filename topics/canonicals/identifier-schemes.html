Identifier Schemes

UUIDv4 / UUIDv7 / Snowflake / KSUID / ULID / hash-of-content / monotonic-sequence / human-readable-slugs. The properties that matter (uniqueness, sortability, forgery-resistance, clock-dependence, size, partition-friendliness) and how to pick. This is the source-of-truth page; scenarios cite this when picking an ID scheme for an entity.

~8 min reference

The Six Properties

Every ID scheme is a particular set of trade-offs across these:

  • Uniqueness without coordination. Can N independent generators produce IDs without consulting a central allocator? (Crucial for high write rates.)
  • Sortability / time-ordering. Does IDs sort in time order? Useful for clustering keys, range scans, "newest first" queries.
  • Forgery resistance. Can an attacker guess valid IDs they don't own? Affects security if IDs are exposed in URLs.
  • Clock dependence. Does the scheme require synchronized clocks? Clock skew → ID collisions or out-of-order IDs.
  • Size on wire / on disk. 8 bytes vs 16 vs 26-character base32. Multiplied by N rows × N indexes, this matters.
  • Partition-friendliness. Does the ID itself shard well? IDs that cluster (e.g., monotonic sequence) create hot partitions; uniform IDs don't.

Pick by which 2–3 properties dominate for the entity. There's no universally "best" scheme.

Per-Scheme Detail

UUIDv4 (random)

  • Format. 128 bits, of which 122 are random. Standard hex with hyphens (36 chars) or base64 (22 chars).
  • Wins. Uniqueness without any coordination — generate in any service, any thread, any time. No collisions in practice (122 bits of entropy means ~1 collision in 2.71 quintillion). Forgery-resistant if exposed in URLs.
  • Loses. Not sortable (random distribution). When used as a primary key in a B-tree, inserts are scattered across the index — defeats sequential write optimization (page splits, lower buffer-cache hit rate). Larger on disk than ints.
  • Pick when. ID needs to be opaque + unguessable + coordination-free, and the storage engine doesn't care about insert order (LSM). Or when sorted-write doesn't matter for performance.
  • Real implementations. Cassandra default, MongoDB ObjectId variants, most modern web APIs.

UUIDv7 (time-ordered)

  • Format. 128 bits: 48 bits Unix-ms timestamp + 74 bits random + 6 version/variant bits.
  • Wins. All UUIDv4 wins (coordination-free, opaque-ish), plus sortability — IDs created later sort after IDs created earlier. Inserts into a B-tree are sequential at the index leaves (no page splits). Replaces UUIDv4 in most modern designs as the better default.
  • Loses. Forgery resistance is reduced (74 bits random vs 122 — still effectively unguessable, but less). Time-clustering creates hot-tail in a range-partitioned store (mitigated by hash-prefixing, but defeats sortability).
  • Pick when. ID needs to be coordination-free, sortable in time order, opaque-ish. The modern default for new applications.
  • Real implementations. Postgres 16+ has gen_random_uuid for v4; v7 generated app-side or via extensions. Newer systems (Linear, etc.) use v7 as the default key.

Snowflake (Twitter-style)

  • Format. 64 bits: 1 bit unused + 41 bits ms timestamp + 10 bits worker-id + 12 bits sequence counter. Fits in an INT64.
  • Wins. Smallest (8 bytes); fastest for B-tree inserts (perfectly sequential). Sortable by time at ms precision. No coordination once worker IDs are assigned.
  • Loses. Forgery: trivially predictable. Anyone with one ID can guess nearby ones. Worker-ID assignment is its own coordination problem (typ. ZK / etcd registration on startup). Clock-skew sensitive: if a worker's clock goes backward, IDs collide.
  • Pick when. Internal IDs not exposed to users; size on disk matters; sortability is critical for clustering or range scans; you can manage worker-ID allocation.
  • Real implementations. Twitter (origin), Discord (variant), Sony, Instagram (variant). Common at companies that hit B-tree-with-UUID pain.

KSUID

  • Format. 160 bits (20 bytes): 32-bit second-precision timestamp + 128 bits random. Base62-encoded for 27 chars.
  • Wins. Sortable to second precision; coordination-free; forgery-resistant (128 bits random); URL-safe alphanumeric.
  • Loses. Larger than UUIDv7. Second precision is too coarse for high-rate IDs (within one second, order is randomized).
  • Pick when. Want sortability + forgery resistance + URL-safe encoding; rate is moderate (≤ thousands/sec per generator).
  • Real implementations. Segment (origin), Stripe-like APIs that prefix IDs with a type code (cus_2K8s...).

ULID

  • Format. 128 bits: 48 bits ms timestamp + 80 bits random. Encoded as 26-char Crockford base32 (URL-safe, no I / L / O / U).
  • Wins. Same size as UUID (16 B); sortable to ms precision; URL-safe encoding human-friendlier than hex; coordination-free.
  • Loses. Within one ms, order is randomized (compared to sequence-based schemes). UUIDv7 is functionally similar with a more standard encoding; ULID is mostly chosen for the prettier text encoding.
  • Pick when. Want UUIDv7-ish properties with a more compact text representation.

Hash of content (content-addressable)

  • Format. A cryptographic hash (SHA-256: 32 B / 64 hex chars) of the entity's content.
  • Wins. Inherent deduplication — identical content has the same ID, so storing twice is a no-op. Tamper detection — content can be verified against ID. Coordination-free.
  • Loses. Not sortable. Larger than time-based schemes. Cannot generate the ID until you have the content. Same content always gets the same ID — fine for backup/dedup, not fine when "two events with the same payload but different times" should be distinct entities.
  • Pick when. Content-addressable storage (backup dedup, Git objects, IPFS, container layers, Merkle trees).
  • Real implementations. Git (SHA-1, migrating to SHA-256), Backblaze B2, Dropbox Magic Pocket, container registries.

Monotonic sequence (auto-increment)

  • Format. 4 / 8-byte integer, allocated by a central counter.
  • Wins. Smallest. Most efficient B-tree insert (perfectly sequential; no page splits). Sortable.
  • Loses. Requires a central allocator (or a per-shard sub-counter, which then complicates uniqueness). Hot-tail: every new insert lands on the latest page (B-tree) or partition (range-sharded). Forgery: trivially predictable. Reveals business volume (sequential IDs leak rate).
  • Pick when. Single-DB-no-shard system; size and B-tree friendliness dominate; the forgery / hot-tail issues don't apply.
  • Anti-pattern at scale. Auto-increment IDs in a sharded system require a coordinating allocator that becomes the bottleneck. Use Snowflake or UUIDv7 instead.

Human-readable slug

  • Format. URL-safe text derived from a name/title (e.g., cats-vs-dogs-2026) plus a uniqueness suffix if needed.
  • Wins. Human-readable URLs; SEO-friendly; recognizable in admin UIs.
  • Loses. Uniqueness is hard (collision on duplicate titles — needs suffixes); regeneration on title change breaks links (or needs redirect tables).
  • Pick when. User-facing URLs (blog posts, articles, product listings); the entity has a natural human name.
  • Pattern. Slugs as aliases alongside an opaque internal ID. Slug for the URL; UUIDv7 for the database PK. Decouples the two.

Decision Heuristics

  1. Is dedup the goal? → hash-of-content.
  2. Internal-only ID, B-tree primary key, size matters? → Snowflake (smallest, sortable). Worker-ID coordination must be in scope.
  3. Public-facing ID in URLs, must be unguessable? → UUIDv4 (max forgery resistance). Or UUIDv7 if you also want sortability and v4's full entropy is overkill.
  4. Default for new apps (modern web service) → UUIDv7. Coordination-free + sortable + acceptable forgery resistance + standard format.
  5. Want UUIDv7 properties + prettier text encoding? → ULID.
  6. API-style with type prefix (cus_, ord_, req_)? → KSUID-like (Stripe-style). Or Snowflake / UUIDv7 with a prefix.
  7. User-facing URL slug for SEO? → Slug + opaque internal ID; never use the slug as the PK.

The Hot-Tail Problem (Time-Ordered IDs)

UUIDv7 / Snowflake / KSUID / ULID all sort in time. That's a feature for sortability, but it creates a hot-tail when used as a partition key in a range-sharded store: every new insert hits the latest partition.

Defenses:

  • Don't partition by the ID. Partition by something else (user_id, tenant_id) and cluster by the ID. The ID's sortability is preserved within each partition; the partition load is spread.
  • Hash-prefix the partition key. Partition key = hash(time-prefix(id)) % N + id. Spreads writes; defeats range-scans by ID.
  • Use hash sharding instead of range. Hash on the ID gives uniform distribution; you lose the natural sorted scans but writes are spread.
  • Auto-split. Modern range stores (CockroachDB, Spanner) auto-split hot partitions; the hot-tail problem self-corrects on a delay.

For B-tree primary indexes (Postgres), time-ordered IDs are a win (sequential leaf inserts; no page splits). For range-sharded distributed stores, time-ordered IDs are a loss. Match scheme to engine.

Anti-Patterns

  • Auto-increment in a sharded system. Either bottlenecks on the allocator or requires per-shard sub-sequences that complicate uniqueness. Use Snowflake or UUIDv7.
  • UUIDv4 as a B-tree primary key on a write-heavy table. Random insertion creates page splits and inflates write amp. Switch to UUIDv7 or Snowflake.
  • Snowflake with no worker-ID coordination plan. The 10-bit worker ID needs unique allocation per generator. Without it, two generators with the same ID will produce duplicates within the same ms.
  • Using a hash-of-content ID for non-content-addressable data. "Two events with the same payload" become the same row — silently drops duplicates that should be distinct.
  • Sequential IDs leaking business volume. Auto-increment IDs in URLs reveal "how many orders does this company process per day" to anyone watching. Use Snowflake or UUIDv7.
  • Slug as PK. Title changes break the PK. Use slug as a secondary lookup; PK is opaque ID.
  • Time-ordered ID as range-shard partition key. Hot-tail on every insert. Use a different partition key (user_id, tenant_id) and cluster by the time-ordered ID within partition.
