Core Entities & Data Modeling

How to derive the 3–8 entities a system traffics in (the S1 step), what minimum shape each needs, how to spot the entity that becomes the partition key, when a relationship deserves to be its own entity. This is the source-of-truth page; scenario S1 sections cite this for the entity-derivation protocol.

~8 min reference

The S1 Step

S1 (Core Entities) is where you prove you understood the domain before drawing boxes. The output is 3–8 nouns the system traffics in, each with the minimum shape that matters for the design — id + 2–3 attributes that drive a decision.

Full schemas live in C9 (the schema sketch). S1 is intentionally light: just enough to pin down what the system is about. The nouns you pick here become the table names, the partition keys, the Kafka topic names, the API resource names. Get them right and the rest of the design has clean joints; get them wrong and every later section fights the model.

The Three-Step Derivation

Step 1: Extract candidate nouns from the FRs

Read the FR list (from functional-requirements.html). Underline every noun. Common buckets:

  • Actors — User, Admin, Tenant, Service.
  • Content / artifacts — Post, Image, Document, Event, File, Metric.
  • Relationships — Follow, Friendship, Membership, Permission, Subscription.
  • Aggregates / collections — Timeline, Inbox, Cart, Playlist, Leaderboard.
  • State / lifecycle — Order, Job, Booking, Reservation, Session, Lease.
  • Audit / log — Activity, AuditEntry, Transaction, Receipt.

Step 2: Collapse to 3–8 first-class entities

Not every noun deserves its own entity. Collapse aggressively:

  • Attributes vs entities. "User name" and "user email" are attributes of User, not separate entities. "Follower count" is a derived attribute, not an entity.
  • Aggregates vs entities. Sometimes an aggregate (Timeline) is its own entity because it has its own identity, lifecycle, and access pattern. Sometimes it's just a query against another (Posts where author_id ∈ followees) — not its own entity.
  • State machines vs entities. If a thing has multiple states (PENDING → ACTIVE → COMPLETED), it's almost certainly its own entity (Order, Booking, Job).

Aim for 3–8 entities. Less than 3 means you've over-collapsed; more than 8 means the model is unfocused (or the system actually has many — fine for #44 object metadata, suspicious for most others).

Step 3: For each entity, write the minimum shape

EntityName {
  id (key shape — see partition-key section below)
  <1–3 attributes that drive a design decision>
}

Don't list every attribute. List only the ones that affect the architecture. "Post has author_id, created_at, body" — body affects storage decisions; author_id affects sharding; created_at affects clustering. Email-address-of-author doesn't.

When a Relationship Becomes Its Own Entity

This is the most-skipped S1 decision. A relationship between two entities can be:

  1. An attribute on one side. Cheap. Comment has post_id.
  2. A many-to-many join table. Cheap. Tag and Post connected by PostTag(post_id, tag_id).
  3. A first-class entity with its own identity, lifecycle, and access pattern. Promote when:
    • The relationship has its own state (Friendship has REQUESTED → ACCEPTED → BLOCKED).
    • The relationship has its own history (Membership joined-on, role-at-joining, role-changes).
    • The relationship has its own access pattern (Follow is queried independently of User and Followee — "who follows X" / "who does Y follow"; an attribute on User wouldn't support both directions).
    • The relationship has its own cardinality concerns (Follow on Twitter — celebrity has 200M; embedding-as-attribute fails).

Twitter's Follow is a first-class entity. Slack's Membership is a first-class entity. A Post's "tags" are usually just an attribute (a list).

Spotting the Partition Key

Among your 3–8 entities, one will dominate the access pattern and become the partition key for the rest. Spotting it early shapes everything downstream.

Heuristics, in order

  1. Which entity does the user-facing query path traverse most? If 90% of reads are "give me this user's X / Y / Z," then user_id is the partition key for X, Y, Z.
  2. Which entity has the strongest natural locality? Tweets by the same author, messages in the same chat, events in the same session — locality favors co-locating to that key.
  3. Which entity, when sharded by it, gives uniform-enough load? user_id usually is uniform (modulo celebrities). post_id is uniform. tenant_id is suspect when tenants are unevenly sized.
  4. Which entity, when sharded by something else, would force cross-shard queries? If your hot query is "all messages in chat X" and you shard by user_id, every read fans out to all users in the chat. Shard by chat_id instead.

The classic conflicts

  • "Posts by author" vs "Posts on timeline." Sharded by author_id: cheap to read author's posts, expensive to assemble timeline. Sharded by reader_id (denormalized into timelines): cheap to read timeline, expensive to update on post (fan-out). The trade is what makes #41 timeline-fanout interesting.
  • "Per-user" vs "per-tenant." Multi-tenant SaaS often shards by tenant_id, but if tenants are uneven (10 huge, 1M small), tenant_id creates hot partitions. Often: tenant_id + hash(user_id) compound key.
  • "Time-series prefix." Anything keyed by created_at is a hot-partition trap (the latest partition cooks; older ones are cold). Add a hash prefix or shard by something else.

Document this in S1

For each entity, write the partition key choice next to it, even if just one word. It's a forward reference to C9 but it pins the decision early so you don't accidentally model an entity in a way the partition can't support.

Output Examples

Example A: Twitter

Core Entities (3–8):

  User {
    user_id (PK; partitions Posts, Follows by author/follower)
    handle, display_name (queried by handle for profile)
  }
  Post {
    post_id (PK; secondary by author_id+created_at)
    author_id, body, created_at, kind
  }
  Follow {
    (follower_id, followee_id) (composite PK; first-class because
                                 dual access pattern + cardinality skew)
    created_at
  }
  Like {
    (user_id, post_id) (composite PK; first-class because audit + count)
    created_at
  }
  Timeline {
    user_id (PK)
    materialized list of post_ids (denormalized for read perf —
                                    promoted to entity because it's
                                    its own access pattern + storage)
  }

Example B: Distributed cache

Core Entities (3–8):

  CacheEntry {
    key (PK; partition by hash(key))
    value (opaque bytes)
    ttl, last_modified, lease_token
  }
  Shard {
    shard_id (PK)
    primary_node, replica_nodes, weight
  }
  Lease (when single-flight is in scope) {
    (key, lease_id) (PK)
    holder_node, expires_at
  }

(That's it — 2–3 entities. Infra primitives often have very few.)

Example C: Web crawler

Core Entities (3–8):

  URL {
    url (PK; partition by hash(host) for politeness grouping)
    host, last_crawled, next_crawl_at, priority, status
  }
  Document (the fetched content) {
    content_hash (PK; content-addressable for dedup)
    url, fetched_at, http_status, body_ref (S3), outlinks[]
  }
  Host {
    host (PK)
    robots_txt, last_crawl_at, rate_limit, politeness_state
  }
  CrawlJob {
    job_id (PK)
    seed_url_set, started_at, completion%, state
  }

Example D: Recommender

Core Entities (3–8):

  User {
    user_id (PK)
    feature_vector (dense ~256d, refreshed hourly)
  }
  Item {
    item_id (PK)
    feature_vector, popularity_score
  }
  Impression {
    impression_id (PK; append-only; partition by user_id+ts)
    user_id, item_id, score_at_serve, position, served_at
  }
  Click (Impression with kind=click — could merge or keep separate
          depending on rate of clicks vs impressions) {
    impression_id (FK)
    clicked_at
  }
  Model {
    model_id (PK)
    artifact_url, trained_at, ndcg, deployed_at
  }

Anti-Patterns

  • Entity list = full schema. S1 is 3–8 entities × 2–3 attributes each. Full schemas (with every column, type, index) live in C9. Mixing them clutters S1 and delays the architecture discussion.
  • Missing the relationship-as-entity decision. Friendship / Membership / Follow modelled as an attribute when it deserves to be its own entity — and the design later has to fight to support both-direction queries.
  • Time-series partition trap. Picking created_at as partition key. Always hot-partitions on the latest bucket.
  • Aggregate-as-attribute. Storing followers as a list field on User. Works at small N; melts at celebrity scale (200M followers in one row).
  • Over-modelled. 15 entities for a Twitter design — the interviewer can't tell what matters. Collapse.
  • Under-modelled. Two entities ("User" and "Stuff") for any non-trivial system. Means you haven't thought about access patterns.
  • Picking a partition key that doesn't survive the dominant query. Sharding messages by user_id when the dominant query is "all messages in chat X" — every read fans out.
  • No partition key noted in S1. Then S1 is incomplete; the entity doesn't pin its layout.
