Bottleneck → Design Decisions

The reverse direction of the bottlenecks page: given the dominant bottleneck, here are the specific architectural decisions that close it. Each entry maps to concrete S5 design moves, not vague "first lever." Use this in the iteration loop to translate diagnosis into action fast.

~10 min reference

How to Use This Page

The bottlenecks page tells you what shape a saturation has. This page tells you which architectural decisions you would change to close it. The two pages are forward / reverse lookups of the same relationship.

Use this page in S5: once an iteration has identified the dominant bottleneck, look it up here and read off the candidate decisions. Pick one (cite C4 alternatives for the others), apply it, recompute, return to the NFR walk.

Each entry below has the same shape:

  • Diagnostic signal — what number told you you're here.
  • Decisions to revisit — the architectural choices that, if changed, close the gap.
  • Likely first move — the lowest-cost decision change to try first.
  • Common combos — moves that pair well or are usually applied together.
  • Anti-move — what looks reasonable but doesn't actually fix this bottleneck.

Latency-failing bottlenecks

Hot key (one key cooks one node)

  • Signal. One key > 100× median QPS; one node pegged regardless of cluster size.
  • Decisions to revisit. Caching strategy (front the origin); single-flight policy on miss; key replication via salting; admission policy (W-TinyLFU); hot-key isolation tier.
  • Likely first move. Cache + single-flight on miss. Solves most cases without further structural change.
  • Combos. Cache + single-flight + jittered TTL + refresh-ahead is the canonical four-pack for read-heavy power-law.
  • Anti-move. Add more shards. The hot key still lands on one of them.
  • Cite. bottlenecks#hot-key; caching; single-flight.

Tail-latency amplification

  • Signal. System p99 ≫ component p99; result needs N parallel calls.
  • Decisions to revisit. Hedging policy (re-issue at p95 to a second backend); fan-out factor (reduce N); deadline propagation; bounded-result-set with partial-OK (return what came back); outlier ejection from LB pool.
  • Likely first move. Hedged requests at p95 deadline. ~5–10% extra origin load buys p99 close to component p95.
  • Combos. Hedging + outlier ejection (kicks out persistently slow nodes) + adaptive concurrency limits (slows callers when latency rises).
  • Anti-move. Reduce component p99 alone. Curve flattens; you can't beat the multiplication of N samples.
  • Cite. bottlenecks#tail-latency.

Network-RTT-bound

  • Signal. p50 ≈ k × RTT; CPU/disk idle; reducing payload doesn't help.
  • Decisions to revisit. Round-trip count (denormalize / batch); placement (move callee closer — read replica in caller's region, edge cache, CDN); pipelining (don't wait between requests); speculative dispatch.
  • Likely first move. Cache or read-replica in the caller's region (cuts cross-region RTT to intra-region).
  • Combos. Edge cache + CDN for user-facing reads; per-region read replica for service-to-service.
  • Anti-move. Faster CPUs. Speed of light unchanged.
  • Cite. bottlenecks#network-rtt-bound; cross-cutting#edge.

Stampede / thundering herd

  • Signal. Origin QPS spike from 1/TTL to N × QPSread on cache expiry.
  • Decisions to revisit. Single-flight policy; TTL jitter; refresh-ahead; request coalescing at edge.
  • Likely first move. Single-flight on miss + jittered TTL (±20%).
  • Combos. Single-flight + refresh-ahead for predictably-hot keys + stale-while-revalidate as the degradation contract.
  • Anti-move. Add more origin capacity. Doesn't address the spike — just makes its cost higher.
  • Cite. bottlenecks#stampede.

Cold start / churn cost

  • Signal. Per-instance warm-up T; (T × churn-rate) > allowable warm-up fraction.
  • Decisions to revisit. Pre-warm policy (shadow traffic before cutover); instance lifetime (longer-lived; less aggressive autoscale-down); tiered cache absorbing cold-start; deployment cadence.
  • Likely first move. Pre-warm via shadow traffic.
  • Combos. Pre-warm + slower scale-down (5–30 min cooldown) + capacity floor + L2 cache that absorbs cold-instance reads.
  • Anti-move. Aggressive scale-up. More cold instances during a burst makes the problem worse, not better, until they warm.
  • Cite. bottlenecks#cold-start.

Cross-region latency / consistency

  • Signal. p50 includes ≥ 1 cross-region RTT (60–200 ms); strong-consistency forces sync replication blowing latency SLO.
  • Decisions to revisit. Per-dataset CAP choice (which datasets need CP, which AP); replication topology (per-region leaders; multi-leader async; per-record region-of-truth); conflict resolution (LWW / CRDT / app-merge); residency policy.
  • Likely first move. Per-dataset CAP audit. Most datasets become AP with bounded staleness; small slice stays CP at the cost of latency.
  • Combos. Per-region leader (single-leader async cross-region) + CRDT for natively-mergeable counters + application-merge for records.
  • Anti-move. "Multi-region active-active strong consistency" without per-dataset audit. Forces every dataset onto worst-latency profile.
  • Cite. bottlenecks#cross-region; replication-decisions#multi-leader-async.

Throughput-failing bottlenecks

Hot partition

  • Signal. Per-shard QPS / bytes > 5× cluster median for > 5 min.
  • Decisions to revisit. Partition key choice (more uniform field); salting (hash-prefix the key); auto-split policy; sharding strategy (hash vs range); composite key with high-cardinality outer dimension.
  • Likely first move. Salt the partition key with a short hash prefix. Defeats range scans on the un-salted dimension; trade.
  • Combos. Salt + auto-split + per-tenant quota (so one tenant doesn't dominate one shard).
  • Anti-move. Add nodes. Uniform sharding doesn't help if key distribution is skewed.
  • Cite. bottlenecks#hot-partition; sharding-strategies.

Coordinator bottleneck

  • Signal. Single leader / sequencer at 100%; system throughput pegged to its capacity.
  • Decisions to revisit. Coordinator placement (metadata path vs data path); lease-based delegation; partition the coordinator (per-tenant / per-region); leaderless / CRDT alternative.
  • Likely first move. Move coordinator off the data path. Workers operate autonomously on lease; coordinator only allocates leases.
  • Combos. Lease delegation + coordinator partitioning + consensus only on metadata.
  • Anti-move. More stateless workers behind one coordinator. Workers idle waiting; coordinator bottleneck unchanged.
  • Cite. bottlenecks#coordinator-bottleneck.

Connection saturation

  • Signal. Caller × backend > 10⁶ TCP connections; FD exhaustion; TLS-handshake CPU dominates.
  • Decisions to revisit. Subsetting policy (each caller talks to k of N); connection multiplexing (HTTP/2, gRPC streams); pooling tuning; sidecar pooling.
  • Likely first move. Deterministic subsetting (Maglev or rendezvous-hash table; k typ. 20–50).
  • Combos. Subsetting + HTTP/2 / gRPC mux + bounded pool with timeout + sidecar.
  • Anti-move. Random per-request connection. Pays TLS handshake on every call (~10 ms).
  • Cite. bottlenecks#connection-saturation; cross-cutting#subsetting.

Lock contention

  • Signal. Latency rises with concurrency; throughput plateaus; profile shows time in lock-acquire.
  • Decisions to revisit. Lock granularity (shard the lock); lock-free / CAS structures; per-thread state with periodic merge; CRDT or append-only log.
  • Likely first move. Shard the lock by the partition key. If you need per-key serialization, partition by key.
  • Combos. Sharded locks + per-thread merging + lock-free CAS counters.
  • Anti-move. Optimize the work inside the critical section. Buys you a constant factor; throughput still serialized.
  • Cite. bottlenecks#lock-contention.

Index bloat / write amp

  • Signal. Per-write IOPS = K × insertions; index size approaches data size; write throughput plateaus.
  • Decisions to revisit. Number of indexes (drop unused); async / eventual index build (CDC to a separate read store); storage engine swap (LSM if write-bound, columnar if scan-bound); secondary engine sidecar.
  • Likely first move. Drop indexes that don't earn their cost; measure query frequency × index maintenance.
  • Combos. Drop low-value indexes + push rare-query paths to Elasticsearch (or columnar) sidecar fed via CDC.
  • Anti-move. "Indexes for completeness." Every index has cost on every write.
  • Cite. bottlenecks#index-write-amp; storage-engines.

Resource-failing bottlenecks

Memory-bandwidth-bound

  • Signal. Bytes-read/s near DRAM/HBM peak; CPU/GPU idle waiting.
  • Decisions to revisit. Working-set size per op (quantize, prune, compress in-memory); arithmetic intensity (batch — more FLOPs per byte loaded); locality (move data closer — L2/L3, NUMA, HBM).
  • Likely first move. Quantize / shrink the working set.
  • Combos. Quantization + batching (LLM decode is the canonical example).
  • Anti-move. More cores. They all stall on the same memory channel.
  • Cite. bottlenecks#memory-bandwidth-bound.

Memory-capacity-bound

  • Signal. Working set > RAM; eviction churn; high page-fault rate.
  • Decisions to revisit. Tier to slower-larger medium (NVMe, S3); shrink representation (compact serialization, delta); shard the dataset; sketch (give up exactness for bounded memory).
  • Likely first move. Tier to NVMe (CacheLib pattern). Often 10× capacity at acceptable latency.
  • Combos. Tier + admission policy (W-TinyLFU keeps the right things in RAM).
  • Anti-move. Smarter eviction policy on a working set that's fundamentally larger than RAM. No policy beats actually having the data.
  • Cite. bottlenecks#memory-capacity-bound; caching.

Disk-IO-bound

  • Signal. IOPS / MB/s saturated; queue depth growing on the device.
  • Decisions to revisit. Storage engine (LSM for sequential writes); fsync batching; tier to faster device (NVMe); precomputed reads (materialized views); async durability (Kafka in front).
  • Likely first move. Switch to LSM if currently on a B-tree under write-heavy load. (Or: stay on B-tree, reduce indexes, batch writes.)
  • Combos. LSM + WAL + batched commit + tiered (hot on NVMe, cold on S3).
  • Anti-move. Add indexes "to speed up reads" when the bottleneck is the resulting write amp.
  • Cite. bottlenecks#disk-io-bound.

Network-bandwidth-bound

  • Signal. NIC at line rate; cross-AZ egress dominates the bill.
  • Decisions to revisit. Compression (Zstd / LZ4); co-location (same AZ); fanout shape (gossip / tree vs broadcast); push-down (compute at the data, not at the caller).
  • Likely first move. Compress on the wire.
  • Combos. Compression + co-location + server-side aggregation.
  • Anti-move. Caching downstream of the bottleneck NIC. Doesn't reduce bytes-on-wire.
  • Cite. bottlenecks#network-bandwidth-bound.

Flow-control bottlenecks

Backlog growth

  • Signal. Producer rate > consumer rate sustained; queue depth monotonically up.
  • Decisions to revisit. Backpressure policy (HTTP 429 / TCP slow-down to producers); shed-by-priority; consumer scaling (auto-scale on queue depth, not CPU); ingress sampling; bounded queue.
  • Likely first move. Backpressure to producers. Never silently buffer.
  • Combos. Bounded queue + shed by priority + autoscale on queue depth + per-tenant quotas.
  • Anti-move. Bigger queue capacity. Bigger buffer = bigger latency at saturation; not fewer drops.
  • Cite. bottlenecks#backlog-growth.

Retry storm

  • Signal. Failure → retry → more failure; load grows during outage.
  • Decisions to revisit. Retry policy (backoff with jitter); per-caller retry budget; circuit breaker thresholds; idempotency keys for safe server-side dedupe; classify which errors are retryable.
  • Likely first move. Exponential backoff with full jitter + retry budget capped at 10% of original calls.
  • Combos. Backoff + budget + circuit breaker + idempotency key + adaptive concurrency limits.
  • Anti-move. Retry without classification. Retrying 4xx is wasted; retrying without budget amplifies failure.
  • Cite. bottlenecks#retry-storm.

Scale / cardinality bottlenecks

Cardinality explosion

  • Signal. Per-key state × N grows unboundedly with input cardinality.
  • Decisions to revisit. Sketch swap (CMS / HLL / t-digest / DDSketch / Bloom); cap N with eviction (LRU); ingress sampling; tiered (exact for top-K hot, sketch for long tail).
  • Likely first move. Pick the right sketch from the sketches selection guide.
  • Combos. Sketch + tiered (Space-Saving for top-K + CMS for long tail) + eviction cap.
  • Anti-move. "Just add more RAM." Input grows faster.
  • Cite. bottlenecks#cardinality-explosion; sketches.

Fanout asymmetry

  • Signal. Top-percentile fanout count orders of magnitude above median (celebrity).
  • Decisions to revisit. Push (write-fanout) vs pull (read-fanin) — pick per-record. Time-bucketed materialization. Differential prefetch (predict who will read what).
  • Likely first move. Hybrid push/pull: push for normals, pull for celebrities. Defines a threshold (e.g., follower count > 1M).
  • Combos. Hybrid + per-user inbox cap (TTL on cold users) + read-time merge for celebrity content.
  • Anti-move. All-push or all-pull. Wrong end of the distribution melts.
  • Cite. bottlenecks#fanout-asymmetry.

How to use this page in S5

Concrete in-iteration usage:

  1. Iteration N's bottleneck analysis identifies the failing NFR and classifies against bottlenecks.
  2. Open this page; jump to the matching entry.
  3. Read "Decisions to revisit" — these become your C4 alternatives.
  4. "Likely first move" is your default choice; the others are alternatives you should explicitly compare against.
  5. "Combos" tells you which moves usually arrive together (saves a future iteration).
  6. "Anti-move" tells you what to not reach for — saves a wrong-path iteration.

This page replaces the open-ended "what should I try next?" question with a small menu of pre-validated options.
