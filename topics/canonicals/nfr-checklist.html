Non-Functional Requirements (NFR) Checklist

How to seek and quantify non-functional requirements (CAP, scalability, latency, durability, security, fault-tolerance, compliance, environment) after functional requirements are settled. This is the source-of-truth page; scenario S0 sections cite this for the NFR clarification protocol.

~10 min reference

The Principle: NFRs Must Be Contextual and Quantified

NFRs are how well the system does what it does. The bar that separates a senior from a staff+ candidate is whether NFRs are contextual and quantified.

  • Vague (avoid): "The system should be low latency." — obvious, not meaningful; nearly all systems should be low latency.
  • Quantified (do this): "The system should have low-latency search, < 500 ms." — identifies which part of the system most needs to be low-latency, and provides a concrete target.

A quantified NFR becomes the gate for an iteration: every later design choice must close the gap between the current numbers and this target. Vague NFRs do not gate anything — they accept any design.

The format

<qualifier> <part-of-system> <dimension> <operator> <quantity> <at-percentile>

Examples:
  read-path     latency        ≤ 200 ms        at p95
  write-path    durability     RPO ≤ 0         (no loss)
  search        latency        < 500 ms        at p99
  per-tenant    error rate     < 0.1%          rolling 5-min window
  global        availability   ≥ 99.95%        (43.8 min/year downtime)

Pick 3–5 NFRs That Are Most Relevant

Coming up with NFRs is challenging if you're not familiar with the domain. The checklist below is the menu — but you don't pick all of them. Pick the 3–5 most relevant to your system; the rest are out-of-scope (named, with one-line reason).

Picking too many NFRs means you can't honor any of them. Picking too few means you'll miss a constraint that the design must respect. 3–5 is the staff+ sweet spot.

The system class drives the shortlist. Different system classes (product / infra / pipeline / algorithmic — see FR system class) surface different NFR priorities. The general checklist applies to all; the per-class additions below contain NFRs that are load-bearing for that class but rarely asked about for others.

The NFR Checklist

For each entry: what it covers, the questions to ask the interviewer (or yourself, if there's no interviewer), and how to quantify the answer. The bottlenecks the entry typically surfaces are linked.

CAP — Consistency vs Availability under Partition

  • What. Should the system prioritize consistency (refuse writes during partition; everyone agrees on the value) or availability (accept writes; reconcile later)? Partition tolerance is a given in distributed systems.
  • Questions to ask. "Under network partition, do we accept writes on the minority side or refuse them?" "Is read-your-writes required? Across regions or just within one?" "Is monotonic-read OK or do users tolerate going-back-in-time briefly?"
  • Quantify. Per-dataset choice: "Inventory: CP, refuse writes during partition. Activity feed: AP with bounded staleness ≤ 5s." Don't pick one CAP setting for the whole system.
  • Surfaces. cross-region latency, coordinator bottleneck, split-brain.

Environment Constraints

  • What. Constraints on the environment in which the system runs. Mobile (limited battery, intermittent connectivity), embedded (limited memory), low-bandwidth networks (3G, satellite), regulated environment (gov-cloud), edge / serverless (cold-start, memory cap).
  • Questions to ask. "Is the client mobile, web, both? What's the assumed bandwidth?" "Are there battery / power constraints?" "Cloud, on-prem, hybrid? Reserved or spot?" "Existing systems we must integrate with (auth, identity, queue, DB)?"
  • Quantify. Concrete: "Mobile clients on 3G; payload ≤ 50 KB per request; tolerate 60s offline with re-sync." Or: "Cloud-only; AWS US-East primary; no on-prem."
  • Surfaces. network bandwidth, cold start, edge compute.

Scalability

  • What. All systems need to scale, but does this system have unique scaling requirements? Bursty traffic at a specific time of day? Seasonal events (holidays, game launches, news)? Read-heavy or write-heavy?
  • Questions to ask. "DAU? MAU? Growth horizon (1y, 5y)?" "Peak QPS vs average QPS? Peak-to-average ratio?" "Read:write ratio?" "Diurnal pattern? Spikes (Black Friday, sports finals)?" "Scale reads, writes, or both?"
  • Quantify. "100M DAU; peak 100k QPS reads, 5k QPS writes; 5× peak-to-average diurnal; 20× peak on Black Friday."
  • Surfaces. hot partition, backlog growth, cold start; the read:write ratio determines whether read-heavy or write-heavy levers apply.

Latency

  • What. How quickly the system responds. Especially relevant for any request that requires meaningful computation (search, ranking, dedup, aggregation).
  • Questions to ask. "What latency is acceptable at p50, p95, p99?" "End-to-end (including network) or server-side budget?" "Cold-start tolerance — first request after deploy?" "Which operations are latency-critical vs background?"
  • Quantify. Always state percentiles, not means. "Search: p95 ≤ 500 ms, p99 ≤ 1 s, server-side. Background re-indexing: no SLO."
  • Surfaces. tail-latency amplification, network-RTT, stampede, cold start.

Durability

  • What. How important that data is not lost. Banks: zero loss. Social networks: tolerate some loss. Telemetry: tolerate substantial loss.
  • Questions to ask. "What's the acceptable RPO (data loss in seconds/minutes)?" "Different RPO for different datasets?" "Is 'eventual durability' OK (e.g., write-behind cache → DB)?"
  • Quantify. "Transactional writes: RPO = 0 (no loss). Analytics events: RPO ≤ 5 min acceptable. Cache state: RPO = ∞ (rebuild from source)."
  • Surfaces. disk-IO (sync writes), backlog overflow (in-memory loss), DR.

Security

  • What. Data protection, access control, authentication, authorization, secret management, encryption, audit logging.
  • Questions to ask. "AuthN / AuthZ model?" "PII handling — encryption at rest, in transit?" "Multi-tenant isolation requirements?" "Audit / immutable-log requirements?" "Is fail-closed required (deny on error) or fail-open with audit?"
  • Quantify. "mTLS between services; JWT validation at gateway, ≤ 2 ms; fail-closed on auth errors. Per-tenant rate limit; one tenant cannot consume > 5% of cluster capacity."
  • Surfaces. cross-cutting security; per-tenant isolation surfaces hot-partition and pool-starvation.

Fault Tolerance

  • What. How well the system handles failures: redundancy, failover, recovery mechanisms.
  • Questions to ask. "Acceptable downtime (RTO) — seconds, minutes, hours per quarter?" "Per-AZ failure tolerance? Per-region?" "Failover automatic or manual?" "Drill cadence — quarterly?"
  • Quantify. "99.95% availability (43.8 min downtime/year). RTO ≤ 30s on AZ failure (active-active). RTO ≤ 5 min on region failure (warm standby). DR drill quarterly."
  • Surfaces. DR, cells, bad deploy, cascading timeout.

Compliance

  • What. Legal or regulatory requirements: GDPR (EU privacy), HIPAA (US health), PCI-DSS (payment cards), SOC2, SOX, region-specific data sovereignty.
  • Questions to ask. "GDPR / HIPAA / PCI / SOC2 in scope?" "Data residency — EU users in EU storage?" "Right-to-erasure — delete-by-user_id within how many days?" "Audit log retention — how many years?"
  • Quantify. "GDPR: erasure within 30 days. EU residency: EU users' data stays in eu-west-1. HIPAA-eligible. SOC2 audit-log retention: 7 years."
  • Surfaces. cross-cutting privacy; data-residency surfaces cross-region latency.

Cost / Unit Economics

  • What. The bill. Specifically: budget per request, per GB stored, per active user, total $/month ceiling.
  • Questions to ask. "What's the cost ceiling — $/month, $/user, $/request?" "Reserved or on-demand capacity?" "Is cost the gate, or latency, or availability?"
  • Quantify. "$/active-user ≤ $0.10/month. Total ceiling $50k/month. Reserved capacity at 70% utilization."
  • Surfaces. cost-shapes (the per-system-class cost dominator); cost ceiling tells you which iteration to stop at.

Multi-Tenancy & Fairness

  • What. Single-tenant vs multi-tenant; isolation level; per-tenant quotas; noisy-neighbor tolerance.
  • Questions to ask. "Single-tenant or multi-tenant?" "Tenant cardinality — 10 huge or 1M small?" "Per-tenant quotas? Hard or soft?" "Noisy-neighbor budget?"
  • Quantify. "50k tenants; per-tenant rate limit; one tenant ≤ 5% of cluster capacity at any moment; per-tenant data isolation via partition key."
  • Surfaces. hot-partition, multi-tenant isolation, pool starvation.

Schema Mutability

  • What. Stable schema vs rapidly evolving. Drives storage choice, migration story, query flexibility.
  • Questions to ask. "Schema fixed or evolving?" "How often do new fields get added?" "Are migrations zero-downtime required?"
  • Quantify. "Schema evolves quarterly; add-column backward compatible; destructive migrations require dual-write window of ≥ 7 days."
  • Surfaces. index bloat (changing indexes); storage engine choice.

Per-Class NFR Additions

Beyond the general checklist, each system class has 2–4 NFRs that are usually load-bearing for that class but rarely surface for others. Walk the relevant block based on the system class fixed in the FR step.

A. Product-shaped NFRs (additional)

  • User-perceived freshness. "How fresh must the read be?" "Timeline can be ≤ 30 s stale; cart contents must be read-your-writes; price displays ≤ 5 s stale." Drives caching aggressiveness vs invalidation cost.
  • Personalization budget. "How much per-user state can the read path touch?" "Per-request: at most 1 user-features lookup, 1 ranking call (≤ 50 ms server-side)." Determines whether to pre-compute or score-on-read.
  • Notification / push semantics. "Real-time push to which users on which events? Best-effort or guaranteed?" Drives connection management (X7) and fan-out architecture.
  • Localization / i18n. "How many locales / languages / currencies? Affects ranking, search, and storage."

B. Infrastructure primitive NFRs (additional)

  • API latency floor. Infra primitives are called from the hot path of N other services. "GET p50 ≤ 1 ms; p99 ≤ 5 ms; SET p99 ≤ 10 ms." Tighter than typical product latencies because every caller's SLO depends on yours.
  • Per-key consistency model. Stated as part of the FR, but quantified here. "Linearizable per-key for CAS; sequential for ordered-stream reads; eventual for cross-partition listings."
  • Per-key throughput limit. "What's the max QPS we can sustain on a single key before the caller must shard?" Determines whether the primitive is honest about hot-key behavior. "Per-key SET ≤ 10k QPS; beyond that, caller must shard or use a different primitive."
  • Operational primitives. "Drain / rebalance / snapshot / restore — exposed to operators? Self-service or ops-only?" Infra primitives are operated by humans far more than products are.
  • Multi-tenancy isolation guarantee. "What's the per-tenant blast radius? Can one tenant exhaust the cluster?" "Per-tenant quotas; one tenant ≤ 5% of cluster capacity at any moment; per-tenant connection pool."
  • Resource overhead per call. "How much CPU / memory / network does each call cost the caller?" Critical for sidecars and in-process libraries.
  • Determinism / debuggability. "Given the same input, will the primitive produce the same output? Can callers trace what happened to their request?" Often more important for infra than for products.

C. Automation / pipeline NFRs (additional)

  • Completeness. "What fraction of input must be processed?" "Crawler: ≥ 95% of seed URLs reached within 24 h. ETL: 100% of source rows materialized within 1 h of arrival." Distinct from latency — completeness is about coverage, not speed.
  • Freshness vs completeness trade. "When inputs arrive faster than we can process, do we drop oldest, drop newest, or fall behind?" "Crawler: process by priority; tail unprocessed forever is OK. Log pipeline: drop oldest after 1 h backlog (sample to disk archive)."
  • Idempotency on retry. Re-running the pipeline on the same input must produce the same output (or a logically-equivalent one). Quantify: "Re-run on the same URL set produces output that differs only where the actual web pages changed."
  • External-source rate-limiting (politeness). "When fetching from external systems we don't control, what's the per-source rate budget?" "≤ 1 req/sec/host by default; respect per-host overrides; back off on 429."
  • Recoverability / replay. "Can we re-process a historical window if a downstream consumer was wrong?" Often the difference between a recoverable bug and a permanent data loss.
  • Progress / observability. "Operators must be able to answer: how far through the input are we; how long until completion; which inputs failed and why."
  • Watermarks / late arrival. For streaming pipelines: "How late can an event arrive and still be processed correctly?" "Tolerate up to 10 min late; emit windows at watermark + 10 min."

D. Algorithmic / batch / ML NFRs (additional)

  • Accuracy / quality target. "What's the offline metric and the threshold?" "Recommender: NDCG@10 ≥ 0.42 on the held-out set; auto-rollback if regresses > 1%." Stated as a hard NFR, not a soft goal.
  • Drift tolerance. "How quickly must we detect that the model has gone stale?" "Drift score evaluated daily; if > threshold for 3 consecutive days, page on-call."
  • Training cadence. "How often is the model retrained? On schedule or triggered by data freshness?" "Daily training; on-demand re-training on flag — completes within 6 h."
  • Cold-start behavior. "What's served for unseen users / items? At what quality bar?"
  • Explainability. "Must each prediction come with an explanation? At what granularity?" Often regulatory (credit / fraud / hiring) — driven by compliance.
  • Fairness / bias bounds. "Are there protected attributes? Quantified disparity limits?" Treated as hard NFRs in regulated domains.
  • Online ↔ offline parity. "How do we ensure offline-trained model and online-served model produce the same output for the same input?" Feature parity, version pinning, shadow traffic.
  • Experiment / A-B-test latency. "How fast can we A/B-test a new model? How long until significance?" Drives whether you need fast deploy / shadow / partial rollout infrastructure.

The Output

A 3–5-NFR shortlist with each NFR quantified, plus the rest of the checklist named as out-of-scope (or "not load-bearing for this design") with a one-line reason. This is what every later iteration is judged against. Pull from both the general checklist and the per-class additions for the system class fixed in the FR step.

Example A: Product (Twitter)

Non-functional requirements (3–5 most relevant):

  1. Latency:     timeline read p95 ≤ 200 ms (server-side); p99 ≤ 800 ms.
  2. Scalability: 500M DAU; peak 100M timeline reads/sec.
  3. Availability: 99.95% (43.8 min/year); active-active multi-region.
  4. Durability:  posts RPO = 0; timeline cache RPO = ∞ (rebuild).
  5. User-perceived freshness: timeline ≤ 30 s stale acceptable.

Out of scope: SOC2 standard; broadband+; stable schema; $0.20/user-mo
ceiling informs but doesn't gate; standard mTLS / JWT.

Example B: Infra primitive (distributed cache)

Non-functional requirements (3–5 most relevant):

  1. API latency:  GET p50 ≤ 0.5 ms, p99 ≤ 5 ms; SET p99 ≤ 10 ms (in-DC).
  2. Per-key throughput: ≤ 100k QPS/key sustained before honest 503.
  3. Consistency: read-your-writes per-client; eventual cross-client.
                  CAS linearizable per-key.
  4. Multi-tenancy: per-tenant quota; one tenant ≤ 5% cluster capacity.
  5. Operability: drain a shard in ≤ 5 min; restore from snapshot ≤ 30 min.

Out of scope: persistence (in-memory only); cross-region (single-region);
schema validation (opaque bytes); compliance (no special regulatory).

Example C: Automation (web crawler)

Non-functional requirements (3–5 most relevant):

  1. Completeness:    ≥ 95% of seed URLs reached within 24 h.
  2. Politeness:      ≤ 1 req/sec/host default; respect robots.txt;
                      back off on 429 / 5xx with jitter.
  3. Idempotency:     re-run on same input set produces equivalent output;
                      output keyed by content-hash, dedup on overlap.
  4. Recoverability:  replay any past 30-day window from input + state.
  5. Cost ceiling:    $/M-URLs-fetched ≤ $X (specific to corpus economics).

Out of scope: real-time freshness (best-effort 30-day re-crawl);
JS rendering; login-walled content; multi-region (single-region).

Example D: Algorithmic (recommender)

Non-functional requirements (3–5 most relevant):

  Online (scoring):
    1. Latency:       p99 ≤ 50 ms server-side per request.
    2. Throughput:    100k scoring QPS peak.
  Offline (training):
    3. Quality:       NDCG@10 ≥ 0.42 on held-out; auto-rollback if regresses
                      > 1% over 24 h shadow.
    4. Training cadence: daily; on-demand re-train completes ≤ 6 h.
    5. Drift:         daily eval; page on-call if drift score > threshold
                      for 3 consecutive days.

Out of scope: explainability (not regulatory in this domain);
fairness audits (separate workstream); cold-start (popularity fallback).

Anti-Patterns

  • Vague NFR. "The system should be fast / scalable / reliable." Always quantify; always identify the part of the system the NFR applies to.
  • NFR as mean rather than percentile. "Average latency 100 ms" hides the tail. Always state percentiles, especially p99.
  • One NFR for the whole system. Different datasets / paths have different NFRs. Read path vs write path vs background path almost always have different latency / durability targets.
  • Wrong NFR vocabulary for the class. Picking "DAU" as a scalability NFR for a distributed cache (callers are services, not users); picking "user-perceived freshness" for a backup pipeline (no user); picking "training cadence" for an OLTP system (no training). Walk the per-class additions for the right vocabulary.
  • Forgetting completeness (pipelines). Pipelines fail silently when nobody asked "what fraction of input must be processed?"
  • Forgetting per-key throughput limit (infra). Without it, the primitive's hot-key behavior is undefined and callers can melt the cluster with one popular key.
  • Forgetting quality / drift (algorithmic). A model with no quality NFR has no rollback criterion — it's a deploy-and-pray system.
  • NFR list of 10+ items. Over-clarification. Pick 3–5 that gate iterations; mark the rest out-of-scope.
  • Skipping the out-of-scope list. Without it, every NFR question from the interviewer surprises you.
  • NFR that doesn't drive a design choice. If a quantified NFR doesn't gate any iteration, it's noise — drop it.
  • Treating CAP as system-wide rather than per-dataset. Real systems mix CP and AP per dataset.
  • RPO / RTO as adjectives. "High durability, fast recovery" isn't an NFR. RPO = X seconds; RTO = Y minutes.
