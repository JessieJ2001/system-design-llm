Bottlenecks

The canonical catalog of bottlenecks a distributed system can hit — what each looks like in numbers, which non-functional requirement (NFR) it violates, and the first techniques you reach for. This is the source-of-truth page; scenario pages diagnose against this taxonomy instead of redefining bottlenecks.

~15 min reference

How to Use This Page

Staff+ system design follows one program:

  1. Start from the simplest design that meets the functional requirements. One process, one DB, no cache, no shard. If functional requirements are satisfied, stop here.
  2. Walk the non-functional requirements (NFRs) from S0. For each NFR — latency, throughput, cost, availability, consistency, durability, cardinality — ask: "does the current design meet this?" If a number says no, that gap is the bottleneck.
  3. Match the gap to a canonical bottleneck below. The catalog gives you the first move to reach for and the cliff at which that move runs out.
  4. Apply the move; recompute the numbers; return to step 2. Iterate until every NFR is satisfied.

This is the no-jumps rule made operational: every iteration in a scenario page traces to this NFR is failing by this much, and every move traces to this is the canonical first lever for that bottleneck shape. If you cannot name the NFR your iteration is closing, you are feature-driven, not bottleneck-driven.

Use this page in two ways:

  • While clarifying requirements (S0). The NFR-class column tells you which bottlenecks each NFR could surface. If your S0 names a 100 ms p99 SLO and an unbounded-cardinality stream, expect tail-latency and cardinality-explosion to bite first.
  • While iterating (S5). The signal column is the diagnostic — per-shard QPS skewed > 5× median for > 5 min classifies as hot partition; the entry tells you what to reach for and what number says you've run out.

Other pages on this site cite here. Do not redefine bottlenecks in scenario pages — link instead.

Per-Bottleneck Detail

Each entry: signal (with numbers; how you diagnose it), mechanism (why it happens), first / second / last-resort moves, anti-moves (what looks reasonable but doesn't fix it), and scenarios where this is the dominant bottleneck.

Compute-bound

  • Signal. CPU or GPU at 100% utilization with FLOP/s near the silicon's peak (e.g., 312 TFLOP/s on H100 BF16 at 40% MFU = 125 TFLOP/s sustained). Adding more memory bandwidth or more disk does nothing; only more compute or less compute-per-op helps.
  • Mechanism. The work itself is arithmetic-bound. Examples: prefill in LLM inference, dense matrix multiply, regex evaluation, hash computation in dedup, video encoding.
  • First move. Vectorize / SIMD; batch to amortize fixed setup; cache the result if the input is reused; horizontal scale with stateless workers.
  • Second move. Reduce work per op — quantize models (BF16 → INT8), prune, distill, approximate.
  • Last resort. Different hardware (GPU → ASIC, CPU → GPU) or different algorithm with lower complexity class.
  • Anti-move. Adding a cache when the cache miss is the compute. Adding more memory when the bottleneck is FLOPs.
  • Scenarios. 34-llm-inference-serving (prefill).

Memory-bandwidth-bound

  • Signal. Bytes-read/s near DRAM bandwidth (≈ 100 GB/s per CPU socket) or HBM bandwidth (≈ 3.35 TB/s on H100). CPU/GPU idle while waiting for memory loads. The classic ratio: arithmetic intensity (FLOP per byte) below the hardware's roofline.
  • Mechanism. Each operation reads a working set from memory once or more per iteration. Examples: LLM decode (autoregressive — every new token re-reads model weights), large hash-table probes, columnar scans.
  • First move. Reduce working-set bytes per op — quantize, prune, compress in memory, or move data closer (L2/L3 cache, HBM vs HBM-far, NUMA-local).
  • Second move. Increase arithmetic intensity — batch operations so more FLOPs share each loaded byte (e.g., decode batch > 1).
  • Last resort. Switch to higher-bandwidth memory tier (DDR → HBM); or change the algorithm to one with higher arithmetic intensity.
  • Anti-move. Adding more cores when the bottleneck is shared memory bandwidth — they all stall on the same channel.
  • Scenarios. 34-llm-inference-serving (decode).

Memory-capacity-bound

  • Signal. Working set > RAM capacity. Symptoms: high page-fault rate, eviction churn, p99 spikes during background swaps, hit ratio of the eviction policy below break-even.
  • Mechanism. The per-instance state grows faster than RAM can hold. Examples: KV cache for many concurrent LLM requests, in-memory feature store with billions of entries, deque per metric series at 200M series.
  • First move. Tier to a slower-but-larger medium (NVMe-backed cache like CacheLib; Redis with disk overflow). Shrink the per-entry representation (compact serialization, delta encoding).
  • Second move. Shard the state across more nodes (the sum of per-node RAM grows with the cluster).
  • Last resort. Move to a streaming / sketched representation that bounds memory regardless of input size — see cardinality explosion.
  • Anti-move. Bigger eviction-policy heuristics on a working set that is fundamentally larger than RAM — no policy beats "actually have the data."
  • Scenarios. 21-distributed-cache (hot-key mitigation is its Iteration 4).

Disk-IO-bound

  • Signal. IOPS or MB/s saturated on the persistent device; iostat shows queue depth growing; sync writes block. Random IOPS for HDD ≈ 100; SATA SSD ≈ 100k; NVMe ≈ 1M. Sequential bandwidth: HDD ≈ 200 MB/s; NVMe ≈ 7 GB/s.
  • Mechanism. Workload pattern doesn't match device strengths. Random reads on HDD; small synchronous writes on any device; un-batched fsync.
  • First move. Convert to sequential writes (LSM with WAL); batch fsync; use NVMe for random IO; precompute reads (materialized views).
  • Second move. Tier hot data to faster storage; cold data to object store / archive.
  • Last resort. Asynchronous durability with separate persistence layer (Kafka in front of DB).
  • Anti-move. Adding indexes to "speed up" reads when the bottleneck is the resulting write amplification — see index bloat.
  • Scenarios. 30-logging-system.

Network-bandwidth-bound

  • Signal. NIC at line rate (10 / 25 / 100 Gbps depending on instance class). Cross-AZ or cross-region egress dominates the cloud bill ($0.01–$0.09/GB depending on path).
  • Mechanism. Bytes-on-wire exceed link capacity, or cross-AZ traffic is priced linearly and large.
  • First move. Compress (Zstd / LZ4); co-locate communicating tiers in the same AZ; reduce fanout (gossip vs broadcast, tree-based aggregation).
  • Second move. Move computation to the data instead of the data to the computation (push-down filters, server-side aggregation).
  • Last resort. Provision higher-bandwidth instances; private network paths (AWS PrivateLink, Direct Connect).
  • Anti-move. Caching at the wrong tier — caching downstream of the bottleneck NIC doesn't help.
  • Scenarios. 10-fan-out-notification (broadcast egress).

Network-RTT-bound

  • Signal. p50 latency dominated by k × RTT (intra-DC ≈ 0.5 ms, intra-region ≈ 1–2 ms, cross-region 60–200 ms). CPU and disk idle waiting for the network. Reducing payload size doesn't help; reducing the number of round-trips does.
  • Mechanism. Each request needs N sequential round-trips (N-tier deep call, chatty protocol).
  • First move. Pipeline / batch — issue the next request without waiting; combine into one. Move data closer (read replica in caller's region, edge cache, CDN).
  • Second move. Reduce round-trips per logical request (fewer service hops, denormalize so one query suffices).
  • Last resort. Speculative execution — issue several variants in parallel, take the first to return — at the cost of extra origin load.
  • Anti-move. Vertical scaling — faster CPUs don't shorten the speed of light.
  • Scenarios. 20-distributed-rate-limiter (RTT to centralized counter).

Lock contention

  • Signal. Latency rises with concurrency rather than throughput rising. CPU usage flat while QPS climbs; profiler shows time spent in futex_wait / lock acquisition; one critical section serialized.
  • Mechanism. Mutual exclusion forces serialization regardless of how much hardware is available.
  • First move. Shard the lock — partition by key so different keys don't contend. Per-thread local state with periodic merge.
  • Second move. Lock-free structures (CAS-based counters, RCU, lock-free queues).
  • Last resort. Re-architect to remove the shared mutable state — append-only log; CRDT; per-actor mailbox.
  • Anti-move. Optimizing the work inside the critical section — even an O(1) critical section serializes throughput at 1/(critical-section-time). Shrinking it 10× only buys you 10×.

Coordinator bottleneck

  • Signal. A single leader, sequencer, scheduler, or metadata service at 100%. The whole system's throughput is pegged to this one component's capacity, regardless of how many workers exist.
  • Mechanism. Some operation requires global ordering / global view (transaction sequencer, ID allocator, scheduler placement).
  • First move. Put the coordinator on the metadata path only, not the data path (e.g., consensus only on shard placement; data writes go directly to shards). Lease-based delegation: coordinator hands out leases for ranges of work that workers execute autonomously.
  • Second move. Partition the coordinator itself (per-tenant sequencer, per-region leader).
  • Last resort. Eliminate the coordinator — switch to leaderless / CRDT-based coordination where order isn't strictly required.
  • Anti-move. Horizontal-scaling stateless workers behind a single coordinator — the workers idle waiting for the coordinator.
  • Scenarios. 23-job-scheduler (single scheduler), 29-resource-scheduler.

Connection saturation

  • Signal. Caller pods × backend pods > 10⁶ TCP connections. Symptoms: file-descriptor exhaustion, TLS-handshake CPU dominating backend, connection pool starvation timeouts.
  • Mechanism. Every caller maintains a connection (or a pool) to every backend. At fleet scale this is an N×M matrix.
  • First move. Deterministic subsetting — each caller talks to a subset of size k of the backend pool; subset chosen by a hash that's stable under churn (rendezvous hashing or a Maglev-style table).
  • Second move. Connection multiplexing (HTTP/2, gRPC streams) so one connection carries many requests.
  • Last resort. Service mesh sidecar that pools on behalf of the application.
  • Anti-move. Random per-request connection — adds TLS handshake latency to every call.
  • Scenarios. Backend-picker, sidecar fleets, microservices meshes (canonical: 40-experiment-bucket-router).

Hot key

  • Signal. One key (or a small handful) at 100× the median QPS. One node pegged at 100% CPU regardless of cluster size; adding nodes doesn't help because the hot key still lands on one.
  • Mechanism. Power-law / Zipfian access — a celebrity user, a viral post, a popular product on a flash sale. See access-patterns#power-law.
  • First move. Client-side L1 cache with short TTL; single-flight on miss to coalesce concurrent fetches.
  • Second move. Replicate the hot key with key suffixing (key#salt0..saltN-1). Writes fan out to all N replicas; reads route to one of N. Trade write amp for read scale.
  • Last resort. Move the hot key to a dedicated tier or cell; isolate from the rest of the cluster.
  • Anti-move. More shards — the hot key still lands on one.
  • Scenarios. 21-distributed-cache (hot-key iteration), 24-realtime-leaderboard, 41-home-timeline-fanout (celebrity).

Hot partition

  • Signal. Per-shard QPS or storage skewed > 5× the median for > 5 minutes. Often caused by a partition key correlated with insertion time (date prefix) or with a popular tenant.
  • Mechanism. The partition key concentrates load. A range partition on created_at means every new write hits the same shard.
  • First move. Salt the partition key with a short hash prefix ({hash(key)%N}#key). For range partitions, auto-split when a partition exceeds a size or QPS threshold.
  • Second move. Choose a different partition key — one with naturally higher cardinality (user_id, request_id) — and accept the loss of co-located scans.
  • Last resort. Move the hot range to a dedicated keyspace or tier.
  • Anti-move. Just adding nodes — uniform sharding doesn't help if the key distribution is skewed.
  • Scenarios. 02-sharding-vs-replication, 18-hotspot-hashing, 44-object-store-metadata (hot prefixes).

Fanout asymmetry

  • Signal. Top-percentile fanout count is orders of magnitude above the median (one user has 200M followers; one notification has 100k recipients). A uniform fan-out-on-write design makes the celebrity case a single-write meltdown.
  • Mechanism. Two correct designs exist for the same problem — push (write to N inboxes) and pull (read by scanning N sources) — and each is wrong for the other end of the distribution. Push wastes work for cold readers; pull melts on celebrity writes.
  • First move. Hybrid: push for normals, pull for celebrities (or the inverse — push for active readers, pull for inactive).
  • Second move. Time-bucket the materialization — fan out once per hour to a per-user inbox of recent items; recompute on read for older.
  • Last resort. Differential prefetch — predict who will read what, and pre-materialize only that intersection.
  • Anti-move. Picking only push or only pull; the wrong end of the distribution always melts.
  • Scenarios. 41-home-timeline-fanout, 10-fan-out-notification.

Tail-latency amplification

  • Signal. A request makes N parallel sub-calls and waits for all (or many) to return. Even if each sub-call has p99 of 10 ms, the system's p99 is the max of N samples, which approaches p99.99 of one. For N = 100, system p99 ≈ component p99.99.
  • Mechanism. The probability that at least one of N independent calls is in the slow tail rises with N: P(any slow) ≈ 1 − (1 − p)N.
  • First move. Hedged / speculative requests — issue the slow sub-call to a second backend after a short deadline, take the first response. Reduces system p99 toward component p95 at ~5–10% extra load.
  • Second move. Reduce N — fewer fan-out branches, denormalize, batch.
  • Last resort. Bound N by quantile — accept partial results past a deadline (return what came back; mark the rest as missing).
  • Anti-move. Reducing component p99 alone — the curve flattens; you can't beat the multiplication.
  • Scenarios. 43-backup-dedup-storage (parallel chunk fetches), 41-home-timeline-fanout (timeline assembly).

Backlog growth

  • Signal. Producer rate > consumer rate, sustained. Queue depth (Kafka lag, SQS depth, in-memory queue length) growing monotonically. End-to-end latency = queue-depth / consume-rate, which goes to infinity if untreated.
  • Mechanism. A consumer that can't keep up turns a queue into a memory leak with extra steps. The system looks healthy (no errors) until it runs out of buffer.
  • First move. Backpressure — propagate "slow down" upstream (HTTP 429, gRPC RESOURCE_EXHAUSTED, TCP window). Shed by priority — drop low-priority requests at the ingress, never silently buffer them.
  • Second move. Scale the consumer (auto-scale on queue depth, not on CPU). Parallelize the consumer (more partitions or more workers per partition).
  • Last resort. Sample at ingress — accept a fraction (head sampling) and drop the rest, with clearly labeled SLO impact.
  • Anti-move. "Just add more queue capacity" — bigger buffer = bigger latency at saturation, not fewer drops.
  • Scenarios. 13-backpressure, 30-logging-system.

Stampede / thundering herd

  • Signal. A cache key's TTL expires; in the next millisecond, N concurrent readers all miss simultaneously and stampede the origin. Origin QPS for that key spikes from 1/TTL to N × QPSread. Origin melts; cache stays cold; readers retry; pile-up cascades.
  • Mechanism. Synchronization on a common expiry boundary. Easy to create accidentally by setting all entries to the same TTL.
  • First move. Single-flight on miss — only the first miss fetches; the rest wait or get stale-while-revalidate.
  • Second move. Jittered TTL — randomize ±20% so expiries don't synchronize. Refresh-ahead — refresh in background before TTL expires.
  • Last resort. Request coalescing at a higher tier (Varnish, edge cache) so the origin only ever sees one request per (key, time-window).
  • Anti-move. "Add more origin capacity" — provisioning for a 1000× spike when the spike is preventable.
  • Scenarios. 27-thundering-herd, 21-distributed-cache.

Retry storm

  • Signal. A failure causes retries from N callers; retries themselves overload, causing more failures, causing more retries. Load grows during the outage instead of shrinking.
  • Mechanism. Retry is amplification under failure. If 1% of calls fail and each is retried 3×, total load is 1.03× — fine. If 50% fail, total load is 2.5× — failure begets failure.
  • First move. Exponential backoff with full jitter; per-caller retry budget (max 10% of original calls); circuit breaker that fails fast when error rate crosses a threshold.
  • Second move. Idempotency keys so the server can dedupe retries server-side; let the client retry without amplifying the work.
  • Last resort. Adaptive concurrency limits (Netflix concurrency-limits, AIMD) — clients reduce concurrency when latency rises.
  • Anti-move. Retrying with no budget; retrying on 5xx without classifying which 5xx are retryable.
  • Scenarios. 26-cascading-failure.

Cardinality explosion

  • Signal. Per-key state × N grows unboundedly with the input stream. Examples: a hashmap of error fingerprints for dedup, a per-user counter for rate limit, a per-(service, endpoint, region) histogram for percentiles. Memory grows linearly with input cardinality, which itself grows with time.
  • Mechanism. The exact-state design is correct at small N but doesn't have a memory bound at large N.
  • First move. Probabilistic sketch: CMS, HLL, t-digest, DDSketch, Bloom. Memory becomes O(1/ε) instead of O(N), at the cost of bounded relative error.
  • Second move. Cap N with eviction (LRU on rare keys); sample at ingress (head sampling preserves overall distribution).
  • Last resort. Tiered: exact state for the top-K hot keys, sketch for the long tail.
  • Anti-move. "Just add more RAM" — the input grows faster.
  • Scenarios. 15-topk-frequent, 36-streaming-percentiles, 42-error-aggregation-dedupe.

Index bloat / write amplification

  • Signal. Each write touches K secondary indexes; per-row write IOPS = K × insertions. Index size approaches or exceeds data size. Write throughput plateaus despite headroom on the primary table.
  • Mechanism. Every secondary index is a parallel structure that must be kept in sync on every write. Index update is the dominant cost when K is high.
  • First move. Drop indexes that don't earn their cost — measure query frequency vs index size. Push rare-query paths to a separate read-optimized store (Elasticsearch, columnar replica) instead of a per-row index.
  • Second move. Async / eventual index build — write to the primary table synchronously, build the index asynchronously from a CDC stream. Trades index freshness for write throughput.
  • Last resort. Switch the storage engine — LSM optimizes writes at the cost of read amp; columnar optimizes scans at the cost of point-lookup latency.
  • Anti-move. Adding "indexes for completeness" — every index has a cost paid on every write.
  • Scenarios. 25-slow-query, 44-object-store-metadata.

Cold start / churn cost

  • Signal. Per-instance warm-up time T (seconds to minutes). Churn rate (rolling deploys, autoscaling, spot interruption) such that the cluster is in warm-up state > X% of the time. Aggregate effective capacity = (steady-state capacity) × (1 − warm-up-fraction).
  • Mechanism. A new instance can't serve at full rate immediately — it needs to load model weights, populate caches, build local indexes, warm JIT, establish connection pools.
  • First move. Pre-warm via shadow traffic before cutover — the new instance receives mirrored production load until its caches and pools are warm.
  • Second move. Longer-lived instances; reduce churn rate (slower rolling deploys, fewer autoscale events with deeper buffer).
  • Last resort. Tiered cache that absorbs cold-start (a shared L2 in front of cold instances).
  • Anti-move. Aggressive autoscaling that spins instances up and down faster than warm-up — more instances make latency worse until they warm.
  • Scenarios. 21-distributed-cache, 34-llm-inference-serving (model load).

Cross-region latency / consistency

  • Signal. p50 latency includes ≥ 1 cross-region RTT (US-EU ≈ 80 ms, US-AP ≈ 150 ms, EU-AP ≈ 200 ms). Synchronous strong-consistency replication forces every write to wait for cross-region quorum, blowing the latency SLO.
  • Mechanism. The speed of light. A write committed to N regions synchronously must traverse the slowest path; reads-your-writes across regions force the read to wait for replication.
  • First move. Per-dataset CAP choice. For most user-facing data: AP with bounded staleness (async replication, 1–10 s lag). For the small slice that needs CP: synchronous quorum, accept the latency, place users near the leader.
  • Second move. Per-region writes with conflict resolution (CRDT, last-writer-wins with logical clocks, or per-record region-of-truth).
  • Last resort. Single-region with read replicas elsewhere — accept that writes are far from non-home users.
  • Anti-move. Default to "multi-region active-active strong consistency" without a per-dataset CAP audit. Some datasets cannot afford it; treating them all uniformly forces the worst latency on every dataset.
  • Scenarios. Multi-region pages (TBD — first canonical to be authored).

Precedence: Which Bottleneck Do You Fix First?

Real systems hit several at once. Three rules of thumb for ordering:

  1. Fix the one that violates the most-load-bearing NFR. If the latency SLO is the gating contract (user-facing real-time UI), fix latency bottlenecks before cost bottlenecks. If the cost SLO is the gating contract (a budget-bound batch system), fix cost first.
  2. Fix the bottleneck whose move also reveals downstream bottlenecks. Adding a queue (for backlog growth) reveals the consumer-throughput bottleneck that was hidden by the synchronous chain. Adding a cache (for read latency) reveals the stampede bottleneck that didn't exist before. Sequence so you don't repeatedly redesign.
  3. Fix the bottleneck whose not-fixing breaks the system, before the one whose not-fixing degrades it. Backlog growth eventually fails (out of memory). Hot key degrades (one node slow but functional). Backlog first.

Common multi-bottleneck patterns and the canonical order:

  • Read-heavy power-law. Hot key first (single-flight + replication), then memory-capacity (tier the long tail), then stampede (jitter + refresh-ahead). Solving hot-key first is the unblocker.
  • Write-burst with strict latency. Backlog first (queue + backpressure), then disk-IO (LSM + batched fsync), then index amp (drop or async).
  • Multi-region active-active. Cross-region consistency first (per-dataset CAP audit), then cold-start in standby region, then fanout (region-local materializations).
  • Streaming aggregation at scale. Cardinality first (sketch), then hot-partition (key salting), then tail-latency in the merge step.
