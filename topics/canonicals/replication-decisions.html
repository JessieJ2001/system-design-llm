Replication & Consistency Decisions

Single-leader vs multi-leader vs leaderless × sync vs async vs quorum × per-key vs cross-key. The decision matrix that tells you which topology survives your consistency requirement, and what it costs. This is the source-of-truth page; scenarios cite this when picking a replication topology.

~12 min reference

The Three Axes

Replication topology is a choice across three orthogonal axes. Pick one value per axis; the combination determines the consistency guarantees you can offer and the failure modes you accept.

Axis 1: Topology — who can write?

  • Single-leader. One node accepts writes; followers replicate. Simplest model; clearest consistency story.
  • Multi-leader. Multiple nodes accept writes; conflicts resolved at merge. Used for cross-region active-active.
  • Leaderless. Any node accepts writes; consistency via quorums + read-repair. Cassandra / Dynamo style.

Axis 2: Synchrony — when does the write return success?

  • Synchronous. Wait for all replicas (or a quorum) before ack. Strong durability; high write latency.
  • Asynchronous. Ack after the leader commits; replicas catch up later. Low latency; potential data loss on leader failure.
  • Quorum. Wait for some subset (R + W > N) before ack. Tunable durability vs latency.

Axis 3: Per-key vs cross-key consistency

  • Per-key strong. A single key's reads/writes are linearizable; cross-key has no joint ordering. Most NoSQL systems.
  • Cross-key (transactional) strong. Multi-key writes are serializable. Spanner, CockroachDB, classic RDBMSes.

Most discussions conflate these axes. Be explicit about all three.

Per-Topology Detail

Single-leader, async replication

  • Mechanism. One leader per shard. Writes go to leader, which commits locally (WAL) and acks the client. A background process ships the WAL to followers. Followers apply asynchronously.
  • Wins. Lowest write latency (only leader-commit on the path). Simplest model — one writable node, clearest "where does the truth live" story. Read scaling for free (followers serve reads). Operationally well-understood — every modern OLTP DB ships with this mode.
  • Loses. Writes that haven't replicated yet are lost on leader crash (RPO > 0). Reads-from-followers are eventual (lag from ms to seconds). Failover requires new-leader election; the new leader may not have all writes. Single leader is the write-throughput ceiling per shard.
  • Pick when. Read-heavy; some write loss on rare leader crash is acceptable; you want simplicity over strong durability; and you want read-replica fan-out for scaling reads.
  • Real implementations. MySQL master/slave, Postgres streaming replication, MongoDB primary/secondaries (default), Redis async replication. Most "managed DB with read replicas" services.
  • Surfaces. leader-loss (RPO > 0); replica divergence on long lag; coordinator bottleneck if writes saturate one leader.

Single-leader, synchronous (or quorum) replication

  • Mechanism. Leader waits for at least one replica (sync) or a quorum (semi-sync; e.g., 2-of-3 acked) before returning success. WAL streamed synchronously to replicas.
  • Wins. RPO = 0 within the quorum: no committed write is lost on single-node failure. Failover is clean — new leader elected from up-to-date replicas; no data loss. Linearizable reads from leader. Quorum reads survive any single replica being stale.
  • Loses. Higher write latency (must wait for at least one replica RTT — typ. 2× single-leader async). Writes block when quorum is unreachable (CP behavior under partition); application must handle write failures. Write throughput limited by slowest in-quorum replica.
  • Pick when. No write loss tolerated (financial, transactional, audit); willing to pay the latency cost; willing to block on partition rather than serve stale.
  • Real implementations. Postgres synchronous_standby_names, Aurora (sync to 4-of-6 storage nodes), most consensus-backed systems (Raft within a shard: Spanner, CockroachDB, etcd, FoundationDB).
  • Surfaces. network partition blocks writes (CP); network-RTT-bound writes when replicas are far apart.

Multi-leader, async (per-region active-active)

  • Mechanism. Each region has its own leader. Writes commit locally and replicate asynchronously to other regions. Conflicts on the same key from different regions are resolved by:
    • Last-writer-wins (LWW) — newest timestamp wins. Simple; loses writes silently.
    • Per-record region-of-truth — each record assigned to one region; cross-region updates rejected or routed.
    • CRDT — data structure that converges deterministically (counters, sets, OR-set, etc.). No data loss; restricted to merge-friendly types.
    • Application-merge — surface the conflict to the app for resolution. Powerful, complex.
  • Wins. Local write latency in every region (no cross-region wait). Survives entire-region loss with bounded data loss (RPO = replication lag). User-perceived latency is low globally. Each region operates independently during partition.
  • Loses. Cross-region eventual consistency only — bounded staleness, typ. seconds. Conflicts on partition heal must be resolved (LWW silently loses; CRDT restricts data shapes; app-merge is complex). Cross-region read-your-writes is hard. Operationally complex (each conflict-resolution strategy has its own bugs).
  • Pick when. Multi-region active-active; latency-sensitive writes from many regions; per-dataset CAP audit says AP with bounded staleness is acceptable for this dataset.
  • Real implementations. DynamoDB Global Tables (LWW); Cassandra cross-DC replication; Riak (vector clocks + app-merge); Cosmos DB; CockroachDB / Spanner partial multi-leader within Raft groups.
  • Surfaces. network partition with divergent writes; replica divergence; clock skew matters for LWW.

Leaderless, quorum-based

  • Mechanism. Any of N replicas can accept writes. A write succeeds when W replicas ack (W ≤ N); a read queries R replicas and reconciles (R ≤ N). With R + W > N, every read sees at least one replica that saw the latest write. Read-repair on conflict; periodic anti-entropy scrubbing closes drift.
  • Wins. No leader = no leader-loss outage. Tunable consistency: pick W and R per workload (e.g., W=1 for max-availability writes; W=N for max-durability). Naturally horizontally scalable. Sloppy quorum (hinted handoff) keeps writes accepted during partition.
  • Loses. No global write ordering — concurrent writes to the same key need version vectors / vector clocks to merge. Read amp on quorum reads (R replicas per read). Read-repair adds latency variance. Application sometimes sees temporary inconsistency before reconciliation.
  • Pick when. Write-heavy with no single hotspot per key; tolerable to lose strict ordering across keys; want availability over linearizability; AP is the right CAP choice for the dataset.
  • Real implementations. Cassandra (per-key), DynamoDB (per-key), Riak, ScyllaDB. The Dynamo paper's classic model.
  • Surfaces. replica divergence handled by read-repair; tail latency on R-of-N quorum reads.

Chain replication

  • Mechanism. Replicas arranged in a chain: head accepts writes, propagates to middle nodes, tail acks. Reads come from the tail (so reads see only fully-replicated writes).
  • Wins. Strong consistency without full Paxos complexity — the chain order itself defines the commit order. Tail-reads are always up-to-date and don't need quorum coordination. Operationally simpler than Paxos for some workloads.
  • Loses. Highest write latency (full chain traversal — N nodes for N-replica chain). Any chain-member death stalls writes until the chain is reformed (typ. seconds; reformation needs a consensus configuration service). Less battle-tested than Raft/Paxos in production.
  • Pick when. Want strong consistency with a simpler invariant than full consensus; chain-length latency is acceptable; the workload has a configuration service available for reformations. Rarely the right choice for new systems vs Raft.
  • Real implementations. Some research systems; Microsoft FaRM; certain internal Dropbox / AWS systems. Not common as a default for new builds.
  • Surfaces. node-loss stalls until reformation.

Consistency Models

Each topology supports different consistency models. State the model your design offers — vague terms hide gaps.

ModelWhat it guaranteesTypical topology
Linearizable (strong)Operations appear instantaneous; global total order; any read after a write sees the writeSingle-leader sync; consensus per key (Raft, Paxos)
SequentialAll operations appear in some total order; not necessarily real-timeSingle-leader async (reads from leader)
CausalCausally-related operations seen in order; concurrent ops may be seen in any orderVector clocks; CRDTs; multi-leader with HLC
Read-your-writes (RYW)A client's reads see its own previous writesSticky session to leader, or write-cookie tracking
Monotonic readsA client's reads don't go backward in timeSticky session to one replica
Bounded stalenessReads at most T seconds (or N versions) staleAsync replication with monitoring; capped lag
EventualIf writes stop, replicas convergeAsync replication; gossip; CRDTs

Per-dataset CAP, not per-system

Real systems mix consistency models per dataset. "Inventory: linearizable per-key (CP). User profile: bounded-staleness ≤ 5s (AP). Social-graph cache: eventual (AP)." Don't pick one CAP setting for the whole system; per-dataset is the staff+ move.

Decision Heuristics

  1. How much write loss is tolerated? Zero → sync. Any → async OK.
  2. Is RYW required across regions? Yes → single-leader (writes to home region) or per-record region-of-truth. No → multi-leader async with conflict resolution.
  3. Are there cross-key transactions? Yes → cross-key strong (Spanner / CockroachDB / RDBMS). No → per-key strong is enough (Cassandra, DynamoDB).
  4. Is global linearizability required? Yes → single-leader sync per shard, plus consensus for shard placement. No → leaderless quorum or multi-leader.
  5. What's the partition behavior? CP-on-partition → block writes (single-leader sync, Spanner). AP-on-partition → accept divergence + reconcile (multi-leader async, Cassandra, DynamoDB).

Common Combinations

  • Postgres / MySQL with read replicas. Single-leader, async. RYW from primary, eventual from replicas. The default OLTP shape.
  • Aurora / Cloud Spanner. Single-leader (per shard), sync. Strong consistency; higher write latency. The default for managed transactional systems.
  • Cassandra / DynamoDB. Leaderless (per-key), tunable quorum. AP by default; CP achievable with quorum-style settings. Default for write-heavy NoSQL.
  • CockroachDB / Spanner. Per-shard Raft (single-leader sync within Raft group); cross-shard transactions via 2PC over Raft groups. Cross-key strong consistency at scale.
  • DynamoDB Global Tables. Multi-leader async, LWW conflict resolution. Per-region writes; eventual cross-region.
  • Kafka. Single-leader (per partition), sync to in-sync replicas. Strong durability per partition; no cross-partition ordering.
  • Redis. Single-leader, async (default). RYW from primary; replicas best-effort. Cluster mode shards by key.

Anti-Patterns

  • "Eventually consistent" without bounded staleness. Eventual is "if writes stop, replicas converge" — but writes don't stop. Quantify the bound: "≤ 5 s replication lag at p99."
  • Linearizable across regions without measuring write latency. Cross-region consensus adds 60–200 ms per write. If your write SLO is < that, you can't have linearizable writes across regions.
  • Multi-leader with LWW for non-LWW-friendly data. "Last-writer-wins" silently loses concurrent updates. For counters, use CRDTs. For records, prefer per-record region-of-truth or app-merge.
  • Read replicas as a substitute for cache. Read replicas have minutes of replication lag at high write loads; they are also expensive (full-instance cost). Use a cache for hot reads; replicas for read scaling and failover.
  • Treating Cassandra as ACID. Cassandra offers per-key consistency at best; cross-key transactions require lightweight transactions (LWT) which are slow.
  • One CAP setting for the whole system. Per-dataset CAP audit is the staff+ move; system-wide CAP is the senior move.
  • Sync replication everywhere. Pays write latency for datasets that don't need it. Match synchrony to durability requirement per dataset.
