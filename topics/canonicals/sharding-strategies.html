Sharding Strategies

Hash, range, geo, lookup-table, hybrid — what each is good at, what breaks each, rebalancing cost, hot-prefix risk. This is the source-of-truth page; scenarios cite this when picking a partition strategy.

~10 min reference

How to Use This Page

Once you've picked an entity as the partition key (see entities-and-data.html), how you partition by it determines how the cluster scales. Hash and range are the two main families; geo and lookup-table are special cases; hybrid is what real systems do.

Three properties to weigh for any choice: uniformity of load (does it avoid hot partitions?), locality of queries (does the dominant query hit one partition or many?), rebalancing cost (when capacity changes, how much data moves?).

Per-Strategy Detail

Hash sharding

  • Mechanism. shard = hash(key) % N, or — better — shard = lookup_in_consistent_hash_ring(key). Each key lands on one shard deterministically; distribution is uniform if the hash is good.
  • Wins. Uniform load distribution by default. Adding/removing shards moves only ~1/N of keys (with consistent hashing + virtual nodes). Simple to reason about.
  • Loses. No locality — adjacent keys in any natural ordering land on random shards. Range scans must fan out to all N shards. Doesn't help with hot keys (one popular key still pegs one shard).
  • Pick when. Point lookup is the dominant query; range scans are rare; load distribution is the priority.
  • Real implementations. Cassandra, DynamoDB, Memcached / Redis Cluster, Kafka (partitioner), most KV stores.
  • Surfaces. hot-key when one key dominates; cite consistent hashing for ring + vnodes detail.

Range sharding

  • Mechanism. Each shard owns a contiguous key range. Keys are sorted; a new key lands on the shard whose range contains it. Auto-split when a shard exceeds size or QPS threshold.
  • Wins. Range scans are sequential (single-shard for in-range queries). Locality preserved — adjacent keys co-located. LIST / scan APIs return naturally ordered results.
  • Loses. Hot-partition risk is high. Any key prefix correlated with insertion time (created_at, sequence numbers) creates a hot tail. Auto-split helps but lags behind sudden spikes.
  • Pick when. Range scan or LIST is the dominant query (object store metadata, time-series with rolling reads, document stores with sorted access).
  • Real implementations. Bigtable, HBase, Spanner, CockroachDB (range with auto-split), TiKV, S3 metadata.
  • Surfaces. hot-partition on prefix-skewed keys; mitigate by salting the key prefix or sub-dividing.

Geo sharding

  • Mechanism. Partition by user's home region or by data residency. Each region's data lives in that region's cluster; cross-region access is the exception.
  • Wins. Latency: in-region access is ~ms; out-of-region adds the cross-region RTT. Compliance: residency requirements (GDPR, sovereignty) are structurally enforced. Per-region cluster failure has bounded blast radius.
  • Loses. Cross-region queries (a user travels; an admin queries globally) must federate or replicate. Per-region capacity must be sized for that region's peak (vs aggregate peak).
  • Pick when. Multi-region active-active with bounded staleness; residency requirements; latency-sensitive user-facing reads.
  • Real implementations. Per-region partitioning in DynamoDB Global Tables, Cosmos DB, Spanner. Pattern: per-region writes, async replication; conflict resolution per dataset.
  • Surfaces. cross-region latency; data residency (X2).

Lookup-table sharding

  • Mechanism. A table maps key (or key range) to shard explicitly. "keys 1–1000 → shard A; keys 1001–2000 → shard B; tenant_X_keys → shard X." Lookup adds one indirection.
  • Wins. Maximum operator control. Hot keys can be moved manually. Uneven shards (big tenants, premium customers) can be carved out. Rebalancing is just a table update.
  • Loses. The lookup table itself is a coordinator (must be available, fast, and consistent). Operator burden — someone has to manage the table.
  • Pick when. Heterogeneous tenant sizes; manual control over hot-key placement; small N (10s–100s of shards) where the table fits in memory and changes rarely.
  • Real implementations. Vitess (lookup-vindex), some custom multi-tenant systems, Notion's tenant→shard map. Pattern: table cached at every router; updates via gossip or pub-sub.
  • Surfaces. coordinator if the table is on the hot path; mitigate by caching aggressively.

Composite key sharding

  • Mechanism. Partition by one dimension, cluster by another within the partition. partition by user_id, cluster by created_at within partition. Two-level addressing within one engine.
  • Wins. Combines uniform partition distribution (if the partition key is high-cardinality) with locality on the secondary dimension (range scans within partition are cheap).
  • Loses. Range scans across partitions still fan out. Choosing partition vs cluster wrong locks you in (changing the partition key is a full rewrite).
  • Pick when. Most queries are "<partition-key>'s range of <cluster-key>" — e.g., "this user's messages from this week."
  • Real implementations. Cassandra (PARTITION KEY + CLUSTERING KEY), DynamoDB (partition key + sort key), Bigtable (row-key composed of partition + cluster).
  • Surfaces. hot-partition if the partition-key half is skewed; the cluster-key half doesn't help.

Hybrid (per-tier)

  • Mechanism. Different sharding for different tiers of data. Hot tier: range; warm tier: hash; archive: by-time. Or: most data hash-sharded, hot keys carved out into dedicated tier.
  • Wins. Each tier optimizes for its dominant access pattern. Hot keys can be isolated from the rest.
  • Loses. Routing complexity — the system must know which tier each key is in. Migration between tiers is its own subsystem.
  • Pick when. Workload has clearly different tiers (hot/warm/cold) with different access patterns; one strategy doesn't fit all.
  • Real implementations. Many caching systems (L1 hash + L2 hash with different placement); hot-key replication as a hybrid where hot keys get more replicas.

Decision Heuristics

Walk these in order; pick the first that fits.

  1. Is range scan the dominant query? → range. (Object store LIST, time-series rolling reads.)
  2. Are there residency / locality requirements? → geo as the outer layer. Within each region, pick another strategy.
  3. Are tenants heterogeneous (very uneven sizes)? → lookup-table for the largest tenants; hash for the rest.
  4. Is the dominant query "this user's range of this thing"? → composite (partition by user, cluster by time/range).
  5. Otherwise (point lookup dominates; uniform-ish keys) → hash with consistent-hashing + vnodes. Default for most stateful services.

The Hot-Key / Hot-Partition Defense

Whatever strategy you pick, hot keys are still possible (Zipfian access — celebrity user, viral post, popular product). Sharding alone doesn't fix it; specific defenses do:

  • Cache in front. The cache absorbs hot reads; the shard sees only cold-tail traffic. Cite caching.
  • Single-flight on miss. Coalesces the burst when a hot key's cache entry expires. Cite single-flight.
  • Hot-key replication via salting. Replicate the key under key#salt0..saltN-1; route reads round-robin across the replicas; pay write-amp to fan out.
  • Hot-key isolation tier. Move detected hot keys to a dedicated cluster (or memory tier).
  • Hot-key detection. CMS sketch tracks per-key QPS; threshold triggers escalation to one of the above. Cite sketches#cms.

Defense should be layered: cache + single-flight handles most cases; salting / isolation handles persistent skew; detection + escalation handles surprises.

Rebalancing

When capacity changes (add/remove shards), how much data moves?

  • Naive hash (mod N). Adding a shard reshuffles ~all keys. Disastrous at scale. Don't use mod-N at scale.
  • Consistent hashing. Adding a shard moves ~1/N of keys. With virtual nodes, distribution stays uniform. Standard for hash-sharded systems.
  • Range with auto-split. Adding a shard takes some hot range and splits it; minimal data motion (just the split range). Standard for range-sharded systems.
  • Lookup-table. Trivial — update the table. Data motion equals the keys whose mapping changed.
  • Geo. Adding a region is a heavy operation (full data set replication). Adding capacity within a region is per-region rebalancing.

Rebalancing is a planned operation, not a hot-path concern — but its cost determines how aggressively you can autoscale. Hash + consistent-hashing is the most autoscale-friendly.

Anti-Patterns

  • Hash sharding by created_at. Defeats the purpose — hash gives uniformity; using a monotonic field as input gives a predictable distribution that creates hot writes on the latest hash bucket regardless of hashing.
  • Range sharding by user_id. The user-id distribution often has prefix correlations (sequentially-allocated IDs); range over them creates time-correlated hot shards.
  • Mod-N hashing at scale. Adding a shard reshuffles everything. Always use consistent hashing.
  • Geo-sharding without per-dataset CAP audit. Treating all data as "EU stays in EU" without distinguishing per-dataset consistency requirements forces every dataset onto the worst-latency profile.
  • Lookup-table on the hot path without caching. The lookup adds an RPC per request; it must be cached in-process at every router.
  • Choosing strategy before access pattern. "Use hash because that's what we always do" — without checking whether range scans are dominant. Match strategy to query.
  • Forgetting that sharding doesn't fix hot keys. All strategies still leave one key on one shard. Hot-key defense is separate from sharding strategy.
