ML / LLM Serving: Comparison & Tradeoffs

"Use Triton" is fine for vision models; for LLMs the answer is vLLM or TGI — the paged-KV-cache + continuous-batching primitives reshape throughput by 5–20×. The decisive question: are you serving a transformer with a KV cache, or anything else?

6. Migration paths

  • OpenAI API → self-host vLLM: driven by cost / privacy / customization. Use the OpenAI-compatible endpoint vLLM exposes; minimal client change. Plan for ops + on-call.
  • TF Serving → Triton: for ensemble pipelines or multi-framework inference. Convert models to TensorRT or keep as TF/PT.
  • TGI → vLLM: usually for richer features (LoRA, scheduling) and broader hardware support.
  • Custom Flask wrapper → Triton/Ray Serve: the most common modernization. Standardize batching + GPU sharing.
  • Self-host LLM → managed: when ops cost outweighs token-bill savings. Common for small teams.

8. Hybrid patterns

Cascading: small first, big fallback

Try a small model first (fast / cheap); fall back to a large model if confidence low or task hard. Cuts average cost dramatically.

RAG = vector DB + retriever + LLM

Embed user query; retrieve top-K passages from vector DB; stuff into LLM context. Three serving systems coordinated; cache embeddings of common queries.

Speculative decoding

Small "draft" model proposes tokens; big "target" model verifies in batch. 2–3× speedup at the cost of more GPU activity.

LoRA per tenant

One base model, hot-swap LoRA adapters per tenant. vLLM supports this natively. Massive cost win for multi-tenant fine-tuning.

KV-cache reuse for prefix

System prompt / shared context cached across requests. Skip prefill for common prefix. vLLM 0.4+ supports automatic prefix caching.

Embedding cache + LLM

Embedding service answers from cache when query repeats; LLM generates the response. Redis ZSET for embedding-hash lookup.

9. Anti-patterns

"We used TF Serving for an LLM"

No paged KV, no continuous batching. Throughput is 5–20× lower than vLLM. Switch.

"Static batching with bursty traffic"

Latency oscillates between batch fill and timeout. Use continuous batching.

"GPU at 30% utilization"

Per-token cost > 3× what it should be. Concurrent requests + continuous batching push utilization to > 80%.

"Synchronous request to LLM in hot path"

P99 of an LLM is 5–30s. Don't block user. Use streaming + UI feedback.

"Loaded the model fresh per request"

Cold start = minutes. Use a long-running serving process; min-replicas always loaded.

"Used Lambda for LLM inference"

Cold-start is unworkable for LLMs; CPU-only Lambda doesn't have GPU. Use SageMaker / Vertex / vLLM on k8s.

"Forgot prompt caching"

Same system prompt sent N times = wasted prefill compute. vLLM auto-caches; Bedrock has explicit prompt caching.

"Per-token bill exploded"

Output token bill is 2–3× input. Set max_tokens; consider self-host past N tokens/month.

10. Decision rules for interviews

  • Default: managed LLM API (Bedrock / Vertex / Azure OpenAI). Self-host only when cost / privacy / fine-tune demand it.
  • Self-host LLM: vLLM (or TGI). Use the OpenAI-compatible endpoint to keep client code stable.
  • Continuous batching + paged KV cache are mandatory for any real LLM workload.
  • Vision / classifier / multi-model: Triton.
  • k8s-native serving: KServe.
  • For RAG, the bottleneck is usually retrieval latency, not LLM — cache embeddings.
  • Always stream LLM tokens to UI. Time-to-first-token is the perceived latency.
  • For multi-tenant fine-tunes: LoRA hot-swap (vLLM) over per-tenant deployments.
  • Output tokens are 2–3× input price. Set max_tokens.
  • Cascading models (small first, big fallback) cuts average cost.
