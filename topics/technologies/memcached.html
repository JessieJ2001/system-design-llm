Memcached

Multi-threaded, in-memory, ephemeral string-only cache. Slab allocator avoids fragmentation; LRU per slab; no persistence; no native cluster (client-side hashing). The simplest possible "key → bytes" remote cache — that simplicity is the feature.

multi-thread • slab + LRU • pure cache • client-hashed

2. Underlying mechanism

  • One process, multiple I/O threads.
  • Slab allocator: memory carved into slab classes (e.g., 96B / 120B / 144B / ... up to 1MB). Each item lands in the smallest fitting slab. Avoids malloc fragmentation; trades space for speed.
  • LRU per slab class: each slab has its own LRU list. Eviction within a class. Modern Memcached has segmented LRU (HOT / WARM / COLD).
  • Hash table indexes all keys to items.
  • No persistence: restart = empty cache.
  • No native cluster: clients use consistent hashing across N Memcached servers; each server independent.
  • Wire protocol: text or binary; recent versions added meta protocol for richer ops.

3. Read & write path

set / get

set key value [flags] [expires]
1. Hash key → pick I/O thread or shared state under lock
2. Find slab class for value size
3. Allocate from free list / evict LRU tail
4. Store; chain into hash table
5. Return STORED

get key:
1. Hash table lookup
2. Touch LRU (refresh recency)
3. Stream value back

cas (check-and-set)

cas adds a version token
client must send the version they read
server rejects if changed since read
— gives atomic compare-and-set without WATCH-style

incr / decr

Atomic numeric ops on existing keys (string-as-int)
Used for counters in classic web stack

multi-get

Client sends batched gets to one server (one round trip)
Or fan-out to multiple servers via consistent hash
Multi-get pipelining is a big perf win for cache-aside

4. Replication & consistency

  • None native. Each server independent.
  • Pattern: replicate hot keys at the client (write to two servers).
  • Some forks (e.g., MemCachier) have replication.
  • EVCache (Netflix) is a Memcached fork with multi-AZ + replication.
  • Consistency: per-key linearizable on one server; cluster-wide is whatever the client does.

5. Partitioning

  • Client-side consistent hashing (ketama is the canonical algo).
  • Add / remove a server = small fraction of keys remap.
  • No coordination protocol, no cluster bus, no slot resharding tooling.
  • Hot key still pegs one server; replicate hot keys client-side.

6. Time complexity

  • get / set: O(1) average (hash table).
  • incr / decr: O(1).
  • multi-get: O(K) for K keys on one server; client may fan out.

7. Per-node capacity envelope

  • ~200k ops/sec/core (multi-thread); typically 1M+ ops/sec/server with multiple cores.
  • P99 latency: < 1 ms LAN.
  • RAM-bound; up to whatever your box has.
  • Single-server scales linearly with cores up to NIC saturation.

8. When to use / not to

Use Memcached when…

  • Pure ephemeral cache (cache-aside).
  • Max throughput per core matters.
  • Memory only stores opaque bytes (string + flags).
  • You don't need data structures or persistence.
  • Existing infra has it (Facebook, Twitter scale).

Don't use Memcached when…

  • Need data structures (use Redis ZSET, Hash, Stream).
  • Need persistence (use Redis AOF / RDB).
  • Need replication / HA (use Redis or EVCache).
  • Need cluster slot management (use Redis Cluster).
  • Need pub/sub or queues.

9. Common gotchas

  • Slab class fragmentation: certain sizes evict prematurely while others have room. Use slab_reassign or restart.
  • Hot server from skewed key distribution; replicate hot keys client-side.
  • Restart loses cache; expect a thundering herd on cold start.
  • Item size limit: 1 MB by default; raise with -I if needed.
  • Connection limit: -c default 1024.
  • LRU "calf" effect: a flood of new keys evicts hot ones; segmented LRU helps.
  • Memcached returns "STORED" but doesn't fsync — restart loses everything (by design).
  • No native ACL until very recent versions; production usually network-isolated.

10. Cloud equivalents

  • AWS: ElastiCache for Memcached.
  • GCP: Memorystore for Memcached.
  • Azure: (no native; use AKS).
  • Open-source: Memcached itself; mcrouter (FB's proxy); EVCache (Netflix fork).

12. Usage patterns

  • Cache-aside in front of slow DB.
  • Session store (small, ephemeral).
  • Page fragment cache.
  • Object cache (serialized).
  • Counter via incr/decr.
  • Negative cache for 404s.
  • Multi-get for batch fetch.
  • NOT distributed lock (no atomics across servers).
  • NOT durable storage.

13. Operational knobs

  • -m memory size.
  • -t threads.
  • -c connection limit.
  • -I max item size (default 1 MB).
  • -e external SSD (extstore) for spillover.
  • -o no_modern / -o slab_reassign.
  • Segmented LRU on by default in modern versions.

14. Capacity planning

  • Memory: items + key/value × 1.5 (overhead).
  • Threads: ~1 per core.
  • Connections: app concurrency × servers; tune -c.
  • Hot keys: replicate client-side.

15. Failure modes

  • Server crash: data lost; ring shrinks; small fraction of keys remap.
  • Network partition: client-side; just cache miss until reachable.
  • Hot key: one server pegged.
  • OOM: eviction.

16. Backup & DR

  • None; rebuild from source.
  • EVCache pattern: write to multiple AZ pools so single-AZ loss doesn't cold-start the cache.
  • Pre-warm cache on deploy from primary store.

17. Migration paths

  • Memcached → Redis: change client lib; usually trivial. Consider Redis if you'll grow into data structures.
  • Memcached → ElastiCache: lift-and-shift.
  • Add mcrouter in front: gradual; no app change.

18. Security

  • SASL auth (5.0+), but rarely used; production is network-isolated.
  • TLS via --enable-tls in modern builds.
  • Bind to private subnet; never expose 11211 to the internet (UDP amplification attack history).
  • Disable UDP unless absolutely needed.

19. Multi-tenancy

  • None. Run one Memcached fleet per tenant tier.

20. Observability

  • stats command: get/set count, hit ratio, evictions, bytes.
  • stats slabs: per-slab usage and evictions.
  • stats items: per-class item info.
  • Hit ratio is the key SLI — aim > 90% for cache-aside.
  • Eviction rate: spike means too small or hot-keyset shifted.

21. Anti-patterns

  • Treating Memcached as durable storage.
  • Single huge value > 1 MB without raising -I.
  • UDP exposure to internet (amplification attack).
  • Restart on deploy → cache stampede on cold start.
  • One key for all features (no segmentation).
  • No TTL → eviction unpredictable.

22. Cost model

  • RAM-only; cheaper than Redis at the same RAM (no replicas).
  • Cloud (ElastiCache Memcached): per-node-hour by RAM tier.
  • No replication = simpler bill.
