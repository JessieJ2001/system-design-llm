Analytics / OLAP / Time-Series: Comparison & Tradeoffs

"Use BigQuery" might be wrong by 10× on cost. The analytics stack has three distinct shapes — cloud warehouses for ad-hoc queries on huge datasets, real-time OLAP for interactive dashboards on streaming data, and TSDBs for monitoring. Picking the wrong shape is the most common mistake.

6. Migration paths

  • Postgres → warehouse: CDC via Debezium → Kafka → warehouse (BQ subscription, Snowpipe Streaming, Redshift Streaming Ingestion). The OLTP DB is freed from analytics.
  • Redshift → Snowflake: common cloud-portability move. Schema-conversion tools; double-write with replication; cut over BI tool.
  • Snowflake / Redshift → BigQuery: typical when GCP is the new cloud. Storage Transfer or BigQuery Data Transfer; rewrite stored procs (each warehouse has its own dialect).
  • Hadoop / HDFS → cloud warehouse: the most common modernization. Move parquet to S3/GCS; use Athena/BigQuery External Tables for cheap query; warehouse for hot tables.
  • Self-host ClickHouse → ClickHouse Cloud: backup + restore; cluster keys may differ.
  • Prometheus → Mimir / VictoriaMetrics: remote-write the existing Prom; cut Grafana to query the new backend; decommission Prom long-term storage.
  • InfluxDB 1.x → 3.0 (or VictoriaMetrics): 1.x → 3.0 needs export/import; many shops jump to VM instead.

8. Hybrid patterns

Postgres + ClickHouse / BigQuery (OLTP + OLAP)

Postgres for OLTP. CDC replicates to a warehouse / OLAP engine. Reports run there; the OLTP DB stays fast.

Lambda architecture (batch + speed)

Stream → real-time OLAP (Druid/CH) for last 24h; same data → warehouse (BQ) for historical. UI queries both, merges on time boundary. Modern variant: Kappa (stream-only) into a single OLAP store.

Iceberg lakehouse (BigQuery + Iceberg + Trino)

Parquet/Iceberg on object storage; warehouse + Trino + Spark all query it. Shared storage, multiple engines, no lock-in.

Materialized views for hot queries

BQ materialized views, Snowflake auto-refresh views, ClickHouse MaterializedView. Pre-aggregate hot dashboards; query MV instead of raw.

Prometheus + Mimir / Thanos

Prom local for short-term; remote_write into Mimir/Thanos for long-term + global query. Single Grafana sees both.

BigQuery + Cloud Functions (event-driven analytics)

Pub/Sub event → Cloud Function → BQ insert (Storage Write API). Sub-second to query in BQ.

9. Anti-patterns

"BigQuery for sub-second dashboards"

BQ is 1–30s typical; not built for < 1s interactive. Use ClickHouse / Druid for that workload, or BI Engine to accelerate.

"SELECT * on a partitioned table"

Reads all partitions; bypasses partition pruning. Always reference partition column in WHERE.

"Snowflake warehouse never auto-suspends"

$$ all weekend even if idle. Set auto_suspend = 60s.

"Used Prometheus to store user IDs as labels"

Series cardinality blows up; OOM. Labels are for grouping, not unique IDs. Use traces or logs for per-user.

"Tried to UPDATE rows in a real-time OLAP"

Druid / Pinot are append-only; updates require rewrites. Use ReplacingMergeTree in CH if you must.

"Joined two huge tables in ClickHouse"

One side must fit in RAM (broadcast join). Either pre-join in Spark, use Distributed engine + colocated keys, or move to BQ/Snowflake for ad-hoc joins.

"Used ClickHouse as an OLTP for low concurrency"

OK for 100 QPS; falls over at 1000+ on point lookups. Use a real KV / SQL DB.

"Forgot data retention"

Bills grow forever. Set TTL / lifecycle on every table; tier to cold storage.

10. Decision rules for interviews

  • Default for analyst SQL on warehouse data: BigQuery (GCP) / Snowflake (multi-cloud) / Redshift (AWS).
  • Sub-second user-facing analytics: ClickHouse. If multi-tenant + streaming-heavy, Druid/Pinot.
  • Application metrics: Prometheus. Long retention: VictoriaMetrics or Mimir.
  • IoT / time-series: TimescaleDB if relational; InfluxDB if pure TS shop.
  • Always partition + cluster on warehouses. Unpartitioned = expensive.
  • Materialized views are the cheapest concurrency lever.
  • For warehouses, separate OLTP from OLAP via CDC. Don't run analytics on the production DB.
  • Lakehouse (Iceberg on S3) avoids lock-in. Same data, multiple engines.
  • Series cardinality — not data volume — drives Prometheus cost.
  • If "ad-hoc query on huge data + sub-second dashboards" both required, run two systems (warehouse + ClickHouse).
