Datastores: Comparison & Tradeoffs

In an interview the question is never "explain Cassandra" — it is "you said Cassandra, why not DynamoDB or Mongo or Postgres?" Postgres is a fine cache, Redis is a fine queue, Cassandra is a fine time-series DB. Overlap is real. This page helps you answer when each is the best fit, not the only fit.

Postgres • MySQL • Spanner • CockroachDB • Redis • Memcached • DynamoDB • MongoDB • Cassandra • Bigtable / HBase • Neo4j

2. Decision flowchart

The first thing to memorize. In an interview, walk down this tree out loud while sketching it on the board.

Glossary for the Redis-vs-Memcached branch. ZSET = Redis sorted set (members ordered by score; O(log N) range queries; powers leaderboards, top-N feeds, sliding-window rate limiters). Streams = Redis append-only log with consumer groups (XADD/XREAD), a lightweight Kafka-inside-Redis for event buses and job queues. AOF = Append-Only File, Redis's durability mode that fsyncs writes to a log on disk so a crash doesn't lose recent data (alternative: RDB point-in-time snapshots). Memcached has none of these — pure ephemeral string KV, restart wipes everything.

The dual-write → cutover ritual.

  1. Dual-write — write to both old and new on every operation. Old remains source of truth.
  2. Backfill — copy historical rows from old to new. Run idempotently; new must be a strict subset of old until done.
  3. Shadow-read & diff — on every read, query both, compare results, log mismatches. Keep going until mismatch rate is zero.
  4. Cutover — flip the read source to new. Keep writing to old as well (rollback insurance).
  5. Decommission — stop writing to old. Wait. Delete.

Skipping any of these steps is how data loss happens. The diff phase is the longest; budget weeks, not days.

8. Hybrid patterns ("use them together")

The most senior-level part of the page. Real systems rarely use one datastore — they layer. These are the canonical combinations.

Postgres + Redis (cache-aside)

The canonical web stack. Postgres is source of truth; Redis caches hot reads. App writes to PG, then invalidates the Redis key. Reads go to Redis first; on miss, fetch from PG and populate.

Failure modes: stale cache after update (TTL bound); cache stampede on TTL expiry (use single-flight or jittered TTL); Redis down → latency cliff but PG still serves.

Postgres + Elasticsearch (CDC search)

PG holds canonical rows; Debezium tails the WAL; ES indexes the text fields. Searches hit ES; full record fetches go back to PG by ID.

Failure modes: CDC lag = stale search results; ES is not source of truth (re-indexable from PG anytime); schema changes need coordinated mapping update.

DynamoDB + S3 (large items)

DDB has a 400 KB item cap. For larger blobs (videos, PDFs, ML embeddings > few KB), store the blob in S3, store the S3 key in the DDB item. Atomic on the item; eventual on the blob.

Failure modes: orphan blobs (write item, fail to write S3) — reverse the order: write blob first, then the item with the pointer. Cleanup orphan blobs via lifecycle policy + tombstones.

Cassandra + Redis (durable + sub-ms)

Cassandra holds the durable activity log; Redis ZSET holds the leaderboard or recent-N feed. Writes go to both (or via Kafka). Reads for "last 100" hit Redis; older reads page back to Cassandra.

Failure modes: Redis cold-start needs replay from Cassandra; backfill cost.

Postgres + ClickHouse / BigQuery (OLTP + OLAP)

PG runs OLTP. CDC (Debezium) replicates to ClickHouse / BigQuery for analytics. Reports run there; the OLTP DB is freed from heavy aggregations.

Failure modes: CDC lag (analytics is "minutes-stale"); schema evolution must propagate; a bad migration in PG can break the warehouse pipeline.

Mongo + Atlas Search / pgvector + tsvector

Single product, two engines. Mongo Atlas Search runs Lucene alongside the doc store; pgvector + tsvector lets Postgres do vector + full-text without a second system. Tradeoff: simpler ops, but each engine inside the product is less feature-rich than a dedicated one.

9. Anti-patterns ("I picked X but should have picked Y")

The mistakes interviewers love to probe. Each one is a real production incident that has happened many times.

"We used MongoDB as a relational DB"

Joins via $lookup are slow; multi-doc transactions limit you to single-shard at high QPS; ACID guarantees you wanted aren't there. You'll fight the tool until you migrate to Postgres. Lesson: if entities are deeply related, start with SQL.

"We used Cassandra for ad-hoc queries"

You'll reach for ALLOW FILTERING, then for secondary indexes, then realize neither works at scale. Cassandra requires you to model around access patterns up front. Lesson: if access patterns aren't known, use SQL or Mongo.

"We used Redis as the source of truth"

Eviction silently deletes data. AOF=everysec loses writes on crash. Cluster slot migrations have edge cases. Lesson: Redis is a cache or a queue. If you can't lose data, durable store first; Redis on top.

"We used Elasticsearch as the primary store"

No transactions; segment merges block writes; refresh interval = stale reads; not durable on crash without explicit fsync. Lesson: ES is a search index. Source of truth elsewhere; CDC into ES.

"We used DynamoDB for analytics"

Scan reads every item; eats RCU; expensive and slow. Even with GSIs, complex aggregations are not its strength. Lesson: export to S3 + Athena, or stream into a warehouse via Kinesis Firehose / DDB Streams.

"We used Postgres for high-throughput append-only events"

Bloat, vacuum churn, WAL pressure. Works fine to ~10k events/sec; breaks past that. Lesson: for sustained heavy append, use Cassandra or Kafka. PG can fan in via batch + COPY for bursts.

"We picked Spanner for a single-region app"

Spanner's value is global consistency. In a single region, regular Postgres has 5× lower latency at lower cost. Lesson: if one region is enough, regional Postgres / MySQL is the right answer.

"We used graph DB for one-hop joins"

If most queries are 1–2 hops, SQL with proper indexes is faster and the team already knows it. Lesson: reach for Neo4j only when traversals dominate (3+ hops, recursive structure, social graph).

10. Decision rules for interviews

The 1-line heuristics interviewers expect to hear. Memorize these.

  • If the data is relational and fits on one box, default to Postgres. Wrong rarely; always defensible.
  • If you need to shard SQL, the answer is Spanner / CockroachDB / Vitess — pick by cloud. Spanner on GCP, CRDB elsewhere, Vitess if you already run MySQL.
  • If access is by primary key and you tolerate eventual consistency, default to Cassandra (self-host) or DynamoDB (AWS-managed). Pick by who runs the cluster.
  • If working set fits in RAM and latency must be sub-ms, Redis. Memcached only when "pure cache, max throughput per core" is non-negotiable.
  • If schema is genuinely unpredictable per row, MongoDB. Otherwise Postgres JSONB.
  • If queries are graph traversals past 2 hops, Neo4j. Otherwise SQL recursive CTEs (Common Table Expressions — WITH RECURSIVE queries that walk a hierarchy by referencing themselves).
  • Distributed locks: Redis (SET NX PX) for single-region cheap; etcd or ZK if fencing tokens are critical.
  • Idempotency tables: DynamoDB if events go through it; Postgres unique index if writes already touch SQL.
  • Never use ES, Redis, or Mongo as source of truth unless you can explicitly defend why.
  • Always ask "single-region or global?" before naming the SQL DB. The answer changes from Postgres to Spanner.
