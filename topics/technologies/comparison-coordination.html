6. Migration paths

  • Zookeeper → etcd: moving off legacy ZK ensemble. Schema is different (znodes vs flat KV); rewrite client code; double-write during migration.
  • Kafka ZK → KRaft: built-in migration tool; rolling, but irreversible. Test in staging.
  • Eureka → Consul / k8s: typical Spring Cloud cleanup. Sidecar approach during transition.
  • Self-host Consul → HCP Consul: cluster federation; tokens migrate; ACL replication.
  • Custom DIY discovery → cloud SD: Cloud Map / Service Directory replace most home-grown solutions.

8. Hybrid patterns

etcd inside k8s + Consul outside

Common pattern: k8s uses its built-in etcd (don't share with apps); apps run Consul for cross-DC discovery.

Service mesh subsumes coordination

Istio / Linkerd / Consul Connect handle service discovery + mTLS. You may not need a separate coordination layer.

Redis lock + etcd as fallback

Redis SET NX for fast path (most cases); etcd-backed lease only when fencing token required.

Feature flags via etcd watches

Tiny shops use etcd watches for live config without paying a feature-flag SaaS. Fine until org grows.

Cloud Map for HTTP services + Consul for gRPC mesh

Cloud Map is cheap for plain DNS; Consul provides richer health + tagging for service mesh.

9. Anti-patterns

"We ran our own ZK for one app"

3-node JVM ensemble + on-call rotation for one feature. Use a managed feature-flag service or k8s ConfigMap.

"We used etcd to store user data"

etcd is for coordination, not data. 8 GB DB cap, 1.5 MB value cap, raft log overhead. Use a real KV.

"We used Eureka in 2026"

Mostly legacy at this point. k8s SD or Consul has surpassed it.

"Mixed app traffic with k8s etcd"

Sharing k8s control-plane etcd with apps causes latency spikes for both. Run a separate etcd if needed.

"Distributed lock without fencing"

Even with etcd, a stalled holder might come back. Leases + monotonic fencing tokens at the resource side. Read Kleppmann.

"One ZK ensemble across regions"

WAN latency tanks consensus. ZK is single-DC; use observers across regions, not voting members.

10. Decision rules for interviews

  • If on k8s, use what k8s gives (etcd, Lease, Service).
  • For new self-host coordination needs, etcd over Zookeeper.
  • Consul if you need multi-DC service discovery + health checks.
  • Eureka is mostly legacy. Don't deploy new.
  • Distributed locks: Redis for cheap; etcd / ZK when fencing tokens matter.
  • Coordination clusters are 3 or 5 nodes — never 2 or 4 (no quorum on tie).
  • Don't store data in coordination services. They're for metadata.
  • Single-DC by default; cross-DC needs Consul or service mesh.
  • Read Kleppmann's "How to do distributed locking" before claiming you have one.
  • Most apps don't need a coordination service. Ask if it's a real requirement.
