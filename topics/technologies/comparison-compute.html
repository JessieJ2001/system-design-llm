6. Migration paths

  • VMs → containers: dockerize each app; deploy on ECS Fargate / Cloud Run first (lowest ops); migrate to k8s only if needed.
  • ECS Fargate → k8s: typical when team grows / multi-cloud needed. Translate task defs to Deployment manifests; redo IAM via IRSA.
  • k8s → Cloud Run: for stateless 12-factor services; cuts ops cost dramatically. Watch for stateful sets, custom controllers, sidecars (Cloud Run has limited sidecar support).
  • Lambda → container: when 15-min limit hits, cold-start is unacceptable, or you outgrow function model. Lambda containers were a stepping stone; Cloud Run / Fargate is the real destination.
  • Self-host k8s → managed (EKS / GKE / AKS): almost always worth it; takes managing the control plane off your plate.
  • k8s → GKE Autopilot: if pods don't need node-level customization. Removes node management entirely.

8. Hybrid patterns

k8s + Cloud Run side by side

k8s for the steady microservice fleet; Cloud Run for spiky/idle services and webhooks. Same team, different scaling shapes.

Lambda glue + Fargate workers

Lambda for event triggers (S3, SQS, EventBridge); Fargate / Cloud Run for long-running workers. Lambda hands work over the line.

Spot for batch + on-demand for serving

Mixed node groups in k8s. Batch jobs on spot for 70% savings; serving stays on-demand for SLOs.

Stateful in managed services + stateless in serverless

DBs, queues, caches in managed services. App = stateless on Cloud Run / Fargate. Match each part of the system to the right shape.

k8s as deploy target + serverless as event surface

Knative / KEDA bring scale-to-zero and event-driven scaling to k8s. Get serverless ergonomics without leaving k8s.

9. Anti-patterns

"k8s for our 5-service startup"

Cluster ops eats engineer time; team would have shipped faster on Cloud Run / Fargate.

"Lambda for a busy 24/7 API"

5–10× the cost of Fargate/Cloud Run at sustained high QPS. And cold starts hurt.

"Stateful workload in serverless"

Filesystem ephemeral; no sticky sessions. Use managed DB / queue, not Lambda local FS.

"k8s nodes always at 30% utilization"

Pay for 70% idle. Bin-pack with VPA / pod packing; consider Autopilot.

"Lambda for > 15-min jobs"

Hits the wall. Use Cloud Run jobs / Fargate / k8s Jobs.

"Forgot reserved concurrency"

Lambda burst exceeds account limit; whole account 429s. Reserve concurrency for important functions.

"App Mesh in production"

AWS App Mesh end-of-life announced. Use Istio/Linkerd directly on EKS.

10. Decision rules for interviews

  • Default progression: VM → ECS Fargate / Cloud Run → k8s only when complexity demands.
  • Lambda / Cloud Functions for event glue, cron, < 15 min stateless.
  • Cloud Run / App Runner / Container Apps for spiky HTTP.
  • k8s for >~10 services, complex topology, multi-cloud.
  • Stateful = managed service first; k8s StatefulSet only if forced.
  • Always say which scaling metric (CPU / RPS / queue depth) drives autoscaler.
  • For HPA, scale-up lag is the main risk; pre-warm with min replicas for spike-prone services.
  • Spot / preemptible for batch + checkpointed work; never for serving alone.
  • Cold-start: Go / Node < Java / .NET. For latency-critical Lambdas, use provisioned concurrency.
  • Always state "managed control plane" when naming k8s in a cloud answer.
