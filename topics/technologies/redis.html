Redis

Redis is a database that keeps your entire dataset in RAM and runs every command on a single thread, one at a time. That's why it answers in microseconds — and that one decision shapes everything else about how it works.

Single-threaded, in-memory data-structure server. The whole dataset lives in RAM (not on disk like a traditional database), so reads and writes finish in microseconds. "Single-threaded" means one main thread handles every command sequentially — no locks, no race conditions, but a slow command stalls everyone behind it. "Data-structure server" means Redis isn't just key→blob; the server natively understands lists, sets, sorted sets, hashes, streams, etc., so you can do things like "add to a leaderboard" or "pop the next job" as one atomic command.

Hash-table indexing for keys. The top-level keyspace is a hash table, so finding any key is O(1) — one step regardless of how many keys you have.

Specialized in-memory representations per type. Each value type uses a layout tuned for its access pattern. Sorted sets use a skiplist (ordered scans + log-time inserts). Small lists/hashes/sets are packed into listpack / ziplist — compact byte arrays that fit in CPU cache and skip pointer overhead. The same logical type can switch encoding as it grows.

Optional AOF + RDB persistence. Redis can periodically snapshot memory to disk (RDB) and/or append every write to a log (AOF). You pick the durability you need: a pure cache turns both off; a durable store enables AOF with fsync.

Leaderless cluster mode with 16,384 hash slots. To scale past one machine, Redis hashes each key into one of 16,384 slots (CRC16(key) mod 16384) and assigns slot ranges to primary nodes. "Leaderless" here means there's no central coordinator routing traffic — clients learn the slot map and talk directly to the primary that owns the key.

Why exactly 16,384 (= 214)? It's a tradeoff fixed at the protocol level, per antirez's design notes:

  • Cluster bus heartbeat size. Every node gossips its slot ownership as a bitmap in heartbeat packets. 16,384 bits = 2 KB per heartbeat; 65,536 slots would be 8 KB — too much, since heartbeats fly constantly between every node pair.
  • Practical cluster cap ~1,000 masters. Redis Cluster isn't designed past ~1,000 master nodes. With 16,384 slots that's still ~16 slots/node — enough granularity for balanced sharding. More slots would be wasted.
  • Bitmap compression. The slot bitmap compresses well when a node owns a contiguous range; smaller bitmaps hit higher compression ratios, which matters for constant gossip traffic.

2. Underlying mechanism

Redis is what you'd write if you were forced to make one decision: everything happens on a single thread. That choice forces every other choice.

Why single-threaded works

RAM is fast; commands finish in microseconds. With one thread there are no locks, no atomic-ops contention, and operations on data structures are linearizable for free. The cost: any one slow command blocks the entire instance. Slow Lua, KEYS *, big ZRANGE, DEBUG SLEEP — all of them stall every other client.

Since 6.0, network I/O (parse / write-back) can run on multiple threads. Command execution itself is still single-threaded.

Per-type encodings (the part most readers miss)

Redis picks the encoding based on size. A small hash uses a flat listpack (cache-friendly array); past hash-max-listpack-entries it converts to a real hash table. Same for set / zset. This is why "small values" run drastically faster than the algorithmic complexity suggests — the inner loop is sequential memory access.

4. Replication & consistency

  • Topology: single-leader with N async replicas. In Cluster mode, each of N primaries has its own followers.
  • Mechanism: command stream over RESP with replication backlog buffer; PSYNC2 handles partial resync after disconnect.
  • Consistency: per-shard linearizable on the primary; replicas are eventually consistent (async repl). WAIT n timeout blocks until n replicas ack — closer to "majority durability" than to consensus.
  • Failure model: primary crash → manual or Sentinel-driven failover. Async repl means up to replication-backlog-size writes can be lost on failover. AOF=everysec adds 1s of tail loss on full crash.

5. Partitioning

  • Strategy (Cluster mode): CRC16(key) mod 16384 → slot. Each primary owns a contiguous range of slots.
  • Rebalancing: live slot migration via MIGRATE. Add a node, move slots gradually; clients receive MOVED / ASK redirects during migration.
  • Hot-key behavior: a hot key pegs one shard's single thread. Cluster doesn't help. Mitigations: replicate the key (suffix-sharded); front with a per-pod L1 cache; use CLIENT REPLY OFF+pipeline for write storms.
  • Hashtags: {user:123}:cart and {user:123}:wishlist hash to the same slot, allowing multi-key commands. Required to keep related keys colocated.

How clients learn the slot map

The slot → node mapping isn't pushed by a coordinator; clients discover and cache it lazily.

  1. Bootstrap. Client is configured with a seed list of any cluster nodes (e.g. node-a:6379, node-b:6379). On startup it calls CLUSTER SHARDS (or older CLUSTER SLOTS) on any reachable seed. The response is the full map: every slot range → (primary endpoint, replica endpoints).
  2. Cache locally. Client stores the map in memory as a slot → node array of length 16,384. Routing a command is CRC16(key) mod 16384 → index into the array → open/reuse a connection to that node. O(1), no extra hop.
  3. Refresh on redirect. If the map is stale (slot moved during rebalancing or failover), the targeted node replies with a redirect:
    • MOVED 3999 10.0.0.5:6379 — "slot 3999 now permanently lives at this node." Client retries there and re-fetches CLUSTER SHARDS to refresh the whole map.
    • ASK 3999 10.0.0.5:6379 — "slot is mid-migration; ask the other node for this one key only." Client retries with an ASKING prefix but does not update its map (the migration isn't complete).

Net effect: the steady-state path is one hop. Redirects are rare (only during topology change) and self-healing — the client never needs an external service-discovery system.

How nodes gossip the topology to each other

Nodes maintain their own view of the cluster over a separate cluster bus port (default 16379, i.e. data port + 10000). It uses a binary protocol, not RESP.

  1. PING/PONG every second. Each node picks a random subset of peers and sends a PING every ~1 second. Each PING/PONG packet carries a header plus a gossip section with info about a few other random nodes the sender knows about. Over a few seconds, every node learns about every other node transitively — classic epidemic gossip.
  2. Slot ownership = bitmap in the header. Every PING/PONG includes the sender's myslots field: a 16,384-bit bitmap (= 2 KB) where bit i is set if this node owns slot i, and 0 otherwise. So a node owning slots 0–5460 has bits 0…5460 set and the rest clear. Receivers walk the bitmap and, for every set bit, write slot[i] = sender_node into their local slot → node table. Why a bitmap? A bitmap is the densest possible encoding when there are exactly two states per slot ("yes I own it" / "no I don't"): 1 bit per slot. The alternative — sending a list of owned slot ranges — would be smaller in the steady state but variable-sized and harder to diff; the fixed 2 KB bitmap makes every heartbeat the same shape and lets the receiver overwrite ownership with a single AND/OR pass. That's also why the slot count was capped at 16,384: bigger bitmaps would balloon the constant heartbeat traffic.
  3. configEpoch breaks ties. Each primary has a monotonically increasing configEpoch (a logical clock). When two nodes claim the same slot (e.g. during failover), receivers trust the one with the higher epoch. This is how a newly-promoted replica's slot ownership wins over the dead primary's stale claim.
  4. Failure detection. If a node hasn't replied within cluster-node-timeout (default 15s), the sender marks it PFAIL (possibly failed) locally and gossips that opinion. Once a majority of primaries report PFAIL for the same node, it's promoted to FAIL cluster-wide and replica failover kicks in.

So the cluster bus carries three things on a single channel: liveness pings, topology (slot bitmap + epoch), and failure votes. There's no Zookeeper, no etcd — just N×N gossip.

6.5 Sophisticated algorithms inside Redis

Redis is small (~150k LOC of C) but punches well above its weight because it picks one clever data structure per problem. This section catalogs them: what problem each solves, the algorithm, and why it's the right choice given the single-threaded constraint.

1. Skiplist (sorted sets)

Problem: back leaderboards / time-ordered indexes — need O(log n) insert + ordered range scan + rank lookup.

Algorithm: a linked list where each node has multiple "express lanes." A node at level k has pointers skipping ahead at every level ≤ k. Levels are assigned by coin flip (geometric distribution: P(level ≥ k) = 1/2k). Search descends levels like a binary tree, expected O(log n).

level 3:  HEAD ----------------> 17 ----------------> NIL
level 2:  HEAD --------> 9 ----> 17 --------> 28 ---> NIL
level 1:  HEAD --> 4 -> 9 ---> 17 --> 23 -> 28 ---> NIL
level 0:  HEAD -> 4 -> 9 -> 12 -> 17 -> 23 -> 28 -> NIL

Why clever: no rotations, no rebalance bookkeeping — just probabilistic levels. Simpler to implement correctly than a red-black tree, and range scans are a flat walk on level 0. Redis also stores a span per pointer so it can answer ZRANK in O(log n) without scanning.

2. Listpack & adaptive encoding

Problem: a 5-element hash shouldn't pay the memory cost of a hash table (buckets, pointers, padding — easily 100 bytes of overhead).

Algorithm: small collections are stored as a listpack: a single packed byte buffer of [total_bytes][num_elements][entry][entry]…[end_byte]. Each entry is a varint-encoded value with a back-length so you can scan in either direction. No pointers, no buckets. When the collection exceeds a threshold (e.g. hash-max-listpack-entries 128 or hash-max-listpack-value 64), Redis transparently converts to a hash table.

Why clever: you get a hash-table API at hash-table cost only when you need it. For the 99% of objects that are tiny, the value sits in one cache line. The adaptive encoding switch is one-way per key, so the amortized cost of conversion is paid once.

3. Quicklist (lists)

Problem: a Redis list needs O(1) push/pop on both ends and good memory density. Pure linked list = pointer overhead per node. Pure array = bad for inserts in the middle.

Algorithm: a doubly-linked list of listpacks. Each list node is a listpack of ~128 elements. Push to the head appends inside the head listpack (O(1) amortized); when full, a new listpack is prepended.

Why clever: CPU cache loves contiguous data (each listpack is one block); cross-block traversal happens only every ~128 elements. Optionally each interior listpack is LZF-compressed (only the head and tail stay raw for fast push/pop) — trades a bit of CPU for a lot of RAM.

4. Incremental rehashing

Problem: the keyspace hash table grows. Doubling it means copying millions of entries. On a single thread, that's a multi-second stall — unacceptable.

Algorithm: keep two hash tables side by side, ht[0] (old) and ht[1] (new, 2× size). On every command, move 1 bucket from old to new. Reads check both tables; writes go to the new. After N ops the migration finishes and ht[0] is freed. The cron tick also moves a few buckets per tick to make progress when traffic is idle.

trigger:  load_factor > 1 (or > 5 if forking)
during:   every command moves 1 old bucket to new table
          lookups query ht[0] then ht[1]
done:     when ht[0] is empty → free it

Why clever: amortizes a stop-the-world cost over millions of operations. The user never sees the resize.

5. HyperLogLog (cardinality estimation)

Problem: "how many unique users visited today?" with a billion events. Storing every user ID = 8 GB+. SADD + SCARD would work but blow up memory.

Algorithm: hash each item to a 64-bit value. Use the first p bits to pick one of 2p "registers" (Redis uses p=14, so 16,384 registers). In each register, store the maximum number of leading zeros seen in the remaining bits. The intuition: if you've seen a hash with k leading zeros, you've probably seen ~2k distinct items. Average across all registers (with bias correction) to estimate cardinality.

PFADD visitors user:1 user:2 ... user:1_000_000_000
PFCOUNT visitors  → 998_421_553  (~0.81% standard error)
total memory: 12 KB regardless of cardinality

Why clever: 12 KB to count up to 264 distinct items with ~0.8% error. PFMERGE is just register-wise max — merge daily HLLs into a weekly one in microseconds. Used everywhere from "unique visitors" to "distinct IPs in a DDoS."

6. Bitmap as a data type (SETBIT / BITCOUNT / BITOP)

Problem: "did user N do X today?" for 100M users. A set of integer IDs costs ~80 bytes/entry. Way too much.

Algorithm: use a Redis string as a bit array. SETBIT dau:2026-05-04 user_id 1 sets one bit. BITCOUNT counts set bits using SIMD popcnt. BITOP AND intersects two days' bitmaps to find users active on both. Storage is dense: 100M users = 12.5 MB regardless of activity rate.

Why clever: reuses the existing string type — no new structure needed. SIMD popcnt counts billions of bits per second on modern CPUs. Sibling to the cluster bitmap discussed in §5: same idea, different problem — whenever the universe is small and every element is a yes/no question, a bitmap is the densest encoding possible.

7. Probabilistic LRU/LFU eviction

Problem: maxmemory is hit; need to evict the least-recently-used key. Classic LRU = doubly-linked list bumped on every access — pointer churn on the hot path, plus O(1) extra memory per key.

Algorithm: on eviction, sample N random keys (default 5), evict the one with the oldest access time. With more samples (e.g. 10), the sampled choice gets within ~5% of true LRU. For LFU, each key has a logarithmic counter that increments probabilistically (decays over time) — capturing access frequency in 8 bits.

Why clever: trades exactness for cost. No eviction list to maintain on the hot path; eviction itself does the work, only when needed. Same trick as the active-expiry sweep — sample, don't scan.

8. Active expiry sampling

Problem: millions of keys have TTLs. Walking them all every second is O(N) and stalls the event loop. Pure lazy expiry leaks memory if keys are never accessed.

Algorithm: at 10 Hz, sample 20 keys with TTLs. Delete expired ones. If > 25% of the sample was expired, loop and sample 20 more — the keyspace is "expiry-rich," keep cleaning. Otherwise break (don't burn CPU on a clean keyspace).

Why clever: adaptive, statistically self-tuning. The 25% threshold means CPU spent on expiry scales with how much expiring there is to do. Same family as probabilistic LRU.

9. Copy-on-write fork() for RDB / AOF rewrite

Problem: snapshot a 64 GB in-memory dataset to disk without freezing writes for 30 seconds.

Algorithm: call Linux fork(). The child inherits the parent's virtual address space via copy-on-write — both processes initially share the same physical pages. The child walks memory and writes RDB to disk at its leisure. The parent keeps serving traffic; pages it modifies get copied lazily by the kernel (only the modified pages, not the whole 64 GB).

Why clever: Redis gets a consistent point-in-time snapshot for free, courtesy of the kernel's COW semantics. The catch: write-heavy workloads can double RAM during the fork (every modified page gets duplicated), and very large datasets make the fork() syscall itself slow due to page-table copy. Production tip: keep shards < 25 GB and use Linux's vm.overcommit_memory=1.

10. Replication backlog as a ring buffer (PSYNC2)

Problem: a replica disconnects briefly. Re-syncing means streaming the entire dataset — minutes of network traffic, primary forks for RDB, etc. For a 5-second blip, this is absurd.

Algorithm: primary keeps a fixed-size circular buffer (repl-backlog-size, default 1 MB — tune to 64–512 MB!) of recent commands, indexed by a monotonic byte offset. When a replica reconnects with offset O: if O still lives in the backlog → resume from there (partial sync). Otherwise → full sync (RDB).

Why clever: O(1) check ("is this offset in [start, end]?"), tiny memory cost, and 99% of disconnects are partial-syncable. PSYNC2 (Redis 4+) also lets a replica that gets promoted keep its backlog, so replicas of the old primary can partial-sync against the new one without a full RDB transfer.

11. Cluster gossip with bitmap + configEpoch

Problem: N nodes need a consistent slot → node map without a coordinator (no Zookeeper, no Raft). Nodes come, go, fail over.

Algorithm: already covered in §5. Each node ships its 16,384-bit ownership bitmap in every PING/PONG. Conflicts (two nodes claiming the same slot) are resolved by configEpoch — a logical clock bumped on every failover. Higher epoch wins. Failure detection is gossip-amplified: PFAIL → majority → FAIL.

Why clever: eventually-consistent topology that converges in O(log N) gossip rounds, with O(N) bandwidth per node per second. No external dependency, no leader election bottleneck. The cost (occasional MOVED redirects to clients during convergence) is paid by the path that's already async.

12. Radix tree (rax) for streams

Problem: Redis Streams append entries with IDs like 1685000000000-0, 1685000000000-1, … You need O(log N) insert and efficient range scans by ID, plus very compact storage for billions of entries.

Algorithm: a radix tree (compressed trie) keyed by stream ID bytes. Common prefixes are stored once. Each leaf points to a listpack of recent entries that share a prefix, so you get the trie's range-scan benefits and listpack's memory density together.

Why clever: stream IDs are heavily prefix-correlated (timestamps move forward in lock-step). The radix tree exploits this so a billion-entry stream uses a small fraction of what a balanced BST would.

13. SDS (Simple Dynamic Strings)

Problem: C strings are \0-terminated, so strlen is O(N) and they can't hold binary data. Plus naive appends are O(N) per call.

Algorithm: SDS = header + buffer. Header stores len and alloc (capacity). Five header variants (sdshdr5/8/16/32/64) are picked based on string length, so a 10-byte string pays a 3-byte header. Appends double capacity until 1 MB, then add 1 MB at a time — amortized O(1) APPEND.

Why clever: O(1) STRLEN, binary-safe, drop-in compatible with C string functions (the buffer is still \0-terminated for safety). The size-class headers save bytes on small strings — matters because Redis has billions of them.

The unifying pattern

Notice the recurring trick across these algorithms: trade exactness or worst-case complexity for amortized cost on the common path. Probabilistic LRU samples instead of tracking. HyperLogLog estimates instead of counting. Active expiry samples instead of scanning. Incremental rehash spreads the cost. Cluster gossip converges instead of consenting. COW fork shares pages instead of copying. The single-threaded constraint forces every choice to be cheap on the hot path; the rest is paid in background ticks, in lazy work at access time, or in tolerable approximation.

8. When to use / not to

Use Redis when…

  • Working set fits in RAM and sub-ms latency matters.
  • You need data structures (ZSET for leaderboards, Streams for queues, hash for session).
  • Atomic ops on single-key state (counters, distributed locks, rate limiters).
  • Pub/sub or fan-out at sub-ms latency.
  • Cache-aside in front of slow primary store.

Don't use Redis as…

  • The source of truth for data you can't lose. AOF=everysec loses up to 1s of writes; eviction silently deletes data.
  • A datastore for working sets > node RAM. Disk-backed Redis (Redis on Flash) is niche.
  • Storage for large blobs (> 1 MB / item). Single-thread blocks during processing.
  • A SQL-like datastore. No joins; no secondary indexes; ad-hoc queries impossible.
  • A durable queue with replay if you don't want to operate Streams + trim policy carefully.

9. Common gotchas

  • KEYS * in production. O(N) blocks the event loop — entire shard stalls. Always SCAN.
  • Big keys. A list with 10M elements: LRANGE 0 -1 blocks for seconds. DEL on a 100 MB key blocks too — use UNLINK.
  • Mass-expire pause. If a million keys expire at the same second, Redis spends time deleting them and blocks. Add jitter to TTLs.
  • Cluster slot resharding. Live but expensive; each migration adds latency to keys in flight. Pre-size the cluster.
  • Multi-key txns across slots. Don't work in Cluster mode; MULTI/EXEC only within one slot. Use hashtags {user:123}.
  • AOF rewrite with huge memory. fork() + copy-on-write can pause for seconds at 100+ GB. Use diskless rewrite or RDB-only.
  • Lua scripts that loop. Block event loop. Set lua-time-limit.
  • Pub/Sub silently drops. No replay, no consumer group; messages lost if subscriber missed them. Use Streams.
  • Sentinel failover with lost writes. Async repl + failover = up to backlog-size of writes lost. Use WAIT for critical writes.
  • Hot key pegs single shard. Cluster doesn't help. Replicate hot key by suffix or front with L1.

13. Operational knobs that matter

  • maxmemory + maxmemory-policy — set both. Default noeviction errors on writes when full. Common: allkeys-lru for cache, volatile-lru when mixing cache + persistent keys.
  • appendfsync — always (slowest, durable per write), everysec (default, lose up to 1s), no (let OS decide).
  • repl-backlog-size — controls how much primary buffers for partial resync. Too small → full sync after every brief disconnect. Set generously (256 MB–1 GB).
  • cluster-require-full-coverage — if yes (default), losing one shard kills the whole cluster. Set no in production for partial availability.
  • cluster-allow-reads-when-down — lets replicas serve reads even when consensus is broken.
  • lua-time-limit / busy-reply-threshold — defends event loop against runaway scripts.
  • tcp-keepalive — default 300s; set lower in cloud environments where idle conns are dropped silently.
  • io-threads — 6.0+ network I/O parallelism. Cap at <= cores; usually 4–8.
  • activedefrag — turn on if you have heavy fragmentation (RSS >> used_memory).
  • save — RDB snapshot triggers; set to empty for AOF-only setups.

14. Capacity planning rules-of-thumb

  • Memory: memory = (avg_value_size + ~50B overhead) × num_keys × replication_factor. Add 30% headroom for fragmentation + COW during AOF rewrite.
  • QPS / shard: ~100k ops/sec/core for simple GET/SET. Shards needed = peak_qps / 80k (leave headroom).
  • Hot-key budget: one hot key ≤ 100k QPS. Past that, replicate the key by suffix and round-robin reads, OR L1 cache in-process.
  • Replication doubles RAM per shard pair. Cluster mode multiplies further (slot-aware).
  • Network: a 1 KB GET at 100k QPS = 100 MB/s = 800 Mbps. Plan NIC for > 2× peak.
  • Connections: a connection is ~20 KB. 10k connections = 200 MB just for sockets. Use connection pools client-side.
  • Stream / ZSET trim: always cap with XADD MAXLEN or periodic ZREMRANGEBYRANK; otherwise unbounded.

16. Backup, restore & disaster recovery

  • Backup primitives: RDB snapshots (point-in-time); AOF (full log). For DR, ship RDB snapshots to S3 hourly + AOF for recent.
  • RPO: RDB-only = last snapshot interval; AOF = up to 1s with everysec; WAIT n+sync repl tightens to "ack of N replicas."
  • RTO: restore is "load RDB into a new instance" (RAM-fill speed) + "replay AOF" (sequential read). 100 GB ≈ 5–15 min.
  • Cross-region DR: async replica in DR region; promote on disaster. Or active-active CRDT (Redis Enterprise).
  • Drill cadence: quarterly: take RDB, restore on a fresh instance, verify DBSIZE. Monthly: simulate Sentinel failover.
  • What's not backed up: Pub/Sub messages (ephemeral); cluster slot map (rebuild from gossip).

17. Upgrade & online migration

  • Major version (e.g., 6 → 7): rolling upgrade — upgrade replica, failover, upgrade old primary. RDB format is forward-compat; AOF format may need conversion.
  • Single → Sentinel: add Sentinels with existing primary; deploy clients with Sentinel-aware libs; verify failover; old single-pointing clients will keep working until cutover.
  • Sentinel → Cluster: requires non-trivial migration; new client lib; scripts to dual-write; eventually decommission Sentinel.
  • Cluster resharding: use redis-cli --cluster reshard; live; one slot at a time. Plan for ~5 min per slot at full Cluster load.
  • License-driven migration (Redis OSS → Valkey): drop-in same protocol; pin exact version; verify command coverage.
  • Migrating away from Redis: dual-write to new store + Redis; shadow-read; cut over; decommission. For caches, just swap the client lib.

18. Security model

  • Authentication: requirepass (single password, legacy). 6.0+ ACLs with user definitions: per-command + per-key-pattern + password.
  • TLS: 6.0+ supports TLS for client + replication + cluster bus.
  • Authorization: ACLs at command level (e.g., deny FLUSHDB, allow GET) and key-pattern level. No row-level.
  • Network: never expose Redis to the internet. Bind to localhost or VPC private subnet; firewall the cluster bus port (16379).
  • Encryption at rest: not native; relies on disk encryption (LUKS, EBS encryption).
  • Audit: none native (commands aren't logged). Use MONITOR for short-term debug; never long-term (it slows everything).
  • Common breach: open Redis instance with no auth and a routable IP → CONFIG SET dir + SAVE writes attacker SSH key. Lock down everything.

19. Multi-tenancy & isolation

  • Database numbers (SELECT 0..15) — legacy logical DBs. Don't use for tenant isolation; not cluster-compatible.
  • ACL users + key-pattern restrictions — reasonable for "team isolation" but no resource quotas (a noisy tenant pegs the shard).
  • Pattern in practice: one Redis cluster per tenant tier (free / pro / enterprise) rather than mixing.
  • Hot tenant protection: nothing native. Use proxy-tier rate limiting (Envoy) or app-level quotas.
  • Per-tenant TTL: apply at the app; Redis just stores keys.

20. Observability — what to monitor

  • used_memory vs maxmemory — before eviction starts, you want to be at 60–75%.
  • evicted_keys — spike means cache is too small or hot keyspace shifted.
  • blocked_clients — pile-up indicates slow command or replication stall.
  • connected_clients — growing past pool size = leak.
  • master_repl_offset − replica's offset — replication lag (bytes).
  • slowlog get 10 — commands taking > 10 ms; investigate immediately.
  • latency-monitor — tracks event-loop pauses and AOF/fork stalls.
  • aof_rewrite_in_progress + aof_current_size — rewrite duration is the biggest stall risk.
  • instantaneous_ops_per_sec — current throughput, vs your headroom plan.
  • Cluster-bus health: CLUSTER INFO → cluster_state:ok; ratio of cluster_known_nodes to active.

21. Anti-patterns & scaling cliffs

  • KEYS * in production. SCAN instead.
  • Mega keys (> 100 MB). Split or move to a DB.
  • EXPIRE on millions of keys at the same instant → mass-expiry pause.
  • Pub/Sub used as durable messaging. Use Streams.
  • Cross-slot multi-key txn in cluster. Use hashtags.
  • Source of truth. Cache or queue, not primary store.
  • Long Lua scripts looping over large sets. Block event loop.
  • Fork-pause at 100 GB+ RAM. Use diskless rewrite or RDB-only or smaller shards.
  • Hot key → one shard pegged regardless of cluster size.
  • Unbounded ZSET / Stream. Always trim.

22. Cost model

  • Compute driver: RAM × replication factor × cluster shards. RAM is the bill.
  • Self-host: instance hours; SSD only matters for AOF replay speed; bandwidth is usually free intra-AZ.
  • ElastiCache / Memorystore: billed per node-hour by RAM tier; replication doubles; cluster mode multiplies.
  • Cross-AZ replication: data charges in some clouds; can dominate if traffic is heavy.
  • MemoryDB: ~3× ElastiCache because of multi-AZ durable log.
  • Hidden costs: Sentinel nodes (3+); proxy tier; backups.
  • Saving levers: shrink working set (TTL aggressively); hash-encoded small values; LFU eviction; in-process L1 cache.
