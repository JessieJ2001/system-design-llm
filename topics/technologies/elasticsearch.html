Elasticsearch / OpenSearch

Distributed search and analytics engine built on top of Apache Lucene. Inverted index for full-text, BM25 ranking, kNN (HNSW) for vectors, rich aggregations, near-real-time indexing. Indexed shards distributed across nodes; primary + replica per shard. OpenSearch is the Apache 2.0 fork after the Elastic license change.

inverted index • Lucene segments • BM25 + kNN • aggregations

2. Underlying mechanism

  • Document → analyzers (tokenize, lowercase, stem) → terms.
  • Inverted index: term → sorted list of document IDs (postings list) + positions / frequencies.
  • Each Lucene segment is an immutable mini-index. New writes → in-memory buffer → new segment.
  • Refresh (default 1s) makes new segments queryable — not flushed to disk yet.
  • Flush persists segments + clears translog (the WAL).
  • Merge background combines small segments into bigger ones.
  • Shard = a Lucene index. Primary + replicas. Index is N shards.
  • Cluster state in master-eligible nodes; data nodes hold shards.

3. Read & write path

Index (write)

1. Coord receives bulk request
2. Routes to primary shards (by routing key, default = doc id)
3. Primary indexes locally + appends to translog
4. Replicates to replica shards (sync; in-sync set)
5. Acks coordinator after replicas
6. Periodically: refresh → new searchable segment
7. Periodically: flush → segments fsynced; translog cleared
8. Background: merge segments

Search (read)

1. Coord parses query (DSL or KQL)
2. Phase 1: query each shard for top-K doc IDs + scores
   (round-robin replicas; adaptive replica selection)
3. Coord merges → global top-K
4. Phase 2: fetch full _source for those K
5. Return

Aggregations: shard-level reduce + final reduce on coord

Vector kNN

HNSW graph index per segment
Search top-K nearest by cosine / L2 / dot product
Filter pushdown for hybrid (term filter + kNN)
Reciprocal Rank Fusion combines BM25 + kNN scores

Cluster coordination

Master-eligible nodes elect a master
Master owns cluster state (mapping, shard placement)
Data nodes hold shards
Coord/ingest nodes can be dedicated for routing

4. Replication & consistency

  • Per-shard primary + replica; sync replication.
  • Near-real-time: 1-second refresh interval default; reads see writes ~1s later (configurable).
  • Translog for durability; index.translog.durability: request for sync fsync per request.
  • Shard re-allocation on node loss; cluster health green / yellow / red.
  • Versioned docs allow optimistic concurrency.

5. Partitioning

  • Index split into N primary shards (chosen at index creation; cannot increase later without reindex). Replicas can change.
  • Routing: doc_id mod num_primary_shards; or custom routing key for colocation.
  • For time-series: rollover indices by date (ILM) instead of one giant index.
  • Search a "data stream" or alias spans many indices; queries fan out.

6. Time complexity

  • Indexed search: O(log N) per term + O(K) merging postings.
  • Document insert: O(small) buffered + amortized merge cost.
  • kNN HNSW: O(log N) per query (approximate).
  • Aggregations: O(N) per shard; coord reduces.
  • Deep pagination: O(from + size) — expensive past 10000.

7. Per-node capacity envelope

  • Indexing: 50k+ docs/sec/node depending on doc size and analyzers.
  • Query QPS: ~10k/shard for simple match; less for aggregations.
  • Latency P99: 10–50 ms for simple queries; aggregations vary.
  • Heap: max 32 GB (compressed pointers); use multiple nodes per box past that.
  • Storage: hot tier limited by RAM/disk ratio; UltraWarm/cold to S3 cuts cost.

8. When to use / not to

Use ES/OS when…

  • Full-text search at scale.
  • Log search (ELK / OS-Log).
  • Faceted ecommerce search.
  • Hybrid (BM25 + vector) retrieval.
  • Rich aggregations on indexed data.

Don't use as…

  • Primary store of record — not transactional, refresh interval = stale, lossy on crash without proper translog.
  • OLTP DB.
  • Analytics warehouse for huge ad-hoc joins (use BQ / Snowflake / ClickHouse).
  • Tiny scale where Postgres tsvector would do.

9. Common gotchas

  • Mapping explosion (dynamic field types) → cluster state ballooning.
  • Refresh interval = visibility lag; cannot read your own write < 1s without forced refresh.
  • Deep pagination (from + size > 10000) errors by default; use search_after.
  • Merge throttle when ingest is heavy.
  • Heap pressure → circuit breakers; dropped requests.
  • Hot shard from skewed routing.
  • Index per tenant explodes shard count; consolidate.
  • Reindexing for mapping change is expensive on huge indices.
  • Stop words / stemmer mismatch causes "why doesn't my exact-match query work?"
  • Aggregation memory overhead bigger than indexing.

10. Cloud equivalents

  • AWS: OpenSearch Service (Amazon's fork).
  • GCP: (no native; use ES on GCE or Vertex Search).
  • Azure: AI Search (different shape; managed cognitive search).
  • Multi-cloud: Elastic Cloud, Bonsai, Aiven.

12. Usage patterns

  • Site search / ecommerce facets.
  • Log + observability stack (ELK / OSL).
  • SIEM / security analytics.
  • Real-time analytics (low-cardinality dimensions).
  • Vector / hybrid retrieval for RAG.
  • Geo search.
  • App search APIs.

13. Operational knobs

  • JVM heap (max 32 GB with compressed oops).
  • Refresh interval (1s default; raise for ingest-heavy logs).
  • Merge policy + throttle.
  • Shard count + size (target 20–50 GB / shard).
  • Replica count.
  • Field data circuit breakers.
  • Indexing back-pressure (queue, threadpool).
  • ILM rollover policy.
  • Index templates (dynamic_mapping: strict).

14. Capacity planning

  • Hot tier RAM >= ~half of hot index size for best perf.
  • Shard size: 20–50 GB target; not too small, not too big.
  • Shard count per node: a few hundred max.
  • Replica count: 1 typical; raise for HA / read scale.
  • Disk: ~2× data size for merge headroom.

15. Failure modes

  • Master loss: re-election ≈ seconds; cluster red briefly.
  • Data node loss: replicas promoted; rebalance.
  • Heap pressure: GC pauses; circuit breakers trip.
  • Hot shard: ingestion / query slowed.
  • Disk watermark: shards moved off; index goes read-only at high watermark.
  • Mapping conflict: rejected docs.
  • Translog corruption: from disk failure; replicas help.

16. Backup & DR

  • Snapshot to S3 / GCS / Azure Blob via repository plugin.
  • SLM (Snapshot Lifecycle Management) automates schedules.
  • Restore creates new indices; alias swap to cut over.
  • Cross-cluster replication for live DR.
  • Searchable snapshots: data lives on S3, queryable via cache.

17. Upgrade & migration

  • Rolling upgrade across minor versions.
  • Major upgrades: deprecation log + reindex if mapping changes.
  • ES → OS: wire-compat through 7.10; later versions diverge. Snapshot + restore is the path.
  • Migrate from Solr: reindex via custom script.
  • From Postgres tsvector: build new ES index via CDC or batch.

18. Security

  • RBAC: roles + privileges.
  • TLS for transport + HTTP.
  • Native realm + LDAP / SAML / OIDC (Enterprise / OpenSearch security plugin).
  • Field-level / document-level security.
  • Encryption at rest: rely on disk encryption.
  • Audit logging.

19. Multi-tenancy

  • Index per tenant: simple but explodes shard count.
  • Filtered alias per tenant on shared index: scales tenants.
  • Document-level security via term-level filter.
  • Quotas: per-role index/cluster privilege; no native rate-limit.

20. Observability

  • JVM heap %.
  • GC pauses.
  • Query rate vs index rate.
  • Refresh time.
  • Merge throttle.
  • Segment count per shard.
  • Search / bulk queue depths.
  • Pending tasks.
  • Cluster health (green/yellow/red).

21. Anti-patterns

  • ES as primary store.
  • Deep pagination via from+size.
  • Mapping explosion (no dynamic: strict).
  • Tiny shards (millions of shards = cluster state bloat).
  • Single huge index for time-series instead of rollover.
  • Aggregations without indexes / on high-cardinality fields.
  • Unbounded scrolls.
  • Heap > 32 GB (loses compressed oops).

22. Cost model

  • RAM-heavy nodes × replicas.
  • UltraWarm / Frozen tier saves 80%+ for old data.
  • Egress + inter-AZ traffic.
  • Snapshots to S3 cheap; restore cost in CPU + time.
