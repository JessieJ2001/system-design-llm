Workflow & Durable Execution: Comparison & Tradeoffs

"Use Airflow" answers a different question than "use Temporal." DAG schedulers (Airflow / Dagster) are for data pipelines — daily / hourly batch jobs. Durable execution engines (Temporal) are for transactional business workflows — orders, payments, multi-step retries. State machines (Step Functions) are managed but limited.

6. Migration paths

  • Hand-coded retry / state machine in app → Temporal: the most common modernization. Wrap existing code as Temporal Activities; workflow defines orchestration.
  • Airflow → Dagster / Prefect: for nicer Python ergonomics + asset lineage. Migrate DAG by DAG; run side-by-side.
  • Cron + scripts → Airflow: typical "we outgrew bash" path.
  • Step Functions → Temporal: when costs hit (millions of transitions/day) or when you need "forever" workflows.
  • Self-host Temporal → Temporal Cloud: Cassandra cluster + history nodes is non-trivial; managed often pays for itself.

8. Hybrid patterns

Airflow for data + Temporal for app

Two distinct workloads, two distinct tools. Don't conflate: Airflow runs nightly batches; Temporal handles transactional business workflows.

Step Functions Express in front of Lambda chains

Replaces hand-rolled "Lambda calls Lambda" with a managed state machine. Better visibility, retries, error handling.

Temporal Signal for human approval

Long-running approval workflow waits via Signal; UI sends signal on click. Workflow proceeds — no polling, no manual state machine.

Argo for ML training + Vertex/SageMaker for serving

Argo orchestrates training pipelines on GPU pods; trained model deployed to managed serving. k8s expertise on training side, managed simplicity on serving.

dbt + Airflow

Airflow schedules a dbt run; dbt does the SQL transforms. dbt models are first-class assets in lineage.

Outbox + Temporal

App writes business event + outbox row in DB transaction; Temporal worker reads outbox via CDC; runs durable workflow with that event. Avoids dual-write pitfalls.

9. Anti-patterns

"We used Airflow for transactional workflows"

Airflow is built for batch / schedule-driven, not for thousands of business workflows running concurrently. Use Temporal.

"We used Step Functions for high QPS"

Standard SF at $0.025/1k transitions = $$. Use Express for short, or Temporal for many.

"We rolled our own state machine"

Initial scope is "just three states", grows to 50, retries become a swamp. Adopt Temporal early.

"Workflows are non-deterministic"

Temporal workflows must be deterministic for replay. Random / now() / IDs generated inline = non-deterministic; move to Activities.

"Forgot ContinueAsNew on long workflows"

Temporal history grows; replay slows. ContinueAsNew checkpoints into a new workflow with reset history.

"Mixed business retries with framework retries"

Temporal retries automatically; if your code also retries, you can get duplicates. Choose one layer to handle retries.

"Used Airflow ExternalTaskSensor everywhere"

Sensors poll; many sensors == many idle workers. Use Datasets / cross-DAG triggers.

10. Decision rules for interviews

  • Data pipelines (DAG, schedule-driven): Airflow / Dagster / Prefect.
  • Business workflows (orders, signups, retries, sagas): Temporal. Step Functions if AWS-only and managed wins.
  • k8s-native CI / ML pipelines: Argo Workflows.
  • Simple cron: don't pick a workflow engine; use the cloud scheduler.
  • Always say which engine handles retries (workflow framework or app code).
  • Temporal workflow code must be deterministic; non-determinism goes in Activities.
  • Long-running with human approval: Temporal Signal or SF callback.
  • For sagas, prefer Temporal's SAGA helper over hand-rolled compensations.
  • Self-host workflow engines have heavy ops; managed almost always justifies cost.
  • If your "workflow" runs in < 1 second, you don't need a workflow engine.
