Messaging: Comparison & Tradeoffs

"Use Kafka" is the wrong answer most of the time. Kafka is a partitioned log; RabbitMQ is a routing broker; SQS is a managed work queue; Pulsar separates broker from storage. They solve different problems and fail differently. This page is the decision tree.

Kafka • Pulsar • RabbitMQ • SQS • Kinesis • Cloud Pub/Sub • NATS / JetStream • Redis Streams

8. Hybrid patterns ("use them together")

Kafka + RabbitMQ (log + work queue)

Kafka holds the immutable event log; a sidecar consumer reads from Kafka and dispatches work into RabbitMQ for low-latency RPC-style processing. Pattern shows up at orgs that started on Rabbit, added Kafka for analytics, and ended up with both.

Outbox pattern (DB → Kafka via Debezium)

App writes business row + outbox row in one DB transaction. Debezium tails the WAL/binlog and publishes outbox rows to Kafka. Solves dual-write atomicity (DB and Kafka) without 2PC.

SQS in front of Lambda + Kafka behind

SQS as the buffer + retry layer; Lambda as the worker; final result published to Kafka for downstream consumers. SQS isolates retry/DLQ from the event log.

Pub/Sub fan-out + Cloud Run subscribers

One topic, N push subscriptions, each to a Cloud Run service. Pub/Sub scales each subscriber independently; backlog is the natural backpressure signal.

Redis Streams as fast-path + Kafka as cold path

Redis Streams for sub-ms intra-region delivery; Kafka downstream for analytics + replay. Useful when latency budget < 5 ms.

NATS for control plane + Kafka for data plane

NATS pub/sub for low-latency commands & service discovery; Kafka for high-volume event log. NATS does not try to be Kafka and vice versa.

9. Anti-patterns

"We used Kafka as a database"

Random reads by key are slow. Kafka is a log, not a KV. Use Kafka Streams + RocksDB to materialize a view, or pair Kafka with an actual DB.

"We used Kafka for low-latency RPC"

5–20 ms typical, even more with batching. For sub-ms, use NATS or gRPC.

"We used RabbitMQ for high-throughput event log"

Mirrored queues stall above ~100k in-flight; classic queues lose messages on crash. Switch to Kafka or quorum queues.

"We used SQS for ordered processing without FIFO"

Standard SQS is at-least-once and unordered. If you assumed order, you have a bug. Switch to FIFO (300 msg/s/group cap) or Kafka.

"We used Pub/Sub push and consumers can't keep up"

Push delivery slows when subscriber returns 5xx, growing backlog. Switch subscription type to pull, or fix the consumer.

"We used NATS core for durable workflows"

NATS core is fire-and-forget. If consumer is down, messages are dropped. Use JetStream.

"We made one Kafka topic per user"

Past ~4k partitions/broker, controller pressure tanks the cluster. Use a key-partitioned shared topic instead.

"We expected exactly-once and it wasn't"

Kafka EOS only inside Kafka (read-process-write). If the side effect is a payment or external API, you still need an idempotency key.

10. Decision rules for interviews

  • "Need replay or many consumers?" → log family (Kafka / Pulsar / Kinesis / Pub/Sub).
  • "Need low-latency work queue?" → RabbitMQ.
  • "On AWS, simple async background jobs?" → SQS.
  • "On GCP, global pub/sub?" → Cloud Pub/Sub.
  • "On AWS but want a log?" → Kinesis or MSK.
  • "Multi-tenant SaaS with millions of topics?" → Pulsar or NATS, not Kafka.
  • "Sub-ms RPC?" → NATS.
  • "Already running Redis, < 1M msg/day?" → Redis Streams.
  • Always ask "do you need replay?" before naming Kafka vs RabbitMQ.
  • Exactly-once is "exactly-once inside Kafka" — external side effects still need idempotency keys.
  • Hot partition is the most-asked failure mode — the answer is composite keys (user_id + time bucket).
