Networking Edge: Comparison & Tradeoffs

"Use a load balancer" doesn't answer the question. The edge has at least five distinct layers — CDN, L4 LB, L7 reverse proxy, API gateway, service mesh — and each terminates TLS, routes, authenticates, and rate-limits. Pick the layer where each concern lives, then pick the product.

CDNs • Cloud LBs (ALB/NLB/GCLB) • Nginx • HAProxy • Envoy • Traefik • Istio / Linkerd • Kong / API Gateway

6. Migration paths

  • Nginx → Envoy: typically when adopting a service mesh. Translate nginx.conf to Envoy YAML / xDS. Run side-by-side via DNS weighted, shift gradually.
  • Self-host LB → Cloud LB (ALB/GCLB): point DNS at the cloud LB; deregister old. Validate sticky sessions, WS, gRPC, and slow-loris timeouts before cutover.
  • Sidecar mesh → sidecarless (Istio Ambient / Cilium): reduces sidecar overhead; needs CNI integration. Roll namespace by namespace.
  • API GW (self-host) → Managed: Kong → AWS API Gateway: rewrite plugins (Lua) as Lambda authorizers. Watch for rate-limit semantics differences.
  • Add CDN in front: change DNS to CDN; CDN points to origin LB. The trick is making sure cache keys are right BEFORE you cut over (otherwise hit rate is 0 and origin gets crushed).
  • Switch CDN providers: warm new CDN with mirrored traffic; DNS cutover with low TTL; surrogate-key purge syntax differs per vendor.
  • Mesh A → Mesh B (e.g., Istio → Linkerd): per-namespace migration; remove old sidecars after verifying mTLS works on new.

8. Hybrid patterns ("layers stack")

CDN → Cloud LB → Nginx ingress → Service Mesh

The canonical 4-layer modern web edge. CDN does TLS, WAF, static caching. Cloud LB does anycast IP and L7 routing. Nginx Ingress does k8s-aware routing. Service mesh handles east-west mTLS, retries, observability.

API Gateway in front, Service Mesh inside

Kong / AWS API GW handles north-south (3rd-party developers, quotas, billing). Inside the cluster, service mesh handles east-west (mTLS between microservices). Two distinct concerns; two distinct products.

Anycast CDN for failover

CDN PoPs serve from nearest healthy region; if a region fails, anycast rerouts in seconds (no DNS TTL wait). Origin cluster is multi-region active-active behind it.

WAF separate from CDN

Use CDN for caching, separate WAF (Cloudflare WAF, AWS WAF) on the LB. Splits concerns when WAF rules are sensitive (PCI, GDPR).

Edge compute for personalization

Cloudflare Workers / Fastly Compute@Edge run before the cache. Personalize on the fly without breaking cache cardinality — e.g., A/B test variant chosen at edge, content cached per variant.

Sidecarless mesh

Istio Ambient / Cilium Service Mesh push L4 mTLS into the kernel via eBPF; L7 only when needed. Cuts sidecar overhead; common at large clusters.

9. Anti-patterns

"We did auth at the L7 LB"

OK for simple cases, but bypasses the API Gateway (auth, quotas, billing). If you have 3rd-party developers, you need an API Gateway.

"We didn't put a CDN in front"

Origin handles all static asset traffic; latency for global users is bad. CDN is the single biggest user-facing latency win.

"We use VARY: Cookie at the CDN"

Cache key explodes; hit rate drops to 0. Either don't cache personalized pages or use edge functions.

"We deployed Istio because it's standard"

Istio's complexity is high. If the team doesn't have a mesh-ops person, Linkerd or going without is often better.

"We rate-limit only at the API gateway"

If a service can be DDoSed by another service in the mesh, you also need east-west rate limiting. Mesh policy or service-level limits.

"We TLS-terminate in N places"

Terminate at the edge (CDN), re-encrypt to origin if compliance demands; don't have 3 separate cert stores. Use ACM / cert-manager.

"WS connections through ALB without idle-timeout tuning"

Default 60s idle timeout disconnects WS. Bump to 4 hours or use heartbeats.

"We made every service its own ALB"

$$ multiplied; complex DNS. Use one ALB with path-based routing, or one Nginx ingress in k8s.

10. Decision rules for interviews

  • Always say which layer terminates TLS. Default: CDN, then re-encrypt to origin if compliance requires.
  • Default edge stack on cloud: CDN → Cloud LB (ALB/GCLB) → Nginx Ingress → service.
  • Internal services need a service mesh once you have >~10 services. Linkerd if simple, Istio if you need policy.
  • API Gateway is for north-south concerns (3rd-party clients). Don't put it in front of internal-only services.
  • L4 vs L7: L4 for raw TCP/UDP, low latency, no HTTP smarts. L7 for path/host routing.
  • WS/gRPC: Envoy is the safest pick.
  • Hot static path = CDN. Personalized = origin.
  • Rate limiting: cheap blanket at CDN; per-key at API Gateway; per-service at mesh.
  • If you're picking Nginx vs Envoy: pick Nginx unless you need dynamic config (xDS) or are running a mesh.
  • Anycast at the CDN solves failover faster than DNS.
