Caches: Comparison & Tradeoffs

"Cache" means at least four different things: a remote KV in front of a database, an in-process LRU inside the app, an HTTP cache in a reverse proxy, and a CDN cache at the edge. Picking the wrong layer is the most common cache mistake.

6. Migration paths

Cache migrations are easier than DB migrations — cache is recoverable from source. The pain is hit-rate during cutover.

  • Memcached → Redis — switch the client library; let warm fill from DB. Optional: dump/restore (less common). Watch for slab semantics: Memcached's per-key size limit is 1 MB, Redis is more generous but hot keys are still dangerous.
  • Redis (single) → Redis Cluster — introduce hashtags before the migration; live-migrate slots; clients must support cluster mode.
  • Caffeine → Redis — usually the trigger is "we need cross-pod consistency." Add Redis as the canonical cache; Caffeine becomes L1 in front of Redis.
  • App-cache → Varnish — move HTTP cacheability into Cache-Control headers; let Varnish sit in front of origin. Test with curl + Age/X-Cache headers.
  • No CDN → CDN — the win is huge but VARY:Cookie kills hit rate. Normalize before turning on.
  • CDN A → CDN B — DNS cutover after warming both; surrogate keys / purge APIs differ.

8. Hybrid patterns ("layers stack")

L1 in-process + L2 Redis (the classic)

Caffeine in each pod for sub-microsecond hits on hottest keys; Redis as L2 for cross-pod sharing. L1 TTL short (seconds); L2 TTL longer (minutes). On L1 miss, single-flight to Redis; on L2 miss, single-flight to DB.

CDN + Varnish + Redis + DB (4-layer web)

Edge CDN for assets & cacheable GETs; Varnish in DC for personalized-but-cacheable; Redis cache-aside for DB rows; DB as source of truth. Each layer protects the next.

Redis as both cache and pub/sub for invalidation

One Redis cluster: keys for the cache, pub/sub channel for "invalidate user:123". Each pod listens, drops Caffeine entries on signal. Solves "stale L1 after another pod's write."

CDN with surrogate keys / cache tags

Each response tagged with surrogate keys (e.g., user-123,product-456). On write, app calls CDN purge by tag; one API call invalidates many URLs. Fastly & Cloudflare native; CloudFront via Lambda@Edge.

Negative cache with short TTL

404s and "not found" results cached for ~1 min. Without this, missing keys hammer the DB during scrapes. Layer it where the misses originate.

Read-through cache library

Caffeine's LoadingCache, Spring Cache, Go's groupcache — the cache itself fetches on miss. Encapsulates single-flight + populate. Avoids duplicate cache-aside logic.

9. Anti-patterns

"We used Redis as the source of truth"

Eviction silently deletes data. AOF=everysec loses up to 1s of writes on crash. Cluster slot migrations have edge cases. Lesson: if you can't lose it, durable store first; cache on top.

"We didn't TTL anything"

Cache fills, evicts unpredictably, hot vs cold blur. Always TTL. Use jittered TTLs to spread expiry.

"All keys expire at the same time"

Stampede — thousands of misses concurrent. Add jitter (TTL = base + random(0, base*0.2)).

"We cached without single-flight"

10,000 requests for the same missing key → 10,000 DB queries. Use Redis SETNX, Caffeine refreshAfterWrite, or app-level mutex.

"VARY: Cookie at the CDN"

Each user gets their own cache entry → CDN hit rate ≈ 0. Either don't cache personalized pages or use edge functions to compute.

"Cache stampede on deploy"

New deploy invalidates Caffeine on every pod restart; concurrent fills hammer Redis or DB. Use rolling restarts; pre-warm cache; honor Cache-Control: stale-while-revalidate.

"Hot key > one node"

One key hitter pegs the shard; entire cluster looks fine but P99 explodes. Replicate the key (write to multiple suffixes), or front it with in-process cache.

"Negative cache forever"

404 cached for 1 hour means the new resource is invisible for 1 hour after it's created. Negative TTL < positive TTL.

10. Decision rules for interviews

  • Pick the layer first, then the product. CDN → reverse proxy → remote KV → in-process. The same logic chooses different products at each layer.
  • "Cache" defaults to Redis cache-aside in front of DB. Memcached only when "max QPS, pure ephemeral" is mandatory.
  • Always say cache-aside vs write-through vs write-behind on purpose. Defaults: cache-aside for reads; write-through if cache-miss is unacceptable; write-behind only with durability story.
  • TTL everything; jitter the TTL.
  • Single-flight on miss; Caffeine has it built-in, Redis needs Lua/SETNX.
  • Hot key > one node = replicate the key or front with in-process L1.
  • Personalized HTTP responses → don't put them in the CDN; put a Redis tier behind it.
  • Invalidation: prefer event-driven (pub/sub or CDN purge) over short TTL.
  • Negative cache shorter than positive cache.
  • If cache is required for correctness, you have a bug. Cache is an optimization — the system must work when cache is empty.
