CDC & ETL: Comparison & Tradeoffs

"How do I get data from the production DB to the warehouse?" has two answers: log-based CDC (Debezium / Maxwell / Canal / cloud DMS) for low-latency, ordered, complete capture; query-based / managed connectors (Fivetran / Airbyte) for SaaS sources and batch comfort. Pick by source type and freshness need.

6. Migration paths

  • Polling → CDC: common modernization. Replace nightly batch dumps with Debezium/DMS for sub-second freshness.
  • Self-host Debezium → managed (DMS / Datastream): when ops cost outweighs control.
  • Fivetran → Airbyte: cost-driven; works for many DB sources but fewer SaaS connectors are battle-tested.
  • Custom dual-write → outbox + Debezium: the canonical "we had data inconsistencies" rewrite.
  • DMS → Zero-ETL (Aurora → Redshift): AWS now supports zero-ETL for some pairs — eliminates the replication-instance entirely.

8. Hybrid patterns

Outbox + Debezium (the gold standard)

App writes business row + outbox row in one DB transaction. Debezium tails the WAL/binlog, publishes outbox rows to Kafka. Solves dual-write atomicity.

CDC → warehouse + ops cache

One Debezium stream forks to two sinks: warehouse (analytics) and Redis / DDB (ops cache). CQRS via CDC.

Debezium → Flink → materialized view

Stream processor builds a projection (denormalized read model) and writes to a serving DB. Read scale + denormalization without dual-writes.

Fivetran for SaaS + Debezium for DB

Most teams end up with two CDC layers: Fivetran for the long tail of SaaS data sources; Debezium for the production DBs.

Reverse ETL (Hightouch / Census)

Take warehouse-computed segments and push them back into ops tools (Salesforce, HubSpot, Iterable). Bidirectional — CDC out, Reverse ETL in.

Snowpipe Streaming

Snowflake's row-stream API ingests Kafka topics directly into Snowflake. Skip the warehouse-load step entirely for streaming sources.

9. Anti-patterns

"Dual-write to DB and Kafka"

Inevitable inconsistencies. Use outbox + CDC.

"Polled the DB every 30s"

Misses deletes; misses fast-changing rows; loads the DB. Switch to CDC.

"Unmonitored replication slot"

Connector dies; slot accumulates WAL; PG runs out of disk. Always monitor slot lag.

"Trust at-least-once without idempotent sink"

Duplicate inserts in warehouse; analytics off. Sinks must be idempotent (upserts on PK).

"Schema change broke the pipeline"

No schema registry; downstream consumers blind to ALTER. Use Confluent Schema Registry or equivalent + compat policies.

"Used CDC for app-to-app (instead of events)"

Coupling: every consumer reads the DB schema. Better to publish domain events from the app.

"Ran Fivetran on a DB without read replica"

Fivetran's full-table queries hammer prod. Use a read replica or switch to log-based CDC.

10. Decision rules for interviews

  • DB → downstream: Debezium (self-host) or DMS / Datastream (managed). Always log-based, never polling.
  • SaaS → warehouse: Fivetran or Airbyte. Pick by cost / connector breadth.
  • Outbox + Debezium for "DB and event bus must agree."
  • CDC sinks must be idempotent. At-least-once + idempotent = exactly-once.
  • Always monitor replication slot / binlog lag.
  • Use a schema registry for streaming events.
  • Don't use CDC for app-to-app integration; publish domain events instead.
  • Reverse ETL is a different shape; use Hightouch / Census, not custom Airflow.
  • Initial snapshot of huge tables is the slowest part — plan parallel snapshot.
  • Stream processor (Flink) on top of CDC builds the read-side projections.
