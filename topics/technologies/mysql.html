MySQL (InnoDB)

Relational SQL DB whose default storage engine InnoDB stores rows in a clustered B+tree on the primary key — the table is the index. Undo log gives MVCC, redo log gives crash recovery, binlog gives replication. The most-deployed open-source DB in the world; the web stack's default.

SQL • clustered-PK lookups • binlog replication

2. Underlying mechanism

The single biggest difference from Postgres: rows live in the leaves of the primary-key B+tree. There is no separate "heap." Look up by PK = traverse one B+tree to the row. Look up by secondary index = secondary tree returns PK, then primary tree returns row (two B+tree walks).

3. Read & write path

Write path

1. Acquire row lock (gap lock for repeatable read isolation)
2. Write change to undo log (for rollback + MVCC)
3. Modify the row (in-memory page in buffer pool)
4. Append to redo log (in memory)
5. On COMMIT: redo log fsync to disk; binlog fsync
   (sync_binlog=1, innodb_flush_log_at_trx_commit=1)
6. Replicas pull binlog and replay
7. Buffer pool dirty pages flushed lazily by background threads

Read path

PK lookup:
  Walk PK B+tree → row (one tree walk)

Secondary index:
  Walk secondary tree → PK
  Walk PK tree → row
  (Covering index returns from secondary tree directly)

Range:
  Find leaf, walk via sibling pointers

Replication

primary: writes to binlog
replica's IO thread: pulls binlog → relay log
replica's SQL thread: replays relay log
GTID since 5.6: each event has unique global tx id

modes: async (default) | semi-sync | group replication

Crash recovery

on startup:
  redo log replay applies committed-but-not-flushed pages
  undo log rolls back uncommitted txns
  buffer pool gets warmed slowly via warmup file (5.7+)

4. Replication & consistency

  • Topology: single-leader binlog replication; cascading + multi-replica common.
  • Modes: async (default), semi-sync (primary waits for at least one replica ack), Group Replication (Paxos-based; multi-primary or single-primary with auto-failover).
  • Consistency: linearizable on primary; replicas eventually consistent. Read-after-write requires session affinity or WAIT_FOR_EXECUTED_GTID_SET.
  • Failover: manual unless using Orchestrator / MHA / cloud-managed (RDS / Aurora). Group Replication enables auto-failover.
  • GTID: 5.6+; replaces "binlog file + position" with globally-unique transaction IDs — mandatory for any modern setup.

5. Partitioning

  • Native: declarative partitioning (range / list / hash / key); same-table abstraction.
  • Sharding: not native. Vitess is the gold standard (YouTube origin, used by Slack / Square / GitHub). App-level sharding also common.
  • Hot row: InnoDB row-level locks; gap locks under REPEATABLE READ. Avoid hot-spot contention via redesign.
  • PK choice matters: auto-increment INT/BIGINT keeps inserts at the end of the B+tree (cheap). Random UUIDs cause page splits all over the tree (expensive); use UUIDv7 (time-ordered) or convert to BINARY(16).

7. Per-node capacity envelope

  • OLTP TPS: ~10–30k for simple ops on tuned hardware.
  • P99 latency (LAN): 1–5 ms point reads.
  • Connections: ~500 direct; thousands via ProxySQL.
  • Storage: 10s of TB single instance; one big buffer pool.
  • Throughput limit: redo log + binlog fsync rate on commit.

8. When to use / not to

Use MySQL when…

  • Web stack default; PHP / Rails / Node ecosystems lean MySQL.
  • Workload dominated by clustered-PK access (e.g., row-by-id).
  • You already have MySQL ops expertise.
  • Need Vitess for SQL sharding at scale.
  • Aurora MySQL for cloud-native.

Don't use MySQL when…

  • You need Postgres-class indexing (GIN / BRIN / partial / expression).
  • Need rich extensions (Postgres has more).
  • Heavy JSONB workloads (Postgres is more mature).
  • Cross-region strong consistency (use Spanner / CRDB).
  • OLAP — use ClickHouse / BigQuery.

9. Common gotchas

  • Random UUID PK → page splits + B+tree fragmentation. Use auto-inc or UUIDv7.
  • Gap locks under REPEATABLE READ can deadlock unexpectedly with mixed read/write txns.
  • Implicit type conversion in WHERE (e.g., string vs int) silently loses index usage.
  • Online DDL but not always online. Some ALTERs lock; use pt-online-schema-change or gh-ost.
  • Secondary index is "PK + secondary key." Big PK = bigger secondary indexes.
  • Replication lag spikes on big single-statement updates (binlog replays serially without parallel slave).
  • InnoDB row format matters for column count / NULL ratio; default DYNAMIC is usually right.
  • SHOW PROCESSLIST is one snapshot; use performance_schema for real diagnosis.
  • SELECT * with TEXT/BLOB → off-page fetch each row.
  • Group Replication split-brain with bad network; understand group_replication_consistency.

10. Cloud equivalents

  • AWS: RDS MySQL; Aurora MySQL (log-as-DB; storage decoupled from compute).
  • GCP: Cloud SQL MySQL.
  • Azure: Azure DB for MySQL Flexible Server.
  • Open-source: MySQL, MariaDB (fork), Percona Server (drop-in).
  • Sharding: Vitess (open-source, used at YouTube/Slack), PlanetScale (managed Vitess).

12. Usage patterns

  • Primary OLTP — users, orders, inventory.
  • Auto-increment ID generation — classic ID source.
  • Write-heavy with INSERT ... ON DUPLICATE KEY UPDATE for upserts.
  • Vitess sharding — horizontally scale beyond one box.
  • CDC via Debezium / Maxwell / Canal reading binlog.
  • Read-heavy workload with replicas + ProxySQL routing.
  • FULLTEXT index for basic full-text (replace with ES at scale).
  • Spatial — native types and indexes for geo (less mature than PostGIS).
  • JSON column — native since 5.7; less indexable than Postgres jsonb.
  • Outbox pattern — outbox table + Debezium on binlog.

13. Operational knobs that matter

  • innodb_buffer_pool_size — ~70% of RAM. The single most important setting.
  • innodb_log_file_size — bigger redo log = fewer flushes; modern default is fine but tune for write-heavy.
  • innodb_flush_log_at_trx_commit — 1 (durable, default), 2 (lose 1s on crash), 0 (lose more).
  • sync_binlog — 1 (durable, slowest), 0 (let OS), N (group commit).
  • binlog_format — ROW (default, recommended for replication safety), STATEMENT (older; risky with non-deterministic statements).
  • gtid_mode — ON always.
  • max_connections — hundreds direct; use ProxySQL past that.
  • innodb_flush_method — O_DIRECT on Linux to avoid double-buffer with OS page cache.
  • transaction_isolation — default REPEATABLE READ; READ COMMITTED is more Postgres-like and avoids gap locks.

14. Capacity planning

  • RAM: buffer pool fits hot data + working set. If > RAM, expect IO-bound performance.
  • Storage: 30% headroom for online DDL + temp tables.
  • Connections: 500 direct typical; 5000+ via ProxySQL.
  • Redo log: large enough to absorb peak write rate without checkpoints stalling.
  • Replication: replica must be at least as fast as primary to keep up.

15. Failure modes

  • Primary crash: replicas have lag; semi-sync minimizes data loss; manual or orchestrator failover.
  • Replica falls behind: SQL thread CPU-bound on big statement; consider parallel slave.
  • Disk full: stops writes; clear binlog or expand disk.
  • Long-running transaction: bloat undo log; history list length grows.
  • Slave gets ahead via direct write: split-brain; use read_only=ON on replicas.
  • Group Replication network partition: minority offline; majority continues.

16. Backup & DR

  • Logical: mysqldump; slow restore.
  • Physical: xtrabackup (Percona); fast, parallel, supports PITR via binlog replay.
  • RPO: binlog interval; near-zero with continuous archive.
  • RTO: bound by restore speed; 1 TB takes hours.
  • Cross-region DR: async replica or RDS / Aurora cross-region replica.

17. Upgrade & migration

  • Minor: in-place; replicas first.
  • Major (5.7 → 8.0): in-place upgrade or rebuild from logical dump; check deprecated features.
  • Online schema change: pt-online-schema-change / gh-ost; build a shadow table, copy data, atomic rename.
  • Sharding migration: introduce Vitess; unsharded keyspace first; gradually shard tables.
  • Migrate to Postgres: Debezium + dual-write; DDL differences (auto-increment vs sequence; index types).

18. Security model

  • Auth: caching_sha2_password (8.0 default), pluggable plugins (LDAP, AD, IAM on RDS).
  • Authz: GRANT system; column-level GRANT exists.
  • TLS: optional for client + replication; mandatory in cloud-managed.
  • Encryption at rest: InnoDB tablespace encryption (8.0).
  • Audit: MySQL Enterprise Audit (commercial); MariaDB Audit Plugin (FOSS).
  • Network: bind to private subnet; never expose 3306 publicly.

19. Multi-tenancy

  • Schema-per-tenant: CREATE DATABASE tenant_42; clean isolation; doesn't scale to thousands.
  • Row-level (shared schema): tenant_id column + per-query filter; most common pattern.
  • Vitess keyspaces: one keyspace per tenant tier with its own sharding.
  • Resource: max_user_connections per user; no native CPU/IOPS isolation.

20. Observability

  • QPS + p99 (performance_schema).
  • Replica lag (Seconds_Behind_Source) — coarse; use pt-heartbeat for accuracy.
  • Buffer pool hit ratio — aim > 99%.
  • Innodb row lock waits.
  • Long-running transactions (information_schema.innodb_trx).
  • Slow query log + pt-query-digest.
  • Connection count vs max_connections.
  • Disk usage per database / largest tables.

21. Anti-patterns

  • Random UUID PKs (page split storms).
  • SELECT * on wide tables (off-page TEXT/BLOB).
  • Heavy single-statement DML on replicated table (replica lag).
  • Many small connections without pooling.
  • Implicit charset/collation conversions losing indexes.
  • Multi-statement transactions without idempotency.
  • Group Replication on flaky network → group thrash.
  • Binlog disabled (loses replication + PITR).

22. Cost model

  • RDS / Cloud SQL: instance + storage + IOPS-provisioning.
  • Aurora MySQL: compute + storage GB + I/O ops; I/O ops can dominate hot workloads.
  • Cross-AZ replication: separate line item.
  • Backup: extra storage cost.
  • Self-host: hardware + on-call.
