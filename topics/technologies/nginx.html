Nginx

Async, event-driven, single-master multi-worker reverse proxy / web server / load balancer / HTTP cache. Each worker is a single thread with an epoll/kqueue loop handling thousands of connections. Static config (reload-required) but extremely battle-tested. The most-deployed reverse proxy in the world.

async event loop • reverse proxy • L4/L7 • nginx.conf

2. Underlying mechanism

  • One master process + N worker processes (one per CPU core typical).
  • Each worker runs an epoll/kqueue event loop: many connections per worker, no thread-per-connection.
  • Master handles signals (reload, upgrade); workers do the actual I/O.
  • Reverse proxy via proxy_pass; load-balance across upstream; supports L4 (stream module) and L7 (http module).
  • HTTP cache via proxy_cache with a file-system cache zone.
  • Config is static; nginx -s reload spawns new workers, drains old.
  • Modules compiled in (or dynamic in 1.9+); Lua via OpenResty for scripting.

3. Read & write path

Reverse proxy request

1. TCP accept on listen port
2. Parse HTTP/1.1, HTTP/2, HTTP/3 (with module)
3. Match server block (server_name) + location
4. Apply rewrite / auth / rate-limit / etc.
5. proxy_pass to upstream:
   - DNS resolve (or use defined upstream)
   - LB algorithm: round-robin / least_conn / ip_hash / hash
   - keep-alive pool to backend
6. Stream response back; gzip, header-mutate
7. Optionally proxy_cache HIT → skip backend

Static file serving

1. Request mapped to filesystem path
2. open() + sendfile() — zero-copy from disk to socket
3. AIO + thread pool for disk read on Linux
4. Range requests handled natively
5. ~50k req/sec/worker for small static files

HTTP cache

proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=z:10m max_size=100g;

On request:
1. Compute cache key (URL + Vary)
2. Lookup in keys_zone (in-memory)
3. HIT: serve from cache file
4. MISS: proxy_pass; on response, store
5. proxy_cache_lock prevents stampede on miss

Reload (zero-downtime)

SIGHUP to master
1. Master parses new config (validates first)
2. Spawns new workers with new config
3. Old workers stop accepting new conns
4. Old workers drain existing conns
5. Old workers exit when done

For binary upgrade: SIGUSR2 + SIGWINCH

4. "Replication" (HA topology)

  • Nginx itself is stateless; HA is "run multiple instances behind a cloud LB or DNS."
  • Active-active: N Nginx in a target group; cloud LB distributes.
  • Active-passive: keepalived + VRRP for VIP failover (on-prem).
  • State (caches, rate-limit counters) is per-instance unless shared (Redis-backed via Nginx Plus or OpenResty Lua).

5. Partitioning / load balancing

  • Algorithms: round-robin (default), least_conn, ip_hash, hash $key, random with two-choices.
  • Sticky sessions: ip_hash, sticky cookie (Plus), hash on consistent.
  • Health checks: passive (fail count from real traffic) by default; active checks in Plus or OpenResty.
  • Upstream blocks define backend pools; weights for traffic split.
  • Slow start: gradual ramp-up after backend recovery (Plus only).

6. Time complexity

  • Per request: O(1) routing on small server config; O(log N) on large location regex sets.
  • Cache lookup: O(1) on key-zone hash.
  • SSL handshake: TLS-bound; session resumption avoids re-handshake.
  • Concurrent connections per worker: bounded by FD limit + epoll efficiency.

7. Per-node capacity envelope

  • ~50k requests/sec/worker on simple proxy (cached static); fewer with TLS handshake-heavy traffic.
  • ~100k+ concurrent connections per worker with sufficient FDs.
  • Latency overhead: ~0.1 ms typical.
  • Memory: ~5 MB / worker + buffers per connection.
  • TLS termination CPU: 30–70% of workload at high RPS without offload.

8. When to use / not to

Use Nginx when…

  • Default L7 reverse proxy / HTTP load balancer.
  • Static file serving + HTTP cache.
  • k8s ingress (Ingress-NGINX).
  • TLS termination + simple routing.
  • Familiar config; large org expertise.

Don't use Nginx when…

  • Need dynamic routing config (no reload) → Envoy + xDS.
  • Service mesh sidecar → Envoy / linkerd-proxy.
  • Pure L4 with millions of pps → cloud NLB / IPVS.
  • Edge compute → Cloudflare Workers / Fastly.
  • API gateway with quota / billing → Kong / managed API GW.

9. Common gotchas

  • proxy_pass trailing slash semantics — with vs without slash changes URI.
  • Default 60s idle timeout disconnects WebSocket; bump proxy_read_timeout.
  • DNS resolve cached at startup for upstream blocks unless you use resolver + variable.
  • Buffering: large request/response bodies buffered to disk by default; tune for streaming.
  • Worker connections cap: increase FD ulimit + worker_connections.
  • TLS session reuse: enable ssl_session_cache shared across workers.
  • Reload — not a graceful drain; clients with long-lived conns may see brief errors.
  • Lua + OpenResty integrates well but adds latency per request.
  • OCSP stapling required for performant TLS.
  • $host vs $server_name vs $http_host — subtle differences.

10. Cloud equivalents

  • Self-host: Nginx (open-source) or Nginx Plus (commercial).
  • Cloud LB analogs: ALB (AWS), GCLB (GCP), Application Gateway (Azure) — not Nginx but solve same L7 problem.
  • Open-source alternatives: HAProxy, Envoy, Traefik, Caddy.
  • Distros: OpenResty (Nginx + LuaJIT + modules); Tengine (Alibaba fork).

12. Usage patterns

  • L7 reverse proxy in front of app servers.
  • Static file serving + sendfile().
  • SSL termination.
  • WebSocket proxy.
  • HTTP cache (proxy_cache).
  • Rate limiting (req_zone, conn_zone).
  • Authentication via auth_request sub-request to auth service.
  • k8s Ingress.
  • API gateway-ish (with OpenResty + Lua); Kong is built on this.
  • Geo redirection / locale routing.

13. Operational knobs

  • worker_processes auto + worker_connections.
  • proxy_buffer_size, proxy_buffers.
  • keepalive in upstream + proxy_http_version 1.1 + proxy_set_header Connection "" for backend connection reuse.
  • proxy_cache_path + proxy_cache_key + proxy_cache_use_stale.
  • SSL: ssl_session_cache shared:SSL:50m; OCSP stapling.
  • Limit modules: limit_req_zone / limit_conn_zone.
  • resolver for DNS-based upstream.
  • Logging: log_format + access_log with buffer + flush.

14. Capacity planning

  • Workers = CPU cores; usually auto.
  • FD limit + worker_connections caps total concurrent.
  • Memory: connections × buffers + cache zone size.
  • Disk: if proxy_cache, plan zone size + IOPS for the cache tier.
  • Network: TLS handshake at 1k+ RPS = significant CPU; consider TLS offload tier.

15. Failure modes

  • Backend down: Nginx 502; proxy_next_upstream retries another upstream.
  • FD exhaustion: 503 / dropped conns.
  • Cache full: LRU eviction, may evict hot keys if zone too small.
  • DNS resolution fail: upstream unavailable (use resolver with valid=).
  • Reload fail: master keeps old config + workers; check nginx -t first.
  • OOM under burst: tune buffers; raise file limits.

16. Backup & DR

  • Nginx is stateless; "backup" = git-versioned config.
  • Cache is rebuildable from origin.
  • DR: deploy in another region behind a global LB.
  • Logs to S3 / log aggregator.

17. Upgrade & migration

  • Binary upgrade with SIGUSR2 + SIGWINCH (zero-downtime).
  • Config reload with nginx -t validate → nginx -s reload.
  • Migrate from Apache: rewrite rules need translation; mod_rewrite → rewrite directives.
  • Migrate to Envoy: typically when joining a service mesh; rewrite config to xDS YAML.

18. Security

  • TLS termination with modern cipher suites (configurable).
  • HTTP/2 + ALPN.
  • HSTS / CSP / security headers via add_header.
  • Rate limiting + connection limits as DDoS first-line.
  • auth_request for OIDC / JWT validation.
  • WAF: ModSecurity v3 module or commercial.
  • mTLS via ssl_verify_client.

19. Multi-tenancy

  • Server blocks per host / SNI.
  • Per-server limits (limit_req_zone keyed by tenant header).
  • No native tenant isolation beyond config.
  • Pattern: per-tenant upstream blocks; routing by header.

20. Observability

  • Access log + error log (parse with Goaccess / Filebeat / OpenTelemetry).
  • stub_status / Plus API for live metrics.
  • Prometheus exporter (nginx-prometheus-exporter or VTS module).
  • Active connections, request rate, response time, upstream errors.

21. Anti-patterns

  • Single Nginx for everything (auth, rate-limit, routing, cache, TLS) → tightly-coupled config.
  • Default 60s idle timeout breaking WebSocket.
  • DNS-resolve at boot; backend IPs change → stale.
  • VARY: Cookie at proxy_cache → hit rate near 0.
  • Hot-reloading on every config change in CI → resource churn.
  • No proxy_cache_lock → cache stampede.
  • Buffering streaming responses (block until full).

22. Cost model

  • Self-host: instance hours.
  • Plus: per-instance license fee.
  • Cloud LB above (NLB/ALB) is the additional tier cost.
  • Disk for proxy_cache.
