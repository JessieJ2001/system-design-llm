PostgreSQL

Relational SQL DB with multi-version concurrency control (MVCC), B+tree indexes over a heap-organized table, write-ahead log (WAL) for durability and replication, process-per-connection model, and the richest extension ecosystem of any open-source DB (PostGIS, pgvector, TimescaleDB, Citus).

SQL • ACID • the default relational database

2. Underlying mechanism

Three concepts explain almost everything Postgres does: heap + B+tree, MVCC via xmin/xmax, and WAL streaming.

Heap + B+tree (no clustered index)

A row lives in the heap, addressed by ctid (page, slot). B+tree indexes hold (key → ctid). PG is not like MySQL InnoDB — rows aren't stored in PK order. Every index is a "secondary" index. Fetch path: B+tree → ctid → heap. With HOT updates, an index can point to a chain of versions that share one heap page.

MVCC via xmin/xmax

Each tuple has xmin (creating txid) and xmax (deleting txid). UPDATE = mark old xmax + insert new tuple. Readers compare against their snapshot's visible txid range — never block writers. Cost: dead tuples accumulate; VACUUM reclaims; on long-running transactions, dead tuples can't be vacuumed.

WAL = the everything log

Every page change goes to WAL before the change is applied. WAL is the basis for crash recovery, streaming replication, logical decoding (CDC), and PITR via archive_command + base backup.

Process-per-connection

Each client connection is a forked Postgres process (~10 MB each). At > 200–300 connections, context switching dominates and you must use a connection pooler (PgBouncer).

3. Read & write path

Write (UPDATE example)

1. Acquire row lock on existing tuple
2. Mark old tuple's xmax = my_txid
3. Insert new tuple with xmin = my_txid
4. Append WAL records (BEFORE the page changes are visible)
5. Update each affected index (HOT update may avoid this)
6. On COMMIT: WAL fsync; tuple becomes visible
7. Bgwriter flushes dirty pages later (asynchronous)
8. VACUUM later marks old tuple slot reusable

Read (SELECT by indexed key)

1. Plan: Index Scan or Bitmap Index Scan
2. Walk B+tree to leaf
3. Get ctid; read heap page (or visibility map check)
4. Re-check visibility (xmin/xmax vs my snapshot)
5. Return tuple

Streaming replication

primary fsyncs WAL → walsender ships records to replica
replica's startup process replays WAL records
replicas can serve hot-standby reads (with replay-lag)
sync_commit = on / remote_apply for stronger durability

VACUUM

autovacuum (default): scans tables, reclaims dead tuples
- updates visibility map
- prevents transaction-id wraparound (XID exhaustion)
- VACUUM FULL rewrites the table (locks!)
Long-running txn = VACUUM cannot reclaim past its xid horizon

4. Replication & consistency

  • Topology: single-leader streaming WAL to N replicas. Optional cascading (replica-of-replica).
  • Modes: async (default; minimal latency, possible data loss on primary loss); sync via synchronous_commit = remote_apply for zero-RPO at higher latency.
  • Logical replication: publish/subscribe by table; useful for cross-version upgrades, multi-master patterns (BDR / pglogical), CDC (Debezium reads via logical decoding).
  • Failover: manual unless using Patroni / Stolon. Promotion triggers timeline switch.
  • Consistency: linearizable on the primary; replicas serve reads with lag (typically < 100 ms but can spike).

5. Partitioning

  • Native: declarative partitioning (range / list / hash) since PG 11. Partitions are first-class; query planner does pruning.
  • Sharding: not native. Options: Citus extension (coordinator + workers), Vitess-for-PG, app-level routing, or change DB to CockroachDB.
  • Rebalance: add a hash partition = re-create with new modulus + redistribute. Citus has online shard moves.
  • Hot row: row-level lock contention → latency spikes. Mitigations: SELECT ... FOR UPDATE SKIP LOCKED (queue pattern), advisory locks, or redesign to append + rollup.

8. When to use / not to

Use Postgres when…

  • Default. The relational answer.
  • Need joins, ACID, foreign keys, transactions.
  • Mixed workload: OLTP + JSONB documents + full-text + geo + vector via extensions.
  • Need rich indexing: GIN, GIST, BRIN, partial, expression, covering.
  • Strong constraints + check / exclusion constraints matter.

Don't use Postgres when…

  • Need to shard SQL across many nodes — switch to Spanner / CockroachDB / Vitess / Citus.
  • Heavy append-only event stream past ~10k/sec sustained — use Cassandra / Kafka.
  • Sub-ms latency at high QPS — need a cache (Redis) on top.
  • Multi-region writes with strong consistency — Spanner / CRDB.
  • Working set fits in RAM and you only do KV — Redis is faster.

9. Common gotchas

  • Connection storms. Use PgBouncer past ~200 conns.
  • Long-running transactions. Block VACUUM → bloat → XID wraparound risk. Monitor pg_stat_activity for old backend_xmin.
  • Bloat. Heavy UPDATE/DELETE creates dead tuples. Tune autovacuum; consider fillfactor < 100 for HOT updates.
  • Missing indexes on FK columns. DELETE on parent does N seq-scans on children.
  • SELECT * on wide rows. Pulls TOAST'd large fields you don't need.
  • Massive WHERE x IN (1, 2, ..., 50000). Plan-time blowup; use a temp table or VALUES + JOIN.
  • count(*) on huge tables. Always seq-scan (no clustered index). Use approximate with pg_class.reltuples.
  • Uncovered ORDER BY ... LIMIT N over large set. Sort-on-the-fly. Add covering index.
  • Logical replication and DDL. DDL doesn't replicate by default; coordinate carefully.
  • Replication slot left behind. WAL accumulates; primary disk fills up.

10. Cloud equivalents

12. Usage patterns

  • Primary OLTP store — users, orders, accounts. Default.
  • JSONB document store — jsonb column + GIN index; pseudo-Mongo without separate system.
  • Full-text search — tsvector + GIN index; good to ~10M docs.
  • Geo — PostGIS extension; production-grade GIS.
  • Vector — pgvector extension; HNSW or IVFFlat for ANN; OK at < 10M vectors.
  • Time-series — TimescaleDB extension; hypertables = time-partitioned chunks.
  • Pub/sub — LISTEN / NOTIFY; small payloads, single-node only.
  • Job queue — SELECT ... FOR UPDATE SKIP LOCKED queue table.
  • Outbox pattern — outbox table + Debezium tails WAL; canonical "DB and Kafka must agree."
  • Idempotency keys — UNIQUE constraint on idempotency-key column.
  • Materialized views — precompute expensive aggregates; refresh CONCURRENTLY.

13. Operational knobs that matter

  • shared_buffers — 25% of RAM typical; PG cache layer.
  • effective_cache_size — planner hint for OS cache; ~75% of RAM.
  • work_mem — per-sort/hash; per operation; total = work_mem × max_connections × ops.
  • maintenance_work_mem — for VACUUM / CREATE INDEX; bigger = faster index builds.
  • max_connections — cap at 200–300 + use PgBouncer.
  • synchronous_commit — on / off / remote_apply; commit-latency dial.
  • wal_level — replica for streaming, logical for CDC.
  • max_wal_size — checkpoint trigger; bigger = fewer checkpoints + more recovery time.
  • autovacuum_* — per-table tuning for hot tables (smaller scale_factor).
  • fillfactor — for tables with heavy updates (e.g., 80) to enable HOT.

14. Capacity planning rules-of-thumb

  • Connections: (2 × cores) + spindles for direct connections. Past ~200, mandatory PgBouncer.
  • RAM: shared_buffers + (work_mem × max_connections × expected ops) must fit comfortably with OS cache reserve.
  • Storage: include 30% bloat headroom + same-size space for VACUUM FULL if you ever need it.
  • WAL: SSD with sustained write throughput; archive_command + storage for PITR window.
  • Replicas: 1 sync + 1–2 async typical; sync replica must be fast or it slows the primary.
  • Backups: base backup + WAL archive; enables PITR. Test restores quarterly.

15. Failure modes & blast radius

  • Primary crash: with sync repl, replica has all committed data → promote, RPO=0. With async, lose recent commits.
  • Storage full from WAL: usually a stuck replication slot. Drop slot or grow disk; PG halts writes when WAL can't be written.
  • VACUUM falling behind: bloat + XID wraparound risk; at < 1B txns left, PG goes read-only to protect data. Monitor txid_age.
  • Long-running query: blocks VACUUM; may also hold locks blocking other writes.
  • Replication slot left by dropped replica: WAL accumulates → disk full.
  • Connection storm: each connection ~10 MB; OOM at thousands.
  • Slow disk: WAL fsync latency → commit latency. Watch for noisy-neighbor on cloud disks.

16. Backup, restore & DR

  • Logical backup: pg_dump / pg_restore. Slow restore for large DBs; preserves data only (not WAL).
  • Physical backup: pg_basebackup + WAL archive → PITR to any second.
  • Tools: pgBackRest, Barman, WAL-G — production-grade with parallel restore + retention.
  • RPO: WAL archive interval (typically minutes); near-zero with continuous archive.
  • RTO: bound by restore time = data size / restore-disk-write speed. 1 TB ≈ 1–2 hours.
  • DR: async replica in another region; promote on disaster. Or a managed cross-region replica.
  • Drill cadence: quarterly restore-from-backup to a fresh instance; verify SELECT count(*) on key tables.

17. Upgrade & online migration

  • Minor versions (e.g., 16.1 → 16.2): binary-compat; dump and restart. Replicas first.
  • Major versions (e.g., 14 → 16): on-disk format may change.
    • pg_upgrade — with --link for in-place hardlinks; minutes of downtime.
    • Logical replication for zero-downtime: set up logical replication from old → new; cut over once caught up. Schema must be compatible.
  • Schema migration: Liquibase / Flyway / Alembic / sqitch. Always additive first (add column nullable), backfill, then drop old.
  • Online index build: CREATE INDEX CONCURRENTLY — no table lock but takes longer.
  • Renaming a column safely: add new column → backfill → dual-write → cut reads → drop old column.

18. Security model

  • Authentication: password (md5 / SCRAM-SHA-256), peer, cert (mTLS), GSSAPI (Kerberos), LDAP, IAM (AWS RDS).
  • Authorization: roles + GRANT (object-level); row-level security (RLS) policies for per-row access.
  • Encryption in transit: SSL/TLS via ssl=on + pg_hba.conf hostssl entries.
  • Encryption at rest: not native; rely on disk encryption (LUKS, EBS encryption); pgcrypto for column-level.
  • Audit: pgAudit extension; logs DDL / DML at session level.
  • Network: never expose 5432 to the internet; bind to private subnet; firewall.

19. Multi-tenancy & isolation

  • Schemas per tenant: CREATE SCHEMA tenant_42; isolation per schema search_path. Doesn't scale past hundreds of schemas (catalog bloat).
  • Row-level (shared schema): tenant_id column + RLS policies. Most common pattern.
  • Database per tenant: heavyweight; but maximum isolation.
  • Resource isolation: per-role statement_timeout, idle_in_transaction_session_timeout; no per-tenant CPU/IOPS native (use cgroups at OS level).
  • Pattern at scale: shared schema + RLS + row-level tenant_id, with monitoring per-tenant; large tenants get dedicated instance.

20. Observability — what to monitor

  • TPS + p99 query duration (from pg_stat_statements).
  • Connections / max_connections.
  • Replication lag (bytes via pg_wal_lsn_diff; seconds via pg_last_xact_replay_timestamp).
  • Bloat per table (use pgstattuple or check_postgres).
  • Cache hit ratio (pg_statio_user_tables); aim > 99%.
  • Checkpoint duration + frequency.
  • Lock waits (pg_locks, pg_stat_activity.wait_event).
  • Autovacuum activity per table; flag tables not vacuumed in > 1 day.
  • WAL write rate; archive backlog.
  • Long-running transactions (pg_stat_activity with state = 'idle in transaction').

21. Anti-patterns & scaling cliffs

  • Long-running transactions blocking VACUUM → XID wraparound.
  • Too many connections without pooling.
  • SELECT * on wide / TOAST'd rows.
  • Missing FK indexes → cascading-delete storms.
  • Mass WHERE col IN (... 50k values ...).
  • Unbounded ORDER BY without index.
  • Heavy ad-hoc analytics on production OLTP.
  • pg_upgrade --copy on huge DB (slow; use --link or logical repl).
  • One Postgres instance for OLTP and OLAP; vacuum / lock contention.
  • Logical decoding without monitoring slot lag → disk full.

22. Cost model

  • Self-host: instance + IOPS-provisioned SSD; backup storage; cross-AZ network for replicas.
  • RDS / Cloud SQL: instance hours + storage + backup + cross-AZ.
  • Aurora: bills storage usage + I/O ops + compute hours; no need to provision IOPS but I/O charge can dominate hot workloads.
  • Surprises: backup storage (continuous snapshots add up); read-replica vs. multi-AZ semantics differ; logical replication slot bloat = silent disk cost.
