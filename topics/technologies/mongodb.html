MongoDB

Document database storing BSON (binary JSON) in B-tree indexes via WiredTiger storage engine. Replica sets for HA; sharded clusters for horizontal scale; oplog for replication and CDC. Schemaless from day one with optional schema validators since 4.0.

document • replica set • sharded cluster • oplog tailing

2. Underlying mechanism

  • BSON document: the row, with rich types (ObjectId, dates, arrays, nested objects). Up to 16 MB per doc.
  • WiredTiger storage engine: B-tree index over collections; copy-on-write blocks; LRU cache of B-tree pages; per-document MVCC via timestamps.
  • Oplog: capped collection on primary recording every write; replicas tail it. Same log used for change streams (CDC), Atlas Stream Processing, etc.
  • Replica set: 3+ nodes; primary takes writes; others are secondaries running automatic failover via Raft-like election.
  • Sharded cluster: mongos router + config servers (replica set storing metadata) + N shards (each itself a replica set). Shard key chosen at table creation.

3. Read & write path

Write (insert / update)

1. Client → mongos (or directly primary in single-rs)
2. mongos hashes shard key → shard
3. Primary of shard:
   - Take collection lock (intent-shared)
   - Write to journal (WAL)
   - Update WiredTiger B-tree page (CoW)
   - Append to oplog
4. With w:majority, wait for majority of replica acks
5. Indexes updated atomically with the doc

Read (find)

1. Client → mongos
2. mongos plans:
   - Targeted: shard key in filter → one shard
   - Scatter-gather: no shard key → all shards
3. Each shard runs query on its data
4. mongos merges + returns

Read concerns: local / available / majority / linearizable

Replication

Each replica tails primary's oplog
Applies operations idempotently
Election when primary unreachable (~10s)
Pre-image / post-image for change streams

Sharding rebalance

Balancer moves chunks between shards (background)
Goal: even chunk count per shard
Online but expensive
4.4+: live re-sharding (change shard key)

4. Replication & consistency

  • Replica set: 1 primary + N secondaries; auto-failover via Raft-like protocol; arbiters allowed but discouraged.
  • Write concern: w:1 (default), w:majority (durable), w:N, j:true (force journal sync).
  • Read concern: local (latest), majority (durable), linearizable (read after majority, strongest), snapshot (txn-bound).
  • Read preference: primary (default), secondary, nearest, secondaryPreferred.
  • Multi-doc transactions (4.0+): single replica set; 4.2+ across shards via 2PC; perf hit at high QPS.

5. Partitioning (sharding)

  • Shard key chosen at collection creation; hash or range.
  • Hash sharding: even distribution; loses ordered range queries.
  • Range sharding: ordered queries fast; hot range possible.
  • Compound shard keys help cardinality.
  • Pre-split with hash sharding for write-heavy from-scratch tables.
  • Zone sharding: pin chunk ranges to specific shards (e.g., geo zones).
  • 4.4+ live re-sharding: change shard key online.

6. Time complexity

  • Indexed find: O(log N).
  • Insert: O(log N) per affected index.
  • Update: O(log N); single-doc atomic.
  • Aggregation: depends on stages; pipeline optimizer pushes filters down.
  • Cross-shard scatter-gather: O(shards × per-shard work).

7. Per-node capacity envelope

  • Throughput: ~5–20k ops/sec/node tuned.
  • P99 latency: 1–5 ms with majority writes.
  • Doc size: 16 MB cap.
  • Per-shard storage: ~1 TB practical before balancer churn.
  • Connections: thousands; mongos handles fan-out.

8. When to use / not to

Use Mongo when…

  • Schema is genuinely flexible; documents vary per row.
  • Need fast development iteration without schema migrations.
  • Embedded doc structure matches your query patterns.
  • Atlas managed cluster fits the budget.
  • Need full-text + geo + change streams in one product.

Don't use Mongo when…

  • Data is naturally relational with joins. Use Postgres.
  • Heavy multi-document transactions at high QPS.
  • Mature relational tooling expected.
  • Strict consistency without majority overhead.
  • JSON-light needs — Postgres JSONB is often enough.

9. Common gotchas

  • Bad shard-key choice = re-sharding pain (eased in 4.4+, still expensive).
  • Unbounded array fields explode doc size.
  • Case-insensitive queries without proper index = collection scan.
  • Cross-shard transactions add coordination overhead; keep transactions to one shard if possible.
  • Stale reads from secondary with default readPreference: primary.
  • Index over wide doc paths can balloon.
  • Aggregation pipeline order matters for performance.
  • Connection pool exhaustion on mongos.
  • Oplog window too small → replica resync from full snapshot.
  • Time-series collections (5.0+) have specific best practices.

10. Cloud equivalents

  • MongoDB Atlas (multi-cloud).
  • AWS DocumentDB (Mongo-API-compat, partial; not the same engine).
  • Azure Cosmos DB Mongo API (compat).
  • Self-host or community edition (SSPL license — restrictions for SaaS providers).

12. Usage patterns

  • Document store for entities with flexible schema.
  • Embed-related data (user with embedded addresses) for one-shot queries.
  • Catalog / inventory.
  • Time-series collections (5.0+).
  • Change streams for real-time event triggers.
  • Geospatial (2dsphere index) for location apps.
  • Atlas Search for full-text + faceted search in-product.
  • Vector search via Atlas Search.
  • NOT primary store for relational data with joins.

13. Operational knobs

  • WiredTiger cache size (default 50% of RAM minus 1 GB).
  • Write concern + read concern defaults.
  • Oplog size (24 hr typical replication window).
  • Compression: snappy (default) / zlib / zstd.
  • Index types: single, compound, multikey, text, 2dsphere, hashed, partial, sparse, TTL, wildcard.
  • Profiler levels for slow query capture.
  • Sharding chunk size (default 128 MB).

14. Capacity planning

  • RAM ≈ working set; index hot paths must fit.
  • Storage: 30% headroom; compression cuts ~3× for typical JSON.
  • Connections: pool per app; 1k+ feasible with mongos.
  • Sharding: pre-split for write-heavy or expect balancer pain.

15. Failure modes

  • Primary loss: replica election ≈ 10s; lose un-replicated writes (w:1).
  • Network partition: minority side rejects writes.
  • Disk full: writes blocked; clear oplog or extend.
  • Balancer storm: chunk moves block writes; throttle balancer window.
  • Long-running query: holds collection lock briefly; then snapshot.
  • Oplog roll-over: replica can't catch up → full resync.

16. Backup & DR

  • Atlas Cloud Backup: continuous + snapshots.
  • mongodump / mongorestore: logical, slow.
  • File-system snapshot of data path (with fsync lock).
  • Ops Manager (Enterprise): full PITR.
  • RPO: oplog tail.
  • RTO: restore + replay oplog.
  • Multi-region replica set = built-in DR.

17. Upgrade & migration

  • Rolling upgrade: secondary first, step-down primary, upgrade last.
  • Bump featureCompatibilityVersion after.
  • Migrate from another DB: typically dual-write + backfill.
  • Atlas Live Migration tool.
  • Sharding migration (re-sharding): 4.4+ live; expensive but no-downtime.

18. Security

  • SCRAM auth; x509 cert; LDAP / Kerberos (Enterprise); IAM (Atlas).
  • RBAC roles; database / collection / action level.
  • TLS for client + cluster.
  • Encryption at rest (Enterprise / Atlas).
  • Field-level encryption (CSFLE) — transparent client-side encryption.
  • Network: VPC peering / private endpoint in Atlas.

19. Multi-tenancy

  • Database per tenant (clean isolation, doesn't scale to thousands).
  • Collection per tenant.
  • Shared collection + tenant_id field + indexed (most common).
  • No native rate limiting per tenant.

20. Observability

  • Operations per second (insert/query/update/delete).
  • Replication lag (rs.printSlaveReplicationInfo() / rs.status()).
  • WiredTiger cache eviction.
  • Page-fault rate.
  • Slow query log (profiler).
  • Connection count.
  • Index hit rate.
  • Balancer activity (sharded).
  • Oplog window remaining.

21. Anti-patterns

  • Unbounded arrays inside documents.
  • Heavy joins via $lookup at high QPS.
  • Transactions across many docs at high QPS (use single-doc atomic ops).
  • Bad shard key (low cardinality, monotonic).
  • Ad-hoc queries on un-indexed fields.
  • Reading from primary by default when secondary would do.
  • Storing large blobs — use S3.
  • Skipping schema validators → data drift.

22. Cost model

  • Atlas: cluster tier (RAM-based).
  • Backup storage extra.
  • Data transfer (egress / cross-region).
  • Sharded clusters multiply.
  • Search nodes / vector search nodes separate.
