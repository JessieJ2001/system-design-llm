Storage: Comparison & Tradeoffs

"Use S3" is wrong half the time. Object, block, and file storage are not interchangeable. Block is for one VM's filesystem; file is for shared filesystem semantics; object is for HTTP-addressable blobs at infinite scale. Pick the wrong one and you're either paying 10× or fighting consistency.

6. Migration paths

  • HDFS → S3: the most common modern migration. Rewrite jobs to use s3a:// URIs; many tools (Hadoop, Spark, Hive) support both. Use distcp; cut over read jobs first, then writers.
  • NFS → S3: only works if app tolerates rewrite. s3fs as a stopgap (POSIX layer over S3) but it's slow and consistency-bumpy. Real fix: change app to use S3 SDK.
  • EBS → EFS: needed when scaling out compute that needs shared state. rsync initial; gradually mount EFS; cut over.
  • S3 → another cloud (GCS / Azure Blob): Storage Transfer Service / AWS DataSync / rclone. Egress dominates the cost — expect tens of cents per GB.
  • Self-host MinIO → S3: trivial — same API. mc mirror.
  • S3 region migration: S3 Cross-Region Replication or Storage Lens, then DNS / config cutover. Existing objects need bulk copy.
  • Lift-and-shift VM disks: snapshot → copy region → restore. Multi-TB disks take hours.

8. Hybrid patterns ("use them together")

S3 + CloudFront (the canonical asset stack)

Bucket as origin; CDN as cache. Sign URLs for private content; lifecycle to IA / Glacier for old blobs. Cost: most static asset traffic served from CDN edge.

EBS + S3 (snapshot DR)

EBS volume for live DB; periodic EBS snapshots replicate to S3-backed snapshot store. Cross-region snapshot copy for disaster recovery. RPO = snapshot frequency.

S3 + DynamoDB (large items)

DDB has 400 KB cap. Store the blob in S3, the metadata + S3 key in DDB. Atomic on the item; blob itself is eventual.

S3 + Athena / BigQuery (data lake)

Parquet / Iceberg / Delta on S3; Athena, Trino, BigQuery External Tables, or Snowflake External Tables query in place. Compute and storage decoupled; pay only for queries.

EFS for shared scratch + S3 for durable

ML training: EFS for shared dataset cache during a training run; S3 for the canonical dataset and checkpoints. EFS is the working set, S3 is the source of truth.

Object Lock + Versioning for ransomware protection

S3 Object Lock makes objects immutable for a retention period; even an attacker with credentials can't delete them. Versioning + MFA Delete add a second layer.

9. Anti-patterns

"We used S3 as a filesystem"

Via s3fs / FUSE: works for casual read; breaks under multi-writer, fails on metadata-heavy workloads, has weird consistency edges. Either rewrite the app to S3 SDK or use EFS.

"All keys had a sequential prefix"

e.g., logs/2026-05-01/... — one prefix takes all writes, hits 3,500 PUT/sec ceiling. Add randomized hex prefix to spread.

"We LISTed the bucket in real-time"

1,000 keys per page, paginated, slow. For a 100M-key bucket, full LIST takes hours. Use S3 Inventory (daily CSV) for batch needs.

"Backup is in the same bucket"

Same credentials, same region. One ransomware event takes both. Replicate to a different region/account; use Object Lock.

"We forgot lifecycle policy"

Multipart upload parts left in the bucket: pay for them forever. Logs accumulate at Standard tier instead of moving to Glacier. Lifecycle is the #1 cost lever.

"We treated EBS like multi-AZ"

EBS is single-AZ. AZ failure = volume offline (data not lost, but unavailable). Multi-AZ is via snapshots or app-level replication.

"We picked EFS at scale because the app uses NFS"

EFS is 13× the per-GB cost of S3. Worth migrating to S3 SDK at > 10 TB.

"We ran HDFS in cloud"

HDFS in EC2 is anti-pattern: pay for 3× replicated EBS/local SSD when S3 already has 11 9s. Use S3 + EMR/Glue/Spark instead.

"VARY: Cookie at the CDN in front of S3"

Cache fragments by cookie; hit rate ≈ 0; origin (S3) gets crushed. Strip cookies before CDN; sign URLs for personalization.

10. Decision rules for interviews

  • Default for blobs / uploads / data lake / static assets → object (S3 / GCS).
  • Default for VM disks / DB storage → block (EBS / PD).
  • File storage (EFS) only when the app explicitly needs POSIX shared filesystem.
  • HDFS only on-prem. In cloud, use S3 + EMR/Dataproc.
  • Always say "with CloudFront in front" for any user-facing object storage.
  • Always set lifecycle policy. "Old objects to IA / Glacier; abort multipart after 7d." This is the #1 cost lever.
  • S3 large items pattern: blob in S3, pointer in DDB / Postgres.
  • S3 throttling = hot prefix. Solution: random hex prefix.
  • DR for object storage = cross-region replication or dual-region bucket.
  • "Backup must survive ransomware" = Object Lock or replicate to different account.
