Apache Kafka

A distributed, partitioned, replicated append-only log. Each topic-partition is an ordered, immutable sequence of records on disk; consumers read by offset. Page cache + zero-copy give it > 600 MB/s/broker. ISR replication guarantees durability; KRaft (5+) replaces ZooKeeper with Raft-based controller quorum.

log • partitioned • pull-based • KRaft (no ZK)

2. Underlying mechanism

Three primitives: topic-partition is an append-only log file, broker is page-cache + sendfile(), and consumer is a cursor (offset). Everything else is bookkeeping around those three.

Why it's fast

Sequential disk write to a single file is > 100× faster than random. Producer batches records; broker appends to active segment in OS page cache; sendfile() serves consumers directly from page cache to NIC, no user-space copy. The only "slow" path is fsync on commit, which Kafka avoids by default.

Durability story

"Durable" means: leader has written + replicas have fetched the record. Not "fsync'd to disk." In a power-loss scenario, an unflushed page-cache write on all replicas is lost. RF=3 across 3 racks/AZs makes simultaneous loss unlikely; flush.messages / flush.ms can force fsync but tank throughput.

3. Read & write path

Produce

1. Producer batches records by partition (linger.ms, batch.size)
2. Send Produce RPC to partition leader
3. Leader appends to active segment in page cache
4. Followers Fetch the new records, append, ack
5. acks=0: don't wait
   acks=1: wait for leader
   acks=all: wait for ISR (durable)
6. Producer ack — offset returned

Fetch

1. Consumer sends Fetch from offset X
2. Broker locates segment, range
3. sendfile() — disk page cache → socket
4. Consumer commits offset (auto or manual)
5. New record at high-water mark only after replicated

Compaction (compacted topics)

For topics with cleanup.policy=compact:
- LogCleaner thread merges segments
- For each key, keep only latest value
- Tombstones (null value) eventually removed
- Used for changelog topics, KTables, etc.

Rebalance

When consumer joins/leaves group:
1. Coordinator detects via heartbeat timeout
2. Stop-the-world (eager) or incremental (cooperative-sticky)
3. Partitions re-assigned
4. Consumers commit then resume

Rebalance time grows with partition count.

4. Replication & consistency

  • Topology: per-partition leader + N-1 followers; replicas distributed across brokers (and ideally racks/AZs).
  • ISR: followers within replica.lag.time.max.ms are "in-sync." Producer with acks=all waits for full ISR.
  • Min ISR: set min.insync.replicas=2 with RF=3 — producer fails if fewer than 2 replicas in sync (better than silently dropping to one).
  • Failover: if leader dies, controller (KRaft) elects a new leader from ISR. unclean.leader.election.enable=false (default modern) prevents promoting an out-of-sync follower at the cost of availability.
  • Consistency: per-partition total order; cross-partition no guarantees. Idempotent producer + transactions give exactly-once within Kafka (read-process-write).

5. Partitioning

  • Key → partition: default murmur2(key) mod partition_count. Same key always goes to same partition (=> ordered).
  • Custom partitioner: set per producer (geo-aware, sticky, etc.).
  • Increasing partition count is destructive for ordering: hash mapping changes; same key may move. Plan partition count up-front.
  • Hot partition = one key with disproportionate traffic; that partition saturates one broker. Composite key (user_id + hour-bucket) is the canonical fix.
  • Cluster-wide cap: ~4k partitions per broker for stable controller behavior (KRaft scales higher than ZK). Beyond that, use Pulsar or split into multiple clusters.

6. Time complexity

  • Append: O(1) (sequential write).
  • Fetch by offset: O(log segment_count) for index lookup, then sequential read.
  • Compaction: O(N log N) per cleaning cycle.
  • Rebalance: O(partition_count × consumer_count).

7. Per-node capacity envelope

  • Throughput: ~600 MB/s sequential write per broker on 10 GbE + NVMe.
  • Replication: same for inbound; cross-AZ network is the practical cap.
  • Latency: typical end-to-end 5–20 ms with acks=all.
  • Disk: a single broker can hold tens of TB; tiered storage moves cold to S3.
  • Partitions/broker: ~4k stable; many more with KRaft + tuning.

8. When to use / not to

Use Kafka when…

  • Distributed event log / event sourcing.
  • Many consumers / replay needed.
  • Buffer between fast producer and slow consumer.
  • CDC pipeline (Debezium).
  • Stream processing input (Flink, Spark Streaming, Kafka Streams).

Don't use Kafka when…

  • Sub-ms RPC; use NATS or gRPC.
  • Random KV reads — not what it does.
  • Millions of low-throughput topics — Pulsar or NATS.
  • Complex routing (topic exchanges) — RabbitMQ.
  • Truly small / tiny scale — Redis Streams or SQS overkill-avoidance.

9. Common gotchas

  • Hot partition kills throughput regardless of cluster size.
  • Partition count is hard to change; over-provision early.
  • Consumer rebalance time grows linearly with partitions.
  • Default acks=1 can lose data on leader crash; use acks=all + min.insync.replicas=2.
  • unclean.leader.election.enable=true (legacy default) trades durability for availability.
  • Schema evolution without a Schema Registry = silent data corruption.
  • Consumer commit offset before processing = "at-most-once" by accident.
  • EOS (exactly-once) is "exactly-once within Kafka," not external side effects.
  • Cross-AZ replication network is often the single biggest line item.
  • Mirror Maker 2 has gotchas with offsets / idempotency.

10. Cloud equivalents

12. Usage patterns

  • Event log / event sourcing.
  • Outbox pattern (DB write + outbox row + Debezium → Kafka).
  • CDC sink (Debezium).
  • Stream-table join with Kafka Streams (KStream × KTable).
  • Pub/sub for many downstream services.
  • Buffer between producer and slow consumer.
  • Replayable analytics / training data.
  • Reprocessing: reset consumer group offset, re-run pipeline.
  • Tiered "infinite" log via tiered storage.

13. Operational knobs

  • acks — all for durable.
  • min.insync.replicas — 2 with RF=3.
  • replica.lag.time.max.ms — how long a follower can lag before being booted from ISR.
  • unclean.leader.election.enable — false for durability.
  • retention.ms / retention.bytes — per topic.
  • compression.type — lz4 or zstd; massive throughput win.
  • linger.ms + batch.size on producer — latency vs throughput.
  • max.poll.records + session.timeout.ms on consumer.
  • num.partitions default for new topics.
  • log.cleaner.* for compacted topics.

14. Capacity planning

  • Disk: retention_hours × peak_MB_per_sec × 3600 × replication_factor.
  • Partitions: at least peak_consumer_throughput / per-partition-throughput.
  • Brokers: at least 3; scale by total throughput / 600 MB/s + headroom.
  • Network: cross-AZ replication doubles or triples network bytes.
  • Consumer groups: more groups = more disk read, but page cache absorbs replay.

15. Failure modes

  • Single broker loss: leader for its partitions re-elected; ISR shrinks; min ISR may pause writes.
  • AZ loss with rack-aware: ISR stays >=2 with RF=3; cluster continues.
  • Cluster-wide controller loss (KRaft): no leader elections possible until quorum restored.
  • Disk full on broker: shrinks ISR; client errors; cluster degraded.
  • Slow consumer: lag grows; disk fills with retention; eventually messages dropped.
  • ZK quorum loss (legacy): no leader elections; stuck.

16. Backup & DR

  • MirrorMaker 2 / Cluster Linking: continuous mirror to DR region.
  • Tiered storage: cold log already in S3 = inherent backup.
  • Topic offsets need to be mirrored separately (or re-derive from data).
  • Restore: cluster from backup snapshot of broker disks + offsets.
  • RPO: replication lag.
  • RTO: cluster spin-up + offset translation + consumer reset.

17. Upgrade & migration

  • Rolling broker upgrade; bump inter.broker.protocol.version after all nodes upgraded.
  • Bump log.message.format.version only after all consumers upgraded.
  • ZK → KRaft migration: multi-stage state machine; documented carefully.
  • Repartitioning a topic (changing partition count): destroys ordering; do via dual-write to a new topic + cutover.
  • Migrate to Pulsar: Pulsar's KoP (Kafka-on-Pulsar) protocol shim or dual-publish.

18. Security

  • SASL: PLAIN / SCRAM / Kerberos / OAUTHBEARER.
  • mTLS at broker port.
  • ACLs per topic / consumer group / cluster operation.
  • Encryption at rest: rely on disk encryption.
  • Network: brokers behind firewall; client connections via TLS.
  • Audit: limited native; commercial extensions.

19. Multi-tenancy

  • One cluster per tenant tier (free / paid / enterprise).
  • Within a cluster: ACLs + quotas (producer_byte_rate, consumer_byte_rate, request_percentage).
  • Pulsar's namespace model is more native multi-tenant than Kafka.

20. Observability

  • Under-replicated partitions (URP) — should be 0.
  • ISR shrink rate.
  • Request handler idle %.
  • Consumer lag per group / topic / partition.
  • Disk %.
  • Controller failover count.
  • Network IO threads idle %.
  • Bytes-in / bytes-out per broker.

21. Anti-patterns

  • One topic per user / per tenant (partition explosion).
  • Tiny messages without batching (kills throughput).
  • Random key (no ordering, but also no skew).
  • Storing huge payloads (multi-MB) in records.
  • Trusting EOS for external side effects.
  • Disabling replication for "speed."
  • Manually managing offsets in app DB without idempotency.
  • No schema registry for non-trivial schemas.

22. Cost model

  • Self-host: brokers + disk + cross-AZ network.
  • Cross-AZ: typically the single biggest line item with RF=3.
  • MSK / Confluent / Managed Kafka: per-broker-hour + ingress GB.
  • Tiered storage: cheaper at scale; complexity tradeoff.
