Apache Cassandra

Leaderless, masterless, eventually-consistent wide-column store with LSM-tree storage and consistent-hash partitioning. Optimized for write-heavy, partition-tolerant, multi-DC workloads. Bigtable + Dynamo — you get the wide-column data model from Bigtable and the leaderless replication from Dynamo.

3. Read & write path

Write

1. Coordinator hashes partition key → token
2. Token → replica set (RF replicas, walk ring clockwise)
3. Coordinator sends to all replicas in parallel
4. Each replica:
   a. Append to commitlog (durable)
   b. Insert into memtable (sorted in-mem)
   c. Ack
5. When <CL> replicas have acked, return success
6. Hinted handoff if a replica is down
7. Memtable eventually flushed to SSTable
8. Compaction merges SSTables in background

Read

1. Coordinator hashes → replicas
2. Send read to <CL> replicas:
   - 1 full read + (CL-1) digest reads
3. Each replica:
   a. Check bloom filter per SSTable
   b. Read from memtable + relevant SSTables
   c. Merge by latest timestamp
4. Coordinator compares digests
5. If mismatch → foreground read repair
6. Return result

Compaction

Strategies:
- STCS (size-tiered): default; merges similar-size SSTables
- LCS (leveled): tighter read amp, more write amp
- TWCS (time-window): time-series friendly
- UCS (unified): newer, balanced

Compaction reclaims tombstones & reduces SSTable count.
Reserve ~50% disk for compaction headroom.

Tombstones

Delete writes a tombstone (marker)
Compaction skips tombstoned data once gc_grace_seconds passed
Long gc_grace = compaction can't reclaim
Tombstone-heavy queries = slow + GC pressure
Avoid TTL columns at high cardinality

4. Replication & consistency

  • Topology: leaderless. Every node is a peer; clients can talk to any node (which becomes coordinator for that request).
  • Replication strategies: SimpleStrategy (single DC) or NetworkTopologyStrategy (multi-DC, replicas spread across racks/AZs).
  • Tunable consistency: R+W>N gives strong consistency. LOCAL_QUORUM for low-latency multi-DC; EACH_QUORUM for global writes (slow).
  • Anti-entropy: hinted handoff (writes), read repair, periodic nodetool repair. Without repair, eventual consistency may diverge over time.
  • Failure model: losing 1 replica with RF=3 + QUORUM = still serves. Losing >= ceil(RF/2)+1 = unavailable for that partition.

5. Partitioning

  • Consistent hash on partition key → token; vnodes (virtual tokens) per node so each node owns many ranges.
  • Adding a node: it claims tokens; data streams from existing replicas.
  • Hot partition: one partition pegs 3 replicas regardless of cluster size. Fix: composite key (e.g., (user_id, hour_bucket)).
  • Partitions should be bounded: < 100 MB / 100k cells per partition.

6. Time complexity

  • Write: O(1) (append to memtable + commitlog).
  • Read: O(log N) per SSTable searched; bloom filter skips most.
  • Range scan within partition: O(log N + K).
  • Range scan across partitions: token-range queries; expensive.
  • Compaction: O(N log K) where K is SSTables.

7. Per-node capacity envelope

  • Writes: ~30k/sec/node typical, > 100k/sec on tuned hardware (LSM strong suit).
  • Reads: lower than writes; depends on SSTable count + bloom fp rate. ~10–50k QPS.
  • Latency P99: 5–15 ms with QUORUM.
  • Disk: tens of TB / node; reserve 50% for compaction.
  • Scales linearly: 2× nodes ≈ 2× throughput.

8. When to use / not to

Use Cassandra when…

  • Write-heavy with predictable access pattern by partition key.
  • Multi-DC active-active with low latency.
  • Time-series, activity feeds, sensor data.
  • Need linear horizontal scale.
  • Tolerate eventual consistency.

Don't use Cassandra when…

  • You don't know access patterns — can't model later.
  • Need joins / transactions / ad-hoc queries.
  • Strong consistency is essential and you can't tolerate quorum costs.
  • Small dataset (single Postgres beats it).
  • You don't have the ops team for compaction + repair tuning.

9. Common gotchas

  • Tombstones during scans: TTL on millions of rows + scan = "tombstone overflow."
  • Repair storms: full repair locks compaction; use incremental repair (4.x).
  • Large partitions: > 100 MB / 100k cells = compaction stalls + GC pause + coordinator timeouts.
  • Secondary indexes on high-cardinality columns: scan all replicas; don't.
  • ALLOW FILTERING: scans entire partitions; performance disaster.
  • BATCH across partitions: not transactional; coordinator overhead.
  • IN with many partition keys: each is a separate read; just loop client-side.
  • Disk full from un-compacted data; always have headroom.
  • Old gossip state: dead nodes lingering; nodetool removenode.
  • JVM GC tuning matters; G1GC default is fine but tune heap size.

10. Cloud equivalents

  • AWS: Keyspaces (Cassandra-compat managed); self-host on EC2; Elasticsearch + DAX combos.
  • GCP: (no native managed Cassandra); use Bigtable for similar shape.
  • Azure: Cosmos DB Cassandra API.
  • Open-source: Cassandra itself; ScyllaDB (C++ rewrite, drop-in compat); Astra DB (DataStax managed).

12. Usage patterns

  • Time-series (sensor / metrics) — partition by (sensor_id, day), cluster by timestamp.
  • Activity feed — partition by user_id; cluster by event_time desc.
  • Write-heavy log of events.
  • Multi-DC active-active with LOCAL_QUORUM.
  • Materialized views (CDC sink for read-side projection).
  • NOT for ad-hoc queries (must model around access patterns up front).
  • NOT for transactions across partitions.

13. Operational knobs

  • concurrent_writes, concurrent_reads — tune per cores.
  • compaction_throughput_mb_per_sec — cap compaction I/O.
  • memtable_flush_writers — parallel flush threads.
  • gc_grace_seconds per table — tombstone retention.
  • compaction strategy per table.
  • read_repair_chance — reduce in 4.x.
  • speculative_retry — avoid waiting for slow replicas.
  • JVM heap size + GC choice.

14. Capacity planning

  • Nodes = peak_writes / 30k.
  • Storage = data × RF.
  • Compaction headroom: 50% disk.
  • Partition size: ≤ 100 MB; key choice critical.
  • RAM: ~25% for memtable + bloom filters; ~50% for OS page cache.

15. Failure modes

  • Single node loss with RF=3, QUORUM: serves; hinted handoff for misses.
  • Lose 2 of 3: QUORUM unavailable for that partition.
  • Slow node: speculative_retry routes around; otherwise tail latency.
  • Partition larger than memtable: write amplification; flush stalls.
  • Compaction backlog: SSTable count grows; reads slow due to bloom fp rate; eventually disk full.
  • Repair storm: heavy I/O blocking flushes; cluster latency spikes.

16. Backup & DR

  • Snapshots: hardlinks of SSTables; fast.
  • Incremental backup: ship new SSTables to S3.
  • Restore: drop SSTables in place + nodetool refresh.
  • RPO: snapshot interval.
  • RTO: SSTable copy speed.
  • Tools: Medusa, manual + S3.
  • Multi-DC = built-in DR (just lose one DC and use the other).

17. Upgrade & migration

  • Rolling upgrade one node at a time.
  • Mixed major versions OK briefly; not long-term.
  • SSTable format upgrade: nodetool upgradesstables.
  • Cassandra → ScyllaDB: similar shape; migrate via dual-write or sstableloader.
  • Cassandra → DynamoDB: different model; usually app-side rewrite + dual-write.

18. Security

  • Auth: PasswordAuthenticator; LDAP via plugins; cert-based.
  • RBAC: GRANT on keyspace / table.
  • TLS: client + node-to-node + DC-to-DC.
  • Encryption at rest: optional (commercial DataStax) or rely on disk encryption.
  • Audit: AuditLogger (4.x).

19. Multi-tenancy

  • Keyspaces + roles for team isolation.
  • No native rate limiting per tenant.
  • One cluster per tenant tier in practice for noisy neighbor protection.

20. Observability

  • Pending compactions per table.
  • Dropped mutations / hints.
  • Read / write latency p99 per table.
  • Hinted-handoff queue size.
  • Blocked native-transport tasks.
  • JVM GC pause.
  • SSTable count per CF.
  • Repair status.

21. Anti-patterns

  • Secondary indexes on high-cardinality columns.
  • ALLOW FILTERING.
  • Large partitions (> 100 MB).
  • IN across many partitions.
  • BATCH across partitions.
  • TTL on millions of rows with same expiry → tombstone storm.
  • Disabling repair to "save I/O."
  • Same compaction strategy for all tables (TWCS for time-series; LCS for read-heavy; STCS for write-heavy).

22. Cost model

  • Self-host: nodes (RAM + SSD) × RF; cross-AZ replication network.
  • Compaction headroom: 2× storage.
  • Keyspaces: per-WCU/RCU.
  • Cosmos Cassandra API: per-RU.
  • Astra DB: managed; per-throughput.
