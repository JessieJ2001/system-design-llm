Stream Processing: Comparison & Tradeoffs

"Use Flink" is fine 30% of the time. Stream processing has three shapes — true streaming (Flink), micro-batch (Spark), and library-not-framework (Kafka Streams). Pick by latency budget, by team's existing stack, and by stateful-vs-stateless.

6. Migration paths

  • Spark Streaming → Flink: typical when sub-second freshness becomes a requirement. Flink SQL or DataStream API; rewrite stateful logic; reuse Kafka source.
  • Kafka Streams → Flink: when state outgrows per-partition local; when you need cross-stream joins over hours of data.
  • Custom in-process → framework: when checkpoints / replay / exactly-once become required. Default to Flink or Streams.
  • Hand-rolled Lambda triggers → Flink: common when stateless transforms grow into windows/joins.
  • Self-host Flink → managed (MSF / Dataflow / Confluent Flink): usually a good trade once team SRE cost exceeds managed price.

8. Hybrid patterns

Flink → ClickHouse / Druid

Flink does the heavy stateful work (windows, joins); writes pre-aggregated rows to a real-time OLAP. UI queries OLAP for sub-second dashboards.

Kafka Streams as service-internal

Each microservice runs a small Streams topology for its own state; no separate cluster. KTables expose state via interactive queries.

Beam pipeline portable across runners

Write once in Beam SDK; run on Dataflow in production, on DirectRunner in tests, on Flink for cost. Hedges runner choice.

Spark for batch + Flink for streaming (lambda)

Spark batch reprocesses historical for correctness; Flink handles real-time. UI merges results at the time boundary.

CDC → Flink → OLAP / OLTP

Debezium → Kafka → Flink (transform / join / aggregate) → sink into warehouse and back into KV for read-side projections (CQRS).

Per-record transform with serverless

Skip the framework: Lambda triggered by Kinesis/SQS for stateless transforms. Move to Flink only when state appears.

9. Anti-patterns

"We picked Flink for stateless transforms"

Overkill. Use Lambda or Kafka Streams stateless map/filter; less ops cost.

"Kafka Streams for cross-region"

Streams expects local Kafka; cross-region adds latency and replication-loop risk. Use Flink with Kafka mirror.

"Spark micro-batch for sub-second"

Spark continuous mode is experimental; default 1s+ batch interval. For real real-time, use Flink.

"State grew unbounded"

Forgot TTL on state. RocksDB grows forever; checkpoints become huge. Always set state TTL where business semantics allow.

"Hot key broke parallelism"

Watermark stuck on one slow operator. Salt the key; aggregate twice (per-salt then merged); consider RebalancedStream.

"Exactly-once means exactly-once everywhere"

Flink EOS only inside Flink + supported sinks. External REST calls / payments still need idempotency keys.

10. Decision rules for interviews

  • Sub-second + stateful: Flink.
  • Already on Spark + minutes-fresh OK: Spark Structured Streaming.
  • Already on Kafka + simple state: Kafka Streams.
  • Stateless per-record transform: Lambda / Cloud Functions.
  • Want runner portability: Beam on Dataflow / Flink.
  • Always set state TTL.
  • Always partition early (over-provision). You can't easily increase partition count later.
  • Exactly-once is "exactly-once inside the framework" — external side effects still need idempotency.
  • Hot keys break stream processing more than anything else. Salt + aggregate-twice is the standard fix.
  • Pre-aggregate stream into OLAP for dashboards; don't query stream state directly.
