Amazon DynamoDB

Fully-managed, serverless KV / wide-column store. Consistent-hash partitioning by partition key; per-partition leader with multi-AZ replicated storage; log-structured. Trades query expressiveness for predictable single-digit-ms latency at any scale.

managed • PK + SK • GSI • auto-scale

2. Underlying mechanism

From the published Dynamo / DynamoDB papers: a request routes by partition key → partition (hash); each partition has its own leader and 2 replica copies across AZs; storage uses a log-structured engine for write amplification; reads either eventually-consistent (any replica) or strongly consistent (leader).

The decisive constraint: one partition is capped at 1,000 WCU / 3,000 RCU. This shapes everything you do with DDB. Your data model is "design partition keys to distribute load."

  • Item: a row. Identified by partition key (PK), optionally + sort key (SK). Up to 400 KB.
  • Partition key (PK / hash key): determines partition.
  • Sort key (SK / range key): within a partition, items sorted by SK; range queries supported.
  • Secondary indexes: GSI (Global) or LSI (Local). GSI is a separate partitioned table maintained async.
  • Streams: change log per table; consumed by Lambda / Kinesis.

3. Read & write path

PutItem / UpdateItem

1. Hash PK → partition
2. Route to partition leader (request router)
3. Leader writes to its log
4. Replicates to 2 followers (other AZs)
5. Quorum of 2 ack → commit
6. Stream record emitted (if enabled)
7. GSIs updated async

GetItem / Query

GetItem (PK + SK):
  Hash → partition → replica (eventual) or leader (strong)
  Read; return item

Query (PK = X):
  All items with same PK, filtered by SK range
  All from one partition

Scan:
  Sequential scan of ALL partitions
  Eats RCU; expensive; avoid

Strong vs eventual

Default reads: eventually consistent (1/2 RCU)
Strongly consistent: 1 RCU per 4 KB
Always-strong on PutItem (returns after quorum)
Cross-region (Global Tables): eventual + LWW

Streams

Each item change → record in stream (24 hr retention)
Consumed by Lambda triggers, Kinesis adapter
Used for CDC, cache invalidation, analytics fan-out

4. Replication & consistency

  • Per-partition leader; 2 replicas across AZs; quorum write. Single-region: strong-consistency option per read.
  • Global Tables: per-region writable; replicated last-write-wins by timestamp; cross-region eventual.
  • Failure: leader loss → one of replicas promoted automatically (single-digit seconds).
  • No notion of "transaction across all partitions": TransactWriteItems supports up to 100 items, single region only, 2× cost.

5. Partitioning

  • Partitioning fully managed; you don't see partitions directly. But you must design PK to spread load.
  • Hot PK = throttling. Solutions: write-shard (suffix PK with random bucket and fan-in on read), or pre-distribute via composite key.
  • Auto-scaling: provisioned mode adjusts capacity within bounds; on-demand mode handles 2× previous peak instantly without configuration.
  • Resharding: invisible to you; DDB splits/joins partitions as needed.

6. Time complexity

  • GetItem / PutItem: O(1) routing + log-structured read/write.
  • Query (within partition): O(log K + M) where M = items returned.
  • Scan: O(N) full-table.
  • GSI write: O(1) but doubles write cost.

7. Per-node capacity envelope

  • Partition cap: 1,000 WCU / 3,000 RCU.
  • Item size: 400 KB max.
  • Latency: P50 ~5–10ms; P99 ~10–30ms; DAX (in-mem cache) cuts P50 to < 1 ms.
  • Throughput: effectively unlimited via partition fan-out (managed).

8. When to use / not to

Use DDB when…

  • Hot KV with predictable PK access.
  • Need fully managed; small ops budget.
  • AWS-native stack; Lambda triggers via Streams.
  • Idempotency table.
  • Session store; user-prefs; metadata.
  • Single-table design with PK+SK + GSIs.

Don't use DDB when…

  • Ad-hoc queries / analytics — use Athena/Redshift via export.
  • Heavy joins — not its model.
  • Items > 400 KB — use S3 + DDB pointer.
  • Multi-region strong consistency.
  • You don't yet know access patterns.

9. Common gotchas

  • Hot partition — uneven PK distribution → throttling.
  • Item-size 400 KB cap — large data must go to S3.
  • Scan cost — 1 RCU per 4 KB read; full table scan = $$.
  • GSI eventual — reading just-written item via GSI may miss.
  • GSI write cost — doubles write capacity.
  • Filter expressions don't reduce RCU — they filter after read.
  • Conditional update with attribute_not_exists for idempotency — must be careful with retries.
  • BatchWriteItem cap: 25 items / 16 MB per call.
  • Backups: PITR window 35 days; on-demand backups separate.
  • Cost: throughput-driven; storage is cheap.

10. Cloud equivalents

  • AWS only (this is the cloud).
  • GCP analog: Bigtable (wide-column) or Firestore (document); shape differs.
  • Azure analog: Cosmos DB Table API.
  • Self-host analogs: Cassandra / ScyllaDB.

12. Usage patterns

  • Hot KV (session, user prefs).
  • Idempotency table (PutItem with attribute_not_exists(idempotency_key)).
  • Single-table design with PK+SK overloading.
  • Streams for CDC into Lambda / S3 / Kinesis.
  • Global tables for multi-region.
  • Saga / distributed transaction state (with TransactWriteItems).
  • Counters via update with ADD; spread by suffix to avoid hot partition.

13. Operational knobs

  • Capacity mode: on-demand vs provisioned.
  • Auto-scaling target utilization.
  • GSI projections: KEYS_ONLY / INCLUDE / ALL.
  • TTL attribute (background expiry; eventual).
  • Streams: NEW_IMAGE / OLD_IMAGE / NEW_AND_OLD / KEYS_ONLY.
  • PITR on/off; export-to-S3 for analytics.
  • DAX cluster size.

14. Capacity planning

  • 1 WCU = 1 KB write/sec; 1 RCU = 4 KB strongly-consistent read/sec (or 2 eventually).
  • Partition cap: 1k WCU / 3k RCU.
  • Throttling = redesign keys to spread.
  • GSI: same WCU as base table for writes; separate RCU for reads.
  • Burst: provisioned has 5-min burst credit; on-demand handles 2× previous peak.

15. Failure modes

  • Leader loss: replica promoted automatically; few seconds.
  • AZ loss: writes continue; replicas re-balance.
  • Region loss: Global Tables continue serving from other regions.
  • Throttling: 4xx ProvisionedThroughputExceeded; client SDK retries.
  • GSI eventual: stale reads on the GSI.

16. Backup & DR

  • PITR (continuous backup): up to 35 days; restore to any second.
  • On-demand backups: snapshots; long retention.
  • Restore creates a new table (cutover required).
  • Export to S3: full data dump for analytics or DR.
  • Global Tables = built-in geo DR.

17. Upgrade & migration

  • No version upgrades (managed).
  • Schema evolution: add attributes anytime; sparse GSIs let you add indexes incrementally.
  • Migration in: AWS DMS or DynamoDB import from S3.
  • Migration out: streams → Lambda → new store; or export to S3.
  • Single-table redesign: hardest migration; usually means dual-write + cutover.

18. Security

  • IAM: per-table, per-action, fine-grained via condition keys (dynamodb:LeadingKeys).
  • Encryption at rest: SSE-S3 / SSE-KMS; on by default.
  • VPC endpoints: keep traffic off the internet.
  • No row-level ACL; use IAM condition on partition key for tenant isolation.

19. Multi-tenancy

  • Tenant ID as PK prefix; IAM dynamodb:LeadingKeys condition for isolation.
  • One table can serve many tenants if PK design spreads them.
  • Hot tenant = hot partition (the canonical DDB problem).

20. Observability

  • Consumed vs provisioned RCU/WCU.
  • Throttled events.
  • Hot partition events (Contributor Insights).
  • GSI throttle.
  • Item collection size (for LSI).
  • Successful request latency p99.
  • System errors (5xx).

21. Anti-patterns

  • Scan-based access patterns.
  • Hot PK (e.g., date as PK).
  • Items close to 400 KB.
  • Filter expression instead of indexed access (still pays read RCU for filtered items).
  • GSIs with low-cardinality PK → hot GSI partition.
  • Treating DDB as a relational DB.
  • BatchGet for many items rather than parallelizing GetItem (BatchGet returns partial; needs retry logic).

22. Cost model

  • Provisioned: per RCU/WCU/hour + storage.
  • On-demand: per-request + storage; ~7× per-unit but no min.
  • GSI: same WCU as base table for writes; doubles cost.
  • Global Tables: 2× writes per region.
  • Streams: per-million read requests by Lambda.
  • PITR: per GB-month of "continuous backup."
  • Storage: cheap; throughput is the bill.
