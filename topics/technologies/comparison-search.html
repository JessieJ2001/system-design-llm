Search: Comparison & Tradeoffs

Full-text search and vector search are siblings — both rank documents by relevance to a query. The split is shifting: every full-text engine now has vectors; every vector DB now has filters. The question is how much of each you need, and whether you can do it inside your existing DB or need a dedicated engine.

6. Migration paths

  • PG tsvector → Elasticsearch: typical "we outgrew it" path. Set up ES; stream PG → ES via Debezium or app-side dual-writes; shadow-search; cut over reads.
  • Elasticsearch → OpenSearch: wire-compatible up to ES 7.10; later versions diverge. Snapshot + restore is the path. Rewrite any X-Pack plugin code.
  • Solr → Elasticsearch: shape is similar (Lucene under both). Reindex, rewrite query DSL.
  • FAISS (offline) → Milvus / Qdrant (online): dump vectors + IDs, bulk-load into Milvus/Qdrant. Re-test recall.
  • pgvector → Milvus: when scale or QPS outgrows PG. Bulk export embeddings + metadata, re-import. Shadow query both, compare recall.
  • Pinecone → self-host (Qdrant): when cost matters. Pinecone exports limited; often re-embed from source.
  • Algolia → Typesense: records JSON is portable; reindex via script. Watch ranking semantics differences.

8. Hybrid patterns

Postgres + Elasticsearch (CDC)

PG is source of truth; Debezium tails WAL; ES indexes searchable fields. Searches hit ES; PG returns full record by ID. ES is rebuildable from PG.

Vector DB + relational metadata

Vector DB stores embeddings + lightweight metadata for filter pushdown. After top-K, hydrate full records from Postgres / DDB by ID. Keeps vector index lean.

Hybrid retrieval (BM25 + vector + reranker)

Two parallel retrievers: BM25 from ES, vector from Qdrant. Combine with RRF, then rerank top-N with a cross-encoder. The modern RAG stack.

FAISS in-process for read replicas

Build the index offline (e.g., nightly Spark job), bake it into the model server image, load into RAM at startup. Lowest latency at the cost of staleness.

Algolia for site search + ES for log search

Two distinct workloads, two distinct products. Don't try to unify.

pgvector + Postgres tsvector together

Single product, two engines. Cheap, simple, works at small scale. Migrate later if needed.

9. Anti-patterns

"We used Elasticsearch as primary store"

No transactions; segment merges block writes; refresh interval = stale; lossy on crash without proper translog. Always have a real DB behind it.

"Mapping explosion"

Dynamic field type detection generated thousands of fields; cluster state ballooned. Define explicit mappings; flatten user JSON before indexing.

"Deep pagination with from+size"

from+size>10000 fails by default; load grows with offset. Use search_after with a tiebreaker.

"Hybrid means UNION two queries"

Just UNIONing BM25 + vector results gives noisy ranking. Use RRF (Reciprocal Rank Fusion) or weighted ranking; better, use ES's built-in hybrid.

"We picked Pinecone for 10k vectors"

Way overkill; pgvector or even FAISS-in-process is faster and cheaper.

"We forgot the reranker"

ANN top-K is approximate; results are close but not always best. A small cross-encoder reranker on top-50 dramatically improves quality.

"Index built once, never updated"

HNSW grows costly to update; many teams batch-rebuild instead. Decide upfront: online updates (smaller HNSW + IVF rebuilds) or scheduled rebuilds (FAISS).

"We didn't filter before vector search"

For 1B vectors with a 1% filter, ANN-then-filter loses recall (most top-K are filtered out). Use filter-then-ANN where supported.

10. Decision rules for interviews

  • Default for full-text search: Elasticsearch / OpenSearch.
  • Site search for an ecommerce or content site: Algolia.
  • Data already in Postgres + low volume: tsvector + GIN.
  • Vector search at < 10M and low QPS: pgvector.
  • Vector search at > 10M or high QPS: Milvus / Qdrant / Weaviate.
  • Modern RAG = hybrid retrieval (BM25 + vector) + cross-encoder reranker.
  • Search engine is never source of truth. Always rebuildable from a real DB.
  • For deep pagination, use cursors (search_after), not offset.
  • For hot queries, cache at the app layer in front of the search engine.
  • Frozen tier / cold tier saves >80% on old logs.
