Apache ZooKeeper

CP coordination service: hierarchical znodes, ZAB consensus protocol (Paxos-like), session-based ephemeral nodes, one-shot watches. The original distributed-systems primitives library — Kafka (until KRaft), HBase, Hadoop, Solr, and many more depend on it. Heavyweight but battle-tested.

CP • ZAB • ephemeral znodes • watches

2. Underlying mechanism

  • Ensemble of 3 or 5 servers (always odd for quorum). One leader, rest followers.
  • ZAB protocol (Atomic Broadcast): leader assigns zxid (transaction id) to every write; broadcasts; quorum acks; commits.
  • Reads can be served by any follower (with relaxed consistency) or only via leader (linearizable). Default reads are sequentially consistent.
  • Hierarchical znodes: filesystem-like tree (/services/myapp/instance-001). Each znode has data, children, ACLs, version, ephemeral flag.
  • Ephemeral znodes: tied to a client session; auto-deleted when session expires. The basis for service discovery + leader election.
  • Sequential znodes: server appends a unique 10-digit suffix; useful for queues + leader-election recipes.
  • Watches: one-shot. Client registers watch on znode; on change, single notification; client must re-register.

3. Read & write path

Write (create / setData)

1. Client → any server (forwards to leader)
2. Leader assigns zxid, proposes to followers
3. Followers write to disk, ack
4. Leader waits for quorum (n/2 + 1)
5. Commits + broadcasts
6. Client gets success
7. Followers apply in zxid order

Read (getData)

Default: sequentially consistent (might be stale)
With watch: server registers callback
sync(): client forces leader-side read for linearizable read
Only sync()+read = "fresh"

Session

Client connects, gets session ID + timeout
Heartbeats keep session alive
Server tracks timeout
On expiry: ephemeral znodes deleted; watches discarded
Client must reconnect with new session

Watch

Set watch on znode
On change: server sends single notification
Client must re-register to see further changes
"Watch storms" possible if many clients watch same znode

4. Replication & consistency

  • ZAB consensus; quorum-based; CP (sacrifices availability under partition).
  • Sequential consistency by default; linearizable via sync() + read.
  • 3 nodes survive 1 loss; 5 nodes survive 2.
  • Cross-DC: not really. Observers are read-only non-voting nodes that span regions.
  • Failover: leader loss triggers election; new leader elected from quorum (~few seconds).

5. Partitioning

  • None. Single ensemble holds the entire data set.
  • Don't store data in ZK; only metadata.
  • If one ensemble isn't enough, run multiple ensembles by domain.

6. Time complexity

  • Write: O(log N) for znode tree manipulation + quorum ack latency.
  • Read: O(log N) tree walk + zxid check.
  • Watch fire: O(N) watchers on a znode.

7. Per-node capacity envelope

  • ~10k writes/sec/cluster (limited by quorum + disk fsync).
  • Reads: 50k+ QPS (followers serve).
  • Latency P99: 5–20 ms.
  • Data size: a few GB total; explicitly not for "data."
  • Connections / watches: thousands per node.

8. When to use / not to

Use ZK when…

  • Existing system requires ZK (Kafka pre-KRaft, HBase, Solr Cloud).
  • Need leader election with watch-based notification.
  • Distributed locks with fencing tokens.
  • Configuration with watch-based push.
  • Service discovery (registration via ephemeral znodes).

Don't use ZK when…

  • You're starting from scratch — etcd is simpler.
  • You don't need it: most apps don't.
  • Application data store (it's not for that).
  • High-throughput writes (~10k/sec ceiling).
  • Cross-region: ZAB tolerates cross-DC poorly.

9. Common gotchas

  • One-shot watches must be re-registered. Forgetting = missed events.
  • Watch storm: 1000 clients watching one znode → 1000 notifications on every change.
  • Session expiry: ephemeral znodes vanish; clients must rebuild state.
  • Default 1 MB znode size cap; raise carefully — large znodes slow everything.
  • Many znodes (millions) → long bootstrap + recovery times.
  • Operator footprint: JVM tuning, GC tuning, snapshot management.
  • Disk fsync latency caps write throughput; SSD essential.
  • Mismatched ZK version across clients → protocol bugs.
  • Leader-election recipes are non-trivial; use Curator instead of rolling your own.
  • Distributed lock anti-patterns: no fencing token by default; you must check version after acquire.

10. Cloud equivalents

  • None managed natively in any cloud.
  • You self-host on EC2/GCE/AKS.
  • Alternatives that are cloud-managed in spirit: AWS Cloud Map (service discovery), GCP Service Directory, Consul on HCP.
  • Open-source alternatives: etcd (Raft, simpler), Consul (Raft + gossip), Eureka (AP).

12. Usage patterns

  • Service registry (ephemeral znode per instance).
  • Leader election (sequential ephemeral; lowest seq is leader).
  • Distributed lock (sequential ephemeral; with fencing tokens).
  • Distributed barrier / latch.
  • Configuration with live update (set + watch).
  • Group membership.
  • Counter / queue (recipes; rarely the right tool).
  • Metadata for distributed systems (Kafka, HBase).

13. Operational knobs

  • tickTime: heartbeat unit.
  • initLimit / syncLimit: follower sync windows.
  • JVM heap (4-8 GB typical).
  • Snapshot interval / autopurge config.
  • Max client connections per host.
  • Disk: snap dir + log dir — fast SSD essential.

14. Capacity planning

  • Ensemble size: 3 or 5; larger = more replication overhead, no big availability win past 5.
  • Memory: in-memory tree fits in RAM (znodes + watches).
  • Disk: snapshots + tx logs.
  • Plan for 10k writes/sec ceiling.

15. Failure modes

  • Quorum loss (lose >= n/2): cluster goes read-only / unresponsive.
  • Leader loss: ~few seconds for election.
  • Slow follower: slows writes (waits for quorum).
  • Disk slow: write latency spikes.
  • Long GC pause: session timeouts; ephemeral znodes vanish.
  • Network partition: minority side becomes read-only.

16. Backup & DR

  • Snapshot files + transaction log; copy off-site.
  • Restore: deploy fresh ensemble, restore data dir.
  • RPO: snapshot interval.
  • RTO: ensemble bootstrap + replay.
  • Cross-region: observers + manual cut over for DR.

17. Upgrade & migration

  • Rolling upgrade: one node at a time; quorum maintained.
  • Major version: read upgrade notes; format may change.
  • Migrate to KRaft (for Kafka): documented multi-stage state machine; once migrated, ZK can go.
  • Migrate to etcd: usually means rewriting client code.

18. Security

  • ACLs per znode: world / auth / digest / IP / SASL.
  • TLS for client + quorum since 3.5+.
  • SASL Kerberos for enterprise auth.
  • No native encryption at rest; use disk encryption.
  • No native audit log; ops via JMX / 4-letter words.

19. Multi-tenancy

  • Per-tenant subtree with ACLs.
  • No native quotas (since 3.5 there's a quota mechanism, rarely used).
  • One ensemble per tenant tier in practice.

20. Observability

  • Four-letter words: stat, mntr, cons, dump.
  • JMX MBeans.
  • Prometheus exporter (since 3.6+).
  • Key SLIs: outstanding requests, watch count, znode count, leader vs follower role, fsync time.

21. Anti-patterns

  • Storing application data in ZK.
  • Many small znodes for high-volume metadata.
  • Many watches on one hot znode (watch storm).
  • Cross-region voting members (latency ruins quorum).
  • Distributed lock without understanding fencing.
  • Rolling your own leader-election recipe (use Curator).
  • Treating session expiry as a transient blip.
  • Big znode payloads — 1 MB cap; even 100 KB is suspect.

22. Cost model

  • 3–5 JVM nodes × instance hours.
  • SSD storage (small).
  • Cross-AZ replication network.
  • Operator time: tuning + on-call — the silent biggest cost.
