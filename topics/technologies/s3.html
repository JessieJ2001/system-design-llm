Amazon S3

Erasure-coded object store with replicated metadata, multi-AZ durability (eleven 9s), HTTP-addressable buckets/keys, virtually unlimited capacity. The default landing pad for blobs, backups, data lakes, logs, anything-immutable. Strong read-after-write consistency since December 2020.

object storage • 11 9s • HTTP API • the data lake floor

2. Underlying mechanism

  • Bucket + key (hierarchical-looking but flat key space; "/" is just a character).
  • Object = blob + metadata + version (if versioning on).
  • Storage class chosen at PUT (Standard / IA / Glacier / Deep Archive / Express One Zone).
  • Internally: erasure-coded across many devices in multiple AZs (specific scheme not public, but ~1.5–1.7× storage overhead).
  • Strongly-consistent metadata index since Dec 2020 (a major change — older docs you'll find online describe the old eventual-consistent LIST behavior).
  • Multipart upload: split large object into parts; upload in parallel; complete with one call.

3. Read & write path

PutObject

1. Client signs request (SigV4)
2. POST/PUT to https://<bucket>.s3.<region>.amazonaws.com/<key>
3. AWS routes to a partition by hash(prefix)
4. Erasure-encode + write to multiple AZs
5. Update metadata index (replicated)
6. 200 OK with ETag (md5 of object)

For large: InitiateMultipartUpload → UploadPart (parallel) → CompleteMultipartUpload

GetObject

1. Client signs GET
2. Metadata lookup → locate object
3. Stream from storage layer
4. Range header for partial fetch (chunked reads)
5. CloudFront in front for static caching

P99 first-byte: 50–200ms; throughput: many GB/s with parallel range reads

List

ListObjectsV2 returns up to 1000 keys, paginated
With prefix to scope; with delimiter to simulate folders
Slow on large buckets — use S3 Inventory CSV (daily) for batch needs

Lifecycle

Per-bucket rules:
- Transition to IA / Glacier / Deep Archive after N days
- Expire (delete) after N days
- Abort incomplete multipart uploads after N days (don't forget!)
- Apply by prefix or tag

4. Replication & consistency

  • Strong read-after-write for new objects + overwrites + deletes (since 2020).
  • Multi-AZ durability by default (Standard / IA); One Zone class skips multi-AZ for cost savings.
  • Cross-Region Replication (CRR) and Same-Region Replication (SRR): async; replicates new objects only by default.
  • Object versioning: every PUT creates a new version; DELETE creates a delete marker.
  • Multi-Region Access Points: route to nearest replicated bucket.

5. Partitioning

  • S3 partitions internally by key prefix (specifically by the leading characters of the key).
  • Per-prefix scaling: 3,500 PUT/COPY/POST/DELETE and 5,500 GET/HEAD requests per second per prefix.
  • Hot prefix → SlowDown 503s. Solution: random hex prefix or distribute keys.
  • Adaptive partitioning: AWS auto-splits hot prefixes, but takes time; for predictable bursts, randomize keys.

6. Time complexity

  • PUT / GET: O(1) request; bandwidth-bound for object body.
  • LIST: O(N / page) paginated; expensive at scale.
  • HEAD: O(1).
  • Multipart: parallel chunks; speedup linear with parts.
  • Glacier restore: minutes to hours (Expedited / Standard / Bulk tiers).

7. Per-node capacity envelope

  • Per prefix: 3,500 PUT/sec / 5,500 GET/sec.
  • Per bucket: effectively unlimited (auto-scaled).
  • Per object: 5 TB max; 5 GB single-PUT (use multipart above).
  • Throughput per object via parallel range reads: many GB/s achievable.
  • Cross-region transfer: bandwidth-driven by network + AWS network.

8. When to use / not to

Use S3 when…

  • User uploads / static assets / images / videos.
  • Backups / log archive / DR.
  • Data lake (Parquet / Iceberg / Delta).
  • ML training data / model artifacts.
  • Stream processor checkpoints.
  • Large blobs paired with KV/SQL pointer.

Don't use S3 as…

  • POSIX filesystem (no real seeks/locks; consistency edges with FUSE).
  • Block device (use EBS).
  • Low-latency KV (50–200ms first byte; use Dynamo / Redis).
  • Search engine (use ES).
  • Many-small-files (LIST is expensive; consider packing).

9. Common gotchas

  • Sequential prefix in keys (e.g., logs/2026-05-01/...) hits 3,500 PUT cap on one prefix.
  • LIST in real-time is expensive; use S3 Inventory CSV.
  • VARY: Cookie at CloudFront in front of S3 destroys hit rate.
  • Forgot lifecycle rule for incomplete multipart uploads → pay forever.
  • Versioning enabled but no expiry rule for old versions → storage grows indefinitely.
  • One bucket per service can hit account bucket limits; use prefixes.
  • Bucket policy + ACL + IAM all interact; permissions confusion is common.
  • Public access by default off (Block Public Access); used to be a major footgun.
  • Deletes are eventual to "show as gone in LIST" if versioning + rules — understand delete markers.
  • CRR doesn't replicate existing objects without batch operations.

10. Cloud equivalents

  • AWS: S3 (this is the cloud).
  • GCP: GCS.
  • Azure: Blob Storage.
  • Open-source / on-prem: MinIO (S3-API compat); Ceph RGW; Swift.
  • Cloud-agnostic: Cloudflare R2 (zero-egress S3).

12. Usage patterns

  • User uploads with pre-signed URL (browser uploads directly).
  • Static asset hosting + CloudFront.
  • Backup destination (DB snapshots, app backups).
  • Data lake (Parquet partitioned by date; Iceberg / Delta tables).
  • Log archive (CloudWatch / Athena / Glue).
  • Cross-region DR.
  • S3 Event Notifications → Lambda / SQS / SNS for event-driven processing.
  • Output for stream processors (Flink checkpoints, Spark output).
  • Large items + KV pointer.

13. Operational knobs

  • Storage class.
  • Lifecycle rules (transition + expire + abort multipart).
  • Versioning + Object Lock.
  • Replication rules (CRR / SRR).
  • Encryption (SSE-S3 / SSE-KMS / SSE-C).
  • Bucket policy + Block Public Access.
  • Access points (per-app).
  • Inventory + Storage Lens.

14. Capacity planning

  • Capacity = unlimited; don't plan; plan cost instead.
  • Throughput per prefix: 3.5k PUT / 5.5k GET; spread keys.
  • Multipart for > 100 MB objects; recommended at 5+ GB.
  • Egress cost is the main bill driver.

15. Failure modes

  • Multi-AZ failure: extremely rare; durability designed for 11 9s.
  • Hot prefix: 503 SlowDown; client should retry with backoff.
  • Region outage: bucket unavailable; CRR to another region for DR.
  • Account-level limits: bucket count, request rate.
  • Permission misconfiguration: most common "outage" cause.

16. Backup & DR

  • S3 itself is durability-as-a-service; "backup" usually means CRR + Object Lock + separate account.
  • For ransomware: replicate to a separate account with no delete permission; use Object Lock.
  • Recover deleted versions via versioning.
  • RPO: replication lag (seconds-minutes).
  • RTO: copy back if needed.

17. Migration paths

  • Cross-cloud: rclone / Storage Transfer Service / DataSync.
  • S3 to S3 (region): CRR + Batch Operations to copy existing.
  • HDFS to S3: distcp + adapter; common modernization.
  • Ceph / MinIO → S3: rclone.

18. Security

  • IAM policies + bucket policies + Block Public Access.
  • Pre-signed URLs for time-limited access.
  • SSE-KMS with customer-managed keys for compliance.
  • VPC endpoints to keep traffic off the internet.
  • CloudTrail data events for object-level audit.
  • Object Lock (compliance / governance) for WORM.

19. Multi-tenancy

  • Bucket per tenant: simple, runs into account limits.
  • Prefix per tenant + IAM condition (s3:prefix) for isolation.
  • Pre-signed URLs scoped per tenant.
  • Access points: per-tenant policy on a shared bucket.

20. Observability

  • CloudWatch metrics: bucket size, request count, 4xx/5xx.
  • S3 Server Access Logs (delivered to a logs bucket).
  • S3 Storage Lens: organization-wide storage analytics.
  • S3 Inventory: daily CSV of objects.
  • CloudTrail data events: per-object PUT/GET audit.

21. Anti-patterns

  • Sequential key prefix at high write rate.
  • LIST in real-time on huge buckets.
  • VARY: Cookie at CDN in front.
  • Forgetting incomplete multipart cleanup.
  • Versioning without expiry → bills grow.
  • Hosting hot, frequently-mutated data (use a real DB).
  • Treating S3 as a filesystem via FUSE.

22. Cost model

  • Storage: per GB-month by class. Standard ~$0.023; IA ~$0.0125; Glacier Deep ~$0.004.
  • Requests: PUT $0.005/1000; GET $0.0004/1000; LIST is "PUT-priced" expensive.
  • Egress dominates at scale; intra-region free; cross-AZ has charges; internet egress most expensive.
  • Lifecycle transitions cost per object; cheap individually but add up on huge buckets.
  • Glacier retrieval has per-GB fee + minimum 30/90/180-day storage charge.
