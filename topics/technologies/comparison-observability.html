6. Migration paths

  • Custom metrics → Prometheus: instrument with the Prom client lib; expose /metrics; scrape. Most apps in < 1 day.
  • Prometheus → Mimir / Thanos / VictoriaMetrics: add remote_write to existing Prom. Grafana points at Mimir for long queries. Decommission local Prom long-term storage.
  • ELK → Loki: typical "logs are too expensive" path. Promtail / OTel collector ships logs to Loki on S3. ELK keeps 30-day indexed; Loki keeps 1-year cheap. Migrate one log source at a time.
  • Vendor APM → OpenTelemetry: the de-vendor move. Replace agent with OTel SDK + collector. Backend pluggable.
  • Datadog → open-source: Prometheus + Loki + Tempo + Grafana ("PLG/T stack"). Operational cost >0 but bill drops dramatically.
  • Self-host Prom → Managed Prometheus: AWS / GCP managed Prom for the data; Grafana Cloud / managed Grafana for the UI.

8. Hybrid patterns

The "PLG" stack: Prometheus + Loki + Grafana

Open-source default for new infra. Add Tempo for traces ("PLGT"). All four products from Grafana Labs; deep integration.

OpenTelemetry as the universal source

Apps emit OTel; collector sends metrics to Prom, logs to Loki, traces to Tempo. Switch any backend without changing instrumentation.

trace_id flowing across pillars

Every log line includes trace_id from OTel context. Click span in Tempo, jump to logs in Loki for that trace. The single biggest debugging UX win.

RED + USE

RED (Rate, Errors, Duration) for services; USE (Utilization, Saturation, Errors) for resources. Two metrics templates that cover >90% of dashboards.

Tail-based sampling for errors

OTel collector buffers spans, decides at trace end: error or slow = keep; otherwise drop. Cuts span volume 10× while preserving the interesting traces.

Logs → metrics derivation

Many platforms (Datadog log-based metrics; Loki recording rules) derive metrics from log patterns. Cheaper than emitting a separate metric.

9. Anti-patterns

"User ID as a Prometheus label"

Cardinality blows up; OOM. Labels are for grouping, not unique IDs. Use traces / logs.

"Logged everything at INFO"

Volume is the bill. Sample / drop noisy logs at the agent. Reserve INFO for state changes; DEBUG for opt-in.

"100% trace sampling"

Performance hit on app + storage cost. 1–10% head-based + tail-based for errors is the standard.

"No correlation between pillars"

Logs without trace_id; metrics without service.name. Each pillar is a silo. Add OTel context everywhere.

"Alerted on raw CPU"

Alert on user-visible SLOs (latency, error rate) not infrastructure metrics. CPU at 90% might be fine.

"Datadog cost surprise"

Custom-metric explosion or log volume spikes. Set hard limits per service; monitor your own observability bill.

"Logs as primary store"

Querying ELK for business analytics — expensive and slow. Use a warehouse for analytics; logs for ops.

10. Decision rules for interviews

  • Default open-source: Prometheus + Loki + Grafana + Tempo, instrumented with OpenTelemetry.
  • Default managed: cloud-native (CloudWatch / Cloud Monitoring) for basic; Datadog for richer.
  • Always use OTel for instrumentation — it decouples app from backend.
  • Cardinality is the cliff for Prometheus / Loki. Don't put unique IDs in labels.
  • RED + USE templates cover most dashboards.
  • Always sample traces. Head-based 1–10% baseline, tail-based for errors.
  • Always include trace_id in logs and span_id in errors.
  • Alert on user-visible SLOs (latency, error rate), not raw CPU.
  • Long retention: tier to cheap storage (S3 via Loki / Mimir); don't pay hot-tier rates for year-old logs.
  • Logs are not analytics. Don't run business queries on the log store.
