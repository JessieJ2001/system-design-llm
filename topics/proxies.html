Proxies

Intermediaries that intercept, route, and transform traffic between clients and servers — from corporate forward proxies to Envoy sidecars powering service meshes.

30 min read

What Is It?

A proxy is an intermediary server that sits between a client and a destination server, forwarding requests on behalf of one side or the other. The word "proxy" literally means "acting on behalf of," and depending on whose behalf the proxy acts, it falls into one of three major categories: forward proxy, reverse proxy, or sidecar proxy.

A forward proxy sits on the client side. It acts on behalf of clients, forwarding their requests out to the internet. Corporate networks use forward proxies (like Squid) to enforce access policies, cache frequently-visited sites, and anonymize outbound traffic. VPNs are conceptually similar — they tunnel all client traffic through an intermediary. The destination server sees the proxy's IP address, not the client's.

A reverse proxy sits on the server side. It acts on behalf of backend servers, receiving inbound requests from the internet and routing them to the appropriate backend. Nginx and HAProxy are the canonical examples. Reverse proxies provide TLS termination (decrypting HTTPS at the edge so backends handle plain HTTP), compression (gzip/brotli responses before sending to clients), connection pooling (maintaining persistent connections to backends so each client request does not trigger a new TCP handshake), and request routing (directing /api to one service and /static to another).

A sidecar proxy is a modern pattern popularized by service meshes like Istio and Linkerd. Instead of a single centralized proxy, every service instance gets its own proxy process (typically Envoy) deployed alongside it in the same pod or host. The sidecar transparently intercepts all inbound and outbound traffic for its service, handling mutual TLS (mTLS), retries, circuit breaking, load balancing, and observability — all without the application code knowing about it. This moves cross-cutting network concerns out of application logic and into infrastructure.

How It Works

Let us trace a request through a reverse proxy step by step. This is the most common proxy pattern in production systems.

Step 1: Client Connection

A client (browser, mobile app, API consumer) opens a TCP connection to the reverse proxy's public IP address on port 443. The client has no direct knowledge of which backend server will ultimately handle its request.

Step 2: TLS Termination

The reverse proxy performs the TLS handshake with the client. It holds the SSL certificate and private key, negotiates cipher suites, and establishes an encrypted channel. This is called TLS termination because the encrypted (TLS) connection terminates at the proxy. Traffic between the proxy and backend servers typically flows over plain HTTP on a private network, avoiding the CPU cost of encryption on every backend. For environments requiring end-to-end encryption, the proxy can re-encrypt traffic to backends (TLS passthrough or TLS re-encryption), though this adds latency.

Step 3: Header Manipulation

The proxy inspects the HTTP request and adds or modifies headers. Critical headers include:

  • X-Forwarded-For: The original client IP address (since the backend will see the proxy's IP as the source)
  • X-Forwarded-Proto: Whether the original request was HTTP or HTTPS
  • X-Request-ID: A unique identifier for distributed tracing
  • Host: May be rewritten for virtual-host-based routing

Step 4: Request Routing

Based on the URL path, Host header, or other criteria, the proxy selects a backend upstream group. For example, requests matching /api/* go to the API service pool, while /static/* goes to a file server or CDN origin. Within the selected pool, the proxy applies a load-balancing algorithm (round-robin, least connections, consistent hashing) to pick a specific backend instance.

Load Balancing Algorithms at the Proxy

Once the proxy has selected the correct upstream group, it must pick which backend instance receives this particular request. Here are the three most common algorithms, each with a concrete walkthrough.

Round-Robin

The simplest strategy: maintain a counter and cycle through backends in order. Given three servers — A, B, and C — here is how 10 requests are distributed:

Request #12345678910
Routed toABCABCABCA

Result: A gets 4 requests, B gets 3, C gets 3. The distribution is nearly perfect — but round-robin ignores how busy each server is. If server A is processing a slow 5-second database query, it still receives the next request in rotation, even while B and C are idle.

Least Connections

Instead of blindly cycling, the proxy tracks how many active (in-flight) connections each backend currently has and sends each new request to the backend with the fewest. This naturally accounts for backends that are slow or processing heavy requests, because their connection count stays high longer.

Walk through this timeline with 3 servers (A, B, C), all starting at 0 active connections:

Notice at t=50ms: server A is slow and has 2 active connections, so new requests automatically avoid it. This is why least-connections is the default choice for most production proxies — it adapts to heterogeneous response times without manual tuning.

Consistent Hashing (When You Need It)

Round-robin and least-connections treat every request independently. But what if your backends have per-server caches? If user Alice's requests go to server A on one call and server B on the next, neither server builds a useful cache for Alice. You want the same user to always hit the same server (as long as that server is healthy).

Consistent hashing solves this by hashing a key from the request (like a user ID or session ID) and mapping it to a server. The critical property: when a server is added or removed, only a small fraction of keys are remapped, not all of them. This is essential for maintaining cache hit rates during scaling events.

In Nginx, this is configured as hash $request_uri consistent; in the upstream block. Envoy supports it via the RING_HASH load balancing policy.

For a full deep-dive with worked examples, virtual nodes, and the ring data structure, see Consistent Hashing.

Step 5: Connection Pooling

Rather than opening a new TCP connection to the backend for every request, the proxy maintains a connection pool — a set of pre-established, persistent (keepalive) connections to each backend. This eliminates the TCP handshake and slow-start overhead for every request. A well-configured proxy reuses connections aggressively: Nginx's upstream keepalive directive, for instance, keeps a specified number of idle connections open per worker process.

Step 6: Response and Compression

The backend sends its response back to the proxy. The proxy can compress the response body (gzip or brotli), add caching headers, strip internal headers that should not leak to clients (e.g., X-Internal-Debug), and send the final response back to the client over the established TLS connection.

Proxy Buffering and Backpressure

Why buffering exists: Backends are typically fast — they generate a response in milliseconds. Clients are typically slow — they download over the internet at limited bandwidth. Without buffering, the backend must hold its connection open until the slow client finishes downloading. This ties up a valuable backend connection (and often an entire backend thread or process) for the duration of the download, not the duration of the computation.

Buffering decouples the fast backend from the slow client. The proxy absorbs the backend's full response into memory (or disk), immediately frees the backend connection, and then drains the buffer to the client at whatever speed the client can handle.

Concrete Scenario: With vs. Without Buffering

Imagine a backend that generates a 10 MB API response (a large data export) in 100 ms. The client downloads at 1 MB/s (a mobile user on a weak connection).

MetricWithout BufferingWith Buffering
Backend connection held for10 seconds (waiting for client to download all 10 MB at 1 MB/s)100 ms (proxy absorbs the full response, then frees the backend)
Backend connections needed (10 concurrent slow clients)10 connections occupied for 10s each = 10 connections tied up10 connections each held for 100ms = ~1 connection on average

Proxy memory costMinimal (pass-through)10 MB per response buffered in RAM (or spilled to disk)

The tradeoff is clear: buffering uses proxy memory but frees backend connections 100x faster in this example. For most web applications where clients are slower than backends, buffering is a net win. The exception is streaming responses (Server-Sent Events, chunked transfers for real-time data) where the proxy should not buffer but instead pass data through immediately.

Backpressure

What happens when the proxy runs out of buffer memory? This is where backpressure comes in. The proxy stops reading from the backend (TCP flow control causes the backend's send buffer to fill, which eventually causes the backend to block on its write call). This propagates the "slow client" problem back to the backend — exactly what buffering was supposed to prevent. To avoid this, configure buffer limits that are large enough for your typical response sizes, and monitor for proxy_buffer warnings in the error log. In Nginx, the proxy_max_temp_file_size directive controls how much response data can be spilled to disk when RAM buffers are full (default: 1 GB).

Sticky Sessions (Session Affinity)

The problem: Some backends store user session state in local memory (login sessions, shopping carts, in-memory caches). If a user's first request goes to server A and their second request goes to server B, server B has no knowledge of the session created on server A. The user appears logged out, their cart is empty, and the experience breaks.

The solution: Sticky sessions ensure that once a user is routed to a specific backend, all subsequent requests from that user go to the same backend for the duration of their session.

Cookie-Based Session Affinity

The most common implementation uses a cookie set by the proxy:

  1. User sends first request. The proxy picks a backend (e.g., server A) using its normal load-balancing algorithm.
  2. The proxy adds a cookie to the response: Set-Cookie: SERVERID=srv-a; Path=/; HttpOnly.
  3. On subsequent requests, the user's browser sends Cookie: SERVERID=srv-a.
  4. The proxy reads the cookie and routes directly to server A, bypassing the load-balancing algorithm.

In Nginx, this is configured with the sticky cookie directive (available in Nginx Plus) or the ip_hash directive for IP-based stickiness. In HAProxy, use cookie SERVERID insert indirect nocache.

What Happens When the Sticky Backend Dies?

If server A crashes or fails its health check, the proxy can no longer route to it. The user's next request is sent to a different backend (say, server B). Since server B has no knowledge of the session that was on server A, the session is lost. The user is effectively logged out and must start over. This is the fundamental weakness of sticky sessions with local state.

Mitigation strategies:

  • Externalize session state: Store sessions in Redis or a database. All backends can read any session, eliminating the need for stickiness entirely. This is the recommended approach for any system that needs high availability.
  • Session replication: Backends replicate session data to each other (e.g., Tomcat session clustering). Adds complexity and network overhead but keeps sessions alive across failovers.
  • Graceful degradation: Accept that failover loses the session. Redirect the user to a login page with a friendly message. This is acceptable for non-critical sessions.

IP-Based Stickiness and Its Limitations

An alternative to cookie-based stickiness is hashing the client's IP address: hash(client_ip) % num_backends. This works without cookies (useful for non-browser clients), but has serious limitations:

  • NAT (Network Address Translation): All users behind the same corporate NAT gateway share one public IP. They all get routed to the same backend, creating a hot spot. A single corporate office with 5,000 employees would send all traffic to one server.
  • Mobile networks: Mobile carriers frequently change the user's IP address (as the device moves between cell towers or switches between Wi-Fi and cellular). The user's "sticky" backend changes mid-session.
  • IPv6 adoption: With IPv6, every device has a unique address, which fixes the NAT problem but means the hash distribution must handle a much larger key space.

For these reasons, cookie-based stickiness is preferred over IP-based whenever the client supports cookies.

When to Use / When NOT to Use

Use a Proxy When

  • TLS termination: You want to offload HTTPS decryption from application servers, centralizing certificate management in one place
  • Connection pooling: Your backends would otherwise be overwhelmed by the sheer number of client TCP connections (e.g., thousands of mobile clients with long-lived connections)
  • Request routing: You need path-based, header-based, or host-based routing to different backend services from a single entry point
  • Caching at the edge: You want the reverse proxy to cache static assets or API responses, reducing backend load by orders of magnitude
  • Rate limiting: You want to throttle abusive traffic before it reaches your application servers
  • Observability: You need a central point for access logs, metrics, and distributed tracing header injection

Avoid a Proxy When

  • Ultra-low-latency (<1ms) requirements: Every proxy hop adds latency (typically 0.5–2ms). For intra-process communication or HFT systems, a proxy in the path is unacceptable
  • Simple single-server setups: If you have one application server and no need for TLS termination, routing, or caching, the proxy adds complexity with no benefit
  • Proxy becomes a SPOF: A single proxy without redundancy is worse than no proxy — it is an additional single point of failure. If you cannot afford to run at least two proxy instances, reconsider
  • End-to-end encryption mandates: Some compliance regimes (PCI-DSS for cardholder data) require encryption from client to backend. TLS termination at the proxy would violate this unless you re-encrypt
  • UDP-heavy workloads: Traditional HTTP reverse proxies do not handle UDP traffic (gaming, VoIP). You need a specialized L4 proxy or direct connections

Choose Forward vs. Reverse Proxy

Forward proxy when you need to control or monitor outbound traffic from a network (corporate firewall, web scraping anonymization, content filtering). Reverse proxy when you need to control inbound traffic to your servers (TLS, routing, load balancing, caching). Sidecar proxy when you have a microservices architecture and want per-service traffic management without modifying application code.

Classic vs. Modern Approaches

Classic: Nginx and HAProxy

Nginx (2004) and HAProxy (2000) are the workhorses of reverse proxying. Both are battle-tested, highly performant, and run in production at enormous scale. Nginx uses an event-driven, non-blocking architecture with a static configuration file (nginx.conf) that is reloaded by sending a signal to the master process. HAProxy excels at L4 proxying and sophisticated health checking. Both require a config reload (and in Nginx's case, a graceful worker restart) to pick up changes — there is no dynamic configuration API out of the box.

Modern: Envoy (Observability-First, Dynamic Config)

Envoy (open-sourced by Lyft in 2016) was designed from the ground up for modern microservices. Its key differentiators:

  • xDS API for dynamic configuration: Envoy does not require restarts or reloads. A control plane (like Istio's Pilot) pushes route tables, cluster definitions, and listener configurations to Envoy at runtime via gRPC streams. This is critical for environments where services scale up and down constantly (Kubernetes).
  • First-class observability: Envoy emits detailed stats (latency histograms, connection counts, retry rates) per upstream cluster, supports distributed tracing (Zipkin, Jaeger, OpenTelemetry) out of the box, and provides structured access logs.
  • Advanced load balancing: Zone-aware routing, automatic retries with budget limits, circuit breaking, and outlier detection (ejecting unhealthy hosts based on consecutive 5xx errors).

Service Mesh: Istio and Linkerd (Sidecar Pattern)

A service mesh deploys a sidecar proxy (usually Envoy for Istio, linkerd2-proxy for Linkerd) alongside every service instance. A control plane (Istiod, Linkerd's destination service) manages the fleet of sidecars, distributing mTLS certificates, routing rules, and traffic policies. The application sends plain HTTP to localhost; the sidecar transparently handles encryption, retries, timeouts, and telemetry. The tradeoff is operational complexity and resource overhead (each sidecar consumes CPU and memory).

WASM Plugins for Envoy

Envoy supports extending its functionality with WebAssembly (WASM) plugins, allowing custom logic (authentication, transformation, rate limiting) to run inside the proxy without recompiling Envoy or writing C++. Languages like Rust, Go, and AssemblyScript compile to WASM modules that Envoy loads at runtime. This decouples the proxy release cycle from custom business logic.

eBPF-Based Proxying: Cilium

Cilium takes a radically different approach: it uses eBPF programs running in the Linux kernel to handle networking decisions before packets reach userspace. For service-to-service traffic within a Kubernetes cluster, Cilium can bypass the sidecar proxy entirely, routing traffic at the kernel level. This eliminates the per-pod sidecar overhead and reduces latency. Cilium still integrates with Envoy for L7 policy enforcement when needed, but avoids the sidecar tax for L3/L4 traffic.

Assumptions & Model

Key Assumptions When Modeling Proxies

  • Proxy adds minimal latency: We assume proxy overhead is sub-millisecond for L4 forwarding and 1–3ms for L7 processing (TLS termination, header inspection, routing). This breaks down under heavy CPU load (TLS with RSA key exchange), when the proxy is saturated, or when complex WASM plugins add per-request processing.
  • Proxy is reliable: We assume the proxy is always available. In practice, a single proxy instance is a SPOF. You need at least two instances behind a virtual IP (VRRP/keepalived) or cloud load balancer (NLB/ALB) to tolerate failures.
  • Proxy has capacity for all connections: We assume the proxy can handle the connection count. A single Nginx instance can handle ~100K–500K concurrent connections depending on configuration. If your client count exceeds this, you need multiple proxy instances or connection coalescing strategies.
  • Network between proxy and backends is fast and reliable: Reverse proxy models assume the internal network has negligible latency and no packet loss. In multi-region or multi-cloud setups, this assumption fails, and you need geo-distributed proxy tiers.
  • Backends are homogeneous: Connection pooling and load balancing models often assume backends have roughly equal capacity. Heterogeneous backends (different instance sizes) require weighted routing.

Sizing & Tradeoffs

Connection Pool Sizing

The goal is to maintain enough idle connections to backends to avoid TCP handshake overhead, without wasting memory on connections that sit unused. A good rule of thumb:

pool_size_per_backend = peak_rps_per_backend × avg_response_time_seconds × 1.5

For example: if a backend handles 500 RPS with a 50ms average response time, you need 500 × 0.05 × 1.5 = 37.5, so set the pool to ~40 connections per backend. The 1.5x multiplier provides headroom for bursts.

TLS CPU Cost

TLS termination is CPU-intensive, especially the handshake. RSA-2048 key exchange costs ~1ms of CPU per handshake; ECDSA P-256 is ~10x cheaper (~0.1ms). For a proxy handling 10,000 new TLS connections per second with RSA-2048, you need roughly 10 CPU cores dedicated to handshakes alone. Modern proxies use TLS session tickets or session caching to avoid full handshakes on reconnections, reducing CPU by 5–10x for returning clients.

Buffer Sizes

Proxies buffer request and response bodies. Key settings:

  • proxy_buffer_size (Nginx): Size of the first buffer for reading the response header. Default 4K–8K, sufficient for most responses.
  • proxy_buffers (Nginx): Number and size of buffers for the response body. Default 8 4k. For large API responses (1MB+), increase to 16 32k.
  • Total memory per connection = sum of all buffers. With 100K concurrent connections and 256K buffers each, that is 25GB of RAM just for buffering.

SLO-Driven Sizing Examples

Example 1: Reverse Proxy for Web App

SLO: p99 < 50ms added proxy latency at 100K RPS.

Backend setup: 40 backend servers, average response time 80ms, 90% HTTPS with 75% TLS session reuse.

Connection pool sizing: 100,000 RPS / 40 backends = 2,500 RPS per backend. Pool size = 2,500 × 0.08 × 1.5 = 300 connections per backend. Total backend connections: 12,000.

Proxy instance count: A single Nginx instance on a 4-vCPU node handles ~30K RPS at p99 < 50ms (with TLS termination). For 100K RPS: ceil(100,000 / 30,000) = 4 instances. Add 1 for redundancy = 5 Nginx instances (4 vCPU, 8GB RAM each).

TLS handshakes: 100K × 0.9 × 0.25 = 22,500 new handshakes/sec. Using ECDSA: 22,500 × 0.1ms = 2.25 CPU cores across the fleet (0.45 per instance — well within budget).

Buffer memory per instance: 100K / 5 = 20K connections × 32KB = 640MB per instance.

Validation: Load test at 120K RPS (1.2x headroom). If p99 exceeds 50ms, add a 6th instance or increase per-instance vCPU to 8.

Example 2: Forward Proxy for Corporate Network

SLO: Support 10K concurrent users with content filtering; page load overhead < 200ms at p95.

Traffic profile: 10K users generate ~5K concurrent connections (50% concurrency ratio), each averaging 3 requests/sec = 15K RPS. Average response size: 50KB. Content filtering (URL categorization + malware scan) adds ~10ms CPU per request.

CPU sizing: Content filtering: 15,000 × 10ms = 150 CPU-seconds/sec = 150 vCPU at full utilization, or ~190 vCPU at 80% target utilization. Spread across proxy nodes: 190 / 8 vCPU per node = 24 Squid/proxy nodes.

Memory sizing: Connection state: 5K connections × 256KB buffer = 1.3GB. URL category cache (10M entries × 200 bytes) = 2GB. Per node: (1.3GB / 24) + 2GB shared cache = ~2.1GB. Allocate 4GB RAM per node (headroom for spikes and OS).

Validation: Monitor p95 page load delta (with proxy vs without). If content filtering latency exceeds 10ms, scale horizontally or offload malware scanning to a dedicated tier.

Example 3: Service Mesh Sidecar (Envoy)

SLO: p99 < 5ms sidecar overhead per hop, 50K RPS per pod.

Baseline overhead: Envoy sidecar at idle consumes ~10MB memory and ~0.01 vCPU. Under load, each request costs ~0.02ms of CPU for L7 proxying + mTLS.

CPU resource limit: 50,000 RPS × 0.02ms = 1.0 CPU-second/sec = 1.0 vCPU at full utilization. To keep p99 < 5ms, target 60% CPU utilization: 1.0 / 0.6 = 1.7 vCPU limit. Round to requests: 1.0, limits: 2.0 vCPU.

Memory resource limit: Base: 10MB. Per-connection state: 50K concurrent connections × 2KB = 100MB. Configuration/routing tables for 200 services: ~20MB. Total: ~130MB. Set requests: 128Mi, limits: 256Mi (2x headroom for spikes).

Fleet impact: 500 pods × 2 vCPU limit = 1,000 vCPU sidecar tax. 500 × 256Mi = 128GB memory tax. At on-demand c6g pricing: ~$1,200/month.

Validation: Track envoy_http_downstream_rq_time histogram. If p99 exceeds 5ms at target RPS, increase CPU limit or enable connection pooling to reduce per-request overhead.

Example 4: API Proxy with TLS Termination

SLO: p99 < 10ms TLS handshake overhead, 200K new connections/sec.

TLS handshake budget: Using ECDSA P-256: each handshake costs ~0.1ms of CPU. 200K × 0.1ms = 20 CPU-seconds/sec = 20 vCPU dedicated to handshakes. At 70% utilization target: 20 / 0.7 = 29 vCPU for TLS.

Total CPU per instance: TLS handshakes + L7 proxying (header parsing, routing). L7 overhead at 200K RPS: ~5 vCPU. Total: 34 vCPU. Using 8-vCPU nodes: ceil(34 / 8) = 5 instances, plus 1 for redundancy = 6 proxy instances.

Session ticket optimization: Enable TLS session tickets with 50MB shared cache (~200K sessions). With 60% returning clients, new handshakes drop to 200K × 0.4 = 80K/sec. CPU drops to 8 vCPU for handshakes — only 3 instances needed (with redundancy).

Memory per instance: TLS session cache: 50MB. Connection buffers: 200K / 6 = 33K connections × 16KB = 528MB. Total: ~600MB. Allocate 2GB RAM per instance.

Validation: Monitor ssl_handshake_time_p99. If > 10ms, check CPU saturation. Consider switching to RSA-free cipher suites or adding instances.

SLO-Driven Capacity Planning

Instead of sizing proxies based on arbitrary rules, start with your SLO and work backwards to derive infrastructure requirements. This approach ensures every resource you provision is justified by a measurable target.

Step 1: Define the Latency Budget

Break your end-to-end latency SLO into a budget per component. For example, if the end-to-end p99 target is 200ms:

  • Network transit: 30ms
  • TLS handshake (amortized): 10ms
  • Proxy overhead (L7 routing, header inspection): 10ms
  • Backend processing: 140ms
  • Serialization/deserialization: 10ms

The proxy gets a 10ms budget. Every sizing decision must keep the proxy contribution under that ceiling at the target percentile.

Step 2: Model Utilization vs. Latency

Proxy latency follows a queuing theory curve: latency stays flat until ~60% CPU utilization, then rises exponentially. The formula latency_multiplier = 1 / (1 - utilization) approximates M/M/1 queue behavior. At 80% utilization, latency is 5x the baseline; at 90%, it is 10x.

Practical rule: Size proxies to run at ≤ 60% CPU utilization at peak traffic to keep latency within your SLO. This means provisioning ~1.7x the CPU you would need at 100% utilization.

Step 3: Derive Instance Count

Use this formula:

instances = ceil(total_cpu_required / (cpu_per_instance × target_utilization)) + redundancy

Where total_cpu_required = TLS handshake CPU + L7 processing CPU + content inspection CPU (if applicable). target_utilization is typically 0.6 for latency-sensitive proxies. redundancy is typically N+1 (add 1 instance) or N+2 for critical paths.

Step 4: Validate with Load Testing

Always load test at 1.2–1.5x your expected peak. Measure the actual p99 latency added by the proxy (difference between client-observed latency and upstream response time). If the proxy exceeds its latency budget under the load test, add instances or increase per-instance CPU until the SLO is met. Document the validated capacity in a runbook so on-call engineers know when to scale.

Scaling Journey

0 – 1K users

No Proxy

Your application server handles everything: TLS (via Let's Encrypt), static files, API requests. One process, one port. No proxy needed. The application framework's built-in server (Flask, Express) is enough.

1K – 100K users

Single Nginx

You add Nginx in front of your app server. It terminates TLS, serves static files, and reverse-proxies API requests. This also enables keepalive connections to the backend. Configuration is a single nginx.conf file.

100K – 10M users

Nginx + Keepalive + Multiple Backends

You scale to multiple backend instances behind Nginx. Upstream keepalive connections prevent TCP handshake storms. You add a second Nginx instance with keepalived (VRRP) for high availability. Response caching at the proxy layer reduces backend load.

10M+ users

Service Mesh with Envoy Sidecars

As you decompose into microservices, each service gets an Envoy sidecar. Istio or Linkerd provides a control plane for mTLS, traffic routing, canary deployments, and per-service observability. An edge proxy (Envoy or a cloud ALB) handles external traffic. eBPF (Cilium) supplements or replaces sidecars for L3/L4 traffic within the cluster.

Migration signal: Move from single Nginx to HA when you can't afford downtime during config reloads. Move to service mesh when the number of inter-service calls makes manual proxy config unmanageable and you need per-service mTLS, retries, and observability without modifying application code.

What Can Go Wrong

Proxy Becomes Bottleneck or SPOF

What happened: A mid-size SaaS company ran a single Nginx instance as their reverse proxy. During a traffic spike from a product launch, Nginx exhausted its worker connections (configured at the default 1024). All new connections were dropped, causing a complete outage for all services behind the proxy.

Root cause: The default worker_connections 1024 was never tuned. Combined with no redundant proxy instance, the proxy was both a bottleneck and a single point of failure.

Mitigation: Increased worker_connections to 65536, increased worker_rlimit_nofile, and deployed a second Nginx instance with keepalived for failover. Added monitoring on nginx_connections_active and nginx_connections_waiting.

Misconfigured Timeouts Causing Cascading Failures

What happened: A reverse proxy was configured with a 60-second proxy_read_timeout, but the backend's request timeout was 30 seconds. When a downstream dependency slowed down, backends started returning 504s after 30s, but the proxy held connections open for the full 60s. This tied up all connection pool slots, causing the proxy to queue requests for healthy backends as well, turning a partial degradation into a full outage.

Root cause: Timeout mismatch between proxy and backend. The proxy should have a timeout slightly shorter than the backend (or equal), not longer. The longer timeout caused connection pool exhaustion.

Prevention: Set proxy timeouts to be ≤ backend timeouts. Implement per-upstream circuit breakers. Monitor connection pool utilization and alert at 80%.

Connection Pool Exhaustion

What happens: A proxy maintains a fixed-size pool of connections to each backend. Under normal conditions, requests flow through quickly and connections are returned to the pool. But when a backend slows down, connections are held longer, the pool drains, and new requests queue up — eventually cascading into a full outage. Here is a step-by-step timeline showing exactly how this unfolds.

t=3s Queue timeout starts firing. The proxy's proxy_connect_timeout is 5s. Requests that have been queuing for over 5s start timing out with 504 Gateway Timeout. The proxy returns errors without the backend ever seeing those requests. 50 / 50 ~1000 504 errors for oldest queued requests. New requests still queuing.
t=5s Cascade. The queue is now over 2,000 deep. Every new request waits 4+ seconds in the queue before even reaching the backend. Effectively, all requests to this backend are timing out. If the proxy has circuit breaking configured, it should trip now and start fast-failing. 50 / 50 2000+ Near-total failure. Most requests return 504.
t=5s+ Collateral damage. If the proxy shares a thread pool or event loop across all upstreams, the overloaded queue for this backend can starve connections to healthy backends. Requests to completely unrelated services start timing out. 50 / 50 Growing Other services affected. Full outage.

The math behind it: Under normal conditions, the pool needs RPS × response_time = 500 × 0.05 = 25 connections. When response time jumps to 2s: 500 × 2.0 = 1,000 connections needed — but only 50 are available. The pool is undersized by 20x, and the deficit accumulates at 500 requests/second.

Prevention:

  • Set a max queue depth (Envoy's max_pending_requests). When the queue exceeds this limit, immediately return 503 instead of queuing. Fast failure is better than slow failure.
  • Use circuit breakers (Envoy's outlier detection, or application-level). When error rate or latency exceeds a threshold, stop sending traffic to the degraded backend entirely.
  • Set aggressive timeouts. The proxy's proxy_read_timeout should be tight enough that slow requests are killed before they occupy connections for too long.
  • Monitor pool utilization. Alert when connection pool usage exceeds 80%. This gives you time to investigate before exhaustion occurs.
  • Isolate upstreams. Use separate connection pools (and ideally separate thread pools) per upstream so a slow backend cannot starve connections to healthy backends. Envoy does this by default; in Nginx, each upstream block has its own pool.

Lost Client IP (X-Forwarded-For Not Propagated)

What happened: After adding a reverse proxy, the application's rate limiter started treating all traffic as coming from a single IP (the proxy's IP). The rate limiter immediately throttled legitimate users while attackers bypassed it by being counted as the same "user."

Root cause: The proxy was not configured to set X-Forwarded-For, and the application was reading the source IP from the TCP connection (which was the proxy's IP) instead of the header. Even when the header was added, the application had to be updated to trust and parse it correctly — including handling multi-hop proxies where the header contains a chain of IPs.

Prevention: Always configure proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for in Nginx. Use the real_ip module to set the client IP from the header, but restrict which proxy IPs are trusted (to prevent header spoofing).

TLS Certificate Expiry

What happened: In 2020, a major company's reverse proxy started rejecting all HTTPS connections when a wildcard certificate expired. The certificate was managed manually, and the renewal reminder email went to a departed employee. The outage lasted two hours while the on-call engineer located the certificate authority credentials.

Prevention: Automate certificate renewal with Let's Encrypt / ACME protocol and certbot. Monitor certificate expiry dates and alert at 30 days, 14 days, and 3 days before expiry. Use ssl_certificate monitoring in your observability stack.

High client latency?

→ Check upstream_response_time. If high, problem is backend, not proxy.

503 Service Unavailable?

→ All backends failed health checks. Verify health check endpoint and thresholds.
→ Check if max_fails and fail_timeout are too aggressive.

Health Check Mechanics

The proxy needs to know which backends are healthy and which are down. Sending traffic to a crashed backend wastes time and returns errors. There are two complementary approaches: active health checks (the proxy probes backends) and passive health checks (the proxy observes real traffic).

Active Health Checks

The proxy periodically sends a probe request (typically GET /health) to each backend. Based on the response, it marks the backend as healthy or unhealthy.

Typical configuration:

  • Interval: Send a probe every 5 seconds.
  • Unhealthy threshold: After 3 consecutive failures (probe returns non-2xx or times out), mark the backend as unhealthy and stop sending traffic to it.
  • Healthy threshold: After 1 success (probe returns 2xx), mark the backend as healthy again and resume sending traffic.
  • Timeout: Each probe has a 2-second timeout. If the backend does not respond within 2 seconds, the probe counts as a failure.

Detection time: With a 5s interval and a threshold of 3 failures, the worst case to detect a crashed backend is 3 × 5s = 15 seconds. During those 15 seconds, real client requests routed to the failing backend will receive errors. To reduce this window, decrease the interval (e.g., 2s) or the threshold (e.g., 2 failures), but be careful — too aggressive and you will flap backends in and out of rotation during minor hiccups.

Passive Health Checks

Instead of sending separate probes, the proxy monitors real traffic responses. If a backend starts returning errors on actual client requests, the proxy ejects it. This is Envoy's outlier detection approach.

Typical configuration (Envoy):

  • Track 5xx response codes from each backend over a 30-second window.
  • If a backend returns 5 or more consecutive 5xx responses, eject it from the load balancing pool.
  • The ejected backend is automatically re-included after 30 seconds.
  • Maximum ejection percentage: Never eject more than 50% of backends (to prevent total loss of capacity during widespread issues).

Advantage over active checks: Passive checks detect problems immediately on real traffic — no waiting for the next probe interval. A backend that starts returning 500 errors is ejected within seconds, not 15 seconds.

Disadvantage: A backend that crashes completely (no response at all) may not be detected as quickly by passive checks, because the proxy must first attempt to route traffic to it and observe the failure. This is why most production setups use both active and passive checks together.

Health Checks During a Rolling Deploy

A rolling deploy replaces backend instances one at a time with a new version. This is where health check timing becomes critical. Here is a scenario with 3 backends (A, B, C) and a rolling deploy of version 2:

Key points:

  • Readiness vs. liveness: In Kubernetes, a readiness probe controls whether traffic is sent to the pod (the proxy respects this). A liveness probe controls whether the pod is restarted. During a rolling deploy, the readiness probe ensures the new pod is fully warmed up before receiving traffic.
  • Graceful shutdown: When a backend receives SIGTERM, it should stop accepting new connections, finish in-flight requests (drain), then exit. The proxy's health check detects the backend is gone and removes it from the pool. Without graceful shutdown, in-flight requests are dropped.
  • Deploy speed vs. safety: Aggressive health check intervals (1s) allow faster deploys but risk flapping. Conservative intervals (10s) are safer but slow down deploy velocity. A common middle ground: 5s interval, 2-failure threshold, 1-success recovery.

Interactions & Pitfalls

Proxy + Load Balancing

In practice, a reverse proxy is a load balancer. Nginx, HAProxy, and Envoy all perform both functions. The distinction is conceptual: "proxy" emphasizes the intermediary role (TLS, header manipulation), while "load balancer" emphasizes traffic distribution. The pitfall is assuming they are separate components when they are usually the same process. Adding a separate load balancer in front of a reverse proxy creates an unnecessary extra hop and latency.

Proxy + Caching

A reverse proxy can cache responses (Nginx's proxy_cache, Varnish in front of Nginx). The pitfall: cached responses must respect Vary headers. If your API returns different content based on Authorization or Accept-Language headers and the proxy does not include these in its cache key, users will see each other's data. Additionally, cache invalidation at the proxy layer is coarse-grained — you typically purge by URL pattern, not by individual resource.

Proxy + Circuit Breakers

Envoy combines proxying with circuit breaking (outlier detection). If you use a separate proxy (Nginx) and a separate circuit breaker (in application code), they must coordinate. Otherwise, the proxy continues sending traffic to a backend that the application-level circuit breaker has already marked as unhealthy, wasting connections and causing errors until the proxy's own health check catches up.

Proxy + Observability

The proxy is the ideal place to inject tracing headers (X-Request-ID, traceparent) because it sees every request. However, the proxy only has L7 visibility if it terminates the connection. An L4 proxy (TCP passthrough) cannot inject headers or log HTTP status codes. If you need observability, you need L7 proxying, which costs more CPU.

Cost Considerations

Cost Analysis

Dominant cost driver: CPU for TLS termination and memory for connection pools/buffers.

Small scale (10K RPS): A single 2-vCPU, 4GB instance running Nginx costs ~$50–$100/month on AWS (c6g.large). This handles TLS termination, keepalive pooling, and basic caching comfortably.

Medium scale (100K RPS): Two c6g.xlarge instances (4 vCPU, 8GB each) behind an NLB: ~$300–$400/month for instances plus ~$50/month for NLB. Adding TLS session caching reduces handshake CPU by 5x.

Service mesh overhead (Envoy sidecars): Each Envoy sidecar consumes ~10–20MB of memory and ~0.1 vCPU at idle. For a cluster with 500 pods, that is 5–10GB of memory and 50 vCPU of overhead. At on-demand c6g pricing, this adds ~$300–$600/month in sidecar tax. Cilium's eBPF approach eliminates most of this overhead for L3/L4 traffic.

Cost of NOT having a proxy: Without TLS termination at a proxy, every backend needs its own certificate and TLS stack, multiplying CPU cost. Without connection pooling, backends waste resources on TCP handshakes. Without caching, every request hits the backend — increasing your compute bill linearly with traffic.

Optimization tips: Use ECDSA certificates (10x cheaper handshakes vs RSA). Enable TLS session tickets. Use ARM instances (Graviton) for better price-performance on TLS workloads. Compress responses at the proxy to reduce bandwidth costs.

Real-World Examples

Nginx at Dropbox

Dropbox runs a customized build of Nginx as their edge proxy, handling millions of concurrent connections. They extended Nginx with custom modules for traffic routing, A/B testing, and rate limiting. Their edge proxy tier terminates TLS, compresses responses, and routes traffic to hundreds of backend services. They moved from a monolithic application behind Nginx to a microservices architecture while keeping Nginx as the single entry point, proving that a well-configured proxy can outlast the backend architecture it was originally built for.

Envoy at Lyft

Lyft created Envoy specifically because Nginx and HAProxy did not meet their needs for a service mesh proxy. They needed dynamic configuration (services scale up and down on Kubernetes), first-class gRPC support, and detailed per-service metrics without instrumenting every application. Envoy now runs as a sidecar on every Lyft service (hundreds of services, thousands of instances), providing mTLS, automatic retries, circuit breaking, and distributed tracing. Lyft credits Envoy with dramatically reducing the time to debug production issues because every service mesh call has detailed metrics and traces.

Cloudflare's Edge Proxy

Cloudflare operates a global network of reverse proxies across 300+ data centers. Every HTTP request to a Cloudflare-protected site hits the nearest edge proxy, which terminates TLS, applies WAF rules, checks the cache, and either serves the cached response or forwards to the origin server. Their proxy handles over 40 million HTTP requests per second globally. They built their own proxy software, initially based on Nginx and later replaced with Pingora (a Rust-based proxy framework they open-sourced in 2022), motivated by Nginx's limitations in connection reuse across requests from different clients to the same origin.

Simple HTTP Reverse Proxy in Python

"""Minimal reverse proxy in Python (~35 lines).
Demonstrates request forwarding, header manipulation,
and connection reuse with a session pool."""

import http.server
import requests

BACKENDS = ["http://localhost:8081", "http://localhost:8082"]
session = requests.Session()  # Connection pooling
counter = 0


class ReverseProxy(http.server.BaseHTTPRequestHandler):
    def do_GET(self):
        global counter
        # Round-robin backend selection
        backend = BACKENDS[counter % len(BACKENDS)]
        counter += 1
        target_url = backend + self.path

        # Forward request with original headers + X-Forwarded-For
        headers = dict(self.headers)
        client_ip = self.client_address[0]
        headers["X-Forwarded-For"] = headers.get(
            "X-Forwarded-For", ""
        )
        if headers["X-Forwarded-For"]:
            headers["X-Forwarded-For"] += f", {client_ip}"
        else:
            headers["X-Forwarded-For"] = client_ip

        resp = session.get(target_url, headers=headers, timeout=10)

        # Send response back to client
        self.send_response(resp.status_code)
        for key, val in resp.headers.items():
            if key.lower() not in ("transfer-encoding", "connection"):
                self.send_header(key, val)
        self.end_headers()
        self.wfile.write(resp.content)

    do_POST = do_GET  # Same forwarding logic for POST


if __name__ == "__main__":
    server = http.server.HTTPServer(("0.0.0.0", 8080), ReverseProxy)
    print("Reverse proxy listening on :8080")
    server.serve_forever()

How to Validate Your Design

Benchmark Proxy Overhead

Use wrk to measure latency and throughput with and without the proxy in the path. Run the benchmark against the backend directly, then through the proxy, and compare p50/p99 latency. The difference is the proxy overhead.

# Direct to backend:
# wrk -t4 -c100 -d30s http://backend:8080/api/health
# Through proxy:
# wrk -t4 -c100 -d30s https://proxy.example.com/api/health
# Compare: latency difference = proxy overhead (TLS + routing)

Key variables to sweep: connection count (10, 100, 1000, 10000), request size (small JSON vs 1MB payload), TLS vs plaintext. Watch for the inflection point where the proxy becomes CPU-bound on TLS.

Test Failover

Use toxiproxy or Chaos Monkey to kill a backend instance while running traffic. Verify that the proxy detects the failure (via health checks), removes the backend from the pool, and redistributes traffic to healthy backends without client-visible errors. Measure how long the failover takes (health check interval + max_fails threshold).

Test Header Propagation

Send a request with a known X-Forwarded-For header and verify that the backend receives both the original value and the proxy-added client IP. Test with multiple proxy hops to ensure the chain is preserved, not overwritten. Verify that X-Request-ID is generated by the proxy and propagated end-to-end.

Chaos Testing Scenarios

  • Kill the proxy process: Does the standby proxy take over via keepalived/VRRP? How fast?
  • Saturate proxy CPU: Send a flood of new TLS connections. Does the proxy degrade gracefully or crash?
  • Expire the TLS certificate: Does monitoring catch it before clients see errors? Does auto-renewal work?
  • Introduce network latency: Add 500ms latency between proxy and one backend. Does the proxy's timeout correctly fast-fail and route to other backends?

Interview Questions

What is the difference between a forward proxy and a reverse proxy?

A forward proxy sits on the client side and acts on behalf of clients. Clients are aware of the proxy and configure their applications to route traffic through it. The destination server does not know a proxy is involved — it sees the proxy's IP. Use cases: corporate firewalls, content filtering, anonymization, caching outbound requests.

A reverse proxy sits on the server side and acts on behalf of servers. Clients are unaware of the proxy — they think they are talking directly to the server. The proxy intercepts inbound traffic and routes it to the appropriate backend. Use cases: TLS termination, load balancing, caching, request routing, DDoS protection.

The key conceptual difference is who the proxy is hiding: a forward proxy hides the client's identity from the server; a reverse proxy hides the server's identity (and architecture) from the client.

Follow-up: Can you have both? Yes — a corporate user behind a forward proxy accessing a website protected by a reverse proxy. The request traverses both proxies. How does this affect X-Forwarded-For?

How does a service mesh work, and why would you use one?

A service mesh is an infrastructure layer that handles service-to-service communication in a microservices architecture. It consists of two parts: a data plane (sidecar proxies like Envoy deployed alongside every service instance) and a control plane (like Istiod) that configures the sidecars.

When Service A calls Service B, the request goes through A's sidecar, which handles mTLS encryption, retries, timeouts, and load balancing. It arrives at B's sidecar, which decrypts and forwards to B. Both sidecars emit metrics and traces. The application code just makes a plain HTTP call to a local address.

You use a service mesh when you have many services (20+) and want consistent cross-cutting concerns (mTLS, retries, circuit breaking, observability) without embedding networking logic in every application. The tradeoff is operational complexity and resource overhead (~10-20MB memory per sidecar, added latency per hop).

Follow-up: What are the alternatives to the sidecar pattern? eBPF-based proxying (Cilium) and proxyless service mesh (gRPC's xDS support, where the application itself speaks to the control plane without a sidecar).

Design a reverse proxy for a microservices platform serving 100K RPS.

Architecture: Deploy two or more Envoy proxy instances behind a cloud Network Load Balancer (NLB) for HA. The NLB handles L4 distribution; Envoy handles L7 processing (TLS, routing, header manipulation).

TLS: Terminate TLS at Envoy using ECDSA certificates for lower CPU cost. Enable TLS session tickets for returning clients. For 100K RPS with 80% session reuse, expect ~20K new handshakes/second = ~2 CPU cores for TLS.

Routing: Use Envoy's xDS API driven by a control plane (Istio or custom) for dynamic route configuration. Route based on path prefix (/users to user-service, /orders to order-service). Within each upstream, use least-connections load balancing with health checks every 5 seconds.

Resilience: Configure per-upstream circuit breakers in Envoy (max connections, max pending requests, max retries). Set retry budgets to prevent retry storms (max 20% of traffic is retries). Add rate limiting at the edge to protect backends from traffic spikes.

Follow-up: How do you handle canary deployments with this setup? Use Envoy's traffic splitting: route 5% of requests to the new version's upstream cluster based on a header or weight, gradually increasing to 100% as confidence grows.

What is TLS termination and why do it at the proxy?

TLS termination means the proxy handles the TLS handshake and decryption/encryption of traffic. The client connects to the proxy over HTTPS; the proxy connects to backends over plain HTTP (on a private network). The TLS connection "terminates" at the proxy.

Why at the proxy? First, centralized certificate management: you manage certificates in one place instead of deploying them to every backend. Second, CPU offloading: TLS handshakes are expensive (especially RSA), and concentrating this on a few proxy instances with optimized crypto libraries is more efficient than every application server doing its own TLS. Third, simplicity: backend applications handle plain HTTP, which simplifies development and debugging.

When NOT to terminate at the proxy: When regulatory compliance (PCI-DSS, HIPAA in certain configurations) requires end-to-end encryption. In that case, use TLS passthrough (the proxy routes encrypted traffic without decrypting) or TLS re-encryption (the proxy decrypts, inspects, and re-encrypts before forwarding).

Follow-up: How does TLS termination interact with HTTP/2? The proxy can accept HTTP/2 from clients (multiplexing) while speaking HTTP/1.1 to backends. This is common because many backends do not support HTTP/2, and the proxy handles the protocol translation transparently.

How do you prevent a proxy from becoming a single point of failure?

Active-passive with VRRP: Run two proxy instances. Use keepalived with VRRP (Virtual Router Redundancy Protocol) to assign a virtual IP (VIP). The primary handles traffic; the standby monitors heartbeats. If the primary fails, the standby claims the VIP within seconds. This is the classic on-premises approach.

Active-active with a cloud LB: Place multiple proxy instances behind a cloud NLB (which itself is managed and redundant). The NLB distributes connections across proxy instances. If one fails the NLB health check, it is removed from the pool. This is the standard approach in AWS/GCP/Azure.

Anycast DNS: For global distribution, use Anycast routing so that clients are directed to the nearest proxy instance. Cloudflare and Google use this approach — the same IP address is advertised from multiple locations, and BGP routing sends clients to the nearest one.

In all cases, you need: health checks (both at the load-balancer level and proxy-to-backend level), graceful connection draining during failover, and shared state for TLS sessions (so a session started on one proxy can resume on another).

Follow-up: What happens to in-flight requests when a proxy fails? They are lost. Clients retry (if the client supports it). Connection draining during planned failovers allows in-flight requests to complete before redirecting traffic.

Compare Nginx, HAProxy, and Envoy for a new project. When would you choose each?

Nginx: Choose when you need a proven, lightweight reverse proxy with excellent static file serving and simple configuration. Best for: monolithic or few-service architectures, static sites with API proxying, when your team already knows Nginx. Limitation: static configuration requires reloads; no built-in dynamic service discovery.

HAProxy: Choose when you need best-in-class L4 proxying, advanced health checking (agent checks, custom health endpoints), and extremely high connection throughput. Best for: TCP-level proxying (database connections, non-HTTP protocols), edge load balancing for very high traffic. HAProxy has a runtime API for some dynamic config but is not as dynamic as Envoy.

Envoy: Choose when you have a microservices architecture on Kubernetes, need dynamic configuration (services scaling up/down constantly), want built-in circuit breaking and outlier detection, or need gRPC-native proxying. Best for: service mesh sidecar, API gateway for microservices, environments where observability is a priority. Limitation: higher memory footprint than Nginx, steeper learning curve, more complex configuration format (YAML/xDS vs. Nginx's conf syntax).

Follow-up: What about Caddy or Traefik? Caddy excels at automatic TLS (built-in ACME/Let's Encrypt). Traefik has native Kubernetes Ingress support and automatic service discovery. Both are good for simpler setups but lack Envoy's advanced traffic management features.

Key Takeaways

  • Forward proxy = client side (hide client), reverse proxy = server side (hide server), sidecar proxy = per-service (service mesh).
  • Reverse proxies provide four core capabilities: TLS termination, connection pooling, request routing, and compression. Most production systems need all four.
  • A proxy without redundancy is a SPOF. Always run at least two instances with failover (VRRP, cloud NLB, or anycast).
  • Timeout alignment is critical. Proxy timeouts must be ≤ backend timeouts, or a slow backend can exhaust the proxy's connection pool and cascade into a full outage.
  • TLS termination is CPU-intensive. Use ECDSA (10x cheaper than RSA), enable session tickets, and size proxy CPU accordingly.
  • Envoy is the modern default for microservices because of its dynamic xDS configuration, first-class observability, and built-in traffic management (retries, circuit breaking, outlier detection).
  • Service meshes trade resources for consistency. The ~10–20MB per-sidecar overhead buys you mTLS, retries, and telemetry across every service without code changes.
  • eBPF (Cilium) is the next frontier, bypassing userspace sidecars for L3/L4 traffic and eliminating much of the sidecar resource tax.
  • Always configure X-Forwarded-For, X-Forwarded-Proto, and X-Request-ID at the proxy. Losing client IPs breaks rate limiting, logging, and compliance.
  • Monitor proxy-added latency, connection pool utilization, TLS handshake time, and 5xx error rate. These four metrics surface 90% of proxy-related issues.
