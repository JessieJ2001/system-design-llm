DNS — Domain Name System

The internet's phone book. DNS translates human-readable domain names into IP addresses, making the entire web navigable without memorizing numbers.

22 min read

What Is It?

The Domain Name System (DNS) is a hierarchical, distributed naming system that maps human-friendly domain names like example.com to machine-readable IP addresses like 93.184.216.34. Without DNS, you would need to remember the IP address of every website you visit. DNS is one of the oldest and most critical pieces of internet infrastructure, and it operates almost entirely invisibly — until it breaks.

DNS is not a single database. It is a globally distributed system organized in a tree structure: root nameservers sit at the top, followed by Top-Level Domain (TLD) servers (.com, .org, .io), and then authoritative nameservers that know the actual IP addresses for specific domains. Each level delegates to the next, and the results are aggressively cached at every layer — your browser, your operating system, your ISP's resolver, and intermediate DNS servers all maintain caches to avoid repeating the full resolution chain for every request.

This caching and delegation design means DNS is remarkably resilient and scalable. Billions of queries per day are handled without centralized coordination. But it also means that changes propagate slowly (governed by TTL values), and a failure at the wrong level — like a DDoS on a major DNS provider — can take down vast portions of the internet.

Animated Diagram: Recursive DNS Resolution

A query for www.example.com travels from the browser through the recursive resolution chain: browser cache, OS cache, recursive resolver, root nameserver, TLD nameserver, and authoritative nameserver. The IP address is then returned back through the chain.

How It Works

Step-by-Step Resolution

  1. User types www.example.com into their browser.
  2. Browser cache check — The browser checks its own DNS cache. If the domain was resolved recently and the TTL hasn't expired, the cached IP is returned immediately (under 1 ms).
  3. OS cache check — If the browser cache misses, the OS resolver (e.g., systemd-resolved, macOS resolver) checks its cache. The OS also checks /etc/hosts for manual overrides.
  4. Recursive resolver query — If the OS cache misses, the query goes to a recursive resolver (typically your ISP's DNS server or a public resolver like 8.8.8.8 or 1.1.1.1). The resolver may have the answer cached from a previous query by another user.
  5. Root nameserver — If the resolver doesn't have it cached, it queries one of the 13 root nameserver clusters (labeled a.root-servers.net through m.root-servers.net). The root server doesn't know the IP for www.example.com, but it knows which TLD servers handle .com and returns a referral.
  6. TLD nameserver — The resolver follows the referral and queries a .com TLD server. The TLD server doesn't know the exact IP either, but it knows which authoritative nameservers are responsible for example.com and returns another referral.
  7. Authoritative nameserver — The resolver queries the authoritative nameserver for example.com. This server has the actual DNS records and returns the A record: 93.184.216.34.
  8. Response propagation — The IP address travels back through the chain. The recursive resolver caches it (respecting the TTL), the OS caches it, the browser caches it, and finally the browser connects to 93.184.216.34.

Notice the cost: a cold lookup requires three network round-trips (root, TLD, authoritative) totaling ~75 ms. But once cached, the same query resolves in under 1 ms. This 75x speedup is why DNS caching is so aggressive — and why TTL matters so much. The recursive resolver at 8.8.8.8 will serve this cached answer to any user who asks for the next 300 seconds, meaning the root and TLD servers see very little repeat traffic.

DNS Record Types: Deep Dive

The table above lists the record types, but understanding why each exists — what problem it solves and how it works mechanically — is essential for designing systems that rely on DNS.

A Record (Address)

The A record is the most fundamental DNS record. It exists because the entire purpose of DNS is to translate names into IPv4 addresses. Without A records, the internet would require humans to memorize 32-bit numbers. When you type example.com, the browser ultimately needs an A record to know where to send packets. A single domain can have multiple A records (returning several IPs), which is the simplest form of DNS-based load balancing — the resolver rotates through them in round-robin order.

AAAA Record (IPv6 Address)

The AAAA record exists because IPv4's 4.3 billion addresses are exhausted. IPv6 uses 128-bit addresses (vs. IPv4's 32 bits), providing 3.4 × 1038 addresses. The name "AAAA" (quad-A) reflects that an IPv6 address is 4× the size of an IPv4 address. Modern clients (browsers, OS resolvers) query for both A and AAAA records simultaneously and prefer IPv6 when available (a behavior called Happy Eyeballs, RFC 8305). If your infrastructure supports IPv6, always publish AAAA records alongside A records.

CNAME Record (Canonical Name)

A CNAME record creates an alias from one domain name to another. It exists because IP addresses change, but domain names are stable contracts. Instead of hardcoding IPs, you point your subdomain to a provider's domain and let them manage IP changes. For example, www.example.com CNAME example.com means "to resolve www.example.com, first resolve example.com and use that answer." The resolver follows the CNAME chain automatically. The critical restriction: a CNAME cannot coexist with any other record type at the same name, and it cannot be placed at the zone apex (the bare domain like example.com) because the apex must have SOA and NS records.

MX Record (Mail Exchange)

MX records exist because email delivery needs to be separate from web hosting. When someone sends an email to user@example.com, their mail server doesn't connect to example.com's web server — it looks up the MX records to find the mail servers responsible for that domain. Each MX record includes a priority value (lower number = higher priority). The sending mail server tries the highest-priority (lowest number) server first, and falls back to lower-priority servers if the primary is unavailable.

Concrete example — mail delivery for example.com:

MX RecordPriorityServerRole
example.com MX10mail1.example.com (203.0.113.10)Primary mail server
example.com MX20mail2.example.com (203.0.113.20)Backup mail server

When alice@gmail.com sends an email to bob@example.com, Gmail's mail server:

  1. Queries example.com MX records and gets both entries.
  2. Sorts by priority: mail1.example.com (priority 10) comes first.
  3. Resolves mail1.example.com to its A record (203.0.113.10).
  4. Opens an SMTP connection to 203.0.113.10 on port 25 and delivers the email.
  5. If mail1 is down: The connection times out (typically after 30 seconds). Gmail then tries mail2.example.com (priority 20) at 203.0.113.20.
  6. If both are down: Gmail queues the email and retries periodically for up to 5 days (per SMTP spec) before sending a bounce-back.

NS Record (Nameserver)

NS records exist to implement DNS's delegation model. They answer the question: "Who is authoritative for this zone?" When the .com TLD server receives a query for example.com, it doesn't know the answer — but it does have NS records that say example.com NS ns1.example.com and example.com NS ns2.example.com. The resolver then queries those nameservers directly. NS records are what make DNS hierarchical: each level delegates to the next. You should always have at least 2 NS records (for redundancy), ideally on different networks.

TXT Record (Text)

TXT records store arbitrary text strings associated with a domain. They were originally intended for human-readable notes, but today they serve critical machine-readable functions: email authentication (SPF, DKIM, DMARC), domain ownership verification (Google Search Console, Let's Encrypt), and security policies.

SPF (Sender Policy Framework) example:

# Query the TXT record
dig example.com TXT

# Response:
example.com.  3600  IN  TXT  "v=spf1 include:_spf.google.com ~all"

This SPF record tells receiving mail servers: "Emails claiming to come from @example.com are legitimate only if they originate from Google's mail servers." Here's how each part works:

  • v=spf1 — Version identifier. This is an SPF record (version 1).
  • include:_spf.google.com — "Also check the SPF record at _spf.google.com." This resolves to a list of Google's mail server IP ranges (e.g., 209.85.128.0/17). If the sending server's IP matches any of these ranges, the email passes SPF.
  • ~all — "Soft-fail everything else." If the sending IP doesn't match any authorized source, mark the email as suspicious (but don't outright reject it). Using -all (hard fail) would cause non-matching emails to be rejected entirely.

Validation flow: When Gmail receives an email claiming to be from bob@example.com, it queries example.com TXT, finds the SPF record, checks whether the sending server's IP is in Google's authorized ranges. If yes, SPF passes. If no, the email is flagged as potentially spoofed. This is one of the primary defenses against email phishing.

SRV Record (Service Locator)

SRV records exist because some services need more than just an IP address — they need a port number, priority, and weight. A and AAAA records only provide IPs; SRV records provide full service discovery information. The format is: _service._proto.name TTL IN SRV priority weight port target.

For example, a SIP (Voice over IP) service:

_sip._tcp.example.com. 3600 IN SRV 10 60 5060 sip1.example.com.
_sip._tcp.example.com. 3600 IN SRV 10 40 5060 sip2.example.com.
_sip._tcp.example.com. 3600 IN SRV 20  0 5060 sip-backup.example.com.

A SIP client resolving this would: (1) try the priority-10 servers first, distributing 60% of connections to sip1 and 40% to sip2 (based on weights); (2) if both priority-10 servers fail, fall back to the priority-20 backup server. SRV records are heavily used in Kubernetes (CoreDNS), XMPP (chat), SIP (telephony), and LDAP (directory services).

TTL and Caching

Every DNS record includes a Time-To-Live (TTL) value, measured in seconds. This tells each caching layer how long it can serve the cached answer before it must re-query. When you set A example.com 3600 93.184.216.34, you're saying "this answer is valid for 1 hour." After 3600 seconds, any resolver holding this record must discard it and fetch a fresh copy.

Caching happens at every layer: the browser (typically 60s), the OS resolver, the recursive resolver (respects the record's TTL), and even some application libraries. This layered caching is what makes DNS fast — most queries never leave the local machine. But it also means that when you change a DNS record, the old answer can linger in caches around the world for up to the previous TTL duration.

TTL Mechanics: A Concrete Timeline

To understand TTL precisely, let's trace what happens when example.com has an A record with TTL=300 seconds (5 minutes). A client first queries at T=0:

Time Event Recursive Resolver Cache OS Cache Browser Cache Result
T=0s First client queries example.com MISS → full resolution (root → TLD → auth). Stores: A=1.2.3.4, TTL=300s Stores: A=1.2.3.4, TTL=300s Stores: A=1.2.3.4, capped at 60s (Chrome default max) 1.2.3.4 (~76 ms)
T=10s Same client queries again Not queried (browser cache hit) Not queried (browser cache hit) HIT (50s remaining in browser cache) 1.2.3.4 (< 1 ms)
T=65s Same client queries again Not queried (OS cache hit) HIT (235s remaining). Re-populates browser cache with remaining TTL EXPIRED (was capped at 60s). Re-filled from OS cache 1.2.3.4 (< 1 ms)
T=130s Same client queries (browser cache expired again at T=125s) Not queried (OS cache hit) HIT (170s remaining) EXPIRED → refilled from OS cache 1.2.3.4 (< 1 ms)
T=299s Same client queries HIT (1s remaining) HIT (1s remaining) Refilled from OS cache 1.2.3.4 (< 1 ms)
T=301s Same client queries EXPIRED → full re-resolution from authoritative. Gets fresh answer (maybe new IP) EXPIRED → forwards to recursive resolver EXPIRED → forwards to OS New IP or same (~20 ms, TLD/root may still be cached)

Key insight — layered caching: The browser, OS, and recursive resolver each maintain independent caches with different behaviors:

  • Browser cache (Chrome, Firefox): Typically caps TTL at 60 seconds regardless of the record's actual TTL. This means even if you set TTL=3600, Chrome will re-query the OS resolver every 60 seconds.
  • OS cache (systemd-resolved, macOS resolver): Respects the record's TTL faithfully. On Linux with systemd-resolved, cached entries are visible via resolvectl statistics.
  • Recursive resolver (8.8.8.8, 1.1.1.1, ISP resolvers): Respects TTL but serves cached answers to all users, not just one client. A popular domain might never cause a cache miss because millions of users keep refreshing the cache entry before it expires.

This layered behavior explains why DNS changes don't propagate instantly: even after the authoritative server returns a new IP, some resolvers still have the old IP cached. The worst-case propagation time equals the TTL of the old record.

When to Use / When NOT to Use

Use DNS For

  • Simple service discovery — Map service names to IPs. Works well for small-to-medium deployments where services don't change IPs frequently.
  • Global load balancing (geo-DNS) — Route users to the nearest data center based on the resolver's location. AWS Route53, Cloudflare, and NS1 all offer geo-DNS.
  • Failover with health checks — Managed DNS providers can health-check endpoints and remove unhealthy IPs from responses automatically.
  • Domain-based routing — Direct api.example.com to your API servers and static.example.com to your CDN, all through DNS records.
  • Email delivery — MX, SPF (TXT), DKIM, and DMARC records are DNS-based and essential for email authentication.

Don't Use DNS When

  • You need sub-second failover — DNS TTLs prevent instant switching. Even with a 30s TTL, clients and resolvers may cache longer. Use a load balancer or Anycast IP instead.
  • You need per-request routing decisions — DNS resolves once and caches. It cannot make decisions per HTTP request (e.g., route based on headers, cookies, or request body). Use an L7 load balancer or API gateway.
  • You need session affinity — DNS round-robin has no concept of sticky sessions. Clients may get a different IP on each resolution. Use a load balancer with session persistence.
  • You need real-time service mesh — In Kubernetes or microservice environments, DNS-based service discovery (even with short TTLs) is too slow. Use a service mesh like Istio or a service registry like Consul.

Classic vs. Modern Approaches

Classic: Standard Recursive Resolution

The original DNS protocol (RFC 1035, 1987) uses unencrypted UDP on port 53. Queries and responses are sent in plaintext, making them visible to anyone on the network path. Recursive resolvers walk the hierarchy (root → TLD → authoritative) and cache results. This design is simple, fast, and has scaled remarkably well for 35+ years, but it has significant privacy and security weaknesses: ISPs can see every domain you visit, and attackers can forge responses (DNS spoofing).

Modern: DNS-over-HTTPS (DoH) and DNS-over-TLS (DoT)

DNS-over-TLS (DoT, RFC 7858) wraps DNS queries in a TLS connection on port 853. DNS-over-HTTPS (DoH, RFC 8484) tunnels DNS queries inside HTTPS requests to a well-known URL (e.g., https://cloudflare-dns.com/dns-query). Both prevent eavesdropping and tampering. DoH is particularly hard to block because it looks like regular HTTPS traffic. Major browsers (Chrome, Firefox) now default to DoH with providers like Cloudflare and Google.

Modern: Anycast DNS

Instead of having nameservers in one location, Anycast DNS advertises the same IP address from many geographic locations using BGP. When a resolver queries the IP, network routing automatically sends the query to the nearest instance. This reduces latency, improves availability (if one site goes down, queries route to the next-nearest), and absorbs DDoS attacks by distributing them globally. Cloudflare's 1.1.1.1 operates from 300+ cities using Anycast.

Modern: EDNS Client Subnet (ECS)

Traditional DNS geo-routing is based on the resolver's location, not the client's. If a user in Tokyo uses Google's 8.8.8.8 resolver (which might resolve from a US datacenter), they could be routed to a US CDN node. EDNS Client Subnet (RFC 7871) fixes this by letting the resolver include a truncated version of the client's IP in the query, so the authoritative server can make geo-routing decisions based on where the actual user is.

Modern: DNSSEC (DNS Security Extensions)

Standard DNS has no way to verify that a response is authentic. When your resolver receives an answer claiming that example.com is at 93.184.216.34, it simply trusts it. An attacker who can inject forged responses (as in the Kaminsky attack) can redirect users to malicious servers. DNSSEC fixes this by adding cryptographic signatures to DNS records, creating a chain of trust from the root zone down to individual records.

The Chain of Trust

DNSSEC works by signing DNS records at every level of the hierarchy. Each zone has two key pairs: a Zone Signing Key (ZSK) that signs individual records, and a Key Signing Key (KSK) that signs the ZSK. The parent zone stores a DS (Delegation Signer) record that is a hash of the child's KSK, linking the two levels together.

Step Zone What Happens Trust Anchor
1 Root zone (.) Root zone is signed with the root KSK (a 2048-bit RSA key, managed by ICANN in a highly publicized key ceremony). The root KSK is the trust anchor — it is hardcoded into every DNSSEC-validating resolver. Root KSK (hardcoded in resolvers)
2 Root → .com Root zone contains a DS record for .com, which is a hash of .com's KSK. Root signs this DS record with its ZSK. Resolver verifies: root's signature is valid (trusted via root KSK) → DS record for .com is authentic. Root KSK → .com DS record
3 .com → example.com .com zone contains a DS record for example.com, signed with .com's ZSK. Resolver verifies: .com ZSK matches the KSK validated via the DS record from step 2. .com KSK → example.com DS record
4 example.com example.com's A record (93.184.216.34) is signed with its ZSK, producing an RRSIG record. Resolver validates: example.com's ZSK matches the KSK validated via the DS record from step 3 → RRSIG on the A record is valid → the A record is authentic. example.com KSK → RRSIG on A record

If any link in this chain is broken (a missing DS record, an invalid signature, an expired key), the DNSSEC-validating resolver returns SERVFAIL instead of a potentially forged answer. The user sees a DNS error rather than being silently redirected to an attacker's server.

Why Adoption Is Low (~30%)

Despite being standardized since 2005, DNSSEC adoption remains limited for several reasons:

  • Complexity: Key management is operationally difficult. Keys must be rotated periodically (ZSK every ~3 months, KSK every ~1-2 years). A botched key rollover can make your domain unresolvable.
  • Increased response size: DNSSEC adds RRSIG, DNSKEY, and DS records to responses, often increasing them from ~100 bytes to ~1000+ bytes. This increases bandwidth and can cause UDP fragmentation, requiring TCP fallback.
  • Amplification attacks: The larger response sizes make DNSSEC-signed domains more effective as amplifiers in DNS reflection DDoS attacks.
  • No confidentiality: DNSSEC provides integrity (responses aren't tampered with) but not privacy (queries are still visible). DoH/DoT solve the privacy problem separately.

Modern: Geo-DNS in Detail

Geo-DNS returns different IP addresses based on where the query originates, routing users to the geographically nearest server. Here's a concrete example of how it works end-to-end:

Concrete Example: Routing cdn.example.com

Step Resolver Location Resolver IP GeoIP Lookup Result Returned A Record PoP Location
Query A Singapore 1.2.3.4 (SingTel resolver) Asia → Southeast Asia → Singapore 203.0.113.1 Singapore PoP
Query B London 5.6.7.8 (BT resolver) Europe → Western Europe → United Kingdom 198.51.100.1 London PoP
Query C São Paulo 200.1.2.3 (Claro resolver) South America → Brazil 192.0.2.50 São Paulo PoP

The authoritative server uses a GeoIP database (such as MaxMind GeoLite2) that maps IP address ranges to geographic locations. When a query arrives, the server looks up the source IP, determines the region, and returns the IP of the nearest PoP (Point of Presence).

The Problem: Public Resolvers and EDNS Client Subnet (ECS)

Geo-DNS works well when users use their ISP's resolver, because the resolver is near the user. But what happens when a user in Singapore uses Google DNS (8.8.8.8)? Google might route the query through a resolver in the US. The authoritative server sees a US IP and returns the US PoP — sending the Singapore user to a server 10,000 miles away.

EDNS Client Subnet (ECS, RFC 7871) fixes this. When Google's resolver forwards the query to the authoritative server, it includes a truncated version of the client's real IP address: 

# What Google's resolver sends to the authoritative server:
;; OPT PSEUDOSECTION:
; EDNS: version: 0, flags:; udp: 512
; CLIENT-SUBNET: 103.252.200.0/24/0
#                ^^^^^^^^^^^^^^^^ Singapore IP prefix (truncated to /24 for privacy)

# The authoritative server now sees:
# - Resolver IP: 8.8.8.8 (Google, could be anywhere)
# - Client subnet: 103.252.200.0/24 (Singapore!)
# Decision: return Singapore PoP IP 203.0.113.1

The /24 prefix means the last octet is zeroed out (103.252.200.0 instead of 103.252.200.47), preserving client privacy while still providing enough geographic precision. The authoritative server's response includes a SCOPE that tells the resolver how specific the geo-routing is (e.g., /24 means "this answer is valid for any client in the 103.252.200.0/24 range"). The resolver caches the answer with this scope, serving it to other clients in the same subnet but re-querying for clients in different subnets.

Modern: DNS-Based Service Discovery (SRV Records)

In Kubernetes, every Service gets a DNS name like my-svc.my-namespace.svc.cluster.local. CoreDNS (the default DNS server in Kubernetes) resolves these names to the ClusterIP or to individual pod IPs (headless services). SRV records add port and priority information, enabling more sophisticated service discovery. However, DNS TTL caching can cause stale endpoints, which is why modern service meshes often bypass DNS entirely for intra-cluster traffic.

Assumptions & Model

Assumptions Behind DNS

  • Clients respect TTL — The model assumes that resolvers and clients will cache DNS responses for exactly the TTL duration, then re-query. In practice, some clients (especially Java's default DNS caching) cache forever or ignore TTL entirely.
  • Resolvers are reliable — DNS assumes recursive resolvers (your ISP or public resolvers) are always available. The 2016 Dyn DDoS attack showed that a single major resolver provider going down can cascade into widespread outages.
  • Propagation is fast — DNS changes are expected to propagate within the TTL window. In reality, propagation can take up to 48 hours due to caches that don't respect TTL, secondary DNS zones that poll infrequently, and registrar-level caching of NS records.
  • Responses are authentic — Standard DNS has no built-in authentication. DNSSEC adds cryptographic signatures, but adoption remains low (~30% of .com domains as of 2024). Without DNSSEC, DNS poisoning attacks can redirect users to malicious servers.

When these break: Clients ignoring TTL cause stale records to persist for hours after a change. Resolver outages (Dyn 2016) take down entire regions. DNS poisoning (Kaminsky attack, 2008) allows attackers to hijack traffic. Long propagation delays during migrations can cause split-brain scenarios where half your users reach the old server and half reach the new one.

Sizing & Tradeoffs

The TTL Tradeoff

TTL is the single most important DNS configuration parameter, and choosing it is a direct tradeoff between query volume (cost and latency) and failover speed (availability).

TTL Failover Time Query Volume Best For
30-60s 30-60 seconds High Active failover, blue-green deployments
300s (5 min) Up to 5 minutes Moderate Good balance for most applications
3600s (1 hour) Up to 1 hour Low Stable services that rarely change IPs
86400s (1 day) Up to 24 hours Minimal Static records (MX, TXT, NS)

Rather than picking TTL or server count from best-practice tables, work backwards from your SLOs using this framework:

Step Question Drives
1. Define availability target How many minutes of DNS downtime per year are acceptable? Number of DNS providers (1 provider → ~99.99%, 2 providers → ~99.9999%); health check aggressiveness
2. Define latency target What is the maximum acceptable p99 resolution time? Anycast PoP count and placement; local cache layer decision; EDNS Client Subnet configuration
3. Define RTO How fast must DNS failover complete after an outage? TTL value (TTL ≤ RTO − detection_time − propagation_time); health check interval
4. Compute query volume Given the TTL from step 3, how many QPS hit authoritative servers? Authoritative server count and sizing; DNS provider plan/tier; cost budget
5. Validate cost Does the resulting QPS fit the budget? If not, relax the RTO or add caching layers to absorb the query volume

Formula: Max TTL = RTO − Detection Time − Propagation Delay − Safety Margin

For example, with an RTO of 60 s, health checks every 5 s (detection = 10 s with 2 failures), and 5 s propagation delay: Max TTL = 60 − 10 − 5 − 15 = 30 s. At 1M users, this yields 1M / 30 = ~33,333 QPS at a Route53 cost of roughly $1,152/day. If that cost is too high, either relax the RTO or deploy a local caching resolver to absorb 95%+ of queries.

Rule of Thumb

Use 300s (5 min) as a default TTL for A records. Drop to 60s before planned migrations or failover events. Use 3600s+ for stable records like MX and NS. Before any DNS change, lower the TTL at least 24-48 hours in advance so old, long-TTL records drain from caches worldwide.

Scaling Journey

0 – 1K Users

Single A Record

One A record pointing to a single server IP. DNS is managed through your registrar's basic DNS panel. TTL set to 3600s. No redundancy, no health checks. If the server dies, you manually update the DNS record and wait for TTL to expire.

1K – 100K Users

Multiple A Records (Round-Robin)

Add multiple A records for the same domain, each pointing to a different server. DNS resolvers rotate through them, distributing load. Move to a managed DNS provider (Route53, Cloudflare). Lower TTL to 300s. No health checking — dead servers still get traffic until records are manually removed.

100K – 10M Users

Geo-DNS + Health Checks

Use a managed DNS provider with geo-routing: users in Europe resolve to EU servers, US users to US servers. Health checks automatically remove unhealthy endpoints from DNS responses. TTL lowered to 60s for faster failover. CNAME to CDN for static assets. Start using ALIAS/ANAME records for zone apex.

10M+ Users

Anycast + Multi-Provider

Use Anycast IP addresses so users automatically route to the nearest PoP via BGP. Run multi-provider DNS (e.g., Route53 + Cloudflare) for redundancy — if one provider goes down, the other answers queries. Implement DNSSEC. Use EDNS Client Subnet for accurate geo-routing through public resolvers.

What Can Go Wrong

Dyn DDoS Attack (October 2016)

What happened: A massive DDoS attack using the Mirai botnet (comprised of ~100,000 IoT devices) targeted Dyn, a major managed DNS provider. The attack generated sustained traffic exceeding 1.2 Tbps against Dyn's Anycast infrastructure.

Blast radius: Dozens of major websites became unreachable, including Twitter, GitHub, Reddit, Netflix, Spotify, and Airbnb. These sites were not attacked directly — their servers were fine — but users couldn't resolve their domain names to reach them.

Root cause: Single DNS provider dependency. All affected companies relied on Dyn as their sole DNS provider. When Dyn went down, there was no fallback.

Prevention: Use multi-provider DNS. Configure NS records to point to at least two independent DNS providers. Route53 + Cloudflare, or Dyn + Google Cloud DNS. The slight increase in complexity and cost is trivial compared to the risk.

DNS Cache Poisoning (Kaminsky Attack, 2008)

What happened: Security researcher Dan Kaminsky discovered that the randomness in DNS query IDs (only 16 bits) was insufficient. An attacker could flood a recursive resolver with forged responses for a domain, and with enough attempts, one would have a matching query ID and be accepted as legitimate.

Blast radius: Every user of the poisoned resolver would be directed to the attacker's server. This could be used for phishing, credential theft, or malware distribution at massive scale.

Prevention: Source port randomization (adding entropy beyond the 16-bit query ID), DNSSEC (cryptographic validation of DNS responses), and DNS-over-HTTPS/TLS (encrypted transport preventing response injection).

TTL Misconfiguration (Stale Records)

What happened: A team migrates their service to a new IP address but forgets that their A record has a 24-hour TTL. They update the DNS record, but resolvers worldwide continue serving the old IP for up to 24 hours. Some users can reach the service; others get connection timeouts.

Root cause: Not lowering the TTL before the migration. The correct procedure is to lower TTL to 60s at least 48 hours before the change (so the old high-TTL records expire from all caches), make the change, verify, then raise the TTL back up.

Detection: Monitor DNS resolution from multiple geographic locations using tools like DNSChecker.org or Catchpoint. Set up synthetic monitoring that resolves your domain and verifies the returned IP.

Interactions & Pitfalls

DNS + CDN

The typical CDN setup uses a CNAME record to point your domain to the CDN's domain: static.example.com CNAME cdn.provider.com. The CDN provider then resolves cdn.provider.com to the nearest edge node. The pitfall: CNAME records are not allowed at the zone apex (example.com itself, without a subdomain) per the DNS spec. This means you can't CNAME your root domain to a CDN. Workarounds include ALIAS/ANAME records (a non-standard extension supported by Route53, Cloudflare, and DNSimple) or flattening the CNAME at the DNS provider level.

DNS + Load Balancing

DNS round-robin (multiple A records) provides basic load distribution, but has significant limitations. The client receives all IPs and typically picks the first one. Some resolvers cache only the first IP. Even when all IPs are returned, there's no weighting, no health awareness, and no feedback loop. The pitfall: If you have 3 servers and one dies, 1/3 of your users get timeouts until the DNS record is updated and the TTL expires. A proper load balancer solves all of these problems — DNS should route to your load balancer's IP, not directly to application servers.

DNS + Failover

DNS-based failover works by monitoring endpoint health and removing unhealthy IPs from DNS responses. The pitfall: Even with a 30-second TTL, actual failover is slower because: (1) many resolvers don't respect low TTLs and cache longer, (2) clients cache at the application level (Java's InetAddress caches for 30s by default, some older versions cache forever), and (3) the health check itself takes time to detect failure. In practice, DNS failover typically takes 1-5 minutes, not the 30 seconds the TTL suggests. For sub-second failover, you need Anycast IP addresses or a load balancer with BGP, not DNS-based switching.

DNS Failover: Step-by-Step Timeline

Let's trace a complete DNS failover scenario. Assume: primary server at 93.184.216.34, backup at 93.184.216.35, health checks every 5 seconds with a 2-failure threshold, and TTL=60s on the A record.

Time Event DNS Provider State Resolver A (cached at T=-30s) Resolver B (cached at T=-5s) New Client (no cache)
T=0s Primary server crashes Health check #1 at T=0s: no response. Count = 1 (need 2 to confirm). Cached: 93.184.216.34 (30s remaining) Cached: 93.184.216.34 (55s remaining) Gets 93.184.216.34 → timeout
T=5s Second health check fails Health check #2: no response. Count = 2 ≥ threshold. Primary marked unhealthy. DNS provider removes 93.184.216.34, now returns only 93.184.216.35. Still cached: 93.184.216.34 (25s remaining) Still cached: 93.184.216.34 (50s remaining) Gets 93.184.216.34 → timeout
T=10s DNS record updated at authoritative Authoritative now serves: example.com A 93.184.216.35 TTL=60. Propagation to all authoritative nodes takes ~1-5s. Still cached: old IP (20s remaining) Still cached: old IP (45s remaining) Gets 93.184.216.35 → success!
T=30s Resolver A's cache expires Serving 93.184.216.35 Cache expired → re-queries → gets 93.184.216.35 → success! Still cached: old IP (25s remaining) Gets 93.184.216.35
T=55s Resolver B's cache expires Serving 93.184.216.35 Serving 93.184.216.35 Cache expired → re-queries → gets 93.184.216.35 → success! Gets 93.184.216.35
T=70s All resolvers updated Serving 93.184.216.35 93.184.216.35 93.184.216.35 93.184.216.35

Failover time budget breakdown:

  • Detection time: health_check_interval × failure_threshold = 5s × 2 = 10 seconds
  • Propagation time: Time to update all authoritative nodes = ~5 seconds
  • Maximum TTL drain: Worst case, a resolver cached the record 1 second before the change = 60 seconds
  • Worst-case total failover time: 10s + 5s + 60s = 75 seconds
  • Best-case total failover time: 10s + 5s + 0s (cache just expired) = 15 seconds

The formula: Worst-case failover = Detection time + Propagation time + Max TTL. With TTL=60s and 5s health checks (threshold 2), worst case is ~75 seconds. To get faster failover, you must lower TTL (increasing query cost) or switch to Anycast/BGP-based failover which operates at the network layer and converges in seconds.

Cost Considerations

DNS Cost Analysis

Hosted DNS pricing (AWS Route53 as reference):

  • $0.50/month per hosted zone (domain)
  • $0.40 per million standard queries (first 1 billion/month)
  • $0.60 per million geo-DNS or latency-based queries
  • Health checks: $0.50/month per endpoint (HTTP/HTTPS)

Typical costs at different scales:

  • Small app (10K users, 300s TTL): ~$1/month total. Essentially free.
  • Medium app (1M users, 300s TTL): ~$50/month. Still very cheap.
  • Large app (100M users, 60s TTL): ~$5,000/month. Significant but tiny relative to compute/storage costs.

Dominant cost driver: Query volume, which is directly proportional to user count and inversely proportional to TTL.

Cost of NOT having proper DNS: Immeasurable. A DNS outage affects 100% of your users. The Dyn 2016 attack cost affected companies millions in lost revenue per hour. Multi-provider DNS costs 2x your DNS bill but provides the most critical layer of redundancy in your stack.

Real-World Examples

Cloudflare 1.1.1.1

Cloudflare operates the world's fastest public DNS resolver (1.1.1.1) from 300+ cities using Anycast. Their architecture is designed for both speed and privacy: they don't log client IP addresses and purge query logs within 24 hours. The 1.1.1.1 resolver supports DNS-over-HTTPS (DoH) and DNS-over-TLS (DoT), and serves as a reference implementation for modern DNS privacy. By placing resolver infrastructure in every major IXP (Internet Exchange Point), Cloudflare achieves median query latency under 10ms globally.

AWS Route53

Route53 is Amazon's managed DNS service, handling authoritative DNS for millions of domains. Its key differentiator is deep integration with AWS services: health-checked failover to S3 static sites, weighted routing for canary deployments, latency-based routing across AWS regions, and ALIAS records (Amazon's solution to the CNAME-at-apex problem). Route53 uses a global Anycast network of nameservers and provides a 100% availability SLA — one of the few AWS services with this guarantee.

Dyn Outage 2016: Case Study

The October 21, 2016 Dyn attack remains the most significant DNS incident in internet history. The Mirai botnet, consisting of ~100,000 compromised IoT devices (cameras, DVRs, routers), launched three waves of DDoS attacks over the course of a day. The first wave hit the US East Coast, taking down major sites for about 2 hours. Dyn mitigated it, but two more waves followed. The key lesson: DNS is a single point of failure that most teams never think about until it's too late. Post-incident, many companies adopted multi-provider DNS strategies, and the industry accelerated the development of Anycast DNS and DDoS mitigation specifically for DNS infrastructure.

Code Example

A simple Python DNS resolver simulation showing recursive resolution with caching:

import time

class DNSResolver:
    """Simulates recursive DNS resolution with TTL-based caching."""

    def __init__(self):
        self.cache = {}  # domain -> (ip, expire_time)
        # Simulated authoritative records
        self.authority = {
            "example.com": ("93.184.216.34", 300),
            "api.example.com": ("93.184.216.35", 60),
            "cdn.example.com": ("104.16.132.229", 3600),
        }

    def resolve(self, domain):
        # Step 1: Check cache
        if domain in self.cache:
            ip, expires = self.cache[domain]
            if time.time() < expires:
                print(f"  Cache HIT: {domain} -> {ip}")
                return ip
            else:
                print(f"  Cache EXPIRED for {domain}")
                del self.cache[domain]

        # Step 2: Recursive resolution (simulated)
        print(f"  Cache MISS: querying root -> TLD -> auth for {domain}")
        if domain in self.authority:
            ip, ttl = self.authority[domain]
            # Step 3: Cache the result with TTL
            self.cache[domain] = (ip, time.time() + ttl)
            print(f"  Resolved: {domain} -> {ip} (TTL={ttl}s)")
            return ip

        print(f"  NXDOMAIN: {domain} not found")
        return None

# Usage
resolver = DNSResolver()
print("First lookup (cache miss):")
resolver.resolve("example.com")
print("\nSecond lookup (cache hit):")
resolver.resolve("example.com")
print("\nLookup non-existent domain:")
resolver.resolve("missing.example.com")

How to Validate Your Design

Manual Testing

  • dig example.com — The go-to tool for DNS debugging. Shows the full response including record type, TTL, and which nameserver answered. Use dig +trace to see the complete resolution chain from root to authoritative.
  • nslookup example.com — A simpler alternative to dig. Useful for quick checks but provides less detail. Works on all platforms including Windows.
  • dig @8.8.8.8 example.com — Query a specific resolver to check what answer it returns. Useful for verifying propagation by checking multiple resolvers.

Propagation Testing

  • DNS propagation checkers (whatsmydns.net, dnschecker.org) — Query your domain from 20+ global locations simultaneously to verify that your DNS change has propagated worldwide.
  • Monitor TTL compliance — Use tcpdump port 53 to observe actual DNS queries on your network. Verify that your resolver is re-querying after the TTL expires and not holding stale records longer than expected.

Load and Chaos Testing

  • dnsperf / queryperf — Benchmark your authoritative DNS servers. Generate thousands of queries per second and measure response latency, throughput, and failure rate under load.
  • Chaos testing — Kill one of your nameservers and verify that queries fail over to the remaining ones. Block port 53 to simulate DNS outages and verify your application handles SERVFAIL gracefully (retries, fallback, circuit breaking).
  • TTL compliance testing — Change a DNS record and measure how long it actually takes for clients across different networks to see the new value. Compare this to your TTL setting to identify resolvers that cache too aggressively.

Interview Questions

Walk through what happens when you type google.com into your browser.

This is the classic interview opener, and the depth of your answer reveals your systems knowledge. The DNS portion goes: the browser checks its internal DNS cache first. If it's a miss, it calls the OS resolver (getaddrinfo()), which checks the OS cache and /etc/hosts. If still a miss, the OS sends a query to the configured recursive resolver (your ISP or a public resolver like 8.8.8.8). The resolver checks its cache; on a miss, it queries a root nameserver (one of 13 clusters), which returns a referral to the .com TLD servers. The resolver follows the referral, queries a .com TLD server, and gets a referral to Google's authoritative nameservers (ns1.google.com, etc.). Finally, the resolver queries Google's authoritative server, which returns the A record (e.g., 142.250.80.46). The IP is cached at each layer with the record's TTL.

After DNS resolution, the browser opens a TCP connection (SYN, SYN-ACK, ACK), performs a TLS handshake (for HTTPS), sends an HTTP GET request, and renders the response. The full process typically takes 100-500ms for a cold start, but under 10ms for a warm cache hit.

Follow-up: "What happens if the DNS resolution fails?" — The browser retries with a different resolver, shows an ERR_NAME_NOT_RESOLVED error, and the user sees a "site can't be reached" page. There's no fallback to IP-based access unless the user knows the IP.

How would you design DNS for a global application?

For a global application, you need geo-aware DNS that routes users to the nearest healthy data center. Start with a managed DNS provider that supports geo-DNS or latency-based routing (Route53, Cloudflare, NS1). Configure location-based routing rules: users resolving from European resolvers get IPs for your EU data center, Asian resolvers get IPs for your APAC data center, etc.

Add health checks so that if a data center goes down, its IPs are automatically removed from DNS responses and users fail over to the next-nearest region. Use EDNS Client Subnet (ECS) to ensure accurate geo-routing even when users are going through a distant public resolver like Google's 8.8.8.8. Set TTLs to 60s for A records to enable reasonably fast failover. Run multi-provider DNS (e.g., Route53 + Cloudflare) to protect against DNS provider outages. Finally, implement DNSSEC for integrity and consider Anycast for your own authoritative servers if you're running them yourself.

Follow-up: "What's the tradeoff of multi-provider DNS?" — Complexity in keeping zone files synchronized across providers. You need automation (Terraform, OctoDNS) to ensure both providers have identical records. If they drift, users get different answers depending on which provider handles their query.

How does geo-DNS work?

Geo-DNS works by returning different DNS responses based on the geographic location of the requesting resolver. When a recursive resolver (e.g., your ISP's DNS server in Tokyo) queries the authoritative nameserver, the authoritative server looks up the resolver's IP in a GeoIP database (like MaxMind) and determines its location. Based on pre-configured routing rules, it returns the IP address of the nearest data center.

The challenge is accuracy: the resolver's location is a proxy for the user's location, and sometimes it's wrong. A user in Paris using Google's 8.8.8.8 might have their query resolved by a Google datacenter in another country. EDNS Client Subnet (ECS) mitigates this by including a truncated version of the client's actual IP in the DNS query, giving the authoritative server a more accurate location signal.

Follow-up: "How does Anycast differ from geo-DNS?" — Geo-DNS returns different IPs per region. Anycast advertises the same IP from multiple locations and lets BGP routing handle the nearest-server selection at the network layer. Anycast is more transparent and faster to fail over (BGP reconvergence vs. TTL expiry), but gives you less control over routing decisions.

What happens when a DNS provider goes down?

If your sole DNS provider goes down, your domain becomes unresolvable. Users can't find the IP address for your site, so they get ERR_NAME_NOT_RESOLVED errors. Your servers are perfectly healthy, but nobody can reach them. Cached records will continue to work until their TTL expires, so users who recently visited may still have access. But new users and those with expired caches are completely blocked.

This is exactly what happened during the Dyn attack in 2016. The mitigation is multi-provider DNS: configure your domain's NS records at the registrar to point to nameservers from two independent providers (e.g., ns1.route53.com and ns1.cloudflare.com). Both providers must have identical zone data. Resolvers will query whichever nameserver they can reach, so if one provider is down, the other answers. This requires keeping zone data synchronized across providers, which is best done through automation (Terraform, OctoDNS, or DNS provider APIs).

Follow-up: "How do you keep two DNS providers in sync?" — Infrastructure-as-code is the best approach. Define your DNS records in Terraform or a similar tool and apply to both providers simultaneously. Some providers also support AXFR zone transfers between primary and secondary DNS, but cross-vendor AXFR support is inconsistent.

CNAME vs A record — when to use which?

An A record maps a domain directly to an IP address: example.com → 93.184.216.34. A CNAME record maps a domain to another domain name: www.example.com → example.com. Use an A record when you know the IP and it's stable. Use a CNAME when you want to point to a service whose IP might change (like a CDN or load balancer): static.example.com CNAME d123.cloudfront.net. The CDN can change its IPs without you updating your DNS.

Key restrictions: A CNAME cannot coexist with any other record type for the same name, and it cannot be used at the zone apex (example.com itself). This is because the DNS spec says if a CNAME exists for a name, that name cannot have any other records (and the apex must have SOA and NS records). To point a zone apex to a service, use ALIAS or ANAME records (provider-specific extensions) or a load balancer with a static IP.

Follow-up: "What is CNAME flattening?" — CNAME flattening is a technique where the DNS provider resolves the CNAME chain at query time and returns the resulting A record directly. This allows CNAME-like behavior at the zone apex without violating the DNS spec. Cloudflare and Route53 (via ALIAS records) both support this.

How to achieve fast failover with DNS?

Pure DNS-based failover is inherently limited by TTL caching. To make it as fast as possible: (1) Set TTL to 30-60 seconds. (2) Use a managed DNS provider with active health checks (10-30 second intervals). (3) Configure the provider to automatically remove unhealthy IPs from responses. With this setup, worst-case failover time is approximately TTL + health check detection time, typically 1-3 minutes.

For truly fast failover (sub-second), DNS alone is not sufficient. You need one of: (a) Anycast with BGP — advertise the same IP from multiple locations; when one location goes down, BGP reconverges in seconds and traffic routes to the next-nearest location. (b) A global load balancer (like Google's Cloud Load Balancer or AWS Global Accelerator) that uses Anycast IPs and monitors backend health. (c) Client-side failover — return multiple A records and let the browser try the next IP if the first one fails (Happy Eyeballs / RFC 8305). Most modern browsers do this, attempting the next IP within 250ms.

Follow-up: "What about lowering TTL to 0?" — A TTL of 0 means "do not cache," but many resolvers impose a minimum TTL (often 30s or 60s) regardless. Some CDNs and libraries also have hardcoded minimums. A TTL of 0 dramatically increases query volume to your authoritative servers and may not even achieve the desired caching behavior in practice.

Key Takeaways

  • DNS is hierarchical and distributed. Root servers delegate to TLD servers, which delegate to authoritative servers. This delegation model scales to billions of queries per day without central coordination.
  • Caching is everywhere. Browser, OS, recursive resolver, and CDN all cache DNS responses. This makes DNS fast but means changes propagate slowly, governed by TTL.
  • TTL is your primary control knob. Short TTLs (60s) enable fast failover but increase query volume. Long TTLs (3600s+) reduce load but slow failover. Use 300s as a sensible default and lower it before planned changes.
  • DNS is a single point of failure you often overlook. If your DNS provider goes down, 100% of users are affected regardless of how healthy your servers are. Use multi-provider DNS for any production application.
  • DNS-based failover is minutes, not seconds. Even with low TTLs, real-world DNS failover takes 1-5 minutes due to resolver caching, client caching, and health check detection time. For sub-second failover, use Anycast or a global load balancer.
  • Modern DNS adds privacy and security. DNS-over-HTTPS (DoH) and DNS-over-TLS (DoT) encrypt queries. DNSSEC adds authentication. EDNS Client Subnet improves geo-routing accuracy. These are now mainstream, not edge cases.
  • The CNAME-at-apex problem is real. You cannot use a CNAME at your zone apex. Use ALIAS/ANAME records or CNAME flattening to point root domains at CDNs and load balancers.
