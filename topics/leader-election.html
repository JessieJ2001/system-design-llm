Leader Election

A coordination mechanism that selects a single node as the authoritative leader among a cluster of peers, ensuring consistent decision-making, single-writer semantics, and orderly failover when the leader fails.

~20 min read

What Is It?

Leader election is the process of choosing one node in a distributed cluster to act as the coordinator or leader. The leader takes on special responsibilities that the other nodes (called followers or replicas) do not: it might be the only node that accepts writes, assigns work to other nodes, or coordinates distributed transactions. Every other node in the cluster either forwards requests to the leader or stands by, ready to take over if the leader fails.

Why do we need a single leader? In many distributed systems, having a single writer dramatically simplifies consistency. If two nodes both accept writes to the same data simultaneously, you get conflicting updates that are expensive to resolve. A designated leader serializes all writes, giving them a total order. The leader also serves as a natural coordination point for tasks like assigning partitions in a message broker, scheduling cron jobs, or driving two-phase commit across shards.

The challenge is that leaders can fail. When the leader crashes, becomes unreachable, or is partitioned from the rest of the cluster, the remaining nodes must detect the failure and elect a new leader quickly and safely. "Safely" means at most one node believes it is the leader at any given time — violating this invariant leads to split-brain, one of the most dangerous failure modes in distributed systems. Modern consensus algorithms like Raft solve this by using term numbers, majority quorums, and heartbeat timeouts to guarantee that exactly one leader exists per term.

Problems That Need a Single Leader

Not every distributed system needs a leader, but these four categories of problems almost always do:

  • Write coordination: Only one node accepts writes to avoid conflicts. A database primary is the canonical example. If two nodes both accept a write to the same row at the same time (say, one sets balance = 50 and the other sets balance = 75), which write wins? With a single leader, all writes are serialized — they get a total order, and every replica applies them in the same sequence.
  • Task scheduling: Only one scheduler assigns tasks to prevent duplicates. Imagine three instances of a cron-job scheduler all running independently. When the "send daily report" job triggers at 9:00 AM, all three instances fire it. The result: three identical emails. A single elected scheduler eliminates this duplication.
  • Lock management: One node manages distributed locks. If multiple nodes independently grant locks, two clients could both believe they hold the same lock. A single lock manager serializes all lock requests, ensuring mutual exclusion.
  • Cluster coordination: One node makes authoritative decisions for the cluster. Kafka's controller broker decides which broker leads each partition. Kubernetes' controller manager decides which pods to schedule. These decisions must come from a single authority to avoid contradictory assignments.

What Goes Wrong Without a Leader: Split-Brain

The worst failure mode in a leader-based system is split-brain — two nodes both believe they are the leader simultaneously. Here is how it happens and why it is so dangerous:

  1. A 3-node cluster has Node A as the leader. Node A is processing writes normally.
  2. A network partition separates Node A from Nodes B and C. Node A can still talk to clients on its side, but not to B or C.
  3. Nodes B and C detect that they have not received heartbeats from Node A. They elect Node B as the new leader.
  4. Now both Node A and Node B think they are the leader. Both accept writes from their respective clients.
  5. Node A receives UPDATE account SET balance = 100 WHERE id = 42. Node B receives UPDATE account SET balance = 200 WHERE id = 42.
  6. When the partition heals, the system has two conflicting histories. Account 42 has balance 100 on Node A's side and balance 200 on Node B's side. Which is correct? Neither can be automatically chosen without risking data loss.

This is why every production leader election system uses either majority quorums (so the old leader cannot commit writes without a majority, which it has lost) or fencing tokens (so storage rejects the old leader's writes), or both.

How It Works

Raft Leader Election (the dominant modern approach)

Raft divides time into terms, each identified by a monotonically increasing integer. Each term begins with an election and, if successful, continues with the elected leader serving until it fails or a new term starts.

  1. Heartbeat timeout: The leader periodically sends heartbeat messages (empty AppendEntries RPCs) to all followers. Each follower maintains an election timeout — a randomized timer (typically 150-300ms). If a follower does not receive a heartbeat before its timeout fires, it assumes the leader has failed.
  2. Becoming a candidate: The follower increments its current term number, transitions to candidate state, votes for itself, and sends RequestVote RPCs to all other nodes. The request includes the candidate's term number and the index/term of its last log entry.
  3. Voting rules: Each node can vote for at most one candidate per term. A node grants its vote only if the candidate's log is at least as up-to-date as its own (this ensures the new leader has all committed entries). Voting is first-come-first-served within a term.
  4. Winning the election: If the candidate receives votes from a majority of nodes (including itself), it becomes the new leader for that term. It immediately sends heartbeats to all nodes to establish authority and prevent further elections.
  5. Split vote handling: If no candidate wins a majority (because multiple candidates split the votes), the election times out, and a new election begins with a higher term. The randomized election timeout makes repeated split votes unlikely.

Key insight: Election timeout randomization (150-300ms) makes split votes rare. Each node picks a new random timeout after every failed election. The probability of two nodes picking timeouts close enough to collide again is roughly 1 / (timeout_range / network_RTT). With a 150ms range and 1ms RTT, the chance of a second consecutive split vote is about 0.7%. The probability of k consecutive split votes drops exponentially — roughly (1/150)^k — making it astronomically unlikely after 2-3 rounds.

Bully Algorithm

One of the earliest leader election algorithms (Garcia-Molina, 1982). The rule is simple: the node with the highest ID always wins. Each node has a unique numeric ID. When a node detects the leader has failed, it sends an "election" message to all nodes with higher IDs. If none respond, it declares itself the leader. If a higher-ID node responds, it takes over the election. The highest-ID surviving node always wins — hence "bully."

The key mechanism: each node only sends ELECTION messages to nodes with higher IDs. If a higher-ID node responds with ALIVE, the lower-ID node backs off. This cascading process ensures the highest surviving node always ends up as leader.

Why the Bully Algorithm Is Not Partition-Tolerant

If a network partition separates {Node 1, Node 2} from {Node 3, Node 4}, both sides will independently elect a leader. On the left side, Node 2 sends ELECTION to 3, 4, 5 — no responses (they are unreachable). Node 2 declares itself leader. On the right side, Node 4 sends ELECTION to 5 — no response. Node 4 declares itself leader. Now there are two leaders: Node 2 and Node 4. The bully algorithm has no mechanism to detect or prevent this split-brain — it assumes that "no response" means "crashed," but it could also mean "partitioned."

Ring Algorithm

Nodes are arranged in a logical ring. When a node detects the leader is down, it sends an election message around the ring, collecting node IDs. When the message returns to the initiator, the node with the highest ID (or some other criterion) is declared leader. Like the bully algorithm, it is simple but unsuitable for partition-prone networks because a broken ring halts the election.

Lease-Based Leadership

A leader holds a time-limited lease (e.g., 10 seconds). It must periodically renew the lease before it expires. If the leader fails to renew — because of a crash, a network partition, or a GC pause — the lease expires and another node can claim leadership. This approach is used by Kubernetes (Lease objects), Google's Chubby lock service, and many cloud-native systems.

The danger zone: Between T=24s and T=25s, Node A's GC pause ends and it resumes execution. It has no idea its lease expired. If it performs any writes during this window, it could corrupt data because Node B is now the legitimate leader. Node A must check the lease validity before every action — but checking a local clock after a GC pause is unreliable (the clock jumped forward during the pause, but the code may not re-read it before acting).

Fencing Tokens: The Last Line of Defense

Because a stale leader cannot always detect its own demotion, we need a mechanism at the storage layer to reject stale writes. This is where fencing tokens come in:

  1. Each lease acquisition gets a monotonically increasing token. Node A acquired lease with token 33. Node B acquired lease with token 34.
  2. Every write from the leader includes its fencing token: WRITE key=balance, value=100, token=33.
  3. The storage system tracks the highest token it has seen. After Node B writes with token 34, the storage records max_token = 34.
  4. When stale leader Node A (with token 33) tries to write, the storage sees 33 < 34 and rejects the write.

Fencing tokens make the system safe regardless of what the stale leader does — even if it does not know it is stale. This is fundamentally more robust than relying on the leader to "check its lease" because the validation happens at the resource, not at the potentially confused leader.

ZooKeeper's Approach: Ephemeral Nodes and Watches

Apache ZooKeeper provides a particularly elegant leader election mechanism using two primitives: ephemeral nodes (nodes that automatically disappear when the client's session dies) and sequential nodes (nodes with auto-incrementing sequence numbers).

Step-by-Step Process

  1. Each candidate creates an ephemeral sequential znode under a designated path:
    • Node A creates: /election/candidate-0001
    • Node B creates: /election/candidate-0002
    • Node C creates: /election/candidate-0003
  2. The node with the LOWEST sequence number is the leader. In this case, candidate-0001 (Node A) has the lowest number, so Node A is the leader.
  3. Each non-leader node watches the node with the NEXT LOWER sequence number (not the leader directly):
    • Node C (candidate-0003) watches candidate-0002 (Node B)
    • Node B (candidate-0002) watches candidate-0001 (Node A)
  4. When Node A crashes, its ZooKeeper session times out, and its ephemeral node (candidate-0001) automatically disappears.
  5. Node B's watch fires — it was watching candidate-0001, which just disappeared. Node B checks the children of /election/ and sees that it now has the lowest sequence number. Node B becomes the leader.
  6. Node C's watch did NOT fire — it was watching candidate-0002 (Node B), which is still alive. Node C is undisturbed.

Why This Design Is Clever

No thundering herd: If all non-leader nodes watched the leader directly, then when the leader dies, all N-1 nodes would simultaneously wake up, check their status, and contend for leadership. This is called the "herd effect" and creates a burst of ZooKeeper traffic. By having each node watch only its predecessor, exactly one node wakes up when the leader dies — the next in line.

Ephemeral nodes prevent stale leaders: If a leader crashes or becomes partitioned from ZooKeeper, its ephemeral node automatically disappears after the session timeout (typically 10-30 seconds). There is no way for a dead leader to leave a stale lock behind — ZooKeeper cleans it up automatically.

Sequential ordering creates a fair queue: Nodes become leader in the order they joined. This is predictable and avoids contention — there is no "election" in the traditional sense, just an orderly succession.

When to Use / When NOT to Use

Use Leader Election When

  • Database primary selection: you need a single-writer primary for strong consistency (PostgreSQL streaming replication, MySQL Group Replication).
  • Distributed lock coordination: a single coordinator must grant locks and detect deadlocks (etcd, ZooKeeper).
  • Partition assignment: one node must assign data partitions to workers — Kafka's controller broker assigns partition leaders to brokers.
  • Cron job scheduling: only one node should run a scheduled task. Without election, multiple instances run the same job, causing duplicate work or data corruption.
  • Coordination-heavy tasks: two-phase commit coordinators, schema migration orchestrators, or workflow engines that need exactly-once execution semantics.

Do NOT Use Leader Election When

  • Leaderless design works: systems using CRDTs or Dynamo-style quorum reads/writes (e.g., Cassandra, Riak) avoid the single-leader bottleneck entirely. Use these when availability matters more than strict ordering.
  • Single leader is a bottleneck: if write throughput exceeds what one node can handle, a single leader becomes a chokepoint. Consider multi-leader or partitioned designs instead.
  • Failover latency is unacceptable: leader election takes hundreds of milliseconds to seconds. If your system cannot tolerate any write downtime during failover (e.g., high-frequency trading), consider active-active with conflict resolution.
  • Stateless services: if all instances are identical and stateless (e.g., web servers behind a load balancer), there is nothing to elect a leader for.

Classic vs. Modern Approaches

Classic: Bully & Ring Algorithms

The bully algorithm (1982) and ring algorithm are the textbook approaches. They are conceptually simple and work well in tightly-coupled systems with reliable failure detection. However, they assume a crash-stop model with no network partitions and provide no protection against split-brain. In modern cloud environments where network partitions are routine, these algorithms are unsafe without additional mechanisms.

Paxos (Theoretical Foundation)

Leslie Lamport's Paxos (1989/1998) was the first algorithm to solve consensus correctly in asynchronous networks with crash failures. Paxos guarantees safety (no two leaders in the same "ballot") even under partitions. However, Paxos is notoriously difficult to understand and implement correctly. Multi-Paxos extends single-decree Paxos to a replicated log but leaves many implementation details unspecified, leading to divergent and error-prone implementations.

Raft (Understandable Consensus)

Diego Ongaro and John Ousterhout designed Raft (2014) specifically to be understandable. Raft decomposes consensus into three clear sub-problems — leader election, log replication, and safety — making it far easier to implement correctly than Paxos. Raft is the algorithm behind etcd, CockroachDB, TiKV, and Consul. Its explicit leader-based design maps naturally to systems that already need a primary node.

ZooKeeper/etcd-Based Election (Delegated)

Rather than building consensus into your application, you can delegate leader election to a dedicated coordination service. ZooKeeper (Zab protocol) and etcd (Raft) provide primitives like ephemeral nodes and lease-based locks that applications use to elect a leader without implementing the consensus algorithm themselves. This is the most common approach in practice — Kafka, HBase, and HDFS all use ZooKeeper for leader election (though Kafka is migrating to its own Raft-based KRaft protocol).

Kubernetes Leader Election (Lease-Based)

Kubernetes provides a lease-based leader election mechanism using its API server (backed by etcd). A pod acquires a Lease resource and periodically renews it. If the lease expires (the pod crashes or becomes partitioned), another pod acquires the lease and becomes the new leader. This is used by the Kubernetes controller manager and scheduler, and is available to any application running in a Kubernetes cluster.

Fencing Tokens for Safety

Even with a correct election algorithm, a stale leader can cause damage. A leader that was partitioned may still believe it is the leader and issue writes. Fencing tokens (monotonically increasing numbers issued with each leadership term) solve this: every write carries the fencing token, and storage systems reject writes with an older token. This is the last line of defense against split-brain data corruption and is essential in any production leader election system.

See the detailed walkthrough in the How It Works section above for a concrete timeline showing how fencing tokens prevent stale leader writes during a GC pause scenario.

Assumptions & Model

Key Assumptions

  • Crash-recovery model (not Byzantine): nodes fail by crashing and may recover, but they do not send arbitrary or malicious messages. Byzantine fault tolerance requires different (and far more expensive) protocols like PBFT.
  • Majority of nodes available: Raft (and Paxos) require a majority quorum to make progress. A 5-node cluster tolerates 2 failures; a 3-node cluster tolerates 1. If a majority is unavailable, the cluster cannot elect a new leader and halts writes.
  • Clocks are roughly synchronized: election timeouts rely on local clocks. While Raft does not require synchronized clocks for safety (it uses logical terms), severely skewed clocks can cause premature elections or election storms if timeouts are too short relative to clock drift.
  • Network partitions are detectable via timeout: the only way a follower knows the leader is down is that heartbeats stop arriving. This means the system cannot distinguish between a crashed leader, a slow leader, and a network partition — it treats them all identically.

When Assumptions Break

  • Byzantine faults: a compromised node could forge votes or heartbeats, undermining the election. This requires BFT protocols, which are impractical for most infrastructure software.
  • Network partition isolates the leader from the majority: the old leader may continue serving reads (or even writes, if not using quorum writes) on its side of the partition, while the majority elects a new leader. This is the classic split-brain scenario. Fencing tokens and quorum-based writes are the mitigations.
  • Clock skew causes premature elections: if a follower's clock runs fast, its election timeout fires early, triggering an unnecessary election. Repeated across nodes, this causes "election storms" — a cascade of failed elections that prevents any leader from being established. Using randomized timeouts and NTP helps, but extreme clock skew can still be disruptive.

Sizing & Tradeoffs

Good sizing starts from SLOs, not from defaults. Work backward: define the availability, recovery time, and latency targets your system must meet, then derive the election parameters that satisfy those targets. The examples below show this process for four common leader-election scenarios.

SLO-Driven Sizing Examples

Example 1 — Distributed Scheduler

SLO: Leader failover < 5 s, 99.99% availability (52.6 min downtime/year).

Derivation:

  • Failover budget breakdown: 5 s total = detection + election + client re-routing. Allocate 3 s for detection, 500 ms for election, 1.5 s for clients to discover the new leader via service discovery.
  • Heartbeat interval: To detect failure within 3 s, the follower must miss several heartbeats. With a 500 ms heartbeat interval and a threshold of 5 missed heartbeats, detection time = 5 × 500 ms = 2.5 s (within the 3 s budget).
  • Election timeout: Set to 2.5–3.5 s (randomized). The randomization range must be wide enough to prevent split votes: at least 1 s spread for scheduler workloads where network jitter is on the order of tens of milliseconds.
  • Cluster size: 99.99% availability requires tolerating 2 simultaneous failures. Use a 5-node cluster (quorum = 3, tolerates 2 failures). Distribute 2-2-1 across 3 AZs so that any single AZ loss preserves quorum.
  • Availability math: With a per-node failure rate of 0.1% per hour and 5 nodes, the probability of losing 3+ nodes simultaneously is ~10-9, well within the 99.99% target. Each failover event contributes ~5 s of downtime; at a leader MTBF of ~30 days, annual failover downtime ≈ 12 × 5 s = 60 s, leaving ample budget.

Example 2 — Database Primary Election (Raft/Paxos)

SLO: RTO < 10 s, RPO < 1 s (at most 1 s of committed writes may be lost).

Derivation:

  • RPO constraint drives replication mode: RPO < 1 s rules out asynchronous replication (which can lag by seconds to minutes). Use synchronous quorum writes (Raft default): every write is committed only after a majority acknowledges it, so RPO = 0 for committed writes. For "acknowledged but uncommitted" writes during leader failure, the pipeline depth times the heartbeat interval must be < 1 s.
  • Heartbeat interval: 200 ms. At this rate, the write pipeline holds at most 1 in-flight batch per heartbeat, limiting uncommitted data exposure to ~200 ms (well under the 1 s RPO).
  • Election timeout: 1–2 s (randomized). This gives 5–10× the heartbeat interval, preventing false elections from transient network blips. Detection at ~1.5 s average.
  • RTO budget: 1.5 s detection + 200 ms election + 1 s log catch-up on new leader + 500 ms client reconnection = ~3.2 s, well within the 10 s RTO.
  • Quorum sizing: A 5-node Raft group is standard for production databases. With quorum writes, any new leader is guaranteed to have all committed entries. For multi-region deployments where cross-region RTT is 50–100 ms, increase the heartbeat interval to 500 ms and election timeout to 3–5 s (RTO rises to ~8 s but stays within budget).

Example 3 — Kafka Partition Leader Rebalance

SLO: Full rebalance < 30 s for 10,000 partitions after a broker failure.

Derivation:

  • Controller detection time: The Kafka controller detects a broker failure via session timeout. Set broker.session.timeout.ms = 6000 (6 s). The controller learns of the failure within 6 s.
  • Partition leader elections: The controller must elect new leaders for all partitions hosted on the failed broker. If the broker hosted 2,000 of the 10,000 partitions, the controller must perform 2,000 leader elections.
  • Batch election throughput: With KRaft, the controller processes leader elections in batches. At ~500 elections/s (metadata log append + ISR lookup per partition), 2,000 elections take ~4 s. With the legacy ZooKeeper controller, sequential ZK writes limit throughput to ~50 elections/s, so the same workload takes 40 s (exceeds SLO — this is why KRaft exists).
  • Controller sizing (KRaft): Use a 3-node KRaft controller quorum (dedicated, not co-located with brokers). Each controller needs 4 vCPU and 8 GB RAM to handle metadata log replication at the required throughput. Total budget: 6 s detection + 4 s elections + 2 s metadata propagation to brokers = 12 s, within the 30 s target.
  • Scaling limit: At 50,000 partitions per failed broker, batch elections take ~100 s. To stay within the 30 s SLO at that scale, shard partitions across more brokers (max ~15,000 partitions per broker) or increase controller resources.

Example 4 — Distributed Lock Service

SLO: p99 lock acquisition < 100 ms, sustained throughput of 50,000 locks/s.

Derivation:

  • Throughput sizing: Each lock acquisition requires a consensus round (leader append + quorum ack). A single Raft group on modern hardware can handle ~10,000–15,000 writes/s (bounded by fsync latency of ~50–100 µs on NVMe). To reach 50,000 locks/s, shard locks across 4–5 independent Raft groups (each a 5-node cluster), partitioned by lock key hash.
  • Latency budget: 100 ms p99 = network RTT to leader + Raft append + quorum replication + response. With intra-AZ RTT of 0.5 ms and fsync at 100 µs: lock request routing ~1 ms + leader append ~0.1 ms + quorum ack (2 followers, parallel) ~1.5 ms + response ~1 ms = ~3.6 ms typical. The p99 accounts for tail latencies from GC pauses and disk contention; keep GC pauses < 50 ms (use ZGC or Shenandoah) to stay within the 100 ms p99.
  • Lease parameters: Lock leases must be long enough that holders can complete work, but short enough to reclaim abandoned locks promptly. Set lease TTL = 10 s with a 3 s renewal interval. If a holder crashes, the lock is available after 10 s. For the lock service's own leader election: heartbeat = 100 ms, election timeout = 500–800 ms, so lock service failover takes ~1 s (clients retry during this window; most will succeed within the 100 ms p99 budget on the next attempt).
  • Cluster sizing total: 5 Raft groups × 5 nodes = 25 nodes. Each node: 4 vCPU, 16 GB RAM, NVMe SSD. Estimated infrastructure cost: ~$3,000–5,000/month on AWS (m6id.xlarge instances).

SLO-Driven Capacity Planning

Use this process to derive leader-election parameters from your SLOs rather than guessing at defaults:

StepInputOutput
1. Define failure budget Target availability (e.g., 99.99%) Allowed annual downtime (e.g., 52.6 min/year)
2. Decompose failover time Max acceptable failover duration (RTO) Time budgets for detection, election, and re-routing
3. Set heartbeat & timeout Detection budget, network RTT P99 Heartbeat interval = detection budget / missed-beat threshold; election timeout = 3–5× heartbeat interval (randomized)
4. Size the cluster Failures to tolerate, AZ count Cluster size = 2f + 1 nodes, distributed across ≥ 3 AZs
5. Validate with math Per-node failure rate, failover duration, leader MTBF Expected annual downtime = (365 / MTBF_days) × failover_seconds. Confirm this fits within the failure budget from step 1.
6. Load test & tune Chaos tests (kill leader, partition AZ) Measured failover time. If it exceeds the budget, tighten heartbeat interval or reduce client re-routing latency (e.g., push-based discovery).

Rule of Thumb

broadcastTime << electionTimeout << MTBF — the Raft paper's original sizing guidance still holds. SLO-driven planning simply adds rigour: your election timeout must be less than RTO minus client-rerouting time, and your cluster size must tolerate enough failures to meet your availability target. Always validate with chaos testing under production-like load.

Scaling Journey

0 – 1K users

Single Server

No election needed. One server handles everything. If it fails, a human restarts it. Downtime is acceptable at this scale.

1K – 100K users

Manual Failover

Primary-replica setup. If the primary fails, an operator promotes a replica manually via runbook. Failover takes minutes. You add monitoring to detect primary failure faster. This works until your SLA demands sub-minute recovery.

100K – 10M users

Delegated Election

Deploy ZooKeeper or etcd (3-node cluster) for automatic leader election. Your application uses the coordination service to elect a primary. Failover is automatic and takes seconds. You start using fencing tokens to prevent stale-leader writes.

10M+ users

Built-in Raft

At scale, you embed Raft directly into your storage layer (like CockroachDB, TiKV). Each data range has its own Raft group, so leader election is per-partition. This eliminates the external dependency on ZooKeeper/etcd and scales horizontally. Multi-region deployments use higher election timeouts to account for cross-region latency (50-100ms RTT).

What Can Go Wrong

Split-Brain: Two Leaders Accept Conflicting Writes

What happened: A network partition isolates the current leader from the rest of the cluster. The leader continues accepting writes on its side of the partition (it hasn't received any indication that it's no longer leader). Meanwhile, the majority elects a new leader and also begins accepting writes. When the partition heals, the system has two divergent histories that cannot be automatically reconciled.

Root cause: The old leader did not check for quorum acknowledgment before committing writes. In Raft, writes require majority acknowledgment, so a partitioned leader cannot commit — but in simpler systems (or systems that allow local reads), the old leader can serve stale or conflicting data.

Mitigation: Require quorum writes (Raft's default), implement fencing tokens so storage rejects writes from the old leader's term, and use leader leases with bounded clock skew so the old leader steps down before its lease expires.

Election Storms: Repeated Failed Elections

What happened: In a cluster with aggressive election timeouts (e.g., 50ms) and high network jitter, multiple followers simultaneously time out and become candidates. Their votes split, no one gets a majority, and a new election starts. This repeats in a cascade — the cluster spends minutes cycling through elections without ever establishing a stable leader. All writes are blocked during this period.

Root cause: Election timeouts were not sufficiently randomized, or the timeout range was too narrow relative to network latency variance. In one real incident, a misconfigured NTP server caused clocks on two nodes to drift in the same direction, causing their timeouts to fire nearly simultaneously every time.

Mitigation: Use wide randomization ranges for election timeouts (Raft's original recommendation of 150-300ms provides 2x range). Implement pre-vote protocol (Raft PreVote extension) where a candidate checks if it can win before incrementing the term. Monitor election frequency and alert if it exceeds 1 per hour.

Stale Leader Corrupts Data (Missing Fencing Tokens)

What happened: A leader experienced a long GC pause (several seconds). The rest of the cluster elected a new leader and processed new writes. When the old leader's GC pause ended, it resumed processing its in-flight requests and wrote to a shared storage backend, overwriting data that the new leader had already updated.

Root cause: The shared storage layer did not validate fencing tokens. The old leader's writes were accepted because the storage had no way to know a newer leader existed.

Mitigation: Always use fencing tokens (monotonically increasing term/epoch numbers). The storage layer must reject any write with a fencing token lower than the highest token it has seen. This is the only reliable defense against stale-leader writes, because you cannot guarantee that a leader will detect its own demotion promptly (GC pauses, VM migrations, and CPU throttling all create blind spots).

Interactions & Pitfalls

Leader Election + Database Replication

The leader in a consensus group is typically the database primary. When a new leader is elected, it must ensure its log contains all committed entries before serving writes. If the new leader's log is behind (possible in some replication configurations), it might acknowledge writes as committed that later get overwritten when the log catches up. Raft's log matching property prevents this, but simpler replication setups (async replication + manual failover) are vulnerable. Always verify that your replication protocol guarantees that the new primary has all committed data.

Leader Election + Load Balancing

Clients must discover and route writes to the current leader. If a load balancer distributes writes across all nodes indiscriminately, followers will reject or forward them, adding latency. Solutions: followers return a redirect to the leader, the load balancer watches the leader's identity via health checks or service discovery, or clients cache the leader's address and refresh on connection error.

Leader Election + Distributed Transactions

If the leader serves as the coordinator for two-phase commit (2PC), a leader change mid-transaction can leave participants in an uncertain state (they have voted "prepare" but haven't received "commit" or "abort"). The new leader must recover in-flight transactions from the log and drive them to completion. This is one of the hardest problems in distributed systems and a major reason why 2PC is avoided when possible.

Leader Election + Service Discovery

The current leader must be discoverable by clients. If service discovery has stale information (e.g., DNS TTL hasn't expired), clients route to the old leader. Use short TTLs, gRPC/HTTP health checks that report leadership status, or push-based updates (watches in etcd/ZooKeeper). The discovery latency adds to the total failover time experienced by clients.

Cost Considerations

Cost Analysis

  • Compute cost is minimal: leader election itself requires negligible CPU. Heartbeat messages are tiny (tens of bytes). A 5-node etcd cluster runs comfortably on 3 small instances (e.g., 2 vCPU, 8 GB RAM) costing ~$150-300/month total on AWS.
  • Dominant cost driver: not the election mechanism itself, but the complexity and risk it introduces. Split-brain incidents can corrupt data, requiring expensive manual recovery. The engineering cost of correctly implementing fencing, monitoring, and testing failover far exceeds the infrastructure cost.
  • Delegated vs. embedded: using a managed coordination service (AWS CloudMap, GCP's managed etcd via GKE) avoids the operational cost of running your own consensus cluster. Managed services cost slightly more in dollars but dramatically less in engineer-hours.
  • Cost of NOT having leader election: without automatic failover, your system's availability depends on how quickly a human can be paged and manually promote a replica. For a 99.9% SLA (8.7 hours downtime/year), manual failover is often insufficient. The cost of a 1-hour outage for a revenue-generating service typically dwarfs the infrastructure cost of a coordination cluster.

Real-World Examples

Kafka Controller Election

Apache Kafka traditionally used ZooKeeper for controller election. One broker is elected as the controller, responsible for assigning partition leaders to brokers, handling broker failures, and managing topic creation. When the controller fails, ZooKeeper's ephemeral node expires and another broker claims the controller role. Since KIP-500 (Kafka 2.8+), Kafka has been migrating to KRaft — an internal Raft-based consensus protocol that eliminates the ZooKeeper dependency. The KRaft controller quorum (typically 3 or 5 nodes) handles metadata management and controller election natively.

etcd and Raft

etcd is a distributed key-value store that uses Raft for consensus. It serves as the backing store for Kubernetes (storing all cluster state). etcd's Raft implementation is one of the most battle-tested in the industry, running in millions of Kubernetes clusters. Each etcd cluster elects a leader that handles all write requests. Followers forward writes to the leader and serve linearizable reads (via read-index protocol) or serializable reads from local state.

ZooKeeper at Yahoo and LinkedIn

Apache ZooKeeper, developed at Yahoo, uses the Zab (ZooKeeper Atomic Broadcast) protocol — a consensus algorithm similar to Paxos with a distinguished leader. ZooKeeper provides leader election as a primitive for applications: a client creates an ephemeral sequential znode, and the client with the lowest sequence number is the leader. LinkedIn used ZooKeeper extensively for Kafka broker coordination, Helix cluster management, and Espresso (LinkedIn's document store) primary election before migrating many of these to more modern approaches.

Kubernetes Controller Manager Leader Election

The Kubernetes controller manager and scheduler use lease-based leader election. Only one instance should be active at a time to avoid duplicate reconciliation. The active instance acquires a Lease object in the Kubernetes API (backed by etcd) and renews it periodically. If the lease expires, another instance acquires it. This pattern is exposed to user workloads via the client-go library's leaderelection package, making it trivial for any Kubernetes application to implement leader election.

Code Example

A simplified Raft election simulation in Python. This models heartbeats, timeouts, vote requests, and leader emergence in a cluster of nodes.

import random, time

class Node:
    def __init__(self, id, peers):
        self.id = id
        self.peers = peers  # list of other Node objects
        self.state = "follower"  # follower | candidate | leader
        self.term = 0
        self.voted_for = None
        self.votes = 0
        self.timeout = random.uniform(0.15, 0.30)  # 150-300ms
        self.last_heartbeat = time.time()

    def receive_heartbeat(self, leader_term):
        if leader_term >= self.term:
            self.term = leader_term
            self.state = "follower"
            self.last_heartbeat = time.time()

    def request_vote(self, candidate_id, candidate_term):
        if candidate_term > self.term and self.voted_for is None:
            self.term = candidate_term
            self.voted_for = candidate_id
            return True  # grant vote
        return False

    def tick(self):
        if self.state == "leader":
            for peer in self.peers:  # send heartbeats
                peer.receive_heartbeat(self.term)
            return
        elapsed = time.time() - self.last_heartbeat
        if elapsed > self.timeout:  # timeout — start election
            self.term += 1
            self.state = "candidate"
            self.voted_for = self.id
            self.votes = 1  # vote for self
            for peer in self.peers:
                if peer.request_vote(self.id, self.term):
                    self.votes += 1
            majority = (len(self.peers) + 1) // 2 + 1
            if self.votes >= majority:
                self.state = "leader"
                print(f"Node {self.id} elected leader (term {self.term})")
                for peer in self.peers:
                    peer.receive_heartbeat(self.term)

# Create 5-node cluster
nodes = [Node(i, []) for i in range(5)]
for n in nodes:
    n.peers = [p for p in nodes if p.id != n.id]
nodes[0].state = "leader"  # initial leader
for _ in range(20):
    for n in nodes:
        n.tick()
    time.sleep(0.05)
print("Simulating leader failure...")
nodes[0].state = "follower"  # simulate crash
nodes[0].last_heartbeat = 0   # force timeout on node 0
for _ in range(20):
    for n in nodes:
        if n.id != 0:  # crashed node doesn't tick
            n.tick()
    time.sleep(0.05)

How to Validate Your Design

Kill the Leader

The most basic test: kill the leader process (or container) and measure the time until a new leader is elected and writes succeed again. Use a simple write loop that records the timestamp of each successful write. The gap in successful writes is your failover time. Target: under 500ms for a same-region cluster. Tools: kill -9, Docker stop, or Kubernetes kubectl delete pod.

Simulate Network Partitions

Use toxiproxy or iptables rules to create a network partition that isolates the leader from the majority. Verify that: (1) the majority elects a new leader, (2) the old leader stops accepting writes (or its writes are rejected by storage via fencing tokens), (3) no data is lost or corrupted when the partition heals. This is the most critical test for split-brain safety.

Clock Skew Testing

Use faketime or container-level clock manipulation to introduce clock skew on individual nodes. Verify that moderate skew (100ms) does not cause unnecessary elections, and that extreme skew (seconds) degrades gracefully rather than causing data loss. The system should prioritize safety over liveness under clock skew.

Chaos Testing at Scale

Use Chaos Monkey or Litmus to randomly kill nodes, introduce latency, and partition networks in a staging environment. Run this continuously and monitor election metrics. The system should converge to a stable leader within seconds under all failure scenarios. Key variables to sweep: election timeout range, heartbeat interval, network latency, and cluster size.

Interview Questions

How does Raft leader election work?

Raft divides time into terms. Each term begins with an election. Every node starts as a follower and waits for heartbeats from the leader. If a follower's randomized election timeout fires without receiving a heartbeat, it becomes a candidate: it increments its term, votes for itself, and sends RequestVote RPCs to all other nodes.

Each node votes for at most one candidate per term, on a first-come-first-served basis, but only if the candidate's log is at least as up-to-date as the voter's log. A candidate that receives votes from a majority of nodes becomes the leader for that term and immediately sends heartbeats to establish authority.

If votes are split and no candidate achieves a majority, the election times out, and a new election begins with a higher term. The randomized timeout range (e.g., 150-300ms) makes repeated split votes statistically unlikely. Safety is guaranteed by the term-based voting: there can be at most one leader per term because a majority can only vote for one candidate.

Follow-up: What happens if a leader is elected but its log is missing some entries? How does Raft prevent this from causing data loss?

What is split-brain and how do you prevent it?

Split-brain occurs when a network partition causes two nodes to both believe they are the leader, each accepting writes independently. When the partition heals, the system has two divergent data histories that may be impossible to reconcile without data loss.

Prevention relies on majority quorums: since a leader must receive votes from a majority of nodes, and two disjoint majorities are impossible, at most one leader can be elected per term. The partitioned old leader cannot commit new writes because it cannot reach a majority.

However, quorums alone are insufficient if the old leader serves cached reads or writes to an external storage layer that doesn't validate leadership. This is where fencing tokens are critical: every leadership term produces a monotonically increasing token, and the storage layer rejects writes with stale tokens. Additionally, leader leases with bounded clock skew ensure the old leader steps down before its lease expires, even if it cannot communicate with the rest of the cluster.

Follow-up: What is the difference between a fencing token and a distributed lock? Why are fencing tokens preferred?

What are fencing tokens and why are they needed?

A fencing token is a monotonically increasing number associated with each leadership term (or lock acquisition). When a leader writes to a shared resource, it includes its fencing token. The resource validates that the token is greater than or equal to the highest token it has previously seen and rejects writes with lower tokens.

Fencing tokens are necessary because a leader cannot always detect its own demotion in real-time. Consider: a leader experiences a 10-second GC pause, during which the cluster elects a new leader (term 5, token 5). When the old leader (term 4, token 4) resumes, it still believes it is the leader and issues writes. Without fencing tokens, the storage accepts these stale writes, overwriting the new leader's data. With fencing tokens, the storage sees token 4 < 5 and rejects the write.

This is fundamentally different from a distributed lock, which relies on the holder releasing it or a timeout expiring. Locks are vulnerable to the same GC-pause problem: a holder's lock expires while it is paused, another process acquires the lock and writes, then the original holder resumes and overwrites. Fencing tokens add a validation step at the resource level, making the system safe regardless of what the holder does.

Follow-up: How would you implement fencing token validation in a database? What if the database doesn't natively support it?

Design a distributed cron job scheduler

The core problem is ensuring each scheduled job runs exactly once, even when multiple scheduler instances are running for high availability. Leader election is the natural solution: one instance is elected the leader (the "scheduler"), and it is the only one that triggers jobs. The others are hot standbys.

Architecture: Deploy 3 scheduler instances. Use etcd or ZooKeeper for leader election. The leader reads job definitions from a database, maintains a time-based priority queue, and triggers jobs at the scheduled time by enqueuing work items into a message queue (e.g., Kafka, SQS). Workers consume from the queue and execute the jobs. The leader stores job execution state (last run time, result) in the database.

Exactly-once execution: The leader writes a job execution record to the database before enqueuing the work item, using a unique execution ID. Workers check this ID for deduplication. If the leader crashes mid-execution, the new leader reads pending execution records and either retries or marks them as failed based on the worker's acknowledgment.

Failover: If the leader crashes, a new leader is elected within ~500ms. It reads all job definitions and pending executions from the database, reconstructs the priority queue, and resumes scheduling. The brief gap may cause a job to run a few hundred milliseconds late, which is acceptable for cron-style workloads.

Follow-up: How would you handle a job that needs to run every second? Is leader election still the right approach, or would you consider a different design?

Raft vs. Paxos — why was Raft created?

Paxos, published by Leslie Lamport in 1989 (republished in 1998), was the first correct consensus algorithm for asynchronous networks. However, Paxos is notoriously difficult to understand. The original paper presents single-decree Paxos (agreeing on one value), and extending it to a replicated log (Multi-Paxos) requires many unspecified design decisions around leader election, log compaction, cluster membership changes, and exactly how to map single-decree consensus to a continuous stream of operations.

Diego Ongaro studied Paxos implementations and found that each one made different design choices, many of which were subtly wrong. He designed Raft (2014) specifically for understandability, decomposing consensus into three clean sub-problems: leader election, log replication, and safety. Each sub-problem can be understood independently.

Key differences: Raft mandates a strong leader (all writes go through the leader, and log entries flow in one direction), while Paxos allows any node to propose values. Raft's leader-based approach simplifies the algorithm but adds a leader election step that Paxos doesn't strictly need. Raft also specifies cluster membership changes (joint consensus) and log compaction (snapshotting) as part of the core algorithm, whereas Paxos leaves these as exercises for the implementer.

In practice, Raft and Multi-Paxos achieve similar performance and fault tolerance. Raft won adoption because implementers could read the paper and build a correct implementation — something that was rarely true of Paxos.

Follow-up: Are there scenarios where Paxos is preferable to Raft? What about Flexible Paxos and its implications for quorum sizes?

How does Kafka elect partition leaders?

Kafka has two levels of leadership: the controller broker and partition leaders. The controller is a single broker responsible for managing all partition leader assignments across the cluster.

Controller election (legacy): In ZooKeeper mode, the controller is elected by creating an ephemeral znode (/controller). The first broker to create it becomes the controller. If that broker fails, the ephemeral node is deleted, and another broker claims it. In KRaft mode (Kafka 3.x+), the controller is elected via an internal Raft quorum among designated controller nodes.

Partition leader assignment: The controller maintains an In-Sync Replica (ISR) set for each partition — the set of replicas that are fully caught up with the leader. When a partition leader fails, the controller selects a new leader from the ISR (typically the first replica in the ISR list, which is the "preferred leader"). This is not a consensus election — it is a unilateral decision by the controller, which is safe because the controller is the single authority for metadata. The new leader is announced to all brokers via a LeaderAndISR request.

If the ISR is empty (all replicas are behind), Kafka can either wait for a replica to catch up (safe but unavailable) or elect an out-of-sync replica (unclean.leader.election.enable=true), which risks data loss. Most production deployments keep unclean election disabled.

Follow-up: What happens if the controller itself is slow or overloaded? How does KRaft improve on ZooKeeper-based controller election?

Key Takeaways

  • Leader election selects one node as the coordinator or single writer in a distributed cluster, simplifying consistency at the cost of requiring failover when the leader fails.
  • Raft is the standard for modern leader election: it uses terms, majority quorums, and randomized timeouts to guarantee at most one leader per term. It was designed for understandability and has proven implementations in etcd, CockroachDB, and Consul.
  • Split-brain is the primary danger: two nodes both acting as leader can corrupt data. Quorum-based election prevents it in theory; fencing tokens prevent it in practice by validating writes at the storage layer.
  • Election timeout tuning is critical: too short causes thrashing (repeated unnecessary elections), too long delays failover. Follow the Raft guidance: heartbeat << election timeout << MTBF.
  • Use odd cluster sizes (3, 5, 7) for clean majority quorums. A 5-node cluster tolerating 2 failures is the production standard.
  • Delegate when possible: unless you are building a database or storage engine, use an existing coordination service (etcd, ZooKeeper, Kubernetes leases) rather than implementing Raft yourself.
  • Test failover relentlessly: kill the leader, simulate partitions, inject clock skew. The value of leader election is only realized if failover actually works when you need it.
