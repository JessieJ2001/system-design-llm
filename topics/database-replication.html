Database Replication

How distributed systems keep identical copies of data across multiple machines to achieve high availability, fault tolerance, and read scalability.

Estimated reading time: 25 minutes

What Is It?

Database replication is the process of keeping copies of the same data on multiple database servers, called replicas. When a user writes data to one server (the primary), that change is propagated to the other servers so they all hold an up-to-date copy. If the primary fails, one of the replicas can take over, preventing data loss and downtime.

There are three core motivations for replication. First, availability: if one server crashes, others can continue serving requests. Second, read scalability: by directing read queries to replicas, you spread the load across multiple machines instead of hammering a single database. Third, geographic distribution: placing replicas in different regions lets users read from a nearby copy, reducing latency from hundreds of milliseconds to single-digit milliseconds.

The two fundamental flavors of replication are synchronous and asynchronous. In synchronous replication, the primary waits for at least one replica to acknowledge the write before confirming it to the client. This guarantees the replica is up-to-date but adds latency to every write. In asynchronous replication, the primary confirms the write immediately and ships the change to replicas in the background. This is faster but means replicas can lag behind, and if the primary crashes before replication completes, those writes are lost.

Replication Modes Compared

There are actually three distinct replication modes, each with a different tradeoff between write speed and data safety. The following table compares them side by side:

Mode Write Latency Durability Data Loss Risk
Synchronous High (wait for ALL replicas) Strongest Zero (all replicas confirmed)
Asynchronous Low (return immediately) Weakest Up to lag duration of data
Semi-synchronous Medium (wait for 1 replica) Good Only if primary + ack'd replica both fail simultaneously

The client waited 24ms — the primary only needed the first replica to confirm. Data is safe on 2 machines (primary + Replica 1). The only way to lose this write is if both the primary and Replica 1 fail simultaneously before Replica 2 catches up, which is extremely unlikely.

How It Works

Replication follows a well-defined sequence of steps regardless of the specific database system:

  1. Write arrives at the primary. A client sends an INSERT, UPDATE, or DELETE to the primary database server. The primary is the sole authority that accepts writes.
  2. WAL entry is created. The primary records the change in its Write-Ahead Log (WAL) — a sequential, append-only file that captures every modification before it is applied to the actual data files. The WAL ensures durability: even if the server crashes immediately after writing, the change can be recovered from the log.
  3. Change is replicated to followers. The primary streams the WAL entries to each replica. In synchronous mode, the primary waits for at least one replica to confirm it received and persisted the WAL entry. In asynchronous mode, the primary sends the entry and continues without waiting.
  4. Followers apply changes. Each replica reads the incoming WAL entries and replays them against its own data files, bringing its state in line with the primary. This replay process is sequential — changes must be applied in the same order the primary produced them.
  5. Reads served from any replica. Once a replica has applied the changes, clients can read from it. In many architectures, a load balancer routes read queries to replicas while directing writes exclusively to the primary.

The gap between step 2 (primary writes the change) and step 4 (replica applies it) is called replication lag. It is the central tension in replication design: lower lag means stronger consistency guarantees but higher write latency and infrastructure cost.

The Read-Your-Own-Writes Problem: A Concrete Scenario

Replication lag causes real user-visible bugs. Here is a step-by-step scenario showing exactly how it happens. Imagine a user named Alice updates her display name from "Alice" to "Bob":

Time Client Action Primary State Replica State What Client Sees
T=0ms Write: UPDATE name='Bob' name='Bob' name='Alice' —
T=1ms Write ACK received name='Bob' name='Alice' (lag!) "Success"
T=2ms Read: SELECT name (routed to replica) — name='Alice' "Alice" ← STALE!
T=50ms (replication catches up) name='Bob' name='Bob' —
T=51ms Read: SELECT name — name='Bob' "Bob" ← correct now

The client wrote "Bob" and immediately read "Alice" back. This violates the read-your-own-writes guarantee. The user thinks their update failed, refreshes the page, sees "Bob" this time (replication caught up), and is thoroughly confused.

Solutions to Read-Your-Own-Writes

  1. Route to primary after write: For N seconds after a write (e.g., 2–5 seconds, matching your expected replication lag), read from the primary instead of a replica. This is simple and effective but increases primary load. Works well when writes are infrequent relative to reads.
  2. Causal consistency token: After a write, the primary returns a token representing its current position in the replication stream (e.g., a Log Sequence Number like LSN=42). The client includes this token in subsequent reads. A replica only serves the read if its replication position is ≥ LSN 42; otherwise, the read is redirected to the primary or the client waits briefly. This is more precise than a time-based approach and avoids unnecessary primary reads.

When to Use / When NOT to Use

Use Replication When

  • Read-heavy workloads: Your application has a high read-to-write ratio (e.g., 10:1 or higher). Replicas can absorb the read load, letting the primary focus on writes.
  • High availability is required: Downtime is unacceptable for your business. Replication enables automatic failover — if the primary dies, a replica is promoted within seconds.
  • Disaster recovery: You need data to survive a data center or region failure. Cross-region replicas act as a warm standby.
  • Geographic distribution: Your users are spread across continents. Placing read replicas near users reduces read latency from 200ms+ to under 10ms.

Don't Use Replication When

  • Write-heavy workloads: Replicas do not help with write throughput — all writes still go to the primary. If writes are your bottleneck, look at sharding instead.
  • Strong consistency everywhere is non-negotiable: Asynchronous replicas always lag. If you need every read to reflect the latest write no matter which server handles it, you must use synchronous replication — but accept the latency cost (1-10ms same region, 50-200ms cross-region per write).
  • Small, single-region deployments: If your database fits on one beefy server and your availability requirements are modest (e.g., an internal tool), the operational complexity of replication may not be worth it.

Classic vs. Modern Approaches

Classic: Primary-Replica Async Replication

The most common setup for decades. A single primary accepts all writes and streams changes asynchronously to one or more read replicas. PostgreSQL streaming replication and MySQL binlog replication both follow this model. It is simple to set up, well-understood, and performant — but it offers only eventual consistency and can lose committed writes if the primary fails before replication completes.

Synchronous Replication (Google Spanner)

Spanner takes the opposite extreme: every write is synchronously replicated using TrueTime (GPS + atomic clocks) and Paxos-based consensus across replicas in multiple regions. This provides external consistency — the strongest possible guarantee — at the cost of higher write latency (typically 10-15ms within a continent). Spanner proves that you can have globally consistent replication if you invest in the infrastructure.

Raft Consensus (etcd, CockroachDB)

Raft is a consensus algorithm designed to be understandable. It elects a leader that accepts writes and replicates them to a majority of followers before acknowledging. If the leader fails, a new election happens automatically. CockroachDB uses Raft at the range level — each range (chunk of data) has its own Raft group, allowing fine-grained replication. Raft simplified consensus compared to Paxos by decomposing the problem into leader election, log replication, and safety — three subproblems that are easier to reason about independently. Paxos, by contrast, presents the entire problem as a monolithic protocol, making it notoriously difficult to implement correctly.

Raft Leader Election: Step-by-Step

Let's walk through a complete leader election in a 5-node Raft cluster. Nodes 1–5 are running normally with Node 1 as the current leader in Term 1. Every node maintains an election timer — a countdown set to a random value between 150ms and 300ms. Each time a node hears from the leader (via a heartbeat or log entry), it resets its timer. If the timer expires, the node assumes the leader is dead.

Step Event Node 1 Node 2 Node 3 Node 4 Node 5
1 Normal operation Leader, Term 1 Follower, Term 1 Follower, Term 1 Follower, Term 1 Follower, Term 1
2 Node 1 crashes DEAD Follower (timer ticking...) Follower (timer ticking...) Follower (timer ticking...) Follower (timer ticking...)
3 Node 3's election timer fires first (it had the shortest random timeout: 180ms) DEAD Follower Candidate, Term 2 Follower Follower
4 Node 3 votes for itself, sends RequestVote RPC to Nodes 2, 4, 5 DEAD Receives RequestVote Votes for self (1 vote) Receives RequestVote Receives RequestVote
5 Nodes 2, 4, 5 see Term 2 > their Term 1, and Node 3's log is up-to-date → they grant their vote DEAD Grants vote to N3 Now has 4 votes Grants vote to N3 Grants vote to N3

The entire election took about 200–400ms. The cluster was unavailable for writes during this window but recovered automatically without human intervention.

Split Vote Scenario

What if two nodes' timers fire at nearly the same time? This is called a split vote, and Raft handles it gracefully. Suppose Node 3 and Node 4 both become candidates simultaneously (Node 1 is still dead):

Step Node 2 Node 3 Node 4 Node 5
1 Follower Candidate Term 2, votes for self Candidate Term 2, votes for self Follower
2 Receives RequestVote from Node 3 first → grants vote to N3 2 votes (self + N2) 2 votes (self + N5) Receives RequestVote from Node 4 first → grants vote to N4
3 — 2 out of 5 (no majority) 2 out of 5 (no majority) —
4 TIMEOUT — neither candidate won. All nodes wait a new random time (150–300ms), then try again in Term 3. Because the timeouts are random, it is extremely unlikely that two nodes will collide again.

Raft prevents perpetual split votes by randomizing election timeouts (150–300ms). The probability of repeated collisions drops exponentially with each round. In practice, elections almost always succeed on the first attempt.

Log Replication

Once a leader is elected, it accepts client writes and replicates them to followers. Here is the step-by-step process:

  1. Client sends write to Leader (Node 3). For example: SET x = 5.
  2. Leader appends entry to its log: {Term: 2, Index: 7, Command: 'SET x=5'}. The entry is not yet committed — it is tentative.
  3. Leader sends AppendEntries RPC to all followers (Nodes 2, 4, 5) containing the new log entry.
  4. Each follower appends the entry to its own log and responds with an ACK to the leader.
  5. Once the Leader receives ACKs from a majority (2 out of 4 followers — remember, the leader itself counts as one of the 5 nodes), the entry is committed. For example, Nodes 2 and 5 ACK, giving 3 out of 5 total (leader + 2 followers) = majority.
  6. Leader applies the entry to its state machine (actually executes SET x=5) and responds to the client: "Write committed."
  7. Leader notifies followers that the entry is committed in the next AppendEntries heartbeat. Followers then apply it to their own state machines.

This majority-based approach is what makes Raft fault-tolerant: even if 2 out of 5 nodes fail, the remaining 3 can still form a majority and continue accepting writes. No data that was committed is ever lost, because any committed entry exists on at least a majority of nodes, and any future leader must have that entry in its log (enforced by the voting rule).

CRDTs for Conflict-Free Multi-Leader (Riak, Redis CRDT)

Conflict-free Replicated Data Types (CRDTs) allow multiple leaders to accept writes simultaneously without coordination. CRDTs are data structures mathematically guaranteed to converge — counters, sets, registers, and maps that can be merged without conflicts regardless of the order updates arrive. Riak uses CRDTs extensively, and Redis added CRDT-based types for its active-active geo-distributed deployments. The tradeoff: CRDTs only work for specific data structures and can consume more storage due to metadata tracking.

G-Counter (Grow-Only Counter)

The simplest CRDT. The problem: you want a distributed counter that multiple nodes can increment without talking to each other. A naive shared integer fails because concurrent increments cause lost updates.

The insight: each node maintains its own counter in a vector. The total count is the sum of all entries. To merge two replicas, take the element-wise maximum.

Let's walk through a 3-node example (Nodes A, B, C). Each node's state is a vector [countA, countB, countC]:

Event Node A state Node B state Node C state
Initial [0, 0, 0] [0, 0, 0] [0, 0, 0]
A increments twice [2, 0, 0] [0, 0, 0] [0, 0, 0]
B increments once [2, 0, 0] [0, 1, 0] [0, 0, 0]
C increments 3 times [2, 0, 0] [0, 1, 0] [0, 0, 3]
A and B sync (merge = element-wise max) [2, 1, 0] [2, 1, 0] [0, 0, 3]
B and C sync [2, 1, 0] [2, 1, 3] [2, 1, 3]
A and C sync [2, 1, 3] [2, 1, 3] [2, 1, 3]

Total count = sum of vector = 2 + 1 + 3 = 6. All nodes converge to the same value. The merge operation (element-wise max) is commutative (A merge B = B merge A), associative ((A merge B) merge C = A merge (B merge C)), and idempotent (A merge A = A). This means the order and frequency of syncs doesn't matter — you always converge to the correct answer.

PN-Counter (Positive-Negative Counter)

A G-Counter can only grow. To support decrements, use two G-Counters: one for increments (P) and one for decrements (N). The actual value = sum(P) − sum(N).

Example: Node A increments 5 times (P_A = 5) and decrements 2 times (N_A = 2). Node B increments 3 times (P_B = 3). After merging: value = (5 + 3) − (2 + 0) = 6.

LWW-Register (Last-Writer-Wins Register)

A register holds a single value (like a variable). When multiple nodes write concurrently, we need a rule to pick the winner. LWW uses timestamps: the write with the highest timestamp wins.

Example:

  • Node A writes x = 'hello' at timestamp T=10
  • Node B writes x = 'world' at timestamp T=12
  • On merge: T=12 > T=10, so x = 'world' wins

The tradeoff: LWW can silently drop writes. If Node A and Node B write at nearly the same time, one write simply disappears. This is acceptable for some use cases (e.g., "last update to a user's status wins") but dangerous for others (e.g., you wouldn't want a bank balance update to be silently discarded). LWW also requires reasonably synchronized clocks — if Node A's clock is ahead by 5 seconds, its writes will always "win" even if they happened earlier in real time.

Chain Replication (FAWN, Microsoft)

In chain replication, replicas are arranged in a chain. Writes enter at the head and propagate down the chain; reads are served from the tail (which has the most up-to-date confirmed data). This separates read and write paths cleanly and provides strong consistency with high throughput. Microsoft's Azure Storage uses a variant of chain replication. The main weakness is that a failure in the middle of the chain requires reconfiguration, and the tail becomes a read bottleneck unless the chain is short.

Chain Replication: How It Works Step by Step

Consider a chain of 3 nodes: HEAD → MIDDLE → TAIL

Write path:

  1. Client sends write (SET x=5) to HEAD
  2. HEAD applies the write to its local state, then forwards it to MIDDLE
  3. MIDDLE applies the write, then forwards it to TAIL
  4. TAIL applies the write, then sends an ACK back up the chain to HEAD
  5. HEAD responds to the client: "Write committed"

Read path:

  1. Client sends read (GET x) to TAIL
  2. TAIL responds with its local state: x = 5

Why this gives strong consistency: The TAIL has applied ALL committed writes (a write isn't committed until TAIL has it and has sent the ACK). So reads from TAIL always see the latest committed state. Stale reads are impossible. Unlike primary-replica setups where replicas may lag, the tail is guaranteed to be fully up-to-date for all committed data.

Failure handling:

  • MIDDLE fails: The chain is reconfigured to HEAD → TAIL. HEAD resends any writes that MIDDLE received but hadn't forwarded to TAIL yet. These in-flight writes are redelivered.
  • HEAD fails: MIDDLE becomes the new HEAD. Any writes that the old HEAD received but hadn't forwarded to MIDDLE are lost (they were never acknowledged to the client, so no committed data is lost).
  • TAIL fails: MIDDLE becomes the new TAIL. But MIDDLE might have writes that it forwarded to TAIL but TAIL hadn't ACK'd yet. These writes must be carefully handled — since they were never ACK'd to the client, they aren't committed, so MIDDLE can safely re-commit them as the new tail.

A separate configuration manager (often using Paxos or Raft) monitors the chain and handles reconfiguration when nodes fail. This is the added complexity compared to simple primary-replica setups.

Assumptions & Model

Key Assumptions

  • Network is reliable enough for replication: The models on this page assume the network between primary and replicas has sufficient bandwidth and low enough latency to keep up with the write rate. In practice, network congestion or cross-region links can cause replication lag to grow unboundedly.
  • Replicas have sufficient resources: Each replica must have enough CPU, memory, and disk I/O to replay the write stream in near-real-time. A replica running on underpowered hardware will fall behind.
  • Clock synchronization (for some methods): Synchronous replication schemes like Spanner's TrueTime depend on tightly synchronized clocks (via GPS/atomic clock hardware). Raft does not require synchronized clocks but does assume bounded message delays for liveness.

When These Assumptions Break Down

  • Network partitions: If the primary cannot reach replicas, synchronous replication blocks all writes (availability sacrifice). Asynchronous replication continues but risks data divergence.
  • Replica hardware too slow: If a replica's disk I/O cannot keep up with the write rate, replication lag grows without bound, eventually making the replica useless for consistent reads.
  • Unbounded replication lag under write spikes: A sudden burst of writes (e.g., a batch import) can cause replicas to fall hours behind. Some systems handle this by throttling writes; others simply accept the lag.

Sizing & Tradeoffs

Number of Replicas

More replicas means better read throughput (linear scaling of read capacity) and higher availability (can tolerate more simultaneous failures). But each additional replica adds write amplification — every write to the primary must be sent to every replica — and multiplies storage cost. For consensus-based replication (Raft, Paxos), you need an odd number: 3 replicas tolerate 1 failure, 5 tolerate 2. Most production systems use 3 or 5 replicas.

Replication Lag Tolerance

Synchronous replication adds latency to every write: 1-10ms within the same region (network round-trip + fsync on replica), 50-200ms for cross-region replication. Semi-synchronous (wait for 1 of N replicas) is a common middle ground: you get durability without waiting for the slowest replica.

SLO-Driven Sizing Examples

Rather than picking replica counts from convention, derive them from your Service Level Objectives. Each example below starts with concrete SLOs and works backward to the replication architecture.

Example 1: E-Commerce Product Reads

SLOs: p99 read latency < 10ms, sustain 200K RPS reads, 99.95% availability.

Derivation: A single read replica handles approximately 30K RPS for index-heavy product lookups before p99 crosses 10ms (benchmarked). To serve 200K RPS: ceil(200,000 / 30,000) = 7 replicas minimum. Add 2 replicas for headroom during rolling deployments and single-node failures: 9 read replicas. Place replicas across 3 availability zones (3 per AZ) so losing one AZ still leaves 6 replicas — enough for 180K RPS, which covers typical traffic with graceful degradation during the failover window.

Replication mode: Asynchronous — product catalog reads tolerate < 500ms staleness, and synchronous replication would add 2-5ms to every write, degrading the merchant content-publishing pipeline.

Example 2: Financial Ledger

SLOs: RPO < 1 second (at most 1s of data loss on failure), RTO < 30 seconds (recovery within 30s).

Derivation: RPO < 1s eliminates pure async replication (typical lag 100ms–5s under spikes). Use semi-synchronous replication: the primary waits for at least 1 replica to acknowledge before committing. This bounds RPO to effectively 0 for the acknowledging replica. Deploy 3 replicas (1 semi-sync in the same region for RPO, 2 async cross-region for disaster recovery). RTO < 30s requires automated failover — use an orchestrator (Patroni, Orchestrator) with a health-check interval of 5s and a failover trigger after 3 missed checks (15s detection + ~10s promotion = ~25s total).

Cost tradeoff: Semi-sync adds 1-3ms write latency. At 2,000 TPS for a ledger workload, this is acceptable. The alternative — synchronous Raft-based consensus (CockroachDB, Spanner) — provides RPO = 0 but adds 5-15ms per write.

Example 3: Global Social Media Feed

SLOs: p95 read latency < 50ms worldwide, 99.99% availability (52 minutes downtime/year).

Derivation: Cross-region network latency alone is 60-200ms, so serving reads from a remote region violates the 50ms SLO. Solution: place read replicas in every major region — US-East, US-West, EU-West, AP-Southeast, AP-Northeast (5 regions minimum). Per region, deploy 3 replicas behind a load balancer for intra-region fault tolerance. Total: 1 primary + 15 read replicas across 5 regions. For 99.99% availability, no single region failure can cause global impact: use DNS-based routing (Route 53 latency routing) to redirect users to the nearest healthy region. Each region must have enough capacity to absorb spillover from an adjacent failed region (~1.5x normal capacity).

Replication mode: Asynchronous with lag monitoring. Feed reads tolerate 2-5s staleness. Use read-your-own-writes routing (sticky sessions to the primary region for 5s after a write) to mask replication delay for the authoring user.

Example 4: Analytics Dashboard

SLOs: p95 query latency < 500ms, tolerate up to 5 minutes of data staleness, support 50 concurrent analysts.

Derivation: Analytics queries are heavy (full table scans, aggregations) and would degrade OLTP performance if run on the primary. Dedicate 2 read replicas sized with 2x the RAM of the primary (to cache large working sets) and fast local NVMe storage. Two replicas allow one to handle queries while the other catches up on WAL replay after a heavy query batch. The 5-minute staleness tolerance means you can use delayed replication — intentionally applying WAL with a 1-minute delay — which also provides a recovery window for accidental data changes (you have 1 minute to cancel a bad DELETE before it reaches the analytics replicas).

Connection management: Use a connection pooler (PgBouncer) with max_client_conn = 50 and default_pool_size = 20 per replica, queuing excess queries rather than overwhelming the replicas.

SLO-Driven Capacity Planning

Use this framework to derive your replication topology from SLOs rather than guesswork:

  1. Define SLOs first. Pin down your p95/p99 latency targets, availability target (number of nines), RPO, and RTO. These are your hard constraints — everything else is derived.
  2. Benchmark a single replica. Load-test one replica with your actual query mix to find its saturation point — the RPS at which your latency SLO is first breached. Call this R_max.
  3. Compute minimum replica count. For read scaling: N_min = ceil(target_RPS / R_max). For availability, add replicas to survive your failure domain (e.g., +1 per AZ if you need to survive an AZ failure).
  4. Choose replication mode from RPO. RPO = 0 requires synchronous or consensus-based replication. RPO < 1s requires semi-synchronous. RPO < 1min allows asynchronous. Map your RPO to the mode that satisfies it with minimum write-latency overhead.
  5. Derive placement from latency SLOs. If your read-latency SLO is tighter than the cross-region round-trip time, you must place replicas in every region you serve. If it is looser, you can centralize replicas and save cost.
  6. Add headroom. Multiply N_min by 1.3-1.5x for burst traffic, rolling deployments, and single-node failures. Validate the final topology against your availability SLO using failure-mode analysis: simulate losing one node, one AZ, and one region, and confirm SLOs are still met.

Rule of thumb: If you cannot write down the SLO that justifies a replica, you probably do not need it. Every replica has a cost — storage, replication bandwidth, operational complexity. Let the SLOs tell you when to stop adding.

Scaling Journey

0 – 1K Users

Single Database

A single PostgreSQL or MySQL instance handles all reads and writes. No replication needed. Back up to cloud storage daily. Downtime during failures is acceptable.

Signal to evolve: Database restarts or hardware failures cause user-facing outages lasting minutes to hours.

1K – 100K Users

Single Read Replica

Add one async read replica. Direct all writes to the primary, offload read-heavy queries (dashboards, reporting) to the replica. This also serves as a hot standby for failover.

Signal to evolve: Read traffic overwhelms the single replica, or you need replicas in a second region for latency.

100K – 10M Users

Multiple Replicas + Read Routing

Deploy 3-5 read replicas behind a connection pooler (PgBouncer, ProxySQL) that routes reads by replica health and lag. Introduce monitoring for replication lag. Consider semi-synchronous replication for the nearest replica.

Signal to evolve: Write throughput hits the primary's ceiling, or you need active-active in multiple regions.

10M+ Users

Multi-Region / Multi-Leader

Deploy primary or leader nodes in multiple regions using Raft-based systems (CockroachDB, Spanner) or multi-leader with CRDT conflict resolution. Combine with sharding to scale writes. Each shard has its own replication group.

Migration pitfall: Switching from single-leader to multi-leader requires resolving write conflicts your application was never designed to handle.

What Can Go Wrong

Replication Lag & Read-Your-Own-Writes Inconsistency

What happened: A user updates their profile, then immediately refreshes the page. The read is routed to a replica that hasn't yet received the write. The user sees their old data and thinks the update was lost. At GitHub, this pattern caused confusion when users pushed code and the web UI showed stale repository state.

Root cause: Asynchronous replication lag combined with read routing that doesn't account for recency.

Mitigation: Implement read-your-own-writes consistency by routing reads to the primary for a short window after a write (using a session token or timestamp), or by checking the replica's replication position before serving the read.

Split-Brain in Multi-Leader Setups

What happened: A network partition between two data centers caused both sides to elect a new primary. Both accepted writes independently. When the partition healed, the databases had divergent data that could not be automatically reconciled. This scenario has occurred at multiple companies running MySQL multi-master replication.

Root cause: Lack of proper fencing — no mechanism to ensure only one primary exists at a time.

Mitigation: Use consensus-based leader election (Raft, ZooKeeper) with fencing tokens. If a node cannot reach a quorum, it must stop accepting writes. STONITH (Shoot The Other Node In The Head) is a last-resort fencing mechanism.

Split-Brain Timeline: How It Happens

Here is a step-by-step timeline showing how split-brain develops and causes data divergence in a 3-node cluster:

Fencing Tokens: Preventing Stale Leaders

Fencing tokens are monotonically increasing numbers assigned to each new leader. When Node B becomes the new primary, it receives fencing token #2 (Node A held token #1). Every write request now includes the fencing token. The storage system tracks the highest token it has seen and rejects writes with older tokens.

So when Node A (with token #1) tries to write SET x = 1 after the partition heals, the storage system says: "I've already seen token #2 from Node B. Token #1 is stale. Write rejected." This prevents the stale leader from causing data divergence, even though it still thinks it is the leader.

STONITH (Shoot The Other Node In The Head)

STONITH is a last-resort fencing mechanism used in high-availability clusters. When Node B suspects that Node A is still acting as primary after a partition, it sends a command to Node A's hardware power management unit (IPMI, iLO, or a cloud API like ec2 stop-instances) to physically power off Node A. This is brutal but effective — it guarantees with 100% certainty that Node A cannot cause split-brain, because Node A is literally turned off.

STONITH is commonly used in Pacemaker/Corosync HA clusters for databases. The tradeoff: it requires out-of-band management access (IPMI network, cloud APIs) and can cause brief unavailability while the fenced node reboots. But when data integrity is non-negotiable (financial systems, medical records), the brief downtime is far preferable to data divergence.

How Raft prevents split-brain entirely: In a properly configured Raft cluster, split-brain cannot occur. Node A is alone on its side of the partition — it is 1 out of 3 nodes, which is not a majority. It cannot win an election and cannot commit any writes (it can't get ACKs from a majority). Nodes B and C are 2 out of 3 — they can form a majority, elect B as leader, and continue serving writes. When the partition heals, Node A discovers that Term has advanced, steps down, and syncs its log from the new leader. No conflicting writes, no data loss, no manual intervention needed.

Replica Promotion Data Loss (Async Replication)

What happened: A primary crashed and the most up-to-date replica was promoted. However, the promoted replica was 3 seconds behind the primary. Those 3 seconds of writes (approximately 15,000 transactions) were permanently lost. This pattern has been documented in PostgreSQL failover incidents.

Root cause: Asynchronous replication by definition allows the primary to be ahead of all replicas.

Mitigation: Use semi-synchronous replication where at least one replica must acknowledge each write. Accept the latency cost. For critical data, consider synchronous replication or Raft-based consensus where writes are only acknowledged after a majority of replicas persist them.

Replication Slot Bloat in PostgreSQL

What happened: A replica went offline for maintenance. The primary's replication slot prevented WAL files from being recycled, causing the primary's disk to fill up over several hours. When the disk was full, the primary could no longer accept writes, causing a full outage.

Root cause: PostgreSQL replication slots guarantee that WAL segments are retained until the replica consumes them. A disconnected replica causes unbounded WAL retention.

Mitigation: Set max_slot_wal_keep_size (PostgreSQL 13+) to cap WAL retention. Monitor WAL directory size. Alert when a replication slot's lag exceeds a threshold. Remove stale slots proactively.

Cascading Replica Failure

What happened: One replica fell behind due to a slow query and was removed from the read pool by the health checker. Its traffic was redistributed to the remaining replicas, which became overloaded and also fell behind. Within minutes, all replicas were removed from the pool, and all read traffic hit the primary, which then became overloaded too.

Root cause: Load-balancing logic that removes lagging replicas without accounting for the capacity reduction.

Mitigation: Implement circuit breakers that keep a minimum number of replicas in the pool even if lagging. Prefer serving slightly stale data over cascading failure. Set a lag threshold that is generous enough to avoid false positives during normal spikes.

Debug Playbook

Is replication lag increasing?

  Yes → Check replica CPU and disk I/O. Is the replica resource-constrained?

    Yes → Scale up replica instance or add IOPS. Consider moving to a dedicated disk.

    No → Check for long-running transactions on the primary that generate large WAL volumes.

  No, lag is stable but high → Check network bandwidth between primary and replica. Is it saturated?

Is lag sudden (spike)?

  Yes → Look for bulk operations (batch inserts, schema migrations, VACUUM) on the primary. These generate massive WAL bursts. Consider throttling or scheduling them during low-traffic windows.

Is a replica disconnected?

  Check pg_stat_replication (PostgreSQL) or SHOW SLAVE STATUS (MySQL). Verify network connectivity. Check if the WAL segment the replica needs has been recycled (if not using replication slots).

Interactions & Pitfalls

Replication + Caching

A common cache-aside architecture: writes go to the primary and the application invalidates the cache key. Reads check the cache first; on a miss, the app reads from a replica and fills the cache. The invalidation itself is correct — the bug is the refill:

T=0     Write X=2 to primary. App invalidates cache key X.        OK
T=1ms   Read request -> cache miss -> fall back to a replica.
        Replica still has X=1 (replication lag).
        Cache filled with X=1.                                     STALE
T=50ms  Replica catches up to X=2. But cache is poisoned with X=1.
        Reads now serve X=1 until the next write to X (hours?).

So invalidation worked perfectly; the next reader re-poisoned the cache by refilling from a lagging replica. The two fixes target the refill, not the invalidation:

  • Delayed re-invalidation: invalidate at write time and again after expected replication lag (e.g., 5s later). The second invalidation evicts any stale value that was refilled from a lagging replica during the window.
  • Refill from primary: for a short window after an invalidation, route cache misses for that key to the primary. The primary is always current, so the refill is correct.

Why not write-through (update the cache with the new value at write time, instead of invalidating)? It avoids the lag race but has its own problems: concurrent writes can race inside the cache and leave the wrong value, many writes don't have a clean "value to cache" (partial updates, multi-row transactions), and it couples the write path to the cache. Cache-aside with one of the two fixes above is the common pattern.

Replication + Sharding

In a sharded database, each shard is an independent database that needs its own replication setup. If you have 8 shards with 3 replicas each, you are managing 32 database instances (8 primaries + 24 replicas). Each shard's replication lag must be monitored independently. Shard-level failover is more complex because promoting a replica for one shard must not affect the others. Systems like CockroachDB and Vitess abstract this complexity:

  • CockroachDB exposes a single SQL endpoint and internally splits data into ~512 MB ranges, each its own Raft group with 3 (or 5) replicas. Range placement, leader election, splitting/merging, and rebalancing across nodes are automatic. The operator runs a cluster, not 32 independent primaries — there is no per-shard primary to fail over because every range elects its own leader.
  • Vitess sits in front of MySQL shards with a query router (VTGate) and a topology service. It presents a single MySQL endpoint and manages shard routing (vschema), replication topology, failover, and resharding workflows under the hood. The application speaks plain MySQL; Vitess handles the 32-instance bookkeeping.

Replication + Load Balancing

A load balancer in front of replicas must understand the read/write split: writes go to the primary, reads go to replicas. If the load balancer is not replication-aware, it might send writes to a replica (which rejects them) or send reads to the primary (defeating the purpose of replicas). Connection poolers like PgBouncer or ProxySQL can enforce this split. Additionally, the load balancer should health-check replicas for replication lag, not just TCP connectivity — a replica that is connected but hours behind is worse than useless for consistent reads.

Cost Considerations

Cost Breakdown

  • Storage: N replicas = N times the storage cost. A 500 GB database with 3 replicas uses 2 TB total. At cloud SSD pricing (~$0.10/GB/month), that is $200/month just for disk.
  • Compute: Replicas that only serve reads can often run on smaller instances than the primary. A primary on an r6g.2xlarge ($0.50/hr) might have replicas on r6g.xlarge ($0.25/hr). Three read replicas: ~$540/month.
  • Cross-region bandwidth: Geo-replicated setups incur data transfer costs. AWS charges $0.02/GB for inter-region transfer. At 5 MB/s replication throughput: 5 MB/s x 86,400 s/day x 30 days = ~13 TB/month = ~$260/month per cross-region replica.
  • Dominant cost driver: For most setups, compute dominates. Storage is secondary. Cross-region bandwidth only matters for high-throughput geo-replicated systems.
  • Cost of NOT replicating: Without replication, a primary failure means downtime (lost revenue) and potential data loss (if backups are hours old). A single hour of downtime for a mid-sized e-commerce site can cost $10,000-$100,000+ — far exceeding the monthly cost of running replicas.

Real-World Examples

PostgreSQL Streaming Replication at Instagram

Instagram runs one of the largest PostgreSQL deployments in the world. Their architecture uses primary-replica streaming replication with dozens of read replicas per shard. They built custom tooling to monitor replication lag per replica and automatically remove lagging replicas from the read pool. For critical reads (like checking if a username is taken during registration), they route directly to the primary to avoid stale reads.

MySQL Replication at GitHub

GitHub uses MySQL with a single primary and multiple replicas per cluster. They developed Orchestrator, an open-source tool for MySQL replication topology management and automated failover. Orchestrator detects primary failures and promotes the most up-to-date replica within seconds. GitHub's engineering blog has documented incidents where replication lag caused visible inconsistencies in pull request comments and issue counts, which they addressed with read-your-own-writes routing at the application layer.

CockroachDB Raft-Based Replication

CockroachDB divides data into 512 MB ranges, each replicated using Raft consensus with a default replication factor of 3. Each range has its own Raft group with an elected leader. Writes require a majority (2 of 3) to acknowledge before being committed. This provides strong consistency without a single point of failure. CockroachDB handles leader re-election transparently — if a node fails, the affected ranges elect new leaders in about 1-3 seconds. DoorDash and Netflix use CockroachDB for workloads requiring multi-region strong consistency.

Code Example

A Python simulation of primary-replica async replication with read-your-own-writes consistency using session tokens. This illustrates the core pattern — not production code.

import time, threading, uuid

class ReplicatedDB:
    def __init__(self, num_replicas=2, lag_seconds=0.5):
        self.primary = {}          # The source of truth
        self.replicas = [{} for _ in range(num_replicas)]
        self.write_log = []        # Simulated WAL
        self.lag = lag_seconds
        self.write_tokens = {}     # session_id -> last_write_timestamp
        self._start_replication()

    def write(self, key, value, session_id=None):
        self.primary[key] = value
        entry = {"key": key, "value": value, "ts": time.time()}
        self.write_log.append(entry)
        if session_id:
            self.write_tokens[session_id] = entry["ts"]
        return entry["ts"]

    def read(self, key, session_id=None):
        # Read-your-own-writes: if this session just wrote,
        # read from primary to guarantee consistency
        if session_id and session_id in self.write_tokens:
            if time.time() - self.write_tokens[session_id] < self.lag * 2:
                return self.primary.get(key)  # Route to primary
        # Otherwise, read from a replica (round-robin)
        replica = self.replicas[hash(key) % len(self.replicas)]
        return replica.get(key)

    def _start_replication(self):
        def replicate():
            applied = 0
            while True:
                time.sleep(self.lag)
                while applied < len(self.write_log):
                    entry = self.write_log[applied]
                    for replica in self.replicas:
                        replica[entry["key"]] = entry["value"]
                    applied += 1
        t = threading.Thread(target=replicate, daemon=True)
        t.start()

# Usage
db = ReplicatedDB(num_replicas=2, lag_seconds=0.5)
session = str(uuid.uuid4())
db.write("user:42", {"name": "Alice"}, session_id=session)
# Immediate read with session token -> reads from primary
print(db.read("user:42", session_id=session))  # {'name': 'Alice'}

How to Validate Your Design

Load Testing

Use pgbench (for PostgreSQL) or sysbench (for MySQL) to generate write traffic at your expected peak rate. Monitor replication lag on all replicas during the test. Gradually increase write throughput until replicas start falling behind — this tells you the maximum sustainable write rate for your replication setup. For application-level testing, use k6 or Locust to simulate realistic read/write mixes.

Chaos Testing: Failover

Test failover by killing the primary process (not a graceful shutdown — use kill -9 or terminate the VM). Measure: How long until a replica is promoted? How many writes are lost (if using async replication)? Does your application reconnect automatically? Use toxiproxy to inject network latency between primary and replica to simulate cross-region replication and test how your system behaves under degraded network conditions.

Benchmarking Methodology

When benchmarking replication, key variables to sweep are: number of replicas, synchronous vs. asynchronous mode, write payload size, and concurrent writer count. Common mistakes include benchmarking with an empty database (no I/O contention), using uniform key distribution (real workloads are skewed), and not running long enough to observe WAL file rotation and checkpointing effects. Run benchmarks for at least 30 minutes to capture steady-state behavior.

Interview Questions

How would you design replication for a global chat application?

A global chat application needs low-latency reads and writes across regions. I would use multi-leader replication with one leader per region. Each region's leader accepts writes from local users, and changes are asynchronously replicated to other regions. For conflict resolution (two users editing the same message simultaneously), I would use last-writer-wins with vector clocks for ordering, or CRDTs for data types that support them (e.g., a grow-only set for message delivery receipts).

For message ordering within a conversation, I would use a logical clock (Lamport timestamp) assigned at the leader that first receives the message, ensuring all regions display messages in the same order. Cross-region replication lag (50-200ms) is acceptable for chat — users tolerate slight delays in seeing messages from other continents.

Each region would have 3 replicas using Raft consensus for local durability and fast failover. A global metadata service (also Raft-replicated) tracks which conversations are active in which regions to optimize replication topology.

Follow-up: How would you handle a user who sends a message from one region and then immediately switches to another region? How do you ensure they see their own message?

Explain read-your-own-writes consistency and how to achieve it.

Read-your-own-writes consistency guarantees that after a client performs a write, any subsequent read by that same client will reflect the write. Other clients may still see stale data (this is weaker than linearizability). Without this guarantee, a user might update their profile picture and then see the old one on refresh, creating a confusing experience.

There are several implementation strategies. Session-based routing: After a write, tag the session with the write's timestamp or log position. On subsequent reads, if the closest replica hasn't reached that position yet, route the read to the primary instead. Sticky sessions: Always route a given user's reads to the same replica (or the primary) for a short window after a write. Causal consistency: Track causal dependencies — a read depends on a prior write, so the system ensures the read only executes on a replica that has applied that write.

In practice, most systems use a combination: route reads to the primary for 2-5 seconds after a write (covering the expected replication lag), then fall back to replicas.

Follow-up: What about cross-device consistency — the user writes on their phone and reads on their laptop? Session-based approaches break here.

When would you use synchronous vs. asynchronous replication?

Use synchronous replication when data loss is completely unacceptable — financial transactions, medical records, or any system where losing even a single committed write has severe consequences. Synchronous replication guarantees that if the primary acknowledges a write, at least one replica has a durable copy. The cost is higher write latency (1-10ms same region, 50-200ms cross-region) and reduced availability (if all synchronous replicas are down, the primary must block writes).

Use asynchronous replication when you can tolerate small amounts of data loss in exchange for lower write latency and higher availability. This is appropriate for most web applications — social media posts, analytics events, user activity logs. A few seconds of data loss during a rare failover is an acceptable tradeoff for consistently fast writes.

Semi-synchronous is the practical middle ground used by most production systems: require one replica to acknowledge synchronously (ensuring at least one backup), but allow the rest to replicate asynchronously. MySQL and PostgreSQL both support this mode. If the synchronous replica goes down, the system can temporarily fall back to asynchronous rather than blocking writes.

Follow-up: If you choose synchronous replication for a financial application, how do you handle the latency impact on user-facing transactions?

How does Raft consensus work at a high level?

Raft is a consensus algorithm that ensures a group of servers agree on a sequence of operations, even if some servers fail. It decomposes the problem into three sub-problems:

1. Leader election: One node is elected leader for a given term (a monotonically increasing number). A node becomes a candidate if it hasn't heard from the leader within a randomized timeout. It requests votes from other nodes; if it gets a majority, it becomes leader for that term. The randomized timeout prevents split votes.

2. Log replication: The leader accepts client writes and appends them to its log. It sends these log entries to all followers. Once a majority of nodes have persisted an entry, the leader considers it committed and applies it to its state machine. Followers apply committed entries to their own state machines in the same order.

3. Safety: Raft guarantees that if an entry is committed, it will be present in the log of every future leader. This is enforced by the voting rule: a candidate can only win an election if its log is at least as up-to-date as the logs of a majority of nodes.

Compared to Paxos, Raft is easier to understand because it separates these concerns cleanly. Paxos describes a single round of consensus and requires significant engineering to build a practical replicated log on top of it. Raft provides the full replicated log protocol directly.

Follow-up: What happens during a network partition in a 5-node Raft cluster? Can both sides accept writes?

What are CRDTs and when would you use them?

Conflict-free Replicated Data Types (CRDTs) are data structures that can be independently updated on different replicas and merged without conflicts. They achieve this by constraining operations to be commutative, associative, and idempotent — meaning the order of operations doesn't matter, and applying the same operation twice has no additional effect.

Common CRDTs include: G-Counter (grow-only counter — each node tracks its own increment count; the total is the sum across all nodes), PN-Counter (supports both increment and decrement), G-Set (grow-only set — elements can be added but never removed), OR-Set (observed-remove set — elements can be added and removed), and LWW-Register (last-writer-wins register — each update carries a timestamp, and the latest wins).

Use CRDTs when you need multi-leader writes without coordination — distributed counters (like counts, upvotes), collaborative editing (like Google Docs, which uses a related approach), or shopping cart merges. They are a poor fit when you need strong consistency (CRDTs are eventually consistent) or when the data model doesn't map to available CRDT types (arbitrary SQL queries, for example).

Riak uses CRDTs extensively for its distributed data types. Redis Enterprise uses CRDTs for its active-active geo-distributed deployment (CRDB). The tradeoff is higher storage overhead (metadata for tracking versions/causality) and the constraint that not every operation can be expressed as a CRDT.

Follow-up: How would you implement a collaborative text editor using CRDTs? What CRDT would you choose?

How to handle split-brain in a multi-leader setup?

Split-brain occurs when a network partition causes two or more nodes to believe they are the primary, both accepting writes independently. When the partition heals, the system has divergent data that cannot be trivially reconciled.

Prevention: The best approach is to prevent split-brain entirely using quorum-based consensus. In a 3-node Raft cluster, a leader needs votes from 2 nodes to win an election. During a partition, only the side with 2+ nodes can elect a leader — the minority side is guaranteed to be read-only. Fencing tokens add another layer: when a new leader is elected, it gets a monotonically increasing token, and storage systems reject requests from stale leaders with older tokens.

Detection: If prevention fails (e.g., in a system without consensus), detect split-brain by monitoring for multiple nodes reporting as primary. Use an external arbitrator (a witness node in a third availability zone) to break ties.

Resolution: If split-brain does occur, you must merge the divergent data. Strategies include: last-writer-wins (simple but may lose data), application-level merge (custom conflict resolution logic — complex but preserves all writes), and CRDT-based merge (automatic but limited to supported data types). In practice, most teams choose to accept the small data loss risk of last-writer-wins rather than build custom merge logic.

Follow-up: Can you design a system where split-brain is impossible? What are the tradeoffs?

Key Takeaways

  • Replication is about copies — keeping the same data on multiple machines for availability, read scaling, and geographic proximity.
  • Synchronous vs. asynchronous is the fundamental tradeoff: consistency and durability vs. latency and availability.
  • Replication does not help write throughput. All writes go to the primary (or leader). To scale writes, you need sharding.
  • Replication lag is inevitable with async replication. Design your application to handle it — especially read-your-own-writes consistency.
  • Raft has become the standard for consensus-based replication, replacing Paxos in most new systems due to its understandability.
  • CRDTs enable multi-leader without coordination but are limited to specific data structures.
  • Failover is where things break. Test it regularly. Measure data loss during promotion. Automate it with tools like Orchestrator or Patroni.
  • Monitor replication lag obsessively. It is the single most important metric in a replicated database system.
