Time Complexity (for LLM serving)

The data-structure choices that decide whether a KV-cache block manager, scheduler, or router stays O(log n) per request or silently degrades to O(n) under load. Every row here is a question an interviewer can ask "which structure, and what does it cost?" — and the wrong answer is usually a sorted array.

~10 min read • data-structure reference

Why This Page Exists

TL;DR: The hot loops of an LLM serving stack are mostly "pick the best/worst item out of n, then update one item's priority" — evict the coldest KV-cache block, admit the highest-priority waiting request, find the least-loaded replica. Get the structure right and each operation is O(log n) or O(1); get it wrong (a sorted array, a linear scan) and you pay O(n) per token, per step, per request.

  • n is not small. A single 80 GB GPU holds tens of thousands of KV-cache blocks; a router fronts thousands of in-flight requests. An O(n) per-token operation inside the decode loop is a latency bug that only shows up under production load.
  • The interview tell. “How do you evict the coldest block?” is really “do you reach for a sorted array (wrong) or a heap/LRU (right), and can you state the cost of each operation?”

Eviction Priority Queue: “Find the Lowest-Score Block”

The setup: a KV-cache block manager keeps n blocks, each with a score (recency, hit count, or a composite priority). On every allocation that runs out of free blocks it must (1) find the lowest-score block and (2) evict it; meanwhile scores keep changing as blocks are touched. The naive instinct — “keep them sorted by score” — is the trap.

Why a sorted container is the wrong answer

  • Array-backed list (e.g. a Python list): O(n) overall. Finding the insert position via binary search is O(log n), but inserting shifts every later element — O(n), and that shift dominates.
  • Linked list: also O(n) overall. The insertion itself is O(1) once you hold the node, but you cannot binary search a linked list (no random access), so finding the spot is an O(n) linear scan.
  • The fix: don't keep the items fully sorted. Use a min-heap (what a real priority-based block evictor does) or a similar O(log n) structure. A heap keeps only the minimum cheap to find, not the whole order — which is all eviction needs.

Plain Min-Heap Operations

A binary min-heap is just an array where parent(i) ≤ children(i). The minimum is always at the root (index 0); every other operation is a logarithmic walk up or down the tree's height (log n).

OpCostWhy
InsertO(log n)Append to the end, then sift up while smaller than parent
Pop-minO(log n)Swap root with last element, drop the last, then sift down the new root
Peek-minO(1)Read the root — no mutation
Find arbitraryO(n)The heap is only partially ordered, so locating a non-root item means scanning the array
Update arbitraryO(n)O(n) to find it + O(log n) to re-sift — the find dominates

The catch for eviction: KV-cache scores change constantly (every block touch bumps recency). A plain heap makes "a block I already hold just got hotter, fix its position" an O(n) operation — because you first have to find it. That find is the bottleneck, and it's what the indexed heap removes.

Indexed Min-Heap (Heap + Hashmap)

The one trick: alongside the heap array, maintain a side hashmap block_id → heap_array_index. Every swap during sift-up/sift-down updates two entries in that map, so you always know where any block lives in the array. The O(n) “find arbitrary” scan becomes an O(1) map lookup — which collapses update and delete from O(n) to O(log n).

The extra map costs O(n) memory and a constant-factor write on every swap — cheap insurance against an O(n) per-touch hot loop.

LRU — Least Recently Used (all O(1))

Implementation: one hashmap (key → node) + one doubly linked list (DLL) ordered by recency, most-recent at the head. The two structures cover each other's weakness: the hashmap gives O(1) lookup, the DLL gives O(1) splice.

  • Add (put): O(1) — hashmap insert + push the node to the head of the DLL.
  • Get / access: O(1) — hashmap lookup, then move the node to the head.
  • Remove: O(1) — hashmap lookup gets the node, then unlink it from the DLL. No scan needed: DLL nodes carry prev/next pointers.
  • Eviction: O(1) — the tail node is the least-recently-used; pop it and delete its hashmap entry.

LRU is the default policy for prefix-cache block reuse in LLM serving: a block touched by a recent prompt is likely to be reused by the next similar prompt, and the tail is exactly what's safe to drop.

LFU — Least Frequently Used (all O(1))

Implementation: a key → node hashmap, a freq → DLL hashmap (one DLL per frequency value), a minFreq integer, and a capacity integer. Each node stores key, value, freq (its current count), plus prev/next pointers for its DLL. Ties within a frequency are broken by LRU (the DLL order inside each bucket).

How the structures interact

  • On get: look up the node via the key→node map → read its freq f → unlink from the freq[f] DLL → append to the freq[f+1] DLL → if freq[f] is now empty and f == minFreq, increment minFreq.
  • On evict: go to the freq[minFreq] DLL → pop its tail node (the least-recently-used among the least-frequent) → delete its entry from the key→node map.

Costs

  • Add (put): O(1) — insert the node into the frequency-1 bucket, update the key→node map, set minFreq = 1.
  • Get / access: O(1) — look up the node, move it from bucket f to bucket f+1, bump minFreq if the old bucket emptied.
  • Remove (eviction): O(1) — minFreq points straight at the least-frequent bucket; evict its tail and delete from the maps.

Caveat: the O(1) design above (key→node map + freq→DLL map + minFreq counter) is the standard interview/production answer. A naive LFU built on a heap or a sorted-by-count list is O(log n) per op — if you reach for a heap here, say why the bucketed version beats it.

Other Complexities Worth Knowing for LLM Serving

The same “which structure, what's the cost” reflex shows up across the stack. These are the high-value ones interviewers probe.

1. KV cache vs. recompute — the reason the cache exists

  • Without a KV cache: generating each new token re-attends over all prior tokens and recomputes their keys/values. Step t costs O(t), so producing n tokens is O(n²) in attention work (and the per-step matmuls make naive recompute even worse).
  • With a KV cache: prior keys/values are stored, so step t only computes attention of the one new query against t cached entries — still O(t) reads, but no recompute of the prefix. The cache trades GPU memory (O(n) per sequence) for cutting redundant compute.
  • Prefill vs. decode: prefill attention over a prompt of length s is O(s²) (every token attends to every earlier token, done in one parallel pass); each decode step is O(s) against the cache. This is why prefill is compute-bound and decode is memory-bandwidth-bound. See K2 (KV cache).

2. Prefix-cache lookup — “has this prompt prefix been seen?”

  • Radix tree (RadixAttention): longest-prefix match is O(L) in the query length L (in blocks), independent of how many prompts are cached. A hash map cannot do longest-prefix match; this is why prefix caches use a trie. See Radix Tree.
  • Block-hash chain: hashing prompt blocks and looking each up in a hash → block map is O(1) per block, O(L) for the whole prefix — the common production shortcut when you only need exact-block reuse, not arbitrary-position matching.

3. Paged KV allocation — the block table

  • Allocate / free a block: O(1) — pull from / push to a free-list stack. PagedAttention's whole point is that physical blocks are interchangeable, so allocation never needs a search or compaction.
  • Logical→physical translation: O(1) per block via the per-sequence block table (an array indexed by logical block number).

4. Scheduler / continuous batching admission

  • Plain FIFO waiting queue: dequeue next request is O(1).
  • Priority scheduling (deadline, fairness, SLO tier): a min-heap over W waiting requests gives O(log W) admit/insert — the eviction-queue argument above applies verbatim, don't sort.
  • Per-step batch assembly: O(B) in the running batch size B to gather the active sequences each decode step.

5. Routing to a replica

  • Consistent hashing (cache-affinity routing, so a tenant's prompts hit the replica that already holds their prefix): O(log T) to find the owning replica via binary search over T points on the ring. See Consistent Hashing.
  • Least-loaded / power-of-two-choices: sampling 2 replicas and picking the lighter is O(1) and avoids the O(T) scan of checking every replica's load.
  • Bloom filter (“does this replica hold tenant X's prefix?” gossip): O(k) membership test for k hash functions — constant, independent of set size. See Bloom Filters.

6. Sampling the next token over vocabulary V

  • Greedy / argmax: O(V) single pass over the logits.
  • Top-k: O(V) with a size-k min-heap (or O(V + k log k) with partial selection) — you don't sort the whole vocabulary, you keep only the k largest.
  • Top-p (nucleus): needs the sorted tail, so O(V log V) in the worst case, though kernels prune aggressively in practice.

Trie vs. Radix Tree — Prefix Structures in Detail

TL;DR: both answer “is this sequence (or a prefix of it) already stored?” in O(L) — independent of how many keys are stored — which is exactly what a prefix cache needs and what a hash map cannot do (a hash map only matches whole keys, not longest prefixes). They differ in space: a trie spends one node per symbol; a radix tree collapses every non-branching run of symbols into a single edge, so its node count tracks the number of keys, not their length. For LLM prefix caches (RadixAttention/SGLang) the “symbols” are KV-block hashes and keys are long, so that collapse is the whole reason the radix tree wins.

Notation

  • L — length of the query/key in symbols (tokens, or KV-cache blocks for a prefix cache).
  • N — number of distinct keys currently stored.
  • Σℓ — total symbols across all stored keys (sum of key lengths).
  • σ — branching factor / alphabet size: how many distinct children a node can have (26 for lowercase text; effectively unbounded for block-hash keys).
  • f — the per-step child-dispatch cost (how you pick the next child), set by the children representation — see the table below.

Same five keys, two shapes

Insert {test, team, toast}. The trie spends a node on every character; the radix tree keeps a node only at the root, at branch points, and at leaves — the runs s‑t, a‑m, o‑a‑s‑t each collapse to one edge. 10 nodes vs. 5.

  • ◯Plain node — internal / branch point; tiny circle at top is the root.
  • ◉Blue node — end of a stored key (test, team, toast).
  • e/oastEdge label — in a radix tree an edge carries a whole substring, not one character.

Time complexity (identical asymptotically)

Both walk at most L symbols from the root, so every operation is O(L) — crucially not a function of N. The radix tree touches fewer nodes on the path (only branch points), but it must compare an edge's whole substring, and those substring lengths sum to ≤ L — so the symbol count, and the bound, is the same.

The per-step factor f — how children are stored

“O(L)” hides the cost of choosing the next child at each of the L steps. That cost is a design choice, and the same choice sets the per-node memory:

Space complexity (where they diverge)

The headline: a trie's node count scales with the total length of all keys (Σℓ); a radix tree's node count scales with the number of keys (N). A compressed trie holding N keys has at most 2N−1 nodes (≤ N leaves + ≤ N−1 branch points), because every internal node has ≥ 2 children. The characters themselves move from nodes onto edge labels, totalling O(Σℓ) stored symbols either way — but the radix tree stops paying a node (with its children container, pointers, and bookkeeping) for every single symbol.

MetricTrieRadix tree
Node countO(Σℓ) — up to one per character (minus shared prefixes)O(N) — ≤ 2N−1, key-count bound

Per-node overheadpaid Σℓ timespaid O(N) times
Worst-case total (array children)O(Σℓ·σ)O(N·σ + Σℓ)

Why a prefix cache picks the radix tree. In RadixAttention/SGLang the “symbols” are KV-cache block hashes (so σ is effectively unbounded → array children are out, hash-map children are mandatory) and keys are long prompts (large L). A plain trie would allocate one node — a child map plus pointers — per block on every non-shared run; across thousands of long prompts that is millions of nodes. The radix tree collapses each unbranched run of blocks into a single edge, so node count tracks the number of distinct prompt shapes, not total blocks cached, while keeping the same O(L) longest-prefix match. A hash map is still out for arbitrary-position matching: it matches whole keys, never longest prefixes — but a chained hash sidesteps that, which is the third option below. Full walk-through on the Radix Tree page.

The third option: block-hash chaining (the flat, production shortcut)

Problem: a tree (trie/radix) is awkward to distribute — sharding a pointer-linked structure across a fleet of KV-store tasks and keeping it consistent is real work. Solution: drop the tree and use a flat block_hash → physical_block hash map, where each block's hash is chained over all preceding blocks. The chaining is the trick that lets a plain hash map do prefix matching after all.

How the chained hash works (worked example)

Tokenize the prompt and pack it into fixed-size KV blocks B₀, B₁, B₂, … (say 16 tokens each). The hash of block i folds in the hash of block i−1, so it encodes the entire prefix up to and including block i, not just block i's own tokens:

  • h₀ = H(B₀)
  • h₁ = H(h₀, B₁)
  • h₂ = H(h₁, B₂)  … and so on.

Two prompts that share the first two blocks produce identical h₀, h₁ and diverge at h₂ — so equal hashes are a shared prefix. That is what makes a flat map able to answer “longest cached prefix” that an unchained per-block hash could not: each key already carries its position and history.

OperationCostWhy
Compute prefix hashes (L blocks)O(L)One chained fold per block, O(1) each (block size is fixed)
Longest cached prefixO(L)Probe h₀, h₁, … in the map, stop at the first miss — O(1) avg per probe, independent of N
Insert a blockO(1)One map put: hₙ → physical block
Collision verify (rare)O(block size)On a hash hit, compare the block's tokens to rule out a false match (needs a strong/wide hash)
Block-hash chain vs. radix tree — same O(L) time, different shape
  • Time: a tie. Both are O(L) longest-prefix match, independent of the number of cached keys N. The chain trades the tree's pointer-walk for a per-block hash computation + flat-map probe.
  • Space: O(M) flat entries (M = distinct cached blocks) with no tree pointers or child containers — a flatter constant factor than the radix tree's O(N) nodes, though it loses the radix tree's prefix de-duplication of the keys themselves.
  • Granularity: block-level only (matches at fixed block boundaries), vs. the radix tree's arbitrary token-level prefixes. Fine here — KV reuse is block-granular anyway.
  • Distribution is the real win: a block hash is a flat, routable key, so you shard the map across a KV-store fleet via consistent hashing and gossip hashes between tasks — far easier than sharding a tree. This is why production exact-reuse caches (vLLM-style automatic prefix caching, and this curriculum's distributed KV cache) chain hashes instead of maintaining a distributed radix tree.
  • What you give up: the tree's structured, subtree-aware eviction and child enumeration. A flat map evicts per block (LRU/LFU over blocks — see above) rather than per subtree.

Key Takeaways

  • Never keep a hot collection fully sorted. A sorted array is O(n) per insert (tail shift); a sorted linked list is O(n) per insert (no binary search). Eviction only needs the extreme, so use a heap.
  • A plain min-heap is O(log n) for insert/pop but O(n) to find or update an arbitrary item — the killer when scores change in place.
  • Add a key → heap-index hashmap to get an indexed heap: find drops to O(1), update/delete to O(log n).
  • LRU and LFU are both achievable at O(1) by pairing a hashmap with doubly linked list(s) — one DLL for LRU, one DLL per frequency (plus a minFreq counter) for LFU. The naive heap-based LFU is only O(log n).
  • Trie and radix tree share the same O(L) time (independent of the number of keys N) for lookup and longest-prefix match — the win a hash map can't give. They diverge on space: a trie spends O(Σℓ) nodes (one per character), a radix tree only O(N) (≤ 2N−1, by compressing non-branching runs onto edges) — which is why long-key prefix caches use the radix tree.
  • The KV cache itself is the headline complexity win: it turns O(n²) attention recompute over a generation into per-step O(t) cache reads, at the cost of O(n) memory per sequence.
  • Across routing, scheduling, prefix lookup, and sampling, the same reflex pays off: name the structure, state the cost, and call out the O(n) trap you avoided.
