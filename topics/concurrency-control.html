Concurrency Control

From OS mutexes to distributed locks, from spinlocks to CRDTs — how concurrency primitives you learned in operating systems map directly to distributed system coordination, and why getting them wrong causes some of the hardest bugs in production.

35 min read

What Is It?

Concurrency control is the set of mechanisms that ensure correct behavior when multiple actors — threads, processes, services, or users — access shared resources simultaneously. Without it, concurrent access produces race conditions, lost updates, torn reads, and data corruption. In a single machine, the OS provides primitives like mutexes, semaphores, and read-write locks. In a distributed system, these same coordination problems exist but are dramatically harder because of network partitions, clock skew, partial failures, and the impossibility of instantaneous global state.

The insight that makes this topic tractable is that every OS concurrency primitive has a direct distributed analogue. A mutex becomes a distributed lock (Redis SETNX, ZooKeeper ephemeral nodes, etcd leases). A semaphore becomes a distributed rate limiter or resource quota. A read-write lock becomes a shared/exclusive distributed lock. Compare-and-swap (CAS) becomes optimistic concurrency control with version vectors or ETags. Even the pathologies — deadlock, livelock, priority inversion — have distributed counterparts that manifest in production as cascading timeouts, circular service dependencies, and lock ordering bugs.

Understanding concurrency control means understanding the spectrum from pessimistic approaches (acquire a lock before touching the resource) to optimistic approaches (proceed without locking, detect conflicts, retry) to lock-free approaches (CRDTs, append-only logs). The right choice depends on contention level, read/write ratio, latency requirements, and failure tolerance. High-contention writes demand pessimistic locking. Low-contention reads with occasional writes favor optimistic concurrency control. Collaborative editing at scale requires CRDTs or operational transformation — locks simply cannot scale to thousands of concurrent editors.

How It Works

Concurrency control in distributed systems is built on six foundational primitives, each a direct analogue of an OS-level mechanism:

1. Mutex → Distributed Lock

An OS mutex guarantees mutual exclusion: only one thread can hold the lock at a time. The distributed analogue is a distributed lock, implemented via:

  • Redis SETNX + TTL: SET lock_key owner_id NX PX 30000 atomically sets the key only if it does not exist, with a 30-second expiry. The caller checks the return value to know if it acquired the lock. Release requires a Lua script that checks ownership before deleting: if redis.call("get",KEYS[1]) == ARGV[1] then return redis.call("del",KEYS[1]) end. The TTL acts as a safety net — if the holder crashes, the lock auto-expires.
  • ZooKeeper ephemeral nodes: The client creates an ephemeral znode at a known path. Only one create succeeds (ZooKeeper guarantees uniqueness within a directory). If the client dies, its session expires and the ephemeral node is automatically deleted, releasing the lock. Other clients set a watch on the node to be notified when it disappears.
  • etcd lease: Similar to ZooKeeper but using etcd's lease mechanism. The client creates a key with a lease (TTL). The client must periodically refresh the lease via a keep-alive. If the client crashes, the lease expires and the key is deleted.

Critical difference from OS mutexes: OS mutexes are reliable — if the holder dies, the OS releases the lock. Distributed locks rely on TTLs, which introduce a fundamental tension: too short a TTL and the lock expires while the holder is still working (due to GC pauses, network delays); too long a TTL and a crashed holder blocks everyone. This is why Martin Kleppmann argues that distributed locks need fencing tokens — monotonically increasing values that the lock service issues on each acquisition. The resource checks the fencing token and rejects stale requests.

Fencing Tokens: Why They Are Essential (Attack Scenario Trace)

Without fencing tokens, a GC pause can cause two clients to act on a shared resource simultaneously, leading to data corruption. Here is a step-by-step trace showing exactly how fencing tokens prevent this:

Time Event Client A Client B Storage
t=0s Lock service grants lock to Client A Holds lock, fence_token=33 Idle last_seen_token=32
t=1s Client A begins processing order Processing (fence_token=33) Idle last_seen_token=32
t=2s Client A hits a GC pause (stop-the-world) PAUSED (unaware, still believes it holds lock) Idle last_seen_token=32
t=12s Lock TTL expires (TTL was 10s). Client A is still paused. PAUSED (lock expired, does not know) Idle last_seen_token=32
t=13s Lock service grants lock to Client B PAUSED Holds lock, fence_token=34 last_seen_token=32
t=14s Client B writes to storage with fence_token=34 PAUSED Write accepted (34 > 32) last_seen_token=34
t=17s Client A resumes from GC pause (15s total) Resumes, tries to write with fence_token=33 Working normally last_seen_token=34
t=17s Storage rejects Client A's write: 33 < 34 Write REJECTED — stale token Working normally last_seen_token=34 (unchanged)

Without fencing tokens: At t=17s, Client A's write would succeed, overwriting Client B's valid data. Both clients would have modified the resource, violating mutual exclusion. The fencing token acts as a logical clock — even though Client A's wall clock shows it acquired the lock "first," the monotonically increasing token lets the storage server determine which lock grant is current.

2. Read-Write Lock → Distributed Read-Write Lock

An OS read-write lock allows multiple concurrent readers but only one writer (and no readers during a write). The distributed analogue uses a lock service that tracks lock mode:

  • Shared mode (read): Multiple clients can hold the lock simultaneously. Implemented by incrementing a counter (INCR readers_count) or creating individual ephemeral znodes under a shared directory.
  • Exclusive mode (write): Only one client can hold the lock. The writer must wait until all readers release. Implemented by checking that the reader count is zero and setting an exclusive flag atomically.

Read-write locks are critical for read-heavy workloads like configuration stores, metadata caches, and catalog services where reads outnumber writes 100:1 or more. Without them, a naive mutex serializes all reads, destroying throughput.

3. Semaphore → Distributed Semaphore

An OS semaphore allows up to N concurrent accessors. The distributed analogue limits concurrent access to a shared resource across the cluster:

  • Rate limiting: A distributed semaphore with N=1000 permits per second limits API calls across all service instances.
  • Connection pool limiting: A distributed semaphore with N=50 limits concurrent database connections across a fleet of application servers, preventing connection exhaustion.
  • Resource quotas: A distributed semaphore with N=10 limits concurrent batch jobs, preventing resource starvation.

Implementation typically uses an atomic counter in Redis (INCR/DECR with WATCH for transactions) or a set of ephemeral znodes in ZooKeeper (the number of znodes represents acquired permits; the semaphore is full when the count reaches N).

4. Condition Variable → Distributed Coordination

An OS condition variable lets a thread wait until a condition is met, signaled by another thread. The distributed analogues are:

  • ZooKeeper watches: A client sets a watch on a znode. When the znode changes, ZooKeeper sends a notification. This is the distributed equivalent of pthread_cond_wait / pthread_cond_signal.
  • etcd watch: Similar semantics — watch a key or key range for changes. The watch provides a revision number for exactly-once event processing.
  • Kafka consumer group rebalance: When a consumer joins or leaves the group, all consumers are notified and partitions are reassigned. This is a distributed condition variable where the condition is "group membership changed."

5. Spinlock → Retry Loops with Exponential Backoff

An OS spinlock busy-waits in a tight loop until the lock is available. This is efficient when contention is low and hold times are short (nanoseconds). In distributed systems, naive spinning is catastrophic:

  • Each "spin" is a network round-trip (1-10ms vs. nanoseconds)
  • Spinning generates massive load on the lock service
  • Under contention, spinning amplifies the problem — N clients spinning at 100 req/s each creates N*100 req/s on the lock service

The distributed equivalent is a retry loop with exponential backoff and jitter: wait 100ms, then 200ms, then 400ms, with random jitter to avoid thundering herd. This is fundamentally different from an OS spinlock — it trades latency for system stability.

6. Compare-and-Swap (CAS) → Optimistic Concurrency Control

An OS CAS instruction atomically compares a memory location to an expected value and, only if they match, swaps in a new value. The distributed analogues are:

  • Version vectors / ETags: Read the resource and its version. Modify locally. Write back with a conditional: "update only if the version is still V." If another client modified the resource in between, the version has changed and the write is rejected. The client re-reads and retries.
  • Conditional writes: DynamoDB ConditionExpression, Cosmos DB If-Match header, PostgreSQL UPDATE ... WHERE version = V.
  • CRDTs (Conflict-free Replicated Data Types): The ultimate optimistic approach. CRDTs are data structures designed so that concurrent updates always converge to the same state without coordination. G-Counters, PN-Counters, OR-Sets, and LWW-Registers are common examples. They eliminate the need for locks entirely at the cost of weaker semantics (eventual consistency, limited data types).

When to Use / When NOT to Use

Use Pessimistic Concurrency Control (Locks) When

  • High write contention: Many writers updating the same resource simultaneously. Optimistic approaches waste work through constant retries.
  • Critical correctness requirements: Financial transactions, inventory management where double-spending must be impossible. "Better to wait than to be wrong."
  • Short lock hold times: The operation completes in milliseconds. Lock overhead is proportionally small.
  • Sequential consistency required: Operations must appear to execute in a strict order. Locks enforce this naturally.
  • Single leader or master pattern: Only one node should perform a task (e.g., cron job, leader election, singleton service).

Use Optimistic Concurrency Control (Versioning/CAS) When

  • Low to moderate contention: Most operations do not conflict. Locking would add unnecessary latency and reduce throughput.
  • Read-heavy workloads: Many readers with occasional writers. Locks serialize readers unnecessarily.
  • Long-running operations: User editing a document for 5 minutes. Holding a lock that long is impractical — detect conflicts at save time instead.
  • Cross-datacenter replication: Locks across datacenters add 50-200ms per operation. OCC lets each datacenter operate independently and reconcile conflicts.
  • Collaborative editing: Multiple users editing the same document. CRDTs or OT (Operational Transformation) handle concurrent edits without locks.

Classic vs. Modern Approaches

Classic: Pessimistic Locking (SELECT FOR UPDATE)

The database itself manages locks. SELECT ... FOR UPDATE acquires a row-level exclusive lock that blocks other transactions from reading or writing the row until the transaction commits. This is the simplest and most widely understood approach. It works well for single-database systems with moderate contention. The downside: it holds database connections and locks for the duration of the transaction, which limits throughput under high concurrency. Long-running transactions can cause lock contention cascades.

Classic: Two-Phase Locking (2PL)

The standard protocol for serializable transactions. In the growing phase, the transaction acquires locks and never releases any. In the shrinking phase, it releases locks and never acquires new ones. Strict 2PL holds all locks until commit, guaranteeing serializability. The problem: 2PL is prone to deadlocks and reduces concurrency because locks are held for the entire transaction duration. This is why most databases default to weaker isolation levels.

Two-Phase Locking: Step-by-Step Deadlock Trace

To see why 2PL is deadlock-prone, consider a concrete example with three transactions transferring money between accounts. Each transaction must lock the source account first (to read the balance), then lock the destination account (to credit it). In the growing phase, locks are only acquired, never released.

Setup: Account A has $1000, Account B has $2000. T1 wants to transfer $100 from A to B. T2 wants to transfer $200 from B to A.

Step T1 (A → B) T2 (B → A) Lock Table Status
1 LOCK(A) — granted — A: T1 (exclusive) OK
2 — LOCK(B) — granted A: T1, B: T2 OK
3 READ A → balance=$1000 READ B → balance=$2000 A: T1, B: T2 OK
4 LOCK(B) — BLOCKED (T2 holds B) — A: T1, B: T2 T1 waiting
5 still waiting for B... LOCK(A) — BLOCKED (T1 holds A) A: T1, B: T2 DEADLOCK

Result: T1 holds A and waits for B. T2 holds B and waits for A. Neither can proceed — this is a deadlock. The wait-for graph has a cycle: T1 → T2 → T1.

Fix — Lock Ordering: Establish a rule: always acquire locks in alphabetical order (A before B). Under this rule:

Step T1 (A → B) T2 (B → A) Lock Table Status
1 LOCK(A) — granted LOCK(A) — BLOCKED (T1 holds A) A: T1 T2 waiting
2 LOCK(B) — granted still waiting for A... A: T1, B: T1 T2 waiting
3 READ A, WRITE A=$900, WRITE B=$2100 still waiting... A: T1, B: T1 T2 waiting
4 COMMIT → UNLOCK(A), UNLOCK(B) LOCK(A) — granted A: T2 OK
5 — LOCK(B) — granted A: T2, B: T2 OK
6 — READ B=$2100, WRITE B=$1900, WRITE A=$1100. COMMIT. (empty) OK

Because both transactions acquire A before B, a cycle can never form. T2 waits for T1 to finish, then proceeds. Lock ordering is simple, provably correct, and is the most common deadlock prevention strategy in practice.

Transaction Isolation Levels: The Four SQL Levels

Databases offer different isolation levels that control what anomalies concurrent transactions can observe. Stronger isolation prevents more anomalies but reduces concurrency (throughput). Understanding these levels is essential because most concurrency bugs in production stem from running at an isolation level that is too weak for the application's invariants.

Level 1: READ UNCOMMITTED (Dirty Reads Allowed)

This is the weakest isolation level. A transaction can read data written by another transaction before that transaction commits. If the writing transaction rolls back, the reader has used data that never officially existed — a dirty read.

Concrete example — dirty read:

Time T1 (Transfer) T2 (Balance check) Account Balance in DB
t1 BEGIN — $1000
t2 UPDATE balance = $500 (not yet committed) — $500 (uncommitted)
t3 — BEGIN $500 (uncommitted)
t4 — SELECT balance → reads $500 $500 (uncommitted)
t5 ROLLBACK (abort!) — $1000 (restored)
t6 — Uses $500 for a decision — WRONG! $1000

T2 read $500, but that value never actually committed. T2 made a decision based on phantom data. This level is almost never used in practice.

Level 2: READ COMMITTED (No Dirty Reads, but Non-Repeatable Reads)

A transaction can only read data from committed transactions. This prevents dirty reads. However, if a transaction reads the same row twice, it might get different values because another transaction committed a change between the two reads — a non-repeatable read.

Concrete example — non-repeatable read:

Time T1 (Report generation) T2 (Account update) Committed Balance
t1 BEGIN — $1000
t2 SELECT balance → reads $1000 — $1000
t3 — BEGIN $1000
t4 — UPDATE balance = $500; COMMIT $500
t5 SELECT balance → reads $500 — $500
t6 T1 sees two different values for the same row! — $500

T1 read the same account balance twice and got different results ($1000 then $500). This is problematic for reports that need a consistent view — the report might show inconsistent subtotals. READ COMMITTED is PostgreSQL's default and is sufficient for many workloads, but not for reports or invariant checks that read the same data multiple times.

Level 3: REPEATABLE READ (No Non-Repeatable Reads, but Phantom Reads)

The transaction sees a consistent snapshot of the data as of the transaction's start time. Re-reading the same row always returns the same value. However, new rows inserted by other transactions can appear — a phantom read.

Concrete example — phantom read:

Time T1 (Scheduling audit) T2 (New hire onboarding) active Doctors Table
t1 BEGIN — 10 rows (Dr. 1 – Dr. 10)
t2 SELECT COUNT(*) WHERE status='active' → 10 — 10 rows
t3 — BEGIN 10 rows
t4 — INSERT new doctor (status='active'); COMMIT 11 rows
t5 SELECT COUNT(*) WHERE status='active' → 11 — 11 rows
t6 T1 counted 10, then 11 — a new "phantom" row appeared — 11 rows

T1's count changed from 10 to 11 because T2 inserted a new row that matched T1's query predicate. The existing rows did not change (repeatable read guarantee), but a new row appeared. Note: PostgreSQL's REPEATABLE READ actually uses snapshot isolation, which prevents phantoms in most cases. The SQL standard's definition of REPEATABLE READ technically allows phantoms.

Level 4: SERIALIZABLE (All Anomalies Prevented)

The strongest isolation level. Transactions behave as if they executed one at a time, in some serial order. No dirty reads, no non-repeatable reads, no phantom reads, no write skew. The tradeoff is reduced throughput — transactions that would conflict are either blocked (under 2PL-based serializable) or aborted and retried (under SSI-based serializable).

Anomalies Prevented at Each Isolation Level

READ UNCOMMITTED Possible Possible Possible Possible Almost never used; some analytics on dirty data
READ COMMITTED Prevented Possible Possible Possible Default for PostgreSQL, Oracle; most OLTP workloads

Modern: Multi-Version Concurrency Control (MVCC)

Instead of locking, the database maintains multiple versions of each row. Readers see a consistent snapshot without blocking writers. Writers create a new version rather than modifying in place. This is how PostgreSQL, MySQL InnoDB, Oracle, and CockroachDB handle concurrency. MVCC enables high read concurrency because readers never block writers and writers never block readers. The tradeoff: more storage (old versions must be retained and garbage collected), and write-write conflicts still require detection.

MVCC Visibility Trace (PostgreSQL-Style)

In PostgreSQL, every row version has two hidden columns: xmin (the transaction ID that created this version) and xmax (the transaction ID that deleted or replaced this version; ∞ means "still alive"). To determine whether a row version is visible to a transaction, PostgreSQL applies visibility rules based on the transaction's snapshot.

Scenario: An accounts table has one row: Alice with balance=$1000, created by committed transaction T95.

Step Action Row Versions on Disk What T100 Sees
1 T100 starts (snapshot: all txns < 100 committed) v1: (xmin=95, xmax=∞, balance=$1000) v1 visible: xmin=95 is committed and < 100, xmax=∞ → $1000
2 T101 starts v1: (xmin=95, xmax=∞, balance=$1000) (T100 unchanged)
3 T101 executes: UPDATE accounts SET balance=500 WHERE name='Alice' v1: (xmin=95, xmax=101, balance=$1000)
v2: (xmin=101, xmax=∞, balance=$500)		 (T100 unchanged)
4 T101 commits v1: (xmin=95, xmax=101, balance=$1000)
v2: (xmin=101, xmax=∞, balance=$500)		 (T100 unchanged)
5 T100 reads Alice's balance again (same as step 4) v1: xmin=95 committed < 100 ✓, xmax=101 > 100 → v1 still visible. T100 sees $1000
6 T102 starts (snapshot: all txns ≤ 101 committed) (same as step 4) T102 checks: v1 has xmax=101, which is committed and ≤ 102 → v1 is dead. v2 has xmin=101, committed and ≤ 102, xmax=∞ → v2 visible. T102 sees $500

Key insight: T100 reads the row after T101 has committed the update, but still sees the old value ($1000). This is snapshot isolation at work — T100 sees a consistent snapshot as of when it started. T102, which started after T101 committed, sees the new value ($500). No locks were needed at any point.

Modern: Optimistic Concurrency Control (OCC)

Proceed without locks. Each transaction reads data and records what it read (the read set) and what it wants to write (the write set). At commit time, the system validates that no other transaction modified the read set. If validation passes, the transaction commits. If it fails, the transaction aborts and retries. OCC shines under low contention: no lock acquisition overhead, maximum parallelism. Under high contention, OCC degrades as transactions repeatedly abort and retry, wasting work.

Modern: CRDTs (Conflict-free Replicated Data Types)

CRDTs are mathematically guaranteed to converge to the same state regardless of the order in which updates are applied. This means replicas can accept writes independently, without coordination, and eventually agree. Common CRDTs include G-Counter (grow-only counter), PN-Counter (increment/decrement), OR-Set (observed-remove set), and LWW-Register (last-writer-wins register). Used in production by Redis (CRDTs in Redis Enterprise active-active), Riak, and collaborative editors like Figma. The limitation: CRDTs can only represent data structures with commutative, associative, and idempotent merge operations — not all business logic fits.

G-Counter Merge Example: 3 Nodes

A G-Counter (grow-only counter) is the simplest CRDT. Each node maintains a vector of counters, one entry per node. To increment, a node only increments its own entry. To get the total, sum all entries. To merge two replicas, take the element-wise maximum.

Setup: 3 nodes (A, B, C), each tracking page view counts. After some local increments:

  • Node A's state: [2, 0, 0] — A saw 2 page views locally
  • Node B's state: [0, 3, 0] — B saw 3 page views locally
  • Node C's state: [0, 0, 1] — C saw 1 page view locally

Merge trace (order 1: A merges with B, then with C):

Step Operation Result Total (sum)
1 merge(A, B) = max([2,0,0], [0,3,0]) [2, 3, 0] 5
2 merge([2,3,0], C) = max([2,3,0], [0,0,1]) [2, 3, 1] 6

Merge trace (order 2: A merges with C first, then with B):

Step Operation Result Total
1 merge(A, C) = max([2,0,0], [0,0,1]) [2, 0, 1] 3
2 merge([2,0,1], B) = max([2,0,1], [0,3,0]) [2, 3, 1] 6

Same result regardless of merge order. This works because max is:

  • Commutative: merge(A, B) = merge(B, A)
  • Associative: merge(merge(A, B), C) = merge(A, merge(B, C))
  • Idempotent: merge(A, A) = A (merging with yourself changes nothing)

These three properties guarantee that any merge order, even with duplicate messages, produces the same final state. This is what makes CRDTs "conflict-free."

PN-Counter: Supporting Decrements

A G-Counter can only grow. To support both increments and decrements, a PN-Counter uses two G-Counters: P (positive/increments) and N (negative/decrements). The actual value = sum(P) − sum(N).

Example: A "likes" counter that supports unlike (decrement):

  • P = [2, 3, 1] — total increments = 2 + 3 + 1 = 6
  • N = [1, 0, 0] — total decrements = 1 + 0 + 0 = 1
  • Value = 6 − 1 = 5

Each G-Counter (P and N) merges independently using element-wise max. Since both P and N converge, their difference also converges. Node A had 2 likes and 1 unlike, so its net contribution is 1. Node B had 3 likes and 0 unlikes. Node C had 1 like and 0 unlikes.

Modern: Serializable Snapshot Isolation (SSI)

SSI combines the performance of snapshot isolation with the correctness of serializability. It allows transactions to proceed on their snapshots without blocking, then detects serialization anomalies (like write skew) at commit time. CockroachDB and PostgreSQL (since version 9.1) implement SSI. It avoids the deadlocks of 2PL while providing the strongest isolation guarantee. The cost: some transactions are aborted when anomalies are detected, requiring retries.

Write Skew: The Hospital On-Call Problem (Detailed Trace)

Write skew is a concurrency anomaly that snapshot isolation (REPEATABLE READ) does not prevent. It occurs when two transactions read overlapping data, make disjoint writes (each writes to a different row), and together violate an invariant that neither violated individually.

Setup: A hospital requires at least 1 doctor on call at all times. Currently, Alice and Bob are both on call.

Time T1 (Alice wants to leave) T2 (Bob wants to leave) on_call Table State
t1 BEGIN (snapshot taken) BEGIN (snapshot taken) Alice=on_call, Bob=on_call (2 on call)
t2 SELECT COUNT(*) WHERE on_call=true → 2 — (unchanged)
t3 — SELECT COUNT(*) WHERE on_call=true → 2 (unchanged)
t4 2 ≥ 2, safe to leave. UPDATE SET on_call=false WHERE name='Alice' — Alice=off, Bob=on_call
t5 — 2 ≥ 2, safe to leave. UPDATE SET on_call=false WHERE name='Bob' Alice=off, Bob=off
t6 COMMIT ✓ COMMIT ✓ 0 doctors on call! INVARIANT VIOLATED

Why snapshot isolation allows this: Each transaction's snapshot showed 2 doctors on call. Each transaction wrote to a different row (T1 wrote Alice's row, T2 wrote Bob's row). There is no write-write conflict — snapshot isolation only detects conflicts when two transactions write the same row. So both commits succeed, and the invariant is violated.

How SSI detects and prevents this:

SSI tracks rw-dependencies (read-write dependencies). A rw-dependency from T1 to T2 exists when T2 writes a row that T1 read. SSI looks for cycles in the dependency graph — a cycle means the transactions cannot be serialized in any order, so one must be aborted.

Dependency Reason
T1 → T2 (rw-dependency) T1 read the on_call rows (including Bob's row). T2 wrote Bob's row.
T2 → T1 (rw-dependency) T2 read the on_call rows (including Alice's row). T1 wrote Alice's row.

Cycle detected: T1 → T2 → T1. SSI aborts one transaction (say T2) at commit time. Bob's request to go off-call fails and must be retried. On retry, T2 reads the count as 1 (Alice already left), sees it is not safe to leave, and the application correctly rejects the request. The invariant is preserved.

Assumptions & Model

Key Assumptions

  • Clocks are imperfect: Distributed locks rely on TTLs, which depend on clocks. Clock skew between nodes means a lock's TTL might expire earlier on one node than another. Google Spanner uses TrueTime (GPS + atomic clocks) to bound clock uncertainty to ~7ms. Most systems do not have this luxury and must design for unbounded clock skew.
  • Processes can pause arbitrarily: A GC pause, a context switch, or a VM migration can pause a process for seconds. During a pause, the process's distributed lock may expire, but the process still believes it holds the lock. This is the fundamental unsafety of distributed locks without fencing tokens.
  • The network is unreliable: Messages can be delayed, duplicated, reordered, or lost. A lock holder might be working correctly but appear dead because its heartbeats are delayed. The lock service expires the lock and grants it to another client. Now two clients believe they hold the lock — the "split-brain" problem.
  • Failures are partial: In an OS, if the machine dies, all threads die with it. In a distributed system, some nodes fail while others continue. A lock holder might crash while the resource it was protecting is still accessible to other nodes.

When These Assumptions Break

  • GC pauses violate lock safety: A Java process acquires a lock with a 30-second TTL, then enters a 40-second GC pause. The lock expires, another client acquires it and modifies the resource. The first client resumes, unaware it lost the lock, and overwrites the changes. Solution: fencing tokens, or design the system to tolerate stale lock holders (idempotent operations, version checks on the resource).
  • Network partitions violate lock liveness: A network partition separates the lock holder from the lock service. The lock service expires the lock (no heartbeats received). The lock holder continues working, believing it still holds the lock. When the partition heals, two clients may have modified the resource. Solution: the resource itself must validate the fencing token, rejecting requests from the old lock holder.
  • Redlock controversy: Redis's Redlock algorithm attempts to provide distributed locks using multiple independent Redis instances. Martin Kleppmann demonstrated that Redlock is not safe under network delays and GC pauses because it fundamentally depends on timing assumptions. The debate (Kleppmann vs. Antirez) highlights that truly safe distributed locks require consensus (ZooKeeper, etcd) rather than timing-based heuristics.

Sizing & Tradeoffs

Lock Granularity

Coarse-grained locks (table-level, service-level) are simple but serialize too much. If you lock the entire inventory table to update one SKU, no other SKU can be updated concurrently. Fine-grained locks (row-level, resource-level) maximize concurrency but increase lock management overhead and deadlock risk. The key metric: lock_contention_rate = concurrent_writers / lock_granularity_units. If 100 writers contend for 1 lock, contention is 100:1. If they contend for 10,000 locks (one per SKU), average contention is 0.01:1 — negligible.

Lock Hold Time

Too short: The operation hasn't completed when the lock expires. Another client acquires the lock and corrupts the resource. Too long: Locks are held unnecessarily, blocking other clients. Rule of thumb: Set the lock TTL to 3-5x the p99 operation time. If the operation takes 50ms at p99, set the TTL to 150-250ms. Implement lock renewal (heartbeat) for operations that might occasionally take longer — but alert on renewals, because they signal an operation is slower than expected.

Contention Level and Strategy Selection

The choice between pessimistic and optimistic concurrency control depends on contention. Below ~5% write conflict rate, OCC is typically faster because it avoids lock acquisition overhead. Above ~30% conflict rate, pessimistic locking wins because OCC wastes too much work on aborted transactions. Between 5-30% is the gray zone where benchmarking is required.

SLO-Driven Sizing Examples

Concurrency control parameters should be derived from your Service Level Objectives, not guessed.

Example 1 — E-commerce Inventory: p99 lock acquisition < 10ms, 50K TPS

Your inventory service must handle 50,000 transactions per second with p99 lock acquisition under 10ms. Each transaction decrements the stock count for a specific SKU.

Derivation:

  • Lock granularity: Per-SKU locks. With 100K SKUs and 50K TPS uniformly distributed, average contention per SKU is 0.5 TPS. Even for hot SKUs (top 1% handling 20% of traffic = 10K TPS across 1000 SKUs = 10 TPS per hot SKU), contention is manageable.
  • Lock service: Redis over ZooKeeper. Redis SETNX round-trip is ~0.5ms (same-region). ZooKeeper ephemeral node creation is 2-10ms. At p99 < 10ms, Redis gives more headroom. A Redis Cluster with 6 nodes handles 500K+ ops/s, well above 50K TPS.
  • Lock TTL: 50ms. The inventory decrement operation (read + conditional write) takes 5-10ms at p99. A 50ms TTL provides a 5x safety margin. If the holder crashes, the lock is released within 50ms — well within the p99 SLO.
  • Fencing tokens: Required. Each lock acquisition returns a monotonically increasing token. The database checks UPDATE inventory SET stock = stock - 1 WHERE sku = ? AND fence_token < ?.
  • Capacity: 50K lock acquire + 50K lock release = 100K Redis ops/s. A single Redis node handles 100K+ ops/s. Use a cluster for redundancy, not throughput.

Example 2 — Collaborative Editor: p99 conflict resolution < 50ms, 1000 concurrent editors per doc

Your collaborative document editor must support 1000 concurrent editors on a single document with p99 conflict resolution under 50ms.

Derivation:

  • Strategy: CRDTs over OCC. With 1000 concurrent editors, OCC would have astronomical conflict rates — nearly every write would conflict with another. CRDTs allow every edit to be applied locally without coordination, then merged.
  • CRDT type: A sequence CRDT (like RGA or Yjs YATA) for text. Each character has a unique ID. Insertions and deletions are commutative operations that converge regardless of order.
  • Sync protocol: Each client maintains a local CRDT replica. Edits are broadcast via WebSocket to a coordination server, which relays to all other clients. At 1000 editors typing ~2 chars/second, the server handles 2000 ops/second per document. Each op is ~100 bytes, so 200 KB/s bandwidth per document.
  • Conflict resolution: CRDTs have no conflicts by design — merge is deterministic. The p99 < 50ms target applies to the end-to-end sync latency: client-to-server (5-20ms) + server-to-client broadcast (5-20ms) = 10-40ms at p99. Achievable with co-located WebSocket servers.
  • Fallback: If CRDT state diverges (due to bugs, not theory), use a periodic full-state sync every 30 seconds as a consistency checkpoint.

Example 3 — Distributed Rate Limiter: p99 < 5ms, 1M RPS

Your API gateway must enforce a global rate limit of 10,000 requests per second per customer, across 50 gateway instances, handling 1M total RPS with p99 decision latency under 5ms.

Derivation:

  • Strategy: Local + periodic sync distributed semaphore. Pure distributed counting at 1M RPS requires 1M Redis ops/s — expensive and fragile. Instead, each gateway instance maintains a local counter and syncs with Redis every 100ms.
  • Local allocation: Each instance gets limit / num_instances = 10,000 / 50 = 200 requests per second as its local quota. Decisions are made locally against this quota with zero network latency.
  • Sync adjustment: Every 100ms, each instance reports its count to Redis and receives an adjusted quota. If some instances are underutilized, their unused quota is redistributed to busy instances. This allows burst handling while maintaining the global limit.
  • Accuracy tradeoff: With 100ms sync intervals, the global count can overshoot by up to 50 instances * 200 rps * 0.1s = 1000 requests (10% overshoot). For most rate limiting, 10% accuracy is acceptable. Tighten by syncing every 50ms (5% overshoot) at the cost of 2x Redis traffic.
  • p99 latency: Local counter check is ~0.01ms. The 5ms budget is met trivially because there is no network round-trip in the hot path. Redis syncs happen in the background.

Example 4 — Banking Ledger: Serializable isolation, p99 < 100ms

Your banking ledger must provide serializable isolation for all transactions (no phantom reads, no write skew) with p99 transaction latency under 100ms.

Derivation:

  • Strategy: Serializable Snapshot Isolation (SSI) via CockroachDB or PostgreSQL with SERIALIZABLE isolation level. SSI provides serializable guarantees without the deadlock-prone 2PL, using conflict detection at commit time.
  • MVCC parameters: PostgreSQL retains old row versions for active transactions. With p99 < 100ms, transactions are short-lived. Set old_snapshot_threshold = 1min to aggressively clean dead tuples. Run autovacuum with autovacuum_naptime = 10s and autovacuum_vacuum_cost_delay = 2ms to prevent bloat.
  • Lock pool sizing: PostgreSQL's max_locks_per_transaction defaults to 64. A banking transfer touches 2-4 rows (debit account, credit account, transaction log, maybe a running balance). 64 is sufficient. Set max_connections = 200 with PgBouncer for connection pooling.
  • SSI abort rate: Under SSI, CockroachDB aborts 1-5% of transactions due to serialization conflicts. With a 100ms p99 budget, one retry (2x the base latency) stays within budget if the base p99 is under 50ms. Monitor txn_aborts_serializable — if it exceeds 10%, reduce contention by partitioning hot accounts or batching updates.
  • Write skew prevention: SSI detects write skew automatically. Example: two transactions both read that total withdrawals < $1000, then both withdraw $800, leaving the account at -$600. SSI aborts one of them. Without SSI, you'd need explicit SELECT FOR UPDATE on the balance check.

What Can Go Wrong

Distributed Deadlocks: The Coffman Conditions in Distributed Systems

Deadlocks require all four Coffman conditions to hold simultaneously. In distributed systems, each condition manifests differently:

  1. Mutual Exclusion: Distributed locks inherently provide mutual exclusion. Any system using Redis locks, ZooKeeper locks, or database SELECT FOR UPDATE satisfies this condition. Unlike OS mutexes, distributed mutual exclusion is not guaranteed under network partitions (split-brain scenarios may allow two holders).
  2. Hold-and-Wait: A service holds lock A while waiting to acquire lock B. This is common in multi-resource transactions: a payment service holds a lock on the customer account while waiting for a lock on the merchant account. In microservices, this manifests as a service holding a database lock while making an RPC to another service that is itself waiting for a lock.
  3. No Preemption: Once a distributed lock is acquired, it cannot be forcibly taken away (unlike OS locks where the kernel can preempt). TTL-based expiry is the closest analogue to preemption, but it is time-based, not on-demand.
  4. Circular Wait: Service A holds lock X and waits for lock Y. Service B holds lock Y and waits for lock X. In microservices, this can span multiple services: Order Service holds the order lock and calls Inventory Service, which holds the inventory lock and calls Order Service back — circular dependency.

Deadlock Prevention Strategies

  • Resource ordering: Assign a global order to all lockable resources (e.g., alphabetical by resource name). Always acquire locks in this order. This breaks the circular wait condition. Simple and effective for systems where all locks are known at transaction start.
  • Timeout-based preemption: All lock acquisitions have a timeout. If a lock cannot be acquired within the timeout, the transaction aborts and releases all held locks, retrying from the beginning. This breaks the no-preemption condition. The timeout must be shorter than the lock TTL.
  • Lock-free algorithms: Use CRDTs, append-only logs, or event sourcing to eliminate locks entirely. If there are no locks, there are no deadlocks. Applicable when eventual consistency is acceptable.

Deadlock Detection: Wait-For Graphs

A wait-for graph has nodes representing transactions and directed edges from T1 to T2 meaning "T1 is waiting for a lock held by T2." A cycle in this graph is a deadlock. Databases like PostgreSQL maintain a local wait-for graph and detect cycles with a background thread, aborting the youngest transaction in the cycle. Distributed deadlock detection requires building a global wait-for graph from information scattered across multiple nodes — a non-trivial problem because the graph changes as messages propagate. Algorithms like the Chandy-Misra-Haas probe-based algorithm detect distributed cycles without building a global graph.

Wait-For Graph: Concrete Deadlock Detection Example

Scenario: Three transactions are running concurrently, each holding one lock and waiting for another:

  • T1 (started at t=100) holds lock on orders table, needs lock on inventory (held by T2)
  • T2 (started at t=102) holds lock on inventory table, needs lock on payments (held by T3)
  • T3 (started at t=105) holds lock on payments table, needs lock on orders (held by T1)

The wait-for graph edges:

Edge Meaning T(waiter) Holds T(waiter) Needs
T1 → T2 T1 is waiting for a lock held by T2 orders inventory
T2 → T3 T2 is waiting for a lock held by T3 inventory payments
T3 → T1 T3 is waiting for a lock held by T1 payments orders

Cycle detection: Following the edges: T1 → T2 → T3 → T1. We return to T1, confirming a cycle. This is a deadlock.

Victim selection: The database must abort one transaction to break the cycle. Common strategies:

  • Youngest transaction (least work lost): T3 started at t=105, the latest. Abort T3. This releases the payments lock, allowing T2 to proceed, which then releases inventory, allowing T1 to proceed.
  • Least locks held: All hold 1 lock each — tie. Fall back to youngest.
  • Least rows modified: If T3 has modified the fewest rows, aborting it wastes the least work.

After aborting T3: The graph becomes T1 → T2 (no cycle). T2 acquires payments and completes. T1 acquires inventory and completes. T3 is rolled back and can retry, now acquiring locks without a cycle because T1 and T2 have finished.

PostgreSQL checks for cycles every deadlock_timeout (default: 1 second). Setting this lower (e.g., 200ms) reduces how long deadlocked transactions wait, but increases CPU overhead from more frequent graph traversals. For most workloads, 1 second is a reasonable default.

GC Pause Lock Expiry

What happened: A Java service acquired a Redis distributed lock with a 30-second TTL to process an exclusive batch job. A 45-second GC pause (full GC on a 32GB heap) caused the lock to expire. Another instance acquired the lock and started the same batch job. When the first instance resumed from GC, both instances were processing the same batch, causing duplicate records in the database.

Root cause: The lock TTL was shorter than the worst-case GC pause. No fencing token was used.

Mitigation: Use fencing tokens. The batch job's database writes include the fencing token: INSERT ... WHERE fence_token > last_seen_fence_token. The stale instance's writes are rejected because its fencing token is older. Additionally, tune GC to limit pause times (use G1GC or ZGC with max pause targets) and set lock TTLs to 5x the worst-case pause.

Optimistic Locking Retry Storm

What happened: A flash sale created extreme write contention on a single product's inventory counter. The system used optimistic locking (version-based conditional updates). With 10,000 concurrent buyers, almost every transaction conflicted and retried. Each retry read the latest version, attempted a write, conflicted again, and retried — creating a retry storm that amplified database load by 50x. The database's CPU hit 100% and all requests failed.

Root cause: OCC assumes low contention. At 10,000 concurrent writers on a single row, the conflict rate was ~99.99%.

Mitigation: For this specific hot-key pattern, switch to pessimistic locking (or a queued/serialized approach). Alternatively, use a pre-allocated token approach: generate 1000 "purchase tokens" and distribute them to requesting clients. Only clients with tokens can attempt the purchase, limiting contention to a manageable level.

Write Skew Under Snapshot Isolation

What happened: A hospital scheduling system used PostgreSQL with REPEATABLE READ (snapshot isolation). The rule: at least one doctor must be on call at all times. Two doctors, both on call, simultaneously checked the on-call count (2), saw it was safe to go off-call, and both updated their status. Now zero doctors are on call. Each transaction saw a consistent snapshot showing 2 on-call doctors, so each believed it was safe to remove one.

Root cause: This is a write skew anomaly — two transactions read the same data, make disjoint writes, and together violate an invariant that neither violated individually. Snapshot isolation does not prevent write skew.

Mitigation: Use SERIALIZABLE isolation (SSI), which detects write skew at commit time and aborts one transaction. Alternatively, use an explicit SELECT ... FOR UPDATE on all on-call rows to force serialization of the check.

Key Metrics to Monitor

Metric What It Tells You Alert Threshold

lock_timeout_rate Percentage of lock attempts that time out. High rates mean deadlocks or excessive contention. Alert if > 1% over 5 minutes
occ_conflict_rate Percentage of optimistic transactions that abort due to conflicts. Alert if > 10% sustained (indicates contention exceeds OCC suitability)
deadlock_detection_count Number of deadlocks detected and resolved per minute. Alert if > 0 (any deadlock warrants investigation)

Debug Playbook

High lock acquisition latency:

1. Check lock_acquisition_p99_ms — which resource or lock key has elevated latency?

2. Check lock_hold_time_p99_ms for that key — are lock holders taking too long? If so, optimize the critical section or increase lock granularity.

3. Check for deadlocks — run SELECT * FROM pg_stat_activity WHERE wait_event_type = 'Lock' to see blocked queries. Use pg_blocking_pids() to find the blocker.

4. If using Redis locks, check SLOWLOG GET for slow Redis operations. A slow Redis instance delays all lock operations.

5. Check client count — has the number of lock contenders increased (new deployment, traffic spike)? Consider increasing lock granularity (per-SKU instead of per-category).

6. If OCC conflict rate is high, consider switching the hot path to pessimistic locking or a queue-based approach.

Interactions & Pitfalls

Distributed Locks + Leader Election

Leader election is often implemented using distributed locks: the leader holds a lock, and when it expires or is released, another node takes over. The pitfall: if the leader's lock expires during a GC pause, two nodes may briefly believe they are the leader. This is identical to the fencing token problem. Use etcd or ZooKeeper for leader election — they provide linearizable guarantees and session-based leadership that handles this correctly. Do not roll your own leader election with Redis SETNX.

Concurrency Control + Database Replication

Pessimistic locks on the primary database do not replicate to read replicas. If you use SELECT FOR UPDATE on the primary and a read query hits a replica, the read bypasses the lock and sees stale data. This is not a bug — replicas are eventually consistent. The pitfall: reading your own writes via a replica after acquiring a lock on the primary. Solution: always route lock-protected reads to the primary, or use synchronous replication.

CRDTs + Undo/Delete Semantics

CRDTs handle concurrent additions well, but deletions are problematic. If one user deletes an element while another concurrently adds the same element, the result depends on the CRDT type. An observed-remove set (OR-Set) gives precedence to add operations (the element reappears after deletion). A remove-wins set does the opposite. Neither may match user expectations. This is the "deleted-but-reappears" bug that plagues collaborative editors using CRDTs. Solution: use tombstones with expiry, or design the UX to handle these edge cases gracefully.

Optimistic Locking + Caching

If you cache database rows and use optimistic locking (version-based conditional writes), a stale cache can cause unnecessary conflicts. Client reads version 5 from cache, modifies it, and writes with WHERE version = 5. But the database is already at version 8. The write fails, not because of a real conflict, but because the cache was stale. Solution: for OCC-protected resources, cache versions must be invalidated on writes, or bypass the cache entirely for the version check.

Cost Considerations

Cost Analysis

Distributed lock infrastructure: A Redis Cluster for distributed locking costs $50-200/month (3-6 nodes, r6g.large). ZooKeeper or etcd clusters for consensus-based locks cost $150-500/month (3-5 nodes, m6i.large — consensus requires odd numbers for quorum). ZooKeeper/etcd is more expensive but provides stronger guarantees.

MVCC storage overhead: MVCC databases (PostgreSQL, CockroachDB) retain old row versions for active transactions. A table with 1M rows, 1KB each, under 100 TPS with average 50ms transactions, retains ~5,000 dead tuples at any time (50ms * 100 TPS = 5 concurrent transactions, each touching ~1,000 rows). Storage overhead is ~5MB — negligible. But long-running transactions (analytics queries spanning 10 minutes) can cause MVCC bloat: 10min * 100 TPS * 1KB = ~60MB of retained dead tuples. Solution: use separate replicas for analytics.

CRDT bandwidth cost: CRDTs replicate operations across all nodes. A G-Counter with 100 nodes sends 99 update messages per increment. At 10K increments/sec, that is 990K messages/sec cluster-wide. At 100 bytes per message, that is ~95 MB/sec. For a 3-region deployment with inter-region bandwidth at $0.02/GB, this costs ~$165/month. Optimization: batch CRDT updates and sync every 100ms instead of per-operation.

Cost of concurrency bugs: A distributed deadlock in production can cause cascading failures affecting all users. A race condition in payment processing can result in double charges, chargebacks, and regulatory scrutiny. Cloudflare's 2019 outage (caused by a global lock contention issue in their WAF) affected millions of websites for 30 minutes. The cost of getting concurrency wrong vastly exceeds the cost of the infrastructure to get it right.

Real-World Examples

Google Spanner: TrueTime and Lock-Free Reads

Google Spanner uses TrueTime (GPS + atomic clock synchronization) to bound clock uncertainty to ~7ms. This allows Spanner to implement externally consistent (serializable + linearizable) transactions across globally distributed data. Writers acquire locks and assign commit timestamps. Readers perform lock-free reads at a snapshot timestamp. The key insight: because TrueTime bounds clock uncertainty, Spanner knows that if it waits 7ms after committing, all other nodes will agree on the ordering of events. This eliminates the need for reader locks — readers just pick a timestamp and read the MVCC version at that timestamp. The result: globally consistent reads with no lock contention.

CockroachDB: Serializable Snapshot Isolation

CockroachDB implements SSI across a distributed SQL database. Each transaction operates on a snapshot. At commit time, CockroachDB checks for serialization anomalies (read-write conflicts that could cause write skew). If anomalies are detected, the transaction is aborted and retried. The retry logic is built into the client driver, making it transparent to application code. In production, CockroachDB reports SSI abort rates of 1-5% under typical workloads, with most retries succeeding on the second attempt. This provides serializable isolation without the deadlock risk of traditional 2PL.

Redis Distributed Locks (and the Redlock Debate)

Redis SETNX is the most widely used distributed lock mechanism due to its simplicity and low latency. However, Redis is a single-point-of-failure for locks. Antirez (Redis creator) proposed Redlock: acquire the lock on N/2+1 out of N independent Redis instances. If the majority succeed, the lock is acquired. Martin Kleppmann's analysis showed that Redlock's safety depends on timing assumptions (bounded message delays, bounded process pauses) that cannot be guaranteed in practice. The practical takeaway: use Redis locks when efficiency matters (preventing duplicate work) but not when correctness matters (preventing data corruption). For correctness-critical locks, use ZooKeeper or etcd, which provide consensus-based coordination.

Figma: CRDTs for Multiplayer Design

Figma uses a custom CRDT implementation for real-time collaborative design editing. Multiple designers can simultaneously move, resize, and modify objects on a shared canvas. Each operation is encoded as a CRDT operation that can be applied in any order and will converge to the same result. Figma chose CRDTs over Operational Transformation (OT) because CRDTs do not require a central server for conflict resolution — conflict resolution is built into the data structure. This allows Figma to scale to hundreds of concurrent editors per file with sub-50ms sync latency. The engineering challenge was designing CRDTs for the complex domain of vector graphics (layers, groups, masks, styles), which required custom CRDT types beyond standard counters and sets.

Distributed Lock with Fencing Token

import uuid, time, random, redis

class DistributedLock:
    ACQUIRE_SCRIPT = """
    if redis.call("SET", KEYS[1], ARGV[1], "NX", "PX", ARGV[2]) then
        local token = redis.call("INCR", KEYS[2])
        return token
    end
    return nil
    """
    RELEASE_SCRIPT = """
    if redis.call("GET", KEYS[1]) == ARGV[1] then
        return redis.call("DEL", KEYS[1])
    end
    return 0
    """

    def __init__(self, client: redis.Redis, lock_key: str, ttl_ms: int = 30000):
        self.redis = client
        self.lock_key = lock_key
        self.ttl_ms = ttl_ms
        self.owner_id = str(uuid.uuid4())
        self.fencing_token = None

    def acquire(self, max_retries: int = 5, base_delay_ms: int = 100):
        for attempt in range(max_retries):
            result = self.redis.eval(
                self.ACQUIRE_SCRIPT, 2,
                self.lock_key, f"{self.lock_key}:fence",
                self.owner_id, self.ttl_ms,
            )
            if result is not None:
                self.fencing_token = int(result)
                return {"acquired": True, "fencing_token": self.fencing_token}
            # Exponential backoff with jitter
            delay = base_delay_ms * (2 ** attempt) / 1000
            time.sleep(delay + delay * 0.5 * random.random())
        return {"acquired": False, "fencing_token": None}

    def release(self) -> bool:
        return self.redis.eval(self.RELEASE_SCRIPT, 1, self.lock_key, self.owner_id) == 1

    def execute_with_lock(self, fn):
        lock = self.acquire()
        if not lock["acquired"]:
            raise RuntimeError("Failed to acquire distributed lock")
        try:
            return fn(lock["fencing_token"])
        finally:
            self.release()

# Usage: inventory update with fencing token protection
lock = DistributedLock(redis_client, "lock:inventory:sku-12345")

def update_inventory(fencing_token):
    row = db.execute("SELECT stock FROM inventory WHERE sku = %s", ("sku-12345",))
    if row["stock"] <= 0:
        raise ValueError("Out of stock")
    # Fencing token ensures stale writes from expired locks are rejected
    db.execute(
        "UPDATE inventory SET stock = stock - 1, last_fence_token = %s "
        "WHERE sku = %s AND last_fence_token < %s",
        (fencing_token, "sku-12345", fencing_token),
    )

lock.execute_with_lock(update_inventory)

Optimistic Concurrency Control with Version Check

import time, random

class OptimisticStore:
    def __init__(self, db):
        self.db = db

    def read_with_version(self, table: str, row_id: str):
        row = self.db.execute(
            f"SELECT *, version FROM {table} WHERE id = %s", (row_id,)
        )
        if row is None:
            return None
        return {"data": row, "version": row["version"]}

    def update_with_version(self, table, row_id, updates, expected_version, max_retries=3):
        for attempt in range(max_retries + 1):
            set_clause = ", ".join(f"{k} = %s" for k in updates)
            params = list(updates.values()) + [row_id, expected_version]

            result = self.db.execute(
                f"UPDATE {table} SET {set_clause}, version = version + 1 "
                f"WHERE id = %s AND version = %s RETURNING *",
                params,
            )

            if result is not None:
                return {"success": True, "data": result}

            # Conflict detected: re-read and retry
            if attempt < max_retries:
                fresh = self.read_with_version(table, row_id)
                if fresh is None:
                    return {"success": False, "reason": "NOT_FOUND"}
                expected_version = fresh["version"]
                delay = 0.05 * (2 ** attempt) + random.random() * 0.05
                time.sleep(delay)

        return {"success": False, "reason": "MAX_RETRIES_EXCEEDED"}

Distributed Semaphore

import uuid, time, random, redis

ACQUIRE_SCRIPT = """
redis.call("ZREMRANGEBYSCORE", KEYS[1], "-inf", ARGV[1])
local count = redis.call("ZCARD", KEYS[1])
if count < tonumber(ARGV[2]) then
    redis.call("ZADD", KEYS[1], ARGV[3], ARGV[4])
    return 1
end
return 0
"""

class DistributedSemaphore:
    def __init__(self, client: redis.Redis, key: str, max_permits: int, ttl_ms: int = 30000):
        self.redis = client
        self.key = key
        self.max_permits = max_permits
        self.ttl_ms = ttl_ms

    def acquire(self, timeout_ms: int = 5000):
        permit_id = str(uuid.uuid4())
        deadline = time.time() + timeout_ms / 1000

        while time.time() < deadline:
            now = int(time.time() * 1000)
            result = self.redis.eval(
                ACQUIRE_SCRIPT, 1, self.key,
                now, self.max_permits, now + self.ttl_ms, permit_id,
            )
            if result == 1:
                return {"acquired": True, "permit_id": permit_id}
            time.sleep(0.05 + random.random() * 0.05)

        return {"acquired": False, "permit_id": None}

    def release(self, permit_id: str) -> bool:
        return self.redis.zrem(self.key, permit_id) == 1

    def current_count(self) -> int:
        now = int(time.time() * 1000)
        self.redis.zremrangebyscore(self.key, "-inf", now)
        return self.redis.zcard(self.key)

if not permit["acquired"]:

        db_pool.putconn(conn)
finally:
    semaphore.release(permit["permit_id"])

# Monitor: alert if semaphore utilization > 80%
usage = semaphore.current_count()
if usage > 0.8 * 50:
    alerting.warn(f"DB semaphore at {usage}/50 ({usage / 50 * 100:.0f}%)")

How to Validate Your Design

Concurrency Correctness Testing

Use Jepsen to test distributed lock implementations against network partitions, clock skew, and process pauses. Jepsen has found bugs in Redis, ZooKeeper, etcd, CockroachDB, and virtually every other distributed system. Run Jepsen against your lock service with nemeses (fault injectors) for network partitions, clock skew, and process kills. Verify linearizability: if a lock is granted to client A and then to client B, B's operations must see all of A's writes.

Deadlock Detection Testing

Intentionally create circular lock dependencies in a staging environment. Verify: (1) the deadlock is detected within the configured timeout, (2) one transaction is aborted cleanly, (3) the aborted transaction retries and succeeds, (4) no data corruption occurs. Automate this test: spawn N threads, each acquiring locks in a random order, and verify that the system makes progress (no permanent hangs) and maintains correctness (no lost updates).

Load Testing Under Contention

Use k6 or Gatling to simulate realistic contention patterns. Create a "hot key" scenario: 80% of writes target 1% of resources (simulating a flash sale). Measure: (1) throughput under contention — does the system degrade gracefully or collapse? (2) p99 latency — is it within SLO? (3) conflict/retry rate — is it within the expected range for the chosen strategy? (4) deadlock count — should be zero under correct lock ordering.

Chaos Testing for Lock Safety

Simulate the failure modes that break distributed locks: (1) Kill the lock holder process after it acquires the lock but before it completes the operation. Verify the lock expires and another client acquires it. (2) Introduce a network partition between the lock holder and the lock service. Verify the lock expires and is safely reacquired. (3) Simulate a GC pause (use SIGSTOP/SIGCONT on the lock holder process). Verify fencing tokens prevent stale writes. (4) Kill a Redis/ZooKeeper/etcd node during lock acquisition. Verify the lock service remains available (quorum-based systems) or that clients handle the failure gracefully (single-node Redis).

Interview Questions

How would you implement a distributed lock? What are the failure modes?

A distributed lock ensures mutual exclusion across processes on different machines. The simplest implementation uses Redis SETNX: SET lock_key owner_id NX PX 30000 atomically sets the key only if it does not exist, with a 30-second TTL. Release uses a Lua script that checks ownership: if GET(key) == owner then DEL(key). This prevents releasing another client's lock.

For stronger guarantees, use ZooKeeper ephemeral nodes or etcd leases. ZooKeeper provides linearizable guarantees backed by ZAB consensus. The client creates an ephemeral znode; if the client's session dies, ZooKeeper auto-deletes the node. Other clients watch the node for deletion.

Failure modes: (1) Lock holder crashes: The lock is stuck until the TTL expires (Redis) or the session times out (ZooKeeper). During this window, the resource is inaccessible. (2) GC pause outlasts TTL: The holder pauses, the lock expires, another client acquires it. The holder resumes and believes it still holds the lock. Two clients now operate on the resource concurrently. Fix: fencing tokens — the lock service issues a monotonically increasing token with each acquisition. The resource checks the token and rejects stale operations. (3) Network partition: The holder is healthy but partitioned from the lock service. The lock expires, another client acquires it. Same two-holder problem. Fix: fencing tokens again. (4) Clock skew (Redis): If Redis nodes have different clocks, TTL expiry may happen earlier or later than expected. Redlock amplifies this problem. Fix: use consensus-based systems (ZooKeeper, etcd) where correctness does not depend on clocks.

Follow-up: "When is a distributed lock not safe enough, and what do you use instead?" — When strict mutual exclusion is required for data safety (not just efficiency), use consensus-based approaches (etcd, ZooKeeper) with fencing tokens. For the highest safety, make the protected resource itself validate the fencing token, so even if the lock is broken, stale writes are rejected at the resource level.

Explain the difference between optimistic and pessimistic concurrency control. When would you use each?

Pessimistic concurrency control assumes conflicts are likely and prevents them by acquiring locks before accessing data. The transaction locks the resource, performs its work, and releases the lock at commit. No other transaction can modify the resource while it's locked. In databases: SELECT ... FOR UPDATE. In distributed systems: acquire a Redis/ZooKeeper lock before proceeding. Pros: simple reasoning, no retries, guaranteed progress for the lock holder. Cons: reduced concurrency (readers block writers and vice versa), deadlock risk, lock management overhead.

Optimistic concurrency control assumes conflicts are rare and detects them at commit time. The transaction reads data and its version, performs work locally, and attempts a conditional write: "update if version is still V." If another transaction modified the data, the version has changed, and the write is rejected. The transaction re-reads and retries. In databases: UPDATE ... WHERE version = V. In distributed systems: DynamoDB ConditionExpression, ETags, CAS operations. Pros: maximum concurrency (no blocking), no deadlocks, readers never block. Cons: wasted work when conflicts occur, retry storms under high contention.

When to choose: Use pessimistic when contention is high (> 20-30% of transactions conflict), operations are non-idempotent (cannot safely retry), or correctness is paramount and throughput is secondary (financial transactions). Use optimistic when contention is low (< 5% conflict rate), the workload is read-heavy with occasional writes, operations span long durations (user editing sessions), or you need maximum throughput and can tolerate retries.

Follow-up: "What about mixed strategies?" — Many production systems use both. Read-heavy paths use OCC for throughput. Write-heavy hot paths use pessimistic locks for predictability. The hot inventory counter during a flash sale gets a pessimistic lock; the product description update uses OCC.

How does MVCC work? Why is it preferred over locking in modern databases?

MVCC (Multi-Version Concurrency Control) maintains multiple versions of each data item. When a transaction writes a row, it creates a new version rather than overwriting in place. Each version has a creation timestamp (or transaction ID) and a deletion timestamp. When a transaction reads a row, it sees the version that was current at the transaction's start time (its snapshot).

How PostgreSQL implements MVCC: Each row has xmin (the transaction ID that created this version) and xmax (the transaction ID that deleted/updated this version, or 0 if still current). A transaction with ID 100 reading a table sees all rows where xmin < 100 and xmax = 0 OR xmax > 100. This is a consistent snapshot. Writers create new row versions; they don't modify existing ones. Old versions are cleaned up by the autovacuum process after all transactions that might need them have completed.

Why MVCC is preferred: (1) Readers never block writers: A long-running analytics query reads a consistent snapshot while concurrent OLTP transactions insert and update without interference. With locking, the analytics query would either block all writes or read inconsistent data. (2) Writers rarely block readers: Since writes create new versions, readers continue seeing the old version. (3) No read locks: Reads never acquire locks, eliminating a major source of contention and deadlocks. (4) Natural snapshot isolation: Each transaction sees a consistent point-in-time view of the database. (5) Higher throughput: Without read-write contention, throughput scales much better under concurrent workloads.

Tradeoffs: MVCC has higher storage overhead (old versions consume space until vacuumed), requires background cleanup (autovacuum in PostgreSQL, compaction in CockroachDB), and does not prevent write-write conflicts (two concurrent writes to the same row still conflict). Write skew anomalies are possible under snapshot isolation — SSI or explicit locking is needed for serializable semantics.

Follow-up: "What is the write skew anomaly and how does SSI prevent it?" — Write skew occurs when two transactions read overlapping data, make disjoint writes, and together violate an invariant. SSI tracks read-write dependencies and aborts one transaction at commit time if a serialization anomaly is detected.

What causes distributed deadlocks and how would you prevent them?

Distributed deadlocks occur when the four Coffman conditions hold across multiple nodes: (1) Mutual exclusion — resources can only be held by one transaction at a time (inherent to locks). (2) Hold-and-wait — a transaction holds one lock while waiting for another. (3) No preemption — locks cannot be forcibly taken from a holder. (4) Circular wait — a cycle of transactions, each waiting for a lock held by the next.

In distributed systems, common causes include: (1) Services acquiring locks in different orders: Service A locks user-account then product-inventory; Service B locks product-inventory then user-account. (2) Circular RPC dependencies: Service A calls Service B (holding lock X), Service B calls Service C (holding lock Y), Service C calls Service A (which is blocked on lock X). (3) Multi-table transactions in different orders: Transaction 1 does UPDATE accounts then UPDATE orders; Transaction 2 does UPDATE orders then UPDATE accounts.

Prevention strategies: (1) Lock ordering: Define a global total order on all resources (e.g., alphabetical by resource name, or numeric by resource ID). Always acquire locks in ascending order. This provably prevents circular wait. (2) Timeout-based preemption: All lock acquisitions have a maximum wait time. If exceeded, abort the transaction, release all held locks, and retry with exponential backoff. This breaks the no-preemption condition. (3) Acquire all locks upfront: Determine all needed locks before starting the transaction and acquire them all at once (atomically or in sorted order). This breaks hold-and-wait. (4) Avoid locks entirely: Use OCC or CRDTs where possible, eliminating the conditions for deadlock.

Detection: If prevention is not feasible, detect deadlocks using a wait-for graph. PostgreSQL detects local deadlocks by checking for cycles in the lock wait graph every deadlock_timeout interval (default: 1 second). For distributed deadlocks, use the Chandy-Misra-Haas algorithm: when a transaction blocks, it sends a probe message along the wait-for chain. If the probe returns to the originator, a cycle exists and one transaction is aborted.

Follow-up: "How do you choose which transaction to abort in a deadlock?" — Abort the youngest transaction (least work wasted), the transaction holding the fewest locks (easiest recovery), or the transaction with the lowest priority (if priority is defined). PostgreSQL aborts the transaction that detected the deadlock.

Design a rate limiter using distributed semaphores that handles 1M RPS.

A pure distributed semaphore at 1M RPS would require 1M Redis operations per second — expensive and introduces a single point of failure. The key insight is to use a two-tier architecture: local semaphores for the hot path, with periodic synchronization to a global distributed semaphore.

Architecture: (1) N gateway instances (e.g., 50 instances). Each instance gets an allocated quota: global_limit / N requests per second per customer. (2) Local token bucket: Each instance maintains an in-memory token bucket per customer with its allocated quota. Request decisions are made locally in ~0.01ms with zero network calls. (3) Global sync layer: Every 100ms, each instance reports its usage to a Redis sorted set and receives an updated quota. Underutilized instances donate unused quota to busy instances. (4) Redis implementation: Use ZADD with score = expiry timestamp for the semaphore. ZREMRANGEBYSCORE cleans expired permits. ZCARD checks current count. Total Redis ops: 50 instances * 10 syncs/sec * 3 commands = 1,500 ops/sec — trivial for Redis.

Accuracy analysis: Between sync intervals, each instance operates independently. Worst case: all 50 instances use their full quota simultaneously before syncing = 50 * (limit/50) = limit — correct. But if traffic is uneven (one instance gets 80% of traffic), that instance exhausts its local quota and must wait for the next sync to get more. The sync interval determines how quickly quotas rebalance. At 100ms sync, the max burst overshoot is ~10%. Tighten to 50ms sync for ~5% accuracy.

Failure handling: If Redis is down, instances continue using their last-known local quota. This degrades to a per-instance rate limiter, which may allow up to N * local_quota = global_limit requests (correct), but won't adapt to uneven traffic distribution until Redis recovers. If an instance crashes, its quota is reclaimed at the next sync interval (100ms) and redistributed to remaining instances.

Scaling: This design handles arbitrary RPS because the hot path is local. To scale beyond Redis's sync capacity, shard the semaphore by customer ID across multiple Redis instances (consistent hashing). Each instance syncs with its assigned Redis shard.

Follow-up: "How do you handle customers with very low rate limits (e.g., 10 RPS) across 50 instances?" — With 50 instances, each gets 0.2 RPS — less than one request per 5 seconds per instance. This is too granular for per-instance allocation. Solution: route low-limit customers to a dedicated subset of instances (sticky routing by customer ID) so each instance has a meaningful quota, or fall back to centralized Redis counting for low-volume customers.

How do CRDTs avoid the need for locks? What are their limitations?

CRDTs (Conflict-free Replicated Data Types) are data structures designed so that concurrent updates always converge to the same state, regardless of the order in which updates are applied, without any coordination between replicas. They achieve this through mathematical properties: operations must be commutative (a + b = b + a), associative ((a + b) + c = a + (b + c)), and idempotent (a + a = a). When these properties hold, replicas can accept writes independently and merge them in any order, always arriving at the same final state.

Common CRDT types: (1) G-Counter (grow-only): each node maintains its own counter; the total is the sum of all node counters. (2) PN-Counter: two G-Counters — one for increments, one for decrements; the value is increments minus decrements. (3) G-Set (grow-only set): elements can only be added, never removed. (4) OR-Set (observed-remove set): elements can be added and removed; each element has a unique tag, so concurrent add+remove is resolved by keeping the element (add-wins semantics). (5) LWW-Register (last-writer-wins): timestamps determine which write "wins"; requires synchronized clocks. (6) Sequence CRDTs (RGA, YATA): ordered sequences for collaborative text editing.

Why no locks are needed: Because merge is deterministic and commutative, there is no "conflict" to resolve — all concurrent updates are simply merged. There is no wrong order. No coordination means no locks, no deadlocks, no contention. Replicas are always writable, even during network partitions.

Limitations: (1) Limited data types: Not all operations are commutative. You cannot build a CRDT for arbitrary database transactions. (2) Eventual consistency only: CRDTs guarantee convergence but not when. A read might return stale data if not all updates have propagated. (3) No global invariants: CRDTs cannot enforce constraints like "account balance must be non-negative" because each replica operates independently and might violate the constraint before hearing about other replicas' operations. (4) Metadata overhead: OR-Sets track tombstones for removed elements, which grow unboundedly without garbage collection. G-Counters maintain a counter per node, scaling with the number of nodes. (5) Conflict resolution is semantic, not syntactic: CRDTs resolve conflicts deterministically, but the resolution may not match user intent. Two users concurrently editing the same paragraph may get a merged result that is grammatically nonsensical.

Follow-up: "How would you implement a shopping cart using CRDTs?" — Use an OR-Set where elements are (product_id, quantity) tuples with add-wins semantics. Adding an item creates a new tagged element. Removing it creates a tombstone for that tag. Concurrent add-and-remove of the same item resolves to "item is in the cart" (add wins). Quantity changes use a PN-Counter per product. This works but cannot enforce inventory limits (a global invariant) without coordination.

Key Takeaways

  • Every OS concurrency primitive has a distributed analogue: Mutexes become distributed locks. Semaphores become distributed rate limiters. CAS becomes optimistic concurrency control. Spinlocks become retry loops with exponential backoff. The problems are the same; the failure modes are worse.
  • Distributed locks are fundamentally unsafe without fencing tokens: GC pauses, network partitions, and clock skew can all cause two clients to believe they hold the same lock simultaneously. Fencing tokens (monotonically increasing values checked by the resource) are the only reliable mitigation.
  • Choose your strategy based on contention level: Low contention (< 5% conflicts) favors optimistic concurrency control. High contention (> 30% conflicts) favors pessimistic locking. Lock-free CRDTs provide consistent throughput regardless of contention but only support eventually consistent semantics.
  • MVCC is the modern standard for database concurrency: PostgreSQL, CockroachDB, MySQL InnoDB, and Oracle all use MVCC. Readers never block writers. But MVCC with snapshot isolation allows write skew — use SSI (Serializable Snapshot Isolation) for the strongest guarantees.
  • The four Coffman conditions apply to distributed deadlocks: Prevent deadlocks by breaking one condition — typically circular wait (via lock ordering) or no-preemption (via timeouts). Detect deadlocks via wait-for graphs.
  • Lock granularity is the primary throughput lever: Coarse-grained locks (table-level, service-level) serialize too much. Fine-grained locks (row-level, resource-level) maximize concurrency. Move from coarse to fine-grained as contention increases.
  • CRDTs eliminate coordination at the cost of semantics: CRDTs are the ideal solution when eventual consistency is acceptable and the data can be modeled with commutative operations. They cannot enforce global invariants or provide strong consistency.
  • Monitor concurrency health: Track lock acquisition latency, conflict rates, deadlock counts, and fencing token rejections. These metrics are your early warning system for concurrency problems before they become outages.
