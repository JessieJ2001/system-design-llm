The NALSD Framework

Non-Abstract Large System Design is Google's internal approach to designing systems. The core idea: never design in the abstract. Every decision must be grounded in concrete numbers.

The NALSD Loop:

  1. Requirements — Quantify everything: QPS, storage, latency SLA, availability target.
  2. Back-of-Envelope Math — Calculate capacity needs from the requirements. (Reference numbers here)
  3. Simplest Design — Start with the dumbest thing that could work. One server, one database.
  4. What Breaks? — Identify the first bottleneck. Is it CPU? Memory? Disk? Network? Availability?
  5. Iterate — Fix the bottleneck. Recalculate. Find the next bottleneck. Repeat.

Every scenario below follows this loop. The goal is not to memorize architectures but to practice the reasoning process so you can apply it to any new problem.

Key Takeaways

  • Always start with numbers. Quantify requirements before designing. "High traffic" means nothing — "50K QPS with 200ms p99 SLA" means everything.
  • Start simple, then iterate. The naive design reveals the real constraints. You can't find bottlenecks in a design that doesn't exist yet.
  • Know your reference numbers. Bookmark the BOE reference page. Server throughput, latency hierarchy, storage growth — these numbers are your design vocabulary.
  • "What breaks first?" is the key question. Every system has a bottleneck. Find it, fix it, find the next one.
  • Tradeoffs, not "best practices." There is no universally correct answer. The right choice depends on the specific numbers in your scenario.
