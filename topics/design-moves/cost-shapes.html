Cost Shapes

What dominates the cloud bill in each class of system. The C7 rubric requires naming the dominant cost line and addressing it; this page is the menu of canonical shapes scenario pages match against.

~10 min reference

How to Use This Page

Every design has a dominant cost line — the single category that, if it doesn't go down, no overall optimization matters. C7 asks you to name it and address it. This page catalogs the canonical shapes by system class so you can:

  1. Anticipate the cost shape from the system's class before you derive numbers, so your iteration arc points at the right move.
  2. Verify your derivation: if your cost shape doesn't match the canonical for the system class, either the system is unusual (state why) or the math is wrong (find it).
  3. Avoid sub-optimization: shrinking 20% off a line that's only 5% of the bill is a waste of iteration budget.

All percentages below are typical at scale; actual mix varies by 10–30% based on workload specifics. Cite M12 pricing-constants for unit prices.

LLM Inference Fleet

Cost shape

  • GPU compute: 80–95%. H100s at $9k/mo each, dozens to thousands deployed. The line that matters; everything else is rounding.
  • Storage (model weights in S3): < 1%.
  • Network (egress to clients): 1–5%.
  • Coordination (orchestrator, autoscaler): < 1%.

The optimization order

  1. Per-request GPU-time. KV cache, PagedAttention, INT8/FP8 quantization, speculative decoding, continuous batching. Each can cut compute 2–10×.
  2. Utilization. Continuous batching and request multiplexing get utilization from ~30% to ~80%; that's a 2.7× cost reduction at the same throughput.
  3. Right-sizing. H100 vs A100 vs L4. For inference of small models, A100/L4 may be 3–4× cheaper per query.
  4. Reservation. 3-yr reserved cuts GPU $/mo by ~55% — but only if utilization stays high.

Anti-patterns

  • Optimizing storage (model weights) when compute is 95% — irrelevant.
  • Reserved capacity at 30% utilization — pay full price for idle hardware.
  • Treating prefill and decode as one workload — they have different cost shapes (prefill is compute-bound, decode is memory-bandwidth-bound).

Storage-Heavy (Backup, Archive, Object Store)

Cost shape

  • Storage: 60–85%. S3, EBS, managed-DB storage. Volume and tier dominate; one PB on S3 Standard is $24k/mo.
  • Restore / read-time egress: 5–20%. Especially for cold tiers (Glacier retrieval cost can be 5–10× the storage cost for actively-restored data).
  • Compute (background jobs: dedup, compaction, replication): 5–15%.
  • API / request pricing: 5–10% (depends heavily on PUT volume).

The optimization order

  1. Tier policy. Hot → warm → cold automatic by age and access pattern. Glacier Deep Archive is 23× cheaper than Standard. Move what doesn't get accessed.
  2. Dedup & compression. Content-defined chunking, Bloom-filtered index, per-chunk compression. 40–80% storage reduction is typical.
  3. Erasure coding instead of replication. 3× replication → 1.5× EC = 50% reduction; trade is repair cost on failure.
  4. Restore cost minimization. Plan restore-time access (locality packs, parallel byte-range, read-ahead) to avoid worst-case Glacier retrieval bills.

Anti-patterns

  • 3× replication on cold-tier object storage — pay 3× for data nobody reads.
  • Hot-tier storage with no lifecycle policy — never moves to Glacier.
  • Frequent restore from Deep Archive — retrieval bill exceeds the storage savings.

Canonical scenarios

43-backup-dedup-storage, 44-object-store-metadata.

Egress-Heavy (CDN, Video Streaming, Public APIs)

Cost shape

  • Egress (Internet + CDN): 50–80%. CloudFront at $0.085/GB (first 10 TB), tiered down. At 1 PB/mo egress: ~$80k/mo just for the wire.
  • Origin compute (transcoding, packaging): 10–30%.
  • Origin storage: 5–15%.
  • Cross-region (replication for global reach): 5–10%.

The optimization order

  1. CDN cache hit rate. Every cache hit is free egress. Going from 80% to 95% hit rate is a 75% reduction in egress cost. Tune TTLs, surrogate keys, cache-key cardinality.
  2. Adaptive bitrate / smart encoding. H.265, AV1, per-title encoding. 30–50% bitrate reduction translates directly to 30–50% egress reduction.
  3. Pre-transcode at upload. Trade upload-time compute for runtime savings.
  4. Negotiate volume tier or commit. Egress is heavily tiered; large customers negotiate sub-list pricing aggressively.

Anti-patterns

  • Bypassing CDN for "personalized" content that could vary by 1-bit-flag — the cache key just needs that flag in it.
  • Origin = single region with global users — every byte crosses an ocean at $0.02/GB minimum.
  • Re-transcoding on the read path — pay compute and egress.

Canonical scenarios

Video streaming systems; CDN-fronted public APIs.

Telemetry / Logging / Observability

Cost shape

  • Hot-tier storage + indexing: 40–60%. Elasticsearch / OpenSearch / ClickHouse — the index dominates because it's typically 50–200% the size of the raw data.
  • Ingest compute: 20–30%. Parsing, enrichment, sampling, routing.
  • Cold-tier archive (S3 Parquet): 5–10%.
  • Egress (queries, exports): 5–10%.

The optimization order

  1. Sampling. Head sampling (drop X% at ingest) and tail sampling (keep all errors / slow traces, sample success). 90% drop reduces hot-tier storage by 90%.
  2. Tiered indexing. Bloom-filter-per-block on cold tier (no index); inverted index only on hot tier. ClickHouse / Loki style.
  3. Per-tenant quota. Without it, one noisy tenant's logs can dominate the bill.
  4. Aggressive TTL. 30 days on hot, 90 on warm, year on cold archive. Older = cheaper tier or delete.

Anti-patterns

  • Indexing every field — write amp explodes; index size exceeds data.
  • Hot-tier retention > 30 days — vast majority of queries are on last 24 hours.
  • No per-tenant quota — one bad service melts the bill.

Canonical scenarios

30-logging-system, 36-streaming-percentiles, 42-error-aggregation-dedupe.

Database-Heavy SaaS

Cost shape

  • Managed DB instance hours: 30–50%. RDS / Aurora / DynamoDB. Multi-AZ doubles, read replicas add per-instance.
  • Storage + IOPS: 20–35%. Often equal to or larger than the instance cost for I/O-heavy workloads.
  • Cross-AZ replication traffic: 5–15%. Inter-AZ is $0.01/GB each way; replication doubles.
  • Backup storage: 5–10%.
  • Cache layer (Redis): 5–15%.

The optimization order

  1. Cache aggressively. Memcached / Redis in front of read-heavy queries — order-of-magnitude QPS reduction on the DB.
  2. Right-size instances. RDS often runs at 20–40% utilization out of paranoia. Track p99 CPU and right-size.
  3. Provisioned IOPS only when needed. gp3 with default IOPS is sufficient for most workloads; io2 is 5× the cost.
  4. Reserved instances. DBs run 24×7; 3-yr reserved cuts ~55%.

Anti-patterns

  • Multi-AZ for dev/staging — pay for HA you don't need.
  • Read replicas without measuring the workload — replicas serving 1% of reads cost 50% of primary.
  • Aurora I/O charges ignored — at 1B I/O/month = $200, easy to overshoot.

Stream Processing & Messaging

Cost shape

  • Kafka brokers + storage: 35–55%. One MSK broker = ~$600/mo + storage; production clusters = 6–30 brokers.
  • Consumer compute (Flink, custom): 25–40%. Stream-processing jobs dominate at high throughput.
  • Cross-AZ replication: 10–20%. Kafka replicates 3×; cross-AZ is $0.01/GB each way.
  • Schema registry, connect, monitoring: 5–10%.

The optimization order

  1. Compression on the wire. Snappy / Zstd 4× reduction; saves both broker storage and inter-AZ bandwidth.
  2. Right-size partitions. Too few = consumer parallelism limited. Too many = broker overhead. Aim for partitions = N × consumer-instance count where N ∈ [1, 4].
  3. Tiered storage (Confluent / MSK Tiered): hot in broker disk, cold in S3. Reduces broker disk cost by 80%.
  4. Consumer co-location. Run consumers in the same AZ as broker leaders; cuts the inter-AZ bandwidth.

Anti-patterns

  • Kafka with 100 brokers because "scale" — can usually do the same job with 10–30.
  • No compression — paying 4× the storage and bandwidth.
  • Consumer in different AZ from broker leader — inter-AZ traffic on every message.

ML Training

Cost shape

  • GPU compute: 75–90%. Same as inference but typically larger jobs (1k–10k GPUs at scale).
  • Inter-node interconnect (EFA / NVLink fabric): bundled with instance.
  • Training data storage (S3): 2–5%.
  • Checkpoint storage (frequent for long runs): 1–3%.
  • Coordination (job scheduler, parameter server): 1–3%.

The optimization order

  1. Utilization. Gang scheduling, fault recovery without restart-from-zero, mixed-precision training. Going from 50% to 90% effective utilization is a 1.8× cost reduction.
  2. Checkpoint frequency vs cost. More frequent = more S3 cost but less wasted compute on failure. Tune to expected MTBF.
  3. Spot instances for fault-tolerant jobs. 60–90% off; works if checkpoint cadence is tight enough.
  4. Quantization-aware training (FP16, BF16, FP8). 2–4× compute reduction at the cost of small accuracy hit.

Anti-patterns

  • On-demand for week-long training jobs — spot can save 70%.
  • Sparse checkpoints + spot — one preemption restarts hours of training.
  • Single-AZ training with EFA — works, but loses HA on AZ failure.

Real-Time Messaging (Chat, Collaborative Editing, Live Push)

Cost shape

  • Connection-handling compute: 30–50%. Long-lived WebSocket / SSE connections at 50–100k per pod; pod count scales with concurrent users, not with QPS.
  • Egress (per-message broadcast): 20–35%. Fan-out multiplies the egress cost per message.
  • Backing store (message history): 10–25%.
  • Cross-AZ for HA: 5–15%.

The optimization order

  1. Per-pod connection density. Tune to file-descriptor / memory ceilings; doubling density halves the pod fleet.
  2. Compress message payloads. Per-message saves 50% bandwidth.
  3. Async fanout via pub-sub (one publish, many subscribers) instead of N publish.
  4. Region-local fanout — don't broadcast to global users from one region.

Anti-patterns

  • WebSocket pod density too low — fleet sized to handle peaks at 5k/pod when 50k is achievable.
  • No message compression — egress doubles.
  • Cross-region broadcast for every message — egress dominates.

Canonical scenarios

10-fan-out-notification, 41-home-timeline-fanout.

Search / Inverted Index

Cost shape

  • Memory-resident index: 40–60%. Elasticsearch / OpenSearch / Lucene. Index typically 30–70% of data size and held in RAM for query latency.
  • Ingest pipeline (parsing, tokenization, enrichment): 15–25%.
  • Query compute (per-query CPU): 10–20%.
  • Storage (replicas, commits): 5–15%.

The optimization order

  1. Right-size shards. Too many small shards = overhead per query; too few = uneven load. Target shard size 30–50 GB.
  2. Frozen tier / searchable snapshots (ES 7.12+). Old indices in S3 with on-demand restore. Cuts memory by 90% for cold shards.
  3. Hot/warm/cold node pool. Different instance classes per tier; cold uses spinning disk + memory fraction of hot.
  4. Drop unused fields from index. Index 5 fields, not 50.

Anti-patterns

  • Index every field — half the field types contribute nothing to queries but dominate index size.
  • Single-tier cluster with 90 days of data hot — paying for memory on data nobody queries.
  • Replica count = 3 on cold data — 3× the storage and memory.

Cache-Heavy (Social Feeds, Timelines, Read-Amplifying Stores)

Cost shape

  • Memory-resident cache + KV nodes: 50–70%. Per-user inboxes, hot keys, working sets that must live in RAM for sub-10 ms p95 reads. Redis / Memcached / Pelikan cluster dominates the bill.
  • Storage (warm tier on NVMe / KV behind cache): 10–25%. Where the long-tail of user state goes once it falls out of the hot tier.
  • App tier compute: 15–30%. Stateless front-ends doing assembly + ranking; usually grows with read QPS.
  • Cross-AZ egress (cache replication, fanout writes): 1–5%. Becomes load-bearing only at very large fanout factors.
  • System-of-record DB: 5–15%. Often a small fraction once the cache absorbs > 90% of reads.

The optimization order

  1. Tier the cache. Hot RAM (recent / active subset) → warm NVMe (long tail) → cold rebuild from system-of-record. Per-byte cost drops ~10× per tier; typical savings 2–4× total.
  2. Right-size the hot tier per access pattern. Most reads consume only the head of each per-user state; capping the hot tier to that head (vs full state) is often a 5–10× memory reduction.
  3. Replicate hot keys, don't shard them. Read-hot keys cannot be split (one user's celebrity feed); replicate 5× and route via deterministic subsetting / Maglev.
  4. Single-flight + jittered TTL. Prevents stampede on TTL expiry concentrating origin QPS spikes; cite single-flight.
  5. Edge-resident filters for small per-user state. Bloom mute filter on cookie removes 1 op/req from the backend at zero infra cost. Cite cross-cutting#edge.

Anti-patterns

  • Caching everything for the same TTL — hot keys and cold keys have different cache economics; uniform TTL wastes RAM on the cold long-tail.
  • Sharding read-hot keys via salting — the read-hot key has one canonical answer; salting helps write-hot, not read-hot.
  • Treating the cache as the system of record — losing cache → data loss, not just latency degradation. Always have a regenerable backing store.
  • Sizing the cache to fit the entire working set — long-tail cardinality means the working set has no upper bound at consumer scale; tier instead.

Canonical scenarios

41-home-timeline-fanout; 21-distributed-cache.
