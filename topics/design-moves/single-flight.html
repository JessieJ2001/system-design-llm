Single-Flight on Miss

When N concurrent requests miss the same key, only one fetches; the rest wait or get stale-while-revalidate. The canonical defense against cache stampede, dogpile, and any "thundering herd to a slow origin" pattern. This is a source-of-truth page; scenarios link here instead of redefining the move.

~6 min reference

When to Reach for It (the Trigger)

You should recognize this move from the shape of the bottleneck, not from the name of the system. The trigger is any of:

  • Read-heavy + power-law + cacheable origin. A hot key's TTL expires; the next millisecond, every reader misses simultaneously and stampedes the origin. The origin's QPS for one key jumps from 1/TTL to N · QPSread.
  • Slow or expensive origin. Origin call costs > 100 ms or > $0.001/call (e.g., LLM inference, ML feature compute, full-text search backend, S3 GET). The cost of N parallel origin calls dwarfs the cost of N − 1 waits.
  • Idempotent fetch. The origin call is read-only or naturally idempotent — the answer doesn't depend on which caller asked.
  • Coalescable result. The fetched value is small enough to broadcast to the waiting callers (not a streaming/chunked response).

Negative trigger — do not reach for single-flight when:

  • The fetch is per-caller (personalized, authenticated to the caller).
  • The origin is cheap (sub-ms in-process compute) — the lease overhead exceeds the saving.
  • You'd block on the slowest waiter and the SLO doesn't tolerate the lease-holder's latency.

Mechanism (One Paragraph)

On miss for key k, the cache atomically issues a lease token to the first caller (CAS / SETNX / Redis SET NX EX). The lease-holder fetches from origin and writes back to the cache. Concurrent callers for the same k see "lease pending" and either (a) wait (block on a Redis pub/sub or short-poll the cache) or (b) immediately return the previous stale value (stale-while-revalidate) and let the lease-holder refresh in the background. Lease has a short TTL (typ. 2–10 s) so a crashed lease-holder doesn't permanently block the key. Lease-holder must release the lease atomically on success or failure.

Worked Example: Cache Stampede Math

Suppose a hot key (one celebrity profile) has read QPS = 50,000. Origin fetch latency = 200 ms. TTL = 60 s.

Without single-flight

At each TTL expiry:
  Concurrent in-flight readers during the 200 ms fetch window:
    50,000 QPS × 0.200 s = 10,000 readers all miss

  All 10,000 issue origin GETs in parallel.
    Origin sees a 10,000-RPS spike on this one key
    (vs steady-state 1/60 = 0.017 RPS)
    Spike multiplier = 10,000 / 0.017 = 600,000×

  Origin must be sized for the spike, not the steady state.
  If the origin can serve 1,000 RPS, the cache layer just DDoS'd it.

With single-flight (lease + wait)

At each TTL expiry:
  First miss acquires lease (Redis SET NX EX 5).
  10,000 − 1 = 9,999 concurrent readers see lease present, wait on pub/sub.
  Lease-holder fetches from origin (200 ms), writes to cache, publishes.
  All 9,999 waiters wake, read from cache, return.

  Origin sees: 1 RPS during the 200 ms window.
  Worst-case latency for waiters: 200 ms (= origin latency, not origin × N).
  Origin spike multiplier = 1 (vs 600,000× without).

With single-flight (stale-while-revalidate)

At TTL expiry, instead of evicting, mark "stale, refresh-in-progress."
  First miss acquires lease and refreshes in background.
  9,999 concurrent readers get the *stale* value immediately (zero added latency).
  When refresh completes (200 ms later), cache flips to the new value.

  Origin spike: 1 RPS.
  Reader p99 latency: unchanged from cache hit case.
  Cost: readers see up to 200 ms-old data during the refresh window.

Alternatives Considered

AlternativeMechanismWhy pick itWhy not pick it
Single-flight (lease + wait) First miss locks; others wait. Strong correctness; one origin call per TTL. Waiters absorb origin latency; lease-holder death blocks for TTL.
Stale-while-revalidate Serve stale; one background refresh. Zero added p99; ideal for cacheable read-mostly content. Up to TTL of staleness visible. Unacceptable for strong-consistency reads.
Refresh-ahead Background job refreshes before TTL expires. No miss ever happens for predictably hot keys. Wastes refresh budget on keys that won't be read again. Pair with hot-key detector.
Probabilistic early expiration Each reader independently rolls a die to refresh slightly before TTL. No coordination needed; works without a lease store. Doesn't bound origin RPS — just smooths the spike. Not enough at extreme scale.
Negative caching only Cache "miss" with short TTL. Defends against floods of invalid keys, not stampedes on valid hot keys. Solves a different problem. Often deployed alongside single-flight.
Do nothing; over-provision origin Size origin for the worst-case stampede. Simplest; works for cheap origins. Cost. For a $0.01-per-call origin and 10,000× spikes, this is the company's biggest line item.

What Breaks (and How to Recognize It in a Design)

  • Lease-holder death. Lease TTL is your bound on how long a key can be blocked. Pick TTL = 2× expected fetch latency; on TTL expiry, next caller acquires a fresh lease.
  • Lease store is itself a stampede target. Single-flight requires an atomic CAS — typically Redis. If every miss in your fleet hits one Redis key, you've moved the stampede from origin to Redis. Mitigate by making the lease store the same shard as the cache value (one round-trip, one node).
  • Wait queue unboundedness. Waiters can pile up indefinitely if origin hangs. Cap waiter count per key; shed beyond cap with 503 + Retry-After.
  • Wrong granularity. Lease per key is correct. Lease per endpoint serializes unrelated requests. Lease per caller defeats the purpose.
  • Personalization sneaks in. The day someone adds a per-caller header to the cache key, single-flight stops coalescing because every caller has a unique key. Audit cache key composition when adding new request dimensions.

Per-Node Complexity

  • Lease acquire: O(1) — single CAS.
  • Wait: O(1) memory per waiter (one pub/sub subscription); O(0) origin cost.
  • Origin call amplification: N concurrent readers → 1 origin call (vs N without).
  • Steady-state lease overhead: 2 extra Redis ops per miss (acquire + release). Negligible vs origin call.
