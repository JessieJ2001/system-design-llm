Microservices Architecture

Decompose a monolithic application into a suite of small, independently deployable services, each owning its own data and communicating through well-defined APIs.

30 min read

What Is It?

Microservices architecture is an architectural style where an application is composed of many small, loosely coupled services. Each service is responsible for a single business capability, runs in its own process, and communicates with other services through lightweight mechanisms such as HTTP REST APIs, gRPC calls, or asynchronous message passing. Critically, each service owns its own data store — there is no shared database.

This stands in contrast to a monolith, where the entire application is built, deployed, and scaled as a single unit. In a monolith, all modules share the same process, the same memory space, and typically the same database. Changes to one module require redeploying the whole application. Microservices trade this simplicity for independent deployability: you can update the payment service without touching the user service, scale the search service independently from the checkout service, and even use different programming languages or databases for different services.

The term was popularized around 2014 by Martin Fowler and James Lewis, though the ideas trace back to service-oriented architecture (SOA). The key difference from SOA is that microservices emphasize smaller service granularity, decentralized governance, and lightweight communication protocols rather than heavyweight enterprise service buses (ESBs).

How It Works

1. Service Boundaries & Domain-Driven Design

Each microservice is aligned with a bounded context from domain-driven design (DDD). A bounded context is a logical boundary within which a particular domain model is defined and applicable. For example, an e-commerce platform might have bounded contexts for Orders, Inventory, Payments, and Users. Each becomes its own service with a clear API contract.

Concrete Example: E-Commerce Bounded Contexts

Consider an e-commerce platform. At first glance, "orders" and "inventory" seem closely related — after all, placing an order reduces inventory. But they are separate bounded contexts for several important reasons:

  • Different data models: The Orders context cares about line items, shipping addresses, order status, and customer payment details. The Inventory context cares about SKU quantities, warehouse locations, reorder thresholds, and supplier lead times. An "item" in Orders is an order line with a price and quantity; an "item" in Inventory is a stock-keeping unit with a bin location and restock schedule. Same word, different meaning — this is exactly what a bounded context captures.
  • Different teams: The Orders team focuses on checkout flow, cart management, and order lifecycle. The Inventory team focuses on warehouse operations, stock reconciliation, and supplier integration. These teams have different release cadences and different priorities.
  • Different scaling needs: During a flash sale, the Orders service might see a 50x traffic spike. The Inventory service sees a spike too, but a much smaller one (batch reservation vs. per-request writes). If they are combined, you must scale the entire combined service to handle the Orders spike, wasting resources on Inventory capacity you don't need.

What happens when they are combined: Suppose Orders and Inventory share a single service and database. A schema migration to add a new column to the inventory table requires redeploying the combined service. The Orders team must coordinate with the Inventory team for every release. A bug in inventory restock logic can take down checkout. Database connection pool exhaustion from inventory batch jobs degrades order placement. You lose all the benefits of independent deployment.

Bounded Context Core Entities Owns Data Key Operations Scaling Profile
Orders Order, LineItem, OrderStatus orders, order_items, order_history Create order, update status, cancel, refund Spiky (flash sales, promotions)
Inventory SKU, StockLevel, Warehouse stock_levels, warehouses, reorder_rules Reserve stock, release stock, restock, transfer Moderate, batch-heavy (nightly reconciliation)
Payments Payment, Refund, PaymentMethod payments, refunds, payment_methods Charge, refund, verify, tokenize Matches Orders (1:1 with checkout)
Users User, Profile, Address users, profiles, addresses, preferences Register, authenticate, update profile Steady, read-heavy (profile lookups)
Shipping Shipment, Carrier, TrackingEvent shipments, tracking_events, carrier_rates Create shipment, track, estimate delivery Grows with fulfilled orders, async

Notice that each context has its own vocabulary. A "User" in the Users context has authentication credentials and preferences. An "Order" in the Orders context references a user by ID but does not store passwords or preferences. The Orders context has a simplified view of a user — just enough to fulfill the order. This is the DDD concept of an anti-corruption layer: each context translates external concepts into its own internal language, preventing one context's model from "corrupting" another's.

2. Data Ownership

Each service owns its data exclusively. The User Service has its own user database; the Order Service has its own orders database. No other service can directly query another service's database. If the Order Service needs user information, it calls the User Service API. This strict data isolation prevents tight coupling and allows each service to choose the storage technology that best fits its needs (polyglot persistence).

Data Ownership Migration: A 4-Phase Plan

Migrating from a shared database to per-service data ownership is one of the hardest parts of a microservices transition. Doing it in one step risks data loss and downtime. Here is a safe, phased approach:

Phase Duration What Changes Risk Level
Phase 1: Dual Reads Weeks 1–2 Both the old monolith and the new service read from the shared database. The new service has no database of its own yet. This validates that the new service can serve the correct data without any data migration. Write path is unchanged — only the monolith writes. Low — read-only, no data moves
Phase 2: CDC Sync Weeks 3–6 The new service gets its own database. A Change Data Capture (CDC) pipeline (using Debezium, DMS, or a custom solution) continuously replicates data from the shared DB into the new service's DB. The new service starts writing to its own DB. Both databases are kept in sync via CDC. Run shadow reads: read from both, compare results, alert on differences. Medium — two data sources exist
Phase 3: API Cutover Weeks 7–8 The old monolith (and any other services that were reading from the shared tables) now reads from the new service's API instead of querying the database directly. The new service is the single source of truth. CDC still runs as a safety net. Monitor for any direct DB queries that slipped through (they'll appear as connections in the shared DB's connection logs). Medium — changing read paths
Phase 4: Decommission Week 9+ Stop the CDC pipeline. Drop the old shared tables (after a backup retention period). The new service fully owns its data. Clean up database permissions to ensure no other service can access the old tables even if they still exist temporarily. Low — cleanup only

What happens if you skip Phase 2 (CDC sync)?

If you jump directly from "shared DB" to "new service owns all data" without the CDC sync phase, you face a dangerous gap: during the cutover moment, writes are going to the new database but the data from the old database hasn't been fully migrated yet. Queries return empty results for historical data. Orders placed yesterday vanish. Customer addresses are gone. The only recovery option is an emergency rollback, which itself risks losing any new data written to the new database during the failed cutover. The CDC phase eliminates this gap by keeping both databases continuously synchronized throughout the migration.

3. API Contracts (REST and gRPC)

Services communicate through well-defined API contracts. REST over HTTP/JSON is the most common approach for its simplicity and wide tooling support. gRPC (using Protocol Buffers) is preferred for internal service-to-service communication where performance matters — it is binary-encoded, supports streaming, and generates strongly typed client libraries automatically. The key principle: contracts are versioned and backward-compatible.

4. Synchronous vs. Asynchronous Communication

Synchronous (request/response): Service A calls Service B and waits for a response. Simple to reason about, but creates temporal coupling — if Service B is down, Service A's request fails. Use for queries where the caller needs an immediate answer.

Asynchronous (event-driven): Service A publishes an event to a message broker (Kafka, RabbitMQ). Service B consumes the event when it is ready. This decouples services in time — Service B can be temporarily unavailable without affecting Service A. Use for commands, notifications, and data synchronization across services.

5. Deployment Independence

Each service is packaged as its own deployable artifact (typically a Docker container), has its own CI/CD pipeline, and can be deployed independently of other services. This means the team owning the User Service can deploy three times a day without coordinating with the Payments team. Deployment independence is the primary operational benefit of microservices.

When to Use / When NOT to Use

Use Microservices When

  • Large engineering teams need to deploy independently. Multiple teams (3+) stepping on each other in a monolith's codebase is a strong signal.
  • Different services need different scaling profiles. Your search service is CPU-bound and needs 20 instances; your user profile service handles 1/10th the traffic.
  • Polyglot persistence is needed. Some services need a graph database, others need a document store, others need a relational database.
  • Organizational boundaries align with service boundaries (Conway's Law). If your teams are already organized around business capabilities, microservices formalize that.
  • You need fault isolation. A bug in the recommendation engine should not take down the checkout flow.

Don't Use Microservices When

  • Your team is small (<10 engineers). The operational overhead of microservices (deployment pipelines, monitoring, debugging) exceeds the benefit. Use a modular monolith instead.
  • The domain is simple and doesn't have clear bounded contexts. If everything is tightly interconnected, splitting into services creates a distributed monolith.
  • You can't afford the operational complexity. Microservices require service discovery, distributed tracing, container orchestration, and sophisticated CI/CD. If you don't have this infrastructure, you'll spend more time fighting tooling than shipping features.
  • You're an early-stage startup iterating fast. Your domain model is changing weekly. Redefining service boundaries is far more expensive than refactoring modules in a monolith.
  • Strong ACID transactions across multiple entities are required. Microservices force eventual consistency and the Saga pattern, which is complex.

Classic vs. Modern Approaches

Classic: REST-Based Microservices

The original microservices pattern (circa 2012-2015): each service exposes a REST API, services call each other directly via HTTP, service discovery uses a registry like Eureka or Consul, and each team manages their own load balancer and health checks. This works, but as the number of services grows, cross-cutting concerns (retries, circuit breaking, mTLS, observability) get duplicated across every service.

Service Mesh (Istio, Linkerd)

A service mesh extracts all networking concerns into a sidecar proxy (typically Envoy) that sits next to every service instance. The mesh handles mTLS, retries, circuit breaking, rate limiting, and distributed tracing transparently. Services don't need any networking libraries — they just make plain HTTP calls to localhost, and the sidecar handles the rest. This solves the "library bloat" problem of classic microservices but adds latency per hop (typically 1-3ms) and significant operational complexity.

How a Service Mesh Works: Sidecar Proxy Architecture

In a service mesh, every service pod gets an additional container — the sidecar proxy (typically Envoy). The application container never talks directly to the network. Instead, all inbound and outbound traffic is intercepted by the sidecar using iptables rules. Here is the request flow:

  1. Service A wants to call Service B. It sends an HTTP request to http://service-b:8080/api.
  2. Service A's sidecar intercepts the outgoing request. It handles: (a) TLS encryption (mTLS — mutual TLS, where both sides present certificates), (b) service discovery (resolving service-b to an actual pod IP), (c) load balancing across Service B's instances, (d) retry policy, (e) recording metrics (request count, latency, error rate), (f) injecting trace headers.
  3. Service B's sidecar receives the encrypted request. It handles: (a) TLS termination and certificate verification, (b) rate limiting, (c) recording inbound metrics, (d) forwarding the plain HTTP request to the application on localhost:8080.
  4. Service B's application processes the request and returns a response. The response traverses both sidecars in reverse.

Latency cost:

Each sidecar adds 1–3ms of processing (TLS, header inspection, metrics recording). Since both the source and destination have sidecars, a single service-to-service call adds 2–6ms total. For most requests (50–500ms), this is negligible. But in a chain of 5 serial calls, the mesh adds 10–30ms — which can be significant for tight SLO budgets.

What the sidecar handles (so your application doesn't have to):

Concern Without Mesh (library-based) With Mesh (sidecar)
mTLS encryption Each service manages its own TLS certificates, key rotation, trust store. Must be implemented per language. Sidecar handles certificate issuance, rotation, and TLS termination automatically. Application uses plain HTTP to localhost.
Retries & circuit breaking Each service imports a resilience library (e.g., resilience4j for Java, polly for .NET). Configuration scattered across services. Configured centrally in the mesh control plane. Applied uniformly to all services without code changes.
Metrics collection Each service instruments its HTTP client and server with metrics middleware. Different languages use different libraries. Sidecar emits standardized metrics (request count, latency histogram, error rate) for every service automatically.
Distributed tracing Each service must import an OpenTelemetry SDK and ensure trace context propagation. Sidecar injects and propagates trace headers. Application still needs to forward headers for in-app spans, but inter-service propagation is automatic.

When is a service mesh worth it?

  • More than ~20 services: Below this, the operational complexity of running the mesh (control plane, sidecar injection, debugging mesh issues) outweighs the benefit. Use libraries instead.
  • Need for mTLS everywhere: If your compliance requirements demand encrypted service-to-service communication, a mesh is far easier than managing certificates per service.
  • Polyglot services: If you have services in Java, Go, Python, and Node.js, implementing consistent retry/circuit-breaking/tracing in four language ecosystems is painful. The mesh provides these uniformly regardless of language.
  • Standardized observability: If you need consistent metrics and tracing across all services without relying on each team to instrument correctly.

Memory overhead: Each Envoy sidecar consumes approximately 50–100MB of RAM. With 200 service instances, that is 10–20GB of cluster memory dedicated entirely to sidecars. Factor this into your capacity planning.

Event-Driven Microservices (CQRS, Event Sourcing)

Instead of synchronous REST calls, services communicate primarily through events. CQRS (Command Query Responsibility Segregation) separates the write model from the read model, allowing each to be optimized independently. Event sourcing stores all state changes as an immutable sequence of events, enabling perfect audit trails and temporal queries. This pattern excels for systems with complex business workflows but has a steep learning curve.

Serverless Microservices

Each "service" is a set of cloud functions (AWS Lambda, Google Cloud Functions) triggered by events or HTTP requests. You get automatic scaling to zero (no cost when idle) and per-invocation billing. The tradeoff: cold start latency (100ms-2s), limited execution duration, and vendor lock-in. Best for event-driven workloads with spiky traffic patterns.

The Modular Monolith: A Middle Ground

A modular monolith enforces service boundaries within a single deployable unit. Each module has its own data access layer, its own domain model, and communicates with other modules through well-defined internal APIs. You get the organizational benefits of clear boundaries without the operational overhead of distributed systems. When the time comes, individual modules can be extracted into true microservices with minimal refactoring. This is increasingly seen as the right starting point for most teams.

Assumptions & Model

Key Assumptions

  • Team size justifies the overhead. You have enough engineers to staff separate teams for separate services. If one person owns five services, you've just created five deployment pipelines for no benefit.
  • Infrastructure exists for service discovery, monitoring, and deployment. You need container orchestration (Kubernetes), distributed tracing (Jaeger/Zipkin), centralized logging (ELK/Datadog), and automated CI/CD pipelines per service.
  • The network is reliable enough for inter-service calls. Microservices turn in-process function calls into network calls. You assume the network will be available, and you build retries, timeouts, and circuit breakers for when it isn't.
  • Domain boundaries are well-understood. You can clearly identify which data and behavior belong to which service. If you can't, you'll end up with services that need to change together, defeating the purpose.

When These Assumptions Break Down

  • Distributed monolith anti-pattern: Services are "separate" but must be deployed together because they share a database, use synchronous call chains, or have circular dependencies. You get all the operational complexity of microservices with none of the benefits.
  • Shared database between services: If two services read/write the same tables, they're coupled at the data layer. Schema changes in one service break the other. This is the single most common microservices anti-pattern.
  • Synchronous call chains: If Service A calls B, which calls C, which calls D synchronously, the overall latency is the sum of all calls, and the availability is the product of all availabilities. A chain of five services each at 99.9% availability gives you 99.5% — a 4.4-hour annual downtime increase.

Sizing & Tradeoffs

How Small Should a Service Be?

The size of a microservice is not measured in lines of code. It is bounded by three constraints:

  • Team size (the 2-pizza rule): A service should be owned by a team of 4-8 people. Larger and communication overhead grows; smaller and on-call burden is unsustainable.
  • Domain boundary: A service should map to exactly one bounded context. If it spans multiple contexts, it should be split. If it's a fragment of a context, it's too small.
  • Deployment independence: You should be able to deploy this service without coordinating with any other team. If you can't, the boundary is wrong.

The Flexibility vs. Overhead Tradeoff

More services = more flexibility in scaling, deploying, and technology choices. But more services also = more network calls, more deployment pipelines, more monitoring dashboards, more things that can fail independently.

SLO-Driven Sizing Examples

The right approach to microservice sizing starts with Service Level Objectives (SLOs), not intuition. Work backwards from the end-to-end SLO to derive per-service budgets, instance counts, and redundancy requirements.

Example 1: E-Commerce Checkout Flow

SLO: End-to-end p99 latency < 500ms across 5 serial services: API Gateway → Cart → Pricing → Inventory → Payment.

Per-service budget derivation: With 5 serial hops, each service gets a budget of 500ms ÷ 5 = 100ms at p99. But network overhead per hop is ~5-10ms, consuming 25-50ms total. Adjusted per-service processing budget: ~80-90ms at p99.

Instance sizing: If each service handles 1,000 RPS at checkout peak and each request occupies a thread for ~80ms, you need at least 1000 × 0.08 = 80 concurrent threads. With a target utilization of 60% to protect tail latency, that is 134 threads, or roughly 9 instances at 15 threads each (plus 2 hot spares for rolling deploys).

Key insight: The Payment service calls an external gateway (Stripe, Adyen) with p99 ~200ms. This blows the 100ms budget. Solution: allocate asymmetrically — give Payment 200ms and reduce Cart and Pricing to 50ms each by adding a Redis cache layer in front of them.

Example 2: Social Media Feed

SLO: p95 < 200ms at 100,000 RPS for the home feed endpoint.

Architecture: Feed Aggregator fans out to 3 services in parallel: Post Service, Social Graph, and Ranking Service. Since calls are parallel, the latency budget is determined by the slowest of the three, not the sum.

Per-service budget: With 200ms total and ~20ms for aggregation + serialization, each parallel service gets 180ms at p95. The Social Graph service has a p95 of 150ms (graph traversal) — it is the critical path.

Connection pool sizing: At 100K RPS, the Feed Aggregator opens 3 connections per request (one to each downstream). With 180ms max hold time, each connection is held for 0.18s. Pool size per downstream = 100,000 × 0.18 / N_instances. With 50 aggregator instances, that is 360 connections per instance per downstream. Set pool max to 400 with a 200ms acquire timeout to fail fast rather than queue.

Service mesh overhead: An Envoy sidecar adds ~1-2ms per hop. With 3 parallel calls, this is negligible (only adds to wall clock once since calls are concurrent), but the memory overhead is 50-100MB per pod × hundreds of pods = significant cluster cost.

Example 3: Payment Processing Pipeline

SLO: 99.999% availability (five nines) — at most 5.26 minutes of downtime per year.

Availability math: If the pipeline has 4 serial services each with independent availability, compound availability = A1 × A2 × A3 × A4. To achieve 99.999% end-to-end, each service must be at least 99.99975% available (since 0.99999754 ≈ 0.99999).

Redundancy derivation: A single instance with 99.9% availability needs N redundant instances where 1 - (1 - 0.999)N ≥ 0.9999975. Solving: N = 3 gives 1 - 10-9 — well above target. Each service runs minimum 3 replicas across 3 availability zones, with no more than 1 replica per AZ.

Blast radius isolation: The Fraud Detection service calls a third-party ML model with only 99.5% availability. This single dependency would drag the pipeline to 99.5%. Solution: run the fraud check asynchronously — approve the payment synchronously, flag for async fraud review, and reverse if fraud is detected. This removes the dependency from the critical path, keeping the synchronous pipeline at five nines.

Example 4: Ride-Sharing Matching

SLO: p99 < 300ms for match + dispatch across 4 services: Location Ingestion → Matching Engine → Pricing Service → Dispatch Service.

Dependency chain analysis: Location Ingestion is a hot path processing 500K GPS updates/sec. It writes to an in-memory geospatial index (not a database — too slow). The Matching Engine reads this index, so these two services are tightly coupled by data. Co-locate them in the same availability zone and use shared memory or a low-latency protocol (Unix domain sockets, ~0.05ms overhead vs. 2-5ms TCP).

Budget allocation: Location lookup: 20ms | Matching algorithm: 150ms (computationally intensive) | Pricing calculation: 50ms | Dispatch notification: 80ms. Total: 300ms. There is zero slack, so the Matching Engine must be over-provisioned to ensure its p99 stays under 150ms even during surge events (2-3x normal traffic at concert endings, sports events).

Surge sizing: During 3x surge, the Matching Engine needs 3x compute. Pre-provision is wasteful. Use predictive auto-scaling based on event calendars: scale up 15 minutes before a known event ends. For unpredictable surges, keep a warm pool of 30% additional capacity and use Kubernetes HPA with a custom metric (matches-per-second) rather than CPU.

SLO-Driven Capacity Planning

Traditional capacity planning asks "how many instances do we need?" SLO-driven planning asks "what latency and availability budgets constrain each service, and how do those budgets translate into resource requirements?"

The process:

  1. Define the end-to-end SLO for each critical user journey (checkout, feed load, search). Express it as latency percentiles (p50, p95, p99) and availability targets.
  2. Map the call graph for each journey. Identify which calls are serial (latency adds up) and which are parallel (latency is the max). This is why the interactive graph above matters: serial chains are latency-multiplicative.
  3. Allocate per-service budgets. Subtract network overhead, then distribute the remaining budget. Services with external dependencies (payment gateways, ML models) get larger allocations. Services that can be cached get smaller ones.
  4. Derive instance counts from budgets. Use Little's Law: concurrent_requests = throughput × latency. Divide by per-instance capacity (threads, event loop throughput) to get instance count. Add a 40-60% headroom factor to protect tail latency.
  5. Validate with load testing. Generate synthetic traffic at 1.5x peak and measure actual percentiles. If a service exceeds its budget, either optimize it, allocate more instances, or redistribute the budget from services with headroom.
  6. Set SLO-based alerts. Alert when a service consumes >80% of its latency budget (burn rate alert), not when it "feels slow." This shifts operations from reactive firefighting to proactive budget management.

The compounding trap: In a serial chain of N services, if each service has a 1% chance of a latency spike (>2x budget), the probability that at least one service spikes is 1 - 0.99N. For N=5, that is 4.9% of requests. For N=10, it is 9.6%. This means your p95 latency is dominated by outliers from any service in the chain. The more services in a serial path, the more aggressively you must control per-service tail latency — not just the median.

Rule of Thumb

Keep synchronous call chains to a depth of 3 or fewer. If a request path touches more than 3 services synchronously, consider whether some of those calls can be made asynchronous (via events), parallelized with fan-out, or whether the services should be merged. Use the SLO budget as the arbiter: if you cannot meet the end-to-end SLO with the current chain depth, reduce the depth.

Scaling Journey

0 – 1K Users

Monolith

A single deployable application. All logic in one codebase, one database. Deploy with a simple git push. This is the right choice — you're still finding product-market fit.

1K – 100K Users

Modular Monolith

Enforce clear module boundaries inside the monolith. Each module has its own data access layer and internal API. No cross-module database queries. This prepares you for extraction without the distributed systems overhead.

100K – 10M Users

First Extractions (Strangler Fig)

Extract the most painful modules first — usually the ones that need independent scaling (e.g., search, media processing). Use the strangler fig pattern: route traffic to the new service while the old module still exists, then cut over. Shared infrastructure: common logging, tracing, deployment tooling.

10M+ Users

Full Microservices + Service Mesh

Dozens to hundreds of services. Service mesh (Istio/Linkerd) handles cross-cutting concerns. Platform team provides golden paths for service creation. Internal developer portal for service discovery. Dedicated SRE teams for infrastructure services.

What Can Go Wrong

1. Cascading Failures

What happened: A single service (e.g., the recommendation engine) becomes slow due to a database connection pool exhaustion. Every upstream service calling it gets blocked waiting for responses. Thread pools fill up across the call chain. Within minutes, the entire platform is unresponsive — even though the root cause is a single misbehaving service.

Root cause: No circuit breakers, no timeouts (or timeouts set too high), and synchronous call chains creating a blast radius far larger than the failing component.

Prevention: Circuit breakers on every inter-service call. Aggressive timeouts (fail fast). Bulkhead pattern to isolate thread pools per dependency. Async communication where possible to decouple failure domains.

Cascading Failure Timeline: A Detailed Trace

Here is exactly how a single slow service can take down an entire platform, second by second:

Time Event Impact
T=0s Recommendation Service database connection pool exhausts. Query latency jumps from 50ms to 2,000ms. Recommendation Service still responds, but 40x slower. From the outside, it looks "healthy" (no errors, just slow).
T=1s Product Service calls Recommendation Service synchronously to fetch "related products." Each request now holds a thread for 2s instead of 50ms. Product Service thread pool starts filling: 12/50 threads occupied (normally 2/50 at this traffic rate).
T=3s At 25 requests/sec, threads accumulate faster than they drain. Thread pool: 50/50 threads occupied, all blocked waiting on Recommendation Service. Product Service can no longer accept ANY requests — even ones that don't need recommendations (like product detail lookups).
T=5s Product Service responds to all new requests with HTTP 503 (Service Unavailable) or times out. Cart Service, which calls Product Service to validate items, starts failing too. Its thread pool begins filling.
T=7s API Gateway's connection pool to Product Service is exhausted. Health checks start failing. API Gateway returns 503 for ALL product-related endpoints. Users see blank product pages, broken search results, and failed checkout.
T=10s Users frantically refresh. Traffic to API Gateway increases 3x (retry storm). API Gateway's own resources are strained. Even User Service and other healthy services experience degraded response times due to shared infrastructure load.

The root cause: One database connection pool in one service. The blast radius: The entire platform. This is why circuit breakers, timeouts, and bulkheads are not optional in a microservices architecture.

Availability Math: Services in Series

When services are called synchronously in sequence, their availabilities multiply. If any single service is down, the entire request fails:

  • 1 service at 99.9% availability = 99.9% (8.76 hours downtime/year)
  • 2 services at 99.9% each = 0.999 × 0.999 = 99.8% (17.5 hours/year)
  • 3 services at 99.9% each = 0.9993 = 99.7% (26.3 hours/year)
  • 5 services at 99.9% each = 0.9995 = 99.5% (43.8 hours/year)
  • 10 services at 99.9% each = 0.99910 = 99.0% (87.6 hours/year — 3.6 full days!)

This is the fundamental reason to keep synchronous call chains short and use asynchronous communication wherever possible. Every additional synchronous hop degrades your end-to-end availability.

2. The Distributed Monolith

What happened: A team "migrated to microservices" by splitting their monolith into 15 services. But the services share a database, must be deployed in a specific order, and a change to the User schema requires updating 8 services simultaneously.

Root cause: Service boundaries were drawn along technical layers (API, business logic, data access) rather than business capabilities. Data ownership was not enforced.

Prevention: Enforce the rule: each service owns its own database. Draw boundaries along business domains, not technical layers. If two services always change together, they should be one service.

3. Data Consistency Nightmares

What happened: A user places an order. The Order Service saves the order, then calls the Inventory Service to decrement stock. The Inventory call fails due to a network timeout. Now there's an order with no corresponding inventory reservation. Or worse: the Inventory Service actually processed the request, but the response was lost, so the Order Service retries, and stock is double-decremented.

Root cause: Distributed transactions across service boundaries without a proper coordination mechanism.

Prevention: Use the Saga pattern with compensating transactions. Make all operations idempotent using idempotency keys. Use outbox pattern to guarantee event publishing. Accept eventual consistency as a design principle.

Debug Playbook

User reports slow response:

1. Check distributed trace (Jaeger/Zipkin) for the request → identify which service is slow

2. Is it one service? → Check that service's CPU/memory/GC metrics → Check its database query latency

3. Is latency spread across multiple services? → Check network latency between services → Check if a downstream dependency is slow (cascading)

4. Is the error rate high? → Check circuit breaker state → Check if a recent deployment caused it (rollback if so)

5. Is it intermittent? → Check for retry storms → Check for connection pool exhaustion → Check for GC pauses

Critical Configuration

Setting Recommended Value Why
timeout 2-5 seconds (per call) Prevents thread exhaustion from slow downstream services. Must be shorter than caller's timeout (timeout cascading).
circuit_breaker_threshold 50% failure rate over 20 requests Opens the circuit before a downstream failure cascades upstream. Too sensitive = false positives; too lenient = cascading failures.
retry_policy 3 retries, exponential backoff (100ms, 200ms, 400ms) + jitter Handles transient failures without creating retry storms. Always add jitter to prevent thundering herd on retries.
connection_pool_size 10-50 per downstream service Too small = connection queueing under load. Too large = resource waste and can overwhelm downstream services.
health_check_interval 5-10 seconds Balances fast failure detection against health check traffic overhead. Use both liveness and readiness probes in Kubernetes.

Interactions & Pitfalls

Microservices + Distributed Transactions

You cannot use traditional ACID transactions across microservice boundaries (there's no shared database to coordinate). Instead, use the Saga pattern: a sequence of local transactions, each publishing an event that triggers the next step. If any step fails, compensating transactions undo the previous steps. Choreography-based sagas (events) are simpler but harder to debug; orchestration-based sagas (a central coordinator) are easier to reason about but introduce a single point of coordination.

Saga Deep Dive: Order Creation Example

Let's trace through a concrete order creation saga with three participating services. The happy path has three forward steps; the failure path triggers compensating actions in reverse order.

Happy Path (all steps succeed):

Step Timestamp Service Action Local DB State
1 T=0ms OrderService Create order #501 with status=PENDING orders: {id:501, status:PENDING, items:[SKU-A ×2], total:$59.98}
2 T=45ms InventoryService Reserve 2 units of SKU-A (stock: 100 → 98) reservations: {order:501, sku:SKU-A, qty:2}, stock_levels: {SKU-A: 98}
3 T=120ms PaymentService Charge $59.98 to card ending 4242 payments: {order:501, amount:$59.98, status:CAPTURED, txn:ch_abc123}
4 T=180ms OrderService Update order #501 status=CONFIRMED orders: {id:501, status:CONFIRMED}

Failure Path (payment fails at Step 3):

Step Timestamp Service Action Result
1 T=0ms OrderService Create order #502 with status=PENDING OK
2 T=45ms InventoryService Reserve 1 unit of SKU-B (stock: 15 → 14) OK
3 T=120ms PaymentService Charge $29.99 to card ending 1234 FAILED — card declined
C2 T=150ms InventoryService Compensate: Release 1 unit of SKU-B (stock: 14 → 15) OK — reservation deleted
C1 T=200ms OrderService Compensate: Update order #502 status=CANCELLED, reason="Payment declined" OK — order cancelled

Choreography approach (event-driven, no central coordinator):

Each service publishes an event after completing its step, and other services react to those events:

Time Event Published Published By Consumed By
T=0ms OrderCreated {orderId:502, items:[SKU-B ×1]} OrderService InventoryService
T=45ms InventoryReserved {orderId:502, reservationId:R-99} InventoryService PaymentService
T=120ms PaymentFailed {orderId:502, reason:"card_declined"} PaymentService InventoryService, OrderService
T=150ms InventoryReleased {orderId:502, reservationId:R-99} InventoryService (logged for audit)
T=200ms OrderCancelled {orderId:502, reason:"payment_declined"} OrderService (notification service sends email)

Problem with choreography: As the number of services grows, the event flow becomes a tangled web. No single place shows the full saga flow. Debugging requires reading logs from all services and mentally reconstructing the sequence.

Orchestration approach (central coordinator):

A Saga Orchestrator (often a separate service or a workflow engine like Temporal) controls the entire flow. It calls each service in sequence, handles failures, and triggers compensation:

Time Orchestrator Action Target Service Response
T=0ms POST /orders {items:[SKU-B], customer:42} OrderService 201 Created {orderId:502, status:PENDING}
T=45ms POST /reservations {orderId:502, sku:SKU-B, qty:1} InventoryService 200 OK {reservationId:R-99}
T=120ms POST /payments {orderId:502, amount:$29.99, card:****1234} PaymentService 402 Payment Failed {reason:"card_declined"}
T=150ms DELETE /reservations/R-99 (compensate) InventoryService 200 OK
T=200ms PATCH /orders/502 {status:CANCELLED} (compensate) OrderService 200 OK

Advantage of orchestration: The entire saga logic lives in one place. You can see the complete flow, add retries, set timeouts per step, and easily add new steps. The tradeoff: the orchestrator is a coordination point (though not a single point of failure if it's replicated and its state is persisted).

Microservices + Service Discovery

Services need to find each other. In a static monolith, everything is at localhost. In microservices, service instances come and go (auto-scaling, rolling deploys). Solutions: client-side discovery (each service queries a registry like Consul/Eureka), server-side discovery (a load balancer fronts each service — the Kubernetes model with kube-proxy and Services), or DNS-based discovery (services register DNS records). The pitfall: stale service registrations causing traffic to dead instances.

Microservices + Observability

In a monolith, a stack trace tells you everything. In microservices, a single user request spans 5-20 services. Without distributed tracing, debugging is nearly impossible. Every request must carry a trace ID propagated through all services (via headers like X-Trace-Id). Tools like Jaeger, Zipkin, or AWS X-Ray assemble traces from spans reported by each service. The pitfall: incomplete traces due to services that don't propagate the trace context, making it appear that requests "disappear" mid-flow.

Trace Propagation: A Concrete Example

Let's trace a single request through four services, showing exactly how trace context propagates via HTTP headers:

  1. Request arrives at API Gateway. No trace headers exist yet. The Gateway generates a new trace_id="abc123" and creates the root span with span_id="s1". It adds headers to the outgoing request:
    traceparent: 00-abc123-s1-01
  2. API Gateway calls Order Service. Order Service reads the incoming traceparent header, extracts trace_id="abc123" and parent_span_id="s1". It creates a new span span_id="s2" with parent="s1". When it calls Payment Service, it forwards:
    traceparent: 00-abc123-s2-01
  3. Order Service calls Payment Service. Payment Service reads the header, creates span_id="s3" with parent="s2". Payment Service also queries its database, creating span_id="s4" with parent="s3".
  4. Each service reports its spans to a collector (Jaeger/Zipkin). The collector assembles all spans with trace_id="abc123" into a single trace tree.

Waterfall Diagram (as seen in Jaeger/Zipkin):

Span Service Operation Start End Duration Visual Timeline (0—150ms)
s1 API Gateway GET /orders/501 0ms 150ms 150ms [================================================]
s2 Order Service getOrder 10ms 140ms 130ms   [==========================================]
s3 Payment Service getPaymentStatus 20ms 80ms 60ms     [====================]
s4 Payment DB SELECT * FROM payments 25ms 75ms 50ms      [================]

Reading this waterfall: the API Gateway span (s1) encompasses the entire request. The Order Service span (s2) is nested inside it. The Payment Service span (s3) is nested inside Order Service. The DB query (s4) is nested inside Payment Service. The bottleneck is immediately visible: the DB query takes 50ms of the total 150ms. If this were slow, you'd see it here.

Critical rule: Every service MUST propagate the traceparent header on all outgoing HTTP calls and message queue publishes. If even one service fails to propagate, the trace breaks — you see two disconnected fragments instead of one complete picture.

Microservices + API Gateway

An API Gateway sits at the edge and routes external requests to internal services. It handles authentication, rate limiting, request/response transformation, and protocol translation (e.g., REST to gRPC). The pitfall: the gateway becomes a monolith itself if too much business logic migrates into it. Keep it thin — routing, auth, and rate limiting only. Also, the gateway is a single point of failure; it must be highly available and horizontally scaled.

Cost Considerations

Cost Analysis

Infrastructure overhead: Each service needs its own compute (containers/pods), monitoring agents, logging pipelines, and often its own database. A 50-service microservices architecture on Kubernetes might cost $5,000-$15,000/month in cloud compute alone (vs. $500-$2,000 for an equivalent monolith), depending on scale.

Engineering overhead: Each service needs its own CI/CD pipeline, its own test suite, its own deployment configuration, its own on-call rotation. For N services, you're maintaining N build pipelines, N sets of integration tests, and N dashboards. The tooling cost is often the dominant expense, not the cloud bill.

Network costs: Inter-service communication generates internal network traffic. At high volumes (millions of RPS), internal load balancer and data transfer charges can add $1,000-$5,000/month. Service mesh sidecars add memory overhead (~50-100MB per pod).

The uncomfortable truth: It is almost always cheaper to run a well-structured monolith until you actually need microservices. The cost of premature decomposition — in engineering time, operational complexity, and cloud spend — far exceeds the cost of refactoring a monolith later. Microservices are an organizational scaling solution, not a cost optimization.

Real-World Examples

Netflix

Netflix operates hundreds of microservices handling over 200 million subscribers. Their migration from a monolithic Java application began in 2009 after a major database corruption incident. Key contributions to the ecosystem include Eureka (service discovery), Hystrix (circuit breaking — now in maintenance mode), Zuul (API gateway), and the Chaos Monkey suite for resilience testing. Their architecture processes over 1 billion API requests per day through their API gateway layer.

Amazon

Amazon's migration to service-oriented architecture began around 2001-2002, driven by a mandate from Jeff Bezos that all teams must expose their functionality through APIs with no exceptions. This architectural decision enabled Amazon to later launch AWS — they realized the internal infrastructure they'd built to support their services could be offered as a product. Amazon's services range from the shopping cart to the recommendation engine to fulfillment, each owned by a "two-pizza team."

Uber's Domain-Oriented Microservice Architecture (DOMA)

By 2020, Uber had grown to over 2,200 microservices and was experiencing the pains of "microservice sprawl": unclear ownership, tangled dependencies, and difficulty understanding how services relate. They introduced DOMA — grouping microservices into larger "domains" (like a Maps domain, a Rides domain) with clear interfaces between domains. Inside a domain, services can call each other freely; across domains, they go through a domain gateway. This is essentially microservices organized into a federation.

Code Example

A minimal service registry and inter-service communication pattern in Python:

import requests
from typing import Dict, List, Optional
import random

class ServiceRegistry:
    """Simple in-memory service registry for demonstration."""

    def __init__(self):
        self._services: Dict[str, List[str]] = {}

    def register(self, name: str, url: str):
        self._services.setdefault(name, []).append(url)

    def deregister(self, name: str, url: str):
        if name in self._services:
            self._services[name] = [u for u in self._services[name] if u != url]

    def discover(self, name: str) -> Optional[str]:
        instances = self._services.get(name, [])
        return random.choice(instances) if instances else None

registry = ServiceRegistry()
registry.register("user-service", "http://user-svc:8001")
registry.register("user-service", "http://user-svc:8002")
registry.register("order-service", "http://order-svc:8003")

def call_service(service_name: str, path: str, timeout: float = 2.0):
    """Call a service with discovery, timeout, and basic retry."""
    url = registry.discover(service_name)
    if not url:
        raise Exception(f"No instances of {service_name} available")
    for attempt in range(3):
        try:
            resp = requests.get(f"{url}{path}", timeout=timeout)
            resp.raise_for_status()
            return resp.json()
        except requests.exceptions.RequestException:
            if attempt == 2:
                raise
    return None

# Usage: Order Service calls User Service
user = call_service("user-service", "/users/42")
print(f"Placing order for {user['name']}")

How to Validate Your Design

Contract Testing

Use Pact (or similar contract testing frameworks) to verify that service APIs conform to their contracts. The consumer defines expectations ("I expect the User Service to return { id, name, email }"), and the provider verifies it satisfies those expectations. Run contract tests in CI — if a provider makes a breaking change, the consumer's tests fail before deployment.

Integration Testing with Service Stubs

Use WireMock or MockServer to create realistic stubs of downstream services. This allows testing a service in isolation without spinning up the entire dependency graph. For end-to-end tests, use a dedicated staging environment with all services deployed, but keep these tests minimal — they are slow and brittle.

Chaos Testing

Kill a service instance during a load test. Does the system degrade gracefully, or does it cascade? Inject network latency between services (using toxiproxy or tc). Introduce partial failures (50% of requests to a dependency fail). Tools: Chaos Monkey (random instance termination), Litmus (Kubernetes-native chaos), Gremlin (commercial chaos platform). The goal: verify that circuit breakers trip, retries work correctly, and fallback responses are served.

Distributed Tracing Validation

Send a synthetic request through the system and verify the resulting trace contains spans from every expected service. If a service is missing from the trace, its instrumentation is broken. Automate this check: "every request that hits the API gateway should produce a trace with at least N spans."

Interview Questions

Design an e-commerce platform — monolith or microservices?

The answer depends on the team size and stage. For a startup with 5 engineers, start with a modular monolith. Define clear module boundaries (User, Product, Cart, Order, Payment, Inventory) with strict internal APIs and separate database schemas per module. This gives you the benefits of clean architecture without distributed systems complexity.

As the team grows past 20-30 engineers and you hit deployment bottlenecks (merge conflicts, long CI pipelines, coordinated releases), begin extracting services. Start with the most independently scalable component — often search or media/image processing. Use the strangler fig pattern: put a reverse proxy in front of the monolith and route specific paths to the new service.

At scale (50+ engineers), you'll likely have services for: Product Catalog, User/Auth, Cart, Orders, Payments (probably integrated with Stripe/PayPal), Inventory, Search (Elasticsearch-backed), Recommendations, Notifications, and Fulfillment. Each owned by a dedicated team with its own deployment pipeline.

Follow-up: How would you handle the shopping cart across the Cart and Inventory services? What consistency guarantees do you need?

How do you handle distributed transactions across microservices?

You cannot use traditional ACID transactions across microservices because there is no shared transaction coordinator. Instead, use the Saga pattern. A saga is a sequence of local transactions where each step publishes an event or command that triggers the next step. If any step fails, compensating transactions are executed to undo the previous steps.

Two approaches: Choreography — each service listens for events and reacts (OrderCreated → InventoryReserved → PaymentProcessed). Simpler but harder to debug as complexity grows. Orchestration — a central saga coordinator tells each service what to do and handles compensation logic. Easier to reason about but introduces a coordination point.

Key requirements: all operations must be idempotent (retrying an operation produces the same result), use the outbox pattern (write events to a local outbox table in the same database transaction, then publish them) to avoid dual-write problems, and design compensating transactions for every step (e.g., if Payment fails, release the Inventory reservation).

Follow-up: What happens if the compensating transaction itself fails? How do you handle that?

What is the strangler fig pattern?

Named after the strangler fig tree that grows around a host tree and eventually replaces it. In software, it's a migration strategy for incrementally replacing a monolith with microservices. You place a routing layer (reverse proxy or API gateway) in front of the monolith. New features are built as microservices. Existing features are gradually extracted one at a time.

The process: (1) Identify a module to extract. (2) Build the new microservice that replicates the module's functionality. (3) Route traffic for that functionality to the new service via the proxy. (4) Verify correctness (you can run both old and new in parallel and compare results). (5) Remove the old code from the monolith. Repeat for the next module.

The advantage: you never do a "big bang" migration. The monolith continues to serve traffic throughout the process. If the new service has problems, you route traffic back to the monolith. Risk is minimized because each step is small and reversible.

Follow-up: How do you handle the database during a strangler fig migration? The monolith and the new service both need the same data.

How do you debug a slow request that spans 10 services?

Start with distributed tracing. Pull up the trace for the slow request in Jaeger or Zipkin. The trace shows a waterfall of spans — one span per service call. Look for the longest span: that's your bottleneck. Common patterns: one service taking 90% of the total time (focus there), or many services each adding moderate latency (architectural problem — too many synchronous hops).

Once you've identified the slow service, check its metrics: CPU utilization, memory pressure, garbage collection pauses, database query latency, connection pool saturation. If it's the database, check the slow query log. If it's CPU, profile the service. If it's a downstream dependency, recursively apply the same analysis.

For intermittent slowness, look at the p99 vs p50 latency. If p99 is 10x the p50, you have tail latency issues — often caused by GC pauses, noisy neighbors, or connection pool exhaustion under load. Consider adding hedged requests (send the same request to two instances, take the first response) for latency-critical paths.

Follow-up: What if the tracing infrastructure itself is the problem — traces are incomplete or missing spans?

Microservices vs. modular monolith — when to choose which?

A modular monolith gives you clean boundaries, independent module development, and clear ownership — without the operational overhead of distributed systems. You still get a single deployment, a single database connection pool, in-process function calls, and simple debugging with stack traces. Choose it when your team is under 30 engineers, your domain model is still evolving, and you don't need independent scaling per module.

Switch to microservices when: (1) deployment conflicts become a daily problem (teams blocking each other), (2) a specific module needs to scale 10x more than others, (3) different modules would benefit from fundamentally different technology stacks, or (4) fault isolation is critical (a bug in one area must not take down the entire system).

The key insight: a well-structured modular monolith is the best preparation for microservices. If your module boundaries are clean, extracting a module into a service is straightforward. If your monolith is a big ball of mud, microservices won't save you — you'll just create a distributed big ball of mud.

Follow-up: How do you enforce module boundaries in a monolith? What stops a developer from directly calling another module's database?

How does Netflix handle microservices at scale?

Netflix runs 700+ microservices processing over 1 billion API calls per day. Their architecture centers on a few key patterns: (1) API Gateway (Zuul) routes and filters all external traffic; (2) Service Discovery (Eureka) allows services to find each other without hardcoded URLs; (3) Circuit Breakers (Hystrix) prevent cascading failures; (4) Chaos Engineering (Chaos Monkey, Chaos Kong) randomly kills services and even entire regions to ensure resilience.

Netflix pioneered the "full cycle developer" model: the team that builds a service also deploys, monitors, and operates it. There is no separate ops team for application services. They also invested heavily in paved roads — opinionated internal tools and templates that make it easy to do the right thing (standard logging format, standard metrics, standard deployment pipeline). Teams that follow the paved road get observability and resilience for free.

A critical lesson from Netflix: they migrated to microservices because they had to (monolith couldn't scale to their growth), not because it was trendy. They also acknowledge it took years, cost hundreds of millions of dollars, and required building an entire internal platform. Most companies don't need this level of decomposition.

Follow-up: Netflix moved away from Hystrix. What replaced it and why?

Key Takeaways

  • Microservices decompose an application into small, independently deployable services, each owning its own data and communicating via APIs.
  • The primary benefit is independent deployment, enabling multiple teams to ship features without coordinating releases.
  • Each service must own its data. A shared database between services is the most common anti-pattern and creates a distributed monolith.
  • Microservices trade monolith simplicity for operational complexity: you need service discovery, distributed tracing, circuit breakers, and sophisticated CI/CD.
  • Start with a modular monolith and extract services only when you have a clear, concrete need (scaling, team independence, fault isolation).
  • Keep synchronous call chain depth to 3 or fewer. Prefer asynchronous, event-driven communication where possible.
  • Use the Saga pattern for distributed transactions, make operations idempotent, and design for eventual consistency.
  • The "right size" for a service is defined by team ownership, domain boundary, and deployment independence — not lines of code.
  • Invest in observability (distributed tracing, centralized logging, per-service metrics) before investing in more services. You cannot debug what you cannot see.
  • Microservices are an organizational scaling pattern, not a technology choice. Use them when your team structure demands it, not because big companies do.
