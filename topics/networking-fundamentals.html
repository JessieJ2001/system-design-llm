Networking Fundamentals

Every system design decision — latency budgets, failure modes, scaling strategies — ultimately rests on how data moves across the network. Understanding networking is not optional; it is the foundation.

30 min read

What Is It?

Networking fundamentals are the core concepts governing how data travels between machines — from one process on your laptop to a server across the planet. For system designers, two protocols dominate: TCP (Transmission Control Protocol) and UDP (User Datagram Protocol). TCP provides reliable, ordered byte streams with built-in flow control and congestion control. UDP is a bare-bones, fire-and-forget protocol: it sends datagrams with no guarantees of delivery, ordering, or duplicate prevention. Nearly every system you will design uses TCP (HTTP, gRPC, database connections), but knowing when UDP is the right choice — and how QUIC merges the best of both — separates good system designers from great ones.

The TCP/IP model (the practical simplification of the 7-layer OSI model) is what matters for system design. The layers you will reason about are: the link layer (Ethernet, ARP), the network layer (IP addressing, routing), the transport layer (TCP, UDP, QUIC), and the application layer (HTTP, gRPC, DNS, TLS). When you draw a system architecture diagram, every arrow between boxes is a network call, and each network call carries the weight of these layers: DNS resolution, TCP handshake, TLS negotiation, serialization, transmission, and deserialization. Each step adds latency and each step can fail.

Why does this matter for system design? Because latency budgets are dominated by network time. A database query might take 2ms to execute, but the network round trip to reach the database could add 0.5ms within the same datacenter — or 70ms if it crosses the Atlantic. Connection management determines whether your service can handle 10 concurrent requests or 10,000. And failure modes in distributed systems are overwhelmingly network failures: packet loss, connection resets, DNS resolution failures, TLS certificate expiry, and the dreaded network partition. Understanding networking gives you the vocabulary and mental models to reason about all of these.

Deep Dive: Sliding Window Algorithm

The sliding window is TCP's core mechanism for achieving high throughput without overwhelming the receiver. Here's exactly how it works, step by step.

The Algorithm

  1. Sender maintains a window of size W (the minimum of cwnd and rwnd). It can send packets with sequence numbers from base to base + W - 1 without waiting for ACKs.
  2. As each packet is sent, it enters the "in-flight" state. A retransmission timer starts for each unacknowledged packet.
  3. When the receiver gets a packet, it sends a cumulative ACK for the highest in-order sequence number received. If packet 3 arrives but packet 2 is missing, the receiver ACKs up to 1 (duplicate ACK).
  4. When the sender receives an ACK, it slides the window forward: base = ACK number. This frees up slots at the right edge of the window, allowing new packets to be sent.
  5. If a timeout occurs (no ACK for the oldest in-flight packet), the sender retransmits that packet and may shrink the window.

Visualizing the Window States

Each packet occupies one of four states: Acknowledged (delivered and confirmed), In-flight (sent, awaiting ACK), Sendable (within window but not yet sent), Blocked (outside window, must wait). Watch as ACKs arrive and the window slides right, allowing new packets to enter the sendable zone.

Python Implementation

This code implements the exact sliding window algorithm shown in the diagram above. Each variable maps directly to what you see in the animation.

"""
TCP Sliding Window Protocol — Sender Side

The window has 4 zones (left to right):
  [ACKed] [In-flight] [Sendable] [Blocked]
  ←─────── window ──────→

- base: left edge of window (oldest unACKed packet)
- next_seq: boundary between in-flight and sendable
- base + window_size: right edge (boundary of sendable/blocked)

Key insight: the sender can keep sending as long as
next_seq < base + window_size, without waiting for ACKs.
"""
import time
import random
from collections import defaultdict

class SlidingWindowSender:
    def __init__(self, total_packets, window_size):
        self.total = total_packets
        self.window_size = window_size  # min(cwnd, rwnd)

        # --- The three pointers that define the window ---
        self.base = 0          # oldest unACKed packet (left edge)
        self.next_seq = 0      # next packet to send
        # right edge = base + window_size

        self.acked = set()              # received ACKs
        self.send_times = {}            # seq -> send timestamp
        self.timeout = 1.0              # retransmit timeout (RTO)

    @property
    def right_edge(self):
        """Right boundary of the window."""
        return min(self.base + self.window_size, self.total)

    def can_send(self):
        """Is there room in the window for more packets?"""
        return self.next_seq < self.right_edge

    def send_next(self):
        """Send the next packet within the window.

        In the diagram: moves a packet from 'Sendable' to 'In-flight'.
        """
        if not self.can_send():
            return None
        seq = self.next_seq
        self.send_times[seq] = time.time()
        self.next_seq += 1
        print(f"  SEND packet {seq}  "
              f"[base={self.base} next={self.next_seq} "
              f"right_edge={self.right_edge}]")
        return seq

    def receive_ack(self, ack_num):
        """Process a cumulative ACK.

        ACK(n) means: 'I have received everything up to n-1,
        send me packet n next.'

        This SLIDES the window: base moves right, opening space
        at the right edge for new packets.
        """
        # Mark all packets up to ack_num as acknowledged
        for seq in range(self.base, ack_num):
            self.acked.add(seq)
            self.send_times.pop(seq, None)

        # Slide the window forward
        old_base = self.base
        self.base = ack_num
        slid = self.base - old_base

        if slid > 0:
            print(f"  ACK {ack_num} → window slid {slid} slots "
                  f"[base={self.base} right_edge={self.right_edge}]")

    def check_timeout(self):
        """Check for timed-out packets and retransmit."""
        now = time.time()
        for seq in range(self.base, self.next_seq):
            if seq not in self.acked and seq in self.send_times:
                if now - self.send_times[seq] > self.timeout:
                    print(f"  TIMEOUT packet {seq} → retransmit")
                    self.send_times[seq] = now
                    return seq  # retransmit this packet
        return None

    def is_complete(self):
        return self.base >= self.total


class SlidingWindowReceiver:
    def __init__(self):
        self.expected = 0       # next expected in-order seq
        self.buffer = set()     # out-of-order packets buffered

    def receive(self, seq, lost=False):
        """Receive a packet and return cumulative ACK.

        The receiver only ACKs the highest in-order sequence.
        Out-of-order packets are buffered but NOT acknowledged
        (triggering duplicate ACKs at the sender).
        """
        if lost:
            print(f"  ✗ Packet {seq} LOST in transit")
            return None

        if seq == self.expected:
            # In order — advance and check buffer
            self.expected += 1
            while self.expected in self.buffer:
                self.buffer.remove(self.expected)
                self.expected += 1
            print(f"  ✓ Received {seq} → ACK {self.expected}")
            return self.expected
        elif seq > self.expected:
            # Out of order — buffer it, send duplicate ACK
            self.buffer.add(seq)
            print(f"  ⚠ Out-of-order {seq} (expected {self.expected})"
                  f" → duplicate ACK {self.expected}")
            return self.expected  # duplicate ACK
        return None  # already received


# --- Run simulation ---
print("=== Sliding Window Simulation ===")
print("Window size = 4, Total packets = 10\n")

sender = SlidingWindowSender(total_packets=10, window_size=4)
receiver = SlidingWindowReceiver()
loss_probability = 0.15  # 15% packet loss
round_num = 0

while not sender.is_complete():
    round_num += 1
    print(f"\n--- Round {round_num} ---")

    # Send all packets within the window
    sent_this_round = []
    while sender.can_send():
        seq = sender.send_next()
        if seq is not None:
            sent_this_round.append(seq)

    # Simulate network: each packet may be lost
    for seq in sent_this_round:
        lost = random.random() < loss_probability
        ack = receiver.receive(seq, lost=lost)
        if ack is not None:
            sender.receive_ack(ack)

    # Check for timeouts on unACKed packets
    timed_out = sender.check_timeout()
    if timed_out is not None:
        ack = receiver.receive(timed_out)
        if ack:
            sender.receive_ack(ack)

print(f"\n✓ All {sender.total} packets delivered "
      f"in {round_num} rounds.")

What the code shows: The base, next_seq, and right_edge pointers directly correspond to the window boundaries in the diagram. When receive_ack() slides base forward, the right_edge automatically moves too, opening slots for new packets. This is the "sliding" in sliding window.

Deep Dive: Congestion Control (AIMD)

While the sliding window prevents overwhelming the receiver, congestion control prevents overwhelming the network. TCP uses AIMD (Additive Increase, Multiplicative Decrease) — the classic sawtooth pattern you see in every networking textbook.

The Three Phases

Slow Start: cwnd doubles every RTT (exponential growth) until hitting ssthresh. Congestion Avoidance: cwnd grows by 1 MSS per RTT (linear/additive increase). Loss Event: cwnd is halved (multiplicative decrease), creating the characteristic sawtooth. The top chart shows cwnd over time; the bottom shows packets in flight with ACK flow.

Python Implementation

This simulation models TCP Reno's congestion control. The cwnd sawtooth pattern in the diagram above is produced by exactly this logic.

"""
TCP Congestion Control — Reno Algorithm

Three phases:
  1. Slow Start:          cwnd doubles each RTT (exponential)
  2. Congestion Avoidance: cwnd += 1 MSS per RTT (linear)
  3. Fast Recovery:        cwnd halved on 3 dup ACKs

The effective sending window = min(cwnd, rwnd).
cwnd is measured in MSS (Maximum Segment Size = 1460 bytes).

The "sawtooth" pattern:
  - cwnd grows linearly in congestion avoidance
  - cwnd is halved on loss → sharp drop
  - then grows linearly again → gradual rise
  - repeat → sawtooth shape
"""

class TCPRenoCongestionControl:
    def __init__(self, initial_cwnd=1, initial_ssthresh=64):
        # --- Core state ---
        self.cwnd = initial_cwnd      # congestion window (in MSS)
        self.ssthresh = initial_ssthresh  # slow start threshold
        self.phase = "slow_start"

        # --- Tracking ---
        self.ack_count = 0            # ACKs received in current RTT
        self.dup_ack_count = 0        # consecutive duplicate ACKs
        self.last_ack = -1

        # --- History for plotting ---
        self.history = []
        self.rtt_num = 0

    def on_ack(self, ack_num):
        """Called when a new ACK arrives.

        Slow Start: each ACK increases cwnd by 1 MSS.
          Since ~cwnd ACKs arrive per RTT, cwnd doubles per RTT.
          This is EXPONENTIAL growth (1→2→4→8→16...).

        Congestion Avoidance: each ACK increases cwnd by 1/cwnd.
          Since ~cwnd ACKs per RTT, total increase = cwnd * (1/cwnd) = 1.
          cwnd grows by 1 MSS per RTT. This is LINEAR growth.
        """
        if ack_num == self.last_ack:
            # Duplicate ACK
            self.dup_ack_count += 1
            if self.dup_ack_count == 3:
                # 3 dup ACKs → packet loss detected (fast retransmit)
                self._on_triple_dup_ack()
            return

        # New ACK
        self.dup_ack_count = 0
        self.last_ack = ack_num
        self.ack_count += 1

        if self.phase == "slow_start":
            # Exponential growth: +1 MSS per ACK
            self.cwnd += 1
            if self.cwnd >= self.ssthresh:
                self.phase = "congestion_avoidance"
                print(f"  → Entering Congestion Avoidance "
                      f"(cwnd={self.cwnd}, ssthresh={self.ssthresh})")

        elif self.phase == "congestion_avoidance":
            # Linear growth: +1/cwnd per ACK → +1 MSS per RTT
            self.cwnd += 1.0 / self.cwnd

        elif self.phase == "fast_recovery":
            # Exit fast recovery on new ACK
            self.cwnd = self.ssthresh
            self.phase = "congestion_avoidance"
            print(f"  → Exiting Fast Recovery "
                  f"(cwnd={self.cwnd:.1f})")

    def _on_triple_dup_ack(self):
        """Three duplicate ACKs: a single packet was lost.

        MULTIPLICATIVE DECREASE: halve the window.
        This is gentler than a timeout (which resets to 1).

        Why halve? The 3 dup ACKs prove the network is still
        delivering packets (just not one specific one). So the
        network isn't totally congested — halving is enough.
        """
        print(f"  ✗ 3 dup ACKs! Fast retransmit. "
              f"cwnd {self.cwnd:.1f} → {self.cwnd/2:.1f}")
        self.ssthresh = max(int(self.cwnd / 2), 2)
        self.cwnd = self.ssthresh + 3  # inflate for 3 dup ACKs
        self.phase = "fast_recovery"

    def on_timeout(self):
        """Retransmission timeout: the worst case.

        No ACKs at all → network might be totally congested.
        Reset cwnd to 1 and restart slow start.
        This is MORE aggressive than triple dup ACK because
        a timeout means even the dup ACKs aren't getting through.
        """
        print(f"  ✗✗ TIMEOUT! cwnd {self.cwnd:.1f} → 1, "
              f"ssthresh → {max(int(self.cwnd/2), 2)}")
        self.ssthresh = max(int(self.cwnd / 2), 2)
        self.cwnd = 1
        self.phase = "slow_start"

    def record(self):
        """Record current state for plotting."""
        self.rtt_num += 1
        self.history.append({
            'rtt': self.rtt_num,
            'cwnd': round(self.cwnd, 2),
            'ssthresh': self.ssthresh,
            'phase': self.phase
        })


# --- Simulation: 30 RTTs with loss events ---
print("=== TCP Reno Congestion Control Simulation ===\n")
cc = TCPRenoCongestionControl(initial_cwnd=1, initial_ssthresh=32)

for rtt in range(40):
    # Simulate one RTT: ~cwnd ACKs arrive
    num_acks = int(cc.cwnd)

    # Inject loss at specific points to show the sawtooth
    loss_at_rtt = {15, 28, 36}  # loss events

    if rtt in loss_at_rtt:
        # Simulate 3 duplicate ACKs (fast retransmit)
        print(f"\nRTT {rtt}: Loss detected!")
        cc.on_ack(rtt)  # dup
        cc.on_ack(rtt)  # dup
        cc.on_ack(rtt)  # 3rd dup → triggers fast retransmit
    else:
        for _ in range(num_acks):
            cc.on_ack(rtt + 1)  # new ACK

    cc.record()
    phase_symbol = {"slow_start": "⟋⟋",  # exponential
                    "congestion_avoidance": "⟋",  # linear
                    "fast_recovery": "↓"}  # drop
    print(f"RTT {rtt:2d}: cwnd={cc.cwnd:6.1f}  "
          f"ssthresh={cc.ssthresh:3d}  "
          f"{phase_symbol.get(cc.phase, '?')} {cc.phase}")

# Print the sawtooth pattern as ASCII art
print("\n=== cwnd Over Time (Sawtooth Pattern) ===\n")
max_cwnd = max(h['cwnd'] for h in cc.history)
chart_width = 50
for h in cc.history:
    bar_len = int(h['cwnd'] / max_cwnd * chart_width)
    phase_char = {'slow_start': 'S',
                  'congestion_avoidance': 'A',
                  'fast_recovery': 'R'}[h['phase']]
    bar = '█' * bar_len
    print(f"RTT {h['rtt']:2d} [{phase_char}] {bar} {h['cwnd']}")

What the code shows: In slow start, cwnd += 1 per ACK gives exponential growth (cwnd ACKs per RTT → cwnd doubles). In congestion avoidance, cwnd += 1/cwnd per ACK gives linear growth (+1 per RTT). On triple duplicate ACK, cwnd is halved (multiplicative decrease). The ASCII chart at the bottom produces the same sawtooth pattern as the animated diagram above.

AIMD: Why It Works

The Math Behind the Sawtooth

AIMD converges to fairness. If two flows share a bottleneck link of capacity C:

  • Additive Increase: Both flows increase by the same amount (+1 MSS/RTT). This moves their combined rate along a 45° line toward the capacity boundary.
  • Multiplicative Decrease: Both flows halve their rate. This moves them toward the fairness line (where both flows get equal share) because halving preserves the ratio while reducing the total.
  • Over repeated cycles of AI+MD, flows converge to C/2 each — regardless of where they started. This is why TCP achieves approximate fairness without any central coordinator.

Throughput formula: In steady state, TCP Reno achieves throughput ≈ (MSS / RTT) × (1.22 / √p) where p is the loss rate. This is the Mathis formula — it shows that throughput degrades as the square root of loss rate and is inversely proportional to RTT.

How It Works

TCP Three-Way Handshake

The client sends SYN, the server responds with SYN-ACK, the client completes with ACK. 1 RTT before data flows.

Before any data flows, TCP establishes a connection with a three-step ritual:

  1. SYN — The client sends a SYN (synchronize) segment with its initial sequence number (ISN). Think of it as "I want to talk, and I will start counting bytes at number X."
  2. SYN-ACK — The server responds with its own SYN and an ACK of the client's ISN+1. "I hear you, here is my starting number Y, and I am ready for your byte X+1."
  3. ACK — The client acknowledges the server's ISN. The connection is now established. This takes one full round-trip time (RTT) before any data can flow.

System design parallel: Every new TCP connection costs 1 RTT. In a microservices architecture with 10 hops per request, those handshakes add up. This is why connection pooling and HTTP/2 multiplexing are so critical — they amortize that setup cost across many requests.

Sliding Window Flow Control

TCP does not send data one byte at a time and wait for an ACK. Instead, it uses a sliding window: the sender can have multiple segments "in flight" (sent but not yet acknowledged) up to the window size. As ACKs come back, the window slides forward, allowing new data to be sent.

The receiver advertises a receive window (rwnd) — how much buffer space it has available. The sender never exceeds this. This prevents a fast sender from overwhelming a slow receiver. In system design terms, this is backpressure: the same concept you see in message queues (consumer lag) and API rate limiting (throttling the producer).

Congestion Control

Flow control protects the receiver; congestion control protects the network. TCP maintains a congestion window (cwnd) that limits how much data is in flight, independent of the receiver's window. The effective window is min(cwnd, rwnd).

  1. Slow start — The cwnd starts small (typically 10 segments, or ~14KB). For every ACK received, cwnd increases by 1 segment, causing exponential growth. The window doubles every RTT until it hits the slow start threshold (ssthresh) or a loss occurs.
  2. Congestion avoidance — Once cwnd reaches ssthresh, growth becomes linear: cwnd increases by 1 segment per RTT (additive increase). This probes for available bandwidth cautiously.
  3. Fast retransmit & fast recovery — If the sender receives 3 duplicate ACKs (indicating a single packet was lost but later packets arrived), it retransmits the lost segment immediately without waiting for a timeout. cwnd is halved (multiplicative decrease), and the sender enters congestion avoidance. This AIMD (Additive Increase, Multiplicative Decrease) pattern is the heartbeat of classic TCP.

System design parallel: TCP slow start means a fresh connection cannot immediately use full bandwidth. The first request on a new connection to a CDN edge is penalized. This is why persistent connections matter, why HTTP/2 multiplexes streams over a single connection, and why QUIC supports 0-RTT resumption.

UDP: When You Do Not Need Reliability

UDP adds only a port number and a checksum on top of raw IP. No handshake, no sequence numbers, no retransmissions. A UDP datagram either arrives or it does not. This makes UDP ideal for:

  • Real-time applications — Video streaming, VoIP, online gaming. A late packet is worse than a lost packet; retransmission is pointless if the moment has passed.
  • DNS queries — A single request-response pair. Opening a TCP connection for one question is wasteful (although DNS-over-TCP and DNS-over-HTTPS are increasingly common for security).
  • Application-level reliability — When you want control over retransmission logic (e.g., QUIC builds its own reliability on top of UDP).

UDP in Practice: Online Multiplayer Game Example

Consider an online multiplayer game sending player position updates 60 times per second via UDP.

Packet #42 (player at position x=100, y=200) is lost in transit. What happens?

  • The game does NOT retransmit packet #42 (UDP has no retransmission).
  • Packet #43 arrives: player at x=102, y=201.
  • The client renders a small "jump" — the player teleports 2 pixels instead of moving smoothly by 1 pixel.
  • This is acceptable! The game prefers a tiny visual glitch over a 100ms delay while waiting for retransmission.

If this were TCP: packet #42 lost → TCP retransmits → packets #43–#45 are queued waiting for #42 → 100ms+ freeze → then all packets delivered at once → player "warps" forward. Much worse user experience.

  t=50ms  Packet #44 arrives  → render player at (103, 202)   ✓

DNS Resolution: How a Domain Name Becomes an IP Address

When you type www.example.com in your browser, your computer does not know the IP address of that server. Before any TCP handshake or TLS negotiation can begin, the domain name must be resolved to an IP address. This is DNS (Domain Name System) — the internet's phone book.

Here is the full resolution chain for an uncached lookup of www.example.com:

Step Who Action Response Cached?
1 Browser Checks browser DNS cache Not found —
2 Operating System Checks OS DNS cache (/etc/hosts, system resolver) Not found —
3 OS → Recursive Resolver (ISP or 8.8.8.8) "What is the IP for www.example.com?" "I don't know yet, let me find out" —
4 Resolver → Root Nameserver (one of 13 clusters) "Who handles .com?" "Try the .com TLD server at 192.5.6.30" Cached for 48h
5 Resolver → TLD Nameserver (.com) "Who handles example.com?" "Try the authoritative server at 93.184.216.34" Cached for TTL
6 Resolver → Authoritative Nameserver (example.com) "What is the IP for www.example.com?" "93.184.216.34, TTL=3600" Cached for 1 hour
7 Resolver → OS → Browser "93.184.216.34" — All levels cache

Total latency for an uncached lookup: ~100–200ms (4 network round trips: to root, to TLD, to authoritative, and back to the client). For a cached lookup (the recursive resolver already knows the answer from step 3): ~1–5ms.

Caching at Every Level

Each DNS response includes a TTL (Time To Live) — the number of seconds the answer may be cached before it must be re-queried. Caching happens at multiple layers:

  • Browser cache — Chrome caches DNS results for up to 60 seconds (configurable). Fastest layer: no network call at all.
  • OS resolver cache — The operating system's stub resolver (e.g., systemd-resolved on Linux, the DNS Client service on Windows) caches responses according to their TTL.
  • Recursive resolver cache — Your ISP's resolver or a public resolver like Google (8.8.8.8) or Cloudflare (1.1.1.1) caches aggressively. This is the most impactful cache — it serves millions of users, so popular domains are almost always cached here.
  • Root hints and TLD delegation — The addresses of root servers and TLD servers change very rarely. Resolvers cache root hints for ~48 hours and TLD referrals for similarly long periods.

Common TTL values and their tradeoffs:

TTL Use Case Tradeoff
60 seconds Services behind a load balancer with frequent IP changes, failover scenarios High DNS query volume, but fast failover
300 seconds (5 min) Most web applications — a good default Balance between freshness and query volume
3600 seconds (1 hour) Stable services with infrequent IP changes Low DNS traffic, but slow failover (up to 1 hour of stale data)
86400 seconds (24 hours) MX records, TXT records for email authentication (SPF, DKIM) Minimal DNS traffic, but changes take a full day to propagate

System design implication: Before a planned migration (e.g., moving to a new CDN or cloud provider), lower the TTL to 60 seconds well in advance (at least 24–48 hours before). This ensures that when you change the DNS record, old cached values expire quickly. After the migration stabilizes, raise the TTL back up to reduce DNS query load.

Why DNS Uses UDP (Usually)

Standard DNS queries use UDP on port 53. A typical DNS query is ~40 bytes and the response is ~100–500 bytes — both fit in a single UDP datagram. Using TCP would require a 3-way handshake (1 RTT) before the query is even sent, tripling the latency for a single lookup. However, DNS falls back to TCP when:

  • The response exceeds 512 bytes (the original UDP limit) or 4096 bytes (with EDNS0 extensions)
  • Zone transfers (AXFR/IXFR) between DNS servers, which transfer large amounts of data
  • DNS-over-TLS (DoT, port 853) and DNS-over-HTTPS (DoH, port 443) for privacy — these always use TCP because TLS requires a connection

TLS Handshake Overhead

TLS 1.2 (left): 2 RTTs — ClientHello/ServerHello + certificate, then key exchange + Finished. TLS 1.3 (right): 1 RTT — key share sent in ClientHello, completed in one flight.

TLS 1.2 adds 2 additional RTTs on top of the TCP handshake before encrypted data can flow. TLS 1.3 reduced this to 1 RTT (and supports 0-RTT for resumed sessions, at the cost of replay attack risk). For a new HTTPS connection: TCP handshake (1 RTT) + TLS 1.3 (1 RTT) = 2 RTTs minimum before the first byte of application data. On a 100ms cross-continent link, that is 200ms of pure setup latency.

TLS 1.2 — Step by Step (2 RTTs)

The TLS 1.2 handshake is a sequential negotiation split across two full round trips:

  1. RTT 1 — Negotiate cipher and exchange certificates
    • ClientHello (Client → Server) — The client sends its supported cipher suites (e.g., TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256), a client random (32 bytes of nonce), and the maximum TLS version it supports. This is the client saying: "Here are all the encryption methods I know. Pick one."
    • ServerHello (Server → Client) — The server selects one cipher suite from the client's list and sends back its own server random nonce. "I picked this cipher. Here is my random number."
    • Certificate (Server → Client) — The server sends its X.509 certificate chain so the client can verify the server's identity. The client checks the certificate against trusted CAs.
    • ServerKeyExchange (Server → Client) — For DHE/ECDHE ciphers, the server sends its Diffie-Hellman public parameters. (Skipped for RSA key exchange, which is now deprecated.)
    • ServerHelloDone (Server → Client) — "I am done sending my handshake messages. Your turn."
  2. RTT 2 — Complete key exchange and verify
    • ClientKeyExchange (Client → Server) — The client sends its half of the key material: either its DH/ECDHE public key, or an RSA-encrypted pre-master secret. Both sides can now independently compute the same master secret from: client random + server random + pre-master secret.
    • ChangeCipherSpec (Client → Server) — "Everything I send from now on will be encrypted with the negotiated cipher." This is a signal, not a negotiation.
    • Finished (Client → Server) — A hash of the entire handshake transcript, encrypted with the new session key. This proves the handshake was not tampered with.
    • ChangeCipherSpec (Server → Client) — Server also switches to encrypted communication.
    • Finished (Server → Client) — Server's own encrypted verify hash. Both sides confirm they derived the same key.

Why 2 RTTs? The cipher negotiation and key exchange are separate phases. The client must wait to learn which cipher and key exchange method the server chose (RTT 1) before it can compute and send its own key material (RTT 2). TLS 1.2 supported many legacy ciphers (RSA, DHE, ECDHE), each requiring different client behavior, so the client could not act until the server decided.

TLS 1.3 — Step by Step (1 RTT)

TLS 1.3 collapses the handshake into a single round trip by making two key design changes:

  1. Removed legacy ciphers. TLS 1.3 only allows a handful of modern key exchange methods (all Diffie-Hellman based: X25519, P-256, P-384). Since there are so few options, the client can guess which one the server will pick and speculatively send its key share upfront.
  2. Merged negotiation and key exchange into a single flight. No separate ChangeCipherSpec message. No ServerKeyExchange. Everything happens in one round trip.

The flow:

  1. ClientHello + KeyShare (Client → Server) — The client sends supported ciphers and pre-computes key shares for the most likely DH groups (e.g., X25519 and P-256). "Here are my cipher preferences, and here is my key material for the ones I think you will pick."
  2. ServerHello + KeyShare + EncryptedExtensions + Certificate + Finished (Server → Client) — The server picks a cipher, sends its DH key share, its certificate, and its Finished hash — all in one flight. Everything after ServerHello is already encrypted using the derived handshake key.
  3. Finished (Client → Server) — The client verifies the server's certificate and Finished hash, then sends its own Finished. Application data can now flow.

Why only 1 RTT? By restricting to modern DH-based ciphers, the client does not need to wait for the server's choice — it sends key shares for all likely options in its very first message. If the server picks one the client already provided, no extra round trip is needed. (In the rare case of a mismatch, the server sends a HelloRetryRequest and it falls back to 2 RTTs.)

TLS 1.3 0-RTT Resumption

If the client has connected to this server before, it can send encrypted application data in the very first message using a pre-shared key (PSK) from the previous session. This is called 0-RTT: the first HTTP request arrives at the server before the handshake is even complete. The tradeoff is that 0-RTT data is vulnerable to replay attacks — an attacker who captures the ClientHello can re-send it. For this reason, 0-RTT should only be used for idempotent requests (GET, not POST).

TCP vs. TLS — When to Use Which

Use Plain TCP When

  • Internal service-to-service in a trusted network — Within a VPC with network-level isolation (security groups, private subnets). The network boundary provides the security layer.
  • Ultra-low-latency requirements — When the handshake cost matters and the channel is already secured at another layer (IPsec, WireGuard VPN).
  • Non-sensitive protocols — Health checks, Prometheus metrics scraping, cluster-internal heartbeats where data is not confidential.
  • Legacy systems — Systems that cannot support TLS, provided there are compensating controls (network segmentation, mTLS sidecar proxies).

Use TLS When

  • Any data crosses the public internet — HTTPS for web traffic, WSS for WebSockets, TLS-wrapped database connections from external clients. No exceptions.
  • Sensitive data in transit — Credentials, PII, financial data, health records — even within a private network if you follow defense-in-depth.
  • User-facing traffic — All HTTP traffic should be HTTPS. Browsers now flag plain HTTP as "Not Secure."
  • Compliance requirements — PCI-DSS, HIPAA, SOC2, and GDPR all mandate or strongly recommend encryption in transit.
  • Zero-trust architectures — In a zero-trust model, the network is never trusted. Every hop uses mTLS (mutual TLS) regardless of whether it is "internal."

Rule of thumb: Default to TLS. Only drop to plain TCP when you have a concrete reason and a compensating security control. The performance cost of TLS 1.3 (1 extra RTT on first connection, 0 on resumption) is negligible for almost all use cases.

System design parallel: This is why internal service-to-service communication within a datacenter often skips TLS (or uses mTLS with connection pooling to amortize the cost). It is also why QUIC is compelling: it combines the transport handshake and TLS handshake into a single RTT.

When to Use / When NOT to Use

TCP — Use When

  • Reliable delivery required — HTTP/HTTPS, REST APIs, gRPC, database connections, file transfers.
  • Ordered data matters — Streaming a web page, transferring a database dump, any protocol where out-of-order data is nonsensical.
  • Long-lived connections — WebSockets, database connection pools, gRPC streaming. TCP's overhead is amortized over many messages.
  • Congestion sensitivity — You want the network to self-regulate. TCP backs off under congestion; UDP does not.

TCP — Avoid When

  • Latency trumps reliability — Real-time voice/video where a lost frame is better than a delayed one.
  • One-shot requests — Tiny request/response pairs where the 1-RTT handshake is a proportionally large overhead (DNS, NTP).
  • Multicast/broadcast — TCP is point-to-point only. Use UDP for multicast scenarios (service discovery, cluster heartbeats).
  • Custom reliability logic — When your application needs different retransmission semantics than TCP provides (e.g., selective retransmission for specific game state).

UDP — Use When

  • Real-time streaming — Video conferencing (WebRTC), VoIP, live game state updates.
  • DNS queries — Standard DNS uses UDP on port 53 for speed. Falls back to TCP for large responses (>512 bytes).
  • Speed over reliability — Metrics collection (StatsD), logging pipelines where occasional loss is acceptable.
  • Building custom protocols — QUIC (HTTP/3) is built on UDP, implementing its own reliability, flow control, and encryption.

UDP — Avoid When

  • Data integrity is critical — Financial transactions, database writes, file transfers. Lost packets mean lost data.
  • Firewall compatibility matters — Many corporate firewalls block or throttle UDP traffic. TCP on port 443 is universally allowed.
  • You do not want to implement reliability yourself — Building reliable delivery, flow control, and congestion control is hard. Use TCP unless you have a compelling reason.

QUIC (HTTP/3): The Best of Both

To understand why QUIC exists, you need to understand the evolution from HTTP/1.1 to HTTP/2, and the fundamental problem HTTP/2 could not solve.

HTTP/2 Stream Multiplexing

HTTP/1.1 problem: One request at a time per TCP connection. To load a web page with 6 resources (CSS, JS, images), you either need 6 separate TCP connections (each with its own handshake overhead) or you wait for each request to finish before sending the next. This is head-of-line (HOL) blocking at the HTTP level — a slow response for /style.css blocks the browser from even asking for /app.js.

HTTP/2 solution: Multiple streams (logical request/response pairs) share a single TCP connection. Each stream has a numeric ID. Frames from different streams are interleaved on the wire:

Single TCP connection carrying 3 HTTP/2 streams:

  Frame: [Stream 1, HEADERS, GET /style.css   ]
  Frame: [Stream 2, HEADERS, GET /app.js       ]
  Frame: [Stream 3, HEADERS, GET /logo.png     ]
  Frame: [Stream 1, DATA, bytes 0-16383 of style.css    ]
  Frame: [Stream 2, DATA, bytes 0-16383 of app.js       ]
  Frame: [Stream 1, DATA, bytes 16384-32767 of style.css ]
  Frame: [Stream 3, DATA, bytes 0-8191 of logo.png      ]
  ...

All 3 resources download in parallel over ONE connection.
No head-of-line blocking at the HTTP level.

BUT: HTTP/2 still has head-of-line blocking at the TCP level. TCP sees all these interleaved frames as a single, ordered byte stream. If TCP packet #5 (carrying a frame for Stream 1) is lost, TCP must retransmit it before delivering packets #6, #7, #8 — even if those packets belong to Streams 2 and 3. Stream 3's data is delayed by Stream 1's lost packet. On lossy networks (mobile, Wi-Fi), this can make HTTP/2 slower than HTTP/1.1 (which used separate TCP connections that failed independently).

HTTP/3: QUIC Solves TCP's Head-of-Line Blocking

QUIC runs on UDP instead of TCP and implements its own reliability layer. The key innovation: each QUIC stream is independently flow-controlled and independently reliable. If a packet belonging to Stream 1 is lost, only Stream 1 stalls:

QUIC connection (over UDP):
  Stream 1: packet lost! Retransmitting...     ⏸ (only this stream waits)
  Stream 2: ✓ continues receiving data, unaffected
  Stream 3: ✓ continues receiving data, unaffected

Compare to HTTP/2 over TCP:

TCP connection:
  Packet lost! ALL streams blocked until retransmit completes.
  Stream 1: ⏸ waiting
  Stream 2: ⏸ waiting (even though its data arrived fine!)
  Stream 3: ⏸ waiting

QUIC's Other Advantages

QUIC is not just about fixing head-of-line blocking. It also includes:

  • Built-in TLS 1.3 — Encryption is not optional and is integrated into the connection handshake. A new QUIC connection takes 1 RTT (transport + crypto combined) vs. TCP's 1 RTT + TLS 1.3's 1 RTT = 2 RTTs. For resumed connections, QUIC supports 0-RTT: application data in the very first packet.
  • Connection migration — TCP identifies connections by the tuple (source IP, source port, dest IP, dest port). If your phone switches from Wi-Fi to cellular, the source IP changes and the TCP connection dies. QUIC identifies connections by a connection ID, so the connection survives network changes seamlessly.
  • User-space implementation — QUIC runs in the application (not the kernel), so protocol improvements ship as library updates rather than OS kernel upgrades. This enables much faster iteration.
Connection setup comparison:

  TCP + TLS 1.3 (HTTP/2):
    Client ──SYN──────────────→ Server       ┐
    Client ←─SYN-ACK──────────  Server       │ 1 RTT (TCP handshake)
    Client ──ACK + ClientHello→ Server       ┘
    Client ←─ServerHello───────  Server       ┐ 1 RTT (TLS handshake)
    Client ──Finished──────────→ Server       ┘
    Client ──HTTP Request──────→ Server       ← data flows (after 2 RTTs)

  QUIC (HTTP/3):
    Client ──Initial (crypto + data)──→ Server  ┐
    Client ←─Handshake─────────────────  Server  │ 1 RTT (combined!)
    Client ──HTTP Request──────────────→ Server  ┘ ← data flows (after 1 RTT)

  QUIC 0-RTT (resumed connection):
    Client ──Initial + 0-RTT data──────→ Server  ← data flows immediately!
    Client ←─Handshake─────────────────  Server  (0 RTTs before first data)

System design implication: On a 100ms cross-continent link, the difference between 2 RTTs (HTTP/2) and 1 RTT (QUIC) is 100ms of saved latency on every new connection. For 0-RTT, you save 200ms. For mobile users who frequently switch networks, QUIC's connection migration eliminates the reconnection penalty entirely.

Classic vs. Modern Approaches

Classic: TCP Reno & Tahoe Congestion Control

TCP Tahoe (1988) and Reno (1990) use loss-based congestion control: they interpret packet loss as a signal of congestion and cut the window. This works well on wired networks where loss almost always means congestion. But on wireless networks or high-bandwidth-delay-product links, it performs poorly — random packet loss triggers unnecessary backoff, and the AIMD probing is too slow to fill fat pipes.

Modern: BBR (Google, 2016)

BBR (Bottleneck Bandwidth and Round-trip propagation time) takes a fundamentally different approach. Instead of reacting to loss, it models the network path by continuously measuring two signals: the maximum bandwidth (delivery rate) and the minimum RTT. It aims to operate at the optimal point where throughput is maximized and queuing delay is minimized — the Kleinrock operating point.

Google reported a 14% throughput improvement on YouTube after deploying BBR, and up to 2700x improvement in throughput on lossy links (where loss-based algorithms collapse). BBR is now the default congestion control algorithm for Google's internal traffic and is available in Linux 4.9+.

System design implication: If you are serving video or large file downloads, switching to BBR at the OS level (a single sysctl command) can yield significant throughput gains with no application changes.

Modern: QUIC & HTTP/3

QUIC moves the transport layer from the kernel into user space, built on top of UDP. This means protocol improvements can be deployed as library updates rather than OS kernel upgrades — a huge advantage for iteration speed. Major CDNs (Cloudflare, Akamai, Fastly) and browsers (Chrome, Firefox, Safari) now support HTTP/3 over QUIC.

Modern: MPTCP (Multi-Path TCP)

MPTCP allows a single TCP connection to use multiple network paths simultaneously (e.g., Wi-Fi + cellular). This improves reliability (failover if one path drops) and throughput (aggregate bandwidth). Apple uses MPTCP for Siri and Maps on iOS. In datacenter networks, MPTCP can exploit multiple paths through a fat-tree topology.

Modern: Kernel Bypass & io_uring

For ultra-low-latency systems (high-frequency trading, real-time ad bidding), the kernel's network stack is too slow. DPDK (Data Plane Development Kit) bypasses the kernel entirely, allowing user-space applications to process packets directly from the NIC. io_uring (Linux 5.1+) provides asynchronous I/O with minimal syscall overhead, achieving near-kernel-bypass performance while remaining easier to use. TCP Fast Open (TFO) allows data in the SYN packet itself, saving 1 RTT on connections to servers you have visited before.

Assumptions & Model

Key Assumptions in TCP's Congestion Control Model

  • Packet loss = congestion — TCP Reno/Tahoe assume that any packet loss is caused by router buffer overflow (congestion). This breaks on wireless networks where packet loss is often due to signal interference, not congestion. BBR addresses this by not relying on loss as a signal.
  • RTT is measurable and stable — TCP timestamps packets to measure RTT and uses smoothed RTT (SRTT) for retransmission timers. On jittery mobile networks or through overloaded middleboxes, RTT measurement becomes noisy, leading to premature retransmissions or excessive timeouts.
  • Endpoints are cooperative — TCP assumes both sides obey congestion control. A malicious sender could ignore cwnd and flood the network. In practice, cloud providers enforce fairness at the hypervisor level.
  • Bandwidth-delay product (BDP) determines optimal window size — The ideal cwnd equals the BDP: BDP = bandwidth × RTT. On a 1 Gbps link with 10ms RTT, BDP = 1.25 MB. If your TCP buffers are smaller than this, you can never fully utilize the link. If they are much larger, you risk bufferbloat.
  • Congestion window growth is appropriate — Slow start's exponential growth assumes the network can absorb doubling traffic every RTT. On very high-bandwidth links, this wastes time; on already-congested links, it causes overshoot. Modern algorithms like CUBIC (Linux default) use a cubic function for more aggressive but smoother probing.

Sizing & Tradeoffs

Good networking decisions start from SLOs, not from defaults. Each example below begins with a concrete SLO and derives the networking configuration required to meet it.

SLO Example 1: Video Conferencing

SLO: p99 RTT < 150ms, 1,000 Concurrent Calls

Scenario: A video conferencing platform must deliver two-way audio/video with p99 round-trip latency under 150ms for 1,000 simultaneous calls.

Bandwidth derivation: Each call uses two media streams (send + receive). With VP8 at 720p the typical bitrate is ~1.5 Mbps per direction, so each call consumes ~3 Mbps. 1,000 calls × 3 Mbps = 3 Gbps aggregate media bandwidth. Provision at least 5 Gbps to absorb bursts and retransmissions.

Codec selection: At 150ms RTT budget, ~50ms is consumed by encode/decode. This leaves ~100ms for network transit. Opus audio (20ms frames) fits comfortably. For video, H.264 Baseline Profile with 100ms keyframe intervals keeps decode latency low. AV1 has better compression but its encode latency (~200ms at real-time preset) would blow the budget — reject it.

Protocol choice: UDP with SRTP, not TCP. A single lost TCP segment stalls the entire stream (head-of-line blocking), which destroys real-time guarantees. With UDP, lost frames are concealed by the codec (Opus PLC, video freeze-frame) rather than blocking subsequent frames.

Server placement: With a 100ms network budget, servers must be within ~50ms one-way of clients. Use latency-based DNS routing to direct users to the nearest media relay. For global coverage, deploy in at least 5 regions (US-East, US-West, EU-West, APAC-South, APAC-East).

SLO Example 2: Real-Time Gaming

SLO: p99 RTT < 50ms, 100K Concurrent Players

Scenario: A real-time multiplayer game requires p99 RTT under 50ms for 100,000 concurrent players across North America.

Server placement: 50ms RTT means ~25ms one-way. Light in fiber travels at ~200 km/ms, giving a radius of ~5,000 km at the physical limit, but real-world routing adds 1.5-3x overhead. Practically, you need servers within ~2,000 km of every player. For NA coverage: deploy in US-East (Virginia), US-Central (Dallas), US-West (Oregon), and Canada-Central (Montreal) — 4 regions.

TCP vs UDP: At 50ms budget, TCP retransmission timeout (~200ms minimum) is 4x the entire budget. A single lost packet causes a catastrophic latency spike. Use UDP with application-level reliability: send game state at 60 Hz (every 16.7ms), make each packet self-contained (no dependency on previous packets), and use client-side prediction to mask jitter.

Packet size optimization: Each game state update is ~200 bytes. Keep packets under the typical Internet MTU of 1,400 bytes to avoid fragmentation. Fragmented UDP packets are reassembled at the destination — if any fragment is lost, the entire datagram is dropped. Bundle up to 6 player updates per packet (6 × 200 = 1,200 bytes payload + 28 bytes UDP/IP header = 1,228 bytes, safely under MTU).

Capacity math: 100K players × 60 packets/sec × 1,228 bytes = ~7.4 Gbps inbound. Each game server handles ~100 players, so you need 1,000 game server instances plus headroom.

SLO Example 3: File Sync Service

SLO: 100 MB File Sync in < 10s on a 100 Mbps Link

Scenario: A file synchronization service (like Dropbox) must sync a 100 MB file in under 10 seconds when the user has a 100 Mbps connection.

Theoretical check: 100 MB = 800 Mb. At 100 Mbps, the minimum transfer time is 8 seconds. This leaves only 2 seconds of budget for overhead — extremely tight.

TCP window sizing: Assume 40ms RTT to the nearest storage edge. BDP = 100 Mbps × 0.04s = 4 Mb = 500 KB. TCP receive window must be at least 500 KB to saturate the link. Default Linux tcp_rmem max of 6 MB is sufficient, but ensure the initial window is large enough — set initcwnd to 32 segments (48 KB) to accelerate slow start.

Chunking strategy: Do not send the file as a single stream. Split into 4 MB chunks and upload via 4 parallel TCP connections. Each connection only needs 125 KB window to reach its 25 Mbps share. Parallel connections also provide resilience — if one connection hits packet loss and enters congestion avoidance, the others continue at full speed. Total throughput degrades gracefully.

Slow-start penalty: TCP slow start from initcwnd=10 takes ~5 RTTs (200ms) to reach the BDP of 500 KB. With 4 connections, this is 200ms × 4 = 200ms total (connections start in parallel). Acceptable within the 2s overhead budget. Use persistent connections (HTTP/2 or long-lived TCP) for subsequent syncs to skip slow start entirely.

Deduplication layer: Before any transfer, hash each 4 MB chunk (SHA-256). Send only chunks the server does not already have. For a file with 90% unchanged content, actual transfer drops from 100 MB to 10 MB, completing in ~1 second.

SLO Example 4: Microservices East-West Traffic

SLO: p99 < 5ms Intra-DC Latency, 1M RPS

Scenario: A microservices mesh within a single datacenter must handle 1 million requests per second with p99 latency under 5ms between any two services.

MTU optimization: Default Ethernet MTU is 1,500 bytes. Intra-DC, enable jumbo frames (MTU 9,000 bytes). A typical gRPC response of 4 KB requires 3 packets at MTU 1,500 but only 1 packet at MTU 9,000. Fewer packets mean fewer interrupts, less CPU overhead, and lower latency. Verify all switches in the path support jumbo frames — a single 1,500-byte link causes fragmentation and performance collapse.

Connection pooling: At 1M RPS across 200 services, each service averages 5,000 RPS. Using Little's Law with 2ms average latency: pool_size = 5,000 × 0.002 = 10 connections per upstream service. Add 2x headroom for p99 variance: 20 connections per upstream. With 200 services, each instance holds ~4,000 connections total. Use HTTP/2 to multiplex requests over fewer TCP connections — a single HTTP/2 connection can handle 100+ concurrent streams.

Kernel tuning for sub-5ms:

  • net.ipv4.tcp_nodelay = 1 — Disable Nagle's algorithm. Nagle buffers small writes for up to 200ms to coalesce them, which is fatal for a 5ms budget.
  • net.ipv4.tcp_low_latency = 1 — Prefer latency over throughput in TCP processing.
  • net.core.busy_poll = 50 — Busy-poll the NIC for 50μs before sleeping. Eliminates interrupt-to-userspace wakeup latency (~10-50μs) at the cost of CPU.
  • SO_REUSEPORT on all listeners — Distribute incoming connections across CPU cores, avoiding single-core bottleneck on the accept queue.

Capacity math: 1M RPS × 2 KB average request/response = 2 GB/s = 16 Gbps. A 25 GbE NIC handles this comfortably. At 10 GbE, you need two NICs with RSS (Receive Side Scaling) to distribute across cores.

SLO-Driven Capacity Planning

Capacity planning that starts from SLOs avoids both over-provisioning (wasted cost) and under-provisioning (breached SLOs). Follow this framework:

Step 1: Define the Latency Budget

Break the end-to-end SLO into per-hop budgets. For a p99 < 200ms API response:

  • Client → Edge: 30ms (governed by geography and DNS routing)
  • Edge → Origin DC: 40ms (governed by backbone RTT)
  • Load balancer + TLS: 2ms (L7 processing + TLS termination)
  • Service processing: 100ms (application logic + DB)
  • Response return path: 28ms (asymmetric routing possible)

Total: 200ms. Each component team owns their slice and monitors against it.

Step 2: Derive Bandwidth from Throughput SLO

Convert request rate SLOs to bandwidth requirements:

bandwidth = RPS × avg_response_size × overhead_factor

Where overhead_factor accounts for TCP/IP headers (~3% for large responses, ~40% for small responses), TLS record framing (~1%), and retransmissions (~1-2% on typical links). For 50K RPS with 5 KB average responses:

50,000 × 5 KB × 1.05 = 262.5 MB/s = 2.1 Gbps

Provision 2x headroom for burst absorption: 5 Gbps NIC capacity.

Step 3: Size Buffers and Pools from the Tail

Always size for the p99, not the average. Connection pools, TCP buffers, and queue depths should accommodate tail latency:

  • Connection pool: pool = RPS × p99_latency × safety_margin. If p99 is 4x the average, your pool must be 4x the Little's Law minimum.
  • TCP buffers: Size to BDP at the p99 RTT (which may be 2-3x the median RTT due to retransmissions and queueing).
  • Queue depth: If your service can absorb bursts for 2 seconds at p99 processing time, queue depth = burst_RPS × 2s × p99_processing_time.

Step 4: Validate with Load Testing at the Tail

Synthetic benchmarks at average load tell you nothing about SLO compliance. Test at p99 conditions:

  • Inject 1% packet loss (using tc netem) to simulate real-world network conditions.
  • Run at 80% of provisioned capacity for 1 hour, not 5 minutes — buffer bloat and GC pauses surface over time.
  • Measure p99.9, not just p99 — many SLO breaches hide in the last 0.1%.
  • Test with realistic payload size distributions, not uniform sizes.

Key Tradeoff: Connection Count

  • Too few connections — Serialization delay. Requests queue behind the pool, adding latency spikes at p99.
  • Too many connections — Memory waste (per-connection buffers), server CPU overhead (context switching between connections), and risk of exhausting server-side limits (file descriptors, max_connections).
  • Sweet spot — Use the Little's Law formula as a baseline, add 50-100% headroom, and monitor queue wait time. If requests spend more than 5ms waiting for a pooled connection, the pool is too small.

Scaling Journey

0 – 1K Users

Single Connection Per Request

Each incoming HTTP request opens a fresh TCP connection to the database, makes its query, and closes the connection. Simple, no pooling libraries needed. At low traffic, the 1ms handshake overhead is invisible.

Migration signal: p99 latency rises as handshake overhead becomes a measurable fraction of request time. Database logs show frequent connection churn.

1K – 100K Users

Connection Pooling

Introduce a connection pool (PgBouncer, HikariCP, or language-native pools). Connections are reused across requests, eliminating handshake overhead. HTTP/1.1 Keep-Alive keeps upstream connections open.

Migration signal: With HTTP/1.1, each in-flight request needs its own connection. At high concurrency, you open hundreds of TCP connections to the same upstream. Head-of-line blocking limits throughput.

100K – 10M Users

HTTP/2 Multiplexing

HTTP/2 multiplexes many logical streams over a single TCP connection. This dramatically reduces connection count to upstreams (load balancers, APIs, CDN origins). gRPC uses HTTP/2 natively, making it ideal for inter-service communication.

Migration signal: TCP head-of-line blocking becomes visible. A single lost packet stalls all multiplexed streams on that connection. Tail latency (p99.9) increases under packet loss.

10M+ Users

QUIC/HTTP/3 & Kernel Bypass

QUIC eliminates TCP head-of-line blocking (each stream is independently reliable). 0-RTT resumption slashes connection setup for returning users. For ultra-low-latency paths (ad bidding, HFT), kernel bypass (DPDK, io_uring) eliminates syscall overhead entirely.

Migration signal: When you need sub-millisecond latency guarantees or when mobile users on lossy networks suffer from TCP performance issues.

What Can Go Wrong

TCP Head-of-Line Blocking

What happened: A major web application migrated from HTTP/1.1 (multiple connections) to HTTP/2 (single multiplexed connection). Under normal conditions, latency improved. But on lossy mobile networks, p99 latency doubled. A single lost TCP segment blocked all HTTP/2 streams on that connection until the segment was retransmitted.

Root cause: TCP treats its byte stream as strictly ordered. HTTP/2 multiplexes independent request/response streams over that single ordered stream. One lost packet stalls everything — even streams whose data arrived fine.

Mitigation: Used multiple HTTP/2 connections (partially defeating the purpose), or migrated to QUIC/HTTP/3 where each stream has independent loss recovery. Some CDNs now automatically fall back from HTTP/2 to HTTP/1.1 on high-loss connections.

Connection Exhaustion (TIME_WAIT Accumulation)

What happened: A microservice under heavy load started returning connection errors intermittently. Monitoring showed 28,000 sockets in TIME_WAIT state, approaching the ephemeral port range limit (typically 28,232 ports on Linux, from 32768-60999).

Root cause: The service was opening and closing short-lived TCP connections to a downstream API at high frequency. TCP's TIME_WAIT state (2 × MSL = 60s by default) exists to prevent old packets from a closed connection from being misinterpreted by a new connection on the same port. But at high churn rates, you run out of (source IP, source port, dest IP, dest port) tuples.

Mitigation: Enabled tcp_tw_reuse (allows reusing TIME_WAIT sockets for new outbound connections), switched to connection pooling (HTTP Keep-Alive), and widened the ephemeral port range (net.ipv4.ip_local_port_range). Long-term fix: moved to HTTP/2 with persistent connections.

Bufferbloat Adding Hidden Latency

What happened: A video streaming service reported that interactive API calls (play/pause, seek) became sluggish during peak streaming hours, even though bandwidth utilization was only 40%. Ping times to the API went from 5ms to 300ms.

Root cause: Network switches and home routers had oversized buffers. During streaming, TCP filled these buffers (because TCP thinks buffered packets = delivered packets). The buffers added hundreds of milliseconds of queuing delay that TCP's congestion control could not detect because no packets were being dropped — they were just sitting in a very deep queue.

Mitigation: Deployed Active Queue Management (AQM) with CoDel or PIE algorithms on network equipment. Switched congestion control to BBR (which measures RTT rather than waiting for loss). Reduced TCP buffer sizes to prevent excessive queuing.

Debug Playbook

Is latency high?

→ Check RTT to the peer → If RTT is normal, the issue is server-side (application, DB). If RTT is elevated:

  → Check retransmission rate → If retransmissions are high, there is packet loss on the path. Check NIC errors, switch port errors, intermediate router health.

  → If retransmissions are low but RTT is high, suspect bufferbloat. Check queue depth on switches/routers. Consider enabling BBR or AQM.

Are connections failing?

→ Check TIME_WAIT count → If high, enable tcp_tw_reuse and implement connection pooling.

→ Check SYN backlog → If ListenOverflows counter is incrementing, increase tcp_max_syn_backlog and somaxconn.

→ Check file descriptor limits → Each socket consumes one fd. Increase ulimit -n and fs.file-max.

The kernel distributes incoming connections across all listeners, improving multi-core scalability for servers like Nginx. This is how high-performance proxies achieve millions of connections per second.

Interactions & Pitfalls

Networking + Load Balancing

L4 load balancers (TCP level) route based on IP/port tuples. They see TCP connections but not HTTP requests. A single HTTP/2 connection carrying 100 multiplexed streams looks like one connection to an L4 LB, causing uneven load distribution. L7 load balancers (HTTP level) can see individual requests and distribute them evenly, but they terminate TCP connections (adding latency) and must manage their own connection pools to backends.

Pitfall: If you use an L4 LB with HTTP/2 clients, you may see one backend drowning while others idle. Solution: use L7 load balancing for HTTP/2, or configure clients to open multiple HTTP/2 connections.

Networking + CDN

CDN edge servers are geographically close to users (low RTT), but the first request to an edge suffers TCP slow start — the connection starts with a small congestion window and must ramp up. For small assets (a 10KB JavaScript file), the entire transfer might complete before slow start finishes ramping up, so you never reach full throughput.

Pitfall: CDN providers tune initcwnd (initial congestion window) to 10 segments (~14KB) or higher. Google found that increasing initcwnd from 3 to 10 reduced page load time by 10%. If you control your origin server, tune this too.

Networking + Microservices

Each hop in a microservice call chain adds at least 1 RTT (more if connections are not pooled). A request that traverses 5 services adds 5 RTTs of network latency. Within a datacenter (0.5ms RTT), this is 2.5ms. Across availability zones (1-2ms RTT), it is 5-10ms. If any hop crosses regions (50-100ms RTT), latency explodes.

Pitfall: "The network is fast" is a common assumption that breaks when you have deep call chains. Always budget for network latency at each hop. Use persistent connections (gRPC with HTTP/2) and co-locate tightly-coupled services in the same availability zone.

Networking + Database

Database connections are expensive to establish (TCP + TLS + authentication + session setup). A PostgreSQL connection setup can take 5-20ms. Without pooling, a service making 1000 QPS would need 1000 connection setups per second — 5-20 seconds of cumulative overhead per second, which is obviously impossible.

Pitfall: Connection poolers like PgBouncer in transaction mode can multiplex many application connections over fewer database connections. But in transaction mode, prepared statements and session-level settings (SET search_path) do not work across requests. Test your ORM compatibility.

Cost Considerations

Network Cost Breakdown

  • Egress is the dominant cost. Cloud providers charge $0.05-$0.09/GB for data leaving their network (egress). Ingress is free. For a service sending 100 TB/month, that is $5,000-$9,000/month in egress alone.
  • Cross-AZ transfer: AWS charges $0.01/GB per direction for cross-AZ traffic. A microservice architecture with chatty cross-AZ calls can accumulate thousands of dollars monthly in transfer costs that are invisible until you check the bill.
  • NAT Gateway costs: AWS NAT Gateways charge $0.045/GB processed, plus $0.045/hour. A service sending 10 TB/month through NAT costs $495/month just for the NAT Gateway. Use VPC endpoints for AWS services and consider NAT instances for cost-sensitive workloads.
  • Compression saves 3-5x. Enabling gzip/Brotli on HTTP responses and using protocol buffers instead of JSON can reduce payload sizes by 60-80%, directly cutting egress costs. A 5 KB JSON payload compressed to 1.2 KB saves $0.00000038 per response — at 1 billion requests/month, that is $380/month saved.
  • The cost of NOT optimizing networking: Without connection pooling, you pay in CPU overhead (connection setup/teardown). Without compression, you pay in bandwidth costs. Without proper TCP tuning, you pay in underutilized expensive links. Without a CDN, you pay in origin bandwidth and higher latency losing users.

Code Example: TCP Sliding Window Simulation

A simplified Python simulation of TCP's sliding window protocol with sender and receiver. This illustrates window management, ACK-based advancement, and congestion response — the same concepts that govern connection throughput in any system you design.

import random

class SlidingWindowSender:
    def __init__(self, total_packets=20, init_cwnd=4):
        self.total = total_packets
        self.cwnd = init_cwnd          # congestion window
        self.base = 0                  # oldest unacked packet
        self.next_seq = 0              # next packet to send
        self.acked = [False] * total_packets
        self.ssthresh = 16             # slow start threshold

    def send_window(self):
        """Send all packets within the current window."""
        sent = []
        while self.next_seq < self.total and \
              self.next_seq < self.base + self.cwnd:
            sent.append(self.next_seq)
            self.next_seq += 1
        return sent

    def receive_ack(self, ack_num):
        """Process an ACK — slide window forward."""
        self.acked[ack_num] = True
        while self.base < self.total and self.acked[self.base]:
            self.base += 1
        # Grow window: slow start or congestion avoidance
        if self.cwnd < self.ssthresh:
            self.cwnd += 1              # slow start: exponential
        else:
            self.cwnd += 1 / self.cwnd  # congestion avoidance: linear

    def handle_loss(self):
        """Packet loss detected — shrink window (multiplicative decrease)."""
        self.ssthresh = max(self.cwnd // 2, 2)
        self.cwnd = self.ssthresh
        self.next_seq = self.base      # retransmit from base

# --- Simulation ---
sender = SlidingWindowSender(total_packets=20, init_cwnd=2)
loss_prob = 0.1  # 10% packet loss — like a congested network

while sender.base < sender.total:
    sent = sender.send_window()
    print(f"Sent: {sent}  cwnd={sender.cwnd:.1f}  base={sender.base}")
    for seq in sent:
        if random.random() > loss_prob:
            sender.receive_ack(seq)
        else:
            print(f"  !! Packet {seq} lost")
            sender.handle_loss()
            break
print(f"All {sender.total} packets delivered.")

How to Validate Your Design

Bandwidth & Throughput Testing

Use iperf3 to measure raw network throughput between two points. Run iperf3 -s on the server and iperf3 -c <server-ip> -t 30 on the client for a 30-second test. Compare results to your expected BDP. If throughput is significantly below link capacity, TCP tuning (buffer sizes, congestion control algorithm) is likely the bottleneck.

For HTTP-level benchmarking, use wrk or k6 to measure requests/second and latency distribution under load. Pay attention to p99, not just p50 — networking tail latency is where the problems hide.

Packet Analysis

tcpdump and Wireshark let you inspect individual packets on the wire. Key things to look for:

  • TCP retransmissions (filter: tcp.analysis.retransmission in Wireshark) — any retransmission > 0.1% deserves investigation.
  • Window size negotiation — is the receive window shrinking to zero (receiver overwhelmed)?
  • TLS handshake duration — is it 1 RTT (TLS 1.3) or 2 (TLS 1.2)?
  • RST (reset) packets — unexpected connection terminations by a firewall or overloaded server.

Fault Injection

toxiproxy is a TCP proxy that lets you inject network faults between your service and its dependencies:

  • Latency — Add 200ms delay to simulate cross-region calls. Verify your timeouts are set correctly.
  • Packet loss — Drop 5% of packets. Does your service degrade gracefully or crash?
  • Bandwidth limiting — Throttle to 1 Mbps. Does your service handle slow clients without holding connections forever?
  • Connection reset — Randomly RST connections. Does your connection pool recover and retry?

Run fault injection tests in staging as part of your CI/CD pipeline. Netflix's Chaos Monkey philosophy applies to networking: if you have not tested it failing, assume it will fail in the worst way at the worst time.

Interview Questions

Why does TCP have a three-way handshake?

The three-way handshake (SYN, SYN-ACK, ACK) serves a fundamental purpose: it allows both sides to agree on initial sequence numbers and confirm that the communication channel works in both directions. Without the third step (the client's ACK), the server would have no proof that the client received its sequence number — it would be committing resources (allocating a socket, buffer memory) based on a one-way message.

The three-way handshake also prevents stale connection requests from causing phantom connections. Imagine a SYN packet delayed by the network for 30 seconds. Without the final ACK, the server might accept this stale SYN and allocate resources for a connection the client no longer wants. The three-way handshake ensures both sides are alive and intentional right now.

System design connection: This 1-RTT cost is why connection pooling exists. In a microservices architecture, a request traversing 5 services would pay 5 RTTs just for TCP setup (plus 5 more for TLS). Connection pools amortize this cost by keeping connections open across requests. HTTP/2 multiplexing reduces it further by sending many requests over a single connection. QUIC eliminates it for returning clients with 0-RTT.

Follow-ups: What is a SYN flood attack and how does SYN cookies defend against it? (SYN floods exhaust the server's SYN backlog with half-open connections. SYN cookies encode connection state in the sequence number, so the server does not allocate memory until the handshake completes.) How does TCP Fast Open avoid the handshake overhead? (TFO sends data in the SYN packet using a cached cookie, saving 1 RTT on repeat connections.)

How does sliding window flow control work, and how does it relate to rate limiting in system design?

TCP's sliding window is a form of credit-based flow control. The receiver advertises how many bytes it can buffer (the receive window, rwnd). The sender can transmit up to that many bytes without waiting for an ACK. As the receiver processes data and frees buffer space, it advertises a larger window. If the receiver is overwhelmed, it advertises a smaller window — potentially zero, which pauses the sender entirely.

This is exactly the backpressure pattern in system design. A message queue with a bounded buffer applies the same principle: producers slow down when consumers cannot keep up. Rate limiting is the application-layer equivalent: the server advertises how many requests a client can make per second (the "window"), and the client must not exceed that rate. Token bucket algorithms are conceptually similar to TCP's window — tokens accumulate over time (like the window opening) and are consumed by requests (like sending data).

The congestion window (cwnd) adds a second dimension: even if the receiver has capacity, the sender also considers network capacity. This dual-window model is analogous to having both per-client rate limits (receiver flow control) and global rate limits (network congestion control) in an API gateway.

Follow-ups: What happens when rwnd reaches zero? (TCP enters "persist mode" and sends small probe segments to detect when the window reopens. Without this, a deadlock would occur — the receiver waits for data, the sender waits for a window update.) How is this different from TCP's Nagle algorithm? (Nagle's algorithm batches small writes into larger segments for efficiency, reducing overhead from tiny packets. It can interact badly with delayed ACKs, causing 200ms latency — disable it with TCP_NODELAY for latency-sensitive applications.)

TCP vs. UDP — give examples of when to use each in system design.

TCP is the default choice for system design. Use it for HTTP/HTTPS (web APIs, REST, GraphQL), gRPC (microservice communication), database connections (PostgreSQL, MySQL, MongoDB), message queue protocols (Kafka uses TCP, AMQP uses TCP), and any scenario where every byte matters (file transfers, financial transactions). TCP's reliability, ordering, and congestion control are exactly what you want when data integrity is critical.

UDP is the right choice when latency matters more than reliability, or when you want to implement custom reliability semantics. Real-world examples: DNS (standard queries on port 53 — a single round trip, no need for connection setup), video streaming (WebRTC uses UDP because a dropped video frame is better than a delayed one; the player just shows the next frame), gaming (game state updates sent 30-60 times per second; old state is irrelevant by the time it is retransmitted), metrics/logging (StatsD sends metrics over UDP because dropping a few data points is acceptable and the simplicity/performance gain is worth it).

The modern answer also includes QUIC: it is technically UDP at the network layer but provides TCP-like reliability at the application layer. Use QUIC/HTTP/3 when you want reliable delivery without TCP's head-of-line blocking problem — especially for mobile users on lossy networks or for multiplexed API calls.

Follow-ups: Why do some DNS implementations fall back to TCP? (When the DNS response exceeds 512 bytes — common with DNSSEC — UDP cannot carry it in a single datagram. DNS over HTTPS/TLS also uses TCP for privacy.) Can you build a reliable protocol on UDP? (Yes, QUIC does exactly this. You get to choose which reliability features you need and skip the ones you do not — at the cost of complexity.)

What is head-of-line blocking and how do HTTP/2 and QUIC address it?

Head-of-line (HOL) blocking occurs when a single slow or lost element blocks everything behind it. In TCP, it manifests because TCP guarantees in-order byte delivery. If segment #5 is lost, segments #6, #7, and #8 (which arrived fine) cannot be delivered to the application until #5 is retransmitted and received. The entire stream stalls.

HTTP/1.1 suffers from HOL blocking at the application layer: each connection handles one request at a time, so a slow response blocks subsequent requests on that connection. Browsers work around this by opening 6 parallel connections per origin — an inefficient hack.

HTTP/2 solves the application-layer HOL blocking by multiplexing many streams over a single TCP connection. Different requests and responses can be interleaved. But it inherits TCP-layer HOL blocking: if one TCP segment is lost, all streams stall because TCP's ordered byte stream sees them as one sequence. Under packet loss, HTTP/2 can actually perform worse than HTTP/1.1 (which had 6 independent TCP connections).

QUIC (HTTP/3) solves both layers. Each QUIC stream has independent reliability: a lost packet for stream A does not stall stream B. QUIC implements its own packet numbering and loss recovery per stream, built on top of UDP (which has no ordering guarantees). This is the fundamental reason HTTP/3 exists.

Follow-ups: Does QUIC completely eliminate HOL blocking? (It eliminates inter-stream HOL blocking but intra-stream HOL blocking remains — within a single request, packets must still be ordered. The improvement is that other requests are unaffected.) How does HOL blocking relate to message queues? (A single poison message blocking a partition in Kafka is conceptually similar — one stuck consumer blocks all messages behind it on that partition.)

How would you size a connection pool?

Start with Little's Law: L = λ × W, where L is the average number of items in the system (connections in use), λ is the arrival rate (requests per second), and W is the average time each request holds a connection. So: pool_size ≥ RPS × avg_latency.

Example: 200 RPS to a database, average query latency 25ms. Minimum pool: 200 × 0.025 = 5 connections. But this assumes perfectly uniform arrival and constant latency. In practice, traffic is bursty and latency varies. Add 2-3x headroom: 10-15 connections. PostgreSQL's own guidance suggests (2 × CPU_cores) + disk_spindles as a per-server maximum, which for a 4-core server is about 9 connections.

The over-provisioning trap: More connections is not always better. Each database connection consumes ~10MB of memory on the server. A pool of 100 connections to PostgreSQL wastes 1 GB of server RAM on connection overhead. Worse, the server spends CPU time context-switching between connections. HikariCP (the popular Java connection pool) recommends a pool size of 2 × CPU_cores + 1 for OLTP workloads, noting that larger pools consistently underperform in benchmarks.

Connection pool hierarchy: With microservices, you need pooling at every layer. If you have 20 application instances each with 15 connections, that is 300 database connections — more than most databases can efficiently handle. Use a connection multiplexer (PgBouncer, ProxySQL) between your application pools and the database to reduce the fan-in.

Follow-ups: What happens when the pool is exhausted? (Requests wait in a queue for a connection. This adds latency and can cascade: if wait times exceed downstream timeouts, you get timeout storms. Set a pool acquisition timeout and shed load early.) How do you size pools differently for read replicas vs. the primary? (Read replicas handle read-only traffic, which is typically the bulk. Allocate 80% of your total pool capacity to replica connections and 20% to the primary.)

Explain BBR and why it is better than traditional congestion control.

Traditional TCP congestion control (Reno, CUBIC) is loss-based: it increases the sending rate until a packet is dropped, then halves the rate. The implicit model is "loss = congestion." This worked well on wired networks in the 1990s, but it has two fundamental problems: (1) it requires filling router buffers to the point of overflow to detect congestion (causing bufferbloat), and (2) on lossy wireless links, random packet loss (from interference, not congestion) causes unnecessary rate reduction.

BBR (Bottleneck Bandwidth and Round-trip propagation time) takes a model-based approach. It continuously measures two signals: the maximum delivery rate (bottleneck bandwidth) and the minimum RTT (propagation delay). It then sets the sending rate to match the measured bottleneck bandwidth while keeping the amount of in-flight data equal to the BDP. This means BBR does not need packet loss to find the optimal rate — it discovers it through measurement.

Results: Google reported 14% throughput improvement on YouTube globally, with the largest gains on congested and lossy paths (developing countries, mobile networks). BBR avoids bufferbloat by design: it does not fill queues because it targets the propagation delay, not the delay including queuing. It also coexists better with other flows on the network (though BBRv1 had fairness issues with loss-based algorithms; BBRv2 addresses this).

System design implication: Switching to BBR is a single command (sysctl net.ipv4.tcp_congestion_control=bbr) and requires no application changes. For any service that transfers large amounts of data (video, file downloads, database replication over WAN), BBR is a near-free performance improvement.

Follow-ups: What are BBRv2's improvements over BBRv1? (BBRv2 adds explicit loss tolerance parameters and fairer bandwidth sharing with CUBIC flows. BBRv1 could starve loss-based flows by being too aggressive.) When should you NOT use BBR? (In controlled datacenter environments where CUBIC performs well and the link is lossless. BBR's probing behavior can cause minor throughput oscillations that CUBIC avoids on stable links.)

Key Takeaways

  • TCP's three-way handshake costs 1 RTT — and this is why connection pooling and HTTP/2 multiplexing exist. Every system design arrow between boxes is a network call; amortize setup costs.
  • The sliding window is backpressure at the transport layer — the same concept appears in message queues (consumer lag), rate limiters (token bucket), and stream processing (bounded buffers).
  • Congestion control determines throughput — BBR (bandwidth-based) outperforms Reno/CUBIC (loss-based) on lossy and high-BDP links. Switching is a single sysctl command.
  • UDP is for speed; TCP is for reliability; QUIC is for both — QUIC eliminates head-of-line blocking, combines transport + TLS handshakes into 1 RTT, and supports connection migration for mobile users.
  • Connection pool sizing follows Little's Law — pool_size = RPS × latency, plus headroom for variance. More connections is not always better; context switching and memory overhead dominate past the sweet spot.
  • Head-of-line blocking is the recurring villain — it appears in TCP (lost segment stalls everything), HTTP/1.1 (one request per connection), message queues (poison message), and databases (long transaction blocking WAL replay). Recognizing the pattern lets you apply the same solutions: parallelism, independent streams, and isolation.
  • Network costs are dominated by egress — cloud egress is $0.05-$0.09/GB. Compression, protocol optimization (protobuf over JSON), and CDNs are your primary cost levers.
  • Tune your kernel, not just your application — tcp_keepalive_time, tcp_tw_reuse, somaxconn, buffer sizes, and congestion control algorithm are the networking knobs that production systems need adjusted from their defaults.
  • Every microservice hop adds network latency — budget for it. Co-locate tightly coupled services, use persistent connections, and measure RTT between every pair of communicating services.
