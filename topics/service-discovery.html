Service Discovery

The mechanism that lets services find each other in a dynamic environment — the phone book for microservices, enabling automatic detection of service instances as they come and go.

20 min read

What Is It?

Service discovery is the process by which services in a distributed system locate each other. In a world where service instances are constantly being created and destroyed — due to auto-scaling, rolling deployments, failures, and container orchestration — hardcoding IP addresses and ports is impossible. Service discovery provides a dynamic registry that tracks which instances of each service are currently running and where they can be reached.

Think of it as the phone book for your microservices. When Service A needs to talk to Service B, it doesn't know Service B's IP address ahead of time. Instead, it asks the service registry: "Where is Service B?" The registry responds with the current list of healthy Service B instances, and Service A picks one to call. When a new instance of Service B spins up, it registers itself. When an instance shuts down or crashes, it gets removed.

The fundamental problem service discovery solves is this: hardcoded IPs don't work when services scale, deploy, or fail. In a traditional monolith deployed to a fixed set of servers, you could put IP addresses in a config file and call it a day. In a modern cloud-native environment with dozens or hundreds of services, each running multiple ephemeral instances across a fleet of machines, you need an automated mechanism to keep track of what's running where — and that mechanism is service discovery.

Why Hardcoded IPs Break Down

In a static environment, you might hardcode service locations in a config file:

# config.yaml — works fine when nothing moves
services:
  payment-service: 10.0.1.5:8080
  inventory-service: 10.0.1.10:8080
  notification-service: 10.0.1.15:8080

But in modern infrastructure, this approach falls apart for several reasons:

  • Containers start and stop dynamically: Kubernetes pods are ephemeral. A pod running at 10.0.1.5 right now might be replaced by a new pod at 10.0.2.37 in a minute.
  • Autoscaling adds and removes instances: Under load, your Payment Service might scale from 3 instances to 10 instances, then back to 5 instances as traffic subsides. Each new instance gets a new IP address.
  • Deployments replace instances: A rolling deployment terminates old instances and starts new ones. The old IP 10.0.1.5 dies, the new instance starts at 10.0.2.12.
  • Instances fail and are replaced: If an instance crashes due to an out-of-memory error, the orchestrator starts a replacement at a different IP.

If Payment Service moves from 10.0.1.5 to 10.0.2.12, every service that talks to it would need to be reconfigured and restarted. With 50 services that depend on Payment Service, that means 50 config changes and 50 restarts — for a single instance change. This does not scale.

Service discovery solves this by providing a layer of indirection: instead of "call 10.0.1.5," the instruction becomes "call payment-service" and a registry tells you which IPs currently map to that name.

Service Discovery in Action

Services register with a central registry. Clients query the registry to locate healthy service instances. Health checks continuously verify liveness. Services coming online register; failing services get deregistered.

How It Works

Discovery Patterns

There are two fundamental patterns for service discovery:

  1. Client-Side Discovery: The client (calling service) queries the service registry directly, receives a list of available instances, and selects one using a load-balancing algorithm (round-robin, random, least connections). The client is responsible for choosing which instance to call. Netflix Eureka and Consul with client libraries use this pattern. Advantage: no intermediate proxy, lower latency, client can use smart routing. Disadvantage: couples the client to the registry, requires a discovery library in every service (and every language).
  2. Server-Side Discovery: The client sends its request to a load balancer or API gateway, which queries the registry and routes the request to a healthy instance. The client doesn't know or care about the registry — it just calls a stable endpoint. AWS ALB with ECS service discovery and Kubernetes Services use this pattern. Advantage: client is decoupled from discovery, simpler client code. Disadvantage: the load balancer/proxy is an extra network hop and a potential bottleneck.

Client-Side Discovery: Deep Dive

In client-side discovery, the client is responsible for finding service instances. The client contains a discovery library that knows how to talk to the registry, cache results, and load-balance across instances. Here is the step-by-step flow:

  1. Service instances register themselves with the registry on startup. When Payment Service starts two instances, each one sends a registration request:
    • Payment-1 registers: {service: "payment", address: "10.0.1.5:8080", healthy: true}
    • Payment-2 registers: {service: "payment", address: "10.0.1.6:8080", healthy: true}
  2. Client (Order Service) queries the registry: "Give me all healthy instances of payment."
  3. Registry returns: ["10.0.1.5:8080", "10.0.1.6:8080"]
  4. Client load-balances between them using round-robin, random selection, least-connections, or any other algorithm implemented in the client library.
  5. Client sends the request directly to the chosen instance — for example, POST http://10.0.1.5:8080/charge.
# Client-side discovery: the client does the work
import random

class DiscoveryClient:
    def __init__(self, registry_url):
        self.registry_url = registry_url
        self.cache = {}        # {service_name: [instances]}
        self.cache_ttl = 10    # seconds
        self.cache_time = {}   # {service_name: last_fetch_time}

    def get_instance(self, service_name):
        """Get one healthy instance using round-robin."""
        instances = self._lookup(service_name)
        if not instances:
            raise Exception(f"No instances found for {service_name}")
        # Simple random load balancing
        return random.choice(instances)

    def _lookup(self, service_name):
        """Query registry (or use cache if fresh)."""
        now = time.time()
        if service_name in self.cache:
            if now - self.cache_time[service_name] < self.cache_ttl:
                return self.cache[service_name]  # cache hit
        # Cache miss — query the registry
        instances = http_get(f"{self.registry_url}/services/{service_name}")
        self.cache[service_name] = instances
        self.cache_time[service_name] = now
        return instances

# Usage: Order Service calls Payment Service
client = DiscoveryClient("http://consul:8500")
payment = client.get_instance("payment")
# payment = {"addr": "10.0.1.5", "port": 8080}
response = http_post(f"http://{payment['addr']}:{payment['port']}/charge", data)

Pros: No single point of failure for routing (the client does it). The client can use sophisticated load balancing strategies (latency-based, zone-aware). One fewer network hop (client talks directly to the service). Examples: Netflix Eureka + Ribbon, HashiCorp Consul + client library.

Cons: Every client language needs a discovery library. If you have services in Python, Java, Go, and Node.js, you need four discovery libraries. Service discovery logic is duplicated across all clients. The client becomes more complex.

Server-Side Discovery: Deep Dive

In server-side discovery, a load balancer or router sits between the client and the service instances. The client stays simple — it just needs to know the address of the load balancer. Here is the step-by-step flow:

  1. Service instances register with the registry (same as client-side — Payment-1 and Payment-2 both register).
  2. Client sends request to a well-known load balancer address: POST http://payment-lb:8080/charge. The client does not know about the registry at all.
  3. Load balancer queries the registry for healthy instances of "payment" and receives ["10.0.1.5:8080", "10.0.1.6:8080"].
  4. Load balancer forwards the request to a chosen instance (e.g., 10.0.1.5:8080) using its own load-balancing algorithm.
  5. Response flows back through the load balancer to the client: instance → LB → client.

Pros: Client stays simple — it just needs to know one address (the LB). Service discovery logic is centralized in the LB/router. Language-agnostic: any client that can make HTTP calls can use it. Examples: AWS ALB + ECS, Kubernetes Service + kube-proxy, Nginx + Consul.

Cons: Load balancer is an extra network hop (adds 1-2ms latency). The LB itself can become a bottleneck or single point of failure (must be replicated). Load balancing flexibility is limited to what the LB supports.

Network hops 1 (client → service) 2 (client → LB → service)

DNS-Based Discovery: Deep Dive

The simplest form of service discovery: service instances register as DNS records, and clients find them via standard DNS lookups. Every programming language and framework already knows how to do DNS — no special library required.

How it works step by step:

  1. Service instance starts and registers a DNS A record: payment.internal.example.com → 10.0.1.5
  2. When a second instance starts, the DNS record is updated to include both IPs: payment.internal.example.com → 10.0.1.5, 10.0.1.6 (multiple A records for the same name)
  3. Client does a DNS lookup for payment.internal.example.com and receives both IPs
  4. Client picks one IP (DNS round-robin rotates the order of returned records) and sends the request directly

The Staleness Problem

DNS responses are cached at every level: the operating system DNS cache, the language runtime (Java's JVM caches DNS forever by default), the application-level HTTP client, and any intermediate resolvers. This creates a fundamental tension:

If you set TTL=60s and an instance dies at time T=0, clients may still send traffic to the dead IP for up to 60 seconds — until their cached DNS entry expires and they re-resolve. During that window, every request to the dead IP fails with a connection refused error.

  • Low TTL (e.g., 5s): Fresh data — clients pick up changes quickly. But this generates a lot of DNS queries. If you have 1,000 clients each resolving every 5 seconds, that is 200 DNS queries/second for a single service name.
  • High TTL (e.g., 300s): Fewer DNS queries, lower DNS infrastructure load. But clients may send traffic to dead instances for up to 5 minutes after a change.

Kubernetes CoreDNS

Every Kubernetes Service gets a DNS name automatically (e.g., payment.default.svc.cluster.local). CoreDNS, running as pods within the cluster, resolves this name to the Service's ClusterIP (a virtual IP that kube-proxy load-balances to pod IPs) or directly to individual Pod IPs (for headless services where ClusterIP: None). TTL is typically 30 seconds. This is the most common DNS-based discovery in practice — if you are on Kubernetes, you get it for free.

Registration

How do service instances get into the registry?

  • Self-Registration: Each service instance registers itself with the registry on startup and deregisters on shutdown. The service is responsible for sending periodic heartbeats to keep its registration alive. If heartbeats stop, the registry removes the entry after a TTL expires. Simple, but the service must include registration logic, and if the service crashes without deregistering, there's a delay before the stale entry is removed.
  • Third-Party Registration: An external process (the "registrar") monitors the deployment environment, detects new instances (e.g., by watching Docker events or Kubernetes pod lifecycle), and registers them. The service itself has no knowledge of the registry. Kubernetes uses this approach — the kubelet and kube-proxy manage service endpoints automatically. Cleaner separation of concerns, but requires the registrar to be reliable.

Health Checking

A registered service isn't useful if it's down. Health checks verify that registered instances are actually able to serve traffic. The registry must know when a service instance is healthy. There are two fundamental approaches:

  • Active health checks: The registry (or a separate health checker) periodically pings each instance — HTTP GET to a /health endpoint, TCP connect, or gRPC health check. If the check fails N consecutive times, the instance is deregistered.
  • Passive health checks (heartbeats): The service sends periodic heartbeats to the registry. If the registry doesn't receive a heartbeat within the TTL, the instance is removed. Less network traffic from the registry's perspective, but the service must actively send heartbeats.

Graceful Deregistration

When an instance is shutting down — during a deployment, scale-down, or maintenance — it should actively deregister by sending a DELETE request to the registry before stopping. This removes it from the pool immediately, so no traffic is sent to an instance that is about to die. The sequence is:

  1. Instance receives SIGTERM (shutdown signal)
  2. Instance stops accepting new requests (closes the listening socket or returns 503)
  3. Instance finishes processing in-flight requests (connection draining, typically 5-30 seconds)
  4. Instance sends deregistration request to the registry: DELETE /services/payment/instances/pay-1
  5. Registry removes the instance from its list immediately
  6. Instance exits

Crash Without Deregistration

If an instance crashes (OOM kill, hardware failure, kernel panic), it cannot deregister gracefully — it is gone before it can send any final request. In this case, the registry relies on TTL-based cleanup: after N missed heartbeats or N failed health checks, the instance is automatically removed.

The detection time equals interval × threshold. For example, with a 10-second health check interval and a threshold of 3 consecutive failures, the instance is removed after 30 seconds. During those 30 seconds, the registry still considers the instance healthy and may route traffic to it. Clients hitting the dead instance will get connection refused errors and must retry on another instance.

This is the fundamental tradeoff: shorter intervals detect failures faster but generate more network traffic. Longer intervals reduce overhead but leave dead instances in the pool longer.

When to Use / When NOT to Use

Use Service Discovery When

  • Microservices with dynamic scaling: Auto-scaling groups, container orchestration, or serverless functions create and destroy instances constantly. You need a way to track what's running.
  • Container/Kubernetes environments: Pod IPs are ephemeral. Every time a pod restarts, it gets a new IP. You need DNS or a service abstraction to provide stable addressing.
  • Multi-environment deployments: Dev, staging, and production each have different service locations. Discovery lets the same code run in any environment without config changes.
  • Polyglot architectures: When services are written in different languages, a central registry provides a language-agnostic way to find services.
  • Blue-green or canary deployments: Routing traffic between old and new versions of a service requires dynamic instance tracking.

Don't Use Service Discovery When

  • Static infrastructure with fixed IPs: If you have 3 servers that never change, a config file with their IPs is simpler and more reliable than running a discovery infrastructure.
  • Monolith with single deployment: If everything runs in one process on one server, there's nothing to discover. Service discovery adds complexity with no benefit.
  • DNS-based discovery is sufficient: For simple cases, DNS A/SRV records with low TTL can provide basic discovery without a dedicated registry. If you only need "which IPs serve this service?" and updates are infrequent, DNS is enough.
  • Small team, few services: If you have 2-3 services and a small team, the operational overhead of running Consul or etcd isn't justified. Use environment variables or a config file.

Classic vs. Modern Approaches

Classic: Hardcoded Configuration

The simplest approach — store service addresses in config files, environment variables, or property files. Works for static environments with a handful of services that rarely change. Breaks immediately when instances scale dynamically, because someone has to manually update the config and redeploy every time an instance is added or removed.

Classic: DNS-Based Discovery

Use DNS A records or SRV records to map service names to IP addresses. Update DNS records when instances change. This is simple and universally supported — every language and framework knows how to do DNS lookups. However, DNS has TTL-based caching, so stale records persist until the TTL expires. Low TTLs increase DNS query load; high TTLs mean slow propagation of changes. DNS doesn't support health checking natively — you need external health checks to update DNS records.

Dedicated Service Registries: Consul, etcd, ZooKeeper

HashiCorp Consul is a full-featured service discovery and configuration tool. It provides service registration, health checking, DNS and HTTP APIs for lookups, key-value storage, and multi-datacenter federation. Consul uses the Raft consensus protocol for strong consistency. Its DNS interface means any application can use it without a client library. etcd is a distributed key-value store (used by Kubernetes) that can serve as a service registry. Strongly consistent (Raft), but lower-level — you build the registration and health check logic yourself. ZooKeeper was the original distributed coordination service (used by Kafka, Hadoop). It provides ephemeral nodes that automatically disappear when the client disconnects — a natural fit for service registration. However, ZooKeeper is complex to operate, and its ZAB consensus protocol has higher latency than Raft.

Kubernetes Native Discovery

Kubernetes has service discovery built in. When you create a Kubernetes Service object, the control plane automatically creates DNS entries (via CoreDNS) and maintains a list of endpoint IPs (healthy pod IPs). Any pod can reach a service by its DNS name (e.g., payment-service.default.svc.cluster.local). kube-proxy handles the load balancing. This is the most common modern approach because if you're already running Kubernetes, discovery is free — no additional infrastructure. Limitation: it only works within the cluster.

Service Mesh Control Plane (Istio, Linkerd)

Service meshes build on Kubernetes discovery and add a data plane (Envoy sidecar proxies) that handles traffic routing, load balancing, retries, and circuit breaking. The control plane (Istiod, Linkerd control plane) aggregates service discovery information and pushes configuration to sidecars. This provides advanced traffic management (canary routing, traffic splitting, mutual TLS) but adds significant operational complexity.

Netflix Eureka

A REST-based service registry designed for resilience. Eureka uses eventual consistency (AP in CAP terms) — it favors availability over consistency. If a Eureka server goes down, clients can still use cached registrations. Peers replicate registrations asynchronously. Eureka was designed for the cloud, where network partitions are expected. Widely used in Spring Cloud/Netflix OSS architectures.

Gossip-Based Discovery: Serf, Memberlist

Decentralized approaches where nodes discover each other by gossiping membership information. No central registry — each node maintains its own view of the cluster by periodically exchanging state with random peers. Serf (by HashiCorp) and Memberlist (the Go library underneath Serf and Consul) use the SWIM protocol for failure detection. Advantages: no SPOF, scales to thousands of nodes. Disadvantage: eventual consistency, convergence time depends on cluster size.

Assumptions & Model

Key Assumptions

  • The registry is highly available: The entire service discovery model assumes the registry is always reachable. If the registry goes down, no new lookups can be performed. Clients with cached results continue to work, but new services can't register and stale entries can't be removed. This is why registries like Consul and etcd run as multi-node clusters with consensus protocols.
  • Health checks are accurate: The model assumes that a passing health check means the service can actually serve traffic. This breaks when the health endpoint is simple (returns 200) but the service has a broken database connection, exhausted thread pool, or other dependency failures. Deep health checks (that verify dependencies) are more accurate but slower and can cause cascading failures if the dependency is slow.
  • Network allows service-to-registry communication: Every service instance must be able to reach the registry to register, send heartbeats, and perform lookups. In split-network scenarios or across firewalled environments, this connectivity may not exist.
  • Service names are stable and unique: The model assumes a consistent naming scheme. If different teams register services with inconsistent names, or the same name maps to different services in different environments, discovery breaks down.

When These Assumptions Break

  • Registry goes down (SPOF): If the registry cluster loses quorum, new registrations fail and health checks stop. Clients using cached data continue to work for a while, but eventually stale entries cause failures. Mitigation: run a 3- or 5-node cluster, spread across availability zones, and implement client-side caching with graceful degradation.
  • Stale health check data: A service passes its health check at time T, then crashes at T+1s. The next health check runs at T+5s. For 4 seconds, the registry considers the service healthy and routes traffic to a dead instance. Mitigation: shorter health check intervals (at the cost of more overhead) and client-side retry with failover to another instance.
  • Split-brain in registry cluster: A network partition splits the registry cluster into two groups, each believing the other is down. Both groups accept registrations, leading to divergent views of the service topology. When the partition heals, conflicting registrations must be reconciled. Mitigation: Raft-based registries (Consul, etcd) handle this by only allowing the majority partition to accept writes.

Sizing & Tradeoffs

Every sizing decision should start from an SLO, not a rule of thumb. Work backwards: define the latency, freshness, or accuracy target, then derive the infrastructure parameters that satisfy it.

Example 1 — Microservices Platform

SLO: p99 discovery latency < 10 ms • 5,000 service instances

  • Registry sizing: 5,000 instances × ~2 KB metadata each = ~10 MB working set. A 3-node Consul/etcd cluster with 4 GB RAM per node keeps the entire dataset in memory, yielding single-digit-ms reads. Use 5 nodes if you need multi-AZ fault tolerance without crossing the 10 ms budget during leader failover.
  • Cache TTL: To stay under 10 ms at p99, each service caches its dependency list locally with a 5-second TTL. Cache misses hit the registry over gRPC (p99 ~3 ms in-region), so even a miss stays within budget. A 30-second TTL would be cheaper but risks stale entries during rapid deploys.
  • Health check interval: 5,000 instances / 3-second interval = ~1,667 checks/sec. Each check is ~500 bytes → ~0.8 MB/s network — acceptable. A 3-second interval means a failed instance is detected in 6-9 seconds (2-3 missed checks), keeping stale-route exposure short enough that p99 user-facing latency is not impacted.

Example 2 — Auto-Scaling Web Tier

SLO: New instance discoverable < 5 s • 99.9 % routing accuracy

  • Registration protocol: Self-registration on boot with a synchronous HTTP PUT to the registry. The instance's readiness probe gates registration — the PUT fires only after the app passes its own health check, so the < 5 s budget covers boot time + one round-trip to the registry (~50 ms). If the registry is unreachable, retry with jitter every 500 ms for up to 4 s before failing the startup probe.
  • Propagation parameters: The registry pushes change events over long-poll / watch channels. Subscribers receive the new instance within 200 ms of registration. For DNS-based consumers, set TTL = 3 s so the worst-case staleness is 3 s — well within the 5 s window. Combined with the registration round-trip, end-to-end discovery is ~3.3 s at p99.
  • Routing accuracy: 99.9 % means at most 1 in 1,000 requests may hit a stale endpoint. With a 3-second health check interval and 2-miss deregistration threshold, a dead instance is removed in ~6 s. If the service handles 500 req/s per instance, that is ~3,000 requests to a dead instance — too many. Solution: pair registry-based discovery with client-side circuit breakers (trip after 3 consecutive failures in 1 s) to meet the 99.9 % target.

Example 3 — Multi-Region Service Mesh

SLO: Cross-region discovery < 100 ms • 10,000 endpoints across 3 regions

  • Federated registry topology: Run a 3-node registry cluster per region (~3,300 local endpoints each). Federate via asynchronous replication — each cluster gossips its catalog to peers every 2 s. Inter-region RTT is 60-80 ms (e.g., us-east to eu-west), so a single replication round-trip fits within the 100 ms budget. Full convergence after a change takes at most one gossip interval + one RTT = ~2.08 s worst case.
  • Local-first routing: Clients resolve services from their local registry replica (p99 < 5 ms). Cross-region lookups are only needed for services not deployed locally. The mesh control plane (e.g., Istio) maintains a merged endpoint table — 10,000 endpoints × 2 KB = ~20 MB, easily held in the sidecar proxy's memory.
  • Failure isolation: If one region's registry becomes unreachable, the other two continue operating with their last-known snapshot. Set a staleness threshold of 30 s: if a region's catalog has not been refreshed in 30 s, mark its endpoints as degraded and shift traffic to healthy regions.

Example 4 — Large Kubernetes Cluster

SLO: Pod discoverable < 2 s after readiness probe passes • 50,000 pods

  • etcd sizing: 50,000 pods generate ~50,000 Endpoints objects (or EndpointSlice objects, ~100 endpoints per slice = 500 slices). Each slice is ~5 KB → ~2.5 MB of endpoint data. etcd recommendation: 3 nodes with SSD-backed storage, 8 GB RAM, and a 10,000 rev/s write budget. At this scale, use EndpointSlices (not legacy Endpoints) to avoid O(n) updates on every pod change.
  • Watch channel parameters: kube-proxy and CoreDNS subscribe via the Kubernetes watch API. A pod becoming ready triggers an EndpointSlice update → the API server fans it out over watch channels within ~200 ms. CoreDNS picks up the change and serves the new record. Total path: readiness probe pass → kubelet updates Pod status (~100 ms) → endpoint controller updates EndpointSlice (~200 ms) → watch delivery (~200 ms) = ~500 ms typical, well within the 2 s budget.
  • Scaling limit: At 50,000 pods with average churn of 500 pods/min, the API server handles ~8 watch events/sec for endpoints alone. If you exceed 100,000 pods, shard with multiple clusters and use multi-cluster service discovery (e.g., Kubernetes MCS API) to stay within etcd's performance envelope.

SLO-Driven Capacity Planning

Use this framework to translate a discovery SLO into concrete infrastructure parameters:

SLO Dimension Governing Parameter How to Size
Discovery latency (p99) Registry node count, cache TTL Measure baseline p99 with load test. If > target, add local caching or switch from polling to watch-based push. Each caching layer subtracts ~50-70 % of tail latency.
Freshness (new instance discoverable in X s) Registration protocol, DNS TTL, push interval Total freshness = registration delay + propagation delay + client cache TTL. Set each component so their sum < SLO. For < 5 s freshness, you need push-based propagation (watch/SSE) and cache TTL ≤ 3 s.
Routing accuracy (% requests to healthy instances) Health check interval, deregistration threshold, circuit breaker config Stale window = health_check_interval × miss_threshold. During that window, traffic to a dead instance = stale_window × rps_per_instance. Set interval and threshold so stale requests / total requests < (1 − accuracy SLO). Layer client-side circuit breakers for the last mile of accuracy.
Registry availability Node count, quorum size, client-side cache fallback A 3-node Raft cluster tolerates 1 failure (99.9 % availability with standard MTBF). For 99.99 %, use 5 nodes across 3 AZs and ensure clients cache last-known-good state to survive full registry outages for up to 5 minutes.
Cross-region convergence Replication interval, inter-region RTT Convergence time = replication_interval + RTT. For a 100 ms cross-region SLO with 70 ms RTT, the replication protocol must trigger within 30 ms of a change — requiring event-driven (not periodic) federation.

Capacity planning rule of thumb: Size for 3x your current service count. Service instances grow faster than you expect — auto-scaling, canary deployments, and blue-green deploys can temporarily double your registry's working set. If your SLO math only works at current scale, you are already behind.

Scaling Journey

0 – 1K users

Config Files with IPs

You have 2-3 services on known servers. Service addresses live in environment variables or a YAML config file. Deployments are infrequent, and you manually update the config when an IP changes. This works fine because nothing moves.

1K – 100K users

DNS-Based Discovery

You're running 10-20 services with auto-scaling. IPs change often enough that config files are a burden. You set up DNS records (A or SRV) and update them via your deployment pipeline. Services look up each other by hostname. Low TTLs keep things reasonably fresh. Works until you need sub-second failover or health-aware routing.

100K – 10M users

Dedicated Registry (Consul)

You deploy a Consul or etcd cluster. Services self-register on startup and deregister on shutdown. Health checks run every 5-10 seconds. Consul's DNS interface makes migration easy — existing services just change their DNS resolver. You get a UI dashboard showing all services, their health, and metadata. Multi-datacenter federation handles cross-region discovery.

10M+ users

Kubernetes + Service Mesh

Hundreds of services on Kubernetes. Built-in DNS discovery (CoreDNS) handles basic service-to-service routing. A service mesh (Istio or Linkerd) adds traffic splitting, canary routing, mutual TLS, and observability. The mesh control plane aggregates discovery from multiple clusters. Cross-cluster service discovery via Istio multi-cluster or Consul federation.

What Can Go Wrong

Registry Single Point of Failure

What happened: A company ran a single Consul server (not a cluster) as their service registry. When that server's disk filled up, Consul crashed. All services continued running, but no new deployments could register, health checks stopped, and stale entries accumulated. A rolling deployment that started during the outage left half-deployed services in an inconsistent state — some new instances registered before the crash, old instances were still listed but had been terminated.

Root cause: Single-node registry with no redundancy. Disk space not monitored.

Mitigation: Always run the registry as a multi-node cluster (minimum 3 nodes across availability zones). Monitor disk, memory, and Raft leader election metrics. Implement client-side caching so services can continue using last-known-good addresses during a registry outage.

Stale Registrations (Zombie Entries)

What happened: A deployment script terminated old instances with kill -9 instead of a graceful shutdown. The instances died without sending deregistration requests. The health check interval was 30 seconds with a TTL of 90 seconds. For up to 90 seconds, the registry continued routing traffic to dead instances, causing connection refused errors for 10-15% of requests.

Root cause: Aggressive termination bypassed graceful deregistration. Long health check intervals delayed detection.

Mitigation: Always use graceful shutdown (SIGTERM, wait for deregistration, then SIGKILL after timeout). Shorten health check intervals for critical services. Implement client-side retry with automatic failover — if an instance is unreachable, try the next one from the registry's list.

Thundering Herd on Registry Restart

What happened: A Consul cluster restarted (planned maintenance). When it came back, all 500 services simultaneously attempted to re-register and all clients simultaneously queried for updated service lists. The Consul cluster was overwhelmed by the burst of registrations and lookups, causing slow responses and timeouts. Some services failed their own health checks because their registration calls to Consul timed out.

Root cause: All services had the same retry interval with no jitter, leading to synchronized re-registration.

Mitigation: Add random jitter to registration retry intervals. Implement exponential backoff for registry connection retries. Use client-side caching so services don't block on registry availability during startup.

DNS Caching Causing Stale Endpoints

What happened: Services used DNS-based discovery with a 60-second TTL. When a backend service was redeployed to new instances, the DNS records were updated, but many clients had cached the old IPs for up to 60 more seconds. During that window, requests to old IPs failed. The JVM's built-in DNS cache (which defaults to caching forever for successful lookups) made it worse — some Java services never picked up the new IPs until they were restarted.

Root cause: DNS TTL caching at multiple levels (OS, language runtime, application) created a long propagation delay. JVM's networkaddress.cache.ttl defaulting to -1 (infinite) was a known gotcha.

Mitigation: Set JVM DNS cache TTL explicitly (networkaddress.cache.ttl=30). Use a dedicated discovery client instead of relying on DNS caching. Implement connection draining: keep old instances running for at least one TTL period after DNS records are updated.

Health Check False Positives

What happened: A service's /health endpoint returned 200 OK because the HTTP server was running, but the service couldn't actually process requests — its database connection pool was exhausted. The registry marked the instance as healthy and kept routing traffic to it. Every request to that instance failed with a database connection timeout, but the health check kept passing.

Root cause: Shallow health check that only verified the HTTP server was alive, not that the service could serve traffic end-to-end.

Mitigation: Implement deep health checks that verify critical dependencies (database connectivity, downstream service reachability, thread pool availability). Balance depth against cost — don't make the health check so comprehensive that it becomes slow or creates its own failure mode.

Operating in Production

Key Metrics to Monitor

Metric What It Tells You Alert Threshold
registered_service_count Total number of registered service instances Alert if drops > 20% in 5 minutes (mass deregistration)
health_check_pass_rate Percentage of health checks passing across all services Alert if < 95% sustained for 2 minutes
lookup_latency_p99 Tail latency for service lookups from the registry Alert if > 50ms (should be single-digit ms)
stale_entry_count Instances registered but failing health checks Alert if > 0 sustained for 2 health check intervals
registry_leader_changes How often the Raft leader changes (indicates instability) Alert if > 2 leader elections in 10 minutes

Debug Playbook

Service can't be found by callers:

1. Check registered_service_count — is the target service registered at all?

2. If NOT registered: Check if the service is running. Check its logs for registration errors (registry unreachable? authentication failure?).

3. If registered but unhealthy: Check health_check_pass_rate for that service. Is the health endpoint returning errors? Is the service overloaded?

4. If registered and healthy but callers can't find it: Check DNS propagation (if DNS-based). Check the caller's cached service list — is it stale? Restart the caller's discovery client.

5. If lookup latency is high: Check registry cluster health. Is the Raft leader stable? Is disk I/O slow? Are there too many services causing large response payloads?

Config What It Does Recommended Value

Interactions & Pitfalls

Service Discovery + Load Balancing

Service discovery answers "where are the instances?" while load balancing answers "which instance should I call?" They are complementary but often conflated. In client-side discovery, the client does both: querying the registry and choosing an instance. In server-side discovery, the load balancer handles both. The pitfall: if the discovery cache is stale and the LB sends traffic to a terminated instance, the request fails. Ensure the LB integrates tightly with health-aware discovery — not just DNS resolution — so it removes unhealthy instances in real time.

Service Discovery + Circuit Breakers

When a circuit breaker trips for a specific instance, that instance should ideally be removed from the discovery pool (or at least deprioritized). If discovery and circuit breaking operate independently, the discovery layer keeps serving the address of an instance the circuit breaker has identified as broken. The caller trips the breaker, but the next lookup returns the same broken instance. Solution: client-side instance-level tracking that marks failed instances and avoids them for a cooldown period, or use a service mesh where the sidecar proxy handles both discovery and breaking.

Service Discovery + API Gateway

API gateways (Kong, Ambassador, AWS API Gateway) typically use service discovery to route external requests to backend services. The gateway must keep its routing table in sync with the service registry. If the gateway caches service locations too aggressively, it routes to stale endpoints after a deployment. If it queries the registry on every request, it adds latency. Most gateways subscribe to registry change notifications or poll at short intervals (5-10s) to balance freshness and performance.

Service Discovery + Rolling Deployments

During a rolling deployment, old instances are being terminated while new ones start up. The deployment orchestrator must coordinate with the service registry: register new instances only after they pass readiness checks, and deregister old instances before terminating them. If deregistration happens after termination, there's a window where traffic is routed to dead instances. If registration happens before the instance is ready, traffic hits an uninitialized service. Kubernetes handles this with readiness probes and endpoint controllers, but custom deployment scripts often get the ordering wrong.

Cost Considerations

Cost Analysis

Consul/etcd cluster: A 3-node cluster of small instances (t3.small on AWS, ~$15/month each) costs approximately $45-50/month. For production with 5 nodes across AZs: ~$75-100/month. This is one of the cheapest pieces of infrastructure you'll run.

Kubernetes Services: Free — built into the Kubernetes control plane. No additional cost beyond your existing cluster. CoreDNS runs as pods within the cluster.

Dominant cost driver: Operational complexity, not compute. Running and maintaining a Consul cluster (upgrades, backups, monitoring) requires engineering time. The compute cost is negligible.

Cost optimization: If you're already on Kubernetes, use Kubernetes-native discovery first. Don't add Consul unless you need cross-cluster or multi-cloud discovery. For small setups, DNS-based discovery costs nothing beyond your existing DNS infrastructure.

Cost of NOT having service discovery: Without automated discovery, every deployment requires manual config updates. A missed config change causes an outage. Auto-scaling becomes impossible because new instances can't be found by callers. Engineers spend hours debugging "why can't Service A reach Service B?" — which turns out to be a stale IP in a config file.

Real-World Examples

Netflix Eureka

Netflix built Eureka specifically for cloud-based service discovery. Running hundreds of microservices on AWS, Netflix needed a registry that prioritized availability over consistency — it's better to route to a slightly stale list of instances than to fail because the registry is temporarily unreachable. Eureka uses peer-to-peer replication: each Eureka server replicates registrations to its peers asynchronously. If a server goes down, clients fall back to cached registrations. Netflix reported that Eureka handles tens of thousands of registered instances and millions of lookups per minute across their AWS regions. Eureka is now part of the Spring Cloud ecosystem and widely used in Java microservice architectures.

HashiCorp Consul at Many Companies

Consul is one of the most widely adopted service discovery tools outside of Kubernetes. Companies like Criteo, Stripe, and DigitalOcean use Consul for service discovery, health checking, and configuration management. Consul's key differentiator is multi-datacenter federation: a single Consul deployment can span multiple data centers, with WAN gossip connecting the clusters. Consul also provides a service mesh capability (Consul Connect) with mutual TLS and intention-based authorization. Its DNS interface makes adoption easy — point your service's DNS resolver at the Consul agent, and redis.service.consul resolves to healthy Redis instances.

Kubernetes Services and DNS

Kubernetes is the most common platform for service discovery today. When you create a Service object, Kubernetes assigns it a stable cluster IP and a DNS name. CoreDNS, running in the cluster, resolves service names to cluster IPs (for ClusterIP services) or directly to pod IPs (for headless services). The Endpoints controller watches pod lifecycle events and updates the list of backing pod IPs in real time. When a pod fails its readiness probe, it's removed from the endpoint list within seconds. This tight integration between the scheduler, health checking, and DNS makes Kubernetes the de facto standard for container-based service discovery.

Code Example

A minimal in-memory service registry in Python with registration, deregistration, health checking, and lookup (~40 lines).

import time, threading

class ServiceRegistry:
    def __init__(self, health_ttl=30):
        self.services = {}      # {name: {instance_id: {addr, port, last_heartbeat}}}
        self.health_ttl = health_ttl
        self._start_reaper()

    def register(self, name, instance_id, addr, port):
        if name not in self.services:
            self.services[name] = {}
        self.services[name][instance_id] = {
            "addr": addr, "port": port, "last_heartbeat": time.time()
        }

    def deregister(self, name, instance_id):
        if name in self.services:
            self.services[name].pop(instance_id, None)

    def heartbeat(self, name, instance_id):
        if name in self.services and instance_id in self.services[name]:
            self.services[name][instance_id]["last_heartbeat"] = time.time()

    def lookup(self, name):
        """Return list of healthy instances for a service."""
        if name not in self.services:
            return []
        now = time.time()
        return [
            {"id": iid, "addr": info["addr"], "port": info["port"]}
            for iid, info in self.services[name].items()
            if now - info["last_heartbeat"] < self.health_ttl
        ]

    def _reap_stale(self):
        """Remove instances that missed their heartbeat TTL."""
        now = time.time()
        for name in list(self.services):
            for iid in list(self.services[name]):
                if now - self.services[name][iid]["last_heartbeat"] > self.health_ttl:
                    del self.services[name][iid]

    def _start_reaper(self):
        def run():
            while True:
                time.sleep(self.health_ttl // 3)
                self._reap_stale()
        threading.Thread(target=run, daemon=True).start()

# Usage
registry = ServiceRegistry(health_ttl=15)
registry.register("payment", "pay-1", "10.0.1.5", 8080)
registry.register("payment", "pay-2", "10.0.1.6", 8080)
print(registry.lookup("payment"))
# [{'id': 'pay-1', 'addr': '10.0.1.5', 'port': 8080}, ...]

How to Validate Your Design

Load Testing

Use k6 or vegeta to simulate lookup traffic against your registry. Key scenarios: sustained 1,000 lookups/sec to verify the registry can handle the query load without degrading. Simultaneously register and deregister instances while lookups are running to test consistency under churn. Measure lookup latency percentiles (p50, p95, p99) — lookups should be under 10ms for a healthy registry.

Chaos Testing

  • Kill a service instance (SIGKILL, not SIGTERM) and verify it's deregistered within the expected time (health check interval + deregistration timeout). Route traffic to the service during this window and confirm clients retry on another instance.
  • Test registry failover: Kill the Raft leader node in your Consul/etcd cluster. Verify a new leader is elected within seconds and that registrations and lookups continue to work.
  • Simulate network partition: Use toxiproxy to block traffic between a service and the registry. Verify the service is deregistered after the TTL expires. Restore the connection and verify re-registration.

Benchmarking

  • Rapid scaling test: Register 100 instances simultaneously and measure how long until all are visible via lookups. Consul should handle this in under a second; DNS-based systems take longer due to TTL propagation.
  • Stale entry test: Kill 50 instances simultaneously and measure the time until all are removed from the registry. This validates your health check interval and deregistration timeout settings.
  • Common mistake: Benchmarking with a warm registry cache. Test cold-start scenarios where the client has no cached service list and must query the registry from scratch.

Interview Questions

Client-side vs server-side discovery — what are the tradeoffs?

Client-side discovery means the client queries the service registry directly and picks an instance using its own load-balancing logic. The advantage is fewer network hops (no intermediate proxy) and the ability to implement smart routing in the client (e.g., latency-based, zone-aware). The disadvantage is coupling: every client must include a discovery library, and you need a library for every language in your stack. The client becomes more complex.

Server-side discovery puts a load balancer or proxy between the client and the backend services. The client calls a single stable endpoint, and the proxy handles discovery and routing. The advantage is simplicity for the client — it just makes an HTTP call. The disadvantage is the extra network hop and the proxy being a potential bottleneck or SPOF.

In practice, Kubernetes uses server-side (kube-proxy/iptables), service meshes use a hybrid (sidecar proxy that's co-located with the client), and Netflix-style architectures use client-side (Eureka + Ribbon). The trend is toward service meshes, which give you the benefits of both: smart routing without requiring application-level libraries.

How does Kubernetes service discovery work?

When you create a Kubernetes Service object, several things happen: (1) The Service is assigned a stable virtual IP (ClusterIP) that doesn't change. (2) CoreDNS creates a DNS entry mapping the service name to its ClusterIP. (3) The Endpoints controller watches for pods matching the Service's label selector and maintains a list of their IPs. (4) kube-proxy programs iptables or IPVS rules on every node that route traffic from the ClusterIP to one of the healthy pod IPs.

When a pod starts and passes its readiness probe, its IP is added to the endpoint list. When a pod fails readiness or is terminated, its IP is removed. DNS resolution happens via CoreDNS, which runs as pods in the cluster. Any pod can access a service using its short name (within the same namespace) or FQDN (service-name.namespace.svc.cluster.local).

For headless Services (ClusterIP: None), there's no virtual IP and no kube-proxy load balancing. DNS returns the individual pod IPs directly, letting the client handle load balancing. This is used for stateful workloads like databases where clients need direct pod connections.

Design a service registry that handles 10,000 services.

At 10K services, the core challenges are: health check throughput, query performance, and cluster consistency under load. Start with a 5-node Raft-based cluster (Consul or etcd) across availability zones. Each node should be a compute-optimized instance with fast SSDs (Raft is write-intensive for the WAL).

Health checks: At 10K services with 10-second intervals, that's 1,000 checks/sec. Rather than having the registry check every instance, use self-registration with heartbeats — each service sends a heartbeat, and the registry just tracks timestamps. This moves the work to the services (distributed) instead of centralizing it. The registry only runs a periodic reaper to remove entries that missed their TTL.

Lookups: Most lookups can be served from a local cache. Each client caches the service list for its dependencies and subscribes to change notifications (Consul blocking queries, etcd watches). This reduces query volume from O(requests) to O(changes). For 10K services with an average of 5 instances each and a change rate of ~10 per second (deployments, scaling), the notification load is manageable.

Data model: Key-value store keyed by service name, value is a list of instances with metadata (IP, port, health status, tags, version). Index by service name for O(1) lookups. Total data size: 10K services * ~500 bytes each = ~5MB — fits entirely in memory.

Follow-up: "How would you handle registry failover without losing registrations?"

What happens when the service registry goes down?

The impact depends on whether clients cache the registry data. With good client-side caching (which all production implementations use), existing services can continue communicating using their cached service lists. The immediate impact is: (1) new services can't register, (2) terminated services aren't deregistered (stale entries accumulate), (3) health check state isn't updated, and (4) no new lookups can be served to clients with empty caches.

In a Raft-based cluster (Consul, etcd), "goes down" usually means losing quorum — more than half the nodes are unavailable. The remaining nodes can still serve stale reads (depending on consistency settings) but can't accept writes. If the cluster was 3 nodes and you lost 2, you have a single node that can serve cached reads but can't register new services or update health.

Mitigation strategies: (1) Run enough nodes (5 across AZs) to tolerate failures. (2) Implement client-side caching with long TTLs as a fallback. (3) Eureka's AP design choice is specifically for this — it sacrifices consistency to remain available during partitions. (4) Have a DNS-based fallback: if the registry is unreachable, fall back to DNS resolution of service names. (5) Monitor registry health aggressively — Raft leader elections and cluster membership changes should trigger immediate alerts.

How to handle service discovery across multiple data centers?

Cross-datacenter discovery introduces two key challenges: latency (WAN round-trips are 10-100x slower than LAN) and network reliability (WAN links are less reliable than LAN). There are three main approaches.

Federated registries: Run a local registry in each datacenter. Federate them so each registry knows about services in other DCs. Consul supports this natively via WAN gossip — each DC has its own Consul cluster, and they gossip membership across the WAN. A lookup for a local service hits the local registry; a lookup for a remote service is forwarded to the remote DC's registry. This keeps local lookups fast while still enabling cross-DC discovery.

Global DNS: Use DNS as the cross-DC discovery layer. Each DC publishes its services as DNS records (SRV or A records with geo-aware routing). Clients resolve DNS to find the nearest DC's instances. AWS Route 53 latency-based routing does this automatically. Simple and universal, but limited by DNS TTL propagation delays.

Service mesh multi-cluster: Istio and Linkerd support multi-cluster configurations where the control plane aggregates services from multiple Kubernetes clusters (potentially in different DCs or clouds). Envoy proxies route cross-cluster traffic using mTLS over the WAN. This provides the most seamless cross-DC discovery but requires a service mesh in every cluster.

Follow-up: "How do you handle failover when an entire data center goes down?"

Consul vs Kubernetes DNS — when to use which?

Use Kubernetes DNS when: All your services run in a single Kubernetes cluster (or a few closely coupled clusters). Kubernetes DNS is free, requires zero setup, integrates with the pod lifecycle automatically, and is the path of least resistance. If you're already on Kubernetes, there's no reason to add another discovery layer for intra-cluster communication.

Use Consul when: You have services running outside Kubernetes (VMs, bare metal, Lambda functions), need multi-datacenter or multi-cloud discovery, want a unified discovery layer across heterogeneous infrastructure, or need Consul's additional features (key-value store, service mesh with Connect, ACL-based access control). Consul is also useful as a bridge during migration — it can discover services in both Kubernetes and legacy VM-based deployments.

A common hybrid pattern: use Kubernetes DNS for intra-cluster discovery and Consul for cross-cluster and cross-datacenter discovery. The Consul Kubernetes integration (consul-k8s) automatically syncs Kubernetes services to Consul, so Kubernetes-native services are discoverable via Consul by non-Kubernetes consumers.

Key Takeaways

  • Service discovery is mandatory in dynamic environments. If instances come and go (containers, auto-scaling, deployments), you need automated discovery. Hardcoded IPs are a liability.
  • Client-side vs server-side discovery is the key architectural choice. Client-side gives more control and fewer hops but couples clients to the registry. Server-side is simpler for clients but adds a proxy layer.
  • Health checks are what make discovery useful. A registry without health checking is just a list of IPs — including dead ones. Invest in meaningful health checks that verify actual traffic-serving ability.
  • The registry is a critical dependency — treat it as one. Run it as a multi-node cluster, monitor it aggressively, and implement client-side caching for resilience.
  • Kubernetes has made basic discovery a solved problem. If you're on Kubernetes, use its native Services and DNS. Add Consul or a service mesh only when you outgrow what Kubernetes provides (cross-cluster, multi-cloud, advanced traffic management).
  • DNS caching is the most common source of stale-discovery bugs. Understand TTL behavior at every layer (OS, language runtime, application) and configure them deliberately.
  • Coordinate discovery with deployments. Register after ready, deregister before terminate. Getting the ordering wrong causes traffic to hit instances that aren't ready or are already gone.
