B-Tree vs LSM-Tree — Animated

Step through the four core operations — build from array, insert, update, and delete — on both index structures. The B-tree updates in place; the LSM-tree appends and compacts. Watch what each design pays for.

B-Tree (order 4)

A B-tree of order m = 4 stores up to 3 keys per node and has up to 4 children. Keys within a node are sorted; child i holds keys between keyi-1 and keyi. All leaves sit at the same depth. The tree grows by splitting a full node from the bottom up; it shrinks by merging or borrowing from siblings.

Insert

Walk down to a leaf, insert the key in sorted order. If the leaf overflows (4 keys), split it into two leaves of 2 keys and push the median up to the parent. If the parent overflows, recurse. The root may split, growing the tree by one level.

Update

A B-tree value update is in place: walk to the leaf containing the key, overwrite the value. The tree shape doesn’t change — one logical I/O, one page rewrite. (For sorted-key updates, the engine deletes-then-inserts.)

Delete

Find the key. If it’s in an internal node, swap with its in-order successor from a leaf, then delete from the leaf. If the leaf underflows (< ⌈m/2⌉-1 = 1 key), borrow a key from a sibling through the parent, or merge with a sibling. Merging may cascade up; the root may shrink by one level.

Build from array

Sequential inserts produce an unbalanced shape compared to bulk-loading from sorted input. Watch the splits cluster on the right side of the tree as ascending keys bubble up the median pointer.

LSM-Tree

An LSM-tree (log-structured merge tree) absorbs writes into an in-memory memtable (a sorted skiplist). When the memtable fills, it’s flushed to disk as an immutable SSTable at L0. Background compaction merges overlapping SSTables into larger sorted runs at deeper levels (L1, L2, ...). Reads check memtable → L0 → L1 → ... and return the newest version found.

Insert

Append the (key, value) to the WAL (durability) and put it in the memtable. O(log n) on a skiplist, no disk seek. Writes never modify existing data on disk.

Update

Same as insert: append a new (key, value) entry. The previous version still exists on disk in some lower-level SSTable; the engine keeps both. On read, the newer version (memtable or shallower level) wins. Compaction eventually reclaims space by dropping the older version.

Delete

Append a tombstone: a special record marking the key as deleted. The original key still lives in older SSTables; the tombstone shadows it on read. Both are physically removed only when compaction merges all SSTables that contain that key.

Build / flush / compact

As inserts pile up, the memtable fills, flushes to L0, and L0 SSTables eventually merge into a single sorted L1 run with overlapping ranges resolved by recency. Watch the obsolete entries get dropped during compaction.

Related Topics

SQL vs NoSQL

Most SQL engines use B-trees; most NoSQL KV stores use LSM-trees. The choice ripples into write throughput, read latency, and operational shape.

Caching

The B-tree page cache and the LSM-tree block cache both fight the same fight: keep the working set in RAM.

Bloom Filters

The reason an LSM-tree point lookup doesn’t have to read every level — a per-SSTable bloom filter rules out most levels in O(1).

Database Sharding

Both indexes scale up via sharding; LSM-trees compose more naturally with append-only replication logs.
