WebSockets

Full-duplex, persistent connections that enable real-time bidirectional communication between clients and servers over a single TCP connection.

30 min read

What Is It?

A WebSocket is a communication protocol that provides a full-duplex, persistent connection between a client (typically a browser) and a server over a single TCP socket. Unlike the traditional HTTP request-response model where the client must initiate every exchange, a WebSocket connection allows both sides to send data at any time, independently and simultaneously. Once the connection is established, it stays open until either side explicitly closes it or the network drops.

The connection begins life as a normal HTTP request. The client sends an HTTP Upgrade handshake—a regular GET request with special headers (Upgrade: websocket, Connection: Upgrade, and a Sec-WebSocket-Key). If the server supports WebSockets, it responds with HTTP 101 Switching Protocols, and from that moment the connection is no longer HTTP. It becomes a lightweight, framed, bidirectional binary protocol running over the same TCP connection. This handshake is critical because it lets WebSocket traffic pass through existing HTTP infrastructure—proxies, load balancers, firewalls—that understand HTTP.

Sec-WebSocket-Key Validation: Preventing HTTP/WebSocket Confusion

The handshake includes a cryptographic validation step to ensure both sides genuinely intend to speak WebSocket. Without this, a malicious page could trick a browser into sending a crafted HTTP request to a WebSocket endpoint (or vice versa), potentially poisoning HTTP caches or confusing intermediary proxies. Here is the exact computation defined by RFC 6455:

Step 1: The client generates a random 16-byte nonce and base64-encodes it. For example:

Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==

Step 2: The server takes this key and appends a fixed "magic string" defined in the RFC:

# The magic GUID is hardcoded in RFC 6455 - never changes
magic = "258EAFA5-E914-47DA-95CA-C5AB0DC85B11"
concatenated = "dGhlIHNhbXBsZSBub25jZQ==" + magic
# Result: "dGhlIHNhbXBsZSBub25jZQ==258EAFA5-E914-47DA-95CA-C5AB0DC85B11"

Step 3: The server computes the SHA-1 hash of the concatenated string, then base64-encodes the 20-byte hash digest:

import hashlib, base64

key = "dGhlIHNhbXBsZSBub25jZQ=="
magic = "258EAFA5-E914-47DA-95CA-C5AB0DC85B11"

# SHA-1 hash of the concatenated string
sha1_hash = hashlib.sha1((key + magic).encode()).digest()
# sha1_hash = b'\xb3\x7a\x4f\x2c\xc0\x62\x4f\x16\x90\xf6\x46\x06\xcf\x38\x59\x45\xb2\xbe\xc4\xea'

accept_value = base64.b64encode(sha1_hash).decode()
# accept_value = "s3pPLMBiTxaQ9kYGzzhZRbK+xOo="

Step 4: The server returns this value in the response header:

Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

The client verifies that the Sec-WebSocket-Accept value matches what it expects (it performs the same computation locally). If the values do not match, the client closes the connection immediately.

Why this matters: The magic GUID ensures that only a real WebSocket server (one that knows the RFC 6455 protocol) can produce the correct Sec-WebSocket-Accept header. A regular HTTP server or caching proxy would not know to perform this computation and would either reject the request or return a wrong value. This prevents cache poisoning attacks where an attacker tricks a proxy into caching a WebSocket upgrade response as if it were a normal HTTP response, which could later be served to other users. The nonce (random key) ensures each handshake is unique, preventing replay attacks.

Why not just use HTTP? Consider the alternatives. Short polling means the client sends a request every N seconds asking "anything new?"—wasteful when there is nothing new, and sluggish when there is. Long polling improves on this by having the server hold the request open until it has something to send—but each response still requires a new connection, and you only get server-to-client messages (the client must send another request to talk back). Server-Sent Events (SSE) give you a persistent server-to-client stream, but it is unidirectional—the client cannot push data back over the same channel. WebSockets are the only option that gives you true, low-latency, bidirectional communication over a single persistent connection.

How It Works

1. HTTP Upgrade Handshake

The client sends a standard HTTP GET with upgrade headers. The server validates the request and responds with 101 Switching Protocols. The Sec-WebSocket-Key and Sec-WebSocket-Accept headers prevent accidental upgrades and cross-protocol attacks. After this handshake, the TCP connection is repurposed for WebSocket frames.

# Client request headers
# GET /chat HTTP/1.1
# Host: server.example.com
# Upgrade: websocket
# Connection: Upgrade
# Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
# Sec-WebSocket-Version: 13

# Server response
# HTTP/1.1 101 Switching Protocols
# Upgrade: websocket
# Connection: Upgrade
# Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

2. Frame Format

Once connected, data is sent in frames. Each frame has a small header (2-14 bytes) containing: a FIN bit (is this the last fragment?), an opcode (text=0x1, binary=0x2, close=0x8, ping=0x9, pong=0xA), a masking bit (client-to-server frames must be masked), and the payload length. The framing overhead is minimal—as low as 2 bytes for a small message, versus ~800 bytes for an equivalent HTTP request with headers.

Byte-by-Byte Frame Breakdown

Let us trace how the text message "Hello" (5 ASCII bytes) is encoded in a WebSocket frame sent from a client to a server. Since it is client-to-server, it must be masked (RFC 6455, Section 5.1).

Byte PositionBinaryHexFieldMeaning
Byte 01 000 00010x81FIN + RSV1-3 + OpcodeFIN=1 (final fragment), RSV=000, Opcode=0x1 (text frame)
Byte 11 00001010x85MASK + Payload lengthMASK=1 (masked), Length=5 bytes
Bytes 2-50x37 0xFA 0x21 0x3DMasking key4 random bytes chosen by the client
Bytes 6-100x7F 0x9F 0x4D 0x51 0x58Masked payload"Hello" XOR'd with the masking key

Total frame size: 11 bytes for a 5-byte message. Compare this to an HTTP POST with headers, which would be ~800+ bytes for the same payload.

The XOR Masking Step-by-Step

Masking transforms each byte of the payload by XOR-ing it with the corresponding byte of the 4-byte masking key, cycling through the key:

# Masking key (4 random bytes chosen by client)
mask_key = [0x37, 0xFA, 0x21, 0x3D]

# Original payload: "Hello"
payload = [0x48, 0x65, 0x6C, 0x6C, 0x6F]  # H=0x48, e=0x65, l=0x6C, l=0x6C, o=0x6F

# Masking formula: masked[i] = payload[i] XOR mask_key[i % 4]
masked = []
for i in range(len(payload)):
    masked_byte = payload[i] ^ mask_key[i % 4]
    masked.append(masked_byte)

# Step-by-step:
# i=0: 'H' 0x48 XOR 0x37 = 0x7F   (mask_key[0])
# i=1: 'e' 0x65 XOR 0xFA = 0x9F   (mask_key[1])
# i=2: 'l' 0x6C XOR 0x21 = 0x4D   (mask_key[2])
# i=3: 'l' 0x6C XOR 0x3D = 0x51   (mask_key[3])
# i=4: 'o' 0x6F XOR 0x37 = 0x58   (mask_key[0], cycles back)

# Masked payload: [0x7F, 0x9F, 0x4D, 0x51, 0x58]
# To unmask, apply the SAME XOR again: 0x7F XOR 0x37 = 0x48 = 'H'

Why must client-to-server frames be masked? This is not for encryption (the masking key is sent in the clear). The purpose is to prevent cache poisoning attacks. Without masking, an attacker could craft a malicious web page that opens a WebSocket connection through a transparent HTTP proxy. The proxy does not understand WebSocket frames and interprets the raw bytes as an HTTP request/response. By carefully constructing the payload, the attacker could inject a fake HTTP response into the proxy's cache, which would then be served to other users. Masking with a random key makes it impossible for the attacker to predict the byte sequence on the wire, preventing this class of attacks. Server-to-client frames are not masked because the server is trusted and does not run attacker-controlled code.

Payload Length Encoding

The length field uses a variable-length encoding to keep small frames compact:

Payload SizeLength Field (7 bits of byte 1)Extended LengthTotal Header Overhead

For our "Hello" example (5 bytes), the length fits in the 7-bit field directly, so no extended length bytes are needed. A 1KB message would use the 2-byte extended encoding. A 100KB message would also use the 2-byte extended encoding. Only messages larger than 64KB need the 8-byte extended encoding.

3. Ping/Pong Heartbeats

WebSocket defines control frames for keepalive. The server (or client) sends a Ping frame; the other side must respond with a Pong frame carrying the same payload. This serves two purposes: it detects dead connections (if no pong comes back within a timeout, close the socket) and it keeps the connection alive through intermediaries like proxies and NATs that may close idle connections.

Ping/Pong Timeline: Detecting a Dead Connection

Consider a server configured with ping_interval = 25s and ping_timeout = 5s. Here is a concrete timeline showing a healthy connection that eventually goes dead:

TimeEventDirectionConnection Status
T=0.00sWebSocket connection established—Open
T=25.00sServer sends Ping frame (payload: 0x01)Server → ClientOpen, waiting for pong
T=25.01sClient replies with Pong frame (payload: 0x01)Client → ServerOpen, healthy
T=50.00sServer sends Ping frame (payload: 0x02)Server → ClientOpen, waiting for pong
T=50.01sClient replies with Pong frame (payload: 0x02)Client → ServerOpen, healthy
T=60.00sClient's network drops (phone enters tunnel)—Open (server does not know yet)
T=75.00sServer sends Ping frame (payload: 0x03)Server → (void)Open, waiting for pong
T=80.00s5 seconds elapsed, no Pong received—Server closes connection (dead)

Without ping/pong, the server would never know the client disappeared. The TCP connection would remain in memory indefinitely as a "zombie connection," leaking memory. The server only discovers the client is gone when it tries to write data and the OS eventually reports a TCP error—which can take minutes or even hours depending on TCP keepalive settings.

Why 25 Seconds? Proxy Idle Timeouts

Most intermediaries (reverse proxies, load balancers, NAT gateways, corporate firewalls) have an idle connection timeout—they close connections where no data has flowed for a certain period. Common defaults:

  • Nginx: proxy_read_timeout defaults to 60 seconds
  • AWS ALB: idle timeout defaults to 60 seconds
  • HAProxy: timeout tunnel defaults to no timeout (but often configured to 60-300s)
  • NAT gateways: typically 30-300 seconds depending on the device
  • Corporate firewalls: can be as low as 30 seconds

The ping interval must be shorter than the smallest idle timeout in the entire network path. Since the smallest common timeout is 30 seconds (NAT/firewall), a 25-second ping interval provides a safety margin. Each ping/pong exchange resets the idle timer on every intermediary in the path, keeping the connection alive even when no application data is flowing.

4. Close Handshake

Either side can initiate a graceful close by sending a Close frame with a status code (e.g., 1000 for normal closure, 1001 for going away). The other side responds with its own Close frame, and then both sides close the TCP connection. This two-way close prevents data loss from in-flight messages.

When to Use / When NOT to Use

Use WebSockets When

  • Real-time chat: Messages must be delivered instantly in both directions. WebSocket latency is measured in single-digit milliseconds.
  • Live multiplayer gaming: Game state updates need to flow continuously between server and multiple clients at high frequency (30-60 updates/sec).
  • Live dashboards and monitoring: Metrics, alerts, and status updates should push to the UI the moment they change, not on a poll cycle.
  • Collaborative editing: Google Docs-style editing requires every keystroke or operation to propagate to all participants in real time.
  • Financial tickers: Stock prices, order book updates, and trade notifications demand sub-millisecond delivery of high-frequency data.
  • Presence and typing indicators: Showing "user is typing..." or online/offline status requires lightweight, frequent, bidirectional signals.

Don't Use WebSockets When

  • Simple request-response: If your app is CRUD with occasional reads, HTTP is simpler, better cached, and more widely supported. Don't add complexity you don't need.
  • Infrequent updates: If data changes every 30 seconds or less, SSE or even short polling is simpler and avoids the operational overhead of persistent connections. SSE also auto-reconnects.
  • Fire-and-forget writes: If the client just sends data and does not need a response (telemetry, analytics events), plain HTTP POST is fine. No need for a persistent connection.
  • Server-only push: If data only flows server-to-client (news feed, live scores), SSE is lighter weight, uses standard HTTP, and works through CDNs and proxies without special configuration.
  • Corporate or restricted networks: Some firewalls and proxies block WebSocket upgrades. If you cannot control the network environment, SSE or long-polling with fallback is safer.

Choose WebSocket over SSE when:

You need the client to push data back to the server over the same connection (chat, gaming), you need binary data support (SSE is text-only), or you need the absolute lowest possible latency. Choose SSE over WebSocket when data flow is only server-to-client, you want automatic reconnection built into the browser API, or you need to work through HTTP infrastructure that does not support WebSocket upgrade.

Classic vs. Modern Approaches

Classic: Raw WebSocket API

The browser's native WebSocket API is straightforward: new WebSocket('wss://...') gives you onopen, onmessage, onclose, and onerror callbacks. The server-side equivalent exists in every language (Python's websockets, Go's gorilla/websocket, Node's ws). Raw WebSocket is simple but leaves you to implement reconnection logic, heartbeats, message serialization, room/channel abstraction, and fallback for environments that block upgrades.

Socket.IO (Fallbacks + Rooms)

Socket.IO wraps WebSocket with automatic fallback to HTTP long-polling, built-in reconnection with exponential backoff, room/namespace abstractions for fan-out, and binary support. It is the most popular real-time library (2M+ weekly npm downloads) and works in environments where raw WebSocket fails. The tradeoff: it uses a custom protocol on top of WebSocket, so Socket.IO clients cannot talk to plain WebSocket servers and vice versa.

WebSocket over HTTP/2

RFC 8441 (2018) enables WebSocket connections to be tunneled over HTTP/2 streams. This means multiple WebSocket connections can share a single TCP connection (multiplexing), reducing the connection overhead and playing nicely with HTTP/2 infrastructure. Browser support is growing but not universal.

WebTransport (QUIC-Based)

WebTransport is the newest contender, built on HTTP/3 and QUIC. It provides bidirectional streams (like WebSocket), unidirectional streams, and unreliable datagrams (like UDP)—all multiplexed over a single QUIC connection. QUIC eliminates head-of-line blocking and handles connection migration (switching between WiFi and cellular without reconnecting). WebTransport is ideal for gaming and video streaming where occasional packet loss is acceptable. It is still emerging (Chrome supports it, Firefox is implementing it).

gRPC Streaming

gRPC supports server streaming, client streaming, and bidirectional streaming over HTTP/2. For service-to-service communication, gRPC streaming offers strong typing via Protocol Buffers, built-in flow control, and deadline propagation. It is not suitable for browser-to-server communication without a proxy (gRPC-Web), but for backend microservices, it is often a better choice than raw WebSocket.

Phoenix Channels / LiveView Pattern

Elixir's Phoenix framework pioneered a pattern where WebSocket connections are managed by lightweight BEAM processes, each using only ~2KB of memory. Phoenix Channels provides presence tracking, topic-based pub/sub, and graceful reconnection out of the box. LiveView takes this further by managing UI state entirely on the server and pushing DOM diffs over WebSocket, eliminating the need for a JavaScript SPA. This pattern has inspired similar approaches: Laravel LiveWire, Rails Hotwire/Turbo Streams, and Django Channels.

Assumptions & Model

Key Assumptions

  • Persistent connections are maintainable: The model assumes the network supports long-lived TCP connections. This breaks on mobile networks where connections are frequently dropped during cell tower handoffs, and on networks with aggressive NAT timeout (often 30-60 seconds). You must implement reconnection with exponential backoff.
  • Clients can handle full-duplex: We assume both sides can send and receive simultaneously. In practice, browser clients handle this natively, but embedded devices or constrained environments may not support WebSocket at all.
  • Intermediaries support WebSocket upgrade: Proxies, load balancers, CDNs, and firewalls must understand and pass through the HTTP 101 Upgrade handshake. This assumption breaks frequently: corporate HTTP proxies often strip Upgrade headers, older nginx configurations require explicit proxy_set_header Upgrade directives, and many CDNs (CloudFront until recently) do not support WebSocket.
  • Memory scales linearly with connections: The sizing model assumes each connection has a roughly constant memory footprint. This breaks if connections carry vastly different state sizes (e.g., one connection subscribed to 1 room vs. 10,000 rooms) or if message buffering during backpressure causes memory spikes.
  • Message ordering is guaranteed per connection: TCP guarantees in-order delivery within a single connection. However, this guarantee disappears across reconnections (client disconnects and reconnects, possibly to a different server). You need sequence numbers or vector clocks for cross-reconnect ordering.

Sizing & Tradeoffs

Memory Per Connection

Each WebSocket connection consumes memory for: the TCP socket buffer (~8-16KB read + write), the application-level connection state (session data, subscribed rooms, pending messages), and any framework overhead. A minimal connection uses ~10KB; a typical production connection (with buffers and state) uses ~30-50KB.

Connections Per Server

The upper bound is set by file descriptors (each connection = 1 fd; Linux default is 1024, production servers set this to 1M+) and available RAM. At 30KB per connection, a server with 64GB of RAM dedicated to WebSocket state can theoretically hold ~2M connections.

SLO-Driven Sizing Examples

Example 1: Real-Time Chat App

SLO: p99 message delivery < 100ms, 1M concurrent connections.

Memory sizing: Each chat connection carries user metadata, room subscriptions (avg 5 rooms), and a 4KB outbound buffer = ~40KB per connection. 1M × 40KB = 40GB RAM for connection state alone. Allow 20GB headroom for OS, GC, and application code → 64GB instances (r6g.2xlarge).

Server count: To keep p99 delivery under 100ms, each server should handle no more than 200K connections (reduces tail latency from per-server fan-out). 1M ÷ 200K = 5 servers minimum, deploy 7 for redundancy (survive 2 failures without SLO breach).

Message rate: At 0.1 msg/sec sent per user, fan-out to avg 20 recipients per room message: effective outbound rate per server = 200K × 0.1 × 20 = 400K msg/sec. At 200B avg message, bandwidth = 80MB/s = 640 Mbps — well within 10 Gbps NIC. CPU at 1μs per message = 0.4 cores. This workload is memory-bound.

Example 2: Live Sports Scores

SLO: p95 update latency < 500ms, 5M concurrent viewers.

Fan-out architecture: Score updates are publisher-driven (servers push to all viewers of a match). A single match may have 2M viewers. Broadcasting 2M messages sequentially at 10μs each = 20 seconds — far exceeds the 500ms SLO. Solution: two-tier fan-out. Origin servers publish to N edge relay servers via pub/sub; each edge server fans out to its local connections.

Edge server sizing: Each viewer connection is read-only, minimal state = ~15KB per connection. At 100K connections per edge server: 100K × 15KB = 1.5GB RAM. 5M ÷ 100K = 50 edge servers. Score updates are small (100B) and infrequent (every few seconds per match), so CPU and bandwidth are negligible.

Latency budget: Origin → pub/sub (10ms) + pub/sub → edge server (20ms) + edge fan-out to 100K clients (100K × 5μs = 500ms if serial). To hit p95 < 500ms: use async I/O with write batching (epoll/kqueue), reducing fan-out to ~50ms for 100K connections. Total: ~80ms p95, well within SLO.

Example 3: Collaborative Document Editing

SLO: p99 < 50ms for cursor/keystroke sync, 100K concurrent documents (avg 5 collaborators each = 500K connections).

Connection routing: All collaborators on the same document must connect to the same server (or share state via CRDT/OT with sub-50ms sync). Use consistent hashing on document ID to route connections. Each server handles ~2,000 documents × 5 users = 10K connections.

Server sizing: Each connection carries CRDT state for the active document (~200KB for a large doc) + cursor positions for all collaborators. Memory per connection: ~250KB. Per server: 10K × 250KB = 2.5GB RAM. 100K documents ÷ 2K docs/server = 50 servers.

Latency budget: Keystroke → server (network RTT, ~10ms same region) + OT/CRDT transform (1-5ms) + broadcast to 4 other collaborators (negligible). Total p99: ~20ms within-region. Cross-region collaboration requires regional relay or CRDT with eventual convergence < 50ms via dedicated inter-region links.

CPU is the constraint: OT transforms at high keystroke rates (10 keystrokes/sec × 5 users × 2K docs = 100K transforms/sec per server). At 50μs per transform = 5 CPU cores per server. Use c6g.2xlarge (8 vCPU) instances.

Example 4: Financial Trading Dashboard

SLO: p99 < 10ms tick delivery, 50K concurrent traders.

Dedicated connection pool: At this latency target, no pub/sub backplane — each server directly receives the market data feed. Traders subscribe to specific symbols (avg 50 symbols each). Use a topic-partitioned architecture: partition symbols across N servers, each trader connects to the servers that own their symbols (multi-connection client).

Server sizing: 10K symbols ÷ 5 partitions = 2K symbols per server. Each server handles ~20K trader connections (some traders overlap across partitions). Memory per connection: 20KB state + 8KB buffer = 28KB. Per server: 20K × 28KB = 560MB RAM. Memory is not the bottleneck.

CPU is the bottleneck: Market data arrives at 500K ticks/sec total, 100K ticks/sec per server. Each tick fans out to avg 200 subscribed traders on that server. Outbound: 100K × 200 = 20M messages/sec. At 0.5μs per message write = 10 CPU cores. Use c6g.4xlarge (16 vCPU) with kernel bypass (io_uring or DPDK) to hit p99 < 10ms.

Bandwidth: 20M msg/sec × 100B per tick = 2GB/sec = 16 Gbps per server. Requires 25 Gbps NIC — use network-optimized instances.

SLO-Driven Capacity Planning

Traditional WebSocket sizing starts with "how many connections can a server hold?" SLO-driven capacity planning inverts this: start with the latency and reliability targets, then derive the infrastructure.

Step 1: Define the SLO Budget

Break the end-to-end SLO into per-hop budgets. For a p99 < 100ms message delivery SLO in a chat app:

HopBudgetWhat Drives It
Client → Server (network)30msRegional deployment, CDN/edge termination
Server processing5msCPU speed, event loop efficiency
Pub/sub backplane15msRedis/NATS latency, partition count
Receiving server fan-out20msConnections per server, async I/O model
Server → Client (network)30msRegional deployment

Step 2: Size Per-Server Load from the Fan-out Budget

If you have 20ms for fan-out and each message write takes 5μs (epoll-based async I/O), a single thread can write to 20ms ÷ 5μs = 4,000 connections within budget. With 4 I/O threads, that is 16K connections in a single room. If your largest room has 50K users, you need at least 4 servers sharing that room's fan-out — which means pub/sub-based distribution to multiple servers.

Step 3: Add Headroom for Failure Scenarios

If you run N servers at 80% capacity (your SLO holds at this load), losing one server redistributes its connections to N-1 servers. The new per-server load is N/(N-1) × 80%. For N=5: each surviving server takes 100% load — right at the edge. For N=7: each takes 93% — still within SLO. Rule of thumb: provision at least N+2 servers where N is the minimum for SLO compliance.

Step 4: Validate with Load Testing

Synthetic sizing models assume uniform connection behaviour. In production, connections are non-uniform (power-law room sizes, bursty message patterns, variable message sizes). Always validate your model with a load test that replays production traffic patterns, and measure p99 latency at 80%, 90%, and 100% of target connection count. Your sizing model is only as good as its worst-case validation.

Key Tradeoff: Connection Count vs. Message Rate

A system with millions of mostly-idle connections (presence, notifications) is memory-bound. A system with thousands of high-frequency connections (gaming, trading) is CPU-bound. Design your infrastructure accordingly: use large-memory instances for the first case, high-CPU instances for the second. The SLO examples above illustrate this spectrum: the sports scores system is memory-bound (5M idle viewers), while the trading dashboard is CPU-bound (50K connections but 20M outbound msg/sec per server).

Scaling Journey

0–1K users

Single Server, In-Memory State

All WebSocket connections terminate on one server. Room membership and message routing are stored in a Map<roomId, Set<connection>> in memory. Broadcasting to a room is a simple loop. No external dependencies. This handles thousands of connections trivially.

Signal to evolve: You need more than one server for redundancy or the single server cannot handle the load.

1K–100K users

Sticky Sessions with Load Balancer

Add a load balancer (HAProxy, Nginx, ALB) that routes WebSocket connections. Use sticky sessions (cookie or IP hash) so a client always connects to the same backend. Each server still manages its own connections in memory. Problem: broadcasting to a room only reaches users connected to that specific server.

Signal to evolve: Users in the same room are on different servers and cannot see each other's messages.

100K–10M users

Pub/Sub Backplane

Add a pub/sub system (Redis Pub/Sub, NATS, Kafka) as a backplane. When server A receives a message for a room, it publishes to the backplane. All servers subscribe and forward to their local connections in that room. This is the Socket.IO + Redis adapter pattern. Redis Pub/Sub handles ~1M msg/sec on a single node.

Signal to evolve: Redis Pub/Sub becomes a bottleneck (it is single-threaded), or you need cross-region support.

10M+ users

Dedicated WebSocket Service + Edge Termination

Extract WebSocket handling into a dedicated, horizontally scalable service. Use sharded pub/sub (Kafka partitions, Redis Cluster pub/sub) for the backplane. Terminate WebSocket connections at the edge (Cloudflare, Fastly) to reduce latency. Use consistent hashing to route rooms to specific backend shards, reducing cross-server fan-out. Slack, Discord, and Figma all run dedicated WebSocket tiers at this scale.

What Can Go Wrong

1. Reconnection Storm (Thundering Herd)

What happened: A WebSocket server crashes or is restarted during a deploy. All 500K connected clients detect the connection loss simultaneously and attempt to reconnect at the same time.

Root cause: Clients use a fixed or small reconnection delay. 500K connections hit the server (or load balancer) in a 1-2 second window.

Blast radius: The server or load balancer is overwhelmed by the connection spike, causing cascading failures. New connections timeout and retry, amplifying the storm.

Detection: Spike in TCP SYN packets, load balancer connection rate metrics, HTTP 503 errors on upgrade requests.

Mitigation: Implement jittered exponential backoff on the client. The first retry should be random within [0, 1s], then [0, 2s], [0, 4s], etc. This spreads reconnections over time. Server-side: use rolling deploys so only a fraction of connections are disrupted at once.

Prevention: Always deploy with rolling updates. Client libraries must include jitter. Provision headroom for 2x the normal connection rate during deploys.

2. Memory Leak from Unclosed Connections

What happened: A production Node.js WebSocket server's memory usage grew steadily over days, eventually triggering OOM kills every 48 hours.

Root cause: When a client disconnected uncleanly (network drop, browser crash), the server-side connection object was not properly cleaned up. Event listeners, room subscriptions, and message buffers accumulated. The application relied on the close event which never fires for TCP half-open connections without ping/pong.

Blast radius: Server becomes unresponsive as memory pressure increases, affecting all connected users on that server.

Detection: Monitor RSS memory growth over time. Track connection count vs. expected active users—a divergence indicates leaked connections.

Mitigation: Enable server-side ping/pong with a timeout (e.g., ping every 30s, close if no pong within 10s). Use WeakRef or explicit cleanup in disconnect handlers. Set tcp_keepalive at the OS level as a backstop.

3. Proxy Timeout Killing Idle Connections

What happened: Users reported frequent disconnections every 60 seconds despite no network issues. The WebSocket server logs showed no close frames received.

Root cause: An upstream nginx proxy had proxy_read_timeout 60s configured. Idle WebSocket connections (no data flowing) were silently closed by the proxy after 60 seconds. No close frame was sent to either side—TCP RST was used.

Detection: Pattern of disconnections at exactly the timeout interval. Missing close frames in server logs.

Mitigation: Set proxy_read_timeout to a high value (3600s or more) for WebSocket endpoints. Implement application-level ping/pong at an interval shorter than the smallest proxy timeout in the path (e.g., ping every 25s for a 60s timeout). Document all proxy/LB timeout values in the chain.

4. Backpressure: Slow Consumer Overwhelming the Server

What happened: A client on a slow 2G mobile network (effective throughput: ~50KB/s) was subscribed to a high-volume room producing 100KB/s of messages. The server's memory usage grew unboundedly until OOM kill.

Root cause: The server was sending messages faster than the client could receive them. Here is the mechanical breakdown of what happens:

TimeEventTCP Send BufferApp BufferAction
T=0.0sConnection established, server starts sending 100KB/s0 KB / 64 KB0 KBNormal operation
T=0.5sServer has sent 50KB, client has consumed 25KB25 KB / 64 KB0 KBTCP buffer absorbing the difference
T=1.3sTCP send buffer is full (64KB). write() would block.64 KB / 64 KB0 KBOS stops accepting writes from app
T=1.3s+Async WebSocket library starts queuing messages in application memory64 KB / 64 KBGrowing...App buffer accumulates at ~50KB/s

Detection: Monitor per-connection outbound buffer size. Alert when any connection's buffer exceeds a threshold (e.g., 1MB). Track the bufferedAmount property on the WebSocket object (browser-side) or the equivalent write-buffer metric server-side.

Mitigation strategies:

  • Disconnect slow consumers: Set a maximum outbound buffer per connection (e.g., 4MB). If the buffer exceeds this, close the connection with a close code indicating the reason. The client can reconnect and catch up via message history.
  • Drop messages for slow consumers: For non-critical data (e.g., cursor positions, presence updates), drop the oldest queued messages when the buffer is full. Keep only the most recent state.
  • Check if the send buffer is draining before writing:
import asyncio
import websockets

MAX_BUFFER = 4 * 1024 * 1024  # 4 MB

async def safe_send(ws, message):
    """Send a message, but disconnect if the client is too slow."""
    # Check how much data is queued but not yet sent
    if ws.transport and ws.transport.get_write_buffer_size() > MAX_BUFFER:
        # Client cannot keep up — disconnect it
        await ws.close(4001, "Slow consumer: buffer limit exceeded")
        return False

    try:
        await ws.send(message)
        return True
    except websockets.ConnectionClosed:
        return False

async def broadcast_to_room(room_id, message, rooms):
    """Broadcast to all room members, disconnecting slow consumers."""
    peers = list(rooms.get(room_id, set()))
    results = await asyncio.gather(
        *[safe_send(peer, message) for peer in peers],
        return_exceptions=True
    )
    # Remove disconnected peers
    for peer, ok in zip(peers, results):
        if ok is False or isinstance(ok, Exception):
            rooms[room_id].discard(peer)

Prevention: Always set backpressure_limit in your WebSocket server configuration. Never allow unbounded per-connection buffers. In production, the most common approach is to disconnect slow consumers and let them reconnect, rather than trying to queue indefinitely.

Debug Playbook

Users report disconnections?

→ Check ws_reconnect_rate. If spiking, check for deploys or proxy restarts.

→ Check connection duration distribution. If clustered at a fixed interval (60s, 120s), a proxy timeout is killing idle connections. Increase proxy_read_timeout and verify ping interval.

→ If durations are random and short, check server memory (OOM kills) and CPU (event loop lag on Node.js, GC pauses on JVM).


Messages are delayed?

→ Check ws_message_send_latency_p99. If high, check outbound buffer size per connection (slow consumers).

→ Check pub/sub backplane latency (Redis Pub/Sub lag, Kafka consumer lag). If high, the fan-out system is the bottleneck.

→ Check event loop lag or thread pool saturation on the WebSocket server.


Memory growing steadily?

→ Compare ws_connections_active to expected user count. If connections >> users, you have a connection leak (missing close handlers).

→ Check for undelivered message buffers accumulating on slow connections. Enforce backpressure limits.

Interactions & Pitfalls

WebSockets + Load Balancing

WebSocket connections are long-lived and stateful, which conflicts with how most load balancers work. An L7 load balancer only makes a routing decision at connection time—it cannot rebalance an existing connection. If you add a new server, existing connections do not migrate to it. You need either sticky sessions (route by client ID or IP hash) or connection-aware routing (the load balancer knows which server handles which user). Without this, a client that reconnects may land on a different server and lose its room subscriptions. AWS ALB supports WebSocket natively; Nginx requires explicit Upgrade header forwarding.

WebSockets + Message Queues (Pub/Sub Backplane)

When you scale beyond one server, you need a way to broadcast a message to all servers that have clients in a given room. Redis Pub/Sub is the classic choice: cheap, fast, fire-and-forget. But Redis Pub/Sub has no persistence—if a server is temporarily disconnected from Redis, it misses messages permanently. For durability, use Kafka or NATS JetStream as the backplane, but accept higher latency. The pitfall: using a durable queue for real-time fan-out adds latency and complexity that you may not need. Match the backplane's guarantees to your application's needs.

Step-by-Step Trace: Cross-Server Message Delivery via Redis Pub/Sub

Consider this scenario: User Alice is connected to Server A and sends a message to room "general". Users Bob and Carol are in the same room but connected to Server B. Here is exactly what happens:

StepActionWhereLatency
1Alice sends WebSocket message: {"type":"message","room":"general","text":"Hi!"}Alice → Server A~10ms (network RTT)
2Server A receives the message, validates it, and checks local connections in room "general". Server A finds 12 local members and delivers to them immediately.Server A (local)~0.1ms
3Server A publishes the message to Redis channel room:general via PUBLISH room:general '{"from":"alice","text":"Hi!"}'Server A → Redis~1ms
4Redis delivers the published message to all subscribers of room:general. Server B (and any other servers with members in that room) receives the message via its subscription.Redis → Server B~1ms
5Server B looks up its local connections in room "general", finds Bob and Carol, and sends the WebSocket message to both.Server B (local)~0.05ms
6Bob and Carol receive the message in their browsers.Server B → Bob, Carol~10ms (network RTT)

Total end-to-end latency: ~22ms (Alice's network RTT + Redis round-trip + Bob/Carol's network RTT). The pub/sub backplane adds only ~2ms to the delivery path.

Scale Numbers: How Much Load Does the Backplane Handle?

Consider a chat application with the following parameters:

  • 10,000 rooms (channels/conversations)
  • 50 members average per room (500K total connections across all rooms)
  • 1 message/second average per room
  • 5 WebSocket servers, each handling 100K connections

The backplane sees:

  • 10,000 PUBLISH commands/sec into Redis (1 per room per second)
  • ~50,000 message deliveries/sec from Redis to servers (each PUBLISH fans out to all subscribing servers; with 5 servers, each room has members on ~5 servers, so each PUBLISH produces ~5 deliveries, giving 10K × 5 = 50K)
  • 500,000 WebSocket writes/sec across all servers (each message fans out to 50 room members)

A single Redis instance comfortably handles ~1M pub/sub messages/sec, so 50K deliveries/sec uses only 5% of Redis capacity. The bottleneck at this scale is the WebSocket servers' fan-out (500K writes/sec across 5 servers = 100K writes/sec per server), not the Redis backplane.

Message Ordering Across Reconnections

TCP guarantees in-order delivery within a single connection, but when a connection drops and the client reconnects (possibly to a different server), that ordering guarantee vanishes. Messages sent during the disconnect window are lost, and the new server has no knowledge of what the old server already delivered. Here is a step-by-step trace of the problem and the solution:

The Problem: Lost Messages During Reconnection

StepEventClient StateServer State
1Client connected to Server A. Server A delivers messages with sequence numbers: msg#1, msg#2, msg#3.Last received: seq=3Server A: last sent to this client = seq=3
2Room receives msg#4 and msg#5. Server A delivers both to the client.Last received: seq=5Server A: last sent = seq=5
3Connection drops. Network failure or Server A crashes.Last received: seq=5 (stored locally)Server A is gone. Messages msg#6, msg#7 arrive in the room.
4Client reconnects to Server B (load balancer routes to a different server). Server B has no record of what this client has already received.Sends: "last_ack_seq": 5Server B receives reconnection with seq=5
5Server B queries the message store (Redis Streams / Kafka / database) for all messages in the room with seq > 5.Waiting for replay...Server B finds msg#6, msg#7
6Server B replays msg#6 and msg#7 to the client, then resumes live delivery.Last received: seq=7, caught upServer B: live delivery from seq=8 onward

Implementation Requirements

  • Sequence numbers: Every message in a room gets a monotonically increasing sequence number. The client tracks the highest sequence number it has received.
  • Durable message store: Messages must be persisted to a store that survives server crashes. Redis Streams (with XADD / XRANGE), Kafka (with consumer offsets), or a database table keyed by (room_id, seq) all work.
  • Reconnection protocol: On reconnection, the client sends its last_ack_seq. The server queries the message store for seq > last_ack_seq and replays them before switching to live delivery.
  • Retention window: You do not need to store messages forever. A retention window of 5-15 minutes covers most reconnection scenarios. Clients that reconnect after the retention window can fall back to fetching history via HTTP API.
import redis.asyncio as redis

r = redis.Redis()

async def on_reconnect(ws, room_id, last_ack_seq):
    """Replay missed messages after a reconnection."""
    # Fetch all messages in the room after the client's last-seen sequence
    # Redis Streams: XRANGE room:general +1 +
    missed = await r.xrange(
        f"room:{room_id}",
        min=f"{last_ack_seq + 1}",  # exclusive of last ack
        max="+",                     # up to the latest
    )

    # Replay each missed message
    for msg_id, fields in missed:
        await ws.send(fields[b"payload"].decode())

    # Now subscribe to live messages from the backplane
    # (client is caught up, resume normal delivery)

WebSockets + Authentication

WebSocket does not have a built-in authentication mechanism. You must validate the user's identity during the HTTP upgrade handshake—check a JWT or session cookie in the initial GET request headers. Once the connection is upgraded, there are no more HTTP headers. Do not rely on per-message authentication (it is too expensive). However, tokens expire: if a connection lives for hours, the token used at upgrade time may have expired. Solutions: (1) periodically re-validate tokens server-side, (2) require the client to send a re-auth message before the token expires, (3) set a maximum connection duration and force reconnection.

WebSockets + CDN

Most CDNs are designed for HTTP request-response caching and do not handle WebSocket well. Cloudflare and Fastly now support WebSocket proxying, but they cannot cache WebSocket traffic (it is dynamic by nature). Using a CDN for WebSocket adds another hop of latency and another intermediary that can timeout or fail. The main benefit is edge termination—terminating the TLS and WebSocket connection at a PoP close to the user, then maintaining a persistent backend connection to your origin. This reduces latency for the initial handshake but adds complexity.

Cost Considerations

Cost Analysis

Dominant cost driver: Memory. WebSocket servers are memory-bound for high-connection, low-message-rate workloads (chat, presence). For high-message-rate workloads (gaming, trading), CPU and bandwidth dominate.

Approximate Cloud Costs (AWS)

  • 10K connections: A single t3.medium (4GB RAM) is sufficient. ~$30/month.
  • 100K connections: r6g.large (16GB). ~$75/month. Add a Redis t3.small for pub/sub backplane: ~$15/month. Total: ~$90/month.
  • 1M connections: 2-3x r6g.2xlarge (64GB each) behind an ALB. ~$300/month per instance + ~$200/month for ALB + Redis: ~$1,100/month.
  • 10M connections: 20-30 r6g.2xlarge instances with Redis Cluster or Kafka for backplane. ~$6,000-$10,000/month for compute, ~$1,000/month for backplane. Total: ~$10,000/month.

Bandwidth

At 1M connections with 1 msg/sec average (200 bytes): ~200MB/sec = ~500TB/month. AWS data transfer: ~$0.09/GB for first 10TB, decreasing after that. Roughly ~$5,000-$10,000/month in bandwidth alone at scale. Compression (permessage-deflate) reduces this by 60-80% for text payloads.

Cost of NOT Using WebSockets

Polling a REST API every second for 1M users = 1M HTTP requests/sec. Each request has ~1KB of header overhead = 1GB/sec of unnecessary overhead. This costs 10-20x more in compute and bandwidth than WebSocket for the same real-time experience.

Code Example: Room-Based WebSocket Fan-Out

A minimal WebSocket server in Python using websockets that supports room-based messaging. Clients join a room and all messages are broadcast to room members.

import asyncio
import json
import websockets

# room_id -> set of connected websockets
rooms: dict[str, set] = {}

async def handler(ws):
    room_id = None
    try:
        async for raw in ws:
            msg = json.loads(raw)

            if msg["type"] == "join":
                room_id = msg["room"]
                rooms.setdefault(room_id, set()).add(ws)
                await ws.send(json.dumps({"type": "joined", "room": room_id}))

            elif msg["type"] == "message" and room_id:
                payload = json.dumps({
                    "type": "message",
                    "room": room_id,
                    "from": msg.get("from", "anon"),
                    "text": msg["text"],
                })
                # Fan-out to all room members except sender
                peers = [
                    peer.send(payload)
                    for peer in rooms.get(room_id, set())
                    if peer != ws
                ]
                await asyncio.gather(*peers, return_exceptions=True)
    finally:
        if room_id and room_id in rooms:
            rooms[room_id].discard(ws)
            if not rooms[room_id]:
                del rooms[room_id]

async def main():
    async with websockets.serve(handler, "0.0.0.0", 8765):
        print("WebSocket server running on ws://0.0.0.0:8765")
        await asyncio.Future()  # run forever

asyncio.run(main())

Chaos Testing

  • Kill a WebSocket server mid-operation: Verify that clients reconnect with jitter, re-join rooms, and resume receiving messages. Use toxiproxy to simulate network partitions between the WebSocket server and the pub/sub backplane.
  • Inject latency on the pub/sub backplane: Add 500ms latency to Redis with toxiproxy. Verify that messages still arrive (with delay) and that the system does not buffer unboundedly.
  • Simulate slow consumers: Create a test client that reads messages slowly (or not at all). Verify that the server detects backpressure and disconnects the slow client rather than running out of memory.
  • DNS failover: If using edge termination, simulate a PoP failure and verify that clients re-resolve DNS and connect to a different PoP.

Benchmarking Methodology

When benchmarking WebSocket performance, sweep these variables: connection count (1K, 10K, 100K), message size (64B, 1KB, 64KB), message rate per connection (1/sec, 10/sec, 100/sec), and fan-out ratio (1:1, 1:100, 1:10K). Common mistakes: (1) benchmarking with a single client process that becomes the bottleneck, (2) not measuring tail latency (p99.9), (3) not running long enough to expose memory leaks (run for at least 30 minutes), (4) using localhost which hides network latency.

Interview Questions

Design a real-time chat system for 10M concurrent users.

Start by separating concerns: an HTTP API layer for CRUD operations (create channel, fetch history) and a dedicated WebSocket gateway tier for real-time message delivery. Users connect via WebSocket to the gateway, which maintains an in-memory mapping of userId → connection and channelId → set of userIds.

At 10M connections (~30KB each), you need ~300GB of RAM. Spread this across 10-15 large-memory instances (r6g.4xlarge, 128GB each), each handling 700K-1M connections. Use consistent hashing on userId to route connections, so a given user always connects to the same gateway (simplifies presence tracking). An ALB with WebSocket support handles the initial routing.

For cross-server fan-out, use a sharded pub/sub backplane. Kafka with channel-based partitioning works well: partition by channelId so all messages for a channel go to the same partition, preserving order. Each gateway server consumes from all partitions and delivers to its local connections. For 1:1 messaging, route directly via the user's gateway (look up in a shared registry like Redis).

Message persistence: write to a Cassandra or DynamoDB table keyed by (channelId, timestamp). This is decoupled from real-time delivery—the API layer writes to the database and publishes to Kafka; the gateway tier consumes from Kafka and pushes to WebSocket.

Follow-ups: How do you handle message ordering across reconnections? How do you implement read receipts at this scale? What happens when a gateway server crashes—how do users recover their room subscriptions?

WebSocket vs. SSE — when to use which?

Use SSE when data flows only server-to-client: live scores, stock tickers (read-only), log tailing, notification feeds. SSE is built on standard HTTP, works through all proxies and CDNs without configuration, auto-reconnects with Last-Event-ID for resumption, and supports event types natively. It is simpler to deploy and debug because it is just HTTP with Content-Type: text/event-stream.

Use WebSocket when you need bidirectional communication (chat, collaborative editing, gaming), binary data support (audio/video streaming, file transfer), or the absolute lowest latency (WebSocket frame overhead is 2-14 bytes vs. SSE's text-based framing). WebSocket also supports multiplexing protocols on top (e.g., custom subprotocols like WAMP or STOMP).

The deciding factor is usually: does the client need to push data back to the server over the same connection? If yes, WebSocket. If no, SSE is almost always simpler and sufficient. A common mistake is choosing WebSocket for a server-push use case and then spending days configuring proxies and load balancers to support the upgrade handshake, when SSE would have worked out of the box.

Follow-ups: Can you use SSE with HTTP/2 to get multiplexing? What about HTTP/2 server push vs. SSE? How does browser connection limit (6 per domain) affect SSE?

How to scale WebSocket connections horizontally?

The core challenge is that WebSocket connections are stateful and server-affine. A message sent to "room X" must reach all members of room X, but those members are spread across multiple servers. Horizontal scaling requires solving two problems: (1) routing connections to servers, and (2) cross-server message delivery.

Connection routing: Use a load balancer that supports WebSocket (AWS ALB, HAProxy, Envoy). For simple cases, use sticky sessions (hash client ID to a server). For more control, use consistent hashing on a routing key (userId or roomId) so related connections land on the same server, reducing cross-server traffic.

Cross-server fan-out: Add a pub/sub backplane. Options in order of complexity: (1) Redis Pub/Sub—simplest, no persistence, ~1M msg/sec single node. (2) NATS—higher throughput, built for this use case, with JetStream for persistence. (3) Kafka—durable, ordered, partitioned, but higher latency (~5-10ms). The pattern: when server A receives a message for room X, it publishes to topic room:X on the backplane. All servers that have members of room X subscribe and deliver locally.

Optimization: Maintain a room → set of servers mapping (in Redis or a shared registry). Only publish to the backplane if the room spans multiple servers. For rooms where all members happen to be on the same server, deliver locally without touching the backplane. This optimization matters at scale—most rooms are small and often local to one server.

Follow-ups: How do you handle a server crash without losing message state? How do you rebalance connections when adding a new server? What is the latency overhead of the pub/sub backplane?

How to handle authentication with WebSockets?

WebSocket authentication happens in three layers. Layer 1 (Upgrade handshake): Validate the user during the HTTP upgrade request. The browser sends cookies automatically with the upgrade request, so you can validate a session cookie or JWT in a query parameter (wss://host/ws?token=...). Reject unauthorized upgrades with HTTP 401 before the connection is established. This is the most important layer—it prevents unauthenticated connections from consuming server resources.

Layer 2 (Connection lifetime): Tokens expire, but connections can live for hours. Implement a periodic re-validation: the server checks the token's expiry and sends a "re-auth required" message before it expires. The client sends a fresh token over the WebSocket. If the client fails to re-authenticate within a grace period, the server closes the connection with a 4001 (custom close code).

Layer 3 (Per-message authorization): Even with an authenticated connection, you may need to authorize specific actions (e.g., "can this user send to this channel?"). Do this check when the user joins a room (authorize once, not per-message) and maintain an in-memory ACL. Re-check if permissions change (listen for permission-change events from your auth system).

Common mistake: Putting the JWT in a query parameter means it appears in server logs, proxy logs, and the browser history. Use a short-lived, single-use token for the upgrade, then exchange it for a session on the server side. Never put long-lived secrets in URLs.

Follow-ups: How do you revoke a user's access in real time (e.g., they are banned from a channel while connected)? How do you handle token refresh across reconnections?

Design a live collaborative document editor.

The architecture has three main components: a WebSocket server for real-time operation delivery, a conflict resolution engine, and a persistence layer for document state.

Conflict resolution: Use either Operational Transformation (OT) (as in Google Docs) or CRDTs (as in Figma and Yjs). OT requires a central server to transform concurrent operations against each other; CRDTs can resolve conflicts without a central authority. OT is simpler to implement for text editing but requires careful server-side logic. CRDTs (e.g., Yjs, Automerge) are more complex but enable peer-to-peer collaboration and offline editing.

Data flow: User types a character → local editor generates an operation (insert "a" at position 5) → operation is applied locally immediately (optimistic) → operation is sent via WebSocket to the server → server transforms the operation against any concurrent operations → server broadcasts the transformed operation to all other editors via WebSocket → each client applies the transformed operation to its local state.

Persistence: Store the document as a log of operations (event sourcing). Periodically snapshot the document state for fast loading. New users joining a document receive the latest snapshot plus any subsequent operations. Use a separate storage system (S3 for snapshots, DynamoDB for operation logs) rather than trying to store everything in the WebSocket server's memory.

Scaling: Partition by document—all editors of a single document connect to the same server (or server group). Use consistent hashing on documentId for routing. Most documents have fewer than 10 concurrent editors, so a single server can handle thousands of documents. For very popular documents, the server becomes CPU-bound on OT/CRDT computation.

Follow-ups: How do you handle cursor positions and selections? How do you deal with a user who goes offline and comes back with a large set of changes? How do you show presence (who is viewing which part of the document)?

What happens when a WebSocket server crashes?

When a WebSocket server crashes, every client connected to it loses their connection simultaneously. Here is the sequence and how to handle each phase:

Detection (0-5 seconds): Clients detect the connection loss via the onclose or onerror event. The load balancer's health check detects the server is down (typically within 10-30 seconds, depending on configuration). During this window, the load balancer may still route new connections to the dead server—configure health checks with short intervals (5s) and low thresholds (2 failures).

Reconnection (5-60 seconds): Clients reconnect with jittered exponential backoff to avoid a thundering herd. They connect to a different server (the load balancer routes them to surviving servers). The new server does not know the client's previous state—room subscriptions, undelivered messages, cursor position. The client must re-send its subscriptions (re-join rooms) after reconnecting.

State recovery: Messages sent while the client was disconnected are lost unless you have a message replay mechanism. Solutions: (1) Each message has a sequence number. On reconnect, the client sends its last-seen sequence number, and the server replays missed messages from a durable store (Redis Streams, Kafka). (2) Use CRDT-based state synchronization—the client sends its state vector, the server computes the delta. (3) For simpler systems, just accept that messages during the disconnect window are lost and rely on the client fetching history via HTTP.

Prevention: Use rolling deploys (drain connections before shutting down). Implement graceful shutdown: the server sends a Close frame with code 1001 (Going Away) to all connections, giving clients time to reconnect proactively. Store critical state (room memberships) externally (Redis) rather than only in server memory, so a new server can reconstruct it on reconnection.

Follow-ups: How do you drain connections during a deploy without dropping messages? How do you test this recovery path? What is the impact on presence tracking (users briefly appear offline)?

Key Takeaways

  • WebSockets provide full-duplex, persistent connections with minimal per-message overhead (2-14 bytes). Use them when you need real-time, bidirectional communication.
  • The HTTP upgrade handshake is how WebSocket connections begin. Every proxy, load balancer, and firewall in the path must understand and pass it through.
  • SSE is simpler when data flows one way. Do not reach for WebSocket when SSE or even polling will suffice. The operational complexity of persistent connections is real.
  • Memory is the dominant cost driver for high-connection workloads. Budget ~30KB per connection. At 1M connections, that is 30GB of RAM.
  • Scaling requires a pub/sub backplane (Redis Pub/Sub, NATS, Kafka) for cross-server fan-out. Without it, users on different servers cannot communicate in the same room.
  • Reconnection storms are the #1 operational risk. Always use jittered exponential backoff in clients. Use rolling deploys and graceful shutdown on the server.
  • Ping/pong heartbeats are not optional. They detect dead connections, prevent memory leaks from zombie connections, and keep connections alive through proxy timeouts.
  • Authenticate during the upgrade handshake, not per-message. But plan for token expiry over long-lived connections.
  • Modern alternatives like WebTransport (QUIC-based) and gRPC streaming are worth considering for specific use cases, but WebSocket remains the most widely supported real-time protocol.
