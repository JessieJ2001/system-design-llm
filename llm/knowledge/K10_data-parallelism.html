K10 — Data Parallelism (serving-replica DP — and why it’s not training DP)

How do you build a scalable throughput tier — replicate the whole engine and shard requests across replicas — when the only real design question is the router, not the sync (there is none)? This page derives why serving DP is embarrassingly parallel (unlike training DP) and why its single hard problem is stateful-KV affinity across replicas.

SLIs This Page Reduces

SLIs this page reduces (the reverse index of K1 §3.12):
  • Throughput / Goodput — replicating the engine N× scales aggregate request rate exactly linearly (no sync, no all-reduce).
  • Cache-hit rate — D39: a cache-aware cross-replica router keeps multi-turn / reusing requests on the replica that holds their KV (else the K2-Role-2 win is lost at the DP layer).
  • Resilience (smallest blast radius) — one replica = the smallest fault domain ⇒ DP is the resilience-friendliest axis (K22).

The DP Spine (stated up front, before §1)

This page is a child of K6 and resolves the fits-but-throughput-bound binding of K6’s grid. The chain:

  • Throughput-bound (model fits one replica, aggregate rate too low) → replicate the whole engine N× = DP.
  • Shard requests across replicas — no gradients, no all-reduce, no sync ⇒ cost is exactly linear.
  • New problem: each replica has its own independent KV cache.
  • ⇒ a multi-turn / cache-reusing request must hit the same replica — affinity-vs-even-load tension at the router (D39).
  • ⇒ DP degree = the autoscaling unit (K15); DP alone is insufficient when the model doesn’t fit one replica (→ TP/PP first, back to K6).

Serving DP ≠ training DP (derived — the most common interview confusion)

Problem→Solution stub (the page’s opening derivation, sub-bulleted). Problem: “data parallelism” evokes training’s gradient all-reduce. Solution, stated explicitly: (a) training DP — each replica processes a different micro-batch then does an all-reduce of gradients every step → synchronised, comm-bound; (b) serving DP — each replica serves different requests, no gradients, no all-reduce, no sync ⇒ embarrassingly parallel; the only “shared state” is none (each replica’s KV is private); (c) ⇒ cost is exactly linear in replicas and the sole design question is the router, not the sync; (d) DP is composed outside TP/PP/EP; fault domain = one replica (smallest ⇒ the resilience-friendliest axis, K22); DP alone is insufficient when the model doesn’t fit one replica (→ TP/PP first, K6).number: linear cost ($/replica × replicas) vs training-DP’s all-reduce overhead, on the trace — cites K1 (#cost), K22 (replica = smallest fault domain), K6 (composition)

Deep dive D39 — cross-replica routing & the stateful-KV affinity problem

Deep-dive stub (the page’s one real design question). Problem: replicas have independent KV; a cache-oblivious LB (round-robin) sends turn-2 to a replica that doesn’t hold turn-1’s KV → full recompute → the K2-Role-2 reuse win is lost at the DP layer, invisibly. Solution: the cross-replica router is first-class — session-affinity / consistent-hash(conv-id) / KV-aware, i.e. K2’s (0)–(7) routing spectrum re-applied at the replica granularity (disambiguate: K2 owns within a replica, K10 owns which replica; same spectrum, two scopes). Crossover: affinity hit-value (saved prefill GPU-s) vs replica load imbalance, capped by bounded-load consistent hashing.number: saved prefill GPU-s from a hit vs the imbalance cost on the trace — cites K2 (Role-2 spectrum — cite, don’t re-derive), K1 (#prefill cost) · real: external L7 LB + consistent hash, SGLang router at the replica tier, Ray Serve session affinity + URLs

Rule 11 — the cross-replica routing policy

Selection-sweep stub. round-robin → least-loaded → session-sticky → consistent-hash(conv-id) → KV-aware: this is K2’s spectrum at replica scope — cite K2, state the granularity disambiguation, do not re-derive the spectrum here. 4. Alternatives & When Each Wins

Stub (≥2, disqualifying constraint per alt). (a) More TP/PP instead of DP — disq: adds comm/bubble for no throughput benefit once the model already fits one replica; wins only when the model does not fit one replica (→ K6/K7/K8). (b) A shared global KV pool across replicas — disq: adds a network hop per access + a consistency/scaling problem; wins only when cross-replica reuse value exceeds the hop (cite K2/K5). (c) Cache-oblivious round-robin — disq: ~0 cross-replica hit → silently loses the Role-2 win (D39); wins only for no-shared-prefix identical short requests. Each: when it actually wins.number: the constraint disqualifying each (single-replica fit for (a); hop cost for (b); hit-rate for (c)) — cites K6/K7/K8, K2/K5 · contrast L9 (cross-replica affinity D39 is its spine)
