K13 — Speculative Decoding (break the one-token barrier, losslessly)

How do you build a scalable, lossless decode accelerator — when generating one token costs a full HBM weight-load (arithmetic intensity ≈ 1, the worst roofline) — by verifying many drafted tokens in a single target forward without changing the model’s output distribution, and turning it off the moment the batch makes decode compute-bound? This page derives the rejection-sampling losslessness proof and the acceptance-rate speedup, then D11 (why it helps at batch 4 and hurts at batch 256).

SLIs This Page Reduces

SLIs this page reduces (the reverse index of K1 §3.12):
  • TPOT / ITL (primary) — a memory-bound target forward commits up to E≈1/(1−α) accepted tokens instead of 1 ⇒ fewer weight-load-bound steps per generated token.
  • Throughput (tokens/s, small batch) — the same single weight-load yields multiple tokens while decode is memory-bound.
  • Goodput — only while decode is memory-bound; explicitly does NOT help — and net-hurts — once the batch makes decode compute-bound (the D11 caveat, stated up front so the SLI claim is honest).

The Speculation Spine (stated up front, before §1)

This page is a consumer of K1’s memory-bound decode and K4’s batching. The chain:

  • Decode reads all model weights from HBM to produce one token — arithmetic intensity ≈ 1, the worst roofline (K1).
  • → if several candidate tokens could be verified in one weight-load, the read amortizes.
  • → a cheap draft proposes K tokens; the target verifies all K+1 in one batched forward (same HBM cost as one decode step).
  • New problem: naively taking the draft’s tokens biases the output to the draft’s distribution — lossy.
  • → rejection sampling: accept the longest correct prefix, resample on first mismatch ⇒ output distribution is exactly the target’s (lossless, proved here).
  • Speedup is governed by acceptance α: E≈1/(1−α); α is draft-quality- and workload-dependent → the draft-mechanism spectrum (rule 11).
  • At large batch decode is already compute-bound ⇒ the extra draft+verify FLOPs are pure overhead ⇒ speculation hurts (D11) ⇒ speculate iff batch < B_spec.

1. The Problem It Solves

Problem→Solution stub. Problem: autoregressive decode emits exactly one token per forward, and each forward must read the entire weight set from HBM (arithmetic intensity ≈ 1) — so single-stream / small-batch decode is gated by HBM bandwidth, not FLOPs, and TPOT is bounded below by weights_bytes / HBM_BW no matter how much spare compute the GPU has (cite K1 #decode/#roofline — do not re-derive). Solution (what the page builds): commit several tokens per weight-load by drafting candidates cheaply and verifying them in a single target forward — losslessly, so the model’s output is unchanged.number: TPOT floor = weights_bytes/HBM_BW on the trace (no-speculation baseline) — cites K1 #decode (memory-bound step), #roofline (arithmetic intensity ≈ 1) · canonical: llm-bottlenecks.html (memory-bandwidth-bound decode)

The acceptance math + the losslessness proof (derived — the page’s home, self-contained)

Problem→Solution stub (the page’s opening derivation, sub-bulleted). Problem: if you just keep the draft’s tokens, the output follows the draft’s distribution q, not the target’s p — that is approximate, not lossless. Solution, derived: (a) rejection-sampling correction — for draft token ti with draft prob q(ti) and target prob p(ti), accept with prob min(1, p/q); on reject, sample from the normalised residual (p−q)+ and stop; (b) proof the committed distribution is exactly p — the accept/residual split telescopes to p at every position (this is why it is lossless, not “approximately”); (c) expected accepted tokens/step E=(1−αK+1)/(1−α) ≈ 1/(1−α) for large K, where α = expected per-token acceptance rate; (d) wall-clock speedup ≈ E, capped by draft cost cdraft·K + the K+1-wide verify-batch overhead — derive both directions. Deep dive D11 — speculation under continuous batching (why batch 4 helps, batch 256 hurts; co-host K4)

Deep-dive stub. Problem: the speedup is amortising the weight-load over K draft tokens — that headroom only exists while decode is memory-bound (small batch). At large batch the weight-load is already amortised over B sequences ⇒ decode is compute-bound ⇒ the extra draft+verify FLOPs are pure overhead ⇒ speculation slows it down. Solution: batch-adaptive speculation (shrink K / disable past a batch threshold), tree-verification batching (verify multiple draft branches in one forward), online α monitoring (drop the draft if α decays under distribution shift). Crossover: speculate iff batch < B_spec, where B_spec = the batch at which the speculative path crosses memory→compute-bound (cite K1 ridge B*). Rule of two: K4 owns the scheduler / per-step accepted-token batch budget; K13 owns the speculation mechanism and this crossover.number: B=4 (memory-bound, α=0.7 → ≈3.3× speedup) vs B=256 (compute-bound, ≈ −15%) on the trace — cites K1 #roofline (ridge B*), K4 #d11 (batch-budget side) · real: vLLM spec-decode, TensorRT-LLM, EAGLE-2 + URLs

4. Alternatives & When Each Wins

Stub (≥2, disqualifying constraint per alt). (a) Just raise the batch (the natural throughput lever) — disq: caps at the memory→compute ridge B* and you cannot always get the concurrency (low-traffic / latency-bound single stream); wins at high concurrency where the batch already saturates compute — and there speculation is correctly off by D11 (the L1 contrast). (b) Smaller / distilled target model — disq: changes the output distribution (lossy); wins when the quality budget tolerates it; composes with, does not replace, lossless speculation. (c) Quantization (K14) — disq: lossy, needs an accuracy gate; orthogonal — cuts the per-step byte bill where speculation cuts the step count; wins as a complement, not a substitute. (d) Non-lossless “approximate” speculation (accept-on-threshold) — disq: changes the output distribution; out of scope — this page is the lossless mechanism; flag the boundary, do not cover.number: the crossover disqualifying each (B* for (a); quality delta for (b); orthogonal-axis for (c)) — cites K1 #roofline, K14 (quant, not yet authored) · contrast L1 (huge batch ⇒ speculation off)

5. Failure Modes

Stub (cite, never inline-redefine). α collapse under distribution shift — OOD traffic (code vs prose) → α→low, E→1, draft+verify FLOPs wasted → net slowdown; first move: online α monitor → disable the draft (llm-bottlenecks.html). Speculation left on at large batch — the D11 cliff: throughput regression while the GPU “looks busy” (llm-bottlenecks.html: compute-bound decode + speculation overhead).
