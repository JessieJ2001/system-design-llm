K6 — Parallelism Overview (the umbrella + selection home)

How do you build a scalable multi-GPU serving deployment — which parallelism, at what degree, composed how, when the model (or its KV/activations) doesn’t fit, or fits but is too slow, on the GPUs you have? This page does not implement an axis — it is the umbrella + selection home. The answer is a constrained search (D25) that first tries the no-parallelism escape (D36) and is bounded throughout by the NVLink-domain size (K20) — that fact, not FLOPs, decides the tuple.

SLIs This Page Reduces

SLIs this page reduces (the reverse index of K1 §3.12):
  • Feasibility (does it fit at all) — the selection procedure assembles the minimal tuple whose per-GPU weight + KV/activation footprint fits HBM.
  • TTFT / TPOT — TP cuts per-token latency; the procedure picks the smallest degree that meets the SLO without crossing the cliff that raises it (over-sharded comm).
  • Throughput / Goodput — DP is chosen to exactly meet the aggregate-rate target; the whole tuple is optimised for goodput-per-$ under the SLO, not raw fit.
  • $ / GPU-hours — D36 refuses to shard when quant/offload would fit on one GPU; D25 never adds an axis the binding doesn’t demand — the minimal tuple is the cheapest one that holds the SLO.

The Spine — the Selection Chain (stated up front, before §1)

The umbrella question resolves as a chain: does the model fit on one GPU? → if no, what binds? → the axis that resolves that binding → the new bottleneck it introduces → the next axis — all under the interconnect-topology constraint that bounds every one of them.

  • Weights don’t fit → TP (split within layers) → new problem: a per-layer all-reduce on the critical path → caps at the NVLink-domain boundary (K7) — and a second, often-tighter cap: TP shards KV only down to n_kv_heads (GQA); past that KV is replicated, not split, so KV-memory stops dropping while the all-reduce keeps growing.
  • Still doesn’t fit past one NVLink domain → PP (split layers across domains) → new problem: a pipeline bubble (K8).
  • MoE: huge total params, tiny active → EP (spread experts) → new problem: two all-to-alls per layer (K9).
  • Fits but throughput-bound → DP (replicate) → new problem: linear GPU cost, a private per-replica KV cache (K10).
  • One sequence exceeds all of the above (1M-token context) → CP (split the sequence) → new problem: ring-attention bandwidth (K11).
  • Master constraint over all of it: the NVLink-domain size — and the budget is joint, not per-axis: TP’s all-reduce, EP’s all-to-all and CP’s ring all ride the same NVLink fabric in the same forward pass, so the cap is on their sum, not on each independently. Everything crossing the domain pays the IB tax (K20). This fact, not FLOPs, decides the tuple.

And before the chain even starts: the no-parallelism escape (D36) — quantize-to-fit (K14), KV-offload/tier (K2), or a smaller/distilled model — is tried first; you only enter the chain when its accuracy/latency cost exceeds the multi-GPU cost. The chosen precision is then fixed (it re-parametrises every memory threshold below — it is a search parameter, not only the escape). Co-fixed with precision: the GPU class. A real fleet is heterogeneous (H100 + H200 + spot mix); a parallelism group cannot span GPU classes (the slow class straggles the collective), so D25 also selects which class to run on, bounded by that class’s contiguous-domain inventory — not just “the GPU type” as a single given.

Three framings that bracket the whole chain:

  • It runs per serving phase. Prefill (compute-bound, latency-driven) and decode (memory-bandwidth-bound, throughput-driven) have opposite rooflines ⇒ the binding evaluation and D25 run once per phase; colocated ⇒ one compromise tuple, disaggregated (K5) ⇒ a different tuple per pool. K6 selects per phase; K5 owns the split. But divergence is not free: if prefill-TP ≠ decode-TP the KV is sharded differently on each side, so the K5 handoff is a reshard/transpose, not a copy — that tax feeds back into whether a per-phase-different tuple beats one compromise tuple.
  • D25 is a fixed-point, not a one-pass pipeline. DP = target ÷ per-replica throughput, but per-replica throughput is set by the max KV-resident batch, which depends on the KV budget the TP/PP/EP/CP choices leave — so you iterate; “max TP to N_dom then PP” is the heuristic seed, not the optimum.
  • The objective carries tie-breaks. Goodput per $ under the SLO, then — at equal goodput-per-$ — the smaller fault blast radius (K22) and the tuple that actually gang-packs/autoscales on the cluster (K15).
  • Only one axis is elastic. TP/PP/EP/CP are deploy-time-rigid — resharding a live TP/PP/EP/CP group is a redeploy/fault event (K22/K21), not an autoscale. DP is the only axis that scales with load. So D25 is not one fixed-point over a static rate: the rigid inner tuple (TP×PP×EP×CP) is sized for the worst-case mix and held; DP is the elastic outer loop that absorbs diurnal/burst variance (K10/K15).

The Page as One Decision Tree

The arc: does it fit & run fast on ONE GPU? → D36 escape first (quantize / KV-offload / smaller) → if still no, fix the precision and GPU class then ask — per serving phase (prefill vs decode, K5; differing per-phase tuples cost a KV-reshard on the handoff) — what binds? → the axis per binding (TP/PP/EP/DP/CP), each introducing its own next bottleneck → all bounded by the NVLink domain (the master constraint — a joint budget over the co-resident collectives) → D25 assembles the minimal tuple as a fixed-point (the rigid TP×PP×EP×CP inner tuple sized for the worst-case mix; DP the elastic outer loop — not a one-pass pipeline) → objective = goodput per $ under the SLO, tie-broken by blast radius (K22) and gang-schedulability (K15). Each box is a section below; the body elaborates them in this order.

  • ① Root → one-GPU question — before any sharding, ask whether the model fits & runs fast on one GPU.
  • ② D36 escape, tried first — quantize-to-fit (K14) / KV-offload (K2) / smaller model; if it now fits, you are done (L5 lives here) and never pay inter-GPU comm. And even if it doesn’t fit on one GPU, quant is still a parameter inside D25 — it shifts every memory threshold, so fix the precision before running the search.
  • ③ Problem: what binds — asked PER serving phase — prefill (compute-bound, latency-driven) and decode (memory-bound, throughput-driven) have opposite rooflines, so the procedure runs once per phase (colocated or disaggregated, K5) ⇒ a tuple per phase, not one tuple. The binding (weight mem / KV-act mem / latency / throughput / one-sequence-too-big) then selects the axis.
  • ④ The five axes — TP / PP / EP / DP / CP, each resolving one binding and introducing its own next bottleneck (→ K7–K11). Note TP also shards KV & activation /TP — a KV/act-memory binding can be met by more TP within N_dom before reaching for PP/CP.
  • ⑤ The master constraint — the NVLink-domain size (K20) bounds every axis; TP inside it, crossing pays the IB tax — this decides the tuple, not FLOPs.
  • ⑥ D25 assembles the tuple as a fixed-point — the constrained search produces DP×TP×PP×EP×CP, but DP depends on the per-replica throughput, which depends on the KV budget the other axes leave: iterate. “Max TP to N_dom then PP” is the heuristic seed, not the optimum — the TP/PP boundary is itself a crossover at your batch size.
  • ↻ The feedback loop — if the assembled tuple misses $/goodput, doesn’t fit the target batch, or won’t pack, raise TP (more KV headroom) or quantize and re-run; this is why the search is a fixed-point, not a one-pass pipeline.
  • ⑦ The objective — goodput per $ under the SLO, tie-broken by a smaller fault blast radius (K22) and gang-schedulability/packability on the real cluster (K15/D43): prefer the smaller tuple at equal goodput, not raw fit.
  • ⑧ The hard caps the boxes hide — (i) GQA caps effective TP at n_kv_heads: past it KV is replicated not split, so KV-memory stops dropping while comm rises (a tighter cap than N_dom). (ii) The N_dom budget is joint: TP×EP×CP collectives contend for the same fabric in one forward pass. (iii) A parallelism group cannot span GPU classes — selection picks the class too, bounded by per-class domain inventory.
  • ⑨ Rigid inner, elastic outer — TP/PP/EP/CP are deploy-rigid (reshard = a K21/K22 event); only DP autoscales. Size the rigid tuple for the worst-case mix; let DP absorb variance. A per-phase-different tuple additionally costs a KV-reshard on the K5 handoff — weigh that against one compromise tuple.

2. The Mechanism — the Tokenized-Prompt Walkthrough

Where parallelism lives — a deployment topology over the K0 forward pass, not a model change

Problem→Solution stub (architecture placement, hard req 6). Problem: the K0 forward pass is identical no matter how many GPUs run it — nothing in the math says “split here.” Solution: each axis is a different cut of the same forward pass — TP splits the within-block matmuls (the residual add stays GPU-local); PP cuts between blocks (the residual-stream legal-cut, cite K0/K18); DP replicates the whole graph; EP splits the MoE experts; CP splits the sequence. Same model, a GPU-topology choice the selection procedure governs — not a parallelism axis built into the model.cites K0/K18 (the legal cut points: PP block-boundary, TP within-block, residual add local) · no number (placement fact) · diagram: one K0 block annotated with where each of the five axes legally cuts

The walkthrough on "I like cat, why?" — what each axis does to these 6 tokens

Problem→Solution stub (hard requirement 6 — reader’s first contact). Canonical trace T1 = "I like cat, why?" (6 tokens). TP: every GPU holds a column/row slice of every weight matrix; one all-reduce after attention out-proj and after the MLP down-proj recombines the slices for all 6 tokens, every layer. PP: GPU 0 runs blocks 0..k over all 6 tokens, then ships the residual-stream activation to GPU 1 for blocks k+1.. DP: replica A serves this whole request; replica B serves a different one (no interaction). EP: the router sends each of the 6 tokens to its top-k experts, which may sit on different GPUs (the two all-to-alls). CP: the 6 token positions are split across GPUs; each ring-passes its KV so every query still attends every key. Deep dive D25 — parallelism-tuple selection as a constrained search (the spine)

Deep-dive stub (the page’s central question; the answer is a fixed procedure run per phase, not a guess — and a fixed-point, not a one-pass pipeline). Given (model size, chosen precision, TTFT/TPOT target, GPU type, NVLink-domain size N_dom, target batch/arrival rate), derive the tuple by this order, each step a sub-stub:

  • (0) Precondition — fix the precision and the GPU class first. Quant is not only D36’s “fits-on-1-GPU” escape; it re-parametrises every memory threshold in (1)–(5). FP8 vs FP16 can cut the minimum TP 8→4 and change the whole downstream tuple — so precision↔tuple are co-selected, and the search runs at the chosen precision. On a heterogeneous fleet the GPU class is co-fixed here: HBM/BW differ per class so every threshold below is class-relative, and a parallelism group cannot span classes (the slow class straggles every collective) — so D25 also picks the class, bounded by that class’s contiguous-domain inventory.number: minimum TP for the frontier-MoE weights at FP16 vs FP8 on the trace (it halves) — cites K14 (the cascade), K1 #memory · composes with D36
  • (1) TP = the smallest degree that fits weights and per-GPU KV/activation (TP shards both ≈ /TP), capped at N_dom — and, for the KV term, at n_kv_heads: with GQA, TP>n_kv_heads replicates KV instead of splitting it, so KV-memory plateaus while the all-reduce keeps growing (the binding may then force PP/CP even though raw weight-fit said “more TP”). Beyond N_dom the per-layer all-reduce crosses IB → latency collapse (cite K7/D26). “Max TP to N_dom then PP” is the heuristic seed: at the target batch the all-reduce cost can exceed the PP bubble before N_dom, so the TP/PP boundary is itself a crossover (cite K7/D37, K8 bubble).number: TP for the frontier-MoE weights+KV on the trace vs N_dom; the batch at which per-layer all-reduce > PP bubble — cites K7, K8, K1 #memory #collectives, K20 (N_dom)
  • (2) PP = if still over, ⌈layer-memory / domain-memory⌉ across domains.number: PP stages = model bytes / per-domain budget on the trace — cites K8, K20
  • (3) EP for MoE = the largest degree keeping the all-to-all inside N_dom (cite K9).number: EP that keeps dispatch/combine intra-domain on the trace — cites K9, K20
  • (4) DP = throughput target ÷ per-replica throughput (the outer, linear axis) — but per-replica throughput is not an input: it is set by the max KV-resident batch = (HBM − weight_shard − activation) / KV_per_seq, which depends on the TP/PP/EP/CP chosen in (1)–(3),(5). Hence the procedure is a fixed-point, not a one-pass pipeline. DP is also the only elastic axis: (1)–(3),(5) are deploy-rigid (resharding a live group is a K21/K22 event), so size them for the worst-case mix and let DP be the outer loop that autoscales with the live rate (cite K10, K15) — the rigid tuple is chosen once, DP tracks the curve.number: per-replica goodput as a function of the KV budget the tuple leaves, on the trace — cites K10, K2 (#kv-budget → max batch), K1 #throughput #decode
  • (5) CP only if one sequence’s KV + activation exceeds the TP×PP per-GPU budget.number: the context length at which CP becomes mandatory on the trace — cites K11, K18 (activation footprint)
  • (6) Iterate (the feedback loop). If the assembled tuple misses goodput/$ , the target batch doesn’t fit, or it won’t gang-pack on the cluster: raise TP (buys KV headroom → bigger batch → more per-replica goodput → fewer DP replicas), or drop precision (back to (0)), and re-run. When the per-phase tuples differ (prefill-TP ≠ decode-TP), charge the KV-reshard/transpose on the K5 handoff (not a copy) against the divergence — a per-phase split only wins when its goodput gain exceeds that reshard tax plus K5’s transport cost. 4. Alternatives & When Each Wins

Stub (≥2, disqualifying constraint per alt). (a) Single axis maxed (pure TP) — disq: TP caps at N_dom; beyond it the per-layer all-reduce crosses IB and latency collapses (cross-link K7/D26); wins only when the model fits within one domain at the needed TP. (b) Pure DP (replicate only) — disq: fails the moment the model doesn’t fit one replica; wins when it fits and you are purely throughput-bound (the resilience-friendliest axis, K10/K22). (c) No-parallelism + aggressive quant (D36) — disq: when quant accuracy loss > SLO floor or single-GPU latency > SLO; wins for small / cost-optimised fleets (L5). (d) Training-style tuple (ZeRO/FSDP) — disq: optimises gradient/optimiser-state sharding, which serving has none of (stateless, no backward) — structurally the wrong tool. (e) One tuple for both phases (no per-phase split) — disq: when prefill and decode SLOs are both tight, a single compromise tuple misses one (the prefill-optimal degree starves decode goodput and vice-versa); wins only for small / loose-SLO single-pool deployments where the disaggregation overhead (K5/D35) exceeds the compromise cost. (f) One-pass D25 (no fixed-point iteration) — disq: ignores that DP depends on the KV budget the other axes leave → over-provisions DP or under-uses HBM; never strictly wins, only “close enough” when the model is tiny relative to HBM. (g) Push TP past n_kv_heads for the KV win — disq: with GQA, TP>n_kv_heads replicates KV so the memory win stops while the all-reduce cost keeps rising; never wins for the KV binding (use PP/CP instead), only when the binding is pure weight-fit and latency has headroom. (h) Size the rigid tuple for the mean mix (no elastic-DP headroom) — disq: TP/PP/EP/CP can’t reshard on the diurnal timescale, so a mean-sized rigid tuple SLO-violates at the peak; wins only when the mix is near-constant (Var(ρ) ≈ 0). Each: when it actually wins.number: the constraint value that disqualifies each (N_dom for (a); single-replica fit for (b); SLO floor for (c); the TTFT/TPOT pair that breaks the compromise tuple for (e)) — cites K7/D26, K10, K14, K5/D35 · contrast scenario L1 (the worked tuple, disaggregated) vs L5 (the D36 branch)

  • K0 / K18 — the residual stream → legal cut points (PP on block boundaries, TP within a block).
