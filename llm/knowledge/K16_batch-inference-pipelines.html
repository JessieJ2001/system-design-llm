K16 — Batch-Inference Pipelines (the inverted roofline: throughput is the only goal)

How do you build a scalable offline/bulk generation pipeline (eval, labeling, synthetic data, embedding backfill) — where there is no latency SLO and GPU throughput is the only goal — by inverting every online assumption (push the batch far past B* so decode is compute-bound at peak MFU), running it on the cheapest capacity, and surviving multi-hour preemptible jobs? This page derives the inverted roofline, then D29 (checkpoint/straggler/spot resilience) and D44 (batch on the online fleet’s idle troughs — the economic core).

SLIs This Page Reduces

SLIs this page reduces (the reverse index of K1 §3.12):
  • Throughput (tokens/s) (primary) — remove the latency SLO ⇒ push the batch far past B* ⇒ N× the online tokens/s on the same GPU.
  • MFU — past the compute-bound ridge decode runs at peak MFU (the inverted roofline).
  • Cost-shape ($/job) — run on the cheapest capacity (spot, or the online fleet’s idle troughs at ≈ marginal cost — D44).
  • Explicitly NOT per-request TTFT / TPOT — there is no per-request latency SLO; stating this is the whole point of the page (the inversion).
  • Makespan (job completion time) (constraint, not free) — the per-request SLO is replaced by a job-level deadline; it bounds scale-out degree and the capacity tier (D44), it is not infinite.
  • Determinism (for eval jobs) — batch-size-dependent FP nondeterminism (K17/D10) corrupts “did the model improve or did the batch change?” — an eval-batch gate alongside accuracy.

What This Page Is (and Is Not)

TL;DR. With no per-request latency SLO (a job makespan deadline replaces it) every online assumption inverts: push the batch until HBM-limited (decode becomes compute-bound, MFU → peak), exploit what online can’t — length-sort the whole dataset and reuse the one shared prefix — quantize aggressively (only the offline accuracy/determinism gate binds), run prefill-heavy; then run it on the cheapest capacity and the cheapest GPU type — spot, or the online fleet’s idle troughs at lowest priority — and survive the multi-hour preemptible job with checkpoint/resume.

  • Is: the inverted roofline derived from K1’s B*/MFU; D29 (offline-batch resilience — checkpoint/resume, straggler mitigation, spot-interruption mid-batch); D44 (batch on the online fleet’s idle troughs / preemptible co-tenancy — the economic core: consume only otherwise-wasted GPU-hours); the operating-point rule-11 spectrum.
  • Is not: the online latency-SLO scheduler (K4 — K16 is the no-SLO inversion of the same engine); the DAG orchestration engine (Airflow / Dagster / Flyte — data-eng; K16 owns each GPU stage’s operating point and the fuse-vs-materialize composition, the workflow scheduler that runs the DAG is out) and non-GPU stages (CPU dedup/filter/SQL — data-eng); training-data generation as training (training-side — serving the generation is here, the training that consumes it is not).

The Batch-Inference Spine (stated up front, before §1)

  • Offline/bulk generation where throughput is the goal and there is no per-request latency SLO — but there is a job-level makespan deadline (the dataset must be done before the next training run / the 6 am report). The deadline, not “infinite patience”, is the real time constraint (eval, labeling, synthetic data, embedding backfill).
  • → invert every online assumption: max batch & MFU, no per-request latency, aggressive quant, prefill-heavy.
  • The workload shape sets the operating point (not “remove the SLO” alone): prefill-heavy jobs (classification / extraction / scoring / LLM-as-judge — long in, short out) are already compute-bound at B=1; decode-heavy jobs (synthetic-data / long-form gen) stay memory-bound until B is pushed up. What binds decides what “max batch” buys.
  • And there isn’t one stage — batch jobs are often GPU DAGs (embed→generate→judge, encode(ViT)→generate), each stage a different roofline → run each at its own operating point / precision / GPU type, then decide fuse (overlapped, with per-stage fleets sized to balance throughput — min-stage is the bottleneck) vs materialize (independent, separately-tuned, separately-resumable per-stage jobs; the materialised intermediate is a free D29 stage-boundary checkpoint).
  • The inverted roofline (per stage): push B past B* → decode becomes compute-bound → MFU → peak ⇒ N× the online tokens/s on the same GPU.
  • Offline you hold the entire dataset → length-sort & bucket into homogeneous batches — kills the ragged-batch / decode-convoy MFU loss online serving structurally cannot avoid (it can’t see future requests). The single biggest offline-only throughput lever.
  • One shared prefix covers the whole dataset — the same system / rubric / few-shot prompt prepends every row → compute its KV once, reuse it for every request (K2 Role 2). For prefill-heavy batch this is the dominant cost lever, not batch size.
  • → run it on the cheapest capacity (spot, or the online fleet’s idle troughs at lowest priority) and the cheapest GPU type (batch optimises tokens/$, so an older/cheaper SKU often wins — the opposite of latency-bound online), subject to spot capacity availability (multi-region/cloud chase + on-demand fallback when spot is unobtainable).
  • Problem: multi-hour jobs on preemptible hardware — an interruption at hour 5 loses everything; one slow shard gates the job.
  • → checkpoint/resume + straggler mitigation + spot-interruption handler, with exactly-once output on resume (D29).
  • The economic win: filling idle GPU-hours, not buying new ones — batch as the lowest-priority co-tenant on the online fleet’s troughs (D44).
  • Objective: maximise tokens/$ within the job deadline, at the most aggressive operating point that still passes the offline accuracy gate (and, for eval jobs, the determinism gate — batch-size-dependent FP nondeterminism, K17/D10, makes “did the model improve or did the batch change?” unanswerable).

1. The Problem It Solves

Problem→Solution stub. Problem: a huge class of GPU generation work — offline eval, dataset labeling, synthetic-data generation, embedding backfill — has no per-request latency SLO, only a total-throughput / cost target; running it on an online-tuned stack wastes most of the GPU because the online latency SLO caps the batch far below the compute-bound ridge. Solution (what the page builds): a pipeline that removes the latency constraint, runs the same engine at the opposite operating point (max batch, peak MFU, aggressive quant), on the cheapest capacity, and survives multi-hour preemptible execution.number: online B_SLO « B* ⇒ the N× tokens/s gap on the trace (the throughput the page recovers) — cites K1 #roofline (B*/MFU) · canonical: gpu-inference-cost-shapes.html ($ via filling idle GPU-hours)

The inverted roofline (derived — the page’s home)

Problem→Solution stub (the page’s opening derivation, sub-bulleted). Problem: engineers carry online intuition (small batch, low MFU is “normal”) into batch work and leave most of the GPU idle. Solution, derived: (a) online — a latency SLO caps batch at B_SLO « B* ⇒ decode runs below the compute-bound ridge (memory-bound, low MFU); (b) batch — no latency SLO ⇒ push B until HBM-limited, far past B* ⇒ decode becomes compute-bound, MFU → peak; (c) therefore the same model on the same GPU does N× the tokens/s in batch mode purely by removing the latency constraint (cite K1 B*/MFU — read past the ridge, not re-derived). The multi-stage GPU DAG — each stage its own roofline; fuse vs materialize (generalises D)

Problem→Solution stub (generalises D — not one stage, N). Problem: the page so far assumes one GPU stage, but real batch jobs are DAGs of GPU stages with different rooflines — embed→generate→judge (LLM-as-judge eval), encode(ViT)→generate (multimodal labeling), generate→score. One config / fleet / GPU-type is wrong for at least one stage, and if naively fused the whole pipeline runs at the slowest stage’s throughput while the rest idle. Solution: run each stage at its own operating point (its own batch config, precision, and GPU type — encoder stage compute-bound on a cheap SKU, generate stage decode-bound on H100, judge stage a small model on a tiny GPU), then decide fuse vs materialize: fuse = keep intermediates on-GPU/in-memory, overlap stages behind a bounded-queue backpressure — highest utilisation iff per-stage fleets are sized so stage throughputs match (min-stage is the bottleneck); materialize = write each stage’s output to storage and run the next as an independent, separately-tuned, separately-resumable job — the materialised intermediate is a free D29 stage-boundary checkpoint. Crossover: materialize when roofline/SKU-mismatch idle-cost + coupled-failure cost > intermediate storage + re-read I/O — very different per-stage rooflines or long spot jobs ⇒ materialize; balanced, similar-roofline stages ⇒ fuse.number: a 2-stage generate(decode-bound, H100)→judge(small model) job — naive-fused throughput = min(stage) vs balanced per-stage fleets vs materialized; the I/O-vs-idle crossover on the trace — cites K1 #roofline (per-stage), K6/K12 (encoder-stage roofline — rule of two), K15 #d43 (heterogeneous per-stage fleet/bin-packing), D29 (stage boundary = free checkpoint) · real: Ray Data multi-stage GPU pipelines, NVIDIA NeMo-Curator, Spark-on-GPU, lm-eval-harness LLM-as-judge + URLs · scope: the per-stage GPU operating point + fuse/materialize is K16; the DAG orchestration engine (Airflow/Dagster/Flyte) and non-GPU stages stay out (data-eng)

Length-sort & bucket the whole dataset (B — the defining offline-only lever)

Problem→Solution stub. Problem: a ragged batch of mixed-length requests wastes MFU to the wave-quantization / decode-convoy tail (a long generation holds slots while short ones drain — the K4 convoy); online cannot avoid this because it can’t see future requests. Solution: offline you hold the entire dataset → sort by (predicted) length and bin into homogeneous batches so every batch drains together at near-peak occupancy — the single biggest batch-only throughput lever, structurally unavailable to online serving. Composes with output-length-unknown straggler handling (D29) and the K4 ragged-batch wave-balance (cite, don’t re-derive).number: MFU of a length-mixed batch vs a length-bucketed batch on the trace (the convoy loss recovered) — cites K4 (ragged-batch/convoy — rule of two), K1 (MFU) · real: vLLM offline length-sorting, HF sortish sampler, Ray Data sort + URLs

The shared-prefix lever — one dataset-wide prefix, computed once (C — dominant prefill-heavy cost lever)

Problem→Solution stub. Problem: a labeling / eval / scoring job prepends the same system + rubric + few-shot prompt (often 2–8k tokens) to every one of millions of rows; recomputing that prefill per row is the bulk of the GPU cost in a prefill-heavy job. Solution: compute the shared prefix’s KV once and reuse it for every request (K2 Role 2 prefix reuse) ⇒ per-row prefill collapses to only the unique-suffix tokens; in batch the hit rate is ≈100% (the prefix is identical by construction), so this beats batch size as the prefill-heavy cost lever. Crossover: reuse pays whenever shared_prefix_tokens > unique_suffix_tokens — i.e. essentially always for templated batch.number: per-row prefill GPU-s with vs without shared-prefix reuse for a 4k-token rubric + 200-token row over 1 M rows — cites K2 Role 2 (mechanism; rule of two — K2 owns the cache), K1 #prefill (the saved FLOPs) · real: vLLM/SGLang APC in offline mode + URLs

Deep dive D29 — offline-batch resilience: checkpoint / straggler / spot-interruption

Deep-dive stub. Problem: a 6-hour job on spot loses a node at hour 5 → restart-from-zero is catastrophic; and stragglers come in two kinds — a slow/failed node, and an unknown-length generation (the offline analog of K19’s bimodal output-length problem) — either gates the whole job’s completion. Solution: periodic checkpoint of completed-request offsets (resume from last checkpoint, not zero) + idempotent request keys with exactly-once output semantics on resume (re-emitted rows deduped on the idempotent key — double-counting silently corrupts an eval); node-straggler mitigation (speculative re-exec / work-stealing); length-straggler mitigation (predicted-length sort + a max-token cap); spot-interruption handler (drain + checkpoint on the ~30 s notice — ties D3/K22). Crossover: checkpoint interval ≈ √(2·checkpoint_cost / interrupt_rate) (the classic optimum) — too frequent = overhead, too rare = lost work.number: 6 h job, 30 s spot notice, checkpoint every N → expected wasted work; the √ optimum on the trace — cites K1 (throughput for checkpoint_cost), K22 (interruption), K19 (output-length straggler — rule of two) · real: Ray Data, Spark-on-GPU, AWS Batch, SkyPilot managed-spot + URLs

Deep dive D44 — batch on the online fleet’s idle troughs (preemptible co-tenancy) — the economic core

Deep-dive stub. Problem: a dedicated batch fleet doubles GPU spend; the online fleet is 10:1 diurnal → huge idle troughs sit wasted. Solution: run batch as the lowest-priority tenant on the online fleet, instantly preemptible by interactive (K4 preemption + K15 priority) → batch consumes only otherwise-wasted GPU-hours at ≈ marginal cost. Crossover: dedicated/spot-batch-fleet cost vs trough-capacity throughput net of preemption-churn overhead — when batch volume exceeds the troughs a dedicated/spot fleet wins; below it, co-tenancy is ≈ free.number: trough GPU-hours available at 10:1 vs batch volume → the crossover where a dedicated fleet wins — cites K1 (cost), K15 (priority/placement) · real: harvest/defrag schedulers, borrowed-capacity batch + URLs · cross-link L5 (its core cost lever)

Rule 11 — the operating-point spectrum

Selection-sweep stub. small-batch-FP16 (online-like — wastes batch’s no-SLO freedom) → max-batch-FP8 (throughput-optimal default) → max-batch-INT4 + all-spot (cheapest; accuracy + interruption risk). Two-sided cliff: too conservative forfeits the no-SLO freedom; too aggressive trips the accuracy gate (K14/D16) or the restart cost (D29). Pick the most aggressive point that passes the offline accuracy gate.number: tokens/$ at each operating point on the trace, with the accuracy-gate floor — cites K1 (throughput), K14 #d16 (accuracy gate) · real: vLLM/TRT-LLM offline mode, Ray Data batch + URLs

Knobs & their cliffs (consolidated)

Consolidated stub. Batch size: push to the HBM limit (past B*) — capped by KV+activation memory (the K18/D46 prefill-activation OOM at huge batch × long prompt, not latency). Precision: the rule-11 operating point; floor = the offline accuracy and (eval) determinism gate. Batch composition: length-sorted & bucketed (B) — cliff: too-coarse buckets re-introduce the convoy, too-fine buckets lose batch fill. Shared-prefix reuse: on whenever the templated prefix dominates (C). GPU type: the SKU maximising tokens/$ (often older/cheaper, the opposite of latency-bound online). Checkpoint interval: the D29 √ optimum. Capacity tier: troughs vs spot vs dedicated — the D44 crossover, bounded by the job deadline. Scheduler priority: lowest (preemptible by interactive). 4. Alternatives & When Each Wins

Stub (≥2, disqualifying constraint per alt). (a) Run batch on the online-tuned stack as-is — disq: the online latency-SLO caps the batch → leaves the N× on the table; wins never for pure batch (only if the job piggybacks transparently on live traffic with no separate run). (b) A dedicated on-demand batch fleet — disq: doubles GPU spend vs filling troughs; wins when batch volume durably exceeds the online fleet’s troughs (the D44 crossover). (c) All-spot dedicated — disq: interruption risk + checkpoint overhead; wins for very large, latency-insensitive jobs where spot discount > restart cost. (d) Treat the whole multi-stage job as one fused fleet/config — disq: stages have different rooflines → one config starves the mismatched stage and the pipeline runs at the slowest stage; wins only when stages are throughput-balanced and similar-roofline (else per-stage fleets / materialize — the multi-stage node). The DAG orchestration engine itself (Airflow/Dagster/Flyte) and non-GPU stages remain out of scope — flag that boundary precisely.number: the crossover disqualifying each (N× for (a); trough vs volume for (b); spot-discount vs restart for (c); min-stage vs balanced/materialize for (d)) — cites K1, K15, D29 · contrast L6 (dedicated home)

Stub (cite, never inline-redefine).
