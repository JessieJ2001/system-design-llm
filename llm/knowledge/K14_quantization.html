K14 — Quantization (fewer bytes → the four-effect cascade)

How do you build a scalable serving stack on fewer bytes per parameter — when decode is HBM-bandwidth-bound on the weights (and KV) it re-reads every token — by turning one knob (precision) that cascades into faster decode → bigger batch → smaller fleet → lower $, while an accuracy gate stops the silent quality regression that ships with it? This page derives the four-effect cascade (so you cannot reason about quant locally) and its regime ceiling (the W·A axis, D64 — weight-only quant helps only memory-bound decode), then D16 (rollout safety) and D42 (weight-quant vs KV-quant — different bytes, different cascades).

SLIs This Page Reduces

SLIs this page reduces (the reverse index of K1 §3.12):
  • TPOT / ITL (primary) — the memory-bound decode step is bytes / HBM_BW; halving bytes/parameter ≈ halves the step (cascade effect 1).
  • Throughput (tokens/s) — freed HBM → larger KV pool → larger batch B → more tokens/s (cascade effect 2, cite K1 B*).
  • MBU — fewer bytes moved per token at the same HBM BW raises memory-bandwidth utilisation toward the roofline.
  • Cost-shape ($/mo) — fewer GPUs for the same load (cascade effects 3–4) — with the accuracy gate as the binding constraint, not a free lunch.

The Quantization Spine (stated up front, before §1)

Every section is one link in a naive→problem→fix chain; the decision-tree SVG below is this chain:

  • Decode is HBM-BW-bound on the bytes it re-reads every token (weights + KV) — cite K1 #decode.

  • New problem (the cascade’s own ceiling): the cascade fires only in the memory-bound regime — weight-only quant (W4A16) does nothing for compute-bound prefill or large-batch decode, where the per-tile dequant is pure overhead ⇒ it can be slower there. Effect 2 (bigger batch) eventually crosses K1 B* and cannibalises effect 1 (faster decode).
  • → the W·A axis (D64): use W8A8 / FP8 (the low-precision tensor-core path, ≈2× FLOPs) when compute-bound; weight-only only when memory-bound.
  • New problem: lower precision loses accuracy — and it ships silently (no crash; PTQ calibration↔serving distribution skew is a prime cause).
  • → an accuracy-regression eval gate (must pass within ε before rollout).
  • New problem: different tensors tolerate different precision.
  • → mixed precision (keep attention / embedding / final layers, or outlier channels, higher).
  • New problem: a quant change shifts latency / batch / cost simultaneously (the cascade) — you cannot A/B on latency alone.
  • → canary-on-eval, not canary-on-latency (D16).
  • And: which bytes? — weight-quant (fixed per-step, uniform) vs KV-quant (per-token-growing, long-ctx/large-batch) are different cascades, commonly conflated (D42).
  • Objective: the minimum bytes/parameter whose eval_delta stays under the quality budget — the precision floor.

1. The Problem It Solves

Problem→Solution stub. Problem: the decode step re-reads the entire weight set (and the growing KV) from HBM for every single token — arithmetic intensity ≈ 1 — so TPOT is bounded below by bytes / HBM_BW and the fleet is sized by that byte bill, no matter how much spare FLOP capacity the GPU has (cite K1 #decode / #roofline — do not re-derive). Solution (what the page builds): store each parameter (and optionally each KV element) in fewer bits, so the same model moves fewer bytes per token — and trace the four coupled effects that one change sets off.number: decode TPOT floor = W_bytes/HBM_BW on the trace at FP16 (the baseline the cascade halves) — cites K1 #decode, #roofline · canonical: gpu-inference-cost-shapes.html (the $ is decode GPU-hours)

2. The Mechanism — the Tokenized-Prompt Walkthrough

Where quantization lives — the weight (and KV) read path, dequant before the GEMM

Problem→Solution stub (architecture placement, hard req 6). Problem: the reader needs to know what physically changes — quantization is not a layer. Solution: the stored weight matrices (and, for KV-quant, the cached K/V) are held in a low-bit format; on the read path each tile is dequantized (scale × int) just before the matmul, so the math runs in higher precision but the bytes moved from HBM are fewer. Prefill (compute-bound, weights reused across all prompt tokens) benefits little; decode (memory-bound, one token per weight-load) is where the cascade fires — state this before any math (cite K1 #decode for why decode, not prefill).no number (placement fact) — cites K1 #decode (memory-bound decode), K17 (the dequant kernel + dtype support), K0 (which tensors) · diagram: the forward pass with low-bit weight blocks + an inline dequant op on the read edge into each GEMM

3. The System-Design View

Where quantization sits in the GPU topology

Stub. Quantized weights are produced by an offline PTQ calibration pipeline leg (a calibration dataset → fitted scales) and shipped as the served artifact; the serving engine loads them, the kernel dequantizes per-tile on read (W4A16) or runs the low-precision tensor-core path (W8A8 — D64). KV-quant is an online per-block transform on the K2 cache. The artifact carries per-tensor / per-channel scales; the served model version pins {dtype, W·A scheme, scale scheme, which tensors kept high, calibration-set id}. The calibration leg re-runs per model version → it is on the L7 rollout-cadence critical path, and the calibration set is a gated, version-pinned input (skew → D16 silent regression).no new number — cites K2 #quant (KV-quant as a cache lever), K17 (kernel dtype support + tensor-core path), K0 (per-tensor vs per-channel granularity), L7 (calibration on the rollout-cadence path) · SVG below

Deep dive D16 — quantization rollout safety (accuracy gate, mixed precision, canary-on-eval; co-host L7)

Deep-dive stub. Problem: a quant change is invisible until quality regresses in production (it does not crash) — and the cascade shifts latency/batch/cost simultaneously, so you cannot A/B on latency alone. Two named silent causes: (i) calibration↔serving distribution skew — PTQ (GPTQ/AWQ) fits scales on a calibration set; calibrate on web text, serve code / multilingual / long-context → quality drops on the unseen slice while an aggregate gate passes; (ii) non-uniform degradation — quant craters long-context recall, JSON/tool-call validity, low-resource languages and safety/refusal while MMLU stays flat (KV-quant especially: perplexity barely moves, needle-in-haystack collapses). Solution: an accuracy-regression gate that is a failure-skewed slice suite (long-ctx recall, structured-output validity, low-resource langs, safety — not an aggregate benchmark), passing within ε before rollout; mixed precision (keep attention / embedding / final layers, or outlier channels, higher; quantize the rest); calibration set drawn from the serving distribution and pinned to the model version; canary-on-eval, not canary-on-latency. Crossover: the precision floor where eval_delta = the quality budget on the worst slice, not the mean. Rule of two: L7 owns the rollout/canary scenario; K14 owns the quant-specific gate + mixed-precision policy; which benchmark is application/eval-side (K14 owns “the gate is a slice suite”, not the suite’s contents).number: INT4-weight-only 70B → aggregate-MMLU drop X% vs long-context-recall drop Y% (Y ≫ X) → the per-slice gate threshold on the trace — cites K1 (cascade legs) · real: AWQ, SmoothQuant, GPTQ, FP8 per-tensor/per-channel + URLs

Deep dive D42 — weight-quant vs KV-quant: different bytes, different cascades, commonly conflated

Deep-dive stub. Problem: “quantize” is ambiguous — weight-quant shrinks the fixed per-step weight read (helps every request uniformly; accuracy cost on model quality); KV-quant shrinks the per-token-growing KV read (helps long-context / large-batch most; accuracy cost on long-context recall — it is the K2/D32 boundary). Conflating them mis-prioritises. Solution: weight-quant first (biggest, uniform win); add KV-quant when KV dominates the decode byte budget. Crossover: KV-quant pays iff KV_bytes_per_step > weight_bytes_per_step, i.e. B·s·kv_bytes_tok > W_bytes (cite K1 #decode / #memory) — derive the s,B where it flips. Rule of two: K14 owns the method/precision; K2 owns KV-quant as a cache lever (its cascade consumer).number: the (s,B) crossover where KV-quant overtakes weight-quant on the trace — cites K1 #decode/#memory, K2 #quant · real: FP8 KV (vLLM), KVQuant + URLs

Deep dive D64 — the W·A axis: weight-only vs weight+activation quant, and the cascade’s regime ceiling

Deep-dive stub (sibling of D42; the second “which quant” axis). Problem: the cascade as derived assumes weight-only quant (W4A16: low-bit weights, dequant to FP16, FP16 matmul) — which saves only bytes, so it speeds the memory-bound regime (small-batch decode) but does nothing for compute-bound prefill or large-batch decode, where the per-tile dequant is pure added work ⇒ quant can make those slower. This means cascade effect 2 (bigger batch) eventually crosses K1 B* and cannibalises effect 1 (faster decode) — the four effects are not all mutually-reinforcing. Solution: the W·A axis — weight+activation quant (W8A8 / FP8, e.g. SmoothQuant) runs the matmul on low-precision tensor cores (≈2× FP16 FLOPs on Hopper/Ada/Blackwell) ⇒ it wins in the compute-bound regime too, at a more fragile accuracy cost (activation outliers → SmoothQuant migration / rotations). Crossover: memory-bound (interactive small-batch decode) → weight-only INT4 is the biggest TPOT win; compute-bound (prefill-heavy, batch/offline, large B past B*) → W8A8/FP8 (tensor-core path) — weight-only there is a latency regression. Derive the B where weight-only’s benefit inverts (cite K1 #decode / B*). Rule of two: orthogonal to D42 (which bytes: weight vs KV) and to rule-11 (which bit-depth); K14 owns the W·A choice, K17 owns the tensor-core/dequant kernel.number: the (B,phase) crossover where W4A16 inverts to net-slower vs W8A8 on the trace (decode small-B vs prefill / large-B) — cites K1 #decode/#roofline (B*), K17 (tensor-core path) · real: AWQ/GPTQ (W4A16, weight-only), SmoothQuant + FP8 W8A8 (vLLM/TRT-LLM), QuaRot/SpinQuant (rotations for low-bit activations) + URLs · D64 — wired load-bearing: L1, L5, L6, L8

Rule 11 — the precision/method (bit-depth) spectrum

Selection-sweep stub (the invariant is the cascade, not the dtype; orthogonal to the D64 W·A axis). FP16 (baseline) → FP8 (≈free accuracy, 2× cascade, modern default; cliff: needs Hopper+) → INT8 (SmoothQuant / per-channel; cliff: activation outliers) → INT4-weight-only (AWQ/GPTQ, 4× weight cascade; cliff: measurable quality drop → eval gate mandatory) → FP4/MXFP4 (Blackwell, extends the ladder; cliff: hardware-gated, accuracy frontier). Each: cascade factor, accuracy cost, named method, hardware requirement. 4. Alternatives & When Each Wins

Stub (≥2, disqualifying constraint per alt). (a) A smaller / distilled model — disq: a different model with its own quality profile and re-training; wins when the quality budget is large and you control the model; composes with quant (quantize the small model too). (b) KV-offload / tiering (K2) — disq: pays PCIe/NVMe BW and only helps the KV byte leg, not weights; wins for cold long-context KV where accuracy must be untouched. (c) More HBM / bigger GPU — disq: buys capacity not bandwidth-per-byte and costs $; wins when accuracy cannot be touched at all and budget is unconstrained. (d) Quantization-aware training / QAT — disq: training-side, needs the training pipeline; out of scope — K14 is post-training PTQ; flag the boundary, do not cover.number: the crossover disqualifying each (quality delta for (a); BW cost for (b); $ for (c)) — cites K1 #decode, K2 #tier, K6 #d36 (quantize-to-fit composes here) · contrast L5/L1 (FP8-to-fit)

5. Failure Modes

Stub (cite, never inline-redefine). Silent accuracy regression — quant ships, no crash, quality drops on a traffic slice not in the eval set; first move: the D16 eval gate + per-slice eval (gpu-inference-cost-shapes.html / llm-bottlenecks.html). Calibration↔serving distribution skew — PTQ scales fitted on a mismatched calibration set → degradation only on the unseen serving slice while the aggregate gate passes; first move: calibration set drawn from serving traffic + the D16 slice suite (llm-bottlenecks.html). Weight-only quant slower when compute-bound — W4A16 in prefill or past B* → the dequant is pure overhead, no byte-saving benefit → latency regression that looks like “quant didn’t help”; first move: the D64 W·A axis (W8A8/FP8 tensor-core path) — cite K1 B*, K17 (llm-bottlenecks.html). Activation-outlier blow-up — INT8/INT4 without outlier handling → a few channels saturate → large error; first move: SmoothQuant / rotations / keep outlier channels high (llm-bottlenecks.html). Dtype unsupported by the kernel — falls back to a slow generic path → the cascade’s latency win evaporates (cite K17 — kernel/dtype support; gpu-failure-modes.html). KV-quant long-context recall loss — aggressive KV precision degrades retrieval over long context silently (cite K2 #quant; llm-bottlenecks.html).
