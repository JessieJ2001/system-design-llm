K7 — Tensor Parallelism (split within the layer)

How do you build a scalable within-layer split — when a layer’s weights or compute are too big or too slow on one GPU — without the per-layer all-reduce collapsing your latency? This page derives column/row-parallel and the all-reduce, then shows the one fact that caps the whole technique: the all-reduce is on the critical path of every layer, so TP degree can never exceed the NVLink domain.

SLIs This Page Reduces

SLIs this page reduces (the reverse index of K1 §3.12):
  • Feasibility (weights fit) — splitting every weight matrix across the TP group makes a layer that overflows one GPU’s HBM resident.
  • TPOT / TTFT — TP divides the per-layer matmul across GPUs → lower per-token latency — until the all-reduce crosses the NVLink domain and the sign flips (D26).
  • KV headroom — n_kv_heads shards across the TP group → per-GPU KV ∝ 1/TP (the K2-sizing interaction).
  • MFU — async-TP (D37) hides the all-reduce under the next GEMM so the added comm doesn’t depress achieved utilisation.

The TP Spine (stated up front, before §1)

This page is a child of K6 and resolves the weights-don’t-fit binding of K6’s grid. The chain:

  • A layer too big / too slow on one GPU → split within the layer (column- & row-parallel) = TP.
  • New problem: every layer now needs an all-reduce to recombine the shards.
  • That all-reduce is on the critical path of every layer (twice per block, every token).
  • ⇒ it must run on NVLink (≫ IB bandwidth).
  • ⇒ the NVLink-domain size caps TP degree (the master constraint, K20).
  • Beyond the cap latency collapses (D26) → hand off to PP / DP (back to K6); on NVLink, async-TP (D37) hides the barrier.

The Page as One Decision Tree

The arc: a layer doesn’t fit / is too slow on one GPU → split it within (TP) → new problem: an all-reduce per layer → it sits on the critical path → so it must live on NVLink → so the NVLink domain caps TP → cross the cap and latency collapses (D26) / hide it on NVLink with async-TP (D37) → objective: the smallest TP that meets the SLO, never exceeding N_dom. 1. The Problem It Solves

Problem→Solution stub. Problem: a single transformer layer’s weight matrices (or the matmul over them) exceed one GPU’s HBM / latency budget — K6’s weights-don’t-fit binding lands here. Solution (what the page builds): split every weight matrix within the layer across a TP group so the per-GPU shard fits and the per-token matmul latency drops — at the cost of one all-reduce per matmul-pair, which the rest of the page bounds.number: per-GPU weight bytes at TP=1 vs TP=8 on the trace, vs HBM capacity — cites K1 §3.2 (#memory) · canonical: gpu-inference-cost-shapes.html

2. The Mechanism — the Tokenized-Prompt Walkthrough

Where TP lives — inside every block’s matmuls, the residual add staying GPU-local

Problem→Solution stub (architecture placement, hard req 6). Problem: the K0 forward pass says nothing about splitting a matmul. Solution: TP splits the within-block GEMMs (QKV proj, attention out-proj, MLP up/down proj); the element-wise residual add stays GPU-local (cite K0/K18 — the legal-cut constraint K6 obeys). It is a deployment cut of the same model, not a model change.no number (placement fact) — cites K0/K18 (TP within a block, residual add local) · diagram: one K0 block with the column/row split marked

The walkthrough on "I like cat, why?" — what TP does to these 6 tokens

Problem→Solution stub (hard requirement 6 — reader’s first contact). Canonical trace T1 = "I like cat, why?" (6 tokens). Each GPU holds a column/row slice of every weight matrix. Prefill: all 6 tokens flow through the sliced matmuls in parallel; after attention out-proj an all-reduce sums the partial outputs across the TP group; same after the MLP down-proj — every block. Decode: identical pattern for the 1 new token (same 2 all-reduces/block, tiny payload — latency-bound, sets up D37). Column- and row-parallel + the 2-all-reduces-per-block result (derived, this page’s home)

Problem→Solution stub (the page’s opening derivation, sub-bulleted). Problem: splitting a matmul naively forces communication on every sub-step. Solution, derived: (a) column-parallel — split a weight matrix by output columns → each GPU computes an independent slice, no communication; (b) row-parallel — split by input rows → each GPU a partial sum → one all-reduce to combine; (c) Megatron composition — MLP = column→row = 1 all-reduce; attention = heads split across GPUs → 1 all-reduce after out-proj ⇒ 2 all-reduces per transformer block; (d) per all-reduce volume ≈ B·s·d_model·2 bytes (cite K1’s ring-cost primitive — the pattern is derived here, the generic cost is quoted); (e) TP’s KV effect — n_kv_heads splits across the TP group → per-GPU KV ∝ 1/TP (the K2-sizing interaction).numbers: all-reduce bytes/block on the trace at TP=8; per-GPU KV at TP=1 vs 8 — cites K1 #collectives #memory, K0 (head/d_model shapes), K2 (KV sizing)

Deep dive D26 — the TP-beyond-NVLink-domain cliff

Deep-dive stub. Problem: the per-layer all-reduce is latency-critical; inside NVLink (≈0.9 TB/s/GPU) it hides, but the moment TP spans PCIe/IB (≈50 GB/s) it explodes ≈18× → per-token latency collapses (TP=16 across two nodes tanks vs TP=8 in one). Solution: never let TP cross the NVLink domain; add sequence/tensor-parallel to also cut LayerNorm+residual activation memory inside the domain; past it switch to PP/DP. Crossover: TP degree ≤ N_dom, always.number: all-reduce time on NVLink vs IB on the trace (the ~18×); TP=8-in-domain vs TP=16-cross-domain per-token latency — cites K1 #collectives, K20 (BW tiers, N_dom) · real: Megatron-LM sequence-parallel; NVL72 (enlarged N_dom softens the cliff — state the assumed domain size) + URLs

Deep dive D37 — TP communication–compute overlap (async-TP)

Deep-dive stub (the dominant production all-reduce-hiding technique). Problem: even on NVLink the 2 all-reduces/block are a serial barrier on the decode critical path. Solution: decompose the all-reduce into pipelined reduce-scatter/all-gather steps interleaved with the consuming GEMM so comm hides under the next matmul. Crossover (formula): comm fully hides iff all_reduce_time(TP) ≤ overlappable_GEMM_time(next proj) per block — give the TP degree where it stops hiding.number: all-reduce time vs next-proj GEMM time per block on the trace, and the TP at which it stops hiding — cites K1 #collectives #decode · real: Megatron-Core async-TP, NVIDIA Flux + URLs

Rule 11 — TP-degree selection

Selection-sweep stub. 1 → 2 → 4 → 8 → 16(cross-domain): each step ≈halves per-layer compute latency but adds all-reduce cost that is flat inside N_dom and explosive outside it. Two-sided cliff: too low → doesn’t fit / latency too high; at/just-below N_dom → the sweet spot; past N_dom → comm-bound collapse (D26). Pick the smallest TP that fits weights+KV under the latency SLO, never exceeding N_dom.number: the per-degree latency vs all-reduce-cost curve on the trace, with the N_dom knee — cites K1, K20, K8/K10 (the hand-off past the cap)

4. Alternatives & When Each Wins

Stub (≥2, disqualifying constraint per alt). (a) More TP past N_dom — disq: the all-reduce crosses IB → ~18× → latency collapse (D26); never wins. (b) PP instead of TP — disq: PP adds a bubble and does not lower single-token latency; wins when the model exceeds one domain and throughput, not latency, binds (→ K8). (c) Quantize instead of shard (K6/D36) — disq: when quant accuracy loss > SLO floor; wins when FP8/INT4 makes the layer fit one GPU. (d) DP only — disq: fails the moment one replica doesn’t fit; wins when it fits and you are purely throughput-bound (→ K10).number: the constraint value disqualifying each (N_dom for (a); bubble vs latency for (b); SLO floor for (c)) — cites K8, K10, K14/K6-D36 · contrast L1 (TP≥8 frontier MoE)

5. Failure Modes

Stub (cite, never inline-redefine). The whole TP group is one all-or-nothing fault domain — lose any member and every layer’s all-reduce is incomplete ⇒ the model is down for all in-flight requests on the group (cite K22, gpu-failure-modes.html); TP crossing N_dom → interconnect saturation (the per-layer all-reduce on IB — llm-bottlenecks.html); over-sharding → comm-bound collapse (more GPUs, less goodput).cites llm/canonicals/gpu-failure-modes.html (TP group blast radius), llm/canonicals/llm-bottlenecks.html (interconnect saturation), K22 (the fault-domain derivation) · ACTION: seed/extend these canonical entries in this same change
