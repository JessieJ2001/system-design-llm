K17 — Attention Kernels & CUDA Graphs (where the roofline meets achieved MFU/MBU)

How do you actually hit the K1 roofline — when the same model on the same GPU runs 2–4× faster or slower depending on the kernel (tiling, the paged gather, the split-axis) and the launch strategy (thousands of µs-scale launches vs one CUDA-graph replay) — and how do those two factors impose shape-rigidity that every upstream scheduling decision must respect? This page derives achieved = roofline×kernel-eff×launch-eff, then D51 (the attention-parallelization design space), D22 (graphs under dynamic batching), D10 (batch-invariance), D45 (the shape-coverage cliff).

What This Page Is (and Is Not)

TL;DR. The roofline is a ceiling; achieved MFU/MBU is roofline × kernel_eff × launch_eff. The Kernel Spine (stated up front, before §1)

  • Roofline (K1) is a ceiling; achieved MFU/MBU is set here.
  • Naive attention materializes the [N,N] score matrix → O(N²) HBM → FlashAttention tiles + online-softmax (K0) → O(N) HBM. kernel_eff is also “do you feed the Tensor Cores” — WGMMA/TMA/async-warp-specialization/FP8 (FA-3 ≠ FA-2); and FlashAttention fixes memory, prefill compute is still O(N²) FLOPs. [ADDED]
  • Paged KV is non-contiguous → the kernel must gather (the paging cost; K2/D31 owns the layout).
  • There is no one “attention kernel” — the fast kernel is keyed by the model’s attention algorithm: GQA/MQA-packed decode, MLA (frontier MoE), sliding-window, attention-sink each need a matching kernel (K0 owns the algorithm, K17 the kernel); the kernel must also read FP8/INT8 KV & dequant in-register, never round-trip dequantized KV through HBM. A mismatch is a D45-class cliff at the algorithm axis. [ADDED]
  • Decode is a thin GEMV: one query gives no M-parallelism → a single CTA underfills the GPU → split the KV dimension (FlashDecoding) with an online-softmax epilogue reduction.
  • Batched variable-length decode then needs load-balanced CTA scheduling (fixed-chunk split-KV + persistent / stream-K) so a long sequence doesn’t starve short ones (D51); chunked-prefill’s “append” kernel is a third Q-thickness regime (medium-Q chunk vs long cached KV; K4/D1 requires it). [ADDED]
  • Per-launch overhead exceeds kernel runtime at small batch → CUDA graph replays the whole step as one launch.
  • Graphs need static shapes → bucketing constrains the scheduler (K4/D22) and breaks determinism (D10) — and is worse under tree-attention/variable-accept spec-decode (K13) and graphs containing the TP-group NCCL all-reduce. [ADDED]
  • Determinism couples back: split-KV’s non-fixed epilogue reduction (D51) is itself a source of D10 — the same kernel decision that fills the GPU creates the reproducibility problem. [ADDED]
  • Which kernel is fastest depends on shape → a dispatcher/autotune, with a cliff off the covered shapes (D45); the launch strategy spectrum ends at a persistent megakernel (dynamism in-kernel, dodges D22, answers K4/D33). [ADDED]
  • Objective: the kernel + launch strategy that maximises achieved MFU/MBU under the SLO, with shape-rigidity every upstream layer must respect.

Deep dive [ADDED — candidate D-number, not yet catalogued] — the attention-variant kernel match: GQA/MQA-packed, MLA, sliding-window, sink (K0 owns the algorithm, K17 the kernel)

Deep-dive stub [ADDED]. Problem: there is no single “attention kernel” — the fast kernel is keyed by the model’s attention algorithm, and a mismatched kernel silently falls to a slow generic path (a D45-class cliff at the algorithm axis, not the shape axis). At decode, vanilla-MHA FlashDecoding on a GQA model re-reads each KV head once per query head and wastes the n_q/n_kv bandwidth advantage; the right kernel packs the query-head group sharing a KV head so cached KV is read once (raises decode arithmetic intensity ∝ group size). MLA (DeepSeek-V2/V3 — the dominant frontier-MoE attention, load-bearing in L1) needs a different kernel entirely (absorbed vs materialized low-rank-KV forms — the wrong form blows up either compute or KV bytes). Sliding-window (Mistral/Gemma) needs a windowed gather + tile structure attending only the last W tokens (the K2 block table must also be windowed). Attention sink / StreamingLLM keeps the first few + last W. Solution: a per-variant kernel selected from the model config — GQA-packed decode layout, MLA absorbed-form decode kernel, windowed/sink masks fused into the tile loop; + the kernel must read FP8/INT8 KV and dequant in-register — never round-trip dequantized KV through HBM (that destroys the K2/K14 KV-quant cascade; the kernel side of D31/D42). Deep dive D51 — attention-kernel parallelization: split-M vs split-KV/FlashDecoding vs persistent/stream-K (cites K4/K2/K0)

Deep-dive stub + selection sweep (the big one). Problem: the attention output is a matmul-shaped problem; which dimension you parallelize across CTAs is set by Q-thickness, and the wrong axis underfills the GPU — prefill has a thick Q (1k–100k rows), decode a thin Q (1 row) ⇒ a Q-only kernel produces ~1 CTA per (batch,head) at decode on a ~132-SM GPU → occupancy collapses even with HBM headroom. Define CTA (cooperative-thread-array / threadblock — the unit the grid scheduler hands to an SM); throughput ≈ total_CTAs / num_SMs per wave. Solution — the spectrum, weakest→strongest, each with its two-sided cliff + a named system: (0) split-M / query-parallel (thick-Q prefill, FA-2; cliff: ~1 CTA at decode → GPU starved); (1) split-KV / FlashDecoding (thin-Q decode; split the KV/contraction dim, online-softmax epilogue reduction — the cost split-M doesn’t pay; cliff: reduction tax, only worth it when M-parallelism is insufficient); (2) fixed-chunk split-KV + persistent / stream-K (batched variable-length decode; work unit = a fixed KV chunk, work-steal across SMs; cliff: chunk too small → per-CTA overhead, too large → imbalance returns). Composition + crossover: split-M while M_CTAs ≥ k·N_SM; below that add split-KV with KV_chunks s.t. (M_CTAs·KV_chunks) mod N_SM ≈ 0; switch to fixed-chunk+persistent the moment KV lengths vary. [ADDED] The 3rd Q-thickness regime — the chunked-prefill “append”/context kernel: a medium-Q chunk attending a long cached KV across the chunk boundary with a boundary-spanning causal mask — K4/D1’s chunked prefill requires it (it is neither pure split-M thick-Q nor pure split-KV thin-Q; it uses both). Named: FlashInfer BatchPrefillWithPagedKVCache / vLLM chunked-prefill attention. Rule of two: K4 owns batch-composition / wave-balance and the decision to chunk; K17 owns the split-axis and the append-kernel. K2/D31 owns the per-chunk gather layout; K0 owns the online-softmax that makes the epilogue exact.number: decode S=1, n_heads×batch CTAs vs N_SM → the split-KV chunk count that fills the GPU + the MBU delta on the trace — cites K1 (roofline/ridge), K4 (wave-balance), K2/D31 (gather), K0 #softmax-online · real: FlashDecoding (Stanford CRFM / PyTorch post), vLLM PagedAttention kernel, FlashInfer, CUTLASS stream-K + URLs

Deep dive D22 — CUDA-graph capture under dynamic batching

Deep-dive stub. Problem: continuous batching changes shape every step; a captured graph needs fixed shapes; re-capturing per step is impossible. Solution: capture a set of graphs at bucketed batch sizes / seq tiers; the scheduler pads to the nearest bucket (K4 owns the snap, K17 owns the capture / piecewise-graph machinery — rule of two); piecewise graphs graph the static parts and leave attention dynamic. Crossover: bucket count — more buckets = less pad waste but more capture memory + warm-up; bucket-pad waste vs launch-overhead saved. [ADDED] Two production realities that make this worse: (i) speculative / Medusa decode — a tree-attention kernel (custom non-causal tree mask) and a variable accepted-token count per step mean the per-step shape itself varies (the K13↔K17 boundary: K13 owns the acceptance math, K17 the tree-attn kernel + its bucket set), so the buckets must cover (1+K)-token shapes or spec-decode silently disables graphs; (ii) graphs that contain NCCL collectives — the captured decode step on a TP≥8 group includes the per-layer all-reduce; capture must use the NCCL graph-capture mode / a side stream and pins a memory pool (interacts with the K2 KV allocator) or it serializes/deadlocks — load-bearing in L1. Rule of two: K20 owns the collective, K17 owns capturing it.number: decode kernel 8 µs, launch 6 µs → graph removes ~6 µs/step → TPOT −X% on the trace; bucket-pad vs launch-saved crossover; [ADDED] spec-decode (1+K)-shape bucket blow-up; graph⊗NCCL capture cost — cites K1 (step time), K4 #d22 (snap side), K13 (spec-decode shape), K20 (the TP all-reduce inside the graph) · real: vLLM CUDA-graph capture, TensorRT-LLM, torch.compile(reduce-overhead), NCCL graph-capture mode + URLs

Deep dive D10 — batch-invariance / numeric determinism (co-host K4)

Deep-dive stub. Problem: the same prompt yields a different output at a different batch size — non-associative FP reduction order changes with the kernel’s tiling/batch → breaks eval reproducibility, breaks the K2 prefix-cache assumption (a prefix computed at batch B ≠ a fresh compute at B′), breaks A/B tests. [ADDED] D51 is itself a source (the coupling the original stub missed): split-KV/FlashDecoding combines partial-softmax results across CTAs via an epilogue reduction whose order is not fixed (atomics / variable chunk count) — so the more you split-KV for occupancy (D51), the worse this nondeterminism: the same kernel decision that fills the GPU creates the reproducibility problem (this is the dashed coupling edge in the spine SVG). A batch-invariant kernel must therefore also fix the split-KV reduction order, not just the batch-reduction order. Solution: batch-invariant kernels (fixed reduction order regardless of batch and split-KV chunking) or pin the batch on cache/eval-sensitive paths. Crossover: batch-invariant kernels cost ~5–15% throughput vs the reproducibility / cache-correctness they buy. Rule of two: K17 owns the kernel determinism (incl. the D51 epilogue order); K4 co-hosts the batch-pinning policy.number: output divergence rate across batch sizes and split-KV chunk counts on the trace; the ~5–15% throughput cost of batch-invariant kernels — cites K1 (throughput), K2/D5 (the cache assumes reproducible KV), D51 (the split-KV epilogue is the in-page source) · real: “Defeating Nondeterminism in LLM Inference” (Thinking Machines), batch-invariant FlashInfer + URLs · cross-link L6 (stored/compared embeddings)

Deep dive D45 — the kernel-dispatch / autotune shape-coverage cliff

Deep-dive stub. Problem: a kernel is fastest only over a shape range (head_dim, seq_len, batch); serving sees a wide shape distribution; an autotuned / engine-built config covers the captured shapes → an off-distribution shape (an unusual prompt length, a rare batch) falls to a slow generic kernel → a specific traffic slice is 2–4× slower while aggregate p50 looks fine. Solution: a per-shape kernel dispatcher (FlashInfer’s shape-specialized kernels), autotune over the measured serving shape distribution, a guaranteed-correct slow fallback, and per-shape-bucket latency monitoring (not just aggregate p99). Crossover: how many shapes to specialize/precompile — coverage vs build-time + binary size; cover the shapes carrying the traffic mass.number: the off-distribution slice slowdown (2–4×) while p50 unchanged on the trace; shapes-to-specialize vs build-time crossover — cites K1 (per-shape kernel cost) · real: FlashInfer, TensorRT-LLM engine build, Triton autotune + URLs

Rule 11 — the kernel/compile-strategy spectrum

Selection-sweep stub. eager (flexible, slowest, launch-bound) → CUDA graph (replay, fast, static-shape rigid) → torch.compile / reduce-overhead (fusion + graph, build step, partial dynamism) → TensorRT-LLM / engine build (peak perf, full autotune, heaviest build + most shape-rigid) → [ADDED] persistent megakernel (fuse the whole decode step — all layers — into one persistent kernel that never returns to the host: no per-layer launch, no graph capture, dynamism handled in-kernel ⇒ it dodges the D22 static-shape problem entirely and is the strongest answer to K4/D33’s CPU-stall tax; cliff: hardest to write/debug, one-model-specialized, immature tooling). 4. Alternatives & When Each Wins

Stub (≥2, disqualifying constraint per alt). (a) Naive (materialized-scores) attention — disq: O(N²) HBM → memory-bound and OOM at long context; wins never in serving (only short-seq pedagogy). (b) Eager execution (no graph) — disq: per-launch overhead dominates at small batch; wins when shapes are wildly dynamic (guided decode, heavy spec-decode) and batch is large enough that launch overhead is hidden. (c) One generic kernel for all shapes — disq: the D45 cliff — a traffic slice 2–4× slow; wins only when the shape distribution is genuinely narrow. (d) Hand-writing CUDA/Triton kernels — disq: this page is about kernel selection & its serving consequence, not kernel programming; out of scope — name FlashAttention/FlashInfer, flag the boundary, do not cover.number: the crossover disqualifying each (HBM O(N²) for (a); launch-overhead/batch for (b); slice-slowdown for (c)) — cites K1 #roofline, K0 #softmax-online · contrast L1 (kernel at TPOT≤30 ms)
