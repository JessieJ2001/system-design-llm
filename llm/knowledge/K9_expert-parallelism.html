K9 — Expert Parallelism (MoE — the dominant 2025–26 architecture)

How do you build a scalable MoE deployment — huge total params, tiny active params — spreading experts across GPUs without the two all-to-alls per layer stalling every decode token or a hot expert straggling the barrier? This page derives why MoE flips the memory-vs-compute balance, then bounds the all-to-all tax (D30) and the hot-expert skew (D15).

The EP Spine (stated up front, before §1)

This page is a child of K6 and resolves the MoE: big total / small active binding of K6’s grid. The chain:

  • MoE has huge total weight memory but tiny per-token FLOPs (only top-k experts fire) → weights can’t sit on a few GPUs but compute is cheap.
  • → spread experts across GPUs = EP.
  • New problem: routing each token to its experts needs two all-to-alls per layer (dispatch + combine).
  • Those all-to-alls are a barrier on every decode token ⇒ they must stay inside the NVLink domain.
  • Skewed token→expert routing overloads the GPU holding a hot expert ⇒ its straggler gates the barrier → rebalance (capacity factor / EPLB, D15).
  • ⇒ EP degree is bounded by the NVLink domain, not FLOPs (the master constraint, K20).

1. The Problem It Solves

Problem→Solution stub. Problem: an MoE’s total expert weights (all experts must be resident) far exceed a few GPUs’ HBM, even though only top-k experts fire per token (tiny active FLOPs) — K6’s MoE binding lands here. Solution (what the page builds): shard experts across GPUs so total weight memory is spread — at the cost of two data-dependent all-to-alls per layer, which the rest of the page bounds.number: total vs active params for a frontier MoE on the trace (e.g. DeepSeek-V3); resident weight bytes / GPU at EP=1 vs EP=N — cites K1 §3.2 (#memory) · canonical: gpu-inference-cost-shapes.html

2. The Mechanism — the Tokenized-Prompt Walkthrough

Where EP lives — the MoE-layer FFN, replacing the dense MLP

Problem→Solution stub (architecture placement, hard req 6). Problem: only the FFN sub-layer is replaced by experts; attention is unchanged. Solution: in an MoE block the dense MLP becomes a gate + N experts; EP places those experts on different GPUs, attention/shared-expert stay local (cite K0 for the block structure). It is a deployment cut, not a model change — serving inherits the trained gate.no number (placement fact) — cites K0 (the MoE block) · diagram: a dense block vs an MoE block, experts coloured by GPU

The walkthrough on "I like cat, why?" — what EP does to these 6 tokens

Problem→Solution stub (hard requirement 6 — reader’s first contact). Canonical trace T1 = "I like cat, why?" (6 tokens). At each MoE layer the gate picks top-k experts per token → the 6 tokens scatter to experts on possibly-different GPUs (dispatch all-to-all) → experts run the FFN → results return to the originating token slot (combine all-to-all) for the gate-weighted sum. Prefill: all 6 tokens dispatched in one exchange. Decode: the 1 new token alone — tiny payload, fixed-latency barrier (this is why decode all-to-all is straggler-bound, sets up D30). Why MoE flips the memory-vs-compute balance (derived, the page’s home)

Problem→Solution stub (the page’s opening derivation, sub-bulleted). Problem: reasoning about MoE cost as if it were a dense model mis-prioritises. Solution, derived: (a) dense: active params = total params; (b) MoE: active ≈ (top_k / n_experts)·total; (c) ⇒ weight memory is dominated by total params (all experts resident) while per-token FLOPs are set by active params (tiny); (d) consequence: decode is still memory-BW-bound (read the selected experts’ weights per token) but which experts is data-dependent ⇒ the cost that matters is the data-dependent all-to-all (D30) + the total-param memory footprint, not a GEMM. This is why EP’s design centers on a collective.numbers: total/active param split + per-GPU resident bytes on the trace — cites K1 #memory #decode, K0 (block structure)

Deep dive D30 — the all-to-all communication tax (dispatch + combine)

Deep-dive stub (the EP-defining cost). Problem: routing is data-dependent ⇒ a full rank→rank exchange with a different payload per pair, on the critical path of every token of every decode step. Solution / mechanism: gate picks top-k → dispatch a2a ships hidden vectors to expert GPUs → FFN → combine a2a ships results back. Volume (derived here): per token per MoE layer dispatch moves top_k×d_model×dtype_bytes, combine the same ≈ 2×top_k×d_model×dtype; worked on a frontier MoE (DeepSeek-V3: d_model=7168, top_k=8, FP8 ⇒ ≈57 KB/token/layer; ×~58 layers×2 ≈ 6.6 MB cross-GPU/token) — tiny vs ~37 GB active weights/token ⇒ at decode the a2a is fixed-latency- and straggler-bound, a barrier, not a pipe. Knob (formula): hidden iff all_to_all_time(EP,payload) ≤ overlappable_compute(shared-expert + next-microbatch attention) — both directions + the EP where it flips. Ceiling: largest EP keeping the a2a inside one NVLink/NVSwitch (or NVL72) domain.numbers: the 6.6 MB/token derivation + the overlap-condition EP on the trace — cites K1 #collectives (generic a2a cost), K20/D28 (topology/IB cliff) · real: DeepEP + DeepSeek-V3 report, Microsoft Tutel, Megatron-Core MoE, GShard/GSPMD, GB200 NVL72 + URLs

Deep dive D15 — hot-expert skew & rebalancing

Deep-dive stub (D30 owns the primitive; this owns which expert is hot). Problem: token→expert routing is data-dependent and skewed (some experts “popular”) → the GPU holding a hot expert does more FFN work + larger a2a payloads → it straggles → the a2a barrier waits on the slowest rank → the whole MoE layer stalls. Solution: capacity factor C (cap tokens/expert; drop/reroute overflow — also bounds the statically-sized a2a buffer), EPLB (replicate hot experts onto redundant GPUs / permute placement), online rebalancing. Crossover: C too low → dropped-token quality loss; too high → re-admits the straggler → pick the C where dropped-token quality loss = straggler-latency cost.number: straggler delay vs dropped-token rate at C on the trace — cites K1 (a2a/FFN cost), llm-bottlenecks.html#hot-expert · real: DeepSeek-V3 EPLB + redundant experts, GShard capacity factor, Tutel adaptive a2a + URLs

Rule 11 — EP-degree selection

Selection-sweep stub. 1 → small → up to N_dom → across-IB: more EP = less weight memory/GPU (good) but more off-rank tokens → larger a2a volume + more likely to cross the NVLink domain (the D30/D28 cliff). Two-sided: too low → weights don’t fit; at the N_dom boundary = the ceiling; past it → all-to-all-over-IB collapse. Pick the largest EP whose a2a stays inside one NVLink/NVSwitch (or NVL72) domain, unless weight memory forces it wider.number: weight-mem/GPU vs a2a-volume curve across EP on the trace, with the N_dom knee — cites K1, K20/D28

4. Alternatives & When Each Wins

Stub (≥2, disqualifying constraint per alt). (a) Pure TP on the MoE FFN — disq: replicates the gating + still reads all selected experts; doesn’t spread total weight memory; wins only for a tiny expert count. (b) EP across IB — disq: the a2a lands on IB → the all-to-all-over-IB cliff (D30/D28) → decode barrier explodes; never wins unless weight memory leaves no choice. (c) Dense model instead of MoE — disq: matches MoE quality only at far higher active FLOPs/$; wins when serving infra can’t afford the a2a complexity. Each: when it actually wins.number: the constraint disqualifying each (expert count for (a); IB a2a time for (b)) — cites K7, K20/D28 · contrast L1 (frontier MoE — D30/D15 load-bearing)
