K11 — Context Parallelism (long context — Ring / sequence parallel)

How do you build a scalable single-request path — when one sequence’s KV+activations exceed one GPU even at TP=N_dom (a 1M-token context) — by splitting the sequence across GPUs without the ring communication exceeding the attention compute it overlaps? This page derives Ring Attention and its bandwidth cost, then the CP-vs-alternatives decision.

SLIs This Page Reduces

SLIs this page reduces (the reverse index of K1 §3.12):
  • Feasibility (one sequence fits) — splitting the sequence across GPUs makes a single request whose KV+activations exceed one GPU even at TP=N_dom resident.
  • TTFT (long context) — CP parallelises the long prefill across GPUs; the ring overlaps comm with attention compute.
  • Goodput (mixed traffic) — CP isolates the long request’s footprint so short traffic on the shared fleet isn’t starved.

The CP Spine (stated up front, before §1)

This page is a child of K6 and resolves the one-sequence-too-big binding of K6’s grid. The chain:

  • One sequence’s KV+activation exceeds one GPU even at TP=N_dom (1M-token context) → split the sequence across GPUs = CP.
  • New problem: each GPU holds a position-shard but attention needs all positions.
  • → ring-pass KV around the CP group overlapped with local attention (Ring Attention); online-softmax (K0) makes the streamed combine exact. The communication pattern is itself a choice: Ring P2P (BW-overlapped, any CP degree) vs. all-to-all / DeepSpeed-Ulysses (lower volume but CP ≤ n_heads) vs. a 2D Ulysses×Ring hybrid (rule-11 below).
  • Master constraint: the ring moves the full KV CP−1 times — it is bandwidth-heavy, so it is more domain-sensitive than PP’s thin P2P. CP stays inside the NVLink domain or the ring collapses onto IB (the same N_dom wall as TP/EP, sharing that joint budget — K20).
  • New problem: the causal mask makes later shards attend to more tokens ⇒ load imbalance → zig-zag / striped assignment rebalances it (D9).
  • At decode CP does not cleanly degenerate. seq=1 removes the prefill compute-parallel motive, but the 1M-token KV still doesn’t fit one GPU — CP persists as KV-parallel decode: KV stays sharded across the CP group, the single query is broadcast/ring’d to attend every shard, and a split-KV / FlashDecoding online-softmax epilogue (K0) combines the partials. “Fall back to TP” is wrong here — the KV is precisely why one GPU is impossible (the D40 decision picks which sharded-decode form, not whether CP vanishes).
  • CP composes with chunked prefill — it is not an alternative to it. At 1M tokens you need both: CP for KV/activation capacity, chunked prefill (K4) to bound the per-step activation footprint (K18) — each chunk is itself CP-sharded and rings against the CP-sharded KV of all prior chunks.
  • CP is for when the sequence, not the model, is what doesn’t fit.

The walkthrough on a long prompt — what CP does to the positions

Problem→Solution stub (hard requirement 6 — reader’s first contact). Use the canonical trace scaled long (T1 "I like cat, why?" as the head of a 1M-token document). The positions are split into CP shards, one per GPU. Prefill: each GPU computes local attention against its current KV block, passes it to the ring neighbour, receives the next; after CP steps every query has seen every key — online-softmax combines the streamed partials exactly. Decode: seq=1 so there is no query to shard — but the 1M-token KV is still CP-sharded and still doesn’t fit one GPU, so CP persists as KV-parallel decode: the one new query is broadcast to every CP rank, each computes a partial against its KV shard, a split-KV online-softmax epilogue combines them (the asymmetry — prefill is query-parallel, decode is KV-parallel; sets up D40, not a fall-back to TP). Ring Attention + its BW cost (derived, the page’s home)

Problem→Solution stub (the page’s opening derivation, sub-bulleted). Problem: a sharded query still needs K,V for all prior positions held on other GPUs. Solution, derived: (a) split the sequence into CP shards (one/GPU); (b) ring schedule — each step compute local attention against the current K,V block, pass it to the ring neighbour, receive the next; after CP steps every query has seen every key; (c) online-softmax (cite K0 #softmax-online) makes the streaming combine exact — not approximate; (d) BW — each GPU sends/receives the full KV CP−1 times → per-GPU comm ≈ KV_bytes·(CP−1)/CP, overlapped with local attention; hides iff ring_step_comm ≤ local_attention_compute — derive both directions.number: ring comm vs local-attention time per step at CP=8 on the trace — cites K1 #collectives (generic ring cost), K0 #softmax-online (combine exactness), K18 (activation)

Deep dive D9 — production CP (causal-mask imbalance + coherent multi-rank preempt)

Deep-dive stub. Problem: with a causal mask, later sequence shards attend to more tokens, so a naive contiguous split makes the last GPU do far more work (≈2×) — the slowest shard gates the step. Solution: zig-zag / striped shard assignment (each GPU gets a low+high position pair → balanced attended-token count); the ring P2P schedule overlaps KV blocks in flight with local attention; preempting a CP’d request means swapping KV from all CP ranks coherently (→K4/K2). Crossover: contiguous (simple, ≈2× imbalance) vs zig-zag (balanced, scatter-gather index cost).number: per-shard attended-token count contiguous vs zig-zag at CP=8 on the trace — cites K1 (attention cost), K4/K2 (coherent multi-rank preempt) · real: Ring Attention (Liu et al.), Llama-3 CP, DeepSpeed-Ulysses + URLs

Deep dive D40 — the CP-vs-alternatives decision (derived; parallels D35/D25)

Deep-dive stub (long context has four answers; picking wrong wastes GPUs or blows TTFT). Problem: long context can be served by CP, more-TP, KV-offload/tier (K2), or chunked-prefill-only (K4). Solution / crossover: choose CP iff seq_KV+act > HBM/GPU at TP=N_dom AND prefill/long-context-heavy AND ring_comm hides under compute; more-TP is bounded by N_dom (K7); KV-offload (K2 tier) wins when the long context is cold (reused, not attended every step — offload beats CP’s persistent ring comm); chunked-prefill-only (K4) suffices when context fits one GPU but is merely slow to prefill — note this is the one alternative that also composes with CP (at 1M you run CP and chunked prefill; the others are mutually exclusive with CP). State the inequality both directions. The decode asymmetry, correctly: seq=1 ends query-parallelism but the sharded 1M-token KV still cannot land on one GPU, so decode stays KV-parallel (sharded KV + broadcast query + split-KV combine) — D40 at decode chooses between KV-parallel-CP and KV-offload (K2), not a fall-back to single-GPU TP (which is infeasible by construction).number: the seq/HBM crossover and the cold-vs-hot context boundary on the trace — cites K7 (more-TP bound), K2 (KV-offload), K4 (chunked-prefill) · real: Ring Attention (Liu et al.), Llama-3 CP, DeepSpeed-Ulysses (the all-to-all CP variant) + URLs

Rule 11 — the two CP knobs: shard-assignment and comm-pattern

Selection-sweep stub (a) — shard assignment. contiguous (trivial; worst causal imbalance — last shard attends all, first none, ≈2×) → block-cyclic → zig-zag / striped (each GPU a low+high pair → balanced attended-token count) → load-aware. Two-sided cliff: contiguous = simple but ≈2× imbalance; zig-zag = balanced but scatter-gather index cost. Pick zig-zag for causal attention.number: imbalance factor per policy at CP=8 on the trace — cites K1 (attention cost) · real: Ring Attention striped, Llama-3 CP

Selection-sweep stub (b) — the communication pattern (added; previously only named, not opened). Ring P2P (each step sends the full KV block to the ring neighbour; per-GPU volume ≈ KV·(CP−1)/CP, BW-heavy but fully overlappable, works at any CP) → all-to-all / DeepSpeed-Ulysses (scatter the head dimension instead of streaming KV — lower total volume, latency-bound not BW-bound, but CP ≤ n_heads and very sensitive to the all-to-all-over-IB cliff) → 2D Ulysses×Ring hybrid (Ulysses inside the NVLink domain, Ring across domains — SoTA for >N_dom CP). Two-sided cliff: Ring = scales unboundedly but BW-bound (GQA shrinks KV → easier overlap, MHA harder); Ulysses = cheap comm but head-count-capped and IB-fragile. Pick Ulysses within one domain, Ring (or 2D) once CP must cross domains.number: ring vs all-to-all per-step volume at CP=8, MHA vs GQA, intra- vs inter-domain — cites K1 #collectives, K20 (all-to-all-over-IB cliff) · real: Ring Attention (Liu et al.), DeepSpeed-Ulysses, USP/2D-CP

4. Alternatives & When Each Wins

Stub (≥2, disqualifying constraint per alt — the D40 spine). (a) More TP instead of CP — disq: TP caps at N_dom (cite K7/D26) and shards the model not the sequence; wins only while the sequence fits at TP≤N_dom. (b) KV-offload/tier (K2) — disq: loses to CP when the long context is hot (attended every step → offload thrash); wins when the context is cold (reused, not attended every step). (c) Chunked-prefill-only (K4) — disq: doesn’t add capacity, only schedules; fails when the sequence doesn’t fit one GPU; wins when it fits but is merely slow to prefill. (d) RAG / retrieval to avoid long context — disq: application-layer, changes the task; out of the GPU-serving scope. 5. Failure Modes

Stub (cite, never inline-redefine). The CP group for that request is the fault domain — lose a CP rank and the whole sharded request is lost; recovery economics invert at long context: re-prefilling a 1M-token request is catastrophic (it was huge by definition), so the drop-vs-replay-vs-checkpoint crossover (K22/D23) shifts hard toward periodic CP-rank KV checkpointing / replication — the policy that is overkill for short requests is mandatory here (cite K22, gpu-failure-modes.html); contiguous assignment → causal-mask straggler (one shard ≈2× — llm-bottlenecks.html); ring comm > local attention → comm-bound (the overlap condition failed); preempting a CP’d request without coherent multi-rank swap → inconsistent KV.cites llm/canonicals/gpu-failure-modes.html (CP-group blast radius), llm/canonicals/llm-bottlenecks.html (causal-mask straggler, ring saturation), K22 · ACTION: seed/extend these canonical entries in this same change
