K19 — Reasoning-Model Serving (the bimodal-length mixture, not “long outputs”)

How do you build a scalable serving fleet for a model that emits a variable, unknown-at-admission number of hidden thinking tokens — ~100 (short CoT) to tens of thousands (long CoT) on the same model and endpoint — when the real problem is not length but the bimodal mixture under output-length uncertainty that breaks static capacity planning — and when test-time compute also scales in parallel (best-of-N / self-consistency / search), a second axis the page owns? This page reframes reasoning as test-time compute on two axes, derives why bimodal-unknown-length breaks static capacity, then D47 (output-length prediction & length-aware scheduling), the parallel-width fan-out (shared-prefix KV + adaptive-N + verifier), and the K19 half of D13 (drop reasoning_effort under overload).

SLIs This Page Reduces

SLIs this page reduces (the reverse index of K1 §3.12):
  • Goodput under the bimodal mixture (primary) — length-aware scheduling keeps a few long generators from pushing many short requests past SLO.
  • Batch-occupancy / effective concurrency — a 30k-token generator holds a slot ~150× longer than a 200-token one; predicting/separating length stops the convoy collapse.
  • Queue-depth / head-of-line — short/long pool separation + preempt-long stops long-CoT head-of-line blocking (cite K4).
  • Cost-shape ($/query) — a high-effort query costs ~50× a short one; the reasoning_effort knob is the cost-vs-quality dial (the dominant $ line). The width axis multiplies this by N (best-of-N / self-consistency) — adaptive-N is the width-cost dial.
  • Time-to-first-answer-token (TTFA), not TTFT — the user waits through queue + prefill + the entire thinking duration before the first visible token; this composite (not TTFT) is the reasoning latency SLO. Stated here so the SLI list is honest (the K19-specific reshape of K1’s TTFT).
  • Intra-request shared-prefix KV-hit (width axis) — the N parallel samples share one prompt prefix; the within-request CoW hit-rate is what keeps best-of-N from costing ×N prefill (K2 Role-1 substrate).

TL;DR. Short CoT (~hundreds of tokens) is just the decode-heavy baseline scenario 34 already assumes — the low anchor. The Reasoning-Serving Spine (stated up front, before §1)

  • The same model + endpoint emits 100 → 30k+ hidden thinking tokens, unknown at admission.
  • → this is not “long outputs” — it is test-time compute, scaled at serving time: the model spends GPU to think, and how much is decided per-request.
  • → test-time compute scales on two axes: sequential DEPTH (long CoT — the rest of this spine) and parallel WIDTH (best-of-N / self-consistency / tree-search). The page owns the serving of both; they compose (a deep request can also be wide).
  • Width problem: one logical request fans out into N samples ⇒ fleet load ×N; but the N all share the one prompt prefix.
  • → intra-request shared-prefix KV (the K2 Role-1 CoW substrate — cite, don’t redefine) + adaptive-N early-stop (stop when a majority / a verifier-pass emerges) + an optional verifier model (a 2nd model in the path — a K5-like colocation decision).
  • Depth problem: static capacity planning breaks — provision-for-mean under/over-provisions; batch-occupancy variance explodes (a 30k generator holds a slot ~150× longer), and per-step decode cost itself grows within one long trace (attention over a KV that grows to 30k).
  • → length prediction + short/long separation (D47).
  • Problem: a few long generators starve many short ones (head-of-line, K4).
  • → preempt long / cap the thinking budget; a mis-predicted-short that runs long is migrated, not allowed to starve (D47).
  • The reasoning_effort knob chooses where on the spectrum a request lands (rule 11); a budget-cap forces an answer (inject the stop-think token, decode the answer from the truncated CoT) — it does not merely truncate.
  • Under overload: drop reasoning_effort / fall back to a non-reasoning model — degrade quality of thinking before dropping requests (the K19 half of D13).
  • Economics: a high-effort query costs ~50× a short one; and the 30k thinking-trace KV is discarded at turn end (hidden thoughts are stripped from history) ⇒ it cannot be amortized across multi-turn — the dominant $ line and the cost-vs-quality knob.
  • Objective: meet the SLO across both axes at a controlled $/query — and the SLO is time-to-first-answer-token (queue + prefill + the whole thinking duration), not TTFT.

1. The Problem It Solves

Problem→Solution stub. Problem: a reasoning model emits a hidden chain-of-thought whose length is decided by the query at generation time — a few hundred tokens for an easy question, tens of thousands for a hard one — on the same endpoint; you cannot tell at admission which it will be, so every static-capacity assumption baked into ordinary serving (scenario 34) fails. The two axes of test-time compute — sequential depth vs parallel width [ADDED — gap surfaced in review; see banner note]

Problem→Solution stub (the reframe the rest of the page stands on). Problem: “reasoning serving” is silently scoped to sequential long CoT — but test-time compute is the act of spending more GPU at inference to get a better answer, and it scales on two orthogonal axes, only one of which the depth-spine addresses. Solution — name both, then thread depth through the rest of the page:
• Sequential DEPTH — one long autoregressive CoT (~100 → 30k tokens). This is the bimodal-length problem the rest of the page derives; it is a decode-count phenomenon.
• Parallel WIDTH — one logical request fans out into N independent samples (best-of-N, self-consistency / majority-vote, tree-of-thought / MCTS, verifier-guided search). This is a concurrency phenomenon: fleet load ×N, but the N samples all share the same prompt prefix.
• They compose — a request can be both deep and wide (self-consistency over N long CoTs is the worst case: ×N × 30k decode). Why bimodal-unknown-length breaks static capacity (derived — the page’s home)

Problem→Solution stub (the page’s opening derivation, sub-bulleted). Problem: standard capacity planning assumes a known s_out distribution ⇒ fleet = f(mean s_out). Solution, derived: reasoning’s s_out is bimodal (mode A ~200, mode B ~30k) and the mode is unknown at admission ⇒ (a) provisioning for the mean under-provisions when the long mode spikes and over-provisions otherwise; (b) batch-occupancy variance explodes — a slot held by a 30k-token generator is occupied ~150× longer than a 200-token one; (b′) per-step decode cost also grows within one long trace — at thinking-token n the attention reads a KV of length n, so the decode step at token 30 000 is materially slower than at token 200 (cost ∝ context) ⇒ a long generator is not just a long-lived slot, it is a worsening one, and the TPOT SLO must be defined per-position, not as a single number ⇒ effective concurrency collapses unpredictably (the K4 convoy, driven by output not prompt) ⇒ you must schedule on predicted/observed length, not a static mean (cite K1 for the decode-step cost and its context-growth term, K4 for the convoy). Also state the inversion: decode:prefill flips extremely decode-heavy ⇒ the fleet is almost all decode GPUs and KV grows for the whole trace → K2 tiering dominates at the long end (negligible at the short end — the same fleet serves both).number: the bimodal s_out histogram + the ~150× occupancy ratio + the decode-step cost at token 200 vs 30k (the in-trace growth) + the provision-for-mean error band on the trace — cites K1 #decode, K4 (convoy) · diagram: the bimodal s_out distribution with the “mean” arrow landing in the empty valley between the two modes (the visual of why mean-planning fails)

Deep dive D47 — output-length prediction & length-aware scheduling (K19’s core hard problem)

Deep-dive stub. Problem: you can’t tell at admission whether a request is 200 or 30k tokens, so FCFS lets long generators head-of-line and capacity is unplannable. Solution: predict length (prompt-feature classifier, the reasoning_effort hint as a coarse prior, or the model’s own early-stop signal) → schedule by predicted length: separate short/long pools or shortest-remaining-predicted-first, capped by a thinking-budget. The mis-prediction failure mode + fix: a predicted-short request that runs long is migrated/preempted to the long pool (never allowed to starve the short pool), and the predictor is monitored for drift. Width-axis deep dive — serving parallel test-time compute (best-of-N / self-consistency / verifier-guided search) [ADDED — candidate numbered deep dive; see banner note]

Deep-dive stub (the width axis the depth-spine omits). Problem: a single logical request becomes N generations (N = 8–256). Naively that is ×N fleet load and ×N prefill of the (identical) prompt — the cost the depth-spine never accounts for. Solution, Problem→Solution each:
• Shared-prefix KV across the N samples — the N samples are identical until the first sampled token diverges ⇒ one prefill + one KV copy, refcounted, copy-on-write at divergence (the K2 Role-1 block-table / CoW substrate — cite, don’t redefine; K19 owns the ×N serving consequence, K2 owns the mechanism). • Adaptive N (early-stop the fan-out) — self-consistency can stop sampling once a majority is unambiguous; verifier-guided search prunes losing branches ⇒ N is itself a scheduled, data-dependent knob, not a constant — the width analog of D47’s length prediction.
• The verifier / reward model is a 2nd model in the path — a PRM/ORM scores candidates many times per request; colocate-with-generator vs separate verifier pool is a K5/D35-pattern decision (different roofline: the verifier is short-prefill-of-a-trace, the generator is long-decode) — reuse that inequality, don’t re-derive it.

4. Alternatives & When Each Wins

Stub (≥2, disqualifying constraint per alt). (a) Provision for the mean output length (treat it like scenario 34) — disq: the mean lands in the empty valley between the two modes → chronically wrong; wins only if the workload is actually unimodal (i.e. not reasoning). (b) One hard thinking-budget cap for everyone — disq: truncates the hard problems the model exists to solve; wins for cost-capped tiers where quality ceiling is acceptable. (c) Separate models / endpoints for short vs long — disq: doubles the fleet + the client must choose; wins when the two modes are genuinely different products. (d) RL/training to make CoT shorter — disq: training-side, changes the model; out of scope — K19 serves the trace it is given; flag the boundary, do not cover. (e) Serve only the depth axis (treat parallel test-time compute as out of scope) — disq: best-of-N / self-consistency / verifier search is a first-class accuracy lever a reasoning product commonly turns on, and serving it (fan-out, shared-prefix KV, adaptive-N, verifier) has GPU consequences nothing else owns; wins only for a product that never samples N>1 (then the width sections are inert, not wrong).number: the crossover disqualifying each (mean-error band for (a); quality truncation for (b); 2× fleet for (c); the depth-vs-width budget split for (e)) — cites K1, K4, K5/D35 · contrast 34 (the static-capacity assumption K19 breaks)

5. Failure Modes

Stub (cite, never inline-redefine). Long-CoT head-of-line — a 30k generator blocks a queue of short requests → goodput collapse while GPU “busy”; first move: short/long pools + preempt-long (llm-bottlenecks.html#head-of-line). Predictor drift — D47 classifier degrades under workload shift → mis-routes long into the short pool; first move: drift monitor + migrate-not-starve (llm-bottlenecks.html). Provision-for-mean miss — long-mode spike under a mean-sized fleet → sustained SLO breach; first move: scale on observed length mix + the D13 effort-drop (gpu-inference-cost-shapes.html). Runaway non-terminating thinking — a request never emits stop → one slot burned indefinitely; first move: thinking-budget cap + cancellation (cite K4 D21; gpu-failure-modes.html). Long-end KV exhaustion — KV grows for the whole 30k trace → HBM pressure; first move: K2 tier ladder (llm-bottlenecks.html — KV exhaustion). Reasoning-chain nondeterminism amplification — one batch-nondeterministic token early (non-associative FP reduction order changing with batch, K17/D10) sends the entire 30k chain down a different path → a different final answer; reasoning eval/repro is therefore far more fragile than short-output (error compounds over the chain, it does not average out); first move: batch-invariant kernels / pinned batch on eval-sensitive paths (cite K17/D10, don’t re-derive). Agentic tool-call slot stall — a reasoning request that calls a tool mid-thinking holds its KV/slot idle for the external wait (seconds) → wasted GPU-hours at scale; first move: the KV-retention-vs-evict decision during the stall (keep warm if the wait < recompute cost, else evict via the K4 preemption path) — tool orchestration is out of scope, this serving consequence is not (cite K4 preemption/D21; gpu-inference-cost-shapes.html). Width fan-out correlated release + verifier overload — the N samples of a best-of-N request finish near-simultaneously (a correlated slot-release spike, unlike depth’s staggered finishes) and the verifier model is hit N× per request → bursty load on the verifier pool; first move: stagger/cap N admission + size the verifier pool to the fan-out rate (cite K5/D35; llm-bottlenecks.html).cites llm/canonicals/llm-bottlenecks.html (head-of-line, predictor drift, KV exhaustion, width correlated-release), llm/canonicals/gpu-failure-modes.html (runaway thinking, nondeterminism amplification), llm/canonicals/gpu-inference-cost-shapes.html (provision-for-mean miss, tool-stall waste); cross-links K17/D10, K4/D21, K5/D35 · ACTION: seed/extend these canonical entries in this same change
