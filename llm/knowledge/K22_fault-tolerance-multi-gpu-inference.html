K22 — Fault Tolerance in Multi-GPU Inference (blast radius = the parallelism group)

How do you keep a frontier-scale fleet correct and available — when a GPU in a TP=8 / PP / NVL72 group kills every in-flight request on that group in one of three classes (fail-stop, fail-slow, fail-corrupt), this is a daily and correlated event not a rare independent one, and the worst failures are silent (wrong/slow, no crash) and hidden by CUDA graphs? This page derives the fault taxonomy + blast-radius-per-parallelism-axis + failure-domain-aware placement + predictive drain, then D23 (the group as the fault domain + guarded in-flight recovery) and D50 (silent-data-corruption detection, incl. under CUDA graphs).

The Fault-Tolerance Spine (stated up front, before §1)

  • A GPU in a TP/PP/EP/NVL72 group fails — in one of three classes: fail-stop (dead/hung), fail-slow (degraded-but-alive straggler that gates every collective), fail-corrupt (SDC — plausible-wrong tokens, no crash). Each has a different detection signal and recovery.
  • → every in-flight request on the group dies (fail-stop/corrupt) or silently misses SLO (fail-slow).
  • → at frontier scale this is a daily event — and correlated, not independent: a rack/power/cooling/network/firmware/host event takes a whole physical zone, so the MTBF-independence assumption (and the hot-standby pool sized on it) is false.
  • → blast radius = the group size (= K20’s NVLink domain); the physical fault unit may be GPU / host-of-8 / rack / domain.
  • → minimize it (D61): a DP replica is the smallest group (DP is the resilience-friendliest axis — K10), and failure-domain anti-affinity spreads the group + its standby across independent power/cooling/network/firmware domains (so a correlated event can’t take all replicas and the standby together).
  • → predictive drain (D62): rising ECC-correctable / thermal / NVLink-CRC counters predict the hard fault ⇒ drain before it — converting an unplanned all-in-flight loss into a planned zero-loss drain (the same machinery as L7’s planned deploy-drain and D58’s scale-in).
  • → detect fast — fail-stop (NCCL watchdog/liveness), fail-slow (D60) (per-rank step-time variance, not liveness; the degraded-compute sibling of D57’s degraded link), fail-corrupt (D50) (golden-vector probe; no crash, hidden by CUDA graphs).
  • → evict the whole group (not the GPU).
  • → recover in-flight: drop / replay-from-prompt / KV-replicate (D23).
  • → guard the recovery (D63): rate-limit + backoff + retry-budget (mass replay = a prefill thundering herd) and quarantine the poison request (a request that crashed the group will crash the standby — attribute fault-vs-bad-input before blaming hardware, or the fault walks the fleet).
  • → spare / hot-standby reformation — health-gate the reformed group (golden-vector + weight-hash + warmup) before live traffic; flap-damp re-admission of an evicted GPU.
  • → under a partial fleet, degrade by the K4/D13 contract (fault-triggered).
  • And: NVL72 enlarges the domain ~9× ⇒ ~9× blast radius per fault — resilience now trades against K20/D48 parallelism efficiency.
  • Objective: bound availability + correctness under daily, correlated faults at an acceptable redundancy cost.

1. The Problem It Solves

Problem→Solution stub. Problem: a frontier model is served by a tightly-coupled parallelism group (TP×PP×EP, often a whole NVLink domain); a faulted GPU takes the entire group’s in-flight requests with it, in one of three classes — fail-stop (dead/hung), fail-slow (degraded-but-alive straggler that gates every collective, no crash), fail-corrupt (SDC: plausible-wrong tokens, no crash, hidden by CUDA graphs). At thousands of GPUs the MTBF math makes this a daily event — and the faults are correlated, not independent (a rack/power/cooling/network/firmware/host event takes a whole physical zone, including potentially the hot-standby pool sized on the independence assumption). Solution (what the page builds): the blast-radius model per parallelism axis, failure-domain-aware placement, predictive drain ahead of the hard fault, fast detection of all three classes, whole-group eviction, a guarded in-flight recovery policy (retry-budget + poison-request quarantine), and health-gated hot-standby reformation — bounding availability and correctness under daily, correlated faults.number: MTBF at the trace fleet size → faults/day (independent vs correlated); GPU-seconds of in-flight work lost per fault on a 30k-token reasoning request — cites K20 (domain = blast radius), K1 (work-lost) · canonical: gpu-failure-modes.html (GPU-group blast radius; fail-slow straggler; correlated-domain fault — ACTION: extend these entries in this same change)

2. The Mechanism — the Tokenized-Prompt Walkthrough

The three fault classes — fail-stop / fail-slow / fail-corrupt (the taxonomy everything else branches on)

Problem→Solution stub (the taxonomy, sub-bulleted — precedes blast radius because detection & recovery differ per class). Problem: “a GPU failed” is three different problems with three different signals; treating it as one (the common mistake) makes you blind to the two that don’t crash. Solution, the classes: (1) fail-stop — dead process or hung collective; signal = NCCL watchdog / liveness; the whole group’s in-flight work is lost. (2) fail-slow (the missing class) — alive and correct but degraded (thermal throttle, NVLink renegotiated to lower BW, ECC-correctable retry storm, a bad NIC retransmitting); in a synchronous group the slowest rank gates every collective, so one slow GPU silently tanks the whole group’s TPOT while every liveness check and golden-vector passes; signal = per-rank step-time variance, response = eject the slow rank, don’t wait for it to die. (3) fail-corrupt (SDC) — alive, fast, wrong; plausible-looking bad tokens, no crash, hidden by CUDA graphs; signal = golden-vector probe / weight-hash (D50). The walkthrough on the canonical trace — a GPU dies / NaNs mid-generation, with vs without fault-handling

Problem→Solution stub (hard requirement 6 — reader’s first contact). Canonical trace T1 "I like cat, why?" is mid-generation (decoding token 47 of a long answer) on a TP=8 group. Hard failure, without handling: one GPU dies → the next all-reduce hangs (NCCL stuck collective) → all 8 GPUs’ in-flight requests are stalled/dead, the whole group down. Slow failure (fail-slow), without handling: one GPU thermal-throttles → it is the last to arrive at every all-reduce → T1’s TPOT silently doubles and the whole group’s SLO slips — nothing crashes, every liveness check is green, so a liveness-only design never notices. Silent failure (SDC), without handling: the GPU keeps running but produces plausible-wrong numbers → the request keeps emitting confident garbage with no crash, and a captured CUDA graph means there is no per-step hook to notice. With handling: the predictive-drain path catches a rising ECC/thermal counter and drains the group with zero in-flight loss; failing that, NCCL watchdog / per-rank step-time outlier / golden-vector probe flags the bad GPU → evict the whole group → guarded recovery of T1 (replay-from-prompt or from a KV checkpoint, under a retry-budget, poison-checked) → a health-gated hot-standby reforms the group. Blast radius per parallelism axis (derived — the page’s home)

Problem→Solution stub (the page’s opening derivation, sub-bulleted). Problem: “a GPU failed” has a different meaning per parallelism axis, and that meaning drives every recovery decision. Solution, derived: a request’s state lives across its parallelism group; lose one GPU → the whole group’s in-flight work is lost — (a) TP = all-or-nothing (every layer’s all-reduce needs all members ⇒ 1 down = model incomplete = all group requests die); (b) PP = a stage (the pipeline stalls; that stage’s in-flight micro-batches are lost); (c) EP = an expert shard (its tokens fail / must reroute); (d) DP = one replica only (smallest blast radius ⇒ DP is the resilience-friendliest axis, cite K10) ⇒ blast radius = group size = K20’s NVLink-domain size; NVL72 → ~9× domains → ~9× blast radius per fault (the K20/D48 trade). Two minimization levers, not one: the logical blast radius is the parallelism group (smallest at DP) but the physical fault unit is whatever a correlated event takes (host-of-8 / rack / power / cooling / network / firmware) — which can be larger than the logical group and can swallow the hot-standby; so minimizing blast radius is (a) smallest logical group (DP, K10) and (b) failure-domain anti-affinity placement spreading the group + its standby across independent physical domains (the §3 placement subsection). Deep dive D61 — failure-domain anti-affinity placement: the second blast-radius lever (correlated faults break independence)

Problem→Solution stub. Problem: the MTBF/redundancy math assumes independent faults, but real frontier faults are correlated — a rack power event, a cooling loss, a spine switch, a host/PCIe failure, or a firmware/driver rollout takes a whole physical zone at once; if a model’s replicas (or a replica and its hot-standby) share that zone, “N-way redundancy” provides zero. Solution: failure-domain anti-affinity — spread a model’s parallelism group members and its standby across independent power / cooling / network / firmware domains (the AZ-anti-affinity analog), and size the standby pool against the correlated fault unit, not the per-GPU MTBF. Crossover: tighter spread → lower correlated blast radius but worse collective locality (a TP group spread across racks pays the K20 IB tax) → spread only above the NVLink-domain boundary; co-locate within it. Rule-of-two: K20 owns the locality cost, K22 owns the anti-affinity constraint.number: independent vs correlated faults/day on the trace fleet; redundancy needed for a rack-correlated event vs per-GPU — cites K20 (locality cost of spread), K10 (DP replicas are the spread unit) · canonical: gpu-failure-modes.html (correlated-domain fault — ACTION: seed) · real: K8s topology-spread / anti-affinity, slurm topology-aware placement + URLs

Deep dive D62 — predictive drain: converting the unplanned fault into a planned, zero-loss drain (the L7/D58 unifier)

Problem→Solution stub. Problem: a purely reactive design loses all of a group’s in-flight work on every hard fault — yet most hard faults are preceded by a signature (rising ECC-correctable counts, climbing temperature, NVLink-CRC/replay, XID precursors). Reacting after the crash throws away a free warning. Solution: watch the leading counters (DCGM) and, on threshold, proactively drain the group — stop admitting, let in-flight requests finish (or migrate the longest), then evict and replace — turning an unplanned all-in-flight loss into a planned zero-loss drain. This is the same machinery as L7’s planned deploy-drain (the unplanned↔planned unifier; rule-of-two with D23 — D23 owns the post-fault recovery, this owns the pre-fault avoidance). Crossover (the threshold knob): drain too eagerly → healthy GPUs wasted + churn; too late → the fault beats the drain → pick the counter threshold where `P(hard fault | signal) × in-flight-work-at-risk > drain-cost`.number: fraction of hard faults with a usable precursor (cite Meta/Google SDC-fleet papers); GPU-seconds saved per predicted fault on a 30k-token in-flight request — cites K21 (the replacement reload), L7 (planned-drain twin) · real: DCGM health policy + drain, NVIDIA Resiliency Extension, node-problem-detector + cordon/drain + URLs

Deep dive D23 — the parallelism group as the fault domain + in-flight recovery

Deep-dive stub. Problem: a TP×PP×EP group dies together; in-flight decode state is lost; detection of a cleanly dead process is easy, of a hung collective less so. Solution: NCCL watchdog/timeout (stuck collective) + process liveness → evict the whole group, not the GPU; an in-flight recovery policy (drop-and-retry vs replay-from-prompt vs KV-replication — cost trade); fast group reformation from a hot-standby pool. Two correctness sub-problems the policy hides: (a) idempotency on replay — replay-from-prompt re-fires any side effects (tool/function calls, external writes) the request already made; needs a request idempotency key + dedup so a retried agentic request doesn’t double-act; and a partially-streamed response must choose resume from token N (requires deterministic continuation — fights D10 batch-invariance) vs restart from 0 (client sees duplicate tokens). (b) KV-replication consistency — KV mutates every decode step, so “replicate the KV” is a sync-vs-async/staleness model: resume from the last consistent checkpoint and re-decode the bounded delta, not from a torn mid-step copy. Crossover: KV-replication (continuous BW/storage + a staleness bound) vs replay-from-prompt (re-prefill cost + idempotency machinery) vs drop (lost work + retry latency) — pick per request value × length.number: a 16-GPU TP×PP group, MTBF → faults/day; recovery cost per policy on a 30k-token in-flight reasoning request on the trace; resume-delta size vs checkpoint interval — cites K2/D4 (KV-checkpoint uses the block table), K17/D10 (deterministic resume fights batch-invariance), K20 (group eviction scope) · real: NCCL async-error-handling, fast-restart inference frameworks + URLs

Deep dive D50 — silent data corruption: detecting a bad GPU & weight corruption, incl. under CUDA graphs (user-requested)

Deep-dive stub (the worst fault class). Why it’s the worst & why CUDA graphs worsen it: SDC produces plausible-looking wrong numbers — no exception, no dead process; a captured CUDA graph (K17) replays a fixed kernel sequence with no per-step host/Python hook → the natural validation point is gone ⇒ SDC under graphs is harder to catch than eager. Rule of two: K17 owns the graph machinery; K22 owns detecting corruption through it. Identify a bad GPU (layered): (1) hardware counters (ECC correctable/uncorrectable, XID, NVLink CRC/replay, thermal — DCGM); (2) numeric guards (periodic NaN/Inf scan, output entropy/perplexity sanity); (3) active golden-vector probe (known input→known output canary per GPU, bit-compare — the only reliable catch for “plausible wrong”); (4) redundant cross-check (sample-replay on two GPUs, audit-sampled). Identify weight corruption: a per-shard hash manifest (safetensors + sha256) verified on load and on a timer (catches HBM bit-rot); HBM ECC; the same golden-vector probe. On detection: evict the group → reload weights from the verified artifact (cite K21 fast reload) → recover in-flight (D23). Deep dive D60 — fail-slow GPU: degraded-but-correct compute (the D50/D57 sibling)

Deep-dive stub (the missing third fault class — sibling of D50’s wrong numbers and D57’s degraded link; this is degraded GPU compute). Problem: a GPU thermal/clock-cap-throttles, ECC-correctable-retry-storms, or has a slow SM — it is alive and numerically correct, just slow; in a synchronous group the slowest rank gates every collective ⇒ the whole group’s TPOT silently slips while liveness is green, the golden-vector probe is green (numbers are right), and D57’s link counters are clean (the link is fine) — a liveness-only or correctness-only design never sees it. Solution: detect by per-rank step-time variance / straggler z-score (the K4/D33 per-step time is the signal, not liveness) + per-GPU clock / thermal / ECC-correctable counters (DCGM); eject the slow rank → evict→recover (D23). A throttled GPU may self-recover after cooling, so the policy is quarantine-then-reprobe before RMA (vs D50/D57 which evict hard). Deep dive D63 — guarded recovery: retry amplification & the poison-request crash-loop

Problem→Solution stub (two real crash amplifiers naive recovery creates). Problem A — retry amplification: a domain fault drops thousands of in-flight requests; if they all replay-from-prompt at once, the surviving fleet eats a synchronized prefill thundering herd that can cascade into fleet-wide overload — the recovery causes a second, larger outage. Problem B — the poison-request crash-loop: if the thing that killed the group was a request (a kernel-crashing shape, an OOM context, a bad-input edge case), blindly replaying it onto the hot-standby kills the standby too → the fault walks the fleet. Solution: (A) rate-limited, jittered, backed-off replay under a global retry-budget (composes with the K4/D13 admission contract); (B) fault-vs-bad-input attribution — a per-request crash counter + circuit-breaker: a request that has crashed a group N times is quarantined (failed back to the client) before hardware is blamed, so one bad request can’t take a fleet. Crossover: retry-budget width (too tight drops recoverable work, too wide re-creates the herd); the quarantine N (too low fails transient-fault-victim requests, too high lets a poison request burn N groups).number: replay QPS spike vs steady-state prefill capacity on the trace; groups burned at quarantine N=1..k — cites K4/D13 (admission contract), K1 (prefill cost of the herd) · canonical: llm-bottlenecks.html (recovery storm), gpu-failure-modes.html (poison-request crash-loop — ACTION: seed) · real: retry-budget/circuit-breaker (Envoy/gRPC), request-level circuit-breakers + URLs

Reform health-gating & flap damping — don’t route onto a still-bad group

Problem→Solution stub. Problem: a reformed group (or a GPU that recovered from a transient blip) may still be cold, mis-loaded, weight-corrupted, or about to fault again; routing live traffic onto it immediately re-creates the outage, and an oscillating GPU that auto-re-admits causes a flap storm. Solution: a re-admission gate — before the reformed group takes live traffic it must pass the golden-vector probe + weight-hash verification + a synthetic warmup request (and graph re-capture, cite K17/K21); an evicted GPU is flap-damped (exponential cool-down before it is eligible again, hard-fail after K flaps → RMA, out of scope). Crossover: gate strictness vs reform MTTR (a heavier gate is safer but extends the unavailability window); flap cool-down length vs churn.number: false-readmit rate vs gate depth; MTTR added per gate stage on the trace — cites K21 (warmup/reload), K17 (graph re-capture), D50 (the probe it reuses) · real: readiness-gate / health-check before in-rotation, flap-damping (BGP-style) + URLs

4. Alternatives & When Each Wins

Stub (≥2, disqualifying constraint per alt). (a) No fault handling (let it crash, restart cold) — disq: a daily fault → daily group outage + cold-start tax; wins never at frontier scale (only single-GPU toy). (b) Full request-state replication / hot-standby shadow for everything — disq: ~2× fleet cost; wins only for the highest-value requests where instant failover is mandatory. (c) Bigger NVLink domain for parallelism efficiency — disq: ~9× blast radius per fault (the K20/D48 trade); wins when fault rate is low and efficiency dominates — an explicit resilience-vs-efficiency choice. (d) Reactive-only (no predictive drain) — disq: converts every predictable degradation (rising ECC/thermal) into a full all-in-flight loss; wins only when faults have no usable precursor (rare — cite the SDC-fleet papers). (e) Locality-optimal packing, blast-radius-oblivious — disq: maximizes collective BW but co-locates replicas + standby in one rack ⇒ one correlated event = total outage; wins only when the fabric forces it and correlated-fault rate is provably negligible. (f) Training-style global checkpoint/restart — disq: serving is per-request stateless + in-flight, a different problem; out of scope — flag the boundary, do not cover. (g) The generic DR runbook — cite cross-cutting.html#dr, do not duplicate.number: the crossover disqualifying each (faults/day for (a); ~2× fleet for (b); 9× blast radius for (c)) — cites K20/D48, K10 · contrast L1 (frontier daily faults), L7 (planned drain)

5. Failure Modes

Stub (cite, never inline-redefine). GPU-group blast radius — one GPU lost = the whole TP/PP/CP group down; first move: minimize group size (DP-friendly) + whole-group eviction (gpu-failure-modes.html — GPU-group blast radius). Silent data corruption — plausible-wrong tokens shipped, no crash, hidden by graphs; first move: golden-vector probe + weight hash manifest (gpu-failure-modes.html — SDC). Fail-slow straggler — a degraded-but-alive GPU gates every collective → the whole group’s TPOT silently slips, no crash, liveness green; first move: per-rank step-time-variance detection → eject the slow rank (gpu-failure-modes.html — fail-slow straggler). Stuck-collective hang — a dead member never returns from all-reduce → the group hangs “not slow, frozen”; first move: NCCL watchdog/timeout → evict (llm-bottlenecks.html — interconnect / collective stall). Correlated-domain fault — a rack/power/cooling/network/firmware/host event takes a whole zone (possibly incl. the standby) → N-way redundancy = 0; first move: failure-domain anti-affinity placement + standby sized to the correlated unit (gpu-failure-modes.html — correlated-domain fault). KV-transfer failure mid-handoff — a block lost/evicted mid-disaggregated-handoff → inconsistent recovery; first move: idempotent transfer + checkpoint (gpu-failure-modes.html — KV-transfer failure). Retry-amplification storm — a domain fault triggers synchronized mass replay/reform → a prefill thundering herd → secondary fleet-wide overload; first move: rate-limit + backoff + retry-budget + the K4/D13 partial-fleet contract (llm-bottlenecks.html — recovery storm). Poison-request crash-loop — the request that crashed the group crashes the standby on replay → the fault walks the fleet; first move: per-request crash counter + circuit-breaker/quarantine before blaming hardware (gpu-failure-modes.html — poison-request crash-loop).
