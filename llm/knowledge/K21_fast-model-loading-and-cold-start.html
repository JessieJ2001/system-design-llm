K21 — Fast Model Loading & Cold Start (scale-to-zero is worthless if cold start blows the SLO)

How do you make scale-to-zero / autoscaling actually usable — when a 140 GB model takes tens of seconds to reach HBM and a cold start blows the SLO — by decomposing where the cold-start time goes into a budget, attacking the dominant leg, and (fastest) restoring a fully-warmed snapshot instead of replaying the whole pipeline? This page derives the cold-start budget, then D49 (warm snapshot/restore) and the K21 half of D12 (cold-start time is the autoscale headroom floor).

SLIs This Page Reduces

SLIs this page reduces (the reverse index of K1 §3.12):
  • Cold-start time (primary) — the first-token latency of a freshly-spawned replica; the budget every leg here attacks.
  • TTFT on the first request after scale-up — a blown cold start is a blown TTFT for that request (and every request queued behind it).
  • Cost-shape ($/mo) — a small cold-start budget is what makes scale-to-zero economically usable instead of paying keep-warm idle GPU-hours.
  • Feeds K15’s headroom floor — min_headroom = cold_start × peak_ramp_rate and the spot-reclaim drain window are exactly this budget (the D12 K21-half).

What This Page Is (and Is Not)

TL;DR. Decompose cold start into legs (object/image pull → host→HBM copy → communicator-init → CUDA-context + compile/graph warm-up → KV-pool alloc + profiling forward → first forward); the sum is the cold-start budget — and at frontier scale it is not one sum but max over ranks + barrier for a sharded replica; attack the dominant leg (usually the pull, the HBM copy, or compile-when-cold); the fastest path is restoring a fully-warmed snapshot (skip the whole pipeline, ~1 s) at the price of a large, version-brittle artifact; this budget is exactly K15’s headroom floor.

The Cold-Start Spine (stated up front, before §1)

  • A 140 GB model takes tens of seconds to load to HBM.
  • → scale-to-zero / autoscaling are worthless if a cold start blows the SLO.
  • → decompose where the time goes: object/image pull → host→HBM copy → NCCL communicator init → CUDA-context + compile/graph warm-up (the K17 leg) → KV-pool alloc + profiling forward → first forward — the sum is the cold-start budget.
  • → at scale this is not a single sum: a TP×PP replica is cold until the slowest rank is warm (a barrier); per-rank shard pull is N-way object-store pull (intra-replica thundering herd) vs rank-0 pull + NVLink/IB broadcast.
  • → attack each leg: zero-copy mmap, GPUDirect-Storage (skip the host bounce), weight streaming, local NVMe/host-RAM cache, persisted compile-artifact cache, weights baked into the machine image.
  • → the fastest path: restore a warm snapshot (skip the whole pipeline, ~1 s) (D49).
  • Problem: the snapshot is large + driver/GPU-coupled + captured-graph-invalidation-brittle (D49).
  • And: cold-start time is K15’s autoscale headroom floor and the spot-reclaim drain window (the D12 K21-half).
  • Objective: the mitigation point whose cold-start ≤ the SLO/headroom budget at acceptable cost & portability (the rule-11 endpoint = the scale-to-zero↔keep-warm crossover).

1. The Problem It Solves

Problem→Solution stub. Problem: every cost lever in the curriculum — scale-to-zero, spot, autoscaling — assumes a replica can come online fast; but a 140 GB model takes tens of seconds to pull, copy to HBM, and warm up, so the freshly-spawned replica is useless exactly when it is needed and the first request (plus its queue) blows the SLO. 2. The Mechanism — the Tokenized-Prompt Walkthrough

Where cold start lives — the replica lifecycle before the first forward (not a layer)

Problem→Solution stub (architecture placement, hard req 6). Problem: the reader needs to see that cold start is everything before token 1 of a fresh replica. Solution: it is the replica lifecycle — spawn → pull weights (object/image) → copy to HBM → form NCCL communicators (if sharded) → init CUDA context → compile / capture graphs / autotune (the K17 warm-up leg) → alloc the KV pool + profiling forward → then the first token. No layer changes; what changes is how fast a new replica — possibly a whole TP×PP group, gated on its slowest rank — reaches its first token. Where the cold-start time goes — the budget derived as a sum of legs (the page’s home)

Problem→Solution stub (the page’s opening derivation, sub-bulleted). Problem: “cold start is slow” is not actionable; you cannot attack what you have not decomposed. Solution, derived: the budget is the sum of legs, each = bytes/BW or a fixed cost — (a) object/image pull (S3→host, ~GB/s; the weights may ship inside the container image → the image-pull leg); (b) host→HBM copy (PCIe/NVLink-bound; pageable→pinned doubles it — pinned + double-buffered, or GPUDirect-Storage NVMe→HBM to skip the host bounce); (c) NCCL/communicator init (TP/PP/EP communicator formation + topology discovery + collective warm-up — a fixed cost that grows with group size; absent on one GPU); (d) CUDA-context init + compile / kernel-autotune / graph capture (the K17 warm-up; the first-ever torch.compile/TensorRT engine build is minutes, not seconds, unless a persisted compile-artifact cache is warm); (e) KV-pool alloc + profiling forward (the big cudaMalloc of the paged-KV arena + the max-batch profiling run that sizes it — e.g. vLLM determine_num_available_blocks); (f) first forward. Sum = the cold-start budget; derive the trace number (≈ tens of s for 140 GB; minutes if compile is cold) and which leg dominates (usually the pull or the HBM copy, or compile when its cache is cold) — that is the leg to attack, and this budget is exactly K15’s minimum-headroom input (cite K1 for the per-leg BW).number: per-leg seconds for a 140 GB model on the trace, summed → the dominant leg → K15’s min-headroom input; +the compile-cold → minutes case — cites K1 (BW per leg), K17 (warm-up/compile leg) · diagram: a stacked-bar budget, each leg sized to its trace seconds, the dominant leg highlighted

At scale the budget is not a single sum — distributed (parallel-replica) cold start

Problem→Solution stub (the budget-shape correction at frontier scale — Must-cover, K21 scope; not a new D-number). Problem: a frontier replica is sharded across a TP×PP(×EP) group, so the single-GPU sum above is wrong: the replica is cold until the slowest rank finishes → cold-start = max over ranks(Σ legs) + communicator-init + barrier sync, not one node’s sum — one straggler rank stalls the whole replica. Solution (the design crossover this surfaces): per-rank shard pull is a real choice — (i) N-way object-store pull (every rank pulls its shard from S3 → N× aggregate egress, intra-replica thundering herd, S3/registry throttling) vs (ii) rank-0 pull + NVLink/IB broadcast tree (one egress, then a fast in-fabric broadcast at NVLink BW); plus communicator-init is its own fixed leg that scales with group size. Rule of two: K21 owns the cold-start budget shape; the parallelism degrees are K6/K7/K8’s, the fabric BW is K20’s — cite, don’t re-derive.number: per-rank shard bytes = total/N (parallel pull helps) but cold-start = max-rank + comm-init; N-way-S3 egress vs rank-0+NVLink-broadcast crossover on the trace — cites K1 (BW), K20 (NVLink/IB BW for the broadcast), K6/K7/K8 (the group) · would be the formal-deep-dive home L1 (400B MoE TP×PP×EP replica) / L5 (the cold-start scenario) if promoted to a D-number · diagram: per-rank lifelines with the slowest-rank barrier marked; N-way-pull vs rank-0-broadcast side by side

Deep dive D49 — warm snapshot/restore of a GPU process

Deep-dive stub. Problem: even with streaming, first-token after a cold start is seconds (full pipeline + graph warm-up) → scale-to-zero is unusable under a tight SLO. Solution: snapshot a fully-warmed process (weights in HBM, CUDA context, captured CUDA graphs) and restore in ~1 s. Pitfalls, derived: snapshot size ≈ the full HBM footprint (storage + restore-BW cost); CUDA-context / driver-version coupling (a snapshot is tied to the GPU model + driver — a portability constraint); captured-graph invalidation on any shape/version change. Crossover: snapshot-restore (fast; large, version-brittle artifact) vs weight-streaming (slower first token; no artifact, portable) — pick by SLO tightness × deploy cadence.number: ~30 s full pipeline vs ~1 s restore on the trace; snapshot size = HBM footprint; the SLO×cadence crossover — cites K1 (HBM footprint/BW), K17 (graph invalidation) · real: ServerlessLLM, vLLM sleep-mode, CRIU-CUDA, Run:ai + URLs

Deep dive D12 (K21 half) — cold-start time IS the autoscale budget (co-host K15)

Deep-dive stub. Problem: the autoscaler must add capacity before the SLO breaches, but it cannot add it faster than a cold start. Solution (K21’s owned half): cold-start time sets K15’s min_headroom = cold_start × peak_ramp_rate and the spot-reclaim drain window (K22/L5) — shrink the budget (snapshot/cache) and the required headroom (and idle cost) shrinks with it. 4. Alternatives & When Each Wins

Stub (≥2, disqualifying constraint per alt). (a) Keep every replica warm (never scale to zero) — disq: pays idle GPU-hours 24/7; wins when traffic is steady and the SLO is unforgiving and a snapshot is impractical. (b) Naive serial load on every scale-up — disq: tens of seconds → the autoscaler is structurally too slow (K15); wins never for GPU LLM (only tiny models where load « SLO). (c) Smaller / quantized model to shrink the pull+copy — disq: changes the model / accuracy (K14 gate), and a dequant/repack-on-load step (AWQ/GPTQ repack, FP8 scale compute) partially offsets the pull saving; wins as a complement (the cascade reaches cold start), not a substitute. (d) A faster artifact registry / CDN — disq: that is distribution infra, not the GPU cold-start path; out of scope — name it, flag the boundary, do not cover. (e) Multi-component stacks (target + draft K13; encoder K12) — cold-start = max/sum over components, sized differently; compose, don’t re-derive — cite K13/K12.number: the crossover disqualifying each (idle-$ for (a); load-vs-SLO for (b); accuracy delta minus repack-cost for (c); component-max for (e)) — cites K1, K15 (autoscaler), K14 (quant + repack-on-load counter-cost), K13/K12 (multi-component) · contrast L5 (dedicated home)

5. Failure Modes

Stub (cite, never inline-redefine). Cold-start > SLO on scale-up — the scaled-up replica is useless when it arrives → sustained TTFT breach through the ramp; first move: shrink the budget (snapshot/cache) + size K15 headroom to it (gpu-inference-cost-shapes.html). Snapshot version skew — D49: a snapshot restored on a different driver/GPU/shape → broken graphs or wrong kernels; first move: pin driver+GPU+shape in the snapshot key, fall back to streaming (gpu-failure-modes.html — version-mismatch). Thundering-herd cold start — a burst scales many replicas (or all N ranks of one replica) that pull from the object store / image registry at once → the pull leg dominates even worse and the registry throttles; first move: rank-0 + broadcast, NVMe/host-RAM cache + staggered warm-up (llm-bottlenecks.html). Cold disk on a freshly autoscaled node — a new VM’s local NVMe is empty so the cache lever silently doesn’t apply to the scale-up it was built for → always pays the full pull; first move: bake weights into the machine image / pre-puller DaemonSet / shared RO volume (gpu-inference-cost-shapes.html). Slowest-rank straggler — one slow rank’s load stalls the whole TP×PP replica (barrier); first move: parallel shard pull + per-rank load-time monitoring, evict + reschedule the straggler (llm-bottlenecks.html). Partial / corrupt load served — a truncated S3 object or a bit-flip loads → the replica goes “ready” and ships plausible garbage (worse than a slow load); first move: a post-load golden-output validation probe before “ready” + a per-shard hash manifest (cite K22/D50; gpu-failure-modes.html). Graph re-capture / compile rebuild on every cold start — the compile leg is minutes when its artifact cache is cold; first move: persist the compile-artifact cache + snapshot the captured graphs (cite K17; llm-bottlenecks.html).cites llm/canonicals/llm-bottlenecks.html (thundering-herd pull, slowest-rank straggler, graph/compile rebuild), llm/canonicals/gpu-failure-modes.html (snapshot version skew, partial/corrupt load), llm/canonicals/gpu-inference-cost-shapes.html (cold-start > SLO, cold-disk-on-new-node) · K22/D50 (post-load validation, hash manifest) · ACTION: seed/extend these canonical entries in this same change
