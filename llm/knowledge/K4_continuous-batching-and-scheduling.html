K4 — Continuous Batching & Scheduling

How do you build a scalable, large-scale single-pool scheduler? Keep one GPU’s compute saturated while every request still meets its own TTFT/TPOT SLO — when requests arrive continuously, vary wildly in length, and finish at different steps. This is the single-pool scheduler: what one engine does with the requests it already holds. Fleet-level scaling is K15; the prefill/decode pool split is K5.

SLIs This Page Reduces

SLIs this page reduces (the reverse index of K1 §3.12):
  • Throughput & MFU / batch-occupancy — by refilling freed slots every iteration (continuous batching) so effective batch stays near Bmax instead of decaying to 1 (the convoy).
  • TPOT / ITL — by chunking long prefills and bounding per-iteration work so a 100k-token prompt never head-of-lines in-flight decodes.
  • Goodput (the real objective) — by tuning chunk size, the prefill↔decode split, multi-step k, and admission against requests-meeting-SLO, not raw tokens/s.
  • Queue-depth — by an admission + degradation contract that sheds in a stated order before the backlog grows unbounded.

The Scheduling Spine (the page’s spine)

Every section is one link in a naive→problem→fix chain; the decision-tree SVG below is this chain.

  • Static batch → idle-slot waste. Fix a batch, run it to completion. Problem: requests finish at different steps; a finished slot sits idle until the slowest sequence drains (the convoy effect) → utilization collapses.
  • Continuous (iteration-level) batching → long-prefill head-of-line. Re-decide the batch every iteration, admit a waiting request the instant a slot frees (Orca). Problem: a freshly admitted 100k-token prefill is one uninterruptible kernel that blocks every in-flight decode → TPOT spikes.
  • Chunked prefill → KV exhaustion. Split the prefill into fixed token-budget chunks so decodes interleave between them. Problem: more concurrent sequences ⇒ aggregate KV outgrows HBM, and KV grows every step so a request admitted when HBM looked fine OOMs the pool thousands of steps later (cite K2).
  • KV-aware admission → the projection can be wrong. Gate admission on free KV blocks, reserve minimum decode headroom, and project aggregate in-flight growth before admitting (proactive). Problem: output length is unknown, so the growth projection is an estimate that can still breach the watermark.
  • Preemption → which victim, recompute or swap. Evict a sequence’s KV at a watermark when the proactive estimate was wrong (the reactive fallback). Problem: victim choice and eviction mechanism are themselves a policy with a cost crossover.
  • Many SLO classes & tenants contend → priority + fairness. Problem: priority starves low-priority/long requests; fairness must compose with priority over GPU tokens, not requests.
  • Still over capacity → admission, load-shed, a degradation contract. Problem: what degrades, in what order, must be a stated contract, not an accident.
  • All of it must out-run two hidden taxes: the CPU scheduler stalling the GPU between steps, and the CUDA-graph static-shape constraint — and be tuned for goodput (requests meeting SLO), not raw tokens/s.

1. The Problem It Solves

The bottleneck shape

Problem→Solution stub. Problem: one GPU’s decode step is memory-bandwidth-bound and only pays for itself when many sequences share the one weight-load — but requests arrive continuously, vary wildly in length, and finish at different steps, so a fixed batch decays toward occupancy 1 (the convoy) and a long prefill stalls every decode. Solution (what the page builds): a per-iteration scheduler that keeps effective batch near Bmax, chunks long prefills, preempts under pressure, and sheds in a stated order — all tuned for goodput.number: effective-occupancy decay of a static batch vs continuous on the trace — cites K1 §3.5 (#decode), §3.4 (#roofline) · canonical: llm/canonicals/llm-bottlenecks.html#convoy

2. The Mechanism — the Tokenized-Prompt Walkthrough

Where the scheduler lives — the host-side control loop around the GPU forward step, not a model layer

Problem→Solution stub (architecture placement, hard req 6). Problem: the model (K0 forward pass) computes one step over whatever tokens it is handed — it has no notion of “requests”, queues, or SLOs; nothing inside the transformer decides which requests’ tokens go into the next step. Solution: the scheduler is a CPU-side loop that wraps every GPU forward step — it admits/evicts requests, packs their tokens into the ragged batch, launches the step, harvests outputs, and repeats. It sits around the K0 forward pass, not inside it; this is also why it can become the bottleneck (the D33 CPU-stall tax) and why CUDA graphs constrain it (D22).cites K0 (the forward step it wraps) · no number (placement fact) · diagram: scheduler loop as a ring around the K0 forward-pass block

The scheduler in a multi-GPU engine — every rank must make the identical decision [ADDED — not in plan]

Problem→Solution stub (a correctness fact, not a perf knob — placement-level, hard req 6 sibling). Problem: “one GPU step” is a simplification — when the engine is a TP/PP group the forward step is a collective-synchronized barrier (per-layer all-reduce, K7/K20). The scheduler loop is replicated on every rank; if rank 0 admits request X this step and rank 1 does not, their batched tensors have different shapes → the all-reduce mismatches → the engine hangs (a deadlock, not a slowdown — the worst failure class). Solution: the schedule must be bit-identical across ranks — a driver rank runs the scheduler and broadcasts the decision (batch membership, cu_seqlens, block tables) to all workers before the step (vLLM), or every rank runs a deterministic scheduler over byte-identical replicated state (no rank-local RNG / wall-clock / hash-order in the admission decision). State the placement: the scheduler sits around the K0 forward pass and above the K20 collective — it is a single logical decision physically replicated, not N independent schedulers.no number (correctness placement fact) · failure: rank-schedule divergence → collective deadlock (§5) · cites K7/K20 (the collective the decision must stay consistent with), K22 (group blast radius on the resulting hang) · real: vLLM driver-worker broadcast, TRT-LLM orchestrator

Why batch at all, and why continuous — derived

Problem→Solution stub (the page’s opening derivation, builds on the walkthrough). Sub-bulleted chain: (a) decode at batch 1 is memory-bandwidth-bound — one token’s compute must read all model weights from HBM, arithmetic intensity ≈ 1, MBU tiny (cite K1 #decode/#roofline); (b) batching B amortizes the one weight-load across B tokens — throughput ∝ B until decode crosses the roofline ridge at B* (cite K1; B*≈295 on the trace), so below B* more batch is free throughput; (c) static batching forfeits this — the batch only shrinks, never refills mid-flight, so effective B decays from Bmax toward 1 (the convoy: one 512-token generation among 31×32-token ones runs most of its life at near-1 occupancy); (d) continuous batching = iteration-level admission — refill a freed slot the instant a sequence EOSs → effective B stays near Bmax. This is the entire reason the scheduler exists; everything below manages the costs it creates.numbers: occupancy decay (512 among 31×32) & throughput ∝ B up to B*≈295, cited from K1 #decode #roofline · cites K1 only

3. The System-Design View

The scheduler itself

The iteration loop & max_num_batched_tokens

Problem→Solution stub. Problem: “continuous batching” is not a step — it is a question (when do you admit? how much work per step? where do finished sequences’ resources go?). Solution: the loop — collect finished → free their KV blocks (K2 D4) → admit from the waiting queue under the token budget → build the ragged batch → one GPU step → repeat; max_num_batched_tokens is the per-iteration work cap that makes step time predictable; a bounded waiting queue + backpressure is what happens when admission can’t keep up.number: step time vs token budget on the trace — cites K1 #decode (step cost), K2 D4 (block free/alloc) · SLIs: batch-occupancy, queue-depth (K1 #slis)

Deep dive — the ragged batch: how one GPU runs variable lengths in parallel

Deep-dive stub (the reader’s “how is this physically possible?” question, owed in place — rule 11). Problem: an iteration mixes a few prefills (different prompt lengths) with many decodes (1 token each, different KV lengths); static batching’s only answer is to pad every request to max_seq_len and run a rectangular tensor — compute/HBM wasted ∝ padding (a batch of one 8k-token req + 31×128-token reqs does ≈32× the needed attention work). Solution (three parts): (1) non-attention layers are length-agnostic — QKV/MLP are per-token GEMMs, so all tokens of all requests pack into one flat [total_tokens, d_model] matrix with cu_seqlens (prefix-sum) marking each request’s slice; (2) attention is the length-sensitive part — the kernel tiles into (query-block × KV-block) tiles, each an independent CTA, KV-side non-contiguity handled by the K2 block table (kernel internals = K17); (3) the GPU’s grid scheduler greedily hands each CTA to any free SM — variable length becomes “variable tile count”, not padding. The cliff K4 owns: the grid runs in waves of ceil(num_CTAs/num_SMs); one very long request → a lopsided tile count → a partial final wave with idle SMs = the wave / tile-quantization tail → MFU collapses though the GPU looks busy. This is precisely why max_num_batched_tokens + chunked prefill exist — they cap and regularize per-iteration tile count.number: padded-vs-packed work for {1×8k + 31×128} on the trace; wave-tail MFU loss · cites K0 #sdpa (O(seq²) shape), K2 D4 (non-contiguous KV), K17 (tiling/online-softmax kernel internals — rule of two) · K4 owns batch-composition + wave-balance

Deep dive D1 — chunked prefill: what it is and the four things it improves

Deep-dive stub. What it is: instead of prefilling a prompt in one giant kernel over all its tokens, split it into fixed max_num_batched_tokens-sized chunks processed across successive scheduler iterations. Four improvements, each Problem→Solution with a concrete prompt:

  • (1) Interleaving prefill with decode → no head-of-line TPOT spike. Problem: unchunked, a long prefill is one uninterruptible kernel; every in-flight decode stalls for its whole duration. Concrete: user A is decoding token 47 of the answer to "I like cat, why?"; user B submits a 120 000-token contract-review prompt — unchunked, A’s next token waits the full ~120k-token prefill (seconds). Solution: chunk B at C=2 048 → ~59 chunk-iterations; A’s decode is scheduled between chunks → A’s TPOT stays ≈ its normal ~21 ms.number: A’s TPOT unchunked (seconds) vs chunked (~21 ms) on the trace — cites K1 #decode #prefill
  • (2) Bounds prefill activation/residual-stream memory — the long-context OOM that hits before KV does. Problem: prefill over N tokens materializes the residual stream + attention intermediates ≈ [N, d_model] per layer; at N=200k this activation working set OOMs the GPU before the KV cache does (the hidden OOM). Concrete: a 200 000-token codebase prompt ("here is my repo, find the bug: <200k tokens>") OOMs even though weights+KV fit. Solution: chunk at C=4 096 → peak prefill activation is O(C·d_model), independent of N → 49 chunks, flat bounded footprint.number: activation footprint 200k vs C=4096 on the trace — cites K0/K18 (residual-stream shape), rule of two: K18/D46 owns the crossover
  • (3) CUDA-graph reuse via regular kernel shapes. Problem: unbounded variable-length prefill is a different tensor shape per request → a captured CUDA graph (fixed shapes) can’t be replayed → thousands of kernel launches/iteration, launch overhead dominates at small batch. Concrete: prompts of 6, 4 000, 128 000 tokens would need three prefill shapes; chunked at C=2 048 they all become a stream of identical 2 048-token chunk iterations → one captured graph serves every prompt length. Solution: chunk size = a fixed graph bucket the scheduler snaps to. Rule of two: K4 owns the scheduler-snaps-to-bucket; the capture machinery is K17 (D22).number: kernel launches/iter unchunked vs one-graph chunked — cites K17 D22 (capture machinery, rule of two)
  • (4) Bounded, uniform per-iteration work → predictable TPOT/goodput. Problem: variable per-step work (a fat prefill step vs a thin decode step) → TPOT jitter, p99 ≫ p50, wave-tail MFU collapse. Solution: max_num_batched_tokens caps every iteration’s work → step time ≈ constant → TPOT p99 ≈ p50. This is also how the in-pool prefill↔decode split (D34) is physically enforced and what goodput (D20) is tuned against.number: TPOT p99/p50 with vs without the cap on the trace — cites K1 #slis, forward-links D34/D20

Sub-stub — the chunk-size knob = the TTFT↔TPOT dial (the crossover, with the number). For prompt s_in, chunk C: prefill takes ⌈s_in/C⌉ iterations → TTFT ≈ ⌈s_in/C⌉ · step_time (bigger C → fewer iters → lower TTFT) but each chunk-bearing step adds ≈ prefill_compute(C) to concurrent decodes’ TPOT (bigger C → worse TPOT). Optimal C is where marginal TTFT-gain = marginal TPOT-loss in SLO-violation terms. Preemption/resume happens at chunk boundaries — chunked prefill is the preemption primitive.number: TTFT & TPOT vs C swept on the trace; the crossover C* — cites K1 #prefill #decode, derive both directions · real: vLLM chunked prefill

KV-aware admission & headroom reservation — the proactive twin of preemption [ADDED — not in plan]

Problem→Solution stub (the proactive sibling D3 reacts to; spine node before preemption). Problem: the iteration loop admits “under the token budget” — but max_num_batched_tokens is a compute cap; it says nothing about HBM. KV is allocated per admitted sequence and grows every decode step, so two distinct failures follow: (i) admitting a request with no free blocks for its prompt guarantees an immediate preempt; (ii) a set of requests that all fit at admission can collectively breach the free-block watermark thousands of steps later because every one keeps growing — the preempt storm D3 then has to clean up. Solution — a second admission gate + reservation + projection: (a) gate admission on free KV blocks for the prompt, not just the token budget; (b) reserve a minimum decode-headroom of blocks per admitted sequence so it can make forward progress before competing for growth; (c) project aggregate in-flight growth (Σ over running seqs of expected remaining tokens × block rate) and stop admitting while the projection would cross the watermark within the horizon. D3 (recompute-vs-swap) is the reactive fallback for when this estimate was wrong (output length is unknown — cite K19/D47 for the length prior, K4 owns the headroom/projection gate). The knob: the reserved-headroom fraction — too small → preempt storms (gate was useless); too large → under-admission, batch occupancy & goodput drop.number: preempt-rate & goodput vs reserved-headroom fraction on the trace; the watermark + horizon that minimize preempts without starving admission — cites K1 #decode (block growth rate), K2 D4 (block alloc) / #tier, K19 D47 (length prior) · real: vLLM watermark + can_allocate / GPU-blocks budget, TGI waiting_served_ratio · canonical: llm-bottlenecks.html#kv-exhaustion

Deep dive D3 — preemption: the two strategies and the crossover

Deep-dive stub (K4 owns the policy; the transfer mechanism is K2’s). Problem: at a free-block watermark a victim’s KV must go — but dropping it wastes its prefill, keeping it costs HBM. Solution — two strategies + the crossover: (a) recompute — drop the victim’s KV, re-run prefill on resume; O(1) extra memory, O(prompt) compute; wins for short prompts. (b) swap/offload — copy KV blocks HBM→pinned host RAM via the K2 block table (block-granular scatter DMA, not a contiguous memcpy), restore on resume; O(prompt) PCIe bytes + host RAM; wins for long prompts. Pick the cheaper: recompute = prompt_FLOPs/throughput vs swap = KV_bytes/PCIe_BW (both directions). State the equivalence explicitly: preemption-by-swap is the K2 warm-tier offload — same HBM→pinned-host DMA over the same block table; preemption is just the whole-sequence, scheduler-triggered, restore-on-resume policy over that mechanism, vs K2’s per-block hotness policy. Link K2_kv-cache.html#tier — “same transfer, different trigger.”number: recompute vs swap cost crossover by prompt length on the trace — cites K1 #prefill, K2 D4 (block-granular scatter DMA) / #tier (the equivalence) · canonical: llm-bottlenecks.html#kv-exhaustion

Deep dive D34 — the in-pool prefill↔decode token-budget split

Deep-dive stub + selection-knob sweep (rule 11) — the single-pool analog of K5’s pool ratio. Problem: each iteration’s max_num_batched_tokens is divided between prefill-chunk tokens and decode tokens; all-prefill iterations starve decodes (TPOT spike), all-decode starves prefills (TTFT spike) — the same tension K5 solves with separate pools, here resolved within one pool. Solution — the sweep: (a) decode-first (serve all running decodes, fill remainder with prefill chunks) — protects TPOT, risks TTFT under prefill backlog; (b) prefill-first — protects TTFT, spikes TPOT; (c) fixed ratio — predictable, ignores live SLO slack; (d) SLO-feedback ratio (SoTA) — raise prefill budget as TTFT-slack tightens, lower as TPOT-slack tightens. Crossover: the split where marginal TTFT-gain = marginal TPOT-loss in SLO-violation terms.number: TTFT/TPOT vs split ratio on the trace; the SLO-feedback crossover — cites K1 #prefill #decode · real: vLLM chunked-prefill long_prefill_token_threshold, SGLang · cross-link K5 (the cross-pool sibling)

Deep dive D11 (K4 co-host half) — speculative decoding breaks the uniform-step assumption · co-host with K13 §D11 (the speculation-mechanism half: the losslessness proof + the B_spec on/off crossover)

Deep-dive stub (rule of two: K13/D11 owns when speculation wins/loses — the α-speedup & the B_spec compute-bound cliff; K4/D11 (here) owns how the scheduler accommodates variable-width steps). Problem: the “bounded, uniform per-iteration work” guarantee (D1.4) silently rests on each decode step emits exactly 1 token/sequence — so a slot frees on a predictable cadence and the token budget is exact. Speculative decoding emits 1..K+1 accepted tokens per sequence per step, variable across sequences and across steps (acceptance is data-dependent). Three scheduler invariants break: (i) the per-iteration token budget is no longer num_seqs — a step verifies num_seqs×(K+1) tokens, so max_num_batched_tokens accounting must reserve the draft+verify width; (ii) KV-append count is variable per sequence (only accepted tokens commit; rejected draft KV must be rolled back — cite K2 D4); (iii) slot-free timing is variable (a high-acceptance seq finishes in fewer steps), perturbing the convoy/refill cadence. Solution K4 owns: budget on the speculative width (num_seqs×(K+1), not num_seqs); make KV-append/rollback part of the per-step harvest; treat the accepted-token count as the slot-reclaim signal. The on/off-by-batch decision (the B_spec crossover) is K13/D11 — cite, do not re-derive.number: token-budget inflation factor (K+1) and its effect on safe batch size on the trace — cites K1 #decode, K2 D4 (KV append/rollback), K13 D11 (when speculation helps/hurts — rule of two) · real: vLLM spec-decode scheduler, TensorRT-LLM

Deep dive D33 — the scheduler is on the critical path: the CPU-stall tax

Deep-dive stub (a top production cause of “GPU-util 60% but throughput low”). Problem: the scheduler runs on the CPU between GPU steps (collect EOS, free blocks, admission/eviction, build cu_seqlens + block tables, launch); at high QPS with many small requests the decode kernel is ~5–15 ms but the host scheduling work can rival it → the GPU idles between steps, achieved throughput ≪ roofline even at “high” GPU-util. Solution + the knob: overlap host scheduling with GPU compute — async scheduling (build step n+1 while step n runs) and multi-step scheduling (run k decode steps per schedule, amortizing host cost k×). Crossover: multi-step trades scheduling latency for staleness (a finished request unnoticed for up to k steps → wasted compute + TPOT jitter) → pick the largest k with host_sched_time/k ≤ ε·step_time and k·step_time ≤ TPOT_slack.number: host_sched_time vs decode step on the trace; the k that satisfies both bounds — cites K1 #decode (step time) · real: vLLM multi-step / async scheduler, TGI batching loop

Constrained-decoding & detokenizer host tax [ADDED — not in plan]

Problem→Solution stub (a real-world surface of the D33 host tax the plan didn’t name). Problem: two per-token host operations silently inflate the D33 tax and can break CUDA graphs in production: (i) guided/structured decoding — JSON-schema / grammar-constrained output computes a token-validity mask on the CPU every step from an FSM/pushdown automaton (large-vocab mask build is non-trivial and serializes with the GPU step); (ii) detokenization + stop-string matching — incremental detokenize and multi-stop-sequence scan run on the host per token. At high batch both rival the decode step and partially defeat graph capture (the mask is dynamic). Solution: mask precompute / FSM-state caching & jump-ahead (XGrammar/Outlines-style), async/overlapped detokenization, batched mask kernels, and accounting these into the D33 budget — they are part of host_sched_time, not free.number: per-token mask-build + detok cost vs decode step at high batch on the trace · real: vLLM structured output, XGrammar, Outlines, SGLang · cites D33 (same tax surface), D22 (graph breakage)

Deep dive D22 — the CUDA-graph static-shape constraint the scheduler must respect

Deep-dive stub (K4 owns the snap, K17 owns the capture — rule of two). Problem: continuous batching changes batch composition every iteration, but a captured CUDA graph requires fixed shapes; re-capturing per iteration is impossible. Sequence fan-out & termination-detection latency — the slot accounting continuous batching depends on [ADDED — not in plan]

Problem→Solution stub (two coupled gaps in “1 request = 1 slot, refill the instant it EOSs”; rule of two: K2 owns the CoW, K4 owns slot/budget/reclaim accounting). Problem A — fan-out: one admitted request is not one decode sequence — n>1 / beam / best-of-n fan out after prefill into n concurrently-decoding sequences (best-of-8 = 1 prefill → 8 decode slots + 8× the decode KV). An admission gate that reserves the prompt’s footprint under-reserves by n× and triggers an immediate post-prefill preempt. Problem B — reclaim latency: continuous batching’s entire premise is “refill the slot the instant the sequence ends” — but when is the end detected? max_tokens and the EOS token are exact (free this step); a stop-string is only decidable after incremental detokenization (the host tax above) and can span tokens → the slot is freed one or more steps late, so effective occupancy < nominal. Solution: (a) admission and the token budget reserve the post-fan-out sequence count and KV (the K2 CoW shares the common prefix, but K4 must count the n divergent slots); (b) model reclaim-latency explicitly — EOS-token / max_tokens free in-step, stop-strings free at detok-confirm; size the refill cadence on the confirmed-free signal, not the optimistic one.number: under-reservation factor for best-of-n; occupancy loss from k-step stop-string reclaim latency on the trace — cites K2 D4 (CoW + block free), the detok host-tax stub (stop-string lookback) · real: vLLM n/best_of + stop-string handling, SGLang

Deep dive D2 — request priority & how it maps to scheduling decisions

Deep-dive stub. Problem: “priority” is named but un-actionable until it maps to concrete per-step choices, and naive priority starves long/low-priority requests. Solution: priority classes resolve three choices each scheduler step — who gets the next free batch slot, who is preempted first when blocks run out, deadline/SLO-aware ordering; plus priority inversion + aging so low-priority or very long requests don’t starve, and per-tenant weighted-fair-queueing layered over GPU tokens (not requests) so priority and fairness compose. Deep dive D18 — multi-tenant isolation on the shared GPU

Deep-dive stub (K4 owns the on-GPU slice; fleet placement is K15). Problem: one tenant’s burst of long requests evicts everyone’s KV and monopolizes batch slots — noisy-neighbor on a shared engine. Solution: weighted-fair-queueing over GPU tokens, not requests (a tenant’s share = tokens/iteration, so a few long requests don’t beat many short ones), per-tenant KV-block quota, and priority composing on top of fairness (priority orders within a tenant’s fair share, not across it). Adapter-aware batch composition (multi-LoRA per step) — the scheduling slice [ADDED — not in plan; rule-of-two boundary]

Problem→Solution stub (rule of two stated in place — K15/L4 owns fleet adapter economics & scale-to-zero; K17 owns the SGMV/grouped-GEMM kernel; K4 owns only the per-batch composition constraint). Problem: a shared-base engine batches requests for different LoRA adapters in the same step. The base GEMM is shared, but each request also needs its adapter’s B·A delta — so a batch spanning m distinct adapters needs m adapter weight-sets resident and a grouped/segmented gather. The scheduler picking batch membership blind to adapter identity maximizes adapter diversity per step → more resident adapter weight (HBM) + less efficient grouped-GEMM → throughput collapses, exactly the noisy-neighbor shape of D18 but along the adapter axis. Solution: adapter-aware composition — cap distinct adapters per batch (max_loras), prefer co-batching requests sharing an adapter, and amortize adapter HBM-residency / activation across the steps it stays hot. The knob & its two-sided cliff: adapters-per-batch too low → head-of-line wait for same-adapter peers (latency, like over-strict affinity); too high → the grouped-GEMM + adapter-residency cost (throughput). The crossover is K4’s; the SGMV kernel cost it trades against is K17’s, the cold-adapter swap is K15/K21’s — cite, do not re-derive.number: throughput vs adapters-per-batch on the trace; the max_loras crossover — cites K1 #decode, K17 (SGMV grouped-GEMM cost, rule of two), K15/L4 (fleet adapter economics, rule of two) · real: S-LoRA / Punica, vLLM max_loras · cross-link D18 (same noisy-neighbor shape, tenant axis)

Overload behavior: admission as a rule-11 spectrum + the degradation contract (D13)
