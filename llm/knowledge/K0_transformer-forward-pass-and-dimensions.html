K0 — Transformer Forward Pass & Dimensions

The dimensional vocabulary the rest of this curriculum stands on. One static picture of what flows through the model and what shape it is — input → tokenize → embed → (norm → attention → residual → norm → MLP → residual) → logits → token — with the tensor shape annotated at every stage, and MHA vs MQA vs GQA. This page names and shapes the terms; it does not re-derive the math.

Serving-lens architecture reference • ~12 min

What This Page Is (and Is Not)

Every memory and FLOP number in K1 — KV-cache bytes, the 2× rule, the residual-stream activation footprint — is written in terms of d_model, d_head, n_q_heads, n_kv_heads. This page is the one place those terms are defined, visualized, and where the attention math that produces them is derived. The curriculum is self-contained: no llm/ page sends the reader to an external textbook.

  • Is: the forward-pass pipeline, the tensor shape at each stage, how a dimension is created/consumed, the MHA/MQA/GQA contrast with its KV-memory consequence, and — to serving depth — the attention math itself: scaled dot-product attention and the √d_head scale, softmax and the online-softmax recurrence FlashAttention runs, and RoPE with the position-alignment rule that decides which KV is reusable.
  • Is not: training-side theory (why residual nets train, optimization dynamics — K18 carries only the serving slice) and the proof of the 2× FLOP rule (that is K1 §3.2’s, not restated here).

The Architecture, in One Diagram

Symbols used in this page (long-form everywhere, no short aliases) — b batch (implicit) · s sequence length · d_model residual-stream width (the same number other sources call embedding_dim, d_embed, n_embd, or HuggingFace/PyTorch hidden_size — all aliases for one value: how many numbers represent a token; Llama-3-70B = 8,192. Not d_head, not d_ff, not V) · n_q_heads query heads · n_kv_heads key/value heads (with GQA, n_kv_heads < n_q_heads: multiple Q heads share one KV head) · d_head width of one head (so d_model = n_q_heads · d_head) · d_ff MLP hidden width · V vocabulary size.
Shape notation — the page uses NumPy/PyTorch convention. [d_model] means a rank-1 tensor of d_model numbers — it’s not a row or column vector; it has no orientation. [1, d_model] and [d_model, 1] are the explicit rank-2 row/column versions. In a matmul, x @ W with x as rank-1 [d_model] and W as [d_model, V] produces rank-1 [V] — the framework treats x as a row vector. Data convention is data-first: shapes read as [batch…, features] and matrices apply on the right (x @ W), opposite to the textbook W·x.
  • Blue dashed container — self-attention sub-layer.
  • Teal dashed container — MLP / FFN sub-layer.
  • Outer dashed container — one transformer block (repeats ×L).
  • Green / red / amber slices & columns — the three didactic heads; each column is one head running its own attention on its d_head-wide slice.
  • Yellow boxes — the two softmaxes (one inside attention — per head; one over the vocab).
  • Green SwiGLU box — the MLP's only non-linearity.
  • Pink boxes — RoPE positional encoding (parameter-free; per head, on Q and K).
  • Amber boxes — embedding-table boundaries (W_E in, W_U out).
  • Cyan boxes & teal arcs — residual adds and what's added back.

Two softmaxes, one network. The inner one (inside each head) decides which past tokens to attend to; the outer one turns final logits into a probability over the next token. The MLP has zero attention — it processes each token's vector independently through SwiGLU. LayerNorms and residual adds belong to neither sub-layer; they're the connective tissue that stabilizes training and lets information bypass either sub-layer when needed.

d_model vs d_head, and the Other Dimensions

The single identity that ties attention to everything else — visualized inline at the Reshape / per-head columns / Concat in the master diagram above: d_model = n_q_heads × d_head. The model vector has width d_model; attention slices it into n_q_heads independent heads each of width d_head, runs attention per head, then concatenates back to d_model. Nothing about the residual stream's width changes — only the internal head split does. In plain language: a token is just a list of d_model numbers (the "highway width"); n_q_heads is how many independent slices that width is cut into for attention; d_head is the width of one slice. K and V get only n_kv_heads slices instead of n_q_heads — that is GQA/MQA, in the next section.

Reading the per-head boxes (Q·Kᵀ and · V). Both · are matrix multiplies, never a cross product — transformers have no cross products. The site convention: · = matrix/dot product, × = ordinary scalar multiply (e.g. d_model = n_q_heads × d_head), [a, b] = a tensor shape. The · V step is best read as a weighted sum of value vectors: for one query token, the softmax row gives one scalar weight per past token; scale each token’s value vector v_j by its scalar weight w_j and add them up — the result is one output vector of width d_head. Not element-wise, not column-wise: vectors scaled by scalars, then summed. The worked numbers are in the concrete attention diagram below: attn = 0.10·v_0 + 0.25·v_1 + 0.65·v_2.

The attention matrices, sized once (W_Q/W_K/W_V/Q/K/V/W_O/O)

Every tensor in one attention sub-layer, for a prompt of length s. Same convention as W_E/W_U below: activations are rows, a projection is x @ W, so a weight’s first dim eats d_model and its second dim is the output width. Two derived numbers drive every size: n_q_heads · d_head = 64 × 128 = 8,192 = d_model (queries keep the full width) but n_kv_heads · d_head = 8 × 128 = 1,024 — GQA makes K/V 8× narrower than Q, which is the whole reason the KV cache (K2) is small.

The shape chain end-to-end: x [s, 8192] → (W_Q/W_K/W_V) → Q [s, 64, 128], K,V [s, 8, 128] → RoPE on Q,K (shape unchanged) → scores [64, s, s] → · V → O [s, 64, 128] → concat → [s, 8192] → W_O → out [s, 8192]. Width in = width out = d_model; the per-head d_head split and the GQA n_kv_heads shrink are the only internal shape changes. Under GQA each KV head is shared by n_q_heads / n_kv_heads = 64 / 8 = 8 query heads (broadcast inside the scores matmul, never materialized in HBM).

Does n_q_heads · d_head have to equal d_model? No — it is a near-universal convention, not an architectural requirement. The only hard constraints: (1) attention’s input and output must both be d_model wide, because the residual add x = x + attn(x) needs the delta to match the stream — W_Q [d_model, ·] enforces it going in, W_O [·, d_model] going out; (2) Q and K must share one d_head so Q · Kᵀ per head is a valid dot product. The inner width n_q_heads · d_head is otherwise a free hyperparameter — W_O is shaped [n_q_heads · d_head, d_model] precisely to reconcile any inner width back to d_model. Setting them equal (GPT-2/3, Llama, Mistral, Qwen) only makes W_Q/W_O square [d_model, d_model] and keeps param accounting clean. Counterexample — DeepSeek-V2 MLA: d_model = 5,120, 128 heads, per-head Q/K dim = 128 + 64 = 192, V head dim = 128 → n_heads · d_head = 128 × 192 = 24,576 ≠ 5,120, and W_O is a non-square [16384, 5120] — works fine.

What “residual stream” means

The residual stream is the running activation tensor that flows from the embedding all the way to the LM head (short for Language Model head — the final linear layer that multiplies the post-LN residual by W_U to produce one logit per vocab token; also called the output projection / unembedding layer; "head" is borrowed from multi-task learning, and a causal LLM has exactly one such head). It’s the same tensor at three nested granularities, all labeled x in the pseudocode:

GranularityShapeWhat one element is
One token, one position[d_model]that token’s running vector at this layer (the “per-token slice”)
One sequence (all positions)[s, d_model]the stream stacked across the s positions of one prompt
Batched at serving[b, s, d_model]the actual tensor in HBM when you serve b requests concurrently

The point is that the last dim is always d_model, every layer, every position, every batch element. After each sub-layer the model writes a delta back into the stream instead of replacing it:

# Inference, one prompt of length s. Tensor shapes shown on the right.

x = W_E[token_ids]                 # row-gather              x: [s, d_model]
for layer in 0..L-1:               # L=80 for Llama-3-70B
    x = x + attn_layer(norm(x))    # attention adds a delta  x: [s, d_model]
    x = x + mlp_layer(norm(x))     # MLP adds a delta        x: [s, d_model]
logits = norm(x[-1]) @ W_U         # read out last position  logits: [V]
                                   # (training would do all positions: [s, V])

Three consequences worth burning in: (1) width never changes — always d_model, from embedding to LM head, all 80 layers; the inner d_head / d_ff widths are temporary scratch only. (2) Layers communicate only through this stream — an early layer’s output is visible to every later layer because it’s added in and never erased (the “circuits” view from interpretability work). (3) At serving time, the batched stream [b, s, d_model] is the prefill activation tensor. In inference you only keep the current layer’s slice (no backprop), but a single layer’s stream at b=32, s=32768, d_model=8192, FP16 is 32 × 32,768 × 8,192 × 2 B = 16 GB — which can OOM before the KV cache does on long context (L3).

How d_ff is typically set

The MLP hidden width d_ff is not free — it’s sized by a model-family rule of thumb, then rounded for hardware:

FFN styleRule of thumbExampleWhy
Classic 2-matrix (ReLU/GELU) — GPT-2, GPT-3, BERTd_ff ≈ 4 × d_modelGPT-3 175B: d_model=12,288, d_ff=49,152 (= 4×)Two matrices: W_up [d_model, d_ff] + W_down [d_ff, d_model]. Parameter count of MLP block = 2 · d_model · d_ff = 8 d_model².
SwiGLU 3-matrix (modern) — Llama, Mistral, Qwen, DeepSeekd_ff ≈ (8/3) × d_model, rounded to a multiple of 128/256Llama-3-70B: d_model=8,192, d_ff=28,672 (≈ 3.5×); Llama-3-8B: d_model=4,096, d_ff=14,336 (≈ 3.5×)Three matrices: W_up, W_gate, W_down. Keeping MLP param count equal to the 4× classic FFN gives d_ff = 8/3 × d_model. Rounding to a TP-friendly multiple (Llama uses 256) bumps it slightly above 8/3.

You almost never set d_ff “to optimize accuracy” directly — it’s a derived knob from d_model + FFN style + TP shard alignment. The takeaway for serving: d_ff drives the dominant MLP matmul ([s, d_model] @ [d_model, d_ff]), and a 3-matrix SwiGLU MLP has 3 matmuls per layer in MLP alone, so MLP cost is ~3× an “attention projection” before counting attention itself.

Why the serving numbers care

  • KV cache (K2) scales with n_kv_heads × d_head — not d_model. That gap is the whole point of GQA/MQA below.
  • The 2× rule (K1) counts every weight once per token; weights are dominated by the four attention projections (Q: d_model × d_model; K,V each d_model × (n_kv_heads · d_head); O: d_model × d_model) and the three SwiGLU matrices (each d_model × d_ff).
  • Residual stream activation — the [b, s, d_model] highway is the prefill activation tensor; size b × s × d_model × 2 B in FP16. At b=32, s=32,768, d_model=8,192 that’s 16 GB just for the current layer’s stream — can OOM before the KV cache (L3).

What W_E and W_U are, with a worked example

Both are just 2-D matrices, sized only by the vocabulary and the residual-stream width. They are the only two weight tensors in the whole model where V appears.

MatrixWhere in the pipelineShapeEach row is…Each column is…Llama-3-70B size
W_E (embedding) Step 3 — “Token embedding lookup” [V, d_model] one vocab token’s learned vector (its “identity” in residual-stream space) one of the d_model residual-stream features, read across all tokens 128,256 × 8,192 = 1.05B params (2.1 GB FP16)
W_U (un-embedding / LM head) Last layer — “LM head” [d_model, V] one residual-stream feature, written across all tokens one vocab token’s “read-out direction” (the pattern that, if the final hidden state matches it, makes that token win) 8,192 × 128,256 = 1.05B params (2.1 GB FP16)

Encode & decode, in one worked example

Take the prompt "the cat" through a Llama-3-70B-shaped model (V=128,256, d_model=8,192, L=80). Numeric token ids and vector values shown are illustrative; shapes and operations are exact. Follow the diagram top-to-bottom — encode (left half = W_E row-gather) flows down into 80 blocks, then decode (right half = W_U column dot-product) reads out from the last position.

Symmetry to burn in: W_E’s rows are token vectors going in; W_U’s columns are token vectors coming out. Encoding is a pure index → row gather; decoding is one matmul whose result is read column-by-column. Training’s only job is to nudge those rows and columns so the right column wins the dot product when the context calls for it. That’s — literally — what next-token prediction is, at the matrix level.

Q&A — Weight vs Activation Shapes (are W_Q/W_K/W_V square? is K_h?)

Q — Are W_Q, W_K, W_V always the same square shape? Is K_h not square — do I understand correctly?

Short answer: two different objects are easy to conflate — the weight matrix W_K vs the activation K_h it produces. Their shapes are unrelated. Yes, K_h is essentially never square — you understand correctly.

The weights W_Q / W_K / W_V (fixed projection matrices, [d_model → out]):

  • Pure MHA (n_kv_heads = n_q_heads): all three are [d_model, d_model] — square and identical in shape. This page’s config: [4096, 4096].
  • GQA / MQA (n_kv_heads < n_q_heads — the modern default: Llama-3, most current models): not the same, not all square. W_Q stays square [d_model, d_model]; W_K, W_V are rectangular [d_model, n_kv_heads·d_head] — a narrower output. Llama-3-70B (d_model=8192, n_q=64, n_kv=8, d_head=128): W_Q [8192, 8192] square; W_K, W_V [8192, 1024] — 8× narrower, not square.
  • Even in MHA, a per-head slice W_Qᵃ is [d_model, d_head] (e.g. [4096, 128]) — tall, not square. Only the full fused matrix is square.

The activation K_h (one head’s cached keys — what the decode diagram shows as K_hᵀ [128×8]):

  • Shape is [T, d_head] (or transposed [d_head, T]) — here [8, 128].
  • T = number of cached tokens, grows +1 every decode step; d_head is fixed. So T ≠ d_head in general ⇒ K_h is essentially never square, and its squareness is meaningless — it is data sized by sequence length, not a weight.

The rule: W_Q/W_K/W_V are the same square shape only under pure MHA; under GQA/MQA W_K/W_V are deliberately rectangular — that narrowing is exactly what shrinks the KV cache.

Q — Will Q_h · K_hᵀ (the scores) always be square?

No — not always. The scores have shape [#query positions, #key positions]. The shared d_head is the contracted dimension — summed away, it never appears in the output. So the result is square only when #queries = #keys:

  • Prefill — queries and keys are the same S prompt tokens → [S, S] (e.g. [7, 7]): the causal-masked square. Square because every prompt token is simultaneously a query and a key (equal counts) — not because of d_head or any weight shape.
  • Decode — exactly 1 new query token vs T cached keys → [1, T] (e.g. [1, 8]): a single row, never square. T grows +1 every step; the only degenerate “square” is the first step at T=1 (a trivial [1, 1]).

The rule: scores-squareness is purely a property of how many query tokens vs how many key tokens — an artifact of prefill (all tokens at once) vs decode (one token at a time), with nothing to do with head or weight dimensions.

Q — Will W_O always have the same size as W_Q?

Same parameter count always; same shape only when n_q_heads·d_head = d_model (the standard case — true here).

  • W_Q = [d_model, n_q_heads·d_head] — residual → query space. W_O = [n_q_heads·d_head, d_model] — concatenated per-head outputs → residual. W_O is the mirror/transpose orientation of W_Q, both keyed on the query-head width n_q_heads·d_head.
  • Parameter count: always equal — both hold d_model × n_q_heads·d_head elements (just transposed).
  • Identical shape ([d_model, d_model], square): whenever n_q_heads·d_head = d_model — the overwhelmingly common case (Llama-3, GPT; this page’s MHA config 32×128 = 4096 → both [4096, 4096]). If a model deliberately sets n_q_heads·d_head ≠ d_model (uncommon): same element count, transposed non-square shape, not equal to W_Q.
  • The crucial asymmetry vs W_K/W_V: unlike them, W_O does not shrink under GQA/MQA. GQA shares K/V, but every one of the n_q_heads query heads still produces its own d_head output, all concatenated and projected by W_O. So W_O’s input width is always n_q_heads·d_head, never n_kv_heads·d_head — W_O tracks W_Q, not W_K/W_V.

Q — Is d_model always = n_q_heads · d_head?

Almost always yes — by near-universal convention — but it is not a hard architectural requirement.

  • Why it usually holds. Standard transformers use a full-width query projection: the residual stream of width d_model splits into n_q_heads heads of width d_head with n_q_heads·d_head = d_model, so the per-head outputs concat back to exactly d_model. That is what makes W_Q/W_O square. Holds for GPT-2/3, Llama-1/2/3, Mistral, the original Transformer (512 = 8×64), this page (4096 = 32×128), Llama-3-70B (8192 = 64×128).
  • What the architecture actually requires (weaker than the equality): (a) Q and K share the same per-head width d_head (so Q·Kᵀ contracts); (b) W_O maps n_q_heads·d_head → d_model. n_q_heads·d_head may differ from d_model — then W_Q is [d_model, n_q_heads·d_head] and W_O is [n_q_heads·d_head, d_model] (rectangular); the residual stays d_model, attention just runs in a different inner width. Nothing breaks.
  • When it differs. Models that pin d_head independently (e.g. fixed d_head=128 regardless of d_model/n_heads). Common enough that HuggingFace configs (e.g. LlamaConfig) expose head_dim as an explicit field — it merely defaults to hidden_size // num_attention_heads, signalling it need not equal it.
  • Scope caveat (query side only). This identity is about the query side. Under GQA/MQA the K/V side is n_kv_heads·d_head, deliberately < d_model — so even when d_model = n_q_heads·d_head holds, d_model ≠ n_kv_heads·d_head in any GQA model.

Q — Is n_kv_heads · d_head always = d_model?

No — almost never in modern models. This is exactly the K/V side, and pushing it below d_model is the entire point of GQA/MQA.

  • Pure MHA (n_kv_heads = n_q_heads): n_kv_heads·d_head = n_q_heads·d_head = d_model — equal, but only because the query-side identity also holds. This page’s config: 32·128 = 4096 = d_model.
  • GQA (n_kv_heads < n_q_heads — the modern default: Llama-3, Mistral, most current LLMs): strictly less than d_model, by the group factor n_q_heads / n_kv_heads. Llama-3-70B: 8·128 = 1024 vs d_model = 8192 — 8× smaller, not equal.
  • MQA (n_kv_heads = 1): n_kv_heads·d_head = d_head (e.g. 128) — far below d_model. The extreme case.
  • Why it matters (the serving payoff). n_kv_heads·d_head is exactly the output width of W_K/W_V (the rectangular shape from the first Q above) and the per-token-per-layer KV-cache width: KV bytes/token/layer = 2 · n_kv_heads · d_head · dtype_bytes. Cutting it by n_q_heads/n_kv_heads shrinks the KV cache by that exact factor — the dominant decode-memory lever (developed in K2 — the KV cache).

Step 1 — The Prefill Step (the Whole Prompt in One Pass), Drawn to Scale

Prefill runs the entire prompt "i like to hang out with jessie" (7 tokens) through every layer in one forward pass — all 7 token-rows at once, not one at a time. Same L-shape matmul geometry and same ln(dim) log scale as the decode diagram below. The one structural difference from decode: the left operand is [7×4096] (7 rows, one per prompt token), so the attention scores are a [7×7] causal-masked square instead of a [1×T] row. At the very end only the last position's row is sliced out ([1×4096]) and sent through the LM head — that single row predicts the first generated token, "in". (All 7 positions' K,V are written to the cache for the decode step that follows.)

The two prefill-only differences vs decode (below). (1) The left operand is [7×4096], not [1×4096] — every box that is a 1-row sliver in decode is a 7-row block here, and the attention scores are a [7×7] causal-masked square (token i attends only to tokens 0..i) instead of a [1×T] row. (2) The LM head reads only the last row final[-1] [1×4096] — one prompt yields exactly one next token; the other 6 rows are computed (their K,V are needed) but their logits are discarded. Bridge to decode: "in" is sampled and appended at position 8; the K,V for all 7 prompt tokens (plus "in") now sit in the cache — so the decode step below runs with T=8.

The formula at each numbered step (data-first; · = matmul, × = scalar/shape)

Config: s=7 prompt tokens, d_model=4096, H=32, d_head=128, d_ff=11008, V=32000. The drawn boxes are the matmuls; the plain arrows between them are the no-weight glue (norm / RoPE / softmax / residual add).

  • ① Q projection (K, V identical) — xn = RMSNorm(x), then Q = xn · W_Q, K = xn · W_K, V = xn · W_V. Shape [7,4096] · [4096,4096] → [7,4096]; reshape to [7,32,128] per head; apply RoPE to Q,K (the “reshape” arrow).
  • ② Scores (per head h) — scores_h = (Q_h · K_hᵀ) / √d_head, add the causal mask (0 if j≤i, −∞ if j>i), then weights_h = softmax(scores_h) row-wise. Shape [7,128] · [128,7] → [7,7].
  • ③ Mix values (per head h) — head_out_h = weights_h · V_h. Shape [7,7] · [7,128] → [7,128]; concat the 32 heads → concat [7,4096].
  • ④ Output projection — attn_out = concat · W_O ([7,4096] · [4096,4096] → [7,4096]), then the residual add h_mid = x + attn_out (the “+ x” arrow).
  • ⑤ MLP up + gate (SwiGLU) — hn = RMSNorm(h_mid), gate = hn · W_gate, up = hn · W_up, then act = SiLU(gate) ⊙ up where SiLU(z) = z · σ(z) and ⊙ = elementwise. Shape [7,4096] · [4096,11008] → [7,11008].
  • ⑥ MLP down — mlp_out = act · W_down ([7,11008] · [11008,4096] → [7,4096]), then residual final = h_mid + mlp_out. Repeat ①–⑥ ×N layers, then one final RMSNorm.
  • ⑦ LM head (last position only) — slice final[-1] [1,4096], then logits = final[-1] · W_LM_head ([1,4096] · [4096,32000] → [1,32000]); next = argmax(softmax(logits)) → "in".

What x = [7×4096] actually is — one ROW per token

Yes — the prompt becomes a stack of 7 rows, one row per token; each row is that token's 4096-number embedding vector. It is not one row of dot-separated words.

Step 2 — The Decode Step (decoding “in”), Drawn to Scale as Matmul Geometry

Now decode: the token "in" the prefill produced is fed back as the new input, and the model predicts the next word. Same L-shape rule — weight on top, input row on the left, result in the corner; the touching edges are the shared (contracted) dimension and must be equal; the corner keeps the two outer dimensions. Scale: box side ∝ ln(dim) (log), so a 4096×4096 weight is a true square and 1×4096 > 1×128 > 1×8 stay distinct. Provenance: every number is keyed in the legend. Config: decoding "in" at position 8 — it attends to T=8 cached keys (the 7 prompt tokens + itself).

Read it as L-shapes flowing right. The squares (W_Q, W_O) are square because 4096×4096 is square; W_LM_head is the widest box because V=32000 is the model's largest dimension; the geometry rule is now visually exact — the weight's height equals the left vector's width at every stage (e.g. K_hᵀ's height = Q_h's width, both the shared 128). 1×128 (a wide thin bar) and 128×8 (a narrow tall block) are clearly different shapes; 8 < 128 < 4096 < 11008 < 32000 all read in order. Only the T arm (here 8) grows as generation proceeds. (Scale is ln(dim) — logarithmic — so values from 1 to 32000 stay distinct on one canvas; the label carries the exact value.)

How to read each stage (a matmul is an L)

  • Weight on TOP, input row vector on the LEFT, result in the CORNER where the two arms meet.
  • The two touching edges are the shared (contracted) dimension — the weight's height equals the left vector's width. They must be equal.
  • The corner keeps the two outer dimensions: height from the left operand (always 1 — one decoded token), width from the weight's columns.
  • Each corner is reshaped or normed into the next stage's left operand — follow the arrows. Non-matmul steps (reshape, softmax, concat, residual add, SiLU⊙up, ×32 layers) are drawn as plain arrows, not Ls.

Aside — Continuous Batching vs Chunked Prefill

Scope note. These are scheduling concepts — not the forward pass. K0 owns the math; the owner of this topic is K4 — continuous batching & scheduling (chunked prefill is K4’s D1 deep dive). Shown here only because both build directly on the prefill (Step 1) vs decode (Step 2) distinction above. For the full treatment — the chunk-size crossover, preemption, goodput — see K4.

They are orthogonal and compose — not alternatives. Continuous batching is the scheduling loop (what to do across requests when a slot frees); chunked prefill is a technique inside that loop (what to do within one request so a long prefill doesn’t monopolize an iteration). Continuous batching alone still suffers prefill head-of-line — the exact gap chunked prefill fills.

How heterogeneous shapes pack into ONE forward pass

Diagrams 1–2 show what the scheduler chooses to run in one iteration; this one shows how a single forward pass actually processes the mix. The trick: only attention cares which row belongs to which request. Every other layer in the transformer block — Q/K/V projections, the MLP, the output projection, LayerNorm, residual adds, the activation — is purely per-token, so all N tokens from all in-flight requests pack into one flat [N, d_model] tensor and run as one matmul. Attention is then split per request via cu_seqlens against each request’s own KV cache. After attention, the rows re-pack and the rest of the block runs as one matmul again.

Diagram 3 — the ragged batch. Step ④ (Q/K/V projection) and the symmetric O proj / MLP / norms / residual adds are one matmul each on the packed [464, d_model] tensor — the 464 rows are interchangeable, the kernel never asks who they belong to. Only attention (⑤⑥⑦) is segmented: cu_seqlens partitions the rows; each request’s queries attend only to its own KV cache; the prefill segment carries a causal mask inside its own 462 new keys (the dashed red triangle). Run as one kernel launch by FlashAttention’s varlen API or, in vLLM, per-segment paged-attention reading each sequence’s block table. Then the rows re-pack and the layer continues. Why this matters for scheduling: max_num_batched_tokens caps the flat dimension N (compute / activation memory in steps ④ and after), while max_num_seqs caps the segment count (KV-slot bookkeeping in ⑤⑥⑦) — they bound different things, which is why both knobs exist.

Step ⑨ expanded — what “split rows by cu_seqlens” actually does, mechanically

In the main diagram, step ⑨ is just one arrow with the label “split rows by cu_seqlens →”. That arrow hides the entire indexing operation. The diagram below opens it up on the same numbers (A=1 tok, B=1 tok, C=462 toks; cu_seqlens=[0, 1, 2, 464]).

The plain-English read of cu_seqlens_q = [0, 1, 2, 464]:

  • Tokens [0:1] belong to request A (1 decode token)
  • Tokens [1:2] belong to request B (1 decode token)
  • Tokens [2:464] belong to request C (462 prefill-chunk tokens)

The rule is uniform: each consecutive pair (cu[i], cu[i+1]) defines a half-open row range for segment i, and the segment’s token count is just cu[i+1] - cu[i]. (We say cu_seqlens_q here because there's a sibling cu_seqlens_k for the K/V side — same shape, different per-request key counts; that's what the kernel uses to bound each segment's attention range against the per-request KV cache.)

What happens to each segment after the split: each slice further splits along the last dim into Q, K_new, V_new (each width d_model); Q goes to attention, K_new/V_new are appended to that request’s KV cache, then the attention kernel attends Q against the full extended K/V (old cached keys + the new ones just appended). Worked on the example: A’s Q_A=[1, d_model] attends against K_A of shape [d_head, 200] (the 200 keys A had cached). C’s Q_C=[462, d_model] attends against K_C of shape [d_head, 2000] (1538 cached + 462 just appended).

Diagram 3a — step ⑨ opened up. Three sub-steps. (1) Read cu_seqlens as consecutive pairs — each pair is a half-open row range; the segment’s token count is just cu[i+1] - cu[i]. (2) Slice the packed QKV_packed [464, 3·d_model] tensor by each range — these are zero-copy views, so the “split” itself costs nothing; the real cost is the per-segment attention kernel that runs after. (3) Each slice further splits along its last dim (size 3·d_model) into Q, K_new, V_new, each width d_model — via Q_seg, K_new_seg, V_new_seg = slice_seg.split(d_model, dim=-1). Then per-segment attention runs (steps ⑤⑥⑦ in the main diagram above): Q against the request’s own (possibly large, cached) K/V, K_new/V_new get appended to the request’s KV cache. In production: FlashAttention’s flash_attn_varlen_func doesn’t literally materialize three slices — it reads cu_seqlens directly and tiles attention per-range — but the semantics are exactly what’s drawn above.

How packing prevents cross-user attention — the structural invariant

A natural worry when you first see the ragged batch: “if I stack tokens from many users into one matrix, what stops user A’s tokens from contaminating user B’s output?” Answer: nothing has to actively stop it — cross-user contamination is structurally impossible, because the two types of layers in a transformer block behave totally differently:

  • Per-token layers (Q/K/V projection, MLP, LayerNorm, residual add, activation — ~90% of FLOPs): each output row depends only on its own input row. There’s no notion of “which user” — the matmul literally cannot mix rows. Packing user-A and user-B’s tokens into one tensor is just a throughput trick; nothing would change numerically if we ran them separately.
  • Attention (the only cross-token layer): segregated explicitly by cu_seqlens. Each user’s queries are restricted to attend only against that user’s own keys/values (held in that request’s private KV cache via the block table). User-A’s “jessie” token cannot see user-B’s “cat” token, ever.

Worked end-to-end on two users. Diagrams 3 and 3a above used three abstract segments A/B/C to focus on the slicing mechanism; here’s the same picture re-drawn with two specific users so the “safety” argument is concrete:

  • User A is decoding a continuation of "i like jessie" — just submitted its 4th token; needs 1 new decode-token output. Its private KV cache holds K/V for tokens 0..3.
  • User B just sent the prefill "i like cat" — 3 new prompt tokens (fresh request, no KV cache yet).

Diagram 3b — the cross-user-attention invariant on two concrete users. Same four-row structure as Diagram 3 but instantiated for the two-user case the question targets. What enforces the segregation, in code: the attention kernel (FlashAttention’s flash_attn_varlen_func or vLLM’s paged-attention) takes cu_seqlens as an input. When it tiles the attention computation for segment i, it walks only the K/V block table for segment i. There is no “global K/V tensor over all users” loaded anywhere; per-user KV caches are physically separate blocks of HBM, indexed by separate block tables. The kernel could not mix users even if it tried — it doesn’t have a pointer to the other users’ K/V.

Why the Cache is K,V — Never Q, Never O

TL;DR — cache exactly the tensors that token t writes once and every future token re-reads. In a transformer that set is precisely {K, V}. Read it straight off the decode diagram above:

  • Stage ② (scores). Q_h [1×128] on the left — height 1, the one new token’s query — against K_hᵀ [128×8] on top — width T=8, all cached keys.
  • Stage ③ (mix). Mixes V_h [8×128] — height T=8, all cached values.
  • The growing arm is the cache. That 8 is the only arm that grows each step — the diagram already calls T “the only dimension that changes during generation.” That is the KV cache.
  • Q is never cached. Forever the height-1 sliver: born from one token, used in ②, gone.
  • O is never cached. O = attn_out [1×4096] flows down its own column to one token and is never read again.

The whole argument in one picture: the thick cyan arrow into STEP B is the entire point — K₈,V₈ were computed once in STEP A and are re-read in STEP B (and every step after) without being recomputed. Q₈ and O₈ have no such arrow leaving STEP A — no future step ever reads them, so there is nothing to cache.

When are rows 1–7 (prompt K,V from prefill) used? Same story as K₈,V₈, one step earlier — written once by prefill, then read at scores ② and mix ③ of every decode step, in every one of the 32 layers, for the whole generation:

  • STEP A (decode "in"): the cyan ↘ reads all cached K,V arrow pulls T=8 rows = rows 1–7 + row 8. Rows 1–7 are 7 of the 8 keys Q₈ dots against and 7 of the 8 values it pools.
  • STEP B (decode "the"): T=9 rows = rows 1–7 + row 8 + row 9 — rows 1–7 read again, unchanged.
  • Every later step still re-reads rows 1–7. They are the permanent floor of the growing T arm — exactly what “every step re-reads ALL rows” in the cache box means — and are never recomputed. That reuse is the whole reason prefill writes them to the cache.

Why the math forces exactly this

  • Attention is the only cross-token operation. It is the single place where token t’s computation reads other tokens’ data. QKV-projection, O-projection, both residual adds, and the MLP are all pointwise in the token axis — token t’s value at that stage depends only on token t. Pointwise ⇒ recompute it for free from the one new token ⇒ nothing to cache.
  • K, V are written once and read forever. Kᵗ = xᵗ · W_K depends only on token t’s residual and the frozen weights — so it is identical at every later step. Storing it once and reusing it is exact, not an approximation. Without the cache you would re-run all layers over tokens 0..N−1 every step — O(L³) over a length-L generation (the cost magnitude is K2’s job — see below).
  • Q is the active side — consumed immediately, never revisited. At step N the only query that exists is Qₙ (the height-1 left operand in stage ②). A past Qᵗ has no consumer: the causal mask pairs the current query with past keys only, and token t’s output was already finalized when it was the active token. No reader ⇒ nothing to cache.
  • O and everything downstream is dead after its token. attn_outᵗ → residual → MLP → logits → token t, then discarded. No future token reads token t’s output — future tokens read token t’s K, V (via attention), never its O. Caching O would store data with zero future readers.

The rule, stated once: cache = {written once, read by the future} = exactly {K, V}. The O(L³)→O(L²) cost this buys, and why it lives in attention and not the MLP, are derived in K2 — the KV cache (not re-derived here — this page owns the shape/dataflow answer, K2 owns the cost).

MHA vs MQA vs GQA

All three use the same n_q_heads query heads. They differ only in how many KV heads those queries share — and that single number sets the KV-cache size, which is the dominant memory cost in decode. More query heads sharing fewer KV heads = smaller cache, slightly less expressive attention.

Yes — when n_q_heads > n_kv_heads, multiple query heads literally reuse the same K and V vectors. Each KV head is shared by a group of n_q_heads / n_kv_heads query heads: those queries compute their attention scores against the same keys and weight the same values. In Llama-3-70B that ratio is 64 / 8 = 8 — eight query heads per KV head. Only Q is “full” (d×d); K and V are projected to width n_kv_heads · d_head (in Llama-3-70B: 8 · 128 = 1,024, eight times narrower than d_model = 8,192) and physically broadcast at attention time so each of the 8 sharing queries dots against the same keys. That broadcast is free at compute time but is the entire reason the KV cache shrinks 8×.

Blue = query heads (always 8 here), teal = KV heads. A line means “this query head attends using that KV head's keys/values.” Fewer teal dots ⇒ fewer K,V vectors to store per token.

Does sharing K,V across query heads make the model dumber?

Short answer: theoretically yes, in practice barely — at moderate sharing (GQA-8). At the extreme (MQA), it’s a real cost.

Why GQA-8’s hit is so small. Many attention heads in a trained MHA model end up doing similar things (induction heads, copying heads, positional heads — lots of redundancy). GQA removes the redundancy on the K,V side without removing expressivity on the Q side: each query head still has its own learned W_Q[head], so 8 sharing Q heads can still rotate the same keys differently. You lose "8 different K projections per token" but keep "8 different ways of querying the shared K", which is most of the expressive power.

Why MQA goes too far. One K and one V per token must serve every Q head — tasks needing head-specific fine-grained key discrimination (precise retrieval over long context, multi-hop reasoning) start showing real degradation. That’s why frontier 2024+ models almost all landed on GQA-8 rather than MQA: ~all the memory win, ~none of the quality cost. Rule of thumb: n_q / n_kv ∈ [4, 8] ≈ free; pushing to n_q (full MQA) starts costing measurable points.

How Attention at the Current Position Becomes the Next-Token Prediction

We’ve seen the static pieces (Q, K, V, heads, residual stream, W_E, W_U). Now the dynamic story: at every layer, the current position’s Q dots against the K of every previous position (plus itself), the resulting scores are softmaxed into weights, and those weights pool the past V vectors into a context summary that gets added back into the current position’s residual. Do this 80 times (80 layers), and the last position’s residual stream has integrated everything before it — ready for the LM head to read out as a next-token logit.

Causal mask (one-line definition). Position i may only attend to positions 0..i — never to i+1, i+2, …. Concretely, before the softmax, the score matrix is masked: scores[i, j] = −∞ if j > i, so softmax sets those weights to zero. Why: at training time a model that could peek at future tokens would learn to copy them — zero training loss, useless at inference (no future exists). At serving time, the same mask is why the KV cache works: a new position only needs to read past K,V; it never re-influences them.

One head’s attention at the last position, walked through

Continuing the running example, take prompt "the cat sat" (s=3) and zoom in on one head, one layer, at the last position i = 2 (the " sat" token). All numbers below are illustrative; shapes and ops are exact.

The autoregressive loop — what happens after sampling

Once we sample, say, " on" at the end of "the cat sat", the new token is appended:

Each new step does the same attention computation shown above, just with a longer K,V row to dot against. This is exactly why the KV cache exists — if past K,V were re-computed every step, decode would be O(s²) instead of O(s). The cache turns each decode step into O(s) (one Q dotted against all past K, then one weighted sum over all past V), which is why decode is dominated by HBM bandwidth (loading the cache) rather than compute (see K1 decode-step formula).

Why we cache K and V — but not Q

Look at how each projection is used over time. At decode step t the model computes one new q_t, one new k_t, one new v_t from x[t]. Their lifetimes are wildly asymmetric:

  • Q[t] — transient. Used once, at step t, to ask "given me, which past positions matter?" Step t+1 will compute its own Q[t+1] from scratch and never needs Q[t] again. Caching Q would store something that has no future reader.
  • K[t] — persistent. Used at step t (when the new token attends to itself) and at every future step — positions t+1, t+2, … all need to dot their queries against K[t]. Without caching it, each future step would have to re-project x[t] through W_K again.
  • V[t] — persistent. Same story: every future step reads V[t] as part of its weighted sum.

Q-as-transient asymmetry, in one sentence: Q[t] is used only at position t, to query the past; K[j] and V[j] are used at every position t ≥ j, by whoever queries position j. Cache the things with future readers; recompute the things without.

The KV cache stores vectors, not weights

One more distinction that’s easy to blur: there are two different things on the GPU with the letters K and V in their name — the projection matrices (model weights) and the projection results (per-request activations). The KV cache holds only the second.

So W_K is loaded once per deployment and used by every request — a few hundred MB total across all 80 layers. K[t] is computed once per token per request and lives in the KV cache for the rest of that request — the per-request buffer that grows to many GB at long context. Saying “the model has KV cache” means the second; saying “the model has K,V projections” means the first. Different lifetime, different region, different scaling story.

Why this matters for serving

  • KV cache is monotonic & per-layer. For every layer, every kv-head, every position, we store one K and one V vector. That’s why KV bytes per token = 2 · L · n_kv_heads · d_head · 2 — the "2 · L" is what this section’s loop unrolls (K2).
  • Decode reads ALL past K, V every step. At step t with sequence length s, the attention pulls 2 · L · n_kv_heads · d_head · s · 2 B bytes from HBM — growing linearly with s. This is the dominant decode cost at long context (L3).
  • Causal masking enables prefix caching. Because position i’s K,V never depend on positions > i, two requests sharing a common prefix can share its KV-cache blocks (K2 Role 2 — inter-request reuse) — the foundation of PagedAttention’s prefix-reuse, and the reason a true prefix (not an arbitrary span) is the reusable unit K2’s cache-aware routing targets.
  • Only the last position’s residual matters for next-token prediction at decode. Other positions’ final residuals are computed (they have to be, to update the KV cache) but their logits are discarded. This is why decode is "1 token in → 1 token out" while prefill processes the whole prompt in a single matmul-heavy pass (K1 prefill vs decode).

The Attention Math, Derived to Serving Depth

The walkthrough above showed the shape of attention. This section gives the formula it computes and the three pieces with direct serving consequences: the √d_head scale, the softmax and its streaming form (the reason FlashAttention is possible), and RoPE (which decides what KV is reusable across requests). This is the curriculum’s single home for this math — every other page cites here, none re-derives it, none sends you to an external textbook.

Scaled dot-product attention, and why the √d_head

For one head, with Q, K, V each [s, d_head] (the shapes from the walkthrough):

Attention(Q, K, V) = softmax( Q·Kᵀ / √d_head ) · V
  • Term by term. Q·Kᵀ → [s, s] scores (row i = query i’s affinity to every key); ÷ √d_head rescales; row-wise softmax → weights that sum to 1; · V → [s, d_head], each output row a weighted average of value vectors.
  • Problem — raw scores blow up. A score is a dot product of two d_head-dim vectors. With ~unit-variance entries, the dot product has variance ≈ d_head. On the trace (Llama-3-70B, d_head = 128): score std ≈ √128 ≈ 11. Feeding scores of magnitude ~11 into softmax pushes it to a near one-hot distribution — attention collapses onto a single token and gradients vanish in training.
  • Solution — divide by √d_head. Scaling scores by 1/√d_head returns their variance to ≈ 1 (std 11 → 1), keeping softmax in its smooth, informative regime. That single constant is the entire reason the term exists.
  • The shape K2 stands on. Q·Kᵀ is [s, s] — s² scores, each O(d_head) ⇒ O(s²·d_head) per head per layer. K2’s “space trades compute” argument cites exactly this quadratic shape; it is derived here, not there.

Softmax, and the online recurrence FlashAttention runs

  • Definition. softmax(x)_i = ex_i / Σ_j ex_j — maps a score row to non-negative weights summing to 1, so ·V is a weighted average.
  • Problem — overflow. ex overflows fp16/fp32 for x ≳ 88. Solution — shift by the row max: ex_i−m / Σ ex_j−m with m = max_j x_j. The e−m cancels top and bottom — identical result, never overflows. Every real softmax does this.
  • Problem — the score matrix is O(s²) in HBM. Naive softmax needs the whole row in memory at once (to get m and the denominator ℓ), so attention materializes the full [s, s] score matrix in HBM — the O(s²) traffic that makes long context explode (the bottleneck K17 attacks).
  • Solution — the online (streaming) recurrence. Process keys in blocks, carrying only a running max m, running denominator ℓ, and running output O. For a new block with local max m′, local denom ℓ′, local output O′:
    mₙ = max(m, m′)  ·  ℓₙ = ℓ·em−mₙ + ℓ′·em′−mₙ  ·  Oₙ = O·(ℓ em−mₙ/ℓₙ) + O′·(ℓ′ em′−mₙ/ℓₙ)
    The score matrix is never materialized — HBM traffic drops from O(s²) to O(s). This recurrence is the FlashAttention tiling loop; K17 applies it, this page owns it.

FlashAttention, Visualized — Block-by-Block + Exact Online Softmax

The recurrence above is exactly what FlashAttention runs. Two ideas, two pictures: (1) tile K,V into blocks and compute attention one block at a time, so the [s×s] score matrix is never built; (2) the online-softmax update carries only a running (m, ℓ, O), so the streamed result is bit-identical to the full-matrix softmax — exact, not approximate. (Kernel / SRAM-tiling and CUDA-graph specifics are K17’s; this page owns the math these diagrams show.)

Diagram 1 — the tiling. Naive attention materializes the full [s×s] matrix in HBM (O(s²)). FlashAttention keeps Q resident and loops over KV blocks, computing one small score tile at a time and discarding it — HBM traffic O(s), identical output. The [s×s] matrix never exists.

Softmax, for reference (the formula Diagram 2 below streams over blocks):
  • Definition. softmax(x)i = exi / Σj exj — non-negative weights that sum to 1. Both i and j index the same input vector x:
    • i — the specific position whose weight you’re computing (fixed for one evaluation).
    • j — a dummy summation index walking every position 1..n to build the denominator (same denominator for every i in the row).
    • In attention: x = one query’s row of s key-scores ⇒ the s weights sum to 1, so ·V is a weighted average.
    • In the LM head: x = the [V] logits ⇒ a probability over the next token.
  • Numerically stable form (what is actually computed; ex overflows fp16/fp32 for x ≳ 88): exi−m / Σj exj−m with m = maxj xj. The e−m cancels top and bottom — identical result, never overflows.
  • Why the rescale below works. When a new block reveals a larger max mₙ > m, every old term must be re-based: exi−m = exi−mₙ · emₙ−m ⇒ multiply the old running denominator ℓ by em−mₙ to convert it to the new basis, then add the new block’s terms (already in the new basis).

Diagram 2 — why it’s exact. Each block updates a running max m and denominator ℓ; when a later block raises the max, the old partial is rescaled by em−mₙ so nothing is dropped. The streamed ℓ = 1.578 equals the naive whole-row sum to the last digit — the trick costs zero precision, only one extra rescale per block.

FlashAttention on the GPU — split-M, split-KV, stream-K

Scope note. Same online-softmax math as above — this is how it is mapped onto the GPU. The kernel-internals owner is K17 (D51 — the attention-parallelization design space); the batched-decode wave-balance half is K4. Shown here because it is the direct continuation of the recurrence above. Terms: a CTA (cooperative-thread-array / threadblock) is the unit the GPU hands to one SM (streaming multiprocessor — ~132 on an H100); throughput ≈ total_CTAs / num_SMs per wave, so the design question is always “which axis do we split across CTAs to get enough of them, without a wasteful reduction?”

The split axis is decided by how thick Q is — the number of query rows. Prefill has a thick Q (many rows) so splitting Q alone fills the GPU; decode has a thin Q (1 row) so you must split the KV instead and pay a merge; batched-decode with ragged lengths needs fixed-size work units so a long sequence cannot starve short ones.

Diagram 3 — the geometry, in K0’s L-shape convention. Both scenarios are Q·Kᵀ→S then S·V→O (left = bottom-left, right = top, result = corner; touching edges = the contracted dim). Scenario 1 splits s_q — an outer dim of O — so each CTA owns disjoint output rows and you simply concatenate the blocks (split-M, no reduction). Scenario 2 splits s_k — the contracted dim of S·V — so every CTA emits a full-width partial of the same row that must be summed (split-KV, the online-softmax merge). Scenario 3 is Scenario 2’s geometry with fixed-size chunks + work-stealing for load balance (stream-K). The one rule: split an outer dim → concatenate; split the contracted dim → sum.

Common confusion — “v1 already tiles K/V; isn’t that the same as Flash-Decoding’s split-KV?”

TL;DR — no, and the table above already answers it: look at which column each one sits in. v1’s “K/V tiles” is in the Sequential inside one CTA column; Flash-Decoding’s “KV splits” is in the Parallel across CTAs column. Same K/V pieces, opposite purpose — v1 tiles K/V to save memory; Flash-Decoding splits K/V to manufacture parallel work.

  • v1’s K/V tiling is a loop, not parallelism. One CTA owns an entire (batch, head) and walks the K/V tiles one after another — only so the running (m, ℓ, O) softmax state fits in SRAM (the online-softmax memory trick). The CTA count is unchanged at batch×heads. When the loop ends, that one CTA already holds the complete output row.
  • Flash-Decoding’s split-KV is a grid axis — genuine parallelism. The K/V of a single (batch, head) is handed to several CTAs that run at the same time on different SMs. That is what raises the CTA count to batch×heads×KV-splits — the whole point, because at decode s=1 leaves batch×heads alone far too few to fill the GPU.
  • That is why only Flash-Decoding needs a merge. The v1 CTA saw all of K/V, so its (m, ℓ, O) is already final — nothing to combine. Each Flash-Decoding CTA saw only its slice, so every one emits a partial and a second kernel must merge them (online-softmax across CTAs).
  • Online softmax runs in both — just in different places. v1: inside the one CTA’s K/V loop. Flash-Decoding: inside each CTA’s loop over its slice and again in the cross-CTA merge.

Each variant on its own. The grid (what runs in parallel across CTAs) and the loop nest (what runs sequentially inside one CTA) are what change between v1 → v2 → Flash-Decoding — one graph each.

Diagram 5 — FlashAttention v1. The grid is only batch×heads; the loop nest is K/V-outer, Q-inner, so each CTA repeatedly reloads O,ℓ,m. Both are what v2 changes.

Diagram 6 — FlashAttention v2. Adds the Q-tile axis to the grid (split-M) and swaps the loop so each CTA owns one Q-tile and loops K/V inside — fixing v1 for prefill, but collapsing back to v1 at decode (s=1).

RoPE, and why only a prefix of KV is reusable

  • Why position info is needed. Q·Kᵀ is order-blind — permute the tokens and the scores are unchanged. The model must be told token order. RoPE (rotary position embedding; the scheme in Llama, Mistral, Qwen, DeepSeek) injects it by rotating Q and K by an angle proportional to absolute position.

Mechanism (one 2-D pair shown; every pair of the vector rotates, each at its own fixed frequency θ_p). The vector for a token at absolute position t has its pair rotated by angle t·θ_p — same content, different position ⇒ different rotation ⇒ different K.

  • The key property. The dot product of a RoPE-rotated Q at position i and rotated K at position j depends only on (i − j) — relative position — even though the rotation is applied per-token by absolute position.
  • Problem — can a cached span be reused anywhere? K2 Role 2 wants to reuse a span’s cached K,V across requests. But K is rotated by absolute position: the same tokens sitting at different positions produce different K. A cached span dropped in at a shifted offset has every K wrong.
  • Solution — only a true prefix is reusable as-is. Reuse is valid exactly when the tokens are at the same absolute positions — i.e. a leading prefix (positions 0..k). Arbitrary mid-sequence or substring matches are not free. On the trace: at T3 "I like cat, why? <resp> I don’t like dog", the leading "I like cat, why?…" tokens occupy the same positions 0..k as in T1/T2, so their K,V are bit-identical and reusable; had "I don’t like dog" been prepended instead, every prior K would shift and none would be reusable. This is the exact fact K2’s “why a prefix is the reusable shape” cites — derived here.

Bonus — How Speculative Decoding Works

Scope note. Speculative decoding is owned by K13 — the rejection-sampling correctness proof, the draft-mechanism spectrum (n-gram → Medusa → small draft model → EAGLE-2), and the batch-adaptive crossover (helps at small batch, hurts past K1’s B*) all live there. This diagram is a visual recap on the canonical trace.

One-sentence thesis. Decode is HBM-BW-bound (cite K1 §decode): a normal decode step reads all target weights from HBM to produce one token. Speculation amortizes that single weight-load over K+1 candidate token positions — if most candidates are accepted, the same big-model forward yields many tokens.

Why it’s lossless (the 30-second version)

  • The fix on first mismatch. If position i’s draft token doesn’t match the target’s sample, you don’t just drop the draft — you sample position i from a corrected target distribution, constructed via rejection sampling so the overall output distribution equals the target’s exactly. Positions i+1..K are discarded.
  • Result. Output is statistically identical to running the target alone — speedup is wall-clock only; quality is bit-equivalent in distribution. (Proof & the corrected-distribution formula: K13.)

Acceptance rate α and the speedup

  • α = fraction of draft tokens the target accepts. In this diagram α = 2/4 = 0.5. In practice α depends on draft quality and workload — deterministic chains (code, math, long reasoning, cite K19) push α high.
  • Expected accepted tokens per target step: E = (1 − αK+1)/(1 − α). With α = 0.5, K = 4 ⇒ E ≈ 1.94 (your big-model step now yields ~2 tokens instead of 1). The diagram shows a luckier step (3 tokens) — tail of the same distribution.
  • The catch (cite K13 / D11). Speedup only fires while decode is memory-bound. Past K1’s ridge B* the weight-load is already amortized over B sequences → decode is compute-bound → speculation’s extra draft + verify FLOPs become pure overhead and it slows decode. Adopt iff batch < Bspec.
