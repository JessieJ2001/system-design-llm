K12 — Multimodal & Image Offload (vision-language serving)

How do you build a scalable multimodal serving path — when the vision encoder has the opposite roofline to the LLM and one image explodes into 1–3k prefill tokens — without a ViT idling in every LLM replica and starving the GPU fleet? This page derives the encoder cost profile, then encoder↔LLM disaggregation, the embedding handoff, the embedding cache, and ragged variable-resolution batching.

SLIs This Page Reduces

SLIs this page reduces (the reverse index of K1 §3.12):
  • TTFT / Goodput (image-dominated prefill) — one image is 1–3k prefill tokens; encoder offload + the embedding cache cut the image-token prefill cost off the LLM’s critical path.
  • Throughput / Encoder-MFU — ragged variable-resolution batching (D27) kills the ViT padding waste that collapses encoder MFU.
  • Cost ($) — a separate autoscaled encoder pool (D41) stops a ViT sitting idle in every LLM replica burning HBM + GPU-hours.
  • cache-hit (embedding cache) — a re-uploaded image / re-sent document skips re-encoding entirely (the K12-owned cache; the KV-side twin is K2/D32).

What This Page Is (and Is Not)

TL;DR. An image becomes 1–3k embedding tokens via a ViT whose roofline is the opposite of the LLM’s; give the encoder its own autoscaled pool, cache its output, batch it raggedly, and stream the embeddings to the LLM-prefill pool.

  • Is: why the encoder is a separate GPU cost profile (derived, the K5/D35 argument on modality); D27 (ragged variable-resolution image batching — dynamic tiling); D41 (encoder↔LLM disaggregation + the embedding handoff = the K5/D8 analog); the encoder-output embedding cache (content-addressed); the image-tokenization-granularity rule-11 sweep.
  • Is not: the KV cache for the resulting image tokens + its content-addressed reuse key (K2/D32 — the KV-side twin; same image, two caches, owned apart); semantic / embedding-similarity caching (out of curriculum); the vision model’s training (training-side); OCR / PDF→text parsing before the LLM (application-layer); the embedding-handoff cost primitive / fabric topology (K1 / K20). The Multimodal Spine (stated up front, before §1)

The chain — each link a section below; the decision-tree SVG is this chain:

  • Image/audio enters as raw bytes → a ViT/encoder turns it into embedding “tokens” (1 image → 1–3k LLM tokens).
  • New problem: the encoder has the opposite roofline to the LLM — compute-bound, fixed-depth, no KV, all-forward (vs the memory-BW-bound autoregressive LLM).
  • New problem: colocating a ViT in every LLM replica wastes the fleet — the ViT idles between image requests, burns LLM HBM, and image tokens dominate prefill.
  • → disaggregate the encoder pool — the K5/D35 argument on modality instead of phase; ship the embeddings (D41, the K5/D8 analog).
  • New problem: repeated images are re-encoded every request.
  • → a content-addressed embedding cache (K12-owned; the KV-side twin is K2/D32).
  • New problem: variable image sizes make the ViT batch half padding ⇒ encoder MFU collapses.
  • → dynamic tiling / ragged encoder batch (D27 — the K4 cu_seqlens pattern applied to tiles).
  • Objective: image-dominated prefill served at the SLO, encoder MFU high, no ViT idling in the LLM fleet.

1. The Problem It Solves

Problem→Solution stub. Problem: vision/audio encoders have a different GPU cost profile than the LLM, and one image blows up the prefill token count — 1 image ≈ 1–3k LLM tokens, so a 4-image prompt ≈ 12k prefill tokens and the LLM prefill cost is dominated by image tokens, not text. Colocating a ViT in every LLM replica then wastes the fleet (the ViT idles between image requests; the LLM idles during encode; encoder weights eat LLM HBM). Solution (what the page builds): derive the encoder cost profile, give it its own autoscaled pool, cache its output content-addressed, batch it raggedly, and stream the embeddings to the LLM-prefill pool.number: image-token prefill cost vs text-token prefill cost on the trace (1.5k img tok vs 6 text tok) — cites K1 §3 (#prefill, #roofline) · canonical: gpu-inference-cost-shapes.html

2. The Mechanism — the Tokenized-Prompt Walkthrough

Where the encoder sits — image embeddings enter the residual stream as positions

Problem→Solution stub (architecture placement, hard req 6). Problem: the LLM only consumes token embeddings; an image is raw pixels. Solution: a ViT/encoder maps the image to [n_img_tokens, d_model] embedding vectors that are injected into the sequence as ordinary positions; the LLM then computes Q,K,V over them exactly like text (cite K0 for the [n_img_tokens, d_model] residual-stream shape). The encoder is a pre-stage before the LLM forward pass — that placement is why it can be a separate pool.no number (placement fact) — cites K0 (image embeddings enter the residual stream as tokens; the [n_img_tokens, d_model] shape) · diagram: image → ViT → embedding tokens spliced into the text-token sequence

The walkthrough on a real multimodal prompt — what the encoder does to the sequence

Problem→Solution stub (hard requirement 6 — reader’s first contact, concrete & plain). Canonical trace, multimodal variant: a user sends [a cat photo] + "I like cat, why?". Tokenize: the photo → ViT → ≈1.5k embedding tokens; the text → 6 tokens; concatenated into one ≈1506-position sequence. Prefill: the LLM prefills all ≈1506 positions in parallel (causal mask) — image tokens dominate the prefill compute ≈250× the text. The ViT runs once, all-forward, no KV. Decode: the answer decodes token-by-token; the image is never re-encoded (its embeddings/KV already resident). Why the encoder is a separate GPU cost profile (derived — the page’s home)

Problem→Solution stub (the page’s opening derivation, sub-bulleted). Problem: the ViT and the LLM are run on the same GPU but stress it oppositely. Solution, derived: (a) the ViT is fixed-depth, non-autoregressive, all-forward, no KV → high stable arithmetic intensity → compute-bound, high stable MFU; (b) the autoregressive LLM decode is memory-BW-bound (arithmetic intensity ≈1, cite K1 #decode/#roofline) — the opposite roofline; (c) different roofline ⇒ different scaling unit ⇒ the same K5/D35 disaggregate-or-colocate inequality, but on modality instead of phase; (d) the number that justifies the whole page: 1 image → 1–3k tokens, a 4-image prompt ≈ 12k prefill tokens — prefill is dominated by image tokens. This single derivation motivates encoder offload (D41), the embedding cache, and the K2/D32 KV consequence.number: ViT MFU vs LLM-decode MBU on the trace; 4-image prompt prefill-token count — cites K1 #prefill/#roofline/#decode, K5 #d35 (the inequality this applies on modality)

Deep dive D27 — ragged variable-resolution image batching (dynamic tiling)

Deep-dive stub. Problem: images arrive at different resolutions; padding every image to the max patch count → the ViT batch is half padding → encoder MFU collapses (a 256×256 thumbnail padded to a 4K screenshot’s patch count ≈ 16× wasted encoder FLOPs). Solution: dynamic tiling (split each image into native-resolution tiles, variable tile count/image) + a ragged encoder batch (the K4 cu_seqlens ragged-batch pattern applied to tiles) + bucketed resolutions. Crossover: bucket granularity — few buckets = pad waste, many = poor batch fill; pick the bucket set that keeps encoder MFU above target.number: padding waste for a batch {4K screenshot, 256² thumbnail, 1024² photo} unbatched-padded vs tiled+packed on the trace (≈16×) — cites K1 (encoder roofline), K4 (the cu_seqlens ragged-batch pattern) · real: Qwen2-VL dynamic resolution, InternVL tiling, NaViT (patch-n’-pack) + URLs

Deep dive D41 — encoder↔LLM disaggregation & the embedding handoff (the K5 pattern on modality)

Deep-dive stub (the K5/D8 analog; plan named “encoder↔LLM disaggregation” but never derived the handoff). Problem: the ViT pool and LLM pool have different rooflines and must scale independently, but the encoder output [n_img_tokens, d_model] must cross from the encoder GPU to the LLM-prefill GPU. Solution: separate autoscaled pools + an embedding transfer that is the K5/D8 analog — stream image embeddings while the encoder finishes the next image; GPUDirect-RDMA zero-copy. Crossover (both directions): embedding_bytes / BW_link vs re-running the encoder on the LLM GPU — colocate when the encoder is cheap and the link slow; disaggregate when image-heavy and the link fast.number: 8 img/s × 1.5k tok × d_model=8192 × 2 B per-second embedding volume vs the re-encode FLOPs on the trace — cites K1 (encoder cost), K20 (BW_link), K5 #d8 (the layer-wise pipelined-handoff analog), #d35 · real: NVIDIA Dynamo multimodal, vLLM encoder offload + URLs

The embedding cache — content-addressed (the K12-owned cache; the three-way disambiguation)

Problem→Solution stub + the rule-of-two boundary. Problem: an identical image re-uploaded each turn pays full ViT FLOPs again. Solution: a content-addressed cache keyed by H(raw image bytes) (or post-encoder embedding hash) + preprocessing params (resize, tiling, frame-sampling fps) + encoder model/version — a hit skips re-encoding entirely. The three-way disambiguation (mandatory, prevents the common conflation): ① the encoder-output embedding cache — skip re-encoding — K12 (here); ② the LLM KV cache for the image tokens — skip LLM prefill — K2/D32; ③ semantic / embedding-similarity caching — out of scope (named, not covered). Different keys, hit conditions, owners.number: ViT GPU-seconds saved per cache hit on the trace (1.5k-token image) — cites K2 #d32 (the KV-side twin & its content key) · real: vLLM multimodal prefix-cache / mm_hash, SGLang multimodal RadixAttention + URLs · scope boundary stated in place (rule of two)

Rule 11 — the image-tokenization granularity policy

Selection-sweep stub. fixed-low-res (cheap; cliff: OCR / dense-image quality loss) → fixed-high-res (accurate; cliff: prefill-cost explosion on small images) → dynamic tiling (Qwen2-VL; cliff: variable token count complicates batching/KV → D27) → adaptive token budget / perceiver-resampler (compress to N tokens regardless of resolution; cliff: information bottleneck). Two-sided: too few image tokens → quality; too many → the prefill-cost explosion. Pick per workload — OCR/document → tiling/high-res; thumbnail-heavy → resampler.number: image-token count & prefill cost per policy on the trace (the ≈16× small-image waste) — cites K1 #prefill · real: fixed (LLaVA-1.5), dynamic tiling (Qwen2-VL/InternVL), resampler (Flamingo/Qwen-VL Perceiver) + URLs

4. Alternatives & When Each Wins

Stub (≥2, disqualifying constraint per alt — the D41 spine). (a) Colocated ViT per LLM replica — disq: the ViT idles between image requests and burns LLM HBM with encoder weights; wins only at tiny image volume / cheap encoder / slow link (the D41 crossover read the other way). (b) No embedding cache (re-encode every request) — disq: pays full ViT FLOPs per repeated image; wins only when images are never repeated. (c) Fixed-resolution, no tiling — disq: pad waste or quality loss; wins only when all images are the same size. (d) OCR / PDF→text instead of a visual encoder — disq: application-layer, discards visual information; out of the GPU-serving scope. Each: when it actually wins.number: the crossover value disqualifying each (image volume × link BW for (a); repeat rate for (b); size variance for (c)) — cites K5 #d35, K1, K20 · contrast L2 (dedicated multimodal scenario)
