K2 — The KV Cache (both roles)

How do you build a scalable, large-scale KV cache? The KV cache is the dominant HBM memory and bandwidth resource in LLM serving, and it earns its keep by being reused at two scopes: Role 1 — intra-request reuse (universal: turns O(L³) decode compute into O(L²); then sized, paged, quantized, tiered, evicted); and Role 2 — inter-request reuse (“prefix caching”) plus the distributed-systems problem of routing a request to the GPU that already holds the reusable KV.

SLIs This Page Reduces

SLIs this page reduces (the reverse index of K1 §3.12):
  • Role 1 → TPOT/ITL, MBU, batch-occupancy, throughput — by shrinking the resident per-sequence KV (GQA lever, paging, quant, window, tier/evict) so more sequences batch and the memory-bound decode step moves fewer bytes.
  • Role 2 → KV-hit / cache-hit rate, TTFT, goodput — by reusing already-computed KV across requests and routing each request to where its reusable KV lives, so shared prefill is never recomputed.

The Two Roles (the page’s spine)

A reader must see both roles, and that Role 2 is an optional layer on the universal Role 1 — not a separate subject.

  • Role 1 — avoid recomputation within a single sequence (intra-request). Without a cache, generating token N recomputes K,V for tokens 0..N−1 every decode step → O(L³) over a length-L generation. With a cache, each token’s K,V is computed once → O(L²). Universal — built into every inference engine, every decode step benefits; not a feature. The cost it creates — O(L) HBM per sequence that dominates GPU memory — is why the rest of Role 1 (paging, quant, tier, evict) exists.
  • Role 2 — avoid recomputation across requests (inter-request, “prefix caching”). When requests share a prefix (system prompt, few-shot, growing history), that prefix’s K,V can be reused across them. An optional layer on Role 1 (APC / RadixAttention), useful only when prefixes are shared, plus the distributed problem of routing to the GPU that holds the KV. Prefix-shaped reuse is dominant and deepest-covered — but it is one technique, not the page’s assumption: the page owes the full routing-for-cache-hit spectrum. The route is not assumed to be a hash of the prompt’s prefix.

The Page as One Decision Tree

Every KV-cache decision is one chain: token N needs all prior K,V → cache it (Role 1) → the cache now dominates HBM → shrink/page/quant/tier it → working-set KV and reuse KV now contend for one pool → the overlap across requests is still recomputed → reuse it (Role 2, only if reuse pays for its overhead) → the reusable KV lives on some GPU → route to it → the hot shared prefix saturates that GPU and the shared cache leaks across tenants → replicate/split it + namespace per tenant. 2. The Mechanism — the Tokenized-Prompt Walkthrough

Where the KV cache lives — attention only, never the MLP

Problem→Solution stub. Problem: attention is the only layer where tokens interact — token N’s attention needs K,V from every prior token, so recomputing them each step is the expensive cross-token dependency. Solution: cache K,V (attention tensors). The MLP/FFN is purely per-token — FFN(x)=W₂·act(W₁·x), token N’s output depends only on token N → nothing to cache. State this placement before any sizing math.cites K0 #sdpa (cross-token attention shape) · no number (placement fact) · diagram: tag attention vs MLP on the K0 forward-pass block

The walkthrough on "I like cat, why?" — prefill vs decode, with vs without

Problem→Solution stub (hard requirement 6 — reader’s first contact). Canonical trace T1 = "I like cat, why?" (6 tokens), walked token-by-token. Prefill = one forward pass over all 6 tokens in parallel, one Q·Kᵀ matmul with a causal mask (pos 5 attends 0–5, pos 4 attends 0–4, …); KV cache written here, not read. Decode next token: without cache → re-run the whole forward pass over all prior tokens, recompute K,V each step (the O(N²)/step waste); with cache → K,V for 0..N−1 already stored, compute Q,K,V for only the new token, attend its Q vs cached K + own, append new K,V. Why the cache exists — space-trades-compute, derived

Problem→Solution stub (the page’s opening derivation, builds on the walkthrough). Chain: (a) attention is O(N²) — Q·Kᵀ is [N×N], O(N²·d_head)/head/layer (cite K0, don’t re-derive); (b) no cache ⇒ Σ O(n²) = O(L³) to generate length L; (c) the trade one-liner — without: mem ≈ 0, compute O(L³); with: mem O(L)/seq (dominates GPU memory → why the rest of Role 1 exists), compute O(L²); (d) compute-bound vs memory-bound + the roofline crossover (cite K1 #roofline); (e) batching flips decode’s roofline (AI ∝ B; at large B decode is compute-bound) — sets up K4.numbers: O(L³)→O(L²) on the trace; ridge ≈295 FLOP/byte & B*≈295 cited from K1 #roofline · cites K0 #sdpa, K1 #roofline #decode · forward-links K4

Deep dive D4 — the block table (the one indirection both roles stand on)

Deep-dive stub. Problem: paging, KV-sharing, and swap each look like separate machinery. Deep dive D31 — KV-cache layout for kernel-efficient attention [ADDED — on request]

Deep-dive stub (read-path sibling of D4; K2 owns the layout, K17 owns the kernel). Problem: the decode attention read is a memory-bound thin GEMV against cached K,V (arithmetic intensity ≈ 1) — so achieved MBU is set by the physical KV layout in HBM, not by FLOPs; a correct algorithm on a bad layout moves the same bytes at a fraction of HBM BW (uncoalesced gather → MBU collapse). Working-set KV vs reuse-cache KV share one physical pool [ADDED — not in plan; the Role-1↔Role-2 boundary]

Problem→Solution + selection-knob sweep (rule 11). Problem: the same HBM block pool holds two populations — KV of running sequences (Role 1 working set, losing it preempts the sequence) and KV kept for reuse by finished requests (Role 2, free to drop). Eviction must distinguish them: dropping a reuse block is free; “evicting” a running block is preemption, a K4 scheduler action. Effective reuse capacity = pool − Σ active sequences − pinned-by-in-flight-read (D4); under high concurrency it collapses toward ~0 and Role 2 silently stops helping. Solution: the partition policy — the spectrum (fully shared LRU-across-both → soft-reserve with borrowing → hard split), its two-sided cliff (too much reserved for reuse starves the working set → preemption storms; too little → ~0 hit rate), and the watermark that arbitrates between them.number: reuse-cache capacity vs concurrency on the trace (the collapse point); reserve-fraction crossover · cites K4 (preemption = evicting a running block) · canonical: llm-bottlenecks.html#kv-exhaustion

Deep dive D52 — cross-tenant cache isolation: the prefix-cache timing side-channel [ADDED — on request; the confidentiality gate, sibling to D5’s correctness gate]

Deep-dive stub (selection-knob sweep, rule 11; pairs with D5). Problem: a content-keyed shared cache makes TTFT data-dependent on other tenants’ history — tenant B times its own first token after sending a candidate prefix and learns fast ⇒ tenant A already sent it: a membership / secret-extraction oracle (recover an API key or PII a few tokens at a time by binary-searching prefixes against the hit/miss timing edge). D5 is the correctness gate (a wrong key serves wrong KV); D52 is the confidentiality gate (the right KV, but its mere existence leaks). Solution — the isolation spectrum (weakest→strongest, each with its hit↔leak cliff + a real system): (0) global sharing — max hit, full oracle; (1) share only operator-public prefixes (system / tool-def prompts) while per-tenant-namespacing all user content — keeps the high-value shared-prefix hit, kills the user-content oracle; (2) per-tenant cache namespace folded into the cache key (D5) — zero cross-tenant leak, zero cross-tenant reuse; (3) oracle-masked TTFT (constant-time / added jitter / cache decision off the timed path) — preserves cross-tenant hit, pays latency or a noise budget. The crossover, as a formula: the leak exists while TTFT_hit + noise < TTFT_miss distinguishably; pick the weakest tier whose residual distinguishability is below the deployment’s leakage budget — tier (1) is the usual staff answer (the system-prompt hit is most of the economic win, and the user-content oracle is removed).
