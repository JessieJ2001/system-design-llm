K20 — Interconnect & Collectives (the master constraint, not FLOPs)

Why is the interconnect topology — not FLOPs — the master constraint that decides every parallelism tuple: because each collective (TP all-reduce, EP all-to-all, PP P2P, CP ring) must live on a bandwidth tier it can afford, and the single fact “TP stays inside the NVLink domain; everything that crosses it pays the InfiniBand tax” is what K6 actually obeys? This page derives the bandwidth hierarchy + the collective-cost models, then D28 (NCCL cost & rail-aware placement) and D48 (the NVL72 domain-size shift — the curriculum’s master sensitivity).

SLIs This Page Reduces

SLIs this page reduces (the reverse index of K1 §3.12) — honest framing: this is a master-constraint page; it decides which tier a collective lives on so it does not become the bottleneck:
  • TPOT / ITL (primary) — the TP all-reduce / EP all-to-all is on the decode critical path; on NVLink it hides, across IB it explodes ~18× — this page keeps it on the affordable tier.
  • TPOT p99 specifically — every collective is a barrier, so per-token p99 is the slowest rank’s collective, not the mean; this page bounds that tail (in-domain, rail-optimized, contention-isolated — not just the average case).
  • MFU — a collective stalled on the wrong tier — or an “overlapped” one still stealing SMs/L2 from the GEMM — idles or starves every GPU in the group; correct placement and channel/contention tuning keep compute fed.
  • Max feasible TP/EP within budget — the NVLink-domain size is the ceiling on in-domain parallelism (the D48 master sensitivity).
  • Constrains K6’s tuple — “TP ≤ N_dom; crossing pays the IB tax” is the single fact that decides the parallelism tuple (not FLOPs).

The Interconnect Spine (stated up front, before §1)

  • Every parallelism degree (TP/PP/EP/CP) is bounded by which GPUs can talk to which, how fast — topology, not FLOPs, is the master constraint.
  • → the bandwidth hierarchy: HBM ≫ NVLink/NVSwitch ≫ IB/RoCE ≫ Ethernet.
  • → each collective must live on a tier it can afford (all-reduce/TP on NVLink; P2P/PP survives IB; all-to-all/EP is the hardest).
  • → the design rule: TP stays inside the NVLink domain; everything that crosses it pays the IB tax.
  • → this single fact decides the K6 tuple (not FLOPs).
  • → but the M/BW cost model is idealized: even on the right tier the collective steals SMs/L2 from the GEMM it is supposed to hide under (the NCCL channel tax), and the fabric is not a clean pipe — congestion / all-to-all incast / RoCE-lossless (PFC/ECN/DCQCN) / GPU↔NIC NUMA affinity / non-uniform intra-node links.
  • → and collectives never run alone: TP/PP/EP/DP/CP + (under K5) KV-transfer share the same NICs/links and contend; every collective is a synchronization barrier ⇒ per-step p99 = the slowest rank, not the mean (the decode-TPOT-p99 story, jitter-amplified with group size).
  • Problem: same TP degree, different physical placement → one is 3× slower on the all-reduce (D28).
  • → ring-vs-tree by message size + rail-optimized placement + topology-aware NCCL + in-network reduction (D28).
  • Master sensitivity: NVL72 enlarges the NVLink domain → the IB cliff softens, the tuple shifts, and the fault blast radius grows ~9× (D48, the K22 trade).
  • Objective: place every collective on a tier it can afford; state the assumed N_dom — the formula is the invariant, the number is hardware-dated.

Prefill vs decode are different collective regimes (the asymmetry — mirrors K1/K2’s roofline spine)

Problem→Solution stub (curriculum prefill/decode spine, applied to collectives). Problem: “the all-reduce” is treated as one cost, but prefill and decode hit the same collective in opposite regimes. Solution, derived: the per-block all-reduce message is B·s·d_model·2 B — in prefill s is the whole prompt ⇒ a large message ⇒ bandwidth-bound, the ring’s M/BW term dominates; in decode s=1 (one new token) ⇒ a tiny message ⇒ latency-bound, the ring’s 2(N−1)/N hop term + fixed kernel-launch dominate, and this collective is on the critical path of every token. This asymmetry is why ring-vs-tree (D28) is decided at decode message sizes and why decode p99 is collective-tail-bound, not BW-bound.number: prefill vs decode all-reduce message size on the trace + which ring term dominates each — cites K1 (#prefill/#decode + ring primitive), K2 (the decode message is one token of KV-width activation)

The M/BW model is idealized — two ways the real cost is higher

Problem→Solution stub (NEW — page-level consideration, not a new catalog D-number). The §2 derivation prices every collective at the ideal M/BW. Two production realities make it strictly worse, each Problem→Solution:
(a) The collective is not free even when “overlapped”. Problem: NCCL all-reduce/all-to-all kernels consume SMs + L2 bandwidth that the GEMM needs → an all-reduce hidden under the next matmul still slows that matmul (the comm–compute SM-contention tax) — so “it hides on NVLink” is necessary but not sufficient. Solution: tune the NCCL channel/SM count (NCCL_MIN/MAX_NCHANNELS) so the collective takes just enough SMs to stay hidden without starving compute. Rule of two: K7/D37 & K9/D30 own the overlap technique; K20 owns the SM-contention tax that bounds how much can overlap.
(b) The fabric is not a clean pipe. Problem: IB/RoCE fat-trees suffer static-routing (ECMP) hash collisions, congestion, and all-to-all incast (many senders → one receiver → switch-buffer overflow → drops/retransmit storms); RoCE specifically needs lossless config (PFC/ECN/DCQCN) and is fragile vs native IB; GPUDirect needs GPU↔NIC PCIe/NUMA affinity or it bounces through host RAM; not every intra-node link is NVLink (PCIe-only nodes / NVLink “islands”). Solution: adaptive routing + congestion control, RoCE DCQCN tuning (or pay for native IB), NUMA-aware NIC↔GPU pinning, and treat real achieved BW (not spec) in the cost model.numbers: SM-contention slowdown of the overlapped GEMM (% of a decode step); incast/congestion achieved-BW vs ideal M/BW on the trace — cites K1 (BW table + ring primitive), K5 (KV-transfer shares the fabric) · real: NCCL channel tuning, NCCL/IB adaptive routing, RoCEv2 DCQCN, GPUDirect-RDMA NUMA affinity + URLs · canonical: llm-bottlenecks.html (interconnect saturation)

Collectives don’t run alone — concurrent contention + the barrier tail

Problem→Solution stub (NEW — page-level consideration, not a new catalog D-number).
(a) Concurrent-collective interference. Problem: a frontier deployment runs TP all-reduce (NVLink) inside DP replicas while EP all-to-all + PP P2P + (under K5) the KV-transfer all hit the same IB rails simultaneously → they contend for the same NICs/links, so a collective’s real cost depends on what else is in flight, not just its own M/BW. Solution: dedicate rails to specific collectives, NCCL stream priorities, fabric traffic classes/QoS. Rule of two: K6 picks the tuple; K20 models the link contention between the chosen axes’ collectives (and K5’s KV-transfer as one more flow on the same wires).
(b) Every collective is a barrier ⇒ p99 = the slowest rank. Problem: a collective synchronizes all ranks, so per-step latency is the max over ranks, not the mean; jitter sources (a noisy shared link, thermal throttle on one GPU, an ECC scrub, OS jitter, an unlucky NCCL channel) get amplified as group size grows — and at decode this barrier is on the critical path every token, so it directly sets TPOT p99. Solution: bound the tail at the source (in-domain, rail-optimized, contention-isolated, jitter-monitored), not just the average. Rule of two: K22 owns collective faults (a dead/stuck rank); K20 owns the tail/jitter of a healthy collective.numbers: TPOT p99/p50 spread vs group size from the barrier-max model; the per-token cost of one contended rail on the trace — cites K1 (#decode + ring primitive, the SLI list), K5 (KV-transfer as a co-tenant flow) · canonical: llm-bottlenecks.html (interconnect saturation)

4. Alternatives & When Each Wins

Stub (≥2, disqualifying constraint per alt). (a) Treat the network as one flat fabric — disq: ignores the order-of-magnitude IB-tax cliff → TP across IB collapses; wins never (it is the naive baseline this page refutes). (b) More TP across nodes to fit a bigger model — disq: the per-layer all-reduce crosses IB → ~18× slower; wins never — use PP across domains instead (the K6/K8 escape). (c) Slower Ethernet instead of IB/RoCE for inter-node — disq: even PP P2P struggles; wins only for embarrassingly-parallel DP-replica traffic with no cross-replica collective. (d) Designing the switch/NIC silicon — disq: that is hardware engineering, not serving topology; out of scope — name the parts, flag the boundary, do not cover.number: the crossover disqualifying each (IB-tax for (a)/(b); Ethernet collective cost for (c)) — cites K1 (BW table), K6/K8 (the PP escape) · contrast L1 (tuple depends on N_dom)

5. Failure Modes

Stub (cite, never inline-redefine). Interconnect saturation — TP all-reduce / EP all-to-all exceeds the NVLink-domain BW → the collective is the bottleneck, every GPU in the group idles; first move: keep it in-domain / rail-optimize (llm-bottlenecks.html — interconnect saturation). TP-across-IB collapse — TP exceeded N_dom → ~18× per-token latency; first move: cap TP at N_dom, switch to PP (llm-bottlenecks.html). Non-rail-optimized all-to-all — oversubscribes a switch → the slowest rank gates EP; first move: rail-aware placement (llm-bottlenecks.html — hot-expert/all-to-all). NVL72 blast-radius surprise — bigger domain → a single fault takes down ~9× more GPUs (gpu-failure-modes.html — GPU-group blast radius; the D48/K22 trade). Stale topology assumption — a derivation that hard-coded N_dom=8 is wrong on NVL72 (gpu-failure-modes.html — version/hardware-mismatch). Degraded (not dead) interconnect — silent BW loss (NEW) — an NVLink lane CRC-replaying down to lower BW, IB symbol-error rate climbing, or NCCL routing around a dead link at reduced bisection → throughput drops ~30% with nothing crashing (the interconnect analog of D50’s compute SDC); first move: alert on NVLink replay / IB symbol-error / NCCL-route-change counters, not just on dead processes (gpu-failure-modes.html — degraded-link silent BW loss). Rule of two: K22/D50 owns compute SDC + dead/stuck rank; K20 owns degraded-link detection. Communicator cold-start (NEW) — ncclCommInitRank + the topology all-gather is seconds; a group reformed after a K22 fault must rebuild its communicator before serving → this adds to the K21 cold-start / K22 fast-reformation budget (K20 owns the comm-setup cost; K21/K22 own the budget it feeds).cites llm/canonicals/llm-bottlenecks.html (interconnect saturation, TP-across-IB, all-to-all), llm/canonicals/gpu-failure-modes.html (NVL72 blast radius, stale-topology, degraded-link silent BW loss); cross-link K22/D50, K21 · ACTION: seed/extend these canonical entries (incl. the new degraded-link entry) in this same change
