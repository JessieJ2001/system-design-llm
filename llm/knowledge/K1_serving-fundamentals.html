What This Page Is (and Is Not)

TL;DR. This is the curriculum's single home for the serving math. Every memory, FLOP, throughput, fleet, and dollar number on any other page is computed with a formula defined here, in terms of the dimensions defined in K0.

  • Is: the hardware constant table, the core formulas with their derivations (including the 2× rule — this page is its home), the roofline that decides compute- vs memory-bound, the collective-cost primitives the parallelism pages cite, and the SLI list every scenario's monitoring section cites.
  • Is not: the dimensional vocabulary (d_model, d_head, n_kv_heads, GQA) — that is K0's; this page uses those terms, it does not re-define them. It also does not optimize anything: paging, batching, parallelism, quantization each get their own page that cites these numbers.
Dedup rule. The curriculum is self-contained. The math here is derived to serving depth on this page; no llm/ page defers to an external textbook for a constant. If a number is wrong, it is wrong in exactly one place — here.

The Question This Page Answers, as One Tree

Every serving decision reduces to one chain: which phase am I in → which resource is it bound on → what formula sizes that resource → how many GPUs → how many dollars. The tree is the page's spine; each box below is a numbered section.

  • ① Prompt path — the whole prompt is processed in one compute-bound forward pass (prefill).
  • ② Token path — each subsequent token is one memory-bandwidth-bound step (decode).
  • ③ Roofline — arithmetic intensity decides which resource each phase is bound on.
  • ④ Utilization — MFU/MBU convert peak hardware specs into achievable throughput.
  • ⑤ Throughput → fleet → cost — the chain that turns one node's rate into a monthly bill.
  • ⑥ Collectives & SLIs — the cost of splitting work across GPUs, and how you tell it is healthy.

1. The Problem This Page Solves

TL;DR. Without one frozen sheet, every page invents its own constant for H100 bandwidth, its own FLOP rule, its own cost formula — and the numbers stop agreeing across the curriculum. This page exists so they agree by construction.

The bottleneck shape

  • Problem: a scenario page that re-derives "H100 = ~3 TB/s, call it 3" and another that uses "3.35" produce fleet sizes that differ by 12% — the reader cannot tell which is right.
  • Solution: one table of frozen hardware constants (§3.1) and one set of formulas; every number elsewhere carries a citation back to a row or formula here. A wrong number is then wrong in exactly one editable place.

Why a "number sheet" is itself a scaling concern

  • The curriculum scales to 23 knowledge pages and 9 scenarios. O(pages²) pairwise inconsistencies is the failure mode; a single cited SoT makes it O(pages).
  • This is the same cite-don't-restate discipline as the generic site's back-of-envelope.html and pricing-constants — this page is their GPU-serving analog and cross-links both.

2. The Mechanism: Two Phases, Opposite Rooflines

TL;DR. An LLM request is two different machines glued together. Prefill is a big matmul over the whole prompt — compute-bound. Decode is one tiny matmul per output token that must re-read every weight — memory-bandwidth-bound. They have opposite rooflines, opposite SLOs, and every later page is some consequence of this split.

The running trace (threaded through K1–K2)

One conversation, grown in three stages. Every formula on this page — and the paging (K2 Role 1) and the cross-request reuse (K2 Role 2) — is illustrated on this same trace, so the reader follows one story.

T1 prompt = "I like cat, why?" → tokenize → s₁ = 6 prompt tokens
    decode = "Cats are independent, clean, and affectionate ..." → r₁ = 40 output tokens
T2 history = T1 prompt + T1 response → 6 + 40 = 46 tokens, all now "prior turn"
T3 request = history + "I don't like dog" → prefix = 46 cached tokens + s₃ = 5 new tokens
  • At T1 (prefill): all 6 prompt tokens go through every layer once, in parallel — one matmul of shape [6, d_model]. The work is 2·P·6 FLOPs (§3.2). Time-to-first-token (TTFT) ends when token #1 is sampled.
  • At T1 (decode): 40 steps. Each step processes one token ([1, d_model]) but must load all P weights from HBM to do it. The per-token gap is TPOT (time-per-output-token, a.k.a. ITL, inter-token latency).
  • At T3 (prefill with reuse): the 46-token prefix was already computed at T1/T2. If its KV is still resident (K2 Role 1) and the request is routed to the GPU holding it (K2 Role 2), prefill only does the 5 new tokens — 2·P·5 instead of 2·P·51. This is the single biggest TTFT lever, and it is why K2 exists.

How P is computed — where the 70 B comes from

TL;DR. P is not a given — it is the sum of every weight matrix in the model. Per layer that is the four attention projections + the three FFN matrices; add the embedding tables and multiply by L. Matrix shapes are K0's; the count is owed here because it feeds W_bytes (§3.6).

P = L × (attn_params + ffn_params) + embedding_params

Attention block — one layer (four weight matrices; a matrix of shape [d_in, d_out] has d_in·d_out params, the 2×-rule input from §3.2):

MatrixRoleShape = rows × colsParams
W_Qproject token → query spaced_model × (n_q_heads·d_head) = 8,192 × 8,19267 M
W_Kproject token → key spaced_model × (n_kv_heads·d_head) = 8,192 × 1,0248.4 M
W_Vproject token → value spaced_model × (n_kv_heads·d_head) = 8,192 × 1,0248.4 M
W_Oproject attention output → residual(n_q_heads·d_head) × d_model = 8,192 × 8,19267 M
attn_paramssum of the four≈ 151 M / layer

FFN block — one layer (SwiGLU → three weight matrices):

MatrixRoleShape = rows × colsParams
W_upup-projectiond_model × d_ff = 8,192 × 28,672235 M
W_gategate-projectiond_model × d_ff = 8,192 × 28,672235 M
W_downdown-projectiond_ff × d_model = 28,672 × 8,192235 M
ffn_paramssum of the three≈ 705 M / layer
per layer    = attn 151 M + ffn 705 M        = 856 M

───────────────────────────────────────────────
P           = 68.5 B + 2.1 B            ≈ 70.6 B → the "70 B" headline
  • Why W_K, W_V are 8× smaller than W_Q, W_O. They project into n_kv_heads·d_head = 1,024, not n_q_heads·d_head = 8,192 — that is GQA (n_kv_heads=8 vs n_q_heads=64). The same n_kv_heads lever that shrinks the KV cache (§3.6) also shrinks these two matrices.
  • Embeddings counted twice. Llama-3-70B keeps the input token-embedding and the output unembedding untied — two separate V×d_model tables. Models that tie them (share one matrix) count V×d_model once; then P ≈ 69.5 B. The thin layer-norm / final-norm vectors (~10–6 params each) round off and are dropped.
  • Relation to the §3.3 shortcut. P ≈ 12·L·d_model² is this exact sum approximated (it assumes d_ff = 4·d_model and MHA, ignoring GQA and embeddings) — it gives 12 × 80 × 8,192² ≈ 64 B here, ~9% low. Use the shortcut for crossover algebra (§3.3); use this exact breakdown when you need the real weight-memory number.

3.2  The 2× rule (an approximation — derived here, this is its home)

  • Problem: every FLOP estimate on every page needs a rule for "FLOPs per parameter per token." Asserting "it's 2" is the most-copied unexplained constant in LLM serving.
  • Solution — derive it, and see why the "2" is a convention. A weight matrix W of shape [d_in, d_out] applied to one token vector x [d_in] produces y[j] = Σᵢ x[i]·W[i,j]. Each output y[j] is a length-d_in dot product: d_in multiplies and d_in−1 adds — summing d_in products takes one fewer add than there are products — = 2·d_in−1 FLOPs. Over d_out outputs: d_out·(2·d_in−1) = 2·d_in·d_out − d_out. The standard ML convention rounds the −d_out away — equivalently, it counts each multiply-accumulate (MAC / FMA) as 2 FLOPs — giving 2·d_in·d_out. Since d_in·d_out is exactly W's weight count, a forward pass is ≈ 2× the weights = ≈ 2·P FLOPs per token.
  • So the 2× rule is an approximation — just a very tight one. The exact count is 2·P − Σd_out (one missing add per output element of every weight matrix); the dropped term is a fraction ≈ 1/(2·d_in) of the total — at d_in = d_model = 8,192 that is 0.006%. It is the m=1 case of the general 2·m·n·k matmul convention: cost scales with the contracted dimension k (here always d_in), so operand order matters — [1×k]·[k×1] costs ≈2k, [k×1]·[1×k] costs ≈2k². This is why the rule is written with ≈, never =: precise enough to size a fleet, never an exact FLOP count.
2× rule.   forward-pass FLOPs ≈ 2 · P · n_tokens   (the linear, weight-driven term)
one multiply-accumulate per weight per token, ~2 FLOPs each → factor ≈ 2 (the −d_out rounding is <0.01%).

This counts only matmul-with-weights. Attention's QKᵀ and AV have no weights and are quadratic in sequence length — the correction term in §3.3. Backprop is not serving (the training "6×" rule does not apply here).

3.3  Prefill FLOPs, the quadratic correction, and its crossover

prefill_FLOPs(s) = 2·P·s  +  4·L·s²·d_model
linear (weights, §3.2)  |  quadratic (attention scores QKᵀ + AV, no weights), s = prompt length
  • Crossover (derived algebraically, not eyeballed). Using the transformer identity P ≈ 12·L·d_model², set linear = quadratic: 2·(12·L·d_model²)·s = 4·L·s²·d_model ⇒ s* = 6·d_model. For Llama-3-70B: s* = 6 × 8,192 ≈ 49,000 tokens.
  • Consequence: below ~49K prompt tokens, prefill cost is linear — use 2·P·s and ignore the quadratic term (it is <10% up to ~16K). Above it, attention dominates and long-context serving (K2/L3) is a different regime. State which side you are on; never silently drop the correction at 100K.
  • On the trace: T1 prefill = 2 × 70e9 × 6 = 0.84 TFLOP; the quadratic part is 4 × 80 × 6² × 8192 ≈ 0.0001 TFLOP — utterly negligible at s=6, exactly as the crossover predicts.

3.4  Roofline & arithmetic intensity — which resource binds

  • Definition. Arithmetic intensity AI = FLOPs ÷ bytes moved from HBM. The ridge point of a GPU is AI* = peak_FLOP/s ÷ HBM_BW. AI > AI* ⇒ compute-bound; AI < AI* ⇒ memory-bandwidth-bound.
  • H100 ridge: 990 TF/s ÷ 3.35 TB/s ≈ 295 FLOP/byte. (FP8: 1,979 ÷ 3.35 ≈ 591.)
  • Prefill processes s tokens per weight load → AI ≈ 2·P·s ÷ (2·P) = s ≈ thousands > 295 ⇒ compute-bound. Decode processes 1 token per weight load → AI ≈ 2·P·B ÷ (2·P) = B (B = batch; the 2·P cancels, exactly as in prefill — not 2·B); at B=1, AI≈1 &Lll; 295 ⇒ deeply memory-bound.
  • Crossover batch B\* (where decode stops being memory-bound): set decode AI = ridge ⇒ B* = 295 ⇒ B* ≈ 295 on H100 BF16. Below it, more batch is free (you are wasting bandwidth-loaded weights); above it, decode is compute-bound and more batch costs latency. Every batching decision (K4) lives around this number.

3.5  Decode step = max(load, compute) — always show both

One decode step, per replica — a replica = the TP GPUs that jointly hold one model copy — at batch B, context length c. The TP GPUs run in parallel, so the replica's step time is each GPU's step time; the replica is the honest unit because it owns the batch and the KV pool:

Load  = W_bytes / (TP·HBM_BW)  +  B·c·kv_bytes_tok / (TP·HBM_BW)  (weights + KV re-read every step; TP·HBM_BW = replica aggregate bandwidth)
Compute = (2·P·B  +  4·B·L·c·d_model) / (TP·FLOP/s)  (weights term: 2× rule, one new token per request projected through every weight, ×B requests; attention term: each new Q attends over c cached K/V entries per request — QKᵀ + AV is two matmuls, each MAC=2 ⇒ factor 4 — ×L layers, ×B requests; TP·FLOP/s = replica aggregate compute)
t_step = max(Load, Compute)   — never report only the larger; the reader must see why it dominates.

Worked (Llama-3-70B, H100, TP=2 → a 2-GPU replica, B=1, c small so KV term ≈ 0):

W_bytes = 70e9 × 2 = 140 GB  (§3.6 weight formula)
Load = 140 GB / (2 × 3.35 TB/s) = 140 / 6700 = 20.9 ms ← per replica, weights only
Compute = (2 × 70e9 × 1  +  4 × 1 × 80 × c × 8192) / (2 × 990e12) ≈ 1.4e11 / 1.98e15 = 0.07 ms ← attention term ≈ 0 at small c
t_step = max(20.9, 0.07) = 20.9 ms (load-bound) → TPOT ≈ 21 ms, ~48 tok/s/req
Total decode cost for N generated tokens (per request in a batch of B, prompt length s, cache grows s → s+N−1):
Σj=0N−1  (2·P·B  +  4·B·L·(s+j)·d_model)
= 2·P·B·N  +  4·B·L·d_model · (N·s + N(N−1)/2)  — weights term linear in N; attention term linear in s, quadratic in N (the ∑(s+j) closed form). Long outputs make the attention term dominate even when each step is still load-bound — this is why decode TPOT drifts upward as the cache grows (the K2 problem in its FLOP form).
  • Why this is the central serving fact: the 0.07 ms compute is 99.7% idle — the replica's FLOPs sit idle while it re-reads all 140 GB of weights. Two levers attack it: bigger batch (amortize the weight load over B tokens → the 2× rule numerator grows, denominator fixed — up to B\*≈295, §3.4) and fewer weight bytes (quantization, K14). Both are "make decode less memory-bound." This single max() is why K2, K4, K14 exist.
  • KV term grows with context (both sides of the max). At c=32K, B=1 on the TP=2 replica: Load side — KV bytes = 1 × 32768 × 327,680 ≈ 10.7 GB → 10.7 / 6700 = 1.6 ms added to Load. Compute side — attention FLOPs = 4 × 1 × 80 × 32768 × 8192 ≈ 8.6e10 → 8.6e10 / 1.98e15 = 0.04 ms added to Compute. Even at 32K context, decode is bandwidth-bound, not attention-FLOP-bound — but the attention term goes quadratic in N (total-cost box above), so very long outputs eventually shift the bottleneck. K2 owns the load side; long-context regimes (L3) own when the compute side starts mattering.

3.5 (cont.)  Why the KV cache exists — prefill + decode × with/without cache

§3.3 and §3.5 above implicitly assume the with-cache regime. Writing out the without-cache column makes the cache's job mechanical: for prefill it changes almost nothing; for decode it converts every step from a full prefill into an O(c) increment. That asymmetry — cheap on prefill, decisive on decode — is why every serving stack has a KV cache.

With/without cache differs only by an s·kv_bytes_tok write-back term on Load (e.g. at s=13K: 13000 × 320 KB = 4.16 GB; on TP=2 H100 that's 4.16/6.7 = 0.6 ms — ~3% on top of the 20.9 ms weight read). Compute is identical. Whether you keep K/V or throw them away is a memory-policy question, not a compute question — that's why §3.3 doesn't mention it.
  • Decode is the entire point of caching. Without cache, step j costs prefill(s+j+1) — Compute is quadratic in (s+j+1). With cache, step j costs 2·P·B + 4·B·L·(s+j)·d_model — linear in (s+j). For Llama-3-70B at s=13K, B=1: without-cache step ≈ prefill(13K) = 2.27 PFLOP per output token; with-cache step ≈ 1.74·10¹¹ FLOPs = 0.000174 PFLOP. Ratio: ~13,000× per step, and it grows as O(c) at large c. The total over T_out tokens collapses from O(T_out·(s+T_out)²) to O(T_out·(s+T_out)) — the difference between a serving stack that works and one that doesn't.
  • Cost paid for the win — cache load. The cached K/V must be re-read from HBM every step (the B·c·kv_bytes_tok / (TP·HBM_BW) term on with-cache decode Load). At small B·c this is dwarfed by weight load; at large B·c (long contexts × high batch — the regime K2 owns) it crosses over and dominates Load, which is what forces paged blocks, tiering, and prefix sharing. The four-formula matrix is the spec; K2's mechanisms are the implementation of that spec under HBM pressure.

3.6 (cont.)  What HBM must hold to finish one forward pass

TL;DR. A forward pass touches five kinds of bytes, but only two of them persist in HBM across steps — weights and the KV cache — and those two dominate the budget. The rest (activations, inputs, kernel scratch) are transient: allocated, consumed, and freed within the pass.

  • Weights (persistent) — embedding table, per-layer {W_Q, W_K, W_V, W_O} (attention output projection) + FFN {W_up, W_gate, W_down}, RMSNorm/LayerNorm params, final LM head. Sized by W_bytes above; loaded once, read every step.
  • KV cache (persistent) — per-layer K and V tensors for all prior tokens, shape [L, 2, batch, n_kv_heads, seq_len, d_head]. Sized by kv_bytes_per_token above — the dominant term at long context. K2 owns its paging/tiering.
  • Activations (transient) — input embeddings, Q/K/V projections, attention scores QKᵀ, softmax probs, attention output, FFN intermediate (d_model → 4·d_model), residual stream, logits. Allocated and freed within the pass — see why below.
  • Positional data (persistent, tiny) — RoPE sin/cos cache or a learned positional-embedding table; precomputed once, read-only.
  • Inputs (transient, tiny) — token IDs, attention mask, position IDs for this batch.
  • Kernel workspace (transient) — cuBLAS / FlashAttention scratch buffers and reduction temporaries; handed back to the allocator when the kernel returns.
  • Quant metadata (persistent, if quantized) — per-channel / per-group scales and zero-points; small alongside the weights they de-quantize.
  • Net: in serving, KV cache + weights dominate. Activations stay small if you use FlashAttention (never materializes the full QKᵀ matrix) and free per-layer intermediates as you go.

Why activations are transient (and weights/KV are not)

  • The forward pass is a feed-forward chain. Layer i's output is layer i+1's only input. Once layer i+1 has consumed it, layer i's activations are dead — nothing later in this pass reads them — so HBM peaks at roughly one layer's activations, not L layers'.
  • Inference does no backward pass. Training must keep every layer's activations until the backward pass reuses them to compute gradients (that's what activation-checkpointing trades off). Inference computes no gradients, so there is no future reader — activations can be freed the instant they're consumed.
  • They don't carry across decode steps. The only state the next token's forward pass needs from prior tokens is their K and V — and that is exactly what the KV cache saves. Everything else is recomputed fresh each step, so it need not persist.
  • Contrast — why weights & KV are persistent. Weights are read identically on every step of every request, so reloading them from elsewhere would waste bandwidth — they stay resident. KV is per-request history that every future decode step re-reads (§3.5 Load term), so it must survive between steps. Both have future readers; activations do not.

3.7  MFU vs MBU — peak is a lie, state the fraction

  • MFU (Model FLOPs Utilization) = achieved FLOP/s ÷ peak FLOP/s — the honest fraction for compute-bound work (prefill). MBU (Model Bandwidth Utilization) = achieved HBM bytes/s ÷ peak — the honest fraction for memory-bound work (decode).
  • Always state the value and why. Default assumption for derivations: MFU = 40% for prefill (conservative; real 30–60% with FlashAttention + good kernels, K17), MBU = 60–80% for well-batched decode. A "990 TF/s" number used without an MFU is wrong by ~2–3× — this is the most common hidden-assumption failure (§5).
  • Effective constants: prefill uses 0.40 × 990 = 396 TF/s; decode uses 0.70 × 3.35 = 2.35 TB/s effective. Scenarios state their MFU/MBU explicitly and cite this default.

3.8  Throughput = batch / req_time

req_time = TTFT + (out_tokens − 1) × TPOT
req/sec/node = batch / req_time  (continuous batching; static batching uses longest req in batch)
  • On the trace (TP=2 node = one 2-GPU replica, B=1, no batching yet): TTFT ≈ prefill(6 tok)/effective + 1 step. Prefill compute = 0.84 TFLOP / (2 × 396 TF/s) ≈ 1.06 ms (per replica); +1 decode step 20.9 ms ⇒ TTFT ≈ 22 ms. E2E = 22 + 39 × 20.9 ms ≈ 0.84 s for the 40-token answer. req/sec/node = 1 / 0.84 ≈ 1.2.
  • The batch lever: at B=64 (still < B\*=295, so decode stays load-bound and TPOT barely moves) req/sec/node ≈ 64 / 0.84 ≈ 76 — a 64× throughput gain for ~0 latency cost. This free lunch (until B\*) is the entire motivation for K4.

3.9  Fleet size

peak_qps = (requests/hr ÷ 3600) × peak-factor  (from the scenario's traffic row)
nodes = ⌈ peak_qps / req_per_sec_per_node ⌉  ·  GPUs = nodes × GPUs_per_node

Example: 10M req/hr, peak-factor 1 ⇒ peak_qps = 2,778. At 76 req/s/node ⇒ ⌈2778/76⌉ = 37 nodes × 2 GPUs = 74 H100. Same peak_qps in every iteration of a scenario; GPUs_per_node = TP degree and is restated whenever TP changes (the "node redefinition" pitfall).

3.11  Collective-cost primitives (every parallelism page cites this)

When a model is split across N GPUs, each layer pays a collective. M = bytes of the tensor involved, BW = link bandwidth, α = fixed setup cost per message = one logical point-to-point send between two endpoints on the link (e.g. one chunk in one ring step) — not per byte; one message of b bytes ≈ α + b/BW (see §3.0 notation).

Message size is set by the algorithm, not fixed. Ring all-reduce splits M into N chunks of M/N bytes and runs 2·(N−1) sequential ring steps (reduce-scatter + all-gather), each ring step shipping one chunk between adjacent GPUs ⇒ the table’s 2(N−1)α (one α per step) + 2M/BW (size-proportional) decomposition. Ring all-gather / reduce-scatter run one such phase (N−1 steps, message M/N). All-to-all sends a distinct M/N-byte chunk to each of the other N−1 peers ⇒ (N−1)α on each GPU’s critical path, but N² distinct messages contending globally on the bisection (the bullet below).

CollectiveWhereBytes moved / GPUTime model

  • All-reduce = all-gather + reduce-scatter — hence the 2M. TP's per-layer all-reduce of the attention/MLP output (M = B·s·d_model·2 bytes) is on the critical path of every layer, which is why TP must sit on NVLink, not IB (K20).
  • Pipeline bubble ratio = (stages − 1) / (microbatches + stages − 1) — derived here, applied in K8. 4 stages, 8 microbatches ⇒ 3/11 ≈ 27% idle.
  • Worked example — Llama-3-70B TP=8 all-reduce on NVLink (α ≈ 1.5 μs, BW = 0.9 TB/s, M = B·s·d_model·2, d_model=8192, BF16). Decode and prefill hit the same collective in opposite regimes — the message size flips which term dominates:
    • Decode (B=1, s=1): M = 16 KB, ring chunk M/N = 2 KB, 14 steps → 14 × (1.5 + 2KB/0.9TB/s) μs ≈ 14 × 1.5 ≈ 21 μs — purely α-bound; the 28 KB of bytes moved per GPU are essentially free, the cost is the 14 hops.
    • Prefill (s=1024): M = 16 MB, ring chunk 2 MB, 14 steps → 14 × (1.5 + 2.2) μs ≈ 52 μs — α and BW each ~half.
    This is why the curriculum treats prefill (BW-bound) and decode (latency-bound) as separate collective regimes (K20 spine).
  • All-to-all and all-gather move the same bytes — both M·(N−1)/N ≈ M per GPU. The ≈ M is just (N−1)/N → 1 (0.88M at N=8, 0.98M at N=64) — it depends only on N, so it holds for both rows regardless of which collective. The data patterns look unrelated, yet the count is identical, because per-GPU egress is simply whatever a GPU holds that must leave it:
    • All-gather — a GPU owns one chunk M/N and ships that same chunk to the other N−1 peers ⇒ (N−1)·M/N (the payload is replicated).
    • All-to-all — a GPU owns N−1 distinct chunks of M/N, one per peer, and ships them ⇒ (N−1)·M/N (the payloads differ).
    Same volume; what differs is structure — all-to-all is N² distinct messages ⇒ bisection-bound and latency-dominated, the hardest collective, why EP-across-IB is the cliff (K9/K20). That cost gap lives in the time-model column (N² msgs), not the byte column.

The diagnostic walk — a headline went red, now which diagnostic, which lever

The tables above are the catalog; this is the conditional logic (integrated from the diagnostic playbook). Each bullet is symptom → what it means → first move.

When TTFT degrades — "prefill is slow, or requests are waiting"; the diagnostics say which.

  • Queue time rising → a scheduling/capacity problem, not compute — the model is fine. Check concurrent requests + preemption rate; scale on queue depth (K4/K15).
  • Prefill latency rising while queue flat → a compute problem — input lengths grew (check the length distribution, K0/K2), or kernels regressed (deploy diff, K17), or the GPU is contended.
  • KV-hit rate dropping → Role-2 reuse broke (a prompt-template change or a deploy invalidated cache keys) — large TTFT effect because a hit skips prefill entirely (K2 Role 2).
  • Prefill batch size rising → chunked prefill is bundling more work/iter — may be intentional throughput tuning, but it raises individual TTFT (K4/D1).

When TPOT/ITL degrades — "decode is slow"; decode is memory-bound, so look at memory pressure + batch composition.

  • Effective decode batch dropping → lost weight-load amortization, usually KV pressure forcing preemption (K2/K4).
  • Preemption rate rising → KV utilization too high; recompute/swap extends the victim's and the waiters' TPOT (K4/D3).
  • MBU rising toward 1 → near the hardware ceiling — only shrink the batch, quantize (K14), or upgrade.
  • Decode MFU → low is expected (bandwidth-bound) — a sanity check, never a target.

When goodput degrades — either SLOs are missed or raw throughput dropped; compare the two.

  • Raw tokens/s flat but goodput down → SLOs are being missed → go to the TTFT/TPOT walks. Both down → a capacity or efficiency problem.
  • Per-tenant attainment skewed → one tenant's pathological prompts — isolate with WFQ-over-tokens + per-tenant KV quota (K4/D18, K15).
  • Head-of-line incidence up → long requests clog the pipe → chunked prefill / priority scheduling (K4/D1).
  • Fairness-across-sizes bad → small requests starved; their tail is what kills goodput → WFQ over GPU tokens (K4/D18).

When availability degrades — the error-rate breakdown is the first stop.

  • OOM → KV or activation memory → paging/tier/evict (K2) + chunked-prefill activation bound (K4/K18/D46).
  • CUDA / plausible-but-wrong output → driver/hardware or silent data corruption → fast detect + parallelism-group eviction + hot-standby (K22/D50).
  • Timeouts → a TTFT/TPOT regression crossed the deadline → the two walks above. 429s → capacity → autoscale + degradation contract (K15, K4/D13).
  • Stream-completion drop + OOM near-misses are the leading indicators — act here before the headline goes red.

When KV-cache utilization runs hot (≥~90%) — itself a leading diagnostic for the two above.

  • KV-hit rate dropped → more unique prefixes eating cache (K2 Role 2). Preemptions spike → the direct consequence; relieve with tier/evict/quant (K2/K14).
  • Length distribution shifted → users pasting bigger documents or longer CoT (K19). Concurrency vs size: low concurrency + high util ⇒ huge individual requests; both high ⇒ just load (K4/K15).

When cost / million tokens rises — downstream of efficiency; goodput per GPU is the denominator.

  • MFU low on prefill-heavy work → the model isn't fitting the hardware (small batch / bad kernel / wrong parallelism) (K17/K6).
  • GPU-util low + MFU low → idle → the autoscale signal is wrong (K15/D12). SM high + MFU low → inefficient kernels (K17).
  • Effective batch small → poor amortization → push batch toward B\* (K4) — almost always the lever.
  • KV utilization → 70–85% is the sweet spot (low = paying for unused HBM; high = can't fit more concurrent work) (K2).

Goodput, not throughput, is the optimization target every lever above is tuned against: a config can raise tokens/s while more requests miss SLO. Every disaggregation/batching/quantization decision in the curriculum is judged on goodput per GPU (the cost/Mtok denominator), not raw tokens/s.

4. Convention Choices (and When the Other One Wins)

A number sheet has no "alternative techniques"; it has convention choices that every other page must share. Each is stated with its disqualifying condition.

Reference dtype: FP16 baseline

  • Chosen: FP16/BF16 (2 B/param) as the baseline everywhere — it is the lossless reference; the model ships in it.
  • Flips to FP8/INT4 when a page is specifically about quantization (K14) or a scenario's cost target forces it; then the page states the dtype and recomputes the whole cascade (weight bytes → decode step → batch → fleet). Never silently mix dtypes.

Dense FLOPs, not "with sparsity"

  • Chosen: dense tensor-core TFLOPs (§3.1). Dense LLM inference does not use 2:4 structured sparsity.
  • The vendor "sparsity" number wins only for a sparsity-trained model on sparsity kernels — out of scope; using it here doubles your apparent compute and halves your fleet estimate (a real, common error, §5).

Per-replica basis for step math and memory budget

  • Chosen: max(load, compute) and the memory budget are both computed per replica — a replica being the TP GPUs that jointly hold one model copy. Its aggregate resources are TP·HBM_BW bandwidth, TP·FLOP/s compute, TP×HBM. The TP GPUs run in parallel, so the per-replica step time equals each GPU's — but the replica is the unit that owns the batch B and the KV pool, so it is the honest unit.
  • The fleet basis rolls the replica up — fleet throughput/cost (§3.8–3.10) is req/s/replica × replica count (one TP-replica per node in the worked examples). Every number is labeled per-replica or fleet-total; mixing a per-GPU quantity into per-replica math is the §5 failure.

Ring collective cost model

  • Chosen: bandwidth-optimal ring (§3.11) — the model NCCL uses for large tensors; gives the 2M all-reduce.
  • Tree/double-binary-tree wins for small tensors (latency-bound, α term dominates) — K20 covers the topology-dependent crossover; this page uses ring as the default.

5. How These Numbers Get Misused

  • Peak FLOPs with no MFU. "990 TF/s" used raw makes prefill look 2.5× faster than reality and the fleet 2.5× too small. Always multiply by an MFU and state it (§3.7).
  • Sparsity-inflated TFLOPs. Quoting the "1,979 / 3,958" H100 sparsity FP16 number for dense inference — halves every fleet estimate. Use the dense column (§3.1).
  • Reporting only the larger of load/compute. "Decode = 21 ms (memory-bound)" without showing compute = 0.07 ms — the reader cannot verify which bound, or that batching has headroom up to B\*. Always show both (§3.5).
  • Mixing a per-GPU quantity into per-replica math. Dividing the full model's W_bytes (a per-replica total) by one GPU's HBM_BW instead of the replica's TP·HBM_BW — off by the TP factor. Compute on the per-replica basis throughout and label every quantity (§4).
  • Dropping the quadratic prefill term at long context. Using 2·P·s at s=200K underestimates prefill ~4×. Past s\*≈49K the attention term dominates (§3.3).
  • Static $/GPU/hr drift. One page at $12.29, another at "$8/hr" — cite the pricing-constants line everywhere (§3.10).
