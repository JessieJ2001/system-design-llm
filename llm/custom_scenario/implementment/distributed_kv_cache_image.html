Distributed KV Cache for a Multimodal Claude-Code-Style Assistant

A staff+ NALSD walkthrough — same fleet, same SLOs, same hardware as the text sibling page, with one functional change: requests may now carry image / PDF / video attachments. An image, a PDF page, and a video frame are all the same thing to the serving system: raw bytes that a vision encoder turns into embedding “tokens,” which are spliced into the text token stream and then produce K,V through the very same LLM layers as text (K2 D32). So the entire distributed-KV-cache design of the parent page still holds — tiering, PagedAttention, the cross-task pool, prefix caching, context parallelism — it just now has to absorb (1) a new upstream compute stage with the opposite roofline (the encoder), (2) far more KV per request (one image ≈ 1–3 k tokens; one minute of video ≈ 15 k+; a 100-page PDF ≈ 150 k), and (3) a new, richer reuse axis — the encoder’s output is deterministic and content-addressable, which unlocks cross-user dedup the text design explicitly rejected.

NEW §M  How an image / PDF / video becomes tokens (from zero)

Why this section exists. The reader is assumed to know LLM text serving cold (the parent page) but nothing about vision. So this section defines from zero, walks one concrete image token-by-token, and gives the tensor transform a diagram (not a wall of prose). It is knowledge, not a design iteration — no “what breaks” / SLO table. It cites K12 and K2 D32 for depth but stands alone, so you never have to leave.

M.1  The one load-bearing fact (state it first)

The LLM never sees pixels. A separate model — the vision encoder (a ViT, Vision Transformer) — converts raw pixels into a sequence of embedding vectors in the LLM’s hidden dimension d_model. Those vectors are inserted into the input sequence at the spot the <image> / <video> placeholder occupied.

After insertion they are indistinguishable from text-token embeddings. They flow through the same n_layers decoder blocks, the same attention, and produce the same K,V entries in the KV cache — K2 D32. This is the reason the parent page’s entire KV machinery transfers wholesale.

Consequence, stated up front (so the rest of §M pays off):

  • “Cache the KV of an image” is just “cache KV” — already solved by the parent (tiering, paging, pool, prefix cache all apply to vision K,V for free).
  • The genuinely new caching question is one level up: can we skip re-running the encoder? (Iter M2.)
  • The genuinely new capacity question is: there are a lot more tokens now (the volume deltas to Iter 1a / 3 / 9).

M.2  The vision encoder, walked end-to-end (worked example + diagram)

Anchor architecture for this page. A decoder-only “splice” fusion VLM — the family where vision tokens become regular self-attention KV — built from a SigLIP/CLIP-class ViT encoder + an MLP “projector” + the parent’s Llama-3-70B-class decoder backbone, in the style of LLaVA-OneVision / Qwen2-VL / InternVL (native dynamic-resolution variants).

  • SigLIP = Sigmoid Loss for Language-Image Pre-training; CLIP = Contrastive Language-Image Pre-training — two ways to pre-train a ViT so its output embeddings are semantically aligned with text. Either yields the same serving shape; concrete dims are flagged G-FA and settled at compose.
  • The alternative cross-attention fusion family (where vision is not in the autoregressive KV sequence) is the design choice in §M.5.

Walk one 1024×1024 RGB image through the pipeline

RGB = Red / Green / Blue — the three colour channels of a raw decoded image. Numbers below are illustrative (G-FA for the model dims; the patch arithmetic is D, derived inline).

  1. Preprocess — decode (JPEG / PNG / WebP → raw RGB) and resize/pad to a grid the encoder accepts. (Video adds frame-sampling; PDF adds page-rendering — §M.3 / §M.4.)
  2. Patchify — cut into fixed p×p patches (p = 14): 1024 / 14 ≈ 73 per side → 73 × 73 = 5 329 patches D. Each patch is 14 · 14 · 3 = 588 raw values, linearly projected (a conv2d, 2-D convolution, with stride p) to a d_vision-dim vector (d_vision = 1 152). This is the “tokenizer” for pixels.
  3. + 2-D positional embedding — so the encoder knows where each patch sat (interpolated for native resolution). Tensor so far: [5 329 × 1 152].
  4. ViT encoder — L_vit (~27) transformer layers of bidirectional self-attention over the ~5 329 patch tokens → one d_vision feature vector per patch. No causal mask, no KV cache — it is a single forward pass. (This is the roofline asymmetry of §M.6: the ViT is compute-bound, the decoder is bandwidth-bound.)
  5. Spatial merge / pooling — adjacent patches are merged (2×2 → ÷4) to cut token count: 5 329 / 4 ≈ 1 332 vision tokens D (lands inside K12’s “1–3 k tokens per image”).
  6. Projector / connector — an MLP (Multi-Layer Perceptron; LLaVA uses a 2-layer GELU MLP) maps d_vision (1 152) → d_model (8 192). (Alternative connector = a cross-attention resampler / Perceiver / Q-Former that compresses to a fixed token count — the DC in §M.5.)
  7. Splice — the ~1 332 vectors replace the <image> placeholder in the input embedding sequence: [hidden_sys … ⟨1 332 image vecs⟩ … user_msg].
  8. LLM prefill — from here it is exactly the parent page: prefill over (text + vision) tokens → K,V → decode. Each vision token costs KV_per_tok = 320 KB, identical to a text token.

M.3  How a PDF is parsed (two complementary streams — render-to-image and extract-text)

Problem. A PDF is not pixels and not plain text — it is a layout container (text runs, vector graphics, embedded fonts, tables, scanned images). There are two ways to turn a page into model input, and production feeds both in parallel — each is blind to exactly what the other captures.

Stream A — extract-text / OCR (complementary)

  • OCR = Optical Character Recognition. A parser (pdfminer, pdfplumber) reads the PDF’s embedded text layer (or an OCR engine reads a scan) → fed as ordinary text tokens.
  • What it is for — the exact characters, verbatim. A born-digital PDF carries the real Unicode underneath: ground-truth text the vision encoder can only approximate.
  • Cheap — text tokens are far fewer than the ~1.5 k vision tokens/page, so exact grounding costs almost nothing.
  • But blind to layout — flattens tables to token soup, drops figures/charts, returns nothing on a scanned page.

Stream B — render-to-image (primary)

  • Rasterize each page to an image (~150 DPI, dots per inch) and run it through §M.2 as an image.
  • What it is for — layout, tables, figures, charts, handwriting, spatial arrangement — and it is the only stream that works on a scanned PDF (no text layer at all).
  • One PDF page = one image ≈ ~1.5 k vision tokens G-FA.
  • A 20-page PDF ≈ ~30 k tokens; a 100-page PDF (the §S0.3 p99) ≈ ~150 k — that is only ~48 GB KV, still well inside one task’s HBM-for-KV budget. Only a ~533+ page PDF (≥ 800 k tokens / 256 GB) reaches the context-parallelism fork (Iter 9 delta).
  • Why both, in parallel — complementary failure modes. The vision stream is strong on structure but weak on exact character fidelity (a downsampled 150-DPI image of small dense text makes the encoder misread digits, decimal points, code, IDs); the text stream is exact but blind to layout. Feeding both lets the model use the picture for structure and the text for exact content. Concrete example — a financial-table screenshot: the image shows it is a table and which column is which; the extracted text supplies $1,234,567.89 exactly, so the model quotes the figure without the encoder turning a 3 into an 8.
  • When you genuinely get only one stream (the true fallbacks). A scanned PDF has no text layer → image-only; an oversize doc past the page/token cap forces a single stream (render-to-image is kept, text-extraction is the degraded fallback). For an in-budget born-digital PDF the default is both.
  • KV consequence. A PDF is just “N images back-to-back” → it inherits everything in §M.2 and multiplies the token / KV count by page count. This is why PDFs are the dominant capacity driver.

M.4  How a video is parsed (frame-sample → images + a temporal axis)

Problem. A 1-minute 30 fps (frames per second) clip is 1 800 frames; encoding every frame as a full image is infeasible (1 800 × 1 332 ≈ 2.4 M tokens D — past the 1 M context cap by itself).

  1. Frame-sample at a low rate (fps_sample ≈ 1) → 60 frames for a 1-minute clip.
  2. Encode each frame as an image (§M.2) — hardware video decode (NVDEC, NVIDIA Video Decoder) feeds the frames.
  3. Temporal + spatial merge — merge adjacent frames/patches to cut tokens hard (Qwen2-VL merges to ~256 tokens/frame after a 2×2×2 merge) → 1-minute clip ≈ ~15.4 k tokens D (60 × 256); 10-minute (the §S0.3 p99) ≈ ~154 k (≈ 49 GB KV — still fits one task). Only a ~52+ minute clip crosses ~1 M tokens / 256 GB → context parallelism (Iter 9 delta) from a single attachment.
  4. Temporal position — a 3-D / multimodal RoPE (M-RoPE; RoPE = Rotary Position Embedding) gives the LLM a temporal axis, so “what happened after X” works.
  • KV consequence. Video is the token-explosion king and the reason the parent’s long-context machinery (tiering and the pool at every clip; context parallelism only for the hour-long extreme) gets exercised at turn 1, not just on long chat histories.
  • Page scope flag. Uploaded video files only. Live / real-time video streams (continuous ingest, streaming encode) are a different system → new non-goal (S0 delta).

M.5  Design choice — fusion architecture (the one with a KV-cache consequence)

DC — How do the vision features reach the LLM’s attention?

Problem: the ViT produced [1332 × 8192] vectors. There are two architecturally distinct ways to feed them into the decoder, and they put the vision K,V in different places.

Possible solutions enumerated:

  • (a) Decoder-only “splice” fusion — CHOSEN (main line) — concatenate vision tokens into the input sequence (LLaVA, Qwen2-VL, InternVL, GPT-4V-class; Llama-4’s early fusion is the same family for KV purposes — vision tokens live in the sequence). KV consequence: vision tokens become regular self-attention KV (K2 D32) → the entire parent design applies unchanged. Picked because it makes the “distributed KV cache” framing exact and matches the dominant open-weights family.
  • (b) Cross-attention fusion (Flamingo, Llama-3.2-Vision) — vision features are consumed by dedicated gated cross-attention layers; they are not in the autoregressive KV sequence. KV consequence: a separate “vision KV” read by cross-attention at every layer — still cacheable, even more separable (decode never re-reads it through self-attn), but it splits the cache into two key spaces and changes the prefix-chain math. Named as the alternative with a spelled-out KV consequence; not shipped (it would fork every downstream iteration).

Winning constraint:

  • The page’s thesis is “vision K,V is the parent’s KV” — only splice fusion makes that literally true, so every parent iteration transfers with a one-line delta instead of a fork.

Connector sub-choice (an orthogonal axis — independent of the fusion choice)

Problem: the projector step (§M.2 step 6) can either preserve native token count or compress to a fixed budget.

  • MLP projector — CHOSEN — variable token count ∝ image area; native dynamic resolution (detail scales with the image).
  • Cross-attention resampler / Perceiver / Q-Former — compresses to a fixed N_query tokens; bounds KV per image but caps detail.

Winning constraint:

  • The product needs full-fidelity tables and screenshots (the same reason PDF renders to image), so detail beats a fixed KV bound here; main line = splice + MLP + native dynamic resolution. The fixed-N_query option returns as a lever in the ragged-batching iteration.

M.6  The three serving consequences that drive the entire delta

State these once; every iteration delta downstream traces to exactly one of them. FLOP = Floating-Point Operation; HBM = High-Bandwidth Memory (the GPU’s on-package RAM); MFU = Model FLOPs Utilization (how much of the GPU’s peak FLOP/s the work actually uses); E/P/D = Encode / Prefill / Decode.

  1. A new compute stage with the opposite roofline. The ViT is compute-bound, fixed-depth, no-KV, a single bidirectional forward; the LLM decode is memory-bandwidth-bound, autoregressive (K12). Colocating a ViT in every LLM task (the common baseline) carries encode capacity on every task though only ~15% of requests use it, and lets a compute-bound ViT burst steal cycles from latency-sensitive decode — the HBM cost is not the issue (1.2 GB ViT vs ~140 GB LLM weights, i.e. 0.6 B vs 70 B params × 2 B). → Iter M1 (E/P/D disaggregation, a dedicated encode pool, K12 D41) + Iter M3 (ragged encoder batching) — the scale-out escalation off the colocated baseline, not table stakes (a smaller deployment keeps the ViT colocated; full argument in the Iter M1 note-box).
  2. Far more KV per request. Same KV_per_tok = 320 KB (K2 D31), but the vision-token count per request is 1 k–1 M. → stresses the parent’s tiering, PagedAttention, the cross-task pool at every attachment, and — only at the near-1 M-context extreme — triggers context parallelism from a single attachment; it also shifts the whole workload distribution (S0 delta).
  3. The encoder output is deterministic & content-addressable. Same bytes + same preprocessing + same encoder_version → bit-identical embeddings. → a new cache layer above prefill (Iter M2 vision-embedding cache) and a content-addressed reuse axis the text pool had nothing for: per-user cross-conversation reuse always, plus opt-in cross-user dedup of the public/shareable slice (private uploads stay isolated — Iter M2 DC3). The headline reversal of the text page’s “nothing to dedup” verdict.
Bridge to the iterations. §M.6’s three consequences are the spine of everything below: consequence 1 (opposite roofline) → the encode pool + ragged batching; consequence 2 (far more KV) → the Part-A volume deltas to tiering / paging / pool / context parallelism; consequence 3 (deterministic, content-addressable output) → the content-addressed embedding cache — the headline reversal of the text page’s “nothing to dedup.” The S1–S3 deltas name the new jobs, entities, and API surface those iterations need; the iteration ribbon then slots them in.

Iter 1a delta, expanded — why the watermark band widens for vision

  • The pair, and why a pair (anti-thrash, unchanged in form). Tiering keeps a low watermark (evict/write-down until free-HBM rises above it) and a high watermark (do not re-trigger eviction until used-HBM crosses it). Two thresholds, not one, so the system drains in a batch and then rests — a single threshold would evict-one / admit-one on every block and thrash.
  • The asymmetry (what changes). The gap hi − lo is set by (bytes freed per eviction step) × (burstiness). Text blocks are small and steady; a finished vision attachment frees a huge, bursty tail (a video’s 49 GB drains in one step). To absorb that without immediately re-triggering, the band must be wider for the vision tier — a symmetric pair, asymmetrically thick, sized by drain-burst size, exactly as the parent justified the HBM↔DRAM band thickness by drain speed.

Iter 8 delta, expanded — token/byte-weighted quota

  1. The problem, step by step. (1) The parent token bucket debits 1 per request. (2) A multimodal request is not one unit of work: a 50 MB video debits the same 1 as a one-line chat. (3) So a client can upload a few tiny-looking videos and consume 150 k prefill tokens + encode FLOPs the limiter never sees (BM6).
  2. The solution, step by step. (1) Keep the token-bucket algorithm. (2) Replace the cost function: debit cost = base + vision_tokens·w_kv + encode_FLOPs·w_enc, so the bucket drains in proportion to true fleet cost. (3) Add a cap max_attachment_tokens_per_user_per_window. A 50 MB video now debits its real cost; the limiter sees it.

Iter 9 delta, expanded — context parallelism (CP) fires at turn 1

  • What changes. The parent triggers CP only on long chat history. Now a single near-1 M-context attachment — a ~600-page PDF (~900 k tokens, 288 GB) or a ~1-hour video (~1 M tokens, 320 GB) — can cross the fork at turn 1, off a brand-new conversation.
  • What does not. The threshold CP_FORK_THRESHOLD_BYTES = 256 GB (≈ 800 k tokens) is unchanged (an HBM-bytes threshold); Ring Attention is unchanged. Only the trigger source changes → the Router computes est_kv from T_text + Σ n_vis_tok(attach) before placement.
  • Frequency is still rare. The §S0.3 p99 attachments (100-page PDF / 10-min video ≈ 48–49 GB) are 5× below the 256 GB fork — they fit one task. Only beyond-p99, near-context-cap attachments fork, matching the parent’s “~0.1% of QPS” CP framing. The change is the source (a single upload, not 100 chat turns), not the rate.

NEW Iter M0 — Blob ingress + preprocessing pipeline

M0.1  Architecture (diagram first)

  • ① POST /v1/attachments — control-plane call; small JSON, no bytes.
  • ② Presigned URL — service returns a direct-to-store upload URL.
  • ③ PUT bytes (out-of-band) — MB–GB blob goes client→store, never through the Router.
  • ④ Populate cache — object store fronts a hot-blob cache / CDN for re-reads.
  • ⑤ Pull bytes (hot) — preprocess workers read from the cache.
  • ⑥ Cold-miss read — on a cache miss, workers read straight from the durable store.
  • ⑦ Decoded frames / pages — preprocessed tensors handed to the encode pool.
  • ⑧ Embeddings → serving task — the encode pool streams vision embeddings to the serving-prefill-job task, where they splice into the token stream and become regular KV (K2·D32). The encode pool + handoff are introduced at Iter M1; the serving task is the parent page’s fleet.

Per-component why-exists / stores / updated-when

  • Attachment service — why: mint attach_id + presigned URL; stores: the Attachment row; updated: on upload-init and on status transitions.
  • Object store (S3 / GCS) — why: durable bytes = the RPO=0 (Recovery Point Objective) root of the media path; stores: raw blobs; updated: once on upload, then immutable.
  • Hot-blob cache / CDN — why: serve re-reads (a re-referenced PDF page next turn) without hitting cold storage; stores: recent / popular blobs; updated: on first read + eviction.
  • Preprocess workers (CPU + NVDEC) — why: keep CPU decode off the ViT GPU; stores: nothing durable; updated: per attachment.

M0.2  Why introduce this now (after Iter 1b)

  • Enabling prior: the L4 + Router ingress (Iter 1b) exists; attachments need somewhere to land before any encode.
  • Constraint that flipped: text fits in the request body (KBs); a 50 MB video does not — it would blow the request path and the Router heap. Bytes must move out-of-band.
  • Why not inline base64 in the message? 33% size inflation + holds the HTTP/2 stream open for the whole upload + re-sends on retry. Out-of-band presigned upload is the standard answer.
  • Roofline note: video decode is its own compute (NVDEC on GPU, or CPU ffmpeg); frame-sample + page-render are CPU/GPU work that must not sit on an H100 serving task.

DC1 — Upload path

Problem: a 50 MB video cannot ride the JSON request body.

  • (c) proxy-through-Router — turns the Router into a byte pump.

Winning constraint:

  • Only (b) keeps MB–GB bytes off the latency-critical control plane while staying retry-safe.

DC2 — Blob store

Problem: where do durable attachment bytes live?

  • self-hosted MinIO — works but adds ops cost / on-call.
  • Redis / blob-in-KV-store — RAM cost for cold MBs is absurd.

Winning constraint:

  • Cold MB–GB blobs want cheap durable object storage, not RAM; managed S3/GCS gives RPO=0 + multi-AZ for free.

  • (b) dedicated preprocess workers (CPU + NVDEC) — CHOSEN.
  • (c) on the client — untrusted; format inconsistency.

Winning constraint:

NEW Iter M1 — Encode/Prefill/Decode (E/P/D) disaggregation: the vision-encoder pool

Bottleneck BM1 · cite K12 D41 · placed right after Iter 2’s P/D split · diagram D-M2

M1.1  Architecture (diagram first) — three rooflines, three pools

  • ① HTTP / SSE — client request enters the L4 load balancer.
  • ② Route — L4 hands to a Router task.
  • ③ Attachment → blob/preprocess — Router triggers the Iter M0 path for the bytes.
  • ④ Decoded frames — preprocess feeds the encode pool.
  • ⑤ Encode dispatch — Router routes by CH-on-content_hash → identical attachments coalesce (single-flight).
  • ⑥ Embeddings (gRPC over RDMA) — [n_vis_tok × d_model] ≈ 21 MB/image streamed to prefill.
  • ⑦ Text path — Router sends text tokens to the prefill task chosen by CH-on-conv_id (parent path).
  • ⑧ K,V handoff (RDMA) — prefill streams produced K,V to the decode pool (parent Iter 2 transport).

The whole point in one line: three stages, three different rooflines — encode is compute-bound with no KV, prefill is compute-bound, decode is bandwidth-bound — so each gets its own independently-scaled pool, exactly as Iter 2 split prefill from decode.

M1.2  Why introduce this now (right after Iter 2’s P/D split)

  • Enabling prior: Iter 2 already split rooflines (prefill compute-bound vs decode BW-bound) into separate pools. M1 is the same move, one stage earlier: the encoder is a third roofline (compute-bound, fixed-depth, no KV, single bidirectional forward — §M.6.1).
  • Constraint that flipped: colocation (the common baseline) provisions encode capacity on every serving task — sized to total QPS — but only f_mm = 0.15 of requests carry an attachment, so ~85% of that capacity sits idle. And when a colocated ViT does fire, its compute-bound burst steals SM (Streaming Multiprocessor) cycles from the latency-sensitive prefill/decode sharing the GPU. The HBM cost is not the issue: 1.2 GB of BF16 ViT weights (0.6 B × 2 B) is a rounding error next to the LLM’s ~140 GB (70 B × 2 B).
  • Why disaggregate? Three reasons — all the Iter 2 argument one stage earlier: (1) independent scaling — encode capacity tracks λ_attach, not total QPS (~6.7× less over-provisioning at f_mm = 0.15); (2) no compute interference — the ViT’s compute-bound burst no longer jitters decode; (3) batching — a dedicated pool batches ViT forwards across requests (Iter M3 ragged), which a per-task colocated ViT cannot. Plus a cheaper SKU (no 70 B weights to hold).
  • The handoff (K12 D41): the encode pool emits embeddings ([n_vis_tok × d_model]) over the same gRPC-over-RDMA (Remote Direct Memory Access) transport as the Iter 2 K,V handoff — a smaller payload: 1.3k × 8192 × 2 B ≈ 21 MB/image vs the 1.3k × 320 KB ≈ 416 MB of K,V that prefill produces from them.
Baseline vs. this page’s target. The common production baseline is colocation: vLLM and SGLang run the ViT inside the same serving engine, on the same GPU as the LLM, and a 0.6 B ViT is cheap to carry. Disaggregation is the scale-out move — it earns its extra hop only once attachment volume (λ_attach) is high enough that independent scaling + dedicated ragged batching + a cheaper SKU dominate. This page builds the scaled target; a smaller deployment would stop at colocation.

M1.3  Design choices

DC1 — Encode placement

Problem: where does the ViT run?

  • (a) colocated ViT in every serving task (the common baseline) — encode capacity idle ~85%, compute-bound bursts jitter decode, no cross-request ViT batching.
  • (b) dedicated vision-encoder-job pool, autoscaled — CHOSEN (K12 D41).
  • (c) external 3rd-party vision API — added latency + data-egress + no cache control.

Winning constraint:

  • Only a dedicated pool lets encode capacity track attachment arrival independently of LLM capacity, batch ViT forwards across requests for MFU, and keep the (deterministic) output under our cache control.

DC2 — Encode-task hardware

Problem: which GPU SKU for a compute-bound 0.6 B ViT?

  • (a) same 8×H100 as serving — overkill; no 70 B weights to hold.
  • (b) smaller / cheaper GPU (L40S / A10 / 1–2×H100) — CHOSEN, sized by enc_throughput_per_task.
  • (c) CPU — ViT is GPU work.

Winning constraint:

  • The ViT (1.2 GB BF16 weights) fits a 48 GB L40S with room for big ragged batches; the compute-bound roofline means a cheaper FLOP/$ SKU wins over H100.

DC3 — Embedding handoff transport

Problem: move ~21 MB/image embeddings from encode to prefill.

  • (a) gRPC-stream over RDMA (reuse Iter 2) — CHOSEN — trivial at 21 MB.
  • (b) write to blob store + prefill pulls — adds a hop for a small payload, but this becomes the cache-write path in the next iteration.
  • (c) one-sided RDMA verbs — payload too small to justify; defer.

Winning constraint:

  • 21 MB is negligible on the existing RDMA fabric; reusing the Iter 2 transport adds zero new infrastructure.

DC4 — Encode↔prefill binding

Problem: which encode task, and which prefill task?

  • Router routes the attachment to an encode task by CH-on-content_hash (coalesces identical attachments → single-flight); the encode task streams embeddings to the prefill task chosen by CH-on-conv_id (the parent path). Two different hash keys, one for dedup, one for conv locality.

M3.2  Why introduce this now (after Iter 4 + Iter M1)

  • Enabling priors: Iter M1 gave a dedicated encode pool to optimize; Iter 4 established the ragged / varlen batching pattern (FlashAttention cu_seqlens, end-of-buffer-only padding) — M3 is that pattern applied to the ViT.
  • Constraint that flipped: images / PDF-pages / frames arrive at wildly different resolutions → different patch counts (a thumbnail = 64 patches; a 4K screenshot = 20 k). A naive fixed-shape ViT batch pads every image to the largest → encoder MFU collapses (half the batch is padding).

  • (b) size-bucketed — residual pad + queueing latency.
  • (c) ragged / varlen concat (FlashAttention cu_seqlens) — CHOSEN (K12 D27).
  • (d) fixed N_query resampler — caps detail (the §M.5 connector axis, orthogonal).

Winning constraint:

  • Only ragged concat removes intra-batch padding without bucket-fill latency or detail loss — it recovers MFU directly.

DC2 — Batch admission knob B_enc

Problem: how many images per ragged batch? This knob has a cliff on both sides.

  • Floor cliff: B_enc too small → ViT weight-read dominates, compute-starved (the same per-tick weight read whether 1 or many images).
  • Ceiling cliff: B_enc too large → activation footprint + latency blow up.

Winning constraint:

  • Same shape as Iter 4’s chunk knob; the range is settled at compose against the image-size distribution (S0.3): pick the smallest B_enc past the floor that fits the activation budget.

DC3 — PDF / video intra-attachment batching

  • A PDF’s pages and a video’s frames are independent images → batch them together raggedly (one PDF = one ragged batch of N pages), improving MFU on multi-page / multi-frame attachments — no waiting on other users.

M3.4  Number chain

enc_throughput_per_task → rises by the MFU-recovery factor 1 / (1 − pad_fraction) pad-to-max pad_fraction ≈ 0.5 → recovery ≈ 2× (ragged removes it) ⇒ enc_task_num shrinks proportionally [D, settled at S6]

M3.5  What breaks

  • Adversarial all-same-huge-size batch → activation OOM (Out Of Memory) → cap total patches/batch.
  • Ragged kernel tuning at extreme aspect ratios → pad-to-tile within varlen (the Iter 4 mitigation).
  • Head-of-line: one giant image (20 k patches) in a ragged batch → chunk the ViT forward (mirror Iter 4 chunked prefill) for the encoder.

NEW Iter M2 — Content-addressed vision-embedding cache (the genuinely new cache layer)

Bottleneck BM2 · cite K12 (embedding cache) + K2 D52 (privacy probe) · placed after Iter 5 · diagram D-M3
The headline reversal — and exactly what it buys. Parent §8b DC3 rejected content-addressed pool keys for text (“the one big shared blob hidden_sys is pinned per task, never pooled → nothing to dedup”). For vision a dedup-able content class actually exists, so content-addressing buys two distinct things, and it is important not to conflate them:
  1. Per-user, cross-conversation reuse (the guaranteed win) — the same user’s same attachment hits across different convs and after the conv’s K,V is evicted from the pool. This is a strict superset of the parent’s conv-scoped hit, and it needs no public flag — it is keyed hash(tenant_id ‖ content_hash).
  2. Cross-user dedup (the opt-in win) — byte-identical uploads shared across users, but only the public/shareable slice by default, because DC3 keys private uploads per-user. A viral meme or shared PDF a user uploads privately does not dedup cross-user unless flagged shared.
So the optional key space becomes load-bearing here — strongly for (1), opt-in for (2). Per-modality decision; it does not retroactively change distributed_kv_cache.html / implies no DESIGN_CHOICE_CHANGE.md entry for the text page.
Baseline vs. frontier here too. Layer (i) — caching the encoder output per attachment — is already mainstream: vLLM ships a multimodal encoder-output cache, and per-conversation prefix caching that hashes vision tokens is standard in vLLM / SGLang. What is frontier (large-operator, not the common baseline) is the cross-user, content-addressed dedup behind the privacy gate (layer ii + DC3): keying vision K,V by content_hash across tenants and defending the K2 D52 timing probe. A smaller deployment ships layer (i) + conv-scoped reuse and stops there.

M2.1  Architecture (diagram first) — two cache layers, content-addressed

  • ① Lookup — Router probes the embedding cache by content_hash before any encode.
  • ② HIT → embeddings (green = fast path) — cached embeddings stream straight to prefill; encode skipped.
  • ③ MISS → dispatch — on a miss the Router dispatches an encode (Iter M1, single-flight on content_hash).
  • ④ Write embeddings — the encode result is written back to the cache for the next requester.
  • ⑤ Embeddings → prefill — freshly-encoded embeddings also go to prefill.
  • ⑥ Write vision-K,V — prefill writes the content-addressed vision K,V into the Iter 3b pool (layer ii).
  • ⑦ HIT → skip vision prefill (green) — a pooled vision-K,V hit lets prefill skip the vision span entirely.
  • ⑧ K,V → decode — unchanged parent handoff.

M2.2  Why introduce this now (after Iter 5)

  • Enabling priors: Iter M1 made the encoder output a clean standalone artifact (embeddings, pre-prefill) — a well-defined thing to cache. Iter 5 built the content-hash chain; Iter 3b built the pooled store. M2 reuses both — the cache co-tenants the existing kv-store-job as a second key space (no new job).
  • Constraint that flipped (§M.6.3): the encoder is deterministic — same bytes + same preproc_params + same encoder_version → bit-identical embeddings. So re-encoding the same bytes is pure waste (BM2).
  • What is genuinely net-new over the parent (do not over-count): within an admitted conv, the parent conv-keyed pool already serves a re-referenced attachment (no re-encode, no re-prefill) — so vis_intra_conv_reref = 0.7 is mostly already covered. M2 adds value only where the conv-keyed pool cannot reach: (a) the same user’s attachment across convs / after the conv’s K,V is evicted (per-user key), and (b) the public/shareable slice of vis_cross_user_dup = 0.2 (cross-user key). These two, not the raw 0.7, set the incremental hit rate.
  • What to cache — two layers, both keyed by content_hash:
    • (i) Embedding cache (encoder output, pre-prefill) — saves the ViT FLOPs; small payload (~21 MB/img). This is K12’s embedding cache. Applies to every attachment that re-hits (a 21 MB fetch always beats re-encoding), but ViT FLOPs are only ~3.5% of the per-attachment compute (§S3), so this is the smaller saving.
    • (ii) Content-addressed vision-K,V (post-prefill, in the Iter 3b pool) — saves the LLM prefill FLOPs on the vision span (the larger cost); payload ~416 MB/img; keyed by (content_hash, chain-pos) via the Iter 5 chain delta. Gated by the ~1.9 k recompute-vs-transfer crossover: a lone image (~1.3 k vision tokens) sits below it — re-prefill beats a pool fetch — so layer (ii) pays off for PDFs / video / multi-image, not single images.
    Both ship; (i) is the cheap universal encode-FLOP trim, (ii) the larger prefill-FLOP win where the attachment clears the crossover.

M2.3  Design choices

DC1 — What to cache

Problem: an encode and a vision-prefill are both repeatable work — cache which?

  • (a) nothing (re-encode every time) — wastes ViT FLOPs.
  • (b) embedding cache only — saves ViT, but still re-prefills the vision span.
  • (c) embedding + content-addressed vision-K,V — CHOSEN — saves ViT and vision-prefill.

Winning constraint:

  • The vision span is the dominant prefill cost for a multimodal request (the embedding cache alone trims only the ~3.5% encode FLOPs), so layer (ii) is what makes M2 a material $ saving.
  • But layer (ii) is conditional: below the ~1.9 k recompute-vs-transfer crossover (inherited from the parent length gate), re-prefilling a lone image’s ~1.3 k tokens beats pulling its ~416 MB of K,V over the fabric — so layer (ii) is admitted only for attachments above the crossover (PDFs / video / multi-image). Layer (i) embedding cache still applies to lone images.

DC2 — Key

Problem: what identifies “the same attachment”?

  • (a) attach_id — per-upload; never dedups.
  • (b) content_hash = hash(bytes ‖ preproc_params ‖ encoder_version) — CHOSEN — dedups byte-identical uploads fleet-wide; version-safe.

Winning constraint:

  • Only a content hash collapses re-uploads of identical bytes across users and invalidates cleanly on an encoder upgrade.

DC3 — Privacy (the K2 D52 timing probe, sharper than text)

Problem: cross-user dedup means a user can upload an image and detect a cache hit via latency → learn someone else uploaded it. Worse than the text front-of-chain probe — images are often private photos / documents.

  • (a) dedup everything (max savings) — leaks “this exact file exists in the fleet” for private uploads.
  • (b) no dedup; per-user-keyed everywhere — kills the cross-user win entirely; only intra-user re-reference hits.
  • (c) two-class dedup — CHOSEN — dedup derived / public-ish content freely (fleet-template assets, hidden_sys companions), but per-user-key private uploads by default (seed = tenant_id); promote to shared dedup only on an explicit public/shared flag.

Winning constraint:

  • Media exposure is higher-value than text, so the default flips to isolation — the opposite of the text page’s accepted front-of-chain probe (K2 D52); the cross-user win is opt-in, not default.
  • Consistency with the Iter 5 chain delta: when private-keyed, the vision span’s chain hash includes tenant_id → no cross-user prefix hit either. One switch (seed = tenant_id) governs both cache layers.

DC4 — Storage home

Problem: where do embeddings + vision-K,V live?

  • co-tenant the Iter 3b kv-store-job (second key space) — CHOSEN — same Mooncake-style backend + replication.
  • a new dedicated embedding store — duplicate infra for a blob class the pool already serves.

Winning constraint:

  • Embeddings are just another content-addressed blob class; reusing the pool’s sharding + replication costs nothing new.

DC5 — Eviction (and both watermarks)

  • Policy: content-addressed entries use an LFU-leaning (Least Frequently Used) policy — a viral asset is hot fleet-wide, and recency-only (LRU, Least Recently Used) would drop a periodically-popular logo between bursts. Contrast the parent pool’s capacity-LRU for conv K,V.
  • Both watermarks, and the asymmetry: the hi/lo pair exists to batch-drain then rest (anti-thrash, as everywhere). The dedup set is small + hot → a narrow band (few bytes move per step). The per-user private set is large + cold → a wider band (more bytes per drain). Same symmetric pair, band thickness sized by drain-burst size — consistent with the Iter 1a derivation.

M2.5  Number chain

enc_cache_hit_rate ≈ P(per-user cross-conv / post-eviction reuse) + vis_public_share·vis_cross_user_dup ≈ 0.4 [G-FA — INCREMENTAL over the conv-keyed pool, NOT the raw 0.7 (that is already parent-covered)] ViT FLOPs saved = enc_cache_hit_rate × λ_attach × enc_FLOPs(attach) (enc_FLOPs ≈ 3.5% of per-attach compute → small $ line) ⇒ enc_task_num: 2,760 → ~1,660 tasks (M1 number chain) vision-prefill saved = (layer-ii hit rate; attachments > ~1.9 k only) × vision-prefill_FLOPs ⇒ prefill_task_num shrinks ← the LARGER $ win W_pool += embedding set (~21 MB/img) + vision-K,V set (~416 MB/img, PDFs/video/multi-image only) [re-derive at S6]

M2.6  What breaks

  • encoder_version bump invalidates the whole embedding cache → cold-encode storm → versioned warm-up / staged rollout (the parent’s shadow→canary→ramp).
  • Hash collision on content_hash → wrong image served → use SHA-256 (Secure Hash Algorithm, collision-free at scale), not a fast non-crypto hash.
  • Privacy regression if a private upload is mis-flagged public → default-isolate + audit.
  • Dedup hot-key (viral asset) pull storm → front-replicate in the pool (parent Iter 3b mechanism).

Δ S6 / S7  Final design & takeaways (delta)

S6.1  Final fleet architecture — the parent fleet plus the multimodal stack (D-M6)

This is the parent page’s final-design diagram (D13) with every multimodal addition folded in — not a delta sketch but the whole settled fleet in one picture. The new pieces are drawn in amber: the blob-store + preprocess ingress (Iter M0), the dedicated encode pool with its ragged batcher (Iter M1 + M3), and the embedding cache as a second key space co-tenanted on the existing kv-store-job (Iter M2). The parent five-job fleet (Router, broker, prefill, decode, kv-store, cp_ring) is unchanged in structure — only its inputs widen (a vision token is just a token).

  • ① Clients → L4 LB — HTTP opens an SSE stream; the managed L4 LB terminates TLS and holds the stable VIP (parent).
  • ② L4 LB → router-job — round-robin across the 3 HA Router tasks (parent).
  • ── ③ Router → blob/preprocess — the attachment path (Iter M0): Router issues a presigned URL, bytes land in the object store, NVDEC preprocess workers decode frames/pages.
  • ── ④ Preprocess → encode pool — decoded frames/pages feed the vision-encoder-job (Iter M1).
  • ── ⑤ Router → encode (on cache MISS) — dispatch by CH-on-content_hash over the encode ring, so identical attachments coalesce (single-flight, Iter M2).
  • ┄┄ ⑥ Encode ↔ embedding cache (key space 2) — encode writes back the content-addressed embeddings; on a later HIT the Router (④ lookup) returns these cached embeddings and skips encode entirely (Iter M2).
  • ── ⑦ Encode → prefill — [n_vis_tok × d_model] ≈ 21 MB/image streamed over the Iter 2 gRPC-over-RDMA transport (much smaller than the 416 MB of K,V prefill then produces).
  • ⑧ Router → prefill (text path) — text tokens routed by CH-on-conv_id after the Iter 8 admit gate clears (parent hot path); prefill splices the vision embeddings in at their placeholder positions.
  • ── ⑨ Prefill → decode (K,V handoff) — prefill ships the turn’s K,V (text + vision, 320 KB/tok) to the conv’s sticky decode task (parent Iter 2).
  • ⑩ Decode → Clients (SSE) — output tokens stream back through the L4 LB; the encode/prefill/decode handoffs are invisible to the client.
  • ── ⑪⑫ Serving pools ↔ kv-store key space 1 — write-back on local eviction (async, length-gated > ~1.9 K tok) down; pull on local miss (sync) up; recompute is the RPO = 0 floor (parent Iter 3b).
  • ┄┄ ⑬ Both pools → broker (async gossip) — each task publishes approx_wait + ring epoch every T_gossip = 100 ms (parent Iter 1b).
  • ┄┄ ⑭ Broker → Router (async fan-out) — ring epochs + per-task wait sketches fanned out to every Router (read-replica pattern, parent).
  • ⋯⋯ ⑮ Router → cp_ring (on-demand fork) — a request whose est_kv (now T_text + Σ n_vis_tok) exceeds 256 GB forks onto the 8-rank CP ring; the Iter 9 delta makes this reachable from a single ≥ 800 K-token upload at turn 1.

Read the diagram. The parent fleet (blue/green/purple/pink boxes) is structurally untouched — a vision token becomes regular KV (K2 D32), so prefill, decode, and the conv-keyed pool serve it with no new machinery. Three amber additions carry the entire delta: ③–④ (Iter M0) turns uploaded bytes into decoded frames; the encode pool (Iter M1+M3) runs the ViT on a cheap compute-bound SKU with no KV and a ragged batcher, exactly mirroring how Iter 2 split prefill from decode for a different roofline; and key space 2 (Iter M2) co-tenants a content-addressed embedding cache on the existing kv-store-job — the one genuinely new reuse axis, gated private-vs-shared because cross-user dedup re-opens the K2 D52 timing probe. The only parent boxes whose parameters move are prefill (sized against T_in_blended = 14.8 K) and the CP-fork trigger (est_kv now sums attachment tokens, so one upload can fork at turn 1).

S6.2  Final-design additions (cost & degradation)

  • The final SVG above folds in: blob-store + preprocess (M0), the encode pool (M1), the embedding cache as a second key space on kv-store-job (M2), the ragged encoder batcher (M3), and the E/P/D three-stage pipeline.
  • Optimization-stack table gains the multimodal rows: encode-pool cost, embedding-cache savings (ViT + vision-prefill), ragged-batch MFU recovery.
  • Degradation contract adds: encode pool down → fall back to colocated emergency encode (slow) or reject multimodal with 503; blob store down → reject upload; embedding cache down → re-encode (RPO=0 from blob).
  • Fleet comparison now reports prefill_task_num against T_in_blended = 14.8 k (not 13.3 k) and adds enc_task_num × $_per_L40S_task_per_mo.

S7  New takeaways (added to the parent’s list)

  1. An image, a PDF page, and a video frame are the same object to the system: bytes → ViT → embedding tokens → regular KV (K2 D32). The text page’s KV cache already serves them — the delta is volume, a new upstream stage, and a new reuse axis.
  2. The encoder has the opposite roofline to decode — so it gets its own pool (E/P/D), the same way prefill and decode split at Iter 2.
  3. Content-addressing was rejected for the text pool (nothing to dedup) and is load-bearing for vision — a real shareable content class now exists. It guarantees per-user cross-conversation reuse (a superset of the conv-keyed hit) and adds opt-in cross-user dedup of public/shared assets (private uploads stay isolated). The right cache key is a function of the modality’s reuse pattern, not a global constant.
  4. Caching the encoder output (embeddings) saves ViT FLOPs (~3.5% of per-attach compute — the small win); caching the vision K,V saves prefill FLOPs (the larger win, but only above the ~1.9 k recompute-vs-transfer crossover, so PDFs/video/multi-image) — two layers, two payload sizes, both keyed by content_hash.
  5. Cross-user dedup re-opens the K2 D52 timing probe, and for private media the exposure is higher than for text — so media defaults to per-user keying, dedup only on an explicit shared flag.
  6. A single near-1 M-context attachment (a ~600-page PDF / ~1-hour video, ≥ 256 GB KV) can trigger context parallelism at turn 1 — the long-context machinery the text page built for rare 1 M chat histories is now reachable from one upload. (Typical p99 attachments — 100-page PDF / 10-min video, ~48–49 GB — stay well under the fork.)
