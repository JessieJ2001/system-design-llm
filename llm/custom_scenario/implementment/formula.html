Formula Reference — Distributed KV Cache

A companion cheat-sheet to distributed_kv_cache.html: every formula used in that page’s calculations, collected in one place — the constants they substitute, the reusable building blocks, the crossover/decision formulas that pick a design, and the per-iteration result chain. Each entry links back to where it is derived and used.

Source of truth: distributed_kv_cache.html · upstream: K1 serving fundamentals / K2 KV cache · every number here is given or derived on the source page — nothing new is introduced

How to read this sheet

The source page builds a distributed KV-cache serving fleet in twelve bottleneck-driven iterations. The same handful of formulas are substituted over and over with different inputs. This sheet deduplicates them: the reusable formulas appear once (F-series), the formulas that pick a design appear once (C-series), and the per-iteration outputs are collected in one ledger. Substituted arithmetic is shown once per formula as a representative example; the full chain lives on the source page.

Provenance tags (carried verbatim from the source page): [G] given / specified · [D] derived on the source page · [G-FA] given, flagged assumption (would confirm in interview) · [G — K#] given by an upstream knowledge page.

The calculation chain at a glance

Almost every iteration produces its fleet size and cost by walking the same spine: from per-token compute up to dollars per month. The blue spine is the cost chain; the slate boxes are the feeder constants/formulas that flow into it.

  • █ Blue spine — the cost chain every iteration walks (F1→F2→F10→F9→F11→F12).
  • █ Slate feeders — constants/formulas that flow sideways into a spine box.
  • → solid = spine dependency · → dashed = feeder input.

Iterations branch off this spine: P/D disaggregation (Iter 2) adds xfer(N); continuous batching (Iter 6) replaces 1/req_time with Bdecode/tick(B); prefix caching (Iter 5) shrinks N in F1. The crossover formulas (C-series) decide when each branch is taken.

1. Foundational formulas (reused every iteration)

F1 · Per-request prefill compute §4.1 · from K1 §3.3

prefill_FLOPs(N) = 2·P·N + 4·n_layers·N²·d_model = 1.4e11·N + 2.62e6·N² (P=70e9, L=80, d_model=8192)

N = tokens to prefill this turn. Linear term = one pass through every weight (the “2× FLOPs per parameter” rule); quadratic term = self-attention over the growing context. Quadratic overtakes linear at s* = 6·d_model ≈ 49 K (C2). Worked: N=13K → 2.32 PFLOP; N=256K → 207.5 PFLOP.

F2 · Prefill time & sustained compute §4.1, §6.6

F_sustained(TP) = TP × GPU_peak_BF16 × MFU = TP × 989e12 × 0.40 prefill_time(N, TP) = prefill_FLOPs(N) / F_sustained(TP) TP=8: prefill_time(13K) = 2.32e15 / 3.17e15 = 730 ms (= β_cold(13K)) TP=8: prefill_time(200) = 2.81e13 / 3.17e15 = 8.9 ms (warm re-prefill)

This is the cold-miss / re-prefill cost — equals βcold(N) in the β family. MFU (40%) folds in kernel-launch, memory-stall and collective-wait losses. Batch scaling (no caching): prefill_time(B×N) ≈ B × prefill_time(N) — this is why cold prefill batching blows the TTFT SLO (§6.4 DC4: B=18 → 13.1 s ≫ 400 ms).

F3 · Per-token KV size §4.2 · K2 D31

KV_per_tok = 2 · n_kv_heads · d_head · n_layers · bytes_per_element = 2·8·128·80·2 = 327 680 B ≈ 320 KB/tok (canonical literal)

The 2 = one K + one V tensor. Grouped-query attention keeps n_kv_heads=8 (not 64), the dominant KV-size reduction. The ~2.4% rounding to 320 KB sits inside workload uncertainty.

F4 · KV slab size §4.2, §8.5

slab(N) = N × KV_per_tok 13K → 4.16 GB (cold, T_in_mean) 200 → 64 MB (warm, user msg) 256K → 82 GB (p99 = one H100’s HBM) 1M → 320 GB (p99.9 → CP fork)

The bytes that must be transferred (Iter 2), pooled (Iter 3b), or held resident. Linear in context length N.

F5 · Weights per GPU §6.6

weights_per_GPU(TP) = P × bytes_per_element / TP TP=8: 70e9 × 2 / 8 = 17.5 GB / GPU TP=2: 70e9 × 2 / 2 = 70 GB / GPU per task (unsharded): 140 GB

Tensor parallelism shards the weight matrices across TP GPUs, so each GPU re-reads only its 1/TP slice per decode tick — the lever F6 turns into decode speed.

F6 · Per-output-token decode tick §6.6

β_per_decode_tok = weights_per_GPU / HBM_BW_per_GPU TP=8: 17.5 GB / 3.35 TB/s = 5.22 ms/tok TP=2: 70 GB / 3.35 TB/s = 20.9 ms/tok

Decode at batch B=1 is HBM-bandwidth-bound: the tick is just the time to stream all weight bytes once. TP=8 is 4× faster than TP=2 because each GPU holds 4× fewer weight bytes. The KV-read term is negligible at B=1 (see F8 for B>1).

F7 · Decode tick = max(Load, Compute) §5.4 Step 3 · K1 §3.5

# The honest tick is the max of the two bottlenecks; at B=1 Load wins (F6 is the shortcut). Load(TP, B, c) = weights_per_GPU / HBM_BW_per_GPU + B·c·KV_per_tok / (TP·HBM_BW) Compute(TP, B, c) = (2·P·B + 4·B·n_layers·c·d_model) / (TP × GPU_peak_BF16) β_per_decode_tok = max(Load, Compute) TP=2, B=1, c=13K: Load = 20.9 ms , Compute = 0.09 ms → max = 20.9 ms (load-bound)

c = cached context length (KV entries the new token attends over). Decode-attention uses factor 4 (4·B·n_layers·c·d_model) — QKᵀ + AV are two matmuls, each MAC=2, the same convention as the 2·P weight term and prefill-attention’s factor 4 (F1). Compute uses raw peak (no MFU) as a conservative floor; decode is load-bound across the whole realistic workload, so the F6 weights-only shortcut holds until B grows the KV-read term (F8).

F8 · Continuous-batching decode tick & throughput §12.6

kv_per_seq_per_GPU = (T_in_mean + T_out_mean/2) × KV_per_tok / TP = 13.2K × 40 KB ≈ 528 MB tick(B_decode) = β_per_decode_tok + B_decode × kv_per_seq_per_GPU / HBM_BW_per_GPU = 5.22 ms + 106 × 528 MB / 3.35 TB/s = 5.22 + 16.7 ≈ 22 ms (= TPOT, < 30 ms SLO) thr(B=1) = 1 / β_per_decode_tok = 1 / 5.22 ms ≈ 192 tok/s/task thr(B_decode) = B_decode / tick(B_decode) = 106 / 22 ms ≈ 4 818 tok/s/task multiplier = thr(B_decode) / thr(B=1) ≈ 25× (not 106×: KV reads now dominate the tick) req_throughput = thr(B_decode) / T_out_mean = 4 818 / 400 ≈ 12 req/s/task

Once weights are shared across a batch, the per-tick KV read grows linearly in B and becomes the new bottleneck — the +Tout_mean/2 term averages each slot over its decode arc. The honest speedup is ~25×, not the naive B×.

F9 · Per-task throughput family §5.4, §8.5, §10.5

# Unified pool (Iter 0/1): req/s/task = 1 / req_time # Decode pool (B=1): decode_per_task = 1 / (T_out_mean × β_per_decode_tok) ≈ 0.479 # Decode pool (batched): req_throughput = thr(B_decode) / T_out_mean (F8) # Prefill pool (mean mix): prefill_per_task = 1 / prefill_time_mix ≈ 6.53 # prefill_time_mix = (1−hit)·prefill_time(13K) + hit·prefill_time(200) = 0.2·730 + 0.8·8.9 = 153 ms # Prefill token rate (linear): prefill_tok_throughput = F_sustained / (2·P) = 3.17e15 / 1.4e11 ≈ 22 600 tok/s

The denominator is whatever the pool is bound by: whole-request time (unified), per-token decode tick (decode pool), or prefill kernel time (prefill pool). Disaggregating P/D (Iter 2) lets each pool be sized on its own throughput.

F10 · Request time (and warm/cold mix) §5.4, §7.4, §8.5

req_time = prefill_time(N) + [ xfer(N) ] + T_out_mean × β_per_decode_tok \_____________/ \______/ \_____________________________/ first-token cost P/D only decode cost req_time_mix = (1 − hit) × req_time_cold + hit × req_time_warm (hit = intra_session_hit_rate) Iter 0 (TP=2): 2.93 s + 400·20.9ms = 11.29 s Iter 1a (TP=8): 0.73 s + 400·5.22ms = 2.82 s Iter 1b mix: 0.2·2.82 + 0.8·2.10 = 2.24 s (warm prefill only 200 tok) Iter 2 mix: +xfer (166 ms cold / 2.6 ms warm) = 2.28 s

The xfer(N) term appears only after P/D disaggregation (Iter 2) puts prefill and decode on different tasks. Warm turns re-prefill only the 200-tok user message because prior KV is sticky (CH-on-conv_id, Iter 1b).

F11 · Fleet sizing §1.3, §7.4

fleet_tasks = ⌈ λ_peak × headroom / per_task_throughput ⌉ # Control plane (HA-bounded): router_tasks = max( 3 (HA quorum), ⌈ λ_peak·1.3 / 100 000 dec/s ⌉ ) broker_tasks = 3 (HA quorum, BW-trivial)

Headroom = 1.3 (30%). The control-plane jobs are sized by high-availability quorum, not throughput — one router decision is ~10 µs of pure CPU (100K dec/s/task).

F12 · Monthly cost §4.4, K1 §3.10

$_per_task_per_mo = TP × $_per_GPU_hr × 720 TP=8: 8 × $12.29 × 720 = $70 790 TP=2: 2 × $12.29 × 720 = $17 698 $/mo = fleet_tasks × $_per_task_per_mo

Cost is linear in GPU-count, so cost ratios between iterations are just task-count × TP ratios. This is the single number every iteration is judged on.

F13 · Wait-time — the routing signal §6.5, §8.4, §10.5, §12.5

# Unified (Iter 1a) — the Router consumes this to pick a task: approx_wait_time_ms(task, t) = Σ over queue(task,t) ∪ running(task,t) of β(tier_residency_for(r)) # Prefill pool (Iter 2): Σ of β_prefill(r), β_prefill(r) = prefill_time(N_new(r)) [N_new: 200 warm / 13K cold] # Chunk-granular (Iter 4): approx_wait_prefill = chunks_ahead × β_chunk (β_chunk = prefill_time(4K) ≈ 190 ms) # Decode pool (Iter 6): approx_wait_decode = max(0, queue − free_slots) × (avg_seq_lifetime / B_decode) # avg_seq_lifetime = T_out_mean × β_per_decode_tok ≈ 2.09 s # Speculative (Iter 7): ... / (B_decode × accept_rate_speedup_factor) (×2.0, see C6)

Using β (per-request prep time) instead of “latency” lets queued requests sum cleanly. Chunking removes the head-of-line term (a 256K request no longer blocks behind its full 65 s); batching and speculation divide the per-admit wait.

F14 · KV transfer time (bulk / layer-wise / chunked) §8.5, §10.4, §8.3 DC2

# Per-request bulk (production default): xfer(N) = slab(N) / RDMA_per_flow = N·KV_per_tok / 25 GB/s cold 13K: 4.16 GB / 25 GB/s = 166 ms warm 200: 64 MB / 25 GB/s = 2.6 ms p99 256K: 82 GB / 25 GB/s = 3.28 s # Layer-wise streaming (transfer hides behind next layer’s compute; only last layer exposed): per_layer_compute = prefill_time(N)/n_layers per_layer_xfer = (slab/n_layers)/RDMA_BW 9.1 ms >> 2.1 ms ⇒ exposed xfer ≈ one layer = 2.1 ms (~80× less than bulk) # Chunked pipeline (Iter 4): only the LAST chunk’s KV exposed: exposed_xfer = C × KV_per_tok / RDMA_BW = 4096·320KB / 25 GB/s ≈ 52 ms (166 → 52 ms) # Placement bound: min_BW = slab / latency_budget (cold 4.16GB / 730ms = 5.7 GB/s)

Model transfer as the time-vs-bandwidth tradeoff. The bulk form is the simple slab/BW; layer-wise and chunked variants overlap transfer with compute to shrink the exposed term in TTFT. min_BW back-solves how far apart prefill and decode tasks may sit.

F15 · Little’s law (live population & pool ingest) §1.3, §9b.4

active_concurrent_sessions = λ × (1 − intra_session_hit_rate) × avg_session_lifetime = 100 000 × 0.20 × 150 = 3 M (peak) # Pool working set (Iter 3b): W_pool = p_admit × N_active × mean_admitted_conv_size (see C8) # Gossip load (Iter 1b): gossip_msg_rate = serving_tasks × (1 / T_gossip) = 291.5K × 10 = 2.9 M msg/s

Arrivals of new sessions = λ × (1−hit) because in-session turns don’t open new sessions. This 3 M figure is what every working-set and pool-QPS estimate is sized against.

2. The β family — per-request setup times by tier

Every β is the GPU time to prepare one request to emit its next token. The subscript names which tier the prep work lives in. They are ordered βHBM ≪ βDRAM ≪ βSSD ≪ βcold, and they sum to give the F13 wait-time. Source: §4.0.

3. Crossover & decision formulas (these pick the design)

Where the F-series compute “how much,” the C-series answer “which design wins, and at what threshold.” Each is the equation set equal at the point two alternatives cost the same.

C1 · Tier crossover — recompute vs transfer §6.5

# Set transfer time = re-prefill time, solve for N: N × KV_per_tok / BW_t = (2·P·N + 4·n_layers·N²·d_model) / F_sustained ⇒ N_crossover = ( F_sustained × KV_per_tok / BW_t − 2·P ) / ( 4 × n_layers × d_model )

Above Ncrossover, fetching cached KV beats recomputing it. With these constants DRAM (25 GB/s) beats re-prefill at every N; SSD (7 GB/s) wins only for N ≥ ~1.9 K tokens — which is why short turns are recomputed and long ones fetched.

C2 · Prefill quadratic-vs-linear §4.1 · K1 §3.3

s* = 6 × d_model = 6 × 8192 ≈ 49 K tok

Below ~49 K, prefill is ≈ linear in N; above it the N² attention term dominates. At 256 K, attention is ~5× the linear term (871 s at 1 M → forces CP, C5).

C3 · Decode attention-vs-weights §5.4

T_in + T_out/2 ≈ P / (2 · n_layers · d_model) = 70e9 / (2·80·8192) ≈ 53 K

The cached-context length at which decode-attention FLOPs equal the weight-pass FLOPs (attention = 4·n_layers·c·d_model, both QKᵀ+AV matmuls at MAC=2 — same convention as the 2·P weight term). The workload mean (13K+200) sits at ~25% of this — decode stays weight/BW-bound, validating F6.

C4 · PagedAttention — max batch & the capacity multiplier §9.5

G_KV = HBM_total_per_task − weights − activations = 640 − 140 − 50 = 450 GB/task block_KV_bytes = block_size × KV_per_tok = 16 × 320 KB = 5.12 MB ; total_blocks = G_KV/block_KV_bytes ≈ 87 890 # Pre-PA: static max reservation (must reserve worst-case T_out_max up front) reserved_per_slot = (T_in_mean + T_out_max) × KV_per_tok = 78K × 320KB = 24.96 GB B_max,static = ⌊ G_KV / reserved_per_slot ⌋ = ⌊ 450 / 24.96 ⌋ = 18 slots (p99 admit → only 4!) # Post-PA: pay for tokens actually used (averaged over the decode arc) avg_used_per_slot = (T_in_mean + T_out_mean/2) × KV_per_tok = 13.2K × 320KB = 4.224 GB B_max,paged = ⌊ G_KV / avg_used_per_slot ⌋ = ⌊ 450 / 4.224 ⌋ = 106 slots multiplier = reserved_per_slot / avg_used_per_slot = 24.96 / 4.224 ≈ 5.9× (24.3× at p99 context)

Paging eliminates the must-reserve-the-max waste, lifting the decode batch ceiling from 18 to 106 — the capacity headroom that continuous batching (F8) then converts to throughput.

C5 · Context-parallel fork (1 M-context requests) §15.4

per_req_KV(N) = N × KV_per_tok ; per_req_KV(1M) = 320 GB > G_KV (450 GB only at B=1) CP_FORK_THRESHOLD = 256 GB ≈ 256e9 / 320e3 = 800 K tokens (route to CP path above this) per_rank_KV = per_req_KV / N_CP = 320 GB / 8 = 40 GB/rank (N_CP = 8) tick(1M, CP=8) = β_per_decode_tok + N_CP × (per_rank_KV / HBM_BW_per_rank) + ring_RDMA = 5.22 + 8·(40GB/26.8TB/s) + 0.8 ≈ 17 ms (< 30 ms TPOT SLO)

A single 1 M-token request won’t fit one task at useful batch size, so context parallelism splits the sequence across N_CP ranks (Ring Attention). Only the ~0.1% p99.9 tail forks; the common case never pays.

C6 · Speculative decoding speedup §13.5

# Expected accepted tokens per target forward (truncated geometric, accept prob α, draft len γ): E[accept] = (1 − α^(γ+1)) / (1 − α) = (1 − 0.7⁵) / (1 − 0.7) ≈ 2.77 (α=0.7, γ=4) draft_overhead = γ × (draft_weights/GPU / HBM_BW/GPU) = 4 × (0.25GB / 3.35TB/s) ≈ 0.3 ms/tick accept_rate_speedup_factor = E[accept] × discount = 2.77 × 0.72 ≈ 2.0×

A small draft model proposes γ tokens; the target verifies them in one forward pass. The net ~2× (after draft reads, larger verify-KV, and reject rollback) halves the decode fleet — ~$382 M/mo (C-series feeds the ledger).

C7 · Prefix caching — unmatched tokens & prefill reduction §11.3, §11.6

# Block-hash chain key (the cache identity; seed isolates private content): H_block_i = hash( seed(class) ‖ parent_hash ‖ tokens_i ) [seed = NULL if public, tenant_id if private] # Only the UNMATCHED suffix is re-prefilled: unmatched_mean = (1−hit)·13.3K + hit·200-ish = 0.20·~8.25K + 0.80·200 ≈ 1.85 K tok prefill_token_load: λ_peak·T_in_mean = 1.33 G tok/s → λ_peak·unmatched = 185 M tok/s (86% cut) prefill_task_num_iter5 = prefill_task_num_iter4 × (185M / 1.33G) = 19 909 × 0.139 ≈ 2 767 tasks

Cache hits collapse the prefilled token count from the full prompt to just the new suffix — an 86% prefill-FLOPs cut, shrinking the prefill fleet ~7×. This is the bulk of the prefill-$ saving.

C8 · Pool admission & sizing §9b.4

p_admit = P(out of local) × P(length gate) P(out of local) = P(gap>15min) + P(reroute) # Pareto survival tail (heavy-tailed return gaps): P(gap>15min) = P(gap>2hr)·(2hr/15min)^α = 1.3% · 8^0.7 ≈ 5.6% (α=0.7) p_admit = (5.6% + 2%) × 0.9 ≈ 6.8% W_pool = p_admit × N_active × conv_size = 0.068 × 3M × 10.4 GB ≈ 2.1 PB store_tasks = ⌈ 2.1 PB / 60 TB ⌉ ≈ 36

A population statistic (not a per-request predictor): what fraction of cold-evicted convs return and are worth pooling. Sizes the cross-task KV store as a write-back victim cache.

C9 · Local residency window §11.5

admission_rate = (λ_peak / prefill_task_num) × unmatched(r) × KV_per_tok = 36 req/s × 1.85K × 320KB ≈ 21 GB/s C_local = usable_HBM + usable_DRAM ≈ 450 + 800 ≈ 1.25 TB τ_resident = C_local / admission_rate = 1.25 TB / 21 GB/s ≈ 60 s f_evict = P( inter_turn_gap > τ_resident )

How long new KV stays resident before LRU pushes it out. τresident ≈ 60 s ≫ typical ~30 s return gap → most returns are local hits, so the pool only catches the tail.

C10 · Skew / whale replica cost §7.3

whale_convs = N_active × whale_share = 3M × 1% = 30 K HBM_per_whale = 256K × 320KB = 82 GB extra_HBM = whale_convs × HBM_per_whale × (N_replica − 1) ≈ 2.5 PB extra_tasks = 2.5 PB / 450 GB ≈ 5 600 extra_$ = 5 600 × $70 790 ≈ $400 M/mo

Cost of replicating heavy (“whale”) conversations for load-spreading on the CH ring — ~2% of fleet $, the price of statistical co-location.

C11 · Token-bucket rate-limit state §14.4

bucket_bytes_per_user = capacity(4)+refill(4)+tokens(4) +last_ts(8)+slack ≈ 32 B full_state = DAU × 32B = 50M·32 = 1.6 GB resident = chatting·32B = 3M·32 = 96 MB per_router = 96 MB / 3 ≈ 32 MB # abuser bound (cap=10, refill=1/s, conv≤5): steady ≤ 1 req/s, burst ≤ 10, ~$0 GPU

Per-user abuse caps cost O(1) per admit and ~$0 GPU; only the chatting working set is resident (idle buckets lazily evicted).

C12 · Fault-tolerance recovery (recompute-as-fallback) §16.4

extra_steady_state_HBM = 0% (vs +100% replication / +50% erasure 6+3) # Per-request RTO, two paths: pool-resident: pull(N) = N × KV_per_tok / BW_pull = 13K·320KB / 25 GB/s ≈ 166 ms not-resident: β_cold(N) = prefill_time(13K) = 730 ms (recompute on the rebalanced target) # Fleet-capacity RTO (autoscale a replacement task): fleet_RTO ≈ 10 min = Borg-schedule (3–7 min) + weight-load (60–180 s) + warmup (~100 ms)

The pool turns a lost task’s recovery from a 730 ms recompute into a 166 ms pull, at 0% steady-state storage overhead — failures are rare (< 1/task/week) so paying per-event beats paying for replication always.
