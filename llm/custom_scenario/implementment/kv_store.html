Implementing the Pooled KV Store

PoolConvIndex, the direct-CH-probe lookup path, and the optional membership summary (§S1 sidecar) — the impl-detail page for the kv-store-job introduced at scenario Iter 3b. Tracks (a) every data structure the store holds; (b) the main read path (a direct CH(conv_id, pool_ring) probe on local miss) and the optional §S1 membership summary that pre-checks it under burst load (built on the store tasks, fanned out to the prefill pool); (c) how a warm prefill pulls-or-recomputes.

Sibling impl pages: scheduler.html (the write-back / pull caller) · router.html (the gossip relay) · Cites K2 / K20 · ~25 min

What this page is (and isn’t)

TL;DR. The scenario page justifies why the pooled store exists (bottleneck B12 — the prefill/decode (P/D) split strands the assistant delta on the decode task) and which design choices win (write-back victim cache, conv_id-keyed, Cuckoo over Bloom). scheduler.html implements the caller side (the decode task’s write-back, the prefill task’s pull). This page implements the store interior: the PoolConvIndex map, the Cuckoo filter’s construction and fan-out, the tier layout, and the exact lookup path a warm prefill walks.

  • Is: the Pythonic typed data structures the kv-store-job holds; how PoolConvIndex is written and read; the membership-filter build + the four-hop gossip chain (store → broker → Router → prefill); the lookup decision flow; the message catalog (KVWriteback / KVPull / PoolFilterUpdate); the Borg-style deployment topology; failure modes.
  • Is not: the design-choice debate behind write-back-vs-write-through / conv_id-vs-content-addressing / Cuckoo-vs-Bloom (those live in scenario §9b.3 DC1–DC4); the decode-side admission gate’s P(continue) model and the prefill engine loop (those live in scheduler.html §10b.3); the Router’s ring + wait-time state (router.html); the workload numbers λ / Nactive (scenario §1.3).

1. Where the store sits — the shared cross-task remote tier (L3 front / L5 back)

Every serving task has a local ladder — HBM (High-Bandwidth Memory) → host DRAM — introduced at scenario Iter 1a; the local NVMe tier it once had is retired at Iter 3b (DC7), so the local stack is HBM → DRAM only. That local ladder dies with the task and cannot be reached by a different task. The kv-store-job is the shared tier any serving task can reach over RDMA (Remote Direct Memory Access): on the scenario’s 2-axis tier grid its RDMA DRAM front ranks L3-remote (above where local NVMe used to sit) and its NVMe back ranks L5-remote — it outranks a local cold tier precisely because it survives a reroute.

The shape to hold in your head

  • It is a job, not a tier on a task. The kv-store-job is its own Borg job of kv_store_job_task_num ≈ 36 tasks [D — max(storage ~36, QPS), scenario §9b.4 post-DC7 flow sizing; was 16 pre-DC7], each a no-GPU NVMe-dense host. “Store task” below always means one of these.
  • It is sharded CH-on-conv_id. The conversation-id space is partitioned across the store tasks by the same consistent-hashing (CH) ring discipline the serving fleet uses. Each store task owns a slice of conv_ids and holds the authoritative index + KV blocks + membership filter for that slice only.
  • It is a write-back victim cache, not write-through. A conv is admitted only when it is evicted from all local tiers and clears a predictor-free length gate (cumulative ctx > ~1.9 K; the P(continue) EV refinement is the §S2 sidecar) — so the store holds a small admitted tail (the flow-derived padmit ≈ 7% of active convs post-DC7; the old posited fpool ≈ 3% fraction is retired, D5), not every turn. Recompute from token IDs (Recovery Point Objective, RPO = 0) is always the correctness floor on any miss.
Three CH owners, three rings — the “owning store task” is NOT the decode task. A single conv_id has three independent owners, each computed by consistent hashing but on a different ring. Each “ring” is one CH ring per job (not per task): its members are that job’s tasks, and an individual task owns only the arc of conv_ids that hash to it — not a whole ring. All three are keyed on the same conv_id but have different membership (thousands of decode tasks vs ~36 store tasks), so one conv_id lands on a different task in each. The ring map itself is maintained by the Router and gossiped via the broker; whoever does lookups carries a read-only copy. Don’t conflate them:
  • decode-task owner = CH(conv_id, decode_ring) — a serving-decode-job task (GPU; thousands of tasks). Holds the live conversation in its local HBM/DRAM/SSD tiers and sends the write-back when the conv evicts.
  • prefill-task owner = CH(conv_id, prefill_ring) — a serving-prefill-job task (GPU). Serves the next turn and issues the pull / recompute.
  • store-task owner = CH(conv_id, pool_ring) — a kv-store-job task (no GPU; ~36 tasks post-DC7). Holds the pooled/evicted KV + its PoolConvIndex shard (+ its optional shard_filter, §S1). This is “the owning store task” everywhere on this page — reached on a local miss by a direct CH(conv_id, pool_ring) probe (the main read path; the filter is an optional pre-check).
So the decode task and the store task are different jobs on different hardware: the decode task KVWritebacks its evicted KV to the store-task owner; the store-task owner is the one that writes its index and later serves the KVPull. The three owners coincide only by accident of hashing, never by design — and the prefill task holds gossiped read-only copies of both decode_ring (for the intra-request Hop) and pool_ring (for the cross-turn pull).
Iter 5b — the user_sys overflow rides this same store (a scale event, not new infrastructure). Born at Iter 3b the pool holds one key class: cross-turn conv K,V keyed by conv_id (the B12 stranded-assistant-delta job). At Iter 5 prefix caching keeps each user’s user_sys warm on the prefill task by reuse-recency LRU + ref_count — but at consumer scale the user_sys working set (users_with_active_conv × Tuser_sys_mean × KV_per_tok = 3 M × 1 024 × 320 KB ≈ 0.96 PB, scenario §11.5) far exceeds HBM, so the slice reuse-recency cannot keep resident (long-tail users + the between-turns evicted tail under churn + BLCH spill + autoscale-fresh + brand-new users) overflows onto this already-existing store. The store changes only by widening what it admits:
  • A second key class, same map + same protocol. user_sys blobs are admitted keyed by user_sys_hash (a content hash of the per-user system prompt) alongside the conv_id-keyed conv K,V — same PoolConvIndex map, same CH-on-key sharding across the same store tasks, same KVWriteback / KVPull messages (§6). No new message types, no new infrastructure.
  • Different admission rule for the new class. conv K,V is admitted by the predictor-free length gate (§1); user_sys is admitted by top-N_pool LFU (a per-user prompt is worth keeping iff it recurs) with replication_factor replication for the heaviest entries. Storage grows by pool_storage_bytes += N_pool × Tuser_sys_mean × KV_per_tok × replication_factor — N_pool ⋙ the HBM-resident set because pool storage is NVMe-backed, not HBM-bound. N_pool (hence the exact task-count growth) is settled at S6 against per-task NVMe budget + the §1.3 consumer skew.
  • The hot-path fetch is still the assistant delta, not a 320 MB user_sys storm. On a warm-turn local miss the prefill task pulls the ~400-tok assistant delta (~128 MB, the B12 job) and re-prefills the 1 K-tok user_sys (βcold ≈ 54 ms) when cheaper than a full user_sys pull (scenario §11.5, Change 6). The pool’s user_sys entries serve the structural-miss / cross-session-cold case (a returning heavy user whose user_sys fell out everywhere), so they are a correctness/cost backstop, not the common pull.
Everything else on this page — the PoolConvIndex read path (§3), the gossip chain (§4), the optional §S1 summary (§5), the deployment shape (§7) — is unchanged; user_sys_hash is just another key in the same namespace.
Iter 7–10 — what the later iterations add to the store (mostly nothing; Iter 10 promotes it from cost lever to availability fast-path). The store is a topic reference maintained across iterations; here is the exhaustive delta for the four late iterations.
  • Iter 7 (speculative decoding) — no store change. The draft→verify step is decode-pool-internal (scheduler §11d); it produces the same conv K,V (the decoded output is provably identical to vanilla sampling), so write-back / pull are byte-identical.
  • Iter 8 (abuse prevention) — no store change. The per-user token-bucket gate is Router-side (router §8e); it only thins the admitted Route stream, which marginally lowers the store’s write-back ingest. No new key class, no protocol change.
  • Iter 9 (context parallelism) — a long-context conv writes back as per-rank slices. A 1 M-context CP request (scenario §15) is 320 GB of KV split across N_CP = 8 ranks (~40 GB/rank). On eviction each rank writes back its slice keyed by (conv_id, rank) — a sub-keying within the existing conv_id class, not a new namespace or message type. The predictor-free length gate (> ~1.9 K tokens, §1) admits these trivially (1 M ⋙ 1.9 K). Storage impact is bounded: the CP population is ~0.1% of convs, so even at 320 GB each the aggregate rides the same N_pool sizing settled at S6 (the NVMe-back tier absorbs the bulk; only the hot front competes for RDMA-DRAM). Per-rank slices let a single dead CP-rank pull only its ~40 GB slice on recovery, not the whole 320 GB.
  • Iter 10 (fault tolerance) — the store becomes the per-serving-task recovery fast path. This is the load-bearing change. When a serving task dies (scenario §16), the CH ring migrates its conv_ids to a neighbour, and a neighbour whose conv is pool-resident pulls the prior KV (pull(13 K) ≈ 166 ms, linear I/O) instead of recomputing (βcold(13 K) ≈ 730 ms). So the store, built at Iter 3b purely to offload cross-turn recompute cost, is now also the availability fast path — the KVPull read path of §3 / §5 is reused verbatim by the recovery handler (scheduler §11g / router §8g). Implication for the store’s OWN durability: the store’s per-shard replication (the replication_factor already carried for heavy user_sys entries, Iter 5b) is what keeps that fast path available across a store-task death — distinct from the per-serving-task FT this iteration adds. A store-task loss is not a correctness event: the affected convs simply lose the pull fast path and fall back to βcold recompute (RPO = 0, the floor that has held since Iter 3b). The store stays a cache, never a source of truth; whether conv K,V (not just user_sys) gets a replica is an S6 lever driven by the measured failure-tail RTO.
Net: no new data structure, no new message type on the store for Iter 7–10; Iter 9 adds a (conv_id, rank) sub-key, and Iter 10 re-uses the existing pull path as the availability fast path and raises the store’s own replication to a first-class S6 sizing question.

2.4  Every bound states its cliff

  • CUCKOO_FINGERPRINT_BITS = 12 (false-positive rate, FPR). FPR ≈ 2b / 2f = 8 / 4096 ≈ 0.2% at b = 4. Cliff below: shrink f and FPR climbs → more wasted round-trips to confirm-absent (each a pull then recompute). Cliff above: grow f and the per-key cost rises above ~1.6 B, bloating the replica on every prefill task for diminishing FPR gain. 12 bits sits at the knee.
  • Cuckoo load factor α (insert-failure cliff). A lookup touches 2 buckets at any α, but an insert kicks occupants along an eviction chain, and at b = 4 the practical ceiling is α ≈ 95% — beyond it, chains lengthen and inserts fail. Cliff above ~95%: provision table capacity with headroom (operate at α ≈ ⅓) so the churning victim-cache filter never approaches it; an insert that fails anyway degrades to a false-negative → recompute (RPO=0). Full treatment in §5.6 — the 1.6 B/key figure below is the dense-pack bound (≈95%), not the operating point.
  • T_GOSSIP_MS = 100 (filter staleness). The replica can lag the truth by one cadence. Cliff above (slower gossip): more false-positives (just-evicted convs still “present”) and false-negatives (just-admitted not yet visible) — both fall to recompute, so it costs latency, never correctness. Cliff below (faster gossip): the broker NIC saturates on fan-out. 100 ms matches the existing wait-time tick, so it rides the same pipeline for free.
  • POOL_ADMIT_CTX_TOKENS = 1_870 (write-back admission floor). Below it, recompute is cheaper than a pull round-trip, so admitting wastes write bandwidth. Above it, the quadratic re-prefill dominates and the pull wins decisively. Derivation: the §6.5 SSD-vs-cold crossover, reused for the store.

3. Q1 — how PoolConvIndex is built and used

TL;DR. PoolConvIndex is a map (conv_id → {owning_pool_task, block_ids[], tier}) that is sharded CH-on-conv_id across the store tasks — each store task holds the authoritative shard for its own slice. It is written synchronously by the owning store task on three triggers (admit / evict / rebalance) and read by a warm prefill task in a single RDMA — never a broadcast — because the owner is computed from the key, not looked up.

3.1  The build/use lifecycle (diagram first)

  • ── ①②③ Writers → index (synchronous): all three run on the owning store task and mutate pool_index + shard_filter together — admit inserts, evict deletes, rebalance re-keys. Synchronous so the index is never wrong about its own shard.
  • ◆ ⑤ Internal resolve: index_get(conv_id) turns a present conv_id into exact block_ids[] + tier — an in-process hashmap hit on the one owner, no second hop.
  • ┄┄ ④⑥⑦ Reader (RDMA): the prefill task computes the owner (④, zero messages), sends one KVPull (⑥), and stages the returned blocks (⑦). One known task, one round-trip.
  • ┄┄ R Recompute floor: a KVPullMiss (lost the write-back race, or the conv was evicted) falls to βcold re-prefill — RPO=0, always correct, just slower.

3.2  How it is BUILT (step by step)

The index is not a separate service and not centralized. Each store task is the authority for its CH slice; the map is built incrementally by three local, synchronous mutations:

  1. Admit (insert). A decode task’s KVWriteback(conv_id, block_ids[], tier_hint) lands on the store task that owns conv_id (the decode task reached it by the same CH(conv_id)). The store writes the KV bytes into block_pool, then pool_index[conv_id] = PoolConvEntry(…) — and in the same critical section shard_filter.insert(fp(conv_id)). Index and filter move together so the shard can never say “present” without a resolvable entry behind it.
  2. Evict (delete). When the store task’s own DRAM-front watermark trips (or an invalidate arrives — e.g. a user_sys version change), it demotes blocks FRONT→BACK or drops the conv entirely. A full drop does pool_index.pop(conv_id) + shard_filter.delete(conv_id). This is the line that forces Cuckoo over Bloom — a victim cache deletes on every admit’s matching future evict (§5).
  3. Rebalance (re-key). On a CH ring change (autoscale, a store task added, a store task lost), the slice boundary moves. The losing owner streams the affected (conv_id, blocks) to the new owner, which index_puts them into its shard; the old owner index_dels. owning_pool_task_id on the entry is updated so a stale reader that lands on the old owner gets a clean “not mine” rather than wrong blocks.

3.3  How it is USED (step by step)

The whole point of CH-on-conv_id is that locating a conv costs zero messages — the reader does arithmetic, not a lookup-service round-trip:

  1. Compute the owner (no network). A warm turn arrives on a cold/rebalanced prefill task Pβ that lacks the prior KV. It computes owner = pool_ring.lookup(conv_id) against its local gossiped copy of the kv-store-job ring (pool_ring, ~36 tasks — a different ring from decode_ring/prefill_ring) — an O(log V) bisection, no message to anyone. The owning store task is whichever kv-store-job task this resolves to — never the decode task (§1).
  2. One RDMA to that one owner. It sends a single KVPull(conv_id, missing_block_ids) to the computed owner. Not a broadcast, not a scatter-probe — the cost does not grow with kv_store_job_task_num.
  3. Owner resolves locally. The store task does e = pool_index.get(conv_id). Present → it reads e.block_ids from block_pool (FRONT or BACK) and streams them back as KVPullResp(blocks). Absent → KVPullMiss.
  4. Stage or recompute. Pβ stages the returned blocks into HBM with fresh slot_mapping entries and runs the (now-shortened) prefill; on a miss it recomputes (βcold, RPO=0).
Worked example. Turn 5 of conv:7af3 (8 K cumulative context) re-routes to a cold prefill task after an autoscale. (1) Pβ computes pool_ring.lookup("7af3") = store-task-9 (the kv-store-job ring, not the decode ring). (2) It sends KVPull("7af3", [b12,b13,b14]) to store-task-9 only. (3) store-task-9 does pool_index["7af3"] → {[b12,b13,b14], FRONT}, reads ~2.56 GB from its DRAM front, streams it back. (4) Pβ stages it (~2.56 GB / 25 GB/s ≈ 102 ms [D]) instead of re-prefilling (βcold(8 K) ≈ 407 ms [D]) — a single round-trip to a single task, located by arithmetic.

Why a single RDMA and not a broadcast. The rejected alternative was a hot-path scatter-probe (“does anyone hold conv_id?”) costing O(kv_store_job_task_num) messages per warm turn (scenario DC4). 4. Q2 — who gossips to whom (the OPTIONAL §S1 summary topology)

This hop runs only when the optional §S1 summary is enabled (burst-insurance); the main read path is the filter-free direct CH probe (§5). TL;DR — the four-hop chain. The membership summary is produced on the store tasks (they own the truth) and consumed on the prefill tasks (they decide pull-vs-recompute). It travels: kv-store-job task → broker-job (Redis Streams, sketch fan-out) → router-job (read-replica) → serving-prefill-job task (Router relays PoolFilterUpdate). It rides the same gossip bus that already carries ring-version + per-task wait-time — no new pipeline.

4.1  The gossip topology (diagram first)

  • ┄┄ ① Store task → broker: each store task emits a PoolFilterDelta(shard_id, inserts[], deletes[], version) — the fingerprints admitted/evicted since the last tick — to the broker every Tgossip = 100 ms (or on a large delta). The store tasks are the only writers of membership truth.
  • ┄┄ ② Broker → Router: the broker fans the membership sketch out to every Router on the same read-replica stream that already delivers ring-version + wait-time (scenario Iter 1b). Adding the filter is a new topic, not a new pipeline.
  • ┄┄ ③ Router → prefill task: each Router relays PoolFilterUpdate(blob) to the prefill tasks on the channel it already uses to send Route decisions; the prefill task merges the delta into its union pool_filter and bumps pool_filter_version.

4.2  The chain, hop by hop

  1. Produce (store task). The authoritative membership of a shard is the keyset of its pool_index. Every index_put/index_del mirrors into shard_filter and enqueues an insert/delete fingerprint. A timer flushes the accumulated PoolFilterDelta to the broker every 100 ms.
  2. Fan out (broker). The broker-job (3-task HA Redis Streams) is the cluster’s gossip bus — it already does “sketch fan-out” for the wait-time signal. It republishes each shard’s delta to all Router subscribers. Choosing Redis Streams over Kafka: Kafka’s per-topic overhead is too heavy for ~100 K low-rate gossip msg/s, and Redis Streams is in-memory and already in the observability stack.
  3. Relay (Router). Routers are the broker’s read-replicas. A Router does not consume the filter for routing (routing is by CH + wait-time); it merely relays PoolFilterUpdate downstream to the serving tasks — the one direction (R→S) it already owns because it sends every Route there.
  4. Apply (prefill task). The prefill task applies each shard’s insert/delete deltas into its single local pool_filter — a Cuckoo filter covering all shards’ keys (note: you can’t bit-wise-OR-merge Cuckoo tables the way you can Bloom filters, so “union” here means one filter fed the merged delta stream, not a combined table) — and records the per-shard version. A version gap (a missed delta) triggers a full-blob refresh. This filter is what §5 probes before every pull.
Wire-bandwidth sanity (why the broker NIC, not 1 Gbps). The per-shard delta blob is small, but the broker fans it out to every Router: broker_outbound = broker_inbound × router_task_num. With the filter blob (~10 KB/shard) riding the 100 ms push alongside wait-time, this is why the broker-job is specced with a 25 Gbps NIC, not 1 Gbps (scenario Iter 1b broker sizing). The full fan-out derivation lives on router.html / the broker sizing; here it just explains why the hub exists.

5. Q3 — the membership summary (OPTIONAL §S1 sidecar)

TL;DR. The summary is an approximate “is this conv_id in the pool?” pre-check. It is OPTIONAL: the main read path on a local miss is a direct CH(conv_id, pool_ring) probe to the owner (computed, one RDMA) — affordable on its own (§5.5). The summary is burst-insurance for correlated mass-miss events (rolling deploy / autoscale / fabric blip), and if added an exact conv_id set is preferred over the Cuckoo filter (no FPR, no load-factor cliff; §S1). This section builds out the Cuckoo variant for depth; a wrong answer can only ever cause a recompute, never wrong KV.

This whole section is the §S1 sidecar. It reverses the earlier “filter required / Cuckoo CHOSEN” framing (DESIGN_CHOICE_CHANGE.md §S1). Read “gates every pull” below as “pre-checks every pull when enabled”; the unconditional path is the direct CH probe. The Cuckoo-vs-Bloom analysis (§5.2, §5.6) is the which-summary-if-any sub-choice — and §S1 prefers the exact set unless its ~7× memory bites at consumer scale.

5.1  The lookup path it gates (diagram first)

  • ── ① Probe (local): the prefill task checks its union pool_filter in-process — two candidate-bucket reads, no network. This is the whole optimization.
  • ── ② False ⇒ absent: a Cuckoo “not present” is exact — the conv is definitely not in the pool, so skip it entirely.
  • ┄┄ ③⑥ Recompute leg: absent (③) or a confirmed miss (⑥) both fall to βcold re-prefill, RPO=0.
  • ── ④ True ⇒ maybe present: a Cuckoo “present” is approximate — proceed to CH-locate the owner.
  • ┄┄ ⑤ Pull (RDMA): one round-trip to the computed owner, which resolves blocks via index_get (§3) and streams them back; a hit stages to HBM.

5.2  What a Cuckoo filter is (from zero)

  • The job. A membership structure: answer “is key x in the set?” in a few bytes per key, far smaller than storing the keys. It may answer “present” for an absent key (a false positive, at rate FPR), but never “absent” for a present key — so a “not present” is exact and a “present” is “probably.”
  • How Cuckoo stores it. Instead of a Bloom filter’s k scattered bits, a Cuckoo filter stores a short fingerprint (here f = 12 bits) of each key in a cuckoo hash table. The key here is the conv_id — the filter answers exactly one question, “is this conversation’s KV in the pool?”, so it stores fp(conv_id) and a lookup is maybe_contains(conv_id) (see BUILD §5.3 and the probe in §5.5). Each key maps to two candidate buckets (hence “cuckoo” — on a collision, kick an occupant to its alternate bucket). A lookup reads only those 2 buckets — cache-friendly versus Bloom’s k random probes.
  • Why Cuckoo here and not Bloom (the one reason). A victim cache evicts constantly — every admit has a matching future evict. A plain Bloom filter cannot delete (clearing bits corrupts other keys), so its FPR would rot toward 1 (everything reads “present”) as the pool churns. Cuckoo deletes natively (remove the fingerprint from its bucket). Counting Bloom can delete too but costs ~4× the space — and this filter is replicated to every prefill task, so footprint matters. Cuckoo wins on delete-support × space. This is the only probabilistic filter in the whole design (scenario DC4).
  • The tuning knob. FPR ≈ 2b / 2f (b = bucket size). Wider fingerprint f → lower FPR → more bytes/key. At f = 12, b = 4: FPR ≈ 8/4096 ≈ 0.2% at ~1.6 B/key — the dense-pack density (≈95% load factor); production provisions headroom below that cliff (§5.6).

5.3  How it is BUILT (step by step)

  1. Per-shard, on the store task. Each store task owns one shard_filter covering exactly its conv_ids. There is no central filter — the membership truth is distributed the same way the index is.
  2. Insert on admit. Inside index_put (§3.4), the store does shard_filter.insert(fp(conv_id)) in the same critical section as the index write — the filter never lags its own shard’s truth.
  3. Delete on evict. Inside index_del, shard_filter.delete(fp(conv_id)). Native delete is the property that keeps the FPR flat as the victim cache churns.
  4. Emit a delta. A 100 ms timer flushes the accumulated inserts/deletes as a PoolFilterDelta onto the gossip bus (§4). Downstream, each prefill task applies all shards’ insert/delete deltas into its single local pool_filter (one Cuckoo filter covering every shard’s keys — fed the merged delta stream, not a bit-wise OR of tables).

5.4  Sizing the OPTIONAL summary (Cuckoo variant) — ~9 KB per shard, ~326 KB union

(These size the §S1 Cuckoo variant if it is deployed; the main read path carries no summary. An exact conv_id set is preferred — ~7× this memory, still trivial.) The two figures that look inconsistent at first glance are the per-shard filter (the gossip unit) and the union replica (what each prefill task would hold). They reconcile exactly:

per-shard keys = POOLED_CONVS / KV_STORE_JOB_TASK_NUM = 204 000 / 36 ≈ 5 667 convs/shard [D] shard_filter = 5 667 × CUCKOO_BYTES_PER_KEY = 5 667 × 1.6 B ≈ ~9 KB [D — the gossip blob, §4 hop 1] union pool_filter = POOLED_CONVS × CUCKOO_BYTES_PER_KEY = 204 000 × 1.6 B ≈ ~326 KB [D — resident on EVERY prefill task, if enabled] consumer-scale = (after user_sys overflow, a later scale event) × 1.6 B ≈ low MB [D — scenario §9b.3]
  • ~9 KB = one shard’s filter — the unit a store task gossips. (The impl plan’s “~10 KB blob” is this per-shard figure.)
  • ~326 KB = the union — 36 × ~9 KB, the replica resident on every prefill task. Trivially task-resident; even the consumer-scale union is a rounding error against an 80 GB-HBM host. (An exact set would be ~7× this — still trivial, and the §S1-preferred choice.)
  • Why a filter and not the full index. Replicating the authoritative PoolConvIndex (KBs/conv, mutating every admit/evict) fleet-wide would be ~GBs per prefill task and perpetually stale. The filter is the 1-bit-per-key summary — small enough to replicate, approximate enough to gossip lazily, and backstopped by recompute.

  • The direct probe carries the steady state (§5.5 budget: ≤ λpeak/kv_store_job_task_num ≈ 0.03–0.06% CPU/store-task). The absent case costs one wasted RDMA RTT (~µs) ahead of a hundreds-of-ms recompute — noise.
  • What the OPTIONAL §S1 summary adds, when enabled (burst-insurance). Per-request latency — a confirmed-absent skips the RTT. Pool-side QPS — the store sees only summary-positive pulls, shrinking its query load by 1 / positive_fraction during a correlated mass-miss storm. On the main line (no summary) the direct probe carries it.
  • Why approximate is safe (the asymmetry). The index must be exact (it returns the actual blocks); the filter may be stale by a gossip interval because its only outputs are “skip the pull” or “try the pull,” and the wrong choice costs at most one round-trip + a recompute. The filter is a pure cost optimizer on the miss path, not a correctness component.

5.6  Does Cuckoo hold up at this traffic? (eviction chains + the ~95% load-factor cliff)

The concern is real. A Cuckoo insert places a fingerprint in one of two candidate buckets; if both are full it kicks an occupant to its alternate bucket and repeats — an eviction chain. As the table approaches its practical ceiling (b = 4 buckets → ~95% load factor), those chains lengthen and an insert can ultimately fail (no slot after MaxKicks). So “just pack it to 1.6 B/key” — which is the ~95% density — would park the filter exactly on that cliff. Four reasons it still holds, given one sizing fix:

  1. The hot path is cliff-immune. A lookup reads exactly 2 candidate buckets, O(1), independent of load factor — it never kicks. The probe-before-pull (the only filter op at request rate, ~4 K/s) does not degrade no matter how full the table is. The cliff touches inserts only.
  2. Inserts are off the hot path. Inserts/deletes run on the gossip-apply side at the churn rate (admits + evicts), not the request rate, and not on the decode tick or the prefill critical path. An occasional long kick chain adds microseconds to a background coroutine — invisible to TTFT/TPOT.
  3. Provision headroom — don’t run at 95%. ~1.6 B/key is the dense-pack bound, not the operating point. Size the table capacity for ~⅓ load factor (≈1.5× the slots) so steady-state occupancy sits far below the cliff → inserts stay amortized-O(1), insert-failure ≈ never. The cost: the union grows ~144 KB → ~210 KB per prefill task [D — 90 K × 1.6 B / 0.66] — still a rounding error against 80 GB HBM. A bounded working set (~90 K convs, delete-on-evict) keeps occupancy from drifting upward, and a background resize (rebuild at 2× capacity) is the backstop if a shard ever spikes past headroom.
  4. Even the cliff degrades to recompute. If an insert does fail (table over capacity), that conv is simply absent from the filter → a false negative → the prefill recomputes instead of pulling — still correct (RPO=0). The worst case is “lose the pull optimization for some convs,” never wrong KV and never an outage. Detection: an insert-failure / load-factor metric per shard pages before it matters.
Honest caveat — the cliff is also an argument for not using a probabilistic filter at all. Because the working set is small, an exact hash set of conv_ids (90 K × ~12 B ≈ ~1 MB; ~32 MB at consumer scale) sidesteps the eviction-chain / load-factor cliff entirely — no FPR, no kicks, trivial delete. Cuckoo’s only edge is memory (~7× smaller at consumer scale: ~4.3 MB vs ~32 MB), which is itself noise against HBM. So if the load-factor failure mode is judged not worth the operational weight (it is the only probabilistic component in the design), the cheap escape hatch is to swap the Cuckoo filter for an exact membership set — same protocol, same gossip, no cliff. That is a DC4 reopener, tracked for review rather than changed here.
