Implementing the Cache-Aware Router

Data structures, wait-time-driven routing, and the bidirectional protocol with the per-task scheduler — the impl-detail page for the distributed-KV-cache scenario. Tracks (a) what state the Router holds; (b) what each scheduler pushes to it; (c) what it commands each scheduler to do; (d) how that state evolves iteration-by-iteration.

Sibling impl page: scheduler.html · Cites K1 / K20 / K21 · ~30 min

What this page is (and isn’t)

TL;DR. The scenario page justifies why a cache-aware Router exists, picks CH-on-conv_id over alternatives, sets the BLCH bound, and sizes the fleet. This page is the implementation: every Pythonic data structure the Router holds, every algorithm it runs per-request and per-epoch, every wire-protocol message it exchanges with the scheduler, every cadence on the time axis. If the scenario page is the what + why, this page is the how + when.

  • Is: Pythonic typed data structures with cadence ledger; per-iteration algorithm pseudo-code (10–30 lines); the bidirectional message catalog; the wire-bandwidth budget that picks Router NIC + broker NIC sizes; the physical-deployment Borg topology.
  • Is not: the design-choice debate behind CH-on-conv_id / BLCH spill / autoscale-on-wait-time (those live in scenario §7); the per-task scheduler state, engine loop, or GPU kernels (those live in scheduler.html); the workload numbers (those live in scenario §1.3).
Scope of this revision. This page is composed up through Iter 6 — the Router is introduced at Iter 1b, the ring splits into prefill_ring + decode_ring at Iter 2, Iter 3 is a Router-side no-op (PagedAttention is a scheduler-internal layout fix), Iter 4 is a near-no-op whose only Router effect is a smoother prefill wait signal → lower BLCH spill, Iter 5 adds exactly one structure (the single_flight cold-miss dedup — retention itself stays task-side, no Router pin), and Iter 6 is a Router-side no-op (continuous batching is decode-pool-internal; the decode wait scalar simply drops ~106×). Iter 7–10 are now built: Iter 7 (speculative decoding) is another Router-side no-op (the decode scalar gains an accept-rate factor); Iter 8 is the first Router-side hot-path change since Iter 5 — a per-user token-bucket + max-concurrent-conv admit gate ahead of route(); Iter 9 adds the third CH ring (cp_ring) + the admit-time est_kv > 256 GB fork to it; Iter 10 drops a dead task from the rings on a heartbeat miss and lets the rebalanced Route trigger the scheduler’s Iter 3b pull. The full set is §8d–§8g.

1. The three-layer routing stack

Every request crosses three independent layers between the client’s SSE socket and the GPU that produces its first token. Each layer has different state, runs on different hardware, and recovers from different failure modes — conflating them is the single most common bug.

LayerJobState it holdsWhat it decidesWhere it runs
L1 — L4 LB Managed NLB / GCP L4 Connection table (5-tuple ↔ backend); TLS termination state Which Router task (round-robin) receives the next TCP stream Managed cloud product; one VIP
L2 — Router task router-job CH ring (SortedDict[int, str]); per-task TaskInfo from gossip; per-tenant soft retention-priority map (Iter 5+, optional — off the main line, Change 5); per-user token bucket (Iter 8+) Which serving task a request lands on; whether to BLCH-spill; whether to admit (Iter 8 token-bucket) router_task_num CPU-only tasks behind L4 LB

This page owns L2. scheduler.html owns L3. L1 is managed and stateless; we don’t implement it — we just configure it (TLS cert + target group of router-job tasks).

2.4  Wait-time formula evolution (the routing signal)

The Router consumes the scalar approx_wait_time_ms the scheduler publishes every 100 ms; the shape of that scalar evolves as later iterations change what the scheduler does. The Router’s consumer (BLCH spill predicate) is unchanged across all stages.

Iter 1 (Iter 1a local self-monitor; Iter 1b Router consumer) approx_wait(t) = Σ over queue(t) ∪ running(t) of β(tier_residency_for(r)) where β ∈ { β_HBM ≈ 5 ms, β_DRAM(N), β_SSD(N), β_cold(N) } Iter 2 (P/D split — one signal per pool; same broker pipeline, separate fields) approx_wait_prefill(t_p) = Σ over queue(t_p) of β_prefill(r) β_prefill(r) = prefill_time(N_new(r), TP=8) N_new(r) = 200 tok on warm path (intra_session_hit_rate = 0.80) = 13 K tok on cold path (turn-1 / BLCH spill / autoscale-fresh) approx_wait_decode(t_d) = max(0, queue(t_d) − free_slots(t_d)) × avg_seq_lifetime / B avg_seq_lifetime = T_out_mean × β_per_decode_tok = 400 × 5.22 ms = 2.09 s B = 1 at Iter 2 (raised to B_decode by a later iteration, bounded by B_max,paged ≈ 106) free_slots(t_d) = active_decode_capacity(t_d) − |live_conv_set(t_d)| ∈ {0, 1} at Iter 2 Iter 3 (PagedAttention layout fix — wait-formula UNCHANGED) The scalar's shape is identical to Iter 2; what changes is the kv_cap field reported in TaskStatePush (paged-block headroom; B_max,paged ≈ 106 vs B_max,static ≈ 18 pre-PA). The Router's BLCH-spill predicate and autoscale signal are unchanged. Iter 4 (chunked/packed prefill — the prefill scalar's SHAPE changes; consumer unchanged) β_chunk = prefill_time(C) = prefill_time(4 K) ≈ 190 ms prefill_tok_throughput = F_sustained / (2·P) = 3.17·10¹⁵ / 1.4·10¹¹ ≈ 22 600 tok/s chunks_ahead(t_p) = Σ over live r of ceil(remaining_unmatched_tokens(r) / C) approx_wait_prefill(t_p) = chunks_ahead(t_p) × β_chunk → the max-β_cold HoL term is GONE; a short request's wait now counts only the fair share of CURRENT chunk work ahead, not the full length of any long prefill in flight. (decode-pool formula unchanged from Iter 2; B still = 1 until a later iteration.) Iter 5 (prefix caching — BUILT, §8b): the prefill scalar's INPUT shrinks (consumer unchanged) remaining_unmatched_tokens(r) = r.prompt_tokens − longest_local_match(r.chain_hash, hash_index) mean 13.3 K → ~1.85 K (scenario §11.6) → chunks_ahead falls → prefill wait drops further. Router adds only single_flight (cold-miss dedup); retention is task-side (no Router pin). Iter 6 (continuous batching — BUILT, §8c): decode scalar's B placeholder graduates 1 → B_decode. approx_wait_decode(t_d) = max(0, queue − free) × avg_seq_lifetime / B_decode B_decode ≈ 106 (scenario §12.3 DC2) → published decode wait drops ~106×; consumer unchanged. Iter 7 (speculative decoding — BUILT, §8d): decode scalar's denominator gains accept_rate_speedup_factor. approx_wait_decode(t_d) = max(0, queue − free) × avg_seq_lifetime / (B_decode × accept_rate_speedup_factor) accept_rate_speedup_factor ≈ 2.0 (scenario §13.3 DC1) → published decode wait drops a further ~2×; consumer unchanged. Iter 8 (abuse-prevention — BUILT, §8e): no wait-formula change; a per-user pre-admit GATE runs AHEAD of route(). Iter 9 (context parallelism — BUILT, §8f): est_kv > 256 GB forks to cp_ring (1 M-context tick ≈ 17 ms); normal scalar unchanged. Iter 10 (fault tolerance — BUILT, §8g): no scalar change; a Heartbeat miss drops the dead task from the rings (ring_version++).

scheduler.html §3 owns the full per-iteration evolution; this page repeats Iter 1–4 because those are the stages the Router consumes today. Why Iter 4 matters to the Router even though no Router code changes: the pre-Iter-4 prefill scalar spiked to tens of seconds whenever a long prefill landed (a false-hot signal that forced needless BLCH spill); the post-Iter-4 scalar tracks the fair share of total prefill work, so it is smooth — the same BLCH predicate now spills far less often (see §8).

3. What the Router does — concurrent roles at Iter 1b–10

One Router task runs two concurrent loops at Iter 1b; Iter 2 adds a third (async consumer of PrefillHandoff); Iter 3 adds none; Iter 3b adds no new loop either — it folds a filter-relay branch into the existing gossip consumer (emit PoolFilterUpdate when a scheduler’s pool_filter_version lags). Iter 5 adds no loop — the single_flight cold-miss dedup folds into the hot-path route(), and there is no pin/replicate epoch (the PinUserSys control-plane is removed, Change 5). Iter 6 + Iter 7 add none (both decode-pool-internal). Iter 8 adds the pre-admit abuse gate (admit_iter8, folded ahead of route() on the hot path). Iter 9 folds the CP fork into route() + adds the cp_groups binding map (no new loop). Iter 10 adds the ring-drop-on-heartbeat-miss handler in the existing gossip/control-plane consumer. Each is a single-threaded event loop; concurrency between them is handled by message queues, not shared mutable state.

4. Iter 1a — no Router (single-task scope)

Bottleneck addressed: n/a — Iter 1a is the single-task per-host setup (KV cache + HBM/DRAM/SSD tiering + TP=8). One task = one route; the L4 LB forwards straight through. Router-side delta: none.

  • ① Clients → L4 LB — sync HTTP/SSE; managed NLB terminates TLS.
  • ② L4 LB → serving task — sync forward; with one task in the fleet there is nothing to choose, so no Router and no routing state exist at this iteration.

The scheduler at Iter 1a already computes approx_wait_time_ms locally (scenario §6.5 introduces it as a self-monitor). It just doesn’t publish anywhere — there’s no consumer. Iter 1b introduces the consumer (this Router) and the broker that ferries the signal.

Bottleneck this opens for next iter: B4 (one task = no scale — can’t serve λ_peak = 100 K req/s) + B7 (cache-oblivious LB at scale destroys per-conv locality) + B8 (whale-conv skew on the CH ring). Iter 1b addresses all three.

5. Iter 1b — the Router lands (CH + BLCH + autoscale-on-wait-time)

Bottlenecks addressed: B4 (single → fleet) · B7 (cache-oblivious LB → CH-on-conv_id) · B8 (whale skew → BLCH spill + autoscale). New Router-side state: tasks map · normal_ring · ring_version · Θ_blch. New protocol messages: Route (R→S) · TaskStatePush (S→R) · Heartbeat (S→R). Scenario-page reference: §7.

  • ① Stored — tasks map · normal_ring · ring_version · Θ_blch; every Router task holds a byte-identical replica so the L4 LB can land a request on any of them.
  • ② Sent (R→S, sync): Route — one wire op per request; primary CH pick on conv_id, or BLCH second-choice when primary.wait_ms > Θ·median. Used for: dispatch + cache-locality preservation.
  • ③ Received (LB→R, sync): client request body carrying conv_id — the thing being routed.
  • ┄┄ ④ Received (broker→R, async gossip): TaskStatePush every 100 ms (updates tasks[*].wait_ms/kv_cap → drives BLCH + autoscale) and Heartbeat every 5 s (refreshes last_hb → ring eviction on miss). Comms channel: serving→broker→Router fan-out over Redis Streams, quorum-acked.

5.2  Algorithm — route_iter1b(req) (the hot path)

def route_iter1b(req: Request) -> RoutingDecision: # 1. Primary CH route: O(log V) bisect on the conv_id hash. h_primary = hash64(req.conv_id) # xxHash64 — fast, well-distributed; ring_position is uint64 primary_idx = normal_ring.bisect_right(h_primary) % len(normal_ring) primary_task = normal_ring.values()[primary_idx] # 2. Check BLCH bound on the primary's wait_time. primary_wait = tasks[primary_task].approx_wait_time_ms fleet_median = _global_median_wait_ms() # maintained by gossip consumer (§5.4) if primary_wait <= Θ_blch * fleet_median: # Healthy primary: take the cache-locality win. return RoutingDecision(req.request_id, primary_task, ring_version, tasks[primary_task].epoch) # 3.   • Why the spill budget, not just the spill predicate. A predicate alone is unbounded: if many primaries are simultaneously hot, every request spills to the same second-choice vnode → that vnode becomes hot → cascade. The BLCH_BUDGET_PCT ≤ 5% per-epoch cap (scenario §7.5) is the load-bearing structural guard.
  • Why fleet_median, not fleet_p99. Median is robust to a small number of overloaded tasks (median ignores them); p99 would follow the hot tasks up → Θ × p99 would never trigger. We want the threshold to resist the skew it’s trying to correct.
  • Cliff on Θ_blch. Θ < 1 → every minor wait-time fluctuation triggers a spill → cache locality collapses (every request takes the spill path). Θ → ∞ → whale tasks queue unbounded → queue’s end times exceed TTFT SLO. Θ ≈ 1.5–2.0 is the elbow (scenario §7 DC3).

6.3  Ring-membership reconciliation under split

The Iter 1b normal_ring doesn’t survive the cutover; both new rings boot empty and re-populate from heartbeats. The handover sequence:

  1. Each serving-job task receives a config flip (Borg-side) declaring its pool (prefill or decode); restarts under the new pool config.
  2. On restart it emits Heartbeat(task_id, pool, alive=true); Router’s heartbeat consumer inserts the task into prefill_ring or decode_ring per the pool field; bumps ring_version.
  3. In-flight conversations on the dying unified task fail their next-turn route (the task is no longer in any ring). Client SDK reconnects with Last-Event-ID; next request lands on a fresh prefill task via the new ring (paying β_cold(N) because the conv’s KV was tied to the now-retired unified task).

The cutover is a planned migration, not a hot-path event; the failure semantics are the standard ring-resize path. The Router emits RebalanceHint(action=drain) to the unified pool before any config flip so each task finishes its running set before reincarnating under its new pool.

  • Why the spill rate falls (the actual win). The Router’s BLCH predicate is primary_wait > Θblch × prefill_median. Pre-Iter-4, a whale landing on the CH-target task pushed that task’s scalar to tens of seconds — orders of magnitude above the median — so every request CH-targeted at that task spilled to a neighbour for the whole 65 s, breaking locality for hundreds of conversations behind one long prefill. Post-Iter-4 the same whale leaves primary_wait near the median (it contributes ≤ 1 βchunk ≈ 190 ms of interleave slip), so the predicate stays false and those requests keep their cache-affinity. The HoL false-hot trigger — the dominant Iter 1b–3 spill cause — is gone.
  • Why route_iter2 needs no edit. The function is parametric in the scalar (§6.2): it reads tasks[primary_task].approx_wait_time_ms, compares to a multiple of the per-pool median, and walks the ring on overflow. Whether the scalar is a βcold sum or a chunks_ahead product is opaque to the predicate — ms are ms. Θblch = 1.5 and ε = 0.25 are unchanged; the same threshold simply trips less.
  • Why it is its own iteration anyway. Mirroring Iter 3: the change is real and load-bearing (it retires B6, the long-prefill tail), but it lands on the scheduler. Surfacing it as a distinct Router iteration keeps the “one bottleneck per iteration” spine honest and documents why the Router’s tail-latency and spill telemetry improve at this deploy without any Router code shipping.

8f.3  Bottleneck this opens for next iter

Every functional and latency NFR is now met — the last gap is availability. Max-context = 1 M is served (closing §1.2’s last open NFR), but the steady-state design still drops in-flight requests when a serving task dies, and CP has just added a new failure mode (a CP-rank dying mid-1 M-request). 8g. Iter 10 — Serving-task fault tolerance (Router-side: ring drop on heartbeat miss)

Bottleneck addressed: B11 — a serving-task death mid-flight loses its local KV, and the design as built drops those in-flight requests. Router-side delta: on a Heartbeat miss > 5 s, drop the dead task from prefill_ring / decode_ring / cp_ring (if a member), bump ring_version, and broadcast RebalanceHint(drain) to neighbours; CH’s K/n migration cap bounds the churn. The next per-request Route on the rebalanced ring then lands the conv on a neighbour, which rebuilds it by an Iter 3b pull(N) (if pool-resident) else a βcold(N) recompute (RPO = 0). The Router does not terminate SSE — the gateway does, and the client SDK reopens with Last-Event-ID. Scenario-page reference: §16.

  • ① Heartbeat miss: a serving task stops emitting Heartbeat for > 5 s; its local KV is gone with the process, but RPO = 0 (KV is reproducible from client-held prompt_tokens).
  • ② Ring drop (Router): drop the dead task from prefill_ring / decode_ring / cp_ring, bump ring_version (idempotent — a duplicate epoch broadcast is a no-op), broadcast RebalanceHint(drain); CH’s K/n cap bounds how many conv_ids migrate.
  • ③ Rebuild on the neighbour: the next Route lands the conv on the neighbour, which pulls (~166 ms, pool-resident) or recomputes (730 ms) — the scheduler’s job (scheduler.html §11g); idempotent on conv_id.
