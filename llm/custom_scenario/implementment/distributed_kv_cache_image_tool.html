Distributed KV Cache for a Multimodal, Tool-Using Claude-Code-Style Assistant

A staff+ NALSD (Non-Abstract Large System Design) walkthrough — same fleet, same SLOs (Service-Level Objectives), same 8×H100 hardware as the text page, plus image / PDF / video attachments (inherited from the image page), and one new functional change: the model can call tools, and tool results feed back into the conversation — the agentic loop.

A tool call and a tool result are, to the serving system, just more tokens in the conversation: the model decodes tool-call tokens, then the client’s tool result comes back as input tokens and is prefilled — exactly like a user turn (K2 D5). So the entire distributed-KV-cache design of both parent pages still holds — prefix caching, the conv-keyed pool, tiering, PagedAttention, context parallelism — it just now has to absorb the loop: one user request becomes many model turns interleaved with external, variable-latency (milliseconds → minutes), non-deterministic tool executions. That forces exactly four new things: (1) a KV-residency lifecycle across the pause — the slot frees the instant the turn stops, so the decision is what happens to a paused conversation’s freed KV blocks: dropped and re-warmed from the prefix cache on the client’s re-POST (the production default) or, at scale, actively offloaded to the pool and pulled back on resume (§T.3); (2) tool definitions as a new large, content-addressed, cross-user-shareable shared prefix (the first new fixed-prefix class since hidden_sys); (3) a non-determinism boundary in the prefix chain; and (4) a new abuse axis — one request’s loop can run away into unbounded compute.

1.1  Functional shape & non-goals (delta)

Added to the functional shape

  • A request may declare a toolset (the tool definitions), and the conversation may contain tool-call turns (model output) and tool-result turns (client input). A tool result may be text or an image (the latter routes through the inherited image path, §T.5).
  • Execution is client-side (§T.3): the model emits calls; the client runs them and returns results as the next message. The loop is client-driven (§T.6 (a)).

Removed from non-goals (this is the whole point)

  • Tool execution (handled by the client; not in the serving page). → the loop’s serving consequences are now in scope (execution stays client-side; KV-lifecycle, tool-def caching, structured decode, loop-abuse are in scope).

Added to non-goals (new)

  • Server-side tool execution — code-interpreter sandboxes, platform web-search, persistent per-conv containers (a separate stateful subsystem; §T.3). The fleet never runs tools.
  • Tool planning / selection policy — which tool to call, and when, is model behavior, not serving.
  • Tool result generation by the platform — no platform-run tools means no result bytes the platform must store; results arrive in the message (large results bounded by §1.2 caps).
  • Multi-agent orchestration / sub-agents — a single conversation’s loop only; agent-to-agent fan-out is a future delta.

NEW §T  How tool execution works — the agentic loop (from zero)

Why this section exists. Same contract as the image page’s §M: the reader knows LLM text + vision serving cold but nothing about agentic tool loops. So this section defines from zero, walks one concrete request turn-by-turn, and gives the loop a diagram (not a wall of prose). It is knowledge, not a design iteration — no “what breaks” / SLO table. Acronyms expanded on first use: tool call / tool use / function calling (the model emitting a structured request to invoke an external function), JSON (JavaScript Object Notation — the structured format of a tool call), FSM (Finite-State Machine — used in §T.6 / Iter T3), SSE (Server-Sent Events — the streaming channel that carries decoded tokens to the client).

T.1  The one load-bearing fact (state it first)

A tool call is decode output; a tool result is prefill input. When the model decides to use a tool it generates (decodes) a structured block of tokens — a JSON object naming a tool and its arguments — then stops with stop_reason = tool_use. The client runs the tool and sends the result back as the next message; the serving system prefills those result tokens and the model continues decoding. Both directions are just tokens through the same n_layers decoder, producing the same K,V (K2 D5 keying applies unchanged).

Consequence, stated up front (so the rest of §T pays off): “cache the KV of a tool result” is just “cache KV” — already solved by both parents. The genuinely new thing is the loop, not the tokens. A single user request is no longer one prefill + one decode; it is tool_turns prefill+decode cycles with an external pause between each, where the client runs a tool the serving system neither runs nor times.

This is K19’s reasoning loop, externalized. A reasoning model’s chain-of-thought is a long internal decode; an agentic loop is a long decode whose intermediate steps leave the fleet to run a tool and come back (K19). The serving consequence of “leaving and coming back” is the whole delta.

Read the columns. Every turn re-sends the entire growing context but prefills only the new tail (the latest tool result) as long as the prefix is still cached — this is the reason agentic loops are viable and not quadratic (§4). The “only the new tail” win is therefore conditional: if the prior blocks were evicted, that turn re-prefills the whole history (cold). Between turns 2→3 and 4→5 the conversation is paused for the tool’s wall-clock; what Iter T1 governs is keeping those blocks cheap to get back so the “only the new tail” case stays the common one.

Diagram D-T1 — the agentic loop, with the fleet boundary drawn. Execution is the client’s; the KV lifecycle across the pause is ours. The green edges are the prefix-cache continuity win (“prefill only the new tail” and the loop closing cheaply).

  • ① POST / re-POST → PREFILL (green = the continuity win) — the client sends the conversation; the fleet prefills only the new tail because hidden_sys ‖ tool_defs ‖ history is a prefix-cache hit.
  • ② PREFILL → DECODE — produce K,V, then decode tokens.
  • ③ DECODE → CLIENT — the tool-call JSON is emitted and the turn stops (stop_reason = tool_use), streamed over SSE.
  • ④ DECODE → KV-retention decision — the slot has already freed (the turn completed at tool_use); the decision is where the conv’s freed KV blocks live — kept resident in HBM (fast tool) or offloaded to the pool (slow tool) by the cache-retention policy, not an active slot held open — the Iter T1 policy.
  • ⑤ CLIENT executes — off-fleet, ms→minutes, external, non-deterministic. The fleet does not hold a socket open across it.
  • ⑥ CLIENT → RE-ADMIT — the tool-result message returns (text OR image) and the conversation is re-admitted (swap-in if parked).
  • ⑦ RE-ADMIT → PREFILL (green) — the loop closes; the new result prefills against the cached prefix, and the cycle repeats until end_turn.

T.3  Where the tool runs — the scope boundary (it sets the whole design)

  • Client-side execution (this page). The client runs the tool and POSTs the result back as the next message. The serving fleet is stateless per turn at the API level — it does not hold a socket open across the 30 s run_tests. What it does hold (or choose to drop) is the conversation’s KV cache between turns. That residency choice is the entire Iter T1.
  • What this buys us (why it’s the main line): no server-run sandbox, no egress / security / container fleet, no per-conv durable execution state — the page stays a KV-cache page. The serving system’s only new burden is “how long, and where, do I keep a paused conversation’s KV?”
  • The two non-goals this creates (§1.1): (i) server-side tool execution (code-interpreter sandboxes, web-search service, persistent containers) — a separate stateful subsystem; (ii) tool planning / selection policy — which tool to call is model behavior, not serving.
Baseline vs. this page’s target. The common production default is stateless drop-and-recompute: a stateless chat/agent API (Anthropic Messages, OpenAI chat-completions) does not actively retain a paused conversation’s KV across the tool’s wall-clock. The decode slot frees, the blocks age out under normal LRU, and the client’s re-POST next turn re-warms the prefix opportunistically (automatic prefix cache) or via an explicit prompt-cache TTL pin (Anthropic cache_control). The active parked-conv pool offload (Iter T1 — deliberately moving a slow-tool conv’s blocks to the Iter 3b tier for E[t_tool] and pulling them back on resume) is the scale-out escalation, earning its bookkeeping only once parked_concurrent is large enough that opportunistic re-warm misses too often. A smaller deployment stops at drop-and-recompute + prompt-cache pinning — the parked pool is not table stakes.

T.4  Tool definitions = a new large shared prefix (the second fixed-prefix class ever)

  • What they are. Before the user message, the request carries the tool definitions: the JSON schemas describing every available tool (name, description, parameter types). For a rich Claude-Code-style toolset this is T_tool_def ≈ 4–16 k tokens (G-FA, §1.3) — comparable to hidden_sys.
  • Why this is a caching object, step by step:
    1. The tool defs are identical across every request that uses the same toolset — they are a configuration, not user data.
    2. They are re-sent on every turn of every loop (the model needs them in context to keep calling tools) → re-prefilling them each turn is pure waste (BT2).
    3. They are content-addressable (toolset_hash = hash(schemas ‖ tool_format_version)) and, unlike user_sys, not user-private — schemas are shared infra, so they dedup cross-user freely (no privacy gate; contrast the image plan’s content-addressing, which was privacy-gated).
  • Where they sit in the cache hierarchy. Between the fleet-wide hidden_sys and the per-tenant user_sys. They behave like user_sys (per-config, medium-sized) but with image-style content-addressing (cross-user-shareable by hash) — the best of both. There are n_toolsets ≈ 10 toolsets, so this is a “few medium shared prefixes” caching problem, distinct from hidden_sys’s “one huge fleet-wide prefix” and user_sys’s “millions of tiny per-user prefixes.” → Iter T2.

T.5  Tool results — four properties that drive the delta

State these once; every tool iteration traces to one.

  1. External & variable-latency. The result arrives after a client-side execution the fleet neither runs nor times: a read_file returns in ms; a run_tests / bash build in tens of seconds to minutes. This wall-clock is the pause Iter T1 governs.
  2. Non-deterministic. Two conversations calling web_search("today's news") get different results. Note the mechanism precisely: content-hashed prefix caching (APC) shares any byte-identical span regardless of how it was produced — non-determinism doesn’t create a hard boundary, it just makes a cross-conversation hit statistically vanishing (the tails almost never collide). The actual enforced boundary is a privacy/correctness keying choice (the parent’s per-conv keying, carried here): tool-result blocks are keyed conv-private so a rare collision can’t leak across conversations. Net: a tool result’s KV is reusable within the conversation (fixed once it happened) and treated as private across conversations; cross-user reuse extends through hidden_sys ‖ tool_defs and stops at the first tool result.
  3. Potentially large. A file dump, a grep over a repo, a verbose stack trace, a web_search page → T_tool_result mean small (~500 tok) but p99 large (~50 k) (§1.3). A large result is a mid-loop prefill spike → chunked prefill (Iter 4 delta).
  4. Can be multimodal (the bridge to the image page). A browser / screenshot tool or a chart-render tool returns an image → that result routes through the image page’s encode pool and embedding cache before splicing as vision tokens. Tool results compose with the image delta — they are the most common server-observed source of fresh images.

T.6  Design choice — who owns the loop (the one with a serving consequence)

DC — Where the loop’s control + intermediate state live

Problem: a request is now multi-turn. Where does the loop’s control + intermediate state live?

Alternatives:

  • (a) Client-driven, stateless-per-turn serving — CHOSEN (main line) — the client orchestrates the loop: it receives stop_reason = tool_use, runs the tool, and re-POSTs the full conversation (incl. all prior calls/results) as the next /messages call. The serving fleet treats each turn as an independent request keyed by conv_id; all cross-turn continuity is the existing KV cache (prefix cache + conv-keyed pool). The loop “state” is the KV cache. Caveat that keeps (a) honest: the RPO=0 recompute-from-transcript story assumes the client can re-POST everything — so with a reasoning model the client must echo back the (signed) thinking blocks verbatim; if the platform redacts them, (a) no longer round-trips losslessly and the design tips toward (b) for that traffic (the reason named under (b) below).
  • (b) Server-tracked conversation state — rejected for this page — the fleet holds the conversation server-side as an addressable object (a conv_id / previous_response_id handle); the client sends only the new tool result each turn. This is a real, genuinely-attractive alternative (the OpenAI Responses-API shape), so its wins are named before rejection (the cons alone are not the case):
    • Bandwidth + latency — the client stops re-uploading the growing context every turn (at p99: history near the 1 M cap + 50 k-tok results + image parts = MB on the wire, re-parsed each turn).
    • Guaranteed (not opportunistic) KV continuity — client-driven resume is “hope the prefix cache still holds the blocks and the turn lands on the task that has them”; a TTL expiry / memory-pressure evict / autoscale-down task-drop = a full cold re-prefill of the whole history.
    • Holds state the client can’t reconstruct from tokens — the RPO=0 (Recovery Point Objective, zero data loss) recompute-from-transcript story breaks for redacted / hidden reasoning (K19): if the “thinking” span is summarized or withheld, the client cannot re-POST it to continue coherently, so the server must hold the authoritative copy.
    • Server is authoritative — smaller client tamper / desync surface; simpler client; a natural home for cumulative loop-cost metering (Iter 8).
    Why rejected here anyway: all of the above is content-ownership value, not KV value — on a KV-cache page the KV cache already provides continuity, so a server session adds a per-conv stateful subsystem (session-affinity surviving the parked period + the state-mgmt spectrum + replication + fault-tolerance) for zero continuity gain. It is also the natural home for server-side execution (the §1.1 non-goal).
  • (c) A separate “agent-orchestrator” job — a new control plane for no KV gain.

Winning constraint:

  • (a) adds zero new server-side session state and reuses the parents’ entire stickiness + caching machinery; the only server-side decision left is residency-across-pause (Iter T1).
Orthogonal axis — KV pinning ≠ server-tracked state (the load-bearing distinction). The KV-relevant win people attribute to (b) — guaranteed retention of a conv’s blocks — is a separate axis, reachable while staying fully client-driven (a). That is exactly Anthropic prompt caching: explicit cache_control markers + a TTL pin the prefix blocks server-side while the client still re-sends the content (distinct from vLLM-style APC, Automatic Prefix Caching — automatic content-hash caching). So the two axes are independent:
  • Axis A — who holds the conversation content: client re-POSTs (a) ↔ server holds it (b).
  • Axis B — is the KV retained: opportunistic LRU/TTL ↔ pinned (client-requestable via prompt-cache markers under either A or B).
  • This page’s pick: (a) on Axis A + explicit pin/retain on Axis B (Iter T1’s residency policy + the parent’s prefix-pinning) — we take the pinning benefit without the content-session.
  • Honest tension: (a) is “stateless per turn” for content, but the page still keeps lightweight server-side KV-residency metadata — LoopState (kv_residency, stop_reason, expected_resume_by; §2.1) — which informs Iter T1’s cache-tier retention policy (it is read by the offload/eviction path; the fleet does not hold an open slot or an active per-pause scheduler thread — the slot frees at tool_use). The real line drawn is server-tracked KV-residency metadata (kept) vs server-authoritative conversation content (rejected), not “stateless vs stateful.”

T.7  The serving consequences that drive the delta (the bridge to the iterations)

State these once; every iteration delta traces to one.

  1. A KV-residency lifecycle across an external pause. At stop_reason = tool_use the request completes and the decode slot frees immediately (continuous batching evicts a finished sequence anyway — nothing sits in a slot during the pause). What persists is the conversation’s prefix-cache blocks; keeping every paused conv’s blocks resident in HBM is catastrophic at scale (§4: the parked-KV working set ≫ fleet HBM). So “keep-warm vs park” is not an active per-pause slot decision — it is the cache-tier retention / KV-offload policy for paused-conv blocks (which tier holds them, and for how long). The default is the cheapest policy: drop them and re-warm from the prefix cache (or a prompt-cache TTL pin) on the client’s re-POST (§T.3); the active parked-conv offload is the scale-out escalation Iter T1 adds. → Iter T1, layered on the Iter 6 scheduler + Iter 3b offload tier (K4).
  2. A new content-addressed, cross-user shared prefix. Tool defs (§T.4) re-prefilled every turn are waste; they are shareable by hash with no privacy gate. → Iter T2 (tool-def prefix cache), reusing the Iter 5 chain + Iter 3b pool.
  3. A non-determinism boundary + the loop-reuse continuity win. The growing context re-prefilled per turn would be quadratic; only the latest result is new; cross-user reuse stops at the first tool result. → Iter T2 / Iter 5 chain delta.
  4. Structured tool-call output. Invalid JSON wastes a whole round trip. → Iter T3 (constrained decode + speculative synergy), on the Iter 7 foundation.
  5. A loop abuse axis. A runaway loop = unbounded compute per request, invisible to a per-request limiter. → Iter 8 delta (max-iterations + cumulative-cost quota).

T0.1  Architecture — the loop is the request path re-entered per turn (D-T4)

The full request path already exists (L4 → Router → encode → prefill → decode). A loop is that path re-entered per turn; D-T1 drew the cycle. D-T4 zooms into the re-admission half — what the fleet does the instant a tool result arrives, and where per-turn TTFT is re-paid.

  • ① Result arrives — the client POSTs the conversation + new tool_result (client-driven, §T.6 (a)).
  • ② Resume-affinity — CH-on-conv_id routes to the task whose KV is warm/parked (free from Iter 1b stickiness).
  • ③ Swap-in if parked — on the critical path (the BT7 cost); 0 if the conv was kept warm (Iter T1).
  • ④ Prefill new tail only — the prefix (history + tool defs) is a cache hit; only the new result is prefilled (the linear-not-quadratic win, Iter T2).
  • ⑤ Loop continues — a fresh tool_use stop re-enters the cycle; end_turn ends it.

T0.2  Why introduce this now (after Iter M1)

  • Enabling prior: the full request path (L4 → Router → encode → prefill → decode) exists. A loop is that path re-entered per turn.
  • Constraint that flipped: the parent assumed one request = one prefill + one decode. With tools a request = model_turns_per_req cycles (§1.3) → the page must define what carries continuity across cycles before any optimization.
  • Why not server-tracked conversation state (the obvious “agent runtime”)? → §T.6 (b): it adds per-conv server state + affinity that must survive the parked period, for zero KV-continuity benefit — the KV cache already provides continuity, and the pinning win it is credited with is a separate axis already reachable client-driven (Anthropic prompt caching). Client-driven + stateless-per-turn reuses the parent’s stickiness wholesale.

T0.3  Design choices

DC1 — Loop ownership

Problem: a request is now multi-turn — where does the loop’s control + intermediate state live?

  • (a) client-driven, stateless-per-turn serving — CHOSEN (§T.6 (a)): the client re-POSTs the full conversation each turn; the fleet keys each turn by conv_id; all continuity is the existing KV cache.
  • (b) server-tracked conversation state — per-conv server state (the §1.1 non-goal’s home); its pinning benefit is a separate axis we take without it (§T.6 (b)).
  • (c) a separate “agent-orchestrator” job — a new control plane for no KV gain.

Winning constraint:

  • (a) adds zero new server-side session state and reuses the parent’s entire stickiness + caching machinery — the loop “state” is the KV cache. The only server-side decision left is residency-across-pause (Iter T1).

DC2 — Continuity mechanism

Problem: turn k+1 needs turn k’s context cheaply.

  • (a) the existing conv-keyed KV cache — CHOSEN (prefix cache + Iter 3b pool already make turn k+1 a cache hit on turn k’s context).
  • (b) a side “conversation store” of past turns re-fed each request — duplicates the KV cache; the client already holds the transcript.

Winning constraint:

  • The KV cache already is a content-addressed store of past turns — a second store is pure duplication.

DC3 — Re-admission contract

Problem: the per-turn TTFT SLO must include resumes, or a parked-then-resumed turn silently blows it.

  • Define t_readmit as an owned SLO — CHOSEN, folded under per-turn TTFT (§S3); cold/deep-parked resumes degrade gracefully (the lever Iter T1 tunes).

Winning constraint:

  • Naming re-admission as an SLO is what forces Iter T1 to exist; leaving it implicit hides the BT7 tail.

T2.1  Architecture — the prefix chain folds in tool spans (D-T3)

Iter 5 built the content-hash block chain + APC (Automatic Prefix Caching); Iter 3b built the pool. T2 reuses both: tool defs are just another shared prefix, the tool-result prefill is just another prefill. The new structure is two boundaries in the chain.

  •   Per-tenant — user_sys + history are keyed by tenant_id (the parent’s keying, carried here).
  •   Conv-private boundary — from the first tool result on, blocks carry deterministic=false + tenant_id in the hash → reusable within the conversation, private across conversations.
  •   Deterministic opt-in — provably-pure results stay cross-conv-shareable.

T2.2  Why introduce this now (after Iter 5)

  • Enabling priors: Iter 5 built the content-hash block chain + APC; Iter 3b built the pool. T2 reuses both — tool defs are just another shared prefix; a tool-result prefill is just another prefill.
  • Constraint that flipped: tool defs (~8 k tok) are re-sent every turn of every request (BT2); the growing context is re-sent every turn (BT3). Without prefix reuse the loop is quadratic (§S3) — agentic serving is infeasible. This iteration is what makes the loop cheap.

DC1 — Cache the tool-def prefix

Winning constraint — the headline three-way contrast on content-addressing:

  • Tool defs accept it with NO privacy gate — schemas are shared infra, not user data → cross-user dedup is free and uncontested. The strongest case of the three. (n_toolsets ≈ 10 medium prefixes cover the fleet → trivially resident, like hidden_sys.)
Altitude check (don’t over-sell a commodity as exotic). On the privacy axis tool defs are the strongest dedup case; on the altitude axis they are the mainstream baseline — cross-request prefix caching of system prompts + tool definitions ships today in vLLM / SGLang APC (automatic, content-hashed) and Anthropic prompt caching (explicit cache_control on the tools / system blocks). The three cases rank oppositely on the two axes: tool-def sharing sounds most novel but is the most commodity; the image embedding cache is becoming mainstream; the image cross-user dedup behind the privacy gate is the one genuinely frontier piece.

DC2 — The non-determinism boundary (BT4)

Problem: two conversations calling web_search("today's news") get different results — where does cross-conv reuse stop?

  • Cross-user-shareable span = hidden_sys ‖ tool_defs; after the first tool result the chain is keyed conv-private — CHOSEN.
  • Treat non-determinism as the cache mechanism — wrong model (see below).

Winning constraint — stated honestly:

  • Content-hashed APC would already share any byte-identical span — non-determinism does not create the boundary, it just makes a cross-conv hit statistically vanishing (the tails almost never collide).
  • The enforced boundary is a privacy/correctness keying choice: the deterministic=false block flag + tenant_id in the hash enforce conv-privacy so a rare collision can’t leak across conversations.
  • The reuse axis is nested: tool-defs (cross-user) ⊃ deterministic-result tails (cross-conv, opt-in) ⊃ everything (intra-conv, the loop-reuse win).

DC3 — Privacy probe (K2 D52), much weaker than the image case

Problem: cross-user tool-def dedup could leak which toolset another user has via a timing hit.

  • Dedup by default — CHOSEN: toolsets are product configurations, not secrets (contrast the image plan, which defaults to isolation for private uploads).
  • Per-tenant keying remains a lever for a tenant with a custom private toolset (rare).

Winning constraint:

  • The exposure (which shared toolset a request used) is low-sensitivity, so the cross-user dedup win dominates the probe risk.

DC4 — Tool-result prefill · DC5 — Storage home

  • DC4: large results (T_tool_result_p99 ≈ 50 k) use chunked prefill (Iter 4) — CHOSEN, no new mechanism; an image result defers to the encode pool (image plan) first.
  • DC5: co-tenant the Iter 3b kv-store-job — CHOSEN (tool-def prefix = a third key space alongside conv-KV and the image embedding cache; same Mooncake-style backend) vs a new store.

NEW Iter T1 — KV-residency lifecycle across the tool pause (the flagship)

Bottleneck BT1 + BT7 · placed after Iter 6 · no new job (scheduler retention policy + Iter 3b pool) · cites K4 · diagram D-T2
Framing (load-bearing — match production, don’t overstate). At stop_reason = tool_use the turn completes: the decode slot frees immediately (continuous batching evicts any finished sequence) — the fleet does not hold a slot, a socket, or a scheduler thread open across the pause. What remains is the conversation’s prefix-cache blocks. So Iter T1 is not an active per-pause “park decision” — it is the cache-tier retention / KV-offload policy for those freed blocks: which tier holds them, and for how long, so the next turn is cheap. In production this is largely emergent from generic LRU + KV-offload tiering (vLLM CPU-offload / LMCache / Mooncake / Dynamo) + optional TTL pinning (Anthropic prompt caching). The pause-class prediction below (DC3) is a proposed enhancement beyond today’s commodity baseline (which just ages blocks by LRU), named as such.

T1.1  Architecture — the keep-in-HBM vs offload state machine (D-T2)

  • ① Turn stops — tool_use; the slot frees immediately, the conv’s blocks remain.
  • ② Classify the pause — per-tool-name prior predicts t_tool (DC3); plain LRU is the floor if no prior.
  • ③ Keep warm — t_tool < t_park_lo: fast tools’ blocks stay in HBM.
  • ④ Band — between the watermarks, stay warm (anti-thrash), biased thick toward keep-warm.
  • ⑤ Offload (swap-down) — t_tool > t_park_hi: slow tools’ blocks go to the Iter 3b pool; off the critical path.
  • ⑥ Warm resume — swap_in = 0; rejoin a batch instantly.
  • ⑦ Parked resume (swap-up) — restore from pool; on the TTFT critical path (the BT7 cost).
  • ⑧ Re-admit — the conv re-enters decode as a fresh request (Iter 6 hook).

T1.2  Why introduce this now (after Iter 6)

  • Enabling priors: Iter 6 gave the continuous-batching decode scheduler (slots admit/evict per step); Iter 3b gave the pool to park into; Iter T0 defined the stop_reason signal. T1 ties them together.
  • Constraint that flipped (the §S3 wall): the parked-KV working set ⋙ fleet HBM. Keeping every paused conv’s blocks resident in HBM is impossible by a wide margin (number below). The floor (production default) is drop-and-recompute: let LRU evict the paused blocks and re-warm the prefix on the client’s re-POST (opportunistic APC, or a prompt-cache TTL pin). The policy spectrum T1 adds on top is: deliberate offload-to-pool (mandatory only once opportunistic re-warm misses too often — the scale-out escalation) and keep-in-HBM as a fast-tool-only optimization.
  • Why not keep every paused conv in HBM? It pins HBM for an external wall-clock the fleet doesn’t control; one 60 s bash build’s conv holds ~4 GB unread — at scale the fleet OOMs its KV region on paused convs while active decode starves.
  • Why not offload every paused conv? A 20 ms read_file would pay a pool round-trip (α + β·KV_bytes down and up) for a pause shorter than the transfer — pure overhead, and the resume lands on per-turn TTFT (BT7). The right answer is a retention threshold.

T1.3  Design choices

DC1 — Retention policy for the freed blocks across the pause

Problem: a paused conv’s blocks sit unread for the tool’s wall-clock — keep them where?

  • (a) keep-in-HBM-always — HBM OOM at scale.
  • (b) offload-always — round-trip tax on fast tools + resume-TTFT hit.
  • (c) threshold on expected pause, keyed by stop_reason + tool class — CHOSEN.

Winning constraint:

  • The parked population forces a threshold, and the threshold is derivable (DC2). Baseline note: the commodity floor is plain LRU/TTL on the offload tier (no prediction); (c) adds a pause-class prior on top.

DC2 — The threshold t_park* + watermark pair

Crossover (where to put it):

  • Offload when t_tool > t_park*, where t_park* solves hbm_value(t_tool) = 2·(α + β·KV_bytes) — the value of the HBM capacity reclaimed over the pause (freed for active work; the slot already freed at tool_use) equals the round-trip offload+restore cost.
  • Modeled as α + β·n (latency–bandwidth), not n/BW: α is hypersensitive here because the two sides near-cancel at the crossover.

Why a pair, not a single threshold (anti-thrash):

  • A single t_park* would thrash a conv whose expected pause sits right at it (park → resume immediately → swap back). Use t_park_lo (keep-warm below) and t_park_hi (park above), with the band between kept warm.

Asymmetry of the band (justified, not symmetric):

  • Restore (swap-up) is on the critical path (lands on resume TTFT, BT7); offload (swap-down) is off-path (background). So bias the band toward keep-warm — set the thresholds higher and make the keep-warm side thicker: a wrong “park” costs a user-visible resume stall, a wrong “keep-warm” only costs reclaimable HBM. (Same shape as the parent’s HBM/DRAM watermark asymmetry: symmetric pair, asymmetric thickness — here set by resume-urgency.)

DC3 — Predicting the pause (which class is a conv in)

  • (a) per-tool-name priors — CHOSEN: the model names the tool in the call → look up the t_tool class from telemetry (read_file→fast, run_tests→slow).
  • (b) fixed keep-warm window then park — fallback when no per-tool prior exists.
  • (c) a learned predictor — out of scope (the P(continue) sidecar analogue; named, not shipped).

Winning constraint:

  • The tool name is a free, high-signal label already in the decode output — a per-tool prior is the cheapest accurate predictor.

DC4 — Where the parked KV lands

  • The Iter 3b pool (RDMA-front DRAM) — CHOSEN, not local NVMe (retired at Iter 3b).

Winning constraint:

  • Resume pull is the §8b ρ_fabric-derated path; the N_pull* crossover decides, on a cold deep-parked resume, pull-from-pool vs re-prefill-from-transcript (Iter 10).

T1.4  Wait-time formula delta

add a RESUME term to the decode-pool wait-time (a fourth tier-residency case, PARKED, on top of Iter 1a's HBM/DRAM/pool): β_resume = swap_in_time(KV_bytes) = α + β·KV_bytes if parked ≈ 0 if kept warm folded into the existing β(r) machinery (Iter 1a).

T1.6  What breaks

  • Bimodal tool (usually 20 ms, occasionally 60 s) → mis-classed warm → HBM held too long → escalate to park if a warm conv exceeds t_park_hi (re-check, don’t trust the prior forever).
  • Parked pool fills under a tool-latency spike (CI outage → every run_tests stalls) → parked class overflows → spill to NVMe-back of the pool / shed with 503 on new loops (protect in-flight).
  • Resume storm when a slow external service recovers (thundering herd of returning results) → rate-limit re-admission + the ρ_fabric-derated pull means size headroom against a correlated-resume burst.

NEW Iter T3 — Structured / constrained decoding of tool calls (+ speculative synergy)

Bottleneck BT5 · placed after Iter 7 · no new job (fused logit mask on the decode task) · cites K13 · GAP: no K-page owns constrained decoding
New-K-page gap (rule-of-three). Constrained / structured decoding (grammar / FSM-guided JSON) has no knowledge page. It is derived locally here; if a third scenario page needs it, lift this into a new K-page (candidate K23 structured-decoding-and-tool-use). FSM = Finite-State Machine; CFG = Context-Free Grammar.

T3.1  Architecture — an FSM logit-mask fused into the decode step

No new job and no new diagram: the mechanism is a logit mask fused into the sampling step on the existing decode task (annotate the decode box of D-T1 with the FSM mask). The active toolset’s JSON schema is compiled to an FSM once per toolset_hash (cached like the tool-def prefix, Iter T2); at each decode step the FSM masks the next-token logits to only schema-valid tokens, guaranteeing a parseable call by construction.

T3.2  Why introduce this now (after Iter 7)

  • Enabling prior: Iter 7 built the speculative draft/verify path; tool-call JSON spans are the most predictable tokens on the page (§S3) → the draft nails them → high accept.
  • Constraint that flipped: a tool call must be schema-valid JSON the client can parse and execute. Free decode can emit invalid JSON / a non-existent tool / wrong-typed args → the client rejects it → a wasted full round trip (BT5). At f_tool = 0.4 this is a real tail.

T3.3  Design choices

DC1 — Validity mechanism

Problem: free decode can emit a call the client can’t parse — a wasted turn.

  • (a) free decode + post-hoc validate + retry — a whole wasted turn on failure.
  • (b) constrained / FSM-guided decode — CHOSEN: compile the schema to an FSM that masks next-token logits to schema-valid tokens → a parseable call by construction.
  • (c) grammar-constrained sampling (CFG) — more general than an FSM but heavier; overkill for JSON-schema (regular-enough).

Winning constraint:

  • (b) gives validity for free with no retries; an FSM is sufficient for JSON-schema, so the CFG’s extra power isn’t worth its cost.

DC2 — Where the FSM mask runs

  • On the decode task, fused into the sampling step (a logit mask) — CHOSEN: negligible compute, no new job. The compiled FSM is cached per toolset_hash (built once, like the tool-def prefix).

Winning constraint:

  • A logit mask is a free elementwise op on logits already in HBM; a separate validator job would add a hop for no benefit.

DC3 — Speculative synergy (K13)

  • Draft the rigid spans directly from the FSM — CHOSEN: structural JSON tokens are deterministic given the FSM state → the draft proposes {, "name":, schema keys from the FSM (no draft model needed for them) → near-100% accept exactly where tool calls live.

Winning constraint:

  • The FSM is both a validity guarantee and a free speculative source — one mechanism, two wins.

T3.4  Number chain

wasted-retry rate without constraint ≈ P(invalid JSON) × λ_peak·f_tool ≈ 0.02 × (100k · 0.40) = 0.02 × 40k/s = 800 wasted turns/s [D, P(invalid) ≈ 2% G-FA] → driven to ~0 by the FSM spec-accept on tool-call spans → near-100% on structural tokens (FSM-deterministic) → Iter 7's ~2.0× speedup factor improves on the f_tool-weighted slice (the rigid JSON is the easiest accept on the page). [D — K13]

T3.5  What breaks

  • FSM over-constrains and the model “wants” an out-of-schema token → forced into a valid-but-wrong call → keep the schema permissive (allow any string for free-text args) + validate semantics client-side.
  • FSM build cost on a huge schema → cache per toolset_hash, amortized over all requests on that toolset.
  • A tool needing free-form output (a code-blob arg) → relax the FSM to “string” for that field, not character-level constraint.

Δ S7  Takeaways (new lines, added to the parents’)

  1. A tool call is decode output and a tool result is prefill input — just more tokens (K2 D5). The KV cache of both parent pages already serves them. The delta is the loop, not the tokens: one request becomes many model turns around an external, variable-latency, non-deterministic pause.
  2. The slot frees the instant a turn stops on tool_use — the fleet holds no slot across the pause. The hardest new thing is owning a KV-residency lifecycle for the freed blocks: a cache-tier retention policy on a derived threshold with an anti-thrash watermark pair biased toward keep-in-HBM (a wrong offload stalls the user-visible resume; a wrong keep only wastes reclaimable HBM). Production gets the floor for free from LRU + KV-offload tiering + TTL pinning; the pause-class prediction is the enhancement on top.
  3. Tool definitions are a new shared prefix — the first fixed-prefix class since hidden_sys. Content-addressing was rejected for text, accepted-with-a-privacy-gate for images, and accepted freely for tool defs. The right cache key is a function of the content class’s reuse + privacy profile, not a global constant — and on the altitude axis the free tool-def case is the mainstream baseline (vLLM APC / Anthropic prompt caching), while the image cross-user dedup is the one genuinely frontier piece.
  4. Prefix caching turns the agentic loop from quadratic to linear in turns — conditional on the prior blocks still being cached when the next turn lands. Without prefix reuse agentic serving is infeasible — and keeping the linear win true is exactly what the residency policy buys.
  5. Tool results are non-deterministic, but that doesn’t create the cache boundary — content hashing would share identical spans anyway; non-determinism just makes a cross-conv hit vanishingly rare. The enforced boundary is a privacy keying choice: tool-result blocks are keyed conv-private from the first result on. Only a provably-deterministic slice is opt-in cross-conversation shareable.
  6. Client-side execution means no new server job and makes the client the durability root for in-flight loop state — a recompute-from-tokens story identical to the parent’s RPO=0. The one hole: it assumes the client can re-POST everything, so a reasoning model must echo back its (signed) thinking blocks; if those are redacted, lossless client-side recompute breaks and that traffic tips toward server-tracked state.
  7. Constrained (FSM) decoding makes tool calls valid by construction and doubles as a free speculative source on the rigid JSON spans — validity and speed from the same mechanism.
