Distributed KV Cache for a Multimodal, Tool-Using, Branching Claude-Code-Style Assistant

A staff+ NALSD (Non-Abstract Large System Design) walkthrough — same fleet, same SLOs (Service-Level Objectives), same 8×H100 hardware as the text page, plus image / PDF / video attachments (inherited from the image page), plus tool execution — the agentic loop (inherited from the tool page), and one new functional change: a conversation is no longer a line — it is a tree. The client can fork a conversation at any point and continue it multiple ways (regenerate, edit-and-resubmit, A/B compare, best-of-N).

A fork is, to the serving system, just a second conversation that happens to share a prefix with the first. Because the parents’ prefix cache is content-addressed (block_hash = hash(parent_hash ‖ block_tokens), K2 D5) and content-hashing is path-independent, two branches that share the first k tokens produce byte-identical block hashes for that prefix — so APC (Automatic Prefix Caching) already deduplicates them, with no new cache and no awareness that a “fork” happened. So the entire distributed-KV-cache design of all three parent pages still holds — it just now has to absorb the tree: one conversation is no longer a line of turns but a branching DAG where sibling continuations share an ancestor prefix. That forces exactly four new things: (1) a ref-count lifetime across the tree — the shared ancestor is pinned while ≥1 branch is live, GC’d only when the last branch is gone; (2) copy-on-write of the single diverging block — full shared blocks are immutable and shared by reference, only the partial fork-point block is copied; (3) co-location + parallel-branch batching — all siblings land on the same task and N siblings decode in one batch reading the prefix once; and (4) a new abuse axis — fan-out is combinatorial (breadthdepth), so a tree’s compute/HBM can explode without a per-request limiter ever seeing it.

1.1  Functional shape & non-goals (delta)

Added to the functional shape

  • A conversation may be forked: the client may continue an existing conversation from any prior node, producing sibling branches that share an ancestor prefix. Forks express regenerate, edit-and-resubmit, A/B compare, and best-of-N (§B.2). Multiple branches may be live simultaneously (A/B compare).
  • Forking is client-driven + implicit (§B.3 / §B.6 (a)): the client re-POSTs a sibling keyed by the same conv_id + a branch_id; the fleet shares the prefix via the existing content-hash APC.

Added to non-goals (new)

  • A server-side conversation-tree object / heavyweight fork() RPC (§B.6 (b)) — the fleet holds no authoritative tree; the client owns tree structure. (The fleet does keep lightweight KV ref-count/pin metadata — the line drawn in §B.6.)
  • Branch selection / evaluation policy — which branch to keep, how to score best-of-N, when to prune — is product/UX or model behavior, not serving.
  • Cross-conversation (cross-conv_id) merge / graph editing — branches live under one root conv_id; arbitrary DAG merges across conversations are out of scope.
An inherited non-goal that stays a non-goal. Multi-agent / sub-agent fan-out (tool page §1.1) is still out of scope — branching is one conversation’s tree of continuations, not many agents. Stated so the two fan-outs aren’t conflated: a fork shares a prefix and decodes siblings; a sub-agent is a separate conversation with its own context.

B.1  The one load-bearing fact (state it first)

A fork is two sequences that share a content-hashed prefix — and the prefix cache already shares it. When the client takes a conversation up to node k and continues it two different ways, the two resulting sequences are identical for tokens 1..k and diverge afterward. The parents’ block hash is hash(parent_block_hash ‖ this_block_tokens) (K2 D5) — a pure function of the token path so far, independent of which “branch” produced it. So the shared prefix’s blocks hash identically in both branches → the existing APC deduplicates them with zero new machinery. The KV is computed once and read by both.

Consequence, stated up front (so the rest of §B pays off): “cache the KV shared by two branches” is just “cache KV with ref_count > 1” — already solved by the parents (cross-conversation prefix sharing). The genuinely new thing is the tree, not the sharing. A conversation is no longer one growing line; it is a DAG of nodes where siblings share an ancestor. That changes four things the parents assumed about a linear conversation: who keeps the shared ancestor alive (ref-count lifetime, B1), how the one not-yet-full block at the fork point is handled (CoW, B1), where the siblings run (co-location + batching, B2), and how much compute one conversation can demand (fan-out abuse, B3).

This is K19’s Tree-of-Thoughts, externalized — and K2/K4’s parallel sampling, made user-visible. A reasoning model exploring branches internally (ToT, beam search) and a serving engine sampling best-of-N from one prompt are the same KV-sharing shape the engine already implements (vLLM CoW fork, SGLang RadixAttention). User-visible forking (regenerate, edit-resubmit, A/B compare) is that primitive promoted to a product feature across turns. The serving consequence is identical; only the trigger (a client request) and the time-scale (turns, not tokens) are new.

B.4  The cache-key consequence — the chain becomes a tree (radix), and content-hashing makes it free

  • The parents’ block-hash chain (hidden_sys → tool_defs → user_sys → history…) was a line because a linear conversation has one path. With forks the same hashing rule produces a tree: a shared prefix with multiple divergent tails — i.e. a radix tree of KV prefixes (the structure SGLang RadixAttention maintains natively; vLLM reaches the same end via hash-keyed blocks + CoW).
  • The key fact: nothing about the key changes. block_hash = hash(parent_hash ‖ tokens) is path-independent, so the shared prefix’s blocks collide (intentionally) across branches → APC dedups them. The data structure the engine keeps is a tree/radix; the cache key model is unchanged. This is why branching is “free” at the cache layer and the delta is all about lifetime, placement, and fan-out.
  • Privacy / sharing scope (contrast the tool-def and image cases). Branch sharing is intra-conversation, hence intra-tenant by construction → no privacy gate, no cross-user exposure (unlike the image cross-user dedup, which was gated; unlike tool-defs, which are cross-user-shareable-but-public). It is the safest form of sharing on the whole page: a user sharing a prefix only with their own sibling branches. (If two different users’ branches happen to share a public prefix — hidden_sys ‖ tool_defs — that reuse is already governed by the parents’ cross-user rules; the branch-specific tail is always private.)

B.5  How forks compose with the inherited features (the bridges — state once)

Bridge to tools (the subtle one)

Where a fork diverges depends on the tool determinism boundary (tool page §T.5 / Iter T2):

  • Fork after a tool result (already fixed in the parent, e.g. the failure log at n4): both branches share the result’s KV → full prefix share to the fork point. The common case (you fork because you saw the result).
  • Fork before a non-deterministic tool call (e.g. two branches each independently web_search): the client re-executes the tool in each branch → different results → the branches genuinely diverge at the result, not at the fork point. The content-hash chain handles this automatically (different result tokens → different block_hashes → no false share).

So branching and the determinism boundary compose cleanly: shared up to the first divergent token, private after — exactly the rule already stated for tools.

Bridges to images & the loop (compounding wins)

  • Images. A fork after an image attachment shares the image-embedding KV (content-addressed embedding cache, image page Iter M2) across all branches. The expensive encode is paid once regardless of fan-out → forking is especially cheap for vision-heavy conversations (the most expensive shared prefix is shared the most).
  • The loop continuity win. The tool page turned the loop from quadratic to linear via prefix reuse; branching reuses the same mechanism, so a fork after a deep loop inherits the entire cached loop context for free — fork cost = only the divergent suffix, even if the shared trunk is a 50-turn agentic history.

B.6  Design choice — who owns the tree (the one with a serving consequence)

Problem.  A conversation is now a tree. Where do the tree’s structure + shared-KV lifetime live?

Alternatives (enumerated):

  • (a) Client-driven, implicit forking — picked (main line). The client holds the tree; to fork, it re-POSTs a sibling conversation (the shared prefix + a new tail) keyed by the same conv_id (with a branch_id sub-key, §2.1). The serving fleet treats each branch like any other turn; all sharing is the existing content-hash APC, all cross-branch continuity is the existing conv-keyed pool + prefix cache. Optional tail-only fast path: a client may send parent_node_id + only the divergent tail (a bandwidth opt-in) and fall back to a full re-POST on a 409 cache miss — this stays within (a) because full-resend remains the correctness floor (§3).
  • (b) Server-tracked conversation tree + explicit fork() RPC — rejected for this page. The fleet holds the tree server-side as an addressable object; the client calls POST /conversations/{id}/fork {from_node} and sends only the divergent continuation per branch. A real, attractive alternative (it is what a stateful “responses/threads” API does), so its wins are named before rejection:
    • Bandwidth — the client stops re-uploading the shared prefix once per branch (at high fan-out + a near-1 M shared trunk + image parts, that is fan-out × MB re-uploaded and re-parsed).
    • Guaranteed (not opportunistic) prefix retention — implicit forking hopes the shared prefix survives in cache between sibling requests; a TTL expiry / pressure-evict between branch A and branch B = a cold re-prefill of the whole shared trunk for B. A server tree lets the fleet pin the fork-point deliberately (and this pin is more valuable than in the tool plan — the share factor is fan-out, not 1).
    • Cheaper CoW in place — the server can fork blocks by bumping ref_count instead of the client re-hashing + the engine re-matching the prefix.
    • Authoritative tree structure — natural home for “show me the branch tree,” branch GC policy, and fan-out metering (B3).

Why (a) wins anyway — the winning constraint:

  • All of (b)’s wins are content-ownership / pinning value, not new KV-sharing value. On a KV-cache page the content-hash chain already shares the prefix, so a server tree adds a per-conv stateful subsystem (the tree object + the §2.2 A/B/C/D state-mgmt spectrum + replication + FT) for zero sharing gain.
  • The one genuinely-KV win it is credited with — guaranteed retention — is the orthogonal pinning axis below, reachable while staying client-driven.
  • Its remaining (non-KV) bandwidth win is recovered opt-in by the §3 tail-only fast path (parent_node_id, best-effort, 409-fallback) — so even that needs no server tree.
Orthogonal axis — KV pinning ≠ server-tracked tree (the load-bearing distinction, same as the tool page’s T6). The retention win attributed to (b) is a separate axis reachable under client-driven (a): explicit prompt-cache cache_control markers pin the fork-point prefix while the client still re-sends content. Two distinct layers (don’t conflate): cache_control is an API-level caching tier (explicit breakpoints, TTL-managed, may persist to a separate KV store) — not the same thing as the engine-level automatic APC + CoW (block-internal, ref-counted, LRU). They coexist: cache_control is the client’s lever to pin a prefix the engine would otherwise let age out under LRU.
  • Axis A — who holds the tree content: client re-POSTs siblings (a) ↔ server holds the tree (b).
  • Axis B — is the shared prefix retained: opportunistic LRU/TTL ↔ pinned (client-requestable under either A or B).
  • This page’s pick: (a) on Axis A + explicit pin/retain on Axis B at the fork point (B1’s ref-count + the parent’s prefix-pinning) — i.e. take the pinning benefit (which fan-out makes more valuable) without the server tree. (b)’s only residual value beyond pinning is the bandwidth offload, which this page does not need.
Honest tension to surface. (a) is “client holds the tree,” but the fleet still keeps lightweight server-side KV metadata — branch_id + per-block ref_count + the fork-point pin (§2.1) — because B1’s lifetime management needs it. The real line drawn is server-tracked KV ref-count/pin metadata (kept) vs server-authoritative tree content (rejected), identical to the tool page’s “KV-residency metadata kept, conversation content rejected.”

B.7  The serving consequences that drive the entire delta (the bridge to the iterations)

State these once; every iteration delta below traces to one.

  1. Ref-count lifetime across the tree. The shared ancestor must be pinned while ≥1 branch is live and GC’d only when the last branch dies; an abandoned branch’s unique suffix must be reclaimed. → Iter B1, on the Iter 3 PagedAttention ref-count + Iter 5 chain + Iter 1a coldness gate.
  2. Copy-on-write of the fork-point block. Full shared blocks are immutable + shared by reference; only the partial (not-yet-full) fork-point block must be copied so two branches can append different tokens. → Iter B1.
  3. Sibling co-location + parallel-branch batching. All branches should land on the same task (else each cold-prefills the shared trunk); N siblings can decode in one batch reading the prefix once. → Iter B2, on Iter 1b routing + Iter 6 continuous batching.
  4. Fan-out abuse. A tree is combinatorial (breadthdepth); one conversation’s compute/HBM can explode, invisible to a per-request limiter. → Iter 8 delta + Iter B3 (tree-shape quota).
§B notation recap (symbols this section newly introduces; full typing in the implementation notation later).
  • branch_id — routing / ref-count sub-key under conv_id; set by client per sibling.
  • parent_node_id — the node a branch continues from; a hint for fork-point targeting.
  • ref_count — live readers of a block; now also incremented by intra-conv siblings (mechanism unchanged).
  • cow_partial — bit on the single partial fork-point block (the only block ever copied).
  • fanout — live branches per fork; shared_prefix_frac — share fraction; both declared in §1.3.
  • cache_control — API-level pin marker (Axis B), distinct from engine APC.

2.3  Routing-model delta

Parent (text + image + tool): a client-driven cycle keyed by conv_id. Forking turns the cycle into a client-driven tree fan-out — multiple sibling requests under one conv_id:

  • ➊ sibling co-location — CH-on-conv_id already lands every branch on the task holding the warm trunk (the loop’s gift, reused)
  • ➋ hot-conv overflow (BB6) — when tree_kv_footprint > the task’s KV budget, spread excess branches to a sibling task; serve the shared trunk from the pool
Production note — name the real mechanism. Real prefix-cache-aware routers (SGLang router, llm-d, AIBrix, Mooncake conductor) co-locate by longest-prefix-match on the actual tokens, not by conv_id — which is strictly stronger (it catches cross-conversation prefix sharing too, and targets the exact task holding the matching blocks). CH-on-conv_id is the coarser proxy this fleet inherits; for siblings the two coincide (all branches share the conv’s prefix), so the proxy is adequate here — but the mechanism production actually uses is prefix-affinity routing. The router also adds fork-point targeting: parent_node_id confirms the target task holds the shared-prefix blocks before dispatch.

  • ② Budget remains — admit the next branch.

NEW Iter B1 — Copy-on-write prefix sharing + ref-count lifetime across the tree (the flagship)

B1.1  Why introduce this now (after Iter 5 / T2)

  • Enabling priors: Iter 5 built the content-hash block chain + APC (the dedup that makes sibling sharing free); Iter 3 built PagedAttention’s ref-counted blocks + CoW (the fork primitive); Iter 3b built the pool to overflow into. B1 turns the chain into a tree and manages its lifetime.
  • Constraint that flipped: the parents’ ref_count only ever exceeded 1 via cross-conversation prefix matches.   • Why not copy the whole shared prefix per branch? → it throws away the entire sharing win (fan-out × trunk KV) and re-pays fan-out × trunk prefill — at shared_prefix_frac = 0.85, fan-out 8, that is ~7× wasted HBM + prefill per fork.
  • Why not refuse to evict any shared trunk ever? → a popular root would pin HBM forever; the trunk must be reclaimable once the last branch dies → ref-count, not pinning-forever.

B1.2  Architecture — the CoW + ref-count state machine (D-B2)

  • ① Fork — increment ref_count along the shared path (fork-point block + ancestors).
  • ② CoW — copy only the partial fork-point block (≤ 1 block/fork); full blocks shared read-only.
  • ③ Branch ends — decrement ref_count along its unique suffix (the abandon hint does this early).
  • ④ GC — ref_count == 0 hits the unchanged coldness gate (Iter 1a).
  • ⑤ Pin — pin_until holds the trunk across the fan-out (single TTL, biased short — the client re-POSTs on a late fork, RPO=0).
  • ⑥ Last branch dies — trunk ref_count hits 0 → reclaimed normally.

B1.3  Design choices

DC1 — Shared-block handling: share read-only + CoW only the partial block

Problem: two branches diverge at a not-yet-full block — how to let both append without corrupting each other?

  • (a) copy the prefix per branch — kills the win (fan-out × trunk KV + prefill).
  • (b) share full blocks read-only by reference (ref-counted) + copy-on-write only the partial fork-point block — CHOSEN — full content-hashed blocks are immutable, so two branches read the same physical block safely; the one not-yet-full block at the fork point is the only thing copied. It is the vLLM parallel-sampling/beam-search primitive; one block copied per fork (§4), the rest shared.

Concrete tech — vLLM (hash-block + CoW) vs SGLang RadixAttention (native radix tree):

vLLM — hash-keyed blocks + CoW

  • block_hash = hash(parent_hash ‖ tokens) over full blocks (e.g. page_size = 16).
  • Fork = bump ref_count + CoW the one partial block.
  • Flat, global, shardable key → CH-shard across the cross-task pool, gossip, dedup fleet-wide (the whole kv-store-job / PoolConvIndex design). O(1), paging-native.

SGLang RadixAttention — radix tree of prefixes

  • A radix tree shares branches natively (no separate fork call); token_to_kv_pool at page_size = 1.
  • Can split a node at the exact divergence token → the CoW copy + partial re-prefill disappears.
  • Centralized in-memory structure — awkward to distribute; needs split/merge maintenance; loses the contiguous 16-token kernel gather.

The four radix-vs-block-hash benefits, judged honestly — only #1 is a real capability gap:

  • #1 Token-granular (sub-block) sharing — the real lever is page_size, not the tree alone. Block-hash leaves a partial trailing block unshared at a non-aligned fork → the one CoW + up-to-block_size−1 re-prefilled tokens. The physical unit on HBM is the page (alloc unit and kernel-gather run); a radix tree is an index whose leaves are still whole pages, so the exact-token win is contingent on shrinking the page: at page_size = 16 the child still CoW-copies the mid-page (the tree buys nothing); at page_size = 1 (SGLang) the divergence token gets a fresh slot → no CoW, but page-table count grows ~block_size× and the kernel loses coalescing; page_size = 2–4 bounds the residual loss to page_size−1 tokens. Net: a local, small-page_size option — the distributed pool stays page_size = 16.
  • #2 Structure-aware eviction — NOT a gap; a cheap block-hash upgrade matches it. Flat ref_count is already correct: ref_count is monotone up the chain, so ref_count(block) = 0 ⇒ all descendants are 0 ⇒ no live children — the “never evict an internal node with a live child” invariant falls out of the ref_count == 0 gate alone. Even the eviction ordering is recoverable from scalars: among ref_count = 0 candidates, the faithful policy is the triple (ref_count==0 gate → last_access recency → min_position depth tiebreak), which reproduces SGLang’s “coldest leaf.” (Runnable side-by-side: prefix_cache_impl.html.)
  • #3 One longest-prefix-match traversal — near-wash, and metadata is trivial. Per-block lookup is O(1), so matching a P-token prefix is ~P/block_size probes, comparable to the radix walk. A block-hash entry is ~100 B guarding a ~5 MB block (16 tok × KV_per_tok = 320 KB) → metadata is ~0.002% of the KV (hidden_sys’s ~500 entries ≈ 50 KB vs ~2.5 GB). The uniform-fixed-block invariant (logical block == physical page) is load-bearing for the paged allocator + kernel gather + per-page eviction + chunked-prefill page-by-page commit — not worth trading for a coarse mega-entry to save 0.002%.
  • #4 Native many-way branching. Best-of-N / beam / ToT / user forks are trees; inserting a sibling = adding a child edge at the divergence node — the structure mirrors the request topology.

Winning constraint:

  • Picked vLLM-style (hash-block + CoW) to stay consistent with the parents’ content-hash APC and the shardable pool; adopt a RadixAttention-style local tree per-task where sub-block/partial sharing dominates — which branching, with its arbitrary fork points, is exactly the workload that justifies. They live at different layers: block-hash for the distributed/shardable pool, a radix tree as the local per-task prefix structure.

DC2 — Ref-count lifetime (BB1) — and why no watermark pair is needed

  • ref_count(block) = number of live branch heads whose path includes it. Fork → increment along the shared path; branch ends/abandoned → decrement along its unique suffix; GC when ref_count == 0 (the unchanged coldness gate, Iter 1a). The shared trunk survives until the last sibling dies.
  • No watermark pair needed here (contrast the tool plan’s keep-warm band): ref-count is an exact liveness signal, not a thrash-prone threshold. The only retention tuning is the fork-point pin TTL (pin_until): how long to hold a trunk after the last known branch ends, in case the client forks again — a single TTL, biased short because the client re-POSTs the trunk anyway on a late fork (RPO=0).

DC3 — Block alignment of the fork point

  • (a) fork at arbitrary token offset → always a partial block to CoW.
  • (b) prefer block-aligned fork points — CHOSEN as an optimization → the fork lands on a block boundary → both branches start fresh blocks, zero copy; fall back to (a)’s single-block CoW otherwise.

Winning constraint:

  • Cheap, and makes the common late-fork (regenerate) fully zero-copy.

B1.4  Wait-time formula delta

a branch's prefill wait = chunked_prefill(divergent_tail) + (0 if trunk warm/ref-counted else trunk_pull(α + β·KV_bytes from pool)) the shared trunk contributes 0 to the second branch onward (cache hit), folded into the parent's β(r) tier machinery as the already-resident case.

B1.5  Number chain

tree_kv_footprint = T_shared + Σ branch_tails → sublinear in fan-out at shared_prefix_frac = 0.85, fan-out 8: 0.85 + 8·0.15 = 2.05× a single path (NOT 8×) [D] CoW cost = 1 partial block / fork = 16–32 tok × KV_per_tok = 5.1–10.2 MB [D] ref_count>1 pop = n_branches_concurrent ≈ 300k blocks held alive by intra-conv sharing [D, §1.3] prefill collapse = fan-out × trunk → 1 × trunk (the headline branch win) [D]

B1.6  What breaks

  • Long-lived A/B branch keeps a huge trunk pinned (ref_count never hits 0) → correctly retained, but a zombie branch (client never closes it) leaks → the pin_until / session-idle TTL reclaims it (trunk is RPO=0 re-buildable from transcript).
  • Fork from a partially-evicted block (tail gone, head kept) → re-prefill the missing tail only (chunked, Iter 4).
  • Ref-count race under concurrent fork+abandon → the parent pool’s existing atomic ref_count ops (Cuckoo/PoolConvIndex) handle it — no new concurrency primitive.

NEW Iter B2 — Sibling co-location + parallel-branch batching (+ hot-conv overflow)

Bottleneck BB3 + BB4 + BB6 · placed after Iter 6 / T1 · no new job · cites K4 + Iter 1b (routing) + Iter 3b (pool) · diagram D-B3

B2.1  Why introduce this now (after Iter 6 / T1)

  • Enabling priors: Iter 6 gave the continuous-batching decode scheduler; Iter 1b gave CH-on-conv_id stickiness; Iter 3b gave the pool; B1 gave the ref-counted shared trunk. B2 places the branches and batches them.
  • Constraint that flipped: siblings must (a) land on the task holding the warm trunk (else BB3: each cold-prefills it) and (b) be batched to share the prefix block set (else BB4: N copies of the prefix waste HBM). And a heavily-forked conv can exceed one task (BB6).
  • Why not just let the load balancer spread branches? → spreading siblings across tasks destroys the sharing — each task cold-prefills the whole trunk → fan-out × trunk prefill, the exact waste B1 eliminated.

B2.2  Architecture — N branch slots read one shared prefix block set (D-B3)

B2.3  Design choices

DC1 — Sibling placement

  • (a) prefix-affinity routing → all siblings on one task — CHOSEN: production prefix-cache-aware routers (SGLang router, llm-d, AIBrix, Mooncake conductor) route by longest-prefix-match on the tokens; this fleet’s inherited CH-on-conv_id is the coarser proxy that coincides for siblings (they all share the conv’s prefix) → every branch lands on the one warm trunk + one physical prefix block set for free.
  • (b) spread for load balance — as the default, loses sharing.

Winning constraint:

  • Co-location is the sharing precondition; prefix-affinity is the mechanism, conv-id stickiness the adequate proxy.

DC2 — Parallel-branch batching (BB4) — capacity vs bandwidth

N live siblings admitted as N decode slots whose attention reads the one shared prefix block set (K4) — prefix KV resident 1×, read by N query streams. Two distinct cases, don’t conflate:

  • Best-of-N issued as one n=N request → the engine co-batches the N samples by construction (vLLM/SGLang parallel sampling) — guaranteed, the textbook case.
  • A/B-compare siblings arriving as separate HTTP requests → no special “branch batch” admission; the continuous batcher co-batches them only opportunistically when concurrent + co-located. Real but best-effort.
  • Capacity vs bandwidth. Storing the prefix 1× is a capacity win (lets B_decode grow). By default each branch still reads the shared KV independently → N× read bandwidth; since decode is bandwidth-bound, the read-side win needs a shared-prefix (cascade) attention kernel (FlashInfer cascade inference / flash-attn) that reads the prefix once for all N queries.

Winning constraint:

  • Decode is memory-bound → sharing the prefix block set across siblings is a large HBM-capacity win unconditionally, and a bandwidth win with the cascade kernel.

DC3 — Hot-conv overflow (BB6) — the spread-vs-share crossover

  • When one conv’s tree_kv_footprint > ρ_kv · HBM_task (one task can’t hold the whole tree), spread excess branches to a sibling task and serve the shared trunk from the Iter 3b pool (overflowed branches pull it read-only, ref_count-shared).
  • Crossover: keep co-located below the threshold (share trunk, pay hot-task risk); spread above (pay one trunk pull α + β·KV_bytes per overflow task, not fan-out × full re-prefill). Modeled α + β·n, not n/BW. No watermark pair needed — this is a capacity threshold (does the tree fit?), not a thrash-prone oscillating signal.

B2.4  Wait-time formula delta

parallel-branch batch occupies N slots but 1× prefix KV: slot-demand(forked conv) = fan-out (each branch is a slot) KV-memory-demand(forked conv) = divergent suffixes only (trunk shared 1×) → fold into Iter 6's B_decode admission: forked convs are SLOT-bound, not memory-bound (the asymmetry).

B2.5  Number chain

sibling_batch_rate = λ_peak·f_fork·(fanout_mean − 1) = 100k·0.15·1 = 15k/s [D, §1.3] overflow fires when tree_kv_footprint > ρ_kv · HBM_task (BB6 crossover, §4) parallel-branch batch: prefix resident 1× vs N× → HBM saved = (N−1)× prefix KV [D]

B2.6  What breaks

  • A single conv fans out past one task’s slots (not just KV) → spread branches across tasks (DC3) + the fan-out cap (Iter 8/B3) bounds it.
  • Co-location makes a forked conv a hot task (all its branches’ decode on one GPU) → the overflow path (DC3) sheds branch slots to siblings while keeping the trunk in the pool.
  • Overflow trunk-pull storm if many convs spread at once → the ρ_fabric-derated pull + headroom (inherited from the tool plan’s resume-storm mitigation).

Δ S6  Final design — the whole settled fleet

The final fleet is the tool page’s fleet (itself = parent text fleet + the multimodal stack + the tool deltas) with no new job — the deliberate structural contrast with the image plan (which added vision-encoder-job + blob-store-job) and the consistency with the tool plan. The existing jobs gain tree-awareness; the kv-store-job gains one residency class (not a new key space).

6.1  Final fleet architecture — the whole settled fleet in one picture (D-B6)

This is the tool page’s final fleet with every branch addition folded in — not a delta sketch but the entire fleet. The inherited boxes keep their colors (router blue, broker pink, prefill green, decode purple, kv-store orange, the amber multimodal stack, the cyan tool rows); the branch deltas are teal. The headline structural fact is again no new job — forking is the client’s (the client now holds the whole tree), drawn outside the dashed fleet boundary, and a fork closes through the existing ingress / egress. Every branch delta is a teal row inside an existing box + a residency class on key space 1 of the kv-store-job.

  • ①② Clients → L4 → router-job — HTTP opens an SSE stream; a fork is just a re-POST of a sibling branch (conv_id + branch_id) on this same ingress.
  • ── ③④⑤⑥ image path (inherited) — blob → encode → prefill, and the content-addressed embedding cache (key space 2); a fork after an image re-uses this embedding KV (encode paid once).
  • ⑦ Router → prefill — CH-on-conv_id after the admit + loop + fan-out quota gate; the same hash co-locates every sibling on the warm-trunk task for free.
  • ── ⑧ Prefill → decode — K,V handoff (320 KB/tok). A branch prefills only its divergent tail; the trunk is a cache hit.
  • ⑨ Decode → Clients (egress) — SSE tokens; the client then forks/continues and re-POSTs via ① (the loop + the tree close through the existing ingress/egress).
  • ── ⑩⑪ Serving pools ↔ key space 1 — write-back / pull conv K,V; also carries the parked-conv park/restore and the branch-shared-trunk residency.
  • ┄┄ ⑫⑬ Pools → broker → Router — async gossip of approx_wait + ring epochs; the broker now also gates drain on n_live_branches.
  • ⋯⋯ ⑭ Router → cp_ring — CP forks per root→head path (fan-out is not a CP trigger).
  • ── ⑮ Prefill ↔ key space 3 — the cross-user tool-def prefix (inherited).
  • ── ⑯ Branch-shared-trunk ↔ prefill (NEW) — on hot-conv overflow, a sibling task pulls the shared trunk read-only from the pool (one α + β·n pull, not fan-out × re-prefill).

Read the diagram. The whole fleet is still the parents’ — no new job line. Every branch delta is either a teal row inside an existing box (Router sibling-co-location + fan-out quota + per-path CP; prefill’s divergent-tail-only + CoW + radix chain; decode’s parallel-branch batch + ref-count-aware eviction + cascade read) or the branch-shared-trunk residency class on key space 1 of the kv-store-job. The one genuinely new actor is the same one the tool page drew outside the boundary — the client — which now holds the entire conversation tree and is its durability root. A fork is more traffic on the same fleet, not more fleet.

6.5  Fleet comparison — the no-new-job result

prefill_task_num now reflects shared-trunk dedup (a fork is mostly a cache hit, not a new prefill); B_decode slot demand reflects fan-out for forked convs (each branch is a slot, while KV-memory grows only by the divergent suffixes). No new job line (client-held tree) — the structural contrast with the image plan’s vision-encoder-job, and the consistency with the tool plan. The stateful part that does grow is the kv-store-job pool (branch-shared-trunk class + the fork-point pin), inheriting the parent’s B: sharded, CH-on-conv_id state-management choice.

Δ S7  Takeaways (new lines, added to the parents’)

  1. A fork is just a second conversation that shares a content-hashed prefix — and because the block hash is path-independent (K2 D5), the existing prefix cache already deduplicates siblings with no new cache. The delta is the tree (lifetime, placement, fan-out), not the sharing.
  2. The chain becomes a radix tree of KV prefixes (SGLang RadixAttention; vLLM via hash-blocks + CoW). Forking is the engine’s parallel-sampling / beam-search fork primitive promoted to a user-visible feature — full shared blocks are immutable and shared by reference; only the one partial fork-point block is copy-on-written.
  3. Branching is a new source of ref_count > 1 (intra-conversation), not a new mechanism — the shared trunk lives while ≥ 1 branch is live, GC’d at the last, by the unchanged coldness gate. Unlike the tool plan’s keep-warm band, ref-count is an exact liveness signal, so no anti-thrash watermark pair is needed.
  4. Branch sharing is intra-tenant by construction → the safest sharing class on the page: no privacy gate at all (contrast tool-defs cross-user, image cross-user gated). The cache key’s sharing scope is again a function of the content’s reuse + privacy profile.
  5. Fan-out multiplies decode, not the shared prefill — so co-locate siblings (prefix-affinity routing; CH-on-conv_id is the inherited proxy) and batch them to store the prefix once (1× HBM-capacity for N branches; the matching read-bandwidth win needs a cascade-attention kernel). The win compounds with the image embedding cache (encode paid once for all branches) and the loop (fork inherits the cached agentic trunk).
  6. Fan-out is combinatorial (breadthdepth) → a new abuse axis a per-request limiter can’t see; cap live branches + tree nodes and charge the token-bucket the aggregate-tree cost, composed with the loop and image quotas.
  7. Client-held tree means no new server job and makes the client the durability root for the whole tree (it re-POSTs each branch path) — the same recompute-from-tokens RPO=0 story as all parents, generalized from a line to a tree.
