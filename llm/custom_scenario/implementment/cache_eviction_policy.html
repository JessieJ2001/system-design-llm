Cache Eviction & Tier-Demotion Policy

The mechanics behind distributed_kv_cache.html Iter 1a DC6 (non-cascading tier-skip) and DC5 (per-tier victim policy): exactly when a KV block leaves HBM, who it is, how many leave, and how the DRAM → SSD pre-offload watermark is sized so the HBM eviction never blocks on a disk write.

Scope & where this sits

This is the implementation reference for the eviction / demotion policy of the local KV tier hierarchy. It is linked from the scenario page’s Iter 1a design choices — DC5 (per-tier victim rule) and DC6 (eviction discipline) name the policy; this page builds out the timing, the watermark math, and the failure path that those two design choices only sketch.

Anchored at Iter 1a’s LOCAL 3-tier model. Iter 1a’s cache lives on one serving task as three local tiers: HBM (on-GPU) → CPU RAM / DRAM (host) → local NVMe SSD (host, via GPUDirect Storage). Your two questions — HBM → CPU RAM and CPU RAM → CPU SSD — are exactly this ladder, so that is the spine of the page. A later iteration reworks the tier model (retires local NVMe, inserts a remote DRAM pool); that evolution is summarised in §7, not the main body. This page is maintained across iterations: whenever any iteration changes the eviction policy, this page is updated (per NEW_IMPLEMENTATION_DETAILS.md §0.8).
One correction up front — demotion is pressure-driven, not pass-driven. A common first guess is “evict HBM→DRAM as soon as a forward pass ends, moving the KV that pass just used.” Both halves are backwards:
  • Trigger is capacity pressure (HBM occupancy crosses a high watermark), checked by a background sweep — not the end of any forward pass. A forward pass ending is not a free-memory event.
  • The KV a pass just used is the hottest set, never the victim. It was touched at last_access = now with ref_count > 0; under LRU + ref_count it is the last thing evicted, and for a multi-turn conversation it is deliberately kept resident for the next turn. The victim is the coldest block of some other, idle conversation.

The two questions this page answers

  1. HBM → CPU RAM — when does a block demote, and how many at a time? Is it tied to the end of a forward pass? Does it move the pass’s own KV? And since an HBM eviction cannot wait for CPU RAM to first offload to SSD, do we pre-offload CPU RAM using a watermark — and how is that watermark chosen?
  2. CPU RAM → CPU SSD — when, how frequently, and how many bytes per offload? And what happens when the SSD write cannot keep up?

Short answers, then the derivations:

QuestionAnswerSection

HBM→DRAM: who?The coldest blocks passing the 3-signal gate (last_access old ∧ ref_count==0 ∧ session_idle) — never the active pass’s KV.§3

1. Mental model — two demotions, one pin

TL;DR. Two different events both move bytes downward, and conflating them is what produces the “evict on pass-end” mistake. One is about a conversation going quiet; the other is about a tier filling up. A single correctness pin (ref_count) sits across both.

The two events

  • (a) Lifecycle cool-down (per conversation). A turn finishes (EOS / decode done). Its KV is no longer active, but the conversation may continue, so we want to keep it for the next turn. ref_count drops to 0; the block becomes evictable but is not moved yet. It just stops being pinned.
  • (b) Capacity demotion (per tier). Independently, a tier fills up. When occupancy crosses the tier’s high watermark, a background sweep picks the coldest evictable blocks and moves them down one (or more) tiers. This is the event your Q1 is really about — and its trigger is the watermark, not any single pass finishing.

Event (a) makes a block a candidate; event (b) decides if and when a candidate actually moves. A block can sit evictable-but-resident in HBM for a long time if there is no pressure — that is the whole point of a cache.

The one pin that crosses both: ref_count

ref_count — what: per-block count of live references (active decode slots + shared-prefix users). used for: the correctness gate “may this block be evicted at all?” — a block with ref_count > 0 is pinned, never a victim, regardless of recency. updated when: +1 on fresh allocation or prefix-match reuse, −1 on EOS / preemption / abort. by whom: the CPU scheduler / block-manager (atomic); the GPU never touches it — the GPU only runs attention over whatever the block table points at.

This is why the just-finished pass’s KV is safe: during the pass its slots hold ref_count > 0 and last_access = now. It is simultaneously the most-pinned and most-recent set — the opposite of a victim.

The tier ladder, its watermarks, and the four flows

Static view of the Iter 1a local hierarchy. Solid arrows are synchronous (on the request’s critical path); dashed-grey arrows are asynchronous background work (hidden from the request); the dashed-red arrow is the loss path.

  • 1HBM → DRAM demote — on HBM crossing W_hi: copy the coldest evictable blocks down via PCIe (25 GB/s), free their HBM slots down to W_lo. Async copy engine, hidden behind compute.
  • 2DRAM → SSD pre-offload — on DRAM crossing W_hi(DRAM): background trickle writes the coldest DRAM blocks to NVMe (3–5 GB/s), keeping H_DRAM free ahead of flow ①.
  • 3Drop → recompute — if DRAM is full and SSD cannot drain in time: discard the coldest blocks; the rare next access re-prefills from token IDs (RPO=0). Loss path.
  • 4Promote on hit — next turn of a demoted conv: swap its KV back up into HBM (DRAM→HBM 25 GB/s, or SSD→HBM 7 GB/s GDS) before the first decode tick.

3. Q1 — HBM → CPU RAM

When does a block demote? (the trigger)

The problem, step by step

  1. Decode is HBM-only: the attention kernel reads a sequence’s entire K,V from HBM every tick. There is no “decode from DRAM” path.
  2. So every active slot must hold its full KV in HBM for its whole lifetime — HBM is the scarce tier.
  3. New work (a fresh prefill, a promoted conv) keeps arriving and needs HBM slots.
  4. If we wait until HBM is completely full to react, the allocator stalls the GPU while it scrambles to free space — a bubble on the critical path.

The solution, step by step

  1. Set a high watermark W_hi below 100% of HBM-for-KV.
  2. A background sweep (CPU-side block-manager) checks occupancy every few ticks.
  3. When occupancy crosses W_hi, it demotes coldest evictable blocks now — ahead of the allocator actually needing the space.
  4. The trigger is therefore occupancy, decoupled from any single forward pass finishing. A pass ending frees nothing by itself (multi-turn KV is kept).
Direct answer to “is it as soon as the forward pass ends?” — No. Demotion fires on HBM_used ≥ W_hi, evaluated by a background sweep, not on pass completion. Coupling it to pass-end would either fire constantly (every tick is a pass) and thrash, or fire on the wrong signal entirely (a pass can end with HBM nowhere near full).

Who is the victim? (the coldness gate)

The eviction-candidacy test — the coldness gate, reused by every downstream tier — is a three-signal AND. All three must hold; recency alone is not enough.

evictable(block) = (last_access old) AND (ref_count == 0) AND (session_idle) recency not live/shared conv quiet long enough
SignalWhat / granularityUpdated when · by whom
last_accessrecency timestamp, per blockstamped = now on every engine step for each block the batch touched; CPU scheduler / block-manager (a CPU-visible proxy — the GPU’s per-tick reads are never individually observed)
ref_countlive-reference count, per block+1 alloc / prefix-reuse, −1 EOS / preempt / abort; CPU scheduler (atomic). The correctness pin.
session_idlederived predicate, per conversation: now − last_turn_ts > Tidlebacked by last_turn_ts, reset to now when a turn is admitted and again on its decode completion; CPU scheduler / Router. Tidle set well above the inter-turn reading gap so an actively-conversing user is never judged cold.
Direct answer to “does it move the KV the pass just used?” — No, the opposite. That KV fails the gate on two of three signals: last_access = now (not old) and ref_count > 0 (live). It is the hottest, most-pinned set on the task. The victim is the coldest block of a different, idle conversation — one whose turn finished long enough ago that all three signals fire.

How many leave, and where are the HBM watermarks set?

The HBM demotion is governed by two watermarks, exactly like the DRAM offload in §4. The loop is tiny:

on each background sweep: if HBM_used ≥ W_hi(HBM): demote coldest evictable blocks (by the §3 gate) until HBM_used ≤ W_lo(HBM) # bytes_moved = HBM_used − W_lo(HBM) — NOT "all", NOT "one pass's worth"

First design choice: why a pair (W_hi and W_lo), not a single line?

Problem. We need a rule for when the background reclaimer runs. The obvious one — “evict whenever HBM_used ≥ T” — has a failure mode.

Options:

  • (a) Single threshold T. Free one block → drop just below T → the next allocation crosses T → fire again. The sweep runs on every tick, each time scanning the LRU order to free one block. Constant CPU + zero batching — thrash.
  • (b) Two thresholds W_hi > W_lo — CHOSEN. Start at W_hi, free a batch down to W_lo, then stay quiet until W_hi is crossed again. The gap W_hi − W_lo is hysteresis: it converts continuous per-tick thrash into occasional batched sweeps, amortising the LRU scan over many freed blocks.
  • (c) Time-based (sweep every Δt regardless of occupancy). Decouples eviction from actual pressure: over-evicts a near-empty cache, under-reacts to a load spike between ticks. Occupancy-triggered watermarks track real pressure.

Why (b): the same min/low/high-watermark pattern every production background reclaimer uses (Linux kswapd, JVM GC). The band is the knob that trades sweep frequency against how much headroom we hold idle.

Second design choice: where do the two HBM watermarks sit? (the derivation)

HBM is a leaky bucket too, but with the roles flipped vs DRAM: here the scarce thing we protect is free HBM slots for a critical-path consumer (the GPU allocator — a new prefill admission or a promoted conv swapping back in), and the “drain” that refills free space is flow ① demotion at PCIe 25 GB/s, which is fast and always succeeds (DRAM is kept free by Q2). So both watermarks come out near the top of HBM.

W_hi(HBM) — the headroom above it must absorb whatever the allocator grabs during the demotion’s reaction time, so the allocator never hits 100% and stalls the GPU:

Free_HBM = Cap_HBM_for_KV − W_hi(HBM) ≥ R_alloc_peak × t_react(HBM) # Peak HBM allocation rate = compute-bound prefill token rate × KV_per_tok: R_alloc_peak = F_sustained / (2·P) × KV_per_tok = 3.17e15 / 1.4e11 × 320 KB = 22 600 tok/s × 320 KB ≈ 7.2 GB/s [D] # Reaction = one sweep interval (detect) + first block demote: t_react(HBM) = t_sweep + block/B_PCIe = ~26 ms (5 ticks) + 205 µs ≈ 26 ms reaction_backlog = 7.2 GB/s × 0.026 s ≈ 0.19 GB [D] # Slab-granularity floor (Iter 1a reserves a whole contiguous slab on admit): Free_HBM = max(reaction_backlog, one mean admission) = max(0.19, 4.16) ≈ 4.2 GB ⇒ W_hi(HBM) ≈ 450 − ~5 GB (with margin) ≈ 445 GB [D]

W_lo(HBM) — how far down each sweep frees. The band W_hi − W_lo is sized so the sweep does not immediately re-fire: free enough for k admissions to proceed before the next crossing.

band_HBM = W_hi(HBM) − W_lo(HBM) = k × mean_admission_KV (k ≈ 3) = 3 × 4.16 GB ≈ 12.5 GB [D] ⇒ W_lo(HBM) ≈ 445 − 12.5 ≈ 432 GB [D]
  • Reading it: at 445 GB the sweep fires, frees ~12.5 GB of the coldest idle KV down to 432 GB, then goes quiet. ~3 mean admissions (12.5 GB) can run before it fires again — so the LRU scan is paid once per ~3 admissions, not once per tick.
  • The inputs are the same three as Q2 — drain speed (here PCIe, fast), the batch-KV statistic (mean admission for the band), and per-step latency (sets t_react). Because the HBM drain is fast and reliable, the headroom comes out thin (~5 GB of a 450 GB tier); the contrast with DRAM’s thick headroom is the whole point of the next table.
  • Tuning note for the implementer: use the workload’s mean admission size for the band, a small safety factor on Free_HBM, and raise k if you observe the sweep firing more than ~once per few admissions in production.
Latency is fully hidden. The HBM→DRAM copy runs on an async copy engine: one 5.12 MB block = 205 µs over PCIe, and the decode tick is ~5.2 ms — so ~25 blocks drain behind a single tick. Because the sweep runs ahead of demand (that is what Free_HBM buys), the next forward pass never waits on flow ①. This is why HBM→DRAM is the easy half; the hard half is keeping DRAM itself free (Q2).

Why both watermarks on both tiers — but sized differently

A fair question: if HBM gets a W_hi/W_lo pair, does DRAM? Yes — every background reclaimer needs the anti-thrash pair (the design choice above applies to both tiers). What differs is the headroom thickness and what it defends, and that difference is set by the tier’s drain:

 HBM watermarksDRAM watermarks
W_hi / W_lo pair?Yes (anti-thrash)Yes (anti-thrash) — same pattern
What the headroom protectsfree slots for a critical-path consumer (GPU allocator); a miss stalls the GPUa burst backlog the slow drain cannot remove fast enough
Its draindemote → DRAM: fast (25 GB/s) & always succeeds (DRAM kept free by Q2)write → SSD: slow (3–5 GB/s) & can be overwhelmed (requirement 1)

One-line answer. The watermark pair is symmetric (both tiers are background reclaimers, both need hysteresis); the headroom thickness is asymmetric, set by drain speed × reliability. HBM’s fast, never-failing drain ⇒ a thin headroom whose job is anti-thrash. DRAM’s slow, defeatable drain ⇒ a thick headroom whose job is to absorb a burst the disk cannot — which is exactly what §4 derives.

4. Q2 — CPU RAM → SSD pre-offload (the watermark)

TL;DR. Your instinct is exactly right: an HBM eviction cannot wait for DRAM to free a slot by writing to SSD synchronously (a 4 GB conv at 3.5 GB/s = ~1.1 s of stall). So DRAM runs its own background watermark sweep that pre-offloads cold blocks to SSD ahead of demand — keeping a free headroom H_DRAM that the HBM demotion always lands in. The whole question becomes: how big must H_DRAM be, and how fast must the SSD writer run?

It’s a leaky bucket

Model the DRAM headroom as a buffer between a producer and a consumer:

Sizing the watermark from your three inputs

You named the three inputs exactly. Here is how each enters the math:

Your inputSymbolRole in the watermark
CPU RAM → SSD latency / bandwidthR_out = B_ssd_writethe drain rate — sets requirement (1) and the drain term of (2)
batch KV cache size (avg / median / max)S = N · KV_per_tokthe size of one demotion event — the statistic chosen sets the burst we plan for
avg process latencyt_burst = prefill_time(N)how long a burst lasts — the producer injects at S / t_burst for this long

Requirement (1) — the drain must keep up (the floor)

B_ssd_write ≥ R_demote_mean # mean rate retained KV is pushed out of HBM
  • R_demote_mean is the steady-state rate at which kept KV (multi-turn convs cooling and getting displaced) flows down. Its exact value depends on the retained fraction, which a later iteration derives from the demotion flow — kept symbolic here, not fabricated.
  • But it is bounded by the per-stream KV creation rate, which is fully derivable. For the mean request at B=1: R_create_sustained = (13K + 400)·320 KB / (0.73 s + 400·5.2 ms) = 4.29 GB / 2.81 s ≈ 1.53 GB/s [D].
  • So at single-stream (Iter 1a, B=1): R_demote_mean ≤ 1.53 GB/s ≪ 3.5 GB/s = B_ssd_write — the floor holds with margin. One NVMe keeps up.
The floor is what eventually breaks — and it is a known scale lever, not a new design choice. Requirement (1) scales with concurrent prefill streams: R_demote_mean ≤ 1.53 GB/s × B_prefill. Past B_prefill ≈ 3.5 / 1.53 ≈ 2.3 concurrent streams of fully-retained KV, one NVMe’s write BW is exceeded. The design already has the three escape valves: (a) the admission gate writes down only KV worth more than recompute (so retained ≪ created); (b) a 4-NVMe write stripe (the Iter 1a DC2 alternative flagged for scale); (c) a later iteration retires local NVMe for a remote pool entirely (§7). This matches the scenario page’s “SSD-BW saturation under high eviction churn” risk — this page does not introduce a new choice.

Requirement (2) — the headroom must absorb a burst (sizes H_DRAM)

A prefill burst creates KV faster than the drain removes it. The headroom must hold the backlog that builds during the worst burst. For a request of N tokens, the producer injects N·KV_per_tok over prefill_time(N) while the drain removes B_ssd_write × prefill_time(N). The net backlog is the difference:

# Backlog a single prefill stream of N tokens can pile up: backlog(N) = N·KV_per_tok − B_ssd_write · prefill_time(N) = created during the burst − drained during the burst # This is NOT monotonic in N: long prefills are slow, so the drain catches up. # Maximise over N to find the worst-case burst we must size for.

Substituting the prefill-time formula prefill_time(N) = (2P·N + 4·n_layers·d_model·N²)/F_sustained with the page constants (2P = 1.4×10¹¹, 4·80·8192 = 2.62×10⁶, F_sustained = 3.17×10¹⁵, B_ssd_write = 3.5×10⁹, KV_per_tok = 3.2×10⁵):

backlog(N) = 3.2e5·N − 3.5e9·(1.4e11·N + 2.62e6·N²)/3.17e15 = 3.2e5·N − (1.546e5·N + 2.89·N²) = 1.654e5·N − 2.89·N² [D] # d/dN = 0 ⇒ N* = 1.654e5 / (2·2.89) ≈ 2.86e4 tok (≈ 28.6 K) # backlog(N*) = 1.654e5·28600 − 2.89·28600² ≈ 2.37 GB [D]

Plus the detection lag — the sweep checks the watermark every t_sweep ≈ 5 ticks ≈ 26 ms, during which the drainer may not yet have reacted, so add R_peak(N*) × t_sweep. At N*, prefill_time(28.6K) ≈ 1.94 s ⇒ R_peak = 9.15 GB / 1.94 s = 4.72 GB/s, so the lag term is 4.72 × 0.026 ≈ 0.12 GB:

H_DRAM(per prefill stream) ≈ backlog(N*) + R_peak·t_sweep ≈ 2.37 + 0.12 ≈ 2.5 GB [D] with a 1.5× safety factor ⇒ ~3.7 GB per concurrent prefill stream At Iter 1a (B=1): W_hi(DRAM) = 800 GB − 3.7 GB ≈ 796 GB (~99.5%) General: H_DRAM ≈ B_prefill × 2.5 GB (scales with concurrency)

W_lo(DRAM) — DRAM uses the same W_hi/W_lo pair as HBM (the anti-thrash design choice in §3 applies to every background reclaimer). The band sets how long the SSD writer runs once it wakes: free a worthwhile batch so the drive does sequential bursts rather than start/stopping per block.

band_DRAM = W_hi(DRAM) − W_lo(DRAM) = drain-run length ≈ 2 × H_DRAM # a couple of bursts' worth before going quiet ≈ 2 × 3.7 GB ≈ 7.4 GB (at B=1) [D] ⇒ W_lo(DRAM) ≈ 796 − 7.4 ≈ 789 GB [D] # Contrast with HBM: there the band amortises an LRU SCAN (CPU cost); # here it amortises SSD WRITE setup (sequential-vs-random throughput).

So both tiers carry a full pair. The reason the headroom above W_hi(DRAM) (H_DRAM ≈ 3.7 GB/stream) is so much thicker than HBM’s (~5 GB total) is the drain asymmetry tabulated in §3: DRAM’s drain is slow and can be overwhelmed, so its headroom must hold a real burst; HBM’s drain is fast and never fails, so its headroom is just anti-thrash slack.

The design choice you flagged: avg, median, or max batch size?

This is a real knob, and the answer is different for the two requirements — which is the whole subtlety:

Where the size feeds inWhich statisticWhy
Requirement (1) — sustained drainmeanSteady-state throughput must cover the average demotion rate. Provisioning the drain for the peak wastes drives; provisioning for the median under-serves.
Requirement (2) — burst headroomneither mean nor max — the size that maximises backlog(N) (here N* ≈ 28.6 K, a p90-ish request)The worst burst is not the largest request. A p99 256 K prefill is slow (65.5 s), so the drain keeps up: backlog(256K) = 82 − 3.5×65.5 = 82 − 229 < 0 — no backlog at all. Sizing on “max” over-provisions DRAM for a whale the disk already absorbs. Sizing on “mean” (13 K) slightly under-provisions. The true worst case is the medium-large request that is both big and fast.
Net rule. Provision the SSD write throughput for the mean demotion rate; size the DRAM headroom for the request that maximises created-minus-drained (a medium-large N, derived by maximising backlog(N)), not the largest request. “Just use max” is the trap the curve’s non-monotonicity sets.

5. The loss path — when the SSD can’t drain in time

If a sustained surge violates requirement (1) and exhausts H_DRAM faster than the drainer can refill it, DRAM hits 100% with an HBM demotion still inbound. The bucket overflows. The policy is deliberately simple:

  1. Never stall the GPU to wait for a write. The HBM demotion must land somewhere instantly, so blocking on the SSD write is off the table.
  2. Drop the coldest blocks instead of writing them. The same coldness gate (last_access ∧ ref_count==0 ∧ session_idle) already ranks victims; under overflow, the bottom of that ranking is simply discarded, not written down.
  3. The next access re-prefills from token IDs. A dropped conv that returns pays βcold = prefill_time(N) once, then is cached again. The prompt token IDs are durable, so recompute is always available — RPO = 0.
Direct answer to “we will just lose some cache” — yes, and that is correct by design. Cache loss is a latency event (one extra re-prefill for the unlucky returning conv), never a correctness event. The drop targets the coldest blocks — the ones least likely to be read back — so the expected cost is minimised. The blast radius is bounded to the convs dropped during the surge; everything hot stays resident.
Why not back-pressure admission instead? You could refuse new prefills until DRAM drains — but that converts a cheap, rare, per-conv re-prefill into a queue-wide TTFT spike for every waiting request. Dropping the coldest is strictly more local. (Admission control still exists for throughput protection — it just is not the tier-pressure release valve.)

6. Worked example — one task under steady mean load

One Iter 1a serving task, B=1, serving the workload-mean request (Tin = 13 K, Tout = 400), HBM near its watermark. Trace one demotion end to end.

At single-task B=1 the system is comfortably inside both requirements: HBM demotion is hidden (166 ms async, GPU never stalls), and DRAM drains faster than it fills (3.5 > 1.53 GB/s). The watermark headroom (~3.7 GB) exists for the prefill burst case (a fresh 28 K-ish prefill arriving while DRAM is near W_hi), not the steady state. Pressure on requirement (1) only appears as concurrency grows — the scale lever in §4’s warn-box.

7. How this policy evolves across iterations

This page is anchored at the Iter 1a local model, but the eviction policy changes as later iterations land (and is re-synced here when they do, per NEW_IMPLEMENTATION_DETAILS.md §0.8). The policy moves along three orthogonal axes: the eviction unit (slab → block), the demotion destination (local SSD → remote pool), and the copy semantics (always-copy → clean-drop). Below is each iteration’s eviction policy in full — the mechanism, the data-structure delta, and the exact eviction decision.

Iter 1a — born: conv/slab-granular, local 3-tier

  • Unit: the whole contiguous per-request slab. There is no sub-slab eviction — a conversation’s KV moves as one block of bytes.
  • State: ref_count, last_access, and tier_tag are tracked per slab / per conversation (the slab is the conv’s KV).
  • Policy: per-tier rule (HBM = LRU + ref_count; DRAM/SSD = plain LRU), non-cascading tier-skip on the 3-signal coldness gate, async watermark pre-offload. This is the entire main body above (§3–§5).
# ITER 1a — the eviction loop is born. Unit = whole conv SLAB; always copies down. on HBM_used ≥ W_hi(HBM), coldest-first over conv SLABS: if slab.ref_count > 0: # active / shared — pinned (correctness), never a victim continue if not coldness_gate(slab): # last_access old ∧ ref_count==0 ∧ session_idle continue demote(slab, cheapest_adequate_tier) # HBM→DRAM→SSD, NON-CASCADING tier-skip # ALWAYS copies down — no clean-drop (each tier holds 1 canonical copy), # no length gate (local tiers always accept the spill)

Iter 2 — P/D split: same policy, run on two pools

  • No policy change — a topology change. Prefill-pool tasks and decode-pool tasks each run the same local HBM→DRAM→SSD eviction independently; nothing about the gate or watermarks changes.
  • New constraint it surfaces: on a decode task the active-decode set is hard-pinned (ref_count > 0) and cannot be evicted mid-decode — doing so would cost βDRAM(N) per token. So eviction fires only on paused / finished convs, exactly as the ref_count pin already guarantees. Still slab-granular (paging is Iter 3).

Iter 3 — PagedAttention: the eviction unit becomes the 16-token block

This is the iteration your question is about. KV stops being a contiguous slab and becomes a list of fixed 16-token blocks (5.12 MB each), indexed by BlockTable[seq_id] = [block_id, block_id, …]; free blocks live on a LIFO free-list. The eviction unit, and every signal, follows the block.

Direct answer: yes — ref_count moves from per-slab to per-block. Each physical block_id carries its own ref_count = the number of BlockTable entries (across all sequences) currently pointing at it. The GPU still never touches it; the CPU block-manager maintains it atomically.
  • What ref_count counts at Iter 3. Most blocks are private to one sequence, so ref_count just toggles 0↔1 with that sequence’s life (allocated → freed on EOS). It is still the liveness / correctness pin: a block any live sequence’s table points at cannot be freed. Within-request sharing (copy-on-write for parallel sampling / beam search) already makes ref_count > 1 possible here; cross-request prefix sharing — where one block is pointed at by thousands of convs and ref_count becomes large and load-bearing — arrives at Iter 5.
  • last_access also goes per-block (the scheduler stamps every block the batch touched each engine step). session_idle stays per-conversation — a block inherits its owning conv’s idle predicate (now − last_turn_ts > Tidle); there is no per-block idle clock.
  • The coldness gate now evaluates per block: evictable(block) = last_access[block] old ∧ ref_count[block]==0 ∧ session_idle(conv_of(block)). The sweep ranks individual blocks by LRU within the evictable set.
  • New capability the block unit unlocks: partial eviction. A long sequence’s cold early-context blocks can be demoted while its hot tail stays HBM-resident — impossible with a contiguous slab, which was all-or-nothing. This is why paging makes mid-flight demotion safe.
  • Mechanics: HBM free-list count < low_watermark → sweep picks coldest evictable block_ids → demote each (5.12 MB / 25 GB/s = 205 µs over PCIe) → set tier_tag[block_id]=DRAM, push the freed slot to the free-list. Non-cascading tier-skip, per block. The 205 µs/block and ~25 blocks/tick figures used in §3 become literal here — the 16-tok block is the demotion quantum.
# ITER 3 — Δ from 1a: the UNIT shrinks slab → 16-tok BLOCK; ref_count/last_access go per-block. Decision logic unchanged. on HBM free-list < low_watermark, coldest-first over 16-tok BLOCKS: # ← was conv SLABS if block.ref_count > 0: # ← ref_count now PER-BLOCK (# BlockTable entries pointing at it) continue if not coldness_gate(block): # per-block last_access; session_idle still per conv continue demote(block, cheapest_adequate_tier) # block-granular tier-skip; STILL always copies down # ← partial eviction now possible: a conv's cold early blocks spill while its hot tail stays in HBM

Iter 3b — tier rework: the destination changes + a length-gate admission

  • Local NVMe is retired (Q7/DC7). The cascade becomes HBM → local DRAM → remote pool DRAM front → remote pool NVMe back → recompute — no local SSD.
  • The DRAM pre-offload now drains over the network. §4’s DRAM→down watermark writes to the remote pool via RDMA (25 GB/s) instead of local NVMe (3–5 GB/s). The drain is ~7× faster and the destination is a shared pool with its own capacity-LRU — but it is still async, ahead of demand: a synchronous RDMA spill would land on the decode step’s TPOT, so the trickle stays off the request path exactly as before.
  • Eviction becomes an admission decision for the tier below (Move D). A block that passes the coldness gate is then length-tested: recompute_cost(N) > transfer_cost(N)? (the ~1.9 K-token crossover, transfer_cost measured against the slowest serving tier). Worth it → write down to the pool; not worth it → discard (recompute floor). So not every evicted block is written down — short, cheap-to-recompute convs are dropped rather than shipped across the fabric.
  • Unchanged: ref_count, the 3-signal coldness gate, and the hysteresis-pair watermarks. What is new is only the destination (pool, not local NVMe) and the length gate on write-down.
  • Pool sizing (now derived — flagged model): the posited f_pool fraction is retired (D5); pool size is the outcome of the demotion flow — a derived per-turn admit-probability padmit ≈ 7% → Wpool ≈ 2.1 PB → ~36 store tasks (scenario §9b.4; dominant uncertainty = the inter-turn-gap heavy-tail exponent). (Pool-side internal DRAM-front→NVMe-back demotion lives in kv_store.html.)
# ITER 3b — Δ from 3: down-destination local-NVMe → REMOTE pool; a LENGTH GATE now decides write-down vs discard. on HBM / local-DRAM watermark, coldest-first over BLOCKS: if block.ref_count > 0: continue if not coldness_gate(block): continue if a LOCAL tier below has room: # HBM → local DRAM: free, ungated (unchanged from Iter 3) demote(block, DRAM) else: # ← leaving local ENTIRELY (local NVMe retired, DC7) → the remote pool if recompute_cost(N) > transfer_cost(N): # ← LENGTH GATE (Move D), NEW: the pool is networked + scarce write_down(block, pool) # RDMA to L3 front; pool demotes to its L5 NVMe back internally else: discard(block) # recompute floor (RPO=0) — short conv not worth the fabric # Still NO clean-drop: no content hash yet to prove a lower copy is identical (that's Iter 5).

The length gate is what keeps cheap-to-recompute short convs out of the scarce shared pool, and the RDMA destination + global pool give the long conv something local NVMe never could: its cache survives a rebalance to another task. Iter 5 — prefix caching: clean-drop, reuse-recency retention, and the full decision tree

  • Blocks become content-addressed. block_hashi = hash(parent_hash || tokensi) — chained, content-only (no per-user seed). Identical content → identical hash → one shared physical block. Now ref_count ≫ 1 is the common case for a popular system-prompt prefix (thousands of convs point at it) — this is where the per-block ref_count pin from Iter 3 truly bites: such a block is read once at prefill, looks old by last_access, but its high ref_count keeps it pinned, where pure LRU would evict it and corrupt every conv whose BlockTable points at it.
  • tier_locs bitmap per block (HBM | DRAM | REMOTE); dirty/clean derived from it. A block is clean when an identical copy provably already exists in a lower tier.
  • Clean-drop (Move B2) — the new third eviction path. An HBM eviction can free a block with no copy-down at all when tier_locs shows a clean copy below. Impossible before Iter 5: without a content hash you cannot prove a lower copy is byte-identical, so you always had to copy down. Content-addressing is what makes the proof free.
  • Reuse-recency keeps hot shared prefixes resident — no hard pin (DESIGN_CHOICE_CHANGE.md Change 5). A popular shared prefix is re-hit on essentially every request, so the LRU key (refreshed on cache hit / reuse at admission, not per-token kernel reads) keeps it at the front, and its ref_count rarely reaches 0 — it stays resident because it is reused, not because it is excluded from the candidate set. The watermark-LRU evicts the released (ref_count==0) tail by measured recency. Taxonomy (Move E): shared-prefix blocks (kept by re-hit) vs session-tail blocks (kept by recency) — reuse-recency + ref_count serve both. (An optional soft retention-priority bias is the documented upgrade, off the main line.)

Putting it together, the Iter 5 eviction decision for each candidate block (coldest first) is the page’s fullest form (the scheduler-side evictor_loop() that runs it — _is_clean, the ref_count pin, the length gate — is scheduler.html §11b.3; the admit-side prefix match that creates the shared blocks is §11b.2):

# ITER 5 — Δ from 3b: reuse-recency LRU + ref_count (NO hard pin; Change 5) and + CLEAN-DROP (is_clean → free, no copy-down). on HBM_used ≥ W_hi(HBM), for block in coldest_first: # coldest = reuse-recency order (LRU key = last cache-hit/reuse) if block.ref_count > 0: # live/shared — the only hard pin (correctness); a hot prefix rarely hits 0 continue # not evictable — skip if not coldness_gate(block): # last_access old ∧ ref_count==0 ∧ session_idle continue if block.is_clean(): # tier_locs shows an identical copy below free(block) # CLEAN-DROP: no copy-down (Move B2) else: # dirty: no lower copy yet if recompute_cost(N) > transfer_cost(N): write_down(block, pool) # admit to the tier below (Move D length gate) else: discard(block) # recompute floor (RPO=0)

Iter 6–10 — pressure, not policy (with one real upstream role at Iter 10)

From Iter 6 on, the eviction policy stops evolving — the gate, the tier-skip discipline, the watermark form, and the clean-drop semantics are all settled. What still moves is the pressure on it (how many slots are pinned, how big each pinned region is) and, at Iter 10, the downstream consumer of what it demotes.

  • Iter 6 (continuous batching) raises the active batch to Bdecode ≈ 106, so many more slots are hard-pinned (ref_count > 0) at once and steady demotion pressure rises — but the gate, the tier-skip discipline, and the watermark form are unchanged. The only quantitative knock-on is the one §4 already states: DRAM headroom scales with the number of concurrent prefill streams (H_DRAM ≈ B_prefill × ~2.5 GB).
  • Iter 7 (speculative decoding) — a small fixed HBM reservation, no policy change. The ~1B draft model takes ~10 GB HBM/task, trimming the KV budget G_KV 450 → ~440 GB (so Bdecode 106 → ~104, absorbed in headroom). That is a fixed reservation like the weights — not an evictable block — so it slightly shrinks the cache’s capacity but never enters the demotion path. The speculative tail’s KV is allocated then, on reject, freed by a pointer/length reset on the block table (same tick, no copy-down) — an instantaneous free, not a demotion, so it adds zero watermark pressure. scheduler §11d.
  • Iter 8 (per-user abuse limits) — Router-side; if anything, relieves pressure. The token-bucket + max-concurrent-conv gate (router §8e) rejects abusive requests before they consume a serving slot, so fewer slots get pinned and demotion pressure on the affected tasks drops marginally. Zero eviction-policy change.
  • Iter 9 (context parallelism) — an unusually large pinned region, on a dedicated sub-pool. A CP rank holds a ~40 GB KV slice of a 1 M-context request (320 GB / 8), pinned (ref_count > 0) for the request’s lifetime like any active-decode block — the per-task eviction policy is unchanged. The only nuance: a CP slice is a far larger pinned block than a normal conv, so a task serving one has less evictable headroom — but CP requests run on a dedicated cp_ring sub-pool (router §8f), so this does not perturb the common decode pool’s watermarks. scheduler §11f.
  • Iter 10 (fault tolerance) — no policy change, but the demote-to-pool flow becomes load-bearing. This is the one real connection. The eviction policy’s promote-on-evict branch (the Iter 3b length-gated write-back to the remote pool, §7 Iter 3b) is exactly what makes a conv pool-resident — and pool-residency is precisely what lets Iter 10’s recovery do a pull(N) ≈ 166 ms instead of a βcold 730 ms recompute when a task dies. So the length gate is now also a recovery-RTO knob: the convs it admits to the pool are exactly the ones that get the fast recovery path. The policy itself is unchanged (same gate, same clean-drop); what changed is that its output now feeds availability, not just cross-turn cost. scheduler §11g / kv_store §1.
What does not change — the invariants. Across every iteration: the async-watermark discipline (drain ahead of demand, never block the request path on a downward write), the three-signal coldness gate (last_access ∧ ref_count==0 ∧ session_idle), the hysteresis W_hi/W_lo pair on each tier, and the recompute (RPO=0) floor. Only the unit (slab→block, Iter 3), the destination (local SSD→remote pool, Iter 3b), and the copy semantics (always-copy→clean-drop, Iter 5) evolve — and from Iter 10 the demote-to-pool output additionally backs the fault-tolerance pull-fast-path.
