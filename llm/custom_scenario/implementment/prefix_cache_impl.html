Radix tree vs block-hash: is “structure-aware eviction” a real benefit?

No. Held at the same page granularity, a flat block-hash table reproduces a RadixAttention-style tree’s “evict the coldest leaf, never an internal node with a live child” order exactly — using two extra scalar fields per entry (last_access, min_position) on top of the ref_count it already carries. So structure-aware eviction is a cheap block-hash upgrade, not a tree capability. The only genuine capability gap is benefit #1 (the token-granular match boundary, which is really a page_size knob).

What this page shows (and isn’t)

TL;DR. Iter B1 lists four claimed benefits of a radix tree over a block-hash KV index. Benefit #2 was “structure-aware eviction”: the tree makes “evict the coldest leaf, never an internal node with live children” a first-class operation because the parent/child structure is the sharing graph. This page proves that claim is not a discriminating benefit — both representations can do it, and a flat table matches the tree byte-for-byte on eviction order once it carries two scalars it can compute for free.

  • Is: two small, runnable Python caches — (a) a per-page radix tree with leaf-LRU eviction, (b) a flat block-hash table with the (ref_count==0 gate → last_access → min_position) policy — driven by one fork-tree workload, with an assertion that their eviction sequences are identical; plus the lemma that explains why they must match and exactly which field does the work.
  • Is not: the token-granularity question (benefit #1) — that is a real gap and is held fixed here (both run at the same page size) so we isolate eviction. It is summarised in §5. The distributed-pool keying debate lives on the scenario page.
The point in one line. The radix tree gets eviction order from its shape; the block-hash table gets the same order from data on each entry. Same result, different bookkeeping — a convenience, not a capability.

1. The shared model — page granularity held fixed

To compare only eviction, both caches store KV at the same page granularity (BLOCK = 4 tokens/page) and key each page by the path-dependent prefix hash (hash(tokens[:(i+1)·BLOCK])) — the APC content-hash chain (K2 D5). Identical prefixes collide on the same key, so prefix sharing is identical in both; the only thing that can differ is which dead page each evicts first.

import hashlib BLOCK = 4 # tokens / page. FIXED across both impls => only eviction differs. def page_keys(tokens): """key[i] = hash(tokens[:(i+1)*BLOCK]) -- identical prefixes collide => sharing (K2 D5).""" assert len(tokens) % BLOCK == 0, "demo uses full pages only (benefit #1 is separate)" keys = [] for i in range(0, len(tokens), BLOCK): h = hashlib.blake2b(repr(tokens[: i + BLOCK]).encode(), digest_size=6).hexdigest() keys.append(h) return keys

Each page key has a fixed depth (its page index), because a given prefix has a fixed length. Depth × BLOCK is the page’s min_position — the start offset of its first token. That offset is known at allocation, so it is free.

2. The two implementations

Both expose the same interface: open(keys, t) (admit a head + its path, mark it live), close(keys, t) (the head dies), evict_one() (reclaim one page under pressure). Reading them side by side, the only difference is the body of evict_one and how each stores the structure.

2a. Radix — structure from the tree shape

One node per physical page. A node’s ref = live heads passing through it (incremented along the whole path on open). A leaf = a node with no resident child. Eviction picks the coldest leaf.

class Radix: def __init__(self): self.nodes = {} # key->{parent,children,ref,last,depth} self.evicted = [] self.hits = 0 def _walk(self, head, t, delta): k = head # ref += delta UP the path while k is not None: n = self.nodes[k] n["ref"] += delta if delta > 0: n["last"] = max(n["last"], t) k = n["parent"] def open(self, keys, t): parent = None for depth, k in enumerate(keys): if k in self.nodes: self.hits += 1 # shared page else: self.nodes[k] = dict(parent=parent, children=set(), ref=0, last=t, depth=depth) if parent is not None: self.nodes[parent]["children"].add(k) parent = k self._walk(keys[-1], t, +1) # head is live def close(self, keys, t): self._walk(keys[-1], t, -1) def evict_one(self): # resident LEAF + ref==0, then coldest (LRU), # deepest-first as the recency tie-break. cands = [k for k, n in self.nodes.items() if n["ref"] == 0 and all(c not in self.nodes for c in n["children"])] if not cands: return None v = min(cands, key=lambda k: ( self.nodes[k]["last"], -self.nodes[k]["depth"], k)) n = self.nodes.pop(v) if n["parent"] in self.nodes: self.nodes[n["parent"]]["children"].discard(v) self.evicted.append(v) return v

2b. Block-hash — structure from two scalars

One flat entry per physical page; no parent/child links. It carries ref (flat, incremented along the path), last (recency), and min_pos (depth×BLOCK). Eviction is the triple: ref==0 gate → LRU → deepest-first.

class BlockHash: def __init__(self): self.entries = {} # key->{ref,last,min_pos} self.evicted = [] self.hits = 0 def open(self, keys, t): for depth, k in enumerate(keys): if k in self.entries: self.hits += 1 # shared page else: self.entries[k] = dict(ref=0, last=t, min_pos=depth * BLOCK) for k in keys: # flat ref-count, self.entries[k]["ref"] += 1 # monotone UP the chain self.entries[k]["last"] = max( self.entries[k]["last"], t) def close(self, keys, t): for k in keys: self.entries[k]["ref"] -= 1 def evict_one(self): # (1) ref==0 gate # (2) last asc = LRU primary # (3) largest min_pos = deepest first cands = [k for k, e in self.entries.items() if e["ref"] == 0] if not cands: return None v = min(cands, key=lambda k: ( self.entries[k]["last"], -self.entries[k]["min_pos"], k)) self.entries.pop(v) self.evicted.append(v) return v
The one structural difference. Radix’s candidate set is computed by walking the shape (“no resident child”). Block-hash’s candidate set is a one-line filter (ref==0), and the leaf-first ordering that the shape gave radix for free is recovered by the -min_pos tie-break. Everything else — open, close, the ref bookkeeping, sharing — is line-for-line the same.

3. The driver & the equivalence assertion

One fork-tree workload: a 2-page shared system prefix; b1 and b2 share a tail page (b2 extends b1); b3 and b4 diverge at depth 2. We open all four, then kill b3 (oldest dead), then kill b1 and b2 together (so they share last_access — the tie case that needs min_pos). b4 is left live to prove the ref-gate protects it. Then we drain evictions and compare.

# --- workload: a fork tree (shared sublists => shared prefix hashes) --- sys = [0,1,2,3, 4,5,6,7] # 2 shared system pages (depth 0,1) b1 = sys + [10,11,12,13] # trunk + tail page (depth 2) b2 = b1 + [20,21,22,23] # shares b1's tail, + page (depth 3) b3 = sys + [30,31,32,33] # diverges at depth 2 (own page) b4 = sys + [40,41,42,43, 44,45,46,47] # diverges at depth 2, 2 pages deep keys = {n: page_keys(s) for n, s in {"b1":b1,"b2":b2,"b3":b3,"b4":b4}.items()} events = [(1,"open","b1"),(2,"open","b2"),(3,"open","b3"),(4,"open","b4"), (5,"close","b3"), # b3 dies first (oldest dead) (6,"close","b1"),(7,"close","b2")] # b1,b2 die together -> share last_access; b4 stays LIVE R, H = Radix(), BlockHash() for t, act, br in events: (R.open if act == "open" else R.close)(keys[br], t) (H.open if act == "open" else H.close)(keys[br], t) while R.evict_one() is not None: pass # drain both under full pressure while H.evict_one() is not None: pass assert R.evicted == H.evicted # SAME eviction order assert R.hits == H.hits # SAME prefix sharing live = set(keys["b4"]) assert not (set(R.evicted) & live) # ref-gate protected live b4 assert not (set(H.evicted) & live)

Running it (the exact script lives at tmp/prefix_cache_demo.py):

radix hits: 7 evicted: ['f413ca42a253', 'a74261dfdc32', 'c19f75804dd1'] blockhash hits: 7 evicted: ['f413ca42a253', 'a74261dfdc32', 'c19f75804dd1'] eviction order identical: True same prefix-sharing (hit count): True ALL ASSERTIONS PASS -> block-hash + (ref==0, last_access, min_pos) == radix leaf-LRU

Three dead tail pages are reclaimed in the same order by both; the two shared system pages and b4’s two tail pages survive in both because b4 is live (ref > 0 all the way to the root). The b1/b2 tie (both died at the same instant, equal last_access) is broken deepest-first — b2’s depth-3 page before b1’s depth-2 page — which is exactly what the radix leaf constraint forces.

4. Why they must match — the recency-monotonicity lemma

This is not a coincidence of the workload. There is a one-line invariant that makes the flat policy provably equal to radix leaf-LRU:

The problem, step by step

  1. Radix evicts only leaves (no resident child), so it structurally peels the deepest resident dead page of any chain first, then works inward.
  2. Block-hash has no notion of leaf — its ref==0 gate admits an entire dead chain at once (a shallow dead page and its dead children are all candidates simultaneously).
  3. So the worry is: could the flat policy evict a shallow dead page before its own deeper dead child, violating radix’s leaf-first order and (in a real cache) throwing away a long shared prefix that a future request would re-hit?

The solution, step by step

  1. Recency is monotone up the chain. A page is touched on a superset of the paths/times its descendants are (every head below it also passes through it). So for any ancestor a and descendant d: a.last_access ≥ d.last_access. The deeper page is always at-most-as-recent.
  2. Strict case needs no help. When the inequality is strict, plain LRU (the last_access primary key) already picks the deeper page first — leaf-first falls out for free, no min_pos required.
  3. The tie case is the only place min_pos earns its keep. Releasing a head decrements ref along its whole unique suffix at one instant, so a parent and child can die with equal last_access (exactly the b1/b2 tie). Plain LRU is then ambiguous; the -min_pos tie-break forces deepest-first — reproducing radix’s leaf order precisely.
  4. Correctness needs no field at all. The “never evict an internal node with a live child” invariant is enforced by the ref==0 gate alone, because ref is monotone up: ref(page)==0 ⇒ every descendant is 0 ⇒ no live child. There is nothing for the tree to close here.
So the faithful flat policy is the triple (ref_count==0 gate → last_access primary → min_position tie-break). last_access recovers recency (what radix’s LRU encodes); min_position recovers structure (what radix’s shape encodes) but only matters on exact-recency ties. Both fields are free (recency is a timestamp; min_position is the allocation offset). The radix tree is therefore a convenience for this operation, not a capability — matching Iter B1’s conclusion that only benefit #1 is a genuine gap.
The trap to avoid: depth alone is not recency. If you drop last_access and order purely by min_position (deepest-first), you break on a hot, idle-tailed long conversation whose deep pages outrank a genuinely cold short one’s — you would evict the hot tail. min_position is the tie-break, not the primary key. The triple, in that order, is what reproduces radix.

5. What the radix tree still wins — benefit #1, and it is a page_size knob

Eviction is a wash; the genuine gap is the token-granular match boundary, which this page deliberately held fixed by running both at BLOCK = 4. The honest framing:

#1 Token-granular sharingshares whole pages; a non-aligned fork CoW-copies the trailing page + re-prefills up to BLOCK−1 tokenscan share to the exact divergence token iff page_size→1radix wins — but only with a small/unit page; cost is page-table size + kernel-gather efficiency
#2 Structure-aware eviction(ref==0, last_access, min_pos) — proven equal aboveleaf-LRU from the tree shapewash — this page
#3 One-walk longest-prefix matchP/BLOCK O(1) probesone pointer walknear-wash on time
#4 Native many-way branchingflat entries + refinsert a child edge at the divergence nodeergonomic only

And even #1 reverses at the distributed layer: a content/block hash is a flat, global, shardable key (the whole kv-store-job design); a radix tree is a centralized in-memory structure. So the scenario keeps block-hash for the shardable pool and adopts a radix tree only as the local per-task structure where arbitrary-fork-point sharing dominates — see Iter B1’s “honest tradeoff”.
