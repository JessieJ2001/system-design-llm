Implementing the Per-Task Scheduler

State evolution, the per-tick GPU contract, and the protocol with the Router — the impl-detail page for the distributed-KV-cache scenario. Tracks (a) what data structures the scheduler holds; (b) what it asks the GPU to do each tick — what it gives, what it gets back; (c) what it pushes to the Router every 100 ms and consumes from it; (d) the full CPU-resident vs GPU-resident state snapshot, the KV-cache byte-layout, and the async watermark-eviction machinery — all re-stated in full at every iteration so each one reads stand-alone.

1. The three-layer routing stack — this page owns L3

Every request crosses three independent layers between the client’s SSE socket and the GPU that produces its first token. Each layer has different state, runs on different hardware, and recovers from different failure modes. router.html §1 owns L2; this page owns L3 — the per-task scheduler, where the GPU actually lives.

The scheduler is the only layer that touches the GPU, owns HBM, and decides what runs each tick — so it is the only layer with a CPU↔GPU data-movement contract, a KV-cache byte layout, and an async eviction thread. Those three are the spine of every iteration section below.

2. Notation — the only global block is the physics table

There is no shared up-front state dump on this page. Per the minimalism rule, every queue, free-list, table, and index is born in the iteration that first needs it and re-stated in that iteration’s full state snapshot (template part E). The only block carried globally is the one below: immutable physical facts (hardware bandwidths, model dimensions, the per-token cost constants derived from them). These never change across iterations, so stating them once is not a “shared preamble” — they are constants of the machine.

Why state is born per-iteration, not dumped up front. A reader dropping into any single iteration must see the complete state at that iteration without reconstructing it from earlier deltas. Concretely: Iter 1a has no block abstraction at all (KV lives in a per-sequence contiguous slab; free_blocks_*, block tables, and slot_mapping do not exist yet), and ref_count does not exist until a physical block can be referenced by more than one live sequence (a count is meaningless under single ownership). Introducing either “for later” would be dead state a reader has to mentally carry.

2.3  Wait-time formula evolution (the routing signal this page produces)

The scheduler computes the scalar approx_wait_time_ms and (from Iter 1b) publishes it every 100 ms; the Router consumes it (router.html §2.4). This page owns the full per-iteration evolution — how the scheduler computes the scalar as later iterations change what it does. The Router’s consumer (BLCH spill) is unchanged across every stage; only the scalar’s shape moves.

Iter 1 (Iter 1a computes it locally as a self-monitor; Iter 1b publishes it) approx_wait(t) = Σ over queue(t) ∪ running(t) of β(tier_residency_for(r)) where β ∈ { β_HBM ≈ 5 ms, β_DRAM(N), β_SSD(N), β_cold(N) } — each item costs its swap-in/recompute price by WHERE its KV currently lives. Iter 2 (P/D split — one signal per pool, same broker pipeline, separate fields) approx_wait_prefill(t_p) = Σ over queue(t_p) of β_prefill(r) approx_wait_decode(t_d) = max(0, queue(t_d) − free_slots(t_d)) × avg_seq_lifetime / B avg_seq_lifetime = T_out_mean × β_per_decode_tok = 400 × 5.22 ms ≈ 2.09 s B = 1 at Iter 2 (a placeholder raised to B_decode by a later iteration) // avg_seq_lifetime ≠ avg_session_lifetime (150 s, conversation-level) — different concepts. Iter 3 (PagedAttention — layout fix; wait-formula shape UNCHANGED) Same scalar as Iter 2; only kv_cap (reported headroom) grows as paged blocks replace static-max reservation (B_max,static ≈ 18 → B_max,paged ≈ 106). Iter 4 (chunked/packed prefill — prefill scalar's SHAPE changes; consumer unchanged) approx_wait_prefill(t_p) = chunks_ahead(t_p) × β_chunk chunks_ahead = Σ ceil(remaining_unmatched_tokens(r) / C) → the head-of-line max-β_cold spike is GONE; wait tracks fair share of chunk work. Iter 5 (prefix caching — prefill scalar's INPUT shrinks; consumer unchanged) remaining_unmatched_tokens(r) = r.prompt_tokens − longest_local_match(r.chain_hash, hash_index) mean 13.3 K → ~1.85 K (scenario §11.6) → chunks_ahead falls → prefill wait drops. Iter 6 (continuous batching — decode scalar's B placeholder graduates 1 → B_decode) approx_wait_decode(t_d) = max(0, queue − free) × avg_seq_lifetime / B_decode B_decode ≈ 106 (scenario §12.3) → published decode wait drops ~106×. Iter 7 (speculative decoding — decode scalar's denominator gains a speedup factor) approx_wait_decode(t_d) = max(0, queue − free) × avg_seq_lifetime / (B_decode × accept_rate_speedup_factor) accept_rate_speedup_factor ≈ 2.0 (scenario §13.3) → a further ~2× drop. Iter 8–10 no wait-formula change (abuse gate / CP fork / ring-drop are orthogonal to the scalar).

router.html §2.4 repeats Iter 1–4 because those are the stages the Router consumes today; this block is the authoritative full evolution. Each stage’s one-line “why the pattern changes here” lands in that iteration’s section.

3. What the scheduler does — four concurrent roles

One scheduler task runs up to four concurrent loops. At Iter 1a only roles 1 and (the async half of) the eviction machinery exist; role 2 lights up at Iter 1b; roles 3 and 4 arrive at later iterations. Each is a single responsibility; they share state only through the one scheduler thread (the tick loop) plus carefully-split async threads (eviction) — never through locks on the hot path.

The async evictor only ever touches its own enqueue structures (parked-frame set + cudaEvent map) — the split that lets eviction run concurrently with the tick without a lock is the load-bearing idea of template part G, restated every iteration.

4. Iter 0 — naive run-to-completion (narrative anchor)

Bottleneck addressed: n/a — the strawman. One request at a time, no cache, no tiering, TP=2. Iter 1a immediately replaces it. Shown only so the deltas below have a zero-point.

The naive serving loop holds no state between requests: a request arrives, the model prefills its whole prompt, decodes to EOS, the KV is thrown away, the next request starts cold. Concurrency is 1 by construction; there is no scheduler, no free-list, no eviction.

def serve_naive(req: Request) -> Response: # TP=2 narrative baseline kv = allocate_kv(req.prompt_len + MAX_NEW) # fresh cudaMalloc per request; freed at return hidden = prefill(req.prompt_tokens, kv) # one big prefill kernel; β_cold(N) out = [] while not done(out): tok = decode_step(hidden, kv) # one token; β_per_decode_tok each out.append(tok) free_kv(kv) # KV discarded — next turn re-prefills from scratch return Response(out)

Why it fails. Every turn of a multi-turn conversation re-prefills the entire accumulated context (β_cold(13 K) ≈ 730 ms burned per turn even when 80% of the prompt is identical to last turn), and a single task serves one request at a time — nowhere near λ_peak = 100 K req/s. Bottleneck this opens: there is no KV reuse and no concurrency. Iter 1a fixes the first (a real KV cache + HBM/DRAM/SSD tiering, so a returning turn finds its KV warm); a later iteration fixes the second (a fleet + batching).

5. Iter 1a — single-task KV cache + tiering + async eviction

Bottleneck addressed: no KV reuse (Iter 0 discarded KV). New state: the per-sequence slab cache · HBM/DRAM/SSD free-lists · tier-residency map · the async evictor’s parked-frame set + cudaEvent map. No blocks, no slot_mapping, no ref_count yet. Still B=1, still run-to-completion — but now expressed as an engine tick loop with an async watermark evictor underneath. Scenario reference: §5.

  • ① evictor → HBM (async D2H): cudaMemcpyAsync of a victim slab’s live bytes HBM→DRAM on the evict stream; frame parked, cudaEvent recorded. The evictor touches only its own evicting set.
  • ┄┄ ② evict (D2H, dashed): HBM→DRAM→NVMe demotion of idle slabs when free_slabs_hbm < low_watermark.
  • ③ swap-in (H2D, solid): a warm-turn reuse restores a demoted slab to HBM before the kernel — β_DRAM(N) / β_SSD(N), never β_cold.
  • Tick ribbon: the single-thread engine loop; reap at top-of-tick is the only committer of residency flips and frame frees — the reason no lock is needed.

This is the flagship section and the first to carry the full A–H template. Everything below is the complete picture at Iter 1a — nothing is deferred to a delta.

How this differs from Iter 0. Iter 0 had no loop at all — one function ran a request start-to-finish and freed its KV. Iter 1a wraps the same B=1 work in a tick loop so that (i) the slab is retained after EOS for cross-turn reuse instead of freed, and (ii) an async evictor can reclaim HBM underneath without blocking the step. There is still no _pre_kernel_launch_uploads step (no block table to upload dirty rows for) and no slot_mapping (the slab is contiguous, so the kernel takes a base pointer + length, not a page map).

  • _ensure_resident(active) — H2D if the seq’s slab is non-HBM, swap it in over PCIe (DRAM) or GDS (NVMe) before the kernel.

  • _read_sampled_token(tok) — D2H blocking read of the sampled token id (4 B).

C · Per-function deep dive — summary, then code

eviction_thread() — the enqueue half (async)

Runs on: a dedicated async coroutine, not the scheduler thread.
Wakes: every T_SWAP_MS (≈ tick_period ≈ 5 ms; derived in part G).

What it does, step by step:

  1. Read len(free_slabs_hbm); if ≥ low_watermark, return — nothing to do.
  2. Otherwise pick victim slabs: the least-recently-used idle sequences (retained-but-not-active), oldest first, until enough frames are scheduled to lift free back to high_watermark.
  3. For each victim, issue an async cudaMemcpyAsync (HBM→DRAM, no callback) of only its live bytes, lock the frame off the free-list into the evicting set, and record a cudaEvent.

What it does NOT touch, and why: it never writes slab_tier, never returns a frame to free_slabs_hbm, never touches the active sequence. All of those are mutated only by reap_evictions() on the scheduler thread. The split is the whole point: the evictor schedules copies; the scheduler thread commits their results. Because every host-state write lives on one thread, there is no lock — the race a naive single-threaded evict-and-commit would create (freeing a frame whose copy hasn’t landed, or committing while the tick reads the free-list) cannot occur.

async def eviction_thread(): while True: await asyncio.sleep(T_SWAP_MS / 1000) if len(free_slabs_hbm) >= low_watermark: continue # pool healthy — stay armed, do nothing need = high_watermark - len(free_slabs_hbm) for seq in _lru_idle_sequences()[:need]: # oldest retained-but-idle first frame = slab_frame[seq.conv_id] evt = cuda.Event() cuda.memcpy_async(dram_dst(seq), hbm_src(frame), # only LIVE bytes: seq.len × KV_per_tok nbytes=seq.len * KV_per_tok, stream=evict_stream) evt.record(evict_stream) evicting[frame] = (seq.conv_id, evt) # park — touches ONLY the enqueue set

reap_evictions() — the reap half (scheduler thread, top-of-tick)

Runs on: the scheduler thread, first thing every tick.

What it does, step by step:

  1. For each parked frame, poll its cudaEvent; if the copy hasn’t completed, leave it parked.
  2. On completion: flip slab_tier[seq] = DRAM, return the frame to free_slabs_hbm, drop it from evicting.

What it does NOT touch: it never issues a copy (that is the evictor’s job) and never picks victims. It is purely the commit step. Because it runs on the same thread as _admit_next_if_idle() and _ensure_resident(), the free-list is mutated by exactly one writer — no lock, no torn read.

Difference from Iter 0: Iter 0 had no eviction at all (KV was freed synchronously at request end); this is the first appearance of asynchronous, watermark-driven reclamation.

def reap_evictions(): # ① top of every tick done = [] for frame, (conv_id, evt) in evicting.items(): if evt.query(): # non-blocking: has the D2H landed? slab_tier[conv_id] = DRAM # commit residency flip — scheduler thread only free_slabs_hbm.append(frame) # frame now reusable done.append(frame) for frame in done: del evicting[frame]

_admit_next_if_idle() — B=1 admission

Runs on: the scheduler thread, step ②.

Steps:

  1. If a sequence is still active, return (B=1 — only one in flight).
  2. Else pop the head of waiting.
  3. Take a frame from free_slabs_hbm (guaranteed non-empty because the evictor holds low_watermark ≥ 1 free).
  4. One-shot H2D upload of the prompt token IDs.
  5. Set the seq active with tier = HBM.

Does NOT build a block table or slot_mapping — none exist at Iter 1a; the kernel addresses the slab by base pointer + length.

def _admit_next_if_idle(): if active is not None: return # B=1 — one sequence in flight if not waiting: return req = waiting.popleft() frame = free_slabs_hbm.pop() # non-empty by watermark invariant h2d_upload(req.prompt_token_ids) # one-shot; N_new × 4 B slab_frame[req.conv_id] = frame slab_tier[req.conv_id] = HBM set_active(req)

_ensure_resident(active) — swap-in on warm-turn reuse

Runs on: the scheduler thread, step ③ (pre-launch).

Steps:

  1. If the active seq’s slab is already HBM, return.
  2. Otherwise take a free HBM frame and copy the slab in — over PCIe from DRAM (β_DRAM(N)) or over GDS from NVMe (β_SSD(N)) — before the kernel launches, so the kernel always reads HBM.

This is the path that makes cross-turn reuse pay off: a returning turn whose KV was demoted is restored at swap-in cost instead of recomputed at β_cold.

def _ensure_resident(seq): if slab_tier[seq.conv_id] == HBM: return frame = free_slabs_hbm.pop() if slab_tier[seq.conv_id] == DRAM: h2d_copy(hbm=frame, src=dram_slab(seq), nbytes=seq.len*KV_per_tok) # β_DRAM(N), 200 GB/s agg else: # SSD gds_copy(hbm=frame, src=nvme_slab(seq), nbytes=seq.len*KV_per_tok) # β_SSD(N), 7 GB/s slab_frame[seq.conv_id] = frame slab_tier[seq.conv_id] = HBM

_run_step() (prefill or one decode step) and _read_sampled_token() are the standard forward-pass primitives from K0; no Iter-1a-specific behaviour beyond writing K,V in place into the active slab and returning one sampled token.

F · KV-cache layout + byte-address derivation (Iter 1a slab form)

At Iter 1a KV lives in a per-sequence contiguous slab, statically sized to max_reservation_tokens = T_in_mean + T_out_max (vLLM v0 reservation policy: you must reserve for the worst-case generated length because the slab is contiguous and cannot grow). The address of “token t, layer l, KV-head h, K-or-V selector kv_idx, in sequence s” is a pure offset computation — no indirection:

slab_bytes_per_layer = max_seq_len × n_kv_heads × 2 × d_head × bytes_per_elem // the ×2 is K and V = 78 000 × 8 × 2 × 128 × 2 = 320 MB / layer / slab addr(l, s, kv_idx, h, t) = kv_cache[l] // per-layer base pointer + s × slab_bytes_per_layer // which sequence's slab + kv_idx × (n_kv_heads × max_seq_len × d_head × 2 B) // kv_idx=0 → K half, =1 → V half + h × (max_seq_len × d_head × 2 B) // which KV head + t × (d_head × 2 B) // which token

Sizing the slab (the numbers behind the 18).

max_reservation_tokens = T_in_mean + T_out_max = 13 000 + 65 536 ≈ 78 000 tok [G — vLLM v0 policy] reserved_per_slab = max_reservation_tokens × KV_per_tok = 78 000 × 320 KB = 24.96 GB / slab [D] HBM_slab_frames = floor(G_KV / reserved_per_slab) = floor(450 / 24.96) = 18 [D]

TP sharding note. With n_kv_heads = 8 = TP, each GPU physically holds exactly one KV head, so the per-GPU slab is 1/8 of the figure above (40 MB/layer/GPU → 3.2 GB/GPU → 25.6 GB across 8 GPUs ≈ 24.96 GB). The address math above is the logical layout; each rank applies it to its single head.

Why this layout is the thing that gets fixed later. max_seq_len = 78 K but the mean sequence is ~13.7 K tokens — so each slab reserves 24.96 GB to use ~4.2 GB on average. That ~83% reserved-but-unused headroom is the static-over-allocation waste; it caps the task at 18 concurrent retained sequences when the live footprint would fit far more. A later iteration replaces the slab with paged blocks + a block-table indirection, turning the s × slab_bytes_per_layer term into a block_table[s][t // block_size] lookup and reclaiming that headroom.

G · Async watermark eviction — derive BOTH watermarks

The enqueue/reap split (functions in part C) is driven by two watermarks on len(free_slabs_hbm). The victim unit at Iter 1a is a whole-sequence slab. Both watermarks are derived, never guessed, from a drain rate × an allocation rate.

drain side drain_rate = D2H bandwidth, HBM→DRAM = BW_pcie_agg = 8 × 25 GB/s = 200 GB/s [D] (KV sharded 1 head/GPU ⇒ 8 parallel D2H copies) victim_bytes = LIVE bytes per slab at eviction = (T_in_mean + T_out_mean) × KV_per_tok = (13 000 + 400) × 320 KB = 13 400 × 320 KB = 4.288 GB [D] (evictor only targets idle retained slabs — sequence has finished its turn, so all T_out_mean output tokens are already written; no /2 averaging needed) drain_latency = victim_bytes / drain_rate = 4.288 GB / 200 GB/s = 21.4 ms ≈ 21 ms [D] allocation side req_time_mean = prefill_time(T_in_mean, TP=8) + T_out_mean × β_per_decode_tok = 0.73 s + 400 × 5.22 ms = 2.82 s / req [D — scenario §5 Step 4] alloc_rate = per-task admit rate at B=1 = 1 / req_time_mean ≈ 0.355 slab/s [D] (= eviction_rate once HBM is saturated: one evict per admit) the watermark pair (anti-thrash) low_watermark = ceil(drain_latency × alloc_rate) = ceil(0.0214 × 0.355) = 1 slab [D] hysteresis_band = ceil(alloc_rate × T_SWAP_MS) = ceil(0.355 × 5.22 ms) = 1 slab [D] high_watermark = low_watermark + hysteresis_band = 2 slabs [D]

Why a pair, not one threshold. A single threshold re-arms the evictor every tick it sits just under the line — thrash. low is the floor that guarantees the next admit finds a free frame (the async drain refills the pool within drain_latency, faster than allocation drains it). high is where the evictor stops; the gap between them (hysteresis_band) is sized to one tick’s allocation burst so a single admit can’t immediately re-trip low.

Why the band is asymmetric (thin here). The band’s thickness is set by drain speed vs. refill reliability. At a single B=1 task allocation is slow (~0.355 slab/s) and the drain is fast and reliable (21.4 ms/slab over a dedicated stream), so one slab of hysteresis is enough — low=1, high=2 of 18. The formula, not the number, is the deliverable: the same derivation re-evaluates to large watermarks once a task holds many concurrently-decoding sequences (a much higher alloc_rate), which is exactly when the evictor becomes load-bearing rather than near-idle. At Iter 1a it is built and correct but barely engages.

The wake period T_SWAP_MS (the evictor’s poll cadence, not the copy time). T_SWAP_MS is how often the coroutine wakes to check len(free_slabs_hbm) against low_watermark — distinct from drain_latency = 21.4 ms, which is how long one slab’s D2H copy takes once issued. It is bounded on both sides, not guessed:

floor — don’t poll faster than the free-list can change free-list changes at most once per engine tick (one admit/retire per tick at B=1), so T_SWAP_MS ≥ tick_period = 5.22 ms (finer polling = wasted CPU wakeups) [D] ceiling — must refill before the one guaranteed-free frame is consumed with low_watermark = 1, after a trip there is 1 free frame; it is consumed in 1 / alloc_rate = 1 / 0.355 = 2.82 s. The evictor must wake, detect, AND drain within that window: T_SWAP_MS + drain_latency << 2.82 s ⇒ T_SWAP_MS << 2.80 s [D] choice T_SWAP_MS = tick_period ≈ 5 ms — matches the cadence the free-list actually changes; leaves a ~530× margin on the ceiling (5 + 21.4 ms << 2820 ms). [D]

So “a few ms” is tick_period: the evictor observes a quantity that moves at the tick rate, so it samples at the tick rate. The ceiling is slack at B=1 (the pool drains slowly); it becomes the binding constraint at high alloc_rate, the same regime that widens the watermark pair above.

H · Delta from Iter 0 + next bottleneck

How Iter 1a differs from Iter 0. Iter 0 freed KV at request end and had no loop, no tiers, no eviction. Iter 1a (i) retains the slab after EOS so a returning turn reuses warm KV instead of recomputing it (β_DRAM/β_SSD swap-in vs β_cold recompute); (ii) adds the HBM/DRAM/SSD tier hierarchy with explicit swap-in/swap-out; (iii) introduces the async watermark evictor (enqueue/reap split) so HBM is reclaimed off the tick. State is still B=1 and run-to-completion; the layout is still a contiguous slab.

Bottleneck this opens for the next iteration. One task at B=1 sustains ~0.355 req/s — nowhere near λ_peak = 100 K req/s, and there is no Router consuming the approx_wait_time_ms the scheduler now computes locally. The next iteration deploys the Router fleet and the broker, and the scheduler starts publishing that signal — without any change to the GPU tick contract above.

6.4  Bottleneck this opens for the next iteration

The unified pool now scales out and routes by wait-time, but prefill and decode still share one task: prefill is compute-bound (it saturates the SMs) while decode is HBM-BW-bound (it idles the SMs at MBU ≈ 1%), and forcing them onto the same task means neither runs at its own optimum. The next iteration splits the pools — at which point this page’s wait scalar becomes two scalars (one per pool) and the outbound KV stream (role 4) is born.

Why C = 4 K (the chunk-size knob is the design question).

  • Floor — below ~4 K, MFU collapses. Arithmetic intensity ≈ C FLOP/byte; the H100 roofline ridge is F_peak/HBM_BW ≈ 295 FLOP/byte. Below that ridge the kernel is weight-BW-bound and re-pays the 17.5 GB/GPU weight read N/C times for no extra useful compute. Even above 295, achieved MFU keeps ramping and only plateaus near 40% around C ≈ 4 K for a 70 B model on H100.
  • Ceiling — above ~8 K, activation budget + HoL slip re-grow. Per-chunk transient HBM (Q/K/V/O softmax buffers) scales with C and starts crowding the KV cache above ~8 K; and a chunk is the worst-case wait a short request can sit behind, so larger C undoes the HoL bound chunking was designed to give.
  • The chosen value: C = 4 K — smallest fully compute-bound chunk, minimises HoL while staying at peak MFU. Yields β_chunk = prefill_time(4 K) ≈ 190 ms, the unit the scheduler interleaves and the hard bound on worst-case HoL slip.

10. Iter 5 — Prefix caching (Role 2) — and ref_count is finally born

Bottleneck addressed: every warm turn re-prefills a prompt that is mostly identical to the last turn’s. Why ref_count is born now and not at Iter 3. A count is meaningful only when a physical block can be referenced by more than one live sequence. Through Iter 4 every block had exactly one owner, so “is it in use?” was the boolean owning-seq ∈ running. G · Eviction — reuse-recency LRU + the hard ref_count pin

The enqueue/reap split is unchanged; what changes is the victim-selection predicate.

  • Coldness gate: (last_access old ∧ ref_count == 0 ∧ session_idle). All three conditions must hold before a block is even a candidate for eviction.
  • ref_count == 0 is the only hard pin. A block shared by N live sequences cannot be evicted — correctness. Among eligible (ref_count == 0) blocks the evictor ranks by reuse-recency LRU; last_access is refreshed on cache hit/reuse at admission, not on every kernel read (vLLM free-queue re-append).

  • No hard pin, no Router pin control-plane (Change 5). A hot prefix stays resident because it is re-hit — the platform prompt via boot-time warmup-prefill + constant re-hits; a heavy user’s system prompt via CH-stickiness landing them on the same task. 18. Physical deployment topology (Borg-style)

One serving-*-job task = one bare-metal host (per scenario §1.5). Task counts are symbolic and materialise at scenario S6; the SVG shows the comms that matter and the per-task hardware.

  • Solid black — sync request path (LB → Router → prefill).
  • ┄┄ gossip (async) — TaskStatePush/Heartbeat via the broker to every Router.
  • ⟶ cross-task KV (IB) — KVHandoff gRPC stream prefill→decode; KVWriteback/KVPull RDMA to/from the store.

Why these placements. Serving tasks are TP=8 because the AllReduce fits one NVLink domain (~18× faster than crossing the NIC, K20). The store is a separate job (not co-located on serving hosts) because local tiers die with the task while the pool must survive a reroute — and because a pool-wide fan-in to a central store avoids the N² connection cost of every task talking to every other. Broker tasks need a 25 Gbps NIC because the gossip fan-out (serving_task_num × 10 Hz pushes) saturates 1 Gbps (router.html §10 derives this).

20. Takeaways & cross-links

  • Three spines, restated every iteration. The CPU↔GPU contract, the KV byte-layout, and the async enqueue/reap eviction split are the load-bearing structure of the scheduler; each iteration re-states all three in full so any one section reads stand-alone.
  • State is born where it is first needed. No up-front dump: blocks at Iter 3, ref_count at Iter 5 (first true sharing), the draft model at Iter 7, CP ranks at Iter 9. A count is meaningless under single ownership — that is why ref_count waits for prefix sharing.
  • The layout fix precedes the throughput win. PagedAttention (Iter 3) frees the HBM headroom that continuous batching (Iter 6) consumes — B_max 18 → 106. Bundling them would hide the dependency.
  • The Flash Attention chain. FA v2 varlen prefill (Iter 4) → Flash Decoding (Iter 6) → Ring Attention (Iter 9) are one SRAM-tiling base extended three ways.
  • Recompute is always the floor. Every cross-task retrieval (hand-off, pool pull) degrades gracefully to β_cold recompute — RPO=0 on every failure path.
