Redis vs Memcached — single-threaded vs multi-threaded, and why

Both are in-memory key–value (KV) stores; the headline difference is the concurrency model. Redis runs its command-execution loop on one thread; Memcached spreads work across N worker threads. This page shows what each choice buys, what it costs, and the causal chain that links the threading model to each store’s feature set — the “why,” not just the “what.”

Reference / design-choice page · cross-links the broker-job (which uses Redis Streams) · ~12 min

The one-line answer

TL;DR. Redis is single-threaded on purpose: serializing all data access onto one core makes every command atomic for free and removes locks, contention, and cross-core cache traffic — so per-op latency is tiny and predictable. It still serves millions of ops/sec because the bottleneck was never CPU: the work is in-memory (no disk wait) and I/O is multiplexed (one thread watches thousands of sockets via epoll). Memcached is multi-threaded on purpose: it keeps values opaque and operations trivial, so its lock critical-sections stay short enough that many cores can share the hash table — letting one process saturate a fat machine without sharding. The threading model and the feature set are the same decision viewed from two ends.

The myth this page kills: “single-threaded” sounds like “slow.” It is the opposite. Single-threaded execution is why Redis is fast and predictable — the thread never waits on a lock, never gets descheduled mid-critical-section, and keeps the hot data structures pinned in one core’s L1/L2 cache. The cost is a single-core ceiling per process, not a per-op slowdown.

1. The two threading models, precisely

The distinction is narrow and exact. It is not “Redis can’t use threads” — modern Redis (v6+) offloads network read/write and some background jobs to helper threads. The invariant is about data access:

Redis — serialized command loop 1 core for data

  • One event-loop thread pulls ready commands off an epoll/kqueue queue and executes them one at a time, to completion.
  • All reads and writes of the actual data structures (strings, hashes, sorted sets, streams) happen on this one thread.
  • v6+ may use extra threads to parse/copy bytes off sockets, but the command itself still runs on the single data thread — so the atomicity guarantee holds.

Memcached — N symmetric workers all cores for data

  • Several worker threads, typically one per core, each run their own event loop and accept connections.
  • Every worker reads and writes the same shared hash table, LRU lists, and slab allocator.
  • To stay correct they take fine-grained locks (per-bucket item locks, LRU-lock, slab-lock) around each access — held only for the microseconds of a trivial get/set.

Side-by-side: where the data lives and who touches it

  • Redis (left): many sockets → one epoll loop → one core executes every command serially; the data lives in that core’s cache untouched by anyone else — solid arrows, no red.
  • Memcached (right): sockets spread across 4 worker cores; all four reach into one shared region (red arrows) and take locks — the same cache lines bounce between cores (coherence traffic).

2. Why one thread still serves millions of ops/sec

The bottleneck for a KV store was never raw CPU. A GET touches a few cache lines and returns — nanoseconds of compute. The time goes to waiting on the network and (for disk stores) the disk. Redis removes both waits, so one core’s worth of compute is plenty:

  1. I/O is multiplexed. epoll/kqueue/select let one thread watch thousands of sockets and wake only for the ones with data ready — it never blocks on any single connection, so “one thread” is not “one client at a time.”
  2. Everything is in memory. No disk seek on the hot path → no millisecond stalls to hide, so there’s little idle time that a second data thread could usefully fill.
  3. Commands are intentionally small and atomic. Each command does a tiny, bounded amount of work, so the serial queue drains fast — head-of-line blocking is rare because ops are designed to be O(1)/O(log N).
  4. The kernel and NIC do the heavy lifting. Packet handling, interrupt coalescing, and TCP live below the application thread.
  5. Modern Redis offloads I/O, not data. v6+ threaded I/O parses and writes socket bytes on helper threads, leaving the single data thread to do only command execution — keeping atomicity while relieving the network-syscall cost.
Where the single thread does bite: head-of-line blocking. Because commands run serially, one expensive command — KEYS *, a big SORT, an O(N) op over a huge collection, a heavy Lua script — stalls every other client until it finishes. The fix is operational, not architectural: avoid O(N) commands on large keys, use SCAN over KEYS, and keep Lua short.

2.1  Same resources — can Memcached out-throughput Redis?

Short answer: only when you compare one process to one process. The honest answer hinges on the deployment unit you hold equal, not on the engine — because “same resources” does not force you to run a single Redis process.

What you hold equalWinner for simple get/setWhy
One multi-core box, one process eachMemcached (several×)One Redis process executes commands on a single core and leaves the rest idle for data ops; one Memcached process spreads work across all cores and scales close to linearly with core count.
One box, but you may fill the cores≈ ComparableRun N Redis shards (Redis Cluster), one per core, consistent-hashing the keyspace → aggregate Redis throughput catches up. Memcached may keep a modest edge (no cross-shard hop, slightly lower per-op overhead), but it’s close and benchmark-dependent.

And four things flip even the one-process comparison:

  • Pipelining → Redis. Batching many commands per round-trip lifts Redis’s per-core throughput, often past a non-pipelined multi-threaded Memcached — the network round-trip, not the core, was the limit.
  • Hot keys / write-heavy → Redis (per shard). Memcached’s shared structures suffer lock contention + cache-line bouncing under a hot key; Redis’s serial loop is immune to contention (still capped at one core/shard).
  • Large values → tie. Both go network/memory-bound; the threading difference washes out.
  • Rich ops → not a throughput question. Memcached can’t do ZADD / INCR-on-a-structure at all, so the comparison only ever holds for the trivial get/set subset.
Net. Memcached out-throughputs Redis only in the narrow case of one process vs one process on multi-core hardware for opaque get/set. Shard Redis to fill the same cores and they’re comparable; turn on pipelining or use any non-trivial operation and it tilts back to Redis. The single-thread “limit” is a per-process ceiling you engineer around with sharding — not a per-op tax.

3. The cache-coherence “why” — the hardware reason single-threaded is predictable

This is the part most comparisons miss. The difference isn’t only locks — it’s what the CPU’s cache-coherence protocol (MESI) has to do when multiple cores touch the same memory.

Redis — zero coherence traffic for its data

  • All data access stays on one core, so the hot structures live in that core’s L1/L2 and never get invalidated by another core.
  • No core ever has to ask another to surrender a cache line — no MESI invalidations, no false sharing.
  • Consequence: per-op latency is tiny and low-variance — a big reason Redis’s single-thread performance is so predictable.

Memcached — coherence traffic caps scaling

  • Several cores hammer shared structures — hash-table buckets, LRU-list heads, global counters — that often sit on the same cache lines.
  • Each write by one core invalidates the line in the others’ caches, forcing a cross-core transfer — cache-line bouncing.
  • Consequence: beyond some core count the coherence traffic, not the lock logic itself, limits how well one process scales — especially under hot-key or write-heavy load.
False sharing, defined (first use): two unrelated variables that happen to land on the same 64-byte cache line. Even though no two cores touch the same variable, writing either one invalidates the whole line in every other core’s cache — so the cores fight over a line for no logical reason. Memcached mitigates this with padding and per-core/per-bucket structures; Redis sidesteps it entirely by never sharing.

4. Pros and cons, both sides

Redis — single-threaded

Pros

  • Free atomicity. Every command runs to completion before the next starts, so INCR, LPUSH, MULTI/EXEC transactions, and Lua scripts are atomic with zero locking — no race conditions on shared structures, by construction.
  • No synchronization cost. No locks, no mutex contention, no thread context-switching for data access — simpler code and predictable per-op latency.
  • Cache-friendly. One core keeps the hot structures in its L1/L2; no cross-core coherence traffic, no false sharing.

Cons

  • One-core ceiling per process. Command execution can’t use multiple cores, so a 64-core box running one Redis process leaves ~63 cores idle for data ops (v6+ threads only the network I/O, not data access).
  • Head-of-line blocking. One slow command (KEYS *, big SORT, O(N) on a huge collection, heavy Lua) stalls every other client until it finishes.
  • Multi-core needs sharding. To use a big machine you run several Redis processes and split the keyspace (Redis Cluster), adding cross-shard complexity.

Memcached — multi-threaded

Pros

  • Full machine utilization. One process saturates all cores, so it scales vertically on a fat machine without sharding.
  • Higher per-process throughput for simple get/set on multi-core hardware.
  • Better request isolation. A slow request on one thread doesn’t fully block the others — no global head-of-line stall.

Cons

  • Locking required. The shared hash table, LRU lists, and slab allocator need synchronization (fine-grained item/bucket locks), adding complexity and contention under hot-key or write-heavy load.
  • Cache-line bouncing. Shared structures touched by multiple cores generate coherence traffic that caps scaling.
  • Thin feature set. Keeping lock hold-times short forces values to stay opaque and ops trivial — no rich server-side data structures (see §5).

5. The deeper tradeoff — the threading model causes the feature set

The two choices aren’t independent. Each store’s concurrency model and its feature set are locked together by one constraint: how long is a critical section allowed to be?

Redis chose single-threaded so that it could offer rich atomic operations.

  • A serial loop makes any command atomic regardless of how many steps it takes — so Redis can ship sorted sets, streams, HINCRBY, set intersections, and Lua, all atomic, all for free.
  • Doing those multi-step ops in a multi-threaded store would mean holding a lock across the whole operation — long critical sections that would crush the concurrency benefit.

Memcached chose multi-threaded and therefore kept values opaque and ops trivial.

  • Many cores can share a structure only if each lock is held for a microsecond — which is true exactly when the op is a flat get/set on an opaque blob.
  • Adding rich server-side operations would lengthen lock hold-times and reintroduce the contention multi-threading was meant to avoid — so Memcached deliberately doesn’t have them.

Read it as one decision, two ends: “I want rich atomic data structures” ⇒ serialize on one thread ⇒ accept a per-process core ceiling. “I want to saturate every core with one process” ⇒ keep critical sections tiny ⇒ accept opaque values and trivial ops.

Rule of thumb. If your need is literally “memoize opaque blobs by key, evict by LRU, saturate one big box” — Memcached is a clean fit. The moment you want a counter, a leaderboard, a queue, a TTL set, pub/sub, or persistence, you want Redis — and the single thread is a feature, not a tax.

7. In a system-design interview — what to actually say

TL;DR for the room. The interviewer almost never wants a feature checklist — they want to hear you reason from the workload to the choice. Default to Redis: it does everything Memcached does plus data structures, persistence, and pub/sub, so it’s the safe pick unless you can name a specific reason for Memcached. Then justify in one breath: “Redis, because I need X (counter / sorted set / TTL / pub/sub / durability), and the single-thread ceiling doesn’t bind because my per-shard QPS is Y.”

The 10-second decision rule

  1. Do you need anything beyond get/set on opaque blobs? — a counter, leaderboard, queue, set, TTL index, pub/sub, rate-limiter, or persistence? ⇒ Redis. This is ~90% of interview cases.
  2. Is it purely a multi-core LRU cache for opaque blobs, throughput-maximal on one fat box, no persistence? ⇒ Memcached is the cleaner, lower-overhead answer — and naming it shows you know Redis isn’t the only hammer.
  3. Either way, say why the loser loses. “Not Memcached — I need atomic INCR for the rate-limiter.” / “Not Redis — I just need a dumb sharded blob cache and Memcached’s multi-threading saturates the box without me running N processes.”

The follow-up that earns the senior signal: when you say “Redis,” pre-empt the obvious objection. Volunteer “and to scale past one core I shard with Redis Cluster — consistent-hash the keyspace across N primaries, each single-threaded; I avoid O(N) commands like KEYS * so no shard head-of-line-blocks.” That single sentence shows you understand the cost (one-core ceiling), the fix (sharding), and the operational footgun (head-of-line blocking) — not just the feature list.
Traps to avoid saying:
  • “Redis is slower because it’s single-threaded.” — Wrong direction; it’s fast because it’s single-threaded (§2). Per-process ceiling, not per-op slowdown.
  • “Memcached is just old Redis.” — They’re different design points, not versions of each other.
  • “I’ll use Redis for everything.” with no reason — defensible as a default, but say the workload reason; a flat “Redis always” reads as cargo-culting.
  • Reaching for Memcached to look clever when you actually need a counter or TTL or pub/sub — that’s choosing the weaker tool to seem contrarian.

8. Why this scenario’s broker uses Redis (not Memcached)

The broker-job in this scenario runs Redis Streams as the cluster gossip bus — it fans ring-version, per-task wait-time, and the membership-filter delta out to every Router (kv_store §6). That choice falls straight out of this comparison:

  • Streams is a data structure, not a blob. The consumer-group fan-out (“every Router consumes the same delta exactly once”) is a first-class atomic Redis op — precisely the rich-operation class that single-threaded execution makes free. Memcached has no such primitive.
  • The workload is tiny + bursty, not core-bound. ~100 K low-rate gossip msg/s at ~93 MB/s aggregate (scenario §Iter 1b) is trivial for one in-memory thread — the single-core ceiling is irrelevant here, so Redis pays none of its main cost.
  • Atomic, predictable latency matters for control-plane gossip. The zero-coherence, low-variance profile from §3 is exactly what you want on the path that keeps every Router’s ring view consistent.

So the broker is a textbook case of the §5 logic: the need is “rich atomic fan-out op on a small message stream,” which selects Redis, and the single-thread ceiling never binds because the traffic is light.
