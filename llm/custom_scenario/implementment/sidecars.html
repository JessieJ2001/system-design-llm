Sidecars (S1–S3)

Optional, interview-depth add-ons to the distributed KV cache — the predictor-free, filter-free main line ships without any of them. Each is described once here and linked from the iteration it would attach to on distributed_kv_cache.html. None is load-bearing; none is inherited — every iteration’s running state, numbers, and SLOs are derived as if all three are absent.

Why these are sidecars (and where each attaches)

The main line ships predictor-free: plain LRU per tier + a length-based pool admit gate + a direct CH(conv_id, pool_ring) probe on local miss. The three sidecars below are add-ons to raise only if an interviewer probes — each trades a little complexity for a marginal win that the main line does not need.

§S1 — local “is it in the pool?” membership summary (optional)

Attaches at Iter 3b DC4 (the lookup-path design choice) · DC5 (is it over-engineering) · DC6 (Cuckoo scale).

A compact local summary of “is conv_id in the pool?”, probed before any RDMA. It is optional — the main line carries no filter.

  • Why optional at all. The no-filter path — a direct CH(conv_id, pool_ring) probe on local miss (Iter 3b DC4/DC5) — is fully supportable on its own: the per-task probe load is ≤ ~6 250 /s ≈ 0.06% of a CPU core (derived below), and a wasted RTT on an absent key is noise against a hundreds-of-ms recompute.
  • When a summary earns its keep. Only under correlated consult bursts — a rolling deploy, a large autoscale step, or a fabric blip mass-invalidates local residency → a synchronized wave of absent probes onto the scarce store tier. A local summary absorbs the absent answers locally instead of as an RDMA-probe storm.

Why “affordable” is a calculation, not an assertion — and why the flagged-model pool size (its α-uncertainty) can’t flip it:

# Givens: λ_peak = 100 000 req/s [§1.2]; CPU hashmap ≈ 10 M ops/s [G]; # kv_store_job_task_num = 16 (pre-DC7, storage-bound — post-DC7 only GROWS, see (2)). # (1) Conservative per-task ceiling — each request probes the pool AT MOST once on a local miss: probes/s/store-task ≤ λ_peak / kv_store_job_task_num = 100 000 / 16 = 6 250 /s/task [D] CPU cost = 6 250 / 10 000 000 ≈ 0.06% of one core’s hashmap throughput [D] # (16 = the conservative fewest-tasks bound; the flow-derived ~36 store tasks (§9b.4) gives ~2 800/s ≈ 0.03% — even safer.) # (2) Structurally bounded — this is what makes it robust to the flagged-model pool size (the α-uncertainty in W_pool): kv_store_job_task_num = max( storage-sized , QPS-sized ), QPS-sized = ⌈ pool_QPS / per_kv_store_task_QPS × headroom ⌉ # The QPS term EXISTS to cap each store task's request rate ≤ per_kv_store_task_QPS BY CONSTRUCTION, # so per-task probe load stays affordable AS THE POOL GROWS — whatever the pool size settles to. # (3) Margin: even a 10× consult surge → 62 500 /s/task ≈ 0.6% CPU — still 2+ orders below saturation. ⇒ the direct probe is affordable by ~3 orders of magnitude, so the α-uncertainty in the pool size CANNOT flip the conclusion — the filter is optional burst-insurance, not a requirement. [D]

The latency the filter would save is noise too: it removes one wasted RDMA RTT (~tens of µs) ahead of a hundreds-of-ms recompute. So its only real value is absorbing correlated absent-probe bursts — which is precisely why it is a sidecar, not the main line.

Three options, not two (cheapest-correct first):

  • (i) No filter — the MAIN LINE. The direct-probe numbers above carry it; nothing to build, gossip, or keep consistent.
  • (ii) Exact membership set of conv_ids — PREFERRED if a summary is added. No false-positive rate, trivial delete, no load-factor cliff, at ~7× the memory of (iii) — and that memory (a few MB/task, re-derive off the §9b.4 working set) is noise against a CPU task’s tens of GB of RAM.
  • (iii) Cuckoo filter — only if the exact set’s memory actually bites (~7× smaller; supports deletes unlike Bloom, which a victim cache needs since pool entries are constantly evicted). It is, however, the only probabilistic / load-factor-cliff component in the whole design — the reason it is not the default. Full Cuckoo scaling analysis: Iter 3b DC4/DC6.

Pick order. Prefer the exact set (ii) — it sidesteps the FPR and the load-factor cliff and the memory is noise; reach for Cuckoo (iii) only if that memory genuinely bites at consumer scale. This supersedes the scenario’s earlier “Cuckoo required” DC4. Not inherited: every iteration’s running numbers assume the no-filter direct probe.

§S2 — P(continue)-predicted eviction / admission (LPC — Learned Prefix Caching)

Attaches at Iter 1a DC5/DC6 (an optional learned ordering over the session-tail LRU) and Iter 3b DC2 (an optional EV refinement over the length-gate admission).

A learned continuation-probability estimator that orders session-tail eviction (and refines pool admission) better than raw LRU + the length gate. It is the home for the P(continue) machinery retired from the Iter 3b main line. Three parts, all off the hot path:

  • 1. Async scoring at insert. The predictor runs in parallel to prefill on the serving task (prefill = 100s of ms; CPU prediction = 10–20 ms → score ready before decode finishes). Scored once, at insert: when a sequence’s decode completes, its KV blocks enter a prefix-cache min-heap keyed by predicted P(continue).
  • 2. Async background re-scoring sweep. At a fixed interval (LPC default ~10 s) a sweep re-evaluates P(continue) via a decay function and reorganises the heap. Optional extension: proactively evict blocks whose decayed score falls below a threshold, freeing memory before the cache fills.
  • 3. Predictor-down fallback → LRU. Keep two structures — a plain LRU linked list and the LPC min-heap. All blocks labelled → evict from the min-heap low→high P(continue); half-labelled → evict heap entries scoring < 50% first, else the LRU tail; predictor down (long) → the coordinator migrates heap entries into the LRU list, degrading gracefully to pure LRU.

It also supplies the popularity signal for shared-prefix blocks — which the main line already covers via the Iter 5 pin-list, so this is strictly an upgrade, never a dependency. (One adaptive extension named on the scenario page: folding observed latency / effective BW / queue depth into an EWMA to drive a live recompute-vs-pull crossover — Iter 3b §9b.3.2.)

§S3 — retention_TTL cross-tier privacy / retention sweep

Attaches at Iter 1a DC5 (where TTL-as-a-cache-policy is rejected — its one real use is this privacy sweep) and to the B2C data-retention requirement.

  • A privacy control, NOT a cache-performance policy. It bounds how long a user’s conversation KV may persist anywhere it is stored — local DRAM and the remote pool (front + NVMe back). A low-rate background sweep evicts blocks whose age (or owning conversation’s idle time) exceeds the retention window, regardless of capacity pressure.
  • Why a sidecar. It does nothing for wear, capacity, or correctness — the cache runs fine on pure capacity-LRU without it (KV is immutable, so there is no staleness for a TTL to fix). It exists only where a B2C data-retention / compliance requirement demands a hard upper bound on KV persistence.
  • Granularity. Cleanest at conversation-lifecycle granularity (evict-on-session-end + a max-age cap), not a blind per-block timer. If enabled it also shortens mean residency — most visibly the pool working set Wpool (the D5 sizing).
