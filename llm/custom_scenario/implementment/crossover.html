Family A — transfer vs recompute (the β crossovers)

One question, asked three ways: is it cheaper to move KV bytes in from a slower tier, or to recompute them with a fresh prefill? Set the transfer cost equal to the re-prefill cost and solve.

A1 · Tier swap-in vs re-prefill — the length crossover N*

§6.5  the central crossover — everything else in Family A is a variation on it.

Q: below what context length N does re-prefilling beat swapping the KV in from tier t?

A tier beats re-prefill when its linear transfer cost βt(N) is below the linear + quadratic re-prefill cost βcold(N). Set them equal and divide by N:

# Set tier swap-in = re-prefill, then solve for N: N × KVper_tok / BWt = ( 2·P·N + 4·n_layers·N²·d_model ) / Fsustained N* = ( Fsustained × KVper_tok / BWt − 2·P ) / ( 4 × n_layers × d_model ) # Constants (Llama-3-70B + TP=8): KVper_tok = 3.2·10⁵ B [D — §4.2] 2·P = 1.4·10¹¹ FLOP/tok [G — K1 §3.3] 4×80×8 192 = 2.62·10⁶ Fsustained = 8 × 989·10¹² × 0.40 = 3.17·10¹⁵ FLOP/s [D — §1.5]

Substituting the two non-HBM tiers:

  • SSD vs cold (BWt = 7 GB/s): N* = (3.17e15 × 3.2e5 / 7e9 − 1.4e11) / 2.62e6 = (1.45e11 − 1.4e11) / 2.62e6 ≈ 1 870 tok. Below ~1.9 K, re-prefill is cheaper than an SSD swap-in.
  • DRAM vs cold (BWt = 25 GB/s): numerator goes negative — 4.06e10 − 1.4e11 = −9.94e10. No positive crossover: DRAM beats re-prefill at every N, because the linear prefill term alone already exceeds the DRAM swap-in time.

Decides: the on-miss tier ladder — walk HBM → DRAM → SSD → recompute and stop at the first hit. DRAM always wins; SSD wins for any realistic conv (N ≥ ~1.9 K); only sub-2 K fragments are cheaper to recompute.

Why DRAM’s lead widens with N. Both βDRAM and βSSD are linear in N; βcold carries the quadratic attention term that dominates above ~49 K (that’s crossover B1). At 256 K, cold is 5.6× SSD and 20× DRAM; at 1 M context the gap is ~68× — which is why deeper SSD isn’t the p99.9 answer, context parallelism is.

A2 · Tier vs re-prefill — the bandwidth break-even BW*

§6.5 · §9b.4  the same algebra read as “which tiers have no crossover”.

Q: how fast must a path be to beat recompute at every length (no positive N*)?

N* goes negative exactly when the per-token transfer cost drops below the per-token linear recompute cost. Set those equal:

KVper_tok / BW* = 2·P / Fsustained BW* = KVper_tok × Fsustained / 2·P = 3.2e5 × 3.17e15 / 1.4e11 ≈ 7.2 GB/s [D]
  • RDMA front (25 GB/s) and the 4-SSD stripe (25 GB/s) clear it → pull always beats recompute.
  • single-NVMe (7 GB/s) sits right on it → it only ties recompute at the 8 K median (366 vs 407 ms); that’s why its finite crossover (A1, A4) lands at ~1.9 K rather than at zero.

Decides: which pool serving paths are “always worth it” vs which need the length gate — the RDMA front needs no gate; the NVMe back does.

A3 · The crossover’s α-sensitivity — restoring the fixed overhead

§6.5  why the gate doesn’t move even though the tail says it should.

Q: A1/A2 drop the fixed per-transfer overhead α. How much does N* move once you put it back?

The true cost of the transfer (swapping/pulling the KV cache in, the alternative to recomputing it) is α + β·n — the standard latency–bandwidth model: α = fixed per-transfer overhead paid once regardless of size (setup / first-byte / queueing), β = the per-token transfer cost = KVper_tok/BW (the slope — latency each extra token of history adds), and n = N tokens. So β·n = N·KVper_tok/BW is exactly the size term βt(N); A1/A2 kept only that slope and dropped the α intercept. At the crossover the two linear terms nearly cancel (recompute is only ~3% cheaper per token than transfer), so a small α moves N* a lot. Solving the quadratic a·N² + b·N − α = 0 (where b’s −KVper_tok/BW term is −β):

# Restore α: set recompute_cost = transfer_cost, then move all to one side. recompute = (2·P·N + 4·n_layers·N²·d_model) / Fsustained # linear + quadratic transfer = α + (KVper_tok/BW)·N # intercept + slope(β) # recompute − transfer = 0, grouped by powers of N: (4·n_layers·d_model/Fsustained)·N² + (2·P/Fsustained − KVper_tok/BW)·N − α = 0 \____________ a ____________/ \______________ b ______________/ a = 4·n_layers·d_model / Fsustained ≈ 8.3e−10 # from the N² attention term b = 2·P/Fsustained − KVper_tok/BW ≈ −1.55e−6 # linear recompute − linear transfer; <0 ⇒ a positive root exists # Positive root: N* = ( −b + √(b² + 4·a·α) ) / (2·a) α = 0 → constant gone ⇒ N(aN+b)=0 ⇒ N* = −b/a ≈ 1.9 K (identical to the A1 formula) α = 1 ms → N* ≈ 2.4 K α = 5 ms → N* ≈ 3.6 K

Why A1/A2 drop α in the first place. (1) On the tiers they cover — local swap-ins over PCIe/GDS — α (setup/first-byte) is µs against a β·n of ~87 ms at the crossover, a <0.1% term: the closed form is cleaner without it and N* barely moves. (2) It confines α to A3, the single path where it is large — the remote-pool pull’s p99 PFC stall (α≈5 ms) — so A1 stays the clean slope-vs-curve crossover and A3 owns the sensitivity. (The twist, above: <0.1% in magnitude still swings N* 1.9 K → 3.6 K, because the two linear terms nearly cancel.)

Local swap-in α ≈ µs (negligible vs the ~87 ms β·n at 1.9 K). The remote-pool pull is µs at median (RDMA RTT same-row/same-DC ≈ 1–10 µs [G — NEW_BOE] + CH-probe + index touch) but ~ms at p99: when a switch buffer fills, RoCE’s PFC (Priority Flow Control) sends PAUSE frames upstream (each pause is tens of µs, but cascading pauses / head-of-line blocking compound), and on a drop the go-back-N retransmit RTO adds another ~ms. That inflates the pull’s fixed overhead from µs to α ≈ 5 ms [D — order-of-magnitude p99 estimate, not a measured constant], which — via the α-hypersensitivity above — pushes the break-even length from 1.9 K to ~3.6 K.

The exact 5 ms is not load-bearing. Any ms-scale tail α (3 ms or 7 ms) lands the crossover in the same ~3–4 K band, and the decision is the same either way (below). The number is here to illustrate the α-sensitivity, not to pin a threshold: the crossover is not a constant — 1.9 K on the common fast path, widening to ~3.6 K only on the rare p99 PFC tail.

Decides: the pool admit gate uses the median-α crossover (~1.9 K), not a widened ~3.6 K gate. The p99-α tail is absorbed by an abandon-to-recompute fallback (a pull that blows its TTFT budget is dropped for local recompute), not by changing the threshold.

A4 · Recompute vs remote-pool pull — the write-back length gate

§9b.4 · length-gate [param]  A1 reused for the cross-task store.

Q: a conv is being evicted — is it worth writing back to the pool so a later turn can pull it instead of recomputing?

A pull is just another tier read, so the crossover is the same algebra as A1/A2 with BWpull in place of the tier BW:

  • Bandwidth break-even = the A2 number, BW* ≈ 7.2 GB/s. The RDMA front (25 GB/s) clears it → pull always wins; single-NVMe sits on it.
  • Length crossover (single-NVMe back) = the A1 number, N* ≈ 1 870 tok — pull wins for any conv history > ~1.9 K, i.e. essentially every multi-turn conv.
  • Contention-aware form: ρfabric ∈ (0,1] is the fraction of the nominal 25 GB/s the shared fabric actually delivers (contention, PFC pauses, retransmits), so the effective pull BW is ρfabric × 25 GB/s. There is no finite crossover as long as that still clears the A2 break-even BW* ≈ 7.2 GB/s — i.e. ρfabric × 25 ≥ 7.2 ⇒ ρfabric ≥ 7.2/25 ≈ 0.29. So the 0.29 is not an assumption but the derated-bandwidth point where the A2 result starts to fail: above 29%-of-nominal the effective pull BW still beats recompute at all lengths; below it the crossover is finite and ≥ 1.9 K — so ~1.9 K is the robust floor (worst case, fabric >71% saturated), not a universal constant.
# Why the conservative (slow-tier) crossover is the gate: recompute @256 K = βcold(256 K) ≈ 65 s vs pull @25 GB/s ≈ 3.3 s [D] write-through every turn = λavg × 400 tok × KVper_tok = 30 000 × 128 MB ≈ 3.84 TB/s of mostly-never-read ingest [D] → so: write-BACK, gated at the ~1.9 K SSD crossover, not write-through.

Decides: the predictor-free admission gate — write a conv back only when (a) local residency failed and (b) cumulative context > ~1.9 K. The dominant argument is FLOPs offload (recompute burns the scarce prefill roofline; a pull burns cheap I/O), so with the RDMA front there is effectively no crossover — pull always wins for cross-turn reuse.

Family B — where a quadratic term overtakes a linear one

Transformer compute has a term linear in sequence length (weights) and a term quadratic in it (attention). These crossovers are where the quadratic catches up — the points past which “just make N bigger” stops being cheap.

B1 · Prefill — attention vs linear term (s*)

§4.1 · [G — K1 §3.3]

Q: above what prompt length does prefill stop being ~linear in N?

# Set the two prefill_FLOPs terms equal: 4·n_layers·N²·d_model = 2·P·N with P = 2·n_layers·d_model·... s* = 6 × d_model = 6 × 8 192 ≈ 49 K tok [G — K1 §3.3]

Below ~49 K, prefill is ~linear (weights dominate). The crossover is already visible in the §4.1 table: at 32 K the attention term is ~60% of the linear term; by 256 K it is ~5× the linear term; at 1 M context, βcold(1 M) ≈ 871 s — >2 000× the 400 ms TTFT SLO.

Why no single TP=8 task can serve very long context. Two compounding reasons — one about compute time, one about why TP can’t buy it back:

  • The quadratic term dominates at long N. At 1 M the attention FLOPs (4·n_layers·N²·d_model = 2.62·10¹⁸) are ~19× the weight FLOPs (2·P·N = 1.4·10¹⁷); the 871 s prefill is essentially all attention. The first token cannot stream until that whole prefill finishes — >2 000× past the 400 ms TTFT budget. It is a 3-orders-of-magnitude gap, not a tuning miss.
  • More TP doesn’t rescue it. Fsustained already folds in 8 GPUs. Raising TP further (a) adds an all-reduce per prefill step that, past ~8 ranks, grows faster than the compute it saves — TP scaling flattens; and (b) splits the weight axis, not the sequence axis — every rank still holds the full 1 M-token sequence and computes the full N² attention. The quadratic term is untouched.
  • So the fix targets the sequence axis: context parallelism (Iter 9). Partition the 1 M tokens across CP ranks (Ring Attention) so each rank attends over a slice and the N² work is parallelized across tasks instead of serialized inside one.

Decides: why no single TP=8 task can serve very long context — it forces context parallelism (Iter 9) to partition the sequence dimension. It is also why DRAM/SSD widen their lead over recompute at large N (crossover A1).

B2 · Decode — attention vs weights term

§5.4 · [G — K1 §3.5]

Q: at what context does per-step decode attention FLOPs equal the weights FLOPs?

Decode FLOPs over Tout tokens carry an attention term linear in Tin and quadratic in Tout. The attention-vs-weights ratio is (2·n_layers·d_model/P)·(Tin + (Tout−1)/2); setting it to 1:

# Per decode step at context c (B=1): two FLOP terms, MAC = 2 FLOPs throughout. WEIGHTS = 2·P per token → over Tout tokens: 2·P·Tout ATTN = 4·n_layers·c·d_model per step (1 query × c keys; QKᵀ + P·V = two matmuls; c = Tin+k at step k) # Sum attention over the Tout generated tokens, k = 0 .. Tout−1 (at step k the context is c = Tin+k): Σk=0Tout−1 (Tin + k) = Σ Tin + Σ k # split the constant and the index = Tout·Tin + (0 + 1 + … + (Tout−1)) # Tin added once per step (Tout steps) = Tout·Tin + Tout(Tout−1)/2 # Gauss: 0+…+m = m(m+1)/2, m = Tout−1 # — first term LINEAR in Tin (every output token attends over the prompt); # second term QUADRATIC in Tout (output tokens attend over the growing generated tail). ATTNtotal = 4·n_layers·d_model · [ Tout·Tin + Tout(Tout−1)/2 ] = 4·n_layers·d_model·Tout · [ Tin + (Tout−1)/2 ] # factor Tout back out # Ratio — the (…)·Tout cancels; the 4 over 2 leaves a factor 2: ATTNtotal / WEIGHTStotal = (2·n_layers·d_model / P) · ( Tin + (Tout−1)/2 )

Setting that ratio to 1 and approximating (Tout−1)/2 ≈ Tout/2:

Tin + Tout/2 ≈ P / (2·n_layers · d_model) = 70·10⁹ / (2 × 80 × 8 192) ≈ 53 K [D]

Tin + Tout/2 is the average context length a decode query attends over, across the whole generation — context grows one token per step from Tin (first output token) to Tin+Tout (last), averaging Tin+(Tout−1)/2 ≈ Tin+Tout/2. So the crossover reads: when the mean cached-KV depth exceeds ~53 K, attention FLOPs overtake weight FLOPs over the generation (equivalently, per step, when the instantaneous context c > 53 K).

The workload mean (Tin=13 K, Tout=400) sits at ~25% of this crossover; Tout p99 = 3 K at ~27%; the hard-cap Tout=65 K at ~85%. So decode is weights/HBM-bandwidth-bound across the whole realistic workload — the attention term never overtakes the weights term on the tick (the rare 65 K-output tail gets close, but stays under).

Why factor 4 — and why this is 53 K, not 107 K. Decode attention is two matmuls (QKᵀ then P·V), each a multiply-add (MAC = 2 FLOPs), so the per-step term is 4·n_layers·c·d_model — the same MAC=2 convention as the weight term 2·P (and as K1 §3.5). The trap: counting attention at MAC=1 (2·n_layers·c·d_model) while keeping weights at MAC=2 is a mixed convention — it mis-lands the crossover at 107 K. Count both at MAC=2 and it is 53 K. (Same trap, same fix as the decode batch crossover B* = 295 — the 2·P cancels, so AI = B not 2·B; below.) The distributed §5.4 time estimate is dominated by Load regardless: decode is load-bound, so the ~24%-of-weights attention compute never sets the tick.
Same crossover with or without the KV cache — caching moves the cost, not the crossover. First, c = 53 K is the full context depth and necessarily includes the already-cached tokens — the new query attends back over all prior tokens (that is what attention is), so you can’t exclude them from c. The cache changes the work over those tokens, not how many there are, and it does so asymmetrically: it excludes the cached tokens from the weight pass (only the 1 new token is projected ⇒ weights drop from 2·P·c to 2·P) but cannot exclude them from the attention read (the new query still dots against all c cached keys ⇒ attention stays 4·n_layers·c·d_model, still linear in the full cache). Without a cache, a decode step re-runs a full prefill over the whole c-token context: weights = 2·P·c (all c tokens, not 1) and attention = 4·n_layers·c²·d_model (c queries × c keys). So the cache divides both by c — weights → 2·P, attention → 4·n_layers·c·d_model — and the attention/weights ratio (2·n_layers·c·d_model / P) is unchanged, so the crossover stays put. It is exactly the prefill crossover B1, c* = P/(2·n_layers·d_model) — 53 K with the actual P=70·10⁹, or 6·d_model ≈ 49 K in B1’s parametric form (same formula, the gap is only parametric-vs-actual P). What caching buys is absolute cost: each step c× cheaper, total O(c³) → O(c²) — not a different attention/weights balance. (The roofline is different, though: cacheless decode is a prefill ⇒ compute-bound past ~590 tok, vs AI=g, never, with the cache — see below.)

The roofline asymmetry: prefill is compute-bound, decode is memory-bound — and exactly when. Both fall out of one comparison: a kernel’s arithmetic intensity (AI = FLOPs per byte read from HBM) against the GPU’s ridge point I* (the AI above which it is no longer memory-starved). A forward pass has two reads with independent AIs — the weight read (linear term) and the KV read (attention term); the algebra below does each separately, and the governing rule is the same for both: AI = how many compute ops reuse each byte you loaded.

# Ridge point — hardware constant (peak compute / peak bandwidth, no MFU): I* = peak_FLOPs / HBM_BW = 989·10¹² / 3.35·10¹² ≈ 295 FLOP/byte [D — H100 SXM, dense BF16] # AI of the WEIGHT read alone (the LINEAR term), n tokens sharing ONE weight read: AI_lin(n) = 2·P·n / 2·P = n FLOP/byte # numerator 2 FLOP/param/token (MAC); denominator 2 bytes/param (FP16) — different 2’s, they cancel # Prefill ALSO carries the quadratic attention term; past s* = 6·d_model ≈ 49 K (B1) it dominates the FLOPs: AI_prefill(s) = (2·P·s + 4·n_layers·s²·d_model) / 2·P ≈ s only while s ≪ 49 K; super-linear above it compute-bound ⇔ AI > I* ⇔ n > 295 (attention only pushes prefill deeper in — AI = n is a floor)
  • Prefill: n = s — all prompt tokens share one weight read. From the linear term alone AI ≈ s, so prefill is compute-bound for any s > 295 tok; and once s > 49 K the quadratic attention term takes over (B1), so AI climbs faster than s — AI = s is a conservative floor, never an over-estimate. The 13 K I cited is not a threshold — it is Tin_mean, the workload mean prompt length (a given, §1), ~44× past the ridge. This is why every prefill crossover (B1, A-family) is fought in FLOPs and why prefill is the scarce roofline you offload (A4).
    Why the prefill crossover is 295, not 590 — min, not max, of the two terms’ thresholds. The weight matmuls cross at s>295 (AI=s); the attention kernel only crosses at s>590 (AI≈s/2). But the weight matmuls are the dominant cost of prefill — attention is 2·n_layers·s·d_model/P of the FLOPs, <1% until s approaches the 49 K B1 crossover — so once the dominant term crosses (295), prefill-as-a-whole is compute-bound; the attention kernel lagging to 590 is a sub-1% sliver. max(590) answers the stricter “every kernel compute-bound” question, not the practical “is prefill compute-bound” one. (Moot for the workload: s=13 K is past both.)
  • Decode: n = B — B sequences each advance one token, sharing one weight read ⇒ AI = B. At B=1, AI=1, sitting exactly 295× below the ridge: the 17.5 GB/GPU weight read (~5.22 ms) is ~295× the ~18 µs of per-token compute at peak — deeply memory-bound. Here the linear estimate errs the other way and stays safe: decode attention never dominates the realistic workload (B2), while the growing KV read only adds to the byte denominator — both push decode further below the ridge, never above it. Batching that weight read (C1) is the only lever.
  • The decode compute crossover is B* = I* ≈ 295 (AI = B, never 2·B — by the same algebra as prefill the 2·P cancels; a naive 2·B would mis-place it at 148). Below B*, more batch is “free” (you under-use a weight read already paid for); above it, decode turns compute-bound and more batch only adds latency. But this fleet never reaches B*: Bdecode is capped at 106 by HBM capacity (C1), well below 295, so decode stays structurally memory-bound and the binding ceiling is the 30 ms TPOT SLO, not the roofline.
  • The KV read (attention term) is a separate roofline. The bullets above are the weight read; the attention term reads the KV cache instead, so it has its own AI. What sets it is how many queries reuse each loaded KV element — decode reuses it once (→ exactly g, the GQA group size), prefill reuses it s times (→ ~s/2). The query count differs by phase, and that flips the verdict:
    # DECODE — B sequences, EACH advancing 1 query over its OWN c-token cache: AI_attn_decode = FLOPs / bytes FLOPs = B · 4·n_layers·c·(n_q_heads·d_head) # query side: ALL n_q_heads work (QKᵀ + P·V, MAC=2) bytes = B · 4·n_layers·c·(n_kv_heads·d_head) # KV side: only n_kv_heads stored (K,V × 2 B) = (n_q_heads·d_head) / (n_kv_heads·d_head) # B, n_layers, c, d_head ALL cancel = n_q_heads / n_kv_heads = g = 64/8 = 8 FLOP/byte [D] # B cancels top & bottom ⇒ batch does NOT change attention AI: each sequence reads its OWN # KV (no sharing). Contrast the weight read, where B sequences SHARE one load ⇒ AI = B (grows). # PREFILL — s queries reuse each KV element (FlashAttention loads K,V once): AI_attn_prefill = FLOPs / bytes FLOPs = 4·n_layers·s²·d_model # s queries × s keys (QKᵀ + P·V) bytes ≈ 8·n_layers·s·d_model # Q,K,V,O each s·d_model elems × 2 B (FP16); KV read once = (4·n_layers·s²·d_model) / (8·n_layers·s·d_model) # n_layers, d_model, one s cancel = s / 2 [D]
    • Decode attention can NEVER reach the ridge — with the KV cache. The c cancels (FLOPs and bytes both scale with context), so AIattn = g = 8 is a constant — ~37× below I* = 295 at every c and every B. Batch doesn’t help (each sequence reads its own KV, no sharing across the batch); a longer context only adds bytes, pushing it deeper in. Structurally memory-bound — the only lever is the byte denominator (GQA/MQA, KV quant).
      The cache is what makes this true. The memory-bound verdict is a consequence of caching K,V, not a property of decode itself: the cache trades recompute FLOPs for a byte read, which is exactly what pins the AI at g. Without a KV cache, each step would re-run the prefill over the whole c-token history (O(c²)/step, like a fresh prefill) — AI ≈ c/2, compute-bound past ~590 tok. So decode is memory-bound precisely because we cache; the cache converts a quadratic recompute into a linear read.
    • Prefill attention is compute-bound, and unbounded in s. AIattn = s/2 > I* for any s > ~590 tok — below every real prompt — and it climbs as s grows: ≈ 6.5 K FLOP/byte at the s = 13 K workload mean, ≈ 500 K at 1 M (where prefill is essentially all attention — the 871 s of B1).
    • Both prefill terms point the same way. Weight read AI = s, attention AI = s/2 — both compute-bound, both grow with s. That is why long context is fixed by context parallelism (spread the FLOPs, Iter 9), not more bandwidth.
  • Why this forces P/D disaggregation (Iter 2). A colocated task can only satisfy one roofline at a time: prefill wants B=1 (TTFT-safe), decode wants B≫1 (to amortize the weight read). Splitting the pools lets prefill stay compute-bound at B=1 and decode independently raise B to spend the idle HBM bandwidth.

Decides: why decode optimization targets bandwidth (batch the weight read, crossover C1), not attention FLOPs — the long-output regime that would flip this is a separate scenario (L3).

Family C — the decode batch hits its ceiling

Decode at B=1 reads the whole weight shard to make one token (MBU ~1%). Batching amortizes that read — until either the latency SLO or HBM capacity stops it. Two crossovers bound how far decode throughput can go.

C1 · Decode batch vs TPOT SLO & HBM capacity — Bdecode

Iter 6 / §12 · Bdecode [param]

Q: how wide can the continuous decode batch run before the per-token tick breaks the 30 ms TPOT SLO?

The weight read is fixed per tick; the KV read grows with B. The tick is the sum:

βper_decode_tok = weights_per_GPU / HBM_BW = 17.5 GB / 3.35 TB/s ≈ 5.22 ms [D] kv_per_seq_per_GPU = (Tin_mean + Tout_mean/2) × KVper_tok/8 = 13.2 K × 40 KB ≈ 528 MB [D] tick(B) = βper_decode_tok + B × kv_per_seq_per_GPU / HBM_BW # At the chosen batch (HBM-capacity bound = GKV / avg_kv_per_active_slot = 450 / 4.22): Bdecode ≈ 106 tick(106) = 5.22 ms + (106 × 528 MB / 3.35 TB/s) = 5.22 + 16.7 ≈ 22 ms < 30 ms SLO [D]

Two ceilings meet near 106: the HBM-capacity bound (Bmax,paged ≈ 106, itself unlocked by PagedAttention — static-max would cap at ~18) sets the batch, and the resulting tick (22 ms) lands just under the TPOT SLO so the KV-read term is the binding constraint on any further growth. The per-task throughput multiplier is thr(106)/thr(1) = (106/22 ms)/(1/5.22 ms) ≈ 4 818/192 ≈ ~25× — not the full 106×, because the tick grew ~4× as KV reads came to dominate.

Decides: the realized continuous-batch width (Iter 6’s ~25× per-task decode throughput, the page’s biggest fleet-$ lever) — and that pushing B higher trades throughput for TPOT, so the next throughput lever has to be orthogonal to batch size (→ C2).

C2 · Speculative decode vs vanilla — the accept-rate break-even

Iter 7 / §13.5  the orthogonal lever once C1’s batch is maxed.

Q: at what token accept rate α does speculative decoding stop being a win over vanilla autoregressive?

A draft proposes γ=4 tokens; the target verifies all γ+1=5 in one weight read. The expected accept run is a truncated geometric; the honest speedup discounts the draft forwards, the larger verify KV read, and reject-rollback waste:

E[accept] = (1 − αγ+1) / (1 − α) = (1 − 0.7⁵) / (1 − 0.7) = (1 − 0.16807) / 0.3 ≈ 2.77 tok / target weight-read [D — α=0.7, γ=4] accept_rate_speedup_factor = E[accept] × discount = 2.77 × 0.72 ≈ 2.0× [D] # Break-even: speedup = 1 ⇒ E[accept] = 1/discount ≈ 1.39 ⇒ α well below 0.7

Output is provably identical to vanilla sampling at any α (the verify uses the exact target distribution) — so a low α is never a correctness or quality bug, only a throughput regression. At α far below 0.7 (out-of-distribution traffic, adversarial input) the rollback waste exceeds the accept run and spec-decode goes slower than vanilla.

Decides: the per-tenant fallback policy — monitor accept rate, and automatically revert that traffic slice to vanilla autoregressive once its α crosses below the break-even where the speedup factor = 1.

Two families, one moral. Family A is bandwidth-vs-FLOPs (move bytes or recompute them); Families B and C are FLOPs-vs-FLOPs and bandwidth-vs-latency. Notice the reuse: A1’s 1 870-token crossover appears verbatim three times (A1 tier ladder, A3 α-sensitivity base, A4 pool write-back gate), because it is one piece of algebra — transfer cost equals re-prefill cost — applied wherever the system must choose between holding bytes and rebuilding them.
