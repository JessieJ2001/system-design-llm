Multi-Stage Pipelining for GEMM on Hopper

Why a tiled matrix-multiply (GEMM) streams its inputs through SMEM (shared memory) in a circular ring of stages, how stage count hides load latency, and the fixed-SMEM tradeoff that pulls you back toward memory-bound. Grounded on the H100 (8×H100 SXM5, Llama-3-70B serving fleet).

Acronyms used here — SM Streaming Multiprocessor · SMEM shared memory (the per-SM scratchpad) · CTA Cooperative Thread Array (a thread block, runs on one SM) · MMA Matrix-Multiply-Accumulate (the tensor-core instruction) · TMA Tensor Memory Accelerator (Hopper’s async bulk-copy engine) · HBM High-Bandwidth Memory (off-chip GPU DRAM) · GEMM General Matrix Multiply · AI Arithmetic Intensity (FLOP per byte read from HBM).

1. The execution model: SM, CTA, warp

TL;DR. Before any pipelining: a GPU runs a kernel as a grid of CTAs; CUDA places each CTA onto one SM, and the SM runs its threads in warps of 32 driven by warp schedulers. Two rules from this model drive everything later on this page — CTAs that fit run in parallel, the rest run in waves, and warps interleave to hide latency.

The hierarchy: grid → CTA → warp → thread

  • SM (Streaming Multiprocessor) — the GPU’s physical compute cluster; an H100 has 132. Each SM owns its 256 KB register file, 228 KB SMEM, and 4 warp schedulers.
  • CTA (Cooperative Thread Array = thread block) — the unit CUDA schedules onto one SM. Once placed it stays there until it finishes (no migration), and its registers + SMEM are reserved for its whole life.
  • warp — 32 threads, the hardware’s scheduling and execution granularity. All 32 share one program counter and run in lockstep (SIMT — Single-Instruction, Multiple-Threads); divergent branches execute under a mask.
  • A warp is assigned to one of the SM’s 4 warp schedulers for its entire life and never migrates to another.

CTAs that fit run in parallel; the rest run in waves

  • How many CTAs are co-resident on an SM is the occupancy, capped by registers, SMEM, and hard slot limits (H100: ≤ 64 warps and ≤ 32 CTAs per SM).
  • CTAs that fit launch and run truly in parallel across all 132 SMs.
  • CTAs that don’t fit are serialized into waves — a waiting CTA launches only when a resident one retires and frees its slot + resources.

Fig 1 — CTAs that fit (occupancy) run in parallel as a wave; overflow CTAs serialize into later waves as resident CTAs retire.

Warp scheduling = interleaving, to hide latency

  • Each warp scheduler holds a pool of warps and issues one instruction per cycle from whichever warp is ready; the next cycle it may pick a different warp.
  • That interleaving is the latency-hiding mechanism: when a warp stalls on an HBM load (hundreds of ns), the scheduler issues from another ready warp instead of idling. Same principle as the multi-stage buffers in §4 — overlap a stall with useful work — but at thread granularity.
  • Misconception to drop: warps do not run one-to-completion-then-next. They interleave instruction-by-instruction; “warp 0 finishes before warp 1 starts” is only the degenerate single-warp / no-stall case, not the model.
  • Two real parallelisms underneath: the 4 warp schedulers issue in parallel, and within a warp all 32 threads are lockstep-parallel.

Fig 2 — A warp scheduler issues one instruction/cycle from whichever warp is ready; a stalled warp’s gap is filled by other warps. This thread-level latency hiding is the same idea as the multi-stage buffers later on the page.

Between CTAs: parallel but unsynchronized

  • Different CTAs make progress concurrently but with no ordering guarantee — CTA 0 may start or finish before or after CTA 1.
  • CUDA forbids inter-CTA synchronization by default, for two reasons:
    1. The grid may launch more CTAs than fit on-chip → later waves haven’t started while earlier ones run → a barrier across all CTAs waits forever.
    2. Even within one wave there is no ordering guarantee between CTAs.
  • The portability trap this kills: a kernel with hand-rolled inter-CTA sync might “work” on an H100 (all CTAs co-resident, barrier succeeds) yet deadlock on a smaller RTX 4060 (CTAs run in waves, the barrier waits on CTAs that never launched). Making inter-CTA sync impossible by default eliminates this entire bug class — the kind that passes testing on a big GPU and fails mysteriously in production on a small one.
  • Opt-in exception: Cooperative Groups grid.sync() — but it requires a launch that guarantees all CTAs are co-resident (an occupancy-limited launch), precisely so the wave deadlock can’t happen.
Why this section is first. Occupancy — how many warps/CTAs sit on an SM — is set by the same registers + SMEM budget that §4’s 192 KB stage configuration collapses to 1 CTA/SM. So the SMEM you spend on pipeline stages directly throttles how many warps the scheduler has to hide latency with. “Warps interleave to hide stalls” (here) and “buffers stage ahead to hide loads” (§3) are the same idea at two levels of the machine.

2. The problem: tiles want to be big, SMEM is small

TL;DR. A GEMM tile should be as large as possible — reuse grows with tile size — but the operand tiles don’t fit in SMEM, so you stream them through in chunks. Streaming introduces a load-to-use latency that has to be hidden.

Why large tiles — reuse

  • Dimensions, defined first. M = rows of the output tile, N = its columns, K = the shared contraction dimension that is summed over (the inner index of the matrix product). The operands are A (left, the row source), B (right, the column source), and the output C = A·B. To compute one output tile C[M×N] you read an A-strip [M×K] and a B-strip [K×N] from HBM.
  • Each A-row is reused N times (once per output column); each B-column is reused M times. So the tile’s arithmetic intensity is AI = 2·M·N·K / (2·(M·K + K·N)) = M·N / (M+N) — it grows with tile size.
  • Small tile → small AI → the same operand strips get re-read from HBM for neighbouring output tiles → memory-bound. A 128×128 tile reuses each byte 64× (16384/256); a 64×128 tile only 42.7× (8192/192).

Why you can’t just load the whole tile

  • The contraction dimension K (for a Llama-3-70B feed-forward layer, K = d_model = 8192, where d_model is the model’s hidden/embedding width — the vector length carried between layers) is far too long to stage whole. You slice it into K_chunk (a contiguous slice of the K axis, e.g. 64 elements) and walk the K-loop. This is the GEMM case — feed-forward and QKV/O (Query / Key / Value / Output) projections; attention stages a different axis, covered in §5.
  • While the tensor cores are doing the MMA on the chunk in buffer i, you must not overwrite that buffer — but you can stream the next chunk into a different buffer. That is the whole idea.

Fig 3 — One output tile is fed by two operand strips. Reuse (hence AI) grows with tile size; K is sliced into K_chunk bands streamed one at a time.

3. The mechanism: multi-stage pipelining

TL;DR. Keep a ring of n SMEM buffers (“stages”). (A ring = circular buffer: the n stage-buffers are reused cyclically — buffer = stage mod n — so once the MMA drains buffer i it is freed and the next load wraps back into it. “The ring fills” later means all n buffers are loaded-but-not-yet-drained, so the next load has nowhere to land and must wait.) The TMA engine streams future K_chunks into free buffers while the tensor cores grind the current one. With enough stages, the MMA lane runs back-to-back with no bubbles — the load-to-use latency is fully hidden. This is exactly the Stages parameter in CUTLASS (NVIDIA’s open-source CUDA C++ template library for GEMM kernels); on Hopper the copy engine is TMA + mbarrier (a shared-memory barrier object the TMA signals on completion; older Ampere did it with cp.async, the async global→shared copy instruction).

The key fact that makes it work

  • A single load has high latency (~400–600 ns to first byte) but the memory system can keep many loads in flight at once. So loads overlap each other — they are not serialized.
  • A buffer is occupied from the moment its load is issued until its MMA completes — that is (load_latency + compute_time) of wall-clock. To feed the tensor cores a new chunk every compute_time you need enough buffers to cover that whole window.

Fig 4 — Same workload, 1 vs 2 vs 3 buffers (load latency ≈ 1.5× compute). More stages → more loads in flight → the tensor-core lane closes its bubbles. Once back-to-back, you are compute-bound and extra stages buy nothing.

4. The tradeoff: stages vs tile size, under a fixed SMEM budget

TL;DR. More stages hide more latency — but SMEM is fixed at 228 KB/SM on the H100. Buying more stages out of that fixed budget forces a smaller tile, which lowers reuse and drags you back toward memory-bound. The sweet spot is the fewest stages that fully hide the latency.

How many stages? The rule of thumb

n_buffers = ceil(load_latency / compute_time_per_tile) + 1 # +1 for safety / latency jitter H100 sanity check: compute_per_tile (128×128×64 MMA) = 2·128·128·64 = 2.1 MFLOP ≈ 2.1e6 / 7.5e12 (per-SM tensor-core rate) ≈ 280 ns load_latency (HBM → SMEM) ≈ 450–600 ns n_buffers = ceil(500/280) + 1 = 2 + 1 = 3 → 3–6 stages typical for Hopper GEMM

The SMEM math (BF16 — bfloat16, 2 bytes/element)

Per stage you hold one A-chunk and one B-chunk. The tile one CTA computes is BLOCK_M × BLOCK_N — the concrete chosen values of M and N for a single CTA’s output tile (here 128×128): bytes/stage = (BLOCK_M·K_chunk + K_chunk·BLOCK_N) · 2.

Fig 5 — The fixed 228 KB SMEM ceiling is the budget you spend on stages × tile. Stages buy latency-hiding; spending too many forces a smaller tile (reuse ↓) or kills occupancy.

Why the 192 KB row fails — occupancy, not the accumulator

  • It is not that the output accumulator gets crowded out of SMEM — the accumulator lives in registers. A 128×128 FP32 (32-bit float) accumulator is 16 384 floats spread across the CTA’s threads, held in the 256 KB/SM register file (a budget separate from the 228 KB SMEM), and the epilogue reuses the freed input SMEM.
  • The real cost is occupancy. 228 KB / 192 KB → 1 — only one CTA resides per SM. With a second CTA resident the warp scheduler hides a stalled CTA by running the other’s warps (§1); at occupancy 1 that escape hatch is gone, and any pipeline stall surfaces directly as idle tensor cores — plus there is no margin left for the epilogue or other SMEM users.
  • Net: “192 KB is too much” is right, but the mechanism is occupancy collapse, not accumulator spill.

5. Where the stages live: GEMM vs attention

The K_chunk picture above is the GEMM (feed-forward, QKV/O projections). Attention pipelines over a different axis, and the head dimension is a compile-time constant, not a streamed loop.

Attention symbols, defined first. Q, K, V are the query / key / value matrices, each shaped [s × d_head] — where s = sequence length (number of tokens) and d_head = per-head width = d_model ÷ n_heads (n_heads = the number of attention heads). Kᵀ is K transposed; P = softmax(Q·Kᵀ) is the attention-probability matrix; the attention output is P·V.

TL;DR. In a plain GEMM the contraction dim is d_model = 8192 — long, so it’s sliced into K_chunk and the multistage ring walks the K-loop. In attention’s Q·Kᵀ the contraction dim is d_head — small (64–256), held whole as a kernel template constant. What flash attention actually streams over is the KV sequence length (the P·V contraction), via the online-softmax block loop.

GEMM (feed-forward, projections) — most of prefill FLOP

  • Contracts over K = d_model = 8192 → long → chunked into K_chunk.
  • The multistage ring buffers successive K_chunks (the picture in §3).

Attention Q·Kᵀ then P·V

  • Q·Kᵀ contracts over d_head — small, not chunked; the kernel is templated on HEAD_DIM ∈ {64,128,256} and holds it whole.
  • The streamed axis is the KV sequence (the P·V contraction over context length): the online-softmax loop marches BLOCK_N down the keys/values — that’s where the stages live.

Fig 6 — The K_chunk staging of §3 is the GEMM. Attention holds d_head whole and stages over the KV sequence instead.

On d_head. It is model-dependent, not fixed at 128: 64 (GPT-2/BERT), 128 (Llama 2/3, Mistral, GPT-3 — the de-facto standard for large LLMs), 192 (DeepSeek MLA — Multi-head Latent Attention: 128 no-RoPE + 64 RoPE — RoPE = rotary position embedding — on the QK term), up to ~256 (about the practical ceiling, where the Q/K SMEM tiles start to force a split). The 128 in the sketches is the value for this fleet’s Llama-3-70B (64 heads × 128 = d_model 8192). The structural claim — “d_head is the un-chunked inner dim; the sequence is the streamed axis” — holds for any of these values.

6. Payoff: the steady weight trickle, and the limit on fused decode prefetch

TL;DR. Multi-stage pipelining is exactly why a compute-bound prefill streams its weights as a smooth, low-rate flow instead of a burst — and the leftover SMEM/occupancy headroom from §4 is precisely what decides whether a fused kernel can also prefetch decode KV onto the same SMs (the Option-A question in the design-choice doc).

  • Why prefill streams at ~124 GB/s, steadily. A prefill GEMM at AI ≈ 8000 only needs 989 TFLOP/s ÷ 8000 ≈ 124 GB/s of HBM to keep the tensor cores fed (989 TFLOP/s = the H100’s peak BF16 tensor-core throughput). The multistage ring is what turns the weight load into that continuous trickle that stays one step ahead of the MMA — not a load-then-idle burst. The gap to 3.35 TB/s (≈ 96% of HBM bandwidth) is spare.
  • Why that matters for fusing decode prefetch. Decode is memory-bound and would love that 96% spare bandwidth. But to co-run it in one kernel you must stage decode KV in the same SMEM. If the prefill GEMM already wants 6 stages × a big tile and has pinned occupancy to 1 CTA/SM (§4), there is no SMEM left — so you fall back to spatial SM-partitioning instead of true fusion.
  • So this analysis is the binding constraint on Option A. “Run a little prefill on the decode pool, hidden under the memory-bound tick” is bandwidth-feasible by a wide margin — but it is SMEM-bound. §7 proves the bandwidth feasibility and pins down exactly when the SMEM fits; §8 names the three ways to actually share the GPU.

7. Can prefill & decode run in one kernel? Feasibility

TL;DR. Two separate questions. (a) Is there room? — mostly no if both run their max-tile “fast” configs, because a fast prefill kernel alone grabs almost the whole 228 KB SMEM; it fits only if you re-budget the prefill tile/stages, use a small head dim, or move to bigger-SMEM hardware. (b) Is it worth it? — yes: prefill is compute-bound and leaves ~94% of HBM bandwidth idle, so the memory-bound decode load hides in that idle bandwidth with multiple-× headroom. The limiter is (a), not bandwidth.

7a. Is the shared memory enough? — usually not, for the fast configs

  • Why a fast prefill kernel eats the budget. To feed the MMA units it wants a big tile and a deep stage ring (§3–§4). A 128×128, d_head=128, 3-stage flash-attention prefill CTA costs 32 KB (Q) + 3×64 KB (K,V ring) = 224 KB — almost the entire 228 KB SMEM/SM. (A stage here is one slot of the §3 multistage ring — one in-flight (K,V) sequence block (a BLOCK_N tile of keys+values, §5), not an algorithm phase and not QKᵀ/softmax/PV. “3 stages” = 3 KV blocks buffered, so the MMA on block i overlaps the TMA load of blocks i+1 and i+2; each stage = one K block (128·128·2 = 32 KB) + one V block (32 KB) = 64 KB.) A decode KV stage is ~32 KB. 224 + 32 = 256 > 228 → they don’t co-reside. So yes, the claim is true for max-tile kernels.
  • When it is enough:
    1. Re-budget the prefill ring — drop prefill to 2 stages (160 KB) + a 1-stage decode tile (32 KB) = 192 KB, fits with ~36 KB to spare. Cost: prefill gives up one stage of latency hiding (fine if it is comfortably compute-bound).
    2. Small head dim — at d_head=64 a 3-stage prefill CTA is ~112 KB; + decode ~16 KB = 128 KB, fits easily.
    3. Shallow decode staging — decode’s query is 1 token (tiny); if you stage few KV blocks, its SMEM footprint is small and slots into prefill’s slack.
    4. Bigger-SMEM hardware — Blackwell raises per-SM SMEM and adds 2-SM thread-block clusters with DSMEM (Distributed Shared Memory) reachable by TMA multicast, ~doubling the staging pool so both fast configs fit. This is the “may change in the future” the original note hand-waved.
  • Net: fusing in one CTA needs a deliberate SMEM re-budget (trade prefill tile/stages) or newer hardware — it is not free.

Fig 7 — The SMEM collision. Two fast kernels overflow 228 KB; fusion needs a re-budget (fewer prefill stages, smaller head dim) or Blackwell-class SMEM.

7b. Can the decode load hide under the prefill compute?

  • Disjoint hardware units. Prefill bottlenecks the tensor-core / MMA pipe; a decode KV read bottlenecks the memory pipe (load/store units + TMA + the HBM controller). On Hopper, TMA async copies proceed independently of the MMA units (the very §3 mechanism) — so the load genuinely runs in the shadow of the compute.
  • The roofline gap that makes it free. (The roofline model plots achievable FLOP/s against arithmetic intensity AI: below the ridge intensity AI* — where the HBM-bandwidth ceiling and the compute ceiling meet — you are memory-bound; above it, compute-bound.) Prefill is compute-bound, so it leaves almost all of HBM idle; decode is memory-bound, so it leaves most of the tensor cores idle. Co-running them reclaims both idle resources at once — that is the efficiency win, not just “it fits.”

Which pipe is the real limiter? The formula above is the HBM pipe. But the decode load touches three shared resources — only the last binds:
  • HBM bandwidth (HBM→SMEM via TMA) — ~94% idle under prefill (above). ✓ hides.
  • SMEM bandwidth (writing the KV into shared memory) — per-SM SMEM BW ≈ 31 TB/s ÷ 132 ≈ 235 GB/s; the decode write is the per-SM share of the idle HBM rate, ≈ 3.16 TB/s ÷ 132 ≈ 24 GB/s = ~10% of SMEM BW — and on Hopper the TMA write uses a dedicated async path that does not stall the MMA’s SMEM operand reads. ✓ hides.
  • SMEM capacity (a free buffer to land in) — a buffer must be unused for the TMA to target, which is exactly the §7a budget. ✗ the binding constraint.
So the shared-memory bandwidth pipe hides the decode load fine; it is shared-memory capacity (§7a) that limits fusion, not bandwidth.
“Won’t SMEM fill up if compute can’t drain it fast enough?” — right question, and it splits by stream. “Compute too slow to drain its loads” is the definition of compute-bound — which is prefill, not decode. The fused kernel carries two data streams with opposite producer–consumer balance:
  • Prefill’s own loads (compute-bound — the slow-consumer case). The MMA is the slow consumer, so the TMA loads do outrun it: the prefill ring fills, the TMA then throttles (waits for a free buffer), and HBM goes ~idle. This is not a bug — it is the compute-bound steady state, and it is exactly why ~94% of HBM bandwidth is free. So “SMEM fills, HBM can’t load more” is true for prefill, and it is the precondition the overlap exploits.
  • The decode KV we hide (memory-bound — the fast-consumer case). This stream is drained by decode’s MMA, not prefill’s. Decode attention is ~1–2 FLOP/byte, so draining KV at the idle-HBM rate (3.16 TB/s) needs only ~2 × 3.16e12 ≈ 6 TFLOP/s = ~0.6% of the 989 TFLOP/s tensor cores. The consumer easily outpaces the feed, so this ring does not back up.
The 42 GB is never stored in SMEM — it streams through a ring of a few 32 KB buffers (HBM → SMEM → MMA → evict). By Little’s law, saturating HBM needs only BW × latency = (3.35e12 ÷ 132) × 500e-9 ≈ 12.7 KB in flight per SM (~one buffer). So SMEM capacity binds — but as “is there room for decode’s small ring once prefill’s ring has already filled ~224 KB” (§7a), not as a slow-drain stall.
Little’s law, quickly. A queuing-theory result: in any stable system the average number of items in flight equals throughput × time-in-system, L = λ · W — where L = items in the system at once, λ (lambda) = arrival rate / throughput (items per second), W = time each item spends in the system (its latency). The memory form (the bandwidth-delay product) maps these to L = bytes in flight, λ = bandwidth, W = access latency: to run a pipe at bandwidth BW when each access takes latency to complete its round trip, you must keep BW × latency bytes outstanding at once — fewer in flight and the pipe idles waiting, so sustained throughput falls below BW.
  • Analogy: a checkout that scans 1 item/sec, but each shopper takes 30 s to walk up (latency). To keep the cashier never idle you need ~30 shoppers already in flight; fewer and it waits.
  • Here: BW = per-SM HBM share ≈ 25 GB/s, latency ≈ 500 ns → 25e9 × 500e-9 ≈ 12.7 KB must be outstanding to saturate HBM — about one 32 KB buffer.
  • This is why the multistage ring (§3) and many resident warps (§1) exist: both are just ways to hold BW × latency requests in flight so the pipe never starves.

Fig 8 — Prefill and decode bottleneck on opposite pipes; fusing them reclaims each one’s idle resource (illustrative utilizations, per the formula above).

Closing the proof — does the SMEM capacity fit?

The bandwidth and time are free (above) — but per the three-pipe split, the binding resource is SMEM capacity. So the proof only closes if decode’s ring fits beside prefill’s in one SM’s 228 KB. It does — if prefill cedes one pipeline stage (or uses a small head dim, or runs on Blackwell). At prefill’s max config there is no room. So the overlap is possible, but not free.

What is a “prefill ring” / “decode ring”? A computation’s ring is the set of n SMEM stage-buffers it cycles through to stream operands from HBM — one operand block per buffer, the next loaded ahead while the MMA consumes the current one (§3). The prefill ring holds prefill’s streamed blocks (its flash-attention K,V sequence tiles, or its GEMM K_chunks); the decode ring holds decode’s streamed KV blocks. They are physically distinct SMEM regions, so a fused kernel must carve both out of the same per-SM 228 KB — the budget fight below.

So every buffer count and KB figure on this page — the 224 KB prefill ring, the 12.7 KB Little’s-law number, the budget below — is per SM.   • How small can decode’s ring be? Little’s law (§7b) says only ~12.7 KB need be in flight per SM to saturate HBM. Double-buffered (one block draining while the next loads), that is a 2-stage ring ≈ 32–64 KB (a decode K+V block is 16–32 KB at d_head=128, BLOCK_N=64).

Dropping prefill 3→2 stages spends §4’s “+1” jitter-safety stage: prefill keeps the bare minimum (2) to hide its load latency, so it stays compute-bound but with less margin against load-latency jitter — a small but real tax. Verdict. The overlap is bandwidth-free (§7b) and capacity-feasible (here) — but only with a deliberate re-budget that cedes prefill’s safety stage (or a small head dim, or Blackwell). At prefill’s max 3-stage config it does not fit. So “provably possible” = yes, contingent on that trade — it is not free.

8. The co-execution taxonomy — three ways to share the GPU

TL;DR. Once you accept prefill and decode want to run together, there are exactly three structural ways to place them on the hardware — split SMs, split CTAs, or split warps — trading rising implementation complexity for tighter, more dynamic overlap.

Reading the three

  • 1 · Split SMs (different kernels). Give prefill k SMs and decode 132−k via MPS, stream priority, or Hopper green contexts. Simplest, but the boundary is static: it can’t track the instantaneous prefill/decode mix, and whichever side drains first leaves its SMs idle. As you noted — this is barely different from just running them in parallel.
  • 2 · Split CTAs (same kernel). One persistent/fused kernel launches CTAs, each tagged prefill or decode; the hardware scheduler interleaves them across all SMs, so load is balanced GPU-wide with no stranded partition. This is the level POD-Attention operates at — SM-resource-aware fusion of prefill and decode attention.
  • 3 · Split warps (one CTA). The tightest form: inside a single CTA, producer warps issue TMA loads of decode KV while consumer warps run the prefill MMA, sharing one SMEM ring — exactly the producer/consumer warp specialization of FlashAttention-3, and the intra-device overlap of NanoFlow. It achieves the genuine overlap proven in §7b, but is the hardest to write and is the one gated by the SMEM budget of §7a. Sarathi-Serve’s chunked-prefill is the batch-level relative — it merges decode tokens into the prefill batch so one kernel pass covers both.
