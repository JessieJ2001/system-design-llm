The through-line — and the order to read in

One idea runs through every page: the GPU has fixed shapes it is fast at → kernels are written to fit those shapes → the serving stack schedules work to match the kernels → models are designed to ease that. Each arrow is a forcing function, and the last one loops back. This map is the only place that synthesis lives — each link points at the page that owns it; nothing here is re-derived.

  • ① GPU fixed shapes → Kernels — the roofline and FLOP/byte ratio decide what a kernel must look like to be fast.
  • ② Kernels → Serving — kernel shape-rigidity (CUDA-graph buckets, minimum viable batch, FlashDecoding's KV split) dictates how the scheduler and pools may behave.
  • ③ Serving → Models — serving pain (KV bandwidth, all-to-all cost, context length) motivates the architectural answers.
  • ④ Models → GPU (feedback) — GQA/MoE/sliding-window are designed to relax the original GPU constraint; the same K0/K9/K2 pages, now read as the model's reply.

KV cache & reuse

K2

KV Cache — the two roles

Role 1, intra-request (universal): the cache exists because it turns O(L³) generation into O(L²) — the classic space-trades-compute move; then size it, page it, quantize it, tier it, evict it. Role 2, inter-request ("prefix caching", optional): reuse computed KV across requests and route each request to the GPU that holds it — the full routing-for-cache-hit spectrum (round-robin → sticky → prefix-hash → approx/precise KV-aware → hybrid → disaggregated/cross-region) and which workload picks which. The block table / refcount / CoW is the one indirection both roles stand on. (Absorbs the former K3 Caching & Routing page.)

Core scenarios

L1

Prefill/Decode Disaggregation — Frontier MoE

Serve a ~400B-param MoE behind separate TTFT & TPOT SLOs at high concurrency. The hardest scenario — it stress-tests the whole foundation (K5/K6/K7/K8/K9/K20/K2).

Planned
L2

Multimodal Vision-Language Serving

VLM serving images + text; image-token blowup; repeated-image dedup. Naive (vision tower per replica) → encoder-disaggregated pool + embedding cache.

Planned
L3

Long-Context (1M-token) Serving

1M-token context alongside ordinary short traffic on a shared fleet; KV for one request ≫ one GPU's HBM; mixed short/long must not starve.

Planned
L4

Multi-Tenant LoRA Platform

Thousands of customer LoRA adapters over one shared base on one fleet; per-tenant isolation & fairness; cold-adapter swap; scale-to-zero per adapter.

Planned
L5

Spot-GPU Autoscaling Fleet

Cost-sensitive bursty serving on spot/preemptible GPUs; 10:1 diurnal; reclaim ≤30s notice; SLO holds through reclaim & cold start.

Planned

Extension scenarios

L6

Embedding / Retrieval Service

Encoder-only at scale: no autoregressive decode, no KV cache, throughput-bound not latency-bound — a completely different roofline (strongest contrast).

Planned
L7

Model-Version Canary & Rollout

Roll a new model/quant into a live fleet without dropping in-flight long generations or regressing quality; KV-aware draining, double-fleet, eval gate.

Planned
L8

Both CoT regimes on one endpoint with a reasoning-effort knob; the bimodal unknown-at-admission output-length mixture is the spine.

Planned
L9

High-reuse agent platform where routing — not the GPU kernel — sets the cost; hot-shared-prefix skew; the locality-vs-load-balance tension.
